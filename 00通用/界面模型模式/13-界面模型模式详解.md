<a name="p01-list-report"></a>
## 1. List Report（列表报告）

### 解决的核心问题

**大量对象中快速定位和批量处理**。

这是企业系统最常用的模式。核心矛盾是：列表可能有数万行，用户需要在几秒内找到目标对象。

### 问题根源

```
场景：门店收银员查询今日订单
数据库里有：订单主表、支付表、会员表、商品明细表、物流表……
用户想要：找到"那笔1280元的订单，看看有没有发货"

如果按数据库结构展示 → 用户需要在4个tab之间来回切换
按List Report展示 → 一个表格行聚合了所有关键信息，一眼定位
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **筛选区** | 核心字段默认展开，高级筛选折叠 | Hick's Law：默认选项少，降低决策时间 |
| **行摘要** | 每行展示最能区分对象的字段（订单号、状态、金额、会员） | 一眼区分，不需要点进去 |
| **状态徽章** | 用颜色区分优先级 | 扫描速度：红色=异常，绿色=完成，橙色=待处理 |
| **主操作外露** | 高频动作放在行末按钮 | 减少操作路径 |
| **次操作收敛** | 低频动作放"更多"菜单 | 避免按钮过多（见反模式§4.3） |
| **批量选择** | Checkbox + 批量操作栏 | 减少重复点击 |

### 典型案例

```
场景：门店收银员查询和处理今日订单

┌─ 筛选区 ─────────────────────────────────────────────┐
│ [搜索订单号/会员名___] [门店 ▼] [状态 ▼] [日期范围 ▼] [查询] │
└──────────────────────────────────────────────────────┘

┌─ 批量操作栏 ──────────────────────────────────────────┐
│ 已选 2 项   [批量发货] [批量打印] [批量导出]          │
└──────────────────────────────────────────────────────┘

┌─ 列表 ──────────────────────────────────────────────┐
│ ☑ 订单号        门店      会员    金额      状态        时间     操作 │
│ ────────────────────────────────────────────────────── │
│ ☑ SO202607120001 人民广场  周杰伦  ¥1,280  [已完成]   10:32  [详情][发货] │
│ ☑ SO202607120002 浦东店    林俊杰  ¥456    [待发货]   10:15  [详情][发货] │
│ ☐ SO202607120003 徐汇店    王力宏  ¥89     [退款中]   09:58  [详情][退款] │
└──────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§2 信息分层**：列表行是一级信息（一眼看到），详情页是二级信息（点击进入）
- **§3 状态轴**：筛选条件本质是状态过滤器——"待发货"是一个状态子集
- **反模式§4.3**：表格按钮过多 → 高频外露、低频收敛

<a name="p02-object-page"></a>
## 2. Object Page（对象详情页）

### 解决的核心问题

**把散落在多个数据库表中的数据，聚合成一个用户能理解的业务对象**。

### 问题根源

一个订单在数据库里涉及 8+ 张表：订单主表、商品明细表、支付表、物流表、优惠券表、发票表、操作日志表……

```
❌ 反模式：字段按数据库表结构平铺
  用户看到：支付信息（来自payments表）→ 物流信息（来自logistics表）→ 商品明细（来自order_items表）
  问题：用户不知道这些表之间的关系，不知道先看什么

✅ Object Page：按用户判断顺序重新组织
  用户看到：订单摘要 → 为什么待发货（因为已支付+已锁库）→ 怎么处理（发货/退款）
  优势：用户脑子里有一个完整的"订单"概念，不需要自己拼接
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **对象头部** | 名称/编号 + 主状态 + 关键指标 + 主操作 | 用户一眼知道这是哪个对象、什么状态、现在能干什么 |
| **状态解释** | 说明"为什么是这个状态" | 消除用户疑惑（"待发货？为什么？库存够吗？"） |
| **内容分区** | 按业务理解顺序排列，不按数据库表 | 字段有逻辑关联，不是碎片化的 |
| **关联列表** | 关联对象作为独立区块 | 相关对象不割裂，但也不喧宾夺主 |
| **时间线** | 记录状态变化和操作历史 | 解释"怎么变成这样的"（见§5） |

### 对象头部标准结构

```
┌─────────────────────────────────────────────────────────────────┐
│  订单 #SO202607120001                                           │
│  状态：[待发货]  已支付  已锁库                                 │
│  金额：¥ 1,280.00    门店：人民广场店    会员：周杰伦           │
│  操作：[发货] [退款] [打印]                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 分区设计示例（零售订单详情）

```
1. 订单摘要（核心信息）
2. 商品明细（用户关心买了什么）
3. 支付信息（金额来源）
4. 配送/自提信息（用户关心怎么收货）
5. 优惠和积分（如果有优惠要透明）
6. 状态流转（见Timeline模式）
7. 操作日志（见Audit Log模式）
```

### 与第13章的呼应

- **§1 数据-心理失配**：Object Page 是把数据库语言翻译成用户心理模型的"翻译器"
- **§3 状态三问**：是什么（状态徽章）/ 为什么（时间线前置节点）/ 怎么办（主操作）
- **§5 可追溯性**：分区结构让用户理解对象形成过程

<a name="p03-timeline"></a>
## 3. Timeline（时间线）

### 解决的核心问题

**让用户理解"这个状态是怎么变成这样的"**——即业务事实的可追溯性。

### 问题根源

```
❌ 反模式：详情页只有当前状态
  "订单金额 ¥1,350，用户问为什么不是 ¥1,280"
  客服：不知道……系统里就显示 ¥1,350

✅ Timeline：展示状态形成过程
  "07-12 10:32 系统 支付成功 ¥1,280
   07-12 11:00 人工 客服张三备注 顾客使用追加优惠券
   07-12 11:05 系统 实付金额变更 ¥1,280 → ¥1,350"
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **三层事件区分** | 系统事件 / 人工操作 / 业务状态分别标注 | 系统自动的和人工操作的责任链不同 |
| **状态变化记录前后值** | 金额 ¥1,280 → ¥1,350 | 只显示最终值无法解释原因 |
| **操作人和时间** | 客服张三 / 07-12 11:00 | 责任追溯 |
| **关联单号和附件** | "附件：优惠券截图" | 有据可查 |

### Timeline 标准字段

```
时间 · 事件类型 · 操作人/系统 · 前状态 → 后状态 · 原因 · 关联单号 · 备注/附件
```

### 典型案例

```
┌─ 时间线 ──────────────────────────────────────────────────────┐
│                                                           ●    │
│  07-12 14:30  系统  订单完成                                     │
│                    状态：已发货 → 已完成                          │
│                                                           ●    │
│  07-12 11:05  系统  实付金额变更              [系统]              │
│                    ¥1,280 → ¥1,350                             │
│                    原因：顾客使用追加优惠券抵 ¥70                 │
│                                                           ●    │
│  07-12 11:00  人工  客服张三  添加备注           [人工]            │
│                    备注：顾客使用了追加优惠券，需要核实            │
│                                                           ●    │
│  07-12 10:33  系统  库存锁定                                   │
│                    SKU001 库存 -2                              │
│                                                           ●    │
│  07-12 10:32  系统  支付成功                   [系统]            │
│                    微信支付 ¥1,280                              │
└───────────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§5 时间线：可追溯性的界面表达**：三层事件结构（系统/人工/业务）
- **§5.3 状态流转的完整表达**：时间+事件类型+操作人+前状态+后状态+原因
- **反模式§4.5**：详情页没有历史 → 增加 Timeline

<a name="p04-related-lists"></a>
## 4. Related Lists（关联列表）

### 解决的核心问题

**一个业务对象关联的其他对象，如何在不喧宾夺主的情况下完整呈现**。

### 问题根源

```
订单关联：会员（1个）、支付单（1个）、优惠券（多个）、物流（多个）、发票（0-1个）、售后单（0-N个）……

❌ 反模式：把所有关联都堆进详情页
  → 详情页变成另一个复杂列表页，用户不知道看哪里

❌ 反模式：关联数据完全割裂
  → 用户需要去菜单里找"支付单管理"，输入订单号筛选
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **关系说明** | 明确说明关系类型（"关联会员"、"关联支付单"） | 用户知道这些数据为什么出现在这里 |
| **关键字段** | 只展示最关键字段，完整查询跳独立页 | 信息密度控制，不喧宾夺主 |
| **有限操作** | 关联列表内只做查看、解绑等轻量操作 | 复杂操作跳转到独立页 |
| **大数据量处理** | 默认分页或只展示最近5条 | 避免关联列表本身变成列表页 |

### 典型案例

```
┌─ 关联列表：支付单 ──────────────────────────────────────────┐
│ 支付单号          支付方式     金额     状态        时间      │
│ PAY202607120001  微信支付    ¥1,350  已支付    07-12 10:32  │
│                                             [查看详情]       │
└─────────────────────────────────────────────────────────────┘

┌─ 关联列表：优惠记录 ────────────────────────────────────────┐
│ 优惠类型      优惠名称          优惠金额     时间            │
│ 店铺券        新客满100减20     -¥20.00    07-12 10:32     │
│ 积分抵扣      100积分抵 ¥1.00   -¥1.00     07-12 10:32     │
│ 追加优惠      追加优惠券抵 ¥70   -¥70.00    07-12 11:05     │
│                                         [查看全部优惠]       │
└─────────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§1 信息聚合**：Related Lists 是把关联对象"聚"到主对象下的机制
- **§2 信息分层**：关联列表是二级信息，不是详情页的核心
- **反模式§3.2**：关联列表字段过多，变成另一个复杂列表页

<a name="p05-master-detail"></a>
## 5. Master-Detail（主从布局）

### 解决的核心问题

**连续处理多个对象时，减少页面跳转带来的上下文切换**。

### 问题根源

```
场景：客服处理今日退款申请，需要逐个审核

❌ 反模式：每次点进去处理完，跳回列表，再点下一个
  → 列表 → 点详情 → 处理 → 返回列表 → 点详情 → 处理 → ……
  → 每次跳转丢失上下文（列表筛选、排序、已处理标记）

✅ Master-Detail：选中对象在右侧展示详情，处理完直接切下一个
  → 左侧列表（筛选后的待退款）→ 右侧详情（处理面板）→ 下一个
  → 上下文保留，处理路径缩短
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **当前选中项高亮** | 左侧列表当前行明确标识 | 用户始终知道自己在哪个对象上 |
| **状态同步** | 右侧处理后，左侧摘要更新（如金额变更） | 用户知道结果已生效 |
| **键盘导航** | 支持 ↑↓ 键切换对象 | 熟练用户效率提升 |
| **快速操作** | 详情区提供主操作，不需要跳转 | 减少操作路径 |

### 布局结构

```
┌─ 左侧：列表栏 ───────────────┬─ 右侧：详情栏 ──────────────────┐
│  [筛选] 待退款订单 (3)        │  订单 #SO202607120003            │
│ ───────────────────────────── │  状态：[退款中]                 │
│  ● SO202607120003 王力宏 ¥89  │  金额：¥89.00                  │
│    SO202607120004 张学友 ¥200 │                                 │
│    SO202607120005 刘德华 ¥50  │  退款原因：顾客误拍              │
│                               │  [同意退款] [驳回]              │
└───────────────────────────────┴─────────────────────────────────┘
```

### 与第13章的呼应

- **§2 认知负荷**：减少页面跳转就是减少上下文切换，降低工作记忆负担
- **§3 状态轴**：列表栏本质是状态过滤（只显示"退款中"）
- **适用边界**：详情复杂度高时（如Object Page分区多），应跳转独立页而非Master-Detail

<a name="p06-dashboard"></a>
## 6. Dashboard（角色工作台）

### 解决的核心问题

**角色每天开始工作时，如何在几秒内了解全局状态、快速进入任务**。

### 问题根源

```
❌ 反模式：首页只是菜单快捷入口
  "店长打开系统 → 找菜单 → 点'订单管理' → 看待发货 → 回到首页 → 点'库存管理' → 看预警"
  → 每次都要自己拼凑"今天要关注什么"

✅ Dashboard：角色视角的"任务驾驶舱"
  "店长打开系统 → 首页直接展示：销售额28,450(待处理3笔)、待发货12单、库存预警5个"
  → 用户一眼知道今天要干什么
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **待办和异常优先** | 红色/橙色区域放在最显眼位置 | 用户优先处理紧急事务 |
| **KPI 可解释** | 指标有口径、时间范围、目标值 | 数字不再是无意义的装饰 |
| **指标可钻取** | 点击指标进入明细 | 指标不只是"看到了"，还要能"追下去" |
| **按角色设计** | 店长、财务、客服看到不同内容 | 避免"千人一面"的无效首页 |

### 标准结构

```
┌─ 角色问候 ─────────────────────────────────────────────────┐
│ 早上好，张店长  ·  华东区 > 上海区 > 人民广场店  ·  07-12    │
└────────────────────────────────────────────────────────────┘

┌─ 今日待办与异常 ───────────────────────────────────────────┐
│ [🔴 紧急] 待发货 12 单    [🟠 预警] 库存预警 5 个           │
│ [🟡 注意] 退款申请 3 单    [🟢 正常] 设备状态 8/8 在线     │
└────────────────────────────────────────────────────────────┘

┌─ 核心指标 ─────────────┐ ┌─ 指标趋势 ─────────────────────┐
│ 销售额  ¥28,450  +12.3% │ │        销售趋势（折线图）         │
│ 订单数    156 单   -3   │ │  ▲ 28k                         │
│ 客单价   ¥182    +5.2%  │ │      ↗                           │
│ 完成率    85%    目标33k│ │  ───────────────  目标 33k      │
└────────────────────────┘ └─────────────────────────────────┘
```

### 与第13章的呼应

- **§2 信息分层**：Dashboard 的信息分层是横向的（按优先级排列指标卡片）
- **§7 钻取链路**：每个指标卡片必须可钻取到明细
- **反模式§4.2**：首页堆图表 → 图表很多但没有行动入口

<a name="p07-analytical-list-page"></a>
## 7. Analytical List Page（分析列表）

### 解决的核心问题

**图表和明细使用同一筛选条件，从指标分析直接下钻到业务明细**。

### 问题根源

```
❌ 反模式：图表和明细查询口径不一致
  图表显示"华东区销售额 ¥100万"
  明细表格筛选"华东区"只显示 ¥95万
  → 用户无法信任数据，无法对账

✅ Analytical List Page：图表和明细用同一筛选上下文
  左侧图表 → 点击"人民广场店" → 右侧明细表自动过滤到人民广场店
  → 数据同源，指标和明细可对账
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **统一筛选条件** | 图表和明细表使用同一筛选上下文 | 数据口径一致，可对账 |
| **图表可点击过滤** | 点击图表某区域过滤明细 | 从宏观到微观无缝切换 |
| **明细能解释指标** | 明细字段足以解释指标来源 | 不是"看到数字，不知道为什么" |
| **导出区分** | 明确导出的是汇总还是明细 | 避免用户导错数据 |

### 布局结构

```
┌─ 筛选条件 ─────────────────────────────────────────────────┐
│ [门店 ▼] [日期范围 ▼] [商品类目 ▼]         [查询] [重置]    │
└────────────────────────────────────────────────────────────┘

┌─ 指标摘要 ─────────────────────────────────────────────────┐
│ 销售额 ¥28,450  订单数 156  客单价 ¥182  完成率 85%        │
└────────────────────────────────────────────────────────────┘

┌─ 图表分析 ─────────────────────────────────┬─ 明细表格 ──┐
│  门店销售占比（饼图）                       │  订单号  门店    金额   状态   │
│                                           │  ───────────────────────── │
│  人民广场 42%  ●●●●●●●●●●                 │  SO001  人民广场 ¥12,000 完成  │
│  浦东店    33%  ●●●●●●●●                  │  SO002  浦东店   ¥9,500  完成  │
│  徐汇店    25%  ●●●●●●                    │  点击人民广场行自动过滤 ↓   │
│                                           │  SO003  人民广场 ¥1,200  待发  │
└────────────────────────────────────────────┴───────────────┘
```

### 与第13章的呼应

- **§7 钻取链路**：从汇总指标到明细数据的无缝链路
- **§7.3 Drill-down 的五个设计要求**：每层保留筛选上下文
- **反模式§4.6**：报表不能钻取 → 图表只能看，不能深入

<a name="p08-worklist"></a>
## 8. Worklist（待办工作列表）

### 解决的核心问题

**某个固定状态下的待办任务集合，让用户连续处理而不被打断**。

### 问题根源

```
场景：客服处理今日退款申请

❌ 反模式：用普通列表处理
  列表包含"全部状态"（已完成+退款中+已驳回）
  → 用户需要每次手动筛选"只看退款中"
  → 已被处理的订单干扰视线

✅ Worklist：只展示单一状态
  列表只包含"退款中"状态的订单
  → 用户打开页面就是任务队列，不需要自己筛选
  → 专注处理当前状态，减少状态切换
```

### 与 List Report 的区别

| 特征 | List Report | Worklist |
|---|---|---|
| 列表内容 | 全量 + 筛选器 | 单一状态 |
| 用户目标 | 查找特定对象 | 连续处理待办 |
| 筛选器 | 需要用户主动筛选 | 无需筛选，打开就是任务 |
| 典型用户 | 查询者 | 执行者 |

### 典型案例

```
┌─ 今日退款处理 ──────────────────────────────────────────────┐
│  共 3 单待处理    已处理 0/3    批量 [同意全部] [导出]      │
│ ─────────────────────────────────────────────────────────── │
│  ☐ SO202607120003 王力宏 ¥89  退款原因：误拍  申请时间 09:58 │
│  ☐ SO202607120004 张学友 ¥200 退款原因：商品损坏  申请时间 09:45 │
│  ☐ SO202607120005 刘德华 ¥50  退款原因：地址错误  申请时间 09:30 │
│ ─────────────────────────────────────────────────────────── │
│  [同意选中] [驳回选中] [查看详情]                            │
└────────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§3 状态轴**：Worklist 是"固定状态"的最直接体现——页面即状态
- **§2 认知负荷**：单一状态减少筛选负担，专注处理
- **保守规则§3**：连续处理 → Worklist 或 Master-Detail

<a name="p09-simple-form"></a>
## 9. Simple Form（简单表单）

### 解决的核心问题

**轻量录入（1-7个字段），如何在不离开当前页面的情况下快速完成**。

### 问题根源

```
场景：运营人员为商品添加二级分类

❌ 反模式：跳转到复杂编辑页
  用户当前在"商品列表"页 → 点"新增分类" → 跳转新页面 → 填完 → 返回
  → 路径长、容易忘记之前在干什么

✅ Simple Form：Modal 或 Drawer
  用户当前在"商品列表"页 → 点"新增分类" → 弹窗/抽屉 → 填完 → 关闭
  → 3个字段的轻量任务，在上下文中完成
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **字段数量** | 最多 7 个字段 | 超过7个认知负荷过大，应考虑 Sectioned Form |
| **单任务** | 一个表单只完成一个明确任务 | "保存分类"不是"确定"——标题表达业务动作 |
| **字段级校验** | 错误直接定位到字段，不要顶部通篇提示 | 减少错误修正时间 |
| **不离开上下文** | 提交后刷新原列表，不跳转 | 用户记得之前在做什么 |

### 典型案例

```
┌─ 新增分类 ──────────────────────────────────────────────────┐
│                                                             │
│  分类名称 *                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 进口零食                                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  上级分类                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 零食饮料 ▼                                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  分类编码 *                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ SNACK_IMPORT                                          │   │
│  └─────────────────────────────────────────────────────┘   │
│  自动生成，或手动填写（编码唯一）                            │
│                                                             │
│                    [取消]  [保存分类]                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§6 表单**：表单是"业务承诺"，Simple Form 是最小粒度的承诺
- **§6.3 表单复杂度**：1-3字段 → Modal Form / Simple Form
- **反模式§4.4**：简单表单里混入复杂配置 → 一个表单只服务一个明确任务

<a name="p10-sectioned-form"></a>
## 10. Sectioned Form（分段表单）

### 解决的核心问题

**大量字段（10-40个）如何通过分区降低认知负荷**。

### 问题根源

```
❌ 反模式：所有字段铺成一列
  用户面对40个字段，不知道先填什么、哪些必填、字段之间的关系
  → 认知过载，容易填错

✅ Sectioned Form：按业务含义分区
  用户面对4个分区，每个分区8-10个字段
  → "基础信息"填完 → "价格库存"填完 → ……
  → 分段推进，每段有明确目标
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **按业务含义分区** | 不按数据库表结构 | 用户脑子里就是按业务场景理解的 |
| **分区有锚点导航** | 40+字段有侧边锚点跳转 | 长表单快速定位 |
| **必填字段靠前** | 前两个分区集中必填字段 | 用户先完成最重要的 |
| **互斥字段动态处理** | 当前模式下无关字段隐藏或禁用 | 避免互斥字段同时校验报错 |
| **审计信息只读** | 创建人、更新时间放最后 | 只读信息不干扰录入 |

### 分区结构示例（零售商品资料）

```
┌─ 锚点导航 ─┬─ 分区内容 ────────────────────────────────────┐
│ ▼ 基础信息   │  名称、编码、类目、品牌、规格、单位            │
│   价格库存   │  零售价、会员价、库存、预警值                  │
│   渠道管理   │  销售渠道、上下架状态                         │
│   扩展信息   │  标签、图片、备注                             │
│   审计信息   │  创建人、创建时间、更新时间（只读）           │
└─────────────┴────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§2 认知负荷**：分区是将40个字段的认知负荷拆分为4个8-10字段的分段
- **§6 表单复杂度**：10-40字段 → Sectioned Form
- **反模式§4.4**：巨型表单 → 分区 + 锚点 + 草稿

<a name="p11-drawer-form"></a>
## 11. Drawer Form（抽屉表单）

### 解决的核心问题

**从列表页快速编辑少量字段，如何不跳转页面就完成**。

### 问题根源

```
场景：收银员需要给订单添加备注

❌ 反模式：跳转详情页
  列表页 → 点详情 → 找到备注输入框 → 填完 → 保存 → 返回列表页
  → 跳转2次，处理1个字段

✅ Drawer Form：右侧滑出抽屉
  列表页 → 点备注图标 → 右侧抽屉展开 → 填完 → 关闭
  → 上下文保留，处理完继续看列表
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **任务必须轻** | 抽屉内只做少量字段的查看或编辑 | 复杂详情不应塞进抽屉 |
| **不嵌套** | 抽屉内不要再打开抽屉 | 层级混乱，用户找不到出口 |
| **保存后刷新** | 关闭抽屉后原列表行数据更新 | 用户知道结果已生效 |
| **未保存提醒** | 关闭前对未保存内容做提示 | 防止误关闭丢失输入 |

### 与 Simple Form 的区别

| 特征 | Simple Form | Drawer Form |
|---|---|---|
| 触发来源 | 新增（创建新对象） | 编辑（修改已有对象） |
| 内容 | 从零录入 | 已有数据回填 |
| 宽度 | 固定（Modal宽度） | 可变（Drawer可展开更大） |
| 关闭行为 | 点外部或按钮关闭 | 需明确"取消/保存" |

### 与第13章的呼应

- **§6 表单复杂度**：3-10字段 → Drawer Form / Simple Form
- **§2 认知负荷**：抽屉保留了列表上下文，减少跳转

<a name="p12-modal-form"></a>
## 12. Modal Form（弹窗表单）

### 解决的核心问题

**短确认、填写原因、危险动作二次确认——如何在弹窗中安全承载轻量任务**。

### 问题根源

```
场景：退款确认

❌ 反模式：弹窗里放大表单
  退款弹窗里包含：退款金额、退款原因、退款方式、退款账户……
  → 弹窗变成另一个表单页，用户不知道是确认还是填写

✅ Modal Form：只承载确认意图
  弹窗标题："确认退款 ¥128.00"
  弹窗内容："退款将退回原支付方式，预计1-3个工作日到账"
  输入框：退款原因（必填）
  按钮：[取消] [确认退款（危险样式）]
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **字段少** | 确认类弹窗只填1-2个字段 | 不是另一个表单 |
| **后果明确** | 说明操作的影响范围 | 用户在确认前知道后果 |
| **危险样式** | 高风险操作用红色按钮 | 与普通操作区分 |
| **不嵌套** | 弹窗不打开弹窗 | 层级爆炸 |
| **关闭行为明确** | 必须有取消按钮，关闭不自动提交 | 用户不会误操作 |

### 典型案例

```
┌─ 确认退款 ──────────────────────────────────────────────────┐
│                                                             │
│  订单号：SO202607110045                                    │
│  原支付金额：¥ 128.00                                      │
│  ━━━━━━━━━━━━━━━━━━━━━━━━                                 │
│  退款金额：¥ 128.00  （原路退回）                           │
│                                                             │
│  退款原因 *                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 顾客误拍，已与顾客确认                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ⚠ 退款后将退回原支付方式，预计1-3个工作日到账             │
│                                                             │
│                    [取消]  [确认退款]                       │
│                         (危险样式)                          │
└─────────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§4 操作风险**：Modal Form 是危险操作的标配承载
- **§6 表单**：Modal 是"确认意图"而非"录入意图"
- **反模式§4.4**：弹窗里放大表单、弹窗嵌套弹窗

<a name="p13-wizard"></a>
## 13. Wizard（向导模型）

### 解决的核心问题

**长流程多步骤，如何让用户每次只专注于当前步骤，降低整体认知负荷**。

### 问题根源

```
场景：开通一个新门店

❌ 反模式：单页巨型表单
  表单包含：基础信息（5字段）+ 业务配置（15字段）+ 硬件配置（8字段）+ ……
  → 用户面对30+字段，不知道从哪开始，不知道字段之间的关系

✅ Wizard：拆成4个步骤
  步骤1：填基础信息（5字段）→ 下一步
  步骤2：配置业务规则（15字段）→ 下一步
  步骤3：配置硬件（8字段）→ 下一步
  步骤4：预览全部配置 → 确认提交
  → 用户每步只面对一个明确任务
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **一步一任务** | 每个步骤只解决一个问题 | 聚焦，降低决策难度 |
| **局部校验** | 提交前做当前步骤的校验 | 不等到最后一步才发现前面的错误 |
| **跨步骤依赖可见** | 步骤之间有数据依赖要标注 | 用户知道为什么某些字段是灰色的 |
| **支持草稿** | 中途关闭可保存进度 | 防止长流程数据丢失 |
| **最后一步总览** | 预览全部配置 + 风险提示 | 用户在提交前最后确认 |

### 步骤结构示例

```
┌─ 步骤指示器 ──────────────────────────────────────────────┐
│  [1 基础信息] → [2 业务配置] → [3 硬件配置] → [4 预览确认] │
│        ✓             ○             ○             ○       │
└──────────────────────────────────────────────────────────┘

┌─ 步骤 1/4：基础信息 ─────────────────────────────────────┐
│  门店名称 *        [人民广场店___]                         │
│  门店编码 *        [SHOP_PDH_001__]                        │
│  所属区域 *        [华东 > 上海 > 浦东___]                  │
│  详细地址 *        [上海市浦东新区___]                      │
│                                    [上一步] [下一步]       │
└──────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§2 渐进披露**：用户不需要一次性看完全部内容
- **§6 表单复杂度**：多步骤有顺序依赖 → Wizard
- **保守规则§4**：配置复杂 → Wizard
- **反模式§4.4**：单页巨型表单 → Wizard 拆解

<a name="p14-inline-edit"></a>
## 14. Inline Edit（行内编辑）

### 解决的核心问题

**表格中少量字段（如备注、排序号、状态）如何直接修改，减少操作路径**。

### 问题根源

```
场景：运营人员需要频繁调整商品排序号

❌ 反模式：每改一个排序号都要打开编辑页
  列表 → 点编辑 → 填写排序号 → 保存 → 返回 → 改下一个
  → 5次点击换一个数字

✅ Inline Edit：直接在列表行内修改
  列表 → 点击排序号单元格 → 直接输入 → Enter保存
  → 2次点击换一个数字
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **适合少量字段** | 单字段或2-3个关联字段 | 全字段行内编辑会丢失上下文 |
| **保存状态明确** | 编辑中/已保存状态可见 | 用户知道数据有没有被保存 |
| **字段级校验** | 错误贴近字段，不要顶部提示 | 修正成本低 |
| **高风险字段不用** | 金额、状态等高风险字段建议跳转编辑 | 行内修改没有确认环节 |

### 与第13章的呼应

- **§2 认知负荷**：减少操作路径就是减少认知切换
- **§6 表单**：Inline Edit 是表格场景下的"表单变体"
- **反模式§4.3**：表格里所有字段都可编辑 → 批量行内编辑没有冲突处理

<a name="p15-drill-down-report"></a>
## 15. Drill-down Report（钻取报表）

### 解决的核心问题

**汇总指标如何可解释——用户从数字能追到具体业务对象**。

### 问题根源

```
❌ 反模式：Dashboard 只展示大数字
  "华东区销售额 ¥100万"  →  用户问：哪类商品卖得好？哪个门店拖累了？
  → 页面无法回答，用户无法对账

✅ Drill-down：四层链路
  汇总 ¥100万 → 维度：按门店 → 门店明细 → 具体订单
  → 用户从宏观到微观完整追溯
```

### 钻取链路示例

```
第一层：汇总指标
  华东区销售额 ¥28,450

      ↓ 点击"华东区销售"下钻

第二层：按门店维度
  人民广场 ¥12,000  浦东店 ¥9,500  徐汇店 ¥6,950

      ↓ 点击"人民广场"下钻

第三层：订单明细
  SO202607120001 ¥1,280  SO202607120002 ¥980  SO202607120003 ¥850 …

      ↓ 点击"SO202607120001"下钻

第四层：业务对象（Object Page）
  订单详情 + 时间线 + 关联对象 → 完整解释"为什么 ¥1,280"
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **每层保留上下文** | 下钻后返回，上级筛选条件不变 | 用户不迷路 |
| **汇总和下级能对账** | 明细合计 = 上级数字 | 数据可信 |
| **最后一层落到业务对象** | 不能永远停在数字 | 追到可操作的实体 |
| **返回路径** | 面包屑或返回按钮 | 支持回退 |

### 与第13章的呼应

- **§7 钻取链路**：四层结构的完整设计
- **§7.2 Drill-down 五要求**：每层保留上下文、汇总可对账
- **保守规则§5**：指标需要解释 → Drill-down

## 16. Exception Center（异常中心）

### 解决的核心问题

**系统异常如何从"技术人员看日志"变成"业务人员可处理"**。

### 问题根源

```
❌ 反模式：异常只在日志里
  支付回调失败 → 写入 error.log
  技术人员：知道有错误，排查原因
  业务人员：完全不知道

结果：用户打电话投诉"支付了但订单没变化"，客服无法解释

✅ Exception Center：
  支付回调失败 → 进入异常中心
  → 业务人员看到"3笔支付异常待处理" → 点击处理 → 重新发起支付
  → 异常被业务人员消化，技术人员只处理真正无法解决的
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **异常分级** | 按严重程度分：ERROR / WARN / INFO | 优先处理高风险 |
| **绑定业务对象** | 异常关联到订单/会员/商品 | 异常不孤立，有上下文 |
| **提供处理动作** | 不只是展示错误码，要给操作 | 业务人员有明确行动 |
| **处理留痕** | 谁处理、什么时候、怎么处理的 | 异常可追溯 |
| **重复异常聚合** | 同一接口大量失败聚合成一条 + 数量 | 避免列表刷屏 |

### 典型案例

```
┌─ 异常中心 ─────────────────────────────────────────────────┐
│  全部(12)  支付异常(3)  库存异常(2)  同步异常(7)           │
│ ─────────────────────────────────────────────────────────── │
│  🔴 支付回调失败        SO202607120003  ¥89    10:32       │
│     错误码：TIMEOUT     支付渠道：微信支付                  │
│     [重新发起支付] [标记已处理] [查看详情]                   │
│ ─────────────────────────────────────────────────────────── │
│  🔴 支付回调失败        SO202607120004  ¥200   10:45       │
│     错误码：TIMEOUT     支付渠道：支付宝                    │
│     [重新发起支付] [标记已处理] [查看详情]                   │
│ ─────────────────────────────────────────────────────────── │
│  🟠 库存扣减失败        SKU001        库存不足  09:15      │
│     当前库存：0  需求：2                                   │
│     [补充库存] [忽略] [查看详情]                            │
└────────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§8 异常闭环**：五要素完整实现
- **§8.1 技术 vs 业务异常**：Exception Center 把技术语言翻译成业务语言
- **保守规则§6**：异常需要闭环 → Exception Center
- **反模式§4.7**：异常只在日志 → 建立异常中心

## 17. Draft Handling（草稿保存）

### 解决的核心问题

**长表单填写中途中断，如何恢复工作进度**。

### 问题根源

```
场景：运营人员编辑商品资料（40+字段），填了一半被叫去开会

❌ 反模式：中途关闭页面全部丢失
  用户：填写了15分钟的数据全没了，明天重新来
  → 产生挫败感，可能放弃填写

✅ Draft Handling：
  用户关闭页面 → 系统提示"检测到未保存内容，是否保存草稿？"
  → 用户选择保存 → 草稿进入草稿列表 → 下次进入可恢复
  → 中断不丢数据，用户可以无缝继续
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **退出时提示** | 浏览器关闭/返回前检测未保存内容 | 防止误操作丢失 |
| **自动草稿** | 每30秒自动保存草稿到服务器 | 断电/崩溃也能恢复 |
| **草稿列表** | 支持查看、恢复、删除 | 草稿不污染正式数据 |
| **提交时清理** | 提交成功自动删除关联草稿 | 草稿列表不堆积 |
| **草稿命名** | 支持用户给草稿起名，便于识别 | 多个草稿时知道哪个是什么 |

### 与第13章的呼应

- **§6 表单**：长流程保护——超过5分钟或10+字段必须支持草稿
- **反模式§4.4**：中途关闭页面全部丢失 → 支持草稿
- **§12 错误恢复原则**：草稿是错误恢复的具体实现

## 19. Background Task Center（后台任务中心）

### 解决的核心问题

**后台异步任务（批量导入、数据导出）如何让用户追踪进度并处理结果**。

```
场景：运营人员导入1000个商品

## 20. Permission Matrix（权限矩阵）

### 解决的核心问题

**权限配置如何可视化，如何让管理员批量设置而不漏项**。

### 问题根源

```
场景：管理员为10个角色配置50个菜单权限

❌ 反模式：逐个角色配置，每个角色单独页面
  角色A → 勾选菜单1、菜单2…… → 保存
  角色B → 勾选菜单1、菜单3…… → 保存
  问题：无法横向对比"菜单1对哪些角色开放"，容易出现权限漏洞

✅ Permission Matrix：行列交叉矩阵
  横向：角色（店长/收银员/财务……）
  纵向：菜单/操作（订单管理/退款审核/报表查看……）
  单元格：勾选/不勾选
  → 一眼看出所有权限分布，任何权限变化可预览影响范围
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **权限类型区分** | 查看/新增/编辑/删除分别设置 | 不是"全开或全关" |
| **继承和覆盖可见** | 半选状态说明继承关系 | 权限来源透明 |
| **影响预览** | 保存前展示本次变更影响的人员 | 防止误配造成安全事故 |
| **变更审计** | 每次保存记录变更前后 | 权限变更可追溯 |

### 典型案例

```

│ 订单新增    │  ✓    │   ✓   │   ✗   │  ✗   │              │
│ 订单编辑    │  ✓    │   ✗   │   ✗   │  ✗   │  ← 半选：部分门店  │
│ 退款审核    │  ✓    │   ✗   │   ✓   │  ✗   │              │
│ 报表查看    │  ✓    │   ✗   │   ✓   │  ✓   │              │
│ 退款金额>500│  ✓    │   ✗   │   ✓   │  ✗   │              │
│ ───────────┴───────┴───────┴───────┴──────┘              │
│                                     [取消]  [保存配置]     │
│  ⚠ 本次变更将影响 3 个用户                          [预览] │
└───────────────────────────────────────────────────────────┘
```

### 与第13章的呼应

- **§4 操作风险**：权限是高风险操作，必须有预览和审计
- **§6 配置特殊性**：权限变更必须预览影响范围

## 21. Audit Log（审计日志）

<a name="p24-reconciliation-workbench"></a>
## 24. Reconciliation Workbench（对账工作台）

### 解决的核心问题

**系统流水和第三方流水如何核对，差异如何闭环处理**。

### 问题根源

```
场景：财务每日对账，确认系统订单金额和微信/支付宝流水一致

❌ 反模式：差异只在报表里
  财务：导出系统数据 → 导出微信对账单 → Excel 手动核对
  问题：手工核对效率低，差异行不知道怎么处理

✅ Reconciliation Workbench：
  系统流水和第三方流水同屏展示，差异行自动标红
  差异行可选择"平账/挂账/转投诉"处理
  → 对账工作从手工变成半自动化
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **两方数据同屏** | 系统 vs 第三方同一行对齐 | 不需要手工对照 |
| **差异自动标红** | 金额不一致的行高亮 | 一眼看到差异 |
| **差异处理动作** | 平账/挂账/转投诉 | 差异有明确处理路径 |
| **对账结果日结** | 对账确认后进入日结流程 | 财务闭环 |

### 典型案例

```

## 25. Customer 360（客户全景视图）

### 解决的核心问题

**多渠道的客户数据如何归一呈现，让客服人员快速了解客户全貌**。

### 问题根源

```
场景：客服人员接到电话，需要快速了解这个会员的所有信息

数据来源：
  - 门店POS：小明在线下门店消费过3次
  - 微信商城：小明在线上下单过5次
  - 积分系统：小明有2000积分
  - 优惠券系统：小明有2张未使用优惠券
  - 售后系统：小明有过1次退货记录

❌ 反模式：客服在5个系统里来回切换
  → 耗时且容易遗漏关键信息

✅ Customer 360：
  一个页面聚合所有渠道数据
  客服一眼了解：小明是高频顾客，积分够兑换礼品，上次退货是因为尺码问题
  → 个性化服务有了基础
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **身份归一** | 手机号+会员卡+微信OpenID 合并为同一客户 | 不重复计算，不割裂记录 |
| **渠道区分** | 交易/权益/互动分区块展示 | 不同类型的信息不混在一起 |
| **标签分层** | 系统标签（如"高价值"）和人工标签（如"重要客户"）区分 | 标签来源透明 |
| **时间线聚合** | 所有交互事件按时间排列 | 理解客户历程 |
| **推荐动作** | 根据画像推荐下一步沟通策略 | 服务有方向 |
