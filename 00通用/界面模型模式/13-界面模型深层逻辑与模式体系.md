# 界面模型模式的深层逻辑与模式体系

## 0. 前言

这些界面模式不是为了"好看"或"现代"，而是为了解决企业信息系统（B2B/ERP/CRM）中的核心 UX 问题：**让用户在复杂业务中找得到、看得懂、做得对、不出错**。

本文档从认识论和方法论层面，剖析模式背后的深层逻辑，以及各模式之间的内在关系。

---

## 1. 核心矛盾：数据模型 vs 心理模型

### 1.1 两套语言的失配

企业系统的根本问题在于**两套语言的不匹配**：

```
数据库视角（按范式分解）：
  order_items  (order_id, sku_id, qty, price, discount...)
  payments     (order_id, channel, amount, status...)
  inventory    (sku_id, warehouse_id, qty, locked_qty...)
  → 一个业务事实散在多个表

用户视角（心理模型）：
  "我有一笔订单，1280元，待发货，顾客问什么时候到"
  → 用户脑子里是一个完整的业务对象，不是表
```

**所有模式的本质**是把散落的数据重新聚合成用户能理解的业务对象。界面模型就是数据库语言到业务语言的翻译层。

### 1.2 翻译的两个方向

| 翻译方向 | 对应模式 | 解决什么问题 |
|---|---|---|
| 数据库 → 业务对象 | Object Page、Related Lists | 信息聚合，把多表拼成"一个对象" |
| 操作 → 业务动作 | 表单族（Simple/Drawer/Modal/Wizard） | 把字段输入翻译成"创建/修改一个业务事实" |
| 状态 → 可见反馈 | Status Badge、Timeline、状态流转 | 把状态变化翻译成用户能追踪的事件链 |
| 汇总 → 可解释指标 | Dashboard、Drill-down、Analytical List | 把数字翻译成"发生了什么" |

## 2. 信息的分层决策

### 2.1 Hick's Law 与信息密度

Hick's Law 指出：选项越多，决策越慢。企业页面必须在"信息够用"和"信息过载"之间找到平衡点。这催生了**三层信息优先级**：

```
第一层（必须一眼看到）：
  - 对象身份：谁/什么
  - 主状态：现在什么情况
  - 核心数值：金额、数量、期限
  - 主操作：用户最可能要干什么

第二层（需要理解背景）：
  - 状态来源：为什么是这个状态
  - 关联对象：跟谁有关
  - 历史变化：怎么变成这样的

第三层（排查问题时才看）：
  - 完整字段
  - 技术细节
  - 审计日志
```

### 2.2 模式的信息分层差异

不同模式的核心差异，就在这三层信息的组合方式：

| 模式 | 第一层 | 第二层 | 第三层 |
|---|---|---|---|
| List Report | 表格行摘要 | 筛选条件 | 高级筛选/导出 |
| Object Page | 头部摘要 | 分区详情 | Timeline / 日志 |
| Dashboard | KPI 卡片 | 图表 | 明细钻取 |
| Wizard | 当前步骤 | 已填步骤 | 总览确认 |
| Exception Center | 异常摘要 | 影响对象 | 处理日志 |

### 2.3 信息分层的物理实现

信息分层通过**空间布局**和**渐进展开**来实现：

- **空间分层**：头部 > 内容区 > 底部 > 抽屉/弹窗，越靠上信息越重要
- **渐进展开**：默认折叠次要信息，点击或滚动展开完整内容
- **上下文保留**：用户查看详情后返回，列表的筛选和排序状态不变

## 3. 状态：模式的核心组织轴

### 3.1 B2B 与 B2C 的根本区别

这是理解企业 UI 最重要的一句话：

```
B2C（电商）：
  页面 = 功能集合
  "购物车" "我的订单" "个人中心"
  → 按功能组织，用户找功能

B2B（ERP）：
  页面 = 状态过滤器 + 状态解释 + 状态操作
  "待发货订单"（状态过滤）
  "为什么待发货"（状态解释）
  "发货"（状态操作）
  → 按状态组织，用户找状态下的任务
```

所有反模式的根源，就是没有以状态为中心：

- 一张表一个页面 → 状态信息割裂，用户不知道在哪个状态
- 巨型表单 → 字段平铺，用户不知道先填什么、哪些必填
- 报表不能钻取 → 只有状态结果，没有状态来源

### 3.2 状态的三问框架

好的状态设计必须同时回答三个问题：

| 问题 | 含义 | 界面表达 |
|---|---|---|
| **是什么** | 当前状态是什么 | 状态徽章、主操作按钮 |
| **为什么** | 为什么是这个状态 | 状态来源说明、时间线前置节点 |
| **怎么办** | 用户能做什么 | 主操作、受控的次操作 |

以订单详情为例：

```
状态 = "待发货"

是什么？→ 状态徽章显示"待发货"
为什么？→ 时间线展示"已支付(10:32) → 库存已锁定(10:33)"
怎么办？→ 头部显示"发货"按钮
```

### 3.3 状态与操作的耦合关系

在企业系统中，**状态决定操作**，而不是操作决定状态。这是 B2B 区别于 B2C 的第二个核心逻辑：

```
B2C：用户想干什么就干什么
  "加入购物车" → 随时可操作

B2B：用户能干什么由状态决定
  "发货"按钮 → 只有"已支付且库存已锁定"的订单才可点击
  其他状态下：按钮禁用 + 说明原因（如"库存不足，无法发货"）
```

模式中**操作分层**的设计（主操作/次操作/分区操作），本质上就是**按状态筛选后的操作子集**。

## 4. 操作的风险分层

### 4.1 企业操作不是平等的

B2B 系统里，一个操作可能影响金额、库存、权限、合同。风险天差地别。把所有操作平等对待是反模式的根源。

```
高风险操作（不可逆 / 影响金额或库存）：
  取消订单、退款、删除数据、批量导入、修改价格
  → 危险样式（红色）+ 明确文案 + 二次确认 + 审计记录

中风险操作（影响状态流转）：
  发货、审核通过、驳回、启用/停用
  → 主操作突出 + 确认提示 + 状态立即更新

低风险操作（局部影响）：
  打印、复制、导出、备注
  → 次操作收敛到更多菜单
```

### 4.2 危险操作的七要素

当高风险操作无法避免时，必须完整具备以下要素：

```
1. 危险样式：红色边框/红色背景，与普通操作明显区分
2. 明确文案：说明后果，如"取消后订单将作废，不可恢复"
3. 二次确认：弹窗要求用户再次输入或选择
4. 影响预览：展示本次操作影响的范围（如"将取消 3 个子订单"）
5. 取消入口：确认窗口必须提供"取消"按钮
6. 结果反馈：操作后立即展示结果，不是静默成功
7. 审计记录：操作人、时间、前后值必须记录
```

### 4.3 不可用操作的处理

当操作因状态/权限/业务规则不可用时，**必须说明原因**：

```
❌ 反模式：按钮置灰，无任何解释
✅ 正确做法：按钮置灰 + tooltip "库存不足，无法发货，请先补充库存"
```

## 5. 时间线：可追溯性的界面表达

### 5.1 企业系统必须回答"怎么变成这样的"

财务对账、问题排查、责任追溯，都依赖同一个能力：**追踪业务事实的形成过程**。

```
订单金额 1280 → 1350
  谁改的？→ 张三（运营经理）
  什么时间？→ 07-12 10:32:15
  为什么改？→ 备注：顾客使用了一张追加优惠券
  怎么改的？→ 手动编辑实付金额
  有依据吗？→ 附件：顾客聊天记录截图
```

### 5.2 Timeline 的三层事件

| 事件类型 | 触发来源 | 记录内容 | 用户 |
|---|---|---|---|
| **系统自动事件** | 支付回调、定时任务、同步 | 事件名、时间、结果 | 需要看到 |
| **人工操作事件** | 客服审核、运营修改 | 操作人、时间、前后值、原因 | 必须看到 |
| **业务状态事件** | 状态机流转 | 前状态 → 后状态、触发条件 | 核心关注 |

### 5.3 状态流转的完整表达

一个完整的状态流转记录应该包含：

```text
时间        事件类型    操作人     前状态     后状态     原因
07-12 10:32  系统      支付回调    -        已支付     微信支付成功，¥1280
07-12 10:33  系统      库存锁定    -        库存已锁定  SKU001 库存 -2
07-12 10:35  人工      客服张三    已支付    待发货     -
07-12 11:00  人工      客服张三    -        -         添加备注：顾客要求下午配送
07-12 14:00  系统      物流推送    待发货    已发货     快递单号 SF123456789
```

## 6. 表单：业务承诺的界面

### 6.1 表单不是字段录入容器

填表单不是"录入数据"，而是**创建或修改一个业务事实**。这个事实会被后续流程使用、被财务对账、被审计追溯。

```
订单退款表单  →  退款金额 128 元
  影响：顾客余额 +128、财务应收 -128、库存 +1、提成重新计算
  后续：可能被审计、可能被顾客投诉、影响门店 KPI
```

### 6.2 表单必须回答的七个问题

| 问题 | 含义 | 设计要点 |
|---|---|---|
| 正在创建/修改什么？ | 表单的业务意图 | 标题明确表达业务动作："保存分类"而非"确定" |
| 哪些字段必填？ | 业务约束而非技术约束 | 必填有业务理由，不是"系统要求必填" |
| 字段之间有什么依赖？ | 互斥、联动、计算 | 互斥字段按当前模式动态显示/禁用 |
| 当前输入影响什么？ | 业务影响范围 | 配置变更必须展示影响预览 |
| 错误如何发现和修正？ | 校验及时性 | 字段级校验 + 提交前预检，错误贴近字段 |
| 提交后发生什么？ | 下一步引导 | 成功/失败都有明确反馈和下一步入口 |
| 能中断后继续吗？ | 长流程保护 | 超过 5 分钟或 10+ 字段必须支持草稿 |

### 6.3 表单复杂度与承载方式

| 字段数量 | 推荐形式 | 原因 |
|---|---|---|
| 1-3 个 | Modal Form | 轻量确认，不离开上下文 |
| 3-10 个 | Drawer Form / Simple Form | 独立但不打断主流程 |
| 10-40 个 | Sectioned Form | 分区降低认知负荷 |
| 40+ 个 | Sectioned Form + 锚点 + 草稿 | 必要时拆成 Wizard |
| 多步骤有顺序依赖 | Wizard | 每步一个任务，跨步骤依赖可见 |

### 6.4 配置表单的特殊性

配置页和普通编辑页不同，它改变的是**系统规则**，而非单个业务对象：

```
普通编辑：修改一条记录，影响一个对象
配置变更：修改一条规则，影响一批对象

→ 配置变更必须有预览：展示本次变更影响多少对象
→ 配置变更必须有时间：支持立即生效 / 定时生效
→ 配置变更必须能回滚：支持恢复默认值或回滚到历史版本
→ 配置变更必须可审计：操作人、时间、变更前后值全部记录
```

## 7. 钻取链路：指标的可解释性

### 7.1 数字不解释就是噪音

Dashboard 上一个"销售额 ¥28,450"，用户看到后问："这个数字对吗？为什么比昨天少？哪类商品拖累了？哪个门店没达标？"

如果页面无法回答这些问题，指标就只是装饰。

### 7.2 钻取链路的四层结构

```
第一层：汇总指标（What）
  销售额 ¥28,450  ·  同比 +12.3%

第二层：维度拆解（Which）
  按门店：人民广场 ¥12,000 / 浦东 ¥9,500 / 徐汇 ¥6,950

第三层：明细数据（Who / What）
  人民广场店 → 订单明细表

第四层：业务对象（Why）
  具体订单 → 订单详情 → 状态时间线
```

### 7.3 Drill-down 的五个设计要求

1. **每层保留筛选上下文**：从门店下钻到订单时，门店筛选条件不变
2. **支持返回上一级**：面包屑或返回按钮，不丢上下文
3. **汇总和下级能对账**：点击明细合计，能回到汇总数字
4. **最后一层落到业务对象**：不能永远停在数字，要能进入 Object Page
5. **导出区分汇总和明细**：用户导出时必须知道导的是哪层数据

## 8. 异常闭环：从日志到业务处理

### 8.1 技术异常 vs 业务异常

这是企业系统最容易被忽视的反模式：

```
❌ 反模式：异常只在日志里
  系统：支付回调失败 → 写入 error.log
  技术人员：看到日志，排查原因
  业务人员：完全不知道有这个异常

✅ 正确做法：异常进入 Exception Center
  系统：支付回调失败 → 写入异常中心
  业务人员：看到"3 笔支付异常待处理" → 点击处理 → 重新发起支付
  技术人员：只处理业务人员无法解决的技术问题
```

### 8.2 异常中心的五要素

| 要素 | 内容 |
|---|---|
| **分类分级** | 支付失败 / 库存不足 / 同步超时 / 数据对账差异，分严重程度 |
| **绑定业务对象** | 异常必须关联到订单 / 会员 / 商品，不能孤立存在 |
| **提供处理动作** | 不只是展示错误码，要给出"重新发起支付 / 调整库存 / 忽略" |
| **处理过程留痕** | 谁处理了、什么时候、怎么处理、结果如何 |
| **重复异常聚合** | 同一接口大量失败，聚合为一条异常 + 数量统计 |

## 9. 模式选择的决策树

```
任务是什么？

├── 查找 / 筛选对象
│   ├── 大量对象、多维度筛选 → List Report
│   ├── 固定状态列表、连续处理 → Worklist
│   ├── 多层级结构（组织/类目） → Tree / Tree Grid
│   └── 需要分析指标 + 明细联动 → Analytical List Page
│
├── 理解一个对象
│   ├── 简单对象、少量字段 → Object Page 头部（无需分区）
│   ├── 复杂对象、多分区 → Object Page + 分区
│   └── 多对象横向关联 → Related Lists + 三栏布局
│
├── 操作对象
│   ├── 1-3 个字段、快速完成 → Drawer Form / Inline Edit
│   ├── 大量字段、按业务分区 → Sectioned Form
│   ├── 多步骤有顺序依赖 → Wizard
│   └── 确认 / 填写原因 / 小任务 → Modal Form
│
├── 查看整体状态
│   ├── 指标看板 → Dashboard + KPI Card
│   ├── 指标 + 可解释来源 → Drill-down Report
│   └── 异常处理闭环 → Exception Center
│
└── 配置规则
    ├── 简单开关 → Toggle + 影响说明
    ├── 复杂条件组合 → Rule Builder + 模拟运行
    ├── 数据可见范围 → Data Scope Config + 预览
    └── 权限分配矩阵 → Permission Matrix
```

## 10. 保守规则的底层原理

当不确定时按保守规则执行，背后的原理是最小认知负担原则：

| 保守规则 | 原理 | 错误做法 |
|---|---|---|
| 对象很多 → List Report | 用户需要先找到对象，再理解对象 | Dashboard 首页堆图表，用户找不到具体订单 |
| 对象复杂 → Object Page | 复杂对象需要分区解释，不能塞进弹窗 | Modal 展示全部字段，用户无法聚焦 |
| 连续处理 → Worklist | 减少页面跳转上下文切换 | 每次处理完跳转回列表再点下一个 |
| 配置复杂 → Wizard | 长流程拆步骤降低认知负荷 | 单页巨型表单，用户不知道填到哪了 |
| 指标需解释 → Drill-down | 数字不解释就是噪音 | Dashboard 只展示大数字，用户不知道涨跌原因 |
| 异常需闭环 → Exception Center | 技术日志业务人员无法处理 | 异常只在日志里，异常无法被业务人员发现 |

## 11. 反模式的系统性分类

所有反模式可以归结为**五个失配**：

| 失配类型 | 表现 | 对应反模式 |
|---|---|---|
| **数据-心理失配** | 字段按数据库结构排列，不按业务理解 | "一张表一个页面" |
| **信息-认知失配** | 不分层、不收敛、全量展示 | "首页堆图表"、"表格按钮过多" |
| **状态-操作失配** | 不以状态为中心组织界面 | "巨型表单"、"报表不能钻取" |
| **系统-业务失配** | 技术异常在日志里，业务人员看不到 | "异常只在日志" |
| **当前-历史失配** | 只有当前值，没有形成过程 | "详情页没有历史" |

## 12. 模式的认识论基础

| 原则 | 含义 | 在模式中的体现 |
|---|---|---|
| **Gestalt 完形原则** | 人脑倾向于把碎片聚合成整体 | Object Page 聚合对象，Related Lists 聚合关联，Timeline 聚合过程 |
| **认知负荷理论** | 工作记忆有限，超载即出错 | 信息分层、操作收敛、状态优先、Wizard 拆解 |
| **行动者-对象模型** | 用户是能动主体，对象是被操作目标 | 操作分层、受状态控制、高风险确认 |
| **可追溯性原则** | 每个业务事实必须能解释来源 | Timeline、Audit Log、版本历史、钻取链路 |
| **风险分层原则** | 不同操作风险不同，界面必须体现 | 危险样式、二次确认、操作受控、影响预览 |
| **渐进披露原则** | 按需提供信息，避免过早暴露 | 默认折叠、高级筛选隐藏、详情跳转 |
| **错误恢复原则** | 让用户知道错了，也知道怎么改 | 字段级校验 + 错误定位、草稿保存、回滚能力 |

---

## 13. 总结

### 模式的本质

界面模型模式是**把数据库语言翻译成业务语言的翻译器**。翻译的介质是：

- **结构**：把多表聚合成对象（Object Page、Related Lists）
- **时间**：把状态快照展开成过程（Timeline、Audit Log）
- **空间**：把全量字段分层到优先级（Sectioned Form、信息分层）
- **操作**：把数据录入翻译成业务承诺（表单族 + 风险控制）
- **指标**：把汇总数字展开成可解释链路（Drill-down、钻取）

### 设计好坏的判断标准

一个企业系统页面好不好，问三个问题：

```
1. 用户进来，能一眼知道自己在什么状态、要干什么吗？
2. 用户操作后，能立即知道结果，并且能追溯操作历史吗？
3. 用户遇到异常时，能自己处理闭环，还是只能找技术人员？
```

如果三个答案都是肯定的，这就是一个好页面。
