> 版本：1.0

> 状态：draft（待评审）
>
> 更新日期：2026-07-17
>
> 第一读者：执行研发任务的 AI、技术负责人、产品负责人、测试与验收人员

## 0. 一句话定义

本规范用于把一个大型、复杂或高风险研发目标，转化为一组目标一致、边界明确、尺寸受控、可独立验证、可安全集成的任务切片，并通过项目级总控循环、切片级交付循环和证据化验收，约束 AI 可靠完成任务。

核心公式：

```text
可靠交付
= 正确目标
+ 经验证的项目事实
+ 受控任务切片
+ 每片完整闭环
+ 独立质量裁决
+ 系统级验收
+ 可追溯证据
```

本规范不承诺 AI 永远不犯错。它保证的是：当目标未对齐、事实不足、切片过大、验证失败或证据缺失时，流程不能把任务错误地放行到下一阶段或宣布完成。

## 1. AI 执行入口

### 1.1 适用条件

满足以下任一条件时，应执行本规范：

- 任务属于《05-研发流程规范》L3 多模块功能或 L4 跨系统架构；
- 任务需要多个工作日、多轮 AI 会话或多个执行者协同；
- 任务涉及多个业务能力、多个状态转换或多个独立验收结果；
- 任务涉及新产品、新子系统、复杂交互、数据迁移或公共契约变更；
- 任务涉及金额、库存、订单、权限、多租户、支付、消息、隐私或生产数据；
- 任务即使规模不大，但目标理解存在实质分叉或失败代价高；
- 当前任务切片已经超过本规范的尺寸预算。

以下任务通常不必完整执行本规范：

- 目标、修改点和验证方式均明确的单文件低风险修复；
- 纯文案、格式或不影响运行行为的文档修改；
- 已有等价项目产物且输入基线未变化的重复执行。

低风险任务仍须遵守项目规则、最小验证和证据化完成要求。

### 1.2 与现有规范的关系

本规范不是第二套产品生命周期，也不替代已有文档：

| 文档 | 本规范与其关系 |
|---|---|

| `02-流程-知识库驱动新产品创建与迭代.md` | 继承 P0-P7 产品生命周期；本规范细化大型任务如何由 AI 交付 |
| `05-研发流程规范.md` | 继承需求、设计、测试、实现和评审活动；本规范增强概念对齐、任务切片与分层验收 |
| `06-规范-AI_UI质量保障流程规范.md` | 涉及 UI/交互时嵌入其视觉和交互质量门禁 |
| `08-规范-复杂产品能力完整性保障流程规范.md` | 复杂产品先用其建立能力全集和覆盖关系，再按本规范形成可执行切片 |

### 1.3 最小输入

AI 启动本流程至少需要：

```yaml
project_root: 项目或产品目录
request: 用户原始任务
request_owner: 能确认目标和验收的人
current_baseline: 当前分支、版本或产品基线
known_constraints: 已知时间、技术、合规和兼容约束
```

输入不完整时，AI 自动执行“现状扫描和缺口识别”，不能直接进入实现。

### 1.4 启动动作

AI 接到大型任务后，必须按顺序执行：

```text
1. 读取项目入口规则、相关规范和当前工作区状态。
2. 扫描已有需求、设计、ADR、测试、代码和执行记录。
3. 识别是否存在等价产物，禁止机械重做已通过阶段。
4. 建立执行记录，区分事实、推断、决策和未知项。
5. 从第一个未通过门禁的阶段开始。
```

### 1.5 动作等级

| 等级 | AI 行为 | 典型动作 |
|---|---|---|
| 自动执行 | AI 执行、验证并报告 | 项目扫描、调用链调查、候选切片、测试执行、RTM 更新 |
| 执行后送审 | AI 先产出，未经评审不得成为冻结基线 | PRD、原型、技术方案、测试规格、切片计划 |
| 必须人工决策 | AI 只准备决策包并等待 | 产品定义、范围取舍、重大 ADR、风险接受、上线放行 |

产品目标、用户价值和重大范围取舍不能由 AI 根据“行业一般做法”代替用户决定。

### 1.6 统一状态

项目阶段、批次和切片统一使用：

```text
not_started
in_progress
blocked
awaiting_human
passed
failed
superseded
```

合法迁移：

```text
not_started → in_progress
in_progress → passed | failed | blocked | awaiting_human
failed | blocked | awaiting_human → in_progress
passed → superseded → in_progress
```

禁止跳过 `in_progress` 直接声明 `passed`。输入基线变化导致旧结论失效时，必须标记 `superseded`，不能继续使用旧证据。

## 2. 核心模型：双轨、双循环、三层验收

### 2.1 双轨：发现与交付

大型任务同时存在两条轨道：

```text
发现轨 Discovery：目标 → 场景 → 原型/Spike → 证据 → 决策
交付轨 Delivery：切片 → 测试 → 实现 → 集成 → 发布
```

发现轨负责判断“做什么、为什么、是否可行”；交付轨负责判断“是否正确实现”。发现轨没有形成足够证据前，不能大量投放交付轨。

两条轨道可以并行探索，但未经批准的发现结论不能悄悄改变正在交付的基线。

### 2.2 双循环：项目总控与切片交付

```text
外层项目总控循环
目标对齐 → 事实调查 → 方案/验收设计 → WBS/切片 → 批次集成 → 系统验收

内层切片交付循环
Slice DoR → 失败测试 → 最小实现 → 局部验证 → 独立审查 → Slice DoD
```

外层防止“每片都完成但整体方向错误”；内层防止“大任务一次性执行导致漂移、失控和假完成”。

### 2.3 三层验收

| 层级 | 回答的问题 | 放行依据 |
|---|---|---|
| Slice Acceptance | 单个业务增量是否正确 | 切片验收标准、测试和证据 |
| Integration Gate | 本批切片组合后是否正确 | 契约、数据流、端到端和回归测试 |
| System Acceptance | 整个任务是否实现原始业务目标 | 系统 AC、System DoD、运行证据和独立验收 |

局部测试全部通过，不等于系统验收通过。

### 2.4 V 模型约束

每个左侧定义必须在右侧有对应验证：

```text
业务目标          ↔ 业务结果验收
产品需求          ↔ 用户验收测试
系统设计          ↔ 系统/集成测试
模块设计          ↔ 模块/契约测试
代码实现          ↔ 单元测试和静态检查
```

测试不能在实现结束后才补。定义需求和设计时，必须同步定义“如何证明它成立”。

## 3. 大型任务标准流程 A0-A12

### A0：恢复现场与建立基线

**输入**：项目目录、原始请求、已有产物。

**动作**：

1. 检查项目规则、Git 状态和用户现有修改；
2. 定位已有需求、设计、测试、代码和历史决策；
3. 建立输入文件及哈希，记录规则版本；
4. 判断应从哪个阶段继续。

**输出**：`Execution Record`、已有产物映射、缺口清单。

**门禁**：工作区状态未知、项目规则未加载或用户修改可能被覆盖时，不得继续写入。

### A1：任务章程与风险分级

先把原始请求转换为任务章程：

```yaml
task_id: TASK-001
problem: 要解决的问题
objective: 期望达成的结果
users: 目标用户/参与角色
business_outcomes: 可观察业务结果
scope: 本轮包含内容
non_goals: 本轮明确不做
constraints: 技术、时间、合规、兼容约束
invariants: 绝对不能破坏的事实
system_acceptance: 系统级验收条件
risk_level: L1|L2|L3|L4
rollback: 回滚原则
```

**输出**：任务章程、风险等级、初始系统验收条件、决策人与验收人清单。

**门禁 Project DoR-0**：如果问题、目标用户、业务结果或验收责任人均不明确，进入 A2，不得进入技术方案。

### A2：意图、概念与产品边界对齐

这是防止 AI 从一开始做错产品的关键阶段。

#### A2.1 必须讨论的触发条件

存在以下任一情况时，AI 必须显式暴露理解并与用户讨论：

- 同一个词存在多种产品模型或领域含义；
- 不同理解会改变核心对象、流程、权限、数据归属或交互；
- 用户只给出“灵活、智能、完整、像某产品”等抽象表述；
- AI 需要替用户选择目标用户、业务优先级或重大范围；
- 典型场景不足以判断产品边界；
- AI 与用户对目标的理解尚未形成可验证的共同表述。

例如“表单设计器”可能指企业业务表单、问卷路径表单、页面搭建器，也可能是三者的受控组合。此时不能由 AI 默认选择一种后继续编码。

#### A2.2 讨论、原型和 Spike 的分工

| 不确定性 | 首选方法 | 主要产物 |
|---|---|---|
| 业务目标、角色、数据归属、流程不清 | 场景访谈/问题讨论 | 场景清单、产品定义、开放问题 |
| 页面结构、信息架构、操作方式不清 | 多方案低保真原型 | 原型、走查记录、选择结论 |
| 技术可行性、性能或兼容性不清 | 限时 Spike/POC | 实验、数据、结论和后续建议 |
| 旧系统实际行为不清 | Characterization | 可重复测试和隐性契约清单 |

原型不能替代业务讨论，Spike 不能替代产品决策。

#### A2.3 场景对齐方法

至少收集：

```text
谁在什么场景下
为了完成什么任务
使用什么输入
执行哪些关键动作
获得什么可观察结果
遇到哪些异常和权限边界
目前如何完成、痛点是什么
```

优先使用 3 个真实典型案例和 1 个反例验证产品定义，避免只讨论抽象功能名。

**输出**：一句话产品定义、典型场景、非目标、概念差异表、必要原型。

**人工决策**：产品定义、方案取舍和第一版边界。

**门禁 Alignment Gate**：双方未确认核心模型时，状态为 `awaiting_human`，禁止正式切片和编码。

### A3：现状调查与证据建模

AI 必须调查而不是根据经验猜测项目事实：

```text
代码入口和调用链
核心对象、状态和数据流
现有接口、Schema、配置和依赖
历史逻辑与隐性契约
权限、租户、并发和事务边界
已有测试、运行环境和发布机制
最近相关变更和已知缺陷
```

结论分为：

```yaml
facts: 有源码、数据、运行结果或正式文档支持
inferences: 从事实推导，附推理链
hypotheses: 尚未验证，只能指导下一步实验
unknowns: 当前不知道且可能影响决策
decisions: 已由有权人员或批准流程作出的决定
```

**输出**：事实与未知项清单、现状模型、影响范围、待验证假设。

**门禁 Evidence Gate**：决定架构、兼容和切片边界的关键未知项必须先验证或升级，不能隐藏在假设中。

### A4：风险优先验证与边界冻结

优先处理最不确定、最难回滚、失败代价最高的事项，而不是先写最容易的页面。

#### A4.1 适合 Spike 的问题

- 新技术或第三方接口是否满足能力；
- Schema/协议能否往返、升级和兼容；
- 性能目标在目标规模下是否可达；
- 数据迁移是否可逆、历史数据是否可映射；
- 复杂规则是否存在循环、冲突或不可判定行为。

Spike 必须有问题、时间盒、实验方法、判定阈值和结论。只产生代码而没有结论的 Spike 不算完成。

#### A4.2 单向门与 ADR

以下通常属于单向门：核心数据模型、公共 API、状态机、权限模型、发布协议、迁移策略。必须记录 ADR：

```text
背景与约束
候选方案
选择及理由
负面后果
兼容和迁移措施
撤销成本
重新评估条件
```

**输出**：Spike 报告、Characterization Tests、ADR、冻结边界。

**门禁**：重大 ADR 未批准时，AI 可以继续不依赖该决策的调查，不能开始依赖它的实现。

### A5：方案设计与系统验收设计

设计必须同时包含正向实现和反向验证：

```text
业务流程和状态机
模块边界与依赖方向
数据流、控制流和失败恢复
接口与数据契约
安全、权限和多租户边界
兼容、迁移和回滚
可观测性和运行指标
系统验收场景及测试策略
```

验收标准应使用可执行、可量化表述。推荐 Given/When/Then：

```gherkin
Given 已发布版本 V1 正在被用户填写
When 设计者修改草稿并保存
Then V1 的渲染和提交行为保持不变
And 新草稿只在再次发布后生效
```

**输出**：技术设计、系统测试规格、风险与回滚计划、RTM 初版。

**门禁 Design & Test Gate**：每项系统需求必须至少有一种验证方式；设计和测试规格必须由不同视角审查。

### A6：WBS、用户故事地图与候选切片

#### A6.1 先 WBS，后切片

WBS 回答“范围是否完整”，切片回答“如何逐步交付”。标准层级：

```text
GOAL/Epic：大型业务目标
CAPABILITY：业务能力
FEATURE：可感知功能
SLICE：可独立验收的业务增量
TASK：实现切片的技术步骤
```

AI 不得把 Epic、Capability 或纯技术 Task 当成可直接执行的 Slice。

#### A6.2 用户故事地图

沿用户主链路排列活动和任务，再从上到下划分发布增量：

```text
用户活动：创建 → 配置 → 预览 → 发布 → 使用 → 管理
第一条可用主路径：每个活动只保留最小必要行为
后续增量：规则扩展、异常、更多角色、规模和集成
```

故事地图用于避免“技术模块全部写完，但用户没有完整流程”。

#### A6.3 切片类型

| 类型 | 目的 | 完成产物 |
|---|---|---|
| Discovery Slice | 澄清目标、场景或产品边界 | 决策材料、场景和开放问题 |
| Spike Slice | 验证关键技术假设 | 可复现实验、数据和结论 |
| Characterization Slice | 锁定旧系统真实行为 | 回归测试、隐性契约 |
| Feature Slice | 交付端到端业务结果 | 可运行功能、测试和证据 |
| Enabler Slice | 解锁明确的后续业务片 | 最小基础能力及消费方 |
| Migration Slice | 安全迁移数据或契约 | 迁移、校验和回退证据 |
| Hardening Slice | 补足规模、韧性或安全目标 | 性能、安全、容灾证据 |

Enabler 必须写明它解锁哪个 Feature Slice，禁止无限建设“未来可能有用”的平台能力。

**输出**：WBS、用户故事地图、候选切片清单、范围覆盖检查。

**门禁 Coverage Gate**：每个范围内能力必须映射到候选切片或明确的暂缓决策；每个候选 Feature Slice 必须落在用户旅程或明确消费关系中。

### A7：切片尺寸门禁与批次计划

本阶段执行第 6 章的完整切片协议，形成：

- 切片合同；
- Slice DoR / Slice DoD；
- 依赖 DAG；
- 执行批次；
- 文件或契约所有权；
- 每片验收者；
- 项目 RTM 更新。

只详细规划下一批 1-3 个即将执行的切片，其余保留能力级或候选切片级描述，采用滚动式规划。

**输出**：Slice Contracts、依赖 DAG、批次计划、Context Packs、更新后的 RTM。

**门禁 Slice Plan Gate**：切片过大、无法独立验收、依赖不清或关键契约未冻结时，不得分配给 AI 实现。

### A8：逐片实现循环

每个 Feature/Migration/Hardening Slice 独立执行：

```text
1. Slice DoR 检查
2. 读取版本化 Context Pack
3. 写最小失败测试并确认按预期失败
4. 实现满足验收标准的最小代码
5. 运行局部测试、静态检查和关联回归
6. 检查范围漂移、接口变化和新增依赖
7. 独立 Reviewer 审查
8. 独立 Verifier 复跑验收
9. Slice DoD 裁决
10. 保存 Evidence Pack、更新 RTM 和进度账本
```

AI 实现者不得仅凭自己的说明批准自己的切片。

**输出**：代码与配置变更、自动化测试、评审记录、Slice Evidence Pack、更新后的 Progress Ledger 和 RTM。

**门禁 Slice DoD Gate**：第 9.1 节任一必需项未通过时，该切片不得标记 `passed`，也不得成为依赖切片的有效输入。

### A9：批次集成门禁

每完成一个批次，必须验证切片组合后的行为：

```text
接口/Schema 契约
跨模块数据流与状态流
数据库事务和迁移
权限、租户和身份传递
消息幂等和失败恢复
端到端主路径
关联旧功能回归
性能与资源基线
```

集成失败时，先确定首次错误状态和责任边界，再决定修复哪个切片；禁止无诊断地同时修改多个切片。

**输出**：批次集成报告、端到端证据、回归结果、缺陷归属和批次基线。

**门禁 Integration Gate**：第 9.2 节全部通过后，本批次才能成为下一批的稳定输入；失败证据必须进入修复切片或回退决策。

### A10：系统验收

重新回到 A1/A2 的原始目标验收，而不是只统计切片完成率：

1. 按系统 AC 执行业务验收；
2. 核对 RTM 中需求、设计、代码、测试、证据是否闭合；
3. 执行完整回归、安全、性能和兼容验证；
4. 检查未完成项、豁免和残余风险；
5. 独立验收者给出 `passed/failed/evidence_insufficient`。

**输出**：系统验收报告、闭合 RTM、最终 Evidence Pack、未覆盖项和残余风险。

**门禁 System DoD Gate**：第 9.3 节全部通过且独立验收者批准后，任务才可声明研发完成；`evidence_insufficient` 不得降级为通过。

### A11：发布、观测与回滚

根据风险采用：

- Feature Flag；
- 内部试用或租户白名单；
- Canary/灰度；
- Shadow Mode 或新旧结果对账；
- 业务不变量监控；
- 自动或人工回滚阈值。

发布不是“部署命令成功”。必须证明真实业务指标、错误率和不变量处于允许范围。

**输出**：发布计划、Runbook、监控与告警配置、灰度/对账报告、回滚演练证据。

**人工决策**：生产上线和风险接受。

**门禁 Release Gate**：未取得上线授权、关键业务指标不可观测或回滚不可执行时，禁止全量发布。

### A12：复盘、知识回流与评测集

每个重大任务完成后记录：

```text
哪些初始假设错误
哪些切片过大或依赖判断错误
哪些测试没有发现缺陷
哪些门禁真正阻止了问题
哪些规则造成无效成本
哪些缺陷需要转成回归用例
哪些知识、ADR、Runbook 和模板需要更新
```

AI 特有缺陷应进入企业 Eval Set，例如：错误需求理解、虚构 API、无关修改、测试自证、虚假完成和上下文遗忘。模型、Agent、规则或工具升级后必须重新运行关键评测集。

**输出**：复盘报告、改进项、知识库/ADR/Runbook 更新、缺陷回归用例和 Eval Set 增量。

**门禁 Learning Gate**：重大缺陷没有根因、责任人、验证方式和防复发载体时，复盘不能标记 `passed`。

## 4. DoR：什么时候允许开始

### 4.1 Project DoR

大型任务进入正式设计与交付前，至少满足：

```text
[ ] 问题、目标用户和业务结果明确
[ ] 产品概念和核心模型已经对齐
[ ] 范围与非目标已经确认
[ ] 系统级验收标准可执行
[ ] 关键旧逻辑和影响范围已有初步证据
[ ] 高风险未知项已有验证或处理计划
[ ] 关键不变量、合规和安全边界明确
[ ] 决策人、实现责任和验收责任明确
[ ] 项目基线和回滚原则明确
```

Project DoR 不要求所有细节提前冻结；它要求足以安全形成下一批切片。

### 4.2 Slice DoR

切片交给 AI 实现前，必须满足：

```text
[ ] 一个可观察目标，且不包含第二个独立业务结果
[ ] 父需求、业务规则和验收场景明确
[ ] 前置依赖已通过，所需契约已冻结
[ ] 输入、输出、错误和状态边界明确
[ ] 必须保持的不变量和旧行为明确
[ ] 允许修改范围、禁止事项和变更预算明确
[ ] 测试环境、数据、工具和权限可用
[ ] 关键未知项不超过一个，且不阻断实现
[ ] 验证命令或验收步骤可执行
[ ] Reviewer、Verifier 和回滚方式明确
```

DoR 未通过时可以继续调查、补测试或做 Spike，不能假装进入编码。

## 5. 任务切片方法

### 5.1 合格切片定义

```text
一个 Slice
= 一个可观察业务结果
+ 一组不可分离的实现修改
+ 一组验收场景
+ 一个证据包
+ 一个可控回滚边界
```

合格切片尽量符合 INVEST，并增加 AI 交付约束：

| 属性 | 判定 |
|---|---|
| Independent | 在已声明依赖之外能独立推进，不与其他片争夺未冻结契约 |
| Negotiable | 解决方案可讨论，但目标和不变量不可被实现者擅改 |
| Valuable | 完成后用户、业务或后续明确消费者能获得价值 |
| Estimable | 影响范围和主要步骤可合理估计 |
| Small | 一次 AI 执行周期和一个可审查 PR 内完成 |
| Testable | 有确定输入、行为和可执行验收 |
| Context-complete | AI 所需事实和规则能装入版本化上下文包 |
| Evidence-producing | 完成后产生可复查的运行证据 |
| Reversible | 能独立回滚、关闭或有明确补偿方案 |

### 5.2 优先纵向切片

不推荐：

```text
先建完所有表 → 再写完所有后端 → 再写完所有前端 → 最后统一测试
```

推荐：

```text
一个用户场景 → 必要数据 → 必要服务 → 必要界面/API → 测试 → 可运行结果
```

纵向切片允许同时触碰前后端和数据库，但只能服务于一个可验收结果。

### 5.3 常用切片维度

按以下顺序寻找最自然的边界：

1. 用户旅程中的一个任务；
2. 一个主要业务状态转换；
3. 一个业务动作或命令；
4. 一个角色或权限场景；
5. 一条最简单主路径，再扩展异常路径；
6. 一条基础规则，再扩展复杂规则；
7. 一种数据类型、渠道或集成对象；
8. 单笔到批量、低规模到目标规模；
9. 内部试用到灰度再到全量。

### 5.4 递归拆分算法

AI 发现候选切片过大时，必须依次询问：

```text
包含多个可观察结果吗？          → 按结果拆
包含多个主要状态转换吗？        → 按状态拆
同时服务多个角色吗？            → 按角色拆
主路径和异常扩展混在一起吗？    → 先最小安全主路径，再扩展
简单规则和复杂规则混在一起吗？  → 按规则层级拆
存在关键技术未知项吗？          → 单独做 Spike
功能与迁移/性能/容灾混在一起吗？→ 拆 Feature/Migration/Hardening
仍然无法独立验收或回滚吗？      → 重新定义业务边界
```

不得为了缩小尺寸，把权限、租户隔离、服务端校验、基本幂等和必要事务一致性拆到未来再补；这些是当前业务行为的安全完整性。

### 5.5 反模式

- 按组织或技术层横切，直到最后才产生用户价值；
- 一个切片标题中有多个独立的“以及/同时/并且”；
- 把“完成整个模块”“支持全部场景”当作切片；
- 只有编码步骤，没有验收标准；
- 为了并行，提前虚构尚未冻结的接口；
- 把共用抽象先做成大平台，却没有首个明确消费者；
- 切得过碎，每片只改一个函数但不能独立验收；
- 把安全、测试、可观测性统一留到最后；
- 执行中把新需求悄悄塞入当前片。

## 6. Slice Size Gate：切片尺寸门禁

### 6.1 硬条件

出现以下任一情况，当前对象不是可执行切片，必须重切或重新规划：

- 包含两个及以上可独立验收的业务结果；
- 包含两个及以上可独立发布的主要状态转换；
- 无法写出独立验收、测试或回滚方式；
- 关键未知项会改变总体架构或核心契约；
- 必须等待多个未完成且未冻结的外部依赖；
- 目标只能用“完成某系统/某模块/所有功能”描述；
- AI 无法在当前 Context Pack 中解释清楚目标、约束和验证方法。

### 6.2 尺寸预警线

以下数值是异常检测信号，不是绩效指标：

| 维度 | 默认预警线 |
|---|---:|
| 可观察业务结果 | 1 个 |
| 主要状态转换 | 1 个 |
| 验收场景组 | 不超过 3 组 |
| 关键未知项 | 不超过 1 个 |
| 主要业务/技术模块 | 通常不超过 2 个 |
| 实现步骤 | 通常不超过 5 个 |
| 手工修改文件 | 通常不超过 5-8 个 |
| 人工有效代码差异 | 通常不超过 300-500 行 |
| 连续执行周期 | 通常不超过 2-4 小时或一次 Agent 会话 |

命中一个严重预警或两个普通预警，AI 必须重新评估尺寸并记录“保持/重切”的理由。生成代码、锁文件、快照和机械迁移应单独统计，不能用总代码行数机械否决。

### 6.3 禁止静默扩片

执行中发现新工作时：

```text
1. 判断是否为当前 AC、不变量或安全完整性的必要条件。
2. 若是，更新切片合同并重新过 Size Gate。
3. 更新后超限，停止实现并重切。
4. 若不是，新建 Backlog Slice，不得并入当前片。
5. 触及公共接口、数据库、权限或重大 ADR，重新走影响评审。
```

AI 无权自行批准超限例外。

## 7. 切片合同、依赖与上下文

### 7.1 Slice Contract 模板

```yaml

title: 一句话可观察结果
type: feature|spike|characterization|enabler|migration|hardening
parent_requirement: REQ-002
objective: 本片完成后发生什么变化
business_value: 谁获得什么价值

depends_on: []
scope: []
non_goals: []

inputs: []
outputs: []
invariants: []
compatibility: []

acceptance_criteria:
  - id: AC-003-01
    given: 前置状态
    when: 行为
    then: 可观察结果

change_budget:
  allowed_modules: []
  forbidden_areas: []
  dependency_change: forbidden|review_required|allowed
  public_contract_change: forbidden|review_required|allowed

expected_impact:
  modules: 1
  files: 5
  steps: 4
  unknowns: 0

verification: []
rollback: 回滚/功能开关/补偿方案
implementer: agent-or-person
reviewer: independent-agent-or-person
verifier: independent-agent-or-person
status: not_started
```

### 7.2 依赖 DAG

切片必须形成有向无环依赖图。依赖边应表示真实的输入或契约依赖，而不是团队习惯顺序。

允许并行的条件：

```text
[ ] 前置依赖均 passed
[ ] 共享接口和 Schema 已冻结
[ ] 不修改相同核心文件或迁移序号
[ ] 不争夺同一外部资源或环境
[ ] 各片有独立 Context Pack 和验收者
[ ] 集成点和合并顺序明确
```

如果多个切片高频修改同一核心文件，优先调整边界或串行执行，不要用更多 Agent 放大冲突。

### 7.3 Context Pack

每个 AI 切片只加载完成当前工作所需的版本化上下文：

```yaml
slice_contract: 当前切片合同
source_baseline: commit/branch/hash
authoritative_requirements: []
approved_decisions: []
relevant_code_entrypoints: []
existing_patterns: []
must_preserve_behaviors: []
allowed_files_or_modules: []
forbidden_changes: []
test_commands: []
facts: []
open_questions: []
```

聊天记录不是权威上下文。阶段结论必须落入文件或可追踪系统，长任务每次恢复先读取 Progress Ledger，而不是依赖模型记忆。

### 7.4 Progress Ledger

```yaml
execution_id: EXE-20260717-001
current_stage: A8
current_batch: BATCH-02
current_slice: SLICE-003
status: in_progress
completed: []

new_facts: []
changed_assumptions: []
scope_changes: []
open_questions: []
next_action: 下一条动作
```

## 8. 切片实现与质量控制

### 8.1 测试层次

| 方法 | 解决的问题 | 使用时点 |
|---|---|---|
| ATDD | 业务方如何判断需求已实现 | 需求和切片验收设计时 |
| BDD | 场景、状态和规则如何表述 | 场景复杂或多角色协作时 |
| TDD | 实现行为如何被快速约束 | 编写生产代码之前 |
| Characterization Test | 旧系统现在实际如何工作 | 修改既有行为之前 |
| Contract Test | 服务、消息或 Schema 是否兼容 | 跨模块/跨服务集成前 |
| Property/Invariant Test | 大量输入下不变量是否保持 | 金额、状态、转换和解析逻辑 |
| Mutation Test | 测试是否真能识别错误 | 关键规则和高风险模块 |
| E2E Test | 用户主流程是否完整 | 批次和系统验收时 |

测试数量不是目标；能否证伪关键主张才是目标。

### 8.2 Maker-Checker-Verifier 分离

| 角色 | 责任 | 禁止 |
|---|---|---|
| Maker/Implementer | 按切片合同实现最小变更 | 擅改目标、批准自己的豁免 |
| Checker/Reviewer | 审查需求覆盖、代码、兼容、安全和范围 | 只阅读实现者摘要而不看 diff/证据 |
| Verifier | 独立运行验收并裁决证据 | 把“测试应该通过”当结果 |
| Product/Business Owner | 验收业务目标和范围取舍 | 把技术测试通过等同业务成功 |

低风险任务可由一个人承担多个角色，但验证仍应独立运行。高风险任务必须职责分离并保留人工审批。

### 8.3 代码变更预算

每片必须明确：

- 允许修改的模块和文件类别；
- 能否新增依赖；
- 能否改变公共接口、数据库和配置；
- 必须保留的旧逻辑；
- 超出预算后的重新评审条件。

预算用于发现异常，不鼓励为了满足数字而拆出不完整实现。

### 8.4 事实与幻觉控制

AI 提到接口、字段、配置、依赖或运行行为时，必须能指出来源。无法确认时：

```text
标记为 hypothesis/unknown
→ 执行最小验证
→ 有证据后升级为 fact
```

禁止根据包名、文件名或常见框架习惯推断项目行为并直接修改。

### 8.5 根因与失败处理

遇到错误时：

```text
完整读取错误
→ 固化复现
→ 定位首次错误状态
→ 提出可证伪假设
→ 最小实验
→ 修复根因
→ 原问题验证
→ 关联回归
```

禁止把 `try/catch`、重试、默认值、空判断或延长超时当作通用修复。

### 8.6 供应链与工具安全

AI 新增依赖、插件、Skill、MCP 或构建工具时，至少检查：

- 来源真实性、版本和维护状态；
- 许可证和已知漏洞；
- 权限、网络和数据访问范围；
- 是否锁定版本、能否生成 SBOM；
- 构建与发布产物能否追溯和验证签名；
- 是否将敏感代码或数据发送到未批准服务。

未经批准不得新增依赖或扩大 Agent 权限。

## 9. 验收、门禁与完成定义

### 9.1 Slice DoD

```text
[ ] 所有 Slice AC 有新鲜通过证据
[ ] 失败测试已先失败并在实现后通过，或已说明不适用原因
[ ] 代码只实现当前切片目标，没有未解释的范围外修改
[ ] 必要安全、权限、租户、幂等和事务约束已覆盖
[ ] 旧行为和公共契约已通过关联回归
[ ] 构建、lint、类型检查和相关静态分析通过
[ ] Reviewer 的阻断问题已关闭
[ ] Verifier 独立复跑验收
[ ] RTM、Progress Ledger 和 Evidence Pack 已更新
[ ] 回滚方式可执行，残余风险已记录
```

### 9.2 Integration Gate

```text
[ ] 批次内所有切片均 passed
[ ] 接口、Schema 和事件契约兼容
[ ] 跨切片状态流、数据流和失败恢复正确
[ ] 端到端主路径和关键异常路径通过
[ ] 关联回归、安全和性能基线通过
[ ] 没有未解决的合并冲突或临时兼容代码
[ ] 集成证据来自本批次最终基线
```

### 9.3 System DoD

```text
[ ] 原始问题和业务目标已经重新验收
[ ] 全部范围内需求具有设计、实现、测试和证据映射
[ ] 所有系统 AC 通过，非目标未被偷偷纳入
[ ] 构建、测试、静态分析、安全和供应链门禁通过
[ ] 旧逻辑、历史数据和公共契约满足兼容要求
[ ] 性能、可用性、可访问性和运维目标达到阈值
[ ] 配置、迁移、监控、告警和 Runbook 已完成
[ ] 灰度、回滚或补偿方案已演练或验证
[ ] 豁免均有 owner、补偿措施和到期时间
[ ] 未覆盖风险和未决项显式列出
[ ] 独立验收者已经给出最终结论
```

### 9.4 RTM 双向追溯

最低追溯链：

```text
GOAL → CAPABILITY → REQUIREMENT → SLICE → CODE CHANGE → TEST → EVIDENCE
```

裁决规则：

- 有需求无测试：不能验收；
- 有代码无需求/切片：疑似范围漂移；
- 有测试无实际结果：没有完成证据；
- 有证据但输入基线不一致：证据无效；
- 需求已取消但代码仍存在：必须解释或删除。

### 9.5 Evidence Pack

每片和系统最终交付至少包含：

```yaml
claim: 要证明的主张
baseline: commit/版本/环境
requirements: []
changes: []
verification_commands: []
results: []
artifacts: []
review_findings: []
coverage_and_gaps: []
residual_risks: []
rollback: 回滚证据
verified_by: 验证者
verified_at: 时间
```

AI 的自然语言总结不是证据。命令、结果、截图、日志、测试报告或可复现实验才是证据。

### 9.6 最终裁决状态

| 状态 | 含义 |
|---|---|
| `passed` | 所有必需门禁通过且证据充分 |
| `failed` | 已有证据证明不满足要求 |
| `blocked` | 缺环境、权限或上游条件，当前无法继续 |
| `awaiting_human` | 等待必须人工作出的决策 |
| `evidence_insufficient` | 看似完成但没有足够证据，不得视为通过 |

`evidence_insufficient` 是验收结论，不替代统一执行状态；执行记录状态应设为 `failed` 或 `blocked`，并写明补证动作。

## 10. 需求和范围变化协议

### 10.1 变更分类

| 变化 | 处理 |
|---|---|
| 澄清原有 AC，不改变行为 | 更新切片合同和测试，重新验证 |
| 当前切片完整性必需项 | 更新预算并重过 Size Gate |
| 新的独立业务结果 | 新建 Backlog Slice |
| 修改公共契约、数据模型或重大 ADR | 影响分析并回到 A4/A5 |
| 改变产品定义、目标用户或范围 | 回到 A2，旧计划标记 superseded |
| 生产风险或事实推翻原假设 | 停止相关切片，重新调查和规划 |

### 10.2 变更记录

```yaml
change_id: CHG-001
source: 变更来源
old_baseline: 旧需求/设计版本
new_request: 新内容
impact: [requirements, slices, contracts, tests, rollout]
decision: accept|reject|defer
approved_by: 决策人
superseded_artifacts: []
reverification: []
```

禁止只修改代码而不更新上游规格和下游测试。

## 11. 示例：AI 开发表单设计器

### 11.1 不能直接进入切片

“表单设计器”至少可能指：

- 企业业务表单：对象字段、密集录入、权限和状态；
- 问卷路径表单：问题、分页、跳题、匿名提交；
- 页面搭建器：组件、布局、数据源、事件和脚本。

如果用户和 AI 对该词理解不同，后续 Schema、权限、运行时和交互都会错误。因此先执行 A2：

```text
收集三个真实表单案例和一个不属于本产品的反例
→ 比较三种产品模型
→ 对关键交互制作低保真原型
→ 走查“设计-预览-发布-填写-查看数据”
→ 用户确认主模型、吸收能力和第一版非目标
```

### 11.2 假设确认后的 MVP

假设确认结果为：企业业务表单优先，吸收基础分页/条件能力，不做自由页面搭建。

```text
MVP：文本/数字/日期/单选、分组/双列、校验、条件显隐、草稿、预览、发布、提交
非目标：自由像素布局、任意脚本、完整审批、实时协作、AI 自动生成
```

先验证：

```text
SP-01：设计态 Tree 与运行态 Schema 是否可以无损往返
SP-02：100-200 字段下设计器和运行时性能是否可接受
CH-01：若有旧表单，锁定旧 Schema 和提交行为
```

### 11.3 候选纵向切片

| 切片 | 可观察结果 |
|---|---|
| S0 | 用核心不变量测试锁定 Schema、版本和提交边界 |
| S1 | 给定 Schema，运行时渲染文本字段并提交值 |
| S2 | 设计者添加文本字段、修改标题、保存并重新打开 |
| S3 | 设计者拖动字段排序，预览与运行时顺序一致 |
| S4 | 配置必填和长度校验，前后端得到一致结果 |
| S5 | 通过物料协议增加数字、日期和单选字段 |
| S6 | 添加分组和双列布局并正确预览 |
| S7 | 配置一个条件显隐规则并检测失效引用 |
| S8 | 发布不可变版本，修改草稿不影响已发布表单 |
| S9 | 填写者提交发布版本，服务端拒绝非法字段和值 |
| S10 | 完成权限、审计、版本迁移、性能和发布验收 |

每片都包含当前行为必需的权限和服务端安全校验，不把安全统一留到 S10。S10 负责系统性强化和最终证明。

## 13. 企业治理与度量

### 13.1 应自动化的门禁

- 分支保护和 CODEOWNERS；
- 编译、lint、类型检查和测试；
- SAST、依赖、许可证、密钥和镜像扫描；
- 公共 API/Schema 兼容检查；
- 架构依赖和变更范围检查；
- RTM 完整性和 Evidence Pack 完整性；
- 制品来源、SBOM、签名和发布审批。

高风险约束不能只写在 Prompt 中，应尽量下沉到 CI、权限、数据库约束和部署策略。

### 13.2 推荐指标

同时观察效率、质量和风险：

```text
需求一次理解正确率
切片首次 DoR 通过率
切片超限重切率
一次验收通过率
无关修改率
集成失败率
逃逸缺陷率和回滚率
AI 声称完成但验证失败的比例
人工返工时间
相同根因重复出现率
从 Ready 到验收通过的周期
```

不得用生成代码行数、AI 调用次数或切片数量作为核心绩效目标；这些指标会诱发过度修改和过度切片。

### 13.3 成熟度路线

| 等级 | 能力 |
|---|---|
| M1 辅助 | AI 生成代码，人手工检查；缺统一任务契约 |
| M2 受控 | 有 DoR/DoD、PR、测试和基本质量门禁 |
| M3 可追溯 | 有切片合同、RTM、证据包和独立验证 |
| M4 可度量 | 有企业 Eval Set、质量指标、灰度和自动回滚 |
| M5 自适应 | 缺陷自动回流规则与评测，流程按风险动态裁剪 |

组织应先稳定 M2/M3，再扩大 Agent 自主权限。

## 14. 方法来源与使用边界

本规范综合使用以下成熟方法，但不机械照搬：

| 方法 | 吸收内容 | 使用边界 |
|---|---|---|
| Product Discovery / Dual Track | 先验证问题和方案，再扩大交付 | 不把无限探索当作不交付的理由 |
| Lean UX / 原型验证 | 用最低成本验证交互和价值假设 | 原型不能证明后端、安全和运行质量 |
| User Story Mapping | 沿用户旅程形成最小端到端增量 | 复杂基础设施仍需 Enabler/Spike |
| INVEST | 判断切片是否有价值、够小、可测试 | 安全不变量不能为了独立性被拆掉 |
| 敏捷团队常用 DoR/DoD 实践 | 建立入口和出口门禁 | DoR 不是用来追求所有需求零变化 |
| ATDD/BDD/TDD | 让验收、场景和实现可执行 | 测试必须验证业务主张，不能自证实现 |
| V 模型 | 定义与验证成对设计 | 不要求阶段只能瀑布式单向执行 |
| ADR/RFC | 固化重大决策和后果 | 小型双向门决策无需重文档 |
| RTM | 保证需求到证据双向追溯 | 追溯服务于决策，不能变成表格劳动 |
| Risk-based Testing | 高风险处增加验证强度 | 低风险不等于无需验证 |
| Feature Flag/Canary/Shadow | 降低发布和回滚风险 | 开关不能长期替代清理和迁移 |
| NIST SSDF / SLSA | 安全研发和供应链完整性 | 按组织风险与合规要求落地 |
| NIST AI RMF / OWASP GenAI | AI 风险、提示注入和过度代理治理 | 不用通用清单替代具体威胁建模 |

## 15. AI 自检清单

AI 每次准备宣布大型任务完成前，必须逐项回答：

```text
[ ] 我是否用用户确认的业务目标，而不是自己的默认理解？
[ ] 概念分叉是否已经讨论，必要交互是否经过原型走查？
[ ] 关键结论是否有项目事实或实验支持？
[ ] 我是否先定义系统验收，再进行任务切片？
[ ] 每个切片是否只有一个可观察结果并通过 Size Gate？
[ ] 新发现范围是否创建新片或重新评审，而非静默扩张？
[ ] 每片是否通过独立 Slice DoR、测试和 Slice DoD？
[ ] 批次是否经过集成门禁？
[ ] 最终是否重新按原始目标进行系统验收？
[ ] 完成主张是否有新鲜、可复现、基线一致的证据？
[ ] 旧逻辑、安全不变量、回滚和残余风险是否明确？
[ ] 是否还有任何未处理分支、失败门禁或等待人工决策？
```

任一必需项为“否”，不得声明 `passed`。

## 16. 一句话压缩

```text
先与人对齐“究竟要做什么”，
再用证据确认“系统实际上是什么”，
先设计“怎样证明整体成功”，
再把范围切成一个个可验收的小增量，
每片独立开发和验证，每批集成，最后回到原始目标验收；
没有证据，不算完成。
```
