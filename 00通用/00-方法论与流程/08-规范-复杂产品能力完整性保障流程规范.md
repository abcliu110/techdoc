# 复杂产品能力完整性保障流程规范

> 版本：v1.2  
> 类型：通用产品完整性规范 / 能力到功能到原型的双向追溯与实现真实性门禁  
> 适用对象：复杂 Web、桌面端、移动端、平台型产品、设计器、业务系统、工作台及其他多角色、多状态、多模块产品  
> 上游输入：产品目标、需求规格、领域能力基线、竞品与用户任务证据、项目范围合同  
> 下游衔接：`05-研发流程规范.md`、`06-规范-AI_UI质量保障流程规范.md` 及项目测试、评审和发布流程

---

## 0. 一句话定义

本规范用于证明一个复杂产品承诺的能力是否被完整、真实地实现，并建立以下可重放、可计算、可审计的双向追溯链：

```text
产品目标
→ 产品能力
→ 功能
→ 用户任务/验收场景
→ 命令与状态变化
→ 页面/模式/弹层/API/CLI/事件入口等执行载体
→ 可执行原型单元
→ 测试用例
→ 运行证据
```

本规范解决两个核心问题：

1. **还有哪些能力没有实现、只实现了外壳或被假成功掩盖？**
2. **为了证明本版本范围完整，究竟需要多少个原型应用和可独立验证的原型单元？**

核心原则：

```text
没有独立于实现的期望能力全集，就无法发现“根本没有实现”的功能。
页面存在不等于能力存在，按钮可点击不等于功能实现，状态写入不等于结果被消费。
原型数量由核心任务与必测状态的覆盖需求反推，不按页面、截图或按钮数量拍脑袋确定。
```

### 0.1 AI 最小输入与启动点

```text
最小输入：项目/产品目录（必须）+ 本轮目标（能力完整性检查、未实现功能检查、原型覆盖设计或交付验收之一）。
AI 自动发现：产品目标、需求、领域知识库、范围合同、现有页面/API、实现注册表、测试、运行方式和历史证据。
启动判定：扫描第 16 章标准产物和 C0-C10 状态，从第一个未通过或因输入变化失效的阶段继续。
缺少领域能力基线：先形成候选基线并标记待评审，不能只从当前实现反推期望能力。
缺少项目范围合同：完成候选能力和范围建议后进入 awaiting_human，不能自行决定产品取舍。
```

AI 启动后首先创建或更新 `completeness/execution.yaml`，不得直接从代码检查开始并跳过能力全集与范围冻结。

### 0.2 动作等级

| 动作 | 等级 | 说明 |
|---|---|---|
| 发现资料、提取候选能力、建立矩阵、扫描界面/API、运行验证、计算差集 | 自动执行 | AI 可直接执行并保留证据 |
| 领域能力基线草案、原型覆盖方案、完整性评审报告 | 执行后送审 | AI 生成后标记待评审，不能冒充已批准范围 |
| 产品目标、REQUIRED/PLANNED/EXCLUDED 取舍、核心任务冻结、风险接受、最终放行 | 必须人工决策 | AI 只准备事实、候选、影响和建议，状态进入 `awaiting_human` |

不得因为用户要求“全部自动完成”而绕过产品范围和风险接受的人工决策；已有明确、可追溯的上位决策可直接作为输入，不重复请求。

### 0.3 执行状态与恢复

C0-C10 每个阶段只能使用：

```text
not_started | in_progress | blocked | awaiting_human | passed | failed | superseded
```

合法迁移：

```text
not_started -> in_progress
in_progress -> passed | failed | blocked | awaiting_human
failed | blocked | awaiting_human -> in_progress     # 退出条件已满足
passed -> superseded -> in_progress                  # 输入、范围或规范变化使证据失效
```

禁止跳过 `in_progress` 直接写 `passed`。恢复执行时扫描 `execution.yaml`、输入哈希和阶段产物，从第一个非 `passed` 且依赖满足的阶段继续；不得重做仍有效的阶段。

```yaml
execution_id: COMPLETE-YYYYMMDD-NNN
spec_id: 08-complex-product-completeness
spec_version: "1.2"
input_baseline: []
idempotency_key: spec_id+spec_version+input_hashes
active_stage: C0
stages:
  C0:
    status: in_progress
    inputs: []
    outputs: []
    facts: []
    inferences: []
    decisions: []
    evidence_refs: []
    open_questions: []
    next_action: build-scope-decision-package
    verification: []
    residual_risks: []
```

事实、推断和决策必须分字段记录。没有人工或上位流程作出的范围取舍，不得由 AI 写入 `decisions` 并视为已批准。

### 0.4 评审模式

每轮必须先选择且只能选择一个模式，禁止用规划产物冒充实现验收：

| 模式 | 目的 | 可以声明 | 不得声明 |
|---|---|---|---|
| `COVERAGE_PLANNING` | 从冻结任务反推应建设的验证单元；需要原型时再反推原型 | 规划覆盖完整、候选数量与缺口 | 产品能力已经实现 |
| `PROTOTYPE_AUDIT` | 审核可运行原型是否真实覆盖冻结任务 | 原型范围内的能力成熟度和覆盖结果 | 生产安全、性能或正式发布就绪 |
| `DELIVERY_AUDIT` | 审核待交付实现是否满足冻结能力 | 当前交付范围的产品完整性 | 未执行的生产或上线验证 |

### 0.5 结果域分离

阶段执行状态、流程运行结果和产品完整性结论属于三个不同状态域：

```yaml
stage_status: not_started|in_progress|blocked|awaiting_human|passed|failed|superseded
run_result: PASS|FAIL|BLOCKED
product_completeness: NOT_ASSESSED|PASS|FAIL|BLOCKED
```

- `stage_status=failed` 表示当前阶段门禁失败，但执行器仍须保存 finding 并按回退路由继续，不得因此停止发现其他问题。
- `COVERAGE_PLANNING` 的 `run_result=PASS` 只表示覆盖规划完成，`product_completeness` 必须为 `NOT_ASSESSED`。
- 只有 `PROTOTYPE_AUDIT` 或 `DELIVERY_AUDIT` 才能产生非 `NOT_ASSESSED` 的产品完整性结论，且结论必须带模式限定。

### 0.6 模式专属产物和差集

| 模式 | 必须使用的覆盖集合 | 允许的验证单元 | 必需产物 |
|---|---|---|---|
| `COVERAGE_PLANNING` | `planned_coverage` | 设计中的候选验证单元；不要求已实现 | 冻结场景、覆盖方案、成本与近似/精确标记 |
| `PROTOTYPE_AUDIT` | `verified_prototype_coverage` | 通过资格检查且绑定同源原型构建的原型单元/流 | 原型映射、运行证据、任务验证、原型差集 |
| `DELIVERY_AUDIT` | `verified_delivery_coverage` | UI/API/CLI/批处理/事件等合格验证单元/流 | 实现映射、运行证据、任务验证、交付差集 |

`claimed_coverage` 只记录提交方声称覆盖的场景，不能进入 PASS 公式；它必须与相应 `verified_*_coverage` 对账并输出夸大声明差集。

### 0.7 规范执行真实性门禁

“阅读、引用或口头说明已遵循本文”不等于执行本文。一次有效执行必须同时满足：

```text
存在 completeness/execution.yaml
+ 当前模式的必需标准产物实际存在且非空
+ execution.yaml 的每个 passed 阶段引用真实输出、输入哈希和验证记录
+ artifact-manifest.yaml 对产物路径、Schema版本、内容哈希和生成时间完成登记
+ 所有集合可由产物重算，报告中的数量与差集能反向核对
```

任一条件不满足，生成 `METHOD_NOT_EXECUTED`，`run_result=FAIL`，不得使用“已按本规范检查”“符合本规范”或任何等价表述。此时：

- 尚无产品缺陷证据时，`product_completeness=NOT_ASSESSED`。
- 已确认存在缺失、空壳、假成功或漏检时，`product_completeness=FAIL`；不得用“产物没生成”把已知产品失败降为未评估。
- 标准产物缺失属于执行方内部失败，不得标记为外部 `BLOCKED`。

## 1. 适用边界与职责分工

### 1.1 本规范负责什么

- 从多个独立来源建立产品的期望能力全集。
- 将产品能力拆为可验证的功能、任务、命令、状态与结果。
- 建立能力、界面载体、原型单元、测试和证据之间的双向追溯。
- 判定功能的实现深度与证据成熟度。
- 识别缺失能力、空壳页面、假交互、假成功、模型到消费者断链和任务未闭环。
- 从核心任务、角色、状态和设备覆盖反推所需原型单元。
- 计算 `REQUIRED - VERIFIED` 差集并失败关闭。
- 对门禁漏检执行归因、规范补强和回归验证。

### 1.2 本规范不负责什么

- 不内置某一行业或某一种产品必须拥有的固定功能清单。
- 不规定所有产品必须使用 Schema、持久化、撤销、发布或某种技术架构。
- 不代替需求决策，不擅自把领域基线中的全部候选能力纳入当前版本。
- 不代替 UI 的几何、视觉、响应式、Token 和可访问性审核。
- 不代替生产安全、性能、容量、灾备、法律和上线审批。
- 不用功能数量、原型数量或综合评分抵消核心缺陷。

### 1.3 与现有规范的关系

| 规范 | 核心职责 | 与本规范的衔接 |
|---|---|---|
| `02-流程-知识库驱动新产品创建与迭代.md` | 决定产品机会、版本目标和范围 | 向本规范提供产品目标、范围决策和知识库证据 |
| `05-研发流程规范.md` | 组织需求、设计、实现、评审和测试阶段 | 在需求冻结、设计评审、代码评审、集成与回归阶段调用本门禁 |
| 本规范 | 证明能力范围完整、实现真实、任务闭环 | 输出能力差集、原型覆盖和真实性结论 |
| `06-规范-AI_UI质量保障流程规范.md` | 检查已实现 UI 的视觉、响应式、交互状态和可访问性质量 | 只审核本规范确认存在且需要验证的界面载体和状态矩阵 |

裁决不互相覆盖：

```text
产品完整性 FAIL 不能被 UI 视觉 PASS 覆盖。
UI 质量 FAIL 不能被产品完整性 PASS 覆盖。
研发阶段门禁未通过，不能用原型演示完成替代。
```

## 2. 三层规范模型

本规范必须与领域内容和项目范围分离，形成三层输入。

### 2.1 第一层：通用完整性规范

即本文，定义：

- 能力如何提取、拆解和去重。
- 能力如何映射到功能、任务、界面、原型、测试和证据。
- 实现成熟度、证据成熟度和真实性检查方法。
- 原型数量的反推算法。
- PASS、FAIL、BLOCKED、NOT_APPLICABLE 的裁决规则。

### 2.2 第二层：领域能力基线

领域能力基线描述某一产品类型的候选能力空间，但不直接等于项目需求。每项能力必须包含适用条件和来源证据。对于设计器、编辑器、工作台、复杂控件、流程编排器等“单个对象内部仍包含多个可操作子结构”的产品，领域基线还必须提供 `DOMAIN_CAPABILITY_PROFILE`，把对象类别的最低候选功能剖面递归拆成能力原子；没有领域剖面时，不得用当前实现已声明的动作反推“应验证什么”。

示例目录结构：

```text
领域能力基线/
├── 某类设计器能力基线.yaml
├── 某类运营工作台能力基线.yaml
└── 某类数据产品能力基线.yaml
```

领域能力基线中的能力和能力原子只能作为 `CANDIDATE` 进入范围裁决。未满足 `applicable_when` 或被项目明确排除时，不得强制实现；但排除必须逐项记录事实、影响和批准来源，不能通过省略整个剖面来规避裁决。

### 2.3 第三层：项目范围合同

项目范围合同决定当前版本的真实承诺：

```text
REQUIRED        当前版本必须达到目标成熟度
OPTIONAL        可实现但不影响本版本核心通过
PLANNED         后续版本规划；当前若暴露入口必须明确禁用
EXCLUDED        明确拒绝或不在本版本范围
NOT_APPLICABLE  适用条件不成立，并记录理由和证据
UNRESOLVED      尚未完成范围裁决，不允许整体 PASS
```

不得用 `PLANNED`、`EXCLUDED` 或 `NOT_APPLICABLE` 隐藏界面已经声明的能力。界面存在可操作入口时，该入口自动进入待审集合，除非明确禁用且不产生能力承诺。

### 2.4 不可覆盖的范围变更账本

`scope-contract.yaml` 冻结后不得原地覆盖。每次变更必须追加到 `scope-change-ledger.yaml`：

```yaml
change_id: SCOPE-CHANGE-XXX
previous_scope_hash: sha256:...
new_scope_hash: sha256:...
changed_capabilities: []
reason: 变更原因
impact: 对任务、原型、实现和测试的影响
decision_ref: 上位人工决策或已批准ADR
approver_identity: 可验证身份
approved_at: ISO-8601
entry_state_after_change: removed|disabled|still_enabled
```

曾为 `REQUIRED`、曾暴露或已有 finding 的能力必须永久保留在审计历史中。只有范围变更来源可验证且相关入口已真实移除或禁用，才允许它不阻断新范围；历史 finding 不得删除，只能标记 `resolved_by_scope_change`。无法验证批准来源时保持 `UNRESOLVED`。

## 3. 核心对象与定义

### 3.1 产品目标 `GOAL`

面向用户或业务的可观察结果，回答“为什么做”。目标不能用页面、按钮或技术方案表达。

### 3.2 产品能力 `CAPABILITY`

产品帮助特定角色在约束下完成某类业务结果的稳定能力，回答“用户最终能够做成什么”。

```text
错误：拥有“新增区域”按钮。
正确：用户能够组织并调整多区域内容结构。
```

### 3.3 功能 `FEATURE`

系统为实现能力提供的可操作机制，回答“系统允许用户执行什么”。一个能力通常由多个功能组合实现，一个功能也可能服务多个能力。

### 3.4 用户任务 `TASK`

特定角色在给定前置条件和数据下，通过一系列功能达到业务结果的完整过程。任务必须具有起始态、过程、结果态以及适用的失败/恢复路径。

### 3.5 命令 `COMMAND`

由用户、系统或外部事件触发的一次意图表达。命令必须映射到前置条件、权威状态或业务结果变化、可观察结果和失败/禁用状态。

### 3.6 交互/执行载体 `SURFACE`

用户或系统执行任务的稳定入口，包括页面、路由、工作区、模式、弹层、抽屉、步骤、画布、原生窗口、API、CLI、事件订阅或批处理入口。UI 产品主要使用界面载体，无 UI 产品使用等价执行载体。`SURFACE` 不是原型数量单位。

### 3.7 验证单元 `VERIFICATION_UNIT`

能够独立到达或调用、重置、执行任务并采集证据的最小可验收运行单元。所有产品都使用 `VERIFICATION_UNIT`；有可运行原型时，其特化形式为 `PROTOTYPE_UNIT`。API、CLI、批处理、事件消费者或已交付系统可以直接形成非原型验证单元，不得为了满足本规范虚构 UI 原型。

### 3.8 原型应用 `PROTOTYPE_APPLICATION`

可以独立启动或打开的可运行应用。一个原型应用可以包含多个原型单元。

### 3.9 原型单元 `PROTOTYPE_UNIT`

可以独立到达、重置、执行任务和采集证据的最小可验收运行单元。一个有效原型单元必须同时满足：

```text
有稳定入口
+ 有可重置的初始状态
+ 有明确测试角色与数据
+ 可执行至少一个冻结任务
+ 产生权威状态或业务结果
+ 结果可观察
+ 覆盖适用的失败或禁用状态
+ 可独立采集证据
```

只有截图、按钮、页签、静态说明、不可操作页面或占位内容，不构成有效原型单元。

### 3.10 验证流与原型流 `VERIFICATION_FLOW / PROTOTYPE_FLOW`

跨多个验证单元完成一个端到端任务时，必须建立有序验证流；全部单元都是原型单元时可称原型流。流记录单元顺序、前后置条件、状态/数据交接、身份与权限上下文以及失败恢复；不得把多个互不连通的单元各自覆盖一部分步骤，错误计为完整任务已覆盖。

```yaml
prototype_flow_id: FLOW-XXX
task_id: TASK-XXX
ordered_units: [PROTO-A, PROTO-B]
handoffs:
  - from: PROTO-A
    to: PROTO-B
    contract: HANDOFF-XXX
    transferred_state: []
reset_method: 重置整条流的方法
evidence_refs: []
```

### 3.11 测试用例 `TEST_CASE`

对任务、功能或命令的可重放验证，包含前置条件、步骤、输入、预期状态、预期结果和失败判定。

### 3.12 证据 `EVIDENCE`

能证明某项断言的运行产物，包括结构化状态差异、接口结果、数据库/领域结果、DOM/原生视图状态、截图、录屏、日志、测试报告和独立复验记录。Toast、固定文本、源码字符串和单张最终截图不能单独证明能力实现。

### 3.13 能力变体 `CLAIM_VARIANT`

同一命令或能力下，对用户承诺不同结构、协议、结果或失败语义的选项必须登记为能力变体。多个入口只有在处理器、权威状态、消费者、序列化/契约、校验与失败语义等价且具有参数化契约证据时才可归并；共享事件处理器本身不构成等价证据。

当一组变体通过目录、控件面板、模板市场、连接器清单、策略注册表、插件列表、模型类型、报表类型或类似集合向用户或调用方暴露时，该集合称为 `CLAIM_FAMILY`。目录中的每个可识别条目都是独立 `CLAIM_VARIANT`，即使它们共用按钮、工厂、基类、页面或 Renderer 入口。

“支持全部”“提供 N 种”“组件丰富”等族级声明是对族内适用变体的全称承诺，不得通过抽测一个代表项证明。每个变体必须有独立稳定 ID、范围状态、适用操作、证明义务和验证结果。

### 3.14 权威状态合同 `AUTHORITATIVE_STATE_CONTRACT`

任何宣称达到 M3 及以上的能力必须记录：所有者、事实来源、写入边界、读取/消费边界、持久性或外部副作用、与目标交付运行时的关系、独立结果 oracle，以及 mock/stub/test double 的使用和等价性证据。写入者的返回值或同一临时 store 的回读不能单独证明权威效果。

### 3.15 领域能力剖面、能力原子模板与证明实例 `DOMAIN_CAPABILITY_PROFILE / CAPABILITY_ATOM_TEMPLATE / CAPABILITY_ATOM_INSTANCE`

`DOMAIN_CAPABILITY_PROFILE` 是针对一种产品对象类别、复杂组件类别或业务构件类别，从领域任务、竞品证据、风险和用户承诺推导出的候选功能空间。它描述“这一类对象在适用条件成立时可能需要证明什么”，不得由被审核实现的现有按钮、动作数组、测试名称或 Renderer 反向生成后冒充期望基线。

`CAPABILITY_ATOM_TEMPLATE` 是领域剖面中可复用的最小证明模式，描述一种对象类别可能承担的操作、结果和边界语义。模板本身不能进入覆盖率或 `VERIFIED`。

`CAPABILITY_ATOM_INSTANCE` 是模板绑定到具体审计主体、操作和适用上下文后形成的最小可独立证伪证明义务，必须能够独立回答：

```text
对哪个对象/子结构
+ 在什么前置和适用条件下
+ 执行什么命令或发生什么状态转换
+ 改变哪个权威状态/业务结果
+ 由哪个消费者产生什么可观察结果
+ 有哪些边界、失败、权限和恢复语义
+ 哪些持久化、反显、撤销或运行态消费维度适用
+ 用什么独立 oracle 判定成功
```

能力原子模板和实例都不是按钮、方法名、测试名或笼统状态名。诸如“表格”“配置态”“支持编辑”“有交互”均不合格；“删除列”可以成为模板，“对 `VARIANT-ENTRY-GRID` 在桌面设计态删除指定列，并在至少保留一列的约束下由设计态与运行态按新列集合重新渲染”才是可验证实例。

```text
A_instance = instantiate(
  atom_template_id,
  subject_kind + subject_id,       # 具体变体、对象类型或对象实例
  operation_id,
  applicable_context_key           # 角色、状态、平台、数据边界、失败/权限上下文
)
```

每个实例必须有稳定 `atom_instance_id`。模板相同不代表实例等价；不同变体、对象协议、操作或上下文只有取得覆盖第 4.5.2 节全部维度的等价证书，才能共享行为证明。即使共享证明，每个实例仍须有成员存在性、参数化执行和实例身份绑定证据。

复杂对象必须递归分解：

```text
产品能力
→ 功能
→ 对象/组件类别
→ 子结构或插槽
→ 对子结构的操作与状态转换
→ 边界/失败/权限/恢复
→ CAPABILITY_ATOM_TEMPLATE
→ 按具体主体、操作和上下文实例化为 CAPABILITY_ATOM_INSTANCE
```

父对象、父容器、原子模板或代表实例通过，不能证明后代子结构、兄弟原子或其他变体实例通过。递归仅在实例叶子已经可独立执行、观察、证伪且不再隐藏不同权威状态、消费者或失败语义时停止。

## 4. 能力全集的建立

### 4.1 能力全集公式

候选能力全集不得只来自当前实现或单一需求文档：

```text
U_candidate = D ∪ B ∪ C ∪ E ∪ R
```

其中：

- `D`：需求、产品合同和版本承诺中的能力。
- `B`：适用的领域能力基线。
- `C`：竞品、真实用户任务和业务流程推导出的候选能力。
- `E`：当前界面、公开 API、配置、菜单和文案实际声明的能力。
- `R`：风险、失败模式、法律、安全或兼容约束推导出的必要能力。

当前版本的必验集合由范围裁决产生：

```text
U_required = { c ∈ U_candidate | scope(c) = REQUIRED }
U_claimed  = 所有已暴露且可操作的UI/API/CLI/事件入口所声明的能力
U_audit    = U_required ∪ U_claimed
```

因此，即使需求漏写了某能力，只要界面已经暴露可操作入口，也必须进入真实性检查。

`U_required` 为空时禁止计算覆盖率或产生 PASS。执行者必须区分：

- 产品确实没有需要本规范验收的能力：本轮 `run_result=BLOCKED`，提交范围适用性裁决，不产生产品完整性结论。
- 范围尚未完成：保持 `UNRESOLVED/awaiting_human`。
- 通过排除清空范围：生成 `SCOPE_TAMPERING` 并判 FAIL。

`U_claimed` 中尚未登记范围或目标成熟度的能力标记为 `UNDECLARED_UI_CLAIM`。它的临时目标成熟度取界面/API文案、控件状态和用户可执行路径所承诺的最低成熟度；无法无歧义推导时标记 `UNRESOLVED` 并使门禁 `BLOCKED`。执行者不得把已启用入口默认降为 M1，也不得替产品负责人擅自扩大为 M5/M6。

### 4.2 能力叶子化

能力树的叶子必须能独立回答：

- 谁使用？
- 在什么条件下适用？
- 要完成什么结果？
- 哪些功能共同实现？
- 达到什么成熟度才算本版本完成？
- 用什么任务和证据证明？

只有名词、模块名或过大的“管理全部业务”不得作为可验收能力叶子。

### 4.3 能力基线充分性与停止条件

能力全集只有同时满足以下条件才能冻结：

```text
D/B/C/E/R 五类来源均已执行并记录命中或“无证据”
核心角色和核心任务均能追溯到能力叶子
所有已暴露声明都进入 U_claimed
独立推导新增能力已全部裁决或保持 UNRESOLVED
连续两轮独立推导不再产生新的 REQUIRED/P0 能力
能力依赖闭包已展开且无未裁决循环
```

不能因为“暂时没想到更多能力”结束提取。停止条件证据写入 `capability-reconciliation.yaml`。D/B/C/E/R 每类来源必须在 `source-discovery.yaml` 中记录检索计划、实际位置/查询、时间、工具、输入哈希、命中与未命中结果、失败和重试；仅填写“无证据”而没有可重放发现记录视为未执行。`source_discovery_gap` 非空时 C1/C2 FAIL。

范围分类也必须对账：由产品目标、核心任务、已启用声明、依赖闭包和风险规则确定的候选 REQUIRED/P0/P1 集合，与 Scope Owner 的分类结果比较。任何被降为 OPTIONAL/P2/EXCLUDED/N_A 的候选项都必须有逐项决策和影响证据；否则进入 `scope_classification_gap`，不能从 `U_required/A_required/S_required` 中移除。

### 4.4 去重、依赖与冲突

能力去重按用户结果、适用条件、权威效果、消费者、序列化/协议和失败语义判断，不按按钮名称或共享处理器判断。不同 `CLAIM_VARIANT` 只有满足第 3.13 节等价合同才能归并。出现冲突时保留来源记录，并在范围合同中作出一次显式裁决。未经裁决的冲突能力保持 `UNRESOLVED`。

每个 REQUIRED 能力的 REQUIRED 依赖也必须进入 `U_required`；若依赖被 EXCLUDED/N_A，依赖它的能力不得保持可通过状态。依赖图必须无环；检测到循环时必须拆解能力边界或保持 `UNRESOLVED`。

### 4.5 可枚举能力族与变体闭包

能力提取阶段必须扫描所有可枚举声明源，至少包括适用的：

```text
可见目录、菜单、面板、选项和创建入口
组件/插件/模板/连接器/策略/驱动/模型注册表
Schema 枚举、配置清单、API 类型、命令参数与功能开关
需求、帮助、营销文案和数量声明
运行时实际可创建、导入、选择或执行的类型
```

对每个 `CLAIM_FAMILY` 建立来源集合并执行基数与成员对账：

```text
V_source     = 所有独立来源枚举出的稳定变体 ID 并集
V_inventory  = variant-inventory.yaml 中登记的变体 ID
V_claimed    = 已暴露、可选择、可创建、可拖入、可调用或被族级文案承诺的变体
V_required   = 范围为 REQUIRED 的变体，以及 REQUIRED 族能力下未被合法排除的变体
V_audit      = V_required ∪ V_claimed

unexpanded_variant_gap = V_source - V_inventory
orphan_variant_gap     = V_inventory - V_source - 合法的新增设计项
```

`V_source` 的每个来源必须记录读取位置、来源哈希、发现时间和成员列表。只登记数量而没有成员 ID，或只登记分类而没有叶子变体，均视为未展开。`unexpanded_variant_gap` 非空时生成 `UNEXPANDED_VARIANT_FAMILY`，后续覆盖率不得计算为 PASS。

#### 4.5.1 族级操作承诺传播

族级能力或同一上下文中的可操作流程会向适用变体传播证明义务。例如产品同时声明“可选择某类物料”和“可配置、预览、运行这些物料”，则每个未明确禁用的物料变体都必须验证相应生命周期，而不能只证明它出现在目录中。

```text
O_family   = 从产品合同、界面/API和任务提取的族级操作
O_required(v) = { o ∈ O_family | applicable_when(v, o) = true }
P_required(v) = O_required(v) × 该操作适用的状态/角色/平台/数据边界
```

操作集合由实际声明推导，不预设所有产品都必须具有设计、配置、保存、预览或发布。执行者必须为不适用操作提供事实和范围裁决；未声明不适用时，不得把族级已启用操作从个别变体上静默移除。

#### 4.5.2 等价类与共享证明

变体只有取得 `VARIANT_EQUIVALENCE_CERTIFICATE` 才能共享实现或原型证明。证书至少比较：

```text
命令与处理器契约
权威状态与持久/序列化契约
消费者或 Renderer
配置/属性编辑契约
结构、插槽和父子约束
数据绑定与校验
权限、设备和状态行为
失败与恢复语义
```

任一维度不同即必须拆分等价类并分别验证。共用基类、工厂、事件处理器、通用占位 Renderer、相同截图外观或相同注册表字段不能单独构成等价证据。

即使取得等价证书，每个变体仍必须有成员存在性、适用性和参数化执行证据；证书只允许共享行为证明，不能把未执行的成员记为已验证。证书失效或缺失时，默认每个变体独立进入待审集合。

### 4.6 `applicable_when` 与场景排除协议

适用条件必须使用项目声明的受限表达式语言，并引用冻结的角色、状态、平台、数据边界和风险维度。条件缺失默认 `true`，解析失败或事实不足为 `UNRESOLVED`，不得默认排除。

每个排除项必须记录：

```yaml
scenario_exclusion_id: EXC-XXX
dimension: role|state|platform|data_boundary|failure|recovery|other
value: 被排除值
condition: 受限表达式
fact_refs: []
reason: 排除理由
decision_ref: 批准来源
approver_identity: 可验证身份
expires_at: ISO-8601|null
```

证明剖面和风险基线规定的强制维度不得由执行者排除。独立评审必须反证排除条件；不能证明不适用时进入 `S_required` 或保持 `UNRESOLVED`。

受限表达式只允许：已登记维度标识符、字符串/数字/布尔常量、`== != < <= > >=`、集合包含、`and/or/not` 和括号；禁止任意函数、代码执行、网络/文件访问和时间依赖。表达式、维度字典、求值输入与结果必须进入证据包，保证可重放。

### 4.7 领域功能剖面实例化与能力原子闭包

每个冻结能力、可枚举变体和复杂对象必须选择适用的 `DOMAIN_CAPABILITY_PROFILE`，再结合 `D/B/C/E/R` 五类来源形成期望原子全集。通用规范不内置某一行业或控件的固定原子清单，但强制项目先建立或选择领域剖面，再开始实现映射和测试生成。

```text
T_baseline = 适用领域剖面推导的候选原子模板
T_demand   = 需求、产品合同和冻结任务直接声明的原子模板
T_compete  = 竞品证据与真实用户任务推导的原子模板
T_claimed  = 当前 UI/API/CLI/配置和文案实际承诺的原子模板
T_risk     = 风险、失败模式、权限、恢复和兼容要求推导的原子模板

T_candidate = T_baseline ∪ T_demand ∪ T_compete ∪ T_claimed ∪ T_risk
A_candidate = 对 T_candidate 按具体审计主体、操作和适用上下文实例化后的原子实例全集
A_claimed   = 已暴露声明产生的原子实例
A_required  = { a ∈ A_candidate | scope(a) = REQUIRED }
A_audit     = A_required ∪ A_claimed
A_declared_implementation = 实现合同、动作数组、注册表或测试当前列出的原子实例
A_verified  = 逐实例满足实现成熟度、证据成熟度和独立 oracle 的原子实例
```

必须同时计算：

```text
unmodeled_atom_gap      = A_audit - A_declared_implementation
undeclared_impl_atom_gap = A_declared_implementation - A_candidate - 合法新增设计项
atom_verification_gap  = A_audit - A_verified - 合法禁用或排除的原子
```

`A_declared_implementation` 只能用于对账和实现映射，不能作为 `A_candidate` 或 `A_required` 的唯一来源。若领域剖面缺失，执行器必须先生成候选剖面并由独立领域角色复核；在剖面完成前，C1/C2 不得通过，覆盖率不得计算，任何“完整”结论均失败关闭。

原子模板不能直接与实现、测试或证据相连，必须先形成实例。`A_candidate` 的基数必须能按 `模板 × 适用主体 × 操作 × 上下文等价类` 重算；只登记模板、只执行代表变体或把同一 evidence ID 无条件复制给多个实例，生成 `atom_instance_coverage_gap`。

领域剖面至少按适用性检查以下维度，而不是机械要求全部实现：

```text
结构：子结构、插槽、嵌套、组合、数量与顺序
数据：绑定、类型、默认值、格式、转换、序列化与反显
操作：创建、读取、修改、删除、移动、复制、批量和撤销
状态：默认、选中、焦点、编辑、只读、禁用、空、加载和错误
规则：校验、条件、联动、公式、权限和冲突
运行：预览、真实消费者、提交/执行、刷新和恢复
质量：可访问性、响应式、性能、安全和兼容（命中风险时）
```

每个维度必须记录 `REQUIRED|OPTIONAL|PLANNED|EXCLUDED|NOT_APPLICABLE|UNRESOLVED`，`NOT_APPLICABLE` 必须引用事实和裁决。不能因为实现中没有对应动作就省略该维度。

是否属于复杂对象必须由 `object-inventory.yaml` 的结构事实判定，而不是由实现者手工设置布尔值。命中任一条件即进入复杂对象候选集：具有可变子结构/插槽、可递归嵌套、两个以上独立命令族、独立子状态/子权限/子校验、子结构序列化、或领域剖面/用户任务声明内部可操作结构。候选复杂对象只有经独立评审和可重放事实证明全部条件不成立，才能降为普通对象；否则 `complex_object_profile_gap` 非空并阻断 PASS。

递归必须终止但不得截断语义。每个节点记录 `object_path`、父子关系和拆分判据；达到可独立执行、观察、证伪且不再隐藏不同状态/消费者/失败语义时停止。若继续拆分只会复制相同合同，可用等价证书归并；若超过项目声明的深度/数量预算仍未闭包，状态为 `UNRESOLVED`，不得静默截断或因原子数量过多降低范围。

#### 4.7.1 聚合声明的量词规则

“完整支持”“所有组件可配置”“每个字段可移动”等是全称声明，只有所有适用原子通过才成立：

```text
aggregate_pass = every(atom_instance ∈ A_audit, result(atom_instance) = PASS)
```

`some(...)`、任意一个代表动作、任意一个成员或任意一个状态通过，只能证明显式登记的存在性声明，不能证明全称或完整性声明。使用存在量词替代全称证明时生成 `AGGREGATE_EXISTENTIAL_PROOF`，整体 FAIL。

#### 4.7.2 证据独立性与禁止循环证明

期望原子、实现映射、测试执行和最终裁决必须保留来源血缘：

```text
期望来源：领域基线/冻结需求/任务/风险/界面声明
实现来源：代码、Schema、动作合同、处理器和消费者
oracle 来源：冻结验收结果、领域不变量或独立结果合同
执行证据：对当前构建真实执行所得原始产物
裁决来源：独立 Verifier 对期望、实现和执行证据的对账
```

以下链路属于循环证据，不得进入 PASS：

```text
从实现动作数组生成“期望功能”
→ 由同一动作数组生成测试或 Renderer
→ 检查测试/Renderer包含该动作字符串
→ 反向宣称实现完整
```

生成器可以生成案例和执行脚本，但生成器输出本身不是实现证据；验证器必须读取独立冻结的能力原子与 oracle 合同，实际执行每个适用原子，并通过认证反例证明能够抓住缺失动作、错误消费者、遗漏失败状态和后代子结构缺陷。生成器、被审核实现和最终裁决若共享同一个不完整合同且没有独立基线对账，生成 `CIRCULAR_EVIDENCE`，整体 FAIL。

## 5. 产品能力到功能的拆解

### 5.1 拆解原则

对每个能力使用操作符和结果链拆解功能：

```text
创建 / 读取 / 修改 / 删除
组织 / 排序 / 组合 / 拆分
配置 / 绑定 / 转换 / 校验
执行 / 暂停 / 恢复 / 撤销
导入 / 导出 / 同步 / 发布
授权 / 审计 / 诊断 / 回滚
```

只保留完成能力所必需的功能，禁止为了“显得完整”无证据扩张范围。

### 5.2 功能合同

每项功能至少包含：

```yaml
feature_id: FEAT-XXX
capability_ids: [CAP-XXX]
atom_ids: [ATOM-XXX]
name: 功能名称
scope: REQUIRED
criticality: P0|P1|P2
applicable_when: 适用条件表达式
preconditions: []
commands: [CMD-XXX]
authoritative_effects: []
consumer_effects: []
observable_results: []
failure_states: []
conditional_dimensions:
  persistence: REQUIRED|NOT_APPLICABLE
  undo_redo: REQUIRED|NOT_APPLICABLE
  permissions: REQUIRED|NOT_APPLICABLE
target_maturity: TASK_COMPLETE
task_ids: [TASK-XXX]
```

`conditional_dimensions` 必须由界面声明、产品合同和能力语义推导。不得因为某类产品经常支持保存、撤销或发布，就无条件要求所有项目实现。

### 5.3 完整性不变量

```text
每个 REQUIRED 能力至少映射一个 REQUIRED 功能。
每个 REQUIRED 功能至少映射一个可执行任务或直接可验证命令。
每个 REQUIRED 功能、复杂对象和 `V_audit` 变体的适用能力原子闭包必须展开；未建模原子不能按“不存在”处理。
每个可操作命令必须映射权威效果和可观察结果。
每个权威效果必须有真实消费者或可重读业务结果。
每个核心任务必须映射至少一个原型单元或原型流，并映射测试用例。
每个 V_audit 变体必须映射适用的功能、场景、实现、测试和证据，或有合法禁用/排除记录。
族级完成声明只有在成员闭包已展开且所有适用变体通过时成立。
全称或完整性声明只有在所有适用能力原子通过时成立，禁止用 `some`、代表动作或父级通过替代逐原子证明。
```

全部关系都是多对多关系，不得假设“一项能力对应一个功能”或“一个功能对应一个原型”。标准追溯矩阵的一行表示一条可验证路径，而不是对象间唯一关系。以下映射必须分别检查正向缺口与反向孤儿：

```text
M_goal_capability
M_capability_feature
M_feature_task
M_task_command
M_command_surface
M_task_prototype_unit_or_flow
M_task_test_case
M_test_case_evidence
M_family_variant
M_variant_scenario
M_variant_prototype_unit_or_flow
M_variant_implementation
M_variant_test_case
M_variant_evidence
```

## 6. 功能到任务、界面和原型的映射

### 6.1 先形成任务，不直接按功能生成页面

功能必须组合为真实任务。任务合同至少包含：

```yaml
task_id: TASK-XXX
goal_ids: [GOAL-XXX]
capability_ids: [CAP-XXX]
feature_ids: [FEAT-XXX]
role: ROLE-XXX
criticality: P0|P1|P2
preconditions: []
test_data: DATASET-XXX
start_state: STATE-START
steps:
  - command_id: CMD-001
    expected_state: STATE-001
    expected_result: RESULT-001
success_state: STATE-DONE
failure_states: [STATE-ERROR]
recovery_path: [CMD-RETRY]
required_viewports: []
required_platforms: []
```

### 6.2 任务到交互/执行载体

对每个任务步骤标注发生在哪个 `SURFACE`。同一任务可以跨多个载体；同一载体也可以服务多个任务。

未映射任何任务的可操作载体必须被标记为：

- 有明确辅助价值和需求来源；或
- `UNDECLARED_UI_CLAIM`。

### 6.3 载体到验证单元或验证流

相邻载体满足以下条件时可以合并为一个验证单元；在 `PROTOTYPE_AUDIT` 中该单元必须是原型单元：

- 共享核心对象、角色和任务上下文。
- 用户能连续完成任务，不依赖人工伪造中间状态。
- 共享可重置状态和测试数据。
- 合并后每个任务仍可独立到达、回放和取证。

出现以下任一条件时必须拆分验证单元：

- 用户角色、核心目标或数据上下文显著不同。
- 信息架构、操作模型或运行环境不同。
- 设计态与运行态需要独立证明。
- 一个任务无法独立重置或采集证据。
- 合并后只能用静态页签或手工跳状态模拟任务。
- 单元规模导致关键正常态、失败态或权限态无法验证。

若端到端任务必须跨多个合格验证单元，使用 `VERIFICATION_FLOW` 表达有序组合；全部为原型单元时可使用 `PROTOTYPE_FLOW`。只有整条流的入口、交接、重置、成功结果和失败恢复均可重放时，该任务场景才算覆盖。流不是额外的应用或单元，不重复增加数量。

不同信任域、身份域、发布物、运行时或不可共享重置边界的单元不得为减少数量而合并。最小化目标服从这些硬边界，不得以原型应用数量更少为理由破坏隔离。

## 7. 如何反推需要多少原型

### 7.1 不预设固定数量

规范禁止使用下列算法：

```text
一个功能一个原型
一个页面一个原型
一个菜单一个原型
一个截图一个原型
竞品有多少页面就做多少原型
```

### 7.2 建立验收场景全集

从冻结的核心任务派生验收场景：

```text
S_required = 核心任务 × 适用角色 × 必测业务状态 × 必测平台/设备 × 关键数据边界
             × 适用的 CLAIM_VARIANT 与族级操作证明义务
```

不采用无意义的全笛卡尔积。只有 `applicable_when` 成立且风险要求覆盖的组合进入 `S_required`，并保留排除理由。

### 7.2.1 变体原型覆盖与“独立原型”的判定

每个 `V_audit` 变体必须具有独立的 `prototype_case_id`，记录稳定到达方式、重置方式、测试数据、适用状态和证据。独立验证身份不等于机械创建一个 HTML 文件：一个原型应用可以承载多个变体，一个参数化工作台也可以承载多个原型案例。

只有同时满足以下条件，多个变体案例才可共享同一个 `PROTOTYPE_UNIT`：

- 每个变体可通过稳定路由、参数或可重放选择步骤独立到达和重置。
- 每个变体均执行自己的适用场景并产生独立证据，不能只切换标题或图标。
- 共享单元不会隐藏结构、交互模型、配置面板、运行结果、失败/权限/设备状态的差异。
- 行为证明共享时具有第 4.5.2 节的有效等价证书。

结构、交互模型、配置协议、运行消费者或失败语义不同的复杂变体必须拆分为不同 `PROTOTYPE_UNIT` 或在同一应用中提供不同的可独立验收单元。只有目录条目、Manifest、通用占位框、静态说明或统一“等待绑定”输出的变体，最高只能证明 M1，生成 `SHELL_ONLY` 或 `VARIANT_COVERAGE_GAP`。

### 7.3 构造候选验证单元

每个候选验证单元记录它能规划或真实覆盖的场景集合：

```yaml
verification_unit_id: UNIT-XXX
unit_kind: prototype|ui|api|cli|batch|event|native
application_id: APP-XXX
entry: 路由或稳定入口
reset_method: 重置方式
role_fixture: ROLE-XXX
data_fixture: DATASET-XXX
surface_ids: []
planned_scenario_ids: []
claimed_scenario_ids: []
verified_scenario_ids: []
evidence_capture: []
```

`COVERAGE_PLANNING` 使用 `planned_scenario_ids`，不要求单元已经实现，但必须有可实现性、入口、重置和证据设计；审计模式只能使用实际通过“可到达、可重置、可执行、可观察、可取证”资格检查的 `verified_scenario_ids`。跨单元场景还必须建立合格验证流，其覆盖集合记在流而不是任一单独单元上。

### 7.4 最小充分覆盖算法

目标是选择最少但足够的验证单元集合，并保留覆盖跨单元场景所需的验证流；`PROTOTYPE_AUDIT` 再统计其中的原型应用和原型单元：

```text
寻找 V* ⊆ V_candidate，F* ⊆ F_candidate
使得 Union(Coverage(v), v ∈ V*) ∪ Union(Coverage(f), f ∈ F*) = S_required
且每个 f ∈ F* 的 ordered_units 均属于 V*

按以下字典序最小化：
1. 违反信任域/运行时/重置硬边界的数量（必须为 0）
2. 验证单元数量；仅原型模式再最小化原型应用数量
3. 跨单元切换次数
4. 状态耦合与证据不可独立风险
```

执行前先做可行性检查：规划模式要求每个必验场景至少被一个候选验证单元/流规划覆盖；审计模式要求至少被一个合格验证单元/流实证覆盖。否则生成对应 `planning_gap` 或 `verification_gap`。

候选原型单元不超过 20 个时，应枚举子集或使用确定性集合覆盖求解器，按上述字典序取得全局最优解。候选超过 20 个或求解成本超出项目预算时，可执行确定性贪心近似：

1. 场景权重固定为 P0=100、P1=10、P2=1；同级按场景 ID 排序。
2. 每轮选择“新覆盖场景权重 / 新增单元与切换成本”最高的候选单元或原型流；平局按稳定 ID 排序。
3. 标记其覆盖场景。
4. 重复直到所有核心场景被覆盖。
5. 尝试删除任一单元或原型流；若仍保持 100% 覆盖，则删除该冗余项。
6. 对剩余每个单元和原型流重新执行资格检查。

贪心结果必须标记 `solution_optimality: approximate`，不得声称全局最少；精确求解结果标记 `solution_optimality: exact`。任何算法结果都不能绕过原型单元和原型流资格检查。

### 7.5 必须报告的数量

每次交付必须同时报告：

```text
verification_unit_count      合格验证单元数
verification_flow_count      跨单元端到端验证流数
prototype_application_count  原型模式下可独立启动的原型应用数
prototype_unit_count         原型模式下可独立验收的原型单元数
prototype_flow_count         原型模式下的跨单元原型流数
surface_count                UI/API/CLI/事件等交互与执行载体数
required_scenario_count      必验场景数
covered_scenario_count       已有合格证据的场景数
core_task_coverage           covered / required
uncovered_scenarios          未覆盖场景清单
declared_variant_count       V_audit 变体数
verified_variant_count       达到目标成熟度和证据成熟度的变体数
variant_equivalence_classes  有效等价类数量
variant_prototype_case_count 具有独立验证身份的变体原型案例数
uncovered_variants           未覆盖或证据不足的变体清单
```

只报告“做了 N 个原型”不合格。

## 8. 实现成熟度与证据成熟度

### 8.1 实现成熟度

成熟度必须单调，达到高等级必须满足所有低等级条件：

| 等级 | 名称 | 判定条件 |
|---|---|---|
| M0 | `ABSENT` | 没有实现或无法找到载体 |
| M1 | `SURFACED` | 有可到达界面、API 或入口，但不证明可用 |
| M2 | `WIRED` | 入口连接到真实命令处理器，能够触发行为 |
| M3 | `AUTHORITATIVE_EFFECT` | 命令改变该能力的权威状态或真实业务结果 |
| M4 | `CONSUMED_OUTCOME` | Renderer、下游模块或重新读取真实消费该变化并产生正确结果 |
| M5 | `TASK_COMPLETE` | 与依赖能力组合后可完成冻结任务及适用失败/恢复路径 |
| M6 | `HARDENED` | 已通过本版本要求的安全、性能、兼容、可访问性和恢复验证 |

`M1` 不能称为“功能已实现”；`M2` 不能证明结果真实；`M3` 不能证明结果被消费；只有目标成熟度达到项目范围合同要求时才进入 `VERIFIED`。

目标成熟度不得由项目任意降低。默认下限如下：

| 范围/能力类型 | 完整性 PASS 的最低目标 |
|---|---|
| REQUIRED 且表达用户/业务结果的能力 | M5 `TASK_COMPLETE` |
| REQUIRED 的配置、命令、工作流、集成或复合容器能力 | M5 `TASK_COMPLETE` |
| REQUIRED 的纯展示/导航能力 | M4 `CONSUMED_OUTCOME`，且完成其全部适用展示任务 |
| 涉及安全、权限、金额、数据破坏、合规或正式交付的高风险能力 | M6 `HARDENED` |
| OPTIONAL/PLANNED 探索性里程碑 | 可低于 M4，但不能进入产品完整性 `VERIFIED` |

低于默认下限的交付只能声明 `PARTIAL_MILESTONE`，不能产生产品完整性 PASS。降低默认下限属于风险接受，必须有上位人工决策和到期条件；高风险能力不得豁免到 M5 以下。高层用户结果不得被“页面可见”“按钮接线”等低层任务替代，能力 `user_outcome` 与任务 `success_state` 必须通过语义一致性评审。

### 8.2 证据成熟度

| 等级 | 名称 | 判定条件 |
|---|---|---|
| E0 | `NONE` | 无证据 |
| E1 | `STATIC` | 仅源码、文档或静态结构证据 |
| E2 | `EXECUTED` | 已执行但不可稳定重放或证据不完整 |
| E3 | `REPRODUCIBLE` | 输入、环境、步骤、预期和产物可重放 |
| E4 | `INDEPENDENT` | 由独立执行上下文复验并得到一致结果 |

核心能力至少需要 `E3`；高风险完成声明按上位规范要求提升到 `E4`。源码存在、单元测试直接调用内部函数、Toast、固定文本或最终截图不能单独超过 `E1`。

E3 证据必须同时具备：`artifact_hash`、环境/工具版本、输入与 fixture 哈希、前置状态、可执行步骤、确定性 oracle、原始产物哈希、执行时间和重放命令；任一缺失最高 E2。E4 还必须具备独立执行身份、上下文隔离声明、独立采集的原始产物以及与 E3 结果一致性对账；只复用生成者的报告不构成 E4。

### 8.3 同源构建和测试数据约束

所有进入同一门禁结论的验证单元必须绑定待审 `artifact_hash`、构建配置、依赖锁、功能开关和目标运行环境。原型审计必须使用与交付原型相同的命令处理器和消费者；交付审计必须使用待交付构建或经批准的等价环境。

fixture 只能准备前置数据、身份和外部依赖，不得直接写入被验证的中间态、成功态或跳过任务步骤。必须执行至少一次数据扰动或 fixture 交换，证明结果由真实链路产生；审计专用条件分支、shadow store 或只供测试读取的消费者按 `FAKE_SUCCESS` 处理。

## 9. 按能力选择验证路径

不得一刀切假设所有能力使用同一种架构。每项能力或命令必须基于证据选择主验证路径：

| 路径 | 选择条件 | 核心链路 |
|---|---|---|
| `SCHEMA` | Schema、元数据或页面定义确实参与写入和消费 | 输入 → 控件 → 内存 → Schema → Renderer/生成器 → 结果 |
| `EQUIVALENT_MODEL` | 使用 Store、ViewModel、领域对象、场景树或其他权威模型 | 输入 → 控件 → 内存 → 权威模型 → 消费者 → 结果 |
| `BACKEND_RESULT` | 能力的权威结果位于服务端或外部系统 | 输入 → 请求 → 服务端/外部权威结果 → 重读/下游消费 → 结果 |
| `PRESENTATION_ONLY` | 只声明展示、导航或无状态交互 | 数据来源 → 显示/导航 → 可观察结果及适用错误态 |
| `MIXED` | 一个能力跨多个权威边界 | 分段声明各路径、边界和一致性断言 |
| `UNRESOLVED` | 声明了状态型能力但无法识别真实状态来源或消费者 | 结果为 `BLOCKED`，不得判 PASS |

代码中出现 `schema`、`store` 或请求函数名称不构成路径证据。必须证明该结构参与当前能力的真实写入和消费。

## 10. 能力证明剖面

不同能力需要不同验证维度，不能用一个按钮点击脚本覆盖全部类型。

### 10.1 展示能力 `PRESENTATION`

验证数据来源、正常显示、空态、错误态、长内容、权限隐藏和刷新一致性中适用的部分。

### 10.2 命令能力 `COMMAND`

验证前置条件、命令处理器、权威效果、可观察结果、失败/禁用、重复执行和权限。

### 10.3 配置/编辑能力 `CONFIGURATION_EDITOR`

验证架构路径、属性写入、消费者投影、适用的保存恢复、撤销重做、预览/生成结果及非法配置阻断。

### 10.4 复合容器 `COMPOSITE_CONTAINER`

验证默认内部结构、子结构增删排序、合法嵌套、非法投放、空态、最小/最大数量、序列化、反显和运行态消费。

### 10.5 工作流 `WORKFLOW`

验证状态机、转换条件、角色权限、副作用、并发、超时、失败恢复和审计。

### 10.6 集成能力 `INTEGRATION`

验证真实接口或等价测试环境、契约、错误、重试、幂等、降级和重新读取。

### 10.7 横切能力 `CROSS_CUTTING`

验证权限、审计、国际化、可访问性、响应式等是否覆盖全部适用任务路径，而不是只在一个演示页面存在。

## 11. 双向追溯矩阵

### 11.1 正向追溯

```text
每个 REQUIRED GOAL
→ 至少一个 REQUIRED CAPABILITY
→ 至少一个 FEATURE
→ 至少一个 TASK/COMMAND
→ 至少一个合格 PROTOTYPE_UNIT 或 PROTOTYPE_FLOW
→ 至少一个 TEST_CASE
→ 至少一个新鲜 EVIDENCE
```

任何断点都进入缺口清单。

### 11.2 反向追溯

```text
每个可见/可调用入口
→ 必须映射 COMMAND
→ COMMAND 必须映射 FEATURE/CAPABILITY
→ 必须具有权威效果、可观察结果和证据
```

无法反向追溯的可操作入口为 `UNDECLARED_UI_CLAIM`。明确禁用的规划入口必须映射 `PLANNED` 并展示真实禁用状态。

atom_instance_id: ATOM-INSTANCE-XXX

## 12. 标准执行工作流 C0-C10

### C0 产品章程与候选范围登记

输入：产品目标、角色、版本目标、风险等级和上位范围决策。  
动作：登记已批准决策，生成候选范围和待决策项；不得在能力全集建立前冻结最终范围。  
输出：`product-charter.yaml`、`scope-draft.yaml`。  
门禁：产品类型、核心角色、候选核心任务和非目标明确；未确认项保持 `UNRESOLVED`。

### C1 建立候选能力全集

从 `D/B/C/E/R` 五类来源提取、去重并保留来源；发现并展开所有可枚举 `CLAIM_FAMILY`，逐来源对账成员闭包；为每类复杂对象选择或建立 `DOMAIN_CAPABILITY_PROFILE`，实例化候选能力原子全集。  
输出：`source-discovery.yaml`、`object-inventory.yaml`、`capability-baseline.yaml`、`domain-capability-profiles.yaml`、`capability-atom-templates.yaml`、`capability-atom-instances.yaml`、`variant-inventory.yaml`。  
门禁：每项能力和能力原子有独立于实现的来源、适用条件和结果合同；所有复杂对象均绑定领域剖面；`source_discovery_gap`、`complex_object_profile_gap`、`unexpanded_variant_gap` 为空，任何数量声明均能回溯到稳定成员 ID。缺少剖面、来源发现不可重放或只从实现动作/测试反推原子时 C1 FAIL。

### C2 独立能力推导与对账

由未读取实现结论的独立评审者从目标、任务、领域基线、竞品/用户证据、风险和界面声明重新推导能力、领域剖面、能力原子与变体来源，并与 C1 对账；必须尝试提出被实现合同遗漏的原子。  
输出：`capability-reconciliation.yaml`、`independent-reviews/*.yaml`。  
门禁：新增、缺失、冲突和排除均有裁决或保持 `UNRESOLVED`；连续两轮独立推导不再产生新的 REQUIRED/P0/P1 原子；`independence_gap` 为空；独立推导者未读取实现者的“已完成动作清单”后照抄结论。

### C3 范围与目标成熟度冻结

输入：C0 章程、C1/C2 对账后的能力全集和待决策项。  
动作：由 Scope Owner 决策能力范围，为能力和功能设置关键级别、适用条件、证明剖面和不低于默认下限的目标成熟度。  
输出：冻结版 `scope-contract.yaml`。  
门禁：所有核心能力均为 `REQUIRED` 或有合法排除理由；`scope_classification_gap` 为空；`U_required` 非空；合同首次冻结。后续变化只能走追加式范围变更账本。

### C4 功能、任务和场景拆解

建立能力到功能合同、用户任务和验收场景；将每个复杂对象递归拆到可独立证伪的能力原子；将族级操作承诺传播到每个适用变体和适用原子并生成场景。  
输出：`feature-inventory.yaml`、`scenario-inventory.yaml`。  
门禁：正常、边界、失败和恢复路径按适用性覆盖；每个 `V_audit` 变体的适用操作和每个 `A_audit` 原子均有场景或合法排除记录；父对象或代表原子不得覆盖未展开的子结构义务。

### C5 原型覆盖设计

建立 Surface 清单、候选原型单元、变体原型案例和场景覆盖关系，执行最小充分覆盖算法。  
输出：`prototype-map.yaml`。  
门禁：所有核心场景映射到合格原型单元；`prototype_variant_gap` 与未覆盖集合为空；共享单元具有稳定变体入口，行为共享具有有效等价证书。

### C6 实现映射与架构探测

为每项能力原子、命令和适用变体找到入口、合法载荷、处理器、权威状态、消费者、可观察结果、失败边界及适用的持久化/反显路径，选择验证路径。  
输出：`implementation-map.yaml`。  
门禁：无法识别的核心路径按内部未实现记 FAIL；`unmodeled_atom_gap`、`implementation_atom_gap` 和 `implementation_variant_gap` 为空后才可通过，不得以共用处理器、动作字符串或父级 Renderer 替代逐原子、逐变体消费者证明。

### C7 静态负向扫描

扫描缺失入口、无处理器入口、占位页面、固定结果、旁路状态、无消费者写入、无测试映射、规划能力误启用、未展开能力族、未建模能力原子、聚合声明无逐成员/逐原子证据、存在量词冒充全称证明、循环证据和通用占位 Renderer。  
输出：初始 `missing-or-fake.json`。  
门禁：发现已确认缺陷立即记 finding，不因后续动态覆盖率高而删除。

### C8 动态真实性验证

使用真实浏览器、API、原生运行环境或等价集成环境逐能力、逐适用变体、逐适用能力原子执行证明剖面；对状态型原子验证合法载荷、权威变化、消费者结果、边界/失败，以及命中条件时的保存、刷新反显、撤销和运行态消费。  
输出：`capability-verification.json` 和运行证据。  
门禁：核心能力达到目标成熟度与证据成熟度；`atom_verification_gap` 为空；每个原子都有真实执行记录和独立 oracle，不能只检查 DOM/源码中存在动作名称。

### C9 端到端任务验证

按角色、数据、状态和平台执行冻结的核心任务；验证跨界面和跨模块组合。  
输出：`task-verification.json`。  
门禁：所有 `S_required` 任务与场景均通过，`task_gap` 为空；P0/P1 任一失败立即阻断，失败路径和恢复路径按合同完成。覆盖率或低风险任务通过不得补偿任何 REQUIRED 缺口。

### C10 差集裁决与漏检闭环

计算全部集合、变体闭包与指标，生成最终结论；将独立评审发现与门禁 finding 对账，并执行第 0.7 节规范执行真实性门禁。  
输出：`capability-gate-report.md`、`capability-gate-report.json`、`miss-analysis.yaml`。  
门禁：漏检已归因并形成回归夹具；否则本轮方法执行为 FAIL 或 BLOCKED。

## 13. 缺陷分类

| 代码 | 含义 | 典型证据 |
|---|---|---|
| `MISSING_CAPABILITY` | REQUIRED 能力不存在 | 能力没有任何功能或实现映射 |
| `MISSING_SURFACE` | 需要 UI 载体但不存在 | 核心任务无可到达入口 |
| `SHELL_ONLY` | 只有页面、页签、容器或占位内容 | 可见但无法完成任何有效命令 |
| `NO_COMMAND_WIRING` | 入口可操作但没有真实处理器 | 点击无结果或只有视觉反馈 |
| `NO_STATE_EFFECT` | 命令执行但不改变权威状态/业务结果 | 只有 Toast 或局部临时样式变化 |
| `NO_CONSUMER_EFFECT` | 权威状态已变化但 Renderer/下游不消费 | 模型已更新，运行结果不变 |
| `FAKE_SUCCESS` | 固定文本、静态数字或伪造检查被当成成功 | 假校验、假发布、假权限修复 |
| `PARTIAL_IMPLEMENTATION` | 只完成能力链的一部分 | 能配置但不能反显，能保存但不能执行 |
| `TASK_NOT_CLOSED` | 单功能存在但完整任务无法完成 | 缺依赖功能或跨模块断链 |
| `PLANNED_BUT_ENABLED` | 规划能力仍可操作 | 未实现入口未禁用 |
| `UNDECLARED_UI_CLAIM` | UI/API 声明了范围合同未登记的能力 | 新按钮或菜单无法追溯 |
| `COVERAGE_GAP` | 必验场景未执行 | 角色、状态、设备或边界缺样本 |
| `EVIDENCE_GAP` | 证据不足、陈旧或不可重放 | 只有静态截图或过期报告 |
| `BASELINE_GAP` | 领域或项目能力基线漏项 | 已知必要能力没有进入全集 |
| `VARIANT_COVERAGE_GAP` | 泛化命令隐藏未验证的行为变体 | 只测一个参数却声明全部类型通过 |
| `UNEXPANDED_VARIANT_FAMILY` | 可枚举能力族没有展开为稳定成员集合 | 只记录“106个组件”或分类，不列组件 ID |
| `AGGREGATE_CLAIM_WITHOUT_PROOF` | 族级全称/数量声明没有逐成员或等价类证据 | 一个通用演示宣称全部类型均已设计 |
| `METHOD_NOT_EXECUTED` | 声称执行规范但必需产物、哈希或阶段验证缺失 | 没有 completeness 账本却宣称符合规范 |
| `SHADOW_STATE` | 审计专用状态被伪装成权威状态 | 临时 store 自写自读，目标运行时不消费 |
| `SCOPE_TAMPERING` | 通过覆盖范围合同洗掉承诺或 finding | 缺少可验证的追加式范围变更链 |
| `UNMODELED_CAPABILITY_ATOM` | 领域剖面要求或产品已承诺的原子未进入实现合同 | 表格声明可设计但合同没有列结构操作 |
| `ATOM_VERIFICATION_GAP` | 原子实例已登记但未逐项执行或结果未达到目标 | 只执行增行却宣称整个数据组件通过 |
| `ATOM_INSTANCE_COVERAGE_GAP` | 原子模板未按具体变体/对象/操作/上下文实例化或只验证代表实例 | 106 个组件共用一个新增列模板，只测其中一个组件 |
| `AGGREGATE_EXISTENTIAL_PROOF` | 用任意一个代表动作/成员通过证明全称或完整性声明 | `some(action)` 通过后标记组件完整 |
| `CIRCULAR_EVIDENCE` | 期望、测试和裁决来自同一不完整实现合同 | 从动作数组生成测试再证明动作数组完整 |
| `DESCENDANT_COVERAGE_GAP` | 父对象通过但后代子结构或嵌套层未验证 | 容器通过但列、节点、插槽无法增删排序 |
| `SOURCE_DISCOVERY_GAP` | 来源类别声称无证据但缺少可重放发现记录 | D/B/C/E/R 全填空值后冻结全集 |
| `DIMENSION_ASSESSMENT_GAP` | 剖面维度未逐项判断，空数组混淆未分析与不适用 | structure 为空但没有状态和裁决证据 |
| `INDEPENDENCE_GAP` | 独立评审的身份、输入或上下文隔离不可核验 | 仅填写 reviewer ref 即宣称独立 |
| `COMPLEX_OBJECT_PROFILE_GAP` | 复杂对象未绑定剖面或被错误降为普通对象 | 表格有可变列结构却跳过原子闭包 |
| `SCOPE_CLASSIFICATION_GAP` | 候选核心/高风险项被降级或排除且无合法决策 | 将必要能力标 OPTIONAL/P2 洗出分母 |

### 13.1 严重级别不可降级矩阵

finding 严重级别由能力关键级别、断裂层和用户影响共同取最高值：

- 假成功导致错误发布、越权、安全/合规失守、金额或数据破坏：P0。
- REQUIRED 核心能力缺失、空壳、无权威效果、无消费者或任务不闭环：至少 P1。
- REQUIRED 核心场景、行为变体或证据覆盖缺口：至少 P1。
- REQUIRED 能力原子未建模、未执行、无消费者或被存在量词/循环证据掩盖：至少 P1；若导致错误发布、越权或数据破坏则为 P0。
- REQUIRED 或已启用能力族未展开、族级完成声明无逐成员证据：至少 P1；若掩盖高风险假成功则按 P0。
- OPTIONAL 且不产生错误承诺、不阻断核心任务的局部缺口：可为 P2。

实现者不得自行降低严重级别。降级必须由独立 Verifier 提出事实证据并由 Approver 批准；P0 涉及安全、权限、数据破坏或假发布时不得降为 P2。

## 14. 集合、指标与裁决

### 14.1 必须输出的集合

```text
DECLARED          范围合同和界面声明的全部待审能力
VERIFIED          达到目标实现成熟度和证据成熟度的能力
MISSING_OR_FAKE   缺失、空壳、假交互、假成功或部分实现能力
BLOCKED           仅因外部证据条件无法完成裁决的能力
EXCLUDED          已合法排除或不适用的能力
UNDECLARED_CLAIMS 未登记但已暴露的界面/API能力
```

### 14.2 关键差集

```text
required_gap     = U_required - VERIFIED
claim_gap        = U_claimed - (VERIFIED ∪ 合法禁用的PLANNED)
task_gap         = S_required - S_verified
planning_gap     = S_required - planned_coverage                       # COVERAGE_PLANNING
prototype_gap    = S_required - verified_prototype_coverage            # PROTOTYPE_AUDIT
delivery_gap     = S_required - verified_delivery_coverage             # DELIVERY_AUDIT
overclaim_gap    = claimed_coverage - 当前模式对应的verified_coverage
variant_gap      = V_audit - V_verified - 合法禁用或排除的变体
prototype_variant_gap = V_audit - 具有合格原型案例和当前构建证据的变体
implementation_variant_gap = V_audit - 完成适用实现链证明的变体
aggregate_claim_gap = 声称完成但成员闭包不完整或任一适用成员未通过的能力族
unmodeled_atom_gap = A_audit - A_declared_implementation
undeclared_impl_atom_gap = A_declared_implementation - A_candidate - 合法新增设计项
implementation_atom_gap = A_audit - 完成入口/载荷/状态/消费者/边界映射的原子
atom_verification_gap = A_audit - A_verified - 合法禁用或排除的原子
atom_instance_coverage_gap = 应由模板实例化的原子实例 - 已登记且具有主体/操作/上下文身份的原子实例
circular_evidence_gap = 使用同一不完整合同同时生成期望、测试和裁决的证明链
source_discovery_gap = 必需来源发现步骤 - 具有可重放记录的来源发现步骤
dimension_assessment_gap = 必需剖面维度 - 具有显式状态、事实和裁决的维度
independence_gap = 必需独立评审 - 身份、目标哈希、输入哈希、会话与隔离证据完整的评审
complex_object_profile_gap = 候选复杂对象 - 已合法绑定剖面或经独立证据降级的对象
scope_classification_gap = 候选 REQUIRED/P0/P1 项 - 已保持分类或具有合法降级/排除决策的项
```

### 14.3 诊断指标

```text
capability_coverage = |VERIFIED ∩ U_required| / |U_required|
task_coverage       = |S_verified| / |S_required|
trace_coverage      = 完整追溯链数量 / REQUIRED追溯链数量
evidence_coverage   = 新鲜可重放证据数量 / 必需证据数量
atom_coverage       = |A_verified ∩ A_required| / |A_required|
```

指标只用于诊断和排序，不能覆盖任何 P0/P1 或核心差集。

### 14.4 最终状态

`PASS` 当且仅当：

```text
U_required 非空
required_gap 为空
claim_gap 为空
task_gap 为空
当前模式对应的 planning_gap / prototype_gap / delivery_gap 为空
overclaim_gap 为空
variant_gap / prototype_variant_gap / implementation_variant_gap 为空
aggregate_claim_gap / unexpanded_variant_gap 为空
unmodeled_atom_gap / undeclared_impl_atom_gap / implementation_atom_gap / atom_verification_gap / atom_instance_coverage_gap 为空
circular_evidence_gap 为空
source_discovery_gap / dimension_assessment_gap / independence_gap 为空
complex_object_profile_gap / scope_classification_gap 为空
第 0.7 节规范执行真实性门禁通过
所有核心能力达到目标成熟度
所有核心证据达到目标证据成熟度
P0/P1 为 0
独立复验无未归因漏检
```

`FAIL`：

- 任一 REQUIRED 能力低于目标成熟度。
- 任一核心任务或原型覆盖缺失。
- 当前模式的规划/原型/交付覆盖差集不为空，或存在夸大声明。
- 存在空壳、假交互、假成功或未登记启用入口。
- 存在已确认 P0/P1。
- 已知问题未被本门禁报告发现。
- 任一可枚举能力族未展开、变体差集非空，或使用抽样结果冒充族级完成。
- 任一复杂对象未绑定领域剖面、能力原子未闭包、原子差集非空、实现暴露未登记原子，或用存在量词/循环证据冒充完整性证明。
- 来源发现不可重放、剖面维度未逐项评估、独立评审身份/目标/隔离不可核验、复杂对象被错误降级或范围分类缺少合法裁决。
- 规范必需产物不存在、为空或无法重算，却声称已执行本规范。

`BLOCKED`：

- 没有已确认产品缺陷，但必要外部环境、权限或证据不可获得，导致结论无法形成。
- 不得用 BLOCKED 隐藏已经确认的 FAIL。

内部设计或实现无法识别权威状态、消费者或依赖边界属于 `INTERNAL_UNRESOLVED`，判 FAIL；只有产品链路已有可核验设计和部分证据、但必要外部环境/权限不可控地缺失时，才属于 `EXTERNAL_EVIDENCE_BLOCKED` 并允许 BLOCKED。

`NOT_APPLICABLE`：

- 只用于单项能力或验证维度。
- 必须有适用条件不成立的证据和范围裁决记录。
- 不作为整个复杂产品的最终状态。

## 15. 角色与职责分离

| 角色 | 职责 | 禁止事项 |
|---|---|---|
| Scope Owner | 冻结目标、范围、核心任务和排除项 | 不得用范围缩减掩盖已暴露能力 |
| Capability Analyst | 从多源推导能力和功能 | 不以当前实现作为能力全集唯一来源 |
| Product/Domain Reviewer | 独立推导能力并对账 | 不读取实现者的“已完成功能”结论后照抄 |
| Implementer | 建立界面、状态和消费者 | 不自评最终成熟度 |
| Completeness Verifier | 静态与动态验证，计算差集 | 不修改被审核对象，不临时降低目标成熟度 |
| Adversarial Reviewer | 主动寻找未建模原子、量词偷换、循环证据、后代漏检和最小反例 | 不接受生成器自报 PASS，不以缺少显性需求为停止条件 |
| Approver | 对范围争议、风险接受和最终交付作人类决策 | 不用百分比覆盖核心缺陷 |

在 AI 执行中，能力独立推导者和实现者不得为同一上下文；无法分离时，相关独立性状态为 `BLOCKED`，不得声明独立复验完成。

## 18. 漏检闭环

每轮必须将以下来源中的已知问题与门禁 finding 对账：

- 用户反馈。
- 历史缺陷和回归问题。
- 独立产品评审。
- 独立代码/架构评审。
- 本轮人工观察。

已确认问题未被门禁发现时：

```text
本轮方法执行改判 FAIL
→ 复现并保存漏检证据
→ 归因
→ 补强对应资产
→ 添加修改前漏检、修改后捕获的回归夹具
→ 重跑同一门禁
```

归因类型：

```text
BASELINE_GAP    能力全集或领域基线缺项
DECOMPOSITION_GAP 能力到功能/任务拆解缺项
PROTOTYPE_MAP_GAP 原型覆盖映射错误
TOOL_GAP        检查器能力不足
EXECUTION_GAP   规则存在但未执行
ORACLE_GAP      预期结果或成熟度判定错误
EVIDENCE_GAP    证据不可重放或不足
SPEC_GAP        通用规范缺项
```

领域特有漏项修改领域能力基线；项目范围错误修改项目范围合同；通用机制缺陷修改本文；自动执行缺陷修改 Skill 或脚本。不得把所有漏检都堆进通用规范。

| CE-040 | 领域剖面完全由当前实现动作清单生成，未使用任务、领域、竞品或风险来源 | C1/C2 FAIL；`METHOD_NOT_EXECUTED` 或 `BASELINE_GAP`，不得计算完整率 |
| CE-041 | D/B/C/E/R 五类来源全部填写“无证据”，但没有查询、路径、输入哈希和工具记录 | `SOURCE_DISCOVERY_GAP`，C1/C2 FAIL，不得冻结能力全集 |
| CE-042 | 剖面七个维度使用空数组，未区分未分析和 NOT_APPLICABLE | `DIMENSION_ASSESSMENT_GAP`；每个维度必须有显式状态、事实和裁决 |
| CE-043 | `independent_reviewer_ref` 有值，但评审目标哈希错误、读取了实现结论或没有会话/输入清单 | `INDEPENDENCE_GAP`；独立复验无效，不能 PASS |
| CE-044 | 具有可变子结构和两个命令族的对象被标为普通对象，从而跳过剖面 | `COMPLEX_OBJECT_PROFILE_GAP`；按结构事实进入候选并独立裁决 |
| CE-045 | 核心任务依赖能力被标为 OPTIONAL/P2 且无范围决策，借此从分母移除 | `SCOPE_CLASSIFICATION_GAP`；保持 REQUIRED/P1 或进入 UNRESOLVED，不得洗白 |
| CE-046 | 所有 YAML 能解析但字段为空、引用悬空，且没有规范性 Schema/重算器/认证命令 | `METHOD_NOT_EXECUTED`；机器语义校验与认证夹具未通过 |
| CE-047 | C9 仅执行 P0 任务，遗漏 REQUIRED 的 P1/P2 任务后申请 PASS | `task_gap` 非空，C9 和最终结论 FAIL |
| CE-048 | 106 个变体共享一个能力原子模板，只对其中一个变体实例执行后复制证据 | `ATOM_INSTANCE_COVERAGE_GAP`；按主体/操作/上下文实例化并逐实例验证，整体 FAIL |

方法认证还应包含至少一个简单产品、一个复杂工作台、一个跨模块产品和一个无 Schema 产品，证明通用性而不是只适配某一种设计器。

## 20. AI 执行清单

AI 接到“检查产品是否完整”“还有哪些没实现”“需要多少原型”时，必须按顺序执行：

```text
1. 定位产品目标、领域基线和项目范围合同。
2. 扫描现有界面/API声明，建立 U_claimed。
3. 建立或复核候选能力全集；扫描目录、注册表、Schema、配置和文案，展开所有 `CLAIM_FAMILY/CLAIM_VARIANT`，并为复杂对象选择/建立领域能力剖面，形成独立于实现的能力原子全集；禁止只记录数量、分类或当前动作数组。
4. 对账并冻结 REQUIRED/PLANNED/EXCLUDED/N_A。
5. 建立能力→功能→能力原子→任务→命令/状态→Surface→原型单元→测试→证据链；对族级操作逐变体、逐适用原子传播证明义务并建立追溯链。
6. 从必验场景反推最小充分原型单元集合。
7. 为每项能力选择架构路径和证明剖面。
8. 先静态负向扫描，再逐能力原子动态真实性验证，再执行端到端任务；不能只检查动作字符串或输出哈希。
9. 计算 REQUIRED-VERIFIED、场景缺口、原型缺口、变体缺口、能力原子缺口、循环证据、聚合声明缺口和未登记声明。
10. 由独立领域角色和对抗评审者复验能力全集、能力原子闭包、量词使用、证据独立性和关键结论。
11. 输出集合、数量、findings、证据、限制和下一步，不只报通过率。
12. 执行第 16 章规范性 Schema、引用、集合重算和认证命令，检查标准产物及哈希；缺失或语义空洞时判 `METHOD_NOT_EXECUTED`，禁止声称已执行规范。
13. 若已知问题漏检，先修方法资产并增加回归夹具，再重跑。
```

## 21. Definition of Done

只有同时满足下列条件，才能声明复杂产品在当前冻结范围内“能力完整且实现真实”：

- 三层输入齐全：通用规范、适用领域基线、冻结项目范围合同。
- 候选能力全集经过多源提取和独立对账。
- 所有可枚举能力族均已逐成员展开，来源成员与变体清单完成基数、成员和哈希对账。
- 所有复杂对象均绑定独立于实现的领域能力剖面，适用能力原子递归闭包已展开。
- 所有 REQUIRED 能力均达到目标实现与证据成熟度。
- 所有可操作界面/API声明均可反向追溯，或已合法禁用。
- 所有核心任务和适用失败/恢复路径均有合格原型单元或原型流和新鲜证据。
- 所有 `V_audit` 变体均有适用操作、场景、原型案例、实现、测试和证据，或具有合法禁用/排除记录；族级声明没有以抽样冒充全量。
- 所有 `A_audit` 原子实例均绑定模板、具体主体、操作和上下文，并具有来源、命令/转换、合法载荷、权威效果、消费者、oracle、边界、实现、测试和新鲜证据；不存在未实例化、未建模或未验证原子。
- 不存在存在量词冒充全称证明、父级结果向后代传播或生成器/实现/裁决共享同一不完整合同形成的循环证据。
- 跨多个原型单元的核心任务均有合格原型流和可重放交接证据。
- 原型覆盖差集为空，原型数量报告包含应用、单元、Surface 和场景四种口径。
- P0/P1 为零，且没有用评分或覆盖率掩盖核心缺口。
- UI 质量、安全、性能等下游门禁按适用规范独立通过。
- 独立复验没有发现未进入门禁报告的已确认问题。
- 范围、能力、任务、实现、测试和证据可双向追溯。
- 第 0.7 节规范执行真实性门禁通过，标准产物存在、非空、可重算且报告一致。
- 规范性 Schema、引用校验、集合重算器和 CE 认证夹具由登记命令实际执行并通过。

一句话完成定义：

> 当前版本承诺的每项核心能力，都能通过一个可独立执行的真实任务，从入口一路追溯到权威效果、消费者结果、合格原型单元和可重放证据；任何缺失、空壳、假成功或未覆盖场景都已归零。
