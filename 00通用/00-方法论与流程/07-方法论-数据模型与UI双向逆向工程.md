## 0. AI 读取与执行入口

```text
最小输入：待分析目标 + 分析方向。
  方向 A 至少提供：界面证据或可操作环境，以及已知的产品版本、角色和平台。
  方向 B 至少提供：业务域、目标角色、核心任务和候选模型；只有数据库 Schema 时只能生成待评审草案。
自动发现：证据来源、产品版本、角色/权限、租户、功能开关、业务状态、数据集、平台/视口、语言/时区和已有知识卡片。
自动动作：冻结上下文 → 登记原子观察 → 生成分层候选 → 设计区分性验证 → 输出追溯结果。
送审动作：候选领域模型、UI 投影方案、关系/约束假设、能力矩阵条目。
人工决策：领域语义、聚合与事务边界、产品范围、UI 优先级、是否接受 proposed ADR，以及风险接受。
输出：观察账本 / 候选模型 / UI 投影方案 / 执行期推断记录 / 跨层追溯矩阵 / 未知项与验证队列。
完成判据：执行第 10 章 R0-R7；事实、推断、决策分离；每个关键推断可证伪、可追溯，并有通过、驳回或 evidence_insufficient 状态。
```

**适用范围：**闭源竞品的界面/API 逆向、已有系统的模型/UI 对账、候选领域模型到任务界面的正向设计，以及知识库中的数据模型和 UI 模式卡片建设。

**不适用范围：**仅凭单张截图还原物理数据库、仅凭 Schema 自动决定产品信息架构、绕过授权抓取非公开数据、以本文替代需求范围决策、安全评审、容量测试或生产数据库设计评审。

**规则优先级：**`00-总纲` > `01-方法论` 的证据 Schema 与置信度规则 > 本文核心协议 > 第 11-22 章模式参考 > 项目实例。冲突时必须记录冲突位置、采用规则和理由。本文不另建第二套证据强度或执行状态。

**渐进式读取：**首次只读第 0-10 章。只有当前推断命中具体模式时，才按需读取第 11-22 章；模式章节是候选解释库，不是自动放行规则。

禁止：把 UI 展示字段等同于数据库字段；把页面边界等同于聚合边界；把关系语义等同于物理外键；把未观察到当作不存在；把自产品生成的 UI 当作独立证据；未区分事实、推断和决策；未授权抓包或记录未脱敏凭据与个人信息。

## 1. 目标与问题域

本方法论处理两个不对称方向，二者输入、证据上限和验证方式不同：

```text
方向 A：UI 证据 → 可观察契约 → 候选模型（逆向发现）
  输入：截图/录屏/可操作 UI、HTTP 证据、API 文档及采集上下文
  输出：可观察事实、任务/命令/状态契约、分层候选模型、替代解释和验证队列
  证据上限：UI 直接证明的是当前上下文中的可见行为；API 直接证明的是接口契约。领域边界和物理存储仍需额外证据

方向 B：角色 + 任务 + 状态 + 约束 + 候选模型 → UI 投影（正向设计）
  输入：目标角色、JTBD、任务频率与风险、命令/状态、不变量、权限、平台和候选模型
  输出：信息架构、交互载体、视图/表单模型、状态矩阵、验证场景
  设计约束：数据结构只提供约束和候选内容，不能单独决定页面、导航、字段优先级或控件
```

双向推导的核心约束是**信息永远不对称，且映射跨越多个模型层**。UI 只暴露特定角色、状态和版本下的投影；API 只暴露契约；领域模型描述业务语义和不变量；持久化模型描述存储。目标不是让各层结构相同，而是让语义可追溯、行为可验证、合法的信息损失有说明。

## 2. 理论基础

### 2.1 双向不对称矩阵

| 维度 | UI 视角 | 数据模型视角 |
|---|---|---|
| 表达结构 | 扁平的视图字段，有优先级和分组 | 网状的实体关系，有主键和外键 |
| 语义来源 | 用户语言（"客户姓名"） | 技术语言（`customer_name` / `cname`） |
| 约束可见性 | 只能观察当前角色、状态和命令下的交互约束 | 只能看到输入模型明确表达的约束；DDL 也不包含全部领域、安全和时态约束 |
| 计算派生 | 可见（总价 = 单价 × 数量） | 不一定可见（可能是视图/物化视图） |
| 状态语义 | 可观察部分状态标签和可执行转换 | 结构可能可见，业务含义、合法转换和副作用仍需契约或实测 |
| 历史追溯 | 通常只暴露用户可见历史 | 只有输入包含审计/时态模型时才可见，不能从时间戳推断完整历史 |
| 多对一映射 | 多个页面可能展示同一实体 | 一个实体字段出现在多个上下文 |

### 2.2 模型分层、证据上限与推断距离

本文中的“数据模型”必须标注具体层级，禁止把不同层的对象混为同一事实：

| 层级 | 主要内容 | UI/API 证据能直接证明什么 | 不能直接证明什么 |
|---|---|---|---|
| M0 任务与 UI 状态 | 角色、任务、可见字段、命令、状态反馈 | 当前上下文中看见和执行了什么 | 后端结构、全角色能力、未出现的状态 |
| M1 视图/表单模型 | 页面需要的字段、分组、临时值、校验反馈 | DOM、表单状态、客户端请求前的数据形状 | 领域对象、持久化字段、权威约束 |
| M2 API/消息契约 | 请求、响应、错误码、事件载荷 | 实际流量或正式 Schema 中的契约 | 服务内部聚合、表、索引、事务实现 |
| M3 领域概念模型 | 业务对象、值对象、命令、规则、不变量、聚合 | 需由行为、契约、领域文档或源码共同支持 | 具体表结构和 ORM 映射 |
| M4 逻辑持久化模型 | 实体、关系、键、基数、生命周期、时态 | 需由 Schema、迁移、查询或源码支持 | 数据库引擎中的物理布局与性能策略 |
| M5 物理实现 | 表、列、索引、分区、缓存、队列、锁 | 需由 DDL、执行计划、配置、源码或运行证据支持 | 不能由 UI 外观或单次时延直接推出 |

**证据上限规则：**一条证据默认只直接证明其所在层。跨一层为候选推断；跨两层及以上必须列出中间模型、至少两个替代解释和区分性验证。没有额外证据时，M0/M1 观察不得升级为 M3-M5 的“直接事实”。

**四个正交标注：**

```text
observation_strength：原始观察是否完整、可复现
target_layer：推断要落到 M0-M5 哪一层
inference_distance：证据层到目标层跨越几层
validation_status：hypothesis / supported / refuted / evidence_insufficient / verified
```

证据强度与数值置信度统一引用《01-方法论》第 2.2 节：单一来源推断不超过 0.5，两个独立来源可升至约 0.7，三个及以上独立来源或实测验证后方可达到 0.9 及以上。领域常识只能生成候选，不能提高来源独立性。

### 2.3 关键概念区分

**展示字段 ≠ 存储字段：**
```text
UI 上看到的"客户姓名"可能是：
- 视图模型字段 customerName（M1）
- API 返回字段 full_name（M2）
- 领域对象的展示方法（M3）
- 存储字段 customer.name（M4/M5）
- first_name + last_name、多语言投影或搜索索引中的组合值（M4/M5）

仅凭 UI 文案只能确认“当前页面显示客户姓名”这一 M0 事实，后续每种映射都是候选。
```

**关联可见性三层次：**
```text
第一层（直接可见）：界面上明明白白展示关联对象
  例：订单列表有"客户名称"列 → 确认当前视图包含客户语义
       点击客户名称可跳转客户详情 → 确认存在可导航的客户资源标识
第二层（隐含可见）：界面上不显示关联对象，但操作行为暴露了关联
  例：选择省份后城市下拉自动更新 → 确认城市候选受省份选择约束
       把文件拖入文件夹 → 确认存在“归入分组”的命令
       选择客户后地址自动回填 → 确认客户选择会影响地址视图
第三层（接口可见）：UI 上没有线索，但实际 API 契约暴露关系候选
  例：列表按"客户订单数"排序，但界面上看不到客户列
       → 抓包发现 sort=customer_order_count，只能确认 API 支持该排序键
  例：API 响应里有字段在界面上完全看不到
       → 可能是资源标识、冗余字段、服务端投影或关联展开，需继续验证
```

#### 2.3.1 聚合根：不变量与一致性边界

**聚合根的定义：**

聚合根（Aggregate Root）是 DDD 中负责维护聚合不变量、控制内部对象变更的入口实体。聚合通常定义强一致性边界，但具体事务实现仍受架构和存储能力约束，不能从页面结构直接确定。

页面表示用户任务边界，聚合表示业务一致性边界。二者可以重合，也可以一页跨多个聚合、一个聚合分布在多个页面。独立 CRUD 页面只是聚合候选的弱线索，不是判定条件。

**判断聚合边界的四问法：**

```
问题 1：哪些业务不变量必须由同一个入口维护？
问题 2：哪些变更必须原子成功或失败，才能避免业务上无效的中间状态？
问题 3：内部对象能否脱离候选根独立存在、被外部直接引用和独立演进？
问题 4：并发冲突和版本检查应覆盖哪些对象，才能守住上述不变量？

只有领域规则、命令契约、事务/并发行为或源码能回答这些问题。按钮、路由、级联删除和内嵌列表只能生成待验证假设。
```

**判断示例：**

```
订单行项（OrderItem）：
  候选不变量：行项小计之和必须与订单总额一致
  原子性：修改数量和重算订单总额必须共同生效
  生命周期：行项不能脱离订单存在
  并发：订单版本覆盖行项变更
  → 若以上均由 Order 维护，可确认 OrderItem 是 Order 聚合内部对象

订单（Order）：
  维护订单金额、行项、状态转换等不变量
  对外命令通过 Order 标识执行，内部行项不被独立修改
  事务和版本边界覆盖订单及其内部对象
  → 在有上述证据时，Order 才可确认为聚合根
```

**聚合根为什么重要：**

```
原因 1：事务边界
  聚合根原子维护聚合内不变量；跨聚合一致性需求必须显式协调

原因 2：页面划分受约束但不受限
  聚合边界决定了什么不能跨页面操作
  但具体页面划分由角色/频率/风险/设备/信息密度共同决定

原因 3：跨聚合协调需要显式设计
  根据不变量、失败语义和基础设施选择策略（不是简单”必须分开”）
```

#### 2.3.1.1 事务边界深入拆解

**聚合内原子性：**

```
订单聚合包含：
  Order（订单主体：订单号、金额、客户、状态）
  OrderItem[]（订单行项：商品、单价、数量、小计）

事务内的原子操作：
  1. 插入 Order 记录
  2. 插入 OrderItem[] 记录
  3. 重算并更新 Order 聚合内的订单总额

→ 如果行项插入失败，Order 记录也回滚，不存在孤立的订单
→ 订单行项不能脱离订单单独存在（无法单独创建/修改）
```

**跨聚合为什么默认避免共享事务：**

```
Customer 聚合内部：
  Customer 主体（name, phone, email）
  CustomerContact[]（联系人列表）
  CustomerAddress[]（地址列表）

Order 聚合内部：
  Order 主体（order_number, amount, status）
  OrderItem[]（行项）

如果把 Customer 和 Order 放在同一个事务里：
  修改客户邮箱 + 创建订单 → 要么同时成功，要么同时失败

问题：
  - 客户和订单是不同的业务领域，为什么要绑在一起？
  - 订单创建失败时，客户的邮箱修改也要回滚？
  - 订单创建成功，但客户邮箱修改失败，订单也要回滚？
```

**一致性边界对 UI 提交语义的约束：**

```
规则：页面边界不等于事务边界，单个用户命令也可以协调多个聚合
  - 落在同一聚合内 → 聚合根应原子维护自己的不变量
  - 跨越多个聚合 → 先定义全成功、部分成功或最终一致的业务语义，再选择本地事务、应用协调、Saga/流程管理器或事件
  - 同库技术上可使用同一事务，不代表领域上应扩大一致性边界；跨服务通常不能依赖单个本地事务
  - UI 可以一次提交，但必须明确处理中、部分失败、重试、补偿和最终结果，不能由聚合数量机械决定拆页

订单表单（Order 聚合）：
  订单主体（订单号、金额、状态、客户）→ Order 表
  订单行项（商品、数量、单价）→ OrderItem 表
  → 都在 Order 聚合内 → 一次性提交，事务保证原子性

订单创建时同时修改客户信息：
  订单创建 → Order 聚合（事务A）
  客户订单数 +1 → Customer 聚合（事务B）
  → 跨聚合 → 候选为独立事务加协调/对账；若确有同库原子业务要求，也需显式记录扩大事务范围的代价
```

#### 2.3.1.2 UI 页面划分受约束但不受限

**聚合边界是约束，不是模板：**

```
聚合边界规定的”不能”：
  如果订单提交时要求“至少一个合法行项”恒真，则最终提交必须由 Order 命令原子校验
  → UI 可单页、向导或多工作区组织，也可保存明确标识的草稿；不能让中间草稿冒充有效订单

不受限的地方：
  同一聚合可以用多个页面、多种方式组织
  页面划分由以下因素共同决定
```

**决定页面划分的六个因素：**

```
因素 1：用户角色
  销售录入订单：只需填基本信息 + 选择商品
  财务复核订单：需要看金额明细 + 审批按钮
  仓库发货：需要看商品明细 + 发货操作
  → 同一 Order 聚合，三个角色看到不同的页面

因素 2：使用频率
  高频识别、比较与切换任务 → 候选独立列表/详情或工作台
  低频、短任务 → 候选折叠区、弹层或上下文操作；用任务数据和可用性测试裁决

因素 3：错误恢复成本
  金额/库存类操作 → 高风险 → 需要确认步骤
  标签修改 → 低风险 → 可以直接保存

因素 4：设备
  桌面端 → 宽屏 → 可以左右分栏（主信息 + 操作历史）
  移动端 → 空间、触控和网络约束更强 → 候选单列或渐进披露；平板/横屏等场景可采用多栏

因素 5：信息密度
  字段少但语义复杂时仍需分组；字段多但主要只读、可扫描时也未必需要多 Tab
  以任务相关性、标签长度、错误风险、设备和用户测试决定平铺、分组、折叠或分步，不使用固定字段数门槛

因素 6：操作频率
  高频操作（状态变更）→ 放在页面顶部，常驻可见
  低频操作（导出/归档）→ 放在”更多”菜单里
```

**同一聚合的四种页面划分方式：**

```
Order 聚合（包含 Order 主体 + OrderItem[]）

方式 A：单页完整表单
  一个页面包含订单主体 + 商品明细 + 客户信息
  适用：复杂但完整的订单创建流程

方式 B：向导式（分步骤）
  Step 1：填写订单主体
  Step 2：添加商品明细
  Step 3：选择关联客户
  Step 4：确认并提交
  注意：向导每一步都在操作 Order 聚合，只有最后一步才触发事务
  中间步骤的数据暂存在前端状态里

方式 C：主表单 + 侧边抽屉
  主表单：订单主体（主要信息）
  侧边抽屉：商品明细列表（专注编辑行项）
  适用：行项数量多，需要专注编辑

方式 D：详情页 + Tab
  Tab 1：订单主体（可编辑）
  Tab 2：商品明细（行项列表）
  Tab 3：操作日志
  适用：信息量大，需要按维度拆分

以上四种方式都遵守 Order 聚合的事务边界
但页面划分完全不同，是其他因素决定的
```

#### 2.3.1.3 跨聚合协调策略

跨聚合操作不是简单“分开提交”就完了。业务需求常要求多个聚合协同；应根据资源是否同库、业务原子性、可补偿性、延迟和失败语义显式设计协调策略。不要把“聚合边界”误写成“数据库绝不允许同一事务”，也不要为普通同库操作自动引入分布式流程。

**策略 1：完全独立（最简单）**

```
场景：两个操作没有依赖关系

操作：
  修改客户邮箱
  创建新订单

→ 各自独立事务，互不影响
→ 客户改了，订单也创建了，各自成功各自失败，不互相影响

适用条件：
  - 两个操作没有业务依赖
  - 失败后不需要补偿
  - 用户可以接受部分成功
```

**策略 2：应用层顺序控制**

```
场景：操作有先后顺序，但失败后不需要回滚

操作序列：
  1. 创建订单（Order 聚合）
  2. 更新客户订单数（Customer 聚合）

代码逻辑：
  try {
    order = orderService.create(orderData);            // 事务A
    customerService.incrementOrderCount(customerId);   // 事务B
  } catch (Exception e) {
    // 不回滚订单，已创建的订单保留
    // 记录不一致日志，后续人工处理或定时任务修复
  }

→ 适用：允许最终一致，业务上可以接受轻微不一致
→ 风险：客户订单数可能比实际少一条（需要定时对账）
```

**策略 3：Saga 编排（补偿模式）**

```
核心原则：补偿操作是撤销已经成功完成的步骤
  不是处理”当前步骤的失败”，而是撤销”已经成功的步骤”

正向执行示例：
  Step 1（创建待确认订单）→ Step 2（预留库存）→ Step 3（取得支付授权）

失败触发补偿链（从当前失败步骤往前补偿）：
  Step 3 失败 → 补偿 Step 2（释放库存预留）+ 补偿 Step 1（取消订单）
  Step 2 失败 → 补偿 Step 1（取消订单）
  Step 1 失败 → 无需补偿（什么都没发生）

场景示例：用户下单

Step 1：创建订单（Order 聚合）
  正向：创建 Order，状态=处理中
  补偿：如果后续必要步骤失败，将订单标为已取消并记录原因；不删除审计事实

Step 2：预留库存（Inventory 聚合）
  正向：以幂等命令预留库存并返回 reservation_id
  补偿：以 reservation_id 幂等释放预留，避免简单“加回数量”造成重复补偿

Step 3：取得支付授权（Payment 边界）
  正向：以幂等键取得授权并返回 authorization_id
  补偿：按支付渠道语义撤销授权；若不可立即撤销则进入人工/对账状态

事务完成后发送通知：通知不是该示例的成功不变量，失败只进入重试/死信/人工告警，不触发订单补偿。

执行路径：
  路径 A（全部成功）：
    Step 1 ✓ → Step 2 ✓ → Step 3 ✓ → 完成

  路径 B（支付授权 Step 3 失败）：
    Step 1 ✓ → Step 2 ✓ → Step 3 ✗
    → 补偿 Step 2：释放库存预留
    → 补偿 Step 1：取消订单
    → 订单创建失败，用户看到”下单失败，请重试”

  路径 C（Step 2 失败）：
    Step 1 ✓ → Step 2 ✗
    → 补偿 Step 1：取消订单
    → 订单创建失败，库存未变化

  路径 D（Step 1 失败）：
    Step 1 ✗
    → 无需补偿
    → 订单创建失败

→ 适用：跨资源长事务、步骤可补偿且业务接受补偿后的最终一致；Saga 不提供 ACID 强一致
→ 风险：补偿不等于数据库回滚，可能失败、超时或只能近似撤销；每步需幂等、可重试、可观测，并有人工恢复与对账
```

**策略 4：事件驱动（最终一致）**

```
场景：异步协调，不需要实时同步

操作流程：
  1. 用户提交订单
     → Order 聚合事务保存
     → 发布事件：OrderCreatedEvent

  2. 消息队列接收 OrderCreatedEvent

  3. 库存服务消费事件
     → 扣减库存（独立事务）

  4. 通知服务消费事件
     → 发送通知（独立事务）

→ 用户看到：订单创建成功（立即）
→ 后台处理：库存扣减、通知发送（异步，可能延迟几秒）

→ 适用：不需要实时一致，接受最终一致
→ 优势：性能好，不阻塞用户
→ 劣势：用户看到成功，但后台处理可能失败（需要额外的监控和告警）
```

**正交模式：事件溯源（Event Sourcing，不是跨聚合协调策略）**

```
场景：需要完整的操作历史，支持任意时间点回溯

传统方式：记录最终状态
  订单状态=已发货
  → 丢失了”从待付款到已发货之间发生了什么”

事件溯源：记录完整事件序列
  OrderCreated → OrderPaid → OrderShipped
  → 可以重放任意事件，还原任意时间点的状态
  → 更正或补偿通过追加新的业务事件表达，历史事件不被擦除；新事件不一定是简单“逆向”

→ 适用：事件序列本身是权威事实、审计/时态重建价值足以覆盖复杂度的业务
→ 不因“需要日志”就采用；普通审计表/事件日志通常更简单
→ 代价：投影、版本演进、重放、副作用隔离、删除合规和运维复杂度显著增加
```

**策略选择决策树：**

```
第一步：两个操作有业务依赖吗？
  无依赖 → 策略 1（完全独立）

第二步：失败后需要回滚吗？
  不需要回滚 → 策略 2（应用层顺序控制）

第三步：是否要求单个 ACID 事务内全成功或全失败？
  是 → 优先检查能否在同一权威边界用本地事务完成；不能时应重新审视边界/需求，Saga 不能伪装成强一致
  否，允许延迟或补偿 → 下一步

第四步：步骤之间有明确的先后顺序吗？
  是 → 策略 3（Saga 编排）
  否 → 策略 3 或 策略 4

第五步：怎样验证协调长期可靠？
  为所有策略补幂等键、状态持久化、超时/重试、补偿失败、对账、告警和人工恢复

另行判断：事件序列是否必须成为 Source of Truth？
  只有答案为“是”且收益覆盖演进/运维成本时，才评估事件溯源；它不能替代上述协调策略
```

**聚合根 → UI 页面映射：**

```
聚合根 → 候选命令/保存边界，不自动等于完整管理页面
  Customer 可能由列表、详情、侧栏和批量任务共同操作
  Order 可能按下单、履约、售后任务拆成多个页面

聚合内部成员 → 通常通过根的命令修改，可在内嵌区、向导或独立任务页呈现
  OrderItem（订单行项）可在下单工作台中编辑，但提交仍通过 Order 命令
    内嵌列表上方有"添加商品"按钮
    删除行项在列表内操作，不单独管理

跨聚合引用 → UI 可选跳转、选择器、摘要或组合工作台
  Order 引用 Customer → 订单详情显示"客户：张三"，可点击跳转客户详情
  Order 的表单中选客户 → 搜索选择器选择已有客户或快速新建
```

#### 2.3.1.4 聚合根之间的组织

聚合根不是平铺在系统中的孤岛，而是按三层嵌套结构组织：**限界上下文（最大隔离单元）→ 模块（业务概念分组）→ 聚合（事务边界）**。

##### 限界上下文（Bounded Context）

限界上下文是系统中最大的组织单元。每个上下文有独立的业务模型、独立团队负责、独立存储。同一术语在不同上下文含义可能不同，不能混用。

**为什么需要限界上下文——没有时的混乱：**

```
没有限界上下文时，"客户"在整个系统里含义混杂：
  销售眼里：客户 = 潜在购买者（有销售阶段、成交概率）
  客服眼里：客户 = 需要服务的人（有投诉记录、满意度）
  财务眼里：客户 = 应收账款的债务人（有信用额度、账期）
  运营眼里：客户 = 活跃度衡量对象（有生命周期阶段）

如果都放在同一个模型里：
  Customer {
    name, phone, email
    salesStage, probability, assignedTo    // 销售视角
    creditLimit, accountBalance            // 财务视角
    complaintCount, satisfactionScore     // 客服视角
    lifecycleStage, totalSpent            // 运营视角
  }

→ 销售团队改了 salesStage，财务团队不知道
→ 一个团队加了字段，所有团队都受影响
→ 改动相互污染，大泥球（Big Ball of Mud）
```

**限界上下文识别标准：**

```
标准 1：术语歧义
  同一词（如"订单"）在不同团队业务里含义不同
  → 需要不同的限界上下文

标准 2：变更频率
  两个实体变更频率差异极大
  → 产品类目（很少变）和价格（频繁变）→ 不同上下文

标准 3：团队边界
  两个功能由不同团队独立维护
  → 不同上下文（康威定律）

标准 4：数据所有权
  两个实体归属于不同部门，数据隔离要求严格
  → 不同上下文
```

**限界上下文的边界类型：**

```
类型 1：物理隔离（强烈推荐）
  每个上下文有自己的数据库
  上下文之间只能通过 API 或事件通信

类型 2：逻辑隔离（次选）
  共享同一个数据库，但通过 schema 或租户字段隔离
  上下文之间仍然通过 API 通信

类型 3：共享表（不推荐）
  多个上下文共享同一套表
  → 违反限界上下文初衷，最后变成大泥球
```

**限界上下文之间的映射方式：**

```
方式 1：客户-供应商（Customer-Supplier）
  上游提供数据，下游消费数据
  例：销售上下文（上游）发布产品目录
      订单上下文（下游）订阅并使用产品信息

方式 2：防腐层（Anti-Corruption Layer, ACL）
  下游不直接用上游数据
  而是写一个"翻译层"把上游数据转成自己上下文能理解的形式
  例：销售上下文发来"productId=123, price=99"
      ACL 翻译成 OrderItem{ productId, price, snapshotPrice }

方式 3：开放主机服务（Open Host Service）
  上游提供一个稳定的 API 协议
  下游按协议调用
  例：GET /products/{id}，GET /customers/{id}

方式 4：共享内核（Shared Kernel）
  两个上下文共享一小部分模型
  → 强耦合，只在收益明显时使用
  → 例：日期格式、货币单位等通用概念
```

##### 模块（Module）

在同一限界上下文内，按业务概念分组。模块名应使用业务术语而非技术术语。

```
✗ 技术术语命名（错误）：
  module_order_service
  module_order_repository

✓ 业务术语命名（正确）：
  订单管理（Order Management）
  退货处理（Return Handling）
  发票开具（Invoice Issuance）

模块内的组织原则：
  模块 = 所有与该业务全生命周期相关的概念集合

module: 订单管理
  包含：
    Order（聚合根）
    OrderItem（聚合内部对象）
    OrderStatus（状态枚举）
    OrderCreated（领域事件）
    OrderCancelled（领域事件）
  不包含：
    Payment（属于"支付"模块）
    Refund（属于"退款"模块）
    Inventory（属于"库存"模块）

module ≠ Java package
  package 通常按技术分层：controller / service / repository
  模块按业务概念分组，与技术分层是正交的
  两者可以结合：在业务模块内部按技术分层组织代码
```

##### 聚合（Aggregate）

在同一个模块内，按事务边界分组。聚合应尽量小；大聚合导致事务范围过大、并发冲突严重。

```
✗ 大聚合（错误）：
  OrderAggregate {
    Order（聚合根）
    OrderItem[]
    Payment（另一个聚合根，有完整支付流程）
    Refund（另一个聚合根）
    Customer（另一个聚合根）
    Product[]
  }
  → 任何对象冲突都导致整个事务失败

✓ 小聚合（正确）：
  Order（聚合根）
  Payment（聚合根）
  Refund（聚合根）
  Customer（聚合根）
  Product（聚合根）
  → 各聚合边界清晰，各自独立演进
```

**聚合边界判断：**

```
问题 1：两个对象必须同时创建吗？
  订单 + 订单行项 → 必须同时创建 → 同一聚合
  订单 + 客户 → 不一定 → 不同聚合，用 customerId 关联

问题 2：修改 A 会导致 B 必须一起验证吗？
  订单行项修改数量 → 总金额变化 → 必须验证金额不超限
  → 同一聚合，聚合根统一维护不变量

  订单 + 产品名称 → 修改产品名不影响订单
  → 不同聚合，订单持有 productId 而非直接引用

问题 3：它们的一致性要求是什么？
  订单行项是否需要和产品的当前价格实时一致？
  → 是 → 验证逻辑应在订单聚合内
  → 否，订单历史价格独立 → 订单行项直接存储快照价格

问题 4：并发修改的风险
  A 和 B 经常被不同用户同时修改 → 应在不同聚合（减少并发冲突）
  订单行项被同一用户在编辑订单时修改 → 同一聚合，锁定聚合根即锁定所有行项
```

##### 聚合根之间的引用规则

**核心规则：聚合根只能通过 ID 引用其他聚合根，不能直接引用对象。**

```
✓ ID 引用（正确）：
  Order {
    Long customerId;          // 用 ID，不是对象引用
    List<OrderItem> items;
  }

✗ 对象引用（错误）：
  Order {
    Customer customer;        // 直接引用，耦合事务边界
    List<OrderItem> items {
      Product product;        // 又一个跨聚合引用
    }
  }

对象引用的后果：
  1. 加载 Order 时必须同时加载 Customer
  2. 修改 Customer 时可能意外影响 Order 的事务
  3. 聚合的事务边界完全失效
  4. Customer 和 Order 形成循环依赖
```

##### 聚合根之间的协作方式

**同限界上下文内的协作（三种方式）：**

```
方式 1：直接调用（同步，强耦合）
  Order.return() → 创建 Return 聚合
  适用：强依赖，Order 状态变化直接触发 Return

方式 2：领域事件（异步，解耦）
  Order 内部发布事件：OrderReturnedEvent
  Return 聚合订阅事件，收到后创建 Return
  适用：弱依赖，不需要 Order 等待 Return 完成

方式 3：应用服务协调
  OrderService 创建 Order，ReturnService 创建 Return
  应用层控制两者之间的协作逻辑
  适用：跨聚合的复杂业务流程，需要显式编排
```

**跨限界上下文的协作（只有领域事件一种方式）：**

```
订单上下文（Order Context）
  发布 OrderCreatedEvent
  {
    orderId: "ORD001",
    customerId: "CUST001",
    amount: 300,
    timestamp: "2026-07-12 10:00"
  }

客户上下文（Customer Context）
  订阅 OrderCreatedEvent
  → 收到后更新客户订单数
  → 触发会员等级重新计算

库存上下文（Inventory Context）
  订阅 OrderCreatedEvent
  → 预扣库存

通知上下文（Notification Context）
  订阅 OrderCreatedEvent
  → 发送欢迎短信

→ 订单上下文不知道其他上下文的存在
→ 任何上下文都可以订阅事件，各自独立演进
```

##### 领域事件的完整生命周期

```
Step 1：事件生成（在聚合根内部）
  Order.confirm():
    this.status = OrderStatus.CONFIRMED
    this.confirmedAt = LocalDateTime.now()
    this.registerEvent(new OrderConfirmedEvent(this.id, this.customerId))

Step 2：事件持久化（聚合根保存时）
  框架自动将聚合根内部的事件列表取出，存入事件存储（EventStore）
  事件存储表：domain_event { event_id, event_type, aggregate_id, payload, created_at }

Step 3：事件发布（事件总线）
  事件发布器从事件存储读取事件，发布到消息队列（Kafka / RabbitMQ）

Step 4：事件消费（其他聚合根/上下文）
  客户上下文订阅 order.confirmed → 更新订单数、计算会员等级
  库存上下文订阅 order.confirmed → 预扣库存
  通知上下文订阅 order.confirmed → 发送通知

Step 5：事件回放（可选，仅事件溯源场景）
  新建 BI 系统，从头回放所有历史事件，重建完整业务视图
```

##### 聚合根自身方法 vs 应用服务方法

```
聚合根自身维护的方法（领域逻辑）：
  order.confirm()        // 订单状态转换的验证逻辑
  order.addItem()        // 添加行项并重新计算总价
  order.removeItem()     // 移除行项并重新计算总价
  order.applyDiscount()  // 应用折扣并验证不变量

应用服务的方法（协调逻辑）：
  orderService.createOrder()   // 协调 Customer、Inventory、Order
  orderService.cancelOrder()   // 协调 Order、Inventory、Notification
  orderService.refundOrder()   // 协调 Order、Payment、Refund

区分原则：
  聚合根方法 = 维护聚合内部不变量，任何外部调用者都可以用
  应用服务方法 = 协调多个聚合根，属于应用层，不属于领域层
```

##### 完整业务流程示例：用户下单到收到发货通知

```
涉及的限界上下文：
  销售上下文（Sales）→ Product（商品）
  订单上下文（Order）→ Order（订单）、Refund（退款）
  库存上下文（Inventory）→ Stock（库存）
  物流上下文（Logistics）→ Shipment（发货单）
  通知上下文（Notification）→ Notification（通知）
  客户上下文（Customer）→ Customer（客户）、Subscription（订阅）

事件流（按时间顺序）：

T0: 用户提交订单
  Order.create(customerId, items[])
    → 发布 OrderCreatedEvent

T1: 客户上下文消费 OrderCreatedEvent
  Customer.onOrderCreated()
    → 更新客户订单计数
    → 重新计算会员等级

T2: 库存上下文消费 OrderCreatedEvent
  Stock.reserve()
    → 预扣商品库存（状态=RESERVED）
    → 发布 StockReservedEvent

T3: 库存上下文消费 StockReservedEvent
  Order.confirm()
    → 订单状态变为 CONFIRMED

T4: 客户上下文消费 OrderConfirmedEvent
  Subscription.notify()
    → 检查订阅偏好，发布 EmailNotificationRequestedEvent

T5: 通知上下文消费 EmailNotificationRequestedEvent
  Notification.sendEmail()
    → 查找模板、渲染内容、发送邮件

T6: 商家发货（人工操作）
  Shipment.create(orderId, trackingNumber)
    → 发布 ShipmentCreatedEvent

T7: 订单上下文消费 ShipmentCreatedEvent
  Order.ship(trackingNumber)
    → 订单状态变为 SHIPPED

T8: 通知上下文消费 OrderShippedEvent
  Notification.sendSms()
    → 发送发货通知短信

T9: 物流上下文消费 OrderShippedEvent
  Shipment.update()
    → 订阅物流轨迹，推送状态给客户

→ 全流程涉及 6 个限界上下文、多个聚合根，全部通过事件驱动解耦
→ 没有直接的跨上下文方法调用
```

##### 餐饮点餐系统完整示例

以餐饮点餐系统为例，展示从业务分析到聚合设计的全过程。

**Step 1：识别限界上下文：**

```
上下文 1：门店管理（Store Management）
  管理餐厅门店信息、桌位、营业时间

上下文 2：菜品管理（Dish Management）
  管理菜品、分类、规格、做法

上下文 3：订单交易（Order Transaction）
  管理点餐、支付、优惠

上下文 4：厨房调度（Kitchen Dispatch）
  管理菜品制作、叫号、打印

上下文 5：会员运营（Membership）
  管理会员、积分、储值
```

**Step 2：识别各上下文内的聚合根：**

```
门店管理上下文：
  Store（聚合根）：门店信息、营业时间、地址
  Table（聚合根）：桌位信息、容纳人数、状态
  Area（聚合根）：区域分组（如"一楼"、"包厢"）

菜品管理上下文：
  Category（聚合根）：菜品分类
  Dish（聚合根）：菜品信息、做法、规格
  DishSpec（聚合根）：菜品规格（小份/中份/大份）
  DishMod（聚合根）：做法修改（不要香菜、加辣）

订单交易上下文：
  Order（聚合根）：订单主体、状态、金额、桌位ID
  OrderItem（聚合内部对象，不是聚合根）：菜品、规格、做法、数量
  Payment（聚合根）：支付记录、方式、金额
  Coupon（聚合根）：优惠券、优惠规则

厨房调度上下文：
  Ticket（聚合根）：厨打小票、菜品列表、状态、挂起原因

会员运营上下文：
  Member（聚合根）：会员信息、手机号、积分
  Account（聚合根）：储值账户、余额
```

**Step 3：识别聚合根之间的引用关系：**

```
Order（聚合根）：
  storeId → Store
  tableId → Table
  memberId → Member（可选）
  couponId → Coupon（可选）
  注意：Order 不直接引用 Dish，而是通过 OrderItem（聚合内部对象）间接关联

OrderItem（聚合内部对象，不是聚合根）：
  dishId → Dish
  specId → DishSpec（可选）
  modIds → DishMod[]（多选）
  订单行项是订单聚合的内部成员，不能独立存在

Ticket（聚合根）：
  orderId → Order
  厨打小票和订单是不同上下文，通过 orderId 关联
```

**Step 4：设计聚合根之间的协作：**

```
Order.confirm() → 发布 OrderConfirmedEvent
  客户上下文消费 → 扣减积分/储值
  门店上下文消费 → 更新桌位状态为"用餐中"
  厨房上下文消费 → 创建 Ticket

Ticket.complete() → 发布 TicketCompletedEvent
  订单上下文消费 → 更新订单状态

Payment.receive() → 发布 PaymentReceivedEvent
  订单上下文消费 → 订单状态变为已支付
  会员上下文消费 → 增加积分
```

##### 总结：三层组织结构

```
限界上下文（业务领域隔离）
  ↓
模块（业务概念分组）
  ↓
聚合（事务边界）
  ↓
聚合根（唯一入口）
  ↓
聚合内部对象（不能独立存在）

聚合根之间的组织规则：
  同上下文：直接调用 / 领域事件 / 应用服务协调
  跨上下文：只能通过领域事件解耦
  引用方式：只能用 ID，不能直接引用对象
  聚合边界：尽量小，避免大聚合导致事务范围过大

聚合根 → UI 映射：
  限界上下文 → 侧边栏一级导航
  模块 → 侧边栏二级导航或 Tab
  聚合根 → 页面（列表、详情、新建）
  聚合内部对象 → 内嵌区块
  外键引用 → 跳转链接或选择器
```



### 2.4 关系语义的 UI 展现方式图谱

UI 展示的是业务关系和任务上下文，不直接展示物理外键。以下模式用于生成关系候选，并帮助从已确认关系设计 UI；是否使用外键、中间表、文档嵌入、事件投影或远程引用，必须在 M4/M5 另行验证。

#### 2.4.1 十一种关系 UI 展现方式

| # | 展现方式 | 可观察语义 | 关系候选 | 可编辑 | 典型场景 |
|---|---|---|---|---|---|
| 1 | 下拉选择 | 当前任务要求单值选择 | 枚举/单值引用/查询条件 | 是 | 候选集较小且可扫描 |
| 2 | 搜索选择器 | 当前任务引用可检索对象 | 远程资源/本地实体/搜索投影 | 是 | 候选集较大或需要模糊搜索 |
| 3 | 标签展示 | 多值分类或关系 | 枚举集合/多对多/数组/投影 | 是 | 多选标签 |
| 4 | 跳转链接 | 存在可导航资源标识 | 资源引用/路由参数/外部链接 | 否（跳转操作） | 查看关联对象 |
| 5 | 内嵌列表 | 当前任务需要上下文集合 | 子对象/独立聚合集合/查询投影 | 是 | 上下文管理 |
| 6 | 拖拽到分组/文件夹 | 存在归类或移动命令 | 单值分类/成员关系/路径变更 | 是 | 树形分类 |
| 7 | 头像/图标+名称 | 展示具名对象摘要 | 对象引用/快照/投影 | 部分可改 | 人员、部门 |
| 8 | 颜色标签 | 分类或状态的视觉编码 | 枚举/值对象/配置引用 | 是 | 状态、优先级、类型 |
| 9 | 面包屑路径 | 可导航层级 | 自引用/路径/路由层级 | 否 | 层级导航 |
| 10 | 纯文本展示 | 当前上下文只读值 | 对象引用/历史快照/冗余文本 | 否 | 创建人等 |
| 11 | 弹窗/抽屉预览 | 任意关系 | 任意 | 否 | 快速预览 |

#### 2.4.2 各方式详解

**方式 1：下拉选择**

```
┌────────────────────┐
│ 客户 * [选择客户 ▾] │
└────────────────────┘
```
常用于单值引用或固定枚举。选择器内显示名称和关键信息；是否为 ManyToOne 或物理外键，需从 API Schema、关系基数和持久化证据确认。

**方式 2：搜索选择器**

```
┌────────────────────┬──────────┐
│ 🔍 搜索客户...      │ [+新建] │
└────────────────────┴──────────┘
  └─ 张三 (138xxxx)
  └─ 李四 (139xxxx)
```
候选集无法快速扫描、需要模糊搜索或数据不能一次加载时，可用搜索+列表选择。具体切换阈值由任务频率、标签长度、设备和用户测试决定，不把固定数量作为通用门禁。旁侧"+"按钮可支持快速新建（见 4.1 规则 3）。

**方式 3：标签展示**

```
客户标签：  [VIP ×] [重点客户 ×] [+ 添加]
```
这是多值分类或关系的常见 UI。点击"×"移除当前上下文中的值，点击"添加"增加值；底层可以是中间表、数组、配置值或服务端投影。

**方式 4：跳转链接**

```
客户：张三  ← 点击跳转到客户详情页
```
这是资源引用的只读展示方式。跳转只证明存在可导航标识，不证明关系基数、聚合归属或物理外键。

**方式 5：内嵌列表**

```
┌─ 客户订单 ────────────────────────┐
│ [新建订单]                         │
│ 订单号    | 金额    | 状态         │
│ ORD001   | ¥300    | 已完成        │
└───────────────────────────────────┘
```
内嵌列表表示当前任务需要在某个上下文查看或操作集合。它可以投影 OneToMany，也可以组合多个独立聚合或查询结果；是否属于同一聚合要用不变量和原子性验证。

**方式 6：拖拽到分组/文件夹**

```
📁 电子产品
  └ 📄 产品A    ← 拖入此分组
📁 服装类
  └ 📄 产品B
拖拽产品A到"服装类" → 执行“归入服装类”的命令
```
拖拽表达归类、移动或排序命令。实现可能修改分类引用、成员关系、路径、排序键或调用远程服务；不能把交互动作等同于某个字段写入。

**方式 7：头像/图标+名称**

```
负责人：🧑 张三    所属部门：🏢 销售部
```
这是具名对象摘要的可视化展示。点击可跳转详情，旁侧操作可切换对象；关系基数和存储方式仍需验证。

**方式 8：颜色标签**

```
状态：● 进行中（绿色）    优先级：● 高（红色）
```
颜色标签是分类、状态或优先级的视觉编码。底层可能是枚举、值对象、配置引用或派生状态；颜色本身不证明 ManyToOne 或状态表。

**方式 9：面包屑路径**

```
首页 / 产品 / 电子产品 / 手机 / iPhone
         ↑        ↑         ↑
       pid=1   pid=5    pid=12
```
面包屑表达导航层级。底层可能是自引用关系、物化路径、闭包表、路由配置或跨资源层级；只有 Schema/查询证据才能确认 `parent_id`。

**方式 10：纯文本展示**

```
创建人：张三（不可修改）    创建时间：2026-01-01
```
纯文本只证明当前上下文不可直接编辑。它可能来自对象引用、历史快照、审计投影或冗余文本，也可能因权限或业务状态暂时只读。

**方式 11：弹窗/抽屉预览**

```
客户：张三 [👁]  ← hover 或点击显示客户卡片预览
```
弹窗/抽屉可为多种关系提供快速预览。预览通常只承载低风险、短时操作；是否允许受控编辑或跳转到详情页，应由任务长度、权限、校验复杂度和失败恢复要求决定。

#### 2.4.3 UI 关系表现 → 候选与区分性验证

| UI 表现 | 可直接确认 | 至少保留的候选 | 优先验证 |
|---|---|---|---|
| 单值选择器 | 当前命令最多选一个值 | 枚举、资源引用、查询参数、快照值 | 请求体、API Schema、修改/删除关联对象后的行为 |
| “+”快速新建 | 可在当前任务创建候选对象 | 独立实体、临时草稿、内联值对象 | 创建响应、生成标识、能否独立访问和演进 |
| 拖拽到分组 | 存在归类/移动命令 | FK、成员表、路径、排序键、远程命令 | 操作前后请求与权威状态差异 |
| 内嵌列表 | 当前上下文展示集合 | 子对象、独立聚合、查询投影 | 保存原子性、生命周期、并发版本范围 |
| 颜色标签 | 存在分类/状态视觉编码 | 枚举、值对象、配置引用、派生值 | 编辑契约、状态转换、Schema |
| 面包屑 | 存在导航层级 | 自引用、路径、闭包表、路由配置 | 路由数据、层级编辑和查询契约 |
| 只读文本 | 当前上下文不可编辑 | 权限限制、状态限制、派生值、快照、引用 | 更换角色/状态、API 字段、编辑命令 |
| 异步搜索 | 候选通过查询获得 | 远程资源、本地大集合、权限过滤、搜索投影 | 网络请求、分页契约、权限差异 |

### 2.5 理论框架：UI 作为任务约束下的投影

可以把 UI 理解为数据空间在用户任务维度上的投影：

```text
数据空间：实体 E₁, E₂, ...，关系 R₁, R₂, ...，约束 C₁, C₂, ...
用户任务空间：用户要完成的 JTBD（"当 X 时，我想 Y，以便 Z"）
UI 空间：基于用户任务从数据空间选取字段、分组、排序、操作的投影

UI 证据 → 候选模型：从投影反推多个可竞争解释，再用区分性证据逐层收敛
任务与模型 → UI：在角色、任务、风险和平台约束下选择可验证投影
```

同一个模型可以产生多种 UI，同一个 UI 也可以由不同领域和存储实现产生。推导目标不是尽快选一个“最像的答案”，而是保留竞争性候选，优先消除会改变设计决策的关键不确定性。

### 2.6 推理引擎：从观察到可证伪结论

以下方法按“触发条件命中即执行”接入第 10 章 R2-R6，不是可选术语清单，也不是要求每条微小观察机械跑完整张表。溯因、可证伪、证据更新和证据上限是所有关键推断的底座；其余方法在对应条件成立时启用。

| 方法 | 何时使用 | 必须动作 | 输出/门禁 |
|---|---|---|---|
| 溯因推理 | 一个现象存在多种实现 | 生成至少两个能解释观察的候选；写出各自所需前提 | 单一观察不得直接收敛为唯一 M3-M5 结论 |
| 可证伪性 | 候选会影响模型或架构决策 | 为每个候选写出若为真应观察到什么、若为假会出现什么 | 没有可证伪预测的候选不得升级为 supported |
| 贝叶斯式证据更新 | 获得新截图、流量、Schema 或实测 | 比较“若 H1/H2 为真，出现该证据各有多意外”，更新候选相对排序；数值置信度仍按上位规则，不另建评分制 | 先验/常识不算新证据；相关、转述或同源证据不得重复计票 |
| 信息价值（VoI） | 待验证项多、成本不同 | 按“决策影响 × 区分能力 ÷ 获取成本”排序；先做能改变决策的低成本探针 | 高成本验证若不会改变当前决策，应延后并记录理由 |
| 因果过程追踪 | 从点击推断业务效果或机制 | 记录 `before → action → request → response → authoritative result → rendered state → recovery` | Toast、局部样式或一次 200 响应不能单独证明权威结果 |
| 反事实检查 | 判断某字段/按钮/状态为何出现 | 改变角色、权限、业务状态、数据边界或功能开关，观察结果是否随假设变化 | 不能产生预期差异时，降低或驳回候选 |
| 负面证据 | 声称“不存在/不支持/无权限” | 覆盖 `Role × Permission × Business State × Data Boundary × Viewport × Feature Flag × Version` | 覆盖不全只能写 `not_observed_in_scope`，不得写“不存在” |
| 往返一致性 | 对账候选模型与 UI | 从模型生成任务投影，再从运行 UI 提取可观察契约，对比语义、不变量和行为 | 自己生成的 UI 只能做一致性检查，不能作为独立证据证明模型正确 |
| 第一性原理与不变量 | 准备照搬竞品结构或既有实现 | 去掉页面、框架和表名，先写“必须解决的问题、必须恒真的业务性质、允许牺牲什么” | 不能回答时不得把竞品模式升级为新系统需求 |
| 敏感性/稳健性分析 | 方案依赖角色、规模、网络、阈值或数据分布 | 改变关键假设，检查候选排序、约束和 UI 是否随之改变 | 只在窄条件成立的结论必须标注适用域和失效点 |
| 失败预演（Premortem） | 方向 B 进入评审前 | 假设上线三个月后失败，列出最可能的权限、历史数据、并发、可用性和演进原因及早期信号 | 高后果失败没有检测、恢复或回滚设计时不得通过 |
| 可逆性分级 | 选择 Schema、聚合、对外契约或迁移方案 | 区分单向门/双向门；单向门补 ADR、迁移/回滚和更强证据，双向门走最小方案 | 不把可延后的物理选择伪装成当前必需决策 |
| 历史约束与隐性契约 | 发现反常字段、流程或兼容行为 | 先问它曾防止什么失败、哪些调用方依赖未明说的行为，再决定保留/替换 | 未确认历史作用和消费者前不得直接删除或“优化” |
| 抽样与选择偏差控制 | 准备把单账号、单租户、成功案例或可见页面外推为产品规律 | 记录样本如何被选中，主动补角色/失败/边界/历史/不同规模样本，并限定结论总体 | 方便采到的样本不代表真实分布；缺失机制未知时不得做全局比例结论 |
| 社会技术责任分析 | UI/模型改变审批、排班、绩效、风控或人工复核 | 标出谁录入、谁受益、谁承担错误/补录/申诉/值守成本，寻找线下表格、人工兜底和影子流程 | “系统自动化”不能隐藏实际转移给人的工作与责任 |
| 反馈回路与古德哈特检查 | 指标、排序、推荐、默认值或必填项会影响用户行为和后续数据 | 画出 `展示/规则 → 行为 → 数据 → 下一轮展示/决策`，检查操纵、偏差放大和二阶效应，并设置护栏指标 | 竞品展示某 KPI 不等于该 KPI 适合作为目标；被考核的指标会改变其含义 |

**停止条件：**关键候选已被独立证据验证或驳回；剩余探针信息价值低且不影响当前可逆决策；或证据不可得并明确输出 `evidence_insufficient`。涉及单向门、不变量、安全、租户、金额、库存、权限或对外契约的关键未知不得以低信息价值为由放行。

### 2.7 可辨识性、等价类与最小承诺

逆向工程存在**不可辨识性**：在已覆盖的角色、状态、数据和探针下，多个模型可能产生完全相同的可观察结果。此时证据只能识别一个“观察等价类”，不能识别其中唯一真实实现。

```text
H1：订单直接读取 customer 表
H2：订单保存 customer_name 快照
H3：订单读取搜索投影或 BFF 组合结果

若现有 UI/API 样本始终只返回相同 customerName，H1/H2/H3 暂时观察等价。
删除或重命名客户、读取历史订单、检查契约/血缘，才可能把它们区分开。
```

执行规则：

1. **先建等价类，不强选唯一答案。** 对无法区分的候选记录共同可确认语义和差异点，状态保持 `evidence_insufficient`。
2. **按决策相关性决定是否继续。** 如果候选差异不影响当前任务、不变量、安全、成本和演进，可以基于共同语义继续可逆设计；不得把临时工作模型写成底层事实。
3. **单向门必须区分。** 如果差异会改变数据迁移、审计、授权、金额口径、并发、一致性或对外契约，必须设计能打破等价类的探针，否则阻断该决策。
4. **遵守最小承诺。** 只承诺当前证据和任务必需的结构，推迟不必要的物理实现选择。奥卡姆剃刀可用于选择下一步成本最低的工作假设，不能用于提高其证据强度。

等价类记录至少包含：`equivalence_class_id`、候选成员、共同可确认语义、会改变的决策、区分性证据、当前状态和重新打开条件。

## 3. 方向 A：UI 证据 → 可观察契约 → 候选模型

### 3.0 观察通道与推断距离

L1-L4 表示证据通道与分析深度，不表示置信度高低。置信度由来源独立性、证据质量和验证状态决定：

```text
L1-静态观察：字段、标签、布局、可见状态（直接事实止于 M0/M1）
L2-结构观察：页面、导航、集合和上下文关系（产生 M1-M3 候选）
L3-行为观察：命令、前置条件、转换、失败与恢复（可支持 M2/M3 契约）
L4-接口/实现证据：请求响应、正式 Schema、源码、DDL、执行计划（按证据实际所在层确认）
```

任何通道都可能产生强或弱证据。正式 DDL 可直接证明 M5 表结构；单张截图只能证明 M0 可见事实。不得把“L1”自动标为高置信，也不得把“L4”自动标为低置信。

### 3.1 字段层级推断（L1）

**规则 1：字段类型逆向映射**

UI 控件先映射到视图/表单语义，再生成 API、领域和存储候选：

| UI 控件 | 可直接确认（M0/M1） | 候选模型（M2-M5） |
|---|---|---|
| `<input type="text">` | 当前视图接受文本 | API string；存储可为 VARCHAR/TEXT/派生值 |
| `<input type="number">` | 当前视图接受数字样式输入 | integer/decimal/计量值；仍需精度、单位和范围证据 |
| `<input type="email">` | 当前视图表达邮箱格式意图 | API string + format；权威校验和存储类型待验证 |
| `<input type="date">` | 当前视图采集日期 | DATE/string/value object；时区语义待验证 |
| `<input type="datetime-local">` | 当前视图采集本地日期时间 | timestamp/string/value object；时区转换待验证 |
| `<select>` 单选 | 当前命令最多选择一项 | 枚举/资源引用/查询参数 |
| `<select>` 多选 | 当前命令允许多项 | 数组/集合关系/中间实体；不是 OneToMany 证据 |
| 富文本编辑器 | 当前视图编辑富内容 | HTML/结构化文档/外部内容标识 |
| 文件上传 | 当前任务接受文件 | 对象存储引用/附件实体/BLOB/异步任务 |
| 颜色选择器 | 当前视图接受颜色值 | hex/token/主题引用/结构化颜色 |
| 评分控件 | 当前视图接受有限等级 | 有界整数/枚举/值对象；范围以实测或契约为准 |
| Toggle | 当前视图提供二态操作 | Boolean/二值枚举/命令；领域状态未必只有两态 |
| Slider | 当前视图接受有界连续或离散值 | integer/decimal/计量值；范围以契约为准 |

**规则 2：字段命名语义逆向**

UI 字段名 → 业务语义 → 候选技术命名。名称只用于搜索和对账，不得写成已确认字段：

```text
"客户姓名" → 候选 customer_name / display_name / name 投影
"创建时间" → 候选 created_at / occurred_at / derived timestamp
"所属部门" → 候选 department reference / snapshot name / derived scope
"负责人" → 候选 owner/assignee 关系或历史快照
"订单总额" → 候选 total_amount 派生值或已存储快照
"备注" → remark / note / description（需结合上下文）
```

**规则 3：必填 vs 可选推断**

```text
必填信号：字段旁有红色星号、控件有默认占位符但不接受空白提交
未显示必填标记：只能说明当前界面未显式标记；必须实际提交空值并观察权威响应，不能直接判为可选
```

**规则 4：只读字段识别**

```text
只读信号：灰色文字、下拉变体（不可编辑）、时间戳自动更新
候选：派生值、审计值、权限限制、当前状态限制或服务端控制字段
```

### 3.2 结构层级推断（L2）

**规则 5：列表页 → 可识别对象与任务集合候选**

列表页通常展示一个核心实体的集合。推断依据：

```text
列表标题 → 用户语言中的对象/任务候选
列表列 → 当前任务的识别、比较和决策字段
每行操作 → 当前状态下暴露的命令候选
新建入口 → 创建任务入口；不能单独证明聚合根
```

**规则 6：详情页 → 视图组合与上下文关系候选**

详情页揭示当前任务需要组合查看的内容，不直接揭示聚合边界：

```text
详情页顶部基础信息 → 当前视图的主要识别字段
详情页内嵌区块 → 子对象、独立资源或查询投影候选
详情页外跳链接 → 可导航资源候选；是否跨聚合需用第 2.3.1 节四问法验证
```

**规则 7：关联关系类型推断**

`XxxToYyy` 中左边（Xxx）是当前实体（主体），右边（Yyy）是关联目标。从 UI 行为推断关系类型：

```text
下拉选择 + 只选一个 → 当前命令对该关系的上界是 1；候选包括枚举、资源引用或快照
下拉选择 + 可选多个 → 当前命令接受多值；候选包括多对多、一对多引用集合、数组、重复字段或仅客户端选择集
  例：Customer 可以选多个 Tag；只有反向复用、关系契约或 Schema 证据才能支持 ManyToMany
复选框列表 → 当前任务接受有限候选集中的多选；存储关系待验证
上级变化后下级 UI 变化 → 依赖、派生或联动候选；不等同 ORM cascade
批量选择 + 批量操作 → 存在批量用户命令；后端可为单批接口、并发单项请求或异步任务

从 UI 行为补充推断候选对象的创建能力：
  选择器旁有"+"按钮 → 存在快速创建候选值/对象的入口；是否为独立实体待验证
  点击"+"后弹出新建页面 → 跳转新建（金蝶模式）
  点击"+"后弹出内嵌表单 → 弹窗新建（现代 SaaS 常用）
  新建完成后自动回填 → 关联关系在创建时建立
```

**规则 8：分页/排序/筛选 → 数据量与查询模式**

```text
分页控件 → 当前视图采用分批导航；数据量和分页层级待测
排序选项 → 当前视图/API 支持排序语义；数据库 ORDER BY 和索引待验证
高级筛选 → 存在组合查询能力；实现可能是数据库、搜索引擎、缓存或客户端过滤
筛选默认值 → 确认当前上下文的默认查询意图；来源可能是产品默认、用户偏好或角色配置
```

### 3.3 行为层级推断（L3）

**规则 9：表单提交 → 命令与业务规则**

```text
表单字段缺失时禁止提交 → 当前角色、状态和命令下存在前端必填约束；权威必填规则待查 API/领域验证
字段 A 填值时字段 B 自动填充 → 确认存在联动行为；计算、默认、远程查询或缓存回填是候选
字段 A 选择后字段 B 选项变化 → 确认当前表单有依赖；领域关系和存储实现待验证
提交前弹出确认 → 操作被设计为高后果或防误触；可逆性需实测
提交后显示成功且数据立即可见 → 候选包括乐观 UI、同步写入或缓存回显，需验证权威状态
提交后显示"处理中"且稍后可见 → 存在延迟完成语义；MQ、后台任务、轮询等实现待验证
```

**规则 10：状态流转 → 状态机**

从 UI 上的状态标签和可用操作按钮推断状态机：

```text
当前样本中的状态标签集合 → 已观察到的状态词汇；不能证明穷尽集合或存储为枚举
标签颜色（红/黄/绿）→ 当前 UI 的视觉分组；具体业务含义需结合图例和行为验证
每个状态下可用的操作按钮不同 → 候选包括状态依赖的命令、权限、数据条件或渐进披露
  例："待审核"状态下有"通过"和"拒绝"按钮
       "已通过"状态下只有"撤销"按钮
       → 形成审批状态机候选；还需验证非法转换、角色差异、副作用和恢复路径
状态流转后页面刷新 → 候选包括导航、整页重载、数据重新获取或服务端渲染
状态流转后局部更新 → 候选包括本地状态更新、缓存更新、局部重新获取或推送
```

**规则 11：操作原子性推断**

```text
单次点击作用于一条记录 → 存在单对象用户命令；内部可触发多个写入或异步流程
批量选择 + 批量操作 → 存在多对象用户命令；可由批量端点、并发单项请求、队列任务或分段处理实现
批量导入/导出 → 存在批处理任务语义；是否异步、是否有任务表和结果存储需通过接口与运行证据确认
```

### 3.4 机制层级推断（L4）

**规则 12：API 请求逆向推断**

通过浏览器 DevTools / 抓包获取请求：

```text
实际请求路径 `/api/v1/customers` → 确认当前环境存在该 M2 端点；资源语义和内部实现另行验证
  → POST /customers → 创建
  → GET /customers → 列表
  → GET /customers/{id} → 详情
  → PUT /customers/{id} → 更新
  → DELETE /customers/{id} → 删除
实际请求体字段 → 该调用的 M2 输入契约事实；不等于领域或存储字段
实际响应体字段 → 该调用的 M2 输出契约事实；嵌套结构不等于聚合边界
Authorization / X-Tenant-ID → 存在认证/租户上下文信号；真正隔离范围需跨账号和服务端拒绝行为验证
```

**规则 13：实时行为 → 传输机制候选**

```text
列表不用刷新就出现新数据 → 候选包括 WebSocket、SSE、轮询、长轮询、跨标签广播或本地乐观更新
出现"在线/离线"状态 → 存在在线状态语义；心跳、连接事件或服务端会话实现待验证
```

**规则 14：复用统一证据强度与置信度**

本文不重新定义数值区间，统一执行《01-方法论》第 2.2 节。最低要求：

```text
单一来源推断：confidence ≤ 0.5
两个独立来源：可升至约 0.7
三个及以上独立来源，或经实测/试用验证：方可 ≥ 0.9
仅有官方渠道的单竞品能力断言：confidence ≤ 0.6

同时记录 observation_strength、target_layer、inference_distance 和 validation_status。
```

### 3.5 推断结果输出格式

先写原子观察，再写执行期推断记录。下列 `inference-record.yaml` 用于保留竞争候选、证伪预测和运行状态；它不是《01-方法论》第 14.5 节的知识卡。只有通过 R7 归档门禁后，才映射成《01》规定的证据卡、普通知识卡或 ADR 卡，不把运行字段直接塞入知识卡 frontmatter。

```text
observation_id: OBS-<domain>-<sequence>
context:
  source/version/role/tenant/permission/feature_flag/business_state/data_set
  platform/viewport/locale/timezone/action_sequence/captured_at/content_hash
observed_fact: <只写看见、执行或收到的内容，不夹带底层结论>

inference_id: INF-<domain>-<sequence>
claim: <具体、可被证伪的候选>
evidence: [<证据卡 ID 或 observation_id>]
observation_channel: L1/L2/L3/L4
source_layer: M0-M5
target_layer: M0-M5
inference_distance: <跨层数>
candidate_explanations: [<至少两个，除非已有直接目标层证据>]
falsifiable_predictions: []
discriminating_probe: <最小区分性验证>
confidence: <按 01 第 2.2 节>
validation_status: hypothesis/supported/refuted/evidence_insufficient/verified
valid_until: <证据有效期>
open_questions: []
```

归档映射：`observed_fact + source metadata → evidence card`；经审查的 `claim + evidence + reasoning/boundary/counterexample → ordinary knowledge card`，其 `strength` 按《01》填写；设计选择 → `decision_status=proposed` 的 ADR。`validation_status` 是执行期假设状态，不等同知识卡 `strength/status` 或 ADR `decision_status`。

## 4. 方向 B：角色 + 任务 + 约束 + 候选模型 → UI 投影

### 4.0 映射层级体系

正向设计先确定用户任务，再使用模型约束投影。分四个决策层：

```text
任务层：角色/JTBD/频率/风险/平台 → 要完成什么、何时完成
命令与状态层：前置条件/命令/权威结果/失败恢复 → 交互合同
内容层：ViewModel + 领域语义 → 展示、比较、输入和反馈信息
载体层：页面/工作台/弹层/向导/API 等 → 以何种结构完成任务
```

必须先接入《08-规范-复杂产品能力完整性保障流程规范》的 `TASK → COMMAND → STATE → SURFACE`。只有 ER 图或数据库 Schema 时，产出最高为“候选数据管理界面”，不得声称完成产品 UI 设计。

### 4.1 结构层映射

**规则 1：聚合根 → 候选命令与保存边界**

```text
聚合根 → 决定命令入口、保存原子性、冲突检测和失败恢复的候选边界
  页面可以一页跨多个聚合，也可以把一个聚合拆到多个任务页
聚合内部对象 → 只能经根的命令维护，但 UI 可用内嵌区、向导或专用任务页呈现
值对象 → 按用户任务拆分输入与展示，不要求压成一个字符串
跨聚合引用 → 可使用跳转、选择器、摘要或组合工作台；交互方式由任务连续性决定
```

**规则 2：任务与信息架构 → 页面层级**

```text
先按角色目标和限界上下文形成任务簇，再依据频率、顺序、切换成本和权限设计导航。
实体数量只用于发现复杂度，不作为导航深度阈值。
同一实体若服务下单、履约、售后等不同任务，可以出现在多个任务入口；低频配置实体可以不进入主导航。
```

**规则 3：关系基数 → UI 操作模式**

`XxxToYyy` 中左边（Xxx）是当前实体（主体），右边（Yyy）是关联目标：

```text
OneToOne → 候选内联或组合展示；生命周期、权限和任务独立时仍可单独管理
  例：User 和 UserProfile 是一对一关系，Profile 在 User 详情页内联展示
OneToMany → 候选内嵌列表、专用列表或摘要入口
  例：Customer 是主体（当前页面的聚合根），Order 是多个目标 → 在 Customer 详情页内嵌 Order 列表
  内嵌列表上方有"新建"按钮，在父实体的上下文中创建子实体
ManyToMany → 根据关系是否有属性/生命周期，选择多选器、关系编辑器或独立关联实体页面
  例：Customer 和 Tag 是多对多 → 可用多选器；关系有属性、审批或生命周期时应建独立关系编辑界面
  选择器支持多选，选完后显示为标签列表，支持移除单个标签
自引用（树形）→ 树、路径、分级列表或搜索导航；只有业务允许重排时才提供拖拽
  例：Department 自引用 parent_id → 组织架构树形展示

ManyToOne 的选择器快速新建（企业软件常见模式）：
  选择器旁提供"+"按钮，支持快速创建关联实体
  ┌──────────────────────────┬─────────────┐
  │ 客户 *  [搜索客户...🔍] │ [+ 新建]   │
  └──────────────────────────┴─────────────┘
  点击"+"后两种处理方式：
    - 弹窗新建（金蝶模式）：弹出一个内嵌表单，创建完成后自动回填到选择器
    - 跳转新建：跳转到父实体的创建页，创建完成后再返回
  两种方式的选择依据：
    - 创建任务短、低风险、无需复杂校验且完成后立即回填 → 候选弹窗/抽屉
    - 需要多步骤、强校验、权限切换、附件或后续任务 → 候选独立页面
    - 通过任务测试验证中断成本、错误恢复和移动端表现；不按固定字段数量裁决
```

### 4.2 内容层映射

**规则 4：字段优先级排序**

```text
第一优先级（列表列 + 详情页顶部）：
  - 名称/标题/编号（可识别性）
  - 状态/类型（可分类性）
  - 关联对象名称（可追溯性）
  - 关键业务字段（金额/数量/日期）

第二优先级（详情页主体）：
  - 详细描述字段
  - 次要属性
  - 关联对象列表
  - 历史记录

第三优先级（详情页底部/折叠区）：
  - 系统字段（创建时间/更新时间）
  - 审计字段（创建人/修改人）
  - 冗余展示字段（派生值）
  - 内部备注
```

**规则 5：字段类型 → 控件类型**

| 数据类型 | 建议控件 | 说明 |
|---|---|---|
| BOOLEAN | Toggle 开关 | 二值开关 |
| ENUM（少量且需同时比较） | Radio 单选 / Tag 标签 | 是否平铺由标签长度、设备和任务频率决定 |
| ENUM（可扫描但不宜平铺） | Select 下拉 | 节省空间并保持选择效率 |
| ENUM（难以扫描或需搜索） | Combobox | 支持搜索；阈值由可用性测试和平台约束决定 |
| 外键 FK（少量） | Select 下拉 | 显示关联对象名称 |
| 外键 FK（大量） | AsyncSelect 异步搜索 | 避免加载所有选项 |
| 外键 FK（大量且模糊搜索） | Autocomplete 搜索选择 | 类似客户搜索 |
| 日期（仅日期） | DatePicker | — |
| 日期（带时间） | DateTimePicker | — |
| 日期（仅时间） | TimePicker | — |
| 日期范围 | RangePicker | 筛选常用 |
| DECIMAL / INT | InputNumber | 带增减按钮 |
| 大段文本 | TextArea | 可展开 |
| 富文本 | RichTextEditor | — |
| 金额（精确） | InputNumber + Currency Select | — |
| 手机号 | Input + 手机号校验 | — |
| 邮箱 | Input + 邮箱校验 | — |
| 文件/图片 | Upload 组件 | 预览缩略图 |
| 经纬度 | 地图选择器 | — |
| JSON 结构 | JSON Editor / CodeMirror | 内部使用 |
| 多值列表 | Tag 输入框 / Checkbox 列表 | — |
| 密码 | Password 输入 | — |

**规则 6：字段组合 → 表单布局**

```text
基础信息区块：名称、类型、状态、核心属性 → 放在表单顶部，横向排列
关联对象区块：选择器 + 已选项列表 + 新增按钮 → 放在表单中部
详细描述区块：多行文本、备注 → 放在表单底部，撑满宽度
系统信息区块：创建人/时间、系统字段 → 折叠展示，默认隐藏
操作区块：保存/取消/更多操作 → 固定在表单底部
```

**规则 7：派生字段的 UI 展示策略**

派生字段（计算得出，不独立存储）：

```text
直接展示（只读）：
  例：订单总价 = 商品单价 × 数量 → 显示"¥300.00"，输入时实时计算
作为编辑反馈：
  例：输入"到期日期"后自动显示"剩余 N 天"
折叠隐藏：
  例：派生值不关键时折叠，减少认知负担
明确标注：
  例：显示为灰色背景 + 标注"系统自动计算"
```

### 4.3 交互层映射

**规则 8：分层约束 → 权威执行 + UI 反馈**

| 约束层 | 权威执行位置 | UI 责任 |
|---|---|---|
| 展示/输入约束 | 视图模型 | 提前引导、格式化和即时反馈 |
| API 契约 | 服务端接口 | 发送合法形状，精确展示错误码 |
| 领域不变量 | 领域/应用服务 | 表达前置条件、禁用原因和失败恢复 |
| 持久化约束 | 数据库/权威存储 | 不替代服务端校验；处理并发冲突结果 |
| 权限/租户约束 | 服务端授权与数据范围 | 仅做可用性提示，不能充当安全边界 |
| 跨行/时态约束 | 事务、锁、版本或工作流 | 显示冲突、版本差异和重试/合并路径 |

前端可镜像部分规则改善体验，但 UNIQUE、FK、并发、权限和租户约束必须由权威层再次执行。

**规则 9：状态机 → 操作按钮与状态标签**

```text
状态枚举值 → 状态标签（Tag/Badge），颜色编码
状态依赖的操作 → 不同状态下渲染不同操作按钮
  例："草稿" → [编辑] [提交审批]
       "审批中" → [撤回]
       "已通过" → [发布] [撤回]
       "已发布" → [下架]
禁止操作 → 按钮置灰 + tooltip 说明原因
```

**规则 10：并发控制 → 乐观锁 UI**

```text
如果并发修改会破坏不变量且契约采用版本条件写入：
  - 客户端携带已读取的 version/ETag/修订号，服务端原子比较后写入
  - 冲突时展示差异，并按业务提供刷新、重试、合并或放弃
  - 可展示最后修改时间/人帮助判断，但时间戳本身不提供并发保护
仅存在名为 version 的字段不能证明乐观锁；它也可能是业务版本、文档版本或展示值。
```

**规则 11：软删除 → 列表筛选器**

```text
如果已确认存在可恢复删除/归档语义：
  - 按角色任务决定默认排除、回收站、归档筛选、恢复和永久清理入口
  - 明确保留期、引用完整性、唯一键复用、审计、权限和清理策略
仅存在 deleted / is_deleted 字段不能决定 UI；它可能表达逻辑删除、同步墓碑、来源状态或历史兼容字段。
```

**规则 12：审计字段 → 自动填充 + 隐藏**

```text
created_at / updated_at → 候选只读展示；是否展示由追溯任务决定
created_by / updated_by → 应由权威层填充；UI 可按审计、协作和权限需求只读展示
```

### 4.4 模型 → UI 的补充决策

数据模型只定义结构，UI 映射需要额外补充：

```text
展示优先级：哪些字段必须在列表页展示，哪些只需要在详情页
分页策略：默认每页多少条，允许用户调整吗
排序策略：列表默认按什么字段排序，哪些字段允许排序
筛选策略：哪些字段提供筛选器，哪些是高级筛选
批量操作：哪些操作支持批量（批量删除/批量审核/批量导出）
权限控制：哪些角色可以看到/编辑哪些字段（字段级权限）
国际化：哪些字段需要多语言展示
草稿保存：表单支持草稿自动保存吗
版本历史：是否需要保留编辑历史
```

## 5. 双循环隔离与对账

逆向发现与自产品设计必须分成两条证据链，防止自证循环：

```text
竞品发现循环：
  原始证据 → 原子观察 → 竞争性候选 → 区分性验证 → 已验证知识/未知项

自产品设计循环：
  角色与任务 → 不变量/命令/状态 → 候选模型 → UI 投影 → 契约/任务测试 → 用户或运行验证
```

两条循环可以通过已验证知识和 ADR 连接，但自产品生成的 UI 不能回写为证明其模型正确的独立竞品证据。双向对账追求的是语义可追溯和行为等价，不是表、DTO 与页面结构一一相同。

差异必须分类后处理：

```text
合法投影损失：UI 有意隐藏技术字段或组合数据 → 记录理由，不改模型
语义冲突：同一术语/状态/动作含义不一致 → 阻断并裁决
规则缺失：模型或 UI 无法表达已确认不变量 → 补设计并验证
未知机制：可见行为成立但实现无法判断 → 保留候选和验证队列
版本/上下文差异：角色、权限、功能开关或版本不同 → 分开建模，禁止强行合并
```

## 7. 质量门禁

### 7.1 UI 证据 → 候选模型门禁

```text
[ ] 原始观察与推断分开记录，观察包含版本、角色、权限、状态、数据和平台上下文
[ ] 每条执行期推断标注 source_layer、target_layer、inference_distance、confidence 和 validation_status；归档知识卡再按《01》填写 strength/status
[ ] 跨层推断列出竞争性解释、可证伪预测和最小区分探针
[ ] 关系、状态机和聚合候选分别检查基数、不变量、非法转换、失败恢复与替代实现
[ ] “不存在/不支持”结论通过负面证据覆盖矩阵；否则只写 not_observed_in_scope
[ ] UI、API、领域、逻辑持久化和物理实现没有被混成同一层事实
[ ] 关键未知输出 evidence_insufficient 或进入验证队列，没有用领域常识补成事实
```

### 7.2 任务与模型 → UI 投影门禁

```text
[ ] 输入包含角色、JTBD、任务频率/风险、状态、权限、数据边界和目标平台
[ ] 每个核心任务映射前置条件、命令、权威结果、可观察反馈和失败恢复
[ ] 页面/工作台/弹层/向导由任务连续性决定，不按实体或聚合数量机械生成
[ ] 字段按识别、比较、决策、输入和反馈用途选择；未使用字段允许不映射但需说明
[ ] 领域不变量和服务端约束有 UI 反馈，前端校验没有被当作权威安全边界
[ ] 状态覆盖正常、加载、空、错误、权限、并发、长文本和关键数据边界
[ ] 交互在目标视口、键盘路径和适用角色下可完成，不依赖隐藏技术 ID
[ ] 所有固定阈值均有适用条件、来源或项目基线；没有将经验数值写成通用硬门禁
```

### 7.3 双循环与追溯门禁

```text
[ ] 竞品发现证据与自产品设计产物隔离，自生成 UI 未被当作独立证据
[ ] 追溯链覆盖 OBS → INF/HYP → MODEL → TASK → COMMAND/STATE → SURFACE → TEST → EVIDENCE
[ ] 差异已归类为合法投影损失、语义冲突、规则缺失、未知机制或版本/上下文差异
[ ] 关键推断经过区分性验证；未验证项不能进入 accepted ADR
[ ] 执行期记录经 R7 审查后，按《01-方法论》第 14.5 节 Schema 映射为证据卡、普通知识卡或 proposed ADR
```

## 8. 常见陷阱

### 8.1 UI → 数据模型陷阱

| 陷阱 | 后果 | 规避 |
|---|---|---|
| 把 UI 展示名当字段名 | "客户名称" ≠ `customer_name`，可能叫 `c.name`、`cust.name`、`cname` | 标注为业务语义，不直接映射到技术命名 |
| 把下拉选项当枚举 | 选项可能来自独立配置表，不固定 | 区分"固定枚举"和"动态选项" |
| 把一条记录当一个实体 | 一行可能是 JOIN、聚合、搜索文档或 BFF 投影 | 先确认 M1/M2 标识，再用 Schema/生命周期验证 M3-M5 |
| 把删除 UI 当物理删除 | 隐藏、归档、软删、撤销和延迟清理外观相似 | 观察恢复、查询、引用和 API 语义，物理实现保持未知 |
| 把租户头当隔离证明 | Header 存在不等于服务端始终实施数据隔离 | 用两个授权租户做交叉访问拒绝测试并查服务端证据 |
| 把时间/人员字段当审计证明 | 普通字段不等于不可抵赖、完整的审计轨迹 | 验证追加性、操作者、前后值、时间和访问权限 |
| 把页面边界当聚合边界 | 任务组合与业务一致性边界不是一回事 | 用第 2.3.1 节四问法和事务/并发证据验证 |

### 8.2 数据模型 → UI 陷阱

| 陷阱 | 后果 | 规避 |
|---|---|---|
| 所有字段都展示 | 界面信息过载，核心数据被淹没 | 按规则 4 排序，超出认知负载的折叠 |
| 外键只展示 ID | 用户无法识别关联对象 | 用下拉/跳转替代 ID 展示 |
| 不展示派生值 | 用户需要心算 | 按规则 7 处理派生字段 |
| 状态用数字代码 | 用户无法理解数字含义 | 映射到有语义的状态标签 |
| 忽略操作权限 | 所有用户看到所有按钮 | 建立角色 × 命令 × 状态 × 数据范围权限矩阵，并验证服务端拒绝 |
| 表单一步到位 | 长表单认知负担重 | 按字段组合分区（规则 6） |
| 列表字段和详情字段完全相同 | 列表缺关键信息或详情冗余 | 区分列表列（可识别性）和详情字段（完整性） |

## 10. R0-R7 可执行工作流

### 10.0 状态、幂等与断点恢复

每阶段只使用总纲统一状态：

```text
not_started | in_progress | blocked | awaiting_human | passed | failed | superseded
```

这里只表示**阶段执行状态**。合法迁移为：

```text
not_started → in_progress
in_progress → passed | failed | blocked | awaiting_human | superseded
blocked | awaiting_human → in_progress | superseded
failed → in_progress | superseded
passed → superseded（仅当输入基线、上下文或规则变化使既有结果失效）
superseded → not_started（创建新修订后重新执行，不复活旧记录）
```

四类状态必须分字段保存，禁止互相代替：阶段 `stage_status` 使用上表；执行期候选 `validation_status` 使用 `hypothesis/supported/refuted/evidence_insufficient/verified`；知识卡生命周期 `status` 和 `strength` 使用《01》Schema；ADR 使用 `decision_status=proposed/accepted/rejected/validated/deprecated`。阶段 `passed` 只证明该阶段门禁已满足，不会自动把候选升级为 `verified`，更不会把 ADR 升级为 `accepted`。

`spec_id + spec_version + input_baseline_hash + direction + context_hash` 构成 `idempotency_key`。恢复时扫描执行记录和产物，从第一个非 `passed` 且依赖满足的阶段继续；输入、上下文或规范变化时只将受影响阶段标为 `superseded`，不得无差别重跑。

```yaml
execution_id: RE-YYYYMMDD-NNN
spec_id: 07-model-ui-reverse-engineering
  spec_version: "2.2"
direction: A|B|BIDIRECTIONAL
input_baseline: []
context_hash: ""
idempotency_key: ""
active_stage: R0
stage_status: in_progress
facts: []
inferences: []
decisions: []
evidence_refs: []
open_questions: []
next_action: freeze-context
verification: []
residual_risks: []
```

### 10.0.1 按方向选择执行路径

```text
direction=A（逆向发现）：R0 → R1 → R2 → R3 → R4 → R7
  只有要把受支持候选用于自产品 UI 设计时才进入 R5；只有存在正向投影可对账时才进入 R6。

direction=B（正向设计）：R0 → R5 → R6 → R7
  R0 同时读取已批准需求、领域/契约模型和现有证据；只有输入仍是待证候选时才回到 R3/R4。

direction=BIDIRECTIONAL：R0 → R1 → R2 → R3 → R4 → R5 → R6 → R7
  完整往返用于竞品发现与自产品设计对账，但两条证据链仍隔离，自生成 UI 不增加竞品候选置信度。
```

### R0 范围与上下文冻结

**输入：**目标产品/模型、分析方向、已知证据。  
**动作：**冻结产品版本、账号、角色、权限、租户、功能开关、业务状态、数据集、平台/视口、语言/时区、采集时间和授权范围；声明目标模型层 M0-M5。  
**输出：**`context.yaml`、输入清单与哈希、缺口清单。  
**门禁：**上下文不足时可以继续采集，但所有负面结论自动降为 `not_observed_in_scope`；未授权采集直接 `blocked`。

### R1 证据采集与清洗

**输入：**冻结上下文和采集计划。  
**动作：**按任务顺序采集截图、录屏、DOM/可访问树、请求响应、正式 Schema、日志或源码；脱敏并保留原始来源、时间、哈希和有效期。  
**输出：**证据卡和 `evidence-index.yaml`。  
**门禁：**营销页、同源重复截图和 AI 转述不能冒充独立来源；证据污染或无法追溯时 `failed`。

### R2 原子观察登记

**输入：**清洗后的证据。  
**动作：**只记录可复现事实；行为按 `before → action → request → response → authoritative result → rendered state → recovery` 登记。  
**输出：**`observation-ledger.yaml`。  
**门禁：**每条观察只能描述证据所在层；出现“数据库有”“使用了 MQ”等越层措辞时退回重写。

### R3 分层候选生成

**输入：**观察账本、已有领域知识。  
**动作：**使用溯因推理生成竞争性解释，标注 source/target layer、推断距离、前提、替代假设和不确定性；构建候选概念模型、Command/Query/ViewModel 映射、关系假设矩阵和约束/不变量注册表。  
**输出：**`candidate-models.yaml`、`relation-hypotheses.yaml`、`constraint-register.yaml`。  
**门禁：**跨层候选不足两个且没有目标层直接证据时不得进入 `supported`。

关系假设至少记录：方向、`0/1/N` 上下界、关系属性、生命周期、排序、删除语义、时态、租户范围、聚合候选、存储候选和反证实验。约束至少记录：适用命令/状态、作用域、权威执行层、UI 反馈、错误码、并发策略和证据。

### R4 区分性验证与信息价值排序

**输入：**候选及未决问题。  
**动作：**为每个候选写可证伪预测；以 `决策影响 × 区分能力 ÷ 获取成本` 排序探针；优先执行能区分多个候选的最低成本验证。  
**输出：**`verification-queue.yaml` 和逐项结果。  
**门禁：**同源重复观察不算交叉验证；验证失败必须削弱或驳回候选，不能只收集支持证据。

### R5 任务、模型与 UI 投影

**输入：**已验证/受支持候选、角色、任务、命令、状态和约束。  
**动作：**方向 A 输出可观察契约和候选模型；方向 B 先建立 `TASK → COMMAND → STATE → SURFACE`，再生成 ViewModel、信息架构和 UI 状态矩阵。  
**输出：**候选领域模型、UI 投影方案、状态/权限/数据边界矩阵。  
**门禁：**只有 Schema 而无任务输入时只能输出候选 CRUD 草案；不得把草案标为已验证产品设计。

### R6 双向追溯与反事实对账

**输入：**观察、候选模型、UI 投影和验证结果。  
**动作：**建立 `OBS → INF/HYP → MODEL_ELEMENT → TASK → COMMAND/STATE → SURFACE → TEST → EVIDENCE`；切换角色、权限、状态、数据边界、视口、功能开关和版本做反事实检查；分类所有差异。  
**输出：**`traceability-matrix.yaml`、差异清单和证据覆盖报告。  
**门禁：**关键任务、不变量或负面结论无法追溯时 `failed` 或 `evidence_insufficient`。

### R7 决策、归档与停止

**输入：**R6 结果、未知项和风险。  
**动作：**将满足《01》Schema 与证据门禁的事实/推断映射为知识卡；模型取舍生成 `decision_status=proposed` 的 ADR，风险接受形成待审决策包，阶段进入 `awaiting_human`。只有有权人/批准流程可以把 ADR 改为 `accepted`；记录证据有效期、复查触发器和残余风险。  
**输出：**知识卡、决策包、验证队列、最终报告。  
**停止条件：**关键候选已验证/驳回，或剩余未知不影响当前可逆决策；证据不可得时明确 `evidence_insufficient`。涉及单向门、不变量、安全、租户、金额、库存、权限或对外契约的未知必须阻断相关决策。


### 10.2 模型完整性七透镜

R3-R6 对每个关键对象至少检查以下七个透镜。未命中可以标 `not_applicable`，但必须给理由；不得只画实体关系图就声称模型完整。

| 透镜 | 必答问题 | 主要产物 |
|---|---|---|
| 身份与键 | 业务如何识别同一对象？技术主键、业务编号、外部标识和幂等键如何区分？ | identity/key register |
| 约束与不变量 | 哪些规则必须恒真？由哪个命令和权威层维护？破坏后如何检测与恢复？ | constraint/invariant register |
| 时间语义 | 记录时间、业务生效时间、时区、有效期、迟到事件和历史更正如何表达？是否需要双时态？ | temporal model |
| 权威来源 | 哪个系统/模型是 Source of Truth？哪些是快照、缓存、搜索文档、派生投影或事件？ | authority/projection map |
| 一致性与并发 | 重复提交、并发修改、乱序消息、部分失败如何处理？原子、一致或最终一致的边界在哪？ | consistency/concurrency contract |
| 租户与敏感性 | 所有权、租户范围、字段敏感级别、脱敏、保留、删除和审计要求是什么？ | tenant/permission/sensitivity matrix |
| 演进与迁移 | Schema/API/状态口径变化时，历史数据、旧客户端、缓存、事件消费者和回滚如何兼容？ | evolution/migration plan |

补充分类：明确对象属于主数据、交易、事件、快照还是查询投影；明确金额/单位/精度/舍入、空值/未知/不适用语义，以及状态和关系的 `0/1/N` 上下界。涉及金额、库存、订单、权限、多租户或消息时，这些不是增强项，而是相关决策的阻断门禁。

### 10.3 从模型导出验证性质与判定预言机

模型的价值不在于图画得完整，而在于它能产生可被反驳的行为预测。R3/R5 每增加一个关键模型元素，必须同步生成 `property → scenario → oracle → evidence`；没有预言机的“测试”只能证明流程跑完，不能证明模型正确。

| 模型元素 | 必须导出的验证性质 | 最小场景族 | 优先判定预言机 |
|---|---|---|---|
| 身份、业务键、幂等键 | 同一业务事实不重复生效；不同事实不误合并 | 正常、重复、同键异载荷、重放 | 权威记录、唯一约束、幂等结果契约 |
| 不变量与关系基数 | 合法边界可成立，越界组合始终被拒绝或纠正 | 最小/最大/空值、非法组合、删除被引用对象 | 领域规则、约束计算、权威错误码 |
| 状态机 | 合法迁移可达，非法迁移不可达，终态和回退语义明确 | 每条合法边、非法边、重复命令、终态命令、恢复路径 | 版本化状态迁移表 + 权威状态 |
| 权限、租户与数据范围 | 决策绑定主体、动作、资源和上下文；跨租户/越权始终拒绝 | 同角色跨租户、对象 ID 猜测、字段越权、状态变化后重试 | 服务端授权决策与审计，不以按钮可见性为准 |
| 金额、数量、单位与口径 | 精度、舍入、守恒和汇总口径一致 | 零/负数、边界精度、跨币种/单位、拆分再合并 | 已批准公式、精确数值计算、总分对账 |
| 并发与一致性 | 竞争写入不破坏不变量；冲突、部分成功和恢复可解释 | 同时修改、陈旧版本、超时后重试、成功响应丢失 | 原子条件写入结果、版本/锁证据、最终权威状态 |
| 事件、消息与异步任务 | 重复、乱序、延迟、丢失、重放后结果仍满足契约 | duplicate/out-of-order/delay/drop/replay | 事件契约、幂等消费记录、对账结果、DLQ/补偿轨迹 |
| 派生投影、缓存与指标 | 投影可从权威来源重建；新鲜度和误差在 SLA 内 | 缓存过期、重建、迟到数据、口径版本切换 | Source of Truth、血缘公式、reconciliation diff |
| 时间与演进 | 时区、有效期、迟到更正及新旧版本兼容符合契约 | 跨日/DST、历史更正、旧客户端、新旧 Schema 并存、回滚 | 业务生效时间规则、兼容矩阵、迁移前后对账 |

```text
判定优先级：已批准业务不变量/契约
           > 权威状态与可复现计算
           > 独立对账/审计轨迹
           > API 响应
           > UI 反馈
```

Toast、按钮消失、局部 DOM 更新或单次 HTTP 2xx 都不能单独作为业务成功预言机。高风险主张还必须形成轻量保证论证：`Claim（要证明什么） → Argument（哪些机制共同保证） → Evidence（哪些测试/监控/审计实际支撑）`。

候选只有在目标层证据直接命中，或其关键可证伪预测经过独立、可复现验证且没有未解释反例时，才可进入 `verified`。测试失败要更新候选、边界或模型，不能通过修改期望值让现有模型“通过”。

## 11. 桥接层模式：Schema-First 与 BFF

桥接层负责在任务/UI、API 契约、领域模型和持久化模型之间转换。本章提供常见候选模式；是否适用必须由当前系统证据、任务需求和约束验证，不能把模式存在当作已确认架构。

### 11.0 桥接层的必要性

数据模型和 UI 之间的直接映射存在三个核心矛盾：

```text
矛盾 1：结构 vs 视图
  数据模型是网状实体关系，UI 是扁平列表/表单
  候选解：查询服务、应用服务、BFF、GraphQL resolver 或客户端组合形成 UI 投影

矛盾 2：技术语义 vs 业务语义
  模型字段是 customer_id，UI 上要显示"客户名称"
  候选解：ViewModel、DTO、resolver 或映射层负责语义转换和关联展开

矛盾 3：粗粒度 vs 细粒度
  某些任务希望减少网络往返，而后端能力可能按服务或资源拆分
  候选解：BFF、网关聚合、GraphQL、批量 API、服务端组合或客户端并行请求
```

### 11.1 Schema-First 模式

**核心思想：用结构化 Schema 驱动类型、校验和候选控件生成，再由任务、权限、状态与平台约束完成业务 UI 设计。Schema 可减少机械映射，但不能替代产品决策。**

**11.1.1 JSON Schema → Form**

```javascript
// 定义 JSON Schema
const customerSchema = {
  type: "object",
  required: ["name", "email"],
  properties: {
    name: { type: "string", maxLength: 100, title: "客户名称" },
    email: { type: "string", format: "email", title: "邮箱" },
    status: {
      type: "string",
      enum: ["active", "inactive"],
      title: "状态"
    },
    tags: {
      type: "array",
      items: { type: "string" },
      title: "标签"
    }
  }
}

// Schema → Form 的自动映射规则
// string + no format → <input type="text">
// string + format:email → <input type="email">
// string + enum → <select>
// array + string items → <multi-select> 或 <tag-input>
// required → 必填校验
// maxLength → 可生成前端输入约束；服务端仍需在权威契约/领域层校验
```

代表工具：react-jsonschema-form、angular-schema-form、formily、JSON Forms

**11.1.2 OpenAPI → CRUD UI**

```yaml
# OpenAPI Schema
paths:
  /customers:
    get:
      responses:
        200:
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Customer'
    post:
      requestBody:
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CustomerInput'
      responses:
        201:
          $ref: '#/components/responses/Customer'

components:
  schemas:
    Customer:
      type: object
      properties:
        id: { type: integer }
        name: { type: string }
        email: { type: string }
        status: { type: string, enum: [active, inactive] }
        createdAt: { type: string, format: date-time }
    CustomerInput:
      type: object
      required: [name, email]
      properties:
        name: { type: string }
        email: { type: string }
        status: { type: string, enum: [active, inactive] }
```

Schema → 类型、契约与候选 UI 的生成路径：

```text
OpenAPI spec → 工具链 → 生成内容
─────────────────────────────────────────────
swagger-codegen / openapi-generator
  → TypeScript/Java 类型定义（前端强类型）
  → 客户端 SDK、服务端桩、文档或配置（取决于 generator）
  → 不默认生成具备业务任务、权限和交互语义的 React/Vue CRUD 页面

Stoplight Elements / Swagger UI
  → API 文档 + 在线调试界面

APIToolkit / Optic
  → API 契约测试 + Schema 变更检测

PostgREST / Hasura
  → 数据库 Schema → REST/GraphQL API → 自动 CRUD
```

**11.1.3 GraphQL 自省**

```graphql
# GraphQL Schema 定义后，前端可直接查询自省接口
query IntrospectionQuery {
  __schema {
    types {
      name
      fields {
        name
        type { name kind }
        args { name type { name kind } }
      }
    }
  }
}

# 前端查询示例
query GetCustomer($id: ID!) {
  customer(id: $id) {
    id
    name
    email
    orders {      # 客户端显式选择关联字段；由 resolver 决定如何解析
      id
      amount
    }
  }
}
```

GraphQL 自省允许工具读取类型、字段、参数和指令等 Schema 元数据。它可辅助生成类型、文档或通用候选控件，但不包含业务任务、字段优先级、权限意图和布局语义。Apollo Client 是 GraphQL 客户端与缓存库，不会仅凭自省自动生成业务 UI。

**11.1.4 Prisma Schema → 类型安全数据访问与数据浏览**

```prisma
// Prisma Schema 定义
model Customer {
  id        Int      @id @default(autoincrement())
  name      String
  email     String   @unique
  status    String   @default("active")
  orders    Order[]
  createdAt DateTime @default(now())
}

model Order {
  id         Int      @id @default(autoincrement())
  amount     Decimal
  customerId Int
  customer   Customer @relation(fields: [customerId], references: [id])
}
```

Prisma Schema 可驱动 Prisma Client 的类型安全数据访问，并参与迁移工作流。Prisma 不自动生成业务 CRUD API；Prisma Studio 是用于查看和编辑数据库数据的 GUI，不是产品级管理后台。

**11.1.5 Schema-First 的逆向工程方向**

对竞品进行 Schema-First 分析时：

```text
竞品 API 请求/响应 → 记录本次调用的 M2 观察形状
  → 形成可选性、类型和关系的候选，而不是完整 Schema 事实
  → 经多样本、错误响应、正式文档或源码验证后，再生成候选 UI

竞品没有 Schema 文档时：
  → 解析多个请求/响应样本并保留缺失值、错误和边界样本
  → 将字段类型、必填和枚举记录为候选；单个样本不能证明 required 或穷尽枚举
  → 生成标记 `hypothesis` 的 OpenAPI/JSON Schema 草案
  → 通过契约探测和任务验证后再用于候选 UI
```

### 11.2 BFF 模式（Backend for Frontend）

**核心思想：当不同前端渠道的任务、网络或数据形状显著分化时，可以为渠道设置专属后端适配层。需求相同的渠道可以共享 API，不应为了套模式强制拆分 BFF。**

**11.2.1 为什么需要 BFF**

```text
传统架构（单一 API）：
  Web Client  ──┐
  Mobile Client ──┼──→ REST API ──→ 微服务集群
  Mini Program ──┘

问题：
  Web 需要聚合 Customer + Order + Product 三个服务的数据
  Mobile 需要 Customer + Order 但不需要 Product，且字段更少
  单个 API 要满足所有客户端 → 要么返回过多数据，要么请求多次

BFF 架构：
  Web Client ──→ Web BFF ──→ [Customer Svc / Order Svc / Product Svc]
  Mobile Client → Mobile BFF ──→ [Customer Svc / Order Svc]
  Mini Program → Mini BFF ──→ [Customer Svc / Order Svc / Payment Svc]
```

**11.2.2 BFF 的数据聚合职责**

```text
BFF 接收：多个微服务的细粒度响应
BFF 输出：客户端需要的聚合响应

示例：订单详情页 BFF
  1. 调用 Customer Svc → 获取客户信息
  2. 调用 Order Svc → 获取订单基本信息
  3. 调用 Product Svc → 获取商品详情（订单中的商品）
  4. 调用 Inventory Svc → 获取库存状态
  5. 聚合以上数据 → 返回单一订单详情响应
     {
       orderId, status, totalAmount,
       customer: { name, phone, address },
       items: [
         { productName, price, quantity, inventory: "有货" }
       ]
     }
```

**11.2.3 BFF 与数据模型 → UI 的映射关系**

```text
BFF 是一种面向特定前端的候选适配层，不保证是最后一道转换层：

领域/API 能力层（不等同数据库结构）
       ↓
应用服务、resolver 或其他组合层（聚合边界需由领域证据确认）
       ↓
BFF 层（决定聚合在网络边界上如何呈现）
       ↓
BFF 输出的 DTO 决定了 UI 能拿到什么数据
       ↓
UI 层（根据 DTO 渲染控件和布局）

竞品逆向分析时：
  UI 请求数量和依赖 → 形成 BFF、网关、单体 API、GraphQL 或客户端组合候选
  API 响应结构 → 直接证明 M2 形状；结合主机、路由、文档或源码后才能确认由哪一层聚合
```

**11.2.4 BFF 的技术选型**

| 技术 | 适用场景 | 特点 |
|---|---|---|
| Node.js / Express / Koa | 轻量 BFF | 低门槛，与前端同语言，JSON 处理快 |
| GraphQL Gateway | 多端共用聚合层 | 按需查询，减少过度获取 |
| gRPC Gateway | 高性能内部通信 | 二进制协议，适合微服务间调用 |
| Spring Cloud Gateway | Java 生态 | 与 Spring Boot 微服务天然集成 |

### 11.3 API Gateway 聚合模式

API Gateway 与 BFF 没有固定的上下层拓扑。网关通常承载路由、认证、限流和协议转换；BFF 通常承载特定客户端的任务适配与组合。二者可前后串联、合并部署或只采用其中之一：

```text
职责 1：路由转发
  /api/customers/** → Customer Service
  /api/orders/**    → Order Service

职责 2：可选的响应聚合（与 BFF 没有固定上下层关系）
  将多个微服务的响应合并为一个
  常用于移动端网络受限场景

职责 3：横切关注点
  认证（JWT 验证）
  限流
  监控
  日志

职责 4：协议转换
  外部 REST → 内部 gRPC
  外部 GraphQL → 内部 REST
```

### 11.4 桥接层与双向逆向工程的对应关系

```text
方向 A（UI → 数据模型）需要桥接层知识：
  BFF 聚合策略 → 解释为什么一个页面只需一次请求
  API 响应结构 → 形成视图组合和关联展开候选，不能直接证明聚合根
  GraphQL 查询 → 证明客户端请求了哪些字段；字段顺序不构成业务优先级证据

方向 B（数据模型 → UI）需要桥接层知识：
  Schema-First → 用 Schema 驱动类型、校验和候选控件，再叠加任务/权限/状态设计
  BFF → 设计前端专用 DTO 与任务适配；是否需要该层由多端差异和运维成本决定
  API Gateway → 按路由、认证、限流、协议转换或聚合需求选配；不预设与 BFF 的拓扑
```

### 12.1 整体页面架构

### 12.2 侧边导航结构模式

**按模块分组（最常见）：**

```text

  高频操作放顶部，低频设置放下部
  当前模块高亮，上级父项展开

典型分组：
  概览 / 仪表盘

  业务域 B（订单管理）：
    订单列表 / 退款管理 / 评价管理
  业务域 C（营销管理）：
    优惠券 / 满减活动 / 推送通知

  系统管理（通常放底部）：
    角色权限 / 部门管理 / 系统配置 / 操作日志
```

**导航层级与页面类型的映射：**

|---|---|---|

| 四级导航 | 子模块详情 | /customers/:id/orders |

### 12.3 列表页通用模板

```text
列表页结构：

  □     | 数据| 数据| 数据| 数据| [查看] [编辑] [删除]
  □     | 数据| 数据| 数据| 数据| [查看] [编辑] [删除]

底部分页区（Pagination）
  共 XX 条  [< 1 2 3 ... 10 >]  每页 [20 ▾] 条
```

**列表页的 UI → 可观察契约与候选模型：**

```text
列标题 → 当前视图的用户语言字段；M1/M2/存储名称待验证
列数量与组合 → 当前视口、角色和任务下的信息选择；不构成数据模型维度
每行操作按钮 → 当前状态暴露的命令；不能单独证明 CRUD、聚合或事务边界
批量操作下拉 → 存在多对象用户命令；批量端点、并发单项请求和异步任务均为候选
列配置（用户可自定义列）→ 存在列偏好能力；偏好可存于 URL、浏览器、服务端或配置层
排序指示器（↑↓）→ 当前视图允许排序；请求契约和执行层待验证
```

### 12.6 Dashboard 仪表盘模式

Dashboard 是 SaaS 产品的主入口，常见模式：

```text
┌─────────┬─────────┬─────────┬─────────┐
│ 关键指标卡片（KPI Cards）              │

│                      │                │
│                      │                │
└──────────────────────┴────────────────┘
┌──────────────────────────────────────────┐

└──────────────────────────────────────────┘
```

Dashboard 的设计原则：

```text
关键指标卡片的推断含义：
  - 数字 + 趋势（↑↓）→ KPI 体系完整，有监控和预警
  - 点击卡片跳转 → 指标有 drill-down 详情
  - 颜色编码（红/绿）→ 阈值告警机制

图表类型的推断含义：
  - 折线图 → 随时间变化趋势（订单量/营收）
  - 柱状图 → 对比分析（各品类销量对比）
  - 饼图 → 占比分析（客户类型/订单来源）
  - 漏斗图 → 转化漏斗（线索→成交）
```

### 12.7 看板（Kanban）模式

看板是状态驱动任务流的典型 UI，适用于：

```text
看板的可观察含义与候选：
  列标题 → 当前视图的分组/阶段词汇；可能是状态、负责人、时间段或自定义分类
  列中的卡片 → 当前分组下的视图项；可能来自实体、任务、事件或查询投影
  卡片在列之间的拖拽 → 存在移动/重分类/状态变更命令候选；权威语义看请求和结果
  卡片上的信息 → 当前任务用于识别和决策的字段，不自动是领域核心属性
  列的数量和命名 → 只确认当前角色/配置下可见列，不能证明状态全集
  列之间的可拖拽路径 → 当前样本中的允许路径；单向性、可逆性和副作用需用反向操作及错误契约验证
  拖拽限制 → 候选包括状态前置条件、权限、容量、排序或客户端限制
  不同角色看到不同看板 → 存在上下文差异；权限、偏好、租户配置和功能开关均需区分
```

### 12.8 向导（Wizard）模式

步骤式表单，适用于复杂一次性任务的创建：

```text
多步骤表单的可观察含义与候选：
  步骤数量 → 当前版本把任务切成这些交互阶段，不证明领域流程也按此分段
  步骤之间保留输入 → 存在跨步状态；可在内存、URL、浏览器、草稿 API 或服务端会话保存
  进度指示器 → UI 表达当前位置/完成度；不能据此判断任务客观复杂度
  上一步/下一步 → 当前路径允许导航；回退后能否修改、是否触发补偿需实测
  每步的必填反馈 → 当前阶段的前端推进条件；最终权威必填规则待验证

典型步骤划分模式：
  Step 1：基础信息（名称、类型、关联对象）
  Step 2：详细配置（规则、条件、参数）
  Step 3：预览与确认
  Step 4：完成（成功提示或下一步引导）
```

### 12.9 SaaS 通用结构的逆向推断速查

竞品分析时，按以下顺序逐层登记观察并生成候选；每一步都受第 2.2 节证据上限约束：

```text
整体布局 → 页面区域、响应式行为与交互壳；技术栈只能由脚本、源码映射或运行特征另行识别
    ↓
侧边导航 → 当前角色的入口与任务分组；不等同限界上下文或业务域
    ↓
列表页 → 可比较项、识别字段与当前命令候选
    ↓
详情页 → 视图组合、上下文关系与可导航资源候选
    ↓
表单页 → 输入语义、前端约束与分组；权威规则和存储类型待验证
    ↓
Dashboard → 当前展示的指标、维度和口径线索；完整 KPI 体系待查血缘与定义
    ↓
看板/向导 → 阶段、路径和状态机候选；异常、权限与不可见状态待验证
```

## 13. 布局与导航模式

### 13.1 布局模式分类

**13.1.1 单列布局（Single Column）**

```text
适用场景：表单、文档阅读、设置页
结构：
  ┌────────────────────────────────────┐
  │ 标题                                │
  ├────────────────────────────────────┤
  │ 内容块 1                            │
  ├────────────────────────────────────┤
  │ 内容块 2                            │
  ├────────────────────────────────────┤
  │ 内容块 3                            │
  └────────────────────────────────────┘
候选内容组织：
  表单页 → 某个命令的输入与反馈；可跨聚合，也可仅编辑值对象或偏好
  文档阅读 → 长文本、富媒体、组合内容或外部文档投影
  设置页 → 用户偏好、租户配置、功能开关或系统参数；是否构成领域聚合待验证
```

**13.1.2 两列布局（Two Column）**

```text
适用场景：主从结构、详情+操作面板
结构：
  ┌────────────────────────────────────┐
  │ 标题                                │
  ├──────────────────┬─────────────────┤
  │ 主内容区          │ 侧边面板         │
  │                  │                 │
  │ 列表 / 详情       │ 操作 / 统计 /   │
  │                  │ 快捷入口         │
  └──────────────────┴─────────────────┘
候选内容组织：
  主内容 → 当前任务的主要对象、列表或详情投影
  侧边面板 → 补充命令、关联摘要、历史或指标；布局位置不证明聚合归属
```

**13.1.3 仪表盘布局（Dashboard Grid）**

```text
适用场景：多指标聚合首页
结构：
  ┌───────┬───────┬───────┐
  │ KPI   │ KPI   │ KPI   │  ← 等高卡片行
  ├───────┴───────┼───────┤
  │ 图表（宽）    │ 图表  │  ← 混合格式
  ├───────────────┴───────┤
  │ 列表 / 动态           │  ← 底部信息流
  └───────────────────────┘
候选数据契约：
  每个卡片对应一个有明确口径的指标，不要求按聚合根拆分
  图表需要维度、度量、时间/分类语义和新鲜度；来源可为实时查询、指标服务或预计算
```

**13.1.4 侧边导航 + 主内容（Shell Layout）**

```text
适用场景：几乎所有 SaaS 管理后台
结构：
  ┌────┬───────────────────────────┐
  │    │ 顶部栏                     │
  │侧  ├───────────────────────────┤
  │边  │                           │
  │导  │     主内容区               │
  │航  │                           │
  │    │                           │
  └────┴───────────────────────────┘
可观察契约：
  侧边导航项 → 当前角色可见的任务/资源入口；不等同聚合或领域边界
  主内容区根据路由动态渲染 → 存在路由到视图的映射；视图可以跨多个模型层
```

### 13.2 导航模式

**13.2.1 扁平导航 vs 层级导航**

```text
扁平导航（侧边全部展开）：
  优势：快速切换，一目了然
  劣势：选项多时过长
  适用：核心入口少、名称短、用户需要频繁切换且在目标视口可扫描

层级导航（可折叠分组）：
  优势：可扩展性强
  劣势：需要多一次点击
  适用：入口较多、存在稳定任务分组或角色需要渐进披露

混合导航（侧边栏固定 + 顶部 Tab 切换）：
  适用：同一模块内多个视图（如客户列表 / 客户地图 / 客户漏斗）
```

**13.2.2 面包屑导航**

```text
层级：列表页 > 详情页 > 子模块页
可观察契约：
  面包屑路径 → 导航与任务上下文层级，不等同聚合或物理父子关系
  例：/客户/客户列表/张三/订单列表/ORD001
      → 只确认用户从客户上下文导航到订单；订单仍可为独立聚合或查询投影
```

**13.2.3 Tab 切换**

```text
适用场景：同一任务上下文或资源的多个视图切面
示例：
  客户详情页的 Tab：
    [基本信息] [订单记录] [跟进历史] [文件附件] [操作日志]

数据模型推断：
  每个 Tab → 当前上下文的一个信息/任务切面，可能来自不同资源或查询
  数字徽章 → 确认存在计数语义；实时查询、缓存、预计算或搜索计数均是候选
```

### 13.3 响应式布局模式

**13.3.1 断点系统**

```text
示例断点（不是通用门禁，应按内容压力测试与项目设计系统调整）：
  Desktop ≥ 1200px：完整侧边栏 + 多列表格
  Tablet  768-1199px：折叠侧边栏 + 简化表格
  Mobile < 768px：汉堡菜单 + 单列卡片列表

数据模型 → 响应式映射：
  列表 → 表格（Desktop）/ 卡片列表（Mobile）
  详情 → 双列（Desktop）/ 单列（Mobile）
  筛选器 → 工具栏（Desktop）/ 抽屉（Mobile）
```

**13.3.2 列的自适应映射**

```text
Desktop 多列表格：
  客户名称 | 手机号 | 邮箱 | 订单数 | 累计消费 | 状态 | 操作
    ↓ 压缩到 Mobile
Mobile 卡片列表：
  ┌─────────────────────────┐
  │ 张三                    │
  │ 138xxxx | 活跃 ●        │
  │ 订单 12 笔 | 累计 ¥3,400│
  │ [查看详情]              │
  └─────────────────────────┘

推断原则：Mobile 卡片 = 列表行的"关键信息"字段子集
当前 Mobile 卡片无编辑入口 → 仅能写 `not_observed_in_scope`；还需检查详情、菜单、权限和版本
```

### 13.4 无限滚动 vs 分页

```text
选择决策：
  连续浏览、无需稳定定位或跳页 → 候选无限滚动/加载更多
  需要总数、跳页、可重复定位、批量选择或审计 → 候选分页
  单页渲染节点多且滚动性能成为瓶颈 → 候选虚拟滚动
  最终依据查询延迟、DOM 数量、设备内存、可访问性和真实任务测试，不使用固定行数裁决

UI 推断（方向 A）：
  无限滚动 → 当前任务偏连续浏览；不能据此判断总数据量或总数需求
  分页 + 总数 → 当前任务暴露定位与规模信息；数据量大小需测量
  虚拟滚动表格 → 当前实现优化渲染窗口；服务端数据量与分页策略待验证

数据模型推断：
  实际请求含 page/pageSize 且响应随之变化 → 确认 M2 暴露页码/页大小语义；内部可再转换为 offset、cursor 或其他查询
  无限滚动 → 前端已加载集合、offset、cursor/keyset 或其他 continuation 均是候选；以请求和响应验证
```

## 14. 表单设计模式

表单是数据录入的核心界面，表单设计质量直接影响数据的完整性和准确性。

### 14.1 字段编排模式

**14.1.1 分区块编排**

```text
表单字段按业务含义分组，而非按字母或字段顺序排列：

区块 1：核心标识（必须填写的核心字段）
  - 名称/标题（必填）
  - 类型/分类（必填）
  - 状态（默认选中的值）

区块 2：关联选择（需要关联已有数据的字段）
  - 所属部门（选择器）
  - 负责人（选择器）
  - 标签（多选标签）

区块 3：详细信息（可选的扩展字段）
  - 描述/备注
  - 附件上传
  - 扩展属性

区块 4：系统信息（通常隐藏或只读）
  - 创建时间、创建人
  - 最后修改时间
```

**14.1.2 双列 vs 单列布局**

```text
双列布局（适用字段关联紧密、标签/输入较短、视觉顺序明确且目标视口足够宽）：
  ┌──────────────────────────────────────────┐
  │ 客户名称 *      │ 手机号 *                │
  │ [___________]  │ [___________]          │
  │                                          │
  │ 邮箱            │ 所属部门                │
  │ [___________]  │ [___________]          │
  └──────────────────────────────────────────┘

单列布局（适用长表单或包含大段文本的表单）：
  ┌──────────────────────────────────────────┐
  │ 客户名称 *                                │
  │ [_________________________________]     │
  │                                          │
  │ 手机号 *                                  │
  │ [_________________________________]     │
  │                                          │
  │ 备注                                      │
  │ [_________________________________]     │
  │ [_________________________________]     │
  │ [_________________________________]     │
  └──────────────────────────────────────────┘
```

**14.1.3 渐进式披露（Progressive Disclosure）**

```text
默认状态（简洁）：
  ┌──────────────────────────────────────┐
  │ 客户名称 *    [_______________]      │
  │ 手机号        [_______________]      │
  │                  [显示更多选项 ▾]    │
  └──────────────────────────────────────┘

展开后（完整）：
  ┌──────────────────────────────────────┐
  │ 客户名称 *    [_______________]      │
  │ 手机号        [_______________]      │
  │ 邮箱          [_______________]      │
  │ 详细地址      [_______________]      │
  │ 备注          [_______________]      │
  │               [隐藏更多选项 ▲]       │
  └──────────────────────────────────────┘

数据模型推断：
  默认隐藏字段 → 只确认渐进披露；它仍可能在特定状态下必填、由系统提供或由其他字段派生
  展开后可见字段 → 当前设计将其视为较低频、较低优先级或上下文相关；具体原因待用户任务/埋点验证
  展开/折叠层级 → UI 优先级线索，不直接等同业务重要性、存储可空性或默认值
```

### 14.2 校验模式

**14.2.1 校验时机**

```text
失焦校验（Blur）：
  适用：格式类校验（邮箱、手机号、身份证）
  优势：即时反馈，不打断输入流程
  劣势：可能显示大量错误（如果用户 tab 快速切换）

输入时校验（Input）：
  适用：密码强度、字符计数、实时格式
  优势：用户实时看到约束反馈
  劣势：可能干扰正常输入

提交时校验（Submit）：
  适用：字段间交叉校验、业务规则校验
  优势：避免过多干扰
  劣势：用户提交后才发现错误，体验差

最佳实践：组合使用
  输入时：字符计数、密码强度
  失焦时：格式校验
  提交时：业务规则校验 + 汇总所有错误
```

**14.2.2 错误展示模式**

```text
行内错误（Inline）：
  输入框下方红色提示文字
  适用：简单单字段错误
  [____________]
  ↑ 格式不正确，请输入有效邮箱

字段组错误（Field Group）：
  多字段联合校验失败时
  适用：日期范围（开始<结束）、金额校验
  ┌─ 日期范围 ─────────────────────┐
  │ 开始日期    结束日期            │
  │ [_______]  [_______]           │
  │       ↑ 开始日期不能晚于结束日期 │
  └─────────────────────────────────┘

表单顶部汇总（Form-level Summary）：
  提交后顶部显示错误列表，点击跳转对应字段
  适用：长表单 + 多个错误
```

**14.2.3 校验与数据模型约束的映射**

```text
已确认的权威约束   候选前端反馈（不能替代权威校验）
─────────────────────────────────────────────
NOT NULL           在该命令确实要求输入时显示必填；系统生成/条件必填需按任务处理
UNIQUE             可提前探测以改善反馈，最终仍由服务端原子执行；避免泄露敏感对象存在性
VARCHAR(100)       按业务字符/字节口径设置输入提示；数据库长度不自动等于用户语义长度
DECIMAL(10,2)      按金额/计量规则处理精度、范围、单位和舍入
CHECK (value>0)    提供范围反馈；具体边界和步长来自业务规则
FOREIGN KEY        提供资源选择/标识输入；服务端验证存在性、租户范围、权限和并发删除
自定义业务规则     提交时后端校验 + 前端展示错误
```

### 14.3 联动模式

**14.3.1 字段联动**

```text
联动类型 1：计算联动（自动计算）
  输入"商品单价" + "数量" → 自动显示"小计"
  候选含义：前端临时计算、后端返回派生值、规则服务或缓存投影；权威金额口径必须由服务端执行

联动类型 2：选项联动（级联下拉）
  选择"省份"后，"城市"下拉更新
  候选含义：城市候选受省份约束；物理外键、配置树、远程目录、搜索过滤或客户端数据集均可能

联动类型 3：显示联动（条件显示）
  选择"类型=A"时，显示"字段A"
  选择"类型=B"时，显示"字段B"
  候选含义：类型影响当前 UI 可见性；是否条件必填及权威规则需用提交边界样本验证

联动类型 4：默认值联动
  选择"客户"后，自动填充"客户地址"
  候选含义：地址可能来自关联对象、历史快照、默认规则或远程服务；提交后是否保持快照需验证
```

**14.3.2 联动推断（方向 A）**

```text
从 UI 行为推断联动关系：
  填写 A 后 B 自动填充 → B 受 A 影响；派生、默认、远程查询和缓存回填均为候选
  选择 A 后 B 的选项变化 → B 依赖 A
  选择 A 后出现新字段 C → C 的可见性依赖 A；是否条件必填需以提交和错误契约确认
  填写 A 后 B 的值被清空 → B 与 A 不兼容
```

### 14.4 草稿与持久化模式

**14.4.1 自动保存草稿**

```text
触发时机：
  - 输入后 N 秒无操作（N 通常 2-5 秒）
  - 切换字段时
  - 页面失焦（beforeunload）

存储方式：
  - LocalStorage（前端草稿，不涉及数据模型）
  - 后端草稿表（跨设备同步草稿）

UI 表现：
  - 顶部显示"草稿已保存"提示
  - 草稿恢复弹窗（用户重新打开时）

数据模型推断：
  存在草稿恢复 → 只确认草稿可恢复；LocalStorage、IndexedDB、后端草稿或混合方案均是候选
  自动保存间隔短 → 只确认保存反馈频繁；本地缓存、HTTP 请求、批处理或实时通道需查网络和存储证据
```

**14.4.2 撤销/重做**

```text
撤销/重做栈：
  每个操作产生一个新状态快照
  用户可回退任意步

技术实现：
  - 命令模式（Command Pattern）
  - 不可变状态（每次修改产生新对象）
  - 本地撤销栈（内存）vs 持久化撤销（数据库）

数据模型推断：
  存在撤销/重做按钮 → 确认当前会话提供撤销语义；快照、命令日志、差异栈或服务端版本均是候选
  字段级撤销 → 形成细粒度变更记录候选；审计还需验证持久性、不可篡改性、操作者与访问控制
```

### 14.5 多语言字段模式

**14.5.1 前端多语言**

```text
结构：
  一个字段名，对应多个语言版本的值
  {
    name_zh: "客户",
    name_en: "Customer",
    name_ja: "顧客"
  }

UI 映射：
  根据当前语言设置显示对应版本
  编辑时显示当前语言版本 + 其他语言切换 Tab

适用场景：产品 UI 文案、邮件模板、内容管理
```

**14.5.2 后端多语言**

```text
结构：
  主表 + 多语言表分离
  customer: { id, created_at, ... }
  customer_i18n: { customer_id, locale, name, description }

UI 映射：
  主表字段直接展示
  多语言字段按 locale 筛选展示
  编辑时支持多语言 Tab

适用场景：支持多语言的企业 SaaS
```

### 14.6 表单质量检查单

```text
可用性检查：
  [ ] 必填字段有明确标识（*）
  [ ] 每个字段有标签（placeholder 不是标签）
  [ ] 错误信息描述清楚（"邮箱格式不正确"而非"格式错误"）
  [ ] Tab 键顺序与视觉顺序一致
  [ ] 支持键盘操作（Enter 提交、Esc 取消）

性能检查：
  [ ] 大列表使用异步搜索选择器，不一次性加载所有选项
  [ ] 长表单有懒加载（非首屏字段按需加载）
  [ ] 提交按钮在提交过程中禁用（防重复提交）

数据质量检查：
  [ ] 前端镜像适合提前反馈的规则；服务端/领域/数据库仍执行全部权威约束
  [ ] 字段间交叉校验在后端也有实现
  [ ] 唯一性校验在后端实现（前端无法阻止重复）
  [ ] XSS 注入和特殊字符有过滤

体验检查：
  [ ] 支持 Tab 在字段间切换
  [ ] Enter 键提交或移到下一字段有明确规则
  [ ] 长表单有进度指示
  [ ] 提交成功有明确反馈（不只是 Toast）
  [ ] 提交失败有错误汇总和恢复能力
```

## 15. 状态管理与数据流模式

前端如何持有和管理数据会显著影响 UI 与远程契约之间的同步质量。本章覆盖从简单本地状态到复杂实时协作的常见谱系。

### 15.1 状态分类谱系

理解状态是正确映射的第一步。以下五类是实用分类，不是互斥且穷尽的物理实现；同一值可同时出现在 URL、客户端缓存和服务器响应中：

```text
第一类：服务器状态（Server State）
  - 来源：远程 API、消息或服务，不等同数据库原始记录
  - 特征：服务端/远程端权威、异步获取；是否全局共享、是否缓存由范围与策略决定
  - 工具：React Query / SWR / Apollo Client / RTK Query
  - 契约映射：这类状态是远程契约在前端的投影
    UI 操作 → 命令请求/消息 → 权威结果 → 客户端协调缓存或重新获取 → UI 反馈

第二类：URL 状态（URL State）
  - 位置：Query String / Path Parameters
  - 特征：可分享、可书签、可浏览器前进后退恢复
  - 内容：筛选条件、排序、分页、当前选中项 ID
  - 契约映射：查询参数表达导航/查询意图，不等同 SQL 或存储字段
    ?status=active&page=1&sort=createdAt,desc
    → API、搜索服务、客户端或数据库可用不同方式实现同一过滤/排序语义

第三类：表单状态（Form State）
  - 位置：表单控件内部或表单状态管理库
  - 特征：临时性、中间态、最终提交才影响数据模型
  - 内容：输入值、校验错误、脏字段标记、提交中状态
  - 契约映射：表单状态 ≠ 领域模型或持久化模型
    表单可能是草稿（未提交）、可能是编辑中（diff 未保存）
    即使提交成功，也只能确认权威命令已接受或完成到某个状态；具体持久化需另证

第四类：UI 状态（UI State）
  - 位置：组件本地 useState / Context / Redux
  - 特征：纯前端、不持久、不影响数据模型
  - 内容：Modal 开关、Tabs 当前激活项、下拉菜单展开、hover 态
  - 契约映射：通常不属于业务持久化模型；若需跨设备恢复或审计则可能升级为偏好/会话数据

第五类：导航状态（Navigation State）
  - 位置：路由栈
  - 特征：描述用户"在哪里"和"怎么来的"
  - 内容：当前路由、历史栈、页面进入来源
  - 模型映射：面包屑表达导航层级，不等同聚合或持久化层级
```

### 15.2 数据获取模式

**15.2.1 渲染时获取（Fetch on Render）**

```text
组件挂载 → 发起请求 → 展示 loading → 数据到达 → 渲染

问题：请求发出后才渲染，可能导致瀑布式加载
瀑布：组件 A 挂载 → 请求 A → 响应 A → 组件 B 挂载 → 请求 B → 响应 B
     → 总时间 = sum(所有请求)，用户看到逐步加载

观察含义：请求时间线呈串行；候选包括真实数据依赖、组件挂载顺序、刻意延迟或未并行化，需检查调用因果链
```

**15.2.2 先行加载（Fetch Ahead）**

```text
在路由跳转前预先加载下一页数据
  路由守卫 → 判断即将进入的页面 → 发起预加载 → 导航

观察含义：路由前获取了下一任务所需数据；这些数据可能来自关联资源、独立查询或静态配置
  进入客户详情前预加载其最近订单 → 减少详情页加载时间
```

**15.2.3 缓存优先（Cache-First / SWR）**

```text
缓存中有数据 → 立即渲染缓存 → 后台静默刷新 → 数据变化时更新 UI
Stale-While-Revalidate

优势：缓存命中时可立即显示旧数据并在后台刷新；首次访问、缓存淘汰和错误态仍需 loading/恢复设计
劣势：可能显示过期数据

适用含义：SWR/stale-while-revalidate 适合允许先展示旧副本、再后台更新的读取场景；是否“读多写少”和可接受多长陈旧期由业务 SLA 决定
  例：客户列表 → 允许显示几秒前的数据
  例：支付状态 → 不允许，显示实时状态
```

**15.2.4 预取模式（Prefetch）**

```text
用户 hover 到链接时预取数据
  用户 hover "张三" 链接 → 立即发起客户详情请求
  用户点击时 → 数据已在缓存 → 几乎是即时加载

数据模型推断：
  hover 时触发请求 → 确认存在预取行为；不能据此判断数据量或 JOIN
  预取载荷 → 直接证明 M2 数据形状，来源查询和聚合实现待验证
```

### 15.3 数据变更模式

**15.3.1 乐观更新（Optimistic Update）**

```text
用户操作 → 立即更新本地 UI（乐观）→ 发送后端请求
    ├─ 请求成功 → 确认更新（无感知）
    └─ 请求失败 → 回滚 UI + 显示错误

适用场景：
  - 操作可逆（撤销成本低）
  - 用户体验优先（不想等网络延迟）
  - 不涉及复杂并发校验

不适用场景：
  - 涉及金额/库存等资源扣减（超卖风险）
  - 依赖服务端生成的主键或默认值
  - 需要服务端校验才能确认的操作

数据模型推断：
  乐观更新 → 只确认 UI 先显示预期结果；幂等、并发和服务端校验必须单独测试
  乐观更新 + 回滚 → 确认客户端处理失败响应；失败原因与权威规则需查错误契约
```

**15.3.2 悲观更新（Pessimistic Update）**

```text
用户操作 → 显示 loading → 等待请求完成 → 更新 UI

适用场景：
  - 操作不可逆
  - 需要服务端返回最新状态
  - 涉及资源竞争（库存扣减）
```

**15.3.3 批量变更（Batch Mutation）**

```text
多个字段变更可合并为一次命令，也可以按字段/步骤保存：
  表单中修改了 5 个字段 → 只有网络证据才能确认是单请求、自动保存还是多命令

与批量接口的关系：
  多个记录变更 → 可选批量端点、并发单项命令、分片任务或队列；须定义全回滚/部分成功/遇错即停和幂等语义
  单个记录多个字段变更 → 可选 PUT/PATCH/命令端点或自动保存；由契约决定
```

### 15.4 缓存失效策略

**15.4.1 缓存失效模式**

```text
Cache-Aside（常见模式之一）：
  读：缓存命中 → 返回；未命中 → 查 DB → 写入缓存 → 返回
  写：常见做法是更新权威存储后删除或更新缓存；一致性窗口、失败恢复和并发次序需显式设计

Write-Through：
  写：更新 DB + 更新缓存（同步）
  优点：可缩小常规路径中的不一致窗口
  缺点：增加写延迟和双写失败处理；不能保证“始终新鲜”

Write-Behind：
  写：更新缓存 → 异步批量写 DB
  优点：写入性能高
  缺点：未持久化队列/缓存故障时可能丢数据，并增加回放、顺序和幂等复杂度

Read-Through：
  缓存未命中时，缓存层自动加载数据并返回
```

**15.4.2 前端缓存失效**

```text
React Query / SWR 的失效策略：
  invalidateQueries → 指定 key 的缓存失效，下次访问重新获取
  refetchOnWindowFocus → 窗口重新聚焦时重新获取（处理多人协作数据变更）

数据模型推断：
  页面切换后数据刷新 → 候选包括重新挂载请求、缓存失效、焦点刷新、轮询或推送
  页面切换后数据不刷新 → 只说明当前观察未触发更新；缓存策略需查网络时间线与配置
  刷新按钮 → 手动重新获取/重算意图；是否使缓存失效待查请求和客户端实现
```

### 15.5 状态管理架构对比

| 模式 | 代表工具 | 适用规模 | UI→模型映射意义 |
|---|---|---|---|
| 服务端状态优先 | React Query / SWR | 以远程数据为主的应用 | 远程契约驱动 UI，客户端维护副本和新鲜度 |
| 全局客户端状态 | Redux / Zustand / Pinia | 跨视图客户端状态较多 | 可集中协调，但应避免把远程缓存与纯 UI 状态无边界混合 |
| 原子化状态 | Jotai / Recoil | 细粒度依赖较多 | 状态由原子组成，可按实现追踪依赖 |
| 表单专用状态 | React Hook Form / Formily | 任意 | 表单状态独立管理，不污染全局 |
| URL 优先状态 | React Router / Nuxt | 任意 | URL 即状态，页面刷新可恢复 |

## 16. 权限与安全 UI

权限模型决定了用户能看到什么、能做什么。本章从 UI 推断权限模型，以及如何从数据模型设计权限驱动的 UI。

### 16.1 权限模型层级

**16.1.1 五类授权控制面**

```text
以下是需要逐层检查的控制面，不是固定的五级表结构；RBAC、ABAC、关系授权、策略引擎或混合模型均可承载。

页面/能力控制（主体能否进入某任务或模块）
  权威判断：主体 + 能力 + 租户/环境上下文
  UI 投影：根据可发现性和信息暴露策略隐藏、禁用或解释入口；路由仍由可信边界复核

命令控制（主体能否执行某个业务动作）
  不局限 CRUD；submit/approve/refund/export/impersonate 等命令需独立表达
  UI 投影：只呈现当前状态下有意义的命令并说明不可执行原因；服务端再次授权

字段/属性控制（主体能否读取、输入、修改或导出某属性）
  读取、写入、脱敏和导出可能是不同权限；返回载荷本身必须按可信策略裁剪/脱敏
  UI 投影：隐藏、只读、掩码或可编辑取决于授权结果和任务，不用 CSS 隐藏保护敏感值

对象/数据范围控制（主体能访问哪些对象、行、关系或集合）
  范围可以来自 tenant/owner/org/relationship/policy，不应只靠客户端筛选
  UI 投影：只展示授权集合，并对空结果避免泄露其他对象是否存在

接口/通道控制（请求能否进入可信执行边界）
  端点、GraphQL 字段、消息订阅、导出任务和文件下载都需认证、授权与租户约束
  即使通过 DevTools、重放请求或猜测对象 ID 绕过前端，可信边界仍应拒绝
```

**16.1.2 UI → 权限模型推断（方向 A）**

```text
从 UI 行为推断权限层级：

某个菜单项不见了 → 候选包括页面权限、功能开关、角色配置、版本差异或响应式折叠
某个按钮不见了 → 候选包括操作权限、业务状态、数据条件或渐进披露
某个字段只读 → 候选包括字段权限、状态限制、派生值或系统控制字段
列表数据比预期少 → 候选包括数据范围、租户隔离、筛选、分页或数据不存在
API 返回 403 → 当前请求被服务端拒绝；还需结合身份、资源和策略确认权限模型

从 UI 布局推断权限设计复杂度：
  角色管理页面存在 → 确认产品暴露角色配置概念；RBAC/ABAC/PBAC 实现待验证
  权限配置页面复杂 → 形成细粒度授权候选；不能仅凭布局确认字段级权限
  UI 看不到其他数据 → 不能证明行级权限；必须做授权账号之间的交叉访问拒绝测试
```

**16.1.3 权限模型 → UI 映射（方向 B）**

```text
权限驱动 UI 的渲染决策：

用户登录 → 获取用户角色和权限集合
    ↓
路由守卫：根据已知授权状态改善导航体验；它不是安全边界
    ↓
页面渲染时：
  侧边导航 → 过滤掉无权限菜单项
  工具栏 → 根据操作权限渲染按钮
  表单字段 → 根据字段权限决定编辑/只读/隐藏
  列表数据 → 服务端按已验证的主体、租户、资源和策略执行范围控制；前端筛选不是安全边界
```

所有可由不受信任客户端触达的页面数据、操作、字段和数据范围权限必须在服务端或等价可信边界权威执行。UI 隐藏、禁用或路由守卫只能减少误操作，不能替代 API 鉴权、租户隔离和对象级授权。

### 16.2 角色模型

**16.2.1 常见角色模型**

```text
RBAC（基于角色的访问控制）：
  User → Role → Permission
  优点：简单直观，易于管理
  适用：角色相对稳定的组织

ABAC（基于属性的访问控制）：
  User + Resource + Environment → Policy → Permission
  优点：细粒度，可动态计算
  适用：复杂权限规则（如"经理只能看到自己部门的下属数据"）

PBAC（基于策略的访问控制）：
  策略语言描述权限规则，更灵活
  优点：最灵活，可表达复杂规则
  缺点：复杂，需要策略引擎

数据模型推断：
  竞品有角色管理页面 → 确认暴露“角色”管理概念；底层可为 RBAC、角色模板或混合策略
  竞品权限配置非常精细 → 形成细粒度策略候选；ABAC/PBAC/RBAC 扩展或定制规则均需接口/策略证据
  当前范围未见权限管理 → 只能记录 `not_observed_in_scope`；可能由上级系统、管理员、配置文件或固定策略管理
```

**16.2.2 继承与互斥**

```text
角色继承：
  Admin 继承自 Manager → Admin 拥有 Manager 的所有权限 + 额外权限
  候选模型：role.parent_id、自包含权限集、继承边或策略组合；用变更传播和 Schema 区分

角色互斥：
  用户不能同时拥有"会计"和"出纳"角色
  候选模型：互斥关系表、约束规则、策略引擎或审批校验
  UI 映射：角色选择时互斥角色自动禁用

组织层级继承：
  上级看到下级的数据（部门负责人 → 部门成员）
  候选模型：邻接表、物化路径、闭包表、目录服务或策略属性 + 服务端范围控制
  UI 映射：列表自动过滤，筛选器有"本部门/下级部门"选项
```

### 16.3 权限 UI 的设计模式

**16.3.1 权限控制的可感知性**

```text
隐藏模式：
  无权限的操作完全不显示
  优点：用户只看到能做的事
  缺点：用户不知道存在其他功能
  适用：入口对该用户无意义，或暴露能力本身有风险

禁用模式：
  无权限的操作显示但禁用（灰色按钮）
  优点：用户知道存在其他功能
  缺点：可能泄露能力存在性，也可能造成大量不可操作噪音
  适用：用户理解升级/审批/状态前置条件有价值，且暴露能力名称可接受

尝试后提示模式：
  点击无权限操作 → 弹窗提示"您没有权限，请联系管理员"
  优点：清晰告知缺失权限
  缺点：交互步骤多
  适用：授权依赖最新资源/环境状态而无法可靠预判，或需要保留申请权限路径；不应故意让明显无权操作走到最后才失败
```

**16.3.2 委托授权模式**

```text
临时将权限委托给其他人：
  A 将"审批权限"临时委托给 B（有时间限制）
  候选逻辑模型：delegation(delegator, delegatee, scope, permission, valid_from, valid_to, status, audit_ref)
  UI 映射：
    委托列表页面（管理我的委托/收到的委托）
    委托状态标签（谁在代理我 / 我在代理谁）
    委托时间限制（到某日自动失效）
```

**16.3.3 审批流中的权限**

```text
审批流是多角色权限的典型场景：
  提交人 → 提交申请（create）
    → 部门主管审批（approve/reject）
      → 财务复核（review）
        → 老板最终审批（final_approve）

数据模型推断：
  审批 UI 存在 → 确认当前任务暴露审批状态、命令或历史；状态机、事件流、工作流引擎和表结构均为候选
  每个节点的执行主体候选包括角色、具体用户、组织规则、表达式或外部服务
  委托授权是否影响执行主体需用角色切换、策略或运行轨迹验证

UI 推断：
  待我审批列表 → 当前用户作为审批人的任务
  审批历史 → 审批节点 + 审批人 + 审批意见
```

## 17. 异常与边界状态

好的产品不仅处理正常流程，还要优雅地处理各种异常和边界情况。本章覆盖错误、空状态、加载状态的系统化设计。

### 17.1 状态类型全景图

每个 UI 区域都存在以下状态：

```text
正常态（Nominal）
  └─ 数据加载完成，正常展示

加载态（Loading）
  ├─ 初始加载（Skeleton / Spinner）
  ├─ 操作中（提交中 / 处理中）
  └─ 刷新中（背景更新）

空态（Empty）
  ├─ 首次使用（引导创建第一个数据）
  ├─ 无数据（符合条件的数据为空）
  └─ 已全部删除（数据被清空）

错误态（Error）
  ├─ 网络错误（离线 / 请求失败）
  ├─ 权限错误（401 / 403）
  ├─ 资源不存在（404）
  ├─ 服务端错误（500）
  ├─ 业务错误（参数校验失败 / 业务规则冲突）
  └─ 超时错误

边界态（Edge Case）
  ├─ 数据量极大（需要分页/虚拟滚动）
  ├─ 数据量极少（只有 1 条）
  ├─ 字段值超长（需要截断/展开）
  ├─ 特殊字符（HTML / 脚本 / Emoji）
  ├─ 并发冲突（数据被他人修改）
  └─ 脱敏/加密数据（部分字段不可见）
```

### 17.2 加载状态设计

**17.2.1 骨架屏（Skeleton）**

```text
适用场景：内容结构已知，仅仅是数据未加载
原理：渲染占位形状（灰色块），替代 Spinner
优势：用户看到内容结构，心理等待时间更短

骨架屏推断：
  骨架屏存在 → 确认当前区域采用结构占位；组件库、自研 CSS、图片占位或服务端模板均可实现
  骨架屏精细（包含按钮、标签形状）→ 占位与预期布局较接近；是否来自成熟组件库需查依赖/源码
  骨架屏缺失 → 观察实际 loading、布局稳定性和等待反馈；不能仅凭缺失骨架屏判定体验差
```

**17.2.2 渐进加载（Progressive Loading）**

```text
第一阶段：显示文字骨架（"客户名称" "手机号"）
第二阶段：显示数据骨架（"张三" "138xxxx"）
第三阶段：完全加载

数据模型推断：
  渐进加载 → 确认 UI 分阶段呈现；并行请求、流式响应或客户端调度均是候选
  "客户名称"标签 → 只确认 M0 业务语义，不证明存储有 `name` 字段
```

**17.2.3 操作中的加载状态**

```text
按钮级 loading：
  [保存] → [保存中...]（按钮内 loading）
  适用：局部操作且不需要阻断其他区域；是否使用由任务风险和实测等待感决定

页面级 loading：
  全屏遮罩 + Spinner
  适用：整个页面确实不能安全继续操作；不要只因耗时就全屏阻断

后台处理（不阻塞）：
  Toast："正在处理，请稍候"
  不阻止用户继续操作（乐观更新）
  适用：异步长任务（批量导入/导出/发送）
```

### 17.3 空状态设计

**17.3.1 空状态类型与策略**

```text
类型 1：首次使用（First-Time Use）
  策略：引导用户创建第一个数据
  典型文案："还没有客户，点击下方按钮创建第一个客户吧"
  CTA 按钮：[新建客户] 突出显示

类型 2：筛选无结果（No Filter Results）
  策略：说明筛选条件 + 提供清除筛选的快捷方式
  典型文案："没有符合条件的客户，请调整筛选条件"
  CTA 按钮：[清除筛选]

类型 3：数据已清空（All Deleted）
  策略：说明现状 + 引导下一步
  典型文案："所有客户已被清理，是否需要恢复？"
  CTA 按钮：[查看回收站]
```

**17.3.2 空状态推断**

```text
从空状态设计评估当前任务支持：
  空白/误导 → 当前空数据任务存在可用性缺口
  有说明但无下一步 → 是否缺 CTA 取决于用户在该状态下是否有合法动作
  有合适说明和合法下一步 → 仅证明该空状态被设计，不能代表产品整体成熟

从空状态推断数据模型：
  "点击创建第一个客户" → 确认存在创建客户任务；聚合边界待验证
  "查看回收站" → 确认存在可恢复/查看已移除项的任务；软删字段、归档表或事件恢复均是候选
  "调整筛选条件" → 确认当前空态与筛选上下文有关；过滤可发生在客户端、API、数据库或搜索引擎
```

### 17.4 错误状态设计

**17.4.1 错误分类与处理策略**

```text
网络错误（离线）：
  UI：Banner 或弹窗提示"网络连接已断开，请检查网络"
  行为：对可安全重试的请求采用有上限的退避与抖动，并尊重 Retry-After；非幂等写入不得盲目重试
  离线数据：根据容量、敏感级别和同步语义选择内存、IndexedDB、受保护原生存储或不缓存

权限错误（401/403）：
  401：按认证协议区分会话过期、凭据缺失或刷新失败；Web 页面可跳转登录，API/嵌入式客户端未必适用
  403：保留任务上下文并说明无权；内联提示、受限页面或返回安全位置由风险与恢复路径决定
  观察结论：当前请求进入未认证/未授权错误分支；Token、Session、Cookie 及具体校验层需查协议与服务端证据

资源不存在（404）：
  UI：区分路由不存在、资源已删、标识错误和为防枚举而隐藏的资源；可用页面级或局部空/错误态
  恢复：返回安全上级、重新搜索或刷新，不能一律返回首页
  候选：标识错误、资源删除、租户/权限隐藏、版本迁移或路由失效

服务端错误（500）：
  UI：不泄露内部细节，提供关联 ID 和与幂等性相容的恢复动作；重试按钮只用于安全重试
  可观测性：由可信边界记录结构化错误/追踪；客户端上报需脱敏、采样并防递归失败

业务/校验错误（不固定为 400）：
  UI：具体错误信息（"客户手机号已存在"）
  精准定位到具体字段（表单内联错误）
  不是通用错误页
```

**17.4.2 错误恢复策略**

```text
自动重试：
  仅对瞬时错误且语义可安全重试的操作启用退避 + 抖动
  最大次数、总时限和间隔由任务时限、幂等性、服务端提示和重试预算决定，不设通用固定值

乐观锁冲突：
  UI："数据已被他人修改，请刷新页面重新编辑"
  提供：当前版本 vs 服务器版本对比
  用户决策：放弃当前修改 / 强制覆盖（需二次确认）

表单错误：
  每个错误字段行内显示原因
  表单顶部汇总错误数量
  点击汇总跳转对应字段

全局错误：
  全局错误边界（Error Boundary）捕获未处理异常
  降级 UI 替代崩溃的组件（而非整个页面崩溃）
```

### 17.5 边界状态设计

**17.5.1 数据量边界**

```text
列表数据边界：
  0 条 → 区分首次使用、筛选无结果、无权限和已清空
  少量 → 验证布局不会因分页/工具栏条件消失而跳动
  大量 → 根据查询延迟、DOM 节点、设备内存、定位需求和可访问性选择服务端分页、虚拟化或搜索优先
  不以固定记录数替代性能测量；同样数量在行复杂度、设备和网络不同情况下结论不同

字段长度边界：
  字段值超出容器 → 截断 + Tooltip 显示完整内容
  富文本字段过长 → 折叠显示 + "展开更多"
  图片过长 → 等比缩放 + 点击放大
```

**17.5.2 特殊内容处理**

```text
HTML / XSS：
  后端转义 → 前端显示为纯文本
  富文本编辑器 → 渲染 HTML，但做 XSS 过滤

Emoji：
  正常显示（现代系统均支持）
  但在纯文本导出时可能变成 ? 或方块

国际化特殊字符：
  阿拉伯文/希伯来文 → RTL（从右到左）布局
  中文竖排（罕见）→ 特殊排版

超长数字：
  银行卡号 → 每 4 位空格分隔
  科学计数法 → 大数用 K/M/B 缩写

日期时区：
  显示用户本地时区的时间
  悬停显示当时区的时间
  跨天数据（如美国用户看中国日期）→ 明确标注时区
```

## 18. 实时协作与多人编辑

当多个用户同时使用同一份数据时，UI 与数据模型之间增加了同步层。这一章覆盖实时协作的设计模式和逆向推断。

### 18.1 实时同步模式

**18.1.1 轮询（Polling）**

```text
最简单：定时（每 N 秒）发起请求获取最新数据
  setInterval(() => fetch('/api/status'), 5000)

优点：实现简单，所有 HTTP 环境可用
缺点：延迟 = 轮询间隔，存在浪费请求

数据模型推断：
  网络时间线出现固定间隔请求 → 支持轮询候选
  仅观察到数字周期变化不能排除推送、缓存刷新或其他触发
  刷新按钮只证明手动刷新命令，不是轮询证据
```

**18.1.2 长轮询（Long Polling）**

```text
请求发出 → 服务端 hold 住连接 → 有变化时返回 → 客户端立即发起下一个请求
  Client: GET /api/updates
  Server: (hold, 等待数据变化...)
  Server: 返回 {changed: true}
  Client: 立即发起下一个长轮询

适用：中等实时性需求（秒级），不需要 WebSocket 的场景
```

**18.1.3 WebSocket**

```text
建立持久连接 → 双向实时通信
  连接建立 → 服务端可主动推送 → 客户端可主动发送

推送类型：
  数据变更推送：其他人修改了数据 → 你的 UI 自动更新
  在线状态推送：用户上线/离线
  通知推送：待办事项、新消息
  进度推送：长任务完成百分比

数据模型推断：
  捕获 WebSocket 连接和消息 → 确认该通道承载相应消息
  列表无需刷新出现新数据 → 仍需区分推送、轮询、跨标签广播和本地更新
  "张三正在编辑..." → 确认协作感知语义；硬锁、软 presence 或租约均是候选
```

**18.1.4 Server-Sent Events（SSE）**

```text
单向通道（服务端 → 客户端），类似 WebSocket 但更简单
  适用：只需要服务端推送，不需要客户端频繁发送的场景
  例：仪表盘数字实时更新、通知推送

与 WebSocket 的选择：
  单向事件流、HTTP 基础设施优先 → 可评估 SSE
  低延迟双向消息且协议/代理支持 → 可评估 WebSocket
  还需比较认证、重连、背压、消息顺序、移动网络和运维能力
```

### 18.2 协作冲突处理

**18.2.1 编辑锁（Lock）**

```text
用户 A 进入编辑页 → 服务端加锁 → 其他人看到"张三正在编辑"
  用户 B 进入编辑页 → 提示"该记录正被张三编辑，请稍后再试"
  用户 A 离开/超时 → 自动解锁

候选模型：
  edit_lock(user_id, record_id, fencing_token, locked_at, expire_at) 或等价租约/软 presence

UI 推断：
  显示"正在编辑" → 确认 presence/占用提示，不足以证明服务端硬锁
  超时释放 → 确认存在过期语义；字段、缓存 TTL 或会话租约均是候选
  可强制解锁 → 确认有管理命令；必须验证权限、审计和 fencing 防止旧写者继续提交
```

**18.2.2 OT（Operational Transform）**

```text
用于富文本协作编辑（如 Google Docs）
  用户 A 和用户 B 同时编辑同一段文字
  各自的操作被转换为操作序列
  服务端合并操作，消除冲突

数据模型推断：
  实时协作 → OT、CRDT、服务端串行操作、锁或其他算法均是候选；用离线合并、并发字符插入和消息载荷区分
```

**18.2.3 CRDT（Conflict-free Replicated Data Type）**

```text
无冲突复制数据类型
  多个离线/在线副本可以独立修改
  在满足所选 CRDT 的合并规则、消息传播和元数据条件时可确定性合并；仍可能需要服务器中继、持久化、授权和垃圾回收

适用：离线优先的应用（协作白板、共享笔记）
逆向判断：观察到离线合并仍不能确认 CRDT；需检查操作/更新载荷、并发合并性质或实现证据。Yjs 是候选实现之一，不等同“前端状态管理”通类。
```

**18.2.4 最后写入胜出（Last Write Wins）**

```text
最简单的冲突策略
  以时间戳或版本号决定保留哪个版本
  丢失的变更没有恢复机制

UI 推断：
  编辑冲突 → 提示"数据已被修改，是否覆盖？"，只确认存在冲突处理分支
  无冲突提示 → LWW、服务端串行、锁、CRDT/OT、无并发样本或静默覆盖均是候选
```

### 18.3 通知与消息系统

**18.3.1 通知类型**

```text
系统通知（System）：
  数据变更提醒（"订单 #123 状态已更新为已发货"）
  审批提醒（"您有一个待审批的报销申请"）
  异常告警（"系统检测到 X 异常"）

用户消息（User-to-User）：
  直接消息、评论、@提及

数据模型推断：
  通知中心存在 → 确认存在通知查询/展示能力；表、事件流、第三方服务、按需计算及轮询/推送均为候选
  通知可标记已读 → 确认存在按主体记录阅读状态的语义；布尔值、时间戳、游标或单独 receipt 均可能
  通知可跳转 → 确认通知载荷可解析目标；可用 URL、路由对象、资源引用或命令描述实现
```

**18.3.2 通知 UX 模式**

```text
通知入口：
  顶部栏铃铛图标 + 未读数字徽章
  点击 → 通知列表下拉或全页

通知分组：
  按类型/时间分组（"3 条新订单更新"）
  避免通知列表过长

通知设置：
  用户可配置通知类型和推送渠道
  候选模型：用户/角色/租户级偏好、策略或外部通知服务配置；表结构待 Schema 证据确认
```

## 19. 性能与体验优化

性能由网络、渲染、数据访问、缓存、资源体积和运行环境共同决定，不能归结为数据模型与 UI 的“距离”。本章提供候选优化模式；结论必须来自性能时间线、资源清单、服务端追踪、查询计划或可复现基准。

### 19.1 加载性能优化模式

**19.1.1 分片加载（Code Splitting）**

```text
按路由分片：每个页面的代码独立打包
  用户访问 /customers 时才加载 customers 相关 JS/CSS
  首屏加载只包含公共代码

验证方式：
  页面切换白屏 → 候选包括代码分片、数据等待、渲染阻塞、字体/资源加载或错误恢复
  用网络瀑布、bundle manifest、Performance trace 和缓存状态区分
  性能目标以项目 SLO、设备和网络基线定义，不设通用首屏秒数
  配置化 UI 仍需加载引擎、配置和组件代码，不是分片延迟的通用替代方案
```

**19.1.2 预加载与预取**

```text
预连接（Preconnect）：
  <link rel="preconnect" href="//api.example.com">
  提前建立 TCP/TLS 连接

预加载关键资源（Preload）：
  preload 用于提前获取关键资源；SSR 是独立渲染策略，不能与 preload 等同

预取下一页数据（Prefetch）：
  <link rel="prefetch" href="/api/next-page-data">
  用户可能访问的页面提前获取数据
```

**19.1.3 虚拟列表（Virtual Scrolling）**

```text
适用：DOM/原生节点数量和行复杂度导致可复现的渲染、滚动或内存问题
原理：只渲染可见行 + 上下占位
  1 万条数据 → 实际 DOM 只有 ~20 个节点

观察结论：
  虚拟列表存在 → 确认当前实现只渲染可见窗口附近内容
  是否还需服务端分页取决于总数据、查询成本、排序/筛选和内存，不由虚拟化本身决定

前端实现：react-window / react-virtualized / ant-design Table 的虚拟配置
后端分页可用 cursor/keyset 或 offset。cursor/keyset 常利于深分页和稳定遍历，offset 支持跳页且在小偏移时可接受；必须按排序稳定性、数据变化和查询计划选择，不能笼统声称一种始终更快。
```

**19.1.4 图片与媒体优化**

```text
图片加载策略：
  懒加载（Lazy Load）：进入视口才加载
  渐进加载：先加载模糊缩略图 → 再加载高清图
  响应式图片：不同屏幕宽度加载不同分辨率

数据模型推断：
  列表页缩略图 → 确认 UI 使用缩略资源；URL 字段、媒体实体、CDN 参数或构建产物均是候选
  头像有多个尺寸 → 候选包括图片服务、静态多版本、客户端缩放或第三方资源
  懒加载存在 → 确认资源被延后获取/解码；可能为数量、带宽、首屏优先级、内存或平台策略所驱动
```

### 19.2 交互性能优化

**19.2.1 防抖与节流**

```text
防抖（Debounce）：
  输入停止 N 毫秒后才执行
  适用：搜索输入（用户停止输入后才查询）

节流（Throttle）：
  每 N 毫秒最多执行一次
  适用：滚动事件、窗口 resize、拖拽

数据模型推断：
  搜索框输入后延迟显示结果 → 防抖、网络延迟、服务端处理、调度或显式提交均是候选；检查按键与请求时间线
  拖拽过程中请求频率受限 → 节流、批处理、松手提交或队列合并均是候选；“实时保存”文案不足以确认
```

**19.2.2 前端缓存**

```text
内存缓存（In-Memory）：
  JS 变量缓存请求结果
  页面刷新消失

LocalStorage / SessionStorage：
  页面刷新可恢复
  存储格式：JSON 字符串

IndexedDB：
  结构化大数据存储（离线应用）
  例：PWA 离线缓存

数据模型推断：
  已直接检查到草稿位于 LocalStorage → 可确认该版本使用浏览器键值存储；敏感性、容量和清理策略仍需评审
  离线可查看 → HTTP 缓存、service worker、Cache Storage、IndexedDB、LocalStorage、内存快照或原生存储均是候选
  登录状态跨刷新保留 → 存在会话恢复机制；可能是受保护 Cookie、内存 + 刷新会话、原生安全存储等，不能据此推断 Token 位于 LocalStorage
```

**19.2.3 服务端渲染（SSR）与静态化**

```text
SSR（Server-Side Rendering）：
  首屏 HTML 由服务端渲染，数据已嵌入
  可改善部分场景的首屏内容与爬虫可见性；实际性能取决于 TTFB、流式/水合、缓存和客户端 JS 成本

静态站点生成（SSG）：
  页面在构建时生成，部署为静态文件
  候选用于内容不频繁变化且可在构建/增量阶段生成的页面；性能仍取决于资源与分发

客户端渲染（CSR）：
  浏览器下载 JS → 执行 → 渲染
  首屏成本受包体、缓存、网络和渲染策略影响，不能笼统认定一定最慢

逆向判断：
  初始响应含完整 HTML → SSR、SSG、预渲染、边缘渲染或服务端模板均是候选；结合响应变化、构建特征和水合标记区分
  初始响应主要是根节点 → 支持 CSR/客户端注入候选，但还需排除加载失败、壳页面或后续流式片段
  可抓取 HTML → 只确认初始内容可见性；页面类型和 SEO 目标需另证
```

### 19.3 数据加载优化

**19.3.1 N+1 查询问题与解决**

```text
N+1 问题：
  获取 100 个客户，每个客户需要显示其订单数
  1 次查询获取客户 + 100 次查询获取每个客户的订单数
  = 101 次数据库查询

解决：
  预加载（Eager Loading）：1 次查询获取所有客户 + 订单（JOIN）
  聚合查询：在数据库层面 COUNT → 1 次查询

UI 推断：
  列表/详情慢或快都不能单独证明 N+1、JOIN 或预聚合
  用服务端 trace、查询计数、执行计划和数据量对照确认；先排除网络、资源、序列化和渲染瓶颈
```

**19.3.2 字段裁剪**

```text
列表接口只返回列表页需要的字段（避免返回完整聚合）：
  列表：GET /customers?fields=id,name,status
  详情：GET /customers/:id（全量）

GraphQL 的字段裁剪优势：
  客户端声明需要的字段；是否减少数据库/下游工作取决于 resolver、数据加载和服务实现

REST 的字段裁剪实现：
  ?fields=id,name,status
  后端根据 fields 参数选择性返回
```

**19.3.3 增量加载**

```text
初始只加载列表前 20 条
用户滚动到底部 → 加载下一批
  （无限滚动）或
用户点击"加载更多" → 加载下一批
  （分页按钮）

数据模型推断：
  无限滚动/加载更多 → API 可使用 cursor/keyset、offset 或其他 continuation token
  显式分页按钮 → 可能使用 offset，也可能用带页码映射的其他策略
  通过请求参数、响应 continuation 和数据变化时的稳定性验证具体实现
```

## 20. 搜索与筛选模式

搜索和筛选是数据模型在 UI 上的查询入口。本章覆盖从简单关键字搜索到高级搜索的所有模式。

### 20.1 搜索模式谱系

**20.1.1 即时搜索（Instant Search）**

```text
输入即搜索（每输入一个字符触发）
  适用：小数据集或服务端性能足够

优化手段：
  防抖（Debounce）避免请求过多
  搜索结果高亮匹配词

数据模型推断：
  输入与请求之间有延迟 → 候选包括防抖、节流、网络延迟或服务端处理
  搜索结果实时更新 → 只确认交互行为；缓存与触发策略需查请求时间线
```

**20.1.2 搜索建议（Autocomplete / Typeahead）**

```text
输入时显示下拉建议列表
  输入"张" → 显示 ["张三", "张四", "张经理"]

数据模型推断：
  建议列表 → 存在建议数据源；可来自本地集合、通用搜索 API 或独立端点
  建议显示额外信息 → M2 载荷包含这些字段；JOIN、搜索文档、缓存或冗余投影均是候选

技术实现：
  前缀匹配：LIKE '张%'
  拼音首字母匹配：pinyin 首字母索引
  纠错建议：编辑距离 / 拼音相似度
```

**20.1.3 高级搜索（Advanced Search）**

```text
多条件组合搜索：
  客户名称：[包含] [张]
  手机号：[精确] [138xxxx]
  创建时间：[介于] [2026-01-01] - [2026-06-30]
  状态：[等于] [活跃]

数据模型推断：
  高级搜索存在 → 确认组合查询能力；索引、搜索引擎和执行策略需用查询计划验证
  搜索条件可保存 → 候选包括用户偏好表、配置文档、URL 或浏览器存储
  搜索条件可分享 → 观察实际分享载荷后确认 URL、服务端短链或已保存搜索标识

字段类型 → 筛选器类型映射：
  字符串 → 文本输入 / 下拉枚举
  日期 → DatePicker / DateRangePicker
  数字 → NumberRange 输入
  枚举 → Checkbox / Radio / Tag 选择
  外键 → AsyncSelect（异步搜索选择）
```

### 20.2 筛选器布局模式

**20.2.1 工具栏筛选（Toolbar Filter）**

```text
列表页顶部的水平工具栏：
  [搜索框] [状态 ▾] [时间范围 ▾] [更多 ▾] [重置]

适用：高频条件能在目标视口完整显示、标签清楚且键盘/移动端路径可用；不按固定数量裁决
优点：一眼可见，无需展开
缺点：条件多时拥挤
```

**20.2.2 抽屉筛选（Filter Drawer）**

```text
点击"筛选"按钮 → 右侧抽屉展开
  包含所有筛选条件（可能很多）
  支持重置全部和逐项清除

适用：筛选条件多，或条件复杂（日期范围、价格区间、分类树）

数据模型推断：
  抽屉筛选 → 当前设计将筛选从主界面渐进披露；条件数量、移动适配、视觉简化或低频均是候选原因
  筛选条件有默认值 → 确认当前上下文存在默认查询意图；是否“常见”需使用数据/研究证据
```

**20.2.3 列内筛选（Column Filter）**

```text
列表列头有筛选图标：
  客户名称 [▼] → 点击展开筛选器
  手机号 [▼] → 点击展开筛选器

适用：用户需要按任意列快速筛选
优点：无需离开列表页
缺点：筛选条件不够丰富
```

**20.2.4 筛选器持久化**

```text
URL 参数化：
  ?status=active&dateFrom=2026-01-01&page=1
  筛选条件在 URL 中，可分享、书签、浏览器前进后退

LocalStorage：
  记住用户上次使用的筛选条件
  下次进入自动恢复

数据模型推断：
  刷新后筛选条件保留 → URL、浏览器存储、服务端偏好或默认配置均是候选
  不同用户条件不同 → 可能来自用户偏好、角色默认、租户配置或历史会话
```

### 20.3 排序模式

**20.3.1 单字段排序**

```text
点击列头 → 升序
再次点击 → 降序
再次点击 → 取消排序

UI 推断：
  列头有排序图标（↑↓）→ 该字段支持排序
  默认排序 → 当前视图有默认次序；可由后端、搜索引擎或客户端产生
  用户可选排序 → 观察 URL、请求参数和本地状态后确认传递方式
```

**20.3.2 多字段排序**

```text
先按状态排序，再按创建时间排序
  状态降序（异常优先）+ 创建时间降序（最新优先）

UI 实现：
  拖拽调整排序优先级
  或：排序字段列表（可添加/移除/调整顺序）
```

**20.3.3 排序与数据模型**

```text
UI 排序 → 数据模型映射：
  UI 排序语义 → API/查询层需提供稳定、可分页的排序契约
  实现可使用数据库排序、搜索引擎、预计算键或应用层排序；以数据量和执行计划选择

富文本、文件和 JSON 也可按提取键、元数据或表达式排序，但必须先定义用户可理解的排序语义；不能笼统写成数据库无法排序。
```

## 21. 数据可视化模式

数据可视化是将数据模型转化为人类可理解图形的方式。本章覆盖图表类型与数据模型的映射关系。

### 21.1 图表类型与数据模型

**21.1.1 维度与度量**

```text
维度（Dimension）：用于切片、分组、过滤或定位观察粒度的语义属性；可以是计算维度
  例：日期、产品类别、地区、客户类型
  → 可来自枚举、引用、日期派生、层级、表达式或语义层，不等同某种存储类型

度量（Measure）：按明确粒度与聚合规则计算的量；可加、半可加或不可加
  例：订单金额、订单数量、利润率
  → 可来自事实值、事件、表达式或指标服务；利润率等比率不能按普通数值直接求和

图表选择还取决于用户问题、数据类型/分布、样本量、排序、时间连续性、可访问性和误导风险；维度/度量数量只用于生成候选，不能单独决定图表。
```

**21.1.2 图表类型选择指南**

| 图表类型 | 维度要求 | 度量要求 | 回答的问题 | 数据模型映射 |
|---|---|---|---|---|
| 折线图 | 有序/连续 X 轴（常为时间） | ≥1 个度量 | 趋势变化 | 有序点序列；不要求物理时间字段 |
| 柱状图 | 分类或离散分桶 | ≥1 个度量 | 对比大小 | 分组/分桶结果，可由多种执行层产生 |
| 饼图/环形图 | 分类维度 | 1 个度量（正值） | 占比 | GROUP BY + 百分比计算 |
| 散点图 | 2 个度量（X/Y） | 1-2 个度量（大小/颜色） | 相关性 | 全量或聚合数据点 |
| 热力图 | 2 个分类维度 | 1 个度量 | 密度/强度 | 双维度聚合 |
| 漏斗图 | 有序分类维度 | 1 个度量 | 转化流失 |  funnel 数据（每步的度量） |
| 仪表盘 | 单一度量 | 1 个度量 | 达标状态 | 单值聚合（COUNT/SUM） |
| 树图/旭日图 | 层级维度 | 1 个度量 | 层级占比 | 层级契约；可来自路径、边集、语义层或嵌套载荷 |
| 河流图 | 时间 + 分类 | 1 个度量 | 成分变化趋势 | 多维度时间序列 |
| 箱线图 | 分类维度 | 1 个度量 | 分布特征 | 需要原始数据或五数概括 |

**21.1.3 图表 → 指标契约候选**

```text
折线图存在 → 确认图表表达有序 X 轴上的变化；X 轴通常是时间，也可能是其他连续序列
  例：每日订单量趋势 → 确认“日”和“订单量”指标口径，数据可能来自实时查询、指标服务或预计算

饼图存在 → 确认当前视图表达部分与整体；分类字段和计算口径需查图例、tooltip 与 API

漏斗图存在 → 确认产品声明有序步骤及度量；不能直接推出单一 funnel_stage 字段

仪表盘存在 → 确认 UI 展示指标；KPI 定义、时区、过滤、去重、迟到数据和数据源必须逐项验证
```

### 21.2 图表交互模式

**21.2.1 钻取（Drill-Down）**

```text
点击图表元素 → 深入下一层详情
  客户地区分布（全国）→ 点击"华北"→ 华北各省分布
  → 点击"北京"→ 北京各区分布

数据模型推断：
  钻取存在 → 确认分析维度有导航层级
  自引用、层级维度表、路径编码或语义模型均是候选
```

**21.2.2 联动筛选（Linked Filtering）**

```text
图表之间的联动：
  饼图点击"活跃客户"→ 其他图表自动过滤为活跃客户数据
  例：仪表盘所有卡片都反映"活跃客户"这一个筛选上下文

数据模型推断：
  联动存在 → 确认图表共享当前筛选语义
  筛选上下文可在前端状态、URL、查询会话或服务端语义层，需观察请求和刷新行为
```

**21.2.3 图表缩放与范围选择**

```text
时间范围选择：
  折线图可拖拽选择时间段 → 放大查看局部细节
  迷你图 + 全局视图 → 上下对比

数据模型推断：
  范围选择存在 → 确认 UI 可改变观察窗口；可能对已加载数据缩放，也可能重新请求
  自动切换粒度 → 形成客户端或服务端粒度选择候选；通过请求参数与载荷验证
```

### 21.3 看板/仪表盘模式

**21.3.1 仪表盘布局策略**

```text
Grid 布局系统：
  仪表盘被划分为等宽列（12 列或 6 列）
  每个卡片占据 2/3/4/6/12 列
  卡片高度根据内容自适应

典型布局：
  ┌────┬────┬────┐
  │ KPI │ KPI │ KPI │  ← 3 个等宽 KPI 卡片
  ├─────────┬────┤
  │ 折线图   │ 饼图│  ← 1 个宽图 + 1 个窄图
  ├─────────┴────┤
  │ 列表/排行榜  │  ← 全宽内容卡片
  └──────────────┘
```

**21.3.2 看板可配置化**

```text
用户可自定义看板布局（拖拽调整大小和位置）
候选逻辑模型（也可存为配置文档、浏览器偏好或外部个性化服务）：
  dashboard_widget(user_id, dashboard_id, widget_id, config)
  widget_config {col, row, width, height, config_json}

用户可保存多个看板（销售看板 / 运营看板 / 我的看板）
候选逻辑模型：
  dashboard(user_id, name, is_default)
```

### 21.4 BI 与报表的数据模型

**21.4.1 星型与雪花模型**

```text
星型模型（Star Schema）：
  事实表（Fact）居中 + 多个维度表（Dimensional）围绕
  事实表：订单事实表（订单金额、成本、数量、时间、外键...）
  维度表：日期维度、产品维度、客户维度、渠道维度

雪花模型（Snowflake Schema）：
  维度表进一步规范化，拆分为多层
  产品维度 → 产品类别维度 → 产品大类维度

报表工具的映射：
  许多分析数仓和语义模型采用星型建模，但 Tableau / Power BI / Metabase 也可连接其他模型；工具存在不证明底层是星型
  维度表 = 筛选/分组的字段
  事实表 = 度量来源
```

**21.4.2 聚合表（Materialized View / 汇总表）**

```text
预计算常用的聚合结果，存储为独立表：
  日销售汇总表（daily_sales_summary）：
    date, product_category, region, sales_amount, order_count

查询时直接读汇总表，避免每次实时聚合：
  性能收益必须通过当前数据量、刷新策略和查询计划基准测量，不预设固定提升

数据模型推断：
  仪表盘加载快 → 候选包括缓存、预聚合、小数据、并行请求、SSR 或高性能实时查询
  数字与明细一致 → 只证明当前样本口径一致；预聚合及同步机制需查接口、血缘和更新延迟
```

## 22. 移动端与多端适配模式

现代企业软件通常需要同时支持桌面端、移动端、甚至小程序端。本章覆盖多端适配的设计模式和推断方法。

### 22.1 多端策略

**22.1.1 响应式 vs 自适应**

```text
响应式（Responsive）：
  同一套代码，根据屏幕宽度自动调整布局
  CSS Media Query 控制断点
  例：Ant Design Grid / Bootstrap Grid

自适应（Adaptive）：
  多套独立的 UI（Desktop / Tablet / Mobile）
  服务器端根据 User-Agent 返回不同页面
  例：传统 PC 网站 + 独立 M 站

逆向判断：
  同一 URL 在不同视口内容变化 → CSS 响应式、客户端条件渲染、服务端设备适配、权限/配置差异均是候选
  手机端布局简化 → 确认移动投影不同；不证明共享或独立代码库
  手机端能力较少 → 候选包括任务裁剪、平台限制、渐进披露、版本/权限差异或独立实现；查资源、路由、发布物和契约区分
```

**22.1.2 移动优先设计**

```text
设计顺序：
  1. 先设计移动端（最受限）
  2. 再扩展到桌面端（更宽裕）

移动端的约束：
  屏幕空间有限 → 字段优先级更严格
  触摸交互 → 命中区域需满足所采用平台/无障碍基线；常见基线约 44 CSS px，但应引用项目标准并实机验证
  无 hover 态 → 状态用颜色/文字而非悬停效果
  键盘输入不便 → 减少文本输入，用选择器替代
  网络不稳定 → 需要离线能力
```

**22.1.3 PWA（Progressive Web App）**

```text
PWA 特征：
  - 可添加到主屏幕（App Icon）
  - 支持离线访问（Service Worker 缓存）
  - 推送通知
  - 后台更新

数据模型推断：
  可添加到主屏幕 → 形成 Web App Manifest / PWA 候选，需检查 manifest 和 service worker
  离线仍可查看 → 存在离线资源/数据来源；可能是 service worker、浏览器缓存或原生壳
  离线可修改并在联网后同步 → 存在本地变更队列与同步语义；IndexedDB 只是候选之一
```

### 22.2 移动端特有的 UI 模式

**22.2.1 手势交互**

```text
手势类型：
  - 滑动（Swipe）：删除/归档（列表项左滑）
  - 下拉刷新（Pull-to-Refresh）：刷新列表
  - 长按（Long Press）：显示上下文菜单
  - 双击（Double Tap）：缩放图表/图片
  - 拖拽排序（Drag & Drop）：移动列表项顺序

数据模型推断：
  列表项左滑显示删除/归档 → 确认存在手势入口；权威命令与撤销语义待验证
  下拉刷新 → 确认用户可主动重新获取/重算；不证明轮询、WebSocket 或其他实时机制
  长按显示菜单 → 确认存在上下文命令入口；是否还有不可见操作需完整枚举
```

**22.2.2 移动端表单**

```text
与桌面端的差异：
  双列表单 → 单列（屏幕宽度限制）
  大量字段 → 分步骤向导或折叠区块
  日期选择 → 原生 DatePicker（体验更好）
  文件上传 → 直接调用系统相机/相册
  签名 → 专用签名控件

优化策略：
  使用原生输入控件（input type="date" 等）
  减少输入，使用选择器代替文本输入
  利用手机传感器（扫码、GPS、NFC）
```

**22.2.3 移动端导航**

```text
底部 Tab 栏（移动端替代侧边栏）：
  ┌────┬────┬────┬────┐
  │首页│客户│订单│我的│
  └────┴────┴────┴────┘

汉堡菜单（侧边抽屉）：
  屏幕左上角点击 → 侧边导航抽屉滑出

数据模型推断：
  底部 Tab 4 个 → 确认该角色/版本/设备下有 4 个一级入口，不等于 4 个业务域
  汉堡菜单有更多选项 → 确认其他入口被折叠；频率、权限或空间约束均可能解释
```

### 22.3 跨端代码共享

**22.3.1 跨端技术方案**

| 方案 | 技术栈 | 特点 | 数据模型映射影响 |
|---|---|---|---|
| Web 响应式 | React/Vue + CSS | 一套前端按断点适配 | 可共享 API/领域契约，但各端 ViewModel 仍可能不同 |
| 跨端框架 | Taro / Uni-app | 共享部分代码并编译到多端 | 共享程度受平台能力、权限和交互差异约束 |
| Hybrid | WebView + Native | 可组合 Web 迭代与原生能力 | 共享边界由桥接、安全、离线和性能约束决定 |
| React Native | React Native | 原生组件渲染 | 可共享部分 TypeScript/业务契约；模型和组件共享程度取决于架构 |
| Flutter | Flutter | 自有渲染与跨端工具链 | 可共享 Dart 层代码；领域/API 模型是否共享仍由多端任务与契约决定 |

**22.3.2 多端数据一致性**

```text
核心挑战：
  各端必须遵守同一业务语义和授权规则；因任务、延迟和离线状态不同，可显示不同投影与新鲜度
  任何一端的权威变更应按明确的一致性 SLA 被其他相关端感知，不要求瞬时相同

解决策略：
  契约治理：共享稳定的领域语义；各端可用共享 API 或各自 BFF/ViewModel
  明确 Source of Truth、版本策略、同步游标、冲突规则和数据新鲜度
  需要低延迟感知时：按单/双向通信、移动网络、认证、重连、背压和运维约束评估 WebSocket / SSE / 轮询；不要求推送到无关端
  乐观更新：各端独立更新，本地优先 + 服务端校验
```

## 23. 核心要点总结

```text
方向 A：
  UI 直接证明当前上下文中的可见事实，API 直接证明契约；二者都不自动证明领域或物理存储
  先写原子观察，再用溯因推理生成竞争性候选，以区分性实验逐层验证
  负面结论必须限定 Role/Permission/State/Data/Viewport/Flag/Version 范围

方向 B：
  先有角色、任务、命令、状态、不变量和平台约束，再让候选模型参与 UI 投影
  聚合根约束命令和一致性，不机械等于独立管理页面
  前端校验负责反馈，API/领域/数据库和授权层负责权威约束

双循环：
  竞品发现循环与自产品设计循环隔离，自生成 UI 不得自证模型正确
  目标是语义可追溯、行为可验证和合法信息损失有说明，不是各层结构一一相同
  最终产出包含已验证知识、竞争性候选、被驳回假设、evidence_insufficient 和下一步验证队列

思考底线：
  证据上限 + 推断距离 + 可证伪性 + 信息价值 + 因果过程 + 反事实 + 往返一致性
  再用身份、约束、时间、权威、一致性、隔离和演进七透镜检查模型完整性
```
