## 术语表

| 术语 | 全称 / 含义 |
|---|---|
| SDLC | Software Development Life Cycle，软件开发生命周期，从需求到运营的全过程阶段划分 |
| BRD | Business Requirements Document，业务需求文档，回答"为什么做、做给谁、商业目标" |
| PRD | Product Requirements Document，产品需求文档，描述产品功能范围与用户需求 |
| SRS | Software Requirements Specification，软件需求规格说明书，定义系统必须提供的可测试、可追溯的功能与非功能特性 |
| RTM | Requirements Traceability Matrix，需求追溯矩阵，建立需求到设计、代码、测试用例的双向追溯 |
| ADR | Architecture Decision Record，架构决策记录，记录一个设计决策的背景、候选、取舍与结论；accepted 状态只在循环 2 产生 |
| DoD | Definition of Done，完成定义，一份产出被视为完成必须满足的可检查条目集合 |
| PoC | Proof of Concept，概念验证，用最小实现证明某方案在关键路径上可行 |
| Spike | 时间盒探查，在固定时间内对高风险未知项做调研或原型，到期必须输出结论而非产品代码 |

## 1. 总体架构：双循环 + 横切质量体系

整个研发体系由两条循环和一套横切质量体系构成：

```text
循环 1：知识供给循环（知识库建设与持续完善）
循环 2：产品交付循环（经典 SDLC：需求 → 设计 → 实现 → 验证 → 运营）
横切：质量保障体系（对循环 2 的每个产出统一适用）

接口 1（供给）：知识库 → 最佳设计决策输入包 → 产品交付各阶段
接口 2（回流）：产品复盘 → 证据回写 → 知识库
```

```text
┌─────────────── 循环 1：知识供给循环 ───────────────┐
│ 竞品/开源证据采集 → 领域抽象 → 能力矩阵 → 候选输入  │
│ 行业标准与最佳实践采集 → 失败模式与设计约束清单      │
└──────────────┬──────────────────▲──────────────────┘
               │ 最佳设计决策输入包 │ 证据回写
               ▼                  │
┌─────────────── 循环 2：产品交付循环（V 模型）──────┐
│ P1 业务需求 → P2 需求规格 → P3 架构设计 → P4 详细设计│
│ → P4a 实现/单测 → P4b 集成测试 → P5 系统测试        │
│ → P6 验收/上线运营 → P7 复盘与回写                  │
└────────────────────────────────────────────────────┘
横切：产出质量保障工作流 ＋ 统一设计决策工作流 ＋ 追溯矩阵（RTM）＋ 变更控制
```

## 2. 职责边界

### 2.1 循环 1：知识供给循环只做三件事

```text
采证据：竞品官网 / 截图 / 实测 / 源码 / 社区 / 行业标准 / 最佳实践
做抽象：领域对象、流程、状态机、规则、权限、UI 模式、指标、非功能、失败模式
出候选：能力矩阵、优缺点、机会清单、候选方案对照、最佳设计决策输入包
```

知识库阶段**不为任何一个具体新软件做最终设计决策**：

```text
不产生 accepted 状态 ADR
不定稿具体产品的架构设计、详细设计、UI 规范或组件规范
不生成具体产品的需求范围
```

知识库的出口只有一个：**最佳设计决策输入包**（候选输入 + 证据 + 行业对照 + 风险 + 待验证项）。

执行文档：《01-方法论-多竞品逆向学习与产品设计知识库.md》《01A-工作流-知识库建设与持续完善.md》。

### 2.2 循环 2：产品交付循环拥有全部决策权

所有"为这个产品做取舍"的活动都发生在循环 2：

```text
需求取舍（做什么、不做什么）
架构决策（accepted ADR 只在这里产生）
数据模型定稿
功能设计定稿
UI 规范与组件规范定稿
```

执行文档：《02-流程-知识库驱动新产品创建与迭代.md》。

阶段模型的唯一口径是 P0-P7。《05-研发流程规范.md》把 P2-P5 展开为 11 个工程活动，用于安排交付物和门禁，但不另建产品生命周期；《06-规范-AI_UI质量保障流程规范.md》只在产出涉及 UI/交互时作为专项子流程嵌入 P4 详细设计、实现和验证。发生编号或门禁冲突时，先按 P0-P7 判定所处阶段，再应用 05 的工程活动和 06 的专项门禁；专项规范可以加严，不得放宽总纲门禁。

### 2.3 失败模式与设计约束清单：两条循环共用的风险资产

```text
循环 1 生产：竞品缺点、源码边界、错误提示、issue、事故线索 → 失败模式候选卡
循环 2 消费：架构评审、设计评审、测试设计必须对照失败模式清单逐项检查
循环 2 回流：上线后事故与缺陷 → 新失败模式 → 回写清单
```

执行文档：《03-方法论-系统级失败模式与设计约束清单.md》《03A-方法论-失败模式与设计约束清单建设.md》《04-清单-通用系统级失败模式与设计约束.md》《04A-清单-复杂平台级失败模式与设计约束.md》。

### 2.4 回写接口规范（P7 → 知识库）

P7 复盘阶段的证据回写是接口 2（回流）的落地机制，必须满足以下规范，确保回流产物可被知识库自动接收、校验和入库。

```text
1. 触发条件
   - P7 复盘完成且产出《复盘报告》并经评审通过。
   - 出现以下任一情形即触发回写：
     a. 上线后出现事故或缺陷，产生新的失败模式或设计约束。
     b. 验证结果与知识库推断不一致，需要纠偏已有知识卡片。
     c. 复盘发现新的竞品证据、行业最佳实践或设计模式可沉淀。
     d. ADR 状态发生变更（如 proposed → accepted / rejected / validated）。

2. 数据格式
   - 回写产物必须是符合《01-方法论-多竞品逆向学习与产品设计知识库.md》14.5 节 Schema 的独立 Markdown 文件，文件头部使用 YAML frontmatter。
   - 证据回写使用 14.5.1 证据卡片 Schema；知识纠偏使用 14.5.2 普通知识卡片 Schema；决策回写使用 14.5.3 ADR 卡片 Schema。
   - 三类卡片不混用；普通知识卡片必须填写 strength 和 confidence；ADR 卡片必须填写 decision_status，不得填写 strength。
   - 回写卡片须在 frontmatter 中标注 source_channel: self-trial 或 post-mortem，source_owner 标注实际来源团队，便于与竞品证据区分。

3. 目标存储路径
   - 回写卡片统一存入知识库对应领域目录下：
     知识库根目录/<领域对象>/cards/<卡片ID>.md
   - 失败模式与设计约束回写存入：
     知识库根目录/_failure-modes/<领域对象>/FM-<序号>.md
   - 回写时必须在知识库的 RTM 索引中登记追溯关系（复盘报告编号 ↔ 卡片 ID）。

4. CI 校验规则
   - Schema 校验：frontmatter 字段必填项、枚举值、ID 前缀命名规则必须通过校验脚本（参见 01 文档 14.8 节 KB-as-Code 自动化质量门禁）。
   - 证据链校验：普通知识卡片和 ADR 卡片的 evidence 字段引用的证据卡片必须存在，不得引用不存在的 ID。
   - 置信度校验：回写卡片的 confidence 值必须符合 2.2 节交叉验证规则（单一来源 ≤ 0.5，两个独立来源 ≤ 0.7，三个及以上或实测验证方可 ≥ 0.9）。
   - 覆盖度门禁：回写后知识库成熟度不得下降；若回写导致已有卡片被标记 pending-review，必须同时提交纠偏说明。
   - 校验失败的回写不得入库，CI 必须输出失败原因和修复建议，禁止绕过校验直接提交。
```

## 3. 循环 2：V 模型产品交付流程

### 3.1 阶段结构

左臂逐层分解，右臂逐层验证；每个左臂产出在产生时就必须定义右臂对应层级的验证标准。

```text
左臂（分解）                              右臂（验证）
P1 业务需求（产品蓝图 / BRD）    ←验收测试→  P6 验收与上线运营
P2 需求规格（PRD / SRS）        ←系统测试→  P5 系统测试
P3 架构设计（架构+数据模型+ADR） ←集成测试→  P4b 集成测试
P4 详细设计（模块+接口+UI+组件） ←单元测试→  P4a 实现与单元测试

P7 复盘与知识库回写（闭环回循环 1）
```

### 3.2 需求分三层

"功能设计"属于需求工程，必须与架构设计分离：

| 层级 | 产物 | 回答的问题 |
|---|---|---|
| 业务需求 | 产品蓝图 / BRD | 为什么做、做给谁、商业目标、成功指标 |
| 用户需求 | 用例 / 用户故事 | 用户在什么场景下要完成什么任务 |
| 需求规格 | PRD / SRS | 系统必须提供哪些功能和非功能特性，可测试、可追溯 |

规则：

```text
下层需求必须能追溯到上层需求。
每条需求规格必须附带验收准则（写需求的同时写验证标准）。
非功能需求（性能、安全、可用性、可观测、合规）与功能需求同等强制。
```

### 3.3 设计分两层

| 层级 | 产物 | 决策内容 |
|---|---|---|
| 架构设计（P3） | 架构视图、数据模型、accepted ADR | 系统怎么分解、质量属性怎么满足、单向门决策 |
| 详细设计（P4） | 模块设计、接口契约、UI 设计、UI 组件设计 | 每个模块内部怎么实现、界面怎么呈现 |

规则：

```text
P3 未通过架构评审，P4 不得开工。
架构决策必须对照失败模式与设计约束清单逐项检查。
UI 设计和 UI 组件设计属于详细设计，必须走统一设计决策工作流。
```

## 4. 横切一：产出质量保障工作流

循环 2 中任何阶段的任何产出，一律走同一条五步流水线：

```text
1. 入口检查：上游基线产物齐备且已通过评审；无基线不开工。
2. 生产：按模板生产；产出必须引用上游追溯编号。
3. 自检：对照该产物的 DoD 检查单（完整性 / 一致性 / 可测试性 / 可追溯性）。
4. 评审：AI 预审 + 人工评审 → 通过 / 有条件通过 / 打回。
5. 基线：通过后纳入基线；之后修改只能走变更流程；同步更新追溯矩阵。
```

配套机制：

```text
追溯矩阵（RTM）：
业务需求 ↔ 需求规格 ↔ 架构/ADR ↔ 详细设计 ↔ 代码 ↔ 测试用例。
任何一层变更必须先查 RTM 波及面，再改动。

变更控制：
基线后的修改必须走：变更申请 → 影响分析 → 评审 → 更新基线 → 更新 RTM。
禁止绕过变更流程直接改基线产物。
```

门禁：

```text
没有通过五步流水线的产出，不得作为下游阶段的输入。
没有登记 RTM 的产出，不得声明完成。
```

### 4.1 DoD 检查单通用模板

循环 2 中所有产出在自检阶段（五步流水线第 3 步）必须对照以下四维度检查单逐项核验。各阶段可在此基础上扩展阶段专属条目，但不得删除以下通用条目。

```text
维度一：完整性
  [ ] 所有必需章节/字段均已填写，无空缺项。
  [ ] 上游基线产物齐备且已通过评审（入口检查通过）。
  [ ] 非功能需求（性能、安全、可用性、可观测、合规）已覆盖，无遗漏。
  [ ] 失败模式与设计约束清单已逐项对照检查并记录结论。
  [ ] 行业标准与最佳实践对照表已建立（设计类产出适用）。

维度二：一致性
  [ ] 产出内容与上游基线无矛盾；如发生变更，已走变更流程。
  [ ] 术语、命名、编号在文档内前后一致，与知识库命名规范一致。
  [ ] 数据模型、接口契约、状态机在文档内自洽，无冲突定义。
  [ ] RTM 追溯关系已更新，上下游编号可双向查询。

维度三：可测试性
  [ ] 每条需求规格附带验收准则，验收准则可被执行测试。
  [ ] 设计中的每个接口、状态、规则均可映射到至少一条测试用例。
  [ ] 非功能需求有可量化的验证标准（如响应时间阈值、并发数）。
  [ ] 假设和未验证项已显式标注，并附验证计划或 Spike 计划。

维度四：可追溯性
  [ ] 每条需求可追溯到上游业务需求（向下追溯）。
  [ ] 每条设计决策可追溯到知识库证据卡片和/或行业最佳实践来源。
  [ ] 每个产出在 RTM 中已登记，编号唯一且可检索。
  [ ] ADR 的 evidence 字段引用的证据卡片均存在且未失效。
```

规则：

```text
四维度全部通过方可声明完成；任一维度有未勾选项，产出标记"待完善"，不得作为下游输入。
自检结果必须随产出一同提交，禁止只提交产出不提交自检单。
```

## 5. 横切二：统一设计决策工作流

架构设计、功能设计、数据模型、UI 设计、UI 组件设计五类设计问题，共用同一个决策工作流：

```text
1. 定义问题：设计问题陈述 + 质量属性场景 + 约束条件。
2. 收集候选：三路候选，候选总数 ≥ 2——
   a. 知识库最佳设计决策输入包（竞品证据支撑的候选）
   b. 行业标准与最佳实践（独立于竞品的行业基准）
   c. 自主创新方案（至少提出一个试图超越现有候选的方案）
3. 权衡分析：权衡矩阵（收益 / 代价 / 风险 / 失败模式 / 回滚成本 / 演进成本）。
4. 降险验证：高风险候选做 Spike / 原型 / PoC，用证据说话。
   4a. License 合规检查与专利排查（必要子步骤，不得跳过）：
       - 对候选方案涉及的第三方库、框架、组件、开源代码，逐一核查 License 类型（MIT / Apache-2.0 / GPL / LGPL / BSD 等），
         判断是否与自身产品的发布模式和商业策略兼容。
       - 对候选方案涉及的核心算法、数据结构、交互模式、UI 设计，进行专利排查，
         检索是否有已授权专利构成侵权风险；排查范围至少覆盖目标市场的主要专利数据库。
       - License 判断属于"必须人工决策"动作等级（见 6.2 节），AI 只准备排查材料（License 清单、专利检索结果、风险提示），
         停下等人确认；禁止 AI 自行做出 License 可用或专利安全的结论。
       - 排查结论写入 ADR 的风险段落；存在不可接受风险的候选直接淘汰，不进入决策归档。
5. 决策归档：ADR（accepted 状态只在此处产生），写明每个落选候选的拒绝理由。
6. 基线与追溯：决策纳入基线，登记 RTM。
```

创新保障规则：

```text
不许只抄竞品：候选中必须有行业标准对照。
不许只凭经验：候选中必须有知识库证据支撑的方案。
必须尝试创新：候选中必须有至少一个自主创新方案参与比较；
             创新方案落选必须写明拒绝理由，不得默认淘汰。
```

## 6. AI 自动执行契约

本目录所有工作流和流程文档的第一读者是 AI 执行者。用户把文档交给 AI 后，AI 必须能自动执行工作流并产出结果，不依赖人工逐步指挥。因此所有工作流文档必须满足以下契约：

### 6.0 AI 优先阅读协议

AI 不应默认把整份长文一次性装入上下文。统一采用渐进式披露：

```text
第一层：先读文档头、AI 执行入口、适用边界、完成定义。
第二层：只读当前阶段及其直接引用的规则、模板和清单。
第三层：发生冲突、缺证或高风险决策时，再回查上位规范和证据原文。
```

规则优先级固定为：总纲 > 专项方法论 > 执行工作流 > 项目实例 > 临时提示。下位规则可以加严，不得放宽上位硬门禁；冲突时必须输出冲突位置、采用规则和理由，禁止静默选边。

### 6.1 可执行性要求

```text
执行入口明确：每份工作流文档必须有"执行入口"章节，说明启动条件和启动提示词。
最小输入契约：明确用户最少提供什么（如：知识库名称 / 产品名称），其余判断由 AI 自行完成。
分支自判定：新建还是完善、进入哪个阶段，由 AI 根据现状检查自行判定，不得反问用户。
步骤确定性：每个阶段必须有明确的输入、动作、输出文件路径和完成判据。
门禁可自检：所有门禁必须写成 AI 可自行核对的检查单，而不是需要主观判断的口号。
产出可验收：每个产出有 DoD 检查单和模板；AI 完成后必须逐项自检并输出自检结果。
```

### 6.1.1 AI 通过结构化检索访问知识库

AI 执行者在循环 2 各阶段需要从知识库获取候选输入时，必须通过结构化检索访问，不得仅凭通读文档获取知识。知识库的卡片均带 14.5 节 Schema 定义的 YAML frontmatter 元数据，AI 应按以下方式检索：

```text
检索层级（按需逐层增强，详见 01 文档 14.7 节）：
1. 元数据过滤：按 domain_object、type、strength、competitors、status 等字段精确过滤。
   示例：查询 domain_object=Order 且 type=state-machine 且 status=active 的全部卡片。
2. 全文检索：在卡片正文范围内做关键词检索，补充元数据过滤无法覆盖的语义。
3. GraphRAG：沿知识图谱的 evidence（支持）边和 links（引用）边扩展关联卡片，
   把证据链和影响链一并纳入上下文。
4. 影响分析查询：当某证据被纠偏时，沿"支持"边查出所有受影响的推断和 ADR。

检索规则：
- AI 访问知识库时必须同时返回卡片正文和 frontmatter 元数据，不得只取正文。
- 检索结果中每条知识必须带证据强度（strength）和置信度（confidence）标注，
  AI 在引用时必须如实传递标注，禁止抹除强度信息后当成等可信事实使用。
- 覆盖度要求：对当前设计问题涉及的 domain_object，AI 必须检索该对象下全部 type 的卡片，
  不得只检索一种类型就声称"已参考知识库"。
- 检索无结果时，AI 必须显式写入缺口清单，标注"知识库无对应证据"，不得静默跳过。
```

### 6.2 AI 执行的三种动作等级

| 等级 | 含义 | 示例 |
|---|---|---|
| 自动执行 | AI 直接做，做完报告 | 证据采集、卡片草案、矩阵初稿、自检、RTM 更新 |
| 执行后送审 | AI 做完，产出标记"待评审"，等人工评审通过才能作为下游输入 | PRD、架构设计、详细设计、UI 规范 |
| 必须人工决策 | AI 只准备决策材料，停下等人 | 需求取舍、accepted ADR、风险接受、上线放行、License 判断 |

每份工作流文档中的每个步骤必须能归入以上三级之一。AI 遇到"必须人工决策"的点必须停下并输出决策材料（候选、证据、权衡、建议），禁止代替人做决策后继续执行。

### 6.3 AI 执行的通用禁令

```text
禁止跳过门禁：门禁未通过不得进入下一阶段，不得"先做后补"。
禁止虚报完成：没有落到文件的产出不算完成；声明完成前必须输出产物路径清单。
禁止把推断当事实：所有结论必须带证据强度标注。
禁止绕过人工决策点：等待确认时只能停止或处理其他不依赖该决策的任务。
禁止静默降级：某个要求做不到时必须显式写入缺口清单，不得静默跳过。
```

### 6.4 断点续跑

AI 执行可能被中断。所有工作流必须支持从中断点恢复：

```text
每个阶段完成时，阶段产物即是执行状态的持久化记录。
恢复执行时，AI 先扫描已有产物，判定当前处于哪个阶段，
再从第一个未完成的阶段继续，不得重做已通过门禁的阶段。
```

### 6.5 AI 原生执行模型

所有可执行规范必须实现以下模型；方法论文档和清单也必须提供对应的 AI 读取与产出协议。

| 能力 | 强制要求 |
|---|---|
| 规范即策略 | `必须/禁止/仅当` 是硬约束；`建议/可以` 是软约束。AI 不得把软约束提升为门禁，也不得把硬约束降为建议。 |
| 确定性状态机 | 每阶段只能使用 `not_started / in_progress / blocked / awaiting_human / passed / failed / superseded` 状态；必须声明合法迁移。 |
| 幂等执行 | 每轮生成 `execution_id` 和 `idempotency_key`；同一输入基线和规则版本重复执行时复用已通过产物，不重复产生副作用。 |
| 证据-推断-决策分离 | 原始事实、AI 推断、人工/流程决策必须分字段记录；无证据不得写成事实，AI 不得把建议写成 accepted 决策。 |
| 反例优先 | 声明通过前，先主动寻找至少一个反例、边界条件或失败路径；找不到证据时输出 `evidence_insufficient`，不得猜测通过。 |
| 上下文包 | 每阶段只接收版本化输入清单、规则版本、证据引用、未决问题和上游结论，禁止依赖未记录的对话记忆。 |
| 结构化升级 | 阻塞或需人工决策时，输出问题、已验证事实、候选方案、权衡、建议、缺失权限/信息和恢复条件。 |
| 可复现验证 | 完成声明必须附验证命令或检查步骤、时间、输入基线、结果和未覆盖风险；没有新鲜证据不得声明完成。 |
| 例外有期限 | 豁免必须记录 owner、原因、影响、补偿控制、到期时间和复审条件；过期自动转为 blocked。 |

每轮执行至少维护以下结构化状态；可保存为 JSON/YAML，也可嵌入项目既有状态文件：

```yaml
execution_id: EXE-YYYYMMDD-NNN
spec_id: 文档编号或稳定文件名
spec_version: 版本号或文件哈希
input_baseline: 输入文件及哈希
idempotency_key: spec_id+spec_version+input_baseline
stage: 当前阶段
status: in_progress
facts: []
inferences: []
decisions: []
evidence_refs: []
open_questions: []
next_action: 下一条可执行动作
verification: []
residual_risks: []
```

### 6.6 文档类型的 AI 合格标准

| 文档类型 | 必须具备 |
|---|---|
| 总纲 | 规则优先级、统一状态、动作等级、冲突处理、执行记录 Schema |
| 方法论 | AI 读取入口、适用性判断、输入、推理步骤、输出结构、人工复核点、禁止事项 |
| 工作流/流程 | 最小输入、分支判定、阶段状态机、逐步输入/动作/输出/完成判据、断点与升级条件 |
| 清单 | 适用性筛选算法、逐项回答格式、证据要求、阻断规则、汇总输出 |
| 专项规范 | 上游/下游映射、模式选择规则、可自动检查项、人工判断项、证据包和退出门禁 |

任何缺少上述必需项的文档只能标记为 `draft`，不得被 AI 用作自动放行依据。

## 8. 一句话压缩

用循环 1 把竞品与行业知识加工成带证据的候选输入，用循环 2 按 V 模型把候选输入转化为经过分层验证的产品，用横切质量体系保证每一个产出都过五步流水线、每一个设计决策都经过三路候选比较，最后用复盘回写让两条循环互相喂养、持续变强。

## 9. 横切方法论加强项

以下方法论思想横跨两条循环，适用于本体系全部文档。

### 9.1 成熟度门禁联动

知识库成熟度（01 文档 10.1 节 L0-L3）不是声明性约束，而是执行性门禁。AI 执行者在进入 P3 架构设计前，自动检查知识库对应模块的成熟度是否达到 L1，未达标则阻断并输出差距清单。这把"知识库够不够好"从人工判断变成流程门禁。

### 9.2 失败模式覆盖率作为验收门禁

失败模式与设计约束清单（04/04A）不是参考材料，而是验收门禁。架构设计文档必须逐条回应清单中的每一项（采用/规避/接受风险），测试用例必须覆盖每一条失败模式。未覆盖的条目在 CI 中自动标记为 gap。

### 9.3 KB-as-Code 推广到全部产出

知识库入 Git 走 CI 校验（01 文档 14.8 节）的理念推广到循环 2 的全部产出：PRD 必须引用 RTM 编号、架构设计必须引用失败模式清单编号、测试用例必须引用需求规格编号。所有产出入 Git 并走 CI 校验，横切质量保障工作流的"五步流水线"可部分自动化。

### 9.4 增量升级 vs 推倒重建决策模型

01 文档 12.4 节的"浅则增量、错则重建"模型推广为通用的知识资产健康度评估框架，不仅用于知识库升级，也可用于代码库、架构文档、测试用例库等任意知识资产的质量评估和升级决策。

### 9.5 创新候选池与 revisit 机制

统一设计决策工作流的"三路候选"中被拒绝的创新方案不丢弃，进入"创新候选池"并标注拒绝原因和 revisit 条件（如"当用户规模超过 X 时重新评估"），CI 定时扫描创新候选池检查 revisit 条件是否触发。防止创新方案被永久埋没。

### 9.6 证据链→推断→决策的三层分离与反向影响分析

知识图谱（01 文档 14.6 节）不仅支持正向追溯（从决策回查证据），还支持反向影响分析：当一条证据被纠偏时，自动生成"影响传播树"（谁依赖了被纠偏的推断 → 谁依赖了那些依赖 → 直到叶子节点），让人工评审者快速判断纠偏波及范围。

### 9.7 质询式评审

设计评审和知识库评审中，评审方必须生成至少 5 个尖锐"为什么"质询，每个质询针对一个设计决策或证据推断的关键假设。这是对抗 AI 生成内容表面合理性的有效手段，防止"看起来都对但经不起追问"的产出通过门禁。

### 9.8 V 模型左右臂同步定义验证标准

左臂产出时同步定义右臂验证标准——设计即测试。P3 架构设计产出时定义 P5 系统测试的验收准则，P4 详细设计产出时定义 P4a 单元测试的覆盖要求。这确保每个设计产物都有可验证的退出条件，避免"设计完了但不知道怎么验收"。

## 10. 战略级元方法论

本章回答的不是“怎样多做一份文档”，而是“怎样让组织在不确定性中持续做出可纠正、可证明、可演进的决策”。每项思想必须落到产物、门禁和反馈，不得只作为评审口号。

### 10.1 控制论：把研发系统设计成可观测、可调节的闭环

研发流程本质上是一个控制系统：目标是期望状态，交付物是控制动作，验证与运行指标是传感器，复盘是反馈，流程调整是控制器。任何闭环都必须明确：

```text
目标变量：希望系统达到什么可测状态。
领先指标：问题发生前能否预警。
滞后指标：结果是否真的改善。
采样频率：多久获得一次可信反馈。
反馈延迟：从决策到看见效果需要多久。
控制动作：偏离目标时允许调整什么。
稳定边界：怎样防止频繁调整造成振荡。
```

门禁：关键目标没有指标、指标没有数据源、反馈周期长于决策有效期，或只能看到滞后指标时，不得声明形成闭环。连续两次同方向偏差时必须复审目标、模型或控制动作，而不是只加大执行强度。

### 10.2 社会技术系统：同时设计软件、组织与责任边界

技术方案不能假设人会永远正确执行。每个关键能力必须同时描述四个相互作用的层面：技术机制、业务流程、角色权限、激励与问责。设计评审必须询问：谁输入、谁批准、谁受影响、谁能绕过、谁发现异常、谁负责恢复。

强制产物为“社会技术责任图”，至少包含角色、权限、信息流、决策权、升级路径和职责冲突。出现同一角色既生产又放行、责任与权限不对等、失败无人承接或指标诱导错误行为时，阻断基线。

### 10.3 可逆性与期权价值：在不确定性中保留未来选择

决策不只比较当前收益与成本，还必须比较锁定成本、退出成本和未来选择权。ADR 增加以下字段：

```text
reversibility：easy / bounded / hard / irreversible
commitment_horizon：在何时之前仍可低成本改变
exit_cost：数据、合同、迁移、培训和停机代价
option_value：延迟承诺或保留兼容层能保留哪些未来选择
revisit_trigger：什么新证据出现时重新决策
```

原则：高不确定性优先选择可逆方案和小额真实实验；不可逆决策必须提高证据门槛并由人工确认。可逆性不是无限抽象，只有当退出成本已被识别且替代路径真实存在时才成立。

### 10.4 不变量驱动：先定义绝不能被破坏的事实

需求与设计不能只描述功能，还必须声明业务、安全、数据和运行不变量。例如“已结算金额守恒”“主体边界不可穿透”“同一意图最多产生一次不可逆副作用”。

每个关键不变量必须形成完整链路：

```text
invariant_id → 来源/理由 → 适用边界 → 可能破坏它的动作
→ 预防机制 → 检测机制 → 恢复机制 → 测试/监控证据
```

门禁：关键对象、状态机、权限边界或资金/库存类流程没有不变量；不变量没有破坏性测试；或者实现、数据约束和监控均未承接时，不得通过设计评审。

### 10.5 保证论证：用“主张-论据-证据”证明可以放行

测试数量不能证明系统足够可信。高风险能力必须建立轻量 Assurance Case（保证论证）：

```text
Claim：要证明的安全、质量或业务主张。
Argument：为什么现有设计和控制足以支撑主张。
Evidence：测试、分析、审计、演练、监控或形式化约束证据。
Assumption：论证依赖的前提。
Defeater：什么反例会推翻主张。
Residual risk：证据覆盖后仍保留的风险。
```

AI 可以生成论证草案和寻找 defeater，但不能批准高风险主张。上线门禁检查的是关键主张是否有足够证据以及反例是否已处理，而不是文档和用例数量是否达标。

### 10.6 证据保鲜：知识有效性随时间和环境衰减

证据不是永久真理。每条关键证据除采集时间外，还必须声明适用版本、环境、有效期和失效触发器。依赖升级、法规变化、用户群变化、规模跨阈值、架构边界变化或反例出现时，相关证据自动进入 `pending_review`，并通过反向影响分析标记受影响的推断、ADR、测试和运行门禁。

原则：证据过期不等于结论必错，但等于不能继续无条件支撑新决策。高风险决策引用过期或环境不匹配证据时必须阻断并补证。

### 10.7 认知债务：治理“团队以为自己知道什么”

除代码债务外，还要管理认知债务：隐含假设、术语歧义、无人能解释的决策、只存在于对话中的上下文、未验证但被反复引用的结论。统一登记：

```text
debt_id：唯一编号
unknown_or_assumption：未知项或假设
decision_at_risk：受影响决策
cost_of_ignorance：继续不知道的代价
resolve_by：最晚解决时间或触发条件
owner：责任人
resolution_evidence：关闭所需证据
```

当认知债务影响不可逆决策、安全边界、合规或上线放行时立即阻断；其他债务按“无知成本 × 发生概率 × 暴露时长”排序。复盘必须检查本轮是否减少了关键未知，而不只检查交付数量。

### 10.8 元方法论统一门禁

关键设计和发布评审增加以下七问，任一回答为“未知”时必须记录认知债务，涉及高风险时阻断：

| 维度 | 必答问题 | 主要产物 |
|---|---|---|
| 控制论 | 我们如何知道决策生效或失效，反馈是否足够快？ | 指标与反馈回路 |
| 社会技术 | 谁决定、执行、监督、受影响和恢复，是否职责冲突？ | 社会技术责任图 |
| 可逆性 | 决策何时锁定，退出成本和 revisit 条件是什么？ | ADR 可逆性字段 |
| 不变量 | 哪些事实绝不能被破坏，怎样预防、检测和恢复？ | 不变量追溯链 |
| 保证论证 | 放行主张由什么论据和证据支持，什么反例会推翻它？ | Assurance Case |
| 证据保鲜 | 证据是否仍适用于当前版本、环境和规模？ | 证据有效性记录 |
| 认知债务 | 哪些未知正威胁决策，何时、由谁、用什么证据关闭？ | 认知债务台账 |

AI 执行时将七项结果写入阶段状态的 `facts / inferences / decisions / open_questions / residual_risks`；不得另建互不追溯的孤立表格。小变更可标记不适用，但必须写理由。
