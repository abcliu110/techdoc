## 0. 一句话定义

用户只需用一句自然语言表达想做什么，AI 必须依据本元规范主动主持任务启动：先帮助用户拆解并确认目标、相关角色、真实场景或问题现象、范围和非目标，再主动发现或规划工作空间，确认需求基线、原型、源码、任务和证据分别放在哪里；上述启动输入确认后，AI 先确定需求基线类型并锁定任务级 DoR/DoD，再生成、评审、批准并登记 PRD 或等价需求基线，最后形成可供后续研发直接使用的启动包。

核心链路：

```text
一句话需求
→ AI 主持逐问式访谈
→ 目标、场景与边界确认
→ AI 发现并提出工作空间方案
→ 用户确认工作空间
→ AI 确定需求基线类型，并预先锁定质量合同、任务 DoR/DoD、评审人和批准人
→ AI 生成需求基线草案（适用时生成 PRD Draft）
→ 用户或有权负责人评审并批准一个明确版本
→ AI 核验批准版本和哈希，登记为权威需求基线
→ AI 形成后续设计、切片、实现与验收可直接使用的启动包
```

本元规范解决的是“用户还说不清、也不知道该提供哪些目录和文档时，AI 如何把任务正确接住”。它不要求用户预先理解 PRD、DoR、ADR、任务切片或项目目录等专业概念。

对用户而言，本元规范的核心交付只有三件事：

```text
1. 帮我把目标说清楚；
2. 帮我把本次做与不做的边界说清楚；
3. 帮我确定需求、原型、源码、任务和证据分别放在哪里。
```

适用完整 PRD 时，PRD 是这三项确认后由 AI 主持形成的后续产物，不是用户启动本元规范前需要准备的输入；但 AI 必须先锁定质量合同和任务 DoR/DoD，不能直接写 PRD。其他任务使用同样原则形成与任务类型匹配的等价需求基线。

### 0.1 执行导航：SOP、PRD 与 DoD 在哪里

本文不是只解释概念，而是把一句话需求变成受控研发任务的可执行规范。执行者应按以下入口使用，不得只阅读某个孤立章节：

| 需要解决的问题 | 执行位置 | 得到什么 |
|---|---|---|
| AI 收到一句话后按什么顺序工作 | 第 3.4、3.5 节“标准作业程序（SOP）” | M0-M8 的 DoR、动作、产物、阶段 DoD、失败处理和下一阶段 |
| 每个可独立执行的小任务怎样受控 | 第 3.6 节“独立任务切片合同” | 任务级 DoR、DoD、责任人、验证证据和失败处理 |
| 什么任务应该生成 PRD | 第 10.2 节“需求基线适用性判定” | `prd`、`incremental_prd` 或等价需求基线的选择结果 |
| PRD 应怎样生成和评审 | 第 10.3-10.10 节“PRD 工作包规则”及第 11 节“评审、批准与基线登记” | 预置质量合同、用户需求、PRD、RTM、自检、批准与基线记录 |
| PRD 的完成标准何时确定、何时算完成 | 第 10.3.1、10.3.2 节“PRD Quality Contract” | 草拟前锁定的 RB-1/RB-2/RB-3 DoD，以及整个 PRD 工作包的完成条件 |
| 本元规范什么时候算执行完成 | 第 12.5 节“任务启动 Definition of Done” | Task Startup Package 能否标记为 `passed` 的逐项判据 |

三者的关系是：

```text
SOP = AI 应按什么步骤做
PRD = 新产品或重大产品变更的权威需求基线
DoD = 用什么证据证明某阶段、任务切片或整个任务启动已经完成
Task Contract = 在小任务开始前把目标、DoR、DoD、责任人和证据锁定下来
```

本元规范中的最终 DoD 只证明“研发任务已经被正确启动并可移交下一活动”，不证明“产品已经开发、测试或上线完成”。后续架构、任务切片、实现、集成、系统验收和发布必须分别建立自己的 DoR、SOP 与 DoD。

### 0.2 输入与核心输出合同

```yaml
input:
  original_request: 用户的一句话需求

core_outputs:
  confirmed_goal_and_boundary:
    - problem
    - objective
    - affected_users_or_stakeholders
    - primary_scenarios_or_symptoms
    - scope
    - non_goals
  confirmed_workspace:
    - product_root
    - requirement_root
    - prototype_root
    - code_roots
    - task_root
    - evidence_root
    - knowledge_sources

outputs_after_confirmation:
  - user_requirements_when_applicable
  - requirement_quality_contract
  - authoritative_requirement_baseline
  - prd_draft_when_applicable
  - traceability_record
  - task_contracts
  - policy_baseline
  - baseline_manifest
  - next_activity_contract
```

执行顺序是硬约束：`confirmed_goal_and_boundary` 未通过，不得确认工作空间；`confirmed_workspace` 未通过，不得执行 RB-0；需求基线类型、适用策略和质量合同未预先锁定，不得草拟 PRD 或等价需求基线；需求基线未完成评审批准与基线登记，不得进入架构、正式切片、生产编码或生产发布。唯一例外是第 5.6 节定义的受控基线前 Spike。任何可独立执行的后续任务没有 Task Contract 或 DoR 未全部通过，也不得开始。

### 0.3 自包含原则与术语

本元规范必须能单独执行。运行时必需输入只有：本文、用户的一句话需求，以及 AI 在已授权工作区内只读取得的事实。其他方法、模板或流程可以作为参考，但不得成为完成 M0-M8 的硬依赖；外部资料不存在时，AI 仍应按本文完成启动，不得把寻找其他规范的责任交给用户。

AI 可以读取其他资料补充知识，但必须把采用的内容转化为本任务可追溯的事实、约束、候选、决策或验收项；不得把“请先阅读或执行某份外部流程文件”作为本元规范的输出、门禁条件或下一步动作。

| 术语 | 本文中的含义 |
|---|---|
| Goal and Scope Contract | 目标与边界合同；记录已确认的问题、目标、相关角色、场景或现象、范围、非目标、成功判断和决策人 |
| Workspace Contract | 工作空间合同；记录产品、需求、原型、源码、任务、证据和知识来源分别位于哪里及其职责 |
| PRD | 产品需求文档；适用于新产品或显著改变产品行为的任务，描述做什么、为什么做和怎样验收，不是技术设计 |
| SOP | 标准作业程序；把阶段的 DoR、规定动作、产物、阶段 DoD、异常处理和下一阶段组织为可重复执行的步骤 |
| RTM | 需求追溯矩阵；把目标、场景、需求、验收标准和证据关联起来，防止孤儿需求和漏验收 |
| 需求基线 | 当前任务经过批准并完成版本、哈希、权威位置和批准记录登记的需求输入；可以是 PRD，也可以是问题合同、变更规格、回归合同或研究合同 |
| Gate | 门禁；进入下一阶段前必须满足的可检查条件 |
| DoR | 开始条件；某项活动允许启动前必须具备的输入、责任人、边界和验收条件 |
| DoD | 完成条件；某项活动允许声明完成前必须具备的结果和验证证据 |
| Task Contract | 任务合同；在一个可独立执行和验收的任务切片开始前，锁定目标、范围、DoR、DoD、责任人、证据和失败处理 |
| Policy Baseline | 策略基线；冻结本任务必须遵守的组织规则、项目规则、安全/合规要求及其版本、优先级和适用理由 |
| Approval Manifest | 送审清单；记录提交人工批准的需求内容、追溯、质量合同和自检证据，不包含批准记录；批准人绑定其清单哈希 |
| Baseline Manifest | 基线清单；在 Approval Manifest 基础上加入批准记录，逐项记录权威产物的路径、版本和 SHA-256，并对清单本身计算哈希 |
| ADR | 架构决策记录；记录重大技术选择、备选方案、理由和后果，属于后续架构活动，不是启动前输入 |
| Spike/POC | 限时可行性实验；用于验证关键技术假设，不直接作为生产实现 |
| Characterization | 既有行为刻画；通过可重复观察或测试锁定旧系统实际行为和隐性契约 |
| Task Startup Package | 任务启动包；汇总已批准目标、边界、工作空间、需求基线、风险和下一活动入口条件 |

“流程路由”在本文中只表示判定下一类研发活动及其入口条件，不表示引用或强制跳转到某一份外部流程文件。

## 1. 用户零配置入口

### 1.1 最小输入

本规范的最小输入只有一项：

```yaml
original_request: 用户用自然语言表达的一句话需求
```

有效示例：

```text
请按照《AI研发任务单句启动与流程路由元规范》启动任务：
我需要设计一个企业表单设计器。
```

更短的表达也必须被接受：

```text
按照元规范启动：我要做一个企业表单设计器。
```

用户不需要预先提供：

- 完整需求；
- PRD；
- 产品目录；
- 代码仓库；
- 原型目录；
- 技术方案；
- 任务切片；
- 验收标准；
- 专业流程术语。

“一句话启动”不代表后续不需要用户参与。用户仍需回答 AI 每次提出的一个关键问题，并对目标、范围、工作空间和需求基线等重大决策作出确认；但用户不需要自己主持流程或一次性写长篇需求。

### 1.2 AI 收到一句话后的第一原则

AI 必须把用户的一句话视为“待发现的原始意图”，不能视为完整需求。

AI 不得立即：

- 生成正式需求基线或 PRD；
- 决定产品模型；
- 选择技术架构；
- 创建多个源码仓库；
- 进行正式任务切片；
- 编写生产代码；
- 把行业经验写成用户已确认需求。

AI 应立即进入 M0-M3 的目标与需求发现流程。

### 1.3 AI 的第一次响应

第一次响应必须简短、可回答，不得倾倒完整问题清单。标准结构：

```text
当前阶段：目标与需求发现

原始需求：
复述用户的一句话。

当前可确认的事实：
只写用户原话或已有权威资料能证明的内容。

我的初步理解：
标记为推断，不冒充确认结果。

当前禁止：
正式需求基线、架构、切片和编码。

第一个问题：
只问一个最能降低目标或边界不确定性的问题。
```

对于“企业表单设计器”，首问推荐为：

> 先不讨论功能清单。请描述一个你最希望它解决的真实工作：谁创建什么表单、谁填写、填写后的数据用来做什么？

如果用户说不清，AI 必须提供通俗的候选案例供选择，而不是要求用户补交完整需求。

## 2. 职责边界

### 2.1 本元规范负责什么

```text
接收一句话需求
主持目标和需求访谈
解释当前决策所需的概念
识别任务类型、产品模型或期望行为分叉
必要时生成低保真原型帮助判断
区分事实、推断、假设、未知和人工决策
形成目标与边界合同
发现已有任务/产品资料和代码候选
提出工作空间划分方案
确认任务/产品、需求基线、原型、源码、执行记录和证据目录
生成与任务类型匹配的需求基线，适用时生成 PRD
组织需求基线人工评审、批准与基线登记
判断任务类型、规模和风险
形成后续研发活动入口合同
```

### 2.2 本元规范不负责什么

在启动包完成前，本元规范不负责：

- 最终技术架构；
- 最终数据库和接口设计；
- 正式任务切片；
- 生产代码实现；
- 完整测试执行；
- 生产上线；
- 最终系统验收。

这些工作应在启动包批准后，由后续研发活动按本规范输出的目标、边界、目录和验收基线继续执行。本元规范不要求任何特定流程文件存在。

本元规范虽然不替后续活动完成正式任务切片，但必须规定切片的最低治理要求：每个可独立分配、执行和验收的任务切片，都要在开始前建立 Task Contract；后续活动不得把未经 DoR 检查、未预先锁定 DoD 的工作直接交给 AI 执行。

### 2.3 人与 AI 的职责

| 角色 | 负责 | 不负责 |
|---|---|---|
| 用户/业务负责人 | 描述真实工作、纠正 AI 理解、确认目标和业务规则 | 预先写完整 PRD、设计目录体系、掌握流程术语 |
| 产品/需求负责人 | 确认目标、范围、优先级和需求基线 | 代替技术人员判断实现细节 |
| AI 主持者 | 提问、解释、归纳、查证、提出方案、记录决策、生成草案 | 替用户作产品取舍或把建议冒充确认结果 |
| 技术负责人 | 确认可行性、约束和工作空间中的代码责任边界 | 擅自改变业务目标 |
| 测试/验收者 | 判断需求是否可验证、验收标准是否充分 | 用“看起来合理”代替实际证据 |

### 2.4 动作等级

| 等级 | AI 行为 | 本规范中的示例 |
|---|---|---|
| 自动执行 | 直接执行并报告 | 读取资料、整理回答、搜索候选目录、生成需求基线草案、自检和后续活动判定 |
| 执行后送审 | 先生成草案，未经确认不能作为基线 | 目标与边界合同、工作空间合同、PRD 或等价需求基线 |
| 必须人工决策 | AI 给出候选、影响和建议后等待 | 目标与范围取舍、工作空间采用方案、需求基线批准、风险接受 |

AI 不得以“用户要求全自动”为由越过必须人工决策点。

## 3. 元流程状态机 M0-M8

### 3.1 阶段总览

| 阶段 | 名称 | 关键结果 |
|---|---|---|
| M0 | 单句接收与任务建档 | 原始请求被准确记录 |
| M1 | 只读资料与候选环境发现 | 已有事实和候选目录被识别 |
| M2 | 目标与需求访谈 | 用户、场景、问题和价值逐步清晰 |
| M3 | 目标与边界确认 | Goal and Scope Contract 通过 |
| M4 | 工作空间发现与规划 | AI 提出目录和仓库划分方案 |
| M5 | 工作空间确认与建档 | Workspace Contract 通过并持久化 |
| M6 | 需求基线准备与草拟 | 先锁定基线类型和质量合同，再形成 PRD 或等价需求基线草案及追溯关系 |
| M7 | 需求基线评审、批准与登记 | 草案被人工批准后，批准版本被登记为权威需求基线；退回时返回草拟任务 |
| M8 | 后续活动判定与移交 | Startup Package 形成并可直接执行 |

### 3.2 统一状态

每个阶段只能使用：

```text
not_started
in_progress
blocked
awaiting_human
passed
failed
superseded
```

合法迁移：

```text
not_started → in_progress
in_progress → passed | failed | blocked | awaiting_human
failed | blocked | awaiting_human → in_progress
passed → superseded → in_progress
```

未经用户确认的目标与边界、工作空间或需求基线只能是 `awaiting_human`，不能由 AI 自行标记 `passed`。

### 3.3 事实与决策分类

AI 在全过程维护：

```yaml
facts: 有用户明确陈述、正式文档、源码、数据或运行证据支持
inferences: AI 从事实推出的理解，必须给出推理依据
hypotheses: 尚未验证，只能指导下一步提问或实验
unknowns: 当前不知道且可能影响产品或实现的事项
decisions: 用户或有权负责人已明确作出的决定
superseded_decisions: 已被新决定替代的旧结论
```

AI 的推荐不能直接写入 `decisions`。

### 3.4 M0-M8 标准作业程序（SOP）

本表是本文的主执行入口。AI 必须逐阶段执行；各阶段详细字段和规则由第 4-12 节给出。本表中的“阶段 DoD”是进入下一阶段的最低完成条件，不得用“已经生成文档”替代。

| 阶段 | DoR（允许开始的条件） | AI 必须执行的动作 | 必须产物 | 阶段 DoD | 未通过时的处理 | 通过后 |
|---|---|---|---|---|---|---|
| M0 单句接收 | 收到非空自然语言需求 | 原样记录；拆分事实、推断和未知；禁止提前设计 | Intake Record | 原始表达无损保留；当前只确认用户明确表达的事实；禁止动作已登记 | 需求连建设对象或待解决问题都无法识别时，只问一个澄清问题，停留 M0 | M1 |
| M1 只读发现 | M0 `passed`；具备相应只读权限 | 最小范围检查已有产品资料、需求、原型、源码、任务和规则；识别任务类型、复杂度、风险和强制策略候选 | Discovery Record、候选路径、Policy Candidate List 及证据引用 | 已有事实、候选环境和强制策略有来源；未找到的内容标为未知；没有把同名目录当作已确认工作空间 | 无权限时记录限制；不阻断目标澄清则继续，阻断时标记 `blocked` 并说明所缺权限 | M2 |
| M2 目标访谈 | M1 已完成或已记录不阻断的发现限制 | 每轮只问一个最高价值问题；围绕角色、场景、问题、结果、范围和非目标收敛；必要时给候选或低保真比较材料 | Alignment Log、角色/场景/问题/成功判断候选 | AI 能准确复述“谁在什么场景遇到什么问题、本轮希望得到什么可观察结果”，主要分叉已显式呈现 | 仍有重大歧义时继续逐问，不得生成正式 PRD、架构或切片 | M3 |
| M3 目标与边界确认 | M2 阶段 DoD 通过；确认人可参与决策 | 草拟并解释 Goal and Scope Contract；逐项处理矛盾、假设和阻断未知；确认风险事实对应的策略适用性；请求明确批准 | Approved Goal and Scope Contract、Policy Applicability Record、批准记录 | 问题、目标、角色、主场景、范围、非目标、成功判断、决策人和高风险策略适用性完整；有权人员明确批准 | 被纠正或拒绝时回到 M2；AI 不得自批 | M4 |
| M4 工作空间规划 | M3 `passed` | 只读发现现有目录和仓库；提出产品、需求、原型、源码、任务、证据和知识来源的职责与候选位置；说明复用、创建及冲突 | Workspace Proposal | 必要资产均有候选位置和职责；现有目录已核实；源码仓库拓扑没有被提前虚构 | 证据不足或职责冲突时继续 M4，只把方案标为候选 | M5 |
| M5 工作空间确认 | M4 阶段 DoD 通过；用户可确认目录和授权范围 | 逐项确认权威来源、读写边界、生产源码候选、策略基线保存位置及资产归属；批准后才持久化启动材料 | Approved Workspace Contract、Policy Baseline Draft、批准记录、Execution State | 工作空间合同经人确认；必要路径与策略引用可解析或有明确创建动作；写入范围和权威来源明确 | 未确认时标记 `awaiting_human`；路径、策略或职责需改时返回 M4 | M6 |
| M6 基线准备与草拟 | M3、M5 `passed`；task_type 已判定 | 先执行 RB-0，确定 baseline_type 并锁定质量合同、责任人和路径；RB-0 通过后再执行 RB-1 生成 PRD 或等价基线草案、验收和追溯 | Baseline Decision、Quality Contract、RB-0/RB-1/RB-2/RB-3 Task Contract、PRD Draft 或等价基线 Draft、RTM、自检记录 | RB-0 与 RB-1 各自 DoD 全部通过；标准在草拟前已锁定；满足第 10.10 节草案门禁 | RB-0 未通过不得草拟；RB-1 未通过继续修复或回到最早受影响阶段 | M7 |
| M7 评审、批准与基线登记 | M6 阶段 DoD 通过；送审版本、评审人及批准权明确 | 执行 RB-2 多视角评审并批准具体版本；RB-2 通过后执行 RB-3，登记版本、哈希、权威位置和替代关系 | Approval Record、Baseline Record、最终 PRD/等价基线、最终 RTM、RB-2/RB-3 DoD 证据 | RB-2 与 RB-3 各自 DoD 全部通过；只有有权人员批准；批准版本与登记版本和哈希一致 | 评审退回 RB-1；登记失败保持 `approved`，不得标为 `baselined` | M8 |
| M8 路由与移交 | M3、M5、M7 均 `passed`；产物引用可解析 | 确认任务类型、复杂度、风险和下一活动；为第一项可执行任务预置 Task Contract、DoR、DoD 与验证方法；组装并校验启动包 | Task Startup Package、Next Activity Contract、First Task Contract | 第 12.5 节任务启动 DoD 全部通过，启动包状态为 `passed`；第一任务的 Task Contract 完整 | 从第一个未通过或已失效的阶段恢复；不得生成“部分通过”的正式移交包 | 校验第一任务 DoR 后执行 `first_action` |

### 3.5 SOP 执行规则

1. **从最早缺口继续**：新任务从 M0 开始；恢复任务先校验产物、批准状态和输入哈希，从第一个非 `passed` 或已失效阶段继续，不重复已经确认的问题。
2. **阶段不得越过**：上游 DoD 未通过，下游 DoR 就不成立。M3、M5、M7 的人工门禁不得跳过，也不得由同一个 AI 代替有权人员批准。
3. **状态必须真实**：`in_progress` 表示正在处理；`awaiting_human` 表示只等待人工决定；`blocked` 表示缺少不可替代的输入、权限或外部条件；`failed` 表示检查未通过且已有明确修复项；`passed` 表示 DoD 及证据全部满足。
4. **证据随状态记录**：每次状态变化必须记录输入版本、产物引用、检查结果、决定人和时间。没有证据引用的 `passed` 无效。
5. **一次只推进一个关键决定**：与用户交互时每轮只提出一个最能降低不确定性的问题；可同时展示该问题所需的事实、候选和影响，但不得倾倒整份问题清单。
6. **失败回到最早受影响阶段**：例如 PRD 评审发现产品边界错误，应重开 M2/M3，而不是只润色 M6 文档；目录变化只重开 M4/M5 及依赖其路径的产物。
7. **草案不等于完成**：生成文件、补齐标题或通过 AI 自检都不能替代人工门禁。草案状态最多为 `draft` 或 `under_review`。
8. **启动完成后受控续接**：M8 通过后重新校验第一任务合同、Policy Baseline、输入哈希和全部 DoR；只有第一任务 `dor_status: passed` 才执行 `first_action`。遇到新的重大产品取舍、目录授权变化或风险接受决策时进入 `awaiting_human`。

### 3.6 独立任务切片的 DoR/DoD 合同

#### 3.6.1 哪些工作必须建立任务合同

只要一项工作同时满足以下特征，就视为“可独立验收的任务切片”，开始前必须具有 Task Contract：

```text
有一个主要、可观察的结果；
可以明确由谁或哪个 AI 负责；
可以独立判断通过或退回；
失败后可以单独重试、回滚或重新规划；
其输出会被后续任务依赖。
```

阶段任务、PRD 草拟、PRD 评审、基线登记、架构决策、原型页面、代码功能、缺陷修复、测试任务和发布任务均适用。读取文件、搜索代码、执行一条命令、修改一个变量等任务内部操作步骤，通常不单独建立 DoR/DoD，除非它本身具有独立风险、交付物或批准要求。

第 3.4 节 M0-M8 SOP 表中的每一行就是对应阶段的规范级 Stage Contract；实际执行时由 Execution State 记录该阶段的输入、DoR/DoD 证据和状态，不再为同一阶段机械复制一份内容相同的 Task Contract。阶段内部拆出的独立任务仍必须按本节建立 Task Contract。

#### 3.6.2 最小 Task Contract

```yaml
task_id: TASK-唯一编号
contract_version: 任务合同版本
created_at: 创建时间
locked_at: 进入 in_progress 前的锁定时间
parent_stage_or_task: 上级阶段或任务
parent_acceptance_refs: []
objective: 本任务唯一的可观察结果
scope: []
non_goals: []
owner: 责任人或执行 AI
reviewer: 评审人
approver: 需要批准时填写
dependencies: []
inputs:
  - ref: 输入路径或标识
    required_status: approved|baselined|passed|available
dor:
  - criterion: 开始前必须满足的条件
    evidence_ref: 证明位置
    status: pending|passed|failed
dod:
  - criterion: 完成时必须满足的结果
    evidence_ref: 证明位置
    status: pending|passed|failed
verification_methods: []
output_refs: []
failure_or_rollback_action: 失败、退回或回滚动作
policy_baseline_ref: 本任务适用的策略基线
execution_fingerprint:
  source_revision: 源码提交或工作区版本
  rules_and_prompt_versions: []
  model_or_agent_id: 可取得时记录
  tool_versions: []
  environment_ref: 环境或容器标识
status: not_started|in_progress|blocked|awaiting_human|failed|passed|superseded
```

L1 低风险任务可以把 Task Contract 内嵌在任务记录中，只保留目标、DoR、DoD、责任人和验证方法；不得因为任务小而省略开始条件与完成条件本身。

#### 3.6.3 开始、完成与变更规则

1. **先定标准再开始**：Task Contract、DoR 和 DoD 必须在任务进入 `in_progress` 前存在；DoR 全部 `passed` 后才允许执行。
2. **DoD 不能事后补写**：任务完成后才创建 DoD，或者为了让当前结果通过而降低 DoD，均视为流程失败。
3. **标准变化要留痕**：执行中确需改变 DoD 时，先暂停任务，记录原因、影响和批准人，生成新版本 Task Contract，再重新判断 DoR；旧版本不得覆盖。
4. **证据先于通过**：每个 DoD 条目必须指向测试、评审、批准、运行结果或可检查产物；没有证据的条目不能标记 `passed`。
5. **子任务向上闭合**：子任务 DoD 必须映射到父任务或阶段的某个验收项；全部子任务通过不自动等于父任务通过，父任务仍要执行集成或整体 DoD。
6. **一个切片一个主要结果**：若任务包含多个可独立接受或失败的结果、跨越多个责任边界，或者无法在一次聚焦执行与验证循环中完成，AI 必须继续拆分。
7. **失败回到责任边界**：DoR 不满足时任务保持 `not_started/blocked`；DoD 不满足时标记 `failed` 或返回 `in_progress` 修复，不能以“主体已完成”跳过剩余验收项。
8. **证据必须可复现**：验证证据至少记录执行命令或方法、时间、环境、退出码/结论和报告位置；只写“测试通过”“已验证”无效。

Task Contract 的“创建与登记”是父阶段或父任务的规划动作，不递归要求先为“创建 Task Contract”再建立另一份 Task Contract。该例外只终止治理递归，不允许被用于跳过实际研发、评审、测试或发布任务的 DoR/DoD。

#### 3.6.4 Slice Budget（防止 AI 切片过大）

每个任务切片在进入 `in_progress` 前必须给出以下预算与实际判断：

```yaml
slice_budget:
  primary_outcomes: 1
  responsibility_boundaries: 预计跨越的模块/服务/团队边界数量
  independent_failure_points: 可独立失败或回滚的结果数量
  acceptance_scenarios: 验收场景数量
  context_inputs: 必须同时阅读的源码、设计、契约和测试清单
  rollback_unit: 可独立回滚的最小单位
  within_single_context: true|false
  split_decision: keep|split
  split_reason: 保留或继续拆分的证据
```

任一条件成立时必须继续拆分，除非技术负责人记录不可拆原因和额外验证措施：

- `primary_outcomes > 1`；
- `independent_failure_points > 1`；
- 跨越两个以上责任边界，且不能作为同一原子变更共同验证和回滚；
- 子结果可以被不同责任人独立接受、拒绝或上线；
- 必要上下文无法在一次执行中完整读取并保持；
- 无法给出单一回滚单位或单一主要验收结论。

文件数或代码行数只能作为风险信号，不能单独决定切片；关键判据是责任边界、独立失败点、验收和回滚是否一致。

## 4. M0：单句接收与任务建档

### 4.1 输入

```yaml
original_request: 用户原始的一句话
received_at: 时间
requester: 当前用户或业务发起人
```

### 4.2 动作

1. 原样保留用户表述；
2. 提取其中明确出现的对象、动作和目标；
3. 判断它是完整需求还是模糊目标；
4. 默认按模糊目标处理，除非验收和边界已经明确；
5. 创建内存中的 `Intake Record`，此时不要求用户提供目录。

### 4.3 输出

```yaml
intake_id: INTAKE-YYYYMMDD-NNN
original_request: 原文
confirmed_facts: []
initial_inferences: []
unknowns: []
current_stage: M1
prohibited_actions:
  - formal_requirement_baseline
  - architecture
  - slicing
  - coding
```

### 4.4 门禁

只要原始请求能表达一个想解决的问题或想建设的对象，M0 即可通过。AI 不得因为用户信息少而拒绝启动，也不得要求用户改写成长篇提示词。

## 5. M1：只读资料与候选环境发现

### 5.1 目的

在提问前先利用已有事实，避免询问可以从本地资料直接查到的问题。

### 5.2 AI 自动动作

AI 按最小范围只读检查：

```text
工作区入口规则和项目映射
与原始需求同名或相关的产品方案目录
相关领域知识库
候选代码仓库的项目说明、入口规则和实际职责
已有 PRD、原型、ADR、任务和测试
历史同类产品或组件
组织强制规则、项目级规则、安全/合规要求及其版本或哈希
```

搜索必须有范围和结果上限。禁止从工作区根目录无边界递归扫描，也禁止把同名目录直接当成正式项目。

### 5.3 已有项目与新产品判定

AI 形成候选而不是直接决定：

```yaml
candidate_product_roots: []
candidate_code_roots: []
candidate_knowledge_sources: []
existing_artifacts: []
task_type_hypothesis: new_product|existing_product_change|bugfix|refactor|research
```

任务类型按实际目标判定：

| 类型 | 判定标准 | 需求基线 |
|---|---|---|
| `new_product` | 建立此前不存在的产品或独立产品能力 | 完整产品蓝图、用户需求、PRD 和 RTM |
| `existing_product_change` | 改变现有产品的用户行为、范围或业务规则 | 增量 PRD、受影响用户故事和 RTM 差异 |
| `bugfix` | 现有行为偏离已知合同或可证明的期望行为 | 问题合同、现状证据、期望行为和回归验收 |
| `refactor` | 保持外部行为不变，改善内部结构或质量 | 行为不变量、影响范围和回归合同 |
| `research` | 关键事实或技术可行性未知，目标是获得证据 | 研究问题、假设、实验方法、阈值和结论格式 |

### 5.4 复杂度与风险分级

| 等级 | 判定标准 | 启动要求 |
|---|---|---|
| L1 | 单一文档、原型或局部模块，依赖少，不改变共享契约 | 目标、边界、目录和局部验收明确 |
| L2 | 单产品或单仓库内的多文件、多组件变更，存在有限集成 | 补齐影响范围、接口关系、回归范围和回滚方式 |
| L3 | 跨模块、跨仓库或跨团队，改变共享接口、数据模型或关键流程 | 必须形成系统验收、依赖图、分批策略，并由负责人确认风险 |
| L4 | 涉及权限、隐私、多租户、支付、金额、库存、生产数据迁移、不可逆决策或重大合规风险 | 必须进行专项风险评审、独立验收和可执行回滚/恢复设计，未经授权不得实现或上线 |

自动升级规则：跨仓库、共享契约或关键数据模型至少为 L3；触及 L4 所列任一边界立即升级为 L4；无法确定影响范围时不得按 L1 处理。复杂度只决定流程强度，不降低事实准确性、兼容性和验证要求。

### 5.5 输出与门禁

**输出**：事实摘要、候选资料、候选目录、待确认任务类型和 `Policy Candidate List`。

**门禁**：候选目录只能用于只读调查，M5 通过前不得作为正式写入范围。

### 5.6 基线前受控 Spike

当关键技术未知会直接影响产品边界、验收阈值或 PRD 能否批准，且仅靠资料查证无法消除时，允许在需求基线完成前执行受控 Spike。它是第 0.2 节“不得提前实验”的唯一例外。

**允许条件（全部满足）：**

```text
[ ] 待验证假设会改变需求、架构可行性或风险等级
[ ] 只读查证和现有证据不足
[ ] Spike Task Contract 已预先锁定 DoR、DoD、时间/成本预算和停止条件
[ ] 工作目录为隔离的临时/实验目录，不写生产源码和生产数据
[ ] 输出仅为证据、结论和局限，不作为可发布实现
[ ] L3/L4 或涉及敏感数据、安全边界时，技术负责人已批准实验范围
```

最小合同：

```yaml
spike_id: SPIKE-唯一编号
hypothesis: 待验证的单一技术假设
decision_affected: 该结论会影响的目标、需求或风险决定
method: 最小实验方法
budget: 时间、成本和资源上限
environment: 隔离环境
production_code_allowed: false
production_data_allowed: false
success_and_failure_thresholds: []
stop_conditions: []
evidence_refs: []
limitations: []
disposition: discard|retain_as_nonproduction_reference
status: not_started|in_progress|passed|failed|blocked
```

Spike 结束后必须删除或明确隔离实验性实现，把证据、否定结论、局限和受影响决策回灌 M2/M3、复杂度风险判断及 RB-0 输入。不得复制实验代码进入生产实现来绕过后续编码 DoR/DoD。

### 5.7 Policy Baseline（强制策略基线）

“元规范自包含”只表示完成 M0-M8 不依赖其他方法论文档，不表示组织政策、项目规则、安全底线或法规可以忽略。AI 在 M1 发现、M3 风险确认和 M5 工作空间确认过程中，必须形成：

```yaml
policy_baseline_id: POLICY-唯一编号
scope: 当前任务和后续活动
mandatory_policies:
  - ref: 文件路径、规则标识或法规条款
    version_or_hash: 版本或 SHA-256
    precedence: 优先级
    applicability_reason: 命中事实与适用范围
    mandatory_controls: []
advisory_references:
  - ref: 仅供方法参考的资料
    version_or_hash: 版本或 SHA-256
conflicts: []
approved_exceptions: []
bundle_sha256: 对规范化后的范围、强制策略、控制项、参考资料、冲突裁决和批准例外计算的 SHA-256
approved_by: 有权策略批准人
approved_at: 批准时间
approval_ref: 绑定 bundle_sha256 的批准证据
validated_at: 时间
status: draft|awaiting_human|passed|superseded
```

`bundle_sha256` 的规范化输入必须覆盖 `scope`、每项 `mandatory_policies`（含 `mandatory_controls`）、`advisory_references`、全部冲突及其裁决、全部批准例外及其范围/期限/补偿控制；不包含批准记录本身，避免自引用。批准证据必须绑定该哈希。只有引用可解析、版本已冻结、冲突已裁决、例外已获有权人员批准且批准证据可验证时，状态才能为 `passed`。

Policy Baseline 未达到 `passed` 时，不得开始 RB-0、草拟任何需求基线或通过 M8。后续任务必须引用同一 Policy Baseline；需要例外时必须记录批准人、范围、期限和补偿控制，AI 不得自行批准策略、接受例外或把强制规则降为参考建议。

## 6. M2：AI 主持目标与需求访谈

### 6.1 访谈原则

```text
一次只问一个关键问题
优先问会改变任务类型、产品模型、期望行为或范围的问题
优先从真实场景、问题现象或研究问题开始，不从功能清单开始
用户说不清时提供 2-4 个通俗选项和例子
每轮回答后先归纳，再选择下一个问题
不重复已经回答的问题
专业概念只在影响当前决策时即时解释
涉及交互分歧时用低保真原型，不用术语争论
```

禁止一次性向用户发送“请回答以下 20 个问题”。

### 6.2 提问优先级

AI 根据当前最大不确定性动态选择，通常依次覆盖：

1. 最典型的真实工作、问题现象或待回答问题；
2. 谁受影响、谁使用结果、哪个系统或对象被改变；
3. 当前状态、期望状态及两者差距；
4. 输入、触发、关键动作和可观察结果；
5. 数据归属、后续业务动作或证据用途；
6. 主要角色、权限和系统边界；
7. 最小可交付行为、必须保持的不变量或最小实验；
8. 哪些能力、行为或结论明确不属于本轮；
9. 成功、修复、无行为变化或研究结论如何判定；
10. 时间、成本、合规、兼容和停止条件等重要约束。

这些是覆盖维度，不是固定问卷。已有证据明确的内容不再询问。

### 6.3 用户说不清时的处理

AI 应把抽象问题改写成可选择的真实场景。例如“企业表单设计器”的首个分叉可以表达为：

```text
A. 管理员设计客户、订单等业务数据录入表单
B. 员工填写费用、请假等申请并进入审批
C. 企业通过公开链接收集报名、问卷或线索
D. 管理员搭建带表格、按钮和数据源的完整业务页面
```

AI 同时解释各选项会怎样改变产品，但不得暗示用户必须四选一；用户可以选择组合并指定主次。

### 6.4 逐轮对齐记录

每次用户回答后，AI 更新并向用户展示必要摘要：

```yaml
new_facts: []
updated_understanding: 一句话
new_decisions: []
remaining_key_unknown: 下一项最重要未知
next_question: 仅一个
```

每完成一个会改变任务类型、产品模型、期望行为或范围的分叉，AI 必须复述决定及其直接后果，防止双方对同一句话理解不同。

### 6.5 原型触发条件

满足以下任一条件时，AI提出 2-3 个低保真方案：

- 页面结构影响产品模型；
- 用户无法仅靠文字判断操作方式；
- 多个交互模型均合理且取舍明显；
- 真实场景走查暴露双方理解不一致。

原型只用于验证产品和交互假设，不能证明权限、安全、数据一致性或后端可行性。

M5 通过前，原型只能作为对话内或临时预览中的比较材料，不得写入候选任务、产品、源码或正式原型目录；M5 通过后，才将选定方案、走查记录和批准结果持久化到 `prototype_root`。

### 6.6 输出与门禁

**输出**：`Discovery Ledger`、相关角色/系统、场景或现象、当前问题、结果假设、范围候选、非目标候选和开放问题。

**门禁**：只有当 AI 能清晰复述“针对谁或什么对象、当前问题或未知是什么、本轮产生什么可观察结果、范围和非目标是什么”时，才能进入 M3。

## 7. M3：目标与需求边界确认

### 7.1 Goal and Scope Contract

AI 根据访谈生成：

```yaml
task_name: 任务或产品暂定名称
task_type: new_product|existing_product_change|bugfix|refactor|research
complexity: L1|L2|L3|L4
risk_reasons: []
problem_statement: 当前问题
objective: 期望的可观察结果
affected_users_or_stakeholders: []
primary_scenarios_or_symptoms: []
secondary_scenarios: []
anti_scenarios_or_counterexamples: []

scope: []
non_goals: []
success_criteria: []
constraints: []
invariants: []
assumptions: []
open_questions: []

task_specific:
  product_or_change:
    product_definition: 一句话产品定义
    primary_product_model: 主要产品模型
    included_models_or_capabilities: []
    excluded_models_or_capabilities: []
    mvp_scope: []
  bugfix:
    observed_behavior: 已复现或有证据的现状
    expected_behavior: 可证明的期望行为
    affected_baseline: 受影响版本或环境
  refactor:
    behavior_invariants: []
    internal_quality_objective: 内部质量目标
    affected_surface: []
  research:
    research_question: 待回答问题
    hypotheses: []
    evidence_threshold: 判定所需证据
    stop_condition: 停止条件

source_refs: []
approved_by: null
status: awaiting_human
```

`task_specific` 只保留与 `task_type` 匹配的分支：`new_product` 和 `existing_product_change` 使用 `product_or_change`，其余类型使用同名分支。不得要求 Bug、重构或研究任务填写虚假的产品模型和 MVP，也不得让新产品跳过产品定义、目标用户和首版范围。

### 7.2 Alignment Gate

所有任务至少满足：

```text
[ ] 问题、目标结果和相关角色已确认
[ ] 至少一个首要场景、现象或研究问题已完整描述
[ ] 范围、非目标和关键约束已确认
[ ] 成功、修复、保持不变或研究结论的判断方式已定义
[ ] 复杂度和风险原因已根据确认后的边界重新判定
[ ] 已根据风险、数据、权限、技术栈和项目边界确认强制策略的适用性；冲突和例外已登记
[ ] 阻断工作空间规划的未知项已消除
```

L3/L4 或触及安全、数据、权限、金额、库存、生产迁移等高风险边界时，复杂度和风险接受必须由有权负责人确认；AI 不能自行降级。

再按任务类型追加：

```text
new_product / existing_product_change：
[ ] 一句话产品定义、目标用户、主要产品模型和首版范围已确认
[ ] 关键场景和至少一个反例已确认

bugfix：
[ ] 现状有复现或证据，期望行为有合同、需求或负责人决定支撑
[ ] 受影响基线和明确不改的行为已确认

refactor：
[ ] 外部行为不变量、影响范围和内部质量目标已确认

research：
[ ] 研究问题、假设、证据阈值、时间/成本边界和停止条件已确认
```

“三个场景加一个反例”是复杂新产品的推荐做法，不作为所有任务的机械数量门槛；门禁关注是否足以确定产品边界。

### 7.3 人工确认

AI 必须用通俗语言展示合同摘要，并只要求用户确认或纠正目标与边界决定。用户或有权目标负责人明确认可后：

```yaml
status: passed
approved_by: 用户/目标负责人
approved_at: 时间
```

未经确认，状态保持 `awaiting_human`，不得进入工作空间写入或需求基线生成。

## 8. M4：工作空间发现与规划

### 8.1 为什么在目标确认后规划目录

目录划分取决于任务是新产品、现有变更、Bug、重构、研究、跨仓库能力还是单仓库修改。目标与边界未确认时提前创建目录，会把错误假设固化成组织结构。

M3 通过后，AI 才能确定要管理哪些类型的资产，并提出工作空间方案。

本规范中的“工作空间”不是某一个总文件夹，而是本次任务使用的一组已确认目录、仓库及职责映射。它回答：任务或产品资料在哪里、需求基线在哪里、原型在哪里、源码在哪里、执行状态在哪里、验证证据在哪里，以及各自谁是权威来源。

### 8.2 工作空间中的八类根目录

| 字段 | 职责 | 权威内容 |
|---|---|---|
| `product_root` | 任务所属产品或主题资料 | 目标、范围、需求基线、重大决策、发布和复盘 |
| `requirement_root` | 权威需求基线 | PRD、变更规格、问题合同、回归合同、研究合同和追溯记录 |
| `prototype_root` | 产品与交互验证 | 原型、交互状态、走查记录和批准基线 |
| `code_roots` | 生产源码仓库 | 实际代码、构建、测试、迁移和运行配置 |
| `task_root` | 本次任务执行账本 | 合同、切片、进度、追溯记录和任务证据 |
| `evidence_root` | 验收证据 | 测试报告、截图、日志、性能和安全结果 |
| `knowledge_sources` | 领域知识和外部证据 | 领域对象、竞品、规则、失败模式和证据卡 |
| `policy_root` | 强制策略与例外 | Policy Baseline、规则引用/快照、冲突裁决和批准例外 |

这些逻辑职责可以在同一物理仓库，也可以分布在多个仓库。AI 必须优先沿用现有项目结构，不能为了“标准化”擅自拆仓或迁移。每个字段都必须存在于合同中；不适用时，单路径字段使用 `required: false`、`path: null` 和 `not_applicable_reason`，列表字段使用 `required: false`、`items: []` 和 `not_applicable_reason`，禁止删除字段或虚构目录。

### 8.3 AI 自动发现规则

AI 必须先：

1. 查询工作区项目入口映射；
2. 检查候选产品目录是否已有同名/相关产品；
3. 检查候选代码仓库的项目说明、入口规则和实际职责；
4. 判断原型是文档附件、HTML 原型还是独立可运行项目；
5. 检查是否已有任务与证据目录约定；
6. 识别重复目录、镜像目录或已废弃版本；
7. 输出候选方案和采用理由。

AI 不得直接问用户一个含糊问题“请提供项目/产品目录”。

### 8.4 新产品的默认建议结构

只有在没有可复用结构时，AI 才提出默认结构：

```text
产品资料根目录/
├── 00-启动与范围/
│   ├── Original-Request.md
│   ├── Goal-and-Scope-Contract.yaml
│   └── Workspace-Contract.yaml
├── 01-知识库盘点/
├── 02-机会与蓝图/
├── 03-需求/
│   ├── 02-用户需求-用例与用户故事.md
│   ├── PRD-产品需求规格说明书.md
│   └── 需求追溯表.md
├── 03A-原型与体验验证/
│   ├── low-fidelity/
│   ├── high-fidelity/
│   ├── interaction-spec/
│   └── review-records/
├── 04-架构决策/
├── 05-详细设计/
├── 06-任务与测试/
├── 07-知识库同步/
├── 08-上线运营/
└── 09-复盘回写/
```

生产源码默认放在独立代码仓库或现有业务仓库，不直接混入产品文档目录：

```text
代码工作区/
├── <product-or-module>-ui/       # 确实需要独立前端仓库时
├── <product-or-module>-service/  # 确实需要独立服务时
└── <existing-monorepo>/          # 已有单仓结构时优先复用
```

仓库拆分属于架构和团队边界决策。M4 可以提出候选源码位置，但不能仅凭产品名称提前创建多个仓库；最终仓库拓扑应由现有架构或后续 ADR 确认。

### 8.5 原型放置规则

```text
低保真图、截图、Figma 链接、走查记录
→ prototype_root

简单 HTML/CSS/JS 概念原型
→ prototype_root 下的独立子目录

具有构建依赖、测试和持续演进的可运行原型
→ 独立 prototype code root，并在 Workspace Contract 登记

生产实现
→ code_roots，禁止把未经评审的原型代码直接当生产代码复制
```

### 8.6 工作空间候选方案输出

AI 应主动向用户展示：

```text
我发现的已有目录：
- 路径、当前用途、证据

我建议采用的目录：
- 任务或产品资料放哪里
- 权威需求基线和追溯记录放哪里
- 原型放哪里
- 生产源码放哪个现有仓库或候选工作区
- 当前任务和验收证据放哪里

需要新建的内容：
- 只列必要目录，不提前创建生产仓库

存在的冲突或未知：
- 重复产品目录、职责不清的仓库等
```

然后只问用户一个确认问题：

> 是否采用以上工作空间方案？需要修改时，请指出哪一项路径或职责不符合实际。

### 8.7 输出与门禁

**输出**：已有目录事实清单、候选工作空间方案、推荐方案、目录职责说明、待确认冲突和拟新增目录清单。

**Workspace Proposal Gate**：只有当每类必要资产都有明确候选位置、现有目录没有被误判、源码仓库职责没有被提前虚构、方案可以由用户用一次确认或局部纠正完成时，M4 才能进入 M5。

## 9. M5：工作空间确认与建档

### 9.1 Workspace Contract

```yaml
workspace_id: WORKSPACE-001
task_or_product_name: 任务或产品名称

product_root:
  required: true|false
  path: 任务所属产品/主题资料根目录或 null
  authority: 目标、范围、需求基线、重大决策、发布与复盘
  not_applicable_reason: null

requirement_root:
  required: true
  path: 权威需求基线和追溯记录目录
  authority: 已批准并完成基线登记的 PRD 或等价需求基线，以及对应质量合同、验收/判定标准和追溯记录

prototype_root:
  required: true|false
  path: 原型目录、仓库或 null
  authority: 已批准的产品交互与视觉基线
  not_applicable_reason: null

code_roots:
  required: true|false
  items:
    - path: 源码仓库根目录
      responsibility: 本仓库在任务或产品中的职责
      status: existing|planned
      write_allowed: true|false
  not_applicable_reason: null

task_root:
  path: 当前任务目录
  authority: 任务状态、切片、RTM和任务证据

evidence_root:
  path: 验收证据目录
  authority: 新鲜验证结果

policy_root:
  path: Policy Baseline、规则快照/引用和例外记录目录
  authority: 当前任务必须遵守的策略版本、优先级和适用范围

knowledge_sources:
  required: true|false
  items:
    - path_or_url: 领域知识来源
      evidence_scope: 可支撑的结论
  not_applicable_reason: null

source_of_truth:
  goal_and_scope: approved Goal and Scope Contract
  requirement_baseline: baselined PRD or equivalent requirement baseline
  interaction: approved prototype and interaction spec
  architecture: accepted ADR
  runtime_behavior: source code, tests and runtime evidence
  completion: evidence pack and acceptance report

conflict_policy:
  - 权威来源冲突时停止受影响的后续动作并提交人工裁决
  - 决策后同步更新所有受影响产物

approved_by: null
status: awaiting_human
```

### 9.2 Workspace Gate

```text
[ ] 任务所属产品/主题资料有唯一权威根目录，或已说明不适用原因
[ ] 需求基线、判定标准和追溯记录有明确的 requirement_root
[ ] 原型及评审记录保存位置明确，或已说明不适用原因
[ ] 需要源码时，每个候选源码仓库职责明确且已确认真实存在或待后续决策；不需要时已说明原因
[ ] 当前任务和证据保存位置明确
[ ] Policy Baseline、规则引用/快照、冲突和例外记录的保存位置明确
[ ] 相关知识来源已登记，或已说明不适用原因
[ ] 各类冲突的权威来源和处理方式明确
[ ] 不会覆盖或迁移用户现有文件
```

### 9.3 写入权限

工作空间方案经用户确认后，AI 才能：

- 创建必要的产品资料和任务目录；
- 写入 M0-M3 已确认结果；
- 生成 `Workspace-Contract.yaml`；
- 进入 M6，由父阶段初始化 RB-0 Task Contract，先确定需求基线类型并锁定质量合同，再按门禁生成草案。

AI 创建目录前必须再次检查是否已存在，不得覆盖已有文件。生产源码仓库的新建、拆分或迁移若属于重大架构选择，应延后到后续架构决策。

### 9.4 输出与门禁

**输出**：已批准的 `Workspace-Contract.yaml`、Policy Baseline Draft、必要目录创建记录、资产到目录的映射以及未创建的候选源码仓库清单。

**Workspace Gate**：第 9.2 节全部通过且用户确认后，M5 才能标记 `passed`。仅有 AI 推荐路径、尚未得到用户确认时，状态必须为 `awaiting_human`。

## 10. M6：需求基线准备与草拟

### 10.1 正确顺序

本节完整定义需求基线和 PRD 的生成规则，不依赖其他流程文件。对新产品或重大产品能力，生成完整产品蓝图、用户需求、PRD 和需求追溯表；对既有产品的小范围变更，复用已批准产品基线并生成增量需求或变更 PRD。

工作空间通过后，AI 必须先按第 10.2 节选择目标需求基线类型，不需要反问用户“是否要写 PRD”。“选择 `baseline_type: prd`”表示本任务最终要形成 PRD 权威基线，不表示此时已经有一份可用 PRD。

适用完整或增量 PRD 时，M6-M7 必须拆为四个可独立验收的任务：

```text
RB-0 基线准备
→ 确定 baseline_type、PRD 的权威用途、模板、质量合同/DoD、评审人、批准人和保存位置

RB-1 PRD 草拟
→ 生成产品蓝图、用户需求、PRD Draft、RTM 和自检证据

RB-2 PRD 评审与批准
→ 按 RB-0 预先锁定的 DoD 评审，关闭阻断意见，批准一个明确版本

RB-3 PRD 基线登记
→ 登记批准版本、哈希、权威位置、批准记录和替代关系，状态改为 baselined
```

顺序是硬约束：先决定要形成哪类基线并锁定完成标准，再生成草案；草案生成后只能按既定标准检查，不能临时定义或降低 DoD。PRD 不能从一句话直接推导，必须经过产品目标、用户场景和用户需求层。

### 10.2 需求基线适用性判定

| `task_type` | RB-0 选择的目标 `baseline_type` | RB-3 最终应登记的权威需求基线 |
|---|---|---|
| `new_product` | `prd` | 完整产品蓝图、用户需求、PRD 和 RTM |
| `existing_product_change` | `incremental_prd` | 增量 PRD/变更规格、受影响用户故事和 RTM 差异 |
| `bugfix` | `problem_contract` | 问题合同、现有行为证据、期望行为、明确不改项和回归验收 |
| `refactor` | `regression_contract` | 行为不变量、影响范围、内部质量目标和回归合同 |
| `research` | `research_contract` | 研究问题、假设、实验方法、判定阈值、停止条件和结论格式 |

本元规范的目标是自动形成“适合当前任务的权威需求基线”，不是为每项工作机械制造长篇 PRD。用户的一句话是“设计企业表单设计器”时，按新产品处理，必须生成完整 PRD；小型 Bug、纯重构和研究任务不得伪装成产品需求。

第 10.2 节只决定“最终要形成哪一种权威基线”，不产生基线本身。只有对应草案经过评审批准，并由 RB-3 完成版本、哈希和权威位置登记后，该产物才是可供下游使用的需求基线。

### 10.3 RB-0：确定基线类型并预先锁定 PRD 质量合同

RB-0 是独立任务，必须在 RB-1 草拟 PRD 之前完成。

**DoR（开始 RB-0 前）**：

```text
[ ] RB-0 Task Contract 已建立，并预置本节的 DoR、DoD 和验证方法
[ ] Goal and Scope Contract 已批准
[ ] Workspace Contract 已批准
[ ] Policy Baseline 已由有权人员批准且状态为 `passed`；`bundle_sha256` 与批准证据可验证
[ ] task_type 及判断证据已记录
[ ] 目标、范围、非目标和主要场景足以判断需求基线类型
[ ] 需求负责人和业务决策人已明确
```

**必须动作**：

1. 按第 10.2 节确定 `baseline_type`，记录选择理由及不采用其他类型的理由；
2. 明确该 PRD 最终作为哪项产品或任务的权威需求基线，以及下游消费者；
3. 实例化 `PRD-Quality-Contract.yaml`，在其中预先锁定第 10.3.1 节的完成标准；
4. 明确草拟责任人、评审视角、评审人、最终批准人和职责边界；
5. 明确 PRD、用户需求、RTM、自检、评审、批准和基线记录的保存位置；
6. 建立输入清单、证据缺口以及 RB-1、RB-2、RB-3 Task Contract，三个任务初始状态均为 `not_started`。

**DoD（RB-0 完成条件）**：

```text
[ ] Baseline-Decision-Record 已记录 baseline_type、选择理由和权威用途
[ ] Policy Baseline 的版本、哈希和批准证据已冻结，并写入 PRD-Input-Manifest
[ ] PRD-Quality-Contract 已有唯一编号、版本和锁定时间
[ ] PRD 的草拟 DoD、评审批准 DoD、基线登记 DoD 已在生成草案前确定
[ ] 评审人、批准人、保存位置和验证方式明确
[ ] PRD-Input-Manifest 已区分可用输入、缺口、假设和阻断项
[ ] RB-1、RB-2、RB-3 Task Contract 均已建立，且各自 DoR/DoD 引用本质量合同
[ ] 无阻断 RB-1 的未知项
```

RB-0 未通过时不得生成 PRD Draft。质量合同在 RB-1 开始后如需修改，必须按第 3.6.3 节暂停、版本化、记录理由和批准人，再重新检查 RB-1 DoR。

#### 10.3.1 PRD Quality Contract（生成前锁定的 PRD DoD）

以下标准必须在 RB-0 写入 `PRD-Quality-Contract.yaml`，后续 RB-1、RB-2、RB-3 只执行和记录检查结果，不得事后创建完成标准。

**RB-1 草拟 DoD：**

```text
[ ] PRD 标识、适用产品/任务、草案版本、状态和变更历史完整
[ ] Goal and Scope Contract、Workspace Contract 和输入资料的版本引用可解析
[ ] 背景和问题有事实或证据支撑，推断与假设未伪装成事实
[ ] 产品目标、成功判断、目标用户、角色、主场景和异常场景明确
[ ] 本期 must/should/could、非目标、后续候选和决策来源明确
[ ] 每条功能需求有唯一 requirement_id，表达单一、可观察行为
[ ] 业务规则、状态、权限、数据边界、异常和适用的非功能阈值明确
[ ] 每条 must/should 需求具有可执行验收标准和系统测试要点
[ ] RTM 覆盖目标 → 场景/故事 → 需求 → 验收标准 → 证据位置
[ ] 高优先级孤儿需求、悬空故事、未映射验收项和未裁决冲突均为 0
[ ] 依赖、风险、假设和未知项有责任人、处理动作和阻断性判断
[ ] PRD Self Review 与 Review Pack 已生成，送审版本与自检版本一致
```

**RB-2 评审与批准 DoD：**

```text
[ ] 业务/产品、技术、测试及适用的安全/合规视角完成评审
[ ] L3/L4 的主要起草者与最终技术/安全评审者不是同一执行身份；身份和独立意见已记录
[ ] 评审意见均有编号、责任人、结论和证据
[ ] 所有阻断意见已关闭，接受的遗留风险有明确决策人
[ ] 修改后的 PRD、RTM、原型和相关产物不存在未裁决冲突
[ ] Approval Manifest 覆盖 PRD、用户需求、RTM、质量合同和自检证据等全部送审产物，并逐项记录 SHA-256
[ ] 有权负责人明确批准一个具体基线版本及其 approval_manifest_sha256
[ ] PRD-Approval.md 记录批准人、时间、版本、approval_manifest_sha256、附带条件和遗留风险
```

**RB-3 基线登记 DoD：**

```text
[ ] Approval Manifest 的版本和哈希与 RB-2 批准记录完全一致
[ ] Baseline Manifest 已包含 Approval Manifest、批准记录及其 SHA-256，不产生自引用
[ ] baseline_id、baseline_version、baseline_manifest_sha256、baselined_at 和权威保存位置已登记
[ ] 替代旧基线时，supersedes/superseded_by 关系和变更摘要已记录
[ ] 权威基线只读或受变更控制，后续修改必须产生新版本
[ ] 基线记录已形成稳定、可解析的引用，可供 M8 Task Startup Package 使用
```

上述条目全部有证据后，RB-3 Task Contract 才能由 `in_progress` 改为 `passed`；DoD 本身不得以“任务已经 passed”作为自己的前提。

#### 10.3.2 PRD 总体 DoD

只有 RB-0、RB-1、RB-2、RB-3 四个任务的 DoD 全部通过，PRD 工作包才算完成。RB-1 通过只表示“草案可送审”，RB-2 通过只表示“特定版本已获批准”，RB-3 通过才表示“该批准版本已经登记为当前权威需求基线”。

### 10.4 PRD 工作包任务 SOP（适用完整或增量 PRD 时）

| 任务 | DoR | 必须动作与输出 | 任务 DoD | 未通过时 | 下一任务 |
|---|---|---|---|---|---|
| RB-0 基线准备 | RB-0 Task Contract 已建立；Policy Baseline 为 `passed`；第 10.3 节 RB-0 DoR 全部通过 | 决定基线类型；锁定质量合同；指定责任人、评审人、批准人、路径和证据；创建 RB-1/RB-2/RB-3 Task Contract | 第 10.3 节 RB-0 DoD 全部通过 | 停留 RB-0；不得创建 PRD Draft | RB-1 |
| RB-1 PRD 草拟 | RB-1 Task Contract 已建立；RB-0 `passed`；Policy Baseline 版本/哈希未变化且仍为 `passed`；质量合同版本已锁定；输入清单可用；无阻断缺口 | 按 P1-P8 生成产品蓝图、用户需求、PRD Draft、RTM、自检和 Review Pack | 第 10.3.1 节 RB-1 草拟 DoD 全部通过并有证据 | 修复草案；若上游目标错误则返回 M2/M3 | RB-2 |
| RB-2 PRD 评审与批准 | RB-2 Task Contract 已建立；RB-1 `passed`；送审版本、评审人和批准人明确 | 多视角评审；关闭意见；记录风险决定；生成 Approval Manifest 并批准具体版本和 approval_manifest_sha256 | 第 10.3.1 节 RB-2 DoD 全部通过并有证据 | `rejected` 返回 RB-1；不得进入基线登记 | RB-3 |
| RB-3 PRD 基线登记 | RB-3 Task Contract 已建立；RB-2 `passed`；批准记录、版本和 approval_manifest_sha256 可验证 | 核验 Approval Manifest，加入批准记录形成 Baseline Manifest；登记基线标识、版本、权威位置和替代关系 | 第 10.3.1 节 RB-3 DoD 全部通过并有证据 | 保持 `approved`，修复登记；不得声称 `baselined` | M8 |

RB-1 内部按以下操作步骤执行。这些是同一个草拟任务中的操作，不机械拆成八份独立任务合同；当其中某一步被单独分配、可独立验收或跨责任边界时，再按第 3.6 节提升为子任务。

| 步骤 | AI 动作 | 输出/检查点 | 不通过时 |
|---|---|---|---|
| P1 汇集批准输入 | 读取 Goal and Scope Contract、Workspace Contract、质量合同、已有产品基线、证据和批准记录 | `PRD-Input-Manifest`，每项输入含版本、状态和引用 | 返回最早缺失的 M3、M5 或 RB-0，不得从一句话补写 |
| P2 建立产品与用户模型 | 归纳业务目标、相关角色、真实场景、痛点、期望结果；分开事实、推断、假设 | 产品蓝图、角色与场景清单 | 关键角色或主场景不清时回到 M2/M3 |
| P3 形成用户需求 | 将主场景写成用例和用户故事，补主流程、异常流、业务价值和验收候选 | 用户需求文档；每个故事有唯一 `story_id` | 没有来源或价值的故事删除或降为候选 |
| P4 定义版本边界 | 明确本期范围、优先级、非目标、后续候选、依赖和发布边界 | 版本范围表；`must/should/could` 有决策来源 | 重大取舍逐项请求人工决定，不得由 AI 猜测 |
| P5 编写可验收需求 | 将用户需求转成单一可观察行为；补业务规则、状态、权限、异常、边界、数据和集成要求 | PRD Draft；每条需求有唯一 `requirement_id` | 拆分复合需求；把不可验证词语改成具体行为或阈值 |
| P6 补非功能与运行约束 | 按场景识别性能、可用性、安全、隐私、审计、兼容、迁移和回滚要求 | 可测量 NFR 与运行约束 | 无证据的数值标为待决策，不伪造阈值 |
| P7 建立验收与追溯 | 为每条 `must/should` 需求建立 Given/When/Then 或等价验收；关联目标、故事、需求、系统测试要点和证据 | RTM；孤儿项数量为 0 | 缺映射或验收项时回到 P3-P6 |
| P8 自检与送审 | 按已锁定的 PRD Quality Contract 检查一致性、完整性和可测性；生成变更摘要、决策清单和风险摘要 | PRD Self Review、Review Pack，状态改为 `under_review` | RB-1 DoD 未全部通过不得送 RB-2 |

AI 不得因为模板字段多而一次性向用户询问全部内容。能从已批准资料和授权工作区查证的先自动查证；只有产品取舍、事实纠正、风险接受和批准需要用户回答。

### 10.5 用户需求产物（适用 PRD 时）

每个用户故事至少包含：

```yaml
story_id: STORY-001

scenario: 真实业务场景
goal: 用户希望完成的任务
business_value: 为什么有价值
preconditions: []
main_flow: []
exception_flows: []
acceptance_criteria: []
source_refs: []
status: draft
```

### 10.6 PRD 标准目录与必备内容

PRD 正文至少使用以下结构。允许按产品规模合并相邻章节，但不得省略对应信息和追溯关系：

```text
1. 文档控制
   - PRD 标识、产品/任务标识、版本、状态、作者、负责人
   - 变更历史、输入基线、评审人和批准记录引用
2. 背景与问题
   - 业务背景、当前问题、证据、受影响角色和现状基线
3. 产品目标与成功判断
   - 目标、可观察业务结果、指标口径、基线值/目标值/观察窗口
4. 用户、角色与场景
   - 角色职责、主场景、前置条件、期望结果和异常场景
5. 本期范围
   - `must/should/could`、非目标、后续候选、版本边界和决策来源
6. 用户旅程与关键流程
   - 主流程、异常流、状态变化、关键原型引用和未决交互
7. 功能需求
   - 唯一 `requirement_id`、来源故事、行为、规则、异常、验收标准
8. 业务规则、状态与权限
   - 规则优先级、状态机约束、角色权限、数据可见范围和审计要求
9. 数据与外部集成边界
   - 业务数据对象、所有权、输入输出、外部系统、兼容和迁移约束
10. 非功能需求
    - 性能、容量、可用性、安全、隐私、审计、可访问性及可测量阈值
11. 发布、迁移与回滚
    - 发布条件、兼容要求、迁移对象、回滚触发条件和数据恢复原则
12. 依赖、风险、假设和未知项
    - 责任人、影响、验证动作、截止点和是否阻断下一阶段
13. 验收与追溯
    - 系统验收目标、需求验收标准、测试要点和 RTM 引用
```

### 10.8 等价需求基线最小合同

不适用完整 PRD 时，AI 必须生成以下通用合同，并按 `baseline_type` 填写对应专有字段：

```yaml
baseline_id: BASELINE-001
baseline_type: incremental_prd|problem_contract|regression_contract|research_contract
task_type: existing_product_change|bugfix|refactor|research
objective: 期望的可观察结果
scope: []
non_goals: []
source_refs: []
constraints: []
acceptance_or_decision_criteria: []
traceability:
  goal_and_scope_ref: Goal-and-Scope-Contract.yaml
  evidence_refs: []
  verification_refs: []
status: draft

type_specific:
  incremental_prd:
    changed_user_stories: []
    changed_requirements: []
    compatibility_requirements: []
  problem_contract:
    observed_behavior: 有复现或证据的现状
    expected_behavior: 有来源的期望行为
    affected_baseline: 版本或环境
    invariants_and_non_regression: []
  regression_contract:
    behavior_invariants: []
    affected_surface: []
    internal_quality_objective: 可测量的内部质量目标
    regression_scope: []
  research_contract:
    research_question: 待回答问题
    hypotheses: []
    experiment_method: []
    evidence_threshold: 判定阈值
    budget_and_stop_condition: 时间、成本边界和停止条件
```

`type_specific` 只保留与 `baseline_type` 匹配的分支。每个合同都必须说明“怎样证明结果成立”；只有研究合同允许以“获得否定结论”为成功结果，但仍必须满足预先定义的实验和证据阈值。

等价需求基线不得绕过任务级治理。它们复用 RB-0 → RB-1 → RB-2 → RB-3 的顺序，但把 `PRD-Quality-Contract` 替换为与类型匹配的质量合同：

| `baseline_type` | RB-0 预先锁定的质量重点 | RB-1 草拟结果 | RB-2 评审重点 | RB-3 登记结果 |
|---|---|---|---|---|
| `incremental_prd` | 变更范围、兼容边界、旧基线引用、增量验收 | 增量 PRD、受影响故事、RTM 差异 | 业务变更、技术影响、测试回归和兼容评审 | 已批准增量版本及其哈希、权威位置、替代关系 |
| `problem_contract` | 复现证据、现状/期望、不改项、回归标准 | 问题合同和回归验收草案 | 期望行为、影响范围和回归风险评审 | 已批准问题合同的版本、哈希和权威位置 |
| `regression_contract` | 行为不变量、影响面、内部质量目标、回归范围 | 回归合同草案 | 不变量、依赖面、测试充分性和回滚评审 | 已批准回归合同的版本、哈希和权威位置 |
| `research_contract` | 研究问题、假设、方法、判定阈值、预算和停止条件 | 研究合同草案 | 方法有效性、证据阈值、成本和风险评审 | 已批准研究合同的版本、哈希和权威位置 |

每种类型都必须在 RB-0 建立四个 Task Contract，并在开始 RB-1 前锁定其草拟、评审批准和基线登记 DoD。不得因为“不写 PRD”而在工作完成后补验收标准。

### 10.9 AI 禁止事项

- 不得把竞品功能自动变成需求；
- 不得把自己的建议写成用户已确认范围；
- 不得写“功能正常、交互友好、性能良好”等不可验收表述；
- 不得在 PRD 中提前锁定数据库、框架或类设计；
- 不得为追求“完整”加入没有用户故事来源的功能；
- 不得把未验证能力列为必须交付；
- 不得自行将任何需求基线状态改为 approved。

### 10.10 输出与门禁

新产品或重大产品能力保存以下文件到 Workspace Contract 指定位置：

```text
Baseline-Decision-Record.yaml
PRD-Quality-Contract.yaml
PRD-Input-Manifest.yaml
01-业务需求-产品蓝图.md
02-用户需求-用例与用户故事.md
PRD-产品需求规格说明书.md
需求追溯表.md
PRD-Self-Review.md
PRD-Approval.md
PRD-Baseline-Record.yaml
```

需求内容文件初始状态为 `draft`；RB-0 产物按其 Task Contract 记录状态；`PRD-Approval.md` 和 `PRD-Baseline-Record.yaml` 在对应任务开始前只能是空框架或 `not_started`，不得预填通过结论。

对于不适用完整 PRD 的任务，输出第 10.2 节规定的等价需求基线和追溯记录，并在 `Next-Activity-Contract.yaml` 中记录基线类型和不生成 PRD 的理由。等价基线初始状态同样为 `draft`。

`PRD-Approval.md` 只能由 RB-2 记录评审结论、批准人、时间、批准版本、`approval_manifest_sha256`、附带条件和遗留风险；`PRD-Baseline-Record.yaml` 只能由 RB-3 在核验 Approval Manifest、批准记录及最终 `baseline_manifest_sha256` 后登记，不得把“已批准”直接当成“已基线化”。

**Requirement Baseline Draft Gate**：适用完整或增量 PRD 时，只有 RB-0 DoD 和第 10.3.1 节 RB-1 草拟 DoD 全部通过，才能送入 M7 执行 RB-2；多视角评审、风险接受和人工批准只能在 RB-2 完成，基线登记只能在 RB-3 完成。不适用 PRD 时，也必须先建立与基线类型匹配的质量合同和 Task Contract；只有问题/假设、边界、期望结果、验证方式、追溯和停止条件完整后，才能送入 M7。草案不能自批，批准状态不能替代基线登记。

## 11. M7：PRD 或等价需求基线评审、批准与基线登记

### 11.1 AI 自检

```text
[ ] 基线类型是否与 task_type 一致
[ ] 目标、范围和非目标与 Goal and Scope Contract 一致
[ ] 每项待交付行为、问题或研究假设有唯一编号
[ ] 每项内容能追溯到目标、场景、缺陷证据或研究问题
[ ] 高优先级内容有业务决定或证据来源
[ ] 每项内容均有可执行验收标准、回归标准或判定阈值
[ ] 权限、状态、异常和边界已覆盖
[ ] 假设、未知和确认事实已分开
[ ] 没有孤儿条目、重复、矛盾或不可验证表述
[ ] 原型与适用的需求基线没有未裁决冲突
[ ] 追溯记录覆盖本期全部高优先级内容
[ ] 适用 PRD 时，每条功能需求追溯到用户故事，每个用户故事追溯到目标或真实场景
[ ] 不适用 PRD 时，现状/期望、不变量或实验停止条件已按任务类型完整记录
```

### 11.2 多视角评审

| 视角 | 检查内容 |
|---|---|
| 用户/业务 | 问题、场景、规则和结果是否真实 |
| 目标/需求负责人 | 目标、范围、优先级和非目标是否正确 |
| 技术 | 是否存在明显不可行、冲突或重大依赖 |
| 测试 | 每条需求是否明确、可观察、可验证 |
| 安全/合规 | 涉及权限、隐私、租户、支付等边界是否充分 |

### 11.3 评审交互

AI 不应要求用户逐字审阅整篇长文后自行找问题。AI 应：

1. 先展示需求基线摘要、基线类型和范围差异；
2. 列出需要人决定的少数关键项；
3. 每次只确认一个重大决定；
4. 自动修改草案并重新运行一致性检查；
5. 最后展示可执行验收和未决风险摘要。

### 11.4 RB-2：Requirement Approval Gate

需求基线状态：

```text
draft → under_review → approved
                    ↘ rejected → draft
approved → baselined
baselined → superseded → draft/under_review → approved → baselined
```

只有用户或有权目标/需求负责人明确批准后，才能写入：

```yaml
status: approved
baseline_type: prd|incremental_prd|problem_contract|regression_contract|research_contract
approved_version: 版本
approved_manifest_ref: Approval-Manifest.yaml
approval_manifest_sha256: 被批准的送审清单哈希
approved_by: 决策人
approved_at: 时间
```

上述记录只证明 RB-2 已通过。适用 PRD 时 PRD 未批准前，不得进入 RB-3；不适用 PRD 时，等价需求基线未批准前同样不得进入基线登记。AI 不得把沉默、无异议或 AI 自检当作批准。

### 11.5 RB-3：Requirement Baseline Registration Gate

RB-3 只能登记 RB-2 已明确批准的具体基线版本，不再评审需求内容，也不得修改质量标准。AI 必须重新计算 Approval Manifest 中每个送审产物的 SHA-256 和 `approval_manifest_sha256`，与批准记录逐字核对；随后将 Approval Manifest 和批准记录加入 Baseline Manifest，再写入：

```yaml
status: baselined
baseline_id: BASELINE-唯一编号
baseline_type: prd|incremental_prd|problem_contract|regression_contract|research_contract
baseline_version: 与批准版本一致
approval_manifest_ref: Approval-Manifest.yaml
approval_manifest_sha256: 与批准记录一致
baseline_manifest_ref: Baseline-Manifest.yaml
baseline_manifest_sha256: 最终基线清单哈希
authority_location: 权威保存位置
artifacts:
  - path: 权威产物路径
    sha256: 文件 SHA-256
approval_ref: PRD-Approval.md 或等价批准记录
quality_contract_ref: PRD-Quality-Contract.yaml 或等价质量合同
dod_evidence_refs: []
baselined_by: 登记责任人
baselined_at: 时间
supersedes: 可为空
```

RB-3 的 DoD 使用第 10.3.1 节预先锁定的“基线登记 DoD”。所有条目通过且证据已记录后，状态才能从 `approved` 改为 `baselined`。以下任何情况都不算 PRD 工作包完成：

- PRD 仍为 `draft` 或只有 AI 自检，没有人工批准；
- 只有功能清单，没有用户、场景、业务规则或可执行验收标准；
- 使用“支持完善、性能良好、体验友好”等无法验证的表述；
- 高优先级需求没有来源、存在孤儿需求或 RTM 只是空表；
- 原型与 PRD 冲突但尚未裁决；
- 阻断未知项仍开放，或者批准记录没有对应明确版本；
- 批准版本、Approval Manifest 逐文件 SHA-256 或 `approval_manifest_sha256` 不一致；
- Baseline Manifest 未包含批准记录，或 `baseline_manifest_sha256` 无法复算；
- PRD 已批准但尚未完成 RB-3 基线登记；
- AI 把自己的推荐、竞品能力或行业惯例写成了用户已批准需求。

## 12. M8：后续活动判定与任务移交

### 12.1 后续活动判定

| `task_type` | 默认下一活动 | 启动条件 |
|---|---|---|
| `new_product` | `architecture`：架构与总体设计 | PRD 已完成 RB-3 基线登记；关键未知已登记；系统验收目标存在；首个架构任务 Task Contract 的 DoR/DoD 已预置 |
| `existing_product_change` | `impact_analysis`：现状与影响分析 | 增量 PRD 已基线化；旧行为证据、兼容边界、回滚原则和验收基线存在；首个影响分析任务 Task Contract 已建立 |
| `bugfix` | `characterization` 或 `implementation` | 问题合同已基线化；旧行为尚未复现或隐性契约未知时先刻画；直接实现时证据、期望行为、影响范围、回归标准、代码目录和首个实现任务合同均明确 |
| `refactor` | `impact_analysis` | 回归合同已基线化；行为不变量、依赖面、回归范围、代码目录、回滚方式和首个影响分析任务合同明确 |
| `research` | `spike` | 研究合同已基线化；假设、实验方法、通过阈值、时间/成本预算、停止条件及首个实验任务合同明确 |

若任务同时包含 UI/交互设计，在主活动中加入 `prototype` 子活动；若新产品或产品变更的能力边界仍不完整，先把 `first_action` 设为能力建模，不得直接进入架构或切片。

AI 输出活动类型和启动条件，不写死其他规范或流程文件的名称。组织已有流程时，可在移交后由执行者选择适用规范，但不能改变本元规范已经批准的目标、边界、目录和验收基线。

### 12.2 启动基线映射

M0-M7 已形成后续活动需要的完整基线：

```text
M0-M3 Goal and Scope Contract
→ 为什么做、为谁做、做什么和不做什么

M4-M5 Workspace Contract
→ 各类资料、源码、任务和证据在哪里

M6-M7 PRD 或等价需求基线及其追溯记录
→ 要交付、修复、保持或验证什么，以及怎样判定结果
```

后续活动必须先校验这些产物的引用、基线状态、批准版本、逐文件 SHA-256、`approval_manifest_sha256`、`baseline_manifest_sha256` 和 Task Contract。证据有效时直接复用，不得重复访谈或重写需求基线；存在缺失、失效或冲突时，只重开最早受影响的元阶段或任务切片。

### 12.2.1 后续任务类型化最低质量合同

M8 只负责为第一项可执行任务实例化质量合同，不代替后续阶段执行。通用 Task Contract 不能使用“完成开发”“测试通过”等空泛 DoD；必须按任务类型合并以下最低项。某项不适用时记录事实依据，不得删除字段。

#### A. 架构与设计任务

```text
DoR：需求基线、Policy Baseline、约束/不变量、规模量级、现状架构与重大未知可解析。
DoD：组件边界、数据流/控制流、接口与数据契约、失败模式、安全边界、容量假设、兼容/迁移/回滚方案完整；单向门决策有 ADR；关键假设有 Spike 或证据；技术、测试及适用安全视角评审通过。
证据：设计产物、ADR、评审记录、契约校验或适应度函数位置。
```

#### B. 正式任务切片与计划任务

```text
DoR：批准的设计/影响分析、需求追溯、依赖图、代码边界、系统验收目标可解析。
DoD：每个切片只有一个主要结果；满足第 3.6.4 节 Slice Budget；依赖和执行顺序明确；每个切片预置类型化 DoR/DoD、责任人、验证方法、回滚单位和父级验收映射；没有孤儿需求或无法验证切片。
证据：切片清单、依赖关系、需求到切片到测试的追溯矩阵、切片预算检查结果。
```

#### C. 编码、缺陷修复与重构任务

```text
DoR：需求/问题/回归基线、Policy Baseline、现有行为证据、影响面、调用方、数据/接口契约、兼容不变量、代码目录、回滚方法和测试方案明确；新增或改变行为时，先定义最小失败测试或可重复失败验证。
DoD：最小范围实现完成；旧逻辑和兼容边界按合同保留；失败测试经历红→绿；单元、集成及相关回归测试通过；lint/typecheck/静态分析按项目能力通过；代码评审阻断项关闭；没有未批准依赖和秘密；验证命令、环境、退出码、报告、变更文件、未覆盖风险和回滚步骤已记录。
证据：红灯与绿灯结果、构建/测试/静态检查输出、代码评审、差异清单、回滚验证或可执行回滚说明。
```

#### D. 安全与供应链任务/门禁

```text
DoR：资产、数据分类、信任边界、权限/租户模型、外部输入、依赖和制品来源明确。
DoD：适用的威胁建模、权限/租户隔离测试、密钥扫描、SAST、SCA、许可证检查和依赖锁定完成；适用时生成 SBOM、制品签名/来源证明；高危问题为 0，接受的例外有责任人、期限和补偿控制。
证据：工具版本、规则集、扫描命令、报告路径、漏洞/许可证处置记录、SBOM 与制品标识。
```

#### E. 集成、系统验收与性能任务

```text
DoR：所有输入切片通过、部署/测试环境和数据准备完成、系统验收目标及阈值已基线化。
DoD：端到端主路径、异常、边界、权限、兼容、迁移和回滚场景通过；性能/容量阈值适用时通过；失败用例已处置或有批准例外；需求→切片→测试→证据追溯闭合。
证据：用例范围、执行命令、通过/失败数量、报告、性能结果、遗留风险与批准记录。
```

#### F. 发布、回滚与运维任务

```text
DoR：发布制品不可变且可追溯；前置质量门禁通过；发布窗口、权限、备份、迁移、灰度和回滚责任人明确。
DoD：发布/灰度结果、迁移校验和冒烟测试通过；回滚触发条件与步骤可执行并已演练或验证；结构化日志、关键指标、告警、追踪和 Runbook 按风险要求存在；观察窗口结束并记录业务/技术指标；失败时已回滚或进入事故流程。
证据：制品摘要、部署记录、迁移和冒烟报告、监控截图/查询、告警测试、回滚证据和观察期结论。
```

M8 根据 `next_activity.type` 选择对应质量画像并写入 `first_task_contract_ref`：

| `next_activity.type` | 必须应用的质量画像 |
|---|---|
| `architecture` | A 架构与设计；触及安全/供应链时叠加 D |
| `impact_analysis` | A 架构与设计中的现状/影响/契约/回滚项；输出切片时叠加 B |
| `slicing` | B 正式任务切片与计划 |
| `implementation` | C 编码、缺陷修复与重构；按风险叠加 D、E |
| `prototype` | A 架构与设计中的交互边界、状态和异常；DoD 追加目标场景走查、可用性/可访问性检查和原型证据，不得冒充生产实现 |
| `characterization` | C 的现有行为证据、失败复现、兼容不变量和回归测试分支；不得改变生产行为 |
| `spike` | 第 5.6 节受控 Spike 专用合同；基线后的研究 Spike还须引用 Research Contract |

若第一任务跨多个类型，必须拆分或同时满足所有命中画像；不得以“后续流程再补”为由留下空白。

`Approval-Manifest.yaml` 必须列出 PRD/等价基线、用户需求、RTM、质量合同和送审 DoD 证据等全部送审产物，每项使用 SHA-256；它不包含批准记录，避免自引用。人工批准必须绑定规范化后的 `approval_manifest_sha256`。RB-3 再将 Approval Manifest 和批准记录加入 `Baseline-Manifest.yaml`，计算 `baseline_manifest_sha256`。两种清单都必须定义路径规范化、字段排序和哈希算法。

`execution_reproducibility` 的六类字段不得使用空数组或空值。每类至少记录一个可解析值；执行环境确实无法取得时，必须保留该项并填写具体 `not_available_reason`，同时记录已采用的替代复现证据。L3/L4 的不可取得项还须由独立评审者接受并留证，不能由执行 AI 自行豁免。

存在可用 Schema/校验器时，`machine_validation.status` 必须为 `passed`；不可用时必须写明原因，并由独立评审者逐项检查等价门禁，不能留空或由起草 AI 自证通过。

M8 送审前可以先生成 `status: awaiting_human` 的启动包草案。正式移交包中四类 `baseline_mappings` 必须分别覆盖 `intent`、`workspace`、`requirements` 和 `acceptance`，且全部为 `passed`；任何 `partial` 或 `missing` 都属于缺口清单，只能记录在 Execution State 中并将 `first_action` 指向最早缺口，不能把启动包标记为 `passed` 或移交后续活动。

对于“设计企业表单设计器”这类新产品，如果 M0-M7 已全部通过，推荐移交为：

```yaml
next_activity:
  type: architecture
  entry_conditions:
    - PRD、RTM、批准记录和基线登记记录完整，状态为 baselined
    - 已确认工作空间及源码候选边界
    - 关键产品未知项不阻断总体设计
    - 系统级验收目标已定义
  first_task_contract_ref: architecture-task-contract.yaml
  first_action: 读取启动包并建立架构问题、约束和风险清单
```

若启动基线存在缺口，启动包状态必须为 `awaiting_human`、`blocked` 或 `in_progress`（按实际原因选择），`first_action` 必须指向最早缺口，不得固定进入架构或实现。

### 12.4 移交门禁

```text
[ ] Goal and Scope Contract 已批准
[ ] Workspace Contract 已批准
[ ] 与任务类型匹配的需求基线先经批准，再完成基线登记，当前状态为 baselined；批准记录绑定 Approval Manifest 哈希，Baseline Manifest 包含批准记录
[ ] 适用 PRD 时，PRD 和 RTM 已基线化且无高优先级孤儿需求或悬空故事
[ ] 不适用 PRD 时，问题/回归/研究合同及其验证方式已基线化
[ ] 基线类型、质量合同及每个独立任务切片的 DoR/DoD 均在对应任务开始前确定
[ ] 任务类型、复杂度和风险已识别
[ ] 下一类研发活动及其入口条件明确
[ ] 下一项可执行任务已有 Task Contract，DoR、DoD、责任人和验证方式明确
[ ] 第一任务引用命中的类型化质量画像，DoR 全部通过或明确保持 pending/blocked，未通过时禁止执行
[ ] Policy Baseline 已冻结且状态为 passed，没有未裁决强制规则冲突
[ ] Schema/校验器已通过；不可用时有原因和独立人工等价检查证据
[ ] 已完成基线具有产物与门禁证据映射
[ ] 未解决问题不阻断下一阶段
[ ] 后续执行者明确知道哪些行为仍被禁止
```

通过 Startup DoD 只表示“可以尝试进入第一任务”，不授权绕过第一任务门禁。AI 必须重新读取 `first_task_contract_ref`，验证合同版本、Policy Baseline、输入哈希和全部 DoR 证据；只有 `dor_status: passed` 才能执行 `first_action`。状态为 `pending/failed/blocked` 时必须停留在任务入口并处理对应缺口，不得让用户重新描述已经确认的需求。

### 12.5 任务启动 Definition of Done（Startup DoD）

本 DoD 是本元规范的整体完成条件。它验收的是“任务已从一句话变成可受控执行的研发启动基线”，不是验收产品已经开发完成。

```text
[ ] 原始一句话需求已无损保存，事实、推断、假设、未知和决定可区分
[ ] Goal and Scope Contract 已由有权人员批准，目标、角色、场景、范围、非目标和成功判断完整
[ ] Workspace Contract 已由用户或有权负责人批准，产品、需求、原型、源码、任务、证据和知识来源的位置与职责明确
[ ] 任务类型、复杂度、风险原因和升级条件已有证据或明确判断依据
[ ] 与任务类型匹配的 PRD 或等价需求基线已按预置质量合同完成草拟、评审批准和基线登记，状态为 baselined
[ ] 适用 PRD 时，RB-0、RB-1、RB-2、RB-3 Task Contract 及对应 DoD 证据完整
[ ] 目标、场景、需求、验收与证据位置之间的追溯闭合，不存在 `partial` 或 `missing` 映射
[ ] 所有已批准产物及批准记录引用可解析，版本/输入哈希能够证明彼此对应
[ ] Approval Manifest 覆盖全部送审产物并与批准记录一致；Baseline Manifest 包含 Approval Manifest 和批准记录，两个清单哈希均可复算
[ ] Policy Baseline、执行指纹和机器/人工校验记录已进入启动包
[ ] 开放问题均有责任人和下一动作，且没有阻断下一活动的未知项
[ ] 下一活动类型、第一项可执行任务的 Task Contract、类型化质量画像、DoR、DoD、责任人、验证方式和仍被禁止的行为已明确
[ ] Task Startup Package 的 `intent`、`workspace`、`requirements`、`acceptance` 四类映射全部为 `passed`
[ ] Startup DoD 检查证据已保存，Task Startup Package 状态为 `passed`
```

以下任一情形出现时，本元规范不得声明完成，也不得进入正式架构、切片或编码：

- 目标与边界只是 AI 的理解，尚未获得人工确认；
- 工作目录只是候选方案，尚未确认资产职责、授权或源码边界；
- PRD 或等价需求基线仍是草案、被退回或由 AI 自行批准；
- PRD 虽已批准，但尚未完成版本、哈希和权威位置的基线登记；
- Approval Manifest 未覆盖全部送审产物、批准记录未绑定其哈希，或 Baseline Manifest 未包含批准记录；
- Policy Baseline 缺失、失效或存在未裁决冲突；
- 任一可独立任务切片在开始后才补写或降低 DoD；
- RTM、验收标准或批准记录缺失，或者引用无法解析到对应版本；
- 启动包存在 `partial`、`missing`、空引用或阻断级开放问题；
- 只写了“下一步做架构/切片/开发”，却没有第一任务的 Task Contract、类型化质量画像、DoR、DoD 和验证方法；
- 把“任务启动完成”表述成“产品研发完成”。

若 Startup DoD 未通过，AI 必须列出未通过项、证据缺口、责任人和最小修复动作，并把执行状态恢复到最早受影响的 M0-M8 阶段。

## 13. 工作空间权威与冲突处理

### 13.1 权威来源

| 主张 | 默认权威来源 |
|---|---|
| 任务为什么做、范围是什么 | Approved Goal and Scope / Requirement Baseline |
| 用户怎样操作和看到什么 | Approved Prototype + Interaction Spec |
| 重大架构为什么这样选择 | Accepted ADR |
| 接口和数据契约是什么 | Approved Design/Schema Contract |
| 系统当前实际上怎样运行 | Source Code + Tests + Runtime Evidence |
| 本次任务是否完成 | Evidence Pack + Acceptance Report |

### 13.2 冲突协议

发现 PRD、原型、设计、源码或测试互相冲突时：

```text
记录冲突和各自版本
→ 判断冲突影响的主张
→ 确定对应权威来源
→ 无法自动裁决时进入 awaiting_human
→ 人工作出决定
→ 同步更新所有受影响产物
→ 重开相应验证
```

禁止 AI 静默选择自己更容易实现的一方。

## 14. 断点续跑与变更

### 14.1 持久化时点

M0 开始后 AI 必须维护临时 Session State，防止 M2/M3 多轮访谈因会话中断丢失。优先使用当前授权环境提供的会话状态目录；没有可写位置时，在对话中输出可复制恢复的结构化检查点，并明确尚未落盘。临时状态不得包含未授权秘密或敏感原文。

```yaml
session_execution_id: SESSION-YYYYMMDD-NNN
storage: 临时状态路径或 conversation_checkpoint
active_stage: M0-M5
status: in_progress
original_request: 原始需求
facts: []
inferences: []
hypotheses: []
decisions: []
open_questions: []
alignment_log: []
candidate_workspace: []
input_hashes: []
updated_at: 时间
```

M5 工作空间确认后，AI 必须把临时 Session State 迁入正式 Execution State，记录迁移来源和迁移前哈希，并把 M0-M4 的确认结果写入正式目录。此后每个阶段更新：

```yaml
startup_execution_id: META-YYYYMMDD-NNN
imported_session_ref: 临时状态引用
imported_session_sha256: 迁移前哈希
active_stage: M0-M8
status: in_progress
input_hashes: []
facts: []
inferences: []
hypotheses: []
decisions: []
open_questions: []
artifact_refs: []
next_question_or_action: 下一步
```

### 14.2 恢复规则

恢复时：

1. M5 前读取 Session State，M5 后读取正式 Execution State；
2. 校验已批准合同和输入哈希；
3. 从第一个非 `passed` 且依赖满足的阶段继续；
4. 不重复询问已有明确答案；
5. 基线变化时把受影响阶段标记 `superseded`；
6. 临时状态与正式状态冲突时保留两者，按权威来源裁决，不静默覆盖。

### 14.3 用户改变想法

用户可以在任何阶段修正目标。AI 应：

```text
记录新决定
→ 标记旧决定 superseded
→ 影响分析 Goal and Scope、Workspace、需求基线和后续产物
→ 只重开受影响阶段
→ 保留历史决策，不覆盖证据链
```

## 16. 失败模式与禁止行为

| 失败模式 | 禁止行为 | 正确处理 |
|---|---|---|
| 用户只说一句话 | 直接生成完整正式 PRD | 进入逐问式需求发现 |
| 用户说不清 | 要求用户提交完整需求清单 | 给真实案例、选项或原型 |
| 术语有歧义 | AI 选择默认理解 | 暴露分叉并确认 |
| 未知工作目录 | 直接要求用户给路径 | AI 先搜索并提出候选方案 |
| 找到同名目录 | 自动认定为正式项目 | 展示证据并请求确认 |
| 工作空间未确认 | 创建生产源码仓库 | 只读调查，不写入 |
| PRD 完成标准尚未锁定 | 先写 PRD，完成后再补 DoD | 先完成 RB-0 并版本化质量合同 |
| PRD 很完整 | AI 自行标记 approved | 保持 draft，人工批准 |
| PRD 已获批准 | 直接写成 baselined | 执行 RB-3，核对版本和哈希并登记权威位置 |
| 小任务很简单 | 执行后再补 DoR/DoD | 在任务进入 in_progress 前建立轻量 Task Contract |
| 竞品有某功能 | 自动加入最高优先级 | 进入候选并验证业务价值 |
| 原型很漂亮 | 认为需求已确认 | 用真实场景走查并记录决定 |
| 用户改变范围 | 覆盖旧内容继续 | 记录 superseded 并做影响分析 |
| 后续活动明确 | 仍询问“是否继续” | 按启动包继续，只有决策门禁才暂停 |

## 19. 版本历史

### 19.1 规范生命周期治理

本文从 `draft` 变为 `approved/effective` 前，必须由有权流程治理人员完成以下记录；AI 只能生成候选和验证证据，不能自行批准本规范：

```yaml
standard_id: AI-RD-META-STARTUP
version: 1.3
owner: 规范所有者
approved_by: 批准人
approved_at: 批准时间
effective_from: 生效时间
applies_to: 适用组织、项目和 AI 执行环境
review_due_at: 下次复审时间，不晚于生效后 12 个月
supersedes: 被替代版本
change_control: 修改、紧急例外和废止流程
status: draft|approved|effective|deprecated|superseded
```

规范修改必须生成新版本、变更摘要、影响分析和回归验证；不得覆盖已生效版本。紧急例外必须记录批准人、范围、期限、风险和补偿控制。

### 19.2 规范自身 Definition of Done
