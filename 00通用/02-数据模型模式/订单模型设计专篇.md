# 订单模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦订单建模。订单是交易系统的中枢：向上承接商品、价格、促销，向下驱动支付、库存、履约、结算、发票。订单模型的质量直接决定整个交易链路能不能演进。

## 0. 一句话总纲

> 订单不是一张表，而是"一次交易承诺"的完整证据链：买了什么、什么价、怎么付、怎么送、改过什么、退过什么。

## 1. 订单的骨架：一单多层

最小完整结构不是"订单 + 明细"两张表，而是五个部分：

```text
订单主表（order）          —— 交易整体：买家、状态、应付/实付总额、下单时间
订单行（order_line）       —— 买了什么：SKU、数量、单价、行金额、行状态
金额分解（order_amount）   —— 钱怎么构成：商品额、运费、税、优惠明细、分摊结果
支付记录（order_payment）  —— 怎么付的：支付方式、金额、渠道单号（一单可多笔组合支付）
履约记录（order_fulfillment）—— 怎么交付：包裹、物流单、发货行与数量
```

**为什么行级如此重要**：部分发货、部分退款、行级开票、按行分摊优惠、按行结算佣金——成熟交易的一切复杂性都发生在行级。主表只存汇总，任何计算以行为准（呼应总览篇"主表表达整体，明细表达可计算颗粒"）。

```sql
CREATE TABLE ord_order (
  id             BIGINT PRIMARY KEY,
  order_no       VARCHAR(32) NOT NULL,
  buyer_id       BIGINT NOT NULL,
  shop_id        BIGINT NOT NULL,
  order_status   VARCHAR(24) NOT NULL,
  goods_amount   DECIMAL(18,4) NOT NULL,   -- Σ行金额
  freight_amount DECIMAL(18,4) NOT NULL,
  discount_amount DECIMAL(18,4) NOT NULL,  -- Σ优惠
  payable_amount DECIMAL(18,4) NOT NULL,   -- 应付 = 商品+运费-优惠
  paid_amount    DECIMAL(18,4) NOT NULL DEFAULT 0,
  buyer_remark   VARCHAR(500),
  ordered_at     DATETIME NOT NULL,
  version        BIGINT NOT NULL DEFAULT 0,
  UNIQUE KEY uk_order_no (order_no),
  KEY idx_buyer_time (buyer_id, ordered_at),
  KEY idx_shop_status_time (shop_id, order_status, ordered_at)
);

CREATE TABLE ord_order_line (
  id             BIGINT PRIMARY KEY,
  order_id       BIGINT NOT NULL,
  sku_id         BIGINT NOT NULL,
  -- ↓ 交易快照：下单时固化，永不随主数据变化（总览篇 §11）
  sku_title      VARCHAR(255) NOT NULL,
  sku_spec       VARCHAR(255),
  sku_image      VARCHAR(255),
  category_snapshot VARCHAR(64),
  list_price     DECIMAL(18,4) NOT NULL,   -- 标价
  deal_price     DECIMAL(18,4) NOT NULL,   -- 成交单价（分摊优惠后）
  quantity       INT NOT NULL,
  line_amount    DECIMAL(18,4) NOT NULL,   -- deal_price × quantity ± 尾差
  line_status    VARCHAR(24) NOT NULL,     -- 行级状态：可与主单不同步
  refunded_qty   INT NOT NULL DEFAULT 0,   -- 累计退货数量（防超退）
  refunded_amount DECIMAL(18,4) NOT NULL DEFAULT 0,
  KEY idx_order (order_id)
);
```

## 2. 状态机：主单状态 × 行状态 × 单据状态

### 2.1 三层状态，不要一个字段硬扛

- **主单状态**：待支付 → 已支付 → 履约中 → 已完成 / 已关闭。只表达交易主流程。
- **行状态**：行级发货、行级退款导致行与主单不同步（3 行里发了 2 行）。
- **衍生单据自带状态**：退款单、发货单有自己的状态机，不要把"退款中""部分发货"塞进主单状态。

反面教材：一个 status 字段塞 30 个值（待支付/已支付/部分发货/部分退款中/退款完成部分发货……），状态组合爆炸后无人敢改。正确做法：主单状态 ≤ 8 个，其余用行状态和衍生单据状态表达，页面展示状态由多个状态**计算**得出（展示层拼装，不落库）。

### 2.2 状态机纪律（呼应总览篇 §9）

- 合法迁移表配置化：`当前状态 + 动作 → 目标状态`，代码里只有一个迁移函数，禁止散落的 `UPDATE order SET status=?`。
- 状态变更 = 乐观锁 CAS：`UPDATE ... SET order_status=?, version=version+1 WHERE id=? AND order_status=? AND version=?`，影响行数 0 即并发冲突，重读重试。
- 每次迁移写状态流转记录表（谁、何时、从哪到哪、因什么动作、关联单号）。
- 超时动作（未支付关单、超时确认收货）是定时任务驱动的正常迁移，同样走状态机、留记录。

## 3. 金额模型：应付的构成与优惠分摊

### 3.1 金额恒等式

订单金额必须满足可校验的恒等式：

```text
应付 = Σ行(标价×数量) + 运费 - Σ优惠
Σ行成交金额 = Σ行(标价×数量) - Σ已分摊到行的优惠
实付 = Σ支付记录(成功) = 应付   （货到付款/部分支付另议）
```

落库前校验恒等式，不满足直接拒绝——这是防"价格计算 bug 造成资损"的最后闸门。

### 3.2 优惠分摊到行

平台券、满减、店铺折扣都是订单级优惠，但退款、结算、开票、毛利全在行级，所以**所有订单级优惠必须分摊到行**：

```sql
CREATE TABLE ord_discount (
  id            BIGINT PRIMARY KEY,
  order_id      BIGINT NOT NULL,
  order_line_id BIGINT NOT NULL,          -- 分摊到的行
  discount_type VARCHAR(32) NOT NULL,     -- PLATFORM_COUPON/SHOP_COUPON/FULL_REDUCE/...
  activity_id   BIGINT,                   -- 活动/券ID
  rule_version  VARCHAR(32),              -- 命中的规则版本（总览篇 §17）
  amount        DECIMAL(18,4) NOT NULL,   -- 分摊到本行的金额
  cost_bearer   VARCHAR(16) NOT NULL      -- 出资方：PLATFORM/SHOP/SUPPLIER —— 结算的关键
);
```

要点：

- 按行金额比例分摊，尾差给最后一行（尾差规则全局统一，呼应账务篇 §8）。
- **出资方（cost_bearer）必须记录**：平台券退款时钱由平台补给商家还是直接不结算，全靠它。漏了这个字段，结算对账永远算不平。
- 命中的活动 ID + 规则版本固化，回答"当时为什么便宜了 7 块"。

### 3.3 退款按行按比例

行退款金额 = 行成交金额 × (退货数量/购买数量)，累计校验：`refunded_amount + 本次 ≤ line_amount`、`refunded_qty + 本次 ≤ quantity`（原子条件 UPDATE 保护）。分摊过优惠的行，退款同时回退分摊优惠并按出资方结转（联动账务篇冲正）。

## 4. 逆向单据：取消、退款、退货

呼应总览篇 §14，订单域的落地形态：

```text
取消单（未支付/未发货取消）：释放库存锁定、关闭支付
退款单（仅退款）：金额逆向，关联订单行，走账务冲正
退货退款单：先退货（库存回收+质检）后退款，两阶段状态机
换货单：退货入 + 新发出，金额可能补差
```

纪律：

- 每种逆向是**独立单据 + 独立状态机**，关联正向单和正向行。
- 一行可多次部分退，防超退靠行上累计字段 + 原子条件更新（双保险）。
- 退款单驱动账务（账务篇冲正凭证）和库存（库存篇退货入库），订单域只管发起和跟踪状态，不直接改余额改库存——**跨域协作走事件/Saga**（总览篇 §8、§14）。

## 5. 订单与外部域的边界

| 协作域 | 订单侧持有 | 对方侧持有 | 一致性 |
|---|---|---|---|
| 支付 | 支付记录（渠道单号、金额、状态） | 支付单、渠道交互 | 支付回调幂等推进订单状态；对账兜底掉单 |
| 库存 | 无库存数据 | 锁定记录 | 下单锁定失败则订单创建失败（同步）；关单释放（异步+超时） |
| 履约 | 履约记录（包裹、行、数量） | WMS/物流详情 | 发货事件推进行状态 |
| 结算 | 行金额、优惠出资方 | 结算单、佣金 | T+N 按行生成结算明细 |
| 发票 | 开票状态 | 发票单 | 行级开票，红冲联动退款 |

核心原则：订单表**不冗余其他域的过程数据**（不存物流轨迹、不存库存数），只存"与本次交易承诺相关的结果引用"。

## 6. 高并发下单要点

- **订单号预生成 + 幂等**：客户端携带下单 token（服务端预发），重复提交命中唯一索引返回原单，防重复下单。
- **下单事务最小化**：事务内只做订单落库 + 库存锁定，营销核销、积分、通知全部事后异步（Outbox）。
- **买家维度分库分表**：分片键 buyer_id（买家查单是最高频查询）；卖家视角查询走异构副本（按 shop_id 分片的读模型或搜索引擎），双写靠 binlog/事件同步——这是总览篇 §37/§38 在订单域的标准应用。
- 订单快照大字段（商品图文快照）拆到独立表或对象存储，主表保持窄行。

## 7. 订单评审清单

```text
1. 五层结构（主单/行/金额分解/支付/履约）是否完整？行级是否承载全部可计算颗粒？
2. 交易快照是否在行上固化（标题/规格/价格/类目）？
3. 主单状态是否 ≤8 个？行状态与衍生单据状态是否分离？展示状态是否计算而非落库？
4. 状态迁移是否配置化 + CAS + 流转记录？
5. 金额恒等式是否落库前强校验？
6. 订单级优惠是否分摊到行？出资方是否记录？尾差规则是否统一？
7. 逆向是否独立单据？部分退/多次退是否防超退？
8. 跨域（支付/库存/履约/结算）是否事件协作而非直接改对方数据？
9. 下单是否幂等？事务是否最小化？
10. 分片键是否为 buyer_id？卖家查询是否走异构读模型？
```

## 8. 总结

订单建模五条纪律：

1. **行级为王**：一切金额、状态、逆向、结算的计算颗粒在行，主表只是汇总。
2. **快照固化**：下单一刻的商品、价格、优惠、规则版本全部定格。
3. **状态分层**：主单极简、行状态独立、衍生单据自治，展示状态靠计算。
4. **金额可证**：恒等式强校验，优惠分摊到行且记出资方。
5. **域界清晰**：订单是交易中枢但不是数据黑洞，跨域走事件，各域自治。

> 订单模型的好坏，看它能不能在三年后新增"组合支付 + 部分退款 + 平台券 + 行级开票"时不动骨架。

## 9. 需要增强的企业级内容

本篇已经覆盖订单主干、订单行、金额、支付、履约、优惠分摊、逆向单据和跨域边界。大型企业系统中，订单还需要补强“售前、履约、税务、合同、版本、审计、读模型”几类能力。

### 9.1 购物车、报价单、订单三阶段

**为什么需要**：订单不是用户选择商品后的唯一形态。B2C 有购物车，B2B 有报价单、合同、审批和账期。

**标准做法**：

```text
购物车：用户意向，可频繁变化，不承诺价格和库存
报价单：销售承诺，有有效期、价格、客户、审批
订单：交易承诺，固化商品、价格、库存、收货、支付
```

**解决的问题**：

- 购物车价格和下单价格混淆。
- 报价后价格变动无法解释。
- B2B 客户审批、账期、合同价无法承载。

### 9.2 地址、联系人和履约承诺快照

**为什么需要**：订单履约依赖收货地址、联系人、配送方式、承诺送达时间、门店自提点等信息。这些信息下单后应固化。

**标准做法**：

- 订单保存收货地址快照，而不是永远引用客户最新地址。
- 订单行或履约单保存履约承诺，例如发货仓、配送方式、预计送达。
- 地址变更必须作为订单变更记录，而不是静默覆盖。

**解决的问题**：

- 用户改地址后历史订单地址变化。
- 物流纠纷无法追溯当时承诺。
- 多包裹、多仓发货无法解释。

### 9.3 订单变更和版本

**为什么需要**：企业订单经常发生改价、改数量、改地址、拆单、合单、延期、取消部分行等变化。

**标准做法**：

```text
订单原始版本 -> 变更申请 -> 变更审批 -> 新版本生效 -> 变更日志
```

需要记录：

- 变更前后差异。
- 变更原因。
- 操作者。
- 是否影响金额、库存、税务、履约。
- 是否需要重新确认。

**带来的复杂度**：

- 订单不是单次写入后静止对象。
- 金额、库存、支付、履约可能都要联动。
- 需要定义哪些状态允许变更。

### 9.4 订单读模型和搜索模型

**为什么需要**：买家查单、商家查单、客服查单、财务查单、运营查单关注字段不同。订单写模型不应该直接承担所有查询。

**标准做法**：

- 写模型围绕订单、行、支付、履约、退款保持规范。
- 读模型按角色建立聚合视图：买家订单列表、商家订单列表、客服订单检索、财务订单明细。
- 搜索字段同步到搜索引擎或宽表。

**解决的问题**：

- 订单主表越来越宽。
- 多角色查询互相拖累。
- 列表页需要跨很多表拼装。

### 9.5 税务、发票与订单关系

**为什么需要**：订单金额、支付金额、开票金额、退款金额不一定同步。行级开票、部分退款红冲是企业系统常态。

**标准做法**：

- 订单行记录税率和税额快照。
- 发票单独建模，支持部分开票、行级开票、红冲。
- 退款单联动红字发票或待红冲状态。
- 报表区分销售额、收款额、开票额、含税/不含税金额。

### 9.6 订单异常和客服介入模型

**为什么需要**：真实订单会出现支付异常、库存不足、发货失败、地址异常、退款争议、商户拒绝等情况。

**标准做法**：

- 建立订单异常表或工单关联。
- 异常类型、责任方、处理状态、处理人、证据独立记录。
- 异常不应被塞进订单主状态，避免状态爆炸。

**底层思想**：

> 订单主流程要保持清晰，异常要独立建模、可追踪、可关闭。
