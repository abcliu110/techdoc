# 国际化与多币种模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

国际化与多币种模型的本质，是把系统中所有隐含的本地默认值显式化。币种、汇率、时区、语言、税制、地址、法域都不能再假设为单一默认。

核心抽象：

```text
站点/法域 -> 币种 -> 汇率 -> 时区 -> 语言内容 -> 税费合规 -> 地址规则
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| i18n_site 站点 | site_code(唯一), 法域, 语言集, 默认币种, timezone(IANA 时区名), 含/不含税展示模式, 数据驻留区域, 法人主体, status | 国际化配置的收口实体：法域×语言×币种×时区×合规的组合 |
| i18n_currency 币种元数据 | ccy_code(ISO 4217), minor_unit 小数位, 舍入规则, 符号, 展示格式 | 精度与舍入的唯一定义处，消灭硬编码 ×100 |
| fx_rate 汇率 | base_ccy, quote_ccy, rate_type(PRICING/SETTLE/ACCOUNTING), rate, valid_from, source | 版本化事实源：只增不改，三种用途分型防汇兑损益混算 |
| ord_currency_snapshot 交易三币快照 | order_id, 交易币/结算币/记账币各自的 ccy+amount, fx_rate_id 各自汇率版本, 换算时点 | 下单固化三币三率；退款按原汇率退，汇差入汇兑损益科目 |
| prd_spu_i18n 翻译表 | spu_id, locale(BCP 47), title, selling_point, trans_status(MACHINE/REVIEWED/NATIVE), version 源内容版本 | 业务数据多语言版本；源变更后译文标记待重审，缺失回退源语言 |
| i18n_resource 系统文案 | resource_key, locale, text, version | key×locale 键值方案，带回退链(zh-HK→zh-TW→zh→默认) |
| tax_rate_config 税费法域配置 | 税种(VAT/GST/销售税), 法域, 类目, 税率, 含税模式, effective_from, effective_to | 税按法域×类目配置带生效区间；订单行固化税率税额快照 |
| geo_region 行政区划 | country_code, 层级, region_code, 名称, version | 按国家维护带版本，支撑分国地址校验 |
| addr_address 地址 | country_code, 一级区划, 邮编, address_line1/2, 校验规则版本 | 结构化字段+弹性行，各国校验规则配置化 |
| compliance_config 合规配置 | site_id, config_key(实名/发票制度/退货期限/数据驻留), config_value, effective_from | 合规分叉是配置不是代码分支，杜绝 if(country=="DE") |

### 2.2 实体关系

```text
i18n_site (1) ──< compliance_config (N)      合规差异按站点配置分叉
i18n_site (N) >── (1) i18n_currency          站点默认币种引用币种元数据
i18n_currency (1) ──< fx_rate (N)            币种对×用途×生效区间的汇率版本
fx_rate (1) ──< ord_currency_snapshot (N)    订单固化当时汇率版本
prd_spu (1) ──< prd_spu_i18n (N)             主表存源语言，翻译表一语言一行
i18n_site (1) ──< tax_rate_config (N)        税率按站点法域挂接
geo_region (1) ──< addr_address (N)          地址引用分国区划
i18n_site (1) ──< i18n_resource (N)          文案按站点语言集取值回退
外部引用：order_id → 订单域（三币快照挂单）；spu_id → 商品域；
         价格按 site 维度定价 → 价格域；汇兑损益科目 → 账务域；
         门店/仓库/租户时区属性 → 各主体域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：ccy_code 唯一；(base_ccy, quote_ccy, rate_type, valid_from) 汇率区间唯一；
       (spu_id, locale) 译文唯一；(resource_key, locale) 文案唯一；
       (税种, 法域, 类目, effective_from) 税率区间唯一
版本固化：fx_rate 只增不改（左闭右开区间）；订单固化三币三率快照；
        订单行固化税率税额；退款按原汇率，汇差单独入科目
金额纪律：每个金额字段旁必有币种字段；计算高精度 DECIMAL(18,6)，
        呈现/入账按 i18n_currency.minor_unit 舍入，尾差有分摊去处
时间纪律：存储一律 UTC；主体(站点/门店/仓)必带 IANA 时区；
        报表带按主体时区预算的 biz_date，促销生效时间按 site 时区解释
译文状态机：源 version 变更 → 译文置 stale 待重审 → 人审通过恢复；
          缺失/过期回退源语言且显式标记
数据驻留：全球共享(商品主数据) 与区域私有(用户/订单) 明确划分，
        跨区同步只走脱敏/聚合数据
```

## 3. 核心张力

最大张力是：

```text
全球统一系统 vs 本地法律、语言、货币和时间差异
```

系统希望一套代码和模型服务全球业务；不同国家和地区要求不同币种、税法、语言、时区、发票、数据驻留和地址格式。

## 4. 不变量

- 金额必须带币种。
- 汇率必须版本化并在交易时固化。
- 时间必须 UTC 存储，本地时区解释。
- 业务日不能等同于自然日。
- 多语言内容必须版本化。
- 税费和合规必须按法域配置。
- 站点是国际化配置的收口实体。

## 5. 为什么这样设计

金额离开币种没有意义，时间离开时区会产生错误，文案离开语言版本会变成内容事故，税费离开法域会导致合规风险。

记录事实时固化当时上下文，是为了让历史订单、退款、报表、税务和审计能按当时条件解释。

## 6. 失败模式

- 金额没有币种，JPY 和 USD 混算。
- 代码里硬编码乘 100 转分，遇到无小数币种出错。
- 订单只存本地时间，跨时区报表错误。
- 促销按服务器时区生效，海外站点提前或延后。
- 翻译内容源文变更后译文仍显示旧含义。
- 税率按代码分支写死，法域变化难维护。

## 7. 适用边界

适用：

- 跨境电商、国际 SaaS、多币种支付、海外门店、全球供应链。

可简化：

- 单国家、单币种、单语言系统可简化，但建议金额和时间模型预留币种、时区字段。

## 8. 设计取舍

优先保证：

- 历史可还原。
- 税费合规。
- 多区域扩展。
- 多语言可治理。

牺牲：

- 字段数量。
- 配置复杂度。
- 价格和报表计算复杂度。

## 9. 同一模型的多业务形态建模

国际化模型要同时支撑多种出海形态，核心方法是：**统一骨架 + 站点作为配置收口 + 差异下沉到法域/币种配置**。

### 9.1 统一骨架（所有形态不变的部分）

```text
站点（site 标识形态与法域）→ 币种元数据 → 三币三率交易时固化 → UTC 存储 + 时区解释 → 翻译表 → 法域税费配置
不变量全集：金额带币种、汇率版本化固化、UTC 存储、多语言版本化、税费按法域配置
```

骨架决不为某个形态特化——单国站和全球站的"这笔钱是什么币种、当时什么汇率、当地几点、显示哪个语言版本"都用同一套模型回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 单国出海站点 | 单币种单语言单时区 | 一个 site 记录收口全部配置；金额仍带币种、时间仍 UTC——形态简化靠配置只有一行，不靠模型退化 |
| 多站点全球化 | 多法域并行、站点间价格独立 | 每法域一个 site；价格表按 site 维度独立定价而非汇率换算；促销生效时间按 site 时区解释 |
| 跨境电商（购汇结汇） | 标价币/支付币/结算币分离 | 三币三率落订单：display/pay/settle 各存币种+金额+当时 fx_rate 版本号；结汇差异走汇兑损益科目 |
| 本地化合资 | 本地法人、数据驻留、独立税制 | site 挂法人主体 + 数据驻留区域标识；税费走该法域配置行；发票模板按法域挂接，不改公共结构 |
| 多语言内容站 | 内容量大、译文治理 | 翻译表（prd_spu_i18n 模式）：源文版本号 + 语言 + 译文 + 翻译状态；源文变更后译文标记 stale，回退默认语言 |

### 9.3 建模决策规则

- **能用 site + 法域配置解决的，不建新模型**：差异只在币种、税率、语言、时区时，交易链路一套。
- **配置按维度垂直拆**（币种元数据 / fx_rate / 税费法域表 / 翻译表），不在业务表堆本地化字段——每类差异有自己的收口表。
- **形态简化靠配置行数减少，不靠删字段**：单国站也存币种和 UTC，为未来多站点保留还原能力。
- **拆独立域的信号**：当法域要求独立账套、独立清结算规则（如金融牌照区域），那是独立法人系统，不是一个 site。
