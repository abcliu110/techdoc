# 商品模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

商品模型的本质，是把“卖的概念”和“库存的实物”分离，并为价格、库存、订单、搜索、报表提供稳定引用。

核心抽象：

```text
SPU -> SKU -> 类目 -> 属性模板 -> 商品版本 -> 价格域 -> 库存域
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| prd_spu 标准产品 | spu_code(唯一), title, category_id 后台类目, brand_id, shop_id, status, current_version | 卖的概念；唯一挂靠后台类目，展示/搜索/评价的锚点 |
| prd_sku 库存量单位 | sku_code(唯一), spu_id, spec_json, spec_fingerprint 规格指纹, 计量单位, status | 交易最小颗粒；订单/库存/价格全部挂它，规格指纹防重复 SKU |
| prd_category 后台类目 | parent_id, path 路径, level, 属性模板引用, 资质模板, status | 稳定管理树，决定属性模板/资质/佣金/报表口径 |
| prd_front_category 前台类目挂靠 | front_category_id, spu_id, 渠道/人群, sort, 生效状态 | 导购展示树多对多挂靠，随运营调整不污染管理口径 |
| prd_attr 属性定义 | attr_name, input_type(单选/多选/文本/数值), is_sale_attr, is_search_filter, 值字典 | 受控 EAV 的元数据：有类型、有字典、可校验 |
| prd_category_attr 类目属性模板 | category_id, attr_id, is_required, sort | 类目决定商品有哪些字段，新品类不改表 |
| prd_attr_value 商品属性值 | spu_id/sku_id, attr_id, value_id/value_text | 商品具体取值的权威源；筛选查询走搜索引擎 |
| prd_spu_version 商品版本 | spu_id, version_no, 内容哈希, audit_status, 提交人/审核人, 发布时间 | 编辑不影响线上；审核通过后原子切换 current_version |
| prd_channel_onsale 渠道上架 | sku_id, channel_id, onsale_status, 上下架时间 | 上下架是渠道级状态，不是 SPU 上的布尔字段 |
| prd_combo_bom 组合明细 | 售卖 sku_id, 子 sku_id, qty, 分摊权重 | 套装按一个 SKU 售卖，库存/退款按子项展开 |

### 2.2 实体关系

```text
prd_category (1) ──< prd_spu (N)            SPU 唯一挂靠后台类目
prd_spu (1) ──< prd_sku (N)                 单规格也建 SKU，引用统一
prd_spu (N) >──< prd_front_category (M)     前台类目多对多挂靠
prd_category (1) ──< prd_category_attr (N)  类目挂属性模板
prd_attr (1) ──< prd_category_attr (N)      属性被多个类目复用
prd_spu/prd_sku (1) ──< prd_attr_value (N)  非销售属性挂 SPU，销售属性定 SKU
prd_spu (1) ──< prd_spu_version (N)         版本只增，线上指针切换
prd_sku (1) ──< prd_channel_onsale (N)      一 SKU 多渠道各自上下架
prd_sku (1) ──< prd_combo_bom (N)           组合 SKU 展开为子 SKU 明细
外部引用：sku_id 被订单/库存/价格/搜索域引用（值靠订单快照固化）；
         shop_id → 商家域；brand_id → 品牌主数据
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：spu_code、sku_code 全局唯一；(spu_id, spec_fingerprint) 防重复 SKU；
       (sku_id, channel_id) 渠道上架状态唯一
状态机约束：草稿→待审核→通过→上架→下架→逻辑删除；
           已被订单引用的 SKU 永不物理删除，状态挡新交易；
           有交易的 SKU 禁改销售属性，只能下架重建
版本化规则：编辑只产生新 version，审核通过原子更新 current_version；
           订单快照引用版本号/内容哈希，历史可还原
属性受控：属性必须有类型和字典，取值落 prd_attr_value 校验；
         搜索筛选走 ES 倒排，DB 属性值为权威源
```

## 3. 核心张力

商品模型最大的张力是：

```text
品类无限变化 vs 下游引用稳定
```

运营希望不断增加类目、属性、规格、渠道、内容和组合商品；订单、库存、价格、报表又要求 SKU 引用长期稳定。

## 4. 不变量

- 订单、库存、价格挂 SKU。
- 展示、搜索、评价主要挂 SPU。
- 类目属性模板受控，不是自由 JSON。
- 已被订单引用的 SKU 不能物理删除。
- 商品编辑不能直接影响线上版本。
- 订单行必须固化商品快照。
- 价格不是商品字段，而是独立模型。

## 5. 为什么这样设计

SPU/SKU 分离是为了同时满足展示聚合和交易精确性。一个 iPhone 15 是展示概念，黑色 256G 才是库存、价格、订单的最小颗粒。

前后台类目分离，是为了让运营导购灵活变化，同时保证管理类目、报表口径、资质要求和佣金规则稳定。

属性模板化，是为了让系统支持无限品类，但仍保持校验、筛选、搜索和结构化能力。

## 6. 失败模式

- 价格挂 SPU，导致不同规格无法差价。
- 订单挂 SPU，导致库存和履约无法精确。
- 类目只有一棵树，运营调整污染报表。
- 属性随意 JSON 化，后续无法筛选和统计。
- 商品改名污染历史订单。
- SKU 删除导致历史订单无法展示。

## 7. 适用边界

适用：

- 零售、电商、POS、WMS、ERP、B2B 采购销售。

可简化：

- 单一商品、无规格、无库存、无历史订单的轻量系统可简化，但仍建议保留 SKU 思想。

## 8. 设计取舍

优先保证：

- 下游引用稳定。
- 品类扩展能力。
- 历史交易可还原。

牺牲：

- 商品录入复杂度。
- 类目和属性治理成本。
- 搜索同步复杂度。

## 9. 同一模型的多业务形态建模

商品模型要同时支撑多种业务形态，核心方法是：**统一 SPU/SKU 骨架 + 类目属性模板承载品类差异 + 商品类型标识裁剪履约语义**。

### 9.1 统一骨架（所有业态不变的部分）

```text
SPU（product_type 标识形态）→ SKU → 类目 → 属性模板 → 商品版本/快照
不变量全集：交易挂 SKU、展示挂 SPU、属性模板受控、SKU 不物理删除、订单快照固化
```

骨架决不为某个业态特化——任何业态的"卖的是什么概念、交易的最小颗粒是什么"都用 SPU/SKU 回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 标品（3C） | 规格标准、条码唯一、比价强 | 标准形态（骨架原型）：SPU 一到少量 SKU，条码表挂 SKU，属性走类目模板 |
| 非标多规格（服装） | 颜色×尺码组合爆炸、销售属性主导 | 销售属性（颜色/尺码）标记在类目模板中，笛卡尔积生成 SKU；SKU 编码规则含属性值，规格矩阵是生成器不是新表 |
| 称重生鲜 | 按重量计价、非整数量、损耗大 | SKU 加计量单位+是否称重标识；价格挂单位重量；条码用店内码规则（编码内嵌重量/金额）解析后仍落到 SKU |
| 服务商品 | 无库存实物、履约是核销/预约 | product_type=service，库存维度裁剪（不建库存项）；SKU 挂服务时长/次数扩展表；订单行仍挂 SKU 保证报表口径统一 |
| 组合套装 BOM | 一个售卖单元对应多个实物 SKU | 组合商品表：售卖 SKU + BOM 明细行（子 SKU+数量）；下单挂组合 SKU，库存和履约按 BOM 展开到子 SKU |
| 虚拟商品 | 卡密/充值/权益，即买即发 | product_type=virtual，SKU 挂发放规则扩展表（卡密池/直充接口）；无物流属性，库存退化为名额或卡密数 |
| 平台 POP 商品 | 商家各自建品、平台要统一类目治理 | 商家商品表 + 平台商品库两层：商家品挂后台类目走审核，发布生成平台侧 SKU 引用；治理规则在平台层不下沉 |

### 9.3 建模决策规则

- **品类差异优先由类目属性模板吸收**：新品类 = 新类目 + 新模板，不改表结构；只有属性模板表达不了的行为差异才动 product_type。
- **product_type 只裁剪履约语义，不改变交易颗粒**：服务/虚拟商品可以没有库存和物流，但订单、价格仍必须挂 SKU。
- **形态扩展表按类型垂直拆**（sku_service_ext / sku_combo_bom），主表不堆可空字段。
- **拆独立域的信号**：当"售卖概念与交易颗粒分离"这一本质不成立（如金融产品按份额净值、内容按授权计费），才是独立域，不是商品的一种类型。
