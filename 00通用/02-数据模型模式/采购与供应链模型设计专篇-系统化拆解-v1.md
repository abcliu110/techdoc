# 采购与供应链模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

采购与供应链模型的本质，是企业支出和供给形成的证据链。它从需求开始，经过供应商选择、采购承诺、收货质检、入库、发票、应付和付款，最终让“钱为什么付出去、货是否真的收到、价格是否正确”可以被追溯。

核心抽象：

```text
供应商 -> 请购 -> 采购单 -> 收货 -> 质检 -> 入库 -> 发票 -> 应付 -> 付款
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| pur_supplier 供应商主档 | supplier_id, supplier_code, supplier_name, status, category, settle_cycle | 采购链条的交易对手主档；准入、冻结、黑名单都围绕它控制 |
| pur_supplier_cert 供应商资质 | supplier_id, cert_type, cert_no, valid_from, valid_to, audit_status | 供应商准入和持续合规的证据 |
| pur_supplier_account 供应商收款账户 | supplier_id, account_type, account_no, bank_name, status, approve_order_no | 付款去向主档；账户变更必须审批留痕 |
| pur_price_list 供货价格 | supplier_id, sku_id, currency, unit_price, tax_rate, min_order_qty, effective_start, effective_end, version | 下采购单时的供货目录和价格版本来源 |
| pur_request 请购单 | request_no, request_org_id, requester_id, source_type, status, approved_at | 需求入口；请购和采购承诺分离 |
| pur_request_line 请购行 | request_id, sku_id, request_qty, converted_qty, required_date | 行级授权量；采购行引用它并留量防超转 |
| pur_order 采购订单 | po_no, supplier_id, purchase_org_id, receive_org_id, currency, status, approved_at | 采购承诺主单；供应商、币种、税率下单即固化 |
| pur_order_line 采购订单行 | po_id, request_line_id, sku_id, qty, received_qty, returned_qty, invoiced_qty, unit_price, tax_rate | 核心颗粒；收货、开票、退供都引用行级留量 |
| pur_receive 收货单 | receive_no, po_id, warehouse_id, receiver_id, received_at, status | 实物到达事实；收货不等于可用库存 |
| pur_receive_line 收货行 | receive_id, po_line_id, received_qty, rejected_qty | 行级收货证据；原子回写采购行 received_qty |
| pur_qc 质检单 | receive_id, qc_no, inspector_id, status, checked_at | 质量确认主单；合格、不合格、让步接收都从这里分流 |
| pur_qc_line 质检行 | qc_id, receive_line_id, qualified_qty, unqualified_qty, disposal_type | 入库、退供、报损的质量依据 |
| pur_inbound 入库单 | inbound_no, qc_id, warehouse_id, inventory_txn_id, status | 合格品进入库存域的凭证 |
| pur_invoice 采购发票 | invoice_no, supplier_id, invoice_type, currency, amount, tax_amount, status | 供应商要款的票据事实 |
| pur_match_record 三单匹配记录 | match_no, po_line_id, receive_line_id, invoice_line_id, diff_amount, diff_reason, status | 付款前的资金安全闸门；差异挂起并审批 |
| pur_payable 应付单 | payable_no, supplier_id, source_type, source_no, amount, paid_amount, due_date, status | 三单匹配后形成的应付债务 |
| pur_payment 付款单 | payment_no, supplier_id, payable_id, pay_amount, pay_account_id, status, paid_at | 应付冲减动作；必须能追溯到供应商账户和三单匹配 |
| pur_return 退供单 | return_no, po_line_id, receive_line_id, return_qty, return_amount, reason, status | 逆向一等公民；冲减库存和应付 |

### 2.2 实体关系

```text
pur_supplier (1) ──< pur_supplier_cert (N)       供应商多资质，资质带有效期
pur_supplier (1) ──< pur_supplier_account (N)    供应商多收款账户，变更审批留痕
pur_supplier (1) ──< pur_price_list (N)          供应商×SKU×时段的供货价格
pur_request (1) ──< pur_request_line (N)         需求按行授权
pur_request_line (1) ──< pur_order_line (N)      请购行可拆分给多个采购行
pur_supplier (1) ──< pur_order (N)               采购单必须引用已准入供应商
pur_order (1) ──< pur_order_line (N)             采购承诺按行落量和价格
pur_order_line (1) ──< pur_receive_line (N)      分批收货，留量防超收
pur_receive (1) ──< pur_receive_line (N)         收货主单与收货行
pur_receive_line (1) ──< pur_qc_line (N)         收货后可分批质检
pur_qc (1) ──< pur_qc_line (N)                   质检主单与质检行
pur_qc (1) ──< pur_inbound (N)                   合格品入库凭证
pur_order_line / pur_receive_line / pur_invoice_line ──< pur_match_record 三单匹配
pur_match_record (1) ──< pur_payable (N)         匹配通过后形成应付
pur_payable (1) ──< pur_payment (N)              应付可部分付款
pur_order_line (1) ──< pur_return (N)            退供引用原采购行和收货行
```

外部引用：supplier_id → 供应商主档；sku_id → 商品/SKU 主档；warehouse_id → 仓库主档；org_id → 组织主档；inventory_txn_id → 库存域事务；payment_no → 资金账务域凭证。

### 2.3 关键约束与不变量落点

```text
唯一键：supplier_code 全局唯一；request_no/po_no/receive_no/qc_no/inbound_no/invoice_no/payable_no/payment_no 全局唯一；
       pur_price_list 使用 (supplier_id, sku_id, effective_start, version) 防同源价格覆盖；
       pur_match_record 使用 (po_line_id, receive_line_id, invoice_line_id) 防重复匹配。

原子约束：pur_request_line.converted_qty + 本次转采购量 <= request_qty；
         pur_order_line.received_qty + 本次收货量 <= qty；
         pur_order_line.invoiced_qty + 本次开票量 <= received_qty - returned_qty；
         pur_payable.paid_amount + 本次付款金额 <= amount。

状态机约束：供应商 POTENTIAL→AUDITING→QUALIFIED→FROZEN/BLACKLIST/EXITED；
           请购 DRAFT→APPROVED→PART_CONVERTED/FULL_CONVERTED/CLOSED；
           采购单 DRAFT→APPROVED→PART_RECEIVED/FULL_RECEIVED/CLOSED；
           应付 PENDING→PART_PAID→PAID/CANCELED。

主档引用：采购链所有执行表只引用 supplier_id，不复制供应商名称作为权威字段；
         票面、订单、付款需要展示时保存 supplier_snapshot 作为历史快照，快照不得反向覆盖主档。

审计落点：供应商状态、银行账户、采购订单、匹配差异、应付付款都必须挂审批单号或操作流水。
```

## 3. 核心张力

采购模型最大的张力是：

```text
业务采购效率 vs 资金支出控制
```

业务希望快速采购、快速收货、快速付款；企业治理要求供应商合规、价格可比、收货真实、发票匹配、付款可控。

## 4. 不变量

- 供应商是主数据，准入、冻结、黑名单必须状态机化。
- 采购链条必须行级引用，而不是只在单据头关联。
- 下游执行量不得超过上游授权量。
- 收货不等于入库，质检不等于付款。
- 三单匹配是付款前的资金安全闸门。
- 应付和付款必须能追溯到发票、收货、采购和请购。

## 5. 为什么这样设计

行级引用是为了支持一张请购拆多张采购单、多张请购合并采购、一张采购单部分收货、部分开票、部分付款。

三单匹配是为了确保企业只为“订过、收过、票据正确”的东西付款。

供应商生命周期建模是为了防止不合格、黑名单、资质过期或银行账户异常的供应商继续进入采购和付款链路。

## 6. 失败模式

- 供应商只是一个名称字段，资质、银行账户、黑名单无法治理。
- 采购单和收货单只在单据头关联，无法行级防超收。
- 收货和入库混为一谈，不合格品直接进入库存。
- 发票金额和收货金额不匹配仍能付款。
- 付款单不能追溯到原始请购和定标依据。

## 7. 适用边界

适用：

- ERP、SRM、采购系统、供应链系统、WMS 入库前链路、B2B 采购。

可简化：

- 小型企业可合并请购和采购、收货和入库，但不能省略供应商治理、行级数量、应付来源和付款追溯。

## 8. 设计取舍

优先保证：

- 资金安全。
- 供应商合规。
- 收货和付款可追溯。
- 应付准确。

牺牲：

- 流程简单性。
- 付款速度。
- 单据数量。
- 业务培训成本。

## 9. 同一模型的多业务形态建模

采购模型要同时支撑多种采购形态，核心方法是：**统一单据链骨架 + 采购类型标识 + 匹配策略配置化吸收差异**。

### 9.1 统一骨架（所有业态不变的部分）

```text
请购 →（行级引用+留量）→ 采购单（po_type 标识形态）→ 收货 → 质检 → 入库 → 发票 → 应付 → 付款
不变量全集：行级引用留量防超收超付、收货≠入库、匹配是付款闸门、付款可沿链追溯到请购
```

骨架决不为某个形态特化——任何形态的"订了多少、收了多少、该付多少、付了没有"都用同一条行级留量链回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 自营采购 | 完整链条、三单匹配 | po_type=STANDARD，全链单据 + 行级三单匹配（标准形态，骨架原型） |
| 寄售代销 | 收货不产生应付，售出才结算 | 入库进寄售库存（货主=供应商的库存维度）；应付触发点改为"销售/耗用报告单"，匹配策略配置为销售量×协议价，收货单不挂应付 |
| JIT 直送 | 供应商直送门店/客户，不经中心仓 | 收货单收货地点为门店/客户地址，入库退化为"直送确认"一条移动记录；三单匹配以门店签收量为收货事实 |
| 集中采购分仓配 | 总部签单、多仓收货 | 一张采购单多张收货单（行级引用天然支持）；采购行带分配仓库计划子表，收货按仓聚合校验不超采购行留量 |
| 生产领料 | 需求来自工单不是请购 | 请购来源类型 source_type=WORK_ORDER，行引用工单行；入库为原料库存，耗用走领料出库与工单成本归集 |
| 服务类采购 | 无实物收货 | 匹配策略按品类配置为两单匹配（采购单+发票）；"收货"事实由验收单（服务完成确认，带验收人审批）替代，应付以验收金额为基础 |

### 9.3 建模决策规则

- **能用 po_type + 匹配策略配置解决的，不建第二条单据链**：差异在"应付何时形成、以什么为收货事实"时，改匹配策略配置，链条一套。
- **收货事实允许替换载体（签收/验收/销售报告），但留量校验机制不变**：任何形态下游执行量都不得超上游授权量。
- **库存归属差异下沉到库存域维度（货主/仓库），不在采购单据上造库存语义**：采购链只负责"该不该付钱"。
- **拆独立域的信号**：当"采购"的对价不是货款而是长期产能/资产（如设备租赁、基建工程进度款），付款依据是合同里程碑而非三单匹配，才是独立的合同结算域。
