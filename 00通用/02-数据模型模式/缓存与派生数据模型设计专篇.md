# 缓存与派生数据模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，把总览篇"单一事实源 + 派生副本"思想在缓存与派生数据上实操展开。缓存、计数器、排行榜、预聚合宽表、物化视图——它们都是同一类东西：**为读性能而生的副本**。副本的原罪是会不一致，本篇讲的就是如何让不一致"有界、可测、可修"。

## 0. 一句话总纲

> 派生数据三问定生死:事实源是谁（永远唯一）、不一致窗口多大能接受（按数据分级）、坏了怎么重建（必须有全量重算路径）。答不出三问的缓存，是定时炸弹不是优化。

## 1. 派生数据的谱系

| 形态 | 例子 | 更新方式 | 不一致窗口 |
|---|---|---|---|
| 旁路缓存 | 商品详情 Redis 缓存 | 失效驱动（删缓存） | 毫秒~秒 |
| 计数器 | 未读数、库存余量展示、点赞数 | 增量累加 + 定期校准 | 秒~小时 |
| 排行榜/集合 | 销量榜、粉丝列表 | 增量维护 ZSET/SET | 秒~分钟 |
| 预聚合宽表 | 用户统计表、卖家订单读模型 | 事件驱动/CDC 异步刷 | 秒~分钟 |
| 物化视图/汇总表 | 日销汇总 | 批任务重算 | 小时~天 |
| 搜索索引 | 商品索引（搜索篇） | CDC 聚合 | 分级（搜索篇 §4） |

共同纪律（全谱系适用）：

- **事实源唯一**：每份派生数据登记"我从哪来"（源表/事件），禁止派生喂派生再喂派生的暗链（链条越长，坏数据传播越难追）。
- **可重建**：任何派生数据必须有全量重算任务（批处理篇），且演练过——不能重建的缓存等于第二份事实源，两边都不敢删。
- **交易不读派生**：下单、支付、扣减走事实源（DB 原子操作），派生只服务展示与筛选（搜索篇"交易回源"的推广）。

## 2. 旁路缓存：删除优于更新

标准模式 Cache-Aside：读时缓存未命中回源并回填；**写时删缓存而不是更新缓存**。

- **为什么删不改**：并发写时"更新缓存"会出现旧值覆盖新值（两次写 DB 与两次写缓存的顺序交错），删除让下次读回源，天然收敛。
- **删除的时序**：先更新 DB 再删缓存（Cache-Aside 标准序）；删除失败要有补偿（重试队列/订阅 binlog 兜底删——"双删"或"延迟删"都是对失败窗口的补丁，binlog 驱动失效是更系统的答案）。
- **缓存对象的粒度 = 事实源聚合粒度**：缓存整个商品聚合（SPU+SKU+图），而不是缓存 SQL 结果——按查询缓存的系统失效逻辑无法枚举。
- 防护三件套进设计而非事后补：**穿透**（空值缓存/布隆过滤器挡不存在的 key）、**击穿**（热 key 过期用互斥重建/逻辑过期）、**雪崩**（过期时间加随机抖动、缓存集群分片）。
- **TTL 是兜底不是机制**：所有缓存必须有 TTL（哪怕一天），保证失效逻辑有漏洞时误差有上界——永不过期 + 手工失效的缓存，漏一处就永久脏。

## 3. 计数器与排行榜：增量 + 校准双通道

计数器（未读数、销量、领券量）的标准结构：

```text
增量通道：业务事件 → 原子 INCR（Redis/DB 计数列），幂等（事件ID去重）
校准通道：定时任务从事实源 COUNT/SUM 重算，覆盖计数器（批处理篇）
展示层：读计数器；对账层：比较计数器与重算值，偏差超阈值告警
```

- **增量会漂移是公理**（消息丢失、重复、程序 bug），校准不是可选项——没有校准通道的计数器，运行一年后的数字没人敢信。
- 涉及资金/资格的计数（配额用量、限领次数）**判定走事实源或 DB 原子计数**，Redis 计数只做展示与预筛（营销篇领券、多租户篇配额同款分工）。
- 排行榜：增量维护 ZSET + 周期全量重建（榜单本质是物化视图），榜单快照落库（昨日榜单是运营证据，出具即冻结）。

## 4. 预聚合宽表与读模型

订单篇"卖家查询走异构读模型"、会员篇"业务侧统计数独立预聚合"的通用方法：

```sql
CREATE TABLE rpt_seller_order_summary (
  seller_id    BIGINT NOT NULL,
  biz_date     DATE NOT NULL,
  order_count  INT NOT NULL,
  pay_amount   DECIMAL(18,4) NOT NULL,
  refund_amount DECIMAL(18,4) NOT NULL,
  refreshed_at DATETIME NOT NULL,      -- 数据新鲜度可见
  source_version BIGINT NOT NULL,      -- 消费到的事件位点/版本
  PRIMARY KEY (seller_id, biz_date)
);
```

- **更新走事件/CDC 异步投影**（Outbox 出来的事件消费），投影消费必须**幂等**（事件 ID 去重表或按位点重放语义）且**记录消费位点**——位点是"重建从哪续、对账到哪了"的依据。
- **新鲜度显式暴露**（refreshed_at 给到前端"数据截至 X 分钟前"）：用户对"明确说了延迟"的容忍远高于"以为实时结果不对"。
- 读模型 schema 演进 = 新建表全量重放 + 切换（搜索篇双索引别名切换同款），不在线改造旧读模型。

## 5. 一致性分级与对账

把搜索篇的字段分级推广为派生数据通用分级：

| 级别 | 不一致容忍 | 手段 | 例子 |
|---|---|---|---|
| L0 交易判定 | 零容忍 | 不用派生，事实源原子操作 | 扣库存、扣余额、限领判定 |
| L1 强展示 | 秒级 | 失效驱动 + binlog 兜底删 | 价格、上下架、订单状态 |
| L2 一般展示 | 分钟级 | 事件投影 + 新鲜度标注 | 统计数、列表页 |
| L3 分析 | 小时级+ | 批量重算 | 报表、榜单 |

- **每份派生数据登记级别**（与指标字典同构的"派生数据登记表"：名称、事实源、级别、更新链路、重建任务、owner）——治理先于优化，登记表就是缓存世界的血缘图。
- **对账巡检**：L1/L2 抽样比对（随机 key 回源比对，偏差率是链路健康度指标），L3 靠批任务天然重算——派生数据的正确性要被测量，而不是被假设。

## 6. 缓存评审清单

```text
1. 每份派生数据能否答出三问（事实源/窗口/重建）？是否有登记表和 owner？
2. 交易判定是否零依赖派生数据？
3. 旁路缓存是否删除优于更新？删除失败是否有 binlog 兜底？是否全部有 TTL？
4. 穿透/击穿/雪崩防护是否设计在前？
5. 计数器是否增量+校准双通道？资金资格类计数是否走事实源判定？
6. 读模型投影是否幂等、记位点、暴露新鲜度？
7. 派生 schema 演进是否走重建切换而非在线改造？
8. 是否有抽样对账巡检并度量偏差率？
9. 是否杜绝派生喂派生的暗链？
```

## 7. 总结

派生数据建模五条纪律：

1. **三问先行**：事实源、窗口、重建，答不全不上线。
2. **交易回源**：判定走事实源，派生只管看和筛。
3. **删除优于更新**：失效驱动 + binlog 兜底 + TTL 保底。
4. **增量必配校准**：漂移是公理，对账是义务。
5. **登记与分级**：每份副本有档案、有级别、有巡检。

> 缓存不是把数据复制一份那么简单——它是你对"这个数字允许错多久"的一次显式承诺。所有缓存事故，本质都是当初没把这个承诺说出口。
