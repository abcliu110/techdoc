# 履约物流模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

履约物流模型的本质，是把订单承诺转化为可追踪的实物交付过程。它连接订单、库存、仓库、承运商、用户签收和售后。

核心抽象：

```text
订单行 -> 履约计划 -> 履约单 -> 包裹 -> 运单 -> 物流事件 -> 签收/异常
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| ful_order 履约单 | 履约单号(唯一), order_id, delivery_type(快递/即时配/自提/送装/虚拟), warehouse_id, status, 计划发货时间 | 一次履约计划；订单与交付分离的锚点 |
| ful_line 履约行 | fulfillment_id, order_line_id, sku_id, 计划 qty, shipped_qty, signed_qty | 行级数量追踪，部分发货/部分签收的颗粒 |
| ful_sourcing 寻源记录 | order_id, 分配仓库/门店, 分配原因(就近/成本/时效), 分配时间, 是否人工改派, 失败原因 | "为什么从这个仓发"可解释、可追溯 |
| ful_package 包裹 | 包裹号(唯一), fulfillment_id, 重量/体积, status, 发出时间 | 实际发出的物理单元，一单可多包裹 |
| ful_package_line 包裹明细 | package_id, ful_line_id, qty | 包裹与履约行的数量桥接，防超发 |
| ful_waybill 运单 | 运单号, package_id, carrier_code 承运商, 段类型(头程/清关/尾程), 运费, status | 承运商合同凭证；跨境一包裹多段运单 |
| ful_track_event 物流事件 | carrier_code, 运单号, 事件时间, 事件状态(揽收/运输/签收/拒收), 事件描述, 去重键(唯一) | 外部轨迹只增不改，幂等去重落库 |
| ful_exception 履约异常 | fulfillment_id/package_id, 异常类型(缺货/拒收/丢件/地址异常), status, 处理方案, 处理人 | 异常是业务对象，有独立生命周期，不写备注 |

### 2.2 实体关系

```text
订单 order (1) ──< ful_order (N)           一单多仓/拆单履约
ful_order (1) ──< ful_line (N)             履约行关联订单行+数量
订单 order (1) ──< ful_sourcing (N)        寻源决策留痕（含改派）
ful_order (1) ──< ful_package (N)          一次履约多包裹
ful_package (1) ──< ful_package_line (N)   包裹明细桥接履约行
ful_package (1) ──< ful_waybill (N)        跨境多段运单
ful_waybill (1) ──< ful_track_event (N)    轨迹事件只增
ful_order/ful_package (1) ──< ful_exception (N)  异常独立单据
外部引用：order_id/order_line_id → 订单域；sku_id → 商品域；
         warehouse_id → 库存/仓储域；carrier_code → 承运商主数据
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：履约单号、包裹号 全局唯一；
             ful_track_event (carrier_code, 运单号, 事件时间, 事件状态) 唯一去重
留量校验：Σful_line 计划 qty ≤ 订单行 qty - 已履约 qty（原子 UPDATE 条件）；
         Σ包裹明细 qty ≤ 履约行计划 qty（防超发）
恒等式：订单行 shipped_qty = Σ该行所有包裹明细已发 qty
状态机约束：待分配→待拣货→待复核→待打包→已发货→运输中→已签收→已完成，
           只进不退；异常不占用主状态位，由 ful_exception 承载；
           delivery_type 允许裁剪环节（自提无运单）但不新增终态
分片键：order_id（订单侧查询主路径）；轨迹事件按运单号路由
```

## 3. 核心张力

最大张力是：

```text
订单承诺完整性 vs 现实交付不确定性
```

订单希望一次成交完整履约，但现实中会出现多仓、拆包、缺货、拒收、丢件、延迟、部分签收等情况。

## 4. 不变量

- 订单和履约要分离。
- 履约必须关联订单行和数量。
- 一单可多仓、多包裹、多运单。
- 物流轨迹是外部事件，必须幂等处理。
- 履约异常不能只写备注。

## 5. 为什么这样设计

订单表达交易承诺，履约表达交付执行。二者分离后，订单主流程可以保持稳定，交付过程的复杂性由履约单、包裹、轨迹、异常模型承载。

## 6. 失败模式

- 订单表只放一个物流单号，无法支持多包裹。
- 发货不关联订单行数量，部分发货不可追踪。
- 物流轨迹覆盖更新，异常节点丢失。
- 缺货、拒收、丢件没有独立处理流程。

## 7. 适用边界

适用：

- 电商、零售配送、仓储发货、门店配送、B2B 发货。

简化：

- 无实物交付的虚拟商品或服务订单可省略包裹和物流模型，但仍需交付状态。

## 8. 同一模型的多业务形态建模

履约物流模型要同时支撑多种交付形态，核心方法是：**统一履约行骨架 + 履约方式标识 + 寻源与轨迹环节按形态替换**。

### 8.1 统一骨架（所有业态不变的部分）

```text
订单行 → 履约计划（寻源留痕）→ 履约单（delivery_type 标识方式）→ 履约行（关联订单行+数量）→ 包裹 → 运单 → 物流事件（幂等去重）→ 签收/异常
不变量全集：订单履约分离、行级数量追踪、寻源结果可追溯、轨迹事件幂等、异常独立建模
```

骨架决不为某个形态特化——任何形态的"这行商品从哪发、发了多少、到哪一步"都用同一套履约行 + 事件回答。

### 8.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 快递电商件 | 仓发、承运商事件推送 | delivery_type=EXPRESS，完整状态机 待分配→…→已签收（标准形态，骨架原型） |
| 同城即时配 | 运力平台派单、分钟时效 | 运单对接运力平台，事件源换成骑手节点（接单/到店/取货/送达），幂等键仍是 承运商+运单号+时间+状态；寻源退化为"就近门店" |
| 门店自提 | 用户到店、无运输 | 寻源结果=指定门店；包裹=备货记录，运单省略；签收替换为提货码核销事件，超期未提进异常模型转退款 |
| 大件安装配送 | 预约制、送装两段 | 履约单拆干线/送装两段子单，履约行引用同一订单行分段追踪；预约与改约独立记录；签收事件带安装凭证附件 |
| 跨境物流 | 多段承运、清关事件 | 一包裹多运单按段建模（头程/清关/尾程段类型）；清关状态作为外部事件接入同一事件表，报关单号挂包裹扩展 |
| 虚拟履约 | 无实物、即时交付 | 包裹运单省略，履约行直连发放记录（卡密/开通流水）作为交付凭证；发放失败进异常模型自动重试 |

### 8.3 建模决策规则

- **能用 delivery_type + 环节省略解决的，不建第二套履约模型**：差异在"有没有包裹和运输段"时，环节退化，履约行和事件机制一套。
- **事件源可替换（承运商/骑手/核销/发放），幂等去重规则不变**：任何外部事件都按 来源+单号+时间+状态 去重落库。
- **状态机按形态裁剪但不新增终态**：自提跳过运输状态，终态仍收敛到 已签收(核销)/异常。
- **拆独立域的信号**：当交付是持续性权益（订阅解锁、服务期开通）而非一次实物/虚拟动作时，归权益域，不是履约方式的一种。
