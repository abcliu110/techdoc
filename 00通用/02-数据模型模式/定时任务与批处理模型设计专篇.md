# 定时任务与批处理模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦任务调度建模。前面每一篇都在依赖它：券过期扫描、积分过期、对账任务、日切、索引重建、报表跑批、补偿重试。批处理是"系统的夜间心跳"，它的建模难点：任务不能重跑出双份、不能漏跑丢一天、失败要能从断点续起、亿级数据要能分片推进。

## 0. 一句话总纲

> 批处理模型三件套：任务定义（做什么、何时做）、执行实例（每一次运行的档案）、分片进度（跑到哪了）。一切设计围绕四个字：幂等、可续。

## 1. 三层结构：定义、实例、分片

```sql
CREATE TABLE job_def (
  id           BIGINT PRIMARY KEY,
  job_key      VARCHAR(64) NOT NULL,    -- coupon_expire_scan
  cron_expr    VARCHAR(32) NOT NULL,
  timezone     VARCHAR(32) NOT NULL,    -- 调度时区显式声明（国际化篇）
  timeout_sec  INT NOT NULL,
  retry_policy JSON,                    -- 重试次数/退避
  concurrency  VARCHAR(16) NOT NULL,    -- FORBID禁止并行/ALLOW/REPLACE
  alarm_config JSON,                    -- 失败/超时/漏跑告警
  status       VARCHAR(16) NOT NULL,    -- ENABLED/PAUSED
  owner        VARCHAR(32) NOT NULL,    -- 每个任务必须有 owner
  UNIQUE KEY uk_key (job_key)
);

CREATE TABLE job_instance (
  id           BIGINT PRIMARY KEY,
  job_key      VARCHAR(64) NOT NULL,
  biz_date     DATE NOT NULL,           -- 业务日期（数据归属日，非执行日）
  trigger_type VARCHAR(16) NOT NULL,    -- SCHEDULED/MANUAL/RETRY/BACKFILL补跑
  status       VARCHAR(16) NOT NULL,    -- RUNNING/SUCCESS/FAILED/TIMEOUT/CANCELED
  started_at   DATETIME,
  ended_at     DATETIME,
  stats        JSON,                    -- 处理量/成功/失败/跳过计数
  error_msg    TEXT,
  operator     VARCHAR(32),             -- 手工触发人
  UNIQUE KEY uk_job_date_seq (job_key, biz_date, id),
  KEY idx_status (status, started_at)
);
```

要点：

- **执行实例是档案**：每次运行（含手工触发和补跑）一条记录，statistics 记处理量——"昨天的过期任务跑了吗、处理了多少"是查表不是查日志。
- **biz_date 与执行时间分离**（报表篇迟到数据同款思想）：补跑 3 天前的任务，biz_date 是 3 天前——任务处理"哪天的数据"由参数决定，不由"今天是几号"决定。**所有任务代码禁止内部取 `now()` 当业务日期**，业务日期必须来自调度参数，这是可补跑的前提。
- 并行策略显式声明：日切、对账类必须 FORBID（上一轮没跑完不开新轮）。

## 2. 幂等：重跑必须无害

批任务一定会重跑（失败重试、手工补跑、调度抖动双触发），每个任务设计时先回答"重跑一遍会怎样"：

| 手段 | 适用 |
|---|---|
| 处理结果带幂等键落唯一索引 | 产生流水/单据的任务（过期流水、账单生成）——与在线幂等同一纪律 |
| 状态条件更新（CAS） | 状态迁移类（`WHERE status='UNUSED' AND valid_end < ?`），重跑扫不到已处理的 |
| 先删后插 / INSERT ON DUPLICATE | 重建派生数据类（汇总表按 biz_date 分区，重跑先清本分区） |
| 实例级互斥 | FORBID 并发 + 分布式锁，防同一实例双跑 |

> 靠"保证只调度一次"来保证正确性是幻觉——调度系统本身就是分布式系统，at-least-once 是常态，幂等是唯一可靠答案。

## 3. 大数据量：分片与断点续跑

亿级表扫描（券过期、消息重推）的标准模式：

```sql
CREATE TABLE job_shard_progress (
  id           BIGINT PRIMARY KEY,
  instance_id  BIGINT NOT NULL,
  shard_key    VARCHAR(64) NOT NULL,   -- 分片标识：ID区间/哈希桶/租户/仓库
  status       VARCHAR(16) NOT NULL,   -- PENDING/RUNNING/SUCCESS/FAILED
  checkpoint   VARCHAR(64),            -- 断点：本分片处理到的最大 ID
  processed    BIGINT NOT NULL DEFAULT 0,
  UNIQUE KEY uk_inst_shard (instance_id, shard_key)
);
```

- **分片并行**：按 ID 区间/哈希/业务维度（租户、仓库）切分，多节点认领执行（认领即 CAS 抢占），失败**只重跑失败分片**。
- **游标推进而非深分页**：`WHERE id > #{checkpoint} ORDER BY id LIMIT 1000`，checkpoint 每批持久化——`LIMIT 100000, 1000` 式深分页在大表上是灾难。
- **限流让路**：批任务跑在业务库上必须限速（批间 sleep/令牌桶），避开业务高峰（这正是"夜间跑批"的原因）；重型任务读走只读副本，写控制事务大小（每批一个小事务，禁止亿级大事务）。

## 4. 依赖编排：任务是有 DAG 的

日切 → 对账 → 报表是链条，不是三个独立 cron：

- **依赖声明**：任务 B 声明依赖任务 A 的同 biz_date 实例 SUCCESS——用"检查上游实例状态"而不是"错开两小时"（用时间差表达依赖，上游一慢就全线错乱，是批处理事故第一大来源）。
- 上游失败：下游不启动并告警，上游修复重跑成功后下游自动/手工接续。
- **数据就绪信号**：跨系统依赖（等上游文件/分区到达）用就绪标记表（哪个数据集的哪个 biz_date 就绪了），下游轮询或订阅——比"猜他应该跑完了"可靠。
- **漏跑检测**：调度器故障可能整段漏触发，独立巡检任务比对"应跑清单 vs 实例表"，缺了告警——漏跑比失败更危险，因为没有报错。

## 5. 补跑与数据订正

- **补跑（backfill）是一等功能**：指定 job_key + biz_date 区间批量生成补跑实例，按依赖顺序执行——上线新指标回刷历史 90 天，靠的就是它。
- 补跑的安全边界：影响已出具快照（月结账单、已报送报表）的补跑要**走订正流程**（新版本 + 差异说明），不能静默覆盖（快照特例规则，总览篇 §13）。
- 手工触发一律留 operator——"谁在周日下午重跑了结算"必须可查。

## 6. 延迟任务与轻量调度

超时关单（30 分钟未支付）、券到期前提醒，这类"一次性的未来动作"与周期任务不同：

- **延迟任务表**方案（中小量级）：`(biz_type, biz_id, execute_at, status, idempotent_key)`，轮询到期执行，处理走 CAS——简单可靠可审计。
- 量大用延迟消息（MQ 延迟队列），但**DB 兜底扫描仍要保留**（消息会丢，超时关单漏一单就是资损）——与 Redis 预扣 + DB 对账同构：快的通道 + 准的兜底。
- 取消语义：订单支付成功后关单任务作废——执行时 CAS 检查业务状态（`WHERE status='UNPAID'`），而不是追求"精确取消任务"。

## 7. 批处理评审清单

```text
1. 任务是否有定义/实例/分片三层？每个任务是否有 owner 和告警？
2. biz_date 是否来自调度参数？代码是否杜绝 now() 当业务日期？
3. 每个任务是否回答过"重跑会怎样"？幂等手段是否落实到唯一索引/CAS？
4. 大表扫描是否分片 + 游标 + checkpoint？失败能否只重跑失败分片？
5. 是否限流、避峰、小事务？重型读是否走副本？
6. 依赖是否显式声明而非时间差？是否有数据就绪信号和漏跑巡检？
7. 补跑是否一等功能？影响已出具快照的补跑是否走订正流程？
8. 延迟任务是否有 DB 兜底？执行时是否 CAS 校验业务状态？
9. 手工操作是否全部留 operator？
```

## 8. 总结

批处理建模五条纪律：

1. **实例即档案**：每次运行有记录、有统计、有归属业务日期。
2. **幂等是底线**：重跑无害靠唯一索引和 CAS，不靠"只跑一次"的幻觉。
3. **分片可续跑**：游标推进、断点持久化、失败分片单独重来。
4. **依赖靠声明**：DAG 明确、就绪有信号、漏跑有巡检。
5. **补跑走正门**：业务日期参数化，触碰已出具快照必须走订正。

> 在线系统的正确性靠事务，批处理系统的正确性靠幂等加可续。白天的每一笔交易能对上账，一半的功劳在夜里那些默默重跑也不出错的任务。
