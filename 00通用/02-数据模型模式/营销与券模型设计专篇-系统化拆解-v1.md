# 营销与券模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

营销与券模型的本质，是一套“规则驱动的权益发放和核销系统”。它把营销活动拆成活动目标、发放规则、用户资格、券资产、核销行为和风控约束。

核心抽象：

```text
营销活动 -> 规则 -> 发放对象 -> 券资产 -> 核销记录 -> 风控审计
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| mkt_activity 营销活动 | activity_id, campaign_type 玩法类型, 名称, 目标人群, 预算引用, 起止时间, status | 运营单元；预算、周期、复盘的锚点 |
| mkt_rule 优惠规则 | activity_id, rule_version, scope 适用范围, condition 门槛, action 优惠方式, cap 力度上限, 互斥组, status | 计算视角的纯函数定义；只发新版本，已发资产锁旧版本 |
| mkt_coupon_template 券模板 | activity_id, coupon_type(CASH/DISCOUNT/EXCHANGE/FREIGHT), face_value, threshold, scope_type, total_count 发行总量, issued_count 已发量, per_user_limit, valid_type/valid_start/valid_end, cost_bearer 出资方, status | 一批券的定义；发行量是预算硬上限，出资方定结算 |
| mkt_user_coupon 用户券 | coupon_no(唯一), template_id, user_id, status(UNUSED/LOCKED/USED/EXPIRED/FROZEN), issue_idem_key 发放幂等键, locked_order_no, used_at, valid_start/valid_end | 准资金资产；一张券一行，完整生命周期状态机 |
| mkt_coupon_flow 券流水 | coupon_id, 动作(发放/锁定/核销/退回/过期/冻结), 关联订单号, 设备号/IP/风控结果, operator, 时间 | 只增不改的证据链；风控字段支撑黑产圈定 |
| mkt_qualification 资格记录 | activity_id, user_id, 资格类型(新客/人群包), 判定依据, 判定时间 | 资格与领取分离，事后可审计发给了谁、凭什么 |
| mkt_budget 预算账户 | activity_id, cost_bearer, total_budget, used_budget, frozen_budget | 活动成本上限；原子扣减防穿透 |
| mkt_group_session 组局表 | activity_id, 发起人, 参与记录, 成团状态, 超时时间 | 拼团/砍价等多人玩法的过程数据，达成后走标准发放链路 |

### 2.2 实体关系

```text
mkt_activity (1) ──< mkt_rule (N)             规则版本只增
mkt_activity (1) ──< mkt_coupon_template (N)  一活动可发多批券
mkt_activity (1) ──< mkt_budget (N)           按出资方分账预算
mkt_coupon_template (1) ──< mkt_user_coupon (N)  模板发出实例券
mkt_user_coupon (1) ──< mkt_coupon_flow (N)   券状态迁移全留痕
mkt_activity (1) ──< mkt_qualification (N)    资格判定独立记录
mkt_activity (1) ──< mkt_group_session (N)    多人玩法的过程表
外部引用：user_id → 会员域；locked_order_no / 优惠分摊 → 订单域（ord_discount）；
         scope 中的 sku_id/类目 → 商品域；核销成本 → 账务域营销费用户
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：coupon_no 全局唯一；issue_idem_key 唯一（防重复发放）；
       (template_id, user_id) 计数校验 per_user_limit
留量校验：发放原子 UPDATE：issued_count = issued_count + 1
         WHERE issued_count < total_count（并发热点分桶）；
         预算扣减同款条件 UPDATE，防超发
状态机约束：UNUSED → LOCKED（下单锁定，记订单号）→ USED（支付成功核销）
           / 退回 UNUSED（取消/超时）；过期/冻结为终态迁移记流水；
           核销 CAS：WHERE status='LOCKED' AND locked_order_no=?，一券一核销
版本化规则：规则修改发新 rule_version；订单固化 activity_id + rule_version；
           已发券锁定领取时的规则版本，活动改规则不污染存量券
恒等式（对账）：模板 issued_count = Σ用户券；
             Σ核销券成本 = 账务营销费用户发生额；
             LOCKED 超时未释放 = 泄露锁，告警
```

## 3. 核心张力

最大张力是：

```text
增长效率 vs 风控和成本控制
```

营销希望多发券、多转化、多触达；系统必须控制预算、防止重复领取、防止刷券、防止异常核销，并确保退款、过期、回收可解释。

## 4. 不变量

- 活动、规则、券资产要分层。
- 券是资产，不是简单优惠字段。
- 券必须有状态机。
- 发放和核销必须幂等。
- 预算和库存必须可控。
- 退款时券的回收、作废或返还规则必须明确。

## 5. 为什么这样设计

活动和规则分离，是为了同一个活动可以承载多种发放、使用、预算和人群规则。券资产化，是为了支持领取、锁定、核销、过期、退款回滚等完整生命周期。

## 6. 失败模式

- 券只是一条优惠记录，无法追踪状态。
- 发券无幂等，重复领取。
- 核销无锁定，重复使用。
- 预算不受控，营销资损。
- 退款后券是否返还没有规则。

## 7. 适用边界

适用：

- 优惠券、满减券、折扣券、兑换券、实物券、会员权益券。

简化：

- 一次性固定折扣且无需领取、核销、退款回滚时，可用促销规则而非券资产。

## 8. 同一模型的多业务形态建模

营销玩法层出不穷，但支撑它们的骨架只有一套：**活动 + 规则 + 资产三层，玩法差异下沉到规则参数和资产类型**。

### 8.1 统一骨架（所有玩法不变的部分）

```text
营销活动（campaign_type 标识玩法）→ 规则（版本化）→ 券/权益资产（状态机）→ 发放/核销记录 → 预算账户
不变量全集：券资产化、状态机完整、发放核销幂等、预算受控、退款回收规则明确
```

骨架决不为某个玩法特化——任何玩法的"谁有资格、发了什么、用没用、花了多少预算"都用同一套表回答。

### 8.2 各玩法差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 现金券 | 固定面额直接抵扣 | coupon_type=CASH，face_value 生效；标准形态，骨架原型 |
| 折扣券 | 按比例优惠、需要封顶 | coupon_type=DISCOUNT，规则参数存折扣率 + max_discount 封顶值；分摊时按实算金额落 ord_discount |
| 兑换券 | 核销结果是商品不是金额 | coupon_type=EXCHANGE，模板挂 sku_id；核销生成 0 元订单行 + 库存扣减，成本记营销出资方 |
| 运费券 | 只作用于运费层 | coupon_type=FREIGHT，叠加分组固定为运费级；分摊只落运费行，不参与商品行分摊 |
| 满减活动 | 无券资产、下单实时命中 | 只用活动 + 规则两层，不生成 mkt_user_coupon；命中记录直接写订单优惠明细，预算在下单时原子扣减 |
| 秒杀直降 | 改的是价格不是优惠 | 活动只做资格与限购（per_user_limit），价格本身交给价格域的活动价规则；营销侧不落优惠分摊 |
| 拼团砍价 | 多人协作才达成资格 | 新增"组局表"（团/砍价局：发起人、参与记录、成团状态、超时）；成团事件触发标准发券或改价，后链路复用骨架 |
| 裂变邀请 | 奖励在他人行为完成后发放 | 邀请关系表（邀请人、被邀请人、绑定时间、唯一约束防循环）；奖励发放挂"被邀请人首单完成"事件，发放幂等键 = 邀请关系 ID |

### 8.3 建模决策规则

- **能用 coupon_type + 规则参数解决的，不建新券表**：现金/折扣/兑换/运费四种券共用模板和实例表，差异全在类型字段和规则 JSON。
- **无资产的玩法裁剪掉资产层，而不是造假资产**：满减、直降只用活动 + 规则两层，不为它们生成空壳券。
- **多人协作类玩法加"过程表"，不改骨架**：拼团、砍价、裂变的组局/关系是玩法私有的过程数据，达成后仍走标准发放和核销链路。
- **拆独立域的信号**：当玩法的核心不再是"权益发放与核销"而是价格本身（如动态定价）或资金本身（如返现打款），应交给价格域或资金域，营销只保留资格判定。
