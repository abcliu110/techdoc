# 缓存与派生数据模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

缓存与派生数据模型的本质，是为读性能、查询体验和统计效率建立的副本系统。它不是新的事实源，而是从唯一事实源派生出来、允许有限不一致、必须可重建的数据投影。

核心抽象：

```text
事实源 -> 变更事件 -> 派生副本 -> 新鲜度窗口 -> 对账校准 -> 全量重建
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| drv_registry 派生数据登记 | 派生数据名称(唯一), 事实源(源表/事件), 一致性级别(L0-L3), 更新链路(删缓存/事件投影/批算), 重建任务 job_key, owner | 缓存世界的血缘图；答不出三问不上线 |
| drv_projection_offset 投影位点 | 派生数据 ID, 消费位点/source_version, refreshed_at | "重建从哪续、对账到哪了"的依据；新鲜度显式暴露 |
| drv_event_dedup 事件去重 | 事件 ID(唯一), 派生数据 ID, 消费时间 | 投影消费幂等的落点 |
| drv_counter 计数器 | 计数键(user×场景等), count, 最近校准时间, 校准偏差 | 增量通道的持久落点；漂移是公理 |
| drv_recon_log 对账巡检记录 | 派生数据 ID, 抽样范围, 偏差率, 巡检时间, 是否告警 | 正确性被测量而非被假设 |
| rpt_seller_order_summary 预聚合宽表(例) | seller_id, biz_date, order_count, pay_amount, refund_amount, refreshed_at, source_version | 读模型范例：主键=维度组合，带新鲜度与位点 |
| drv_rank_snapshot 榜单快照 | 榜单 ID, biz_date, 榜单内容, 生成时间 | 昨日榜单是运营证据，出具即冻结 |

### 2.2 实体关系

```text
drv_registry (1) ──< drv_projection_offset (N)  一份派生数据一条（或分片多条）位点
drv_registry (1) ──< drv_event_dedup (N)        幂等去重按派生数据隔离
drv_registry (1) ──< drv_recon_log (N)          巡检记录挂登记项，偏差率可趋势化
drv_registry (1) ──< drv_counter (N)            计数器也是登记在册的派生数据
drv_registry (1) ── 预聚合宽表/榜单快照          登记项指向具体物理副本表
外部引用：事实源 → 各业务域源表/Outbox 事件；重建任务 job_key → 批处理域 job_def；
         owner → 权限域用户（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：派生数据名称唯一（一份副本一条档案）；
             事件 ID 去重唯一（投影幂等）；宽表主键=维度组合（seller_id, biz_date）
版本固化：宽表带 source_version 记消费位点；榜单/报表快照按 biz_date 固化，
         重跑生成新版本不覆盖已出具快照
一致性分级落点：登记表的级别字段决定链路——L0 不建派生（交易回源），
              L1 删缓存+binlog 兜底，L2 事件投影+新鲜度标注，L3 批量重算
校准约束：计数器必配校准通道（定时从事实源重算覆盖），偏差超阈值告警；
         所有缓存必有 TTL 兜底
重建约束：每条登记必挂重建任务且演练过；schema 演进走新表全量重放+切换
分片键：宽表按主维度（seller_id 等）分片；缓存 key 加随机抖动防雪崩
```

## 3. 核心张力

最大张力是：

```text
读取性能 vs 数据一致性
```

缓存和派生数据提高读性能，但副本一定可能落后、漂移或损坏。因此设计重点不是消灭不一致，而是让不一致有界、可测、可修。

## 4. 不变量

- 每份派生数据必须有唯一事实源。
- 派生数据不能成为交易判定依据。
- 每份派生数据必须能全量重建。
- 缓存必须有 TTL。
- 写入事实源后优先删除缓存而不是更新缓存。
- 计数器必须有增量和校准双通道。
- 派生数据必须登记 owner、事实源、一致性级别和重建任务。

## 5. 为什么这样设计

事实源唯一，是为了避免多个副本互相成为真相。可重建，是为了当缓存、索引、宽表或计数器错误时能从权威数据恢复。交易不读派生，是为了避免副本延迟造成资金、库存、资格错误。

## 6. 失败模式

- 缓存永不过期，失效漏一处后永久脏。
- 更新缓存而不是删缓存，并发下旧值覆盖新值。
- Redis 计数器漂移后无人校准。
- 派生宽表坏了无法重建。
- 下单扣库存依赖缓存库存，造成超卖。
- 派生喂派生，坏数据链式传播。

## 7. 适用边界

适用：

- 缓存、未读数、点赞数、排行榜、商品索引、订单读模型、报表汇总、物化视图。

不适用：

- 余额扣减、库存扣减、支付成功判断、领券资格等强一致交易判定。

## 8. 设计取舍

优先保证：

- 读性能。
- 可重建。
- 不一致可控。
- 事实源清晰。

牺牲：

- 强一致。
- 架构简单性。
- 同步链路维护成本。

## 9. 同一模型的多业务形态建模

派生数据模型要同时支撑多种副本形态，核心方法是：**统一治理骨架 + 一致性分级 + 差异下沉到同步链路与校准方式**。

### 9.1 统一骨架（所有形态不变的部分）

```text
事实源三问（源是谁/多旧可接受/怎么重建）→ 一致性分级 → 派生数据登记（owner+事实源+级别+重建任务）
不变量全集：事实源唯一、交易回源、可全量重建、TTL 保底、增量加校准
```

骨架决不为某类副本特化——任何形态的"这份数据从哪来、允许多旧、坏了怎么修"都用登记表的同三问回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 详情页缓存 | 秒级新鲜度、读多写少 | Cache-Aside：写库后删 key 不更新 key；TTL 保底（分钟级）；登记级别=最终一致（标准形态，骨架原型） |
| 计数器（点赞/未读） | 高频增量、必然漂移 | Redis INCR 增量通道 + 定时批任务从事实源 COUNT 校准回写；登记校准任务 ID，漂移阈值告警 |
| 排行榜 | 窗口计算、可整体重算 | ZSET 增量更新 + 周期全量重建（从事实源重算整榜替换）；榜单本身不做资格判定，发奖回源核验 |
| 卖家读模型 | 跨域聚合、查询维度多 | 事实源变更事件驱动更新宽表（rpt_seller_order_summary 模式）；带 last_event_time 水位；重建任务按卖家分片全量刷 |
| 搜索索引 | 结构异构、更新链路长 | binlog/事件流增量写索引 + 每日对账任务比对 ID 集合补齐差异；全量重建走新索引别名切换，不原地重刷 |
| 报表汇总表 | 按 biz_date 固化、口径版本化 | 批任务从事实源按 biz_date 生成，重跑生成新版本不覆盖已出具快照；口径变更靠版本号区分，不改历史行 |

### 9.3 建模决策规则

- **能用一致性分级 + 登记解决的，不建新同步框架**：差异只在链路（删缓存/事件/批算）和校准频率时，治理模型一套。
- **同步链路按副本垂直选择**（Cache-Aside / 事件驱动 / 批量重算），选择记录在登记表，不隐藏在代码里。
- **新鲜度允许按形态放宽但不新增语义**：报表允许 T+1，但不得因此免登记、免重建能力。
- **拆独立域的信号**：当"副本"被业务直接写入、成为交易判定依据时，它已不是派生数据——要么升级为事实源并明确归属域，要么改造回源。
