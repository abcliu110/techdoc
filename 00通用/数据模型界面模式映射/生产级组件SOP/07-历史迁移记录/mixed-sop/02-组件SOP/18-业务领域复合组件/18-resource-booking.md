## 2. 原型直接事实

### 2.1 定义与模型

- 摘要：房间 / 资源预订器把领域对象、关系与操作组合成可连续完成任务的复合界面。
- 模型：`Resource × TimeRange → Booking`
- 原型状态：`date, selectedSlot`
- 不变量：对象身份、状态与用户操作必须保持一致；任何切换都不能静默丢失未提交内容。
- 适用边界：适用于业务领域复合组件中的高频任务；简单展示场景应使用更轻量的基础组件。

### 2.2 业务上下文

- 角色：李明｜实施顾问；赵峰｜系统管理员
- 任务：协同完成房间 / 资源预订器的校验、确认与下游交接
- 对象：会议室 A；10:30-11:30 时段；项目评审日程
- 规则：会议室 A通过当前规则校验且10:30-11:30 时段版本未变化后，才允许更新项目评审日程
- 原型异常：10:30-11:30 时段发生并发变化、约束冲突或审批拒绝时冻结下游动作并保留现场
- 预期影响：会议室 A的处理结果同步到项目评审日程，并记录金额、库存、排期、权限或 SLA 影响

职责：

- 李明｜实施顾问：录入并提交业务事实
- 赵峰｜系统管理员：复核规则、影响和例外

事件/操作时间线：

- 李明｜实施顾问载入会议室 A
- 系统校验10:30-11:30 时段与关联约束
- 赵峰｜系统管理员确认后更新项目评审日程

## 3. 生产风险基线

- 暂定等级：**R2**。
- 判定依据：组件包含持久化、复杂草稿、异步数据、协作冲突或可补偿的业务状态影响。
- 类别升级条件：命中权限、多租户、金额、库存、订单、支付、医疗隐私、安全告警或不可逆动作即 R3；其余跨系统流程至少 R2。
- 最高后果复核：Gate 1 必须检查权限、多租户、敏感数据、金额、库存、订单、支付、不可逆操作和跨系统一致性；命中任一 R3 条件即升级，禁止降级绕过门禁。
- 冻结规则：最终等级、理由和批准人写入契约卡与机器索引后，才能进入 Gate 2。

## 4. 生产交互契约

### 4.1 核心状态

类别状态模型：核心对象、关联对象、规则快照、审批、提交、下游状态、审计与补偿状态

本组件至少覆盖：

- 初始：对象身份、权限、版本和当前 date, selectedSlot 可解释。
- 编辑/操作中：每个用户动作产生可观察但未误报成功的状态。
- 校验/提交中：防重复触发，可取消的操作提供取消能力。
- 成功：结果关联原业务对象并说明下游影响。
- 异常：真实触发组件自身失败，不调用统一壳层伪造。
- 已恢复：撤销房间 / 资源预订器的未完成事务，恢复10:30-11:30 时段快照并取消尚未执行的项目评审日程下游动作
- 只读、禁用、无权限、数据过期和冲突：适用性在 Gate 2 逐项冻结，不适用项必须说明理由。

### 4.2 主路径

1. 载入并核对 会议室 A、10:30-11:30 时段、项目评审日程 的身份、版本和当前状态；此动作属于组件业务前置，不得用统一“开始”按钮代替。
2. 通过组件自身控件完成核心操作：房间 / 资源预订器把领域对象、关系与操作组合成可连续完成任务的复合界面。
3. 按“会议室 A通过当前规则校验且10:30-11:30 时段版本未变化后，才允许更新项目评审日程”校验；校验通过后才产生“会议室 A的处理结果同步到项目评审日程，并记录金额、库存、排期、权限或 SLA 影响”。

### 4.3 异常与恢复路径

1. 使用可重复夹具真实触发：10:30-11:30 时段发生并发变化、约束冲突或审批拒绝时冻结下游动作并保留现场
2. 组件显示原因、影响对象和可执行恢复动作；不得清空未提交输入或把失败写成成功。
3. 执行恢复：撤销房间 / 资源预订器的未完成事务，恢复10:30-11:30 时段快照并取消尚未执行的项目评审日程下游动作
4. 恢复后验证焦点、对象版本、草稿和重复副作用；补偿语义为：撤销房间 / 资源预订器的未完成事务，恢复10:30-11:30 时段快照并取消尚未执行的项目评审日程下游动作

### 4.4 键盘与辅助技术

- 原型要求：Tab 依序到达对象、校验和提交控件，Enter 执行，Esc 返回上一个可恢复状态
- React 契约必须在 Gate 2 细化每个键、焦点入口、焦点返回和读屏播报文本。
- 主路径、异常路径和恢复路径均须只用键盘完成；远端状态更新不得抢夺用户焦点。

## 5. 组件专属验证

### 5.1 必须断言

- 主路径必须实际完成“协同完成房间 / 资源预订器的校验、确认与下游交接”，不能只改变状态文案或调用统一壳层动作。
- 不变量断言：对象身份、状态与用户操作必须保持一致；任何切换都不能静默丢失未提交内容。
- 异常断言：真实触发“10:30-11:30 时段发生并发变化、约束冲突或审批拒绝时冻结下游动作并保留现场”，并验证业务动作被正确阻断。
- 恢复断言：执行“撤销房间 / 资源预订器的未完成事务，恢复10:30-11:30 时段快照并取消尚未执行的项目评审日程下游动作”后，输入、对象身份、焦点和可继续操作性符合契约。
- 结果断言：主体区域明确呈现“会议室 A的处理结果同步到项目评审日程，并记录金额、库存、排期、权限或 SLA 影响”，并能关联到 会议室 A、10:30-11:30 时段、项目评审日程。
- 边界断言：适用于业务领域复合组件中的高频任务；简单展示场景应使用更轻量的基础组件。
- 键盘断言：Tab 依序到达对象、校验和提交控件，Enter 执行，Esc 返回上一个可恢复状态

### 5.2 类别附加测试

- 验证至少三个业务对象和双职责边界
- 验证重复、并发、部分成功、下游失败与补偿
- 验证金额/库存/排期/权限/SLA 影响和完整事件时间线

### 5.3 失败模式

- 关联对象版本变化或跨模块规则冲突
- 部分提交、重复提交或下游超时
- 权限、金额、库存、排期、SLA 或审计不一致

### 5.4 性能基线

按组件契约登记真实数据规模和下游延迟；本地反馈 p95 不高于 100ms，长事务必须显示阶段、允许安全取消或进入可恢复后台状态。

Gate 2 必须记录设备档位、数据规模、运行次数、p95 口径和公共子路径 gzip 增量预算；未登记预算不得进入 PR Gate。

### 5.5 旧原型特征化

- 旧观察点：`#prototypeState`
- 旧变化指纹：`booking:conflict;resource:room-A;start:10:30;end:11:30`
- 旧重置指纹：`booking:unchecked;resource:room-A`
- 使用规则：这些值只锁定旧原型行为。React 测试必须转为用户可见结果、公开事件和业务对象断言，不能继续把状态字符串变化作为唯一通过条件。

## 7. 证据包与发布条件

证据目录：`quality/evidence/18-resource-booking/<component-version>/`

必须包含：

- `manifest.json`：组件键、契约版本、源码修订、产物版本、完整性和 SOP 版本。
- `test-results/`：RED 原始失败、GREEN、回归、浏览器和失败注入结果。
- `accessibility/`：自动扫描，以及主/异常/恢复路径的键盘和读屏人工记录。
- `visual/`：桌面、移动、相邻断点、默认/暗色/密度和关键状态矩阵。
- `performance/`：登记场景的原始测量与 p95 结果。
- `package/`：ESM、类型、CSS、SSR、子路径、Tree Shaking 和隔离安装结果。
- `approvals.json`：与最终风险等级匹配的独立批准。
- `rollback.md`：撤回、降级、消费方通知和数据/草稿兼容说明。

晋级条件：所有 Gate 通过、零绝对红线、候选产物与证据绑定；R2 还需代表性消费项目完整验收，R3 还需受控灰度和回滚演练。
