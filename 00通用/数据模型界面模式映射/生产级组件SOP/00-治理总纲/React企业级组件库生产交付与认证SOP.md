# React 企业级组件库生产交付与认证 SOP

> 文档状态：待书面复核
>
> 适用对象：`@company/ui`、`@company/tokens`、规范演示站及其发布设施
>
> 适用范围：现有 309 项复杂 UI 交互规范迁移，以及后续新增 React + TypeScript 组件
>
> 规范级别：发布阻断级
>
> 版本：1.0.0

## 1. 目的

本 SOP 把“生产级”定义为一种可复核的认证结果，而不是对完成度的主观描述。

组件只有同时满足以下条件，才能标记为生产级：

1. 公开契约明确且可测试。
2. 统一质量底线全部通过。
3. 与生产风险匹配的附加门禁全部通过。
4. 测试、文档、评审、发布和回滚证据可追溯。
5. 实际发布产物与被测试产物完全相同。
6. 已进入 `Stable` 生命周期状态。

任何强制证据缺失、失败、超预算或不可追溯，都等同于未完成。不得先发布后补证据。

## 2. 核心模型

### 2.1 本质

本 SOP 是一套“统一底线 + 双轴分层 + 组件认证 + 发布晋级”的质量控制系统：

- 统一底线规定所有组件必须具备的最低生产能力。
- `B/C` 表示交互原型复杂度，决定设计和测试工作量。
- `R1/R2/R3` 表示生产失败风险，决定发布门禁强度。
- 单组件七道关口控制从立项到候选产物。
- 仓库四级门禁控制从代码提交到稳定版本晋级。
- 生命周期治理控制实验、稳定、弃用和退役。

### 2.2 不变量

以下规则在任何项目、进度和版本压力下均不得改变：

1. 无证据不得声明通过。
2. 测试对象必须是实际发布的同一不可变产物。
3. 风险只能向上取级，不能为赶进度向下降级。
4. 新能力不得暗中破坏旧公开契约。
5. 前端交互防护不能被描述成服务端安全保证。
6. R3 不允许偏差发布。
7. 绝对红线在任何风险等级下都不得豁免。

### 2.3 关键取舍

本 SOP 优先保证正确性、可访问性、兼容性、可恢复性和可追溯性，因此接受更长的稳定版晋级周期和更高的证据维护成本。实验能力可以快速迭代，但不能绕过实验入口进入关键生产流程。

## 3. 适用范围与边界

### 3.1 纳入范围

- `@company/ui` 的 React 组件、Hooks、样式、类型和子路径导出。
- `@company/tokens` 的设计令牌、主题、密度、动效和兼容矩阵。
- `apps/docs` 中对公开 API 的真实业务演示、规范和迁移说明。
- 单元、契约、交互、无障碍、视觉、响应式、性能、产物和消费集成测试。
- 私有 npm Registry 的预发布、稳定发布、撤回、弃用和退役。
- 现有 309 个 HTML 交互规范到 React 组件的契约映射。

### 3.2 不纳入范围

- 业务服务端的权限、租户、事务、幂等和领域规则具体实现。
- 消费项目自身的页面编排、业务流程和接口实现。
- 把现有 HTML 原型作为 npm 运行时代码直接发布。

### 3.3 前后端责任边界

React 组件负责：

- 防止重复触发和并发交互造成前端状态紊乱。
- 呈现加载、成功、失败、部分成功、只读、禁用和冲突状态。
- 输入约束、确认语义、焦点管理、草稿保护和恢复入口。
- 提供明确的受控状态、事件、错误和可观测钩子。
- 不在日志、DOM、错误提示或事件载荷中泄露敏感数据。

业务服务必须最终负责：

- 身份认证、权限授权和多租户隔离。
- 幂等、并发控制、事务和一致性。
- 金额、库存、订单、支付及其他领域不变量。
- 审计记录和跨系统补偿。

组件不得通过隐藏按钮、禁用控件或客户端校验声称已经完成安全控制。

## 4. 术语与标记

| 术语 | 定义 |
|---|---|
| 契约卡 | 单个组件的可测试规格，包含身份、API、状态、交互、风险、兼容和责任信息。 |
| 证据包 | 与组件版本绑定的机器结果、人工复核记录、产物信息和批准记录。 |
| 候选产物 | 源码、依赖和构建环境锁定后产生，已进入认证但尚未晋级稳定版的不可变包。 |
| 偏差 | 对非红线要求的限期例外；不是永久豁免。 |
| 晋级 | 将同一候选产物从实验或预发布通道提升到更稳定的发布通道。 |
| 撤回 | 阻止问题版本继续被新消费方安装或晋级，并提供已验证的回退或修复路径。 |

## 5. 双轴分层

### 5.1 原型复杂度 B/C

`B/C` 只衡量交互规格的复杂度，不表示生产风险。

- `B`：任务真实。围绕一个明确角色和任务形成可操作闭环。
- `C`：系统真实。串联多个业务对象、角色、模块或下游结果。

复杂度影响：组件拆分、场景数量、实现批次、契约组合和测试矩阵。

### 5.2 生产风险 R1/R2/R3

风险按最坏可信失败后果判定，而不是按视觉大小或代码行数判定。

#### R1：局部且可逆

同时满足以下条件才能定为 R1：

- 只影响当前视图或本地临时状态。
- 用户可以立即撤销、重试或刷新恢复。
- 不提交持久化业务状态。
- 不触达敏感数据、权限、租户、金额、库存、订单或跨系统动作。

#### R2：影响业务状态但可补偿

命中任一项即至少为 R2：

- 异步提交或持久化编辑。
- 批量选择、批量修改或长流程草稿。
- 并发刷新、版本冲突、重复操作或部分成功。
- 错误会造成工作丢失或业务状态不一致，但存在明确补偿路径。
- 大数据量、虚拟化或性能退化会阻断主要任务。

#### R3：关键业务或安全风险

命中任一项即为 R3：

- 权限、身份、多租户或敏感数据。
- 金额、库存、订单、支付、开票、结算或关键审批。
- 不可逆批量操作或不可恢复数据损失。
- 跨系统一致性、补偿、审计或合规要求。
- 错误可能造成越权、跨租户、财务损失或关键业务中断。

### 5.3 分级规则

1. 按最高命中项定级。
2. 无法证明为低风险时向上取级。
3. `C` 不自动等于 R3，`B` 也可能是 R3。
4. 风险变化时立即重新认证；旧证据不能自动沿用。
5. 每次发布必须记录风险判断依据和批准人。

## 6. 全组件统一底线

所有组件，无论 B/C 或 R 等级，都必须满足以下要求。

### 6.1 TypeScript 与公开 API

- 启用 TypeScript 严格模式。
- 公开 API 不允许隐式 `any`、无边界的 `unknown` 或内部类型泄漏。
- 受控与非受控模式必须有明确优先级，不得在运行中静默切换。
- 事件必须描述“发生了什么”，不能要求消费方读取内部 DOM 推断状态。
- 子路径导出、类型导出、样式入口和 peer dependency 必须声明并测试。
- 公共 API、DOM 语义、焦点顺序、默认值和设计令牌均属于兼容面。

### 6.2 状态完整性

组件必须按其能力覆盖适用状态：

- 初始、空、加载、成功、失败、部分成功。
- 禁用、只读、无权限、数据过期、冲突。
- 未保存、提交中、重试中、取消中。
- 异步响应乱序、卸载后返回和重复触发。

不适用的状态必须在契约卡中说明原因，不能静默遗漏。

### 6.3 异常与恢复

- 错误必须说明发生了什么、影响什么以及下一步可执行动作。
- 失败不得无提示吞掉用户输入、选择、草稿或焦点上下文。
- 重试必须明确是否会重复副作用。
- 取消和关闭不得伪装成成功。
- 恢复后焦点返回符合用户任务上下文。
- 组件错误边界不能隐藏会破坏业务正确性的异常。

### 6.4 无障碍

- 目标范围遵守 WCAG 2.2 AA。
- 使用原生语义或等价、完整且经过验证的 ARIA 模式。
- 所有主要路径、异常路径和恢复路径可用键盘完成。
- 焦点顺序、焦点可见性、弹层焦点约束和关闭后的焦点返回必须稳定。
- 状态变化、校验错误和异步结果能够被辅助技术理解。
- 自动扫描不得存在严重或高危问题。
- 关键路径必须进行人工键盘和读屏复核。

### 6.5 视觉、主题与响应式

- 默认主题、品牌令牌覆盖、暗色模式和密度模式不得破坏可用性。
- 文本不得溢出、遮挡或依赖固定文案长度。
- 桌面、平板、移动及断点相邻宽度不得出现阻断式裁切或不可达滚动。
- 弹层、菜单、提示和拖拽反馈不得超出可视区域或覆盖关键操作。
- 动效遵守减少动态效果偏好，且不能成为理解状态的唯一方式。

### 6.6 运行质量

- 关键演示与测试期间无未处理异常、控制台错误或 React 警告。
- 事件监听、计时器、订阅、请求和观察器在卸载时正确清理。
- 用户内容通过安全渲染路径输出，不执行注入内容。
- 服务端渲染不访问浏览器专属对象；客户端水合不得产生未解释差异。
- 导入单个子路径不得引入无关组件模块。

### 6.7 文档与可用性

每个组件必须在规范演示站中提供：

- 一句话用途和明确的不适用边界。
- 真实业务场景，而不是孤立控件陈列。
- API、状态矩阵、键盘行为、错误恢复和无障碍说明。
- 默认主题、暗色、密度和响应式示例。
- 旧 ID 映射、版本状态、兼容说明和迁移示例。
- 已知限制；不得使用模糊的“后续补充”。

规范站只能消费发布包公开 API，禁止直接引用包内源码或测试专用后门。

## 7. 现有 309 项迁移契约

### 7.1 保留内容

现有 HTML 套件继续作为需求与回归基线，不进入 npm 运行时。默认保留：

- 309 个组件 ID。
- 角色、任务、业务对象和业务规则。
- 主路径、异常路径、恢复动作和业务结果。
- 键盘行为、焦点返回和响应式契约。
- B/C 复杂度标记。

视觉允许统一重做，但不得因为视觉重做削弱交互契约。

### 7.2 迁移追踪

每个旧 ID 必须唯一映射到：

```text
legacyId
React 导出名
包子路径
规范站场景
契约测试
风险等级
发布批次
生命周期状态
证据包位置
```

旧契约需要改变时，必须形成显式差异记录，说明原因、受影响消费方、兼容策略和版本级别。不得通过“重新设计”静默删除旧行为。

### 7.3 三级 SOP 体系与专用目录

309 项迁移采用三级 SOP，不以一份总纲替代单组件执行记录：

1. 全库治理 SOP：统一底线、风险模型、认证、发布、偏差和撤回。
2. 类别 SOP：13 个实际类别（01-09、15-18）各自的状态模型、失败模式、性能规模和风险升级规则。
3. 单组件 SOP：309 个组件各自的身份、原型事实、不变量、主路径、真实异常、恢复、暂定风险、专属断言、七道 Gate 和证据条件。

全部 SOP 集中存放在：

```text
00通用/数据模型界面模式映射/生产级组件SOP/
├─ 00-治理总纲/
├─ 01-类别SOP/
├─ 02-组件SOP/
├─ 03-机器索引/
├─ 04-模板与证据规范/
└─ 05-维护工具/
```

机器索引必须反向验证每个 catalog contract 恰有一份组件 SOP，且每份组件 SOP 恰对应一个 contract。React 导出名、包路径、Owner 和最终风险等级在实施 Gate 冻结前不得被生成器推断。

全部单组件 SOP 初始状态必须是 `Draft / not-certified`。生成 309 份文档只代表迁移与认证要求已经逐项登记，不代表 309 个组件已经通过生产认证。

## 8. 角色与职责

| 角色 | 强制职责 |
|---|---|
| 需求/交互责任人 | 确认任务、状态、异常、恢复、键盘和旧契约差异。 |
| 组件 Owner | 对契约、实现、证据、维护和事故响应负责。 |
| 备份 Owner | 在主 Owner 不可用时承担维护和响应；无人备份不得进入 Stable。 |
| 组件库维护者 | 审核包边界、兼容性、依赖、版本和发布资格。 |
| 测试/无障碍评审人 | 独立复核测试策略、风险场景、键盘和读屏证据。 |
| 领域/安全责任人 | R3 必须参与，确认服务端边界、权限、租户、关键数据和回滚。 |
| 发布负责人 | 锁定候选产物、执行晋级、撤回、通知和证据归档。 |

作者不得成为唯一批准人。R2 至少需要组件库维护者独立批准；R3 需要组件库维护者、测试/无障碍评审人和领域/安全责任人共同批准。

## 9. 单组件七道强制关口

七道关口必须按顺序完成。后续关口不能替代前序证据。

### 9.1 Gate 1：登记与定级

输入：组件需求或旧组件 ID。

必须产出：

- 组件名称、唯一追踪 ID、Owner 和备份 Owner。
- 用户、任务、适用边界和不适用边界。
- B/C、R1/R2/R3 及判定依据。
- 最坏可信失败后果。
- 依赖、目标发布批次和生命周期目标。

通过条件：范围、责任和风险均可被独立评审人理解，没有未归属的关键决策。

### 9.2 Gate 2：契约冻结

必须产出契约卡，至少包含：

- Props、事件、方法、插槽和类型。
- 受控/非受控状态模型。
- 正常、空、加载、错误、冲突和恢复状态。
- 键盘模型、焦点模型和读屏语义。
- 异步取消、乱序响应和重复操作策略。
- 主题、密度、响应式和国际化约束。
- 兼容面与弃用策略。

通过条件：API、交互和无障碍评审通过；每条强制行为都能映射到验证方法。

### 9.3 Gate 3：RED

在生产实现前，先写最小失败测试并实际运行。

RED 证据必须证明：

- 测试因目标行为尚未实现而失败。
- 失败不是语法、环境、夹具或选择器错误。
- 测试观察的是公开行为，不是内部实现细节。

没有观察到正确 RED，不得进入 GREEN。

### 9.4 Gate 4：GREEN

实现满足契约的最小生产代码，并使目标测试通过。

要求：

- 复用既定令牌、基础设施和公共模式。
- 不加入契约外能力，不进行无关重构。
- 适用状态和恢复动作同时完成，不能只交付主路径。
- 保持旧契约兼容；需要改变时执行版本治理。

通过条件：目标测试和受影响回归测试通过，且没有新增运行时警告。

### 9.5 Gate 5：风险加固

R1 必须完成：

- 契约、交互、键盘、焦点、无障碍、视觉、响应式和包产物测试。

R2 在 R1 基础上必须完成：

- 请求乱序、取消、重试和重复操作。
- 版本冲突、部分成功、草稿恢复和失败注入。
- 大数据或高频交互性能预算。
- 代表性消费项目集成测试。

R3 在 R2 基础上必须完成：

- 权限拒绝、租户边界和敏感数据暴露检查。
- 幂等、并发、审计和跨系统契约验证。
- 服务端最终裁决边界验证。
- 隔离环境演练、灰度条件、可观测指标和回滚演练。

组件测试只能验证前端是否正确表达和调用这些契约，不能替代服务端领域验证。

### 9.6 Gate 6：规范与迁移

必须在规范演示站完成真实业务场景、状态矩阵、API、错误示例、限制、迁移和版本信息。

通过条件：

- 规范站只使用公开 API。
- 示例可以复制到隔离消费项目运行。
- 旧 ID、契约测试和证据包之间可双向追踪。
- 文档与实际发布候选产物一致。

### 9.7 Gate 7：认证与发布候选

必须完成：

- 全部适用机器门禁。
- 风险等级要求的独立评审和签署。
- 变更集、版本级别、兼容报告和升级说明。
- 不可变候选产物、完整性信息和隔离安装验证。
- 回滚步骤、未覆盖风险和支持责任人。

通过 Gate 7 只获得候选资格，不自动等于 Stable。

## 10. 仓库四级机器门禁

### 10.1 Commit Gate

阻断条件：

- 格式、静态分析、严格类型或单元测试失败。
- 公开 API 类型测试失败。
- 包边界、循环依赖、重复导出或变更集检查失败。
- 新增代码没有对应追踪 ID。

### 10.2 PR Gate

必须验证：

- 全部受影响契约与回归测试。
- 浏览器交互、键盘、焦点、异常恢复和响应式。
- 自动无障碍扫描和适用的人工复核。
- 视觉回归、主题、暗色和密度。
- 性能、包体积、依赖安全与许可证。
- 独立评审签署。

### 10.3 Publish Gate

必须从候选产物创建隔离消费项目，验证：

- ESM、类型、CSS 和子路径导入。
- Tree Shaking 与无关模块不被引入。
- SSR 导入和客户端水合边界。
- peer dependency 范围和令牌兼容矩阵。
- 产物完整性、来源记录、版本和变更说明。

### 10.4 Promote Gate

- R1：候选产物和认证证据通过后，可进入正常稳定发布批次。
- R2：必须经过预发布通道，并在至少一个代表性消费项目中完成一次完整验收循环。完整验收循环必须使用候选产物，执行已登记的主路径、非理想路径和恢复路径，并由该消费项目责任人记录结论。
- R3：必须完成隔离环境演练、受控消费方灰度、监控和回滚演练。灰度是指只有预先登记的消费方和业务场景可以安装该候选版本，且具备停止继续采用和回退到已验证版本的能力；仅发布 npm 预发布标签不等于完成灰度。
- 晋级只能提升同一候选产物，禁止重新构建。
- 候选产物生成后，只要源码、依赖锁文件、构建配置、环境镜像或产物内容发生变化，原候选资格立即失效，必须生成新候选并重新执行受影响的全部认证。

## 11. 机器质量门禁

### 11.1 零容忍项

以下结果必须为零：

- 未处理异常。
- 控制台错误和 React 警告。
- 严重或高危无障碍问题。
- 已知高危依赖漏洞。
- 未声明的破坏性公开 API 变化。
- 丢失的组件到契约、场景或测试映射。

### 11.2 行为覆盖

不得仅用代码覆盖率证明完成。每个组件必须按契约覆盖：

- 主路径。
- 至少一个与组件本质相关的非理想路径。
- 对应恢复路径。
- 键盘与焦点路径。
- 所有被声明为适用的异步和边界状态。

R2/R3 按第 9.5 节增加风险场景。

### 11.3 性能与包预算

所有预算必须在版本控制中登记，缺少预算即阻断 PR。默认预算如下；组件契约可以更严格，不能无批准放宽：

- 直接用户操作到可见反馈的 p95 不高于 100ms，不含明确展示的网络等待。
- 输入、拖拽、滚动等连续交互不得因组件自身逻辑持续产生超过 50ms 的长任务。
- 大数据组件必须声明基准数据量、设备档位和操作场景，并在该基准下满足可操作性。
- 每个公共子路径必须登记 gzip 增量预算；超预算必须拆分、优化或经非 R3 偏差流程批准。
- 导入一个组件不得打包未使用的其他组件和演示站代码。

性能测量必须记录环境、数据量、运行次数和统计口径。单次最快结果不能作为通过证据。

### 11.4 视觉回归

- 基线只能由明确的设计或缺陷修复变更更新。
- 动态区域必须使用稳定数据或显式掩码；不得扩大全局容差隐藏差异。
- 超出组件已登记像素容差的差异必须人工判定并关联变更原因。
- 布局、焦点、错误和响应式等语义问题即使像素差异低于阈值也不得通过。

## 12. 证据包与可追溯格式

每个组件版本使用以下逻辑目录；具体构建工具可以实现等价路径，但字段不可缺失：

```text
quality/evidence/<component-id>/<component-version>/
  manifest.json
  contract.json
  test-results/
  accessibility/
  visual/
  performance/
  package/
  approvals.json
  rollback.md
```

`manifest.json` 至少包含：

```text
componentId
legacyId
exportName
packageSubpath
prototypeComplexity
productionRisk
lifecycle
owner
backupOwner
sourceRevision
artifactVersion
artifactIntegrity
contractVersion
testRunIds
approvals
knownLimitations
rollbackEntry
```

证据必须满足：

1. 与源码修订和产物完整性绑定。
2. 能由另一名维护者复核。
3. 自动结果保留原始机器输出，不只保留摘要截图。
4. 人工复核记录执行人、环境、时间、步骤和结论。
5. 失败后重跑不得删除旧失败记录；必须保留修复链路。

## 13. 生命周期治理

### 13.1 状态

| 状态 | 允许范围 | 兼容承诺 |
|---|---|---|
| Experimental | 探索和内部验证；从实验入口显式导入 | API 可变，不得进入关键生产流程 |
| Alpha | 契约已成形，完成内部演示和失败注入 | 不承诺升级兼容 |
| Beta | 代表性项目试用，开始收集运行证据 | 启用兼容审查和迁移记录 |
| Stable | 完成认证、支持矩阵和回滚 | 遵守语义化版本与支持期限 |
| Deprecated | 保持运行并发出开发期告警 | 提供替代 API、迁移路径和截止版本 |
| Retired | 已从新主版本移除 | 提供历史文档和可执行迁移清单 |

### 13.2 稳定版准入

进入 Stable 必须同时具备：

- 全部适用认证证据。
- Owner 与备份 Owner。
- 兼容和支持矩阵。
- 至少一个隔离消费验证。
- R2/R3 要求的预发布、灰度和演练证据。
- 可执行回滚路径。

## 14. 版本与兼容

### 14.1 语义化版本

- Patch：修复缺陷，不改变公开类型、DOM 语义、焦点顺序、默认行为或视觉尺寸契约。
- Minor：增加向后兼容能力；默认行为不改变，或新行为明确选择启用。
- Major：允许破坏性修改，但必须完成弃用周期、消费方扫描、迁移工具和回滚方案。

`0.x` 阶段仍必须显式标记破坏性变更，不得以“尚未 1.0”为理由省略兼容说明。

### 14.2 兼容面

以下变化都需要兼容审查：

- Props、类型、事件和导出路径。
- 默认值、默认交互和受控状态优先级。
- DOM 语义、ARIA、键盘、焦点和读屏输出。
- CSS 选择器、设计令牌、主题变量和布局尺寸契约。
- 错误类型、异常恢复和异步时序。

### 14.3 tokens 与 ui 联动

`@company/tokens` 与 `@company/ui` 独立发布，但必须维护版本兼容矩阵。`ui` 不得隐式依赖未声明的令牌版本。令牌删除、重命名或语义变化按公开 API 破坏处理。

## 15. 偏差机制

偏差只用于不影响功能、安全、无障碍、数据正确性和兼容性的非红线项。

偏差申请必须包含：

- 未满足的具体条款。
- 原因和影响范围。
- 风险证明和补偿措施。
- Owner、批准人和到期版本。
- 到期前的修复任务和验证方式。

规则：

1. R1 偏差至少由组件库维护者批准。
2. R2 偏差至少由组件库维护者和测试/无障碍评审人批准。
3. R3 不接受偏差。
4. 偏差最长存续一个 Minor 版本。
5. 到期未关闭时自动阻断后续发布。
6. 不允许用扩大视觉容差、关闭检查或降低风险等级规避失败。

## 16. 绝对红线与停止线

### 16.1 绝对红线

任一命中都禁止发布和偏差：

- 跨租户、越权或身份边界失效。
- 敏感信息泄露。
- 金额、库存、订单、支付或关键审批错误。
- 不可恢复的数据或用户工作丢失。
- 关键键盘路径不可达。
- 未处理异常或候选产物未验证。
- 已知高危漏洞、许可证不合规或产物来源不可信。
- 破坏性变更未按 Major 和弃用流程处理。

### 16.2 单组件停止线

任一命中立即停止晋级：

- 契约未冻结。
- RED 未正确观察。
- R2/R3 证据缺失。
- 存在未解释回归。
- 回滚不可执行。
- 文档站绕过公开 API。
- 实际候选产物与测试对象不一致。

### 16.3 全库发布冻结

出现以下情况立即冻结整个组件库的发布晋级：

- 绝对红线问题可能影响多个组件。
- 错误版本或令牌变更造成系统性兼容破坏。
- 构建、Registry 或供应链完整性无法证明。
- 同类缺陷可能存在于多个公共基础设施组件。

解冻前必须完成影响扫描、修复验证、消费方通知和发布负责人批准。

## 17. 发布、观察与撤回

### 17.1 发布列车

1. 锁定源码、依赖、构建环境和候选产物哈希。
2. 完成 Publish Gate。
3. 发布到预发布通道。
4. 按风险等级完成观察：
   - R1：完成候选认证和隔离消费验证。
   - R2：至少在一个代表性消费项目中使用候选产物完成第 10.4 节定义的完整验收循环。
   - R3：完成隔离演练、受控消费方灰度、监控和回滚演练。
5. 通过 Promote Gate 后，将同一产物晋级稳定通道。

### 17.2 发布后观测

每次稳定发布必须具备：

- 已知消费方和版本使用清单。
- 错误、性能、安装失败和兼容问题的反馈入口。
- R2/R3 对应的业务或技术观测指标。
- Owner 响应路径和问题分级规则。

### 17.3 撤回顺序

发现严重问题时：

1. 阻止问题版本继续被新安装或晋级。
2. 冻结相关发布列车并扫描影响范围。
3. 选择已验证的旧产物回退，或构建、认证并发布修复版本。
4. 通知已知消费方，提供明确的升级或降级命令和风险说明。
5. 验证消费方恢复后再解除冻结。
6. 完成事故复盘并把缺失检查转为自动门禁。

禁止直接删除仍被消费项目锁定的版本，也禁止在事故中临时发布未经认证的新构建。

## 18. 组件完成定义

组件只有全部满足以下条件才能标记完成：

```text
[ ] 唯一 ID、旧 ID 映射、Owner、备份 Owner 和风险等级完整
[ ] 契约卡冻结，所有强制行为均有验证方法
[ ] RED 证据有效，GREEN 与受影响回归通过
[ ] 统一底线全部通过
[ ] R 等级附加门禁全部通过
[ ] 规范站只使用公开 API，文档与候选产物一致
[ ] 类型、包边界、子路径、SSR 和隔离安装验证通过
[ ] 兼容报告、变更集、升级说明和回滚路径完整
[ ] 指定角色签署完成
[ ] 候选产物、源码修订和证据包可追溯
[ ] 已进入符合目标的生命周期状态
```

完成代码、完成 Story 或测试通过其中任一项，都不能单独代表组件完成。

## 19. 首批 0.1.0 的执行约束

首批采用“生产基线 + 完整业务切片”，至少包含：

- 设计令牌与默认企业主题。
- 基础控件、弹层和表单基础设施。
- 表格、树及其异步、空、错误和恢复状态。
- 一组代表性业务复合组件，用于证明公开 API、风险门禁、规范站和发布链路完整可用。

0.1.0 不以组件数量为完成标准，而以以下闭环为完成标准：

```text
旧 ID → React API → 真实场景 → RED/GREEN → 风险认证
→ 候选产物 → 隔离消费 → 预发布 → 同产物晋级 → 可执行回滚
```

首批不得为了追求数量并行铺开 309 个未稳定 API。后续批次只有复用已经通过认证的基础设施，才能降低重复风险。

## 20. 失败预演

| 失败方式 | 预防措施 |
|---|---|
| 为追求 309 数量复制同质组件，API 快速失控 | 首批只做完整业务切片；每个组件先冻结契约并验证复用边界。 |
| 证据很多但无法说明用户任务是否正确 | 强制契约行为覆盖，不用代码覆盖率替代业务路径。 |
| 文档站效果很好，npm 包无法真实安装 | Publish Gate 使用打包产物创建隔离消费项目。 |
| 前端禁用按钮被误认为完成权限控制 | 明确服务端最终裁决边界；R3 由领域/安全责任人签署。 |
| 为通过视觉测试不断放宽容差 | 容差按组件登记；禁止全局放宽；语义问题不受像素阈值豁免。 |
| `0.x` 频繁破坏兼容导致消费方不敢升级 | 0.x 仍执行兼容报告、弃用和破坏性变更标记。 |
| R3 流程太慢，团队私下降级为 R2 | 风险依据入证据包；降级必须重新评审，红线命中不可降级。 |
| 发布时重新构建导致测试对象与产物不同 | 候选产物不可变，只允许同一产物跨通道晋级。 |

## 21. SOP 自身变更规则

本 SOP 属于发布阻断级规范。修改时必须：

1. 说明修改原因、受影响门禁和迁移期限。
2. 不得让正在执行的 R2/R3 认证静默失效。
3. 降低门禁强度必须经过组件库维护者、测试/无障碍评审人和领域/安全责任人共同批准。
4. 新规则先进入一个发布批次的影子验证，再转为阻断；绝对红线修订除外。
5. 保留历史版本和生效日期，证据包引用其执行时的 SOP 版本。

## 22. 后续设计输入

后续组件库技术设计和实施计划必须从本 SOP 推导，至少落实：

- 工作区包边界和依赖方向。
- 契约卡与证据包的机器可读 Schema。
- 测试工具、浏览器矩阵、无障碍和视觉回归设施。
- 变更集、版本、私有 Registry 和不可变产物晋级方式。
- 0.1.0 代表性业务切片清单。
- 309 项迁移台账和批次顺序。

技术设计不得降低本 SOP 的底线和风险门禁；如工具能力不足，应标为阻塞，而不是把要求改成自愿检查。
