# 表格组件族实施设计

> 文档状态：已采纳方案，待书面复核
>
> 技术栈：React + TypeScript + TanStack Table（仅内部无头内核）
>
> 对应类别：02 表格类，共 34 项目录能力
>
> 风险基线：DataGrid R2；涉及金额、库存、订单写入或不可逆批量操作时升级 R3

## 1. 决策摘要

表格组件族采用以下架构：

```text
业务项目
  -> @company/ui 自有公开 API
  -> DataGrid / TreeGrid / 预设组件
  -> 内部能力模块与数据策略
  -> TanStack Adapter
  -> TanStack Table
```

本设计冻结以下决策：

1. 只保留一份[React 组件生产交付 SOP](../../03-生产SOP/React组件生产交付SOP.md)，不为 34 种表格复制 34 份 SOP。
2. `DataGrid` 是第一批唯一核心表格组件；排序、筛选、选择、分页、固定列和列宽调整以内部能力模块组合。
3. TanStack Table 只作为内部状态与行模型引擎，不向消费方暴露其类型、实例、事件或 RowModel。
4. 首版不开放任意第三方插件 API。扩展面先保持内部可重构，稳定后再评估是否公开。
5. `TreeGrid` 是第二批派生组件，复用 DataGrid 的行列、渲染、焦点、状态呈现和发布基础设施，不把层级语义塞入首版 DataGrid。
6. 透视、OLAP 和电子表格式表格使用独立内核；它们只能复用公共基础设施，不能强行实现为 DataGrid 开关。
7. 首版支持本地数据和受控的服务端分页数据；虚拟滚动、无限加载在后续批次通过数据/视口策略接入。
8. `@company/ui/data-grid` 的 gzip 增量预算为 35 KB，统计其运行时依赖，排除 React 与 React DOM peer dependency。

## 2. 文档职责

这份文件是表格组件族的技术设计与实施边界，不是第二份生产 SOP。

```text
统一生产规范       定义所有 React 组件必须满足的结果底线
表格类规范         定义表格共同的数据身份、编辑、选择和性能不变量
本实施设计         定义 34 项如何归并、采用什么架构、按什么批次实现
单组件规范         定义 DataGrid、TreeGrid 等具体组件必须是什么
唯一生产 SOP       定义每个组件如何 RED、实现、验证、发布和回滚
执行记录与证据包   记录某个版本是否实际通过
```

本设计不能单独授权编码。只有单组件规范达到 `ImplementationReady` 且机器索引给出 `implementationAllowed=true`，才可进入生产 SOP 的 RED 阶段。

## 3. 目标与非目标

### 3.1 第一批目标

交付一个能在普通企业后台中独立使用的 `DataGrid`：

- 稳定行 ID 和列 ID
- 本地数据与受控服务端分页数据
- 单列/多列排序
- 列筛选
- 单选/多选行
- 分页
- 固定表头和固定列
- 列宽调整
- 初始加载、刷新、空、初始错误、刷新错误和重试
- ARIA Grid 键盘导航、焦点恢复和屏幕阅读语义
- 默认、暗色、紧凑和减少动态效果主题
- 390px 起的可达横向滚动，不把二维数据静默改成卡片

### 3.2 第一批非目标

- 单元格编辑、批量编辑、区域复制粘贴和撤销栈
- 树形层级、异步子节点和父子选择联动
- 虚拟滚动与无限加载
- 透视聚合、OLAP、公式和电子表格行为
- 业务服务端的查询、权限、多租户、事务和幂等实现
- 允许消费方注入任意插件或访问 TanStack Table 实例

## 4. 34 项目录的实现归类

34 项是产品能力目录，不等于 34 套互不相关的 React 源码。

| 实现类型 | 数量 | 目录项 | 产物策略 |
|---|---:|---|---|
| 核心组件 | 1 | 数据表格 | 第一批实现 `DataGrid` |
| 能力模块 | 19 | 可编辑、批量编辑、分组、固定表头与列、多级表头、列分组、列管理、列宽调整、列拖拽、行拖拽、展开行、行内详情、行选择、单元格选择、区域复制粘贴、表头筛选、多列排序、汇总小计、冻结窗格 | 复用核心，通过明确能力配置接入；不复制核心 |
| 数据与视口策略 | 3 | 虚拟滚动、无限加载、服务端分页 | 通过 DataSource/Viewport 策略接入 |
| 派生组件 | 2 | 树形表格、主从表格 | 复用核心基础设施，具有独立公开规范 |
| 独立内核组件 | 4 | 透视表、交叉表、多维分析表、电子表格式表格 | 独立状态模型和内核，复用令牌、单元格、错误态与发布设施 |
| 场景预设 | 5 | 属性表格、键值表格、对比表格、矩阵表格、数据差异表格 | 组合 DataGrid 能力并冻结场景语义；动态聚合矩阵转入透视表 |

归类变化必须先回答“状态模型是否仍是稳定二维行列”。若答案为否，不能继续作为 DataGrid 能力堆叠。

## 5. 架构边界

### 5.1 稳定公开层

第一批允许对业务项目公开：

```ts
DataGrid
DataGridProps<TRow>
DataGridColumn<TRow, TValue>
DataGridDataSource<TRow>
DataGridSorting
DataGridFilters
DataGridRowSelection
DataGridPagination
DataGridColumnPinning
DataGridColumnSizing
DataGridError
DataGridRef
```

能力使用具名配置对象，避免无限平铺 props，也不暴露通用插件协议：

```tsx
<DataGrid
  data={dataSource}
  columns={columns}
  getRowId={(row) => row.id}
  sorting={{ value, onChange, mode: 'multiple' }}
  filtering={{ value: filters, onChange: setFilters }}
  rowSelection={{ value: selectedIds, onChange: setSelectedIds }}
  pagination={{ value: page, onChange: setPage }}
  columnPinning={{ defaultValue: { left: ['name'] } }}
  columnSizing={{ resizable: true }}
/>
```

每个能力配置遵守同一受控规则：提供 `value` 时由消费方控制；仅提供 `defaultValue` 时由组件控制；挂载后禁止在两种模式间静默切换。

### 5.2 数据源边界

首版只提供两个判别联合分支：

```ts
type DataGridDataSource<TRow> =
  | {
      kind: 'local';
      rows: readonly TRow[];
    }
  | {
      kind: 'remote-page';
      rows: readonly TRow[];
      rowCount: number;
      loading: boolean;
      error: DataGridError | null;
      onQueryChange: (query: DataGridQuery) => void;
      onRetry: (query: DataGridQuery) => void;
    };
```

组件不直接发网络请求。服务端分支只产生查询意图并显示消费方传入的结果状态，因此请求取消、认证、缓存和服务端幂等仍由业务数据层负责。

`remote-page` 模式禁止再次执行客户端排序、筛选和分页。相同查询的事件必须可去重，刷新失败不得清空上一份成功数据和当前选择。

### 5.3 内部能力层

内部能力模块可以组合以下扩展点：

- 状态切片及受控状态适配
- 行模型变换
- 命令和键盘绑定
- 表头、行、单元格与工具栏渲染槽
- ARIA 属性和状态播报
- 查询意图与观测事件

内部模块不得直接修改其他模块状态。跨能力协作通过核心命令和稳定 ID 完成，例如排序不能直接清空选择模块状态。

### 5.4 TanStack 隔离层

TanStack Adapter 负责：

- 把自有列定义转换为内部 `ColumnDef`
- 把自有状态转换为 TanStack 状态
- 读取行模型并转换为自有渲染模型
- 把 TanStack 回调转换为自有语义事件
- 封装版本差异和升级适配

以下内容不得出现在公开 `.d.ts`、包导出或业务示例中：

- `ColumnDef`
- `Table` 实例
- TanStack Row/Cell/RowModel
- TanStack 专属状态与事件类型
- `@tanstack/*` 子路径

通过公开类型快照与隔离消费测试阻断泄漏。升级 TanStack 时必须先通过 Adapter 契约测试，不允许把升级迁移成本转嫁给消费项目。

## 6. 核心不变量

1. 行身份只能来自 `getRowId`，不能来自当前数组索引。
2. 列身份来自显式列 ID；重排、隐藏和固定不能改变列身份。
3. 排序、筛选、分页和刷新只能改变可见集合或顺序，不能把选择迁移到另一业务行。
4. 重复行 ID 时必须阻断行级操作，不能静默合并。
5. 刷新失败时保留上一份成功数据、查询、选择和焦点上下文，并明确标记数据可能过期。
6. `remote-page` 不执行客户端查询变换；`local` 不伪造远程加载状态。
7. 固定列、列宽和横向滚动必须保持表头与单元格几何对齐。
8. 任意时刻最多一个活动单元格进入 tab 序列；活动节点不存在时不得留下悬空 `aria-activedescendant`。
9. 禁用、只读和加载不能只靠颜色表达，也不能吞掉已有数据和用户上下文。
10. 前端选择、隐藏和禁用不构成权限、多租户或数据范围控制。

## 7. 能力组合规则

第一批必须覆盖以下高风险组合，不以单能力测试通过代替：

| 组合 | 必须证明的结果 |
|---|---|
| 排序 + 选择 | 排序后仍按 rowId 选择同一业务行，不额外触发选择事件 |
| 筛选 + 选择 | 隐藏已选行不删除选择；界面明确区分“已选总数”和“当前可见已选数” |
| 分页 + 选择 | 翻页后选择不按页内索引漂移；全选语义明确为当前页或全部结果 |
| 服务端 + 排序/筛选 | 只发一次查询意图，不执行客户端二次变换 |
| 刷新错误 + 选择 | 旧数据和选择保留，重试不重复追加行 |
| 固定列 + 列宽调整 | 左右固定区与滚动区表头、行高和列边界对齐 |
| 键盘 + 横向滚动 | 活动单元格始终可见，焦点不跳出表格或落入卸载节点 |

“全选全部查询结果”不在首版默认提供。首版只允许明确的当前页/当前本地结果选择；跨页全选必须在后续规范中定义排除集、数据版本和服务端提交语义。

## 8. TreeGrid 扩展边界

第二批 `TreeGrid` 复用以下部分：

- 自有列定义、单元格渲染和主题令牌
- 稳定行 ID、错误态、加载态和观测事件
- 固定列、列宽调整、横向滚动
- 活动单元格、焦点可见和发布测试设施

`TreeGrid` 自己拥有：

- 父子关系和层级路径
- 展开/折叠状态
- `aria-level`、`aria-expanded`、`aria-setsize` 和 `aria-posinset`
- 左右方向键的层级导航
- 异步子节点加载与失败恢复
- 父子选择联动和半选语义
- 同级排序、父路径保留筛选和根节点分页语义

以下决策在 TreeGrid 单组件规范中冻结，不由 DataGrid 猜测：

1. 筛选命中子节点时保留完整父路径。
2. 排序只在同级兄弟节点内执行。
3. 默认选择互相独立；父子联动作为显式配置。
4. 未加载后代不参与客户端半选计算，界面必须标记选择范围未完全加载。
5. 分页按根节点计算，展开后的可见后代不单独消耗页额。
6. 折叠保留子节点选择和草稿；卸载不能等同于删除状态。

这些规则证明 TreeGrid 可以在当前架构上实现，同时避免污染首版 DataGrid 的二维状态模型。

## 9. 第一批生产执行剖面

每次实现仍执行唯一生产 SOP，DataGrid 第一批按以下顺序进入关口：

### Gate 0：冻结规范

- 用本设计修订 `02-data-grid.spec.json`。
- 冻结自有公开 API、35 KB 预算、本地/远程分页边界和首批非目标。
- 清空开放决策并完成组件维护、UX/无障碍和测试角色审批。
- 只有机器索引给出 `ImplementationReady` 和 `implementationAllowed=true` 才进入 RED。

### Gate 1：核心 RED

先建立并观察最小失败测试：

- 重复 rowId 阻断行级操作
- 排序后选择保持业务行
- 刷新失败保留旧数据和焦点
- 单一活动单元格键盘导航
- TanStack 类型没有泄漏到公开类型

失败原因必须是目标行为尚未实现，不得用语法错误或坏夹具冒充 RED。

### Gate 2：核心 GREEN

只实现行列模型、语义渲染、状态容器、数据源分支、错误恢复和 Adapter，使 Gate 1 通过。此阶段不实现未进入首批的能力。

### Gate 3：逐能力增量

按以下顺序每次完成一轮 RED、GREEN、回归：

```text
排序 -> 筛选 -> 行选择 -> 分页
-> 固定表头/列 -> 列宽调整
-> 完整键盘与状态播报
```

不得并行铺开多个未稳定公开 API。

### Gate 4：组合与视觉验证

- 执行第 7 节组合矩阵。
- 验证 390、768、1280 和 1440 像素，以及每个断点相邻宽度。
- 验证默认、暗色、紧凑和减少动态效果。
- 人工键盘复核完整主路径、错误路径和重试焦点恢复。
- 自动与人工无障碍验证目标为 WCAG 2.2 AA。

### Gate 5：性能与产物

- 本地基准：1,000 行 x 20 列。
- 远程分页基准：100,000 逻辑行，单页 100 行 x 20 列。
- 选择、排序意图、键盘移动和列宽反馈 p95 不高于 100ms。
- 子路径 gzip 增量不超过 35 KB。
- 隔离消费项目只能从打包产物的 `@company/ui/data-grid` 导入。

### Gate 6：演示、候选与回滚

- 规范站提供真实数据浏览场景，能实际演示排序、筛选、选择、分页、刷新失败和重试。
- 规范站只消费公开包 API，不读取源码内部路径。
- 候选产物、源码修订、测试结果和批准记录绑定同一完整性摘要。
- 验证降级到上一稳定版本的命令、数据状态兼容和消费方恢复路径。

## 10. 测试结构

| 层级 | 重点 |
|---|---|
| 类型契约 | 泛型推断、判别数据源、受控状态、公开导出、TanStack 类型隔离 |
| 单元测试 | 行列身份、状态转换、查询去重、选择集合和错误恢复 |
| 组件交互 | 鼠标、键盘、焦点、排序、筛选、选择、分页和重试 |
| 组合测试 | 第 7 节的能力交叉不变量 |
| 无障碍 | axe 自动扫描、ARIA 关系、人工键盘与读屏复核 |
| 视觉响应式 | 主题、密度、长文本、横向滚动、固定列与断点邻域 |
| 性能 | 指定数据规模下的 p95、长任务、重复渲染和包体 |
| 消费集成 | 打包、子路径导入、类型解析、SSR 导入与独立安装 |

不以代码覆盖率替代行为 oracle。每个强制行为都必须至少有一个能观察错误结果的测试。

## 12. 失败预演与停止线

| 失败方式 | 识别信号 | 阻断措施 |
|---|---|---|
| 公开 API 被 TanStack 绑死 | `.d.ts` 或示例出现 `@tanstack/*` | 类型快照和隔离消费测试失败即停止发布 |
| 为“插件化”提前设计万能协议 | 首版出现任意注册器、生命周期和依赖解析 | 首版只允许内部具名能力模块 |
| 34 项变成 34 份复制代码 | 相同排序/选择逻辑在多个组件重复 | 回到第 4 节归类，不通过架构评审 |
| 远程模式出现双重排序 | 服务端结果又在客户端重排 | 数据源分支契约测试阻断 |
| 选择随索引漂移 | 排序或翻页后指向另一业务行 | rowId 不变量测试阻断 |
| 固定列在复杂组合下错位 | 调整列宽后表头与单元格边界不一致 | 组合视觉与几何断言阻断 |
| TreeGrid 污染 DataGrid 核心 | DataGrid 首版出现层级、异步子节点语义 | 拆回派生组件，停止冻结 API |
| 包体和性能后期失控 | 单能力引入无关模块或 p95 超预算 | 子路径包体和逐能力性能门禁阻断 |

以下任一条件出现时不得开始或继续生产实现：

- DataGrid 规范仍有开放决策或公开 API 未冻结。
- 规定角色审批未绑定当前规范版本。
- 机器索引未允许实现。
- 业务方要求跨页全选、库存/金额写入等首批外语义，却未升级规范和风险。
- 实际发布产物不是测试使用的同一不可变产物。

## 13. 设计验收标准

本设计通过书面复核后，下一步只能是：

1. 修订 DataGrid 单组件规范，使其与本设计一致。
2. 运行规范校验并完成角色审批。
3. 生成针对实际 React 组件库仓库的逐文件 TDD 实施计划。
4. 通过准入后开始代码 RED。

不得直接从本文件跳到一次性生成 34 个组件。
