## 1. 目的与边界

本 SOP 规定 `DataGrid` 特有的实现顺序和验证方法。它不重复总 SOP 的发布治理，也不定义正确结果；正确结果以对应单组件规范为准。

首批只实施：

- 稳定行列身份与二维语义渲染
- 本地数据与受控远程分页数据
- 排序、筛选、行选择和分页
- 固定表头/列与列宽调整
- 初始加载、刷新、空、初始错误、刷新错误和重试
- 键盘导航、焦点恢复、无障碍、主题和响应式

下列内容不在本次执行范围：编辑、批量编辑、区域复制粘贴、虚拟滚动、无限加载、树形层级、透视、OLAP 和公式。出现范围外需求时停止执行，先修订规范、风险和实施 SOP。

## 3. 全程不变量

每个步骤都必须守住：

1. `rowId` 只能来自 `getRowId`，不能来自数组索引。
2. `columnId` 必须稳定；重排、隐藏、固定和调整列宽不得改变身份。
3. 排序、筛选、分页和刷新不得把选择迁移到另一业务行。
4. 重复 `rowId` 时阻断选择和行操作，不静默合并。
5. 远程分页只发查询意图，不执行客户端二次排序、筛选或分页。
6. 刷新失败保留上一份成功数据、查询、选择和焦点上下文。
7. TanStack 类型、实例和 RowModel 不得出现在公开导出或生成的 `.d.ts` 中。
8. 前端隐藏、禁用或选择不构成服务端权限、租户或数据范围控制。
9. 任何能力都先有正确 RED，再写最小生产代码；一次只增加一个能力切片。

任一不变量失败，停止当前 Gate，保留失败证据，不得以已知限制或后续修复继续晋级。

## 4. Gate D0：准入与仓库绑定

### 操作

1. 运行规范验证与分离校验。
2. 在机器索引中读取 `02:data-grid`，确认 `ImplementationReady` 和 `implementationAllowed=true`。
3. 核对公开 API 已冻结、开放决策为空，审批记录绑定当前规范版本且作者不是批准人。
4. 读取目标 React 仓库的规则、现有包结构、令牌、测试设施、发布流程和工作区状态。
5. 在执行清单中写入第 2 节全部真实输入。
6. 保存规范、总 SOP、组件实施 SOP 和源码修订的完整性摘要。

### 通过证据

- 规范和分离校验原始输出
- 机器索引中的 DataGrid 条目
- 审批记录及其规范版本
- 目标仓库工作区状态和已绑定命令
- 执行清单

### 停止条件

机器索引不是 `ImplementationReady` 或 `implementationAllowed` 不为 `true` 时，此 Gate 必须停止。缺少目标 React 仓库、真实命令、Owner、审批或准入中的任一项也必须停止；不得进入 D1。

## 5. Gate D1：锁定公开类型和 TanStack 隔离

### RED

先写类型契约测试，至少证明下列期望尚未实现：

- `DataGrid<TRow>` 从 `rows` 和列访问器正确推断 `TRow` 与单元格值类型。
- 本地与远程分页数据源是可穷尽判断的判别联合。
- 每个能力配置遵守 `value`/`defaultValue` 受控规则。
- 公开类型和 `.d.ts` 不包含 `@tanstack/`、`ColumnDef`、`Table`、TanStack Row/Cell/RowModel。
- `@company/ui/data-grid` 子路径可独立解析类型。

运行目标仓库的类型测试，保存因公开 API 尚未实现而失败的原始输出。语法错误、缺包、错误路径或坏夹具不算 RED。

### GREEN

1. 定义规范中冻结的自有公开类型和子路径导出。
2. 建立内部 TanStack Adapter，只在内部完成列定义、状态、回调和行模型转换。
3. 不实现排序、筛选等行为，只让类型契约和隔离测试通过。
4. 运行类型测试和受影响回归。

### 通过证据

- RED 与 GREEN 原始输出
- 公开 `.d.ts` 快照
- 子路径隔离类型消费结果
- TanStack 泄漏扫描结果为零

## 6. Gate D2：行列核心与基础状态

按下面顺序逐项完成 RED、确认失败原因、最小 GREEN、相关回归，不得一次性提交整段实现。

### D2.1 稳定身份

RED oracle：

- 三行排序或替换数组后仍以相同 `rowId` 关联业务对象。
- 两行返回同一 `rowId` 时，选择和行操作不可用，并出现表格级契约错误。
- 列重排后单元格仍关联相同 `columnId`。

GREEN 操作：实现行列注册、重复 ID 检查和自有渲染模型。生产错误态不得展示敏感行对象。

### D2.2 语义渲染与状态机

RED oracle：

- `loadingInitial`、`ready`、`empty`、`refreshing`、`errorInitial`、`errorRefresh` 都有可观测输出。
- 刷新错误保留旧行、查询、选择和活动单元格。
- 重试只发一次与失败查询一致的 `querySnapshot`。

GREEN 操作：实现最小状态容器、grid/row/columnheader/gridcell 语义、错误区域和重试命令。

### 通过证据

- 身份和状态机单元测试
- 组件交互测试
- 重复 ID 与刷新失败的失败注入记录
- 无新增 React 警告或控制台错误

## 7. Gate D3：能力切片

能力严格按以下顺序实施；前一项未通过组合回归，不得开始下一项：

```text
排序 -> 筛选 -> 行选择 -> 分页 -> 固定表头/列 -> 列宽调整
```

### D3.1 排序

RED：激活列头产生一次自有排序事件；ARIA `aria-sort` 正确；远程模式不在客户端重排。

GREEN：通过 Adapter 接入排序状态和本地行模型；远程分支只产生查询意图。

### D3.2 筛选

RED：本地筛选改变可见集合但不删除选择；远程筛选只产生一次查询意图；筛选结果数可被辅助技术获知。

GREEN：实现列筛选配置、查询快照和结果播报。

### D3.3 行选择

RED：排序、筛选和翻页后选择仍指向相同 `rowId`；界面区分已选总数和当前可见已选数；重复 ID 时阻断选择。

GREEN：实现按 ID 的选择集合和单选/多选模式。首版全选只允许当前页或当前本地结果，不实现跨全部查询结果全选。

### D3.4 分页

RED：本地分页使用本地行模型；远程分页不重复切片；翻页不按页内索引漂移选择；越界页在数据收缩后恢复到有效页。

GREEN：实现本地与远程分页分支、页码边界和查询事件。

### D3.5 固定表头与列

RED：水平和垂直滚动后固定区域仍与滚动区行高、列边界和表头对齐；键盘可以进入和离开固定区。

GREEN：实现固定区布局和滚动同步，不复制可聚焦单元格导致重复语义。

### D3.6 列宽调整

RED：鼠标和键盘调整遵守列最小/最大宽度；调整过程中表头与单元格对齐；受控配置只发事件不篡改外部状态。

GREEN：实现列宽命令和视觉反馈，只更新受影响列。

### 每个能力的必备证据

- 本能力的 RED 与 GREEN 原始输出
- 本能力公开事件和受控状态测试
- 已实现能力的完整组合回归
- 无契约外公开 API

## 8. Gate D4：组合、键盘与无障碍

### 组合矩阵

至少覆盖：

| 组合 | 必须通过的 oracle |
|---|---|
| 排序 + 选择 | 选择保持相同业务行且排序不额外触发选择事件 |
| 筛选 + 选择 | 隐藏已选行不删除选择，两个选择计数正确 |
| 分页 + 选择 | 翻页后不按页内索引漂移 |
| 远程 + 排序/筛选/分页 | 每次操作只发一次查询意图，没有客户端二次变换 |
| 刷新错误 + 选择 | 旧数据和选择保留，重试不追加重复行 |
| 固定列 + 列宽 | 固定区和滚动区几何对齐 |

### 键盘与焦点

1. 使用单一 roving tabindex 活动单元格。
2. 验证方向键、Home/End、Ctrl+Home/End、Space 和 Enter。
3. 活动单元格被筛选、翻页或卸载时，焦点移到规范定义的最近有效目标，不留下悬空引用。
4. 刷新失败时进入重试按钮；成功后返回原活动表头或单元格。

### 无障碍

- 自动扫描严重和高危问题为零。
- `aria-rowcount`/`aria-colcount` 表示逻辑规模。
- 排序、筛选结果和失败使用 `polite` 播报，不逐行播报刷新。
- 人工键盘完成主路径、错误路径和恢复路径。
- 人工读屏复核表头关联、选中状态、加载、错误和重试。

通过证据包括自动报告、人工执行人/环境/步骤/结论和焦点轨迹。

## 9. Gate D5：视觉、响应式与性能

### 视觉与响应式

使用稳定业务夹具验证默认、暗色、紧凑和减少动态效果主题，以及 `390`、`768`、`1280`、`1440` 像素和每个断点相邻宽度：

- 文本、工具栏、分页和错误操作不重叠或不可达。
- 390px 保持可达横向滚动，不把二维关系改成卡片。
- 固定列、滚动条和焦点环不遮挡关键内容。
- 加载、空、刷新和错误状态不改变表格外框导致页面跳动。

### 性能

记录设备、浏览器、数据、运行次数和统计方法：

- 本地：1,000 行 x 20 列。
- 远程分页：100,000 逻辑行，当前页 100 行 x 20 列。
- 选择、键盘移动、排序意图和列宽反馈 p95 不高于 100ms。
- 组件自身连续交互不持续产生超过 50ms 的长任务。
- `@company/ui/data-grid` 运行时 gzip 增量不超过 35 KB，排除 React/React DOM peer dependency。

任何超预算都必须停止；优化、拆分或按总 SOP 申请适用偏差后重新验证，不能改测试阈值掩盖结果。

## 10. Gate D6：打包、演示、候选与回滚

### 打包与隔离消费

1. 从锁定源码和依赖生成候选产物并记录完整性摘要。
2. 在隔离消费项目只从 `@company/ui/data-grid` 导入 ESM、类型和样式。
3. 验证 SSR 导入不访问浏览器专属对象，客户端水合无未解释差异。
4. 验证导入 DataGrid 不包含其他组件或演示站代码。

### 规范演示

使用候选包公开 API 演示：排序、筛选、选择、分页、固定列、列宽、初始加载、刷新、空、初始错误、刷新错误和重试。异常必须通过可重复夹具真实触发，不能只改变说明文字。

### R2 候选与回滚

1. 在至少一个代表性消费项目使用同一候选产物完成主路径、失败路径和恢复路径。
2. 执行降级到上一稳定版本的真实命令。
3. 验证受控状态、查询和消费方页面恢复；不删除仍被锁定的发布版本。
4. 候选生成后源码、依赖、构建环境或产物变化时，候选作废并重跑受影响 Gate。

### 完成条件

只有总 SOP 全部适用门禁、D0-D6、规范 oracle、R2 证据和发布批准都通过，才能晋级 Stable。完成代码或单测通过不代表完成组件。
