# 微信小程序扫码点餐系统分析方法

## 1. 写作目的

本文说明如何从架构师、业务分析师、测试设计师的角度分析微信小程序扫码点餐系统。

扫码点餐不是一个简单的“扫码 -> 点菜 -> 下单 -> 支付”线性流程。源码显示，它是由二维码业务类型、前端页面状态、桌台账单、支付方式、会员状态、门店配置、异常恢复共同驱动的信息系统。

因此，分析这类系统不能只依赖一张流程图，而应使用一组互补的图和分析技术。

## 2. 总体分析原则

### 2.1 不用一张图描述全部复杂度

流程图适合描述顺序动作，但扫码点餐系统的复杂度主要来自状态和条件：

- 当前二维码是什么业务类型。
- 当前是否已经绑定桌台。
- 当前桌台是否需要拉单。
- 当前订单是先付还是后付。
- 当前视图是否允许点餐或结账。
- 当前支付中心可用哪些支付方式。
- 当前会员余额是否足够。
- 支付失败或储值后如何回到原桌台。

这些问题如果全部塞进流程图，会导致图越来越大，最终无法指导实现和测试。

### 2.2 先分层，再画图

推荐先把系统拆成六层：

1. 系统边界层：系统和外部服务的关系。
2. 业务能力层：系统能完成哪些业务能力。
3. 页面状态层：小程序当前处于哪个页面状态。
4. 接口调用层：前端、后端、微信支付、CRM 之间如何调用。
5. 数据与金额层：关键数据和金额如何产生、变化、展示。
6. 异常恢复层：失败后如何重试、等待、跳转或报错。

每一层使用不同的图或表，而不是强行合并。

## 3. 推荐图谱组合

### 3.1 系统上下文图

系统上下文图用于回答：这个扫码点餐系统和哪些外部系统交互。

应该包含：

- 顾客微信小程序。
- Taro 前端。
- 二维码/商品服务。
- 订单服务。
- CRM 会员服务。
- 支付服务。
- 微信支付。
- 门店 POS 或线下开台系统。

它的价值是明确系统边界，防止分析只停留在前端页面。

### 3.2 业务能力地图

业务能力地图用于回答：系统到底提供哪些业务能力。

扫码点餐系统至少包含：

- 扫码识别。
- 业务类型分流。
- 桌台绑定。
- 点菜。
- 购物车。
- 创建订单。
- 拉取桌台账单。
- 查询已下单状态。
- 结账。
- 支付。
- 会员余额判断。
- 储值。
- 支付回流。

业务能力地图不关心页面先后顺序，它关注系统能力边界。

### 3.3 页面状态机图

页面状态机图是分析小程序自动化最重要的图。

它用于回答：当前页面是什么状态，允许执行什么动作，动作后会进入哪个状态。

核心状态包括：

- `ScanCode`：扫码页。
- `SelectTable`：选择桌台页。
- `PullSingle`：拉单 / 桌台账单页。
- `ORDER_FOODS`：点菜页。
- `CART`：购物车页。
- `ORDER_STATUS`：已下单页。
- `PAY_CENTER`：支付中心。

页面状态机图必须标注状态转换条件，例如：

- 是否存在 `bindusertaiid`。
- `handlePull()` 是否为真。
- `viewMode` 是否允许点餐或结账。
- `payType` 是先付还是后付。
- `g_scanTblQrcodeToHome` 是否开启。

对自动化测试平台来说，页面状态机比普通流程图更重要。

### 3.4 时序图

时序图用于回答：前端、后端、微信支付、CRM 之间谁先调用谁。

建议至少画四张：

1. 扫码识别时序图。
   - 小程序扫码。
   - 解析 `q`、`scene`、`code`、`mid`。
   - 调用二维码绑定查询。
   - 写入前端状态。

2. 点菜购物车时序图。
   - 查询菜品。
   - 加入购物车。
   - 修改菜品数量。
   - 查询购物车金额。

3. 下单和拉单时序图。
   - 购物车提交。
   - 创建订单。
   - 后付进入已下单页。
   - 查询订单 / 结账时重新拉取桌台账单。

4. 支付和储值时序图。
   - 进入支付中心。
   - 选择微信、会员卡、组合支付。
   - 订单支付。
   - 调起微信支付。
   - 支付成功确认。
   - 支付中心内储值或会员中心储值回流。

时序图能暴露跨系统边界和接口依赖。

### 3.5 数据流图

数据流图用于回答：数据从哪里来，在哪里变化，最终在哪里展示。

扫码点餐系统中最重要的数据流包括：

1. 二维码数据流。
   - `q` / `scene` / `code` / `mid`
   - 二维码绑定查询
   - `billStore`
   - 页面状态渲染

2. 购物车数据流。
   - 点菜页选择菜品
   - `addDish`
   - `modDishNum`
   - `cartDishs`
   - 购物车金额

3. 账单数据流。
   - `getOfflineOrderInfo`
   - `foodList`
   - `payList`
   - `bill.order`
   - 当前台账单金额

4. 支付数据流。
   - `paidAmount`
   - `payList`
   - 优惠券
   - 组合支付分摊
   - 微信支付金额
   - 会员卡支付金额

数据流图的重点是解释“这个值如何产生”，特别适合分析金额、状态、断言口径。

### 3.6 决策表

决策表用于回答：多个条件共同决定什么结果。

扫码点餐系统中建议建立以下决策表：

1. 二维码业务类型决策表。

| `businesstype` | 中文含义 | 目标页面 |
|---|---|---|
| `order` | 堂食桌台点餐 | `ORDER` 点餐 tab |
| `book_mgr` | 预先点菜 | `ORDER_FOODS` 点菜页 |
| `takeout` | 外卖 | `ORDER_FOODS` 点菜页 |
| `self_lifting` | 自提 | `ORDER_FOODS` 点菜页 |
| `pay` | 快捷付款 | `QUICKPAYMENT` 快捷付款页 |
| `evaluate` | 评价码 | `EVALUATE` 评价页 |
| `recharge` | 充值码 | `RECHARGE` 储值页 |

2. 拉单条件决策表。

| 条件 | 含义 |
|---|---|
| `businesstype == order` | 必须是堂食 |
| `payType == 1` | 后付 |
| `payType == 0 && mustOfflineOpenTbl == true` | 先付但必须线下先开台 |

只有满足以上拉单条件，才进入完整的桌台拉单逻辑。

3. 按钮显示决策表。

| 条件 | 显示按钮 |
|---|---|
| `viewMode == 0 || viewMode == 1` | 点餐 |
| `handlePull() == true` | 刷新账单 |
| `viewMode == 0 || viewMode == 2` | 结账 |

4. 购物车提交后跳转决策表。

| 条件 | 跳转 |
|---|---|
| `payType == 0` | 支付中心 |
| `qrCodeType == takeout` | 支付中心 |
| `qrCodeType == self_lifting` | 支付中心 |
| 其他后付堂食场景 | 已下单页 |

决策表能让复杂条件保持可读，也方便转成测试用例。

### 3.7 金额口径模型

扫码点餐系统必须单独建立金额口径模型。

至少区分：

1. 购物车金额。
   - 表示本次准备下单的菜品金额。
   - 出现在购物车页。
   - 不能代表整桌账单。

2. 当前台账单金额。
   - 表示当前桌台所有已送单菜品的账单金额。
   - 可能包含服务费、优惠、已支付记录。
   - 出现在拉单 / 桌台账单页。

3. 支付中心应付金额。
   - 以账单金额为基础。
   - 可能扣减已支付记录、优惠券、组合支付分摊。
   - 是支付断言的最终口径。

自动化测试不能跨页面复用金额。每进入一个页面，都应该重新读取当前页面的金额。

### 3.8 领域模型 / ER 图

领域模型用于回答：系统里有哪些核心业务对象，它们之间是什么关系。

核心对象包括：

- 门店。
- 桌台。
- 二维码。
- 会员。
- 会员卡。
- 菜品。
- 购物车。
- 订单。
- 桌台账单。
- 支付记录。
- 储值记录。
- 优惠券。

领域模型不一定等同数据库 ER 图。前期可以先画概念模型，后续再对齐数据库表和接口返回结构。

### 3.9 异常恢复矩阵

异常恢复矩阵用于回答：失败后应该怎么办。

建议按以下列设计：

| 异常场景 | 可能原因 | 当前页面 | 推荐动作 | 是否重试 | 是否需要重新拉单 |
|---|---|---|---|---|---|
| 扫码失败 | 二维码无效或网络失败 | 扫码页 | 提示失败并重新扫码 | 是 | 否 |
| 下单失败 | 菜品售罄或接口失败 | 购物车页 | 停留购物车并提示 | 可选 | 否 |
| 拉单失败 | 桌台账单接口失败 | 拉单页 | 提示并允许刷新 | 是 | 是 |
| 支付取消 | 用户取消微信支付 | 支付中心 | 保留支付中心状态 | 是 | 否 |
| 余额不足 | 会员卡余额不足 | 支付中心 | 组合支付或储值 | 否 | 视回流方式 |
| 储值成功但余额未刷新 | 异步延迟 | 储值页或支付中心 | 等待并刷新会员信息 | 是 | 回流后需要 |

异常矩阵能直接指导自动化平台的恢复策略。

### 3.10 模型驱动测试图

模型驱动测试用于把系统分析结果变成可执行自动化。

每个状态应定义：

- 状态名称。
- 识别条件。
- 可执行动作。
- 动作前置条件。
- 动作后可能进入的状态。
- 成功断言。
- 失败恢复策略。

示例：

| 状态 | 可执行动作 | 成功后状态 | 关键断言 |
|---|---|---|---|
| 扫码页 | 扫桌台码 | 点餐 tab 或首页 | `billStore` 已写入，页面跳转成功 |
| 点菜页 | 加菜 | 点菜页 | 购物车数量增加 |
| 点菜页 | 去下单 | 购物车页 | 购物车金额可见 |
| 购物车页 | 现在下单 | 支付中心或已下单页 | 创建订单成功 |
| 已下单页 | 查询订单 / 结账 | 拉单页 | 桌台账单可见 |
| 拉单页 | 结账 | 支付中心 | 应付金额可见 |
| 支付中心 | 会员付款 | 支付成功或余额不足 | 支付结果明确 |

这样自动化不是按固定步骤点击，而是根据当前状态选择合法动作。

## 4. 推荐分析顺序

建议按以下顺序分析：

1. 画系统上下文图。
   - 确认系统边界和外部依赖。

2. 画业务能力地图。
   - 拆分扫码、点菜、下单、支付、储值等能力。

3. 画页面状态机图。
   - 建立页面状态和动作之间的关系。

4. 建决策表。
   - 把 `businesstype`、`payType`、`viewMode`、配置开关等条件从流程图中拆出来。

5. 画关键时序图。
   - 描述扫码、下单、拉单、支付、储值回流的接口调用顺序。

6. 画数据流图和金额口径模型。
   - 解释关键数据和金额如何产生。

7. 建异常恢复矩阵。
   - 定义失败场景和恢复策略。

8. 建自动化测试状态模型。
   - 把分析结果转换成可执行测试模型。

## 5. 适合本文系统的最终文档结构

如果要继续完善扫码点餐业务文档，推荐使用以下目录结构：

```text
1. 系统背景和边界
2. 业务能力地图
3. 系统上下文图
4. 页面状态机图
5. 二维码业务类型决策表
6. 下单与支付时序图
7. 数据流和金额口径
8. 领域模型
9. 异常恢复矩阵
10. 自动化测试状态模型
11. 测试用例生成规则
```

## 6. 架构师结论

扫码点餐系统的核心复杂度不是页面数量，而是状态条件、接口边界和金额口径。

因此，单纯流程图只能作为入口说明，不能作为完整系统分析依据。

更合理的分析方式是：

```text
系统上下文图
  -> 业务能力地图
  -> 页面状态机图
  -> 决策表
  -> 时序图
  -> 数据流图
  -> 金额口径模型
  -> 领域模型
  -> 异常恢复矩阵
  -> 自动化测试状态模型
```

这套组合能同时服务三类人：

- 架构师：理解系统边界、模块职责和跨服务调用。
- 业务分析师：理解业务能力、规则和异常场景。
- 测试工程师：把页面状态、动作和断言转换成自动化测试模型。
