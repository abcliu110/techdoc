# 开源表单设计器候选池

日期：2026-07-07

## 选择原则

本候选池只服务“低代码平台表单设计器”专题，不替代主竞品分级。候选对象分两类：

```text
可视化表单设计器：直接提供拖拽、设计器、JSON/schema 编辑或可嵌入 Creator。
Schema 表单内核：不一定提供可视化设计器，但可支撑低代码平台的运行时渲染、校验、主题、扩展和多端适配。
```

## 阿里 Formily / Designable 重点研究

Formily / Designable 不应只作为普通开源表单库处理。它更接近“低代码表单设计器内核 + 可视化设计器生态”的组合，适合作为下一轮 A 级深拆对象。

重点研究问题：

```text
1. Formily core 如何表达 Form / Field / ArrayField / ObjectField 等字段状态。
2. reactive 与字段级状态管理如何避免整棵表单重渲染。
3. JSON Schema / x-component / x-decorator / x-reactions 如何表达控件、布局和联动。
4. validator、effects、reaction 如何承载校验和副作用。
5. Designable 的 designer tree、物料协议、属性面板和 schema transform 如何组织。
6. 设计态 schema 与运行态 schema 如何互转，哪些信息不能直接进入运行态。
7. 与 Ant Design、Vue、React 组件库绑定时，如何抽象 renderer registry。
```

## 暂不进入主矩阵的原因

这些项目大多是表单设计器或表单运行时内核，不是完整低代码平台。它们适合补强表单专题，但不能替代 Frappe、NocoBase、金蝶这类业务低代码主线竞品。

## 本轮结论

开源表单生态的主流路线集中在三种模式：

```text
JSON Schema 驱动：rjsf、JSON Forms、Form.io、react-json-schema-form-builder
字段状态/联动驱动：Formily、ngx-formly、FormKit、Vueform
设计器/Creator 驱动：SurveyJS Creator、Form.io Builder、Designable、Vueform Builder
```

对自研低代码平台的直接启发是：表单设计器不应只保存画布 JSON，而应明确区分 data schema、ui schema、rule schema、renderer registry、validator、submission 和 publish snapshot。
