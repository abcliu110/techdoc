# 证据：金蝶插件扩展模型

## 原始观察

“插件整体介绍”说明，使用设计器开发业务对象可以全程配置，但配置语义需要预先定义，遇到未覆盖的业务场景时需要插件补足。文档明确提出通常用业务对象设计器实现大部分业务语义，再用开放插件承担剩余复杂工作。

插件只能在系统功能运行到特定时刻时收到通知并进行处理，不能干预系统功能全过程。插件拿到的上下文和可调用控制方法也经过系统封装，属于受约束的适度干预。

插件类型按应用对象和业务过程拆分，包含动态表单插件、移动端表单插件、单据界面插件、基础资料界面插件、标准单据列表插件、左树右表单据列表插件、单据操作插件、单据转换插件、单据反写插件、报表取数插件、报表界面插件、工作流插件、引入引出插件、开放 API 插件、后台任务插件等。各类插件有不同注册入口。

文档还说明金蝶 AI 苍穹是 B/S 结构：交互界面运行在客户端，业务逻辑和业务插件运行在服务端。插件不能直接获取前端控件句柄，而是通过系统封装的视图模型接口、控件代理对象和数据模型间接访问、控制界面与数据。表单数据模型持有主实体模型、子实体、字段属性和基于主实体模型构建的数据包。

## 证据强度

直接事实：官方插件文档明确说明插件存在的原因、受限事件点、上下文边界、插件类型、注册入口，以及服务端视图模型 / 控件代理 / 数据模型的概念。

推断边界：该证据不能证明每个事件方法的源码实现，只能证明公开扩展机制和运行时分层。

## 可抽取知识

- 企业级低代码平台需要“配置优先、插件兜底”的扩展战略。
- 插件应运行在受控事件点上，而不是允许任意改写运行时全过程。
- 插件运行在服务端，通过视图模型、控件代理和数据模型访问页面，避免直接操作前端 DOM。
- 插件类型应围绕页面类型、列表、操作、转换、反写、报表、工作流、开放 API、后台任务等企业业务过程设计。
- 自研平台应优先定义扩展点、上下文对象、执行顺序和安全边界，再开放脚本或 Java 插件能力。
