# ERP 信息系统数据模型评价与知识库驱动新系统设计方法

## 0. 使用指南

一句话压缩：**从既有系统和竞品中提炼业务事实与设计决策（而非表结构），沉淀为可治理的知识卡片，按业务量级裁剪数据分层，用决策卡管住不可逆决策，用适应度函数防止上线后腐化，用双写灰度对账保证切换存活。**

按场景进入：

| 你要做什么 | 阅读章节 |
|---|---|
| 评价一个既有模块的数据模型 | §2 基本判断 → §3 评价维度 → §8 评分表 → §9 前后端统一评价 |
| 分析竞品 / 行业巨头产品 | §5 分析模板 → §20 巨头分析（含合法边界）→ §21 界面提取（字段级） |
| 设计一个新模块 | §4 数据分层（先看 4.1 量级门槛）→ §12 提炼模型 → §13 优化模型 → §14–16 UI 模型 → §18 决策卡 |
| 建设知识库 | §6 卡片类型与治理 → §10 落地架构 → §23 组织治理 |
| 新老系统切换 | §22 迁移与对账 |
| 从零启动整个方法 | §7 设计流程 → §17 工作流程 → §24 试点建议 |

三套分析模板的关系（逐层细化，不重复）：

```text
§5  系统级模板：一个系统整体的设计选择（问题/模型/取舍/边界）。
§20 产品级模板：行业巨头某业务域的逻辑模型与扩展机制。
§21 界面级模板：从菜单/列表/表单/按钮/报表逐字段反推逻辑模型。
使用顺序：先 §5 建立整体判断，需要深入某域时用 §20，拿不到内部资料时用 §21 从界面反推。
```

## 1. 背景

在 ERP、CRM、POS、库存、支付、报表等信息系统中，很多长期复杂度并不是来自代码本身，而是来自早期业务模型和数据模型设计不合理。

常见问题包括：

- 核心实体边界不清，后续业务只能不断加字段、加特殊判断。
- 状态、金额、库存、积分、券等关键业务缺少流水或事件记录，导致历史不可追溯。
- 业务表同时承担交易写入、流程查询、报表统计和审计追溯，最终 SQL 复杂、性能下降。
- 报表维度不是模型内生能力，只能通过多表拼接、临时宽表或手工补数实现。
- 新增业务类型时，需要改大量表结构、接口、页面和统计逻辑。
- 前端页面、后端接口、数据库模型之间没有统一业务边界，导致模块越做越重。

因此，设计新系统时不应只参考单个老系统，也不应直接照抄竞品表结构。更合理的方法是：分析多个既有系统和第三方系统，提炼业务事实、设计模式、反模式和适用边界，再基于目标业务重新建模。

## 2. 基本判断：模型没有绝对正确，只有场景下的合理性

数据库模型没有脱离业务目标的绝对“正确”。一个模型是否合理，必须放在具体约束下评价：

- 当前业务边界是什么？
- 未来需要支持哪些扩展？
- 哪些数据必须强一致，哪些可以最终一致？
- 哪些查询是高频交易查询，哪些是离线分析查询？
- 哪些历史状态必须可还原？
- 报表需要哪些维度、指标和钻取能力？
- 系统是否支持多组织、多门店、多租户、多渠道？
- 旧数据、旧接口、旧流程是否需要兼容？

如果模型只能支撑当前页面录入，却不能支撑状态追溯、业务扩展和报表分析，那么它在短期内可用，但长期成本会很高。

## 3. 数据模型评价的核心维度

### 3.1 业务事实表达

评价模型前，先不要看表结构，而要确认业务事实是否被正确表达。

典型业务事实包括：

- 客户是谁。
- 商品是什么。
- 库存在哪里。
- 订单发生了什么。
- 价格如何形成。
- 权益如何产生、核销、回滚。
- 资金如何流转。
- 状态如何变化。
- 责任主体是谁。
- 时间点和业务归属是什么。

评价问题：

- 核心实体是否稳定？
- 实体之间的关系是否清晰？
- 状态是否可追踪？
- 历史是否可还原？
- 金额、库存、积分、券是否有流水？
- 枚举和类型是否可扩展？
- 报表维度是否天然可查？
- 权限、租户、组织边界是否内生支持？

### 3.2 扩展性压力测试

不能只问“当前需求能不能实现”，还要用未来场景压测模型。

例如会员权益模型至少要验证：

- 积分。
- 返现。
- 返券。
- 等级。
- 储值。
- 实物券。
- 组合优惠。
- 反结账。
- 退款。
- 部分退款。
- 跨门店核销。
- 报表统计。
- 历史追溯。

评价问题：

- 新增一种业务类型，是新增配置，还是新增字段？
- 新增一种状态，是新增枚举，还是改主流程？
- 新增一个报表维度，是直接聚合，还是需要跨多张表拼接？
- 历史数据是否能按当时规则还原？
- 新规则是否影响老业务？
- 旧逻辑是否可以并行保留或灰度切换？

如果每增加一种权益、支付方式、库存动作或审批类型，都需要修改大量表结构和报表 SQL，说明模型抽象层级可能不合理。

### 3.3 状态机与生命周期

复杂业务系统必须显式表达生命周期。

需要关注：

- 状态是否完整覆盖业务流程。
- 状态变化是否有合法路径。
- 是否支持撤销、反审、反结账、退款、作废等逆向操作。
- 状态变更是否记录操作者、时间、原因和来源。
- 当前状态和历史事件是否分离。
- 报表是否依赖状态快照，还是依赖事件流水重算。

不建议只在主表上放一个状态字段，然后让所有历史和报表都围绕这个字段推导。主表状态适合表达“当前结果”，事件或流水适合表达“为什么变成这样”。

### 3.4 流水、事件与审计

凡是涉及金额、库存、积分、优惠券、会员权益、审批、支付、结算、退款的业务，都应该优先考虑流水或事件模型。

典型分层：

- 账户表：表示当前余额或当前状态。
- 流水表：表示每一次变化。
- 业务单据表：表示变化来源。
- 事件表：表示系统间传播、异步处理和重算依据。
- 快照表：表示某个时间点的统计结果。

评价问题：

- 当前值能否由流水解释？
- 发生争议时能否追溯每一步？
- 退款、反结账、冲正是否有独立记录？
- 是否能区分业务发生时间、入账时间、统计时间？
- 是否能支持幂等、重试和异步补偿？

### 3.5 报表模型

很多信息系统的后期痛点来自报表。业务表如果没有为分析维度留下结构，后续只能靠复杂 SQL 和临时逻辑拼凑。

常见报表维度：

- 时间。
- 门店。
- 商户。
- 组织。
- 商品。
- 分类。
- 会员。
- 渠道。
- 支付方式。
- 优惠类型。
- 操作员。
- 单据状态。
- 活动或规则。

评价问题：

- 常见报表是否可以从明细表自然聚合？
- 主交易表是否记录了必要维度？
- 历史维度变化后，老报表是否仍按当时口径展示？
- 是否需要事实表、维度表、宽表或指标表？
- 高频报表是否有预聚合或异步计算机制？

对于复杂 ERP 或 SaaS 系统，不应强迫一套业务表同时承担交易写入和复杂 BI 分析。更合理的是建立“交易模型 + 分析模型”。

## 4. 推荐的数据分层方式

一个长期可扩展的信息系统，通常应将数据模型分为几类：

| 层级 | 作用 | 示例 |
|---|---|---|
| 主数据 | 稳定业务对象 | 商品、客户、门店、供应商、会员 |
| 交易单据 | 业务发生记录 | 订单、采购单、销售单、库存单、支付单 |
| 明细行 | 交易的可计算颗粒 | 订单明细、库存明细、支付明细、优惠明细 |
| 流水 | 金额、库存、权益等变化过程 | 库存流水、积分流水、余额流水、券核销流水 |
| 状态记录 | 生命周期变化 | 审批记录、订单状态记录、结账记录 |
| 事件 | 系统间同步和异步处理 | 订单已支付、库存已扣减、权益已发放 |
| 快照 | 某个时点的结果 | 日结快照、库存快照、会员等级快照 |
| 报表模型 | 面向统计分析 | 销售事实表、支付事实表、会员权益事实表 |
| 配置规则 | 驱动业务变化 | 价格规则、促销规则、积分规则、审批规则 |

核心原则：

- 主表表达当前业务对象。
- 明细表达计算颗粒。
- 流水表达变化过程。
- 事件表达系统间传播。
- 快照表达时点结果。
- 报表模型表达分析口径。
- 配置规则表达可变业务策略。

### 4.1 分层的量级门槛

九层不是全选，方案复杂度必须与业务量级匹配。先做数量级估算（日均单量、门店数、租户数、增长率），再裁剪分层：

| 量级 | 必做层 | 可缓做层 |
|---|---|---|
| 单店 / 日均千单以下 | 主数据、交易单据、明细、流水（金额/库存/权益） | 事件用同步调用替代；快照用定时汇总替代；报表用视图直查明细 |
| 多门店 / 日均千到十万单 | 上述 + 状态记录、事件、快照、预聚合宽表 | 独立数仓可用宽表过渡；配置规则先枚举后引擎 |
| 多租户 / 日均十万单以上 | 全部九层 + 数仓分层 + 指标血缘 + 规则引擎 | — |

裁剪原则：

- **流水层任何量级都不可省**：流水缺失是不可逆损失，历史无法事后补造。
- 事件、快照、数仓是可后补的可逆决策，业务量到了再加，成本可控。
- 判断口诀：**丢了补不回来的先做，晚做只是花钱的后做**。

## 5. 多系统与竞品分析方法

参考老系统和第三方系统时，不要照抄表结构，而要提炼“设计选择”。

每个被分析系统都应形成以下内容：

```text
1. 业务问题：它要解决什么业务？
2. 核心模型：它用哪些实体表达业务？
3. 状态模型：业务生命周期如何变化？
4. 流水模型：关键数据如何追溯？
5. 报表模型：统计口径如何实现？
6. 扩展方式：新增类型、规则、渠道的成本如何？
7. 性能策略：高频查询和历史数据如何处理？
8. 优点：哪些设计值得复用？
9. 缺点：哪些设计造成复杂度？
10. 适用边界：适合什么场景，不适合什么场景？
11. 迁移启示：如果重做，应该保留什么、修正什么？
```

示例：

```text
问题：会员权益扩展困难。
系统 A 做法：积分、返券、返现分别建表，业务简单但扩展重复。
系统 B 做法：统一权益流水，追溯能力强，但核销模型不足。
竞品 C 做法：权益账户 + 权益事件 + 权益核销记录。
推荐方案：权益账户、权益规则、权益流水、权益核销、权益回滚分层。
适用场景：积分、储值、返券、实物券、跨门店核销。
风险：统一模型不能过度抽象，具体权益仍需要保留业务语义。
```

## 6. 知识库应该沉淀什么

为了让 AI 基于老系统和竞品快速设计新系统，知识库不应只存源码片段或长篇总结，而应沉淀结构化知识卡片。

建议至少包含以下卡片类型：

| 卡片类型 | 内容 |
|---|---|
| 业务能力卡片 | 某个模块解决的业务能力、边界、输入输出 |
| 数据模型卡片 | 实体、关系、主键、状态、生命周期 |
| 报表模型卡片 | 指标、维度、事实表、统计口径 |
| 状态机卡片 | 状态、转换、逆向操作、异常路径 |
| 流水事件卡片 | 金额、库存、积分、券、审批等变化记录 |
| 设计模式卡片 | 可复用的建模方法和适用条件 |
| 反模式卡片 | 已验证会带来复杂度或性能问题的做法 |
| 竞品对比卡片 | 多系统方案的优缺点和边界 |
| 决策记录卡片 | 为什么选某个方案，拒绝了哪些方案 |
| 迁移方案卡片 | 老模型到新模型如何兼容和迁移 |

知识卡片示例：

```yaml
type: design_pattern
name: 权益账户与权益流水分离
domain: CRM / 会员权益
problem: 积分、返现、储值、券等权益需要统一追溯和报表统计
solution: 当前值放账户表，变化过程放流水表，发放和核销通过事件关联业务单据
reuse_when:
  - 需要反结账、退款、冲正
  - 需要会员权益报表
  - 需要跨门店或跨渠道核销
avoid_when:
  - 只有一次性简单优惠
risks:
  - 统一模型过度抽象会掩盖具体权益差异
  - 流水幂等和冲正规则必须明确
evidence:
  - 参考系统 A 的积分流水
  - 参考系统 B 的券核销记录
```

知识库治理规则（防止退化为文档坟场）：

- 每张卡片有 owner 和复核周期（建议 6 个月），过期未复核自动标记 stale，stale 卡片不得作为新设计依据。
- 卡片有状态：draft / active / deprecated（须注明被哪张卡片替代）。
- 警惕古德哈特定律：**不考核卡片数量**——数量成为目标后必然灌水。考核的是"新设计引用了多少张卡片、被拒绝方案是否有卡片支撑、有多少卡片被证伪后及时废弃"。
- 卡片被实践证伪时，废弃动作和证伪证据本身就是一张新的反模式卡片。

## 7. 新系统设计流程

推荐采用以下流程：

```text
旧系统/竞品源码与文档
        ↓
AI 分析
        ↓
业务能力卡片
数据模型卡片
报表模型卡片
扩展性问题卡片
设计模式卡片
反模式卡片
        ↓
目标系统设计
        ↓
数据库模型 / 后端模块 / 前端页面 / 报表模型 / 测试用例
```

具体步骤：

1. 选定一个业务域，不要一开始覆盖整个 ERP。
2. 收集老系统、竞品、行业资料和现有报表。
3. 用统一模板分析每个系统。
4. 提炼业务事实，而不是直接抄表。
5. 对多个方案做横向对比。
6. 建立目标业务能力模型。
7. 设计领域实体、状态机、流水、事件和报表模型。
8. 用未来场景做扩展性压力测试。
9. 输出数据库模型、接口模型、前端页面模型和测试模型。
10. 将最终决策回写知识库，形成可复用设计模式。

AI 参与本流程时的分工边界：

- AI 按《系统化拆解方法论-AI执行版-v2》第 5 节逆向协议执行分析和生成，输出须带证据强度标注（直接事实/主流共识/经验判断/推断/假设）。
- 步骤 1–5（分析、提炼、对比）AI 可主导，人负责抽查证据。
- 步骤 6–7 中的**单向门决策（实体边界、状态机、口径、契约）必须由人评审拍板**，AI 只提供备选方案和取舍分析，不代替决策。
- AI 生成的知识卡片默认 confidence 不高于 medium，经人工复核后方可升级。

第一版建议选择一个模块跑通，例如：

- 会员 / CRM。
- 订单。
- 库存。
- 支付。
- 报表。

不要一开始试图覆盖整个 ERP。先拿一个模块验证“分析旧系统 -> 提炼模型 -> 评分 -> 生成新设计”的闭环。

## 8. 数据模型评分表

可以使用以下评分表评价一个模块的数据模型。每项 1 到 5 分：

| 维度 | 评价问题 | 分值 |
|---|---|---|
| 业务事实清晰度 | 实体、关系、行为是否准确表达业务 | 1-5 |
| 生命周期完整性 | 状态机、逆向操作、异常路径是否明确 | 1-5 |
| 历史可追溯性 | 是否有流水、事件、审计记录 | 1-5 |
| 报表友好度 | 常见指标和维度是否自然支持 | 1-5 |
| 扩展成本 | 新增类型、规则、渠道是否低成本 | 1-5 |
| 性能可控性 | 高频查询、聚合、历史数据是否有策略 | 1-5 |
| 一致性边界 | 强一致、最终一致、异步补偿是否明确 | 1-5 |
| 前后端契约 | 页面、接口、DTO、权限是否匹配业务边界 | 1-5 |
| 兼容迁移能力 | 老数据、老接口、老流程是否可迁移 | 1-5 |
| 可测试性 | 是否容易构造用例验证业务正确性 | 1-5 |

评分解释：

- 1 分：只能支撑当前功能，依赖大量特殊逻辑。
- 2 分：基本可用，但扩展和报表明显困难。
- 3 分：能支撑主要业务，部分复杂场景需要补偿设计。
- 4 分：模型边界清晰，扩展、报表和追溯能力较好。
- 5 分：模型稳定，可解释、可扩展、可迁移、可验证。

评分不是为了追求满分，而是为了暴露设计风险。低分项应该进入设计决策记录，明确是否立即修正，还是保留为后续演进点。

一票否决项（不参与平均分，出现即必须修正）：

- 涉及金额、库存、权益的业务，"历史可追溯性" ≤ 2 分：流水缺失是不可逆损失，无法事后补造。
- 有逆向操作（退款/反结账/冲正）的业务，"生命周期完整性" ≤ 2 分：上线后补状态机等于重做。
- 多组织/多租户业务，交易记录未固化归属边界：历史归属会被后续配置变更污染。

其余维度低分可以有意识地接受并排期演进；一票否决项不能。区分标准就是 4.1 的口诀：**丢了补不回来的必须现在做**。

### 8.1 从一次性评分到持续校验（适应度函数）

评分是设计期的人工检查，只发生一次；模型上线后会持续腐化。应把关键评分维度转成**可自动执行的校验**，纳入发布流水线或每日任务：

| 评分维度 | 适应度函数（自动化校验） |
|---|---|
| 历史可追溯性 | 每日校验"账户当前值 = 期初 + 流水累计"，偏差即告警 |
| 生命周期完整性 | 扫描单据表，非法状态跳转数必须为 0 |
| 报表友好度 | 同一指标在报表层与明细层重算，偏差超阈值告警 |
| 一致性边界 | 幂等重放测试：重复请求不产生重复流水，纳入回归用例 |
| 性能可控性 | 核心单据提交 P99 阈值压测，纳入发布流水线 |
| 前后端契约 | 自动化用例遍历"状态 × 可执行动作"矩阵，UI 可点但后端拒绝的动作数必须为 0 |

规则：**低分项若决定暂不修正，必须配一条适应度函数监控它是否恶化**——接受风险可以，失明不可以。

## 9. 前端、后端和业务设计的统一评价

数据模型不能孤立评价。一个好模型必须能支撑前端、后端和报表的统一表达。

### 9.1 前端设计

评价问题：

- 页面是否对应清晰的业务对象？
- 表单字段是否直接映射业务语义，而不是数据库偶然字段？
- 状态变化是否能在界面上解释？
- 列表筛选项是否对应稳定报表维度？
- 权限和组织边界是否能被前端自然表达？

### 9.2 后端设计

评价问题：

- Controller、Service、DAO 是否围绕业务能力划分？
- DTO、VO、Entity 是否保持业务语义一致？
- 枚举是否端到端建模，而不是散落魔法值？
- 业务规则是否配置化、版本化或可灰度？
- 老逻辑是否可并行保留？

### 9.3 具体业务设计

评价问题：

- 一个业务动作会影响哪些实体、流水、状态和事件？
- 是否有幂等键？
- 是否有回滚或冲正机制？
- 是否支持异常补偿？
- 是否能输出字段级证据验证结果？

## 10. 推荐落地架构

第一阶段采用轻量方案：

```text
Markdown 知识库
+ 统一分析模板
+ 数据模型评分表
+ Git 版本管理
```

第二阶段增加检索能力：

```text
PostgreSQL + pgvector
+ Markdown 知识卡片
+ 元数据检索
+ 向量检索
```

第三阶段增加代码关系图谱：

```text

+ 类、方法、接口、表、枚举、调用链关系
+ 模块影响范围分析
```

第四阶段形成系统生成器：

```text

        ↓
检索相关知识卡片
        ↓
生成 PRD
        ↓
生成领域模型
        ↓
生成数据库设计
        ↓
生成接口设计
        ↓
生成前端页面设计
        ↓
生成测试计划
```

## 11. 关键原则

1. 不要只存答案，要存设计决策。
2. 不要只存表结构，要存业务事实和适用边界。
3. 不要只看当前功能，要用未来场景压力测试。
4. 不要让业务表同时承担所有报表责任。
5. 涉及金额、库存、积分、券、状态流转的业务必须有流水或事件。
6. 报表维度应该在模型设计时提前考虑。
7. 参考竞品时不要照抄实现，要提炼模式和反模式。
8. 新系统设计必须能说明被拒绝方案和拒绝原因。
9. 所有模型设计都要能落到前端页面、后端接口、数据库和测试用例。
10. 知识库最终目标不是问答，而是支撑更高质量的新系统设计。
11. 方案复杂度必须与业务量级匹配：先估数量级，再选分层（见 4.1），不为千级业务建百万级架构。
12. 不可逆决策（实体边界、状态机、口径、契约）重评审慢落锤，可逆决策快速试错（见第 18 节决策分级）。
13. 新系统成败在切换：双写、灰度、对账、回滚必须是设计的一部分，不是上线前的补丁（见第 22 节）。
14. 模型和知识库必须有明确责任人，无人负责的模型必然腐化（见第 23 节）。

## 12. 如何提炼模型

模型提炼的目标不是把现实业务完整复制进系统，而是找到稳定、可表达、可扩展、可验证的业务结构。优秀的架构设计和产品设计都要先完成“提炼”，再进入数据库、接口和页面设计。

### 12.1 从业务语言中提炼概念

先收集业务人员、页面、报表、接口和代码中反复出现的名词与动词。

名词通常候选为实体、值对象、角色、资源或维度：

- 会员、客户、门店、商品、订单、库存、券、账户。
- 等级、渠道、组织、仓库、批次、活动、规则。

动词通常候选为业务动作、状态变化或事件：

- 下单、支付、核销、退款、反结账、入库、出库、调拨、审批。
- 发放、冻结、解冻、冲正、作废、同步、结算。

提炼时要区分：

- 业务对象：长期存在，有身份，例如会员、商品、订单。
- 值对象：没有独立身份，依附于对象，例如金额、地址、规格、时间范围。
- 业务动作：会改变状态或产生记录，例如支付、退款、核销。
- 业务规则：决定动作是否合法或结果如何计算，例如积分规则、价格规则。
- 业务事件：动作完成后发生的事实，例如订单已支付、库存已扣减。
- 报表维度：用于统计和筛选的稳定分类，例如门店、渠道、分类、操作员。

### 12.2 从业务流程中提炼生命周期

每个核心对象都要画出生命周期，而不是只给一个状态字段。

提炼步骤：

1. 找出对象从创建到结束的完整过程。
2. 标出每个可见状态。
3. 标出状态转换动作。
4. 标出逆向动作，例如取消、退款、反审、反结账。
5. 标出异常动作，例如超时、失败、补偿、人工处理。
6. 标出每次转换需要记录的操作者、原因、时间和来源。

如果一个状态变化无法解释“谁在什么时间因为什么动作造成”，说明生命周期模型不完整。

### 12.3 从报表反推事实模型

报表不是后期附属物，而是检验模型是否合理的重要工具。

提炼方法：

1. 收集目标报表和历史报表。
2. 拆出每个指标，例如销售额、实收额、退款额、库存成本、权益核销额。
3. 拆出每个维度，例如时间、门店、商品、会员、渠道、支付方式。
4. 追问每个指标来自哪类业务事实。
5. 判断业务发生时是否已经记录了这些维度。

如果一个报表指标需要从多个不稳定字段拼接推导，说明事实模型可能缺失。应考虑增加明细、流水、事件、快照或事实表。

### 12.4 从异常场景中提炼边界

普通流程只能验证模型可用，异常流程才能验证模型是否健壮。

必须压测：

- 重复提交。
- 支付成功但回调失败。
- 部分退款。
- 跨日退款。
- 反结账后再结账。
- 规则变更后老订单如何展示。
- 门店、组织、租户变更后历史数据如何归属。
- 商品改名、分类调整后历史报表如何展示。
- 异步同步失败后如何补偿。

异常场景会暴露模型是否需要幂等键、事件表、快照、版本化规则、冲正流水和审计记录。

### 12.5 从多个系统中提炼共性与差异

分析老系统和竞品时，用同一模板比较，不要直接比较表名。

对每个业务域提炼：

- 共性实体：所有系统都存在的核心对象。
- 差异实体：某些系统才有的对象，说明它服务特殊业务。
- 共性动作：所有系统都支持的业务行为。
- 差异动作：体现产品能力差异的行为。
- 共性报表：所有系统都需要的统计口径。
- 差异报表：体现管理精细度的统计口径。
- 成功模式：扩展成本低、语义清楚、报表友好的设计。
- 失败模式：靠字段堆叠、SQL 拼凑、状态混乱支撑的设计。

最终沉淀的不是“某系统有哪张表”，而是“为什么它这样建模、解决了什么问题、牺牲了什么能力”。

## 13. 如何优化数据模型

数据模型优化不是简单范式化或反范式化，而是在业务表达、扩展能力、查询性能和历史追溯之间做清晰取舍。

### 13.1 优化实体边界

常见问题：

- 一个表承担多个业务对象。
- 一个对象被拆得过碎，导致所有查询都要多表拼接。
- 主表堆满只对少数业务有效的字段。
- 类型字段过多，实际隐藏了多个生命周期不同的对象。

优化方法：

- 如果两个概念生命周期不同，应拆开。
- 如果两个概念总是一起创建、一起修改、一起删除，可以合并或作为从属明细。
- 如果字段只对某个类型有效，优先考虑子表、扩展表或配置化结构。
- 如果对象需要独立权限、审计、状态和报表，应独立建模。

### 13.2 优化关系模型

关系设计要表达真实业务约束。

优化方法：

- 一对一关系要确认是否真的是同生命周期。
- 一对多关系要确认明细是否需要独立状态。
- 多对多关系要显式建中间表，并记录关系属性，例如生效时间、来源、权重、排序。
- 层级关系要确认是否需要历史版本、移动记录和路径查询。
- 组织、门店、租户、渠道等边界字段应在交易发生时固化，避免历史归属被当前配置影响。

### 13.3 优化状态与流水

主表状态只能表达当前结果，不能替代流水。

优化方法：

- 当前状态放主表。
- 状态变化放状态记录表。
- 金额、库存、权益变化放流水表。
- 可撤销业务使用冲正流水，而不是直接修改历史流水。
- 规则计算结果要记录规则版本或计算快照。
- 所有外部回调、异步消息、同步任务要有事件记录和幂等键。

### 13.4 优化报表能力

报表优化应从模型层开始，不要只靠 SQL 调优。

优化方法：

- 将交易明细设计成可聚合颗粒。
- 在交易发生时固化关键维度。
- 对高频统计建立事实表或宽表。
- 对跨日、跨月统计建立快照。
- 对规则变更敏感的数据记录当时口径。
- 区分实时看板、经营报表、财务核算和审计追溯，不同目标使用不同模型。

### 13.5 优化扩展能力

扩展性不是把所有东西都做成 JSON，也不是无限加配置。

优化方法：

- 稳定核心字段结构化。
- 变化频繁但不参与核心查询的属性可扩展化。
- 业务类型使用枚举和策略，不使用散落魔法值。
- 规则配置要版本化，历史单据引用规则快照。
- 插件化或扩展点只用于明确存在多实现的地方。
- 新增业务类型时，优先评估是否是同一业务抽象下的新策略，还是应该成为新实体。

## 14. 如何提炼 UI 模型

UI 模型不是页面堆叠，而是用户任务、业务对象、状态反馈和操作路径的可视化表达。好的 UI 模型应该让用户自然理解业务，而不是暴露数据库结构。

### 14.1 从用户任务提炼页面

先定义用户要完成的任务，而不是先画菜单。

提炼问题：

- 用户是谁？
- 他在什么场景下进入系统？
- 他要完成什么任务？
- 他判断成功的标准是什么？
- 他需要看到哪些信息才能做决定？
- 他下一步最可能做什么？

页面应该围绕任务组织，例如：

- 查询和筛选。
- 创建和编辑。
- 审核和处理。
- 对账和核验。
- 异常处理。
- 报表分析。

不要把数据库表一张张映射成页面。很多表是技术结构，不是用户心智模型。

### 14.2 从业务对象提炼信息架构

UI 信息架构要匹配业务对象和生命周期。

常见结构：

- 列表页：发现对象、筛选对象、批量处理。
- 详情页：理解对象、查看状态、追踪历史。
- 编辑页：创建或修改核心信息。
- 操作弹窗：完成局部动作。
- 工作台：聚合待办、异常、指标和快捷动作。
- 报表页：围绕指标、维度、趋势和钻取组织。

评价问题：

- 用户能否一眼知道当前对象是什么？
- 关键状态是否可见？
- 下一步动作是否清晰？
- 历史记录是否容易追溯？
- 异常原因是否可解释？
- 批量操作是否有明确反馈？

### 14.3 从状态机提炼交互

复杂业务 UI 必须跟状态机一致。

设计方法：

- 每个状态下只展示合法动作。
- 危险动作需要二次确认和影响说明。
- 异步动作要展示处理中、成功、失败和重试。
- 逆向动作要展示影响范围，例如退款会影响库存、权益、报表。
- 状态变化后要展示证据，例如操作记录、流水、回调结果。

如果 UI 上能点击一个后端不允许的动作，说明前后端状态模型没有对齐。

### 14.4 从报表目标提炼分析界面

报表 UI 不是把表格字段全部展示出来，而是帮助用户回答经营问题。

提炼步骤：

1. 明确报表要回答的问题。
2. 确定核心指标。
3. 确定筛选维度。
4. 确定默认时间范围。
5. 确定钻取路径。
6. 确定异常提示和解释口径。

报表页面通常应包含：

- 核心指标区。
- 趋势区。
- 维度分布区。
- 明细列表。
- 钻取入口。
- 口径说明。

但口径说明不应替代模型正确性。若口径需要大量解释才能理解，往往说明数据模型或指标模型不清晰。

## 15. 如何优化 UI 模型

UI 优化不是美化页面，而是降低用户完成任务的认知成本和操作成本。

### 15.1 优化任务路径

评价问题：

- 高频任务是否一步可达？
- 低频危险任务是否被适当收敛？
- 是否存在重复录入？
- 是否需要用户记住上一步信息？
- 是否把系统内部流程暴露给了用户？

优化方法：

- 高频动作放在主路径。
- 危险动作放在受控入口。
- 相关信息就近展示。
- 可从上下文推导的内容不要让用户重复输入。
- 批量任务提供预览、结果反馈和失败明细。

### 15.2 优化信息密度

企业系统不应盲目追求大留白，也不应把所有字段挤在一起。

优化方法：

- 列表页优先展示识别对象和决策所需字段。
- 详情页按业务分组，而不是按数据库字段顺序。
- 重要状态使用稳定位置和一致样式。
- 次要信息折叠到更多区域。
- 金额、库存、状态、异常等高风险信息要明确可见。

### 15.3 优化错误与异常体验

企业系统的异常体验必须可操作。

优化方法：

- 错误提示要说明原因、影响和下一步。
- 校验错误要定位到字段。
- 异步失败要支持重试或转人工。
- 权限不足要说明缺少什么权限。
- 业务规则不满足要说明是哪条规则阻止。
- 批量失败要展示成功数、失败数和失败明细。

### 15.4 优化前后端契约

好的 UI 模型需要后端接口支持，而不是前端自己拼业务。

优化方法：

- 列表接口返回页面需要的状态摘要。
- 详情接口返回业务对象、明细、状态、可执行动作和关键历史。
- 操作接口返回操作结果和下一状态。
- 报表接口返回指标值、口径、维度和钻取参数。
- 枚举、权限、可执行动作由后端按业务状态输出，前端负责展示和交互。

## 16. 数据模型与 UI 模型的联合设计

数据模型和 UI 模型应该互相校验。

从数据模型到 UI：

- 实体决定页面对象。
- 状态机决定可执行动作。
- 流水决定历史记录。
- 事件决定进度反馈。
- 报表事实决定分析界面。

从 UI 到数据模型：

- 页面任务反推业务能力。
- 用户筛选反推报表维度。
- 详情字段反推对象属性和快照需求。
- 操作按钮反推状态转换。
- 异常处理反推补偿机制。

如果一个重要页面找不到清晰的数据实体，说明数据模型可能缺少业务对象。  
如果一个重要数据对象没有任何用户任务使用，说明它可能只是技术中间结构，不应该暴露为核心 UI。

## 17. 模型优化的工作流程

推荐在设计或重构业务模块时使用以下流程：

```text
1. 收集业务资料、老系统、竞品、报表和异常案例
2. 提炼业务词汇：名词、动词、规则、事件
3. 识别核心对象、生命周期、关系和边界
4. 设计初版数据模型和 UI 模型
5. 用未来场景和异常场景压力测试
6. 用报表指标反推事实和维度是否完整
7. 用前端任务路径反推接口和状态是否合理
8. 对比多个系统的优缺点
9. 记录被拒绝方案和拒绝原因
10. 输出目标模型、迁移策略、验证用例和知识卡片
```

模型优化必须保留证据链：

- 哪些业务事实支持这个模型。
- 哪些旧系统或竞品方案被参考。
- 哪些方案被拒绝。
- 哪些扩展场景已经压测。
- 哪些报表可以自然支持。
- 哪些性能风险需要后续验证。

## 18. 设计决策必须回答的问题

企业级系统设计不能只写“应该这样做”，必须回答“为什么这样做、解决什么问题、引入什么问题、如何治理”。否则设计文档无法指导研发、评审、测试、运维和后续演进。

每一个关键设计都应使用以下决策卡表达：

```text
设计项：
决策类型：单向门 / 双向门（不可逆 / 可逆）
撤销成本：如果决策错了，改回来要付出什么
为什么这样设计：
要解决的问题：
不这样设计会怎样：
带来的新问题：
大型企业标准做法：
适用边界：
不适用场景：
验证方式：
回滚或演进方式：
参考系统：
```

决策分级规则（审慎资源集中在单向门上）：

- **单向门**（错了改不回或改动代价巨大）：核心实体边界、状态机结构、流水模型、指标口径、对外接口契约、元数据 Schema。必须写完整决策卡并评审，落锤前用最小实验（原型/影子表）验证关键假设。
- **双向门**（错了低成本改回）：页面布局、报表样式、规则参数、字段展示。团队自决、快速试错，事后可查即可，不占用评审资源。
- 能推迟的单向门决策尽量推迟：用双向门方案先顶住，等信息足够时再落锤——推迟期间积累的信息就是决策质量。

示例：为什么要引入流水表。

```text
设计项：账户当前值与业务流水分离。

为什么这样设计：
当前值只能说明“现在是多少”，不能说明“为什么变成这样”。金额、库存、积分、储值、券等业务需要支持追溯、冲正、退款、审计和报表。

要解决的问题：
- 主表直接改余额导致历史不可追溯。
- 退款、反结账、冲正时无法解释变化来源。
- 财务、库存、会员权益报表无法还原过程。
- 系统间同步失败后缺少补偿依据。

不这样设计会怎样：
- 后期只能从日志或订单反推变化。
- 报表口径会越来越复杂。
- 一旦发生争议，无法提供字段级证据。
- 数据修复风险高，容易直接改坏当前值。

带来的新问题：
- 表数量增加。
- 写入链路变长。
- 需要处理幂等、冲正和流水与账户一致性。
- 查询当前值时不能每次全量汇总流水，需要账户表或快照。

大型企业标准做法：
- 账户表保存当前余额或当前状态。
- 流水表保存每次变化。
- 业务单据表保存来源。
- 冲正不改历史流水，而是写反向流水。
- 关键流水有幂等键、业务来源、操作人、业务时间和入账时间。
- 财务和库存类系统通常会有日结、月结、快照或事实表。

验证方式：
- 当前值能由流水解释。
- 退款、反结账、冲正后流水可追溯。
- 报表指标能从明细或事实表自然聚合。
- 重复请求不会产生重复流水。
```

示例：为什么要做交易模型和报表模型分离。

```text
设计项：交易表与报表事实表分离。

为什么这样设计：
交易模型服务在线业务写入，报表模型服务统计分析。两者关注点不同，强行共用会导致交易表越来越臃肿，报表 SQL 越来越复杂。

要解决的问题：
- 业务表承担过多统计责任。
- 报表跨多表拼接，性能差。
- 历史维度变化后报表口径不稳定。
- 新增报表会反向污染交易模型。

带来的新问题：
- 需要同步、ETL 或异步构建事实表。
- 实时性和一致性需要定义。
- 报表口径需要版本管理。

大型企业标准做法：
- 交易库负责写入和当前业务查询。
- 明细和流水提供可追溯事实。
- 数仓、宽表、事实表、指标表负责报表分析。
- 核心指标有统一口径和数据血缘。
- 对高频经营报表做预聚合或快照。
```

示例：为什么 UI 要围绕任务而不是围绕数据库表。

```text
设计项：UI 按用户任务和业务对象组织，而不是一张表一个页面。

为什么这样设计：
用户关心的是完成任务，不关心数据库表。数据库表是技术结构，直接映射成页面会导致操作路径割裂、信息不完整、用户需要自己拼业务上下文。

要解决的问题：
- 页面数量多但任务不闭环。
- 用户在多个页面之间来回复制信息。
- 状态、可执行动作和异常原因不清晰。
- 前端暴露技术字段，业务人员难以理解。

带来的新问题：
- 后端接口需要按页面任务聚合数据。
- 前后端需要共同定义 DTO、VO、可执行动作和状态摘要。
- 页面设计需要更多业务分析，而不是简单 CRUD。

大型企业标准做法：
- 列表页支持发现、筛选和批量处理。
- 详情页聚合对象、状态、明细、历史和可执行动作。
- 工作台聚合待办、异常和关键指标。
- 报表页围绕指标、维度、趋势和钻取组织。
- 后端返回当前状态下允许的动作，前端只展示合法动作。
```

## 19. 大型企业的标准解决方法

大型企业的软件体系通常不会依赖单一数据库模型解决所有问题，而是将业务能力、数据治理、流程管理、集成能力和分析能力分层处理。

### 19.1 领域模型与主数据管理

标准做法：

- 建立核心领域模型，例如客户、供应商、商品、组织、门店、仓库、账户、订单。
- 对核心对象建立主数据管理机制。
- 明确对象编码、唯一性、归属组织、生命周期和变更历史。
- 将主数据与交易数据分离。
- 对跨系统共享的数据建立标准编码和同步规则。

解决的问题：

- 多系统客户、商品、组织口径不一致。
- 报表无法按统一维度汇总。
- 接口集成时字段含义不同。

带来的问题：

- 主数据治理成本高。
- 需要审批、同步、去重、合并和数据质量检查。
- 初期比单系统自建字典更复杂。

### 19.2 交易、流水、事件、快照分层

标准做法：

- 交易单据记录业务发生。
- 明细行记录可计算颗粒。
- 流水记录金额、库存、权益等变化。
- 事件记录系统间传播和异步处理。
- 快照记录某个时点的统计结果。

解决的问题：

- 当前状态与历史过程混在一起。
- 业务回滚、冲正、审计困难。
- 报表和对账缺少稳定事实来源。

带来的问题：

- 数据链路更长。
- 一致性、幂等、补偿、重算需要明确机制。
- 需要更强的测试和监控。

### 19.3 流程引擎与状态机治理

标准做法：

- 对审批、采购、销售、结算、退款等流程建立显式状态机。
- 状态变化必须由动作触发。
- 每次动作记录操作者、时间、原因、来源和前后状态。
- 复杂审批使用流程引擎或规则引擎。
- 前端根据状态展示合法动作。

解决的问题：

- 状态字段随意变化。
- 逆向操作不可控。
- 用户看不到为什么不能操作。
- 流程逻辑散落在多个服务中。

带来的问题：

- 流程配置和代码逻辑需要边界清晰。
- 过度流程化会降低简单业务效率。
- 流程版本升级会涉及历史实例兼容。

### 19.4 规则引擎与配置版本化

标准做法：

- 价格、促销、积分、审批、权限、分润等规则配置化。
- 规则必须版本化。
- 历史单据保存当时命中的规则版本或计算快照。
- 规则发布需要灰度、回滚和审计。

解决的问题：

- 新业务规则都要改代码。
- 历史单据无法按当时规则还原。
- 多租户、多门店、多渠道差异难以维护。

带来的问题：

- 规则配置错误会造成大范围业务错误。
- 需要规则测试、发布流程和权限控制。
- 规则过度通用会变得难懂。

### 19.5 数据仓库、指标体系与数据血缘

标准做法：

- 在线业务库不直接承担复杂经营分析。
- 建立 ODS、DWD、DWS、ADS 或等价分层。
- 建立指标口径、维度字典和数据血缘。
- 高频报表使用宽表、事实表、汇总表或指标表。
- 财务、库存、销售等核心报表支持对账和追溯到原始单据。

解决的问题：

- 报表慢。
- 指标口径不统一。
- 同一个销售额在不同页面结果不同。
- 新增统计需求影响在线业务库。

带来的问题：

- 数据链路复杂。
- 实时性与准确性需要取舍。
- 需要数据治理和质量监控。

### 19.6 API 契约、集成平台与事件总线

标准做法：

- 对外接口版本化。
- 内部服务通过清晰 API、RPC、消息或事件集成。
- 重要事件有唯一业务键和幂等控制。
- 跨系统同步有重试、补偿、死信和监控。
- 接口契约、字段含义、枚举 code 有统一文档。

解决的问题：

- 系统间直接读库或互相耦合。
- 字段含义不一致。
- 同步失败无法恢复。
- 老客户端和新服务不兼容。

带来的问题：

- 集成治理成本增加。
- 事件最终一致会带来延迟。
- 需要监控、告警和补偿工具。

## 20. 行业巨头产品分析方法：以金蝶、用友、SAP、Oracle 为例

分析金蝶、用友、SAP、Oracle 等行业巨头产品的目标，不是复制其数据库或界面，而是提炼它们对业务对象、流程、权限、报表和扩展点的建模方法。

必须区分三类模型：

| 模型 | 含义 | 获取方式 | 用途 |
|---|---|---|---|
| 物理数据库模型 | 真实表、字段、索引、约束 | 仅限授权环境、公开资料、合法导出 | 了解实现细节，不能直接照搬 |
| 逻辑业务模型 | 业务对象、关系、生命周期、规则 | 产品文档、页面、接口、报表、演示环境、业务流程 | 提炼可复用设计 |
| UI 任务模型 | 页面结构、任务路径、状态反馈、操作方式 | 演示系统、公开截图、培训资料、试用环境、用户手册 | 改进交互和信息架构 |

### 20.1 合法获取信息的渠道

可以使用：

- 官方产品文档。
- 官方帮助中心。
- 官方 API 文档。
- 官方培训材料。
- 公开演示系统。
- 授权试用环境。
- 企业已购买并授权访问的部署环境。
- 系统自带的元数据导出、表单配置导出、报表配置导出。
- 合法的截图、操作录像、用户手册。
- 公开招投标文档、实施方案、产品白皮书。

不应使用：

- 绕过授权获取数据库。
- 反编译或破解商业软件。
- 未经授权复制客户生产库。
- 直接搬运竞品私有表结构、代码、页面资源。
- 将竞品专有字段、命名、流程照搬成自己的实现。

如果拿不到物理数据库模型，也可以通过页面、接口、报表和业务流程反推出“逻辑业务模型”。新系统真正需要的是逻辑模型和设计思想，不是竞品原始表结构。

### 20.2 数据模型分析步骤

分析一个行业巨头产品的数据模型时，按以下顺序进行：

1. 选定业务域，例如采购、销售、库存、财务、会员、报表。
2. 收集该业务域的页面、单据、字段、流程、报表和接口。
3. 识别核心业务对象，例如客户、物料、仓库、单据、科目、组织。
4. 识别对象关系，例如订单与明细、库存与批次、客户与组织。
5. 识别生命周期，例如草稿、提交、审核、关闭、作废、红冲。
6. 识别业务动作，例如新增、审核、下推、反审、结算、冲销。
7. 识别流水或凭证，例如库存流水、应收应付、财务凭证。
8. 识别报表维度和指标，例如组织、期间、物料、客户、金额、数量。
9. 识别扩展机制，例如自定义字段、单据类型、插件、审批流、规则配置。
10. 输出逻辑模型和设计决策，不直接输出照搬表结构。

分析输出应包含：

```text
业务对象：
对象关系：
生命周期：
关键动作：
单据与明细：
流水与凭证：
报表事实与维度：
扩展点：
权限与组织边界：
优点：
代价：
适用边界：
可借鉴设计：
不应照搬部分：
```

### 20.3 UI 模型分析步骤

分析金蝶、用友等产品 UI 时，重点看它如何帮助企业用户完成复杂任务。

分析步骤：

1. 记录用户角色，例如财务、采购、仓管、销售、老板、系统管理员。
2. 记录入口结构，例如工作台、菜单、单据、报表、基础资料。
3. 分析高频任务路径，例如新增采购单、审核订单、查询库存、生成报表。
4. 分析列表页字段、筛选项、批量操作和状态展示。
5. 分析详情页如何组织主信息、明细、状态、附件、审批、日志。
6. 分析操作按钮如何随状态变化。
7. 分析异常提示和校验方式。
8. 分析报表页的指标、维度、钻取和导出。
9. 分析可配置能力，例如自定义字段、列表列配置、打印模板、审批流程。
10. 提炼 UI 任务模型，而不是复制页面样式。

分析输出应包含：

```text
用户角色：
核心任务：
页面结构：
信息分组：
状态反馈：
可执行动作：
异常处理：
报表分析路径：
配置能力：
值得借鉴：
不适合照搬：
```

### 20.4 从行业巨头方案转化为自己的设计

转化时必须做抽象，不做复制。

推荐流程：

```text
竞品现象
  ↓
它解决了什么企业问题
  ↓
它采用了什么业务抽象
  ↓
这个抽象的代价是什么
  ↓
是否适合我们的目标客户和系统规模
  ↓
转化为自己的领域模型、数据模型和 UI 任务模型
```

示例：

```text
竞品现象：大型 ERP 通常有“单据类型”和“单据状态”。

解决的问题：
同一类业务存在多个业务变体，例如普通采购、退货采购、委外采购，不可能每种都完全新建一套流程。

业务抽象：
单据类型决定字段、流程、规则和后续动作；单据状态表达生命周期。

带来的代价：
单据类型配置复杂，过度配置会导致实施难度上升。

自己的转化：
对核心业务引入单据类型，但只开放必要配置；状态机代码化或半配置化；历史单据保存单据类型和规则版本。
```

每次分析金蝶、用友、SAP、Oracle 或其他系统后，应沉淀知识卡片。

```yaml

```

### 20.6 分析结果如何为我所用

行业巨头产品的价值在于提供高质量参照：

- 帮助识别企业系统的通用业务对象。
- 帮助发现遗漏的状态、逆向流程和异常路径。
- 帮助设计主数据、单据、明细、流水、凭证、报表之间的关系。
- 帮助设计适合企业用户的任务型 UI。
- 帮助判断哪些功能应该配置化，哪些应该代码化。
- 帮助建立大型企业标准做法的知识库。

最终产物不是“金蝶表结构复刻版”或“用友页面复刻版”，而是：

```text
经过抽象、取舍、验证后的自有领域模型
+ 自有数据模型
+ 自有 UI 任务模型
+ 自有接口契约
+ 自有报表模型
+ 自有扩展机制
```

## 21. 如何通过软件界面提取数据模型和 UI 模型

很多时候无法直接获得竞品或行业系统的数据库结构，但可以通过软件界面、操作流程、报表、导入导出、接口响应和用户手册反推出它的逻辑数据模型和 UI 任务模型。

> 本节是 20.3 的字段级深化。所有信息来源必须符合 20.1 的合法渠道边界：官方文档、公开演示、授权试用、合法截图；不绕授权、不反编译、不照搬私有实现。

这种方法的目标不是还原对方真实数据库，而是提炼：

- 它把业务拆成了哪些对象。
- 对象之间是什么关系。
- 每个对象有哪些生命周期。
- 哪些字段是主数据，哪些字段是交易数据。
- 哪些维度支撑报表。
- 哪些操作代表状态变化。
- 哪些 UI 结构值得借鉴，哪些只是历史包袱。

### 21.1 从菜单结构提取业务能力地图

菜单通常反映系统对业务能力的划分。

分析方法：

1. 记录一级菜单、二级菜单和三级菜单。
2. 判断每个菜单是主数据、交易单据、流程处理、报表分析、系统配置还是运维工具。
3. 合并同类能力，形成业务能力地图。
4. 标出菜单之间的上下游关系。

示例：

```text
基础资料：客户、供应商、商品、仓库、组织
交易业务：采购订单、销售订单、入库单、出库单
流程处理：审核、反审、结算、作废
报表分析：库存报表、销售报表、应收应付报表
系统配置：单据类型、审批流、权限、打印模板
```

评价问题：

- 菜单是按用户任务划分，还是按数据库表堆叠？
- 高频任务是否容易进入？
- 报表和业务操作是否有清晰边界？
- 配置入口是否能解释业务扩展方式？

### 21.2 从列表页提取对象集合模型

列表页通常暴露系统认为重要的业务对象。

需要记录：

- 列表名称。
- 主键或业务编号。
- 列字段。
- 筛选条件。
- 排序字段。
- 批量操作。
- 状态列。
- 操作按钮。

从列表页可以推断：

- 这个页面对应哪个业务对象。
- 哪些字段是对象识别字段。
- 哪些字段是报表维度。
- 哪些字段是状态或生命周期。
- 哪些动作是对象支持的业务行为。

示例：

```text
销售订单列表
- 识别字段：订单号、客户、门店
- 时间维度：下单时间、审核时间
- 状态字段：订单状态、支付状态、发货状态
- 金额字段：应收金额、实收金额、优惠金额
- 操作按钮：审核、反审、发货、关闭、作废
```

评价问题：

- 列表字段是否服务用户决策？
- 筛选项是否对应稳定业务维度？
- 状态是否表达完整生命周期？
- 操作按钮是否和状态机一致？
- 是否把太多技术字段暴露给用户？

### 21.3 从详情页提取实体关系模型

详情页通常能反推出主表、明细表、关联对象和历史记录。

需要观察：

- 页面标题和主对象。
- 基本信息区。
- 明细表格。
- 附件、备注、标签。
- 审批记录。
- 操作记录。
- 关联单据。
- 支付、物流、库存、权益、发票等子区域。

推断方法：

```text
基本信息区 -> 主对象属性
明细表格 -> 一对多明细关系
关联单据 -> 上下游业务关系
审批记录 -> 状态变更或流程实例
操作日志 -> 审计模型
支付/库存/权益区域 -> 交易衍生事实或流水
```

评价问题：

- 主对象和明细关系是否清晰？
- 详情页是否能解释当前状态为什么形成？
- 是否能看到历史动作和责任人？
- 是否能从详情追溯到上下游单据？
- 是否把报表、审计、交易结果混在一起，导致理解困难？

### 21.4 从新增和编辑表单提取字段模型

表单能反推出字段必填性、默认值、枚举、引用关系和业务规则。

需要记录：

- 字段名称。
- 字段类型。
- 是否必填。
- 默认值。
- 枚举选项。
- 字段联动。
- 只读条件。
- 校验规则。
- 可选择的数据来源。

推断方法：

```text
下拉框 -> 枚举或引用主数据
搜索选择 -> 外键或业务引用
必填字段 -> 创建对象的最小业务事实
只读字段 -> 系统计算或状态控制
联动字段 -> 业务规则或约束
隐藏/显示字段 -> 类型驱动的差异模型
```

评价问题：

- 字段是否属于当前对象，还是被错误塞进表单？
- 字段联动是否暗示缺少独立业务对象？
- 枚举是否稳定，还是应该配置化？
- 必填项是否真的符合业务最小事实？
- 同一字段在不同页面含义是否一致？

### 21.5 从操作按钮提取状态机和业务动作

按钮是 UI 中最直接的业务动作入口。

需要记录：

- 哪些状态下出现哪些按钮。
- 点击按钮后是否弹窗确认。
- 是否需要填写原因。
- 是否触发审批。
- 是否产生新单据。
- 是否改变金额、库存、权益、状态。
- 成功后页面状态如何变化。

推断方法：

```text
按钮可见性 -> 状态机合法动作
确认弹窗 -> 高风险动作
原因输入 -> 审计或冲正需求
动作后生成单据 -> 上下游单据关系
动作后状态变化 -> 生命周期转换
```

评价问题：

- UI 是否只展示当前状态下合法动作？
- 危险动作是否说明影响范围？
- 逆向动作是否有原因和记录？
- 状态变化是否有可追溯证据？
- 是否存在“按钮可点但后端不允许”的前后端模型不一致？

### 21.6 从报表页提取分析模型

报表页是反推数据模型质量的重要入口。

需要记录：

- 报表名称。
- 指标。
- 维度。
- 筛选条件。
- 分组方式。
- 钻取路径。
- 导出字段。
- 口径说明。
- 数据刷新频率。

推断方法：

```text
指标 -> 事实表或明细可聚合字段
维度 -> 主数据或交易发生时固化字段
筛选项 -> 常用分析维度
钻取 -> 汇总到明细的数据血缘
口径说明 -> 模型边界或历史兼容问题
```

评价问题：

- 指标是否能追溯到原始单据？
- 同一指标在不同报表是否口径一致？
- 维度是否在交易发生时固化？
- 报表是否依赖复杂临时拼接？
- 是否有实时、日结、月结等不同统计层级？

### 21.7 从导入导出和打印模板提取字段边界

导入导出和打印模板通常暴露企业系统真正关心的数据边界。

需要分析：

- 导入模板字段。
- 必填列。
- 错误提示。
- 导出字段。
- 打印字段。
- 单据头和单据体。
- 汇总字段和小计字段。

可推断：

- 哪些字段是外部交换契约。
- 哪些字段是业务人员真正使用的字段。
- 哪些字段必须保持兼容。
- 哪些字段是计算结果。
- 哪些字段是报表或财务口径。

评价问题：

- 导入字段是否和页面字段一致？
- 导出字段是否能解释业务结果？
- 打印模板是否暴露了缺失的业务概念？
- 字段命名是否清楚表达业务语义？

### 21.8 从错误提示和校验提取业务规则

错误提示和校验规则能反推出系统的业务约束。

需要记录：

- 必填校验。
- 唯一性校验。
- 权限校验。
- 状态限制。
- 金额、库存、时间范围校验。
- 跨对象约束。
- 异常处理建议。

推断方法：

```text
不能删除 -> 存在下游引用或历史约束
不能反审 -> 状态机或后续单据已生成
库存不足 -> 库存可用量模型
金额不平 -> 财务平衡约束
无权限 -> 角色、组织、数据范围模型
```

评价问题：

- 错误提示是否解释了原因和下一步？
- 规则是在前端硬编码，还是后端业务规则？
- 校验是否有字段级定位？
- 是否能从规则反推出缺失的数据模型？

### 21.9 软件界面分析输出模板

### 21.10 通过界面提取模型后的评价方法

评价时不能只看“界面好不好看”，而要同时评价数据模型和 UI 模型。

数据模型评价：

- 是否有清晰核心对象。
- 是否有稳定对象关系。
- 是否有生命周期和状态机。
- 是否有明细、流水、事件和审计。
- 是否天然支持报表维度。
- 是否支持逆向操作和异常补偿。
- 是否支持扩展类型和规则版本。

UI 模型评价：

- 是否围绕用户任务组织。
- 高频任务是否路径短。
- 状态和可执行动作是否清晰。
- 详情页是否能解释业务结果。
- 报表页是否能回答经营问题。
- 错误和异常是否可操作。
- 前端是否暴露了过多技术结构。

综合评价：

```text
如果 UI 任务清楚，但数据追溯弱：
说明产品体验较好，但底层模型可能会在报表、审计、退款、冲正时暴露问题。

如果数据模型完整，但 UI 路径复杂：
说明底层能力强，但用户完成任务成本高，需要优化信息架构和任务流。

如果报表强，但交易操作复杂：
说明分析模型可能完善，但在线业务模型或操作流不够友好。

如果配置能力很强，但普通业务难理解：
说明系统偏实施平台化，适合大型企业，但中小企业使用成本高。
```

评价结果的收口（接回评分与监控闭环）：

- 提取出的逻辑数据模型，直接套用第 8 节评分表逐维度打分，与自有系统同表对比。
- 借鉴其设计并落地后，低分风险项按 8.1 配置适应度函数持续监控。
- 每次界面分析的产出按 20.5 沉淀为标杆分析知识卡片，标注 confidence 和证据来源。

### 21.11 改进并转化为自己的模型

从别人的软件界面提取模型后，要经过二次设计，不能直接照搬。

转化步骤：

1. 识别它解决的真实业务问题。
2. 判断这个问题是否也是我们的目标问题。
3. 分析它为什么这样设计。
4. 识别它带来的复杂度和使用成本。
5. 判断我们的客户规模、业务复杂度、团队能力是否承受得起。
6. 保留核心业务抽象，重做命名、边界、页面任务和扩展方式。
7. 用我们的报表、异常场景、性能目标做压力测试。
8. 输出自有数据模型、UI 模型、接口模型和测试用例。

示例：

```text
观察：某 ERP 的销售订单详情页包含基本信息、明细、收款、发货、审批、操作日志、关联单据。

它解决的问题：
让用户在一个页面理解订单当前状态、历史过程和上下游影响。

带来的复杂度：
详情页信息密度高，后端接口需要聚合多个领域数据。

为我所用：
保留“订单详情聚合视图”的思想，但按我们的用户任务重组信息：
- 首屏只展示订单状态、金额、客户、关键动作。
- 明细、支付、发货、权益、日志分区展示。
- 后端提供订单详情聚合接口和可执行动作列表。
- 底层仍保持订单、支付、库存、权益、日志分表，不为了页面方便合成一张大表。
```

### 21.12 顶级架构师和设计师的判断标准

优秀的提炼不是“看到了什么就抄什么”，而是能回答：

- 这个界面背后的业务对象是什么？
- 它为什么把信息这样分组？
- 它隐藏了哪些底层复杂度？
- 它用什么方式表达状态和动作？
- 它是否牺牲了扩展性换取简单体验？
- 它是否牺牲了使用体验换取强配置能力？
- 它是否适合我们的客户、团队和系统阶段？
- 如果我们吸收它，应该保留思想、重做实现，还是完全拒绝？

最终目标：

```text
从软件界面中提炼业务模型
+ 从业务模型中设计数据模型
+ 从用户任务中设计 UI 模型
+ 从报表目标中校验事实模型
+ 从异常场景中优化状态机和流水
+ 从竞品优缺点中形成自己的标准方案
```

## 22. 新老系统切换与数据迁移

新系统死亡率最高的环节不是设计，而是切换。迁移方案必须与新模型**同时评审**——它本身就是单向门决策，不能等上线前临时补。

### 22.1 迁移设计原则

- 新老并行是默认，一刀切是例外。
- 每一步可回滚，且回滚路径要先于切换路径演练。
- 对账是迁移的验收标准，不是可选项。
- 迁移期间老系统是事实基准，差异默认判老系统赢，除非有证据证明是老数据错误。

### 22.2 标准切换路径

```text
1. 存量迁移：历史数据按新模型清洗导入；不可映射的数据显式进差异表并标注原因，禁止静默丢弃。
2. 双写期：老系统为主，新系统影子写入；每日自动对账（单据数、金额合计、状态分布、流水可解释性）。
3. 灰度切读：按门店/租户/业务类型逐步把查询切到新系统，写仍在老系统。
4. 灰度切写：同维度逐步切写，老系统降级为只读兜底；每个灰度批次设停留期和回退开关。
5. 静默期：老系统保留只读 N 个月供争议追溯，期满归档下线。
```

### 22.3 对账清单

- 主数据：数量一致、编码映射完整、去重合并有记录。
- 单据：总数、金额合计、按状态分布逐项相等。
- 流水：新老两边分别满足"当前值 = 期初 + 流水累计"。
- 报表：核心经营指标新老口径偏差小于阈值，且每笔偏差可解释（精度差异/口径差异/历史脏数据）。
- 所有差异进差异表并标注原因，**禁止不留痕的"手工调平"**。

### 22.4 兼容边界

- 老接口保留版本化适配层，并设定明确下线日期，不做永久兼容。
- 老单据在新系统按"当时规则版本"展示，不用新规则重算历史。
- 迁移脚本、映射表、差异表和对账结果本身沉淀为迁移方案卡片，供下一个业务域复用。

## 23. 组织与决策治理

康威定律：系统结构最终映射组织沟通结构。模型边界不清、口径不一致的技术问题，根因常常是责任边界不清的组织问题。

- **每个业务域指定模型 Owner**：负责该域实体边界、状态机、指标口径的最终裁决。没有 Owner 的模型，"加字段"就是最快路径，模型必然腐化。
- **决策分级执行**（对应第 18 节）：单向门决策必须评审 + 决策卡；双向门决策团队自决，事后可查。评审会只审单向门，避免评审疲劳。
- **平台与业务的边界即团队边界**：稳定结构（对象/单据/状态机/口径）归平台团队维护，业务差异归扩展代码；边界争议按"多业务复用的进平台，单业务专用的留扩展"裁决。
- **知识库治理按第 6 节执行**：卡片有 owner、复核周期、状态流转；不考核卡片数量。

## 24. 建议的下一步

建议先选择一个业务域做试点，例如“会员 / CRM”。

试点输出物：

- 旧系统会员模型分析。
- 第三方或竞品会员模型对比。
- 会员业务能力模型。
- 会员数据模型评分表。
- 会员权益、等级、积分、券、储值的推荐模型。
- 会员报表模型。
- 设计模式和反模式知识卡片。
- 新会员系统的数据库、接口、前端页面和测试设计草案。
- 会员数据迁移与对账方案（含双写、灰度、回滚路径）。
- 单向门决策卡集与适应度函数清单。

当一个业务域跑通后，再复制方法到订单、库存、支付和报表模块。
