# 界面模型模式的深层逻辑与模式体系

## 0. 前言

这些界面模式不是为了"好看"或"现代"，而是为了解决企业信息系统（B2B/ERP/CRM）中的核心 UX 问题：**让用户在复杂业务中找得到、看得懂、做得对、能恢复、可协作**。

本文档从认识论和方法论层面，剖析模式背后的深层逻辑、适用边界和组合关系。它讨论的是上位模型；具体页面模式和交互细节仍以本目录 `00-12` 文档为准。25 个典型模式的案例拆解见 [`13-界面模型模式详解.md`](13-界面模型模式详解.md)，静态高保真原型见 [`13-界面模型模式详解.html`](13-界面模型模式详解.html)。

## 1. 核心张力：系统模型 vs 用户任务

### 1.1 两套语言的失配

企业系统的一个基础问题在于**两套语言的不匹配**：

```
数据库视角（按范式分解）：
  order_items  (order_id, sku_id, qty, price, discount...)
  payments     (order_id, channel, amount, status...)
  inventory    (sku_id, warehouse_id, qty, locked_qty...)
  → 一个业务事实散在多个表

用户视角（心理模型）：
  "我有一笔订单，1280元，待发货，顾客问什么时候到"
  → 用户脑子里是一个完整的业务对象，不是表
```

对象聚合解决了其中一类重要问题，但不能解释全部界面模式。长任务、多人冲突、权限治理、命令面板和异常处置等模式，还在解决任务推进、协作控制、风险约束和反馈恢复问题。

因此，更完整的定义是：

> 企业界面模型是面向特定角色和任务，对业务现实所做的**可操作投影和控制面**。它既翻译数据，也支持判断、行动、反馈、协作与恢复。

### 1.2 映射的四个维度

| 翻译方向 | 对应模式 | 解决什么问题 |
|---|---|---|
| 数据库 → 业务对象 | Object Page、Related Lists | 信息聚合，把多表拼成"一个对象" |
| 操作 → 业务动作 | 表单族（Simple/Drawer/Modal/Wizard） | 把字段输入翻译成"创建/修改一个业务事实" |
| 状态 → 可见反馈 | Status Badge、Timeline、状态流转 | 把状态变化翻译成用户能追踪的事件链 |
| 汇总 → 可解释指标 | Dashboard、Drill-down、Analytical List | 把数字翻译成"发生了什么" |

### 1.3 任务-行动-反馈闭环

用户不是为了浏览数据而进入系统，而是为了推进一个目标。一个完整界面至少要支撑以下闭环：

```text
目标
  → 感知当前事实
  → 理解原因和约束
  → 预测下一步影响
  → 选择并执行动作
  → 观察执行结果
  → 成功后继续，失败后恢复
```

这个闭环对应 Norman 行动循环中的执行鸿沟与评估鸿沟，也对应企业系统里的两类常见失败：

- **执行鸿沟**：用户知道目标，却找不到合法动作，或不知道动作会影响什么。
- **评估鸿沟**：系统已经处理，用户却不知道是否开始、是否成功、当前数据是否最新。

因此，页面设计不能只问“展示哪些字段”，还要问：

1. 用户此刻要完成什么任务？
2. 用户需要哪些证据才能判断？
3. 当前允许做什么，为什么？
4. 系统如何表达处理中、成功、失败、部分成功和可恢复路径？

## 2. 信息的分层决策

### 2.1 Hick-Hyman 定律与信息选择

Hick-Hyman 定律说明：在需要从若干选项中作选择时，选项数量和概率结构会影响决策时间。它能解释菜单和操作收敛，但不能单独推出“信息越少越好”。熟练度、任务频率、可扫描性、比较需求和风险同样影响企业页面的合理密度。

企业页面应在“证据够用”和“认知过载”之间取得平衡。常用做法是建立**三层信息优先级**：

```
第一层（当前任务通常应优先看到）：
  - 对象身份：谁/什么
  - 主状态：现在什么情况
  - 核心数值：金额、数量、期限
  - 主操作：用户最可能要干什么

第二层（需要理解背景）：
  - 状态来源：为什么是这个状态
  - 关联对象：跟谁有关
  - 历史变化：怎么变成这样的

第三层（排查问题时才看）：
  - 完整字段
  - 技术细节
  - 审计日志
```

### 2.2 模式的信息分层差异

不同模式的核心差异，就在这三层信息的组合方式：

| 模式 | 第一层 | 第二层 | 第三层 |
|---|---|---|---|
| List Report | 表格行摘要 | 筛选条件 | 高级筛选/导出 |
| Object Page | 头部摘要 | 分区详情 | Timeline / 日志 |
| Dashboard | KPI 卡片 | 图表 | 明细钻取 |
| Wizard | 当前步骤 | 已填步骤 | 总览确认 |
| Exception Center | 异常摘要 | 影响对象 | 处理日志 |

### 2.3 信息分层的物理实现

信息分层通过**空间布局**和**渐进展开**来实现：

- **空间分层**：头部 > 内容区 > 底部 > 抽屉/弹窗，越靠上信息越重要
- **渐进展开**：默认折叠次要信息，点击或滚动展开完整内容
- **上下文保留**：用户查看详情后返回，列表的筛选和排序状态不变

### 2.4 任务-表征认知适配

认知适配理论强调：信息形式要匹配当前任务，而不是追求一种“最佳可视化”。同一份数据在不同任务中需要不同表征：

| 用户任务 | 优先表征 | 不适合单独使用 |
|---|---|---|
| 精确查值、逐项对账 | 表格、明细、差异列 | 只有趋势图 |
| 判断趋势、拐点和异常波动 | 折线、区间带、基线 | 只有汇总数字 |
| 比较方案和权衡 | Decision Matrix、并排对比 | 分散在多个页面的卡片 |
| 理解先后与持续时间 | Timeline、Gantt、阶段视图 | 无时间轴的事件清单 |
| 理解空间、网络或上下游关系 | 地图、关系图、依赖图，并保留列表替代 | 只用坐标或 ID 表达关系 |

表征选择还要考虑用户当前采用的认知模式。Rasmussen 的 SRK 模型描述的是**同一用户在不同情境中的行为模式**，不是三种固定用户类型：

| 认知模式 | 典型场景 | 界面支持 |
|---|---|---|
| **技能型（Skill-based）** | 熟悉、重复、高频操作 | 键盘、批量、固定位置、最少打断、快速反馈 |
| **规则型（Rule-based）** | 有明确规则和标准处置路径 | 检查清单、阈值、原因解释、标准动作、升级条件 |
| **知识型（Knowledge-based）** | 新异常、规则失效、首次决策 | 约束与不变量、原始证据、假设对比、模拟、决策记录 |

普通 CRUD 不需要为了 SRK 引入复杂工作台。认知工作分析和生态界面设计主要适用于实时、高风险、异常密集的控制台；其价值是把系统约束、资源余量和手段-目的关系显露出来，使用户在预案之外仍能判断，而不是用更多图形装饰页面。

## 3. 状态与能力：模式的核心组织轴

### 3.1 企业系统的常见组织倾向

企业系统通常比消费系统更强调长生命周期对象、角色分工、规则约束和可追溯性，因此更常按“状态下的任务”组织页面：

```
B2B / ERP 的常见组织方式：
  页面 = 状态过滤器 + 状态解释 + 状态操作
  "待发货订单"（状态过滤）
  "为什么待发货"（状态解释）
  "发货"（状态操作）
  → 按状态组织，用户找状态下的任务
```

这是一种倾向，不是 B2B 与 B2C 的绝对分界。消费系统同样有支付、履约和售后状态，企业系统也会按角色、任务、对象或功能组织。状态中心设计主要用于生命周期长、规则多、需要连续处理或追责的对象。

忽略状态会产生一类典型反模式：

- 只显示状态码，不解释来源、时间和下一节点
- 所有状态展示同一组按钮，点击后才由后端拒绝
- 只显示“成功/失败”，不区分排队中、部分成功和可重试
- 状态变化后不更新责任人、SLA 或关联对象

### 3.2 状态的四问框架

好的状态设计应帮助用户回答四个问题：

| 问题 | 含义 | 界面表达 |
|---|---|---|
| **是什么** | 当前状态是什么 | 状态徽章、主操作按钮 |
| **为什么** | 为什么是这个状态 | 状态来源说明、时间线前置节点 |
| **接下来会怎样** | 正在等待什么、何时超时、可能进入什么状态 | 进度、预计完成时间、SLA、后续节点 |
| **怎么办** | 用户能做什么 | 主操作、受控的次操作 |

以订单详情为例：

```
状态 = "待发货"

是什么？→ 状态徽章显示"待发货"
为什么？→ 时间线展示"已支付(10:32) → 库存已锁定(10:33)"
接下来会怎样？→ 等待仓库接单，2 小时后升级为超时异常
怎么办？→ 头部显示"发货"按钮
```

### 3.3 能力模型：当前到底能做什么

状态是动作可用性的必要输入，但不是唯一输入。企业系统中的可执行动作通常由多项约束共同决定：

```
availableActions = f(
  业务状态,
  角色与权限,
  数据范围,
  业务不变量,
  锁与并发版本,
  时间窗,
  策略与风险
)

nextState = T(currentState, action, externalEvents)
```

例如，“待发货”订单不代表任何人都能发货：还要满足当前用户具有仓库权限、订单属于其数据范围、库存锁有效、订单未被其他人处理且未超过时间窗。模式中的操作分层，本质上是对当前能力集合按任务频率和风险排序。

### 3.4 多维状态与时间真相

一个状态徽章往往只表达业务状态，无法代表系统的完整事实。复杂页面至少要区分以下维度：

| 状态维度 | 回答的问题 | 示例 |
|---|---|---|
| **业务状态** | 对象处于生命周期哪一步 | 待审核、已支付、已发货 |
| **执行状态** | 当前动作是否真正完成 | 排队中、处理中、部分成功、失败 |
| **数据状态** | 页面中的事实是否新鲜可信 | 截至 10:32、同步延迟、来源系统不可用 |
| **协作状态** | 谁正在负责、是否被占用 | 待认领、张三处理中、等待财务 |
| **编辑状态** | 当前版本是否可以安全提交 | 草稿、未保存、冲突、已被他人更新 |
| **异常状态** | 哪个约束被破坏、如何恢复 | 库存不足、对账差异、回调超时 |

界面应明确展示关键状态的来源、时间和版本，不能把“最后一次已知事实”伪装成“当前事实”。草稿、长任务和冲突解决的具体模式见 `08-高级企业交互模型.md`。

### 3.5 韧性状态与控制权交接

正常流程只能说明系统在预期条件下怎么工作。韧性工程进一步追问：依赖故障、数据延迟、自动化失效或容量不足时，关键任务如何继续或安全停止。界面需要支持四种能力：

```text
预见：哪些依赖或资源即将触及边界？
监测：关键能力、数据和外部依赖当前是否可信？
响应：继续、降级、排队、切换人工还是安全停机？
学习：事后如何沉淀触发条件、处置结果和新阈值？
```

可用统一状态描述：

```text
resilienceState = (
  criticalCapability,
  dependencyHealth,
  dataConfidence,
  degradedMode,
  recoveryPoint,
  controlOwner
)
```

| 模式 | 解决的问题 | 关键表达 |
|---|---|---|
| **Degraded Mode Banner** | 系统还能做什么、不能做什么 | 降级原因、受限能力、数据时间、预计恢复 |
| **Dependency Health Panel** | 哪个依赖导致当前限制 | 依赖状态、最近成功时间、影响范围 |
| **Recovery Point** | 中断后从哪里继续 | 最后已确认步骤、未提交动作、重放边界 |
| **Control Handoff** | 自动化和人工谁在控制 | 当前控制者、接管原因、接管窗口、回退和安全状态 |

降级不能伪装成正常成功；自动化失去关键输入时，也不能把旧结果继续显示为当前结论。离线队列、长任务和重试的具体模式见 `08-高级企业交互模型.md` 与 `12-更多经典模式速查表.md`。

## 4. 操作的风险分层

### 4.1 企业操作不是平等的

B2B 系统里，一个操作可能影响金额、库存、权限、合同，风险差异显著。把所有操作平等对待会产生一类高代价反模式。

```
高风险操作（不可逆 / 影响金额或库存）：
  取消订单、退款、删除数据、批量导入、修改价格
  → 硬约束 + 影响预览 + 可逆/补偿路径 + 必要的门禁与审计

中风险操作（影响状态流转）：
  发货、审核通过、驳回、启用/停用
  → 主操作突出 + 影响说明 + 状态和结果及时更新

低风险操作（局部影响）：
  打印、复制、导出、备注
  → 次操作收敛到更多菜单
```

### 4.2 风险自适应的防护层级

高风险设计的目标不是增加弹窗，而是降低错误发生概率和错误后果。按优先级选择与风险匹配的控制：

```
1. 消除：不向不需要该能力的角色提供入口
2. 硬约束：用状态机、不变量、权限和额度阻止非法结果
3. 安全默认：默认选中低风险范围，不预选破坏性选项
4. 影响预览 / 模拟：提交前展示影响对象、金额和不可逆后果
5. 可逆与补偿：优先提供撤销、软删除、反向流水或恢复版本
6. 职责分离：大额、高权限操作采用审批或二次认证
7. 确认：仅在用户能理解且确认能实质降低风险时使用
8. 反馈与审计：展示执行结果；有追责或合规要求时记录前后值
```

危险样式、明确后果、取消入口、结果反馈、影响预览、确认和审计都是可选控制，不要求每个动作机械具备全部要素。频繁、无差别的确认会造成确认疲劳；低风险且可逆的动作通常优先使用 Undo，高风险且不可逆的动作才需要更强门禁。

### 4.3 不可用操作的处理

当用户有权知道该操作、但因状态或业务规则暂不可用时，应说明原因和恢复路径：

```
❌ 反模式：按钮置灰，无任何解释
✅ 正确做法：按钮置灰 + tooltip "库存不足，无法发货，请先补充库存"
```

如果操作本身对该角色不可见，隐藏入口通常比暴露权限信息更合适。原因说明不能只依赖 tooltip，还应支持键盘和触屏用户；权限与数据范围的具体模式见 `09-企业治理与权限配置模型.md`。

## 5. 时间线：可追溯性的界面表达

### 5.1 企业系统需要回答"怎么变成这样的"

财务对账、问题排查、责任追溯，都依赖同一个能力：**追踪业务事实的形成过程**。

```
订单金额 1280 → 1350
  谁改的？→ 张三（运营经理）
  什么时间？→ 07-12 10:32:15
  为什么改？→ 备注：顾客使用了一张追加优惠券
  怎么改的？→ 手动编辑实付金额
  有依据吗？→ 附件：顾客聊天记录截图
```

### 5.2 Timeline 的三层事件

| 事件类型 | 触发来源 | 记录内容 | 用户 |
|---|---|---|---|
| **系统自动事件** | 支付回调、定时任务、同步 | 事件名、时间、结果 | 需要看到 |
| **人工操作事件** | 客服审核、运营修改 | 操作人、时间、前后值、原因 | 高风险或需交接时重点看到 |
| **业务状态事件** | 状态机流转 | 前状态 → 后状态、触发条件 | 核心关注 |

### 5.3 状态流转的完整表达

一个完整的状态流转记录应该包含：

```text
时间        事件类型    操作人     前状态     后状态     原因
07-12 10:32  系统      支付回调    -        已支付     微信支付成功，¥1280
07-12 10:33  系统      库存锁定    -        库存已锁定  SKU001 库存 -2
07-12 10:35  人工      客服张三    已支付    待发货     -
07-12 11:00  人工      客服张三    -        -         添加备注：顾客要求下午配送
07-12 14:00  系统      物流推送    待发货    已发货     快递单号 SF123456789
```

## 6. 表单：业务承诺的界面

### 6.1 表单不是字段录入容器

填表单不是"录入数据"，而是**创建或修改一个业务事实**。这个事实会被后续流程使用、被财务对账、被审计追溯。

```
订单退款表单  →  退款金额 128 元
  影响：顾客余额 +128、财务应收 -128、库存 +1、提成重新计算
  后续：可能被审计、可能被顾客投诉、影响门店 KPI
```

### 6.2 表单应回答的七个问题

| 问题 | 含义 | 设计要点 |
|---|---|---|
| 正在创建/修改什么？ | 表单的业务意图 | 标题明确表达业务动作："保存分类"而非"确定" |
| 哪些字段必填？ | 业务约束而非技术约束 | 必填有业务理由，不是"系统要求必填" |
| 字段之间有什么依赖？ | 互斥、联动、计算 | 互斥字段按当前模式动态显示/禁用 |
| 当前输入影响什么？ | 业务影响范围 | 影响广或高风险的配置变更展示影响预览 |
| 错误如何发现和修正？ | 校验及时性 | 字段级校验 + 提交前预检，错误贴近字段 |
| 提交后发生什么？ | 下一步引导 | 成功/失败都有明确反馈和下一步入口 |
| 能中断后继续吗？ | 长流程保护 | 任务耗时长、可被打断或输入难以重建时支持草稿 |

### 6.3 表单复杂度与承载方式

字段数量只能作为早期信号，不能单独决定表单形式。还要考虑字段依赖、决策点数量、错误代价、可逆性、操作频率、跨步骤比较需求和中断概率。

| 场景特征 | 推荐形式 | 原因 |
|---|---|---|
| 少量、低风险、无需跨字段比较 | Modal Form | 轻量完成，不离开上下文 |
| 中等复杂度、需要参考当前页面 | Drawer Form / Simple Form | 保留主任务上下文 |
| 字段多但需要整体比较 | Sectioned Form + 锚点 | 分区但不割裂全局关系 |
| 耗时长、易中断、输入重建成本高 | Sectioned Form + 草稿 | 支持离开和恢复 |
| 多步骤有顺序依赖 | Wizard | 每步一个任务，跨步骤依赖可见 |
| 步骤间需要频繁来回比较 | 优先单页分区 | 避免 Wizard 隐藏前后信息 |

### 6.4 配置表单的特殊性

配置页和普通编辑页不同，它改变的是**系统规则**，而非单个业务对象：

```
普通编辑：修改一条记录，影响一个对象
配置变更：修改一条规则，影响一批对象

→ 影响广或高风险时提供预览：展示本次变更影响多少对象
→ 存在时效要求时表达生效时间：支持立即生效 / 定时生效
→ 可逆变更提供直接回滚；不可逆变更提供补偿、版本迁移或恢复方案
→ 高风险、合规或需追责的变更进入审计：记录操作人、时间和前后值
```

## 7. 钻取链路：指标的可解释性

### 7.1 数字不解释就是噪音

Dashboard 上一个"销售额 ¥28,450"，用户看到后问："这个数字对吗？为什么比昨天少？哪类商品拖累了？哪个门店没达标？"

如果页面无法回答这些问题，指标就只是装饰。

### 7.2 钻取链路的四层结构

```
第一层：汇总指标（What）
  销售额 ¥28,450  ·  同比 +12.3%

第二层：维度拆解（Which）
  按门店：人民广场 ¥12,000 / 浦东 ¥9,500 / 徐汇 ¥6,950

第三层：明细数据（Who / What）
  人民广场店 → 订单明细表

第四层：业务对象（Composition / Evidence）
  具体订单 → 订单详情 → 状态时间线
```

钻取通常能回答“这个数字由什么构成、证据在哪里”，但不能自动证明“为什么发生”。因果解释还需要对照基线、业务机制、实验或其他独立证据；不要把相关变化直接写成原因。

### 7.3 Drill-down 的五个设计要求

1. **每层保留筛选上下文**：从门店下钻到订单时，门店筛选条件不变
2. **支持返回上一级**：面包屑或返回按钮，不丢上下文
3. **汇总和下级能对账**：点击明细合计，能回到汇总数字
4. **最后一层落到业务证据**：优先支持进入业务对象；没有单一对象时提供可核对的明细或血缘
5. **导出区分汇总和明细**：用户导出时清楚当前导出的是哪层数据

### 7.4 从报表到决策工作台

报表回答“发生了什么”，决策工作台还要帮助用户在约束下选择方案。有限理性意味着用户不可能穷尽所有信息；界面应把注意力放在硬约束、关键差异和最有价值的补充证据上。

| 决策场景 | 推荐模式 | 核心内容 |
|---|---|---|
| 熟悉、时压、线索稳定 | **Cue Panel + Action Review** | 关键线索、典型模式、拟采取动作、快速心智模拟 |
| 陌生、高影响或参与者分歧大 | **Decision Workbench** | 硬约束、备选项、准则、证据、不确定性、理由 |
| 少数假设决定结论 | **Scenario Compare / Sensitivity Analysis** | 假设范围、临界点、结论何时翻转 |
| 需要持续调查 | **Evidence Workspace** | 事实、假设、反证、未知项、来源和分析上下文 |
| 需要复盘和治理 | **Decision Log** | 当时信息、选择理由、反对意见、责任人和复盘时间 |

Recognition-Primed Decision（RPD）说明，专家在熟悉且时压的情境中常根据线索识别典型模式，再通过心智模拟检验一个可行动方案，而不是穷举并打分所有选项。但 RPD 不适用于陌生、分歧大、不可逆或证据薄弱的决策，此时应切换到结构化比较。

决策界面还应避免结果偏差：**好的决策过程可能遇到坏结果，坏的决策过程也可能偶然成功**。复盘应评价当时可得证据、假设和约束，而不是只按事后结果倒推过程质量。权重、评分和单点概率都是模型输入，不应伪装成客观精确值。

## 8. 异常闭环：从日志到业务处理

### 8.1 技术异常 vs 业务异常

这是企业系统最容易被忽视的反模式：

```
❌ 反模式：异常只在日志里
  系统：支付回调失败 → 写入 error.log
  技术人员：看到日志，排查原因
  业务人员：完全不知道有这个异常

✅ 正确做法：异常进入 Exception Center
  系统：支付回调失败 → 写入异常中心
  业务人员：看到"3 笔支付异常待处理" → 点击处理 → 重新发起支付
  技术人员：只处理业务人员无法解决的技术问题
```

### 8.2 异常中心的五要素

| 要素 | 内容 |
|---|---|
| **分类分级** | 支付失败 / 库存不足 / 同步超时 / 数据对账差异，分严重程度 |
| **绑定影响范围** | 能定位具体对象时关联订单 / 会员 / 商品；系统性异常关联服务、任务和受影响范围 |
| **提供处理动作** | 不只是展示错误码，要给出"重新发起支付 / 调整库存 / 忽略" |
| **处理过程留痕** | 谁处理了、什么时候、怎么处理、结果如何 |
| **重复异常聚合** | 同一接口大量失败，聚合为一条异常 + 数量统计 |

### 8.3 异常判别与告警疲劳

异常检测不是“提示越多越安全”。阈值设计必须同时考虑：

| 判断 | 业务代价 | 设计要求 |
|---|---|---|
| **真正异常并告警** | 正常发现问题 | 给出证据、影响范围、责任人和处理动作 |
| **正常却告警（误报）** | 打断工作、消耗信任 | 支持聚合、抑制、调阈值和反馈误报 |
| **异常却未告警（漏报）** | 风险继续扩大 | 对高损失场景降低阈值或增加独立校验 |
| **正常且不告警** | 无需处理 | 不占用注意力 |

低基准率事件即使检测器准确，也可能产生大量误报。异常中心应显示规则或模型依据、阈值、数据时间窗，以及经校准的置信信息（若可提供），并记录“处理、忽略、误报、升级”的结果，用于后续校准。AI 异常解释还应遵循 `11-AI辅助与智能化界面模型.md` 的证据和人工确认要求。

## 9. 协作：业务事实由多人共同完成

### 9.1 从单用户界面到社会技术系统

订单、审批、对账和异常处理通常跨角色、班次与系统完成。知识并不只在某个人脑中，而是分布在界面、记录、规则、人员和外部系统之间。界面的责任是维护可共享的任务状态，而不是假设一个用户从头做到尾。

协作界面至少要回答：

```text
谁负责？
正在做什么？
等待谁或什么？
截止时间是什么？
已经尝试过什么？
下一位接手者需要哪些上下文？
发生冲突时以哪个版本为准？
```

### 9.2 协作状态的最小表达

| 维度 | 界面表达 |
|---|---|
| 责任 | 待认领、负责人、协作者、升级对象 |
| 进展 | 未开始、处理中、等待外部、已阻塞、已完成 |
| 时限 | SLA、截止时间、剩余时间、超时升级规则 |
| 交接 | 处理摘要、未决项、最近证据、下一步建议 |
| 并发 | 当前编辑者、版本差异、锁、合并或放弃路径 |
| 通知 | 谁需要知道、为什么通知、是否已读或确认 |

评论、提及、指派、关注和共享视图等具体模式见 `12-更多经典模式速查表.md`；草稿和冲突解决见 `08-高级企业交互模型.md`。在总论层面，判断标准是：换人、换班或跨系统后，任务能否在不依赖口头记忆的情况下继续。

### 9.3 共同基础与承诺闭环

协作不只是“发消息”和“改负责人”。共同基础（Common Ground）要求参与者共享足够的目标、事实、计划、约束和完成定义；承诺闭环则要求系统区分请求、接受、执行、交付和验收：

```text
请求方提出任务与完成条件
  → 执行方接受 / 协商 / 拒绝
  → 执行方推进并暴露阻塞
  → 执行方声明已交付
  → 请求方验收 / 退回 / 关闭
```

“执行方点了完成”不等于“业务结果已被验收”。高风险审批、跨部门工单、对账差异和客户承诺尤其需要区分这两个状态。

```text
coordinationState = (
  sharedGoal,
  requester,
  owner,
  commitment,
  dependency,
  handoff,
  acknowledgement,
  acceptance,
  escalation
)
```

| 模式 | 核心用途 |
|---|---|
| **Shared Task Summary** | 固化目标、当前事实、约束、完成定义和未决项 |
| **Commitment / Acceptance** | 区分已认领、已承诺、已交付和已验收 |
| **Dependency & Blocker** | 说明等待对象、阻塞原因、责任方和预计解除时间 |
| **Handoff with Acknowledgement** | 接手方确认收到上下文和责任，不把“已发送”当“已交接” |
| **Escalation Path** | 超时或风险扩大时明确升级对象、依据和后续状态 |

### 9.4 中断恢复与前瞻记忆

草稿只恢复数据，不一定恢复任务。用户在中断后还需要知道“我为什么做到这里、刚才在判断什么、接下来要做什么”。对长流程、调查和复杂配置，应保存最小恢复包：

```text
任务目标
停止位置与停止原因
已确认事实和最近证据
未完成输入 / 未提交动作
未决问题和依赖
建议的下一动作
保存时间、版本和保存人
```

Resume Point、Investigation Workspace、Handoff Summary 等模式解决的是意图连续性，不应被简化为“重新打开同一个 URL”。

### 9.5 商业完整性与职责分离

高风险业务不仅要“谁有权限”，还要保证数据只能通过受控业务动作改变，并避免同一人完成相互制约的步骤。Clark-Wilson 商业完整性模型可转化为以下界面判据：

- 金额、库存、权限和主数据通过受控动作修改，不提供绕过业务规则的任意编辑入口。
- 创建与审批、制单与付款、申请与授权按风险需要分离，禁止自批。
- 批量变更、导入和紧急操作先校验、再执行，并保留可对账的结果。
- 审计记录不能由同一业务动作的执行者静默删除或改写。

| 模式 | 适用场景 | 边界 |
|---|---|---|
| **Maker-Checker** | 资金、权限、敏感主数据 | 仅在错误代价或合规要求足够高时启用 |
| **Controlled Transaction** | 退款、调库存、改权限 | 用业务动作和影响预览替代任意字段编辑 |
| **Effective Access View** | 权限继承复杂 | 展示当前实际权限、来源、范围和到期时间 |
| **Time-bound Delegation / Break-glass** | 临时代办、应急处置 | 限时、限范围、强理由、显著标识、事后复核 |
| **Reconciliation Panel** | 导入、结算、对账 | 展示应有、实有、差异、处置和最终平衡 |

紧急访问不能伪装成普通管理员权限，也不能成为永久后门。权限配置的具体模式见 `09-企业治理与权限配置模型.md`。

### 9.6 上下文完整性与最小披露

隐私不是简单“隐藏字段”。上下文完整性（Contextual Integrity）要求信息流符合具体场景中的角色、数据类型、接收者和传递原则：同一数据可用于履约，不代表可以用于营销、导出或跨组织共享。

界面在展示、搜索、导出、复制、通知和 AI 上下文注入时，都应回答：

```text
谁在什么业务角色下查看？
为了什么任务？
需要哪一层精度？
是否可以导出、复制或继续传播？
依据和有效期是什么？
```

最小披露的目标是提供完成任务所需的最少数据，而不是把所有可见字段都返回给页面。权限拒绝说明也应保持最小充分：告诉用户恢复路径，但不泄露其无权知道的对象、角色或策略细节。

## 10. 运行流与一致性：对象如何安全穿过系统

对象页和状态机解释“一个对象现在怎样”，但企业系统还必须解释“一批对象如何在有限资源、并行步骤和分布式依赖中完成”。这类问题不能只用页面布局或单对象状态描述，需要把控制、排队、工作流和事务一致性共同纳入界面模型。

### 10.1 控制回路：界面是业务控制系统的一部分

从控制论看，企业操作是一个闭环：目标和约束形成参考值，界面展示测量值，用户或自动化选择控制动作，系统执行后产生新状态，再通过反馈校正偏差。

```text
目标 / 策略 / 不变量
  → 比较当前值与目标值
  → 用户或自动化选择动作
  → 系统执行与外部扰动
  → 观测新状态、偏差和副作用
  → 继续、停止、回退或调整
```

界面因此不仅要显示结果，还要表达：**目标是什么、当前偏差多大、测量是否可信、动作是否生效、控制权在谁、何时必须停止或接管**。反馈过慢会导致用户重复操作，反馈失真会导致错误修正，自动化频繁切换则会形成“控制权振荡”。

| 模式 | 解决的问题 | 关键表达 |
|---|---|---|
| **Target vs Actual** | 当前离业务目标多远 | 目标、当前值、偏差、趋势、阈值 |
| **Control Loop Status** | 动作是否进入闭环并产生预期效果 | 指令、执行状态、测量时间、结果、残余偏差 |
| **Intervention Threshold** | 何时观察、干预或升级 | 阈值依据、持续时间、风险和责任人 |
| **Manual Override / Safe Stop** | 自动化失效时如何接管 | 控制者、接管原因、互斥状态、回退条件 |

### 10.2 排队与流动：Worklist 不是过滤后的列表

待办、工单、审批、异常和后台任务都构成排队系统。Little 定律给出稳定系统中的基本关系：

```text
在制品数量（WIP） = 有效平均流率（λ） × 平均流动时间（Flow Time）
```

在稳定系统的长期平均下，到达率、完成率和有效流率才会相等；系统持续积压时不能把它们混为一个数。这个关系提醒设计者：只把待办按优先级排序，不能解决积压；增加同时打开的任务还可能拉长完成时间。Worklist 应把“流”而不只是“记录”变得可见：

| 需要看见的事实 | 对应界面模式 |
|---|---|
| 当前积压、到达速度、完成速度和老化程度 | Queue Health、Aging Buckets、Throughput Trend |
| 哪个环节形成瓶颈 | Stage WIP、Bottleneck Highlight、Blocked Reason |
| 优先级是否导致低优先任务长期饥饿 | Priority Policy、Oldest-item Guardrail、Fairness Indicator |
| 领取过多是否造成任务停滞 | WIP Limit、Pull Next、Work-in-progress Warning |
| 批量处理是否出现部分失败 | Batch Progress、Per-item Result、Retry Failed Only |

优先级、SLA 和分派策略是业务策略，不是纯视觉排序。界面应解释排序依据、例外规则和超时升级，避免用户通过反复改优先级制造新的拥塞。

### 10.3 并行工作流：完成不等于所有前置条件已汇合

真实业务常包含并行分支、等待外部事件、取消、超时和补做。单线 Timeline 容易把并行过程误画成顺序过程。可用工作流网 / Petri Net 的基本语义检查以下问题：

```text
Case：一个端到端业务实例
Token：该实例在流程网中当前位置的标记；并行分叉后一个 Case 可以有多个 Token
Place：等待条件或中间状态
Transition：满足前置条件后可发生的业务动作
Fork / Join：并行拆分与汇合
```

由此派生的界面要求：

- 明确哪些步骤可以并行，哪些步骤必须全部完成后才能汇合。
- 区分“本分支完成”“主流程可继续”“整个实例完成”。
- 显示正在等待的事件、超时边界、取消传播和替代路径。
- 检测死锁、孤儿任务和永远无法满足的汇合条件，并提供恢复动作。

适合的模式包括 Parallel Stepper、Dependency Graph、Join Gate、Waiting-for-event、Deadlock / Orphan Warning。普通线性表单和简单审批不需要引入工作流网表达。

### 10.4 分布式事务：技术成功不等于业务完成

跨支付、库存、物流、CRM 和财务的动作通常无法一次原子提交。Saga 模型把长事务拆为多个本地事务，并为已完成步骤定义补偿；补偿不是数据库回滚，而是新的业务事实。

```text
业务动作
  → 步骤 A 成功
  → 步骤 B 成功
  → 步骤 C 失败
  → 补偿 B
  → 补偿 A
  → 最终完成 / 已补偿 / 需人工介入
```

用户界面至少要区分：

| 状态 | 含义 | 推荐表达 |
|---|---|---|
| **已接受** | 系统已接单，不代表完成 | Request Receipt、幂等键、可追踪任务号 |
| **执行中** | 多个步骤正在推进 | Step Progress、最近成功步骤、当前依赖 |
| **部分完成** | 一部分副作用已发生 | 已完成项、未完成项、业务影响 |
| **补偿中 / 已补偿** | 系统正在或已经做反向业务动作 | 补偿步骤、结果、无法还原的剩余影响 |
| **最终一致** | 各系统已收敛到可接受状态 | 对账结果、最终时间、来源系统 |
| **需人工介入** | 自动恢复无法继续 | 原始证据、安全动作、责任人、升级路径 |

重试按钮必须与幂等语义配套；“重试失败项”不能重复执行已经成功且不可重复的副作用。跨系统页面还应明确哪个系统是当前字段或状态的权威来源，避免把不同来源的快照拼成一个虚假的原子事实。

### 10.5 边界对象与语义互操作

同一个“客户”“订单”“完成”在销售、仓储、财务和客服中含义可能不同。边界对象（Boundary Object）理论说明，跨角色协作需要一个各方都能共同引用、又允许保留本地视角的稳定对象；它不是强迫所有部门使用完全相同的页面和术语。

界面应显式表达：

- 共享身份：跨系统稳定 ID、对象对应关系和来源。
- 共享核心：各方都承认的状态、承诺、时间和完成定义。
- 本地视图：角色专属字段、术语和操作，不伪装成全局真相。
- 转换规则：状态映射、字段换算、版本和无法映射的例外。
- 语义冲突：同名不同义、不同来源不一致时展示差异，而不是静默覆盖。

由此可派生 Cross-system Identity、Source Badge、Field Mapping、Semantic Diff、Shared Core + Role View 等模式。它们特别适用于主数据、跨系统同步和跨部门交接；单系统、单角色的小型 CRUD 不必增加这套复杂度。

### 10.6 分布式事件顺序：一条时间线不等于唯一真相

跨服务事件没有天然的全局顺序。支付回调、库存消息、人工操作和定时补偿的本地时间可能漂移，消息还可能延迟、重复或乱序。Timeline 不应仅按展示时间排序后暗示严格因果。

```text
eventIdentity = (
  eventId,
  correlationId,
  causationId,
  sourceSystem,
  sourceSequence,
  occurredAt,
  recordedAt
)
```

- `correlationId` 把同一业务链路的事件关联起来。
- `causationId` 表达“哪个事件直接触发了我”，区别于只是同一链路；它必须由执行链路显式记录，不能根据时间邻近自动推断。
- `sourceSequence` 只保证来源系统内的局部顺序；跨来源无法确定时应显示“并行 / 顺序未知”。
- `occurredAt` 与 `recordedAt` 分别表达发生时间和系统知道的时间。

对应模式包括 Correlated Timeline、Causal Link、Parallel Branch、Late-arriving Event Badge、Source Sequence。只有能证明因果或顺序时才画连接箭头；否则用分支、来源泳道或“不确定顺序”表达。

```
任务是什么？

├── 查找 / 筛选对象

│   ├── 多层级结构（组织/类目） → Tree / Tree Grid
│   └── 需要分析指标 + 明细联动 → Analytical List Page
│

│   └── 多对象横向关联 → Related Lists + 三栏布局
│
├── 操作对象
│   ├── 少量、低风险、局部修改 → Drawer Form / Inline Edit
│   ├── 需要整体查看或比较 → Sectioned Form

│   └── 异常处理闭环 → Exception Center
│
├── 协作推进任务
│   ├── 一批对象需要认领和处理 → Worklist + Assignment
│   ├── 多人围绕对象沟通 → Activity Feed + Comment / Mention
│   ├── 请求、交付与验收分离 → Commitment + Acceptance

│
├── 复杂判断 / 选择方案
│   ├── 熟悉且时压的异常处置 → Cue Panel + Mental Simulation + Action Review
│   ├── 多方案、多准则权衡 → Decision Matrix + Scenario Compare
│   ├── 结论取决于关键假设 → Sensitivity Analysis + What-if
│   ├── 需要提出因果主张 → Causal Evidence Card + Experiment / Cohort Compare
│   └── 需要复盘与追责 → Evidence Workspace + Decision Log
│
├── 管理运行流 / 跨系统执行
│   ├── 待办积压、吞吐或瓶颈 → Queue Health + Aging + WIP Limit
│   ├── 并行分支和汇合条件 → Dependency Graph + Join Gate
│   ├── 长事务部分成功或补偿 → Step Progress + Compensation + Reconciliation
│   ├── 跨系统同名对象或状态映射 → Source Badge + Semantic Diff + Role View
│   └── 跨服务事件乱序或因果追踪 → Correlated Timeline + Source Lane + Causal Link
│
├── 分配人工与自动化
│   ├── 只辅助判断、不代替决策 → Advisory
│   ├── 新规则或模型上线前验证 → Shadow Mode
│   ├── 自动准备、人工决定是否执行 → Approval Gate
│   └── 可监测、可补偿且边界明确 → Guardrailed Automation + Override
│
└── 配置规则
    ├── 简单开关 → Toggle + 影响说明
    ├── 复杂条件组合 → Rule Builder + Lint + Simulation + Version Diff
    ├── 数据可见范围 → Data Scope Config + 预览
    └── 权限分配矩阵 → Permission Matrix
```

## 12. 保守规则的底层原理

当缺少更多任务证据时，可以先按以下保守规则形成低成本原型，再用真实任务验证；这些规则不是不可推翻的定律：

| 保守规则 | 原理 | 错误做法 |
|---|---|---|
| 对象很多 → List Report | 用户需要先找到对象，再理解对象 | Dashboard 首页堆图表，用户找不到具体订单 |
| 对象复杂 → Object Page | 复杂对象需要分区解释，不能塞进弹窗 | Modal 展示全部字段，用户无法聚焦 |
| 连续处理 → Worklist | 减少页面跳转上下文切换 | 每次处理完跳转回列表再点下一个 |
| 配置有明确阶段依赖 → Wizard | 长流程拆步骤降低认知负荷 | 单页巨型表单，用户不知道填到哪了 |
| 指标需解释 → Drill-down | 数字不解释就是噪音 | Dashboard 只展示大数字，用户不知道涨跌原因 |
| 异常需闭环 → Exception Center | 技术日志业务人员无法处理 | 异常只在日志里，异常无法被业务人员发现 |
| 待办有积压 → Queue Health + WIP Limit | 流动时间取决于到达、吞吐和在制品，不只是排序 | 只改优先级，积压和瓶颈继续扩大 |
| 流程有并行汇合 → Dependency Graph + Join Gate | 分支完成不等于实例完成 | 用单线进度条伪装并行流程 |
| 跨系统长事务 → Step Progress + Compensation | 接受、局部成功、补偿和最终一致是不同事实 | 一个“成功”Toast 掩盖部分失败 |

## 13. 反模式的系统性分类

常见反模式至少可以从以下 **18 类失配**解释：

| 失配类型 | 表现 | 对应反模式 |
|---|---|---|
| **数据-心理失配** | 字段按数据库结构排列，不按业务理解 | "一张表一个页面" |
| **信息-认知失配** | 不分层、不收敛、全量展示 | "首页堆图表"、"表格按钮过多" |
| **状态-操作失配** | 动作和反馈没有反映对象的多维状态 | 所有状态显示同一组按钮、处理中仍可重复提交 |
| **系统-业务失配** | 技术异常在日志里，业务人员看不到 | "异常只在日志" |
| **当前-历史失配** | 只有当前值，没有形成过程 | "详情页没有历史" |
| **任务-页面失配** | 页面围绕功能或表结构，不能完成一个闭环任务 | 用户跨多个页面手工拼上下文 |
| **角色-能力失配** | 操作只按状态显示，忽略权限、范围和责任 | 按钮可见但后端拒绝、越权提示混乱 |
| **时间-真相失配** | 把旧快照、处理中或部分成功显示成当前事实 | 重复提交、误判同步结果 |
| **个人-协作失配** | 只记录对象，不记录责任、交接和冲突 | 换班后任务丢失、多人覆盖修改 |
| **任务-表征失配** | 信息形式不支持查值、比较、趋势或关系任务 | 用饼图做精确对账、用表格隐藏趋势 |
| **承诺-完成失配** | 把已发送、已交付或执行方完成当成业务验收 | 工单无人接收、交接无回执、结果未验收 |
| **正常-降级失配** | 依赖故障或数据过期时仍显示正常态 | 用户基于旧数据操作、自动化失控后无人接管 |
| **指标-目标失配** | 指标成为目标后只优化数字，不再代表原业务目的 | 拆单刷完成率、压低告警数量而扩大漏报 |
| **目标-反馈失配** | 控制目标、测量值和执行反馈不一致 | 用户反复操作、自动化在错误信号下持续修正 |
| **到达-处理失配** | 只管理优先级，不管理积压、在制品和瓶颈 | 队列越处理越长、低优先任务永久饥饿 |
| **分支-完成失配** | 把单个分支完成误认为主流程或实例完成 | 并行审批未汇合就进入下一状态 |
| **局部-全局失配** | 单系统步骤成功却被显示为端到端业务完成 | 支付成功但库存或财务未收敛，重试造成重复副作用 |
| **术语-语义失配** | 跨角色、跨系统使用同名字段却含义不同 | 状态静默映射、来源冲突被覆盖、交接双方理解不同 |

## 14. 模式的理论层次

### 14.1 基础理论与方法框架

| 理论 | 核心命题 | 能解释的模式问题 |
|---|---|---|
| **行动循环与执行/评估鸿沟** | 用户要把目标转成动作，再从反馈判断结果 | 动作可发现性、影响预览、处理中状态、结果与恢复 |
| **情境意识** | 决策依赖对当前事实的感知、理解和后续预测 | 状态四问、SLA、风险预警、下一节点 |
| **分布式认知与 CSCW** | 认知分布在人、界面、记录、规则和系统中 | 指派、交接、共享视图、活动流、冲突解决 |
| **人因错误理论** | 操作失误、认知错误和违规的成因与控制不同 | 硬约束、安全默认、Undo、确认、审批与审计 |
| **认知负荷与外部认知** | 界面应承担记忆、比较和线索保存，而非把负担留给用户 | 信息分层、保存视图、并排比较、上下文恢复 |
| **信息觅食理论** | 用户依据信息线索，在收益和查找成本间选择路径 | 搜索、分面筛选、信息气味、钻取和相关对象 |
| **信号检测理论** | 异常识别需要权衡误报、漏报、基准率和阈值 | 异常分级、告警聚合、阈值说明、反馈校准 |
| **测量理论与数据血缘** | 指标只有在概念、尺度、口径和来源明确时才构成可解释证据 | 指标定义、粒度、分母、时间窗、血缘、可加性与对账 |
| **认知适配与 SRK** | 表征应匹配任务；同一用户会在技能、规则和知识模式间切换 | 表格/图形选择、专家捷径、检查清单、异常调查工作台 |
| **认知工作分析 / 生态界面设计** | 高风险动态系统应显露工作域约束、不变量和手段-目的关系 | 资源余量、边界、依赖、故障降级和预案外处置 |
| **自然情境决策与意义建构** | 专家按线索识别并模拟方案；陌生问题通过证据与假设逐步成形 | Cue Panel、Scenario Compare、Evidence Workspace、Decision Log |
| **有限理性与信息价值** | 决策资源有限，应优先获取可能改变选择的证据 | 硬约束、关键差异、敏感性和待验证项 |
| **韧性工程** | 系统要能预见、监测、响应并从偏差中学习 | 降级模式、恢复点、依赖健康、人工接管、安全停机 |
| **共同基础与承诺闭环** | 协作需要共享目标、事实和完成定义，并区分请求、交付与验收 | Shared Task Summary、Commitment、Acknowledgement、Acceptance |
| **商业完整性与职责分离** | 关键数据只能通过受控事务改变，相互制约的职责按风险分离 | Maker-Checker、禁止自批、受控动作、对账和不可变审计 |
| **上下文完整性** | 信息流是否合适取决于角色、场景、数据类型、接收者和传递原则 | 最小披露、目的限制、导出控制、AI 上下文边界 |
| **双时态事实模型** | 业务有效时间与系统记录时间是两条不同时间轴 | As-of View、追溯更正、修订状态、版本差异 |
| **Goodhart / Campbell 效应** | 被用于奖惩和控制的指标会改变参与者行为并逐渐失真 | 护栏指标、反指标、分布、操纵检查和定期复核 |
| **因果推断与证据层级** | 构成、相关、干预和反事实是不同强度的主张，不能由钻取或同期变化互相替代 | Causal Evidence Card、Cohort Compare、Experiment Result、Counterfactual Scenario |
| **反馈控制与必要多样性** | 控制依赖及时可信的反馈，处置能力还要覆盖环境扰动的种类 | Target vs Actual、控制回路状态、阈值、人工接管和安全停机 |
| **排队论与 Little 定律** | 积压、吞吐和流动时间相互约束，增加在制品通常不会自动增加完成量 | Queue Health、Aging、WIP Limit、瓶颈和公平性 |
| **工作流网 / Petri Net** | 并行、汇合、等待和取消需要显式的令牌与前置条件语义 | Parallel Stepper、Join Gate、依赖图、死锁和孤儿任务检测 |
| **Saga 与最终一致性** | 跨系统长事务由本地步骤和业务补偿组成，技术接受不等于业务完成 | 分步进度、部分成功、补偿状态、幂等重试和对账 |
| **边界对象与语义互操作** | 跨群体协作需要稳定共享核心，同时允许角色化视图和显式转换 | 跨系统身份、来源标记、字段映射、语义差异和角色视图 |
| **认知维度框架** | 表征的可见性、一致性、黏滞性和渐进求值等性质决定修改与理解成本 | Rule Builder、低代码配置、查询构建器、即时预览和变更传播 |
| **自动化功能分配与自动化层级** | 应分别决定谁获取信息、分析、选择和执行，而不是笼统选择人工或自动 | 建议模式、批准后执行、自动执行后复核、控制权和回退路径 |
| **分布式事件与部分顺序** | 跨系统事件通常只有局部顺序，相关链路不自动等于因果链路 | Correlated Timeline、Causal Link、来源泳道、迟到事件和顺序未知 |
| **模式语言** | 模式是特定情境和张力下可组合、带后果的解法 | 模式选择、组合、替代和反模式边界 |

### 14.2 派生设计原则

| 原则 | 含义 | 在模式中的体现 |
|---|---|---|
| **Gestalt 完形原则** | 人脑倾向于把碎片聚合成整体 | Object Page 聚合对象，Related Lists 聚合关联，Timeline 聚合过程 |
| **可观察性** | 用户能看到当前事实、来源、时间和执行进度 | 状态徽章、数据时间、Progress、任务中心 |
| **可控性** | 用户知道当前能力、影响范围和退出路径 | 能力模型、影响预览、安全默认、取消入口 |
| **可恢复性** | 错误和中断后能回到可工作的状态 | Undo、草稿、重试、补偿、冲突解决 |
| **责任可归属** | 任务和变更能确定负责人、依据与交接过程 | Assignment、Timeline、Audit Log、Handoff |
| **可追溯性原则** | 关键业务事实应能解释来源和形成过程 | Timeline、Audit Log、版本历史、钻取链路 |
| **业务不变量可见** | 页面解释哪些约束不可被操作破坏 | 金额守恒、库存可用量、审批顺序、权限范围 |
| **任务-表征适配** | 信息形式匹配查值、比较、趋势、关系或决策任务 | 表格、趋势图、Decision Matrix、Timeline、关系视图 |
| **韧性与显式降级** | 正常能力不可用时，系统仍能安全继续或停止 | Degraded Mode、Recovery Point、Control Handoff |
| **承诺可验证** | 请求、承诺、交付、验收和交接回执是不同事实 | Commitment、Acceptance、Acknowledgement、Escalation |
| **最小披露** | 只在当前角色、任务和传递规则下提供必要数据 | 脱敏、字段裁剪、导出控制、AI 上下文边界 |
| **时间语义明确** | 当前值、历史有效值和当时系统所知不能混为一谈 | As-of View、Effective Timeline、Revision Diff |
| **信任校准** | 自动化和 AI 的确定性、证据和边界与真实能力匹配 | 来源、置信度、替代方案、人工接管、AI 参与审计 |
| **包容性与等价交互** | 语义、键盘、焦点、颜色和错误表达支持不同用户 | 语义控件、焦点管理、非颜色编码、错误摘要 |
| **流动可见与在制品受控** | 界面展示积压、老化、瓶颈并限制同时推进的任务 | Queue Health、Aging、WIP Limit、Pull Next |
| **部分完成不伪装成完成** | 多步骤和跨系统动作必须表达局部成功、汇合、补偿和最终收敛 | Join Gate、Partial Success、Compensation、Reconciliation |
| **共享核心、角色化视图** | 跨边界对象保留共同身份和语义，同时按角色表达本地信息 | Shared Core、Source Badge、Role View、Semantic Diff |

### 14.3 局部启发式及其边界

渐进披露、按钮收敛、字段数量阈值、危险色、二次确认和 Wizard 拆步，都是局部启发式，不是上位理论。使用前应回到任务频率、角色经验、风险、可逆性和比较需求：

- **新手、规则型用户、专家需要不同通道**：引导说明、稳定标准路径、键盘/批量捷径可以并存。
- **渐进披露不能隐藏决策所需证据**：需要横向比较的内容应同时可见。
- **颜色不能独立承载状态和风险**：还要有文本、图标或结构化语义。
- **确认不能替代约束和恢复**：可逆低风险动作优先 Undo，不可逆高风险动作才加强门禁。
- **可访问性不是事后补丁**：复杂表格、树、对话框、Tabs 和 Combobox 应从模式选择时就遵循语义与键盘契约。

#### 自动化功能分配：不是“人工或自动”二选一

自动化应分别决定信息获取、分析、方案生成、选择、执行和复核由谁负责。风险、可逆性和证据质量不同，适合的协作级别也不同：

| 模式 | 自动化做什么 | 人保留什么 | 适用边界 |
|---|---|---|---|
| **Advisory** | 汇总证据、给建议 | 判断与执行 | 高影响、证据仍需语境解释 |
| **Shadow Mode** | 在后台给出结果并与人工比较 | 全部控制权 | 新模型、新规则上线前验证 |
| **Approval Gate** | 准备完整动作和影响预览 | 批准、修改或拒绝 | 可审查但不宜自动提交的动作 |
| **Guardrailed Automation** | 在额度、范围和不变量内自动执行 | 配置护栏、观察、暂停 | 高频、可监测、可补偿任务 |
| **Post-action Review** | 自动执行并抽样或按风险送审 | 复核异常与追责 | 低单笔风险、高吞吐任务 |
| **Override / Appeal** | 提供机器决策和依据 | 接管、申诉、纠正 | 自动拒绝、排序、风控等影响用户权益的决策 |

界面必须明确当前自动化级别、控制者、护栏、最近一次人工接管和恢复条件，避免用户以为自己在批准，实际只是查看已经执行的结果。

#### 认知维度：复杂配置的修改成本

Rule Builder、查询构建器、低代码流程和权限表达式不仅是“字段很多”，还具有高黏滞性、隐藏依赖、一处多表述和延迟反馈等认知成本。配置界面应按主要困难补相应模式：

| 认知困难 | 模式 |
|---|---|
| 依赖不可见 | Dependency Map、引用查找、影响对象列表 |
| 规则冲突或不可达 | Rule Lint、冲突检测、不可达分支提示 |
| 修改后果要到发布后才知道 | Simulation、样本回放、即时预览 |
| 改动传播范围过大 | Change Impact、分阶段发布、作用域限制 |
| 新旧版本难比较 | Version Diff、语义差异、回滚点 |
| 用户被迫过早决定细节 | Draft、占位符、渐进求值、待补项清单 |

“可视化”本身不保证更易用；如果连线增加隐藏依赖和视觉搜索成本，表格、文本表达式或分层视图反而更合适。

### 14.4 指标真实性与因果边界

报表和看板不仅要可钻取，还要表达“这个数字是什么事实”。关键指标至少应明确：

```text
业务定义 / 分子 / 分母
统计粒度 / 时间窗 / 时区
数据来源 / 最后更新时间
是否可加、可拆、可对账
过滤范围 / 权限范围
修订或回补规则
```

钻取、血缘和明细对账支持构成解释；相关性、同期变化和 AI 摘要不自动构成因果解释。

#### 因果证据的层级

界面应把“构成、相关、干预和反事实”分开：

```text
构成证据：数字由哪些明细组成？
相关证据：哪些变量一起变化？
干预证据：改变 X 后，Y 是否发生可归因变化？
反事实判断：如果当时不做 X，Y 会怎样？
```

钻取和分群对比最多直接支持前两层；随机实验、自然实验或有明确识别假设的因果模型才能加强干预与反事实结论。由此可派生：

| 模式 | 应展示什么 |
|---|---|
| **Causal Evidence Card** | 主张、证据类型、识别假设、效应范围和不确定性 |
| **Cohort Compare** | 分组规则、基线差异、时间窗和混杂提醒 |
| **Experiment Result** | 分配方式、样本量、主要指标、护栏指标和停止规则 |
| **Counterfactual Scenario** | 模型、关键假设、适用人群和结论翻转条件 |

没有足够证据时，界面应写“相关”“分解”或“待验证假设”，不能用“原因”“贡献了”制造超出证据的确定性。

#### 双时态事实与追溯更正

对价格、组织归属、权限、合同、账务和规则等会追溯更正的事实，只显示“更新时间”不够。界面应区分：

- **业务有效时间（valid time）**：该事实在业务世界何时成立。
- **系统记录时间（recorded time）**：系统何时知道或记录该事实。

```text
displayedFact =
  value
  + validTime
  + recordedTime
  + source
  + transformation
  + quality
  + verificationStatus
  + version
```

由此派生：

| 模式 | 用途 |
|---|---|
| **As-of View** | 查看“截至当时系统所知”的对象和权限状态 |
| **Effective Timeline** | 查看事实在业务上何时生效、失效或被追溯更正 |
| **Revision Badge / Diff** | 区分原始值、修订值及修订原因 |
| **Data Quality & Reconciliation Panel** | 展示缺失、延迟、冲突、校验和差异处置状态 |

这不是要求所有 CRUD 采用双时态；只有历史还原、追溯更正、审计或对账需要区分两条时间轴时才使用。

#### 预测不确定性与指标反馈效应

预测、预算、风险评分和 AI 建议不是已发生事实。界面应根据实际可用证据表达：点估计、合理区间、情景范围、数据覆盖、模型版本和主要假设；不能用虚假的小数精度制造确定感。用户还应能看到“什么变化会让结论翻转”。

当 KPI 被用于排名、奖惩或资源分配后，界面本身会改变行为和未来数据：

```text
展示 / 目标 / 默认值
  → 用户行为与策略调整
  → 新数据
  → 下一轮指标、推荐和排序
```

这就是 Goodhart / Campbell 风险。关键 KPI 建议配套：

- **KPI Contract**：业务目的、口径、适用范围、负责人和修订规则。
- **护栏指标**：防止优化主指标时牺牲质量、安全或长期价值。
- **反指标与分布**：同时观察可被隐藏的副作用，不只展示平均值。
- **操纵性检查**：列出用户如何通过改变行为“赢得指标”而不改善业务。
- **定期复核**：指标成为目标后，重新验证它是否仍代表原目的。

### 14.5 模式语言的记录格式

本目录中的模式在深化时，建议统一记录：

| 字段 | 回答的问题 |
|---|---|
| **Context** | 在什么角色、任务和环境下出现？ |
| **Problem / Forces** | 哪些目标、约束和张力同时存在？ |
| **Solution** | 模式如何组织对象、状态、动作和反馈？ |
| **Consequences** | 得到了什么，又增加了什么成本？ |
| **Failure modes** | 在什么情况下会失效或被误用？ |
| **Related patterns** | 与哪些模式组合、替代或互斥？ |
| **Evidence** | 用什么任务指标、错误数据或测试证明有效？ |

这样可以避免把模式降格为组件清单，也避免为了套模式忽略真实任务。

## 15. 总结

### 模式的本质

界面模型是面向角色与任务，对业务现实的**可操作投影和控制面**。数据库语言到业务语言的翻译是其中一个维度，而不是全部。它通过以下介质工作：

- **结构**：把多表聚合成对象（Object Page、Related Lists）
- **时间**：把状态快照展开成过程（Timeline、Audit Log）
- **空间**：把全量字段分层到优先级（Sectioned Form、信息分层）
- **操作**：把数据录入翻译成业务承诺（表单族 + 风险控制）
- **指标**：把汇总数字展开成可解释链路（Drill-down、钻取）
- **能力**：把状态、权限、规则和风险收敛为当前可执行动作
- **协作**：把责任、交接、并发和通知沉淀为共享任务状态
- **反馈**：把同步与异步执行过程表达为可观察、可恢复的闭环
- **决策**：把约束、证据、假设和不确定性组织成可复盘的选择过程
- **治理**：把职责分离、时间语义、最小披露和指标反馈转成界面边界
- **流动**：把积压、吞吐、瓶颈和在制品转成可管理的工作队列
- **一致性**：把并行汇合、部分成功、补偿和跨系统收敛表达为真实业务状态
- **语义**：把跨角色、跨系统的共享身份、来源和转换规则显式化

### 设计好坏的判断标准

一个企业系统页面好不好，至少问以下问题：

```
1. 这个页面服务哪个角色的什么任务？
2. 用户能否感知当前事实、来源、时间和下一步？
3. 当前动作是否符合状态、权限、范围、不变量和风险约束？
4. 操作影响是否可预见，结果是否可观察，失败是否可恢复？
5. 系统降级或自动化失效时，受限能力、恢复点和控制权是否明确？
6. 换人、换班或中断后，目标、承诺、未决项和下一动作能否继续？
7. 高风险业务是否通过受控事务、职责分离、验收和对账保护完整性？
8. 指标能否对账，并区分有效时间、记录时间、修订和数据质量？
9. 预测和方案比较是否表达关键假设、不确定性、敏感性和反证？
10. KPI 成为目标后，是否有护栏、反指标和操纵性检查？
11. 数据展示、导出、复制和 AI 使用是否符合当前目的并最小披露？
12. 新手、专家、键盘和辅助技术用户是否都有等价路径？
13. AI 或自动化是否展示证据、边界并保留人工接管？
14. 待办和后台任务是否展示积压、老化、瓶颈，并控制在制品？
15. 并行或跨系统流程是否区分分支完成、部分成功、补偿和最终一致？
16. 跨角色和跨系统的同名对象、字段与状态是否具有明确来源和映射？
17. 跨服务时间线是否区分链路相关、直接因果、局部顺序和顺序未知？
18. 自动化级别是否与风险匹配，并明确审批、复核、接管和申诉路径？
```

这些问题共同成立时，页面才不仅“可用”，还具备企业系统需要的可解释、可控、可恢复和可协作能力。
