## 扩展模式族

25 个核心模式覆盖常见的“找对象、看对象、改对象、看整体、处理异常”任务，但不能替代以下模式族。选型时先确定任务，再叠加协作、运行流、治理、自动化和行业约束。

| 模式族 | 典型模式与模型 | 何时进入 | 专题入口 |
|---|---|---|---|
| **韧性与高级交互** | Conflict Resolution、Degraded Mode、Control Handoff、Interruption Recovery | 存在并发编辑、依赖失效、中断恢复或人工接管 | [`08-高级企业交互模型.md`](08-高级企业交互模型.md) |
| **治理与配置** | Effective Access、Maker-Checker、Data Scope、Rule Builder、Feature Flag | 权限、规则或配置会影响一批对象，且需要预览、复核、审计或回滚 | [`09-企业治理与权限配置模型.md`](09-企业治理与权限配置模型.md) |
| **行业作业** | Inventory Board、Scheduling Board、POS Console、Financial Voucher、Map Operations | 任务受行业不变量、设备、空间、班次或凭证规则约束 | [`10-行业业务专用UI模型.md`](10-行业业务专用UI模型.md) |
| **AI 与自动化** | Explainable Recommendation、Human-in-the-loop、Shadow Mode、Approval Gate、Override | 系统给出预测、建议或自动执行，必须表达依据、不确定性和控制权 | [`11-AI辅助与智能化界面模型.md`](11-AI辅助与智能化界面模型.md) |
| **协作与决策** | Assignment、Commitment、Handoff、Decision Matrix、Evidence Workspace、Causal Evidence | 事实由多人共同完成，或结论依赖方案比较、假设和证据 | [`13-界面模型深层逻辑与模式体系.md`](13-界面模型深层逻辑与模式体系.md) §9、§7.4、§14.4 |
| **运行流与一致性** | Queue Health、WIP Limit、Join Gate、Compensation、Reconciliation、Correlated Timeline | 存在积压、并行汇合、部分成功、补偿或跨系统乱序 | [`13-界面模型深层逻辑与模式体系.md`](13-界面模型深层逻辑与模式体系.md) §10 |
| **组合与局部模式** | Simulation、Version History、Retry Queue、Empty State、Bulk Action、Virtual List | 核心页面模式需要补足反馈、恢复、安全或大数据能力 | [`12-更多经典模式速查表.md`](12-更多经典模式速查表.md) |

这些模式不是互斥模板。一个真实页面通常是“核心页面模式 + 任务局部模式 + 风险/协作/恢复机制”的组合，例如：`Worklist + Assignment + Aging + Handoff`，或 `Rule Builder + Simulation + Version Diff + Maker-Checker`。

<a name="p01-list-report"></a>
## 1. List Report（列表报告）

### 解决的核心问题

**大量对象中快速定位和批量处理**。

这是企业系统最常用的模式。核心矛盾是：列表可能有数万行，用户需要在几秒内找到目标对象。

### 问题根源

```
场景：门店收银员查询今日订单
数据库里有：订单主表、支付表、会员表、商品明细表、物流表……
用户想要：找到"那笔1280元的订单，看看有没有发货"

如果按数据库结构展示 → 用户需要在4个tab之间来回切换
按List Report展示 → 一个表格行聚合了所有关键信息，一眼定位
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **筛选区** | 核心字段默认展开，高级筛选折叠 | Hick's Law：默认选项少，降低决策时间 |
| **行摘要** | 每行展示最能区分对象的字段（订单号、状态、金额、会员） | 一眼区分，不需要点进去 |
| **状态徽章** | 状态文本/图标为主，颜色为辅助；状态与优先级分开表达 | 便于扫描，也避免色觉差异和“状态=优先级”的误判 |
| **主操作外露** | 高频动作放在行末按钮 | 减少操作路径 |
| **次操作收敛** | 低频动作放"更多"菜单 | 避免按钮过多（见深层文档 §13“信息-认知失配”） |
| **批量选择** | Checkbox + 批量操作栏 | 减少重复点击 |

### 典型案例

```
场景：门店收银员查询和处理今日订单

┌─ 筛选区 ─────────────────────────────────────────────┐
│ [搜索订单号/会员名___] [门店 ▼] [状态 ▼] [日期范围 ▼] [查询] │
└──────────────────────────────────────────────────────┘

┌─ 批量操作栏 ──────────────────────────────────────────┐
│ 已选 2 项   [批量发货] [批量打印] [批量导出] [影响预览] │
└──────────────────────────────────────────────────────┘

┌─ 列表 ──────────────────────────────────────────────┐
│ ☑ 订单号        门店      会员    金额      状态        时间     操作 │
│ ────────────────────────────────────────────────────── │
│ ☑ SO202607120001 人民广场  周杰伦  ¥1,280  [已完成]   10:32  [详情][发货] │
│ ☑ SO202607120002 浦东店    林俊杰  ¥456    [待发货]   10:15  [详情][发货] │
│ ☐ SO202607120003 徐汇店    王力宏  ¥89     [退款中]   09:58  [详情][退款] │
└──────────────────────────────────────────────────────┘
```

### 理论桥接

- **§2 信息分层**：列表行是一级信息（一眼看到），详情页是二级信息（点击进入）
- **§3 状态轴**：筛选条件本质是状态过滤器——"待发货"是一个状态子集
- **§13 反模式分类**：“信息-认知失配”要求高频外露、低频收敛，避免表格按钮过多

<a name="p02-object-page"></a>
## 2. Object Page（对象详情页）

### 解决的核心问题

**把散落在多个数据库表中的数据，聚合成一个用户能理解的业务对象**。

### 问题根源

一个订单在数据库里涉及 8+ 张表：订单主表、商品明细表、支付表、物流表、优惠券表、发票表、操作日志表……

```
❌ 反模式：字段按数据库表结构平铺
  用户看到：支付信息（来自payments表）→ 物流信息（来自logistics表）→ 商品明细（来自order_items表）
  问题：用户不知道这些表之间的关系，不知道先看什么

✅ Object Page：按用户判断顺序重新组织
  用户看到：订单摘要 → 为什么待发货（因为已支付+已锁库）→ 怎么处理（发货/退款）
  优势：用户脑子里有一个完整的"订单"概念，不需要自己拼接
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **对象头部** | 名称/编号 + 主状态 + 关键指标 + 主操作 | 用户一眼知道这是哪个对象、什么状态、现在能干什么 |
| **状态解释** | 说明"为什么是这个状态" | 消除用户疑惑（"待发货？为什么？库存够吗？"） |
| **内容分区** | 按业务理解顺序排列，不按数据库表 | 字段有逻辑关联，不是碎片化的 |
| **关联列表** | 关联对象作为独立区块 | 相关对象不割裂，但也不喧宾夺主 |
| **时间线** | 记录状态变化和操作历史 | 解释"怎么变成这样的"（见§5） |

### 对象头部标准结构

```
┌─────────────────────────────────────────────────────────────────┐
│  订单 #SO202607120001                                           │
│  状态：[待发货]  已支付  已锁库                                 │
│  金额：¥ 1,280.00    门店：人民广场店    会员：周杰伦           │
│  操作：[发货] [退款] [打印]                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 分区设计示例（零售订单详情）

```
1. 订单摘要（核心信息）
2. 商品明细（用户关心买了什么）
3. 支付信息（金额来源）
4. 配送/自提信息（用户关心怎么收货）
5. 优惠和积分（如果有优惠要透明）
6. 状态流转（见Timeline模式）
7. 操作日志（见Audit Log模式）
```

### 理论桥接

- **§1 数据-心理失配**：Object Page 是把数据库语言翻译成用户心理模型的"翻译器"
- **§3.2 状态四问**：是什么、为什么、怎么办、如果失败或中断如何恢复
- **§5 可追溯性**：分区结构让用户理解对象形成过程

<a name="p03-timeline"></a>
## 3. Timeline（时间线）

### 解决的核心问题

**让用户理解"这个状态是怎么变成这样的"**——即业务事实的可追溯性。

### 问题根源

```
❌ 反模式：详情页只有当前状态
  "订单最终实付 ¥1,280，用户问为什么原支付是 ¥1,350"
  客服：不知道……系统里只显示最终金额

✅ Timeline：展示状态形成过程
  "07-12 10:32 系统 支付成功 ¥1,350
   07-12 11:00 人工 客服张三记录 原承诺优惠券未抵扣
   07-12 11:05 人工 主管李四批准售后优惠 ¥70
   07-12 11:06 系统 原路退款 ¥70，最终实付 ¥1,280"
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **三层事件区分** | 系统事件 / 人工操作 / 业务状态分别标注 | 系统自动的和人工操作的责任链不同 |
| **状态变化记录前后值** | 金额 ¥1,280 → ¥1,350 | 只显示最终值无法解释原因 |
| **操作人和时间** | 客服张三 / 07-12 11:00 | 责任追溯 |
| **关联单号和附件** | "附件：优惠券截图" | 有据可查 |

### Timeline 标准字段

```
时间 · 事件类型 · 操作人/系统 · 前状态 → 后状态 · 原因 · 关联单号 · 备注/附件
```

### 典型案例

```
┌─ 时间线 ──────────────────────────────────────────────────────┐
│                                                           ●    │
│  07-12 14:30  系统  订单完成                                     │
│                    状态：已发货 → 已完成                          │
│                                                           ●    │
│  07-12 11:06  系统  原路退款成功                [系统]              │
│                    退款 ¥70，最终实付 ¥1,280                    │
│                    关联退款单：RF202607120001                   │
│                                                           ●    │
│  07-12 11:05  人工  主管李四  批准售后优惠         [人工]            │
│                    应付金额：¥1,350 → ¥1,280；原因：承诺优惠未抵扣   │
│                                                           ●    │
│  07-12 11:00  人工  客服张三  提交售后优惠申请     [人工]            │
│                    备注：顾客反馈承诺优惠券未抵扣，待主管核实        │
│                                                           ●    │
│  07-12 10:33  系统  库存锁定                                   │
│                    SKU001 库存 -2                              │
│                                                           ●    │
│  07-12 10:32  系统  支付成功                   [系统]            │
│                    微信支付 ¥1,350                              │
└───────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§5 时间线：可追溯性的界面表达**：三层事件结构（系统/人工/业务）
- **§5.3 状态流转的完整表达**：时间+事件类型+操作人+前状态+后状态+原因
- **§13 反模式分类**：“当前-历史失配”要求用 Timeline 补足状态形成过程

<a name="p04-related-lists"></a>
## 4. Related Lists（关联列表）

### 解决的核心问题

**一个业务对象关联的其他对象，如何在不喧宾夺主的情况下完整呈现**。

### 问题根源

```
订单关联：会员（1个）、支付单（1个）、优惠券（多个）、物流（多个）、发票（0-1个）、售后单（0-N个）……

❌ 反模式：把所有关联都堆进详情页
  → 详情页变成另一个复杂列表页，用户不知道看哪里

❌ 反模式：关联数据完全割裂
  → 用户需要去菜单里找"支付单管理"，输入订单号筛选
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **关系说明** | 明确说明关系类型（"关联会员"、"关联支付单"） | 用户知道这些数据为什么出现在这里 |
| **关键字段** | 只展示最关键字段，完整查询跳独立页 | 信息密度控制，不喧宾夺主 |
| **有限操作** | 关联列表内只做查看、解绑等轻量操作 | 复杂操作跳转到独立页 |
| **大数据量处理** | 默认分页或只展示最近5条 | 避免关联列表本身变成列表页 |

### 典型案例

```
┌─ 关联列表：支付单 ──────────────────────────────────────────┐
│ 支付单号          支付方式     金额     状态        时间      │
│ PAY202607120001  微信支付    ¥1,350  已支付    07-12 10:32  │
│                                             [查看详情]       │
└─────────────────────────────────────────────────────────────┘

┌─ 关联列表：优惠记录 ────────────────────────────────────────┐
│ 优惠类型      优惠名称          优惠金额     时间            │
│ 店铺券        新客满100减20     -¥20.00    07-12 10:32     │
│ 积分抵扣      100积分抵 ¥1.00   -¥1.00     07-12 10:32     │
│ 追加优惠      追加优惠券抵 ¥70   -¥70.00    07-12 11:05     │
│                                         [查看全部优惠]       │
└─────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§1 信息聚合**：Related Lists 是把关联对象"聚"到主对象下的机制
- **§2 信息分层**：关联列表是二级信息，不是详情页的核心
- **§13 反模式分类**：“信息-认知失配”要求关联列表只保留当前任务所需信息

<a name="p05-master-detail"></a>
## 5. Master-Detail（主从布局）

### 解决的核心问题

**连续处理多个对象时，减少页面跳转带来的上下文切换**。

### 问题根源

```
场景：客服处理今日退款申请，需要逐个审核

❌ 反模式：每次点进去处理完，跳回列表，再点下一个
  → 列表 → 点详情 → 处理 → 返回列表 → 点详情 → 处理 → ……
  → 每次跳转丢失上下文（列表筛选、排序、已处理标记）

✅ Master-Detail：选中对象在右侧展示详情，处理完直接切下一个
  → 左侧列表（筛选后的待退款）→ 右侧详情（处理面板）→ 下一个
  → 上下文保留，处理路径缩短
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **当前选中项高亮** | 左侧列表当前行明确标识 | 用户始终知道自己在哪个对象上 |
| **状态同步** | 右侧处理后，左侧摘要更新（如金额变更） | 用户知道结果已生效 |
| **键盘导航** | 支持 ↑↓ 键切换对象 | 熟练用户效率提升 |
| **快速操作** | 详情区提供主操作，不需要跳转 | 减少操作路径 |

### 布局结构

```
┌─ 左侧：列表栏 ───────────────┬─ 右侧：详情栏 ──────────────────┐
│  [筛选] 待退款订单 (3)        │  订单 #SO202607120003            │
│ ───────────────────────────── │  状态：[退款中]                 │
│  ● SO202607120003 王力宏 ¥89  │  金额：¥89.00                  │
│    SO202607120004 张学友 ¥200 │                                 │
│    SO202607120005 刘德华 ¥50  │  退款原因：顾客误拍              │
│                               │  [同意退款] [驳回]              │
└───────────────────────────────┴─────────────────────────────────┘
```

### 理论桥接

- **§2 认知负荷**：减少页面跳转就是减少上下文切换，降低工作记忆负担
- **§3 状态轴**：列表栏本质是状态过滤（只显示"退款中"）
- **适用边界**：详情复杂度高时（如Object Page分区多），应跳转独立页而非Master-Detail

<a name="p06-dashboard"></a>
## 6. Dashboard（角色工作台）

### 解决的核心问题

**角色每天开始工作时，如何在几秒内了解全局状态、快速进入任务**。

### 问题根源

```
❌ 反模式：首页只是菜单快捷入口
  "店长打开系统 → 找菜单 → 点'订单管理' → 看待发货 → 回到首页 → 点'库存管理' → 看预警"
  → 每次都要自己拼凑"今天要关注什么"

✅ Dashboard：角色视角的"任务驾驶舱"
  "店长打开系统 → 首页直接展示：销售额28,450(待处理3笔)、待发货12单、库存预警5个"
  → 用户一眼知道今天要干什么
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **待办和异常优先** | 红色/橙色区域放在最显眼位置 | 用户优先处理紧急事务 |
| **KPI 可解释** | 指标有口径、时间范围、目标值 | 数字不再是无意义的装饰 |
| **指标可钻取** | 点击指标进入明细 | 指标不只是"看到了"，还要能"追下去" |
| **按角色设计** | 店长、财务、客服看到不同内容 | 避免"千人一面"的无效首页 |
| **数据新鲜度** | 展示统计截止时间、来源和刷新/延迟状态 | 避免把旧快照当成当前事实 |
| **指标治理** | 提供 KPI Contract：口径、目标、护栏指标、可操纵性和责任人 | 防止只优化数字而伤害真实业务目标 |

### 标准结构

```
┌─ 角色问候 ─────────────────────────────────────────────────┐
│ 早上好，张店长  ·  华东区 > 上海区 > 人民广场店  ·  07-12    │
└────────────────────────────────────────────────────────────┘

┌─ 今日待办与异常 ───────────────────────────────────────────┐
│ [紧急 · !] 待发货 12 单   [预警 · !] 库存预警 5 个         │
│ [注意 · !] 退款申请 3 单   [正常 · ✓] 设备状态 8/8 在线    │
└────────────────────────────────────────────────────────────┘

┌─ 核心指标 ─────────────┐ ┌─ 指标趋势 ─────────────────────┐
│ 销售额  ¥28,450  +12.3% │ │        销售趋势（折线图）         │
│ 订单数    156 单   -3   │ │  ▲ 28k                         │
│ 客单价   ¥182    +5.2%  │ │      ↗                           │
│ 完成率    85%    目标33k│ │  ───────────────  目标 33k      │
└────────────────────────┘ └─────────────────────────────────┘
```

### 理论桥接

- **§2 信息分层**：Dashboard 的信息分层是横向的（按优先级排列指标卡片）
- **§7 钻取链路**：每个指标卡片必须可钻取到明细
- **§13 反模式分类**：“信息-认知失配”和“任务-页面失配”解释了为什么首页不能只堆图表

<a name="p07-analytical-list-page"></a>
## 7. Analytical List Page（分析列表）

### 解决的核心问题

**图表和明细使用同一筛选条件，从指标分析直接下钻到业务明细**。

### 问题根源

```
❌ 反模式：图表和明细查询口径不一致
  图表显示"华东区销售额 ¥100万"
  明细表格筛选"华东区"只显示 ¥95万
  → 用户无法信任数据，无法对账

✅ Analytical List Page：图表和明细用同一筛选上下文
  左侧图表 → 点击"人民广场店" → 右侧明细表自动过滤到人民广场店
  → 数据同源，指标和明细可对账
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **统一筛选条件** | 图表和明细表使用同一筛选上下文 | 数据口径一致，可对账 |
| **图表可点击过滤** | 点击图表某区域过滤明细 | 从宏观到微观无缝切换 |
| **明细能解释指标** | 明细字段足以解释指标来源 | 不是"看到数字，不知道为什么" |
| **导出区分** | 明确导出的是汇总还是明细 | 避免用户导错数据 |

### 布局结构

```
┌─ 筛选条件 ─────────────────────────────────────────────────┐
│ [门店 ▼] [日期范围 ▼] [商品类目 ▼]         [查询] [重置]    │
└────────────────────────────────────────────────────────────┘

┌─ 指标摘要 ─────────────────────────────────────────────────┐
│ 销售额 ¥28,450  订单数 156  客单价 ¥182  完成率 85%        │
└────────────────────────────────────────────────────────────┘

┌─ 图表分析 ─────────────────────────────────┬─ 明细表格 ──┐
│  门店销售占比（饼图）                       │  订单号  门店    金额   状态   │
│                                           │  ───────────────────────── │
│  人民广场 42%  ●●●●●●●●●●                 │  SO001  人民广场 ¥1,280  完成  │
│  浦东店    33%  ●●●●●●●●                  │  SO002  浦东店   ¥980    完成  │
│  徐汇店    25%  ●●●●●●                    │  点击人民广场行自动过滤 ↓   │
│                                           │  SO003  人民广场 ¥1,200  待发  │
└────────────────────────────────────────────┴───────────────┘
```

### 理论桥接

- **§7 钻取链路**：从汇总指标到明细数据的无缝链路
- **§7.3 Drill-down 的五个设计要求**：每层保留筛选上下文
- **§7.3 钻取要求**：图表应能保留上下文地下钻到可核对的明细或业务证据

<a name="p08-worklist"></a>
## 8. Worklist（待办工作列表）

### 解决的核心问题

**把当前角色真正可处理的任务组织成队列，让用户看清责任、优先级、等待时间和下一步**。

### 问题根源

```
场景：客服处理今日退款申请

❌ 反模式：用普通列表处理
  列表包含"全部状态"（已完成+退款中+已驳回）
  → 用户需要每次手动筛选"只看退款中"
  → 已被处理的订单干扰视线

✅ Worklist：展示满足可处理条件的任务
  条件包括：退款待审核 + 属于当前数据范围 + 已分配/可认领 + 前置证据齐全
  → 用户打开页面就是有责任边界的任务队列，不需要自己拼筛选条件
  → 同时展示老化时间、截止时间和阻塞原因，避免任务饥饿
```

### 与 List Report 的区别

| 特征 | List Report | Worklist |
|---|---|---|
| 列表内容 | 全量 + 筛选器 | 按状态、权限、分配、依赖和时效计算出的可处理任务 |
| 用户目标 | 查找特定对象 | 连续处理待办 |
| 筛选器 | 需要用户主动筛选 | 无需筛选，打开就是任务 |
| 典型用户 | 查询者 | 执行者 |

### 典型案例

```
┌─ 今日退款处理 ──────────────────────────────────────────────┐
│  共 3 单待处理    已处理 0/3    [认领下一单] [导出]         │
│ ─────────────────────────────────────────────────────────── │
│  ☐ SO202607120003 王力宏 ¥89  退款原因：误拍  申请时间 09:58 │
│  ☐ SO202607120004 张学友 ¥200 退款原因：商品损坏  申请时间 09:45 │
│  ☐ SO202607120005 刘德华 ¥50  退款原因：地址错误  申请时间 09:30 │
│ ─────────────────────────────────────────────────────────── │
│  [查看证据] [逐单审核]                                      │
└────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§3 状态与能力**：Worklist 只呈现当前角色在状态、权限、范围和依赖上真正可执行的任务
- **§10.2 排队与流动**：除优先级外，还应展示老化、截止时间、阻塞和在制品
- **§12 保守规则**：连续处理 → Worklist；需并排查看详情时可组合 Master-Detail

<a name="p09-simple-form"></a>
## 9. Simple Form（简单表单）

### 解决的核心问题

**少量、低风险且关系简单的录入，如何在不离开当前页面的情况下快速完成**。

### 问题根源

```
场景：运营人员为商品添加二级分类

❌ 反模式：跳转到复杂编辑页
  用户当前在"商品列表"页 → 点"新增分类" → 跳转新页面 → 填完 → 返回
  → 路径长、容易忘记之前在干什么

✅ Simple Form：Modal 或 Drawer
  用户当前在"商品列表"页 → 点"新增分类" → 弹窗/抽屉 → 填完 → 关闭
  → 3个字段的轻量任务，在上下文中完成
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **字段与决策点** | 保持少量且关系简单；同时评估依赖、风险和比较需求 | 字段数相同，认知复杂度也可能完全不同 |
| **单任务** | 一个表单只完成一个明确任务 | "保存分类"不是"确定"——标题表达业务动作 |
| **字段级校验** | 错误直接定位到字段，不要顶部通篇提示 | 减少错误修正时间 |
| **不离开上下文** | 提交后刷新原列表，不跳转 | 用户记得之前在做什么 |

### 典型案例

```
┌─ 新增分类 ──────────────────────────────────────────────────┐
│                                                             │
│  分类名称 *                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 进口零食                                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  上级分类                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 零食饮料 ▼                                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  分类编码 *                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ SNACK_IMPORT                                          │   │
│  └─────────────────────────────────────────────────────┘   │
│  自动生成，或手动填写（编码唯一）                            │
│                                                             │
│                    [取消]  [保存分类]                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§6 表单**：表单是"业务承诺"，Simple Form 是最小粒度的承诺
- **§6.3 表单复杂度**：字段数只是信号；少量、低风险且无需跨字段比较时可用 Modal Form / Simple Form
- **§13 反模式分类**：“任务-页面失配”要求一个简单表单服务一个明确任务

<a name="p10-sectioned-form"></a>
## 10. Sectioned Form（分段表单）

### 解决的核心问题

**字段或决策点较多时，如何通过分区降低认知负荷并保留整体比较能力**。

### 问题根源

```
❌ 反模式：所有字段铺成一列
  用户面对40个字段，不知道先填什么、哪些必填、字段之间的关系
  → 认知过载，容易填错

✅ Sectioned Form：按业务含义分区
  用户面对4个分区，每个分区8-10个字段
  → "基础信息"填完 → "价格库存"填完 → ……
  → 分段推进，每段有明确目标
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **按业务含义分区** | 不按数据库表结构 | 用户脑子里就是按业务场景理解的 |
| **分区有锚点导航** | 长表单提供稳定的侧边锚点和错误定位 | 快速定位且不会遗漏错误分区 |
| **必填字段靠前** | 前两个分区集中必填字段 | 用户先完成最重要的 |
| **互斥字段动态处理** | 当前模式下无关字段隐藏或禁用 | 避免互斥字段同时校验报错 |
| **审计信息只读** | 创建人、更新时间放最后 | 只读信息不干扰录入 |

### 分区结构示例（零售商品资料）

```
┌─ 锚点导航 ─┬─ 分区内容 ────────────────────────────────────┐
│ ▼ 基础信息   │  名称、编码、类目、品牌、规格、单位            │
│   价格库存   │  零售价、会员价、库存、预警值                  │
│   渠道管理   │  销售渠道、上下架状态                         │
│   扩展信息   │  标签、图片、备注                             │
│   审计信息   │  创建人、创建时间、更新时间（只读）           │
└─────────────┴────────────────────────────────────────────┘
```

### 理论桥接

- **§2 认知负荷**：分区按业务语义和决策依赖形成可理解单元，同时保留需要横向比较的上下文；字段数只用于提醒复核，不是分区阈值
- **§6.3 表单复杂度**：字段较多但需要整体比较时，优先 Sectioned Form + 锚点
- **§13 反模式分类**：巨型表单应按业务任务分区；任务易中断或输入难重建时再增加草稿

<a name="p11-drawer-form"></a>
## 11. Drawer Form（抽屉表单）

### 解决的核心问题

**从列表页快速编辑少量字段，如何不跳转页面就完成**。

### 问题根源

```
场景：收银员需要给订单添加备注

❌ 反模式：跳转详情页
  列表页 → 点详情 → 找到备注输入框 → 填完 → 保存 → 返回列表页
  → 跳转2次，处理1个字段

✅ Drawer Form：右侧滑出抽屉
  列表页 → 点备注图标 → 右侧抽屉展开 → 填完 → 关闭
  → 上下文保留，处理完继续看列表
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **任务必须轻** | 抽屉内只做少量字段的查看或编辑 | 复杂详情不应塞进抽屉 |
| **不嵌套** | 抽屉内不要再打开抽屉 | 层级混乱，用户找不到出口 |
| **保存后刷新** | 关闭抽屉后原列表行数据更新 | 用户知道结果已生效 |
| **未保存提醒** | 关闭前对未保存内容做提示 | 防止误关闭丢失输入 |

### 与 Simple Form 的区别

| 特征 | Simple Form | Drawer Form |
|---|---|---|
| 触发来源 | 新增（创建新对象） | 编辑（修改已有对象） |
| 内容 | 从零录入 | 已有数据回填 |
| 宽度 | 固定（Modal宽度） | 可变（Drawer可展开更大） |
| 关闭行为 | 点外部或按钮关闭 | 需明确"取消/保存" |

### 理论桥接

- **§6.3 表单复杂度**：中等复杂度且需要参考当前页面时，可用 Drawer Form / Simple Form
- **§2 认知负荷**：抽屉保留了列表上下文，减少跳转

<a name="p12-modal-form"></a>
## 12. Modal Form（弹窗表单）

### 解决的核心问题

**短确认、填写原因、危险动作二次确认——如何在弹窗中安全承载轻量任务**。

### 问题根源

```
场景：退款确认

❌ 反模式：弹窗里放大表单
  退款弹窗里包含：退款金额、退款原因、退款方式、退款账户……
  → 弹窗变成另一个表单页，用户不知道是确认还是填写

✅ Modal Form：只承载确认意图
  弹窗标题："确认退款 ¥128.00"
  弹窗内容："退款将退回原支付方式，预计1-3个工作日到账"
  输入框：退款原因（必填）
  按钮：[取消] [确认退款（危险样式）]
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **字段少** | 确认类弹窗只填1-2个字段 | 不是另一个表单 |
| **后果明确** | 说明操作的影响范围 | 用户在确认前知道后果 |
| **风险表达** | 高风险操作用明确动词、风险图标和后果说明，颜色只作辅助 | 与普通操作区分且不依赖色觉 |
| **不嵌套** | 弹窗不打开弹窗 | 层级爆炸 |
| **关闭行为明确** | 必须有取消按钮，关闭不自动提交 | 用户不会误操作 |

### 典型案例

```
┌─ 确认退款 ──────────────────────────────────────────────────┐
│                                                             │
│  订单号：SO202607110045                                    │
│  原支付金额：¥ 128.00                                      │
│  ━━━━━━━━━━━━━━━━━━━━━━━━                                 │
│  退款金额：¥ 128.00  （原路退回）                           │
│                                                             │
│  退款原因 *                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 顾客误拍，已与顾客确认                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ⚠ 退款后将退回原支付方式，预计1-3个工作日到账             │
│                                                             │
│                    [取消]  [确认退款]                       │
│                         (危险样式)                          │
└─────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§4 操作风险**：Modal Form 是危险操作的标配承载
- **§6 表单**：Modal 是"确认意图"而非"录入意图"
- **§13 反模式分类**：“任务-页面失配”包括弹窗里放大表单和弹窗嵌套弹窗

<a name="p13-wizard"></a>
## 13. Wizard（向导模型）

### 解决的核心问题

**长流程多步骤，如何让用户每次只专注于当前步骤，降低整体认知负荷**。

### 问题根源

```
场景：开通一个新门店

❌ 反模式：单页巨型表单
  表单包含：基础信息（5字段）+ 业务配置（15字段）+ 硬件配置（8字段）+ ……
  → 用户面对30+字段，不知道从哪开始，不知道字段之间的关系

✅ Wizard：拆成4个步骤
  步骤1：填基础信息（5字段）→ 下一步
  步骤2：配置业务规则（15字段）→ 下一步
  步骤3：配置硬件（8字段）→ 下一步
  步骤4：预览全部配置 → 确认提交
  → 用户每步只面对一个明确任务
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **一步一任务** | 每个步骤只解决一个问题 | 聚焦，降低决策难度 |
| **局部校验** | 提交前做当前步骤的校验 | 不等到最后一步才发现前面的错误 |
| **跨步骤依赖可见** | 步骤之间有数据依赖要标注 | 用户知道为什么某些字段是灰色的 |
| **支持草稿** | 中途关闭可保存进度 | 防止长流程数据丢失 |
| **最后一步总览** | 预览全部配置 + 风险提示 | 用户在提交前最后确认 |

### 步骤结构示例

```
┌─ 步骤指示器 ──────────────────────────────────────────────┐
│  [1 基础信息] → [2 业务配置] → [3 硬件配置] → [4 预览确认] │
│        ✓             ○             ○             ○       │
└──────────────────────────────────────────────────────────┘

┌─ 步骤 1/4：基础信息 ─────────────────────────────────────┐
│  门店名称 *        [人民广场店___]                         │
│  门店编码 *        [SHOP_PDH_001__]                        │
│  所属区域 *        [华东 > 上海 > 浦东___]                  │
│  详细地址 *        [上海市浦东新区___]                      │
│                                    [上一步] [下一步]       │
└──────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§2 渐进披露**：用户不需要一次性看完全部内容
- **§6 表单复杂度**：多步骤有顺序依赖 → Wizard
- **§12 保守规则**：配置存在明确阶段依赖时 → Wizard
- **§6.3 表单复杂度**：只有步骤存在顺序依赖时才用 Wizard；需要频繁跨步骤比较时优先单页分区

<a name="p14-inline-edit"></a>
## 14. Inline Edit（行内编辑）

### 解决的核心问题

**表格中少量字段（如备注、排序号、状态）如何直接修改，减少操作路径**。

### 问题根源

```
场景：运营人员需要频繁调整商品排序号

❌ 反模式：每改一个排序号都要打开编辑页
  列表 → 点编辑 → 填写排序号 → 保存 → 返回 → 改下一个
  → 5次点击换一个数字

✅ Inline Edit：直接在列表行内修改
  列表 → 点击排序号单元格 → 直接输入 → Enter保存
  → 2次点击换一个数字
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **适合少量字段** | 单字段或2-3个关联字段 | 全字段行内编辑会丢失上下文 |
| **保存状态明确** | 编辑中/已保存状态可见 | 用户知道数据有没有被保存 |
| **字段级校验** | 错误贴近字段，不要顶部提示 | 修正成本低 |
| **高风险字段不用** | 金额、状态等高风险字段建议跳转编辑 | 行内修改没有确认环节 |

### 理论桥接

- **§2 认知负荷**：减少操作路径就是减少认知切换
- **§6 表单**：Inline Edit 是表格场景下的"表单变体"
- **§13 反模式分类**：“状态-操作失配”包括批量行内编辑没有保存状态和冲突处理

<a name="p15-drill-down-report"></a>
## 15. Drill-down Report（钻取报表）

### 解决的核心问题

**汇总指标如何可解释——用户从数字能追到具体业务对象**。

### 问题根源

```
❌ 反模式：Dashboard 只展示大数字
  "华东区销售额 ¥100万"  →  用户问：哪类商品卖得好？哪个门店拖累了？
  → 页面无法回答，用户无法对账

✅ Drill-down：四层链路
  汇总 ¥100万 → 维度：按门店 → 门店明细 → 具体订单
  → 用户从宏观到微观完整追溯
```

### 钻取链路示例

```
第一层：汇总指标
  华东区销售额 ¥28,450

      ↓ 点击"华东区销售"下钻

第二层：按门店维度
  人民广场 ¥12,000  浦东店 ¥9,500  徐汇店 ¥6,950

      ↓ 点击"人民广场"下钻

第三层：订单明细
  SO202607120001 ¥1,280  SO202607120002 ¥980  SO202607120003 ¥850 …

      ↓ 点击"SO202607120001"下钻

第四层：业务对象（Object Page）
  订单详情 + 时间线 + 关联对象 → 完整解释"为什么 ¥1,280"
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **每层保留上下文** | 下钻后返回，上级筛选条件不变 | 用户不迷路 |
| **汇总和下级能对账** | 明细合计 = 上级数字 | 数据可信 |
| **最后一层落到业务对象** | 不能永远停在数字 | 追到可操作的实体 |
| **返回路径** | 面包屑或返回按钮 | 支持回退 |

### 理论桥接

- **§7 钻取链路**：四层结构的完整设计
- **§7.3 Drill-down 五要求**：每层保留上下文、汇总可对账
- **§12 保守规则**：指标需要解释 → Drill-down
- **§13 反模式分类**：“任务-页面失配”包括报表只有结果、没有来源或行动入口

<a name="p16-exception-center"></a>
## 16. Exception Center（异常中心）

### 解决的核心问题

**系统异常如何从"技术人员看日志"变成"业务人员可处理"**。

### 问题根源

```
❌ 反模式：异常只在日志里
  支付回调失败 → 写入 error.log
  技术人员：知道有错误，排查原因
  业务人员：完全不知道

结果：用户打电话投诉"支付了但订单没变化"，客服无法解释

✅ Exception Center：
  支付回调失败 → 进入异常中心
  → 业务人员看到"3笔支付状态待核验" → 先按商户、订单和支付流水查询渠道结果
  → 已支付则幂等补记订单；未支付且原交易明确失败时，才允许引导顾客重新支付
  → 业务人员处理有明确预案的异常，超出预案或证据冲突时升级给技术/财务
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **异常分级** | 按严重程度分：ERROR / WARN / INFO | 优先处理高风险 |
| **绑定业务对象** | 异常关联到订单/会员/商品 | 异常不孤立，有上下文 |
| **提供受控动作** | 显示前置证据、幂等键、影响预览和适用条件，再提供核验/补记/升级 | 业务人员能行动，又不会重复产生副作用 |
| **处理留痕** | 谁处理、什么时候、怎么处理的 | 异常可追溯 |
| **重复异常聚合** | 同一接口大量失败聚合成一条 + 数量 | 避免列表刷屏 |

### 典型案例

```
┌─ 异常中心 ─────────────────────────────────────────────────┐
│  全部(12)  支付异常(3)  库存异常(2)  同步异常(7)           │
│ ─────────────────────────────────────────────────────────── │
│  🔴 支付回调失败        SO202607120003  ¥89    10:32       │
│     错误码：TIMEOUT     支付渠道：微信支付                  │
│     [查询渠道结果] [幂等补记] [升级处理]                    │
│ ─────────────────────────────────────────────────────────── │
│  🔴 支付回调失败        SO202607120004  ¥200   10:45       │
│     错误码：TIMEOUT     支付渠道：支付宝                    │
│     [查询渠道结果] [幂等补记] [升级处理]                    │
│ ─────────────────────────────────────────────────────────── │
│  🟠 库存扣减失败        SKU001        库存不足  09:15      │
│     当前库存：0  需求：2                                   │
│     [核验可售库存] [转人工分配] [升级处理]                   │
└────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§8 异常闭环**：五要素完整实现
- **§8.1 技术 vs 业务异常**：Exception Center 把技术语言翻译成业务语言
- **§12 保守规则**：异常需要闭环 → Exception Center
- **§13 反模式分类**：“系统-业务失配”要求把日志异常转成业务可处理任务

<a name="p17-draft-handling"></a>
## 17. Draft Handling（草稿保存）

### 解决的核心问题

**长表单填写中途中断，如何恢复工作进度**。

### 问题根源

```
场景：运营人员编辑商品资料（40+字段），填了一半被叫去开会

❌ 反模式：中途关闭页面全部丢失
  用户：填写了15分钟的数据全没了，明天重新来
  → 产生挫败感，可能放弃填写

✅ Draft Handling：
  用户关闭页面 → 系统提示"检测到未保存内容，是否保存草稿？"
  → 用户选择保存 → 草稿进入草稿列表 → 下次进入可恢复
  → 中断不丢数据，用户可以无缝继续
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **退出时提示** | 浏览器关闭/返回前检测未保存内容 | 防止误操作丢失 |
| **自动草稿** | 按风险、网络和输入重建成本选择定时或事件触发保存，并显示最后保存时间 | 中断后可恢复，也避免给出无法兑现的固定频率承诺 |
| **草稿列表** | 支持查看、恢复、删除 | 草稿不污染正式数据 |
| **版本与冲突** | 恢复时校验对象版本；检测到他人修改时提供差异对比 | 避免旧草稿静默覆盖新事实 |
| **提交后处置** | 提交成功后归档或按留存策略清理关联草稿 | 草稿列表不堆积且满足审计/隐私要求 |
| **草稿命名** | 支持用户给草稿起名，便于识别 | 多个草稿时知道哪个是什么 |

### 理论桥接

- **§6.2 表单七问**：任务耗时长、可被打断或输入难以重建时支持草稿，而不是按字段数机械判断
- **§13 反模式分类**：“正常-降级失配”要求中断或故障后存在可恢复点
- **§12 错误恢复原则**：草稿是错误恢复的具体实现

## 18. View Management（视图管理）

### 解决的核心问题

**用户如何保存和复用个人筛选偏好，避免每次重复筛选**。

### 问题根源

```
场景：客服人员 A 每天只看"待发货"订单，客服人员 B 只看"退款中"

❌ 反模式：每次打开列表都重新筛选
  客服 A：打开列表 → 筛选"待发货" → 处理 → 关闭
  明天：打开列表 → 筛选"待发货" → 处理 → 关闭

✅ View Management：
  客服 A：筛选"待发货" → 保存视图为"我的待发货"
```

| **视图共享** | 支持将视图共享给团队 | 团队标准化工作流 |
| **最近使用优先** | 最近使用的视图排在前面 | 常用视图快速访问 |
| **上下文保留** | 切换视图不丢失当前页码和选中状态 | 视图是"筛选+状态"的组合 |

### 理论桥接

- **§2 认知负荷**：视图管理消除了重复筛选的认知负担
- **§2 信息分层**：保存视图把重复筛选转化为可复用的任务表征
- **§12 保守规则**：视图是按角色和任务逐步定制工作环境的方式

## 19. Background Task Center（后台任务中心）

### 解决的核心问题

**后台异步任务（批量导入、数据导出）如何让用户追踪进度并处理结果**。

### 问题根源

```
场景：运营人员导入1000个商品

❌ 反模式：提交后无反馈
  用户：点"导入" → 等待10秒 → 页面显示"成功" → 关闭窗口
  实际问题：可能只成功了200个，失败800个，用户完全不知道

✅ Background Task：
  用户：点"导入" → 立即展示任务ID和预估时间
  页面：显示"导入中 356/1000 (35%)，预计剩余2分钟"
  完成后：通知用户"导入完成，成功900条，失败100条可下载修正"
  → 用户完整知道任务状态和结果
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **可信进度** | 有可靠总量时显示百分比；否则显示阶段、已处理数和更新时间，并标注估算 | 避免伪精确进度和持续回退 |

| **完成后通知** | 页面不打开时通过系统通知提醒 | 用户不需要一直盯着 |
| **任务历史** | 列表展示所有历史任务 | 可追溯、可复查 |

### 理论桥接

- **§6 表单反馈**：提交后必须解释发生了什么
- **§8.2 异常中心五要素**：失败条目是另一种形式的"异常"，需要可处理
- **§10.1 控制回路**：后台任务必须把接受、执行进度、部分成功和最终结果反馈给用户

## 20. Permission Matrix（权限矩阵）

### 解决的核心问题

**权限配置如何可视化，如何让管理员批量设置而不漏项**。

### 问题根源

```
场景：管理员为10个角色配置50个菜单权限

❌ 反模式：逐个角色配置，每个角色单独页面
  角色A → 勾选菜单1、菜单2…… → 保存
  角色B → 勾选菜单1、菜单3…… → 保存
  问题：无法横向对比"菜单1对哪些角色开放"，容易出现权限漏洞

✅ Permission Matrix：行列交叉矩阵
  横向：角色（店长/收银员/财务……）
  纵向：菜单/操作（订单管理/退款审核/报表查看……）
  单元格：勾选/不勾选
  → 一眼看出所有权限分布，任何权限变化可预览影响范围
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **权限类型区分** | 查看/新增/编辑/删除分别设置 | 不是"全开或全关" |
| **继承和覆盖可见** | 分别标记直接授权、继承授权和显式拒绝；半选只表示一组子项部分选中 | 权限来源透明且不误导 |
| **影响预览** | 保存前展示本次变更影响的人员 | 防止误配造成安全事故 |
| **变更审计** | 每次保存记录变更前后 | 权限变更可追溯 |

### 典型案例

```
┌─ 权限矩阵 ─────────────────────────────────────────────────┐
│            │ 店长  │ 收银员 │ 财务  │ 客服 │              │
│ ───────────┼───────┼───────┼───────┼──────┤              │
│ 订单查看    │  ✓    │   ✓   │   ✓   │  ✓   │              │
│ 订单新增    │  ✓    │   ✓   │   ✗   │  ✗   │              │
│ 订单编辑    │  ✓    │   ◐   │   ✗   │  ✗   │  ← ◐：部分子权限  │
│ 退款审核    │  ✓    │   ✗   │   ✓   │  ✗   │              │
│ 报表查看    │  ✓    │   ✗   │   ✓   │  ✓   │              │
│ 退款金额>500│  ✓    │   ✗   │   ✓   │  ✗   │              │
│ ───────────┴───────┴───────┴───────┴──────┘              │
│                                     [取消]  [保存配置]     │
│  ⚠ 本次变更将影响 3 个用户                          [预览] │
└───────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§4 操作风险**：权限是高风险操作，必须有预览和审计
- **§6 配置特殊性**：权限变更必须预览影响范围
- **§9.5 商业完整性与职责分离**：高风险权限变更应按风险采用 Maker-Checker，并记录审计日志

<a name="p21-audit-log"></a>
## 21. Audit Log（审计日志）

### 解决的核心问题

**高风险操作（金额修改、权限变更、配置调整）如何追溯**。

### 问题根源

```
场景：财务发现某笔订单金额被改过，但不知道谁改的、什么时候改的、为什么改

❌ 反模式：系统没有记录变更历史
  财务：只能问人，技术人员查数据库才能回答

✅ Audit Log：
  财务：打开审计日志 → 筛选"订单金额变更" → 看到完整记录
  "07-12 11:05 张三 订单SO202607120001 实付金额 ¥1,280 → ¥1,350"
  "原因：顾客使用追加优惠券"
  → 有据可查，有责可追
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **记录前后值** | 不只是"改了什么"，还要展示改前和改后 | 完整还原变更过程 |
| **按对象筛选** | 能查到某个订单/商品/会员的所有变更 | 快速定位问题 |
| **防篡改与受控处置** | 不参与普通业务删除；按适用法规、组织策略和法务保全执行留存、归档、访问与删除 | 在可追溯、隐私和法定义务之间保持可证明的治理 |
| **可导出** | 支持导出给审计或法务 | 配合合规要求 |

### 理论桥接

- **§5 可追溯性**：审计日志是 Timeline 的"管理员版本"（更长周期、更高风险）
- **§5.2 三层事件**：人工操作事件是审计日志的核心
- **§9.5 商业完整性与职责分离**：高风险变更需要留痕、复核和职责边界

<a name="p22-config-page"></a>
## 22. 配置页（Configuration Page）

### 解决的核心问题

**配置变更如何做到"改规则，影响一批对象"，而不只是"改一个对象"**。

### 问题根源

```
场景：运营人员把"满减门槛"从 ¥100 改成 ¥150

❌ 反模式：只改配置值，不说明影响范围
  用户：改了门槛 → 保存 → 不知道影响多少订单/活动
  结果：可能恰好有一个活动依赖旧门槛，改完直接崩

✅ 配置页：
  用户：改门槛 → 实时预览"本次变更将影响：3个进行中活动、12个待结算订单"
  → 改前知道影响，改后有预期
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **影响预览** | 保存前展示影响范围 | 防止改规则影响预期外的数据 |
| **生效时间** | 支持立即生效/定时生效 | 避免高峰期变更造成问题 |
| **版本化** | 每次变更为一个版本，支持回滚 | 改错了能恢复 |
| **变更审计** | 记录操作人、时间、变更前后值 | 可追溯 |
| **当前值说明** | 不只是"开/关"，要说明影响范围 | 配置意图透明 |

### 理论桥接

- **§6 配置特殊性**：配置变更是"改规则，影响一批"，不是"改一个对象"
- **§6.4 配置表单**：预览、回滚、审计三要素
- **§4 操作风险**：高风险配置 → 影响预览 + 审计记录

<a name="p23-comparison-view"></a>
## 23. Comparison View（对比视图）

### 解决的核心问题

**多对象横向对比时，如何让差异一目了然**。

### 问题根源

```
场景：店长对比 5 家门店的今日销售数据

❌ 反模式：5个门店各自一个数据卡
  用户：需要记住每个店的数字，然后自己比较
  → 记忆负担重，差异容易被忽略

✅ Comparison View：横向对齐排列
  所有门店用同一列对齐，数字差异一目了然
  → 一眼看出"浦东店落后了"
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **对比维度稳定** | 所有对象展示相同的字段 | 苹果对苹果 |
| **差异突出** | 用数值、方向图标和颜色共同标注相对基线的差异 | 异常自动突出且不只依赖颜色 |
| **量纲一致** | 不同量纲（金额/数量/率）不混在一个表 | 避免误导 |
| **时间维度** | 支持切换时间范围对比 | 趋势对比 |

### 典型案例

```
┌─ 门店销售对比 ─────────────────────────────────────────────┐
│            │ 今日销售额 │ 同比    │ 订单数 │ 完成率 │       │
│ ───────────┼───────────┼─────────┼────────┼────────┤       │
│ 人民广场   │ ¥12,000   │ +15.2%  │  82    │  91%   │  🟢   │
│ 浦东店     │  ¥9,500   │  -3.1%  │  65    │  78%   │  🔴   │
│ 徐汇店     │  ¥6,950   │  +8.7%  │  48    │  85%   │  🟡   │
│ ───────────┴───────────┴─────────┴────────┴────────┘       │
│  平均      │ ¥9,483    │  +6.9%  │  65    │  85%   │       │
└───────────────────────────────────────────────────────────┘
  ↑ 超均值   ≈ 接近均值   ↓ 低于均值（颜色仅作辅助）
```

### 理论桥接

- **§2 信息分层**：对比视图将信息从"分散"变成"对齐"，让差异可见
- **§2.4 任务-表征认知适配**：对比任务应对齐相同维度，并明确口径、单位和基线

<a name="p24-reconciliation-workbench"></a>
## 24. Reconciliation Workbench（对账工作台）

### 解决的核心问题

**系统流水和第三方流水如何核对，差异如何闭环处理**。

### 问题根源

```
场景：财务每日对账，确认系统订单金额和微信/支付宝流水一致

❌ 反模式：差异只在报表里
  财务：导出系统数据 → 导出微信对账单 → Excel 手动核对
  问题：手工核对效率低，差异行不知道怎么处理

✅ Reconciliation Workbench：
  系统流水和第三方流水同屏展示，差异行以状态文本、图标和颜色共同突出
  差异行先完成证据核验与归因，再按权限选择"生成调整单/挂账待查/升级复核"
  → 对账工作从手工变成半自动化
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **两方数据同屏** | 系统 vs 第三方同一行对齐 | 不需要手工对照 |
| **差异突出** | 金额、状态或时间不一致时用状态文本、图标和颜色共同标识 | 一眼看到差异且不只依赖颜色 |
| **证据与归因** | 展示双方流水号、来源时间、匹配规则和差异原因 | 处理动作建立在可核验事实上 |
| **受控处理动作** | 生成调整单/挂账待查/升级复核；高风险调整进入 Maker-Checker | 差异有明确路径且不能无证据直接改账 |
| **对账结果日结** | 全部差异已处理或有受控挂账后才允许确认日结 | 财务闭环不掩盖未决项 |

### 典型案例

```
┌─ 支付对账工作台 ───────────────────────────────────────────┐
│  对账日期：2026-07-11    系统流水：¥28,450   第三方：¥28,430│
│  差异金额：¥20.00（2笔）           [开始对账] [导出差异]    │
│ ─────────────────────────────────────────────────────────── │
│  状态    订单号       系统金额  第三方金额  差异   操作      │
│ ─────────────────────────────────────────────────────────── │
│  ✅ 一致  SO202607110001  ¥1,280  ¥1,280   ¥0   —          │
│  ✅ 一致  SO202607110002  ¥456    ¥456     ¥0   —          │
│  ! 差异   SO202607110003  ¥89     ¥79     ¥10   [生成调整单]│
│  ! 差异   SO202607110004  ¥200    ¥210    ¥10   [升级复核]  │
│ ─────────────────────────────────────────────────────────── │
│  对账结果：3笔一致，2笔差异待处理      [确认对账（不可用）] │
└────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§8 异常闭环**：差异是另一种异常，需要有处理动作
- **§10.4 分布式事务**：跨系统局部成功必须通过补偿、重试和对账收敛
- **§13 反模式分类**：“局部-全局失配”要求把对账差异变成可处理、可追踪的任务

<a name="p25-customer-360"></a>
## 25. Customer 360（客户全景视图）

### 解决的核心问题

**多渠道的客户数据如何归一呈现，让客服人员快速了解客户全貌**。

### 问题根源

```
场景：客服人员接到电话，需要快速了解这个会员的所有信息

数据来源：
  - 门店POS：小明在线下门店消费过3次
  - 微信商城：小明在线上下单过5次
  - 积分系统：小明有2000积分
  - 优惠券系统：小明有2张未使用优惠券
  - 售后系统：小明有过1次退货记录

❌ 反模式：客服在5个系统里来回切换
  → 耗时且容易遗漏关键信息

✅ Customer 360：
  一个页面聚合所有渠道数据
  客服一眼了解：小明是高频顾客，积分够兑换礼品，上次退货是因为尺码问题
  → 个性化服务有了基础
```

### 核心设计要素

| 要素 | 设计要求 | 解决什么问题 |
|---|---|---|
| **身份解析** | 通过已验证的标识映射、置信度和冲突流程关联手机号/会员卡/OpenID；不确定时保留候选身份 | 减少重复记录，也避免误合并不同客户 |
| **渠道区分** | 交易/权益/互动分区块展示 | 不同类型的信息不混在一起 |
| **标签分层** | 系统标签和人工标签区分，并显示来源、更新时间、适用目的和撤销方式 | 标签来源透明且不会成为永久身份判断 |
| **时间线聚合** | 只聚合当前角色和任务允许查看的事件，并保留渠道来源 | 理解客户历程且符合最小披露 |
| **推荐动作** | 展示依据、置信度、禁用规则和人工覆盖；敏感特征不得越过用途边界 | 服务有方向，同时避免自动化越权或不当画像 |

### 典型案例

```
┌─ 客户全景 ─────────────────────────────────────────────────┐
│  头像  周杰伦  138****8888  黄金会员  积分 2000            │
│  标签：[高价值] [零食爱好者] [VIP] [重要客户]               │
│ ─────────────────────────────────────────────────────────── │
│  消费概览    │ 权益信息         │ 最近活动                 │
│  累计消费 ¥3,280 │ 积分 2,000  │  满100减20活动中        │
│  订单数 8 单    │ 优惠券 2 张   │  有效期至 07-31         │
│ ─────────────────────────────────────────────────────────── │
│  交互时间线                                            ▼    │
│  07-12  微信下单  ¥89.00  订单#SO202607120003          │
│  07-11  门店消费  ¥156.00  人民广场店                   │
│  07-08  积分兑换  -200积分  兑换咖啡券                   │
│  07-05  退货申请  ¥80.00   尺码问题，已完成退款          │
│  06-28  投诉记录  商品缺货  已解决                        │
│ ─────────────────────────────────────────────────────────── │
│  建议：下次推荐服饰时提醒核对尺码                           │
│  依据：07-05 尺码问题退货 · 置信度：中 · [采用] [忽略]      │
└────────────────────────────────────────────────────────────┘
```

### 理论桥接

- **§1 数据聚合**：Customer 360 是"把散落在多渠道的数据聚合成一个完整客户画像"
- **§1.2 翻译方向**：多渠道数据 → 统一客户视图
- **§10.5 边界对象与语义互操作**：统一客户视图必须保留来源、身份映射和语义差异

## 选型框架

不要从组件名称或“字段数量”开始选模式。先用任务类型确定核心模式，再逐项叠加状态、风险、协作、时间和系统边界。

| 任务主线 | 首选模式 | 必须继续追问 |
|---|---|---|
| 查找、筛选、连续处理对象 | List Report / Worklist / Master-Detail / Analytical List | 是否有责任分配、老化、阻塞、批量影响和恢复点？ |
| 理解单个或关联对象 | Object Page / Timeline / Related Lists / Customer 360 | 当前事实来自哪里、何时发生、怎样形成、是否存在语义冲突？ |
| 录入、修改或配置 | Simple / Drawer / Modal / Sectioned Form / Wizard / Inline Edit / Config Page | 是否有顺序依赖、跨字段比较、草稿、冲突、模拟、复核和回滚？ |
| 监控、分析和比较 | Dashboard / Drill-down / Comparison View | 指标口径、数据时效、钻取证据和可执行动作是否明确？ |
| 处理异常或跨系统差异 | Exception Center / Background Task / Reconciliation Workbench | 是否区分接受、执行、部分成功、重试、补偿和最终收敛？ |
| 管理权限与可追溯性 | Permission Matrix / Audit Log | 是否展示有效权限、职责分离、影响范围、留存和最小披露？ |

在核心模式之后，执行以下五个门禁：

1. **状态门禁**：页面能否回答当前状态、形成原因、可执行动作以及失败或中断后的恢复方式？
2. **风险门禁**：动作是否按影响范围、可逆性和自动化程度提供预览、确认、复核、撤销或补偿？
3. **协作门禁**：是否明确责任人、承诺、交接、验收、冲突与共同证据，而不把“已发送”误当“已完成”？
4. **时间门禁**：是否区分发生时间、记录时间、数据时效、部分成功和跨系统事件顺序？
5. **控制权门禁**：自动化或 AI 失效时，用户是否知道依据、不确定性、边界，并能暂停、覆盖、接管或升级？

上述框架是筛选问题，不是机械决策树。最终选择必须用真实角色、真实任务、异常路径和恢复路径验证；理论依据及完整决策树见深层逻辑文档 §11。
