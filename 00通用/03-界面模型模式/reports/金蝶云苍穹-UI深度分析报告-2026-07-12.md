## 三、任务闭环分析

### 3.1 应用与页面设计任务

```text
入口：业务云 → 应用卡片 → 选中卡片 → 再点击标题进入
任务：选择功能分组 → 按设备/类型筛选 → 找到页面 → 进入设计/预览
出口：保存或预览 → 回到页面资产列表

闭环评估：基本闭环
优点：业务云、应用、分组、页面构成稳定资产层级。
断点：卡片首次点击仅选中；进入动作反馈弱；部分叠层操作后上下文收起；返回时上下文保持不稳定。
```

### 3.2 基础资料维护任务

```text
入口：业务工作台 → 维护物品分类
任务：确认自动编码 → 填写名称 → 保存或提交
出口：继续新增或退出

闭环评估：基本闭环
优点：快捷入口直接打开新增表单，减少列表中转。
断点：保存、保存并新增、提交的业务差异缺少就地说明；提交后的状态与下游影响不透明。
```

### 3.3 办公用品领用任务

```text
入口：应用看板 → 一键申请领用
任务：确认申请人上下文 → 填写事由 → 增加物品明细 → 提交
出口：进入审批或继续新增

闭环评估：基本闭环
优点：单据头与明细分录在同一弹层完成。
断点：页面未说明审批人、下一节点、库存预占、撤回条件和提交失败恢复方式。
```

### 3.4 管理者看板任务

```text
入口：办公用品领用应用首页
任务：查看库存、分类、来源、领用满足率和异常
出口：钻取明细或发起处理

闭环评估：断裂闭环
直接证据：多个卡片显示“该卡片已被删除”，多个方案显示“方案已删除”。
影响：看板仍占据大面积首屏，但无法回答指标、钻取或处理问题。
```

## 六、优缺点深度分析

### 6.1 优点

1. **业务对象优先，而不是控件优先**  
   深层原因：企业应用的核心不变量是对象、状态、权限和流程，而不是页面像素。  
   成立条件：业务具有主数据、单据、组织隔离、审批或审计要求。  
   效果：同一对象可派生 PC/移动表单、列表、报表和流程操作。  
   边界：简单一次性表单会承担过高建模成本。

2. **页面类型是业务模板，不是空白画布别名**  
   基础资料预置编码、名称、状态与组织语义；单据预置单号、状态、分录和附件；报表预置过滤与查询结构。  
   这降低了企业应用从零拼装造成的不一致风险。

3. **资产追溯信息完整**  
   页面卡片显示编码、类型、源页面、修改人和时间，满足管理员理解来源与责任的需求。  
   边界：尚缺“修改会影响哪些页面/菜单/流程”的依赖预览。

4. **筛选模型与大型资产库匹配**  
   设备、页面类型、排序和卡片/列表视图形成完整的 List Report 骨架。实测基础资料和单据筛选能准确收敛结果。

5. **运行态单据采用成熟的头体模型**  
   申请单将申请上下文与多行物品明细集中在同一任务面，适合企业业务录入。

### 6.2 缺点

1. **P0：看板失效资产没有闭环治理**  
   表现：多个卡片与方案显示已删除，但仍占据大面积首屏。  
   影响链：看不到指标 → 无法钻取 → 不知道影响对象 → 无修复入口 → 管理看板失去决策价值。  
   根因：配置资产删除与引用方运行时缺少自动降级和治理闭环。

2. **P1：应用卡片“选中”和“进入”反馈不足**  
   表现：第一次单击只选中，第二次点击标题才进入。  
   影响链：用户误判点击失败 → 重复点击或改点图标 → 增加探索与误操作风险。  
   根因：卡片同时承担选择容器和导航入口，但没有显式区分。

3. **P1：高风险与低风险操作处于同一图标层**  
   启停、扩展、预览、菜单、脚本等图标视觉权重接近。启停会改变运行状态，却没有在卡片上体现风险等级和影响范围。

4. **P1：单据状态只回答“是什么”**  
   暂存状态缺少来源、下一节点、审批人、库存影响与可撤回条件。状态四问覆盖度仅 1.5/4。

5. **P1：上下文保持不稳定**  
   部分叠层或应用内部页面在后续操作后返回应用管理，筛选和当前分组容易丢失，不利于连续处理多个页面资产。

6. **P2：筛选结果缺少摘要**  
   虽然筛选有效，但未清楚显示当前条件、结果数量和一键清除，资产规模增大后会增加确认成本。

7. **P2：工作台混合任务入口与决策看板**  
   快捷发起、业务向导、指标卡片、关注告警和布局工具同屏，角色与任务优先级没有充分分离。

8. **P2：复合设计器的选中与落点反馈不足**  
   深层容器容易命中父节点；拖放命令成功不等于结构创建成功。用户需要同时观察画布、大纲和属性面板才能确认结果。

## 八、模式选择合理性评估

| 模式 | 判断 | 理由 | 优化方向 |
|---|---|---|---|
| 业务云 → 应用 → 分组 → 页面 | 合理 | 与平台资产层级和权限边界一致 | 增加跨层搜索、最近使用和上下文恢复 |
| 应用卡片库 | 勉强合理 | 应用数量中等时扫描效率高 | 标题单击直接进入；操作按风险收敛 |
| 页面 List Report | 合理 | 异构页面需要多维筛选与排序 | 增加结果计数、条件摘要、保存视图 |
| 可配置 Dashboard | 设计不足 | 能承载任务与指标，但缺失引用治理 | 删除资产自动降级并提供修复动作 |
| 基础资料紧凑表单 | 合理 | 高频稳定档案字段少、语义明确 | 解释保存与提交差异 |
| 单据头 + 分录 | 合理 | 主对象与多行明细必须共同提交 | 补齐状态四问和提交影响预览 |
| 五区设计器 | 合理 | 复杂建模需要控件、实体、画布与属性协作 | 强化合法落点、层级导航和批量编辑 |

## 九、改进建议

### 9.1 优先级排序

| 优先级 | 问题 | 建议 | 预期收益 | 成本 | 验证方式 |
|---|---|---|---|---|---|
| P0 | 看板引用已删除资产 | 自动移除失效卡片；显示影响对象；提供替换/恢复/移除动作 | 恢复决策看板可用性 | 中 | 删除被引用卡片后，运行态无空洞且管理员能闭环修复 |
| P1 | 卡片进入方式不清楚 | 标题表现为链接，单击直接进入；选中态只用于批量管理 | 降低首次进入错误率 | 低 | 新用户一次点击进入成功率 ≥95% |
| P1 | 操作风险未分层 | 预览/进入外露；脚本/菜单/扩展收进菜单；启停独立并展示影响 | 降低误操作 | 中 | 启停误触率、撤销率下降 |
| P1 | 状态四问缺失 | 状态区显示来源、下一节点、责任人、超时和可用动作 | 减少流程咨询与错误提交 | 中 | 用户无需外部帮助即可回答状态四问 |
| P1 | 上下文容易丢失 | 持久化应用、分组、筛选、排序和滚动位置 | 提升连续资产处理效率 | 中 | 返回后上下文恢复率 100% |
| P2 | 筛选无结果摘要 | 显示条件标签、结果数量、清除与保存视图 | 降低大资产库确认成本 | 低 | 用户能在 2 秒内说出当前范围和数量 |
| P2 | 设计器落点反馈弱 | 合法落点高亮；非法时说明父子约束；创建后聚焦新节点 | 降低拖放失败与结构误解 | 中高 | 拖放后结构确认无需同时检查三处 |
| P2 | 工作台角色混杂 | 按业务用户/管理员提供任务型与分析型默认首页 | 降低首屏认知负荷 | 中 | 关键任务入口首屏可见率 ≥90% |

### 9.2 改进路线图

```text
Phase 1（可逆、低成本）
  - 卡片标题单击直达
  - 筛选条件标签、结果数量与清除
  - 保存/提交操作说明
  - 图标工具提示与可访问名称

Phase 2（任务闭环）
  - 单据状态四问区
  - 上下文持久化
  - 操作风险分层与影响预览
  - 失效看板引用治理

Phase 3（设计器效率）
  - 合法落点与父子约束可视化
  - 复合控件层级快速选择
  - 批量状态/权限配置
  - 跨端规则一致性检查
```

## 十、总体评价

| 维度 | 评分 | 说明 |
|---|---|---|
| 任务匹配 | 4/5 | 业务对象、单据、页面资产和治理模型与企业场景匹配 |
| 信息架构 | 4/5 | 层级清晰，但大型应用分组多、上下文恢复不足 |
| 认知负荷 | 2.5/5 | 高密度卡片、图标操作和复杂设计器偏向专家用户 |
| 操作效率 | 3/5 | 筛选能力完整，但进入反馈、上下文保持和批量治理有缺口 |
| 状态与反馈 | 2/5 | 运行态状态四问、设计态落点反馈和操作后果解释不足 |
| 异常闭环 | 1.5/5 | 失效看板资产可见但不能聚合、修复或追溯 |
| 企业治理 | 4.5/5 | 版本、来源、权限、插件、规则、组织和多端资产基础扎实 |

**综合评价**：金蝶云苍穹选择了正确的企业业务低代码路线，核心优势是将业务对象、状态、组织、权限、流程和多端布局作为统一资产治理，而不是把平台简化为页面拖拽器。当前 UI 的主要问题不是能力不足，而是复杂能力的交互解释和失效治理没有同步达到同等成熟度。系统适合经过培训的企业平台团队；若面向更广泛业务人员，需要进一步降低入口歧义、补齐状态与操作后果、恢复上下文，并把异常处理做成完整闭环。
