列表类

1. List Report 列表报告

解决的核心问题

大量对象中快速定位和批量处理。当列表可能有数万行时，用户需要在几秒内找到目标对象。

深层逻辑

按数据库结构展示需要用户在多个标签页间切换；按 List Report 展示，一个表格行聚合完成当前任务所需的关键信息。理论依据见深层逻辑 §2 信息分层：列表行是一级信息，详情页是按需进入的二级信息。

设计要点：
  • 筛选区默认展示核心字段，高级筛选折叠（符合Hick定律）
  • 每行聚合最能区分对象的字段（订单号、状态、金额、会员）
  • 状态使用文字和图标表达，颜色只作辅助；状态与优先级分开呈现
  • 高频操作（发货、详情）外露，低频操作（打印、导出）放更多菜单
  • 批量操作栏在有选中项时出现
详情类

2. Object Page 对象详情页

解决的核心问题

把散落在多个数据库表中的数据，聚合成一个用户能理解的业务对象。一个订单在数据库里涉及8+张表，用户脑子里却是一个完整的"订单"概念。

深层逻辑

Object Page 是把数据库语言翻译成用户任务语言的边界对象。它应回答状态四问：是什么、为什么、怎么办、失败或中断后如何恢复。理论依据见深层逻辑 §1 与 §3.2。

  • 操作受状态控制：只有"已支付且库存已锁定"才能发货
状态类

3. Timeline 时间线

解决的核心问题

让用户理解"这个状态是怎么变成这样的"——即业务事实的可追溯性。财务对账、问题排查、责任追溯，都依赖这个能力。

深层逻辑

时间线应区分系统自动事件、人工操作事件和业务状态事件，并按场景记录时间、主体、前后状态、原因、关联对象和证据。跨系统场景还需区分发生时间与记录时间；理论依据见深层逻辑 §5 与 §10.6。

详情类

4. Related Lists 关联列表

解决的核心问题

一个业务对象关联的其他对象，如何在不喧宾夺主的情况下完整呈现。订单关联：会员、支付单、优惠券、物流、发票、售后单……

深层逻辑

Related Lists 把当前任务需要的关联对象聚合到主对象下。关联列表属于二级信息，只展示能支持识别和行动的关键字段；完整查询进入独立页面。理论依据见深层逻辑 §1 与 §2。

  • 关联列表内只做轻量操作，复杂操作跳转
  • 大数据量默认分页或只展示最近5条
布局类

5. Master-Detail 主从布局

解决的核心问题

连续处理多个对象时，减少页面跳转带来的上下文切换。用户打开页面就是任务队列，不需要自己筛选。

深层逻辑

减少页面跳转就是减少上下文切换，降低工作记忆负担。列表栏本质是状态过滤（只显示"退款中"），右侧详情提供主操作。适用边界：详情复杂度高时应跳转独立页而非Master-Detail。

工作台类

6. Dashboard 角色工作台

解决的核心问题

角色每天开始工作时，如何在几秒内了解全局状态、快速进入任务。这是"任务驾驶舱"，不是"菜单快捷入口"。

深层逻辑

角色工作台应围绕一个角色的任务闭环组织状态、解释和行动，回答优先事项、异常和指标变化，而不是堆放图表。理论依据见深层逻辑 §2、§3 与 §7。

分析类

7. Analytical List Page 分析列表

解决的核心问题

图表和明细使用同一筛选条件，从指标分析直接下钻到业务明细。解决"图表和明细查询口径不一致"的问题。

深层逻辑

指标、图表和明细必须共享口径与筛选上下文，点击图表后应能落到可核对的数据和业务对象。理论依据见深层逻辑 §7 钻取链路。

状态类

8. Worklist 待办工作列表

解决的核心问题

把当前角色真正可处理的任务组织成队列，明确责任、优先级、等待时间和下一步，而不只是按单一状态过滤。

深层逻辑

Worklist由状态、权限、数据范围、分配、依赖和时效共同决定。它减少筛选负担，也要防止任务饥饿和责任不清。对应深层文档的“状态与能力”及“排队与流动”。

  • 高风险退款逐单查看证据并审核，不提供无差别批量同意
  • 每行展示金额、原因、等待时间、截止时间和阻塞状态
  • 主操作随能力条件变化；条件不足时显示原因和升级路径

15. Drill-down Report 钻取报表

解决的核心问题

汇总指标如何可解释——用户从数字能追到具体业务对象。"销售额 ¥28,450"只是一串数字，必须能解释为什么是这个数字。

深层逻辑

钻取链路通常经过汇总指标、维度拆解、明细数据和业务对象。每层保留筛选上下文，最终落到可核对或可操作的实体；理论依据见深层逻辑 §7。

  • 汇总和下级能对账（明细合计 = 上级数字）
  • 最后一层必须落到业务对象（Object Page）
  • 面包屑或返回按钮支持回退
异常类

16. Exception Center 异常中心

解决的核心问题

系统异常如何从"技术人员看日志"变成"业务人员可处理"。技术异常必须翻译成业务语言，并提供处理动作。

深层逻辑

异常只在日志里会造成系统-业务失配。Exception Center让业务人员看到“3笔支付状态待核验”，先查询渠道结果，再按证据执行幂等补记或升级处理，不能因回调超时直接再次扣款。

  • 异常要绑定业务对象（订单号、商品编号）
  • 处理动作必须展示前置证据、适用条件、幂等语义和影响范围
  • 处理过程要留痕（谁处理、什么时候、怎么处理的）
  • 重复异常要合并或聚合

设计好坏的最小检查：
  1. 状态：能否回答当前状态、形成原因、可执行动作，以及失败或中断后的恢复方式？
  2. 风险：是否按影响范围、可逆性和自动化程度提供预览、复核、撤销或补偿？
  3. 协作：是否明确责任、承诺、交接、验收、冲突与共同证据？
  4. 时间：是否区分发生时间、记录时间、数据时效、部分成功和跨系统事件顺序？
  5. 控制权：自动化失效时，用户能否理解依据与边界，并暂停、覆盖、接管或升级？
这些问题是筛选门禁，不是机械决策树。全部成立只是最低门槛，还需用真实角色、任务、异常路径和恢复路径验证。
