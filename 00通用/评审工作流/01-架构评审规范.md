# 架构评审规范

> 版本：v2.1
> 日期：2026-07-12
> 定位：架构评审规则、评审理念、反模式
> 适用范围：架构设计评审、ADR规范

## 1. 评审触发条件

满足任一条件即触发架构评审：
- 新系统/新模块的架构设计
- 技术栈重大变更（如更换数据库、框架）
- 架构模式变更（如单体→微服务、同步→异步）
- 引入新的外部依赖（MQ、缓存、搜索引擎）
- 性能目标发生数量级变化
- 安全等级提升（公网暴露、处理敏感数据）
- 跨模块/跨系统集成方案

## 2. 评审核心理念

### 2.1 对抗性评审

每个架构决策必须有反对者（Devil's Advocate），强制双方辩论并记录权衡过程。

```
决策：使用Redis作为缓存
├─ 支持者（Proponent）：
│  └─ 理由：性能好、支持丰富数据结构
│
└─ 反对者（Devil's Advocate）：
   ├─ 质疑1：Redis不可用时怎么办？
   ├─ 质疑2：多实例缓存如何同步？
   ├─ 质疑3：为什么不用本地缓存+DB轮询？
   ├─ 质疑4：Redis内存不足时会影响主业务吗？
   └─ 质疑5：运维成本如何？
```

### 2.2 权衡矩阵法

每个架构决策点必须列出 ≥3 个候选方案，且每个落选方案必须有明确拒绝理由。

| 架构点 | 候选A | 候选B | 候选C | 采纳 | 拒绝理由 |
|---|---|---|---|---|---|
| 元数据存储 | 少表+JSON聚合 | 子定义全部拆表 | JSON聚合+影子表 | A+影子表 | B事务成本高 |
| 业务数据 | 动态表 | EAV | JSON宽表 | A+SchemaSync | EAV查询复杂；JSON损失约束 |

强制规则：
- 必须有 ≥3 个候选
- 每个落选方案必须有明确拒绝理由
- 决策必须记录到 ADR

### 2.3 质量属性量化

质量目标必须可度量、可验收，禁止模糊描述。

```
❌ 错误：系统要"性能好"、"安全"、"可扩展"
✅ 正确：
- 性能：冷启动<2s，list P95<500ms，并发1000 QPS
- 安全：通过OWASP Top 10测试，无SQL注入/XSS漏洞
- 可扩展：新增字段类型不改核心代码，通过SPI注册
```

质量属性模板：

| 维度 | 定义 | 可验收标准 | 验证方式 |
|---|---|---|---|
| 正确性 | 元数据、物理结构、缓存、快照四方一致 | 发布后Reconciler对账无差异 | 集成测试 |
| 安全性 | 六层权限统一判定，恶意输入有测试集 | 通过14类安全测试 | 安全测试 |
| 性能 | 冷加载<2s，list P95<500ms | 基准数据集测试通过 | 性能测试 |
| 可扩展性 | 新增类型=实现SPI+注册 | 22种字段类型通过SPI实现 | 契约测试 |

### 2.4 陷阱驱动评审

基于历史陷阱清单，逐项检查防线。

```
架构变更：引入动态DDL
涉及陷阱：
  - A1: DDL非原子
  - A2: DDL不可回滚
  - A3: 动态表爆炸
  - A5: MDL锁雪崩

检查防线：
  A1 → 是否有发布状态机？✓
  A1 → 是否持久化DDL Plan？✓
  A1 → 是否有Reconciler？✓
  A2 → 是否定义回滚边界？✓
  A3 → 是否有表数量上限？✓
  A5 → 是否有MDL预检？✓
```

详见：`../工程规范/低代码平台规范/06-陷阱覆盖矩阵.md`

## 3. ADR规范

### 3.1 ADR必须包含的内容

```markdown
# ADR-<项目>-<领域>-<序号>-<简短描述>.md

## 状态
proposed | accepted | deprecated | superseded

## 背景
为什么要做这个决策？

## 决策
我们决定采用什么方案？

## 候选方案
- 方案A：...（采纳）
- 方案B：...（拒绝，理由：XXX）
- 方案C：...（拒绝，理由：XXX）

## 权衡
| 维度 | 方案A | 方案B | 方案C |
|---|---|---|---|
| 性能 | 高 | 中 | 低 |
| 复杂度 | 中 | 高 | 低 |
| 成本 | 低 | 高 | 低 |

## 后果
### 正面影响
- ...

### 负面影响
- ...

## 验证方式
- 性能测试：XXX
- 集成测试：XXX
- 安全测试：XXX

## 参考资料
```

### 3.2 ADR命名规范

```
ADR-<PROJECT>-<DOMAIN>-<序号>-<简短描述>.md

示例：
ADR-LOWCODE-PUBLISH-001-persistent-publish-pipeline.md
ADR-LOWCODE-PERM-001-access-view-permission-core.md
```

## 4. 架构评审检查点

### 4.1 通用检查点

- 业务目标：架构是否支持业务目标？是否有明确的成功标准？
- 架构决策：每个决策点是否 ≥3 个候选？每个落选方案是否有明确拒绝理由？是否记录到 ADR？
- 模块边界：模块职责是否单一？依赖方向是否正确？是否有循环依赖？
- 质量属性：8 个维度是否都有可度量目标？每个目标是否有验证方式？
- 陷阱覆盖：是否逐项检查了陷阱清单？每个陷阱是否有防线？
- 风险与缓解：是否识别了所有风险？每个风险是否有缓解措施？是否有回滚方案？
- 可实施性：里程碑划分是否合理？依赖项是否清晰？团队能力是否匹配？

### 4.2 安全检查点

- 认证方式、会话管理策略、CSRF 防护、权限模型
- 所有外部输入是否校验？SQL 注入、XSS、SSRF 防护
- 多租户数据隔离策略、tenantId 传递、fail-fast 机制
- 敏感数据识别、加密存储、传输加密、日志脱敏
- 关键操作是否审计？审计日志是否防篡改？

### 4.3 性能检查点

- N+1 查询、索引策略、慢查询预防、连接池配置
- 缓存策略、缓存失效机制、缓存雪崩防护
- 并发控制策略、乐观锁/悲观锁、死锁预防
- 内存泄漏、连接泄漏、文件句柄、线程池配置

### 4.4 可扩展性检查点

- 是否支持多实例？状态如何共享？是否有单点？
- 扩展点是否 SPI 化？扩展点是否有示例？
- API/数据/配置兼容策略

## 5. 评审结论门禁

| 结论 | 条件 |
|------|------|
| APPROVE | 无 BLOCKER，2/3 以上同意，质量目标可达 |
| COMMENT | 无 BLOCKER，有中等风险或待补证据，记录负责人和截止时间 |
| NEEDS HUMAN DECISION | 需要业务/安全/架构负责人裁决，记录负责人和截止时间 |
| REQUEST CHANGES | 有重大架构缺陷，需要重新设计 |

## 6. 架构评审反模式

### 6.1 橡皮图章评审

- 评审会上没人提问题，大家都"看起来不错"，10分钟走过场
- **正确做法**：指定魔鬼代言人强制质疑，每个决策必须辩论

### 6.2 权威压制

- 权威者提出方案，下属不敢反对
- **正确做法**：技术问题用技术说话，鼓励反对意见，权威者最后发言

### 6.3 事后评审

- 代码都写完了才开评审会
- **正确做法**：设计阶段就评审，编码前评审通过

### 6.4 完美主义

- 满足门禁的方案因主观审美要求重做
- **正确做法**：满足门禁即可通过，关注关键风险而非主观审美

### 6.5 形式主义

- PPT 做得漂亮，但没有权衡矩阵、ADR、陷阱覆盖
- **正确做法**：内容重于形式，必须有权衡过程、ADR、陷阱覆盖

## 7. 架构约束规则（ArchUnit）

```java
// 禁止循环依赖
@ArchTest
public static final ArchRule modules_should_not_have_cycles =
    slices().matching("com.lowcode.(*)..").should().beFreeOfCycles();

// Service不依赖Controller
@ArchTest
public static final ArchRule service_should_not_depend_on_controller =
    noClasses().that().resideInPackage("..service..")
        .should().dependOnClassesThat().resideInPackage("..controller..");
```

## 8. 评审度量

- **过程指标**：评审覆盖率、评审周期、质疑密度
- **结果指标**：架构变更率、架构缺陷密度、ADR 完整性
