# AI代码评审SOP

> 版本：v2.4
> 日期：2026-07-12
> 定位：AI执行代码评审的唯一入口SOP
> 适用范围：既有系统代码评审、PR评审、目录/文件级代码审查

## 0. 定位

本文档是 AI 执行代码评审的**唯一入口**。

AI 读取本文档后，按角色分工执行评审、输出证据化结论。
涉及业务取舍、风险接受、上线决策时，必须由人类负责人确认。

## 1. 评审角色体系

代码评审由 **8 个角色**共同完成，缺一不可。

### 1.1 角色总览

| 角色 | 职责 | 核心问题 | 视角 |
|------|------|----------|------|
| **code-reviewer** | 综合评审：逻辑、陷阱、红线、旧逻辑 | "这段代码有什么问题？" | 攻击者视角 |
| **quality-reviewer** | 代码质量：可读性、可维护性、重复代码 | "这段代码是否容易理解和维护？" | 匠人视角 |
| **api-reviewer** | 接口契约：DTO/VO一致性、API规范、返回值 | "接口契约是否完整和正确？" | 接口消费者视角 |
| **security-reviewer** | 安全审计：注入、越权、敏感信息、XSS/CSRF | "有没有安全漏洞？" | 黑客视角 |
| **performance-reviewer** | 性能审计：数据库查询、N+1、缓存、并发 | "这段代码有没有性能陷阱？" | 压测工程师视角 |
| **style-reviewer** | 风格规范：命名、注释、格式化、Java/TS规范 | "代码是否符合团队规范？" | 主编码规范官视角 |
| **test-engineer** | 测试评审：测试覆盖、边界用例、Mock合理性 | "测试是否充分覆盖了变更？" | 测试工程师视角 |
| **verifier** | 验证评审：修复确认、回归检查、缺陷关闭 | "修复是否真正解决了问题？" | 收口确认者视角 |

> **对抗性评审理念**：对抗者（Devil's Advocate）不是独立角色，而是贯穿所有评审阶段的思维原则。
> 每个角色在执行自己的评审时，必须同时扮演质疑者角色——不信任未经验证的假设，不放过"快速通过"的诱惑。
> verifier 阶段的最终结论中包含对前期评审结论的再次质疑。

### 1.2 角色分组

```
评审阶段（6 个评审角色）
├── code-reviewer     — 综合评审
├── quality-reviewer  — 代码质量
├── api-reviewer     — 接口契约
├── security-reviewer — 安全审计
├── performance-reviewer — 性能审计
└── style-reviewer   — 风格规范

支持阶段（2 个支持角色）
├── test-engineer     — 测试评审
└── verifier          — 验证评审
```

八个角色**独立执行**，不共享中间结论，完成后合并。

## 3. 输入确认

评审开始前确认：
- **评审范围**：PR / 分支 diff / 单文件 / 目录
- **项目根目录**：实际业务仓库根目录
- **变更目标**：需求、bug、重构、配置
- **风险领域**：金额、库存、订单、权限、多租户、支付、MQ
- **本次参与角色**：8 个角色全部参与（特殊情况需人工决定角色裁剪）

## 4. 执行流程

### 阶段一：六个评审角色独立执行

#### 角色1：code-reviewer（综合评审）

**职责**：正面扫描，按检查清单逐项检查。

**Step 1 确认评审范围**
1. 读取 git diff 或用户指定文件
2. 列出变更文件、变更类型（新增/修改/删除/重命名）
3. 避免把无关改动误判为本次变更

**Step 2 识别旧逻辑和隐性契约**
必须回答：
- 旧入口在哪里？旧调用方有哪些？
- 历史数据是否兼容？
- 配置、定时任务、消息链路是否受影响？
- 是否有新逻辑启用条件？是否有回滚方案？

无法确认时，结论写为"未知"，并给出最小验证动作。

**Step 3 定位陷阱和风险域**
根据变更内容，对照陷阱矩阵筛选本次涉及的陷阱，逐条确认是否有对应防线。
详见 `03-代码评审规范.md` §3。

**Step 4 执行红线检查**
详见 `03-代码评审规范.md` §2 安全红线。

发现以下问题即为 **BLOCKER**：
- 跨租户读写风险
- 权限绕过
- SQL/表达式注入
- 金额/库存/订单状态错误
- 无幂等的支付/订单/MQ消费
- 核心旧逻辑被删除/改变
- 密钥/token/凭证硬编码
- 敏感信息明文日志

**Step 5 输出 code-reviewer 结论**
```
## code-reviewer 发现

### BLOCKER
### HIGH
### MEDIUM
### LOW
### 未覆盖风险
```

#### 角色2：quality-reviewer（代码质量评审）

**职责**：评估代码的可读性、可维护性，识别重复代码和坏味道。

**Step 6 代码质量专项检查**

**可读性**
- 方法是否过长（建议不超过 80 行）？
- 变量命名是否清晰、符合业务语义？
- 嵌套层级是否过深（建议不超过 3 层）？
- 是否有不必要的类型转换或魔法值？

**可维护性**
- 是否存在重复代码（DRY 原则）？
- 是否有过于复杂的条件判断可简化为卫语句？
- 是否过度设计（为假想需求添加了不必要的抽象）？
- 异常处理是否合理（不要吞掉异常、不要用异常做流程控制）？

**依赖管理**
- 是否引入了不必要的依赖？
- 循环依赖是否被引入？
- 是否引入了与项目技术栈不一致的库？

**Step 7 输出 quality-reviewer 结论**
```
## quality-reviewer 结论

### HIGH（影响可维护性）
### MEDIUM（影响可读性）
### LOW（代码风格）
### 未覆盖的质量风险
```

---

#### 角色3：api-reviewer（接口契约评审）

**职责**：确保接口契约的完整性和一致性。

**Step 8 接口契约专项检查**

**DTO/VO 规范**
- DTO、VO 是否符合项目规范（注释、字段顺序、命名）？
- 是否复用既有 DTO 而非新建？
- 新增字段是否有中文注释？
- 枚举是否端到端（DTO/VO/Entity/Service 使用同一种枚举类型）？

**接口一致性**
- RESTful 风格是否规范？
- 请求方法（GET/POST）使用是否正确？
- 分页、排序参数是否规范？
- 错误码和状态码是否一致？

**Feign 接口**
- Feign 接口与实现是否一致？
- 参数传递是否完整（mid/sid/lid 是否漏传）？
- 超时、重试配置是否合理？

**数据库变更**
- 新增字段是否向后兼容？
- 是否修改了现有字段的语义或默认值？
- 索引变更是否影响查询性能？

**Step 9 输出 api-reviewer 结论**
```
## api-reviewer 结论

### HIGH（接口契约风险）
### MEDIUM（一致性风险）
### LOW（规范建议）
### 未覆盖的接口风险
```

---

#### 角色4：security-reviewer（安全审计）

**职责**：专职安全视角，深挖其他评审角色可能遗漏的安全问题。

**Step 10 专职安全检查**
不依赖其他评审角色结论，独立执行以下安全检查：

**身份与认证**
- 是否有新接口？新接口是否走统一鉴权入口？
- 认证方式是否正确？token 校验是否完整？
- 是否有未鉴权的接口被引入？

**权限控制**
- /meta、list、update 鉴权是否一致？
- 是否有越权风险？（横向越权、纵向越权）
- 超级管理员是否使用了 break-glass 机制？

**输入安全**
- 所有外部输入是否校验？
- SQL 注入：表名、列名、op 操作符是否白名单/枚举？用户值是否参数化？
- XSS：富文本输入是否净化？
- SSRF：是否有 URL 外跳？是否有内网探测风险？
- 命令注入：是否有系统命令调用？

**敏感信息**
- 是否有新的密钥、token、凭证硬编码？
- 是否有手机号、身份证、token 明文日志？
- 敏感数据是否加密存储？

**多租户**
- tenantId 是否 fail-fast？
- 查询是否强制注入 tenant_id？
- 异步/MQ/定时任务是否显式传递 tenantId？

**Step 11 输出 security-reviewer 结论**
```
## security-reviewer 结论

### BLOCKER（安全漏洞）
### HIGH（高危安全风险）
### MEDIUM（中等安全风险）
### LOW（安全建议）
### 未覆盖的安全风险
```

---

#### 角色5：performance-reviewer（性能审计）

**职责**：识别性能陷阱，确保变更不会引入性能退化。

**Step 12 性能专项检查**

**数据库查询**
- 是否有 N+1 查询？
- 是否有不必要的全表扫描？
- 索引是否合理（尤其是新增查询条件）？
- 分页是否正确实现（避免大 offset）？
- 是否在循环内执行数据库操作？

**并发与锁**
- 是否有长事务或锁等待风险？
- 分布式锁是否正确使用（超时、防重入）？
- 是否有乐观锁冲突风险？
- 是否有死锁可能？

**缓存**
- 缓存 key 设计是否合理？
- 是否有缓存雪崩/击穿风险？
- 缓存与数据库一致性是否被破坏？

**异步与 MQ**
- 异步任务是否正确处理失败和重试？
- MQ 消费是否幂等？
- 消息积压风险是否存在？

**内存**
- 是否有大对象频繁创建？
- 是否有内存泄漏风险？

**Step 13 输出 performance-reviewer 结论**
```
## performance-reviewer 结论

### BLOCKER（性能严重退化）
### HIGH（性能风险）
### MEDIUM（优化建议）
### LOW（可优化项）
### 未覆盖的性能风险
```

---

#### 角色6：style-reviewer（风格规范评审）

**职责**：确保代码符合团队编码规范。

**Step 14 编码规范检查**

**Java 规范**（如有 Java 代码变更）
- 是否遵循 `.claude/java.md` 规范？
- 包名、类名、方法名命名是否符合规范？
- 注解使用是否正确（一行一个纵向排列）？
- Javadoc 注释是否完整（Service 方法、定时任务、消息消费）？
- Lombok 使用是否与项目一致？

**TypeScript/Vue 规范**（如有前端代码变更）
- 是否遵循前端编码规范？
- 是否禁止使用 `any` 类型？
- 是否遵循 Vue/uni-app 模板规范（`<template v-if>` 等）？
- 用户可见文案是否国际化？

**通用规范**
- 文件编码是否为 UTF-8 无 BOM（Java 源码）？
- 缩进是否一致？
- 是否引入了未使用的导入？

**Step 15 输出 style-reviewer 结论**
```
## style-reviewer 结论

### HIGH（违反规范）
### MEDIUM（规范偏离）
### LOW（格式建议）
```

---

### 阶段二：测试评审

#### 角色7：test-engineer（测试评审）

**职责**：评估测试覆盖的充分性，识别测试缺口。

**Step 16 测试覆盖专项检查**

**单元测试**
- 核心业务逻辑是否有单元测试？
- 边界条件是否被覆盖（空值、极值、异常值）？
- Mock 是否合理（是否 mock 了不该 mock 的依赖）？

**集成测试**
- 涉及数据库的变更是否有集成测试？
- Feign 调用是否有 Mock 或 TestContainer？
- MQ 消息处理是否有测试？

**变更针对性**
- 新增逻辑是否有对应测试？
- 修改的逻辑是否更新了对应测试？
- 删除的逻辑是否删除了对应测试？

**测试质量**
- 测试是否可重复执行？
- 测试之间是否相互独立？
- 断言是否有意义（不是只 assert "!= null"）？

**Step 17 输出 test-engineer 结论**
```
## test-engineer 结论

### HIGH（测试缺口）
### MEDIUM（覆盖不足）
### LOW（测试优化建议）
### 建议补充的测试用例
```

---

### 阶段三：对抗性质疑与合并

#### 角色8：verifier（验证评审 + 对抗质疑）

**职责**：
1. 在所有评审角色完成后，对整体结论进行对抗性质疑
2. 开发者修复后，验证问题是否真正关闭
3. 确认无新风险引入

**Step 18 对抗性质疑（第一轮：合并前）**

verifier 在合并前必须质疑：

**对"无问题"的质疑**：
- 是否有评审角色跳过了某类检查？
- 陷阱矩阵涉及的陷阱是否被逐条确认？
- 是否存在评审者未识别的旧逻辑依赖？
- 是否有未声明的隐性契约被本次变更破坏？

**强制质疑问题**：
1. code-reviewer 是否检查了租户隔离？（B1）
2. code-reviewer 是否检查了 SQL 注入？（C1）
3. 是否识别了旧调用方和回滚路径？
4. security-reviewer 是否独立发现了所有安全风险？
5. performance-reviewer 是否检查了并发锁风险？
6. api-reviewer 是否检查了枚举端到端一致性？
7. 是否有评审角色在"快速通过"压力下跳过了检查？

**Step 19 合并六个评审角色结论**

按以下规则合并：
1. **驳回**：被任意角色驳回的问题，从问题清单中移除，记录驳回理由
2. **补充**：被任意角色补充的问题，加入问题清单
3. **升级**：任何角色认为 HIGH，按 HIGH 处理；安全审计的 BLOCKER 结论优先级最高
4. **分歧**：两个角色对同一问题结论相反，提交人类裁决

**优先级规则**：security-reviewer 的 BLOCKER 结论 > 其他角色。

合并建议只允许使用：
- **APPROVE**：无 BLOCKER，HIGH 已修复或已接受，verifier 无补充异议
- **COMMENT**：无 BLOCKER，有 MEDIUM/LOW 需记录，verifier 无补充异议
- **REQUEST CHANGES**：存在 BLOCKER，或 HIGH 未处理且无风险接受
- **NEEDS HUMAN DECISION**：角色间存在分歧，需要人类裁决

**重要约束**：
- 任意评审角色补充了 HIGH 问题 → 禁止直接 APPROVE
- security-reviewer 发现 BLOCKER → 必须 REQUEST CHANGES
- 兼容性风险（api-reviewer 发现旧逻辑被破坏且无回滚方案）→ 必须 REQUEST CHANGES

**Step 20 输出 verifier 最终结论（合并后）**

```
## verifier 最终结论

### 最终合并问题清单
（六个评审角色结论合并后的完整问题清单）

### 驳回记录
（被驳回的问题及驳回理由）

### verifier 补充质疑
（对抗性质疑中发现的新问题）

### 最终评审结论
APPROVE / COMMENT / REQUEST CHANGES / NEEDS HUMAN DECISION
```

---

### 阶段四：修复-再评审循环

**Step 21 修复后验证**

开发者修复后，重新进入 verifier 角色验证：

1. 确认问题是否真正关闭（不只是注释掉或日志打印）
2. 确认修复是否引入新风险
3. 确认测试是否同步更新
4. 复审发现新问题，继续修复循环

**循环停止条件**：
- 无 BLOCKER
- HIGH 已修复或有正式风险接受
- verifier 无补充异议
- 测试覆盖已补充
- MEDIUM/LOW 已进入后续计划

**Step 22 最终收口**

所有问题关闭后，verifier 输出最终收口报告：
```
## 最终收口报告

### 关闭的问题清单
### 残留风险（已知且接受）
### 未覆盖风险（无法验证，需人工确认）
### 最终结论：APPROVE
```

## 6. AI自检清单

### code-reviewer 自检
```
□ 是否从本文档进入执行流程？
□ 是否加载了必读文档？
□ 是否识别了旧逻辑、调用方、配置和回滚路径？
□ 是否逐条检查了本次涉及的陷阱？
□ 每个问题是否有文件/行号/证据/影响/修复建议/验证方式？
□ 是否说明了已验证和未验证项？
```

### quality-reviewer 自检
```
□ 是否检查了方法长度和嵌套层级？
□ 是否识别了重复代码和坏味道？
□ 异常处理是否合理？
□ 是否引入了不必要的依赖？
□ 是否存在难以测试的代码结构？
```

### api-reviewer 自检
```
□ 是否检查了 DTO/VO 注释和命名？
□ 是否复用了既有 DTO 而非新建？
□ 枚举是否端到端（DTO/VO/Entity/Service 使用同一种枚举）？
□ Feign 接口与实现是否一致？
□ 数据库变更是否向后兼容？
□ RESTful 风格是否规范？
```

### security-reviewer 自检
```
□ 是否独立执行了安全检查，未依赖其他评审角色结论？
□ 是否检查了身份认证和权限控制？
□ 是否检查了输入安全（SQL/XSS/SSRF/命令注入）？
□ 是否检查了敏感信息保护？
□ 是否检查了多租户隔离？
□ 是否有新的攻击面未被发现？
□ 是否误报了不存在的安全风险？
```

### performance-reviewer 自检
```
□ 是否检查了 N+1 查询风险？
□ 是否检查了长事务和锁等待风险？
□ 缓存与数据库一致性是否被破坏？
□ 是否有大对象频繁创建？
□ 异步任务是否正确处理失败？
□ MQ 消费是否幂等？
```

### style-reviewer 自检
```
□ 是否遵循 Java/TypeScript 编码规范？
□ 文件编码是否正确（UTF-8 无 BOM）？
□ 注释是否完整（Javadoc、方法级注释）？
□ 是否引入了未使用的导入？
□ 前端代码是否遵循 Vue/uni-app 规范？
```

### test-engineer 自检
```
□ 核心业务逻辑是否有单元测试？
□ 边界条件是否被覆盖？
□ Mock 是否合理？
□ 变更是否同步更新了测试？
□ 删除的逻辑是否删除了对应测试？
□ 测试是否可重复执行且相互独立？
```

### verifier 自检
```
□ 是否对"无问题"通过进行了对抗性质疑？
□ 是否回答了7个强制质疑问题？
□ 是否合并了六个评审角色结论？
□ 驳回问题是否有充分理由？
□ 修复后是否验证了问题真正关闭？
□ 是否确认修复未引入新风险？
□ 最终收口是否完整？
```
