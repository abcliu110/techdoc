# 评审检查清单

> 版本：v2.1
> 日期：2026-07-12
> 定位：评审检查项定义、验收标准
> 适用范围：架构评审、代码评审、设计评审、UI评审、安全评审

**级别说明**：BLOCKER 必须修复后合并；HIGH 默认修复，延期需记录风险接受人、截止时间和补偿措施。

---

## 一、架构评审检查清单

### 架构决策
- 每个决策点是否有 ≥3 个候选方案？
- 每个落选方案是否有明确拒绝理由？
- 是否记录到 ADR？
- 是否有权衡矩阵？

### 模块边界
- 模块职责是否单一？
- 模块间依赖方向是否正确？
- 是否有循环依赖？
- 是否有 ArchUnit 规则？

### 质量属性
- 8 个维度是否都有可度量目标？
  - 正确性/一致性、安全性、可扩展性、性能、可测试性、可审计性、易用性、可理解性
- 每个目标是否有验证方式？

### 陷阱覆盖
- 是否逐项检查了陷阱清单？
- 每个陷阱是否有防线？
- 是否有机器防线 + 人工 checklist？

### 安全红线
- 认证方式、权限模型、租户隔离策略
- SQL 注入防护、XSS 防护、CSRF 防护

### 性能目标
- QPS 目标、响应时间目标、并发用户数、数据规模
- 是否有性能测试计划？

### 风险与缓解
- 是否识别了所有风险？每个风险是否有缓解措施？
- 是否有回滚方案？是否有单点故障？

### 可实施性
- 里程碑划分是否合理？依赖项是否清晰？团队能力是否匹配？

## 二、代码评审检查清单

### 第一部分：陷阱检查

本次 PR 涉及的陷阱（在低代码平台陷阱覆盖矩阵中查询）：
- B1: 租户隔离
- B2: data_scope 正确性
- B3: 权限一致性
- C1: N+1 查询
- H1: SQL 注入
- 其他陷阱：

### 第二部分：多租户与安全（L1 必查）

**SQL 防注入（17-001，H1）**
- 表名是否白名单？
- 列名是否白名单？
- op 是否枚举？
- 用户值是否参数化？

**tenantId fail-fast（17-010，B1）**
- 查询前是否检查 tenantId？
- tenantId 为空是否抛异常？

**异步传递 tenantId（17-011，B1）**
- @Async 方法是否显式传递 tenantId？
- MQ 消息是否包含 tenantId？
- 定时任务是否有租户上下文？

**权限统一判定（17-020，B3）**
- 是否走 AccessView？
- /meta、list、update 鉴权是否一致？

**data_scope AST 层合并（17-004，B2）**
- 是否防止 OR 短路？

### 第三部分：数据库设计（11-数据库）

**标准列（11-012）**
- 是否有 tenant_id、lid (ULID)、revision (乐观锁)？
- 是否有 deleted / deleted_at / delete_token？
- 是否有 create_time/create_by、update_time/update_by？

**软删除唯一约束（11-043，D2）**
- unique 是否用 delete_token？未删除=0，删除=id或唯一值

**索引 tenant_id 前缀（11-040，B1）**
- 核心查询索引格式：(tenant_id, deleted, 业务字段, id)

**禁止项（11-003）**
- 是否有物理外键、触发器、存储过程？

### 第四部分：Java 编码（10-Java）

- 对外接口是否有 Javadoc？（参数、返回值、异常说明）
- 状态是否用枚举，不用魔法值？
- 事务注解是否正确？@Transactional(rollbackFor = Exception.class)
- 是否有乐观锁？revision CAS 更新
- 资源是否正确关闭？try-with-resources
- 是否直接捕获 Exception？不应 catch Exception

### 第五部分：业务逻辑

- 空值是否处理？参数 null 检查、返回值 null 说明
- 异常是否正确分类？BizException、错误码定义
- 枚举是否覆盖所有分支？switch 是否有 default
- 状态机是否完整？状态转换是否校验

### 第六部分：事务与并发

- 事务边界是否合理？事务范围是否最小？外部调用是否在事务外？是否有长事务风险？
- 并发是否安全？是否有竞态条件？CAS 是否正确？是否有死锁风险？

### 第七部分：性能（23-运行时）

- 是否有 N+1 查询？（23-030，C1）循环中是否有查询？是否改为批量查询？
- 是否有深分页？（23-082，C4）offset 是否过大？
- 循环中是否有 IO？网络调用、文件操作
- 是否有内存泄漏风险？集合无限增长、资源未释放

### 第八部分：测试（14-测试）

- 关键代码是否有测试？L1/L2 代码要求 100% 覆盖
- 是否有陷阱相关测试？双租户隔离测试（14-031，B1）、SQL 注入测试、权限一致性测试
- 是否有边界值测试？null 输入、空字符串、边界数值
- 是否有并发测试？CAS 冲突测试（14-041，F1）

### 第九部分：可观测性（27-可观测）

- 关键操作是否有审计日志？创建/修改/删除
- 错误是否有 traceId？
- 敏感信息是否脱敏？日志不含密码/token（17-051）

### 第十部分：代码质量

- 命名是否清晰？注释是否充分？代码是否易读？
- 方法是否过长？（>50 行预警）
- 嵌套是否过深？（>3 层预警）
- 是否有重复代码？

## 三、设计评审检查清单

### 接口契约
- 接口是否完整定义？请求参数、返回值、错误码、异常
- 接口是否有 Javadoc？参数校验规则是否明确？幂等性是否说明？并发控制是否说明？

### 数据模型
- 实体是否定义完整？字段列表、字段类型、字段约束
- 关系是否明确？一对一、一对多、多对多
- 索引策略是否合理？分区策略是否考虑？

### 状态机
- 状态是否枚举定义？状态转换是否完整？非法转换是否阻断？状态图是否提供？

### 业务规则
- 业务规则是否明确？校验规则是否完整？计算公式是否正确？边界条件是否说明？

### 异常处理
- 异常分类是否合理？错误码是否定义？降级策略是否考虑？重试策略是否说明？

### 事务设计
- 事务边界是否合理？隔离级别是否说明？分布式事务是否考虑？

### 性能设计
- 性能目标是否量化？缓存策略是否考虑？批量操作是否支持？分页是否合理？

### 安全设计
- 认证方式是否明确？权限控制是否完整？敏感数据是否加密？审计日志是否考虑？

## 四、UI 评审检查清单

### 可访问性（WCAG 2.1 AA）
- 是否支持键盘导航？Tab 键顺序合理、Enter/Space 触发操作、Esc 关闭弹窗
- 是否有焦点指示器？
- 颜色对比度是否达标？（≥4.5:1）
- 是否有替代文本？图片 alt、按钮 aria-label
- 表单是否可访问？label 关联 input、错误提示清晰
- 是否支持屏幕阅读器？

### 交互设计
- 交互是否一致？按钮位置、操作反馈、错误提示
- 操作是否可撤销？删除确认、撤销功能
- 加载状态是否提示？Loading 动画、骨架屏
- 错误提示是否友好？"人话"提示，具体到字段

### 前端安全（G4）
- 是否防 XSS？用户输入转义、富文本净化、危险属性过滤
- 是否防 CSRF？CSRF token
- 敏感数据是否脱敏？手机号、身份证号
- 附件预览是否隔离？

### 响应式设计
- 是否支持移动端？触摸友好、布局自适应
- 是否支持多分辨率？是否考虑横竖屏？

### 性能优化
- 首屏加载时间？（<3 秒）
- 是否懒加载？图片懒加载、组件懒加载
- 是否虚拟滚动？（大列表）
- 是否代码分割？

## 五、安全评审检查清单

### OWASP Top 10
- A01: 访问控制失效 — 是否有权限校验？是否防止越权？
- A02: 加密失效 — 敏感数据是否加密？传输是否加密（HTTPS）？
- A03: 注入 — SQL 注入、XSS、命令注入防护
- A04: 不安全设计 — 是否有威胁建模和安全需求？
- A05: 安全配置错误 — 默认密码是否修改？错误信息是否泄露？
- A06: 易受攻击的组件 — 依赖是否有已知漏洞？是否定期更新？
- A07: 认证失败 — 密码策略、登录失败锁定、会话管理
- A08: 软件和数据完整性失效 — 是否验证签名？是否防止篡改？
- A09: 日志记录和监控失效 — 是否记录安全事件？是否有告警？
- A10: 服务器端请求伪造（SSRF） — URL 是否白名单？是否防止内网探测？

### 多租户安全
- 数据隔离是否完整？资源隔离是否完整？跨租户访问是否阻断？

### 审计与合规
- 关键操作是否审计？审计日志是否防篡改？个人信息是否合规处理？数据保留期是否明确？

## 六、快速评审检查表（5 分钟版）

适用于小 PR 的快速评审。发现安全边界、核心风险或需要记录的非阻断问题时，必须转完整评审。

- CI 是否通过？
- 测试覆盖是否达标？
- 是否涉及安全边界？（如是，转完整评审）
- 是否有 SQL 拼接？是否有 tenantId 检查？是否有权限检查？
- 是否有明显性能问题？代码是否易读？

快速评审结论：
- APPROVE：快速评审通过，无安全边界和核心风险
- 转完整评审：不在快速评审内给出合并结论

---

## 评审结论门禁

| 结论 | 条件 |
|------|------|
| APPROVE | 无 BLOCKER，验证充分 |
| COMMENT | 无 BLOCKER，有需记录的非阻断问题 |
| REQUEST CHANGES | 存在 BLOCKER 或未接受的 HIGH |
| NEEDS HUMAN DECISION | 需要负责人裁决 |
