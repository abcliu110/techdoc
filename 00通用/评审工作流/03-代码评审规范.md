# 代码评审规范

> 版本：v2.2
> 日期：2026-07-12
> 定位：代码评审规则、陷阱检查、安全红线
> 适用范围：所有代码变更评审

## 1. 风险分层

### 1.1 代码层级

| 层级 | 范围 | 评审强度 |
|------|------|---------|
| L1 安全边界 | SQL拼装、权限、租户、认证、支付凭证、敏感数据 | 逐行 + 双人复核 |
| L2 核心状态 | 订单、库存、支付、MQ、事务、状态机、数据迁移 | 逐行 + 场景验证 |
| L3 核心业务 | Service、领域规则、计费、同步、外部接口 | 重点逐行 + 影响分析 |
| L4 普通业务 | CRUD、查询、后台配置、低风险接口 | 重点抽查 + 工具扫描 |
| L5 低风险代码 | DTO、VO、枚举、工具类、测试辅助 | 抽查 + 规范扫描 |

无法判断层级时，按更高风险处理。

### 1.2 严重级别

| 级别 | 合并策略 | 示例 |
|------|---------|------|
| BLOCKER | 禁止合并，必须修复 | SQL注入、越权、跨租户、资金/库存错误、破坏旧逻辑 |
| HIGH | 默认修复后合并；延期需风险接受 | N+1、并发风险、兼容性不明、测试缺失 |
| MEDIUM | 建议本次修复；可带记录合并 | 复杂度高、错误处理不足、日志不完整 |
| LOW | 可选优化 | 命名、注释、局部可读性 |

**关键**：不是所有规范违反都阻断，只有造成红线风险或实质影响的才阻断。

## 2. 安全红线（BLOCKER）

### 2.1 多租户隔离

- tenantId 必须 fail-fast（为空立即抛异常）
- 查询必须强制注入 tenant_id 条件
- 异步/MQ/定时任务必须显式传递 tenantId
- 跨租户读取或写入 → BLOCKER

### 2.2 权限控制

- 必须走统一鉴权入口
- 禁止绕过鉴权路径
- /meta、list、update 鉴权必须一致
- 权限绕过 → BLOCKER

### 2.3 SQL/输入安全

- 用户输入必须参数化
- 表名/列名必须白名单
- op 操作符必须枚举
- SQL注入 → BLOCKER

### 2.4 金额/库存/订单

- 金额计算必须使用 BigDecimal
- 库存扣减必须使用乐观锁或悲观锁
- 订单状态流转必须有状态机控制
- 金额/库存/订单状态错误 → BLOCKER

### 2.5 幂等性

- 支付、订单、MQ消费、重复提交必须可重入
- 无幂等的写操作 → BLOCKER

### 2.6 敏感信息

- 密钥、token、支付凭证禁止硬编码
- 手机号、身份证、token 禁止明文日志
- 密钥泄露 → BLOCKER

## 3. 陷阱检查

### 3.1 陷阱类型映射

根据变更内容，对照陷阱矩阵（见相关文档）检查对应风险：

| 变更内容 | 需检查陷阱 |
|----------|-----------|
| 查询/Mapper/报表 | B1 租户隔离、C1 N+1查询 |
| 权限/菜单/角色 | B3 权限一致性 |
| 订单/支付/库存 | F1 并发幂等 |
| MQ/定时任务/同步 | B1 租户上下文丢失 |
| DDL/数据迁移 | D2 软删除连带语义 |
| 前端输入/富文本 | XSS、CSRF、输入校验 |

### 3.2 常见陷阱速查

详见陷阱覆盖矩阵：`../工程规范/低代码平台规范/06-陷阱覆盖矩阵.md`

主要陷阱 ID 速查：
- **B1 租户隔离**：tenantId 是否 fail-fast？异步是否传递 tenantId？
- **C1 N+1查询**：循环中是否有查询？是否改为批量查询？
- **F1 并发幂等**：金额/库存扣减是否有锁？是否有重复提交保护？
- **D2 软删除唯一约束**：unique 是否用 delete_token？

## 4. 兼容优先原则

既有系统默认存在隐性契约。

### 4.1 必须确认

- 旧入口是否保留？
- 旧调用方是否兼容？
- 历史数据是否兼容？
- 配置、定时任务、消息链路是否受影响？
- 是否有新逻辑启用条件？
- 是否有回滚方案？

### 4.2 禁止行为

- 无证据删除或重写旧逻辑
- 修改接口契约但不说明兼容性
- 数据库变更无回滚方案

## 5. 验证要求

| 风险类型 | 验证方式 |
|----------|---------|
| 租户隔离 | 双租户隔离测试 |
| SQL注入 | 恶意输入测试 |
| 权限一致性 | 多角色权限测试 |
| 并发安全 | 并发场景测试 |
| 幂等性 | 重复请求测试 |

L1/L2 代码覆盖率要求详见：`06-工具与自动化规范.md`

## 6. 评审输出要求

每个问题必须包含：
- **位置**：文件、行号
- **证据**：代码片段或规范依据
- **影响**：为什么造成风险
- **级别**：BLOCKER/HIGH/MEDIUM/LOW
- **建议**：最小修复方向
- **验证**：如何证明修复有效
