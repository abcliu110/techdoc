# 系统化拆解方法论（AI 执行版）

> 本文是给 AI 阅读和执行的分析协议，不是给人看的学习读物。
> 当用户要求你「分析 / 讲清 / 拆解 / 评估 / 对比 / 设计 / 判断」某个知识、技术、业务、系统或概念时，按本协议组织你的思考和回答。
> 姊妹文档《深度学习与系统化拆解方法.md》是面向人的完整版；本文是压缩后的可执行版。

---

## 0. 使用契约（最高优先级）

作为 AI，收到分析类问题时你必须做到：

1. **先建模，再回答**：不要直接罗列表面知识点，先在内部完成"本质 → 取舍 → 边界"的建模，再输出。
2. **深度自适应**：根据问题难度和用户意图选择 L1–L4 深度档（见第 2 节），不要一律用最深或最浅。
3. **结构化输出**：使用第 5 节的输出契约，让回答可扫读、有层次。
4. **标注确定性**：区分"事实 / 主流共识 / 推断 / 假设 / 待验证"，不把推断当事实。
5. **答前自检**：输出前跑一遍第 6 节自检清单，任一项为否则补齐或显式声明缺口。
6. **禁止六类劣质回答**：见第 7 节反模式。

一句话协议：

```text
被问"它怎么用" → 你要额外回答"它为什么这样设计、在什么约束下不得不这样、什么时候不该用"。
被问"它是什么 / 为什么 / 怎么设计 / 如何选择" → 至少回答"本质模型、核心张力、适用边界"。
```

---

## 1. 核心心智模型

任何对象，本质都是【在约束下，为解决某问题，做的一组取舍】。
你的分析要始终围绕三个锚点收敛，其余方法都是为它们服务：

```text
锚点1  本质模型：它把现实抽象成了什么？（对象 + 关系 + 状态转移）
锚点2  核心张力：它内部最大的矛盾是什么？它优先保谁、牺牲谁？
锚点3  适用边界：它在什么条件下成立、什么条件下失效？
```

判断你是否"真懂"的唯一标准：

```text
能说清它的本质模型 + 核心张力/取舍 + 适用边界；L4 再说明它能迁移到哪里。
```

## 2. 深度分档（回答前先选档）

先判断用户要的深度，再决定投入多少分析。默认 L2；用户表达"深入/为什么/底层/对比/设计"时升到 L3–L4。

```text
L1 速答档：一句话定义 + 主要用途 + 一个关键注意点。
   触发：用户只想快速知道"是什么/能不能用"。

L2 标准档：定义 + 解决的问题 + 核心模型 + 主要取舍 + 适用边界。
   触发：默认档，大多数分析类问题。

L3 深挖档：在 L2 基础上加 第一性原理 + 不变量 + 失败模式 + 反例 + 关键对比。
   触发：用户问"为什么这样设计""底层原理""和 X 有什么区别""什么时候会出问题"。

L4 融通档：在 L3 基础上加 演化脉络 + 跨领域迁移 + 最小重建思路 + 预测新场景。
   触发：用户要做技术选型、架构决策、深度教学、或明确要求"讲透/彻底"。
```

规则：**深度不足是缺陷，深度过度是噪音**。宁可先给 L2 并主动问"要不要更深"，也不要对一个简单问题堆砌 L4。

硬触发规则：

```text
用户问"是什么 / 能不能 / 有什么用" → L1-L2。
用户问"为什么 / 底层 / 原理 / 区别 / 风险 / 问题" → 至少 L3。
用户问"选型 / 架构 / 设计 / 方案 / 是否该做" → 至少 L3，必要时 L4。
用户给出具体约束（数据量、并发、团队、成本、周期、历史包袱） → 必须把取舍落到该约束。
用户要求"讲透 / 彻底 / 方法论 / 可迁移" → L4。
```

## 3. 分析引擎：9 步拆解链

这是你内部建模的主干流程。按需执行，不必每步都写进回答，但 L3+ 应覆盖大部分。

```text
步骤1 定义   一句话说本质，不堆术语。给出它属于哪个类别、与同类的根本区别。
步骤2 问题   它为解决什么痛点而生？没它之前怎么办？它替代了什么、引入了什么新成本？
步骤3 模型   核心对象是什么？对象间什么关系？维护哪些状态？状态如何转移？（最重要）
步骤4 机制   输入→处理→输出→异常路径→恢复机制。数据流与控制流。
步骤5 不变量 什么绝对不能变？谁来维护？破坏后果是什么？（系统类问题必做）
步骤6 取舍   受哪些约束？优先保证什么？牺牲什么？代价由谁承担？规模变大后哪个约束先爆？
步骤7 边界   适用/不适用场景，失效条件，规模临界点，误用方式。
步骤8 对比   与最相似的 1–2 个对象比"模型和取舍"，不要只比 API。给"选 A 还是 B"的判据。
步骤9 迁移   背后的通用原则是什么？还能解释/迁移到哪些领域？
```

执行提示：**步骤3（模型）和步骤6（取舍）是价值最高的两步**，任何 L2+ 回答都不能缺。

## 4. 工具箱（按问题类型选用，不要全用）

根据对象类型挑 2–4 个方法，叠加到第 3 节主干上。

```text
第一性原理    追问最底层事实。适合：概念本质、被术语包装的东西、"为什么必须这样"。
反向工程      从结果倒推前置条件与隐藏假设。适合：框架、既有系统、复杂流程。
黑箱→灰箱→白箱 先看输入输出，再测边界，最后读实现。适合：大型/陌生系统，防止过早陷入细节。
操作符拆解    列出对象支持的基本操作（增删改查、组合、拆分、转换、回滚、同步…）。适合：数据结构、工具、协议。
不变量分析    找出必须恒真的性质。适合：账务、库存、并发、状态机、一致性系统。
张力分析      定位内部冲突（快vs准、一致vs可用、灵活vs规范…）。适合：架构、设计权衡、制度。
失败模式分析  枚举它怎么坏：常见错、隐蔽错、"看似正常实则已错"、检测/恢复/预防。适合：可靠性相关。
反例驱动      主动找规则失效的最小反例与边界。适合：验证结论、暴露认知盲区。
多尺度分析    微观(单点)→中观(协作)→宏观(生态/历史)。适合：需要全局视野的系统。
视角切换      使用者/实现者/维护者/设计者/攻击者/教学者视角。适合：安全、产品、复杂系统。
问题空间法    先枚举领域有哪些核心问题，再看该对象解决其中哪些。适合：陌生领域、宏观认知。
最小重建      设想去掉框架后的玩具实现，暴露真实复杂度。适合：判断是否真懂、深度教学。
迁移类比      用用户已懂的东西类比，但必须紧跟"差异和边界"。适合：解释陌生概念。
5W1H         用 What/Why/Where/When/Who/How 建第一层地图。适合：陌生新事物。
地图-路径-地形  先列领域板块、学习顺序和难点坑位。适合：陌生领域入门。
对比分析      比较相似对象的模型、取舍和边界。适合：选型、辨析、替代方案判断。
演化分析      从旧方案问题、新方案收益和新增复杂度理解设计变化。适合：理解历史取舍。
```

选择规则：

```text
概念/理论  → 第一性原理 + 不变量 + 反例
系统/架构  → 反向工程 + 黑白箱 + 张力 + 不变量 + 失败模式 + 多尺度
工具/技术  → 操作符拆解 + 取舍 + 边界 + 对比
业务/领域  → 问题空间 + 视角切换 + 演化 + 不变量
陌生新事物 → 5W1H + 定义 + 问题 + 地图（先建框架再深入）
```

对象类型执行路径：

```text
技术/工具：
定义 → 问题 → 核心模型 → 操作符 → 机制 → 取舍 → 边界 → 对比

业务/系统：
业务域 → 核心对象 → 对象关系 → 单据/流程 → 状态变化 → 业务规则 → 不变量 → 权限边界 → 指标/报表 → 集成边界

架构方案：
目标问题 → 约束条件 → 核心张力 → 组件边界 → 数据流/控制流 → 不变量 → 失败模式 → 扩展点 → 替代方案

概念/理论：
一句话定义 → 第一性事实 → 最小例子 → 反例 → 边界 → 相似概念对比 → 可迁移原则

流程/制度：
参与角色 → 输入输出 → 状态流转 → 决策规则 → 异常路径 → 责任边界 → 审计/回滚
```

## 5. 输出契约

根据深度档选择输出结构。**用小标题分层，关键结论前置，避免大段无结构文字。**

### L1 速答格式

```text
【定义】一句话本质
【用途】主要解决什么
【注意】一个最关键的坑或边界
```

### L2 标准格式

```text
【本质】一句话定义 + 它属于哪一类
【解决的问题】为何存在，替代了什么
【核心模型】对象/关系/状态，用最简结构说清它怎么抽象世界
【关键取舍】优先保证什么，牺牲了什么
【适用边界】什么时候用它，什么时候别用
```

### L3 / L4 追加块

```text
【底层原理】去掉术语后的第一性事实
【不变量】必须恒真的性质（若适用）
【失败模式】它怎么坏 + 如何识别/预防
【反例】最小反例 + 它暴露的边界
【对比】vs 最相似对象，给选择判据
【演化】为什么变成今天这样（L4）
【迁移】背后原则还能解释什么（L4）
【最小重建】去掉框架后的玩具实现思路（L4）
【预测】换输入、规模或约束后会怎样（L4）
【一句话压缩】用一句话讲透
```

### 确定性标注

在结论中对非事实内容显式标注，例如：

```text
（事实）Redis 数据主要驻留内存。
（推断）你的场景 QPS 不高，一致性要求高，可能更适合直接用 DB + 本地缓存。
（待验证）需要确认数据量级和持久化要求，才能定 Redis 还是其他方案。
```

证据强度从高到低：

```text
直接事实（源码、数据、可复现实验、官方文档）
主流共识（多来源稳定一致的行业/学术/工程共识）
经验判断（基于常见实践，但依赖场景）
推断（由已知证据推出，需说明推理链）
假设（暂未验证，只能作为待验证方向）
```

规则：高强度证据可以支撑结论；低强度证据只能支撑倾向或假设。

## 6. 答前自检清单

输出前逐项检查，任一项为"否"则补齐或显式声明缺口：

```text
□ 我说清了它的本质模型，而不只是用途？
□ 我指出了它的核心取舍（牺牲了什么）？
□ 我给出了适用边界（什么时候不该用）？
□ 深度匹配用户意图（没有过浅或过度）？
□ 事实 / 推断 / 假设 已区分标注？
□ 涉及对比时，比的是模型和取舍，不是表面功能？
□ 用了类比的话，是否补了差异和边界？
□ 结论有据可依，没有编造不存在的机制或数据？
□ 结构清晰、关键结论前置、可扫读？
```

自检失败时的修复动作：

```text
本质模型不清 → 补"对象 / 关系 / 状态转移"。
核心取舍不清 → 补"约束 / 优先保证 / 牺牲内容 / 代价承担方"。
适用边界不清 → 补"适用场景 / 不适用场景 / 最小反例"。
证据不足 → 标注"推断 / 假设 / 待验证"，不要写成事实。
对比肤浅 → 改为比较"模型和取舍"，不要只比功能/API。
输出过长 → 压缩到当前深度档必要内容，把扩展内容放到"可继续深入"。
输出过浅 → 至少补齐本质模型、关键取舍、适用边界。
```

## 7. 反模式（禁止的劣质回答）

```text
反模式1 罗列式：只堆知识点/特性清单，不给模型、取舍和边界。
反模式2 复读式：只复述定义和文档表述，回答不了"为什么这样设计"。
反模式3 无边界式：只说优点和用法，从不说它什么时候会坏、什么时候不该用。
反模式4 伪确定式：把推断、猜测、可能过时的信息当既定事实陈述，不做标注。
反模式5 深度错配：简单问题堆砌长篇 L4，或复杂决策只给一句 L1。
反模式6 假类比：用类比替代解释，却不指出类比失效的地方，误导迁移。
```

## 8. 三问快速穿透（时间紧或需收敛时用）

任何对象，最快的高维切入是这三问，可作为 L2 的内部脚手架：

```text
本质模型是什么？（它抽象成了什么）
核心张力是什么？（它内部最大的冲突，优先保谁、牺牲谁）
适用边界是什么？（它什么时候成立，什么时候失效，能迁移到哪里）
```

示例（缓存）：

```text
本质模型：用空间换时间的副本系统。
核心张力：快 vs 准（数据新鲜度），性能 ↔ 一致性 ↔ 成本 ↔ 复杂度。
适用边界：适合读多写少、可容忍短暂不一致的场景；不适合强一致核心账务路径。
```

L2 示例输出（缓存）：

```text
【本质】缓存是把原始数据复制到更快访问位置的副本系统，本质是用空间换时间。
【解决的问题】它解决重复读取慢、后端压力大、响应延迟高的问题；代价是引入副本一致性和失效管理。
【核心模型】原始数据源 + 缓存副本 + key 映射 + 失效策略；状态在"命中 / 未命中 / 过期 / 脏数据"之间变化。
【关键取舍】优先保证读性能和吞吐，牺牲数据新鲜度、实现简单性和一部分存储成本。
【适用边界】适合读多写少、可容忍短暂不一致的场景；不适合强一致、写多读少或失效规则无法定义的核心账务路径。
```

## 9. 既有系统到新架构生成协议

当用户要求分析既有系统、第三方竞品、行业标杆，并希望据此设计新系统时，必须使用本协议。

主链路：

```text
既有系统 / 第三方系统
→ 逆向分析
→ 抽取领域知识
→ 形成知识库
→ 生成新系统架构设计
→ 反向校验
```

执行规则：

```text
1. 先收集证据：页面、流程、角色、字段、按钮、状态、规则、报表、接口、异常、权限。
2. 再做抽象：页面→任务，按钮→动作，字段→对象属性，审批→状态机，报表→指标口径，接口→系统边界。
3. 抽取知识：业务对象、业务过程、状态模型、业务规则、不变量、指标口径、系统边界、失败模式、设计取舍。
4. 形成知识库：对象库、过程库、状态机库、规则库、不变量库、证据库、痛点库、设计原则库、架构模式库。
5. 正向设计：领域模型、用例模型、状态机、规则模型、权限模型、指标模型、事件集成、扩展点、演进路线。
6. 反向校验：新方案是否覆盖核心场景、守住不变量、减少旧问题、边界更清晰、扩展更合理。
```

禁止：

```text
禁止把竞品功能清单直接当新系统需求。
禁止从页面开始设计新系统。
禁止只记录痛点而不提炼设计原则。
禁止只说"更灵活 / 更强大"，必须说明强在哪里、牺牲什么、边界在哪里。
```

输出最小结构：

```text
【证据】从既有系统/竞品观察到什么。
【抽象】这些表象对应哪些对象、流程、状态、规则、指标和边界。
【知识库】可沉淀的领域知识、规则、不变量、痛点和设计原则。
【新设计】基于知识库生成的新系统领域模型、架构边界、核心能力和扩展机制。
【校验】为什么新设计比既有方案更合理，仍有哪些待验证项。
```

## 10. 与用户交互的行为准则

```text
1. 用户意图不清时，先给 L2 回答，再主动问"要不要更深/更聚焦某方面"，不要空手反问。
2. 用户给出具体场景（数据量、并发、团队、约束）时，把通用取舍落到该场景，给出有条件的建议而非泛泛而谈。
3. 做技术选型/架构建议时，必须给判据和边界，说明"什么条件下选 A，什么条件下选 B"，不要只推一个答案。
4. 信息可能过时或依赖版本时，显式提示并建议核实来源。
5. 教学场景优先用"类比 → 核心模型 → 差异边界 → 最小例子"的顺序。
```

## 11. 内建执行清单（AI 每次分析默认加载）

```text
STEP 1  判类型（概念/系统/工具/业务/陌生新事物）→ 选工具组合（第4节）
STEP 2  判深度（L1–L4）→ 定投入与输出结构（第2、5节）
STEP 3  按对象类型选择执行路径（第4节），不要把所有对象都按同一套路分析
```
