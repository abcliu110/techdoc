# 世界级系统模型设计重磅补充：从业务能力到 AI 生成系统

## 1. 这篇文档解决什么问题

数据库模型和 UI 模型只是系统设计的两个结果。真正决定系统质量的是更上层的建模能力：

- 是否理解业务能力。
- 是否找到稳定领域对象。
- 是否识别生命周期和状态机。
- 是否设计了可追溯的数据事实。
- 是否让 UI 围绕任务和对象组织。
- 是否能从老系统、竞品、行业巨头中提炼模式。
- 是否能把这些知识沉淀给 AI，用于设计新系统。

因此，世界级系统设计不是“先画页面，再建表，再写接口”，而是：

```text
业务能力
  -> 领域模型
  -> 数据模型
  -> UI 任务模型
  -> 接口契约
  -> 报表模型
  -> 测试模型
  -> 知识库沉淀
  -> 新系统生成
```

底层思想：

> 系统设计的本质，是把复杂业务转化为稳定、可解释、可扩展、可验证的模型。

## 2. 业务能力建模：比数据库和 UI 更上层的设计

### 方法

先把系统拆成业务能力，而不是拆成页面或表。

业务能力示例：

- 客户管理。
- 商品管理。
- 价格管理。
- 订单履约。
- 库存核算。
- 会员权益。
- 支付结算。
- 退款冲正。
- 报表分析。
- 权限治理。
- 组织管理。

每个业务能力都要回答：

```text
它解决什么业务问题？
它的输入是什么？
它的输出是什么？
它影响哪些对象？
它产生哪些事实？
它依赖哪些规则？
它需要哪些报表？
它有哪些异常和逆向流程？
```

### 为什么重要

页面会变，表会变，技术会变，但业务能力相对稳定。一个企业系统真正能复用的不是表结构，而是业务能力划分。

### 解决的问题

- 避免按菜单或数据库表拆系统。
- 避免模块边界混乱。
- 避免一个模块承担多个业务责任。
- 帮助 AI 理解“这个系统到底能做什么”。

### 带来的问题

- 前期需要更多业务抽象。
- 需要区分“能力”和“功能入口”。
- 能力边界划错会影响后续架构。

### 蕴含思想

> 业务能力是系统的骨架，数据库和 UI 是能力的表达。

## 3. 领域对象分级：核心对象、支撑对象、技术对象

### 方法

把对象分为三类：

| 类型 | 含义 | 示例 |
|---|---|---|
| 核心对象 | 直接承载业务价值 | 订单、库存、会员、账户、商品 |
| 支撑对象 | 支撑核心业务运行 | 分类、标签、规则、组织、渠道 |
| 技术对象 | 为实现服务，不应直接暴露给用户 | 同步任务、消息记录、缓存、导入批次 |

### 为什么重要

很多系统失败，是因为把技术对象当业务对象暴露给用户，或把核心对象藏在技术结构后面。

### 解决的问题

- 明确系统真正要保护和演进的对象。
- 避免 UI 直接暴露技术表。
- 避免报表围绕技术字段组织。
- 帮助数据库模型分清主次。

### 带来的问题

- 需要判断对象是否真的有独立生命周期。
- 某些对象既有业务属性又有技术属性，需要拆分视角。

### 蕴含思想

> 核心对象决定系统价值，支撑对象决定扩展能力，技术对象决定工程质量。

## 4. 时间建模：企业系统最容易被低估的设计

### 方法

所有关键业务都要区分时间类型：

- 创建时间。
- 修改时间。
- 业务发生时间。
- 生效时间。
- 失效时间。
- 入账时间。
- 审核时间。
- 支付时间。
- 统计时间。
- 同步时间。

### 为什么重要

企业系统里，“什么时候发生”往往比“现在是什么”更重要。财务、库存、报表、审批、会员权益都会受到时间口径影响。

### 解决的问题

- 跨日报表不准。
- 退款按哪个日期统计不清楚。
- 规则变更后历史计算无法还原。
- 库存、财务、经营报表口径冲突。

### 带来的问题

- 字段更多。
- 报表口径需要明确。
- 开发不能只用 `created_at` 解决所有时间问题。

### 蕴含思想

> 时间不是一个字段，而是一组业务口径。

## 5. 逆向流程建模：退款、反审、反结账、冲正

### 方法

设计正向流程时，同时设计逆向流程：

```text
下单 -> 取消
支付 -> 退款
审核 -> 反审
结账 -> 反结账
入库 -> 红冲/退货
发券 -> 作废/回收
扣库存 -> 释放/冲正
```

每个逆向流程必须回答：

- 是否允许。
- 何时允许。
- 谁允许。
- 影响哪些数据。
- 是否生成反向流水。
- 是否影响报表。
- 是否通知外部系统。
- 是否能再次正向操作。

### 为什么重要

低质量系统通常只设计正向流程，逆向流程靠补丁实现。ERP、支付、库存、会员权益系统真正复杂的是逆向、异常和补偿。

### 解决的问题

- 退款后权益、库存、报表不一致。
- 反审后下游单据无法处理。
- 反结账后金额和流水解释不清。
- 人工改数据破坏审计链。

### 带来的问题

- 状态机更复杂。
- 流水和事件必须支持冲正。
- UI 要解释逆向操作影响范围。

### 蕴含思想

> 成熟系统不是只会前进，而是能安全、可解释地后退。

## 6. 事实、状态、视图三分法

### 方法

把数据分成三种：

| 类型 | 含义 | 示例 |
|---|---|---|
| 事实 | 已经发生且不应随意修改 | 支付成功、库存扣减、积分发放 |
| 状态 | 事实累积后的当前结果 | 订单已完成、库存余额、会员等级 |
| 视图 | 为用户或报表组织的展示结果 | 订单详情、会员画像、销售看板 |

### 为什么重要

很多系统把状态当事实，把视图当模型，导致历史无法追溯、报表无法解释、UI 改动影响数据库。

### 解决的问题

- 区分不可变事实和可变状态。
- 避免为了页面展示污染底层模型。
- 支持从事实重建状态和视图。
- 支持审计、回放和修复。

### 带来的问题

- 需要更多数据层次。
- 需要明确哪些数据可以修改，哪些只能冲正。

### 蕴含思想

> 事实记录过去，状态服务现在，视图服务使用。

## 7. 语义层：让报表、接口和 AI 使用同一套业务语言

### 方法

建立统一语义层，定义：

- 指标名称。
- 指标公式。
- 维度定义。
- 枚举含义。
- 时间口径。
- 过滤规则。
- 数据来源。
- 适用范围。

示例：

```text
销售额 = 已完成订单的商品成交金额，不含退款，不含储值充值
实收额 = 实际收到的支付金额，含现金、微信、支付宝、会员余额
退款额 = 统计期间完成退款的金额，按退款完成时间归属
```

### 为什么重要

没有语义层时，同一个指标在不同页面、报表、接口、AI 回答中会出现不同口径。

### 解决的问题

- 报表口径不一致。
- 业务和研发对字段理解不同。
- AI 检索到数据但不知道含义。
- 新系统设计缺少统一语言。

### 带来的问题

- 需要持续治理。
- 指标变更需要版本管理。
- 口径争议需要业务负责人确认。

### 蕴含思想

> 数据只有被赋予稳定语义，才真正成为知识。

## 8. 复杂度预算：不是所有企业级方法都要一次上

### 方法

每个设计都评估复杂度预算：

```text
业务复杂度是否足够高？
客户规模是否需要？
团队能否维护？
是否有明确收益？
是否可以后续演进？
是否存在更简单替代？
```

### 为什么重要

世界级设计不是把所有高级方法堆上去。对小系统过度设计，会比简单设计更糟糕。

### 解决的问题

- 防止过度架构。
- 防止为了未来可能性牺牲当前交付。
- 让架构选择有边界。

### 带来的问题

- 需要判断什么时候该简单，什么时候该复杂。
- 某些未来扩展如果低估，后期迁移成本会高。

### 蕴含思想

> 架构不是越复杂越高级，而是在正确的位置承载正确的复杂度。

## 9. 设计债务：承认并记录没有解决的问题

### 方法

每次设计都记录：

- 当前没有解决的问题。
- 为什么暂时不解决。
- 未来触发重构的条件。
- 当前方案的风险。
- 迁移路径。

示例：

```text
当前不引入规则引擎，因为只有两种积分规则。
触发条件：规则超过 5 类，或需要按租户配置，或规则需要业务人员发布。
迁移路径：将当前硬编码规则抽象为版本化规则表。
```

### 为什么重要

没有完美设计。世界级系统不是没有债务，而是知道债务在哪里、为什么存在、什么时候偿还。

### 解决的问题

- 防止后人误以为当前设计是最终形态。
- 防止重复讨论已拒绝方案。
- 支持系统渐进演进。

### 带来的问题

- 需要纪律记录。
- 需要定期回顾，否则债务文档会过期。

### 蕴含思想

> 成熟架构不是消灭妥协，而是管理妥协。

## 10. 企业级 UI 的高级形态：任务流、对象流、异常流

### 方法

UI 不只设计页面，还要设计三种流：

| 流 | 解决的问题 | 示例 |
|---|---|---|
| 任务流 | 用户如何完成目标 | 新增订单、审核采购、处理退款 |
| 对象流 | 对象如何跨页面被理解 | 订单从列表到详情到关联单据 |
| 异常流 | 失败后如何恢复 | 支付失败、库存不足、审批驳回 |

### 为什么重要

很多 UI 看起来完整，但用户一旦遇到异常就不知道下一步。企业系统的体验质量主要由异常流决定。

### 解决的问题

- 页面孤立。
- 任务中断。
- 异常无恢复路径。
- 用户不知道结果是否可信。

### 带来的问题

- 设计成本更高。
- 需要产品、前端、后端共同设计。
- 需要测试覆盖异常路径。

### 蕴含思想

> 页面是静态结构，流程才是真实体验。

## 11. 评审设计时必须问的十个重磅问题

```text
1. 这个设计保护了哪些业务事实？
2. 它让哪些未来变化变容易了？
3. 它让哪些事情变复杂了？
4. 不这样设计，三年后会付出什么代价？
5. 这个模型能否解释历史？
6. 这个 UI 能否解释状态和异常？
7. 这个报表能否追溯到事实？
8. 这个设计是否适合当前客户规模和团队能力？
9. 它是否可以渐进演进，而不是一次性豪赌？
10. AI 以后根据这套知识生成新系统时，会不会误解它？
```

## 12. 面向 AI 的知识库沉淀格式

AI 不是靠长文档自动变聪明，而是靠结构化、可检索、可复用的知识。

每个重要设计都应沉淀为知识卡：

```yaml
type: model_decision
domain: 会员权益
design: 账户当前值与权益流水分离
problem_solved:
  - 当前权益可快速查询
  - 历史变化可追溯
  - 退款和反结账可冲正
why:
  - 当前值表达结果
  - 流水表达证据
tradeoffs:
  - 写入复杂度增加
  - 需要幂等和对账
enterprise_standard:
  - 账户表 + 流水表 + 业务来源单据 + 冲正流水
ui_impact:
  - 详情页可展示权益变化历史
  - 异常时可解释权益来源
report_impact:
  - 可按权益来源、门店、时间统计
avoid_when:
  - 一次性简单优惠且无需追溯
verification:
  - 当前值可由流水解释
  - 退款生成反向流水
  - 报表能从流水聚合
```

### 为什么重要

只有这样，AI 才能在新系统设计时检索到“设计理由”和“适用边界”，而不是只检索到“某张表怎么建”。

### 蕴含思想

> 给 AI 的知识库不是资料堆积，而是设计判断的结构化记忆。

## 13. 从竞品到自有系统的转化公式

```text
观察竞品现象
  -> 识别它解决的问题
  -> 提炼它使用的业务抽象
  -> 分析它的代价和边界
  -> 判断是否适合自己
  -> 转化为自己的模型
  -> 用自己的业务场景验证
  -> 沉淀为知识卡
```

示例：

```text
竞品现象：
金蝶/用友类 ERP 普遍使用“单据 + 单据明细 + 审批 + 下推 + 反审”。

解决的问题：
企业业务需要把采购、销售、库存、财务串成可追溯的流程。

业务抽象：
单据是业务发生的载体，明细是计算颗粒，审批是责任确认，下推是上下游关系，反审是逆向治理。

代价：
流程复杂，配置复杂，用户学习成本高。

为我所用：
保留“单据、明细、状态、上下游关系、逆向动作”的思想；
根据目标客户规模简化审批和下推；
不要照搬巨头产品的全部配置复杂度。
```

## 14. 系统成熟度分级

### L1：功能可用

- 页面能录入。
- 数据能保存。
- 基础查询可用。

风险：

- 模型通常围绕页面设计。
- 报表、扩展、追溯能力弱。

### L2：业务闭环

- 正向流程完整。
- 基础状态清楚。
- 主要报表可用。

风险：

- 逆向流程和异常处理不足。

### L3：可追溯

- 有流水、事件、日志、状态记录。
- 能解释历史。
- 能处理退款、冲正、反审。

风险：

- 数据链路复杂，需要治理。

### L4：可扩展

- 规则版本化。
- 业务能力边界清楚。
- 支持多组织、多渠道、多租户。
- UI 和接口契约稳定。

风险：

- 配置和扩展点需要控制复杂度。

### L5：可生成

- 知识库沉淀了能力模型、数据模型、UI 模型、报表模型、决策理由。
- AI 可以基于知识生成新系统设计草案。
- 生成结果可通过评审清单验证。

风险：

- 知识质量决定生成质量。
- 错误知识会被 AI 放大。

## 15. 最终方法论

世界级系统设计不是单点技巧，而是一套闭环：

```text
看业务
  -> 抽能力
  -> 找对象
  -> 定生命周期
  -> 记事实
  -> 设计状态
  -> 分离读写
  -> 支撑 UI
  -> 支撑报表
  -> 处理逆向
  -> 记录决策
  -> 沉淀知识
  -> 生成新系统
  -> 用真实场景验证
```

最终目标：

> 不是做一个“能用”的系统，而是做一个可以解释、可以演进、可以复用、可以被 AI 学习并持续生成更好系统的设计体系。
