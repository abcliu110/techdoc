# T-207 全新表单设计器：产品定位与范围

> 状态：v0.3
> 日期：2026-07-09
> 上游依据：`../00-参考产品拆解/T-207-全新表单设计器-商业产品参考研究.md`

---

## 1. 一句话定位

T-207 全新表单设计器是：

```text
面向企业 PC Web 单据、复杂表单、主数据维护、审批页和业务工作台的
IDE 级可见即可得页面设计器。
```

它不是普通表单 Builder，也不是报表设计器，而是一个围绕业务对象、页面布局、表单元素、强表格、规则权限、流程状态和发布治理构建的**运行时 Schema 驱动**企业级设计平台。

---

## 2. 核心设计理念

### 2.1 可见即可得

中央画布使用 RuntimeRenderer 渲染真实页面，DesignerOverlay 只负责选择框、吸附线、投放点等编辑辅助。设计态、预览态、运行态使用同一套渲染引擎，确保所见即所得。

### 2.2 Schema 驱动

Schema 是运行时的核心元数据，不是设计态的副产品：

```
Schema 定义页面（Design）
    ↓ 发布
Schema 驱动页面（Runtime）
    ↓ 热更新
Schema 变更即时生效
```

| 理解 | 说明 |
|------|------|
| Schema = 运行时元数据 | 发布后的页面由 Schema 解释执行，不是编译后的代码 |
| Schema 是版本化的 | 每个 Schema 有版本历史，支持回滚和灰度 |
| Schema 是可分析的 | 规则、权限、依赖关系可被 Analyzer 分析 |

### 2.3 为什么重新设计

旧方案的问题不是"原型不够漂亮"，而是产品定义还不够锋利：

| 问题 | 后果 |
|---|---|
| 容易退化成字段拖拽器 | 无法设计真实复杂单据和业务页面 |
| 布局能力不足 | 无法表达复杂 PC Web 页面结构 |
| 表格能力不足 | 无法支撑分录、母子嵌套、复杂表头和批量录入 |
| 设计态和运行态分离 | 原型看似可行，发布后可能错位、遮挡、截断 |
| 规则权限缺少治理 | 字段联动、权限、流程状态容易变成不可解释的脚本网络 |
| 缺少发布前分析器 | 企业平台缺少质量门禁 |

新设计必须从产品架构出发，而不是从单个原型页面出发。

|---|---|---|

## 4. 核心场景

### 4.1 复杂业务单据

例如销售订单、采购申请、费用报销、出库单。

必须支持：

1. 顶部命令栏。
2. 单据状态区。
3. 单据头字段。
4. 分录表格。
5. 汇总区。
6. 右侧审批/附件/关联单据面板。
7. 底部操作栏。
8. 字段权限、流程状态、业务规则。

### 4.2 主数据维护页面

例如客户、供应商、物料、组织、员工。

必须支持：

1. 基本信息。
2. 扩展属性。
3. 关联列表。
4. 审计信息。
5. 附件和备注。
6. 多页签和侧栏布局。

### 4.3 审批任务页面

例如审批详情、审批意见、流程轨迹。

必须支持：

1. 表单只读/编辑状态切换。
2. 审批意见区。
3. 流程节点轨迹。
4. 附件、签章、审批动作。
5. 基于角色和状态的字段可见性。

### 4.4 业务工作台页面

例如客户经营工作台、项目管理工作台、采购执行工作台。

必须支持：

1. 左右分栏。
2. 局部滚动。
3. 主从联动。
4. 数据卡片、表格、表单、页签混排。
5. 固定操作区和响应式断点。

## 5. 产品边界

### 5.1 T-207 是什么

| 定义 | 说明 |
|---|---|
| PC Web 页面设计器 | 面向桌面浏览器企业页面，不以移动端为首要目标 |
| Schema 驱动的运行时平台 | Schema 是运行时的核心元数据，不是设计态副产品 |
| 业务对象驱动 | 字段、分录、状态、权限来自业务对象和元数据 |
| 可见即可得 | 中央画布渲染真实运行态页面 |
| 可发布平台能力 | 设计、校验、预览、发布、回滚、影响分析形成闭环 |

### 5.2 T-207 不是什么

| 非目标 | 原因 |
|---|---|
| 报表设计器 | 报表、套打、BI 透视不是第一阶段目标 |
| 营销网站设计器 | 不追求 Webflow 式视觉自由度和营销动画 |
| 移动端 App Builder | 第一阶段聚焦 PC Web 企业页面 |
| Excel 替代品 | 吸收 Excel-like 编辑体验，但不做完整电子表格 |
| 普通问卷表单 Builder | 目标是复杂企业业务页面，不是轻量收集表 |
| Schema 只是设计态产物 | Schema 是运行时元数据，驱动页面渲染 |

## 6. Schema 体系

### 6.1 Schema 类型

| Schema 类型 | 职责 | 说明 |
|------------|------|------|
| PageSchema | 页面整体结构 | 包含页面元信息、布局、组件、数据绑定、规则、权限 |
| LayoutSchema | 布局配置 | Grid、Flex、Dock、Split、Tabs 等布局参数 |
| ComponentSchema | 组件配置 | 字段、按钮、容器等组件的属性配置 |
| TableSchema | 表格配置 | 列定义、编辑器、汇总、分页、嵌套明细 |
| RuleSchema | 规则配置 | 显隐、必填、只读、计算、校验规则 |
| PermissionSchema | 权限配置 | 页面、区块、字段、按钮、行列权限 |
| DataBindingSchema | 数据绑定配置 | 数据源、字段映射、级联刷新 |

### 6.2 Schema 约束

| 约束项 | 值 | 说明 |
|--------|---|------|
| 格式 | JSON | UTF-8 编码 |
| 最大单 Schema size | 500KB | 超过需拆分页面 |
| 最大组件数/页 | 200 个 | 超过需拆分页面 |
| 嵌套表格最大深度 | P0=Level 1，P1=Level 2，P2=Level 3 | 深度受控，防止性能问题；Level 0 为主表 |
| Schema 版本状态 | draft / published / archived | 三态流转 |
| 保存前归一化 | 必须 | 保存和发布前必须经 SchemaNormalizer 归一化；其中 LayoutNormalizer 负责布局子树，确保 children 引用可解析、默认值已补齐、非法属性已清理 |

### 6.3 PageSchema 结构

```json
{
  "schemaVersion": "1.0",
  "pageId": "page_sales_order",
  "version": 1,
  "status": "draft",
  "metadata": {
    "title": "销售订单",
    "businessObject": "BO_SalesOrder",
    "createdBy": "user_001",
    "createdAt": "2026-07-09T10:00:00Z",
    "componentsCount": 45,
    "schemaSize": 128000
  },
  "root": "page_root",
  "components": {
    "page_root": {
      "id": "page_root",
      "type": "PageRoot",
      "layout": {
        "direction": "vertical"
      },
      "children": ["section_basic", "entry_sales_items"]
    },
    "section_basic": {
      "id": "section_basic",
      "type": "Section",
      "children": ["grid_basic"]
    },
    "grid_basic": {
      "id": "grid_basic",
      "type": "GridLayout",
      "children": ["field_customer"]
    },
    "field_customer": {
      "id": "field_customer",
      "type": "Field",
      "binding": "salesOrder.customerId"
    },
    "entry_sales_items": {
      "id": "entry_sales_items",
      "type": "EntryTable",
      "dataSource": "salesOrder.entries",
      "columns": []
    }
  }
}
```

`version` 是正整数版本号，`status` 是 `draft / published / archived` 枚举；二者不能混用。P0 持久化结构以 `root + components` 为准，不再使用顶层 `layout` 树。

Schema 版本状态与业务流程状态必须区分：`status` 仅表示 Schema 生命周期；业务单据状态在运行态中使用 `bizState` 表示，例如 `bizState=draft/submitted/approved/archived`。

### 6.4 Schema 验证规则

| 规则 | 说明 |
|------|------|
| 循环引用检测 | 字段A依赖B、B依赖A时拒绝 |
| 字段绑定完整性 | 字段必须有 dataBinding 或 staticValue |
| 规则影响环检测 | 规则链形成环时拒绝 |
| 权限冲突检测 | 同一字段显隐和只读冲突时警告 |
| Schema size 检查 | 超过 500KB 时警告 |

## 7. 运行时集成

### 7.1 Schema 持久化

| 项目 | 说明 |
|------|------|
| 存储位置 | 数据库表 `page_schema` |
| Schema API | `POST /api/design/schema/save` 保存 |
| 版本 API | `GET /api/design/schema/versions/{pageId}` 获取版本历史 |
| 发布 API | `POST /api/design/schema/publish` 发布草稿 |

保存和发布链路必须先执行 Schema 归一化：

```text
Draft Schema
  -> SchemaNormalizer normalize
     -> LayoutNormalizer normalize
  -> SchemaValidator validate
  -> PublishAnalyzer check
  -> save / publish
```

归一化后才能写入持久化表，避免运行态拿到悬浮 ID 引用、缺省布局属性或非法 children 结构。

### 7.2 运行时 Schema 下发

| 场景 | 方式 | 说明 |
|------|------|------|
| 页面加载 | 全量拉取 | `GET /api/runtime/page/{pageId}` 获取最新已发布版本 |
| Schema 更新 | WebSocket 推送 | 发布后运行态通过 WebSocket 收到刷新通知 |
| 指定版本 | URL 参数 | `?schemaVersion=xxx` 可访问历史版本（需权限） |
| 多实例一致性 | 中央数据库 | 所有运行实例从同一数据源获取 Schema |

### 7.3 热更新策略

| 步骤 | 说明 |
|------|------|
| 1. 设计态发布 | Schema 写入 published 状态，version +1 |
| 2. WebSocket 广播 | 所有运行态实例收到 `schema:updated` 事件 |
| 3. Schema 重新拉取 | 运行态重新请求最新 Schema |
| 4. 页面热刷新 | 运行态重新拉取全量 Schema，RuntimeRenderer 在本地尽量只重渲染受影响子树 |
| 5. 最大延迟 | 热更新生效延迟 ≤ 5 秒 |

### 7.4 并发编辑控制

| 场景 | 策略 |
|------|------|
| A 和 B 同时编辑同一页面 | 后保存者需处理冲突（类似 Git） |
| A 正在编辑，B 发布 | A 的草稿变为 stale，需重新拉取 |
| 冲突检测时机 | 保存时检测，冲突时提示用户 |

## 8. 能力分层

```
业务对象层
  -> 页面 Schema 层
    -> 布局引擎层
      -> 组件/字段层
        -> 表格/分录层
          -> 规则权限流程层
            -> 发布治理层
              -> RuntimeRenderer
```

| 层 | 关键能力 |
|---|---|
| 业务对象层 | BusinessObject、Field、Relation、EntryModel、StateMachine |
| 页面 Schema 层 | PageSchema、LayoutSchema、ComponentSchema、RuleSchema |
| 布局引擎层 | Grid、Flex、Dock、Split、Tabs、Collapse、Scroll、Sticky、Portal |
| 组件/字段层 | Text、Number、Date、Money、ReferencePicker、Attachment、Signature、Stamp |
| 表格/分录层 | EntryTable、NestedDetailTable、ColumnSchema、CellEditor、复杂表头 |
| 规则权限流程层 | 显隐、必填、只读、计算、校验、权限、审批状态 |
| 发布治理层 | Analyzer、Version、Diff、Impact、Rollback |
| 运行渲染层 | 设计态、预览态、运行态同源渲染 |

| 引擎 | 职责 | P0 支持 |
|------|------|--------|
| LayoutEngine | Grid/Flex/Dock/Split/Scroll/Sticky/Portal 布局计算 | ✓ |

| PermissionEngine | 页面、区块、字段、按钮、表格、行列单元格权限 | ✓ 入口 |
| DataBindingEngine | 字段-数据列映射、级联刷新、跨表单引用 | ✓ P0 基础绑定；级联刷新和跨表单引用为 P1 |

| PublishAnalyzer | 保存/发布前质量门禁 | ✓ |
| RuntimeRenderer | 设计态、预览态、运行态同源渲染 | ✓ |
| TemplateEngine | 行业模板、布局模板、组件组合模板 | P1 |

## 12. 性能约束

| 指标 | 约束值 | 说明 |
|------|--------|------|
| 单页最大组件数 | 200 | 超过需拆分页面 |
| 单页最大布局容器数 | 100 | 统计 Section、GridLayout、FlexLayout、Tabs、SplitPane、ScrollContainer、StickyBar 等布局容器；自动生成且唯一的 PageRoot 不计入 100 个容器限制；不含 Field、Button、TableColumn |
| 最大 Schema size | 500KB | 超过需拆分页面 |
| 嵌套表格最大深度 | P0=Level 1，P1=Level 2，P2=Level 3 | 深度受控；Level 0 为主表 |
| 设计器冷启动时间 | ≤ 3 秒 | 从打开到可编辑 |
| 画布渲染帧率 | ≥ 30 fps | 拖拽操作流畅度 |
| Schema 保存时间 | ≤ 2 秒 | 单次保存操作 |
| 运行态页面加载 | ≤ 2 秒 | Schema 拉取 + 首屏渲染 |
| 热更新延迟 | ≤ 5 秒 | 发布后生效最大延迟 |

## 14. 设计原则

### 14.1 Schema 优先

Schema 是运行时的核心契约，设计器的所有输出都是 Schema。所有功能必须能转化为 Schema 可表达的形态。

### 14.2 业务优先

页面不是从空白画布开始，而是从业务对象开始。字段、分录、状态、权限、规则应有稳定元数据来源。

### 14.3 运行态同源

设计态画布必须使用 RuntimeRenderer 渲染真实页面，DesignerOverlay 只负责编辑辅助。

### 14.4 Web 布局优先

页面主体使用 Grid、Flex、Dock、Split、Tabs、Scroll、Sticky 等 Web 布局能力，不使用绝对定位堆叠字段。

### 14.5 表格是一等公民

企业单据的分录表格不是普通控件，而是与表单同级的核心引擎。

### 14.6 治理内建

规则、权限、数据绑定、版本、发布校验不能作为后补功能，必须从第一期进入设计器骨架。

### 14.7 企业视觉克制

界面采用灰白工具风格、信息密集、低装饰、清晰边界。

### 14.8 性能约束内置

单页组件数、Schema size、嵌套深度等约束在设计时就强制校验，而非运行时才发现问题。

## 15. 第一版原型必须表达什么

第一版原型不是展示"控件很多"，而是证明 T-207 能设计复杂企业页面。

必须展示：

1. 一个完整销售订单设计场景。
2. 五区 IDE 工作台。
3. 左侧资源区含布局容器、业务字段、单据组件、表格组件、大纲。
4. 中央画布显示真实 PC Web 销售订单页面。
5. 页面中有单据头、基础信息、分录表格、汇总、右侧审批/附件面板。
6. 选中字段、容器、表格列时右侧属性面板变化。
7. 表格展示冻结列、行操作、SummaryRow、Level 1 DetailPanel；列组/复杂表头仅作为 P1 参考能力，不在 P0/09 原型中表达为已完成能力。
8. 底部 Analyzer 展示布局、规则、权限、性能问题。
9. 视觉克制，接近企业 IDE，不花哨。
10. 数据绑定效果：字段显示 BO 数据。

## 16. 成功标准

| 标准 | 验收问题 | 可量化指标 |
|------|---------|-----------|
| 定位清晰 | 用户是否一眼看出这是企业 PC Web 单据设计器 | 5个用户测试，3秒内描述正确 |
| 布局强 | 是否能表达复杂页面 | 支持8种以上布局容器类型 |
| 表格强 | 是否能支撑真实分录、母子明细 | 支持1层嵌套，可配置汇总行 |
| 治理强 | 是否能看见规则、权限、发布校验和版本控制 | 5种规则类型可配置 |
| 运行一致 | 设计态是否接近真实运行态页面 | 关键布局框、字段位置、表格列宽与运行态一致；差异需由截图评审记录 |
| 企业感 | 是否克制、密集、专业 | 主色/语义色≤5种；仅保留必要交互反馈动效 |
| 性能达标 | 是否满足性能约束 | 单页≤200组件，Schema≤500KB |

## 17. 关键风险

| 风险 | 说明 | 控制方式 |
|---|---|---|
| 范围过大 | P0/P1/P2 如果不分层，会变成巨型工程 | 第一版只做 P0，P1/P2 作为路线图 |
| 竞品照搬 | 直接复制 DevExpress 或金蝶会偏题 | 只吸收能力层，不照搬外观和业务边界 |
| 表格复杂度失控 | 嵌套表格容易引发性能和交互问题 | P0 只做到 Level 1，P1 做到 Level 2，Level 3 放 P2 |
| 规则不可治理 | 自由 JS 会变成黑盒 | P0 只做声明式规则入口 |
| 原型过度视觉化 | 花哨会掩盖企业工具目标 | 原型用灰白 IDE 风格 |
| Schema 膨胀 | 复杂页面 Schema 超过 500KB | 设计期强制校验，超限提示拆分 |
| 并发编辑冲突 | 多人同时编辑同一页面 | 后保存者处理冲突，类似 Git |
