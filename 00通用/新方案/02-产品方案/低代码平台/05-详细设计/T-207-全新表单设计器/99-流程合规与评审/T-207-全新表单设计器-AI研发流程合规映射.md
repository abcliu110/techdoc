本轮已按 06 门禁对 `PT-025 左侧工具栏详细原型` 做局部回跑：

### 1A.3 T-207 逐组件即时视觉闭环

T-207 从 `PT-029` 起强制执行两份通用规范新增的逐迭代视觉门禁，后续工具箱、画布、Inspector、强表格、列管理、规则、权限、发布工作台均按同一规则执行：

```text
修改一个组件或工作区
-> 立即刷新真实浏览器
-> 1440×900 与 1920×1080 运行态截图
-> 初始态、关键交互态、阻断态、恢复态检查
-> 专业前端 / UX 评审
-> 自动化点击 / 拖拽与状态断言
-> 对抗性 Critic + Verifier 复核
-> 阻断清零后进入下一组件
```

项目级硬规则：

1. 不允许把整页 HTML 生成完成后再集中补审。
2. 浏览器命令成功必须再由 scoped selector 下的 DOM 状态证明；所有 PT 专项自动化必须限定在对应 `#pt-XXX` 根节点。
3. 1440 与 1920 必须分别检查工具条裁切、面板比例、文字换行、横向滚动、浮层定位和底部状态条。
4. 每次视觉修复必须产生新截图，不得以旧截图证明新代码。
5. 竞品超过结论仍需 source_id / screenshot_id / task_id 和人工终审；本地视觉门禁通过不等于超过竞品。

`PT-029` 已按该闭环发现并修复：1440 工具条裁切、表格超宽静默截断、列菜单锚点不可见、冻结预算只声明未执行、冻结超限修复建议不匹配等问题。对应证据必须写入 `PT-029-列菜单列管理06门禁`，不能只保留在对话中。

## 2. 项目复杂度判定

### 2.1 判定等级

T-207 全新表单设计器复杂度判定为：

```text
qualityLevel = L4 平台核心
```

### 2.2 判定原因

| 触发条件 | 是否满足 | 说明 |
|---|---|---|
| 涉及 3 个以上用户角色 | 是 | 业务顾问、实施顾问、前端开发、后端开发、测试/运维 |
| 涉及多运行模式 | 是 | 设计态、预览态、运行态 |
| 涉及权限、审计、发布、回滚 | 是 | 规则权限、版本状态、发布门禁、schemaHash |
| 涉及复杂 UI 工作台 | 是 | 五区 IDE 工作台、画布、Inspector、Analyzer |
| 涉及平台元模型或运行时基础设施 | 是 | PageSchema、RuntimeRenderer、SchemaNormalizer |
| 失败影响多个业务模块 | 是 | 表单设计器产物会驱动多个单据、主数据、审批、工作台页面 |
| 需要兼容历史版本 | 是 | 旧 `layout.children`、旧 `EntryTable.children` 迁移 |

### 2.3 启用的加严门槛

| 阶段 | 加严要求 |
|---|---|
| 阶段 1 需求规格 | 必须有角色-场景-目标矩阵、P0 业务闭环样例、需求质量矩阵 |
| 阶段 2 功能设计 | 必须有页面原型编号、交互流、状态矩阵、权限矩阵、异常矩阵 |
| 阶段 3 详细设计 | 必须有架构级 Schema、API、数据结构、兼容策略、迁移策略、回滚策略 |
| 阶段 4 自我评审 | 必须汇总多轮评审问题、修复状态、剩余风险 |
| 阶段 5 测试规格 | 必须能直接生成自动化用例和发布门禁 |
