## 1. 设计目标

测试验收体系要回答一个核心问题：

```text
如何证明 T-207 表单设计器既能被人高效使用，
又能被自动化平台稳定模拟、验证、回归和发布门禁阻断。
```

T-207 不是普通表单 Builder。它的质量标准必须覆盖四层：

1. 设计器工作台本身是否稳定、可操作、可回放。
2. 设计器产出的 PageSchema 是否结构正确、可归一化、可分析。
3. RuntimeRenderer 是否能在设计态、预览态、运行态同源渲染。
4. 发布门禁是否能阻断布局、字段、表格、规则、权限、性能风险。

## 2. 测试原则

| 原则 | 说明 |
|---|---|
| 用户路径优先 | P0 自动化必须模拟真实点击、拖拽、输入、调整，不直接改内部状态 |
| Schema 证据优先 | 每次设计态操作都必须能断言 Schema diff |
| 门禁前置 | Error 级问题必须在发布前被 Analyzer 阻断 |
| 同源渲染 | 设计态、预览态、运行态使用 RuntimeRenderer 的同一渲染契约 |
| 稳定标识 | 自动化只依赖 `data-testid`、`schemaPath`、`nodeId`，不依赖中文标题和 CSS 类 |
| 分层验证 | 单元测试、Schema 测试、集成测试、端到端测试、视觉回归各司其职 |
| 可追责 | 每次发布都保存 Analyzer 结果、schemaHash、测试报告和操作者信息 |

## 3. 测试分层

```text
Static Check
  -> Unit Test
  -> Schema Snapshot Test
  -> Analyzer Test
  -> Interaction Test
  -> Runtime Integration Test
  -> Visual Regression Test
  -> Publish Gate
```

| 层级 | 目标 | P0 |
|---|---|---|
| Static Check | 文档、Schema、配置、命名规范静态检查 | 是 |
| Unit Test | LayoutEngine、FieldResolver、TableEngine、Normalizer 等纯逻辑 | 是 |
| Schema Snapshot Test | 操作前后 PageSchema diff 和归一化结果 | 是 |
| Analyzer Test | 发布门禁规则的 Error/Warning/Suggestion 输出 | 是 |
| Interaction Test | Playwright 模拟人操作设计器 | 是 |
| Runtime Integration Test | 设计态产物进入预览态和运行态渲染 | 是 |
| Visual Regression Test | 截图对比关键布局、表格、字段状态 | 是 |
| Performance Test | 大页面、大表格、拖拽帧率、加载时间 | 是 |
| Publish Gate | 发布前自动执行质量门禁 | 是 |

### 4.2 禁止的自动化方式

| 禁止 | 原因 |
|---|---|
| 直接调用内部 store 修改 Schema | 不能验证真实用户路径 |
| 依赖中文标题定位 | 多语言、改名、同名节点会失效 |
| 依赖 CSS class | 样式重构会破坏测试 |
| 只用坐标定位起点和终点 | 画布缩放、滚动、断点会导致漂移 |
| 跳过 DesignerOverlay 抓手 | 无法验证抓手命中区域和交互质量 |

坐标可以用于拖拽过程，但起点和目标必须来自稳定 testId 的元素矩形。

### 4.3 自动化动作语义

| 动作 | 模拟用户行为 | 必须断言 |
|---|---|---|
| `clickNode(nodeId)` | 点击画布节点 | 选中框出现，Inspector 切换 |
| `dragFromPalette(assetId, targetDropzoneId)` | 从 Component Palette 拖入组件资产 | 新节点出现，Schema 新增，`sourceAssetId` 可追溯 |
| `dragNode(nodeId, targetId)` | 拖动画布节点 | CommandStack 新增 MoveNode |
| `resizeField(fieldId, targetCell)` | 拖字段右侧抓手 | `components.{fieldId}.layout.span` 改变 |
| `resizeSplit(splitId, dx)` | 拖 Split 分隔线 | `secondarySize` 改变 |
| `resizeColumn(tableId, columnId, dx)` | 拖表格列宽抓手 | `columns[n].layout.width` 改变 |
| `dragEntryFieldToTableColumn(fieldPath, tableId)` | 从 BO Field Tree 的字段资产拖分录字段到 `designer-dropzone-{tableId}-columns` | `columns[]` 增加或更新，TableColumn 写入 binding/editor，不生成顶层组件节点 |
| `expandDetail(tableId, rowKey)` | 展开明细 | 明细区域可见，状态不污染 Schema |
| `setInspectorProp(nodeId, propName, value)` | 在属性面板修改属性 | Schema diff 精确命中 schemaPath |
| `clickAnalyzerIssue(issueId)` | 点击 Analyzer 问题 | 定位到节点和 Inspector 属性 |
| `publishPage(pageId)` | 点击发布 | 无 Error 才能生成 published 版本 |

## 5. Schema 测试

### 5.1 PageSchema 基线校验

P0 每个 PageSchema 必须通过以下校验：

| 规则 | 级别 |
|---|---|
| `schemaVersion` 必填 | Error |
| `pageId` 必填且全局唯一 | Error |
| `version` 必须为正整数 | Error |
| `status` 必须为 `draft/published/archived` | Error |
| draft 必须记录 `basePublishedVersion` | Error |
| `root` 必须指向 `components` 中的 PageRoot | Error |
| 所有可渲染节点必须存放于 `components.{nodeId}` | Error |
| `components.{nodeId}.id` 与 key 一致 | Error |
| `children` 引用必须可解析 | Error |
| TableColumn 不作为 `components.{id}` 顶层节点 | Error |
| EntryTable 使用 `columns/detail`，不使用 `children` 表达列 | Error |
| `schemaHash` 发布时必须存在 | Error |
| P0 必备 testId 必须可生成 | Error |

### 5.2 归一化快照

SchemaNormalizer 必须有快照测试：

| 输入 | 期望输出 |
|---|---|
| 缺失 testId 的节点 | TestIdNormalizer 补齐节点、抓手、投放点、错误区稳定 testId |
| 缺失布局默认值 | 补齐 span、gap、minWidth、labelWidth |
| 旧 `layout.children` 树 | 提升为 `root + components` |
| 旧 `EntryTable.children` | 可迁移时转为 `columns/detail` |
| 不可迁移旧表格结构 | Analyzer Error |
| P1 动态规则字段 | P0 标记 inactive 或 Warning |
| 字段绑定缺失 | 保留结构，但 Analyzer Error |

### 5.3 schemaPath 覆盖

所有 Analyzer、Inspector、Outline、CommandStack 修改必须使用统一 schemaPath：

| 场景 | schemaPath |
|---|---|
| 页面根 | `components.page_root` |
| Grid 子节点 | `components.grid_basic.children[0]` |
| 字段绑定 | `components.field_customer.binding` |
| 字段布局 | `components.field_customer.layout.span` |
| 表格数据源 | `components.entry_sales_items.dataSource` |
| 表格列 | `components.entry_sales_items.columns[3]` |
| 表格列宽 | `components.entry_sales_items.columns[3].layout.width` |
| 表格明细 | `components.entry_sales_items.detail` |
| 规则 | `rules.rule_credit_required` |
| 权限 | `permissions.perm_sales_amount.static.readonly` |
| 状态流转 | `stateMachine.transitions[1]` |

## 6. Analyzer 测试

### 6.1 Analyzer 输出契约

Analyzer 输出必须稳定：

```json
{
  "id": "issue_field_customer_binding_missing",
  "severity": "error",
  "code": "FIELD_BINDING_MISSING",
  "message": "客户字段缺少数据绑定。",
  "nodeId": "field_customer",
  "schemaPath": "components.field_customer.binding",
  "quickFix": "绑定到 salesOrder.customerId"
}
```

| 字段 | 要求 |
|---|---|
| id | 全局唯一，可用于 `analyzer-issue-{issueId}` |
| severity | error / warning / suggestion |
| code | 稳定错误码，不随语言变化 |
| nodeId | 可定位到画布节点 |
| schemaPath | 可定位到 Inspector 属性 |
| quickFix | P0 可为空；存在时必须可回放 |

### 6.2 P0 Analyzer 用例矩阵

| Analyzer | Error 用例 | Warning 用例 |
|---|---|---|
| LayoutAnalyzer | 缺少 PageRoot、非法嵌套、Grid span 越界、断点横向溢出 | label 截断风险、Scroll 嵌套风险 |
| FieldAnalyzer | 字段绑定缺失、编辑器类型非法、必填但无校验 | label 过长、默认 span 风险 |
| TableAnalyzer | 列 ID 重复、列宽非法、Level 2 明细进入 P0 | 列过多、冻结列过多、虚拟滚动风险 |
| RulePermissionAnalyzer | 动作 target 不存在、权限 target 不存在、P0 不支持规则启用 | 静态权限冲突、状态显示不完整 |
| PublishAnalyzer | 任一 Error 未解决 | Warning 未填写发布说明 |

### 6.3 发布阻断规则

| 条件 | 发布结果 |
|---|---|
| 存在 Error | 阻断发布 |
| 仅存在 Warning | 允许发布，但必须记录说明 |
| 仅存在 Suggestion | 允许发布 |
| draft stale | 阻断发布 |
| schemaHash 生成失败 | 阻断发布 |
| RuntimeRenderer 预览失败 | 阻断发布 |
| P0 必备 testId 缺失 | 阻断发布 |
| 首屏或拖拽性能低于 P0 约束 | 阻断发布 |

## 8. Web 布局验收

### 8.1 P0 布局能力

| 布局控件 | 必须验证 |
|---|---|
| PageRoot | 单页面唯一，最小宽度、密度、滚动模式生效 |

| GridLayout | columns、span、labelWidth、gap、自动换行正确 |
| FlexLayout | direction、align、justify、gap 正确 |
| Tabs | 当前 Tab 内容投放和切换正确 |
| SplitPane | 两子区、分隔线拖拽、最小宽度约束正确 |
| ScrollContainer | 局部滚动、sticky 不遮挡内容 |
| StickyBar | 顶部/底部固定区稳定，不覆盖主体 |

### 8.2 断点验收

P0 支持 1280 / 1440 / 1600 / 1920 四类桌面断点：

| 断点 | 验收 |
|---|---|
| 1280 | 无横向溢出 Error，关键字段不截断 |
| 1440 | 默认设计断点，布局与 Schema 快照一致 |
| 1600 | 右侧辅助区和表格可用宽度增加 |
| 1920 | 页面不被拉伸到不可读，最大宽策略有效 |

### 8.3 视觉回归

每个核心页面保存截图基线：

| 截图 | 目的 |
|---|---|
| 设计态全屏 | 验证五区工作台和画布 |
| 选中字段 | 验证选中框、抓手、Inspector |
| 选中表格列 | 验证列抓手和列属性 |
| Analyzer 展开 | 验证问题列表和定位 |
| 预览态页面 | 验证隐藏 DesignerOverlay 后的运行页面 |
| 运行态页面 | 验证真实数据和权限结果 |

P0 允许少量抗锯齿差异，但不允许布局位移、遮挡、文本截断、抓手缺失。

| 静态权限 | read/edit/readonly/action 配置生效 |
| 动作引用 | ActionButton target/serviceRef 可解析 |
| StateMachine 展示 | 当前业务状态可显示，但 P0 不驱动动态字段状态 |

### 11.2 P1 保留入口

| 能力 | P0 处理 |
|---|---|
| 动态 RuleSchema | 可保存入口，不执行复杂动态求值 |
| 角色预览 | P1 PreviewState，不作为 P0 RuntimeState |
| 流程状态驱动字段 | P1 |
| 数据权限 | P1 |
| 规则影响分析 | P1 |

### 11.3 权限冲突测试

| 场景 | 期望 |
|---|---|
| readonly=true 且 editable=false | finalReadonly=true |
| disabled=true 且 required=true | Warning，提示业务矛盾 |
| action 无权限 | 按钮不可用或隐藏 |
| permission target 不存在 | Error |
| rule target 不存在 | Error |

## 12. RuntimeRenderer 验收

### 12.1 同源渲染

| 模式 | 验收 |
|---|---|
| design | 显示 DesignerOverlay、抓手、投放点 |
| preview | 隐藏设计辅助，使用样例数据 |
| runtime | 加载真实数据和权限结果 |

同一 PageSchema 在三种模式下的布局结构必须一致。设计态允许多出 overlay DOM，但不允许用不同布局模拟运行态。

### 12.2 RuntimeState 边界

| 行为 | 应写入 |
|---|---|
| 输入字段值 | RuntimeState |
| 编辑表格单元格 | RuntimeState |
| 展开表格明细 | RuntimeState |
| 调整字段 span | Schema |
| 调整列宽 | Schema |
| 修改权限配置 | Schema |
| 角色预览 | PreviewState，P1 |

RuntimeState 不得回写 Schema。测试必须对比操作前后 Schema，确认用户输入没有污染设计元数据。

## 13. 发布门禁

### 13.1 发布流程门禁

```text
click publish
  -> check stale draft
  -> SchemaNormalizer full normalize
  -> Schema static validate
  -> AnalyzerPipeline
  -> RuntimeRenderer preview smoke test
  -> schemaHash generate
  -> create published version
  -> broadcast schema:updated
```

### 13.2 门禁结果

| 门禁 | 阻断条件 |
|---|---|
| StaleGate | 当前 draft 基于旧 published 版本 |
| SchemaGate | PageSchema 结构非法 |
| AnalyzerGate | 存在 Error |
| RuntimeGate | 预览态渲染异常 |
| TestIdGate | P0 必备 testId 缺失 |
| PerformanceGate | 首屏或拖拽性能低于 P0 约束 |
| HashGate | schemaHash 生成失败 |
| PerformanceGate | 冷启动 >3 秒、运行态页面加载 >2 秒、拖拽帧率 <30 fps、Schema size >500KB |

### 13.3 发布证据包

每次成功发布必须生成证据包：

| 证据 | 内容 |
|---|---|
| schema snapshot | 归一化后的 PageSchema |
| schemaHash | SHA-256 |
| version metadata | version、status、basePublishedVersion、createdAt |
| Analyzer report | Error/Warning/Suggestion 汇总 |
| Runtime smoke report | 预览态渲染结果 |
| Test report | 自动化测试结果 |
| Visual report | 截图回归结果 |
| Publish metadata | 发布人、时间、说明、版本号 |

## 14. 性能验收

| 指标 | P0 目标 |
|---|---|
| 设计器冷启动 | ≤ 3 秒 |
| 单页最大组件数 | ≤ 200 |
| 布局容器数 | ≤ 100 |
| Schema size | ≤ 500KB |
| Schema 保存 | ≤ 2 秒 |
| 运行态页面加载 | ≤ 2 秒 |
| 热更新生效 | ≤ 5 秒 |
| 拖拽帧率 | ≥ 30 fps |
| 表格 1000 行虚拟滚动 | 无明显卡顿 |
| Analyzer 增量检查 | 单节点变更 ≤ 300ms |

Schema size 由 CI 或发布门禁通过归一化后的 PageSchema JSON UTF-8 字节数计算，不以 `metadata.schemaSize` 为唯一来源。自动化动作 `assertSchemaSize(pageId, maxKB)` 必须从 Schema API 获取归一化结果并计算字节数，超过 500KB 时阻断发布。

性能测试必须覆盖：

1. 200 组件复杂页面。
2. 100 列表格配置态。
3. 1000 行运行态表格。
4. Split 拖拽和字段 span 拖拽。
5. Analyzer 同步刷新。
6. 预览态切换。

## 17. CI 与回归流程

### 17.1 提交流程

```text
开发提交
  -> lint
  -> typecheck
  -> unit test
  -> schema snapshot test
  -> analyzer test
  -> playwright interaction test
  -> visual regression
  -> performance smoke
  -> publish gate dry-run
```

### 17.2 回归套件分级

| 套件 | 触发 |
|---|---|
| quick | 每次提交，≤ 5 分钟 |
| standard | 合并前，≤ 20 分钟 |
| full | 每日或版本发布前 |
| migration | SchemaNormalizer 或版本结构变更时 |

### 17.3 失败处理

| 失败 | 处理 |
|---|---|
| 单元测试失败 | 阻断合并 |
| Schema 快照差异 | 必须人工确认或修复 |
| Analyzer 用例失败 | 阻断发布 |
| Playwright 交互失败 | 阻断合并 |
| 视觉回归失败 | 需截图评审 |
| 性能回归 | 超过阈值阻断发布 |

## 19. 缺陷分级

| 级别 | 定义 | 示例 |
|---|---|---|
| Blocker | 发布或核心路径不可用 | 有 Error 仍发布成功 |
| Critical | 核心能力错误 | RuntimeRenderer 与设计态布局不一致 |
| Major | 重要能力缺陷 | 表格列宽无法持久化 |
| Minor | 局部体验缺陷 | 提示文案不够清晰 |
| Trivial | 不影响功能 | 非关键视觉细节 |

P0 不允许遗留 Blocker/Critical。Major 必须有明确修复计划和影响说明。
