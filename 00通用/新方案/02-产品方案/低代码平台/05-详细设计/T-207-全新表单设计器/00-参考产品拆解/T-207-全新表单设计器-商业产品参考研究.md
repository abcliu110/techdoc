# T-207 全新表单设计器：商业产品参考研究

> 状态：v0.3 研究基线（含深度拆解补充，已对齐 P0/P1/P2 边界）
> 日期：2026-07-09
> 目的：为 T-207 全新表单设计器提供参考产品、能力吸收方向和禁止照搬边界。

---

## 1. 研究结论

T-207 不应定义为普通表单 Builder，也不应定义为报表设计器。

它应定义为：

```text
面向企业 PC Web 单据、复杂表单、主数据维护、审批页和业务工作台的
IDE 级可见即可得页面设计器。
```

参考产品应分层吸收：

1. **金蝶**提供企业业务元模型、动态表单、单据、分录、规则、权限、流程的业务语义。
2. **DevExpress / Visual Studio**提供设计器工作台骨架和专业编辑交互。
3. **Oracle APEX / Infor Mongoose / ServiceNow UI Builder**提供企业 Web 页面设计器和组件树范式。
4. **Power Apps / Salesforce / Pega / Appian / Creatio**提供模型驱动表单、流程表单、可见性规则和业务规则范式。
5. **AG Grid / SpreadJS / DevExtreme / Kendo / Syncfusion / DevExpress VCL Grid**提供强表格、分录、母子嵌套和高效录入范式。
6. **Webflow / Plasmic / Builder.io**提供强 Web 布局、组件注册和设计系统约束启发。

最终目标不是复制某一家产品，而是组合成一个比普通低代码和金蝶表单设计器更强的企业级 Web 单据设计平台。

## 3. S 级重点参考产品

### 3.1 金蝶 BOS（传统 IDE）

金蝶 BOS 是传统企业开发平台，适合作为**强约束业务语义**的参考。

| 可吸收点 | T-207 转化 |
|---|---|
| 业务对象定义 | BusinessObject / Field / Relation |
| 单据头体 | HeaderForm + EntryTable + Summary + ActionBar |
| 分录（多层级） | EditableTable / EntryTable / NestedDetailTable |
| 权限规则 | PermissionOverlay / FieldPermission / ActionPermission |
| 流程审批 | StateMachine / ApprovalPanel / OperationPolicy |
| 发布机制 | Draft / Version / Publish / Rollback |
| BOS 实体单据体 | BillEntity / BillHead / BillBody / BillEntry |

**吸收边界**：BOS 的强约束是优点（企业级规范），也是缺点（灵活性差）。T-207 应吸收其业务语义完整性，但放宽数据源绑定限制。

### 3.1.1 金蝶云苍穹（现代架构）

金蝶云苍穹采用微服务 + 前端组件化架构，适合作为**现代企业 SaaS**参考。

| 云苍穹能力 | T-207 转化 |
|---|---|
| 动态领域模型 | BusinessObject / Field / Relation / EntryModel |
| 动态表单 | PageSchema / FormSchema / RuntimeRenderer |
| 前端组件化 | ComponentRegistry / DesignSystem / Variant |
| 服务编排 | FlowEngine / ServiceTask / DataMapping |
| 规则引擎 | RuleEngine / Condition / Action |
| 多端适配 | PCSchema / MobileSchema / ResponsiveLayout |

**核心差异**：
- BOS 是桌面 IDE，云苍穹是纯 Web
- BOS 单据强绑定金蝶实体，云苍穹支持外部数据源
- BOS 规则在 IDE 配置，云苍穹规则可运行时热更新

### 3.3 Visual Studio Form Designer

Visual Studio 的价值是编辑体验：

| VS 能力 | T-207 转化 |
|---|---|

### 3.4 Oracle APEX Page Designer

Oracle APEX Page Designer 是成熟企业 Web 页面设计器，核心理念是**"页面是数据库的视图"**。

#### 3.4.1 核心数据模型

```
Page Definition
├── Page Items (与Session变量绑定)
├── Regions (SQL渲染的HTML区域)
│   ├── Static Content
│   ├── Classic Report
│   ├── Interactive Report
│   └── Dynamic Content
├── Items (表单输入项)
│   ├── Text Field
│   ├── Select List
│   ├── Checkbox
│   └── Date Picker
├── Buttons (操作按钮)
├── Processes (服务端处理流程)
│   ├── Data Manipulation
│   ├── Ajax Callback
│   └── PL/SQL Execution
├── Dynamic Actions (客户端交互逻辑)
│   ├── Event (Click, Change, Load...)
│   ├── Condition
│   └── Actions (Show, Hide, Set Value...)
└── Branches (页面导航)
```

#### 3.4.2 关键设计理念

| APEX 理念 | T-207 转化 | 说明 |
|---------|-----------|------|
| Page Item = Session 变量 | FieldNode = DataBinding | 字段与数据双向绑定 |
| Region = SQL 渲染区域 | Section = DataSource Query | 区域内容由数据查询驱动 |
| Dynamic Action = 前端交互 | RuleAction = 可见性/联动规则 | 规则声明式配置，不写JS |
| Process = 服务端处理 | ServerAction = 保存/校验管道 | 服务端逻辑可配置 |
| Branch = 页面导航 | Navigation = 路由 + 状态机 | 页面跳转受控 |

#### 3.4.3 Schema 结构

```json
{
  "pageId": 100,
  "name": "CustomerEdit",
  "regions": [
    {
      "id": "r1",
      "type": "form",
      "sql": "SELECT * FROM customers WHERE id = :P100_ID"
    }
  ],
  "items": [
    {
      "id": "P100_NAME",
      "type": "text",
      "region": "r1",
      "binding": "NAME"
    }
  ],
  "buttons": [...],
  "dynamicActions": [
    {
      "event": "P100_TYPE_CHANGE",
      "actions": [
        { "type": "show", "target": "P100_DISCOUNT" }
      ]
    }
  ],
  "processes": [...],
  "branches": [...]
}
```

**T-207 核心吸收**：后端规则应能用 Schema 表达，而非必须写 Java 代码。

Infor Mongoose 接近传统企业软件开发平台，对 PC 端企业表单很有参考价值。

| Mongoose 思想 | T-207 转化 |

### 3.6 ServiceNow UI Builder

ServiceNow UI Builder 是 2019 年后成熟的现代产品，采用 React 架构。

#### 3.6.1 核心概念

| 概念 | 说明 | T-207 对应 |
|------|------|-----------|
| Page | 页面声明式定义 | AppPage |
| Container | 布局容器 | LayoutContainer |
| Component | 可复用 UI 组件 | ComponentRegistry |
| Data Resource | 数据源抽象 | DataSource |
| Event | 组件间通信 | EventBus |
| Client State | 运行时状态 | RuntimeState |

#### 3.6.2 Data Resource 抽象（关键创新）

```json
{
  "dataResources": [
    {
      "id": "dr1",
      "name": "CustomerRecord",
      "type": "TableRecord",
      "table": "sn_customers",
      "fields": ["id", "name", "type", "discount"]
    },
    {
      "id": "dr2",
      "name": "OrderList",
      "type": "TableRecord",
      "table": "sn_orders",
      "query": "customer_id = :dr1.id"
    }
  ]
}
```

**关键创新**：同一个页面组件可以绑定不同 Data Resource，数据源切换不改变 UI。T-207 应借鉴此设计，实现**数据源与 UI 分离**。

#### 3.6.3 页面 Schema 结构

```json
{
  "id": "page_customers_edit",
  "containers": [
    {
      "id": "header",
      "type": "Header",
      "components": ["CustomerTitle", "CustomerActions"]
    },
    {
      "id": "body",
      "type": "TwoColumn",
      "left": ["CustomerForm"],
      "right": ["OrderList"]
    }
  ],
  "dataResources": [...],
  "events": [
    {
      "trigger": "CustomerForm.save",
      "actions": [
        { "type": "call", "resource": "CustomerRecord.update" },
        { "type": "notify", "message": "保存成功" }
      ]
    }
  ]
}
```

### 3.7 风险与避免倾向

以下内容不是否定对应产品，而是提示 T-207 在吸收时需要避开的倾向。后续需要通过官方文档、截图和试用环境继续补强证据。

| 产品 | 需要避免的倾向 | 风险 | T-207 对应方案 |
|------|-----------|------|---------------|
| Salesforce Force.com | 过度绑定单一平台对象模型 | 若只围绕某一类对象建模，会限制多数据源和异构系统集成 | 支持多数据源（REST/SOAP/DB直连） |
| OutSystems | 过度依赖编译期强约束 | 对动态表单、热更新和多租户差异化配置不够友好 | Schema 运行时解释 + 版本管理 |
| Mendix | 过度模型化 | 复杂业务若全部进入模型，可能增加治理和理解成本 | 规则声明式 + 代码扩展点 |
| 活字格 / 简道云 | 面向小白的简洁设计 | 限制专业场景扩展性 | 区分"配置模式"和"代码模式" |
| 金蝶 BOS 传统版 | 桌面 IDE 模式 | 不能纯 Web 化 | 必须是纯 Web 设计器 |
| 传统报表工具 | 绝对定位布局 | 不适合响应式 Web | Grid/Flex 布局为主 |

## 6. Schema 运行时模式对比

文档中各产品的 Schema 性质差异是选型的关键维度：

| 产品 | Schema 性质 | 表单能力边界 | T-207 定位 |
|------|-----------|-------------|-----------|
| Power Apps | 运行时解释型 | 强但无法突破组件库 | 参考其易用性 |
| Salesforce Lightning | 编译+运行时混合 | 受 SObject 强约束 | 吸收权限模型 |
| OutSystems | 编译时强类型 | 可以突破但需专业模式 | 避免其复杂度 |
| Mendix | 模型驱动 | 强业务语义，弱布局 | 吸收业务模型 |
| Oracle APEX | 运行时声明式 | 配置即代码 | **核心参考** |
| ServiceNow UI Builder | 运行时声明式 | 受 Table / Record / Data Resource 模型约束 | **核心参考** |

**T-207 Schema 设计原则**：
- **运行时声明式 + 扩展点机制**：既能快速配置，又能支持代码组件扩展
- **Schema 描述 UI + 数据 + 规则，不描述实现**：后端逻辑通过扩展点注入
- **支持 Schema 版本管理和灰度发布**

## 7. 中国市场特殊性考量

T-207 的参考产品以欧美为主，但中国企业场景有独特需求：

| 需求 | 欧美产品方案 | 中国企业需求 | T-207 应对 |
|------|------------|-------------|----------|
| 复杂表头 | 通常无 | ✓✓✓（采购申请需部门+项目+供应商+物料多级表头） | TableEngine 支持合并单元格 |
| 打印套打 | 通常无 | ✓✓✓（发票、出库单必须套打） | 暂不做，但 Schema 应能描述"仅数据"vs"含布局"两种模式 |
| 批量操作 | Excel 为主 | ✓✓✓（一次性处理 100 行分录） | 分录支持批量编辑模式 |
| 手写签名 | 无 | ✓✓（审批流中常见） | 预留 Signature 组件 |
| 套打纸模板 | 无 | ✓✓✓ | 暂不做，需预留扩展点 |
| 多级审批意见 | 无 | ✓✓✓（部门长→总监→总经理多级签章） | 审批面板支持多级嵌套 |
| 印章图片 | 无 | ✓✓（合同审批需盖章图片） | 预留 Stamp 组件 |

## 9. T-207 推荐产品形态

### 9.1 工作台结构

```text
+--------------------------------------------------------------------------------+
| Top Command Bar：保存 / 撤销 / 预览 / 校验 / 发布 / 角色预览 / 断点              |
+----------------------+----------------------------------------+----------------+
| Left Resource Dock    | Center Runtime Design Surface           | Right Inspector|
| - Layout Containers   | - 真实 PC Web 页面                       | - 属性          |
| - BO Fields           | - Designer Overlay                      | - 布局          |
| - Form Elements       | - Snaplines / Drop Target               | - 表格          |
| - Bill Components     | - Selection / Resize Handles            | - 规则          |
| - Table Widgets       | - Runtime Preview                       | - 权限          |
| - Page Outline        |                                        | - 数据绑定      |
| - Data Sources        |                                        | - 性能          |
+----------------------+----------------------------------------+----------------+
| Bottom Analyzer：错误 / 警告 / 性能 / 权限冲突 / 规则影响 / 数据绑定断裂        |
+--------------------------------------------------------------------------------+
```

### 9.2 核心引擎

| 引擎 | 职责 |
|---|---|
| LayoutEngine | Grid/Flex/Dock/Split/Scroll/Sticky/Portal 布局计算 |
| DesignerOverlay | 从真实 DOM 读取坐标，绘制选中框、吸附线、投放点 |
| ComponentRegistry | 注册布局容器、字段控件、业务组件、表格组件 |
| TableEngine | 分录、强表格、嵌套明细、虚拟滚动、编辑状态、复杂表头 |
| RuleEngine | 字段联动、显隐、必填、只读、计算、校验 |
| PermissionEngine | 页面、区块、字段、按钮、表格、行列单元格权限 |
| **SchemaVersionEngine** | Draft/Published/Archived 多版本管理，依赖树分析，破坏性变更检测 |
| **DataBindingEngine** | 字段-数据列映射、级联刷新、跨表单引用 |
| **TemplateEngine** | 行业模板（采购单、销售单）、布局模板、组件组合模板 |
| PublishAnalyzer | 保存/发布前质量门禁 |
| RuntimeRenderer | 设计态、预览态、运行态同源渲染 |

### 9.3 Schema 版本管理模型

```json
{
  "schemaId": "page_customer_edit_v1",
  "versions": [
    {
      "version": 12,
      "status": "draft",
      "basePublishedVersion": 11,
      "createdBy": "user_001",
      "createdAt": "2026-07-09T10:00:00Z",
      "content": { /* 完整 Schema */ }
    },
    {
      "version": 11,
      "status": "published",
      "publishedBy": "user_001",
      "publishedAt": "2026-07-09T12:00:00Z",
      "content": { /* 发布版本 */ },
      "dependencies": ["ds_customer", "rule_discount_calc"]
    },
    {
      "version": 10,
      "status": "archived",
      "archivedBy": "user_002",
      "archivedAt": "2026-07-10T08:00:00Z",
      "content": { /* 历史版本 */ }
    }
  ]
}
```

`version` 只表示正整数版本号，`status` 表示 `draft / published / archived` 生命周期状态，二者不能混用。

## 10. 设计取舍

### 10.1 必须做（路线图总目标）

本节表示 T-207 全新表单设计器相对竞品必须具备的总能力，不等同于 P0 全部完成范围。P0 / P1 / P2 的落地边界以《产品定位与范围》为准。

1. 五区 IDE 工作台。
2. 中央真实 Web 页面画布。
3. 强 Web 布局容器。
4. 表单字段和业务对象绑定。
5. 强表格 / 分录 / 母子嵌套：P0 支持 Level 1，P1 支持 Level 2，P2 支持 Level 3。
6. 属性、大纲、画布三向同步。
7. 规则、权限、流程、状态一体化。
8. 发布前分析器。
9. Schema 版本管理（Draft/Published/Archived）。
10. 多数据源绑定支持：P1。
11. 中国式复杂表头支持：P1。

### 10.2 暂不做

1. 报表设计。
2. 打印套打（预留扩展点）。
3. 移动端优先设计。
4. BI 透视分析。
5. 无限嵌套表格（P0=Level 1，P1=Level 2，P2=Level 3）。
6. 任意 JS 脚本自由执行。
7. Excel 导入导出（预留扩展点）。

### 10.3 明确禁止

| 禁止 | 原因 |
|---|---|
| 主体绝对定位 | 不适合复杂 Web 页面和响应式 |
| 彩色卡片墙控件库 | 不符合企业实施工具 |
| 表格用普通 HTML table 表示 | 无法支撑企业分录 |
| 规则散落在控件脚本里 | 无法分析和治理 |
| 只设计静态 UI | 无法形成可发布运行的平台能力 |
| 强绑定单一数据源 | 必须支持多数据源灵活切换 |
| Schema 无版本管理 | 企业平台必须有版本控制 |
