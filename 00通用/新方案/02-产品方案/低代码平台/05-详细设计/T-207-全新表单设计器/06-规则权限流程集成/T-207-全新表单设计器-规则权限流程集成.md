## 1. 设计目标

规则权限流程集成要回答一个核心问题：

```text
设计器如何把字段、表格、按钮、页面和流程状态连接起来，
让企业单据在不同角色、不同业务状态、不同数据条件下表现正确、可分析、可测试、可发布。
```

T-207 的规则不是控件脚本，也不是任意 JS。它必须是可声明、可分析、可定位、可回放的业务配置。

规则权限流程必须做到：

1. 字段、表格、按钮、区块的显隐、必填、只读、可编辑可由规则驱动。
2. 页面操作必须受状态机和权限控制。
3. 规则结果必须能合并到 FieldNode、TableColumn、ActionButton 的最终运行态。
4. Analyzer 必须能检查规则循环、目标不存在、权限冲突、状态不可达和发布风险。
5. 自动化测试平台必须能模拟角色、状态、字段输入、按钮点击和 Analyzer 定位。

## 3. 总体架构

```text
RuleSchema
PermissionSchema
StateMachineSchema
ActionPolicySchema
  ↓
RuleEngine
PermissionEngine
StateMachineEngine
ActionEngine
  ↓
RuntimeStateResolver
  ↓
Field / Table / Action / Section Final State
  ↓
RuntimeRenderer
  ↓
RulePermissionAnalyzer
```

| 模块 | 职责 | P0 |
|---|---|---|
| RuleRegistry | 注册规则类型和目标对象类型 | 入口 |
| RuleEngine | 求值 visible/required/readonly/compute/validate | P1 |
| PermissionEngine | 求值角色、流程、数据状态权限 | P1 |

| RulePermissionAnalyzer | 发布前检查规则权限流程问题 | 是 |

## 4. RuntimeState 合并协议

### 4.1 最终状态来源

一个字段、列、按钮或区块的最终状态来自多层输入：

```text
finalState =
  staticState
  + staticPermission
  + ruleResult(P1)
  + permissionResult(P1)
  + stateMachineResult(P1)
```

P0 只实现 `staticState + staticPermission` 的合并协议，P1 接入动态结果。

### 4.2 字段最终状态

| 状态 | 合并规则 |
|---|---|
| finalVisible | static.visible AND staticPermission.visible AND rule.visible(P1) |
| finalRequired | state.required OR validation.required OR rule.required(P1) |
| finalReadonly | state.readonly OR permission.readonly OR permission.editable=false OR rule.readonly(P1) OR stateMachine.readonly(P1) |
| finalDisabled | state.disabled OR stateMachine.disabled(P1) |
| finalMasked | permission.mask OR dataPolicy.mask(P1) |

P0 只求值静态部分：

```text
P0 finalReadonly = state.readonly OR permission.readonly OR permission.editable=false
P0 finalDisabled = state.disabled
```

`rule.*`、`PermissionEngine.*`、`stateMachine.*` 只在 P1 动态求值时参与。P0 可以声明 StateMachineSchema，但状态机不驱动字段只读或禁用。

冲突处理：

| 冲突 | 结果 |
|---|---|
| hidden + required | Analyzer Warning，P1 动态规则中可升级为 Error |
| readonly + required 且无默认值 | Analyzer Warning |
| disabled + required 且无默认值 | Analyzer Error |
| mask + editable | Analyzer Warning |

### 4.3 表格最终状态

| 对象 | P0 静态状态 | P1 动态扩展 |
|---|---|---|
| EntryTable | visible、editable、readonly | 按角色/流程/数据状态控制 |
| TableColumn | visible、editable、readonly、mask | 行列级动态权限 |
| RowAction | visible、enabled | 行级状态策略 |
| DetailPanel | visible、readonly | 行级展开条件 |

P0 不做行级动态权限，只保留状态合并协议和 Analyzer 入口。

## 5. RuleSchema

### 5.1 定位

RuleSchema 是 P1 完整规则能力的核心。P0 只展示规则摘要和入口，不允许配置不可分析的复杂规则。

### 5.2 Schema 示例

```json
{
  "id": "rule_credit_required",
  "type": "Rule",
  "name": "creditRequiredForKeyCustomer",
  "trigger": {
    "type": "fieldChange",
    "source": "field_customerType"
  },
  "condition": {
    "op": "eq",
    "left": { "type": "field", "path": "customerType" },
    "right": { "type": "literal", "value": "KEY" }
  },
  "actions": [
    {
      "type": "setRequired",
      "target": "field_creditLimit",
      "value": true
    }
  ],
  "stage": "P1",
  "enabled": true
}
```

### 5.3 P1 规则类型

| 规则 | 说明 |
|---|---|
| visibleWhen | 条件满足时显示对象 |
| requiredWhen | 条件满足时必填 |
| readonlyWhen | 条件满足时只读 |
| disabledWhen | 条件满足时禁用 |
| computeValue | 计算字段值 |
| validate | 自定义声明式校验 |
| filterOptions | 过滤枚举/引用选择项 |
| expandDetailWhen | 控制明细展开入口 |

P1 规则仍禁止任意 JS。表达式必须使用受控表达式 AST。

### 5.4 表达式 AST

```json
{
  "op": "and",
  "args": [
    {
      "op": "gt",
      "left": { "type": "field", "path": "amount" },
      "right": { "type": "literal", "value": 10000 }
    },
    {
      "op": "eq",
      "left": { "type": "context", "path": "currentUser.role" },
      "right": { "type": "literal", "value": "manager" }
    }
  ]
}
```

P0 不实现表达式编辑器，但 Analyzer 必须能识别规则对象存在、目标对象存在和 P0 不可发布 P1 未启用能力。

## 6. PermissionSchema

### 6.1 权限对象

| 对象 | P0 静态权限 | P1 动态权限 |
|---|---|---|
| Page | visible、access | role、org、state |
| Section | visible、readonly | role、state |
| FieldNode | visible、editable、readonly、mask | role、state、data condition |
| EntryTable | visible、editable、readonly | role、state、data condition |
| TableColumn | visible、editable、readonly、mask | role、state、row condition |
| RowAction | visible、enabled | role、row state |
| ActionButton | visible、enabled | role、workflow state |

### 6.2 Schema 示例

```json
{
  "id": "perm_sales_amount",
  "type": "Permission",
  "target": "field_amount",
  "scope": "field",
  "static": {
    "visible": true,
    "editable": false,
    "readonly": true,
    "mask": false
  },
  "dynamic": {
    "stage": "P1",
    "roles": ["sales_manager"],
    "states": ["submitted", "approved"],
    "effect": {
      "readonly": true
    }
  }
}
```

P0 只保存和求值 `static`。`dynamic` 在 P0 中允许作为不可编辑占位，但 Analyzer 不能把它当作已生效能力。

### 6.3 权限矩阵

| 权限 | 可见 | 可编辑 | 可执行 | 说明 |
|---|---|---|---|---|
| visible=false | 否 | 否 | 否 | 不渲染或占位隐藏 |
| readonly=true | 是 | 否 | 否 | 只读展示 |
| editable=false | 是 | 否 | 否 | 和 readonly 同向，语义偏权限 |
| enabled=false | 是 | 否 | 否 | 按钮/动作禁用 |
| mask=true | 是 | 否 | 否 | 脱敏展示 |

字段、列、表格对象中 `editable=false` 等价于静态只读来源之一，最终进入 `finalReadonly`。如果同一对象同时配置 `editable=true` 和 `readonly=true`，以 `readonly=true` 为准，并由 Analyzer 输出 `PERMISSION_STATIC_CONFLICT` Warning。按钮和动作对象使用 `enabled=false` 表达不可点击，不使用 `readonly`。

## 7. StateMachineSchema

### 7.1 P0 最小状态

| 状态 | 说明 |
|---|---|
| draft | 草稿，可编辑 |
| submitted | 已提交，只读或部分只读 |
| approved | 已审核，只读 |
| archived | 已归档，不可编辑 |

P0 只提供状态显示和基础动作入口，不交付完整流程引擎。

### 7.2 Schema 示例

```json
{
  "id": "sm_sales_order",
  "type": "StateMachine",
  "initialState": "draft",
  "states": [
    { "id": "draft", "title": "草稿" },
    { "id": "submitted", "title": "已提交" },
    { "id": "approved", "title": "已审核" },
    { "id": "archived", "title": "已归档" }
  ],
  "transitions": [
    {
      "id": "submit",
      "from": "draft",
      "to": "submitted",
      "action": "submitBill"
    },
    {
      "id": "approve",
      "from": "submitted",
      "to": "approved",
      "action": "approveBill"
    }
  ]
}
```

### 7.3 状态对 UI 的影响

| 状态 | 默认影响 |
|---|---|
| draft | 页面和字段默认可编辑 |
| submitted | 主字段只读，审批动作可见 |
| approved | 页面只读，反审核动作按权限显示 |
| archived | 全页面只读，隐藏编辑类动作 |

完整状态到字段/表格/按钮的动态策略归 P1。

## 8. ActionButton 与 ActionPolicy

### 8.1 ActionButton Schema

```json
{
  "id": "action_submit",
  "type": "ActionButton",
  "title": "提交",
  "action": {
    "type": "submitBill",
    "transition": "submit"
  },
  "state": {
    "visible": true,
    "enabled": true
  },
  "permission": {
    "visible": true,
    "enabled": true
  },
  "confirm": {
    "enabled": true,
    "message": "确认提交当前单据？"
  }
}
```

### 8.2 P0 动作

| 动作 | 说明 |
|---|---|
| saveDraft | 保存草稿 |
| validate | 触发 Analyzer 校验 |
| submitBill | 提交单据 |
| approveBill | 审核单据，占位动作 |
| rejectBill | 驳回单据，占位动作 |
| archiveBill | 归档单据 |

P0 动作只定义 UI 和状态入口。真正工作流、审批流、回调动作、插件动作归 P1/P2。

### 8.3 ActionPolicy

| 策略 | P0 | 说明 |
|---|---|---|
| visible | 是 | 是否展示动作 |
| enabled | 是 | 是否可点击 |
| confirm | 是 | 是否二次确认 |
| analyzerBeforeRun | 是 | 执行前是否触发 Analyzer |
| transitionGuard | P1 | 状态流转条件 |
| roleGuard | P1 | 角色条件 |
| dataGuard | P1 | 数据条件 |

发布动作必须经过 Analyzer，Error 未清零时不能发布。

P0 的 `analyzerBeforeRun` 指设计时 Schema Analyzer：在动作执行前检查 ActionButton 绑定的 action、transition、target 是否存在且处于可发布状态。它不是运行时业务校验引擎，也不执行 P1 规则表达式。

## 9. Inspector 设计

### 9.1 规则 Tab

P0 中规则 Tab 是摘要和入口，不是完整规则编辑器。

| 区域 | P0 |
|---|---|
| 规则摘要 | 显示对象被多少规则影响 |
| 规则入口 | 提示 P1 开启后可编辑 |
| 静态状态提示 | 显示当前对象静态 visible/required/readonly |
| Analyzer 问题 | 显示与该对象相关的规则问题 |

P1 规则 Tab 才显示触发器、条件、动作、目标对象和影响分析。

### 9.2 权限 Tab

| 区域 | P0 | P1 |
|---|---|---|
| 静态权限 | visible、editable、readonly、mask、enabled | 是 |
| 角色矩阵 | 只读占位 | 可编辑 |
| 流程状态矩阵 | 只读占位 | 可编辑 |
| 数据条件 | 不显示 | 可编辑 |

### 9.3 流程 Tab / 状态面板

| 区域 | P0 |
|---|---|
| 当前状态 | draft / submitted / approved / archived |
| 可用动作 | 根据静态状态显示 |
| 状态流转图 | P1 |
| 审批节点配置 | P1 |

## 10. Analyzer

### 10.1 Analyzer 输出结构

```json
{
  "severity": "error",
  "code": "RULE_TARGET_MISSING",
  "message": "规则目标对象不存在。",
  "nodeId": "rule_credit_required",
  "schemaPath": "rules.rule_credit_required.actions[0].target",
  "quickFix": "选择有效目标对象"
}
```

### 10.2 P0 规则清单

| 规则 | 级别 | 说明 |
|---|---|---|
| RULE_P1_NOT_ENABLED | Error | P0 发布版本包含启用的 P1 动态规则 |
| RULE_TARGET_MISSING | Error | 规则目标对象不存在 |
| RULE_TRIGGER_MISSING | Error | P0 仅做导入/遗留 RuleSchema 结构检查；P1 规则编辑器中实际触发 |
| RULE_ACTION_UNSUPPORTED | Error | P0 仅检查导入/遗留 RuleSchema 是否含当前阶段不支持动作；P1 规则编辑器中实际触发 |
| RULE_CYCLE_DETECTED | Error | 规则之间形成循环依赖 |
| PERMISSION_TARGET_MISSING | Error | 权限目标对象不存在 |
| PERMISSION_STATIC_CONFLICT | Warning | 静态权限冲突 |
| ACTION_TRANSITION_INVALID | Error | 按钮绑定的状态流转不存在 |
| ACTION_PERMISSION_CONFLICT | Warning | 动作可见但不可执行 |
| STATE_UNREACHABLE | Warning | 状态机存在不可达状态 |
| STATE_TRANSITION_BROKEN | Error | 状态流转 from/to 不存在 |

### 10.3 P1 Analyzer 扩展

| 规则 | 说明 |
|---|---|
| RULE_CONDITION_INVALID | 条件表达式非法 |
| RULE_TARGET_TYPE_MISMATCH | 动作与目标对象类型不匹配 |
| RULE_IMPACT_TOO_BROAD | 单条规则影响对象过多 |
| PERMISSION_ROLE_MISSING | 引用的角色不存在 |
| PERMISSION_STATE_CONFLICT | 角色/流程/数据权限冲突 |
| RULE_PERFORMANCE_RISK | 规则求值成本过高 |

### 10.4 schemaPath 规范

| 场景 | schemaPath 示例 |
|---|---|
| 规则 | `rules.rule_credit_required` |
| 规则条件 | `rules.rule_credit_required.condition.args[0]` |
| 规则动作 | `rules.rule_credit_required.actions[0]` |
| 权限 | `permissions.perm_sales_amount.static.readonly` |
| 状态流转 | `stateMachine.transitions[1]` |
| 动作按钮 | `components.action_submit.action.transition` |

Analyzer 必须能从问题定位到对应规则、权限、状态或按钮，并打开相应 Inspector Tab。

本文扩展 04/05 的 schemaPath 命名空间：

| 命名空间 | 说明 |
|---|---|
| `components.{id}` | 页面组件、字段、表格、按钮 |
| `components.{id}.columns[n]` | 表格列 |
| `rules.{id}` | 规则 |
| `permissions.{id}` | 权限配置 |
| `stateMachine.*` | 状态机 |

完整统一规范由 07-Schema 与运行时架构文档集中定义，本文不得另起不兼容路径格式。

## 11. 自动化测试契约

### 11.1 DOM 标识

| 对象 | data-testid |
|---|---|
| 规则列表项 | `rule-item-{ruleId}` |
| 权限矩阵行 | `permission-row-{targetId}` |
| 状态徽标 | `state-badge-{stateId}` |
| 动作按钮 | `action-button-{actionId}` |
| 规则问题 | `analyzer-issue-{issueId}` |

这些 testId 属于工作台统一自动化命名空间的扩展：字段沿用 `field-*`，表格根节点沿用 `designer-node-*`，表格内部使用 `table-*`，规则权限流程使用 `rule-*`、`permission-*`、`state-*`、`action-*`。完整规范由 07 文档统一收口。

### 11.2 自动化动作

| 动作 | 模拟用户行为 | 预期断言 |
|---|---|---|
| clickAction(actionId) | 点击动作按钮 | 动作执行或阻断原因显示 |
| switchState(stateId) | 预览态切换状态 | 字段/表格/按钮最终状态刷新 |
| switchRole(roleId) | P1 角色预览 | 权限结果刷新 |
| inspectRule(ruleId) | 点击规则列表项 | Inspector 打开规则摘要 |
| assertFinalState(targetId, state) | 检查运行态最终状态 | visible/readonly/required 等状态正确 |
| clickAnalyzerIssue(issueId) | 点击问题 | 定位规则/权限/动作目标 |

### 11.3 Playwright 示例

```ts

  targetId: 'field_amount',
  readonly: true
});

await page.getByTestId('analyzer-issue-rule_target_missing').click();
await expect(page.getByTestId('rule-item-rule_credit_required')).toBeVisible();
```

## 12. 发布门禁

P0 发布必须满足：

| 门禁 | 规则 |
|---|---|
| 无 Analyzer Error | 所有 Error 清零 |
| 不启用 P1 动态规则 | P0 版本不能含 enabled=true 的 P1 RuleSchema |
| 权限目标存在 | Permission target 必须能定位到 Page/Section/Field/Table/Column/Action |
| 动作目标存在 | ActionButton 绑定的 action 和 transition 必须有效 |
| 状态机基本可达 | initialState 存在，transition from/to 有效 |
| Error 不可忽略 | 只能忽略 Warning/Suggestion |

## 14. P0 验收标准

| 标准 | 验收方式 |
|---|---|
| 静态权限可配置 | 字段/表格/按钮可设置 visible/editable/readonly |
| 动作按钮可配置 | 保存、提交、审核、归档动作有 Schema |
| 状态可显示 | 页面可显示 draft/submitted/approved/archived |
| P1 规则不误当 P0 | P0 发布启用 P1 规则时 Analyzer Error |
| 权限冲突可提示 | hidden+required、readonly+required 等被 Analyzer 发现 |
| 规则目标可定位 | 规则目标不存在时 Analyzer 可定位 |
| 状态流转可检查 | transition from/to 无效时 Error |
| 自动化可模拟 | 可点击动作、切换状态、定位 Analyzer 问题 |
