> 版本：v0.1

## 2. T-107 业务事件中心

### 2.1 元数据

```text
EventDef:
  eventCode
  sourceType: platform/object/action/schedule
  objectCode
  payloadSchemaVersion
  syncAllowed
  retentionPolicy

EventSubscriptionDef:
  eventCode
  subscriberType: rule/workflow/webhook/plugin/notification
  subscriberRef
  phase: before/after
  mode: sync/async
  filterCondition
  retryPolicy
```

### 2.2 执行流程

```text
DataWriteService / ActionExecutor / PublishService / Scheduler
  -> EventPublisher.publish(event)
  -> before sync subscriptions
  -> business transaction
  -> after async subscriptions into outbox
```

### 2.3 必测

- before 订阅可阻断事务，阻断时无业务写入。
- after 订阅进入 outbox，重复投递按 eventId 幂等。
- payload 含 metaHash/traceId，敏感字段按 B6 脱敏。
- rule 消费者不再直接绑定对象 trigger，而是消费 EventSubscription。

## 3. T-207/T-208 规则动作与定时自动化

### 3.1 create_record/update_record

规则 config：

```text
targetObjectCode
fieldMappings[]
executionPhase: after_event
idempotencyStrategy
maxDepth
onFailure: retry/dead_letter/block
```

约束：

- 目标写入必须走 DataWriteService。
- 不允许直接拼 SQL 或绕过目标对象权限。
- 默认异步执行；同步执行必须有 ADR 或白名单。

### 3.2 schedule

规则 config：

```text
cron
tenantTimezone
targetObjectCode
filters
actionCode
missedPolicy: skip/run_once/run_all_limited
concurrencyPolicy: forbid/replace/parallel_limited
```

必测：

- 双实例同一 schedule 只触发一次。
- 租户时区边界正确。
- 目标 filters 不能绕过 data_scope。

## 4. T-209 rollup

### 4.1 FieldDef options

```text
sourceKind: table/link/multilink
sourceField
aggregateField
aggregateFn: count/sum/avg/max/min
filterCondition
materialized: true
recomputePolicy: same_transaction/outbox
```

### 4.2 阻断与实现边界

- M2 前 `field_type=rollup` 可保存，发布阻断。
- M2 实现前必须补 rollup ADR，选择同事务重算或 outbox 重算。
- rollup 不得泄露源字段无权限值。

必测：

- 子表增删改触发主表汇总变化。
- 循环依赖被阻断。
- 大批量重算进入配额和告警。

## 5. T-108 复合唯一与 IdentityProvider SPI

### 5.1 复合唯一

BusinessObject options：

```text
uniqueConstraints:
  - code
    fieldCodes[]
    message
```

DDL：

```text
unique key uk_xxx (tenant_id, app_id, field_a, field_b, delete_token)
```

必测：

- 未删除记录重复组合拒绝。
- 软删除后同组合可重建。
- 错误返回 constraintCode 和 fieldCodes。

### 5.2 IdentityProvider SPI

M1 只预留接口：

```text
IdentityProvider:
  authenticate()
  mapUser()
  syncOrg()
  rotateSecret()
```

M3 落地前必须补组织同步冲突策略：外部组织删除、重命名、合并、员工转岗均不得直接重写历史 owner 快照。

### 6.1 T-210 业务数据 Excel 导入导出执行设计

导入元数据：

```text
ImportTask:
  taskNo / tenantId / appId / objectCode
  sourceFileRef / fileHash / status
  mappingVersion / columnMappings / dryRunReportRef
  batchSize / idempotencyKey / createdBy / traceId

ImportRowResult:
  rowNo / status / recordLid / errorCode / errorMessage / fieldPath

ExportTask:
  taskNo / objectCode / filtersAst / selectedFields / accessViewHash
  status / fileRef / expireAt / downloadedCount / traceId
```

执行流程：

```text
upload
-> virus/size/type check
-> header parse
-> column mapping
-> dry-run through FieldTypeHandler + AccessView + DataWriteService validator
-> commit by batches
-> row result report
```

规则：

1. 导入默认分批事务，每批最多 500 行；批内失败回滚该批，已成功批次保留并进入结果报告。
2. 试跑和正式导入必须使用同一套 FieldTypeHandler 转换、必填校验、唯一约束校验和 AccessView 写权限，不允许导入链路另写校验器。
3. 导入可显式声明规则豁免清单；豁免只允许跳过通知类副作用，不允许跳过类型转换、租户注入、唯一约束、权限和审计。
4. 导出字段集合必须来自 AccessView，MASKED 字段按脱敏值导出；NONE 字段即使在保存视图中出现也必须剔除。
5. CSV/Excel 单元格以 `=`, `+`, `-`, `@` 开头时按安全策略转义；导出文件设置保留期、下载鉴权和 EXPORT 事件。

### 6.2 T-211 OptionSet 执行设计

```text
OptionSetDef:
  code / name / scope(app)
  options: [{code, label, i18nKey, color, sortNo, status}]
  sourceKind / version / status
```

规则：

1. select/multiselect 可引用 OptionSet，也可继续使用字段内联 options；同一字段二者只能选一种。
2. option code 只增不复用；停用选项不得影响历史数据展示，历史值显示为“已停用”状态。
3. OptionSet 变更进入发布流程和 Version.snapshot，运行态只读已发布版本。
4. 选项显示名支持 i18nKey，但 M1~M3 只保证中文值可用，完整语言包进入 M4。

### 6.3 T-212 SavedView 与看板扩展执行设计

```text
SavedViewDef:
  viewCode / objectCode
  ownerType: user/role/app
  filtersAst / columns / sorts / groupBy / density
  isDefault / version / status
```

规则：

1. SavedView 保存的是用户偏好，不是权限凭证；每次使用必须重新套当前 AccessView。
2. 无权限字段出现在历史 SavedView 中时，运行时静默剔除并在视图设置界面提示“字段已不可用或无权限”。
3. 看板 groupBy 字段允许 state/select/user/link；字段必须可读且具备离散值能力，富文本、附件、子表、formula 默认不可分组。
4. 看板拖动改变分组值时必须走 DataWriteService update，不能在前端直接 patch 字段。

### 6.4 T-213 Masked 权限执行设计

AccessView 字段视图扩展为：

```text
NONE < MASKED < READ < WRITE
```

规则：

1. MASKED 是可读的降级态，不允许提交写入；WRITE 隐含 READ，但不隐含查看密钥明文。
2. 脱敏策略由字段数据分级和 fieldType 决定，例如手机号、邮箱、证件号、金额、token 分别注册策略。
3. API、列表、详情、导出、打印、通知、审计 diff、API Token 访问必须消费同一个 masking service。
4. MASKED 字段不得作为精确筛选条件暴露给无 READ 权限用户，避免通过查询结果反推明文。

### 6.5 T-214 站内消息执行设计

```text
MessageDef:
  messageLid / receiverUserLid
  sourceEventId / targetRef(app, object, recordLid)
  title / summary / readStatus / priority
  expireAt / createdAt / traceId

SubscriptionSetting:
  userLid / eventCode / channel(in_app) / enabled
```

规则：

1. notify 默认通道为站内消息；邮件、企微、钉钉等外部通道进入 T-306。
2. 消息 summary 只能保存脱敏摘要，不保存完整字段 diff 或敏感明文。
3. @ 提及、关注记录、规则 notify 都通过 EventSubscription 生成 Message，不允许直接写消息表。
4. 消息跳转到目标记录时重新执行 AccessView；无权限时只展示“记录不可访问或已删除”的模糊提示。

### 7.1 T-311 ConvertRule 单据下推执行设计

```text
ConvertRuleDef:
  code / sourceObjectCode / targetObjectCode
  actionCode / fieldMappings / lineMappings
  sourceRefPolicy / idempotencyPolicy / enabled

LinkTrace:
  sourceObjectCode / sourceRecordLid
  targetObjectCode / targetRecordLid
  convertRuleCode / traceType / quantity / amount
```

规则：

1. M3 只支持 1:1 单级下推；1:N、合并下推、拆分下推和跨应用下推进入 M4。
2. 下推动作先校验源单 read/action 权限，再校验目标单 create 权限；字段映射结果仍按目标对象 FieldTypeHandler 转换。
3. 重复下推使用 `sourceRecordLid + convertRuleCode + actionCode` 幂等键；是否允许同源多次下推必须由 ConvertRule 显式声明。
4. LinkTrace 参与权限裁剪：上查/下查只能显示当前用户对目标对象有 read 权限的记录。

### 7.2 T-312 打印模板执行设计

```text
PrintTemplateDef:
  templateCode / objectCode / templateVersion
  layout / fieldPlaceholders / tableBlocks / headerFooter
  archivePolicy / status
```

规则：

1. 打印输入是对象记录快照 + AccessView；模板不能绕过字段权限、MASKED 脱敏和 data_scope。
2. 模板表达式只允许 lowcode-expression 的安全子集和白名单格式化函数，禁止自由 HTML、脚本、iframe、远程资源和事件属性。
3. 服务端渲染 HTML/PDF，渲染结果写 PRINT/EXPORT 类审计事件；需要归档时记录 templateVersion 和 metaHash。
4. 旧记录回放打印默认使用当时归档模板；未归档时必须提示模板已升级，不能伪装成历史原版。

### 7.3 T-313 Webhook 与外部事件订阅执行设计

```text
WebhookSubscription:
  subscriptionCode / eventCode / targetUrl
  secretRef / signatureAlg / retryPolicy / enabled
```

规则：

1. Webhook 是 EventSubscription 的 async subscriber，必须通过 outbox 投递。
2. targetUrl 必须经过 Connector URL Guard、DNS rebinding guard、协议白名单和租户级出网配额。
3. payload 只包含事件契约字段和脱敏摘要；敏感字段必须按 B6 分级裁剪。
4. 签名失败、目标 5xx、超时进入重试；超过阈值进入死信，支持人工重放。

### 7.4 T-314 API Token 执行设计

```text
ApiTokenDef:
  tokenCode / hashedSecret / scope(app, object, operation, fieldMask)
  expiresAt / rotatePolicy / rateLimit / status / lastUsedAt
```

规则：

1. token 明文只在创建时返回一次，数据库只存 hash；日志和审计只记录 tokenCode。
2. token 认证后构造 ExecutionContext，actor 类型标记为 api_token，并复用 AccessView。
3. scope 只能缩小权限，不能扩大 token 所属主体权限；fieldMask 与 AccessView 取交集。
4. 撤销、过期、轮换、超限、失败鉴权全部写安全审计事件。

### 7.5 T-315 聚合视图与基础图表执行设计

规则：

1. group by API 只允许对 AccessView 可读且 FieldTypeHandler 声明可聚合/可分组的字段执行。
2. 聚合查询必须先合并 data_scope，再执行 group by；不得用“总数差异”泄露无权限行存在性。
3. 图表 schema 只描述展示：chartType、dimension、metric、filters、sort；指标口径来自后端查询契约。
4. M3 只提供指标卡、柱状图、折线图、饼图；钻取、复杂多表指标、数据仓库语义进入 M4 或商业报表模块。

### 7.6 T-316 全局搜索执行设计

规则：

1. 搜索候选对象来自 MetaGraph 中启用 search 的对象，字段来自 title_field、autonumber 和 search_fields。
2. 每个候选对象单独计算 AccessView；无 read 权限对象不参与搜索，无字段 READ 权限的字段不参与匹配和摘要。
3. 首版使用 LIKE/前缀匹配，限制对象数、字段数、返回数和单租户频率；全文索引、拼音、相关性排序进入 M4。
4. 搜索结果只返回 recordLid、objectCode、title、masked summary 和跳转信息，不返回完整记录。

### 7.7 T-317 协作与记录时间线执行设计

```text
Comment:
  commentLid / objectCode / recordLid / content / mentions / status
Follow:
  objectCode / recordLid / userLid / followType
TimelineItem:
  sourceType(comment/audit/transition/message) / sourceId / occurredAt
```

规则：

1. 评论内容按富文本安全策略净化，禁止脚本、事件属性、危险链接和内嵌未授权附件。
2. 评论可见性不得高于记录 read 权限；用户失去记录权限后不再看到历史评论。
3. @ 提及只生成站内消息，不直接外发；外部通知由 T-306 通道治理决定。
4. 时间线复用 PlatformEvent、transition_log、Comment，不复制业务字段明文。

### 7.8 T-318 SSO 落地执行设计

规则：

1. M3 至少选择 OIDC、钉钉、企微之一落地；未落地的 provider 只保留 SPI 配置结构。
2. 外部账号映射必须有 stable externalId，不得只靠手机号或显示名。
3. 组织同步采用“外部变更生成同步计划 -> 预览冲突 -> 提交”的流程；删除/合并/重命名不得直接改写历史 owner 快照。
4. 用户禁用、退出租户、token 撤销的最大生效时限必须与 F7 权限缓存窗口一致。

### 7.9 T-319 回收站执行设计

规则：

1. 回收站只展示 soft deleted 且用户仍有 delete/restore 权限的记录。
2. restore 前必须校验唯一约束、引用完整性、父子表关系和目标对象状态；冲突时给出字段级影响报告。
3. purge 是高风险动作，必须二次确认、审计、保留期策略检查；默认不物理删除审计和 LinkTrace。
4. 恢复后 revision 增加，并写 DATA_WRITE/RESTORE 事件。

### 7.10 T-320 官方模板库执行设计

规则：

1. 官方模板以应用包形式分发，至少包含 CRM、请假、采购三类，安装复用 T-304 的签名、兼容矩阵、安全扫描和影响报告。
2. 模板 manifest 必须声明对象、字段、权限、页面、规则、所需插件、License 能力和升级策略。
3. 模板安装默认生成 customer sourceKind，不覆盖租户内同 code 应用；冲突时要求重命名或显式映射。
4. 模板升级必须生成差异报告，禁止覆盖客户扩展字段、规则、页面和权限。

## 8. M4 边界

M4 能力必须先补 ADR 后实现：

- 移动端/H5：移动端组件白名单、响应式布局、移动端权限裁剪。
- 多环境/灰度：环境模型、版本提升、回滚、灰度范围。
- 跨应用主数据：跨 app 引用、权限、版本和导入导出兼容。
- 自由脚本沙箱：语言、资源限制、依赖、审计和退出机制。
- 丰富 Connector 库：生态扩展、认证、配额、License。
- 国际化完整落地：语言包导入导出、运行时语言切换、打印模板多语言和审计语言回放。

M1~M3 的预留边界：

| 能力 | M1~M3 允许做 | M1~M3 禁止伪装成已完成 |
|---|---|---|
