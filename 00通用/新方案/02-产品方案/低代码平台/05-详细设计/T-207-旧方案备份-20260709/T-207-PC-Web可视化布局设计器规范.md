# T-207 PC Web 可视化布局设计器规范

> 适用范围：T-207 表单设计器、单据设计器、复杂业务页面设计器  
> 版本：v1.0  
> 日期：2026-07-08  
> 定位：以金蝶表单设计器为参考，但核心目标是设计可见即可得的 PC 端 Web 业务页面，布局能力必须超过普通表单搭建器。

---

## 1. 核心结论

T-207 不能只做“字段拖拽到表单”。它必须是一个 **PC Web 可视化布局设计器**。

它面向的不是简单表单，而是复杂企业页面：

| 页面类型 | 必须能设计出的结构 |
|---|---|
| 动态表单 | 多分组、多列字段、联动规则、权限态、局部折叠 |
| 业务单据 | 顶部命令栏、单据头、状态条、分录表格、汇总区、底部操作栏 |
| 主数据维护 | 基本信息、扩展属性、关联信息、审计信息、附件面板 |
| 审批页面 | 表单主体、流程轨迹、审批意见、附件、关联单据 |
| 复合工作台 | 左右分栏、局部滚动、页签、列表详情联动、固定操作区 |

判断是否超过金蝶，不看视觉是否更花，而看：

1. 是否能表达真实 PC Web 页面布局。
2. 设计态是否与运行态一致。
3. 是否支持 Grid、Flex、嵌套容器、局部滚动、Sticky、复杂组件组合。
4. 是否能校验布局约束，避免发布后错位、遮挡、截断。
5. 是否能把复杂单据页面设计成可维护的 schema，而不是一堆绝对定位节点。

---

## 2. 产品定位

### 2.1 正确定义

T-207 是：

```text
面向 PC 端企业 Web 页面的可见即可得设计器。
以表单和单据为核心场景，以强 Web 布局引擎为底座，
通过 RuntimeRenderer 保证设计态、预览态、运行态一致。
```

### 2.2 错误定义

T-207 不是：

| 错误理解 | 问题 |
|---|---|
| 控件拖拽器 | 只能摆字段，无法设计复杂页面结构 |
| 彩色组件展示墙 | 看起来丰富，但不能支撑生产配置 |
| 简单栅格表单 | 无法承载分录、审批、附件、关联单据等复合区域 |
| 纯绝对定位画布 | 与 Web 响应式和运行态 DOM 脱节，维护成本高 |
| 只面向移动端或小屏 | 本项目目标是 PC Web 企业页面 |

## 3. 目标页面模型

### 3.1 复杂业务单据页面

设计器必须能可视化设计以下页面：

```text
+--------------------------------------------------------------------------------+
| 顶部命令栏：保存 / 提交 / 审核 / 打印 / 预览 / 发布                              |
+--------------------------------------------------------------------------------+
| 单据状态条：编号、状态、版本、审批流、风险提示                                    |
+--------------------------------------+-----------------------------------------+
| 主内容区                              | 右侧辅助区                               |
| - 基本信息 FormGrid                   | - 审批记录 ApprovalPanel                  |
| - 客户/组织/日期/金额字段              | - 附件 AttachmentPanel                    |
| - 页签 Tabs                           | - 关联单据 RelatedBillPanel               |
| - 折叠分组 Collapse                   |                                         |
| - 分录 Subtable                       |                                         |
| - 汇总 SummaryBar                     |                                         |
+--------------------------------------+-----------------------------------------+
| 底部固定操作栏：保存草稿 / 提交审批 / 关闭 / 更多                                |
+--------------------------------------------------------------------------------+
```

### 3.2 必须支持的页面区块

| 区块 | 布局要求 |
|---|---|
| TopToolbar | 顶部 Sticky，支持主按钮、次按钮、更多菜单、状态入口 |
| StatusBar | 支持横向信息密集展示，支持告警与校验入口 |
| FormGrid | 支持 12/24 列、span、label 对齐、字段组、跨列说明 |
| Subtable | 支持横向滚动、固定列、列宽、行高、批量操作、行级状态 |
| SummaryBar | 可固定在分录下方或页面底部，支持金额合计、税额、折扣 |
| SidePanel | 支持审批、附件、关联单据、日志等辅助内容 |
| BottomActionBar | 底部 Sticky，支持主操作和状态机操作 |
| Drawer/Dialog | 发布影响、字段冲突、规则冲突、AI Patch 进入抽屉或弹窗 |

## 4. Web 布局能力

### 4.1 CSS Grid 能力

| 能力 | 要求 |
|---|---|
| 列系统 | 支持 12 列、24 列、自定义列宽 |
| span | 字段、分组、子表、按钮区均可配置 span |
| gap | 支持 rowGap、columnGap，使用设计 token |
| named area | 支持 header、main、aside、footer 等命名区域 |
| nesting | Grid 内可嵌套 Grid、Flex、Tabs、Subtable |
| minmax | 支持 min/max 宽度，避免小屏挤压失真 |
| auto flow | 支持按顺序自动排布，也支持手动指定区域 |

### 4.2 Flex 能力

| 能力 | 要求 |
|---|---|
| direction | row、column |
| alignment | justify、align、gap |
| wrap | 支持按钮组、标签组、辅助字段自动换行 |
| grow/shrink | 支持填满、收缩、固定、内容自适应 |
| basis | 支持固定基准宽度和百分比基准宽度 |
| order | 支持局部排序，不破坏 schema 节点关系 |

### 4.3 尺寸约束

| 类型 | 说明 |
|---|---|
| fixed | 固定宽高，适合工具栏、侧栏、操作栏 |
| fill | 填充剩余空间，适合主内容区、表格区 |
| hug | 根据内容自适应，适合按钮、标签、状态 |
| min/max | 所有容器和复杂组件必须支持最小/最大约束 |
| aspect | 图片、二维码、签章区可配置比例 |

### 4.4 滚动能力

| 场景 | 要求 |
|---|---|
| 页面滚动 | 主工作区可纵向滚动 |
| 局部滚动 | Subtable、审批记录、附件列表可独立滚动 |
| 横向滚动 | 分录表格列超宽时横向滚动 |
| 滚动同步 | 设计态 overlay 坐标必须跟随滚动同步 |
| 滚动阴影 | Sticky 区域出现滚动阴影，提示层级 |

### 4.5 Sticky 能力

| Sticky 区域 | 要求 |
|---|---|
| 顶部命令栏 | 始终固定在设计器顶部 |
| 表格表头 | Subtable 滚动时表头可固定 |
| 汇总区 | 可固定在分录底部或页面底部 |
| 底部操作栏 | 长单据滚动时操作按钮可固定 |
| 侧栏页签 | 右侧辅助区可局部固定 |

### 4.6 复杂容器

| 容器 | 能力 |
|---|---|
| Section | 标题、描述、折叠、权限态、校验态 |
| FieldGroup | 标签对齐、字段密度、跨列提示 |
| Tabs | 页签内嵌 Grid/Subtable/Panel |
| SplitPane | 左右或上下分栏，支持固定/弹性宽度 |
| Collapse | 多区块折叠，支持默认展开和权限隐藏 |
| Subtable | 表格列配置、固定列、横向滚动、行级操作 |
| ActionBar | 主次按钮、分组、更多菜单、状态机绑定 |
| SummaryBar | 汇总字段、公式展示、权限脱敏 |
| Drawer | 发布影响、规则冲突、权限冲突、AI Patch |

### 4.7 绝对定位边界

绝对定位只允许用于编辑辅助或特殊叠层：

| 可用场景 | 说明 |
|---|---|
| 选中框 | DesignerOverlay 根据真实 DOM 坐标绘制 |
| 拖放提示 | drop indicator、吸附线、尺寸标注 |
| Tooltip/Dropdown | 运行态浮层 |
| 打印套打 | 套打模板可单独使用坐标系 |

业务页面主体不得依赖绝对定位堆叠字段。复杂页面必须优先使用 Grid、Flex、容器和运行态组件。

## 5. LayoutEngine 职责

LayoutEngine 是 T-207 的核心引擎，不能退化成普通拖拽逻辑。

| 职责 | 说明 |
|---|---|
| DropTarget 计算 | 根据当前节点类型、容器能力、鼠标坐标计算合法投放点 |
| 吸附线 | 提供列线、行线、边距线、同级对齐线 |
| 尺寸标注 | 拖拽和 resize 时显示 span、px、百分比、gap |
| 约束校验 | 校验 span 超限、容器不兼容、局部滚动冲突、Sticky 冲突 |
| schema-to-CSS | 将 PageSchema layout 映射为稳定 CSS Grid/Flex 样式 |
| DOM-to-overlay | 从运行态 DOM 读取真实坐标，绘制编辑 overlay |
| 响应式断点 | PC 默认 1440，支持 1280/1600/1920 设计宽度检查 |
| 设计运行一致性 | 生成设计态、预览态、运行态布局差异报告 |

## 6. Schema 约定

### 6.1 LayoutNode

```json
{
  "nodeId": "section_basic",
  "type": "container",
  "component": "Section",
  "layout": {
    "display": "grid",
    "columns": 24,
    "rowGap": 12,
    "columnGap": 16,
    "padding": "16 20",
    "minWidth": 720,
    "overflow": "visible"
  },
  "children": []
}
```

### 6.2 FieldNode

```json
{
  "nodeId": "field_customer",
  "type": "field",
  "component": "ReferenceSelect",
  "fieldRef": "customerId",
  "layout": {
    "gridColumn": "span 8",
    "labelWidth": 96,
    "labelPosition": "left",
    "minWidth": 240,
    "overflow": "hidden"
  }
}
```

### 6.3 PageLayout

```json
{
  "layout": {
    "display": "grid",
    "templateAreas": [
      "toolbar toolbar",
      "status status",
      "main aside",
      "footer footer"
    ],
    "templateColumns": "minmax(0, 1fr) 320px",
    "templateRows": "52px 40px minmax(0, 1fr) 48px",
    "height": "100vh"
  }
}
```

## 7. 设计器交互要求

| 交互 | 要求 |
|---|---|
| 拖入字段 | 自动识别 BO 字段类型，生成默认控件和合法 span |
| 拖入容器 | 显示容器可接受的子类型，不合法时直接阻断 |
| resize | 支持按列拖动、按像素拖动、最小宽高约束 |
| 嵌套选择 | 支持从字段选中到父容器、兄弟节点、页面区块 |
| 大纲同步 | 画布选中、右侧属性、大纲树必须同步 |
| 局部滚动 | overlay 坐标必须与滚动容器同步，不得漂移 |
| Sticky 配置 | 右侧属性显示 sticky top/bottom、z-index、所属滚动容器 |
| 布局问题定位 | 校验报告点击后定位到具体节点和父容器 |

## 8. Figma 原型验收标准

后续 Figma 重画必须新增 `09 Web Layout / PC Web 布局能力` 画板，并满足：

| 编号 | 标准 |
|---|---|
| WEB-LAYOUT-001 | 原型第一眼是企业 PC Web 设计器，不是宣传页 |
| WEB-LAYOUT-002 | 至少展示一个完整复杂单据页面：工具栏、状态栏、FormGrid、Subtable、SummaryBar、右侧面板、底部操作栏 |
| WEB-LAYOUT-003 | 明确展示 Grid、Flex、SplitPane、Tabs、Collapse、Scroll、Sticky 的配置入口 |
| WEB-LAYOUT-004 | 控件库是列表/树，不能再做彩色卡片墙 |
| WEB-LAYOUT-005 | 中央画布必须像真实运行态 Web 页面，不是抽象示意图 |
| WEB-LAYOUT-006 | 文本不得换行错乱、截断不可读、相互重叠、底部裁切 |
| WEB-LAYOUT-007 | 右侧属性面板必须展示布局属性：display、columns、span、gap、size、overflow、sticky |
| WEB-LAYOUT-008 | 分录表格必须展示横向滚动、固定列、表头、行级操作 |
| WEB-LAYOUT-009 | 视觉必须克制，灰白企业工具风格，语义色小面积使用 |
| WEB-LAYOUT-010 | 原型必须表现“强 Web 布局能力是关键”，而不是只表现控件数量 |

## 9. 不通过判定

出现以下任一情况，原型或方案不得交付：

| 不通过项 | 原因 |
|---|---|
| 只能设计简单两列表单 | 没有达到 PC Web 页面设计器定位 |
| 页面主体依赖绝对定位 | 运行态难维护，不能响应式适配 |
| 无局部滚动和 Sticky | 复杂单据页面无法成立 |
| 无 Subtable 复杂表格能力 | 无法支撑单据分录 |
| 无 LayoutEngine 约束校验 | 发布后容易错位、遮挡、截断 |
| Figma 文本换行、重叠、裁切 | 原型质量不达标 |
| 视觉花哨但操作入口不真实 | 不符合企业设计器规范 |
