## 1. 目标

实现元模型领域服务：对象、字段、关系、状态、动作、规则的草稿态 CRUD、校验、引用索引维护。

T-003 不执行动态 DDL，不加载运行态 MetaGraph，不实现完整权限判定链。

> 决策门禁：REQ-070~REQ-078 的业务不变量是强制项；来自 proposed ADR 的 `ObjectExtensionValidator` 等类名、字段结构和合并/授权算法是候选。只有通过 `README.md` 的 ADR 状态门禁后才进入生产接口和阻断性测试，否则仅使用契约中性校验保护来源、不可覆盖边界和 runtime-disabled 状态。

## 2. 服务边界

| 服务 | 职责 |
|---|---|
| `MetaObjectService` | 对象聚合根创建、更新、查询、删除 |
| `FieldModelService` | 字段批量保存、字段配置校验 |
| `RelationModelService` | link/table/multilink 与 RelationDef 同步 |
| `StateModelService` | State/Transition 草稿维护 |
| `ActionModelService` | ActionDef 草稿维护 |
| `RuleModelService` | RuleDef 草稿维护 |
| `RoleModelService` | M0 最小 Role/PermissionDef 草稿 CRUD，保证 Version.snapshot 可包含 roles |
| `MetaValidationService` | 单对象校验与应用全图校验 |
| `MetaRefService` | `lc_meta_ref` 重建与影响分析 |
| `JsonDefinitionCodec` | Entity JSON 与 Def DTO 互转、升级 |
| `CommercialMetadataGuard` | REQ-070~REQ-078 补充 DTO 的 M0 承载校验和误执行阻断 |

## 3. API 草案

M0 设计态 API 仅供后续前端/测试使用，可先实现最小 Controller：

```text
POST /api/designer/object/add
POST /api/designer/object/update
POST /api/designer/object/get
POST /api/designer/object/list
POST /api/designer/object/del
POST /api/designer/object/validate
POST /api/designer/app/validate
```

请求 DTO 不直接暴露 Entity。

## 4. 核心 Command

```java
public record AddObjectCommand(
    Long tenantId,
    Long appId,
    String code,
    String name,
    ObjectTypeEnum objectType,
    List<FieldDef> fields
) {}

public record UpdateObjectCommand(
    Long tenantId,
    Long appId,
    String objectCode,
    Long revision,
    List<FieldDef> fields,
    List<RelationDef> relations,
    List<StateDef> states,
    List<ActionDef> actions,
    List<RuleDef> rules
) {}
```

规则：

- `tenantId` 实现阶段可来自 TenantContext，DTO 不一定显式传，但 service 内部必须有。
- update 必须带 revision。
- code 创建后不可修改。

## 5. 保存流程

```text
1. 捕获 TenantContext
2. 读取现有 MetaObjectEntity
3. 校验 revision
4. 反序列化 JSON DTO 并升级到当前版本
5. 执行单对象校验
6. 从字段生成/同步 RelationDef
7. 在同一事务内删除该对象旧 `lc_meta_ref` source 记录并插入新引用集合
8. CAS 保存 lc_meta_object 整行 JSON
9. 返回新 revision 和校验摘要

事务边界：步骤 2~8 必须在同一数据库事务内完成。任何一步失败时，`lc_meta_object` 与 `lc_meta_ref` 必须同时回滚，禁止出现对象 JSON 已更新但引用索引仍为旧版本的状态。
```

## 6. 校验器链

接口：

```java
public interface MetaValidator<T> {
  ValidationTarget target();
  List<ValidationError> validate(T model, ValidationContext context);
}
```

校验器：

| 校验器 | 规则 |
|---|---|
| `ObjectCodeValidator` | code 格式、应用内唯一、创建后禁改 |
| `FieldCodeValidator` | 字段 code 格式、对象内唯一、禁关键字、禁改 |
| `FieldTypeOptionsValidator` | 通过 `FieldTypeHandler.validateOptions` 校验 22 种 field_type options 必填项和能力约束 |
| `RelationValidator` | link/table/multilink 目标存在、关系类型合法 |
| `FetchFromValidator` | `link字段.目标字段` 路径存在且目标字段可读 |
| `ExpressionSyntaxValidator` | depends_on、validate_rule、condition 语法预检接口 |
| `StateMachineValidator` | initial 唯一、transition from/to 存在、final 不可编辑 |
| `ActionValidator` | action_code 唯一、transition action 与状态机一致 |
| `RuleValidator` | trigger/rule_type/config 匹配 |
| `PageRefValidator` | 页面 schema 引用字段/动作存在 |
| `PermissionRefValidator` | 权限引用对象、字段、动作存在 |
| `M0CapabilityValidator` | M0 不支持执行的能力必须在发布前阻断，例如 multilink DDL |
| `ObjectExtensionValidator` | 标准对象、行业模板和客户扩展的来源层、基准对象、禁止覆盖清单校验 |
| `DocumentConversionValidator` | Conversion/WriteBack/LinkTrace 的源目标对象、字段映射和一等元数据结构校验 |
| `FlexOrgCodeRuleValidator` | FlexField/OrgRelation/CodeRule 的维度、组织职能、编码范围结构校验 |
| `ReportPrintMenuI18nValidator` | Report/PrintTemplate/Menu/I18nResource 不得落入自由 PageSchema props |
| `PackageLicenseValidator` | PackageManifest/LicensePolicy 的离线授权边界、依赖和降级策略结构校验 |

T-003 必须实现 accepted M0 契约的结构校验。表中消费 proposed ADR 的商用补充校验器只有通过 ADR 状态门禁后才实现；未通过时不得为满足清单创建候选类。表达式语法预检可先定义接口，M1 实现。

字段类型校验不得在 `FieldTypeOptionsValidator` 内散落 `switch(field_type)`。校验器只按 `field_type` 找到对应 `FieldTypeHandler`，再调用处理器声明的 options 校验、能力声明和 M0 发布能力。

## 7. 错误模型

```java
public record ValidationError(
    String path,
    String code,
    String message,
    ErrorLevel level
) {}
```

path 示例：

```text
fields[3].code
fields[2].options.targetObject
states[1].transitions[0].toState
rules[4].condition
```

错误码：

| 错误 | code |
|---|---|
| code 重复 | LC-META-1001 |
| code 格式非法 | LC-META-1002 |
| 引用不存在 | LC-META-1101 |
| 字段类型配置非法 | LC-META-1201 |
| 状态机非法 | LC-META-1301 |
| revision 冲突 | LC-META-4090 |
| 标准对象禁止覆盖 | LC-META-2001 |
| 多层继承禁止 | LC-META-2002 |
| 单据转换映射非法 | LC-META-2101 |
| 反写策略缺失 | LC-META-2102 |
| 弹性域无 schema | LC-META-2201 |
| 组织关系类型非法 | LC-META-2202 |
| 报表/菜单误入 PageSchema | LC-META-2301 |
| License 在线强绑定 | LC-META-2401 |

## 8. Relation 同步规则

字段到关系：

| field_type | RelationDef |
|---|---|
| link | many_to_one |
| table | one_to_many，目标对象为子表对象 |
| multilink | many_to_many，需 through_object |

规则：

- Field 是表单视角，Relation 是模型视角，两者必须一致。
- 保存字段时自动生成或更新 RelationDef。
- 删除 link/table/multilink 字段时，不物理删除历史 RelationDef，草稿态标记 disabled，发布前给影响报告。

## 9. lc_meta_ref 规则

每次保存对象后重建该对象作为 source 的引用。

引用类型：

```text
FIELD_LINK_OBJECT
FIELD_FETCH_FROM
STATE_TRANSITION_ACTION
RULE_FIELD
RULE_ACTION
PAGE_FIELD
PAGE_ACTION
PERMISSION_FIELD
PLUGIN_EXTENSION
OBJECT_EXTENSION_BASE
DOCUMENT_CONVERSION_OBJECT
WRITEBACK_OBJECT
LINK_TRACE_OBJECT
FLEX_FIELD_OBJECT
ORG_RELATION_OBJECT
CODE_RULE_OBJECT
REPORT_DATASET
PRINT_TEMPLATE_OBJECT
MENU_TARGET
I18N_RESOURCE
PACKAGE_DEPENDENCY
```

规则：

- `lc_meta_ref` 可重建，损坏时可按全部元数据重算。
- 删除对象/字段/动作前必须查 target 引用，存在引用则阻断或生成影响清单。
- 对象保存时重建 source 引用必须与对象 JSON 更新同事务提交。
- 并发保存同一对象时，只有 revision CAS 成功的一方可以提交对应 ref；失败方不得删除成功方 ref。
- 商用补充 DTO 必须提取引用进入 `lc_meta_ref`，至少覆盖 base object、转换源/目标对象、反写源/目标对象、引用追踪源/目标对象、弹性域适用对象、组织关系对象、编码规则对象、报表数据集、打印对象、菜单目标和 i18n resource key。

伪代码：

```text
begin transaction
  object = select ... for update by tenant/app/code
  assert object.revision == command.revision
  newJson = buildObjectJson(command)
  newRefs = refExtractor.extract(newJson)
  delete from lc_meta_ref where tenant_id=? and app_id=? and source_type='OBJECT' and source_code=?
  insert newRefs
  update lc_meta_object set fields=?, relations=?, ..., revision=revision+1 where id=? and revision=?
commit
```

## 10. 并发策略

M0 采用整对象乐观锁：

```sql
update lc_meta_object
set fields = ?, relations = ?, revision = revision + 1
where id = ? and revision = ?;
```

规则：

- 两人同时编辑同一对象，后提交者收到 revision 冲突。
- 不做字段级合并，避免 lost update。
- 前端后续负责冲突展示。

## 11. 单对象校验 vs 全图校验

| 阶段 | 校验强度 | 允许保存 |
|---|---|---|
| 草稿保存 | 单对象结构正确，引用可暂时缺失但必须 warning | 可保存 |
| 发布前全图校验 | 所有引用必须闭合，危险变更和 M0 不支持能力必须报告 | 不通过则阻断 |

T-003 提供两个入口：

```java
ValidationReport validateObject(ObjectCode code);
ValidationReport validateApp(AppId appId);
```

M0 发布阻断项：

| 能力 | 草稿保存 | 发布前全图校验 |
|---|---|---|
| `multilink` 字段 | 允许保存，便于设计态表达关系 | 阻断，提示 M0 不执行 through 表 DDL |
| 表达式实际求值 | 允许保存表达式文本 | 只做语法预检接口，不做语义执行 |
| 插件扩展点 | 允许保留配置结构 | 阻断启用，插件系统未实现 |
| ObjectExtension | 允许保存受控扩展层 DTO | 阻断继承展开、标准对象升级合并和客户扩展运行时覆盖 |
| Conversion/WriteBack/LinkTrace | 允许保存 DTO | 阻断转换执行、反写副作用和运行时上查/下查 |
| FlexField/OrgRelation/CodeRule | 允许保存 DTO | 阻断弹性域查询、组织关系运行时判权和复杂编码取号 |
| Report/PrintTemplate/Menu/I18nResource | 允许保存 DTO | 阻断报表运行、打印生成、门户菜单运行时和多语言发布强校验 |
| PackageManifest/LicensePolicy | 允许保存 DTO | 阻断包安装执行、市场分发、在线 License 校验和授权降级执行 |

## 11.1 商用补充能力校验边界

REQ-070~REQ-078 的候选详细设计落点如下。表中精确类名和字段仅在对应 ADR/决策记录 accepted 后生效；无论是否终审，右侧“运行时不得误启用”和 PRD 安全不变量始终生效：

| 需求 | T-003 校验 | 发布前阻断 |
|---|---|---|
| REQ-070/071 标准对象扩展与继承 | `ObjectExtensionValidator` 校验 `sourceKind/baseObjectCode/extensionPolicy`，禁止 customer 直接覆盖 system/vendor 字段 | 阻断无限继承、继承展开、标准包升级合并 |
| REQ-072/073 单据转换、反写、引用追踪 | `DocumentConversionValidator` 校验源目标对象和字段映射存在，提取 `lc_meta_ref` | 阻断转换执行、反写副作用、上查/下查运行时 |
| REQ-074/075 弹性域与多组织 | `FlexOrgCodeRuleValidator` 校验维度、组织职能和编码范围结构，不允许无 schema JSON | 阻断弹性域查询、组织关系判权、复杂编码运行时 |
| REQ-076/077 报表打印菜单移动端 i18n | `ReportPrintMenuI18nValidator` 校验独立元数据位置，不允许写入 PageSchema 自由 props | 阻断报表运行、打印生成、菜单运行时、多语言发布强校验 |
| REQ-078 应用包/License | `PackageLicenseValidator` 校验 manifest、依赖、离线授权和降级策略结构 | 阻断包安装升级执行、市场分发、License 运行时 |

## 12. 测试

测试类：

```text
MetaObjectServiceTest
FieldModelServiceTest
RelationModelServiceTest
MetaValidationServiceTest
MetaRefServiceTest
MetaObjectConcurrencyTest
CommercialMetadataGuardTest
ObjectExtensionValidatorTest
DocumentConversionValidatorTest
FlexOrgCodeRuleValidatorTest
ReportPrintMenuI18nValidatorTest
PackageLicenseValidatorTest
```

必测：

- 重复对象 code 拒绝。
- 重复字段 code 拒绝。
- link 目标不存在报字段路径。
- 字段 options 校验通过 `FieldTypeHandler.validateOptions` 执行，非法 decimal 精度、缺失 link target、multiselect 高性能筛选限制均能返回字段路径。
- fetch_from 路径不存在报字段路径。
- 状态机 initial 多个拒绝。
- revision 冲突拒绝。
- 删除被页面引用字段时能查到影响。
- 并发保存同一对象时，失败方不会删除成功方 `lc_meta_ref`。
- 对象保存中途异常时，`lc_meta_object` 与 `lc_meta_ref` 同时回滚。
- 通过 OBJECT-EXT ADR 状态门禁后，标准对象扩展按 accepted 契约保存；无论采用何种结构，客户扩展都不能覆盖系统、供应商或 final 字段。
- 多层继承或继承展开配置在发布前被阻断。
- Conversion/WriteBack/LinkTrace 能保存并提取引用，但任何 runtime enabled 标记被阻断。
- FlexField/OrgRelation/CodeRule 能保存并提取引用，但禁止作为普通 JSON 字段或普通 autonumber 运行。
- Report/PrintTemplate/Menu/I18nResource 能保存，且菜单/报表/i18n 不允许落入 PageSchema 自由 props。
- 通过 APP-PACKAGE ADR 状态门禁后，包与 License 候选 DTO 可保存；无论采用何种结构，在线 License 强绑定和未治理的直接包安装执行都被阻断。

## 13. 验收标准

- [ ] 对象/字段/关系可保存和读取。
- [ ] RelationDef 与 link/table/multilink 字段一致。
- [ ] `lc_meta_ref` 可重建。
- [ ] 对象保存与 `lc_meta_ref` 重建同事务，且并发保存不会丢引用。
- [ ] 校验错误有 path。
- [ ] 字段类型 options 校验只通过 `FieldTypeHandler`，无散落 field_type 分支。
- [ ] 整对象 revision 乐观锁生效。
- [ ] 不执行任何 DDL。
- [ ] REQ-070~REQ-078 的产品不变量和 M0 误执行阻断测试齐全；精确 DTO 校验与引用提取只验收已通过 ADR 状态门禁的条目。
