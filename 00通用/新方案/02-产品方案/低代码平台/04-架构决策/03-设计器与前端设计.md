## 1. 双层构建器（ADR-LOWCODE-UI-001 落地）

```text
┌── 业务建模构建器（Model Builder）──────────────┐
│ 对象管理：对象列表 → 字段表格编辑器（类 Frappe    │
│ DocType 编辑：一行一字段，行内改类型/必填/选项）    │
│ 关系图：对象间 ER 可视化（只读，M2）              │
│ 状态机设计：状态节点 + 流转连线（图形化）          │
│ 规则配置：触发器 + 条件 + 动作的表单式配置         │
│ 权限矩阵：Role × Object × 操作 的勾选矩阵 +       │
│           数据范围/字段权限抽屉                   │
├── 页面编排构建器（Page Builder）────────────────┤
│ 默认页面：建对象即自动生成 列表/表单/详情          │
│ 布局覆盖：拖拽调整字段分组/分栏/顺序、隐藏字段、    │
│           配置列表列/筛选/按钮                    │
│ 自定义页面：区块级拼装（M2，参照 NocoBase Block）  │
└────────────────────────────────────────────────┘
```

**边界铁律**（ADR-LOWCODE-UI-001 验证条件）：页面层只能"覆盖表达"，不能"绕过语义"——页面上隐藏字段 ≠ 免校验；页面不配按钮 ≠ 动作可通过 API 越权调用。所有语义强制在运行时服务端（02 文档管线）。

## 2. 页面 Schema（View.schema 结构）

自研精简 Schema（不引入 amis/formily 全量协议，避免被外部协议锁死；结构参照 NocoBase UI Schema 的"组件树+绑定"思想，UI-NOCOBASE-SRC 卡）：

```json
{
  "viewType": "form",
  "bindObject": "order",
  "layout": [
    {"type": "group", "title": "基本信息", "columns": 2, "items": [
      {"type": "field", "field": "order_no"},
      {"type": "field", "field": "customer_id", "props": {"showSearch": true}},
      {"type": "field", "field": "amount", "hidden": false, "readonlyOverride": null}
    ]},
    {"type": "subtable", "field": "order_items", "columns": ["product_id","qty","price"]}
  ],
  "actions": ["save", "submit"]
}
```

规则：

1. Schema 中只允许引用元数据中存在的 field/action（保存时服务端校验，防悬空引用）。
2. `hidden/readonlyOverride` 只能**收紧**不能放开：字段无读权限时即使 Schema 声明显示也不返回数据（/meta 接口已裁剪）。
3. 列表视图 Schema 额外含 `columns / filters / sorts / batchActions / rowActions`。
4. Schema 版本随 App Version 快照发布，运行时读快照。
5. Schema 不得绑定固定像素宽度作为唯一布局依据；桌面 Web 是 M1~M3 主目标，但组件和布局必须保留 M4 响应式/移动端改造空间。
6. 报表、打印、菜单、门户、移动端和 i18n 不得塞入 PageSchema 自由 props，必须引用独立元数据。

## 3. 运行时渲染器（packages/renderer）

数据流（唯一元数据入口是 `/api/data/{app}/{object}/meta`）：

```text
路由 /app/{app}/page/{page}
  → 拉取页面 Schema + 对象 meta（含当前用户字段权限/动作权限裁剪结果）
  → FieldRegistry 按 field_type 映射 AntD 组件（22 种字段类型 → 组件表）
  → 表单：值联动（depends_on 表达式前端求值镜像）+ fetch_from 回填 + 提交前本地校验
  → 列表：ProTable 风格（服务端分页），筛选项由 in_filter 字段自动生成
  → 动作按钮：按 meta 返回的 allowed actions 渲染，transition 类按钮附确认框
```

表达式前端镜像：depends_on/visible_condition 需要前端即时求值——`packages/shared` 内实现与后端语义一致的表达式子集求值器（仅逻辑/比较/字段引用），**校验类表达式不在前端复制**（防止双端漂移，校验以服务端为准，前端只做类型级即时校验）。

| 界面 | 设计要点 | 竞品参照 |
|---|---|---|
| 字段编辑器 | 表格式批量编辑（一屏看全部字段），行内下拉改类型，抽屉改高级配置 | Frappe DocType 编辑器（UI-FRAPPE-* 卡） |
| 状态机设计器 | 画布节点+连线，连线属性面板配置角色/条件 | Frappe Workflow / 金蝶流程设计 |

## 4.2 设计器能力边界

1. 导入向导、模板库安装、ConvertRule、rollup、API Token、IdentityProvider、Webhook 等高风险能力，设计器只负责配置采集和预检展示；执行必须由后端统一管线完成。
2. rollup、复合唯一、选项集、保存视图、字段脱敏都必须在模型构建器中有显式配置入口，不能藏在 JSON 高级配置里作为首选路径。
3. 字段脱敏在权限矩阵中作为独立档位展示：NONE、MASKED、READ、WRITE。
4. 事件中心必须有订阅列表、事件 payload 预览和订阅执行日志入口；Webhook 和插件订阅默认在 M3 前不可启用。
5. API Token 管理界面必须只显示一次明文 token，后续只显示脱敏值、最后使用时间、scope 和撤销按钮。

## 5. 前端工程

- monorepo：pnpm workspace；`builder`/`renderer` 独立构建产物（运行态私有化部署可只部署 renderer）。
- 技术：React 18 + TS + AntD 5 + zustand（状态）+ @dnd-kit（拖拽）；图形化设计器（状态机/关系图）用 reactflow。
- API 层：统一 POST 客户端 + 租户/应用上下文注入；类型从后端元模型 OpenAPI 生成。
- 多语言：字段 name 等元数据多语言 JSON，界面文案 i18n 资源（首版中文，结构预留）。

## 6. 待实测修订项

知识库 UI 证据目前为"官方 demo/文档截图"级（未实测）。以下设计点在竞品本地实测后复核修订：

1. NocoBase 区块级页面拼装的实际交互成本 → 影响 M2 自定义页面的粒度设计
2. Frappe 字段表格编辑器在 100+ 字段对象上的可用性 → 影响字段编辑器分组/搜索设计
3. 权限矩阵在角色多、对象多时的可维护性 → 可能改为"角色详情页"模式
