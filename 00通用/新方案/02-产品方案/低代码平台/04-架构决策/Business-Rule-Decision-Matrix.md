# Business-Rule-Decision-Matrix

> 阶段：5A 统一设计决策工作流  
> 质量流水线：入口检查已读取 PRD、运行时引擎、状态机/规则详细设计、Outbox ADR、M1/M2 测试规格；本文件约束业务规则、状态、自动化与副作用。

| 设计点 | 候选A(知识库/竞品证据) | 候选B(行业标准/最佳实践) | 候选C(自主创新方案) | 采纳结论 | 各落选候选拒绝理由 | 验证方式 | 承接ADR |
|---|---|---|---|---|---|---|---|
| 表达式与规则入口 | 竞品支持公式、显隐、校验、默认值、自动化条件 | 解释器模式 + 沙箱 + 单一门面，避免表达式语义漂移 | Expression Compatibility Ledger：每次新增表达式函数都登记适用场景、风险等级、前后端支持矩阵和回放用例 | M1 前必须补表达式 ADR；Ledger 作为候选增强 | 直接暴露通用脚本风险过高；多处各自解析会产生双端漂移 | 沙箱阻断反射/IO/网络；前端仅实现可见性子集；表达式版本回放 | 待补 ADR-LOWCODE-EXPR-001、T-101、T-201 |
| 状态机与动作执行 | Frappe Workflow / 金蝶流程模型均用状态和动作表达业务流转 | 状态机模式：动作合法性、角色、条件、后置规则必须串行可审计 | Transition Explain Report：发布和模拟时生成“为何可见/不可见/可执行/被拒绝”的状态、权限、条件解释 | 采纳显式状态机；Explain Report 进入 T-204/T-104 | 页面按钮控制不能防 API 越权；自由流程引擎首版复杂度过高 | 状态机非法配置阻断；模拟销售只能看本人订单；action API 与 UI 结果一致 | 02-运行时引擎设计.md、T-104、T-204 |
| 规则链与循环防护 | 竞品常见 before/after 规则、自动化、rollup | 规则引擎必须有拓扑、最大轮数、幂等和事务边界 | Rule Impact Graph：发布前绘制字段依赖、对象依赖和级联深度，超阈值阻断并给出裁剪建议 | M1/M2 采纳最大轮数和循环检测；Impact Graph 作为 M2 增强 | 无限制自动化会造成规则风暴；首版完整图分析可先 Spike | RuleLoopGuardTest、RuleCascadeDepthGuardTest、发布前 warning/blocking 报告 | T-104/T-105、M1/M2 测试规格 |
| 跨对象副作用 | 竞品支持通知、Webhook、外部 API、消息等副作用 | Outbox Pattern：业务事务只写本地事件，异步投递支持幂等和重试 | SideEffect Receipt：每个副作用生成收据，记录租户、来源记录、幂等键、脱敏 payload、投递状态和人工重放边界 | 倾向采纳 Outbox；Receipt 作为 Outbox 必填审计字段候选 | 同步直接调用会污染主事务；各模块自建重试导致一致性不可解释 | outbox 幂等、失败重试、死信、人工重放、敏感字段脱敏测试 | ADR-LOWCODE-OUTBOX-001、T-107、T-306 |
| 单据转换、反写与汇总 | 金蝶式单据链路、Frappe/NocoBase 的关联与 rollup 能力 | 企业应用必须保护超额控制、反写一致性、来源追踪 | Conversion Sandbox：先试跑字段映射、权限、状态、数量/金额约束，输出可执行计划；发布后执行仍走统一写入口 | M2/M3 proposed；M0 只预留和阻断误执行 | 直接跨表写会绕过权限/状态/审计；首版实作全部单据链路范围过大 | Conversion/WriteBack/LinkTrace DTO 保存但运行阻断；M3 超额控制测试 | ADR-LOWCODE-CONVERSION-001、T-307~T-310 |
