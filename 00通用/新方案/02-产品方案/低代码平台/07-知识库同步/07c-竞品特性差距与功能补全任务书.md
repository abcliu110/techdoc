## 1. P0：单据类业务系统刚需（缺失=核心场景不成立）

平台定位是"单据类业务系统"（订单、审批、主数据）。以下能力是该场景的地基，竞品全部具备。

### GAP-01 业务事件中心（事件发布/订阅）🔥用户已点名

**竞品**：Frappe hooks.py 的 doc_events（before_save/after_save/on_submit/on_cancel 应用级订阅）；金蝶事件中心（业务事件跨应用订阅解耦）；NocoBase Workflow 事件触发。
**现状**：只有 Rule 挂在固定 trigger 上，规则与对象强耦合；没有"事件"作为一等公民——无法跨对象/跨应用订阅，无法对外（webhook）推送，插件无事件挂接点。
**设计方向**：
- 元模型新增 **Event 语义**：平台内置事件集（record.created/updated/deleted、state.transitioned、version.published 等）+ 对象级自定义事件（Action 可显式发布事件）。
- **EventSubscription 订阅模型**：订阅者类型 = rule（现有规则改造为事件的一种消费者）/ webhook（M3）/ plugin handler（M3）/ workflow 触发（M3）。同步订阅（before_ 类，可否决事务）与异步订阅（after_ 类，走 07b ARCH-4 Outbox 投递）语义分开定死。
- 事件 payload 契约：含 tenantId/objectCode/recordLid/diff 摘要/metaHash/traceId，字段脱敏遵循 B6 分级。
- 这是 Rule/Workflow/Webhook/Plugin 四者的统一底座——**先定事件模型，再谈其余三者**，避免各自为政。
**落点**：01 新增"§Event 与订阅模型"；02 规则章节重构为"事件与消费者"；PRD 新增 REQ；04 M1 补事件内核卡（内置事件+规则消费者），M3 补 webhook/插件订阅卡。**里程碑：事件内核 M1，外部订阅 M3。**

### GAP-02 规则动作扩展：跨对象创建/更新记录

**竞品**：Frappe Server Script / NocoBase Workflow 节点 / 简道云类产品的"新建关联数据"——单据联动的标准解法。
**现状**：Rule.rule_type 只有 validate/assign/notify/call_api，assign 仅限当前记录。订单确认后自动生成发货通知单这类最常见的联动无法配置。
**设计方向**：rule_type 新增 `create_record` / `update_record`（目标对象 + 字段映射表达式）。硬约束：必须走 07b ARCH-4 统一写管线（触发目标对象自身的权限/规则/审计）；级联深度与轮数上限复用 F2 的循环检测；同步/异步执行位置定死（建议 after 事件异步 + outbox）。
**落点**：01 Rule 定义、02 §4、PRD REQ、04 M1/M2 任务卡。**里程碑 M2。**

### GAP-03 定时触发自动化（schedule trigger）

**竞品**：Frappe scheduler_events；各自动化平台的 cron 触发。
**现状**：规则只有数据事件触发；"每天 9 点提醒逾期订单""每月 1 号生成对账单"无法实现。07a-C6 只解决了调度引擎底座，产品特性未立项。
**设计方向**：Rule trigger 新增 `schedule`（cron/周期 + 目标对象 filters + 动作），运行在 07b ARCH-4 统一调度器上；租户时区语义、错过补跑、租户配额按 07a-C6 的答案执行。
**落点**：01/02、PRD REQ、04 M2 任务卡。**里程碑 M2。**

### GAP-04 汇总字段（rollup：子表→主表聚合）

**竞品**：NocoBase/NocoDB rollup、Frappe 通过 fetch/脚本实现——"订单总额=明细求和"是单据系统的第一需求。
**现状**：22 类型有 formula 但无 rollup 语义；总额只能靠用户手填或未来脚本。
**设计方向**：新增字段类型 `rollup`（源=子表/关联对象 + 聚合函数 count/sum/avg/max/min + 过滤条件），作为 07b ARCH-5 FieldTypeHandler SPI 的一个实现；求值语义与 07a-D4（formula）一并定死（建议：子表行变更时同事务重算物化——单据场景行数有限）。
**落点**：01 §2.2（字段类型 22→23）、07a-D4 条目扩展覆盖 rollup、PRD REQ、04 M2 卡。**里程碑 M2。**

### GAP-05 单据转换/下推（BOTP）

**竞品**：金蝶 BOTP（订单→发货单→发票的字段映射下推 + 上下游关联追溯）——金蝶路线的标志性能力。
**现状**：完全缺失，且 GAP-02 只解决"自动创建"，不解决"人工发起 + 映射模板 + 来源追溯"。
**设计方向**：新增 **ConvertRule 元模型对象**：源对象→目标对象、字段映射（支持表达式/子表映射）、携带上下游引用（目标记录记 source_lid，形成单据链）、下推动作挂 Action(action_type=convert)。执行走统一写管线；单据链查询支撑"上查/下查"。首版可只做 1:1 单级下推，1:N/合并下推声明为 M4。
**落点**：01 新增对象（19→20，需增补 ADR-LOWCODE-DM-001 修订记录）、02、PRD REQ、04 M3 卡。**里程碑 M3。**

### GAP-06 打印模板

**竞品**：Frappe Print Format、金蝶打印模板——发票/合同/送货单打印是单据系统交付底线。
**现状**：完全缺失。
**设计方向**：Page 新增 view_type=print 或独立 PrintTemplate：基于对象元数据的模板（字段占位 + 子表表格 + 页眉页脚），服务端渲染 PDF/HTML。安全：模板表达式走 lowcode-expression 门面（禁自由 HTML 注入，联动 G4）；打印受 AccessView 字段裁剪；打印行为进审计事件。
**落点**：01/03、PRD REQ、04 M3 卡；07 陷阱流程登记"打印模板注入"候选条目。**里程碑 M3。**

### GAP-07 复合唯一约束

**竞品**：数据库常识 + Frappe/NocoBase 均支持。"客户+年月 唯一"类需求首版就会遇到。
**现状**：只有字段级 unique_flag。
**设计方向**：BusinessObject 新增 unique_constraints: [[field_code,...]]；SchemaSync 生成复合唯一索引（与软删除 delete_token 方案、D2 口径对齐）；违约错误码定位到约束名并给人话提示。
**落点**：01 §2.1、T-004、`../../../../工程规范/通用规范/02-数据与API规范.md`、`../../../../工程规范/低代码平台规范/01-元模型与物理存储契约.md`、PRD REQ。**里程碑 M1（元模型 M0 预留字段）。**

### GAP-08 业务数据 Excel 导入导出

**竞品**：全部竞品标配（导入向导：列映射→校验预览→执行报告）。
**现状**：PRD 只有"应用包"导入导出，业务数据导入导出未立项（07a-F10 只覆盖了风险面）。
**设计方向**：作为产品特性立项：导入向导（模板下载、列映射、试跑校验报告、部分成功策略）+ 列表导出（受 AccessView 裁剪）。全部安全语义按 07a-F10 执行。
**落点**：PRD REQ、03 交互、04 M2 卡。**里程碑 M2。**
