# 低代码平台 · 平台级陷阱与高难度问题清单

> 版本：v1.1
> 日期：2026-07-05
> 地位：**06-规范补全任务书的强制输入**。细则编写 AI 必须对本清单每个问题给出"设计答案 + 对应细则条款 + 强制/检测方式"，答不出的必须列入《未决事项清单》上报，禁止跳过或用风格条款敷衍。
> 判断标准：一份规范如果只能防"代码写得丑"，防不了本清单里的事故，就是不合格的规范。
> 每个陷阱标注等级：🔴 数据损坏/越权级（出事故无法挽回） 🟠 停机/性能崩溃级 🟡 演进受阻级（半年后才爆发）
> 范围说明：本清单不只记录狭义"架构陷阱"，还包括产品语义、设计器交互、代码实现、安全合规、运维交付、AI 协作等会导致平台事故或长期腐化的陷阱。
> 方法论关系：本文是《04A-清单-复杂平台级失败模式与设计约束.md》在低代码平台领域的实例化；新增、合并、升级或降级条目时，应按《03-方法论-系统级失败模式与设计约束清单.md》和《03A-方法论-失败模式与设计约束清单建设.md》判断证据来源、适用范围和承接闭环。
> v1.1 增补：A5-A6、C6、D4-D8、F9-F11、G6-G7（formula/fetch_from 语义、MDL 锁、Redis/分布式锁、调度引擎、认证与 CSRF、业务数据批量导入导出、通知出域、预览边界、错误信息泄露等）。

陷阱类型口径：

- **架构/一致性陷阱**：元数据、物理结构、缓存、版本、发布回滚、备份恢复等系统不变量被破坏。
- **产品/业务语义陷阱**：用户以为能力成立，但平台没有定死语义边界，如回滚、组织变更、工作流在途实例。
- **设计器/前端陷阱**：配置能保存但运行不可用，或渲染器、Schema、表达式、组件系统产生安全和一致性问题。
- **实现/测试陷阱**：代码表面符合规范但绕过统一入口，测试只覆盖主路径，AI 生成过度抽象。
- **安全/合规陷阱**：租户隔离、权限、超级管理员、个人信息、附件、Connector、插件等边界失控。
- **运维/交付陷阱**：容量、噪声租户、环境漂移、SaaS/私有化分叉、可观测性和恢复演练缺失。

---

## A. 元数据与物理表的一致性（本平台最核心的难题）

### A1 🔴 MySQL DDL 不可回滚，发布事务是"假原子"

**陷阱**：MySQL 的 DDL 隐式提交、不能包进事务。发布流程"写 Version 快照 + 执行 Schema Sync DDL + 更新元数据状态"看似一个操作，实际是**非原子的三段**。中途失败（第 3 张表 DDL 报错/进程被杀）会留下"元数据说有列、物理表没有"或反之的**半应用状态**。
**细则必须回答**：
- 发布流程的状态机设计（每一步落库状态，失败可从断点恢复还是必须人工介入？）
- **对账器（Reconciler）**：元数据 vs information_schema 的差异检测的触发时机（启动时？发布前强制？）、差异的分类处置（自动补齐 / 阻断发布 / 工单）
- DDL 执行日志表的设计（执行了什么、结果、耗时）

### A2 🔴 版本回滚回滚不了 DDL

**陷阱**：Version.snapshot 回滚元数据是一行 update，但已加的物理列、已建的索引回滚不掉（按"只做加法"原则也不该删）。回滚后运行 v1 元数据 + v2 物理表。若 v2 曾把字段改语义（options 变更），v1 代码读 v2 格式数据会出错。
**细则必须回答**：
- "回滚"的准确语义定义：哪些能回滚（页面/规则/权限）、哪些不能（物理结构、已按新格式写入的数据）——必须写成给用户看的能力边界，不是内部备注
- 字段语义变更（select 选项集收窄、decimal 精度改小）的禁止/兼容规则清单
- 回滚前置检查器：检测本次回滚跨越的版本区间内是否有不可回滚变更，有则阻断并给出影响清单

### A3 🟠 动态表数量爆炸

**陷阱**：每对象一张表 × 每租户 N 应用 × SaaS 上千租户 = 数万~数十万张表。MySQL 在十万级表时：information_schema 查询变慢、buffer pool 元数据开销、备份时间失控、Schema Sync 的 diff 全表扫描超时。
**细则必须回答**：
- 容量模型与硬上限：单实例表数上限、达到上限的扩展路径（按租户分库？——这决定 ID 策略和路由层，是现在就要留的口子）
- Schema Sync 的 diff 必须走自维护的"物理结构登记表"而不是每次查 information_schema——登记表设计
- 表生命周期：应用删除/租户注销时物理表的处置流程（立删？归档期？）

### A4 🟡 JSON 聚合列的演进与并发

**陷阱1**：`_v` 版本升级器（JsonUpgrader）如果只在读取时惰性升级、不回写，系统里永远存在 N 个版本并存，某天删升级器代码就炸。
**陷阱2**：两个设计者同时编辑同一对象的不同字段，整行乐观锁使后提交者全量失败，体验差；若为了体验改成"读-改-写合并"，会静默丢失对方的修改（lost update 变种）。
**细则必须回答**：
- 惰性升级的回写策略与旧版本淘汰机制（如何证明库里已无 `_v=1`）
- 设计器并发编辑的冲突策略：整对象锁（谁在编辑）？字段级合并？冲突提示 UI？——定死一种
- Version.snapshot 里的 JSON 是否随 `_v` 升级？（快照必须能被当前代码读取——历史快照升级策略）

### A5 🟠 DDL 的 MDL 元数据锁雪崩

**陷阱**：大表 online DDL（即使是 INSTANT/INPLACE）在开始与结束时都要拿 MDL 排他锁。共享库 SaaS 下，任何一个未提交的长事务或慢查询持有该表的 MDL 读锁，DDL 就会排队等待；而排队中的 DDL 又会阻塞其后**所有**对该表的新查询——一个租户的发布操作可以让全库对该表的访问雪崩。E3 只回答了 INSTANT/INPLACE 判定，没有回答锁等待链问题。
**细则必须回答**：
- 发布 DDL 前的强制预检：检测目标表上的长事务/慢查询（information_schema.innodb_trx / processlist），存在则阻断或延迟。
- DDL 会话必须设置 lock_wait_timeout 上限（定死具体秒数），超时后的状态机处置（回到 A1 的断点恢复流程）。
- 大表判定阈值与低峰执行窗口策略；共享库下"租户 A 发布影响租户 B"是否纳入 C5 噪声租户的 DDL 配额（与 C5 关联）。
- 演练用例：模拟长事务持锁时发起发布，验证阻断/超时/恢复路径。

### A6 🟠 Redis 不可用与发布分布式锁的正确性

**陷阱1**：MetaGraph 版本号依赖 Redis。Redis 宕机/被清空后，若直接回源 DB 会产生回源风暴；若拒绝服务则全平台不可用；若各实例各自缓存不刷新，则版本收敛机制（C3）整体失效。
**陷阱2**：发布/DDL 用 Redis 分布式锁互斥，但锁有过期时间：持有者因 GC 停顿或慢 DDL 超过过期时间后，第二个发布者拿到锁并发执行——双写 DDL，直接制造 A1 的半应用状态。
**细则必须回答**：
- Redis 不可用时的降级策略定死一种：本地缓存 + 版本号轮询降级 / 只读模式 / 拒绝写入，以及恢复后的收敛流程。
- 版本号缓存更新失败（DB 已提交、Redis 未更新）时的检测与收敛机制。
- 发布锁的续期（watchdog）或 fencing token 方案定死一种；锁失效后旧持有者的 DDL 执行必须能被识别并中止或安全完成。
- 测试集：锁过期双持有者并发发布、Redis 闪断期间的读写行为。

## B. 多租户与安全（事故不可挽回类）

### B1 🔴 租户上下文在异步链路丢失

**陷阱**：TenantContext 用 ThreadLocal，在 @Async、线程池、CompletableFuture、定时任务、MQ 消费、after_save 异步规则里**静默丢失或串位**（线程复用读到上一个请求的租户）——数据串租户是 SaaS 死刑级事故，且测试环境单租户根本测不出来。
**细则必须回答**：
- 上下文传播的统一机制（TaskDecorator/TransmittableThreadLocal/显式传参——选定一种并给全场景模板：@Async/MQ/定时/规则异步执行）
- **兜底**：动态数据访问层在 SQL 组装处对 tenant_id 为空直接抛异常（fail-fast，宁可报错不可串数据）——这条的单测和 ArchUnit 检测怎么写
- 多租户混测的测试规范：核心链路必须有"双租户并发+断言隔离"用例

### B2 🔴 数据范围条件注入的 AST 正确性

**陷阱**：data_scope 条件与用户 filters 合并时，若在 SQL 字符串层拼接 `AND`/`OR`，用户 filters 里一个顶层 OR 就能让权限条件被短路（`WHERE scope AND a OR b` ≠ `WHERE scope AND (a OR b)`）——经典越权漏洞，代码评审肉眼很难发现。
**细则必须回答**：
- 强制在 filters AST 层合并（scope 作为根 AND 的强制子节点），禁止任何字符串层条件拼接——违规的检测方式
- 越权测试集的构造方法：专门构造"试图短路 scope"的恶意 filters 用例族

### B3 🔴 /meta 裁剪与实际执行的一致性

**陷阱**：字段权限在 /meta 接口裁剪了一份、在 list/get 返回时裁剪一份、在 update 提交时校验一份——三处逻辑分开写必然漂移，漂移=泄露或越权写。
**细则必须回答**：
- 字段可见性/可写性判定必须收敛为单一函数（判定器唯一入口），三处只许调用它——ArchUnit 或代码结构上如何强制
- link 展开（`customer_id.level`）跨对象时的权限传递规则：目标对象无 read 权限时展开字段返回什么？（定死：不返回 or 报错）

### B4 🟠 表达式沙箱的间接逃逸

**陷阱**：Aviator 禁了反射，但白名单函数本身可能成为跳板（如提供 `getRecordById` 类函数→绕过数据范围权限读任意记录）；表达式在 data_scope 里可引用 `$user`，若 `$user` 对象暴露过多（含 token/密码哈希字段）也是泄露面。
**细则必须回答**：
- 白名单函数的准入评审规则：函数不得进行 IO/查库；需要数据的场景由引擎预取后注入上下文
- `$user/$record/$env` 上下文对象的字段白名单（显式 DTO，不是直接扔 Entity 进去）

### B5 🔴 超级管理员与运维越权入口

**陷阱**：平台管理员、租户管理员、运维排障、代客操作很容易变成"万能后门"。一旦 super 角色可以绕过租户隔离、字段权限、数据范围或审计，任何业务权限设计都会失效；若审计可被同一角色删除或修改，事后也无法追责。
**细则必须回答**：
- break-glass/代客操作的触发条件、审批方式、有效期、授权范围和强制二次确认。
- super 角色能绕过哪些层、不能绕过哪些层（tenant_id 隔离、字段脱敏、密钥明文必须默认不可绕过）。
- 越权运维操作必须全量审计且审计不可被同一权限主体修改/删除；审计字段至少包含操作者、被代理用户、理由、工单号、traceId、前后数据摘要。
- 测试集：普通管理员跨租户失败、break-glass 到期失效、审计不可篡改、DataSource 密钥不可见。

### B6 🔴 个人信息、敏感数据与审计保留边界

**陷阱**：低代码平台会承载客户、联系人、用户、组织、附件、审计日志和导出包。若没有个人信息分级、脱敏、保留期、删除/匿名化规则，日志、审计、附件、导出包会长期保存超范围敏感数据，形成合规和泄露风险。
**细则必须回答**：
- 数据分级清单：手机号、身份证、邮箱、地址、token、密钥、附件、审计 diff、导出包分别属于哪个敏感等级。
- 最小化与脱敏规则：API 返回、日志、审计、导出、前端错误、截图/预览分别如何处理敏感字段。
- 保留期与删除策略：审计日志、规则日志、附件、导出包、临时文件、备份中的个人信息保留多久，如何过期清理。
- 用户删除/租户注销/应用删除时，个人信息是删除、匿名化、归档还是因审计要求保留，必须写成产品边界。

### B7 🔴 组织变更后的历史数据范围语义

**陷阱**：data_scope=self/dept/dept_tree 依赖用户与组织关系。员工调岗、部门合并、组织树调整后，历史单据到底按"当前部门"可见，还是按"创建时部门快照"可见？如果不定死，同一条历史数据会随组织调整改变可见范围，造成越权或业务不可追溯。
**细则必须回答**：
- 每类数据范围的时间语义：本人、本部门、本部门及以下、角色范围、自定义表达式分别按当前组织还是历史快照。
- 记录创建/流转/转交时是否保存 owner_user_lid、owner_dept_lid、owner_org_path 快照；组织迁移时是否重算。
- 历史数据范围变更的用户可见说明、审计记录和批量修复流程。
- 测试集：员工调岗、部门删除、部门合并、组织树移动后历史记录可见性。

## C. 运行时性能（上线后才爆发类）

### C1 🟠 动态查询的 N+1 与展开爆炸

**陷阱**：列表 100 行 × 每行 3 个 link 字段展开 + fetch_from 链 = 数百次点查；子表(table 字段)在列表场景被误加载。元数据驱动的查询引擎没有 ORM 的 lazy/eager 语义，全靠自己设计。
**细则必须回答**：
- link 展开的批量化规则（收集 id 集合→ IN 一次查→内存装配），引擎层强制而不是靠调用者自觉
- 展开深度上限（已定一层）之外：单请求展开的**对象种类上限、总行数上限**
- 子表加载策略：列表禁自动加载子表，详情按需——写进 API 行为定义

### C2 🟠 审计/规则日志的写放大

**陷阱**：一次业务写 = 业务行 + 审计(字段级 diff) + 规则日志 + 修订快照 ≈ 4 倍写放大，全同步则 TPS 崩，全异步则宕机丢审计（合规问题）。
**细则必须回答**：
- 分级策略：哪些必须同事务（审计主记录？）、哪些可异步（规则日志）、异步通道的可靠性等级（本地表 outbox / 允许丢失）
- 大字段 diff 的截断规则（富文本改一次存两份全文？）
- 日志表的滚动分区与归档细则（保留期、归档介质、查询降级）

### C3 🟠 MetaGraph 多实例一致性与发布时的请求视图

**陷阱1**：多实例部署，发布后 A 实例已加载 v2、B 实例还在 v1，负载均衡下同一用户前后两个请求行为不一致，甚至前端拿 v2 的 /meta 提交到 v1 的实例。
**陷阱2**：单请求处理中途缓存刷新，前半段用 v1 校验、后半段用 v2 持久化。
**细则必须回答**：
- 请求级版本固定：请求入口取定 MetaGraph 引用，整个请求生命周期不换（代码模式怎么写、怎么防止中途重新 get）
- 实例间收敛机制（版本号轮询/广播）与收敛窗口的可接受上限；窗口期内 /meta 响应携带元数据版本号、写接口校验版本号过旧则要求前端刷新——是否采用，定死
- 冷加载 2s 期间的请求处理策略（阻塞等待/旧版本服务）

### C4 🟡 autonumber 与深分页

- autonumber 并发取号：DB 行锁计数器（瓶颈）vs 号段缓存（宕机跳号）——业务单号允许跳号吗？定死并写清代价
- 动态列表的深分页（offset 10 万）：是否强制游标式分页/限制最大页码——定死 API 行为

### C5 🟠 租户级资源配额与噪声租户

**陷阱**：共享库 SaaS 下，一个租户的大查询、大导出、规则循环、批量导入、频繁发布 DDL、Connector 慢调用或附件上传，就可能占满数据库连接、线程池、异步队列、文件存储和 CPU，拖垮其他租户。只做全局限流无法防噪声租户。
**细则必须回答**：
- 租户级配额：请求 QPS、并发查询、导出行数、规则执行时长、Connector 调用、附件容量、异步队列积压、DDL 发布频率。
- 超额策略：拒绝、排队、降级、只读、人工审批的触发条件和错误码。
- 资源隔离指标与告警必须按租户维度可见，不能只有全局平均值。
- 压测/演练：单租户大导出、规则死循环、慢 Connector、批量导入时，其他租户核心 CRUD 仍可用。

### C6 🟠 定时调度引擎的平台级设计缺失

**陷阱**：autonumber 按日重置、规则异步重试、通知重发、工作流超时节点、日志归档、导出任务、附件清理全部隐式依赖"定时调度"，但平台没有统一调度设计。各处自己起 @Scheduled 的结果：多实例重复执行（重复通知、重复取号重置）、实例宕机漏触发无补跑、调度线程里租户上下文丢失（触发 B1）。
**细则必须回答**：
- 统一调度机制定死一种（DB 抢锁 / 分布式调度框架 / 单实例调度角色），禁止业务代码散落 @Scheduled。
- 漏触发的补跑语义按任务分类定死：必须补跑（归档、outbox 投递）/ 跳过即可（缓存预热）；补跑的幂等要求（与 F1 关联）。
- 租户时区语义："每天 0 点"是平台时区还是租户时区——按任务类型逐一定死（与 D1 时区策略对齐）。
- 调度任务的租户上下文注入模板（与 B1 的全场景模板合并）与租户级执行配额（与 C5 关联）。
- 测试集：双实例并发调度防重、实例宕机后补跑、跨时区租户的按日任务。

## D. 类型系统与数据语义（隐蔽但长痛类）

### D1 🔴 字段类型转换的静默数据损坏

**陷阱**：动态层做 22 种类型的字符串→Java→JDBC 转换：时区（DATETIME 无时区，SaaS 跨时区租户）、BigDecimal 精度静默截断、空字符串 vs NULL 的歧义、boolean 的 "0"/"false"/0 多形态。任何一个歧义都是静默数据损坏。
**细则必须回答**：
- 22 种 field_type 的**类型转换真值表**：每种类型的合法输入形态、非法输入的行为（拒绝，不猜）、NULL/空串语义——逐类型定死
- 时区策略：存储时区、传输格式、租户时区配置的换算点在哪一层——定死
- 精度：decimal 入库前校验超精度直接拒绝（不四舍五入）——定死

### D2 🟡 软删除的连带语义

**陷阱**：软删除记录的 link 引用还指着它——列表展开显示已删客户？unique 约束被已删记录占用？on_delete=restrict 检查含不含已删行？
**细则必须回答**：软删除与 unique 索引方案（deleted 参与唯一键 or 墓碑值）、link 展开对已删目标的显示规则、restrict 判定口径——三项定死。

### D3 🟡 对象/字段的 rename 语义

**陷阱**：改 field code = 物理列改名（危险 DDL）+ 页面 Schema/规则表达式/fetch_from 里的引用全部失效——这是低代码平台最高频的用户操作之一。
**细则必须回答**：code 是否允许改（建议禁改，只改显示名——定死）；若禁改，创建时的 code 质量约束和提示规范；引用完整性检查器（lc_meta_ref）在改名/删除时的阻断行为。

### D4 🔴 formula 字段的求值语义与循环依赖

**陷阱**：formula 是 22 种字段类型之一，但求值语义完全未定：读时求值则无法下推 SQL（不能过滤/排序/分页，或只能内存排序导致深分页语义崩塌）；写时物化则源字段变更后需要级联重算（何时算、算多少行、失败怎么办）。formula 引用 formula、经 link 跨对象引用会形成循环依赖；源字段被删/改类型后 formula 静默失效或返回错值都是数据事故。
**细则必须回答**：
- 求值时机定死一种：读时求值 / 写时物化 / 混合（并写明各自对过滤、排序、导出、聚合的能力边界，作为产品语义公开）。
- 若物化：源字段变更触发重算的时机（同步/异步/惰性）、批量重算的行数与事务边界、重算失败的状态与告警。
- 循环依赖检测：保存时全图检测（含跨对象 link 链），深度上限定死；检测必须纳入 lc_meta_ref 引用索引（与 G1 关联）。
- 源字段删除/改类型/改权限时 formula 的行为：发布阻断（推荐）还是运行时降级为 null——定死并给用户可读的影响报告。
- formula 结果的字段权限：引用了用户无权字段的 formula，结果对该用户是否可见——定死（与 B3 判定器单一入口关联）。

### D5 🔴 fetch_from 的快照 vs 实时语义

**陷阱**：fetch_from（从 link 目标对象带出字段值）的语义未定死：是"写入时拷贝快照"还是"读取时实时取"？两者对审计、历史单据、报表的含义完全不同。快照语义下源记录后来改名，历史订单显示旧客户名是正确行为还是 bug，用户会持相反预期；实时语义下则产生 C1 的读放大且历史数据随源漂移、不可追溯。
**细则必须回答**：
- 语义已定为写入时快照落列，并写成给用户看的产品能力边界，不是内部备注。
- 快照语义下：是否提供"刷新带出值"的显式动作；源记录更新是否提供可选的级联刷新规则（及其写放大代价，与 C2 关联）。
- link 目标被软删除/用户对目标无 read 权限时，已落地的 fetch_from 快照值是否仍显示——与 B3、D2 的口径对齐，三处不得矛盾。
- fetch_from 链（A.b 的 b 又是 fetch_from）是否允许，允许则深度上限与取值时机定死。

### D6 🟡 many_to_many 中间表的物理设计

**陷阱**：multilink/many_to_many 需要物理中间表，但中间表不是元模型"对象"，容易游离在所有系统性设计之外：没有 tenant_id、没有软删除口径、不进物理结构登记表、不进备份清单、审计不知道怎么记（改一次 multilink = N 行中间表增删）。
**细则必须回答**：
- 中间表的命名规则、系统列集合（tenant_id 必须有，deleted/revision 是否有——定死）、unique 约束设计。
- 中间表必须登记进物理结构登记表（A3）与备份对象清单（F6）。
- multilink 修改的审计粒度：记录最终集合 diff（加了谁删了谁），不逐行记中间表变更。
- 两端记录软删除/恢复时中间表关系的连带口径（与 D2 对齐）。

### D7 🟡 on_delete=cascade/set_null 的应用层实现

**陷阱**：动态表没有真外键，cascade/set_null 由引擎在应用层实现。未定死的问题：级联深度上限、单事务级联行数上限（删一个客户级联十万订单行？）、级联删除是否触发目标对象的规则/审计/权限校验、cascade 链上遇到 restrict 时整体如何裁决。D2 只回答了 restrict 的判定口径。
**细则必须回答**：
- 级联深度与单次级联行数上限，超限的行为（阻断并提示，不静默截断）。
- 级联操作是否走统一写入口（触发规则、审计、权限）——定死：必须走（性能代价写明）或明确豁免项清单。
- 混合链（cascade→restrict）的裁决顺序：先全链预检再执行，禁止执行到一半才发现 restrict。
- 级联属于软删除还是硬删除，与 D2 的软删除连带语义对齐。

### D8 🟡 collation 与 code/unique 的大小写语义、ID 生成陷阱

**陷阱1**：utf8mb4 的 ci 类 collation 下 `Abc` 与 `abc` 在 unique 索引上冲突，但 Java 层 equals 认为不同——对象 code 唯一性、text 字段 unique 的大小写口径若不与 DB collation 定死一致，会出现"代码判重通过、入库唯一键炸"或反向的静默重复。
**陷阱2**：雪花 ID 时钟回拨导致重复/乱序；ULID 同毫秒内多实例生成的单调性不保证——依赖 lid 排序的逻辑会出错。
**细则必须回答**：
- 动态表与元数据表的 collation 定死（含 code 列是否用 bin collation）；Java 层判重逻辑必须与之一致的检测方式。
- 雪花时钟回拨的处理策略（等待/报错/备用位），ULID 生成器的单调性声明与禁止依赖 lid 做时间排序的规范（或提供保证）。

### D9 🔴 标准单据扩展与客户定制被升级覆盖

**陷阱**：正式商用平台一定会有标准对象、行业模板、客户扩展和插件扩展。如果所有定制都直接改 BusinessObject，平台升级、行业包升级或插件升级会覆盖客户字段、规则、页面和权限；如果用无限继承模拟扩展，又会让字段合并、状态机、权限、Schema Sync 和发布回滚不可预测。
**细则必须回答**：
- 标准对象、行业模板、客户扩展、插件扩展的来源和优先级如何记录。
- 哪些字段、规则、状态、动作、页面允许扩展，哪些禁止覆盖。
- 扩展合并报告如何生成，冲突时是阻断、人工选择还是自动合并。
- M0/M1 是否只允许模板复制 + 受控扩展层；如支持父子单据继承，必须先有 ADR。
- 测试集：标准对象升级不覆盖客户扩展、插件禁用后扩展消失、冲突扩展被阻断。

### D10 🔴 单据转换、反写与引用追踪缺失

**陷阱**：ERP 型业务低代码的核心不是单表 CRUD，而是订单下推出库、出库转收款、下游审核反写上游数量/金额/状态、上查下查引用链。若把它简化成复制数据或普通 link，就无法处理分组合并、部分转换、超额控制、关闭控制、反写幂等和审计追踪。
**细则必须回答**：
- LinkConfig、Conversion、WriteBack、LinkTrace 是否是一等元模型，而不是规则里的临时 JSON。
- 转换字段映射、选单条件、分组策略、排序策略、插件策略和表单服务策略如何表达。
- 反写触发动作、反写公式、反写模式、多源单分配、超额控制和关闭控制如何表达。
- 引用追踪如何在权限裁剪、导入导出、回滚和删除时保持一致。
- 测试集：部分下推、防超额、重复审核幂等、反写失败补偿、上查/下查权限裁剪。

### D11 🟠 弹性域、辅助属性与多组织业务关系被简化

**陷阱**：物料规格、批次维度、辅助属性、多组织职能和组织间业务关系是企业业务平台的一等模型。若把弹性域简化为 JSON 大字段，后续校验、字段映射、报表查询、权限和单据转换都会失去结构；若组织只做 `org_id`，采购/销售/库存/核算组织关系无法表达。
**细则必须回答**：
- 弹性域维度、属性组合、适用对象、校验规则、字段映射和报表查询边界。
- 多组织分类、组织职能、组织业务关系、生效时间和历史组织变更语义。
- CodeRule 是否能按组织、条件、对象和字段生成业务编码。
- M0 哪些只是 DTO 预留，M2/M3 前哪些必须补 ADR。
- 测试集：维度组合校验、组织职能权限、组织关系变更、编码规则按组织隔离。

### G8 🟠 报表、打印、菜单、门户、移动端和多语言被页面 schema 吞掉

**陷阱**：报表不是列表导出，打印不是页面截图，菜单不是页面树，多语言不是硬编码 label，移动端也不是压缩桌面表单。如果这些能力都塞进 page schema，后续权限裁剪、版本归档、模板升级、门户入口、移动端交互和 i18n 回放都会失控。
**细则必须回答**：
- Report、PrintTemplate、Menu、I18nResource 是否有独立元数据和版本。
- 报表聚合口径、维度指标、权限裁剪和缓存边界。
- 打印模板的版本归档、签章/水印、敏感字段脱敏和审计。
- 菜单/门户/移动端入口权限是否独立于页面 schema。
- 测试集：报表权限裁剪、打印旧模板回放、菜单权限、移动端 schema 适配、多语言缺失 fallback。

### F12 🟠 应用包、行业模板、市场与 License 生命周期缺失

**陷阱**：正式商用平台会交付行业模板、插件、应用包和私有化授权。如果没有安装、升级、禁用、卸载、迁移、License、兼容检查和审计模型，交付会退回人工复制元数据；升级失败后也无法判断哪些对象来自模板、哪些来自客户扩展。
**细则必须回答**：
- 应用包/行业模板/插件包的 manifest、版本、依赖、License、迁移脚本和兼容范围。
- 安装/升级/回滚/卸载时如何处理对象扩展、菜单、页面、权限、报表、转换规则和数据迁移。
- License 离线授权、审计、过期、降级和应急策略。
- 市场分发是否允许第三方后端代码，若允许，安全扫描和隔离边界是什么。
- 测试集：旧包升级、License 过期降级、模板升级与客户扩展冲突、卸载保留数据。

## E. 工程与演进（半年后爆发类）

### E1 🟡 内部 API 与元模型 DTO 的兼容纪律

**陷阱**：FieldDef 等 JSON DTO 同时被 元数据存储/Version 快照/前端 meta 接口/导入导出 四方消费，随手加删字段会破坏历史快照反序列化和已导出的应用包。
**细则必须回答**：
- JSON DTO 的字段增删规则（只增不删不改语义、废弃标记流程）、Jackson 反序列化对未知字段的策略（忽略但告警？）、导入导出包的版本兼容承诺范围。
- 平台代码升级导致字段类型转换/表达式求值行为变化时，对存量数据的语义漂移检测与升级说明义务（与 D1 真值表版本化关联）。

### E2 🟡 "临时逃逸"的架构腐蚀

**陷阱**：交付压力下最容易发生"这个场景元模型不支持，先在 Java 里硬编码特判某租户/某对象"——第一个特判出现后平台会迅速腐化成项目制代码。
**细则必须回答**：禁止在引擎代码中出现 tenant/app/object 的字面量特判（检测：代码扫描 tenant_id == / code.equals 常量）；表达力不足的正规出口（登记需求→评估进元模型 or 进插件机制）。

### E3 🟡 测试环境与生产的结构漂移

**陷阱**：动态 DDL 意味着**每个环境的物理结构由各自的元数据操作历史决定**，测试库和生产库结构必然漂移，Flyway 只管 lc_meta_*，动态表没人管——"测试过了、生产炸了"。
**细则必须回答**：环境间应用迁移的唯一合法通道（导出/导入包，禁手工 SQL 搬运）；导入时 Schema Sync 的预检报告（将执行哪些 DDL，人工确认）；生产 DDL 的执行窗口与大表在线 DDL（INSTANT/INPLACE 判定）规则。

### E4 🟡 SaaS 与私有化部署模式漂移

**陷阱**：设计目标是同一套代码支持多租户 SaaS 和私有化部署。若配置、存储、文件、连接器出网、License、升级、备份、监控在两种部署形态下各走一套逻辑，半年后会变成两个产品，补丁、规范、测试都无法复用。
**细则必须回答**：
- SaaS/私有化差异只能通过明确的 deployment_mode、配置和适配层表达，禁止业务代码到处 `if private` 特判。
- 两种模式下 tenant_id、权限、文件存储、DataSource、插件、备份恢复、License、升级流程的差异边界。
- CI/回归必须至少有 SaaS shared 模式和私有化单租户模式两套最小验收。
- 模式特有能力必须进入 ADR 和产品边界，不能只靠部署文档说明。

## F. 外部边界与副作用（真实商用后最容易爆雷）

### F1 🔴 动作/状态流转的并发、幂等与重复执行

**陷阱**：状态流转和 Action 是业务动作入口。用户重复点击、前端超时重试、网关重放、两个审批人同时操作同一记录，都可能让同一动作执行多次：重复流转、重复通知、重复生成单号，甚至未来接入库存/支付后重复扣减。
**细则必须回答**：
- 哪些接口必须强制幂等键（transition/action/batch action/import/publish），幂等键的作用域、过期时间、重复请求返回语义。
- 状态流转写入必须带当前状态 CAS 或 revision 条件，不能只按 id 更新；并发冲突的错误码和前端提示。
- Action 的副作用执行顺序：状态变更、规则、审计、通知、外部调用之间谁同事务、谁异步、失败如何补偿。
- 幂等与并发测试集：重复提交、并发审批、超时后重试、批量动作部分成功。

### F2 🔴 规则链副作用、重试与补偿

**陷阱**：规则从 validate/assign 扩展到 notify/call_api 后，会把业务事务和外部副作用绑在一起。若事务提交后通知丢失，业务不可追踪；若外部调用成功但本地事务回滚，会出现外部系统已执行、本地无记录；若失败后盲目重试，又可能重复执行。
**细则必须回答**：
- before_save/after_save/before_transition/after_transition 中哪些规则允许有副作用，哪些必须纯函数。
- 外部副作用必须走 outbox/任务表还是可同步调用；重试次数、退避、死信、人工补偿、幂等键规则。
- 规则循环与级联上限：assign 触发 fetch_from、规则触发规则时的最大轮数、循环检测、错误处理。
- 规则执行日志必须能串联业务记录、规则版本、输入摘要、输出摘要、异常、traceId。

### F3 🔴 导入/导出包的安全、完整性与引用重映射

**陷阱**：应用导入/导出是跨环境迁移的唯一合法通道，也会成为绕过设计器校验的入口。恶意包可携带危险表达式、悬空引用、冲突 code、伪造版本、过大 JSON、跨租户引用，甚至把 DataSource 密钥打包外泄。
**细则必须回答**：
- 导出包格式、版本、清单、校验和/签名、大小上限；哪些内容禁止导出（密钥明文、token、运行时业务数据）。
- 导入时 id/code/引用的重映射规则：对象、字段、页面 schema、规则、权限、插件、DataSource 的冲突处理。
- 导入必须先跑全图校验和 Schema Sync 预检，输出影响报告，禁止边导入边半应用。
- 恶意包测试集：表达式注入、路径穿越文件名、超大 JSON、重复 code、跨租户 id、未知 DTO 字段。

### F4 🔴 DataSource/Connector 的 SSRF、凭据泄露与出网治理

**陷阱**：DataSource/Connector 一旦支持 HTTP/MySQL 等外部连接，平台就具备主动访问网络的能力。用户配置一个内网地址即可探测云元数据服务、Redis/MySQL 内网、管理后台；连接失败日志还可能泄露账号、URL、SQL、响应体。
**细则必须回答**：
- 出网白名单/黑名单策略：禁止访问 localhost、私网段、云 metadata 地址、保留地址；DNS rebinding 如何防。
- 每类 Connector 的超时、重试、限流、最大响应体、最大结果集、连接池隔离和租户级配额。
- 凭据加密、脱敏、轮换、最小权限、运行态解密边界；设计器用户是否能测试连接、能看到哪些错误。
- Connector 调用必须走统一审计与 trace，不允许插件或规则绕过出网治理。

### F5 🔴 attachment 文件字段安全与生命周期

**陷阱**：attachment 是用户可上传二进制内容的入口。若只把它当普通字段，会引入路径穿越、恶意文件、跨租户下载、超大文件打爆存储、软删除后文件泄露、导出包夹带敏感文件等问题。
**细则必须回答**：
- 文件类型白名单、扩展名与 MIME 双校验、大小上限、病毒扫描/异步隔离区策略。
- 存储路径/对象 key 必须包含租户隔离维度且不可由用户输入直接拼接；下载必须重新走权限判定。
- 文件引用计数、软删除、恢复、永久删除、归档、导入导出时的处理规则。
- 文件上传/预览/下载日志脱敏规则，禁止把原始路径、签名 URL、内部 bucket 信息暴露给前端或日志。

### F6 🔴 元数据、动态表、业务数据的一致性备份恢复

**陷阱**：备份恢复不是简单备份 MySQL。低代码平台的数据正确性依赖三件事同一逻辑时间点一致：Version.snapshot、动态表物理结构、动态表业务数据。只恢复其中一部分，会出现元数据引用不存在的列、业务数据字段语义错位、已发布版本无法加载。
**细则必须回答**：
- 备份对象清单：lc_meta_*、lc_rt_*、动态业务表、物理结构登记表、文件存储、插件包、配置密钥。
- PITR 恢复的逻辑时间点如何选择，恢复后 Reconciler 如何强制对账，差异如何阻断运行态。
- 单租户恢复、单应用恢复、全实例恢复分别是否支持；如果不支持，必须写成产品边界。
- 定期恢复演练和验收脚本：恢复后必须跑元数据加载、Schema Sync dry-run、样例应用 CRUD、权限隔离测试。

### F7 🔴 权限缓存失效与组织变更越权窗口

**陷阱**：权限不只由 Role 定义决定，还依赖用户状态、部门、组织树、岗位、数据范围表达式和租户成员关系。用户调岗、禁用、移出部门、角色撤销后，如果只按 app 粗粒度失效缓存，会在窗口期继续读写旧范围数据。
**细则必须回答**：
- 权限缓存 key 必须包含哪些版本：用户、角色、组织树、数据范围、应用权限、字段权限。
- 用户禁用/退出租户/部门迁移/角色变更/权限发布时的失效事件和最大生效时限。
- 运行中请求是否使用请求入口固定的权限视图；权限变化与 MetaGraph 版本变化的先后关系。
- 安全测试集：调岗前后 self/dept/dept_tree 范围变化、禁用用户 token、角色撤销后旧缓存。

### F8 🔴 工作流在途实例与流程定义版本漂移

**陷阱**：工作流进入 M3 后，流程定义会升级，但历史实例可能仍在旧节点等待。若运行时总按最新流程定义执行，旧实例的节点、条件、角色、表单字段可能已不存在；若永远冻结旧定义，又会留下无法迁移的长尾实例。
**细则必须回答**：
- 工作流实例创建时是否冻结 workflow version；节点、条件、角色、表单 schema、动作权限随实例冻结还是动态读取。
- 流程定义升级时，在途实例的迁移策略：继续旧版、人工迁移、自动映射、终止重开；每种策略的适用条件。
- 定时节点、等待节点、会签/或签、撤回、退回、转交、超时升级的幂等和补偿规则。
- 测试集：旧流程实例遇到新版定义、节点删除、角色删除、字段删除、定时任务重复触发。

### F9 🔴 认证会话与全 POST 接口的 CSRF

**陷阱**：B 章全部在讲授权，认证层零覆盖。全 POST + cookie 会话是经典 CSRF 靶子；用户被禁用/移出租户后已签发的 token 若不失效，与 F7 的权限缓存窗口叠加成越权窗口；未来若开放第三方 API 调用而没有预留 token 体系，会被迫复用浏览器会话机制——安全模型直接崩坏。
**细则必须回答**：
- CSRF 防护方案定死一种（SameSite + 自定义 header / CSRF token），并给出全 POST 接口的统一强制点与检测方式。
- token/会话生命周期：签发、过期、续期、用户禁用/退出租户/改密后的强制失效时限（与 F7 的最大生效时限对齐为同一承诺）。
- 登录防护基线：失败锁定、验证码触发条件、会话固定防护；若复用既有身份系统，写明边界与集成契约。
- 开放 API 的产品边界定死：首版是否支持第三方 token；不支持则明确写入产品边界，支持则 token 的作用域/限流/审计要求。
- 测试集：CSRF 攻击用例、禁用用户旧 token、跨租户 token 重放。

### F10 🟠 业务数据批量导入导出（Excel/CSV）

**陷阱**：F3 只覆盖了"应用元数据包"，业务数据的 Excel/CSV 导入导出是另一个怪兽：1 万行错 3 行是全回滚还是跳过？导入触发规则/审计 = 写放大 ×4×1 万行；不触发 = 绕过校验与权限的第二入口（与 H1 同类风险）。大文件全量读入内存直接 OOM。导出如果不走字段权限与数据范围判定，就是最大的数据泄露口。
**细则必须回答**：
- 部分成功语义定死：分批事务大小、错误行的收集与错误报告格式、失败后的幂等重导（与 F1 幂等键关联）。
- 导入必须走统一写入口（类型转换真值表 D1、校验、权限、租户注入），规则/审计的触发策略按行分级定死（哪些规则导入时豁免须显式声明并审计）。
- 流式解析强制（禁止全量载入内存）、单次导入行数/文件大小上限（纳入 C5 租户配额）。
- 导出必须复用 B3 的字段判定器与 B2 的数据范围注入——同一判定器，禁止导出链路另写一份；导出文件的保留期、下载鉴权、审计（与 B6、F5 对齐）。
- 测试集：部分失败重导幂等、越权字段导出裁剪、超大文件、恶意 CSV（公式注入 =cmd、超长单元格）。

### F11 🟠 通知的数据出域与通道治理

**陷阱**：notify 把字段值发进邮件/钉钉/企微，数据就离开了平台权限体系——收件人可能对该记录根本无 read 权限；模板里引用手机号/金额等敏感字段等于主动外泄（B6 的分级形同虚设）；通道失败静默丢通知则业务不可追踪，盲目重试则轰炸用户。
**细则必须回答**：
- 通知模板可引用字段的白名单机制：与 B6 数据分级联动，敏感等级以上字段禁入模板或强制脱敏。
- 收件人权限校验策略定死：发送前校验收件人对记录的 read 权限（推荐）或明确声明通知内容视为授权披露并写入产品边界。
- 通道可靠性：复用 F2 的 outbox/重试/死信机制（禁止另建一套），通知发送日志与业务记录、traceId 串联。
- 租户级通知频率配额与合并策略（防轰炸，纳入 C5）。
- 测试集：无权限收件人、敏感字段模板、通道超时重试不重复发送。

## G. 设计器与前端一致性（配置能保存不代表运行时能用）

### G1 🟠 设计器保存的配置不是运行时可执行配置

**陷阱**：对象、字段、页面 schema、规则、权限、状态机分散在不同设计器页面。局部页面保存成功，不代表整张应用图可发布：页面引用已删字段、规则引用旧 code、按钮绑定不存在 action、权限引用不存在角色，都会在发布或运行时才炸。
**细则必须回答**：
- 设计态保存校验与发布前全图校验的职责边界；哪些错误保存即拒绝，哪些允许草稿存在但发布阻断。
- lc_meta_ref 或等价引用索引必须覆盖 field/action/state/rule/page/permission/plugin/connector 的引用关系。
- 发布前必须输出可读的引用完整性报告，错误定位到具体对象、字段、页面节点或规则。

### G2 🟠 前端镜像表达式与后端表达式语义漂移

**陷阱**：depends_on/visible_condition 需要前端即时求值，但最终校验在后端。如果前后端对 null、空串、数字精度、日期时区、字符串比较、in/contains 的语义不一致，就会出现前端显示可填但后端拒绝，或前端隐藏字段导致用户无法修复后端错误。
**细则必须回答**：
- 前端表达式子集的语法、类型转换、空值、比较、日期、数字精度必须有对照表，禁止"大致兼容"。
- 契约测试：同一批表达式和输入样例必须在前端求值器与后端表达式门面得到一致结果。
- 当前端无法支持某表达式时，设计器如何限制配置，运行时如何降级显示。

### G3 🟡 大对象设计器性能与协作冲突

**陷阱**：单对象允许数百字段后，字段编辑器、权限矩阵、页面 schema、状态机画布都会变成大 JSON 整体编辑。多人同时编辑、浏览器卡顿、局部修改覆盖整体、权限矩阵无法扫描，都会把设计器变成事故入口。
**细则必须回答**：
- 大对象阈值：字段数、规则数、权限项、页面节点数达到多少必须启用搜索/分组/虚拟列表/分页。
- 设计器编辑锁、乐观锁、草稿合并、冲突展示策略；禁止静默覆盖他人修改。
- 保存粒度：整对象保存还是局部 patch；若局部 patch，如何保持 JSON `_v` 与引用索引一致。

### G4 🔴 前端渲染器 XSS 与 Schema 注入

**陷阱**：运行时渲染器会渲染 richtext/code/attachment/link 展开和页面 schema。若 schema props 可透传到组件、富文本未净化、附件预览直接内嵌、错误信息含 HTML，攻击者可通过低代码配置或业务数据注入脚本，影响所有访问该应用的用户。
**细则必须回答**：
- richtext/code/markdown/html 类内容的允许范围、净化库、服务端/前端双层净化职责和禁止项。
- 页面 schema 的 props 白名单：禁止透传危险属性（如 dangerouslySetInnerHTML、style 表达式、事件 handler、iframe/srcdoc）。
- attachment 预览隔离：下载 vs 预览、Content-Type、Content-Disposition、sandbox iframe、签名 URL 有效期。
- XSS 测试集：字段值、字段 label、错误消息、富文本、附件文件名、导入 schema、插件前端组件。

### G5 🟠 OpenAPI 与动态对象类型生成的边界

**陷阱**：前端接口类型从 OpenAPI 生成，但低代码对象字段是运行时动态元数据。若把动态对象伪装成静态 DTO，前端会误以为类型安全已经覆盖字段、权限、版本和 schema 变化，导致运行时字段缺失、旧类型提交新对象、权限裁剪后访问 undefined。
**细则必须回答**：
- 哪些 API 可以生成静态类型（平台管理、元模型 DTO），哪些只能生成 runtime schema + unknown/record 结构。
- 动态对象的字段类型、必填、权限裁剪、版本号如何在前端运行时校验，而不是依赖编译期类型。
- OpenAPI 变更与 meta version/schema version 的关系；前端缓存旧类型或旧 meta 时如何刷新。
- 契约测试：旧前端类型访问新 meta、新前端访问旧 meta、权限裁剪字段缺失、导入旧 schema。

### G6 🟡 设计态预览与草稿数据边界

**陷阱**：设计器"预览"功能的数据边界未定：直连生产业务表 = 预览时的增删改污染真实数据、且草稿元数据绕过了发布版本机制直接作用于生产数据；用假数据预览 = 表达式、规则、权限行为失真，"预览通过、发布后炸"。
**细则必须回答**：
- 预览的读写边界定死成产品语义：只读真实数据 / 沙箱数据 / 禁止预览带写操作——选定一种并写明代价。
- 预览使用草稿元数据还是已发布元数据；草稿元数据参与运行时求值的隔离机制（禁止污染 MetaGraph 已发布视图，与 C3 关联）。
- 预览请求的审计与标识（防止预览操作与真实业务操作在审计中混淆）。

### G7 🟡 错误信息的信息泄露与探测面

**陷阱**："对象不存在"与"无权限"返回不同错误码，攻击者即可枚举探测租户内对象/记录的存在性；错误信息带 SQL 片段、物理列名、内部类名、堆栈，则直接泄露物理结构与实现细节。前端为了体验要细错误，安全要求模糊错误——不定死双层口径，两边会各写一套。
**细则必须回答**：
- 错误码体系的双层口径定死：对外（不存在与无权限归一为同一响应）、对内（traceId 关联完整错误落日志）；哪些场景允许对外精确错误（如表单校验）的白名单。
- 禁止对外错误信息包含：SQL、物理表列名、堆栈、内部类名、文件路径、连接串——检测方式（响应体扫描测试/日志脱敏规则，与 B6 对齐）。
- 与 H2 的状态机错误码、G2 的前端错误展示衔接：前端对模糊错误的统一提示文案规范。
- 测试集：越权探测归一化响应、错误响应体敏感信息扫描。

## H. 代码实现反模式（AI 最容易写出来但评审不一定能看见）

### H1 🔴 动态 SQL 白名单"看似校验，实际绕过"

**陷阱**：代码里有白名单类不等于所有 SQL 都安全。常见绕过包括：sort 字段未校验、op 直接拼接、JOIN path 未校验、函数/聚合字段放行、分页参数不限制、表别名可注入、link 展开临时 SQL 绕过统一组装器。
**细则必须回答**：
- 表名、列名、别名、op、sort、join path、函数、分页、group by、having 的逐项白名单入口。
- 所有动态 SQL 只能通过统一 QueryBuilder/SqlAssembler；禁止 mapper XML 或字符串拼接另开口子。
- 注入测试集必须覆盖字段名、op、sort、link path、分页、批量 ids、导出查询、统计查询、聚合查询（group by 结果对无权行的存在性泄露）。

### H2 🟠 `catch Exception` 破坏发布/规则/DDL 状态机

**陷阱**：发布、DDL、规则、异步任务这类流程依赖状态机记录真实结果。若代码 catch 后只 log 或返回 false，上层会误以为成功或无法恢复，A1/F2 里的恢复设计全部失效。
**细则必须回答**：
- 哪些流程禁止吞异常，必须落库失败状态并携带错误码、traceId、可恢复建议。
- 允许捕获后降级的白名单场景；降级必须有告警和可观测指标。
- Checkstyle/扫描规则：禁止空 catch、禁止 `catch Exception` 后不 rethrow/不转业务异常/不记录状态。

### H3 🟠 测试只测单租户、单实例、主路径

**陷阱**：低代码平台最严重事故通常不在单租户单实例主路径出现。只测一个租户、一个应用、一个实例、一次发布、一次提交，会漏掉串租户、版本窗口、缓存漂移、并发动作、异步上下文丢失。
**细则必须回答**：
- 核心链路测试矩阵必须包含：双租户、双应用、双实例元数据版本、并发写、发布中请求、异步规则、权限变更后旧缓存。
- 哪些测试必须用 Testcontainers 真 MySQL/Redis，哪些可单测；禁止 H2 覆盖动态 DDL 语义。
- 回归测试数据必须包含恶意 filters、旧 Version.snapshot、旧导出包、软删除 link、超精度 decimal、跨时区 datetime。

### H4 🟡 AI 生成代码的通用抽象膨胀

**陷阱**：AI 很容易为"未来可扩展"提前生成通用框架、万能工具类、过深继承、过宽接口、重复的 Result/Exception/Mapper 封装。短期看完整，长期会绕过元模型、增加调试成本、让真正的扩展点被伪抽象淹没。
**细则必须回答**：
- 新增抽象的准入标准：至少两个真实调用方、能减少重复或隔离明确变化点；否则保持局部实现。
- common 模块禁止承载业务语义；support 包禁止跨模块引用；工具类禁止持有可变全局状态。
- 代码评审必须检查"是否有未被需求证明的抽象/扩展点/配置项"，发现则删除或降级为局部代码。

## 使用方式（对 06 任务书的增补要求）

1. 细则 AI 产出的每份细则文档开头必须附**《陷阱覆盖表》**：本文档全部条目中与该细则相关的条目 → 对应细则编号。全部细则文档合并后必须覆盖全部条目，未覆盖 = 验收不通过。
2. 每个 🔴 级条目至少要有一条**机器可检测**的防线（ArchUnit/扫描/强制单测），做不到的要写明"为什么只能靠人工评审 + 评审checklist条目"。
3. 本清单本身随实现推进而增补：执行 AI 在实现中发现新陷阱 → 报告 → 人类确认后由维护者追加（AI 不得自行增删条目）。
