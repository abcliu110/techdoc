## 0. 文档边界

T-207 表单设计器不是普通问卷设计器，也不是只输出前端 JSON 的页面搭建器。它是企业低代码平台中的动态表单/单据页面设计器，必须连接 BO 元模型、页面类型、多视图、字段控件、属性、规则、权限、插件、保存校验、发布快照、影响分析和运行态渲染。

本项目的核心竞争力由三类能力共同构成：

1. **强大的 Web 布局能力**：能设计真实 PC Web 业务页面，不退化成简单两列表单。
2. **丰富的表单元素**：字段、选择、引用、上传、展示、业务语义、布局容器、操作命令都有统一契约。
3. **强大的表格控件**：面向企业单据分录和宽表，支持冻结列、横向滚动、单元格编辑、行列权限、公式汇总和大数据量性能。

本文回答“做什么、给谁用、做到什么程度才算可验收”。它不声称代码、测试执行或上线已完成。

## 1. 背景与目标

### 1.1 问题

低代码平台如果缺少企业级表单设计器，会出现三个问题：

1. 设计器只能拖控件，不能表达复杂 PC Web 布局、单据、规则、权限和发布语义。
2. 设计态和运行态分裂，导致预览可用但上线行为不一致。
3. 表格能力弱，无法支撑企业单据分录、宽表录入、冻结列、批量录入和行列权限。
4. 发布前缺少元数据、权限、规则、插件和影响分析校验，埋下运行风险。

金蝶公开资料显示，成熟企业表单设计器围绕动态表单基类、单据设计链路、页面属性、规则权限、插件扩展和高密度工作台形成体系。T-207 的目标是学习这种架构思想，同时在 Web 布局引擎、表单元素完整性、表格控件、权限模拟、一致性报告、AI 可解释辅助和版本影响分析上超过金蝶。

### 1.2 目标用户

| 用户 | 目标 |
|---|---|
| 应用设计者 | 拖字段、配属性、配规则、预览并发布业务表单。 |
| 业务建模者 | 将 BO 字段、子表和关系生成多视图页面。 |
| 租户管理员 | 管理字段权限、操作权限、发布权限和运行入口。 |
| 业务用户 | 使用发布后的表单录入、保存、提交、审批和查询。 |
| 插件开发者 | 在受控事件点扩展复杂逻辑。 |
| UI/交互设计师 | 定义高密度、可扫描、可验收的设计器体验。 |
| 运维/审计人员 | 审计版本、发布、权限、插件和运行态错误。 |

## 5. 约束与假设

### 5.1 约束

- 必须集成到现有低代码平台，不另起孤立产品线。
- Designer 必须复用 Runtime Renderer，避免设计态和运行态长期分裂。
- 强 Web 布局能力、丰富表单元素、强表格控件必须作为首版 P0 验收主轴，不得退化为普通控件拖拽器。
- PageSchema 不能放宽后端对象权限、字段权限、状态机和校验规则。
- 金蝶资料只能作为公开证据和设计参考，不复制界面资产、图标、文案和闭源实现。
- AI 能力不能成为发布、权限和校验的权威来源。

### 5.2 假设

- 平台已有或将提供 BO、字段、权限、表达式、发布、动态数据 API。
- 首版以桌面端企业后台设计器为主，移动端设计能力先预留。
- 首版以销售订单复杂单据样例验证核心闭环，必须覆盖主表、多区块布局、分录表格、汇总、权限、规则和发布。
- 插件先做注册声明、事件点和边界校验，复杂运行时按后续阶段深化。

---

## 6. 超出范围

- 首版不复制金蝶具体 UI、图标、颜色值、模板图和文案。
- 首版不实现完整单据转换、反写、套打、工作流、部署包生态。
- 首版不开放任意前端脚本或自由 JavaScript 执行业务逻辑。
- 首版不做完整移动端/H5 表单设计器。
- 首版不做通用 BI 报表设计器。
- 首版不承诺所有长尾组件一次性交付；但 P0 表单元素、PC Web 布局能力和强表格控件必须支撑销售订单闭环。
