## 1. 产品定位

本平台定位为独立商用的企业业务低代码平台，核心是“业务对象元模型驱动应用”，不是只拼页面和 API 的内部工具构建器。

目标客户：

- 有多个业务系统建设需求的中大型企业。
- 需要交付行业应用或客户定制应用的 ISV。
- 需要 SaaS 与私有化两种部署形态的产品团队。

核心价值：

1. 业务对象、字段、权限、状态、规则和页面统一建模。
2. 建模后自动获得 API、默认页面和运行态能力。
3. 通过版本发布把设计态变更安全进入运行态。
4. 支持长期演进、插件扩展、多租户隔离、审计和运维治理。

非目标：

- 不做通用 BI 平台。
- 不做纯流程引擎产品。
- 不做 Appsmith/Retool 类“连接器 + 页面脚本”优先的内部工具平台。
- 首版不支持自由 JavaScript 业务脚本作为逃逸口。

## 2. 用户角色

| 角色 | 核心诉求 | 必须能做 | 不能默认做 |
|---|---|---|---|
| 平台管理员 | 管理租户、版本、插件、运维安全 | 租户管理、全局配置、审计查看、break-glass 审批 | 绕过审计查看密钥明文 |
| 租户管理员 | 管理本租户应用与成员 | 创建应用、授权角色、发布应用、查看租户内审计 | 跨租户读取或导出数据 |
| 应用设计者 | 无代码/低代码建业务应用 | 建对象、字段、状态、规则、页面、权限 | 绕过运行时校验或直接改动态表 |
| 业务用户 | 使用已发布应用处理业务 | 列表、查询、新增、编辑、流转、附件、导出 | 访问无权限字段或他人数据 |
| 插件开发者 | 扩展平台能力 | 注册字段类型、动作、校验器、页面块、通知通道 | 直接访问动态表、关闭租户隔离 |
| 运维/审计人员 | 稳定运行、排障、合规审计 | 查看指标、日志、trace、运行 Runbook、恢复演练 | 删除或篡改自身越权操作审计 |

## 3. 范围与里程碑

### 3.1 MVP / M0：元模型内核

必须交付：

- 租户、工作区、应用、业务对象、字段、关系、版本等元模型。
- 22 种字段类型的元数据定义。
- 元数据表、动态业务表、Schema Sync、MetaGraph。
- “客户-订单”样例对象可建模、发布并生成动态表。

验收标准：

- 新建对象能生成物理表。
- 新增字段能生成物理列。
- 重复发布幂等。
- 危险变更被阻断并产生可读影响说明。

### 3.2 M1：运行时引擎

必须交付：

- `/api/data/*` 动态 CRUD、meta、action 接口。
- 类型转换、参数校验、动态 SQL 白名单。
- 租户隔离、对象权限、字段权限、数据范围。
- 状态机、动作、before/after 规则、审计日志。

验收标准：

- 双租户数据隔离测试通过。
- 字段权限在 `/meta/list/get/update` 四处一致。
- 恶意 filters 不能短路 data_scope。
- 重复提交和并发流转不能重复执行副作用。

### 3.3 M2：设计器与运行时渲染器

必须交付：

- 对象与字段设计器。
- 默认列表、表单、详情页面。
- 页面 schema 布局覆盖。
- 权限矩阵、状态机配置界面。
- 运行时 renderer 按 `/meta` 与 page schema 渲染。

验收标准：

- “客户-订单-审批”样例应用全程零代码完成。
- 页面 schema 引用不存在字段时，草稿可提示，发布必须阻断。
- 前端显示/隐藏不影响后端权限和校验。
- 富文本、schema、附件预览 XSS 测试通过。

### 3.4 M3：商业扩展能力

必须交付或形成可执行设计：

- 工作流在途实例版本策略。
- 插件生命周期、插件隔离、插件兼容矩阵。
- DataSource/Connector 出网治理。
- 导入导出包、跨环境迁移、备份恢复演练。
- 业务对象扩展与标准单据继承策略。
- 单据转换、反写、引用追踪与单据链路查询。
- 弹性域/辅助属性、多组织业务关系、编码规则、菜单/门户/移动端/多语言、报表/打印的可执行设计。

验收标准：

- 旧工作流实例不因流程定义升级而不可执行。
- 插件禁用后菜单、路由、表达式函数、字段组件均不可用。
- Connector SSRF 测试通过。
- 导入恶意包被阻断并给出影响报告。

## 4. 核心业务需求

### 4.1 应用与对象建模

REQ-001：平台必须支持租户内创建多个应用，每个应用包含多个业务对象。

REQ-002：业务对象必须至少支持实体对象、子表对象、视图对象三类。

REQ-003：对象编码、字段编码、动作编码、状态编码创建后默认不可修改；允许修改显示名。

REQ-004：对象发布前必须完成引用完整性校验，覆盖字段、关系、状态、动作、规则、页面、权限、插件、Connector。

### 4.2 字段与关系

REQ-010：首版字段类型固定为 22 种：text、textarea、richtext、code、integer、decimal、percent、currency、date、datetime、time、select、multiselect、checkbox、link、table、multilink、autonumber、user、org、attachment、formula。

REQ-011：每种 field_type 必须定义 API 输入、存储类型、空值语义、非法值处理、前端组件和测试样例。

REQ-012：link/table/multilink 字段必须生成关系元数据；关系删除行为必须显式选择 restrict、set_null 或 cascade。

REQ-013：动态字段不得静默转换用户输入；非法输入必须拒绝。

### 4.3 页面与运行态

REQ-020：建对象后必须自动获得默认列表、表单、详情页面。

REQ-021：页面 schema 只能覆盖展示和布局，不能放宽字段权限、对象权限、状态机、校验规则。

REQ-022：运行时 renderer 的唯一授权元数据来源是服务端 `/meta`，前端不得自行推导放权。

REQ-023：动态对象数据前端不能伪装成固定 DTO；必须按运行时 schema 校验。

### 4.4 权限与多租户

REQ-030：任何业务数据查询和写入必须绑定 tenant_id，tenantId 缺失必须 fail-fast。

REQ-031：权限判定必须覆盖租户成员、应用访问、对象权限、数据范围、字段权限、动作权限。

REQ-032：字段权限在 `/meta/list/get/update/export/audit` 中语义必须一致。

REQ-033：组织调岗、部门移动、角色变更、用户禁用后，历史数据范围语义必须明确为当前组织或历史快照，不得由实现偶然决定。

REQ-034：超级管理员和运维代客操作必须使用 break-glass 流程，默认不得绕过审计、租户隔离、字段脱敏和密钥不可见边界。

### 4.5 状态、规则与副作用

REQ-040：启用状态机的对象只能通过 transition 改变状态，不能通过普通 update 直接改 `state_code`。

REQ-041：transition/action/import/publish 必须支持幂等，重复请求不得重复执行副作用。

REQ-042：规则分为纯函数类和副作用类；外部副作用必须有 outbox、重试、死信、补偿和幂等键策略。

REQ-043：规则链必须有最大执行轮数和循环检测。

### 4.6 版本、发布与回滚

REQ-050：设计态变更不直接影响运行态，必须通过发布生成版本快照。

REQ-051：发布必须有状态机、DDL 计划、执行日志、失败恢复和 Reconciler 对账。

REQ-052：回滚必须定义用户可理解的能力边界：元数据快照可回滚，已执行 DDL 和已写入新格式的数据不承诺物理回滚。

REQ-053：旧版本快照、旧页面 schema、旧导出包必须有回放测试。

### 4.7 导入导出、插件与 Connector

REQ-060：跨环境迁移唯一合法通道是导出/导入包，禁止手工 SQL 搬运动动态业务表结构。

REQ-061：导出包不得包含密钥明文、token、运行时业务数据和跨租户引用。

REQ-062：插件不得绕过租户隔离、权限判定、SQL 白名单、表达式沙箱、出网治理。

REQ-063：Connector 必须防 SSRF、DNS rebinding、凭据泄露、超时失控和噪声租户资源占用。

### 4.8 商用业务平台补充能力

REQ-070：平台必须支持“标准对象 + 客户扩展”的演进模型。标准对象、行业模板对象、客户扩展字段、扩展规则、扩展页面必须可区分来源和归属，平台升级不得直接覆盖客户扩展。

REQ-071：单据继承首版不得采用无限多层继承。M0/M1 只允许模板复制和受控扩展层；如后续支持父子单据类型继承，必须先形成 ADR，明确字段、状态、权限、规则、页面、动态表和升级合并策略。

REQ-072：平台必须支持单据转换和反写的模型预留。销售订单转出库单、出库单转收款单这类链路不能简化成复制数据；必须能表达源单、目标单、字段映射、分组策略、选单条件、反写规则、超额控制和关闭控制。

REQ-073：平台必须支持引用追踪和单据链路查询。用户必须能从下游单据追溯上游来源，也能从上游单据查看下游生成记录；删除、回滚、导入导出和权限裁剪必须保留链路语义。

REQ-074：平台必须预留弹性域/辅助属性模型。物料规格、批次维度、组织维度等组合属性不能简化为普通 JSON 大字段；必须支持维度定义、属性组合、字段映射、校验和后续报表查询。

REQ-075：平台必须预留多组织业务关系模型。组织不只是 `org_id` 字段，还包括组织分类、组织职能、业务关系、核算/库存/销售/采购等职能边界；M1 权限和数据范围设计不得阻断后续多组织扩展。

REQ-076：平台必须预留报表、打印和账表能力。报表/打印不能等同于列表导出；必须区分运行态查询、聚合口径、权限裁剪、打印模板、归档版本和审计。

REQ-077：平台必须预留菜单、门户、移动端和多语言能力。页面 schema 不等同于导航信息；菜单权限、门户入口、移动端视图和 i18n 文案必须有独立元数据位置和权限边界。

REQ-078：平台必须预留应用包、行业模板、插件市场和 License 的生命周期治理。安装、升级、禁用、卸载、迁移、兼容检查、授权校验和审计必须进入产品边界，不能靠人工复制元数据。

### 4.9 定价、打包、运营指标与竞品复核

REQ-079：平台必须定义产品打包维度。至少区分社区/标准/企业/私有化四类交付形态，并明确每类可用的应用数、对象数、运行用户数、设计用户数、插件数、Connector 数、环境数、审计保留期、工作流能力和私有化能力。

REQ-080：平台必须支持 License 能力开关。License 不能只校验有效期，还必须能表达功能包、容量额度、离线授权、宽限期、应急授权、审计要求和超限处理策略。

REQ-081：平台必须支持私有化授权生命周期。离线 License 文件必须有签名、机器或组织绑定策略、续期流程、吊销列表、应急临时授权和审计留痕；不得默认依赖在线 License 服务。

REQ-082：平台必须支持商业化计量指标采集。至少采集租户数、应用数、已发布应用数、业务对象数、运行用户数、设计用户数、工作流执行数、Connector 调用数、插件安装数、存储用量和导出包次数。

REQ-083：平台必须支持产品健康指标。至少采集模型复用率、页面生成率、脚本逃逸率、发布失败率、权限拦截数、流程失败率、Connector 失败率、平均构建周期、平均发布耗时和回滚次数。

REQ-084：运营指标只能使用脱敏、聚合、最小化数据。SaaS 模式下不得跨租户泄露业务对象名称、字段值、用户身份、密钥、附件内容和导出包内容；私有化模式下指标上报必须可关闭。

REQ-085：平台必须支持应用包版本兼容矩阵。包的安装、升级、回滚和卸载必须校验平台版本、插件版本、字段类型能力、表达式语法版本、页面 schema 版本、权限模型版本和导入导出包格式版本。

REQ-086：平台必须支持竞品证据复核门禁。凡设计依据来自知识库中标注“未验证”“源码初证”“高可信推断”的能力，在进入不可逆实现前必须补充本地实测、源码证据或 ADR 风险接受记录。

REQ-087：平台必须支持表达式/公式能力边界说明。公式字段、校验表达式、显示条件、数据范围条件、状态机条件和规则条件必须共用版本化表达式语法，并明确循环检测、依赖图、执行超时、权限泄漏防护、跨对象引用和聚合限制。

REQ-088：平台必须支持元数据迁移与版本升级策略。元模型 JSON、页面 schema、导入导出包、插件 manifest、表达式语法、权限快照和工作流定义必须有升级器、兼容等级和回放测试。

REQ-089：平台必须支持租户生命周期治理。租户创建、停用、冻结、注销、数据导出、数据清理、保留期、备份恢复、独立库迁移和共享库回迁必须有明确状态机、审计和 Runbook。

REQ-090：平台必须支持产品级成功验收。M2 结束时不仅要跑通“客户-订单-审批”，还必须给出页面生成率、脚本逃逸率、发布失败率、权限拦截数和平均构建周期的基线报告；M3 结束时必须补齐商业打包、License 和私有化验收报告。

### 4.10 竞品特性差距补全需求

REQ-091：平台必须支持业务事件中心。业务对象写入、状态流转、版本发布、导入导出、定时任务和自定义 Action 必须能发布标准事件；事件 payload 必须包含 tenantId、appId、objectCode、recordLid、diff 摘要、metaHash、traceId 和脱敏后的业务摘要。

REQ-092：平台必须支持事件订阅模型。订阅者类型至少包括 rule、workflow、webhook、plugin handler 和 notification；同步订阅可否决事务，异步订阅必须进入 outbox，不得各自实现投递机制。

REQ-093：规则动作必须支持跨对象创建和更新记录。`create_record` 与 `update_record` 必须声明目标对象、字段映射、执行时机、幂等键、最大级联深度和循环检测，所有写入必须走统一写管线。

REQ-094：规则必须支持定时触发。schedule 规则必须声明 cron/周期、租户时区、目标对象 filters、错过补跑策略、并发策略和租户配额；执行必须走统一调度器。

REQ-095：平台必须支持 rollup 汇总字段。rollup 必须声明源子表或关联对象、聚合函数、过滤条件、物化策略、重算时机和权限边界；不得用普通 formula 或脚本伪装汇总字段。

REQ-096：平台必须支持复合唯一约束。BusinessObject 必须能声明字段组合唯一，Schema Sync 生成唯一索引时必须与 `delete_token` 软删除语义兼容，违反约束时错误必须定位到约束名和字段组合。

REQ-097：平台必须支持全局选项集。select/multiselect 可引用应用级 OptionSet；选项支持启用、停用、排序、多语言显示名和只增不复用 code，内联 options 仍允许但不得替代共享字典。

REQ-098：平台必须支持保存的筛选视图。列表页用户可以保存 filters、columns、sort、group、density 等视图配置，并区分个人视图、角色共享视图和应用默认视图。

REQ-099：平台必须支持更多视图类型。看板分组字段不得只绑定状态字段，应支持 select/user/link 等受控字段；日历视图必须以 date/datetime 字段为时间轴，并受字段权限和数据范围裁剪。

REQ-100：平台必须支持轻量聚合视图和基础图表。首版只做分组聚合、柱状图、折线图、饼图和指标卡，不做通用 BI；聚合查询必须受 AccessView 字段权限、数据范围和导出安全约束。

REQ-101：平台必须支持全局搜索。首版按对象 title_field、autonumber 和配置的 search_fields 做跨对象检索，结果必须按 AccessView 裁剪；全文检索和复杂相关性排序放入 M4。

REQ-102：平台必须支持记录协作能力。评论、@、关注、分配和记录时间线必须作为平台内置能力，不占用用户业务对象配额；评论内容要防 XSS，@ 通知必须走事件订阅和消息中心。

REQ-103：平台必须支持 IdentityProvider SPI。认证体系应预留 LDAP、AD、OAuth2/OIDC、SAML、钉钉、企业微信等身份源接入；组织同步必须定义冲突策略，并保留历史组织快照语义。

REQ-104：平台必须支持开放 API Token。API token 必须有作用域、有效期、密钥轮换、限流、审计、撤销和最后使用时间；不得复用浏览器 cookie 会话，不得绕过 AccessView。

REQ-105：字段权限必须支持 masked 档。字段可见性不只区分 READ/NONE，还要支持脱敏半可见，例如手机号、证件号、邮箱；导出、打印、通知、审计和 API Token 访问必须一致执行脱敏策略。

REQ-106：平台必须支持回收站。软删除记录可在应用级回收站查看、恢复、彻底删除或按保留期清理；恢复和彻底删除必须校验权限、引用和审计。

REQ-107：平台必须支持站内消息中心。notify 默认通道为站内消息，支持已读/未读、订阅设置、来源记录跳转、批量已读和保留期；邮件、企微、钉钉等外部通道作为 M3 扩展。

REQ-108：平台必须支持应用模板库。官方模板至少覆盖 CRM、请假、采购三类样例；模板安装必须复用应用包导入、安全扫描、版本兼容和 License 规则。

REQ-109：业务数据 Excel 导入导出必须作为产品能力提供。导入向导包含模板下载、列映射、试跑校验、错误报告、部分成功策略和幂等重导；导出必须按 AccessView 字段裁剪、防 CSV 公式注入并写审计。

REQ-110：单据下推必须作为 ConvertRule 能力设计。ConvertRule 必须表达源对象、目标对象、字段映射、子表映射、来源引用、下推动作、权限裁剪和幂等；M3 首版只承诺 1:1 单级下推，1:N 与合并下推进入 M4。

REQ-111：打印模板必须作为正式产品能力设计。打印模板必须支持对象绑定、模板版本、字段占位、子表表格、页眉页脚、权限裁剪、脱敏、归档和审计；不得把打印实现为页面截图。

REQ-112：M4 必须承接商用后增强边界。移动端/H5、多环境 dev/test/prod 与灰度、国际化完整落地、跨应用共享主数据、自由脚本沙箱和丰富 Connector 库均为 M4 候选，不得让用户误以为 M1~M3 默认包含。

REQ-113：首版移动端只做结构预留。M1~M3 不承诺完整移动端/H5，但页面 schema、组件 schema 和布局能力不得绑定固定像素或桌面专用组件，必须能被 M4 响应式改造复用。

REQ-114：首版多环境能力采用独立部署实例 + 导入导出包迁移。租户内 dev/test/prod、多环境灰度、一键提升版本进入 M4，不得在 M1~M3 以隐式环境字段替代正式设计。

REQ-115：首版跨应用对象引用不开放。对象默认只能应用内引用；跨应用主数据共享进入 M4，并必须先补权限、版本、导入导出和 lc_meta_ref 跨 app 引用 ADR。

REQ-116：首版不做自由脚本沙箱。表达力阶梯为表达式、规则动作、ConvertRule、插件 SPI；只有这些机制不能覆盖且经安全 ADR 通过后，才允许讨论自由脚本。

REQ-117：首版国际化只做结构预留。M1~M3 以中文界面交付为准，字段显示名等元数据可保存多语言 JSON，前端界面文案必须从 i18n 资源目录读取并允许后续补语言包；完整多语言翻译、运行时语言切换、导入导出语言包和多语言审批/打印模板进入 M4。

REQ-118：首版 Connector 库只承诺 SPI 与少量内置连接器。M1~M3 只允许实现 mysql/http 等经 SSRF、凭据、限流、审计治理覆盖的连接器；50+ 数据源生态、连接器市场、三方认证模板和批量连接器适配进入 M4，不得把单个连接器实现写成绕过 Connector SPI 的特例。

## 5. 非功能需求

| 类别 | 需求 |
|---|---|
| 隔离性 | SaaS shared 模式下任一租户的大查询、大导出、慢 Connector、规则循环不得拖垮其他租户核心 CRUD |
| 性能 | 应用级 MetaGraph 冷加载目标 < 2s；万级数据 list P95 目标 < 500ms；发布后多实例 MetaGraph 版本收敛窗口 ≤ 60 秒 |
| 可用性 | 发布失败必须进入可恢复状态，不得留下无状态记录的半应用变更 |
| 安全 | 动态 SQL、表达式、附件、Connector、导入包、前端渲染器必须有安全测试集 |
| 合规 | 个人信息必须分级、脱敏、最小化、保留期清理，租户注销/用户删除的边界必须可说明 |
| 兼容 | 公开 API、元模型 JSON、页面 schema、插件 SPI、导入导出包必须定义兼容等级 |
| 可观测 | REST、异步任务、规则、DDL、插件、前端错误必须能用 traceId 串联 |
| 可运维 | 必须提供 Reconciler、只读模式、插件禁用、outbox 重放、备份恢复 Runbook |
| 商业化 | License、功能包、容量额度、离线授权、超限处理、指标采集和私有化交付必须可审计 |
| 竞品复核 | 未经实测或仍为推断的竞品结论不得直接变成不可逆实现，必须通过 ADR 或测试证据关闭风险 |

## 6. 贯穿验收样例

样例应用：客户-订单-审批。

必须覆盖：

1. 客户对象：客户名称、等级 select、联系人、附件。
2. 订单对象：订单号 autonumber、客户 link、订单明细 table、金额 currency、审批意见 richtext。
3. 状态机：草稿 -> 已提交 -> 已审批 / 已驳回。
4. 规则：金额大于 10000 必填审批意见；选择客户后带出客户等级。
5. 权限：销售只能看自己的订单；文员可看订单但金额只读；经理可审批。
6. 页面：默认列表/表单/详情，表单可分组布局。
7. 发布：v1 发布后修改字段不影响运行态；v2 发布后生效；回滚说明清楚不可物理回滚项。
8. 安全：双租户隔离、字段权限裁剪、OR 短路攻击、XSS、附件越权下载、重复审批、并发审批全部通过。

## 7. 明确暂不做

- 首版不做自由 JS 业务脚本。
- 首版不做多数据库运行时适配；MySQL 8.0 是唯一数据库。
- 首版不做完整 BI 报表和复杂数据仓库建模。
- 首版不承诺工作流所有高级节点；M3 前只形成版本冻结和迁移策略。
- 首版不支持租户自行上传未审查后端 jar 插件。
- M1~M3 不承诺完整移动端/H5；只做 schema 与组件能力预留，M4 再做响应式运行态。
- M1~M3 不承诺租户内多环境和灰度发布；首版环境隔离通过独立部署实例和导入导出包完成。
- M1~M3 不开放跨应用业务对象引用；跨应用主数据共享进入 M4 并要求 ADR。
- M1~M3 不承诺完整国际化落地；首版中文交付，只保留元数据多语言 JSON、前端 i18n 资源目录和后续语言包入口。
- M1~M3 不承诺 50+ Connector 库；首版只定义 Connector SPI 与 mysql/http 等少量连接器。
- 不做通用 BI；只提供轻量聚合视图和基础图表，复杂指标建模进入后续商业版本。

## 9. 需求变更规则

1. 新增需求必须标注影响的角色、里程碑、验收样例和涉及的 07 陷阱。
2. 改变兼容边界、安全边界、元模型对象、数据语义、部署模式的需求，必须形成 ADR。
3. 实现中发现需求缺口时，先补本文，再补架构/细则/测试；不得只在任务卡或聊天记录中说明。
