# 人工评审清单

用于承接 `15/18/22/24/25/27/28` 中无法完全机器化的条款。发布前必须逐项过一遍，不能只看 CI 绿灯。

## 清单

### 规则编号：15-060 / 18-041

- 为什么不能机器化：是否“新增了未证明抽象”需要结合上下文和需求目标判断。
- 评审问题：本次改动是否新增了只为“显得完整”而存在的脚本、目录层级、抽象包装或长期无人维护的占位物？
- 通过标准：每个新增文件都能对应到明确上线门禁、runbook 或合规要求。
- 失败示例：为了记录一个命令单独造一层工具脚手架，最终没人使用。
- 对应陷阱：H4 AI 通用抽象膨胀。

### 规则编号：18-010 / 18-050

- 为什么不能机器化：回滚方案是否真实可执行，必须由人审查上下游依赖和环境差异。
- 评审问题：发布说明里是否写清“元数据可回、物理 DDL 不自动回、数据默认保留”的边界？
- 通过标准：回滚粒度、不可回滚项、人工确认点都已写清。
- 失败示例：只写“如有问题回滚版本”。
- 对应陷阱：A2 版本回滚回滚不了 DDL。

### 规则编号：22-030 / 22-031

- 为什么不能机器化：字段语义收窄、删列、插件迁移等单向门是否可回，需要业务和 DBA 联合判断。
- 评审问题：本次发布是否跨越了任何不可回滚的字段语义、插件数据迁移或 License 策略变化？
- 通过标准：人工确认结果已记录，必要时阻断发布。
- 失败示例：明知字段类型收窄仍强行放行。
- 对应陷阱：A1、A2、F12。

### 规则编号：24-021 / 27-040

- 为什么不能机器化：灰度与回滚条件是否足够清楚，取决于业务承受能力和运维值守方式。
- 评审问题：每个灰度或发布动作是否绑定了观测指标、回滚阈值和升级联系人？
- 通过标准：能明确回答“看到什么指标变化时回滚”。
- 失败示例：只有“观察一段时间”这种口头描述。
- 对应陷阱：A6、C3、C5。

### 规则编号：25-040 / 25-043

- 为什么不能机器化：对外错误口径是否足够安全且不影响排障，需要结合产品文案和日志实践判断。
- 评审问题：对外报错是否只暴露安全分类和 traceId，内部又是否有足够诊断信息？
- 通过标准：用户可处理信息明确，内部排障信息不丢。
- 失败示例：把 SQL、堆栈、物理表名直接展示给外部调用方。
- 对应陷阱：G7、H2。

### 规则编号：27-090

- 为什么不能机器化：runbook 是否真能被值班人拿来执行，需要人工通读。
- 评审问题：每份 runbook 是否都包含症状、止血动作、恢复步骤、回滚、验证和联系人？
- 通过标准：值班人不看源码也能完成首轮止血。
- 失败示例：只有概念介绍，没有动作步骤。
- 对应陷阱：A1、A5、F6、F10、F11。

### 规则编号：22-010 / 22-015 / 27-034

- 为什么不能机器化：发布计划、锁风险和回滚阈值是否足够清楚，必须结合本次变更上下文判断。
- 评审问题：`docs/review/release-checklist.md` 是否写清本次 DDL / 元数据发布范围、观测指标和 stop-loss 条件？
- 通过标准：能明确回答“发布后看什么、看到什么时停止、谁来拍板回滚”。
- 失败示例：只写“发布后观察”或“如异常回滚”。
- 对应陷阱：A1、A5、A6。

### 规则编号：27-020 / 27-021 / 27-039

- 为什么不能机器化：观测项是否足够覆盖本次发布影响面，依赖人对变更和运行链路的判断。
- 评审问题：是否已指定核心指标、错误告警、traceId 检索入口和观察时窗？
- 通过标准：release checklist、发布 runbook 和回滚 runbook 的观测口径一致。
- 失败示例：只列指标名，不写阈值、负责人和观察窗口。
- 对应陷阱：C3、G7、H3。

### 规则编号：28-001 / 28-060

- 为什么不能机器化：依赖“是否值得引入”不是纯 License 判断，还涉及替代方案、退出成本和权限边界。
- 评审问题：新增依赖是否真的不可避免，是否已有更轻方案？
- 通过标准：依赖准入表填写完整，能说明替代方案为何不选。
- 失败示例：只因为一个命令方便就引入大型工具链。
- 对应陷阱：E2、F12。

### 规则编号：28-113 / 28-114

- 为什么不能机器化：离线授权、降级只读和数据保留边界需要产品、交付和法务共识。
- 评审问题：私有化环境断网时，License 行为是否仍符合“可读、可审计、不删数据”底线？
- 通过标准：README、合规文档和插件 runbook 三处口径一致。
- 失败示例：License 过期直接清业务数据。
- 对应陷阱：F12、E4。

### 规则编号：28-010 / 28-090

- 为什么不能机器化：依赖变更是否需要重新做准入、License 复核或 SBOM 更新，必须结合发布内容判断。
- 评审问题：本次发布是否已明确“无依赖差异”或给出 `Dependency / License / SBOM` 结论？
- 通过标准：PR 模板、release checklist 和合规文档三处口径一致。
- 失败示例：依赖已变更，但仍沿用旧的清单结论。
- 对应陷阱：E2、F12。
