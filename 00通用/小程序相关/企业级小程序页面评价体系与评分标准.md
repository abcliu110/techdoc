# 企业级小程序页面评价体系与评分标准

## 1. 文档目的

本文用于评价一个微信小程序页面是否达到企业级标准。

本文不是单纯的视觉审美清单，也不是只看页面是否“能用”。企业级页面评价必须同时判断：

```text
用户能不能完成任务。
页面信息是否清楚。
业务对象是否明确。
状态是否可理解。
操作是否安全。
异常是否可恢复。
测试是否可自动化。
后端结果是否可证明。
上线是否可治理。
```

适用场景：

- 小程序页面设计评审。
- 开发提测前自检。
- 测试准入判断。
- UI 自动化可测性评估。
- 交易、支付、退款、会员、订单、核销类高风险页面评审。
- 老页面升级企业级页面契约时的差距评估。

## 2. 评价体系来源

本评价体系来自五类约束，而不是个人审美偏好。

### 2.1 业务风险

页面风险越高，评价标准越严格。

例如结账单、支付、退款、反结账、修改支付方式等页面，涉及财务和审计，必须优先评价：

```text
当前操作对象是否明确。
当前业务状态是否明确。
这个状态下能否执行当前操作。
操作结果是否有后端证据。
操作是否可审计、可回滚、可追踪。
```

交易类页面不能只评价“页面好不好看”。

### 2.2 信息架构

企业级工作台页面必须帮助用户完成任务，而不是堆信息。

通用任务链路：

```text
定位对象
-> 理解状态
-> 查看细节
-> 执行动作
-> 得到反馈
-> 验证结果
```

例如结账单页面：

```text
筛选账单
-> 选择账单
-> 确认当前账单
-> 查看小票和支付明细
-> 执行重打、退款、反结账等动作
-> 查看操作结果
```

### 2.3 状态机和页面契约

企业级页面必须具备状态机和页面契约。

评价时必须检查：

```text
页面有哪些状态。
每个状态显示什么。
状态如何流转。
点击后状态是否可观察。
测试是否能读取 data-state。
错误状态是否可恢复。
```

状态标签不是装饰，而是页面状态机的用户可见表达。

### 2.4 企业级工作台设计原则

POS、收银、交易后台、运营后台类页面通常是高密度工作台。

这类页面设计重点是：

```text
高密度但不混乱。
当前对象必须明确。
金额和状态必须突出。
危险操作必须可识别。
列表和详情必须强绑定。
操作必须可审计。
```

因此不应套用营销页、内容页、普通商城首页的评价标准。

### 2.5 自动化测试和证据链

如果页面需要快速测试，就必须评价：

```text
页面状态能不能等。
业务对象能不能读。
操作对象能不能确认。
接口请求能不能追踪。
后端结果能不能查询。
失败原因能不能归类。
```

页面显示成功只是 UI 证据，后端 evidence 才能证明业务结果真实发生。

## 3. 总体评价模型

企业级页面评价采用 100 分制。

| 维度 | 权重 | 核心问题 |
|---|---:|---|
| 任务完成效率 | 15 | 用户能否快速完成主要任务 |
| 信息架构清晰度 | 15 | 用户是否知道当前看什么、操作什么 |
| 状态完整性 | 15 | 页面状态是否完整、可理解、可等待 |
| 操作安全性 | 15 | 高风险操作是否可控、防误触、可恢复 |
| 业务上下文与证据链 | 15 | 页面能否关联业务对象、接口、后端证据 |
| 自动化可测性 | 10 | 测试能否稳定定位、等待、操作、验证 |
| 视觉专业度 | 5 | 信息层级、对齐、密度、状态标签是否专业 |
| 性能与平台适配 | 5 | 首屏、滚动、分包、基础库、真机表现是否可靠 |
| 安全隐私 | 5 | 是否泄漏敏感信息，权限边界是否清楚 |

## 4. 评级标准

| 等级 | 分数 | 结论 |
|---|---:|---|
| A | 90-100 | 企业级优秀页面，可作为标准页面 |
| B | 80-89 | 可上线，主要链路清楚，但有优化项 |
| C | 70-79 | 基本可用，但测试、状态或风险控制不足 |
| D | 60-69 | 不建议发布，高风险缺口明显 |
| F | 60 以下 | 不合格，业务、测试或安全风险不可接受 |

### 4.1 A 级页面特征

```text
页面身份明确。
入口完整可测。
参数契约清楚。
状态机完整。
主业务对象明确。
操作权限和状态联动。
危险操作有确认和审计。
页面状态可被自动化等待。
后端 evidence 可查询。
错误可恢复。
视觉层级清楚。
正式包不泄漏敏感信息。
```

### 4.2 C 级页面常见表现

```text
人工能操作，但页面契约不完整。
按钮可以点，但点击后状态不可观察。
列表和详情有关联，但测试无法读取业务对象。
错误只有 toast，没有页面状态。
测试依赖固定 sleep。
后端结果无法自动查询。
危险操作没有足够的状态限制。
```

## 5. 评价维度详解

### 5.1 任务完成效率

检查问题：

```text
页面的主任务是什么？
用户是否一眼知道下一步？
主要操作是否突出？
次要信息是否干扰主任务？
完成任务需要多少步？
```

高分标准：

```text
主任务明确。
主操作清楚。
关键字段完整。
用户路径短。
用户不需要猜下一步。
```

低分信号：

```text
按钮很多但主次不分。
页面信息很多但不知道下一步。
用户必须反复对照多个区域才能确认操作对象。
```

### 5.2 信息架构清晰度

检查问题：

```text
页面是否按用户任务组织信息？
是否有清楚的信息分组？
当前对象是否突出？
列表、详情、操作区关系是否明确？
合计、统计、金额是否说明口径？
```

高分标准：

```text
筛选区、列表区、当前对象摘要、详情区、操作区边界明确。
当前选中对象清楚。
金额和状态重点突出。
统计口径有说明。
```

低分信号：

```text
当前选中对象只能靠高亮猜。
详情区和列表区绑定不明显。
合计金额不知道是当前页还是全部筛选结果。
父子记录混在一起。
```

### 5.3 状态完整性

检查问题：

```text
页面有哪些状态？
loading、empty、error、ready、submitting 是否完整？
状态标签是否有业务语义？
状态之间如何流转？
状态能否被测试读取？
```

高分标准：

```text
页面有完整状态机。
每个状态有明确 UI。
状态标签语义清晰。
状态能通过 data-state 暴露。
中间状态不会长期卡住。
```

低分信号：

```text
使用“-”表示未知或特殊状态。
所有错误都显示系统异常。
提交中没有可观察状态。
测试只能 sleep。
```

### 5.4 操作安全性

检查问题：

```text
哪些操作是高风险？
按钮是否按状态启用或禁用？
禁用原因是否明确？
危险操作是否二次确认？
操作是否幂等？
失败后能否恢复？
```

高风险操作包括：

```text
退款
反结账
修改支付方式
删除
核销
支付
取消订单
```

高分标准：

```text
按钮权限和账单状态联动。
危险操作视觉上可识别。
禁用按钮有 disabledReason。
操作中防重复点击。
操作失败有可恢复路径。
```

低分信号：

```text
所有按钮永远可点击。
退款、反结账和普通按钮视觉相同。
提交中仍能连点。
失败后页面状态不明确。
```

### 5.5 业务上下文与证据链

检查问题：

```text
页面当前处理哪个业务对象？
主业务对象和关联对象是否明确？
是否暴露外部业务编号？
是否有 traceId / requestId？
是否能查询后端 evidence？
```

高分标准：

```text
页面根节点暴露 data-biz-type / data-biz-no。
列表项暴露自己的业务编号。
详情区和选中行使用同一业务编号。
按钮暴露 data-action 和操作对象。
后端 evidence 可按业务编号查询。
```

低分信号：

```text
页面显示成功但不知道是哪张单。
小票预览和列表行只能靠人工对照。
按钮操作对象不明确。
后端结果无法自动验证。
```

### 5.6 自动化可测性

检查问题：

```text
页面根节点是否有 data-qa？
关键元素是否有 data-qa？
状态是否可等待？
业务编号是否可读取？
点击后状态是否可观察？
是否依赖固定 sleep？
```

高分标准：

```text
测试能进入页面。
测试能等待 ready。
测试能点击 data-action。
测试能等待 submitting / success / error。
测试能读取 data-biz-no。
测试能查询后端 evidence。
```

低分信号：

```text
靠文案、坐标、截图定位。
靠固定 sleep 等接口。
不能判断当前选中对象。
不能判断点击后是否触发。
```

### 5.7 视觉专业度

检查问题：

```text
信息层级是否清楚？
表格密度是否合理？
状态颜色是否有语义？
危险操作是否与普通操作区分？
按钮是否过多且无分组？
```

高分标准：

```text
高密度但不拥挤。
对齐清晰。
状态标签语义和颜色一致。
操作按钮按风险分组。
文字不遮挡、不溢出。
```

低分信号：

```text
所有按钮同色。
状态标签颜色随意。
表格横向滚动但重点列不固定。
金额、状态、对象不突出。
```

### 5.8 性能与平台适配

检查问题：

```text
首屏是否快速可见？
表格滚动是否顺畅？
筛选是否会触发过多请求？
长列表是否分页？
页面隐藏后是否停止轮询？
是否适配小程序基础库和真机？
```

### 5.9 安全隐私

检查问题：

```text
data-* 是否暴露敏感信息？
日志是否脱敏？
操作权限是否只靠前端判断？
后端是否再次校验权限？
正式包是否剥离测试桥？
```

## 6. 交易后处理页面专项标准

适用页面：

```text
结账单
订单详情
支付结果
退款
反结账
修改支付方式
发票
核销
```

### 6.1 必须突出当前操作对象

页面必须显示当前选中对象摘要：

```text
当前账单：2026070310035
状态：部分退款
台号：16
实收：81.40
结账时间：2026-07-03 11:44:30
```

目的：

```text
防止误退款。
防止误反结账。
防止测试无法确认操作对象。
```

### 6.2 必须区分原单和子单

例如退款记录不能和原结账单完全平级展示。

推荐：

```text
2026070310035      部分退款      实收 81.4
  ↳ 2026070310035_1 退款记录      -47.3
```

### 6.3 必须定义状态标签

推荐主状态：

| 主状态 | 含义 |
|---|---|
| 已结账 | 正常完成 |
| 部分退款 | 已发生部分退款 |
| 已全退 | 全部金额已退 |
| 已反结账 | 结账已撤销 |
| 退款记录 | 当前行是退款子记录 |
| 异常 | 金额、支付或状态不一致 |
| 未知 | 状态无法识别，需要排查 |

禁止用 `-` 作为业务状态。

### 6.4 状态标签必须和按钮权限联动

示例：

| 状态 | 重打 | 反结账 | 备注 | 开发票 | 改支付 | 退款 |
|---|---|---|---|---|---|---|
| 已结账 | 可 | 可 | 可 | 可 | 可 | 可 |
| 部分退款 | 可 | 禁用 | 可 | 受限 | 受限 | 可 |
| 已全退 | 可 | 禁用 | 可 | 禁用 | 禁用 | 禁用 |
| 已反结账 | 可 | 禁用 | 可 | 禁用 | 禁用 | 禁用 |
| 退款记录 | 可 | 禁用 | 可 | 禁用 | 禁用 | 禁用 |
| 异常 | 可 | 禁用 | 可 | 禁用 | 禁用 | 禁用 |

禁用必须有原因：

```text
已部分退款，不能反结账。
退款记录不能再次退款。
已全退账单不能开发票。
```

### 6.5 合计口径必须明确

金额合计必须说明口径：

```text
当前页合计
当前筛选结果合计
全部账单合计
当前选中账单金额
```

不能只显示一个孤立金额。

## 7. 结账单页面示例评价

基于典型结账单页面，评价时应重点检查：

```text
当前选中账单是否明确。
小票预览是否和选中行强绑定。
原单和退款子单关系是否清楚。
状态标签是否有业务语义。
底部按钮是否按风险分组。
按钮是否根据账单状态启用/禁用。
合计金额口径是否明确。
测试能否读取 selectedBillNo。
测试能否读取 receiptBillNo。
测试能否按 billNo 查询后端 evidence。
```

推荐信息架构：

```text
顶部筛选区
  日期 / 流水号 / 食品卡号 / 台名 / 金额 / 更多筛选

结果摘要区
  共 53 条 | 筛选总实收 ¥3713.70 | 当前页 20 条

主体区
  左侧：小票预览
  右侧：
    当前选中账单摘要
    账单表格

底部操作区
  常规操作：重打 / 备注 / 开发票
  交易变更：修改支付方式 / 退款 / 反结账
  页面操作：刷新 / 退出
```

## 9. 最终结论

企业级小程序页面评价的核心不是“好不好看”，而是：

```text
对象明确。
状态明确。
动作安全。
证据可查。
测试可等。
异常可恢复。
上线可治理。
```

对于交易、支付、退款、结账、核销类页面，评价优先级必须是：

```text
业务安全 > 状态清晰 > 证据链 > 自动化可测 > 视觉美观
```
