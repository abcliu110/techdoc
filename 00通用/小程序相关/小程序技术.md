# 小程序技术

本文整理微信小程序开发的核心知识：它为什么这样设计、通用项目结构、页面路径规则、tabBar 与分包原理，以及在本地 `taro-mall` 项目中的对应关系。

## 1. 小程序解决什么问题

微信小程序的目标不是提供一个完全自由的 Web 环境，而是在微信里提供一种受控应用形态：

- 不需要安装，扫码、搜索、分享即可打开。
- 比普通 H5 更接近 App，可使用微信登录、支付、定位、扫码、相机等能力。
- 比原生 App 开发成本低，统一适配 iOS 和 Android。
- 运行环境受微信管控，便于审核、安全、权限、性能和包体治理。

可以理解为：

```text
Vue-like 的数据驱动开发体验
+
微信 App 内的受控运行环境
+
微信统一提供的登录、支付、权限、审核和分发能力
```

## 2. 小程序设计原理

小程序采用“配置先行、逻辑和视图分离、数据驱动视图”的模型。

```text
app.json / page.json 先声明页面和能力
          |
微信客户端创建页面
          |
page.js 处理数据、事件、接口
          |
setData / 框架更新
          |
wxml + wxss 渲染界面
```

这解释了几个关键约束：

- 页面必须先在配置中声明，微信需要提前知道页面地图。
- 原生小程序不能随便操作 DOM，逻辑层和视图层隔离。
- 修改页面数据要通过 `setData()`，否则视图层不知道数据变化。
- 系统能力必须通过 `wx.*` API 调用，例如 `wx.login`、`wx.request`、`wx.requestPayment`。
- 网络域名、隐私权限、类目和页面内容受微信平台审核约束。

## 3. 通用项目结构

一个原生小程序通常由这些文件组成：

```text
app.js
app.json
app.wxss
pages/
  index/
    index.js
    index.json
    index.wxml
    index.wxss
```

| 文件 | 作用 | 设计原因 |
|---|---|---|
| `app.js` | 小程序全局入口，注册 `App`，处理启动、前后台、全局状态 | 微信需要统一生命周期入口 |
| `app.json` | 声明页面列表、窗口样式、tabBar、权限等 | 微信必须提前知道页面地图，才能路由、预加载、审核、分包 |
| `app.wxss` | 全局样式 | 避免每个页面重复写通用样式 |
| `pages/index/index.js` | 页面逻辑，注册 `Page`，处理数据、事件、接口 | 逻辑层独立运行，不直接操作 DOM |
| `pages/index/index.wxml` | 页面结构模板 | 视图层用受控模板描述，不开放完整 HTML |
| `pages/index/index.wxss` | 页面样式 | 页面局部样式隔离 |
| `pages/index/index.json` | 页面局部配置，如标题、组件引用 | 页面可以覆盖全局配置 |

## 4. 最小页面示例

### app.js

```js
App({
  globalData: {
    userInfo: null,
    token: ''
  },

  onLaunch() {
    console.log('小程序启动')
  },

  onShow() {
    console.log('小程序进入前台')
  },

  onHide() {
    console.log('小程序进入后台')
  }
})
```

### app.json

```json
{
  "pages": [
    "pages/index/index"
  ],
  "window": {
    "navigationBarTitleText": "点餐宝",
    "navigationBarBackgroundColor": "#ffffff",
    "navigationBarTextStyle": "black",
    "backgroundColor": "#f5f5f5"
  }
}
```

### app.wxss

```css
page {
  background: #f5f5f5;
  font-size: 28rpx;
  color: #333;
}

.primary-btn {
  background: #07c160;
  color: white;
}
```

### pages/index/index.js

```js
Page({
  data: {
    title: '欢迎使用点餐宝',
    count: 0
  },

  onLoad() {
    console.log('首页加载')
  },

  addCount() {
    this.setData({
      count: this.data.count + 1
    })
  }
})
```

### pages/index/index.wxml

```xml
<view class="container">
  <text class="title">{{title}}</text>

  <view class="counter">
    当前数量：{{count}}
  </view>

  <button class="primary-btn" bindtap="addCount">
    +1
  </button>
</view>
```

### pages/index/index.wxss

```css
.container {
  padding: 40rpx;
}

.title {
  display: block;
  font-size: 36rpx;
  font-weight: bold;
  margin-bottom: 32rpx;
}

.counter {
  margin-bottom: 24rpx;
}
```

### pages/index/index.json

```json
{
  "navigationBarTitleText": "首页",
  "enablePullDownRefresh": true
}
```

运行过程：

```text
微信读取 app.json
  ↓
知道首页是 pages/index/index
  ↓
加载 app.js，触发 onLaunch
  ↓
进入首页，加载 index.json / index.js / index.wxml / index.wxss
  ↓
index.js 提供 data
  ↓
index.wxml 使用 {{count}} 显示数据
  ↓
点击按钮触发 addCount
  ↓
addCount 调用 setData
  ↓
视图更新
```

## 5. 和 Vue 的相似与区别

小程序和 Vue 表面相似，是因为二者都采用数据驱动视图：

```text
数据 data
  ↓
模板 template / wxml
  ↓
事件 click / bindtap
  ↓
修改数据
  ↓
界面更新
```

区别在运行环境：

```text
Vue
  ↓
浏览器 JS
  ↓
DOM / CSSOM
  ↓
页面渲染

小程序
  ↓
小程序 JS 逻辑层
  ↓
setData / 事件桥
  ↓
微信客户端
  ↓
小程序视图层
```

关键区别：

- Vue 是网页框架，跑在浏览器里。
- 小程序是微信框架，跑在微信受控运行时里。
- Vue 可以直接使用部分浏览器能力，小程序需要通过 `wx.*` API。
- Vue 的路由主要由代码管理，小程序页面必须提前在配置里声明。
- Vue 更自由，小程序受包体、权限、审核、域名、隐私合规限制。

## 6. 页面路径规则

页面路径规则可以记成一句话：

```text
页面路径必须先在配置里注册，跳转时用这个注册路径，不写文件后缀。
```

如果文件是：

```text
pages/index/index.js
pages/index/index.wxml
pages/index/index.wxss
pages/index/index.json
```

页面路径写：

```json
"pages/index/index"
```

不要写：

```text
/pages/index/index
pages/index/index.js
pages/index/
```

`app.json` 示例：

```json
{
  "pages": [
    "pages/index/index",
    "pages/order/order",
    "pages/user/user"
  ]
}
```

规则：

- `app.json` 里的页面路径不写 `/` 开头。
- 不写 `.js`、`.wxml`、`.json`、`.wxss` 后缀。
- 每个页面都要注册。
- 默认情况下，第一个页面是首页。
- 路径大小写要和真实文件一致。

跳转时建议使用从根目录开始的绝对路径：

```js
wx.navigateTo({
  url: '/pages/order/order?id=1001&type=takeout'
})
```

目标页面接收参数：

```js
Page({
  onLoad(options) {
    console.log(options.id)   // "1001"
    console.log(options.type) // "takeout"
  }
})
```

复杂对象需要先序列化和编码：

```js
const data = encodeURIComponent(JSON.stringify({ id: 1, name: '张三' }))

wx.navigateTo({
  url: `/pages/detail/detail?data=${data}`
})
```

目标页面：

```js
onLoad(options) {
  const data = JSON.parse(decodeURIComponent(options.data))
}
```

## 7. 路由 API 规则

| API | 能跳 tabBar 页面 | 能带参数 | 用途 |
|---|---|---|---|
| `wx.navigateTo` | 不能 | 可以 | 打开普通页面，保留当前页 |
| `wx.redirectTo` | 不能 | 可以 | 替换当前普通页面 |
| `wx.switchTab` | 只能跳 tabBar | 一般不能 | 切换底部 tab |
| `wx.reLaunch` | 可以 | 普通页可带参数 | 关闭所有页面重新打开 |
| `wx.navigateBack` | 不写路径 | 不适用 | 返回上一页 |

常见坑：

- `navigateTo` 不能跳 tabBar 页面。
- `switchTab` 不能直接传 query 参数。
- tabBar 页面要传数据，通常用全局状态或本地缓存。
- 新增页面后必须注册到 `app.json` 或分包配置。
- 不能运行时新增一个没有注册过的页面路径。
- `navigateTo` 会压页面栈，深层流程要用 `redirectTo` 或 `reLaunch` 控制页面栈。

## 8. tabBar 技术原理

`tabBar` 不是普通页面组件，而是微信客户端托管的原生底部导航。

配置示例：

```json
{
  "pages": [
    "pages/home/index",
    "pages/order/index",
    "pages/user/index"
  ],
  "tabBar": {
    "list": [
      {
        "pagePath": "pages/home/index",
        "text": "首页",
        "iconPath": "assets/home.png",
        "selectedIconPath": "assets/home-active.png"
      },
      {
        "pagePath": "pages/order/index",
        "text": "订单",
        "iconPath": "assets/order.png",
        "selectedIconPath": "assets/order-active.png"
      }
    ]
  }
}
```

运行机制：

```text
微信启动小程序
  ↓
读取 app.json
  ↓
发现 tabBar 配置
  ↓
微信客户端创建底部原生导航栏
  ↓
点击 tab
  ↓
微信切换到对应 pagePath 页面
```

设计目的：

- 稳定一级导航。
- 让微信提前知道核心页面结构。
- 便于页面栈管理、审核、预加载和原生导航体验。
- 避免一级入口完全由 JS 动态生成导致不可控。

规则：

- `tabBar.list` 通常配置 2 到 5 个 tab。
- 每个 `pagePath` 必须先出现在 `pages` 中。
- tabBar 页面通常放在主包。
- 跳 tabBar 页面必须用 `wx.switchTab`。
- 可以用 `wx.setTabBarItem`、`wx.setTabBarStyle` 动态修改显示效果。
- 不能动态新增未注册页面。

## 9. 分包技术原理

分包解决的是包体和首屏启动速度问题。

核心思想：

```text
主包：启动必须用的代码
分包：进入某些功能时再下载的代码
```

配置示例：

```json
{
  "pages": [
    "pages/home/index",
    "pages/order/index"
  ],
  "subpackages": [
    {
      "root": "packageMember",
      "pages": [
        "coupon/index",
        "balance/index"
      ]
    },
    {
      "root": "packageMall",
      "pages": [
        "goods/index",
        "pay/index"
      ]
    }
  ]
}
```

最终页面路径是：

```text
/packageMember/coupon/index
/packageMember/balance/index
/packageMall/goods/index
/packageMall/pay/index
```

运行机制：

```text
用户打开小程序
  ↓
微信只下载主包
  ↓
启动首页 / tabBar 页面
  ↓
用户进入分包页面
  ↓
微信下载对应分包
  ↓
加载分包页面
```

主包通常放：

- `app.js`
- `app.json`
- `app.wxss`
- 首页
- tabBar 页面
- 登录页
- 启动页
- 公共工具
- 公共组件
- 全局 store
- 首屏必须使用的资源

分包通常放：

- 订单详情
- 支付中心
- 优惠券详情
- 充值页
- 积分商城
- 活动页
- 设置页
- 低频功能
- 体积大的业务页面

## 10. 主包配置和分包配置的区别

主包配置写在 `pages` 中，表示小程序启动就必须具备的页面。

分包配置写在 `subpackages` 中，表示进入对应功能时再加载的页面。

```json
{
  "pages": [
    "pages/home/index",
    "pages/order/index"
  ],
  "subpackages": [
    {
      "root": "packageMember",
      "pages": [
        "coupon/index",
        "balance/index"
      ]
    }
  ]
}
```

对应关系：

```text
主包页面：
pages/home/index
pages/order/index

分包页面：
packageMember/coupon/index
packageMember/balance/index
```

配置写法区别：

```text
主包 pages：
直接写完整页面路径，从小程序根目录开始，不写文件后缀。

分包 subpackages：
root 写分包根目录，pages 只写 root 下面的相对路径。
最终路径 = root + page。
```

错误写法：

```json
{
  "root": "packageMember",
  "pages": [
    "packageMember/coupon/index"
  ]
}
```

正确写法：

```json
{
  "root": "packageMember",
  "pages": [
    "coupon/index"
  ]
}
```

加载时机区别：

```text
主包：
用户打开小程序 -> 微信下载主包 -> 主包页面可立即访问

分包：
用户打开小程序 -> 微信下载主包 -> 用户跳转到分包页面 -> 微信下载对应分包 -> 打开分包页面
```

依赖方向：

```text
分包可以依赖主包公共能力。
主包不要反向依赖分包里的代码。
```

否则会破坏按需加载意义。

## 11. 如何做小程序分发

做小程序分发，本质是做四件事：

```text
1. 把代码包发布到微信
2. 选一个固定入口页
3. 生成能打开这个入口页的入口物料
4. 入口页根据参数进入真实业务
```

### 11.1 发布代码包

以 `taro-mall` 为例：

```text
src 源码
  ↓
npm run build:weapp
  ↓
dist 小程序代码包
  ↓
微信开发者工具上传
  ↓
微信后台审核
  ↓
发布线上版
```

发布后，微信平台知道：

```text
AppID = 这个小程序是谁
版本 = 当前线上代码包
页面 = app.json 里声明了哪些页面
```

### 11.2 选一个分发入口页

例如 `taro-mall` 使用：

```text
/pagesIndex/pageIndex/index
```

这个页面必须在主包 `pages` 里注册。它的作用不是展示最终业务，而是做入口中转：

```text
接收参数
解析二维码 / 分享 / 外部链接参数
登录
获取门店 / 商户 / 业务类型
跳到点餐、支付、优惠券、充值等页面
```

入口页路径可以自定义，不必须叫 `pagesIndex/pageIndex/index`。关键是：

```text
必须是已注册页面
最好在主包
所有分发入口统一指向它
```

### 11.3 生成分发入口

常见入口有三类。

第一类：微信内分享。

```ts
useShareAppMessage(() => {
  return {
    title: '邀请你来点餐',
    path: `/pagesIndex/pageIndex/index?bizId=${bizId}&sid=${sid}&code=${code}`,
    imageUrl: shareImgUrl
  }
})
```

用户点击分享卡片后，微信会打开：

```text
AppID + /pagesIndex/pageIndex/index + bizId/sid/code
```

第二类：小程序码 / 桌台码 / 门店码。

后端调用微信接口生成小程序码时，核心参数类似：

```json
{
  "page": "pagesIndex/pageIndex/index",
  "scene": "bizId=10001&sid=20002&code=abc"
}
```

用户扫码后，入口页收到 `scene`，再解析业务参数。

第三类：外部渠道链接。

短信、H5、广告页要打开小程序时，可以使用微信 URL Link / URL Scheme。它最终也是指向：

```text
AppID + 页面路径 + 参数
```

### 11.4 入口页处理参数

入口页通常做这些事：

```ts
const params = getRouteParams()

const { bizId, sid, code, redirect } = params

setStorage('bizId', bizId)
setStorage('sid', sid)

const data = await queryQrBing({
  bizId,
  qrcodeKey: code
})

if (data.businesstype === 'order') {
  navigateTo('/pagePop/orderfoods/index')
}

if (data.businesstype === 'pay') {
  navigateTo('/pagePop/quickpayment/index')
}
```

完整链路：

```text
用户扫码 / 点分享 / 点短信链接
  ↓
微信识别 AppID
  ↓
微信打开线上代码包
  ↓
进入 /pagesIndex/pageIndex/index
  ↓
入口页解析 bizId / sid / code
  ↓
请求后端查询业务
  ↓
跳真实页面
```

关键理解：

```text
小程序分发不是发一个网页地址。
小程序分发是生成“打开哪个小程序、进入哪个已注册页面、带什么参数”的入口。
```

推荐结构：

```text
分发入口统一指向主包：
/pagesIndex/pageIndex/index

真实业务再跳：
/pagePop/orderfoods/index
/pagePop/paycenter/index
/pagePop/coupon/index
/pagePop/recharge/index
```

这样入口页可以统一处理登录、门店、参数、异常兜底和分包跳转。

### 11.5 使用二维码分发

二维码分发要先区分两种码：

```text
小程序码：
微信生成的太阳码，天然绑定 AppID、page、scene。

普通二维码：
普通 URL 二维码，需要通过微信后台的小程序码规则或扫码参数解析进入小程序。
```

二维码分发的本质是：

```text
二维码不是小程序代码，也不是网页页面。
二维码只是一个入口索引。
```

这个入口索引让微信知道三件事：

```text
打开哪个小程序：AppID
进入哪个页面：page path
携带什么参数：scene / query / q
```

用户扫码后的通用链路：

```text
用户扫码
  ↓
微信识别二维码类型
  ↓
微信确认要打开哪个 AppID
  ↓
微信打开该小程序的线上版本
  ↓
下载或使用本地缓存的主包
  ↓
进入固定入口页
  ↓
入口页解析 scene / query / q
  ↓
入口页拿短参数请求后端
  ↓
后端返回门店、桌台、业务类型等完整业务信息
  ↓
前端跳转到点餐、支付、优惠券等真实业务页
```

因此二维码里最好只放短参数，例如：

```text
code=TBL_8_ABCD
```

不要把完整业务数据放进二维码。更稳的设计是让后端维护短码映射：

```text
TBL_8_ABCD
  -> 商户 10001
  -> 门店 20002
  -> 桌台 8
  -> 业务类型 order
```

这样二维码印出去后也能在后端调整业务含义，例如禁用、过期、换桌、换门店、统计扫码来源。

#### 11.5.1 小程序码分发

小程序码适合桌台码、门店码、优惠券码、活动码。

后端生成小程序码时，核心参数是：

```json
{
  "page": "pagesIndex/pageIndex/index",
  "scene": "bizId=10001&sid=20002&code=abc"
}
```

用户扫码后的流程：

```text
用户扫小程序码
  ↓
微信识别 AppID
  ↓
打开线上小程序
  ↓
进入 page 指定的入口页
  ↓
入口页 onLoad 收到 scene
  ↓
decode scene
  ↓
根据 bizId / sid / code 请求后端
  ↓
跳到真实业务页
```

入口页接收示例：

```js
Page({
  onLoad(options) {
    const scene = decodeURIComponent(options.scene || '')
    // scene: bizId=10001&sid=20002&code=abc
  }
})
```

Taro 中也可以通过路由参数或当前实例读取：

```ts
const curRoute = Taro.getCurrentInstance()
const scene = curRoute?.router?.params?.scene
```

注意点：

- `page` 不要以 `/` 开头，通常写 `pagesIndex/pageIndex/index`。
- `page` 必须是已注册页面。
- `scene` 长度有限，不适合塞大对象。
- 复杂业务数据不要全放进 `scene`，建议只放短 ID，例如 `code`、`shareId`、`tableId`。
- 入口页拿到短 ID 后，再请求后端换取完整业务数据。

推荐参数设计：

```text
scene = code=abc123
```

后端维护映射：

```text
code=abc123
  -> bizId=10001
  -> sid=20002
  -> tableId=8
  -> businesstype=order
```

这样二维码短、稳定、可回收、可失效，也方便后端控制业务含义。

#### 11.5.2 普通二维码分发

普通二维码本质是一个 URL，例如：

```text
https://example.com/qrcode?code=abc123
```

如果这个 URL 被配置为小程序扫码规则，微信扫码后可能会把原始二维码内容作为 `q` 参数带给入口页。

入口页常见解析方式：

```ts
const curRoute = Taro.getCurrentInstance()
const q = curRoute?.router?.params?.q
const rawUrl = decodeURIComponent(q || '')
```

然后把 URL 里的参数转成业务参数：

```text
https://example.com/qrcode?code=abc123
  ↓
code=abc123
  ↓
请求后端 queryQrBing
  ↓
得到门店、桌台、业务类型
  ↓
跳业务页
```

普通二维码适合兼容已有线下二维码体系。小程序码适合新建小程序专用入口。

#### 11.5.3 桌台码示例

桌台扫码点餐可以这样设计：

```text
二维码只放短码：
code=TBL_8_ABCD

后端保存：
TBL_8_ABCD -> 商户 10001 / 门店 20002 / 桌台 8 / 业务类型 order
```

扫码流程：

```text
顾客扫码
  ↓
进入 /pagesIndex/pageIndex/index
  ↓
入口页拿到 code=TBL_8_ABCD
  ↓
调用 queryQrBing({ qrcodeKey: 'TBL_8_ABCD' })
  ↓
后端返回 businesstype='order'、sid、桌台信息
  ↓
前端保存门店和账单上下文
  ↓
跳转点餐页或购物车页
```

这种设计的好处：

- 二维码内容短，不容易超长。
- 桌台、门店、业务类型可以在后端改，不用重新印码。
- 可以做二维码禁用、过期、换桌、换门店等控制。
- 可以统一统计扫码来源。

#### 11.5.4 二维码分发的关键原则

```text
二维码不要承担业务全部信息。
二维码只承担入口索引。
完整业务信息由后端根据短码返回。
```

推荐：

```text
二维码 / 小程序码 -> 入口页 -> 短码 -> 后端查询 -> 真实业务页
```

不推荐：

```text
二维码里塞完整业务 JSON
二维码直接承载价格、商品、权限、订单状态
多个页面分别解析不同格式二维码
```

## 12. 小程序场景概念

小程序里的“场景”有两个容易混淆的概念：

```text
微信入口场景值：微信告诉你用户从哪里进入小程序。
小程序码 scene：开发者生成二维码时写入的业务参数。
```

二者名字都叫 `scene`，但含义完全不同。

### 12.1 微信入口场景值

入口场景值是微信系统给的数字，不是开发者自定义的。

例如用户可以从这些入口进入小程序：

```text
微信搜索
聊天分享卡片
群分享卡片
扫码
长按识别小程序码
公众号菜单
支付完成页
另一个小程序跳转
```

微信会在启动参数里给一个场景值：

```js
App({
  onLaunch(options) {
    console.log(options.scene)
  },

  onShow(options) {
    console.log(options.scene)
  }
})
```

常见场景值示例：

```text
1001：发现栏小程序主入口
1007：单人聊天会话中的小程序消息卡片
1008：群聊会话中的小程序消息卡片
1047：扫描小程序码
1048：长按图片识别小程序码
1049：扫描手机相册中选取的小程序码
```

入口场景值适合用来：

```text
统计用户来源
区分扫码、分享、搜索等入口
做入口兜底逻辑
分析营销渠道效果
```

不要把入口场景值当业务权限。比如 `scene === 1047` 只能说明用户是扫码进入，不能说明用户一定有某个桌台、订单或优惠券权限。业务权限仍然必须由后端校验。

### 12.2 小程序码里的 scene 业务参数

生成小程序码时，后端可以传一个自定义 `scene`：

```json
{
  "page": "pagesIndex/pageIndex/index",
  "scene": "code=TBL_8_ABCD"
}
```

这里的 `scene` 不是微信入口场景值，而是开发者自定义的业务参数。

页面接收：

```js
Page({
  onLoad(options) {
    const scene = decodeURIComponent(options.scene || '')
    console.log(scene) // code=TBL_8_ABCD
  }
})
```

它适合传：

```text
桌台短码
门店短码
活动短码
分享短码
优惠券领取码
```

例如：

```text
code=TBL_8_ABCD
```

后端解释：

```text
TBL_8_ABCD -> 商户 10001 / 门店 20002 / 桌台 8 / 业务类型 order
```

### 12.3 两种 scene 的区别

| 位置 | 含义 | 类型 | 谁生成 | 用途 |
|---|---|---|---|---|
| `App.onLaunch(options).scene` | 微信入口场景值 | 数字 | 微信 | 判断用户从哪里进入 |
| `Page.onLoad(options).scene` | 小程序码业务参数 | 字符串 | 开发者 / 后端 | 传业务短码 |

完整例子：用户扫描一个桌台小程序码。

```text
微信入口场景值：
1047，表示扫描小程序码进入。

业务 scene 参数：
code=TBL_8_ABCD，表示桌台短码。
```

流程：

```text
用户扫码
  ↓
App.onLaunch 里 options.scene = 1047
  ↓
进入 pagesIndex/pageIndex/index
  ↓
Page.onLoad 里 options.scene = code=TBL_8_ABCD
  ↓
前端拿 code 请求后端
  ↓
后端返回桌台、门店、业务类型
  ↓
跳点餐页
```

一句话总结：

```text
场景值 scene：微信告诉你用户从哪里来。
二维码 scene：你告诉小程序这次业务参数是什么。
```

## 13. 生命周期与启动机制

小程序生命周期的核心不是记函数名，而是理解三件事：

```text
小程序什么时候进前台 / 进后台
页面什么时候被创建 / 隐藏 / 销毁
页面栈如何变化
```

很多小程序 bug 都来自把 `onLoad`、`onShow` 用错，例如扫码进入、分享进入、tab 切换、从详情页返回列表页后状态没有刷新。

### 13.1 冷启动和热启动

冷启动：

```text
用户第一次打开小程序
或小程序已经被微信销毁后重新打开
```

冷启动的大致流程：

```text
App.onLaunch
  ↓
App.onShow
  ↓
Page.onLoad
  ↓
Page.onShow
  ↓
Page.onReady
```

热启动：

```text
小程序之前在后台，没有被销毁
用户再次打开
```

热启动的大致流程：

```text
App.onShow
  ↓
当前页面 Page.onShow
```

关键区别：

```text
App.onLaunch 只在冷启动执行一次。
App.onShow 每次进入前台都会执行。
```

所以扫码、分享、公众号菜单、外部链接再次进入小程序时，不要只在 `App.onLaunch` 里处理参数。用户第二次进入可能是热启动，只触发 `App.onShow`。

### 13.2 App 生命周期

```js
App({
  onLaunch(options) {
    // 冷启动，只执行一次
  },

  onShow(options) {
    // 每次进入前台都会执行
  },

  onHide() {
    // 进入后台
  }
})
```

`App.onLaunch` 适合：

```text
初始化全局配置
初始化日志
读取本地缓存
准备全局 store
```

`App.onShow` 适合：

```text
处理本次进入来源
读取 scene / path / query
恢复前台状态
检查更新
刷新全局登录态
```

例如：

```js
App({
  onShow(options) {
    console.log(options.scene)
    console.log(options.path)
    console.log(options.query)
  }
})
```

`App.onHide` 适合：

```text
暂停定时器
停止轮询
保存临时状态
上报停留时间
```

### 13.3 Page 生命周期

```js
Page({
  onLoad(options) {
    // 页面第一次创建时执行
  },

  onShow() {
    // 页面每次显示时执行
  },

  onReady() {
    // 页面首次渲染完成
  },

  onHide() {
    // 页面隐藏
  },

  onUnload() {
    // 页面销毁
  }
})
```

`onLoad`：

```text
一个页面实例只执行一次。
能拿到 URL query 参数。
适合做页面初始化。
```

示例：

```js
onLoad(options) {
  this.orderId = options.orderId
  this.queryDetail(options.orderId)
}
```

`onShow`：

```text
页面每次显示都会执行。
从下个页面返回会执行。
tab 切回来会执行。
小程序从后台回前台也可能执行。
```

适合：

```text
刷新可能变化的数据
检查登录态
读取 storage 里的跨页面参数
处理返回后的刷新
```

### 13.4 路由方式和生命周期

`navigateTo`：打开新页面，保留当前页。

```text
A 页面 wx.navigateTo 到 B
  ↓
A.onHide
  ↓
B.onLoad
  ↓
B.onShow
```

`navigateBack`：返回上一页。

```text
B 返回 A
  ↓
B.onUnload
  ↓
A.onShow
```

结论：

```text
返回上一页后，A 不会重新 onLoad，只会 onShow。
```

`redirectTo`：关闭当前页，打开新页。

```text
A redirectTo B
  ↓
A.onUnload
  ↓
B.onLoad
  ↓
B.onShow
```

`switchTab`：切换 tabBar 页面。

```text
普通页 / 当前 tab
  ↓
切到目标 tab
  ↓
目标 tab 第一次打开：onLoad + onShow
目标 tab 已经打开过：只触发 onShow
```

所以 tabBar 页面不要指望每次切回来都触发 `onLoad`。

`reLaunch`：关闭所有页面，重新打开某页。

```text
所有旧页面 onUnload
  ↓
新页面 onLoad
  ↓
新页面 onShow
```

### 13.5 返回页面后的刷新

常见错误理解：

```text
从详情页返回列表页，列表页 onLoad 会重新执行。
```

实际：

```text
不会。
只会执行列表页 onShow。
```

如果详情页修改了数据，列表页要刷新，可以在返回前设置上一页状态：

```js
const pages = getCurrentPages()
const prevPage = pages[pages.length - 2]

prevPage.setData({
  needRefresh: true
})

wx.navigateBack()
```

上一页在 `onShow` 中处理：

```js
onShow() {
  if (this.data.needRefresh) {
    this.queryList()
    this.setData({ needRefresh: false })
  }
}
```

也可以用 storage / 全局 store 标记刷新：

```js
wx.setStorageSync('orderListNeedRefresh', true)
wx.navigateBack()
```

列表页：

```js
onShow() {
  const shouldRefresh = wx.getStorageSync('orderListNeedRefresh')

  if (shouldRefresh) {
    wx.removeStorageSync('orderListNeedRefresh')
    this.queryOrderList()
  }
}
```

### 13.6 tabBar 页面刷新时机

tabBar 页面通常长期存在，不会频繁销毁。

例如订单 tab：

```text
第一次进入订单 tab：onLoad + onShow
切到首页：订单页 onHide
再切回订单 tab：订单页 onShow，不会 onLoad
```

所以订单列表、会员中心、购物车角标这类数据，应该放在 `onShow` 或 Taro 的 `useDidShow` 里刷新。

但不要无脑每次都全量请求。推荐做刷新标记：

```js
onShow() {
  const shouldRefresh = wx.getStorageSync('orderListNeedRefresh')

  if (shouldRefresh) {
    wx.removeStorageSync('orderListNeedRefresh')
    this.queryOrderList()
  }
}
```

### 13.7 扫码 / 分享入口的生命周期问题

扫码和分享入口经常携带参数：

```text
/pagesIndex/pageIndex/index?bizId=10001&sid=20002&code=abc
```

冷启动时：

```text
App.onLaunch 能拿到入口参数。
Page.onLoad 也能拿到页面参数。
```

热启动时：

```text
App.onLaunch 不会再执行。
App.onShow 会拿到新的入口信息。
```

所以扫码、分享、外部链接进入时，稳妥做法是：

```text
App.onShow 记录本次入口来源。
入口页 Page.onLoad / Page.onShow 解析业务参数。
不要只依赖 App.onLaunch。
```

### 13.8 Taro 对应关系

Taro 中常用 Hook 对应原生生命周期：

```text
useLoad       -> Page.onLoad
useDidShow    -> Page.onShow
useDidHide    -> Page.onHide
useUnload     -> Page.onUnload
```

实战原则：

```text
初始化页面参数：onLoad / useLoad
每次页面可见要检查：onShow / useDidShow
返回后刷新：onShow / useDidShow
tab 切换刷新：onShow / useDidShow
启动入口来源：App.onShow
冷启动全局初始化：App.onLaunch
清理资源：onHide / onUnload
```

一句话总结：

```text
onLoad 管“页面第一次创建”。
onShow 管“每次重新看到页面”。
```

小程序里凡是返回、tab 切换、热启动、扫码分享再次进入，都优先考虑 `onShow`。

## 14. 登录、身份与手机号授权

小程序登录要分清两件事：

```text
微信身份：这个微信用户是谁。
业务身份：这个用户在系统里是谁。
```

小程序登录流程就是把这两件事接起来。

### 14.1 登录链路

完整链路：

```text
小程序前端
  ↓ wx.login()
拿到临时 code
  ↓ 把 code 发给业务后端
业务后端
  ↓ 带 appid + appsecret + code 请求微信 code2Session
微信服务器
  ↓ 返回 openid / session_key / unionid
业务后端
  ↓ 根据 openid 找或创建业务用户
生成自己的 token
  ↓ 返回 token 给小程序
小程序后续请求
  ↓ header 带 token
业务后端识别当前用户
```

`wx.login()` 拿到的 `code` 是临时登录凭证：

```text
它不是用户 ID。
它不是业务 token。
它不能直接当登录态。
它只能给后端换 openid / session_key。
```

前端可以知道 `appid`，但不能知道 `appsecret`。`appsecret` 必须只放后端，因为小程序前端代码包会下载安装到用户微信客户端中，存在被反编译、抓包和调试的风险。

边界必须是：

```text
前端：调用 wx.login，拿临时 code。
后端：保存 appsecret，用 code 调微信 code2Session。
后端：生成自己的业务 token。
前端：保存并携带业务 token 请求接口。
```

### 14.2 openid、session_key、token 的区别

```text
openid：
用户在当前小程序下的唯一标识。

session_key：
微信会话密钥，后端用于处理微信敏感数据，不能下发给前端。

token：
业务系统自己的登录态，用于识别当前业务用户。
```

同一个微信用户，在不同小程序里可能有不同 `openid`。业务系统通常会用 `openid` 找到或创建自己的用户，再返回业务 `token`。

推荐理解：

```text
openid：识别这个微信用户。
token：识别这个业务用户。
手机号：绑定业务账号或会员。
头像昵称：展示资料。
```

### 14.3 登录不等于授权手机号或头像昵称

用户可以已经登录，但还没有授权手机号或头像昵称。

原因是登录和资料授权是不同能力：

```text
wx.login：
识别这是哪个微信用户，通常可静默调用。

getPhoneNumber：
用户主动点击授权后，才允许获取手机号。

chooseAvatar / nickname 输入：
用户主动选择或填写头像昵称。
```

所以用户状态可能是：

```text
openid：已知
token：已生成
手机号：未知
头像：未知
昵称：未知
```

点餐场景示例：

```text
用户扫码
  ↓
wx.login
  ↓
后端拿到 openid
  ↓
后端创建临时顾客
  ↓
返回 token
```

此时系统可以维护购物车、桌台上下文和订单草稿，但还不知道用户手机号。只有当用户开通会员、绑定账号或业务要求手机号时，才触发手机号授权。

### 14.4 获取手机号流程

获取手机号不是静默能力，必须由用户点击授权组件触发。

原生写法：

```xml
<button open-type="getPhoneNumber" bindgetphonenumber="getPhoneNumber">
  获取手机号
</button>
```

流程：

```text
用户点击“获取手机号”按钮
  ↓
微信弹出授权 / 确认界面
  ↓
用户同意
  ↓
前端收到 e.detail.code
  ↓
前端把 phoneCode 发给后端
  ↓
后端调用微信 phonenumber.getPhoneNumber
  ↓
微信返回手机号
  ↓
后端绑定业务用户 / 会员
```

前端只拿一次性 `phoneCode`，真正手机号由后端向微信换取。`wx.login()` 的 `code` 和手机号授权返回的 `code` 不能混用。

拒绝授权时，前端拿不到可用 `code`，需要兜底：

```text
手动输入手机号
稍后再绑定
只允许浏览，不允许开通会员 / 下单
按业务配置决定是否强制获取手机号
```

### 14.5 为什么看起来像“自动获取手机号”

实际项目里经常会让人误以为手机号是自动获取的，原因通常有三类。

第一类：按钮文案不是“获取手机号”，但按钮本身是手机号授权按钮。

例如按钮显示“注册/登录”，但代码是：

```tsx
<Button
  openType='getPhoneNumber'
  onGetPhoneNumber={async (phoneInfo) => register(phoneInfo)}
>
  注册/登录
</Button>
```

用户点击这个按钮时，技术上已经触发了微信手机号授权组件。

第二类：前一个页面已经拿到手机号授权 `code`，后续页面只是复用。

典型流程：

```text
用户点击注册 / 登录按钮
  ↓
按钮触发 getPhoneNumber
  ↓
前端拿到 detail.code
  ↓
如果需要付费开卡，先跳注册会员付款页
  ↓
付款成功后，注册会员页用之前的 code 换手机号并完成开卡
```

所以用户在注册会员页看起来像“自动获取手机号”，但授权动作已经发生在前一个按钮点击里。

第三类：后端或会员系统已经有手机号。

如果当前业务用户、会员卡、第三方会员系统里已经保存了手机号，页面可能只是自动回填或自动注册外部会员，并不是重新从微信静默获取手机号。

### 14.6 taro-mall 中的手机号链路

`taro-mall` 中的 `LoginBtn` 组件使用了手机号快速验证按钮：

```tsx
openType='getPhoneNumber'
onGetPhoneNumber={async (phoneInfo) => register(phoneInfo)}
```

`register` 方法中明确处理了拒绝授权：

```ts
if (_.isEmpty(detail?.code)) {
  if (orderMustRegister === 'GetPhoneNotMust') {
    onGetPhoneNotMust && onGetPhoneNotMust()
  }
  return
}
```

这说明如果用户没有授权，前端拿不到 `detail.code`，不会继续调用手机号换取接口。

拿到 `detail.code` 后，前端调用：

```ts
getPhoneNoInfo({
  ...detail,
  headimgurl: placeholderImages.default_avatar,
})
```

对应接口：

```text
/scrm/user/wxMaGetPhone
```

后端再调用微信手机号接口，拿到 `phoneNumber` 后绑定业务用户或会员。

如果需要付费开卡，`LoginBtn` 会把手机号授权返回的 `code`、`encryptedData`、`iv` 等参数带到注册会员页。注册会员页付款成功后再调用 `getPhoneNoInfo` 完成注册。因此这条链路看起来像注册页自动拿手机号，但本质上是复用前面用户点击按钮后得到的授权结果。

### 14.7 安全原则

```text
前端不能伪造手机号。
前端不能传手机号就直接信。
手机号必须由后端用微信返回的 code 换取，或由用户手动输入后经过短信验证。
订单金额、会员权益、优惠券资格、桌台权限都不能只信前端参数。
```

一句话总结：

```text
登录解决“识别用户”。
手机号授权解决“获取用户手机号”。
看起来自动，通常是因为授权按钮被包装成注册按钮，或后端已有手机号。
```

## 15. 网络请求与合法域名

小程序网络请求的核心原则：

```text
小程序前端不是自由浏览器，不能随便请求任何 URL。
微信会先检查这个域名是不是在小程序后台登记过的可信服务器。
```

它解决的是安全、审核、用户数据保护和线上服务可控问题。

### 15.1 为什么要配置合法域名

普通 H5 可以直接请求：

```js
fetch('https://api.xxx.com/order/list')
```

小程序请求：

```js
wx.request({
  url: 'https://api.xxx.com/order/list'
})
```

微信会先检查：

```text
https://api.xxx.com 是否在当前小程序的 request 合法域名中？
```

如果没有配置，线上会失败，常见错误类似：

```text
request:fail url not in domain list
```

小程序请求链路：

```text
小程序代码
  ↓ wx.request / Taro.request
微信客户端检查合法域名
  ↓
检查通过
  ↓
真正发 HTTPS 请求
  ↓
后端返回数据
```

### 15.2 合法域名类型

微信后台通常区分不同网络能力的域名：

```text
request 合法域名：
wx.request / Taro.request 普通接口请求。

uploadFile 合法域名：
wx.uploadFile 上传图片、文件。

downloadFile 合法域名：
wx.downloadFile 下载文件。

socket 合法域名：
wx.connectSocket，通常要求 wss。
```

不是配置了一个域名，所有网络能力都能用。接口请求能用，不代表上传文件也能用。

### 15.3 线上请求常见限制

线上一般要满足：

```text
必须配置到微信后台合法域名。
通常必须 HTTPS。
证书必须有效。
域名不能随便用 localhost。
域名不能随便用内网 IP。
协议、域名、端口要匹配。
```

例如配置了：

```text
https://api.example.com
```

通常可以请求：

```text
https://api.example.com/order/list
```

但下面这些可能失败：

```text
http://api.example.com/order/list       // 协议不对
https://api2.example.com/order/list     // 域名不同
https://api.example.com:8443/order/list // 端口不同，除非配置了这个端口
http://localhost:8080/order/list        // 线上不允许
```

开发者工具里可以临时勾选“不校验合法域名”，只适合本地调试，线上无效。

### 15.4 为什么要封装请求

如果每个页面都直接写：

```js
wx.request({
  url: 'https://api.xxx.com/order/list',
  data: {},
  success() {},
  fail() {}
})
```

很快会重复处理大量横切逻辑：

```text
baseURL
token
bizId / appId / sid / mid
loading
错误提示
登录过期
接口日志
重试
响应格式转换
```

真实项目一般会封装一层：

```text
页面
  ↓
service/order.ts
  ↓
request.ts
  ↓
Taro.request / wx.request
  ↓
后端
```

页面不直接关心 token、baseURL、错误日志，只调用业务 service。

### 15.5 标准请求封装示例

简化版：

```ts
async function request(options) {
  const token = getToken()
  const appId = getAppId()
  const bizId = getBizId()

  const res = await wx.request({
    url: BASE_URL + options.url,
    method: options.method || 'POST',
    data: {
      bizId,
      appId,
      ...options.data
    },
    header: {
      'content-type': 'application/json',
      Authorization: token ? `Bearer ${token}` : ''
    }
  })

  return res.data
}
```

业务页面只写：

```ts
const res = await request({
  url: '/order/list',
  data: { sid }
})
```

### 15.6 token 过期重登

小程序常见登录态：

```text
wx.login -> 后端换 openid -> 后端发 token
```

后续请求都带 token。token 过期时，请求封装要处理：

```text
请求前发现 token 已过期
  ↓
重新 wx.login
  ↓
后端重新发 token
  ↓
继续原请求
```

或者：

```text
请求返回 401 / 未登录
  ↓
重新登录
  ↓
重放原请求
```

重点：并发请求不能各自重登。

错误情况：

```text
页面同时发 10 个请求
  ↓
10 个请求都发现 token 过期
  ↓
触发 10 次 wx.login
  ↓
状态互相覆盖
```

更稳的做法是维护全局刷新 Promise：

```ts
let refreshingPromise = null

async function ensureLogin() {
  if (!isTokenExpired()) return getToken()

  if (!refreshingPromise) {
    refreshingPromise = doLogin().finally(() => {
      refreshingPromise = null
    })
  }

  return refreshingPromise
}
```

这样多个请求会等待同一次登录刷新。

### 15.7 错误重试

不是所有请求都能重试。

适合重试：

```text
网络超时
临时 502 / 503
查询类接口
明确幂等的接口
```

不适合随便重试：

```text
支付
下单
核销优惠券
扣库存
扣余额
注册会员
```

因为这些接口重复执行可能造成重复扣款、重复下单、重复核销。

建议策略：

```text
GET / 查询类 / 明确幂等 POST：可以重试 1-2 次。
写操作 / 支付 / 核销：不要前端自动重试，交给后端幂等处理。
```

### 15.8 loading 控制

loading 不应该每个接口都随便弹。

常见策略：

```text
普通静默请求：不弹 loading。
用户主动点击按钮：按钮 loading。
关键流程：全屏 loading。
支付 / 下单：必须 mask，防重复点击。
```

错误做法：

```text
页面一进来 5 个接口，每个都 showLoading/hideLoading。
```

这会导致 loading 闪烁或提前消失。

更稳的是计数器：

```ts
let loadingCount = 0

function showGlobalLoading() {
  loadingCount++
  if (loadingCount === 1) {
    wx.showLoading({ title: '加载中', mask: true })
  }
}

function hideGlobalLoading() {
  loadingCount--
  if (loadingCount <= 0) {
    loadingCount = 0
    wx.hideLoading()
  }
}
```

### 15.9 接口日志

接口日志分两类。

前端调试日志：

```text
请求 URL
请求参数
响应结果
耗时
```

生产环境不要打印敏感信息：

```text
token
手机号
openid
session_key
支付签名
身份证
完整地址
```

后端业务日志：

```text
用户是谁
商户 / 门店是谁
请求哪个接口
失败原因
traceId
订单号
业务状态
```

接口失败时，可以上报简化日志：

```text
url
业务参数摘要
错误码
错误信息
用户 token 对应 userId
traceId
```

不要上报完整 token 和手机号明文。

### 15.10 taro-mall 中的请求封装

`taro-mall` 中统一请求封装在：

```text
src/common/utils/request.ts
```

它承担了这些职责：

```text
拼接 baseURL
区分 H5 和小程序环境
token 过期时重新登录
自动带 token header
自动带 bizId / appId
特定接口显示 loading
统一调用 Taro.request
统一转换响应数据
失败时调用 bizLog 上报
```

关键逻辑：

```text
拼接真实请求 URL
判断 token 是否过期
H5 微信登录
小程序 wxLogin
设置 token header
自动补 bizId / appId
调用 Taro.request
接口失败时上报 bizLog
```

### 15.11 实践原则

```text
页面不要直接 wx.request。
所有请求走统一 request 封装。
合法域名必须提前在微信后台配置。
线上接口使用 HTTPS。
token 过期只做一次集中刷新。
查询接口可有限重试，支付 / 下单 / 核销不要乱重试。
loading 要按用户动作和关键流程控制。
日志要能排查问题，但不能泄露敏感信息。
```

一句话总结：

```text
小程序网络请求 = 微信平台域名校验 + 统一请求封装 + 后端 token 鉴权 + 安全日志与异常处理。
```

## 16. 动态渲染的边界

动态渲染是合规技术能力，但不能用来绕过审核。

合规用途：

- 后端配置页面内容。
- 动态主题。
- 动态 tabBar 文案、图标、颜色。
- 多门店展示不同商品。
- 登录后展示会员内容。
- 合规 A/B 测试。

不合规用途：

- 审核员看到空页面，真实用户看到另一套内容。
- 审核时隐藏未备案类目。
- 审核后远程打开违规功能。
- 按账号、地区、时间、审核特征切换审核版本。

动态渲染的底线：

```text
动态出来的内容、服务类目、权限、支付、会员、优惠券、点餐等业务
必须和审核提交信息一致。
```
