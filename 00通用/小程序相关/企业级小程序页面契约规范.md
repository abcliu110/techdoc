# 企业级小程序页面契约规范

## 1. 文档目的

本文定义企业级微信小程序页面应具备的页面契约。

在企业级小程序中，一个页面不应只是一个 `.vue`、`.tsx`、`.wxml` 或页面目录，而应是一个可治理的工程单元。

一个页面必须能被以下角色和系统共同理解：

```text
产品：知道页面完成什么任务。
设计：知道页面有哪些状态。
开发：知道页面如何初始化、流转和调用接口。
测试：知道页面如何进入、等待、操作和验证。
后端：知道页面用什么业务编号查询证据。
日志系统：知道错误发生在哪个页面、哪个入口、哪个业务对象。
构建系统：知道页面是否注册、是否在正确分包。
发布系统：知道页面是否涉及灰度、兼容和回滚风险。
```

因此，企业级页面交付物不是“能打开的 UI”，而是：

```text
页面身份
+ 页面入口
+ 参数契约
+ 状态机
+ 业务上下文
+ 操作定义
+ 接口契约
+ UI 状态
+ 错误定义
+ 测试契约
+ 后端证据
+ 安全隐私
+ 性能规范
+ 生命周期规范
+ 观测日志
+ 发布兼容
```

## 2. 核心原则

### 2.1 页面必须可识别

页面必须有稳定唯一的身份，而不是只依赖路径、标题或截图。

```text
错误：用 /pages/order/detail/index 或标题“详情”识别页面。
正确：用 pageId = order.detail 识别页面。
```

### 2.2 页面必须可进入

页面必须声明允许从哪些入口进入，包括普通路由、分享、扫码、订阅消息、支付回跳等。

页面不能只在“从上一页点击进入”时可用。

### 2.3 页面必须可恢复上下文

分享、扫码、订阅消息等入口可能是冷启动。页面必须能恢复登录态、门店、租户、会员、业务对象等上下文。

### 2.4 页面必须可观察

页面必须暴露状态、入口、业务编号、traceId、errorCode 等必要非敏感信息，供测试、日志和排障使用。

### 2.5 页面必须可测试

测试框架应能读取页面契约，验证：

```text
页面身份是否正确。
入口是否正确。
参数是否有效。
状态流转是否合法。
关键动作是否可观察。
业务结果是否有后端证据。
```

### 2.6 页面必须可证明业务结果

页面显示成功不等于业务成功。关键业务必须能通过后端 evidence 证明。

```text
页面状态证明前端显示。
业务编号证明业务对象。
traceId 证明请求链路。
后端 evidence 证明业务结果。
```

| 序号 | 契约项 | 回答的问题 | 主要使用者 |

| 2 | 页面入口 | 页面从哪里来？ | 开发、测试、产品 |
| 3 | 参数契约 | 页面启动需要什么输入？ | 开发、测试 |
| 4 | 状态机 | 页面有哪些状态，如何流转？ | 开发、测试、设计 |
| 5 | 业务上下文 | 页面当前处理哪个业务对象？ | 测试、后端、日志 |
| 6 | 操作定义 | 页面有哪些关键动作？ | 开发、测试 |
| 7 | 接口契约 | 页面依赖哪些接口？ | 开发、后端、测试 |
| 8 | UI 状态 | 每个状态显示什么？ | 设计、开发、测试 |
| 9 | 错误定义 | 错误如何分类和展示？ | 开发、测试、客服 |
| 10 | 测试契约 | 测试如何验证页面？ | 测试、开发 |
| 11 | 后端证据 | 业务结果如何证明？ | 测试、后端 |

| 16 | 发布兼容 | 上线是否安全？ | 开发、测试、发布 |

## 4. 页面身份

### 4.1 定义

页面身份用于稳定识别一个页面。

页面身份不是页面标题，也不是文件路径。标题可能变化，路径可能迁移，页面可能从主包迁到分包，但 `pageId` 应保持稳定。

### 4.2 必须字段

```ts
pageIdentity: {
  pageId: 'order.detail',
  routeName: 'OrderDetail',
  pagePath: '/pages/order/detail/index',
  rootQa: 'order.detail.page',
  title: '订单详情',
  domain: 'order',
  pageType: 'detail',
  owner: 'order-team',
  version: '1.0.0',
  package: 'orderPackage',
  tabBar: false
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `pageId` | 全局唯一页面 ID |
| `routeName` | 代码层路由名称 |
| `pagePath` | 小程序真实页面路径 |
| `rootQa` | 页面根节点测试标识 |
| `title` | 页面标题 |
| `domain` | 所属业务域 |
| `pageType` | 页面类型，例如 list、detail、form、result |
| `owner` | 负责团队或模块 |
| `version` | 页面契约版本 |
| `package` | 所属主包或分包 |
| `tabBar` | 是否 tabBar 页面 |

### 4.3 根节点暴露

页面根节点必须暴露核心身份：

```vue
<view
  data-page-id="order.detail"
  data-page-type="detail"
  data-domain="order"
  data-qa="order.detail.page"
>
  ...
</view>
```

### 4.4 校验要求

构建或测试工具应校验：

```text
pageId 全局唯一。
pagePath 已注册到 app.json 或 subpackages。
tabBar 配置与契约一致。
分包归属与契约一致。
页面根节点 data-page-id 与契约一致。
页面根节点 data-qa 与契约一致。
```

## 5. 页面入口

### 5.1 定义

页面入口用于说明页面允许从哪些来源进入，每个入口需要什么参数、上下文和校验规则。

同一个页面从不同入口进入，初始化逻辑和风险不同。

例如订单详情页：

```text
从订单列表进入：已有登录态，已有门店上下文。
从分享进入：可能冷启动，可能未登录。
从扫码进入：参数在 scene 中，需要解析。
从订阅消息进入：订单状态可能已经变化。
从支付回跳进入：需要刷新支付状态。
```

### 5.2 入口定义

```ts
entries: [
  {
    entryId: 'fromOrderList',
    type: 'route',
    sourcePageId: 'order.list',
    method: 'navigateTo',
    requiredParams: ['orderNo'],
    optionalParams: ['source'],
    authRequired: true,
    permissionsRequired: ['order:view'],
    contextRequired: ['storeCode'],
    coldStart: false,
    invalidFallback: {
      state: 'error',
      errorCode: 'INVALID_ORDER_NO'
    }
  },
  {
    entryId: 'fromShare',
    type: 'share',
    requiredParams: ['orderNo'],
    authRequired: true,
    permissionsRequired: ['order:view'],
    contextRequired: [],
    coldStart: true,
    invalidFallback: {
      state: 'error',
      errorCode: 'SHARE_PARAM_INVALID'
    }
  },
  {
    entryId: 'fromScan',
    type: 'scan',
    requiredParams: ['scene'],
    sceneParser: 'parseOrderScene',
    authRequired: true,
    permissionsRequired: ['order:view'],
    coldStart: true,
    invalidFallback: {
      state: 'error',
      errorCode: 'SCAN_PARAM_INVALID'
    }
  }
]
```

### 5.3 入口校验责任

入口校验分层完成：

| 层级 | 责任 |
|---|---|
| `router.open` | 校验内部跳转是否合法 |
| `resolveEntryContext` | 校验页面真实入口是否合法 |
| service / api / 后端 | 校验登录、权限和业务真实性 |
| 测试框架 | 校验每个入口是否按契约可用 |

### 5.4 内部入口控制

禁止散落手写：

```ts
wx.navigateTo({
  url: `/pages/order/detail/index?orderNo=${orderNo}`
})
```

必须使用：

```ts
router.open('order.detail', {
  entryId: 'fromOrderList',
  params: {
    orderNo
  }
})
```

`router.open` 至少校验：

```text
pageId 是否存在。
entryId 是否存在。
requiredParams 是否齐全。
跳转方法是否正确。
tabBar 页面是否使用 switchTab。
页面路径是否已注册。
```

### 5.5 页面真实入口控制

所有入口最终必须在页面初始化时经过 `resolveEntryContext`。

```ts
onLoad(options) {
  const entryContext = resolveEntryContext(orderDetailPageContract, options)

  if (!entryContext.valid) {
    this.setData({
      pageState: entryContext.fallback.state,
      errorCode: entryContext.fallback.errorCode
    })
    return
  }

  this.loadPage(entryContext.normalizedParams)
}
```

`resolveEntryContext` 至少负责：

```text
识别 entryId / entryType。
查找 contract.entries。
校验 requiredParams。
解析 scene。
标准化参数 normalizedParams。
恢复登录态。
恢复门店、租户、会员上下文。
入口非法时返回 fallback 状态和 errorCode。
```

### 5.6 根节点暴露

```vue
<view
  data-page-id="order.detail"
  data-qa="order.detail.page"
  :data-entry-id="entryContext.entryId"
  :data-entry-type="entryContext.type"
  :data-state="pageState"
>
  ...
</view>
```

## 6. 参数契约

### 6.1 定义

参数契约回答：

```text
这个页面启动时，需要哪些输入？
这些输入从哪里来？
输入缺失、非法、过期或无法解析时，页面进入什么状态？
```

参数契约发生在页面入口阶段，也就是页面还没有真正加载业务数据之前。

### 6.2 示例

```ts
params: {
  required: {
    orderNo: {
      type: 'string',
      source: ['route', 'share', 'scene'],
      validate: 'isOrderNo',
      invalidState: 'error',
      invalidErrorCode: 'INVALID_ORDER_NO'
    }
  },
  optional: {
    source: {
      type: 'string',
      default: 'unknown'
    }
  }
}
```

### 6.3 必填参数

| 页面 | 常见必填参数 |
|---|---|
| 订单详情页 | `orderNo` |
| 支付页 | `payNo` 或 `orderNo` |
| 退款页 | `refundNo` |
| 门店页 | `storeCode` |
| 会员页 | `memberNo` |

### 6.4 参数来源

```text
route
tabBar
share
scan
subscribeMessage
paymentReturn
urlLink
```

扫码入口必须将 `scene` 解析为标准参数：

```text
scene=type=order&no=ORD001
  -> normalizedParams.orderNo = ORD001
```

### 6.5 非法参数处理

| 场景 | 页面状态 | 错误码 |
|---|---|---|
| 缺少必填参数 | `error` | `MISSING_REQUIRED_PARAM` |
| 格式错误 | `error` | `INVALID_PARAM` |
| scene 解析失败 | `error` | `SCENE_PARSE_FAIL` |
| 参数过期 | `error` | `PARAM_EXPIRED` |
| 未登录 | `unauthorized` | `LOGIN_EXPIRED` |
| 无权限 | `forbidden` | `FORBIDDEN` |

## 7. 状态机

### 7.1 定义

状态机用于定义页面有哪些状态、状态如何流转、哪些状态可以被测试等待。

### 7.2 示例

```ts
stateMachine: {
  initialState: 'idle',
  states: [
    'idle',
    'loading',
    'ready',
    'empty',
    'error',
    'submitting',
    'success',
    'unauthorized'
  ],
  stableStates: ['ready', 'empty', 'error', 'success', 'unauthorized'],
  transientStates: ['loading', 'submitting'],
  transitions: {
    idle: { LOAD: 'loading' },
    loading: {
      LOAD_SUCCESS: 'ready',
      LOAD_EMPTY: 'empty',
      LOAD_FAIL: 'error',
      LOGIN_EXPIRED: 'unauthorized'
    },
    ready: { SUBMIT: 'submitting' },
    submitting: {
      SUBMIT_SUCCESS: 'success',
      SUBMIT_FAIL: 'error',
      LOGIN_EXPIRED: 'unauthorized'
    }
  }
}
```

### 7.3 要求

```text
页面根节点必须暴露 data-state。
测试框架必须知道状态全集。
中间状态不能长期不退出。
非法状态流转必须记录。
关键操作必须产生可观察状态变化。
```

## 8. 业务上下文

### 8.1 定义

业务上下文回答：

```text
这个页面当前展示或操作的是哪个业务对象？
这个对象属于什么类型？
它和哪些关联对象有关？
如何用这些编号查询后端证据？
```

参数契约是“进门票”，业务上下文是“进门后正在处理哪件业务”。

### 8.2 示例

```ts
bizContext: {
  primary: {
    type: 'order',
    noField: 'orderNo',
    requiredWhen: ['ready', 'success']
  },
  related: [
    { type: 'store', field: 'storeCode', requiredWhen: ['ready', 'success'] },
    { type: 'member', field: 'memberNo', requiredWhen: ['ready', 'success'] },
    { type: 'request', field: 'clientRequestId', requiredWhen: ['submitting'] },
    { type: 'trace', field: 'traceId', requiredWhen: ['loading', 'submitting', 'error'] }
  ]
}
```

### 8.3 根节点暴露

```vue
<view
  data-qa="order.detail.page"
  :data-state="pageState"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
  :data-store-code="order.storeCode"
  :data-member-no="order.memberNo"
  :data-trace-id="traceId"
>
  ...
</view>
```

### 8.4 外部编号优先

推荐暴露：

```text
orderNo
payNo
refundNo
memberNo
storeCode
couponNo
```

谨慎暴露：

```text
orderId
memberId
数据库自增 ID
雪花内部 ID
```

禁止暴露：

```text
token
session_key
AppSecret
openid
unionid
手机号明文
身份证号
完整地址
支付密钥
```

## 9. 操作定义

### 9.1 定义

操作定义用于说明页面有哪些关键动作，以及这些动作在哪些状态允许执行。

### 9.2 示例

```ts
actions: {
  submitOrder: {
    qa: 'order.confirm.submit',
    action: 'submitOrder',
    allowedStates: ['ready'],
    pendingState: 'submitting',
    successEvent: 'SUBMIT_SUCCESS',
    failEvent: 'SUBMIT_FAIL',
    idempotent: true,
    dangerous: false
  },
  refundOrder: {
    qa: 'order.detail.refund',
    action: 'refundOrder',
    allowedStates: ['paid'],
    pendingState: 'submittingRefund',
    successEvent: 'REFUND_SUCCESS',
    failEvent: 'REFUND_FAIL',
    idempotent: true,
    dangerous: true
  }
}
```

### 9.3 要求

```text
关键按钮必须有 data-qa。
关键按钮必须有 data-action。
非 allowedStates 不得触发动作。
点击后必须立即进入 pendingState。
提交中必须禁用重复点击。
危险操作必须二次确认。
```

## 10. 接口契约

接口契约用于说明页面依赖哪些接口，以及接口结果如何影响页面状态。

```ts
apis: {
  loadOrder: {
    timing: 'onLoad',
    requestParams: ['orderNo'],
    successEvent: 'LOAD_SUCCESS',
    failEvent: 'LOAD_FAIL',
    timeoutState: 'error',
    needTraceId: true
  },
  submitOrder: {
    timing: 'userAction',
    requestParams: ['orderNo', 'clientRequestId'],
    idempotent: true,
    successEvent: 'SUBMIT_SUCCESS',
    failEvent: 'SUBMIT_FAIL',
    needTraceId: true
  }
}
```

要求：

```text
页面不得散落裸 wx.request。
接口成功必须判断 HTTP 状态码和业务码。
接口失败必须映射到明确页面状态。
登录失效必须进入 unauthorized 或统一重登流程。
关键接口必须有 traceId 或 requestId。
```

## 11. UI 状态

UI 状态定义每个页面状态下显示、隐藏、启用、禁用哪些元素。

```ts
uiStates: {
  loading: {
    visible: ['order.confirm.loading'],
    hidden: ['order.confirm.submit']
  },
  ready: {
    visible: ['order.summary', 'order.confirm.submit'],
    enabled: ['order.confirm.submit']
  },
  submitting: {
    visible: ['order.summary', 'order.confirm.submit'],
    disabled: ['order.confirm.submit'],
    buttonText: '提交中'
  },
  error: {
    visible: ['order.confirm.error', 'order.confirm.retry']
  }
}
```

要求：

```text
loading 不能白屏。
empty 必须说明为什么为空以及下一步。
error 必须有错误提示和重试或返回路径。
submitting 必须禁用重复提交。
success 必须给出下一步。
```

## 12. 错误定义

错误定义用于说明错误如何分类、如何映射状态、如何展示。

```ts
errors: {
  unauthorized: {
    codes: [401],
    state: 'unauthorized',
    message: '登录已过期'
  },
  forbidden: {
    codes: [403],
    state: 'forbidden',
    message: '无权限访问'
  },
  orderNotFound: {
    codes: ['ORDER_NOT_FOUND'],
    state: 'error',
    message: '订单不存在'
  }
}
```

要求：

```text
禁止所有错误统一显示“系统异常”。
错误状态必须暴露 data-error-code。
错误必须可被测试构造和验证。
```

## 13. 测试契约

测试契约定义测试框架如何验证页面。

```ts
testContract: {
  rootQa: 'order.confirm.page',
  waitForReadyState: 'ready',
  forbiddenSleep: true,
  criticalStates: ['loading', 'ready', 'submitting', 'success', 'error'],
  requiredEvidence: ['orderEvidence']
}
```

要求：

```text
不依赖固定 sleep。
等待 data-state。
读取 data-entry-id / data-entry-type。
读取 data-biz-no。
查询后端 evidence。
输出状态覆盖率和入口覆盖率。
```

## 14. 后端证据

后端证据定义页面成功后如何证明业务结果真实发生。

```ts
evidence: {
  orderEvidence: {
    queryBy: 'orderNo',
    expected: {
      status: 'CREATED'
    }
  },
  paymentEvidence: {
    queryBy: 'payNo',
    expected: {
      payStatus: 'SUCCESS'
    }
  }
}
```

要求：

```text
页面显示成功不等于业务成功。
下单、支付、退款、领券、积分、会员等关键业务必须查后端证据。
接口超时或状态不确定时，应能通过 clientRequestId 或 traceId 查询结果。
```

## 15. 安全隐私

页面契约、`data-*` 属性、日志和测试桥都不得泄漏敏感信息。

```ts
security: {
  allowInternalIdInProd: false,
  forbiddenDataAttrs: [
    'token',
    'session_key',
    'openid',
    'unionid',
    'phone',
    'idCard',
    'address'
  ]
}
```

禁止暴露：

```text
token
session_key
AppSecret
openid
unionid
手机号明文
身份证号
完整地址
支付密钥
验证码
密码
```

## 16. 性能规范

页面性能规范必须定义：

```text
首屏关键接口。
非首屏延迟接口。
setData 更新范围。
图片压缩和懒加载策略。
长列表分页或虚拟化策略。
定时器、监听器、Socket、Worker 清理规则。
```

要求：

```text
首屏不被低优先级接口阻塞。
不高频全量 setData。
页面隐藏后停止无用更新。
长列表不能一次性全量渲染。
```

## 17. 生命周期规范

生命周期必须有明确职责：

| 生命周期 | 职责 |
|---|---|
| `onLoad` | 参数校验、入口解析、一次性初始化 |
| `onShow` | 必要刷新，例如订单状态、购物车数量 |
| `onReady` | 依赖渲染完成的节点操作 |
| `onHide` | 暂停定时器、轮询、动画、音视频 |
| `onUnload` | 清理定时器、监听器、Socket、Worker、请求回调标识 |

禁止：

```text
所有逻辑都堆在 onShow。
页面隐藏后继续高频 setData。
页面卸载后异步回调继续更新状态。
```

## 18. 观测日志

页面必须记录关键观测信息：

```text
页面进入。
入口 entryId / entryType。
参数校验结果。
状态变化。
关键点击。
接口请求和响应。
业务编号。
traceId / requestId。
错误码。
```

日志必须脱敏，不得记录 token、手机号明文、身份证号、完整地址等敏感信息。

## 19. 发布兼容

发布前必须确认：

```text
页面已注册到 app.json 或 subpackages。
分包归属正确。
tabBar 配置正确。
基础库最低版本明确。
新 API 有降级方案。
分享、扫码、订阅消息入口不受影响。
正式包不泄漏测试桥和敏感字段。
核心交易页面有灰度和回滚策略。
```

## 23. 最终标准

一个企业级小程序页面的交付标准是：

```text
有身份，可识别。
有入口，可冷启动。
有参数，可校验。
有状态，可等待。
有业务上下文，可关联。
有动作，可观察。
有接口，可追踪。
有 UI 状态，可验收。
有错误定义，可恢复。
有测试契约，可自动化。
有后端证据，可证明。
有安全边界，不泄密。
有性能约束，不劣化。
有生命周期，不泄漏。
有日志观测，可排障。
有发布兼容，可上线。
```

一句话：

```text
企业级页面不是“写出 UI”，而是交付一个有身份、有入口、有状态、有业务语义、有测试契约、有证据链、有安全边界、有发布治理的工程单元。
```
