# 企业后台经典界面设计模式知识库

## 1. 文档定位

本文是一套后端管理界面、ERP、SaaS 后台、业务中台的经典界面设计模式知识库。

它不是某个项目的页面说明书，也不是组件库文档。

它沉淀的是更底层的东西：

```text
经典页面结构为什么成立？
它解决什么业务认知问题？
它适合什么数据关系？
它的交互规则是什么？
它在工程上要暴露什么契约？
它如何被测试？
它有哪些反模式？
```

本文的目标是：以后设计任何后端管理界面时，不从空白画布开始，而是先从知识库选择合适的经典模式。

## 2. 世界级后台界面的根本原则

### 2.1 后台界面不是展示界面，而是业务操作系统

消费级页面常常以吸引、浏览、转化为目标。

后端管理界面的目标完全不同：

```text
让业务对象可识别。
让业务状态可判断。
让业务操作可执行。
让业务风险可控制。
让业务过程可追溯。
让多人协作可审计。
让异常处理可恢复。
让自动化测试可验证。
```

因此，后台界面设计不是“好看优先”，而是：

```text
结构优先。
对象优先。
状态优先。
效率优先。
证据优先。
```

### 2.4 后台配色不是装饰，而是业务语义系统

企业后台、ERP、金蝶类系统的配色原则和营销网站完全不同。

营销网站常用颜色制造情绪、品牌记忆和转化冲动。

后台系统用颜色表达：

```text
对象状态。
操作风险。
业务优先级。
异常程度。
权限边界。
数据变化。
流程进度。
```

所以后台配色的第一原则不是“漂亮”，而是：

```text
稳定。
克制。
可读。
可解释。
可测试。
长期看不疲劳。
```

颜色必须成为业务语言的一部分。

错误做法：

```text
为了页面活泼，把每个模块都配不同颜色。
为了强调，把所有按钮都做成高饱和色。
为了品牌，把整套后台染成一种强品牌色。
```

正确做法：

```text
大面积使用中性背景。
品牌色只服务主操作和关键识别。
状态色只表达稳定语义。
风险色只用于真实风险。
图表色用于分类和比较，不抢业务状态色。
```

### 2.2 经典结构来自数据关系，不来自视觉偏好

左树右表不是因为“左边放树好看”。

它成立是因为业务数据存在这种关系：

```text
范围 -> 对象集合
```

主从表不是因为“上下分区常见”。

它成立是因为业务数据存在这种关系：

```text
主对象 -> 明细对象集合
```

看板不是因为“卡片拖拽酷”。

它成立是因为业务对象存在这种关系：

```text
对象 -> 状态阶段
```

时间轴不是因为“竖向展示直观”。

它成立是因为业务事实存在这种关系：

```text
事件 -> 时间顺序
```

所以顶级设计师和架构师在做后台页面前，先判断数据关系，而不是先画布局。

### 2.3 每个经典结构都必须同时服务四类人

一个企业后台页面不是只给最终操作员用。

它同时服务：

```text
业务用户：完成录入、审批、处理、查询、分析。
管理者：看到进度、风险、异常、结果。
开发者：理解页面契约、状态、接口、扩展点。
测试者：稳定定位对象、状态、动作、结果和证据。
```

所以一个经典结构必须同时具备：

```text
清晰的信息架构。
稳定的交互行为。
明确的业务状态。
可落地的工程契约。
可自动化的测试证据。
```

## 4. 模式一：左树右表

### 4.1 一句话定义

左树右表用于表达：

```text
左侧确定业务范围，右侧处理该范围下的对象集合。
```

它是后端管理界面最经典、最重要的结构之一。

### 4.2 解决的问题

企业后台经常有大量数据，不可能平铺给用户。

用户真正的任务通常是：

```text
先确定范围。
再查看范围内对象。
再筛选、编辑、批量处理这些对象。
```

例如：

```text
先选组织，再看员工。
先选商品分类，再看商品。
先选门店区域，再看门店。
先选仓库，再看库存。
先选菜单节点，再看权限点。
```

左树右表把“范围选择”和“对象处理”分离，降低用户认知成本。

### 4.3 底层数据关系

左树右表的底层关系是：

```text
TreeNode(scope) -> TableRows(objects)
```

左树不是导航装饰，而是数据作用域。

右表不是全量列表，而是当前作用域下的对象集合。

### 4.4 适用场景

适合：

```text
组织树 -> 员工表。
部门树 -> 岗位表。
区域树 -> 门店表。
商品分类树 -> 商品表。
仓库树 -> 库存表。
菜单树 -> 权限点表。
会计科目树 -> 明细表。
客户分组树 -> 客户表。
设备分组树 -> 设备表。
```

共同特征：

```text
数据有天然层级。
用户常常先按层级定位范围。
右侧对象数量较多，需要表格、筛选、批量操作。
范围切换频繁。
```

### 4.5 不适用场景

不适合：

```text
没有层级范围的数据。
树节点很少，普通筛选即可解决。
树层级过深，用户难以展开。
用户主要任务是搜索，而不是按层级浏览。
移动端小屏高频使用。
右侧对象和左侧树没有强作用域关系。
```

如果只是为了让页面“看起来专业”而放一棵树，就是误用。

### 4.6 信息架构

左树右表包含四个核心区域：

```text
页面身份区：说明当前管理的对象类型。
范围树区：展示层级范围。
查询操作区：对右侧对象集合进行筛选和操作。
对象表格区：展示当前范围下的对象集合。
```

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 商品管理                                      │
├──────────────┬───────────────────────────────┤
│ 商品分类      │ 当前范围：食品 / 饮料            │
│              ├───────────────────────────────┤
│ 全部商品      │ 搜索 | 状态 | 品牌 | [新增] [导出] │
│ ├ 食品        ├───────────────────────────────┤
│ │ ├ 饮料      │ 商品编码  名称  状态  库存  操作   │
│ │ └ 零食      │ P001    可乐  启用  120  编辑   │
│ └ 日用品      │ P002    雪碧  停用  80   查看   │
└──────────────┴───────────────────────────────┘
```

### 4.7 核心交互

必须定义清楚这些规则：

```text
点击树节点后，右表是否立即刷新？
切换树节点后，分页是否回到第一页？
切换树节点后，右侧筛选条件是否保留？
点击父节点时，是否包含所有子节点数据？
节点数量统计是否包含子节点？
新增对象时，是否默认归属当前树节点？
批量操作影响当前节点，还是包含子节点？
无权限节点是隐藏、禁用，还是显示但不可进入？
```

这些不是细节，而是业务契约。

### 4.8 状态模型

左树右表至少有两组状态。

树状态：

```text
treeLoading。
treeLoaded。
treeLoadFailed。
nodeSelected。
nodeExpanded。
nodeCollapsed。
nodeNoPermission。
```

表状态：

```text
tableLoading。
tableLoaded。
tableEmpty。
tableLoadFailed。
tableFiltering。
tableBatchSelecting。
tableBatchOperating。
```

范围状态：

```text
scopeSelected。
scopeChanged。
scopeNoData。
scopeNoPermission。
```

### 4.9 权限与风险

左树右表最容易出问题的是权限和范围。

必须处理：

```text
用户能否看到该树节点？
用户能否看到节点下的数据？
用户能否对当前范围新增对象？
用户能否批量操作当前范围数据？
用户能否导出当前范围数据？
父节点权限和子节点权限是否一致？
```

高风险点：

```text
批量删除。
批量导出。
批量停用。
跨组织查看。
跨门店操作。
包含子节点统计。
```

### 4.10 工程契约

建议页面暴露以下契约：

```ts
type TreeTablePageContract = {
  pageId: string;
  objectType: string;
  scopeType: string;
  tree: {
    nodeIdField: string;
    parentIdField: string;
    labelField: string;
    includeChildrenPolicy: 'include' | 'exclude' | 'configurable';
  };
  table: {
    rowKey: string;
    columns: string[];
    defaultSort?: string;
    defaultFilters?: Record<string, unknown>;
  };
  actions: {
    pageActions: string[];
    rowActions: string[];
    batchActions: string[];
  };
};
```

页面根节点建议暴露：

```html
<div
  data-page-id="productManage"
  data-pattern="tree-table"
  data-object-type="product"
  data-scope-type="productCategory"
  data-current-scope-id="cat_1001"
  data-current-scope-name="饮料"
>
```

表格行建议暴露：

```html
<tr
  data-row-id="P001"
  data-object-type="product"
  data-scope-id="cat_1001"
  data-status="enabled"
>
```

### 4.11 测试要点

必须测试：

```text
默认选中节点是否正确。
点击节点后右表是否刷新。
父节点是否按约定包含子节点数据。
切换节点后分页是否重置。
筛选条件保留/清空是否符合契约。
新增对象是否默认带当前范围。
批量操作是否只影响当前范围。
无权限节点是否按设计处理。
导出是否只导出当前范围或明确的选择范围。
```

### 4.12 反模式

```text
左树只是装饰，右表数据和树无关。
右表没有显示当前范围，用户不知道自己在看哪里。
点击父节点时是否包含子节点没有说明。
树节点和表格权限不一致。
切换节点后筛选条件残留造成误解。
批量操作跨范围误伤数据。
移动端硬塞左树右表。
```

### 4.13 评价标准

左树右表达到 A 级必须满足：

```text
当前范围清楚。
右侧数据范围清楚。
范围切换规则清楚。
筛选、分页、排序规则清楚。
新增、批量、导出都受当前范围约束。
权限差异可解释。
自动化测试能读取当前 scope 和表格对象。
```

## 5. 模式二：上筛下表

### 5.1 一句话定义

上筛下表用于表达：

```text
先定义查询条件，再展示命中的对象集合。
```

它是后端管理界面最普遍的数据检索结构。

### 5.2 解决的问题

当数据没有明显层级，或者用户主要通过条件查找数据时，应使用上筛下表。

用户任务是：

```text
输入查询条件。
获得结果集合。
排序、分页、批量处理。
进入详情。
导出结果。
```

### 5.3 底层数据关系

```text
Filters -> ResultSet
```

查询条件定义结果集，结果集承载后续操作。

### 5.4 适用场景

```text
订单列表。
单据列表。
流水列表。
会员列表。
客户列表。
凭证列表。
库存明细。
支付记录。
退款记录。
导入记录。
操作日志。
```

### 5.5 信息架构

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 订单列表                                      │
├──────────────────────────────────────────────┤
│ 查询区                                        │
│ 单据号 | 日期范围 | 状态 | 门店 | 客户 | [查询] │
│ [更多筛选] [重置]                             │
├──────────────────────────────────────────────┤
│ 操作区                                        │
│ [新增] [批量审核] [导出] [打印]                │
├──────────────────────────────────────────────┤
│ 结果表                                        │
│ 单号  日期  客户  金额  状态  操作              │
├──────────────────────────────────────────────┤
│ 已选 N 条 | 共 M 条 | 分页                     │
└──────────────────────────────────────────────┘
```

### 5.6 核心交互

必须定义：

```text
默认查询条件是什么？
进入页面是否自动查询？
重置是清空所有条件，还是回到默认条件？
高级筛选是否折叠？
查询后 URL 是否同步参数？
导出的是当前页、全部查询结果，还是选中项？
批量操作是否基于选中项？
分页切换是否保留条件？
```

### 5.7 状态模型

```text
idle。
querying。
querySuccess。
queryEmpty。
queryFailed。
filterDirty。
batchSelected。
batchOperating。
exporting。
```

### 5.8 工程契约

```ts
type QueryTableContract = {
  pageId: string;
  objectType: string;
  defaultQuery: Record<string, unknown>;
  resetPolicy: 'empty' | 'default';
  filters: Array<{
    name: string;
    type: 'keyword' | 'dateRange' | 'enum' | 'numberRange' | 'selector';
    required?: boolean;
    defaultValue?: unknown;
  }>;
  table: {
    rowKey: string;
    columns: string[];
    pageable: boolean;
    sortable: boolean;
  };
  exportPolicy: 'currentPage' | 'queryResult' | 'selectedRows';
};
```

### 5.9 测试要点

```text
默认条件是否正确。
查询参数是否正确传给接口。
重置策略是否符合契约。
空结果和接口失败是否区分。
分页是否保留查询条件。
排序是否保留查询条件。
批量操作是否只处理选中项。
导出范围是否正确。
```

### 5.10 反模式

```text
筛选条件过多，首屏全是表单。
默认全量查询导致性能和权限风险。
重置后查询全公司数据。
状态筛选和表格状态口径不一致。
导出按钮没有说明范围。
批量操作失败只提示“失败”。
```

## 6. 模式三：主从表

### 6.1 一句话定义

主从表用于表达：

```text
一个主对象，对应一组从属明细对象。
```

### 6.2 解决的问题

企业数据经常是主从关系：

```text
订单和订单明细。
入库单和入库分录。
客户和联系人。
门店和设备。
对账单和流水。
活动和商品。
```

主从表让用户在不离开页面的情况下，查看主对象和从对象的对应关系。

### 6.3 底层数据关系

```text
MasterRecord -> DetailRows
```

主记录决定从表范围。

### 6.4 文本原型

```text
┌──────────────────────────────────────────────┐
│ 主表：订单                                    │
│ 单号    客户    金额    状态                  │
│ SO001  客户A   1000   已审核                  │
│ SO002  客户B   800    待审核                  │
├──────────────────────────────────────────────┤
│ 从表：SO001 的订单明细                         │
│ 商品    数量    单价    金额                  │
│ 可乐    10      3       30                    │
│ 雪碧    20      3       60                    │
└──────────────────────────────────────────────┘
```

### 6.5 核心交互

```text
点击主表行后，从表刷新。
主表当前行必须明显。
从表标题必须显示当前主对象。
主表分页切换后，从表如何处理必须明确。
主表多选时，从表是否显示合并明细必须明确。
从表操作是否反向影响主表状态必须明确。
```

### 6.6 状态模型

```text
masterLoading。
masterSelected。
detailLoading。
detailLoaded。
detailEmpty。
detailFailed。
masterChanged。
```

### 6.7 工程契约

```ts
type MasterDetailContract = {
  pageId: string;
  master: {
    objectType: string;
    rowKey: string;
    defaultSelectedPolicy: 'firstRow' | 'none';
  };
  detail: {
    objectType: string;
    foreignKey: string;
    rowKey: string;
  };
  selectionPolicy: 'singleMaster' | 'multiMaster';
};
```

### 6.8 测试要点

```text
默认选中主记录是否正确。
切换主记录后从表是否刷新。
从表接口参数是否使用当前主记录 ID。
主记录为空时从表是否为空态。
主表分页后从表是否清空或重新选择。
从表操作是否正确影响主表状态。
```

### 6.9 反模式

```text
主表当前行不明显。
从表没有显示属于哪个主对象。
主表切换后从表旧数据残留。
主表多选和从表内容关系混乱。
从表行操作改变主表但无反馈。
```

## 7. 模式四：左列表右详情

### 7.1 一句话定义

左列表右详情用于表达：

```text
从一组待处理对象中选择一个，并在右侧快速查看和处理。
```

### 7.2 适用场景

```text
审批中心。
消息中心。
工单中心。
客服会话。
待办任务。
异常处理。
订单快速处理。
```

### 7.3 底层数据关系

```text
QueueItems -> CurrentItemDetail
```

列表是队列，详情是当前处理对象。

### 7.4 文本原型

```text
┌──────────────┬───────────────────────────────┐
│ 待审批列表     │ 报销单 BX001                  │
│              │ 金额：1000                     │
│ BX001 高风险  │ 申请人：张三                    │
│ BX002 普通    │ 附件：3 个                      │
│ BX003 超期    │ 审批意见：                      │
│              │ [同意] [驳回] [转交]             │
└──────────────┴───────────────────────────────┘
```

### 7.5 核心交互

```text
点击列表项后右侧详情切换。
当前项必须高亮。
处理完成后列表状态要同步。
处理完成后是否自动进入下一条要明确。
列表筛选和详情状态要一致。
```

### 7.6 状态模型

## 10. 模式七：分步向导

### 10.1 一句话定义

分步向导用于表达：

```text
复杂任务必须按阶段完成，每一步都有明确输入、校验和结果。
```

### 10.2 适用场景

```text
数据导入。
活动创建。
流程配置。
系统初始化。
批量生成单据。
开户。
支付渠道配置。
```

### 10.3 底层数据关系

```text
Step1 -> Step2 -> Step3 -> Confirm -> Result
```

### 10.4 文本原型

```text
┌──────────────────────────────────────────────┐
│ 1 上传文件 -> 2 字段映射 -> 3 数据校验 -> 4 导入 │
├──────────────────────────────────────────────┤
│ 当前步骤内容                                  │
├──────────────────────────────────────────────┤
│ [上一步] [保存草稿] [下一步]                  │
└──────────────────────────────────────────────┘
```

### 10.5 设计原理

分步向导的核心不是“拆页面”，而是“拆风险”。

每一步都应该有边界：

```text
上传文件：只校验文件格式。
字段映射：只校验字段是否可匹配。
数据校验：校验业务规则并输出失败明细。
确认导入：展示影响范围。
导入结果：展示成功、失败、可下载明细。
```

### 10.6 状态模型

```text
stepIdle。
stepEditing。
stepValidating。
stepValid。
stepInvalid。
stepCompleted。
wizardSubmitting。
wizardSuccess。
wizardFailed。
```

### 10.7 测试要点

```text
不能跳过必经步骤。
每一步校验失败能定位原因。
返回上一步数据不丢失。
保存草稿后能恢复。
最终提交前有确认。
失败后能回到具体步骤修复。
```

### 10.8 反模式

```text
只是视觉分步，所有校验堆到最后。
没有草稿能力。
返回上一步数据丢失。
最后一步没有影响范围确认。
```

## 11. 模式八：工作台

### 11.1 一句话定义

工作台用于表达：

```text
一个角色每天最需要处理的任务、异常、指标和入口。
```

### 11.2 解决的问题

后台系统常见错误是把首页做成门户：

```text
公告。
轮播。
宣传图。
常用链接堆叠。
```

真正的工作台应该回答：

```text
我今天要处理什么？
什么异常最紧急？
哪些指标需要关注？
下一步去哪里操作？
```

### 11.3 适用场景

```text
财务工作台。
采购工作台。
销售工作台。
仓库工作台。
审批工作台。
运营工作台。
门店工作台。
老板驾驶舱。
```

### 11.4 文本原型

```text
┌──────────────────────────────────────────────┐
│ 待办 18 | 超期 3 | 异常 2 | 今日完成 40          │
├──────────────────────────────────────────────┤
│ 快捷入口：新增订单 | 审批 | 对账 | 导入          │
├──────────────────────────────────────────────┤
│ 我的待办                                      │
│ 单据号  类型  状态  截止时间                   │
├──────────────────────────────────────────────┤
│ 关键指标                                      │
│ 销售额 | 应收 | 库存预警 | 退款                 │
└──────────────────────────────────────────────┘
```

### 11.5 设计原理

工作台必须按角色设计。

```text
财务：待付款、待记账、异常凭证、资金风险。
仓库：待入库、待出库、库存预警、盘点差异。
采购：采购申请、待下单、待入库、供应商异常。
销售：待跟进客户、待发货订单、应收风险。
审批人：待审批、超期、驳回、已处理。
```

### 11.6 反模式

```text
所有角色看到同一个首页。
指标不能下钻。
待办不能直达对象。
异常没有优先级。
快捷入口只是菜单复制。
```

## 17. 模式十四：审批处理页

### 17.1 一句话定义

审批处理页用于表达：

```text

```

### 17.2 它不是普通详情页

普通详情页关注“看完整事实”。

审批页关注：

```text
是否应该同意？
风险在哪里？
金额是否异常？
预算是否足够？
附件是否完整？
历史意见是什么？
下游影响是什么？
```

### 17.3 信息架构

```text
对象身份：单据号、申请人、组织、状态。
决策摘要：金额、风险、预算、异常。
业务明细：字段、分录、附件。
流程上下文：当前节点、历史意见、下一节点。
操作区：同意、驳回、转交、加签。
```

文本原型：

```text
┌──────────────────────────────────────────────┐

│ 决策摘要：金额 1000 | 超预算 200 | 附件 3      │

│ [同意] [驳回] [转交] [加签]                    │
└──────────────────────────────────────────────┘
```

### 17.4 反模式

```text
审批人只看到同意/驳回按钮，看不到业务风险。
审批意见不是必填但驳回没有原因。
附件隐藏很深。
历史审批意见不可见。
审批后状态不刷新。
```

## 18. 模式十五：导入向导

### 18.1 一句话定义

导入向导用于表达：

```text
批量数据进入系统前，必须经过模板、映射、校验、确认、结果反馈。
```

### 18.2 标准结构

```text
下载模板 -> 上传文件 -> 字段映射 -> 数据校验 -> 确认导入 -> 导入结果
```

### 18.3 设计原理

导入是高风险操作。

它可能影响：

```text
大量主数据。
库存。
价格。
权限。
财务单据。
客户资料。
```

所以导入必须先验证，再写入。

### 18.4 文本原型

```text
┌──────────────────────────────────────────────┐
│ 商品导入                                      │
│ 1 下载模板 -> 2 上传 -> 3 校验 -> 4 导入        │
├──────────────────────────────────────────────┤
│ 校验结果：成功 980 行，失败 20 行              │
│ 失败明细：第 12 行 商品编码为空                │
│ [下载失败明细] [返回修改] [确认导入成功数据]    │
└──────────────────────────────────────────────┘
```

### 18.5 反模式

```text
上传后直接入库。
失败只提示导入失败。
没有失败行号和字段。
没有下载失败明细。
重复导入没有幂等控制。
```

### 19.4 反模式

## 20. 模式选择决策树

设计页面前，先用这棵决策树。

```text
1. 用户是否先选择业务范围？
   是 -> 左树右表。

2. 用户是否维护层级对象本身？
   是 -> 树表一体 / 左树右详情。

3. 用户是否通过条件查找数据？
   是 -> 上筛下表。

4. 用户是否围绕一个主对象查看明细？
   是 -> 主从表 / 详情页明细分组。

5. 用户是否连续处理队列任务？
   是 -> 左列表右详情 / 三栏工作界面。

6. 用户是否按状态推进对象？
   是 -> 看板。

7. 用户是否完成复杂多阶段任务？
   是 -> 分步向导。

8. 用户是否看指标并追溯原因？
   是 -> 图表钻取。

9. 用户是否选择一批对象形成范围？
   是 -> 双栏穿梭。

10. 用户是否做审批决策？
    是 -> 审批处理页。

11. 用户是否批量导入数据？
    是 -> 导入向导。

12. 用户是否配置复杂规则？
    是 -> 配置矩阵 + 预览发布。
```

## 21. 模式组合原则

真实后台页面常常不是单一模式，而是模式组合。

### 21.1 左树右表 + 详情抽屉

适合：

```text
商品分类树 + 商品表 + 商品快速详情。
组织树 + 员工表 + 员工详情。
```

原则：

```text
左树控制范围。
右表处理集合。
抽屉只做轻量查看或编辑。
复杂编辑跳转详情页。
```

### 21.2 上筛下表 + 批量操作 + 导入导出

适合：

```text
订单列表。
单据列表。
会员列表。
流水列表。
```

原则：

```text
筛选定义结果集。
批量操作基于选中项。
导出范围必须明确。
导入必须用导入向导，不要混在列表页里直接上传。
```

### 21.3 工作台 + 列表详情 + 审批处理

适合：

```text
审批中心。
任务中心。
异常中心。
```

原则：

```text
工作台展示任务和异常。
列表详情负责连续处理。
审批页负责决策。
处理结果回写工作台指标。
```

### 21.4 图表钻取 + 上筛下表 + 详情页

适合：

```text
销售分析。
库存分析。
财务分析。
```

原则：

```text
图表是入口。
明细表是解释。
详情页是证据。
```

## 22. 面向工程的统一页面契约

经典结构要想长期稳定，必须工程化。

页面至少暴露：

```text
pageId：页面身份。
pattern：页面结构模式。
objectType：业务对象类型。
objectId/objectNo：业务对象 ID 或编号。
scopeType/scopeId：当前范围。
state：页面状态。
actionState：当前动作状态。
permissionState：权限状态。
errorCode：结构化错误码。
```

示例：

```html
<div
  data-page-id="inventoryManage"
  data-pattern="tree-table"
  data-object-type="inventoryItem"
  data-scope-type="warehouse"
  data-scope-id="wh_001"
  data-state="tableLoaded"
>
```

这让页面从“人工可看”变成：

```text
可测试。
可观测。
可审计。
可复用。
可治理。
```

## 23. 面向测试的统一断言

测试不应该依赖视觉猜测。

应该围绕模式断言：

```text
左树右表：断言 scope 和 table rows 的关系。
上筛下表：断言 filters 和 result set 的关系。
主从表：断言 master 和 details 的关系。
看板：断言 status column 和 card state 的关系。
导入向导：断言 step、校验结果、失败明细。
审批页：断言业务事实、风险、审批动作、流程状态。
矩阵表：断言行列交叉配置和保存结果。
图表钻取：断言指标、钻取条件、明细和原始对象。
```

每个模式都应该有自己的自动化测试模板。

## 24. 反模式总表

| 反模式 | 后果 |
|---|---|
| 为了好看使用复杂结构 | 用户成本上升，开发和测试复杂 |
| 没有根据数据关系选模式 | 页面看起来合理，业务操作别扭 |
| 范围不清 | 批量操作、导出、权限容易出错 |
| 状态不清 | 按钮可用性混乱，测试不稳定 |
| 错误不结构化 | 用户无法恢复，测试无法判断 |
| 导出范围不清 | 数据泄漏和误操作风险 |
| 批量失败无明细 | 企业效率低，客服和运维成本高 |
| 审批页缺少业务事实 | 审批人只能盲批 |
| 图表不能下钻 | 报表变成装饰，无法追责 |
| 扩展不进入契约 | 后续测试、权限、导出、打印全断裂 |

## 25. 企业后台配色原理

### 25.1 配色的本质

企业后台配色不是美术问题，而是信息编码问题。

颜色要帮助用户在高密度数据中快速判断：

```text
什么是当前对象？
什么是可操作项？
什么是状态？
什么是风险？
什么是异常？
什么是成功？
什么是禁用？
什么是需要立即处理？
```

如果颜色只负责装饰，就会造成噪声。

如果颜色能稳定表达业务语义，就会成为系统语言。

### 25.2 金蝶/ERP 类后台为什么通常克制

ERP 用户每天长时间面对系统。

典型工作场景：

```text
财务连续处理凭证和付款。
仓库连续处理入库、出库、盘点。
采购连续看订单和供应商。
审批人连续处理待办。
运营连续看报表和异常。
```

因此配色必须避免视觉疲劳。

它通常呈现为：

```text
大面积中性浅色背景。
低饱和边框和分割线。
文字以深灰为主，而不是纯黑大面积压迫。
品牌色只用于主按钮、当前选中、关键链接。
状态色小面积使用。
风险色严格控制。
```

这不是保守，而是企业软件的专业性。

### 25.3 企业后台颜色角色

后台颜色至少分 8 类。

| 颜色角色 | 作用 | 使用位置 |
|---|---|---|
| 中性色 | 承载结构和文字 | 背景、表格、边框、分割线、正文 |
| 品牌色 | 表达系统识别和主操作 | 主按钮、选中态、链接、焦点 |
| 成功色 | 表示成功、启用、完成 | 成功提示、已完成、启用状态 |
| 警告色 | 表示注意、待处理、临界风险 | 待审核、库存预警、即将超期 |
| 危险色 | 表示失败、错误、高风险、破坏性操作 | 删除、作废、驳回、错误 |
| 信息色 | 表示普通提示或进行中 | 处理中、同步中、提示 |
| 图表色 | 区分类别和系列 | 报表、趋势、占比、排行 |
| 状态背景色 | 弱化状态强调 | 标签底色、轻提示、行背景 |

关键原则：

```text
同一种颜色只能表达稳定含义。
不要让红色有时表示删除，有时表示热销，有时表示重点。
不要让绿色有时表示启用，有时表示折扣，有时表示普通分类。
```

### 25.4 中性色系统

中性色是后台系统的骨架。

建议分层：

```text
页面背景：最浅，用于承载整个页面。
容器背景：白色或接近白色，用于表格、卡片、表单。
悬停背景：浅灰，用于 hover 和当前行轻反馈。
选中背景：品牌色极浅透明度，用于当前范围、当前行。
边框线：低对比灰，用于分割而不是抢注意。
主文字：深灰。
次文字：中灰。
辅助文字：浅灰。
禁用文字：更浅灰。
```

设计原理：

```text
后台界面信息密度高，中性色必须先把层次建立好。
如果结构层次靠大色块表达，页面会变得吵。
```

### 25.5 品牌色使用原则

品牌色不能铺满后台。

品牌色适合：

```text
主按钮。
当前选中菜单。
当前树节点。
当前 Tab。
文本链接。
焦点边框。
关键操作入口。
```

品牌色不适合：

```text
大面积背景。
所有图标。
所有标题。
所有卡片边框。
所有状态标签。
```

后台品牌色的正确心智：

```text
品牌色是导航和主动作信号，不是装饰底色。
```

### 25.6 状态色语义

ERP 和后端管理界面必须定义状态色字典。

推荐语义：

| 语义 | 建议色彩角色 | 示例状态 |
|---|---|---|
| 成功/完成 | 绿色 | 已审核、已完成、启用、已收款 |
| 进行中 | 蓝色/信息色 | 审核中、处理中、同步中 |
| 待处理 | 橙色/警告色 | 待审核、待付款、待入库 |
| 驳回/失败 | 红色/危险色 | 已驳回、失败、异常 |
| 关闭/终止 | 灰色 | 已关闭、已作废、已停用 |
| 草稿/暂存 | 灰蓝或中性 | 草稿、暂存 |
| 部分完成 | 蓝橙混合语义，通常用警告色弱化 | 部分入库、部分结算、部分收款 |

重点：

```text
颜色只是辅助，状态文案必须存在。
不要只靠颜色表达状态。
自动化测试也不能只靠颜色判断状态。
```

### 25.7 风险色原则

红色和强橙色必须珍惜。

适合使用红色：

```text
删除。
作废。
反审核。
驳回。
失败。
严重异常。
超期严重。
财务风险。
权限拒绝。
```

不适合使用红色：

```text
普通重点字段。
普通金额。
普通标题。
普通图表分类。
只是为了醒目的按钮。
```

原因：

```text
如果红色被滥用，真正危险时用户会忽略。
```

### 25.8 表格配色

表格是后台系统最高频区域。

表格配色原则：

```text
表头背景轻微区分。
行分割线要弱。
hover 行要轻。
选中行要清楚但不刺眼。
异常行可以弱色背景，但必须配合状态或图标。
金额、状态、操作不要全部高亮。
```

表格中的颜色层级：

```text
第一层：当前选中行。
第二层：异常/风险状态。
第三层：可点击链接。
第四层：普通文本。
```

反模式：

```text
奇偶行用强烈底色。
每列都用不同颜色。
金额一律红色。
操作按钮全是彩色。
异常行只靠背景色不写原因。
```

### 25.9 左树右表配色

左树右表最重要的是“当前范围”。

配色要表达：

```text
当前选中树节点。
树节点 hover。
树节点禁用。
树节点无权限。
右侧表格属于当前节点。
```

推荐：

```text
当前树节点：品牌色浅背景 + 品牌色文字或左侧竖线。
hover 节点：浅灰背景。
无权限节点：灰色文字 + 锁图标或提示。
右侧范围提示：使用中性文字，不要强色块。
```

不要：

```text
每一级树用不同颜色。
树节点图标全部彩色。
选中态只靠文字变粗。
右表没有任何当前范围提示。
```

### 25.10 表单配色

表单配色要帮助用户完成录入，而不是制造视觉花样。

原则：

```text
必填标识小面积使用。
错误状态用红色边框 + 错误文案。
警告状态用警告色 + 解释。
只读字段用浅背景或禁用样式。
分组标题用文字层级，不靠大色块。
当前聚焦字段用品牌色边框。
```

表单错误必须做到：

```text
颜色提示。
文字解释。
字段定位。
提交后自动滚动到第一个错误。
```

### 25.11 审批页配色

审批页的颜色要服务决策。

重点突出：

```text
风险。
超预算。
异常金额。
缺失附件。
历史驳回。
当前审批节点。
```

不应突出：

```text
普通字段。
普通说明。
普通装饰图标。
```

审批操作颜色建议：

```text
同意：主按钮或成功色，但不要过度刺激。
驳回：危险色。
转交/加签：次按钮。
查看附件/历史意见：普通链接。
```

### 25.12 报表和图表配色

图表色和业务状态色要分离。

图表色用于：

```text
区分类别。
比较系列。
表达趋势。
```

状态色用于：

```text
成功、失败、警告、危险、禁用。
```

不要把状态色随意作为图表分类色。

例如：

```text
红色在系统中表示失败，就不要用红色表示“华东区销售额”。
绿色在系统中表示成功，就不要用绿色表示“退款金额”。
```

报表配色原则：

```text
指标卡不应全彩。
异常指标才用警告或危险色。
趋势上涨/下降要结合业务语义，不是上涨一定绿、下降一定红。
图例颜色要稳定。
点击图表后的钻取条件要明确显示。
```

### 25.13 深色模式慎用

企业后台可以有深色模式，但不能为了“高级感”默认深色。

深色模式适合：

```text
监控大屏。
夜间调度。
数据中心。
指挥中心。
低环境光场景。
```

不适合：

```text
财务录入。
长时间表格处理。
复杂表单。
高密度审批。
大量打印预览。
```

原因：

```text
深色模式在高密度文本、表格、输入场景下更容易疲劳。
```

### 25.14 可访问性和色盲友好

后台颜色必须可访问。

要求：

```text
颜色不能作为唯一信息。
状态必须有文字。
错误必须有文案。
风险必须有图标或说明。
文字和背景对比度要足够。
禁用态不能低到无法阅读。
图表系列不能只靠红绿区分。
```

### 25.15 配色与测试契约

测试不能断言颜色本身，除非测试的是视觉规范。

业务测试应断言：

```text
data-status="approved"
data-risk-level="high"
data-action-state="submitting"
data-permission="denied"
```

颜色只是这些状态的视觉映射。

正确关系：

```text
业务状态 -> 语义 token -> 视觉颜色
```

而不是：

```text
颜色 -> 猜业务状态
```

### 25.16 配色 token 建议

企业后台应建立语义 token，而不是到处写色值。

示例：

```text
color.bg.page
color.bg.container
color.bg.hover
color.bg.selected
color.border.default
color.text.primary
color.text.secondary
color.text.disabled
color.brand.primary
color.status.success
color.status.processing
color.status.warning
color.status.danger
color.status.closed
color.risk.high
color.risk.medium
color.risk.low
color.chart.series.1
color.chart.series.2
```

好处：

```text
设计一致。
主题可替换。
状态语义稳定。
测试和代码不绑定具体颜色。
品牌升级成本低。
```

### 25.17 配色评价标准

一个后台配色达到 A 级，应满足：

```text
大面积克制，中性色层次清楚。
品牌色只用于主操作和选中态。
状态色语义稳定。
风险色没有滥用。
表格长时间阅读不疲劳。
表单错误能快速定位。
审批风险能被看见。
图表色不和业务状态色冲突。
颜色不是唯一信息来源。
有语义 token，而不是散落色值。
```

不合格表现：

```text
页面像营销页一样花。
所有按钮都是彩色。
所有金额都是红色。
状态色没有统一语义。
图表色和状态色冲突。
选中态不明显。
禁用态看不清。
错误只靠红框没有文字。
```

## 26. 知识库使用方式

设计一个新页面时，按这个流程：

```text
1. 识别业务对象。
2. 识别数据关系。
3. 从知识库选择经典模式。
4. 判断是否需要模式组合。
5. 写出页面契约。
6. 写出状态模型。
7. 写出权限和风险点。
8. 写出测试断言。
9. 再开始画原型。
```

注意顺序：不是先画原型，再补规则。

顶级设计师和架构师的顺序是：

```text
业务关系 -> 页面模式 -> 交互规则 -> 工程契约 -> 测试证据 -> 视觉细节
```

## 27. 最终结论

经典页面结构是企业后台设计的基础知识库。

它们不是 UI 套路，而是业务关系的视觉化表达：

```text
左树右表：范围 -> 对象集合。
上筛下表：条件 -> 结果集合。
主从表：主对象 -> 明细集合。
左列表右详情：队列 -> 当前处理对象。
三栏工作界面：队列分类 -> 对象 -> 处理。
树表一体：层级对象集合。
分步向导：阶段性任务。
工作台：角色 -> 任务、异常、指标、入口。
看板：状态 -> 对象。
矩阵表：行列 -> 配置结果。
图表钻取：指标 -> 明细 -> 原始对象。
时间轴详情：对象 -> 事件链。
双栏穿梭：候选集合 -> 已选集合。
审批处理页：业务事实 -> 风险判断 -> 流程动作。
导入向导：外部数据 -> 校验 -> 写入结果。
配置矩阵 + 预览发布：规则编辑 -> 影响评估 -> 受控生效。
```

一个后端管理界面是否专业，首先看它有没有选对模式。

一个团队是否成熟，首先看它能不能把这些经典模式沉淀成知识库，而不是每次重新发明页面。
