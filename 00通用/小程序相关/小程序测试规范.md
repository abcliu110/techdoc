# 小程序测试规范

## 1. 文档目的

本文用于判断小程序是否可以快速测试、如何设计测试、如何保留证据、如何判断是否允许发布。

测试目标不是简单点一遍页面，而是确认：

```text
入口正确
状态正确
路由正确
接口正确
数据正确
权限正确
异常可恢复
证据可复现
风险可拦截
```

本文与 `小程序开发规范.md` 配套使用。开发规范判断代码和页面是否具备可测试性；测试规范判断测试是否覆盖关键风险并形成可复查证据。

## 2. 测试分层

小程序测试分为五层：

| 层级 | 目标 | 主要方式 |
|---|---|---|
| 静态检查 | 发现配置、路由、包体、权限、代码规范问题 | 代码审查、构建检查 |
| 单元测试 | 验证工具函数、状态计算、接口适配、业务纯逻辑 | JS 单测或框架单测 |
| 组件测试 | 验证组件输入、输出、状态和事件 | 组件测试、页面片段测试 |
| UI 自动化测试 | 验证页面流程、用户动作、页面状态 | miniprogram-automator / DevTools 自动化 |
| 业务证据测试 | 验证后端真实业务结果 | 接口查询、数据库查询、业务后台证据 |

合格的小程序测试必须至少覆盖静态检查、核心主流程 UI 测试和业务结果验证。

## 3. 一票否决项

出现以下任一问题，测试结论不能判定为通过：

| 问题 | 原因 |
|---|---|
| 页面无法稳定进入 | 基础可测性不成立 |
| 关键元素没有稳定定位方式 | 自动化测试无法可靠执行 |
| 页面加载完成只能靠固定 sleep 判断 | 测试不稳定 |
| 关键接口失败但页面仍显示成功 | 用户和业务结果不一致 |
| 页面显示成功但后端业务数据未成功 | 不能只相信前端表现 |
| 登录、授权、支付等关键能力只有 mock，无真实链路验证 | 无法证明线上可用 |
| 测试证据没有操作、接口、页面状态、业务结果 | 失败无法复盘 |
| 测试环境和正式环境配置混用 | 发布风险不可控 |

## 4. 测试准入

开发提测前，测试必须确认以下条件：

| 类别 | 准入条件 |
|---|---|
| 构建 | 测试包可正常构建 |
| 页面 | 涉及页面已注册，能从真实入口进入 |
| 路由 | tabBar 和普通页面跳转方式正确 |
| 接口 | 测试环境接口可用，域名配置正确 |
| 数据 | 测试账号、门店、商品、订单、会员等数据准备完成 |
| 权限 | 测试账号具备对应角色、门店、租户权限 |
| 自动化 | 关键元素有 data-qa 或等价稳定标识 |
| 状态 | 页面提供 loading、ready、empty、error、success 等可等待状态 |
| 证据 | 能采集截图、操作日志、接口日志、后端业务证据 |

未满足准入条件时，测试可以拒绝提测，并按 `小程序开发规范.md` 要求开发补齐。

## 5. 测试环境规范

### 5.1 环境分类

至少区分：

```text
dev：开发联调
test：测试环境
uat：验收环境
prod：生产环境
```

### 5.2 环境检查

测试开始前必须记录：

- 小程序 AppID。
- 构建环境。
- 代码版本。
- 后端环境。
- 基础库版本。
- 微信开发者工具版本。
- 真机系统和微信版本。
- 测试账号和角色。
- 关键配置，例如门店、商户、租户、支付模式。

### 5.3 禁止做法

- 禁止测试包连接生产业务接口，除非是明确的生产验证任务。
- 禁止正式包依赖测试开关。
- 禁止测试账号和真实用户混用。
- 禁止用生产支付、生产库存、生产优惠券做破坏性测试。

## 6. 测试数据规范

### 6.1 数据准备

测试数据必须可复现，至少记录：

- 账号。
- 门店。
- 商品。
- 会员。
- 优惠券。
- 库存。
- 支付方式。
- 订单号。
- 业务状态。

### 6.2 数据隔离

测试数据应满足：

- 不污染生产数据。
- 不依赖上一次测试残留。
- 可重复执行。
- 可清理或可重置。
- 关键业务有唯一标识方便查询。

### 6.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 可复现 | 其他人按记录能重新准备或找到数据 |
| 可查询 | 业务结果能通过接口、数据库或后台验证 |
| 可隔离 | 多人并发测试不互相污染 |
| 可清理 | 测试后有清理或归档方案 |

## 7. 用例设计规范

### 7.1 用例必须覆盖的维度

每个功能至少考虑：

```text
正常路径
空数据
参数缺失
接口失败
接口超时
登录失效
权限不足
授权拒绝
重复点击
返回再进入
切后台再回来
弱网
低版本基础库
不同机型
```

### 7.2 用例字段

用例必须包含：

| 字段 | 说明 |
|---|---|
| 用例编号 | 唯一编号 |
| 功能模块 | 所属模块 |
| 前置条件 | 账号、数据、权限、配置 |
| 入口路径 | 从哪个页面或场景进入 |
| 操作步骤 | 用户动作 |
| 预期页面结果 | 页面展示、状态、提示 |
| 预期接口结果 | 请求、响应、业务码 |
| 预期业务结果 | 后端数据变化 |
| 证据要求 | 截图、日志、接口、数据库或后台证据 |
| 自动化等级 | 可自动化 / 半自动化 / 仅手工 |

### 7.3 自动化等级判定

| 等级 | 判定标准 |
|---|---|
| 可自动化 | 元素可定位、状态可等待、数据可准备、结果可查询 |
| 半自动化 | 需要扫码、支付、授权等人工介入，但其余步骤可自动化 |
| 仅手工 | 依赖真实外设、真实支付、平台审核或不可模拟能力 |

## 8. 路由测试规范

### 8.1 必测项

- 默认首页。
- tabBar 切换。
- 普通页面跳转。
- 返回上一页。
- 多级返回。
- `redirectTo` 后返回行为。
- `reLaunch` 后页面栈。
- 分享入口。
- 扫码入口。
- 订阅消息入口。
- 分包页面首次进入。

### 8.2 tabBar 测试

必须验证：

- tabBar 页面只能通过 `switchTab` 切换。
- 切换 tab 后非 tab 页面栈被关闭。
- tab 页面回来时触发刷新逻辑符合预期。
- tab 页面不会因重复进入产生重复请求或重复监听。

### 8.3 页面参数测试

必须验证：

- 正常参数。
- 缺失参数。
- 非法参数。
- 过期参数。
- 分享参数。
- 二维码 scene 参数。

### 8.4 入口契约测试规范

测试框架必须按页面 contract 中的 `entries` 验证入口，而不是只验证一个常用点击路径。

入口测试要覆盖四类校验：

```text
入口是否允许。
参数是否完整。
上下文是否可恢复。
非法入口是否进入明确状态。
```

#### 8.4.1 内部跳转入口测试

对内部入口，例如订单列表进入订单详情，测试必须验证：

```text
router.open 是否使用正确 pageId。
entryId 是否正确。
requiredParams 是否传齐。
跳转方法是否正确，例如 navigateTo / switchTab。
页面根节点 data-entry-id 是否等于预期。
页面最终是否进入 ready / error / unauthorized / forbidden 等稳定状态。
```

示例断言：

```ts
await routerOpen('order.detail', {
  entryId: 'fromOrderList',
  params: {
    orderNo: 'ORD202607030001'
  }
})

await page.assertAttr('order.detail.page', 'data-entry-id', 'fromOrderList')
await page.waitForAnyState('order.detail.page', ['ready', 'error', 'unauthorized'])
```

#### 8.4.2 外部入口测试

分享、扫码、订阅消息、URL Link、支付回跳等入口不经过内部 `router.open`，必须直接构造真实打开路径或 `scene` 参数测试。

必须覆盖：

| 入口类型 | 测试重点 |
|---|---|
| 分享入口 | 冷启动、未登录、参数缺失、业务对象不存在 |
| 扫码入口 | `scene` 正常、非法、过期、解析失败 |
| 订阅消息入口 | 消息 ID、业务状态已变化、登录态过期 |
| 支付回跳入口 | 支付成功、支付取消、支付处理中、后端回调延迟 |
| URL Link / Scheme | 参数编码、缺失参数、登录恢复 |

外部入口测试不得只用内部页面跳转代替。

#### 8.4.3 `resolveEntryContext` 结果测试

页面初始化后必须能暴露入口解析结果。

页面根节点应可读取：

```text
data-page-id
data-entry-id
data-entry-type
data-state
data-error-code
```

测试框架必须断言：

```text
当前 pageId 是否正确。
当前 entryId 是否正确。
当前 entryType 是否正确。
非法参数是否进入 error。
未登录是否进入 unauthorized。
无权限是否进入 forbidden。
```

#### 8.4.4 入口异常用例

每个关键页面至少覆盖以下异常入口：

```text
缺少必填参数。
参数格式错误。
scene 解析失败。
业务对象不存在。
登录态失效。
无业务权限。
上下文缺失，例如门店、租户、会员上下文不存在。
```

异常入口不能白屏，必须进入明确页面状态，并暴露 `data-error-code`。

#### 8.4.5 入口覆盖率

如果页面 contract 定义了多个入口，测试报告应输出入口覆盖率。

示例：

```json
{
  "pageId": "order.detail",
  "totalEntries": 4,
  "coveredEntries": [
    "fromOrderList",
    "fromShare",
    "fromScan"
  ],
  "missingEntries": [
    "fromSubscribeMessage"
  ],
  "coverage": "75%"
}
```

入口覆盖率不足时，测试结论必须说明未覆盖风险。

## 9. 页面状态测试规范

每个页面必须测试以下状态：

| 状态 | 测试内容 |
|---|---|
| loading | 请求中有明确反馈 |
| ready/success | 数据加载成功，页面可操作 |
| empty | 无数据时不白屏 |
| error | 接口失败有提示和重试 |
| submitting | 提交中按钮禁用或防重复 |
| unauthorized | 未登录或授权缺失时可恢复 |
| forbidden | 无权限时提示明确 |
| offline/timeout | 弱网或超时时可重试 |

自动化测试必须优先等待页面状态，不得默认使用固定等待时间作为通过依据。

## 10. 接口测试规范

### 10.1 前端请求验证

必须验证：

- 请求 URL。
- 请求 method。
- 请求参数。
- header 和 token。
- 超时配置。
- HTTP 状态码。
- 业务 code。
- data 结构。
- 错误提示。
- 登录失效处理。

### 10.2 后端业务验证

关键链路不能只看前端页面，必须验证后端结果。

示例：

| 功能 | 必须验证的后端结果 |
|---|---|
| 登录 | 用户身份、openid 绑定、业务 token |
| 下单 | 订单号、金额、商品、状态、库存、优惠 |
| 支付 | 支付单、订单支付状态、支付回调 |
| 退款 | 退款单、订单退款状态、原支付状态 |
| 领券 | 领券记录、券状态、使用限制 |
| 积分 | 积分流水、余额变化 |
| 会员 | 会员信息、等级、权益 |

### 10.3 异常接口测试

必须覆盖：

- 500。
- 404。
- 401/登录失效。
- 403/无权限。
- 业务失败码。
- 超时。
- 断网。
- 返回字段缺失。
- 返回空数组。
- 返回大数据量。

## 11. 登录与授权测试规范

### 11.1 登录

必须测试：

- 首次进入未登录。
- 已登录冷启动。
- 登录态过期。
- 切换账号。
- 清缓存后进入。
- `wx.login` code 换取业务 token。
- code 重复使用失败。

### 11.2 授权

涉及授权能力时必须测试：

- 首次授权同意。
- 首次授权拒绝。
- 拒绝后再次触发。
- 系统设置中关闭权限。
- 授权后业务继续。
- 授权失败兜底。

### 11.3 隐私合规

必须检查：

- 隐私指引是否配置。
- 页面文案是否说明用途。
- 后台配置和实际调用能力是否一致。
- 用户拒绝后是否仍能使用不依赖该权限的功能。

## 12. 表单与提交测试规范

必须覆盖：

- 必填为空。
- 格式错误。
- 超长输入。
- 特殊字符。
- 前后空格。
- 重复提交。
- 提交中返回。
- 提交中切后台。
- 提交成功。
- 提交失败。
- 服务端校验失败。

判断标准：

```text
前端能拦截基础错误
服务端能拦截权威错误
重复点击不会产生重复业务
失败后用户能继续修改并重新提交
```

## 13. 列表与分页测试规范

必须覆盖：

- 首次加载。
- 下拉刷新。
- 上拉加载更多。
- 无数据。
- 无更多。
- 接口失败。
- 加载中重复触发。
- 删除或变更后列表刷新。
- 从详情返回列表。
- 大数据量。

判断标准：

- 不重复插入数据。
- 不漏数据。
- 页码或游标正确。
- 空状态和错误状态明确。
- 滚动性能可接受。

## 14. 支付、订单、退款类关键链路测试

这类链路必须做业务级证据验证。

### 14.1 必须覆盖

- 正常成功。
- 用户取消。
- 支付失败。
- 支付中断。
- 重复点击支付。
- 支付后返回小程序。
- 后端回调延迟。
- 前端显示和后端状态不一致。
- 订单关闭。
- 退款成功。
- 退款失败。

### 14.2 必须留存证据

- 页面截图。
- 操作日志。
- 请求和响应。
- 订单号。
- 支付单号。
- 支付状态。
- 后端查询结果。
- 回调日志或状态更新时间。

## 15. 分包与包体测试规范

必须验证：

- 首次启动只加载主包。
- 进入分包页面时能正常下载和展示。
- 分包页面刷新、返回、分享正常。
- 分包预下载符合预期。
- 网络较差时分包加载有提示。
- 主包和分包大小不超过限制。
- 低频页面没有误放主包导致首包过大。

## 16. 性能测试规范

### 16.1 启动性能

必须关注：

- 首屏可见时间。
- 首页接口数量。
- 主包大小。
- 首屏图片大小。
- 登录态恢复耗时。
- 分包加载耗时。

### 16.2 运行性能

必须关注：

- 滚动是否卡顿。
- 大列表是否卡顿。
- 频繁点击是否卡顿。
- 倒计时是否高频 `setData`。
- 切后台是否停止无用更新。
- 页面来回切换是否越来越慢。

### 16.3 性能问题判定

以下现象必须记录为问题：

- 首屏长时间白屏。
- loading 无明确结束。
- 滚动明显掉帧。
- 页面切换延迟明显。
- 输入卡顿。
- 列表加载后内存明显增长且不释放。
- 后台页面仍持续刷新。

## 17. 兼容性测试规范

至少覆盖：

| 类型 | 要求 |
|---|---|
| 开发者工具 | 基础功能可运行 |
| iOS 真机 | 核心链路通过 |
| Android 真机 | 核心链路通过 |
| 不同屏幕 | 关键页面不遮挡、不溢出 |
| 低基础库 | 新 API 有降级 |
| 弱网 | 页面可恢复 |

如项目面向 PC、小程序平板、HarmonyOS 或特定设备，还必须增加对应测试。

## 18. UI 自动化测试规范

### 18.1 自动化必须依赖稳定契约

优先使用：

- `data-qa`。
- 页面状态。
- 业务 ID。
- 请求 trace。
- 后端业务查询。

禁止优先依赖：

- 坐标。
- 截图相似度。
- 页面文案。
- 不稳定层级。
- 固定 sleep。

### 18.2 自动化操作规范

每一步操作必须记录：

- actionName。
- selector。
- 页面路径。
- 操作前页面状态。
- 操作后页面状态。
- 截图。
- 关联请求。
- 失败原因。

### 18.3 自动化等待规范

等待优先级：

```text
页面状态 ready/success
关键元素出现
关键请求完成
业务结果可查询
最后才允许短暂固定等待
```

固定等待只能作为补充，不能作为主要判断依据。

### 18.4 操作反馈与状态流转断言

自动化测试不能只验证“点击后最终出现某个文案”。对关键操作，必须验证状态流转。

以提交订单为例，推荐断言链路：

```text
进入页面
-> 等待 data-state="ready"
-> 点击 data-qa="order.confirm.submit"
-> 等待 data-state="submitting"
-> 等待 submitting 结束
-> 最终进入 submitted / error / unauthorized 中的一个稳定状态
-> 如果期望成功，则断言最终状态为 submitted
-> 查询后端订单结果
```

伪代码：

```ts
await page.waitForState('order.confirm.page', 'ready')
await page.tapQa('order.confirm.submit')

await page.waitForState('order.confirm.page', 'submitting')

const finalState = await page.waitForAnyState('order.confirm.page', [
  'submitted',
  'error',
  'unauthorized'
], {
  timeout: 10000
})

expect(finalState).toBe('submitted')
```

测试框架必须检查三类问题：

| 检查项 | 失败判定 |
|---|---|
| 点击后是否进入处理中状态 | 点击后仍停留在 `ready`，说明事件未触发或无可观察反馈 |
| 处理中状态是否退出 | 长时间停留在 `submitting`、`paying`、`loadingMore`，说明接口卡死或异常未处理 |
| 最终状态是否符合预期 | 期望 `submitted`，实际进入 `error`、`unauthorized` 或其他状态 |

常见业务状态流：

```text
提交订单：
ready -> submitting -> submitted / error / unauthorized

加载更多：
ready -> loadingMore -> ready / noMore / loadMoreError

支付：
submitted -> paying -> paySuccess / payFailed / payCancelled / payPending

登录：
ready -> loggingIn -> loginSuccess / loginFailed / unauthorized

退款：
ready -> submittingRefund -> refundSuccess / refundRejected / error
```

固定 sleep 的使用限制：

- 只能用于动画、平台弹窗、系统能力返回等短暂缓冲。
- 必须有明确上限，通常不超过 300 到 500 毫秒。
- 不能作为接口完成、页面加载完成、提交完成、支付完成的主要判断依据。
- 如果没有可等待状态，应判定为开发可测试性不足，而不是在测试脚本里堆 sleep。

测试报告中必须记录：

```text
操作前状态
操作动作
操作后第一个状态
最终稳定状态
等待耗时
是否出现非法状态
是否超时停留在中间状态
```

## 20. 失败分类规范

失败必须分类，不能只写“测试失败”。

| 分类 | 说明 |
|---|---|
| `frontend.route` | 页面跳转、页面栈、tabBar 问题 |
| `frontend.state` | loading、empty、error、success 状态问题 |
| `frontend.render` | 页面渲染、样式、遮挡、溢出 |
| `frontend.event` | 点击、输入、事件绑定问题 |
| `frontend.validation` | 前端校验问题 |
| `api.request` | 请求参数、域名、超时、header 问题 |
| `api.response` | 接口返回结构、业务码、错误处理问题 |
| `backend.business` | 后端业务结果不符合预期 |
| `auth.login` | 登录态、token、code 换取问题 |
| `auth.permission` | 授权、权限、隐私问题 |
| `automation.selector` | 元素无法定位 |
| `automation.wait` | 等待条件不稳定 |
| `environment` | 环境、账号、数据、配置问题 |

## 21. 回归测试规范

每次修复后必须回归：

- 原失败用例。
- 同模块相邻功能。
- 同类页面。
- 同类接口错误处理。
- 涉及的主流程。
- 涉及的自动化用例。

不能只验证“当前按钮能点了”。

## 23. 快速测试能力判定

用以下表格判断一个小程序是否可以快速测试：

| 能力 | 合格标准 | 结果 |
|---|---|---|
| 页面可进入 | 入口清晰，路由稳定 | 是 / 否 |
| 入口可校验 | 页面 contract 定义 entries，页面暴露 entryId/entryType，测试可覆盖内部和外部入口 | 是 / 否 |
| 元素可定位 | 关键元素有稳定 QA 标识 | 是 / 否 |
| 状态可等待 | 页面有 loading、ready、error、success 状态 | 是 / 否 |
| 操作可观察 | 点击后能观察到 submitting、paying、loadingMore 等状态变化 | 是 / 否 |
| 无固定等待依赖 | 测试主要等待页面状态、请求或业务证据，不依赖固定 sleep | 是 / 否 |
| 数据可准备 | 测试数据可复现、可隔离 | 是 / 否 |
| 接口可追踪 | 请求有 requestId 或 traceId | 是 / 否 |
| 结果可验证 | 后端业务结果可查询 | 是 / 否 |
| 失败可归因 | 有截图、动作、接口、后端证据 | 是 / 否 |
| 环境可切换 | 测试环境和正式环境隔离 | 是 / 否 |
| 自动化可运行 | DevTools 自动化和测试包能力可用 | 是 / 否 |

判定：

```text
11-12 项为“是”：可快速自动化测试
8-10 项为“是”：可部分自动化，测试效率中等
少于 8 项为“是”：不可快速测试，需要先补开发可测试性
```
