# 小程序开发规范

## 1. 文档目的

本文用于判断一个微信小程序项目的开发是否规范、是否容易维护、是否具备快速测试条件。

本文不是微信小程序知识介绍，而是开发验收规范。评审小程序代码、页面、接口联调和测试准备度时，按本文逐项检查。

配套文档：

- `小程序测试规范.md`：定义测试准入、用例设计、自动化测试、证据留存和发布门禁。
- `小程序技术.md`：解释小程序基础结构、配置、运行机制和常见概念。
- `小程序tabBar页面栈机制.md`：解释 tabBar、页面栈和路由 API 行为。
- `小程序UI自动化框架设计与测试结果证据规范.md`：定义 UI 自动化框架和证据包设计。

## 2. 总体原则

小程序开发必须满足以下目标：

```text
页面可注册
路由可预期
状态可恢复
接口可追踪
授权可解释
异常可兜底
性能可接受
兼容可降级
测试可定位
结果可验证
```

判断一个小程序是否规范，不只看页面能不能打开，还要看它是否能稳定复现、快速定位问题、自动化测试和安全发布。

## 3. 一票否决项

出现以下任一问题，视为不符合开发规范，原则上不得进入提测或发布：

| 问题 | 影响 |
|---|---|
| 页面未在 `app.json` 或分包配置中注册 | 页面无法稳定路由、无法审核完整页面地图 |
| tabBar 页面使用 `navigateTo` 打开 | 路由行为错误，页面栈异常 |
| 普通页面使用 `switchTab` 打开 | 路由 API 使用错误 |
| 登录态只存在页面内存，刷新或冷启动丢失 | 用户链路不稳定 |
| 前端保存或暴露 AppSecret、session_key | 严重安全问题 |
| 接口域名未配置合法 HTTPS 域名 | 真机或线上请求失败 |
| `wx.request success` 中不判断 HTTP 状态码和业务码 | 错误响应被当作成功 |
| 页面无 loading、empty、error、success 状态 | 测试无法稳定等待，用户体验不可控 |
| 关键按钮、输入框、列表项无稳定测试标识 | 自动化测试难以定位 |
| 高频 `setData` 或一次性 `setData(this.data)` | 性能风险和卡顿风险 |
| 授权、手机号、定位等隐私能力无用途说明和拒绝兜底 | 审核和合规风险 |
| 支付、下单、退款等关键链路无幂等和防重复提交 | 业务数据风险 |

## 4. 工程结构规范

### 4.1 必须有清晰目录边界

推荐结构：

```text
src/
  app.js
  app.json
  app.wxss
  pages/
  packages/
  components/
  services/
  api/
  utils/
  stores/
  constants/
  qa/
```

如果使用 Taro、uni-app 或其他框架，可以保留框架目录，但必须能映射到小程序页面、组件、接口、状态和测试契约。

### 4.2 判断标准

| 检查项 | 合格标准 |
|---|---|
| 页面目录 | 页面按业务域或入口归类，不能所有页面堆在一个目录 |
| 组件目录 | 公共组件和业务组件分开 |
| 接口目录 | 接口定义集中管理，禁止页面内散落 URL 字符串 |
| 常量目录 | 路由、枚举、状态码、QA 标识集中管理 |
| 测试契约目录 | 自动化测试标识、页面状态、测试桥配置有独立位置 |

### 4.3 禁止做法

- 禁止页面直接拼散落的接口 URL。
- 禁止在多个页面复制同一套登录、请求、错误处理逻辑。
- 禁止把业务状态、接口返回码、路由路径写成无法追踪的魔法字符串。
- 禁止为了一个页面新增大而空的通用框架层。

## 5. 配置规范

### 5.1 `app.json`

必须明确配置：

- `pages`：主包页面列表。
- `entryPagePath`：默认首页，未配置时必须确认 `pages[0]` 就是首页。
- `window`：统一导航栏、背景色、下拉样式。
- `tabBar`：主导航页面。
- `subpackages`：分包结构。
- `networkTimeout`：网络超时。
- `permission`：授权用途说明。
- `requiredPrivateInfos`：涉及隐私接口时必须配置。
- `sitemapLocation`：搜索收录配置。
- `lazyCodeLoading`：自定义组件按需注入，视项目能力启用。

### 5.2 页面配置

页面 `.json` 只能配置该页面真正需要的能力：

- 页面标题。
- 自定义组件。
- 下拉刷新。
- 背景色。
- 页面方向。
- 导航栏样式。

禁止为了省事把所有页面都复制同一份配置。

### 5.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 页面注册 | 每个可路由页面都在主包或分包中注册 |
| 首页 | 默认首页明确，符合业务入口 |
| tabBar | tabBar 页面必须在 `pages` 中定义，数量 2 到 5 个 |
| 分包 | 非首屏、低频、重资源页面进入分包 |
| 权限说明 | 定位、相机、相册、手机号等能力有明确用途说明 |
| 超时 | 请求、上传、下载、Socket 有合理超时 |

## 6. 页面与路由规范

### 6.1 页面分类

页面必须先分类：

```text
tabBar 页面：一级入口，如首页、分类、购物车、我的
普通页面：详情、编辑、确认、支付结果等业务页面
分包页面：低频、独立、重资源或团队独立维护页面
独立分包页面：可独立启动的低耦合场景
```

### 6.2 路由 API 使用规范

| API | 使用场景 | 禁止场景 |
|---|---|---|
| `wx.navigateTo` | 打开普通页面，新增页面栈 | 打开 tabBar 页面 |
| `wx.redirectTo` | 替换当前普通页面 | 打开 tabBar 页面 |
| `wx.switchTab` | 切换 tabBar 页面 | 打开普通页面 |
| `wx.reLaunch` | 重置整个链路，如登录失效、切换门店 | 普通返回链路 |
| `wx.navigateBack` | 返回上一页或多级返回 | 代替业务状态跳转 |

### 6.3 页面参数规范

- URL 参数只传页面启动所需的最小参数，例如 `id`、`scene`、`source`。
- 禁止通过 URL 传大对象、敏感信息、完整订单信息。
- 页面参数必须在 `onLoad(options)` 中校验。
- 参数缺失、非法、过期必须展示错误态，不允许白屏。

### 6.4 判断标准

| 检查项 | 合格标准 |
|---|---|
| 页面入口 | 每个页面能说明由哪个入口进入 |
| 参数校验 | `onLoad` 中校验必填参数 |
| 页面栈 | 返回路径符合用户预期 |
| tabBar | tab 切换不会残留普通页面栈 |
| 异常参数 | 非法参数有错误页或兜底提示 |

## 7. 生命周期规范

### 7.1 App 生命周期

`App.onLaunch` 适合做：

- 读取本地登录态。
- 初始化全局配置。
- 初始化日志、埋点、请求追踪。
- 检查版本更新。

`App.onShow` 适合做：

- 处理从分享、扫码、订阅消息、外部入口进入的场景值。
- 恢复前后台状态。
- 检查需要前台刷新的一次性状态。

`App.onHide` 适合做：

- 暂停定时器。
- 保存临时状态。
- 记录离开时间。

### 7.2 Page 生命周期

| 生命周期 | 推荐用途 |
|---|---|
| `onLoad` | 读取和校验页面参数，初始化页面静态状态 |
| `onShow` | 刷新需要每次回到页面都更新的数据 |
| `onReady` | 读取节点信息、初始化依赖渲染完成的能力 |
| `onHide` | 暂停轮询、倒计时、音视频等 |
| `onUnload` | 清理页面级资源 |

### 7.3 禁止做法

- 禁止在 `onLoad` 中重复做每次展示都应该刷新的逻辑。
- 禁止在 `onShow` 中无条件重复请求大量接口。
- 禁止页面隐藏后仍高频定时 `setData`。
- 禁止页面卸载后继续回调更新页面状态。

### 7.4 判断标准

| 检查项 | 合格标准 |
|---|---|
| 初始化 | 参数初始化和业务刷新职责清晰 |
| 资源清理 | 定时器、监听器、Socket、Worker 有清理 |
| 返回刷新 | 从子页面返回后刷新策略明确 |
| 后台行为 | 小程序切后台后不继续浪费渲染资源 |

## 8. 数据与状态规范

### 8.1 页面状态必须显式建模

页面必须至少能表达：

```text
idle
loading
success
empty
error
submitting
submitted
```

业务复杂页面可以扩展：

```text
unauthorized
forbidden
expired
offline
retrying
partialSuccess
```

### 8.2 `data` 使用规范

- `data` 只放渲染相关数据。
- 非渲染数据放到实例字段，例如 `this.requestId`、`this.pendingParams`。
- 纯计算缓存、请求锁、埋点上下文不要放入 `data`。
- 使用 `setData` 只更新变化字段。

### 8.3 `setData` 规范

必须遵守：

- 合并连续更新。
- 不传大对象。
- 不使用 `this.setData(this.data)`。
- 更新数组或对象局部字段时优先使用数据路径。
- 高频更新元素应拆成独立组件。
- 页面隐藏后停止或延迟无意义更新。

### 8.4 判断标准

| 检查项 | 合格标准 |
|---|---|
| 状态 | 页面状态可以被用户和测试感知 |
| 请求中 | 请求期间有 loading 或 skeleton |
| 空数据 | 空列表不是白屏 |
| 错误 | 接口失败有错误态和重试入口 |
| 提交中 | 提交按钮有防重复状态 |
| 自动化等待 | 测试能等待稳定状态，而不是固定 sleep |

## 9. 接口与网络规范

### 9.1 请求封装

必须统一封装请求层，至少处理：

- baseUrl。
- token 或自定义登录态。
- 超时。
- HTTP 状态码。
- 业务返回码。
- 错误提示。
- 登录失效。
- 请求 ID 或 trace ID。
- 重试策略。
- 自动化测试请求记录。

### 9.2 请求判断

`wx.request` 进入 `success` 只表示收到服务器响应，不代表业务成功。必须继续判断：

```text
HTTP statusCode
业务 code
业务 data 是否符合预期
```

### 9.3 域名与证书

- 生产请求必须使用 HTTPS。
- 域名必须在小程序后台配置。
- 不允许前端直接请求 `api.weixin.qq.com` 调用服务端 API。
- 不允许在正式包使用测试域名。
- 不允许正式包依赖“不校验合法域名”。

### 9.4 弱网与失败处理

每个关键接口必须设计：

- 超时提示。
- 失败重试。
- 防重复提交。
- 服务端幂等。
- 失败后页面状态回滚。
- 用户可继续操作或明确退出路径。

### 9.5 判断标准

| 检查项 | 合格标准 |
|---|---|
| 封装 | 页面不直接调用裸 `wx.request` |
| 成功判断 | HTTP 和业务码都判断 |
| 错误分类 | 网络失败、业务失败、登录失效分开处理 |
| trace | 关键请求可关联页面动作 |
| 幂等 | 下单、支付、退款、领券等关键操作有防重复 |

## 10. 登录、授权与隐私规范

### 10.1 登录流程

标准流程：

```text
wx.login 获取 code
-> 前端把 code 发送到业务后端
-> 后端调用 code2Session
-> 后端保存 openid/session_key 关系
-> 后端生成业务登录态
-> 小程序只保存业务 token
```

### 10.2 安全要求

- `code` 只能使用一次。
- `session_key` 不允许下发到小程序。
- AppSecret 不允许出现在前端代码、配置、日志和测试包。
- token 存储要有过期处理。
- 登录失效必须有统一重登路径。

### 10.3 授权规范

涉及以下能力时必须有明确业务用途、拒绝兜底和隐私配置：

- 用户信息。
- 手机号。
- 地理位置。
- 相机。
- 相册。
- 麦克风。
- 蓝牙。
- 通讯地址。

### 10.4 判断标准

| 检查项 | 合格标准 |
|---|---|
| 登录态 | 冷启动、热启动、token 过期都有处理 |
| 敏感密钥 | 前端没有 AppSecret/session_key |
| 授权拒绝 | 用户拒绝授权后有可理解提示和备用路径 |
| 隐私指引 | 涉及隐私能力时后台配置和页面说明一致 |

## 11. 组件开发规范

### 11.1 组件边界

适合抽组件：

- 多页面复用。
- 复杂交互独立。
- 高频渲染区域。
- 业务含义明确，例如商品卡片、门店选择器、优惠券弹窗。

不适合抽组件：

- 只用一次且逻辑很薄。
- 为了“看起来统一”抽空壳。
- 把整个页面包成一个巨型组件。

### 11.2 组件接口

组件必须明确：

- `properties` 输入。
- `data` 内部状态。
- 对外事件。
- 默认值。
- 空数据表现。
- 禁用态。
- 加载态。
- 错误态。

### 11.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 输入 | properties 类型和默认值明确 |
| 输出 | 通过事件通知父级，不直接改父级数据 |
| 样式 | 组件样式不污染外部 |
| 状态 | loading、empty、disabled 可测试 |
| 性能 | 频繁更新在组件内局部 setData |

## 12. 表单与交互规范

### 12.1 表单

必须处理：

- 初始值。
- 必填校验。
- 格式校验。
- 最大长度。
- 提交中状态。
- 防重复提交。
- 接口错误回填。
- 成功后跳转或重置。

### 12.2 交互

- 按钮必须有 disabled/submitting 状态。
- 删除、退款、取消订单等危险操作必须二次确认。
- 列表分页必须处理加载中、无更多、失败重试。
- 关键结果页必须可返回明确入口，不依赖用户猜测。

### 12.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 输入 | 错误输入能被前端拦截 |
| 提交 | 连点不会产生重复业务 |
| 反馈 | 用户知道当前操作成功、失败或处理中 |
| 恢复 | 操作失败后页面仍可继续使用 |

## 13. 分包与包体规范

### 13.1 分包原则

主包只放：

- 默认启动页面。
- tabBar 页面。
- 所有分包都需要的公共资源。
- 登录、请求、基础组件等必要公共代码。

分包放：

- 低频页面。
- 重资源页面。
- 独立业务域。
- 团队独立维护模块。

### 13.2 判断标准

| 检查项 | 合格标准 |
|---|---|
| 主包 | 首屏相关代码为主，不堆低频业务 |
| 分包 | 业务边界清晰 |
| 资源 | 图片、字体、模型等重资源不随意进主包 |
| 预下载 | 高频下一跳页面可配置分包预下载 |
| 体积 | 主包、单包和总包体积不超过微信限制 |

## 14. 性能规范

### 14.1 启动性能

必须关注：

- 主包体积。
- 首页接口数量。
- 首屏阻塞资源。
- 登录态恢复速度。
- 分包加载策略。
- 首屏骨架屏或 loading。

### 14.2 运行时性能

必须关注：

- `setData` 数据量和频率。
- 长列表渲染。
- 图片尺寸和懒加载。
- 定时器清理。
- 页面隐藏后的任务暂停。
- 大对象序列化。
- 复杂动画和滚动区域。

### 14.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 首屏 | 用户能快速看到有效反馈 |
| 列表 | 大列表分页或虚拟化处理 |
| 图片 | 图片尺寸、格式、懒加载合理 |
| 更新 | 不高频全量 setData |
| 后台 | 后台页不抢前台渲染资源 |

## 15. 兼容规范

### 15.1 基础库兼容

使用新 API 或新配置时必须确认：

- 最低基础库版本。
- 目标用户微信版本分布。
- 是否需要 `wx.canIUse`。
- 是否需要降级方案。

### 15.2 多端兼容

需要按项目范围确认：

- iOS。
- Android。
- 微信开发者工具。
- PC 小程序。
- iPad。
- HarmonyOS。

### 15.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 新能力 | 有基础库版本判断 |
| 降级 | 不支持时有替代路径 |
| 平台差异 | iOS/Android 真机至少覆盖关键链路 |
| 样式 | 不依赖单一机型尺寸 |

## 16. 安全规范

### 16.1 前端安全

禁止：

- 存放 AppSecret。
- 存放 session_key。
- 明文存放敏感个人信息。
- 打印 token、手机号、身份证、地址等敏感日志。
- 信任前端传入的金额、权限、折扣、库存等关键值。

### 16.2 后端协同

关键业务必须由后端校验：

- 用户身份。
- 门店、租户、商户权限。
- 商品价格。
- 优惠券有效性。
- 库存。
- 订单状态。
- 支付金额。
- 退款权限。

### 16.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 密钥 | 前端无服务端密钥 |
| 日志 | 敏感信息脱敏 |
| 权限 | 前端只控制展示，后端控制权限 |
| 金额 | 金额以服务端计算为准 |

## 17. 可测试性开发规范

这是判断能否快速测试的核心。

### 17.1 页面必须具备页面契约

一个可测试、可维护的小程序页面，不能只是“页面 + 接口 + UI”。它必须是一份清晰的页面契约。页面契约用于说明：

```text
这个页面是谁。
这个页面从哪里进入。
这个页面需要什么参数。
这个页面有哪些状态。
这个页面展示或操作什么业务对象。
这个页面有哪些关键动作。
这个页面依赖哪些接口。
这个页面每个状态下显示什么。
这个页面如何处理错误。
这个页面如何被自动化测试。
这个页面成功后如何验证后端业务结果。
这个页面不能暴露哪些敏感信息。
```

页面契约不是额外文档负担，而是开发、设计、测试、后端和排障共同使用的行为说明。没有页面契约，页面就会退化成一堆 Vue/React 代码，测试只能靠截图、文案、固定等待和人工经验猜测页面行为。

#### 17.1.1 页面契约必须包含的 16 类定义

| 类别 | 说明 | 主要用途 |
|---|---|---|
| 页面身份 | 页面 ID、路径、标题、根 QA 标识 | 判断是否进入正确页面 |
| 页面入口 | tabBar、普通路由、分享、扫码、订阅消息等入口 | 验证页面能否冷启动 |
| 参数契约 | 必填参数、可选参数、参数来源、非法参数处理 | 防止页面依赖上一页内存对象 |
| 状态机 | 状态全集、初始状态、稳定状态、中间状态、状态流转 | 支持自动化等待和状态覆盖 |
| 业务上下文 | 主业务对象、关联业务对象、外部编号、traceId | 关联后端证据 |
| 操作定义 | 关键按钮、允许点击状态、点击后状态、成功失败事件 | 防重复提交和测试动作断言 |
| 接口契约 | 请求时机、参数来源、成功状态、失败状态、幂等性 | 关联接口和页面状态 |
| UI 状态 | 每个状态下显示、隐藏、禁用哪些元素 | 保证设计、开发、测试一致 |
| 错误定义 | 登录失效、无权限、业务失败、网络失败、超时等 | 避免所有错误都变成“系统异常” |
| 测试契约 | 根节点 QA、关键元素 QA、等待状态、禁止 sleep、覆盖状态 | 自动化测试直接使用 |
| 后端证据 | 用什么编号查询什么 evidence，期望后端状态是什么 | 证明业务结果真实发生 |
| 安全隐私 | 禁止暴露 token、openid、手机号、身份证等 | 防止测试属性泄漏敏感信息 |
| 性能规范 | 首屏接口、setData 范围、图片、列表、资源清理 | 保证页面稳定运行 |
| 生命周期规范 | onLoad、onShow、onReady、onHide、onUnload 职责 | 防止初始化、刷新、清理逻辑混乱 |
| 观测日志 | 页面进入、入口、状态变化、动作、请求、错误 | 出问题后能定位 |
| 发布兼容 | app.json 注册、分包、tabBar、基础库、灰度 | 保证上线安全 |

#### 17.1.2 页面身份定义

页面身份用于说明页面是谁。

必须定义：

```text
pageId
pagePath
data-qa
pageTitle
pageType
```

示例：

```ts
{
  pageId: 'order.detail',
  pagePath: '/pages/order/detail/index',
  qa: 'order.detail.page',
  title: '订单详情',
  pageType: 'detail'
}
```

用途：

```text
测试框架知道当前是不是进入了正确页面。
日志系统知道用户在哪个页面。
问题截图能定位到页面和业务模块。
页面路径变更时能统一更新契约。
```

#### 17.1.3 页面入口定义

页面入口用于说明页面可以从哪里来。

常见入口：

```text
tabBar 入口
普通路由入口
分享入口
扫码入口
订阅消息入口
支付回跳入口
外部 URL Link / Scheme 入口
```

示例：

```ts
entry: {
  allowedSources: ['order.list', 'payment.result', 'share', 'scan'],
  requiredParams: ['orderNo'],
  optionalParams: ['source', 'scene']
}
```

开发要求：

- 页面不能只在“从上一页点击进入”时可用。
- 支持分享、扫码、订阅消息的页面必须能冷启动。
- 冷启动时必须能恢复登录态、门店上下文和必要业务数据。
- 不允许依赖上一页临时内存对象作为唯一数据来源。

##### 17.1.3.1 入口校验责任分层

入口校验不能只靠页面自己做，也不能只靠统一路由做。企业级小程序应按入口来源分层校验：

```text
router.open：校验内部跳转是否合法。
resolveEntryContext：校验页面真实入口是否合法。
service / api / 后端接口：校验登录、权限和业务真实性。
测试框架：校验入口是否按页面契约可用。
```

各层责任如下：

| 层级 | 负责对象 | 必须校验 |
|---|---|---|
| `router.open` | 小程序内部主动跳转 | `pageId` 是否存在、`entryId` 是否存在、参数是否齐全、跳转方法是否正确 |
| `resolveEntryContext` | 页面 `onLoad` 收到的真实入口 | 入口是否允许、参数是否有效、`scene` 是否可解析、登录态和上下文是否可恢复 |
| service / api / 后端 | 业务真实性和权限 | 用户是否有权访问业务对象、门店/租户是否匹配、业务状态是否允许操作 |
| 测试框架 | 入口契约落地情况 | 每个入口能否进入、非法参数是否进入错误态、未登录是否进入 `unauthorized` |

##### 17.1.3.2 内部入口必须走 `router.open`

内部页面跳转禁止散落手写 `wx.navigateTo` 拼 URL。

禁止：

```ts
wx.navigateTo({
  url: `/pages/order/detail/index?orderNo=${orderNo}`
})
```

推荐：

```ts
router.open('order.detail', {
  entryId: 'fromOrderList',
  params: {
    orderNo
  }
})
```

`router.open` 至少负责：

```text
查找 pageId。
查找 entryId。
确认该入口允许内部 route 跳转。
校验 requiredParams。
根据页面类型选择 navigateTo / redirectTo / switchTab / reLaunch。
拼接 pagePath 和标准入口参数。
注入 entryId、entryType、sourcePageId。
记录入口埋点和路由日志。
```

内部跳转经过 `router.open` 后，最终生成的小程序路径应包含标准入口信息：

```text
/pages/order/detail/index?entryId=fromOrderList&orderNo=ORD202607030001
```

##### 17.1.3.3 页面真实入口必须走 `resolveEntryContext`

页面不能直接信任 `onLoad(options)`。所有入口，无论来自内部路由、分享、扫码、订阅消息还是支付回跳，最终都必须在页面初始化时经过 `resolveEntryContext`。

推荐：

```ts
onLoad(options) {
  const entryContext = resolveEntryContext(orderDetailPageContract, options)

  if (!entryContext.valid) {
    this.setData({
      pageState: entryContext.fallback.state,
      errorCode: entryContext.fallback.errorCode
    })
    return
  }

  this.loadPage(entryContext.normalizedParams)
}
```

`resolveEntryContext` 至少负责：

```text
读取 entryId / entryType。
查找页面 contract.entries 中对应入口。
校验 requiredParams。
解析扫码 scene。
标准化参数 normalizedParams。
恢复登录态。
恢复门店、租户、会员等上下文。
判断是否冷启动入口。
入口非法时返回 fallback 状态和 errorCode。
```

必须有 `resolveEntryContext` 的原因：

```text
分享、扫码、订阅消息、URL Link、支付回跳不会经过 router.open。
用户可能直接从外部打开页面。
测试也可能直接构造路径打开页面。
页面 onLoad 是前端入口校验的最后一道关口。
```

##### 17.1.3.4 外部入口的标准化

分享入口不能手写散落 path，应使用统一方法构造入口路径。

```ts
onShareAppMessage() {
  return {
    title: '订单详情',
    path: buildEntryPath('order.detail', {
      entryId: 'fromShare',
      params: {
        orderNo: this.data.order.orderNo
      }
    })
  }
}
```

扫码入口通常只有 `scene`，必须在入口契约中声明解析器。

```ts
{
  entryId: 'fromScan',
  type: 'scan',
  requiredParams: ['scene'],
  sceneParser: 'parseOrderScene',
  coldStart: true
}
```

解析后页面只能使用标准化参数：

```text
entryContext.normalizedParams.orderNo
```

订阅消息、支付回跳、URL Link 也必须转换成标准入口上下文，不能在页面中各写一套初始化逻辑。

##### 17.1.3.5 页面根节点必须暴露当前入口

页面根节点除页面身份和状态外，还应暴露当前入口信息。

```vue
<view
  data-page-id="order.detail"
  data-qa="order.detail.page"
  :data-entry-id="entryContext.entryId"
  :data-entry-type="entryContext.type"
  :data-state="pageState"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
  :data-trace-id="traceId"
>
  ...
</view>
```

测试框架可以据此断言：

```text
当前页面是 order.detail。
当前入口是 fromShare。
当前入口类型是 share。
当前状态是 ready。
当前业务对象是订单 ORD202607030001。
```

##### 17.1.3.6 禁止事项

入口控制中禁止：

- 禁止页面里散落手写 `wx.navigateTo` 拼业务 URL。
- 禁止页面直接信任 `options`，不做入口校验。
- 禁止分享 path 手写散落参数。
- 禁止扫码 `scene` 解析散落在多个页面里。
- 禁止入口缺参、参数非法、权限不足时白屏。
- 禁止不同入口各写一套重复初始化逻辑。
- 禁止把前端入口校验当作最终权限校验，服务端必须再次校验业务权限。

#### 17.1.4 参数契约

参数契约用于说明进入页面需要什么参数。

参数契约回答的是：

```text
这个页面启动时，需要哪些输入？
这些输入从哪里来？
输入缺失、非法、过期或无法解析时，页面进入什么状态？
```

参数契约发生在页面入口阶段，也就是页面还没有真正加载业务数据之前。它的目标是保证页面可以从普通路由、分享、扫码、订阅消息、URL Link 等入口稳定启动，而不是依赖上一页刚好传过来的内存对象。

必须定义：

```text
必填参数
可选参数
参数类型
参数来源
默认值
非法参数进入哪个状态
```

示例：

```ts
params: {
  required: {
    orderNo: {
      type: 'string',
      source: ['route', 'share', 'scene'],
      validate: 'isOrderNo',
      invalidState: 'error',
      invalidErrorCode: 'INVALID_ORDER_NO'
    }
  },
  optional: {
    source: {
      type: 'string',
      default: 'unknown'
    }
  }
}
```

参数契约必须覆盖以下内容：

##### 17.1.4.1 必填参数

必填参数是页面启动不可缺少的参数。缺失时页面不得继续执行业务请求，必须进入明确错误状态。

示例：

| 页面 | 常见必填参数 |
|---|---|
| 订单详情页 | `orderNo` |
| 支付页 | `payNo` 或 `orderNo`，按业务流程统一 |
| 退款页 | `refundNo` |
| 门店页 | `storeCode` |
| 会员页 | `memberNo` |
| 优惠券页 | `couponNo` 或 `couponCode` |

##### 17.1.4.2 可选参数

可选参数可以影响页面行为，但没有也必须能使用默认逻辑。

示例：

```text
source
campaignCode
couponNo
shareUserId
messageId
channel
```

可选参数必须定义默认值或缺省行为，不能让页面逻辑依赖“参数刚好存在”。

##### 17.1.4.3 参数来源

参数来源必须明确。常见来源：

```text
route：普通页面跳转
tabBar：tabBar 入口
share：分享卡片
scan：扫码 scene
subscribeMessage：订阅消息
paymentReturn：支付回跳
urlLink：外部链接
```

同一个业务字段可以来自不同入口。例如 `orderNo` 可以来自普通路由、分享参数，也可以来自扫码 `scene` 解析结果。

扫码入口尤其特殊，因为微信常把参数压到 `scene` 中。页面不得直接依赖原始 `scene` 字符串，必须解析成标准参数：

```text
scene=type=order&no=ORD001
  -> normalizedParams.orderNo = ORD001
```

页面业务逻辑只能使用标准化后的 `normalizedParams`。

##### 17.1.4.4 参数校验

参数校验不能只判断是否存在，还要判断是否合法。

必须校验：

```text
是否为空。
类型是否正确。
格式是否正确。
是否能从 scene 中解析。
是否属于当前入口允许的参数。
是否过期。
```

示例：

```ts
orderNo: {
  type: 'string',
  pattern: /^ORD\d{12}$/,
  invalidState: 'error',
  invalidErrorCode: 'INVALID_ORDER_NO'
}
```

##### 17.1.4.5 非法参数处理

参数非法时禁止白屏，禁止继续请求接口制造混乱。

推荐映射：

| 场景 | 页面状态 | 错误码 |
|---|---|---|
| 缺少必填参数 | `error` | `MISSING_REQUIRED_PARAM` |
| 订单号格式错误 | `error` | `INVALID_ORDER_NO` |
| scene 解析失败 | `error` | `SCENE_PARSE_FAIL` |
| 参数已过期 | `error` | `PARAM_EXPIRED` |
| 未登录 | `unauthorized` | `LOGIN_EXPIRED` |
| 无权限 | `forbidden` | `FORBIDDEN` |

页面根节点必须能暴露错误状态：

```vue
<view
  data-qa="order.detail.page"
  :data-state="pageState"
  :data-error-code="errorCode"
>
  ...
</view>
```

开发要求：

- `onLoad` 或页面初始化阶段必须校验必填参数。
- 参数缺失、非法、过期时必须进入明确错误状态。
- URL 参数只传最小上下文，例如 `orderNo`、`storeCode`、`scene`。
- 禁止通过 URL 传完整业务对象、敏感信息、金额、权限、库存等权威值。
- 页面不得依赖上一页内存对象作为必需参数来源。
- 扫码、分享、订阅消息入口必须能被解析为标准化参数。

#### 17.1.5 状态机定义

状态机用于说明页面有哪些状态，以及状态如何流转。

必须定义：

```text
initialState
states
stableStates
transientStates
errorStates
finalStates
transitions
```

示例：

```ts
stateMachine: {
  initialState: 'idle',
  states: [
    'idle',
    'loading',
    'ready',
    'empty',
    'error',
    'submitting',
    'success',
    'unauthorized'
  ],
  stableStates: ['ready', 'empty', 'error', 'success', 'unauthorized'],
  transientStates: ['loading', 'submitting'],
  transitions: {
    idle: { LOAD: 'loading' },
    loading: {
      LOAD_SUCCESS: 'ready',
      LOAD_EMPTY: 'empty',
      LOAD_FAIL: 'error',
      LOGIN_EXPIRED: 'unauthorized'
    },
    ready: { SUBMIT: 'submitting' },
    submitting: {
      SUBMIT_SUCCESS: 'success',
      SUBMIT_FAIL: 'error',
      LOGIN_EXPIRED: 'unauthorized'
    }
  }
}
```

开发要求：

- 不允许用多个互相冲突的布尔值拼页面状态。
- 每个状态必须有对应 UI 表现。
- 中间状态必须有退出路径。
- 非法状态流转必须记录日志。
- 测试框架必须能通过 `data-state` 或等价字段读取当前状态。

#### 17.1.6 业务上下文定义

业务上下文用于说明页面展示或操作哪个业务对象。

业务上下文回答的是：

```text
这个页面当前展示或操作的是哪个业务对象？
这个对象属于什么类型？
它和哪些关联对象有关？
如何用这些编号查询后端证据？
```

业务上下文通常发生在参数解析之后、业务数据加载中或加载完成后。参数契约保证页面能正确进来，业务上下文保证页面能证明自己正在处理什么业务。

必须区分：

```text
主业务对象
关联业务对象
请求链路对象
错误对象
```

示例：

```ts
bizContext: {
  primary: {
    type: 'order',
    noField: 'orderNo',
    requiredWhen: ['success']
  },
  related: [
    { type: 'store', field: 'storeCode', requiredWhen: ['ready', 'success'] },
    { type: 'member', field: 'memberNo', requiredWhen: ['ready', 'success'] },
    { type: 'request', field: 'clientRequestId', requiredWhen: ['submitting'] },
    { type: 'trace', field: 'traceId', requiredWhen: ['loading', 'submitting', 'error'] }
  ]
}
```

业务上下文必须覆盖以下内容：

##### 17.1.6.1 主业务对象

主业务对象是当前页面最核心的业务对象。

| 页面 | 推荐主业务对象 |
|---|---|
| 订单详情页 | `order` |
| 订单确认页 | 提交前无订单编号，提交后为 `order` |
| 支付页 | `payment` 或 `order`，按系统主流程统一 |
| 支付结果页 | `payment` |
| 退款页 | `refund` |
| 会员详情页 | `member` |
| 优惠券详情页 | `coupon` |
| 门店页 | `store` |

主业务对象通过以下属性表达：

```text
data-biz-type
data-biz-no
```

示例：

```vue
<view
  data-qa="order.detail.page"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
>
  ...
</view>
```

##### 17.1.6.2 关联业务对象

一个页面可能同时涉及多个业务对象，必须区分主对象和关联对象。

订单详情页可能关联：

```text
storeCode
memberNo
payNo
refundNo
couponNo
```

退款详情页示例：

```vue
<view
  data-qa="refund.detail.page"
  data-biz-type="refund"
  :data-biz-no="refund.refundNo"
  :data-order-no="refund.orderNo"
  :data-pay-no="refund.payNo"
>
  ...
</view>
```

含义：

```text
主对象：refund。
关联对象：order、payment。
```

##### 17.1.6.3 请求链路和证据链

业务上下文不只包含业务编号，还必须能关联请求和证据。

常见链路字段：

```text
traceId
requestId
clientRequestId
errorCode
```

提交类页面在提交前通常没有后端业务编号，因此必须使用 `clientRequestId` 或 `requestId` 串联链路：

```text
提交前：clientRequestId / requestId
提交中：clientRequestId / traceId
提交成功：bizNo / traceId
提交失败：clientRequestId / traceId / errorCode
```

示例：

```vue
<view
  data-qa="order.confirm.page"
  :data-state="pageState"
  data-biz-type="order"
  :data-biz-no="orderNo || ''"
  :data-client-request-id="clientRequestId"
  :data-trace-id="traceId"
  :data-error-code="errorCode"
>
  ...
</view>
```

##### 17.1.6.4 外部编号优先

业务上下文中优先使用外部业务编号，不优先使用内部数据库 ID 或雪花 ID。

推荐：

```text
orderNo
payNo
refundNo
memberNo
storeCode
couponNo
```

谨慎：

```text
orderId
memberId
数据库自增 ID
雪花内部 ID
```

原因：

```text
外部编号适合测试、客服、运营、日志和后端 evidence 查询。
内部 ID 对人没有语义。
直接暴露内部 ID 会增加前后端耦合和信息泄露风险。
```

如果当前系统只有内部 ID，没有外部编号，应优先推动后端返回稳定外部编号。过渡期只能在测试包中受控暴露内部 ID。

##### 17.1.6.5 安全边界

业务上下文不能塞敏感信息。

可以暴露：

```text
订单号
退款单号
支付单号
会员编号
门店编码
优惠券编号
traceId
requestId
clientRequestId
errorCode
非敏感业务状态
```

禁止暴露：

```text
token
session_key
AppSecret
openid
unionid
手机号明文
身份证号
完整地址
支付密钥
密码
验证码
内部权限数据
```

##### 17.1.6.6 参数契约和业务上下文的区别

| 对比项 | 参数契约 | 业务上下文 |
|---|---|---|
| 回答的问题 | 页面启动需要什么输入 | 页面当前代表哪个业务对象 |
| 发生阶段 | 页面进入时 | 参数解析后、业务加载中或加载后 |
| 来源 | route、share、scene、message | 接口返回、页面状态、业务流程 |
| 典型字段 | `orderNo`、`scene`、`source` | `data-biz-type`、`data-biz-no`、`traceId` |
| 主要用途 | 校验入口合法性 | 关联测试、日志、后端证据 |
| 异常处理 | 缺参、非法、解析失败 | 对象不存在、无权限、状态不一致 |
| 测试重点 | 构造正常、缺失、非法参数 | 读取业务编号并查后端 evidence |

简单说：

```text
参数契约是“进门票”。
业务上下文是“进门后正在处理哪件业务”。
```

完整流程示例：

```text
1. 用户从分享进入订单详情。
2. options 中带 orderNo。
3. resolveEntryContext 校验参数契约。
4. 参数合法，得到 normalizedParams.orderNo。
5. 调用 loadOrder(orderNo)。
6. 接口返回订单详情。
7. 页面进入 ready。
8. 页面根节点暴露业务上下文：
   data-biz-type="order"
   data-biz-no="ORD202607030001"
   data-store-code="S001"
   data-trace-id="trace_abc"
9. 测试框架读取 data-biz-no。
10. 测试框架查询后端 evidence 验证订单真实状态。
```

开发要求：

- 页面根节点必须暴露主业务对象类型，例如 `data-biz-type="order"`。
- 页面根节点优先暴露外部业务编号，例如 `data-biz-no="ORD202607030001"`。
- 复杂页面必须显式暴露关联业务编号，例如 `data-order-no`、`data-pay-no`、`data-refund-no`。
- 提交前没有业务编号时必须暴露 `clientRequestId` 或 `requestId`。
- 错误时必须暴露 `traceId` 或 `errorCode`，便于排查。
- 测试必须能用业务编号或 traceId 查询后端证据。
- 禁止在业务上下文属性中暴露敏感信息。

必须定义：

```text
调用接口

失败事件

```

示例：

```ts
actions: {
```

开发要求：

- 按钮必须有稳定 `data-qa`。
- 关键操作按钮必须有 `data-action`。
- 不允许在非允许状态触发提交、支付、退款等动作。

#### 17.1.8 接口契约

接口契约用于说明页面依赖哪些接口，以及接口如何影响状态。

必须定义：

```text
接口名

请求参数来源
成功事件
失败事件
超时处理
登录失效处理
是否幂等
```

```ts
apis: {

    failEvent: 'LOAD_FAIL',
```

开发要求：

- 页面不得散落裸 `wx.request`。
- 接口成功必须同时判断 HTTP 状态码和业务码。
- 接口失败必须映射到明确页面状态。
- 登录失效必须统一进入 `unauthorized` 或统一重登流程。
- 关键接口必须记录 `traceId` 或 `requestId`。

#### 17.1.9 UI 状态定义

UI 状态用于说明每个页面状态下显示、隐藏、启用、禁用哪些元素。

示例：


- `error` 必须有错误提示和重试或返回路径。
- `submitting` 必须禁用重复提交。
- `success` 必须给出下一步，例如查看订单或返回首页。

错误定义用于说明不同错误如何映射到页面状态。

- 登录失效、无权限、业务失败、网络失败、超时必须可区分。
- 错误状态必须暴露 `data-error-code`。
- 错误必须可被测试用例构造和验证。

#### 17.1.11 测试契约

测试契约用于说明测试框架如何验证页面。

必须定义：

```text

禁止固定 sleep
```


开发要求：

- 页面根节点必须有稳定 `data-qa`。
- 页面当前状态必须可读取。
- 关键元素必须有稳定 `data-qa`。
- 自动化测试必须等待状态、请求或后端证据，不得依赖固定 sleep。
- 页面必须提供足够信息让测试框架定位失败原因。

#### 17.1.12 后端证据定义

后端证据定义用于说明页面成功后如何证明业务结果真实发生。

示例：

```ts
evidence: {
  orderEvidence: {
    queryBy: 'orderNo',
    expected: {
      status: 'CREATED'
    }
  },
  paymentEvidence: {
    queryBy: 'payNo',
    expected: {
      payStatus: 'SUCCESS'
    }
  }
}
```

开发要求：

- 页面显示成功不等于业务成功。
- 下单、支付、退款、领券、积分、会员等关键业务必须能查询后端证据。
- 页面必须暴露能查询证据的非敏感编号，例如 `orderNo`、`payNo`、`refundNo`。
- 接口超时或状态不确定时，应能通过 `clientRequestId` 或 `traceId` 查询后端结果。

#### 17.1.13 安全与隐私定义

安全与隐私定义用于说明页面属性、日志、测试桥不能暴露哪些信息。

开发要求：

- 禁止暴露手机号明文、身份证号、完整地址。
- 内部雪花 ID 只能在必要且受控的测试环境谨慎使用。
- 正式包必须剥离测试桥和不必要调试字段。

### 17.4 操作反馈必须可观察

用户每一次关键操作都必须让页面进入一个可观察状态。所谓可观察，不只是用户肉眼看到按钮变化，还包括测试框架可以读取到状态变化。

关键操作包括：

```text
提交订单
发起支付
申请退款
保存表单
领取优惠券
修改购物车
加载更多
刷新列表
登录
授权
切换门店
```

页面必须满足以下要求：

| 要求 | 说明 |
|---|---|
| 点击前有可操作状态 | 例如 `ready`，表示页面允许提交 |
| 点击后立即进入处理中状态 | 例如 `submitting`、`paying`、`loadingMore` |
| 处理中禁止重复操作 | 按钮 disabled 或请求锁拦截 |
| 成功后进入成功状态 | 例如 `submitted`、`paySuccess` |
| 失败后进入错误状态 | 例如 `error`、`payFailed` |
| 登录失效进入登录状态 | 例如 `unauthorized` |
| 状态暴露给测试框架 | 页面根节点必须有 `data-state` 或等价状态字段 |

示例状态流：

```text
ready
  用户点击提交
    -> submitting
      接口成功
        -> submitted
      接口失败
        -> error
      登录失效
        -> unauthorized
```

页面根节点推荐写法：

```vue
<view
  data-qa="order.confirm.page"
  :data-state="pageState"
  :data-error-code="errorCode"
  data-biz-type="order"
  :data-biz-no="orderNo"
  :data-trace-id="traceId"
>
  ...
</view>
```

关键按钮推荐写法：

```vue
<button
  data-qa="order.confirm.submit"
  :disabled="pageState !== 'ready'"
  @tap="submitOrder"
>
  {{ pageState === 'submitting' ? '提交中' : '提交订单' }}
</button>
```

禁止做法：

- 禁止点击后页面没有任何状态变化，只在后台偷偷发请求。
- 禁止提交中按钮仍然可连续点击。
- 禁止接口失败后停留在提交中。
- 禁止成功、失败、登录失效都用同一个模糊状态。
- 禁止只靠 toast 表达关键业务结果，页面状态不变化。

开发自测时必须能说明：

```text
点击前页面是什么状态？
点击后立刻进入什么状态？
请求成功进入什么状态？
请求失败进入什么状态？
测试框架等待哪个状态？
```

### 17.5 页面业务上下文标记规范

页面业务上下文标记是指页面根节点、列表项节点、关键操作节点上的 `data-*` 语义属性。它不是给用户看的 UI，不是 CSS 样式选择器，也不是后端权限依据，而是用于自动化测试、问题定位、日志关联和后端证据查询的页面可观测性契约。

核心目标：

```text
让测试框架知道当前页面是谁。
让测试框架知道当前页面处于什么状态。
让测试框架知道当前页面展示的是哪个业务对象。
让测试框架能用业务编号、traceId、errorCode 关联接口日志和后端证据。
```

#### 17.5.1 基本定义

页面业务上下文标记推荐使用小程序节点的 `data-*` 属性表达。

示例：

```vue
<view
  data-qa="order.detail.page"
  :data-state="pageState"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
  :data-trace-id="traceId"
  :data-error-code="errorCode"
>
  ...
</view>
```

含义：

| 属性 | 含义 |
|---|---|
| `data-qa` | 页面或元素的稳定测试定位标识 |
| `data-state` | 当前页面状态，例如 `loading`、`ready`、`submitting`、`error` |
| `data-biz-type` | 当前主业务对象类型，例如 `order`、`payment`、`refund`、`member` |
| `data-biz-no` | 当前主业务对象外部业务编号，例如订单号、支付单号、退款单号 |
| `data-trace-id` | 当前页面关键请求链路 ID |
| `data-error-code` | 当前错误码，便于错误分类和证据关联 |

这些属性构成页面的“语义元数据”。普通 UI 只能告诉测试框架“这里有一个 view、一个 text、一个 button”；业务上下文标记可以告诉测试框架：

```text
这是订单详情页。
当前状态是 ready。
当前展示的是订单 ORD202607030001。
最近一次关键请求 traceId 是 trace_abc。
当前没有错误，或错误码是 ORDER_NOT_FOUND。
```

#### 17.5.2 放置位置

不同节点放不同粒度的上下文，不要把所有字段都塞到所有节点上。

##### 页面根节点

页面根节点表示整个页面的当前业务上下文。

推荐放：

```text
data-qa
data-state
data-biz-type
data-biz-no
data-trace-id
data-error-code
```

示例：

```vue
<view
  data-qa="order.detail.page"
  :data-state="pageState"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
  :data-trace-id="traceId"
  :data-error-code="errorCode"
>
  ...
</view>
```

##### 列表项节点

列表项节点表示每一行、每一张卡片、每一个可点击业务对象。

推荐放：

```text
data-qa
data-biz-type
data-biz-no
data-status
```

示例：

```vue
<view
  v-for="item in orders"
  :key="item.orderNo"
  data-qa="order.list.item"
  data-biz-type="order"
  :data-biz-no="item.orderNo"
  :data-status="item.status"
>
  ...
</view>
```

测试框架可以验证：

```text
列表中存在订单 ORD202607030001。
点击该列表项后，详情页 data-biz-no 仍然是 ORD202607030001。
```

##### 关键操作按钮

关键操作按钮表示这个按钮会对哪个对象执行什么动作。

推荐放：

```text
data-qa
data-action
data-biz-type
data-biz-no
```

示例：

```vue
<button
  data-qa="order.detail.pay"
  data-action="pay"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
  :disabled="pageState !== 'ready'"
  @tap="payOrder"
>
  去支付
</button>
```

这能让测试报告记录得更清楚：

```text
用户点击了 order.detail.pay。
动作是 pay。
操作对象是 order:ORD202607030001。
点击后页面进入 paying。
```

##### 弹窗和确认框

弹窗也应暴露语义，尤其是删除、退款、取消订单、确认支付等危险动作。

示例：

```vue
<view
  v-if="showRefundDialog"
  data-qa="refund.confirm.dialog"
  data-action="confirmRefund"
  data-biz-type="order"
  :data-biz-no="order.orderNo"
>
  ...
</view>
```

#### 17.5.3 主业务对象和关联业务对象

一个页面可能涉及多个业务对象，必须区分主对象和关联对象。

原则：

```text
data-biz-type + data-biz-no 表示当前节点的主业务对象。
关联对象使用明确命名的属性。
```

示例：退款详情页。

```vue
<view
  data-qa="refund.detail.page"
  :data-state="pageState"
  data-biz-type="refund"
  :data-biz-no="refund.refundNo"
  :data-order-no="refund.orderNo"
  :data-pay-no="refund.payNo"
  :data-trace-id="traceId"
>
  ...
</view>
```

含义：

```text
主对象：refund，编号 refund.refundNo。
关联订单：refund.orderNo。
关联支付单：refund.payNo。
```

常见页面主对象建议：

| 页面 | 主业务对象 |
|---|---|
| 订单详情页 | `order` |
| 订单确认页 | 提交前无主订单对象，提交后为 `order` |
| 支付页 | `payment` 或 `order`，按系统主流程统一 |
| 支付结果页 | `payment` |
| 退款页 | `refund` |
| 会员详情页 | `member` |
| 优惠券详情页 | `coupon` |
| 门店页 | `store` |

#### 17.5.4 外部编号优先，内部 ID 谨慎

业务上下文标记中的 `data-biz-no` 优先使用外部业务编号，而不是内部数据库 ID 或雪花 ID。

原因：

- 外部编号更适合人工识别、客服沟通和测试报告。
- 内部雪花 ID 对人没有语义。
- 直接暴露内部 ID 会增加前后端耦合。
- 某些内部 ID 可能带来枚举、越权排查或信息泄露风险。

推荐：

```text
订单：orderNo
支付：payNo
退款：refundNo
会员：memberNo
优惠券：couponNo 或 couponCode
门店：storeCode
```

不推荐优先使用：

```text
orderId = 982374982374982374
memberId = 982374982374982375
```

如果系统当前只有内部 ID，没有外部业务编号，应优先推动后端返回稳定外部编号。过渡期可以在测试包中受控暴露内部 ID，但不得把它作为长期页面契约。

过渡示例：

```vue
<view
  data-qa="order.detail.page"
  data-biz-type="order"
  :data-biz-no="order.orderNo || ''"
  :data-internal-id="isTestBuild ? order.orderId : ''"
>
  ...
</view>
```

#### 17.5.5 创建前没有业务编号时如何处理

很多页面在用户点击提交前还没有订单号、退款单号或支付单号。此时必须使用客户端请求标识或幂等标识串联链路。

推荐状态：

```text
提交前：clientRequestId / idempotencyKey
提交中：clientRequestId / traceId
提交成功：bizNo / traceId
提交失败：clientRequestId / traceId / errorCode
```

示例：

```vue
<view
  data-qa="order.confirm.page"
  :data-state="pageState"
  data-biz-type="order"
  :data-biz-no="orderNo || ''"
  :data-client-request-id="clientRequestId"
  :data-trace-id="traceId"
  :data-error-code="errorCode"
>
  ...
</view>
```

提交流程：

```text
ready
  生成 clientRequestId
  -> submitting
    请求携带 clientRequestId
    -> submitted
      后端返回 orderNo
```

测试证据链：

```text
点击提交
-> 记录 clientRequestId
-> 记录 traceId
-> 接口返回 orderNo
-> 页面暴露 data-biz-no
-> 测试框架用 orderNo 查询后端 evidence
```

如果接口超时，也可以用 `clientRequestId` 或 `traceId` 查询后端是否已经创建业务对象，避免重复提交导致脏数据。

#### 17.5.6 推荐命名

通用属性：

```text
data-qa
data-state
data-action
data-biz-type
data-biz-no
data-trace-id
data-request-id
data-client-request-id
data-error-code
```

关联对象属性：

```text
data-order-no
data-pay-no
data-refund-no
data-member-no
data-coupon-no
data-store-code
data-activity-no
```

状态属性：

```text
data-status
data-pay-status
data-refund-status
data-order-status
```

命名要求：

- 使用小写短横线命名。
- 不使用缩写到无法理解的名称。
- 不把内部字段名直接暴露成测试契约。
- 同一业务对象在全项目使用同一种命名。

#### 17.5.7 安全边界

可以暴露：

```text
订单号
支付单号
退款单号
会员编号
门店编码
优惠券编号
活动编号
traceId
requestId
clientRequestId
errorCode
非敏感业务状态
```

禁止暴露：

```text
token
session_key
AppSecret
openid
unionid
手机号明文
身份证号
完整地址
支付密钥
内部权限数据
密码或验证码
```

谨慎暴露：

```text
数据库自增 ID
雪花内部 ID
微信支付 transaction_id
用户内部账号 ID
员工 ID
```

正式包和测试包可以有不同策略：

```text
测试包：保留完整测试契约、traceId、clientRequestId、非敏感业务编号。
正式包：保留必要非敏感语义属性，剥离 testBridge、调试日志和不必要内部字段。
```

#### 17.5.8 判断标准

一个页面的业务上下文标记合格，应满足：

```text
页面根节点暴露 data-qa。
页面根节点暴露 data-state。
页面根节点暴露主业务对象类型 data-biz-type。
页面根节点优先暴露外部业务编号 data-biz-no。
复杂页面显式暴露关联业务编号，例如 data-order-no、data-pay-no。
提交前暴露 clientRequestId 或 requestId。
请求后暴露 traceId。
错误时暴露 errorCode。
列表项暴露每一项的业务类型、业务编号和状态。
关键操作按钮暴露 data-action 和操作对象。
不暴露 token、openid、手机号、身份证等敏感信息。
测试框架能用业务编号或 traceId 查询后端证据。
```

最终目标：

```text
data-state 证明前端当前处于什么状态。
data-biz-type / data-biz-no 证明页面展示的是哪个业务对象。
data-trace-id 证明是哪次请求导致当前结果。
后端 evidence 证明业务结果是否真实发生。
```

### 17.6 请求追踪规范

关键接口必须记录：

- actionName。
- url。
- method。
- requestId。
- traceId。
- request payload 脱敏摘要。
- response code。
- error message。
- startTime/endTime。

### 17.7 正式包剥离

测试辅助能力必须可通过构建环境控制：

```text
测试包：保留 data-qa、testBridge、request trace、pageState
正式包：保留不影响安全和性能的必要标识，剥离测试桥和调试输出
```

### 17.8 判断标准

| 检查项 | 合格标准 |
|---|---|
| 元素定位 | 关键元素有稳定 QA 标识 |
| 状态等待 | 页面有可等待状态 |
| 操作反馈 | 点击后页面状态立即可观察 |
| 防固定等待 | 测试可等待 `data-state`，不依赖固定 sleep |
| 业务上下文 | 页面根节点、列表项和关键按钮暴露必要非敏感业务语义属性 |
| 动作追踪 | 用户动作能关联接口请求 |
| 业务验证 | 页面结果能关联后端业务对象 |
| 构建隔离 | 测试能力不会泄漏敏感信息到正式包 |

## 18. 日志与问题定位规范

### 18.1 必须记录

- 页面进入。
- 页面参数。
- 关键点击。
- 关键接口请求和响应。
- 登录态变化。
- 授权结果。
- 支付、下单、退款等关键业务结果。
- 异常码和错误信息。

### 18.2 禁止记录

- token 明文。
- session_key。
- AppSecret。
- 完整手机号。
- 身份证号。
- 地址明文。
- 支付敏感数据。

### 18.3 判断标准

| 检查项 | 合格标准 |
|---|---|
| 可定位 | 失败时能知道停在哪个页面、哪个动作、哪个接口 |
| 可关联 | 前端 traceId 能关联后端日志 |
| 可脱敏 | 日志无敏感信息泄漏 |

## 19. 提测准入标准

开发提测前必须满足：

| 类别 | 准入标准 |
|---|---|
| 功能 | 主流程自测通过 |
| 页面 | 页面状态完整，不白屏 |
| 路由 | tabBar、普通页面、返回链路正确 |
| 接口 | 成功、失败、超时、登录失效都有处理 |
| 授权 | 同意和拒绝都测试过 |
| 数据 | 测试数据准备方式明确 |
| 自动化 | 关键元素有 QA 标识，页面有状态契约，页面根节点和关键业务节点有必要业务上下文标记 |
| 日志 | 关键动作和请求可追踪 |
| 兼容 | 至少通过开发者工具和一台真机 |
| 文档 | 影响范围、接口、测试点已说明 |

## 20. 开发自检清单

开发完成后逐项打勾：

```text
[ ] 页面已注册到 app.json 或分包配置
[ ] 页面路径、页面参数、返回路径符合预期
[ ] tabBar 页面和普通页面路由 API 使用正确
[ ] 页面具备 loading / empty / error / success 状态
[ ] 接口统一封装，判断 HTTP 状态码和业务码
[ ] 登录失效、授权拒绝、弱网、超时有兜底
[ ] 关键提交按钮有 submitting 和防重复提交
[ ] 关键业务操作有后端幂等保护
[ ] 页面根节点暴露 data-qa、data-state 和必要业务上下文
[ ] 主业务对象优先暴露外部业务编号 data-biz-no，而不是内部雪花 ID
[ ] 列表项和关键操作按钮暴露 data-biz-type、data-biz-no、data-action 等必要语义属性
[ ] 创建前无业务编号的提交类页面暴露 clientRequestId 或 requestId
[ ] 没有泄漏 AppSecret、session_key、token 明文
[ ] 没有在 data-* 属性中暴露 openid、unionid、手机号明文、身份证号、完整地址等敏感信息
[ ] 没有高频或全量 setData
[ ] 定时器、监听器、Socket、Worker 已清理
[ ] 新 API 已确认基础库兼容
[ ] 关键元素有稳定 data-qa
[ ] 页面状态可被自动化测试等待
[ ] 关键请求有 traceId 或 requestId
[ ] 真机主流程已自测
```

## 21. 规范等级判定

| 等级 | 判定标准 |
|---|---|
| A | 功能完整、状态完整、自动化契约完整、日志证据完整，可快速自动化回归 |
| B | 功能完整、状态基本完整、关键元素可定位，可人工测试和部分自动化 |
| C | 功能可用但状态、日志、测试契约不足，测试效率低 |
| D | 路由、接口、登录、安全或性能存在明显问题，不建议提测 |
