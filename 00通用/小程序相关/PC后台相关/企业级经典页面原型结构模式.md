# 企业级经典页面原型结构模式

## 1. 文档目的

企业系统、金蝶类 ERP、SaaS 后台、小程序管理页中，有一批非常经典的页面原型结构。

这些结构不是视觉风格，而是信息组织方式。它们决定：

```text
用户先看什么？
用户如何定位对象？
用户如何筛选、比较、批量处理？
用户如何从总览进入明细？
页面是否适合自动化测试？
页面能否承载企业级复杂业务？
```

本文总结企业级页面最常用的经典原型结构，特别补充“左树右表”这类高频结构。

## 2. 原型结构总览

| 结构模式 | 核心形态 | 最适合场景 |
|---|---|---|
| 左树右表 | 左侧分类树，右侧数据表 | 组织、分类、仓库、门店、部门、权限、物料分类 |
| 左树右详情 | 左侧树定位对象，右侧对象详情 | 组织架构、菜单权限、区域配置、分类详情 |
| 左列表右详情 | 左侧对象列表，右侧详情/编辑 | 消息、审批、工单、客户、会员、订单快速处理 |
| 上筛下表 | 顶部筛选区，下方表格 | 单据列表、订单列表、流水列表、报表明细 |
| 上摘要下明细 | 顶部指标摘要，下方明细列表 | 经营看板、库存预警、结算汇总、营销效果 |
| 主从表 | 主表一行对应下方从表 | 订单-明细、入库单-分录、对账单-流水 |
| 三栏结构 | 左导航，中列表，右详情 | 邮件、工单、审批中心、客服工作台 |
| 分步向导 | 按步骤完成复杂任务 | 开户、配置活动、创建营销规则、导入数据 |
| 工作台 | 指标、待办、快捷入口组合 | 管理首页、门店工作台、财务工作台 |
| 看板 | 按状态分列管理对象 | 线索、商机、任务、售后工单、订单履约 |
| 矩阵表 | 行列交叉表达规则 | 权限矩阵、价格表、排班表、库存分布 |
| 树表一体 | 表格行本身可展开为树 | 科目、物料 BOM、组织层级、菜单资源 |
| 图表钻取 | 图表总览，下钻到明细 | BI、销售分析、库存分析、财务分析 |
| 时间轴详情 | 按时间展示事件链 | 订单轨迹、审批记录、物流、售后处理 |
| 双栏穿梭 | 左可选，右已选 | 授权、分配人员、选择门店、选择商品 |
| 抽屉/弹窗辅助 | 主页面不离开，局部处理 | 快速编辑、查看日志、选择对象、轻量审批 |

## 3. 左树右表

### 3.1 结构定义

左树右表是企业系统最经典的结构之一。

```text
左侧：分类、组织、层级、目录、范围。
右侧：当前左侧节点下的数据列表。
```

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 页面标题                                      │
├──────────────┬───────────────────────────────┤
│ 左侧树        │ 查询条件 / 操作按钮             │
│              ├───────────────────────────────┤
│ 全部          │ 表格                           │
│ ├ 华东区      │ 编号  名称  状态  负责人  操作   │
│ │ ├ 上海      │ 001  门店A 启用  张三    编辑   │
│ │ └ 杭州      │ 002  门店B 停用  李四    查看   │
│ └ 华南区      │                               │
└──────────────┴───────────────────────────────┘
```

### 3.2 适用场景

左树右表适合有明确层级分类的数据：

```text
组织架构 -> 员工列表
部门树 -> 岗位列表
门店区域 -> 门店列表
商品分类 -> 商品列表
仓库分类 -> 库存列表
菜单树 -> 权限资源列表
客户分组 -> 客户列表
会员等级 -> 会员列表
科目树 -> 凭证明细
```

### 3.3 为什么它经典

左树右表解决的是“范围先行”的问题。

企业数据通常不是一张平铺列表，而是必须先确定范围：

```text
看哪个组织？
看哪个部门？
看哪个门店？
看哪个商品分类？
看哪个仓库？
看哪个权限节点？
```

用户先在左侧树确定业务范围，再在右侧表处理对象。这样比把所有范围都塞进筛选条件更直观。

### 3.4 设计标准

| 维度 | 标准 |
|---|---|
| 树节点 | 节点名称清晰，层级不宜过深 |
| 当前节点 | 必须有明显选中态 |
| 数据范围 | 右表必须明确展示当前树节点的数据范围 |
| 查询联动 | 切换树节点后，右表查询条件和分页要有明确规则 |
| 计数 | 树节点可展示数量，但数量必须口径清楚 |
| 权限 | 无权限节点要隐藏或禁用并说明 |
| 空态 | 当前节点无数据时，要说明是无数据还是无权限 |
| 批量操作 | 批量操作必须限制在当前树节点范围内 |

### 3.5 常见错误

```text
1. 左树选中了节点，但右表标题不显示当前范围。
2. 切换树节点后，右表筛选条件残留导致用户误判。
3. 树节点太深，用户展开成本过高。
4. 树节点和表格数据权限不一致。
5. 批量操作没有说明影响的是当前节点还是全部节点。
6. 小程序里照搬左树右表，导致屏幕拥挤。
```

### 3.6 小程序适配

小程序不适合直接做宽屏左树右表，应改成：

```text
顶部范围选择器 + 下方列表
侧边抽屉树 + 右侧列表
分级选择页 + 结果列表页
Tab 分类 + 列表
```

小程序文本原型：

```text
┌──────────────────────────────┐
│ 当前范围：华东区 / 上海  [切换] │
├──────────────────────────────┤
│ 搜索 | 状态筛选 | 更多          │
├──────────────────────────────┤
│ 门店A   启用   负责人 张三       │
│ 门店B   停用   负责人 李四       │
└──────────────────────────────┘
```

切换范围时弹出：

```text
┌──────────────────────────────┐
│ 选择范围                      │
├──────────────────────────────┤
│ 全部                          │
│ ├ 华东区                       │
│ │ ├ 上海                       │
│ │ └ 杭州                       │
│ └ 华南区                       │
└──────────────────────────────┘
```

### 3.7 测试关注点

```text
当前树节点 data-scope-id 是否可读取。
右表是否带 data-scope-id。
切换节点后是否重新加载列表。
分页是否重置。
筛选条件是否按契约保留或清空。
无权限节点是否不可操作。
批量操作是否只影响当前范围。
```

## 4. 左树右详情

### 4.1 结构定义

左树右详情用于管理层级对象本身，而不是管理节点下的数据列表。

文本原型：

```text
┌──────────────┬───────────────────────────────┐
│ 组织树        │ 组织详情                       │
│              │ 名称：上海分公司                │
│ 总部          │ 编码：ORG001                   │
│ ├ 华东区      │ 上级：华东区                    │
│ │ ├ 上海      │ 状态：启用                      │
│ │ └ 杭州      │ [新增下级] [编辑] [停用]         │
└──────────────┴───────────────────────────────┘
```

### 4.2 适用场景

```text
组织架构维护。
菜单资源维护。
商品分类维护。
区域层级维护。
权限资源维护。
科目体系维护。
```

### 4.3 设计重点

```text
左侧树负责定位对象。
右侧详情负责展示和编辑当前节点。
新增同级、新增下级、移动、停用必须有清晰入口。
删除或停用必须说明对子节点和关联数据的影响。
```

### 4.4 常见错误

```text
1. 用户不知道当前编辑的是哪个节点。
2. 新增同级和新增下级混淆。
3. 停用父节点对子节点影响不说明。
4. 拖拽移动后没有保存状态和失败恢复。
```

## 5. 左列表右详情

### 5.1 结构定义

左列表右详情用于快速浏览一组对象并处理当前对象。

文本原型：

```text
┌──────────────┬───────────────────────────────┐
│ 待办列表      │ 当前待办详情                    │
│              │ 单据号：BX001                  │
│ BX001 待审批  │ 金额：1000                     │
│ BX002 待审批  │ 申请人：张三                    │
│ BX003 已驳回  │ 审批意见：                      │
│              │ [同意] [驳回] [转交]             │
└──────────────┴───────────────────────────────┘
```

### 5.2 适用场景

```text
审批中心。
消息中心。
工单处理。
客服会话。
订单快速处理。
会员详情快速查看。
```

### 5.3 设计重点

```text
左侧列表要展示状态和优先级。
右侧详情要突出当前对象。
处理完成后要自动进入下一条或保持当前结果，规则必须明确。
列表状态与详情状态必须同步。
```

### 5.4 小程序适配

小程序通常拆成两页：

```text
列表页 -> 详情页 -> 处理结果 -> 返回列表刷新
```

也可以使用底部弹层做轻量详情：

```text
列表卡片点击 -> 半屏详情 -> 快速处理
```

## 6. 上筛下表

### 6.1 结构定义

上筛下表是最常见的数据检索结构。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 查询条件                                      │
│ 单据号 | 日期 | 状态 | 门店 | 客户 | [查询] [重置] │
├──────────────────────────────────────────────┤
│ 操作栏：新增 | 导出 | 批量审核                 │
├──────────────────────────────────────────────┤
│ 表格                                          │
│ 单据号  日期  客户  金额  状态  操作            │
└──────────────────────────────────────────────┘
```

### 6.2 适用场景

```text
订单列表。
单据列表。
流水列表。
库存列表。
会员列表。
优惠券列表。
结算记录。
对账明细。
```

### 6.3 设计标准

| 维度 | 标准 |
|---|---|
| 默认筛选 | 默认条件必须符合业务高频场景 |
| 高级筛选 | 低频条件折叠到更多筛选 |
| 查询结果 | 必须显示命中数量、分页、排序 |
| 重置规则 | 重置是回到默认条件，不一定是全部为空 |
| 操作栏 | 新增、导出、批量操作位置稳定 |
| 表格列 | 关键身份、状态、金额、日期优先 |
| 行操作 | 高频操作外露，低频操作收进更多 |

### 6.4 常见错误

```text
1. 筛选条件过多，占据首屏。
2. 默认查询范围太大，性能差。
3. 重置后变成全量查询，引发风险。
4. 状态筛选文案和表格状态文案不一致。
5. 导出没有说明导出当前页还是全部结果。
```

## 7. 主从表

### 7.1 结构定义

主从表用于表达“一个主对象对应多条明细”。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 主表：订单列表                                │
│ 订单号  客户  金额  状态                       │
│ SO001  客户A 1000 已审核                      │
├──────────────────────────────────────────────┤
│ 从表：订单明细                                │
│ 商品  数量  单价  金额                         │
│ A     2     100   200                         │
│ B     1     800   800                         │
└──────────────────────────────────────────────┘
```

### 7.2 适用场景

```text
订单和订单明细。
入库单和分录。
对账单和流水。
会员和消费记录。
门店和设备列表。
活动和参与商品。
```

### 7.3 设计重点

```text
主表选中行必须明显。
从表标题必须说明当前主对象。
主表切换时，从表加载状态必须可观察。
从表为空时要说明当前主对象无明细。
主表批量操作和从表行操作不能混淆。
```

## 8. 三栏结构

### 8.1 结构定义

三栏结构适合高频处理型工作台。

文本原型：

```text
┌──────────┬──────────────┬────────────────────┐
│ 分类/队列 │ 对象列表      │ 当前对象详情/操作    │
│ 待我处理  │ 工单001 高    │ 问题描述             │
│ 已超期    │ 工单002 中    │ 处理记录             │
│ 已完成    │ 工单003 低    │ [处理] [转派] [关闭] │
└──────────┴──────────────┴────────────────────┘
```

### 8.2 适用场景

```text
工单中心。
客服中心。
审批中心。
消息处理。
异常处理中心。
```

### 8.3 设计标准

```text
第一栏解决队列和范围。
第二栏解决对象选择和优先级。
第三栏解决详情判断和动作。
三栏之间必须有明确选中态。
完成当前对象后，下一对象选择规则必须明确。
```

## 9. 分步向导

### 9.1 结构定义

分步向导用于把复杂任务拆成连续步骤。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 步骤：1 基本信息 -> 2 规则配置 -> 3 预览 -> 4 发布 │
├──────────────────────────────────────────────┤
│ 当前步骤内容                                  │
├──────────────────────────────────────────────┤
│ [上一步] [保存草稿] [下一步]                  │
└──────────────────────────────────────────────┘
```

### 9.2 适用场景

```text
创建营销活动。
导入数据。
开户。
配置支付渠道。
新建审批流程。
发布小程序页面。
```

### 9.3 设计标准

```text
每一步只解决一个明确目标。
步骤之间有校验边界。
允许保存草稿。
最后一步必须有预览或确认。
失败时能回到具体步骤修复。
```

### 9.4 常见错误

```text
1. 步骤只是视觉拆分，业务校验仍然集中在最后。
2. 用户返回上一步后数据丢失。
3. 发布前没有总览确认。
4. 不支持草稿，长流程中断成本高。
```

## 10. 工作台

### 10.1 结构定义

工作台是角色任务入口，不是广告首页。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 今日待办 18 | 异常 3 | 超期 2 | 已完成 40       │
├──────────────────────────────────────────────┤
│ 快捷入口：新增订单 | 审批 | 盘点 | 对账          │
├──────────────────────────────────────────────┤
│ 我的待办                                      │
│ 单据号  类型  状态  截止时间                   │
├──────────────────────────────────────────────┤
│ 关键指标                                      │
│ 销售额 | 应收 | 库存预警 | 退款                 │
└──────────────────────────────────────────────┘
```

### 10.2 设计标准

```text
按角色配置内容。
待办和异常优先。
指标必须可下钻。
快捷入口必须服务高频任务。
不要把低频公告、宣传图占据核心区域。
```

## 11. 看板

### 11.1 结构定义

看板按状态列组织对象。

文本原型：

```text
┌──────────┬──────────┬──────────┬──────────┐
│ 待处理    │ 处理中    │ 待确认    │ 已完成    │
│ 订单001   │ 订单004   │ 订单007   │ 订单010   │
│ 订单002   │ 订单005   │          │ 订单011   │
└──────────┴──────────┴──────────┴──────────┘
```

### 11.2 适用场景

```text
线索阶段。
商机阶段。
任务状态。
售后工单。
订单履约。
配送状态。
```

### 11.3 设计标准

```text
列就是状态。
拖拽或移动必须等价于状态变更。
状态变更要有权限和校验。
每列数量、超期、异常要可见。
```

## 12. 矩阵表

### 12.1 结构定义

矩阵表用于表达行列交叉规则。

文本原型：

```text
┌──────────┬──────┬──────┬──────┐
│ 角色/权限 │ 查看  │ 新增  │ 删除  │
├──────────┼──────┼──────┼──────┤
│ 店长      │  √   │  √   │  ×   │
│ 收银员    │  √   │  ×   │  ×   │
│ 财务      │  √   │  ×   │  ×   │
└──────────┴──────┴──────┴──────┘
```

### 12.2 适用场景

```text
权限配置。
价格配置。
排班表。
库存分布。
门店商品配置。
渠道能力配置。
```

### 12.3 设计标准

```text
行列含义必须稳定。
批量勾选必须有范围提示。
冻结首行首列。
变更要有保存状态。
权限矩阵必须支持审计。
```

## 13. 树表一体

### 13.1 结构定义

树表一体是在表格中表达层级。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 名称                 类型     状态     操作    │
│ ▾ 商品管理            菜单     启用     编辑    │
│   ├ 商品列表          页面     启用     编辑    │
│   └ 商品新增          按钮     启用     编辑    │
│ ▸ 订单管理            菜单     启用     编辑    │
└──────────────────────────────────────────────┘
```

### 13.2 适用场景

```text
菜单资源。
权限点。
会计科目。
物料 BOM。
组织层级。
分类带属性。
```

### 13.3 左树右表与树表一体的区别

```text
左树右表：左侧树是范围，右侧表是范围内对象。
树表一体：表格行本身就是层级对象。
```

选择标准：

```text
如果用户先选范围，再处理数据，用左树右表。
如果用户直接维护层级对象，用树表一体。
```

## 14. 图表钻取

### 14.1 结构定义

图表钻取从指标总览进入明细。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 销售额趋势图 / 分类占比 / 门店排行             │
├──────────────────────────────────────────────┤
│ 点击某个指标后展示明细                         │
│ 门店  销售额  订单数  客单价                    │
└──────────────────────────────────────────────┘
```

### 14.2 设计标准

```text
指标口径必须清楚。
图表筛选和明细筛选必须一致。
点击图表后要显示当前钻取条件。
明细数据要能回溯到业务单据。
```

## 15. 时间轴详情

### 15.1 结构定义

时间轴详情按时间展示事件链。

文本原型：

```text
┌──────────────────────────────────────────────┐
│ 订单轨迹                                      │
│ 10:00 创建订单  张三                          │
│ 10:05 支付成功  微信支付                       │
│ 10:10 门店接单  店员A                         │
│ 10:30 配送完成  骑手B                         │
└──────────────────────────────────────────────┘
```

### 15.2 适用场景

```text
订单轨迹。
审批记录。
物流记录。
售后处理。
支付退款流水。
任务处理日志。
```

### 15.3 设计标准

```text
每个事件有时间、动作、人、结果。
关键失败事件要突出。
时间轴不是日志堆砌，必须服务业务追溯。
```

## 16. 双栏穿梭

### 16.1 结构定义

双栏穿梭用于从候选集合中选择目标集合。

文本原型：

```text
┌──────────────────┬────────┬──────────────────┐
│ 可选门店           │  >     │ 已选门店           │
│ 门店A             │  <     │ 门店C             │
│ 门店B             │  >>    │ 门店D             │
│ 门店C             │  <<    │                  │
└──────────────────┴────────┴──────────────────┘
```

### 16.2 适用场景

```text
分配门店。
分配人员。
选择商品。
授权角色。
配置活动参与范围。
```

### 16.3 设计标准

```text
左右两侧都要支持搜索。
已选数量必须可见。
批量移动必须有范围说明。
保存前要能预览影响范围。
```

## 17. 抽屉和弹窗辅助

### 17.1 结构定义

抽屉和弹窗用于不离开主页面完成轻量任务。

适合：

```text
查看日志。
快速编辑少数字段。
选择客户、商品、人员。
查看关联单据摘要。
轻量审批。
```

不适合：

```text
复杂长表单。
多步骤流程。
高风险操作。
需要大量上下文判断的任务。
```

### 17.2 设计标准

```text
弹窗只能承载一个明确任务。
弹窗标题必须说明正在处理什么对象。
关闭前要处理未保存变更。
弹窗内错误不能只显示在主页面。
```

## 18. 原型结构选择规则

### 18.1 先看用户任务

```text
用户是在选范围？优先左树右表。
用户是在维护层级对象？优先树表一体或左树右详情。
用户是在查找数据？优先上筛下表。
用户是在处理一个队列？优先左列表右详情或三栏结构。
用户是在录入复杂对象？优先表单页或分步向导。
用户是在看一个业务对象完整事实？优先详情页。
用户是在看指标并追明细？优先图表钻取。
用户是在按状态推进对象？优先看板。
用户是在分配范围？优先双栏穿梭。
```

### 18.2 再看数据关系

```text
树状层级：左树右表、左树右详情、树表一体。
主从关系：主从表、上主下从、详情页明细分录。
状态流转：看板、状态机详情页、审批页。
时间链路：时间轴详情。
指标到明细：图表钻取。
候选到已选：双栏穿梭。
```

### 18.3 最后看设备形态

PC 端适合：

```text
左树右表。
三栏结构。
主从表。
矩阵表。
宽表格。
复杂筛选。
```

小程序适合：

```text
范围选择器 + 列表。
卡片列表 + 详情页。
分步向导。
底部操作栏。
折叠分组。
Tab 分区。
轻量时间轴。
```

## 19. 小程序对经典结构的降维改造

小程序屏幕小，不能机械复制 PC 原型。经典结构要改成移动端结构。

| PC 经典结构 | 小程序改造 |
|---|---|
| 左树右表 | 范围选择器 + 列表，或抽屉树 + 列表 |
| 左树右详情 | 选择树页面 + 详情页 |
| 左列表右详情 | 列表页 + 详情页 |
| 上筛下表 | 顶部搜索 + 筛选抽屉 + 卡片列表 |
| 主从表 | 详情页主信息 + 明细卡片 |
| 三栏结构 | 队列 Tab + 列表 + 详情 |
| 矩阵表 | 分组配置页，必要时横向滚动但要谨慎 |
| 双栏穿梭 | 选择页 + 已选页 + 底部确认 |
| 图表钻取 | 指标卡片 + 趋势图 + 明细列表 |
| 时间轴详情 | 纵向时间线 |

## 20. 原型结构评价标准

判断一个页面原型是否选对结构，看 8 个问题：

```text
1. 用户的第一动作是什么，结构是否支持？
2. 页面主对象是什么，是否一眼可见？
3. 数据关系是树、表、主从、状态、时间还是指标？
4. 高频操作是否在最短路径上？
5. 风险操作是否被降级并确认？
6. 状态和范围是否可被用户感知？
7. 自动化测试能否稳定定位范围、对象、状态、动作？
8. 小程序端是否做了移动适配，而不是硬塞 PC 布局？
```

## 21. 补充思考：我的原型设计观点

以下是我对原型结构设计的一些个人思考和经验补充。

### 21.1 模式选择的"三问法"

在选择页面模式之前，我会先问三个问题：

**问题1：用户现在处于什么任务阶段？**

```text
探索阶段 -> 需要概览，引导用户找到入口
执行阶段 -> 需要高效操作，减少干扰
追溯阶段 -> 需要完整事实，清晰证据
审批阶段 -> 需要决策支持，风险可见
```

不同阶段对应不同模式：
- 探索阶段：工作台、图表钻取
- 执行阶段：上筛下表、左树右表、双栏穿梭
- 追溯阶段：时间轴详情、图表钻取
- 审批阶段：审批处理页、左列表右详情

**问题2：数据关系是什么？**

```text
一对多 -> 主从表
多对一 -> 左树右表
同层同质 -> 看板
层级异质 -> 树表一体
候选集合 -> 双栏穿梭
规则矩阵 -> 矩阵表
```

**问题3：用户是高频还是低频使用？**

```text
高频用户 -> 宽页面，减少跳转，一屏处理更多
低频用户 -> 深页面，减少认知，每步只做一件
```

### 21.2 模式组合的"主从原则"

当需要组合多个模式时，必须分清"主模式"和"辅模式"。

**主模式**：用户在这个页面最主要的目的是什么？
**辅模式**：为了完成主目的，用户需要哪些辅助功能？

```text
主：上筛下表（用户要查订单）
辅：批量操作 + 导入导出 + 详情抽屉

主：左树右表（用户要管理商品分类）
辅：新增/编辑表单 + 排序 + 导入导出

主：工作台（用户要看今天要做什么）
辅：指标卡 + 快捷入口 + 看板
```

辅模式不能喧宾夺主。如果辅模式占据了页面核心区域，说明模式选择有问题。

### 21.3 移动端适配的"降维不是减法"

小程序适配 PC 页面，不是简单地把"大布局"改成"小布局"，而是重新思考"在小屏幕下如何表达同样的业务关系"。

| PC 模式 | 小程序降维思路 | 常见错误 |
|---|---|---|
| 左树右表 | 范围选择器 + 列表 | 直接缩小左树，右表挤成一团 |
| 三栏结构 | Tab 分类 + 列表 + 详情页 | 试图在一个屏幕塞三个区域 |
| 矩阵表 | 分组配置页或横向滚动 | 把几十行几十列直接塞进手机 |
| 看板 | 状态 Tab + 列表 | 保留拖拽但不改变交互方式 |

**降维的核心思路**：
1. **按维度拆分**：多维信息按优先级分配到不同页面
2. **按阶段拆分**：复杂流程按步骤分配到不同页面
3. **按密度拆分**：高密度信息按分类折叠

### 21.4 原型的"可测试性"前置

好的原型设计，在画线框图之前就应该考虑测试契约。

**我的做法**：
1. 先定义页面的 `data-page-id` 和 `data-pattern`
2. 再定义表格行的 `data-row-id` 和 `data-status`
3. 再定义按钮的 `data-action-id` 和 `data-action-state`
4. 最后才画线框图

这样设计的页面天然可测试，不需要后期"打补丁"加测试属性。

### 21.5 原型评审的"六个问题"

评审一个原型时，我通常问这六个问题：

```text
1. 焦点问题：用户的视觉焦点在哪里？这个焦点是否符合业务优先级？
2. 路径问题：用户完成核心任务需要几步？每步之间的跳转是否清晰？
3. 状态问题：页面有多少种状态？每种状态之间的切换是否都有明确触发条件？
4. 异常问题：如果操作失败，错误信息是否具体？用户是否知道如何恢复？
5. 边界问题：如果数据量很大（比如 10000 条），原型是否能承载？
6. 测试问题：如果要自动化测试，我能从 DOM 中读取到足够的定位信息吗？
```

### 21.6 常见原型评审扣分项

以下是我在评审原型时经常扣分的点：

```text
扣 10 分：没有考虑空状态（没有数据、没有权限、没有查询结果）
扣 10 分：没有考虑加载状态（loading、submitting）
扣 10 分：批量操作没有说明范围（当前页？选中项？全部结果？）
扣 10 分：风险操作（删除、作废）没有二次确认
扣 5 分：表单提交失败后没有保留用户输入
扣 5 分：详情页/列表页/表单页之间跳转没有考虑返回路径
扣 5 分：分页信息不明确（当前页/总页数/总条数）
扣 5 分：表格列的优先级没有区分（重要信息被压缩在窄列里）
```

### 21.7 总结：好原型的三个标志

1. **选对模式**：根据数据关系和用户任务选择了正确的模式
2. **画对契约**：定义了稳定的页面契约（状态、参数、权限）
3. **测对场景**：考虑了空态、加载态、错误态等边界场景
