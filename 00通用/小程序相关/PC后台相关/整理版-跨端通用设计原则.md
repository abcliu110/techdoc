# 整理版：跨端通用设计原则

## 1. 文档定位

本文整理同时适用于小程序和 PC 后端管理界面的通用设计原则。

不管页面运行在：

```text
PC 后台。
ERP 管理端。
SaaS 管理端。
业务中台。
微信小程序。
移动 H5。
Taro / uni-app。
```

只要它是业务系统页面，就必须遵守本文原则。

本文不讨论具体终端布局，例如左树右表、tabBar、分包、宽表格、底部操作栏。这些放到专用文档。

## 2. 通用原则总览

| 原则 | 说明 |
|---|---|
| 业务对象优先 | 页面必须让用户知道正在操作什么对象 |
| 页面身份明确 | 页面必须有稳定 pageId、pageType、bizType |
| 入口来源明确 | 页面必须知道用户从哪里进入 |
| 参数契约明确 | 页面参数必须可校验、可降级、可追踪 |
| 状态机驱动 | 页面状态决定内容、按钮、字段和测试 |
| 业务上下文明确 | 页面必须暴露业务编号、业务状态、组织、来源 |
| 操作反馈可观察 | 点击后必须有 loading、success、fail、pending |
| 错误可恢复 | 错误要具体到字段、行、权限、状态或规则 |
| 权限可解释 | 不可见、不可点、不可导出、不可审批要有治理 |
| 证据可追溯 | 关键动作必须能追到后端证据 |
| 自动化可测试 | 测试不能依赖固定 sleep 和视觉猜测 |
| 发布可兼容 | 新旧入口、字段、状态、接口必须兼容 |

## 3. 业务对象优先

任何业务页面都不是孤立 UI。

页面必须明确：

```text
这个页面操作什么对象？
对象类型是什么？
对象编号是什么？
对象状态是什么？
对象属于哪个组织、门店、会员、客户或订单？
对象和哪些上下游有关？
```

典型业务对象：

```text
订单。
会员。
商品。
门店。
优惠券。
库存。
支付单。
退款单。
审批单。
报销单。
应收单。
应付单。
```

如果页面没有业务对象，测试和运维就无法定位问题。

## 4. 页面身份

页面必须有稳定身份。

推荐字段：

```text
pageId：页面唯一标识。
pageType：列表、详情、表单、审批、结果、工作台。
bizType：业务对象类型。
module：业务模块。
version：页面契约版本。
```

页面身份不是给用户看的，而是给：

```text
路由治理。
埋点。
日志。
自动化测试。
问题定位。
灰度发布。
权限治理。
```

## 5. 页面入口

同一个页面可能从多个入口进入。

例如订单详情：

```text
从订单列表进入。
从支付结果进入。
从消息通知进入。
从扫码进入。
从审批待办进入。
从报表钻取进入。
```

入口影响：

```text
默认视图。
返回路径。
可见操作。
参数要求。
错误处理。
埋点归因。
测试用例。
```

因此页面必须定义入口契约：

```ts
type PageEntry =
  | 'fromList'
  | 'fromSearch'
  | 'fromMessage'
  | 'fromApproval'
  | 'fromReport'
  | 'fromScan'
  | 'directUrl';
```

## 6. 参数契约

页面参数必须被当作接口契约处理。

必须定义：

```text
参数名。
参数类型。
是否必填。
默认值。
来源入口。
非法值处理。
缺失值处理。
是否影响权限。
是否影响查询范围。
```

错误做法：

```text
页面直接读取 query.id，不校验。
参数缺失就白屏。
参数错误就请求接口失败。
```

正确做法：

```text
先解析入口。
再校验参数。
再解析业务上下文。
再决定进入 loaded、notFound、noPermission、invalidParams 状态。
```

## 7. 页面状态机

好的业务页面应该像一个小型状态机。

通用状态：

```text
initial。
loading。
loaded。
empty。
submitting。
success。
failed。
noPermission。
notFound。
offline。
expired。
```

状态决定：

```text
显示什么内容。
哪些按钮可见。
哪些按钮可点。
哪些字段可编辑。
是否允许离开。
自动化测试等待什么。
```

不要让状态散落在多个 boolean 中：

```text
isLoading。
isSubmit。
isError。
isEmpty。
isDisabled。
```

更好的方式是定义明确状态：

```ts
type PageState =
  | 'loading'
  | 'loaded'
  | 'empty'
  | 'submitting'
  | 'submitSuccess'
  | 'submitFailed'
  | 'noPermission'
  | 'notFound';
```

## 8. 业务上下文

页面必须暴露业务上下文。

推荐：

```text
bizType：业务类型。
bizNo：外部业务编号。
bizId：内部 ID，必要时使用。
bizStatus：业务状态。
orgId：组织。
storeId：门店。
memberNo：会员号。
orderNo：订单号。
traceId：链路 ID。
entry：入口来源。
```

原则：

```text
给用户看的优先是外部业务编号。
给系统内部关联的可以是内部 ID。
给测试和日志的必须稳定可读。
```

## 9. 操作反馈

所有动作都必须可观察。

动作至少有：

```text
idle。
confirming。
submitting。
success。
failed。
retrying。
```

点击后必须发生可观察变化：

```text
按钮 loading。
页面 actionState 变化。
状态标签变化。
结果提示。
错误提示。
后端证据更新。
```

自动化测试应等待状态，而不是固定 sleep。

## 10. 错误恢复

错误必须结构化。

```ts
type PageError = {
  code: string;
  scope: 'page' | 'field' | 'row' | 'action' | 'permission' | 'network';
  field?: string;
  rowIndex?: number;
  message: string;
  recoverable: boolean;
  retryAction?: string;
};
```

错误必须告诉用户：

```text
哪里错了。
为什么错。
如何修复。
是否能重试。
重试会不会重复提交。
```

## 11. 权限和审计

权限不是简单隐藏按钮。

跨端通用权限包括：

```text
页面访问权限。
数据查看权限。
按钮操作权限。
字段编辑权限。
导出权限。
审批权限。
删除/作废等高风险权限。
```

关键动作必须留痕：

```text
谁操作。
什么时候操作。
操作了什么对象。
操作前后状态。
失败原因。
traceId。
```

## 12. 测试契约

页面要天然可测试。

推荐暴露：

```html
<div
  data-page-id="orderDetail"
  data-page-type="detail"
  data-biz-type="order"
  data-biz-no="SO202607030001"
  data-biz-status="paid"
  data-state="loaded"
  data-entry="fromList"
>
```

测试应围绕：

```text
页面身份。
入口。
参数。
状态。
业务编号。
业务状态。
操作结果。
错误码。
后端证据。
```

## 13. 配色通用原则

小程序和 PC 后台都应遵守语义配色。

通用原则：

```text
颜色表达语义，不做装饰。
品牌色用于主动作和选中态。
成功、警告、危险、禁用语义稳定。
风险色不能滥用。
状态不能只靠颜色表达，必须有文字。
测试不能靠颜色判断业务状态。
```

## 14. 跨端通用评价标准

一个页面如果跨端达到 A 级，必须满足：

```text
业务对象清楚。
页面身份清楚。
入口和参数清楚。
状态机清楚。
业务上下文可读取。
操作反馈可观察。
错误可恢复。
权限可治理。
证据可追溯。
自动化可测试。
```

## 15. 补充思考：我的跨端设计观点

以下是我对跨端设计的一些个人思考和经验补充。

### 15.1 跨端的本质是"业务抽象层"的复用

很多人认为跨端就是"一套代码多处运行"，但这只是技术层面的理解。

更本质的理解是：**跨端是"业务抽象层"的复用**。

业务抽象层包括：
- 业务对象模型（订单、会员、商品...）
- 业务状态定义（待支付、已支付、已完成...）
- 业务规则（满减规则、库存校验规则...）
- 页面契约（参数、状态、返回值...）

这些业务抽象层在 PC 和小程序之间应该是完全一致的，不应该有差异。

有差异的应该是：
- 布局（PC 宽屏 vs 小程序竖屏）
- 交互（鼠标 vs 触摸）
- 组件（Ant Design vs 小程序原生组件）

**我的核心观点**：如果你的 PC 端和小程序端的"业务状态定义"不一致，那跨端就是空谈。

### 15.2 "状态机"的三层设计

我在实践中发现，状态机应该分三层设计：

**第一层：加载状态（系统级）**
```ts
type LoadState = 'idle' | 'loading' | 'loaded' | 'error';
```
这层状态所有页面都有，关注的是"数据有没有加载成功"。

**第二层：业务状态（领域级）**
```ts
type OrderStatus = 'draft' | 'pending' | 'paid' | 'shipped' | 'completed' | 'cancelled';
type ApprovalStatus = 'pending' | 'approved' | 'rejected';
```
这层状态和具体业务相关，不同业务有不同的状态集合。

**第三层：操作状态（动作级）**
```ts
type ActionState = 'idle' | 'confirming' | 'submitting' | 'success' | 'failed';
```
这层状态关注的是"用户当前在做什么操作"。

**三层状态的联动**：
```
用户进入页面
  -> LoadState: loading
  -> API 返回
  -> LoadState: loaded
  -> OrderStatus: pending（从数据中读取）
  -> ActionState: idle（初始状态）

用户点击"支付"按钮
  -> ActionState: confirming（显示确认弹窗）
  -> 用户确认
  -> ActionState: submitting
  -> API 返回成功
  -> ActionState: success
  -> OrderStatus: paid（状态更新）
  -> ActionState: idle
```

### 15.3 "错误设计"的五个层次

好的错误设计应该分五个层次：

**第一层：错误预处理（防止错误发生）**
```text
表单校验 -> 提交前校验 -> 业务规则校验
```
在用户犯错之前就拦截。

**第二层：错误说明（帮助用户理解错误）**
```ts
{
  code: 'ORDER_STOCK_INSUFFICIENT',
  message: '库存不足',
  detail: '商品「可口可乐」当前库存 5 件，您需要 10 件',
  suggestion: '可以尝试：1. 减少购买数量 2. 等待补货后重试'
}
```
错误信息要具体到"什么商品"、"需要多少"、"当前多少"。

**第三层：错误恢复（让用户能修复）**
```text
参数错误 -> 自动填充 -> 用户修改 -> 重新提交
权限不足 -> 引导申请权限 -> 权限通过 -> 重试
网络错误 -> 自动重试 -> 重试失败 -> 手动重试入口
```

**第四层：错误兜底（防止系统崩溃）**
```text
未知的错误码 -> 显示通用错误 -> 记录日志 -> 通知研发
```

**第五层：错误追踪（帮助排查问题）**
```text
错误必须关联 traceId
错误必须记录用户操作路径
错误必须记录当时的业务上下文
```

### 15.4 "参数契约"的最小化原则

页面参数应该遵循"最小化暴露"原则：

**应该暴露的参数**：
- 业务 ID（bizId, orderNo, memberId...）
- 入口来源（entry: 'fromList' | 'fromScan' | 'fromMessage'）
- 业务上下文（orgId, storeId...）
- 视图模式（viewMode: 'detail' | 'edit'）

**不应该暴露的参数**：
- 数据库主键（id, pk, _id...）
- 内部状态标识（internalState, debugFlag...）
- 调试信息（debug_userId, test_mode...）

**我的原则**：参数名应该是业务语言，而不是技术语言。用户和测试都能读懂的参数名才是好的参数名。

### 15.5 "权限"的渐进式设计

权限设计应该采用"渐进式披露"：

```text
Step 1: 不可见
  -> 用户根本不知道有这个功能
  -> 适用于：高级功能、隐藏菜单

Step 2: 可见但不可操作
  -> 用户知道有这个功能，但点击后提示无权限
  -> 适用于：大部分业务权限

Step 3: 可见且可操作
  -> 用户可以正常使用功能
  -> 适用于：已授权用户

Step 4: 引导申请
  -> 用户知道功能存在，且有申请入口
  -> 适用于：可申请的权限
```

**为什么不是一刀切**：
- 如果所有无权限都隐藏，用户不知道系统能做什么 -> 无法提出权限申请
- 如果所有无权限都显示，用户会点击后被拒绝 -> 体验差
- 渐进式披露让用户知道系统能力，同时有申请路径

### 15.6 "入口来源"决定"返回路径"

同一个页面从不同入口进入，返回路径应该不同：

```ts
// 订单详情页
const getBackPath = (entry: string) => {
  switch (entry) {
    case 'fromList':
      return '/order/list'; // 返回列表
    case 'fromApproval':
      return '/approval/pending'; // 返回审批待办
    case 'fromScan':
      return undefined; // 扫码进入没有返回路径
    case 'fromMessage':
      return '/message/list'; // 返回消息列表
    default:
      return '/'; // 默认返回首页
  }
};
```

**我的经验**：入口来源不仅仅影响返回路径，还影响：
- 默认视图和展开状态
- 可见的操作按钮
- 埋点的归因
- 测试用例的设计

### 15.7 "可测试性"的三个检查点

在设计阶段就考虑测试，可以显著降低后期测试成本。

**检查点1：页面是否暴露了足够的 data-* 属性？**
```html
<div
  data-page-id="orderDetail"
  data-biz-no="SO202607120001"
  data-state="loaded"
>
```
测试必须能从 DOM 读取到页面状态，而不是依赖视觉判断。

**检查点2：状态变化是否通过 DOM 属性体现？**
```html
<!-- 提交前 -->
<button data-action-id="submit" data-action-state="idle">提交</button>

<!-- 提交中 -->
<button data-action-id="submit" data-action-state="submitting" disabled>提交中...</button>

<!-- 提交后 -->
<button data-action-id="submit" data-action-state="success" disabled>提交成功</button>
```
测试应该等待 `data-action-state` 变化，而不是等待固定时间。

**检查点3：表格行是否暴露了足够的业务信息？**
```html
<tr
  data-row-id="SO001"
  data-object-type="order"
  data-status="pending"
  data-amount="1000"
>
```
测试应该能从行属性读取到业务状态，而不是解析文本内容。

### 15.8 总结：好跨端设计的三个层次

1. **业务抽象层一致**：状态、规则、契约在 PC 和小程序之间完全一致
2. **表现层适配**：布局、交互、组件根据终端特性适配
3. **体验层优化**：根据终端特性做额外的体验优化（如小程序的分享、微信支付）

## 16. 最终结论

跨端通用设计原则是企业级业务系统的基础规范。

不管页面运行在 PC 后台、ERP、移动端还是小程序：

```text
业务对象必须清晰。
页面身份必须明确。
参数契约必须稳定。
状态机必须可观察。
错误必须可恢复。
权限必须可治理。
证据必须可追溯。
测试必须可自动化。
```

做到这些，跨端就不再是问题。
