# 数据归档与冷热分离模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

数据归档与冷热分离模型的本质，是为数据设计生命周期。不同阶段的数据服务不同目标：热数据服务在线交易，温数据服务低频查询，冷数据服务审计、合规和法律保留。

核心抽象：

```text
数据温度 -> 归档策略 -> 迁移批次 -> 校验删除 -> 查询路由 -> 回热 -> 销毁留痕
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| arch_policy 归档策略档案 | table_name(唯一), archive_predicate(终态+时间谓词), aggregate_tables(连带子表), hot_retention, warm_retention, legal_retention, destroy_policy(物理删除/匿名化), owner, status | 每张大表一份档案；"这张表留多久"建表评审即回答 |
| arch_batch 迁移批次 | 批次号(唯一), table_name, 条件范围, 行数, 校验和, status(COPIED/VERIFIED/DELETED/ROLLBACK) | 搬家的账本，永久保留；三步分离可安全重来 |
| arch_route_config 查询路由配置 | table_name, 温度分界线(时间), 热/温/冷目标数据源, 路由方案(双入口/应用层/联邦) | 按数据温度路由查询，归档对使用方显式或透明 |
| arch_reheat_task 回热任务 | 申请依据(诉讼/审计), 数据范围, 临时查询区, 期限, 审批人, 清理时间 | 冷数据调取走正式流程，用后清理留痕 |
| arch_destroy_log 销毁留痕 | 销毁批次号, 范围, 行数, 依据策略版本, destroy_policy, 执行时间, 审批人, 备份副本处理 | "证明数据已删"的合规证据，永久记录 |

### 2.2 实体关系

```text
arch_policy (1) ──< arch_batch (N)        一份策略驱动多次迁移批次
arch_policy (1) ──< arch_route_config (1) 温度线与保留期决定路由分界
arch_policy (1) ──< arch_destroy_log (N)  销毁引用策略版本作为依据
arch_batch (1) ──< arch_reheat_task (N)   回热按批次/范围调取冷数据
外部引用：table_name/aggregate_tables → 各业务域大表（订单/流水/审计日志）；
         迁移执行 → 批处理域 job_instance；owner/审批人 → 权限域用户
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：table_name 策略唯一；批次号唯一；
             归档端按源主键幂等写入，删除按批次范围条件删，重跑无害
归档条件约束：archive_predicate 必须是状态白名单（终态）+时间线双条件，
             禁止只按时间；聚合根与 aggregate_tables 同批次号整体迁移
状态机约束：批次状态 COPIED→VERIFIED→DELETED 只进不退，
           校验（行数+校验和）通过才允许删除在线端；失败可 ROLLBACK
留存/分级策略落点：legal_retention 压倒存储成本，到期才进可删池；
                 销毁按 destroy_policy 执行且覆盖备份副本轮转；
                 匿名化保骨架（个人字段抹除、统计骨架保留）
删除物理约束：小事务限速分块删；分区表用 DROP PARTITION 代替 DELETE
分片键：大表按时间 RANGE 分区（建表时前移归档设计）
```

## 3. 核心张力

最大张力是：

```text
在线库性能 vs 历史数据可查可信
```

历史数据会拖慢在线库，但企业又必须能查询、审计、诉讼、合规保留和按期销毁。

## 4. 不变量

- 归档条件必须是终态 + 时间，不能只按时间。
- 聚合数据必须整体迁移。
- 每张大表必须有归档策略档案。
- 迁移必须先落地、校验、再删除。
- 迁移批次必须永久可查。
- 查询必须按数据温度路由。
- 销毁必须按法规执行并留痕。

## 5. 为什么这样设计

终态 + 时间条件，是为了避免把仍在履约、退款、诉讼、审批中的历史数据移走。聚合整体迁移，是为了防止主表和明细表分居两地形成孤儿数据。

迁移批次留痕，是为了证明某批数据搬到了哪里、搬了多少、校验是否通过。

## 6. 失败模式

- 超过 6 个月就归档，结果把退款中的订单搬走。
- 主单归档了，明细还在线上，查询断链。
- 归档后没有查询路由，客服查不到历史订单。
- 大 DELETE 锁表，拖垮在线库。
- 数据到期删除但无销毁记录，无法证明合规。
- 备份中仍保留应删除数据。

## 7. 适用边界

适用：

- 订单、流水、审计日志、报表明细、支付记录、库存流水、消息记录等大表。

可简化：

- 小数据量系统可暂不拆冷热，但建表时应定义保留期和未来归档策略。

## 8. 设计取舍

优先保证：

- 在线库性能。
- 历史可查。
- 迁移可信。
- 合规保留和销毁。

牺牲：

- 查询简单性。
- 运维复杂度。
- 存储分层管理成本。

## 9. 同一模型的多业务形态建模

归档模型要同时支撑多种数据形态，核心方法是：**统一温度骨架 + 每表一份策略档案 + 差异下沉到迁移手段与保留规则**。

### 9.1 统一骨架（所有形态不变的部分）

```text
温度模型（热/温/冷）→ arch_policy（每张大表一份档案）→ 三步迁移（落地→校验→删除）→ 迁移批次留痕 → 查询按温度路由
不变量全集：终态+时间才归档、聚合整体迁移、先落地校验再删除、批次永久可查、销毁留痕
```

骨架决不为某类数据特化——任何形态的"这批数据搬去哪了、搬了多少、能不能查回来"都用同一套策略档案和批次记录回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 订单归档（聚合连带） | 主单+行+支付+履约必须同批走 | arch_policy 声明聚合根及连带表清单；条件=终态（完成/关闭）+N 月；按订单号整体迁移到归档库，同批次号（标准形态，骨架原型） |
| 流水分区归档 | 单表巨量、按时间自然切分 | 表按月 RANGE 分区；迁移=分区数据导出落地校验后 DROP PARTITION，避免大 DELETE 锁表；批次记录分区名和行数 |
| 审计日志冷存 | 只增不改、极少查询 | 热库只留近 N 月，其余压缩转对象存储（按月文件+哈希清单）；查询路由到冷存检索接口，接受分钟级延迟 |
| 日志类直写冷存 | 事后分析、无在线查询 | 不进热库：采集链路直写对象存储/日志系统，arch_policy 仍登记保留期与销毁规则；温度模型退化为只有冷层 |
| 合规长期保留（发票凭证） | 法定年限、到期必须销毁 | arch_policy 配法定保留期 + 禁提前删除；冷存带 WORM（不可改写）属性；到期销毁走审批并写销毁留痕，覆盖备份副本 |

### 9.3 建模决策规则

- **能用 arch_policy 参数解决的，不建新迁移链路**：差异只在条件、目的地、保留期时，三步迁移和批次留痕一套。
- **迁移手段按表的物理形态垂直选择**（逐批 DELETE / DROP PARTITION / 直写冷存），选择写进策略档案，不隐藏在脚本里。
- **三步迁移允许按形态裁剪但不新增语义**：直写冷存跳过"从热库删除"一步，但不得跳过保留期登记和销毁留痕。
- **拆独立域的信号**：当"历史数据"需要重新计算和多维分析（而不只是查得到），那是数仓域的事——归档保证原始可信，分析交给数仓。
