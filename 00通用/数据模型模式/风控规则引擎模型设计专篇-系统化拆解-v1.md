# 风控规则引擎模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

风控规则引擎模型的本质，是对业务事件进行风险判断、记录依据、执行处置并支持事后重放的决策系统。

核心抽象：

```text
风控事件 -> 特征 -> 规则/策略/模型 -> 决策记录 -> 名单 -> 处置/申诉
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| rc_event 风控事件 | event_type, event_time, user_id, device_id, ip, ip_region, biz_order_no, amount, ext(JSON) | 风控的 ODS，只增不改；环境要素接入时强制采集 |
| rc_feature_def 特征定义 | 特征编码(唯一), 口径描述, 计算窗口, 数据来源, 负责人, status | 注册制口径字典，防"同名不同数"误判 |
| rc_rule 规则 | rule_code, 条件表达式, 分数/动作, version, status(试运行/生效/停用), 生效时间 | 版本化判断逻辑；shadow mode 只记分不动作 |
| rc_policy 策略 | scene, 规则集, 组合方式(评分累加/最坏单条), 阈值动作映射, policy_version, 降级预案(放行/拒绝) | 场景级编排；降级是显式配置不是 catch 块 |
| rc_decision 决策记录 | event_id, scene, policy_version, hit_rules(JSON), feature_snapshot(JSON), score, decision(PASS/CHALLENGE/REVIEW/REJECT), final_result | 版本+命中+快照三件套，任何决策可重放 |
| rc_list 名单条目 | 名单类型(黑/白/灰), 维度(用户/设备/IP/地址), 值, 来源(规则/案件/人工), 有效期, 操作人 | 入单有来源、出单有记录，默认带有效期 |
| rc_case 案件 | decision_id, 上下文聚合, 审核结论(通过/拒绝/加黑), 审核人, 结论时间 | REVIEW 决策的人工闭环；结论回填并成为规则效果标注 |
| rc_appeal 申诉单 | 原 decision_id/处置单据号, 申诉理由, 复核结论, 是否误报标记 | 误报救济通道，误报率的数据来源 |
| rc_rule_stat 规则命中统计 | rule_id, version, 统计周期, 命中量, 拦截量, 误报/漏报数 | 规则效果度量，无度量不上线、不敢删的规则由此清理 |
| rc_ext_call 外部调用留痕 | 调用方特征/规则, 三方接口, 请求/返回摘要, 费用, 时间 | 外部数据成本核算 + 合规留痕 |

### 2.2 实体关系

```text
rc_event (1) ──< rc_decision (N)        一个事件可经多场景决策
rc_policy (1) ──< rc_rule (N)           策略按场景编排规则集（版本对版本）
rc_decision (1) ──- rc_case (0..1)      REVIEW 决策落人工案件
rc_decision (1) ──< rc_list (N)         决策/案件结论生成名单条目（来源可追）
rc_decision (1) ──- rc_appeal (0..1)    申诉引用原决策，复核结论回填
rc_rule (1) ──< rc_rule_stat (N)        按版本×周期累计命中效果
rc_feature_def (1) ──< 特征取值          实时值在 KV，决策时刻快照固化进 rc_decision
外部引用：user_id → 会员域；biz_order_no → 订单/支付域；
         处置动作 → 各业务域正式单据（冻结账户/冻券/取消订单），
         引用不落外键，风控不直改业务表
```

### 2.3 关键约束与不变量落点

```text
只增不改：rc_event 无 UPDATE 路径；决策记录 final_result 为唯一可回填字段
版本固化：rc_decision 必存 policy_version + hit_rules(含 rule version) + feature_snapshot，
         三样缺一决策不可重放，落库校验非空
唯一键：特征编码唯一；(名单类型, 维度, 值, 场景范围) 唯一防重复入单
有效期约束：名单条目默认非永久，查询按 有效期 > now() 过滤
状态机：规则 试运行(shadow)→生效→停用，只进不退；决策同步产出、记录异步落库不可丢(Outbox)
分片键：rc_event 按 user_id/event_time；名单查询走布隆过滤器+KV 缓存，DB 为权威源
```

## 3. 核心张力

最大张力是：

```text
风险拦截强度 vs 正常用户体验
```

风控拦得越严，资损越少但误伤越多；放得越松，体验更好但黑产更容易穿透。

另一个张力：

```text
实时决策性能 vs 决策可解释可回放
```

决策链路必须快，但每一次拒绝、挑战、人工审核都必须能解释依据。

## 4. 不变量

- 风控事件只增不改。
- 事件接入必须有 schema 和关键环境要素。
- 特征必须注册口径。
- 规则、策略、模型必须版本化。
- 决策记录必须保存命中规则、特征快照和策略版本。
- 名单必须有来源和有效期。
- 风控处置必须走业务域正式接口或单据。

## 5. 为什么这样设计

事件是风控的事实源，特征是对事件的聚合画像，规则和模型是判断逻辑，决策记录是可解释证据，名单和处置是风险结果沉淀。

决策快照是为了事后能回答“当时为什么拦截这个用户”。没有快照，特征变化后无法重放历史判断。

## 6. 失败模式

- 事件缺设备、IP、渠道等关键字段，黑产识别无基础。
- 特征没有口径，同名指标不同数。
- 规则直接上线全量拦截，误伤用户。
- 决策只存 PASS/REJECT，不存命中依据。
- 黑名单无有效期，正常用户长期被误伤。
- 风控系统直接改业务表，造成跨域耦合和审计缺失。

## 7. 适用边界

适用：

- 登录、注册、领券、下单、支付、提现、退款、营销、防刷、防薅羊毛。

可简化：

- 低风险内部系统可使用简单规则，但只要涉及资金、优惠、账号安全，就应保留事件、规则版本和决策记录。

## 8. 设计取舍

优先保证：

- 风险可控。
- 决策可解释。
- 规则可灰度。
- 误伤可申诉。

牺牲：

- 实时链路复杂度。
- 规则治理成本。
- 特征计算成本。

## 9. 同一模型的多业务形态建模

风控模型要同时支撑多种风控形态，核心方法是：**统一五件套骨架 + 场景（scene）标识 + 特征与处置按场景配置**。

### 9.1 统一骨架（所有业态不变的部分）

```text
事件（event_type + 环境要素）→ 特征（注册制口径）→ 规则/策略（scene × 版本化）→ 决策记录（版本+命中+特征快照）→ 名单（来源+有效期）→ 处置/申诉
不变量全集：事件只增不改、特征口径注册、规则版本化灰度、决策快照可重放、处置走业务正门
```

骨架决不为某个场景特化——任何场景的"发生了什么、怎么判的、为什么拦、结果沉淀到哪"都用同一套五件套回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 营销反薅羊毛 | 团伙聚集、设备/地址关联 | scene=COUPON_CLAIM，特征以关联图为主（设备/收货地址关联账号数）；处置=批量冻券单据 + 设备黑名单（带有效期），降级预案配置为放行+事后批量追回 |
| 支付反欺诈 | 毫秒级同步决策、资损直接 | scene=PAY，同步链路 p99 预算最严，特征走 KV 预计算；决策动作含 CHALLENGE（3DS/短信验证），final_result 由挑战结果回填；降级预案=放行+事后审 |
| 账号安全（盗号） | 行为突变检测、保护性处置 | 特征以状态突变为主（异地登录、设备更换、改密改绑序列）；处置=保护性冻结账户+二次验证解冻，申诉通道权重最高（误伤即锁真人） |
| 内容风控 | 判断对象是内容不是交易 | 事件 ext 挂内容指纹/文本特征，机器审+人工审两级：REVIEW 决策落人工审核队列，结论回填并作为模型标注数据；名单维度扩展"内容指纹" |
| 信贷风控 | 一次决策长期生效、监管强合规 | 决策记录升级为授信快照（额度/利率/依据特征全量固化，留存年限按监管）；外部征信调用留痕计费是硬要求；贷中走定时批量重评估事件 |
| 商家风控 | 主体是商家、周期性巡检 | 事件维度从 user_id 扩展 merchant_id；特征多为离线批计算（刷单率/投诉率/资金异动）；处置=降权/冻结提现/清退，全部走商家域单据并可申诉 |

### 9.3 建模决策规则

- **能用 scene + 策略配置解决的，不建第二套引擎**：差异在特征集、阈值动作、降级预案时，全部是 policy 配置，五件套一套。
- **决策主体维度可扩展（用户/设备/商家/内容），事件表和名单维度跟随扩展，机制不变**：名单永远是 类型×维度×来源×有效期。
- **同步/异步是链路差异不是模型差异**：支付同步、商家巡检批量，都产出同构的决策记录。
- **拆独立域的信号**：当决策结果本身是长期金融契约（授信额度、保单核保）需要独立生命周期管理时，风控只提供决策，契约归信贷/保险域。
