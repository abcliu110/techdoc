# 开放平台与 OpenAPI 模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦开放平台建模。当系统要对外提供 API（ISV 接入、商家自研对接、渠道集成），就多出一个"半可信的外部开发者世界"：应用身份、授权、签名、配额、回调、版本。开放平台的建模难点：权限要按应用×商家双维度收敛，回调要按外部渠道的谦卑纪律对待，每一次调用都要可计量可审计。

## 0. 一句话总纲

> 开放平台模型四件套：应用（谁在调）、授权（替谁调、能调什么）、调用契约（怎么调：签名/限流/版本）、回调订阅（我怎么通知你）。核心思想：外部开发者既是客户也是风险源，能力给出去，控制留在手里。

## 1. 应用与凭证：外部调用方的身份

```sql
CREATE TABLE open_app (
  id           BIGINT PRIMARY KEY,
  app_key      VARCHAR(32) NOT NULL,
  app_name     VARCHAR(128) NOT NULL,
  developer_id BIGINT NOT NULL,        -- 开发者主体（企业实名）
  app_type     VARCHAR(16) NOT NULL,   -- SELF_USE自用/ISV第三方服务商
  status       VARCHAR(16) NOT NULL,   -- DEVELOPING/AUDITING/ONLINE/SUSPENDED/OFFLINE
  UNIQUE KEY uk_key (app_key)
);

CREATE TABLE open_app_secret (
  id          BIGINT PRIMARY KEY,
  app_id      BIGINT NOT NULL,
  secret_hash VARCHAR(128) NOT NULL,   -- 密钥不明文存储
  status      VARCHAR(16) NOT NULL,    -- ACTIVE/DEPRECATED/REVOKED
  created_at  DATETIME NOT NULL,
  expire_at   DATETIME,
  KEY idx_app (app_id)
);
```

- **开发者 → 应用 → 凭证三层**：一个开发者多个应用，一个应用多个密钥——密钥表独立且**支持多密钥并存**，是密钥轮换（新旧密钥重叠期）的前提；泄露时 REVOKED 单个密钥而不是换 app_key（换 key 等于让所有对接方重新开发）。
- 应用生命周期状态机：开发中（沙箱）→ 审核 → 上线 → 暂停/下线，SUSPENDED 即时切断调用但保留配置（与租户暂停同构）。
- **沙箱环境是模型的一部分**：沙箱应用/沙箱数据隔离标记，让开发者不用真实数据联调。

## 2. 授权：应用 × 商家 × 权限包

自用型应用调自己的数据，ISV 型应用**替商家**调数据——授权是三方关系：

```text
权限包（scope）：API 按业务域打包（订单读/订单写/商品管理/结算查询），带敏感分级
应用申请权限包 → 平台审核（敏感包人工审）
商家授权应用（OAuth）：商家 × 应用 × 权限包 × 授权时间 × 有效期 → access_token
```

```sql
CREATE TABLE open_authorization (
  id          BIGINT PRIMARY KEY,
  app_id      BIGINT NOT NULL,
  merchant_id BIGINT NOT NULL,         -- 授权主体（商家/租户）
  scopes      JSON NOT NULL,           -- 授权的权限包快照
  status      VARCHAR(16) NOT NULL,    -- ACTIVE/EXPIRED/REVOKED商家解除
  granted_at  DATETIME NOT NULL,
  expire_at   DATETIME NOT NULL,
  UNIQUE KEY uk_app_merchant (app_id, merchant_id)
);
```

- **每次 API 调用双重校验**：应用有此 scope + 商家授权了此 scope 且未过期——应用能力上限和商家授权范围取交集。
- 授权/解除授权全部留痕（授权历史只增不改）："这个 ISV 凭什么拉走了我三个月订单"必须可答。
- **数据隔离等价于多租户纪律**：ISV 调用只能取到授权商家的数据，merchant_id 由 token 解析注入上下文，不由请求参数传入（多租户篇 tenant_id 同款铁律——参数传商家 ID 的开放平台迟早出越权事故）。

## 3. 调用契约：签名、幂等、限流

- **签名**：参数排序 + 时间戳 + nonce + 密钥签名（HMAC/RSA），时间戳窗口（±5 分钟）+ nonce 去重防重放。验签失败计入应用风险档案。
- **写接口强制幂等**：外部请求带业务幂等键（对接方生成），落唯一索引——网络超时后对接方一定会重试，没有幂等约定的开放写接口等于邀请重复下单。
- **限流配额分层**：平台级（保护系统）→ 应用级（按套餐/等级：QPS + 日调用量）→ 敏感接口单独限——配额是应用的商业属性（付费提额），配额定义与用量计量按多租户篇 §6 同款模型（计数器检查 + 流水结算）。
- **调用日志**：每次调用记录（应用、商家、接口、耗时、结果码、trace_id），是计量、排障、审计三合一的数据源；量大分级留存（成功聚合统计 + 失败逐条，参考通知篇分级留痕）。

## 4. 回调与消息推送:反向通道

平台主动通知对接方（订单状态变了、退款发生了），角色互换——我们成了"渠道"，对接方成了"商户"：

```text
订阅配置：应用 × 事件类型 × 回调地址（地址要验证所有权）× 签名密钥
推送任务：事件 → 按订阅展开推送任务（幂等键）→ 重试（退避策略）→ 死信
```

- **推送必重试 + 必有死信**：对接方服务挂了是常态，指数退避重试 N 次后进死信，提供**主动拉取补偿接口**（对接方按时间段拉漏掉的事件）——推 + 拉双通道是开放平台消息可靠性的标准答案。
- 推送内容**只带标识不带敏感明细**（订单号 + 变更类型，详情让对方凭 token 反查），降低回调泄露的影响面。
- 对接方处理成功的判定：HTTP 200 + 约定应答体——"应答了才算送达"，与支付回调纪律互为镜像。
- 推送记录留存（何时推的、几次、对方应答什么），客服处理"你们没通知我"的对接纠纷全靠它。

## 5. 版本与废弃:对外契约的生命周期

- **API 版本显式化**（路径或参数），版本是**契约不是代码分支**：字段只加不改不删（加字段兼容），破坏性变更 = 新版本。
- **废弃流程建模**：版本状态（ACTIVE → DEPRECATED（保留期，返回废弃警告头）→ RETIRED），废弃公告 + 剩余调用方清单（从调用日志统计谁还在用旧版）+ 定向通知——"直接下线旧版"等于单方面撕毁合同。
- 字段级变更记录变更日志（changelog 对外发布），沙箱先行新版本。

## 6. 开放平台评审清单

```text
1. 开发者/应用/密钥是否三层？密钥是否多版本并存支持轮换、可单独吊销？
2. ISV 授权是否应用×商家×scope 三方建模、授权历史留痕？
3. 调用是否双重校验（应用 scope ∩ 商家授权）？merchant_id 是否由 token 注入而非参数？
4. 签名是否防重放？写接口是否强制业务幂等键？
5. 限流配额是否分层、与计量计费打通？
6. 调用日志是否支撑计量/排障/审计？是否分级留存？
7. 回调是否重试+死信+主动拉取补偿？推送是否只带标识不带敏感明细？
8. API 版本是否只加不破坏？废弃是否有保留期、存量调用方清单和定向通知？
9. 沙箱是否隔离？应用暂停能否即时切断？
```

## 7. 总结

开放平台建模五条纪律：

1. **身份三层**：开发者、应用、密钥分离，轮换吊销不伤对接。
2. **授权三方**：应用能力 ∩ 商家授权，主体标识由 token 注入。
3. **契约刚性**：签名防重放、写必幂等、配额即商业条款。
4. **推拉双保**：回调重试进死信，拉取接口兜底，记录可对质。
5. **版本即承诺**：只加不破坏，废弃走流程，旧版调用方看得见。

> 开放平台把系统的能力变成别人的依赖。每开放一个接口，就多一份"不能随便改"的承诺——模型的严谨程度，决定这些承诺十年后还能不能兑现。
