# 文件与附件存储模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

文件与附件存储模型的本质，是二进制内容、文件元数据和业务引用关系的分离治理。文件本体存对象存储，文件档案存数据库，业务对象通过引用关系使用文件。

核心抽象：

```text
对象存储 -> 文件元数据 -> 业务引用 -> 访问授权 -> 生命周期策略
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| sys_file 文件元数据 | file_key(对象存储 key，不可猜测), content_hash(SHA-256), size_bytes, mime_type, storage(集群/区域), status(UPLOADING/ACTIVE/ARCHIVED/FROZEN/DELETED), uploaded_by, tenant_id | 一行一个文件的档案；哈希去重与完整性校验的落点 |
| sys_file_ref 业务引用 | file_id, biz_type(SPU_IMAGE/CONTRACT/INVOICE...), biz_id, purpose(主图/正本), sort_order, deleted_at(软删) | "谁在用"的唯一依据；删除决策查它不靠猜 |
| sys_file_access_log 访问日志 | file_id, 访问人, 访问时间, 访问方式(签名 URL/下载), 业务上下文 | 敏感文件"谁看过"的留痕 |
| sys_file_lifecycle 生命周期策略 | biz_type, 保护期, 法定保留年限, TTL, 归档层级规则 | 删除是策略不是操作；法定保留优先于零引用 |
| sys_file_delete_log 删除流水 | file_id, content_hash, 触发策略, 删除时间, 二次校验结果 | 真删留证据，"为什么被删了"可答 |
| sys_file_derived 固化派生文件 | 源 file_id, 派生 file_id, 处理参数, 出具场景 | 仅登记出具即证据的派生（盖章 PDF），可重算的不入册 |

### 2.2 实体关系

```text
sys_file (1) ──< sys_file_ref (N)         一个文件被多处业务引用
sys_file (1) ──< sys_file_access_log (N)  敏感文件逐次访问留痕
sys_file (1) ──< sys_file_delete_log (1)  真删一次，留哈希与策略依据
sys_file (1) ──< sys_file_derived (N)     源文件-固化派生文件关联
sys_file_lifecycle (1) ──< sys_file (N)   策略按 biz_type 作用于文件回收
外部引用：biz_type+biz_id → 各业务域单据；uploaded_by/访问人 → 权限域用户；
         tenant_id → 租户域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：(content_hash, tenant_id) 租户内去重唯一；
             file_key 含随机段不可猜测，禁止自增 ID 当 key
版本/固化：文件本体不可变，重传即新文件新哈希；
          出具类派生文件（盖章 PDF）独立登记并关联源文件
状态机约束：UPLOADING→ACTIVE 由确认/回调驱动（哈希+大小校验），
           超时 UPLOADING 由清理任务收敛（对象存储侧同步清碎片）；
           安全扫描不过转 FROZEN；状态只进不退
删除/留存策略落点：业务删除只软删 sys_file_ref；
                 零引用+超保护期+满足法定保留期才进可删池，删前二次校验引用；
                 冷热分层按最近访问时间转 ARCHIVED，元数据记当前层级
访问约束：私有读默认，URL 短时效签发（分钟级）；敏感 biz_type 强制写访问日志
分片键：tenant_id（租户隔离主路径）
```

## 3. 核心张力

最大张力是：

```text
文件使用便利性 vs 权限、安全和生命周期治理
```

业务希望随处上传、随处引用、直接访问；系统必须保证文件不乱删、不泄露、可迁移、可审计、可归档。

## 4. 不变量

- 二进制、元数据、业务引用必须分离。
- 业务表不直接存永久 URL。
- 文件本体以哈希去重和校验。
- 删除由引用和保留期驱动。
- 私有读是默认。
- 访问 URL 必须短时效授权。
- 敏感文件访问必须留痕。

## 5. 为什么这样设计

三层分离是为了避免文件 URL 散落在业务表里，导致迁移、删除、权限和引用关系不可控。引用表决定文件是否还被使用，生命周期策略决定能否归档或删除。

## 6. 失败模式

- 业务表直接存 URL，更换存储域名要全库刷数据。
- 文件无人引用但不敢删，存储无限增长。
- 文件有引用却被误删，历史合同或发票丢失。
- 私有文件使用长期外链，泄露后无法控制。
- 上传未确认形成孤儿文件。

## 7. 适用边界

适用：

- 商品图片、合同、发票、审批附件、报表导出、证件、用户上传文件。

可简化：

- 纯公开静态资源可简化访问控制，但仍建议保留元数据和引用关系。

## 8. 设计取舍

优先保证：

- 权限安全。
- 文件可追溯。
- 可迁移。
- 可删除治理。

牺牲：

- 上传流程复杂度。
- 元数据维护成本。
- 授权访问开销。

## 9. 同一模型的多业务形态建模

文件模型要同时支撑多种文件形态，核心方法是：**统一三层骨架 + 文件类别标识 + 差异下沉到访问策略与生命周期配置**。

### 9.1 统一骨架（所有形态不变的部分）

```text
对象存储 → sys_file（file_category 标识形态 + 内容哈希）→ sys_file_ref（业务引用）→ 访问授权 → 生命周期策略
不变量全集：三层分离、不存永久 URL、哈希去重校验、引用驱动删除、私有读默认、访问留痕
```

骨架决不为某类文件特化——任何形态的"文件是谁的、被谁引用、谁能看、什么时候能删"都用同一套表回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 商品图片（公开 CDN） | 公开可读、多尺寸、高频访问 | file_category=public_asset，访问策略配置为公开读 + CDN 域名；衍生尺寸作为子文件记录挂原图 file_id，哈希各自独立 |
| 合同证照（私密合规） | 强权限、长保留、访问留痕 | 私有读 + 短时签名 URL（分钟级）；sys_file_ref 挂合同单据；生命周期策略配保留期 N 年禁删；每次访问写审计 |
| 报表导出（临时文件） | 生成即用、短命 | 生命周期策略配 TTL（如 7 天）自动清理；引用挂导出任务记录，任务删除即引用归零可回收 |
| 用户上传 UGC（安全扫描） | 内容风险、先审后可见 | sys_file 加 scan_status 状态（pending/pass/reject）；扫描通过前访问授权拒绝下发 URL；reject 文件保留证据不物理删 |
| 发票票面（出具固化） | 出具后不可变、法定保留 | 哈希写入业务单据作为固化凭证，重传视为新文件新哈希；生命周期按税法保留期配置；删除必须走销毁留痕 |

### 9.3 建模决策规则

- **能用 file_category + 访问策略 + 生命周期配置解决的，不建新文件表**：差异只在可见性、时效、保留期时，元数据和引用链路一套。
- **策略按维度垂直配置**（访问策略 / 扫描策略 / 保留策略各自独立），不在 sys_file 堆业务字段；业务语义放引用方单据。
- **访问流程允许按形态裁剪但不新增语义**：公开图片跳过签名 URL，但删除仍必须走引用计数，不得直删。
- **拆独立域的信号**：当文件内容本身需要结构化解析和检索（文档协作、知识库），那是内容域，文件模型只管它的二进制载体。
