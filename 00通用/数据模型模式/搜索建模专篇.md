# 搜索建模专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦搜索侧建模。搜索引擎（ES/OpenSearch 等）是最典型的"派生副本"（总览篇 §38）：DB 是权威源，索引是为查询而重塑的投影。本篇讲的不是搜索算法，而是**数据模型如何喂养搜索**：文档结构设计、同步链路、一致性边界、以及搜索运营数据的回流。

## 0. 一句话总纲

> 搜索建模三件事：把规范化的 DB 模型"打宽"成面向查询的文档、让同步链路可靠且可重建、承认并管理索引与 DB 之间的最终一致延迟。

## 1. 索引文档设计：为查询建模，不是为存储建模

DB 按业务域规范化（商品/价格/库存/评价各自成表），索引按**查询场景反规范化**：

```text
商品搜索文档 = SPU 核心字段
  + SKU 列表（嵌套：规格、价格区间）
  + 类目路径（前台类目多条，冗余全路径名用于面包屑与筛选）
  + 属性键值对（筛选项，来自受控 EAV，商品篇 §3）
  + 统计字段（销量、评价数、好评率 —— 派生自其他域）
  + 营销标记（是否有券、是否促销 —— 高频变化，见 §4 分级更新）
  + 排序特征（人气分、质量分 —— 算法产出）
```

纪律：

- **文档粒度 = 搜索结果粒度**：搜商品出 SPU 就以 SPU 为文档（SKU 嵌套），搜门店商品出"店铺×商品"就以此为文档——先定结果长什么样，再定文档。
- **筛选项字段化**：品牌、颜色、价格区间是索引里的显式字段（keyword/数值），不是让引擎解析 JSON——受控 EAV 中"是否搜索筛选项"标记（商品篇）决定哪些属性进索引。
- 索引里**只放查询和展示需要的**：详情大文本不进索引（点击后回源查 DB/缓存），索引膨胀直接拖慢全量重建。

## 2. 同步链路：CDC 聚合宽表

```text
各域 DB 变更（binlog/CDC 或领域事件）
  → 变更分发：哪个表变了影响哪些文档（依赖倒排：SKU 价格变 → 找到所属 SPU 文档）
  → 聚合构建：回查各域拼装完整文档（或增量部分更新）
  → 写入索引（带版本号，防乱序覆盖）
```

要点：

- **变更影响分析是核心难点**：一个品牌改名影响十万商品文档——大范围变更走批量重刷任务而非逐条风暴；变更 → 文档的映射关系显式配置。
- **乱序防护**：文档带版本（源数据更新时间或版本号），索引写入用外部版本控制，旧版本写入被拒——CDC 链路乱序是常态。
- **删除同步**：DB 软删除/下架 → 索引删除或打不可见标记，靠 CDC 的删除事件 + 每日全量比对兜底（索引里的"幽灵商品"是最常见事故：已下架还能搜到）。
- 同步任务失败重试 + 死信队列，单文档失败不阻塞整体。

## 3. 全量重建：索引的可重算性

索引必须能随时从 DB 全量重建（总览篇"派生可重算"）：

- **双索引 + 别名切换**：新索引后台全量构建（DB 快照 + 构建期间增量追平）→ 数据校验（条数、抽样字段比对）→ 别名原子切换 → 旧索引保留一段回滚窗口。
- 重建的触发场景：mapping 变更（引擎的 schema 变更几乎都要重建）、数据订正、事故恢复——**没有演练过重建的搜索系统等于没有备份**。
- 全量取数走只读副本或数仓，不打业务库。

## 4. 一致性分级：不同字段不同鲜度

索引是最终一致的，但"多终"要按字段分级：

| 字段类 | 例子 | 鲜度要求 | 方案 |
|---|---|---|---|
| 生死状态 | 上下架、删除 | 秒级 | CDC 最高优先级 + 查询时二次校验兜底 |
| 交易关键 | 价格、是否有库存 | 秒~分钟级 | 独立轻量更新通道（partial update），不重建整文档 |
| 描述内容 | 标题、图片、属性 | 分钟级 | 常规聚合链路 |
| 统计特征 | 销量、评分、人气分 | 小时~天级 | 批量离线更新 |

- **查询时兜底**：搜索结果页/下单前对价格、库存、上架状态**回源校验**（以 DB 为准），索引负责"找得到"，交易数据以权威源为准——这条边界划清楚，索引延迟就从事故降级为体验问题。
- 有无库存进索引只做布尔/分层（有货/紧张/无货），精确库存数永远不进索引。

## 5. 搜索运营模型：词表与规则也是数据

搜索不只是索引，还有一组需要建模的运营数据（都在 DB 中管理、版本化、同步给引擎）：

- **同义词/纠错词表**：同义词组（"手机=移动电话"）、纠错对，带生效状态与生效记录——改词表影响面大，需灰度与回滚。
- **搜索干预**：指定词的置顶/加权/屏蔽（"iphone"置顶某商品），带生效区间和操作留痕——干预是运营动作，要能回答"为什么这个商品排第一"。
- **热搜/底纹词**：离线统计产出 + 人工审核（敏感词过滤）双通道。
- **查询日志**：查询词、结果数、点击行为——搜索效果优化和"无结果词"运营的数据基础，进数仓不进业务库。

## 6. 搜索评审清单

```text
1. 文档粒度是否等于搜索结果粒度？筛选项是否显式字段化？
2. 索引是否只含查询所需？大文本是否回源？
3. 同步是否 CDC 驱动、变更影响映射显式、乱序有版本防护？
4. 删除/下架同步是否有全量比对兜底？
5. 全量重建是否双索引+别名切换？是否演练过？
6. 字段鲜度是否分级？价格库存是否独立快速通道？
7. 交易关键数据是否查询时回源校验？精确库存是否不进索引？
8. 同义词/干预/热搜是否版本化、留痕、可灰度回滚？
9. 查询日志是否采集进数仓支撑效果分析？
```

## 7. 总结

搜索建模五条纪律：

1. **为结果建文档**：文档粒度、字段取舍全部由查询场景倒推。
2. **索引是派生**：CDC 喂养、版本防乱序、随时可全量重建。
3. **鲜度分级**：生死秒级、价格快道、统计慢道，各走各的通道。
4. **交易回源**：索引管找到，DB 管准确，边界清楚就没有资损。
5. **运营即数据**：词表、干预、排序规则版本化留痕，排序结果可解释。

> 搜索引擎再强大也只是 DB 的一面镜子。镜子可以磨得更亮，但永远不要把镜子当成实物本身。
