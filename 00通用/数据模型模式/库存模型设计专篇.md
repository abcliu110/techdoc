# 库存模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦库存建模。库存是 ERP/零售/电商中最容易出错的领域：它同时具有账务性（数量守恒、可追溯）、高并发性（秒杀扣减）和物理性（实物盘点会打脸系统数据）。

## 0. 一句话总纲

> 库存模型的本质是"数量维度的账务系统"+"承认系统与实物必然有差异的纠偏机制"。

## 1. 先分清四种"库存数"

一个"库存"字段走天下是所有库存事故的根源。至少区分：

| 库存数 | 含义 | 谁改它 |
|---|---|---|
| 实物库存（on-hand） | 仓库里真实存在的数量 | 入库、出库、盘点 |
| 可用库存（available） | 可以被销售承诺的数量 = 实物 - 锁定 - 不可售 | 派生值，不单独存储 |
| 锁定库存（locked/reserved） | 已被订单占用但未出库 | 下单锁定、支付确认、超时释放 |
| 在途库存（in-transit） | 已发出未到达（调拨、采购在途） | 调拨单、采购单 |

另有**渠道库存/共享库存**（同一实物池分配给不同渠道的可售额度）属于销售策略层，建在实物库存之上而不是混进去。

**为什么**：销售看"能不能卖"（可用），仓库看"有没有货"(实物)，财务看"值多少钱"（实物×成本）。三个视角改同一个字段，必然互相踩踏。

## 2. 库存的核心维度：库存项

库存不是"商品有多少个"，而是：

```text
库存项 = SKU × 仓库/门店 × 库存状态（可售/残次/待检/冻结） [× 批次] [× 库位] [× 货主]
```

```sql
CREATE TABLE inv_stock (
  id           BIGINT PRIMARY KEY,
  sku_id       BIGINT NOT NULL,
  warehouse_id BIGINT NOT NULL,
  stock_status VARCHAR(16) NOT NULL DEFAULT 'SELLABLE', -- SELLABLE/DEFECTIVE/QC/FROZEN
  batch_no     VARCHAR(32) NOT NULL DEFAULT '',          -- 无批次管理时为空串（勿用NULL，参与唯一键）
  owner_id     BIGINT NOT NULL DEFAULT 0,                -- 货主（3PL/多货主仓）
  on_hand      INT NOT NULL DEFAULT 0,
  locked       INT NOT NULL DEFAULT 0,
  in_transit   INT NOT NULL DEFAULT 0,
  version      BIGINT NOT NULL DEFAULT 0,
  updated_at   DATETIME NOT NULL,
  UNIQUE KEY uk_item (sku_id, warehouse_id, stock_status, batch_no, owner_id),
  CHECK (on_hand >= 0 AND locked >= 0 AND locked <= on_hand)
);
```

要点：

- 维度是否启用（批次/库位/货主）决定粒度，**上线后加维度是大手术**（相当于账户体系重构），设计期就要判断三年内是否需要。
- 残次品转可售、可售转冻结等，是**状态间的库存移动**（一减一增两条流水），不是改字段。
- `CHECK` 与原子 UPDATE 双保险防负库存。

## 3. 库存流水：数量守恒的账本

每次库存变动记流水，结构与账务流水同构：

```sql
CREATE TABLE inv_flow (
  id            BIGINT PRIMARY KEY,
  stock_id      BIGINT NOT NULL,
  biz_type      VARCHAR(32) NOT NULL,   -- PURCHASE_IN/SALE_OUT/TRANSFER_IN/TRANSFER_OUT/
                                        -- ADJUST/RETURN_IN/LOCK/UNLOCK/STATUS_MOVE...
  biz_order_no  VARCHAR(64) NOT NULL,   -- 来源单据号
  biz_line_id   BIGINT NOT NULL,        -- 来源单据行（行级追溯）
  idempotent_key VARCHAR(64) NOT NULL,
  change_qty    INT NOT NULL,           -- 带符号
  qty_after     INT NOT NULL,           -- 变动后 on_hand（链式对账）
  unit_cost     DECIMAL(18,4),          -- 本次变动的成本单价（见 §6）
  created_at    DATETIME NOT NULL,
  operator      VARCHAR(32),
  UNIQUE KEY uk_idem (idempotent_key),
  KEY idx_stock_time (stock_id, created_at)
);
```

纪律与账务篇相同：只增不改、幂等键落唯一索引、`qty_after` 链条连续性即对账手段。

**守恒式**：任何 SKU，`Σ入库 - Σ出库 = Σ各仓各状态 on_hand + 在途`。调拨、状态转移在守恒式中两边抵消（一出一入），凭空多货少货立即暴露。

## 4. 单据驱动：一切库存变动来自单据

库存永远不被"直接修改"，只被单据驱动：

| 单据 | 动作 |
|---|---|
| 采购入库单 | 待检 +，质检后转可售 |
| 销售出库单 | 锁定 → 实扣 |
| 退货入库单 | 待检/可售 + |
| 调拨单 | 出仓 -、在途 +、入仓时在途 - 实物 + |
| 盘点单 | 差异调整（见 §7） |
| 报损/报溢单 | 调整（需审批） |

每张单据行级关联库存流水（`biz_order_no + biz_line_id`），实现"这个 SKU 为什么少了 3 个"的完整回答链：库存流水 → 单据行 → 单据 → 上游业务（订单/审批）。

> 没有单据的库存变动等于没有凭证的记账——审计上不存在，出错时无法追责。

## 5. 扣减时序：下单锁定、支付确认、超时释放

电商/零售标准三段式（与账务冻结同构）：

```text
下单：   locked += n     原子UPDATE ... WHERE on_hand - locked >= n   （防超卖核心）
支付成功：on_hand -= n, locked -= n   （产生出库流水；实物出库由 WMS 后续执行）
取消/超时：locked -= n
```

变体选择：

- **拍下减库存**（下单即锁定）：用户体验好，需超时释放对抗恶意占库存。
- **付款减库存**（支付才锁定）：无占用问题，但可能超卖后退款。大促常用"拍下减 + 短超时（15 分钟）"。
- **秒杀热点**：单行 UPDATE 扛不住时，用总览篇 §16 方案——库存分桶（一个 SKU 拆 N 行）、Redis 预扣 + 异步落库（Redis 是派生副本，以 DB 为准对账）、请求排队。

**锁定记录表**必须独立存在（订单号、SKU、数量、过期时间、状态），仅靠 locked 汇总字段无法回答"锁定被谁占着"，也无法做超时释放。

## 6. 库存成本核算

库存的钱一面。核算方法决定成本流水怎么记：

| 方法 | 逻辑 | 适用 |
|---|---|---|
| 移动加权平均 | 每次入库重算平均单价：新均价 =（原值 + 入库值）/（原量 + 入库量），出库按当前均价 | 零售/电商主流，实现简单 |
| 批次实际成本（分批认定） | 出库按所属批次的实际入库成本 | 高值商品、医药、食品 |
| 先进先出 FIFO | 出库依入库顺序逐层消耗成本层 | 财务要求严格的场景 |

要点：

- 出库流水固化 `unit_cost`（当时成本快照）——毛利报表 = 售价 - 当时成本，成本后续变化不回溯（呼应总览篇历史快照）。
- 移动加权在**负库存或乱序入库**时会算出荒谬均价，因此：要么禁止负库存，要么负库存期间成本挂账、期末重算。
- 成本调整（发票晚到、运费分摊）走成本调整单，产生成本流水，不改历史出库成本。

## 7. 盘点与差异：承认系统会错

库存与账务最大的不同：**存在物理世界这个"对手方"**，实物才是终审。

盘点模型：

```text
盘点单：仓库、范围（全盘/循环盘点抽SKU）、快照时间、状态
盘点明细：SKU/批次/库位 | 账面数（快照）| 实盘数 | 差异数 | 差异原因 | 复盘标记
```

纪律：

- 盘点开始时**冻结账面快照**，盘点期间的正常业务变动要么暂停（停业盘点），要么记录并在差异计算中抵消（动态盘点）。
- 差异确认后生成**报损/报溢单**驱动调整流水（biz_type=ADJUST），差异原因必填、大额审批——盘点差异是损耗率、盗损、流程漏洞的核心经营指标，不是"把数字改对"。
- 循环盘点（每天盘一小部分，高值高频 SKU 盘得更勤）优于一年一次全盘。

> 系统数字与实物的差异不是异常，是常态。世界级库存系统的标志是差异被测量、被解释、被收敛，而不是被掩盖。

## 8. 多仓、调拨与寻源

- **调拨在途**：调出仓 `on_hand -`、在途 `+`；调入仓收货时在途 `-`、`on_hand +`。在途余额长期不清零 = 丢货告警（与账务在途户同构）。
- **寻源（sourcing）**：一张订单从哪个仓发货，是独立的决策层（就近、成本、分单），不要把寻源逻辑耦合进库存扣减——扣减只接受"仓库已确定"的指令。
- **渠道共享库存**：实物池 + 各渠道配额/比例，超卖风险在配额层控制，实物层守恒不破坏。

## 9. 库存评审清单

```text
1. 实物、锁定、可用、在途是否分离？可用是否是计算值？
2. 库存项维度（状态/批次/库位/货主）是否覆盖三年内需求？
3. 所有变动是否由单据驱动、行级关联流水？
4. 扣减是否原子 UPDATE + 条件校验？锁定是否有独立记录和超时释放？
5. 数量守恒式是否可校验？qty_after 链条是否连续？
6. 秒杀/爆款是否有热点方案？Redis 预扣是否以 DB 为准对账？
7. 成本核算方法是否明确？出库成本是否固化快照？负库存对成本的影响是否处理？
8. 盘点流程是否有账面快照和差异单据？差异原因是否强制记录？
9. 调拨在途是否被监控清零？
10. 渠道/共享库存是否建在实物层之上而非混入？
```

## 10. 总结

库存建模五条纪律：

1. **四数分离**：实物、锁定、可用、在途各司其职，可用永远是算出来的。
2. **单据驱动**：没有单据不动库存，行级流水全程可追溯。
3. **原子扣减**：条件更新一条 SQL 防超卖，热点靠拆分不靠祈祷。
4. **量价同录**：数量流水同时固化成本快照，库存账实一体。
5. **实物终审**：盘点差异是常态指标，纠偏走单据留证据。

> 库存系统 = 数量账本 + 物理世界的谦卑。账可以完美，货一定会出入，模型必须为"账实不符"预留正规出口。

## 11. 需要增强的企业级内容

本篇已经覆盖库存四数、库存项、流水、单据、锁定、成本、盘点、多仓和调拨。大型企业库存模型还需要补强批次效期、序列号、库位作业、可承诺库存、补货、质检和财务集成。

### 11.1 批次、效期和序列号

**为什么需要**：食品、医药、电子设备、高值商品都需要追踪批次、效期或单品序列号。

**标准做法**：

```text
批次：同一批入库、同一生产日期或供应商批号
效期：生产日期、有效期、到期预警、先进先出
序列号：单件唯一追踪，销售、退货、维修都要绑定
```

**解决的问题**：

- 临期商品无法预警。
- 质量召回无法定位影响范围。
- 高值商品串货、丢失、维修无法追踪。

**带来的复杂度**：

- 库存粒度从 SKU 升级为 SKU + 批次/序列号。
- 出入库必须扫描或选择批次。
- 报表和成本核算更复杂。

### 11.2 库位和仓内作业模型

**为什么需要**：仓库不是一个抽象地点，大型仓库有库区、货架、库位、拣货、复核、打包、移库等作业。

**标准做法**：

```text
仓库 -> 库区 -> 货架 -> 库位
入库上架任务
拣货任务
复核任务
打包任务
移库任务
```

库存余额可以在仓库级、库位级分别管理。仓库级用于销售可用，库位级用于仓内作业。

**解决的问题**：

- 系统知道有货，但仓库找不到货。
- 拣货路径低效。
- 移库无记录，账实不符。

### 11.3 ATP 可承诺库存

**为什么需要**：销售承诺不只看当前可用库存，还要考虑采购在途、生产计划、已锁定订单、未来到货和安全库存。

**标准做法**：

```text
ATP = 当前可用 + 预计入库 - 已承诺出库 - 安全库存
```

适用于预售、B2B 订单、大件商品、供应链周期较长的业务。

**带来的复杂度**：

- 需要采购、生产、调拨、销售预测数据。
- 承诺时间和数量都要建模。
- 承诺失败需要异常处理。

### 11.4 安全库存和补货模型

**为什么需要**：库存系统不仅记录库存，还要帮助企业保持合理库存水平。

**标准做法**：

- SKU + 仓库维度配置安全库存、最高库存、补货点、补货批量。
- 根据销售速度、采购周期、季节性计算建议补货量。
- 生成补货建议单，再转采购单或调拨单。

**解决的问题**：

- 缺货靠人工发现。
- 滞销积压无法预警。
- 门店补货不稳定。

### 11.5 质检、冻结和隔离库存

**为什么需要**：入库商品、退货商品、异常商品不应直接进入可售库存。

**标准做法**：

```text
采购入库 -> 待检
质检通过 -> 可售
质检不通过 -> 残次/退供/报损
争议商品 -> 冻结
```

状态间转移必须走单据和流水，不能直接改状态字段。

### 11.6 库存与财务成本集成

**为什么需要**：库存数量和库存金额必须能对上。库存变化通常影响成本、资产和损益。

**标准做法**：

- 入库形成库存资产增加。
- 出库形成成本结转。
- 盘亏形成损失。
- 盘盈形成待处理收益或差异。
- 成本调整形成成本调整凭证。

**底层思想**：

> 库存是数量账，财务是金额账。企业级库存必须能让数量和金额互相解释。
