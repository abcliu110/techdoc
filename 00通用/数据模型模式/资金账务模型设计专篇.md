# 资金账务模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦资金与账务建模。总览篇讲"为什么"（复式记账、守恒、对账、热点的思想），本篇讲"怎么建"：表结构、记账流程、完整案例。适用于支付、储值、钱包、积分、分账、结算等一切"钱和类钱"系统。

## 0. 一句话总纲

> 账务系统的全部设计，都是为了让"每一分钱在任何时刻都能说清楚：从哪来、在哪里、到哪去、谁动的、为什么"。

## 1. 先分清三层：交易、账务、资金

初学者最常见的错误是把三层混成一张"订单表 + 余额字段"。世界级系统严格分层：

| 层 | 回答的问题 | 典型对象 | 特征 |
|---|---|---|---|
| 交易层（业务层） | 发生了什么业务 | 订单、退款单、充值单、提现单 | 面向用户和业务流程，有丰富状态机 |
| 账务层（记账层） | 价值如何在账户间转移 | 账户、分录、账本、余额 | 只认借贷平衡，不关心业务细节 |
| 资金层（渠道层） | 真实资金在银行/渠道如何流动 | 渠道单、银行流水、结算账单 | 受外部系统支配，异步、会出错 |

三层通过单号关联：业务单号 → 记账凭证号 → 渠道单号。

**为什么必须分层**：

- 业务会变（新玩法、新单据类型），但记账规则极稳定——分层后新业务只需定义新的"记账规则"，账务核心不动。
- 资金层不可控（渠道延迟、掉单、退款失败），分层后业务状态和资金状态可以各自推进、互相核对。
- 审计和对账天然需要三层互相印证：业务说发生了、账务说记了、银行说钱动了，三者对上才是真的。

> 交易层描述故事，账务层描述价值，资金层描述现实。三者互为证据。

## 2. 账户体系设计

### 2.1 账户的分类维度

账户 = 账户所有者 × 账户类型 × 币种/资产类型。

**按所有者**：

- 用户账户：储值余额、积分、券、佣金。
- 商户账户：待结算、可提现、保证金。
- 平台内部账户（关键！）：这是初学者最容易缺失的部分。

**必备的平台内部账户（内部户）**：

| 内部户 | 用途 |
|---|---|
| 渠道待清算户 | 用户支付成功但渠道未结算给我方时，钱的"逻辑位置" |
| 银行存款户（备付金户） | 渠道已结算到我方银行账户的钱 |
| 手续费支出户 | 支付给渠道的手续费 |
| 平台收入户 | 平台佣金、服务费收入 |
| 营销费用户 | 平台补贴、红包、立减的出资方 |
| 在途户 | 提现已发起但银行未确认期间的钱 |
| 差错户（挂账户） | 对账差异未定性时的临时挂账 |

**为什么需要内部户**：复式记账要求每笔钱都有对手方。用户充值 100 元，如果只给用户账户 +100，这 100 从哪来？答案是"渠道待清算户"。没有内部户，守恒式就建立不起来，对账就没有基准。

### 2.2 账户表结构

```sql
-- 账户表（当前值层，是流水的派生）
CREATE TABLE acct_account (
  id              BIGINT PRIMARY KEY,
  account_no      VARCHAR(32) NOT NULL,          -- 账户编号，对外
  owner_type      VARCHAR(16) NOT NULL,          -- USER / MERCHANT / PLATFORM
  owner_id        BIGINT NOT NULL,               -- 所有者ID，内部户为固定值
  account_type    VARCHAR(32) NOT NULL,          -- BALANCE/POINT/PENDING_SETTLE/...
  currency        CHAR(3) NOT NULL DEFAULT 'CNY',
  balance         DECIMAL(18,4) NOT NULL DEFAULT 0,  -- 总余额
  frozen          DECIMAL(18,4) NOT NULL DEFAULT 0,  -- 冻结金额
  -- 可用余额 = balance - frozen，派生值不落库，避免三字段互相打架
  balance_dir     TINYINT NOT NULL,              -- 余额方向：1借方户/2贷方户
  status          VARCHAR(16) NOT NULL,          -- NORMAL/FROZEN/CLOSED
  version         BIGINT NOT NULL DEFAULT 0,     -- 乐观锁
  created_at      DATETIME NOT NULL,
  updated_at      DATETIME NOT NULL,
  UNIQUE KEY uk_owner (owner_type, owner_id, account_type, currency)
);
```

要点：

- **可用余额不单独存列**。存三个字段（总额、冻结、可用）必然出现"改了两个忘了第三个"，可用余额永远用 `balance - frozen` 计算。
- **余额方向**：资产类账户（银行存款）借方为正，负债类账户（用户余额——那是平台欠用户的钱）贷方为正。统一记录方向，守恒校验才能写成通用公式。
- **唯一键保证一个所有者同类账户只有一个**，开户用 INSERT IGNORE / ON CONFLICT 幂等。

### 2.3 科目表（可选，账户的分类树）

中大型系统在账户之上建科目（Chart of Accounts）：科目是分类（"用户储值负债"），账户是实例（用户 123 的储值户）。科目挂接财务会计科目，让业务账务能直接出财务报表。小系统可用 account_type 枚举代替。

## 3. 分录与账本：复式记账的落地结构

### 3.1 核心三表：凭证、分录、流水

```sql
-- 记账凭证（一次业务动作产生一张凭证）
CREATE TABLE acct_voucher (
  id            BIGINT PRIMARY KEY,
  voucher_no    VARCHAR(32) NOT NULL,
  biz_type      VARCHAR(32) NOT NULL,   -- RECHARGE/PAY/REFUND/SETTLE/WITHDRAW...
  biz_order_no  VARCHAR(64) NOT NULL,   -- 业务单号
  idempotent_key VARCHAR(64) NOT NULL,  -- 幂等键 = 业务单号 + 动作
  amount        DECIMAL(18,4) NOT NULL, -- 凭证总金额
  currency      CHAR(3) NOT NULL,
  acct_date     DATE NOT NULL,          -- 账务日期（日切决定，≠自然日期）
  created_at    DATETIME NOT NULL,
  memo          VARCHAR(255),
  UNIQUE KEY uk_idem (idempotent_key)   -- 幂等的物理保证
);

-- 分录（一张凭证 2~N 条分录，借贷合计为零）
CREATE TABLE acct_entry (
  id           BIGINT PRIMARY KEY,
  voucher_id   BIGINT NOT NULL,
  account_id   BIGINT NOT NULL,
  direction    TINYINT NOT NULL,        -- 1借 DEBIT / 2贷 CREDIT
  amount       DECIMAL(18,4) NOT NULL,  -- 恒为正数，方向由 direction 表达
  currency     CHAR(3) NOT NULL,
  acct_date    DATE NOT NULL,
  created_at   DATETIME NOT NULL,
  KEY idx_voucher (voucher_id),
  KEY idx_account_date (account_id, acct_date)
);

-- 账户流水（面向单账户的余额变动视角，含变动后余额）
CREATE TABLE acct_flow (
  id             BIGINT PRIMARY KEY,
  account_id     BIGINT NOT NULL,
  entry_id       BIGINT NOT NULL,
  voucher_no     VARCHAR(32) NOT NULL,
  biz_type       VARCHAR(32) NOT NULL,
  change_amount  DECIMAL(18,4) NOT NULL,  -- 带符号：对该账户的余额影响
  balance_after  DECIMAL(18,4) NOT NULL,  -- 变动后余额（快照，对账利器）
  acct_date      DATE NOT NULL,
  created_at     DATETIME NOT NULL,
  KEY idx_account_time (account_id, created_at)
);
```

要点：

- **金额恒为正 + 方向字段**，而不是用正负号表达借贷。正负号方案在"借方户/贷方户余额方向不同"时会把所有人绕晕。
- **flow 表冗余 balance_after**：每条流水记下变动后余额，使"任意时点余额"可直接查询，且相邻流水的 balance_after 连续性本身就是对账手段（断链 = 有并发漏洞或丢流水）。
- **分录表和流水表看似重复，视角不同**：分录面向凭证（这笔业务动了哪些账户），流水面向账户（这个账户经历了什么）。中小系统可合并为一张表同时带 voucher_id 和 balance_after。
- 凭证、分录、流水都是**不可变事实**：只 INSERT，错误走冲正凭证。

### 3.2 记账规则配置化

每种业务类型的记账规则（借哪个户、贷哪个户）不要散落在代码 if/else 里，配置成规则表或规则代码：

```text
biz_type = PAY_BY_BALANCE（余额支付，含平台抽佣3%）:
  借 用户储值户        100%
  贷 商户待结算户       97%
  贷 平台收入户          3%
```

新业务上线 = 新增一条记账规则 + 审核，账务核心代码零改动。规则本身要版本化（呼应总览篇"规则配置必须版本化"）。

## 4. 记账核心流程

### 4.1 单次记账的标准事务

```text
BEGIN;
1. 幂等检查：凭证表按 idempotent_key 查询，存在则直接返回原结果（成功语义）。
2. 按记账规则生成凭证 + 分录组，校验借贷合计为零。
3. 逐账户更新余额：
   UPDATE acct_account
   SET balance = balance + ?, version = version + 1, updated_at = NOW()
   WHERE id = ? AND (balance + ? >= frozen OR 该户允许透支);
   -- 影响行数为 0 → 余额不足，整体回滚
4. 写入分录、流水（flow.balance_after 取自更新后的余额）。
COMMIT;
```

纪律：

- **余额检查和扣减必须是同一条原子 UPDATE**，先 SELECT 再 UPDATE 在并发下必然超扣。
- 幂等键唯一索引是防重复的物理保证，应用层检查只是快速路径。
- 一张凭证的所有账户变动在**同一个本地事务**内完成——这要求参与记账的账户在同一个库。跨库场景见 4.3。
- **锁顺序**：一笔凭证涉及多个账户时，按 account_id 排序后依次更新，防止死锁。

### 4.2 冻结/解冻/扣减三段式

支付、提现、担保交易的标准模式：

```text
预扣（下单/发起提现）：  frozen += X       （校验 balance - frozen >= X）
确认（支付成功/提现成功）：balance -= X, frozen -= X   （此时才产生分录）
取消（超时/失败）：      frozen -= X       （无分录，或记冻结流水）
```

要点：

- 冻结不是记账（价值没有转移），确认才是记账。冻结用独立的冻结流水表追踪（冻结单号、金额、过期时间、状态）。
- 冻结必须有**超时自动释放**任务，否则失败请求会永久占用用户资金。
- 部分确认要支持：冻结 100，实际扣 80，释放 20。

### 4.3 跨库/跨服务记账：账务 Saga

用户余额库和商户账务库分库时，一笔支付无法单事务完成。标准做法：

1. 交易层创建支付单（状态：处理中）。
2. 用户侧记账：借用户储值户，贷"内部在途户"（用户库内本地事务 + Outbox 事件）。
3. 商户侧消费事件记账：借"内部在途户"，贷商户待结算户（幂等）。
4. 两边都成功 → 支付单成功；商户侧失败重试超限 → 冲正用户侧，支付单失败。

关键：**在途户把一个跨库动作拆成两个各自平衡的本地凭证**，任何时刻全局守恒依然成立（钱在在途户上），且在途户余额长期不为零就是告警信号。

## 5. 差错、冲正与红字

- **冲正（反向凭证）**：原凭证的镜像（借贷方向互换），关联原凭证号，理由必填。禁止 UPDATE/DELETE 原凭证。
- **部分冲正**：金额小于原凭证，累计冲正金额 ≤ 原凭证金额，用"原凭证 + 累计已冲正"约束保护（呼应总览篇防超退）。
- **差错户运转**：对账发现的多钱/少钱，先原因不明入差错户（借差错户 或 贷差错户），查明后从差错户结转到正确账户。差错户余额是运营健康度指标，目标趋零。
- **红字与蓝字**：财务口径的负向单据（红字发票、红冲凭证）在账务层就是冲正凭证，业务层是独立的负向单据，两层都留痕。

## 6. 对账体系落地

总览篇讲了三层对账思想，本篇给出模型：

```sql
-- 对账批次
CREATE TABLE recon_batch (
  id           BIGINT PRIMARY KEY,
  recon_date   DATE NOT NULL,           -- 对账账务日
  channel      VARCHAR(32) NOT NULL,    -- 渠道：WXPAY/ALIPAY/BANK...
  bill_file    VARCHAR(255),            -- 渠道账单文件
  our_count    INT, our_amount   DECIMAL(18,4),
  their_count  INT, their_amount DECIMAL(18,4),
  diff_count   INT,
  status       VARCHAR(16) NOT NULL,    -- INIT/RUNNING/BALANCED/DIFF/CLOSED
  UNIQUE KEY uk_date_channel (recon_date, channel)
);

-- 差异明细
CREATE TABLE recon_diff (
  id           BIGINT PRIMARY KEY,
  batch_id     BIGINT NOT NULL,
  diff_type    VARCHAR(16) NOT NULL,    -- LONG长款/SHORT短款/AMT_DIFF金额不符/STATUS_DIFF
  our_order_no VARCHAR(64),
  their_order_no VARCHAR(64),
  our_amount   DECIMAL(18,4),
  their_amount DECIMAL(18,4),
  handle_status VARCHAR(16) NOT NULL,   -- PENDING/PROCESSING/RESOLVED/WRITTEN_OFF
  handle_way   VARCHAR(32),             -- 补单/退款/挂差错户/渠道处理...
  handler      VARCHAR(32),
  resolved_at  DATETIME,
  memo         VARCHAR(255)
);
```

**对账时序要点**：

- 短款（我有他无）常见原因是渠道账单延迟——T 日差异先挂起，T+1 账单到达后自动销账，连续 N 日未销才升级人工。
- 长款（他有我无）= 掉单，标准处置是按渠道单反查补单（补业务单 + 补凭证），补单必须走幂等。
- **总分核对每日必做**：Σ流水变动 = 余额变化（逐户），Σ所有账户余额（按方向）= 0（全局守恒），flow 表 balance_after 链条连续。三项任何一项失败即冻结日切、触发告警。

## 7. 日切与账务日期

- 每笔凭证带 `acct_date`（账务日期），由日切规则决定，而非 created_at 的日期部分。
- 日切流程：切点（如 23:50）后新交易记入次日账务日 → 等待迟到交易的宽限期 → 跑总分核对 → 生成日终余额快照 → 冻结当日。
- 日终快照表：`账户ID | 账务日 | 期初余额 | 借方发生额 | 贷方发生额 | 期末余额`，是报表、审计、快速对账的基础，也让"任意历史日余额"不再需要全量回放流水。
- 快照出具后不可变；发现错误重出新版本快照并留痕（呼应总览篇 §13 特例规则）。

## 8. 多币种与精度

- 金额与币种成对出现，不同币种绝不相加。
- 每币种明确最小单位和存储精度（JPY 无小数，BTC 8 位小数），建议 DECIMAL(18,4) 起步、加密资产另议。
- 换汇是两笔记账 + 一个汇率：借出 A 币种、贷入 B 币种，汇率和换汇时间固化在凭证上；汇兑损益差额记入"汇兑损益户"，守恒式按币种分别成立。
- 分摊除不尽时的**尾差规则**要显式定义：最后一笔吃尾差（如 100 元分给 3 人：33.33/33.33/33.34），并写入记账规则，禁止各处代码各自舍入。

## 9. 完整案例：储值支付 + 分账 + 部分退款

**场景**：用户充值 100 → 用余额买 100 元商品（商家 97 / 平台 3）→ 退款 30（按比例：商家出 29.1 / 平台出 0.9）。

```text
① 充值 100（渠道支付成功回调，幂等键=充值单号+SUCCESS）
   借 渠道待清算户    100
   贷 用户储值户      100
   （渠道 T+1 结算到账后：借 银行存款户 99.4 / 借 手续费支出户 0.6 / 贷 渠道待清算户 100）

② 余额支付 100（幂等键=支付单号+PAY）
   借 用户储值户      100
   贷 商户待结算户     97
   贷 平台收入户        3

③ 部分退款 30（幂等键=退款单号+REFUND，校验累计退款 ≤ 100）
   借 商户待结算户    29.1
   借 平台收入户       0.9
   贷 用户储值户      30

任意时刻验证：所有账户余额按方向求和 = 0 ✓
用户视角：100 - 100 + 30 = 30 ✓  商户视角：97 - 29.1 = 67.9 ✓  平台：3 - 0.9 = 2.1 ✓
```

这个案例覆盖了：内部户、渠道清算分离、抽佣分账、按比例退款、幂等键、防超退、全局守恒——账务系统 80% 的日常就是这五张凭证的变体。

## 10. 账务系统评审清单

```text
1. 交易、账务、资金三层是否分离？单号链路是否贯通？
2. 内部户体系是否完整（待清算、在途、手续费、收入、营销、差错）？
3. 每笔价值变动是否都有借贷平衡的分录？全局守恒式是否可校验？
4. 余额检查与扣减是否同一条原子 UPDATE？可用余额是否是计算值而非存储值？
5. 凭证/分录/流水是否只增不改？错误是否全部走冲正？
6. 幂等键是否落在唯一索引上？重复请求返回什么？
7. 冻结三段式是否有超时释放？是否支持部分确认？
8. 跨库记账是否用在途户拆分？在途户余额是否被监控？
9. 部分冲正/退款是否有累计校验防超退？
10. 三层对账是否每日执行？长短款处置路径是否定义？差错户是否趋零？
11. flow.balance_after 链条是否连续？总分核对失败是否阻断日切？
12. 账务日期与日切规则是否明确？日终快照是否冻结？
13. 多币种是否金额币种成对？尾差规则是否统一定义？
14. 热点账户（平台户、头部商户）是否有缓冲/拆分方案？（见总览篇 §16）
15. 记账规则是否配置化、版本化，而不是散落在代码里？
```

## 11. 总结

资金账务建模的方法可以浓缩为五条纪律：

1. **三层分离**：交易讲故事，账务讲价值，资金讲现实。
2. **复式平衡**：每张凭证借贷合计为零，全局余额守恒可校验。
3. **事实不可变**：凭证分录流水只增不改，纠错走冲正。
4. **原子与幂等**：余额变更一条 UPDATE 完成，幂等键落唯一索引。
5. **对账兜底**：假设一定会错，用三层对账 + 差错户 + 日切让错误当天暴露、有序处置。

> 账务系统的可信度不来自代码没有 bug，而来自：即使有 bug，钱也不会无声无息地错。

## 12. 需要增强的企业级内容

本篇已经覆盖账户、凭证、分录、流水、冻结、对账、日切、多币种和冲正，是资金账务模型的核心骨架。大型企业系统中，还需要补强以下能力，否则在财务、税务、审计、清结算和合规场景中仍会出现断点。

### 12.1 业务账、财务账、渠道账三账关系

**为什么需要**：业务系统关心订单、退款、储值、提现；财务系统关心会计科目、凭证、收入、成本、应收应付；渠道系统关心银行和支付机构流水。三者口径不同，不能用一套表强行表达。

**解决的问题**：

- 业务订单成功但渠道未到账。
- 渠道到账但业务掉单。
- 业务收入和财务收入确认时间不一致。
- 财务凭证无法追溯到业务单据。

**标准做法**：

```text
业务账：订单、退款、充值、提现、结算等业务事实
账务账：账户、凭证、分录、流水，表达价值转移
财务账：会计科目、总账、明细账、财务凭证、收入确认
渠道账：支付渠道、银行流水、清算账单、手续费账单
```

业务账驱动账务账，账务账按规则生成财务账，渠道账通过对账校验业务账和账务账。不要让订单表直接承担财务总账责任。

**带来的复杂度**：

- 单号链路必须完整。
- 财务科目映射要治理。
- 收入确认、成本确认、税务确认可能不同步。

### 12.2 科目体系与子账总账

**为什么需要**：账户体系解决业务资产和负债的明细问题，但大型企业还需要按财务科目出资产负债表、利润表和现金流。

**标准做法**：

```text
业务账户：用户储值户、商户待结算户、渠道待清算户
会计科目：银行存款、预收账款、主营业务收入、手续费支出
映射规则：业务账户类型 + 业务动作 -> 会计科目分录
```

业务账务系统通常是财务总账的子账。子账要能汇总到总账，总账要能追溯回业务凭证。

**评价标准**：

- 每张账务凭证能否映射到财务科目。
- 总账余额能否追溯到子账流水。
- 子账总额与财务总账是否可对账。

### 12.3 收入确认与应收应付

**为什么需要**：收钱不等于收入，付款不等于成本。预收、待结算、已结算、已开票、已确认收入是不同状态。

**标准做法**：

- 用户充值进入预收或负债，不直接确认收入。
- 商品交付或服务完成后才确认收入。
- 商户待结算是平台对商户的负债。
- 平台佣金、服务费、手续费分别入不同收入或费用科目。
- 退款和冲正必须反向影响收入、应收应付和结算。

**带来的复杂度**：

- 需要结算单、收入确认单、开票单等衍生单据。
- 财务期间和业务发生时间可能不同。
- 跨期退款、补贴、手续费需要独立处理。

### 12.4 税务和发票模型

**为什么需要**：真实企业账务离不开税率、税额、发票、红冲、未开票收入、已开票未收款等口径。

**标准做法**：

```text
业务单据 -> 税额计算快照 -> 发票申请 -> 发票开具 -> 红字发票/作废
```

账务模型应记录：

- 含税金额。
- 不含税金额。
- 税额。
- 税率。
- 税收分类。
- 发票状态。
- 红冲关系。

**解决的问题**：

- 销售额、开票额、收款额无法对齐。
- 退款后发票无法红冲。
- 税率变更后历史单据无法还原。

### 12.5 清结算模型

**为什么需要**：平台、商户、供应商、分销员、渠道之间的资金归属不是支付成功时立即完成，而是经过清算、结算、提现、打款等过程。

**标准做法**：

```text
交易明细 -> 清算明细 -> 结算单 -> 提现/打款单 -> 银行回单 -> 对账
```

结算单应记录：

- 结算对象。
- 结算周期。
- 交易金额。
- 退款金额。
- 佣金。
- 手续费。
- 补贴。
- 应结金额。
- 已结金额。
- 结算状态。

**带来的复杂度**：

- T+N 结算和跨期退款。
- 部分结算、冻结结算、争议暂扣。
- 商户账、平台账和渠道账三方核对。

### 12.6 风控、争议和拒付

**为什么需要**：支付和账务系统不仅有成功/失败，还有风控冻结、争议、拒付、人工审核、资金暂扣。

**标准做法**：

- 高风险交易进入冻结或待审核账户。
- 争议金额进入争议户或暂挂户。
- 拒付生成独立拒付单和冲正凭证。
- 人工处理必须记录原因、证据、操作者。

**解决的问题**：

- 争议资金不知归属。
- 人工改账无凭证。
- 拒付后收入、结算、商户余额不一致。

### 12.7 账务权限和操作审计

**为什么需要**：账务系统的人工操作风险极高，不能只靠功能权限。

**标准做法**：

- 查账、调账、冲正、结算、提现、关闭差异分开授权。
- 大额操作双人复核。
- 所有人工操作记录操作前后值、原因、附件和审批记录。
- 生产手工改账禁止直接改余额，必须走调账单或冲正单。

**底层思想**：

> 账务系统不相信隐式正确，只相信凭证、权限、复核和审计链。
