# 消息与通知模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

消息与通知模型的本质，是业务事件到用户触达的可追溯转换系统。它把业务事件、通知模板、触达策略、发送任务、渠道回执和合规控制分离。

核心抽象：

```text
业务事件 -> 模板 -> 触达策略 -> 发送任务 -> 渠道回执 -> 合规审计
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| msg_template 消息模板 | template_key, channel, version, content(变量占位), status(DRAFT/AUDITING/PUBLISHED/OFFLINE) | 内容定义版本化；同 key 多渠道多版本共存 |
| msg_strategy 触达策略 | template_key(场景), msg_category(交易/营销/验证码), 渠道编排链, priority, 重试/频控/免打扰配置 | 场景到渠道的唯一决策点，业务代码不选渠道 |
| msg_send_task 发送任务 | idempotent_key(唯一), template_key, template_version, channel, receiver(脱敏), render_params 快照, status(INIT/SENT/DELIVERED/FAILED/SUPPRESSED), suppress_reason | 一人一次发送的幂等事实；固化版本+参数可还原内容 |
| msg_receipt 渠道回执 | send_task_id, 渠道消息 ID, 回执类型(提交/送达/失败/已读), 回执原文, 回执时间 | 外部世界的回答，异步且不完全可信 |
| msg_channel_config 渠道配置 | 供应商, 通道, 渠道模板 ID 映射, 签名, 单价, 限速, status | 多供应商路由与故障切换的依据 |
| msg_consent 授权记录 | user_id, 授权范围, 渠道, 授权/退订时间, status | "凭什么给他发"的证据；退订实时生效 |
| msg_freq_counter 频控计数 | user_id, 场景, 周期, count(原子累加) | 频控闸门的落点，防打扰防轰炸 |
| msg_blacklist 黑名单 | receiver, 类型(投诉/无效号码/退订), 来源, 拉黑时间 | 发送前强制检查项，灰名单降渠道成本 |
| msg_announce 站内公告 | 公告内容, 生效范围, 生效时间, status | 一对多不逐人生成任务，防写放大 |
| msg_inbox_read 用户已读 | user_id, announce_id/message_id, 已读/删除时间 | 拉模式合并读；未读数走派生计数器 |

### 2.2 实体关系

```text
msg_template (1) ──< msg_send_task (N)        任务固化 template_key+version
msg_strategy (1) ──< msg_send_task (N)        策略决定任务的渠道与优先级
msg_send_task (1) ──< msg_receipt (N)         一次发送多条回执（提交/送达/已读）
msg_channel_config (1) ──< msg_send_task (N)  任务路由到具体通道
msg_consent (1) ──< msg_send_task (N)         营销任务发送前查授权
msg_announce (1) ──< msg_inbox_read (N)       公告-已读状态，读时合并
外部引用：user_id/receiver → 会员域；业务事件 ID（幂等键成分）→ 各业务域；
         渠道模板 ID → 渠道侧资源（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：idempotent_key（业务事件+用户+场景）全局唯一，防重复发送；
             (template_key, channel, version) 模板唯一
版本固化：send_task 固化 template_version + render_params，
         模板后续修改不污染历史发送内容
状态机约束：任务状态只进不退（INIT→SENT→DELIVERED/FAILED），
           回执驱动状态迁移；SUPPRESSED 必带 suppress_reason
合规闸门落点：营销类发送前依次查 msg_consent、msg_blacklist、
             msg_freq_counter（原子累加判上限），任一不过即 SUPPRESSED 留痕
留存/分级策略：交易类逐条留痕；营销类批次+统计+失败明细；任务表按时间分区
分片键：receiver/user_id（发送与查询主路径）
```

## 3. 核心张力

最大张力是：

```text
触达效果 vs 合规、成本和用户打扰
```

业务希望消息必达、多发、及时；用户和监管要求授权、退订、频控、免打扰，企业还要控制短信和推送成本。

## 4. 不变量

- 模板、策略、任务、回执必须分离。
- 业务代码只发事件，不直接选择渠道。
- 发送任务必须幂等。
- 发送时模板版本和参数必须快照。
- 合规检查是发送前强制闸门。
- 拦截也要留记录。
- 渠道回执异步且不完全可信。

## 5. 为什么这样设计

模板版本化是为了能还原当时发了什么内容；策略分离是为了让业务事件和渠道编排解耦；回执记录是为了回答到底有没有提交、送达、失败或已读。

## 6. 失败模式

- 业务代码里到处直接发短信，渠道策略无法统一。
- 模板改了以后，历史发送内容无法还原。
- 没有幂等，订单事件重复导致重复通知。
- 营销短信未授权发送，造成合规风险。
- 拦截消息无记录，业务以为发送了但用户没收到。

## 7. 适用边界

适用：

- 短信、邮件、站内信、App 推送、微信模板消息、企业微信、钉钉通知。

可简化：

- 内部低风险通知可简化渠道回执，但交易类和营销类必须完整留痕。

## 8. 设计取舍

优先保证：

- 可追溯。
- 合规。
- 幂等。
- 渠道可降级。

牺牲：

- 发送链路简单性。
- 存储成本。
- 策略配置复杂度。

## 9. 同一模型的多业务形态建模

消息模型要同时支撑多种通知形态，核心方法是：**统一四层骨架 + 消息类别标识 + 差异下沉到策略配置**。

### 9.1 统一骨架（所有形态不变的部分）

```text
msg_template（版本化）→ 触达策略（msg_category 标识形态）→ msg_send_task（幂等）→ 渠道回执
不变量全集：四层分离、发送幂等、模板参数快照、合规闸门前置、拦截留痕
```

骨架决不为某类消息特化——任何形态的"发了什么内容、走了哪个渠道、有没有送达、为什么被拦"都用同一套表回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 交易通知（必达） | 不可退订、失败要补偿 | msg_category=transactional，策略配置渠道降级链（推送→短信）；失败任务进重试队列，超限告警；合规闸门跳过退订检查但不跳过频控 |
| 营销触达 | 授权、退订、频控、成本控制 | msg_category=marketing，发送前强制查授权与退订表；策略挂日/周频控配额；批量任务分片生成 msg_send_task |
| 验证码 | 时效秒级、防轰炸 | 独立高优先级通道（策略 priority 字段）；同号码滑动窗口限流在策略层配置；回执驱动秒级失败切渠道 |
| 站内公告 | 一对多、无外部渠道 | 不逐人生成 send_task：公告主记录 + 用户已读表（拉模式）；仅站内渠道，回执退化为已读记录 |
| 系统告警 | 收件人是值班组、要升级 | 收件人解析为值班表查询而非用户 ID；策略配置升级链（N 分钟未确认升级下一级）；确认动作写回执表 |
| 对接方回调通知 | 收件人是 URL、签名重试 | 渠道类型=webhook；回执 = 对方 HTTP 响应码；策略配置签名密钥引用 + 指数退避重试次数 |

### 9.3 建模决策规则

- **能用 msg_category + 策略配置解决的，不建新发送链路**：差异只在渠道、频控、优先级、重试时，模板/任务/回执一套。
- **策略配置按类别垂直拆**（合规项、降级链、频控、重试各成配置块），不在任务表堆特化字段。
- **发送流程允许按形态裁剪但不新增语义**：站内公告跳过渠道回执，但不得绕过拦截留痕。
- **拆独立域的信号**：当"消息"本身是业务对象（IM 会话、工单对话），有自己的读写模型和状态机时，才是独立域，不是一种通知类别。
