# 商品模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦商品建模。商品是交易系统的"名词之源"：订单、库存、价格、搜索、报表全部引用它。商品模型设计错误的修复成本全系统最高——因为所有下游都在引用它。

## 0. 一句话总纲

> 商品模型的核心是分清"卖的概念"和"库存的实物"，并让类目和属性能长出无限品类而不改表。

## 1. SPU / SKU：最重要的一次抽象

| 概念 | 含义 | 例子 |
|---|---|---|
| SPU（标准产品单元） | "卖的概念"：一个可展示、可搜索的产品 | iPhone 15 |
| SKU（库存量单位） | "实物的最小识别单位"：规格确定、可定价、可入库 | iPhone 15 黑色 256G |
| 销售属性 | 区分 SKU 的属性 | 颜色、容量 |
| 非销售属性 | 描述 SPU 的属性，不产生新 SKU | 上市年份、屏幕尺寸 |

铁律：

- **订单行、库存、价格挂 SKU；展示、搜索、评价挂 SPU**。挂错层是最常见的结构性错误（价格挂 SPU 导致不同规格无法差价；评价挂 SKU 导致同款商品评价分裂）。
- 单规格商品也建 SKU（一个 SPU 下恰好一个 SKU），保持下游引用统一，不要为"简单商品"走特殊路径。
- SKU 由"SPU + 销售属性值组合"唯一确定，用唯一键约束（spu_id + 规格值指纹）防止重复 SKU。

```sql
CREATE TABLE prd_spu (
  id           BIGINT PRIMARY KEY,
  spu_code     VARCHAR(32) NOT NULL,
  title        VARCHAR(255) NOT NULL,
  category_id  BIGINT NOT NULL,        -- 后台类目（唯一挂靠）
  brand_id     BIGINT,
  shop_id      BIGINT NOT NULL,        -- 平台模式下的卖家
  status       VARCHAR(16) NOT NULL,   -- DRAFT/PENDING/ON/OFF/DELETED
  UNIQUE KEY uk_code (spu_code)
);

CREATE TABLE prd_sku (
  id           BIGINT PRIMARY KEY,
  sku_code     VARCHAR(32) NOT NULL,
  spu_id       BIGINT NOT NULL,
  spec_json    JSON NOT NULL,          -- {"颜色":"黑","容量":"256G"} 展示用
  spec_fingerprint VARCHAR(64) NOT NULL, -- 规格值排序后哈希，防重复
  status       VARCHAR(16) NOT NULL,
  UNIQUE KEY uk_code (sku_code),
  UNIQUE KEY uk_spec (spu_id, spec_fingerprint)
);
```

## 2. 类目：前后台分离

**后台类目（管理类目）**：稳定的树，商品唯一挂靠，决定属性模板、资质要求、佣金费率、报表口径。改动极谨慎（商品迁移类目是大动作）。

**前台类目（导购类目）**：面向消费者的展示树，随运营随时调整，多对多挂靠（一个商品可出现在多个前台类目），甚至按人群/渠道有多套。

**为什么分离**：运营每周都想调导购结构，而费率、资质、报表绑定的分类必须稳定。一棵树同时服务两个目标，结局是既不敢改（怕报表乱）又不得不改（运营催），最后靠人肉例外。分离后前台类目只是"引用商品的标签树"，随便改。

层级建模用总览篇 §35 方案（邻接表 + 路径字段）；订单行固化下单时的后台类目快照，类目调整不污染历史报表。

## 3. 属性系统：类目属性模板

品类间属性天差地别（服装有尺码面料，手机有内存屏幕），不能用宽表硬列。标准方案是**类目属性模板 + 属性值表**：

```text
属性定义（attr）：属性ID、名称、录入类型（单选/多选/文本/数值+单位）、是否销售属性、是否搜索筛选项
类目-属性关联（category_attr）：类目挂哪些属性、是否必填、排序
属性值字典（attr_value）：可选值列表（"黑色""白色"）
商品属性值（spu_attr_value / sku_spec）：商品的具体取值
```

要点：

- 这是**受控的 EAV**（总览篇 §34 说的"只在元数据平台受控使用"正是此处）：属性有类型、有字典、有校验，不是随意的 kv。
- **搜索筛选属性**（价格区间、品牌、颜色筛选）不要指望 EAV 表查询——同步进搜索引擎（ES）做倒排筛选，DB 里的属性值是权威源（总览篇 §38）。
- 销售属性变更要极其谨慎：改销售属性=可能改变 SKU 结构，已有交易的 SKU 禁止修改规格，只能下架重建。

## 4. 商品状态与生命周期

```text
草稿 → 待审核 → 审核通过（未上架）→ 上架 → 下架 → （逻辑删除）
```

- 上下架是**渠道级**的：同一商品可在 App 上架、小程序下架，用"商品-渠道上架表"而不是 SPU 上一个布尔字段。
- **已被订单引用的 SKU 永不物理删除**（总览篇 §27 软删除边界）：删除的 SKU 保留供历史订单展示，用状态挡住新交易。
- 审核流水独立记录（谁提交、谁审、驳回原因），商品每次重大修改重新送审时，**线上继续卖旧版本、审核通过后原子切换**——这需要商品内容版本化（见 §5）。

## 5. 商品版本与快照策略

两个不同需求，两种机制：

- **审核版本**：商品编辑产生新版本（draft version），审核通过后发布为线上版本。线上表 + 版本表，或统一版本表 + current_version 指针。
- **交易快照**：订单行固化下单时的标题/价格/规格/主图（订单篇 §1）。图文详情大快照不复制全文——详情内容本身按**内容地址存储**（内容哈希为键，相同内容只存一份），订单引用哈希，兼顾"当时页面长什么样"的可追溯和存储成本。

## 6. 价格：商品模型的邻居而非属性

价格不是商品表的一个字段，而是独立域：

```text
基础价（SKU 维度）→ 渠道价/门店价（SKU × 渠道/门店）→ 会员价（× 等级）→ 促销价（活动域计算）
```

- 价格表带生效区间（总览篇 §12 双时态：提前录入未来价格、追溯历史价格），改价即插入新区间。
- **下单时的成交价固化在订单行**，价格计算过程（命中哪些价格/优惠）记计算快照。
- 报表口径：标价、成交价、结算价是三个数，分别可追溯。

商品表里放一个 price 字段的系统，在第一个"渠道差价"需求出现时崩塌。

## 7. 平台模式补充：商家商品与平台商品库

平台电商（多商家）比自营多一层：

- **平台标准商品库（产品库）**：平台维护的标准 SPU（条码、标准图文）。
- **商家商品**：商家发布的商品，可挂接标准 SPU（同款聚合、比价）或自建。
- 好处：搜索聚合同款、类目属性质量可控；代价：挂接审核和纠错流程。
- 自营 + POP 混合模式下，货主维度要进商品和库存模型（库存篇 owner_id）。

## 8. 商品评审清单

```text
1. SPU/SKU 是否分离？订单库存价格是否全部挂 SKU？单规格是否也走 SKU？
2. 销售属性与非销售属性是否区分？SKU 唯一性是否有规格指纹约束？
3. 前后台类目是否分离？商品是否唯一挂靠后台类目？
4. 类目属性模板是否受控（类型/字典/必填）？筛选是否走搜索引擎而非 EAV 查询？
5. 上下架是否渠道级？被引用的 SKU 是否永不物理删除？
6. 编辑审核是否版本化？线上版本与草稿是否隔离？
7. 价格是否独立域、带生效区间？订单是否固化成交价和计算快照？
8. 类目/属性调整是否不污染历史订单和报表（快照固化）？
```

## 9. 总结

商品建模五条纪律：

1. **SPU 管展示，SKU 管交易**：引用层级错一处，下游痛十年。
2. **类目前后台分离**：稳定的管理树 + 灵活的导购树。
3. **属性模板化**：受控 EAV 长出无限品类，搜索交给搜索引擎。
4. **版本与快照**：编辑不影响线上，交易定格当时。
5. **价格独立成域**：商品定义"是什么"，价格域定义"卖多少"。

> 商品模型是全系统被引用最多的模型。它每省一次抽象，下游就多十处补丁。

## 10. 需要增强的企业级内容

本篇已经覆盖 SPU/SKU、类目、属性、状态、版本、价格边界和平台商品库。大型企业商品模型还需要补强单位、包装、条码、组合商品、内容资产、合规资质和商品关系。

### 10.1 计量单位与包装规格

**为什么需要**：采购、库存、销售可能使用不同单位。例如箱、包、瓶、克、千克、件之间存在换算。

**标准做法**：

```text
基础单位：库存核算最小单位
销售单位：前台销售单位
采购单位：供应商采购单位
包装单位：箱规、包规、托盘
换算关系：1 箱 = 24 瓶
```

**解决的问题**：

- 采购按箱，销售按瓶，库存按瓶时数量混乱。
- 报表无法统一数量口径。
- 条码和包装规格无法对应。

### 10.2 条码、外部编码和多编码体系

**为什么需要**：同一个 SKU 在企业内部、供应商、门店、平台、电商渠道可能有不同编码。

**标准做法**：

- SKU 内部编码。
- 国际条码或商品条码。
- 供应商货号。
- 门店 PLU。
- 渠道商品编码。
- 外部系统映射编码。

编码映射表应记录来源系统、生效状态、是否主码、冲突处理。

### 10.3 组合商品、套装和 BOM

**为什么需要**：套餐、礼盒、组合销售、餐饮套餐、生产组装都不是单一 SKU。

**标准做法**：

```text
组合销售套装：前台卖一个套装，库存扣多个子 SKU
固定 BOM：子项固定，数量固定
可选套餐：用户在规则范围内选择子项
生产 BOM：用于加工和成本核算
```

**解决的问题**：

- 套餐销售无法正确扣库存。
- 组合商品退款无法按子项处理。
- 成本和毛利无法计算。

**带来的复杂度**：

- 订单行展示商品和库存扣减商品可能不同。
- 价格分摊、优惠分摊、退款分摊都要按子项规则处理。

### 10.4 商品内容与媒体资产

**为什么需要**：商品不仅有结构化字段，还有图片、视频、详情、卖点、说明书、资质文件。

**标准做法**：

- 商品结构化信息和内容资产分离。
- 图片、视频、详情使用版本或内容哈希。
- 审核中的内容和线上内容分离。
- 订单快照只引用当时内容版本，不复制大文本。

### 10.5 合规、资质和禁限售

**为什么需要**：食品、医药、烟酒、生鲜、电子设备等商品可能需要资质、许可证、产地、效期、禁售区域。

**标准做法**：

- 类目挂资质模板。
- 商品提交时校验资质。
- 销售区域、渠道、用户年龄或身份限制独立配置。
- 订单下单时校验当时商品合规状态。

**解决的问题**：

- 不同类目合规要求无法表达。
- 商品可上架但不可合法销售。
- 历史订单无法证明当时资质有效。

### 10.6 商品关系模型

**为什么需要**：替代品、相似品、配件、赠品、升级款、同款聚合都会影响搜索、推荐、采购和售后。

**标准做法**：

```text
商品 A -> 替代商品 B
商品 A -> 配件商品 C
商品 A -> 赠品商品 D
商品 A -> 同款标准商品 E
```

关系要记录关系类型、方向、生效时间、来源和权重。

**底层思想**：

> 商品不是孤立节点，而是交易、库存、搜索、推荐和供应链网络中的核心对象。
