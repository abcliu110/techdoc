# 工作流与审批模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦审批流建模。采购要审、退款要审、调价要审、权限要审——审批是所有企业系统的横切能力。它的建模难点是"流程定义会变而流程实例要按当时的定义走完"，以及"组织在变而审批人要按规则实时解析"。

## 0. 一句话总纲

> 工作流模型三分离：流程定义（模板，版本化）、流程实例（按发起时版本运行）、任务（人的待办）。实例永远跑在发起时的定义版本上，审批人是规则在执行时刻的解析结果。

## 1. 三层结构：定义、实例、任务

```text
流程定义（process definition）—— 模板：节点、连线、审批人规则、条件（版本化！）
    ↓ 发起时绑定当时版本
流程实例（process instance）  —— 一次具体的审批：单据 X 的这一次流转
    ↓ 每到一个节点产生
任务（task）                  —— 某人（们）在某节点的一次待办与决定
```

```sql
CREATE TABLE wf_process_def (
  id          BIGINT PRIMARY KEY,
  def_key     VARCHAR(64) NOT NULL,    -- 流程标识：purchase_approval
  version     INT NOT NULL,            -- 版本号
  def_content JSON NOT NULL,           -- 节点/连线/规则的完整定义
  status      VARCHAR(16) NOT NULL,    -- DRAFT/PUBLISHED/DEPRECATED
  published_at DATETIME,
  UNIQUE KEY uk_key_ver (def_key, version)
);

CREATE TABLE wf_process_instance (
  id           BIGINT PRIMARY KEY,
  def_key      VARCHAR(64) NOT NULL,
  def_version  INT NOT NULL,           -- 固化发起时版本（铁律）
  biz_type     VARCHAR(32) NOT NULL,   -- 关联单据类型
  biz_order_no VARCHAR(64) NOT NULL,   -- 关联单据号
  status       VARCHAR(16) NOT NULL,   -- RUNNING/APPROVED/REJECTED/CANCELED/TERMINATED
  variables    JSON,                   -- 流程变量（金额、部门等条件判断输入，发起时快照）
  started_by   BIGINT NOT NULL,
  started_at   DATETIME NOT NULL,
  ended_at     DATETIME,
  KEY idx_biz (biz_type, biz_order_no)
);

CREATE TABLE wf_task (
  id           BIGINT PRIMARY KEY,
  instance_id  BIGINT NOT NULL,
  node_key     VARCHAR(64) NOT NULL,
  assignee_id  BIGINT,                 -- 处理人（认领后）
  candidate    JSON,                   -- 候选人/候选组（解析结果快照）
  status       VARCHAR(16) NOT NULL,   -- PENDING/CLAIMED/APPROVED/REJECTED/
                                       -- TRANSFERRED转办/TIMEOUT
  action_at    DATETIME,
  comment      VARCHAR(512),
  KEY idx_instance (instance_id),
  KEY idx_assignee (assignee_id, status)
);
```

铁律：

- **实例固化定义版本**：流程改版只影响新发起的实例，在途实例按旧版本走完（总览篇规则版本化的最典型场景）。在途实例太多需要迁移到新版时，走显式的**实例迁移操作**（记录谁批准迁移、从哪版到哪版），不是悄悄改。
- **流程变量发起时快照**：条件分支（金额 > 5 万走总监）用发起时的金额判断，单据后来改了金额要**重新发起审批**，而不是流程中途重算路由。

## 2. 审批人规则：晚绑定 + 结果快照

审批人不写死人名，写**规则**，到达节点时解析：

```text
规则类型：指定角色（财务经理）/ 发起人的直属上级 / 发起人部门的分管领导 /
         上一节点处理人的上级 / 表单字段指定（成本中心负责人）/ 会签组
```

- **晚绑定**：到达节点那一刻按当时组织关系解析（上月配置的流程，今天走到"直属上级"节点，按今天的汇报线找人）。
- **解析结果落快照**（task.candidate）：事后审计"为什么是他审的"能还原当时的解析依据——组织后来变了不影响解释历史。
- 解析为空（岗位空缺）是必然发生的：显式的**兜底规则**（升级到上级/转管理员/挂起告警），不能让流程无声卡死。
- **审批与授权衔接**：审批人的可批范围应与权限体系（权限篇数据权限）一致——能审批某部门采购的人，必须对该部门有数据权限，两套体系引用同一组织事实源。

## 3. 节点类型与流转语义

| 节点 | 语义 | 建模要点 |
|---|---|---|
| 单人审批 | 一人通过即过 | 候选多人时先认领（CLAIMED）防重复处理 |
| 会签 | 多人全部通过 | 每人一条 task，全部 APPROVED 才推进；一票否决即拒 |
| 或签 | 任一人通过 | 一人处理后其余 task 置 CANCELED |
| 条件分支 | 按变量路由 | 条件表达式在定义中，判断输入用实例变量快照 |
| 并行分支 | 多路同时走 | 汇聚节点等待所有分支到达 |
| 抄送 | 知会不阻塞 | 只产生已读记录，不产生待办 |

流转操作全集：通过、驳回（**驳回到哪**要显式：上一节点/发起人/指定节点，重走后续节点）、转办（换人，留转办记录）、加签（临时加一道审批，前加签/后加签）、撤回（发起人撤回，仅限首节点未处理时或按配置）、催办、超时（自动通过/自动升级/仅提醒，按节点配置）。

**每个操作都是流水**：任务表记终态，操作历史表记全过程（谁、何时、什么操作、意见、附件）——审批记录是法律证据级数据，只增不改。

## 4. 与业务单据的关系：审批不吞噬业务

- **单据状态与流程状态分离**：采购单有自己的状态机（草稿/审批中/已审批/已关闭），流程实例通过 biz_order_no 关联。审批结果**回调**业务域推进单据状态，业务域是单据的唯一权威——流程引擎不直接 UPDATE 业务表。
- **审批期间单据锁定**：送审后单据不可编辑（编辑 = 撤回重新发起），保证审批人看到的和最终生效的是同一份内容。严谨做法是送审时对**单据内容做版本/哈希快照**，审批通过时校验未被篡改。
- 一张单据多次送审（驳回后修改再发起）产生多个流程实例，全部保留——"这张单被驳回过两次、为什么"是完整历史。

## 5. 待办列表：工作流的高频查询

- 待办查询（`assignee_id + status=PENDING`，候选组的用候选人展开表或倒排）是全系统最高频查询之一，索引和分页设计对齐总览篇容量估算纪律；大型组织考虑**待办收件箱表**（任务产生时按人展开写入），读写分离。
- 代理/委托：出差期间 A 的待办自动给 B——**委托规则表**（委托人、被委托人、范围、起止时间），任务仍记原责任人 + 实际处理人两个字段，责任不因委托而转移。

## 6. 幂等、并发与恢复

- 流程推进的每一步（完成任务 → 计算下一节点 → 生成新任务）必须**事务内完成或走 Outbox**（总览篇 §14）：任务完成了但下个节点没生成 = 流程卡死，是工作流最常见的事故形态。
- 同一任务并发处理（两人同时点或签）：CAS 更新 `WHERE status='PENDING'`，输者收到"已被处理"。
- 引擎要提供**实例诊断与修复工具**：卡死实例检测（RUNNING 但无 PENDING 任务）、手工推进/跳过节点（管理员操作，全程留痕）——承认引擎会出 bug，给运维留正门。

## 7. 工作流评审清单

```text
1. 定义/实例/任务是否三层分离？实例是否固化发起时定义版本？
2. 流程变量是否发起时快照？改单是否强制重新发起？
3. 审批人是否规则化 + 晚绑定 + 解析结果快照？空缺是否有兜底？
4. 会签/或签/条件/并行/抄送语义是否明确建模？
5. 驳回目标、转办、加签、撤回、超时策略是否显式配置？
6. 操作历史是否只增不改、含意见附件、可作证据？
7. 流程状态与单据状态是否分离？引擎是否不直改业务表？
8. 送审后单据是否锁定或哈希校验防篡改？
9. 待办查询是否有容量设计？委托是否留双人记录？
10. 推进是否事务/Outbox 保证？是否有卡死检测和留痕的人工修复通道？
```

## 8. 总结

工作流建模五条纪律：

1. **三层分离**：定义是模板、实例是运行、任务是待办，各自独立演化。
2. **版本锁定**：实例跑在发起时的版本上，改版不改在途。
3. **晚绑定留快照**：审批人按执行时刻解析，解析依据永久可查。
4. **审批不越权**：引擎管流转，业务管单据，回调是唯一接口。
5. **每步皆流水**：通过驳回转办加签全部留痕，审批记录是证据。

> 审批的价值不是"走了流程"，而是多年以后仍能回答：这件事当时是谁、依据什么规则、看着什么内容、做出的决定。
