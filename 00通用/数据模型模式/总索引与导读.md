## 2. 文档清单与一句话定位

### 总览

| 文档 | 定位 |
|---|---|
| 世界级数据库模型设计常用方法与底层思想 | 全部方法论：数据角色分层、复式记账、一致性边界、双时态、事实不可变、逆向与 Saga、对账、并发热点、物理底线、敏感数据、容量估算、层级与图、存储形态、Schema 即代码、方法论对照、34 项评审清单 |

### 交易核心（钱与货的骨架）

| 专篇 | 一句话总纲 |
|---|---|
| 资金账务 | 交易/账务/资金三层，复式凭证，余额是流水的推论 |
| 订单 | 五层结构、行级为王、金额恒等式、快照定格当时 |
| 商品 | SPU 管展示 SKU 管交易，类目前后台分离，价格独立成域 |
| 库存 | 数量维度的账务系统 + 承认账实必然有差的纠偏机制 |
| 支付渠道与网关 | 不信任外部：UNKNOWN 合法、回调要验、查询确认、对账终审 |

### 增长域（营销、用户与数据）

| 专篇 | 一句话总纲 |
|---|---|
| 营销与券 | 可配置的规则 + 严格的资产（券即准资金）+ 固化的证据 |
| 会员与积分 | One ID、等级可重放、积分储值按准资金记账 |
| 报表与数仓 | 口径统一、链路可溯、随时可重算；分层固化口径 |
| 搜索建模 | 为结果建文档、索引是派生、鲜度分级、交易回源 |

### 企业域（流程、协作与合规）

| 专篇 | 一句话总纲 |
|---|---|
| 权限 | 功能权限（RBAC）+ 数据权限（范围），组织是共同事实源 |
| 工作流与审批 | 定义/实例/任务三层，实例锁定发起时版本，审批人晚绑定留快照 |
| 采购与供应链 | 单据链行级留量，三单匹配守门，付款可溯到请购 |
| 物流与履约 | 订单与包裹解耦，承诺固化执行留痕，异常即工单 |
| 风控规则引擎 | 事件/特征/规则/决策/名单五件套，每次决策可重放 |
| 发票与税务 | 留量防超开，纠错必红冲，发票账务申报三方可核对 |
| 开放平台与 OpenAPI | 身份三层、授权三方、契约刚性、推拉双保、版本即承诺 |

### 架构与基础设施（横切能力）

| 专篇 | 一句话总纲 |
|---|---|
| 多租户 SaaS | tenant_id 贯穿一切，隔离模式可迁移，计量即收入 |
| 国际化与多币种 | 金额带币种、汇率即快照、UTC 加时区、法域即配置 |
| 消息与通知 | 事件进策略出，发送即快照，合规是闸门 |
| 文件与附件存储 | 三层分离、哈希为本、删除走策略、访问即授权 |
| 定时任务与批处理 | 幂等是底线、分片可续跑、依赖靠声明、补跑走正门 |
| 审计日志与操作留痕 | 四类分治、五要素齐全、账本不可改 |
| 缓存与派生数据 | 三问定生死：事实源是谁、窗口多大、坏了怎么重建 |
| 数据归档与冷热分离 | 温度先定义、搬家有账本、销毁也合规 |

## 4. 阅读路径

**首次通读**（建立体系）：总览 → 资金账务 → 订单 → 库存 → 商品。账务是全套纪律的原型（流水、凭证、对账都源于此），先读它事半功倍。

**按角色**：

- 交易系统研发：总览 → 账务 → 订单 → 库存 → 支付网关 → 缓存派生
- 中后台/ERP 研发：总览 → 权限 → 工作流 → 采购供应链 → 审计 → 批处理
- 增长/营销研发：总览 → 营销券 → 会员积分 → 通知 → 风控
- 数据研发：总览 → 报表数仓 → 搜索 → 缓存派生 → 归档
- 架构师/评审人：总览全篇 + 各篇评审清单（见 §5）
- 出海/SaaS 团队：总览 → 多租户 → 国际化 → 开放平台

**按问题查**（常见场景 → 该翻哪里）：

| 你面对的问题 | 去读 |
|---|---|
| 钱对不上/需要对账 | 账务篇 §对账、支付网关篇 §5 |
| 超卖/秒杀热点 | 库存篇 §5、总览 §16 |
| 退款金额算错/超退 | 订单篇金额恒等式与防超退、账务篇退款案例 |
| 改了规则历史数据变了 | 总览 §17、营销篇 §2.2、工作流篇 §1 |
| 同一个指标两张报表两个数 | 报表篇 §3 指标字典 |
| 用户说没收到通知/被骚扰投诉 | 通知篇 §3 合规闸门 |
| 被薅羊毛了 | 营销篇 §4、风控篇全篇 |
| 表太大查询变慢 | 归档篇、总览 §31 容量估算 |
| 缓存和 DB 不一致 | 缓存派生篇 §2/§5 |
| 审计/监管要查三年前的操作 | 审计篇、归档篇 §4 回热 |

## 5. 评审清单汇总

每篇末尾都有该域的评审清单（8~34 项），设计评审时按域取用：

- 新建一个业务域的表：总览 §42（34 项通用清单）+ 对应专篇清单
- 上线前专项检查：涉钱的过账务+支付清单，涉库存的过库存清单，对外开放的过 OpenAPI 清单
- 年度架构体检：容量（总览 §31）、归档策略覆盖（归档篇 §2）、派生数据登记（缓存篇 §5）、审计覆盖（审计篇 §7）

## 6. 维护约定

- 专篇间引用写"篇名 + 章节"；总览内部引用章节号，专篇引用总览用"总览篇 §N"。
- 各篇的"需要增强的企业级内容"章节（商品/库存/订单/账务四篇末尾）为后续深化的路标，增补内容时优先消化这些章节。
- 新增专篇的门槛：该域有独立的表结构、独立的纪律、且被两个以上现有篇引用为"外部系统"——否则作为现有篇的章节扩展。
- 每篇遵循统一结构：一句话总纲 → 分层/核心模型（含 DDL）→ 各专题纪律 → 评审清单 → 五条纪律总结。

> 这套文档的目标不是穷尽所有表怎么建，而是让读者带走那组反复出现的底层纪律——当遇到没有专篇覆盖的新领域时，能自己推导出"这里的流水是什么、快照是什么、幂等键是什么、对账对什么"。
