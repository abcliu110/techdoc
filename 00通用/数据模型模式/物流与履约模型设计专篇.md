# 物流与履约模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦履约建模。支付成功只是承诺的开始，履约是把承诺变成"货到人手"的全过程：拆单、寻源、出库、干线、配送、签收、逆向。履约域的特点是**物理世界主导**——包裹会丢、会破、会拒收，模型必须与承运商、仓库这些"半失控的外部执行者"共存。

## 0. 一句话总纲

> 履约模型的核心：订单与包裹解耦（一单可多包、一包可多单行），每个物理动作都有单据和轨迹，承诺（时效）与执行（轨迹）分离建模、互相对照。

## 1. 履约主链路与单据体系

```text
支付成功 → 履约单（fulfillment order）→ 拆单/寻源 → 出库单（发给WMS）
        → 包裹（parcel）→ 运单（发给TMS/承运商）→ 轨迹 → 签收/异常
```

| 单据 | 归属域 | 职责 |
|---|---|---|
| 履约单 | 履约中台 | 订单的履约视角副本：履约什么、承诺何时到 |
| 出库单 | 仓库(WMS) | 指挥仓内作业：拣、复核、打包 |
| 包裹 | 履约中台 | 物理包裹：重量体积、内含明细 |
| 运单 | 运输(TMS) | 承运商合同：单号、路由、费用 |

**为什么履约单独立于订单**：订单是交易合同（不可变部分多），履约是执行过程（改址、拆合、换仓是常态）。订单直接驱动仓库的系统，第一次"部分缺货分开发"需求就会把订单表改得面目全非。

## 2. 拆单与合单：行级映射是地基

```text
订单行 —— N:M —— 包裹明细行
```

拆单的触发因素：多仓发货（寻源结果不同仓）、大件与小件分运、预售与现货分开、超重拆包。合单：同一用户同一地址的多笔订单合并发货省运费。

```sql
CREATE TABLE ful_parcel_item (
  id             BIGINT PRIMARY KEY,
  parcel_id      BIGINT NOT NULL,
  order_line_id  BIGINT NOT NULL,   -- 订单行
  qty            INT NOT NULL,      -- 本包裹含该行多少件
  KEY idx_parcel (parcel_id),
  KEY idx_line (order_line_id)
);
```

- 订单行维护**已发货量**（`shipped_qty`），拆包时原子校验 Σ包裹明细 ≤ 订单行量（防超发，与防超退同构）。
- 用户视角的"我的包裹"、客服视角的"这行商品在哪个包裹"，全靠这张映射表——它是履约域的事实中枢。
- 部分发货时订单状态的展示（"部分已发货"）是**从包裹状态聚合计算**的，不在订单上手工维护。

## 3. 时效承诺：承诺是数据，不是文案

- **承诺产生在下单前**（"预计明天送达"影响转化），承诺引擎输入：仓库存、仓作业截单时间（cut-off）、承运商时效表（发货地 × 收货地 × 产品的时效矩阵）、节假日日历。
- **下单即固化承诺**：承诺送达时间写入履约单（快照），后续考核以此为准——承诺引擎后来变聪明了，不能改写历史承诺。
- 每个履约节点带**计划时间 vs 实际时间**（计划出库时间/实际出库时间……），逐段偏差定位延误责任段（仓慢了还是配送慢了），这是履约考核与赔付的数据基础。

## 4. 运单与承运商：外部执行者管理

```sql
CREATE TABLE tms_waybill (
  id            BIGINT PRIMARY KEY,
  waybill_no    VARCHAR(48) NOT NULL,   -- 承运商运单号
  carrier_code  VARCHAR(32) NOT NULL,
  parcel_id     BIGINT NOT NULL,
  status        VARCHAR(16) NOT NULL,   -- CREATED/PICKED揽收/IN_TRANSIT/DELIVERING/
                                        -- SIGNED/REJECTED拒收/RETURNED/LOST/EXCEPTION
  est_fee       DECIMAL(18,4),          -- 预估运费（按报价卡）
  billed_fee    DECIMAL(18,4),          -- 承运商账单费用（对账后回填）
  UNIQUE KEY uk_no (carrier_code, waybill_no)
);
```

- 电子面单**取号是资源分配**：号段预取、取号幂等、废弃回收，与发号器纪律一致。
- **承运商报价卡**：承运商 × 线路 × 重量段 × 计费规则（首重续重/抛比），带生效区间。运单固化预估运费，月度与承运商账单**逐单对账**（billed_fee 回填，差异挂差错处理）——运费对账是被普遍忽视的资损点，量大后差额惊人。
- 轨迹表**只增不改**：承运商推送/抓取的每个节点原样落库（时间、地点、动作、原文），状态机从轨迹推导。轨迹乱序（先收到"派送中"后收到"揽收"）是常态，状态推进要按业务时间且只进不退。

## 5. 异常件：履约域的差错户

物理世界的失败形态繁多，全部进**异常工单**统一处理：

```text
异常类型：丢件 / 破损 / 拒收 / 超时未揽收 / 多次派送失败 / 错发 / 少发
工单内容：关联运单+包裹+订单行 | 责任初判（仓/承运商/商家）| 处理动作 | 赔付
```

- **拒收/派送失败的货**走逆向入库（退货入库单 → 质检 → 回可售或残次），货权状态全程有账。
- **丢件**触发双线处理：对客（补发或退款，走正向/逆向流程）与对承运商（索赔单，赔付金额挂应收）——两条线独立结案，不互相等待。
- 补发是**新履约单**（引用原履约单和异常工单），不是把原运单改个号：补发的成本归因（谁的责任谁承担）靠单据链。

## 6. 逆向物流：退货的物理链路

```text
退货申请（订单域）→ 退货物流单（用户寄回/上门取件）→ 逆向入库 → 质检 → 
  合格回可售 / 残次报损 → 触发退款（资金域）
```

- **退款时点策略**显式配置：仓检后退款（防"退回来的是砖头"）/ 签收即退（体验优先）/ 信用免检退（风控分层）——策略是配置和风控规则，不是代码分支。
- 退货入库必须**关联原订单行**（哪一单退回来的），无头件（收到货找不到源单）进无头件池人工认领，账上是"待归属库存"不是凭空入库。

## 7. 运费成本与分摊

- 一个包裹多笔订单（合单）时运费按规则分摊到订单（按重量/金额比例），分摊明细落库——毛利核算到订单粒度的前提。
- 运费三个数分开：**用户支付运费**（订单收入）、**预估成本**（报价卡）、**实际成本**（承运商对账）——三者对照才能算清"包邮到底亏多少"。

## 8. 履约评审清单

```text
1. 履约单是否独立于订单？拆合单是否行级映射（订单行 N:M 包裹明细）？
2. 订单行是否维护已发货量并原子校验防超发？
3. 时效承诺是否下单时固化？各节点是否记录计划 vs 实际时间？
4. 运单号取号是否幂等？轨迹是否只增不改、状态只进不退、容忍乱序？
5. 承运商报价卡是否带生效区间？月度运费是否逐单对账？
6. 异常件是否统一工单？丢件是否对客对承运商双线独立处理？
7. 补发是否新履约单并关联责任归因？
8. 逆向是否关联原订单行？无头件是否有归属流程？退款时点策略是否配置化？
9. 合单运费是否分摊到订单？收入/预估/实际三个运费数是否分离可对照？
```

## 9. 总结

履约建模五条纪律：

1. **订单与包裹解耦**：行级 N:M 映射，拆合单是常态不是例外。
2. **承诺固化、执行留痕**：下单锁定承诺，节点记录计划与实际。
3. **轨迹只增、状态只进**：外部推送原样落库，乱序靠推导吸收。
4. **异常即工单**：每种物理失败都有单据、责任归因和双线结案。
5. **运费三数对照**：收的、估的、付的分开记，逐单对账不吃哑巴亏。

> 履约是电商体验的最后一公里，也是模型与物理世界摩擦最剧烈的地方。好的履约模型不假装世界不出错，而是让每次出错都有单据、有责任人、有闭环。
