# 价格促销模型设计专篇 - 系统化逐行拆解 v2

> 本文不替代原文档，也不修改原文档。本文用于把《价格促销模型设计专篇》中的每个关键句、每个清单项逐条拆开，说明为什么这样设计、解决什么问题、如果不这样会怎样、会带来什么复杂度，以及大型企业系统的标准做法。

## 0. 总纲逐行拆解

### 原文：价格促销模型的核心，是让“为什么这个人此刻以这个价格成交”可以被计算、解释、复现和审计。

**本质**：价格不是一个静态数字，而是由人、货、场、时间、渠道、会员身份、促销规则、优惠资产、税费、运费共同计算出来的结果。

**为什么这样定义**：真实交易中，同一个 SKU 在不同门店、不同渠道、不同会员等级、不同活动时间、不同客户合同下，成交价可能不同。如果系统只保存一个最终价格，就无法解释这个价格是怎么算出来的。

**解决的问题**：

- 客服问“为什么这个客户买得更便宜”时能解释。
- 财务问“优惠是谁承担”时能核算。
- 退款时能按原成交规则回退。
- 审计时能复现当时的价格计算。
- 活动复盘时能分析哪个规则生效。

**如果不这样做**：

- 历史订单只能看到成交价，无法解释价格来源。
- 规则变更后，历史订单无法复算。
- 退款、结算、开票、毛利都会出现口径争议。
- 系统容易出现资损，但找不到是哪条规则导致。

**带来的复杂度**：

- 需要价格规则、促销规则、优惠分摊、计算快照。
- 需要规则版本和生效时间。
- 需要价格计算引擎或统一计算服务。

**大型企业标准做法**：

```text
价格规则表
促销规则表
优惠资产表
价格计算结果快照
订单金额明细
优惠分摊明细
规则版本记录
```

## 1. 逻辑模型

### 1.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| prc_price_rule 价格规则 | price_type(基础/渠道/门店/客户/会员/合同/阶梯/活动), target_scope 适用对象, sku_id/spu_id/category_id, price, currency, effective_start/end, priority, version, status | 成交价的唯一权威来源；对谁、何时、多少钱，多条命中靠优先级 |
| prc_tier_price 阶梯明细 | price_rule_id, min_qty, max_qty, unit_price | 数量区间定档；拆单/退款按原命中档位回退 |
| prm_activity 促销活动 | activity_id, activity_type(满减/满折/买赠/第N件/套餐), 适用商品/门店/渠道/会员范围, 门槛, 优惠方式参数, effective_start/end, rule_version, status | 临时优惠策略；规则只发新版本不原地改 |
| prm_stack_rule 叠加排斥 | activity_id, 优惠分组(商品/订单/支付/运费级), 排斥组ID, 叠加上限 | 多规则组合的确定性，防重复打折 |
| prm_budget 活动预算 | activity_id, cost_bearer 出资方, total_budget, used_budget, frozen_budget | 营销成本上限；原子扣减防超发 |
| prc_calc_snapshot 计算快照 | order_id, 输入快照(用户/渠道/会员等级), 命中规则ID+版本列表, 排斥明细, calculation_detail | "为什么是这个价"的复现依据，规则变了历史仍可解释 |
| prc_discount_allocation 优惠分摊 | order_line_id, 优惠类型, activity_id/coupon_id, cost_bearer, 分摊金额, rule_version, 尾差标记 | 订单级优惠拆到行；退款/结算/开票/毛利的行级依据 |
| prc_price_change_log 调价流水 | price_rule_id, 旧值/新值, operator, 审批单号, 时间 | 价格是高风险配置，改动全留痕 |

### 1.2 实体关系

```text
prc_price_rule (1) ──< prc_tier_price (N)        阶梯价的档位明细
prm_activity (1) ──< prm_stack_rule (N)          活动挂叠加/排斥配置
prm_activity (1) ──< prm_budget (N)              一活动可多出资方分账预算
prc_calc_snapshot (1) ──< prc_discount_allocation (N)  快照展开为行级分摊
prc_price_rule (1) ──< prc_price_change_log (N)  调价流水只增
外部引用：sku_id/spu_id/category_id → 商品域；order_id/order_line_id → 订单域
         （成交价固化在订单行）；会员等级/客户ID → 会员域（下单时固化）；
         coupon_id → 券域（引用不落外键，应用层校验）
```

### 1.3 关键约束与不变量落点

```text
唯一键：(price_type, target_scope, sku_id, effective_start) 防同范围同时段重复规则；
       (order_id) 计算快照一单一份；分摊幂等键 (order_line_id, activity_id/coupon_id)
留量校验：预算扣减原子 UPDATE：
         used_budget = used_budget + x WHERE used_budget + frozen_budget + x ≤ total_budget
恒等式：应付 = 商品成交金额 + 运费 + 税费 - Σ优惠；
       Σ行分摊金额 = 该优惠总额（尾差按固定规则归属，全局一致）；
       行成交金额 - 分摊优惠 ≥ 0（防负价）
版本化规则：price_rule / activity 修改即发新 version，历史订单引用旧版本复现；
           任何来源的价格（算法/人工改价）都必须落规则表和快照
状态机约束：DRAFT → PENDING_APPROVAL → PUBLISHED → DISABLED/EXPIRED，
           编辑中的规则不影响交易；活动到期自动失效
```

## 2. “价格不是商品字段”逐行拆解

### 原文：商品定义“是什么”，价格定义“卖多少”。

**本质**：商品模型表达对象身份，价格模型表达交易条件。二者生命周期不同。

**为什么**：商品名称、规格、品牌、类目相对稳定；价格会随渠道、门店、客户、会员等级、活动、合同、时间频繁变化。

**解决的问题**：

- 支持同一商品不同渠道不同价。
- 支持同一商品不同客户合同价。
- 支持同一商品不同时间段价格。
- 支持价格审批和生效失效。

**如果不这样做**：

- 商品表出现 `price`、`vip_price`、`shop_price`、`channel_price`、`activity_price` 等字段膨胀。
- 每新增一种价格场景都要改商品表。
- 商品主数据变成交易规则容器，越来越难维护。

**底层思想**：

> 商品是名词，价格是关系。关系不应该硬塞进名词本体。

### 原文：价格应独立建模。

**本质**：价格应成为一个独立领域对象，拥有自己的适用范围、时间、优先级、版本和发布流程。

**标准模型**：

```text
price_rule
- price_rule_id
- sku_id / spu_id / category_id
- price_type
- target_scope
- price
- currency
- effective_start
- effective_end
- priority
- version
- status
```

**带来的复杂度**：

- 下单时不能直接读商品价格，要调用价格计算。
- 多条价格同时命中时要有优先级。
- 价格发布要考虑未来生效和历史保留。

## 3. 价格类型逐条拆解

### 基础价

**含义**：SKU 的默认销售价格，是所有价格计算的起点。

**为什么需要**：没有基础价，系统无法在没有活动、没有会员价、没有合同价时给出默认价格。

**解决的问题**：

- 商品默认可售。
- 报表可比较标价和成交价。
- 促销可基于基础价计算折扣。

**风险**：

- 基础价不能等同于最终成交价。
- 基础价变化不能污染历史订单。

### 渠道价

**含义**：同一商品在不同销售渠道上的价格，例如 App、小程序、POS、外卖平台、批发渠道。

**为什么需要**：渠道成本、用户群、补贴方式不同，价格策略自然不同。

**解决的问题**：

- 线上线下不同价。
- 外卖平台加价。
- 批发和零售分离。

**如果没有渠道价**：

- 只能为每个渠道复制商品。
- 或在订单逻辑里写特殊价格判断。

### 门店价

**含义**：同一商品在不同门店或区域的价格。

**为什么需要**：门店租金、消费水平、库存压力、运营策略不同。

**解决的问题**：

- 区域差异化定价。
- 门店自主调价。
- 临期或滞销门店局部促销。

**复杂度**：

- 门店价和渠道价同时存在时，需要优先级。
- 门店调价要有审批和生效时间。

### 客户价

**含义**：针对特定客户的价格，常见于 B2B、经销商、企业客户。

**为什么需要**：B2B 客户有合同、账期、采购量和谈判价格。

**解决的问题**：

- 大客户专属价格。
- 经销商等级价格。
- 合同价格锁定。

**如果没有客户价**：

- 销售只能线下改价。
- 订单价格无法审计。
- 客户争议时无法证明价格依据。

### 会员等级价

**含义**：按会员等级给出的价格。

**为什么需要**：会员体系需要用价格差异体现等级权益。

**解决的问题**：

- VIP 价。
- 会员日价格。
- 等级权益可量化。

**风险**：

- 下单时必须固化当时会员等级。
- 会员升级/降级不能改变历史订单价格。

### 合同价

**含义**：基于合同约定的价格，通常有有效期、客户、商品范围、数量条件。

**为什么需要**：企业采购、供应、批发交易通常以合同为准。

**解决的问题**：

- 合同期间锁价。
- 价格审批。
- 法务和财务可追溯。

**复杂度**：

- 合同价可能和活动价冲突。
- 合同价通常优先级较高，甚至禁止叠加促销。

### 阶梯价

**含义**：购买数量越多，单价越低。

**为什么需要**：批发、采购、B2B 和大宗交易常用。

**示例**：

```text
1-9 件：10 元
10-99 件：9 元
100 件以上：8 元
```

**解决的问题**：

- 鼓励大批量采购。
- 支持批发价格策略。

**风险**：

- 退款后是否仍满足阶梯价要定义。
- 拆单后阶梯计算口径要明确。

### 临时活动价

**含义**：在某个活动期间临时生效的价格。

**为什么需要**：促销活动通常不应该永久修改基础价格。

**解决的问题**：

- 限时抢购。
- 秒杀。
- 节日活动。
- 清仓。

**风险**：

- 活动结束后必须自动失效。
- 活动价和优惠券是否叠加要明确。

## 4. 价格字段逐条拆解

### 适用对象

**含义**：这条价格规则对谁、对什么商品、在哪个范围生效。

**可能包括**：

```text
SKU
SPU
类目
品牌
客户
会员等级
门店
渠道
区域
租户
```

**为什么需要**：价格没有适用对象，就不知道该给谁用。

**如果缺失**：

- 价格规则可能误伤全量用户。
- 活动价可能在不该生效的门店生效。
- 大客户价格可能被普通用户使用。

### 生效时间

**含义**：价格从什么时候开始有效。

**为什么需要**：价格常常提前配置，未来自动生效。

**解决的问题**：

- 支持预约调价。
- 支持活动准点开始。
- 支持价格审批后定时发布。

### 失效时间

**含义**：价格到什么时候结束。

**为什么需要**：活动价、合同价、会员价都有边界。

**如果没有**：

- 活动价可能永久生效。
- 合同到期后仍按旧价销售。
- 历史价格无法区分时段。

### 币种

**含义**：价格使用的货币。

**为什么需要**：多币种系统不能把 CNY、USD、HKD 混算。

**标准做法**：

- 金额必须和币种成对出现。
- 不同币种不能直接比较或相加。
- 汇率必须有时间和来源。

### 优先级

**含义**：多条价格同时命中时，哪条优先生效。

**为什么需要**：同一个 SKU 可能同时命中基础价、门店价、会员价、合同价、活动价。

**如果没有**：

- 不同代码路径可能选出不同价格。
- 历史订单无法复现。
- 价格争议无法解释。

**示例优先级**：

```text
合同价 > 客户价 > 活动价 > 会员等级价 > 门店价 > 渠道价 > 基础价
```

实际优先级必须由业务确认。

### 版本

**含义**：价格规则每次变化都形成一个新版本。

**为什么需要**：历史订单要知道当时命中的是哪一版价格。

**如果没有版本**：

- 改了价格规则后，历史订单无法复算。
- 审计无法确认当时价格是否正确。
- 价格争议只能靠猜。

### 发布状态

**含义**：价格规则是否草稿、待审核、已发布、已停用。

**为什么需要**：价格是高风险配置，不能编辑中就影响交易。

**标准状态**：

```text
DRAFT -> PENDING_APPROVAL -> PUBLISHED -> DISABLED / EXPIRED
```

## 5. 价格计算链路逐行拆解

### 基础价 -> 渠道/门店价

**含义**：先取商品默认价，再根据销售场景替换或调整。

**为什么**：渠道和门店是价格最常见的差异来源。

**风险**：如果渠道价和门店价都命中，必须定义优先级或组合规则。

### 渠道/门店价 -> 客户/会员价

**含义**：在销售场景价格基础上，再考虑购买者身份。

**为什么**：同一个门店里，不同会员或客户可能有不同价格。

**风险**：客户合同价通常不应再被普通会员价覆盖。

### 客户/会员价 -> 活动价

**含义**：活动可能进一步降低价格，也可能覆盖原价。

**为什么**：活动是临时策略，不应该永久改变基础价格。

**风险**：活动价和会员价是否叠加必须明确。

### 活动价 -> 优惠券/满减/折扣

**含义**：在商品价格基础上应用订单级或商品级优惠。

**为什么**：优惠券和满减通常不是价格本身，而是交易优惠。

**风险**：如果优惠作用层级不清楚，退款和结算会算错。

### 优惠 -> 运费/税费

**含义**：应付金额不仅有商品价，还可能有运费和税费。

**为什么**：真实订单金额必须包含全部应收应付构成。

**风险**：税前优惠还是税后优惠，必须定义。

### 运费/税费 -> 应付金额

**含义**：最终用户需要支付的金额。

**不变量**：

```text
应付金额 = 商品成交金额 + 运费 + 税费 - 优惠
```

这个恒等式必须可校验。

## 6. 计算快照逐条拆解

### 命中的价格规则

**为什么记录**：解释基础成交价来自哪条规则。

**例子**：SKU 基础价 100，会员等级价 90，活动价 80。最终采用 80，必须知道为什么。

### 命中的促销规则

**为什么记录**：解释优惠来自哪个活动。

**例子**：满 100 减 20、平台券减 10、会员日 9 折，哪些命中、哪些被排斥，都要留痕。

### 命中的会员权益

**为什么记录**：会员价、积分抵扣、专属券、权益包都和会员身份有关。

**风险**：会员等级后续变化后，历史订单仍要按当时会员权益解释。

### 优惠分摊结果

**为什么记录**：退款、结算、开票、毛利都需要行级金额。

**例子**：订单 100 元，用了 10 元券，买了 A 商品 60、B 商品 40。优惠要分摊为 A 6 元、B 4 元，否则退 A 时不知道退多少优惠。

### 尾差处理

**为什么记录**：金额按比例分摊可能除不尽。

**例子**：10 元优惠分摊到 3 行，可能是 3.33、3.33、3.34。最后 0.01 给谁必须有规则。

### 规则版本

**为什么记录**：规则变化后仍能复现历史订单。

**标准做法**：订单金额快照中保存 `rule_id + rule_version + calculation_detail`。

## 7. 促销规则模型逐条拆解

### 活动类型

**含义**：促销属于满减、折扣、直降、买赠、优惠券、套餐价等哪一种。

**为什么需要**：不同活动类型的计算方式、叠加规则、退款规则不同。

### 适用商品

**含义**：活动对哪些商品生效。

**为什么需要**：不是所有促销都全场通用。

**风险**：商品范围不清会导致不该优惠的商品被优惠。

### 适用门店/渠道

**含义**：活动在哪些销售场景生效。

**为什么需要**：线上活动、门店活动、外卖活动通常不同。

### 适用会员

**含义**：活动是否只对特定会员等级、人群、标签、客户生效。

**为什么需要**：精准营销和会员权益依赖人群范围。

### 使用门槛

**含义**：满多少金额、买多少件、达到什么条件才能用。

**风险**：门槛计算口径必须明确，是优惠前金额还是优惠后金额，是订单金额还是指定商品金额。

### 优惠方式

**含义**：减金额、打折、赠品、免运费、积分抵扣等。

**为什么需要**：不同优惠方式影响订单金额、库存、权益和结算。

### 叠加规则

**含义**：这条优惠能否和其他优惠一起使用。

**为什么需要**：防止多重优惠叠加造成负价或资损。

### 排斥规则

**含义**：这条优惠不能和哪些优惠同时使用。

**例子**：合同价商品不参与平台券；秒杀价不再叠加满减。

### 预算

**含义**：活动最多承担多少成本。

**为什么需要**：营销费用必须受控。

**风险**：预算扣减必须幂等，且高并发下不能超发。

### 生效时间

**含义**：活动什么时候开始和结束。

**风险**：活动开始前提前生效或结束后仍生效，都是资损风险。

### 规则版本

**含义**：活动规则每次修改都形成版本。

**为什么需要**：活动过程中修改门槛或优惠力度后，历史订单仍按旧规则解释。

## 8. 常见促销类型逐条拆解

### 满减

**模型**：满足金额或数量门槛后减固定金额。

**关键口径**：满减门槛按哪些商品、优惠前还是优惠后、是否含运费。

### 满折

**模型**：满足门槛后按比例折扣。

**风险**：折扣结果的分摊和尾差必须明确。

### 直降

**模型**：商品直接降低价格。

**风险**：直降更像价格规则，不一定是订单优惠，要区分。

### 买赠

**模型**：购买指定商品后赠送商品或权益。

**风险**：赠品涉及库存、退货、成本和税务。

### 第 N 件优惠

**模型**：第 2 件半价、第 3 件免费等。

**风险**：多 SKU 组合时，哪件算第 N 件必须明确。

### 套餐价

**模型**：多个商品组合成一个优惠价格。

**风险**：库存扣减是子商品，订单展示是套餐，退款要能拆分。

### 优惠券

**模型**：用户持有的可核销优惠资产。

**风险**：券有发放、领取、锁定、核销、过期、退回状态，不是简单金额字段。

### 会员价

**模型**：按会员等级或身份生效的专属价格。

**风险**：会员等级变化不能影响历史订单。

## 9. 优惠叠加和排斥逐条拆解

### 优惠分组：商品级、订单级、支付级、运费级

**为什么分组**：不同优惠作用对象不同。

```text
商品级：直降、单品折扣
订单级：满减、订单券
支付级：银行卡优惠、支付渠道立减
运费级：免运费券
```

**如果不分组**：所有优惠混在一起，退款、结算、对账无法分清影响范围。

### 优先级：先商品价，再订单优惠，再支付优惠

**为什么**：先确定商品成交价，再算订单级优惠，最后算支付侧优惠，符合交易和资金顺序。

**风险**：如果支付优惠参与商品分摊，会导致商户结算和渠道对账混乱。

### 排斥组：同组不可叠加

**为什么**：同一层级的优惠通常只能选一个最佳方案，避免重复打折。

**例子**：秒杀价和会员折扣不可叠加。

### 叠加组：允许叠加但有上限

**为什么**：有些活动允许同时使用，例如商品直降后还能用平台券，但需要限制最低价、预算和优惠总额。

### 预算控制：平台、商家、供应商分别承担

**为什么**：优惠不是凭空消失的钱，必须知道谁承担成本。

**如果不记录**：

- 平台和商家结算算不平。
- 促销成本无法归因。
- 毛利报表错误。

## 10. 优惠分摊逐条拆解

### 原文：订单级优惠必须分摊到行。

**为什么必须**：订单级优惠发生在订单总额上，但退款、结算、开票、毛利都是行级处理。

**例子**：

```text
订单：
A 商品 60 元
B 商品 40 元
订单满减 10 元

分摊：
A 分摊 6 元
B 分摊 4 元

如果退 A：
应退 A 的成交金额 = 60 - 6 = 54
```

**如果不分摊**：

- 退某一行时不知道退多少优惠。
- 商家结算不知道优惠由哪个商品承担。
- 开票金额无法到行。
- 毛利无法按商品计算。

### 订单行

**含义**：分摊必须落到具体订单行。

**为什么**：订单行是商品、数量、价格、成本、税率、退款、结算的最小颗粒。

### 优惠类型

**含义**：满减、券、折扣、支付优惠等。

**为什么**：不同优惠类型有不同退款、结算和出资规则。

### 活动或券 ID

**含义**：优惠来源。

**为什么**：用于活动复盘、预算扣减、券核销追踪。

### 出资方

**含义**：谁承担这笔优惠成本。

**可能值**：

```text
平台
商家
供应商
渠道
品牌方
```

**为什么必须明确**：优惠影响利润和结算。平台补贴和商家让利在财务上完全不同。

### 分摊金额

**含义**：这条优惠分到该订单行的金额。

**不变量**：

```text
所有行分摊金额之和 = 该优惠总额
```

### 规则版本

**为什么**：同一活动修改规则后，历史分摊仍要按旧版本解释。

### 尾差归属

**为什么**：金额分摊常常除不尽，最后 0.01 必须有归属。

**标准做法**：

- 最后一行吃尾差。
- 或金额最大行吃尾差。
- 或按固定排序最后一行吃尾差。

规则必须全局一致。

## 11. 防资损设计逐条拆解

### 金额恒等式校验

**含义**：

```text
应付金额 = 商品金额 + 运费 + 税费 - 优惠金额
实付金额 = 成功支付金额之和
```

**为什么**：这是防止价格计算错误进入订单的最后闸门。

### 优惠不得超过商品金额

**为什么**：避免商品成交金额变负。

**注意**：如果存在赠品或补贴，需要明确是营销成本还是负价销售，不能模糊处理。

### 预算不得超发

**为什么**：营销活动有成本上限，高并发领取或下单时必须防止预算穿透。

**标准做法**：

- 预算账户化。
- 预算扣减幂等。
- 高并发用原子扣减或预占。

### 券不得重复核销

**为什么**：券是用户资产，重复核销会直接资损。

**标准做法**：

- 核销记录唯一键。
- 券状态 CAS 更新。
- 支付失败释放锁定。

### 叠加规则必须确定

**为什么**：规则不确定会导致不同代码路径算出不同价格。

**标准做法**：优惠计算引擎集中处理，不允许各业务各自实现。

### 计算结果必须可重放

**为什么**：历史订单、客服争议、审计、活动复盘都需要复现价格。

**标准做法**：保存输入快照、命中规则、规则版本、计算明细。

### 高风险活动灰度发布

**为什么**：价格促销错误会快速造成大范围资损。

**标准做法**：

- 先测试环境。
- 再指定门店/渠道/人群灰度。
- 再全量。
- 发布后监控订单金额、优惠总额、预算消耗、负价订单。

## 12. 评审清单逐条拆解

### 1. 价格是否独立于商品？

看的是商品表是否还承担价格规则职责。如果商品表只存展示用参考价可以接受；如果所有成交价都从商品表 price 字段来，就是风险。

### 2. 价格是否有生效区间、适用范围、优先级和版本？

这四个字段分别解决：

```text
生效区间：什么时候有效
适用范围：对谁有效
优先级：多条命中选谁
版本：历史如何复现
```

缺一个都可能导致价格不可解释。

### 3. 价格计算链路是否可解释？

要求能从订单看到：

```text
原价是多少
命中什么价格
用了什么优惠
为什么排斥其他优惠
最终为什么是这个金额
```

### 4. 促销规则是否支持叠加、排斥和预算？

这是防资损核心。只有优惠规则，没有组合规则，系统迟早会在活动叠加时出错。

### 5. 优惠是否分摊到订单行？

这是退款、结算、开票、毛利的前提。

### 6. 出资方是否记录？

这是结算和利润核算的前提。

### 7. 退款、结算、开票是否能按分摊结果处理？

这是验证分摊模型是否真正可用，而不是只存了字段。

### 8. 规则变更后历史订单是否可复现？

这是验证规则版本和计算快照是否足够。

### 9. 是否有防重复核销、防超预算、防负价校验？

这是价格促销系统最基本的资损护栏。

## 13. 对“不变量”逐条加深解释

### 不变量 1：价格不是商品字段。

**真正含义**：商品表不能作为成交价规则的唯一来源。商品可以有参考价、展示价，但成交价必须由价格模型和促销模型计算。

**原因**：价格是“商品 + 人 + 场 + 时间 + 规则”的关系，不是商品本身的固有属性。

### 不变量 2：价格必须有适用范围和生效区间。

**真正含义**：任何价格都必须回答“对谁、对什么、在哪里、什么时候有效”。

**原因**：没有范围就会误用，没有时间就无法历史复现。

### 不变量 3：促销规则必须有版本。

**真正含义**：规则不是被修改，而是发布新版本。历史订单引用旧版本。

**原因**：促销规则变化频繁，历史成交不能被当前规则重新解释。

### 不变量 4：订单级优惠必须分摊到行。

**真正含义**：订单总优惠必须拆到每个订单行，成为行级成交金额的一部分。

**原因**：退款、开票、结算、毛利都以订单行为颗粒。

### 不变量 5：出资方必须明确。

**真正含义**：每一笔优惠金额必须知道由平台、商家、供应商、渠道或品牌方中的谁承担。

**原因**：优惠不是凭空消失，它影响利润、结算、财务和活动预算。

### 不变量 6：成交价必须能复现。

**真正含义**：给定订单当时的用户、商品、渠道、规则版本和优惠资产，系统应能解释最终价格。

**注意**：复现不一定要重新调用当前规则引擎，因为当前规则已变；可以通过订单价格快照复现。

### 不变量 7：优惠不得造成非法负价或超预算。

**真正含义**：促销系统必须有资损边界。

**原因**：促销错误的影响通常是即时、大规模、不可完全追回。

## 14. 最小示例

```text
商品：SKU-A 基础价 100
渠道：小程序渠道价 95
会员：VIP 价 90
活动：满 80 减 10，平台出资
券：用户持有 5 元券，商家出资
运费：0

成交：
基础价 100
命中小程序渠道价 95
命中 VIP 价 90
满减 -10
券 -5
应付 75

快照必须记录：
- 渠道价规则 ID 和版本
- VIP 价规则 ID 和版本
- 满减活动 ID 和版本
- 券 ID 和核销记录
- 满减出资方：平台
- 券出资方：商家
- 分摊到订单行的优惠金额
```

这个例子说明：价格促销模型不是为了“算出 75”，而是为了三个月后仍能解释为什么是 75，以及退款、结算、报表该怎么处理。

## 15. 同一模型的多业务形态建模

定价形态多样，但支撑它们的骨架只有一套：**价格规则 + 计算快照 + 优惠分摊，形态差异下沉到 price_type 和规则参数**。

### 15.1 统一骨架（所有形态不变的部分）

```text
price_rule（price_type 标识形态，带适用范围/生效区间/优先级/版本）→ 价格计算引擎 → 计算快照 → 订单金额明细 → 优惠分摊明细
不变量全集：价格不是商品字段、规则版本化、成交价可复现、订单级优惠分摊到行、出资方明确、防负价防超预算
```

骨架决不为某个形态特化——任何形态的"为什么这个人此刻以这个价格成交"都由同一条规则→快照→分摊链路回答。

### 15.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 零售一口价 | 全场统一、变化少 | 只用基础价一条 price_rule（标准形态，骨架原型），快照仍必须落 rule_id + version |
| B2B 合同价 | 按客户锁价、有审批、禁止叠加促销 | price_type=CONTRACT，target_scope 挂客户 ID + 合同 ID；优先级置顶且排斥组包含全部促销；发布走 DRAFT→PENDING_APPROVAL→PUBLISHED 状态机 |
| 阶梯批发价 | 单价随数量变化 | price_type=TIER，规则参数存数量区间价格表；计算引擎按行数量选档，快照记录命中档位；拆单/部分退款按原命中档位口径回退 |
| 动态定价 | 价格由算法实时给出 | 算法只产出候选价，仍写入 price_rule（短生效区间 + 来源=算法版本号）；不允许绕过规则表直接下单，否则快照无法复现 |
| 秒杀价 | 瞬时生效失效、高并发、限购 | price_type=FLASH，生效区间精确到秒；库存预占和限购在营销域，价格域只保证到点自动失效 + 排斥其他优惠；快照记录活动 ID |
| 套餐价 | 价格属于组合不属于单品 | 套餐作为虚拟 SKU 挂 price_rule；成交后按子商品基础价占比把套餐优惠分摊到行（复用优惠分摊 + 尾差规则），退单品按分摊后金额退 |

### 15.3 建模决策规则

- **能用 price_type + 规则参数解决的，不建新价格表**：合同价、阶梯价、秒杀价共用 price_rule 结构，差异在类型、参数 JSON 和优先级。
- **任何来源的价格都必须过规则表和快照**：算法价、人工改价也要落 rule 记录（来源标识），否则"成交价可复现"不变量被打穿。
- **形态差异优先用优先级和排斥组表达，不写 if-else**：合同价压过活动价、秒杀排斥满减，都是配置而非代码分支。
- **拆独立域的信号**：当定价依赖实时竞价、撮合或汇率清算（如广告竞价、外汇报价），"价格是规则命中结果"这一本质已不成立，应独立成交易撮合域。
