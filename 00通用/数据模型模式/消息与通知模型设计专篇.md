# 消息与通知模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦消息通知建模。短信、推送、邮件、站内信、微信模板消息——通知是所有系统的横切基础设施。它的建模难点：模板与发送分离、多渠道降级、频控合规、以及"到底发没发出去"的可追溯。

## 0. 一句话总纲

> 通知模型四分离：模板（说什么）、触达策略（走哪些渠道、什么顺序）、发送任务（一次具体的发送）、渠道回执（外部世界的回答）。合规（授权与频控）是发送前的强制闸门。

## 1. 四层结构

```text
模板（template）      —— 内容定义：变量占位、多语言版本、渠道适配版本（版本化）
触达策略（strategy）  —— 场景 → 渠道编排：先推送，10分钟未读补短信
发送任务（send_task） —— 一次对一个人的一次发送决定（幂等键）
渠道回执（receipt）   —— 渠道返回：提交成功/送达/失败/用户已读
```

```sql
CREATE TABLE msg_template (
  id           BIGINT PRIMARY KEY,
  template_key VARCHAR(64) NOT NULL,   -- 场景标识：order_paid
  channel      VARCHAR(16) NOT NULL,   -- SMS/PUSH/EMAIL/INBOX/WECHAT
  version      INT NOT NULL,
  content      TEXT NOT NULL,          -- 含变量占位 ${orderNo}
  status       VARCHAR(16) NOT NULL,   -- DRAFT/AUDITING/PUBLISHED/OFFLINE
  UNIQUE KEY uk_key_ch_ver (template_key, channel, version)
);

CREATE TABLE msg_send_task (
  id            BIGINT PRIMARY KEY,
  idempotent_key VARCHAR(64) NOT NULL, -- 业务事件+用户+场景，防重复发送
  template_key  VARCHAR(64) NOT NULL,
  template_version INT NOT NULL,       -- 固化发送时模板版本
  channel       VARCHAR(16) NOT NULL,
  receiver      VARCHAR(128) NOT NULL, -- 脱敏/加密存储
  render_params JSON,                  -- 渲染参数快照（能还原当时发了什么）
  status        VARCHAR(16) NOT NULL,  -- INIT/SENT提交渠道/DELIVERED/FAILED/SUPPRESSED被拦截
  suppress_reason VARCHAR(64),         -- 拦截原因：未授权/频控/黑名单/免打扰
  sent_at       DATETIME,
  UNIQUE KEY uk_idem (idempotent_key),
  KEY idx_status_time (status, sent_at)
);
```

要点：

- **发送任务固化模板版本 + 渲染参数**：客诉"你们短信说满 100 减 50"时能精确还原当时发的内容——通知也是对外承诺，快照纪律适用。
- 短信模板需运营商/平台审核，模板状态机含 AUDITING；渠道模板 ID（微信模板 ID、短信签名）是渠道侧资源，映射表管理。

## 2. 触达策略：编排与降级

- **场景驱动**：业务方只发"业务事件"（订单已发货 + 参数），订什么渠道、什么顺序由策略配置——业务代码里不出现"发短信"三个字。
- **渠道编排**：串行降级（推送未达/未读 N 分钟后补短信）、并行多发（验证码同时走两个短信通道取先达）、择一（站内信兜底）。编排执行记录落库，能回答"为什么他收到两条"。
- **重要性分级**：验证码/交易类（必达，多通道冗余）> 服务通知 > 营销（可丢弃）。级别决定重试策略和是否受免打扰约束——营销短信不该重试三次骚扰用户，验证码不该被频控挡掉。

## 3. 合规闸门：授权、频控、黑名单

发送前的强制检查链，任何一关不过则 SUPPRESSED（留任务记录，不是悄悄丢弃）：

- **营销授权**（会员篇 §6）：营销类消息校验用户授权记录（授权时间、渠道、范围），"凭什么给他发"永远可答。退订指令（回复 TD）实时落黑名单并停发。
- **频控**：用户 × 场景 × 周期的发送计数器（原子累加），营销类每日/每周上限；**免打扰时段**（22:00–8:00 不发营销）延迟到窗口后发送或丢弃。
- **黑名单/灰名单**：投诉用户、无效号码（连续送达失败自动拉灰，减少渠道成本与信誉损耗）。
- 验证码类**防轰炸**：同号码分钟级/小时级/天级三层限频 + 图形验证前置，这是安全问题不只是体验问题。

## 4. 渠道网关：与支付网关同构的谦卑

- 渠道配置（供应商 × 通道 × 签名 × 单价 × 限速）、多供应商路由与故障切换、渠道回执异步回调（送达回执可能延迟数小时甚至没有）——结构与支付渠道管理完全同构，纪律照搬：请求幂等、状态只进不退、回执驱动状态、供应商月度**账单对账**（发送量 × 单价 vs 供应商账单）。
- 发送量大（营销全量推送千万级）：任务表按时间分区、批量生成走异步任务（见批处理篇），推送渠道用批量接口，DB 里记批次 + 抽样明细而非亿级逐条（分级留痕：交易类逐条、营销类批次 + 统计 + 失败明细）。

## 5. 站内信：自有渠道的特殊性

- 站内信不依赖外部渠道，但有**收件箱容量问题**：全员公告不给每人插一行（亿级写放大），用"公告表 + 用户已读/删除状态表"（读时合并），私信才逐条落收件箱。
- 未读数是**派生计数器**（缓存 + 定期校准），不实时 COUNT。

## 6. 通知评审清单

```text
1. 模板/策略/任务/回执是否四层分离？业务代码是否只发事件不选渠道？
2. 发送任务是否幂等？是否固化模板版本和渲染参数？
3. 渠道编排和降级是否配置化并留执行记录？
4. 消息是否分级？营销与交易类的重试/频控/免打扰策略是否不同？
5. 营销授权是否校验留痕？退订是否实时生效？
6. 频控/黑名单/防轰炸是否为强制闸门？拦截是否留 SUPPRESSED 记录？
7. 渠道是否多供应商路由+回执驱动+月度对账？
8. 大批量发送是否批次化留痕？站内公告是否避免写放大？
```

## 7. 总结

通知建模五条纪律：

1. **事件进、策略出**：业务发事件，渠道编排是配置。
2. **发送即快照**：模板版本、参数、内容当时是什么，永远可还原。
3. **合规是闸门**：授权、频控、免打扰前置强制，拦截也留痕。
4. **渠道要谦卑**：幂等、回执驱动、对账，与支付网关同一套纪律。
5. **分级留痕**：交易类逐条可查，营销类批次可算，成本与追溯平衡。

> 每一条发出去的消息都是企业对用户说的一句话。模型要保证这句话说得有依据（授权）、有分寸（频控）、有记录（快照）。
