# 采购与供应链模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦采购与供应链建模。采购是"花钱的正门"：从供应商准入到付款结算，每一步都同时是业务流程和资金控制点。供应链模型的特点是**单据链条极长**（请购→采购→收货→质检→入库→对账→发票→付款），任何一环断链，钱和货就对不上。

## 0. 一句话总纲

> 采购模型的核心：三单匹配（订单、收货、发票）是资金安全的闸门，单据链条逐级引用、逐级留量，任何付款都能沿链条追到最初的需求。

## 1. 供应商：主数据 + 生命周期 + 资质

```sql
CREATE TABLE srm_supplier (
  id            BIGINT PRIMARY KEY,
  supplier_code VARCHAR(32) NOT NULL,
  name          VARCHAR(128) NOT NULL,
  status        VARCHAR(16) NOT NULL,  -- POTENTIAL潜在/AUDITING准入中/QUALIFIED合格/
                                       -- FROZEN冻结/BLACKLIST黑名单/EXITED退出
  category      VARCHAR(32),           -- 供应商分类（生产商/贸易商/服务商）
  settle_cycle  VARCHAR(16) NOT NULL,  -- 结算周期：月结30/60、货到付款、预付
  UNIQUE KEY uk_code (supplier_code)
);
```

- **准入是流程不是插入**：潜在 → 资质审核（营业执照、行业许可、验厂报告）→ 合格，每步留审批记录。资质文件带**有效期**，到期预警、过期自动冻结新单（存量单正常执行）。
- 供应商是主数据（总览篇主数据纪律）：唯一编码、防重（统一社会信用代码唯一键）、变更审批。银行账户变更是**高危操作**（付款诈骗最常见入口），必须双人复核 + 留痕 + 变更后首笔付款加验。
- **绩效档案**：交期达成率、质检合格率、价格竞争力按周期计算并留快照，降级/淘汰有数据依据。
- 一个供应商多个**供货目录条目**（供应商 × SKU：供货价、最小起订量、交期、含税率），价格带生效区间（双时态），采购单引用"当时价"。

## 2. 单据链条：逐级引用、逐级留量

```text
请购单（需求）→ 采购单（合同）→ 发货/到货通知 → 收货单 → 质检单 → 入库单
                                                                    ↓
                          付款单 ← 对账单/应付账款 ← 发票 ←—— 三单匹配
```

核心设计纪律：

- **行级引用 + 数量留量**：每张下游单据行引用上游单据行，上游行维护"已执行量"（请购行的已转采购量、采购行的已收货量、收货行的已开票量）。转换时原子校验 `已执行量 + 本次量 <= 上游量`——与订单防超退、库存防超卖同构，这是整条链不出"超收超付"的机制保障。
- **一对多与多对一并存**：一张请购拆多张采购单（分供应商），多张请购合并一张采购单（集中采购）——因此关联在**行级**建模（上游行 ↔ 下游行的关联表），单据头只是行的容器。
- 每张单据独立状态机 + 审批流（审批见工作流纪律），**改量一律新版本或补充单据**，不直接 UPDATE 已审批单据的数量。

```sql
CREATE TABLE pur_order_line (
  id            BIGINT PRIMARY KEY,
  po_id         BIGINT NOT NULL,
  sku_id        BIGINT NOT NULL,
  req_line_id   BIGINT,                  -- 来源请购行
  qty           INT NOT NULL,            -- 采购量
  received_qty  INT NOT NULL DEFAULT 0,  -- 已收货量（防超收）
  invoiced_qty  INT NOT NULL DEFAULT 0,  -- 已开票量（防超票）
  unit_price    DECIMAL(18,4) NOT NULL,  -- 未税单价（固化当时供货价）
  tax_rate      DECIMAL(6,4) NOT NULL,
  KEY idx_po (po_id)
);
```

## 3. 收货与质检：货权转移的证据链

- **收货 ≠ 入库**：收货单记"供应商送到了多少"（含拒收），质检后合格部分入可售/合格库存，不合格走退供或让步接收（降价接收要留审批）——对应库存篇的待检 → 可售状态移动。
- 收货单是**应付的事实基础**：对账以"实际合格入库量 × 采购价"为准，不以供应商发货量为准。
- 退供单是逆向一等公民（总览篇 §14）：引用原收货/入库行，库存出、应付冲减，链条对称。

## 4. 三单匹配：资金安全的闸门

**Three-Way Match** 是采购付款的世界级标准控制：

```text
采购单（订了什么价、多少量）
收货单（实际合格收了多少）        三者一致（容差内）才允许付款
发票（供应商要多少钱）
```

- 匹配在**行级**做：发票行 ↔ 收货行 ↔ 采购行，数量、单价、税率逐项比对，差异超容差（如 ±1% 或 ±100 元）自动挂起走人工审批。
- 匹配结果落**匹配记录表**（谁和谁匹配、差异额、处理方式），审计的核心抽查对象。
- 变体：两单匹配（服务类无收货，采购单 + 发票）、四单匹配（加质检单）。按品类配置匹配策略，不硬编码。

## 5. 应付与结算：账务纪律的采购侧落地

- **应付账款按账务篇建模**：合格入库即挂应付（借库存/贷应付），发票到达核销暂估、差异调整走凭证，付款冲应付——三步都是凭证驱动，不是改余额。
- **对账单**：按结算周期（月结30）汇总供应商的应付明细，双方确认后冻结为结算依据（快照特例规则）——出具后的对账单不改，有误走调整单。
- 预付款是独立科目（预付户），到货后转应付冲抵；押金/质保金（如货款的 5% 留 1 年）带释放条件和到期日，到期释放走付款流程。
- 付款单支持**多发票合并付款**与**部分付款**，付款行关联发票行，任何一笔付款可追溯到发票 → 收货 → 采购 → 请购。

## 6. 寻源与价格：把"选谁买"变成可审计的决策

- **询比价/招标**：询价单（一次询价多家报价）→ 报价记录（供应商 × SKU × 价格 × 有效期，只增不改）→ 定标记录（选了谁、为什么、审批人）。"为什么选贵的那家"必须有当时的记录可查。
- **价格协议/框架合同**：协议价带生效区间和累计量阶梯（买越多越便宜），采购单价格引擎按"下单时点 + 累计量"取价并固化。
- 寻源决策与采购执行分离（同库存篇寻源纪律）：采购单只接受"供应商和价格已确定"的输入。

## 7. 供应链计划的模型接口

计划算法（MRP/补货）本身是应用逻辑，但模型要提供其输入输出的落点：

- **输入**：可用库存 + 在途（库存篇）、未清采购单（订了没到的量）、销售预测/安全库存参数（库存篇 §11.4）。
- **输出**：**补货建议单**（建议不是命令：SKU、建议量、建议供应商、依据快照），人工确认后转请购/采购——算法建议与人工决策之间留一道单据，算法错了能看到"当时它为什么这么建议"。
- 交期承诺：采购行带承诺到货日，实际到货回填，交期偏差进供应商绩效。

## 8. 采购评审清单

```text
1. 供应商准入/冻结/黑名单是否状态机化？资质是否带效期并自动拦截？
2. 银行账户变更是否双人复核留痕？
3. 供货目录价格是否带生效区间？采购单是否固化当时价？
4. 单据链是否行级引用？上游行是否留量防超转/超收/超票？
5. 收货与入库是否分离？质检不合格是否走退供/让步单据？
6. 三单匹配是否行级、带容差、差异挂起？匹配记录是否可审计？
7. 应付是否凭证驱动（暂估/核销/调整）？对账单出具后是否冻结？
8. 预付、押金、质保金是否独立科目、带释放条件？
9. 付款能否沿链条追溯到请购？
10. 询比价和定标是否留记录？补货建议与人工决策之间是否有单据？
```

## 9. 总结

采购与供应链建模五条纪律：

1. **供应商即主数据**：准入有流程、资质有效期、改账户是高危操作。
2. **链条行级留量**：逐级引用 + 已执行量校验，超收超付在机制上不可能。
3. **三单匹配守门**：数量价格逐行比对，差异挂起，容差可配。
4. **应付凭证驱动**：暂估、核销、调整、付款全走凭证，对账单出具即冻结。
5. **决策留痕**：选谁、什么价、为什么，当时的依据永远查得到。

> 采购是企业最大的支出通道。模型的每一道校验，都是在替公司把住"钱花出去之前的最后一道门"。
