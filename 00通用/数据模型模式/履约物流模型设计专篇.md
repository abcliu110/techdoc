# 履约物流模型设计专篇

> 本篇聚焦履约、发货、配送、收货、签收和物流轨迹。履约模型连接订单、库存、仓库、物流商、售后和用户体验。

## 0. 一句话总纲

> 履约模型的核心，是把“订单承诺”转化为“可追踪的实物交付过程”。

## 1. 订单和履约要分离

订单表达交易承诺，履约表达交付过程。一个订单可以：

- 多仓发货。
- 多包裹发货。
- 部分发货。
- 拆单发货。
- 自提。
- 门店配送。
- 第三方物流。

因此履约应独立建模，不要只在订单表放一个物流单号。

## 2. 履约单、包裹和行

标准结构：

```text
履约单：一次履约计划，关联订单
履约行：关联订单行和数量
包裹：实际发出的包裹
物流单：承运商和运单号
轨迹：物流节点事件
```

**为什么需要行级履约**：部分发货、部分签收、部分退货都发生在行级数量上。

## 3. 履约状态机

典型状态：

```text
待分配 -> 待拣货 -> 待复核 -> 待打包 -> 已发货 -> 运输中 -> 已签收 -> 已完成
```

异常状态：

- 缺货。
- 拣货失败。
- 地址异常。
- 物流拒收。
- 配送失败。
- 丢件。
- 退回。

异常不要塞进订单主状态，应由履约异常模型承载。

## 4. 寻源和分仓

寻源决定从哪里发货：

- 就近仓。
- 成本最低。
- 库存充足。
- 指定门店。
- 拆单最少。
- 时效最优。

寻源结果应记录：

- 分配仓库。
- 分配原因。
- 分配时间。
- 是否人工改派。
- 失败原因。

## 5. 物流轨迹和事件

物流轨迹来自外部系统，不可完全控制。应作为事件记录：

- 已揽收。
- 运输中。
- 到达站点。
- 派送中。
- 已签收。
- 拒收。
- 异常。

轨迹事件必须幂等，按承运商、运单号、轨迹时间和状态去重。

## 6. 评审清单

```text
1. 订单和履约是否分离？
2. 是否支持一单多仓、多包裹、部分发货？
3. 履约是否行级关联订单行和数量？
4. 寻源结果是否可追溯？
5. 物流轨迹是否事件化、幂等化？
6. 缺货、拒收、丢件、地址异常是否有独立异常模型？
7. 履约状态是否反向推动订单展示状态？
```

## 7. 总结

履约物流建模五条纪律：

1. **交易和交付分离**：订单不是物流单。
2. **行级履约**：部分发货和退货必须有数量颗粒。
3. **寻源留痕**：为什么从这个仓发要能解释。
4. **轨迹事件化**：外部物流状态只可信事件，不可信一次查询。
5. **异常独立**：履约失败是业务对象，不是备注字段。
