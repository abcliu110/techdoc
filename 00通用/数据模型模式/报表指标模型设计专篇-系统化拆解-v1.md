# 报表指标模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

报表指标模型的本质，是业务事实到管理决策的语义转换系统。它把交易、库存、会员、支付等事实转换成指标、维度、趋势和钻取路径。

核心抽象：

```text
业务事实 -> 事实表 -> 维度 -> 指标口径 -> 汇总结果 -> 钻取明细
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| metric_atomic 原子指标 | 指标编码(唯一), 业务含义, 来源事实表, 度量字段, 单位, 聚合类型(可加/半可加/不可加), 负责人 | 最细粒度的唯一定义；GMV 只能由它派生不能自立 |
| metric_derived 派生规则 | 派生指标编码(唯一), 原子指标编码, 时间窗, 过滤条件(是否扣退款/含税/含取消), 维度组合 | 派生不改口径；口径差异靠命名显式化（支付 GMV/下单 GMV） |
| metric_caliber_version 口径版本 | 指标编码, version, 计算公式, 时间口径, 更新频率, 生效日期, 变更说明, 账期属性 | 指标变更留痕；关账后该期数值锁定 |
| metric_dimension 维度定义 | 维度编码, 来源维度表, 粒度, 是否拉链 | "按什么看"的稳定来源，防组织调整污染历史 |
| fact_trade 交易事实表(例) | 订单行业务键, biz_date, 成交/支付/退款金额, 各维度键 | 分析从稳定事实聚合，不临时拼业务表 |
| metric_result 汇总结果 | 派生指标编码, 维度组合值, biz_date, 指标值, calc_version, 计算时间 | 回溯型指标重算生成新版本不覆盖 |
| metric_lineage 血缘与钻取路径 | 派生指标编码, 层级(总额→门店→商品→明细), 下钻目标(事实表/单据) | 每个汇总都能回到业务事实，争议可解释 |

### 2.2 实体关系

```text
metric_atomic (1) ──< metric_derived (N)        一个原子指标多条派生规则
metric_atomic/derived (1) ──< metric_caliber_version (N)  口径按版本演进
metric_dimension (N) >── metric_derived (N)     派生规则声明可用维度组合
fact_trade (1) ──< metric_result (N)            结果从事实表按口径聚合
metric_derived (1) ──< metric_result (N)        一条规则产出多期多维结果
metric_derived (1) ──< metric_lineage (N)       钻取路径逐级指向明细
外部引用：来源事实表 → 数仓 DWD/账务事实表；维度表 → 数仓维度层（拉链）；
         钻取明细 → 各业务域原始单据；负责人 → 权限域用户
         （引用不落外键，应用层校验；报表不直查业务库）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：原子/派生指标编码全局唯一，同名不多口径；
             (指标编码, version) 口径版本唯一；
             (派生指标, 维度组合, biz_date, calc_version) 结果唯一
版本固化：口径变更=新 version+生效日期，不改历史定义；
         财务指标关账后冻结，调整走下期调整指标；
         回溯型指标（风控）重算写新 calc_version，报表默认取最新版并展示口径说明
聚合类型约束：不可加指标（UV/比率）禁止跨周期 sum，必须回明细重算；
            比率只存分子分母，展示层再除
跨事实约束：分子分母来自不同事实表时，维度必须对齐到共同粒度才允许相除
钻取约束：每个汇总结果必有 lineage 路径可达原始单据
分层落点：实时/日结/财务/审计报表分链路，实时结果标注"以日结为准"
分片键：biz_date（结果表分区主键）
```

## 3. 核心张力

最大张力是：

```text
指标易用性 vs 口径准确性
```

管理者希望看到简单数字，但每个数字背后都有时间口径、过滤条件、退款处理、税额处理、组织归属和数据延迟。

## 4. 不变量

- 指标必须有公式。
- 维度必须有来源。
- 汇总必须可钻取到原始事实。
- 指标口径必须版本化。
- 实时、日结、财务、审计报表要分层。
- 同名指标不能多口径并存而不说明。

## 5. 为什么这样设计

事实表和维度表分离，是为了让报表从稳定事实中聚合，而不是每次临时跨业务表拼 SQL。指标口径治理，是为了避免“销售额”在不同页面有不同结果。

## 6. 失败模式

- 报表直接查业务库，拖慢交易系统。
- 指标没有口径，争议无法解决。
- 汇总数字不能钻取明细。
- 组织调整污染历史报表。
- 实时报表和财务报表混用。

## 7. 适用边界

适用：

- 经营报表、财务报表、库存报表、会员分析、BI、数据看板。

简化：

- 少量实时查询可直接查业务库，但核心指标仍应定义口径。

## 8. 同一模型的多业务形态建模

指标横跨交易、流量、财务、供应链、风控多个业务面，但支撑它们的骨架只有一套：**原子指标 + 派生规则 + 口径字典，业务面差异下沉到事实来源和口径参数**。

### 8.1 统一骨架（所有业务面不变的部分）

```text
业务事实 → 原子指标（唯一定义：来源事实表 + 度量 + 单位）→ 派生规则（时间窗/过滤/维度组合）→ 口径字典条目（版本化）→ 汇总结果 → 钻取明细
不变量全集：指标有公式、维度有来源、汇总可钻取、口径版本化、同名不多口径
```

骨架决不为某个业务面特化——任何指标的"数字怎么算出来的、能不能钻到明细"都用同一套字典机制回答。

### 8.2 各业务面差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 交易指标（GMV/客单价） | 退款口径、下单口径 vs 支付口径争议多 | 原子指标拆到最细（下单金额/支付金额/退款金额各自独立），"GMV"只能是派生指标，派生规则显式声明是否扣退款（标准形态，骨架原型） |
| 流量指标（UV/转化率） | 去重计算、无法简单累加 | 原子指标标注聚合类型=不可加（去重计数）；派生规则限定去重键和时间窗，跨周期汇总禁止 sum，必须回明细重算 |
| 财务指标（收入/毛利） | 权责发生制、关账后冻结 | 事实来源指向账务事实表而非交易表；口径条目带账期属性，关账后该期数值锁定，调整走下期调整指标 |
| 供应链指标（周转/满足率） | 分子分母来自不同事实表 | 派生规则支持跨事实组合（库存快照事实 + 出库事务事实），维度必须对齐到共同粒度（仓库+SKU+日）才允许相除 |
| 风控指标（拦截率/误杀率） | 依赖事后标注、数值会回溯变化 | 口径条目标注"回溯型指标" + 标注滞后期；结果表带 calc_version，重算生成新版本不覆盖，报表默认取最新版并展示口径说明 |

### 8.3 建模决策规则

- **能用派生规则参数解决的，不新增原子指标**：GMV 含不含退款是两条派生规则，不是两个原子指标，更不是两张表。
- **原子指标必须标注聚合类型**（可加/半可加/不可加），派生层据此决定能否跨维度直接汇总——这是流量指标不出错的机制保证。
- **口径差异用命名显式化，不允许同名共存**："支付 GMV"和"下单 GMV"是两个字典条目，任何页面引用必须落到条目 ID。
- **拆独立域的信号**：当指标需要独立的计算引擎和实时决策（如风控规则实时打分），那是规则引擎域的运行时输出，指标域只沉淀其事后统计口径。
