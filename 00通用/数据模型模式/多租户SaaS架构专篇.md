# 多租户 SaaS 架构专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦多租户建模。SaaS 的本质是"一套系统服务 N 个企业"：租户之间要像不存在彼此一样隔离，又要共享同一套代码、同一次升级。多租户设计错误的代价是最高级别的——数据串租户是 SaaS 公司的生死事故。

## 0. 一句话总纲

> 多租户模型的全部目标：任何一行数据都明确属于一个租户，任何一次查询都不可能越过租户边界，任何一个租户都能独立配置、独立计量、独立迁移。

## 1. 隔离模式：三种方案与选型

| 模式 | 方案 | 隔离强度 | 成本 | 适用 |
|---|---|---|---|---|
| 独立数据库 | 每租户一个库（甚至一个实例） | 最强（物理隔离） | 最高（运维 N 个库） | 大客户/金融/合规要求高 |
| 独立 Schema | 同库不同 Schema | 中（逻辑隔离） | 中 | 中等规模、租户数百级 |
| 共享表 + tenant_id | 所有租户共享表，行级区分 | 最弱（靠纪律） | 最低 | 长尾小租户、租户数万级 |

选型要点：

- **绝大多数 SaaS 是混合模式**：长尾小租户共享表，付费大客户独立库（"专属版"卖点），因此模型设计第一天就要支持**同一套代码跑在两种模式上**——所有 SQL 都带 tenant_id，即使跑在独立库里（独立库里 tenant_id 是冗余，但保证了租户可以在两种模式间迁移）。
- 隔离模式是**租户级属性**（记录在租户元数据里），路由层按租户查路由表决定连哪个库。
- 不要按"现在只有 10 个客户"选独立数据库——租户数破千后，Schema 变更要在 1000 个库上执行，备份恢复监控全部乘以 1000。反过来共享表起步、大客户按需拆出，是被验证最多的路径。

## 2. tenant_id：一列定生死

### 2.1 铁律

- **每张业务表第一列业务字段就是 tenant_id**，NOT NULL，无例外（全局元数据表除外，见 §4）。
- **所有唯一键、所有索引以 tenant_id 打头**：`UNIQUE KEY uk_code (tenant_id, order_no)`——订单号只需租户内唯一，索引前缀也让查询天然限定在租户内。
- **所有查询强制带 tenant_id**，靠机制不靠自觉：
  - ORM 层全局过滤器（如 Hibernate Filter / MyBatis 拦截器）自动追加 `tenant_id = ?`；
  - 数据库层行级安全（PostgreSQL RLS：`CREATE POLICY tenant_isolation ON orders USING (tenant_id = current_setting('app.tenant_id')::bigint)`）作为第二道防线；
  - SQL 审计巡检：定期扫描慢日志/全量日志中不含 tenant_id 条件的语句，视为 P0 缺陷。

```sql
CREATE TABLE ord_order (
  id          BIGINT PRIMARY KEY,
  tenant_id   BIGINT NOT NULL,
  order_no    VARCHAR(32) NOT NULL,
  buyer_id    BIGINT NOT NULL,
  -- ... 业务字段
  UNIQUE KEY uk_order_no (tenant_id, order_no),
  KEY idx_buyer (tenant_id, buyer_id)
);
```

### 2.2 tenant_id 的来源：上下文注入而非参数传递

- 请求入口（登录态/域名/API Key）解析出租户，写入**请求上下文**，数据访问层从上下文取——tenant_id 不作为业务方法参数层层传递（传参迟早有人传错，把 A 租户的 id 传进 B 租户的查询）。
- 后台任务、消息消费、定时任务同样要**先建立租户上下文再干活**；跨租户批处理（平台运营视角）走专门的"平台通道"，显式声明、单独审计。
- 越权测试进 CI：自动化用例用租户 A 的身份访问租户 B 的资源 ID，必须全部 404/403——**串租户测试与单元测试同等地位**。

## 3. 租户元数据：租户本身也是一个域

```sql
CREATE TABLE sys_tenant (
  id            BIGINT PRIMARY KEY,
  tenant_code   VARCHAR(32) NOT NULL,     -- 租户编码（域名前缀等）
  name          VARCHAR(128) NOT NULL,
  status        VARCHAR(16) NOT NULL,     -- TRIAL试用/ACTIVE/SUSPENDED欠费停用/TERMINATED
  plan_id       BIGINT NOT NULL,          -- 订阅套餐
  isolation_mode VARCHAR(16) NOT NULL,    -- SHARED共享表/SCHEMA/DEDICATED独立库
  db_route_key  VARCHAR(64),              -- 路由标识（独立库连接串的键）
  created_at    DATETIME NOT NULL,
  expire_at     DATETIME,                 -- 订阅到期
  UNIQUE KEY uk_code (tenant_code)
);
```

要点：

- **租户生命周期是状态机**：试用 → 付费激活 → 欠费暂停（只读或封禁，数据保留）→ 终止（保留期后按协议删除/导出）。每一步转换记录流水，"欠费停用"绝不是删数据。
- **暂停 ≠ 删除**：SUSPENDED 租户拦在入口层（登录/API 网关），数据原封不动——欠费客户续费后 5 分钟内要能完整恢复。
- **终止与数据退出**：合同通常约定数据保留期（如 90 天）和导出义务，模型上要支持按租户全量导出（这也是 tenant_id 全表贯穿的另一个理由——没有它连"这个租户有哪些数据"都答不出来）。

## 4. 配置分层：平台默认 → 套餐 → 租户 → 用户

SaaS 的配置不是一张表，是一个**逐层覆盖**的体系：

```text
平台默认值（代码/全局配置表）
  ↓ 覆盖
套餐配置（plan：功能开关、配额上限）
  ↓ 覆盖
租户配置（tenant_config：租户管理员的选择）
  ↓ 覆盖
用户偏好（user_preference：个人级）
```

```sql
CREATE TABLE sys_tenant_config (
  id          BIGINT PRIMARY KEY,
  tenant_id   BIGINT NOT NULL,
  config_key  VARCHAR(64) NOT NULL,
  config_value TEXT NOT NULL,
  updated_by  BIGINT NOT NULL,
  updated_at  DATETIME NOT NULL,
  UNIQUE KEY uk_tenant_key (tenant_id, config_key)
);
```

纪律：

- **读取时逐层回退**（租户没配则取套餐值，套餐没配取平台默认），求值逻辑收敛在一个配置服务里，禁止业务代码各自拼装。
- **功能开关（feature flag）与配额（quota）分开**：开关是布尔（这个租户能不能用高级报表），配额是数值（最多 50 个用户、每月 10 万条短信）——配额要配套**用量计量**（见 §6）。
- 配置变更留审计流水（谁改的、改前改后值）——B 端客户的"昨天还能用今天不能用"客诉，九成是配置变更，流水是唯一答案。
- 套餐升降级 = 换 plan_id + 生效时间，降级时**超出新配额的存量数据只读不删**（用户数超了就不让加新用户，不是删旧用户）。

## 5. 定制化：元数据扩展而不是改表

大客户总会要"加个字段"。三级方案按侵入性递增：

| 方案 | 做法 | 适用 |
|---|---|---|
| 预留扩展字段 | ext_json JSON 列（带 JSON Schema 校验） | 少量展示性字段 |
| 自定义字段元数据 | 字段定义表（租户/对象/字段名/类型/校验）+ 值表或 JSON 列，受控 EAV（总览篇 §34） | 平台级能力："自定义字段"作为产品功能 |
| 独立定制 | 大客户独立库上做 Schema 级定制 | 战略客户，单独分支要有版本纪律 |

铁律：**核心交易表结构对所有租户一致**。定制只发生在扩展层，一旦为某租户改了共享表结构，升级发布就永远背上这个包袱。自定义字段要进搜索/报表的，同步进搜索引擎和数仓的通用扩展列，不为单租户改 ETL。

## 6. 计量与计费：SaaS 的收入线

订阅 + 用量是 SaaS 主流计费模型，模型上三件套：

```text
订阅（subscription）：租户 × 套餐 × 周期（生效/到期/续费/升降级），本质是"订单 + 生效区间"
用量流水（usage_record）：每次可计费行为一条（API 调用、短信、存储日快照），只增不改，带幂等键
账单（bill）：周期结算任务从用量流水聚合生成，出具后冻结（快照特例规则，总览篇 §13）
```

纪律：

- **用量流水是计费的事实源**：账单可从流水重算，客户质疑"为什么收我这么多"时逐条可对——计量不准的 SaaS 会同时失去收入和信任。
- 配额检查用**预聚合计数器**（租户 × 计量项 × 周期的当前用量，原子累加），流水异步对账校准计数器——检查走计数器（快），结算走流水（准）。
- 升降级的**按比例折算**（proration）规则显式建模：升级立即生效补差价、降级下周期生效，折算明细进账单行。
- 试用期、赠送额度、代金券按营销资产纪律处理（营销篇），不直接改账单金额。

## 7. 升级与租户版本：一套代码的代价

- **Schema 变更全租户同步**是共享表模式的默认约束，因此 Expand-Contract（总览篇 §39）在 SaaS 是必选项而非最佳实践——不可能停机让一万个租户一起等。
- 独立库模式下要维护**租户 Schema 版本表**（每个租户库当前版本），升级任务分批滚动执行、失败可重试可暂停，允许短期内不同租户跑在相邻版本上（代码必须兼容 N 和 N-1 版本 Schema）。
- **灰度按租户**：新功能先放开给内部租户 → 友好客户 → 全量，功能开关（§4）就是灰度机制本身。

## 8. 跨租户场景：平台视角的正门

有三类合法的跨租户需求，都要走"正门"：

- **平台运营后台**：客服/运营查看租户数据，走独立的平台端应用 + 显式租户切换 + 全程审计（查了谁的什么数据），且遵守最小授权——不是给运营一个"不带 tenant_id 的查询口"。
- **平台级报表**：所有租户汇总的经营分析走数仓（报表篇），ODS 抽取时保留 tenant_id 作为一等维度，分析侧天然跨租户，在线侧永远不跨。
- **租户间协作**（供应商与采购商都是租户且互相下单）：这是业务关系不是隔离豁免——用显式的**关系/授权表**（谁授权谁看什么），双方数据各自留在各自租户内，交换走单据。

## 9. 多租户评审清单

```text
1. 每张业务表是否有 NOT NULL 的 tenant_id？唯一键和索引是否以 tenant_id 打头？
2. 查询是否由 ORM 过滤器/RLS 强制注入租户条件？是否有 SQL 巡检抓漏网语句？
3. tenant_id 是否来自请求上下文而非方法传参？后台任务是否先建立租户上下文？
4. 串租户越权测试是否进 CI？
5. 租户生命周期（试用/激活/暂停/终止）是否状态机化？欠费是否只封不删？
6. 配置是否分层（平台/套餐/租户/用户）且求值收敛在一处？变更是否留流水？
7. 功能开关与配额是否分离？配额是否有用量计量支撑？
8. 定制化是否走扩展层（ext_json/自定义字段元数据）？核心表结构是否全租户一致？
9. 用量流水是否只增不改、可重算账单？账单出具后是否冻结？
10. Schema 升级是否 Expand-Contract？独立库租户是否有版本表和滚动升级机制？
11. 平台侧跨租户访问是否走独立通道并全程审计？
12. 租户数据能否按合同全量导出和按期删除？
```

## 10. 总结

多租户建模五条纪律：

1. **tenant_id 贯穿一切**：每表必有、索引打头、机制强制、巡检兜底。
2. **隔离模式可迁移**：共享起步、大客户拆出，同一套代码跑两种模式。
3. **配置逐层覆盖**：平台→套餐→租户→用户，开关与配额分离，变更留痕。
4. **定制不碰核心表**：扩展层吸收个性化，共享表结构全租户一致。
5. **计量即收入**：用量流水只增不改，账单可重算，出具即冻结。

> SaaS 卖的不是软件，是"你的数据在我这里比在你自己机房更安全"的信任。租户隔离的每一道防线，都是这句承诺的抵押物。
