# 发票与税务模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

发票与税务模型的本质，是交易在税法世界中的法律凭证和税务镜像。它把订单、退款、商品税收分类、抬头资质、税控开票、红冲和申报口径连接起来。

核心抽象：

```text
开票依据 -> 开票申请 -> 发票 -> 发票明细 -> 红冲 -> 税控回执 -> 税务申报
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| inv_title 开票抬头 | owner_id, title_type(个人/企业), title_name, tax_no(格式校验), bank_info(专票必填), addr_phone, is_default | 用户的开票资质档案，活档案；票面以快照为准 |
| inv_basis 开票依据 | order_line_id, 实付金额, 已退款金额, invoiced_amount 已开票金额 | 可开金额 = 实付 - 已退 - 已开，逐行留量防超开 |
| inv_apply 开票申请 | 申请号(唯一, 对税控幂等键), title_id, invoice_type, 申请金额, status, 关联单据集 | 开票意图：可失败重试、拆票、合并开票 |
| inv_invoice 发票 | apply_id, invoice_type, invoice_no(唯一), amount 不含税, tax_amount, total_amount, status(ISSUING/ISSUED/RED_FLUSHED), title_snapshot 抬头快照, red_ref_id, 已红冲金额, file_id | 税控确认的法律凭证，开具后不可变 |
| inv_invoice_line 发票明细 | invoice_id, order_line_id, 商品名称(符合税收分类), 税收分类编码, 数量, 单价, 税率, 金额, 税额 | 对齐订单行与税法口径，尾差落明细行 |
| inv_tax_receipt 税控回执 | apply_id, 通道类型(数电/税控盘), 请求报文, 回执状态, 发票号码/日期, 票面文件引用 | 外部交互原样落库，出具即固化 |
| inv_quota 开票额度 | 纳税主体, 期间(月度), 授信额度, 已用额度 | 数电票额度是资源计数，余量监控预警 |
| inv_input_invoice 进项发票 | 发票号码, 供应商 ID, 金额/税额, 查验记录, 认证状态(未认证/已认证/已抵扣), 关联采购单行 | 销项的镜像方向，三单匹配与抵扣跟踪 |

### 2.2 实体关系

```text
inv_title (1) ──< inv_apply (N)          申请选抬头，票面靠快照
订单行 (1) ── inv_basis (1)              开票依据逐行维护留量
inv_apply (1) ──< inv_invoice (N)        申请-发票 1:N（拆票/重试）
inv_apply (N) >── 订单/订单行 (N)         合并开票多单一申请
inv_invoice (1) ──< inv_invoice_line (N) 票面明细
inv_invoice (1) ──< inv_invoice 红字票 (N) red_ref_id 红蓝对链，双票永久保留
inv_apply (1) ──< inv_tax_receipt (N)    每次税控交互一条回执
inv_quota (1) ──< inv_invoice (N)        数电票消耗期间额度
采购单行 (1) ──< inv_input_invoice (N)   进项票对齐采购行
外部引用：order_line_id → 订单域；税收分类编码 → 商品域类目税务属性；
         退款单 → 订单/支付域（触发红冲）；供应商 ID → 采购域；
         申报口径 → 税务申报域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：invoice_no 唯一；申请号唯一且作为税控幂等键
             （超时只能查询确认，禁止盲发重试）
留量校验：申请原子 UPDATE inv_basis.invoiced_amount
         WHERE 已开 + 本次 ≤ 实付 - 已退（防超开）；
         红冲原子校验 已红冲 + 本次 ≤ 票面金额（防超冲）；
         数电额度原子累加 WHERE 已用 + 本次 ≤ 授信额度
恒等式：Σ发票明细行 = 票面价税合计；不含税额 + 税额 = 价税合计，
       尾差按税法舍入规则落明细行
状态机约束：ISSUING 是合法中间态；ISSUED 后票面不可变，
           纠错一律红冲（红字票为独立负数发票）；
           换开 = 红冲 + 新申请两步留痕；红冲自动回补 inv_basis 留量
分片键：owner_id（用户查票主路径）；额度按纳税主体+期间
```

## 3. 核心张力

最大张力是：

```text
业务交易灵活性 vs 税务凭证不可变
```

订单可以退款、改价、拆分、合并；发票一旦开具就是法律凭证，不能随意修改，纠错必须通过红冲。

## 4. 不变量

- 抬头、申请、发票必须分离。
- 发票必须固化抬头快照。
- 可开金额必须逐行留量防超开。
- 发票明细要对齐订单行和税收分类。
- 发票开具后不可变，纠错走红冲。
- 红字票和蓝字票都必须永久保留。
- 税控交互必须幂等，开具中状态合法。
- 发票、账务、申报三方数字必须能核对。

## 5. 为什么这样设计

申请和发票分离，是因为一次申请可能失败、重试、拆票或合票；申请只是意图，发票才是税控系统确认的法律结果。

可开金额留量，是为了防止同一订单或订单行被重复开票。红冲机制是因为税务世界不允许直接修改已开具发票。

## 6. 失败模式

- 发票直接挂订单，不支持合并开票和拆票。
- 抬头只引用当前档案，历史票面被抬头修改污染。
- 已开票金额不留量，出现超开。
- 退款后未红冲，税务和业务金额不一致。
- 税控请求超时后重复开票，造成税务事故。

## 7. 适用边界

适用：

- 电商、ERP、SaaS、B2B、支付结算、采购进项票、销项票。

可简化：

- 无开票义务的纯内部系统可不建完整发票模型，但只要涉及税务凭证，就必须遵守不可变和红冲纪律。

## 8. 设计取舍

优先保证：

- 税务合规。
- 开票防超。
- 历史票面可追溯。
- 红冲链路完整。

牺牲：

- 流程简单性。
- 退款链路复杂度。
- 税控对接复杂度。

## 9. 同一模型的多业务形态建模

发票与税务要同时支撑多种业务形态，核心方法是：**统一三层骨架 + 票种类型标识 + 税控通道适配层吸收差异**。

### 9.1 统一骨架（所有业态不变的部分）

```text
抬头 → 开票依据（逐行留量）→ 开票申请 → 发票（invoice_type 标识票种）→ 明细 → 红冲 → 税控回执
不变量全集：三层分离、抬头快照固化、可开金额留量防超开、发票不可变、红冲纠错、税控幂等
```

骨架决不为某个票种特化——任何票种的"凭什么开、开了多少、还能开多少、错了怎么纠"都用同一套留量和红冲机制回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| B2C 电子普票 | 抬头简单（个人可只填姓名）、量大自动开 | 标准形态（骨架原型）：申请由用户触发或支付后自动生成，抬头校验规则按票种配置放宽 |
| B2B 增值税专票 | 抬头六要素强校验、可抵扣、纸票有票号库存 | 抬头表加资质字段（税号/开户行/账号），专票申请强制校验+人工审核节点；纸票加票号段领用表管理票量 |
| 数电票（全电发票） | 无票号段、无税控盘、额度制、乐企/RPA 通道 | 税控通道适配层新增数电通道实现；票号段管理裁剪，替换为开票额度表（月度授信额度留量，机制同可开金额留量） |
| 红冲换开 | 抬头错了要先红冲再重开 | 红字票挂原蓝票 ID 形成对链；换开是"红冲 + 新申请"两个动作的编排，可开金额随红冲自动回补留量 |
| 进项收票 | 方向反转：收供应商票、查验、认证抵扣 | 独立进项发票表（不复用销项表，方向不同但结构镜像）；增加查验记录+认证状态字段；与采购单行对齐做三单匹配 |
| 跨境税务 | 零税率/免税、形式发票、无国内税控 | 发票明细税率支持零税率和免税标识（税收分类编码驱动）；形式发票 invoice_type=proforma，跳过税控通道但保留留量和不可变纪律 |

### 9.3 建模决策规则

- **新票种 = invoice_type + 票种校验规则配置 + 税控通道适配器**，不动三层结构：留量、红冲、幂等机制对所有票种一视同仁。
- **票种差异字段进票种扩展表**（invoice_special_ext / invoice_paper_ext），主表保持票面公共要素。
- **允许按票种裁剪税控交互但不允许裁剪留量和不可变**：形式发票没有税控回执，但超开和篡改同样不被允许。
- **拆独立域的信号**：进项与销项方向相反、事实源不同（对方开的票），应镜像建表而非塞进销项模型；申报计算本身是税务申报域，发票模型只负责提供可核对的数字。
