# 权限模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

权限模型的本质，是组织责任边界在系统中的表达。它决定谁能看、谁能做、能操作哪些数据、在什么状态下能操作，以及操作后谁负责。

核心抽象：

```text
用户 -> 组织/岗位 -> 角色 -> 功能权限 -> 数据权限 -> 状态动作权限 -> 审计
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| auth_permission 权限点 | perm_code(唯一, domain:resource:action 语义编码), perm_type(MENU/BUTTON/API), parent_id, status | 随代码发布的枚举；perm_code 是代码引用的稳定标识 |
| auth_role 角色 | role_code, role_name, tenant_id, role_source(平台模板/租户自定义), status | 权限点的集合；租户自定义角色只能在平台授予的权限点子集内选 |
| auth_role_permission 角色-权限 | role_id, perm_id | 功能授权的唯一常规通道，禁止直接给用户挂权限点 |
| auth_user_role 主体-角色 | subject_id, subject_type(用户/岗位/应用), role_id, effective_from, effective_to, grant_operator | 授权带有效期到期自动收回；只增不改，支撑"当时有没有权"回放 |
| auth_data_scope 数据范围 | role_id, resource, scope_type(ALL/ORG_TREE/ORG/SELF/CUSTOM), custom_expr | 角色×资源→行过滤规则；多角色范围取并集 |
| auth_column_scope 列权限 | role_id, resource, field_name, visibility(可见/脱敏/隐藏) | 字段级可见性；查看明文是独立权限点且逐次审计 |
| org_assignment 任职 | user_id, position_id, org_id, assign_from, assign_to | 人挂任职、岗挂角色；带任职期间只增不改，调动不重配权限 |
| auth_delegation 委托代理 | delegator_id, delegatee_id, scope(委托范围), valid_from, valid_to, status | 代理审批独立记录；操作留痕记"实际操作人+以谁的身份" |
| auth_role_exclusion 互斥角色组 | group_code, role_id, exclusion_reason | 职责分离(SoD)约束：出纳与审批人不能同人 |

### 2.2 实体关系

```text
auth_role (1) ──< auth_role_permission (N) >── (1) auth_permission  角色聚合权限点
auth_role (1) ──< auth_user_role (N)       主体(用户/岗位/应用)挂角色，带有效期
auth_role (1) ──< auth_data_scope (N)      角色×资源→行过滤范围
auth_role (1) ──< auth_column_scope (N)    角色×资源×字段→列可见性
auth_permission (1) ──< auth_permission (N)  parent_id 菜单树自引用
org_position (1) ──< org_assignment (N)    人挂任职，岗位再挂角色
auth_role (1) ──< auth_role_exclusion (N)  互斥组授予时校验
外部引用：subject_id → 用户/组织域；org_id/org_path → 组织域；
         业务表 org_id/owner_id/tenant_id 归属字段 → 各业务域；
         授权变更与敏感查看事件 → 审计域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：perm_code 全局唯一；(role_id, resource) data_scope 唯一；
       (subject_id, role_id) 有效期内授权不重叠
双时态：auth_user_role / org_assignment 带时间区间、只增不改，
       审计可回放"操作发生时他有没有权限"
tenant_id 纪律：角色/授权/数据范围全部带 tenant_id；
       租户隔离在会话层强制注入，位于数据权限之下
注入纪律：scope_type 过滤在服务端查询构造层统一注入，不靠手写 where
互斥约束：授予角色时校验互斥组，同一主体不得同时持有互斥角色
```

## 3. 核心张力

最大张力是：

```text
使用便利性 vs 安全隔离和责任追踪
```

权限越简单，越容易越权；权限越精细，配置和排查越复杂。

## 4. 不变量

- 功能权限、操作权限、数据权限要分离。
- 数据权限必须落到业务数据边界。
- 有按钮权限不代表当前状态可操作。
- 多租户隔离必须在权限之下。
- 权限变更和敏感操作必须审计。

## 5. 为什么这样设计

RBAC 解决“能不能进入功能”，但不能解决“能看哪些数据”。企业系统真正复杂的是数据范围和状态动作结合，例如能看本门店订单，但不能反结账已完成订单。

## 6. 失败模式

- 只有菜单权限，没有数据权限。
- 前端隐藏按钮当权限控制。
- 超级管理员逻辑到处散落。
- 组织调整后历史数据归属变化。
- 审批权限和业务操作权限混用。

## 7. 适用边界

适用：

- SaaS、ERP、CRM、POS、后台管理、审批系统。

简化：

- 单人单租户工具可简化，但只要多角色、多组织，就应引入数据权限。

## 8. 同一模型的多业务形态建模

权限模型要同时支撑多种业务形态，核心方法是：**统一骨架 + 授权主体扩展 + 数据范围维度映射吸收差异**。

### 8.1 统一骨架（所有业态不变的部分）

```text
主体（用户/岗位/应用）→ 角色 → 权限点（perm_code）→ 数据范围（scope_type × 维度字段）→ 审计
不变量全集：功能与数据权限分离、租户隔离在权限之下、状态参与授权、授权变更可审计
```

骨架决不为某个业态特化——任何业态的"能做什么动作、能触达哪些行"都用同一套 RBAC + 数据范围回答。

### 8.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 内部员工后台 | 组织树范围、人员调动频繁 | scope_type=ORG_TREE + org_path 前缀过滤；人挂任职、岗挂角色（标准形态，骨架原型） |
| 商家多店铺权限 | 一个商家管多店、店员只见本店 | 数据范围维度换成 shop_id：data_scope 的 resource × 范围指向店铺维度；店员角色为商家在租户内自定义角色 |
| SaaS 租户管理员 | 租户自建角色、不得越过平台 | 平台角色模板 + 租户自定义角色两层；租户可分配权限点 = 平台按套餐授予该租户的权限点子集；tenant_id 硬隔离在权限之下 |
| ISV 开放权限 | 授权对象是应用不是人、接口粒度 | 主体扩展为 app_id；权限点复用 perm_type=API；OAuth scope 映射到 perm_code，授权记录带有效期和数据范围 |
| 门店 POS 收银权限 | 离线可用、高危操作二次授权 | 权限清单随登录下发本地缓存（带版本号，联网刷新）；反结账/整单折扣是独立权限点 + 主管授权记录（实际操作人 + 授权人双字段） |
| 供应商/合作方门户 | 只见与己相关单据 | scope_type=SELF 的维度变体：owner 换成 supplier_id，业务表带 supplier_id 归属字段做行过滤 |

### 8.3 建模决策规则

- **能用 scope_type + 维度字段解决的，不建第二套权限体系**：差异只在"过滤哪个字段"时，换 resource × 维度映射，RBAC 主链复用。
- **新业态先检查业务表有无归属字段**（org/shop/supplier/tenant）：数据权限的上限由字段决定，字段缺失先补业务模型，不在权限层硬造。
- **授权主体可扩展（用户/岗位/应用），权限点和数据范围机制不变**：不为某类主体发明专属鉴权语义。
- **拆独立域的信号**：当授权语义超出"角色 × 数据范围"（如 C 端资源所有权、文件级 ACL 共享授权），才是独立授权域，不是 RBAC 的一种配置。
