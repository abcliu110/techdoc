# 发票与税务模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦发票建模。发票是"交易在税法世界里的镜像"：业务上一笔订单，税务上是一张有法律效力的凭证。它的建模难点：开票金额与订单金额的对齐、红冲而非作废的纠错纪律、抬头资质的管理、以及与税控系统这个"强势外部方"的交互。

## 0. 一句话总纲

> 发票模型的核心：开票依据（可开金额）逐行留量防超开，发票一经开具即不可变（纠错走红冲），业务金额与开票金额的差异永远可逐项解释。

## 1. 三层结构：抬头、申请、发票

```text
开票抬头（invoice_title）—— 用户/企业的开票资质档案（可复用、要校验）
开票申请（invoice_apply）—— 一次开票请求：开哪些单、什么抬头、什么类型
发票（invoice）          —— 税控系统返回的法律凭证：发票号码、代码、开票日期
```

**为什么申请与发票分离**：申请可能失败重试、可能拆分成多张票（超限额拆票）、可能合并多单开一张（合并开票）——申请是意图，发票是结果，1:N 关系。与支付单/渠道交易的分层完全同构。

```sql
CREATE TABLE inv_title (
  id           BIGINT PRIMARY KEY,
  owner_id     BIGINT NOT NULL,        -- 归属用户/企业
  title_type   VARCHAR(16) NOT NULL,   -- PERSONAL个人/COMPANY企业
  title_name   VARCHAR(128) NOT NULL,  -- 抬头名称
  tax_no       VARCHAR(32),            -- 纳税人识别号（企业必填，格式校验）
  bank_info    VARCHAR(255),           -- 开户行及账号（专票必填）
  addr_phone   VARCHAR(255),           -- 地址电话（专票必填）
  is_default   TINYINT NOT NULL DEFAULT 0,
  KEY idx_owner (owner_id)
);

CREATE TABLE inv_invoice (
  id            BIGINT PRIMARY KEY,
  apply_id      BIGINT NOT NULL,
  invoice_type  VARCHAR(16) NOT NULL,  -- SPECIAL专票/GENERAL普票/E_SPECIAL数电专/E_GENERAL数电普
  invoice_no    VARCHAR(32),           -- 发票号码（税控返回）
  amount        DECIMAL(18,2) NOT NULL, -- 不含税金额
  tax_amount    DECIMAL(18,2) NOT NULL, -- 税额
  total_amount  DECIMAL(18,2) NOT NULL, -- 价税合计
  status        VARCHAR(16) NOT NULL,  -- ISSUING开具中/ISSUED/RED_FLUSHED已红冲/FAILED
  title_snapshot JSON NOT NULL,        -- 抬头快照（开票时固化，抬头档案后改不影响）
  issued_at     DATETIME,
  file_id       BIGINT,                -- 票面文件（文件篇：出具即固化登记）
  red_ref_id    BIGINT,                -- 红字票指向原蓝字票
  UNIQUE KEY uk_no (invoice_no),
  KEY idx_apply (apply_id)
);
```

- **抬头快照**：发票固化开票时的抬头内容，抬头档案是活的、票面是死的（快照特例规则）。
- 企业抬头的税号做格式与工商校验，抬头是用户资产（可维护多个、设默认），但**每次开票以申请时选择+快照为准**。

## 2. 开票依据：可开金额逐行留量

发票的资金安全等价于"防超开"，机制与防超退/防超收完全同构：

```text
可开票金额 = 实付金额 - 已退款金额 - 已开票金额（行级或单级维护）
申请开票：原子校验并累加 invoiced_amount，Σ开票 ≤ 可开依据
退款发生在开票后：触发红冲或差额红字（见 §3）
```

- **开票粒度对齐订单行**：发票明细行对应订单行（商品名称、税收分类编码、数量、单价、税率），税收分类编码挂在商品/类目上维护（商品篇类目模板的税务属性）——票面"商品名称"必须符合税收分类，不能随便写。
- **金额恒等**：Σ发票明细行 = 票面价税合计；不含税额 + 税额 = 价税合计，尾差按税法舍入规则处理并落在明细行上（分摊尾差纪律）。
- 优惠与发票：折扣是**票面折扣行**（负数行）还是直接按成交价开，按税务口径配置——报表篇"标价/成交价/结算价三个数"在这里加上第四个"开票价"，四者差异要可解释。

## 3. 红冲：发票世界的冲正

发票开错不能改、跨期不能作废，只能**红冲**（开负数红字票对冲原蓝字票）——这是总览篇"冲正而非 UPDATE"思想在税法上的强制版：

- 红字票是独立发票记录（负数金额，red_ref_id 指向原票），原蓝字票状态置 RED_FLUSHED，**两张票都永久保留**。
- **部分红冲**（退了一部分货）：红字票金额 = 退款对应金额，原票剩余部分继续有效；原票维护"已红冲金额"防超冲。
- 触发链条：退款单 → 判断已开票 → 生成红冲申请 →（专票需先取得红字确认单）→ 红字票——退款流程要感知开票状态，**先票后退的场景红冲是退款的前置或并行步骤**，流程编排显式建模。
- 换开（抬头开错）：红冲原票 + 新开正确票，两步都留痕，不存在"改一下抬头"。

## 4. 税控对接：又一个强势外部方

税控系统（数电票平台/税控服务商）具备外部渠道全部特征，支付网关篇纪律照搬：

- **请求幂等**：申请号作为对外幂等键；开具中（ISSUING）是合法状态，超时不明结果只能**查询确认**，不能重复盲发（重复开票是税务事故）。
- 税控返回的发票号码、开票日期、票面文件（OFD/PDF/XML）原样落库，文件按文件篇"出具即固化"登记。
- **额度管理**：数电票有授信开票额度（月度总额），额度使用是资源计数（原子累加 + 余量监控告警），临近耗尽要预警而不是开票失败才发现。
- 交付：票面文件发送用户（邮箱/短信链接，走通知篇），交付记录留痕——"没收到发票"是高频客诉。

## 5. 进项与税务合规侧

销项（开给客户）之外，企业级还有进项侧：

- **进项发票**（供应商开来的票）：采购篇三单匹配中的"发票"即此——收票登记、查验真伪（税局接口查验记录留痕）、认证抵扣状态跟踪（未认证/已认证/已抵扣）、与应付付款联动。
- **纳税申报口径**：销项税额、进项税额按期汇总，报表口径与账务凭证（账务篇应交税费科目）可核对——发票系统的数字最终要与申报表、总账三方一致。
- 发票文件与数据的**法定保留期**（10 年级别）按审计篇/文件篇的保留策略执行。

## 6. 发票评审清单

```text
1. 抬头/申请/发票是否三层分离？申请与发票是否 1:N（拆票合票）？
2. 发票是否固化抬头快照？票面文件是否按"出具即固化"登记？
3. 可开金额是否逐行留量、原子校验防超开？
4. 发票明细是否对齐订单行、挂税收分类编码？价税合计恒等式与尾差是否处理？
5. 纠错是否一律红冲（含部分红冲防超冲）？换开是否两步留痕？
6. 退款流程是否感知开票状态并编排红冲？
7. 税控对接是否幂等 + ISSUING 中间态 + 查询确认？额度是否监控？
8. 进项票是否登记/查验/认证跟踪并与三单匹配、应付联动？
9. 销项进项能否与账务科目、申报口径三方核对？
10. 发票数据与文件是否按法定年限保留？
```

## 7. 总结

发票建模五条纪律：

1. **三层分离**：抬头是档案，申请是意图，发票是法律结果。
2. **留量防超开**：可开金额行级维护，与防超退超收同一机制。
3. **纠错必红冲**：票面不可变，红蓝对冲永久留双票。
4. **税控要谦卑**：幂等、中间态、查询确认、额度监控。
5. **三方可核对**：发票、账务、申报的数字互相解释得通。

> 发票是整个系统里唯一"每一行都写在税法里"的数据。业务可以灵活，票面必须刻板——刻板到十年后税务稽查翻出任何一张票，链条都完整如初。
