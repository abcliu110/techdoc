# 采购供应商模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

采购供应商模型的本质，是企业补充供给能力的协同系统。它回答向谁买、买什么、什么价、何时到、是否合格、是否入库、是否应付、是否付款。

核心抽象：

```text
供应商 -> 采购申请 -> 采购订单 -> 收货 -> 质检 -> 入库 -> 应付 -> 付款
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| pur_supplier 供应商主数据 | supplier_code(唯一), category(生产商/贸易商/服务商), subject_type(企业/个人/境外), 名称, 账期, 结算方式, 税率, 开票信息, status(潜在/准入中/合格/冻结/黑名单) | 向谁买的权威源；准入是流程不是插入，黑名单跨形态生效 |
| pur_supplier_cert 资质档案 | supplier_id, 资质类型(营业执照/授权书/许可), 证件号, 有效期起止, 审核状态 | 资质按分类配置；到期自动拦截新单 |
| pur_supplier_account 银行账户 | supplier_id, 账户类型(对公/个人), 账号, 变更审批单号, status | 钱付给谁；账户变更双人复核留痕 |
| pur_price_list 供货价格 | supplier_id, sku_id, 单价, currency, 税率, 起订量, effective_start/end, version | 供货目录与价格带生效区间，议价即新版本 |
| pur_request 采购申请 | 申请单号, 申请组织, sku_id, 申请数量, 需求日期, 审批状态 | 需求的起点；申请和订单分离，支持合并下单 |
| pur_order 采购订单 | po_no(唯一), supplier_id, 采购组织, 收货仓库, currency, 税率快照, status | 采购承诺的根；币种税率下单即快照 |
| pur_order_line 采购行 | po_id, sku_id, qty, 单价快照, 税额, 预计到货时间, received_qty 已收, returned_qty 已退, invoiced_qty 已开票 | 核心颗粒；留量字段支撑部分收货/退供/开票/付款 |
| pur_receive 收货单 | 收货单号, po_id, po_line_id, 实收数量, 收货时间, status | 实物到达的事实；收货不等于入库 |
| pur_qc 质检单 | receive_id, 合格数量, 不合格数量, 处理方式(退供/报损/让步接收), 质检人 | 质量确认；不合格走退供或报损路径 |
| pur_inbound 入库单 | qc_id/receive_id, 仓库, sku_id, 入库数量, 库存事务ID, status | 库存可用的证据；入库必须驱动库存域事务 |
| pur_payable 应付单 | supplier_id, 来源类型(订单/入库/发票), 来源单号, 应付金额, 已付金额, currency, 账期到期日, status | 三单匹配后生成；付款冲减，应付必有来源 |
| pur_return 退供单 | 原 po_line_id, 退供数量, 退供金额, 原因(质检不合格/滞销), status | 逆向独立单据，行级引用原采购行 |

### 2.2 实体关系

```text
pur_supplier (1) ──< pur_supplier_cert (N)     资质多项、各带有效期
pur_supplier (1) ──< pur_supplier_account (N)  账户变更留历史
pur_supplier (1) ──< pur_price_list (N)        供货目录按 SKU×时段
pur_request (1) ──< pur_order_line (N)         申请可合并/拆分下单
pur_order (1) ──< pur_order_line (N)           主单-行一对多
pur_order_line (1) ──< pur_receive (N)         分批收货
pur_receive (1) ──< pur_qc (N)                 收货后质检
pur_qc (1) ──< pur_inbound (N)                 合格部分入库
pur_order_line (1) ──< pur_return (N)          退供行级引用原行
pur_supplier (1) ──< pur_payable (N)           应付按供应商归集
外部引用：sku_id → 商品域（价格/税率靠快照固化）；入库/退供 → 库存域事务；
         付款单 → 资金账务域；采购组织/仓库 → 组织域
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：supplier_code、po_no 全局唯一（个体供应商防重键切换为证件号）；
       (supplier_id, sku_id, effective_start) 同 SKU 同时段价格唯一；
       应付单幂等键 (来源类型, 来源单号) 防重复生成
留量校验（原子 UPDATE 条件）：
       Σ收货 received_qty ≤ 行 qty；Σ退供 ≤ received_qty - 已退；
       Σ开票 invoiced_qty ≤ received_qty；已付金额 ≤ 应付金额
恒等式（三单匹配）：应付金额 = 匹配后的 min(订单价×入库量, 发票金额)，
       差异必须落差异单处理，不得静默抹平；
       质检：合格数量 + 不合格数量 = 实收数量
状态机约束：供应商 潜在→准入中→合格→冻结/黑名单/退出，
           冻结/黑名单拦截新采购单但不影响存量应付；
           资质到期自动冻结对应品类供货
版本化/快照规则：价格税率币种在下单时快照进采购行，
       价格表修改只发新生效区间；供应商关键信息变更留审核历史
```

## 3. 核心张力

最大张力是：

```text
供应链灵活性 vs 财务和库存准确性
```

采购希望快速补货和灵活议价；库存和财务要求收货、质检、入库、发票、应付、付款可核对。

## 4. 不变量

- 供应商是主数据。
- 采购行是核心颗粒。
- 收货不等于入库。
- 入库必须影响库存。
- 应付必须有来源。
- 采购价格、税率、币种必须快照。

## 5. 为什么这样设计

采购行级设计，是为了支持部分收货、部分退供、部分开票、部分付款。收货、质检、入库分离，是为了表达实物到达、质量确认、库存可用不是同一件事。

## 6. 失败模式

- 一张采购单直接入库，无法处理质检。
- 没有采购行累计字段，部分收货和退供混乱。
- 供应商资料无资质和账期，后续应付无法治理。
- 发票金额、入库金额、订单金额无法匹配。

## 7. 适用边界

适用：

- ERP、供应链、零售采购、B2B 采购、仓储补货。

简化：

- 小型系统可合并收货和入库，但必须明确是否需要质检和应付。

## 8. 同一模型的多业务形态建模

供应商模型要同时支撑多种供应商形态，核心方法是：**统一主数据骨架 + category 分类标识 + 资质项按分类配置吸收差异**。

### 8.1 统一骨架（所有业态不变的部分）

```text
供应商主数据（supplier_code + category）→ 准入生命周期（潜在→准入中→合格→冻结/黑名单/退出）→ 资质档案（带有效期）→ 银行账户 → 供货目录/价格 → 绩效档案
不变量全集：唯一编码防重、准入是流程不是插入、资质到期自动拦截新单、账户变更双人复核、价格带生效区间
```

骨架决不为某个形态特化——任何形态的"这家能不能合作、资质是否有效、钱付给谁、价格怎么定"都用同一套主数据 + 生命周期回答。

### 8.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 生产商 | 验厂、产能与质量体系认证 | category=MANUFACTURER，资质项配置含验厂报告/ISO 证书（资质项按分类配置，同一张资质表）；绩效加质检合格率权重 |
| 贸易商 | 无产能，看授权与货源 | 资质项配置为品牌授权书（带授权期，到期冻结对应品类供货目录）；供货目录按品类挂授权范围校验 |
| 服务商 | 无实物、按验收结算 | 资质项为行业许可/人员资格；供货目录条目是服务项（无起订量，价格按人天/次）；结算周期挂验收单而非入库 |
| 个体供应商 | 无对公账户、代扣个税 | 主体类型 subject_type=INDIVIDUAL，证照为身份证而非营业执照（防重键切换为证件号）；结算走个人账户 + 代扣税扩展表，单笔限额校验 |
| 跨境供应商 | 外币、多税制、清关资质 | 主数据挂币种与税制扩展表；资质项含出口资质/原产地证明；供货目录价格带币种，付款走跨境通道并留购付汇记录 |
| 集团内关联供应商 | 内部关联交易、定价受控 | 标记 related_party=TRUE 并关联内部法人组织；价格来源锁定为内部协议价（不走询比价）；对账进入关联交易报表口径 |

### 8.3 建模决策规则

- **能用 category + 资质项配置解决的，不建第二套供应商表**：差异在"要什么证、怎么定价、怎么结算"时，资质项/价格/结算配置吸收，主数据和生命周期一套。
- **主体差异（企业/个人/境外）用 subject_type + 防重键切换，不复制表**：统一编码空间保证黑名单跨形态生效。
- **生命周期状态机对所有分类同一套，允许裁剪环节不允许新增终态**：个体供应商可免验厂，但"合格/冻结/黑名单"语义不变。
- **拆独立域的信号**：当对方同时是客户与供应商且核心是双向往来对账（平台二清、渠道返利），应上升为统一交易伙伴主数据，供应商只是其一种角色。
