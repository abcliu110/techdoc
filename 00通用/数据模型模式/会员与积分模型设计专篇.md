# 会员与积分模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦会员身份、等级、积分与储值建模。会员体系是"长期关系"的模型：交易是一瞬间，会员关系持续数年，模型必须扛住身份合并、等级规则变化、积分过期这些时间性难题。

## 0. 一句话总纲

> 会员模型的三个支柱：一个人只有一个身份（One ID）、等级由可重放的规则计算、积分储值按准资金记账。

## 1. 身份模型：One ID 是一切的地基

### 1.1 身份的分层

```text
自然人身份（customer）—— 唯一的"人"
  ├─ 登录凭证（credential）：手机号、微信 unionid、邮箱、Apple ID …（一对多）
  ├─ 渠道身份（channel_identity）：小程序 openid、App 设备、门店会员卡号（一对多）
  └─ 会员身份（membership）：某会员体系中的会员号、等级、权益（可多体系）
```

铁律：

- **凭证与身份分离**：手机号是凭证不是主键——手机号会换、会二次放号。customer_id 是不变的内部主键，所有业务表引用它。
- 每个凭证/渠道身份表带绑定时间、解绑时间（只增不改的绑定历史），回答"这个 openid 在 3 月时属于谁"。

### 1.2 账户合并：设计期就要面对

同一人先后用手机号和微信注册成两个 customer，迟早要合并：

- **合并单**：主身份、被合并身份、合并时间、操作依据，被合并身份标记 MERGED 并指向主身份（不物理删除）。
- 历史交易**不回写**（订单上还是旧 customer_id），查询层通过合并映射聚合——回写亿级历史表既不现实也破坏审计。
- 积分/储值余额合并走**转账凭证**（账务纪律，见 §3），不是 UPDATE 相加。
- 风控介入：合并是盗号和薅羊毛的高发操作，需要验证和留痕。

> 不设计合并的会员系统，等于假设用户永远只用一种方式注册。这个假设第一年就会破产。

## 2. 等级模型：状态是计算结果，不是手工字段

### 2.1 成长值驱动

```text
行为（消费/签到/评价）→ 成长值流水（+/-，只增不改）→ 累计成长值 → 等级
```

- 等级 = f(窗口期内成长值, 等级规则版本)，是**可重放的纯函数**。
- 等级变更记录表：旧等级、新等级、触发原因（升级/保级失败/人工调整）、依据的规则版本、生效时间。
- 人工调级走独立单据（原因、审批、有效期），不直接 UPDATE 等级字段。

### 2.2 时间性难题

- **窗口期**：滚动 12 个月成长值——用成长值流水按窗口聚合，或维护"窗口累计值 + 每日过期扣减"，量大时预计算。
- **保级与降级**：周期结束的批量评估任务，评估结果全部留痕；降级要提前通知（评估记录支撑"差多少保级"的展示）。
- **规则改版**：新规则版本 + 生效日，生效日前按旧规则解释历史等级，禁止用新规则重算历史（总览篇 §17）。
- 订单固化**下单时等级**（订单快照），等级价、等级折扣按当时等级结算。

## 3. 积分：准资金,完整账务纪律

积分套用账务篇模型（账户 + 凭证 + 流水），额外多两个难题：**过期**和**批次**。

### 3.1 积分批次与先进先出过期

积分和钱最大的不同是会过期，而且不同来源的积分过期时间不同，因此必须**按批次记账**：

```sql
CREATE TABLE mbr_point_batch (
  id           BIGINT PRIMARY KEY,
  customer_id  BIGINT NOT NULL,
  biz_type     VARCHAR(32) NOT NULL,    -- 消费赠送/活动奖励/签到/人工调整
  biz_order_no VARCHAR(64) NOT NULL,
  issue_idem_key VARCHAR(64) NOT NULL,
  issued       INT NOT NULL,            -- 本批发放量
  remaining    INT NOT NULL,            -- 本批剩余（消耗和过期从这里扣）
  expire_at    DATETIME NOT NULL,
  UNIQUE KEY uk_idem (issue_idem_key),
  KEY idx_cust_expire (customer_id, expire_at)
);
```

- **消耗按 FIFO 吃批次**（先过期的先用）：一次消耗可能跨多个批次，消耗流水记录"从哪个批次扣了多少"。
- **过期任务**按批次扫（expire_at 索引），过期也是流水（批次 remaining 清零 + 过期流水），用户可查"哪来的积分、什么时候过期了多少"。
- 总余额 = Σ批次 remaining，与账户表余额每日对账。
- **退款回补**：消费返的积分在退款时按比例扣回，扣回可能导致负余额（积分已用掉）——负余额策略要显式定义（允许挂负 / 从后续发放抵扣），不能没想过。

### 3.2 积分的财务属性

积分是平台负债（会计上要计提），积分发放/核销/过期对应账务分录（营销费用/负债冲减/过期利得），年底"积分负债余额"是审计科目——这就是积分必须走账务纪律的根本原因。

## 4. 储值：钱的纪律 + 营销的复杂度

储值余额直接用账务篇模型（用户储值户），叠加两个特殊点：

- **本金与赠送金分户**：充 100 送 20，本金户和赠金户分开记账。退储值只退本金余额，赠金不可提现（合规红线）；消费时扣减顺序（先赠金后本金 or 按比例）写进规则并在流水中体现。
- **可退金额计算**：退卡 = 本金余额 - 已享优惠的追回（按规则），每一步都要有凭证。
- 储值受预付卡监管（备付金、实名、限额），模型上要能按监管口径出报表：发卡量、余额、消费额、退卡额。

## 5. 权益：资格类资产

会籍权益（免邮、折扣、生日礼、优先购）与券的区别是"资格"而非"面额"：

```text
权益定义（等级/付费会籍挂什么权益）→ 权益实例（发给个人，带有效期）→ 权益使用流水（次数型要计数）
```

- 次数型权益（每月 4 张免邮券）：按周期发放实例，用一次记一次，周期末清零——发放、使用、清零全部流水化。
- 付费会员（年卡）：会籍是一张"订单 + 订阅"（生效期、续费、退款按剩余期折算），权益随会籍生效/失效联动，退款时已用权益要折价。

## 6. 隐私与生命周期

会员域是个人数据最密集的域，总览篇 §29 在此处全量适用：

- 手机号/证件加密存储 + 哈希指纹检索；展示默认脱敏。
- **注销**：身份进入注销状态，凭证解绑（手机号释放给二次注册），个人字段匿名化，交易骨架保留（统计与审计），积分储值余额按协议处理并留凭证。
- 营销触达授权（短信/推送同意书）单独记录，带授权时间和渠道——合规检查的第一个问题就是"你凭什么给他发短信"。

## 7. 会员评审清单

```text
1. 凭证与身份是否分离？customer_id 是否为不变内部主键？
2. 绑定/解绑是否留历史？账户合并是否有合并单、映射查询、余额转账凭证？
3. 等级是否由成长值流水 + 规则版本计算？人工调级是否走单据？
4. 等级评估/降级是否留评估记录？订单是否固化当时等级？
5. 积分是否按批次记账、FIFO 过期、过期留流水？
6. 退款扣回积分的负余额策略是否显式定义？
7. 积分负债能否按审计口径出报表？
8. 储值是否本金/赠金分户？退卡规则是否可算可证？
9. 次数型权益是否流水化？付费会籍退款是否按期折算？
10. 个人数据是否加密/脱敏/可注销匿名化？营销授权是否留痕？
```

## 8. 总结

会员建模五条纪律：

1. **One ID**：凭证会变，身份不变；合并是常态功能不是事故处理。
2. **等级可重放**：成长值流水 + 规则版本 = 等级，人工干预走单据。
3. **积分批次化**：发放成批、消耗吃批、过期扫批，负债可审计。
4. **储值双户**：本金赠金分离，退卡可算，监管口径可出。
5. **隐私内建**：加密、脱敏、注销匿名化是模型能力，不是运营承诺。

> 会员体系经营的是以年计的信任。模型里每一条流水，都是未来某次客诉、审计或监管检查时的答案。
