# 数据归档与冷热分离模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦数据生命周期的下半场。订单表五年积累几十亿行，流水表更甚——不做归档，索引膨胀、备份变慢、变更工具超时，在线库被历史数据活活拖死。归档的建模难点：迁走的数据还要能查（客诉翻三年前的订单）、还要合法（法定保留期）、还要可信（迁移不丢不重）。

## 0. 一句话总纲

> 归档的本质是给数据分温度、分居所：热数据伺候交易，温数据伺候查询，冷数据伺候审计与法律。搬家全程有账（迁移批次可对可回滚），查询按温度自动路由，销毁按法规执行并留痕。

## 1. 数据温度模型：先定义再搬家

| 温度 | 定义（按访问特征） | 居所 | 例子 |
|---|---|---|---|
| 热 | 高频读写、参与交易 | 在线主库 | 近 3~6 月订单、活跃流水 |
| 温 | 只读为主、低频查询 | 归档库（同构 DB / 廉价实例） | 6 月~2 年订单 |
| 冷 | 极少访问、合规保留 | 对象存储（列式/压缩格式） | 2 年以上流水、审计日志 |

纪律：

- **温度线按业务状态 + 时间双条件**：不是"超过 6 个月就搬"，而是"**终态**且超过 6 个月"——未完结的订单（履约中、退款中、诉讼中）无论多老都不归档。归档条件是每张表显式声明的谓词（状态白名单 + 时间线），登记在归档策略表里。
- **行级引用完整性优先**：订单归档要连带订单行、支付记录、优惠明细**按聚合整体搬**（一致性边界思想）——主单归档了明细还在线上，两边都成了孤儿。
- 温度线是**产品承诺的一部分**："App 里只能查近 2 年订单，更早的走客服/导出"——归档边界要与产品交互对齐，而不是用户翻页翻到一半突然没了。

## 2. 归档策略登记：每张大表一份档案

```sql
CREATE TABLE arch_policy (
  id            BIGINT PRIMARY KEY,
  table_name    VARCHAR(64) NOT NULL,
  archive_predicate VARCHAR(512) NOT NULL, -- 归档条件：终态+时间线
  aggregate_tables JSON,                   -- 连带归档的子表（聚合边界）
  hot_retention  VARCHAR(32) NOT NULL,     -- 在线保留：6个月
  warm_retention VARCHAR(32) NOT NULL,     -- 归档库保留：2年
  legal_retention VARCHAR(32) NOT NULL,    -- 法定保留：5年/7年/10年（审计篇）
  destroy_policy VARCHAR(32) NOT NULL,     -- 到期处理：物理删除/匿名化
  owner         VARCHAR(32) NOT NULL,
  status        VARCHAR(16) NOT NULL,
  UNIQUE KEY uk_table (table_name)
);
```

- **法定保留期优先于存储成本**（文件篇同款铁律）：财务凭证、发票、审计日志的法定年限压倒一切归档冲动。
- 无策略的大表是隐患：容量巡检（总览篇容量估算）发现的增长大表，必须在策略表里有归属——"这张表打算留多久"是建表评审就该回答的问题（总览篇 §31 的闭环）。

## 3. 迁移作业：搬家全程有账

归档迁移是典型批处理任务（批处理篇全套纪律适用），额外的关键点：

```text
迁移批次：批次号 | 表 | 条件范围 | 行数 | 校验和 | 状态（COPIED/VERIFIED/DELETED/ROLLBACK）
流程：抽取写入归档端 → 校验（行数 + 关键字段校验和）→ 通过后才删在线端 → 批次留档
```

- **先落地、再校验、后删除**——三步分离且批次状态机推进，任何一步失败可安全重来（幂等：归档端按主键幂等写入，删除按批次范围条件删）。
- **删除是低峰限速的小事务**（批处理篇纪律）：大 DELETE 锁表拖复制，按批次分小块删 + 间隔限速；分区表用 `DROP PARTITION` 代替 DELETE（这正是流水表按时间分区建表的最大红利——归档设计要前移到建表时的分区决策）。
- 迁移批次表永久保留："三年前那批订单搬去哪了、校验值多少"可查——搬家的账本本身按审计纪律管理。

## 4. 查询路由：归档对使用方透明或显式

三种由浅入深的方案：

| 方案 | 做法 | 适用 |
|---|---|---|
| 显式双入口 | 产品上分"近期订单/历史订单"，历史查归档库 | 多数场景，简单诚实 |
| 应用层路由 | 按查询时间范围自动选库，跨温度合并结果 | 中台统一查询服务 |
| 联邦查询 | 引擎级透明（分布式 SQL/外部表） | 分析场景，在线慎用 |

- **归档端的索引按查询场景瘦身**：只保留"单号精确查 + 用户+时间范围"两类，不复刻在线库全部索引——归档库的使命是"找得到"，不是"跑报表"（报表走数仓，报表篇）。
- 冷数据（对象存储上的 Parquet/ORC）用扫描引擎按需查，**查询延迟以分钟计要有产品预期管理**（文件篇解冻延迟同款思想）。
- **回热是正式流程**：诉讼/审计要求批量调取冷数据 → 回热任务加载到临时查询区 + 期限 + 用后清理留痕，不是手工 copy。

## 5. 销毁：生命周期的合法终点

- 过了法定保留期的数据按 destroy_policy 处理：**物理删除**或**匿名化保骨架**（会员篇注销同款：个人字段抹除、统计骨架保留）。
- **销毁留痕**：销毁批次（范围、行数、依据的策略版本、执行时间、审批人）永久记录——"证明数据已删"和"证明数据还在"同样是合规义务（GDPR 删除权 vs 财务保留义务的交汇点，按数据分类分别执行）。
- 备份里的数据同步考虑：销毁策略要覆盖备份副本的到期轮转，否则"删了但备份里还有"在严格法域不算删除。

## 6. 归档评审清单

```text
1. 每张大表是否有归档策略档案（条件/各级保留期/销毁方式/owner）？
2. 归档条件是否"终态+时间"双条件？聚合是否整体连带迁移？
3. 温度线是否与产品承诺对齐？
4. 迁移是否三步分离（落地/校验/删除）+ 批次状态机 + 幂等？
5. 删除是否小事务限速？流水大表是否建表时就按时间分区（DROP PARTITION 红利）？
6. 迁移批次账是否永久可查？
7. 查询路由方案是否明确？归档端索引是否按场景瘦身？
8. 冷数据回热是否正式流程带期限清理？
9. 法定保留是否压倒成本考虑？销毁是否留痕并覆盖备份副本？
```

## 7. 总结

归档建模五条纪律：

1. **温度先定义**：终态+时间双条件、聚合整体搬、与产品承诺对齐。
2. **策略即档案**：每张大表的保留期与销毁方式在建表评审时就有答案。
3. **搬家有账本**：落地、校验、删除三步分离，批次永久可查。
4. **查询分温度**：热库交易、温库查证、冷存审计，路由显式。
5. **销毁也合规**：到期依法删或匿名化，删除本身留下证据。

> 数据和人一样有生命周期：年轻时在交易一线，中年退居查询二线，老年进入合规档案，最后体面地注销。归档设计就是提前给每类数据写好这份"一生的安排"。
