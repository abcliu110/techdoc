# 客户会员模型设计专篇

> 本篇聚焦客户、会员、账户、等级、权益、标签和画像建模。客户会员模型连接订单、营销、支付、售后、报表和权限，是企业经营“人”的核心模型。

## 0. 一句话总纲

> 客户会员模型的核心，是区分“人是谁”“他属于谁”“他有什么权益”“他发生过什么行为”“系统如何服务他”。

## 1. 客户、会员、用户、联系人要分清

| 概念 | 含义 | 示例 |
|---|---|---|
| 用户 | 登录系统或小程序的账号主体 | 手机号、微信 openid、员工账号 |
| 会员 | 面向权益、等级、积分、储值的经营对象 | 门店会员、品牌会员 |
| 客户 | B2B 或 CRM 中的交易对象 | 公司客户、经销商、采购方 |
| 联系人 | 客户下的自然人联系人 | 采购经理、财务联系人 |

**为什么需要区分**：

- 一个用户可能绑定多个会员身份。
- 一个客户可能有多个联系人。
- 一个会员可能跨门店消费。
- B2C 和 B2B 的权益、账期、合同、发票逻辑不同。

**错误做法**：一张 `member` 表塞登录账号、客户资料、权益、联系人、开票信息。短期简单，长期无法支持多身份、多组织、多触点和 B2B 关系。

## 2. 身份与归属模型

会员不是孤立对象，必须有归属边界：

- 所属租户。
- 所属品牌。
- 所属门店。
- 注册渠道。
- 推荐人。
- 所属客户经理。
- 所属组织或区域。

交易发生时应固化会员当时等级、归属门店、渠道和客户经理，否则历史报表会被当前归属变更污染。

## 3. 账户与权益分离

会员权益至少分为：

- 积分账户。
- 储值账户。
- 成长值账户。
- 优惠券账户。
- 等级。
- 权益包。

标准模型：

```text
会员主档：身份、基础资料、归属关系
权益账户：当前积分、储值、成长值等当前值
权益流水：每次增加、减少、冻结、过期、冲正
权益规则：发放、消耗、升级、降级、过期规则
权益事件：订单支付、退款、核销、过期等事实
```

**为什么这样设计**：

当前权益值只能说明现在有多少，流水才能说明为什么变化。退款、反结账、券回收、积分过期都需要权益流水和冲正。

## 4. 会员等级模型

等级设计要回答：

- 等级依据是什么：消费金额、消费次数、成长值、人工指定。
- 等级周期是什么：永久、自然年、滚动 365 天。
- 升级是否实时。
- 降级是否允许。
- 保级条件是什么。
- 历史订单按当时等级还是当前等级展示。

标准做法：

```text
等级规则版本
会员当前等级
等级变更记录
等级计算快照
等级权益配置
```

**带来的复杂度**：

- 等级规则变更后历史计算不能被污染。
- 降级会影响权益、价格和营销触达。
- 人工调级需要审计。

## 5. 标签、画像和分群

标签分三类：

- 人工标签：运营或销售手工维护。
- 规则标签：根据消费、频次、金额、品类自动计算。
- 模型标签：由算法或数据分析生成。

分群是标签和规则的组合结果，应记录：

- 分群规则。
- 计算时间。
- 命中人数。
- 规则版本。
- 是否用于营销活动。

**为什么需要版本**：营销活动发券时，必须能解释“当时为什么这个会员被选中”。

## 6. 隐私与合规

客户会员模型包含敏感数据：

- 手机号。
- 身份证。
- 地址。
- 生日。
- 微信 openid/unionid。
- 消费记录。
- 储值余额。

标准做法：

- 敏感字段脱敏展示。
- 敏感字段加密或分级存储。
- 访问记录审计。
- 导出需要权限和水印。
- 删除、匿名化、注销要有流程。

## 7. 评审清单

```text
1. 用户、会员、客户、联系人是否分清？
2. 会员归属、渠道、门店、客户经理是否可追溯？
3. 积分、储值、成长值是否账户与流水分离？
4. 权益发放、消耗、退款、过期、冲正是否有事件和流水？
5. 等级规则是否版本化？等级变更是否有记录？
6. 标签和分群是否有来源、规则版本和计算时间？
7. 敏感数据是否脱敏、加密、审计？
8. 历史订单是否固化当时会员等级和权益规则？
```

## 8. 总结

客户会员建模五条纪律：

1. **身份分层**：用户、会员、客户、联系人不是一回事。
2. **权益留痕**：积分、储值、券、等级都要有流水和事件。
3. **规则版本化**：等级、发券、积分、分群都要能解释当时命中原因。
4. **归属可追溯**：门店、渠道、客户经理变化不能污染历史。
5. **隐私受控**：客户数据是资产，也是责任。
