# 配置、错误恢复与可观测性规范

> 上位治理：`../00-工程治理总纲.md`；SLO、错误预算、反馈延迟和控制动作共同构成运行闭环，不能只建设指标与告警。

> 适用范围：所有需要部署、运行和运维的软件服务。
> 来源：原 `24-配置与特性开关细则.md`、`25-错误处理与恢复细则.md`、`27-可观测性与运维细则.md` 的跨项目规则。
> 迁出边界：低代码元数据版本、动态查询、动态 Schema 同步机制、插件和专属容量阈值进入低代码平台规范。

## 1. 约束等级与适用规则

- `MUST`：在适用范围和条款前提命中时违反即阻断合并或发布。
- `SHOULD`：默认遵守；偏离时记录理由、风险和替代控制。
- `条件规则`：仅在对应能力存在时生效；未使用特性开关、消息、调度、分布式追踪或动态配置的项目不得为套规范引入这些设施。

项目可以采用等价实现。指标名、错误分类、可靠性等级和生命周期名称由项目统一定义，本文只约束必须表达和验证的语义。

## 2. 配置定义与覆盖

1. `MUST`：每个配置项声明 key、类型、是否必填、敏感等级和变更生效方式；默认值、单位、合法范围与允许覆盖层级在适用时声明，禁止用隐含默认值掩盖必填配置缺失。
2. `条件规则`：存在多层配置源时，覆盖优先级必须固定且有测试；禁止同一配置在不同模块使用不同覆盖顺序。
3. `条件规则`：既有系统核心业务配置的默认值保持旧行为；改变默认值按行为变更评审。
4. `MUST`：新增配置 key 优先于改变旧 key 语义；禁止静默改变单位或枚举含义。
5. `MUST`：敏感配置不得进入普通日志、普通前端响应、导出包或源码；存储和传输采用项目批准的秘密管理能力并支持轮换。确需展示时仅通过专用、最小授权、可审计入口脱敏展示，不要求每个项目自建密钥系统。
6. `条件规则`：动态配置需要传播到多实例时必须带版本并声明最大收敛时间；实例遇到未知或不兼容版本时不得静默继续依赖该配置写入。
7. `SHOULD`：配置 schema 由机器校验，并在启动或发布前验证缺失、类型和范围错误。
8. `条件规则`：动态配置更新失败时必须定义 fail-closed、回退到最后已知正确版本或拒绝相关操作中的一种策略；回退版本、原因和影响范围可审计。

## 3. 特性开关

1. `MUST`：开关登记 owner、创建原因、默认值、作用范围、验证信号、回滚条件、过期时间和清理任务；验证信号可以是指标、日志、业务校验或项目已有等价证据。登记位置为项目统一维护的特性开关清单（如配置中心目录、专用注册表或受版本控制的声明文件），确保可检索且与运行时一致。
2. `条件规则`：既有系统用开关引入新逻辑时默认保持旧行为；开关关闭或回滚后，旧版本必须能处理开关开启期间产生的数据，否则必须阻断关闭并走恢复流程。恢复流程失败时执行兜底方案：保留开关开启状态、冻结相关写入或降级为只读，并触发人工介入告警，不得在未确认数据一致性的情况下强制关闭开关。
3. `MUST`：灰度维度来自受控字段，禁止任意 SQL 或不可审计表达式。
4. `MUST`：开关不作为永久客户特判；到期时删除，或迁移为正式配置、策略或扩展契约。
5. `MUST`：紧急或越权能力默认关闭、最小授权、限时有效、全程审计，且不得关闭安全审计和脱敏。
6. `SHOULD`：开关使用实验、灰度、全量、移除中的生命周期，并定期扫描过期项。
7. `MUST`：同一业务操作中的开关判定必须一致；远程判定超时或配置缺失时使用已声明的安全默认值，不得在执行中途随机切换路径。

## 4. 错误契约

1. `MUST`：项目维护稳定错误分类；至少能区分调用方可修复、权限或隔离拒绝、冲突、外部依赖和系统缺陷，具体枚举按业务需要扩展。
2. `条件规则`：机器消费的服务边界对外错误提供稳定 code、安全 message 和关联标识；字段校验错误提供可定位字段信息。命令行或离线任务使用等价的退出码、错误记录和运行标识。
3. `MUST`：对外错误遵守 `04-安全与供应链规范.md` 的信息泄露底线；内部诊断通过关联标识获取，不在响应复制敏感详情。
4. `MUST`：为防止资源探测，在需要保密的查询场景中，"不存在"与"无权访问"必须使用统一的外部响应口径；内部仍保留准确原因。
5. `MUST`：对外稳定错误码只能新增或按版本废弃，不得复用旧 code 表达新语义；内部错误分类可以重构，但不得破坏已承诺的外部映射和历史诊断能力。
6. `MUST`：除明确声明为可丢弃且不承担业务义务的 best-effort 操作外，异常不得只写日志后吞掉；调用方必须收到明确失败，或任务持久化为可恢复状态。

## 5. 重试、补偿与异步恢复

1. `MUST`：自动重试只用于可安全重放的操作，并设置次数、退避、抖动、单次超时和总时限；有副作用的写入必须通过业务唯一键、幂等键、去重记录或状态条件更新防止重复副作用。
2. `MUST`：发生部分成功、未知提交状态或数据不一致时先对账，再决定重试、补偿或人工处理。
3. `条件规则`：存在异步任务时声明可靠性等级，例如与业务同事务、可靠异步或允许丢弃；不得默认所有任务都“尽力而为”。
4. `条件规则`：可靠异步消息记录事件标识、必要的业务隔离上下文、关联标识、重试次数和最后脱敏错误。
5. `条件规则`：消息系统存在死信或隔离队列时，失败项必须可查询、可审计、可重放或标记放弃；重放本身必须幂等并受权限控制。
6. `条件规则`：异步、消息和调度入口恢复执行所需的租户、用户或业务域隔离上下文；关键隔离上下文缺失时 fail-fast。
7. `条件规则`：定时任务可能并发触发或补跑时必须防重，并声明时区、补跑策略、执行标识和资源上限；有副作用的任务具备幂等语义。
8. `条件规则`：存在跨层自动重试时，调用链只允许在明确层级拥有重试责任和总预算，避免客户端、网关、服务和消息层叠加重试放大流量；尊重下游限流与 `Retry-After` 等恢复信号。

## 6. 日志与追踪

1. `条件规则`：跨进程或异步链路使用 traceId、correlationId 或等价关联标识贯穿入口、下游调用、消息和任务；跨信任边界时校验或重新生成，不盲信外部标识。
2. `MUST`：结构化日志至少包含时间、级别、模块、事件和结果；存在可追踪操作时包含关联标识，存在业务隔离时包含必要且经脱敏的隔离标识。
3. `MUST`：日志分类定义字段、级别、脱敏、访问权限和保留期；审计日志与诊断日志不能相互替代。
4. `MUST`：日志不得输出完整敏感对象、密码、token（明确涵盖会话 token、API key、刷新 token 等）、密钥、身份证件、支付凭证或 SQL 参数明文。
5. `MUST`：关键安全拒绝、越权阻断和紧急授权形成可审计事件；高频拒绝可以限流、聚合或采样，但必须保留计数、代表样本和关联信息，不能因防日志洪泛而失去检测证据。涉及敏感范围或会改变关键业务行为的配置变更、数据导出同样审计，普通公开数据导出按项目风险处理。
6. `MUST`：不得直接记录来源不明、无界或含敏感字段的整个对象 `toString`；异常堆栈在负责处理的边界按正确级别记录一次，避免逐层重复打印和用 INFO 记录失败堆栈。
7. `条件规则`：产品确实提供可自主执行写入、访问受控数据或调用外部工具的 Agent 能力，或审计/合规明确要求逐步追溯时，为高风险或有副作用的 Agent 调用建立独立关联标识，并与上层业务 traceId、责任主体、目标工具、结果状态、耗时和失败原因关联。默认只记录满足审计目的的最小元数据；输入、输出或其摘要仍受数据分级、脱敏和保留期约束。纯推理或无副作用的内部子调用可按风险采样或聚合，不得为追溯默认保存完整提示词和输出。

## 7. 指标、告警与容量

1. `MUST`：关键在线链路至少采集吞吐、成功率或错误率、延迟和主要资源饱和度；批处理或离线链路使用完成率、耗时、积压、数据新鲜度或一致性等更合适的信号，不为凑齐统一四指标制造无意义数据。
2. `条件规则`：项目存在发布、迁移、调度、消息、外部依赖、权限拒绝或降级机制时，对影响恢复判断的状态建立指标、事件或等价可查询证据。
3. `MUST`：每个高优先级告警有明确阈值或判定条件、评估窗口、负责人、Runbook 和恢复确认方式；需要立即响应的安全或完整性事件可以把评估窗口设为零。
4. `MUST`：容量阈值来自项目基线、压测或生产证据；通用规范不硬编码某一产品的表数、行数或超时值。
5. `MUST`：指标标签控制基数；记录 ID、用户 ID、完整租户 ID 等高基数字段不进入全量指标标签。确需诊断时进入访问受控且符合隐私策略的日志或 trace，或采用聚合、采样和脱敏形式。
6. `SHOULD`：灰度发布、降级和恢复条件由相同指标驱动，避免发布计划与监控口径不一致。
7. `条件规则`：承担明确可用性目标的服务必须定义用户可感知的 SLI/SLO；告警优先绑定症状和错误预算，组件指标用于定位，不把“有图表”当作可观测性完成证据。

## 8. 降级、备份与 Runbook

1. `MUST`：每个关键外部依赖声明端到端时限、失败处理、降级边界和恢复策略；没有安全降级方式时明确失败关闭。只有故障模型需要时才引入熔断器，不同模块不得各自发明互相冲突的降级语义。
2. `MUST`：只读降级、写入拒绝或缓存回退的边界必须明确，优先保护数据正确性和隔离安全。
3. `条件规则`：有持久化关键数据的系统必须根据业务影响定义 RPO/RTO；备份覆盖恢复目标所需且允许保留的业务数据、配置、元数据和审计记录，恢复能力以演练结果而非“已配置备份”证明。
4. `条件规则`：需要人工值守或干预的关键故障必须有 Runbook，至少包含症状、影响、确认方法、止血、恢复、回滚、验证和升级联系人。
5. `MUST`：项目按风险分析选出关键恢复路径并在上线前或约定周期内演练；存在消息、复杂发布、关键持久化数据或高风险安全边界时，分别覆盖积压恢复、部分发布失败、备份恢复或安全事件响应。未命中的机制不要求为演练而引入。
6. `SHOULD`：Runbook 中的命令定期验证，并避免依赖个人机器路径或未声明权限。

## 9. 最小验证矩阵

只验证项目实际采用且本次变更命中的能力；其余条目标记为不适用并说明依据：

- 配置缺失、越界、旧 key 兼容和多实例传播延迟。
- 开关关闭、灰度命中、紧急回退和过期清理。
- 错误响应敏感信息扫描及资源探测归一。
- 重复消息、重试耗尽、死信重放和部分成功对账。
- 上下游叠加重试、总超时耗尽和限流恢复信号。
- trace 跨 HTTP、消息和异步任务关联。
- 告警触发、Runbook 执行、降级恢复和按 RPO/RTO 验证的备份恢复演练。
