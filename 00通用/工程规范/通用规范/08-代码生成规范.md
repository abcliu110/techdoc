# 代码生成规范

> 适用范围：代码、SQL、配置、schema、类型、文档和测试样板等生成物。
> 来源：原 `26-代码生成与模板细则.md` 的跨项目规则。
> 迁出边界：低代码字段类型映射、动态 DDL 和元模型版本规则进入低代码平台规范。
> 不包含：编译器、打包器内部的临时中间产物和可删除缓存；这类产物只遵守适用的输出边界、供应链和构建安全要求，不要求逐文件版本元数据或变更预览。

## 1. 约束等级与适用规则

- `MUST`：在对应生成器或生成物类型存在时违反即阻断合并、发布或覆盖操作。
- `SHOULD`：默认遵守；偏离时记录理由、风险和替代验证。
- `条件规则`：仅在所述生成方式、人工编辑或执行环境前提成立时生效。

项目可以使用模板、结构化模型、AST、官方生成器或其他等价方式；不得为了套规范引入新的生成框架。本文中的“人工确认”指项目定义的显式批准，可由受控命令参数、评审或审批流程实现，不要求交互式停顿。

## 2. 生成物契约

1. `MUST`：每类生成物声明为可覆盖、需合并或禁止人工修改；未声明时不得自动覆盖现有文件。可覆盖表示生成器每次完整重写该文件且不保留人工修改；需合并表示生成器与人工修改共存，按第4节 marker 规则处理；禁止人工修改表示文件完全由生成器产出，人工不得编辑。
2. `MUST`：生成物可通过锁文件、构建清单、生成记录或文件元数据追溯到模板版本、生成器版本和输入版本/摘要；不要求每个文件重复写头部。生成时间仅在确有审计需求时写入，避免无意义 diff。最小生成记录格式示例：
   ```text
   生成器：<generator-name>@<version>
   模板：<template-ref>@<version>
   输入摘要：<input-files-hash>
   输出清单：<output-file-list>
   ```
3. `MUST`：相同有效输入、工具版本和已声明环境产生稳定输出；排序、换行、locale、时区和序列化顺序不得随机抖动。
4. `MUST`：覆盖受版本控制、需保留人工修改或会直接进入发布制品的已有文件前，通过预览、暂存区或等价机制产生变更摘要；删除、重命名、类型收窄或覆盖人工区域时阻断并要求显式批准。本条是对第1条"可覆盖"和"需合并"声明的执行约束：即使声明为可覆盖，对已受版本控制且含人工修改的文件仍须先产生变更摘要并经显式批准后方可覆盖。
5. `MUST`：可覆盖生成物可由正式命令完整重建；生成命令和前置条件写入项目文档。
6. `条件规则`：生成器会创建多个文件、重命名或清理陈旧文件时，维护输出清单和归属边界；清理时只能删除清单中由当前生成器拥有的路径，不得按目录或模糊匹配删除未知文件。

## 3. 模板治理

1. `MUST`：可复用或非平凡模板集中存放并版本化，禁止在业务逻辑中拼接大段难以测试的模板文本；短小、局部且无复用价值的固定片段可留在调用处。
2. `MUST`：模板变更附至少一个代表性输入，以及输出快照或语义断言，覆盖边界字符、空值和条件分支。
3. `MUST`：模板不得生成密钥、凭证、绝对机器路径、环境私有配置或未经准入的依赖。
4. `MUST`：不可信输入进入代码、SQL、HTML、Shell 或配置模板时优先使用结构化 API，并按目标语言校验标识符和编码数据；禁止把通用字符串转义当作跨语言安全边界。
5. `SHOULD`：模板复用项目已有格式化器和约定，不为生成器另建一套代码风格。

## 4. 人工编辑与合并

1. `MUST`：需要保留人工编辑时使用结构化模型、AST 或稳定 marker 合并；不得依赖模糊文本替换。
2. `MUST`：marker 必须唯一、可校验并禁止嵌套歧义；缺失或损坏时停止生成，不得猜测插入位置。marker 格式示例：`// @gen:begin <region-name>` 与 `// @gen:end <region-name>` 成对出现，或使用等价的、可被生成器解析的结构化标记；同一文件内 region-name 不得重复。
3. `MUST`：生成器升级不能静默重写历史人工区域；先迁移生成格式或要求显式确认。"迁移生成格式"指升级生成器输出结构、marker 语法或模板版本时，对已有生成物中的人工区域按新格式进行受控转换（如 marker 重命名、区域边界调整、占位符替换），转换过程必须产出 diff 预览并经人工确认，不得在升级时静默改变人工区域内容。
4. `SHOULD`：存在真实人工扩展需求时，优先使用项目已有的独立文件、扩展点或受保护区域，减少长期三方合并；不得仅为预想扩展引入继承层或插件机制。

## 5. 输出边界与写入安全

1. `MUST`：声明的输出根目录先解析为固定边界；不可信输入携带的文件名或相对路径必须规范化并限制在该边界内，拒绝输入中的绝对路径、父目录逃逸和符号链接越界。
2. `MUST`：生成器不得读取或覆盖声明输入与输出范围之外的文件；需要网络、执行外部命令或访问凭据时显式声明并最小授权，日志和生成物不得泄露凭据。
3. `条件规则`：多文件或破坏性生成先写入暂存区并完成校验，再原子替换或按清单提交；失败时保留原文件，或提供可验证的恢复清单，避免半生成状态。
4. `MUST`：生成器、模板包、插件和外部二进制按 `04-安全与供应链规范.md` 完成来源、版本、完整性和 License 准入，不因“仅开发时使用”绕过供应链检查。

## 6. SQL、类型与契约生成

1. `MUST`：DDL 或数据迁移生成器先输出计划，不得生成后立即无审计执行生产变更。
2. `MUST`：标识符经过关键字、长度、字符集、冲突和目标数据库规则校验；运行时或不可信数据值必须参数化。固定迁移字面量使用目标方言的结构化编码能力并接受评审，不得自行拼接转义。
3. `MUST`：同一业务类型只有一个权威映射源；前后端、数据库和文档生成器不得各维护一份分叉映射。
4. `MUST`：从 OpenAPI、Schema 或其他契约生成类型时，破坏性变化由契约 diff 和历史样例回放捕获。
5. `MUST`：生成物必须进入其类型适用的编译、解析、lint、安全扫描和相关测试，不得因文件由生成器产生而跳过验证。

## 7. AI 辅助生成的附加边界

1. `条件规则`：使用 AI Agent 或大语言模型辅助生成代码时，生成结果视为“需合并”类生成物（参见第2节第1条），默认不得自动覆盖已受版本控制的文件；提交前按变更风险审查其正确性、安全性和既有契约一致性。
2. `条件规则`：组织合规、客户合同或项目契约明确要求 AI 来源追溯时，在提交、变更单或生成记录层级记录可获得的模型/工具及版本、时间或提交范围、脱敏后的任务摘要和责任审查人，不要求在每个文件或代码段重复标注。未命中该条件时，以版本控制、评审记录和统一质量门禁作为变更证据。
3. `MUST`：AI 辅助生成物进入发布制品前执行与人工编写内容相同的编译、静态检查、安全扫描和测试；审查强度由变更风险决定，不因作者身份自动提高或降低。
4. `MUST`：AI 建议的依赖、命令、配置和外部内容在验证前视为不可信输入；提示上下文和工具参数遵守 `04-安全与供应链规范.md` 的数据出域、凭据和最小授权边界。

## 8. 验证

只执行项目生成器实际具备的能力和输出类型所命中的条目，未命中的条目标记为不适用：

- 同输入连续生成两次，第二次无 diff。
- 模板快照或语义断言覆盖正常、空值、边界字符和恶意输入；格式不构成契约时优先语义断言，避免无意义快照更新。
- 模拟 marker 缺失、人工区域修改和生成器版本升级。
- 验证输出路径逃逸、符号链接越界、半写入恢复和陈旧文件清理边界。
- 扫描生成物中的凭证、绝对机器路径、未准入依赖和未编码输入。凭证扫描应使用项目已采用的密钥扫描工具（如 Gitleaks、TruffleHog 或等价密钥检测能力），扫描范围覆盖生成物全量文件，不得仅依赖生成器自身的过滤逻辑。
- 对生成物运行编译、格式、静态检查和最小行为测试。
- 对生成 SQL 运行语法、锁风险和回滚边界检查，并验证幂等、迁移框架一次性执行或明确断点恢复语义中的一种。

以上是通用要求，不指定某个生成框架或工具。项目应在 README 中记录实际生成命令、模板目录、权威输入源和门禁状态。
