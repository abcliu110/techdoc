# 协作与文档规范

> 适用范围：所有代码仓库和交付型技术文档。
> 来源：原 `16-注释与文档细则.md`、`18-Git与协作细则.md` 的跨项目规则。
> 说明：本文只定义通用底线；任务卡编号、分支命名和提交格式以项目规则为准。

## 1. 约束等级

- `MUST`：违反即阻断合并或发布。
- `SHOULD`：原则上遵守；偏离时必须在评审记录中说明理由和风险。
- `条件规则`：仅在所述前提成立时生效。

规则只在本文适用范围和条款前提命中时生效。项目级规则可以规定等价或更严格的实现方式；工具名、模板字段和示例不是唯一实现。

## 2. 变更边界与协作

1. `MUST`：一次变更只服务一个明确目标，不得混入无关格式化、重命名、依赖升级或重构。
2. `MUST`：开始修改前检查工作区状态和并行修改，禁止覆盖无法确认来源的已有变更。
3. `MUST`：共享模块或公共契约的变更必须列出调用方、依赖方向、兼容影响和回退方案。
4. `MUST`：冲突解决必须逐处理解双方意图，禁止用整文件 `ours/theirs` 覆盖。
5. `SHOULD`：发现范围外问题时记录为独立事项，不在当前变更中顺手处理。
6. `条件规则`：多人或多 Agent 并行时，每个执行者必须声明任务范围、预计修改文件和依赖契约；共享文件由单一责任人整合。由 Agent 生成的变更，人类责任人必须在提交前验证生成内容的正确性和安全性。

## 3. 提交、评审与合并

1. `MUST`：提交首行说明变更目的；同一提交不得混入无关任务。
2. `MUST`：受保护分支禁止强制推送；分支保护策略由项目配置落实。
3. `条件规则`：使用合并请求、变更单或同类评审载体时至少说明：

```text
变更目标：
关联需求/任务卡：
影响范围：
保留的旧行为：
新增行为及启用条件：
兼容性影响：
安全影响：
新增依赖及理由：
测试或验证证据：
回滚方案：
未覆盖风险：
需要确认的未决项：
```

未命中或当前无法执行的字段必须明确写“不适用”或“未执行”及依据，不得留空造成已验证的误解。

4. `MUST`：评审先识别本次命中的业务兼容、安全、隔离、输入、并发、迁移、恢复和回滚风险，再检查风格；未命中的风险标为不适用，不要求为满足清单引入对应机制。
5. `MUST`：测试证据必须与已识别风险匹配；核心链路存在失败、并发或隔离风险时，不得仅凭正常路径通过。
6. `SHOULD`：评审删除无必要抽象和重复实现；新增抽象必须有现存复用需求或明确边界价值。
7. `条件规则`：紧急修复仍需可追溯分支、最小验证和回滚方案；事后补齐回归测试和事故记录，时限为在紧急修复合并后的下一个工作日内或下一次发布前补齐。
8. `MUST`：验证证据必须绑定到被评审版本，记录命令或检查方法、结果和未执行项；不得复用其他提交的结果或把“规范要求”写成“已经通过”。
9. `条件规则`：项目或工作区要求结构化提交格式（如 Conventional Commits 等）、变更单号或其他结构化提交格式时按入口规则执行；`Tested`、`Not-tested`、风险和约束等记录必须与实际证据一致，不得为了满足格式虚构内容。

## 4. 契约先行

1. `MUST`：有独立下游的接口、消息、数据格式或 SPI 变更，先明确契约、兼容等级；只有存在并行版本、独立发布节奏或长期持久化数据时才需要版本策略。
2. `MUST`：契约冻结后的破坏性变更必须通知所有已知消费者，并通过新版本、兼容层或迁移窗口处理。
3. `MUST`：下游不得依赖未公开的实现类、内部包或临时字段。
4. `条件规则`：涉及单向或高撤销成本决策时必须写 ADR，不能只保留在聊天或评审评论中。

## 5. 代码注释

1. `MUST`：注释解释原因、业务边界、风险或兼容承诺，不复述代码语句。
2. `MUST`：保留反直觉历史行为时，注释必须说明保留原因、影响范围和修改风险。
3. `MUST`：安全关键代码的注释必须说明防御对象、关键不变量和失败后果；不得包含真实密钥、凭据、个人信息或可直接滥用的生产操作信息。需要限制传播的攻击细节进入受控威胁模型（存放位置和访问权限由 `04-安全与供应链规范.md` 定义），不得因此省略安全边界。
4. `MUST`：废弃 API 或字段必须标明替代项、废弃版本和计划移除条件。
5. `MUST`：禁止注释掉的死代码、空 Javadoc 和无责任人或无追踪项的裸 TODO。
6. `SHOULD`：公开 API、复杂事务、幂等入口和扩展点应记录调用语义、事务边界、稳定性与禁止事项。
7. `SHOULD`：可以由类型和命名直接表达的信息不再重复写注释。

## 6. README、ADR 与交付文档

1. `MUST`：可独立构建、部署或被外部使用的模块必须有入口文档，至少包含职责、公开接口、依赖边界、运行与测试方式。
2. `MUST`：ADR 至少包含背景、约束、备选方案、决策、后果、撤销成本和验证方式。
3. `MUST`：文档中的路径、命令和链接必须可验证；路径迁移时更新工作区内已知引用。无法同步更新的外部引用才保留跳转或归档说明，避免为每次内部重排永久保留迁移壳。
4. `MUST`：文档必须区分“规范要求”“示例配置”和“已经落地的事实”，不得把脚本骨架或建议工具描述为现有门禁。
5. `MUST`：命令需要特定系统、版本、凭据、网络或生产权限时必须标明前置条件；不得用无法复现的本机状态充当通用步骤。
6. `SHOULD`：交付文档记录稳定事实与决策，不复制易过期的实现细节；版本敏感内容标明适用版本。

## 7. 最小门禁

- 检查变更中是否混入范围外文件。
- 检查合并请求是否具备验证证据和回滚方案。
- 检查公开契约是否附兼容说明。
- 检查新增 TODO、废弃项和历史兼容分支是否可追踪。
- 检查文档链接与命令是否存在并可执行。
- 检查验证证据是否来自当前被评审版本，未执行项是否如实披露。

以上是门禁要求，不代表任一项目已安装对应扫描器。项目必须在 README 或 CI 配置中列出实际启用的门禁及其验证命令。
