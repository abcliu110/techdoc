# 工程规范分层与迁移说明

> 日期：2026-07-10
> 状态：已完成（v1.2 修订于 2026-07-10）
> 目标：把原 09~29 混合规范拆分为通用工程底线、低代码平台契约和参考资料，消除重复强制条款，并删除被替代的旧文档。
> v1.2 修订摘要：AI Agent 协议改为按任务加载和按风险自检；代码与测试使用统一门禁，仅保留 AI 工具特有的数据出域、未验证内容、覆盖和生产副作用边界；外部调用、Agent 追踪和来源记录按故障模型、审计与合规条件启用。

## 1. 分层原则

1. `通用规范/` 只保留跨项目成立的工程底线。项目可以增加更严格的规则，但不得降低安全、兼容、验证和可回滚要求。
2. `低代码平台规范/` 只承载元模型、动态数据、设计器、插件、应用包、发布状态机等低代码产品契约。
3. `参考资料/` 承载方法、示例和建议。参考资料不以“必须”覆盖项目事实或架构决策。
4. 新规范验收通过后删除原始混合规范。历史追溯依赖 Git，不在工作区保留第二套正文或迁移壳。
5. 新规范使用三级约束：`MUST`（违反即阻断）、`SHOULD`（原则上遵守，偏离需记录理由）、`条件规则`（仅在前提成立时生效）。

## 3. 替换策略

- 删除旧文档前，先把工作区内所有有效引用更新到新路径。
- 原 00、09~29 不保留旧文件名；新目录是唯一有效来源。
- 新项目从根 `README.md` 按任务类型加载，不再默认通读全部规范。
- 低代码项目先读通用规范，再叠加低代码平台规范；发生冲突时，以具体项目 ADR 和产品契约为准，但不得降低通用安全底线。

## 4. 验收标准

- 根索引能明确回答“哪些规则对所有项目生效，哪些只对低代码平台生效”。
- 原 00、09~29 每份文档都有新归属，或有明确的删除理由。
- 工作区内不再引用已删除的旧路径。
- README 中的低代码上游文档链接全部指向实际存在路径。
- 新文档不存在乱码、连续问号、重复一级标题或无去向的相对 Markdown 链接。
- “门禁已要求”和“门禁已实现”明确区分，不能把示例骨架当作运行证据。
- 每份强制规范标注适用范围和通用底线声明，关键条款附交叉引用；根 README 包含按需加载协议；代码和测试门禁不按作者身份分叉，AI 工具特有边界在相关规范中有唯一定位。

## 5. 回滚方式

本次变更只涉及 Markdown 文件和目录结构。回滚时从 Git 恢复原 00、09~29 和根 `README.md`，并删除新增分层目录即可，不涉及运行时代码或数据迁移。
