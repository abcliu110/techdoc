# 工程规范体系

> 版本：v1.5（2026-07-10）
> 定位：按任务加载的工程约束索引。通用底线、低代码平台契约和方法参考分层维护，同一规则只保留一个权威来源。

## 1. 规则治理

所有工程任务先遵循 `00-工程治理总纲.md` 的风险、价值流、复杂性、可逆性和证据原则，再按本 README 路由到具体规范。总纲不替代具体控制点；具体规范可以加严，但不得违背总纲的证据、最小必要治理和底线不可降级原则。

- `MUST` 只用于违反后会造成安全、数据正确性、兼容性、可恢复性或已批准产品契约失效的条款。每条 `MUST` 应能说明适用前提、控制点和验收证据；否则降为 `SHOULD` 或条件规则。
- `SHOULD` 是默认选择，允许基于项目事实、替代控制和验证证据偏离。
- 条件规则仅在对应技术、部署或风险真实存在时生效；不得为满足清单引入项目不需要的框架、平台或抽象。
- 规则优先固化为类型、框架默认值、静态检查或 CI；只能人工判断的内容保留评审入口，但不虚构已经实现的自动门禁。
- 固定技术选型、状态名和数值阈值必须来自 PRD、状态为 accepted 的 ADR、版本化项目契约或实测基线。示例和 proposed/draft ADR 不是项目事实。
- 代码、测试和配置的门禁由变更风险决定，不因作者是人或 AI 而提高或降低。AI 工具只叠加数据出域、自动执行、来源不可信和覆盖写入等工具特有边界。
- 定期根据缺陷、误报、执行成本和技术变化修订规则；没有证据的规则不因“更完整”而新增，也不以固定数量或固定周期机械增删。

## 2. 决策优先级

在系统、开发者和用户直接指令确定的范围内，按以下顺序决策：

1. 用户明确需求、验收标准和当前任务范围。
2. 已验证的代码、数据和运行行为，以及项目入口规则、PRD、accepted ADR 和对外契约。
3. 当前场景命中的强制工程规范，尤其是安全、兼容、验证和可回滚底线。
4. 官方资料、AI 知识库和工程经验，用于在上述边界内选择实现方法。
5. 方法类参考资料，只提供启发，不产生新的强制约束。

项目事实决定“做什么、兼容什么”，工程规范限定“不能突破什么”，AI 知识负责“怎么做”。更具体的项目事实可以替代通用默认方案，但不能把安全、兼容、验证和可回滚底线降级。证据不足时标记未知并执行最小验证，不得把训练语料、示例或参考资料写成项目事实。

## 4. 按需加载协议

1. 先读取当前项目入口规则和与任务直接相关的事实源。简单问答、路径查询或局部文档修改不要求加载无关 PRD、架构和 ADR。
2. 从上表选择当前任务命中的规范。普通局部任务通常只需 1~2 份通用规范；低代码任务再叠加一份主契约和风险矩阵相关行。
3. 只有跨域变更、全局架构评审或规范体系评审才扩大读取范围；禁止默认通读全部规范，也不要求扫描未加载文档中的全部条件规则。
4. `MUST` 只在其适用前提已被项目事实或任务范围命中时阻断；`SHOULD` 偏离说明理由和替代控制；未命中的条件规则无需构造 N/A 报告。
5. 引用规则时使用可解析的“相对路径 + 节号”，例如 `通用规范/04-安全与供应链规范.md §7`。正文未定义稳定规则 ID 时不得虚构编号。
6. 设计、高风险变更和评审输出应区分事实、推断/假设、适用范围和未覆盖风险；简单问答按问题直接回答，不强制套固定四段模板。
7. R3/R4 变更必须执行 `00-工程治理总纲.md` 的统一工程评审七问；R1/R2 只记录命中的维度，不为格式完整制造无价值文档。

## 5. 冲突与证据

- proposed/draft ADR 只能作为候选设计；只有 accepted 或明确 `accepted for Mx` 的决策才能约束对应里程碑的代码、DDL、公共 DTO 和阻断性测试。
- 低代码平台契约在其适用范围内叠加于通用规范；项目 accepted ADR 可以替代规范中的默认实现选项，但不能降低安全、兼容、验证和可回滚底线。
- 文档写有门禁、测试骨架或示例，不等于门禁已经实现。交付结论必须引用本次测试、配置、CI 或运行记录。
- 仍无法消解的真实冲突标记为待裁决，说明冲突双方、影响和最小验证，不得静默任选一边。

分层和原文档合并关系见 `规范分层与迁移说明.md`。
