# 小程序壳化与配置协议低代码平台设计

## 一、问题背景

传统小程序开发模式里，页面、组件、交互逻辑和业务流程大多写在小程序代码包中。只要新增页面、调整复杂交互、修改业务入口，就可能需要重新构建、上传体验版、提审、发布。对于商户中心、会员营销、扫码点餐这类高频变化业务，这种发布链路成本很高。

现有“小程序装修”系统已经具备一定配置化能力：后台可以配置页面背景、组件顺序、图片、标题、跳转入口等，小程序端按页面类型拉取配置后动态渲染。但它本质仍然是页面装修器，主要解决“页面长什么样”和“入口跳到哪里”，还没有完全解决页面新增、静态内容管理、事件编排、发布治理和性能缓存这些系统性问题。

更理想的方向是把小程序做成一个稳定的壳：小程序代码包内置运行时引擎、组件库和动作执行器，后端发布页面配置协议。高频业务变化走配置发布，低频底层能力变化才发小程序版本。

## 二、当前装修系统的本质

当前实现可以抽象成三层数据：

```text
wx_page
  页面级配置：页面类型、商户、门店、背景图、背景色、是否全屏、高度

wx_componet
  页面组件：标题、轮播图、分组、会员信息、视频、权益卡等

wx_componet_item
  组件子项：图标、文字、跳转页面、跳转链接、排序
```

后台保存时生成页面和组件数据。小程序打开页面时，根据 `type` 请求后端配置，例如 `HOME`、`MEMBER_CENTER`、`ORDER`。后端按 `mid + sid + type` 查询配置，有门店配置则用门店配置，没有则回退到商户默认配置，再没有则使用系统默认页面。

小程序端拿到 `componetList` 后，根据组件类型执行固定渲染：

```text
TITLE          -> 标题组件
MEMBER_INFO    -> 会员信息组件
GROUP          -> 分组入口组件
SLIDESHOW      -> 轮播图组件
CARD_BENEFITS  -> 卡权益组件
ROLLER_SKATING -> 滑动图组件
VIDEO          -> 视频组件
```

这个模型的优势是简单、稳定、容易理解。缺点是能力边界明显：后台只能配置已经写好的组件和跳转，不能自由新增业务逻辑，也不能真正管理页面版本和发布过程。

## 三、目标架构：小程序壳 + 配置协议

理想架构不是让后端下发代码，而是让后端下发 Schema，小程序内置引擎解释执行。

```text
小程序代码包
  Runtime Engine        运行时引擎
  Component Registry    组件注册表
  Action Executor       动作执行器
  Data Binding          数据绑定
  Rule Engine           规则引擎
  Theme System          主题系统
  Fallback Pages        兜底页面

后端平台
  Page Schema           页面配置
  Component Schema      组件配置
  DataSource Schema     数据源配置
  Action Schema         事件动作配置
  Version Service       版本服务
  Publish Service       发布服务
```

小程序运行流程：

```text
读取 appId / 门店 / 页面类型或 pageId
  ↓
请求后端 published schema
  ↓
运行时引擎解析 schema
  ↓
根据 componentRegistry 渲染内置组件
  ↓
绑定数据源
  ↓
绑定事件动作
  ↓
用户操作时由 actionExecutor 执行白名单动作
```

这种模式下，小程序代码包是稳定壳，页面内容、组件顺序、样式、跳转和部分交互由后台发布配置控制。

## 四、配置协议示例

一个会员中心页面可以描述为：

```json
{
  "pageType": "MEMBER_CENTER",
  "title": "会员中心",
  "theme": {
    "primaryColor": "#2F6BFF",
    "backgroundColor": "#F5F7FB"
  },
  "components": [
    {
      "id": "member-card",
      "type": "MemberCard",
      "props": {
        "showBalance": true,
        "showPoints": true,
        "showLevel": true
      }
    },
    {
      "id": "quick-actions",
      "type": "ActionGrid",
      "props": {
        "columns": 3
      },
      "items": [
        {
          "title": "会员充值",
          "icon": "https://example.com/recharge.png",
          "action": {
            "type": "navigate",
            "target": "RECHARGE"
          }
        },
        {
          "title": "积分商城",
          "icon": "https://example.com/points.png",
          "action": {
            "type": "navigate",
            "target": "POINTS_MALL"
          }
        }
      ]
    }
  ]
}
```

小程序不会执行远程代码，只会识别：

```text
MemberCard -> 使用内置会员卡组件
ActionGrid -> 使用内置宫格组件
navigate   -> 执行内置跳转动作
```

这就是配置协议的关键边界：后端下发的是结构、参数和动作描述，不是 JavaScript 代码。

## 五、新增页面如何处理

新增页面分两类。

第一类是新增配置页面，不需要发版。小程序壳中预置一个通用运行时页面：

```text
/pages/runtime/index
```

后台新增页面配置：

```text
pageId: campaign_20260701
title: 夏季充值活动
components: [...]
actions: [...]
```

小程序跳转时使用：

```text
/pages/runtime/index?pageId=campaign_20260701
```

运行时页面根据 `pageId` 拉取后端配置并渲染。活动页、会员规则页、优惠券落地页、积分专题页、门店介绍页都适合这种模式。

第二类是新增底层能力，需要发版。如果新页面需要地图选点、蓝牙、摄像头、扫码、支付新流程、复杂授权等小程序壳没有内置的能力，就必须新增组件或动作处理器，然后发一次小程序版本。

因此合理边界是：

```text
新增页面 = 通常不发版
新增能力 = 可能要发版
```

## 六、静态页面如何处理

静态页面不应该每个都写成小程序代码文件。小程序壳中只需要一个通用静态页入口：

```text
/pages/static/index
```

后台维护静态页内容：

```json
{
  "slug": "recharge_rule",
  "title": "充值规则",
  "blocks": [
    {
      "type": "heading",
      "props": {
        "text": "充值规则"
      }
    },
    {
      "type": "richText",
      "props": {
        "html": "<p>1. 充值金额不可提现</p><p>2. 赠送金额不可兑换现金</p>"
      }
    },
    {
      "type": "image",
      "props": {
        "src": "https://example.com/rule.png",
        "mode": "widthFix"
      }
    },
    {
      "type": "button",
      "props": {
        "text": "去充值"
      },
      "action": {
        "type": "navigate",
        "target": "RECHARGE"
      }
    }
  ]
}
```

访问方式：

```text
/pages/static/index?slug=recharge_rule
```

静态页渲染器只需遍历 `blocks`，按 `block.type` 找对应内置块组件：

```text
heading  -> 标题块
richText -> 富文本块
image    -> 图片块
button   -> 按钮块
divider  -> 分割线
space    -> 留白
notice   -> 提示条
card     -> 卡片
```

新增会员规则、充值说明、积分规则、门店介绍、活动说明、隐私协议等静态页，都只需要后台新增内容配置，不需要发小程序。

## 七、前端 JS 逻辑如何处理

小程序不能像 Web 一样随便从后端下载一段 JS 执行。这样既有安全风险，也容易触碰平台审核边界。

正确做法是动作白名单：

```text
navigate          跳转页面
navigateToStatic  跳转静态页
navigateToRuntime 跳转配置页
requestApi        调用后端接口
showToast         显示提示
showModal         显示确认弹窗
openPopup         打开弹窗
closePopup        关闭弹窗
setState          写页面状态
refreshComponent  刷新组件
trackEvent        埋点
```

后台只能配置这些动作：

```json
{
  "type": "requestApi",
  "api": "coupon.receive",
  "params": {
    "couponId": "{{item.id}}"
  },
  "onSuccess": [
    {
      "type": "showToast",
      "message": "领取成功"
    },
    {
      "type": "refreshComponent",
      "componentId": "couponList"
    }
  ]
}
```

小程序动作执行器根据 `action.type` 找内置处理器执行。如果已有动作可以表达新逻辑，就不需要发版；如果需要一种全新的动作类型，就要升级小程序壳。

复杂业务逻辑应优先放到后端接口中，小程序只通过 `requestApi` 调用。比如会员升级、领券、充值、订单操作等规则，不应该由前端配置写复杂逻辑，而应该由后端保证一致性。

## 八、真正低代码平台需要哪些能力

页面装修器只解决展示问题，真正低代码平台至少要补齐以下能力：

```text
页面编排：页面、容器、组件、子组件
数据源：接口、参数、字段映射、加载状态、错误状态
事件系统：onClick、onLoad、onShow、onSubmit、onSuccess、onFail
动作系统：跳转、请求、弹窗、状态写入、刷新、埋点
状态管理：页面状态、组件状态、用户状态、门店状态
规则引擎：显示条件、会员等级、时间范围、门店能力
发布治理：草稿、预览、发布、灰度、回滚、版本对比
```

但不建议一开始做通用低代码。更合理的是做领域低代码，例如餐饮、会员、营销、扫码点餐场景。这样组件、数据源、动作和安全边界都可控，成功率更高。

## 九、发布模型

当前很多装修系统把“保存并发布”简化成写库并清 Redis。对于小规模页面可以工作，但长期看治理能力不足。

更合理的发布模型是：

```text
Draft      草稿
Preview    预览包
Published  已发布版本
Archived   历史版本
```

发布流程：

```text
编辑草稿
  ↓
保存草稿
  ↓
发布前校验
  ↓
生成发布包
  ↓
写入 published version
  ↓
清 Redis / CDN 缓存
  ↓
小程序拉取新版本
```

发布前应检查：

```text
图片是否存在
跳转目标是否有效
组件必填项是否完整
数据源是否可用
门店范围是否正确
是否影响线上页面
```

发布系统必须支持：

```text
一键回滚
版本对比
发布记录
发布人
发布时间
影响门店
灰度发布
定时发布
```

最终目标是让高频业务变化通过后台发布配置完成，而不是频繁提交小程序审核。

## 十、性能设计

配置协议本身不是性能问题，运行时设计才是关键。

错误做法：

```text
每次打开页面都实时查多张表并组装
schema 巨大且嵌套过深
所有组件同时渲染
所有数据源同时请求
大量大图和富文本无懒加载
没有本地缓存
没有版本号机制
```

正确做法：

```text
发布时预生成 published schema
运行时直接返回发布包
小程序按 pageId + sid + version 本地缓存
先渲染本地缓存，再后台检查版本
首屏组件优先渲染
非首屏组件延迟渲染
图片懒加载
业务数据和页面结构分离
静态页按 slug + version 缓存
```

页面结构和业务数据要分开：

```text
页面 schema：控制布局，缓存时间长
业务数据：会员余额、优惠券、订单，按需刷新
```

这样二次打开页面时可以非常快。小程序先展示本地最后一次成功配置，再异步检查线上版本号。如果版本没有变化，不需要重新拉完整 schema。

## 十一、与当前系统的升级路径

不建议一开始推翻现有系统。更稳妥的路线是渐进式升级：

第一阶段：前端统一 Schema 和转换层。

```text
编辑器内部使用 PageSchema
保存前转换成现有 WxPage / WxComponet / WxComponetItem DTO
后端接口暂时不动
```

第二阶段：组件注册表。

```text
组件默认值
属性 schema
后台预览
小程序渲染
校验规则
版本迁移
```

第三阶段：编辑体验升级。

```text
左侧页面树
中间可编辑手机画布
右侧属性面板
组件抽屉
选中描边
拖拽排序
发布确认
```

第四阶段：发布包和版本治理。

```text
草稿
预览
发布
回滚
门店灰度
发布记录
```

第五阶段：小程序运行时注册表化。

```text
去掉大 switch
改成 componentRegistry
引入 actionRegistry
引入本地缓存和版本检查
```

这条路线能保留旧系统兼容性，同时逐步把装修器升级为真正的配置协议运行时。

## 十二、最终边界

小程序可以是壳，但不是完全不变的壳。

合理边界是：

```text
页面内容、组件顺序、图片、文字、样式、跳转、显示条件
  -> 后台配置发布，不发小程序

新增静态页、活动页、规则页、专题页
  -> 后台新增 pageId/slug，不发小程序

新增已内置组件的组合方式
  -> 后台配置，不发小程序

新增小程序不认识的组件类型
  -> 升级壳，发小程序

新增小程序不认识的动作类型
  -> 升级壳，发小程序

新增微信原生能力、支付、登录、授权、定位、蓝牙、扫码等底层能力
  -> 升级壳，发小程序
```

一句话总结：

```text
壳低频升级，配置高频发布。
小程序内置稳定引擎，后端发布页面协议。
页面是配置，组件是能力，事件是白名单动作，业务逻辑优先放后端。
```

这套设计的目标不是让小程序永远不发版，而是让绝大多数运营和页面变化不再依赖小程序发版，把微信审核链路从高频业务变化中剥离出来。
