# 微信小程序页面栈、业务单页面状态机、自动化上下文定位

资料基准：以 `D:\mywork\taro-mall` 当前源码为准  
适用场景：小程序自动化、页面状态识别、点餐业务定位、测试平台上下文抽取  
文档目标：把“小程序有 4 个 tab 树”和“点餐业务按单页面状态机理解”统一起来，给自动化脚本一个稳定的定位模型。

## 1. 结论先行

这个小程序在技术上是“多页面 + tabBar + 分包”的结构，但在点餐业务上，自动化不应该把它理解成一堆彼此平铺的页面，而应该理解成一棵业务状态机。

更准确地说：

- 第一层先看当前栈顶页面 `path`，先判断大状态。
- 如果 `path` 已经唯一确定业务状态，就直接返回，不再继续找 DOM。
- 如果是复合状态，尤其是 `pagePop/orderfoods/index`，再按当前最前面的可见覆盖层判断子状态。
- 原子操作只能在当前状态的 root 内找元素，禁止全局扫按钮。

这个结论和源码是一致的：

- `app.config.ts` 里明确只有 4 个 tabBar 树：`page0/page1/page2/page3`。
- `pagePop` 分包里挂着大量业务页面，包括 `orderfoods`、`cart`、`orderstatus`、`paycenter`、`selectnumberofpeople` 等。
- `orderfoods/index.tsx` 内部同时存在主列表、套餐弹层、单品详情弹层、券弹层、图片弹层，说明它本质是一个单页面业务容器，而不是一个纯列表页。

## 3. 点餐业务为什么要按单页面状态机理解

`orderfoods/index.tsx` 不是一个“纯列表页”，而是一个业务容器页，内部同时承载：

- 点菜主列表
- 套餐弹层
- 单品详情弹层
- 券弹层
- 图片弹层
- 门店公告
- 购物车入口

也就是说，用户在 `ORDER_FOODS` 这个业务状态里看到的不是单一页面，而是同一业务树里的一组可叠加子状态。

### 3.1 主状态：点菜列表

主列表容器是：

- `.main-section`
- `.orderfoods-scroll`

这部分是点菜业务的根 DOM 区域，自动化原子操作应优先在这里查找，而不是全局扫整页。

## 4. 覆盖层的层级关系

你给出的自动化规则里，最关键的是这一句：

> 先取栈顶 path；path 能唯一确定就直接返回业务状态；复合 path 再按最前可见覆盖层判断。

这个规则和源码完全对得上。

### 4.1 `orderfoods/index.tsx` 里的覆盖层顺序

源码里可以直接看到以下覆盖结构：

- `DishsSetMeal`
- `DishsDetail`
- `renderCouponPopup()`
- 图片 `Overlay`

它们都挂在同一个页面容器里，所以自动化不该把它们当成多个路由页面处理。

## 5. 自动化上下文定位规则

这里给出推荐的实际判定顺序。

### 5.1 第一步：先看栈顶 path

先判断当前页面属于哪棵树、哪个业务入口。

建议至少分成这几类：

- `pagesIndex/page0/index` - 首页树
- `pagesIndex/page1/index` - 点餐树
- `pagesIndex/page2/index` - 订单树
- `pagesIndex/page3/index` - 我的树
- `pagePop/selectnumberofpeople/index` - 点餐前置状态
- `pagePop/orderfoods/index` - 点餐业务主状态
- `pagePop/orderstatus/index` - 订单状态
- `pagePop/paycenter/index` - 支付中心

如果 path 已经唯一确定业务目标，就直接返回，不再继续解析 DOM。

### 5.2 第二步：复合 path 再看最前可见覆盖层

特别是 `pagePop/orderfoods/index`，应该再按覆盖层判断子状态：

1. `DishsDetail Overlay` 先命中时，状态就是“规格/做法/确认”。
2. `DishsSetMeal Overlay` 先命中时，状态就是“套餐配置”。
3. `coupon Popup` 先命中时，状态就是“券核销”。
4. 都没有覆盖层时，回到点菜主列表。

结合源码，这个优先级是合理的，因为这些层本来就是在同一个页面上叠加出现的。

### 5.3 第三步：原子操作只在当前 root 内找元素

这是最重要的自动化约束。

在 `ORDER_FOODS` 里，脚本不能全局扫描“加入购物车”“确定”“加号”之类按钮，否则容易误点到：

- 套餐层里的按钮
- 单品详情里的按钮
- 券弹层里的按钮
- 图片弹层里的关闭按钮

正确做法是：

- 先确定业务状态。
- 再确定当前可见 root。
- 只在这个 root 内做元素定位。

这条规则和源码中的 root 划分一致：

- 点菜根容器：`.main-section` / `.orderfoods-scroll`
- 单品详情 root：`DishsDetail`
- 套餐 root：`DishsSetMeal`
- 券弹层 root：`Popup`

## 6. 业务状态机的推荐抽象

如果把点餐流程抽象成状态机，可以写成下面这样：

```text
TAB_HOME
TAB_ORDER
TAB_BILL
TAB_MEMBER
    │
    └── TAB_ORDER
          ├── SELECT_NUMBER_OF_PEOPLE
          ├── ORDER_FOODS
          │     ├── ORDER_FOODS_LIST
          │     ├── ORDER_FOODS_SET_MEAL_OVERLAY
          │     ├── ORDER_FOODS_DISH_DETAIL_OVERLAY
          │     ├── ORDER_FOODS_COUPON_POPUP
          │     └── ORDER_FOODS_IMAGE_OVERLAY
          ├── CART
          ├── ORDER_STATUS
          └── PAY_CENTER
```

这个抽象的关键点不是“页面有多少个”，而是“当前可见业务状态是什么”。

### 为什么这样抽象更稳

- 技术路由会变，但页面内状态往往更稳定。
- 同一路由内的 Overlay / Popup 会不断新增，但它们仍然属于同一个业务 root。
- 自动化脚本按状态机写，比按 DOM 绝对路径写更抗改版。

## 7. 落地建议

### 7.1 状态识别优先级

推荐顺序：

1. 栈顶 `path`
2. 页面级覆盖层
3. 当前 root 内部可见 DOM
4. 组件级局部状态

### 7.2 自动化定位边界

建议把定位边界固定成以下几层：

- `TAB` 层：`pagesIndex/page0~page3`
- `PAGE` 层：`pagePop/selectnumberofpeople`、`pagePop/orderfoods` 等
- `OVERLAY` 层：`DishsDetail`、`DishsSetMeal`、券弹层、图片弹层
- `ROOT` 层：列表区、弹层区、按钮区

### 7.3 不建议的做法

- 不要把 `pagePop` 内所有页面都当成平级独立业务。
- 不要在 `orderfoods` 页面上全局扫按钮。
- 不要把 `DishsDetail` 和 `DishsSetMeal` 当成 DOM 父子页面关系，它们是业务上的子状态，不是路由父子。
