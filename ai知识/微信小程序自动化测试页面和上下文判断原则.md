# 微信小程序自动化测试页面和上下文判断原则

## 适用范围

本文总结微信小程序自动化测试中的页面判断、上下文判断和原子操作原则，目标是让脚本始终按“真实可见业务状态”执行，而不是按页面名字、按钮文本或全局 DOM 盲扫执行。

本文只描述测试判断原则，不涉及测试平台代码实现。

## 核心结论

微信小程序自动化不能按浏览器网页那样理解成单层页面树，而要按“`path + tabTree + context`”三元组理解。

判断顺序必须固定为：

1. 先判断当前 `path` 属于哪棵 `tabBar` 根树。
2. 再判断当前 `tabTree` 下处于哪个业务上下文。
3. 再判断目标元素是否位于当前可见子树或弹窗内。
4. 最后才执行 `tap`、`trigger`、`switchTab` 或返回类操作。

只要这三个层级没有先判清，就不允许直接点目标元素。

## 1. 4 个 tabBar 根树

小程序在业务上可以抽象成 4 个 `tabBar` 根树，例如：

- 首页
- 点餐 / 账单入口
- 订单 / 账单列表
- 会员中心

这 4 棵树是自动化测试的最高层边界。

原则上，`switchTab` 只能在 `tabBar` 根页面之间使用。  
如果当前处于子页面、全屏支付中心、弹窗覆盖层或其他非根态，就不能直接 `switchTab` 跳转。

### 正确做法

- 先通过页面顶部的主页、返回或关闭能力回到底层。
- 回到底层后，再从 `tabBar` 根页面执行 `switchTab`。

### 错误做法

- 在子页面里直接 `switchTab`。
- 在弹窗遮罩层上直接切 tab。
- 把支付中心当成普通根页面处理。

## 2. `path` 先于一切

原子能力的第一步不是找元素，而是先判断 `path`。

`path` 的作用是确认当前页面属于哪一棵业务树，以及当前状态是不是已经偏离根页面。

### 建议的判定方式

| 判定项 | 目的 | 结果含义 |
|---|---|---|
| `path` | 确认页面归属 | 判断是否在 `tabBar` 根树或业务子页 |
| `tabTree` | 确认业务入口 | 判断当前属于首页、点餐、订单还是会员 |
| `context` | 确认可见状态 | 判断当前是列表、详情、弹窗还是支付中心 |

只要 `path` 不对，后续的元素定位就没有意义。

## 3. 同一路径内按可见子树判断

同一个 `path` 内部，业务状态可能完全不同。  
因此不能把小程序当成“单页面单状态”处理，而要按可见子树和弹窗状态判断。

典型场景是点餐页：

- 人数选择
- 规格选择
- 购物车
- 下单结果
- 支付中心

这些都属于同一路径内的业务状态或子视图，不是全局独立页面。

### 处理原则

1. 先确认当前是否仍在点餐业务树。
2. 再确认当前是否处于某个可见子树或弹窗。
3. 只有在目标子树可见时，才定位目标元素。
4. 定位范围必须限制在当前 root 内，禁止全局乱扫。

### 为什么不能全局扫

因为同一个按钮文本可能同时出现在：

- 主列表
- 弹窗
- 详情层
- 支付中心
- 购物车层

如果不先判断可见子树，就很容易点到错误按钮。

## 4. 原子能力的执行顺序

每一个自动化原子能力都要遵守下面的顺序：

1. 判断 `path`。
2. 判断 `tabTree`。
3. 判断 `context`。
4. 判断目标元素所在子树是否可见。
5. 只在可见 root 内定位元素。
6. 执行正确的 `tap`、`trigger` 或返回动作。
7. 验证状态是否真的变化。

### 关键要求

- 不能跳过上下文判断直接定位元素。
- 不能把“能找到元素”当成“当前状态正确”。
- 不能把一次点击的结果直接当成业务成功，必须看页面、文本、弹窗或请求状态是否变化。

## 5. 学习原子和正式原子要分离

学习原子的作用只有一个：发现稳定选择器和稳定定位规则。

学习阶段可以用来做：

- 探测页面结构
- 识别可见 root
- 收集稳定的 `selector`
- 归纳业务上下文规则

正式原子不能直接读取学习产物，而要把学习和源码分析得到的稳定定位规则固化到代码里。

### 这条原则的含义

- 学习原子是“发现工具”。
- 正式原子是“执行工具”。
- 两者不能混用。

### 不推荐的做法

- 正式执行时动态依赖学习结果文件。
- 临时扫描结果直接当最终选择器。
- 把实验性 selector 直接复制进稳定业务流。

### 推荐的做法

- 学习阶段输出稳定规则。
- 通过源码分析确认规则来源。
- 将规则写入正式原子代码。
- 正式原子只消费固化后的规则。

## 6. 会员必须有 `openid`

微信小程序的业务操作必须使用有 `openid` 的会员。

这意味着：

- 不能拿匿名用户做正式业务操作。
- 不能用没有绑定身份的账号去跑需要会员态的流程。
- 不能把“页面能打开”误判成“业务可以执行”。

### 原因

很多小程序业务状态依赖会员身份、授权态和服务端用户上下文。  
如果没有 `openid`，页面定位即使成功，业务结果也可能不成立。

### 自动化要求

每次执行业务流前，必须先确认当前账号具备有效会员身份，并且有可用 `openid`。

## 7. 推荐的判断模板

可以把自动化判断模板固定为下面这套顺序：

```text
1. 读取当前 path
2. 识别 tabTree
3. 识别 context
4. 判断当前可见 root
5. 只在 root 内定位元素
6. 执行 tap / trigger / switchTab / 返回
7. 复核页面是否进入目标状态
```

这套模板适用于：

- 根页面切换
- 点餐页操作
- 购物车和下单流程
- 支付中心处理
- 弹窗和覆盖层处理

## 8. 常见误区

### 误区 1：把子页面当 tabBar 根页面

子页面、全屏支付中心、弹窗状态都不是 `tabBar` 根页面。  
它们必须先回到底层，再做 `switchTab`。

### 误区 2：把同一路径内的所有状态当成一个页面

同一路径内可能存在多个可见子树。  
必须先判断当前子树，再找元素。

### 误区 3：全局扫按钮

全局扫会把隐藏层、遮罩层和错误 root 里的按钮一起扫出来，容易误点。

### 误区 4：正式原子直接读取学习结果

学习结果只用于分析，不应成为正式执行时的外部依赖。

### 误区 5：忽略会员身份

没有 `openid` 的账号不适合跑正式业务流，结果不具备可重复性。

## 9. 一句话总结

微信小程序自动化测试的核心，不是“找到一个按钮”，而是“先判断页面树和上下文，再在可见业务子树里执行正确操作”。
