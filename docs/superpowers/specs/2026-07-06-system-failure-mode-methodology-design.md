# 系统级失败模式与设计约束清单方法论设计

## 背景

现有低代码平台已经形成 `07-低代码平台级架构陷阱与高难度问题清单.md`，用于约束 PRD、架构设计、详细设计、工程规范、测试规格和任务验收。该文档本质不是普通“架构风险列表”，而是把高后果失败模式提前显性化，并要求后续设计、规范、测试和评审逐条闭环。

当前问题是：低代码清单已经具备领域实例价值，但通用方法论和工作流引用还不完整。其他产品如果继续复制低代码清单，会把低代码专属问题误当成通用规则；如果完全不引用，又会丢掉“失败模式驱动设计约束”的工程价值。

## 目标

建立一套分层文档体系，用于回答三类问题：

1. 这类文档到底描述什么问题，如何正确命名、生成、维护和使用。
2. 所有软件系统通用的失败模式与设计约束有哪些。
3. 企业级复杂平台通用的失败模式与设计约束有哪些。

低代码平台现有 07 清单保留为领域实例，并在开头声明其来源层级。

## 非目标

- 不把低代码平台专属风险搬到全局通用清单。
- 不把通用方法论变成编码规范正文。
- 不要求编码时重新执行整套方法论。
- 不用通用清单替代领域清单；通用清单只提供起点和校验框架。

## 推荐命名

通用命名采用“失败模式与设计约束清单”，而不是单独使用“架构陷阱清单”。

原因：

- “失败模式”强调系统在什么条件下会坏。
- “设计约束”强调后续设计、规范、测试必须回答什么。
- “陷阱”适合领域实例标题，但过于口语化，不适合作为方法论主名。

### 1. 系统级失败模式与设计约束清单方法论.md

用途：定义这类文档是什么、什么时候使用方法论、什么时候使用清单。

核心章节：

- 定义：失败模式、设计约束、高后果风险、机器防线、人工评审。
- 使用时机：产品设计前、架构设计阶段、规范编写阶段、知识库升级后、重大变更前、事故复盘后。
- 清单使用时机：PRD、架构设计、详细设计、编码规范、测试设计、任务验收、复盘回写。
- 条目结构：编号、等级、失败模式、触发条件、事故后果、设计必须回答、机器防线、人工评审、证据来源、承接文档。
- 维护规则：新增条目必须有证据来源；领域条目不能直接上升为通用条目；清单随知识库和复盘持续演进。

### 2. 通用系统级失败模式与设计约束清单.md

用途：作为所有软件系统的基础风险清单。

建议分类：

- 数据一致性与状态不变量
- 权限、安全与隐私
- 并发、幂等与重复执行
- 接口契约与兼容演进
- 配置、发布、迁移与回滚
- 错误处理、补偿与恢复
- 可观测性、审计与运维
- 性能、容量与资源隔离
- 测试盲区与环境漂移
- AI 协作与过度抽象

### 3. 复杂平台级失败模式与设计约束清单.md

用途：面向低代码、CRM、ERP、SaaS、工作流、数据平台等企业级复杂平台。

建议分类：

- 多租户与数据范围
- 元数据、配置驱动与动态模型
- 插件、扩展点与生态兼容
- 工作流、状态机与业务副作用
- 导入导出、开放 API 与外部集成
- 组织、角色、权限与历史语义
- 平台级发布、私有化、License 与升级
- 运行时资源配额、噪声租户与容量模型
- 设计器、配置器、模板与运行态一致性
- 商用交付、行业模板与客户定制保护

## 验收标准

- 三篇通用文档存在且边界清晰。
- 低代码 07 清单开头明确其领域实例地位。
- 三个方法论/工作流文档引用通用方法论和清单。
- 旧的“架构风险清单”说法在关键流程中升级为“系统级失败模式与设计约束清单”。
- 工作流明确区分：生成和维护清单时用方法论；设计、编码、测试、评审时用清单。
- 全文无空泛口号，每个清单条目都要求有承接文档或防线。

## 风险与处理

- 风险：通用清单写得过细，变成另一个领域清单。
  处理：通用清单只保留跨系统稳定成立的问题，领域细节放到领域实例清单。
- 风险：工作流引用过多，执行成本变高。
  处理：工作流只把通用方法论作为生成和维护清单的标准，不要求每次编码重复执行。
- 风险：低代码现有 07 与通用清单重复。
  处理：低代码 07 保留领域细节，通用清单只保留抽象问题族。
