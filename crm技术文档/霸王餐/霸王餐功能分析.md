# 霸王餐功能分析

## 1. 结论摘要

霸王餐是 CRM 储值体系里的一个特殊营销能力，本质上是“按消费金额命中规则后，赠送一笔只能按霸王餐规则使用的余额”。

- 后端主链路已具备：规则配置、规则查询、规则命中、充值提交、赠送余额入账、参与次数限制、订单收尾。
- `taro-mall` 主支付链路已接入：点餐页、支付中心、收银页都能触发霸王餐。
- `nms4pos`（POS 后端）、`nms4pos-ui`（POS 前端）、`uni4merchant` 当前未检索到可触发霸王餐的实现。
- `nms4pos-ui` POS 前端对 `dashBalance` 完全无感知，不会主动展示霸王餐余额或提示整额使用限制。
- `taro-mall` 的 `pointsmallpaycenter` 页面里，霸王餐入口代码存在但处于注释/未启用状态。

总体判断：**核心功能基本可用，但多端覆盖不完整，存在局部链路未启用与接口语义不够统一的问题。**

## 2. 功能本质

霸王餐不是普通充值套餐，而是一个特殊储值类型：

- 前端提交时用 `ruleType = -2` 标识霸王餐。
- 后端查询命中规则后，不直接返回真实充值套餐，而是构造一个虚拟储值规则。
- 该虚拟规则的 `lid = -2`，表示这是霸王餐充值。
- 真实命中的霸王餐规则主键会放在 `ruleTypeLid`，用于后续参与次数统计。

命中后，系统会：

1. 按账单金额和倍数生成赠送金额。
2. 生成充值任务与任务子项。
3. 把霸王餐赠送金额记入会员卡 `dashBalance`。
4. 记录该会员卡对该霸王餐规则的参与次数。

### 2.1 霸王餐和普通充值的关系

霸王餐和普通充值不是两套完全独立的系统，关系更接近：

- 普通充值：复用 CRM 充值框架，按普通储值规则充值并入普通余额体系
- 霸王餐：同样复用 CRM 充值框架，但走 `ruleType = -2` 的特殊分支，命中霸王餐规则后把金额记入 `dashBalance`

可以概括为：**霸王餐是挂在普通充值框架上的一个特殊营销型充值分支。**

直接证据：

- 两者都走充值服务 `ChargeService`，霸王餐只是其中一个特定分支：
  - `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/ChargeService.java:183-210`
- 霸王餐不会直接返回真实充值套餐，而是构造一个虚拟规则：
  - `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/card/CrmCardOpServicePlus.java:2268-2276`
- 充值任务模型与普通充值共用同一套 `CrmDealTask` / `CrmDealTaskItem`：
  - `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/ChargeService.java:339-377`
  - `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/ChargeService.java:390-403`
- 霸王餐的差异不在“是否走充值框架”，而在“充值后的余额落点和使用限制”：
  - 入账到 `dashBalance`：`nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/balance/CardBalanceService.java:1797-1802`
  - 使用时强制整额核销：`nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/balance/CardBalanceService.java:1032-1042`

归纳：

- 相同点：都走 CRM 充值任务模型、都能形成会员卡资金变动、后续都走会员卡消费链路
- 不同点：普通充值主要处理普通余额；霸王餐处理 `dashBalance`，并附带一次性参与次数和整额核销限制

### 3.2 需要配置的规则内容

商户配置霸王餐时，核心是定义：

- 哪些会员卡可参加
- 哪些门店可参加
- 消费金额在哪个区间内可触发
- 触发后赠送几倍
- 活动有效期
- 是否限制“同一张卡只能参加一次”

主要配置字段见：

- `CrmCardLevelAndOverlordMealAddDTO`
- `CrmCardLevelAndOverlordMealUpdateDTO`
- `CrmCardLevelAndOverlordMeal`

核心字段包括：

- `name`：霸王餐名称
- `cardType` / `cardTypeCode`：会员卡类型
- `cardTypeLevel` / `cardTypeLevelCode`：会员卡等级
- `minAmount` / `maxAmount`：可触发的消费金额区间
- `rechargeMultipleTimes`：充值倍数
- `validBeginTime` / `validEndTime`：有效期
- `remarks`：前端描述文案
- `once`：是否仅限参与一次
- `overlordStoreList`：适用门店列表

门店关联单独落表：

- 表实体：`CrmOverlordStore`
- 关键字段：`overlordLid`、`sid`

## 4. 会员侧如何使用

### 4.1 使用前提

会员侧要触发霸王餐，需要同时满足：

1. 当前用户已开通会员并持有会员卡。
2. 当前账单金额大于 0。
3. 当前门店在霸王餐适用门店内。
4. 当前会员卡类型/等级命中霸王餐规则。
5. 当前日期在活动有效期内。
6. 消费金额落在规则的 `minAmount ~ maxAmount` 之间。
7. 如果规则设置了 `once=true`，则该卡之前不能参加过该规则。
8. 如果是线下自动结账场景，该订单此前不能已经做过霸王餐充值。

### 4.3 小程序 API 接口层

小程序所有霸王餐相关的网络请求都封装在：

```typescript
// GET /SCRM/crm_card_op/customer/get_overlord_rules
export const get_overlord_rules = (data?: any): Promise<Response<any>> => {
  return request({
    url: `${SCRM}/crm_card_op/customer/get_overlord_rules`,

  })

```

#### 提交霸王餐充值（`ruleType = -2`）
```typescript

export const charge_commit = (data?: any): Promise<Response<any>> => {
  return request({
    url: `${SCRM}/charge/commit`,
    data,
  })

```

#### 通知充值成功（后端才真正执行入账）
```typescript
// POST /SCRM/charge/success
export const charge_success = (data?: any): Promise<Response<any>> => {
  return request({
    url: `${SCRM}/charge/success`,
    data,
  })
}
```

**关键语义**：
- `ruleType = -2`：前后端约定的霸王餐充值标识，`ChargeService.charge()` 根据此值走霸王餐专属分支
- `get_overlord_rules`：只负责”查询当前是否可以触发霸王餐”，不涉及余额操作
- `charge_commit`：提交充值预下单，唤起微信支付
- `charge_success`：支付完成后通知后端，后端才真正执行 `dashBalance` 入账

## 5. 后端命中规则的逻辑

核心方法：

- `CrmCardOpServicePlus#getOverlordRules`

命中步骤：

1. 校验 `billAmount > 0`
2. 校验 `sid` 存在
3. 如果没有 `cardLid`，直接返回空
4. 如果 `notCheckOut = true`，则检查订单是否已做过霸王餐
5. 读取会员卡信息，拿到卡类型与卡等级
6. 在 `crm_card_level_and_overlord_meal` 中查符合条件的规则
7. 在 `crm_overlord_store` 中校验当前门店是否适用
8. 如果 `once = true`，则去 `crm_overlord_once` 校验是否参加过
9. 构造一个虚拟的 `CrmSaveRule` 返回给前端/充值服务

构造出来的关键值：

- `lid = -2`
- `saveAmount = billAmount * rechargeMultipleTimes`
- `giveAmount = billAmount`
- `giveCouponNum = rechargeMultipleTimes`
- `ruleTypeLid = overlordMeal.lid`
- `description = overlordMeal.remarks`

这里要特别注意：

- `lid = -2` 是“霸王餐充值类型”的系统标识
- `ruleTypeLid` 才是真正命中的霸王餐规则 ID

## 6. 充值与入账链路

核心服务：

- `ChargeService`
- `CardBalanceService`

### 6.1 提交充值

当前端传入 `ruleType = -2` 时：

1. 后端要求 `billAmount` 不能为空
2. 后端要求 `orderId` 不能为空
3. 如果 `notCheckOut = true`，则还要求 `billId` 不能为空
4. 再次调用 `getOverlordRules` 获取当前可用霸王餐
5. 若无命中规则，报“没有找到对应的霸王餐储值”

### 6.2 创建充值任务

充值任务创建时：

- `CrmDealTask.saveRuleCode = -2`
- `CrmDealTask.dashAmount = 霸王餐赠送金额`
- `CrmDealTaskItem.saveRuleCode = -2`
- `CrmDealTaskItem.ruleTypeLid = 真实霸王餐规则 lid`

这意味着：

- 主任务层记录“这是霸王餐充值”
- 子项层既记录“这是霸王餐充值”，也记录“具体命中了哪条霸王餐规则”

### 6.3 余额入账

会员卡入账时，霸王餐金额不会走普通赠送余额，而是进入：

- `dashBalance`

对应逻辑在 `CardBalanceService` 中：

- `dash_balance = dash_balance + dashAmount`

会员卡/会员查询对象里也暴露了这个字段：

- `CrmCardVO.dashBalance`
- `CrmMemberVO.dashBalance`

## 7. 霸王餐余额怎么使用

霸王餐余额有严格限制，不是普通余额：

1. 只要会员卡上存在 `dashBalance > 0`
2. 本次会员卡支付金额必须和 `dashBalance` 完全相等
3. 不允许部分使用，不允许和普通余额随意混用

后端逻辑会强制：

- `task.setGiveAmount(amount)`
- `task.setPrincipalAmount(BigDecimal.ZERO)`
- `task.setDashAmount(amount)`

也就是说，霸王餐余额存在时，本次会员卡支付必须走霸王餐赠送余额整笔核销。

### 10.3 `nms4pos`（POS 后端）

状态：**无法触发霸王餐，间接路径也不存在**

POS 后端位于 `nms4pos/ms4cloud-pos2plugin`，是 POS 前端（`nms4pos-ui`）调用的后端服务，通过 Feign 接口与 CRM 后端通信。

#### 结论：POS 后端无论直接还是间接，都无法触发霸王餐

证据：

**1. POS 后端 Feign 客户端没有霸王餐相关接口**

`Nms4CloudCrmService.java`（POS 调用 CRM 的 Feign 接口定义）中，**没有任何霸王餐相关方法**：

```java
// POS 后端 Feign 客户端有的方法：
✅ memberCheck          — 会员信息校验
✅ cardConsume         — 会员卡消费
✅ cardRepay           — 会员卡回款
✅ cardCharge          — 会员卡充值（普通充值）
✅ revokeConsume       — 消费退款
✅ adjustPoints        — 积分调整

❌ getOverlordRules   — 查询霸王餐规则（不存在）
❌ chargeCommit       — 充值提交（不存在）
```

**结论**：POS 后端**没有**调用 `getOverlordRules` 的路径，无法查询霸王餐规则。

**2. POS 后端 `cardCharge` 走普通充值链路，不走霸王餐链路**

POS 后端有 `/cardCharge` 接口（`MemberForBizController.java:573`），最终调用链：

```
POS 前端调用 /cardCharge
  → MemberForBizController.cardCharge()
  → OrderServiceUtil.dealCharge()
  → nms4CloudCrmService.cardCharge()   ← Feign 调用 CRM
  → CrmCardOpServicePlus.cardCharge()  ← CRM 后端
```

CRM 后端 `CrmCardOpServicePlus.cardCharge()` 的逻辑：

```java
// CrmCardOpServicePlus.java:1526-1532
.setCardLid(request.getLid())
.setSaveRuleCode(request.getRuleLid())  // ← ruleLid 被传入
```

CRM 后端确实接受 `ruleLid` 参数，但**不走霸王餐的 `ruleType = -2` 分支**。

霸王餐的正确触发路径是：

```
ruleType = -2
→ ChargeService.charge() 的霸王餐专属分支
→ 创建 dashAmount 入 dash_balance
```

而 POS 的 `cardCharge` → `CrmCardOpServicePlus.cardCharge()` → `crtTask()` 创建的是**普通充值任务**，`dashAmount` 不会被设置，`dash_balance` 不会增加。

**3. POS 后端 `dealCharge` 没有传入 `ruleLid`**

`OrderServiceUtil.dealCharge()`（第 3328–3351 行）调用 `cardCharge` 时：

```java

chargeDTO.setPrincipalAmount(dwdBill.getPaidAmount());  // 实付金额
chargeDTO.setGiveAmount(dwdBill.getSendAmount());      // 赠送金额
// ⚠️ 没有调用 setRuleLid() → ruleLid 为 null
```

POS 的充值单处理根本不传 `ruleLid`，即使传了也不走霸王餐分支。

**4. 全仓库关键词检索为空**

| `ruleType.*-2` | 空 |

**结论**：POS 后端（`nms4pos/ms4cloud-pos2plugin`）**无法直接或间接触发霸王餐功能**。

#### POS 后端被动参与霸王餐消费核销

POS 后端虽然不能触发霸王餐充值，但通过 `cardConsume` 接口**被动参与**霸王餐消费核销：

```
taro-mall 会员小程序（唯一触发入口）：
  用户消费 100 元 → charge_commit (ruleType=-2)
  → CRM 写入 dashBalance = 100
  → taromall 用 dashBalance 支付 200 完成本单

POS 收银台（被动核销）：
  会员卡有 dashBalance > 0
  → POS 调用 cardConsume
  → CRM CardBalanceService.checkBalance() 检查到 dashBalance > 0
  → 要求本次支付金额 == dashBalance，否则抛异常
```

| 维度 | POS 后端 (`nms4pos`) |
|------|----------------------|
| 能查询霸王餐规则 | ❌ Feign 客户端没有 `getOverlordRules` |
| 能触发霸王餐充值 | ❌ 没有 `chargeCommit`/`ruleType=-2` 路径 |
| `cardCharge` 是否霸王餐 | ❌ 走普通充值链路，不走 `ruleType=-2` |
| 被动参与消费核销 | ✅ `cardConsume` → CRM 后端强制校验 `dashBalance` |

- 支付成功后通知 CRM 后端收尾，由后端继续处理余额入账与后续自动结账链路

### 13.3 推荐本地测试入口

本地测试霸王餐时，建议优先验证以下 3 个入口：

#### 入口 1：`pagePop/paycenter/index`

这是当前更成熟的支付中心入口。

证据：

- `src/app.config.ts` 已注册 `pagePop/paycenter/index`
- 页面内包含完整的霸王餐查询、插入 `BWC`、发起 `charge_commit`、触发 `charge_success` 逻辑

#### 入口 2：`pagePop/paymentcenter/index`

这是较新的支付中心入口，也已接入霸王餐，但源码注释写明“（新）支付中心还没弄完”，因此应作为补充验证入口。

证据：

- `src/app.config.ts` 注释：`paymentcenter/index // （新）支付中心还没弄完`

#### 入口 3：点餐页 `PullSingle`

点餐页会展示“霸王餐 / 马上充值”卡片，适合验证 `notCheckOut = true` 的自动结账预处理分支。

证据：

- `src/pages/Order/components/PullSingle/index.tsx`

### 13.4 支付中心本地测试步骤

#### 第一步：确认是否显示霸王餐入口

进入 `paycenter/index` 或 `paymentcenter/index` 后，页面加载时会先请求：

- `POST /scrm/crm_card_op/customer/get_overlord_rules`

请求字段：

| 字段 | 中文名 | 含义 |
|---|---|---|
| `sid` | 门店编号 | 当前订单所属门店 |
| `billAmount` | 账单金额 | 当前待支付金额 |

展示条件：

- 当前用户必须有 `cardLid`
- 接口返回 `data.length > 0`
- 支付顺序配置 `g_paytypeorder` 里必须包含 `0`

满足后，前端会把一项支付方式插入到列表：

- `label = 霸王餐`
- `value = BWC`
- `image = BWC.png`

并展示描述文案：

- `overlordList?.[0]?.description`

直接证据：

- `paymentcenter/index.tsx:225-266`
- `paycenter/index.tsx:229-266`

#### 第二步：点击霸王餐支付方式

当 `curPay === 'BWC'` 时，页面不会走普通订单支付 `payOrder(...)`，而是跳到：

- `handleRecharge()`

随后发起请求：

- `POST /scrm/charge/commit`

关键请求字段：

| 字段 | 中文名 | 含义 |
|---|---|---|
| `sid` | 门店编号 | 当前门店 |
| `billAmount` | 账单金额 | 当前待支付金额 |
| `ruleType = -2` | 储值类型 | 表示霸王餐充值 |
| `orderId` | 订单编号 | 当前订单号 `saasOrderKey` |
| `h5` | 是否 H5 | 决定支付调用方式 |

测试时要确认：

- Network 中确实发起了 `/scrm/charge/commit`
- 请求体中的 `ruleType` 是否为 `-2`
- 返回体中是否包含支付参数和 `taskLid`

直接证据：

- `paymentcenter/index.tsx:434-524`
- `paycenter/index.tsx:447-524`

#### 第三步：支付成功后是否通知后端

微信支付成功后，前端会调用：

- `POST /scrm/charge/success`

关键请求字段：

| 字段 | 中文名 | 含义 |
|---|---|---|
| `sid` | 门店编号 | 当前门店 |
| `terminalTrace` | 终端流水号 | 这里传 `taskLid` |
| `needComment` | 是否需要评论配置 | 是否获取评价跳转配置 |

测试时要确认：

- Network 中是否发起了 `/scrm/charge/success`
- 请求体里的 `terminalTrace` 是否等于 `/charge/commit` 返回的 `taskLid`

直接证据：

- `paymentcenter/index.tsx:483-502`
- `paycenter/index.tsx:495-514`

### 13.5 点餐页 `PullSingle` 本地测试步骤

点餐页的霸王餐不是支付方式列表，而是一张营销卡片。

#### 第一步：确认是否显示霸王餐卡片

页面进入后会调用：

- `POST /scrm/crm_card_op/customer/get_overlord_rules`

关键请求字段：

| 字段 | 中文名 | 含义 |
|---|---|---|
| `sid` | 门店编号 | 当前订单门店 |
| `billAmount` | 账单金额 | `bill.order.paidAmount` |
| `orderId` | 订单号 | 这里传 `bill.order.orgBillId` |
| `notCheckOut = true` | 是否自动结账 | 标识线下自动结账场景 |

如果返回 `data.length > 0`，页面就显示：

- 标签：`霸王餐`
- 文案：`充值消费{giveCouponNum}倍（¥{saveAmount}）立享免单!`
- 按钮：`马上充值`

直接证据：

- `PullSingle/index.tsx:217-241`
- `PullSingle/index.tsx:787-810`

#### 第二步：点击“马上充值”

点击后会发起：

- `POST /scrm/charge/commit`

关键请求字段：

| 字段 | 中文名 | 含义 |
|---|---|---|
| `sid` | 门店编号 | 当前门店 |
| `ruleType = -2` | 储值类型 | 表示霸王餐充值 |
| `billAmount` | 账单金额 | 当前订单金额 |
| `orderId` | 订单编号 | `bill.order.saasOrderKey` |
| `billId` | 线下订单编号 | `bill.order.orgBillId` |
| `notCheckOut = true` | 是否自动结账 | 表示走自动结账预处理分支 |

测试时要特别确认：

- `ruleType` 是否等于 `-2`
- `billId` 是否已传

这是点餐页路径与支付中心路径最大的差异。

直接证据：

#### 第三步：支付成功后的前端表现

随后：

- 页面会等待约 2 秒
- 提示“充值成功”
- 调 `onMemberInfoRefresh()` 刷新会员信息

测试时应重点观察：

- 是否真正发起了 `/scrm/charge/success`
- 支付成功后会员信息是否刷新

直接证据：

- `PullSingle/index.tsx:384-404`

### 13.6 本地测试时的验收标准

#### 正向验收

满足以下条件时，可认为前端霸王餐链路通过基础验证：

1. 会员已登录，且 `cardLid` 存在
2. 支付中心能看到 `BWC / 霸王餐` 支付方式
3. 点餐页能看到“霸王餐 / 马上充值”卡片
4. 点击后会发起 `POST /scrm/charge/commit`
5. `charge/commit` 中 `ruleType = -2`
6. 支付成功后会发起 `POST /scrm/charge/success`
7. 支付中心页面能继续进入 `handleSuccess(...)`
8. 点餐页支付成功后能看到会员信息刷新

#### 负向验收

以下情况不应显示霸王餐入口：

1. 当前用户未开通会员，没有 `cardLid`
2. 后端接口 `/get_overlord_rules` 返回空数组
3. `g_paytypeorder` 不包含 `0`
4. 当前账单金额不满足霸王餐规则区间

### 13.7 本地测试时的注意事项

#### 1. `pointsmallpaycenter` 当前不适合用来测试霸王餐

虽然页面保留了 `BWC` 的展示代码和 `charge_commit` 代码，但查询霸王餐列表和插入 `BWC` 的关键逻辑已被注释。

因此当前不应把它当作主测试入口。

直接证据：

- `pointsmallpaycenter/index.tsx:189-206`

#### 2. 支付中心和点餐页测试目标不同

- `paycenter/paymentcenter` 更适合验证“霸王餐作为支付方式如何展示、如何发起充值、支付成功后如何继续结账”
- `PullSingle` 更适合验证“霸王餐营销卡片如何展示、如何带 `billId + notCheckOut = true` 发起充值”

#### 3. 本地前端测试不能直接证明后端已经完成自动结账

前端本地能验证的是：

- 是否发起了正确请求
- 请求参数是否正确
- 页面是否按预期展示和跳转

而后端的 MQ、余额入账、自动关单，仍需要配合服务端日志或接口结果综合确认。
