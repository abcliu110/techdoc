# 霸王餐功能分析

## 概述

**霸王餐**是 nms4cloud CRM 系统中的一个营销功能，允许商户为特定会员卡等级提供"免费赠送"的储值优惠。会员在消费时，如果账单金额符合霸王餐规则，系统会自动赠送相应的储值金额和优惠券。

---

## 核心概念

### 1. 霸王餐的本质

霸王餐是一种**条件性储值赠送**机制：
- **触发条件**：会员在消费时，账单金额落在霸王餐规则的金额范围内
- **赠送内容**：
  - 赠送储值金额（账单金额 × 充值倍数）
  - 赠送优惠券（数量 = 充值倍数）
- **使用限制**：赠送的储值金额只能在当前订单中使用（称为"当餐限制"）

### 2. 关键术语

| 术语 | 说明 |
|------|------|
| **充值倍数** | 账单金额乘以该倍数 = 赠送的储值金额 |
| **金额范围** | 霸王餐适用的账单金额范围（最小金额 ~ 最大金额） |
| **一次性** | 如果启用，同一会员卡对同一霸王餐规则只能参与一次 |
| **赠送余额** | 通过霸王餐赠送的储值金额，在数据库中用 `dashBalance` 字段表示 |
| **当餐限制** | 赠送的储值金额只能在当前订单中消费，不能跨订单使用 |

## 数据模型

### 1. 核心表结构

#### `crm_card_level_and_overlord_meal`（霸王餐规则表）

```sql
CREATE TABLE crm_card_level_and_overlord_meal (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT,
  mid BIGINT,                          -- 商户ID
  sid BIGINT,                          -- 门店ID
  lmnid BIGINT,                        -- 逻辑ID（唯一标识）
  name VARCHAR(255),                   -- 霸王餐名称
  status_ INT,                         -- 状态（0=禁用, 1=启用）
  card_type VARCHAR(255),              -- 会员卡类型
  card_type_code VARCHAR(255),         -- 会员卡类型代码
  card_type_level VARCHAR(255),        -- 会员卡等级
  card_type_level_code VARCHAR(255),   -- 会员卡等级代码
  min_amount DECIMAL(10,2),            -- 最小账单金额
  max_amount DECIMAL(10,2),            -- 最大账单金额
  recharge_multiple_times INT,         -- 充值倍数
  valid_begin_time DATETIME,           -- 有效期开始
  valid_end_time DATETIME,             -- 有效期结束
  remarks TEXT,                        -- 备注
  once BOOLEAN,                        -- 是否一次性（true=只能参与一次）
  created_time DATETIME,
  updated_time DATETIME,
  deleted INT
);
```

**关键字段说明**：
- `card_type_code` 和 `card_type_level_code`：可为空或空字符串，表示对所有卡类型/等级适用
- `min_amount` 和 `max_amount`：定义账单金额范围
- `recharge_multiple_times`：充值倍数，例如 2 表示账单金额的 2 倍
- `once`：如果为 true，同一会员卡只能参与一次

#### `crm_overlord_store`（霸王餐门店关联表）

```sql
CREATE TABLE crm_overlord_store (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT,
  mid BIGINT,                          -- 商户ID
  sid BIGINT,                          -- 门店ID
  lid BIGINT,                          -- 逻辑ID
  overlord_lid BIGINT,                 -- 霸王餐规则ID（关联 crm_card_level_and_overlord_meal.lmnid）
  created_time DATETIME,
  updated_time DATETIME,
  deleted INT
);
```

**说明**：定义霸王餐规则在哪些门店生效。

#### `crm_overlord_once`（霸王餐参与记录表）

```sql
CREATE TABLE crm_overlord_once (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT,
  mid BIGINT,                          -- 商户ID
  sid BIGINT,                          -- 门店ID
  lid BIGINT,                          -- 逻辑ID
  times INT,                           -- 剩余参与次数
  card_lid BIGINT,                     -- 会员卡ID
  overlord_lid BIGINT,                 -- 霸王餐规则ID
  type_ VARCHAR(50),                   -- 类型（OVERLORD=霸王餐, PETTY=小额储值）
  created_time DATETIME,
  updated_time DATETIME,
  deleted INT
);
```

**说明**：记录会员卡对霸王餐规则的参与情况（用于"一次性"限制）。

## 业务流程

### 1. 霸王餐查询流程

```
用户消费 (账单金额 = 100 元)
    ↓
调用 getOverlordRules(billAmount=100, cardLid=xxx, sid=yyy)
    ↓
[1] 查询会员卡信息 → 获取卡类型和等级
    ↓
[2] 查询霸王餐规则
    条件：
    - 商户ID 匹配
    - 卡类型/等级 匹配（或为空表示全部）
    - 账单金额 在 [min_amount, max_amount] 范围内
    - 当前日期 在 [valid_begin_time, valid_end_time] 范围内
    ↓
[3] 检查门店是否适用
    查询 crm_overlord_store，确认该门店有该霸王餐规则
    ↓
[4] 如果启用"一次性"，检查会员卡是否已参与过
    查询 crm_overlord_once，检查 times > 0
    ↓
[5] 构建虚拟储值规则
    rule.lid = -2L  // 用 -2 代表霸王餐
    rule.saveAmount = 100 × 2 = 200 元  // 账单金额 × 充值倍数
    rule.giveAmount = 100 元  // 赠送金额 = 账单金额
    rule.giveCouponNum = 2  // 赠送优惠券数 = 充值倍数
    ↓
返回规则列表
```

### 2. 霸王餐消费流程

```
用户确认支付
    ↓
调用 ChargeService.charge(ruleType=-2, billAmount=100, ...)
    ↓
[1] 验证参数
    - billAmount 不能为空
    - orderId 不能为空
    ↓
[2] 检查订单是否已充值过霸王餐
    Redis key: "CRM_PAY_ORDER_NOT_CHECKOUT:{mid}:{sid}"
    如果已存在，抛出异常："每个订单霸王餐只能充值一次"
    ↓
[3] 调用 getOverlordRules() 获取适用规则
    ↓
[4] 执行储值逻辑
    - 创建储值任务（CrmDealTask）
    - 赠送金额记录到 dashBalance（赠送余额）
    - 赠送优惠券
    ↓
[5] 标记订单已充值
    Redis 中记录该订单已参与霸王餐
    ↓
[6] 如果启用"一次性"，更新参与次数
    UPDATE crm_overlord_once SET times = times - 1
    WHERE card_lid = xxx AND overlord_lid = yyy
```

### 3. 赠送余额使用流程

```
会员使用赠送余额支付
    ↓
检查 card.dashBalance > 0
    ↓
如果有赠送余额，必须全部使用
    条件：支付金额 == dashBalance
    否则抛出异常："本次会员卡支付只能使用霸王餐赠送余额【xxx】"
    ↓
执行支付
    - 从 dashBalance 中扣除
    - 记录交易类型为 OVERLORD（霸王餐）
    ↓
支付完成后，更新参与次数
    UPDATE crm_overlord_once SET times = times - 1
```

## 关键代码实现

### 1. 查询霸王餐规则

**文件**：`CrmCardOpServicePlus.java`

```java
public List<CrmSaveRule> getOverlordRules(MemberOverlordRuleGetDTO request) {
    // 1. 验证参数
    BigDecimal billAmount = request.getBillAmount();
    Assert.isTrue(billAmount.compareTo(BigDecimal.ZERO) > 0, "金额必须大于0");

    // 2. 检查订单是否已充值过
    if (Boolean.TRUE.equals(request.getNotCheckOut())) {
        String keyOfNotCheckOut = RedisConstants.getKeyOfNotCheckOut(request.getMid(), request.getSid());
        if (redisTemplatePlus.hasHashKey(keyOfNotCheckOut, request.getOrderId())) {
            return Collections.emptyList();  // 已充值过，返回空
        }
    }

    // 3. 查询会员卡
    CrmCard card = getOne(request.getMid(), request.getCardLid());

    // 4. 查询适用的霸王餐规则
    LambdaQueryWrapper<CrmCardLevelAndOverlordMeal> query =
        new LambdaQueryWrapper<CrmCardLevelAndOverlordMeal>()
            .eq(CrmCardLevelAndOverlordMeal::getMid, request.getMid())
            // 卡类型/等级匹配（支持通配符）
            .and(i -> i.eq(CrmCardLevelAndOverlordMeal::getCardTypeLevelCode, card.getCardTypeLevelCode())
                      .or().isNull(CrmCardLevelAndOverlordMeal::getCardTypeLevelCode)
                      .or().eq(CrmCardLevelAndOverlordMeal::getCardTypeLevelCode, ""))
            // 金额范围匹配
            .le(CrmCardLevelAndOverlordMeal::getMinAmount, billAmount)
            .ge(CrmCardLevelAndOverlordMeal::getMaxAmount, billAmount)
            // 日期范围匹配
            .le(CrmCardLevelAndOverlordMeal::getValidBeginTime, LocalDate.now())
            .ge(CrmCardLevelAndOverlordMeal::getValidEndTime, LocalDate.now())
            .orderByDesc(CrmCardLevelAndOverlordMeal::getValidBeginTime);

    CrmCardLevelAndOverlordMeal overlordMeal = iCrmCardLevelAndOverlordMealService.getOne(query, false);
    if (Objects.isNull(overlordMeal)) {
        return Collections.emptyList();
    }

    // 5. 检查门店是否适用
    var storeQueryWrapper = new QueryWrapper<CrmOverlordStore>()
        .lambda()
        .eq(CrmOverlordStore::getMid, overlordMeal.getMid())
        .eq(CrmOverlordStore::getSid, request.getSid())
        .eq(CrmOverlordStore::getOverlordLid, overlordMeal.getLid());

    if (iCrmOverlordStoreService.count(storeQueryWrapper) < 1) {
        return Collections.emptyList();
    }

    // 6. 检查"一次性"限制
    Boolean once = overlordMeal.getOnce();
    if (Boolean.TRUE.equals(once)) {
        if (iCrmOverlordOnceService.count(
            new LambdaQueryWrapper<CrmOverlordOnce>()
                .eq(CrmOverlordOnce::getMid, request.getMid())
                .eq(CrmOverlordOnce::getCardLid, request.getCardLid())
                .eq(CrmOverlordOnce::getOverlordLid, overlordMeal.getLid())
                .eq(CrmOverlordOnce::getType, OverlordOnceTypeEnum.OVERLORD)
                .gt(CrmOverlordOnce::getTimes, 0)) > 0) {
            return Collections.emptyList();
        }
    }

    // 7. 构建虚拟储值规则
    CrmSaveRule rule = new CrmSaveRule();
    rule.setLid(-2L);  // 用 -2 代表霸王餐
    rule.setSaveAmount(NumberUtil.mul(billAmount, overlordMeal.getRechargeMultipleTimes()));
    rule.setGiveAmount(billAmount);
    rule.setGiveCouponNum(overlordMeal.getRechargeMultipleTimes());
    rule.setName("霸王餐储值");
    rule.setRuleTypeLid(overlordMeal.getLid());

    return Collections.singletonList(rule);
}
```

### 2. 赠送余额使用限制

**文件**：`CardBalanceService.java`

```java
// 检查是否有霸王餐赠送余额，如果有霸王餐余额，必须使用霸王餐赠送余额
BigDecimal dashBalance = card.getDashBalance();
if (NumberUtilPlus.gtZero(dashBalance)) {
    BigDecimal amount = task.getAmount();
    if (amount.compareTo(dashBalance) != 0) {
        throw new BizException(
            String.format("本次会员卡支付只能使用霸王餐赠送余额【%s】",
                NullSafeUtils.formatZero(dashBalance)));
    }
    task.setGiveAmount(amount);
}
```

## 数据流向示例

### 场景：会员消费 100 元，霸王餐规则为 2 倍

```
消费前：
  会员卡余额 = 500 元
  赠送余额 = 0 元

消费时触发霸王餐：
  账单金额 = 100 元
  霸王餐倍数 = 2

系统计算：
  saveAmount = 100 × 2 = 200 元
  dashAmount = giveAmount = 100 元
  赠送优惠券 = 2 张

消费后：
  会员卡余额 = 500 元（未变）
  霸王餐专项余额 dashBalance = 100 元（新增）

会员使用赠送余额：
  必须支付 200 元（全部使用赠送余额）
  不能支付 100 元（部分使用）

使用后：
  会员卡余额 = 500 元
  赠送余额 = 0 元
```

## 关键业务规则

| 规则 | 说明 |
|------|------|
| **每订单一次** | 同一订单只能充值一次霸王餐 |
| **赠送余额全用** | 如果有赠送余额，必须全部使用，不能部分使用 |
| **当餐限制** | 赠送的储值金额只能在当前订单中使用 |
| **一次性限制** | 如果启用，同一会员卡对同一霸王餐规则只能参与一次 |
| **金额范围** | 账单金额必须在 [min_amount, max_amount] 范围内 |
| **日期范围** | 当前日期必须在 [valid_begin_time, valid_end_time] 范围内 |
| **门店限制** | 霸王餐规则必须在该门店启用 |
| **卡类型匹配** | 会员卡类型/等级必须匹配霸王餐规则（或规则为空表示全部） |

## POS 前端问题发现

### ⚠️ 重要问题：POS 结账页面未展示霸王餐余额

**问题描述**：`CheckOut/index.tsx`（POS 结账核心组件）只读取 `memberInfo.balance`，完全不使用 `dashBalance` 字段。

**影响**：收银员无法从 POS 界面看出会员卡是否有霸王餐赠送余额（`dashBalance`），只能看到普通余额（`balance`）。当会员卡有霸王餐余额时，CRM 后端强制要求"本次会员卡支付只能使用霸王餐赠送余额"，但 POS 收银员对此毫无感知，容易导致消费失败时找不到原因。

**证据 1：`dashBalance` 在类型定义中存在，但 POS 未使用**

- **类型定义**：`nms4pos-ui/api/pos4plugin/src/typings/ICrmCard.ts` 第 57–59 行
  ```typescript
  invoiceBalance?: string;
  /** 霸王餐余额 */
  dashBalance: number;
  /** 未到账赠送金额（冻结金额） */
  unpaidGiveAmount: number;
  ```

- **POS 结账页面使用 `balance` 而非 `dashBalance`**：`nms4pos-ui/app/pos4desktop/src/components/CheckOut/index.tsx`
  ```typescript
  // 第 716–717 行：只读 balance
  const cardBalanceBefore = !isEmpty(memberInfo)
    ? parseFloat(formatNumber(memberInfo?.balance))
    : parseFloat(...);

  // 第 1460–1465 行：会员选择时只传 balance
  if (!memberInfo?.balance || parseFloat(memberInfo?.balance) == 0) {
    message.warning('会员余额不足');
    return;
  }
  setSelectedMember(memberInfo);
  updatePaymentTypeList(parseFloat(memberInfo?.balance || '0'), {...memberInfo,...});

  // 全文件 grep 结果：完全没有 `dashBalance` 或 `dash_balance` 的引用
  ```

- **CRM 后端强制校验**：`CardBalanceService.java` 第 1033–1043 行
  ```java
  BigDecimal dashBalance = card.getDashBalance();
  if (NumberUtilPlus.gtZero(dashBalance)) {
      BigDecimal amount = task.getAmount();
      if (amount.compareTo(dashBalance) != 0) {
          throw new BizException(
              String.format("本次会员卡支付只能使用霸王餐赠送余额【%s】",
                  NullSafeUtils.formatZero(dashBalance)));
      }
      task.setGiveAmount(amount);
      task.setPrincipalAmount(BigDecimal.ZERO);
      task.setDashAmount(amount);
      return null;
  }
  ```

**证据总结**：

| 位置 | 文件 | 行号 | 内容 |
|------|------|------|------|
| 类型定义 | `ICrmCard.ts` | 59 | `dashBalance: number` ✅ 定义存在 |
| POS 读余额 | `CheckOut/index.tsx` | 717 | `memberInfo?.balance` ✅ 只读 balance |
| POS 选会员 | `CheckOut/index.tsx` | 1460, 1465 | `memberInfo?.balance` ✅ 只传 balance |
| POS 全文件 | `CheckOut/index.tsx` | — | 无 `dashBalance` 引用 ❌ |
| 后端校验 | `CardBalanceService.java` | 1038 | `"本次会员卡支付只能使用霸王餐赠送余额【%s】"` ✅ 强制校验 |

**结论**：CRM 后端在 `dashBalance > 0` 时强制要求整额使用，但 POS 前端对 `dashBalance` 字段完全无感知——既不展示，也不传给收银员，导致收银员无法预判消费限制。

**建议**：POS 前端应展示 `dashBalance`，或在会员卡有霸王餐余额时给予明确提示。

### 6.1 推荐调试时序

如果你要一次把霸王餐主流程看完，建议按下面顺序下断点，不要跳着看：

```text
1. CardOpForCustomerController.getOverlordRules(...)
   -> CrmCardOpServicePlus.getOverlordRules(...)
   目的：确认前端为什么能看到/看不到霸王餐选项

2. CrmChargeController.commit(...)
   -> ChargeService.commit(...)
   目的：确认提交时是否真的进入 ruleType = -2 的霸王餐分支

3. ChargeService.createDealTask(...)
   目的：确认是否创建 crm_deal_task / crm_deal_task_item，
        以及 dashAmount 到底写成多少

4. CrmOrderCloseQueueConsumer.onMessage(...)
   目的：确认支付成功后是否真的进入充值入账链路

5. CardBalanceService.changeCardBalance(...)
   目的：确认 crm_card.dash_balance 是否真正增加

6. CardBalanceService.buildRecord(...)
   目的：确认是否生成 crm_card_record 流水

7. CardBalanceService.dealOtherItemTask(...)
   目的：确认 crm_overlord_once 是否新增/累加

8. CardBalanceService.checkBalance(...)
   目的：确认会员消费时为什么必须整额使用霸王餐余额
```

### 6.2 调试时怎么判断卡在哪一阶段

```text
能查到霸王餐规则，但没进 ChargeService.commit
=> 只是前端展示了霸王餐选项，还没真正发起充值

进了 createDealTask，但 crm_deal_task.trade_state 还是 USERPAYING
=> 任务已创建，正在等支付成功回调

支付成功后进了 CrmOrderCloseQueueConsumer，但没进 changeCardBalance
=> MQ 收到了消息，但余额入账链路中断

changeCardBalance 执行了，但 crm_card_record 没有数据
=> 要继续看 buildRecord 或后续 saveBatch/save 是否失败

crm_card.dash_balance 已增加，但消费时报"只能使用霸王餐赠送余额"
=> 去看 CardBalanceService.checkBalance(...) 的整额限制判断
```

### 6.3 实际请求入口与关键参数

霸王餐主流程至少有两个直接入口，一个是"查询是否有霸王餐规则"，一个是"确认发起霸王餐储值"。

#### 入口一：查询霸王餐规则

Controller:

```text
POST /crm_card_op/customer/get_overlord_rules
-> CardOpForCustomerController.getOverlordRules(...)
-> CrmCardOpServicePlus.getOverlordRules(...)
```

关键请求字段：

| 字段 | 说明 |
|---|---|
| `mid` | 商户 ID |
| `sid` | 门店 ID |
| `cardLid` | 会员卡 lid |
| `billAmount` | 账单金额 |
| `orderId` | 订单号，可选 |
| `notCheckOut` | 是否未结账单 |

示例：

```json
{
  "mid": 10001,
  "sid": 20001,
  "cardLid": 30001,
  "billAmount": 100.00,
  "orderId": "ORD202604270001",
  "notCheckOut": true
}
```

这一步的目标不是创建储值记录，而是确认当前卡、当前门店、当前账单金额是否能命中霸王餐规则。

#### 入口二：提交霸王餐储值

Controller:

```text
POST /charge/commit
-> CrmChargeController.commit(...)
-> ChargeService.commit(...)
```

关键请求字段：

| 字段 | 说明 |
|---|---|
| `mid` | 商户 ID |
| `sid` | 门店 ID |
| `cardLid` | 会员卡 lid |
| `ruleType` | 霸王餐固定传 `-2` |
| `billAmount` | 账单金额 |
| `orderId` | 订单号，霸王餐必填 |
| `billId` | 线下账单号，`notCheckOut=true` 时必看 |
| `notCheckOut` | 是否未结账单 |
| `h5` | 是否 H5 支付 |
| `appId` | 小程序/H5 AppId，可由后端补 |

示例：

```json
{
  "mid": 10001,
  "sid": 20001,
  "cardLid": 30001,
  "ruleType": -2,
  "billAmount": 100.00,
  "orderId": "ORD202604270001",
  "billId": "BILL202604270001",
  "notCheckOut": true,
  "h5": false
}
```

这一步才会真正创建：

```text
crm_deal_task
crm_deal_task_item
```

### 6.4 请求和 SQL 观察点对照

```text
请求 /crm_card_op/customer/get_overlord_rules
=> 重点看 crm_card_level_and_overlord_meal / crm_overlord_store / crm_overlord_once

请求 /charge/commit
=> 重点看 crm_deal_task / crm_deal_task_item

支付成功后 MQ 回调
=> 重点看 crm_card / crm_card_record / crm_overlord_once

会员卡消费
=> 重点看 crm_card.dash_balance 和 CardBalanceService.checkBalance(...)
```

建议按这个顺序观察数据库：

1. 先查霸王餐规则是否命中
2. 再查是否创建了 `crm_deal_task`
3. 再查是否生成了 `crm_card_record`
4. 最后查 `crm_card.dash_balance` 是否变化
5. 若涉及一次性限制，再查 `crm_overlord_once.times`

### 7. 关键判断

如果只有：

```text
crm_deal_task
crm_deal_task_item
```

但没有：

```text
crm_card_record
crm_card.dash_balance 增加
```

通常说明支付任务已创建，但还没有完成支付成功后的余额入账。

如果 `crm_overlord_once` 有记录，也不能说明已入账。它只说明该卡对某个霸王餐规则有参与次数记录，真正的储值任务和余额流水仍然要看 `crm_deal_task`、`crm_deal_task_item`、`crm_card_record`。

---

## 总结

霸王餐是一个**条件性储值赠送**功能，核心特点是：

1. **自动触发**：消费时自动检测是否符合霸王餐条件
2. **倍数赠送**：赠送金额 = 账单金额 × 充值倍数
3. **当餐限制**：赠送的储值金额只能在当前订单中使用
4. **灵活配置**：支持按卡类型、金额范围、日期范围、门店等多维度配置
5. **一次性选项**：可选择同一会员卡只能参与一次

这个功能主要用于**促进消费**和**增加客户粘性**的营销策略。
