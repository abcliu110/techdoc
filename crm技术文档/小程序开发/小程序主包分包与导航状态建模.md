# 小程序主包、分包与导航状态建模

本文档记录 `taro-mall` 小程序在测试平台中的框架级建模方式，避免自动化测试靠页面文本或按钮猜测导航。

## 1. 主包和分包的关系

微信小程序只有一个主包。主包不是 4 个，主包里包含启动页、全局配置、公共代码和 `app.json.pages` 声明的页面。

当前 `D:\mywork\taro-mall\dist\app.json` 中，主包页面包括：

```text
pagesIndex/pageIndex/index
pagesIndex/page0/index      首页 tab
pagesIndex/page1/index      点餐 tab
pagesIndex/page2/index      订单 tab
pagesIndex/page3/index      我的 tab
pagesIndex/pagePop/index
pagesIndex/loginForReview/index
```

`subpackages` 是微信小程序官方的分包配置。当前项目只声明了一个分包：

```json
{
  "root": "pagePop",
  "pages": [
    "orderfoods/index",
    "cart/index",
    "orderstatus/index",
    "paycenter/index",
    "recharge/index"
  ]
}
```

真实页面路径由 `root + page` 拼出来：

```text
pagePop/orderfoods/index
pagePop/cart/index
pagePop/orderstatus/index
pagePop/paycenter/index
pagePop/recharge/index
```

所以 `subpackages` 是微信框架配置项，`pagePop` 是项目给分包 root 起的名字，`pagePop/...` 才是具体业务页面路径。

## 2. tabBar 和分包不是父子关系

4 个底部 tab 不是 4 个主包，也不是各自拥有一个分包。真实结构是：

```text
小程序
├─ 主包 pages
│  ├─ pagesIndex/page0/index   首页 tab
│  ├─ pagesIndex/page1/index   点餐 tab
│  ├─ pagesIndex/page2/index   订单 tab
│  └─ pagesIndex/page3/index   我的 tab
│
└─ 分包 pagePop
   ├─ orderfoods/index
   ├─ cart/index
   ├─ orderstatus/index
   ├─ paycenter/index
   ├─ recharge/index
   └─ ...
```

`pagePop` 分包被整个小程序共享。不同 tab 可以通过业务按钮进入同一个分包里的不同页面。

## 3. 点击 tab 后页面如何决定

点击底部 tab 后，微信框架只负责进入固定的 tab 壳页面：

```text
首页 tab -> pagesIndex/page0/index
点餐 tab -> pagesIndex/page1/index
订单 tab -> pagesIndex/page2/index
我的 tab -> pagesIndex/page3/index
```

这些 `pagesIndex/pageX/index` 不是最终业务页面，而是容器。容器页面读取后台导航配置 `navigationStore.value`：

```ts
const pagePath = wxNavigation.find((i) => i.idx === 1)?.pagePath
const pageKey = pagePath == null ? 'ORDER' : pagePath
const Page = pages[pageKey]
```

默认关系是：

```text
idx=0 首页 tab -> HOME
idx=1 点餐 tab -> ORDER
idx=2 订单 tab -> BILL
idx=3 我的 tab -> MEMBER_CENTER
```

然后业务组件再通过 `handlePopPage(...)` 进入分包页。

典型点餐路径：

```text
点击点餐 tab
-> pagesIndex/page1/index
-> 渲染 ORDER 组件
-> 点击桌台/扫码点餐
-> pagePop/selectnumberofpeople/index
-> pagePop/orderfoods/index
-> pagePop/cart/index
-> pagePop/orderstatus/index
-> pagePop/paycenter/index
```

典型充值路径：

```text
点击我的 tab
-> pagesIndex/page3/index
-> 渲染 MEMBER_CENTER 组件
-> 点击充值
-> pagePop/rechargenew/index 或 pagePop/recharge/index
```

## 4. 加载性能关系

小程序启动时只加载主包，不会立即加载 `pagePop` 分包。第一次进入任意 `pagePop/...` 页面时，微信才加载 `pagePop` 分包。

当前项目只有一个较大的 `pagePop` 分包，所以第一次进入点餐、充值、支付等任意 `pagePop` 页面时，可能加载整个分包，首次进入会比 tab 切换慢。加载过一次后，分包内页面切换通常会稳定很多。

测试平台遇到 `pagePop/...` 空白页时，不能一直等待，也不能直接猜按钮点错。超过 10 秒没有 UI、request、member 监控变化时，必须先截图，再分析 DOM、请求和会员状态。

## 5. 测试平台状态机规则

测试平台应按微信框架层和业务流层分开建模。

框架层：

```text
tabBar 页面：
- pagesIndex/page0/index 首页
- pagesIndex/page1/index 点餐
- pagesIndex/page2/index 订单
- pagesIndex/page3/index 我的

分包业务页：
- pagePop/selectnumberofpeople/index 选择人数
- pagePop/orderfoods/index 点菜
- pagePop/cart/index 购物车/下单
- pagePop/orderstatus/index 订单状态/接单成功
- pagePop/paycenter/index 支付中心
- pagePop/recharge/index 充值
- pagePop/rechargenew/index 新充值
```

动作规则：

```text
只有 tabBar.list 里的页面可以 switchTab。
pagePop/... 页面不能当 tab 切换。
pagePop/orderstatus/index 且页面有“接单成功”时，是已下单状态，应进入支付流程，不应重新开台点餐。
pagePop/selectnumberofpeople/index 应选择人数并确认，再进入点菜。
pagePop/orderfoods/index 点菜后必须看到购物车请求、规格弹窗确认或数量变化，不能把未变化的页面当作加菜成功。
余额不足时，必须通过 POS 充值或小程序充值 UI 处理，禁止 API 改余额。
```

核心结论：

```text
tab 决定主入口。
navigationStore 决定 tab 容器渲染哪个业务组件。
业务组件和 handlePopPage 决定进入哪个 pagePop 分包页面。
测试平台必须用状态机判断当前状态和下一步动作。
```
