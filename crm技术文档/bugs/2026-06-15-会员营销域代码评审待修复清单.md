## 二、严重问题

### S1 储值赠品异步发放竞态 + 幂等闸门提前消费

- 文件：`nms4cloud-crm-service/.../service/charge/DepositPlanChargeService.java:716-774`

**问题**
`onPaymentSuccess()` 标注 `@Transactional`，流程为：余额入账 → `markChargeRecordSuccessOnce`（把购买记录从 USERPAYING 原子置为 SUCCESS，作为防重闸门）→ `CompletableFuture.runAsync(handlePostSuccessBenefits)` 异步发券/升级/赠分。

三重缺陷：
1. **竞态**：`runAsync` 在主事务提交前于 ForkJoinPool 线程启动，`handlePostSuccessBenefits` 内 `getChargeRecordByTaskLid`/`crmCardOpServicePlus.getOne` 可能读不到未提交数据，导致 `tier==null` 直接 return。
2. **赠品永久丢失**：闸门已把记录推进到 SUCCESS（幂等资格已消费）。异步发券失败仅 `catch` 打日志（770-773 行），不回滚、不重试。POS 重放 `retryExistingPosCharge` 再次进入会被闸门 return 拦截，**赠券再也不会补发**，且系统无"欠发"记录。
3. 异步线程无事务上下文、无 MDC/登录态。

**触发场景**：发券底层异常（库存不足、会员升级 `parseLevelRank` 抛错等）、或 MQ 重投时序竞态。

**修复方案**（最小改动、保留老逻辑）：
- 核心原则：**让"标记成功"不早于"赠品落库"不可逆地消费幂等资格**。
- 推荐改法：去掉 `CompletableFuture.runAsync`，改为同步执行 `handlePostSuccessBenefits`；并调整顺序为「发券成功 → 才 `markChargeRecordSuccessOnce`」。这样发券失败时闸门保持 USERPAYING，MQ/POS 重试能再次进入补发。
- **前提**：发券/升级/赠分三步必须各自幂等（同一 task 发过不再发第二次）。需先确认 `handleGiftIssuance`/`couponOpAdd` 是否已按 task/订单去重；若未幂等，需先为发券补幂等键，否则"先发券再盖章"会导致重试重复发券。
- 备选（影子/兼容）：保留异步，但为赠品发放建立独立的发放状态记录（schedule/record 各自 SUCCESS 标记），由补偿任务重扫 USERPAYING+未发放的记录补发。
- 同步发放会延长 MQ 消费 RT，需评估；可加开关 `feature.depositPlan.syncBenefit.enabled` 控制新旧路径。

**回滚**：开关关闭即回到异步老路径；无数据库结构变更。

**验证**：补特征化测试覆盖 ①首次成功 ②发券失败后重试补发 ③重复回调幂等 ④USERPAYING 重放。

### S2 储值周期发券失败被吞，计划仍推进导致赠券永久丢失

- 文件：`nms4cloud-crm-service/.../service/charge/DepositPlanCouponService.java:205-208, 234-279, 464, 695`

**问题**
`issueOneTime` 内部 `issueCoupons`/`issueCouponPackages` 用 try-catch 吞掉发券异常（464、695 行），仅记日志；但 `executeScheduledIssuance` 随后**无条件** `issuedTimes+1` 并 `updateById`，甚至将计划置为已完成。本期发券失败 → 计划照常前进 → 会员永久丢失该期赠券且无补偿。`firstIssueImmediate`（205-208 行）同样问题。

**触发场景**：定时发券时券库存不足、券模板已终止、网络抖动等。

**修复方案**：
- 发券失败时**不推进 `issuedTimes`**：让异常向上抛出令事务回滚，或把本期落为"失败/待重试"状态，由下次调度或补偿任务重试。
- 区分"未命中/无需发"与"发放失败"两类：前者可正常推进，后者必须保留重试资格。
- 建议加开关控制新行为，默认开启失败不推进。

**回滚**：开关关闭恢复"无条件推进"老行为。

**验证**：测试覆盖 ①正常发放推进 ②发放失败不推进且可重试 ③计划完成边界。

### H2 消费赠券回查未带 roundKey，多轮同券串轮误撤

- 文件：`nms4cloud-crm-service/.../service/CrmConsumeCouponOpServicePlus.java:182, 498-514`；`revokeIfNecessary` 466-496

**问题**
发券后回查"本轮发了哪些券"靠 `listCouponOrderLids`，查询条件为 `mid + order_bill_id(=outOrderId) + unionid(phone) + coupon_code`，**不含 roundKey**。同一账单多轮赠券（部分退款重结）若命中同一券模板，回查会把**其他轮次**的券一并纳入本轮 `couponOrderLidsJson`，撤销时（`revokeIfNecessary` 按该 lids 批量作废，commit `99187ec`）误废其他轮的券。

**根因**：底层 `couponOpAdd` 返回的 `CouponOpVO` 只带 orderId，不返回本次新发券的 lid，只能事后回查。

**已核实事实**：`coupon_order` 表无 roundKey 字段；`record` 表有 round_key；多轮 outOrderId 相同、roundKey 不同。

**修复方案**（三选一，按风险）：
- **方案 C（最小改动，推荐起步）**：`doGrant` 发券前查一次该券已有 lid 集合，发券后再查一次，**取差集**得到本轮新发券 lid。零数据库变更、不动核心发券类。串行场景完全正确；同账单同券**并发**多轮时有竞态窗口（实际多为串行，概率低）。
- **方案 B（根因修复，最严谨）**：`CouponOpVO` **新增字段** `couponOrderLids`（新增字段，老调用方忽略，向后兼容）；`CrmCouponOpServicePlus.issuance` 把本次 `saveBatch` 的 lid 回传；`doGrant` 改用返回值不再回查。动到被储值赠券/MQ 复用的核心发券类，但只加字段不改老逻辑。
- **方案 A（不推荐）**：`coupon_order` 加 roundKey 列 + 写入 + 回查带 roundKey。动核心券表，影响老券/核销/报表，违反最小改动。

**待确认**：业务上是否存在"同账单 + 同券模板 + 多轮发券"场景（退款重结/分批结账）。若不存在则不触发，但代码缺陷确凿。

**回滚**：方案 C/B 均可 git 回滚；方案 B 新增字段对老调用方无影响。

### H5 nms4pos 补偿任务依赖门店上下文，异常静默跳过

- 文件：三服务 `isConsumeBenefitEnabled()` 无参版（coupon 107-117、points 170-180 等）

**问题**：定时补偿通过 `sycConfigDataService.getMid()/getSid()` 取当前门店判断开关，捕获 `UnsupportedOperationException` 后直接 `return false`。补偿线程上下文缺失/异常时，**所有待补偿记录被静默跳过且不告警**，与"补偿兜底"目标冲突。

**修复方案**：补偿场景按 round 自带的 mid/sid 逐条判断开关，不依赖线程级当前门店；取不到上下文时至少 warn 一次。

## 四、中等问题

### M1 isAvailableByCycle 在 values 为 null 时 NPE

### M2 executeScheduledIssuance 无并发护栏
- 文件：`DepositPlanCouponService.java:234-279`
- 未校验 `status==ACTIVE`、`nextIssueDate<=today`，`updateById` 无乐观锁，定时任务并发/重试可能重复发券。
- **修复**：发放前按 `lid+revision` 条件更新抢占，或校验状态与到期日。

### M3 handleMemberUpgrade 两次 update 无事务包裹
- 文件：`DepositPlanCouponService.java:382-408`
- 等级更新与 `over_time` 延期是两次独立 update，第二次失败则等级升了但有效期没延。
- **修复**：确认调用方事务边界，或方法加 `@Transactional` 包裹两次更新。

### M4 星期+时段组合限制可能漏判时段
- 文件：`CrmConsumeCouponOpServicePlus.java:387-420`（matchLimit）
- `if(week) else if(time)` 结构下，同一行同时配星期+时段时只评估星期，时段被跳过 → 时段限制被绕过。
- **待确认**：`crm_consumption_coupon_limit` 单行能否同时承载星期+时段。若星期/时段分行存储则现逻辑正确。
- **修复**：若单行可承载双类限制，改为同一行内星期、时段各自独立判定累计。

### M5 积分规则同步失败阻断门店全量同步
- 文件：`nms4cloud-pos3boot-biz/.../service/sync/FullSyncDataService.java:267-273, 341-363`
- 仅 `CrmPointsRule` 走 CRM 接口，其余走 pos4cloud；任一表 `downloadPage` 失败即抛异常，CRM 抖动会连带阻断菜品/桌台等核心基础数据全量同步。
- **修复**：`CrmPointsRule` 同步失败时降级为"跳过该表 + 告警"，不阻断整轮；用开关 `feature.fullSync.pointsRuleDegradable.enabled` 控制（默认开启降级）。保留其它表失败仍抛异常的旧行为。或拆为独立同步任务。
- **回滚**：开关关闭恢复"任一失败即整轮失败"老行为。

### M6 赠券退款命中口径与积分不一致
- 文件：`CrmConsumeCouponLocalService.java:248-268`（recalculateConsumeCouponAfterRefund）
- `refundPays` 形同虚设；有 active round 即撤销重发，部分退款未命中赠券基数时也触发撤销，与积分侧 `calculateRefundEligibleAmountFromSnapshot` 精确判定口径不一。
- **待确认**：是否产品决策。若是，注释写明"赠券任意退款即全量撤销重发"；若否，统一三权益退款命中判定口径。
