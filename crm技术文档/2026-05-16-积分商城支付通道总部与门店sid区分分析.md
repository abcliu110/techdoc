# 积分商城支付通道总部与门店 sid 区分分析

## 问题

积分商城微信支付报错：

```text
请设置门店号对应的支付通道
```

当前排查重点是：商城传入 `sid = 1` 时，问题到底是商城传错了，还是商户后台支付通道保存逻辑没有保存 `store_no = 1` 的记录。

## 结论

现有系统没有一个独立的“总部商城 / 非总部商城”支付配置开关。

系统实际靠 `sid` 的取值约定区分总部和门店：

- 页面装修、导航配置层：`sid = -1` 表示总部 / 全局配置，`sid = 具体门店 sid` 表示门店配置。
- 支付通道配置层：总部虚拟门店前端展示来自 `sid = -1`，但保存支付通道时会改成 `store_no = mid`；普通门店保存为 `store_no = sc_store.sid`。
- 积分商城支付层：当前没有按总部 / 门店重新判断，而是直接用 `currentUser.sid` 作为订单和支付的 `sid`。

因此，`pay_store_and_channel` 没有 `store_no = 1` 的记录是符合当前商户后台保存逻辑的。总部支付通道不保存成 `store_no = 1`，而是保存成 `store_no = mid`。

当前问题更倾向于商城前端 / 订单侧传入的 `sid` 来源不一致：积分商城支付传了 `currentUser.sid = 1`，但支付通道查询期望的是 `store_no = mid` 或真实门店 `sid`。

## 证据链

### 1. 商城页面配置如何区分总部和门店

前端加载微信页面配置时，会把当前门店 `shopStore.curShop.sid` 带给后端：

```ts
export const getByAppIdWxPage = (data) => {
  const {curShop} = shopStore
  return request({
    url: apiUrl('getByAppId'),
    data: {
      ...data,
      sid: curShop?.sid?.replace('-', '')
    },
  })
}
```

代码位置：

```text
D:\mywork\taro-mall\src\common\service\crud\WxPageservice.ts
```

后端 `WxPageServicePlus.getByAppId` 的逻辑是：

- 如果 `sid > 0`，先查该门店页面配置。
- 如果查不到，或使用默认配置，则改查 `sid = -1` 的总部配置。
- 如果请求本身不是正数 sid，也直接按请求 sid 查询。

核心逻辑：

```java
Long sid = request.getSid();
if (sid > 0) {

  if (Objects.isNull(data) || NullSafeUtils.nullSafe(data.getUseDefault())) {
    request.setSid(-1L);
    result = get(request);
  }
} else {
  result = get(request);
}
```

### 2. 商户后台支付通道如何保存总部记录

后台支付通道页面加载门店列表时，会请求 `withParent: true`：

```ts
const res = await listSysStore({
  ...params,
  withParent: true,
  pageSize: 1000,
});
```

代码位置：

```text
D:\mywork\nms4cloud-biz-ui\src\pages\OperationalInfoMrg\components\PayChannel\index.tsx
```

后端门店列表在 `withParent = true` 时，会额外加入一个虚拟总部：

```java
if (NullSafeUtils.nullSafe(request.getWithParent())) {
  page.getRecords().addFirst(new ScStore().setMid(request.getMid()).setSid(-1L).setName("总部"));
  page.setTotalRow(page.getTotalRow() + 1);
}
```

代码位置：

```text
D:\mywork\nms4cloud\nms4cloud-app\2_business\nms4cloud-biz\nms4cloud-biz-service\src\main\java\com\nms4cloud\biz\service\ScStoreServicePlus.java
```

但前端拿到总部虚拟门店后，会把 `sid = -1` 改成 `sid = mid`：

```ts
res.data = res.data?.map((v) => {
  if(v.sid === '-1'){
    return {
      ...v,
      sid: v.mid
    }
  }
  return v
})
```

然后保存支付通道时使用：

```ts
storeNo: record.sid,
storeName: record.name,
channelNo: val || '',
```

所以总部支付通道保存成：

```text
pay_store_and_channel.store_no = mid
```

普通门店支付通道保存成：

```text
pay_store_and_channel.store_no = sc_store.sid
```

不会自然保存成：

```text
pay_store_and_channel.store_no = 1
```

## 关键差异

| 场景 | 当前使用的 sid | 总部含义 | 门店含义 |
|---|---:|---|---|
| 页面装修 / 导航配置 | `-1` 或门店 sid | `sid = -1` | `sid = sc_store.sid` |
| 支付通道后台配置 | `mid` 或门店 sid | `store_no = mid` | `store_no = sc_store.sid` |
| 积分商城下单 / 支付 | `currentUser.sid` | 未显式转换 | 未显式使用当前门店 |

这三个地方的 sid 约定不完全一致，尤其是积分商城支付使用 `currentUser.sid`，没有对齐支付通道表的 `store_no` 约定。

## 对当前问题的判断

如果当前商城传的是：

```text
sid = 1
```

支付服务最终会按订单 sid 去查支付通道：

```text
pay_store_and_channel.store_no = 1
```

但商户后台总部支付通道实际保存的是：

```text
pay_store_and_channel.store_no = mid
```

所以查不到 `store_no = 1`，就会报：

```text
请设置门店号对应的支付通道
```

因此目前更支持的判断是：

```text
商城前端 / 订单侧 sid 来源不正确或未做总部场景转换。
```

不是：

```text
商户后台支付通道保存时漏保存 store_no = 1。
```

## 未完全确认项

还需要业务侧确认积分商城的产品定义：

1. 积分商城是否应当永远按总部商城处理。
2. 是否允许门店级积分商城。
3. 如果允许门店级积分商城，进入积分商城前是否必须有明确的 `shopStore.curShop.sid`。
4. `currentUser.sid = 1` 在当前会员登录态中具体代表什么，是默认值、平台 sid、总部账号 sid，还是历史兼容字段。

这些不影响当前代码链路结论，但会影响后续修复方案选择。

## 后续排查建议

只读确认优先级：

1. 查当前商户 `pay_store_and_channel`，确认总部记录是否为 `store_no = mid`。
2. 查积分商城订单表中失败订单的 `sid`，确认是否落成 `1`。
3. 查当前会员登录接口返回的 `currentUser.sid` 来源。
4. 查积分商城入口是否有门店上下文，如果没有，应明确总部场景应传 `mid`。
