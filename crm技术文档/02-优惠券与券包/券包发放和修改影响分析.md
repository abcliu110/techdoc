# 券包发放和修改影响分析

## 1. 发券包时的数据记录

### 1.1 核心结论

**crm_coupon_order 表只记录子券，不记录券包本身！**

### 1.2 数据流程

```
发放券包A（包含2张子券B和3张子券C）
    ↓
查询券包配置（crm_coupon_map）
    ├─ 子券B × 2
    └─ 子券C × 3
    ↓
遍历每个子券，调用 toNormal() 创建订单
    ├─ 创建2条子券B的订单记录
    └─ 创建3条子券C的订单记录
    ↓
在每条子券订单中标记券包信息
    ├─ pkgCoupon = "券包A"
    └─ pkgCouponCode = 100001
    ↓
保存到 crm_coupon_order 表（共5条记录）
```

### 1.3 数据示例

```
crm_coupon_order 表：
┌─────┬────────────┬──────────────┬──────────────┬─────────────────┬──────────┐
│ lid │ coupon     │ couponCode   │ pkgCoupon    │ pkgCouponCode   │ faceValue│
├─────┼────────────┼──────────────┼──────────────┼─────────────────┼──────────┤
│ 001 │ 满100减20  │ 200001       │ 新人大礼包   │ 100001          │ 20.00    │
│ 002 │ 满100减20  │ 200001       │ 新人大礼包   │ 100001          │ 20.00    │
│ 003 │ 8折券      │ 200002       │ 新人大礼包   │ 100001          │ 0.00     │
│ 004 │ 8折券      │ 200002       │ 新人大礼包   │ 100001          │ 0.00     │
│ 005 │ 8折券      │ 200002       │ 新人大礼包   │ 100001          │ 0.00     │
└─────┴────────────┴──────────────┴──────────────┴─────────────────┴──────────┘

说明：
- ❌ 没有券包本身的记录
- ✅ 只有5条子券记录
- ✅ 每条记录都标记了来源券包
```

### 1.4 查询SQL

```sql
-- 查询会员从券包获得的所有券
SELECT * FROM crm_coupon_order
WHERE member_code = '会员卡LID'
  AND pkg_coupon_code = '100001'
ORDER BY get_date DESC;

-- 统计券包发放情况
SELECT
    pkg_coupon,
    coupon,
    COUNT(*) as total,
    SUM(CASE WHEN coupon_status = 'WHX' THEN 1 ELSE 0 END) as unused,
    SUM(CASE WHEN coupon_status = 'YHX' THEN 1 ELSE 0 END) as used
FROM crm_coupon_order
WHERE pkg_coupon_code = '100001'
GROUP BY pkg_coupon, coupon;

-- 查询所有从券包发放的券
SELECT * FROM crm_coupon_order
WHERE pkg_coupon_code IS NOT NULL;
```

## 2. 券包修改的影响分析

### 2.1 核心原理：快照机制

**发券时，系统会将券的所有信息复制到 crm_coupon_order 表中，形成快照。**

```java
// 快照字段（发券时复制）
order.setCoupon(coupon.getName());           // 券名称
order.setPkgCoupon(packageCoupon.getName()); // 券包名称
order.setFaceValue(coupon.getFaceValue());   // 面值
order.setValidDate(validDate);               // 开始时间
order.setDeadline(deadline);                 // 截止时间

// 关联字段（仅用于追溯）
order.setCouponCode(coupon.getLid());        // 券ID
order.setPkgCouponCode(packageCoupon.getLid()); // 券包ID
```

### 2.2 修改影响矩阵

| 修改内容 | 已发券影响 | 新发券影响 | 风险等级 |
|---------|-----------|-----------|---------|
| 券包名称 | ❌ 不受影响（使用快照） | ✅ 使用新名称 | 🟡 低（数据不一致） |
| 券包有效期 | ❌ 不受影响（本来就不用） | ❌ 不受影响 | 🟢 无 |
| 券包面值 | ❌ 不受影响（本来就不用） | ❌ 不受影响 | 🟢 无 |
| 子券数量 | ❌ 不受影响（使用快照） | ✅ 使用新配置 | 🟡 中（用户困惑） |
| 新增子券 | ❌ 不受影响（没有新券） | ✅ 包含新券 | 🟡 中（用户困惑） |
| 删除子券 | ❌ 不受影响（仍可使用） | ✅ 不再包含 | 🟡 中（用户困惑） |
| 子券名称 | ❌ 不受影响（使用快照） | ✅ 使用新名称 | 🟡 低（数据不一致） |
| 子券面值 | ❌ 不受影响（使用快照） | ✅ 使用新面值 | 🔴 高（金额不一致） |
| 子券有效期 | ❌ 不受影响（使用快照） | ✅ 使用新有效期 | 🟡 中（时间不一致） |

### 2.3 详细场景分析

#### 场景1：修改券包名称

```
时间线：
2026-03-01：创建券包"新人大礼包"
2026-03-10：用户A领取券包
2026-03-15：修改券包名称为"新人专享礼包"
2026-03-20：用户B领取券包

结果：
用户A的券：pkgCoupon = "新人大礼包" ✅
用户B的券：pkgCoupon = "新人专享礼包" ✅

问题：
- 同一个券包ID，不同的名称
- 统计报表可能混乱
```

#### 场景2：修改子券配置

```
时间线：
2026-03-01：创建券包A（包含2张券B + 3张券C）
2026-03-10：用户A领取券包
2026-03-15：修改券包A（改为1张券B + 3张券C + 2张券D）
2026-03-20：用户B领取券包

结果：
用户A获得：2张券B + 3张券C ✅
用户B获得：1张券B + 3张券C + 2张券D ✅

问题：
- 用户A和用户B获得的内容不同
- 用户可能投诉：为什么我的券包少了？
```

#### 场景3：修改子券面值

```
时间线：
2026-03-01：创建券B（面值20元）
2026-03-10：用户A领取券包（包含券B）
2026-03-15：修改券B面值为30元
2026-03-20：用户B领取券包（包含券B）

结果：
用户A的券B：faceValue = 20元 ✅
用户B的券B：faceValue = 30元 ✅

问题：
- 同一个券，不同的面值
- 用户A可能投诉：为什么我的券面值少了？
- 风险等级：🔴 高
```

#### 场景4：删除券包中的子券

```
时间线：
2026-03-01：创建券包A（包含券B + 券C）
2026-03-10：用户A领取券包
2026-03-15：从券包A中删除券B
2026-03-20：用户B领取券包

结果：
用户A获得：券B + 券C ✅（券B仍然可用）
用户B获得：券C ✅（没有券B）

问题：
- 用户A的券B仍然有效，可以正常核销
- 但新用户无法获得券B
```

### 2.4 快照字段 vs 关联字段

| 字段 | 类型 | 修改影响 | 用途 |
|------|------|---------|------|
| `coupon` | 快照 | ❌ 不受影响 | 显示券名称 |
| `pkgCoupon` | 快照 | ❌ 不受影响 | 显示券包名称 |
| `faceValue` | 快照 | ❌ 不受影响 | 核销时使用 |
| `validDate` | 快照 | ❌ 不受影响 | 判断是否可用 |
| `deadline` | 快照 | ❌ 不受影响 | 判断是否过期 |
| `couponCode` | 关联 | ⚠️ 仅追溯 | 关联券模板 |
| `pkgCouponCode` | 关联 | ⚠️ 仅追溯 | 关联券包模板 |

## 3. 最佳实践建议

### 3.1 券包修改策略

#### 策略1：版本化管理（推荐）

```
❌ 不推荐：直接修改现有券包
券包A（v1）→ 修改 → 券包A（v2）

✅ 推荐：创建新版本
券包A（v1）→ 停用
券包A（v2）→ 创建新券包

优点：
- 数据一致性好
- 用户体验清晰
- 便于追溯和统计
```

#### 策略2：小修改可以直接改

```
✅ 可以直接修改的内容：
- 券包名称（轻微调整）
- 券包描述
- 券包图片
- 券包排序

❌ 不建议直接修改的内容：
- 子券配置（数量、种类）
- 子券面值
- 子券有效期
```

### 3.2 修改前的检查清单

**修改券包前，请检查：**

1. ✅ 是否有用户已经领取了此券包？
   ```sql
   SELECT COUNT(*) FROM crm_coupon_order
   WHERE pkg_coupon_code = '券包ID';
   ```

2. ✅ 修改是否会影响用户体验？
   - 面值减少 → 🔴 高风险
   - 数量减少 → 🟡 中风险
   - 名称修改 → 🟢 低风险

3. ✅ 是否需要创建新版本？
   - 重大修改 → 创建新版本
   - 轻微调整 → 可以直接修改

### 3.3 修改后的处理

**修改券包后，建议：**

1. ✅ 记录修改日志
   ```
   修改时间：2026-03-15
   修改内容：将券B数量从2张改为1张
   影响范围：2026-03-15之后领取的用户
   ```

2. ✅ 通知用户（如果是重大修改）
   ```
   "亲爱的用户，券包内容已更新，
   您已领取的券不受影响，可以正常使用。"
   ```

3. ✅ 更新前端展示
   - 确保前端显示的券包内容与实际一致

### 3.4 删除券包的注意事项

**删除券包前，请注意：**

1. ⚠️ 已发出去的券不受影响
   - 即使删除券包，已发券仍然有效
   - 可以正常使用和核销

2. ⚠️ 建议使用"停用"而不是"删除"
   ```sql
   -- 停用券包
   UPDATE crm_coupon
   SET status_ = 0
   WHERE lid = '券包ID' AND is_pkg = 1;
   ```

3. ⚠️ 如果必须删除，先检查
   ```sql
   -- 检查是否有已发券
   SELECT COUNT(*) FROM crm_coupon_order
   WHERE pkg_coupon_code = '券包ID';

   -- 如果有已发券，建议不要删除
   ```

## 4. 常见问题

### Q1: 修改券包后，已发出去的券会变吗？

**A:** 不会。已发券使用快照数据，完全不受影响。

### Q2: 删除券包后，已发出去的券还能用吗？

**A:** 可以。券订单中保存了所有必要信息，不依赖券包模板。

### Q3: 修改子券面值后，已发出去的券面值会变吗？

**A:** 不会。券订单中的 `faceValue` 是快照数据。

### Q4: 如何查询某个券包发放了多少券？

**A:**
```sql
SELECT COUNT(*) FROM crm_coupon_order
WHERE pkg_coupon_code = '券包ID';
```

### Q5: 如何区分不同版本的券包？

**A:** 建议在券包名称中加入版本号，例如：
- "新人大礼包 v1"
- "新人大礼包 v2"

## 5. 总结

### 核心结论

1. ✅ **crm_coupon_order 只记录子券，不记录券包本身**
2. ✅ **已发券完全不受券包修改影响（快照机制）**
3. ⚠️ **修改券包可能导致数据不一致和用户困惑**
4. ✅ **重大修改建议创建新版本，而不是直接修改**

### 设计优点

- 已发券独立存在，不依赖券模板
- 即使券模板被删除，已发券仍然有效
- 核销流程简单，直接读取券订单数据

### 设计缺点

- 修改券包可能导致数据不一致
- 同一个券包ID，不同时间发放的内容可能不同
- 缺乏版本管理机制

### 改进建议

1. 增加券包版本管理
2. 修改券包时提示影响范围
3. 重大修改时强制创建新版本
4. 记录券包修改历史
