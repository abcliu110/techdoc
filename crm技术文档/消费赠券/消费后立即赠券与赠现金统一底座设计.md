# 消费完成当下赠券与赠现金统一底座设计

日期：2026-05-12  
范围：`nms4cloud` CRM / Order / POS 消费完成当下赠券、赠现金、撤销、补偿、审计与报表口径  
目标：把“消费完成当下赠券”和“消费完成当下赠现金”做成同一套消费权益底座下的两个独立业务域，支持线上订单和 POS / 线下清台统一接入，同时保留现有赠券、返现、会员余额、券核销等老逻辑。

## 1. 结论

推荐采用：

```text
消费权益中心底座
  + 消费赠券域
  + 消费赠现金域
```

其中：

- 底座统一：消费事件、反结账事件、Inbox / Outbox、业务幂等、任务状态机、补偿任务、审计字段、监控告警。
- 业务域分开：赠券和赠现金各自保留规则、任务、明细、撤销、报表事实表。
- 发放时点统一：赠券和赠现金都以消费完成当下为业务触发点；异步投递、重试和补偿只改变技术执行路径，不改变权益归属时间。
- 执行互不阻塞：赠券失败不影响赠现金，赠现金失败不影响赠券；失败项进入各自补偿队列。
- 反结账按事实回滚：撤销时不按当前规则重算，只按原始任务和明细回收券或冲正现金。
- 财务口径分离：券是营销权益，现金是余额 / 金额类资产，不能混成同一张业务账本。

一句话：**同一个事件底座，两个独立权益域，三套可审计事实：事件事实、赠券事实、赠现金事实。**

## 2. 当前现状

### 2.1 消费赠券现状

当前仓库已有部分赠券能力：

- 订单服务中存在 `sendConsumeCoupon(...)`，但主结账流程调用被注释，注释说明通过线下清台赠送。
- CRM 侧存在 `/consumerGiveCoupon` 外部签名入口，可由 POS / 清台链路调用。
- CRM 侧存在 `CRM_CONSUMER_COUPON_QUEUE` 消费者，收到消息后按消费赠券规则调用通用发券能力。

主要缺口：

- 线上订单消费完成后默认不触发消费赠券。
- 缺少消费赠券专属任务台账和明细台账。
- 缺少基于消费业务键的强幂等。
- 发券失败没有稳定补偿任务。
- 反结账回收没有完整闭环，尤其缺少“按原任务事实回收”的台账。

### 2.2 消费赠现金现状

当前返现能力更接近“返现明细 + 定时任务 + 会员余额入账”：

- 正向链路已有：消费生成 `CrmRebateDetail`，`RebateExecTask` 扫描后调用 `cardCashBack()`，最终落到会员余额。
- 撤销能力部分存在：`revokeCashBack()` 和余额服务 `CancelCashBack` 分支说明系统具备冲正能力。
- 现有 `undoConsume()` 主要覆盖未执行返现明细，已执行返现和原返现明细之间的自动强绑定回滚不完整。

主要缺口：

- 运行时返现链路和返现规则配置之间边界不够清晰。
- 返现明细没有完整保存余额入账任务号、余额流水号、冲正流水号。
- 已执行返现的自动冲正闭环不足。
- 返现失败、冲正失败缺少统一补偿和人工处理台账。

## 3. 设计目标

1. 同一笔消费完成后，可以同时触发赠券和赠现金。
2. POS / 线下清台与线上订单使用统一消费权益事件模型。
3. 同一消费事件重复投递、接口重试、人工补推时，只产生一次业务效果。
4. 赠券和赠现金独立执行，任何一方失败不阻塞另一方。
5. 反结账、退款、撤销消费时，按原任务事实回收券或冲正现金，不按当前规则重算。
6. 规则变更不影响历史任务重试、撤销和报表统计。
7. 报表可以统一看“消费权益”，但事实口径必须按赠券和赠现金分开统计。
8. 保留现有 `/consumerGiveCoupon`、返现明细、通用发券、会员余额入账能力，新增底座以包装和扩展为主，不重写老逻辑。

## 4. 非目标

- 不把赠券和赠现金合并成同一张业务账本。
- 不重写现有通用发券、券核销、会员余额、返现入账基础能力。
- 不把规则配置层强行统一成一个“大权益规则中心”。
- 不改变已有券模板、券订单、会员余额流水的基础语义。
- 不在第一阶段处理复杂组合权益，例如券 + 积分 + 会员等级成长值的统一编排。

## 5. 方案对比

### 方案 A：完全合并成一套权益任务和明细

做法：

- 一张权益任务表记录赠券、赠现金。
- 一张权益明细表记录券、现金等所有结果。
- 用 `benefit_type` 区分业务类型。

优点：

- 表面结构统一。
- 查询统一入口简单。

缺点：

- 券和现金生命周期不同，状态机会被迫抽象到过宽。
- 财务口径容易污染，现金余额不能和券权益混账。
- 反结账策略不同，明细模型会越来越复杂。
- 后续报表解释成本高。

结论：不推荐。

### 方案 B：底座统一，业务域分开

做法：

- 统一消费事件、Inbox / Outbox、幂等、任务调度、补偿、审计。
- 赠券域独立保存赠券任务和赠券明细。
- 赠现金域独立保存现金任务和现金明细。
- 两个执行器订阅同一消费权益任务或同一消费事件。

优点：

- 复用重复投递、补偿、监控、审计等基础能力。
- 保留赠券和现金各自财务 / 报表语义。
- 更容易兼容现有代码，迁移风险较低。
- 支持赠券和赠现金独立成功 / 失败。

缺点：

- 表数量比方案 A 多。
- 统一看板需要聚合两套事实。

结论：推荐。

### 方案 C：完全分开两套链路

做法：

- 消费赠券独立一套事件、任务、补偿。
- 消费赠现金独立一套事件、任务、补偿。

优点：

- 每个业务域内部最简单。
- 对现有代码改动最少。

缺点：

- POS / 线上订单接入会重复建设。
- 幂等、补偿、审计、监控重复。
- 同一消费事件的全链路追踪困难。

结论：可作为短期过渡，不作为目标架构。

## 6. 推荐总体架构

```text
POS / 线下清台 / 线上订单
  |
  | 消费完成事件 / 反结账事件
  v
消费权益事件入口
  |
  | 1. 参数规范化、业务键生成
  v
消费权益 Inbox / Outbox
  |
  | 2. 去重、落事件、派发子任务
  v
消费权益调度器
  |
  +----------------------------+
  |                            |
  v                            v
消费赠券域                    消费赠现金域
  |                            |
  | 规则匹配、任务落库          | 规则匹配、任务落库
  | 发券执行、明细落库          | 入账执行、明细落库
  | 回收、补偿、异常单          | 冲正、补偿、异常单
  v                            v
券订单 / 券明细                会员余额流水
```

关键原则：

- 事件底座只负责“这笔消费事实是什么、是否重复、派发到哪些权益域、处理到什么阶段”。
- 赠券域只负责“这笔消费应该发哪些券、实际发了哪些券、撤销时如何回收”。
- 赠现金域只负责“这笔消费应该赠多少钱、实际入账多少、撤销时如何冲正”。

## 7. 统一消费权益事件模型

### 7.1 事件类型

建议先定义三类事件：

| 事件类型 | 说明 |
|---|---|
| `CONSUME_COMPLETED` | 消费完成当下触发赠券和赠现金 |
| `CONSUME_REVOKED` | 反结账 / 撤销消费，触发回收和冲正 |
| `CONSUME_REPLAY` | 人工补推或系统补偿重放 |

### 7.2 来源类型

| 维度 | 取值 | 说明 |
|---|---|---|
| `origin_source` | `ONLINE_ORDER` / `POS_CLEAR` | 原始业务来源，只参与幂等和回查，对应 `CrmConsumeBenefitOriginSourceEnum` |
| `delivery_source` | `NORMAL` / `MANUAL_REPLAY` / `SYSTEM_RETRY` | 投递或重放来源，不参与幂等，对应 `CrmConsumeBenefitDeliverySourceEnum` |

### 7.3 核心业务键

统一业务键建议：

```text
mid + sid + originSource + originLifecycleId + eventType + outOrderId + cardLid
```

说明：

- `mid`：商户隔离维度，必须参与幂等。
- `sid`：门店隔离维度，必须参与幂等和回查。
- `originSource`：原始业务来源，只能取 `ONLINE_ORDER` / `POS_CLEAR`。
- `deliverySource`：投递来源，只能取 `NORMAL` / `MANUAL_REPLAY` / `SYSTEM_RETRY`，不参与幂等。
- `originLifecycleId`：原始消费生命周期 ID，POS 建议使用清台 / round / bill 生命周期 ID，线上订单使用订单 LID。
- `eventType`：正向发放和撤销必须分开。
- `outOrderId`：外部订单号或账单号，作为追溯字段。
- `cardLid`：会员卡 ID，避免同一订单内多会员身份时互相污染。

如果 POS 场景存在一次清台拆多会员、多账单或多支付主体，必须把 `originLifecycleId` 定义到“权益应发放的最小业务单元”，不能只用门店日结号。

发放时点约束：

- 赠券和赠现金都在消费完成时触发，业务时间使用消费完成时间。
- Outbox / MQ / Inbox 只是可靠投递和去重机制，不允许把“日结后、定时任务后、人工补推后”定义为新的发放时点。
- 补偿重放只能恢复消费完成时本应发生的权益结果，不能按重放时的规则或时间重新计算。

### 7.5 枚举契约

| 业务字段 | 枚举 | 说明 |
|---|---|---|
| 事件类型 | `CrmConsumeBenefitEventTypeEnum` | `CONSUME_COMPLETED` / `CONSUME_REVOKED` / `CONSUME_REPLAY` |
| 原始来源 | `CrmConsumeBenefitOriginSourceEnum` | `ONLINE_ORDER` / `POS_CLEAR` |
| 投递来源 | `CrmConsumeBenefitDeliverySourceEnum` | `NORMAL` / `MANUAL_REPLAY` / `SYSTEM_RETRY` |
| 底座状态 | `CrmConsumeBenefitEventStatusEnum` | `RECEIVED` / `DISPATCHING` / `DISPATCHED` / `PARTIAL_FAILED` / `DONE` / `FAILED` |
| 赠券任务状态 | `CrmConsumeCouponTaskStatusEnum` | `PROCESSING` / `SUCCEEDED` / `FAILED` / `REVOKE_PENDING_GRANT` / `REVOKING` / `REVOKED` / `REVOKE_FAILED` |
| 赠券明细发放状态 | `CrmConsumeCouponGrantStatusEnum` | `PENDING` / `SUCCEEDED` / `FAILED` |
| 赠券明细回收状态 | `CrmConsumeCouponRevokeStatusEnum` | `NOT_REQUIRED` / `REVOKED` / `REVOKE_FAILED` / `USED_CANNOT_REVOKE` |
| 赠现金任务状态 | `CrmConsumeCashTaskStatusEnum` | `PROCESSING` / `SUCCEEDED` / `FAILED` / `REVERSE_PENDING_GRANT` / `REVERSING` / `REVERSED` / `REVERSE_FAILED` |
| 赠现金明细发放状态 | `CrmConsumeCashGrantStatusEnum` | `PENDING` / `SUCCEEDED` / `FAILED` |
| 赠现金明细冲正状态 | `CrmConsumeCashReverseStatusEnum` | `NOT_REQUIRED` / `REVERSED` / `REVERSE_FAILED` |
| 撤销策略 | `CrmConsumeRevokeStrategyEnum` | `BLOCK` / `EXCEPTION` / `PARTIAL_COMPENSATE` |

说明：

- 数据库存枚举 `code`，不存枚举常量名。
- DTO / VO / Entity / Service 对同一业务字段必须使用同一种枚举类型。
- 所有 `@Schema` 必须说明枚举 code 的中文含义。
- 所有事件入口统一使用 `POST`，禁止 `GET` / `PUT` / `DELETE`。
- `eventType`、`originSource`、`deliverySource`、`status`、`revokeStrategy` 这五类固定字段不得再用裸字符串描述。
- `originSource` / `deliverySource` / `originLifecycleId` / `outOrderId` / `cardLid` 这组契约字段名必须作为统一口径使用，其他文档只能引用，不再重新定义别名。

### 7.4 事件快照

消费事件必须保存快照，至少包括：

- 商户、门店、品牌、渠道。
- 订单 / 账单 / 清台生命周期 ID。
- 会员卡、会员、卡类型、卡等级。
- 实付金额、可参与权益金额、支付方式拆分。
- 商品明细、分类、排除项摘要。
- 事件来源和原始请求摘要。
- 触发时间、业务日期。

快照目的：

- 历史任务重试不依赖当前订单状态。
- 反结账按原任务事实回滚。
- 后续报表和审计可复盘当时规则命中依据。

## 8. 底座表设计建议

### 8.1 消费权益事件表 `crm_consume_benefit_event`

用于记录原始消费权益事件和统一处理状态。

建议字段：

| 字段 | 说明 |
|---|---|
| `pid / mid / sid / lid` | 标准四 ID |
| `origin_source` | 来源：ONLINE_ORDER / POS_CLEAR |
| `delivery_source` | 投递来源：NORMAL / MANUAL_REPLAY / SYSTEM_RETRY |
| `event_type` | CONSUME_COMPLETED / CONSUME_REVOKED / CONSUME_REPLAY |
| `origin_lifecycle_id` | 原始消费生命周期 ID |
| `out_order_id` | 外部订单号或订单 LID，对应 DTO 字段 `outOrderId` |
| `bill_lid` | 账单 LID |
| `card_lid` | 会员卡 LID |
| `business_key` | 统一业务幂等键（不含 delivery_source） |
| `event_snapshot` | 事件快照 JSON |
| `status` | RECEIVED / DISPATCHING / DISPATCHED / PARTIAL_FAILED / DONE / FAILED |
| `coupon_status` | NOT_REQUIRED / PENDING / SUCCEEDED / FAILED |
| `cash_status` | NOT_REQUIRED / PENDING / SUCCEEDED / FAILED |
| `retry_count` | 底座重试次数 |
| `next_retry_time` | 下次重试时间 |
| `error_code / error_message` | 最近错误 |
| `created_by / created_time / updated_by / updated_time / deleted / revision` | 审计字段 |

建议唯一键：

```text
uk_consume_benefit_event:
mid + sid + business_key + deleted
```

### 8.2 消费权益 Outbox

建议在事件发起方落本地 Outbox：

| 字段 | 说明 |
|---|---|
| `aggregate_type` | ORDER / POS_CLEAR |
| `aggregate_lid` | 订单或清台业务 ID |
| `event_type` | CONSUME_COMPLETED / CONSUME_REVOKED |
| `business_key` | 统一业务键 |
| `payload` | 事件 JSON |
| `status` | PENDING / SENT / FAILED |
| `retry_count` | 发送重试次数 |
| `next_retry_time` | 下次发送时间 |

Outbox 负责“不丢事件”。订单或清台业务成功提交后，即使 MQ 发送失败，也能靠 Outbox 补发。

### 8.3 消费权益 Inbox

建议 CRM 侧统一 Inbox：

| 字段 | 说明 |
|---|---|
| `message_id` | MQ 消息 ID，可选 |
| `business_key` | 业务唯一键 |
| `payload_hash` | 请求体摘要 |
| `status` | RECEIVED / PROCESSING / DONE / FAILED |
| `retry_count` | 处理重试次数 |
| `last_error` | 最近错误 |

建议唯一键：

```text
uk_consume_benefit_inbox:
mid + sid + business_key + deleted
```

Inbox 负责“挡重复消息”，但不能只依赖 MQ messageId，必须以业务键作为幂等核心。

## 9. 赠券域设计

### 9.1 赠券任务表 `crm_consume_coupon_task`

用于记录一笔消费在赠券域内的处理事实。

关键字段：

| 字段 | 说明 |
|---|---|
| `mid / sid / lid` | 标准 ID |
| `benefit_event_lid` | 关联消费权益事件 |
| `origin_source` | 来源 |
| `delivery_source` | 投递来源 |
| `event_type` | `CONSUME_COMPLETED` / `CONSUME_REVOKED` / `CONSUME_REPLAY` |
| `origin_lifecycle_id` | 原始消费生命周期 ID |
| `out_order_id / bill_lid` | 订单或账单追溯字段 |
| `card_lid / card_no` | 会员卡快照 |
| `rule_lid` | 命中的消费赠券规则 |
| `rule_snapshot` | 规则快照 |
| `amount_snapshot` | 金额快照 |
| `coupon_plan_snapshot` | 应发券列表与数量 |
| `status` | `PROCESSING` / `SUCCEEDED` / `FAILED` / `REVOKE_PENDING_GRANT` / `REVOKING` / `REVOKED` / `REVOKE_FAILED` |
| `grant_task_lid` | 撤销任务关联的原发放任务 |
| `retry_count / next_retry_time` | 补偿控制 |
| `error_code / error_message` | 错误记录 |

建议唯一键：

```text
uk_consume_coupon_grant:
mid + sid + origin_source + origin_lifecycle_id + event_type + card_lid + out_order_id + deleted
```

撤销任务建议：

```text
uk_consume_coupon_revoke:
mid + sid + origin_source + origin_lifecycle_id + event_type + grant_task_lid + deleted
```

### 9.2 赠券明细表 `crm_consume_coupon_grant_detail`

用于记录每张券的实际发放和回收事实。

关键字段：

| 字段 | 说明 |
|---|---|
| `coupon_task_lid` | 赠券任务 |
| `coupon_template_lid` | 券模板 |
| `coupon_order_lid` | 实际券订单 / 券明细 |
| `coupon_name_snapshot` | 券名称快照 |
| `grant_status` | `PENDING` / `SUCCEEDED` / `FAILED`，对应 `CrmConsumeCouponGrantStatusEnum` |
| `revoke_status` | `NOT_REQUIRED` / `REVOKED` / `REVOKE_FAILED` / `USED_CANNOT_REVOKE`，对应 `CrmConsumeCouponRevokeStatusEnum` |
| `grant_time` | 发券时间 |
| `revoke_time` | 回收时间 |
| `error_message` | 最近错误 |

反结账时只看此表，不重新按当前赠券规则计算。

### 9.3 赠券执行策略

正向流程：

1. 底座收到 `CONSUME_COMPLETED`。
2. 创建或复用赠券任务。
3. 按事件快照匹配赠券规则。
4. 保存规则快照和应发券计划。
5. 调用现有通用发券能力。
6. 保存每张券实际结果。
7. 任务成功或失败进入对应状态。

失败处理：

- 规则未命中：任务可标记为 `SUCCEEDED`，明细为空，并记录未命中原因。
- 发券部分失败：任务标记 `FAILED` 或 `PARTIAL_FAILED`，成功券明细保留，失败明细进入补偿。
- 重试时只补失败明细，不重复发已成功的券。

## 10. 赠现金域设计

### 10.1 赠现金任务表 `crm_consume_cash_task`

用于记录一笔消费在赠现金域内的处理事实。

关键字段：

| 字段 | 说明 |
|---|---|
| `mid / sid / lid` | 标准 ID |
| `benefit_event_lid` | 关联消费权益事件 |
| `origin_source` | 来源 |
| `delivery_source` | 投递来源 |
| `event_type` | `CONSUME_COMPLETED` / `CONSUME_REVOKED` / `CONSUME_REPLAY` |
| `origin_lifecycle_id` | 原始消费生命周期 ID |
| `out_order_id / bill_lid` | 追溯字段 |
| `card_lid / card_no` | 会员卡快照 |
| `rule_lid` | 命中的返现 / 赠现金规则 |
| `rule_snapshot` | 规则快照 |
| `amount_snapshot` | 金额和支付拆分快照 |
| `cash_plan_snapshot` | 应赠现金计算结果 |
| `cash_amount` | 应赠金额 |
| `status` | `PROCESSING` / `SUCCEEDED` / `FAILED` / `REVERSE_PENDING_GRANT` / `REVERSING` / `REVERSED` / `REVERSE_FAILED` |
| `grant_task_lid` | 撤销任务关联的原发放任务 |
| `retry_count / next_retry_time` | 补偿控制 |
| `error_code / error_message` | 错误记录 |

建议唯一键：

```text
uk_consume_cash_grant:
mid + sid + origin_source + origin_lifecycle_id + event_type + card_lid + out_order_id + deleted
```

### 10.2 赠现金明细表 `crm_consume_cash_grant_detail`

用于记录实际余额入账和冲正事实。

关键字段：

| 字段 | 说明 |
|---|---|
| `cash_task_lid` | 赠现金任务 |
| `card_lid` | 会员卡 |
| `cash_amount` | 入账金额 |
| `deal_task_lid` | 会员卡操作任务号 |
| `balance_record_lid` | 余额流水号 |
| `revoke_deal_task_lid` | 冲正任务号 |
| `revoke_balance_record_lid` | 冲正余额流水号 |
| `grant_status` | `PENDING` / `SUCCEEDED` / `FAILED`，对应 `CrmConsumeCashGrantStatusEnum` |
| `reverse_status` | `NOT_REQUIRED` / `REVERSED` / `REVERSE_FAILED`，对应 `CrmConsumeCashReverseStatusEnum` |
| `grant_time / revoke_time` | 入账和冲正时间 |
| `error_message` | 最近错误 |

### 10.3 赠现金执行策略

正向流程：

1. 底座收到 `CONSUME_COMPLETED`。
2. 创建或复用赠现金任务。
3. 按事件快照匹配赠现金规则。
4. 保存规则快照和应赠金额。
5. 调用现有 `cardCashBack()` 或等价余额入账能力。
6. 保存入账任务号和余额流水号。

撤销流程：

1. 底座收到 `CONSUME_REVOKED`。
2. 找到原发券任务。
3. 根据原赠现金明细调用 `revokeCashBack()` 或等价冲正能力。
4. 保存冲正任务号和余额流水号。

注意：

- 已执行现金必须走反向冲正，不能直接改原明细为取消。
- 冲正金额以原明细为准，不按当前规则重算。
- 如果会员余额不足，默认进入可追踪异常状态；负余额冲正只能通过商户级显式开关启用。

## 11. 统一状态机

### 11.1 底座事件状态

```text
RECEIVED
  -> DISPATCHING
  -> DISPATCHED
  -> DONE

DISPATCHING / DISPATCHED
  -> PARTIAL_FAILED
  -> FAILED
```

含义：

- `DONE`：赠券域和赠现金域都已完成，或明确不需要执行。
- `PARTIAL_FAILED`：至少一个权益域失败，但另一个权益域成功或不需要。
- `FAILED`：底座无法完成分发或两个权益域都失败。

### 11.2 赠券任务状态

```text
PROCESSING
  -> REVOKE_PENDING_GRANT
  -> SUCCEEDED
  -> FAILED
  -> REVOKING
  -> REVOKED
  -> REVOKE_FAILED
```

### 11.3 赠现金任务状态

```text
PROCESSING
  -> REVERSE_PENDING_GRANT
  -> SUCCEEDED
  -> FAILED
  -> REVERSING
  -> REVERSED
  -> REVERSE_FAILED
```

状态设计原则：

- 失败状态可重试。
- 成功状态不可被覆盖，只能追加撤销 / 冲正事实。
- 撤销任务必须关联原正向任务。
- 人工处理也要写回状态和操作记录。
- 现金类人工动作必须双人审批，非现金类人工动作至少要求理由码和原始事件快照。

### 11.4 底座与子域状态收敛矩阵

| 底座状态 | 赠券状态 | 赠现金状态 | 说明 |
|---|---|---|---|
| `DONE` | `SUCCEEDED` / `NOT_REQUIRED` | `SUCCEEDED` / `NOT_REQUIRED` | 两个权益域都完成或明确不需要执行 |
| `PARTIAL_FAILED` | `FAILED` / `REVOKE_FAILED` / `REVOKE_PENDING_GRANT` | `FAILED` / `REVERSE_FAILED` / `REVERSE_PENDING_GRANT` | 至少一个权益域失败，另一个成功或不需要 |
| `FAILED` | `FAILED` | `FAILED` | 底座分发失败或两个权益域都失败 |

## 12. 幂等策略

幂等分四层：

### 12.1 事件幂等

`crm_consume_benefit_event` 以 `mid + sid + business_key + deleted` 做唯一约束。

同一消费事件重复投递时：

- 如果已有事件为 `DONE`，直接返回成功。
- 如果已有事件为 `FAILED`，允许进入补偿重试。
- 如果已有事件为 `PROCESSING`，返回处理中或由调度器继续处理。

### 12.2 赠券任务幂等

赠券任务以消费业务键 + 权益域 + 会员卡做唯一约束。

重复执行时：

- 已成功发出的券不重复发。
- 只补偿失败明细。
- 撤销只对原任务明细执行一次。

### 12.3 赠现金任务幂等

赠现金任务以消费业务键 + 权益域 + 会员卡做唯一约束。

重复执行时：

- 已存在余额入账流水则不重复入账。
- 已存在冲正流水则不重复冲正。
- 如果底层余额服务支持幂等请求号，应把 `cash_task_lid` 或 `cash_detail_lid` 作为请求幂等号。

### 12.4 下游业务幂等

不能只依赖底座幂等。通用发券和余额入账仍应有业务请求号或来源单号，避免网络超时后重复调用造成重复资产。

## 13. 反结账 / 撤销设计

反结账流程：

```text
CONSUME_REVOKED
  -> 查原消费权益事件
  -> 查原赠券任务和赠现金任务
  -> 赠券域按明细回收
  -> 赠现金域按明细冲正
  -> 汇总撤销结果
```

关键原则：

- 不按当前规则重算。
- 不按订单号模糊删除。
- 不直接修改历史成功事实。
- 只追加回收 / 冲正事实。

异常场景：

| 场景 | 建议处理 |
|---|---|
| 原消费事件不存在 | 记录 `NO_ORIGINAL_EVENT`，允许人工处理 |
| 原赠券已使用 | 标记 `USED_CANNOT_REVOKE`，由业务决定阻断反结账或生成异常单 |
| 原赠现金余额不足 | 标记 `BALANCE_NOT_ENOUGH`，默认进入 `REVERSE_FAILED` 并生成异常单；负余额冲正仅允许商户级显式开关，默认关闭 |
| 原任务部分成功 | 只撤销成功部分，失败部分无需撤销 |
| 重复撤销 | 命中撤销任务唯一键，返回已有结果 |

默认策略：

- POS 当场反结账遇到已使用赠券时默认 `BLOCK`；线上退款默认 `EXCEPTION`。
- 部分撤销按原任务事实处理，只撤销成功部分，失败部分不反向补撤。
- 赠现金已消费导致余额不足时默认进入异常单；负余额冲正仅允许商户级显式开关，默认关闭。

## 14. 失败补偿策略

### 14.1 独立补偿

赠券和赠现金失败互不影响。

例如：

- 赠券成功，赠现金失败：事件状态为 `PARTIAL_FAILED`，现金任务进入补偿。
- 赠券失败，赠现金成功：事件状态为 `PARTIAL_FAILED`，赠券任务进入补偿。
- 两者都失败：事件状态为 `FAILED`，两个任务各自补偿。

### 14.2 重试策略

建议：

- 前 3 次快速重试。
- 后续指数退避。
- 超过最大次数进入人工异常单。
- 每次重试必须记录最近错误和重试次数。

### 14.3 人工处理

人工处理入口至少要能看到：

- 原消费事件。
- 赠券任务、赠券明细。
- 赠现金任务、余额流水。
- 失败原因、重试次数。
- 可执行动作：重试、跳过、标记已人工处理、人工回收、人工冲正。

人工动作必须写操作人和操作说明。

## 15. 规则快照与规则变更

正向发放时必须保存规则快照。

赠券快照建议包括：

- 规则 ID、规则名称、规则版本。
- 门店 / 卡类型 / 卡等级 / 金额门槛。
- 满额规则、券模板、发放数量。
- 生效时间和渠道限制。

赠现金快照建议包括：

- 规则 ID、规则名称、规则版本。
- 返现方式：固定金额 / 比例 / 分档。
- 金额基数：实付金额、可参与金额、排除金额。
- 上限、取整、特殊排除条件。

规则变更后：

- 历史任务重试按快照执行。
- 反结账按原任务明细执行。
- 新消费按新规则执行。

## 16. 与现有系统的兼容策略

### 16.1 保留老入口

保留：

- `/consumerGiveCoupon`
- `CRM_CONSUMER_COUPON_QUEUE`
- 现有 `CrmRebateDetail + RebateExecTask + cardCashBack()` 正向返现链路
- 通用 `couponOpAdd()` / `couponOpRevoke()`
- 通用 `cardCashBack()` / `revokeCashBack()`

新增底座优先包装这些能力，不直接替换。

### 16.2 灰度接入

建议新增开关：

```text

feature.crm.consume-benefit.online-order-enabled
feature.crm.consume-benefit.pos-clear-enabled
feature.crm.consume-benefit.coupon-enabled
```

2. 灰度门店开启 POS 清台来源。
3. 灰度线上订单来源。
4. 稳定后扩大范围。

### 16.3 老逻辑保护

- 开关关闭时，现有赠券和返现逻辑保持不变。
- 新底座只在明确来源和开关开启时接管。
- 旧 MQ 消费者可以先写入新事件表做影子记录，不直接改变发券结果。
- 旧返现明细可以先补充关联字段或旁路映射，不直接改变现有定时任务行为。

## 17. 报表与财务口径

### 17.1 统一维度

可统一维度：

- 商户、门店、品牌。
- 订单、账单、消费生命周期 ID。
- 会员卡、会员、卡类型、卡等级。
- 来源渠道、业务日期、规则版本。

### 17.2 事实分开

必须分开：

- 赠券事实：应发券、实发券、回收券、已使用不可回收券、过期券。
- 赠现金事实：应赠金额、实赠金额、冲正金额、冲正失败金额、未闭环金额。

### 17.3 不允许的口径

不建议：

- 把券面值和现金金额直接相加成“权益总额”。
- 用赠券成功率代表整体权益成功率。
- 用返现金额口径统计券成本。
- 把已使用券简单等同于现金支出。

## 19. 测试计划

### 19.1 正向发放

- 命中赠券、命中赠现金，两者都成功。
- 只命中赠券，不命中赠现金。
- 只命中赠现金，不命中赠券。
- 两者都不命中，事件完成但不产生明细。

### 19.2 独立失败

- 赠券失败、赠现金成功，赠券补偿任务生成。
- 赠现金失败、赠券成功，赠现金补偿任务生成。
- 两者都失败，各自补偿互不覆盖。

### 19.3 幂等

- 同一消费完成事件重复投递，只产生一套赠券和赠现金业务效果。
- MQ 重投、接口重试、人工补推使用同一业务键。
- 发券成功后网络超时重试，不重复发券。
- 入账成功后网络超时重试，不重复入账。
- Outbox 已提交但发送前进程崩溃，重启后必须补发。
- Inbox 已接收但任务状态未落库，重放后必须识别已处理事实，不重复发放。
- 同一 `sid` 下不同门店不得串账。
- 一次清台拆多会员 / 拆单，只影响对应最小权益单元。

### 19.4 反结账

- 原赠券未使用，反结账成功回收。
- 原赠券已使用，进入已使用不可回收状态或阻断，按业务配置处理。
- 原赠现金已入账，反结账成功冲正。
- 原赠现金余额不足，默认进入异常单，不允许负余额冲正。
- 重复反结账只产生一次回收和冲正。
- 先到撤销、后到发放时，必须进入挂起态，直到原任务闭环。

### 19.5 规则快照

- 正向发放后修改规则，补偿重试仍按原快照执行。
- 正向发放后修改规则，反结账仍按原明细回收 / 冲正。

### 19.6 来源追踪

- POS 场景按 `originLifecycleId` 追溯。
- 线上订单场景按订单 LID 追溯。
- 同一会员同一订单不同权益域可分开追踪。
- 灰度期必须按 `mid / sid / originSource / originLifecycleId` 对账影子记录与现网结果。

## 20. 分阶段落地建议

### 第一阶段：只建底座和影子记录

目标：

- 接收 POS / 线上订单消费完成事件。
- 生成统一业务键。
- 写入事件表、Inbox / Outbox。
- 不改变现有赠券和返现执行结果。

价值：

- 先验证事件完整性、业务键、来源字段。
- 降低对老逻辑影响。

### 第二阶段：接入赠券域

目标：

- 新增赠券任务和明细。
- 包装现有 `couponOpAdd()`。
- 支持赠券补偿和反结账回收。
- POS / 清台来源灰度启用。

### 第三阶段：接入赠现金域

目标：

- 新增赠现金任务和明细。
- 包装现有 `cardCashBack()` 和 `revokeCashBack()`。
- 保存余额流水关联。
- 支持已执行返现冲正闭环。

### 第四阶段：统一看板和异常处理

目标：

- 消费权益事件看板。
- 赠券异常单。
- 赠现金异常单。
- 统一链路追踪和告警。

## 21. 关键风险

| 风险 | 说明 | 建议 |
|---|---|---|
| 业务键定义不稳定 | POS / 线上订单生命周期不同 | 先梳理每个来源的最小权益单元 |
| 下游无幂等 | 发券 / 入账网络超时后可能重复 | 给下游传业务请求号，并落明细 |
| 反结账策略不统一 | 已用券、余额不足需要业务裁决 | 做成明确异常状态，不静默吞掉 |
| 规则快照不完整 | 重试和撤销无法复盘 | 快照保存命中所需字段 |
| 报表混口径 | 券和现金混算 | 事实表分开，统一看板只做聚合展示 |
| 老逻辑被误改 | 现有 POS / 清台和返现链路可能依赖旧行为 | 开关默认关闭，先影子记录 |

## 22. 上线前仍需确认的问题

以下问题不改变本文的技术契约，只影响上线范围、字段映射或业务配置默认值：

1. POS 清台场景的 `originLifecycleId` 使用清台 LID、round ID 还是账单 LID 作为最小权益单元。
2. 线上订单与 POS 清台的上线顺序：默认先灰度单一来源，再扩大到另一来源。
3. 赠现金规则沿用现有返现规则，还是新建“消费赠现金规则”。
4. 哪些商户允许开启负余额冲正显式开关；默认不开启。
5. 人工补偿只能按事件快照执行，不允许重新按当前规则计算；如需例外，必须单独审批。

## 23. 推荐实施顺序

推荐先做：

1. 统一消费权益事件模型和业务键。
2. 底座事件表、Inbox、Outbox、补偿状态机和枚举契约。
3. 赠券域任务表和明细表，包装现有发券。
4. 赠券反结账回收。
5. 赠现金域任务表和明细表，包装现有余额入账。
6. 赠现金已执行冲正闭环。
7. 统一异常看板和报表聚合。

不要先做：

- 大一统规则中心。
- 大一统权益账本。
- 直接替换现有返现定时任务。
- 默认开启线上订单发券 / 赠现金。

## 24. 最终建议

本需求不应被理解为“在消费完成后多调用两个方法”。它本质是一个跨订单、POS、CRM、券、会员余额、反结账、报表的权益资产闭环。

最佳路径是：

- 用统一消费权益底座承接事件、幂等、补偿和审计。
- 用赠券域处理券资产生命周期。
- 用赠现金域处理余额 / 金额资产生命周期。
- 用规则快照和明细台账保证历史可追溯。
- 用开关和影子记录保护老逻辑。

这样既能支持“消费完成当下赠券 + 赠现金”，也能避免后续在反结账、重复投递、财务报表上形成不可解释的隐患。
