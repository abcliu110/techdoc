## 一、背景与目标

### 1.1 现状

消费赠券在系统中存在两套链路：

| 链路 | 现状 | 问题 |
|------|------|------|
| **线上（nms4cloud-order）** | 有 Outbox + RocketMQ + CRM 消费者 + 幂等 Task，代码已实现但线上结账主流程中调用被注释 | 结账不触发送券，线上链路不可用 |
| **门店（POS）** | Feign 接口定义了但从未调用，无本地状态机，无补偿 | POS 侧完全不可用 |

消费积分（POS + 线上）已有完整链路，可作为消费赠券的实现参照。

### 1.2 目标

在门店和线上都可以**可靠**实现消费赠券发放与反结账回收：

- 可靠 = **不丢**（本地落库 + 失败补偿）+ **不重**（业务键幂等）+ **可恢复**（定时补偿）+ **可排查**（操作流水）

## 二、需求范围

### 2.1 纳入范围

| 功能 | 说明 |
|------|------|
| POS 结账赠券 | 会员消费满额，结账成功后自动赠券 |
| POS 反结账退券 | 反结账成功后退回未使用券，已使用券按策略处理 |
| 线上自助结账赠券 | 会员线上消费满额，结账成功后自动赠券 |
| 线上反结账退券 | 线上反结账后退回未使用券 |
| 定时补偿 | 网络异常/CPU 高负载导致发放失败时，定时重试 |
| 操作流水 | 每步操作均有记录，便于排查 |

### 2.2 不纳入范围（本次）

- 消费返现（赠现金）：正向链路已有，撤销链路需单独讨论，本次不实现
- 消费赠券的 CRM 后台管理页面（规则配置已有）
- 异常单人工处理页面

### 2.3 验收标准

1. POS 结账后，CRM 有 `crm_coupon_order` 记录（券流水）
2. POS 反结账后，券被正确回收（未使用）/进入异常单（已使用）
3. 线上结账后，CRM 有 `crm_coupon_order` 记录
4. 线上反结账后，券被正确回收
5. CRM 调用超时/失败后，定时补偿能重试成功
6. 重复结账不重复赠券（幂等）
7. 先反结账后发券（乱序场景），撤销能等待发券完成后回收

## 三、功能需求

### 3.1 POS 侧消费赠券

#### 3.1.1 结账成功赠券

**触发时机**：POS 结账事务提交成功后（afterCommit），调用 CRM 发券

**处理流程**：
1. 查询消费赠券规则（本地镜像表），判断是否满足门槛（billAmount / crmAmount）
2. 计算应赠券计划（发哪些券、多少张）
3. 幂等创建本地轮次记录（`crm_consume_coupon_round`，status = PENDING）
4. 调用 CRM `POST /crm_card_op/consumerGiveCoupon`
5. 成功 → status = GRANTED，记录 CRM task_lid
6. 失败 → status = FAILED，进入补偿队列

**幂等键**：`mid + sid + "POS" + billLid + "GRANT" + orderId + cardLid`

#### 3.1.2 反结账退券

**触发时机**：POS 反结账事务提交成功后

**处理流程**：
1. 查本地轮次记录
2. 状态判断：
   - GRANTED → 调用 CRM 撤销，记录 status = REVOKE_PENDING
   - PENDING/FAILED → 停止发放，标记 REVOKED
   - REVOKED → 幂等返回
3. 撤销成功 → status = REVOKED
4. 撤销失败 → status = REVOKE_FAILED，进入补偿

#### 3.1.3 定时补偿

每 60 秒扫描一次，扫描 PENDING / FAILED / REVOKE_PENDING / REVOKE_FAILED 状态的轮次，按指数退避重试，重试 N 次后保留失败状态。

### 3.2 线上侧消费赠券

#### 3.2.1 结账成功赠券（恢复）

**现状**：代码已有（`OrderConsumeBenefitOutboxService` + `enqueueAndDispatchGrant`），但主流程中被注释。

**需求**：恢复主流程调用，确保结账事务提交后通过 Outbox + RocketMQ 发送到 CRM。

#### 3.2.2 反结账退券（恢复）

**现状**：代码已有（`enqueueAndDispatchRevoke`）。

**需求**：恢复主流程调用。

### 3.3 CRM 侧增强

#### 3.3.1 POS 来源幂等

在 `consumerGiveCoupon` 接口中增强幂等：先按 business_key 查 `crm_consume_benefit_task`，已执行则幂等返回，未执行则新建任务 + 执行 `couponOpAdd`。

#### 3.3.2 撤销幂等

在 `revokeCoupon` 接口中增强幂等：先按原 GRANT business_key 查 Task，执行 `couponOpRevoke`。

## 四、数据模型

### 4.1 POS 侧新建表

```
crm_consume_coupon_session   # 会话：管理一轮消费（含多次结账/反结账）
crm_consume_coupon_round     # 轮次：每个赠券操作
crm_consume_coupon_event     # 事件：操作流水
```

参照现有 `crm_consume_points_session/round/event` 三表结构，枚举字段改用通用枚举（见 4.2）。

### 4.2 通用枚举

消费赠券、消费赠积分共用同一套通用枚举，命名以 `CrmConsumeBenefit` 为前缀：

| 枚举 | 说明 |
|------|------|
| `CrmConsumeBenefitSessionStatusEnum` | 会话状态：ACTIVE / COMPLETED / CLOSED |
| `CrmConsumeBenefitRoundStatusEnum` | 轮次状态：PENDING / GRANTED / FAILED / REVOKE_PENDING / REVOKING / REVOKED / REVOKE_FAILED / RETRYABLE |
| `CrmConsumeBenefitEventTypeEnum` | 事件类型：GRANT / REVOKE / RETRY / COMPENSATION |
| `CrmConsumeBenefitEventStatusEnum` | 事件状态：SUCCESS / FAILED / PENDING |
| `CrmConsumeBenefitSourceEnum` | 来源：POS / ORDER |

### 4.3 幂等统一

CRM 侧复用 `crm_consume_benefit_task.business_key` + 唯一索引，POS 和线上使用完全一致的幂等键格式：

```
mid|sid|source|lifecycleId|eventType|outOrderId|cardLid
```

## 五、非功能需求

| 需求 | 要求 |
|------|------|
| 幂等 | 重复结账不重复赠券，CRM 侧保证 |
| 不丢 | POS 本地事务写入轮次表 + Feign 调用失败后轮次表标记 FAILED + 定时重试 |
| 不重 | CRM 按 business_key 幂等 + POS 轮次表 GRANTED 时跳过 |
| 可恢复 | 定时补偿任务扫描 PENDING/FAILED/REVOKE_FAILED |
| 可排查 | 轮次表有快照 + 事件表有操作流水 |
| 事务边界 | afterCommit 后才发券，避免业务回滚但券已发 |
| 乱序处理 | revoke 时找不到 GRANT 任务 → RETRYABLE 等 GRANT 完成 |

## 六、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| POS 通信方式 | Feign 直调（不走 RocketMQ） | 消费积分已用此方式，稳定；不需要额外 outbox 复杂度 |
| POS 表结构 | 三表（Session/Round/Event） | 完全复用消费积分成熟模式，可追溯性高 |
| CRM 幂等 | 复用 CrmConsumeBenefitTask | 已存在，business_key 结构完整，POS 和线上共享同一张幂等表 |
| 消费返现 | 本次不实现 | 正向链路已有，撤销链路需更多业务策略讨论 |
