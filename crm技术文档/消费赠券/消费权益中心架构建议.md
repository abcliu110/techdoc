# 消费权益中心架构建议

## 结论

“消费赠券”和“消费返现”**可以共用同一套架构方法论**，但**不能共用同一套业务账本、财务口径和报表事实表**。

业界更合理的做法是：

- 共用基础能力：事件接入、Inbox/Outbox、幂等、任务状态机、补偿、审计、监控。
- 分开业务域：赠券域和返现域各自维护独立任务表、明细表、撤销表和报表事实表。
- 分开财务口径：券是权益资产，返现是余额/金额资产，两者不能用同一套业务状态直接混算。
- 统一发放时点：赠券和返现都以消费完成当下为业务触发点；异步投递和补偿只保证可靠执行，不改变权益归属时间。

## 一、为什么可以共用底座

两条链路的共性足够高：

1. 都是“消费触发的权益发放”。
2. 都存在重复通知、MQ 重投、人工补推、补偿恢复等问题。
3. 都需要可追溯的任务台账。
4. 都需要撤销、冲正、回收、人工异常单。

所以底层架构可以统一成：

```text
消费事件/反结账事件
  -> Inbox / Outbox
  -> 业务任务台账
  -> 执行器
  -> 业务明细台账
  -> 补偿 / 冲正 / 人工处理
```

这和现有赠券架构文档的方向一致，也和返现链路当前“消费明细 + 执行任务 + 撤销能力”的形态一致。

## 二、为什么不能共用业务账本

### 1. 业务对象不同

- 赠券：处理的是“券的发放、核销、回收、过期”。
- 返现：处理的是“金额入账、余额变化、冲正、回滚”。

这两类对象的生命周期不同，状态也不同，不能只靠一个通用状态机硬合并。

### 2. 财务风险不同

赠券本质上是营销权益，不应直接等同为现金。  
返现本质上是余额/金额类资产，必须满足更强的财务可追溯要求。

因此：

- 赠券侧要强调“发券事实 + 使用事实 + 回收事实”。
- 返现侧要强调“入账事实 + 冲正事实 + 余额变化事实”。

### 3. 报表口径不同

如果把两者混成一张事实表，报表会出现口径污染：

- 赠券报表关心券发放量、核销率、回收率、过期率。
- 返现报表关心返现金额、入账成功率、冲正金额、未闭环金额。

共用维度可以，混用事实不行。

## 三、财务标准建议

推荐采用“**不可变流水 + 反向冲正**”的财务思路。

### 1. 赠券

- 任务表记录业务意图。
- 明细表记录每张券的实际发放和回收。
- 撤销不改历史事实，只追加回收事实或状态迁移事实。

### 2. 返现

- 返现明细要保留发放事实。
- 已执行返现需要独立冲正流水。
- 不能只依赖原始消费明细的 `update` 来假装回滚完成。

### 3. 统一原则

所有对外可审计的权益动作，都应满足：

- 有原始触发事件。
- 有执行结果。
- 有撤销/冲正记录。
- 有可复算的业务主键。
- 有规则快照，不依赖当前配置重算历史。

## 四、报表设计建议

### 1. 维度统一

建议统一维度表或公共维度字段：

- 商户
- 门店
- 会员
- 订单
- 来源渠道
- 规则版本
- 业务日期
- 操作人

### 2. 事实分开

建议分开两套事实模型：

- 赠券事实表：发券、核销、回收、过期。
- 返现事实表：返现、入账、冲正、撤销。

### 3. 汇总层统一

可以在“消费权益中心”下面做统一看板，但展示层应按业务类型拆分：

- 赠券看板
- 返现看板
- 风险与异常看板

不要把两类指标直接求和后当成一个“权益总额”，那会破坏财务解释性。

## 五、推荐目标架构

```text
                消费权益中心
                      |
         +------------+------------+
         |                         |
      赠券域                    返现域
         |                         |
  任务表 / 明细表 / 撤销表   明细表 / 余额流水 / 冲正表
         |                         |
      执行器 / 补偿             执行器 / 冲正 / 补偿
         |                         |
              统一报表与监控
```

共享的是平台能力，不共享的是业务账。

## 六、落地建议

1. 先定义统一的“消费权益中心”事件模型、业务键和任务状态规范。
2. 赠券和返现各自保留独立表结构和独立唯一键。
3. 报表层统一维度，事实层分开建模。
4. 财务相关字段全部保留快照和冲正链路。
5. 历史规则、历史任务、历史回收必须按事实回放，不按当前配置重算。
6. 事件契约必须和主设计文档保持一致，`origin_source` / `delivery_source` / `origin_lifecycle_id` / `sid` 以及 `CrmConsumeBenefitEventTypeEnum` / `CrmConsumeCouponTaskStatusEnum` / `CrmConsumeCashTaskStatusEnum` / `CrmConsumeRevokeStrategyEnum` 不能再重新定义。

## 七、最终判断

如果问题是“能不能用同样的架构设计思想”，答案是：**可以**。

如果问题是“能不能共用同一套业务表、同一套财务账和同一套报表事实”，答案是：**不可以**。

最优解是：

> 共用底座，分离账本，统一维度，分开事实。
