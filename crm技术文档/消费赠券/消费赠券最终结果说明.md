# 消费赠券最终结果说明

日期：2026-05-12  
范围：消费完成当下发券、Inbox/Outbox 去重、反结账回收券

## 1. 最终结论

当前仓库里，`消费完成当下自动发券` 不是线上订单结账的默认行为。

更准确地说：

- 订单服务里有 `sendConsumeCoupon(...)` 实现，但线上结账主流程里这行调用被注释掉了。
- 现有可触发消费赠券的入口，主要是 CRM 的 `/consumerGiveCoupon` 外部签名接口，或等价的 POS / 线下清台调用链。
- CRM 侧有 MQ 消费者，会在收到 `CRM_CONSUMER_COUPON_QUEUE` 后按规则发券。

所以，当前实现更像“具备消费送券能力”，不是“所有消费自动送券”。

## 2. Inbox / Outbox 是什么

### Outbox

发件箱。由**业务发起方**写入。

作用：

- 先把“要发送的消费事件”落到本地数据库。
- 业务事务提交成功后，再异步把这条事件发到 MQ。
- 避免“业务成功了，但消息没发出去”。

### Inbox

收件箱。由**消费方**写入。

作用：

- 消费者收到消息后，先按业务唯一键写入收件记录。
- 如果同一事件重复投递，第二次插入会因唯一键冲突而被跳过。
- 避免“MQ 重投导致重复发券”。

## 3. 它们如何保证只处理一次

严格说，不是靠名字保证，而是靠四层共同保证。

### 3.1 Outbox 保证“不丢”

业务数据和待发送事件放在同一个本地事务里提交。

只要业务成功，事件就一定在库里；如果发送器宕机，后面还能补发。

### 3.2 Inbox 保证“不重”

消费者先查/写 inbox，inbox 上有唯一索引。

同一条事件重复到达时，第二次会命中唯一键冲突，消费者直接认为“已处理”，不再执行业务。

### 3.3 业务唯一键保证“业务只生效一次”

不能只依赖 MQ messageId。

需要用业务键做幂等，例如：

`mid + sid + originSource + originLifecycleId + eventType + outOrderId + cardLid`

这样即使：

- MQ 重投
- 接口重试
- 人工补发

也只会命中同一笔业务。

### 3.4 状态机保证“可恢复”

Outbox 和 Inbox 都要有状态：

- Outbox：`PENDING -> SENT / FAILED`
- Inbox：`RECEIVED -> DONE / FAILED`

这样中途崩溃时，可以靠状态继续补发或重试，而不是靠人工猜。

## 4. 反结账如何回收券

反结账不能只按订单号模糊撤销，应该按“原消费赠券任务 + 赠券明细”回收。

推荐做法：

1. 反结账事件先找到原发券任务。
2. 根据原任务的赠券明细逐张处理。
3. 未使用券，直接回收。
4. 已使用券，按业务策略处理：
   - 阻断反结账
   - 允许反结账但生成异常单
   - 部分回收 + 补偿单

核心原则：

- 回收要有幂等键。
- 回收要有状态。
- 回收失败要能重试。
- 已使用券不能靠简单删除解决。

## 5. 一句话总结

`Outbox` 负责把事件安全送出去，`Inbox` 负责把重复消息挡在门外，业务唯一键和状态机负责让“消费送券”和“反结账回收券”最终都只生效一次。

后续实现统一以《消费完成当下赠券与赠现金网统一底座设计》为权威契约：

- 赠券和赠现金都在消费完成当下触发，异步消息、重试、补偿不改变业务发放时点
- 幂等键使用 `mid + sid + originSource + originLifecycleId + eventType + outOrderId + cardLid`
- `MANUAL_REPLAY` / `SYSTEM_RETRY` 只作为 `deliverySource`
- 所有状态、事件类型、来源、撤销策略都必须用主设计文档中的枚举
- 所有事件入口统一 `POST`
- 现金冲正默认不允许负余额，余额不足进入异常单

本说明只解释概念，不再单独定义另一套不同口径的键和状态。
