# 消费赠券企业级架构设计

日期：2026-05-12  
范围：`nms4cloud` CRM / Order / POS 消费完成当下赠券与反结账回收券链路  
目标：把“消费完成当下赠券”从可被调用的异步逻辑，升级为可审计、可重试、可幂等、可补偿、可回收的企业级闭环。

## 1. 当前结论

当前代码中已经存在消费赠券的部分能力：

- CRM 侧存在 `CRM_CONSUMER_COUPON_QUEUE` 消费者，收到消息后会按消费赠券规则筛选并调用发券逻辑。
- CRM 侧存在 `/consumerGiveCoupon` 外部签名接口，可以由 POS / 线下清台系统调用后进入 MQ。
- 订单服务里存在 `sendConsumeCoupon(...)` 方法，但线上结账路径中该调用被注释，注释说明“通过线下清台来赠送，不在此进行赠送”。

因此当前真实状态不是“所有消费都会自动赠券”，而是：

- 线上订单结账链路默认不触发消费赠券。
- 若 POS / 线下清台调用 CRM 的 `consumerGiveCoupon` 接口，才可能触发消费赠券。
- 发券后缺少独立的赠券任务台账、券明细关联、反结账回收闭环、可靠补偿和可观测性。

## 2. 设计目标

1. 消费完成当下，在配置允许、规则命中、会员信息有效的情况下触发发券；异步投递和补偿只保证可靠执行，不改变业务发放时点。
2. 同一笔消费重复通知、MQ 重投、接口重试时，只产生一次业务赠券效果。
3. 反结账、退款、撤销消费时，可以按原赠券任务回收券。
4. 回收失败、券已使用、消息失败等情况必须有明确状态、补偿任务和人工处理入口。
5. 规则变更不影响历史任务重试和反结账处理。
6. 保留现有 `/consumerGiveCoupon` 与 MQ 消费者能力，逐步迁移，不破坏旧 POS / 清台调用方。

## 3. 非目标

- 不改变优惠券模板本身的发放、核销、库存基本语义。
- 不直接重写现有 `couponOpAdd`、`couponOpRevoke` 等通用发券/回收能力。
- 不把消费赠券和消费赠积分混成同一张任务表。两者可参考同一套幂等/补偿模式，但业务台账应独立。

## 4. 总体架构

建议采用“事件驱动 + 业务任务台账 + 发券明细台账 + 补偿任务”的架构。

```text
订单/清台/POS
  |
  | 1. 消费成功事件 / 反结账事件
  v
消费赠券事件入口
  |
  | 2. Inbox 去重 + 业务事实校验
  v
消费赠券任务服务
  |
  | 3. 规则匹配 + 数量计算 + 任务落库
  v
发券执行器
  |
  | 4. 调用通用发券 couponOpAdd
  v
赠券明细台账
  |
  | 5. 反结账时按台账回收 couponOpRevoke
  v
回收任务 / 异常补偿 / 人工处理
```

核心思想：

- MQ 只保证“至少投递”，业务“只执行一次”由数据库唯一键和任务状态保证。
- 发券不是直接调用完就结束，必须记录“这笔消费命中了哪条规则、发了哪些券、每张券后续状态如何”。
- 反结账不重新算规则，而是回查原赠券任务和赠券明细，按历史事实回收。

## 5. 核心领域模型

### 5.1 消费赠券任务表

建议新增或重建专用表：`crm_consume_coupon_task`。

关键字段：

| 字段 | 说明 |
|---|---|
| `mid` | 商户 ID |
| `sid` | 门店 ID |
| `origin_source` | `ONLINE_ORDER` / `POS_CLEAR` |
| `delivery_source` | `NORMAL` / `MANUAL_REPLAY` / `SYSTEM_RETRY` |
| `event_type` | `CONSUME_COMPLETED` / `CONSUME_REVOKED` / `CONSUME_REPLAY` |
| `origin_lifecycle_id` | 原始消费生命周期 ID，POS 可用清台/round ID，线上订单可用订单 LID |
| `out_order_id` | 外部订单号 |
| `bill_lid` | 账单 LID，可空但 POS 场景建议必填 |
| `card_lid` | 会员卡 LID |
| `card_no` | 会员卡号快照 |
| `rule_lid` | 命中的消费赠券规则 |
| `rule_snapshot` | 规则快照 JSON |
| `amount_snapshot` | 金额快照 JSON |
| `coupon_plan_snapshot` | 本次应发券列表和数量 JSON |
| `status` | `PROCESSING` / `SUCCEEDED` / `FAILED` / `REVOKE_PENDING_GRANT` / `REVOKING` / `REVOKED` / `REVOKE_FAILED` |
| `retry_count` | 重试次数 |
| `next_retry_time` | 下次重试时间 |
| `error_code` / `error_message` | 最近失败原因 |
| `grant_task_lid` | 撤销任务关联的原发券任务 |

建议唯一键：

```text
uk_consume_coupon_task_grant:
mid + sid + origin_source + origin_lifecycle_id + event_type + out_order_id + card_lid
```

对于撤销任务：

```text
uk_consume_coupon_task_revoke:
mid + sid + origin_source + origin_lifecycle_id + event_type + grant_task_lid
```

### 5.2 消费赠券明细表

建议新增：`crm_consume_coupon_grant_detail`。

关键字段：

| 字段 | 说明 |
|---|---|
| `task_lid` | 原消费赠券任务 |
| `coupon_lid` | 券模板 LID |
| `coupon_order_lid` | 实际生成的会员券/券订单 LID |
| `coupon_code` | 如系统存在券码，记录券码 |
| `grant_status` | `PENDING` / `SUCCEEDED` / `FAILED`，对应 `CrmConsumeCouponGrantStatusEnum` |
| `revoke_status` | `NOT_REQUIRED` / `REVOKED` / `REVOKE_FAILED` / `USED_CANNOT_REVOKE`，对应 `CrmConsumeCouponRevokeStatusEnum` |
| `write_off_order_id` | 如已核销，记录核销订单 |
| `revoke_task_lid` | 关联回收任务 |
| `error_message` | 回收失败原因 |

这张表是反结账回收券的核心依据。不能只靠 `outOrderId` 去通用券表里模糊查，因为规则变更、重复发券、部分发券、跨门店核销都会让后续回收不可控。

### 5.3 消息 Inbox / Outbox

建议保留 MQ，但增加数据库可靠消息表。

订单/POS 侧：

- `consume_coupon_outbox`：业务事务内写入消费成功/反结账事件。
- 后台发布器异步发送 MQ。
- 发送成功后标记 SENT；失败保留重试。

CRM 侧：

- `consume_coupon_inbox`：按 `event_id` 或业务唯一键去重。
- 同一事件重复投递时直接返回已有任务结果。

## 6. 消费完成当下发券流程

### 6.1 事件产生

订单、POS、清台系统在消费完成当下产生 `ConsumeCouponGrantEvent`。

建议事件字段：

| 字段 | 说明 |
|---|---|
| `eventId` | 全局唯一事件 ID |
| `origin_source` | ONLINE_ORDER / POS_CLEAR |
| `delivery_source` | NORMAL / MANUAL_REPLAY / SYSTEM_RETRY |
| `mid` / `sid` | 商户、门店 |
| `origin_lifecycle_id` | 原始消费生命周期 ID |
| `outOrderId` | 外部订单号 |
| `billLid` | 账单 ID |
| `cardLid` / `cardNo` | 会员卡信息 |
| `checkoutTime` | 结账时间 |
| `billAmount` | 账单总额 |
| `crmAmount` | 会员卡支付金额 |
| `otherAmount` | 其他支付金额 |
| `online` | 是否线上订单 |
| `operator` | 操作人 |
| `traceId` | 链路追踪 ID |

事件发送要求：

- 不能在订单事务提交前直接发 MQ。
- 订单/POS 侧先写 outbox，事务提交后由发布器发送。
- 线上订单是否启用自动赠券必须由明确配置控制，不建议再依赖注释或隐式约定。
- Outbox / MQ 延迟只代表技术投递延迟，不代表赠券业务时点后移；任务和明细必须记录消费完成时间。

### 6.2 CRM 入站处理

CRM 消费者收到事件后：

1. 写入 inbox 或基于唯一键查询已有任务。
2. 校验配置开关。
3. 校验会员卡是否存在且属于当前商户。
4. 校验事件金额、订单号、门店、会员卡信息是否完整。
5. 创建发券任务，状态 `PROCESSING`。

如果唯一键冲突：

- 已成功：直接返回，不再发券。
- 处理中：不重复执行，等待重试或返回处理中。
- 失败：按策略允许补偿任务重试，不新建第二个业务任务。

### 6.3 规则匹配

规则匹配应独立成 `CrmConsumeCouponRuleMatcher`，不要写在 MQ 消费者里。

匹配维度：

- 商户、门店。
- 会员卡类型和等级。
- 固定有效期 / 相对有效期。
- 周几限制、时段限制。
- 金额类型：会员卡支付金额、非会员卡支付金额、账单总额。
- 满额门槛。
- 递增赠券规则。
- 单次赠券上限。

重要设计：

- 命中后把规则关键字段写入 `rule_snapshot`，后续重试和反结账都使用快照，不再重新匹配当前规则。
- 如果多条规则命中，必须有明确优先级。建议按 `fullAmount` 最大优先，其次规则优先级字段，其次创建时间。
- 如果需要“叠加赠送”，应显式建模为规则策略，不能由代码隐式只取一条。

### 6.4 发券执行

发券执行器 `CrmConsumeCouponGrantExecutor` 做三件事：

1. 根据规则快照计算应发券列表。
2. 调用现有 `couponOpAdd`，发券渠道使用 `ConsumeGive`。
3. 把实际生成的券写入 `crm_consume_coupon_grant_detail`。

调用 `couponOpAdd` 时必须传业务幂等号，且建议直接复用底座业务键，至少要包含 `mid` / `sid` / `originSource` / `originLifecycleId` / `eventType` / `outOrderId` / `cardLid`：

```text
consume-give:{mid}:{sid}:{originSource}:{originLifecycleId}:{eventType}:{outOrderId}:{cardLid}
```

正向发券场景下，`eventType` 固定为 `CONSUME_COMPLETED`。

如果现有 `couponOpAdd` 不支持返回券明细或幂等号，需要在外围补一层适配：

- 发券前记录任务和预期数量。
- 发券后按 `mid + sid + originSource + originLifecycleId + eventType + outOrderId + cardLid` 查询实际券明细。
- 查不到明细时任务不能标记成功，只能进入待补偿状态。

## 7. 反结账回收券流程

反结账不能重新按当前规则计算，也不能只按订单号模糊撤销。必须按原发券任务和赠券明细回收。

### 7.1 反结账事件

反结账、退款、撤销消费产生 `ConsumeCouponRevokeEvent`。

关键字段：

| 字段 | 说明 |
|---|---|
| `eventId` | 撤销事件 ID |
| `origin_source` | `ONLINE_ORDER` / `POS_CLEAR` |
| `delivery_source` | `NORMAL` / `MANUAL_REPLAY` / `SYSTEM_RETRY` |
| `mid` / `sid` | 商户、门店 |
| `origin_lifecycle_id` | 原消费生命周期 ID |
| `revoke_lifecycle_id` | 本次反结账生命周期 ID |
| `outOrderId` | 原订单号 |
| `cardLid` / `cardNo` | 原会员卡 |
| `reason` | 反结账原因 |
| `operator` | 操作人 |

### 7.2 回收任务创建

CRM 收到反结账事件后：

1. 按 `mid + sid + origin_source + origin_lifecycle_id + event_type(CONSUME_COMPLETED) + out_order_id + card_lid` 查原 `crm_consume_coupon_task`（原发券任务）。
2. 如果原任务不存在：
   - 若发券事件可能尚未到达，创建 `REVOKE_PENDING_GRANT` 状态，稍后重试。
   - 不建议直接返回成功，否则后续发券事件到达会发出无法回收的券。
3. 如果原任务存在，创建撤销任务。
4. 根据 `grant_detail` 逐张判断券状态。

### 7.3 回收策略

券状态分三类处理：

1. 未使用券：直接调用通用券回收能力，成功后标记 `REVOKED`。
2. 已过期但未使用券：仍应回收或标记无效，避免前端继续展示为历史可疑权益。
3. 已使用券：不能简单删除。必须走企业级异常策略。

已使用券建议支持三种业务策略，由商户配置决定：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| `BLOCK` | 存在已使用赠券时，不允许反结账成功 | 财务强一致要求高 |
| `EXCEPTION` | 原消费撤销成功，已使用赠券进入人工处理/追偿 | 门店体验优先 |
| `PARTIAL_COMPENSATE` | 未使用券自动回收，已使用券生成负向权益或补偿记录 | 连锁门店常用折中 |

推荐默认策略：

- POS 当场反结账：优先 `BLOCK`，提示“消费赠券已被使用，需人工处理”。
- 线上退款：优先 `EXCEPTION`，允许主退款继续，但生成 `CONSUME_COUPON_REVOKE_EXCEPTION` 异常单，财务/运营处理。

### 7.4 回收幂等

反结账重复请求时：

- 同一原赠券任务只能有一个成功的回收任务。
- 已回收券再次回收直接返回成功。
- 回收失败后允许同一任务重试，但不能新建第二套回收任务。

## 8. 配置设计

现有 `g_enableConsumerCoupon` 使用字母值，含义不够自解释。建议保留旧值兼容，同时引入枚举化配置。

建议新枚举：

| 新值 | 含义 |
|---|---|
| `OFF` | 不启用消费赠券 |
| `ONLINE_ONLY` | 仅线上订单启用 |
| `OFFLINE_ONLY` | 仅线下/POS 清台启用 |
| `ALL` | 线上、线下都启用 |

兼容适配：

- 读取旧配置值时在配置服务内转换为新枚举。
- 业务代码只比较枚举，不直接比较 `"N"`、`"F"`、`"X"`。
- 迁移期间保留旧配置写入，但新页面展示明确语义。

## 9. 代码设计

建议新增包结构：

```text
com.nms4cloud.crm.service.consume.coupon
  CrmConsumeCouponServicePlus
  CrmConsumeCouponGrantServicePlus
  CrmConsumeCouponRevokeServicePlus
  CrmConsumeCouponRuleMatcher
  CrmConsumeCouponGrantExecutor
  CrmConsumeCouponRevokeExecutor
  ICrmConsumeCouponTaskService
  ICrmConsumeCouponGrantDetailService
  CrmConsumeCouponCompensationJob

com.nms4cloud.crm.api.dto.consume.coupon
  CrmConsumeCouponGrantEventDTO
  CrmConsumeCouponRevokeEventDTO
  CrmConsumeCouponTaskQueryDTO
  CrmConsumeCouponTaskVO
```

职责划分：

| 类 | 职责 |
|---|---|
| `CrmConsumeCouponServicePlus` | 入口编排，处理 grant/revoke 事件 |
| `CrmConsumeCouponRuleMatcher` | 只负责规则匹配和数量计算 |
| `CrmConsumeCouponGrantServicePlus` | 创建赠券任务、维护任务状态 |
| `CrmConsumeCouponGrantExecutor` | 调用通用发券并落明细 |
| `CrmConsumeCouponRevokeServicePlus` | 创建回收任务、查原任务 |
| `CrmConsumeCouponRevokeExecutor` | 调用通用回收并处理已使用券策略 |
| `CrmConsumeCouponCompensationJob` | 扫描失败/超时任务并重试 |

MQ 消费者只做三件事：

1. 解析消息。
2. 调用 `CrmConsumeCouponServicePlus` 入口编排服务。
3. 根据入口编排服务结果决定 ACK / 重试 / 死信。

不建议继续把规则匹配、发券、异常吞掉都写在 MQ consumer 里。

## 10. 可靠性设计

### 10.1 消息可靠

- 订单/POS 侧使用 outbox，避免业务提交成功但 MQ 发送失败。
- CRM 侧使用 inbox，避免 MQ 重复投递导致重复发券。
- MQ 消费失败应抛出或返回失败，让 MQ 重试；不应只打日志吞异常。
- 超过最大重试进入死信或补偿表，必须有告警。

### 10.2 业务幂等

幂等不依赖 MQ messageId，必须依赖业务唯一键：

```text
mid + sid + originSource + originLifecycleId + cardLid + outOrderId + eventType
```

原因：

- 同一业务事件可能被多个系统重发。
- MQ messageId 变了不代表业务事件变了。
- POS 弱网重试、清台补偿、人工重推都必须命中同一业务键。

### 10.3 任务状态机

建议状态流：

```text
赠券任务状态（`CrmConsumeCouponTaskStatusEnum`）：

```text
PROCESSING -> SUCCEEDED
           -> FAILED
           -> REVOKE_PENDING_GRANT
           -> REVOKING -> REVOKED
                      -> REVOKE_FAILED
```

返现任务状态（`CrmConsumeCashTaskStatusEnum`）：

```text
PROCESSING -> SUCCEEDED
           -> FAILED
           -> REVERSE_PENDING_GRANT
           -> REVERSING -> REVERSED
                      -> REVERSE_FAILED
```

底座状态（`CrmConsumeBenefitEventStatusEnum`）：

```text
RECEIVED -> DISPATCHING -> DISPATCHED -> DONE
                             -> PARTIAL_FAILED
                             -> FAILED
```
```

所有状态变更记录操作时间、错误原因和操作者。

## 11. 反结账一致性策略

不同业务场景对一致性的要求不同，应在产品配置层明确。

### 11.1 强一致模式

反结账前同步检查赠券状态：

- 未使用：先回收券，再允许反结账。
- 已使用：阻断反结账。

优点：财务最干净。  
缺点：POS 体验受 CRM 可用性影响。

### 11.2 最终一致模式

反结账先成功，CRM 异步回收赠券。

优点：不阻塞收银。  
缺点：需要完善异常单、补偿任务和运营处理。

### 11.3 推荐模式

- POS / 线下清台：强一致优先，CRM 不可用时允许进入“待反结账补偿”但必须在 POS 明确展示。
- 线上订单退款：最终一致优先，生成回收任务和异常单，后台补偿。

## 12. 监控与运营后台

必须提供消费赠券任务查询页。

查询维度：

- 商户、门店、会员卡、订单号。
- 任务类型：发券 / 回收。
- 状态：成功、失败、处理中、待人工。
- 时间范围。
- 规则 LID、券模板 LID。

操作能力：

- 查看规则快照。
- 查看实际发券明细。
- 手动重试失败任务。
- 手动标记人工处理完成。
- 导出异常任务。

告警指标：

- `consume_coupon_grant_failed_count`

## 14. 迁移路径

### 阶段一：补台账，不改变入口

- 保留现有 `/consumerGiveCoupon` 和 `CRM_CONSUMER_COUPON_QUEUE`。
- 新增任务表和赠券明细表。
- CRM 消费者改为调用 `CrmConsumeCouponServicePlus` 入口编排服务。
- 先只对线下/POS 入口启用。

### 阶段二：补反结账回收

- 增加反结账事件和回收任务。
- 对未使用券自动回收。
- 对已使用券进入异常单。

### 阶段三：补可靠消息

- 订单/POS 侧引入 outbox。
- CRM 侧引入 inbox。
- 增加补偿 job 和死信处理。

### 阶段四：决定是否启用线上订单自动赠券

- 明确配置枚举。
- 在线上订单结账链路中按配置触发消费赠券。
- 上线前先灰度：指定商户、指定门店、指定订单来源。

## 15. 关键设计取舍

### 不建议只打开当前注释

直接恢复 `sendConsumeCoupon(order, orderPayList)` 的风险：

- 没有专用赠券任务台账。
- 反结账无法可靠回收。
- MQ 失败可能丢事件。
- MQ 重投可能造成重复发券，取决于下游发券幂等是否完整。
- 规则变更后历史重试可能按新规则发券。

### 不建议只依赖 outOrderId 回收

同一个订单可能存在：

- 多会员卡。
- 多次清台补偿。
- 多张赠券。
- 部分券已使用。
- 规则变更。

只靠 `outOrderId` 回收会造成边界不清，应以赠券明细台账为准。

### 推荐先保证线下，再开放线上

当前代码注释已经表达“通过线下清台来赠送”。如果业务仍然坚持这个边界，应先把 POS / 清台链路做成完整闭环，再评估线上订单是否接入。

## 16. 最终推荐方案

推荐采用：

```text
Outbox 可靠事件
+ CRM Inbox 去重
+ 消费赠券任务表
+ 赠券明细台账
+ 规则快照
+ 发券幂等号
+ 反结账回收任务
+ 失败补偿与人工异常单
```

这样能达到企业级要求：

- 消费完成当下命中规则时应发必发。
- 重复通知不会重复发。
- 发券失败可以补偿。
- 反结账可以找到原券并回收。
- 已使用券有明确业务策略。
- 所有动作可查询、可审计、可告警。
