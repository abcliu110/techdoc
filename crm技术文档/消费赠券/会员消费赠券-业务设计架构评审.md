## 2. 结论摘要

结论：当前“会员消费赠券”**不满足可上线的完整闭环标准**。

核心原因不是单点 bug，而是整条链路同时存在以下问题：

1. 主业务入口未真正接通
2. MQ 消费失败会静默丢失
3. 发券没有消费赠券专属幂等保护
4. 门店 `sid` 维度没有在规则命中和部分统计/撤销中正确生效
5. 规则删除存在明确实现错误
6. 缺少消费赠券链路自动化测试

综合判断：

- 业务评审结果：`阻断`
- 设计评审结果：`阻断`
- 架构评审结果：`阻断`

## 3. 业务评审

### 3.1 业务闭环是否成立

当前业务闭环**不成立**。

按一个合理的消费赠券闭环，应至少包含：

1. 会员消费成功
2. 命中正确的消费赠券规则
3. 只发一次券
4. 发券失败可重试或可补偿
5. 撤销消费时只撤销本次消费赠出的券
6. 门店规则只作用于对应门店

而当前源码中：

- 线上支付成功后，订单侧本应触发消费赠券，但调用已被注释
- CRM 侧虽然有 `consumerGiveCoupon` 入口和 MQ 消费者，但仓内看不到稳定、统一的内部成功链路调用它
- 发券失败只记录日志，不进入失败任务或补偿流程

因此，当前更像“存在若干可调用能力”，而不是“系统已经完成业务闭环”。

### 3.2 业务阻断问题

#### 问题 1：线上主链路未触发消费赠券

证据：

- `PayOrderServiceImpl.java:900-901`

说明：

- `sendShare(order);` 正常执行
- `sendConsumeCoupon(order, orderPayList);` 被直接注释
- 注释文字明确写了“通过线下清台来赠送，不在此进行赠送”

业务影响：

- 在线上自助结账场景下，会员消费成功后不会自动赠券
- 如果外部没有额外补调 CRM 接口，则赠券根本不会发生

#### 问题 2：CRM 内没有稳定的自闭环成功后处理入口

证据：

- `CrmCardOpController.java:174-178`
- `CrmCardOpServicePlus.java:2056-2082`

说明：

- 当前 `consumerGiveCoupon` 更像外部接口入口
- 仓内没有明确、稳定的“消费成功 -> 一定调用 consumerGiveCoupon”的统一主路径

业务影响：

- 功能依赖上游系统额外配合
- 一旦上游没有按约调用，CRM 内部不会自动补齐

## 4. 设计评审

### 4.1 设计完整性结论

当前设计不完整，主要缺失：

- 统一业务键
- 消费赠券专属任务状态
- 明确幂等策略
- 明确失败补偿策略
- 明确门店隔离策略

### 4.2 设计问题清单

#### 问题 3：消费赠券没有专属幂等设计

证据：

- `CrmConsumerCouponQueueConsumer.java:237-248`
- `CrmCouponOpServicePlus.java:102-153`
- `CrmCouponOpServicePlus.java:318`

说明：

- `outOrderId` 被传递到发券服务，并写入 `CrmCouponOrder.orderBillId`
- 但发券前没有看到基于 `orderId + channel + member` 的查重
- 也没有消费赠券专属任务表或唯一约束兜底

设计问题本质：

- `orderId` 只是“落库字段”
- 不是“幂等键”

结果：

- MQ 重投
- 人工重放
- 上游重复调用

都可能导致重复发券。

#### 问题 4：规则和限制缺少清晰的门店隔离设计

证据：

- `CrmConsumptionCouponRule.java:45-49`
- `CrmConsumerCouponQueueConsumer.java:89-101`
- `CrmConsumerCouponQueueConsumer.java:140-192`

说明：

- 规则实体和限制实体都有 `sid`
- 但消费执行时查询限制和规则都主要按 `mid` 查
- 没有看到明确的 `sid` 过滤或“总部规则 / 门店规则优先级回退”设计

设计风险：

- A 店结账可能命中 B 店规则
- 总部规则和门店规则优先级不明确

#### 问题 5：规则删除联动设计存在实现错误

证据：

- `CrmConsumptionCouponRuleServicePlus.java:227-230`
- `CrmConsumptionCouponLimit.java:63-64`

说明：

- 删除规则后，代码按 `CrmConsumptionCouponLimit.lid in request.getLids()` 删除限制
- 但限制表关联规则的字段是 `ruleLid`

结果：

- 删除规则后，对应限制记录大概率留存
- 形成孤儿数据

这不是优化问题，而是明确实现错误。

## 5. 架构评审

### 5.1 架构结论

当前消费赠券链路没有形成可靠的事件驱动闭环。

理想结构应为：

1. 消费成功产生业务事件
2. 事件持久化并带唯一业务键
3. 消费者幂等处理
4. 发券成功 / 失败状态可审计
5. 可按状态重试或人工补偿
6. 撤销时按来源单号 + 渠道 + 门店精确回滚

而当前结构更接近：

`某些入口 -> 直接发 MQ -> 消费者直接发券 -> 异常吞掉`

这是典型的 `fire-and-forget` 风格，不适合营销权益发放。

### 5.2 架构问题清单

#### 问题 6：MQ 消费失败被吞掉，链路不可靠

证据：

- `CrmConsumerCouponQueueConsumer.java:250-252`

说明：

- 捕获异常后只打日志
- `throw e` 被注释

架构影响：

- RocketMQ 不会重试这类失败
- 消费赠券会静默丢失
- 系统中也没有失败任务状态给后续补偿

#### 问题 7：撤销链路没有按渠道和门店收口

证据：

- `CrmCardOpController.java:160-168`
- `CrmCouponOpServicePlus.java:55-73`
- `CrmCouponOrder.java:170-176`

说明：

- 撤销时主要按 `mid + phone + orderId + status` 更新券单
- 没有限制 `getChannel = ConsumeGive`
- 也没看到基于 `sid` 的精确过滤

架构风险：

- 同一订单号下其他来源券单可能被误撤
- 多门店数据可能被串撤

#### 问题 8：消费者从“当前时间”开始消费，不利于历史恢复

证据：

- `CrmConsumerCouponQueueConsumer.java:277-280`

说明：

- `prepareStart` 中将消费起点设置为 `CONSUME_FROM_TIMESTAMP`
- 这类设置对历史积压恢复不友好

架构风险：

- 新 consumer group 首次启动
- offset 异常
- 历史消息补偿

这些场景下恢复能力偏弱。

## 6. 测试评审

### 6.1 当前测试现状

当前没有找到消费赠券链路对应的有效自动化测试覆盖。

对比发现：

- 消费赠分已有幂等测试
- 但消费赠券没有对应的 queue / service / rule CRUD / revoke 测试

### 6.2 缺失的关键测试

至少缺少以下测试：

1. 重复消息不重复发券
2. 同一订单只发一次 `ConsumeGive`
3. 规则按 `sid` 正确命中
4. 撤销只撤本次消费赠券
5. 删除规则时同步删除限制
6. 库存不足时的失败行为
7. MQ 消费异常时的重试或补偿行为

## 7. 问题清单汇总

### 7.1 阻断问题

1. 线上主链路未接通，消费成功不自动赠券
2. MQ 消费失败被吞掉，失败后无重试无补偿
3. 发券没有消费赠券专属幂等保护
4. CRM 内部没有稳定、自闭环的“消费成功后自动触发消费赠券”主路径，当前更依赖外部接口或额外链路配合
5. 撤销/补偿不是 race-safe，“先撤销、后发券”会静默失效

### 7.2 高风险问题

1. 规则命中忽略 `sid`，存在跨门店串用风险
2. 限领统计忽略 `sid`，存在跨门店串算风险
3. 撤销忽略 `getChannel` / `sid`，存在误撤风险
4. 删除规则时限制删除条件写错字段
5. 无手机号会员卡在 CRM 发券层会直接失败，且失败被上层静默吞掉
6. “单次最多赠券数量”字段疑似在消费者查询阶段未查出，封顶规则可能失效
7. 周几 + 时段限制的判断实现有缺陷，同一条限制同时配置两类条件时，时间条件可能被跳过
8. 消费者从“当前时间”开始消费，offset 丢失或新组启动时恢复能力弱
9. 缺少消费赠券自动化测试

## 8. 补充问题说明

### 8.1 无手机号会员卡会导致静默漏发

证据：

- `CrmCouponOpServicePlus.java:108-116`

说明：

- 发券时强依赖 `card.getPhone()`
- 只有 `openid.length() == 11` 才会被当作手机号兜底
- 对正常微信 `openid` 基本不成立

结果：

- 会员卡如果没有手机号，消费赠券会在 CRM 发券层抛错
- 而消费赠券消费者上层又会吞掉异常
- 最终表现为静默漏发

### 8.2 “单次最多赠券数量”可能失效

证据：

- `CrmConsumerCouponQueueConsumer.java:143-149`
- `CrmConsumerCouponQueueConsumer.java:222-231`
- `CrmConsumptionCouponRule.java:72-73`

说明：

- 规则查询时没有把 `giftMaxQuantity` 选出来
- 但后续又使用 `giftMaxQuantity` 做赠券数量封顶

风险：

- 运行时该字段可能是 `null/0`
- 梯度赠券场景下配置的“单次最多赠券数量”可能不生效

### 8.3 周几 + 时段限制判断有逻辑缺陷

证据：

- `CrmConsumerCouponQueueConsumer.java:110-128`
- `CrmConsumptionCouponRuleServicePlus.java:261-276`

说明：

- 消费者侧用 `if ... else if ...` 处理周几和时间段
- 规则维护侧允许同一条限制记录同时配置 `weekType` 和 `start/endTime`

风险：

- 业务本意可能是“周几且在某个时间段”
- 运行时却可能只校验了其中一半

结果：

- 规则命中变宽
- 出现本不该发券的时段仍发券

### 8.4 撤销抢跑会导致闭环断裂

证据：

- `CrmCouponOpServicePlus.java:55-73`
- `CrmCardOpController.java:160-169`
- `CardOpForCustomerController.java:861-868`
- `CrmConsumerCouponQueueConsumer.java:237-248`

说明：

- 撤销逻辑只是直接按条件更新 `crm_coupon_order`
- 没有校验更新条数
- 没有“未发先撤”的挂起状态
- 控制器即使 0 行命中也直接返回成功

风险：

- 撤销请求先到
- 正向发券消息后到

这种乱序下，撤销会先“成功返回”，但券随后仍然发出来。

### 8.5 从当前时间开始消费，历史恢复能力偏弱

证据：

- `CrmConsumerCouponQueueConsumer.java:278-281`

说明：

- 消费者启动时将消费起点设为 `CONSUME_FROM_TIMESTAMP`
- 时间值取当前启动时刻

风险：

- 新 consumer group 首次启动
- broker offset 丢失
- 需要回放历史消息

这些场景下，历史待补消息可能被直接跳过。

## 9. 根因总结

根因不是某一行代码写错，而是消费赠券没有被设计成“可审计、可幂等、可补偿”的权益任务。

当前链路缺少：

- 统一业务主键
- 持久化任务状态
- 明确失败状态
- 明确补偿入口
- 明确门店隔离策略

`orderId` 目前只是普通字段，不是系统真正依赖的幂等控制点。

## 10. 建议方向

### 9.1 第一优先级

1. 建立消费赠券专属任务状态
2. 使用统一底座定义的业务幂等主键 `mid + sid + originSource + originLifecycleId + eventType + outOrderId + cardLid`
3. MQ 消费失败不能吞异常，必须可重试或可补偿

### 9.2 第二优先级

1. 规则和限制命中统一纳入 `sid`
2. 明确总部规则与门店规则优先级
3. 撤销强制按 `sid + getChannel + orderBillId` 精确过滤

### 9.3 第三优先级

1. 修正规则删除联动逻辑
2. 补齐消费赠券自动化测试

## 11. 本次评审结论

当前“会员消费赠券”不能认定为已完整实现。

更准确的判断是：

> CRM 内部已经存在部分规则和发券能力，但业务入口、可靠性、幂等、门店隔离、删除联动和测试保护都未收口，因此整体上仍属于未完成闭环的高风险实现。
