# 消费赠券规则命中算法详解

本文说明 POS 普通消费结账后，CRM 如何判断是否赠券、命中哪一条消费赠券规则，以及最终赠送几张券。

## 1. 总体流程

消费赠券不是只看某一条页面配置，而是按下面链路逐层判断：

1. POS 普通消费结账成功。
2. POS 确认账单已绑定会员。
3. POS 汇总本次账单金额和支付方式金额。
4. POS 调用 CRM 消费赠券接口。
5. CRM 根据会员、门店、有效期、金额口径匹配消费赠券规则。
6. CRM 根据命中的规则计算赠券数量。
7. CRM 调用通用发券服务生成券实例。
8. BI 报表从券实例记录中展示券发行明细。

## 2. POS 传给 CRM 的三个金额

POS 会把同一笔消费拆成三个金额口径传给 CRM：

| 代码 | 中文含义 | 说明 |
|---|---|---|
| H | 会员卡支付金额 | 本次结账中使用会员卡余额支付的金额。 |
| F | 非会员卡支付金额 | 现金、微信、支付宝等非会员卡支付金额。 |
| B | 账单整单金额 | 本次账单总消费金额。 |

示例：

| 支付方式 | 账单总额 | 会员卡支付 | 非会员卡支付 |
|---|---:|---:|---:|
| 全部会员卡支付 | 100 | 100 | 0 |
| 全部现金支付 | 100 | 0 | 100 |
| 混合支付 | 100 | 30 | 70 |

## 3. CRM 的金额口径匹配顺序

CRM 匹配规则时，固定按下面顺序查找：

```text
H（会员卡支付金额） -> F（非会员卡支付金额） -> B（账单整单金额）
```

具体含义：

1. 如果 `H（会员卡支付金额）` 大于 0，先找 `amountMethod = H` 的规则。
2. 如果没有 H 规则命中，并且 `F（非会员卡支付金额）` 大于 0，再找 `amountMethod = F` 的规则。
3. 如果 H 和 F 都没有命中，最后才找 `amountMethod = B` 的规则。

因此，会员卡支付时并不是优先看整单金额 B。只要 H 规则命中，B 规则不会再参与。

混合支付也是一样：先用会员卡支付金额匹配 H，再用非会员卡支付金额匹配 F，最后才用整单金额匹配 B。

## 4. 规则基础过滤条件

CRM 在匹配规则前，会先过滤掉不适用的规则：

| 判断项 | 命中要求 |
|---|---|
| 商户 | 规则必须属于当前商户。 |
| 门店 | 当前门店必须在规则适用范围内，或规则配置为全部门店。 |
| 会员类型 | 规则未限制会员类型，或会员类型与当前会员一致。 |
| 会员等级 | 规则未限制会员等级，或会员等级与当前会员一致。 |
| 有效期 | 当前结账时间必须在规则有效期内。 |
| 星期限制 | 如果规则配置了星期限制，当前日期必须满足。 |
| 时段限制 | 如果规则配置了时段限制，当前结账时间必须满足。 |
| 金额口径 | 当前正在匹配 H、F 或 B 时，只看对应金额口径的规则。 |
| 满额门槛 | 当前金额必须大于等于规则的 `fullAmount（满额门槛）`。 |

只有通过这些过滤的规则，才会进入最终命中选择。

## 5. 同一金额口径下如何选择规则

同一个金额口径下，如果有多条规则都满足，CRM 会优先选择 `fullAmount（满额门槛）` 最大的一条。

示例：当前匹配 B（账单整单金额），账单金额为 100。

| 规则 | 金额口径 | 满额门槛 | 赠送方式 |
|---|---|---:|---|
| 规则 A | B（账单整单金额） | 50 | 每满 50 送 1 张，最多 2 张 |
| 规则 B | B（账单整单金额） | 100 | 固定送 1 张 |

这时两条规则都满足金额条件，但 CRM 会优先选择满额门槛更高的规则 B。

结果是：实际固定送 1 张，而不是按规则 A 计算 2 张。

## 6. 赠券数量如何计算

命中规则后，CRM 才计算赠券数量 `couponNum`。

### 6.1 固定赠送

如果没有开启“每满递增”，赠券数量直接等于规则配置的赠送数量：

```text
couponNum = giftQuantity（赠送数量）
```

示例：

```text
满 100 送 1 张
消费金额 100
couponNum = 1
```

### 6.2 每满递增

如果开启“每满递增”，按命中的金额口径除以 `giftEveAmount（每满金额）` 向下取整，再乘以 `giftQuantity（每档赠送数量）`：

```text
couponNum = floor(命中金额 / giftEveAmount（每满金额）) * giftQuantity（每档赠送数量）
```

如果配置了 `giftMaxQuantity（最大赠送数量）`，还要再封顶：

```text
couponNum = min(couponNum, giftMaxQuantity（最大赠送数量）)
```

示例：

```text
金额口径：B（账单整单金额）
账单金额：100
满额门槛 fullAmount：50
每满金额 giftEveAmount：50
每档赠送 giftQuantity：1
最大赠送 giftMaxQuantity：2

couponNum = floor(100 / 50) * 1 = 2
couponNum = min(2, 2) = 2
```

注意：`fullAmount（满额门槛）` 用于判断规则能不能命中；`giftEveAmount（每满金额）` 用于计算每满递增的档数。

## 7. 券模板校验

计算出 `couponNum` 后，CRM 会调用通用发券服务。消费赠券仍然受券模板限制。

通用发券会校验：

| 券模板条件 | 说明 |
|---|---|
| 已审核 | 未审核券不能发。 |
| 未终止 | 已终止券不能发。 |
| 领取时间 | 当前时间必须在券模板可领取时间内。 |
| 库存 | 剩余库存必须大于等于本次发券数量。 |
| 每人每日限领 | 如果配置了每日限领，同一会员当天不能超过限制。 |
| 每人总限领 | 如果配置了总限领，同一会员累计不能超过限制。 |

消费赠券不会跳过会员每日限领和总限领。

储值方案赠券有特殊标记，可以跳过券模板的单人日限领和单人总限领；消费赠券没有这个特殊标记。

## 8. 正常发券结果

如果最终传入通用发券服务的 `couponNum = 2`，普通券正常会生成 2 条券实例记录。

也就是说，如果只看到 1 张券，需要优先检查：

1. CRM 实际命中了哪条消费赠券规则。
2. CRM 实际计算出的 `couponNum` 是多少。
3. 券模板是否因库存、限领、时间、状态导致发券失败。
4. BI 报表是否漏查或过滤了部分券实例。

## 9. 排查时优先看的数据

### 9.1 CRM 消费赠券幂等记录

优先查 `crm_consume_coupon_op_record`。

重点字段：

| 字段 | 含义 |
|---|---|
| `order_id` | POS 订单号。 |
| `round_key` | POS 本地消费赠券轮次幂等键。 |
| `card_no` / `phone` | 会员标识。 |
| `status_` | CRM 处理状态。 |
| `rule_lid` | 实际命中的消费赠券规则。 |
| `coupon_lid` | 实际发放的券模板。 |
| `coupon_num` | CRM 实际计算出的赠券数量。 |
| `coupon_order_lids_json` | 本轮生成的券实例编号。 |
| `request_json` | POS 传入的金额快照，包括 H、F、B。 |
| `error_msg` | 发券失败原因。 |

建议 SQL：

```sql
SELECT order_id,
       round_key,
       card_no,
       phone,
       status_,
       rule_lid,
       coupon_lid,
       coupon_num,
       coupon_order_lids_json,
       request_json,
       response_json,
       error_msg,
       created_time,
       updated_time
FROM crm_consume_coupon_op_record
WHERE phone = '会员手机号'
ORDER BY created_time DESC
LIMIT 30;
```

### 9.2 券实例记录

再查 `crm_coupon_order`。

重点字段：

| 字段 | 含义 |
|---|---|
| `lid` | 券号。 |
| `coupon_code` | 券模板编号。 |
| `unionid` | 会员手机号。 |
| `get_channel` | 发券渠道，消费赠券通常为消费赠送。 |
| `order_bill_id` | 关联订单号。 |
| `get_date` | 发券时间。 |
| `coupon_status` | 未核销、已核销、已作废等状态。 |

建议 SQL：

```sql
SELECT lid,
       coupon_code,
       unionid,
       get_channel,
       order_bill_id,
       get_date,
       coupon_status
FROM crm_coupon_order
WHERE unionid = '会员手机号'
  AND coupon_code = '券模板编号'
ORDER BY get_date DESC
LIMIT 50;
```

## 10. 本次“期望 2 张但实际 1 张”的原因模型

本次排查中，券模板本身不支持“只能发 1 张”的解释：

- 券模板有剩余库存。
- 没有每日限领。
- 总限领不是 1。
- 券已审核、未终止、在领取期内。

更符合现象的原因是规则命中不符合 E2E 期望。

E2E 期望命中：

```text
B（账单整单金额）满 50
开启每满递增
每满 50 送 1 张
最多 2 张
账单金额 100
期望 2 张
```

但 CRM 实际可能优先命中了另一条同为 B 的规则：

```text
B（账单整单金额）满 100
固定送 1 张
```

因为 CRM 在同一金额口径下优先选择满额门槛更高的规则，所以满 100 的固定赠 1 张规则会压过满 50 每满递增规则。

这时最终表现就是：账单金额 100，但只发 1 张券。

## 11. 结论

消费赠券最终命中不是简单按“我刚配置的规则”计算，而是受以下因素共同影响：

1. 支付金额口径优先级：`H（会员卡支付金额） -> F（非会员卡支付金额） -> B（账单整单金额）`。
2. 会员类型、会员等级、门店、有效期、星期、时段过滤。
3. 同一金额口径下优先命中满额门槛最高的规则。
4. 命中后才按固定赠送或每满递增计算赠券数量。
5. 最后还要经过券模板库存、状态、领取时间、限领校验。

因此排查赠券数量不对时，不能只看页面上的某一条规则配置，必须结合 `crm_consume_coupon_op_record.rule_lid` 和 `coupon_num` 确认 CRM 实际命中的规则和实际计算出的张数。
