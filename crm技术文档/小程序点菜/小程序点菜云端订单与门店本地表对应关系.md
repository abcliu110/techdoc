# 小程序点菜云端订单与门店本地表对应关系

## 一句话结论

微信小程序点菜不是把云端 `order_*` 表直接同步到门店 `dwd_*` 表。实际链路是：

1. 小程序先把菜品写入云端 Redis 购物车。
2. 创建订单时，云端把购物车计算成正式订单，落云端 `order_bill`、`order_food`、`order_taste`，付款后落 `order_pay` 和支付中心 `pay_order`。
3. 云端把 `OrderMsgDTO` 消息推到门店。
4. 门店 POS 收到 `DO_ORDER` 或 `CASH_PAY` 消息后，再本地创建/更新 `dwd_bill`、`dwd_food`、`dwd_pay` 等表。

所以排查时要把三个编号分清：

| 名称 | 所在侧 | 典型字段 | 生成/用途 |
| --- | --- | --- | --- |
| 云端订单号 | 云端订单库 | `order_bill.saas_order_key` | 小程序下单时云端生成，消息 `msgId` 和 `data.id` 通常使用它。 |
| 门店本地营业单号 | 门店本地库 | `dwd_bill.saas_order_key` | POS 本地开台/开单时重新生成，规则是营业日期 + 操作类型 + 4 位流水。 |
| 云端与门店关联号 | 门店本地库 | `dwd_bill.other_order_key`、外卖/自提的 `dwd_bill.takeout_order_no` | 门店处理云端消息后写入云端 `saas_order_key`，用于回查关联。 |

这也是“某个订单号比其他订单号长”的常见原因：它很可能是云端 `saas_order_key`，由 `IdWorkerPlus.getIdStr()` 生成，长度会明显长于门店本地按营业日生成的 `yyyyMMdd + opType + 0001` 风格单号。

## 消息如何从云端传到门店

消息不是数据库同步，而是业务消息推送。

```text
小程序
  -> nms4cloud-order /order_bill/crt_order, /pay_order
  -> OrderMsgDTO
  -> OrderUtil.sendMessageToShop
  -> mqMqttMsgFeign.sendMqttMsg
  -> 门店 POS ActiveMQ/JMS topic SCAN_ORDER_MSG:DO_ORDER 或 SCAN_ORDER_MSG:CASH_PAY
  -> OnlineOrderActiveMQListener
  -> DoOrderHandler / CashPayHandler
  -> 本地 dwd_bill / dwd_food / dwd_pay
```

`OrderUtil.sendMessageToShop` 对消息做统一包装：

| 消息字段 | 赋值 |
| --- | --- |
| `OrderMsgDTO.time` | 当前时间格式化。 |
| `OrderMsgDTO.type` | `msgType.name().toLowerCase().replace("_", "-")`。 |
| `OrderMsgDTO.timeout` | 默认当前时间 + 1 小时。 |
| `MqMqttMsgAddDTO.mid/sid` | `orderMsg.mid/sid`。 |
| `MqMqttMsgAddDTO.msgId` | `orderMsg.msgId`。 |
| `MqMqttMsgAddDTO.msg` | `OrderMsgDTO` JSON。 |
| `MqMqttMsgAddDTO.msgType` | `ORDER`。 |
| `needConfirm` | 由消息类型决定是否需要确认。 |

代码证据：`OrderUtil.java:36-71`。

云端发送消息的关键场景：

| 场景 | 消息类型 | `msgId` | `data.id` | 代码证据 |
| --- | --- | --- | --- | --- |
| 堂食后付创建订单后 | `DO_ORDER` | `order.saasOrderKey` | `order.saasOrderKey` | `CrtPostOrderServiceImpl.java:241-255` |
| 先付订单支付完成后先下单 | `DO_ORDER` | `order.saasOrderKey` | `order.saasOrderKey` | `PayOrderServiceImpl.java:647-660` |
| 后付订单支付完成后结账 | `CASH_PAY` | `order.saasOrderKey` | `order.saasOrderKey` | `PayOrderServiceImpl.java:665-678` |

门店 POS 侧监听三个队列：

| 队列 | 处理器 | 作用 |
| --- | --- | --- |
| `SCAN_ORDER_MSG:DO_ORDER` | `DoOrderHandler` | 把云端订单拉到门店并点菜。 |
| `SCAN_ORDER_MSG:CASH_PAY` | `CashPayHandler` | 把云端支付结果同步到门店并结账。 |
| `SCAN_ORDER_MSG:CASH_REQUEST` | `CashRequestHandler` | 扫码拉单。 |

`OnlineOrderActiveMQListener` 收到消息后会解析 JSON 为 `OrderMsgDTO`，把 `mid/sid` 取绝对值，然后用 `msgId` 加锁调用处理器。

代码证据：`OnlineOrderActiveMQListener.java:27-67`。

## 与基础资料 Canal 同步的区别

这条小程序点菜链路不是基础资料同步。

基础资料同步一般是把商品、门店、配置等资料从云端同步到本地，而小程序订单是通过 `OrderMsgDTO` 消息触发门店业务处理器，再由门店代码按 POS 规则生成本地 `dwd_*` 数据。

证据点：

- 云端订单发出的是 `MqMqttMsgAddDTO`，`msgType=ORDER`，消息体是 `OrderMsgDTO` JSON，见 `OrderUtil.java:62-71`。
- 门店监听的是 `SCAN_ORDER_MSG:DO_ORDER`、`SCAN_ORDER_MSG:CASH_PAY`，见 `OnlineOrderActiveMQListener.java:27-40`。
- 门店业务处理器显式调用本地下单和付款逻辑，见 `DoOrderHandler.java:163-214`、`CashPayHandler.java:297-587`。

## 排查建议

### 查云端订单

用云端订单号查：

```sql
select * from order_bill where saas_order_key = '云端订单号';
select * from order_food where saas_order_key = '云端订单号';
select * from order_taste where saas_order_key = '云端订单号';
select * from order_pay where saas_order_key = '云端订单号';
```

如果是支付问题，还要查支付中心：

```sql
select * from pay_order where out_order_no = '云端订单号' or order_no = '云端订单号';
```

具体字段名以支付库实际表结构为准。

### 查门店本地订单

优先用云端订单号在本地查关联字段：

```sql
select * from dwd_bill
where other_order_key = '云端订单号'
   or takeout_order_no = '云端订单号';
```

查到本地单头后，再用本地 `lid/saas_order_no/saas_order_key` 查明细：

```sql
select * from dwd_food where saas_order_no = 本地dwd_bill.saas_order_no;
select * from dwd_pay where saas_order_no = 本地dwd_bill.lid;
```

代码里 `CashPayHandler` 查询 `dwd_pay` 使用的是：

```java
.eq(DwdPay::getMid, mid)
.eq(DwdPay::getSaasOrderNo, dwdBillLid)
```

证据：`CashPayHandler.java:297-301`。

### 查消息是否到门店

门店会把消息处理状态落 `dwd_order_msg`：

```sql
select * from dwd_order_msg where id = '云端订单号';
```

字段含义：

| 字段 | 说明 |
| --- | --- |
| `id` | `OrderDataDTO.id`，通常就是云端 `saas_order_key`。 |
| `content` | 原始 `OrderMsgDTO` JSON。 |
| `revision` | 处理阶段/状态，代码里 `revision == 2` 表示已经处理过。 |
| `saas_order_key` | 门店本地生成的 `dwd_bill.saas_order_key`。 |
| `table_name` | 门店本地单名/桌台名。 |

代码证据：`DoOrderHandler.java:92-121`、`:215-220`。
