# 会员消费积分企业级重做方案

日期：2026-05-10

## 1. 背景

当前 POS 消费积分已经有一版本地补偿实现，但模型仍以 `crm_consume_points_grant` 单表承载发放、撤销、补偿和退款重算状态。该模型可以覆盖简单结账、反结账和部分晚回包场景，但面对多轮部分退款、反结账后重结账、跨系统超时和审计对账时，生命周期边界仍不够清晰。

本轮为本地开发阶段，尚未上线，因此不做历史数据迁移，不做旧表兼容，不保留旧消费积分补偿表数据。旧表不用的直接删除。

本轮暂缓处理老储值链路切断：

```text
暂缓：
  不改 CardBalanceService
  不删 CrmDealTask.givePoint
  不承诺本轮彻底治理老储值链路双退
```

因此本轮目标是重建“新 POS 消费积分”自身的生命周期和账务模型，确保新链路内部不重复赠分、不重复撤销、可补偿、可审计。

## 2. 设计目标

本轮必须覆盖：

```text
1. 正常结账只赠一次积分。
2. 重复结账或补偿重试不重复赠分。
3. 反结账或全额退款只撤一次积分。
4. grant 未返回时先反结账，晚回包不能覆盖撤销意图。
5. 部分退款支持多轮撤旧重发。
6. CRM 超时、POS 重启后可以继续补偿。
7. 正负积分流水可以强关联。
8. 新 POS 消费积分路径不主动写 CrmDealTask.givePoint。
```

本轮不覆盖：

```text
1. 老 CardBalance 反结账是否还会基于历史 givePoint 退积分。
2. 老储值交易链路的彻底双退治理。
3. 会员账户、PLUS 汇总、BI 报表全量口径迁移。
```

## 3. 总体模型

最终模型采用：

```text
POS:
  crm_consume_points_session
    订单 + 会员 + 来源维度的总生命周期。

  crm_consume_points_round
    一轮 grant/revoke 生命周期。
    部分退款时关闭旧 round，再创建新 round。

  crm_consume_points_event
    弱事件、审计、唤醒。
    不作为最终状态源。

CRM:
  crm_consume_points_task
    只做 GRANT / REVOKE 幂等账务任务。

  crm_card_points_record
    积分账本。
    负向流水通过 related_points_record_lid 指向原正向流水。
```

核心原则：

```text
session.lid = 订单会员维度生命周期根。
round.lid = 一轮积分权益生命周期 ID，也是 CRM lifecycle_id。
event 不参与最终状态判断。
POS 补偿任务扫描 round，不扫描 event。
CRM task 不判断 POS 生命周期，只保证同一 round 的 grant/revoke 幂等。
CRM 必须同事务写 crm_card.points、crm_card_points_record、crm_consume_points_task。
```

## 4. 为什么必须引入 round

如果只有一条 session 状态，部分退款多轮场景会混乱：

```text
原订单现金 220，赠分 +2200
第一次部分退款后剩余现金 110：
  撤销 -2200
  重发 +1100

第二次部分退款后剩余现金 50：
  撤销 -1100
  重发 +500
```

正确生命周期应表达为：

```text
session 1
  round 1: +2200 -> -2200
  round 2: +1100 -> -1100
  round 3: +500
```

因此：

```text
session 表示整张订单同一会员的消费积分生命周期。
round 表示当前有效的一轮权益。
同一 session 下只能有一个 active round。
新 round 必须等旧 active round 撤销成功后再创建。
```

## 5. 职责边界

### 5.1 POS session

`crm_consume_points_session` 是订单会员维度的生命周期根。

它只回答：

```text
这张订单、这张会员卡、这个来源，是否还存在消费积分生命周期。
当前有效 round 是哪一个。
生命周期整体是否 OPEN / CLOSED / FAILED。
```

### 5.2 POS round

`crm_consume_points_round` 是本轮积分权益的唯一运行态。

它负责：

```text
grant/revoke 状态机。
当前有效积分。
正向积分流水 LID。
负向积分流水 LID。
规则和支付快照。
补偿重试字段。
```

### 5.3 POS event

`crm_consume_points_event` 只记录发生过什么。

它不负责：

```text
不判断当前有效积分。
不决定是否需要补偿。
不作为状态恢复依据。
```

补偿恢复只能看 round。

### 5.4 CRM task

`crm_consume_points_task` 只做账务动作幂等。

它不负责：

```text
不判断 POS 当前是否已退款。
不判断是否要创建新 round。
不承载 POS 生命周期状态。
```

同一 `round.lid` 下：

```text
GRANT 最多成功一次。
REVOKE 最多成功一次。
```

### 5.5 CRM points record

`crm_card_points_record` 是积分账本。

规则：

```text
正向赠分流水：
  amount > 0
  related_points_record_lid = null

负向撤销流水：
  amount < 0
  related_points_record_lid = 被撤销的正向流水 LID
```

## 6. 老链路暂缓边界

本轮不修改：

```text
CardBalanceService
CrmDealTask.givePoint
老储值消费撤销链路
```

但新链路必须满足：

```text
新 POS 消费积分不主动写 CrmDealTask.givePoint。
新 POS 消费积分以 lifecycleId，也就是 POS round.lid 作为主锚点。
CRM task 幂等键包含 task_lid 作为兼容维度；新 POS 链路传空并统一落库为 0，老储值链路仍保留在旧 CardBalanceService 分支。
```

本轮验收表述必须保持准确：

```text
可以声明：
  新 POS 消费积分生命周期已独立。
  新链路自身不制造重复赠分或重复撤销。

不能声明：
  已彻底解决老储值链路双退。
```
