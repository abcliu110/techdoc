﻿# checkOut 与 close 收口链路分析

## 1. 文档目的

这份文档回答 7 个问题：

1. 老会员充值链路里，前端实际调用了哪些后端接口。
2. 这些后端接口分别做什么。
3. `checkOut`、`queryOrderPay`、`pollingClose`、`dealCharge` 各自处在什么位置。
4. 为什么老会员充值真正的主调度中心是 POS，而不是前端，也不是 CRM。
5. 充值单二维码是怎么生成的。
6. 真实二维码与假二维码的区别是什么。
7. `DwdApplyPay` 这张表是谁创建、谁更新、状态如何流转。

## 2. 核心结论

老会员充值不是“前端直接调用 CRM 充值”。

真实链路是：

```text
前端发起充值
-> POS 创建本地充值单并进入统一结账流程
-> POS 处理支付
-> POS 在支付成功后的收口阶段调用 CRM cardChargeSign
-> POS 回写 CRM 返回结果并完成整单关闭
```

所以：

- 前端负责发起请求、展示二维码、查询状态、轮询界面结果。
- CRM 负责执行会员充值入账。
- POS 负责控制时序，决定什么时候允许真正调用 CRM。

## 3. 关键概念

### 3.1 `close` 的含义

`CheckOutVO.close` 表示这次结账调用是否已经把订单推进到“完成收口”的阶段。

- `CheckOutVO.close=false`：支付已发起，但订单还没有完成最终收口。
- `CheckOutVO.close=true`：订单已经进入或完成正式收口。

它不是单纯的“支付是否成功”标志，而是“是否已经走完 POS 收口主链路”的标志。

### 3.2 `queryOrderPay` 与 `pollingClose` 的区别

这两个接口最容易混淆，但职责完全不同。

#### `queryOrderPay`

作用：

- 查询第三方支付状态。
- 判断顾客是否已经支付成功。
- 在支付成功时触发 POS 收口。

它关心的是：

- `tradeState == SUCCESS` 吗？

#### `pollingClose`

作用：

- 轮询本地订单是否已经关闭。
- 给前端“支付弹窗是否可以结束”提供判断依据。

它关心的是：

- `DwdBill.orderStatus == CLOSED` 了吗？

一句话区分：

- `queryOrderPay` = 查支付结果并推动收口。
- `pollingClose` = 查收口结果是否已经落库完成。

### 3.3 `DwdApplyPay` 是什么

`DwdApplyPay` 是 POS 本地表 `dwd_apply_pay` 对应的实体，不是服务。

它表示：

- 一张订单发起的一次支付申请记录

它和 `DwdBill` 的区别是：

- `DwdBill` 是订单主单
- `DwdApplyPay` 是该订单下某次支付申请的状态记录

它的几个关键字段含义是：

- `DwdApplyPay.payTerminalTrace`（表 `dwd_apply_pay.pay_terminal_trace`）
  - 本次支付申请的唯一流水号
- `DwdApplyPay.payQrCode`（表 `dwd_apply_pay.pay_qr_code`）
  - 这次支付申请对应的二维码内容
- `DwdApplyPay.tradeState`（表 `dwd_apply_pay.trade_state`）
  - 第三方支付状态
- `DwdApplyPay.deal`（表 `dwd_apply_pay.deal`）
  - 这笔成功支付是否已经被 POS 收口处理完成
- `DwdApplyPay.refundState`（表 `dwd_apply_pay.refund_state`）
  - 退款处理状态

## 4. 前端实际调用了哪些后端接口

这一节只列老会员充值场景下，前端真实会用到的接口。

### 4.1 充值前准备类接口

#### `member_ops_get`

前端调用位置：`MemberRecharge/index.tsx`

用途：

- 查询会员信息。
- 返回会员卡号、手机号、会员类型、充值前余额、充值前积分等。

#### `member_ops_listRules`

前端调用位置：`MemberRecharge/index.tsx`

用途：

- 查询可选充值规则。
- 返回如“充1000送100”之类的充值套餐。

#### `member_ops_getDealingTask`

前端调用位置：`MemberRecharge/index.tsx`

用途：

- 查询是否存在进行中的充值单或未结束任务。

### 4.2 发起充值/结账类接口

#### `member_ops_cardCharge`

前端调用位置：`MemberRecharge/index.tsx`

用途：

- 发起老会员充值。
- 这是老会员充值场景的前端首入口。

前端收到返回后：

- 如果 `CheckOutVO.close=true`，说明本次已经同步收口成功。
- 如果 `CheckOutVO.close=false` 且 `order.payQrCode` 有值，说明进入二维码支付场景，前端打开支付弹窗。

#### `billCheckOut`

前端调用位置：`PaymentCodeModal/index.tsx`

用途：

- 付款码场景直接发起结账。
- 例如扫码枪录入顾客付款码后，直接走付款码支付。

### 4.3 支付后确认/观察类接口

#### `queryOrderPay`

前端调用位置：`PaymentCodeModal/index.tsx`

用途：

- 主动查询第三方支付状态。
- 支付成功时推动后端收口。

#### `pollingClose`

前端调用位置：`PaymentCodeModal/index.tsx`

用途：

- 自动轮询订单是否已经关闭。
- 默认每 3 秒一次。

#### `revokeSettle`

前端调用位置：`PaymentCodeModal/index.tsx`

用途：

- 撤销当前未完成支付/结算流程。

## 5. 这些后端接口分别做什么

### 5.1 `member_ops_get`

后端控制器：`MemberForBizController`

职责：

- 调 CRM 或会员服务查询会员卡信息。
- 返回会员基础资料、余额、积分、卡类型等。

它不做：

- 不创建本地充值单。
- 不发起支付。
- 不调用 CRM 充值入账。

### 5.2 `member_ops_listRules`

后端控制器：`MemberForBizController`

职责：

- 查询老会员充值规则。
- 返回给前端展示充值套餐。

### 5.3 `member_ops_getDealingTask`

后端控制器：`MemberForBizController`

职责：

- 查询当前会员是否存在进行中的充值任务或处理中订单。

### 5.4 `member_ops_cardCharge`

后端位置：`MemberForBizController.cardCharge(...)`

这是老会员充值的后端首入口，但它不是“直接充值到 CRM”。

它主要做：

1. 校验请求参数。
2. 查询会员信息 `memberCheck`（会员查询结果对象，不是表字段）。
3. 查询充值规则 `listRules`（充值规则查询动作，不是表字段）。
4. 创建或复用本地充值单 `DwdBill`。
5. 生成本地充值明细 `DwdFood`。
6. 把本金、赠送、充值前余额、充值前积分等写入本地单据。
7. 最后调用 `dwdBillOpsService.checkOut(...)`。

其中这一步显式写入到本地充值单 `DwdBill`（表 `dwd_bill`）的字段，至少包括：

| 字段 | 所属对象 | 所属表 | 说明 |
| --- | --- | --- | --- |
| `DwdBill.billName` | 订单主单 | `dwd_bill` | 本地充值单名称 |
| `DwdBill.cardNo` | 订单主单 | `dwd_bill` | 会员卡号 |
| `DwdBill.phone` | 订单主单 | `dwd_bill` | 会员手机号 |
| `DwdBill.memberName` | 订单主单 | `dwd_bill` | 会员名称 |
| `DwdBill.cardTypeLid` | 订单主单 | `dwd_bill` | 会员卡类型标识 |
| `DwdBill.memberTypeName` | 订单主单 | `dwd_bill` | 会员类型名称 |
| `DwdBill.cardBalanceBefore` | 订单主单 | `dwd_bill` | 充值前余额 |
| `DwdBill.cardPrincipalBefore` | 订单主单 | `dwd_bill` | 充值前本金余额 |
| `DwdBill.cardGiveBefore` | 订单主单 | `dwd_bill` | 充值前赠送余额 |
| `DwdBill.cardPointsBefore` | 订单主单 | `dwd_bill` | 充值前积分 |
| `DwdBill.deviceCode` | 订单主单 | `dwd_bill` | 发起充值的设备编码 |

它的本质是：

- 先把“会员充值业务”包装成一张 POS 本地订单。

它不直接做：

- 不直接调用 CRM `cardChargeSign`。

### 5.5 `billCheckOut`

后端位置：`DwdBillOpsForBizController.checkOut(...)`

service 位置：`DwdBillOpsServiceImpl.checkOut(...)`

它的职责是统一收银/结账入口，主要做：

1. 整理支付方式。
2. 校验支付组合是否合法。
3. 处理付款码或二维码支付。
4. 回写设备、收银员、支付交易号等信息。
5. 支付成功后处理挂账、会员卡、积分抵现。
6. 调 `OrderServiceUtil.checkOut(...)` 做正式收口。
7. 打印、写日志、必要时触发同步。

### 5.6 `queryOrderPay`

后端位置：`DwdBillOpsForBizController.queryOrderPay(...)`

service 位置：`DwdBillOpsServiceImpl.queryOrderPay(...)`

它主要做：

1. 查本地 `DwdApplyPay`。
2. 调支付网关 `nms4CloudPayService.query(...)`。
3. 读取第三方返回的 `tradeState`。
4. 如果 `tradeState == SUCCESS`：
   - 更新本地支付申请单状态。
   - 调 `OrderServiceUtil.close(dwdApplyPay)`。

它的本质是：

- 查询第三方支付结果。
- 在成功时推进 POS 收口主链路。

### 5.7 `pollingClose`

后端位置：`DwdBillOpsForBizController.pollingClose(...)`

核心依赖：`OrderServiceUtil.isClosed(mid, lid)`

它主要做：

1. 查询本地订单是否还处于未关闭状态。
2. 如果订单已经 `CLOSED`，返回 `true`。
3. 如果订单还没有 `CLOSED`，返回 `false`。

它的本质是：

- 观察“收口是否已经完成”。

它不做：

- 不查支付网关。
- 不推进收口。
- 不直接调用 `dealCharge(...)`。

### 5.8 `revokeSettle`

后端位置：`DwdBillOpsForBizController.revokeSettle(...)`

它主要做：

- 撤销未完成支付/结算。
- 清理当前支付申请和订单状态。

## 6. 老会员充值的完整主链路

### 6.1 同步成功路径

同步成功一般出现在：

1. 没有走微信/支付宝移动支付。
2. 走了付款码支付，并且支付网关在当前请求内直接返回成功。

链路如下：

```text
前端 member_ops_cardCharge
-> MemberForBizController.cardCharge
-> 创建/复用本地充值单 DwdBill
-> dwdBillOpsService.checkOut(...)
-> DwdBillOpsServiceImpl.checkOut(...)
-> dealMobile(...) 返回 close=true
-> updateCash / dealCredit / dealCard / dealCardPoint
-> OrderServiceUtil.checkOut(...)
-> OrderServiceUtil.dealCharge(...)
-> CRM /crm_card_op/cardChargeSign
-> 回写本地 DwdBill
-> 正式结账 / 打印 / 上传 / 广播刷新
```

### 6.2 异步二维码支付路径

二维码支付场景下，第一次通常返回 `CheckOutVO.close=false`。

链路如下：

```text
前端 member_ops_cardCharge
-> 后端建本地充值单并返回 payQrCode
-> 前端打开 PaymentCodeModal
-> 前端每 3 秒调用 pollingClose
-> 用户也可以手动调用 queryOrderPay
-> queryOrderPay 查支付网关 tradeState
-> 如果 tradeState == SUCCESS
   -> OrderServiceUtil.close(dwdApplyPay)
   -> CloseMpScHandler.runInner(...)
   -> OrderServiceUtil.checkOut(...)
   -> OrderServiceUtil.dealCharge(...)
   -> CRM cardChargeSign
   -> 本地订单变为 CLOSED
-> pollingClose 返回 data=true
-> 前端停止轮询并提示支付成功
```

## 7. `OrderServiceUtil.checkOut(...)` 做了什么

`OrderServiceUtil.checkOut(...)` 是真正的业务收口核心。

它主要做四件事：

1. `dealCharge(...)`
   - 如果是充值单 `OrderOpTypeEnum.E`，在这里处理充值入账。
   - 老普通充值走 CRM `cardChargeSign`。
   - 新储值方案走 `DepositPlanChargeUtil`。
2. `dealRepay(...)`
   - 如果是回款单 `OrderOpTypeEnum.P`，在这里处理回款。
3. 真正结账
   - 按 `checkType` 调 `normalCheckOut(...)` 或 `zeroCheckOut(...)`。
4. 结账后的后处理
   - 上传订单、上传质检、广播通知设备刷新。

所以：

- `DwdBillOpsServiceImpl.checkOut(...)` 是支付流程与收口编排入口。
- `OrderServiceUtil.checkOut(...)` 是真正的业务落账核心。

## 8. `dealCharge(...)` 为什么在 POS 结账成功之后才调用 CRM

### 8.1 触发条件

`dealCharge(...)` 只在 `OrderOpTypeEnum.E`，也就是充值单时执行。

老普通充值场景下，如果：

- `depositPlanLid == null`

则会走老 CRM 充值接口：

- `/crm_card_op/cardChargeSign`

### 8.2 它组装了什么参数

`dealCharge(...)` 会组装 `CrmCardOpChargeDTO`，核心字段包括：

- `orderId = dwdBill.getSaasOrderKey()`
- `cardNo = dwdBill.getCardNo()`
- `principalAmount = dwdBill.getPaidAmount()`
- `giveAmount = dwdBill.getSendAmount()`
- `ruleLid / ruleLName`
- `comment = 支付方式说明`
- `operator = user.getName()`

然后调用：

- `nms4CloudCrmService.cardCharge(chargeDTO)`

对应 CRM 接口：

- `/crm_card_op/cardChargeSign`

### 8.3 为什么不能在入口直接调 CRM

如果在 `member_ops_cardCharge` 入口一进来就直接调 CRM 充卡，会有风险：

1. 顾客还没付款，CRM 就先加钱。
2. 二维码支付还没完成，CRM 已经入账。
3. 后续支付失败、撤销、超时，会导致对账和回滚都变复杂。

所以系统设计成：

```text
先建本地充值单
-> 先完成 POS 支付和收口
-> 再在收口阶段调用 CRM 充值入账
```

### 8.4 CRM 返回后会回写什么

CRM 返回 `CardBalanceVO` 后，会把这些结果回写到本地 `DwdBill`：

- `cardBalanceAfter`
- `cardPrincipalAfter`
- `cardGiveAfter`
- `points`
- `cardPointsAfter`
- `cardTaskLid`

其中 `cardTaskLid` 很关键：

- 它是 CRM 会员任务号。
- 后续撤销充值时，要靠它去 CRM 回查和回滚。

## 9. 为什么说 POS 是主调度中心

### 9.1 不是前端主调度

前端只能做：

- 发起充值请求。
- 展示支付二维码。
- 查询支付状态。
- 轮询订单是否关闭。

前端做不了：

- 决定什么时候真正给会员卡加钱。
- 决定什么时候允许调用 CRM 入账。

### 9.2 不是 CRM 主调度

CRM 只负责：

- 在被调用时执行会员充值入账。
- 返回充值后的余额、积分、任务号。

CRM 做不了：

- 不知道顾客是不是已经在 POS 付款成功。
- 不知道当前这张本地订单是否已经允许收口。

### 9.3 POS 才是主调度

POS 负责：

1. 建本地充值单。
2. 处理支付流程。
3. 判断支付是否成功。
4. 判断是否允许正式收口。
5. 在收口阶段调用 CRM `cardChargeSign`。
6. 把 CRM 返回结果回写到本地单据。
7. 完成结账、打印、上传、广播刷新。

所以准确说法是：

- 前端 = 入口和界面层。
- POS = 总控和时序编排中心。
- CRM = 被 POS 调用的账务执行方。

## 10. 充值单二维码是如何生成的

### 10.1 前端并不生成二维码内容

前端支付弹窗 `PaymentCodeModal` 只是把后端返回的 `order.payQrCode` 渲染成二维码图片。

前端本质上是：

```tsx
<QRCode size={250} value={billOrderDetail?.payQrCode!} />
```

这说明：

- 二维码真正的内容字符串由后端生成。
- 前端不自己拼二维码 URL。
- 前端只是读取 `order.payQrCode` 并显示。

### 10.2 前端显示二维码时读取了哪些参数

支付弹窗展示时，前端会读取以下数据：

1. `order.payQrCode`
   - 二维码实际内容
2. `order.paidAmount`
   - 顶部应收金额展示
3. `pays`
   - 支付方式和支付金额展示
4. `order.cardNo`
   - 会员卡号展示
5. `order.phone`
   - 手机号展示
6. `order.memberName`
   - 会员名称展示
7. `order.cardBalanceBefore`
   - 充值前余额展示
8. `order.cardPointsBefore`
   - 充值前积分展示
9. `foods`
   - 充值套餐名称、笔数、金额等明细展示

其中真正决定二维码内容的关键字段只有一个：

- `order.payQrCode`??? VO ??????? `DwdBill.payQrCode` / ? `dwd_bill.pay_qr_code`?

### 10.3 后端在哪一步生成 `payQrCode`

二维码是在 `DwdBillOpsServiceImpl.dealMobile(...)` 的二维码支付分支里生成的。

流程是：

1. 识别当前是移动支付。
2. 如果没有顾客付款码 `authNo`，说明要生成顾客扫码用的收款二维码。
3. 创建 `DwdApplyPay`。
4. 调 `OrderServiceUtil.createQrcode(dwdApplyPay, dwdBill, dwdPay, null)`。
5. 把 `dwdApplyPay.payQrCode` 回写到 `dwdBill.payQrCode`。
6. 返回给前端，前端再把 `order.payQrCode` 画成二维码。

这里要注意：

- `DwdApplyPay.payQrCode`?? `dwd_apply_pay.pay_qr_code`? 最初是生成并保存在 `DwdApplyPay` 上
- `dwdBill.payQrCode` 只是为了给订单主单和前端展示使用，做了一次回写

### 10.4 `createQrcode(...)` 读取了哪些参数

`OrderServiceUtil.createQrcode(...)` 主要从三个对象上取值：

#### 从 `dwdPay` 读取

1. `dwdPay.getAmount()`
   - 支付金额
2. `dwdPay.getType()`
   - 判断支付方式是微信还是支付宝

#### 从 `dwdBill` 读取

1. `dwdBill.getOrderOpType()`
   - 判断这是普通消费单还是充值单
2. `dwdBill.getMid()`
   - 商户 ID
3. `dwdBill.getSid()`
   - 门店 ID
4. `dwdBill.getLid()`
   - 本地订单 ID
5. `dwdBill.getSaasOrderKey()`
   - POS 订单号，作为支付侧订单标识
6. `dwdBill.getCheckoutBy()`
   - 收银员

#### 从 `dwdApplyPay` 读取

1. `dwdApplyPay.getPayTerminalTrace()`
   - 本次支付流水号 `terminalTrace`

因此可以把两者关系理解成：

- `DwdApplyPay.payTerminalTrace`?? `dwd_apply_pay.pay_terminal_trace`? 先在 `DwdApplyPay` 创建时生成
- `DwdApplyPay.payQrCode`?? `dwd_apply_pay.pay_qr_code`? 再基于这次支付申请生成

### 10.5 它实际传给支付服务的参数

如果不是“假二维码”模式，系统会组装 `PrePayRequest` 并调用支付服务预下单。

核心参数包括：

1. `notifyUrl`
   - 支付回调地址 `/merchant/dwd_bill_ops/notify`
2. `dealType`
   - 普通消费走 `CONSUME`
   - 充值单走 `RECHARGE`
3. `totalFee`
   - 支付金额，单位分
4. `terminalTrace`
   - 本次支付流水号
5. `openId`
   - 当前调用里通常传 `null`
6. `DwdApplyPay.payType`?? `dwd_apply_pay.pay_type`?
   - 微信或支付宝
7. `saasOrder`
   - POS 订单号 `saasOrderKey`
8. `cashier`
   - 收银员

这里最关键的一点是：

如果这张单是充值单 `OrderOpTypeEnum.E`，二维码创建时会明确走：

- `DealTypeEnum.RECHARGE`

也就是说，充值单二维码走的是“充值支付通道”，不是普通消费通道。

## 11. 假二维码场景下，`payQrCode` 的 URL 前缀从哪里来

### 11.1 假二维码的拼接方式

当系统配置 `g_cashierReceiptsDuplicateUsed=true` 时，`createQrcode(...)` 不会去支付网关申请真实支付链接，而是直接在本地拼一个 URL 给 `payQrCode`。

拼接逻辑是：

```text
{server}/?S={sid62}&M={mid62}&B={lid}
```

其中：

- `S` = 门店 `sid` 的 62 进制编码
- `M` = 商户 `mid` 的 62 进制编码
- `B` = 本地订单 `lid`

### 11.2 URL 前缀来自哪里

这个 `server` 前缀来自配置类：`PayProperties`

绑定的配置前缀是：

```text
nms4cloud.pay.server
```

也就是说，假二维码的 URL 前缀不是前端写死的，也不是 `nms.common.api-server`，而是：

- `nms4cloud.pay.server`

### 11.3 为什么不是 `nms.common.api-server`

代码里也存在：

- `nms.common.api-server`

但那是另一套公共 API 地址配置。

假二维码拼接使用的是：

- `properties.getServer()`
- 而这个 `properties` 对应的是 `PayProperties`
- `PayProperties.PREFIX = "nms4cloud.pay"`

所以假二维码 URL 的前缀来源很明确：

```text
nms4cloud.pay.server
```

### 11.4 不同环境下的实际前缀示例

在不同环境配置里，这个前缀会不同。例如：

1. 生产环境：`https://p.lmnsaas.com`
2. 广州酒家环境：`https://p.gzjjzhy.com`
3. 自测环境：`http://clovia.top`

因此假二维码最终显示出来的内容会随着当前运行环境不同而不同。

## 12. 真实二维码与假二维码的对照图

### 12.1 真实二维码生成链路

真实二维码的意思是：

- 后端去支付网关发起预下单
- 支付网关返回真实支付链接 `payUrl`
- 后端把这个 `payUrl` 作为 `payQrCode` 返回给前端

链路如下：

```text
member_ops_cardCharge / checkOut
  -> DwdBillOpsServiceImpl.dealMobile(...)
  -> OrderServiceUtil.createQrcode(...)
  -> 组装 PrePayRequest
  -> nms4CloudPayService.prePay(prePayRequest)
  -> 支付网关返回 PayOrderVO.payUrl
  -> dwdApplyPay.payQrCode = payUrl
  -> dwdBill.payQrCode = dwdApplyPay.payQrCode
  -> 前端 QRCode 组件渲染 order.payQrCode
```

### 12.2 假二维码生成链路

假二维码的意思是：

- 不去支付网关申请真实 `payUrl`
- 后端本地直接拼接一个 URL 放进 `payQrCode`

触发条件是：

- 配置 `g_cashierReceiptsDuplicateUsed=true`

链路如下：

```text
member_ops_cardCharge / checkOut
  -> DwdBillOpsServiceImpl.dealMobile(...)
  -> OrderServiceUtil.createQrcode(...)
  -> 命中 fake 模式
  -> 使用 nms4cloud.pay.server 作为前缀
  -> 拼接 /?S={sid62}&M={mid62}&B={lid}
  -> dwdApplyPay.payQrCode = 本地拼接后的 URL
  -> dwdBill.payQrCode = dwdApplyPay.payQrCode
  -> 前端 QRCode 组件渲染 order.payQrCode
```

### 12.3 两种二维码的核心区别

#### 真实二维码

来源：

- 支付网关返回的 `payUrl`

#### 假二维码

来源：

- POS 本地基于配置和订单参数拼出来的 URL

一句话总结：

- 真实二维码：`payQrCode = 支付网关返回的 payUrl`
- 假二维码：`payQrCode = nms4cloud.pay.server + sid/mid/lid 本地拼接 URL`

前端不区分这两种来源，它统一只做一件事：

- 读取 `order.payQrCode`，然后把它渲染成二维码图片。

## 13. `DwdApplyPay` 状态是如何被更新的

`DwdApplyPay` 不是一个服务，而是 POS 本地表 `dwd_apply_pay` 对应的实体。

它可以理解为：

- 一张订单发起的一次支付申请记录

这张表的状态不是由单一方法统一维护，而是由 POS 本地多个环节按支付生命周期分段更新。

### 13.1 创建时由谁初始化

创建入口在：`OrderServiceUtil.createApplyPay(...)`

初始化时会写入：

- `mid`
- `sid`
- `dwdBillId`
- `dwdBillLid`
- `dwdBillName`
- `payAmount`
- `DwdApplyPay.payTerminalTrace`?? `dwd_apply_pay.pay_terminal_trace`?
- `DwdApplyPay.tradeState = USERPAYING`?? `dwd_apply_pay.trade_state`?
- `devId`
- `devName`
- `createdBy`
- `DwdApplyPay.payType`?? `dwd_apply_pay.pay_type`?
- `dealType`
- `DwdApplyPay.deal = false`?? `dwd_apply_pay.deal`?
- `refundState = U`
- `orderOpType`

其中最关键的初始状态是：

- `DwdApplyPay.tradeState = USERPAYING`?? `dwd_apply_pay.trade_state`?
- `DwdApplyPay.deal = false`?? `dwd_apply_pay.deal`?

表示：

- 这是一条刚创建的支付申请，尚未完成支付，也尚未完成收口处理。

### 13.2 `payTerminalTrace` 是怎么来的

`payTerminalTrace` 在创建 `DwdApplyPay` 时生成：

```java
dwdApplyPay.setPayTerminalTrace(IdWorkerPlus.getIdStr());
```

它本质上是：

- 本次支付申请的唯一流水号

后续用途包括：

1. 发起支付时传给支付网关。
2. 支付回调时用它反查本地支付申请单。
3. 手动查询支付状态时也用它去支付网关查询。

### 13.3 `payQrCode` 是怎么来的

`payQrCode` 也是在支付申请阶段生成，但分两种来源：

#### 真实二维码

- POS 调支付网关 `prePay`
- 支付网关返回 `payUrl`
- 系统把 `payUrl` 存入 `payQrCode`

即：

- `payQrCode = 支付网关返回的 payUrl`

#### 假二维码

如果命中 fake 模式：

- 不调用支付网关拿 `payUrl`
- 直接按本地规则拼 URL

格式是：

```text
{nms4cloud.pay.server}/?S={sid62}&M={mid62}&B={lid}
```

即：

- `payQrCode = POS 本地拼出来的 URL`

### 13.4 支付状态 `tradeState` 是谁更新的

`tradeState` 主要由两个入口更新：

#### 1. 手动查询支付状态时更新

入口：`DwdBillOpsServiceImpl.queryOrderPay(...)`

流程：

1. 根据 `DwdApplyPay.payTerminalTrace` 去支付网关查询。
2. 取回第三方返回的 `tradeState`。
3. 如果与数据库里的状态不同，就更新本地 `DwdApplyPay`。

更新字段包括：

- `DwdApplyPay.tradeState`?? `dwd_apply_pay.trade_state`?
- `DwdApplyPay.payAt`?? `dwd_apply_pay.pay_at`?，成功时写入
- `DwdApplyPay.payType`?? `dwd_apply_pay.pay_type`?，成功时写入
- `DwdApplyPay.updatedTime`?? `dwd_apply_pay.updated_time`?
- `updatedBy = "支付手动查询"`

#### 2. 支付通知时更新

入口：`OrderServiceUtil.notify(...)`

流程：

1. 根据支付回调里的 `terminalTrace` 找到本地 `DwdApplyPay`。
2. 再主动去支付网关查询一次真实状态。
3. 如果状态变化，就更新本地记录。

更新字段包括：

- `DwdApplyPay.tradeState`?? `dwd_apply_pay.trade_state`?
- `DwdApplyPay.payTradeOrderNo`?? `dwd_apply_pay.pay_trade_order_no`?
- `DwdApplyPay.payAt`?? `dwd_apply_pay.pay_at`?
- `DwdApplyPay.payType`?? `dwd_apply_pay.pay_type`?
- `DwdApplyPay.updatedTime`?? `dwd_apply_pay.updated_time`?
- `updatedBy = "支付通知"`

所以：

- `DwdApplyPay.tradeState`?? `dwd_apply_pay.trade_state`? 不是靠前端轮询直接改的。
- 它是由 POS 本地查询支付网关后写回本地表的。

### 13.5 收口完成后，谁把 `deal` 改成 `true`

当订单真正结账完成后，由 `CheckOutServiceImpl` 更新成功支付申请记录：

- `deal = true`
- `dealAt = checkoutTime`

而且只更新：

- `tradeState = SUCCESS` 的记录

这一步的含义是：

- 这条支付申请虽然之前已经支付成功，
- 但直到 POS 正式完成结账收口之后，才算“已处理完成”。

所以：

- `tradeState = SUCCESS` 表示钱已经支付成功
- `deal = true` 表示这笔成功支付已经被 POS 收口处理完了

### 13.6 撤销时，谁把状态改成 `REVOKED`

`tradeState = REVOKED` 主要有三种来源：

#### 1. 手工撤销支付/结算

入口：`revokeSettle(...)`

如果当前支付申请还处于：

- `USERPAYING`
- `NOTPAY`

则会更新为：

- `DwdApplyPay.tradeState = REVOKED`?? `dwd_apply_pay.trade_state`?
- `DwdApplyPay.cancelBy`?? `dwd_apply_pay.cancel_by`?
- `DwdApplyPay.cancelAt`?? `dwd_apply_pay.cancel_at`?

#### 2. 超过 30 分钟自动撤销

入口：`OrderServiceUtil.notify(...)`

如果支付状态长期未成功，并且超过 30 分钟，会自动更新为：

- `DwdApplyPay.tradeState = REVOKED`?? `dwd_apply_pay.trade_state`?
- `updatedBy = "超过30分钟撤销"`

#### 3. 结账时自动撤销其他未成功记录

入口：`CheckOutServiceImpl`

当某一笔支付成功并且订单已经正式结账后，系统会把同订单下其他仍处于：

- `USERPAYING`
- `NOTPAY`

的支付申请自动改成：

- `DwdApplyPay.tradeState = REVOKED`?? `dwd_apply_pay.trade_state`?
- `updatedBy = "结账时自动撤销"`

这样可以避免同一订单残留多条未完成支付申请。

### 13.7 退款状态是谁更新的

退款时，更新的是 `refundState` 系列字段。

入口：`refundPay(...)`

更新内容包括：

- `DwdApplyPay.refundState`?? `dwd_apply_pay.refund_state`?
- `DwdApplyPay.refundBy`?? `dwd_apply_pay.refund_by`?
- `DwdApplyPay.refundedAmount`?? `dwd_apply_pay.refunded_amount`?
- `DwdApplyPay.refundAt`?? `dwd_apply_pay.refund_at`?

所以：

- `DwdApplyPay.tradeState`?? `dwd_apply_pay.trade_state`? 管支付申请状态
- `DwdApplyPay.refundState`?? `dwd_apply_pay.refund_state`? 管退款处理状态

这是两套不同维度的状态。

### 13.8 这张表是在云端还是 POS 本地

`DwdApplyPay` 对应的是 POS 本地数据库表：

- `dwd_apply_pay`

并且插入、查询、更新都通过 POS 本地 Mapper 完成：

- `DwdApplyPayMapper`
- `dwdApplyPayMapper.insert(...)`
- `Chain.forQuery(dwdApplyPayMapper)`
- `Chain.forUpdate(dwdApplyPayMapper)`

所以它不是云端支付平台的表，而是：

- POS 本地维护的支付申请表

云端支付网关负责的是：

- 接收支付请求
- 返回支付状态
- 返回 `payUrl` / `outTradeNo` 等结果

但本地支付申请生命周期状态，仍然落在 POS 本地表 `dwd_apply_pay` 上。

## 14. 状态是如何流转的，以及流转条件是什么

这一节把文档里出现的几个关键状态统一拆开说明：

1. `CheckOutVO.close`
2. `DwdBill.orderStatus`
3. `DwdApplyPay.tradeState`
4. `DwdApplyPay.deal`
5. `DwdApplyPay.refundState`

这些状态分别属于不同层次，不能混着看。

### 14.1 `CheckOutVO.close` 的流转

`close` 是接口返回值，不是数据库字段。

它的作用是告诉前端：

- 这次请求结束时，订单是否已经进入或完成最终收口

#### `close = false`

出现条件：

1. 当前走的是移动支付。
2. 没有顾客付款码 `authNo`，即二维码支付场景。
3. 或者有付款码，但支付网关返回的不是 `SUCCESS`，而是：
   - `USERPAYING`
   - `NOTPAY`
   - 其他未成功状态

此时系统会：

- 创建 `DwdApplyPay`
- 生成 `payQrCode` 或记录付款码支付申请
- 返回给前端 `CheckOutVO.close=false`
- 后续等待：
  - 支付回调 `notify`
  - 或手动查询 `queryOrderPay`
  - 或前端继续轮询 `pollingClose`

#### `close = true`

出现条件：

1. 没有走微信/支付宝移动支付。
2. 或者付款码支付在当前请求内就同步返回 `SUCCESS`。
3. 或者异步支付成功后，收口链路已经跑完，前端再次查询时看到订单已关闭。

所以：

- `CheckOutVO.close=false` = 还没完成最终收口
- `CheckOutVO.close=true` = 已进入或完成最终收口

### 14.2 `DwdBill.orderStatus` 的流转

`DwdBill.orderStatus` 表示整张订单的业务状态。

对于充值二维码支付这条链，最重要的是：

- `UNSETTLED`
- `CLOSED`
- `CANCEL`

#### `UNSETTLED -> CLOSED`

流转条件：

1. 支付已经成功
   - 同步支付成功，或者
   - 异步支付成功后被 `queryOrderPay` / `notify` 确认
2. 进入 POS 收口流程
   - `OrderServiceUtil.close(dwdApplyPay)`
   - `CloseMpScHandler.runInner(...)`
3. 收口过程中完成：
   - 挂账处理
   - 会员卡处理
   - 积分抵现处理
   - `OrderServiceUtil.checkOut(...)`
   - 对充值单执行 `dealCharge(...)`
4. 结账服务成功落账

只有这些动作完成后，订单才应该变成：

- `CLOSED`

这就是为什么：

- 支付成功 != 订单已关闭

#### `UNSETTLED -> CANCEL`

流转条件：

- 支付流程被人工撤销或业务取消
- 或订单被整体取消

这个状态不是二维码支付成功主链的正常终态，而是中断态。

### 14.3 `DwdApplyPay.tradeState` 的流转

`tradeState` 是支付申请状态，属于支付申请层，不是订单层。

最常见的流转是：

```text
创建支付申请
-> USERPAYING
-> SUCCESS / NOTPAY / USERPAYING
-> REVOKED（撤销、超时、清理时）
```

#### 初始状态：`USERPAYING`

流转条件：

- 创建 `DwdApplyPay` 时直接写入

表示：

- 已经发起支付申请，等待支付结果

#### `USERPAYING / NOTPAY -> SUCCESS`

流转条件：

1. `queryOrderPay()` 查询支付网关时返回 `SUCCESS`
2. 或 `notify()` 查询支付网关时返回 `SUCCESS`
3. 然后把本地 `DwdApplyPay.tradeState` 更新为 `SUCCESS`

表示：

- 这次支付申请已经确认支付成功

但这时仍然不代表整单已经收口完成。

#### `USERPAYING / NOTPAY -> REVOKED`

流转条件有三类：

1. 人工撤销支付/结算
   - `revokeSettle(...)`
2. 超过 30 分钟仍未支付成功
   - `notify()` 中自动撤销
3. 同订单已有一笔支付成功并完成结账
   - 结账时自动撤销其他未成功记录

表示：

- 这条支付申请已经失效，不再继续参与后续收口

### 14.4 `DwdApplyPay.deal` 的流转

`deal` 表示：

- 这条支付申请是否已经被 POS 收口处理完

它和 `tradeState` 不是一个维度。

#### 初始值：`false`

流转条件：

- 创建 `DwdApplyPay` 时初始化为 `false`

表示：

- 不管这笔支付后续会不会成功，当前都还没有完成收口处理

#### `false -> true`

流转条件：

1. 该支付申请已经 `tradeState = SUCCESS`
2. 对应订单已经完成正式结账
3. `CheckOutServiceImpl` 在结账成功后批量更新成功支付申请记录

表示：

- 这笔成功支付不但付成了，而且已经被 POS 收口处理过了

所以：

- `tradeState = SUCCESS` = 付钱成功
- `deal = true` = 收口也完成了

### 14.5 `DwdApplyPay.refundState` 的流转

`refundState` 是退款处理状态，不是支付状态。

常见值可以理解为：

- `U`：未退款
- `P`：部分退款
- `A`：全部退款

#### 初始值：`U`

流转条件：

- 创建支付申请时初始化为未退款

#### `U -> P`

流转条件：

- 发起退款，且退款金额小于剩余可退款金额

表示：

- 这条支付申请已经部分退款

#### `U / P -> A`

流转条件：

- 发起退款，且本次退款后累计退款金额达到支付金额上限

表示：

- 这条支付申请已经全部退款

### 14.6 状态流转总图

#### 订单收口视角

```text
member_ops_cardCharge
  -> 创建本地充值单 DwdBill
  -> orderStatus = UNSETTLED
  -> 发起支付
  -> 支付成功后进入 close 收口
  -> OrderServiceUtil.checkOut(...)
  -> dealCharge(...)
  -> orderStatus = CLOSED
```

#### 支付申请视角

```text
createApplyPay
  -> tradeState = USERPAYING
  -> deal = false
  -> 生成 payTerminalTrace
  -> 生成 payQrCode
  -> 查询支付 / 支付通知
  -> tradeState = SUCCESS / NOTPAY / USERPAYING / REVOKED
  -> 如果订单正式结账成功
     -> deal = true
  -> 如果发生退款
     -> refundState = U / P / A
```

### 14.7 一句话总结

状态流转要分层看：

- `close`：接口层，表示当前请求是否已经完成收口
- `DwdBill.orderStatus`：订单层，表示整张单是否已经关单
- `DwdApplyPay.tradeState`（表 `dwd_apply_pay.trade_state`）：支付申请层，表示某次支付申请是否成功
- `DwdApplyPay.deal`（表 `dwd_apply_pay.deal`）：收口处理层，表示这笔成功支付是否已经被 POS 处理完
- `DwdApplyPay.refundState`（表 `dwd_apply_pay.refund_state`）：退款层，表示这笔支付申请退款到了哪一步

| `DwdApplyPay.payQrCode` | 支付申请单 | `dwd_apply_pay` | 二维码原始内容 |
| `DwdApplyPay.tradeState` | 支付申请单 | `dwd_apply_pay` | 第三方支付状态 |
| `DwdApplyPay.deal` | 支付申请单 | `dwd_apply_pay` | 是否已完成 POS 收口处理 |
| `DwdApplyPay.refundState` | 支付申请单 | `dwd_apply_pay` | 退款处理状态 |
| `DwdApplyPay.payTradeNo` | 支付申请单 | `dwd_apply_pay` | 支付平台交易号 |

| `DwdFood` 相关明细 | 充值/菜品明细 | `dwd_food` | 前端 `foods` 列表来源 |

## 16. 一句话总结

老会员充值的真实流程不是“前端直接调 CRM 充值”，而是“前端发起 -> POS 建本地充值单并完成支付/收口 -> POS 在收口阶段调用 CRM 入账 -> POS 回写结果并完成关单”；二维码本身也不是前端生成的，而是后端在 `checkOut -> dealMobile -> createQrcode` 链路里生成并返回给前端展示。
