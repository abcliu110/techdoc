## 术语表

| 术语 | 定义 |
|---|---|
| 积分获取规则 | 会员消费后按比例获得积分的规则配置 |
| 积分抵扣规则（积分抵现）| 会员使用积分抵扣消费金额的规则配置 |
| 积分清零规则 | 积分到期自动清零的策略配置 |
| 积分规则说明 | 向会员公示的积分规则文字说明 |
| 会员等级 | 系统中已配置的会员层级（如VIP1、vip2），来自会员等级管理模块 |
| 多倍积分 | 在特定时段（生日/会员日）按倍率额外赠送积分的机制 |
| 可用周期 | 积分规则生效的时间周期粒度（按日/按周/按月） |
| 企微会员专享 | 限定积分策略仅对满足企业微信好友/社群条件的会员生效 |
| 逐步清零 | 积分按清零频次（日清/月清）批量到期，而非一次性全部清零 |
| 抵扣倍数 | 每次抵扣必须为N的整数倍积分 |
| 来源标签 | 📸=截图可见 / 👤=用户确认 / 🤖=AI推断 |

## 一、业务概述

### 1.1 系统定位

「设置积分权益」是全来店商户中心的积分规则全局配置页面，商家在此统一设定会员积分的获取规则、抵扣规则、清零规则及公示说明。本页影响所有会员的积分计算、抵现、失效等核心权益。📸

### 1.2 表单形态

- **顶层开关**：积分规则 Toggle（全局启停，开启后各子规则生效）📸
- **四个规则区块**：积分获取规则 / 积分抵扣规则 / 积分清零规则 / 积分规则说明，各区块独立可配置 📸
- **保存机制**：整页统一保存，顶部右上角「保存」按钮，点「取消」还原 📸
- **左侧导航**：四区块对应左侧四项导航（当前页内锚点导航，不跳转） 📸

## 二、页面结构

| 区块编号 | 区块名称 | 主要交互元素 |
|---|---|---|

### 2.2 信息流向

```
积分规则（全局开关）
  └── B-1 积分获取规则（是否启用消费送积分）
        ├── 积分模式（按支付方式 / 按商品实收）
        │     └── 适用商品（按商品实收时显示）→ 商品选择[P1]
        ├── 会员等级积分比率 [D]（多行）
        ├── 单笔获取上限
        ├── 多倍积分（生日/会员日）
        ├── 可用周期 → 联动日期选择
        ├── 可用时段 → 联动时段输入
        ├── 企微会员专享
        ├── 订单场景限制
        └── 订单渠道限制
  └── B-2 积分抵扣规则（是否启用积分抵现）
        ├── 积分抵扣比例
        ├── 起扣积分
        ├── 抵扣倍数
        ├── 抵扣金额取整
        ├── 抵扣上限 → 联动输入
        ├── 可用周期 → 联动日期选择
        ├── 可用时段 → 联动时段输入
        ├── 商品限制 → 商品范围
        │     ├── 指定商品参与/不参与 → 商品选择[P1]
        │     ├── 指定分类参与 → 分类选择[P2] + 排除以下商品 → 商品选择[P1]
        │     └── 指定分类不参与 → 分类选择[P2] + 排除以下商品 → 商品选择[P1]
        ├── 订单场景限制
        └── 订单渠道限制
  └── B-3 积分清零规则（是否启用积分清零）
        ├── 清零模式（自发放日期/按年度）
        └── 是否消息提醒 → 联动「距清零N天消息提醒」输入
  └── B-4 积分规则说明
        └── 积分规则Textarea
```

### 2.3 负空间

- 积分规则全局 Toggle **关闭**时：四个子区块标题仍可见，每块折叠为仅含各自子 Toggle（灰色禁用），所有详细字段均隐藏 📸
- 积分获取规则「是否启用消费送积分」Toggle **关闭**时：积分模式/会员等级等所有子字段隐藏 📸
- 积分抵扣规则「是否启用积分抵现」Toggle **关闭**时：抵扣比例/起扣积分等所有子字段隐藏（行为与获取/清零规则一致）📸
- 积分清零规则「是否启用积分清零」Toggle **关闭**时：清零模式/时限/频次等所有子字段隐藏 📸
- 积分规则说明 Textarea 在全局 Toggle 关闭时仍然可见可编辑 📸

## 三、数据模型

### 3.1 数据容器识别

| 标识 | 类型 | 说明 |
|---|---|---|
| [M] | 主表单 | 积分权益设置（整页一次保存） |
| [D] | 明细行 | 会员等级积分比率（积分获取规则下，每行=一个等级的积分获取比率） |
| [P1] | 弹窗子实体 | 「选择商品」弹窗（product_scope_type=商品类时触发）|
| [P2] | 弹窗子实体 | 「选择分类」弹窗（product_scope_type=分类类时触发）|
| [J] | 多选关联 | 积分抵扣规则与商品/分类的N:N关联 |

📸 IMG-Q3、IMG-Q9b、IMG-Confirm5a/b

### 3.2 实体层级

```
积分权益配置 [M]（1个商户1份配置）
  ├── 会员等级积分比率 [D]（1:N，每个会员等级设置一行）
  ├── 获取适用商品关联 [J]（N:N，earning_mode=BY_PRODUCT 且 scope≠ALL 时）
  ├── 抵扣商品/分类关联 [J]（N:N，product_limit_enabled=TRUE 时）
  └── 抵扣排除商品关联 [J]（N:N，product_scope_type=CATEGORY_IN 或 CATEGORY_OUT 时，从分类中排除特定商品）
```

| expiry_mode | expiry_duration+expiry_frequency / expiry_annual_* | = FROM_ISSUE_DATE/ANNUAL 时分别显示 |

### 3.6 跨区依赖

- 会员卡类型（plan_lid）→ 积分权益规则必须挂载在一个已存在的会员卡类型下，保存时需校验 `plan_lid` 在当前商户下存在且有效 📸
- 会员等级设置 → 引用「会员等级管理」模块的等级数据（外键）📸
- 选择商品 → 引用「商品管理」模块的商品数据（外键），保存时需校验所有引用的商品 ID 在当前商户下存在且有效 📸
- 订单渠道/场景 → 与订单模块的渠道/场景枚举对齐 🤖
- 会员日具体日期（周几/几号）→ 在当前积分权益页面直接配置；按周时选周几，按月时选几号 📸

### 3.7 事务边界

- 整页表单为单事务，点击「保存」一次提交所有区块配置 📸
- 「选择商品」弹窗内有独立「确定/取消」，确定后同步到主表单，随主表单一起保存 📸

### 4.2 不变量

- deduction_rate > 0 📸
- 固定积分上限：0.01 ≤ value ≤ 999999 📸
- 账单比例：0.01 ≤ value ≤ 100 📸
- 清零时限枚举固定5项，不可自定义 📸
- 订单渠道固定7项 📸
- expiry_remind_days > 0（提醒天数为正整数）🤖
- 会员等级明细行 ≥ 1 行 📸

### 4.3 计算逻辑

- **积分获取**：单次获得积分 = floor(实收金额 / consume_amount) × earn_points（按会员等级），若有上限再取 min(计算值, limit) 🤖
- **多倍积分**：特殊日积分 = 基础积分 × multiplier；生日与会员日重叠时取 max(birthday_multiplier, member_day_multiplier)，不叠加 👤
- **积分抵现**：抵扣金额 = 使用积分 ÷ deduction_rate（按 rounding 规则取整），受 ceiling 限制 🤖

### 4.4 执行层分类

| 规则 | 执行层 |
|---|---|
| 积分获取计算 | 服务端（支付完成后触发）|
| 积分抵扣校验 | 服务端校验 + 客户端预计算展示 |
| 积分清零 | 定时任务（按 expiry_frequency）|
| 商品范围过滤 | 服务端（下单时判断）|

## 九、工程建议

1. **配置版本化**：积分权益配置变更建议记录历史版本，便于追溯积分计算依据 🤖
2. **接口幂等性**：保存接口设计为幂等（同一 merchant_id 覆盖更新）🤖
3. **时间段验证**：服务端校验 end_time > start_time，且多段时间段不重叠 🤖
4. **明细行排序**：明细行建议增加 sort_order，保证展示顺序一致 🤖
5. **MyBatis 性能优化（防内存溢出/OOM）**：采用关联表 `points_benefit_item_rel` 后，若商户圈选了上万个商品，使用 MyBatis 查询配置详情时，**严禁将关联记录 Eager Load（立即加载）或查出全量 List 放入 Java `Config` 实体类中**。建议：
   - **校验时采用 EXISTS 查询**：当订单结算计算某商品是否可用积分时，在数据库层面直接做 `SELECT 1 FROM points_benefit_item_rel WHERE config_id = ? AND item_id = ? LIMIT 1`，判断结果返回布尔值。
   - **Redis 高性能缓存**：若业务高并发，在后台保存此类规则时，将对应的海量 `item_id` 写入 Redis 的 `Set` 结构。结算引擎计算时，使用 `SISMEMBER` （时间复杂度 O(1)）直接在缓存判断商品是否在集合内，彻底让 Java 堆内存减负 🤖
