# 储值方案发券功能分析

> 适用范围：`/deposit-plan/charge/*`、储值方案档位赠券、周期发券、撤销赠券
>
> 关联模块：`nms4cloud-crm-service`、`nms4cloud-crm-api`
>
> 依据：当前代码中的 `DepositPlanChargeService`、`DepositPlanCouponService`、`CrmCouponOpServicePlus`、现有设计文档


## 1. 功能概述

当前储值方案发券功能，已经不是老充值那种“送一张券”的简单模式，而是独立的一套权益发放链路。

它支持：

1. 普通券发放
2. 券包发放
3. 一次性发放
4. 周期性发放
5. 首次立即发放
6. 撤销时按订单作废赠券

整体调用关系如下：

```text
储值方案充值成功
  -> DepositPlanChargeService.onPaymentSuccess(...)
  -> handlePostSuccessBenefits(...)
  -> depositPlanCouponService.handleGiftIssuance(...)
  -> 按档位配置分流
     -> issueOneTime(...)
        -> issueCoupons(...)
        -> issueCouponPackages(...)
     -> createPeriodicSchedule(...)
        -> 后续由定时任务 executeScheduledIssuance(...)
  -> 最终统一进入 CrmCouponOpServicePlus.couponOpAdd(...)
  -> 落库到 crm_coupon_order
```


## 2. 配置模型

储值方案的赠券配置来源于档位规则 `DepositTierRule.TierConfig`，而不是老充值的 `giveCouponId/giveCouponNum`。

核心字段包括：

1. `coupons`

- 直接发放的普通券列表

2. `couponPackages`

- 直接发放的券包列表

3. `isGiftEnabled`

- 是否开启赠券

4. `issueRuleType`

- `ONE_TIME`
- `PERIODIC`

5. `issueCycleType`

- `DAILY`
- `WEEKLY`
- `MONTHLY`

6. `issueCycleValues`

- 周期发放的具体取值

7. `issueTime`

- 周期发放执行时间

8. `issueTimes`

- 总发放次数

9. `firstIssueImmediate`

- 是否首次立即发放

这部分模型能力是完整的，配置表达力已经明显强于老充值链路。


## 3. 发券触发入口

储值方案支付成功后的主编排在：

- [DepositPlanChargeService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanChargeService.java)

支付成功后先做：

1. 余额入账
2. 购买记录更新成功状态
3. 方案销量统计更新

然后异步处理后置权益：

```java
CompletableFuture.runAsync(() -> handlePostSuccessBenefits(task, notifyAttach));
```

后置权益包括：

1. 发券
2. 升级
3. 积分

这里的发券入口是：

```java
depositPlanCouponService.handleGiftIssuance(
    task, tier, card,
    notifyAttach.getPlanLid(), notifyAttach.getTierIndex(), chargeRecordLid);
```


## 4. 发券主流程

发券服务在：

- [DepositPlanCouponService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanCouponService.java)

入口方法：

```java
handleGiftIssuance(...)
```

逻辑很清楚：

1. `isGiftEnabled=false`

- 直接返回，不发券

2. `issueRuleType=ONE_TIME`

- 调 `issueOneTime(...)`

3. `issueRuleType=PERIODIC`

- 调 `createPeriodicSchedule(...)`


## 5. 一次性发券

一次性发券在：

- [DepositPlanCouponService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanCouponService.java#L147)

执行顺序：

1. 先发普通券

```java
issueCoupons(task, tier.getCoupons(), card);
```

2. 再发券包

```java
issueCouponPackages(task, tier.getCouponPackages(), card);
```

### 5.1 普通券发放

普通券发放方法：

- [DepositPlanCouponService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanCouponService.java#L417)

处理方式：

1. 遍历 `tier.getCoupons()`
2. 每张券取：

- `couponLid`
- `issueQtyPerTime`

3. 组装为：

```java
new CouponIssuanceDTO()
    .setCouponCode(coupon.getCouponLid())
    .setCouponNum(coupon.getIssueQtyPerTime())
```

4. 调统一发券服务：

```java
crmCouponOpServicePlus.couponOpAdd(...)
```

### 5.2 普通券有效期处理

普通券发放不是简单拿模板直接发，而是会按当前档位配置重算本次发券实例的有效期。

相关逻辑：

- `buildIssueCouponMap(...)`
- `applyCouponValidity(...)`
- `calculateBeginUseTimeByValidity(...)`
- `calculateEndUseTimeByValidity(...)`

这意味着：

1. 券模板是基础定义
2. 本次储值发出去的券实例，会按储值方案里的礼品配置覆盖生效时间/失效时间

这一点设计是合理的，也是储值方案灵活性的关键。

### 5.3 券包发放

券包发放方法：

- [DepositPlanCouponService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanCouponService.java#L636)

处理方式：

1. 遍历 `tier.getCouponPackages()`
2. 每个券包取：

- `packageLid`
- `packageQty`

3. 同样组装成 `CouponIssuanceDTO`
4. 交给统一发券服务 `couponOpAdd(...)`

这说明当前新储值方案的券包发放，已经和老充值统一到同一个执行引擎里了。


## 6. 周期发券

周期发券入口：

- [DepositPlanCouponService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanCouponService.java#L170)

核心思路是：

1. 不在支付成功时一次性发完所有券
2. 先创建一条周期计划 `CrmDepositCouponSchedule`
3. 把当前档位快照 `tierSnapshot` 存进去
4. 后续由定时任务按计划继续发

### 6.1 周期计划保存了什么

计划表里保存的关键数据有：

1. `planLid`
2. `tierIndex`
3. `cardLid`
4. `chargeRecordLid`
5. `tierSnapshot`
6. `issueCycleType`
7. `issueCycleValues`
8. `issueTime`
9. `totalIssueTimes`
10. `issuedTimes`
11. `nextIssueDate`
12. `firstIssueImmediate`

### 6.2 为什么 `tierSnapshot` 很重要

因为它把充值当时的档位配置冻结了。

好处是：

1. 后续方案被修改，不会影响历史订单
2. 周期发券可以严格按“购买时约定的权益”继续执行

这是正确设计，必须保留。

### 6.3 首次立即发放

如果：

```java
firstIssueImmediate = true
```

那么在创建周期计划时，会先执行一次：

```java
issueOneTime(task, tier, card);
schedule.setIssuedTimes(1);
```

然后剩余次数交给后续计划。

### 6.4 周期执行

后续由定时任务调用：

- [DepositPlanCouponService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanCouponService.java#L239)

方法：

```java
executeScheduledIssuance(...)
```

它会：

1. 从 `tierSnapshot` 反序列化出档位配置
2. 构造最小化 `CrmDealTask`
3. 查询会员卡
4. 再调用 `issueOneTime(...)`
5. 更新 `issuedTimes`
6. 计算下一次发放日期
7. 若次数完成则置为完成状态


## 7. 统一发券引擎

无论是：

1. 普通券
2. 券包
3. 老充值
4. 新储值方案

最终都统一走：

- [CrmCouponOpServicePlus.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/card/CrmCouponOpServicePlus.java)

统一入口：

```java
couponOpAdd(CouponOpAddDTO)
```

这层统一带来的好处：

1. 普通券和券包不会各写一套落库逻辑
2. 券包展开、子券生成、库存处理都收口到同一层
3. 最终都统一落表 `crm_coupon_order`

这部分架构是对的。


## 8. 当前实现的优点

## 8.1 功能能力完整

相比老充值，你的储值方案发券能力明显更强：

1. 支持多张普通券
2. 支持多个券包
3. 支持一次性和周期性
4. 支持首发立即发
5. 支持和升级、积分一起作为组合权益

## 8.2 新老链路最终统一

虽然业务编排不同，但最终都复用统一发券引擎，避免了两套发券执行器并行维护。

## 8.3 周期发券使用快照

`tierSnapshot` 的设计保证了历史订单权益不受后续方案修改影响，这是非常关键的正确点。

## 8.4 普通券实例有效期支持动态覆盖

普通券发放时会基于储值方案配置重新计算有效期，而不是机械复制券模板，这让产品配置更灵活。


## 9. 当前实现的主要风险

## 9.1 发券是异步后置逻辑

在：

- [DepositPlanChargeService.java](/D:/mywork/nms4cloud/nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/DepositPlanChargeService.java#L596)

发券、升级、积分是在：

```java
CompletableFuture.runAsync(...)
```

里异步执行的。

这意味着：

1. 余额成功入账后，主流程就算成功
2. 发券失败不会回滚余额
3. 用户可能出现“充值成功但券没到账”

这不是代码 bug，而是当前事务策略的业务取舍。

### 风险结论

如果没有补偿机制，这会成为线上真实问题。


## 9.2 发券失败只打日志

普通券发放里：

```java
catch (Exception e) {
  log.error(...)
}
```

券包发放里也是同样处理。

问题在于：

1. 失败没有回写业务状态
2. 失败没有单独落补偿记录
3. 后续只能查日志排查

这会导致运营和开发都很难快速确认：

- 哪一单失败了
- 失败了哪类权益
- 是否补发过


## 9.3 周期发券依赖定时任务稳定执行

`executeScheduledIssuance(...)` 本身逻辑没问题，但周期计划一定依赖定时任务的可靠性。

潜在风险：

1. 定时任务漏跑
2. 定时任务重复跑
3. 多实例并发执行同一计划
4. 执行中途失败没有明确补偿机制

如果没有额外分布式锁、状态 CAS 或批次幂等保护，就容易出现重复发或漏发。


## 9.4 配置错误会在运行时暴露

例如：

1. `couponLid` 不存在
2. `packageLid` 不存在
3. 券模板状态异常

这些问题有一部分是在发券执行时才抛出来，而不是在方案配置保存时全部拦截。

结果就是：

- 方案能配
- 充值能成功
- 发券阶段才失败


## 9.5 撤销边界需要业务明确

当前设计里，储值方案撤销时：

1. 余额会回滚
2. 赠券会按订单作废
3. 升级、积分、销量统计不回滚

这个边界在技术上是可实现的，但必须业务明确接受。否则后续会被当成缺陷反复追问。


## 10. 我对这套功能的判断

如果只问“功能是不是能用”，答案是：

- 能用，而且设计总体是合理的

如果问“是不是已经足够稳”，答案是：

- 主流程稳
- 权益补偿能力还不够强

更准确地说：

1. 配置能力没问题
2. 发券编排没问题
3. 统一发券引擎复用方式没问题
4. 风险集中在异步失败后的可追踪和可补偿


## 11. 建议优化项

## 11.1 给发券结果落状态

不要只打日志，建议至少记录：

1. `taskLid`
2. `planLid`
3. `tierIndex`
4. `chargeRecordLid`
5. 发券状态
6. 失败原因
7. 最近重试时间

这样可以形成“待补偿清单”。

## 11.2 增加补发机制

建议增加后台补发能力，至少支持：

1. 按 `taskLid` 补发
2. 按 `chargeRecordLid` 补发
3. 只补普通券
4. 只补券包
5. 只补周期计划某一批次

## 11.3 周期计划增加幂等保护

建议周期执行增加：

1. 分布式锁
2. 批次唯一约束
3. 执行状态标记
4. 失败重试次数

否则多实例环境下容易重复发券。

## 11.4 方案保存时加强预校验

建议在方案保存或启用时预校验：

1. 券是否存在
2. 券包是否存在
3. 券是否启用
4. 券包映射是否完整

尽量把问题拦在“配置阶段”，不要让它延迟到“充值成功后”。

## 11.5 明确撤销边界

建议把以下内容写进正式技术文档和验收标准：

1. 撤销回滚余额
2. 撤销作废赠券
3. 不回滚升级
4. 不回滚积分
5. 不回滚销量统计


## 12. 最终结论

你的储值方案发券功能，从架构和代码组织上看，已经具备上线级别的主流程能力：

1. 模型完整
2. 流程清晰
3. 新老链路统一到同一发券引擎
4. 支持普通券、券包、一次性、周期性、首发立即发

当前最主要的问题不是“不会发”，而是：

1. 发券失败后如何补偿
2. 周期任务如何防重、防漏
3. 业务边界如何对外说清楚

一句话总结：

- 主功能设计是对的。
- 可靠性建设还需要补一层。
