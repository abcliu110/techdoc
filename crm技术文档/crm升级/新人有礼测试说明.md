# 新人有礼测试说明

## 1. 测试目标

验证“新人有礼/新人礼包”从配置、生效、注册新会员、发券到领取记录落库的完整链路。

本测试重点确认：

1. 新人礼包配置在有效期内可以被命中。
2. 新会员注册成功后，按配置自动发放新人礼包。
3. 同一商户同一手机号只能领取一次新人礼包。
4. 会员侧通过礼包入口手动领取时，也能正常发券。
5. 领取记录、会员卡、发券记录可追溯。

## 2. 相关入口

后台配置入口：

```text
商户中心 -> 会员营销 -> 卡券营销 -> 新人礼包
```

后台接口：

```text
POST /api/scrm/crm_new_gift/list
POST /api/scrm/crm_new_gift/add
POST /api/scrm/crm_new_gift/update
POST /api/scrm/crm_new_gift/get
```

会员侧接口：

```text
POST /api/scrm/crm_new_gift/get_customer_gift
POST /api/scrm/crm_new_gift/take_gift
```

临时注册测试接口：

```text
POST /api/scrm/user/wxLoginTmp
```

核心后端逻辑：

```text
CrmNewGiftServicePlus.giftCoupon()
UserLoginService.login()
UserLoginService.register()
CrmNewGiftController.takeGift()
```

## 3. 测试前准备

### 3.1 新人礼包配置

在后台新增或确认一条有效的新人礼包：

```text
新人礼包名称：测试新人有礼
起始时间：早于当前时间
截止时间：晚于当前时间
券名称/券编号：必须选择一个有效优惠券或券包
领取数量：建议先填 1
适用门店：包含当前测试门店
```

注意：发放时后端会查询当前商户下有效礼包，要求：

```text
use_begin_time <= 当前时间
use_end_time > 当前时间
coupon_code 不为空
```

如果同一商户同时存在多条有效新人礼包，实际命中结果可能受查询排序和数据影响。测试时建议只保留一条当前有效的测试礼包，或确保目标礼包时间范围最明确。

### 3.2 自动发放开关

注册新会员后自动发放新人礼包，需要满足以下任一条件：

```text
商户 mid = 171696376262485939
或 商户配置 g_memberRegisterGiftCoupon = true
```

普通测试商户建议先确认配置项：

```text
g_memberRegisterGiftCoupon = true
sid = -1
```

如果该开关未开启，新会员注册仍会成功，但注册后的异步新人礼包发放会直接跳过。

### 3.3 会员开卡费用

注册链路会先检查会员开卡费用。如果当前默认会员卡类型需要付费，接口会返回“需要支付开卡费才能注册会员”，新人礼包不会发放。

测试前建议使用无需开卡费的默认会员卡类型，或在测试环境临时调整默认卡类型配置。

### 3.4 测试手机号规则

使用虚假的测试手机号时，必须保证 11 位且每次唯一。

推荐格式：

```text
199 + 时间戳后 8 位
示例：19962415301
```

同一次测试还要保证以下字段唯一：

```text
phone
openId
unionId
```

原因：注册逻辑会先按 `phone`、`openId`、`unionId` 查询 `crm_card_map` 和 `crm_card`。只要命中已有会员，就不会再创建新会员，也不会按“新会员注册”链路测试。

## 4. 测试方式一：通过临时注册接口测试自动发放

### 4.1 请求

```http
POST /api/scrm/user/wxLoginTmp
Content-Type: application/json
```

请求体示例：

```json
{
  "mid": 你的商户ID,
  "cardType": "WECHAT_MINI",
  "appId": "test-appid",
  "openId": "test-openid-20260624153001",
  "unionId": "test-unionid-20260624153001",
  "phone": "19962415301",
  "name": "新人有礼测试",
  "notNeedPhone": false,
  "needSaveOpenId": true
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `mid` | 当前测试商户 ID |
| `cardType` | 建议使用 `WECHAT_MINI` |
| `appId` | 测试 appId，可使用固定测试值 |
| `openId` | 必须唯一 |
| `unionId` | 必须唯一 |
| `phone` | 必须唯一，11 位测试手机号 |
| `name` | 测试会员名称 |
| `notNeedPhone` | 必须为 `false`，确保走手机号注册 |
| `needSaveOpenId` | 建议为 `true`，便于保存映射关系 |

### 4.2 预期结果

接口返回成功，并返回会员信息，至少应包含：

```text
cardLid
cardId
phone
newed = true
```

注册完成后，后端会在 `finally` 中异步执行新人礼包发放。因为是异步逻辑，接口返回后建议等待 1-3 秒再查库。

## 5. 测试方式二：通过会员侧手动领取

适用于验证用户点击“领取新人礼包”的场景。

### 5.1 获取礼包详情

```http
POST /api/scrm/crm_new_gift/get_customer_gift
Content-Type: application/json
```

请求体：

```json
{
  "mid": 你的商户ID,
  "lid": "新人礼包lid"
}
```

预期返回礼包详情和 `couponList`。

### 5.2 手动领取礼包

```http
POST /api/scrm/crm_new_gift/take_gift
Content-Type: application/json
```

请求体：

```json
{
  "mid": 你的商户ID,
  "lid": "新人礼包lid"
}
```

注意：该接口依赖当前登录的会员用户。用户必须已经开通会员卡，且会员手机号不能为空。否则会返回：

```text
需要开通会员卡后才能领取礼包
或 手机号码为空不能领取礼包
```

## 6. 数据验证

将下面 SQL 中的 `?` 替换为测试商户 ID，手机号替换为本次测试手机号。

### 6.1 验证会员已创建

```sql
select *
from crm_member
where mid = ?
  and phone = '19962415301';
```

预期：存在一条新会员记录。

### 6.2 验证会员卡已创建

```sql
select *
from crm_card
where mid = ?
  and phone = '19962415301';
```

预期：存在一条会员卡记录，且 `lid` 与接口返回的 `cardLid` 一致。

### 6.3 验证登录映射已创建

```sql
select *
from crm_card_map
where mid = ?
  and id in (
    '19962415301',
    'test-openid-20260624153001',
    'test-unionid-20260624153001'
  );
```

预期：存在手机号、openId、unionId 对应的映射记录，并关联同一个 `card_lid`。

### 6.4 验证新人礼包领取记录

```sql
select *
from crm_member_gift_record
where mid = ?
  and member_phone = '19962415301'
order by created_time desc;
```

预期：

```text
存在一条领取记录
new_gift 为新人礼包名称
new_gift_code 为新人礼包 lid
gift_coupon_code 为被发放的券或券包编号
gift_num 为领取数量
```

### 6.5 验证优惠券发放记录

根据本项目实际券记录表查询该会员卡的发券记录，重点检查：

```text
会员卡 lid = 注册返回的 cardLid
渠道 = NewGift / 新人礼包
订单号或来源单号 = crm_member_gift_record.lid
券编号 = gift_coupon_code
```

如果发的是券包，还需要确认券包子券是否展开发放，数量是否与券包配置一致。

## 7. 重复领取测试

使用同一个 `mid + phone` 再执行一次注册或领取。

预期：

```text
不会重复发放新人礼包
crm_member_gift_record 不新增第二条同手机号记录
接口或日志提示已经领取礼包，不能重复领取
```

后端重复判断条件：

```text
crm_member_gift_record.mid = 当前商户
crm_member_gift_record.member_phone = 当前手机号
```

## 8. 常见失败原因

| 现象 | 可能原因 | 检查方式 |
|---|---|---|
| 注册成功但未发新人礼包 | `g_memberRegisterGiftCoupon` 未开启 | 查商户配置 |
| 注册接口提示需要支付开卡费 | 默认会员卡类型有开卡费 | 检查默认卡类型配置 |
| 提示礼包已失效 | 当前没有有效礼包或 `couponCode` 为空 | 查 `crm_new_gift` |
| 提示已经领取礼包 | 该手机号已有领取记录 | 查 `crm_member_gift_record` |
| 没有创建新会员 | `phone/openId/unionId` 命中了已有会员 | 查 `crm_card_map`、`crm_card` |
| 手动领取提示手机号为空 | 当前登录会员没有手机号 | 查 `crm_card.phone` |
| 手动领取提示需开通会员卡 | 当前用户没有 `cardLid` | 检查登录态返回 |

## 9. 清理测试数据

测试环境可以按本次测试手机号清理数据。清理前先记录 `card_lid`、`member_lid`、`gift_record_lid` 和券记录，避免误删其他测试数据。

建议清理顺序：

```text
1. 删除或作废该会员卡发放的测试优惠券记录
2. 删除 crm_member_gift_record 中该手机号的测试记录
3. 删除 crm_card_map 中该手机号/openId/unionId 的映射
4. 删除 crm_card 中该手机号的测试会员卡
5. 删除 crm_member 中该手机号的测试会员
```

不要在生产环境直接删除测试数据。如果必须在线上验证，只能使用明确授权的测试商户、测试手机号和可审计的清理方案。

## 10. 最小回归用例

每次调整新人礼包相关代码后，至少执行以下回归：

1. 新手机号注册新会员，自动发放新人礼包成功。
2. 同一手机号重复注册或重复领取，不重复发放。
3. 礼包过期后，新会员注册不发放该礼包。
4. 未配置 `couponCode` 的礼包不会发放。
5. 关闭 `g_memberRegisterGiftCoupon` 后，普通商户注册不自动发放。
6. 会员侧 `take_gift` 在已开卡且有手机号时可以领取。
7. 会员侧 `take_gift` 在无手机号或无会员卡时拒绝领取。
