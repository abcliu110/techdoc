## 一、总体结论

当前测试平台的方向是正确的：用声明式目录描述场景，用 flow 层编排流程，用 helper/action 层复用 UI 操作。这个结构比每个场景自己写一套 UI 操作更灵活，本质上接近组合模式：场景只组合节点，节点复用通用 UI 行为，平台负责失败记录、重试、轮询和执行报告。

已固化的能力：

- 场景目录（`scenarios/*.json`）仍是单一事实源，支持 `--print-plan` / `catalog:check` 干跑校验。
- Web 平台入口和 CLI 入口已共用 `scripts/platform-runner-core.mjs`，避免两套套件执行逻辑漂移。
- 套件默认失败后继续执行，并落盘 `results.jsonl`、`failures.jsonl`、`summary.json`；需要失败即停时显式使用 `--stop-on-failure`。
- flow 通用层支持节点级重试、失败后继续、异步证据轮询、Mermaid 可视化路径图。
- 场景层守门测试继续强制：POS e2e spec 只能进入 flow 编排，不能直接定义 locator/click/fill 等 raw UI 操作，也不能定义新的可复用 UI 组件。
- 商户后台会员详情核对已按“先登录 -> 再进入基础信息/会员档案 -> 查询会员 -> 打开详情”建模，不依赖直接访问详情 URL。

结论：架构主线可以继续使用，不建议全面重写。现有 helper/action 是可复用资产，应逐步拆分和收敛，而不是推倒重来。

## 二、已修复问题

### P0-1【已修复】`startSuiteRun` 使用未定义变量 `id`

- 原位置：`scripts/platform-server.mjs`
- 原问题：HTTP 平台触发 `POST /api/suite/run` 时，套件成功路径中使用 `runs.set(id, run)`，`id` 未定义，会导致 ReferenceError。
- 当前状态：已改为 `runs.set(run.id, run)`，并补充 `npm run platform:check` 中的 HTTP 套件成功路径检查。
- 验证：`npm run platform:check` 已通过。

### P0-2【已修复】套件失败即停，不符合“失败记录并继续”

- 原问题：用户要求测试不过直接跳过/记录，然后继续下一个测试；原 `user-full-combo`、`user-member-benefit-regression` 仍是 fail-fast 行为。
- 当前状态：
  - `user-full-combo.stopOnFailure=false`
  - `user-member-benefit-regression.stopOnFailure=false`
  - 对应 case 参数移除 `--fail-fast`
  - `platform-suite-runner.mjs` 默认继续执行，失败写入 `failures.jsonl`
- 验证：`tests/platform/platform-runner.spec.ts`、`tests/pos/pos-matrix-runner.spec.ts` 已覆盖。

### P0-3【已修复】平台缺少通用 flow 能力

- 原问题：节点重试、异步后台数据轮询、失败后继续、路径可视化如果散落在场景里，会重新走回“每个场景自己写 UI 操作”的老路。
- 当前状态：统一放在 `tests/flow/ui-flow.ts`：
  - `retry`
  - `continueOnError`
  - `pollUiEvidence`
  - `renderUiTreeMermaid`
- 验证：`tests/flow/ui-flow.spec.ts` 已覆盖。

## 三、仍需关注的问题

### P1-1【高风险】源码仍有真实域名/默认凭据

- 文件：`tests/helpers/env.ts`
- 风险：默认远程域名和密码存在源码默认值，容易误连真实环境或泄露凭据。
- 建议：把真实域名和密码完全迁移到 `.env`，源码只保留空默认值或安全测试默认值；缺失时用 required env 明确报错。
- 当前未改原因：这是环境配置变更，可能直接影响当前真实业务测试执行。建议在当前业务验证完成后单独处理。

### P2-1【中】helper 文件过大

- 典型文件：
  - `tests/helpers/pos-member-discount-consume-report-actions.ts`
  - `tests/helpers/pos-member-recharge-wechat-actions.ts`
  - `tests/helpers/pos-physical-coupon-actions.ts`
  - `tests/helpers/pos-member-recharge-batch-actions.ts`
- 风险：helper 已经承担了太多动作细节，长期会难维护。
- 建议：按业务子流程拆分：结账、支付策略、后置动作、商户后台会员详情核对、BI/报表核对、实物券核销。
- 边界：拆分后仍属于通用 helper/action 层，不能把 UI 操作退回到场景 spec。

### P2-2【中】部分测试仍是源码字符串断言

- 文件：`tests/pos/pos-matrix-runner.spec.ts`
- 说明：架构守门类字符串检查有价值，例如禁止 scenario spec 直接写 UI 操作；但部分对 helper 内部实现文本的断言过细，会阻碍正常重构。
- 建议：保留架构守门；逐步把实现文本断言替换为 `--print-plan`、`--print-cases`、运行结果文件等行为断言。

### P3-1【低】执行链仍较深

- 当前链路：platform -> suite runner -> matrix runner -> Playwright spec -> flow -> helper/action。
- 说明：这不是阻断问题，分层本身合理；问题是入口多、调试路径长。
- 建议：保留平台主入口，把其他 runner 的定位写清楚；长期减少重复入口。

### P3-2【低】Web 平台运行历史仍以内存为主

- 文件：`scripts/platform-server.mjs`
- 风险：HTTP server 重启后 `/api/runs` 不会重建历史。
- 建议：启动时扫描 `.tmp/platform-runs/*/summary.json`，恢复历史列表；或者在 UI 上明确只展示当前进程运行记录。
