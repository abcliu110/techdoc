# 幂等键问题

## 背景

POS 储值充值接口 `/deposit-plan/charge/commitByPos` 使用 POS 订单号 `orderId` 作为 CRM 交易任务表 `crm_deal_task.out_trade_no`，用于充值任务幂等。

问题场景是：同一商户下不同门店可能产生相同的 POS 订单号。如果 CRM 侧幂等判断只按 `mid + orderId` 查询，就可能把 A 门店的充值任务误判为 B 门店的同一笔订单。

## 影响接口

- 后端模块：`nms4cloud-app/2_business/nms4cloud-crm`
- 服务类：`DepositPlanChargeService`
- 方法：`commitByPos(DepositPlanPosChargeDTO request)`
- 接口入口：`DepositPlanChargeController`

`DepositPlanChargeController` 会把 POS 签名信息写入请求对象：

- `merchantNo` 写入 `mid`
- `terminalId` 写入 `sid`

因此 POS 储值充值幂等键具备门店维度，不能只使用商户维度。

## 根因

`commitByPos` 的幂等查询如果只按下面条件命中旧任务：

- `CrmDealTask.mid = request.mid`
- `CrmDealTask.outTradeNo = request.orderId`

那么同一商户不同门店共用同一个 `orderId` 时，会命中其他门店的任务。

命中后的表现取决于旧任务状态：

- 旧任务为 `SUCCESS`：直接返回旧任务余额结果，导致当前门店未真正充值但接口看起来成功。
- 旧任务为 `USERPAYING`：复用旧任务进入阶段 2 重试，可能把当前门店请求绑定到其他门店任务上。
- 其他状态：返回“订单处理中，请勿重复提交”。

## 修复方案

POS 储值充值的幂等范围调整为同一门店内唯一：

```text
mid + sid + orderId
```

对应代码修复点：

1. Redis 锁 key 增加 `sid`：

```java
String lockKey =
    "depositPlan:pos:charge:"
        + request.getMid()
        + ":"
        + request.getSid()
        + ":"
        + request.getOrderId();
```

2. `CrmDealTask` 幂等查询增加 `sid` 条件：

```java
.eq(CrmDealTask::getMid, request.getMid())
.eq(CrmDealTask::getSid, request.getSid())
.eq(CrmDealTask::getOutTradeNo, request.getOrderId())
```

## 保留的旧逻辑

同一门店内的同一 `orderId` 仍保持原来的三态幂等处理：

- `SUCCESS`：直接返回已有充值任务结果。
- `USERPAYING`：调用 `retryExistingPosCharge(request, existing)` 重放阶段 2，避免已收款订单卡住。
- 其他状态：提示订单处理中，禁止重复提交。

这次修改只收紧 POS 幂等命中范围，不改变旧余额入账、储值权益发放、任务状态流转逻辑。

## 表索引核查

仓库内 `crm.pdma.json` 模型中查到两个相关索引：

- `idx_mid_out_trade_no`
- `idx_report_date_mid_out_trade_no`

核查结果：

- `idx_mid_out_trade_no` 是普通索引，`unique: false`，字段为 `mid + out_trade_no`。
- `idx_report_date_mid_out_trade_no` 也是普通索引，`unique: false`。
- 仓库内迁移脚本未发现 `crm_deal_task.out_trade_no` 相关唯一索引。

因此从仓库模型和迁移脚本看，不存在会阻止不同门店使用相同 `orderId/out_trade_no` 的唯一索引。

注意：以上结论未连接生产库执行 `SHOW INDEX FROM crm_deal_task`，如果生产库曾手工加过索引，需要 DBA 额外核查。

## 后续建议

1. 在生产库执行只读索引核查：

```sql
SHOW INDEX FROM crm_deal_task;
```

重点确认是否存在未纳入仓库模型的唯一索引，例如：

- `UNIQUE(mid, out_trade_no)`
- `UNIQUE(report_date, mid, out_trade_no)`

2. 如果未来要在数据库层增加 POS 充值幂等唯一约束，应使用包含门店维度的组合键，例如：

```sql
UNIQUE KEY uk_crm_deal_task_pos_order (mid, sid, out_trade_no, deleted)
```

上线前必须先清洗或确认历史数据，否则可能因为旧重复数据导致加索引失败。

## 本次教训

1. 幂等键不能只按“字段名看起来像订单号”来设计，必须先确认业务唯一域。

   POS 的 `orderId` 是门店侧订单号，不天然保证在同一商户下全局唯一。对 CRM 来说，`mid + orderId` 不是可靠的 POS 充值幂等键，真实业务唯一域应至少包含 `sid`，也就是 `mid + sid + orderId`。

2. 看到 `out_trade_no` 不能直接套用支付系统的全局唯一假设。

   在微信、支付宝等支付链路里，外部交易号通常会被设计成商户维度或平台维度唯一；但这里的 `out_trade_no` 来自 POS 订单号映射，来源系统的唯一性规则不同。跨系统字段复用时，要以来源系统的生成规则为准。

3. 排查幂等问题时，要同时看三层证据。

   - 请求入口：确认 `mid`、`sid`、`orderId` 分别从哪里来。
   - 业务查询：确认代码实际用哪些字段做幂等命中。
   - 数据约束：确认数据库唯一索引是否真的约束了同一业务域。

   只看其中一层容易误判。例如只看到 `idx_mid_out_trade_no` 这个索引名，不能推断它是唯一索引；必须确认 `unique: false/true` 或真实库的 `SHOW INDEX` 输出。

4. 普通索引只影响查询效率，不提供幂等安全。

   本次仓库模型里的 `idx_mid_out_trade_no` 和 `idx_report_date_mid_out_trade_no` 都是普通索引。它们不会阻止同一商户不同门店写入相同 `out_trade_no`，所以不能把索引存在当成“数据库已经兜底”。

5. Redis 锁和数据库查询必须使用同一个业务幂等域。

   如果锁 key 是 `mid + orderId`，查询是 `mid + sid + orderId`，或者反过来，都会留下并发或误命中的边界问题。本次修复要求两处都统一为 `mid + sid + orderId`。

6. 保守修复优先缩小命中范围，不要顺手改旧流程。

   旧逻辑中 `SUCCESS` 直接返回、`USERPAYING` 重试阶段 2，是已有业务恢复机制。问题只在“命中了谁”，不是“命中后怎么处理”。因此本次只加 `sid` 隔离，不重写充值状态流转。

7. 以后遇到 POS、门店、终端相关单号，默认先问“是否门店内唯一”。

   对 POS 场景，订单号、轮次号、终端流水号、离线单号都可能只在门店、设备或营业日范围内唯一。接入 CRM、订单、支付、积分、发券等中心化服务时，幂等键要显式带上这些范围字段，不能只依赖单号本身。

## 回滚方案

如果新逻辑出现兼容问题，代码层可回滚为旧的 `mid + orderId` 幂等范围。但回滚会重新暴露跨门店同订单号误命中的风险。
