## 2. 已确认 Bug：消费赠券每满场景少发券

### 2.1 现象

用例名称：

`coupon B every-full + capped rate cashback, member-card pay`

规则配置：

| 字段 | 值 |
|---|---|
| 消费赠券金额口径 | `B` |
| 支付方式 | 会员卡支付 |
| 消费金额 | 100 |
| 满赠金额 | 50 |
| 是否每满递增 | 是 |
| 每满金额 | 50 |
| 每档赠券数量 | 1 |
| 最多赠券数量 | 2 |
| 消费返现类型 | 比例返现 |
| 返现比例 | 10% |
| 返现封顶 | 5 |

按规则计算：

```text
floor(100 / 50) * 1 = 2
最多赠券数量 = 2
期望本次消费新增券数量 = 2
```

实际结果：

- 60 秒内 `member_coupon_issue_detail` 只匹配到 1 张新增消费赠券。
- 矩阵测试连续两次复现同一问题。
- 外层矩阵测试没有被阻断，已按“确认程序问题后记录并继续”的规则写入 E2E 已知问题文件。

### 2.2 失败报表

失败报表：

`/api/bi/ck/member_coupon_issue_detail`

报表名称：

会员券发行明细 / 券发行明细

断言失败摘要：

```text
member coupon issue detail report after consume 在 60000ms 内没有返回足够的匹配数据行。
期望匹配行数至少 = 2。
实际新增匹配行数 = 1。
```

本问题也会影响：

- 券发行数量汇总
- 依赖 `crm_coupon_order` 新增券记录数量的后续核销测试
- 会员消费权益对账

### 2.3 为什么判定为程序 Bug

这不是 token、报表查询条件或测试配置问题，原因如下：

1. 同一个矩阵中其他消费赠券用例可以正常新增 1 张券。
2. 该用例消费金额、满赠金额、每满金额、赠券数量、上限都满足发 2 张的计算条件。
3. 报表接口能查到本次消费赠送的券，只是数量不足。
4. 券模板本身能持续发券，测试中同一券模板已有多条“消费赠送”记录。
5. 消费返现同用例配置不是失败点，失败集中在券发行数量。

### 2.4 初步怀疑方向

需要继续排查下面链路：

1. POS 本地消费赠券 round 里记录的 `couponNum` 是否已经是 2。
2. CRM 消费赠券幂等记录 `crm_consume_coupon_op_record.coupon_num` 是否为 2。
3. CRM 实际调用发券服务时传入的数量是否为 2。
4. `CrmCouponOpServicePlus.toNormal(..., number)` 是否在当前场景完整循环创建了 2 条 `crm_coupon_order`。
5. 券模板领取限制、库存限制或幂等键是否在同一笔消费的第二张券上发生拦截。
6. 报表 `member_coupon_issue_detail` 是否只按某个维度展示了 1 条，导致 `crm_coupon_order` 中实际已有 2 条但报表聚合/过滤后少展示。

建议优先查表：

```sql
-- CRM 消费赠券执行记录
SELECT order_id, card_no, phone, status_, rule_lid, coupon_lid, coupon_num,
       request_json, response_json, remark, create_time
FROM crm_consume_coupon_op_record
WHERE card_no = '18923865943'
ORDER BY create_time DESC
LIMIT 20;

-- 实际券发行记录
SELECT lid, mid, coupon_code, unionid, get_channel, get_date, coupon_status
FROM crm_coupon_order
WHERE unionid = '18923865943'
ORDER BY get_date DESC
LIMIT 50;
```

## 3. 消费返现测试结论

本轮矩阵测试中，消费返现没有确认程序 Bug。

已覆盖场景：

- 固定金额返现。
- 比例返现。
- 比例返现封顶。
- 会员卡支付返现。
- 现金支付绑定会员返现。
- 混合支付下与消费赠券同时触发。

当前测试期望已按真实报表行为调整：

- 现金支付绑定会员时，会员消费详情不新增会员卡消费行。
- 如果触发返现，会员账户详情会新增 `CashBack` 流水。
- 会员账户汇总、会员余额汇总按返现导致的余额变化核对。

## 5. 后续处理建议

1. 先用数据库确认 `crm_consume_coupon_op_record.coupon_num` 与 `crm_coupon_order` 实际新增数量。
2. 如果 `coupon_num = 2` 但 `crm_coupon_order` 只有 1 条，重点查 CRM 发券服务和券模板限制。
3. 如果 `coupon_num = 1`，重点查消费赠券规则计算逻辑，尤其是 `amountMethod = B` 且 `giftByIncrease = true` 的分支。
4. 如果 `crm_coupon_order` 有 2 条但报表只有 1 条，重点查 POS 报表 `member_coupon_issue_detail` 的过滤、分页、去重或分组逻辑。
5. 修复后保留该矩阵用例作为回归测试，避免每满递增再次退化成只发 1 张。
