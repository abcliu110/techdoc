# 乐观锁机制详解

生成时间：2026-03-09

---

## 🔒 什么是乐观锁

### 基本概念

**乐观锁（Optimistic Locking）**是一种并发控制机制，假设多个事务在处理数据时不会发生冲突，只在提交时检查是否有冲突。

**对比悲观锁：**
- **悲观锁**：假设一定会冲突，先加锁再操作（如 `SELECT ... FOR UPDATE`）
- **乐观锁**：假设不会冲突，操作时不加锁，提交时检查版本号

## 🎯 在储值方案中的作用

### 使用场景：销售统计更新

```java
// CrmDepositPlan.java
@TableField("revision")
@Version  // 乐观锁注解
private Integer revision;

@TableField("total_sold_count")
private Integer totalSoldCount;  // 已售总数量

@TableField("total_sold_amount")
private BigDecimal totalSoldAmount;  // 已售总金额
```

### 问题场景

**假设没有乐观锁：**

```
时间线：
T1: 用户A购买套餐，读取 total_sold_count = 100
T2: 用户B购买套餐，读取 total_sold_count = 100
T3: 用户A更新 total_sold_count = 101
T4: 用户B更新 total_sold_count = 101  ❌ 错误！应该是102

结果：丢失了一次更新，统计数据不准确
```

**使用乐观锁后：**

```
时间线：
T1: 用户A购买套餐，读取 total_sold_count = 100, revision = 5
T2: 用户B购买套餐，读取 total_sold_count = 100, revision = 5
T3: 用户A更新 total_sold_count = 101, revision = 6 ✅ 成功
T4: 用户B更新 total_sold_count = 101, revision = 6 ❌ 失败（revision不匹配）
T5: 用户B重试，读取 total_sold_count = 101, revision = 6
T6: 用户B更新 total_sold_count = 102, revision = 7 ✅ 成功

结果：数据准确，没有丢失更新
```

## 🔧 工作原理

### 1. MyBatis-Plus 自动处理

```java
// 更新时 MyBatis-Plus 自动生成的 SQL
UPDATE crm_deposit_plan
SET total_sold_count = 101,
    total_sold_amount = 10100.00,
    revision = 6                    -- 版本号+1
WHERE lid = 100001
  AND revision = 5                  -- 必须匹配当前版本号
```

**关键点：**
- WHERE 条件包含 `revision = 5`（当前版本）
- SET 语句包含 `revision = 6`（新版本）
- 如果 revision 不匹配，更新失败（影响行数 = 0）

### 2. 更新方法实现

```java
// CrmDepositPlanServicePlus.java
public NmsResult<String> updateSoldStatistics(Long planLid, BigDecimal amount) {
    // 1. 查询当前数据（包含 revision）
    CrmDepositPlan plan = iService.getOne(
        new QueryWrapper<CrmDepositPlan>().lambda()
            .eq(CrmDepositPlan::getLid, planLid)
    );

    // 此时 plan.revision = 5

    // 2. 修改数据
    plan.setTotalSoldCount(plan.getTotalSoldCount() + 1);
    plan.setTotalSoldAmount(plan.getTotalSoldAmount().add(amount));

    // 3. 更新（MyBatis-Plus 自动处理乐观锁）
    boolean success = iService.updateById(plan);

    if (!success) {
        // revision 不匹配，更新失败
        return NmsResult.error("更新失败，请重试");
    }

    return NmsResult.success("更新成功");
}
```

## 📊 实际案例分析

### 案例1：高并发购买场景

**场景描述：**
- 某热门储值套餐，1秒内有100个用户同时购买
- 需要更新 `total_sold_count` 和 `total_sold_amount`

**没有乐观锁：**
```
初始值：total_sold_count = 0
100个并发请求同时读取 count = 0
100个请求都更新为 count = 1
最终结果：count = 1 ❌ 错误！应该是100
```

**使用乐观锁：**
```
初始值：total_sold_count = 0, revision = 1

请求1: 读取 count=0, rev=1 → 更新 count=1, rev=2 ✅ 成功
请求2: 读取 count=0, rev=1 → 更新 count=1, rev=2 ❌ 失败（rev不匹配）
请求2: 重试，读取 count=1, rev=2 → 更新 count=2, rev=3 ✅ 成功
请求3: 读取 count=0, rev=1 → 更新 count=1, rev=2 ❌ 失败
请求3: 重试，读取 count=2, rev=3 → 更新 count=3, rev=4 ✅ 成功
...

最终结果：count = 100 ✅ 正确！
```

### 案例2：限量抢购场景

**场景描述：**
- 套餐限量1000份（`totalSaleLimit = 1000`）
- 需要检查是否超过限量

**代码示例：**
```java
public NmsResult<String> purchaseWithLimit(Long planLid, BigDecimal amount) {
    // 1. 查询当前数据
    CrmDepositPlan plan = getById(planLid);

    // 2. 检查限量
    if (plan.getTotalSoldCount() >= 1000) {
        return NmsResult.error("已售罄");
    }

    // 3. 更新统计（乐观锁保护）
    plan.setTotalSoldCount(plan.getTotalSoldCount() + 1);
    boolean success = updateById(plan);

    if (!success) {
        // 并发冲突，重试
        return purchaseWithLimit(planLid, amount);
    }

    return NmsResult.success("购买成功");
}
```

**乐观锁的作用：**
- 防止超卖：即使并发很高，也不会超过1000份
- 数据准确：每次购买都会正确累加

## ⚡ 性能影响

### 优点

1. **无锁等待**
   - 不需要等待其他事务释放锁
   - 并发性能好

2. **数据库压力小**
   - 不需要维护锁表
   - 只是多了一个 WHERE 条件

3. **死锁风险低**
   - 不会出现死锁问题

### 缺点

1. **冲突时需要重试**
   - 高并发场景下，重试次数可能较多
   - 需要实现重试逻辑

2. **不适合长事务**
   - 事务时间越长，冲突概率越高

## 🔄 重试策略

### 简单重试

```java
public NmsResult<String> updateWithRetry(Long planLid, BigDecimal amount) {
    int maxRetries = 3;
    int retryCount = 0;

    while (retryCount < maxRetries) {
        NmsResult<String> result = updateSoldStatistics(planLid, amount);

        if (result.isSuccess()) {
            return result;
        }

        retryCount++;
        if (retryCount < maxRetries) {
            try {
                // 指数退避：100ms, 200ms, 400ms
                Thread.sleep(100 * (1 << retryCount));
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    return NmsResult.error("更新失败，已重试" + maxRetries + "次");
}
```

### Spring Retry

```java
@Retryable(
    value = {OptimisticLockingFailureException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 100, multiplier = 2)
)
public void updateWithSpringRetry(Long planLid, BigDecimal amount) {
    NmsResult<String> result = updateSoldStatistics(planLid, amount);

    if (!result.isSuccess()) {
        throw new OptimisticLockingFailureException("更新失败");
    }
}
```

## 📈 与其他方案对比

### 方案对比表

| 方案 | 并发性能 | 实现复杂度 | 数据准确性 | 适用场景 |
|------|---------|-----------|-----------|---------|
| **乐观锁** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 读多写少，冲突少 |
| 悲观锁 | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 写多读少，冲突多 |
| Redis 分布式锁 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | 分布式环境 |
| 数据库行锁 | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 单机环境 |
| 无锁（不安全） | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ | 不关心准确性 |

### 为什么选择乐观锁？

**储值方案的特点：**
1. **读多写少**：查询套餐信息频繁，购买相对较少
2. **冲突概率低**：即使热门套餐，也不会每秒几千次购买
3. **性能要求高**：不能因为统计更新影响购买体验
4. **数据准确性要求高**：销售统计必须准确

**结论：** 乐观锁是最佳选择 ✅

## ⚠️ 注意事项

### 1. revision 字段初始化

```sql
-- 必须有默认值
`revision` INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号'
```

### 2. 更新时必须先查询

```java
// ❌ 错误：直接 new 对象更新
CrmDepositPlan plan = new CrmDepositPlan();
plan.setLid(100001L);
plan.setTotalSoldCount(101);
updateById(plan);  // revision 为 null，乐观锁不生效

// ✅ 正确：先查询再更新
CrmDepositPlan plan = getById(100001L);  // 包含 revision
plan.setTotalSoldCount(plan.getTotalSoldCount() + 1);
updateById(plan);  // 乐观锁生效
```

### 3. 批量更新不支持乐观锁

```java
// ❌ 乐观锁不生效
update(new UpdateWrapper<CrmDepositPlan>()
    .eq("lid", 100001L)
    .set("total_sold_count", 101));

// ✅ 必须使用 updateById
CrmDepositPlan plan = getById(100001L);
plan.setTotalSoldCount(101);
updateById(plan);
```

### 4. 事务中的使用

```java
@Transactional
public void purchaseOrder(Long orderId) {
    // 1. 更新订单状态
    orderService.updateStatus(orderId, "PAID");

    // 2. 更新销售统计（乐观锁）
    NmsResult<String> result = depositPlanService.updateSoldStatistics(planLid, amount);

    if (!result.isSuccess()) {
        // 乐观锁失败，整个事务回滚
        throw new RuntimeException("更新统计失败");
    }
}
```

## 📊 总结

### 乐观锁在储值方案中的作用

1. **保证销售统计数据准确** ✅
   - 防止并发更新导致的数据丢失
   - 确保 `total_sold_count` 和 `total_sold_amount` 准确

2. **防止超卖** ✅
   - 配合限量检查，防止超过 `totalSaleLimit`
   - 即使高并发也能保证不超卖

3. **提升并发性能** ✅
   - 无锁等待，性能好
   - 适合读多写少的场景

4. **降低死锁风险** ✅
   - 不需要数据库锁
   - 避免死锁问题

### 关键要点

- ✅ 必须在 `revision` 字段上添加 `@Version` 注解
- ✅ 更新前必须先查询（获取当前 revision）
- ✅ 更新失败时需要实现重试逻辑
- ✅ 适合读多写少、冲突概率低的场景

**乐观锁是保证销售统计数据准确性的关键机制！** 🔒✨
