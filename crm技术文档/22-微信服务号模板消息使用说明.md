# 微信服务号模板消息使用说明

## 一、结论

微信服务号可以发送微信体系内的模板消息，但不能直接发送手机短信模板消息。

服务号模板消息发送到用户的微信会话中，发送对象是当前公众号下的用户 `openid`，不是手机号。若业务需要发送手机短信验证码、短信通知或营销短信，需要另行接入短信服务商，例如腾讯云短信、阿里云短信或运营商短信平台。

## 二、适用场景

服务号模板消息适合发送与用户操作或业务状态强相关的服务通知，例如：

- 订单支付成功通知。
- 订单发货或取餐通知。
- 预约成功、预约变更、预约提醒。
- 审核通过或审核失败通知。
- 会员权益到账、积分变动、储值到账等业务通知。

不适合用于：

- 广告营销。
- 无用户触发背景的批量推广。
- 频繁打扰用户的运营消息。
- 与模板内容不匹配的自定义营销内容。

## 三、前置条件

使用服务号模板消息前，需要满足以下条件：

1. 公众号为已认证服务号。
2. 已在微信公众平台开通模板消息或对应通知能力。
3. 已在微信公众平台选择行业类目并添加模板。
4. 已取得模板对应的 `template_id`。
5. 后端已取得公众号的 `AppID` 和 `AppSecret`。
6. 后端服务器 IP 已加入公众号接口 IP 白名单。
7. 业务系统已能拿到用户在当前服务号下的 `openid`。
8. 用户通常需要已关注该服务号，否则模板消息可能发送失败。

## 四、整体调用流程

整体流程如下：

1. 用户关注服务号或通过公众号网页授权进入业务页面。
2. 业务系统保存用户在该服务号下的 `openid`。
3. 业务发生关键状态变化，例如支付成功、审核完成、权益到账。
4. 后端获取并缓存公众号 `access_token`。
5. 后端按微信模板字段组装模板消息内容。
6. 后端调用微信模板消息发送接口。
7. 微信返回发送结果，业务系统记录 `msgid`、错误码和错误信息。

## 五、获取 openid

模板消息发送对象是 `openid`。

常见获取方式：

- 用户关注服务号时，微信会向服务器推送关注事件，事件中的 `FromUserName` 就是该用户对当前公众号的 `openid`。
- 用户从公众号菜单、模板消息、二维码等入口进入网页时，可通过公众号网页授权获取 `openid`。
- 已有粉丝数据时，也可通过微信用户管理相关接口获取。

注意事项：

- 同一个用户在不同公众号下的 `openid` 不同。
- 不要用手机号、会员 ID、unionid 直接当作模板消息接收人。
- 如果系统内同时存在多个公众号，需要明确当前业务通知使用哪个公众号的 `openid`。

## 六、获取 access_token

后端调用微信接口获取公众号 `access_token`：

```http
GET https://api.weixin.qq.com/cgi-bin/token?grant_type=client_credential&appid=APPID&secret=APPSECRET
```

返回示例：

```json
{
  "access_token": "ACCESS_TOKEN",
  "expires_in": 7200
}
```

处理要求：

- `access_token` 必须由后端统一获取和缓存。
- 不要把 `AppSecret` 或 `access_token` 暴露给前端。
- 不要每次发送消息都重新获取 `access_token`，应按过期时间缓存。
- 如果多个服务都会调用公众号接口，应避免并发刷新导致 token 被覆盖或频繁失效。

## 七、发送模板消息

发送接口：

```http
POST https://api.weixin.qq.com/cgi-bin/message/template/send?access_token=ACCESS_TOKEN
```

请求体示例：

```json
{
  "touser": "用户openid",
  "template_id": "模板ID",
  "url": "https://example.com/order/detail?id=202605210001",
  "data": {
    "first": {
      "value": "您的订单已支付成功"
    },
    "keyword1": {
      "value": "订单号：202605210001"
    },
    "keyword2": {
      "value": "88.00元"
    },
    "keyword3": {
      "value": "2026-05-21 14:30"
    },
    "remark": {
      "value": "点击查看订单详情"
    }
  }
}
```

成功返回示例：

```json
{
  "errcode": 0,
  "errmsg": "ok",
  "msgid": 123456789
}
```

## 八、跳转小程序

如果模板消息点击后需要跳转小程序，可以在请求体中加入 `miniprogram`：

```json
{
  "touser": "用户openid",
  "template_id": "模板ID",
  "miniprogram": {
    "appid": "小程序appid",
    "pagepath": "pages/order/detail?id=202605210001"
  },
  "data": {
    "first": {
      "value": "您的订单已完成"
    },
    "keyword1": {
      "value": "202605210001"
    },
    "remark": {
      "value": "点击查看详情"
    }
  }
}
```

注意：

- 小程序 `appid` 需要与公众号具备合法关联关系。
- `pagepath` 应使用小程序内真实存在的页面路径。
- 如果同时配置网页 `url` 和小程序跳转，需要按微信当前规则确认最终跳转优先级。

## 九、常见错误

| 错误现象 | 常见原因 | 处理方式 |
|---|---|---|
| `invalid credential` | `access_token` 失效或不属于当前公众号 | 重新获取并检查 AppID、AppSecret |
| `require subscribe` | 用户未关注服务号 | 引导用户关注后再发送 |
| `invalid template_id` | 模板 ID 错误或不属于当前公众号 | 检查微信公众平台模板配置 |
| `invalid openid` | openid 与公众号不匹配 | 确认 openid 来源公众号 |
| 内容发送失败 | 字段与模板不匹配 | 按模板字段名称和顺序重新组装 |
| 调用被拒绝 | IP 白名单未配置或接口权限不足 | 检查公众号基本配置和接口权限 |

## 十、CRM 业务接入建议

CRM 场景下建议将模板消息作为会员服务通知能力接入，不要和短信能力混用。

建议设计原则：

1. 会员基础档案中保存服务号 `openid` 绑定关系。
2. 通知发送前先判断会员是否已绑定微信身份。
3. 对支付、储值、积分、券到账等关键事件建立明确的通知触发点。
4. 发送记录应落库，至少记录业务单号、模板 ID、openid、发送结果、错误码、错误信息和微信 `msgid`。
5. 失败后按错误类型处理，用户未关注或 openid 无效不应无限重试。
6. 同一业务事件应做好幂等控制，避免重复发送。
7. 模板文案应保持服务通知属性，避免营销化表达。

## 十一、与短信能力的区别

| 能力 | 服务号模板消息 | 手机短信 |
|---|---|---|
| 到达位置 | 微信会话 | 手机短信收件箱 |
| 接收标识 | openid | 手机号 |
| 发送主体 | 公众号 | 短信服务商或运营商通道 |
| 常见场景 | 订单、预约、权益、审核等服务通知 | 验证码、强提醒、无微信绑定用户通知 |
| 是否可营销 | 不建议，且受微信规则限制 | 需要用户授权并遵守短信合规要求 |
| 失败原因 | 未关注、openid 错误、模板不匹配 | 手机号错误、签名模板审核、通道失败 |

## 十二、落地检查清单

接入前确认：

- 服务号已认证。
- 模板已添加并取得 `template_id`。
- 后端已能稳定获取并缓存 `access_token`。
- 业务系统已保存正确的服务号 `openid`。
- 已配置服务器 IP 白名单。
- 发送内容与模板字段完全匹配。
- 已记录发送日志和错误码。
- 已实现业务幂等，避免重复发送。
- 已区分微信模板消息和手机短信能力。
