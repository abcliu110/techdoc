# 交给其他AI执行的标准指令包

## 1. 使用目的

本文档用于把“既有 Spring Cloud + React 系统的功能源码理解工作”稳定交给其他 AI 执行。

目标是让不同 AI 在同一方法论下工作，减少以下问题：

- 分析口径不一致
- 只看一层代码
- 没有证据支撑
- 输出内容不适合人类阅读
- 主流程清楚但异常流缺失

## 2. 推荐交付方式

把任务交给其他 AI 时，不要只发一句：

- “帮我看下这个功能”

而要一次性给出以下信息：

1. 功能名称
2. 业务背景
3. 用户角色
4. 已知入口
5. 期望输出格式
6. 证据要求
7. 是否允许多轮分析

## 3. 最小可执行输入模板

把下面这段直接发给其他 AI 即可：

```text
请分析既有系统中的【功能名称】。

系统形态：
- 后端：Spring Cloud
- 前端：React

分析目标：
完整理解该功能的前后端实现方式，并输出一份给人类阅读的功能源码理解文档。

已知信息：
- 用户角色：
- 使用场景：
- 已知入口页面/菜单：
- 已知接口/类名（如有）：
- 特别关注点：

执行要求：
1. 必须从“用户动作触发的一条完整业务链路”来理解功能。
2. 必须同时覆盖前端入口、接口契约、后端调用链、业务规则、数据变化、状态流转、权限配置、异常分支。
3. 所有关键结论必须附源码证据，不允许只依据命名猜测。
4. 必须明确区分：已证实 / 高概率推断 / 待验证。
5. 先分析主流程，再补分支流程、异常流程、异步流程。
6. 输出必须适合人类阅读，不要写成零散源码笔记。

输出结构：
1. 功能摘要
2. 用户入口与操作路径
3. 前端实现链路
4. 接口契约与后端调用链
5. 核心业务规则
6. 数据模型与状态流转
7. 配置、权限、租户影响
8. 异常流程与补偿机制
9. 关键源码索引
10. 证据表
11. 未确认问题
12. 总结
```

## 4. 标准强约束版提示词

如果你希望其他 AI 严格按步骤执行，用下面这版：

```text
你现在不是开发者，而是“企业系统功能理解分析师”。

任务目标：
基于现有 Spring Cloud + React 源码，完整理解【功能名称】的实现方式，并输出一份给人类阅读的功能源码理解文档。

你必须按多轮方式执行，不允许跳步。

第0轮：先输出功能定义卡，确认分析边界。
第1轮：只做入口定位，输出前端入口地图和 API 初始清单。
第2轮：只做主链路穿透，输出前后端调用链。
第3轮：只抽取业务规则，输出规则表。
第4轮：只分析数据与状态，输出数据影响表和状态流转。
第5轮：只分析异常与边界，输出异常矩阵和待验证问题。
第6轮：整编成人类可读文档，并附证据表。

工作要求：
1. 不允许猜测业务结论，每个关键结论必须附源码证据。
2. 必须同时分析前端入口、接口契约、后端调用链、数据变化、状态流转、权限配置、异常分支。
3. 先找主流程，再找分支和异常，不要一开始陷入细节。
4. 必须识别同步调用、异步事件、远程服务依赖、缓存、副作用。
5. 明确哪些结论已经证实，哪些只是推断，哪些仍需验证。
6. 输出时优先面向人类理解，而不是面向机器检索。

每轮结束都必须输出：
- 本轮已证实结论
- 本轮待验证问题
- 下一轮目标

最终输出结构：
1. 功能摘要
2. 用户视角
3. 前端实现链路
4. 接口契约
5. 后端调用链
6. 核心业务规则
7. 数据模型
8. 状态流转
9. 配置、权限、租户影响
10. 异常与分支
11. 时序图/流程说明
12. 关键源码索引
13. 证据表
14. 未确认问题
15. 总结
```

## 5. 多Agent版本任务指令

如果其他 AI 支持多 Agent 或多个并行窗口，可以使用下面这套拆解：

```text
请以多 Agent 协作方式分析【功能名称】。

角色分工：
- Lead Agent：负责定范围、分工、合并和最终输出
- Frontend Agent：负责 React 前端入口、交互、状态、API 发起
- Backend Agent：负责 Spring Cloud 后端接口、服务编排、微服务调用、异常
- Data Agent：负责表、字段、状态、缓存、配置、副作用
- Verifier Agent：负责冲突检查、证据补漏、置信度评估

执行要求：
1. Frontend / Backend / Data 先并行输出各自分析结果。
2. Lead Agent 先做一次合并。
3. Verifier Agent 基于合并结果做交叉校验。
4. Lead Agent 再输出最终人类文档。
5. 所有关键结论必须附源码证据。
6. 必须明确标记已证实 / 待验证。
7. 不允许把猜测写成结论。

最终交付：
- 功能源码理解报告
- 证据表
- 冲突与未确认问题清单
```

## 6. 给其他AI时必须补充的上下文

为了让其他 AI 少走弯路，建议在提示词外再补这些信息：

### 6.1 功能上下文

- 这是新增功能、老功能还是问题排查
- 当前关注的是“理解功能”还是“准备改造”
- 本次只看主流程，还是要连异常流程一起看

### 6.2 技术上下文

- 前端仓库路径
- 后端仓库路径
- 网关模块名
- 目标微服务名
- 可能涉及的表名、类名、菜单名

### 6.3 输出上下文

- 输出成 Markdown 还是 Word
- 是否要画时序图
- 是否要写给业务、开发还是架构师

## 7. 推荐给其他AI的执行规则

你可以直接追加下面这些约束，提升执行质量：

```text
附加约束：
1. 不要只看 Controller 或页面文件，必须打通完整链路。
2. 不要把代码分支直接抄成结论，要翻译成业务规则。
3. 不要省略配置、权限、租户、缓存、消息等影响项。
4. 如果前后端契约不一致、状态定义冲突、接口命名误导，必须单独指出。
5. 如果证据不足，不允许直接定结论，必须标记待验证。
6. 最终输出要先给高层摘要，再给技术细节，最后附证据表。
```

## 8. 如何判断其他AI执行得好不好

检查下面 8 项：

- 是否明确给出功能入口
- 是否打通前后端主链路
- 是否抽取了业务规则，而不是只贴代码
- 是否分析了数据变化
- 是否提炼了状态流转
- 是否覆盖了异常流程
- 是否附了证据表
- 是否单列了未确认问题

如果缺任意两项以上，说明分析质量不合格。

## 9. 推荐的执行流程

建议你给其他 AI 下任务时，采用下面顺序：

1. 先让 AI 只做入口定位和主链路分析
2. 再让 AI 补业务规则、数据影响、状态流转
3. 再让 AI 补异常流程、配置影响、未确认问题
4. 最后让 AI 生成适合人类阅读的完整文档

这样比一次性让 AI 全做更稳定。

## 10. 直接可复制的最终交付口令

如果你只想快速交给另一个 AI，可以直接复制下面这段：

```text
请按“功能闭环分析法”分析【功能名称】。

要求你从用户入口开始，打通 React 前端交互、API 调用、Spring Cloud 后端调用链、数据变化、状态流转、配置权限影响、异常分支，并输出一份适合人类阅读的功能源码理解文档。

所有关键结论都必须附源码证据，并明确区分：
- 已证实
- 高概率推断
- 待验证

最终至少输出：
1. 功能摘要
2. 用户入口与操作路径
3. 前端实现链路
4. 后端调用链
5. 核心业务规则
6. 数据与状态流转
7. 异常与风险
8. 关键源码索引
9. 证据表
10. 未确认问题
```

## 11. 结论

把任务交给其他 AI 时，最关键的不是“让它去看代码”，而是：

- 给它明确边界
- 给它明确输出结构
- 给它明确证据要求
- 给它明确多轮步骤

只要这四点明确，其他 AI 才更容易稳定执行，并产出可复核的高质量分析结果。
