# 会员充值前端接口替换对照说明

本次前端改动的核心是：会员充值新页面不再继续走原有接口，而是统一切到新的储值方案充值接口前缀。

## 1. 接口层新增前缀

新增接口前缀：

- `/api/merchant/member_deposit_plan_ops/`

对应文件：

- `D:/mywork/nms4pos-ui/api/pos4plugin/src/services/pos4desktop/member/index.tsx`

在该文件中新增了 3 个接口方法：

- `member_deposit_plan_ops_getDealingTask`
- `member_deposit_plan_ops_listAvailableDepositPlans`
- `member_deposit_plan_ops_cardCharge`

这 3 个新方法在当前前端改造里的使用方式，默认就是对齐原有页面调用场景来替换：

- 入参结构延续原页面原有调用方式
- 返回值用法也延续原页面原有处理方式
- 这次改动的重点是“接口入口切换”，不是“页面处理逻辑重写”

更明确地说：

- `member_ops_getDealingTask` 和 `member_deposit_plan_ops_getDealingTask` 的参数定义一致，返回值类型一致
- `recharge_plan_listAvailable` 和 `member_deposit_plan_ops_listAvailableDepositPlans` 的参数定义一致，返回值类型一致
- `member_ops_depositPlanCharge` 和 `member_deposit_plan_ops_cardCharge` 的参数定义一致，返回值类型一致

## 2. 页面层需要替换的接口

对应页面文件：

- `D:/mywork/nms4pos-ui/app/pos4desktop/src/pages/FunctionPanel/pages/MemberRechargeNew/index.tsx`

该页面里，老接口和新接口的对应关系如下：

### 2.1 查询可用充值方案

老接口：

- `recharge_plan_listAvailable`

新接口：

- `member_deposit_plan_ops_listAvailableDepositPlans`

替换方式：

- `recharge_plan_listAvailable({ cardLid: selectedMember?.cardLid })`
- 改为 `member_deposit_plan_ops_listAvailableDepositPlans({ cardLid: selectedMember?.cardLid })`

说明：

- 老接口参数：
  - `sid?: string`
  - `cardLid?: string`
  - `brandLid?: string`
- 新接口参数：
  - `sid?: string`
  - `cardLid?: string`
  - `brandLid?: string`
- 结论：
  - 这一组接口的调用参数定义一致
  - 返回值类型都还是 `ListDwdBillResponse`
  - 当前页面实际调用时仍然只传 `cardLid`

### 2.2 发起储值方案充值

老接口：

- `member_ops_depositPlanCharge`

新接口：

- `member_deposit_plan_ops_cardCharge`

替换方式：

- 原来调用 `member_ops_depositPlanCharge(...)`
- 改为调用 `member_deposit_plan_ops_cardCharge(...)`

说明：

- 老接口参数：
  - `cardNo: string`
  - `authNo?: string`
  - `lid?: string`
  - `planLid: string`
  - `tierIndex: number`
  - `saveAmount: number`
  - `giftAmount?: number`
  - `giftPoints?: number`
  - `planName?: string`
  - `pays: { payWayLid: string; amount: string }[]`
- 新接口参数：
  - `cardNo: string`
  - `authNo?: string`
  - `lid?: string`
  - `planLid: string`
  - `tierIndex: number`
  - `saveAmount: number`
  - `giftAmount?: number`
  - `giftPoints?: number`
  - `planName?: string`
  - `pays: { payWayLid: string; amount: string }[]`
- 结论：
  - 这一组接口的调用参数定义一致
  - 返回值类型都还是 `ListDwdBillResponse`
  - 前端这里可以按原参数直接替换调用方法名

### 2.3 查询本设备未完成充值单

老接口：

- `member_ops_getDealingTask`

新接口：

- `member_deposit_plan_ops_getDealingTask`

替换方式：

- `member_ops_getDealingTask({ orderOpType: 'E' })`
- 改为 `member_deposit_plan_ops_getDealingTask({ orderOpType: 'E' })`

说明：

- 老接口参数：
  - `orderOpType: 'E' | 'P'`
- 新接口参数：
  - `orderOpType: 'E' | 'P'`
- 结论：
  - 这一组接口的调用参数定义一致
  - 返回值类型都还是 `ListDwdBillResponse`
  - 当前页面传参仍然是 `orderOpType: 'E'`

## 3. 一句话说明给前端

会员充值新页面现在要统一改走新的储值方案充值接口分支：

- `recharge_plan_listAvailable` -> `member_deposit_plan_ops_listAvailableDepositPlans`
- `member_ops_depositPlanCharge` -> `member_deposit_plan_ops_cardCharge`
- `member_ops_getDealingTask` -> `member_deposit_plan_ops_getDealingTask`

并且这些新接口都挂在统一前缀下：

- `/api/merchant/member_deposit_plan_ops/`

补充说明：

- 这次前端调整可以直接理解为：老接口名替换成新接口名，调用参数不变，返回值类型不变
- 如果后端协议没有额外调整，那么前端主要改方法名和接口前缀即可
