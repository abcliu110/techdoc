# 小程序会员付款账单未同步仍允许支付问题

## 当前结论

截至 2026-06-29，本问题的早期证据已确认受到测试平台污染：测试平台曾通过 `reLaunch` 直接打开 `pagePop/paycenter/index`，导致“下单后出现立即支付”的现象不是自然 UI 路径产生的可靠证据。

因此，不能再基于这次被污染的链路直接断定业务程序存在“下单后自动进入支付中心并允许旧金额支付”的 bug。该业务问题仍需要在只使用可见 UI `tap` 的自然路径下重新复现。

## 已确认的测试平台问题

测试平台存在以下不合规行为：

1. 使用代码控制小程序页面跳转，例如 `reLaunch`、`switchTab`、`callWxMethod('reLaunch')`。
2. 在业务流程中尝试清理小程序 storage 或重置运行时上下文。
3. 通过接口直接修改购物车或发起支付，例如 `shopping_cart/clear`、`shopping_cart/modDishNum`、`order_bill/pay_order`。
4. 存在非真实用户操作的兜底动作，例如 synthetic trigger/touch/dispatch。

这些行为会制造自然用户不会走到的页面状态，进而产生假 bug 或错误归因。

## 测试用例

小程序开台点餐，小程序会员付款，要求有积分、赠券、返现。

## 待复现的业务风险

需要在自然 UI 路径下重新确认：

1. 小程序下单成功后，是否会通过正常页面流转出现支付入口。
2. 支付中心金额是否与 POS/后端应收金额一致。
3. 如果账单未同步，是否仍允许点击“立即支付”并提交会员支付。
4. 成功会员付款后，积分、赠券、返现断言是否全部通过。

## 5 Why：测试平台为什么会制造错误现象

1. 为什么会看到“下单后立即支付”页面？
   - 因为测试平台在下单后使用代码直接跳转到了支付中心。

2. 为什么测试平台会主动跳转支付中心？
   - 因为脚本为了继续支付流程，在找不到自然结账入口时启用了 direct route/reLaunch 兜底。

3. 为什么会有这种兜底？
   - 因为平台把“继续完成用例”放在了“保持真实用户路径”之前，允许能力执行器直接控制页面状态。

4. 为什么这个设计会导致误判？
   - 因为页面状态不是用户通过可见 UI 操作产生的，支付中心金额、订单上下文和 POS 账单状态可能不同步。

5. 为什么必须修复测试平台？
   - 因为测试平台只应验证业务程序，不应通过代码跳转、写接口、改 storage 来创造业务状态；否则会持续制造假 bug。

## 测试平台修复要求

1. 小程序业务流程只能通过可见元素 `tap` 推进。
2. 禁止使用代码控制页面跳转：`reLaunch`、`navigateTo`、`redirectTo`、`switchTab`。
3. 禁止通过 direct-route 打开业务页面。
4. 禁止 synthetic trigger/touch/dispatch 代替真实点击。
5. 禁止直接调用小程序业务写接口来清购物车、改数量、下单或支付。
6. 会员付款前不能主动选择券支付，也不能自行计算金额或构造支付数据。
7. 支付中心如出现账单不确定状态，只能通过可见“刷新账单”“返回账单”“刷新订单”“结账入口”等 UI 操作恢复；无法证明同步时应停止并记录证据。

## 后续业务验证要求

测试平台修复后，重新跑该用例时必须满足：

- 全流程不使用代码跳转或直接业务写接口。
- 下单后停留在自然 UI 流程中，由可见按钮进入后续页面。
- 如果出现“立即支付”，记录它是由哪个可见 UI 操作进入的。
- 支付前不主动选券、不直接发支付请求、不自行计算支付金额。
- 会员支付成功后，再验证积分、赠券、返现。

## 当前状态

测试平台侧已按上述原则修复并需要通过本地守卫与回归测试验证。业务程序侧的账单未同步问题保留为待自然路径复现项，不能再引用被 direct route 污染的证据作为结论。
