# 消费赠券、消费返现多门店适用范围方案

## 背景

普通券可以配置适用门店，消费赠券和消费返现原先只有主表 `sid`，无法像券一样选择多个适用门店。当前需求是在不引入新关系表的前提下，让这两类营销规则支持“所有门店”或“指定多个门店”。

## 存储方案

本次采用主表 JSON 字段方案。

- `crm_consumption_coupon_rule` 新增 `is_all_store`、`store_sids`
- `crm_cash_back` 新增 `is_all_store`、`store_sids`
- `is_all_store=true` 表示全部门店适用
- `is_all_store=false` 表示只适用 `store_sids` 中的门店
- `store_sids` 使用 MySQL JSON 数组保存门店 `sid`，例如 `[10001,10002]`
- 原有 `sid` 保留，继续作为历史单门店规则和管理范围字段

## 兼容规则

历史数据不回填，运行时兼容：

- `is_all_store IS NULL` 且 `store_sids IS NULL` 时，继续使用原 `sid` 匹配门店
- 新数据选择“全部门店”时，保存 `is_all_store=true`，并清空 `store_sids`
- 新数据选择“指定门店”时，保存 `is_all_store=false`，并要求 `store_sids` 非空

## 匹配逻辑

POS 触发消费赠券或消费返现时：

1. 先按原有商户、状态、会员卡类型、会员卡等级、有效期、金额等条件筛选候选规则。
2. 再按门店适用范围过滤：
   - 全部门店：直接命中
   - 指定门店：POS 请求 `sid` 必须在 `store_sids` 内
   - 历史规则：继续要求规则 `sid` 等于 POS 请求 `sid`

## 为什么不用关系表

普通券使用 `crm_store_in_coupon` 保存券和门店关系，适合券实例量和既有券链路。本次两类规则只需要随主规则读取和写入，门店数量有限，不需要按门店反查规则报表，因此主表 JSON 字段更小、更直接。

Java + MySQL 场景下，本方案的性能特点：

- 查询候选规则仍主要依赖已有 `mid`、状态、有效期、会员条件和金额条件。
- 门店匹配在 Java 内存中对候选规则做 `List.contains`，候选规则量通常较小。
- 不需要额外 join 或维护关系表增删，保存和读取链路更简单。

如果未来需要高频按门店反查、门店维度统计、规则量达到很大规模，再考虑把 `store_sids` 拆为关系表并补索引。
