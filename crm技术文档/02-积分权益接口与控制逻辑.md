# 积分权益接口与控制逻辑



> 适用范围：`/crm_points_rule/*`



>



> 依据：当前代码中的 Controller、DTO/VO、Model、Service 校验逻辑



>



> 说明：本文档描述的是**当前代码实际行为**，不是需求理想态。凡是代码里未实现的规则，不会在本文中写成“已生效逻辑”。



## 1. 接口清单



1. `POST /crm_points_rule/get` 获取积分权益规则详情



2. `POST /crm_points_rule/list` 分页查询积分权益规则列表



3. `POST /crm_points_rule/add` 新增积分权益规则



4. `POST /crm_points_rule/update` 修改积分权益规则



5. `POST /crm_points_rule/save` 保存积分权益规则（存在则修改，不存在则新增）

6. `POST /crm_points_rule/del` 删除积分权益规则


## 2. 统一响应格式



```jsonc



{



  "code": 0, // 业务码：0=成功



  "msg": "ok", // 提示信息



  "data": {} // 业务数据（对象/数组/字符串）



}



```



分页接口（`/list`）通常还会带分页字段（以运行环境返回为准）：



```jsonc



{



  "code": 0, // 业务码：0=成功



  "msg": "ok", // 提示信息



  "data": [], // 当前页数据



  "current": 1, // 当前页



  "pageSize": 20, // 页大小



  "total": 100, // 总条数



  "pages": 5 // 总页数



}



```



## 3. 核心控制逻辑总览



### 3.1 通用控制逻辑



1. 所有接口均使用 `POST`



2. Controller 不信任前端传入的 `mid/sid`



3. Controller 会从登录态 `getForceAdmin()` 中回填 `request.mid` 和 `request.sid`



4. 所有查询、修改、删除都按 `mid/sid` 做租户隔离



5. 新增、修改、删除、详情均要求后台权限；列表接口要求登录态



6. **互斥字段自动清空**：保存时后端在 Convert 层根据主控字段自动将无意义的从属字段置为 `null`，前端无需手动清空。例如：



   - `earningMode=1`（按支付方式）时，`earningProductScopeType`、`earningSpecifiedProductLids` 自动清空



   - `availableCycle=1`（按日）时，`availableDaysOfWeek`、`availableDaysOfMonth` 自动清空



   - `bonusEnabled=0` 时，生日/会员日所有字段自动清空



   - `productLimitEnabled=0` 时，商品范围所有字段自动清空



   - 其他互斥关系以此类推，互斥字段即使传值也会被忽略



### 3.2 新增控制逻辑 `POST /crm_points_rule/add`



1. `mid`、`sid` 由后端登录态回填



2. `planLid` 必填



3. 同一商户、同一门店、同一会员方案下，只允许存在一条积分权益规则



4. **`planLid` 存在性校验**：`planLid` 对应的会员卡类型必须在当前商户（`mid`）下存在且有效；若不存在则报错「会员卡类型不存在或不属于当前商户」（RULE-029，已实现校验）



5. 若 `pointsRuleEnabled != 1`，只校验 `ruleDescription` 长度，其他子规则字段不做强校验



6. 若 `pointsRuleEnabled == 1`，会继续校验：



   - 积分获取规则



   - 积分抵现规则



   - 积分清零规则



7. **商品存在性校验**：当 `earningSpecifiedProductLids`、`deductSpecifiedProductLids`、`deductExcludeProductLids` 非空时，所有引用的商品 ID 必须在当前商户、当前门店的商品中心 `pt_dish` 中存在且有效；若存在无效 ID 则报错「商品不存在或不属于当前商户，请检查商品配置」（RULE-030，已实现校验）。注意：后端保存前会自动清空互斥字段，例如 `earningMode=1` 时 `earningSpecifiedProductLids` 会被清空，不会触发商品校验



8. 新增成功后，后端自动生成 `lid`



### 3.3 修改控制逻辑 `POST /crm_points_rule/update`



1. `mid`、`sid` 由后端登录态回填



2. `lid`、`planLid` 必填



3. 先按 `mid/sid/lid` 查询待修改记录，查不到则报错



4. 修改后的 `planLid` 不能和其他记录冲突



5. **`planLid` 存在性校验**：`planLid` 对应的会员卡类型必须在当前商户（`mid`）下存在且有效（RULE-029，已实现校验）



6. 修改时会重复执行与新增相同的字段校验，包括：



   - 全局开关关闭时仅校验 `ruleDescription`



   - 积分获取规则的必填、范围、时段、等级明细校验



   - 积分抵现规则的必填、范围、时段、商品范围校验



   - 积分清零规则的模式联动与提醒天数校验



7. **商品存在性校验**：与新增相同，所有引用的商品 ID 必须在当前商户、当前门店的商品中心 `pt_dish` 中存在且有效（RULE-030，已实现校验）。互斥字段自动清空后再校验



8. 更新时不会修改已有记录的 `lid`



### 3.4 保存控制逻辑 `POST /crm_points_rule/save`

1. `mid`、`sid` 由后端登录态回填

2. 请求体字段与新增接口 `POST /crm_points_rule/add` 保持一致

3. 以后端 `mid + sid + planLid` 作为会员方案维度的业务唯一键

4. 若当前会员方案下不存在积分权益规则，则自动走新增逻辑

5. 若当前会员方案下已存在积分权益规则，则自动走修改逻辑

6. 若按 `mid + sid + planLid` 查询命中多条记录，则返回错误「同一会员方案存在多条积分权益规则，请联系管理员处理」

7. 保存成功后返回详情型响应；若存在商品展示列表，则单个商品对象全量返回 7 个字段：`dishLid`、`dishCode`、`dishName`、`unit`、`status`、`dishType`、`dishTypeCode`

### 3.5 删除控制逻辑 `POST /crm_points_rule/del`


1. `mid`、`sid` 由后端登录态回填



2. `lid` 必填



3. 先按 `mid/sid/lid` 查询，查不到则报错



4. 查到后执行删除



### 3.6 查询控制逻辑


#### 3.6.1 详情 `POST /crm_points_rule/get`


支持的查询参数：



| 参数 | 类型 | 必填 | 说明 |



|------|------|------|------|



| `mid` | Long | 是 | 商户ID，后端登录态自动回填，前端可不传 |



| `sid` | Long | 否 | 门店ID，有值时加入查询条件做门店级过滤 |



| `pointsRuleLid` | Long | 二选一 | 积分权益规则逻辑编号，有值时优先按此精准查询 |



| `planLid` | Long | 二选一 | 会员方案逻辑编号，`pointsRuleLid` 为空时按此查询 |



控制逻辑：



1. `mid`、`sid` 由后端登录态回填



2. `mid` 必填



3. `pointsRuleLid` 和 `planLid` 至少传一个，两个都不传则报错「必须指定规则编号或会员方案编号」



4. 查询优先级：



   - 传了 `pointsRuleLid` → 按 `mid + sid + pointsRuleLid` 精准查询



   - 未传 `pointsRuleLid` 但传了 `planLid` → 按 `mid + sid + planLid` 查询



   - 两个都传 → 优先使用 `pointsRuleLid`，忽略 `planLid`



5. 若按 `planLid` 查询命中多条记录，则返回错误「同一会员方案存在多条积分权益规则，请联系管理员处理」

6. 查询不到时返回 `data: null`，不报错

#### 3.6.2 列表 `POST /crm_points_rule/list`


1. `mid`、`sid` 由后端登录态回填



2. `current`、`pageSize` 必填



3. `planLid` 可选



4. 如果传了 `planLid`，则按该方案过滤

5. 如果不传 `planLid`，返回当前商户门店下全部积分权益规则

6. 列表响应不补齐商品展示信息；如需商品的 7 个展示字段，请使用详情型响应（`/get`、`/add`、`/update`、`/save`）


## 4. 已实现的 CRUD 业务校验



### 4.0 枚举传值约定



1. 当前积分权益接口中的开关、模式、范围、类型字段，前端统一传**枚举编码整数值（code）**



2. 不传枚举名，例如不要传 `BY_PRODUCT`、`ANNUAL`、`ENABLED`



3. 若传入非法枚举编码，当前代码会在 JSON 反序列化阶段直接报错，不会静默转为 `null`



4. 典型字段示例：



   - `earningMode`：`1-按支付方式实收积分`，`2-按商品实收积分`



   - `availableCycle`：`1-按日`，`2-按周`，`3-按月`



   - `deductionCeilingType`：`1-不限制`，`2-固定积分`，`3-账单比例`



   - `expiryMode`：`1-自发放日起`，`2-按年度清零`



### 4.1 文本与基础约束



1. `ruleDescription`



   - 可为空



   - 有值时最大 600 字节



   - 按 UTF-8 字节长度计算



### 4.2 积分获取规则



仅当 `pointsRuleEnabled = 1` 且 `earningEnabled = 1` 时生效：



1. `earningMode` 必填



2. 若 `earningMode = 2`（按商品实收积分）：



   - `earningProductScopeType` 必填



   - 若 `earningProductScopeType = 2` 或 `3`，`earningSpecifiedProductLids` 必须非空



3. `levelRates` 必须至少 1 行



4. `levelRates[].memberLevelId` 不可为空，且同一配置中不可重复



5. `levelRates[].consumeAmount > 0`



6. `levelRates[].earnPoints > 0`



7. 若 `singleEarnLimitType = 2`，`singleEarnLimitValue > 0`



8. 若 `bonusEnabled = 1`：



   - `birthdayEnabled` 和 `memberDayEnabled` 至少有一项为 `1`



   - 若 `birthdayEnabled = 1`，则 `birthdayPeriod` 必填，且仅允许 `1-当天`、`2-当周`、`3-当月`，`birthdayMultiplier > 0`



   - 若 `memberDayEnabled = 1`，则 `memberDayPeriod` 必填，且仅允许 `1-每周`、`2-每月`

   - 若 `memberDayPeriod = 1`，`memberDayDaysOfWeek` 必须非空（数字数组，合法值：1-周一，2-周二，3-周三，4-周四，5-周五，6-周六，7-周日）

   - 若 `memberDayPeriod = 2`，`memberDayDaysOfMonth` 必须非空（数字数组，合法值：1-31）

   - `memberDayMultiplier > 0`



9. `availableCycle` 必填



10. 若 `availableCycle = 2`，`availableDaysOfWeek` 必须非空（数字数组，合法值：1-周一，2-周二，3-周三，4-周四，5-周五，6-周六，7-周日）



11. 若 `availableCycle = 3`，`availableDaysOfMonth` 必须非空（数字数组，合法值：1-31）



12. `availableTimeType` 必填



13. 若 `availableTimeType = 2`，`availableTimeSlots` 必须非空，且每段：



   - `startTime` 必填



   - `endTime` 必填



   - 格式必须为 `HH:mm`



   - `endTime > startTime`



14. 若 `wecomExclusiveEnabled = 1`，`wecomExclusiveType` 必填，且当前允许值为：



    - `1`：企微好友或企微群客



    - `2`：企微好友



    - `3`：企微群客



    - `4`：企微双粉



### 4.3 积分抵现规则



仅当 `pointsRuleEnabled = 1` 且 `deductionEnabled = 1` 时生效：



1. `deductionRate > 0`



2. 若 `minDeductionType = 2`，`minDeductionValue > 0`



3. 若 `deductionMultipleType = 2`：



   - `deductionMultipleValue` 必填



   - `deductionMultipleValue > 0`



4. 若 `deductionCeilingType = 2`（固定积分上限）：



   - `deductionCeilingPoints` 必填



   - 范围：`0.01 <= value <= 999999`



5. 若 `deductionCeilingType = 3`（账单比例上限）：



   - `deductionCeilingRatio` 必填



   - 范围：`0.01 <= value <= 100`



6. `deductCycle` 必填



7. 若 `deductCycle = 2`，`deductDaysOfWeek` 必须非空（数字数组，合法值：1-周一，2-周二，3-周三，4-周四，5-周五，6-周六，7-周日）



8. 若 `deductCycle = 3`，`deductDaysOfMonth` 必须非空（数字数组，合法值：1-31）



9. `deductTimeType` 必填



10. 若 `deductTimeType = 2`，`deductTimeSlots` 必须非空，且每段：



    - `startTime` 必填



    - `endTime` 必填



    - 格式必须为 `HH:mm`



    - `endTime > startTime`



11. 若 `productLimitEnabled = 1`：



    - `productScopeType` 必填



    - 若 `productScopeType = 2` 或 `3`，`deductSpecifiedProductLids` 必须非空



    - 若 `productScopeType = 4` 或 `5`，则 `deductPosCategoryLids` 和 `deductMiniappCategoryLids` 至少一组非空



12. 上述开关、类型、范围字段若传入非法枚举编码，会在 JSON 反序列化阶段直接失败



### 4.4 积分清零规则



仅当 `pointsRuleEnabled = 1` 且 `expiryEnabled = 1` 时生效：



1. `expiryMode` 必填



2. 若 `expiryMode = 1`（自发放日期）：



   - `expiryDuration` 必填



   - `expiryFrequency` 必填



3. 若 `expiryMode = 2`（年度清零）：



   - `expiryAnnualYears` 必填，且 `> 0`



   - `expiryAnnualDate` 必填



4. 若 `expiryNotifyEnabled = 1`：



   - `expiryRemindDays` 必填



   - `expiryRemindDays > 0`



5. `expiryMode`、`expiryFrequency`、`expiryEnabled`、`expiryNotifyEnabled` 若传入非法枚举编码，会在 JSON 反序列化阶段直接失败

## 6. 核心模型



### 6.1 会员等级积分规则 `MemberLevelRateRule`



```jsonc



{



  "memberLevelId": 60001, // 会员等级ID；新增/修改时必填；同一配置下不可重复



  "consumeAmount": 10.00, // 每消费X元；新增/修改时必填；必须大于0



  "earnPoints": 1.00, // 获得Y积分；新增/修改时必填；必须大于0



  "sortOrder": 1 // 排序号；可为空；为空时后端当前不会自动重排



}



```



### 6.2 时段模型 `PointsRuleTimeSlot`



```jsonc



{



  "startTime": "10:00", // 开始时间；availableTimeType=2 或 deductTimeType=2 时必填；格式 HH:mm



  "endTime": "14:00" // 结束时间；availableTimeType=2 或 deductTimeType=2 时必填；格式 HH:mm；必须晚于 startTime



}



```



### 6.3 商品展示模型 `PointsRuleProductVO`



详情型响应（`/get`、`/add`、`/update`、`/save`）返回时，以下三个字段会携带商品的完整展示信息，字段来源于商品中心 `pt_dish`：


- `earningSpecifiedProducts`：积分获取规则中指定的商品列表（`earningProductScopeType=2/3` 时有值）



- `deductSpecifiedProducts`：积分抵扣规则中指定的商品列表（`productScopeType=2/3` 时有值）



- `deductExcludeProducts`：积分抵扣规则中分类模式下排除的商品列表（`productScopeType=4/5` 且配置了排除商品时有值）



每个商品对象的结构如下：



```jsonc



{



  "dishLid": 50001,          // 商品逻辑ID（lid）；商品的唯一标识



  "dishCode": "B020101001",  // 商品编码；前端列表展示用



  "dishName": "土豆猪肉盖饭", // 商品名称；前端列表展示用



  "unit": "份",              // 售卖单位；前端列表展示用



  "status": 1,               // 当前状态：0-禁用，1-启用；前端列表展示用



  "dishType": "主食",         // 所属小类名称；前端列表展示用



  "dishTypeCode": "B0201"    // 所属小类编码；前端列表展示用



}



```



> **注意**：新增/修改/保存接口（`/add`、`/update`、`/save`）的请求体中，商品仍以 ID 列表形式传入（`earningSpecifiedProductLids`、`deductSpecifiedProductLids`、`deductExcludeProductLids`），这里的商品 ID 指商品中心 `pt_dish.lid`。商品展示 VO 仅在详情型响应（`/get`、`/add`、`/update`、`/save`）中返回，`/list` 不返回商品展示信息。


## 7. 各接口请求/响应格式（JSONC 中文行内注释）



## 7.1 获取详情 `POST /crm_points_rule/get`



```jsonc



// 请求



{



  "mid": 10001, // 商户ID；前端可传但会被后端登录态覆盖；必填



  "sid": 20001, // 门店ID；前端可传但会被后端登录态覆盖；有登录态时通常会被覆盖



  "planLid": 30001, // 会员方案逻辑编号；pointsRuleLid 为空时按此查询；pointsRuleLid 有值时此字段被忽略



  "pointsRuleLid": 900001 // 积分权益规则逻辑编号；有值时优先按此精准查询；与 planLid 至少传一个



}



```



```jsonc



// 响应（查询成功）



{



  "code": 0, // 业务码：0=成功



  "msg": "ok", // 提示信息



  "data": {



    "pointsRuleLid": 900001, // 规则逻辑ID；详情查询成功时有值；查询不到时整个 data 为 null



    "mid": 10001, // 商户ID；查询成功时有值



    "sid": 20001, // 门店ID；查询成功时有值



    "planLid": 30001, // 所属会员方案逻辑ID；查询成功时有值



    "name": null, // 规则名称；当前新增/修改接口没有提供该字段，所以大多数情况下可能为空；只有库中已有值时才有值



    "pointsRuleEnabled": 1, // 积分规则全局开关；1=开启，0=关闭；有记录时有值



    "earningEnabled": 1, // 是否启用消费送积分；pointsRuleEnabled=1 时通常有值；关闭时可能为 0 或 null



    "earningMode": 2, // 积分模式；earningEnabled=1 时应有值；earningEnabled!=1 时可能为空
```

## 7.3 新增 `POST /crm_points_rule/add`



### 场景速查表



不同场景下关键字段的传值差异：



| 场景 | `earningMode` | 需额外传的字段 | 后端自动清空的字段 |



|------|--------------|--------------|-----------------|



| 按支付方式积分 | `1` | — | `earningProductScopeType`、`earningSpecifiedProductLids` |



| 按商品积分（指定商品参与） | `2` | `earningProductScopeType=2`、`earningSpecifiedProductLids` | `deductPosCategoryLids`、`deductMiniappCategoryLids` |



| 按商品积分（指定商品不参与） | `2` | `earningProductScopeType=3`、`earningSpecifiedProductLids` | 同上 |



| 场景 | `productLimitEnabled` | `productScopeType` | 需额外传的字段 | 后端自动清空的字段 |



|------|----------------------|-------------------|--------------|-----------------|



| 不限制抵扣商品 | `0` | — | — | `productScopeType` 及所有商品/分类列表 |

## 8. 前后端联调重点



1. `mid/sid` 前端可以传，但最终以后端登录态回填为准



2. 新增和修改都不会接收 `name` 字段，因此查询返回中的 `name` 当前很可能为空



3. `pointsRuleEnabled != 1` 时，当前代码只校验 `ruleDescription` 长度，不会校验子规则字段



4. `availableTimeSlots` 和 `deductTimeSlots` 现在已经是强类型结构，不再是 `Object` 数组



5. 详情接口中 `pointsRuleLid` 和 `planLid` 至少传一个；前端如果同时传两个，当前代码会优先按 `pointsRuleLid` 查



6. 当前 CRUD 只负责配置保存校验，不代表积分获取、抵现、清零、提醒、退款链路已经实现
