# 会员积分权益：后台页签展示与 0 金额流水问题分析

## 1. 背景

排查日期：2026-05-06

本次排查围绕后台会员详情弹窗中“积分、优惠券、消费、储值、账户”页签的取数方式，以及 POS 消费后自动积分补写导致消费页签出现 `0` 金额流水的问题。

现场现象：

- 后台会员详情弹窗的“消费”页签中，同一账单号附近出现 `本次消费金额 = 0` 的记录。
- 截图中正常消费记录金额为 `-110`，但同时间或相近时间出现多条 `0` 金额记录。
- POS 日志显示部分账单存在重复提交或失败补记现象，但不是所有 `0` 金额记录都能直接归因于 POS 端重复扣款。

### 3.1 积分流水接口

`list_point_record` 调用 `CrmMemberServicePlus.listPointRecord`。

核心过滤：

```java

CrmCardPointsRecord::getMid = request.mid
CrmCardPointsRecord::getCardIdAlias = request.cardId
  or CrmCardPointsRecord::getPhone = request.phone

```

该接口返回积分流水 VO。

### 3.2 余额流水接口

`list_balance_record` 调用 `CrmMemberServicePlus.listBalanceRecord`。

通用过滤：

```java
CrmCardRecord::getYingyeriqi between beginDate and endDate
CrmCardRecord::getMid = request.mid
CrmCardRecord::getIfDealSuccess = true
cardIdAlias / phone / cardId 匹配会员卡
order by createTime desc
```

按 `type` 增加过滤：

| type | 页签 | 后端过滤 |
| --- | --- | --- |
| 0 | 账户 | 不限制 operationModel |
| 1 | 储值 | Charge、ChargeCancel |
| 2 | 消费 | Consume、CancelOrder、Refund |

重要缺口：

```text
listBalanceRecord 没有过滤 amount != 0。
```

因此 `crm_card_record` 中只要存在成功状态、操作类型属于消费类、金额为 0 的记录，就会进入消费页签。

已观察到的现象：

- `2026050610025`：日志中只看到一次结账请求和一次成功响应，不支持“该账单 POS 端提交了两次扣款”的判断。

- POS 端确实存在个别账单重复提交或补记失败迹象。
- 但后台消费页签出现 `0` 金额记录，不能简单归因于 POS 端重复扣款。
- 对于截图中的 `0` 金额消费记录，更直接的解释是后端余额流水表中生成了 `operation_model = Consume` 且 `amount = 0` 的记录。

## 5. 两个问题的边界

### 5.1 问题一：消费页签展示 0 金额记录

直接原因：

```text
消费页签查询 crm_card_record，并且 type=2 只按 operation_model 过滤，不过滤 amount=0。
```

表现：

```text
crm_card_record 中的 0 金额 Consume 记录被展示为“本次消费金额 0”。
```

风险：

- 前端展示上会误导用户认为 POS 端发生了一笔 0 元消费。
- 如果这些记录来自积分补记或积分相关逻辑，则余额流水与积分流水的职责边界被混淆。

### 5.1.1 这是不是最近新增代码引入的 bug

当前结论：

```text
不是“消费补记积分”新代码直接把余额流水写了两次。
```

代码证据：

- `CrmCardOpServicePlus.grantConsumePoints(...)` 只写积分流水表 `crm_card_points_record`。
- 该方法内部创建的是 `CrmCardPointsRecord`，保存调用是：

```java
iCrmCardPointsRecordService.save(pointsRecord)
```

- 该方法没有调用：

```java
iCrmCardRecordService.save(...)
iCrmCardRecordService.saveBatch(...)
```

对应位置：

```text
D:\mywork\nms4cloud\nms4cloud-app\2_business\nms4cloud-crm\nms4cloud-crm-service\src\main\java\com\nms4cloud\crm\service\card\CrmCardOpServicePlus.java
```

另一方面，真正会生成 `crm_card_record` 的主链路在：

```text
CardBalanceService.execute(...) -> genRecord(...) -> genRecordForMinus(...)
```

而 `git blame` 显示 `genRecordForMinus(...)` 里这段会生成 `0` 金额消费流水的逻辑最早来自 2023 年的旧提交，不是本次积分补记改造新增。

结论：

- 本次新增的补记积分代码没有直接把余额流水表写两次。
- `0` 金额消费流水属于旧余额引擎在特定分层场景下的历史缺陷。
- 这次积分改造更多是把问题暴露得更明显，而不是直接制造该问题。

### 5.1.2 以前是不是没有这个问题

当前不能下“以前没有”的结论。

更准确的说法是：

```text
这个问题在旧余额流水生成逻辑中一直存在，只是以前不一定容易触发，或者触发后不容易被注意到。
```

原因：

- 旧逻辑只有在会员卡余额被拆成多条 `crm_card_balance` 分层记录时，才更容易出现。
- 如果会员卡只有单一余额层，通常只会生成一条真实消费流水，不会暴露出这个问题。
- 一旦会员卡存在本金层、赠送层、多门店层或冻结导致的拆层，流水生成就会按层遍历，更容易出现附带的 `0` 金额流水。

## 5.1.3 什么情况下会出现这个问题

问题出现通常需要同时满足以下条件：

1. 会员卡余额不是单层，而是多条 `crm_card_balance` 记录。
2. 这些记录中存在赠送余额层，或者其他会被遍历到但本次不实际扣减的余额层。
3. 本次消费最终没有从某一条赠送层扣钱，即：

```text
task.getGiveAmount() == 0
```

或者在遍历到该层时已经被前面的层消耗完，导致：

```text
gift = min(cardBalance.getGift(), task.getGiveAmount()) = 0
principal = min(cardBalance.getPrincipal(), task.getPrincipalAmount()) = 0
```

但老逻辑仍然会：

- 因为 `cardBalance.getGift() > 0` 先创建一条 `record`
- 最后把这条 `record` 以 `amount = 0` 加入 `cardRecords`
- 再统一 `saveBatch(cardRecords)` 持久化到 `crm_card_record`

于是后台“消费”页签就会看到：

- 1 条真实消费记录，例如 `-110`
- N 条 `0` 金额记录

### 5.1.4 典型触发场景

更容易触发的业务场景包括：

- 会员卡同时存在本金余额和赠送余额。
- 扣款规则要求只扣本金。
- 当天存在冻结赠送金额，导致赠送余额不可用。
- 门店余额被拆成多条层记录，本次只实际命中其中一部分。
- 多条赠送层存在，但本次消费金额全部由本金承担。

不容易触发的场景：

- 会员卡只有一条本金层余额。
- 本次消费正好真实使用了所有被遍历到的余额层，没有产生“创建了记录但未实际扣减”的分支。

### 5.1.5 代码级根因

根因不在补记积分接口，而在旧余额流水引擎的扣减逻辑：

- `genRecordForMinus(...)` 遍历每一条 `crm_card_balance`
- 只要满足：

```java
if (gtZero(cardBalance.getGift())) {
    record = buildRecord(task, cardBalance, card);
}
```

就会先创建一条 `CrmCardRecord`

- 之后即使计算结果为：

```java
gift == 0
principal == 0
```

也没有把这条空记录丢弃

- 最终仍会执行：

```java
record.setAmount(record.getPrincipalAmount() + record.getGiveAmount());
cardRecords.add(record);
```

因此会生成一条：

```text
operation_model = Consume
amount = 0
```

的消费流水。

### 5.2 问题二：补写方法没有补写到对应流水，而是新增流水

当前观察到的风险点：

- 新增消费积分补记逻辑应补到积分流水或关联原消费上下文。
- 如果通过旧的余额变更统一链路生成流水，可能在 `amount = 0`、`operationModel = Consume` 的情况下额外写入一条 `crm_card_record`。
- 这类记录不会改变余额，却会被消费页签按消费流水展示。

需要继续精确核对的点：

- `CrmCardOpServicePlus.grantConsumePoints(...)` 是否只写 `crm_card_points_record`，还是间接触发旧统一链路写入 `crm_card_record`。
- `CardBalanceService.execute(...)` 或相关旧链路在 `givePoint != 0` 且余额变动金额为 0 时，是否仍会调用 `genRecord(...)` 写入 `crm_card_record`。
- 原消费流水和补记积分流水之间是否有稳定关联字段，例如 `orderBillId`、`sourceOrderId`、`dealTaskId` 或业务单号。

## 6. 建议修复方向

### 6.1 后端查询层防止误展示

消费页签对应的 `listBalanceRecord(type=2)` 建议增加金额过滤：

```java
amount != 0
```

适用范围：

- 仅对 `type = 2` 的消费页签生效。
- 不建议直接影响 `type = 0` 账户总览，除非产品确认账户页也不需要展示 0 金额余额流水。

注意：

- 这是展示层防护，能避免 0 金额记录误出现在消费页签。
- 不能单独作为根因修复，因为数据库中仍可能存在职责不清的 0 金额余额流水。

### 6.2 写入层避免生成 0 金额消费余额流水

积分补记应优先写入：

```text
crm_card_points_record
```

不应因为补记积分而新增：

```text
amount = 0
operation_model = Consume
```

的 `crm_card_record`。

如果业务需要保留补记痕迹，应使用更明确的积分操作类型和备注，例如：

```text
operationModel = ConsumePointGrant 或类似明确的积分补记类型
comment = 消费后补记积分
```

不要复用余额消费类型 `Consume` 表示积分补记。

### 6.3 原流水关联

补记积分需要与原消费单稳定关联：

- 使用 `orderBillId` 关联原消费账单。
- 必要时补充 `dealTaskId` 或来源单号字段。
- 避免通过“同一时间、同一卡号、同金额”进行弱关联。

如果要在后台消费页签展示“该消费获得多少积分”，推荐方案是：

- 消费页签仍以 `crm_card_record` 展示真实余额消费。
- 通过 `orderBillId` 关联积分流水，追加展示“本次获得积分”。
- 不要把积分补记伪装成一条 0 金额消费流水。

## 7. 当前结论

1. 后台会员详情弹窗中，`积分` 页签查 `crm_card_points_record`，`消费/储值/账户` 页签查 `crm_card_record`。
2. `消费` 页签传 `type = 2`，后端只按 `operationModel in (Consume, CancelOrder, Refund)` 过滤，没有过滤 `amount = 0`。
3. 前端没有计算或制造 `0` 金额，只是把后端返回的 `amount` 原样展示为“本次消费金额”。
4. POS 日志只能证明个别账单存在重复提交或补记失败迹象，不能直接解释所有 `0` 金额消费记录。
5. `grantConsumePoints(...)` 本身只写积分流水，不直接写余额流水；它不是“余额表写两次”的直接来源。
6. 更高概率的根因是：旧 `CardBalanceService` 在扣减多层余额时生成了 `amount = 0`、`operationModel = Consume` 的余额流水，导致后台消费页签误展示。
7. 这个问题属于旧余额引擎中的历史缺陷，不是本次补记积分代码首次引入；本次改动更多是让问题更容易被观察到。
8. 修复应分两层：查询层避免 0 金额消费流水误展示；写入层避免旧余额引擎保存 `amount = 0` 的消费流水，并建立与原消费单的稳定关联。
