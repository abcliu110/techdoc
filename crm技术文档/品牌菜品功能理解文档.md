## 1. 一页摘要

品牌菜品功能是集团/品牌维度的菜品主数据管理能力。它不是一套独立前端实现，而是复用门店菜品管理组件 `DishManage`，通过 `forBrand=true` 把数据上下文从“门店”切换到“品牌”。

该功能支持：

- 按品牌查询菜品列表；
- 按 POS 分类 / 线上分类过滤菜品；
- 新增、编辑、批量编辑菜品；
- 切换菜品状态、上架状态、楼面分单等开关；
- 删除菜品；
- 导入、导出菜品；
- 从套餐中移除菜品；
- 品牌菜品批量发布到菜谱；
- 在开启 `P1` 审批时，走菜品售价审批入口。

当前最关键的技术判断：

1. 前端入口明确：`/admin/groupdish/branddish`。
2. 前端页面明确：`BrandDishPage` 只渲染 `<DishManage forBrand={true} />`。
3. 查询接口明确：列表查询走 `/api/pos4cloud/merchant/pt_dish/list`，通过 `sidList: [品牌lid]` 查询品牌维度菜品。
4. 写操作接口前端走新版 `/merchant/pt_dish_new/*`，在 `nms4cloud` 仓库中未找到对应 Controller/Service 实现；补查 `D:\mywork\nms4pos` 后，已找到品牌同步和菜谱发布相关后端实现。
5. `pt_dish_type/sync_to_brand` 位于 `nms4pos` 的 `nms4cloud-pos2plugin`，实际委托 `nms4cloud-pos4cloud` 的 `GroupDishBookReleaseService.syncStoreToBrand()` 执行。
6. 品牌菜品有独立权限码 `brand:dish:add/update/delete`，但当前可见基础后端 Controller 使用的是 `pt:dish:*` 权限，权限映射关系需要继续确认。
7. 参考文档与当前代码能相互印证：品牌菜品和门店菜品不是两张不同主表，而是共用 `sc_dish / sc_dish_type`，差异主要在 `shop_id` 这个容器 ID 的语义：
   - 品牌菜品：`shop_id = brand_lid`
   - 门店菜品：`shop_id = store_sid`
8. 从前端与 `nms4pos` 后端实现看，品牌菜品分配到门店是“三容器复制/发布”：
   - 门店菜品可同步到品牌库；
   - 品牌菜品先发布到品牌菜谱；
   - 品牌菜谱再通过 `pos_group_dish_book_release.store_lids` 发布到指定门店。
9. `pt_dish_new`、`pt_dish_type/listTree` 在前端仓库的可见历史里并不是最近才出现；`nms4cloud` 仓库缺失这些实现，但 `nms4pos` 仓库补齐了 `sync_to_brand`、`release2Book`、`pos_group_dish_book_release` 的后端证据。

## 5. 前端实现链路

### 5.1 路由

`config/routes.ts` 中，品牌菜品菜单位于 `/admin/groupdish` 下：

- `/admin/groupdish/branddishtype`：品牌类库；

- `/admin/groupdish/groupdishbook`：菜谱方案。

### 5.2 页面组件

`BrandDishPage` 内容很薄：

```tsx
const BrandDishPage : React.FC = () => {
  return <DishManage forBrand={true} />
};
```

这说明品牌菜品的核心能力来自 `DishManage`，品牌语义由 `forBrand` 参数驱动。

### 5.3 品牌上下文

`DishManage` 中：

- 门店列表来自 `useStore()`；
- 品牌列表来自 `useBrand()`；
- 当 `forBrand=true` 时，当前 `sid` 使用 `curBrand`；
- 品牌选择框展示 `brandList`；
- 切换品牌时调用 `setCurBrand(value)`。

关键逻辑：

```tsx
form.setFieldValue('sid', forBrand ? curBrand : curStore);
options={(forBrand ? brandList : storeList) as any[]}
onChange={(value) => {
  forBrand ? setCurBrand(value) : setCurStore(value);
}}
```

### 5.4 查询逻辑

列表查询通过 `listPtDish` 发起：

```tsx
listPtDish({
  sidList: [sid],
  foodTypes,
  name,
  business,
  startPrice,
  endPrice,
  pageSize,
  current,
  moreTypes,
  dishLabels,
  ...(support_mode === 'POS' ? { dishTypeCodeList } : { onlineTypeCodeList }),
  ...(isBoolean(enable) && { enable }),
  ...(isBoolean(shelve) && { shelve }),
})
```

这说明品牌菜品并没有单独的 `brand_dish` 查询接口，而是使用同一个菜品查询接口，以 `sidList` 区分当前品牌维度。

### 8.4 品牌菜品与非品牌菜品的关系

结合参考文档 `11-品牌选择后按sid查询商品分类逻辑分析.md` 和当前代码，可以得到一个更稳的模型：

| 维度 | 品牌菜品 | 非品牌菜品（门店菜品） |
|---|---|---|
| 物理表 | `sc_dish` / `sc_dish_type` | `sc_dish` / `sc_dish_type` |
| 容器标识 | `shop_id = brand_lid` | `shop_id = store_sid` |
| 用途 | 品牌标准菜品模板 / 品牌主数据 | 门店实际经营使用菜品 |
| 查询入口 | 选择品牌后仍沿用字段名 `sid`，但值实际是 `brand_lid` | 选择门店后 `sid` 就是门店 `sid` |

这意味着：

- 品牌菜品和门店菜品不是通过“不同表名”区分；
- 它们是同一套商品模型在不同容器下的两份数据；
- 品牌菜品是门店菜品的上游模板来源，而不是一个完全脱离门店的独立体系。

### 8.5 品牌菜品如何分配到门店

补查 `D:\mywork\nms4pos` 后，这条链路已经能从后端实现层确认。它不是简单“品牌菜品直接分配到门店”，而是围绕 `sid/shop_id` 这个容器字段做三次数据搬运：

#### 第一步：门店菜品可以同步到品牌库

前端 `PtDishType` 提供了“同步到品牌库”按钮，仅在非品牌模式下显示。后端入口位于 `nms4pos`：

- 接口：`POST /api/pos4cloud/merchant/pt_dish_type/sync_to_brand`
- 请求参数：`{ sid: curSid, brandSid: syncTargetBrand }`
- 页面提示：`仅同步品牌库中不存在的分类、菜品和做法`
- Controller：`PtDishTypeForBizController.syncToBrand()`
- Service：`PtDishTypeServicePlus.syncToBrand()`
- 实际执行：`GroupDishBookReleaseService.syncStoreToBrand(mid, storeSid, brandSid)`

真实规则是：品牌库已有菜品分类时拒绝同步；然后读取来源门店完整菜品数据，把目标容器 `sid` 改成 `brandSid`，并把 `pid` 置空后批量保存。这个动作覆盖/写入 11 类数据：

| 数据 | 逻辑表/实体 | 核心字段变化 |
|---|---|---|
| 菜品分类 | `pt_dish_type` / `PtDishType` | `sid: storeSid -> brandSid`, `pid = null` |
| 菜品主数据 | `pt_dish` / `PtDish` | `sid: storeSid -> brandSid`, `pid = null` |
| 菜品规格/单位 | `pt_dish_unit` / `PtDishUnit` | `sid: storeSid -> brandSid`, `pid = null`, 保留 `dish_code` |
| 套餐/组合关系 | `pt_dish_map` / `PtDishMap` | `sid: storeSid -> brandSid`, `pid = null`, 保留 `main_dish_code/sub_dish_code` |
| 菜品区域/餐段 | `pt_dish_area` / `PtDishArea` | `sid: storeSid -> brandSid`, `pid = null` |
| 做法类别 | `pt_cookway_type` / `PtCookwayType` | `sid: storeSid -> brandSid`, `pid = null` |
| 做法 | `pt_cookway` / `PtCookway` | `sid: storeSid -> brandSid`, `pid = null` |
| 做法类型 | `pt_cook_type` / `PtCookType` | `sid: storeSid -> brandSid`, `pid = null` |
| 做法引用 | `pt_cook_ref` / `PtCookRef` | `sid: storeSid -> brandSid`, `pid = null` |
| 口味类别 | `pt_flavor_type` / `PtFlavorType` | `sid: storeSid -> brandSid`, `pid = null` |
| 菜品口味 | `pt_dish_flavor` / `PtDishFlavor` | `sid: storeSid -> brandSid`, `pid = null` |

因此，品牌库不是只同步分类，而是一次性复制“分类、菜品、规格、套餐关系、做法、口味”等完整菜品生态。

#### 第二步：品牌菜品发布到品牌菜谱

品牌菜品页的“批量发布到菜谱”：

- 先选择 `bookLids`；
- 再选择 `bookReleaseRule`、`dishReleaseRule`；
- 提交时带 `dishLids` 和 `sid = brandLid`；
- 接口：`POST /api/pos4cloud/merchant/pos_group_dish_book_release/release2Book`

后端 `GroupDishBookReleaseService.release2Book()` 会校验 `sid` 必须是品牌 `SysBrand.lid`，并校验所选 `bookLids` 都属于该品牌。随后通过 `queryDataByDishLids(mid, sid, dishLids)` 读取品牌容器里的菜品及套餐子菜、分类、规格、做法等关联数据，再按发布规则写入目标菜谱容器：

```text
源容器：sid = brand_lid
目标容器：sid = pos_group_dish_book.lid
```

也就是说，品牌菜谱本身也是一个菜品容器，菜谱明细不是单独的明细表，而是继续复用 `pt_dish / pt_dish_type / pt_dish_unit / pt_dish_map ...`，只是 `sid` 换成菜谱 `lid`。

#### 第三步：品牌菜谱再发布到门店

在 `PosGroupDishBookPage` 中，菜谱创建和菜谱发布都存在 `storeLids`：

- 创建菜谱时可设置 `默认发布门店`；
- 发布菜谱时明确选择 `发布门店`；
- 发布记录实体 `IPosGroupDishBookRelease` 中也有 `storeLids` 字段。

这说明从品牌到门店的分配动作是：

```text
品牌菜品
 -> 品牌菜谱方案
 -> 菜谱发布记录（带 storeLids）
 -> 指定门店
```

后端 `GroupDishBookReleaseService.release()` 会读取 `pos_group_dish_book_release.store_lids`，拆成多个门店 `sid`，然后查询菜谱容器数据：

```text
queryDishData(mid, bookLid, brandLid)
```

再根据 `bookReleaseRule` 执行：

| 发布规则 | 行为 |
|---|---|
| `MERGE` | 合并到门店 |
| `ONLY_UPDATE_SAME` | 仅更新门店已有相同菜品 |
| `COMPLETE_REPLACE` | 完全替换门店菜谱数据 |
| `ONLY_REMOVE_SAME` | 删除门店中与菜谱相同的菜品 |

其中 `MERGE` 下还会继续看 `dishReleaseRule`，例如发布新菜忽略相同菜、覆盖相同菜、保留价格、仅更新图片等。

因此，当前最稳妥的业务结论是：

> 品牌菜品不是直接“分配”到门店，而是先被复制进品牌菜谱容器，再由菜谱发布记录按 `store_lids` 和发布规则复制/合并/替换到门店菜品容器。

发布完成后，后端会把 `pos_group_dish_book_release.done` 置为 `true`，写入 `done_time`，同时更新 `pos_group_dish_book.published_time`、`published_by`、`status = published`，并对每个目标门店清理菜品缓存、发送 `WMS_SYNC_PRODUCT` 商品同步消息。

## 11. 异常、边界、并发与幂等

| 场景 | 当前证据 | 风险 |
|---|---|---|
| 删除菜品 | 前端提示“不可逆”；基础后端直接 `remove(query)` | 是否校验菜谱/套餐/订单引用待确认 |

## 17. 给后续开发者的修改提示

修改品牌菜品前，必须先确认当前环境到底使用的是 `pt_dish_new/*` 还是基础 `pt_dish/*`。前端主写操作已经全部指向 `pt_dish_new`，而这个接口族在当前后端仓库和已知远端分支中都没有实现。如果只改 `PtDishController` / `PtDishServicePlus`，很可能不会影响真实线上品牌菜品写入链路。

高风险点：

1. 不要把品牌 `sid` 和门店 `sid` 混用。
2. 不要只看 `PtDishController` 就断定品牌菜品后端行为。
3. 删除菜品前必须确认菜谱、套餐、门店同步、订单历史引用。
4. 修改售价必须确认 `P1` 审批是否开启，以及审批通过后的写回链路。
5. 品牌到门店不是“直接复制”，而更像“品牌菜品 -> 品牌菜谱 -> 发布门店”；改任何一段都要把整条链路一起看。
6. 发布到菜谱必须确认异步任务的幂等、失败明细和部分成功处理。
7. 品牌利润部门字段需要同时处理 `departmentCode` 和 `department`。

当前最应该补充的证据是：找到并阅读 `pt_dish_new/*` 和 `pt_dish_type/listTree` 的真实后端实现，并确认它们与 `nms4pos` 里的品牌菜谱发布链路是否部署在同一运行环境。
