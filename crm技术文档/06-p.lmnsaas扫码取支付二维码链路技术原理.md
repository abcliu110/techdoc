# p.lmnsaas扫码取支付二维码链路技术原理

## 1. 文档目的

本文用于说明客户扫码 `https://p.lmnsaas.com/?S=...&M=...&B=...` 后，系统如何从线上页面联动到门店 POS，再由 POS 调支付中心生成真实支付二维码，并最终把结果返回给客户手机。

重点解释以下问题：

- 这条链路涉及哪些项目
- 每一步使用什么通信方式
- `nms4cloud-mq` 项目在这里扮演什么角色
- Netty 的 `subscribe`、`Topic`、`MsgID` 分别解决什么问题
- 为什么客户扫码后不是直接去支付中心，而是要先经过 POS
- 超时通常会卡在哪一步

## 2. 参与项目与职责

这条链路至少涉及 5 个部分。

## 3. 这条链路用了哪些通信方式

本链路不是单一通信模式，而是混合了 3 种通信方式。

### 3.1 浏览器 HTTP

手机 H5 页面和线上服务之间，使用普通 HTTP/HTTPS。

包括：

- 手机访问 `p.lmnsaas.com`
- H5 调 `/pay/get_qrcode_from_offline`
- POS 调 `/pay/receive_qrcode_from_offline` 回推结果

### 3.2 Netty 长连接

`nms4cloud-mq` 项目和 POS 之间，使用 Netty 长连接。

用途不是做支付，而是做“在线指令分发”：

- POS 启动时订阅主题
- `nms4cloud-mq` 根据主题将“生成二维码”任务推送到指定 POS

### 3.3 POS 到支付中心的 HTTP/Feign

POS 本地业务调用 `nms4cloud-payment` 的 `prePay()`，使用 HTTP/Feign。

这一跳才是真正向支付中心发起预下单请求。

## 4. 为什么扫码后不是直接调支付中心

因为客户扫到的不是最终支付二维码，而是一个“外层中转二维码”。

这个外层二维码由 POS 生成，格式类似：

```text
https://p.lmnsaas.com/?S=...&M=...&B=...
```

它的作用不是直接发起支付，而是：

- 先定位到哪一家门店、哪一张单
- 再由 POS 本地根据当前账单状态、支付方式、门店配置去生成最新的一次真实支付二维码

这样设计通常有几个原因：

- 可复用一张打印在小票上的二维码
- 真正下单前仍然以 POS 本地账单状态为准
- 可以兼容消费单、充值单等不同业务分支
- 可以让 POS 在本地完成支付方式决策、金额校验、交易号生成

## 5. 完整链路时序

下面按步骤解释每一步的技术原理。

### 第 1 步：客户手机打开 `p.lmnsaas.com`

客户扫码后，浏览器打开 H5 页面。

前端页面代码在：

- `D:\mywork\nms4pos-ui\app\pos4pay\src\pages\index\index.tsx`

页面做的第一件事是解析 URL 查询参数：

- `S`
- `B`

当前代码只显式使用了 `S` 和 `B`，`M` 在该页面内没有参与取码请求。

原理上，这一步只是“把小票二维码上的参数交给 H5 页面读取”，还没有和 POS 通信。

### 第 2 步：H5 调 `/pay/get_qrcode_from_offline`

前端通过 `service.ts` 发起 POST 请求：

```text
/api/sc/pay/get_qrcode_from_offline
```

请求体大致为：

```json
{
  "SHOP_ID": "S",
  "billId": "B",
  "isAlipay": false
}
```

线上接收方是 `nms4cloud-mq-app` 中的：

- `GetQrCodeFromOffline`

这一步仍然是浏览器到云端的普通 HTTP 调用。

原理上，这是一次“我要真实二维码”的同步请求，但服务端不会立刻返回结果，而是先把它挂起，等待 POS 处理完成。

### 第 3 步：`nms4cloud-mq` 生成 Netty 指令

`GetQrCodeFromOffline` 接收到 H5 请求后，会立刻组装一条内部指令消息。

核心字段如下：

```json
{
  "Cmd": "GenQrcodeForBill",
  "Topic": "xxx_server",
  "BillId": "2046101681026846721",
  "openid": null,
  "appId": null,
  "MsgID": "netty_xxx",
  "isAlipay": false
}
```

每个字段的技术含义如下：

- `Cmd`
  - 业务命令名
  - 告诉 POS 这次要执行什么动作
  - 当前命令是 `GenQrcodeForBill`

- `Topic`
  - 路由键
  - 告诉 `nms4cloud-mq` 应该把消息发给哪个 POS

- `BillId`
  - POS 本地业务要处理的账单标识

- `MsgID`
  - 本次云端请求的唯一编号
  - 用来把 POS 回推的结果与最初那次 H5 请求关联起来

这里最重要的两个概念是：

- `Topic` 解决“发给谁”
- `MsgID` 解决“结果回给谁”

### 第 4 步：POS 是如何被定位到的

这一步不是消息队列消费者模式，而是 `nms4cloud-mq` 项目里的 Netty 服务端按 `Topic` 路由。

#### 4.1 POS 启动后主动建立 Netty 长连接

POS 启动后，`nms4cloud-pos3boot` 中的 `NettyClient` 会主动连接 `nms4cloud-mq` 的 Netty 服务端。

#### 4.2 POS 发送 `subscribe` 消息

连接成功后，POS 会发一条控制消息：

```json
{
  "Cmd": "subscribe",
  "Topic": ["uuid_server", "sid_server"]
}
```

这里的 `subscribe` 指的是：

- 订阅 `nms4cloud-mq` 项目里 Netty 服务端维护的 Topic 消息
- 不是订阅 RocketMQ、Kafka、ActiveMQ 的主题

#### 4.3 `nms4cloud-mq-service` 保存 Topic 与连接的关系

服务端 `ServerHandler` 收到 `subscribe` 后，会维护两个映射：

- `topicToContext`
  - 某个 Topic 对应哪些在线 Netty 连接

- `contextToTopic`
  - 某个连接当前订阅了哪些 Topic

这样一来，当 `GetQrCodeFromOffline` 生成一条消息并指定 `Topic=xxx_server` 时，`ServerHandler` 就能从 `topicToContext` 中查出应该发给哪个 POS。

#### 4.4 按 Topic 发给 POS

`nms4cloud-mq-app` 中的 `SendMsgController.send()` 最终会调用 `nms4cloud-mq-service` 中的 `ServerHandler.sendMessage()`。

它会：

- 根据 `Topic` 找到对应连接
- 把 JSON 编码为 Netty 帧
- 将帧写到目标 POS 的长连接上

因此，第 4 步真正的原理不是“谁消费了队列”，而是：

- POS 先用 `subscribe` 登记自己关心的 Topic
- 云端再按 Topic 把消息精确推给对应 POS

### 第 5 步：POS 收到 Netty 消息后如何执行业务

POS 收到 Netty 帧后，进入 `ClientHandler`。

`ClientHandler` 会根据 `Cmd` 做映射：

- `GenQrcodeForBill` -> `/merchant/dwd_bill_ops/createQrcode`

注意，这里并不是直接在 Netty 回调里写全部业务逻辑，而是：

- 将 Netty 指令转换成一次 POS 本机 HTTP 调用
- 调本机地址 `http://127.0.0.1:{port}/merchant/dwd_bill_ops/createQrcode`

这样设计的好处是：

- Netty 通道层只负责传输和路由
- POS 的核心业务仍然收敛在原有 Web 接口中
- 业务入口统一，便于日志、权限、事务和复用

所以：

- Netty 层是“任务投递层”
- POS 本地 Controller/Service 是“任务执行层”

### 第 6 步：POS 本地业务生成支付二维码

POS 本地接口最终进入：

- `DwdBillOpsServiceImpl.createQrcode()`

该方法会执行以下工作：

1. 查询账单
2. 校验订单状态
3. 查询/生成支付方式
4. 创建 `DwdApplyPay`
5. 调 `OrderServiceUtil.createQrcode()`

而 `OrderServiceUtil.createQrcode()` 内部分两种情况：

#### 6.1 走“可重复使用小票二维码”

如果配置启用了 `g_cashierReceiptsDuplicateUsed`，则 POS 不会立刻调用支付中心，而是再次生成一个外层二维码 URL 保存到 `payQrCode`。

这适合某些重复扫码场景。

#### 6.2 走真实支付中心预下单

如果没有走外层复用分支，则会创建 `PrePayRequest`，然后同步调用：

```java
nms4CloudPayService.prePay(prePayRequest)
```

这一步是 POS 到 `nms4cloud-payment` 的 HTTP/Feign 调用。

技术原理是：

- POS 本地把账单金额、交易号、支付类型、回调地址等打包
- 交给支付中心
- 支付中心再去选具体支付通道
- 通道返回真实 `payUrl`
- POS 把真实 `payUrl` 填入 `dwdApplyPay.payQrCode`

### 第 7 步：支付中心做了什么

支付中心入口在：

- `PosDealController.prePay()`

它会根据请求中的：

- `sid`
- `dealType`
- `channelNo`

选择实际支付通道。

尤其在充值场景下，会优先使用 `channelNoForRecharge`。

然后具体支付通道（微信官方、聚合支付、第三方服务商等）会向上游支付系统发起预下单请求，最终返回：

- `payUrl`
- `outTradeNo`

其中 `payUrl` 就是后续手机真正跳转的目标地址。

因此，“客户扫码外层二维码”与“最终真实支付链接”是两层结构：

- 外层二维码：打印在小票上，负责联动 POS
- 内层真实支付链接：支付中心生成，负责真正支付

### 第 8 步：POS 把结果回推给 `nms4cloud-mq-app`

POS 在本地拿到结果后，不会直接回给手机，因为最初的手机请求是在云端 `nms4cloud-mq` 那里挂起的。

所以 POS 会调用：

- `/api/sc/pay/receive_qrcode_from_offline`

请求体大致如下：

```json
{
  "MsgId": "netty_xxx",
  "data": {
    "qr_code": "真实支付链接"
  }
}
```

这里最关键的是：

- `MsgId` 必须沿用最初的 `MsgID`

因为云端要靠它找到“这条结果应该回给哪一个手机请求”。

### 第 9 步：`nms4cloud-mq` 如何把结果关联回原请求

#### 9.1 原请求在第 3 步已经被挂起

在 `GetQrCodeFromOffline` 中，H5 请求不会立即结束，而是注册到 `MonoMgr`。

注册时保存了：

- `msgId`
- `dataKey = GenQrcodeForBill:msgId`
- `CallBackForSink`

含义是：

- 用 `msgId` 标识一笔挂起请求
- 约定未来 Redis 中只要出现 `GenQrcodeForBill:msgId`，就说明 POS 已经回结果

#### 9.2 POS 回推结果后写入 Redis

`ReceiveQrCodeFromOffline` 收到结果后，会把它写入 Redis：

- key：`GenQrcodeForBill:msgId`
- value：POS 回传的 `data`
- ttl：20 秒

#### 9.3 `MonoMgr` 轮询 Redis 唤醒请求

`MonoMgr` 每秒检查一次：

- 是否超时
- Redis 中是否已经有对应 key

如果读到 Redis 中的结果，则：

1. 删除 Redis key
2. 调用 `call(msgId, json)`
3. 把结果写回最初挂起的 `Mono`
4. HTTP 响应返回给 H5

这一步就是整个链路中 `MsgID` 的真正价值：

- `MsgID` 不是给支付中心看的
- 也不是给 POS 做业务主键的
- 它是云端“异步联动结果回填”的关联键

### 第 10 步：H5 收到真实二维码并跳转

当 H5 请求收到：

```json
{
  "qr_code": "真实支付链接"
}
```

页面会执行：

```javascript
window.location.href = qrCode;
```

到这里，客户手机才真正离开 `p.lmnsaas.com` 这个外层中转页，进入支付中心返回的真实支付页。

## 6. 整条链路的核心机制总结

可以把这条链路压缩为两条主线。

### 6.1 路由主线：靠 `Topic`

`Topic` 用来解决：

- 这条“生成二维码”任务应该发给哪个 POS

链路如下：

```text
POS启动 -> subscribe(topic)
-> nms4cloud-mq记录 topicToContext
-> H5请求到来
-> nms4cloud-mq生成 Topic
-> 按 Topic 找到对应 POS 连接
-> 通过 Netty 推送任务
```

### 6.2 结果主线：靠 `MsgID`

`MsgID` 用来解决：

- POS 处理完之后，结果应该还给哪一个客户手机请求

链路如下：

```text
H5请求到来
-> nms4cloud-mq生成 MsgID
-> 挂起该 HTTP 请求
-> POS 收到任务并原样带着 MsgID 回推结果
-> nms4cloud-mq按 MsgID 找回原请求
-> 把结果返回给 H5
```

## 7. 为什么这里不是传统 MQ 组件消费模型

虽然项目名叫 `nms4cloud-mq`，但在这条链路里，真正承担核心联动的是：

- `nms4cloud-mq` 项目中的 HTTP 接口
- `nms4cloud-mq` 项目中的 Netty 服务端
- `MonoMgr + Redis` 的挂起/回填机制

因此这条链路的核心不是：

- RocketMQ 生产者/消费者
- ActiveMQ 订阅/消费

而是：

- HTTP 触发
- Netty 按 Topic 路由
- HTTP 回推结果
- Redis + Mono 关联响应

所以 `subscribe` 在这里的准确含义是：

- POS 向 `nms4cloud-mq` 的 Netty 服务端登记自己关注哪些 Topic
- 不是订阅传统 MQ 组件的 Topic

## 8. 常见超时点与技术含义

客户手机出现超时时，通常意味着下列某一步没有在 20 秒内完成。

### 8.1 POS 不在线

表现：

- `ServerHandler` 找不到对应 `Topic` 的在线连接

技术含义：

- POS 没有成功连上 `nms4cloud-mq` Netty 服务端
- 或者订阅未建立成功

### 8.2 Topic 错误

表现：

- 云端把消息发到了错误 Topic
- POS 虽然在线，但不是这个 Topic 的订阅者

技术含义：

- 路由键错了，任务发错目标

### 8.3 POS 本地业务执行过慢

表现：

- POS 收到命令，但 `createQrcode()` 没能在 20 秒内完成

技术含义：

- 查询账单慢
- 本地事务慢
- 调支付中心慢
- 上游第三方支付通道慢

### 8.4 POS 回推失败

表现：

- POS 已经生成二维码，但 `receive_qrcode_from_offline` 没有成功写入 Redis

技术含义：

- POS 到线上 `api.lmnsaas.com` 网络不通
- 回推接口异常
- `MsgID` 丢失或不一致

### 8.5 `MonoMgr` 超时

表现：

- H5 收到超时错误

技术含义：

- 对应 `MsgID` 的 Redis key 在超时窗口内没有被写入

## 9. 结论

客户扫码取支付二维码的链路，本质上是一个“线上 HTTP 入口 + 云端 Netty 路由 + POS 本地业务执行 + HTTP 回推 + 云端异步结果回填”的联动模型。

其技术核心可以概括为：

- `nms4cloud-mq` 项目负责联动调度
- Netty 负责把任务从云端推到 POS
- POS 本地业务负责真正生成二维码
- 支付中心负责真正向支付通道预下单
- `Topic` 负责定位目标 POS
- `MsgID` 负责定位原始手机请求

只要理解了这两条主线：

- `Topic = 发给谁`
- `MsgID = 回给谁`

就可以准确理解这条链路中每一步的技术原理。
