# 微信点菜到门店接单链路说明

日期：2026-05-31

## 1. 结论

微信小程序点菜到门店接单不是前端直接把完整订单推给 POS。

实际链路是：

1. 小程序调用云端 `/order_bill/crt_order` 创建订单。
2. 云端订单服务保存 `order_bill`、`order_food`、`order_taste`、`order_pay` 等订单数据。
3. 云端向门店发送 `DO_ORDER` 接单通知，通知里主要带订单索引，不带完整菜品明细。
4. 门店 POS 收到 `DO_ORDER` 后，调用云端 `/order_bill/do_get_order` 反查完整订单。
5. POS 把云端返回的 `OrderBillVO`、`OrderFoodVO`、`OrderTasteVO` 转成本地 `DwdBillSendDTO` / `DwdFoodCreateDTO`。
6. POS 调 `DwdBillServicePlus.toOrder()` 落本地 `DwdBill`、`DwdFood`、`DwdCook`、`DwdTaste`。
7. POS 落单成功后回调云端 `confirm_order`，确认接单成功。

因此，任何需要进入门店本地订单的数据，必须满足两段透传：

- 云端 `do_get_order` 必须返回该字段。
- POS 接单代码必须把该字段从 `OrderFoodVO` 映射到本地 DTO，并在落单时持久化。

平台商品券核销凭证也是这个原则：只在云端 `order_food` 保存还不够，必须继续进入 POS 的 `OrderFoodVO -> DwdFoodCreateDTO -> DwdCoupon` 链路。

## 5. POS 门店监听 DO_ORDER

门店监听入口：

- `D:\mywork\nms4pos\nms4cloud-pos3boot\nms4cloud-pos3boot-biz\src\main\java\com\nms4cloud\pos3boot\listeners\OnlineOrderActiveMQListener.java`
- 方法：`doOrder`
- 监听队列：`SCAN_ORDER_MSG:DO_ORDER`

关键逻辑：

```java
@JmsListener(destination = Pos2pluginMqTopic.SCAN_ORDER_MSG + ":DO_ORDER")
public void doOrder(String msg) {
  handle(doOrderHandler, msg, Pos2pluginMqTopic.SCAN_ORDER_MSG + ":DO_ORDER");
}
```

`handle()` 会做几件事：

1. 解析 `OrderMsgDTO`。
2. 把 `mid/sid` 转为正数，适配 POS 本地口径。
3. 使用 `msgId` 加锁，避免同一订单并发接单。
4. 调用 `DoOrderHandler.handle()`。
5. 如果处理失败且 `dealTimes < 3`，10 秒后重新投递。
6. 超过 3 次后调用 `handler.handled()` 清理处理标记。

这说明 POS 接单有基础幂等和重试机制。

## 7. POS 转换并落本地订单

POS 取到 `DoGetOrderVO` 后：

## 9. 先付与后付的区别

后付点菜：

- 小程序下单后，云端创建订单。
- 云端立即发送 `DO_ORDER`。
- POS 接单后落本地台账。
- 结账发生在门店或后续支付链路。

先付订单：

- 支付完成后才触发门店接单。
- `PayOrderServiceImpl.postHandle()` 中判断订单状态已结账，再发送 `DO_ORDER`。
- POS 收到订单后会发现 `payList` 非空，`hadPay = true`。
- `DoOrderHandler` 落菜品后，如果 `hadPay` 为 true，会把消息推进 `CASH_PAY` 队列继续结账处理。

对应云端位置：

- `D:\mywork\nms4cloud\nms4cloud-app\3_customer\nms4cloud-order\nms4cloud-order-service\src\main\java\com\nms4cloud\order\service\c\order\PayOrderServiceImpl.java`
- 方法：`postHandle`

## 10. 与平台商品券凭证传递的关系

门店接单链路决定了平台券凭证的传递边界。

平台商品券正式核销后，云端如果只把 `certificate_id`、`verify_id/write_off_id` 保存到 `order_food`，POS 仍然拿不到，除非满足以下条件：

1. 云端 `OrderFoodVO` 暴露这些字段。
2. 云端 `do_get_order` 的 `OrderFood` 查询包含这些字段。
3. POS 侧 `OrderFoodVO` 有同名字段接收 JSON。
4. `DoOrderHandler.toCreateDTO()` 把字段复制到 `DwdFoodCreateDTO`。
5. `DwdBillServicePlus.toOrder()` 或接单后置逻辑把这些字段转换成 `DwdCoupon`。

POS 本地撤销平台券时依赖 `DwdCoupon`，不是依赖云端 `order_food`。

对齐 POS 的关键字段建议：

| 云端字段 | POS 本地字段 | 说明 |
|---|---|---|
| `platformCertificateId` | `DwdCoupon.couponNo` | POS 撤销时作为平台 `certificate_id` 使用 |
| `platformWriteOffId` | `DwdCoupon.writeOffId` | 平台正式核销返回的 `verify_id` / 凭证结果 |
| `writeOffChannel` | `DwdCoupon.writeOffChannel` | 撤销时选择平台渠道 |
| `platformPrice` | `DwdCoupon.faceAmount` | 券面额或平台优惠金额 |
| `platformPaidAmount` | `DwdCoupon.paidAmount` | 券支付金额，需与 POS 本地平台核销口径一致 |
| `platformCount` / `foodNumber` | `DwdCoupon.numbers` | 核销数量 |
| `productCouponBusinessType` | `DwdCoupon.couponType` | 区分 WP/MP/DP 等业务类型 |

其中最关键的是：

- `DwdCoupon.couponNo` 应保存平台返回的 `certificate_id`。
- 不能用用户扫码的原始券码替代 `certificate_id`，否则 POS 撤销时会把原始券码当作 `certificate_id` 调平台，导致撤销失败。

## 11. 排查接单问题的顺序

如果门店没有收到小程序点菜订单，建议按下面顺序排查：

1. 小程序是否成功调用 `/order_bill/crt_order`。
2. 云端 `order_bill`、`order_food` 是否已保存。
3. 云端 `postHandle()` 是否发送 `DO_ORDER`。
4. `OrderUtil.sendMessageToShop()` 是否成功，失败时是否降级投递 `MQTT_SEND_MSG`。
5. 门店是否监听到 `SCAN_ORDER_MSG:DO_ORDER`。
6. POS `DoOrderHandler` 是否成功调用 `/order_bill/do_get_order`。
7. 云端 `do_get_order` 是否返回完整 `foodList/tasteList/payList`。
8. POS `DwdOrderMsg.revision` 是否显示已处理或处理中。
9. POS `DwdBillServicePlus.toOrder()` 是否保存了本地 `DwdFood`。
10. POS 是否调用 `confirm_order` 回写接单成功。

## 12. 当前设计边界

这条链路的核心边界是：

- `DO_ORDER` 是通知，不是完整订单数据。
- `do_get_order` 是门店接单时的数据源。
- POS 本地落单只认识自己的 DTO 和本地表。
- 云端新增字段如果没有进入 POS DTO，JSON 反序列化时会被丢弃。
- 平台券凭证如果没有沉淀为 POS `DwdCoupon`，后续 POS 撤单/撤销核销无法可靠执行。
