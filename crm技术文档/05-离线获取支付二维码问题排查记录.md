## 1. 问题背景

用户本地通过 PowerShell 调用：

```powershell
Invoke-WebRequest `
  -Method POST `
  -Uri 'http://127.0.0.1:9898/api/sc/pay/get_qrcode_from_offline' `
  -ContentType 'application/json' `
  -Body '{"SHOP_ID":"20stR5mZoJz","billId":"2046101681026846721","isAlipay":false}'
```

目标是调用“从线下获取支付二维码”的接口。

用户最初关注点在：

1. `nms4cloud-netty` 是否有问题
2. Nacos 日志里的连接 reset 是否影响了本次请求
3. 本地 nginx 代理是否有问题

## 2. 现象时间线

### 2.1 第一阶段：连接失败

最初 PowerShell 调用报错：

```text
Invoke-WebRequest : 无法连接到远程服务器
```

这说明当时 `127.0.0.1:9898` 不可达，可能原因：

1. nginx 未启动
2. nginx 未监听 9898
3. 本地后端服务未启动且没有代理

### 2.2 第二阶段：HTTP 已通，但接口返回 `No endpoint`

之后再测时，HTTP 已可访问，出现两种返回：

#### 1. 访问 GET

```powershell

```

返回：

```json

  "code": -1,
  "errorMessage": "No endpoint GET /api/sc/pay/get_qrcode_from_offline.",
  "status": "error",

```

#### 2. 访问 POST

```powershell
Invoke-WebRequest `
  -Method POST `
  -Uri 'http://127.0.0.1:9898/api/sc/pay/get_qrcode_from_offline' `
  -ContentType 'application/json' `
```

```json

  "code": -1,
  "errorMessage": "No endpoint POST /api/sc/pay/get_qrcode_from_offline.",
  "status": "error",
  "success": false

```

这说明：

1. `9898` 上确实已经有 HTTP 服务
2. 请求已经被某个后端应用接收
3. 但当前收到请求的应用里不存在路径 `/api/sc/pay/get_qrcode_from_offline`

## 3. 相关代码链路

### 3.1 真正的二维码接口

接口代码位于：

- [GetQrCodeFromOffline.java](D:/mywork/nms4cloud/nms4cloud-app/1_platform/nms4cloud-mq/nms4cloud-mq-app/src/main/java/com/nms4cloud/mq/app/pay/GetQrCodeFromOffline.java)

核心定义：

```java
@PostMapping(value = "/pay/get_qrcode_from_offline")
public Mono<?> get_qrcode_from_offline(@Valid @RequestBody Body body)
```

结论：

1. 原始后端接口路径是 `/pay/get_qrcode_from_offline`
2. 代码中没有 `/api/sc` 或 `/sc` 前缀

### 3.2 接口内部行为

接口逻辑大致如下：

1. 读取请求体中的 `SHOP_ID`
2. 生成 topic：

```java
String sid = StringUtils.trimToEmpty(body.SHOP_ID);
String topic = sid + "_server";
```

3. 构造 Netty 消息：

```java
nettyMsg.put("Cmd", "GenQrcodeForBill");
nettyMsg.put("Topic", topic);
nettyMsg.put("BillId", billId);
```

4. 调用 `SendMsgController.send(...)`
5. 注册 `Mono` 回调，等待设备返回二维码结果

所以这条链路依赖两个条件：

1. HTTP 请求能正确进到 `nms4cloud-mq`
2. `SHOP_ID + "_server"` 对应的 topic 必须真有设备订阅

## 5. nginx 配置核查结果

本地使用的配置文件：

- [nginx.conf](D:/mywork/nginx-1.19.5-nms4cloud/conf/nginx.conf)

本地 nginx 目录：

- `D:\mywork\nginx-1.19.5-nms4cloud`

与本次问题直接相关的配置为：

```nginx
location ^~ /api/sc/pay/ {
    proxy_pass http://127.0.0.1:18120/pay/;
}
```

理论上，这段配置的目标是把：

- `/api/sc/pay/get_qrcode_from_offline`

转发为：

- `http://127.0.0.1:18120/pay/get_qrcode_from_offline`

但从实际返回的报文看，当前接收方回的是：

- `No endpoint POST /api/sc/pay/get_qrcode_from_offline`

这说明至少在当时那次请求中，后端接收到的仍然是原始路径 `/api/sc/pay/...`，而不是重写后的 `/pay/...`。

可能原因：

1. 当前生效的 nginx 配置不是这份 `nginx.conf`
2. nginx 改完配置后没有 reload
3. 请求被其他 `location` 匹配到了
4. 实际返回 `No endpoint` 的不是预期中的本地 `18120` 目标服务，而是本机上的其他本地服务或本地错误转发目标

因此，nginx 是否完全正确，当前仍需进一步验证。

## 6. Netty topic 订阅核查结果

### 6.1 `pos3boot` 客户端实际订阅逻辑

代码位于：

- [NettyClient.java](D:/mywork/nms4pos/nms4cloud-pos3boot/nms4cloud-pos3boot-biz/src/main/java/com/nms4cloud/pos3boot/netty/client/NettyClient.java)

### 6.3 与当前请求体的冲突

用户传入的请求体是：

```json
{
  "SHOP_ID":"20stR5mZoJz",
  "billId":"2046101681026846721",
  "isAlipay":false
}
```

而 `GetQrCodeFromOffline` 会把它转成：


- `20stR5mZoJz_server`

这意味着：

2. `SHOP_ID` 的取值与 Netty topic 约定很可能不一致

## 7. Nacos / Netty 其他日志判断

用户前面还给过一批 Nacos 和 Netty 相关日志，核心结论如下：

### 7.1 Nacos `Connection reset`

表现为：

1. `StatusRuntimeException: UNAVAILABLE: io exception`
2. `Caused by: java.net.SocketException: Connection reset`
3. 随后马上出现重连成功日志

判断：

1. 这是一次瞬时 gRPC 长连接中断
2. 客户端已自动重连成功
3. 当前没有直接证据表明这就是二维码接口失败的根因

### 7.2 `subscribe` 日志打成 ERROR

例如：

```text
ERROR ... {"Cmd":"subscribe","Topic":[...]}
```

判断：

1. 这更像是日志级别用错了
2. 不是“订阅失败”的证据
3. 真正语义上它更接近普通 `info/debug` 日志

## 8. 当前最有可能的问题排序

按当前掌握的信息，最值得优先排查的原因排序如下：

### 第一优先级：`SHOP_ID` 取值错误

证据：

1. 代码明确把 `SHOP_ID` 直接用于拼 topic
2. 当前传入的 `20stR5mZoJz` 与设备实际订阅的 topic 前缀不一致

结论：

如果 `SHOP_ID` 不改成实际已订阅的：

1. 设备 UUID
2. 或门店 sid

那么即使 HTTP 路径正确，也很可能收不到二维码回包。

### 第二优先级：nginx 路径转发或生效配置问题

证据：

1. 接口实际路径是 `/pay/get_qrcode_from_offline`
2. nginx 配置理论上应把 `/api/sc/pay/...` 转成 `/pay/...`
3. 但实际返回却像是后端收到的仍然是 `/api/sc/pay/...`

结论：

当前仍需确认：

1. nginx 是否已 reload
2. 当前是否真的在使用 `D:\mywork\nginx-1.19.5-nms4cloud\conf\nginx.conf`
3. 该请求是否命中了预期的 `location ^~ /api/sc/pay/`

### 第三优先级：Netty/Nacos 本身故障

当前证据不足。

因为：

1. `nms4cloud-netty` 已正常监听端口
2. `pos3boot` 已发出订阅 topic
3. Nacos 有瞬时 reset，但已经自动恢复

所以暂不应把根因优先归结为 Netty 或 Nacos 故障。

需要强调的是：

1. 这里的判断是在“所有服务均为本地部署”的前提下成立。
2. 因此当前更应优先检查本机内部链路是否对齐：
   - 本地 nginx 是否使用了预期配置
   - 本地 mq 服务是否收到正确路径
   - 本地 netty 服务是否收到正确 topic
3. 不应把问题优先扩展成“远端网络不通”或“外部服务故障”。

## 10. 当前阶段结论

当前这次“离线获取支付二维码失败”的问题，不能直接归因于：

1. `nms4cloud-netty` 服务没起
2. Nacos reset

更值得优先怀疑的是：

1. `SHOP_ID` 与门店实际订阅 topic 不一致
2. nginx 的 `/api/sc/pay/` 转发是否真正生效

在这两个问题确认之前，不建议继续把根因扩大到 Netty/Nacos 层面。
