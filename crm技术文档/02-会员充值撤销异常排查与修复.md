# 会员充值撤销异常排查与修复

> 适用范围：`/api/merchant/member_ops/revokeCharge`、`/crm_card_op/revokeChargeSign`
>
> 关联模块：`nms4cloud-pos2plugin-biz`、`nms4cloud-crm`
>
> 依据：当前代码中的 Controller、AOP、工具类封装、实际联调现象


## 1. 问题概述

本次会员充值撤销问题，前后出现了两类异常：

1. 第一次撤销时报错：

```json
{
  "code": -1,
  "errorMessage": "全局异常，Cannot invoke \"com.mybatisflex.core.update.UpdateChain.set(com.mybatisflex.core.util.LambdaGetter, Object, boolean)\" because \"this.updateChain\" is null",
  "showType": 0,
  "status": "error",
  "success": false,
  "sync": false
}
```

2. 修复后再次测试，出现：

```json
{
  "code": -1,
  "errorMessage": "请求重复，请稍后重试",
  "showType": 0,
  "status": "error",
  "success": false,
  "sync": false
}
```

这两类问题不是同一个根因：

- 第一个问题是代码误用了 MyBatis-Flex 的查询链和更新链。
- 第二个问题是 `@Idempotent` 幂等拦截器把请求判定成了重复提交。


## 2. 接口与调用链路

前端实际调用入口是：

1. `POST /api/merchant/member_ops/revokeCharge`

该接口定义在 POS 侧：

1. `MemberForBizController.revokeCharge(...)`

主要流程如下：

```text
前端
  -> /api/merchant/member_ops/revokeCharge
  -> MemberForBizController.revokeCharge(...)
  -> revokeInner(taskLid, OrderOpTypeEnum.E, user)
  -> revokeBill(dwdBill, user)
  -> nms4CloudCrmService.revokeCharge(request)
  -> CRM: /crm_card_op/revokeChargeSign
```

关键点：

- POS 侧先撤销本地订单、支付、菜品等状态。
- 然后再调用 CRM 执行会员余额撤销。
- 所以第一次异常不一定是 CRM 抛出，也可能发生在 POS 侧撤销前半段流程。


## 3. 第一个问题：`updateChain is null`

## 3.1 现象

最初报错内容为：

```text
Cannot invoke "com.mybatisflex.core.update.UpdateChain.set(...)"
because "this.updateChain" is null
```

这个错误的含义非常明确：

- 代码进入了 `Chain.set(...)`
- 但当前 `Chain` 内部没有初始化 `updateChain`
- 也就是当前拿到的不是“更新链”，而是“查询链”


## 3.2 直接出错点

出错代码位于 `MemberForBizController.revokeBill(...)` 中，原始写法如下：

```java
OrderServiceUtil.forFoodQuery()
    .set(DwdFood::getOrderStatus, OrderStatusEnum.CANCEL)
    .eq(DwdFood::getMid, dwdBill.getMid())
    .eq(DwdFood::getSid, dwdBill.getSid())
    .eq(DwdFood::getSaasOrderNo, dwdBill.getLid())
    .update();
```

而 `OrderServiceUtil.forFoodQuery()` 的定义是：

```java
public static Chain<DwdFood> forFoodQuery() {
  return forQuery(dwdFoodMapper);
}
```

可以看到：

- `forFoodQuery()` 返回的是查询链
- 查询链可以 `.eq(...)`
- 但不能 `.set(...)`、`.update()`

因此这段代码在运行到 `.set(...)` 时，会进入 `Chain.set(...)`，最终触发：

```java
this.updateChain.set(getter, value, isEffective);
```

此时 `updateChain == null`，所以抛出空指针。


## 3.3 根因分析

根因不是请求参数错误，也不是数据库数据异常，而是 API 使用错误。

本项目对 MyBatis-Flex 做了统一封装：

- `Chain.forQuery(...)` 或 `forFoodQuery()`：查询链
- `Chain.forUpdate(...)` 或 `forFoodUpdate()`：更新链

两者对业务代码的外观很像，都是 `Chain<T>`，因此容易出现误用：

```java
forQuery().set(...).update()
```

这种写法编译能通过，但运行时一定会报错。


## 3.4 为什么第一次失败、第二次可能成功

这个现象并不矛盾，原因通常是：

1. 第一次请求进入撤销流程后，前半段已经执行了部分逻辑。
2. 执行到 `forFoodQuery().set(...).update()` 时报错中断。
3. 第二次再发同样请求时，前置状态已经变化，可能绕过了第一次触发异常的路径，或者执行分支不同。

因此：

- 第二次成功，不代表第一次的错误不存在。
- 这类问题本质上仍然是代码缺陷。


## 3.5 修复方案

正确写法应该使用更新链：

```java
OrderServiceUtil.forFoodUpdate()
    .set(DwdFood::getOrderStatus, OrderStatusEnum.CANCEL)
    .eq(DwdFood::getMid, dwdBill.getMid())
    .eq(DwdFood::getSid, dwdBill.getSid())
    .eq(DwdFood::getSaasOrderNo, dwdBill.getLid())
    .update();
```

本次已修复为 `forFoodUpdate()`。


## 3.6 修复结论

第一个问题的最终结论：

1. 异常发生在 POS 侧 `nms4cloud-pos2plugin-biz`，不是最初怀疑的 CRM 主链路。
2. 根因是把查询链 `forFoodQuery()` 错当成更新链使用。
3. 修改为 `forFoodUpdate()` 后，该异常根因已消除。


## 4. 第二个问题：“请求重复，请稍后重试”

## 4.1 现象

在修复 `forFoodQuery()` 之后，撤销接口返回：

```json
{
  "code": -1,
  "errorMessage": "请求重复，请稍后重试",
  "showType": 0,
  "status": "error",
  "success": false,
  "sync": false
}
```

这个错误不来自撤销业务代码本身，而是来自 AOP 幂等拦截。


## 4.2 触发位置

`revokeCharge(...)` 方法带了 `@Idempotent`：

```java
@Idempotent
@PostMapping(value = "/revokeCharge")
public NmsResult<CardBalanceVO> revokeCharge(@RequestBody CrmCardOpRevokeChargeDTO request) {
    ...
}
```

`@Idempotent` 只是标记，真正生效的是切面：

```java
@Around("@annotation(idempotent)")
public Object aroundMethod(ProceedingJoinPoint joinPoint, Idempotent idempotent)
```


## 4.3 判重规则

切面会提取以下信息生成幂等 key：

1. 当前登录 token：`tokenValue`
2. 请求体中的 `devId`，若为空则取 `-1`
3. 请求体中的 `source`，若为空则取 `PC`
4. 当前 `@PostMapping` 的路径，即 `"/revokeCharge"`

生成规则如下：

```java
String key = String.format("%s::%s::%s::%s", tokenValue, devId, source, cmd);
```

也就是：

```text
tokenValue::devId::source::/revokeCharge
```

随后执行：

```java
long now = SystemClock.now();
long startAt = idempotentLockMap.computeIfAbsent(key, k -> now);
if (startAt != now) {
    Assert.isTrue(now - startAt > 15 * 1000, "请求重复，请稍后重试");
    idempotentLockMap.put(key, now);
}
```

含义如下：

1. 第一次请求进入时，`key` 不存在，放入 `idempotentLockMap`。
2. 如果同一个 `key` 在 15 秒内再次进入，则命中旧值。
3. 命中旧值后，如果 `now - startAt <= 15秒`，直接抛出：
   `请求重复，请稍后重试`


## 4.4 为什么它容易误判

当前 key 粒度过粗，只包含：

- token
- devId
- source
- 接口路径

没有包含实际业务主键，例如：

- `taskLid`
- `cardNo`
- `lid`

这会导致：

1. 同一个操作员，在同一设备上，对同一接口发起不同撤销单，也可能被视为“同一个请求”
2. 接口完成后短时间再次点击，也容易被拦截
3. 前端自动重试、双击、界面回调二次触发，都可能命中判重


## 4.5 `finally` 中的延迟删除逻辑

切面在 `finally` 中还有一段逻辑：

```java
put(key, 3 * 1000, 0);
```

它来自 `AbstractMpScHandler` 的延迟队列机制，作用是：

- 3 秒后异步删除这个幂等 key

也就是说：

1. 请求处理结束后，不是立刻删除 key
2. 而是延迟 3 秒删除

因此短时间内再次点击，更容易触发“重复请求”。


## 4.6 当前实现的风险

当前 `@Idempotent` 实现有两个主要风险：

1. 业务粒度过粗

- 不同 `taskLid` 的撤销，只要 token、devId、source、路径相同，就会被判为重复

2. 时间窗口偏保守

- 内存 map 中按 15 秒窗口判重
- 同时又有 3 秒延迟删除
- 对“撤销、退款、充值”这类操作来说，可能造成误伤


## 5. 建议修复方案

## 5.1 第一优先级：缩小幂等 key 粒度

对于 `revokeCharge` 这类接口，建议把业务主键一起放进 key，例如：

```java
String taskLid = Optional.ofNullable(request.getString("taskLid")).orElse("");
String key = String.format("%s::%s::%s::%s::%s", tokenValue, devId, source, cmd, taskLid);
```

这样：

- 同一笔撤销单仍然能防重复
- 不同 `taskLid` 不会互相误伤


## 5.2 第二优先级：明确幂等语义

建议按接口类型区分幂等策略：

1. 支付、退款、撤销类接口

- 按业务单号幂等
- 例如 `taskLid`、`orderId`、`outTradeNo`

2. 纯查询接口

- 一般不需要 `@Idempotent`

3. 新增、保存类接口

- 可按请求单号、客户端唯一流水号或业务对象主键幂等


## 5.3 第三优先级：优化时间窗口

如果前端存在自动重试或操作链较长，建议评估：

1. 是否需要把 15 秒窗口缩短
2. 是否需要成功后立即清理 key，而不是统一延迟 3 秒
3. 是否需要对“失败”和“成功”区分不同的锁释放策略


## 5.4 第四优先级：增加日志

建议在判重时输出关键日志，至少打印：

- key
- tokenValue 摘要
- devId
- source
- cmd
- taskLid
- startAt
- now

这样后续如果再出现“请求重复”，可以快速确认是否是误判。


## 6. 本次修复与后续建议

## 6.1 已完成修复

本次已完成的代码修复：

1. 将 `revokeBill(...)` 中错误的 `forFoodQuery()` 改为 `forFoodUpdate()`


## 6.2 建议继续处理

建议继续做以下优化：

1. 调整 `IdempotentAspect` 的 key 生成逻辑，把 `taskLid` 纳入判重维度
2. 对充值撤销、消费撤销、退款等敏感接口统一梳理幂等策略
3. 全局排查类似误用：

- `forXXXQuery().set(...)`
- `forXXXQuery().update()`

4. 为撤销链路补充回归测试，覆盖：

- 首次撤销成功
- 短时间重复点击
- 不同 `taskLid` 连续撤销
- 撤销后本地订单/支付/菜品状态正确更新


## 7. 最终结论

本次问题最终可以归纳为两部分：

1. 最开始的 `updateChain is null`

- 根因：`forFoodQuery()` 被误用于更新
- 位置：POS 侧 `revokeBill(...)`
- 结果：运行时抛出 `this.updateChain is null`
- 处理：改为 `forFoodUpdate()`

2. 后续出现的“请求重复，请稍后重试”

- 根因：`@Idempotent` 切面按 `token + devId + source + 接口路径` 判重
- 问题：未纳入 `taskLid`，判重粒度过粗
- 结果：短时间重复撤销或不同单据撤销都可能被误判
- 建议：将 `taskLid` 等业务主键纳入幂等 key

一句话总结：

- 第一个问题是代码写错了。
- 第二个问题是幂等策略设计得太粗。
