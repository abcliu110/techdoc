# 离线获取支付二维码 - 完整流程与修复方案

> 接口：`POST /pay/get_qrcode_from_offline`
>
> 涉及服务：`nms4cloud-mq`、`nms4cloud-netty`、`nms4cloud-pos3boot`、本地 nginx
>
> 排查日期：2026-04-20

## 1. 完整调用链路

```
前端/调用方
    │
    │ POST /api/sc/pay/get_qrcode_from_offline
    │ Body: {SHOP_ID, billId, isAlipay}
    ▼
nginx (9898)
    │
    │ location ^~ /api/sc/pay/ → proxy_pass http://127.0.0.1:18120/pay/
    │ 路径重写: /api/sc/pay/xxx → /pay/xxx
    ▼
nms4cloud-mq (18120)  ── GetQrCodeFromOffline.java
    │
    │ 1. topic = SHOP_ID + "_server"
    │ 2. 构造消息: {Cmd:"GenQrcodeForBill", Topic, BillId, MsgID, isAlipay}
    │ 3. serverHandler.sendMessage(topic, msg)
    │ 4. 注册 Mono 回调(CallBackForSink)，等待设备回包
    ▼
消息转发层（由 useNettyService 配置决定）
    │
    │ useNettyService=true:  NettyReactiveFeign → HTTP → nms4cloud-netty (18190)
    │ useNettyService=false: 直接从本地 topicToContext 找 channel 发送
    ▼
nms4cloud-netty (TCP 9999)
    │
    │ 根据 topic 在已订阅设备中查找对应 channel
    │ 通过 TCP 长连接推送消息到设备
    ▼
pos3boot (门店收银设备)
    │
    │ 启动时通过 NettyClient 连接 netty:9999
    │ 订阅 topics: [uuid+"_server", sid+"_server"]
    │ 收到 Cmd:"GenQrcodeForBill" 后:
    │   → 本地调用支付渠道生成二维码
    │   → 通过 netty TCP 连接回包（携带原始 MsgID）
    ▼
nms4cloud-netty → nms4cloud-mq (回包)
    │
    │ 回包通过 topic_for_http_sync 主题传回 mq
    │ MonoMgr 根据 MsgID 匹配到等待中的 CallBackForSink
    │ Mono 被 resolve，返回二维码数据
    ▼
调用方拿到二维码 → 展示给顾客扫码支付
```

## 2. 关键代码位置

| 组件 | 文件 | 作用 |
|------|------|------|
| 接口入口 | `nms4cloud-mq/.../app/pay/GetQrCodeFromOffline.java` | 接收请求，拼 topic，发消息，等回包 |
| 消息发送 | `nms4cloud-mq/.../app/netty/controller/SendMsgController.java` | 调用 serverHandler.sendMessage |

## 3. 关键配置（Nacos: nms4cloud-mq.yaml）

```yaml
netty:
  server:
    host: 10.11.252.71      # netty 服务地址（NettyClient 连接目标）
    port: 9999              # netty TCP 端口
    useNettyService: false  # false=本地发送, true=转发到独立 netty 服务
```

## 4. 关键依赖条件

| 条件 | 说明 | 验证方式 |
|------|------|----------|
| SHOP_ID 正确 | 必须是设备实际订阅的 topic 前缀（门店 sid 或设备 uuid） | 对比 pos3boot 日志中的 subscribe topic |
| nginx 路径转发 | `/api/sc/pay/` → `/pay/` | `curl http://127.0.0.1:18120/pay/get_qrcode_from_offline` |
| mq → netty 可达 | mq 能把消息送到 netty 服务 | 检查 useNettyService 配置和网络连通性 |
| 设备在线 | pos3boot 已连接 netty 并订阅了对应 topic | `netstat` 查看 9999 端口 ESTABLISHED 连接 |
| 回包链路通 | 设备回包能通过 MsgID 匹配回 mq 的 Mono | 检查 mq 的 NettyClient 是否连上 netty |

## 5. 本次问题根因

### 直接原因

`useNettyService: false` 且 mq 服务本地无设备 TCP 连接。

`ServerHandler.sendMessage()` 在 `useNettyService=false` 时从本地 `topicToContext` 查找设备 channel，但 pos3boot 连接的是独立 netty 服务（PID 12268，端口 9999），不是 mq 服务本身。mq 的 `topicToContext` 为空，消息被静默丢弃，Mono 等待超时。

### 附加问题

- `netty.server.host: 10.11.252.71` 不可达，mq 的 NettyClient 一直 SYN_SENT
- 即使消息能发出，请求中 `SHOP_ID: "20stR5mZoJz"` 与设备实际订阅的 topic 前缀不匹配

### 端口与进程确认

| 进程 | PID | 监听端口 | 说明 |
|------|-----|----------|------|
| nms4cloud-mq | 25744 | 18120 (HTTP) | 仅 HTTP，无 TCP netty 端口 |
| nms4cloud-netty | 12268 | 9999 (TCP) + 18190 (HTTP) | 独立 netty 服务 |
| pos3boot | 27516 | - | 作为客户端连接 192.168.1.192:9999 |
| nginx | 62360 | 9898 | 反向代理入口 |

## 6. 修复方案

### 方案 A（推荐：改 Nacos 配置）

修改 Nacos 中 `nms4cloud-mq.yaml`：

```yaml
netty:
  server:
    host: 192.168.1.192     # 改为本地 netty 地址
    port: 9999
    useNettyService: true   # 改为 true，通过 Feign 转发到独立 netty 服务
```

效果：
- `useNettyService: true` → mq 通过 `NettyReactiveFeign` 调用 `nms4cloud-netty` 的 HTTP 接口（18190）发消息
- `host: 192.168.1.192` → mq 的 NettyClient 能连上本地 netty 接收回调
- Nacos 服务发现：`nms4cloud-netty` 已注册在默认命名空间（172.19.16.1:18190，已确认可达）

### 方案 B（停独立 netty，用 mq 内置）

1. 杀掉独立 netty 服务（PID 12268）
2. 保持 `useNettyService: false`，改 `host: 192.168.1.192`
3. 确认 mq 能自行启动 netty server 绑定 9999
4. pos3boot 重连后直接连 mq

风险：需确认 mq 模块有完整的 netty server 启动逻辑。

### 修复后还需验证

修复 `useNettyService` 后，还需用正确的 SHOP_ID 测试：

```powershell
# 用门店 sid 作为 SHOP_ID
Invoke-WebRequest -Method POST `
  -Uri 'http://127.0.0.1:9898/api/sc/pay/get_qrcode_from_offline' `
  -ContentType 'application/json' `
  -Body '{"SHOP_ID":"1684817396849500161","billId":"2046101681026846721","isAlipay":false}'
```

设备实际订阅的 topic 前缀：
- UUID 维度：`d8bc82872426af9accb678a0862db6e3_server`
- 门店 sid 维度：`1684817396849500161_server`
