# 离线获取支付二维码 - 完整流程与修复方案

> 接口：`POST /pay/get_qrcode_from_offline`
>
> 涉及服务：`nms4cloud-mq`、`nms4cloud-netty`、`nms4cloud-pos3boot`、本地 nginx、frpc
>
> 排查日期：2026-04-22

## 1. 完整调用链路

```
前端/调用方
    │
    │ POST /api/mq/pay/get_qrcode_from_offline
    │ Body: {SHOP_ID, billId, isAlipay}
    ▼
nginx (9898 / 80)
    │
    │ location ^~ /api/mq → proxy_pass http://127.0.0.1:18120/
    │ 路径重写: /api/mq/pay/xxx → /pay/xxx
    ▼
nms4cloud-mq (18120)  ── GetQrCodeFromOffline.java
    │
    │ 1. topic = SHOP_ID + "_server"
    │ 2. 构造消息: {Cmd:"GenQrcodeForBill", Topic, BillId, MsgID, isAlipay}
    │ 3. serverHandler.sendMessage(topic, msg)
    │    useNettyService=false → 从本地 topicToContext 查 channel 直接发送
    │ 4. 注册 Mono 回调(CallBackForSink)，等待 pos3boot 回包
    ▼
nms4cloud-mq 内置 NettyServer (TCP 9999)
    │
    │ topicToContext 中存有 pos3boot 订阅的 channel
    │ 通过该 channel 将消息推送到 pos3boot
    ▼
frpc 隧道（本地 9999 ← 远程 clovia.top:9999）
    │
    │ pos3boot 连接 clovia.top:9999
    │ frpc 将远程 9999 流量转发到本地 127.0.0.1:9999（mq 的 NettyServer）
    ▼
nms4cloud-pos3boot
    │
    │ 启动时通过 NettyClient 连接 clovia.top:9999（经 frpc 穿透到本地 mq:9999）
    │ 订阅 topics: [uuid+"_server", sid+"_server"]
    │ 收到 Cmd:"GenQrcodeForBill" 后:
    │   → 调用支付渠道生成二维码（createQrcode）
    │   → HTTP POST 回包到 clovia.top:9898/api/sc/pay/receive_qrcode_from_offline
    ▼
nginx (9898) — location ^~ /api/sc/pay → proxy_pass http://127.0.0.1:18120/pay
    │
    │ 路径重写: /api/sc/pay/xxx → /pay/xxx
    ▼
nms4cloud-mq (18120)  ── ReceiveQrCodeFromOffline.java
    │
    │ 根据 MsgID 找到等待中的 Mono（MonoMgr / CallBackForSink）
    │ Mono 被 resolve，携带二维码数据返回
    ▼
调用方拿到二维码 → 展示给顾客扫码支付
```

## 2. 关键架构说明

### 2.1 进程与端口对应关系

| 进程 | PID（示例） | 监听端口 | 说明 |
|------|------------|----------|------|
| nms4cloud-mq | 36316 | 18120 (HTTP) + 9999 (TCP NettyServer) + 8720 | mq 服务，同时内置 netty server |
| nms4cloud-netty | 32720 | 18190 (HTTP) + 8723 (TCP) | 独立 netty 服务，本地开发不参与此流程 |
| nms4cloud-pos3boot | 38492 | 9160 / 9180 / 9986 | pos 收银服务，作为 netty 客户端 |
| nginx | - | 9898 / 80 | 反向代理入口 |
| frpc | 24016 | - | 内网穿透客户端 |

### 2.2 useNettyService 的正确配置

**必须使用 `useNettyService: false`（本地模式）**

- `false`：mq 使用自己内置的 NettyServer（9999），从本地 `topicToContext` 找 channel 发消息
- `true`：mq 通过 Feign 调用独立的 nms4cloud-netty 服务（18190）发消息

本地部署时 pos3boot 通过 frpc 连接到 mq 的 9999，topic 注册在 mq 自己的 `topicToContext` 里。若改为 `true`，mq 去查独立 netty（18190）的 `topicToContext`，两套映射不共享，消息永远找不到 channel。

### 2.3 frpc 穿透配置

pos3boot 配置连接 `clovia.top:9999`，通过 frpc 穿透到本地 mq 的 9999：

```toml
# D:\frp_0.68.1_windows_amd64\frpc.toml
[[proxies]]
name = "netty-9999"
type = "tcp"
localIP = "127.0.0.1"
localPort = 9999      # mq 的内置 NettyServer
remotePort = 9999     # pos3boot 连接的远程端口
```

### 2.4 回包路径

pos3boot 生成二维码后，通过 HTTP 回包：
```
POST http://clovia.top:9898/api/sc/pay/receive_qrcode_from_offline
```

经 frpc 穿透 → 本地 nginx(9898) → `/api/sc/pay/` 路由 → mq(18120)/pay/receive_qrcode_from_offline

### 2.5 为什么需要 mq 中转，而不是直接和 netty 通信

**通信模型不匹配**：

- 前端是 **HTTP 同步请求**，需要等待二维码结果后才能返回响应
- pos3boot 是 **TCP 长连接客户端**，基于异步消息驱动

netty 只是"智能交换机"，负责消息路由投递，不处理业务语义。要将前端的同步 HTTP 请求与 pos3boot 的异步回包对应起来，需要 mq 完成以下三件事：

| 职责 | 实现方式 |
|------|---------|
| MsgID 绑定 | 每个请求生成唯一 `MsgID`，pos3boot 回包时携带，mq 按 ID 匹配 |
| 同步转异步 | 用 Reactor `Mono` 挂起 HTTP 请求，等回包后 resolve |
| 超时控制 | Mono 等待超时返回 `over time` 错误 |

### 2.6 ServerHandler 核心数据结构

**topicToContext**（代码位置：`ServerHandler.java:398`）：

```java
// topic → 订阅该 topic 的所有 channel（Set 支持多设备广播）
Map<String, Set<ChannelHandlerContext>> topicToContext = new ConcurrentHashMap<>();

// channel.hashCode() → 该 channel 订阅的 topic 列表（反向索引，用于断连时清理）
Map<Integer, List<String>> contextToTopic = new ConcurrentHashMap<>();

// channel.hashCode() → 该 channel 订阅的被禁用 topic 数量（快速过滤禁用设备）
Map<Integer, Integer> contextForbiddenCount = new ConcurrentHashMap<>();
```

两个 Map 互为反向索引，支持双向快速查找。

**广播场景**：

同一 topic 可被多个设备订阅（Set 结构），发消息时遍历 Set 逐个推送。适用于连锁门店场景：总部向所有门店广播通知，所有门店设备订阅同一 topic，一次发送全部收到。

```java
// 发送时遍历所有订阅该 topic 的 channel
for (ChannelHandlerContext context : contexts) {
    context.writeAndFlush(buf);
}
```

**同一 channel 上的消息区分**：

一个 TCP 连接（channel）承载所有消息，靠字段区分：

| 字段 | 作用 |
|------|------|
| `Cmd` | 消息类型（subscribe / GenQrcodeForBill 等） |
| `Topic` | 路由目标 topic |
| `MsgID` | 请求响应匹配 ID |
| `ResponseTopic` | 回包走哪个通道（`topic_for_http_sync` 或 `str_topic_for_web_socket`） |

**特殊内置 topic**：

| topic | 用途 |
|-------|------|
| `topic_for_http_sync` | pos3boot 回包给 mq 的 Mono（同步请求响应） |
| `str_topic_for_web_socket` | 回包给 WebSocket 连接 |

**集群支持**：

当前 `topicToContext` 是纯内存结构（`ConcurrentHashMap`），**多节点间不共享**，没有实现跨节点路由。生产多实例部署时需确保同一设备的请求和连接路由到同一节点。

### 4.4 nginx: nginx.conf（9898 和 80 两个 server 均需配置）

```nginx
http {
    large_client_header_buffers 4 64k;   # 防止回包请求头过大 400 错误
    client_header_buffer_size 64k;

    server {
        listen 9898;
        ...
        location ^~ /api/sc/pay {
            proxy_pass http://127.0.0.1:18120/pay;   # pos3boot 回包路由
        }
        location ^~ /api/mq {
            proxy_pass http://127.0.0.1:18120/;      # 前端请求路由
        }
        ...
    }
}
```

**为什么需要 `/api/sc/pay/` 这条规则：**

生产环境有业务网关 `yd4cloud-gateway`（端口 33009），其路由配置为：

```yaml
- id: a04-nmx-chain
  uri: lb://a04-nmx-chain
  predicates:
    - Path=/*api/sc/**
  filters:
    - StripPrefix=2   # 去掉前两段路径，/api/sc/pay/xxx → /pay/xxx
```

网关负责解析 `/api/sc/**` 路由，通过 `StripPrefix=2` 将路径处理后转发给对应服务。pos3boot 回包请求 `POST /api/sc/pay/receive_qrcode_from_offline` 经网关处理后变为 `/pay/receive_qrcode_from_offline`，最终到达 mq。

**本地开发环境没有 `yd4cloud-gateway`**，因此必须在 nginx 中手动模拟网关的路由行为，直接将 `/api/sc/pay` 转发到本地 mq（18120），等价于网关的 `StripPrefix=2` 效果。

## 5. 本次问题根因（2026-04-22）

### 问题一：frpc 缺少 9999 穿透配置

- **现象**：pos3boot 连 `clovia.top:9999` 一直 `SYN_SENT`，无法建立连接
- **根因**：frpc.toml 只配了 80 和 9898，没有配 9999
- **修复**：frpc.toml 加 `netty-9999` 代理

### 问题二：nginx 回包请求头过大（400 错误）

- **现象**：pos3boot 生成二维码后回包到 nginx，nginx 返回 `400 Request Header Or Cookie Too Large`，fastjson 解析 HTML 报错，Mono 永远收不到回包
- **根因**：nginx 默认 `large_client_header_buffers 4 8k`，Cookie/请求头超出限制
- **修复**：nginx.conf 加 `large_client_header_buffers 4 64k`

### 问题三：nginx 缺少 `/api/sc/pay/` 路由

- **现象**：pos3boot 回包请求 `/api/sc/pay/receive_qrcode_from_offline` 走到 nginx 兜底规则，转发错误
- **根因**：nginx 没有 `/api/sc/pay/` 的 location，兜底 `/api/` 规则转发到 `127.0.0.1:80/api/`，循环或 502
- **修复**：nginx.conf 加 `location ^~ /api/sc/pay → 127.0.0.1:18120/pay`

### 干扰项：useNettyService 配置

- **现象**：改为 `useNettyService: true` 后仍然超时
- **根因**：`true` 时 mq 通过 Feign 查 nms4cloud-netty（18190）的 topicToContext，但 pos3boot 连接的是 mq 的 9999，两套 topicToContext 不共享
- **结论**：本地部署必须保持 `useNettyService: false`
