# 会员积分问题分析与设计建议

日期：2026-05-09

## 1. 背景

当前 POS 反结账场景中，会员储值消费后，会员积分明细出现两条扣减积分流水：

- `订单取消 -积分`
- `POS反结账撤销消费赠分 -积分`

同时还存在另一个业务问题：会员折扣、会员价、现金/微信/支付宝支付等非会员卡储值支付场景，没有老的会员卡扣款任务 `cardTaskLid`，导致消费赠积分无法稳定落地。

本问题涉及 CRM、POS、BI、Report11 以及前端会员报表页面。

## 2. 当前核心表职责

### 2.1 `crm_card`

会员卡主表，当前积分余额使用 `points` 字段。

积分增加和撤销都应继续维护：

```text
crm_card.points
```

### 2.2 `crm_card_points_record`

会员积分流水表，是会员积分明细和积分详情报表的主要数据源。

积分增加和撤销都应写入：

```text
balance_before
amount
balance_after
order_bill_id
operation_model
comment
```

其中 `order_bill_id` 应写 POS 原账单号，用于后续和账单关联。

现有 `crm_card_points_record` 没有适合表示“负向撤销流水关联哪条正向赠分流水”的字段。

不建议复用以下字段承载正负流水关系：

```text
order_bill_id：只表示 POS 原账单号，同一账单反结账后重结账会出现多轮积分流水
out_order_bill_id：语义是外部系统单号，不应改成原积分流水号
comment：只适合人工备注，不适合作为可靠关联键
```

因此，本次设计不新增 `crm_card_points_record` 字段。正向流水和负向流水的闭环关系由消费积分任务表记录。

### 2.3 `crm_card_record`

会员卡储值金额流水表，用于充值、消费、退款、反结账、分账、余额汇总等金额报表。

不建议新 POS 消费赠积分逻辑写入此表，否则可能污染储值金额报表。

### 2.4 `crm_deal_task`

老会员卡交易任务表，也就是 Card Task。它是 `CardBalanceService` 的核心任务表。

主要用于：

- 会员卡充值任务
- 会员卡消费任务
- 会员卡撤销任务
- 生成 `crm_card_record`
- 通过 `give_point` 生成积分流水
- 支撑老会员卡交易查询

它不是纯积分任务表，不建议用于新 POS 消费积分。

## 3. 当前双退积分根因

当前存在两条积分撤销路径。

### 3.1 老储值余额引擎

储值消费反结账会调用老接口撤销会员卡消费。

老引擎基于 `CrmDealTask.givePoint` 自动生成积分撤销流水：

```text
CardBalanceService.cancelOrder
  -> genPointsRecord
  -> crm_card_points_record: 订单取消 -积分
```

### 3.2 新 POS 积分补偿逻辑

POS 反结账时又调用新的消费赠积分撤销接口：

```text
revokeConsumePointsSign
  -> crm_card_points_record: POS反结账撤销消费赠分 -积分
```

### 3.3 根因结论

同一笔 POS 消费赠积分同时被两套引擎管理：

```text
老 CardBalance 引擎
新 POS 积分引擎
```

两套引擎都认为自己负责撤销这笔积分，最终导致同一笔订单退了两次积分。

## 4. 会员折扣积分问题

会员折扣、会员价、非储值支付场景通常没有会员卡扣款任务 `cardTaskLid`。

如果继续依赖老 `crm_deal_task`，这些场景没有自然任务承载积分，导致：

- 结账时无法稳定赠积分
- 反结账时无法找到原赠积分
- 无法防止重复赠送或重复撤销
- 无法稳定关联账单和积分流水

因此需要一个独立的 POS 消费积分任务或关联表。

## 5. 是否继续使用现有积分余额模型

建议继续使用现有积分余额模型：

```text
crm_card.points
crm_card_points_record.balance_before
crm_card_points_record.amount
crm_card_points_record.balance_after
```

赠积分：

```text
before = crm_card.points
amount = +赠送积分
after = before + amount
```

撤销积分：

```text
before = crm_card.points
amount = -原赠送积分
after = before + amount
```

需要避免的是使用老储值交易模型承载新 POS 消费积分：

```text
不要写 CrmDealTask.givePoint
不要为了积分新增 crm_card_record
不要依赖 CardBalanceService.cancelOrder 自动撤销 POS 消费积分
```

## 6. 为什么需要新积分任务表

新积分任务表不是为了报表，而是为了业务一致性。

它应承担以下职责：

### 6.1 防重复赠积分

POS 可能重试、补偿任务可能重跑。需要通过唯一键保证一次结账赠分生命周期只赠一次。

反结账后重结账仍然使用同一个 POS 账单号，因此不能继续用 `order_id` 或 `bill_lid` 作为唯一幂等边界。

建议唯一约束：

```text
mid + source + idempotency_key + card_lid
```

字段含义：

```text
order_id：POS 原账单号，用于会员详情展示、账单追踪和报表查询
idempotency_key：一次结账赠分生命周期的唯一键，用于防重复赠分和防重复撤销
```

### 6.2 按原赠送积分撤销

反结账不能重新按当前规则计算积分，因为积分规则可能已经变化。

撤销时应读取任务表中记录的原始赠送积分：

```text
grant_points
grant_points_record_lid
```

### 6.3 防重复撤销积分

反结账也可能重试。任务表需要记录：

```text
revoke_points_record_lid
revoke_time
status
```

### 6.4 关联账单和积分流水

建议记录：

```text
mid
sid
bill_lid
order_id
idempotency_key
card_lid
card_no
grant_points
grant_points_record_lid
revoke_points_record_lid
status
grant_time
revoke_time
```

其中：

```text
order_id
  写 POS 原账单号，例如 2026050910008
  负责会员详情、账单查询、报表展示

idempotency_key
  写 POS 本地消费赠分补偿记录 lid
  负责区分同一账单的多次结账赠分生命周期

grant_points_record_lid
  记录正向赠分流水的 crm_card_points_record.lmnid

revoke_points_record_lid
  记录负向撤销流水的 crm_card_points_record.lmnid
```

因此，正向流水和负向流水的关联关系不放在 `crm_card_points_record` 内，而是通过 `crm_consume_points_task` 查询。

### 6.5 反结账后重结账幂等边界

同一张 POS 账单可能经历：

```text
结账 -> 赠积分
反结账 -> 撤销积分
重新结账 -> 再次赠积分
```

这三步的 `order_id` 和 `bill_lid` 可以相同，但第三步必须产生新的消费赠分生命周期。

推荐边界：

```text
POS 本地 crm_consume_points_grant.lid
  -> 作为 CRM grantConsumePoints / revokeConsumePoints 的 idempotency_key
```

处理规则：

```text
1. 首次结账：创建本地赠分记录，使用该记录 lid 作为 idempotency_key。
2. 结账重试：复用未撤销的本地赠分记录，不创建新 idempotency_key。
3. 反结账：使用原本地赠分记录 lid 撤销原 CRM 任务。
4. 反结账后重结账：旧记录已经 REVOKED，允许创建新的本地赠分记录，生成新的 idempotency_key。
```

## 7. 报表影响分析

### 7.1 BI / Report11 会员积分详情

会员积分详情接口：

```text
member_points_detail
```

数据源是：

```text
crm_card_points_record
```

因此新积分流水会进入积分详情报表。

当前问题是现有报表没有把 `order_bill_id` 查询和展示出来。若要和账单关联，需要补充：

```text
select order_bill_id
```

并增加账单号列或查询条件。

如果需要查看正向赠分流水和负向撤销流水的闭环关系，应通过 `crm_consume_points_task` 的 `grant_points_record_lid` 和 `revoke_points_record_lid` 追溯，不从 `crm_card_points_record` 本身直接反查。

### 7.2 会员充值汇总 / 明细

数据源是：

```text
crm_card_record
```

主要操作类型：

```text
Charge
ChargeCancel
AccountingBluePatch
AccountingRedFlush
```

只写积分流水不会影响充值金额报表。

### 7.3 会员消费汇总 / 明细

数据源是：

```text
crm_card_record
```

主要操作类型：

```text
Consume
CancelOrder
Repay
Reverse
```

只写积分流水不会影响储值消费金额报表。

### 7.4 会员账户详情和 PLUS 汇总

部分报表中的积分字段当前来自：

```text
crm_card_record.give_point
```

如果新积分不再写 `crm_card_record.give_point`，这些报表里的积分字段不会自动体现新 POS 消费积分。

建议后续将这些积分字段改为从 `crm_card_points_record` 汇总。

## 8. 推荐设计

### 8.1 引擎边界

建议明确两套引擎边界：

```text
CardBalance 引擎：
  只负责会员卡储值金额
  维护本金、赠送金额、余额
  写 crm_card_record

POS 消费积分引擎：
  只负责 POS 消费赠积分和撤销积分
  写 crm_card.points
  写 crm_card_points_record
  写积分任务表
```

### 8.2 新 POS 积分引擎职责

结账时：

```text
只要订单使用会员身份且满足积分规则：
  先创建 POS 本地赠分补偿记录
  使用本地赠分记录 lid 作为 idempotency_key 调用 CRM
  CRM 按 idempotency_key 生成或返回积分任务
  更新 crm_card.points
  写 crm_card_points_record 正向积分流水
  积分任务记录 grant_points_record_lid
```

反结账时：

```text
根据积分任务找到原赠送记录：
  若已赠且未撤销，按原 grant_points 撤销
  更新 crm_card.points
  写 crm_card_points_record 负向积分流水
  积分任务记录 revoke_points_record_lid
  更新积分任务状态为已撤销
```

反结账后重结账时：

```text
同一 POS 账单号允许再次结账
旧赠分记录已 REVOKED 时，不再复用旧 idempotency_key
重新创建本地赠分记录
使用新的 idempotency_key 生成新的 CRM 积分任务和正向积分流水
```

### 8.3 不建议的设计

不建议继续将新 POS 消费积分写入：

```text
CrmDealTask.givePoint
crm_card_record.give_point
```

原因：

- 会触发老反结账自动撤销积分
- 容易再次双退
- 会员折扣场景没有自然 Card Task
- 会扩大储值金额链路影响面

也不建议为了表达正负流水关系去复用 `crm_card_points_record.out_order_bill_id` 或 `comment`。正负流水关系应由 `crm_consume_points_task` 保存，积分流水表继续保持现有展示和报表职责。

## 9. 短期兼容策略

如果短期不能立刻改所有报表，可以先保证业务正确：

```text
1. 新 POS 消费积分只写 crm_card.points 和 crm_card_points_record
2. 新增或复用积分任务表做幂等和撤销，幂等键使用 idempotency_key
3. 不再把 POS 消费赠积分写入 CrmDealTask.givePoint
4. 不新增 crm_card_points_record 关联字段，正负流水闭环由任务表记录
5. 先保证不双退、折扣积分可赠可撤销、反结账后重结账可再次赠分
6. 后续再补会员账户详情 / PLUS 汇总的积分口径
```

## 10. 赠分未返回时反结账的并发处理方案

### 10.1 问题场景

存在一个高风险并发场景：

```text
1. POS 结账后创建本地赠分记录，状态为 PENDING
2. POS 调用 CRM 赠分接口，HTTP 请求已经发出，但结果还没有返回
3. 门店操作反结账
4. POS 立即发起撤销积分
5. 此时 CRM 可能还没有创建消费积分任务，撤销接口查不到任务
6. 随后原赠分请求返回成功，CRM 实际已经给会员加了积分
```

如果把“CRM 查不到任务”当作撤销成功，POS 会把本地记录标记为 `REVOKED`，但云端后续可能已经完成正向赠分，最终形成：

```text
本地认为已撤销
CRM 实际已加分
会员积分没有被扣回
```

所以，`PENDING` 状态下的反结账不能直接进入 `REVOKED`，必须进入可重试的撤销中间态。

### 10.2 推荐原则

推荐把反结账设计成“本地撤销意图 + 云端撤销确认”两阶段：

```text
反结账发生时：
  先把本地赠分记录标记为 REVOKE_PENDING
  表示业务上已经要求撤销

CRM 撤销成功时：
  再把本地赠分记录标记为 REVOKED
  表示云端负向积分流水已经落库
```

核心原则：

```text
REVOKED 只能表示云端撤销已确认。
不能用 REVOKED 表示“用户已经点了反结账”。
用户点了反结账但云端还未确认撤销时，只能是 REVOKE_PENDING 或 REVOKE_FAILED。
```

### 10.3 本地状态转换要求

本地表：`crm_consume_points_grant`

正向赠分请求发出后：

```text
lid = 本地赠分记录主键，也是传给 CRM 的 idempotency_key
order_id = POS 账单号
status_ = PENDING
grant_points_record_lid = null
revoke_points_record_lid = null
next_retry_time = 当前时间
```

反结账发生时，不管赠分结果是否已经返回，都先落撤销意图：

```text
status_ = REVOKE_PENDING
last_retry_time = 当前时间
next_retry_time = 当前时间或下次重试时间
last_error_msg = null
```

如果赠分请求稍后返回成功，不能无条件把状态改成 `GRANTED`，而是要按当前状态判断：

```text
当前 status_ = PENDING / GRANT_FAILED:
  status_ = GRANTED
  grant_points_record_lid = CRM 正向积分流水 lid
  grant_time = CRM 赠分成功时间

当前 status_ = REVOKE_PENDING / REVOKE_FAILED:
  status_ 保持 REVOKE_PENDING
  grant_points_record_lid = CRM 正向积分流水 lid
  grant_time = CRM 赠分成功时间
  next_retry_time = 当前时间
  继续等待撤销重试

当前 status_ = REVOKED:
  不允许被晚返回的赠分结果覆盖成 GRANTED
```

如果撤销请求成功：

```text
status_ = REVOKED
revoke_points_record_lid = CRM 负向积分流水 lid
revoke_time = CRM 撤销成功时间
next_retry_time = null
last_error_msg = null
```

如果撤销请求失败：

```text
status_ = REVOKE_FAILED
last_error_msg = 撤销失败原因
next_retry_time = 退避后的下次重试时间
```

### 10.4 CRM 撤销接口要求

CRM 表：`crm_consume_points_task`

撤销接口必须按 `idempotency_key` 查找原消费积分任务：

```text
idempotency_key = POS 本地 crm_consume_points_grant.lid
source_ = POS 消费积分来源
card_lid = 会员卡 lid
```

CRM 查到任务并且任务已赠分：

```text
crm_consume_points_task.status_ = REVOKED
crm_consume_points_task.revoke_points_record_lid = 负向积分流水 lid
crm_card_points_record 增加一条负向积分流水
crm_card.points 扣回原赠送积分
```

CRM 查到任务并且任务已经撤销：

```text
保持幂等成功
不重复扣积分
返回已有撤销结果
```

CRM 查不到任务时，不能返回成功：

```text
返回失败或可重试错误
含义：正向赠分任务尚未生成，稍后重试撤销
POS 收到该结果后必须保持 REVOKE_PENDING / REVOKE_FAILED
不能把本地状态改为 REVOKED
```

### 10.5 为什么不建议用锁作为主方案

不建议用本地长锁包住赠分 HTTP 请求和反结账操作。

原因：

```text
HTTP 请求可能超时，但 CRM 仍然执行成功
POS 进程可能重启，本地锁会丢失
反结账和定时补偿可能来自不同线程
长时间持锁会影响收银操作体验
锁无法解决“请求超时但云端成功”的不确定性
```

更可靠的方案是：

```text
本地状态机表达业务意图
CRM idempotency_key 保证正向赠分幂等
CRM REVOKED 状态保证撤销幂等
POS 定时补偿负责最终一致
```

### 10.6 必须验证的场景

实现后至少验证：

```text
正常结账：
  PENDING -> GRANTED
  CRM 生成正向积分流水

正常反结账：
  GRANTED -> REVOKE_PENDING -> REVOKED
  CRM 生成负向积分流水

赠分未返回时反结账：
  PENDING -> REVOKE_PENDING
  CRM 查不到任务时不能标记 REVOKED
  赠分晚返回后保持 REVOKE_PENDING
  后续重试撤销成功后进入 REVOKED

反结账后赠分晚返回：
  late grant response 不允许覆盖 REVOKE_PENDING / REVOKED 为 GRANTED

重复赠分：
  相同 idempotency_key 不重复加分

重复撤销：
  已 REVOKED 时不重复扣分
```

## 11. 退款路径的积分撤销要求

本次现场验证发现一个独立入口缺口：

```text
POS 前台点击“退款”按钮，账单状态变成“全部退款”
但该路径没有生成负向积分流水
```

原因不是 CRM 撤销接口本身，而是 POS 入口不一致：

```text
反结账入口：
  DwdBillOpsServiceImpl.cancel
  已调用 revokeConsumePointsForBill

日结批量撤销入口：
  DwdBillOpsServiceImpl.batchRevoke
  已调用 revokeConsumePointsForBill

退款按钮入口：
  DwdBillOpsServiceImpl.refundByOrder
  原逻辑只处理退款订单、退款付款、退款菜品、打印和上传
  未调用 revokeConsumePointsForBill
```

因此，积分撤销不应只绑定“反结账”，还应绑定“参与积分的付款方式发生退款”的业务事实。

### 11.1 按付款方式重算

消费赠分的正向计算口径是按付款方式，不是按菜品金额。

退款后也必须保持同一口径：

```text
如果本次退款影响参与积分的付款方式金额：
  先撤销原赠分
  再按退款后的剩余付款方式重新计算应赠积分

如果本次退款只影响不参与积分的付款方式：
  不处理消费赠分
```

字段含义：

```text
bill_lid：必须使用原账单 lid，不使用退款单 lid
grant_points_record_lid：原正向积分流水
revoke_points_record_lid：CRM 撤销成功后返回的负向积分流水
status_：先进入 REVOKE_PENDING / REVOKE_FAILED，云端确认后才能进入 REVOKED
```

### 11.2 整单退款

整单退款是付款方式全部退完的特殊情况。

处理结果：

```text
先撤销原赠分
剩余可积分付款金额 = 0
不再重发新积分
```

### 11.3 部分退款

部分退款不按菜品金额扣积分，而是按退款后的剩余付款方式重算。

示例：

```text
原现金付款 220，正向赠分 2200
本次退款现金 110

积分处理：
  先撤销原赠分：-2200
  再按剩余现金 110 重算赠分：+1100
```

即使重算后的积分数和原积分数相同，只要本次退款影响了可积分付款方式，也要执行撤销和重发，用流水表达退款后的积分重算过程。

### 11.4 调用位置约束

撤销调用必须放在本地退款数据落库之后：

```text
先更新原订单退款状态
再插入退款单、退款付款、退款菜品等本地明细
最后登记积分撤销意图
撤销成功后按当前剩余付款方式重发
```

这样可以降低跨系统不一致：

```text
不能在退款事务尚未完整落库时，先让 CRM 生成负向积分流水
如果 CRM 撤销失败，本地记录保留 REVOKE_PENDING / REVOKE_FAILED
后续由补偿任务继续重试撤销
撤销确认成功后，若原订单仍为已结账且部分退款，再按当前剩余付款方式重发积分
```

## 12. 完整生命周期模型设计

### 12.1 设计目标

开发阶段不再只做最小修补，而是按“可追踪、可补偿、可对账”的方式设计 POS 消费积分生命周期。

需要覆盖的场景：

```text
结账赠分
反结账撤销积分
反结账后再次结账重新赠分
再次反结账再次撤销
部分退款先撤销当前有效积分，再按剩余订单事实重新赠分
全额退款只撤销当前有效积分，不再重发
CRM 调用失败后可补偿
POS 重启后可继续补偿
```

核心原则：

```text
同一订单的同一种 POS 消费积分权益，只能由新 POS 消费积分引擎负责。
POS 本地 session 是唯一运行态。
CRM task 只负责云端幂等和积分流水落地保护。
event 只做不可变审计和投递日志，不作为第二套状态机。
CRM 积分流水保留正负关联，方便审计和对账。
```

### 12.2 为什么不直接用双状态机

不建议让 POS session、POS event、CRM task 同时管理完整生命周期状态。

原因：

```text
POS session 如果有状态，event 也有状态，CRM task 也有状态，
失败或超时时就会出现三处状态不一致。

例如：
  session = REVOKE_PENDING
  event = FAILED
  CRM task = REVOKED

此时到底谁是事实源会变得不清晰。
```

因此，职责要拆清楚：

```text
POS session：
  唯一运行态和补偿调度依据。

POS event：
  append-only 日志，记录发生过什么、投递过什么、返回了什么。

CRM task：
  云端幂等保护，保证同一个事件不会重复生成积分流水。

crm_card_points_record：
  真正的积分增减流水。
```

### 12.3 POS session 表

建议新增 POS 本地表：

```text
crm_consume_points_session
```

它表示同一订单的一轮积分生命周期。

关键字段：

```text
lid：session 主键
mid：商户 ID
sid：门店 ID
bill_lid：POS 原账单 lid
order_id：POS 订单号 / saas_order_key
card_lid：会员卡本地 lid
card_no：会员卡号
session_seq：同一订单同一会员的第几轮生命周期
previous_session_lid：上一轮 session
origin_session_lid：第一轮 session，便于追整条链
status_：session 当前状态
trigger_type：触发来源，CHECKOUT / REVOKE / REFUND / COMPENSATE
trigger_bill_lid：触发本轮变化的退款单或反结账单 lid
trigger_order_id：触发本轮变化的业务单号
grant_points：本轮应赠积分
current_effective_points：本轮当前有效积分
grant_points_record_lid：CRM 正向积分流水 lid
revoke_points_record_lid：CRM 负向撤销流水 lid
current_effective_record_lid：当前有效正向积分流水 lid
rule_lid：本次计算使用的积分规则 lid
rule_revision：本次计算使用的规则版本
eligible_amount_snapshot：本次可积分金额快照
pay_snapshot_json：本次参与积分计算的支付事实快照
retry_count：补偿重试次数
next_retry_time：下次补偿时间
last_error_msg：最近失败原因
last_failed_event_lid：最近失败事件
```

`pay_snapshot_json` 必须保存聚合后的支付事实，不能继续使用 `Map<PayWayEnum, DwdPay>` 覆盖式建模。

支付快照至少包含：

```text
payType
amount
payAmount
principal
give
pointsDeductAmount
canIntegral
```

会员卡支付必须保留本金和赠送金额拆分，否则部分退款后无法按规则正确重算。

### 12.4 POS event 表

建议新增 POS 本地表：

```text
crm_consume_points_event
```

它只做不可变事件日志和投递日志，不承担主状态机。

关键字段：

```text
lid：event 主键
mid：商户 ID
sid：门店 ID
session_lid：所属 session
event_type：GRANT / REVOKE / RECALC / DISPATCH
event_status：NEW / PUBLISHED / SUCCEEDED / FAILED / WAIT_RETRY / DEAD
points_delta：本次积分变化，正数赠分，负数撤销
idempotency_key：传给 CRM 的事件幂等键
related_event_lid：当前事件关联的上一事件
target_points_record_lid：撤销事件要撤销的正向 CRM 流水
produced_points_record_lid：当前事件产生的 CRM 流水
request_snapshot_json：请求快照
result_snapshot_json：返回结果快照
dispatch_state：投递状态
lease_until：事件租约时间，避免多个补偿线程同时处理
attempt_no：第几次投递
last_worker：最近处理线程或设备标识
last_error_msg：最近失败原因
operator：操作人
comment：业务备注
```

event 的作用：

```text
排查时知道谁触发了积分变化。
补偿时知道哪次投递失败。
对账时能还原生命周期过程。
```

event 不负责判断“当前有效积分是多少”，这个判断只看 session。

### 12.5 CRM 流水关联字段

建议给 CRM 积分流水表增加字段：

```text
crm_card_points_record.related_points_record_lid
```

字段含义：

```text
正向赠分流水：
  related_points_record_lid = null

负向撤销流水：
  related_points_record_lid = 被撤销的正向赠分流水 lid
```

这个字段只负责审计关联，不负责幂等。

重复撤销的防护仍然由 CRM `crm_consume_points_task` 或事件幂等键控制。

### 12.6 session 状态机

session 是唯一运行态，建议状态如下：

```text
INIT：已创建，尚未发起赠分
GRANT_PENDING：等待赠分或赠分补偿中
GRANTED：CRM 已确认正向赠分
REVOKE_PENDING：等待撤销或撤销补偿中
REVOKED：CRM 已确认撤销，本轮生命周期结束
SUPERSEDED：本轮已被部分退款后的新 session 替代
FAILED：超过最大补偿次数，需要人工处理
CANCELLED：本地事务或业务取消后废弃
```

核心转换：

```text
结账：
  INIT -> GRANT_PENDING -> GRANTED

反结账：
  GRANTED -> REVOKE_PENDING -> REVOKED

全额退款：
  GRANTED -> REVOKE_PENDING -> REVOKED

部分退款：
  旧 session: GRANTED -> REVOKE_PENDING -> SUPERSEDED
  新 session: INIT -> GRANT_PENDING -> GRANTED
```

注意：

```text
只有旧 session 的 CRM 撤销已经确认成功，才能创建并出站新 session 的赠分。
不能在旧 session 只是 REVOKE_PENDING 时就给新 session 发积分。
```

### 12.7 幂等键设计

推荐使用稳定的 session / event 标识作为幂等根，不建议只拼 `order_id + version`。

建议：

```text
赠分幂等键：
  POS:SESSION:{session_lid}:GRANT

撤销幂等键：
  POS:SESSION:{session_lid}:REVOKE
```

原因：

```text
session_lid 是创建 session 时一次性生成的稳定 ID。
同一 session 重试时幂等键不变。
反结账后再次结账会创建新的 session_lid，不会和旧生命周期冲突。
```

`session_seq` 只用于业务阅读和排序，不作为唯一幂等依据。

### 12.8 当前有效 session 唯一性

同一个订单、同一个会员卡，同一时间只能有一个当前有效 session。

不能只靠“先查后插”保证唯一。

建议使用数据库约束或 CAS 锁保证：

```text
mid + order_id + card_lid + active_flag
```

其中：

```text
active_flag = 1 表示当前有效或正在关闭中的 session
active_flag = 0 表示 REVOKED / SUPERSEDED / CANCELLED / FAILED 终态
```

创建新 session 时必须满足：

```text
不存在 active_flag = 1 的旧 session
或者旧 session 已经完成 CRM 撤销并进入可关闭状态
```

这样可以防止：

```text
重复点击结账生成两个有效 session
退款补偿和人工操作并发生成两个新 session
旧 CRM 回包污染新 session
```

### 12.9 事务边界和 outbox 原则

POS 本地事务内必须同时写入：

```text
订单 / 退款事实
session
首个 event
```

CRM 调用必须在本地事务提交后执行：

```text
本地事务提交前：
  只能写本地事实
  不能调用 CRM 改积分

本地事务提交后：
  dispatcher 扫描 PENDING event
  调用 CRM
  成功后更新 session 和 event
  失败后设置 next_retry_time 等待补偿
```

这样可以避免：

```text
POS 退款事务回滚，但 CRM 已经扣积分
POS 结账事务回滚，但 CRM 已经赠积分
```

### 12.10 部分退款处理原则

部分退款不按退款金额比例扣积分，而是按完整生命周期重算：

```text
1. 找到当前有效 session
2. 创建 REVOKE event，撤销当前有效正向积分
3. CRM 确认撤销成功后，旧 session 进入 SUPERSEDED
4. 按退款后的订单剩余支付事实重新计算积分
5. 如果应赠积分 > 0，创建新 session 和 GRANT event
6. 如果应赠积分 = 0，不创建新 session
```

示例：

```text
原订单现金 220，赠分 +2200
部分退款现金 110

旧 session:
  +2200 -> -2200

新 session:
  剩余现金 110 -> +1100

最终净效果：
  +1100
```

再次部分退款：

```text
撤销当前有效的 +1100
按新的剩余订单事实重新计算
如果剩余应赠 +500，则新建下一轮 session
```

### 12.11 规则快照要求

部分退款重算必须可解释。

如果只读取当前规则，会出现：

```text
结账时规则 A
退款时规则 B
重算结果按规则 B
用户或财务无法解释为什么退款后积分变化不是原规则口径
```

因此 session 必须记录本次计算快照：

```text
rule_lid
rule_revision
eligible_amount_snapshot
pay_snapshot_json
grant_points
```

是否按原规则重算，还是按退款时当前规则重算，需要业务明确。开发阶段为了稳定和可解释，建议使用 session 创建时的规则快照作为本轮依据；新 session 创建时重新记录新的规则快照。

### 12.12 新模型下的 bug 覆盖

该模型可以覆盖当前问题：

```text
结账赠分：
  session_lid 固定，GRANT 幂等，不重复赠分

反结账撤销：
  只撤销当前有效 session，负向流水关联正向流水

再结账：
  旧 session 已结束，创建新 session，不复用旧幂等键

再次反结账：
  撤销最新 session，不会误撤旧 session

部分退款：
  先撤销当前有效积分，再按剩余支付事实重新赠分

CRM 超时：
  event 保留投递记录，session 保留补偿状态

POS 重启：
  补偿任务从 session / event 继续恢复
```

### 12.13 设计评审后的定稿建议

评审后不建议直接上“双运行态”的 session/event。

定稿建议：

```text
强 session：
  保存唯一运行态、当前有效积分、正负流水、快照、补偿字段。

弱 event：
  只做不可变审计和投递日志。

CRM task：
  只做云端幂等，不参与 POS 生命周期判断。

CRM points record：
  增加 related_points_record_lid，用于正负流水关联。
```

必须先补齐以下硬约束再进入开发：

```text
1. 新 POS 消费积分拥有唯一所有权，不能再进入老 CrmDealTask.givePoint 撤销链。
2. session 和首个 event 必须和 POS 订单 / 退款事实同事务落库。
3. 当前有效 session 必须有数据库唯一约束或 CAS 锁。
4. 旧 session 撤销确认成功后，才能创建新有效 session。
5. 支付事实和会员卡本金 / 赠送拆分必须快照化。
6. CRM 负向流水必须关联原正向流水。
```

## 13. 最终结论

当前积分问题的核心不是余额模型问题，而是责任边界问题。

建议：

```text
保留现有积分余额模型。
拆分 POS 消费积分任务。
不要把新 POS 消费积分挂进老 Card Task。
不要把 POS 账单号当作结账赠分生命周期幂等键。
不要复用 crm_card_points_record 现有字段表达正负流水关系，应新增 related_points_record_lid 记录撤销流水对应的正向流水。
金额报表继续读 crm_card_record。
积分明细报表继续读 crm_card_points_record。
需要账单关联时，补齐 order_bill_id 查询和展示。
需要正负流水闭环时，优先通过 crm_card_points_record.related_points_record_lid 追溯，session / task 中的 grant_points_record_lid、revoke_points_record_lid 作为生命周期索引。
```

最关键的设计原则：

```text
同一笔订单的同一种积分权益，只能由一个引擎负责赠送和撤销。
同一次结账赠分生命周期，正向赠分只能有一个稳定 idempotency_key，撤销也只能有一个稳定 idempotency_key。
反结账后重结账属于新的结账赠分生命周期，应允许再次赠分。
```

## 14. 2026-05-10 最新定稿：强 session、弱 event，两张表重建

本节覆盖前面关于单表补偿、`crm_consume_points_task` 直接承载 POS 生命周期、以及是否复用积分流水字段的旧讨论。

会员消费积分属于新开发功能，上一版实现已经验证存在边界问题，因此新方案允许重建 POS 本地积分表结构。

### 14.1 总体原则

```text
强 session：
  POS 本地 session 主表是消费积分生命周期的唯一事实源。
  当前状态、当前有效积分、正向流水、负向流水、重试字段都以 session 为准。

弱 event：
  event 只记录业务动作和唤醒处理器。
  event 不直接决定最终状态，不作为幂等主键，不替代 session 状态机。
  event 丢失或处理失败时，定时补偿仍能从 session 恢复。
```

### 14.2 新建 POS 本地表

```text
crm_consume_points_session
  一次结账赠分生命周期一条记录。
  负责幂等、状态机、正负流水、支付快照、补偿重试。

crm_consume_points_session_event
  一次业务动作一条记录。
  例如结账赠分请求、反结账撤销请求、退款重算请求、处理成功、处理失败。
  只做审计和弱事件唤醒。
```

### 14.3 session 主表关键字段

```text
lid：
  POS 本地 session LID。
  传给 CRM 作为 idempotency_key。

order_id：
  POS 原账单号。
  同一账单反结账后重结账会复用该值，不能作为幂等唯一键。

bill_lid：
  POS 原账单 LID。

card_lid / card_no：
  会员卡身份。

status_：
  GRANT_PENDING
  GRANT_PROCESSING
  GRANTED
  REVOKE_PENDING
  REVOKE_PROCESSING
  REVOKED
  FAILED

grant_points：
  本次生命周期实际赠送积分。

grant_points_record_lid：
  CRM 正向积分流水 LID。

revoke_points_record_lid：
  CRM 负向撤销积分流水 LID。

pay_snapshot_json：
  结账或退款后剩余支付事实快照。
  必须包含支付方式、金额、会员卡本金、会员卡赠送金额、是否参与积分。

next_retry_time / retry_count / last_error_msg：
  补偿处理字段。

parent_session_lid：
  部分退款撤旧后重算时，新 session 指向被撤销的旧 session。
  反结账后重结账也可以为空，因为它是新的结账生命周期。
```

### 14.4 event 表关键字段

```text
lid：
  event LID。

session_lid：
  归属的积分 session。

event_type：
  CHECKOUT_GRANT_REQUESTED
  REVOKE_REQUESTED
  REFUND_RECALCULATE_REQUESTED
  CRM_GRANT_SUCCEEDED
  CRM_GRANT_FAILED
  CRM_REVOKE_SUCCEEDED
  CRM_REVOKE_FAILED

event_payload_json：
  记录本次业务动作的必要快照。
  不作为最终状态依据。

status_：
  NEW
  HANDLED
  FAILED

error_msg：
  弱事件处理失败原因。
```

### 14.5 状态流转

```text
结账：
  创建 session = GRANT_PENDING
  创建 event = CHECKOUT_GRANT_REQUESTED
  事务提交后弱事件处理赠分
  CRM 成功后 session = GRANTED

反结账 / 全退：
  当前有效 session = REVOKE_PENDING
  创建 event = REVOKE_REQUESTED
  事务提交后弱事件处理撤销
  CRM 成功后 session = REVOKED

部分退款：
  当前有效 session = REVOKE_PENDING
  创建 event = REFUND_RECALCULATE_REQUESTED
  先撤销旧 session
  旧 session = REVOKED 后，按剩余支付快照创建新 session = GRANT_PENDING
  新 session 再走正常赠分流程
```

### 14.6 幂等与并发

```text
CRM 幂等键：
  使用 session.lid。

POS 当前有效 session：
  同一 mid + order_id + card_lid 只允许存在一个未终态 session。
  终态 REVOKED 后，反结账再结账可以创建新 session。

弱事件重复投递：
  只允许根据 session 当前状态推进。
  不能因为 event 重复而重复调用赠分或撤销。

赠分未返回时反结账：
  session 先从 GRANT_PENDING / GRANT_PROCESSING 推进到 REVOKE_PENDING。
  正向赠分晚返回时只补 grant_points_record_lid，不覆盖回 GRANTED。
  后续补偿按 REVOKE_PENDING 执行撤销。
```

### 14.7 设计结论

```text
订单动作只负责写 session 和 event。
event 只负责唤醒处理。
处理器只按 session 状态推进。
定时补偿只扫描 session，不依赖 event 必达。
CRM 只负责云端幂等和积分余额流水。
```
