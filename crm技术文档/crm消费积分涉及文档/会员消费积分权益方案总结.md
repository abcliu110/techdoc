# 会员消费积分权益方案总结

日期：2026-05-10

## 1. 背景

本次讨论的问题是 POS 账单使用会员身份后，通过现金、微信、支付宝、会员卡等付款方式结账时，如何稳定发放、撤销和重算会员消费积分权益。

已验证的典型场景包括：

```text
结账赠分
反结账撤销积分
反结账后再次结账再次赠分
再次反结账再次撤销
部分退款后先撤销原积分，再按剩余付款方式重算积分
赠分请求未返回时 POS 已反结账
CRM 超时、POS 重启、补偿任务重跑
```

核心目标不是单纯“加积分”或“扣积分”，而是保证每一次结账积分生命周期都有独立、可追踪、可补偿、可幂等的业务闭环。

## 2. 方案一：继续复用老会员卡交易任务

### 2.1 方案说明

继续使用老的 `crm_deal_task` / `CardBalanceService` 链路，让会员卡消费任务中的 `givePoint` 承载 POS 消费赠分。

### 2.2 优点

```text
改动少。
可以复用老的会员卡余额、积分、撤销逻辑。
储值消费场景天然有 cardTaskLid。
```

### 2.3 问题

```text
非储值支付没有 cardTaskLid，例如现金、微信、支付宝、会员价、会员折扣。
反结账时老引擎可能自动撤销 givePoint，新 POS 消费积分逻辑又撤销一次，导致双退积分。
老任务表语义是会员卡交易，不是纯积分权益。
退款后按剩余付款方式重算积分很难表达。
同一账单多次反结账、重结账无法稳定区分不同积分生命周期。
```

### 2.4 结论

不采用。

新 POS 消费积分不应继续挂进老 `crm_deal_task.givePoint`，否则责任边界不清，容易出现重复赠分或重复撤销。

## 3. 方案二：只用 CRM 任务表做幂等

### 3.1 方案说明

在 CRM 新增 `crm_consume_points_task`，POS 每次赠分和撤销都调用 CRM，由 CRM 通过幂等键防止重复赠分、重复撤销。

### 3.2 优点

```text
CRM 能统一记录正向积分流水和负向撤销流水。
云端能防止 POS 重试导致重复加积分。
对非储值支付场景更友好，不依赖 cardTaskLid。
```

### 3.3 问题

```text
POS 本地缺少强状态机时，无法稳定处理 POS 离线、重启、网络超时。
POS 反结账可能早于正向赠分返回，CRM 任务尚未生成时不能正确撤销。
同一 POS 账单反结账后再次结账，如果只按 order_id 幂等，会误认为已经赠分。
退款后撤旧重算需要 POS 侧知道当前有效积分生命周期，CRM 单表无法替 POS 管理本地业务会话。
```

### 3.4 结论

只做 CRM 任务表不够。

CRM 任务表适合作为云端幂等和积分流水索引，但不适合作为 POS 本地业务生命周期的唯一事实源。

## 4. 方案三：POS 本地单补偿表

### 4.1 方案说明

POS 本地新增 `crm_consume_points_grant`，记录赠分状态、撤销状态、重试次数、CRM 正向流水号和负向流水号。

上一版实现大体属于这个方案。

### 4.2 优点

```text
能支持 POS 本地补偿。
能用本地记录 lid 作为 CRM idempotency_key。
反结账后重结账可以在旧记录 REVOKED 后创建新记录。
比只依赖 CRM 任务表更可靠。
```

### 4.3 问题

```text
表名和语义偏“赠分记录”，实际已经承担 session、状态机、事件、审计多个职责。
部分退款的“撤旧后重算”意图缺少清晰表达。
业务动作历史不完整，排查时只能看最终状态。
event 和 session 混在同一张表中，后续扩展退款、重算、异常补偿会越来越复杂。
```

### 4.4 结论

不作为最终方案。

该方案能修部分 bug，但模型不够清晰，不适合开发阶段追求稳定无 bug 的目标。

## 5. 方案四：强 session、弱 event，两张表重建

### 5.1 方案说明

POS 本地重建两张表：

```text
crm_consume_points_session
crm_consume_points_session_event
```

`session` 是消费积分生命周期的唯一事实源。

`event` 只记录业务动作和唤醒处理器，不决定最终状态。

### 5.2 核心原则

```text
强 session：
  当前积分权益状态只看 session 主表。
  正向流水、负向流水、重试状态、支付快照都落在 session。

弱 event：
  event 只记录结账、反结账、退款、CRM 成功、CRM 失败等动作。
  event 可以重复、可以失败、可以延迟。
  定时补偿不依赖 event 必达，只扫描 session。
```

### 5.3 典型流程

```text
结账：
  创建 session = GRANT_PENDING
  创建 event = CHECKOUT_GRANT_REQUESTED
  事务提交后弱事件唤醒赠分
  CRM 成功后 session = GRANTED

反结账：
  当前有效 session = REVOKE_PENDING
  创建 event = REVOKE_REQUESTED
  事务提交后弱事件唤醒撤销
  CRM 成功后 session = REVOKED

部分退款：
  当前有效 session = REVOKE_PENDING
  创建 event = REFUND_RECALCULATE_REQUESTED
  旧 session 撤销成功后，按剩余支付快照创建新 session = GRANT_PENDING
```

### 5.4 优点

```text
能区分同一账单多次结账积分生命周期。
能处理赠分未返回时 POS 先反结账的乱序问题。
能支持部分退款先全退积分再按剩余付款方式重算。
能支持 POS 重启、CRM 超时、补偿任务重跑。
event 可用于审计和排查，但不会污染状态判断。
```

### 5.5 代价

```text
需要重建 POS 本地表。
需要替换上一版单补偿表实现。
需要定义更明确的状态机和补偿处理器。
开发量高于单表方案。
```

### 5.6 结论

采用。

当前处于开发阶段，目标是尽量稳定、减少隐藏 bug。两张表模型虽然开发量更高，但边界清晰，适合承载后续反结账、退款、重算和补偿场景。

## 6. CRM 侧职责

CRM 仍然需要保留云端任务表：

```text
crm_consume_points_task
```

它的职责是：

```text
按 POS session.lid 做云端幂等。
记录 grant_points。
记录 grant_points_record_lid。
记录 revoke_points_record_lid。
保证 CRM 自身不会重复加积分或重复扣积分。
```

CRM 不负责判断 POS 当前账单是否应该再次赠分。

POS 是否创建新积分生命周期，只能由 POS 本地 session 判断。

## 7. 积分流水表职责

`crm_card_points_record` 继续作为积分明细和报表来源。

积分余额模型保持不变：

```text
赠分：
  balance_before = 当前积分
  amount = 正积分
  balance_after = 当前积分 + 正积分

撤销：
  balance_before = 当前积分
  amount = 负积分
  balance_after = 当前积分 - 原赠分
```

正向和负向流水的生命周期关系优先通过任务表或 session 表追溯：

```text
session.grant_points_record_lid
session.revoke_points_record_lid
crm_consume_points_task.grant_points_record_lid
crm_consume_points_task.revoke_points_record_lid
```

是否在 `crm_card_points_record` 增加 `related_points_record_lid`，属于报表和审计增强，不是 POS 本地 session 模型成立的前提。

## 8. 最终推荐

最终推荐模型：

```text
POS：
  crm_consume_points_session
  crm_consume_points_session_event

CRM：
  crm_consume_points_task
  crm_card_points_record
  crm_card.points
```

最终职责边界：

```text
POS session：
  决定一次结账积分生命周期是否存在、是否有效、是否要撤销、是否要重算。

POS event：
  记录动作和唤醒处理，不决定业务状态。

CRM task：
  做云端幂等和积分流水索引。

CRM points record：
  写积分明细和报表数据。
```

最终设计原则：

```text
同一笔订单的同一次结账积分生命周期，只能有一个 session。
同一个 session 只能成功赠分一次。
同一个 session 只能成功撤销一次。
反结账后重结账必须创建新 session。
部分退款必须先撤销旧 session，再按剩余支付事实创建新 session。
```
