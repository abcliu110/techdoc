# POS 实物券单位稳定 ID 修复说明

日期：2026-05-30

适用范围：CRM 实物券、商户后台实物券配置页、POS 实物券加载与核销链路。

## 1. 背景

当前 CRM 实物券绑定菜品时，会保存菜品单位名称，例如“份”“盒”“斤”。POS 加载或核销实物券时，也会用这个单位名称去匹配门店本地菜品单位。

这个设计在单位名称不变时可以工作，但如果商户后续修改了商品单位名称，就会出现下面的问题：

1. CRM 券上保存的是旧单位名称。
2. POS 本地菜品单位已经同步成新单位名称。
3. POS 按旧名称查询单位时找不到匹配项。
4. 实物券加载、匹配或核销失败。

因此，本问题的根因不是 POS 单点校验错误，而是实物券保存了会变化的展示值，缺少稳定的单位选项 ID。

## 2. 当前链路问题

### 2.1 CRM 保存的是单位名称

实物券菜品绑定关系当前主要保存：

| 字段 | 含义 |
| --- | --- |
| `dishLid` | 菜品 lid |
| `dishName` | 菜品名称 |
| `dishUnit` | 菜品单位名称 |

其中 `dishUnit` 是展示名称，不是稳定标识。单位改名后，旧券上的 `dishUnit` 不会自动变化。

### 2.2 POS 需要按单位定位价格与规格

POS 侧菜品可能存在默认单位和多单位。不同单位可能对应不同价格、换算关系或销售规格，因此实物券不能只按菜品匹配，仍然需要确认券绑定的是哪个单位。

如果跳过单位校验，只按 `dishLid` 核销，会带来错核风险。例如同一个菜品的“份”和“斤”可能价格不同，券本来绑定“份”，但实际核销到了“斤”。

### 2.3 名称不是可靠匹配键

单位名称存在以下变化风险：

- 商户改名，例如“份”改成“例”。
- 多单位名称调整，例如“盒”改成“小盒”。
- 历史券保存旧名称，新菜品资料保存新名称。
- POS 本地数据同步后只保留当前名称。

所以单位名称适合用于展示，不适合作为跨系统长期匹配键。

## 3. 稳定 ID 设计

商品单位接口已经存在稳定单位选项 ID：`PtDishUnitVO.lid`。

这里的 `lid` 不是单纯的“单位表主键”，而是“菜品单位选项 ID”。它有两种来源：

| 单位类型 | 稳定 ID 来源 | 说明 |
| --- | --- | --- |
| 默认单位 | 菜品 `lid` | 默认单位没有独立 `pt_dish_unit` 记录，用菜品 lid 表示该菜品的默认单位选项 |
| 多单位 | `pt_dish_unit.lid` | 多单位有独立单位记录，使用单位记录 lid |

因此建议新增字段 `dishUnitLid`，语义是“菜品单位选项 ID”：

```text
默认单位：dishUnitLid = 菜品 lid
多单位：dishUnitLid = pt_dish_unit.lid
```

同时继续保留 `dishUnit`：

- `dishUnit` 用于页面展示、旧数据兼容、日志和人工排查。
- `dishUnitLid` 用于新链路的稳定匹配。

## 4. 为什么需要新增字段

新增字段是为了不破坏旧接口语义，同时让新券具备稳定匹配能力。

不建议直接复用或替换 `dishUnit`，原因如下：

| 方案 | 问题 |
| --- | --- |
| 继续只保存单位名称 | 单位改名后仍然失败，无法解决根因 |
| 把 `dishUnit` 改成保存 ID | 破坏旧接口语义，历史数据和页面展示会混乱 |
| 核销时只按菜品不校验单位 | 会放宽业务规则，可能不同单位错核 |
| 动态按旧单位名称反查当前单位 | 单位改名后旧名称已经不可查，除非额外维护历史别名表，改动更大 |

因此，新增 nullable 字段是最小兼容方案：

```text
新券：保存 dishUnit + dishUnitLid，优先按 dishUnitLid 匹配。
旧券：没有 dishUnitLid，继续按原 dishUnit 名称匹配。
```

## 5. 修复方案

### 5.1 CRM 数据模型

在 `crm_coupon_dish` 增加 nullable 字段：

```sql
ALTER TABLE crm_coupon_dish
  ADD COLUMN dish_unit_lid BIGINT NULL COMMENT '菜品单位选项ID：默认单位保存菜品lid，多单位保存pt_dish_unit.lid' AFTER dish_unit;
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `dish_unit_lid` | BIGINT | 否 | 菜品单位选项 ID。默认单位保存菜品 lid，多单位保存 `pt_dish_unit.lid` |

该字段允许为空，保证历史数据不需要立即回填。

### 5.2 CRM 接口契约

在实物券相关 DTO/VO 增加 `dishUnitLid`：

| 对象 | 新字段 | 用途 |
| --- | --- | --- |
| `CrmCouponDishAddDTO` | `dishUnitLid` | 后台创建/编辑实物券时提交单位选项 ID |
| `CrmCouponDishVO` | `dishUnitLid` | CRM 返回券绑定菜品单位 ID |
| `CrmCouponOrderVO` | `dishUnitLid` | POS 加载券详情时接收单位选项 ID |

同时在校验入参 `CheckCouponFoodDTO` 增加 `foodUnitLid`，用于 POS 或其他调用方传入订单菜品实际单位选项 ID。

### 5.3 商户后台

实物券配置页选择单位时，需要同时保存：

| 字段 | 值 |
| --- | --- |
| `dishUnitLid` | 单位选项 ID |
| `dishUnit` | 单位展示名称 |

单位下拉选项应使用：

```text
value = 单位选项 lid
label = 单位名称
```

这样新建或编辑后的实物券，都会保存稳定 ID。

### 5.4 CRM 校验逻辑

实物券校验单位时按下面优先级处理：

1. 如果券上有 `dishUnitLid`，且订单菜品有 `foodUnitLid`，优先比较两个 ID。
2. 如果任一方缺少单位 ID，则回退到旧逻辑，继续比较单位名称。
3. 如果券没有绑定单位，保持现有“不限定单位”的逻辑。

伪代码：

```text
if couponDish.dishUnitLid is not null and food.foodUnitLid is not null:
    match by dishUnitLid == foodUnitLid
else:
    match by dishUnit == food.unit
```

这样可以保证：

- 新券不受单位改名影响。
- 旧券仍按旧规则运行。
- 不放宽单位匹配规则。

### 5.5 POS 加载链路

POS 从 CRM 加载实物券详情时，应接收 `dishUnitLid`。

POS 本地已有 `DwdCoupon.productUnitNo` 字段，可优先复用该字段保存单位选项 ID 的字符串形式，避免新增 POS 本地表字段。

加载券商品时按下面优先级解析单位：

1. 如果 `productUnitNo` 有值：
   - 等于菜品 `lid`：表示默认单位。
   - 不等于菜品 `lid`：按 `pt_dish_unit.lid` 查询多单位。
2. 如果 `productUnitNo` 为空：
   - 保持旧逻辑，按 `productUnit` 单位名称匹配。

这样 POS 对新券可以使用稳定 ID，对旧券继续使用名称。

## 6. 影响范围

### 6.1 CRM 后端

主要影响：

- 实物券菜品绑定保存。
- 实物券详情返回。
- 可用券/核销券校验。
- 给 POS 返回的券详情字段。

影响性质：

- 新增字段，兼容旧请求。
- 校验逻辑增加 ID 优先匹配。
- 旧数据没有 ID 时仍走旧逻辑。

### 6.2 商户后台

主要影响：

- 实物券创建表单。
- 实物券编辑表单。
- 单位下拉选项的 value 从单位名称调整为单位选项 ID。
- 提交参数增加 `dishUnitLid`。

影响性质：

- 页面展示仍使用单位名称。
- 老数据没有 `dishUnitLid` 时仍可以展示旧名称。
- 编辑保存后可补上新 ID。

### 6.3 POS 后端

主要影响：

- POS 接收 CRM 券详情。
- POS 本地券对象保存单位选项 ID。
- POS 加载券商品时的单位解析。
- POS 调 CRM 校验时补传订单菜品单位选项 ID。

影响性质：

- 优先使用稳定 ID。
- 旧券或旧 CRM 未返回 ID 时保持名称匹配。
- 不需要新增 POS 本地表字段。

### 6.4 数据库

只新增 CRM 表字段：

```text
crm_coupon_dish.dish_unit_lid
```

不回填历史数据，不修改历史券语义。

## 7. 发布与兼容策略

建议按下面顺序发布：

1. CRM 后端先发布，支持保存和返回 `dishUnitLid`，同时兼容旧请求。
2. 商户后台发布，创建/编辑实物券时开始提交 `dishUnitLid`。
3. POS 发布，加载和校验新券时优先使用稳定 ID。

兼容行为：

| 场景 | 行为 |
| --- | --- |
| 历史券没有 `dishUnitLid` | 继续按 `dishUnit` 名称匹配 |
| 新后台提交 `dishUnitLid` | CRM 保存稳定 ID |
| POS 未升级 | 仍使用 `dishUnit` 名称，行为不变 |
| POS 已升级但 CRM 未返回 ID | 回退名称匹配 |
| CRM 与 POS 都升级 | 优先按稳定 ID 匹配 |

## 8. 验收场景

### 8.1 默认单位改名

1. 创建菜品 A，默认单位为“份”。
2. 创建实物券，绑定菜品 A 默认单位。
3. 确认券保存 `dishUnitLid = 菜品 A 的 lid`。
4. 将菜品 A 默认单位从“份”改成“例”。
5. POS 同步最新菜品资料。
6. POS 加载或核销该实物券。

预期：

- POS 能识别默认单位。
- 不因为旧单位名称“份”失败。
- 实物券仍绑定菜品 A 的默认单位选项。

### 8.2 多单位改名

1. 创建菜品 B，默认单位为“份”，多单位为“盒”。
2. 创建实物券，绑定菜品 B 的“盒”单位。
3. 确认券保存 `dishUnitLid = 盒对应的 pt_dish_unit.lid`。
4. 将多单位名称从“盒”改成“小盒”。
5. POS 同步最新菜品资料。
6. POS 加载或核销该实物券。

预期：

- POS 按 `pt_dish_unit.lid` 找到多单位。
- 不因为旧单位名称“盒”失败。
- 使用当前最新单位名称和价格信息。

### 8.3 历史券兼容

1. 准备一张历史实物券，只有 `dishUnit`，没有 `dishUnitLid`。
2. POS 加载或核销该券。

预期：

- 仍按旧逻辑比较单位名称。
- 不因为新增字段为空而报错。
- 历史券行为不被放宽。

### 8.4 不允许跨单位错核

1. 同一菜品存在默认单位和多单位。
2. 实物券绑定默认单位。
3. 订单里选择该菜品的多单位。
4. POS 或 CRM 执行可用券校验。

预期：

- 单位选项 ID 不一致时不可用。
- 不能只因 `dishLid` 相同就核销成功。

## 9. 回滚方案

如果上线后发现异常，可按下面方式回滚：

1. 应用层先回滚后台和 POS 代码。
2. CRM 后端保留 nullable 字段，不影响旧逻辑运行。
3. 新产生的 `dish_unit_lid` 数据可以暂时保留，不参与旧逻辑。
4. 如需完全回退，确认无代码依赖后再删除 `crm_coupon_dish.dish_unit_lid` 字段。

该方案的关键点是新增字段不改变旧字段含义，因此回滚时不需要迁移历史 `dishUnit` 数据。

## 10. 结论

本次修复不是简单把 POS 的名称匹配放宽，而是把实物券绑定单位从“可变展示名称”升级为“稳定单位选项 ID”。

最终口径：

```text
dishUnit：单位名称，用于展示和兼容旧券。
dishUnitLid：单位选项 ID，用于新券稳定匹配。
```

这样可以在不破坏历史券、不回填历史数据、不放宽单位校验的前提下，解决新建实物券在单位改名后 POS 加载或核销失败的问题。
