# 实物券菜品绑定与核销链路分析

> 状态说明：本文是 POS/CRM 既有链路分析和历史背景文档。小程序当前实现口径以 `03-小程序商品券核销设计文档.md` 和 `04-小程序商品券后端接口实现说明.md` 为准。

## 1. 结论

CRM 系统中的实物券由 `CouponTypeEnum.SWQ` 表示，枚举 code 为 `3`，描述为“实物券”。
业务概念上需要区分：

| 概念 | 业务含义 | 典型表现 |
|---|---|---|
| 单品券 | 指定某一个菜品，只能兑换这个菜品 | 一张券绑定一个明确菜品 |
| 实物券 | 可以兑换实物类商品/菜品，通常可以配置多个可兑换菜品候选 | 一张券可在候选菜品中命中一个进行核销 |

因此，`SWQ` 不应简单理解为“单品券”。它更准确的语义是“实物兑换券/菜品兑换券”。单品券只是实物券的一种更窄的业务形态。

## 2. 当前代码实现状态

当前代码里实物券菜品绑定主要使用 `crm_coupon_dish` 表。

保存逻辑在 `CrmCouponServicePlus.addCouponDish(...)`：

- 只有 `couponType == CouponTypeEnum.SWQ` 才处理 `dishList`。
- 每个 `dishList` 项代表一个门店菜品绑定。
- 绑定前会校验菜品存在、门店存在、单位存在。
- 修改时会先删除旧绑定，再保存请求里的新绑定。

需要特别注意：当前实现用 `Set<Long> sids` 校验“同一个门店只能出现一条有效绑定”。也就是说，从代码现状看：

- 一张实物券可以配置多条绑定记录。
- 但这些记录不能属于同一个门店。
- 在同一个门店下，当前代码只允许绑定一个菜品。

如果业务要求“同一个门店的一张实物券可以在多个菜品中任选其一兑换”，当前实现和业务定义不一致，需要调整 `addCouponDish(...)` 的同门店唯一校验和核销命中逻辑。

## 3. 绑定入口

后台新增优惠券：

```text
POST /crm_coupon/add
```

后台修改优惠券：

```text
POST /crm_coupon/update
```

控制器 `CrmCouponController` 会补齐当前登录商户、门店、操作人信息，然后调用：

```java
crmCouponServicePlus.add(request);
crmCouponServicePlus.update(request);
```

新增和修改 DTO 中都有：

```java
private List<CrmCouponDishAddDTO> dishList;
```

`CrmCouponDishAddDTO` 关键字段：

```java
private Long sid;        // 门店
private Long dishLid;    // 菜品 LID
private String dishUnit; // 菜品单位，可为空
```

## 4. 保存流程

新增时：

```java
addCouponDish(request.getDishList(), entity, true);
iCrmCouponService.save(entity);
```

修改时：

```java
addCouponDish(request.getDishList(), entity, false);
iCrmCouponService.update(entity, query);
```

`addCouponDish(...)` 关键逻辑：

1. 非 `SWQ` 券直接忽略 `dishList`。
2. 清空优惠券主表上的旧菜品字段：`dishUnit`、`dishShopId`、`dishName`、`dishCode`。
3. 遍历请求中的 `dishList`。
4. `dishLid == null` 的项会被跳过。
5. `sid` 必须不为空。
6. 当前代码用 `Set<Long> sids` 限制同一个门店只能绑定一个菜品。
7. 每个有效项调用 `addDish(...)` 生成 `CrmCouponDish`。
8. 如果没有任何有效菜品，报错“实物券，必须要设置对应的菜品”。
9. 修改场景先删除旧 `crm_coupon_dish` 记录。
10. 批量保存新的 `crm_coupon_dish` 记录。

## 5. 菜品和单位校验

`addDish(...)` 会调用商品服务确认菜品：

```java
ptDishFeign.getPtDish(new PtDishGetDTO()
    .setMid(coupon.getMid())
    .setSid(couponDish.getSid())
    .setLid(dishCode));
```

校验规则：

- `dishLid` 必须存在。
- 商品服务必须能查到该菜品。
- 如果传了 `dishUnit`，且与菜品主单位不同，则必须存在于菜品多单位列表中。
- 门店必须存在。

最终保存到 `crm_coupon_dish`：

```java
dish.setMid(coupon.getMid());
dish.setSid(couponDish.getSid());
dish.setCouponLid(coupon.getLid());
dish.setDishLid(dishCode);
dish.setDishName(dishVO.getName());
dish.setDishUnit(couponDish.getDishUnit());
dish.setShopName(shop.getName());
dish.setLid(IdWorkerPlus.getId());
```

## 7. 查询时如何带出绑定菜品

查询优惠券详情时，系统按 `mid + couponLid` 查询 `crm_coupon_dish`，映射成 `CrmCouponDishVO` 后放入：

```java
couponVO.setDishList(dishVOS);
```

会员券订单装配时，会把优惠券上的 `dishList` 放进订单券：

```java
orderVO.setDishList(NullSafeUtils.nullSafe(j.getDishList()));
```

核销校验使用的就是 `CrmCouponOrderVO.dishList`。

## 8. 核销时如何校验

核销入口：`CheckCouponUtil.check(...)`

当订单券类型为：

```java
Objects.equals(type, CouponTypeEnum.SWQ.name())
```

系统进入实物券分支：

1. 从 `orderVO.getDishList()` 获取绑定菜品列表。
2. 如果列表为空，返回“该菜品券没有设置菜品”。
3. 从绑定列表中找当前消费门店 `sid` 对应的菜品。
4. 如果当前门店没有绑定菜品，返回“菜品券所绑定的菜品不属于该门店”。
5. 把命中的绑定菜品写回订单券的 `dishCode`、`dishName`、`dishUnit`、`dishShopId`。
6. 如果 `notNeedCheckFood == true`，只校验绑定关系，不检查订单商品。
7. 否则从订单商品 `foodList` 中查找 `foodNo == dishLid` 的商品。
8. 如果订单中没有该商品，返回“商品券不满足使用条件，订单中无此商品”。
9. 如果绑定了 `dishUnit`，订单商品单位必须一致。
10. 命中后按该商品价格计算本次抵扣金额。

当前代码在第 3 步只按门店取第一条绑定：

```java
dishList.stream().filter(i -> ObjectUtil.equal(i.getSid(), sid)).findAny();
```

这也说明当前实现更接近“每门店一个兑换菜品”。如果要支持同门店多个可兑换菜品，需要把这里改成“取当前门店下所有候选菜品，再与订单商品逐个匹配”。

## 9. 与“限抵商品”的区别

系统里还有另一套“优惠券限抵商品”逻辑，表为 `crm_consume_coupon_food`，实体是 `CrmConsumeCouponFood`。

这套逻辑不是实物券绑定菜品，而是用于普通优惠券限定抵扣范围：

- `couponFoodList` 为空：不限制商品。
- `couponFoodList` 不为空：只统计命中的商品金额。
- 校验时按 `foodId + unit` 与订单商品匹配。

在 `CheckCouponUtil.check(...)` 中，`SWQ` 实物券分支会提前 `return`。因此实物券不会继续走后面的 `checkLimitFood(...)` 限抵商品逻辑。

区别如下：

| 逻辑 | 表 | 适用场景 | 核销含义 |
|---|---|---|---|
| 实物券菜品绑定 | `crm_coupon_dish` | `SWQ` 实物券 | 兑换命中的菜品 |
| 优惠券限抵商品 | `crm_consume_coupon_food` | 现金券、比例券等普通优惠券 | 限定哪些商品金额可参与抵扣 |

## 10. 对业务定义的修正

如果按业务口径：

- “单品券”是指定某个菜品。
- “实物券”可以兑换多个菜品。

那么当前代码里的 `SWQ` 名称应归为“实物券”，不要归为“单品券”。

但当前实现还有两个限制：

1. 同一个门店只允许配置一个绑定菜品。
2. 核销时同一个门店只取一个绑定菜品进行匹配。

所以，若业务要实现“同门店多个可兑换菜品候选”，需要改动保存和核销两处：

- 保存：移除或调整 `Set<Long> sids` 的同门店唯一限制。
- 核销：把 `findAny()` 改成当前门店下候选菜品列表，与订单商品逐个匹配。

## 12. POS 端当前核销实物券的实际流程

POS 端实物券核销不是“最终结账时再处理”，而是在添加优惠券接口中完成核销、加菜、送单。

接口入口：

```text
POST /dwd_bill_ops/addCoupons
```

控制器：

```java
DwdBillOpsForBizController.addCoupons(...)
```

业务实现：

```java
DwdBillOpsServiceImpl.addCoupons(...)
```

当前流程：

1. POS 扫券或录入券码，调用 `/addCoupons`。
2. POS 找到当前订单；快餐模式下如果没有订单，会先开一个快餐订单。
3. POS 根据券码从本地 `DwdCoupon` 查券记录。
4. POS 按券类型取得对应 `CouponHandler`。
5. 对 NMS CRM 实物券，调用 `NmsCouponHandler.writeOff(...)`。
6. `NmsCouponHandler.writeOff(...)` 调用 CRM：

```text
POST /crm_coupon_order/writeOffSign
```

7. 如果是实物券，POS 会把本地券记录中的 `productNo` 转成 `dishLid` 传给 CRM。

```java
writeOffOneDTO.setDishLid(Long.valueOf(dwdCoupon.getProductNo()));
```

8. CRM 核销成功后，POS 将券标记为已核销。
9. POS 根据券对应菜品生成 `DwdFoodCreateDTO`。
10. POS 调用 `dwdBillServicePlus.toOrder(...)`，把菜品加入当前订单。
11. POS 调用 `OrderServiceUtil.send(...)`，把菜品送单，厨房开始制作。
12. POS 调用 `OrderServiceUtil.calc(...)` 重算账单。
13. 后续再进入正常结账流程。

因此，POS 当前更接近以下业务顺序：

```text
扫码/录券
  -> 核券
  -> 核销成功
  -> 自动新增券对应菜品到订单
  -> 厨房出单
  -> 订单重算
  -> 后续结账
```

## 13. POS 端是重新下单菜品，不是匹配已点菜

当前 NMS CRM 实物券路径下，POS 不是查找订单中已经点过的菜品再抵扣，而是在核销成功后重新向订单追加券对应的菜品。

关键代码：`DwdBillOpsServiceImpl.addCoupons(...)`

```java
PtDish product = couponHandler.getProduct(writeOffCoupon);
foods.add(
    new DwdFoodCreateDTO()
        .setFoodNo(product.getLid())
        .setFoodUnit(product.getUnit())
        .setCouponLid(writeOffCoupon.getLid())
        .setCouponNo(writeOffCoupon.getCouponNo())
        .setCouponName(writeOffCoupon.getCouponName())
        .setFoodNumber(...));
```

然后：

```java
dwdBillSendDTO.setFoods(foods);
dwdBillServicePlus.toOrder(dwdBillSendDTO, user);
OrderServiceUtil.send(...);
```

业务含义：

- 券核销成功后，POS 自动把券对应菜品下到当前订单。
- 这道菜会带上 `couponLid`、`couponNo`、`couponName`。
- 厨房按新增菜品出单。
- 它不是对已有点菜单行做抵扣标记。

例外情况：

- 如果前端请求里已经传了 `DwdCouponCreateDTO.foods`，POS 会使用请求里的菜品列表。
- 这通常更接近套餐券或需要前端选择菜品的场景。
- 但当前 NMS CRM 实物券默认路径仍然是根据券上的 `productNo` 自动新增菜品。

## 14. POS 端金额以什么为准

当前实现不是直接把券菜品“单价改为 0”。

更准确的实现是：

1. 券菜品按 POS 本地菜品资料正常下单。
2. 菜品行仍然计算原价、菜品金额、应收金额。
3. 因为菜品行带有 `couponLid`，重算账单时会把这道菜的金额累计到券金额。
4. 订单层面再用优惠券金额抵扣应付金额。

POS 计算逻辑：`GeneralCalcOrderServiceImpl`

### 14.1 菜品金额正常计算

菜品金额按本地 POS 菜品价格计算：

```java
dwdFood.setFoodAmount(
    NumberUtil.mul(dwdFood.getPaidNumber(), dwdFood.getFoodProPrice())
        .setScale(2, RoundingMode.DOWN));
```

然后计算菜品应收：

```java
dwdFood.setPaidAmount(
    NumberUtil.add(
        dwdFood.getFoodAmount(),
        dwdFood.getProcessingFee(),
        dwdFood.getPromotionAmount().negate(),
        dwdFood.getServiceChargeAmount(),
        tasteServiceAmount));
```

### 14.2 券菜品金额累计到券金额

如果菜品行有 `couponLid`：

```java
if (Objects.nonNull(couponLid)) {
  BigDecimal couponAmount = NullSafeUtils.nullSafe(couponMap.get(dwdFood.getCouponLid()));
  BigDecimal couponFoodAmount;
  if (g_groupCouponTakeServiceAmount) {
    couponFoodAmount =
        NumberUtil.sub(
            dwdFood.getPaidAmount(),
            dwdFood.getServiceChargeAmount(),
            dwdFood.getProcessingFeeService());
  } else {
    couponFoodAmount = dwdFood.getPaidAmount();
  }
  couponMap.put(couponLid, NumberUtil.add(couponAmount, couponFoodAmount));
  dwdFood.setCouponAmount(couponFoodAmount);
}
```

也就是说，券菜品不是直接变成 0，而是：

- 菜品本身有金额。
- 菜品行记录 `couponAmount`。
- 该金额会汇总到对应券的 `faceAmount`。

### 14.3 订单应付再扣券金额

重算订单时：

```java
BigDecimal totalCouponAmount =
    oldCouponMap.values().stream()
        .map(DwdCoupon::getFaceAmount)
        .map(NullSafeUtils::nullSafe)
        .reduce(BigDecimal.ZERO, BigDecimal::add);
BigDecimal paidAmount = NumberUtil.sub(order.getPaidAmount(), totalCouponAmount);
order.setCouponAmount(totalCouponAmount);
order.setPaidAmount(paidAmountNew);
```

业务展示上可以理解为：

```text
券菜品正常下单
券菜品金额计入券抵扣
订单应付金额扣掉券抵扣
```

所以，“菜品价格变为 0”不是当前数据库和计算层的实现方式。当前实现是“菜品正常计价，券抵扣同等金额，使顾客应付不增加”。

## 15. POS 当前对多菜品实物券的限制

如果实物券业务定义为“可兑换多个菜品，顾客从多个候选菜品中选择一个”，当前 POS 端还不完整。

当前 POS 对 NMS CRM 路径只接收旧字段：

- `dishCode`
- `dishName`
- `dishUnit`

本地 `CrmCouponOrderVO` 没有接收 `dishList`。因此 POS 无法展示多个候选菜品，也无法让收银员或顾客选择具体兑换哪个菜品。

当前 POS 能处理的是：

```text
CRM 返回一个 dishCode
  -> POS 根据 dishCode 找本地菜品
  -> 核销
  -> 自动新增这个菜品
```

如果要支持“多个菜品任选”，需要补齐：

1. CRM 返回实物券候选菜品列表 `dishList`。
2. POS DTO 接收 `dishList`。
3. POS 前端展示候选菜品。
4. 用户选择一个菜品。
5. POS 将选中的 `dishLid` 传给 CRM 核销。
6. POS 按选中菜品新增订单菜品。
7. 重算时按选中菜品的 POS 本地价格生成券抵扣金额。
