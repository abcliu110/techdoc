**评审日期**：2026-06-01

**评审方式**：基于工作区未提交代码的静态走查 + 跨模块链路核对（云端 nms4cloud + POS nms4pos）
**评审结论**：主链路已闭合，可进入联调/行为测试阶段；遗留 1 个 MEDIUM、3 个 LOW 改进项，均不阻断后端主链路。前端 taro-mall 接入为独立任务，未在本次后端范围内。

## 1. 评审对象与链路

本次功能实现"小程序点餐页扫码/选券 → 预核销写购物车草稿 → 下单时正式核销 → 凭证回写 order_food → POS 接单沉淀 DwdCoupon → 复用 POS 平台撤销链路"的完整后端主链路。

涉及模块与关键文件：

| 模块 | 文件 | 职责 |
|---|---|---|
| nms4cloud-order | `CartProductCouponService` | 购物车预核销（校验+写草稿，不正式核销） |
| nms4cloud-order | `ShoppingCartController` | `/shopping_cart/get_available_product_coupon`、`/pre_write_off_coupon` |
| nms4cloud-order | `CrtPostOrderServiceImpl` | 下单正式核销、凭证回写 order_food、失败补偿 |
| nms4cloud-order | `CalcGenericOrderServiceImpl` | 订单金额核算（商品券优惠口径） |
| nms4cloud-order | `OrderBillServicePlus.doGetOrder` | 接单查询透传平台券字段 |
| nms4cloud-order | `OrderFoodServicePlus` | 公共菜品接口内部券字段防护 |
| nms4cloud-product | `PtDishServicePlus.matchPlatformCouponDish` | 平台券 SKU → 本地菜品 extNames 匹配 |
| nms4pos | `DoOrderHandler.toCreateDTO` | 接单透传平台券字段 + 设置 DwdFood.couponNo |
| nms4pos | `CloudPlatformCouponService` | 沉淀 DwdCoupon（凭证不全抛异常） |
| nms4pos | `DwdBillServicePlus.toOrder` | 接单流程中调用券沉淀 |

## 2. 已闭合的关键设计点（含代码证据）

### 2.1 预核销与正式核销职责分离（对齐需求）

`CartProductCouponService.buildPreWriteOffFood` 仅做校验 + 平台 `prepare` + 写购物车草稿，不调用 CRM `writeOffCouponInner` 或平台 `verify`。正式核销统一在 `CrtPostOrderServiceImpl.writeOffPreWriteOffCoupons`（下单 `/order_bill/crt_order` 时）执行。

意义：避免"未下单先核销"，预核销失败不污染外部券状态。

### 2.2 平台券扫码自动匹配菜品（对齐 POS extNames，P0-02 已修复）

- `CartPreWriteOffCouponRequest` 的 `foodId/unitId` 已从必填降级为可选（仅保留 `@Schema`，无 `@NotNull`）。平台券（MP/DP）只需传 `couponCode`。
- `autoMatchPlatformCouponDish` 在 `prepare` 后取 `sku.sku_id / coupon_name / title`，调用 product 侧 `PtDishServicePlus.matchPlatformCouponDish` 复刻 POS `MtCouponHandler/DyCouponHandler` 的候选查询与"首个命中"规则。
- 匹配成功后回填 `foodId = matchedDish.dishCode`、`unitId = 默认单位`，复用会员券同一套写草稿逻辑。
- 若前端同时传了 `foodId/unitId`，会与自动匹配结果做一致性校验（不一致直接拒绝），兼容旧调用。

### 2.3 平台凭证完整解析与可信落地（P0-03/P0-04 已修复）

`buildPreWriteOffFood` 中：

- `verify_token` 取自平台 `prepare` 返回，正式核销时使用；不随机生成、不信任前端。
- 多凭证拒绝：`Assert.isTrue(certificates.size() == 1, "暂不支持多凭证券包，请逐张核销")`，不静默取 `[0]`。
- `certificate_id`、`encrypted_code`、`channel`、`count` 均从平台返回落到购物车券菜品行。
- 现金券拦截：`groupon_type == 2` 直接拒绝点餐页核销。
- 面额/实收分别取值：`faceAmount = list_market_amount/100`；`paidAmount` 对 DP 优先取 `coupon_pay_amount/100`，缺失退回 `pay_amount/100`。两者缺失时 `Assert` 失败，不静默写空值。

### 2.4 正式核销解析 verify_results 并校验（P0-04 已修复）

`CrtPostOrderServiceImpl.writeOffSinglePreWriteOffCoupon`：

- `verify` 后用 `parsePlatformWriteOffId` 解析 `verify_results`，逐条校验 `verify_id` 非空，任一为空返回 null。
- 调用方对 `platformWriteOffId` 与 `platformCertificateId` 双重 `Assert.isTrue(StrUtil.isNotBlank(...))`，杜绝"平台已核销、本地无凭证"的坏单。
- 撤销数量按 `platformCount`（= `certificate.count`）传递，`encryptedNums = List.of(verifyNum)`，不再固定 1。

### 2.5 凭证持久化为精确一行更新（P1-03 行更新精度已修复）

`persistProductCouponWriteOffSuccesses` 按主键 `lid`（预核销已生成）+ `mid/sid/saasOrderNo/couponWriteOffTraceNo` 定位 order_food，并断言 `matched.size() == 1`，命中 0 行或多行均抛异常，避免静默更新错误数据。

### 2.6 doGetOrder 透传平台券字段（P1-01 已修复）

`OrderBillServicePlus.doGetOrder` 的 OrderFood `select(...)` 已显式补齐 `couponNo / couponWriteOffTraceNo / platformCertificateId / platformWriteOffId / productCouponBusinessType / platformPrice / platformPaidAmount / writeOffChannel`，POS 反查订单可拿到全部沉淀字段。

### 2.7 POS 接单沉淀 DwdCoupon（P0 移植主目标已闭合）

- `DoOrderHandler.toCreateDTO` 透传上述全部平台券字段；并对 MP/DP 设置 `DwdFood.couponNo = platformCertificateId`、`couponName = foodName`，使券菜品受 POS 现有退菜/改数量"先取消券"保护，不被绕过。
- `CloudPlatformCouponService.saveCloudPlatformCoupons` 沉淀 DwdCoupon：
  - 凭证不全（certificateId/writeOffId/channel 任一缺失）**直接抛 BizException**，让 `toOrder` 事务回滚、依赖 DO_ORDER 重试，避免静默跳过形成坏单。
  - 幂等键 `mid + dwdBillLid + couponNo + couponType`，DO_ORDER 重试时更新而非重复插入。
  - `couponNo / writeOffId / writeOffChannel / faceAmount / paidAmount / numbers` 与 POS 本地扫码核销口径一致。

### 2.8 撤销凭证格式与 POS 撤销处理器对齐（关键一致性）

核对 POS `MtCouponHandler.cancelOff` 与 `DyCouponHandler.cancelOff`：

- MP/餐道：`writeOffId` 存整个 `verify_results` JSON 数组（POS 用 `JSON.parseArray` 解析），撤销时 `certificateId` 用 `verify_id`。
- DP/抖音：`writeOffId` 存逗号分隔 `verify_id`（POS 用 `split` 解析），撤销时 `certificateId` 用 `couponNo`（= certificate_id）。

云端 `parsePlatformWriteOffId` 与下单失败补偿 `cancelWrittenOffProductCoupon` 的写入/解析格式与上述 POS 口径完全一致。

### 2.9 订单金额无双扣（P0-06 主要风险已澄清）

`CalcGenericOrderServiceImpl`：

- 订单头 `paidAmount` 基于 `foodAmount`（原价汇总）+ 服务费 - 普通折扣计算；`applyProductCouponAccounting` 之后只对商品券优惠扣减一次（`promotionAmount += totalCouponDiscount`、`paidAmount -= totalCouponDiscount`），不存在行级与订单头双扣。
- `coupon_amt` 为"纯券优惠"：`pureCouponAmount = 行 promotionAmount - 行普通折扣 - 做法折扣`，负值归零，普通折扣不计入券金额。

### 2.10 内部凭证字段防伪造（P1-04 已修复）

- 公共 `OrderFoodAddDTO/OrderFoodUpdateDTO` 的 `couponWriteOffTraceNo` 加 `@JsonIgnore` + `hidden=true`，外部无法注入。
- `OrderFoodServicePlus.add/update` 调用 `clearProductCouponFields` 清空 `couponNo / couponWriteOffTraceNo / platformCertificateId / platformWriteOffId / productCouponBusinessType`，双重防护，防止把普通菜品伪装成券菜品。

### 2.11 DDL 迁移脚本（P0-08 已修复）

`docs/sql/migration/V20260529__order_product_coupon_columns.sql` 覆盖 order_food 8 个字段与 order_bill 3 个快照字段，做存在性检查（幂等可重复执行）。POS 侧 `dwd_coupon` 复用既有列，`write_off_id` 为 `@ColumnPlus(jdbcType=BLOB)`，足以容纳 verify_results JSON（LOW-1 已确认）。

## 3. 遗留问题清单

### MEDIUM-1：旧后付链路 `ProductCouponOrderVO` 仍向前端返回核销凭证

- **位置**：`OrderCouponServicePlus.preWriteOffProductCoupon` → 返回 `ProductCouponOrderVO`（`api/vo/ProductCouponOrderVO.java`）。
- **现状**：该 VO 仍包含 `writeOffResultRawData`、`verifyId`、`certificateId` 字段，并由 `OrderBillController#/write_off_product_coupon_order` 暴露给前端（`@SaCheckLogin`）。
- **风险**：这是历史后付订单链路（guoyun_liu 既有代码），**不影响小程序点餐主链路**（点餐走 `/shopping_cart/pre_write_off_coupon`，正式核销在 `crt_order`，凭证只落库不返回前端）。但旧接口仍把平台撤销凭证（rawData/verify_id/certificate_id）下发到客户端，存在凭证泄露与重放面。
- **建议**：对 `ProductCouponOrderVO` 的 `writeOffResultRawData / verifyId / certificateId` 三个字段加 `@JsonIgnore`（仅内部使用，不序列化给前端），或在旧接口出参处脱敏。属于历史链路收敛，建议与 guoyun_liu 确认后处理，不在本次主链路强行改动其行为。

### MEDIUM-2（观察项）：可用商品券列表语义偏宽

- **位置**：`OrderCouponServicePlus.getAvailableProductCoupon`，使用 `loadAllCoupon=true` + `notNeedCheckFood=true`，Order 侧只过滤 `SWQ`。
- **现状**：`loadAllCoupon=true` 返回 XJQ+SWQ 是 CRM 老接口既有语义，本次未引入；Order 侧已过滤只返回 SWQ。
- **风险**：点餐页可能展示当前购物车/门店并不可用的券，用户预核销时才失败（体验问题，非正确性问题）。
- **建议**：v1 可接受（预核销阶段会再次校验并失败兜底）；后续如需收敛，可在列表查询时按购物车菜品做可用性预过滤。

### LOW-1：POS `dwd_coupon` 字段容量已确认，无需新增列

- `DwdCoupon.writeOffId` 标注 `@ColumnPlus(jdbcType = JdbcType.BLOB)`，可容纳 MP 的 `verify_results` JSON 数组；`couponNo` 同为 BLOB。POS 侧复用既有表结构，本次无需 POS DDL 变更，仅需在发布说明中标注确认目标库列类型。

### LOW-2：平台券预核销移除时未显式释放 prepare 状态

- **位置**：`validateAndRemoveExpiredPreWriteOff` 对 MP/DP 不做二次校验；`modDishNum` 移除券菜品时只删购物车行。
- **现状**：平台 `prepare` 通常有时效（5-10 分钟），过期后下单 `verify` 会失败提示重核。当前未调用平台释放/取消。
- **风险**：若平台 prepare 存在短时锁定语义，用户移除后立即重扫可能短暂冲突。实际平台多为 prepare 无锁、verify 才占用，风险低。
- **建议**：保持现状，文档标注；如平台确有 prepare 锁定，再补释放调用。

### LOW-3：行为测试缺失

- 当前缺少端到端行为测试（预核销写 Redis、重复幂等、多 certificate 拒绝、count>1、正式核销保存 verify、失败补偿、金额不双扣、POS 沉淀 DwdCoupon）。
- **建议**：在联调环境补关键路径行为测试，替代源码字符串断言型测试。

## 7. 补充评审（2026-06-01 第二轮，已修复）

### HIGH-A：`CloudPlatformCouponService.java` 未纳入版本控制（提交风险，已确认）

- **证据**：`git status` 返回 `?? nms4cloud-pos2plugin/.../service/admin/CloudPlatformCouponService.java`（未跟踪）。
- **依赖**：`DwdBillServicePlus` 第 755 行调用 `cloudPlatformCouponService.saveCloudPlatformCoupons(...)`，第 2111 行 `@Autowired` 注入。
- **影响**：若提交时漏掉该新文件，POS 端 `nms4cloud-pos2plugin-biz` 将因找不到类/Bean 编译失败。
- **处理**：代码逻辑正确，属提交纪律问题。**提交时必须 `git add` 该文件**。（需人工在提交环节确认，AI 未代为执行 git 操作。）

### HIGH-B：多数量平台券 faceAmount 口径未对齐 POS（count>1 金额少算，已修复）

- **根因**：
  - POS 本地扫码核销在 `DwdBillOpsServiceImpl:2480` 落库前把 `faceAmount` 改写为 `faceAmount × numbers`（总额）。
  - 下游 `GeneralCalcOrderServiceImpl:665` 用 `sum(DwdCoupon.faceAmount)` 当订单券抵扣总额（不再乘 numbers）；报表（DaySale/Shift/NoDiscount Provider）也直接累加 `face_amount`。
  - `OrderServiceUtil.getSumCoupons` 中 `sum(face_amount)` 直接累加，`sum(paid_amount*numbers)` 仅对 paidAmount 乘 numbers。
  - 原 `CloudPlatformCouponService` 只写单份 `platformPrice` 到 `faceAmount`，云端 `CartProductCouponService` 券菜品 `promotionAmount` 也只写单份。
- **后果**：当 `certificate.count > 1` 时，POS FJ 券支付金额、订单 `couponAmount`、券报表金额、以及云端订单头抵扣均**少算 (numbers-1) 份**。
- **修复**：
  - POS `CloudPlatformCouponService`：`faceAmount = platformPrice × numbers`（总额，与 POS 本地核销口径一致）；`paidAmount` 保持单份（由 `getSumCoupons` 的 `paid_amount*numbers` 自乘，不在写入处预乘）。新增/更新两个分支均已修正。
  - 云端 `CartProductCouponService`：平台券券菜品行 `promotionAmount(couponAmount) = platformPrice × quantity`，使订单头抵扣与 POS 总额口径一致。
- **验证**：两个文件 `getDiagnostics` 无错误；编译验证中。

### 结论更新

新增两项均已处理：HIGH-A 为提交纪律（必须 add 新文件），HIGH-B 为已修复的金额口径 BUG。建议在提交前确认 `CloudPlatformCouponService.java` 已被 git 跟踪，并补一条 count>1 的金额对齐行为测试。
