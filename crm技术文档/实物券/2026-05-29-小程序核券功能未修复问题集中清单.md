总体结论：当前仍应 `REQUEST CHANGES`。小程序商品券点餐核销主链路尚未闭合，平台券 POS 能力移植不完整，正式核销缺少可补偿闭环，订单金额、凭证持久化、DDL 和行为测试仍存在上线风险。

## 阻断问题

### P0-01 小程序点餐页未接入购物车预核销主链路

未修复依据：

- `taro-mall/src/common/service/order/cart.ts` 仍只有加菜、查购物车、改数量、清空、设置人数等购物车接口封装，没有 `/shopping_cart/get_available_product_coupon` 和 `/shopping_cart/pre_write_off_coupon`。
- `taro-mall/src/common/service/order/bill.ts` 暴露的是旧订单维度 `/order_bill/*product_coupon*` 接口，不是点餐购物车草稿预核销接口。
- 在 `taro-mall/src` 未检索到点餐页调用 `pre_write_off_coupon`、`preWriteOffCoupon`、`couponBusinessType` 的证据。

影响：用户无法在点餐页完成“扫码或选择券 -> 写入购物车草稿 -> 购物车展示预核销菜品 -> 现在下单正式核销”的需求路径。

### P0-02 外部平台券仍要求前端传 `foodId/unitId`

未修复依据：

- `CartPreWriteOffCouponRequest.java` 仍对 `foodId`、`unitId` 标记必填。
- `CartProductCouponService.java` 先用请求里的 `foodId/unitId` 查询菜品和单位，再处理会员券或平台券。
- POS 原链路是由后端根据平台 `sku_id/productName` 识别菜品，小程序当前没有完整移植该能力。

影响：抖音/美团扫码场景下，前端只有券码，不应承担识别菜品的职责。当前实现可能无法扫码核销，或因前端传错菜品导致错核券。

### P0-03 平台券凭证、渠道和 token 处理不完整

未修复依据：

- 平台券去重把原始券码塞进 `writeOffChannel`，正式核销时又把 `writeOffChannel` 当平台 `channel` 传出。
- 平台 prepare 后没有可靠沉淀 `channel`、`certificate_id`、`count` 等 POS 会保存的关键字段。
- `verifyToken` 缺失时存在随机生成或依赖前端传入的风险，正式核销不能证明只使用服务端可信凭证。
- 购物车 VO 曾被评审指出会把 `encryptedCode/verifyToken/platformPrice/writeOffChannel` 等核销凭证返回给前端，缺少后续修复证据。

影响：平台 verify/cancel 可能因为 channel 或 token 错误失败；敏感凭证暴露给客户端后存在篡改、重放和日志泄露风险。

### P0-04 平台正式核销结果没有解析和持久化

未修复依据：

- `CrtPostOrderServiceImpl.java` 调用 `douYinCouponFeign.verify(...)` 后只判断 `NmsResult.isSuccess()`。
- 没有解析 `verify_results` 中的 `verify_id/certificate_id/channel/count`。
- `OrderFood` 当前只落 `coupon_no` 和 `coupon_write_off_trace_no`。
- 平台取消接口需要 `verifyId/certificateId`，但下单链路没有保存这些值。

影响：订单取消、退款、退菜、反结账、异常补偿时，后端无法从本地订单恢复平台 cancel 所需参数。

### P0-05 外部正式核销在本地事务内执行，缺少补偿闭环

未修复依据：

- `crtOrder()` 带本地数据库事务，先保存订单和菜品，再逐张执行 CRM/平台正式核销。
- 任一券核销失败时只抛异常回滚本地事务。
- 没有记录已成功核销项，也没有失败后按成功列表执行反核销/取消的证据。

影响：第一张外部券核销成功、第二张失败，或外部核销成功后本地后续步骤异常时，会出现“本地订单回滚但外部券已核销”的账实不一致。

### P0-06 订单金额和商品券优惠口径仍不可靠

未修复依据：

- POS 移植分析指出订单金额存在疑似双扣：行级已扣商品券优惠，订单头又可能再次扣减商品券优惠。
- 早期深挖清单指出商品券菜品的普通折扣和券优惠混在同一个 `promotionAmount`，`coupon_amt` 可能把普通折扣也计入券金额。
- 平台券金额曾被指出可受请求 `platformPrice` 影响，而需求要求前端不传最终优惠金额。

影响：订单头、订单行、支付金额、CRM/POS 核算快照、退款分摊和报表口径可能不一致；极端情况下可能少收款或出现负应付。

### P0-07 多凭证、多数量平台券处理不完整

未修复依据：

- POS 会遍历 `certificates`，保存每个凭证和 `count`。
- 小程序预核销只取 `certificates[0]` 的证据仍存在。
- 下单正式核销平台券 `encryptedNums` 固定为 `List.of(1)`，没有按平台凭证数量或菜品数量传递。

影响：团购券包、多份券、套餐券场景下，可能只处理第一张凭证或只核销数量 1，导致本地菜品数量与平台核销数量不一致。

### P0-08 新增数据库字段缺少迁移 SQL

未修复依据：

- 复评检索 `coupon_write_off_trace_no`、`product_coupon_items`、`coupon_items`、`product_items`、`verify_id`、`certificate_id`、`write_off_channel`、`platform_price`、`verify_token`、`encrypted_code` 等 SQL 迁移脚本未找到匹配。
- `OrderFood` 和 `OrderBill` 已映射新增字段或快照字段。

影响：目标环境未人工加列时，保存订单或商品券快照可能出现 `Unknown column`；即使环境已手工加列，也缺少可审计、可回滚的发布脚本。

## 高风险问题

### P1-01 支付页仍展示并允许选择 `SWQ` 商品券

未修复依据：

- 支付提交侧已有过滤 `SWQ` 的逻辑。
- 但 `paycenter/index.tsx`、`paymentcenter/index.tsx` 仍在普通优惠券列表中渲染 `SWQ`，并允许点击进入选中态。

影响：用户会误以为支付页可使用商品券；实际提交时又不会作为 `COUPON` 付款项生效，容易造成金额和交互预期不一致。

### P1-02 移除预核销菜品只删除购物车行，未明确释放平台 prepare 状态

未修复依据：

- `modDishNum` 对预核销菜品数量小于等于 0 时只从购物车移除。
- 未看到移除时调用平台释放、取消 prepare 或记录废弃状态。

影响：如果平台 prepare 有短时锁定、占用或 token 有效期语义，用户移除后重扫、换桌或换用户可能出现平台侧状态不一致。

### P1-03 会员商品券预核销规则仍可能弱于 CRM/POS 老规则

未修复依据：

- 购物车预核销会员券金额计算被复评指出绕过 `CheckCouponUtil.check`，手写 `FREE/DISCOUNT/AMOUNT_OFF` 规则。
- 早期复评指出预核销阶段曾缺少单位校验、券状态、有效期、会员归属、门店/时段等完整校验。
- POS 会员商品券正式核销会传 `dishLid` 进行菜品绑定兜底校验，小程序正式核销路径没有同等边界。

影响：折扣/立减/免费券金额、单位、多门店、会员归属和历史兼容规则可能与 CRM/POS 老逻辑漂移。

### P1-04 商品券识别条件过宽，订单快照缺少业务类型

未修复依据：

- 商品券识别仍主要依赖 `couponNo` 或 `couponWriteOffTraceNo`。
- `OrderFood` 未持久化明确的商品券业务类型或来源字段。
- 商品券快照没有保存足够的平台凭证、业务类型、渠道和核销结果原文。

影响：历史或其他业务使用 `coupon_no` 的订单行可能被误归类为商品券；报表、审计、售后和反核销无法可靠定位券来源和平台记录。

### P1-05 新旧两套商品券核销接口并存，容易走错链路

未修复依据：

- `/shopping_cart/pre_write_off_coupon` 是最新需求需要的购物车草稿预核销接口。
- `/order_bill/write_off_product_coupon_order` 旧接口仍存在，并且早期评审指出该接口路径可能提前正式核销。
- 前端新增封装曾指向 `order_bill` 商品券接口，而不是购物车接口。

影响：后续集成或维护时可能在点餐阶段误用旧订单维度接口，造成未下单先核销、重复核销或绕过购物车预核销设计。

### P1-06 支付链路缺少后端兜底防二次抵扣证据

未修复依据：

- 早期问题指出支付校验可能接受客户端传入的 `COUPON` 付款项。
- 最新复评只确认前端提交侧过滤 `SWQ`，没有看到后端支付接口拒绝商品券或拒绝已在 `productCouponItems` 中出现券号的证据。

影响：恶意或异常客户端仍可能把商品券作为普通 `COUPON` 支付项提交，导致“券菜品优惠 + 支付券抵扣”双重扣减。

### P1-07 平台适配层可能把平台业务失败包装为 Feign 成功

未修复依据：

- 早期深挖清单指出微信/平台适配层在平台业务失败时可能 `return NmsResult.data(...)`。
- Order 侧多处只判断 `result.isSuccess()`。
- 最新清单仍指出正式核销只判断 `NmsResult.isSuccess()`，没有校验 `verify_results`。

影响：平台返回券不可用、核销失败、状态异常时，Order 侧可能仍继续生成券菜品或落单。

## 中低风险问题

### P2-01 可用商品券列表语义过宽

未修复依据：

- 可用券查询曾被指出使用 `loadAllCoupon=true`、`notNeedCheckFood=true`，只过滤 `SWQ`。
- 最新 POS 分析仍将“可用商品券查询过宽，未按购物车/菜品过滤”列为中风险。

影响：点餐页可能展示当前购物车、当前门店菜品或单位并不可用的商品券，用户预核销时才失败。

### P2-02 Controller 参数校验分组存在漏校验风险

未修复依据：

- 早期深挖清单指出新增接口使用 `@Validated(NeedTbl.class)`，但 DTO 的 `@NotBlank/@NotNull` 未声明同一分组。
- 后续复评没有明确该项已修复。

影响：必填字段可能不能在 Controller 层稳定拦截，错误会下沉到 Service、Feign 或外部平台调用阶段。

### P2-03 测试主要是源码字符串断言，缺少行为覆盖

未修复依据：

- `ProductCouponOrderEndpointContractTest` 多数通过 `Files.readString(...).contains(...)` 判断源码文本。
- 图谱变更检测显示 `nms4cloud` 和 `taro-mall` 仍存在大量测试缺口。

缺失的关键行为测试包括：

- 购物车预核销成功写入 Redis foodList。
- 重复预核销幂等。
- 移除预核销菜品后清理或释放状态。
- 平台 prepare 异常、多 certificate、count > 1。
- 下单正式核销成功后保存平台 verify 结果。
- 外部核销成功后本地失败的补偿。
- 商品券金额是否双扣。
- 支付页和后端支付接口不把商品券作为普通 `COUPON` 提交。

### P2-04 CRM 应用仍开启 MyBatis stdout SQL 日志

未修复依据：

- `nms4cloud-crm-app/src/main/resources/bootstrap.yml` 中仍有 `mybatis-plus.configuration.log-impl: org.apache.ibatis.logging.stdout.StdOutImpl` 的复评证据。

影响：生产或共享环境可能输出会员、券码、手机号、订单号等敏感 SQL 日志，增加性能、成本和数据泄露风险。

### P2-05 新增商品券 DTO/VO 缺少中文类注释和字段说明

未修复依据：

- 最新复评列出 `ProductCouponAvailableDTO`、`ProductCouponPrepareDTO`、`ProductCouponWriteOffDTO`、`ProductCouponCancelDTO`、`ProductCouponOrderVO`、`ProductCouponFoodVO`、`ProductCouponPrepareVO` 等类缺少中文类注释和字段说明。

影响：违反工作区 DTO/VO 中文注释规范，前后端联调和后续维护容易误解高风险字段来源、必填条件、单位和枚举 code。

## 建议修复顺序

1. 先闭合小程序主链路：点餐页接入 `/shopping_cart/get_available_product_coupon` 和 `/shopping_cart/pre_write_off_coupon`，支付页不允许选择 `SWQ` 商品券。
2. 后端统一链路：点餐阶段只购物车预核销，下单接口统一正式核销，旧 `order_bill` 商品券接口限定用途或显式隔离。
3. 平台券按 POS 方式移植：后端识别菜品，保存 prepare 凭证，支持多 certificate/count，正式核销解析并保存 `verify_results`。
4. 为外部核销增加幂等、状态记录和失败补偿，不能只依赖本地事务回滚。
5. 修正金额模型：区分普通折扣、商品券优惠、平台券面额和订单头金额，补行为测试证明不双扣。
6. 补齐数据库迁移 SQL、平台核销流水或订单扩展字段，并提供回滚脚本。
7. 补最小行为测试，再处理 SQL 日志、DTO/VO 注释和可用券列表体验收敛。
