# 小程序商品券后端接口实现说明

**日期**：2026-05-29  
**状态**：按最新源码校准。本文只保留小程序点餐当前应接入的购物车预核销主链路。

## 1. 接口总览

| 顺序 | 接口 | 方法 | 中文注释 |
|---|---|---|---|
| 1 | `/shopping_cart/get` | POST | 查询购物车；返回前会清理失效的会员商品券预核销菜品。 |
| 2 | `/shopping_cart/get_available_product_coupon` | POST | 查询当前会员可用商品券；只返回 CRM `SWQ` 商品券。 |
| 3 | `/shopping_cart/pre_write_off_coupon` | POST | 预核销商品券；校验券并把券菜品写入购物车 Redis 草稿。 |
| 4 | `/order_bill/crt_order` | POST | 创建订单；对购物车中 `preWriteOff=true` 的券菜品执行正式核销。 |

主入口：

- `ShoppingCartController#getAvailableProductCoupon`
- `ShoppingCartController#preWriteOffCoupon`
- `ShoppingCartServicePlus#preWriteOffCoupon`
- `CartProductCouponService#buildPreWriteOffFood`
- `CrtPostOrderServiceImpl#writeOffPreWriteOffCoupons`
- `CalcGenericOrderServiceImpl#applyProductCouponAccounting`

## 3. `/shopping_cart/get_available_product_coupon`

### 3.1 用途

点餐页打开“核销券/商品券”面板时调用，查询当前会员名下可用于点餐的系统商品券。

### 3.2 请求字段

请求 DTO：`CartAvailableCouponRequest extends CartBaseRequest`

| 字段 | 类型 | 必填 | 示例 | 中文注释 | 取值规则 | 兼容备注 |
|---|---|---|---|---|---|---|
| `phone` | `String` | 否 | `"13800000000"` | 会员手机号。 | 前端可传；不传时后端从登录用户上下文取手机号。 | 用于 CRM 查询该会员名下可用商品券。 |
| `mid` | `Long` | 是 | `10001` | 商户编号。 | 见“购物车请求通用字段”。 | 必须与当前登录上下文一致。 |
| `sid` | `Long` | 是 | `20001` | 门店编号。 | 见“购物车请求通用字段”。 | 必须与当前购物车门店一致。 |
| `tblId` | `String` | 是 | `"88001"` | 桌台编号。 | 见“购物车请求通用字段”。 | 用于定位购物车上下文。 |
| `openId` | `String` | 通常是 | `"o_xxx"` | 小程序用户 openId。 | 见“购物车请求通用字段”。 | 不直接传给 CRM 查询。 |
| `cardLid` | `Long` | 否 | `30001` | 会员卡逻辑编号。 | 后端转为字符串后作为 CRM 查询用 `cardId`。 | 与 `OrderBaseDTO.cardId` 不是同一字段类型。 |

### 3.3 后端处理

`ShoppingCartController` 先调用 `setCartInfo(request, CartOpModeEnum.NONE)` 补齐购物车上下文，然后转换为 `ProductCouponAvailableDTO`：

- `mid`、`sid` 直接取购物车上下文。
- `phone` 优先取请求值，不传则取登录用户手机号。
- `cardId` 由 `cardLid` 转字符串。

随后复用 `OrderCouponServicePlus#getAvailableProductCoupon`：

- 调 CRM `getAvailableCoupon`。
- `totalAmount=0`。
- `loadAllCoupon=true`，确保 CRM 返回包含 `SWQ` 的券列表。
- 不传 `foodList`，商品券绑定菜品由预核销接口校验。
- 后端只返回 `type == SWQ` 的券。

### 3.4 响应字段

响应为 CRM 券实例列表，元素映射为 `CrmCouponOrderVO`。小程序预核销会员商品券时至少需要使用以下字段：

|---|---|---|---|---|---|---|
| `lid` | `Long` | 是 | `70001` | 券领取记录逻辑编号。 | CRM 券实例返回。 | 前端预核销 WP 时传为 `couponNo`。 |
| `name` | `String` | 否 | `"单品兑换券"` | 微商城优惠券记录名称。 | CRM 券实例返回。 | 仅展示用。 |
| `coupon` | `String` | 否 | `"咖啡兑换券"` | 优惠券名称。 | CRM 券实例返回。 | 仅展示用。 |
| `couponCode` | `String` | 否 | `"CP001"` | 优惠券编号。 | CRM 券实例返回。 | 不等同于平台扫码 `couponCode`。 |
| `type` | `String` | 是 | `"SWQ"` | 券类型。 | 商品券接口只返回 `SWQ`。 | 非 `SWQ` 不应进入点餐预核销。 |
| `couponStatus` | `CouponStatusEnum` | 是 | `WHX` | 券状态。 | 预核销时要求未核销 `WHX`。 | 非 `WHX` 会被后端拒绝或清理。 |

## 4. `/shopping_cart/pre_write_off_coupon`

### 4.1 用途

点餐页用户选择会员商品券或扫码平台商品券后调用。该接口只做预核销：

- 校验券有效性。
- 计算券优惠金额。
- 构造一条券菜品行。
- 写入购物车 Redis 草稿。

该接口不调用 CRM `writeOffCouponInner`，也不调用平台 `verify`。

### 4.2 请求字段

请求 DTO：`CartPreWriteOffCouponRequest extends CartBaseRequest`

| 字段 | 类型 | 必填 | 示例 | 中文注释 | 取值规则 | 兼容备注 |
|---|---|---|---|---|---|---|
| `couponBusinessType` | `ProductCouponBusinessTypeEnum` | 是 | `WP` | 券业务类型。 | `WP` 会员商品券，`MP` 美团商品券，`DP` 抖音商品券。 | `WV/MV/DV` 现金券会被拒绝。 |
| `couponNo` | `Long` | WP 必填 | `70001` | 会员券记录编号。 | 从 `/shopping_cart/get_available_product_coupon` 返回的 `lid` 获取。 | 平台券不传。 |
| `couponCode` | `String` | MP/DP 必填 | `"812345678901"` | 平台券扫码原始券码。 | 用户扫码后传入。 | 会员券不传；平台券用它做幂等去重。 |
| `foodId` | `Long` | WP 必填；MP/DP 可不传 | `12345` | 券绑定菜品编号。 | 会员券从返回券的 `dishCode` 获取；平台券由后端根据平台 `prepare` 返回的 `sku_id/title/coupon_name` 匹配 `PtDish.extNames` 自动推导。 | 兼容旧调用：MP/DP 如果仍传 `foodId`，只能作为后端二次校验或兜底参考，不能覆盖平台 SKU 匹配结果。 |

| `quantity` | `Integer` | 否 | `1` | 核销数量。 | 不传或小于等于 0 时按 1 处理。 | 当前需求为一券一份。 |
| `encryptedCode` | `String` | 否 | `"enc_xxx"` | 平台券加密码。 | 通常不传，后端会在 prepare 阶段获取。 | 保留给特殊场景，当前主流程前端不需要传。 |
| `encryptedCodes` | `List<String>` | 否 | `["enc_xxx"]` | 平台券加密码列表。 | 当前主流程不使用。 | 预留给多凭证券包场景。 |
| `verifyToken` | `String` | 否 | `"token_xxx"` | 平台券验证 token。 | 通常不传，后端会从 prepare 返回中获取。 | 正式核销必须使用后端保存的 token。 |
| `platformPrice` | `BigDecimal` | 否 | `12.00` | 平台券价格。 | 后端忽略前端传值，只使用平台 prepare 返回价格。 | 前端不能用该字段控制优惠金额。 |
| `writeOffChannel` | `String` | WP 可选 | `"wechat-miniapp"` | 核销渠道。 | 会员券可传；平台券会被 prepare 返回的 channel 覆盖。 | 平台券不要依赖前端传值。 |
| `mid` | `Long` | 是 | `10001` | 商户编号。 | 见“购物车请求通用字段”。 | 必须与购物车一致。 |
| `sid` | `Long` | 是 | `20001` | 门店编号。 | 见“购物车请求通用字段”。 | 必须与购物车一致。 |
| `tblId` | `String` | 是 | `"88001"` | 桌台编号。 | 见“购物车请求通用字段”。 | 用于定位购物车 Redis 草稿。 |
| `openId` | `String` | 通常是 | `"o_xxx"` | 小程序用户 openId。 | 见“购物车请求通用字段”。 | 用于购物车 key 和后续下单上下文。 |

### 4.3 会员商品券 WP 处理

后端处理步骤：

1. 查询 `foodId` 对应菜品和 `unitId` 对应单位。
2. 调 CRM `mutGetCoupon` 查询 `couponNo` 对应券实例。
3. 校验券存在。
4. 校验 `type == SWQ`。
5. 校验 `couponStatus == WHX`。
6. 校验券绑定菜品 `dishCode` 与请求 `foodId` 一致。
7. 如果券绑定了 `dishUnit`，校验其与菜品单位名称一致。
8. 按券优惠类型计算优惠金额：
   - `FREE` 或空：优惠金额 = 菜品单价。
   - `DISCOUNT`：优惠金额 = 原价 × `(10 - dishDiscountValue) / 10`。
   - `AMOUNT_OFF`：优惠金额 = `dishDiscountValue`，最高不超过菜品价格。
9. 构造 `OrderFoodVO`，标记 `preWriteOff=true`。
10. 写入购物车。

会员券幂等去重规则：购物车中已有 `preWriteOff=true` 且 `couponNo` 相同的券菜品时，返回“该券已在购物车中，不能重复使用”。

### 4.4 平台商品券 MP/DP 处理

后端处理步骤：

1. 校验 `couponCode` 非空。
2. 按券类型选择平台：
   - `MP`：`ThirdPlatformTypeEnum.CAN_DAO`。
   - `DP`：`ThirdPlatformTypeEnum.DOU_YIN`。
3. 调微信服务团购券 `prepare`。
4. 解析平台返回：
   - `verify_token`
   - `certificates[0].encrypted_code`
   - `certificates[0].channel`
   - `certificates[0].sku.sku_id`
   - `certificates[0].sku.title`
   - `certificates[0].sku.coupon_name`
   - `certificates[0].sku.original_amount`
   - `certificates[0].sku.groupon_type`
5. 校验 `certificates` 非空。
6. 校验 `groupon_type != 2`。`groupon_type == 2` 表示现金券，不能在点餐页核销。
7. 调 `nms4cloud-product` 内部商品匹配能力，按 `mid/sid/sku_id/title/coupon_name` 解析 `PtDish.extNames` 并匹配本地菜品；匹配行为必须和 POS 端 `MtCouponHandler#getProduct`、`DyCouponHandler#getProduct` 一致。
8. 匹配成功后取 `foodId=PtDish.lid`，`unitId=-1`，再查询菜品和默认单位。
9. 如果前端兼容旧版本传了 `foodId/unitId`，后端只能用于一致性校验；与平台 SKU 自动匹配结果不一致时应拒绝，不能按前端值覆盖。
10. 优惠金额以平台返回 `original_amount` 为准。
11. 构造 `OrderFoodVO`，标记 `preWriteOff=true`。
12. 写入购物车。

平台 SKU 匹配规则：

| 优先级 | 匹配条件 | 中文注释 |
|---|---|---|
| 1 | `extNames[].id == sku_id` | 首选规则，表示平台 SKU 或外部商品 ID 与本地菜品绑定。 |
| 2 | `extNames[].name == sku_id` | 兼容部分历史数据把平台 SKU 写入名称字段。 |
| 3 | `extNames[].name == title/coupon_name` | 平台商品名或券商品名匹配。 |
| 4 | `PtDish.name == title/coupon_name` | 与 POS 一致，最后使用菜品名称匹配平台商品名。 |

候选查询和命中行为也要和 POS 一致：

1. 先按 `mid/sid` 查询候选菜品。
2. 候选条件为 `PtDish.extNames` 非空，且 `extNames like sku_id` 或 `extNames like title/coupon_name` 或 `PtDish.name == title/coupon_name`。
3. 遍历候选菜品，解析 `extNames` JSON 数组。
4. 对每个候选菜品按上表优先级判断，命中后返回首个匹配菜品。

未命中时返回“平台券商品未绑定本店菜品”。为了和 POS 保持一致，不额外增加“多命中则拒绝”的新规则；如果业务后续要收紧为唯一命中，必须同时评估 POS 端行为是否同步调整。

`takeout_food_map` 不作为主路径。该表是外卖商品映射，商户可能没有数据，且当前查询缺少 `channelType` 维度，跨平台 SKU 相同时有串映射风险。小程序平台商品券扫码核销应使用 POS 已验证的 `PtDish.extNames` 逻辑。

平台券幂等去重规则：购物车中已有 `preWriteOff=true`、券类型不是 `WP`，且 `originalCouponCode` 相同的券菜品时，返回“该券已在购物车中，不能重复使用”。

## 5. `/shopping_cart/get`

购物车查询接口不是商品券专用接口，但商品券预核销依赖它做状态维护。

返回购物车前，`ShoppingCartServicePlus` 会调用 `CartProductCouponService#validateAndRemoveExpiredPreWriteOff`：

- 只检查 `WP` 会员商品券。
- 如果会员券不再是未核销状态，则从购物车中移除对应预核销券菜品。
- 移除后写回 Redis。
- `MP/DP` 平台券不做二次校验；prepare 凭证过期由下单正式 `verify` 报错。

`/shopping_cart/get` 响应仍为 `ShoppingCartVO`，字段中文注释见 4.5。

## 6. `/order_bill/crt_order`

### 6.1 用途

创建订单。对商品券来说，它是正式核销边界。

### 6.2 正式核销时机

`CrtPostOrderServiceImpl` 在保存订单、菜品、做法后调用 `writeOffPreWriteOffCoupons`：

1. 从购物车 `foodList` 中筛选 `preWriteOff=true` 的菜品。
2. 按 `preWriteOffBusinessType` 分支正式核销。
3. 任一券核销失败，抛出 `BizException`，提示“商品券核销失败[菜品名]: 失败原因”。

## 7. 订单核算字段

`CalcGenericOrderServiceImpl#applyProductCouponAccounting` 根据最终订单菜品生成商品券核算快照。

识别券菜品规则：

- `couponNo` 非空；或

| 字段 | 类型 | 必填 | 示例 | 中文注释 | 取值规则 | 兼容备注 |

| `coupon_amt` | `BigDecimal` | 是 | `18.00` | 核算明细中的券优惠金额。 | 行级 `promotionAmount` 扣除普通折扣和做法折扣后得到。 | 小于 0 时按 0 处理。 |
| `food_lid` | `Long` | 是 | `91001` | 订单菜品行编号。 | 取最终订单菜品 `lid`。 | 用于定位商品券对应菜品。 |
| `food_no` | `Long` | 是 | `12345` | 菜品编号。 | 取最终订单菜品 `foodNo`。 | 与商品服务菜品编号一致。 |

| `food_number` | `BigDecimal` | 是 | `1` | 菜品数量。 | 取最终订单菜品 `foodNumber`。 | 当前商品券通常为 1。 |
| `food_amt` | `BigDecimal` | 是 | `18.00` | 菜品金额。 | 取最终订单菜品 `foodAmount`。 | 与订单行金额保持一致。 |

金额处理：

- 券优惠金额 = 行级 `promotionAmount` - 普通折扣 - 做法折扣。
- 商品券优惠总额追加到订单头 `promotionAmount`。
- 商品券优惠总额从订单头 `paidAmount` 扣减。

## 8. 异常边界

| 场景 | 中文注释 |
|---|---|
| 现金券调用预核销 | 拒绝，提示现金券不能在点餐页核销。 |
| 会员券不存在 | 拒绝，提示优惠券不存在。 |
| 会员券不是 `SWQ` | 拒绝，提示只有商品券可以在点餐页核销。 |
| 会员券状态不是 `WHX` | 拒绝，提示该券已被使用或已失效。 |
| 会员券绑定菜品不匹配 | 拒绝，提示券与菜品不匹配。 |
| 会员券绑定单位不匹配 | 拒绝，提示券与菜品单位不匹配。 |
| 同一会员券重复加入购物车 | 拒绝，提示该券已在购物车中，不能重复使用。 |
| 平台券码为空 | 拒绝，提示平台券码不能为空。 |
| 平台 prepare 无有效凭证 | 拒绝，提示平台券无有效凭证。 |
| 平台返回 `groupon_type == 2` | 拒绝，提示现金券不能在点餐页核销，请在支付页使用。 |
| 平台券无 `sku_id` 且无可匹配商品名 | 拒绝，提示平台券未返回可匹配商品信息。 |

| 同一平台券码重复加入购物车 | 拒绝，提示该券已在购物车中，不能重复使用。 |
| 下单时平台券缺少 `verifyToken` | 拒绝，提示平台券预核销已失效（缺少 verifyToken），请重新核销券。 |
| 下单时平台券缺少 `encryptedCode` | 拒绝，提示平台券预核销已失效（缺少 encryptedCode），请重新核销券。 |
| 下单正式核销失败 | 下单失败，提示商品券核销失败和具体菜品名。 |

## 9. 前端接入要点

- 会员商品券列表走 `/shopping_cart/get_available_product_coupon`。
- 会员商品券预核销传 `couponBusinessType=WP`、`couponNo`、`foodId`、`unitId`。
- 平台商品券预核销传 `couponBusinessType=MP/DP`、`couponCode`；前端不再要求用户选菜，也不必传 `foodId/unitId`。
- 平台商品券的菜品匹配由后端通过平台 `prepare` 返回的 SKU 信息和 `PtDish.extNames` 完成。
- 前端不传最终优惠金额；平台券价格以后端 prepare 返回为准。
- 预核销成功后以前端收到的 `ShoppingCartVO` 刷新购物车。
- 删除预核销商品券按购物车菜品删除/重算处理，不调用反核销接口。
- 支付页只处理现金券；`SWQ` 商品券不能作为 `COUPON` 付款项提交。
