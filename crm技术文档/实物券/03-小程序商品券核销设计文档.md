# 小程序商品券核销设计文档

**日期**：2026-05-29  
**范围**：`taro-mall` 微信小程序 / `nms4cloud-order` 订单服务 / CRM `SWQ` 商品券 / POS 商品券核销口径

## 1. 结论

小程序商品券按“用券点菜”设计，不按“支付页现金券抵扣”设计。

最新源码中的主流程不是点餐页直接正式核销，而是：

1. 点餐页查询可用商品券。
2. 点餐页预核销，把券菜品行写入购物车 Redis 草稿。
3. 用户下单时由 `/order_bill/crt_order` 统一正式核销，并生成订单菜品、订单券明细和金额快照。

现金券仍在支付页作为付款方式处理，不生成菜品行。商品券不能作为支付页 `COUPON` 付款项二次扣减。

## 2. 券类型边界

| 类型 | 含义 | 小程序处理 |
|---|---|---|
| `WP` | 会员商品券 | 点餐页预核销，生成购物车券菜品行；下单时正式核销 |
| `MP` | 美团商品券 | 点餐页预核销，生成购物车券菜品行；下单时正式核销 |
| `DP` | 抖音商品券 | 点餐页预核销，生成购物车券菜品行；下单时正式核销 |
| `WV` | 会员现金券 | 支付页付款方式模型 |
| `MV` | 美团现金券 | 支付页付款方式模型 |
| `DV` | 抖音现金券 | 支付页付款方式模型 |

购物车预核销和订单商品券接口都只接受 `WP/MP/DP` 作为点餐商品券类型。如果识别为 `WV/MV/DV`，不能生成券菜品行。

## 3. POS 对齐规则

小程序不直接调用 POS `/addCoupons` 或 POS 插件接口，但业务规则对齐 POS：

- 券码以 `http` 开头，或长度 `>=15 && <19`，默认按抖音券识别。
- 券码长度 `>9 && <15`，默认按美团券识别。
- 其他券码默认按会员券识别。
- 平台返回 `groupon_type == 2` 时识别为现金券，否则识别为商品券。
- 商品券生成菜品行；现金券作为付款方式。
- 商品券点餐页只做预核销；正式核销发生在下单流程。
- 支付页不得再次把商品券作为 `COUPON` 付款项提交。

## 4. 后端接口

### 4.1 小程序点餐主流程

接口落在 `nms4cloud-order` 的 `ShoppingCartController`，使用小程序登录态和购物车上下文。

| 接口 | 用途 |
|---|---|
| `POST /shopping_cart/get_available_product_coupon` | 查询当前购物车上下文下会员可用商品券，过滤 CRM `SWQ` |
| `POST /shopping_cart/pre_write_off_coupon` | 校验商品券，构造预核销券菜品行并写入购物车 Redis |
| `POST /order_bill/crt_order` | 下单时遍历 `preWriteOff=true` 的菜品，正式核销会员券或平台券 |

`/shopping_cart/pre_write_off_coupon` 不调用 CRM `writeOffCouponInner`，也不调用平台 `verify`。

### 4.2 保留的订单商品券接口

`OrderBillController` 仍保留以下接口，但当前前端页面未检索到实际调用点，不能再作为小程序点餐主链路描述：

| 接口 | 用途 |
|---|---|
| `POST /order_bill/get_available_product_coupon` | 查询会员可用商品券，过滤 CRM `SWQ` |
| `POST /order_bill/prepare_product_coupon` | 根据券码识别会员 / 美团 / 抖音，以及商品券 / 现金券 |
| `POST /order_bill/write_off_product_coupon_order` | 返回券菜品行和核销结果；兼容旧订单商品券流程 |
| `POST /order_bill/cancel_product_coupon_write_off` | 取消旧订单商品券流程产生的核销记录 |

## 5. 接口调用关系

### 5.1 小程序点餐主链路

| 步骤 | 小程序动作 | 调用接口 | 后端处理 | 下一步依赖 |
|---|---|---|---|---|
| 1 | 打开点餐页或刷新购物车 | `POST /shopping_cart/get` | 返回购物车；后端会检查已预核销会员券是否仍有效，失效则移除并写回 Redis | 展示当前菜品和已预核销商品券 |
| 2 | 打开商品券选择面板 | `POST /shopping_cart/get_available_product_coupon` | 根据购物车上下文补齐 `mid/sid/phone/cardId`，复用 CRM 可用券查询并过滤 `SWQ` | 前端展示会员商品券列表 |
| 3 | 用户选择会员商品券 | `POST /shopping_cart/pre_write_off_coupon` | 校验 `WP` 券状态、`SWQ` 类型、绑定菜品和单位，生成 `preWriteOff=true` 的券菜品行 | 返回最新购物车，前端刷新购物车 |
| 4 | 用户扫码美团/抖音商品券 | `POST /shopping_cart/pre_write_off_coupon` | 校验 `MP/DP`，调用平台 `prepare` 获取 `sku_id/title/coupon_name/encryptedCode/verifyToken/channel/price`，再按 `PtDish.extNames` 自动匹配本店菜品，生成预核销券菜品行 | 返回最新购物车，前端刷新购物车 |
| 5 | 用户取消预核销券 | 购物车删除/重算接口 | 预核销阶段未正式核销，只删除购物车券菜品行，不调用 CRM 反核销或平台 `cancel` | 返回最新购物车 |
| 6 | 用户点击下单 | `POST /order_bill/crt_order` | 保存订单和菜品后，遍历 `preWriteOff=true` 菜品正式核销；随后清空购物车 | 下单成功进入订单/支付流程 |
| 7 | 进入支付页 | 支付页现有接口 | 支付页只处理现金券；`SWQ` 商品券不进入 `orderPayList` 的 `COUPON` 付款项 | 避免商品券二次抵扣 |

关键关系：

- `/shopping_cart/get_available_product_coupon` 的输出用于会员券 `WP` 预核销，前端从券列表中取得 `couponNo/foodId/unitId`。
- 平台券 `MP/DP` 的主流程不再要求前端选菜，前端扫码后只传 `couponBusinessType + couponCode`；`foodId/unitId` 由后端根据平台 `prepare` 结果和 `PtDish.extNames` 推导。
- `/shopping_cart/pre_write_off_coupon` 的输出是完整购物车，其中新增的券菜品行带有 `preWriteOff=true`。这个字段是 `/order_bill/crt_order` 正式核销的识别条件。
- 平台券 `MP/DP` 的 `encryptedCode/verifyToken/channel/platformPrice` 由后端在预核销阶段从平台 `prepare` 返回值中取得，正式核销时由后端继续使用，前端不应计算或覆盖最终优惠金额。
- `/order_bill/crt_order` 是正式核销边界。预核销成功只代表券菜品已写入购物车，不代表 CRM 或平台已经核销成功。

### 5.2 平台券自动匹配关系

平台商品券扫码自动预核销对齐 POS 已验证的 `extNames` 绑定逻辑，不以 `takeout_food_map` 作为主路径。

| 关系 | 说明 |
|---|---|
| 券码到平台券 | 小程序扫码只得到 `couponCode`，由 `/shopping_cart/pre_write_off_coupon` 调平台 `prepare` 解析券信息。 |
| 平台券到平台 SKU | 后端从 `prepare` 返回中读取 `certificates[0].sku.sku_id`，并同时保留 `title/coupon_name` 作为辅助匹配信息。 |
| 平台 SKU 到本地菜品 | 后端调用 `nms4cloud-product` 内部能力，由商品服务解析 `PtDish.extNames` 并匹配本地 `foodId=PtDish.lid`。 |
| 本地菜品到单位 | 平台券当前按 POS 口径使用默认单位，`unitId=-1`；后续如果平台 SKU 能绑定具体规格，再扩展映射结构。 |
| 菜品行到正式核销 | 预核销菜品行保存平台 `encryptedCode/verifyToken/channel/platformPrice/originalCouponCode`，下单时 `/order_bill/crt_order` 使用这些字段调用平台 `verify`。 |

`extNames` 自动匹配必须和 POS 端 `MtCouponHandler#getProduct`、`DyCouponHandler#getProduct` 保持一致：

1. `extNames[].id == sku_id`。
2. `extNames[].name == sku_id`。
3. `extNames[].name == title/coupon_name`。
4. `PtDish.name == title/coupon_name`。

POS 当前逻辑会先按 `mid/sid` 查询 `extNames` 包含 `sku_id` 或平台商品名、或 `PtDish.name` 等于平台商品名的候选菜品，再按上述顺序遍历候选列表，找到第一个匹配菜品后返回。小程序后端实现要复刻这个行为，不能改成另一套更严格或更宽松的规则，否则同一张平台商品券在 POS 和小程序可能命中不同菜品。

`takeout_food_map` 不作为主链路的原因：

- 该表是外卖商品映射，不是平台商品券绑定主数据。
- 商户可能只维护 POS 菜品 `extNames`，没有外卖映射数据。
- POS 美团/抖音商品券核销本身使用 `extNames`，没有依赖 `takeout_food_map`。
- 现有 `takeout_food_map` 查询按 `sid + foodIdInChannel`，未体现 `channelType`，跨平台 SKU 相同时有串映射风险。

### 5.3 旧订单商品券接口关系

`/order_bill/*product_coupon*` 接口仍存在，但和购物车主链路不要混用：

| 接口 | 与主链路关系 |
|---|---|
| `/order_bill/get_available_product_coupon` | 功能上和购物车可用券查询相近，但使用 `OrderBillController.setOrderInfo()`，不是购物车上下文主入口 |
| `/order_bill/prepare_product_coupon` | 可识别券码和平台券现金/商品类型；购物车主链路中平台券 prepare 已内聚到 `/shopping_cart/pre_write_off_coupon` |
| `/order_bill/write_off_product_coupon_order` | 旧流程会直接调用会员券核销或平台 `verify` 并返回券菜品行；不写购物车草稿 |
| `/order_bill/cancel_product_coupon_write_off` | 只用于取消旧流程已经正式核销的券；购物车预核销取消不应调用它 |

因此，小程序点餐新接入应优先走购物车接口；只有维护旧页面或兼容旧调用方时，才使用 `order_bill` 商品券接口。

## 6. 预核销、正式核销和落表规则

### 6.1 预核销写购物车

预核销阶段由 `CartProductCouponService` 构建 `OrderFoodVO`，主要沉淀字段：

| 位置 | 用途 |
|---|---|
| `OrderFoodVO.preWriteOff` | 标记该菜品是商品券预核销写入，尚未正式核销 |
| `OrderFoodVO.preWriteOffBusinessType` | 记录 `WP/MP/DP`，下单正式核销时分支使用 |
| `OrderFoodVO.couponNo` | 会员商品券编号 |
| `OrderFoodVO.couponWriteOffTraceNo` | 预核销菜品追踪号 |
| `OrderFoodVO.promotionAmount` | 券优惠金额 |
| `OrderFoodVO.encryptedCode` | 平台券加密码 |
| `OrderFoodVO.verifyToken` | 平台券验证 token |
| `OrderFoodVO.platformPrice` | 平台券价格 |
| `OrderFoodVO.writeOffChannel` | 核销渠道，平台券以 prepare 返回值为准 |
| `OrderFoodVO.originalCouponCode` | 平台券原始券码，用于购物车去重 |

会员券预核销会校验券存在、类型为 `SWQ`、状态为 `WHX`、绑定菜品和单位匹配。平台券预核销会调用平台 `prepare` 获取正式核销凭证，并拒绝 `groupon_type == 2` 的现金券；随后根据平台 SKU 走 `PtDish.extNames` 自动匹配本地菜品，匹配不到时不写购物车。匹配顺序和首个命中行为必须与 POS 端一致。

### 6.2 下单正式核销

`CrtPostOrderServiceImpl` 在保存订单和菜品后，遍历购物车 `foodList` 中 `preWriteOff=true` 的菜品：

- `WP`：调用 CRM `writeOffCouponInner`，`orderId` 使用真实订单 `saasOrderKey`。
- `MP/DP`：调用微信服务团购券 `verify`，使用预核销保存的 `verifyToken/encryptedCode/channel/platformPrice`。
- 正式核销失败时抛出业务异常，订单流程失败，提示用户重新处理商品券。

### 6.3 订单核算快照

最终订单需要沉淀以下数据：

| 位置 | 用途 |
|---|---|
| `OrderFood.coupon_no` | 会员商品券与券菜品行关系 |
| `OrderFood.coupon_write_off_trace_no` | 平台券或无 `couponNo` 场景的稳定定位键 |
| `OrderFood.promotion_amount` | 券菜品行展示优惠金额 |
| `OrderBill.coupon_items` | 券维度核算明细 |
| `OrderBill.product_items` | 商品维度核算明细 |
| `OrderBill.product_coupon_items` | 商品券与商品关系明细 |

金额规则：

- 免费商品券优惠金额等于菜品价格。
- 折扣券按折扣值计算优惠金额。
- 立减券按立减值计算优惠金额，最多减到菜品价格。
- 平台券优惠金额以平台 `prepare` 返回价格为准。
- 券菜品保留菜品原价，真实优惠额单独落 `promotionAmount`。
- 订单头 `promotionAmount` 增加商品券优惠额，`paidAmount` 扣减商品券优惠额。
- 商品券不能作为支付方式二次扣减，否则会导致订单头、订单行和付款方式不平。

## 7. 小程序处理

小程序职责：

- 展示商品券入口、券列表和预核销结果。
- 调用购物车商品券主流程接口。
- 根据后端返回刷新购物车、菜品行和商品券展示列表。
- 支付页展示已进入订单菜品的商品券结果。
- 扫码平台商品券后直接提交券码，不要求用户再手动选择菜品。

小程序不得处理：

- 不计算商品券优惠金额。
- 不自行调整菜品价格。
- 不直接调用普通加菜接口生成券菜品。
- 不解析平台 SKU，也不在前端维护 `extNames` 匹配规则。
- 不解析 `coupon_items/product_items/product_coupon_items` 做核算。
- 不把商品券放入 `orderPayList` 的普通 `COUPON` 付款项。

支付页只把非 `SWQ` 券转换为 `COUPON` 付款项；`SWQ` 商品券只作为点餐/订单结果展示。

## 8. 取消和异常

预核销阶段尚未正式核销，取消选择商品券应删除购物车中的预核销券菜品并触发购物车重算，不应调用 CRM 反核销或平台 `cancel`。

购物车返回前会校验会员商品券是否仍为未核销状态；如果失效，后端会移除对应预核销券菜品并写回 Redis。平台券不做二次校验，`prepare` 凭证过期时由下单 `verify` 失败提示用户重新核销。

已正式核销后的整单取消、退菜、退款、反结账等动作仍由后端订单流程联动处理，不能让小程序绕过后端直接改券状态或订单行。

保留的 `/order_bill/cancel_product_coupon_write_off` 只适用于旧订单商品券接口产生的核销记录：

- 会员券调用 CRM 精确反核销。
- 平台券调用微信服务 `cancel`，需要 `verifyId`、`certificateId` 等取消参数。
