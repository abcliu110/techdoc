# 小程序点餐页单品券预核销需求文档

日期：2026-05-28

适用范围：`taro-mall` 微信小程序点餐页、购物车页、后端购物车/下单接口

## 1. 背景

当前小程序扫码点餐流程中，用户在点餐页操作的是购物车草稿。正式订单 ID 不是在点餐页生成，而是在购物车点击“现在下单”后，由后端创建订单并返回。

客户要求单品券需要在点餐页就可以使用，但点餐页没有正式订单 ID，因此本需求将点餐阶段的券使用定义为“预核销”，正式核销由后端在下单接口中统一处理。

## 2. 需求目标

在微信小程序点餐页新增“核销券”入口，支持用户在点餐阶段使用单品券。

用户在点餐页完成的操作是预核销：

- 前端调用后端预核销接口。
- 后端校验券并把对应券菜品写入购物车草稿。
- 购物车展示该预核销菜品。
- 用户移除该菜品时，后端自动取消预核销。
- 用户点击“现在下单”时，后端在现有下单接口中完成真正核销。

## 3. 名词定义

| 名词 | 说明 |
|------|------|
| 单品券 | 一张券只对应一个指定菜品的券。 |
| 实物券 | 系统中的 `SWQ` 类型券，本需求中的系统单品券属于实物券。 |
| 预核销 | 点餐阶段调用后端接口，把券和对应菜品写入购物车草稿，但不完成最终订单核销。 |
| 正式核销 | 用户点击“现在下单”后，后端基于正式订单和购物车草稿完成券核销。 |
| 预核销菜品 | 由预核销券自动写入购物车的菜品行。 |
| 购物车草稿 | 当前扫码点餐上下文中的购物车状态，由 `/api/sorder/shopping_cart/get` 返回。 |

## 4. 已确认业务规则

1. 点餐页需要新增“核销券”入口。
2. 点餐页做的是预核销，不做最终核销。
3. 点餐页预核销必须调用后端预核销接口。
4. 一个单品券只对应一个菜品，不存在前端选择多个菜品的场景。
5. 单品券对应系统发放的实物券，可支持免费、折扣、立减等规则。
6. 系统内发放的实物券可以通过后端接口获取可用券列表。
7. 抖音券、美团券等外部券需要用户扫码获取券码。
8. 预核销成功后，后端需要将对应券菜品写入购物车草稿。
9. 购物车中不要求展示完整优惠后的金额，避免改动过大。
10. 用户移除购物车菜品时，如果该菜品是预核销菜品，后端在移除菜品逻辑中自动取消预核销。
11. 取消预核销不需要前端额外调用取消接口。
12. 点击购物车“现在下单”时，后端在现有下单接口中完成真正核销。
13. 前端不需要在下单时额外调用正式核销接口。
14. 前端不自行计算最终优惠金额，金额以后端返回为准。

## 5. 用户流程

### 5.1 系统实物券预核销

1. 用户进入点餐页。
2. 用户点击“核销券”入口。
3. 前端打开核销券面板。
4. 前端请求后端获取当前用户可用单品券列表。
5. 用户选择一张单品券。
6. 前端调用后端预核销接口。
7. 后端校验券有效性及绑定菜品。
8. 后端将预核销券及对应菜品写入购物车草稿。
9. 前端刷新购物车数据。
10. 点餐页/购物车展示预核销菜品。

### 5.2 抖音券/美团券预核销

1. 用户进入点餐页。
2. 用户点击“核销券”入口。
3. 用户在核销券面板中扫码。
4. 前端获取扫码结果中的券码。
5. 前端调用后端预核销接口，并传入券码及券来源。
6. 后端校验外部券是否有效，并识别对应菜品。
7. 后端将预核销券及对应菜品写入购物车草稿。
8. 前端刷新购物车数据。
9. 点餐页/购物车展示预核销菜品。

### 5.3 取消预核销

1. 用户在购物车中移除菜品。
2. 前端继续调用现有购物车改数量/移除接口。
3. 如果被移除菜品是预核销菜品，后端自动取消该菜品对应的预核销状态。
4. 后端返回最新购物车状态。
5. 前端刷新展示。

取消预核销复用现有接口：

```text
POST /api/sorder/shopping_cart/modDishNum
```

### 5.4 正式核销

1. 用户进入购物车。
2. 用户点击“现在下单”。
3. 前端继续调用现有下单接口。
4. 后端创建正式订单。
5. 后端读取购物车草稿中的预核销券。
6. 后端完成真正核销、落单、金额计算。
7. 后端返回下单结果。

前端不额外调用正式核销接口。

## 6. 页面要求

### 6.1 点餐页

点餐页新增“核销券”入口。

入口点击后打开核销券面板。面板建议包含：

- 扫码入口：用于抖音券、美团券等外部券。
- 系统单品券列表：用于展示用户当前可用的系统实物券。
- 已预核销券/菜品展示：用于查看当前购物车草稿中已经使用的券。

### 6.2 购物车页

购物车需要展示预核销菜品。

购物车不要求展示完整优惠后的金额。首版可以只展示：

- 菜品名称
- 数量
- 原菜品价格或后端当前返回价格
- “单品券”或“预核销”标识

如果后端已经在购物车返回中提供优惠说明，前端可以展示；如果没有，首版不强制展示优惠后金额。

### 6.3 支付页

支付页不需要新增单品券正式核销逻辑。

正式核销已经在购物车点击“现在下单”时由后端处理。支付页只展示后端返回的订单结果。

## 7. 后端接口需求

### 7.1 获取可用单品券列表

用途：点餐页核销券面板展示系统内发放的可用实物券。

接口路径待后端确认。

建议返回内容：

| 字段 | 说明 |
|------|------|
| `couponNo` | 券号或领券记录 ID。 |
| `couponName` | 券名称。 |
| `couponType` | 券类型，系统实物券为 `SWQ`。 |
| `foodId` | 绑定菜品 ID。 |
| `unitId` | 绑定单位 ID。 |
| `foodName` | 绑定菜品名称。 |
| `discountType` | 免费、折扣、立减等优惠类型。 |
| `discountValue` | 折扣值或立减金额。 |
| `available` | 是否可用。 |
| `unavailableReason` | 不可用原因。 |

### 7.2 预核销接口

用途：点餐页选择系统券或扫码外部券后，调用后端进行预核销。

接口路径待后端确认。

建议请求字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `sid` | 是 | 门店 ID。 |
| `tblId` | 是 | 当前点餐上下文，沿用购物车接口。 |
| `couponNo` | 是 | 券号。系统券为领券记录 ID，外部券为扫码券码。 |
| `couponSource` | 是 | 券来源，如 `SYSTEM`、`DOUYIN`、`MEITUAN`。 |

说明：

- 前端不传最终优惠金额。
- 前端不传正式订单 ID。
- 一个券只对应一个菜品，菜品识别由后端完成。
- 预核销成功后，后端需要返回最新购物车状态，或前端随后重新调用 `/shopping_cart/get` 获取最新状态。

### 7.3 获取购物车

现有接口：

```text
POST /api/sorder/shopping_cart/get
```

新增要求：

- 返回预核销菜品。
- 返回预核销菜品标识，便于前端展示。
- 后端在返回购物车前，需要判断购物车草稿中的预核销券是否仍有效。
- 如果发现预核销券失效，后端应自动移除失效券及对应预核销菜品。

可能的失效场景：

- 券已过期。
- 券已被收回。
- 券已被其他渠道使用。
- 券不再适用于当前门店。
- 券对应菜品不可用。

### 7.4 移除购物车菜品

现有接口：

```text
POST /api/sorder/shopping_cart/modDishNum
```

新增要求：

- 如果被移除菜品是预核销菜品，后端自动取消该菜品对应的预核销。
- 前端不单独调用取消预核销接口。
- 后端返回最新购物车状态或前端随后重新调用 `/shopping_cart/get`。

### 7.5 现在下单

现有下单接口：

```text
POST /api/sorder/order_bill/crt_order
```

新增要求：

- 后端在该接口中读取购物车草稿里的预核销券。
- 后端创建正式订单后完成真正核销。
- 后端完成券菜品落单和金额计算。
- 如果正式核销失败，后端返回明确失败原因。
- 前端不额外调用正式核销接口。

## 8. 前端职责

1. 点餐页新增“核销券”入口。
2. 打开核销券面板。
3. 查询系统可用单品券列表。
4. 支持扫码获取抖音券、美团券券码。
5. 调用后端预核销接口。
6. 预核销成功后刷新购物车数据。
7. 展示预核销菜品及必要标识。
8. 购物车移除菜品时继续调用现有 `modDishNum`。
9. 点击“现在下单”时继续调用现有 `crt_order`。
10. 不自行计算最终优惠金额。
11. 不自行调用正式核销接口。
12. 不自行调用取消预核销接口。

## 9. 后端职责

1. 提供系统单品券查询能力。
2. 提供预核销接口。
3. 预核销时校验券有效性。
4. 预核销时识别券对应菜品。
5. 将预核销券和对应菜品写入购物车草稿。
6. 在 `/shopping_cart/get` 中清理失效预核销券。
7. 在 `/shopping_cart/modDishNum` 中处理预核销取消。
8. 在 `/order_bill/crt_order` 中完成真正核销。
9. 返回前端可识别的预核销菜品标识。
10. 返回明确错误原因，便于前端提示用户。

## 10. 金额展示规则

首版购物车不要求展示完整优惠后的金额。

推荐展示原则：

- 购物车展示预核销菜品。
- 预核销菜品带“单品券”或“预核销”标识。
- 不强制展示折扣后单价。
- 不要求前端重算购物车总金额。
- 如果后端返回优惠说明，前端可展示。
- 最终金额以后端正式下单后的返回结果为准。

## 11. 异常处理

### 11.1 预核销失败

后端返回失败原因，前端提示用户。

常见原因：

- 券不存在。
- 券已过期。
- 券已被使用。
- 券已被收回。
- 券不适用于当前门店。
- 券绑定菜品不可用。

### 11.2 购物车刷新时券失效

后端在 `/shopping_cart/get` 中自动移除失效券及对应菜品。

前端展示后端返回的最新购物车状态。

如后端返回失效提示，前端可提示用户“部分预核销券已失效，已从购物车移除”。

### 11.3 移除预核销菜品

前端调用现有 `modDishNum`。

后端识别该菜品为预核销菜品后，自动取消预核销。

### 11.4 下单时正式核销失败

后端返回明确失败原因。

前端提示用户，并刷新购物车。

## 12. 验收标准

1. 点餐页可以看到“核销券”入口。
2. 用户可以打开核销券面板。
3. 用户可以查看系统内可用单品券列表。
4. 用户可以通过扫码录入抖音券、美团券券码。
5. 用户选择或扫码后，前端调用后端预核销接口。
6. 预核销成功后，购物车出现对应预核销菜品。
7. 购物车中预核销菜品有可识别标识。
8. 购物车不要求展示完整优惠后的金额。
9. 用户移除预核销菜品时，前端调用现有 `modDishNum`。
10. 后端在移除菜品时取消对应预核销。
11. `/shopping_cart/get` 会清理失效预核销券。
12. 点击“现在下单”时，前端仍调用现有下单接口。
13. 后端在下单接口中完成真正核销。
14. 前端不额外调用正式核销接口。
15. 前端不额外调用取消预核销接口。

## 13. 待后端确认项

1. 系统单品券列表接口路径、请求字段、响应字段。
2. 预核销接口路径、请求字段、响应字段。
3. `couponSource` 枚举是否使用 `SYSTEM`、`DOUYIN`、`MEITUAN`。
4. 外部券扫码结果是否统一直接作为 `couponNo`。
5. 购物车中预核销菜品的标识字段名称。
6. 购物车中预核销菜品是否需要返回原券信息。

8. `/shopping_cart/modDishNum` 移除预核销菜品后，是否直接返回最新购物车状态。
9. `/order_bill/crt_order` 正式核销失败时的错误码和错误信息格式。
