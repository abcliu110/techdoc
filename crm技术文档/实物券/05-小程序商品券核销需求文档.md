# 小程序商品券核销需求文档

**日期**：2026-05-28  
**范围**：微信小程序点餐、订单服务、CRM 商品券、平台团购券  
**当前口径**：商品券使用 `product_coupon` 接口体系；现金券仍走支付页付款方式体系。

## 1. 需求背景

当前小程序支付页主要按现金券模型处理优惠券。商品券的业务语义不同于现金券：商品券是“用券兑换或优惠一个绑定菜品”，核销后必须在订单中形成券菜品行，并沉淀券与菜品、券与订单金额的核算明细。

如果商品券继续在支付页按现金券处理，会产生以下风险：

- 订单没有券菜品行，后续退菜、退款、反结账无法准确处理。
- 商品券被当作付款方式二次扣减，导致订单头、订单行和付款方式不平。
- 美团、抖音团购券和会员券的商品券 / 现金券边界不清。
- 小程序前端承担金额计算，容易和 POS 口径分叉。

因此，小程序需要支持结算前直接核销商品券，由后端按 POS 口径完成核销、落菜、金额重算、取消和异常处理。

## 2. 需求目标

1. 小程序支持会员商品券、美团商品券、抖音商品券的核销。
2. 商品券在点餐阶段核销，并生成券菜品行。
3. 现金券不走商品券核销接口，仍作为支付页付款方式处理。
4. 小程序不计算商品券优惠金额，不自行修改菜品价格。
5. 后端统一完成券校验、核销、订单行写入、金额重算和券明细沉淀。
6. 支付页不把已核销商品券再次作为 `COUPON` 付款项提交。
7. 支持取消已核销商品券，并保证券状态、券菜品行和订单金额一致。

## 3. 业务范围

### 3.1 包含范围

| 范围 | 说明 |
|---|---|
| 商品券识别 | 识别会员、抖音、美团券码，并区分商品券和现金券 |
| 可用券查询 | 查询当前用户可用会员商品券 |
| 商品券核销 | 核销 `WP/MP/DP` 并生成券菜品行 |
| 平台券核销 | 支持美团、抖音商品券核销和取消 |
| 订单金额重算 | 商品券优惠由后端按 POS 口径计算和沉淀 |
| 支付页分流 | 商品券不进入普通现金券付款项 |
| 商品券取消 | 支持取消某一张已核销商品券 |
| 展示结果刷新 | 核销或取消后返回订单结果供小程序刷新 |

### 3.2 不包含范围

| 非范围 | 说明 |
|---|---|
| 现金券核销改造 | 现金券继续沿用现有支付页逻辑 |
| 前端金额计算 | 小程序不计算商品券优惠额、折扣额、应付额 |
| 前端构造券菜品 | 小程序不直接调用普通加菜接口创建券菜品行 |

## 4. 券类型规则

| 类型 | 含义 | 处理方式 |
|---|---|---|
| `WP` | 会员商品券 | 点餐页核销，生成券菜品行 |
| `MP` | 美团商品券 | 点餐页核销，生成券菜品行 |
| `DP` | 抖音商品券 | 点餐页核销，生成券菜品行 |
| `WV` | 会员现金券 | 支付页付款方式 |
| `MV` | 美团现金券 | 支付页付款方式 |
| `DV` | 抖音现金券 | 支付页付款方式 |

平台券识别规则对齐 POS：

- 券码以 `http` 开头，或长度 `>=15 && <19`，默认识别为抖音券。
- 券码长度 `>9 && <15`，默认识别为美团券。
- 其他券码默认识别为会员券。
- 平台返回 `groupon_type == 2` 时识别为现金券，否则识别为商品券。

## 5. 用户流程

### 5.1 核销商品券

1. 用户进入小程序点餐流程。
2. 用户输入或扫描券码，或从可用商品券列表选择商品券。
3. 小程序调用后端预识别接口，确认券来源和券类型。
4. 如果是现金券，提示用户到支付页使用或按现金券流程处理。
5. 如果是商品券，小程序调用商品券核销接口。
6. 后端核销成功后生成券菜品行，并返回订单结果。
7. 小程序刷新菜品列表、订单金额和商品券展示结果。

### 5.2 支付商品券订单

1. 用户进入支付页。
2. 支付页展示已核销商品券摘要。
3. 支付页可继续选择现金券、会员卡、微信支付等支付方式。
4. 支付提交时，只把非 `SWQ` 券转换为 `COUPON` 付款项。
5. 后端以订单已落商品券明细为准完成最终校验。

### 5.3 取消商品券

1. 用户在允许状态下取消某一张已核销商品券。
2. 小程序调用取消商品券核销接口。
3. 后端定位券菜品行，回滚订单金额和券明细。
4. 后端调用 CRM 或平台取消核销能力。
5. 小程序刷新取消后的订单结果。

## 6. 功能需求

### 6.1 可用商品券查询

- 系统应查询当前会员在当前门店、当前订单上下文下可用的会员商品券。
- 查询结果只返回商品券，不返回现金券。
- 查询结果用于展示，不作为最终核算依据。

### 6.2 商品券预识别

- 系统应根据券码识别券来源：会员、美团、抖音。
- 系统应根据平台返回结果识别商品券或现金券。
- 预识别结果应返回业务类型，供小程序决定后续流程。

### 6.3 商品券核销

- 核销接口只允许 `WP/MP/DP`。
- 会员商品券需要校验 CRM 券实例、`SWQ` 类型、绑定菜品、绑定单位。
- 美团、抖音商品券需要调用平台团购券核销能力。
- 核销成功后，后端必须生成券菜品行。
- 核销成功后，后端必须写入订单券核算明细。
- 核销成功后，后端必须返回完整订单结果或等价刷新数据。

### 6.4 金额处理

- 免费商品券沿用 POS 老逻辑。
- 折扣商品券、立减商品券保留菜品原价。
- 商品券优惠额单独沉淀到券优惠金额字段和订单券明细。
- 订单头应付金额只增加 `券菜品行金额 - 券优惠金额`。
- 商品券不得作为支付方式再次扣减。

### 6.5 支付页处理

- 支付页可以展示已核销商品券结果。
- 支付页不得把商品券放入普通现金券列表做金额抵扣。
- 支付页提交 `orderPayList` 时，只提交非 `SWQ` 券作为 `COUPON` 付款项。

### 6.6 取消核销

- 取消接口应优先使用 `couponWriteOffTraceNo` 定位券菜品行。
- 缺失 `couponWriteOffTraceNo` 时，可使用 `couponNo` 定位。
- 取消时必须删除对应券菜品行。
- 取消时必须回滚订单金额。
- 取消时必须移除对应券明细。
- 会员商品券调用 CRM 精确反核销。
- 平台商品券调用平台取消能力。

## 7. 数据要求

商品券核销后，订单需要具备以下数据：

| 数据 | 说明 |
|---|---|
| 券菜品行 | 表示商品券兑换或优惠的菜品 |
| `couponNo` | 会员商品券编号 |
| `couponWriteOffTraceNo` | 平台券或无券号场景下的稳定追踪号 |
| `promotionAmount` | 券菜品行优惠金额 |
| `coupon_items` | 券维度核算明细 |
| `product_items` | 商品维度核算明细 |
| `product_coupon_items` | 券与商品关系明细 |

## 8. 验收标准

1. 会员商品券可以在小程序核销成功，并生成券菜品行。
2. 美团商品券可以在小程序核销成功，并生成券菜品行。
3. 抖音商品券可以在小程序核销成功，并生成券菜品行。
4. 美团、抖音现金券不能走商品券核销接口生成菜品行。
5. 免费商品券金额表现与 POS 口径一致。
6. 折扣商品券、立减商品券保留菜品原价，优惠额单独沉淀。
7. 支付页不会把已核销商品券作为 `COUPON` 付款项提交。
8. 商品券和现金券混用时，订单金额、付款方式和券明细保持平衡。
9. 取消某一张商品券时，只影响该券对应的菜品行和券明细。
10. 重复点击核销不会重复生成券菜品行。
11. 后端契约测试通过。
12. 小程序契约测试通过。

## 9. 风险和约束

- 商品券核销必须依赖稳定 `orderId`，否则自动解锁、取消和对账无法可靠追踪。
- 平台商品券没有 CRM `couponNo` 时，必须使用 `couponWriteOffTraceNo` 作为定位键。
- 订单字段由本地服务自动加字段能力处理，上线前需要确认目标环境字段存在。
