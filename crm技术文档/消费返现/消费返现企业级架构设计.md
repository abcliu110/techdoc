# 消费返现企业级架构设计

日期：2026-05-16  
范围：CRM / POS 消费返现的配置、发放、撤销、补偿与审计  
目标：把消费返现做成独立闭环，参考消费积分/消费赠券的生命周期思路，但不复用它们的表和实现；不改 `order-app` / `order-service`。

## 1. 结论

当前 `crm_cash_back` / `crm_cash_back_food` 主要承担规则配置和菜品绑定，不是运行时返现执行主链路。  
消费返现应采用“门店驱动、CRM 执行”模式，由 POS/门店在消费完成、反结账、退款等事实节点触发返现事件，CRM 负责规则匹配、任务入账、撤销和补偿。

## 2. 设计目标

1. 消费完成后可可靠触发返现。
2. 同一消费事件重复投递、重试、补偿，不重复返现。
3. 反结账、退款、撤销时，可按原始返现事实回收。
4. 规则匹配、发放、撤销、补偿、审计分层清晰。
5. 保留现有返现配置页和历史直调能力，兼容旧流程。
6. 不动订单模块，返现不从 `order-service` 触发。

## 3. 总体架构

### 3.1 分层

- POS / 门店层：产出消费事实、反结账事实、补偿事件。
- CRM 入口层：接收返现事件，做幂等和快照落库。
- 规则匹配层：按 `crm_cash_back` / `crm_cash_back_food` 计算命中规则。
- 任务层：生成返现任务、撤销任务、补偿任务。
- 账务执行层：调用现有 `cardCashBack / revokeCashBack` 入账或冲正。
- 审计层：保存任务、明细、快照、流水号、失败原因。

### 3.2 边界

- POS 只负责消费事实和补偿重试，不碰余额表。
- CRM 负责规则、任务、明细、撤销、补偿。
- 订单模块不参与这条新链路。

## 4. 核心模型

### 4.1 生命周期

建议新增独立生命周期模型，不复用积分/赠券：

- `crm_consume_cash_session`
- `crm_consume_cash_round`
- `crm_consume_cash_event`

含义：

- `session`：订单 + 门店 + 会员 + 来源维度的总生命周期根。
- `round`：一轮返现生命周期，部分退款时可新起一轮。
- `event`：消费完成、反结账、退款、补偿等弱事件。

### 4.2 CRM 任务与明细

建议新增：

- `crm_consume_cash_task`
- `crm_consume_cash_detail`

任务表保存：

- 幂等键
- 订单号 / 账单号 / 会员卡
- 命中的返现规则快照
- 返现金额快照
- 状态
- 执行结果

明细表保存：

- 实际发放流水号
- 实际撤销流水号
- 关联原始任务
- 失败原因
- 回收状态

## 5. 规则模型

### 5.1 规则来源

继续使用现有配置表作为规则来源：

- `crm_cash_back`
- `crm_cash_back_food`

但在消费完成时必须保存规则快照，后续重试和撤销都只看快照，不重新匹配当前配置。

### 5.2 规则维度

建议纳入：

- 商户 / 门店
- 会员卡类型 / 等级
- 消费金额区间
- 返现类型：固定金额 / 比例金额
- 最大返现金额
- 有效期
- 延期天数
- 不参与返现菜品

### 5.3 优先级

若多条规则命中，建议显式定义优先级：

1. 门店级优先于商户级
2. 更具体的卡等级优先于卡类型
3. 区间更窄的规则优先
4. 创建时间更早的规则优先

## 6. 发放流程

### 6.1 消费完成

1. POS / 门店在消费完成后生成返现事件。
2. 写入 outbox / 本地事件表。
3. CRM 返现入口接收事件，先做幂等。
4. 匹配 `crm_cash_back` 规则，生成 `round` 和 `task`。
5. 事务提交后执行返现入账。
6. 成功后写返现明细和余额流水。
7. 失败进入补偿队列。

### 6.2 反结账 / 退款 / 撤销

1. 查原始 `session` / `round` / `task`。
2. 不重新计算当前规则。
3. 按原始返现事实发起撤销。
4. 撤销成功后标记明细与任务为已回收。
5. 若已被使用或无法回收，进入异常单。

## 7. 接口设计

建议新增 CRM 专用接口：

- `POST /crm_consume_cash/grantSign`
- `POST /crm_consume_cash/revokeSign`

接口职责：

- 只接收幂等业务键和事实快照
- 不让调用方直接操控余额表
- 不暴露规则计算细节
- 重复请求返回已存在任务结果

## 8. 兼容策略

- 现有 `crm_cash_back` 配置页保留。
- 现有卡余额直调接口保留，用于人工补单或历史兼容。
- 新链路不复用积分、赠券的表和实现类。
- 不改 `order-app` / `order-service`。

## 9. 测试重点

1. 同一消费事件重复投递不重复返现。
2. 返现成功后可撤销。
3. 撤销晚于发放回包时不覆盖事实。
4. 规则变更后，历史任务仍按快照回收。
5. 菜品黑名单、金额门槛、卡等级条件生效。
6. CRM 超时、POS 重启、重试可补偿恢复。
7. 订单模块不受影响。

## 10. 风险与约束

- 不能把返现撤销做成“重新计算当前规则再撤销”。
- 不能让 `crm_cash_back` 继续只做纯 CRUD，否则还是无法闭环。
- 不能复用积分/赠券表，否则后续返现审计会和别的权益混账。
- 返现任务和明细必须保留原始事实快照，便于对账和回收。
