# 消费返现字段说明

## 1. 文档范围

本文说明 SaaS 后台「会员营销 - 营销管理 - 消费返现」页面中新增、修改弹窗的字段含义，并结合当前前后端代码说明字段保存、查询、匹配和返现金额计算规则。

截图对应页面标题为「修改消费返现」。前端页面位于 `nms4cloud-biz-ui/src/pages/CrmMarketingMrg/components/CrmCashBack`，后端接口位于 `nms4cloud-app/2_business/nms4cloud-crm` 的 `CrmCashBackController`、`CrmCashBackServicePlus`、`CrmCashBackAddDTO`、`CrmCashBackUpdateDTO`、`CrmCashBackVO`、`CrmCashBack`。

## 2. 页面入口与接口

前端路由：`/membermarketing/crmmarketingmrg/crmcashback`

前端接口前缀：`/api/scrm/crm_cash_back/`

后端 Controller 路径：`/crm_cash_back`

主要接口：

| 操作 | 前端调用 | 后端方法 | 说明 |
|---|---|---|---|
| 列表 | `POST /list` | `listCrmCashBack` | 查询消费返现规则列表 |
| 详情 | `POST /get` | `getCrmCashBack` | 获取单条规则详情，同时返回已设置的返现商品列表 |
| 新增 | `POST /add` | `addCrmCashBack` | 新增消费返现规则 |
| 修改 | `POST /update` | `updateCrmCashBack` | 修改消费返现规则 |
| 删除 | `POST /del` | `delCrmCashBack` | 按规则 `lid` 删除 |
| 返现商品 | `POST /add_cash_back_dish` | `addCashBackDish` | 设置该规则关联的菜品列表 |

## 4. 字段详细说明

### 4.1 消费返现名称

`name` 是规则名称。新增和修改时，后端按当前商户 `mid`、当前登录门店 `sid` 和名称做重复校验；新增时同名规则不允许保存，修改时排除当前 `lid` 后仍不允许重名。

截图中该字段显示为「会员卡等级和返现关系表名称」，新增表单显示为「消费返现名称」。两者落到同一个字段 `name`。

### 4.2 活动开始时间、活动结束时间

`validBeginTime` 和 `validEndTime` 用于规则有效期判断。POS 发起消费返现时，CRM 使用 POS 传入的结账时间 `checkOutTime` 匹配规则：

- `validBeginTime <= checkOutTime` 或开始时间为空时满足开始条件。
- `validEndTime > checkOutTime` 或结束时间为空时满足结束条件。

代码现状注意点：

- 修改接口会把前端选择的结束日期转成当天 `23:59:59`。
- 新增接口当前代码把 `validEndTime` 设置为 `validBeginTime.plusDays(1).minusSeconds(1)`，也就是按开始日期生成当天 `23:59:59`，没有使用前端传入的结束日期。若业务要求新增时支持跨天活动，需要单独核对这段实现。

### 4.3 会员类型、会员等级

前端提交的是编号：

- `cardTypeCode`：会员卡类型编号，值来自会员卡类型列表中的 `lid`。
- `cardTypeLevelCode`：会员等级编号，值来自会员等级列表中的 `lid`。

后端保存前会校验：

- `cardTypeCode` 不为空时，必须能在当前商户、当前门店下查到会员卡类型。
- `cardTypeLevelCode` 不为空时，必须能在当前会员卡类型下查到会员等级。
- 校验通过后，后端会把名称冗余保存到 `cardType`、`cardTypeLevel`，便于列表展示和历史快照。

POS 消费返现匹配时，会优先匹配会员卡档案上的类型和等级。规则中的类型或等级为空时，代码允许作为通用规则参与匹配。

### 4.4 最小消费金额、最大消费金额

`minAmount`、`maxAmount` 是消费金额区间。当前 POS 消费返现按整单参与返现的实付金额 `billAmount` 判断：

- `minAmount` 为空，表示没有下限。
- `maxAmount` 为空，表示没有上限。
- 命中条件为 `billAmount >= minAmount` 且 `billAmount <= maxAmount`。

当前配置界面虽然还有「返现商品」设置入口，但 POS 返现执行服务的注释和代码明确说明：当前消费返现配置不按商品或菜品维度计算，金额区间和返现金额只按 POS 传入的整单实付金额计算。

### 4.5 返现计算类型

`cashBackType` 使用枚举 `CashBackTypeEnum`：

| 枚举值 | code | 页面文案 | 计算含义 |
|---|---:|---|---|
| `AMOUNT` | 1 | 按金额 | 直接取规则中的 `cashBackAmount` 作为返现金额 |
| `RATE` | 2 | 按比例 | 使用 `billAmount * cashBackAmountRate` 计算返现金额 |

前端枚举文件使用字符串枚举 `AMOUNT = '按金额'`、`RATE = '按比例'`，表单实际提交枚举名；后端 DTO 和 Entity 使用 `CashBackTypeEnum`。

### 4.6 返现金额

`cashBackAmount` 是固定返现金额。只有当 `cashBackType=AMOUNT` 时，当前计算逻辑会读取该字段。

示例：规则配置 `cashBackType=AMOUNT`、`cashBackAmount=100`，命中规则后本次应返现金额先取 `100`，再受 `cashBackMaxAmount` 限制。

### 4.7 返现金额比例

`cashBackAmountRate` 是比例返现参数。页面使用百分比展示和输入，例如截图中的 `10%`；后端保存前会除以 `100`，因此数据库中保存为 `0.10`。

列表和详情返回时，后端会把数据库小数乘以 `100` 再返回给前端展示。

当 `cashBackType=RATE` 时，返现金额计算公式为：

```text
返现金额 = floor_to_2_decimal(参与返现账单金额 * cashBackAmountRate)
```

代码使用 `RoundingMode.DOWN` 向下保留 2 位。

### 4.8 最大返现金额

`cashBackMaxAmount` 是最终返现金额封顶值。无论按金额还是按比例计算，只要该字段大于 0，最终返现金额都不能超过它。

示例：账单金额 `1000`，比例 `10%`，初算返现 `100`；如果 `cashBackMaxAmount=80`，最终返现为 `80`。

### 4.9 延期使用天数

`deferredDayNum` 是规则表中的配置字段，页面可填写，数据库保存为 `deferred_day_num`。从当前 POS 消费返现执行链路看，规则匹配和返现金额计算没有读取该字段。

因此当前可确认的代码语义是：该字段会保存和展示，但不影响 POS 消费返现是否命中、也不影响返现金额。

### 4.10 适用门店、选择门店

`isAllStore` 决定规则适用范围：

- `true`：所有门店。前端清空 `storeSids`，后端保存时也会把 `storeSids` 置空。
- `false`：选择门店。前端展示多选门店字段 `storeSids`；后端会去重、过滤空值，并要求至少选择一个门店。

POS 消费返现匹配门店时：

- `isAllStore=true` 直接命中门店范围。
- `storeSids` 不为空时，POS 请求门店 `sid` 必须在列表中。
- 为兼容旧数据，如果没有多门店字段，则回退比较规则原有 `sid` 与请求 `sid` 是否相等。

## 6. 规则命中顺序

POS 发放消费返现时，CRM 先查出当前商户下状态为启用的规则，并按以下条件过滤：

1. 会员卡类型匹配，或规则未配置会员卡类型。
2. 会员等级匹配，或规则未配置会员等级。
3. 结账时间在有效期内。
4. 门店范围匹配。
5. 账单金额满足最小、最大消费金额区间。

规则排序逻辑：

1. 会员等级编号倒序。
2. 会员类型编号倒序。
3. 最小消费金额倒序。
4. `pid` 升序。

遍历排序后的规则，第一条同时满足门店和金额条件的规则即为命中规则。

## 7. 返现发放与撤销关系

消费返现实际发放入口在 `CrmConsumeCashOpServicePlus`。POS 结账成功后传入会员卡、结账时间、账单实付金额和幂等键，CRM 命中规则后调用旧会员卡返现能力 `cardCashBack`，把返现金额加入会员卡余额。

当前发放策略：

- 返现金额小于等于 0 时按未命中处理，不调用余额变动。
- 发放成功后记录命中规则 `ruleLid`、返现金额 `cashBackAmount` 和旧余额任务 `taskLid`。
- 撤销时必须依赖原返现任务 `taskLid`，避免误撤同卡其它余额操作。
- 退款后第一版策略是撤销原返现轮次，不按剩余金额自动重发。

## 10. 需要重点核对的实现点

1. 新增接口当前会用开始日期生成结束时间，没有使用前端提交的结束日期；如果业务期望新增时支持任意结束日期，需要调整或确认。
2. 前端必填项主要包括名称、活动开始时间、活动结束时间、会员类型、会员等级、适用门店；金额、计算类型、返现金额等字段当前页面未设置必填规则。
3. 后端计算时未看到对 `cashBackType`、`cashBackAmount`、`cashBackAmountRate` 的强业务校验；配置不完整时可能得到 0 或空值兜底结果。
4. 「返现商品」当前不参与 POS 消费返现计算，不能把该配置理解为菜品级返现过滤规则。
