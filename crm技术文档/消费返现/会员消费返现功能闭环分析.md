# 会员消费返现功能闭环分析

## 结论

当前项目中的“会员消费返现”是**部分闭环**，不是完整自动闭环。

- 正向返现链路已经打通：消费生成返现明细，定时任务触发发放，最终落到会员卡余额。
- 逆向撤销链路只覆盖了“未执行”的返现明细；已执行返现没有看到与返现明细强绑定的自动回滚闭环。
- `crm_cash_back` 主要是返现规则配置与菜品绑定管理，不是运行时返现执行主链路。

## 一、两层“返现”概念

### 1. 配置层 `crm_cash_back`

`CrmCashBackController` / `CrmCashBackServicePlus` 负责返现规则的增删改查、菜品绑定、卡类型校验等配置管理，接口集中在 `/crm_cash_back/*`。

参考：

- `nms4cloud-crm-app/src/main/java/com/nms4cloud/crm/app/controller/CrmCashBackController.java:45-85`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/CrmCashBackServicePlus.java:43-205`

### 2. 运行层会员消费返现

真正的会员消费返现运行链路走的是 `CrmRebateDetail` + `RebateExecTask` + `cardCashBack()` + `CardBalanceService`。

参考：

- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/CrmShareholderServicePlus.java:540-590`
- `nms4cloud-crm-app/src/main/java/com/nms4cloud/crm/app/task/RebateExecTask.java:53-117`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/card/CrmCardOpServicePlus.java:1723-1744`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/balance/CardBalanceService.java:1735-1745`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/balance/CardBalanceService.java:1967-1979`

## 二、正向链路

1. `CrmShareholderServicePlus.consume()` 根据消费信息生成 `CrmRebateDetail`，其中 `rebateType = C`，`done = false`。
2. `RebateExecTask` 扫描 `done = false`、`cancel = false`、`rebateType = C` 的返现明细，调用 `cardCashBack()`。
3. `CrmCardOpServicePlus.cardCashBack()` 创建 `CrmDealTask`，操作模式为 `OperationModelEnum.CashBack`，再执行卡余额变更。
4. `CardBalanceService` 在 `CashBack` 分支中更新 `sum_of_back_times`、`sum_of_back`、`back_balance`。

这说明“消费 -> 生成返现明细 -> 定时发放 -> 入账”这条正向链路是闭合的。

## 三、撤销链路与缺口

### 1. 已有撤销能力

- `CrmShareholderServicePlus.undoConsume()` 只能把 **尚未执行** 的返现明细标记为 `cancel = true`。
- `CrmCardOpServicePlus` 里存在 `revokeCashBack()`，`CardBalanceService` 里也有 `CancelCashBack` 分支，说明系统具备返现冲正能力。

参考：

- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/CrmShareholderServicePlus.java:595-606`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/card/CrmCardOpServicePlus.java:1747-1757`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/balance/CardBalanceService.java:1972-1979`

### 2. 现有缺口

- `undoConsume()` 没有把已执行返现一起反向处理。
- `CrmRebateDetail` 只记录了 `done / cancel / doneAt / cancelAt`，没有看到保存 `cardCashBack()` 生成的任务号或余额操作号，后续精确回滚链路不够直接。
- 没有发现 `crm_cash_back` 配置表直接驱动运行时返现执行的证据。

参考：

- `nms4cloud-crm-dao/src/main/java/com/nms4cloud/crm/dao/entity/CrmRebateDetail.java:51-137`
- `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/CrmShareholderServicePlus.java:595-606`

## 四、补充观察

订单侧还有一条独立的分享返现路径：

- `ShareCloseQueueConsumer` 在订单关闭后，会直接调用 `cardCashBackInner()` 给会员返现。

这条链路也是直接入卡余额，不是通过 `crm_cash_back` 配置表驱动。

参考：

- `nms4cloud-order-app/src/main/java/com/nms4cloud/order/app/task/ShareCloseQueueConsumer.java:218-231`

## 五、最终判断

如果“闭环”指的是“消费后自动发放返现”，当前是闭合的。

如果“闭环”指的是“配置、发放、撤销、追踪、冲正都能在同一条业务链路里完整打通”，当前**还不是完全闭环**，更准确地说是：

> 正向发放闭环已完成，逆向撤销闭环未完全打通。
