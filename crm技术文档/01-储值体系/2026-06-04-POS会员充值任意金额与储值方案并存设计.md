# POS 会员充值任意金额与储值方案并存设计

## 1. 背景

POS 会员充值当前主要入口已经切到新会员充值页：

- `nms4pos-ui/app/pos4desktop/src/pages/FunctionPanel/index.tsx`
- `nms4pos-ui/app/pos4desktop/src/pages/Bill/component/MoreFunction/index.tsx`

新页面为：

- `nms4pos-ui/app/pos4desktop/src/pages/FunctionPanel/pages/MemberRechargeNew/index.tsx`

新页面已支持储值方案充值，也保留了手动输入金额的 UI，但提交时当前强制要求选择储值方案档位。因此收银员不选择储值方案时，无法按老方式直接输入任意金额充值。

老充值页和老后端链路仍然支持任意金额充值：

- 前端旧接口：`member_ops_cardCharge`
- 后端旧入口：`/member_ops/cardCharge`
- 普通充值规则：`ruleLid = -1`

老后端在 `ruleLid < 0` 时直接使用前端传入的 `principalAmount` 和 `giveAmount`。旧 POS 页面实际手动输入任意金额时，`giveAmount` 固定为 `0`，即手动任意金额只增加本金。

## 2. 设计目标

在不切回旧页面的前提下，让新会员充值页同时支持：

1. 普通任意金额充值。
2. 储值方案档位充值。

设计约束：

- 不修改后端接口路由和请求协议结构。
- 后端必须补充防御性校验和收口分流保护，避免普通充值哨兵值 `-1` 写入储值方案字段。
- 不修改 CRM 入账接口。
- 不修改数据库字段。
- 不修改路由入口。
- 普通任意金额充值与储值方案充值必须互斥，不能在同一笔充值单里混用。
- 保留旧普通充值链路，避免破坏历史行为。

## 3. 当前链路分析

### 3.1 普通任意金额充值旧链路

旧页面手动输入金额后，会加入一条普通充值规则：

```ts
{
  ruleLid: '-1',
  principalAmount: inputAmount,
  giveAmount: 0,
  times: 1,
}
```

提交时走旧接口：

```ts
member_ops_cardCharge({
  authNo: '',
  cardNo,
  rules,
  pays,
})
```

后端旧接口处理规则：

- `ruleLid < 0`：使用前端传入的 `principalAmount` 和 `giveAmount`。
- `ruleLid > 0`：查询 CRM 充值规则，使用规则里的本金和赠送金额。

支付成功收口时，如果本地充值单没有 `depositPlanLid`，继续走旧 CRM 充值接口 `cardChargeSign`。

### 3.2 储值方案充值新链路

新页面选择储值方案档位后，提交走新接口：

```ts
member_deposit_plan_ops_cardCharge({
  authNo: '',
  cardNo,
  planLid,
  tierIndex,
  saveAmount,
  giftAmount,
  planName,
  pays,
})
```

后端新接口要求：

- `planLid` 必填，且必须大于 `0`。
- `tierIndex` 必填，且必须大于等于 `0`。
- `saveAmount` 必填。

本地充值单会写入：

- `depositPlanLid`
- `depositTierIndex`

支付成功收口时，只有本地充值单 `depositPlanLid > 0` 才能走新储值方案 CRM 入账接口 `deposit-plan/charge/commitByPos`。

### 3.3 不能混用的原因

普通任意金额充值和储值方案充值最终收口链路不同：

- 没有 `depositPlanLid`：老普通充值链路。
- `depositPlanLid > 0`：新储值方案链路。
- `depositPlanLid <= 0`：异常数据，不能按储值方案处理。

当前新储值方案后端模型是“单方案 + 单档位”，不是“多规则混合”模型。如果一笔单里同时存在普通金额和储值方案档位，会导致本地单据、CRM 入账、赠送规则、防篡改校验等边界变复杂。

因此本设计要求两者在同一笔充值单中互斥。

## 4. 目标交互

### 4.1 普通任意金额充值

操作流程：

1. 收银员输入会员卡。
2. 不选择储值方案。
3. 在金额输入框输入任意金额。
4. 点击添加。
5. 选择支付方式。
6. 点击确定。

系统行为：

- 页面生成 `ruleLid = -1` 的普通充值规则。
- 提交时走旧接口 `member_ops_cardCharge`。
- 后端按旧普通充值链路入账。

### 4.2 储值方案充值

操作流程：

1. 收银员输入会员卡。
2. 选择储值方案。
3. 选择一个档位。
4. 选择支付方式。
5. 点击确定。

系统行为：

- 页面生成储值方案档位规则。
- 提交时走新接口 `member_deposit_plan_ops_cardCharge`。
- 后端按新储值方案链路入账。

## 5. 互斥规则

必须在页面交互和提交前都做互斥保护。

### 5.1 已添加普通金额后禁止选择储值方案

判断条件：

```ts
displayRules.some((rule) => rule.ruleLid === '-1')
```

提示文案：

```text
已添加普通充值金额，请先移除后再选择储值方案
```

### 5.2 已选择储值方案后禁止添加普通金额

判断条件：

```ts
displayRules.some((rule) => rule.ruleLid !== '-1')
```

提示文案：

```text
已选择储值方案，请先移除后再添加普通充值金额
```

### 5.3 提交前再次校验

提交前防御性判断：

```ts
const planRule = displayRules.find((rule) => rule.ruleLid !== '-1');
const manualRule = displayRules.find((rule) => rule.ruleLid === '-1');

if (planRule && manualRule) {
  return message.warning('普通充值和储值方案不能同时充值，请先移除其中一项');
}
```

## 6. 前端修改方案

修改文件：

```text
nms4pos-ui/app/pos4desktop/src/pages/FunctionPanel/pages/MemberRechargeNew/index.tsx
```

### 6.1 增加旧接口导入

新增：

```ts
member_ops_cardCharge,
member_ops_getDealingTask,
```

仍从以下模块导入：

```ts
@yd/api-pos4plugin/services/pos4desktop/member
```

### 6.2 修改提交逻辑

当前提交逻辑强制要求必须存在储值方案档位。需要改为按 `planRule` 是否存在分流。

伪代码：

```ts
const planRule = displayRules.find((rule) => rule.ruleLid !== '-1');
const manualRule = displayRules.find((rule) => rule.ruleLid === '-1');

if (planRule && manualRule) {
  return message.warning('普通充值和储值方案不能同时充值，请先移除其中一项');
}

await checkPermission({
  label: '会员充值',
  requestFn: (token) => {
    if (!planRule) {
      return member_ops_cardCharge({
        authNo: '',
        cardNo: selectedMember.id,
        rules: displayRules,
        pays,
        ...token,
      });
    }

    return member_deposit_plan_ops_cardCharge({
      authNo: '',
      cardNo: selectedMember.id,
      planLid: planRule.ruleLid,
      tierIndex: planRule.tierIndex,
      saveAmount: planRule.principalAmount,
      giftAmount: planRule.giveAmount,
      planName: planRule.name,
      pays,
      ...token,
    });
  },
});
```

### 6.3 修改手动金额添加逻辑

在点击“添加”普通金额前增加判断：

```ts
if (displayRules.some((rule) => rule.ruleLid !== '-1')) {
  return message.warning('已选择储值方案，请先移除后再添加普通充值金额');
}
```

通过后继续保持旧普通充值结构：

```ts
{
  ruleLid: '-1',
  principalAmount: inputAmount,
  giveAmount: 0,
  times: 1,
}
```

### 6.4 修改储值方案选择逻辑

在打开储值方案档位弹窗前增加判断：

```ts
if (displayRules.some((rule) => rule.ruleLid === '-1')) {
  return message.warning('已添加普通充值金额，请先移除后再选择储值方案');
}
```

在档位确认时也做同样校验，防止弹窗打开后页面状态变化导致混用。

### 6.5 修改未完成单恢复逻辑

当前新页面只查询储值方案未完成单：

```ts
member_deposit_plan_ops_getDealingTask({ orderOpType: 'E' })
```

建议改为：

1. 先查询新储值方案未完成单。
2. 如果没有支付二维码，再查询旧普通充值未完成单。

伪代码：

```ts
const planRes = await member_deposit_plan_ops_getDealingTask({ orderOpType: 'E' });
if (planRes.data?.order?.payQrCode) {
  setBillDetail(planRes.data);
  setPaymentCodeModalShow(true);
  return;
}

const legacyRes = await member_ops_getDealingTask({ orderOpType: 'E' });
if (legacyRes.data?.order?.payQrCode) {
  setBillDetail(legacyRes.data);
  setPaymentCodeModalShow(true);
}
```

这样普通任意金额充值扫码出码后，如果未完成支付，重新进入新页面也能恢复支付弹窗。

## 7. 后端防御性修改方案

前端分流是主路径，但后端不能信任前端。已出现过新储值方案接口收到 `planLid = -1`、`tierIndex = -1` 后把本地单写成 `deposit_plan_lid = -1` 的坏数据；后续收口按 `depositPlanLid != null` 分流，最终拿 `-1` 查询储值方案并报“储值方案不存在”。

### 7.1 储值方案入口校验

修改文件：

```text
nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/controller/biz/MemberDepositPlanForBizController.java
```

在 `cardCharge(...)` 开始处理前增加业务校验：

```java
Assert.isTrue(
    Objects.nonNull(request.getPlanLid()) && request.getPlanLid() > 0,
    "储值方案ID不能为空");
Assert.isTrue(
    Objects.nonNull(request.getTierIndex()) && request.getTierIndex() >= 0,
    "储值方案档位不能为空");
```

说明：

- `@NotNull` 只能拦截空值，不能拦截 `-1`。
- `planLid = -1` 是普通充值旧规则的哨兵值，只允许出现在旧接口 `rules[].ruleLid`。
- 新储值方案接口里 `planLid <= 0` 必须视为非法请求。

### 7.2 收口分流保护

修改文件：

```text
nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/util/OrderServiceUtil.java
```

当前分流不能只判断 `depositPlanLid != null`，需要改成：

```java
Long depositPlanLid = dwdBill.getDepositPlanLid();
if (Objects.nonNull(depositPlanLid) && depositPlanLid <= 0) {
  throw new BizException("储值方案充值单数据异常");
}
if (Objects.nonNull(depositPlanLid)) {
  DepositPlanChargeUtil.dealDepositPlanCharge(dwdBill, orderOpType, user);
  return;
}
```

等价规则：

- `depositPlanLid == null`：普通任意金额充值，走旧 `cardChargeSign`。
- `depositPlanLid > 0`：储值方案充值，走 `commitByPos`。
- `depositPlanLid <= 0`：异常数据，阻断并提示，不静默降级。

### 7.3 旧普通充值复用单保护

修改文件：

```text
nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/controller/biz/MemberForBizController.java
```

旧 `/member_ops/cardCharge` 查询未完成充值单时必须只复用普通充值单：

```java
OrderServiceUtil.forOrderQuery()
    .eq(DwdBill::getMid, request.getMid())
    .eq(DwdBill::getSid, request.getSid())
    .between(DwdBill::getReportDate, startDate, endDate)
    .eq(DwdBill::getOrderOpType, OrderOpTypeEnum.E)
    .notIn(DwdBill::getOrderStatus, OrderStatusEnum.CLOSED, OrderStatusEnum.CANCEL)
    .eq(DwdBill::getDeviceCode, request.getDevId())
    .isNull(DwdBill::getDepositPlanLid)
    .orderByDesc(DwdBill::getLid)
    .onlyOne();
```

这样旧普通充值不会复用已被标记为储值方案充值的未完成单。

### 7.4 历史坏数据处理

如果本地 POS 库已存在 `deposit_plan_lid = -1` 的未完成充值单，需要按实际单据确认后修复。示例：

```sql
SELECT pid, mid, sid, lid, saas_order_key,
       order_op_type, order_status, device_code,
       deposit_plan_lid, deposit_tier_index,
       report_date, card_no, paid_amount, created_time, updated_time
FROM dwd_bill
WHERE order_op_type = 4
  AND order_status NOT IN (4, 5)
  AND deposit_plan_lid <= 0
ORDER BY lid DESC;
```

确认属于普通任意金额充值后，将储值方案标记清空：

```sql
UPDATE dwd_bill
SET deposit_plan_lid = NULL,
    deposit_tier_index = NULL
WHERE lid = :billLid
  AND order_op_type = 4
  AND order_status NOT IN (4, 5)
  AND deposit_plan_lid <= 0;
```

枚举落库值：

- `order_op_type = 4`：充值订单。
- `order_status = 4`：已结账。
- `order_status = 5`：已取消。

## 8. 不修改范围

本方案不修改：

- CRM 充值接口
- CRM 储值方案接口
- 数据库字段
- POS 路由入口
- 旧充值页

## 9. 兼容性说明

### 9.1 老逻辑保留

普通任意金额充值继续走旧接口 `member_ops_cardCharge`，请求结构和历史语义保持一致。

### 9.2 新逻辑保留

储值方案充值继续走新接口 `member_deposit_plan_ops_cardCharge`，不改变 `planLid/tierIndex/saveAmount/giftAmount` 语义。

### 9.3 新旧链路隔离

通过前端互斥、提交分流和后端防御保证：

- 普通充值单不写入 `depositPlanLid`。
- 储值方案单继续写入有效的 `depositPlanLid` 和 `depositTierIndex`。
- `ruleLid = -1` 只能用于旧普通充值规则，不能进入新接口 `planLid` 或本地单 `depositPlanLid`。

后端 `OrderServiceUtil.dealCharge` 的分流规则必须以 `depositPlanLid == null`、`depositPlanLid > 0`、`depositPlanLid <= 0` 三类明确区分。

## 11. 风险与边界

### 11.1 不支持同一笔混合充值

本方案不支持同一笔充值单中同时包含：

- 普通任意金额。
- 储值方案档位。

原因是当前后端收口链路按 `depositPlanLid` 判断普通充值和储值方案充值，混合模型需要重新设计本地单据、CRM 入账和防篡改校验。

### 11.2 任意金额赠送暂不开放

旧后端接口支持 `ruleLid < 0` 时传入 `giveAmount`，但旧 POS 页面手动任意金额历史行为是 `giveAmount = 0`。

本方案保持该历史行为，不新增手动赠送金额输入框。

### 11.3 未完成单恢复优先级

建议先恢复储值方案未完成单，再恢复普通充值未完成单。这样不会改变新储值方案页面的优先行为。

### 11.4 后端防御不能省略

即使前端已经正确分流，后端也必须保留 `planLid > 0`、`tierIndex >= 0` 和收口分流校验。原因是 POS 页面、自动化脚本、旧缓存或外部调用都可能绕过前端互斥逻辑直接请求接口。

## 12. 回滚方案

如上线后出现异常，优先回滚前端文件：

```text
nms4pos-ui/app/pos4desktop/src/pages/FunctionPanel/pages/MemberRechargeNew/index.tsx
```

回滚内容：

- 移除旧接口导入。
- 恢复提交时必须选择储值方案档位的逻辑。
- 移除普通充值未完成单兜底查询。
- 移除普通充值与储值方案互斥提示。

后端防御校验属于数据保护逻辑，原则上不建议回滚。若必须回滚后端代码，应同步确认：

- 没有前端或脚本继续向新储值方案接口传 `planLid <= 0`。
- 本地 `dwd_bill` 不存在 `deposit_plan_lid <= 0` 的未完成充值单。
- 已有异常单已按 7.4 清理。
