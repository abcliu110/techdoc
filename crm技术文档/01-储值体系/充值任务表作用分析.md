# CrmDealTask 表作用分析

## 1. 核心定位

**CrmDealTask 是会员储值/消费交易的任务表（订单表）**，在整个交易流程中扮演**中间状态持有者**的角色。

### 1.1 为什么需要 CrmDealTask？

在会员储值/消费流程中，从用户发起交易到交易完成，存在以下时间差：

```
用户提交储值请求
    ↓
创建支付订单（等待支付）⏳
    ↓
用户完成支付
    ↓
支付回调通知
    ↓
更新会员余额 ✅
```

**问题：** 在等待支付的过程中，如何保存交易信息？

**解决方案：** 使用 CrmDealTask 表保存交易任务信息，作为临时状态持有者。

## 3. 完整生命周期

### 3.1 储值流程（Charge）

```
阶段1：创建任务（ChargeService.commit）
    ↓
创建 CrmDealTask
    ├─ operation_model = Charge
    ├─ trade_state = USERPAYING（待支付）
    ├─ card_lid = 会员卡ID
    ├─ save_rule_code = 储值规则ID
    ├─ principal_amount = 本金
    ├─ give_amount = 赠送金额
    ├─ give_point = 赠送积分
    ├─ give_coupon_id = 赠送券ID
    └─ created_time = 当前时间
    ↓
保存到数据库
    ↓
返回支付URL给用户

阶段2：用户支付（等待中）⏳
    ↓
CrmDealTask 状态：USERPAYING
    ↓
10分钟后未支付 → 自动关单（trade_state = CLOSED）

阶段3：支付回调（CardBalanceService.execute）
    ↓
查询 CrmDealTask（通过 lid）
    ↓
验证任务状态（必须是 USERPAYING）
    ↓
更新会员余额
    ├─ card.principal_balance += task.principal_amount
    ├─ card.give_balance += task.give_amount
    └─ card.points += task.give_point
    ↓
发放优惠券（如果有）
    ↓
升级会员等级（如果配置了）
    ↓
更新任务状态
    ├─ trade_state = SUCCESS
    └─ success_time = 当前时间
    ↓
完成 ✅
```

### 3.2 消费流程（Consume）

```
阶段1：创建任务（POS收银或线上消费）
    ↓
创建 CrmDealTask
    ├─ operation_model = Consume
    ├─ trade_state = USERPAYING
    ├─ card_lid = 会员卡ID
    ├─ amount = 消费金额（负数）
    ├─ principal_amount = 扣除本金（负数）
    └─ give_amount = 扣除赠送金额（负数）
    ↓
保存到数据库

阶段2：支付回调
    ↓
查询 CrmDealTask
    ↓
扣除会员余额
    ├─ card.principal_balance -= task.principal_amount.abs()
    └─ card.give_balance -= task.give_amount.abs()
    ↓
更新任务状态
    ├─ trade_state = SUCCESS
    └─ success_time = 当前时间
    ↓
完成 ✅
```

### 3.3 会员升级流程（AsUpgradeCost）

```
阶段1：创建任务
    ↓
创建 CrmDealTask
    ├─ operation_model = Charge
    ├─ as_upgrade_cost = true（标记为升级费用）
    ├─ card_type_code = 升级到的会员类型ID
    ├─ org_type_code = 原会员类型ID
    └─ amount = 升级费用
    ↓
保存到数据库

阶段2：支付回调
    ↓
查询 CrmDealTask
    ↓
判断 as_upgrade_cost = true
    ↓
升级会员等级
    ├─ card.card_type_code = task.card_type_code
    └─ card.card_type = 新会员类型名称
    ↓
发放升级赠送券（如果有）
    ↓
更新任务状态
    ├─ trade_state = SUCCESS
    └─ success_time = 当前时间
    ↓
完成 ✅
```

**关键逻辑：**

## 6. 状态机设计

### 6.1 交易状态（TradeStateEnum）

```
USERPAYING（待支付）
    ↓
    ├─ 用户支付成功 → SUCCESS（成功）
    ├─ 用户取消支付 → CLOSED（关闭）
    ├─ 超时未支付 → CLOSED（关闭）
    └─ 管理员撤销 → REVOKED（已撤销）
```

### 6.2 状态转换规则

| 当前状态 | 允许的操作 | 目标状态 |
|---------|-----------|---------|
| USERPAYING | 支付成功 | SUCCESS |
| USERPAYING | 取消支付 | CLOSED |
| USERPAYING | 超时 | CLOSED |
| SUCCESS | 撤销 | REVOKED |
| CLOSED | 无 | - |
| REVOKED | 无 | - |

## 7. 新旧流程对比

### 7.1 旧流程（CrmSaveRule）

```
用户选择储值规则
    ↓
ChargeService.commit()
    ├─ 查询 CrmSaveRule
    ├─ 计算金额
    └─ 创建 CrmDealTask
        ├─ save_rule_code = 规则ID
        ├─ principal_amount = 本金
        ├─ give_amount = 赠送金额
        ├─ give_point = 赠送积分
        └─ give_coupon_id = 赠送券ID（单个）
    ↓
支付回调
    ↓
CardBalanceService.normalOrder()
    ├─ 更新余额
    ├─ 发放积分
    └─ 发放单个券
```

### 7.2 新流程（CrmDepositPlan）

```
用户选择储值方案
    ↓
用户选择储值档位
    ↓
ChargeService.commitWithPlan()（新接口）
    ├─ 查询 CrmDepositPlan
    ├─ 获取档位配置
    └─ 创建 CrmDealTask
        ├─ deposit_plan_lid = 方案ID ⭐
        ├─ deposit_tier_index = 档位索引 ⭐
        ├─ deposit_tier_snapshot = 档位快照（JSON）⭐
        ├─ principal_amount = 本金
        ├─ give_amount = 赠送金额
        └─ give_point = 赠送积分
    ↓
支付回调
    ↓
CardBalanceService.processDepositPlan()（新方法）
    ├─ 读取档位快照
    ├─ 更新余额
    ├─ 发放积分
    ├─ 批量发放券 ⭐
    ├─ 批量发放券包 ⭐
    └─ 升级会员等级 ⭐
```

### 7.3 兼容性处理

```java
// 在支付回调时判断使用哪种流程
if (task.getDepositPlanLid() != null) {
    // 新流程：使用 CrmDepositPlan
    processWithDepositPlan(task);

    // 旧流程：使用 CrmSaveRule
    processWithSaveRule(task);
```

## 8. 快照机制

### 8.1 为什么需要快照？

**问题：** 如果储值方案在用户支付后被修改，会影响已提交的任务吗？

**解决方案：** 使用快照机制，在创建任务时保存档位配置的完整副本。

### 8.2 快照字段：deposit_tier_snapshot

**类型：** JSON

**内容：** 档位配置的完整副本

**示例：**
```json
{
  "name": "充100送20",
  "saveAmount": 100.00,
  "giftAmount": 20.00,
  "giftPoints": 50,
  "isUpgradeMemberLevel": false,
  "upgradeMemberLevelLid": null,
  "coupons": [
    {
      "couponLid": 300001,
      "couponName": "满100减20券",
      "issueQtyPerTime": 1
    }
  ],
  "couponPackages": [
    {
      "packageLid": 400001,
      "packageName": "新人大礼包",
      "packageQty": 1
    }
  ]
}
```

### 8.3 快照的作用

```
时间线：
2026-03-01：创建储值方案A（充100送20 + 1张券）
2026-03-10：用户提交储值
    ↓
创建 CrmDealTask
    ├─ deposit_plan_lid = 方案A的ID
    ├─ deposit_tier_index = 0
    └─ deposit_tier_snapshot = 档位配置快照 ✅
    ↓
2026-03-15：修改储值方案A（改为充100送30 + 2张券）
    ↓
2026-03-20：用户完成支付
    ↓
支付回调
    ↓
读取 deposit_tier_snapshot（快照）
    ↓
按照快照内容发放（充100送20 + 1张券）✅
    ↓
不受方案修改影响 ✅
```

## 9. 防重机制

### 9.1 订单级别防重

**机制：** 使用 Redis 缓存订单处理状态

**代码位置：** `CardBalanceService.executeInner` 第116-118行

```java

    return NmsResult.ok("redis中有相应记录，说明订单已被处理过");

```

**Key 格式：** `order:processed:{lid}`

**过期时间：** 2小时

## 10. 统计和报表

**作用：** 提高统计查询性能

**示例：**
```sql

WHERE report_date = '2026-03-11'
  AND operation_model = 'Charge'
  AND trade_state = 'SUCCESS';
```

### 10.2 常用统计查询

**储值统计：**
```sql
SELECT
    DATE(success_time) as date,
    COUNT(*) as count,
    SUM(principal_amount) as total_principal,
    SUM(give_amount) as total_give
FROM crm_deal_task
WHERE operation_model = 'Charge'
  AND trade_state = 'SUCCESS'
  AND success_time BETWEEN '2026-03-01' AND '2026-03-31'
GROUP BY DATE(success_time);
```

**消费统计：**
```sql
SELECT
    DATE(success_time) as date,
    COUNT(*) as count,
    SUM(ABS(amount)) as total_amount
FROM crm_deal_task
WHERE operation_model = 'Consume'
  AND trade_state = 'SUCCESS'
  AND success_time BETWEEN '2026-03-01' AND '2026-03-31'
```

**储值方案统计（新）：**

## 11. 总结

### 11.1 CrmDealTask 的核心作用

1. **交易状态持有者**
   - 保存交易从提交到完成的所有信息
   - 跟踪交易状态变化

2. **支付回调的数据源**
   - 支付回调时查询任务信息
   - 根据任务信息更新会员余额

3. **快照机制的载体**
   - 保存档位配置快照
   - 防止方案修改影响已提交任务

4. **防重机制的基础**
   - 通过状态检查防止重复处理
   - 通过 Redis 缓存防止并发问题

5. **统计报表的数据源**
   - 提供储值/消费统计数据
   - 支持按日期、方案、门店等维度统计

### 11.2 设计优点

- ✅ 状态清晰，易于追溯
- ✅ 支持异步支付流程
- ✅ 快照机制保证数据一致性
- ✅ 防重机制保证幂等性
- ✅ 支持新旧流程兼容

### 11.3 设计缺点

- ⚠️ 表数据量大，需要定期归档
- ⚠️ JSON 字段查询性能较差
- ⚠️ 缺少版本管理机制

### 11.4 改进建议

1. **数据归档**
   - 定期将历史数据归档到历史表
   - 保留近期数据在主表

2. **索引优化**
   - 增加复合索引：`(mid, trade_state, success_time)`
   - 增加分区：按 `report_date` 分区

3. **版本管理**
   - 增加 `version` 字段
   - 记录任务修改历史
