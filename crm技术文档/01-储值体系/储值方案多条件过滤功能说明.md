# 储值方案多条件过滤功能说明

## 更新日期
2026-03-12

## 功能概述
为储值方案列表查询接口添加了多条件过滤功能，支持按活动名称、类型、状态、适用组织、品牌、有效期等多个维度进行筛选。

## 新增过滤条件

### 1. 方案类型（planType）
**字段名称：** `planType`  
**数据类型：** `DepositPlanTypeEnum`  
**可选值：**
- `1` - 储值分期送礼
- `2` - 智能储值
- `3` - 储值套餐

**查询逻辑：** 精确匹配

**示例：**
```json
{
  "planType": 1
}
```

### 2. 品牌过滤（brandLids）
**字段名称：** `brandLids`  
**数据类型：** `List<Long>`  
**说明：** 查询包含指定品牌的储值方案

**查询逻辑：** 
- 支持多个品牌ID
- 使用OR逻辑，只要方案包含任一指定品牌即返回
- 查询JSON字段 `brand_rule.brandIds`

**SQL实现：**
```sql
JSON_CONTAINS(brand_rule->'$.brandIds', JSON_ARRAY(?))
```

**示例：**
```json
{
  "brandLids": [1001, 1002, 1003]
}
```

### 3. 门店过滤（storeLids）
**字段名称：** `storeLids`  
**数据类型：** `List<Long>`  
**说明：** 查询适用于指定门店的储值方案

**查询逻辑：**
- 支持多个门店ID
- 使用OR逻辑，只要方案适用于任一指定门店即返回
- 同时返回"全部门店"类型的方案
- 查询JSON字段 `applicable_orgs.storeIds`

**SQL实现：**
```sql
(applicable_orgs->>'$.type' = 'ALL' 
 OR JSON_CONTAINS(applicable_orgs->'$.storeIds', JSON_ARRAY(?)))
```

**示例：**
```json
{
  "storeLids": [2001, 2002]
}
```

### 4. 有效期范围过滤（validityStartDate / validityEndDate）
**字段名称：** 
- `validityStartDate` - 有效期查询开始日期
- `validityEndDate` - 有效期查询结束日期

**数据类型：** `LocalDateTime`  
**说明：** 查询在指定日期范围内有效的储值方案

**查询逻辑：**
- 查询方案的有效期与指定日期范围有交集
- 方案的结束时间 >= 查询开始时间
- 方案的开始时间 <= 查询结束时间

**SQL实现：**
```sql
end_time >= ? AND begin_time <= ?
```

**示例：**
```json
{
  "validityStartDate": "2026-03-01T00:00:00",
  "validityEndDate": "2026-03-31T23:59:59"
}
```

**场景说明：**
- 查询3月份有效的方案
- 包括：
  - 3月1日开始，3月31日结束的方案
  - 2月开始，4月结束的方案（跨月）
  - 3月中旬开始的方案
  - 3月中旬结束的方案

## 技术实现细节

### 应用层过滤（当前实现）

为了避免JSON字段查询的性能问题，品牌和门店过滤采用应用层过滤方式：

**实现流程：**
1. 数据库查询基础条件（商户、状态、分类、有效期等）
2. 在Java代码中过滤品牌和门店
3. 重新组装分页结果

**品牌过滤逻辑：**
```java
// 应用层过滤：品牌
if (request.getBrandLids() != null && !request.getBrandLids().isEmpty()) {
  final Set<Long> brandLidSet = new HashSet<>(request.getBrandLids());
  list = list.stream()
      .filter(plan -> {
        // 品牌规则为空，不匹配
        if (plan.getBrandRule() == null) {
          return false;
        }
        // 全部品牌类型，匹配所有
        if (BrandRuleType.ALL == plan.getBrandRule().getType()) {
          return true;
        }
        // 指定品牌类型，检查是否有交集
        if (plan.getBrandRule().getBrandIds() == null) {
          return false;
        }
        return plan.getBrandRule().getBrandIds().stream()
            .anyMatch(brandLidSet::contains);
      })
      .collect(Collectors.toList());
}
```

**门店过滤逻辑：**
```java
// 应用层过滤：门店
if (request.getStoreLids() != null && !request.getStoreLids().isEmpty()) {
  final Set<Long> storeLidSet = new HashSet<>(request.getStoreLids());
  list = list.stream()
      .filter(plan -> {
        // 门店规则为空，不匹配
        if (plan.getApplicableOrgs() == null) {
          return false;
        }
        // 全部门店类型，匹配所有
        if (StoreRuleType.ALL == plan.getApplicableOrgs().getType()) {
          return true;
        }
        // 指定门店类型，检查是否有交集
        if (StoreRuleType.SPECIFIED_STORES == plan.getApplicableOrgs().getType()) {
          if (plan.getApplicableOrgs().getStoreIds() == null) {
            return false;
          }
          return plan.getApplicableOrgs().getStoreIds().stream()
              .anyMatch(storeLidSet::contains);
        }
        return false;
      })
      .collect(Collectors.toList());
}
```

**优点：**
- 避免JSON字段查询的性能问题
- 代码逻辑清晰，易于维护
- 支持复杂的业务规则（如全部品牌、全部门店）

**适用场景：**
- 储值方案数量 < 1000条
- 查询频率不是特别高
- 需要灵活的过滤逻辑

### 有效期范围查询
使用日期范围交集判断：

```java
if (request.getValidityStartDate() != null || request.getValidityEndDate() != null) {
  query.and(wrapper -> {
    if (request.getValidityStartDate() != null) {
      // 方案的结束时间 >= 查询开始时间
      wrapper.ge(CrmDepositPlan::getEndTime, request.getValidityStartDate());
    }
    if (request.getValidityEndDate() != null) {
      // 方案的开始时间 <= 查询结束时间
      wrapper.le(CrmDepositPlan::getBeginTime, request.getValidityEndDate());
    }
  });
}
```

## 性能优化建议

### 1. 当前实现的性能特点
- 数据库查询：使用普通字段索引，性能良好
- 应用层过滤：适合数据量 < 1000条
- 内存占用：单次查询加载一页数据（默认20条）

**方案A：添加缓存**

```java
  // ... 现有逻辑
}
```

**方案B：使用关联表（数据量 > 1000条时）**
- 创建 `crm_deposit_plan_brand` 关联表
- 创建 `crm_deposit_plan_store` 关联表
- 使用JOIN或子查询，利用索引

**方案C：使用虚拟列 + 多值索引（MySQL 8.0.17+）**

```sql
ADD COLUMN brand_ids_virtual JSON

  (CAST(brand_ids_virtual AS UNSIGNED ARRAY))
);
```

### 2. 查询优化
- 避免同时使用过多过滤条件
- 优先使用普通字段过滤（状态、分类、方案类型）
- 品牌和门店过滤在应用层执行，性能良好
- 合理设置分页大小（建议20-50条）

### 3. 缓存策略
对于热门查询条件，可以考虑添加缓存：
- 按品牌查询的结果（缓存5分钟）
- 按门店查询的结果（缓存5分钟）

### 4. 监控建议
- 监控单次查询返回的数据量
- 如果单页数据 > 100条，考虑优化查询条件
- 如果总数据量 > 1000条，考虑使用关联表方案

## 注意事项

### 1. 应用层过滤的特点
- 品牌和门店过滤在Java代码中执行
- 先从数据库查询基础条件，再在内存中过滤
- 适合数据量 < 1000条的场景
- 性能稳定，不依赖JSON索引

### 2. 日期范围查询
- 永久有效的方案（date_type=2）的日期范围是1970-01-01到2099-12-31
- 查询时会包含这些永久有效的方案

### 3. 门店过滤逻辑
- `type=ALL` 的方案会被所有门店查询返回
- `type=SPECIFIED_STORES` 的方案只返回给指定门店
- `type=SPECIFIED_REGIONS` 的方案需要额外的区域判断逻辑（暂不支持）

### 4. 品牌过滤逻辑
- `type=ALL` 的方案适用于所有品牌
- `type=SPECIFIED` 的方案只适用于指定品牌
- 使用交集判断：只要方案包含任一查询品牌即返回

### 5. 性能考虑
- 当储值方案数量 < 1000条时，应用层过滤性能良好
- 如果数据量增长，建议切换到关联表方案
- 可以通过缓存进一步提升性能
