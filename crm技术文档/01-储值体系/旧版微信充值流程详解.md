# Legacy WeChat Charge Flow

## Scope

This document describes the old member WeChat recharge flow based on `/charge/*` and `CrmSaveRule`.

It focuses on:

- Request and processing path for old WeChat recharge
- Core classes and methods involved
- Table writes and field changes by stage
- Differences between "submit order" and "balance actually credited"
- SQL snippets for troubleshooting

The new `/deposit-plan/charge/*` flow is not the subject of this document.

## Entry Points

HTTP entry:

- `/charge/commit`
- `/charge/query_charge_status`
- `/charge/success`

Main files:

- `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-app/src/main/java/com/nms4cloud/crm/app/controller/charge/CrmChargeController.java`
- `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/ChargeService.java`
- `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-app/src/main/java/com/nms4cloud/crm/app/task/CrmOrderCloseQueueConsumer.java`
- `nms4cloud-app/2_business/nms4cloud-crm/nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/balance/CardBalanceService.java`

## High-Level Flow

The old flow is split into two phases:

1. Submit payment order
2. Consume payment success and actually credit member balance

This means:

- `/charge/commit` does not directly change member balance
- It creates a pending recharge task and calls the payment service
- Real balance update happens later in MQ consumer logic

## Request Model

Request DTO:

- `ChargeCommitRequest`

Important fields:

- `ruleType`
- `ruleIds`
- `couponLid`
- `billAmount`
- `orderId`
- `billId`
- `h5`
- `notCheckOut`

Meaning of `ruleType`:

- `0`: normal recharge rules
- `-1`: petty recharge
- `-2`: overlord meal recharge

Important consequence:

- The old API can match multiple `CrmSaveRule` rows in one request
- Payment amount is the sum of all matched `saveAmount`
- The real rule-level breakdown is stored in `crm_deal_task_item`

## Full Sequence

### 1. HTTP submit

Client calls:

- `POST /charge/commit`

Controller delegates to:

- `CrmChargeController.commit()`
- `ChargeService.commit()`

Inside `ChargeService.commit()`:

1. Get `CUserVO` from login session
2. Force `request.mid = user.mid`
3. Validate `sid`
4. Validate member card existence from login state
5. Load rules by `ruleType`

### 2. Rule loading

There are three loading branches.

#### 2.1 Normal recharge: `ruleType = 0`

- Load rules from `crm_save_rule` by `ruleIds`
- If requested rule count and loaded rule count do not match, return error
- If a rule is an upgrade cost rule, extra validation is performed

#### 2.2 Petty recharge: `ruleType = -1`

- Requires `billAmount`
- Requires `orderId`
- Can dynamically load petty recharge rules based on `billAmount`
- Can also combine dynamic petty rules with additional `ruleIds`

#### 2.3 Overlord recharge: `ruleType = -2`

- Requires `billAmount`
- Requires `orderId`
- If `notCheckOut = true`, requires `billId`
- Loads overlord rules from member context

## Why Old Flow Can Use Multiple Rules

The old design is intentionally multi-rule:

- Request field is `Set<Long> ruleIds>`
- Query uses `IN (...)`
- A single payment task creates multiple `crm_deal_task_item` rows
- Downstream coupon, points, and overlord count logic iterates each item separately

So the old model is not "one payment equals one rule".

It is closer to:

- one payment
- one `crm_deal_task`
- many `crm_deal_task_item`

## Payment Request Phase

### 3. Total amount calculation

In `ChargeService.commit()`:

- `total = sum(rules.saveAmount) * 100`

This means:

- Amount unit in rules is yuan
- Amount passed to payment is fen

### 4. Build payment attach

`ChargeNotifyAttach` carries old-flow context:

- `cardLid`
- `cardId`
- `phone`
- `name`
- `openId`
- `orderId`
- `billId`
- `notCheckOut`

The old flow does not set:

- `depositPlan`
- `planLid`
- `tierIndex`

Those belong to the new deposit plan flow.

### 5. Call payment service

Payment service methods:

- `dealReactiveFeign.jsPay()`
- `dealReactiveFeign.miniPay()`

Common request fields:

- `mid`
- `sid`
- `subAppid`
- `openId`
- `totalFee`
- `terminalTrace = tradeNo`
- `dealType = RECHARGE`
- `notifyUrl = CRM_ORDER_CLOSE_QUEUE`
- `attach = JSON(ChargeNotifyAttach)`

### 6. Persist pending recharge task

After payment order is created successfully:

- insert one row into `crm_deal_task`
- insert one or more rows into `crm_deal_task_item`
- send delayed MQ message to `CRM_ORDER_CLOSE_QUEUE`

At this stage, member balance is still unchanged.

## Database Changes by Stage

## Stage A: Before payment success

### Table: `crm_deal_task`

Inserted in `ChargeService.createDealTask()`.

Important initial fields:

- `mid`
- `sid`
- `shop_name`
- `lid = tradeNo`

- `card_lid`

- `card_type_code`
- `as_upgrade_cost`
- `unpaid`
- `principal_amount`

- `comment`
- `dash_amount`
- `pay_way_code = -1`

Notes:

- `amount` is not finalized here
- `amount` is written later after actual balance processing

Important fields:

- `lid`

- `card_lid`

- `give_point`
- `first_charge_gift_coupon`
- `give_coupon`
- `give_coupon_id`
- `give_coupon_num`

No member balance table is changed in Stage A.

## Stage B: Payment success consumed by MQ

Entry:

- `CrmOrderCloseQueueConsumer.onMessage()`

Processing logic:

1. Parse MQ payload
2. Check Redis idempotency key
3. Query `crm_deal_task`
4. Query payment result if needed
5. If not paid, close payment order and mark task closed
6. If paid, execute `CardBalanceService.execute()`

### Table: `crm_deal_task`

If payment is not successful after delay:

- `trade_state` becomes `CLOSED`

If payment is successful and old flow proceeds:

- `trade_state` becomes `SUCCESS`
- `balance` is updated to credited card balance
- `amount` is written
- `principal_amount` may be corrected
- `give_amount` may be corrected
- `give_point` may be corrected
- `dash_amount` may be corrected
- `unpaid`
- `give_unpaid`
- `unpaid_at`

Important point:

- `crm_deal_task` is the source-of-truth status table for payment processing
- `USERPAYING` means waiting for payment result or waiting for MQ processing
- `SUCCESS` means credited
- `CLOSED` means payment not successful and order closed
- `REVOKED` means recharge reversed later

### Table: `crm_card`

This is the member summary balance table.

`CardBalanceService.changeBalance()` updates:

- `balance`
- `principal_balance`
- `give_balance`
- `dash_balance`
- `points`
- `sum_of_points`
- `sum_of_save_times`
- `sum_of_save`
- `sum_of_save_give`
- `invoice_balance`
- `unpaid_amount`
- `unpaid_give_amount`
- `unpaid_at`

Depending on branch, later async processing may also update:

- `card_type_code`
- `card_type`
- `card_type_level_code`
- `card_type_level`
- `first_gift_coupon_done`
- `rebate_rule_lid`
- `rebate_ratio`

## Failure Path

If MQ consumer determines payment is not successful:

1. Call payment close API
2. Update `crm_deal_task.trade_state = CLOSED`

No balance credit happens.

Tables not changed in this path:

- `crm_card`
- `crm_card_balance`
- `crm_card_record`
- `crm_card_points_record`

## Cancel/Revoke Path

Although not part of initial crediting, it is important for understanding later state changes.

Method:

- `cancelOrder()`

Tables and changes:

### `crm_card`

Reverse:

- `balance`
- `principal_balance`
- `give_balance`
- `dash_balance`
- `points`
- cumulative save/consume fields
- unpaid related fields

### `crm_card_balance`

Reverse by merging negative balance rows back into buckets.

### `crm_card_record`

- mark original rows: `is_cancel = true`
- insert reversal ledger rows

### `crm_card_points_record`

- insert negative points record

### `crm_deal_task`

Updated fields:

- `trade_state = REVOKED`
- `canceled = true`
- `cancel_time`
- `cancel_by`

### `crm_coupon_order`

Coupon revoke logic updates:

- `coupon_status = YZF`
- `abandon_staff`
- `abandon_time`

Condition:

- `order_bill_id = taskLid`

### `crm_overlord_once`

Reverse count:

- `times = times - 1`

### `crm_rebate_detail`

Mark rebate detail as cancelled.

### `crm_card`

If rebate relation changed on recharge, restore:

- `rebate_rule_lid`
- `rebate_ratio`

## State Meaning Cheat Sheet

### `crm_deal_task.trade_state`

- `USERPAYING`: payment created, not yet credited
- `SUCCESS`: payment credited successfully
- `CLOSED`: unpaid or payment failed and closed
- `REVOKED`: recharge later reversed

### Old flow success criteria

Usually all of the following should hold:

- `crm_deal_task.trade_state = SUCCESS`
- `crm_card_record` exists for `task_lid`
- `crm_card` balance fields have changed
- Redis order processed key exists

### Common half-done patterns

#### Only task exists, no card record

Meaning:

- payment order was created
- but balance credit did not finish

#### `crm_deal_task.trade_state = USERPAYING`

Meaning:

- waiting for payment result
- or MQ consumer has not completed processing

#### `crm_deal_task.trade_state = CLOSED`

Meaning:

- payment timed out or not successful

#### Task success, no coupon order

Meaning:

- main balance credit completed
- async coupon issue failed later

## Troubleshooting SQL

Replace:

- `@mid`
- `@taskLid`
- `@cardLid`
- `@outTradeNo`

```sql
SET @mid = 10001;
SET @taskLid = 190000000000000001;
SET @cardLid = 200000000000000001;
SET @outTradeNo = '420000xxxxxxxxxxxx';
```

### 1. Main task

```sql
SELECT
  mid, sid, lid, card_lid, operation_model, trade_state, out_trade_no,
  principal_amount, give_amount, amount, give_point, save_rule, save_rule_code,
  as_upgrade_cost, unpaid, give_unpaid, unpaid_at, balance, comment,
  created_by, created_time, updated_time
FROM crm_deal_task
WHERE mid = @mid
  AND lid = @taskLid;
```

### 2. Task items

```sql
SELECT
  mid, sid, lid, task_lid, operation_model, save_rule, save_rule_code,
  rule_type_lid, principal_amount, give_amount, amount, give_point,
  give_coupon, give_coupon_id, give_coupon_num, first_charge_gift_coupon, created_time
FROM crm_deal_task_item
WHERE mid = @mid
  AND task_lid = @taskLid
ORDER BY created_time, lid;
```

### 3. Task by payment order number

```sql
SELECT
  mid, sid, lid, trade_state, out_trade_no, card_lid,
  principal_amount, give_amount, amount, created_time
FROM crm_deal_task
WHERE mid = @mid
  AND out_trade_no = @outTradeNo
ORDER BY created_time DESC;
```

### 4. Card summary

```sql
SELECT
  mid, sid, lid, id AS card_no, member, phone,
  card_type, card_type_code, card_type_level, card_type_level_code,
  balance, principal_balance, give_balance, dash_balance, invoice_balance,
  unpaid_amount, unpaid_give_amount, unpaid_at, points,
  sum_of_save_times, sum_of_save, sum_of_save_give,
  sum_of_consume, sum_of_consume_give,
  first_gift_coupon_done, rebate_rule_lid, rebate_ratio, updated_time
FROM crm_card
WHERE mid = @mid
  AND lid = @cardLid;
```

### 5. Card ledger

```sql
SELECT
  company_id, shop_id, lmnid, task_lid, operation_model, name,
  save_rule, save_rule_code, principal_amount, give_amount, amount, give_point,
  balance_before, balance_after,
  principal_amount_before, principal_amount_after,
  give_amount_before, give_amount_after,
  order_bill_id, out_order_bill_id, pay_way, is_cancel, create_time
FROM crm_card_record
WHERE company_id = @mid
  AND task_lid = @taskLid
ORDER BY create_time, lmnid;
```

### 6. Points ledger

```sql
SELECT
  company_id, shop_id, lmnid, operation_model, amount,
  balance_before, balance_after,
  save_rule, save_rule_code, order_bill_id,
  member_name, phone, create_time
FROM crm_card_points_record
WHERE company_id = @mid
  AND order_bill_id = @outTradeNo
ORDER BY create_time, lmnid;
```

### 7. Balance buckets

```sql
SELECT
  mid, sid, lid, cno, total, principal, gift, source, out_trade_no,
  created_time, updated_time
FROM crm_card_balance
WHERE mid = @mid
  AND cno = @cardLid
ORDER BY created_time, lid;
```

### 8. Issued coupon orders

```sql
SELECT
  company_id, shop_id, lmnid, coupon, coupon_code, pkg_coupon, pkg_coupon_code,
  coupon_status, get_channel, order_bill_id, member, member_code, unionid,
  valid_date, deadline, get_date, abandon_time, abandon_staff
FROM crm_coupon_order
WHERE company_id = @mid
  AND order_bill_id = CAST(@taskLid AS CHAR)
ORDER BY get_date, lmnid;
```

### 9. Overlord / petty counts

```sql
SELECT
  mid, sid, lid, card_lid, overlord_lid, type_, times, created_time, updated_time
FROM crm_overlord_once
WHERE mid = @mid
  AND card_lid = @cardLid
ORDER BY updated_time DESC, lid DESC;
```

### 10. Upgrade cost records

```sql
SELECT
  mid, sid, lid, task_lid, id AS card_no, name, phone,
  rule_name, rule_lid, org_type_lid, org_type, card_type_lid, card_type,
  amount, points, coupon_lid, coupon, coupon_num,
  cancel, cancel_by, cancel_at, created_time
FROM crm_card_upgrade_record
WHERE mid = @mid
  AND task_lid = @taskLid
ORDER BY created_time, lid;
```

### 11. Rebate detail

```sql
SELECT
  mid, sid, lid, consume_card_lid, consume_phone, consume_card_id, consume_member_name,
  consume_amount, rule_lid, rebate_card_lid, rebate_phone,
  rebate_amount, rebate_ratio, done, done_at,
  cancel, cancel_at, cancel_by, created_time
FROM crm_rebate_detail
WHERE mid = @mid
  AND lid = @taskLid
ORDER BY created_time;
```

## Practical Reading Order

When troubleshooting one old WeChat recharge, read in this order:

1. `crm_deal_task`
2. `crm_deal_task_item`
3. `crm_card`
4. `crm_card_record`
5. `crm_card_balance`
6. `crm_card_points_record`
7. `crm_coupon_order`
8. `crm_overlord_once`
9. `crm_card_upgrade_record`
10. `crm_rebate_detail`

If you only need one quick success verdict, the fastest combination is:

- `crm_deal_task.trade_state`
- `crm_card_record` existence by `task_lid`
- `crm_card.balance/principal_balance/give_balance`
