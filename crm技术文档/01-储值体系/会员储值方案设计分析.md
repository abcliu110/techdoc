# 会员储值方案（CrmDepositPlan）设计分析

## 1. 概述

会员储值方案（CrmDepositPlan）是新设计的储值配置系统，用于替代旧的储值规则（CrmSaveRule）。它使用 JSON 字段存储复杂的业务规则，提供更灵活的配置能力。

**设计日期：** 2026年
**当前状态：** 已实现，但未集成到储值流程中
**目标：** 替代旧的 CrmSaveRule，统一储值配置

## 3. JSON规则详解

### 3.1 储值档位规则（DepositTierRule）- 核心

**字段：** `tier_rule`

**结构：**
```json
{
  "tiers": [
    {
      "name": "档位名称",
      "imageUrl": "档位图片",
      "recommendTitle": "推荐标题",
      "saveAmount": 100.00,
      "giftAmount": 20.00,
      "giftPoints": 50,
      "isUpgradeMemberLevel": false,
      "upgradeMemberLevelLid": null,
      "isFirstChargeOnly": false,
      "coupons": [
        {
          "couponLid": 300001,
          "couponCode": "COUPON001",
          "couponName": "优惠券1",
          "couponType": 1,
          "issueQtyPerTime": 1,
          "categoryName": "通用券",
          "validDays": 30,
          "effectiveTime": "2026-01-01T00:00:00",
          "countInAmount": 1,
          "sortNo": 1
        }
      ],
      "couponPackages": [
        {
          "packageLid": 400001,
          "packageCode": "PKG001",
          "packageName": "券包1",
          "categoryName": "通用券包",
          "packageQty": 1,
          "countInAmount": 1,
          "sortNo": 1
        }
      ],
      "sortNo": 1
    }
  ]
}
```

**说明：**
- `saveAmount`: 储值金额（本金）
- `giftAmount`: 赠送金额
- `giftPoints`: 赠送积分
- `isUpgradeMemberLevel`: 是否升级会员等级
- `upgradeMemberLevelLid`: 升级到的会员等级ID
- `isFirstChargeOnly`: 是否仅首充可用
- `coupons`: 赠送的优惠券列表
- `couponPackages`: 赠送的券包列表

### 3.2 周期规则（CycleRule）

**字段：** `cycle_rule`

**结构：**
```json
{
  "enabled": true,
  "weekdays": [1, 2, 3, 4, 5]
}
```

**说明：**
- `enabled`: 是否启用周期规则
- `weekdays`: 可用的星期几（1-7，1=周一，7=周日）

### 3.3 时段规则（TimePeriodRule）

**字段：** `time_periods`

**结构：**
```json
{
  "enabled": true,
  "periods": [
    {
      "startTime": "09:00:00",
      "endTime": "12:00:00"
    },
    {
      "startTime": "14:00:00",
      "endTime": "18:00:00"
    }
  ]
}
```

**说明：**
- `enabled`: 是否启用时段规则
- `periods`: 可用时段列表

### 3.4 渠道规则（ChannelRule）

**字段：** `channels`

**结构：**
```json
{
  "enabled": true,
  "channelCodes": ["MINI_PROGRAM", "H5", "APP"]
}
```

**说明：**
- `enabled`: 是否启用渠道规则
- `channelCodes`: 可用渠道代码列表

### 3.5 门店规则（StoreRule）

**字段：** `applicable_stores`

**结构：**
```json
{
  "enabled": true,
  "storeIds": [1001, 1002, 1003]
}
```

**说明：**
- `enabled`: 是否启用门店规则
- `storeIds`: 可用门店ID列表

### 3.6 品牌规则（BrandRule）

**字段：** `brand_rule`

**结构：**
```json
{
  "enabled": true,
  "brandIds": [2001, 2002]
}
```

**说明：**
- `enabled`: 是否启用品牌规则
- `brandIds`: 可用品牌ID列表

### 3.7 会员等级规则（MemberLevelRule）

**字段：** `member_level_rule`

**结构：**
```json
{
  "enabled": true,
  "memberPlanLids": [3001],
  "memberLevelLids": [3101, 3102]
}
```

**说明：**
- `enabled`: 是否启用会员等级规则
- `memberPlanLids`: 可用会员方案ID列表
- `memberLevelLids`: 可用会员等级ID列表

### 3.8 购买限制规则（PurchaseLimitRule）

**字段：** `purchase_limit_rule`

**结构：**
```json
{
  "enabled": true,
  "limitType": "PERSON_CYCLE",
  "limitValue": 1,
  "cycleType": "MONTH",
  "totalLimit": 1000
}
```

**说明：**
- `enabled`: 是否启用购买限制
- `limitType`: 限制类型（PERSON_CYCLE-单人周期限制，TOTAL-总量限制）
- `limitValue`: 单人周期限制值
- `cycleType`: 周期类型（DAY-天，WEEK-周，MONTH-月）
- `totalLimit`: 总量限制值

## 4. 核心业务逻辑

### 4.1 查询可用方案（listAvailable）

**方法：** `CrmDepositPlanServicePlus.listAvailable()`

**过滤条件（按顺序）：**

1. **基础过滤**
   - 状态必须为启用（status = 1）
   - 如果指定渠道，必须匹配渠道规则

2. **有效期检查**
   - 当前时间必须在 beginTime 和 endTime 之间

3. **周期规则检查**
   - 如果启用，当前星期几必须在 weekdays 列表中

4. **排除日期检查**
   - 当前日期不能在 excludeDates 列表中

5. **时段规则检查**
   - 如果启用，当前时间必须在某个 period 范围内

6. **门店规则检查**
   - 如果启用，当前门店ID必须在 storeIds 列表中

7. **品牌规则检查**
   - 如果提供品牌ID且启用规则，品牌ID必须在 brandIds 列表中

8. **会员等级规则检查**
   - 如果提供会员等级且启用规则，会员方案和等级必须匹配

**代码位置：** `CrmDepositPlanServicePlus.java` 第175-233行

### 4.2 新增方案（add）

**方法：** `CrmDepositPlanServicePlus.add()`

**验证步骤：**

1. 验证必填字段
2. 验证协议ID列表
3. 验证活动分类是否存在
4. 验证名称唯一性
5. 验证日期类型和日期字段的一致性
6. 验证周期规则
7. 验证时段规则
8. 验证渠道规则
9. 验证门店规则
10. 验证品牌规则
11. 验证会员等级规则
12. 验证购买限制规则
13. 验证金额限制规则
14. **验证储值档位规则**（核心）

**代码位置：** `CrmDepositPlanServicePlus.java` 第238-307行

### 4.3 储值档位规则验证（validateTierRule）

**验证内容：**

1. **档位基本验证**
   - 至少有一个档位
   - 档位名称不能为空
   - 储值金额必须大于0

2. **赠送内容验证**
   - 赠送金额 >= 0
   - 赠送积分 >= 0

3. **会员升级验证**
   - 如果启用升级，必须指定升级等级ID

4. **优惠券验证**
   - 券ID必须存在
   - 券必须已审核
   - 券未终止
   - 发放数量 > 0

5. **券包验证**
   - 券包ID必须存在
   - 券包必须已审核
   - 券包未终止
   - 发放数量 > 0

## 5. 与旧系统的对比

### 5.1 CrmSaveRule vs CrmDepositPlan

| 特性 | CrmSaveRule（旧） | CrmDepositPlan（新） |
|------|------------------|---------------------|
| 数据结构 | 扁平化字段 | JSON字段存储规则 |
| 储值档位 | 单一档位 | 多档位支持 |
| 赠送内容 | 固定金额+积分+单个券 | 金额+积分+多个券+券包 |
| 有效期控制 | 简单的开始/结束时间 | 支持周期、时段、排除日期 |
| 渠道控制 | 无 | 支持多渠道配置 |
| 门店控制 | 无 | 支持多门店配置 |
| 品牌控制 | 无 | 支持品牌限制 |
| 会员等级控制 | 简单 | 支持方案+等级双重控制 |
| 购买限制 | 简单 | 支持单人周期+总量双重限制 |
| 可选券 | 支持 | 未明确（需确认） |
| 首充限制 | 支持 | 支持（档位级别） |
| 会员升级 | 支持 | 支持（档位级别） |

### 5.2 数据迁移考虑

**从 CrmSaveRule 迁移到 CrmDepositPlan：**

1. **基础信息映射**
   - name → name
   - saveAmount → tierRule.tiers[0].saveAmount
   - giveAmount → tierRule.tiers[0].giftAmount
   - givePoint → tierRule.tiers[0].giftPoints

2. **赠券信息映射**
   - giveCouponId → tierRule.tiers[0].coupons[0].couponLid
   - giveCouponNum → tierRule.tiers[0].coupons[0].issueQtyPerTime
   - giveCouponIsMap → 判断是券还是券包

3. **会员升级映射**
   - upgradeCardTypeCode → tierRule.tiers[0].upgradeMemberLevelLid
   - asUpgradeCost → tierRule.tiers[0].isUpgradeMemberLevel

4. **首充限制映射**
   - firstChargeGiftCoupon → tierRule.tiers[0].isFirstChargeOnly

5. **新增功能**
   - 周期规则：需要新配置
   - 时段规则：需要新配置
   - 渠道规则：需要新配置
   - 门店规则：需要新配置

## 8. 技术要点

### 8.1 JSON字段处理

**使用 MyBatis-Plus 的 JacksonTypeHandler：**

```java
@TableField(value = "tier_rule", typeHandler = JacksonTypeHandler.class)
private DepositTierRule tierRule;
```

**优点：**
- 自动序列化/反序列化
- 类型安全
- 易于维护

**缺点：**
- 查询性能较差
- 不支持复杂的JSON查询

### 8.2 规则验证

**分层验证：**
1. 字段级验证（非空、格式）
2. 业务级验证（唯一性、存在性）
3. 规则级验证（逻辑一致性）

### 8.3 可用方案过滤

**性能考虑：**
- 先查询所有启用的方案
- 在内存中进行规则过滤
- 适合方案数量不多的场景

**优化方向：**
- 使用缓存减少数据库查询
- 使用索引优化查询性能
