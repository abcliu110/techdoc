# 微信消息模板定义与发送机制

## 1. 文档目的

本文说明 nms4cloud 系统中微信消息模板是如何定义、初始化、启用和发送的，重点覆盖 CRM 相关的公众号模板消息与小程序订阅消息。

本文基于以下代码路径分析：

- `nms4cloud-app/1_platform/nms4cloud-wechat/`
- `nms4cloud-app/2_business/nms4cloud-crm/`
- `nms4cloud-app/3_customer/nms4cloud-order/`
- `nms4cloud-app/2_business/nms4cloud-biz/`
- `nms4cloud-app/2_business/nms4cloud-product/`

## 2. 总体结论

系统中的微信消息模板不是由业务模块直接保存真实微信 `template_id` 使用，而是分为三层：

1. 平台模板库：由 `tmpl.json` 初始化到 `wx_msg_tmpl`、`wx_msg_tmpl_item`，`mid = PLATFORM_MID`。
2. 商户模板实例：商户启用模板时，根据平台模板向微信后台添加模板，微信返回真实 `template_id` 后保存到商户自己的 `wx_msg_tmpl`。
3. 业务代码引用：业务代码通常只引用短模板编号 `tid`，例如 `WechatConsts.TmplMsgTid.ORDER_SUCCESS = "43747"`。

发送时，业务模块组装 `WxMpTemplateMessage`，把 `templateId` 临时放成业务短编号 `tid`，把 `clientMsgId` 放成 `mid` 或平台 `appId`。微信服务消费 MQ 后，再把 `mid + tid` 转换成真实微信 `template_id`，最终调用微信 SDK 发送。

## 3. 核心数据模型

### 3.1 模板主表：wx_msg_tmpl

实体类：

`nms4cloud-app/1_platform/nms4cloud-wechat/nms4cloud-wechat-dao/src/main/java/com/nms4cloud/wechat/dao/entity/WxMsgTmpl.java`

表名：

`wx_msg_tmpl`

关键字段：

| 字段 | 含义 |
|---|---|
| `mid` | 商户 ID。平台模板库使用 `PLATFORM_MID`。 |
| `sid` | 门店 ID。 |
| `lid` | 模板逻辑 ID。 |
| `product_type` | 产品类型，对应 `MsgTmplProductTypeEnum`。 |
| `biz_type` | 业务类型，对应 `MsgTmplBizTypeEnum`。 |
| `template_id_short` | 微信模板库中的短模板编号，注释说明可能是 `TM**`、`OPENTMTM**` 或纯数字 ID。 |
| `tid` | 系统内常用的业务短编号。当前业务代码大量使用纯数字 `tid`。 |
| `title` | 模板标题。 |
| `disable` | 启停字段。注意发送逻辑中 `Boolean.FALSE.equals(disable)` 会直接不发，不能只按字段名理解。 |
| `template_id` | 微信后台返回的真实模板 ID，最终发送时必须使用这个值。 |
| `jump_type` | 跳转类型，对应 `MsgTmplJumpTypeEnum`。 |
| `appid` | 公众号或小程序 appId。 |
| `page_path` | 跳转页面路径。 |
| `app_type` | 应用类型，对应 `WeappTypeEnum`，用于区分公众号、小程序等。 |
| `revision` | 乐观锁版本。 |

实体类：

`nms4cloud-app/1_platform/nms4cloud-wechat/nms4cloud-wechat-dao/src/main/java/com/nms4cloud/wechat/dao/entity/WxMsgTmplItem.java`

`wx_msg_tmpl_item`

关键字段：

| 字段 | 含义 |

| `sid` | 门店 ID。 |

| `biz_type` | 业务类型。 |
| `tmlp_lid` | 关联 `wx_msg_tmpl.lid`。字段名为 `tmlp_lid`，注意拼写。 |
| `kid` | 微信关键词 ID。 |
| `name` | 关键词名称，例如订单号、商品名称、支付金额等。 |

| `content` | 微信字段格式，例如 `{{thing1.DATA}}`。 |
| `example` | 示例值。 |
| `color` | 关键词颜色，商户启用模板时可覆盖。 |

| `revision` | 乐观锁版本。 |

## 6. 平台模板库初始化

入口：

`nms4cloud-app/1_platform/nms4cloud-wechat/nms4cloud-wechat-app/src/main/java/com/nms4cloud/wechat/app/controller/WxMsgTmplController.java`

接口：

`POST /wx_msg_tmpl/init`

处理过程：

1. 控制器读取 classpath 下的 `tmpl.json`。
2. 将 JSON 解析为 `JSONArray`。
3. 调用 `WxMsgTmplServicePlus.initPlatTmpl(jsonArray)`。
4. 服务层把 JSON 里的 `pub_tmpl` 转成 `WxMsgTmpl`。
5. 把 `keyword_list.list` 转成多条 `WxMsgTmplItem`。
6. 所有平台模板主记录和关键词记录都写入 `mid = PLATFORM_MID`。
7. 初始化前会先删除平台旧模板和旧关键词，再批量插入新数据。

初始化文件：

`nms4cloud-app/1_platform/nms4cloud-wechat/nms4cloud-wechat-app/src/main/resources/tmpl.json`

`tmpl.json` 的结构来自微信公共模板库，核心结构如下：

```json
{
  "first_category_name": "餐饮服务",
  "second_category_name": "餐饮服务场所/餐饮服务管理企业",
  "pub_tmpl": {
    "class_id": "632",
    "tid": 43747,
    "title": "订单完成通知",
    "type": 4,
    "version": 24,
    "keyword_list": {
      "list": [
        {
          "kid": 1,
          "name": "first",
          "rule": "first",
          "content": "{{first1.DATA}}",
          "example": "..."
        }
      ]
    }
  }
}
```

实际文件中部分中文在当前终端显示为乱码，但代码按 `UTF-8` 读取。

## 7. tid 到业务分类的硬编码映射

平台模板初始化时，系统会通过 `WxMsgTmplServicePlus.TID_TO_TYPE` 给模板补充产品类型和业务类型。

代码位置：

`nms4cloud-app/1_platform/nms4cloud-wechat/nms4cloud-wechat-service/src/main/java/com/nms4cloud/wechat/service/WxMsgTmplServicePlus.java`

示例：

```java
TID_TO_TYPE.put(43747, Pair.of(ONLINE_RESTAURANT, TAKE_OUT));
TID_TO_TYPE.put(44466, Pair.of(MEMBER, MEMBER_REGISTER));
TID_TO_TYPE.put(47974, Pair.of(MEMBER, TRANSCATION));
TID_TO_TYPE.put(42715, Pair.of(MEMBER, OAUTH_CODE));
```

这意味着：如果新增一个平台模板，但没有补充 `TID_TO_TYPE`，平台模板仍可落库，但产品分类和业务分类可能为空，后续后台筛选、展示或业务判断会受影响。

## 8. 商户启用公众号模板

商户新增或启用模板的接口最终进入：

`WxMsgTmplServicePlus.add(WxMsgTmplAddDTO request)`

入参 DTO：

`nms4cloud-app/1_platform/nms4cloud-wechat/nms4cloud-wechat-api/src/main/java/com/nms4cloud/wechat/api/dto/WxMsgTmplAddDTO.java`

关键入参：

| 字段 | 含义 |
|---|---|
| `mid` | 商户 ID |
| `sid` | 门店 ID |
| `tid` | 要启用的短模板编号 |
| `jumpType` | 跳转类型 |
| `appid` | 跳转小程序或公众号相关 appId |
| `pagePath` | 跳转路径 |
| `items` | 关键词项配置，主要用于颜色 |
| `disabled` | 启停状态 |

处理分两种情况。

### 8.1 商户模板已存在

如果按 `mid + tid` 能查到商户模板：

1. 更新 `appid`、`pagePath`、`jumpType`、`disable`。
2. 遍历 `items`，按 `kid` 更新关键词颜色。
3. 删除本地 Redis 缓存。
4. 返回模板 VO。

### 8.2 商户模板不存在

如果商户模板不存在：

1. 从平台模板库按 `PLATFORM_MID + tid` 查 `WxMsgTmpl`。
2. 从平台关键词库按 `PLATFORM_MID + tmlp_lid` 查 `WxMsgTmplItem`。
3. 取平台关键词的 `name` 组成 `keyword_name_list`。
4. 调微信公众号接口 `TEMPLATE_API_ADD_TEMPLATE`。
5. 请求体包含：

```json
{
  "template_id_short": 43747,
  "keyword_name_list": ["first", "订单号", "商品名称"]
}
```

6. 微信返回真实 `template_id`。
7. 系统复制平台模板为商户模板，写入商户 `mid` 和真实 `templateId`。
8. 系统复制平台关键词为商户关键词，并按入参覆盖颜色。
9. 删除 Redis 缓存。

## 9. 小程序订阅消息模板

小程序订阅消息模板不是从 `tmpl.json` 初始化，而是在 `WxMsgTmplServicePlus` 里有一组内置定义：

```java
temps.add(new MaMsgTemp().setTid(340).setBizType(WAIT_IN_LINE_MA).setKids(List.of(2, 3, 6, 8, 9)));
temps.add(new MaMsgTemp().setTid(1619).setBizType(GOODS_TAKE_MA).setKids(List.of(1, 2, 5)));
temps.add(new MaMsgTemp().setTid(543).setBizType(ORDER_FINISH_MA).setKids(List.of(14, 5, 4, 6, 3)));
```

入口方法：

`WxMsgTmplServicePlus.addMaTemps(long mid, String appId)`

调用位置：

`WxMerchantConfigServicePlus` 在配置商户小程序时会调用 `wxMsgTmplServicePlus.addMaTemps(mid, appId)`。

处理逻辑：

1. 查询该商户、该小程序 appId 下是否已有对应 `tid` 的模板。
2. 没有则调用小程序订阅模板接口 `WxMaSubscribeService.addTemplate`。
3. 微信返回真实模板 ID。
4. 保存到 `wx_msg_tmpl`，`app_type = MINIAPP`，`appid = appId`。
5. 删除 `appId` 对应 Redis 缓存。

## 10. 发送时如何从 tid 转真实 templateId

业务侧发送公众号模板消息时常见写法：

```java
wxMpTemplateMessage.setClientMsgId(String.valueOf(mid));
wxMpTemplateMessage.setTemplateId(WechatConsts.TmplMsgTid.ORDER_SUCCESS);
wxMsgTmplFeign.sendTmplMsg(wxMpTemplateMessage);
```

这里：

- `clientMsgId` 临时承载 `mid`。
- `templateId` 临时承载业务短编号 `tid`。

微信服务消费 `WX_SEND_TMPL_MSG` 后：

1. 解析为 `WxMpTemplateMessage`。
2. 如果 `clientMsgId` 是平台 appId，则按 appId 直接发送。
3. 否则把 `clientMsgId` 解析为 `mid`。
4. 把 `templateId` 解析为 `tid`。
5. 调用 `sendTemplateMsg(mid, tid, openid, msg)`。
6. 从 Redis 或数据库查询 `mid + tid + appType=MP` 对应的真实 `templateId`。
7. 构造新的 `WxMpTemplateMessage`。
8. 调用微信 SDK `sendTemplateMsg`。

## 13. Redis 缓存

模板查询有两类缓存键：

1. 发送公众号模板时按 `mid + tid` 缓存：

```java
AppNamesConstants.WECHAT_SERVICE + ":msgTmpl:" + mid + ":" + tid
```

2. 查询小程序订阅消息真实模板 ID 时按 `appId` 缓存：

```java
AppNamesConstants.WECHAT_SERVICE + ":msgTmpl:" + appId
```

商户模板新增、更新、小程序模板新增后都会删除对应缓存，避免继续使用旧 `templateId` 或旧配置。

## 14. 新增模板时的正确理解

新增公众号模板通常不是直接写业务代码就可以完成，还需要确保以下数据闭环：

1. 微信公共模板信息已经进入 `tmpl.json`，或平台表中已有对应 `PLATFORM_MID + tid` 模板。
2. `WxMsgTmplServicePlus.TID_TO_TYPE` 已补充该 `tid` 到产品类型、业务类型的映射。
3. 如果业务代码要直接引用该模板，`WechatConsts.TmplMsgTid` 应增加常量。
4. 商户需要通过模板配置接口启用该模板，系统才会向微信后台添加模板并保存真实 `templateId`。
5. 业务发送时填充的 `WxMpTemplateData` 字段名必须与关键词规则匹配，例如 `thing1`、`time2`、`amount3`。
6. 如果是小程序订阅消息，需要在 `temps` 中定义 `tid + kids + bizType`，并确保商户小程序配置流程会执行 `addMaTemps`。

## 15. 常见排查路径

### 15.1 业务发了消息但微信收不到

按下面顺序排查：

1. 业务代码是否调用了 `WxMsgTmplFeign.sendTmplMsg` 或 `sendSubscribeMsg`。
2. MQ 是否进入 `WX_SEND_TMPL_MSG` 或 `WX_SEND_PLATFORM_TMPL_MSG`。
3. `clientMsgId` 是不是正确：商户消息应为 `mid`，平台消息应为 appId。
4. `templateId` 是否只是短 `tid`，并且数据库中是否有 `mid + tid` 对应真实 `template_id`。
5. `wx_msg_tmpl.disable` 当前值是否会导致不发送。
6. 用户是否有公众号 openId 或小程序订阅授权。
7. 关键词字段名是否与微信模板关键词匹配。

### 15.2 商户模板配置失败

重点看：

1. `PLATFORM_MID + tid` 平台模板是否存在。
2. 平台模板下 `wx_msg_tmpl_item` 是否有关键词项。
3. 调微信 `TEMPLATE_API_ADD_TEMPLATE` 返回的 `errcode/errmsg`。
4. 商户公众号 appId 是否已配置并可获取 `WxMpService`。

### 15.3 小程序订阅消息查不到 templateId

重点看：

1. `wx_msg_tmpl` 是否存在 `appid + tid + app_type=MINIAPP` 的记录。
2. `template_id` 是否非空。
3. `WECHAT_SERVICE:msgTmpl:{appId}` Redis hash 是否缓存了空值。
4. 商户小程序配置流程是否调用过 `addMaTemps`。

## 17. 结论

系统中微信消息模板的定义核心不是单个常量，而是由以下内容共同决定：

- `tmpl.json`：平台公共模板来源。
- `wx_msg_tmpl`：模板主定义和商户真实 `templateId`。
- `wx_msg_tmpl_item`：模板关键词定义。
- `WechatConsts.TmplMsgTid`：业务代码引用的常用短模板 ID。
- `TID_TO_TYPE`：短模板 ID 到产品/业务类型的分类映射。
- `temps`：小程序订阅消息的内置模板定义。

因此，排查或新增微信消息模板时，不能只看业务代码里的 `WechatConsts.TmplMsgTid`。必须同时确认平台模板库、商户模板实例、关键词项、真实微信 `template_id`、启停状态和发送链路。
