# AI源码功能理解方法论与多轮分析SOP

## 1. 目标

本文档用于指导 AI 在既有 `Spring Cloud + React` 信息系统中，以稳定、可追溯、可审阅的方式理解一个特定功能，并输出给人类阅读的技术文档。

目标不是“解释代码”，而是“还原功能如何工作”。

输出必须满足三个要求：

- 面向业务和技术人员都可读
- 所有关键结论可追溯到源码证据
- 能支撑后续开发、测试、排障、交接

## 2. 适用范围

适用于以下类型的既有系统功能分析：

- Spring Cloud 微服务后端
- React 前端页面、路由、状态管理、API 调用
- 需要跨前后端、跨服务、跨数据层进行链路梳理的业务功能

适用场景包括：

- 理解老功能
- 接手他人模块
- 变更前影响面分析
- 编写技术交接文档
- 输出功能说明书
- 排查“功能为什么这样工作”

## 3. 方法论核心

### 3.1 以功能闭环为阅读单位

不要以类、文件、模块为单位读源码。

正确单位是：

- 谁在什么场景下
- 从哪个入口触发
- 执行什么动作
- 系统经过哪些前后端链路
- 最终产生什么结果

### 3.2 先主流程，后分支和异常

优先打通一条真实主链路：

`页面入口 -> 交互事件 -> API -> Gateway -> Controller -> Service -> DAO -> DB/外部服务`

主链路打通后，再补：

- 条件分支
- 异常路径
- 异步路径
- 配置影响
- 权限和租户影响

### 3.3 先证据，后结论

AI 不允许依靠命名猜测业务结论。

每个关键结论都必须有证据来源，例如：

- 文件路径
- 类名
- 方法名
- 字段名
- 路由定义
- 接口路径
- 配置项
- SQL 或表结构

### 3.4 分离“代码逻辑”和“业务规则”

源码里看到的是 `if/else`，文档里要输出的是“业务规则”。

例如，不要写：

- `if (status == 2) return error`

应该写成：

- 当单据状态为“已审核”时，禁止再次编辑

### 3.5 必须形成闭环

一个功能被真正理解，至少要同时回答：

- 入口在哪里
- 谁发起请求
- 后端谁负责主业务
- 数据改了什么
- 状态如何流转
- 异常如何处理
- 是否影响缓存、消息、外部系统
- 是否受权限、配置、租户影响

## 4. 分析产物清单

每次分析必须至少产出以下 8 个工件：

- `A0 功能定义卡`
- `A1 入口地图`
- `A2 主调用链`
- `A3 业务规则表`
- `A4 数据影响表`
- `A5 状态流转表`
- `A6 异常矩阵`
- `A7 证据表`

如果这 8 个工件不完整，则功能理解不算完成。

## 5. 六轮分析SOP

### 第0轮：任务定标

目标是先把分析边界钉死。

输入信息：

- 功能名称
- 用户角色
- 触发动作
- 预期结果
- 已知入口
- 已知异常

输出工件：`A0 功能定义卡`

模板如下：

```md
功能定义卡
- 功能名：
- 用户角色：
- 触发动作：
- 预期结果：
- 上下游关联：
- 本次范围：
- 非本次范围：
```

退出标准：

- 能用一句话说清“谁在什么场景下做什么动作，系统产生什么结果”

### 第1轮：入口定位

目标是找到功能从哪里开始。

重点查看：

- 菜单配置
- 路由定义
- 页面组件
- 按钮、弹窗、入口文案
- 权限包裹
- API 模块引用

输出工件：

- `A1 入口地图`
- API 初始清单

退出标准：

- 已明确用户从哪个页面进入
- 已明确是哪个按钮、链接或动作触发
- 已找到至少一条真实 API 发起点

本轮禁止事项：

- 不要深挖 Service 细节
- 不要过早下业务规则结论

### 第2轮：主链路穿透

目标是打通一条主流程。

标准链路：

`页面事件 -> API 方法 -> Gateway -> Controller -> Service -> DAO/Repository -> DB/外部服务`

输出工件：

- `A2 主调用链`
- 微服务参与清单
- 同步/异步边界说明

建议格式：

```md
主链路
1. 用户在 {页面} 点击 {按钮}
2. 前端调用 {api.ts 方法}
3. 请求进入 {gateway/path}
4. {Controller#method} 接收参数
5. {Service#method} 执行业务编排
6. {DAO/Repository} 读写 {表名}
7. 如有，调用 {Feign/MQ/第三方}
8. 返回前端并更新页面状态
```

退出标准：

- 可以用 5 到 10 步讲清主流程
- 每一步都能落到具体类和方法

### 第3轮：规则抽取

目标是抽取业务规则，而不是停留在代码层。

重点提炼：

- 参数校验
- 角色权限
- 状态约束
- 幂等和防重
- 事务边界
- 分支条件
- 枚举含义
- 租户隔离
- 灰度或开关控制

输出工件：`A3 业务规则表`

建议格式：

```md
| 规则编号 | 业务规则 | 触发条件 | 系统行为 | 证据位置 | 状态 |
|---|---|---|---|---|---|
```

退出标准：

- 关键 `if/else` 已经翻译成业务规则语句
- 不再只是描述“代码怎么写”，而是说明“业务为什么这样做”

### 第4轮：数据与状态建模

目标是从执行过程提升到系统语义。

重点查看：

- 关键实体和 DTO/VO/Entity
- 表与字段变化
- 状态字段和枚举
- 缓存读写
- 消息事件
- 外部副作用
- 配置项、租户、权限依赖

输出工件：

- `A4 数据影响表`
- `A5 状态流转表`
- 副作用矩阵

退出标准：

- 已明确改了哪些表和字段
- 已明确功能状态如何流转
- 已明确是否存在异步最终一致性

### 第5轮：异常与边界验证

目标是补齐非 happy path。

重点检查：

- 参数错误
- 权限不足
- 数据不存在
- 重复提交
- 并发冲突
- 远程调用失败
- MQ 消费失败
- 回滚、补偿、降级
- 前后端契约不一致

输出工件：

- `A6 异常矩阵`
- 待验证问题清单

建议格式：

```md
| 场景 | 触发条件 | 后端行为 | 前端表现 | 恢复方式 | 证据 |
|---|---|---|---|---|---|
```

退出标准：

- 已覆盖正常流程、失败流程、边界流程
- 所有当前无法 100% 证实的点已单独列出

### 第6轮：整编成人类文档

目标是把分析结果转成可读的功能说明书。

最终报告建议结构：

1. 功能摘要
2. 用户入口与操作路径
3. 前端实现链路
4. 接口契约与后端调用链
5. 核心业务规则
6. 数据模型与状态流转
7. 配置、权限、租户影响
8. 异常流程与补偿机制
9. 关键源码索引
10. 证据表
11. 未确认问题
12. 结论与风险

退出标准：

- 产品能看懂这个功能做什么
- 开发能定位关键代码
- 测试能据此设计测试点
- 架构师能识别风险和依赖

## 6. 证据规则

每条重要结论都要尽可能带以下信息：

- 文件路径
- 类名
- 方法名
- 字段名
- 接口路径
- 配置键
- 表名
- 状态值或枚举

结论必须分级：

- `已证实`：有直接源码证据
- `高概率推断`：有间接证据，但仍缺一环
- `待验证`：当前没有足够证据下结论

证据表模板：

```md
| 结论 | 证据文件 | 方法/符号 | 证据类型 | 置信度 |
|---|---|---|---|---|
```

## 7. 多Agent协作建议

建议按以下角色拆分：

- `Lead Agent`：定范围、拆任务、合并结论、控质量
- `Frontend Agent`：分析 React 入口、交互、状态、API 发起
- `Backend Agent`：分析 Controller、Service、DAO、Feign、MQ
- `Data Agent`：分析表、字段、状态、缓存、配置、副作用
- `Verifier Agent`：做冲突检查、证据补漏、置信度评估

协作原则：

- 一个 Agent 只负责一个视角
- 子 Agent 不擅自扩范围
- 冲突结论必须回到证据层处理
- 最终结论只能由 Lead Agent 汇总后输出

## 8. 完成判定标准

一个功能只有同时满足以下条件，才算“已被理解”：

- 已找到真实入口
- 已打通前后端主链路
- 已抽取核心业务规则
- 已识别数据变化和状态流转
- 已覆盖异常流程
- 已检查配置、权限、租户影响
- 已附完整证据表
- 已单列未确认问题

## 9. 常见失败模式

以下情况说明分析质量不合格：

- 只看类名和方法名就下结论
- 只分析主流程，不分析异常流
- 只看 Controller，不看真实 Service 编排
- 只看前端页面，不看权限和配置影响
- 把技术流程当业务规则
- 忽略 MQ、缓存、定时任务、外部副作用
- 输出一堆源码摘录，但没有形成业务理解

## 10. 推荐交付格式

建议每次交付都分三层：

### 第一层：一页摘要

面向产品、架构、管理角色，快速说明：

- 功能做什么
- 谁使用
- 主流程是什么
- 关键风险是什么

### 第二层：技术详解

面向开发、测试、实施，说明：

- 前后端实现链路
- 业务规则
- 数据变化
- 状态流转
- 异常处理

### 第三层：证据附录

面向复核和后续维护，提供：

- 关键源码索引
- 证据表
- 未确认问题

## 11. 最终结论

对既有系统做功能理解，最重要的不是“看了多少代码”，而是是否建立了下面这条闭环：

`用户入口 -> 前端交互 -> 接口契约 -> 后端编排 -> 数据变化 -> 状态流转 -> 异常处理 -> 证据回溯`

只要沿着这条闭环执行，并坚持多轮分析、证据优先、结论分级，AI 就能稳定地产出可被人类使用的功能理解文档。
