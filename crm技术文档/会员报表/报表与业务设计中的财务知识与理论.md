# 报表与业务系统设计中必须体现的财务知识与理论

> 本文档从"技术人员视角"系统梳理：哪些财务理论**不是财务部门的事**，而是**必须嵌入到你的代码、表结构和报表逻辑中**的硬性约束。
> 每个理论均结合 CRM 积分/储值/券等业务场景，说明"不遵守会怎样"。

---

## 一、会计基础原则——决定你的数据怎么存

### 1.1 复式记账法（Double-Entry Bookkeeping）

**核心思想**：每一笔经济业务至少涉及两个科目，且借方总额 = 贷方总额。

**对业务系统的要求**：

```
会员充值 ¥100：
  借：银行存款    ¥100（资产增加）
  贷：预收账款    ¥100（负债增加）

会员消费 ¥60：
  借：预收账款    ¥60（负债减少）
  贷：主营业务收入 ¥60（收入增加）
```

**在你的系统中意味着什么**：

| 理论要求 | 系统设计映射 | 反模式 |
|---------|------------|--------|
| 每笔交易至少影响两个科目 | 储值消费应同时写 `crm_card_record`（余额变动）和财务凭证（收入确认） | 只扣余额不生成凭证 |
| 借贷必须平衡 | `balance_before + amount = balance_after` 恒等式 | 余额直接覆盖写，无法校验 |
| 不允许单边记账 | 积分发放必须同时记录"积分增加"和"合同负债增加" | 只在会员卡上加积分 |

> [!IMPORTANT]
> 你现有的 `crm_card_points_record` 中 `balance_before + amount = balance_after` 校验，本质上就是**复式记账的简化形式**——确保每一笔变动都是自洽的、可验证的。这个恒等式**绝对不能删除或放松**。

### 1.2 权责发生制 vs 收付实现制

**权责发生制（Accrual Basis）**：收入和费用在**经济事实发生时**确认，而非在收到/支付现金时确认。

```
场景：会员5月1日充值¥1000，5月15日消费¥300

收付实现制（错误的财务口径）：
  5月1日确认收入 ¥1000 ← 这是错的！钱收到了但服务还没提供

权责发生制（正确的财务口径）：
  5月1日：收到现金 ¥1000 → 确认预收账款（负债）¥1000
  5月15日：提供服务 ¥300 → 减少预收账款 ¥300，确认收入 ¥300
  月末：预收账款余额 ¥700（未来还要提供服务的义务）
```

**在你的系统中意味着什么**：

| 业务场景 | 权责发生制要求 | 系统设计影响 |
|---------|-------------|------------|
| 储值充值 | 充值不是收入，是预收款（负债） | 充值报表不能叫"收入报表" |
| 积分发放 | 积分是合同负债，不是费用 | 积分报表的"本期增加"= 负债增加 |
| 积分过期 | 过期当期确认收入（负债释放） | 必须有过期流水记录 |
| 跨月消费 | 收入归属于消费发生月份 | `yingyeriqi`（业务日期）才是归期口径 |

> [!WARNING]
> **`yingyeriqi` vs `create_time` 的区分，本质上就是权责发生制的技术实现**。如果你的报表用 `create_time` 归期，就变成了收付实现制——5月28日的消费因为系统延迟在6月1日才入库，按 `create_time` 就归入了6月，财务口径就错了。

### 1.3 会计分期假设（Accounting Period）

**核心思想**：企业的持续经营活动被人为划分为等长的期间（月/季/年），每个期间独立结算。

**对系统的硬性约束**：

```
关键规则：
1. 已关账期间的数据不可修改（只能通过当期调整分录处理）
2. 每个期间必须有明确的"开始-结束-关账"状态
3. 跨期间的交易必须有明确的归期规则
```

**系统设计清单**：

- [ ] 日历维度表（`dim_calendar`）包含会计期间状态
- [ ] 关账操作不可逆（或需要高级审批才能反关账）
- [ ] 迟到数据的处理策略（归入当期 or 调整分录）
- [ ] 月末截止时间明确定义（是自然日 23:59:59 还是次日凌晨 N 点？）

### 1.4 配比原则（Matching Principle）

**核心思想**：收入与产生该收入的成本/费用必须在同一期间确认。

```
场景：会员消费 ¥300，系统发放 30 积分（1积分 = ¥0.01）

配比原则要求：
  收入确认：¥300（扣除积分负债后的净额，见 IFRS 15）
  积分负债：¥0.30（30积分 × ¥0.01）
  → 两者必须在同一期间（5月）确认

违反配比原则的错误做法：
  5月确认收入 ¥300
  积分负债"等到积分被使用时"才确认 ← 违反！
```

**报表设计影响**：R3 期间汇总报表中，如果要展示"积分成本率"，分子（积分负债）和分母（关联收入）必须来自同一期间。

### 1.5 重要性原则（Materiality）

**核心思想**：如果一个信息的遗漏或错误会影响使用者的决策，那它就是重要的。

**在报表中的应用**——这是"容差阈值"的理论依据：

| 场景 | 重要性判断 | 系统处理 |
|------|----------|---------|
| `balance_before + amount - balance_after = ±1 分` | 不重要（浮点精度） | `AUTO_RECONCILED`，不需人工 |
| 对账差异 ¥0.50（月总积分 100万） | 不重要（万分之零点五） | 自动核销 |
| 对账差异 ¥500（月总积分 100万） | 重要（万分之五） | 必须人工排查 |
| 对账差异 ¥50,000 | 非常重要 | 阻断关账，升级处理 |

**容差阈值应该按比例设定，而不是固定金额**：

```
建议阈值策略：
  自动核销：|差异| ≤ MAX(1, 月积分总额 × 0.001%)
  人工复核：|差异| > MAX(1, 月积分总额 × 0.001%) 且 ≤ 月积分总额 × 0.1%
  阻断关账：|差异| > 月积分总额 × 0.1%
```

### 1.6 谨慎性原则（Prudence / Conservatism）

**核心思想**：不高估资产/收入，不低估负债/费用。在不确定时，选择对企业不利的那个估计。

**在积分业务中的体现**：

| 场景 | 谨慎性要求 | 错误做法 |
|------|----------|---------|
| 积分兑换率估计 | 估计偏高（更多积分会被使用） → 更多负债 | 故意低估兑换率以减少负债 |
| 积分过期预测 | 不确定是否会过期时，假设不会过期 → 维持负债 | 提前确认过期收入 |
| 差异处理 | 不确定差异原因时，不能假设"是浮点误差" | 未查明原因就自动核销大额差异 |

## 二、收入确认理论——决定"什么时候算钱"

### 2.1 IFRS 15 / ASC 606 五步法模型

这是全球通用的收入确认标准，**直接影响你的报表口径**：

```
第一步：识别合同
  → 会员充值/消费/积分发放都是与客户的合同

第二步：识别履约义务
  → 一次消费可能包含多个履约义务：
     ① 提供商品/服务（即时履约）
     ② 未来积分兑换权利（未来履约）

第三步：确定交易价格
  → 客户实际支付的总金额

第四步：将交易价格分摊到各履约义务
  → 按各义务的"单独售价"比例分摊
  → 这就是为什么 ¥100 消费不能全部确认为收入

第五步：在履约时确认收入
  → 商品交付时确认商品收入
  → 积分兑换时确认积分对应的递延收入
  → 积分过期时确认积分对应的递延收入
```

**对报表系统的影响——积分负债的完整生命周期**：

```
                    ┌─────────────────────────────────┐
                    │         积分负债池                │
                    │   (合同负债 / 递延收入)           │
                    └─────────────────────────────────┘
                          ↑              ↓        ↓
                       发放增加        兑换释放   过期释放
                      (负债+)        (负债-     (负债-
                                     收入+)     收入+)

  R3 期间汇总应展示：
    期初负债 → 本期新增负债 → 本期兑换释放 → 本期过期释放 → 期末负债

  R4 对账应验证：
    期末负债 = 期初负债 + 新增 - 兑换释放 - 过期释放 ± 调整
```

### 2.2 递延收入的后续计量

**比例法 vs 断裂法**（积分过期时如何确认收入）：

| 方法 | 原理 | 计算方式 | 适用场景 |
|------|------|---------|---------|
| 比例法 | 随着积分被使用，按比例确认收入 | 每次兑换时：确认收入 = 本次兑换积分 / 预计总兑换积分 × 总递延收入 | 能合理估计兑换率时 |
| 断裂法 | 在积分失效时一次性确认未使用部分的收入 | 过期时：确认收入 = 未使用积分的对应递延收入 | 无法合理估计兑换率时 |

> [!TIP]
> 大多数中国企业的积分系统在实务中使用**简化处理**：积分发放时全额计入"销售费用"或"营业外支出"，不做递延收入处理。这种简化在企业规模较小或积分金额不重要时是可接受的，但如果积分负债金额较大（如超过营收的 1%），审计师可能会要求按 IFRS 15 处理。

## 三、成本会计理论——决定"积分值多少钱"

### 3.1 积分的成本计量

积分的"价值"在不同场景下有不同的计量方式：

| 计量口径 | 定义 | 使用场景 | 举例 |
|---------|------|---------|------|
| 面值（Face Value） | 积分规则定义的兑换价值 | 客户端展示、运营分析 | 1积分 = ¥0.01 |
| 公允价值（Fair Value） | 考虑兑换率后的预期成本 | 财务报表（IFRS 15） | 1积分 × 60%兑换率 = ¥0.006 |
| 增量成本（Incremental Cost） | 兑换时实际产生的边际成本 | 管理会计、预算 | 积分兑换商品的采购成本 |
| 历史成本（Historical Cost） | 积分发放时的成本计价 | 成本追溯 | 发放时按规则版本对应的成本 |

**报表设计影响**：

- **运营报表**（R1/R3）：通常使用面值（`amount × 单位面值`）
- **财务报表**（R4 对账）：应使用公允价值或增量成本
- **管理报表**：可能需要同时展示面值和公允价值，提供对比分析

### 3.2 积分成本分摊

```
场景：门店 A 的消费产生了积分，但积分在门店 B 兑换使用

问题：积分成本应该归属于哪个门店？

方案一（发放归属）：成本归 A（谁产生的积分谁承担成本）
方案二（兑换归属）：成本归 B（谁接受兑换谁承担成本）
方案三（总部承担）：成本归总部（积分视为总部营销费用）
```

**报表设计影响**：R3 期间汇总报表的 `sid`（门店维度）分组时，"本期减少积分"的门店归属必须明确是发放门店还是兑换门店。

## 四、内部控制理论——决定"谁能看、谁能改"

### 4.1 COSO 内部控制五要素

| 要素 | 在报表系统中的体现 |
|------|-----------------|
| **控制环境** | 管理层对数据准确性的态度、财务人员的专业能力、报表口径的正式审批流程 |
| **风险评估** | 识别报表数据可能出错的环节（如并发记账、迟到数据、枚举漂移） |
| **控制活动** | 恒等式校验、关账审批流、职责分离、对账机制 |
| **信息与沟通** | 异常告警通知、报表变更公告、数据契约 |
| **监督活动** | 月度对账、年度审计、报表准确率 KPI |

### 4.2 职责分离矩阵（SOD Matrix）

```
角色定义：
  R = 记录（Record）   谁录入数据
  A = 审批（Approve）   谁批准操作
  V = 验证（Verify）    谁核实正确性
  C = 保管（Custody）   谁持有资产/系统权限

规则：同一个人不能同时承担 R+A，或 R+C，或 A+V
```

| 操作 | 记录(R) | 审批(A) | 验证(V) | 保管(C) |
|------|---------|---------|---------|---------|
| 积分手动调账（蓝补/红冲） | 客服/运营 | 运营主管 | 财务 | 系统管理员 |
| 月结快照生成 | 系统自动 | — | 财务 | DBA |
| 月结关账 | 财务 | 财务主管 | — | 系统管理员 |
| 差异处理 | 财务 | 财务主管 | 审计 | — |
| 报表口径变更 | 技术 | 财务 + 技术 | 审计 | — |

### 4.3 审计痕迹（Audit Trail）的深度要求

不只是"谁在什么时间做了什么"，而是要满足**完整可追溯**：

```sql
-- 审计日志表设计
CREATE TABLE audit_trail (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    event_time      DATETIME(3) NOT NULL,      -- 精确到毫秒
    event_type      VARCHAR(32) NOT NULL,       -- QUERY / EXPORT / MODIFY / APPROVE
    operator_id     VARCHAR(64) NOT NULL,       -- 操作人ID
    operator_name   VARCHAR(128),               -- 操作人姓名
    operator_ip     VARCHAR(45),                -- 操作IP
    target_report   VARCHAR(64),                -- 报表标识
    query_params    JSON,                       -- 查询条件（含时间范围、筛选条件）
    result_count    INT,                        -- 返回/导出的数据行数
    data_scope      VARCHAR(256),               -- 数据范围（哪些门店/会员）
    export_format   VARCHAR(16),                -- 导出格式（XLSX/CSV/PDF）
    session_id      VARCHAR(64),                -- 会话ID（关联前端操作链路）
    INDEX idx_event_time (event_time),
    INDEX idx_operator (operator_id, event_time)
);
```

> [!IMPORTANT]
> 审计日志本身**不可修改、不可删除**。通常建议审计日志表使用独立的数据库实例，且 DBA 对审计库只有读权限。

## 五、税务相关理论——决定"要不要交税、怎么交"

### 5.1 增值税与积分的关系

| 场景 | 增值税处理 | 报表影响 |
|------|----------|---------|
| 充值 | 充值时不开票（预收款不纳税） | 充值报表标注"不含税" |
| 消费 | 消费时开票（确认收入时纳税） | 消费报表需区分含税/不含税金额 |
| 积分兑换商品 | 视同销售，可能需要缴纳增值税 | 积分兑换报表需展示"视同销售金额" |
| 积分抵扣现金 | 属于折扣，减少销售额 | 报表需展示折扣前/折扣后金额 |

### 5.2 发票与报表的勾稽

```
财务级报表必须能与发票系统对账：

  报表"本月消费收入" 
    = 本月开票金额（含税）/ (1 + 税率)
    + 本月未开票收入
    - 本月红冲发票对应收入

如果对不上 → 要么报表口径有问题，要么开票有遗漏
```

## 六、管理会计理论——决定"报表给谁看、看什么"

### 6.1 责任会计（Responsibility Accounting）

**核心思想**：每个管理者只对自己可控范围内的收入/成本负责。

```
门店店长：只看自己门店的积分发放/兑换数据
区域经理：看所辖区域所有门店的汇总
总部运营：看全国汇总 + 异常门店预警
财务总监：看对账报表 + 差异分析
CEO：看积分负债总额趋势 + 兑换率趋势
```

**报表设计影响**：

- 权限模型必须支持**层级式数据隔离**（门店→区域→总部）
- 不同角色看到的**指标不同**（运营看笔数/转化率，财务看金额/负债）
- 同一报表的**下钻深度不同**（店长可下钻到流水，CEO 只看趋势）

### 6.2 差异分析框架（Variance Analysis）

财务报表不只是展示数字，还要回答"为什么和预期不一样"：

```
差异分解公式（以积分发放为例）：

  实际积分发放量 - 预算积分发放量 = 总差异

  总差异可分解为：
    ① 数量差异 = (实际交易笔数 - 预算交易笔数) × 预算单笔积分
    ② 价格差异 = 实际交易笔数 × (实际单笔积分 - 预算单笔积分)
    ③ 混合差异 = (实际笔数 - 预算笔数) × (实际单笔 - 预算单笔)
```

**报表设计影响**：R3 期间汇总报表如果要服务管理层决策，最好能支持"实际 vs 预算"的对比展示，并自动计算差异。

### 6.3 关键财务指标（KPI）

报表系统应能计算并展示以下积分相关 KPI：

| KPI | 计算公式 | 意义 |
|-----|---------|------|
| 积分发放率 | 本期发放积分 / 本期消费金额 | 积分成本占比 |
| 积分兑换率 | 本期兑换积分 / 期初可用积分 | 积分活跃度 |
| 积分沉淀率 | 期末未使用积分 / 累计发放积分 | 负债风险 |
| 积分负债率 | 积分负债总额 / 营业收入 | 对利润的潜在影响 |
| 积分过期率 | 本期过期积分 / 到期应过期积分 | 预测准确性 |
| 人均积分余额 | 总积分余额 / 活跃会员数 | 会员价值 |

## 七、理论到代码的映射总表

| 财务理论 | 核心要求 | 你的代码/表结构中怎么体现 |
|---------|---------|----------------------|
| 复式记账 | 借贷平衡 | `balance_before + amount = balance_after` |
| 权责发生制 | 按业务发生时间归期 | `yingyeriqi` 做归期口径，非 `create_time` |
| 会计分期 | 期间独立、关账不可逆 | `dim_calendar.is_closed`、关账审批流 |
| 配比原则 | 收入与成本同期确认 | 积分负债与关联收入在同一期间 |
| 重要性原则 | 容差阈值 | 对账差异 ≤ 阈值 → `AUTO_RECONCILED` |
| 谨慎性原则 | 不低估负债 | 积分兑换率估高不估低 |
| IFRS 15 | 收入分摊+递延 | 积分负债池、过期释放收入 |
| 红冲蓝补 | 历史不可改 | `AccountingRedFlush` / `AccountingBluePatch` |
| 职责分离 | R≠A≠V | 操作人≠审批人≠复核人 |
| 审计痕迹 | 全程可追溯 | `audit_trail` 表、`related_points_record_lid` |
| 责任会计 | 按权责范围报告 | 行级权限（sid 过滤）、角色差异化视图 |
| 差异分析 | 解释偏差原因 | R4 差异金额 + 差异流水追溯 + 差异原因字段 |
