# 报表设计的原则与基本要素

> 本文档系统讲解企业级报表设计中的通用原则和必备要素。
> 适用于任何业务领域的报表，同时结合 CRM 积分报表场景举例。

---

## 一、报表的六大基本要素

每一张合格的企业级报表，都必须包含以下六个结构要素：

```
┌─────────────────────────────────────────────────┐
│ ① 表头区（Header）                               │
│    报表名称 / 所属组织 / 报表期间 / 生成时间        │
├─────────────────────────────────────────────────┤
│ ② 筛选条件区（Filter Bar）                        │
│    时间范围 / 门店 / 业务类型 / 会员 / ...          │
├─────────────────────────────────────────────────┤
│ ③ 列定义区（Column Definition）                   │
│    列名 / 数据类型 / 对齐方式 / 排序 / 宽度         │
├─────────────────────────────────────────────────┤
│ ④ 数据区（Data Body）                             │
│    N 行 × M 列的结构化数据                         │
├─────────────────────────────────────────────────┤
│ ⑤ 合计/小计区（Summary Row）                      │
│    全表合计 / 分组小计 / 占比 / 均值                │
├─────────────────────────────────────────────────┤
│ ⑥ 脚注区（Footer）                               │
│    口径说明 / 数据来源 / 免责声明 / 页码             │
└─────────────────────────────────────────────────┘
```

### 各要素详解

#### ① 表头区

| 必备项 | 说明 | 示例 |
|--------|------|------|
| 报表名称 | 唯一标识，含业务主题 | "会员积分流水明细报表" |
| 所属组织 | 商户/公司名称 | "XX餐饮集团" |
| 报表期间 | 明确的起止时间 | "2026-05-01 至 2026-05-31" |
| 生成时间 | 数据截止时点 | "生成于 2026-06-01 08:30:15" |
| 报表版本 | 口径变更版本 | "V2.1（2026-05起启用新口径）" |

> [!WARNING]
> **没有"生成时间"的报表是危险的**。财务人员在不同时间打开同一报表，如果底层数据在持续变动（如未关账期间），看到的数字可能不同。"生成时间"告诉读者"这个数字反映的是截至何时的状态"。

#### ② 筛选条件区

**设计原则**：

- **必选条件**：时间范围（`yingyeriqi`）、商户（`mid`）——不能允许无条件全量查询
- **可选条件**：门店、会员、业务类型、处理状态等
- **条件回显**：导出的 Excel/PDF 必须在表头展示"当前筛选条件"
- **默认值**：时间默认为"当月"或"最近7天"，避免用户误查全量

#### ③ 列定义

每一列必须有明确的规范：

| 属性 | 说明 | 示例 |
|------|------|------|
| 列标题 | 业务语义名称，非数据库字段名 | "本次发生额"（非 `amount`） |
| 数据类型 | 决定展示格式和对齐 | 金额→右对齐两位小数；日期→左对齐 |
| 可排序 | 是否支持点击列头排序 | 金额列、日期列通常可排序 |
| 可筛选 | 是否支持列内筛选 | 业务类型、门店等枚举列 |
| 固定/冻结 | 是否固定在视口左侧 | 会员姓名、卡号通常固定 |
| 脱敏规则 | 敏感数据的展示规则 | 手机号→`138****1234` |
| 空值展示 | NULL 和 0 的区分 | NULL→`—`；0→`0.00` |

#### ④ 数据区

- **行粒度必须明确声明**：1行代表什么（一笔流水？一个账户？一个门店的月汇总？）
- **排序稳定性**：ORDER BY 必须有确定性兜底列（如 `lid DESC`），避免分页跳行/重复
- **空状态处理**：无数据时显示"暂无数据"而非空白或报错

#### ⑤ 合计/小计区

| 类型 | 位置 | 计算方式 |
|------|------|---------|
| 全表合计 | 表格底部固定行 | 对**当前筛选条件下全部数据**求和，非仅当前页 |
| 分组小计 | 每个分组的最后一行 | 对该分组内数据求和 |
| 页面合计 | 当前页底部 | 仅对当前页数据求和（较少使用） |

> [!IMPORTANT]
> **合计行必须对全量数据计算，不能只对当前分页求和**。这是报表开发中最常见的错误之一。正确做法是后端返回一个独立的 `summary` 对象，前端固定展示在表格底部。

#### ⑥ 脚注区

| 必备项 | 内容 |
|--------|------|
| 口径说明 | "本报表仅统计 if_deal_success=1 的有效流水" |
| 数据来源 | "数据来源：crm_card_points_record 事实表" |
| 归期说明 | "归期口径：以业务日期（yingyeriqi）为准" |
| 精度说明 | "金额精确到分，积分为整数" |
| 免责声明 | "本报表数据截至生成时间，后续可能因补录/冲正发生变化" |

## 二、十大报表设计原则

### 原则一：单一事实来源（Single Source of Truth）

```
同一个指标，不管出现在哪张报表里，计算口径和数据源必须一致。

正确：R2 的"本期获得积分" 和 R3 的"本期增加积分"
      → 都来自 crm_card_points_record 的 SUM(amount > 0)

错误：R2 从流水表算，R3 从另一张汇总表算
      → 两张表的 ETL 时间差导致数字对不上
```

### 原则二：口径先于实现（Definition Before Implementation）

```
开发顺序：
  ① 财务/业务定义报表口径（用自然语言 + 公式）
  ② 技术评审口径的可实现性
  ③ 双方签字确认口径文档
  ④ 开发实现
  ⑤ UAT 阶段用手工数据交叉验证

错误顺序：
  ① 技术直接写 SQL
  ② 上线后财务发现"数字不对"
  ③ 反复扯皮修改
```

### 原则三：可验证性（Verifiability）

```
任何一个汇总数字，都必须能"打开"看到构成它的明细。

R3 显示"门店A 本期增加积分 5,000"
  → 用户点击 → 展开 R1 明细：显示构成这 5,000 的 N 笔流水
  → 用户可以手动加总验证：确实是 5,000

如果无法下钻验证 → 这个数字不可信 → 财务不会认可
```

### 原则四：归期一致性（Period Consistency）

```
同一报表体系内，所有报表用同一个时间字段做归期。

正确：R1/R2/R3/R4 全部用 yingyeriqi 做归期
错误：R1 用 create_time，R3 用 yingyeriqi
      → 同一笔流水在 R1 归入5月，在 R3 归入4月 → 永远对不上
```

### 原则五：精度一致性（Precision Consistency）

```
全链路（数据库 → 后端 → 前端 → 导出）的数值精度必须一致。

数据库：DECIMAL(20,2)
后端：  BigDecimal，不允许 float/double
前端：  展示两位小数，千分位格式（如 12,345.67）
导出：  Excel 单元格格式为数值型，非文本型
```

### 原则六：NULL ≠ 0 ≠ 空字符串

```
在财务报表中，三者含义完全不同：

NULL  = 该数据不存在/未发生（该会员本月没有任何交易）
0     = 发生了，但值为零（该会员本月积分增减恰好抵消）
""    = 数据质量问题（字段存在但值异常）

展示规范：
  NULL → 显示 "—" 或 "-"
  0    → 显示 "0" 或 "0.00"
  ""   → 不应出现，如出现应标记为异常
```

### 原则七：先聚合后分页（Aggregate Then Paginate）

```
正确：
  SELECT 维度, SUM(amount), COUNT(*) 
  FROM 事实表 
  WHERE 条件 
  GROUP BY 维度 
  ORDER BY ... 
  LIMIT :pageSize OFFSET :offset  ← 对聚合结果分页

错误：
  SELECT * FROM 事实表 LIMIT 100  ← 先取100行
  然后在应用层 GROUP BY           ← 聚合的只是100行，不是全量
```

### 原则八：导出即所见（WYSIWYG Export）

```
导出的 Excel/PDF 必须与屏幕上看到的一致：
  - 相同的筛选条件
  - 相同的排序
  - 相同的列顺序和列名
  - 包含合计行
  - 包含表头信息（报表名称、期间、筛选条件）
  - 包含脚注（口径说明）
```

### 原则九：幂等可重跑（Idempotent Regeneration）

```
同一份报表，在相同条件下重新生成 N 次，结果完全一致。

前提：
  - 查询的数据源在关账后不再变化
  - 或查询时锁定快照版本
  
反模式：
  - 报表 SQL 中包含 NOW() 或 RAND()
  - 报表查询实时数据但不标注数据截止时间
```

### 原则十：权限最小化（Least Privilege）

```
每个用户只能看到其权限范围内的数据：

店长：WHERE sid = :userSid          ← 只看本店
区域：WHERE sid IN (:regionSids)    ← 看所辖门店
总部：不加 sid 限制                  ← 看全部

敏感列（手机号、身份证）默认脱敏，需特殊权限才能查看明文。
```

## 三、字段设计规范

### 3.1 字段命名规范

| 类别 | 命名规范 | 正确示例 | 错误示例 |
|------|---------|---------|---------|
| 期初/期末 | `opening_` / `closing_` | `opening_balance` | `start_balance` |
| 增加/减少 | `earned_` / `used_` | `earned_points` | `add_points` |
| 本期发生 | `period_` 前缀 | `period_net_change` | `this_month_change` |
| 计算值 | `computed_` 前缀 | `computed_closing` | `calc_end` |
| 快照值 | `snapshot_` 前缀 | `snapshot_balance` | `card_points` |
| 差异值 | `difference_` 前缀 | `difference_amount` | `diff` |
| 计数 | `_count` 后缀 | `record_count` | `num` |
| 比率 | `_rate` 后缀 | `redemption_rate` | `ratio` |
| 标志位 | `is_` / `has_` 前缀 | `is_new_account` | `new_flag` |

### 3.2 字段分类

每张报表的字段可分为四类：

| 类别 | 用途 | 示例 |
|------|------|------|
| **维度字段** | 分组、筛选、下钻 | `sid`、`operation_model`、`yingyeriqi` |
| **度量字段** | 求和、平均、计数 | `amount`、`earned_points`、`record_count` |
| **属性字段** | 仅展示、不参与计算 | `member_name`、`shop_name`、`phone` |
| **审计字段** | 追溯、校验 | `create_time`、`operator`、`related_lid` |

> [!TIP]
> 维度字段可以出现在 GROUP BY 和 WHERE 中；度量字段只能出现在聚合函数中；属性字段只能出现在 SELECT 中（且通常需要 MAX/MIN 包裹）。设计阶段明确分类，可以避免很多 SQL 错误。

## 四、口径管理规范

### 4.1 口径定义文档模板

每张报表必须有一份对应的口径说明：

```yaml
报表名称: 会员积分期间汇总报表 (R3)
报表编号: RPT-POINTS-003
版本: V2.1
生效日期: 2026-05-01
口径变更记录:
  - V1.0 (2026-01): 初版
  - V2.0 (2026-03): 增加期初/期末余额
  - V2.1 (2026-05): 拆分异常笔数为增加/减少方向

数据源: crm_card_points_record (if_deal_success = 1)
归期字段: yingyeriqi
统计口径: 
  - 本期增加: SUM(amount) WHERE amount > 0
  - 本期减少: SUM(ABS(amount)) WHERE amount < 0
  - 期初余额: 期间开始前最后一笔的 balance_after
排除条件:
  - amount IS NULL 或 amount = 0 的骨架行
  - balance_before IS NULL 或 balance_after IS NULL 的异常行

签字确认:
  财务负责人: ___________  日期: __________
  技术负责人: ___________  日期: __________
```

### 4.2 口径变更影响评估

```
口径变更申请 → 影响评估检查清单：

□ 是否影响其他报表的勾稽关系？（R1↔R3 的聚合关系是否还成立？）
□ 是否影响已关账期间的历史数据？（历史数据需要重跑吗？）
□ 是否需要并行运行新旧口径？（建议至少并行1个月）
□ 下游消费方（总账、风控）是否知晓？
□ 导出模板是否需要同步更新？
```

## 五、异常与边界场景处理

| 场景 | 处理方式 | 报表展示 |
|------|---------|---------|
| 查询结果为空 | 返回空数据 + 提示 | "当前条件下暂无数据" |
| 数据量超大 | 限制最大导出行数 + 异步导出 | "数据量较大，已提交异步导出任务" |
| 关账前查看 | 标注"未关账，数据可能变动" | 表头醒目提示⚠️ |
| 部分数据异常 | 正常数据正常展示，异常行标红 | 异常行底色标红 + 异常原因列 |
| 跨时区 | 统一转换为业务时区 | 所有时间按商户所在时区展示 |
| 并发查询 | 读写分离，报表走从库 | 用户无感知 |

## 六、报表分类体系

| 分类维度 | 类型 | 特点 | 示例 |
|---------|------|------|------|
| **按时效** | 实时报表 | 秒级延迟，查最新 | 当前积分余额 |
| | 准实时报表 | 分钟级延迟 | 今日积分发放统计 |
| | T+1 报表 | 次日可用 | 日报/月报 |
| **按粒度** | 明细报表 | 1行=1笔交易 | R1 流水明细 |
| | 汇总报表 | 1行=1个维度组合 | R3 期间汇总 |
| | 快照报表 | 1行=1个实体在某时点的状态 | R2 余额报表 |
| **按受众** | 操作报表 | 一线人员日常使用 | 客服查账 |
| | 管理报表 | 管理层决策 | KPI 仪表盘 |
| | 法定报表 | 外部监管/审计 | 资产负债表 |
| | 内控报表 | 风控/合规 | R4 对账报表 |
| **按触发** | 自助报表 | 用户按需查询 | 在线报表页面 |
| | 推送报表 | 系统定时生成推送 | 每日积分日报邮件 |
| | 异常报表 | 异常触发生成 | 对账差异超阈值告警 |

---

> [!NOTE]
> **核心总结**：一张合格的企业级报表 = **六要素完整**（表头/筛选/列/数据/合计/脚注）+ **十原则遵守**（单一来源/口径先行/可验证/归期一致/精度一致/空值区分/先聚合后分页/导出即所见/幂等可重跑/权限最小化）。缺少任何一个要素或违反任何一条原则，在财务审计时都可能被标记为"控制缺陷"。
