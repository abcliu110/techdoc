# 会员积分账本体系与四类报表设计说明

日期：2026-05-13  
评审补充：2026-05-14

## 1. 设计目标

会员积分不是普通展示数据，而是会员账户账本的一部分。  
这份设计要解决的不是“页面上怎么显示积分”，而是“积分如何记账、如何查账、如何汇总、如何关账”。

本设计坚持以下原则：

- 以 `crm_card_points_record` 作为唯一事实表。
- 记账方向和业务类型分离。
- 余额可以由流水重算，历史不回写。
- 汇总必须能下钻到明细。
- 对账必须能闭环到差异和调整。
- 当前可实现能力和目标态财务能力必须分开描述，不能把实时查询结果当作正式关账结果。
- 正式关账结论必须由独立快照、关账批次和复核记录承载，不能只依赖事实表实时重算。
- 录入、复核、审批、关账、解锁必须职责分离，所有动作可追溯。

这套设计更接近一般企业财务软件的思路，而不是运营展示页的思路。

### 1.1 参考规范

本设计不是法定总账系统，但应借鉴财务软件的基本设计原则：凭证可追溯、账簿可复算、报表可解释、关账可冻结、内控可验证。

参考依据：

- 财政部《会计信息化工作规范》（财会〔2024〕11号）：强调会计数据处理、电子会计资料、系统安全和监督要求。
- 财政部《会计软件基本功能和服务规范》（财会〔2024〕12号）：强调会计软件应能生成凭证、账簿、报表等会计资料，并保障数据真实、完整、安全和可追溯。
- 财政部等五部委《企业内部控制基本规范》（财会〔2008〕7号）：强调风险防范、职责分离、授权审批、记录保存和可验证性。

## 2. 账本分层

| 层级 | 报表 | 解决的问题 | 财务角色 | 不能替代什么 |
|---|---|---|---|---|
| 证据层 | 流水明细报表 | 这笔积分为什么发生 | 审计、客服 | 不能替代余额 |
| 账户层 | 余额报表 | 某个时点还剩多少积分 | 客服、财务 | 不能替代汇总 |
| 分析层 | 期间汇总报表 | 这段期间怎么变、为什么变 | 运营、管理层 | 不能替代对账 |
| 关账层 | 对账报表 | 账是否平、差异在哪 | 财务、内控 | 不能替代明细 |

这四张报表不是四个页面，而是一套从记账到关账的积分账务体系。

## 3. 唯一事实表与业务值域

### 3.1 唯一事实表

`crm_card_points_record` 是积分账本的唯一事实表。  
它保存的是一次积分记账事件，不是静态余额。

### 3.2 关键字段的财务含义

| 字段 | 财务语义 | 说明 |
|---|---|---|
| `mid` | 商户ID / 租户ID | 账本归属商户。 |
| `sid` | 门店ID | 门店稳定主键，门店名称仅用于展示。 |
| `operation_model` | 业务类型 | 表示这笔积分因什么业务发生，不表示方向。 |
| `amount` | 记账发生额 | 积分变动值，正负号表达增减方向。 |
| `balance_before` | 变动前余额 | 本次记账前的积分余额。 |
| `balance_after` | 变动后余额 | 本次记账后的积分余额。 |
| `create_time` | 记账时间 | 系统入账时间。 |
| `yingyeriqi` | 业务日期 | 业务发生日期，用于日结和期间归属。 |
| `shop_name` | 发生门店 | 本次记账发生在哪个门店。 |
| `order_bill_id` | 业务单号 | 原始业务单据号。 |
| `related_points_record_lid` | 关联流水 | 冲正、反向、补记时的追溯链路。 |
| `if_deal_success` | 有效标识 | 是否纳入正式报表口径。 |
| `is_third_party` | 外部来源标识 | 是否来自第三方系统。 |
| `record_status` | 记录状态 | 报表层建议派生，用于表达成功、失败、待处理、冲正、红冲、蓝补等状态。 |
| `subject` | 科目/分类 | 业务归类字段，便于财务分组。 |
| `operator` | 操作人 | 记账责任人。 |
| `comment` | 备注 | 补充说明。 |

### 3.3 `operation_model` 值域

`operation_model` 是业务类型字典，不是方向字段。  
报表展示时应统一翻译成中文业务名称，不要混用旧文本字段。
如果写入层存在多套业务类型口径，报表层必须先标准化成统一的“标准业务类型”，原始值仅保留追溯用途，不直接参与汇总分组。

| 值 | 含义 | 方向倾向 |
|---|---|---|
| `Charge` | 充值 | 增加 |
| `ChargeCancel` | 撤销充值 | 减少 |
| `Consume` | 消费 | 减少 |
| `CancelOrder` | 订单取消 | 增加 |
| `Refund` | 订单退款 | 增加 |
| `LevelUpg` | 会员升级 | 增加 |
| `Join` | 会员开卡 | 增加 |
| `CashBack` | 会员返现 | 增加 |
| `CancelCashBack` | 撤销返现 | 减少 |
| `Exchange` | 积分兑换 | 减少 |
| `TransferOut` | 会员卡转出 | 减少 |
| `TransferIn` | 会员卡转入 | 增加 |
| `Return` | 会员卡退卡 | 减少 |
| `Withdrawal` | 会员卡提现 | 减少 |
| `Repay` | 会员卡回款 | 增加 |
| `Reverse` | 会员卡反回款 | 增加 |
| `AccountingBluePatch` | 会计蓝补 | 增加 |
| `AccountingRedFlush` | 会计红冲 | 减少 |
| `Initial` | 期初 | 视导入规则而定 |

### 3.4 财务软件的核心规则

1. `operation_model` 只表示业务类型，不表示记账方向。
2. 记账方向由 `amount` 正负和业务规则共同表达。
3. 正式报表默认只看有效记账记录，通常以 `if_deal_success = 1` 作为主口径。
4. 历史记录不直接改写，冲正、撤销、红冲、蓝补通过新增记录表达；这类调整记录必须参与余额闭环，并在汇总与对账中单列为调整类，不得与正常业务发生额混成一类。
5. 余额类报表必须能被明细反推，不能只存结论。
6. “收入 / 支出”只能作为展示别名，不能作为核心模型字段。
7. 正式对账和月结口径优先按 `yingyeriqi` 归期，`create_time` 仅作为记账时点和审计追溯字段，不得在同一张报表里混用两个归期口径。
8. 若未来要做业务日与记账日分离，必须新增独立 `biz_date`，不能把 `create_time` 既当记账日又当业务日混用。
9. `related_points_record_lid` 只能作为追溯字段使用；如果现有写入链路不能稳定写入，不得把它写成正式对账闭环的唯一承诺。

### 3.5 字段落地等级与状态矩阵

#### 3.5.1 字段落地等级

| 等级 | 含义 | 典型字段 |
|---|---|---|
| `L1` 事实直出 | 当前事实表已有字段，SQL 可直接取 | `mid`、`sid`、`operation_model`、`amount`、`balance_before`、`balance_after`、`create_time`、`yingyeriqi`、`member_id`、`phone`、`card_id`、`card_id_alias`、`order_bill_id`、`out_order_bill_id`、`related_points_record_lid`、`if_deal_success`、`is_third_party`、`subject`、`operator`、`comment` |
| `L2` SQL 派生 | 仅依赖事实表，通过窗口/聚合可得 | 记账方向、当前余额、期初余额、期末余额、本期增加、本期减少、净发生额、行级校验结果 |
| `L3` 字典映射 | 需要稳定映射表，不应写死在 SQL | 标准业务类型、业务中文名、状态中文名、`subject` 标准分类 |
| `L4` 快照/关账 | 需要独立落表保存，不能靠重跑重建 | 期末账面余额、是否已关账、关账批次、对账状态、差异原因、差异明细归档 |
| `L5` 维表补充 | 当前事实表无字段，需补主数据/维度 | `member_name`、零流水账户清单、渠道、会员组、对账范围名称 |

#### 3.5.2 状态矩阵

| 状态 | 定义 | 是否进明细 | 是否进余额 | 是否进期间汇总 | 是否进正式对账 | 是否进异常池 |
|---|---|---|---|---|---|---|
| `SUCCESS` | 已入账有效 | 是 | 是 | 是 | 是 | 否 |
| `FAIL` | 入账失败 | 是 | 否 | 否 | 否 | 是 |
| `PENDING` | 待入账/待处理 | 是 | 否 | 否 | 否 | 是 |
| `REVERSED` | 已冲正 | 是 | 是 | 是（单列调整类） | 是（单列差异源） | 否 |
| `RED_FLUSH` | 红冲 | 是 | 是 | 是（单列调整类） | 是（单列差异源） | 否 |
| `BLUE_PATCH` | 蓝补 | 是 | 是 | 是（单列调整类） | 是（单列差异源） | 否 |
| `DUPLICATE` | 重复/疑似重复 | 是 | 否 | 否 | 否 | 是 |
| `SKELETON` | 骨架/测试 | 是 | 否 | 否 | 否 | 是 |
| `SOURCE_EXCEPTION` | 来源异常 | 是 | 否 | 否 | 否 | 是 |

说明：`if_deal_success` 只作为入账成功的基础标识，正式口径必须以 `record_status` 状态矩阵和调整链路共同判断。`record_status` 由 `if_deal_success`、`operation_model` 和调整链路共同派生，不应直接由单一布尔值决定。

## 4. 四张报表总览

| 报表 | 建议接口名 | 粒度 | 核心用途 | 说明 |
|---|---|---|---|---|
| 流水明细报表 | `POST /crm/card/points/ledger/list` | 1条记录 = 1条积分事实 | 追溯、审计、客服查账 | 账本原始明细。 |
| 余额报表 | `POST /crm/card/points/balance/list` | 1个账户在1个时点的余额 | 余额查询、账户核对 | 账户快照。 |
| 期间汇总报表 | `POST /crm/card/points/summary/list` | 期间 × 商户 × 门店 × 科目 × 标准业务类型 × 方向 | 管理分析、门店对比 | 发生额分析账。 |
| 对账报表 | `POST /crm/card/points/recon/list` | 期间 + 统计范围 | 月结、核对、差异定位 | 关账控制表。 |

所有接口都应保持 `POST`，与 CRM 模块规范一致。
每张报表都应对应独立 DTO、独立 SQL、独立服务方法，不复用通用查询 DTO 直接兜底。

## 5. 流水明细报表

### 5.1 解决的问题

这张报表回答的是“这笔积分为什么发生、发生了多少、发生后余额是多少”。

### 5.2 字段清单

| 字段类别 | 推荐字段名 | 说明 |
|---|---|---|
| 商户 | 商户ID | `mid`。 |
| 门店 | 门店ID | `sid`。 |
| 时间 | 记账时间 | `create_time`。 |
| 时间 | 业务日期 | `yingyeriqi`。 |
| 门店 | 发生门店 | `shop_name`，仅展示。 |
| 会员 | 会员姓名 | `member_name`。 |
| 会员 | 会员ID | `member_id`。 |
| 会员 | 手机号 | `phone`。 |
| 账户 | 会员卡号 | `card_id_alias`。 |
| 账户 | 会员卡主键 | `card_id`。 |
| 业务 | 标准业务类型 | 由 `operation_model` 标准化后得到。 |
| 业务 | 业务类型 | `operation_model`。 |
| 业务 | 记账方向 | 由 `amount` 正负派生。 |
| 金额 | 本次发生额 | `amount`。 |
| 金额 | 变动前余额 | `balance_before`。 |
| 金额 | 变动后余额 | `balance_after`。 |
| 单据 | 业务单号 | `order_bill_id`。 |
| 单据 | 外部单号 | `out_order_bill_id`。 |
| 追溯 | 关联流水 | `related_points_record_lid`。 |
| 责任 | 操作人 | `operator`。 |
| 说明 | 备注 | `comment`。 |
| 状态 | 处理状态 | `if_deal_success`。 |
| 来源 | 外部系统标识 | `is_third_party`。 |

注：本节描述的是报表可见字段，不等同于事实表物理字段清单。`record_status`、`标准业务类型`、`对账状态` 等均属于报表层派生或关账结果层字段，不应和事实直出字段混写成同一层级。

### 5.3 统计口径

- 1 条记录对应 1 次积分记账。
- 明细只收正式入账行，必须排除 `amount` 为空、`amount = 0`、`balance_before` 为空、`balance_after` 为空的骨架行。
- `amount > 0` 记为增加，`amount < 0` 记为减少。
- `operation_model` 只做业务归类，不参与方向判断。
- `balance_before + amount = balance_after` 应作为行级校验公式。
- `if_deal_success != 1` 的记录不进入正式余额和对账口径，但要保留在异常明细中。
- 正式报表默认按 `yingyeriqi` 归期；只有在审计追溯和时点分析场景下才使用 `create_time`。

### 5.4 查询条件

- 业务日期或记账时间区间。
- 商户ID。
- 门店ID。
- 发生门店。
- 会员ID、手机号、卡号。
- 标准业务类型。
- 业务类型。
- 记账方向。
- 业务单号、外部单号、关联流水号。
- 处理状态（成功、失败、待处理、冲正、红冲、蓝补）。
- 是否第三方。
- 操作人。

### 5.5 不能替代什么

- 流水明细不能替代余额。
- 流水明细不能替代对账。

## 6. 余额报表

### 6.1 解决的问题

这张报表回答的是“某个时点这个会员账户还剩多少积分”。

### 6.2 建议粒度

默认按会员卡账户做粒度。  
会员维度只作为关联展示或筛选条件，不允许在不同报表里自由切换主账户粒度。
账户主键建议固定为 `mid + card_id`；`member_name`、`phone` 只做展示和辅助检索，不作为主键口径。

### 6.3 字段清单

| 字段类别 | 推荐字段名 | 说明 |
|---|---|---|
| 商户 | 商户ID | `mid`。 |
| 账户 | 会员姓名 | 账户归属人。 |
| 账户 | 会员ID | 账户归属会员，辅助关联字段，不是主键。 |
| 账户 | 手机号 | 账户辅助标识。 |
| 账户 | 会员卡号 | 账户主标识。 |
| 时间 | 截止时点 | 查询快照时间。 |
| 金额 | 当前余额 | 查询时点余额。 |
| 金额 | 期初余额 | 若按期间查询，则为期间开始前余额。 |
| 金额 | 本期增加 | 期间内所有正向发生额合计。 |
| 金额 | 本期减少 | 期间内所有负向发生额绝对值合计。 |
| 金额 | 期末余额 | 若按期间查询，则为期间结束时余额。 |
| 时间 | 最新记账时间 | 最近一次有效记账时间。 |
| 业务 | 最近业务类型 | 最近一次有效记账的业务类型。 |
| 门店 | 最近发生门店 | 最近一次有效记账门店。 |

### 6.4 统计口径

- 余额以最后一笔有效记账记录为准。
- 当前余额取最新有效 `balance_after`。
- 期初余额取期间开始前最近一笔有效余额。
- 期末余额取期间结束时最近一笔有效余额。
- 本期增加 = `sum(amount > 0)`。
- 本期减少 = `sum(abs(amount < 0))`。
- 如无历史记录，默认余额为 0。
- 同一时点多笔记录时，按 `create_time desc, pid desc` 的稳定排序取最后一笔有效记录；当前版本以 `pid` 作为稳定排序主键，若未来引入跨系统补录或乱序回传，应补充 `posting_seq` 作为更稳定的过账序号。
- 明细过滤规则同流水报表，必须排除 `amount` 为空、`amount = 0`、余额字段为空的骨架行。

### 6.5 查询条件

- 截止时点。
- 商户ID。
- 会员ID、手机号、卡号。
- 发生门店。
- 业务类型。
- 处理状态（成功、失败、待处理、冲正、红冲、蓝补）。
- 是否第三方。

### 6.6 不能替代什么

- 余额报表不能替代流水明细。
- 余额报表不能替代期间汇总。

## 7. 期间汇总报表

### 7.1 解决的问题

这张报表回答的是“在某个期间内，各门店、各业务类型、各方向一共发生了多少积分”。

### 7.2 建议粒度

建议按“期间 + 商户 + 门店 + 科目/分类 + 标准业务类型 + 记账方向”汇总。  
不要把汇总报表做成纯会员明细，否则会退化成流水明细的重复。  
门店名称只展示，不参与 group by；门店维度以 `sid` 为准。

### 7.3 字段清单

| 字段类别 | 推荐字段名 | 说明 |
|---|---|---|
| 商户 | 商户ID | `mid`。 |
| 时间 | 统计期间 | 日、周、月或自定义区间。 |
| 门店 | 发生门店ID | `sid`。 |
| 门店 | 发生门店名称 | 仅展示。 |
| 分类 | 科目/分类 | `subject`，正式财务汇总建议纳入维度。 |
| 业务 | 标准业务类型 | 由 `operation_model` 标准化后得到。 |
| 业务 | 业务类型 | 原始值，仅追溯。 |
| 业务 | 记账方向 | 增加 / 减少。 |
| 金额 | 期初余额 | 期间开始时余额。 |
| 金额 | 本期增加 | 期间内增加积分合计。 |
| 金额 | 本期减少 | 期间内减少积分合计。 |
| 金额 | 净发生额 | 本期增加 - 本期减少。 |
| 金额 | 期末余额 | 期间结束时余额。 |
| 笔数 | 发生笔数 | 期间内有效流水条数。 |
| 笔数 | 异常笔数 | 失败、冲正、待处理笔数。 |

注：`subject` 目前只是业务分类文本，不等同正式会计科目编码；若后续要做总账级科目汇总，需要另建 `subject_code` 字典表，再由报表层进行映射。

### 7.4 统计口径

- 先按明细归集，再做维度汇总。
- `期末余额 = 期初余额 + 本期增加 - 本期减少`。
- 汇总报表必须能下钻到流水明细。
- 冲正、红冲、蓝补应单独统计为调整类，并在余额闭环中按净额影响处理，不得与正常业务发生额混成一类。
- `subject` 作为业务分类维度的展示字段，适合正式汇总前做映射；方向始终是派生字段，不是独立口径。
- 汇总接口分页必须基于聚合后的结果集，不能先对事实行分页再聚合。

### 7.5 查询条件

- 统计周期。
- 商户ID。
- 门店ID。
- 门店。
- 科目/分类。
- 标准业务类型。
- 业务类型。
- 记账方向。
- 会员ID。
- 是否第三方。
- 处理状态（成功、失败、待处理、冲正、红冲、蓝补）。

### 7.6 不能替代什么

- 期间汇总不能替代流水明细。
- 期间汇总不能替代对账报表。

## 8. 对账报表

### 8.1 解决的问题

这张报表回答的是“账是否平、差异在哪里、能不能关账”。

### 8.2 粒度说明

对账报表默认按“期间 + 统计范围”出一条主记录即可。  
如果范围扩展到门店、会员组或账户组，则一条主记录对应一个范围，不必强行拆成多行。
对账范围必须结构化表达为 `scope_type + scope_id + scope_name`，不能只放一段文本。

### 8.3 字段清单

| 字段类别 | 推荐字段名 | 说明 |
|---|---|---|
| 商户 | 商户ID | `mid`。 |
| 时间 | 对账期间 | 账期或查询区间。 |
| 范围 | 对账范围类型 | `scope_type`。 |
| 范围 | 对账范围ID | `scope_id`。 |
| 范围 | 对账范围名称 | `scope_name`。 |
| 金额 | 期初余额 | 账期开始余额。 |
| 金额 | 本期增加 | 账期内增加积分。 |
| 金额 | 本期减少 | 账期内减少积分。 |
| 金额 | 调整类发生额 | 冲正、红冲、蓝补等调整类发生额，展示上单列，余额闭环中必须计入。 |
| 金额 | 期末账面余额 | 关账时锁定的系统余额快照；当前事实表不直接提供独立快照，本字段属于目标态字段。 |
| 金额 | 计算余额 | 按账本重算得到的余额。 |
| 金额 | 差异金额 | 账面余额 - 计算余额。 |
| 笔数 | 差异笔数 | 不能闭环的流水条数。 |
| 追溯 | 差异流水LID | `difference_record_lid`，指向差异源流水。 |
| 状态 | 对账状态 | `recon_status`，平衡、存在差异、待复核。 |
| 说明 | 差异原因 | `difference_reason`，人工复核结果。 |

### 8.4 统计口径

- 正式对账口径必须先过滤 `if_deal_success = 1`；失败、无效、待处理流水单列异常区，不参与平衡公式。
- 计算余额 = `期初余额 + 本期增加 - 本期减少`。
- `期末账面余额` 来源于关账时锁定的余额快照，不由计算余额反推；当前版本仅支持账内平衡核对，正式关账必须依赖独立快照。
- `差异金额 = 期末账面余额 - 计算余额`。
- 只要有差异，必须能下钻到差异门店、差异业务类型、差异流水。
- 对账状态建议自动判定：`差异金额 = 0` 且无异常则为平衡；差异不为 0 为存在差异；账面快照缺失或未关账为待复核。
- 对账报表不能只给结论，必须给差异来源。
- 调整类记录在对账中单列展示，但不得从余额闭环中剔除；如果它们改变余额，就必须体现在计算余额里，只是不与正常业务混算。

### 8.5 查询条件

- 对账期间。
- 对账范围。
- 对账范围类型。
- 对账范围ID。
- 对账范围名称。
- 门店。
- 科目/分类。
- 是否第三方。
- 处理状态（成功、失败、待处理、冲正、红冲、蓝补）。
- 是否纳入冲正和异常流水。
- 是否只看正式口径（`if_deal_success = 1`）。
- 是否已关账。

### 8.6 不能替代什么

- 对账报表不能替代流水明细。
- 对账报表不能替代余额报表。
- 对账报表不能替代独立账面余额快照来源。

### 8.7 锁账语义

- 日结锁账后，当日业务归期只读，不能回写原流水。
- 月结锁账后，月度报表只读，不能回写已关闭期间的历史记录。
- 关闭期间发现差异时，只能通过当前开放期间新增冲正、撤销或补记流水进行修正。
- 跨期间修正必须保留原单据和关联流水，不能直接覆盖历史余额。

## 9. 审计与关账场景

### 9.1 日结

- 以业务日期归集当天全部有效流水。
- 对当天发生额、冲正额、异常额进行核对。
- 记录差异原因和处理人。

### 9.2 月结

- 以月为单位冻结统计口径。
- 月结后历史不回写。
- 任何修正通过后续冲正、撤销、补记表达。

### 9.3 关账规则

- 关账后历史不直接修改。
- 所有修正都必须保留原始轨迹。
- 关账后如有差异，只能新增调整记录，不可覆盖原记录。
- 关账期间内不允许补录回写，补录只能进入当前开放期间并通过追溯字段指向原单据。
- 对账状态未锁定前，只能作为核对结论；锁账后才可作为关账结果。

### 9.4 审计要求

- 能按单据追溯到每一条积分流水。
- 能按流水追溯到原始业务单据。
- 能按对账差异追溯到调整记录。
- 能说明某一时点余额为什么成立。

### 9.5 职责分离与审批链

- 录入人、复核人、审批人、关账人、解锁人必须分离。
- 未复核不得置为正式关账状态。
- 关账、解锁、跨期补记必须记录 `prepared_by`、`reviewed_by`、`approved_by`、`approved_at`、`reopen_reason`、`reopen_by`、`reopen_at`。
- 调整类记录必须保留原始单据和关联流水，不能直接覆盖历史。

## 10. 财务控制点与内控校验

| 控制目的 | 校验规则 | 异常处理 |
|---|---|---|
| 单据可追溯 | `order_bill_id`、`out_order_bill_id`、`related_points_record_lid` 必须可定位 | 进入异常明细 |
| 方向可计算 | `amount` 正负必须可解释 | 进入待复核队列 |
| 余额可复算 | `balance_before + amount = balance_after` | 触发重算和告警 |
| 口径可分层 | 明细、余额、汇总、对账必须分层 | 不允许混层展示 |
| 历史不可回写 | 关账后历史不允许直接修改 | 只能冲正或补记 |
| 有效性可过滤 | `if_deal_success` 决定正式口径 | 无效流水不进正式报表 |
| 内外部可区分 | `is_third_party` 必须可筛选 | 分开统计和审计 |
| 业务日/记账日可分离 | `yingyeriqi` 与 `create_time` 必须同时存在 | 避免日结失真 |
| 关账可冻结 | 独立快照表必须记录关账批次、锁定时点、复核人 | 当前版本只输出账内核对结论 |
| 幂等可控 | `source_system + source_event_id/原单号 + account_key + operation_model` 唯一 | 重复请求幂等返回 |
| 状态可判定 | `record_status` 由规则派生且可追溯 | 进入异常池或调整类 |
| 顺序可复现 | 同账户稳定排序键必须存在 | 乱序补录进入异常池 |

## 12. 最终口径建议

如果只保留一版标准口径，建议统一为：

- 事实层：`crm_card_points_record`
- 商户与门店：`mid + sid`
- 账户主键：`mid + card_id`
- 业务类型：`operation_model`
- 标准业务类型：由 `operation_model` 标准化得到
- 方向：由 `amount` 正负派生
- 余额：`balance_before / balance_after`
- 业务时间：`yingyeriqi`
- 记账时间：`create_time`
- 发生门店：`sid + shop_name`
- 状态口径：以 3.5.2 状态矩阵为准，`if_deal_success` 只作为基础入账标识
- 调整类：冲正、红冲、蓝补必须参与余额闭环，但在汇总和对账中单列
- 正式报表：只看有效记账记录和已定义的调整类记录
- 关账规则：关账后历史不回写，所有修正通过后续冲正/补记表达
- 对账定位：当前事实表条件下先做账内平衡核对，正式关账对账需要独立账面余额快照来源
- 稳定排序：当前以 `create_time + pid` 作为过账顺序；若未来出现跨系统补录，应补充 `posting_seq`
- 科目口径：`subject` 只是业务分类文本，不等同正式会计科目编码，若要上升为正式会计科目汇总，需要另建 `subject_code` 字典表
- 正式关账前提：至少需要独立账面余额快照表、关账批次表、对账结果表和复核链路；当前版本只支持账内平衡核对，不支持正式关账结论输出
