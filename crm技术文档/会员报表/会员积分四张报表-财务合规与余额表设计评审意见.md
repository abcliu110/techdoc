# 会员积分四张报表——财务合规与余额表设计评审意见

**日期**: 2026-05-14
**评审人**: Claude
**评审文件**: `会员积分四张报表-基于事实表的实现说明.md`（V5，第四轮复审版）
**评审维度**: 财务规范符合性、数据库实际结构一致性、报表设计合理性

---

## 一、总体评价

文档在**积分账本体系设计**上是严谨的，方向定义（amount 正负）、归期口径（yingyeriqi）、历史不回写原则、快照机制均是财务级规范的核心要素。但在与**实际数据库结构**的对应、以及**快照表（crm_card_points_snapshot）的落地可行性**上存在重大缺口，不解决则无法进入实现阶段。

---

## 二、严重问题（Must Fix）

### F-01：`crm_card_points_snapshot` 表和实体不存在

**问题描述**：

R4 对账报表以 `crm_card_points_snapshot` 表为月结快照来源，R2 余额报表也引用该表做 `snapshot_balance` 字段。但经过对 CRM 源码（`nms4cloud-crm-dao/src/main/java/com/nms4cloud/crm/dao/entity/`）和整个 nms4cloud 项目的搜索：

- **`CrmCardPointsSnapshot.java` 实体类不存在**
- **`crm_card_points_snapshot` 数据库表不存在**
- 文档中定义的快照 INSERT 语句（R4 5.5 节）无法落地

**影响**：R4 对账报表和 R2 余额报表（路径 C）**核心依赖缺失，整套月结机制无法运转**。

**建议**：
1. 设计并创建 `crm_card_points_snapshot` 表（含 snapshot_date/snapshot_type/mid/card_id_alias/points 字段及复合唯一索引）
2. 在 CRM DAO 层新增 `CrmCardPointsSnapshot.java` 实体
3. 在报表服务层封装快照生成 API，供月结流程调用
4. 文档中需补充该表的 ER 图和字段说明

---

### F-02：文档所述字段精度与实际数据库不一致

**问题描述**：

文档 V2 修复记录（S-01）称已将 `balance_before/balance_after` 改为 `DECIMAL(20,2)`、`amount` 改为 `BIGINT`。但经过对源码和 SQL 文件的交叉验证：

| 字段 | 文档所述精度 | 实际精度 |
|------|------------|---------|
| `crm_card_points_record.balance_before` | DECIMAL(20,2) | 无 `@NmsTableField` 元数据注解，无法直接确认；需 DBA 确认 DDL |
| `crm_card_points_record.balance_after` | DECIMAL(20,2) | 同上 |
| `crm_card_points_record.amount` | BIGINT | Java 源码用 `BigDecimal`，若 DDL 用 BIGINT 则存在类型不匹配风险 |
| `crm_card.points` | 未明确 | 源码：`BigDecimal points`（无精度注解）；V20260509 SQL 用 `DECIMAL(18,2)` |
| `crm_card_points_round.points` | 未明确 | V20260509 SQL：`DECIMAL(18,2)` |

**影响**：若 `amount` 在数据库层实际是 `DECIMAL` 而非 `BIGINT`，文档所称 S-01 修复并未真正落地，报表 SQL 中的 SUM 和 ABS 逻辑不受影响（均兼容），但数据类型的声明应与数据库保持一致。

**建议**：
1. 由 DBA 确认 `crm_card_points_record` 各字段的 DDL 精度
2. 在文档中明确标注"精度以 DBA 最终 DDL 为准"，避免声明值与实际不符
3. 若实际 DDL 为 DECIMAL，报表 SQL 中的 `ABS(amount)` 写法完全兼容，无需修改

---

### F-03：`frozen_points` 字段在 `crm_card` 中不存在

**问题描述**：

R2 余额报表（字段清单第 319 行）引用 `crm_card.frozen_points` 作为"冻结积分余额"字段。经过对 `CrmCard.java` 实体源码的完整搜索，**该字段不存在**。

**影响**：R2 余额报表中 `available_points = current_points - frozen_points` 和独立 `frozen_points` 字段**无法落地**，文档用括号注明"冻结机制暂未实现"是一种合理的过渡处理，但应在评审记录中标记为"待建"而非"已有"。

**建议**：
1. 若短期无冻结机制：保留 `available_points = current_points`，新增字段 `frozen_points = 0`，注释"冻结机制建设中"
2. 若短期有冻结需求：将 `frozen_points` 字段加入 `crm_card` 表设计，并在 CRM 实体中同步添加

## 三、一般问题（Should Fix）

### F-04：R4 对账报表的 `adjustment_amount` 字段与主勾稽公式的关系表述歧义

**问题描述**：

R4 字段定义（R5.2 第 652 行）说明 `adjustment_amount`：
> "不单独参与 balance = 期初 + earned - used 的校验，但参与 difference 差异分析"

但在 R4 对账状态判定逻辑（5.3 节）的主公式中：
```
期末积分余额 = 期初 + earned - used
```

与扩展勾稽公式（1.3 节）相比：
```
期末积分负债余额 = 期初 + earned - used - expired + refund_reverse + adjustment
```

两份公式**不一致**：主公式没有包含 adjustment_amount，但差异计算 `difference_amount = snapshot_balance - computed_closing` 中 snapshot 来自 `crm_card.points` 快照（已包含所有调整后的实时值），而 computed_closing 仅包含期初+earned-used，两者天然存在差异，调整额未进主公式会导致 difference 的口径解释复杂化。

**建议**：
1. 统一勾稽公式表述，将 adjustment_amount 纳入主勾稽公式或明确说明差异金额 = snapshot_balance - (期初 + earned - used) 的物理含义是"快照中包含而流水未记录的调整"
2. 差异金额字段应增加说明：若 adjustment_amount ≠ 0 且未进流水，需人工确认差异是否由蓝补/红冲未落流水导致

---

### F-05：R3 期间汇总报表 `closing_balance` 计算未包含 adjustment_amount

**问题描述**：

R3 SQL（步骤 4.3，步骤 556-606）中，`closing_balance` 计算为：
```sql
sid_opening_balance + earned_points - used_points
```

与 R4 主勾稽公式一致，但 R3 的 `adjustment_amount`（蓝补/红冲/冲正）**未参与期末余额计算**，导致同一期间内 R3 和 R4 的期末余额数字可能不一致（蓝补/红冲未落流水时）。

**建议**：在 R3 汇总层增加 `adjustment_amount` 字段（蓝补正向 + 红冲负向 + 冲正代数和），并在 `closing_balance` 计算中纳入：
```sql
closing_balance = sid_opening_balance + earned_points - used_points + adjustment_amount
```

---

### F-06：快照生成语句缺少幂等性保证和事务边界

**问题描述**：

R4 5.5 节的快照 INSERT 语句注释中写有：
```sql
-- ON DUPLICATE KEY UPDATE points = VALUES(points)  -- 幂等重跑
```

但该行被注释掉，未生效。同时缺少：
- **幂等性**：相同 `snapshot_date + mid + card_id_alias + snapshot_type` 重复插入时将触发主键冲突
- **事务边界**：未说明快照生成是单条 INSERT 还是批量 BATCH INSERT
- **删除旧快照**：重跑场景下是否先删后插（TRUNCATE vs DELETE）

**建议**：提供完整的幂等重跑 SQL：
```sql
DELETE FROM crm_card_points_snapshot
WHERE mid = :mid AND snapshot_date = :endDate AND snapshot_type = 'MONTHLY_CLOSE';

INSERT INTO crm_card_points_snapshot (...) VALUES (...);
```

或在 INSERT 中使用 `ON DUPLICATE KEY UPDATE`（需先建唯一索引）。

## 四、建议问题（Nice to Fix）

### F-07：`crm_card.sum_of_points` 与 `crm_card.points` 的关系未明确

**问题描述**：

`crm_card` 表中存在两个相关字段：
- `points`：当前积分余额
- `sum_of_points`：累计积分

R2 余额报表取 `crm_card.points` 作为 `current_points`，但未说明 `sum_of_points` 的用途。财务口径上，两者分别代表"负债余额"和"历史累计"，报表中若需"历史累计发放积分"字段，应从 `crm_card_points_record` 重算而非直接用 `sum_of_points`（后者未在 R2/R3/R4 中出现）。

**建议**：在文档 1.2 节补充说明 `sum_of_points` 与 `points` 的区别，明确报表不依赖 `sum_of_points`，均从流水重算。

---

### F-08：快照机制缺少"快照补打"机制说明

**问题描述**：

月结后发现历史数据问题（蓝补/红冲），可能需要补打快照。但文档未说明：
- 是否支持"补打历史快照"
- 补打后是否需要重新对账
- 快照与对账的版本管理

**建议**：在 5.5 节补充"快照补打"场景处理流程。

## 五、合规亮点（保持）

以下设计已符合财务规范，应予保持：

| 设计点 | 符合规范说明 |
|------|------------|
| 方向由 `amount` 正负派生，`operation_model` 不参与方向判断 | ✅ 借贷分离原则，与储值账本一致 |
| `yingyeriqi` 作为正式归期，`create_time` 仅审计追溯 | ✅ 权责发生制，日志/审计时间分离 |
| 历史流水禁止直接修改，蓝补/红冲通过新增记录表达 | ✅ 审计轨迹不可篡改 |
| 月结前必须先快照，再以快照值对账，禁止用实时 `crm_card.points` | ✅ 防止期间内变动污染对账数据 |
| 浮点容差 ±1 分，差异 > 1 进入异常区 | ✅ 合理的容差处理 |
| R2/R3/R4 期初余额统一取 `balance_after` | ✅ 避免多笔累加导致期初虚高 |
| 积分作为"应付客户对价/合同负债"定性 | ✅ 与 IFRS 15 处理口径一致 |
