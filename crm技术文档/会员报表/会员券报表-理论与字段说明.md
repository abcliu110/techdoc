# 会员券报表理论与字段说明

日期：2026-05-13

## 1. 文档范围

本文档用于说明 CRM 会员券体系的财务理论、报表模型和字段含义，覆盖以下场景：

- 储值时发券
- 积分商城兑换券
- 营销赠券
- 消费核销
- 过期转销
- 退款冲回
- 积分转券对账

本文档中的表模型是建议的逻辑模型，物理表名可按项目命名规范调整，但应保持同一套业务口径。

## 2. 会员券理论口径

### 2.1 券的财务本质

会员券不是简单的营销数据，而是带有财务含义的未来权益。不同来源的券，对应的会计属性可能不同：

- 储值赠券：通常对应未来履约义务或合同负债。
- 积分兑换券：通常对应积分负债向券负债的转换。
- 营销赠券：可能作为销售折扣、营销费用或应付客户对价处理。
- 第三方补贴券：可能对应应收补贴方款项，而不是直接冲减收入。

### 2.2 处理原则

1. 发券不等于确认收入。
2. 核销才是负债释放和收入确认的核心时点。
3. 过期券必须明确如何处理未使用权益。
4. 积分转券必须打通积分负债和券负债。
5. 业务余额和财务负债必须分开建模，不能混成一个字段。
6. 历史流水应只追加，不应直接改写，调整通过冲销或调整事件处理。

### 2.3 标准财务系统中的券分类

在标准财务系统里，券不应只当作“营销工具”，而应按经济实质分成几类处理：

| 券类型 | 财务本质 | 标准处理 |
|---|---|---|
| 充值赠券 | 客户取得的未来权益 | 作为单独履约义务或合同负债管理 |
| 积分兑换券 | 积分权益转换成券权益 | 从“积分负债”转入“券负债”，兑换时通常不直接进损益 |
| 现金购买券 / 储值券 | 预收款 | 记合同负债，使用时确认收入 |
| 消费抵扣券 | 折扣 / 应付客户对价 | 使用时冲减收入或释放负债 |
| 免费营销券 | 营销激励 | 视规则计入销售折扣或营销费用，不能和预收款券混在一起 |
| 第三方补贴券 | 代收代付 / 应收补贴 | 使用时确认应收补贴方款项，不能直接冲掉门店收入 |

核心原则是：**券发放不等于收入，券使用才触发收入、折扣、费用或负债释放。**

### 2.4 标准财务系统中的建模原则

标准财务系统建议这样建模：

1. **券台账是财务子账**  
   每张券要有唯一编号、来源、面值、余额、有效期、适用范围、发放单据、使用订单、作废/过期记录。
2. **按券来源分会计科目**  
   不同券不能混用一个“优惠券金额”字段，至少要区分：
   - 储值赠券负债
   - 积分兑换券负债
   - 现金购买券合同负债
   - 第三方券核销对应的应收补贴款
3. **发券时不直接确认收入**  
   发券只意味着未来权益形成，不能因为“发了券”就直接确认收入。
4. **用券时按订单行释放**  
   消费时要拆到商品、税率、门店、订单行：
   - 储值 / 购买券使用：释放合同负债，确认收入和税额。
   - 折扣券使用：冲减收入或确认营销费用。
   - 第三方券使用：确认应收补贴款。
5. **过期券不能随便进收入**  
   要按准则处理未使用权利。能合理估计客户不会使用的部分，按使用模式确认；否则通常等客户行权可能性极低时再转收入或冲减负债。
6. **月结必须能对账**  
   期初券负债 + 本期新增 - 本期核销 - 本期过期 - 本期冲销 = 期末券负债。
7. **券报表要服务总账**  
   CRM 的券系统要能生成：
   - 券发行明细表
   - 券核销明细表
   - 券余额表
   - 券负债余额表
   - 券过期转销表
   - 积分转券对账表
   - 门店 / 渠道 / 税率维度券核销表

这几条原则和后面的报表模型是一一对应的：明细负责追溯，余额负责对账，统计负责月结，总账负责入账。

### 2.5 记账处理口径

会员券的记账不是“发券即收入”，而是按券的经济实质分阶段处理：

1. **发券时先确认负债**  
   券一旦发放，说明企业对客户形成了未来履约义务。  
   如果券对应储值或预收款，还要把税额单独拆出来，不要和本金混在一起。
2. **核销时再转收入或费用**  
   券在消费时才释放负债。  
   如果券属于交易价格减让，就体现为收入抵减；如果券属于营销补贴，就可能进入销售费用。
3. **过期时处理未使用权利**  
   对长期未使用且可合理估计不会使用的券，可以按预计失效率处理；否则继续挂账，直到满足确认条件。
4. **退款或冲销时做反向分录**  
   如果原订单退款或核销被撤销，必须冲回原先释放的负债、收入或费用，不能直接改历史流水。

#### 2.5.1 常见券的记账方向

| 券类型 | 发放时 | 核销时 | 过期时 |
|---|---|---|---|
| 现金购买券 / 储值券 | 记合同负债，并单列待转销项税额 | 释放合同负债，确认收入和销项税额 | 按未使用权利处理，转收入或继续挂账 |
| 充值赠券 | 若构成客户重大权利，则单独挂负债 | 按履约进度转收入或转费用 | 按预计失效率或未使用权利处理 |
| 积分兑换券 | 先由积分负债转入券负债 | 再按券核销逻辑处理 | 按积分与券的剩余权益处理 |
| 消费抵扣券 | 可能形成折扣或券负债 | 通常冲减收入或确认费用 | 按规则处理未使用部分 |
| 免费营销券 | 多数情形下作为营销补贴管理 | 视政策确认销售折扣或营销费用 | 未使用部分按规则转销 |
| 第三方补贴券 | 可能形成应收补贴方款项 | 同步确认补贴应收和门店收入，不能直接冲掉门店收入 | 按补贴协议处理 |
| 第三方补贴券-发券时 | 可能形成应收补贴方款项（补贴方确认前先挂负债） | — | — |
| 第三方补贴券-核销时 | 同步确认补贴应收和门店收入，不能直接冲掉门店收入 | 贷应收补贴方款项 | 贷主营业务收入（门店口径） |

#### 2.5.2 与总账的对应关系

券系统最终要落到总账，核心对应关系通常如下：

| 券系统字段 | 财务含义 | 总账方向 |
|---|---|---|
| `liability_principal_amt` | 合同负债本金 | 发券时贷方，核销时借方 |
| `liability_tax_amt` | 待转销项税额 | 发券时贷方，核销或转销时借方 |
| `principal_release_amt` | 负债本金释放金额 | 核销时借合同负债，贷收入或相关科目 |
| `tax_release_amt` | 负债税额释放金额 | 核销时转入销项税额 |
| `discount_amount` | 优惠金额 | 作为收入抵减或营销费用 |
| `expense_confirm_amt` | 确认费用金额 | 营销费用或销售费用借方 |
| `revenue_confirm_amt` | 确认收入金额 | 主营业务收入贷方 |
| `gl_voucher_no` | 总账凭证号 | 说明已生成哪张财务凭证 |
| `gl_sync_status` | 总账同步状态 | 说明是否已过账 |

#### 2.5.3 月结记账公式

券负债月结通常要满足下面的勾稽关系，需区分**本金**和**税额**两条独立勾稽线：

**本金勾稽公式：**
`期初本金负债 + 本期新增本金 - 本期核销本金 - 本期过期转销本金 - 本期冲销本金 + 本期退款冲回本金 = 期末本金负债`

**税额勾稽公式：**
`期初税额负债 + 本期新增税额 - 本期核销税额 - 本期过期转销税额 - 本期冲销税额 + 本期退款冲回税额 = 期末税额负债`

> **说明：** 退款冲回项（+本期退款冲回本金/税额）是必须项。退款场景下，原核销释放的负债需原路冲回，若不对退款冲回做单独记录，期末负债余额会失准。本金和税额必须分别勾稽，不能只看净额。

## 5. 各表字段说明

### 5.1 `crm_coupon_type`

**中文名：券类型主数据表**

**模型定位：** 定义一类券的业务规则和会计属性，是券实例、券事件和报表汇总的上游主数据。

**主键声明**：`coupon_type_code VARCHAR(32) PRIMARY KEY`（业务编码为主键，不使用自增ID，以支持分布式部署下的码值唯一性）

| 字段 | 中文名 | 作用 |
|---|---|---|
| `coupon_type_code` | 券类型编码 | 业务唯一编码，用于区分不同券种 |
| `coupon_type_name` | 券类型名称 | 报表展示名称和业务名称 |
| `accounting_class` | 会计分类 | 决定该券按合同负债、折扣券还是营销券处理 |
| `source_type` | 来源类型 | 说明券来自储值、积分兑换、营销赠送等哪种业务 |
| `face_amount` | 券面额 | 券对客户展示的金额或权益面值 |
| `can_partial_use` | 是否支持部分核销 | 说明券是否允许拆分多次使用 |
| `stackable_flag` | 是否可叠加使用 | 说明券是否可与其他优惠同用 |
| `tax_rate_policy` | 税率策略 | 说明该券核销时如何拆分税额 |
| `validity_type` | 有效期类型 | 说明是固定日期、固定天数还是相对有效期 |

| `issue_doc_no` | 发券单据号 | 追溯发券来源业务单 |

| `source_biz_no` | 来源业务单号 | 追溯原始业务单据 |
| `store_id` | 门店ID | 说明券由哪个门店或核算门店发出 |

| `face_amount` | 券面额 | 券的原始权益金额 |

| `current_tax_balance` | 当前负债税额余额 | 说明尚未释放的税额负债 |
| `created_at` | 创建时间 | 券实例入库时间 |
| `updated_at` | 更新时间 | 券实例最后变更时间 |

### 5.3 `crm_coupon_event`

**中文名：券事件流水表**

**模型定位：** 记录券生命周期中的每一次动作，是发券、核销、过期、作废、退款冲回的原始事实层。

**索引策略：**
- `UNIQUE INDEX uk_event_no (event_no)`：业务流水号唯一约束，保证每条事件有唯一追溯码，不可重复。
- `idx_coupon_instance_id`：关联查询（按券实例查找所有事件）。
- `idx_event_time`：按时间范围查询（月结批量处理）。
- `idx_event_type`：按事件类型筛选（统计某类事件）。
- 复合索引 `idx_instance_type_time`：覆盖（券实例ID + 事件类型 + 事件时间）组合查询。

**存储引擎与分区策略：**
- 存储引擎：`InnoDB`（必须，不可用 Memory，因财务数据要求事务一致性和持久化）。
- 主键策略：`id` 使用雪花算法生成（如 SnowFlake），避免高并发写入时 AUTO_INCREMENT 聚簇索引尾部锁竞争。
- 分区策略：按 `event_time` 月分区（`PARTITION BY RANGE (TO_DAYS(event_time))`），配合 `idx_event_time` 索引，提升月结批量扫描效率。建议保留最近12个月分区，超期数据归档至冷存储。
- 写入模式：纯追加（INSERT ONLY），禁止 UPDATE/DELETE 操作；调整通过新增冲销事件（event_type=5 退款冲回）实现。

**重要约束：** 历史事件只追加（INSERT），不允许 UPDATE 或 DELETE。调整通过新增冲销事件（`event_type=5` 退款冲回）实现，保证审计追溯完整性。

### 5.4 `crm_coupon_redemption_alloc`

**中文名：券核销分摊表**

**模型定位：** 一次券核销往往要分摊到订单行、商品、税率和门店，因此这里记录的是核销的最细分摊层。

**分摊规则说明**（重要）：
- 分摊方式：按订单行商品金额比例分摊，即该行 `redeem_amount` = 券核销金额 ×（本行订单金额 / 订单总金额），各行加总必须等于核销事件的 `event_amount`。
- 分摊校验：所有分摊行的 `redeem_amount` 之和 = 核销事件 `event_amount`（允许 ±0.01 元容差，用于处理除不尽情况）。
- 部分核销：`coupon_instance.current_face_balance` 在核销事件后由应用层实时更新；过期/作废后置零；余额试算以 `crm_coupon_event` 事件流实时聚合为准。
- 税率优先级：`crm_coupon_redemption_alloc.tax_rate` 为订单行实际适用税率（来自商品税率），与 `crm_coupon_instance.tax_rate`（券类型默认税率）可能不同；若商品行税率变更（如跨境商品），以本字段为准。

`difference_amt` = coupon_face_amount - points_value_amt，表示积分价值与券面值之间的差额。差额处理规则：
- 若 coupon_face_amount > points_value_amt（券面额大于积分价值）：差额由营销费用承担，贷记"销售费用-积分兑换补贴"。

| `expense_confirm_amt` | 确认费用金额 | 记录当天确认了多少费用 |
| `source_hash` | 来源数据哈希 | 校验统计数据是否来自同一批明细 |
| `freeze_time` | 报表冻结时间 | 说明统计结果何时冻结 |
| `created_at` | 创建时间 | 统计行写入时间 |

## 6. 这些表之间的关系

以下为外键依赖链的显式声明，财务合规场景下建议在应用层维护参照完整性（不建议依赖数据库外键约束，以避免跨库迁移时失效）：

| 子表 | 父表 | 外键字段 | 约束说明 |
|------|------|---------|---------|
| `crm_coupon_instance` | `crm_coupon_type` | `coupon_type_id` | 券实例必须关联已定义的券类型，禁止悬挂 |
| `crm_coupon_event` | `crm_coupon_instance` | `coupon_instance_id` | 事件必须关联已存在的券实例 |
| `crm_coupon_event` | `crm_coupon_type` | `coupon_type_id` | 事件记录时可携带券类型快照以便汇总 |
| `crm_coupon_redemption_alloc` | `crm_coupon_event` | `redeem_event_id` | 核销分摊必须关联到一次核销事件 |
| `crm_coupon_redemption_alloc` | `crm_coupon_instance` | `coupon_instance_id` | 核销分摊必须关联到具体券实例 |

**数据一致性保障说明**：
- `coupon_instance` 表的 `coupon_status`、`current_face_balance`、`current_principal_balance`、`current_tax_balance` 由业务系统在事件发生时同步更新（同一事务内）。
- 事件消费和实例更新是同一个事务边界，不允许出现"事件已记录但实例状态未更新"的情况。
- 异常场景修复：通过新增反向冲销事件（event_type=5）修复，不允许直接 UPDATE 历史事件或实例状态。

## 7. 月结闭环

券报表的月结顺序建议如下：

1. 收集当月所有券事件。
2. 按券实例生成发行明细和核销分摊。
3. 按日生成余额快照和负债快照。
4. 汇总积分转券对账结果。
5. 冻结当期报表批次。
6. 生成总账凭证或过账数据。
7. 做总账对账。
8. 历史期间只允许反向冲销，不允许直接改旧流水。

## 8. 结论

会员券报表的核心不是“把券统计出来”，而是把券从业务权益、财务负债、核销收入、税额分摊到总账过账完整串起来。  
只要这套模型成立，后续无论是储值发券、积分兑换券，还是营销赠券和过期转销，都可以在同一条闭环里解释清楚。
