## 三、问题详情

### 3.1 🔴 P0 — 门店名称全部为空

**现象：** 左侧表格中"门店名称"列 8 行数据全部显示 `—`。

**影响：**
- 无法按门店汇总收款金额
- 财务对账时无法按门店轧账
- 报表失去了最基本的地域分析维度

**根本原因推测：** 线上充值订单的门店归属逻辑未定义，或门店字段未从业务表映射到报表层。

---

### 3.2 🔴 P0 — 赠送金额逻辑极度异常

**现象：** 部分记录的赠送金额远高于合理范围，导致实收金额极低或异常：

| 充值本金 | 赠送金额 | 实收金额 | 折扣率 | 问题 |
|---------|---------|---------|:-----:|------|
| 100.01 | 100.00 | **0.01** | ≈0% | 本金几乎全返，财务无法处理 |
| 100.00 | 100.00 | 0.00 | 0% | 相当于充100送100 |
| 600.00 | 100.00 | 500.00 | 83% | 正常折扣 |
| 500.00 | 100.00 | 400.00 | 80% | 正常折扣 |
| 100.00 | 100.00 | 0.00 | 0% | 相当于充100送100 |
| 600.00 | 100.00 | 500.00 | 83% | 正常折扣 |
| 100.00 | —（空） | 100.00 | 100% | 无赠送，原价充值 |
| 100.00 | 100.00 | **200.00** | 200% | 实收金额超过充值本金 |

**具体问题：**

1. **行1（充值100.01元）：** 赠送100元后实收仅0.01元。在会计处理上需要拆分为两笔记账分录——本金收入 0.01 元 + 赠送负债 100 元（待兑换的营销成本），但报表中无法体现这种拆分逻辑。

2. **行8（充值100元，实收200元）：** 实收金额超过充值本金，相当于全额退款后额外赠送100元。这超出了常规充值赠送的会计处理范围，需确认是否为系统Bug或营销配置错误。

3. **行7（无赠送）：** 同样金额（100元），但赠送金额字段为空，与其他行口径不一致。

**财务影响：** 赠送金额合计 600 元，占充值本金（2200.01元）的 27.3%，占比偏高，若赠送策略与财务核算口径不匹配，将导致营销成本无法准确归集。

---

### 3.3 🟠 P1 — 单号格式异常

**现象：**

| 行号 | 单号显示值 | 问题 |
|:---:|-----------|------|
| 1 | 0 | ❌ 无效单号，0不应为业务单号 |
| 2 | 0 | ❌ 无效单号 |
| 3 | 10005 | 疑似自增序号，非标准业务流水号 |
| 4 | 10004 | 序号格式，非业务单号 |
| 5 | 10003 | 序号格式 |
| 6 | 10002 | 序号格式 |
| 7 | 10001 | 序号格式 |
| 8 | 10000 | 序号格式 |

**问题分析：**
- 前两行单号为 0，属于无效数据，需排查是测试遗留还是系统写入错误。
- 后续 6 行虽为 5 位数字，但属于自增序号，而非带日期前缀或门店前缀的标准业务流水号，追溯能力弱。
- 财务对账要求单号可追溯到具体收款凭证，当前单号无法满足此要求。

---

### 3.4 🟠 P1 — 操作人全部为空

**现象：** "操作人"列 8 行数据全部为空。

**财务风险：**
- 充值涉及资金收取，无操作人记录意味着无法追溯责任人。
- 内部财务审计无法确认每笔充值由谁操作。
- 若出现金额异常，难以定责。

**根本原因：** 会员卡充值操作在写入时未同步记录操作人信息，需在充值交易写入逻辑中补充操作人字段。

---

### 3.5 🟠 P1 — 无法区分充值与退款

**现象：**
- 查询条件中存在"操作类型"筛选项（支持按充值/退款筛选）
- 但表格中无"操作类型"列，无法直观区分正向充值和负向退款

**风险：** 若存在退款记录（金额为负或单独标记），其金额可能被混入正向汇总，导致总收入数字失真。

---

### 3.6 🟡 P2 — 实时余额格式不规范

**现象：** "实时余额"列当前格式为 `0 → 100`（变动前余额 → 变动后余额合并显示）。

**问题：**

| 问题 | 说明 |
|------|------|
| 信息过载 | 一格塞入两值，不符合单列单值的财务规范 |
| 无列头区分 | 无法判断 0 是"前"还是"后" |
| 大数值截断 | 余额较高时（如 27400 → 27500）可能溢出显示 |
| 无法直接引用 | 财务人员无法直接引用该数据入账 |

**建议：** 拆分为两列——`充值前余额` + `充值后余额`，各列独立显示单值。

---

### 3.7 🟡 P2 — 备注列全部为空

**现象：** "备注"列 8 行全部为空。

**影响：** 特殊交易（如促销优惠、异常退款、冲正等）无任何说明，财务核对时无法判断笔笔异常。

---

### 3.8 🟡 P2 — 交易无时间精度

**现象：** "交易日期"仅显示到日期（2026-05-13），无时分秒。

**影响：**
- 无法按收银班次（早班/中班/晚班）进行日结对账
- 跨日交易（如23:50充值次日确认）会产生日期归属争议

---

### 3.9 🟡 P2 — 付款方式口径不统一

**现象：** 表中出现的付款方式为"微信""现金""储值卡"。

**问题：** 口语化表达无统一编码，后续按付款方式做财务汇总时，容易因口径不一致导致数据失真。

**建议统一为：**

| 当前写法 | 建议口径 |
|---------|---------|
| 微信 | 微信支付 / WeChat Pay |
| 现金 | 现金 / CASH |
| 储值卡 | 储值卡 / Stored Value Card |

---

### 3.10 🟡 P3 — 会员姓名无识别度

**现象：** 8 条记录中"会员姓名"几乎全为"微信会员"。

**原因分析：** 系统从微信渠道获取的会员信息未要求实名认证，姓名字段未从微信 API 拉取或同步。

**影响：** 报表无法识别具体会员身份，财务核对和会员营销均无法使用此字段。

---

## 四、数据链路追踪

```
微信充值入口
    ↓
会员注册（仅采集手机号，无姓名认证）
    ↓
充值订单创建
    ├── 单号 = 0（错误）或 自增序号（非标准流水号）
    ├── 门店 = 空（未写入）
    ├── 操作人 = 空（未写入）
    └── 备注 = 空（未记录）
    ↓
会员卡变更记录
    ├── 充值本金 = 正常
    ├── 赠送金额 = 异常（逻辑不统一）
    └── 实时余额 = 格式不规范
    ↓
报表查询
    ├── 缺失字段：门店名称、操作人、备注
    ├── 异常字段：单号、赠送金额
    └── 格式问题：实时余额、交易时间精度
```

> **结论：** 这张报表的问题不是某一层单点故障，而是**整条业务链路每个环节都存在数据缺失或写入不规范**。门店、操作人、备注三个字段在底层就没有写入，建议从**充值交易创建的那一刻**开始排查数据埋点和写入逻辑。

## 六、建议改进方向

### 立即修复（P0）

1. **修复门店字段映射** — 排查充值订单表与门店表的关联关系，确保门店信息正确写入。
2. **重新审核赠送金额计算逻辑** — 确认促销规则配置是否正确，充值本金与赠送金额的比例是否在合理范围内（如不超过 1:1）。
3. **修复单号为0的数据** — 排查前两行订单的写入逻辑，确认为何单号为0。

### 短期修复（P1）

4. **补充操作人字段** — 在会员卡充值操作写入时，同步记录操作人信息。
5. **规范单号格式** — 改为带日期前缀的标准业务流水号（如 `RC-20260513-00001`）。
6. **增加操作类型列** — 在表格中增加"操作类型"列，区分充值/退款/赠送。
7. **实时余额拆列** — 拆分为"充值前余额"和"充值后余额"两列。

### 中期优化（P2）

8. **补充时分秒精度** — 交易时间精确到分，便于班次对账。
9. **统一付款方式编码** — 建立标准化的支付方式字典。
10. **备注字段补录** — 为所有充值记录补充必要的业务备注。

### 8.4 第三组：财务金额信息（核心）

| 15 | 优惠金额 | discount_amount | DECIMAL(12,2) | 计算字段：充值本金 - 实收金额 | 本次优惠总额（含赠送折算），营销费用归集依据 | 同上 | ≥0 | 显示"-" | ✅ |

**字段间勾稽关系（必须校验）：**

```
充值场景：
  充值本金 = 实收金额 + 优惠金额
  赠送金额 ≤ 充值本金（建议规则：赠送比例不超过1:1）
  实收金额 > 0（不允许0元充值，除非营销活动）

退款场景：
  本金退款金额 ≤ 本金余额
  赠送退款金额 = 0（赠送不可退，或按系统规则）

汇总校验：
  期末本金余额 = 期初本金余额 + 本期充值本金合计 - 本期本金退款合计 - 本期消费本金合计
  期末赠送余额 = 期初赠送余额 + 本期新增赠送合计 - 本期赠送消费合计 - 本期过期赠送合计
```

## 九、财务视角：余额应如何展示（详细说明）

> 本节从财务会计角度，系统说明会员账户余额的展示原则和业务逻辑依据。

### 9.1 会员账户的财务本质

会员账户在财务上不是单一的账户，而是**两个性质完全不同的子账户**的组合：

```
会员账户余额（对外展示的总余额）
├── 本金子账户  →  会计科目：预收账款 — 客户存款
│   └── 性质：会员自有资金，商家对客户的负债，原则上可退款
└── 赠送子账户  →  会计科目：应付客户奖励 / 递延营销成本
    └── 性质：商家赠送给客户的营销成本，已承诺须兑现，通常不可退款
```

| 余额类型 | 会计科目 | 退款政策 | 有效期 | 消费限制 |
|---------|---------|---------|--------|---------|
| 本金余额 | 预收账款 | ✅ 可退款 | 依会员协议 | 无 |
| 赠送余额 | 应付客户奖励 | ❌ 不可退 | 通常有有效期 | 可能有（见系统规则） |
| 冻结余额 | 冻结款项 | 依冻结原因 | 依冻结原因 | 不可用 |

> **重要结论：** 本金余额和赠送余额在会计处理、退款政策、有效期管理上完全不同，**不能合并为一个总余额显示**，否则财务入账将产生错误。

---

### 9.2 单笔交易中的余额展示结构

以充值交易为例，财务视角的完整展示如下：

```
【交易信息】
  交易日期：2026-05-13 14:32:15
  单号：RC-20260513-00001
  操作人：李四（收银员A）
  门店：XX门店

【金额信息】
  充值本金：   +¥600.00    →  预收账款增加
  赠送金额：   +¥100.00    →  应付客户奖励增加
  实收金额：   ¥500.00      →  银行存款增加

【余额变动】
  ┌─────────────────────────────────────────────────────┐
  │ 充值前  │  本金余额：¥0.00    │  赠送余额：¥0.00     │  可用总余额：¥0.00 │
  ├─────────────────────────────────────────────────────┤
  │ 本次    │  +¥600.00          │  +¥100.00           │  +¥700.00         │
  ├─────────────────────────────────────────────────────┤
  │ 充值后  │  本金余额：¥600.00  │  赠送余额：¥100.00  │  可用总余额：¥700.00│
  └─────────────────────────────────────────────────────┘
```

> 每一笔交易必须清晰展示：充值前余额 → 本次变动 → 充值后余额，且本金和赠送**分行独立显示**。

---

### 9.3 消费扣减时的余额展示规则

当会员消费时，本金和赠送的扣减顺序必须明确，并体现在报表中：

**扣减优先级规则：**

| 优先级 | 扣减顺序 | 规则说明 |
|:---:|---------|---------|
| 1 | 先扣赠送余额 | 赠送余额通常有有效期限制，先消耗临近过期的赠送 |
| 2 | 再扣本金余额 | 本金无有效期限制，后消耗 |

**消费场景示例：**

```
会员消费 ¥300.00，当前余额构成：
  本金余额：¥600.00
  赠送余额：¥100.00
  可用总余额：¥700.00

本次消费扣减：
  赠送余额：  ¥100.00 → ¥0.00    （扣减 ¥100.00，绿色标注 +¥100.00 转入已消耗）
  本金余额：  ¥600.00 → ¥400.00  （扣减 ¥200.00）
  可用总余额：¥700.00 → ¥400.00

余额变动列展示：
  ┌────────────────────────────────────────┐
  │ 消费前本金余额：¥600.00                │
  │ 消费前赠送余额：¥100.00                │
  │ 消费后本金余额：¥400.00                │  ← 红色标注 -¥200.00
  │ 消费后赠送余额：¥0.00                  │  ← 红色标注 -¥100.00
  │ 可用总余额：¥400.00                   │
  └────────────────────────────────────────┘
```

> **禁止规则：** 赠送余额的扣减顺序不能由会员自行选择，否则存在套现风险。扣减规则必须由系统强制执行。

### 9.5 报表余额列的财务展示标准

综合以上所有原则，报表中余额相关列的展示必须满足以下标准：

| 标准 | 说明 |
|------|------|
| **分类展示** | 本金余额和赠送余额必须分行/分列展示，不能合并 |
| **方向标注** | 增加用绿色 +，减少用红色 -，方向一目了然 |
| **前后对照** | 每笔交易必须同时展示充值前余额和充值后余额 |
| **实时查询** | 支持查询任一时刻的当前实时余额（本金/赠送/总余额） |
| **有效期标注** | 赠送余额必须标注到期日期（或显示"永久"） |
| **可追溯** | 每笔余额变动必须可追溯到具体的交易单号 |
| **可加总** | 报表底部的本金余额合计和赠送余额合计必须与科目账一致 |

**禁止的展示方式：**

```
❌ 禁止 1：合并总余额，不区分本金和赠送
   余额：¥700.00

❌ 禁止 2：一格塞入多值，无法对齐和加总
   余额：0 → 700

❌ 禁止 3：只展示变动额，不展示前后余额
   本次变动：+¥700.00（无法判断充值前是多少）

❌ 禁止 4：不标注余额增减方向
   充值后本金余额：¥600.00（看不出是增是减）
```
