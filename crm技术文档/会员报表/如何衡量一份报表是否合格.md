# 如何衡量一份报表是否合格

> 报表不是"能跑出数据"就算完成。一份合格的企业级报表必须通过**五个维度**的系统化验收。
> 本文提供可直接用于评审的检查清单和评分标准。

---

## 一、评估框架总览

```
                        报表质量五维模型

          数据准确性 ★★★★★（一票否决）
         ╱
        ╱   财务合规性 ★★★★
       ╱   ╱
      ╱   ╱   可追溯性 ★★★★
     ╱   ╱   ╱
    ╱   ╱   ╱   可用性 ★★★
   ╱   ╱   ╱   ╱
  ╱   ╱   ╱   ╱   安全性 ★★★
 ╱   ╱   ╱   ╱   ╱
━━━━━━━━━━━━━━━━━━━━
```

| 维度 | 权重 | 核心问题 | 不合格的后果 |
|------|------|---------|------------|
| **数据准确性** | 一票否决 | 数字对不对？ | 财务报告失真、审计意见保留 |
| **财务合规性** | 30% | 符不符合会计准则？ | 监管处罚、审计不通过 |
| **可追溯性** | 25% | 能不能查到来源？ | 无法审计、无法排错 |
| **可用性** | 20% | 好不好用？ | 用户弃用、效率低下 |
| **安全性** | 25% | 数据安不安全？ | 数据泄露、合规风险 |

## 二、维度一：数据准确性（一票否决）

### 检查清单

| # | 检查项 | 验证方法 | 通过标准 |
|---|--------|---------|---------|
| A1 | **恒等式校验** | `balance_before + amount = balance_after` 逐行校验 | 差异行 = 0（容差 ±1） |
| A2 | **期间接续** | 本期期初 = 上期期末 | 严格相等 |
| A3 | **交叉验证** | R1 明细求和 = R3 汇总对应值 | 严格相等 |
| A4 | **外部对账** | 报表汇总 = 总账科目余额 | 差异在重要性阈值内 |
| A5 | **手工抽样** | 随机抽取 N 笔明细，人工追溯原始单据核实 | 抽样准确率 100% |
| A6 | **边界值测试** | 首日/末日/跨月/零积分/负余额等极端场景 | 结果符合预期 |
| A7 | **精度一致性** | 数据库→后端→前端→导出全链路精度一致 | 无精度丢失 |
| A8 | **合计行验证** | 合计行 = 全量数据求和，非仅当前页 | 严格相等 |

**自动化验证 SQL 示例**：

```sql
-- A1：恒等式全量校验
SELECT COUNT(*) AS violation_count
FROM crm_card_points_record
WHERE mid = :mid
  AND if_deal_success = 1
  AND ABS(balance_before + amount - balance_after) > 1;
-- 通过标准：violation_count = 0

-- A2：期间接续验证
SELECT 
    curr.recon_period,
    curr.opening_balance,
    prev.computed_closing AS prev_closing,
    curr.opening_balance - prev.computed_closing AS break_amount
FROM rpt_points_recon curr
JOIN rpt_points_recon prev 
  ON curr.mid = prev.mid
  AND curr.recon_period = DATE_FORMAT(
      DATE_ADD(STR_TO_DATE(CONCAT(prev.recon_period,'-01'),'%Y-%m-%d'), 
      INTERVAL 1 MONTH), '%Y-%m')
WHERE ABS(curr.opening_balance - prev.computed_closing) > 1;
-- 通过标准：0 行返回

-- A3：R1→R3 交叉验证
SELECT 
    r3.sid,
    r3.earned_points,
    SUM(CASE WHEN r1.amount > 0 THEN r1.amount ELSE 0 END) AS r1_earned,
    r3.earned_points - SUM(CASE WHEN r1.amount > 0 THEN r1.amount ELSE 0 END) AS diff
FROM r3_result r3
JOIN crm_card_points_record r1 
  ON r3.sid = r1.sid AND r3.mid = r1.mid
WHERE r1.if_deal_success = 1
  AND r1.yingyeriqi BETWEEN :startDate AND :endDate
GROUP BY r3.sid, r3.earned_points
HAVING ABS(diff) > 0;
-- 通过标准：0 行返回
```

## 三、维度二：财务合规性

### 检查清单

| # | 检查项 | 通过标准 |
|---|--------|---------|
| F1 | **口径文档完整** | 有正式的口径定义文档，含计算公式，有财务签字 |
| F2 | **归期口径正确** | 使用权责发生制归期（`yingyeriqi`），非入库时间 |
| F3 | **红冲蓝补机制** | 历史数据不直接修改，通过冲正/补记处理 |
| F4 | **关账控制** | 已关账期间数据不可变，有关账状态标识 |
| F5 | **NULL 与 0 区分** | NULL（未发生）和 0（发生但为零）展示有区别 |
| F6 | **负债口径正确** | 积分负债 = 期初 + 新增 - 兑换 - 过期（IFRS 15） |
| F7 | **多期间可比** | 口径变更时提供新旧口径对照说明 |
| F8 | **快照机制** | 对账使用快照值而非实时值 |

### 常见不合格案例

| 问题 | 表现 | 影响 |
|------|------|------|
| 口径未经财务确认 | 技术自定义"收入"="充值金额" | 充值不是收入（是预收款），报表数字虚高 |
| 归期混用 | 同一报表中 WHERE 条件混用两个时间字段 | 同一笔交易可能被重复统计或遗漏 |
| 直接修改历史 | UPDATE 了已关账月份的流水记录 | 该月报表不可重现，审计无法追溯 |
| 合计行含异常数据 | 合计行包含了 `if_deal_success=0` 的数据 | 汇总金额偏高 |

## 四、维度三：可追溯性

### 检查清单

| # | 检查项 | 通过标准 |
|---|--------|---------|
| T1 | **下钻链路完整** | 汇总 → 明细 → 原始单据，全链路可穿透 |
| T2 | **字段血缘清晰** | 每个字段可追溯到源表.源字段 + 加工规则 |
| T3 | **冲正可追溯** | 冲正/补记记录通过 `related_points_record_lid` 关联原始记录 |
| T4 | **操作日志完整** | 谁在何时查询/导出了什么报表，有完整日志 |
| T5 | **口径版本可查** | 口径变更历史可查，能知道某个月用的是哪个版本的口径 |
| T6 | **数据源可标识** | 报表脚注注明数据来源、归期口径、筛选条件 |

### 追溯链路示例

```
R4 对账报表发现差异 ¥50
  ↓ 查看差异流水
R1 发现 3 笔异常记录（balance_before + amount ≠ balance_after）
  ↓ 查看原始单据
crm_deal_task 找到对应的 task_id
  ↓ 查看业务上下文
dwd_bill 找到对应的消费账单
  ↓ 定位根因
POS 端并发记账导致 balance_before 取值错误
  ↓ 修复
通过蓝补/红冲生成修正记录，重新对账
```

> [!IMPORTANT]
> **如果追溯链路中任何一环断裂（如找不到 task_id、关联流水 lid 为空），这份报表就不具备审计可追溯性，属于不合格。**

## 五、维度四：可用性

### 检查清单

| # | 检查项 | 通过标准 |
|---|--------|---------|
| U1 | **查询响应时间** | 常规查询 P95 < 3秒；大数据量 < 10秒 |
| U2 | **分页正确** | 前后翻页无重复、无遗漏 |
| U3 | **导出功能** | 支持 Excel 导出，含表头/合计/脚注，格式正确 |
| U4 | **筛选条件回显** | 用户能看到当前报表使用了什么筛选条件 |
| U5 | **空数据提示** | 无数据时显示"暂无数据"，非空白或报错 |
| U6 | **排序稳定性** | 多次查询同一条件，行顺序不变 |
| U7 | **列格式规范** | 金额右对齐+千分位；日期统一格式；手机号脱敏 |
| U8 | **表头信息完整** | 报表名称、期间、生成时间、组织名称齐全 |

## 六、维度五：安全性

### 检查清单

| # | 检查项 | 通过标准 |
|---|--------|---------|
| S1 | **行级权限** | 店长只看本店，区域看所辖，总部看全量 |
| S2 | **列级脱敏** | 手机号/身份证默认脱敏（`138****1234`） |
| S3 | **导出审计** | 导出操作记录在审计日志中（谁/何时/什么条件/多少行） |
| S4 | **接口鉴权** | 报表 API 需登录 + 权限校验，无匿名访问 |
| S5 | **SQL 注入防护** | 所有参数使用预编译（`:param`），无字符串拼接 |
| S6 | **导出量限制** | 单次导出有最大行数限制（如 10 万行） |

## 七、综合评分表

可直接用于报表评审会议：

| 评估维度 | 满分 | 得分 | 关键发现 |
|---------|------|------|---------|
| 数据准确性（一票否决） | 必过 | □通过 □不通过 | |
| 财务合规性（8项） | 40分 | /40 | |
| 可追溯性（6项） | 20分 | /20 | |
| 可用性（8项） | 20分 | /20 | |
| 安全性（6项） | 20分 | /20 | |
| **总分** | **100** | **/100** | |

**评级标准**：

| 等级 | 分数 | 说明 |
|------|------|------|
| A（合格） | ≥ 85 且准确性通过 | 可上线 |
| B（有条件合格） | 70-84 且准确性通过 | 限期整改后上线 |
| C（不合格） | < 70 或准确性不通过 | 不允许上线 |

> [!CAUTION]
> **数据准确性是一票否决项**。即使其他四个维度都满分，只要恒等式校验不通过、或交叉验证有差异，报表评级直接为 C（不合格），不允许上线。这是财务级报表与普通业务报表最本质的区别。
