## 1. 数据来源架构

### 1.1 数据流向

```
POS 本地结账
  dwd_bill（账单主表） → crm_consume_points_grant（本地积分发放记录）
  dwd_pay（支付明细）
  biz_pay_way（支付方式配置）
  crm_points_rule（POS积分规则镜像）
          ↓ 上传
云端 CRM 落账
  crm_deal_task（消费任务 lid / operation_model / give_point）
  crm_card（会员卡主表：points 当前余额 / sum_of_points 累计）
  crm_card_points_record ← 【唯一事实表】（积分流水：amount / balance_before / balance_after）
```

### 1.2 各表在报表中的角色

| 表 | 报表角色 | 核心用途 |
|---|---------|---------|
| `crm_card_points_record` | 唯一事实表 | 四张报表的共同数据源，记录每一次积分记账事件 |
| `crm_card` | 账户余额来源 | 提供 `points`（当前积分余额）作为"账面余额"字段 |
| `crm_deal_task` | 消费任务上下文 | `operation_model`、`give_point`、`task_id`，关联储值消费流水 |
| `dwd_bill` | POS 消费明细 | 账单金额、菜品明细，用于分析积分与消费的关系 |
| `crm_member` | 会员维度 | 姓名、手机号、会员等级 |
| `crm_card_type` | 卡类型维度 | 卡类型名称、等级 |
| `biz_pay_way` | 支付方式维度 | `can_integral` 标记是否参与积分 |

### 1.3 积分财务口径

积分在财务上属于**应付客户对价**，是一种合同负债/或有负债，其生命周期如下：

```
积分发放 → 积分负债增加（应付积分义务形成）
    ↓
积分核销(消费/兑换) → 积分负债释放（权益兑现，冲减收入或进入成本）
    ↓
积分过期 → 积分负债释放（未使用权利，按 IFRS 15 处理）
```

**积分负债月结勾稽公式**：

```
期末积分负债余额 = 期初积分负债余额 + 本期新增积分负债 - 本期核销积分 - 本期过期积分
```

> ⚠️ 阶段一无独立积分负债表时，`期末积分负债余额` 以 `crm_card.points` 的快照值为准（见 R4 对账报表）。月结前必须先对 `crm_card.points` 做快照锁定，再用快照值与流水计算值对账，禁止直接用实时 `crm_card.points` 进行月结对账。

**扩展勾稽公式（含退款/冲正场景）**：

```
期末积分负债余额 = 期初 + 本期新增 - 本期核销 - 本期过期 + 本期退款冲回 + 本期冲正调整
```

- **退款冲回（Refund/CancelOrder）**：原核销已减少负债，退款时负债回滚（+）。`crm_card_points_record` 中记录为 `amount > 0` 的退款流水，直接进入"本期增加"或单独列示"退款冲回"。
- **蓝补（AccountingBluePatch）**：正向调增，`amount > 0`，增加负债。
- **红冲（AccountingRedFlush）**：逆向调减，`amount < 0`，减少负债。
- **冲正（Reverse）**：对原错误记录的抵销，按实际 `amount` 方向处理，正值加、负值减。
- ⚠️ 历史期间关账后，**禁止直接修改历史流水**，所有修正通过新增冲正/补记记录表达。

**年度关账**：

- 年度关账时，所有月度快照汇总为年度快照，未过期积分负债继续挂账。
- 年度关账前必须完成12个月的对账，确保月结闭环全部收敛。
- 年度关账凭证通常以月结凭证汇总生成，不需要对积分系统单独出年度凭证。
- 若系统中存在跨年积分（如2026年发放2028年才过期），年度关账不影响其有效期，负债按原有效期挂账。

### 1.4 积分与储值的关系

`crm_card_points_record`（积分流水）与 `crm_card_record`（储值消费流水）通过 `crm_deal_task.task_id` 关联，但**独立落账**：

- 储值消费（`crm_card_record.operation_model` 为 `Consume/Charge` 等）触发积分发放时，`crm_deal_task` 同时生成 `crm_card_record` 和 `crm_card_points_record`，两者共享同一个 `task_id`。
- 退款时：`crm_card_record` 和 `crm_card_points_record` **分别处理**，储值退款不自动冲销积分，积分退款也不自动冲销储值。
- 两者各自的余额（`crm_card.balance` 和 `crm_card.points`）独立维护，不可互相推导。

> ⚠️ 阶段一无跨表退款关联机制时，退款场景需人工确认积分是否需要同步处理。

### 1.5 事实表关键字段

`crm_card_points_record` 是唯一事实表，以下字段在四张报表中均使用：

| 字段 | 财务语义 | 说明 |

| `amount` | 记账发生额 | 正=增加，负=减少，由正负表达方向 |
| `balance_before` | 变动前余额 | 本次记账前积分余额 |

## 2. 第一张：流水明细报表

### 2.1 报表定位

流水明细报表回答：**"这笔积分为什么发生、发生了多少、发生后余额是多少"**。

这是账本原始明细，服务审计、客服查账、问题追溯。

### 2.3 标准业务类型标准化映射

`operation_model` 是原始业务类型字典，报表展示需标准化：

| 原始 operation_model | 标准业务类型 | 方向 |
|---------------------|------------|------|
| `Charge` | 充值 | 增加 |
| `ChargeCancel` | 撤销充值 | 减少 |
| `Consume` | 消费 | 减少 |
| `CancelOrder` | 订单取消 | 增加 |
| `Refund` | 订单退款 | 增加 |
| `Exchange` | 积分兑换 | 减少 |
| `TransferOut` | 卡转出 | 减少 |
| `TransferIn` | 卡转入 | 增加 |
| `Return` | 退卡 | 减少 |
| `Withdrawal` | 提现 | 减少 |
| `Repay` | 回款 | 增加 |
| `Reverse` | 反回款 | 增加 |
| `LevelUpg` | 会员升级 | 增加 |
| `Join` | 开卡 | 增加 |
| `CashBack` | 返现 | 增加 |
| `CancelCashBack` | 撤销返现 | 减少 |
| `AccountingBluePatch` | 蓝补 | 增加 |
| `AccountingRedFlush` | 红冲 | 减少 |
| `Expire` | 积分过期 | 减少 |
| `Initial` | 期初 | 视导入规则 |

> **积分过期（Expire）说明**：若积分存在有效期，过期时应产生一条 `amount < 0` 的过期记录，标准业务类型记为"积分过期"。过期积分在 R3 期间汇总中单独统计（纳入"本期减少"），在 R4 对账中按"积分负债释放"口径处理。

### 2.4 SQL 实现

```sql
SELECT
    cpr.sid                                  AS 发生门店ID,
    cpr.shop_name                            AS 发生门店名称,
    cpr.yingyeriqi                           AS 业务日期,
    cpr.create_time                          AS 记账时间,
    cpr.member_name                          AS 会员姓名,
    cpr.phone                                AS 会员手机号,
    cpr.card_id_alias                        AS 会员卡号,
    -- 标准业务类型（由 operation_model 标准化）
    CASE cpr.operation_model
        WHEN 'Charge'             THEN '充值'
        WHEN 'ChargeCancel'       THEN '撤销充值'
        WHEN 'Consume'            THEN '消费'
        WHEN 'CancelOrder'        THEN '订单取消'
        WHEN 'Refund'             THEN '订单退款'
        WHEN 'Exchange'           THEN '积分兑换'
        WHEN 'TransferOut'        THEN '卡转出'
        WHEN 'TransferIn'         THEN '卡转入'
        WHEN 'Return'             THEN '退卡'
        WHEN 'Withdrawal'         THEN '提现'
        WHEN 'Repay'             THEN '回款'
        WHEN 'Reverse'           THEN '反回款'
        WHEN 'LevelUpg'          THEN '会员升级'
        WHEN 'Join'              THEN '开卡'
        WHEN 'CashBack'          THEN '返现'
        WHEN 'CancelCashBack'    THEN '撤销返现'
        WHEN 'AccountingBluePatch' THEN '蓝补'
        WHEN 'AccountingRedFlush'  THEN '红冲'
        WHEN 'Initial'           THEN '期初'
        ELSE cpr.operation_model
    END                                       AS 标准业务类型,
    cpr.operation_model                       AS 业务类型,
    -- 记账方向（由 amount 正负派生）
    CASE WHEN cpr.amount > 0 THEN '增加' ELSE '减少' END
                                              AS 记账方向,
    cpr.balance_before                        AS 变动前余额,
    cpr.amount                                AS 本次发生额,
    cpr.balance_after                         AS 变动后余额,
    cpr.order_bill_id                         AS 业务单号,
    cpr.out_order_bill_id                     AS 外部单号,
    cpr.related_points_record_lid              AS 关联流水,
    cpr.operator                              AS 操作人,
    cpr.comment                               AS 备注,
    CASE cpr.if_deal_success WHEN 1 THEN '成功' ELSE '失败' END
                                              AS 处理状态,
    CASE cpr.is_third_party WHEN 1 THEN '第三方' ELSE '内部' END
                                              AS 来源类型,
    cpr.subject                               AS 科目,
    cpr.mid                                   AS 商户ID
FROM crm_card_points_record cpr
WHERE cpr.mid = :mid
  -- 正式报表只看有效入账记录，排除骨架行
  AND cpr.amount IS NOT NULL
  AND cpr.amount <> 0
  AND cpr.balance_before IS NOT NULL
  AND cpr.balance_after IS NOT NULL
  -- 正式归期口径用 yingyeriqi，create_time 仅审计追溯
  AND cpr.yingyeriqi BETWEEN :beginDate AND :endDate
  -- 可选：AND cpr.sid IN (:sidList)
  -- 可选：AND cpr.card_id_alias = :cardNo
  -- 可选：AND cpr.operation_model = :operationModel
  -- 可选：AND cpr.if_deal_success = 1  -- 严格口径时启用
ORDER BY cpr.yingyeriqi DESC, cpr.create_time DESC, cpr.lid DESC
```

### 2.5 行级校验

流水明细每行应满足 `balance_before + amount = balance_after`：

```sql
-- 校验失败记录查询（差异 > 1 视为异常）
SELECT
    cpr.lid,
    cpr.card_id_alias,
    cpr.operation_model,
    cpr.amount,
    cpr.balance_before,
    cpr.balance_after,
    cpr.yingyeriqi,
    ABS(cpr.balance_before + cpr.amount - cpr.balance_after) AS balance_diff
FROM crm_card_points_record cpr
WHERE cpr.mid = :mid
  AND cpr.if_deal_success = 1
  AND cpr.yingyeriqi BETWEEN :startDate AND :endDate
  AND ABS(cpr.balance_before + cpr.amount - cpr.balance_after) > 1  -- 浮点容差 ±1
ORDER BY cpr.yingyeriqi DESC, cpr.create_time DESC
```

**异常处理策略**：
- 差异 > 1 的记录进入异常明细区，报表主查询默认排除。
- 差异 ≤ 1 的记录视为浮点精度问题，正常展示并在表头标注"存在 ±1 浮点容差记录"。
- 异常记录由运营人员人工确认后，通过蓝补/红冲（`AccountingBluePatch` / `AccountingRedFlush`）修复。

## 3. 第二张：余额报表

### 3.1 报表定位

余额报表回答：**"某个时点这个会员账户还剩多少积分"**。

这是账户快照，服务客服查账和财务核对。

### 3.2 字段清单

| 字段 | 命名 | 来源/计算方式 |
|------|------|-------------|
| 商户ID | `mid` | `mid` |
| 会员姓名 | `member_name` | 关联 `crm_card` 或 `crm_member` |
| 会员手机号 | `phone` | 关联 |
| 会员卡号 | `card_id_alias` | 账户主标识 |
| 截止时点 | `snapshot_time` | 查询参数 |
| 期初积分余额 | `opening_balance` | 期间开始前最近一笔有效余额的 `balance_after` |
| 本期获得积分 | `earned_points` | 期间内 `sum(amount > 0)` |
| 本期消耗积分 | `used_points` | 期间内 `sum(abs(amount < 0))` |
| 本期净变动 | `net_change` | 本期获得 - 本期消耗 |
| 期间最后余额 | `period_last_balance` | 期间内最后一笔有效记录的 `balance_after`（用于校验） |
| 重算期末余额 | `computed_closing` | 期初 + 本期获得 - 本期消耗（独立计算，不依赖 `balance_after`） |
| 当前积分余额 | `current_points` | 直接取 `crm_card.points`（实时余额） |
| 可用积分余额 | `available_points` | `current_points - frozen_points`（若 `crm_card.frozen_points` 存在），否则等于 `current_points` 并注明"冻结机制暂未实现" |
| 冻结积分余额 | `frozen_points` | 取 `crm_card.frozen_points`，若不存在则填 0 并注明 |
| 最新记账时间 | `latest_record_time` | 最近一次有效记账的 `yingyeriqi` |
| 最近业务类型 | `latest_operation` | 最近一次有效记账的标准化业务类型 |
| 最新发生门店 | `latest_shop` | 最近一次有效记账的 `shop_name` |

> **字段命名说明**：`期间最后余额`（取最后一笔 `balance_after`）和 `重算期末余额`（独立计算）是两个不同的概念，前者用于校验 `balance_before + amount = balance_after` 是否成立，后者用于与 `current_points` 对比验证账户一致性。

### 3.3 数据来源与计算方式

余额报表有两种取数路径：

**路径 A：实时查询（适合查询截止时点为当天的场景）**

```sql
-- 当前余额：取 crm_card.points（实时余额）
SELECT c.points AS current_points
FROM crm_card c
WHERE c.company_id = :mid
  AND c.member_code = :memberCode
  AND c.status_ = 1
```

**路径 B：期间余额（适合有历史追溯需求的场景）**

期末余额的精确计算：

```sql
-- 期末积分余额：取期间内最后一笔有效记录的 balance_after
SELECT cpr.balance_after AS closing_balance,
       cpr.yingyeriqi    AS latest_record_time,
       cpr.operation_model AS latest_operation,
       cpr.shop_name     AS latest_shop
FROM crm_card_points_record cpr
WHERE cpr.mid = :mid
  AND cpr.card_id_alias = :cardIdAlias
  AND cpr.if_deal_success = 1
  AND cpr.yingyeriqi <= :endDate
ORDER BY cpr.yingyeriqi DESC, cpr.create_time DESC, cpr.lid DESC
LIMIT 1
```

期初余额的计算：

```sql
-- 期初积分余额：取期间开始前最近一笔有效记录的 balance_after
-- ⚠️ 注意：用 balance_after 而非 balance_before
-- 数学原理：最后一笔的 balance_before（本次记账前）= 上一次记账后的余额 = 前一记录的 balance_after
-- R2/R3/R4 三张报表统一使用 balance_after，避免口径不一致
SELECT cpr.balance_after AS opening_balance
FROM crm_card_points_record cpr
WHERE cpr.mid = :mid
  AND cpr.card_id_alias = :cardIdAlias
  AND cpr.if_deal_success = 1
  AND cpr.yingyeriqi < :startDate
ORDER BY cpr.yingyeriqi DESC, cpr.create_time DESC, cpr.lid DESC
LIMIT 1
```

**路径 C：完整余额查询 SQL（单账户多期间）**

> 📌 **V4 修复说明**：本 SQL 已重构为 CTE+JOIN 模式，替代原标量子查询方式，避免对 `crm_card_points_record` 重复全表扫描。期初余额统一取 `balance_after`，与 R4 口径完全一致。

```sql
WITH period_flows AS (
    -- 汇总指定期间内该卡的所有积分流水
    SELECT
        cpr.card_id_alias,
        cpr.yingyeriqi,
        cpr.operation_model,
        cpr.amount,
        cpr.balance_after,
        cpr.create_time,
        cpr.lid
    FROM crm_card_points_record cpr
    WHERE cpr.mid = :mid
      AND cpr.if_deal_success = 1
      AND cpr.amount IS NOT NULL
      AND cpr.amount <> 0
), period_summary AS (
    SELECT
        pf.card_id_alias,
        SUM(CASE WHEN pf.amount > 0 THEN pf.amount ELSE 0 END) AS earned_points,
        SUM(CASE WHEN pf.amount < 0 THEN ABS(pf.amount) ELSE 0 END) AS used_points,
        -- 取期间最后一条有效记录的 balance_after（用于校验，不作为账本余额）
        MAX(pf.yingyeriqi) AS last_record_date,
        MAX(pf.create_time) AS last_create_time
    FROM period_flows pf
    WHERE pf.yingyeriqi BETWEEN :startDate AND :endDate
    GROUP BY pf.card_id_alias
), pre_period_last AS (
    -- 取期间开始前最近一条有效记录的 balance_after（期初余额）
    SELECT
        pf.card_id_alias,
        pf.balance_after AS opening_balance
    FROM period_flows pf
    WHERE pf.yingyeriqi < :startDate
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY pf.card_id_alias
        ORDER BY pf.yingyeriqi DESC, pf.create_time DESC, pf.lid DESC
    ) = 1
)
SELECT
    cc.company_id                              AS 商户ID,
    cc.member                                  AS 会员姓名,
    cc.phone                                   AS 会员手机号,
    cc.id                                      AS 会员卡号,
    :snapshotTime                              AS 截止时点,
    COALESCE(pl.opening_balance, 0)            AS 期初积分余额,
    COALESCE(ps.earned_points, 0)              AS 本期获得积分,
    COALESCE(ps.used_points, 0)                AS 本期消耗积分,
    COALESCE(ps.earned_points, 0) - COALESCE(ps.used_points, 0)
                                              AS 本期净变动,
    COALESCE(pl.opening_balance, 0) + COALESCE(ps.earned_points, 0) - COALESCE(ps.used_points, 0)
                                              AS 重算期末余额,
    cc.points                                  AS 当前积分余额,
    COALESCE(cc.points, 0) - COALESCE(cc.frozen_points, 0)
                                              AS 可用积分余额,
    COALESCE(cc.frozen_points, 0)              AS 冻结积分余额,
    ps.last_record_date                        AS 最新记账时间,
    ps.last_create_time                        AS 最近记账时间,
    COALESCE(pl.opening_balance, 0)
        + COALESCE(ps.earned_points, 0)
        - COALESCE(ps.used_points, 0)         AS 期间最后余额   -- 用于校验 balance_before + amount = balance_after 是否成立
FROM crm_card cc
LEFT JOIN pre_period_last pl ON cc.id = pl.card_id_alias
LEFT JOIN period_summary  ps ON cc.id = ps.card_id_alias
WHERE cc.company_id = :mid
  AND cc.status_ = 1
  -- 可选：AND cc.id = :cardIdAlias
  -- 可选：AND cc.member LIKE CONCAT('%', :keyword, '%')
  -- 可选：AND cc.phone LIKE CONCAT('%', :keyword, '%')
```

### 3.4 余额校验

```
期末积分余额 = 期初积分余额 + 本期获得积分 - 本期消耗积分
当前积分余额 ≈ 期末积分余额（允许 ±1 浮点误差）
```

若 `current_points` 与 `computed_closing` 差异不为 0，说明存在未落入 `crm_card_points_record` 的积分变动（如手动调账未写流水）。

## 4. 第三张：期间汇总报表

### 4.1 报表定位

期间汇总报表回答：**"在某个期间内，各门店、各业务类型、各方向一共发生了多少积分"**。

这是发生额分析账，服务运营分析和管理层决策。

### 4.2 字段清单

| 字段 | 命名 | 来源/计算方式 |
|------|------|-------------|
| 统计期间 | `stat_period` | 查询参数（日报：yyyy-MM-dd；月报：yyyy-MM） |
| 商户ID | `mid` | `mid` |
| 发生门店ID | `sid` | `sid`（分组口径） |
| 发生门店名称 | `shop_name` | 仅展示 |
| 科目/分类 | `subject` | `subject`（财务分类维度） |
| 标准业务类型 | `standard_operation` | 由 `operation_model` 标准化 |
| 业务类型 | `operation_model` | 原始值，追溯用 |
| 记账方向 | `direction` | 增加 / 减少 |
| 期初积分余额 | `opening_balance` | 期间开始时余额（按 `sid` 汇总） |
| 本期增加积分 | `earned_points` | `sum(amount > 0)` |
| 本期减少积分 | `used_points` | `sum(abs(amount < 0))` |
| 本期净变动 | `net_change` | 本期增加 - 本期减少 |
| 期末积分余额 | `closing_balance` | 独立计算：`sid_opening_balance + earned_points - used_points`（不依赖 `balance_after`） |
| 发生笔数 | `record_count` | 期间内有效流水条数 |
| 异常笔数 | `abnormal_count` | 失败/冲正/待处理笔数（`if_deal_success != 1`） |
| 异常增加笔数 | `abnormal_earned_count` | 增加方向异常笔数（方向+状态双重过滤，用于定位具体问题） |
| 异常减少笔数 | `abnormal_used_count` | 减少方向异常笔数（同上，方向+状态双重过滤） |
| 本期新账户数 | `new_account_count` | 本期新开卡（`operation_model = 'Join'`）人数 |

### 4.3 SQL 实现

> 📌 **V4 修复说明（S-02）**：R3 汇总层（按 sid 分组）不应用标量子查询计算期初余额（同一 sid 下多卡余额无法归因）。改为 CTE 计算全局期初/期末，再关联到分组维度。同时补齐 `closing_balance` 和 `new_account_count` 字段。

```sql
WITH period_flows AS (
    -- 期间内所有有效积分流水
    SELECT
        cpr.mid,
        cpr.sid,
        cpr.card_id_alias,
        cpr.operation_model,
        cpr.amount,
        cpr.yingyeriqi,
        cpr.if_deal_success
    FROM crm_card_points_record cpr
    WHERE cpr.mid = :mid
      AND cpr.if_deal_success = 1
      AND cpr.amount IS NOT NULL
      AND cpr.amount <> 0
      AND cpr.yingyeriqi BETWEEN :startDate AND :endDate
), pre_period AS (
    -- 期间开始前各卡最近一笔的 balance_after（全局期初）
    SELECT
        cpr.mid,
        cpr.sid,
        cpr.card_id_alias,
        cpr.balance_after AS opening_balance
    FROM crm_card_points_record cpr
    WHERE cpr.mid = :mid
      AND cpr.if_deal_success = 1
      AND cpr.yingyeriqi < :startDate
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY cpr.mid, cpr.sid, cpr.card_id_alias
        ORDER BY cpr.yingyeriqi DESC, cpr.create_time DESC, cpr.lid DESC
    ) = 1
), sid_opening AS (
    -- 按 sid 汇总期初余额（同一 sid 下多卡之和）
    SELECT mid, sid, SUM(opening_balance) AS sid_opening_balance
    FROM pre_period
    GROUP BY mid, sid
), period_grouped AS (
    -- 期间发生额（按分组维度）
    SELECT
        pf.mid,
        pf.sid,
        pf.operation_model,
        CASE WHEN pf.amount > 0 THEN '增加' ELSE '减少' END AS direction,
        SUM(CASE WHEN pf.amount > 0 THEN pf.amount ELSE 0 END) AS earned_points,
        SUM(CASE WHEN pf.amount < 0 THEN ABS(pf.amount) ELSE 0 END) AS used_points,
        SUM(pf.amount)                                             AS net_change,
        COUNT(*)                                                   AS record_count,
        SUM(CASE WHEN pf.if_deal_success != 1 THEN 1 ELSE 0 END)  AS abnormal_count,
        SUM(CASE WHEN pf.amount > 0 AND pf.if_deal_success != 1 THEN 1 ELSE 0 END) AS abnormal_earned_count,
        SUM(CASE WHEN pf.amount < 0 AND pf.if_deal_success != 1 THEN 1 ELSE 0 END) AS abnormal_used_count,
        -- 本期新开卡数（Join 操作）
        SUM(CASE WHEN pf.operation_model = 'Join' THEN 1 ELSE 0 END) AS new_account_count
    FROM period_flows pf
    GROUP BY
        pf.mid, pf.sid, pf.operation_model,
        CASE WHEN pf.amount > 0 THEN '增加' ELSE '减少' END
)
SELECT
    :statPeriod                                          AS 统计期间,
    pg.mid                                               AS 商户ID,
    pg.sid                                               AS 发生门店ID,
    MAX(cpr.shop_name)                                   AS 发生门店名称,
    MAX(cpr.subject)                                     AS 科目,
    CASE pg.operation_model
        WHEN 'Charge'             THEN '充值'
        WHEN 'ChargeCancel'       THEN '撤销充值'
        WHEN 'Consume'            THEN '消费'
        WHEN 'CancelOrder'        THEN '订单取消'
        WHEN 'Refund'             THEN '订单退款'
        WHEN 'Exchange'           THEN '积分兑换'
        WHEN 'TransferOut'        THEN '卡转出'
        WHEN 'TransferIn'         THEN '卡转入'
        WHEN 'Return'             THEN '退卡'
        WHEN 'Withdrawal'         THEN '提现'
        WHEN 'Repay'             THEN '回款'
        WHEN 'Reverse'           THEN '反回款'
        WHEN 'LevelUpg'          THEN '会员升级'
        WHEN 'Join'              THEN '开卡'
        WHEN 'CashBack'          THEN '返现'
        WHEN 'CancelCashBack'    THEN '撤销返现'
        WHEN 'AccountingBluePatch' THEN '蓝补'
        WHEN 'AccountingRedFlush'  THEN '红冲'
        WHEN 'Initial'           THEN '期初'
        ELSE pg.operation_model
    END                                                   AS 标准业务类型,
    pg.operation_model                                    AS 业务类型,
    pg.direction                                          AS 记账方向,
    COALESCE(sid_open.sid_opening_balance, 0)            AS 期初积分余额,
    pg.earned_points                                      AS 本期增加积分,
    pg.used_points                                        AS 本期减少积分,
    pg.net_change                                         AS 本期净变动,
    COALESCE(sid_open.sid_opening_balance, 0) + pg.earned_points - pg.used_points
                                                         AS 期末积分余额,
    pg.record_count                                       AS 发生笔数,
    pg.abnormal_count                                     AS 异常笔数,
    pg.abnormal_earned_count                              AS 异常增加笔数,
    pg.abnormal_used_count                                AS 异常减少笔数,
    pg.new_account_count                                  AS 本期新账户数
FROM period_grouped pg
LEFT JOIN sid_opening sid_open ON pg.mid = sid_open.mid AND pg.sid = sid_open.sid
LEFT JOIN crm_card_points_record cpr
    ON pg.mid = cpr.mid AND pg.sid = cpr.sid
    AND pg.operation_model = cpr.operation_model
WHERE pg.mid = :mid
  -- 可选：AND pg.sid IN (:sidList)
  -- 可选：AND pg.operation_model = :operationModel
ORDER BY pg.sid, pg.direction DESC, pg.net_change DESC
```

### 4.4 期末余额计算

```sql
-- 期末余额 = 期初余额 + 本期增加 - 本期减少（独立计算，不依赖 balance_after）
```

## 5. 第四张：对账报表

### 5.1 报表定位

对账报表回答：**"账是否平、差异在哪里、能不能关账"**。

这是关账控制表，服务财务月结和内控。

### 5.2 字段清单

| 字段 | 命名 | 来源/计算方式 |
|------|------|-------------|
| 对账期间 | `recon_period` | 查询参数 |
| 对账范围类型 | `scope_type` | `MERCHANT` / `STORE` / `MEMBER_GROUP` |
| 对账范围ID | `scope_id` | 商户ID / 门店ID / 会员组ID |
| 对账范围名称 | `scope_name` | 商户名 / 门店名 / 会员组名 |
| 期初积分余额 | `opening_balance` | 从 `crm_card_points_record` 计算（取期间开始前最后一笔的 `balance_after`） |
| 本期增加积分 | `earned_points` | `sum(amount > 0)` |
| 本期减少积分 | `used_points` | `sum(abs(amount < 0))` |
| 冲正/补记发生额 | `adjustment_amount` | 蓝补（正向增加）+ 红冲（负向减少）+ 冲正（方向依业务规则）的代数和。净增为正，净减为负，用于判断本期调整是净补入还是净冲销。本字段不单独参与 balance = 期初 + earned - used 的校验，但参与 difference 差异分析。 |
| 计算期末余额 | `computed_closing` | 期初 + 增加 - 减少 |
| 期末快照余额 | `snapshot_balance` | **月结前锁定的 `crm_card.points` 快照**（详见 5.5 节快照机制） |
| 差异金额 | `difference_amount` | `snapshot_balance - computed_closing` |
| 差异笔数 | `difference_count` | 无法闭环的流水条数 |
| 差异流水追溯 | `difference_records` | 差异流水的 `lid` 列表 |
| 对账状态 | `recon_status` | `BALANCED` / `AUTO_RECONCILED` / `HAS_DIFF` / `PENDING` |
| 差异原因 | `difference_reason` | 人工复核结果 |
| 对账人 | `recon_user` | 执行本次对账的操作员 |
| 对账时间 | `recon_time` | 本次对账的执行时间 |
| 复核人 | `review_user` | 差异复核确认的操作员 |
| 复核时间 | `review_time` | 差异复核完成时间 |
| 处理措施 | `resolution` | 差异处理方式（如蓝补调账、确认忽略等） |

> ⚠️ **差异金额方向说明**：差值 > 0 表示快照余额高于计算余额（可能有未落流水的增加、重复退款等）；差值 < 0 表示快照余额低于计算余额（可能有重复记录或未冲销的减少）。差值绝对值在容差范围内（默认 ±1 分）时，状态为 `AUTO_RECONCILED`，无需人工干预。

> ⚠️ **重要**：`期末快照余额`（`snapshot_balance`）**不是**实时 `crm_card.points`，而是月结前锁定的快照值。若直接用实时值进行月结对账，期间内任何未写流水的积分调整都会导致差异被误判为"数据问题"。月结前必须先执行快照生成（见 5.5 节），再用快照值进行对账。

### 5.3 对账状态判定逻辑

```
差异金额 = 0 → BALANCED（账平）
|差异金额| ≤ 容差阈值（默认 ±1 分） → AUTO_RECONCILED（自动核销，无需人工）
差异金额 ≠ 0 且 > 容差阈值 → HAS_DIFF（需人工处理）
账面余额快照缺失 OR 未关账 → PENDING（待处理）
```

> 📌 **容差说明**：积分计算中存在浮点误差（如 `balance_before + amount - balance_after` 的 ±1 分误差）和时间差（快照时点后有新流水落地），差异金额在容差阈值内可视为自动平衡，无需人工干预。

### 5.4 SQL 实现

```sql
-- 步骤1：计算期初余额（取期间开始前最后一笔有效记录的 balance_after）
WITH opening_balance AS (
    SELECT
        cpr.mid,
        cpr.balance_after AS opening_balance
    FROM crm_card_points_record cpr
    WHERE cpr.mid = :mid
      AND cpr.if_deal_success = 1
      AND cpr.yingyeriqi < :startDate
      AND cpr.card_id_alias IS NOT NULL
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY cpr.mid
        ORDER BY cpr.yingyeriqi DESC, cpr.create_time DESC, cpr.lid DESC
    ) = 1
),
-- 步骤2：汇总期间发生额
period_data AS (
    SELECT
        cpr.mid,
        SUM(CASE WHEN cpr.amount > 0 AND cpr.if_deal_success = 1 THEN cpr.amount ELSE 0 END) AS earned_points,
        SUM(CASE WHEN cpr.amount < 0 AND cpr.if_deal_success = 1 THEN ABS(cpr.amount) ELSE 0 END) AS used_points,
        SUM(CASE WHEN cpr.operation_model IN ('AccountingBluePatch','AccountingRedFlush','Reverse')
                 AND cpr.if_deal_success = 1 THEN cpr.amount ELSE 0 END) AS adjustment_amount,
        COUNT(CASE WHEN cpr.if_deal_success != 1 THEN 1 END) AS abnormal_count
    FROM crm_card_points_record cpr
    WHERE cpr.mid = :mid
      AND cpr.yingyeriqi BETWEEN :startDate AND :endDate
      AND cpr.card_id_alias IS NOT NULL
    GROUP BY cpr.mid
),
-- 步骤3：从快照表读取期末快照余额（⚠️ 禁止用实时 crm_card.points）
snapshot_balance AS (
    SELECT
        mid,
        SUM(points) AS snapshot_balance
    FROM crm_card_points_snapshot
    WHERE mid = :mid
      AND snapshot_date = :endDate
      AND snapshot_type = 'MONTHLY_CLOSE'
    GROUP BY mid
)
-- 步骤4：合并计算差异
SELECT
    ob.mid                                        AS 商户ID,
    :reconPeriod                                  AS 对账期间,
    'MERCHANT'                                    AS 对账范围类型,
    ob.mid                                        AS 对账范围ID,
    (SELECT company_name FROM company WHERE id = ob.mid)
                                                      AS 对账范围名称,
    COALESCE(ob.opening_balance, 0)                AS 期初积分余额,
    COALESCE(pd.earned_points, 0)                AS 本期增加积分,
    COALESCE(pd.used_points, 0)                   AS 本期减少积分,
    COALESCE(pd.adjustment_amount, 0)             AS 冲正/补记发生额,
    COALESCE(ob.opening_balance, 0)
        + COALESCE(pd.earned_points, 0)
        - COALESCE(pd.used_points, 0)             AS 计算期末余额,
    COALESCE(sb.snapshot_balance, 0)              AS 期末快照余额,
    COALESCE(sb.snapshot_balance, 0)
        - (COALESCE(ob.opening_balance, 0)
           + COALESCE(pd.earned_points, 0)
           - COALESCE(pd.used_points, 0))          AS 差异金额,
    COALESCE(pd.abnormal_count, 0)                AS 异常笔数,
    CASE
        -- 容差自动核销（差异在 ±1 分以内）
        WHEN ABS(
            COALESCE(sb.snapshot_balance, 0)
            - (COALESCE(ob.opening_balance, 0)
               + COALESCE(pd.earned_points, 0)
               - COALESCE(pd.used_points, 0))
        ) <= 1
        THEN 'AUTO_RECONCILED'
        WHEN COALESCE(sb.snapshot_balance, 0)
             - (COALESCE(ob.opening_balance, 0)
                + COALESCE(pd.earned_points, 0)
                - COALESCE(pd.used_points, 0)) = 0
         AND COALESCE(pd.abnormal_count, 0) = 0
        THEN 'BALANCED'
        WHEN sb.snapshot_balance IS NULL
        THEN 'PENDING'
        ELSE 'HAS_DIFF'
    END                                           AS 对账状态
FROM opening_balance ob
LEFT JOIN period_data pd ON ob.mid = pd.mid
LEFT JOIN snapshot_balance sb ON ob.mid = sb.mid
```

> 📌 **R4 SQL 修正说明**（V2）：
> - **期初余额**：`SUM(balance_before)` 改为取最后一笔的 `balance_after`（通过 `QUALIFY ROW_NUMBER()` 实现），避免多笔流水累加导致期初虚高。
> - **期末快照余额**：从 `crm_card_points_snapshot`（月结快照表）读取，禁止用实时 `crm_card.points`。
> - `book_balance_data` 已移除（由 `snapshot_balance` CTE 替代）。

### 5.5 月结快照机制

**禁止直接用实时 `crm_card.points` 进行月结对账**。月结前必须按以下步骤生成快照：

```sql
-- 步骤1：月结前生成积分余额快照
INSERT INTO crm_card_points_snapshot (mid, card_id_alias, points, snapshot_date, snapshot_type, created_at)
SELECT
    cc.company_id           AS mid,
    cc.id                   AS card_id_alias,
    cc.points               AS points,
    :endDate                AS snapshot_date,
    'MONTHLY_CLOSE'         AS snapshot_type,
    NOW()                   AS created_at
FROM crm_card cc
WHERE cc.company_id = :mid
  AND cc.status_ = 1
  -- ON DUPLICATE KEY UPDATE points = VALUES(points)  -- 幂等重跑
```

**快照表 `crm_card_points_snapshot` 字段说明**：

| 字段 | 说明 |
|------|------|
| `mid` | 商户ID |
| `card_id_alias` | 会员卡号 |
| `points` | 锁定的积分余额快照 |
| `snapshot_date` | 快照日期（月末） |
| `snapshot_type` | 快照类型（`DAILY_CLOSE` / `MONTHLY_CLOSE`） |
| `created_at` | 快照生成时间 |

> ⚠️ 快照生成后，`crm_card.points` 仍可继续变化（实时余额），但 R4 对账始终以快照值为准。

当 `差异金额 ≠ 0` 时，需要定位差异来源：

```sql
-- 差异流水追溯：查找期间内余额不平衡的记录
SELECT
    cpr.lid,
    cpr.card_id_alias,
    cpr.operation_model,
    cpr.amount,
    cpr.balance_before,
    cpr.balance_after,
    cpr.yingyeriqi,
    (cpr.balance_before + cpr.amount - cpr.balance_after) AS balance_diff
FROM crm_card_points_record cpr
WHERE cpr.mid = :mid
  AND cpr.if_deal_success = 1
  AND cpr.yingyeriqi BETWEEN :startDate AND :endDate
  AND ABS(cpr.balance_before + cpr.amount - cpr.balance_after) > 1  -- 浮点容差
ORDER BY cpr.yingyeriqi DESC
```

## 8. 关键实现约束

1. **归期严格区分**：`yingyeriqi` 是正式归期，`create_time` 仅审计追溯，同一张报表不得混用两个归期口径。
2. **方向由 `amount` 正负派生**：`operation_model` 只做业务归类，不参与方向判断。
3. **骨架行必须排除**：`amount IS NULL`、`amount = 0`、`balance_before IS NULL`、`balance_after IS NULL` 的记录不进正式报表。
4. **余额可复算**：每行 `balance_before + amount = balance_after`，允许 ±1 浮点误差。
5. **历史不回写**：关账后所有修正通过新增冲正/补记记录表达。
6. **差异必须可追溯**：对账报表给出差异金额的同时，必须能下钻到差异流水。
7. **汇总接口分页**：必须基于聚合后的结果集分页，不能先对事实行分页再聚合。
8. **`if_deal_success = 1`**：正式对账和月结口径以此为主过滤条件。
