# 积分过期功能重新设计方案

版本：v2
时间：2026-06-04
适用系统：

- 前端：`D:\mywork\nms4cloud-biz-ui`
- 后端：`D:\mywork\nms4cloud`

## 1. 设计结论

本次不恢复原来的“积分清零规则”复杂能力，改为一个简单的“积分有效期”配置，并保留清零前消息提醒配置。

最终口径：

- 界面学阿里：在“设置积分权益”页面放一个“积分过期设置”卡片，运营只需要选择“永久有效”或“每年固定日期清零”。
- 规则口径保持简单：到清零日时，清掉会员卡当前可用积分余额，不再按每笔积分获得时间、自然月、积分桶或 FIFO 计算。
- 不新增表：不新增积分批次表、任务表、过期明细表、月度汇总表。
- 不增加复杂结构：只在 `crm_points_rule` 主表增加少量有效期配置字段。
- 不加任务状态字段：不增加 `expiry_effective_time`、`last_expiry_time`、`task_cursor` 这类字段。
- 以当前规则为准：任务执行时只看当前 `crm_points_rule` 配置；规则变更后，下次按新规则执行，不追溯恢复已经清掉的积分。
- 过期通过负数流水入账：积分过期不是删除流水，也不是改旧流水，而是在 `crm_card_points_record` 写一条负数流水，并同步扣减 `crm_card.points`。
- 清零前消息提醒只保存简单配置：是否提醒、提前几天提醒；不新增提醒任务表。

本方案不做“精确到获得批次的积分过期”。如果以后要做到“某一年获得的积分未使用部分才过期”，就需要重新引入流水归属计算或积分批次结构，这不符合本轮“不增加复杂结构”的约束。

## 2. 为什么不再叫“积分清零规则”

“积分清零规则”容易让人联想到一组复杂的任务、提醒、清零范围和历史规则生效时间。当前需求只需要表达积分是否长期有效，以及是否每年固定日期把余额清掉。

新功能建议统一命名为：

- 前端配置区标题：`积分过期设置`
- 后端配置字段前缀：`validity*`
- 流水操作类型：继续保留 `OperationModelEnum.PointsExpire`

保留 `OperationModelEnum.PointsExpire` 的原因：历史积分流水可能已经存在该操作类型，直接删除枚举会影响历史记录展示、导出和筛选。流水展示文案可以继续显示“积分清零”，也可以调整为“积分过期”。

## 3. 前端界面设计

### 3.1 页面位置

在“设置积分权益”页面中新增一个卡片：

- 左侧锚点顺序：
  - 积分获取规则
  - 积分过期设置
  - 积分规则说明
- 页面主体顺序同上。
- 当前隐藏的“积分抵扣规则”继续保持隐藏，不因本次改动恢复。

### 3.2 卡片内容

卡片标题：`积分过期设置`

单选项一：永久有效

```text
永久有效
```

单选项二：每年固定日期过期清零

```text
每年的 [月] 月 [日] 日24:00时积分过期清零
```

说明：

- `24:00` 按次日 `00:00:00` 后执行。
- 例如选择 `12月31日24:00`，任务在下一天 `1月1日00:00` 后处理。
- 前端日期下拉按月份限制合法日；2 月最多到 28 日，不支持 2 月 29 日，避免闰年分支。

清零前消息提醒：

```text
是否过期提醒：● 不提醒（禁用） ○ 提醒（禁用）  微信模板消息待接入此功能，暂不可开启。
```

说明：

- 只有选择“每年固定日期清零”时，消息提醒配置才有业务意义；本轮仅展示禁用态。
- 选择“永久有效”时，前端禁用或隐藏提醒输入，后端保存前清空提醒天数字段。
- 本轮提醒入口在前端禁用，并展示“微信模板消息待接入此功能，暂不可开启。”；后端字段保留但保存时按关闭处理，不新增提醒任务表。

### 3.3 默认值

如果没有设置积分有效期规则，默认视为：

```text
永久有效
```

这样可以兼容两类情况：

- 老数据没有新字段。
- 商户一开始没有配置过规则。

后端也必须做相同兜底：`validityMode == null` 时按永久有效处理。

## 4. 规则口径

### 4.1 永久有效

不执行积分过期处理。

配置字段：

- `validityMode = 1`
- 固定日期从属字段清空为 `null`
- 消息提醒字段清空为 `null` 或按关闭处理

### 4.2 每年固定日期清零

规则：

- 每年固定月日的 `24:00` 到达后，处理当前规则适用会员方案下有积分余额的会员卡。
- 对单张会员卡，过期金额直接取 `crm_card.points` 当前可用积分余额。
- 写一条负数积分流水，流水金额为 `-crm_card.points`。
- 同步把 `crm_card.points` 扣到 `0`。
- 不读取旧积分流水做归属计算。
- 不修改历史获得积分流水。
- 不扣减 `crm_card.sum_of_points`，除非产品确认累计积分也要随过期减少。
- 如果开启消息提醒，则在清零日前 `validityRemindDays` 天触发提醒发送；提醒发送不能改变积分余额。

示例：

- 配置：每年 `12月31日24:00`
- 执行时间：`2027-01-01 00:00` 后
- 会员当前 `crm_card.points = 100`
- 系统写入一条 `amount = -100` 的积分过期流水
- 会员当前积分余额变为 `0`

### 4.3 关于“本年度积分”的说明

如果目标是“每年清零”，最简单且最容易解释的口径是清当前可用余额，不需要计算本年度产生了多少积分。

原因：

- 现有 `crm_card.points` 已经是当前可用积分余额投影。
- 年度清零的业务语义通常是“到年底把剩余积分清掉”，不是“只清某个获得批次的剩余部分”。
- 只写一条负数流水，用户和运营都容易理解。

如果改成“只清本年度获得且未使用的积分”，就必须解决消费积分归属问题：用户花掉的积分到底是去年积分还是今年积分。这会重新引入 FIFO 或积分批次，本轮不采用。

## 5. 数据模型设计

### 5.1 不新增表

不新增以下表：

- 积分批次表
- 积分过期任务表
- 积分过期明细表
- 月度积分汇总表

原因：

- 当前需求只需要简单年度余额清零。
- `crm_card.points` 已经能表示当前可用余额。
- 写一条负数积分流水即可保留审计记录。
- 增加批次表会引入迁移、历史余额拆分、消费归属、撤销归属等复杂问题。

### 5.2 在 `crm_points_rule` 增加的字段

建议只增加五个字段：

```text
validity_mode
validity_fixed_month
validity_fixed_day
validity_notify_enabled
validity_remind_days
```

字段说明：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| `validity_mode` | TINYINT | 积分有效期模式：1-永久有效，2-每年固定日期清零 |
| `validity_fixed_month` | TINYINT | 每年固定日期清零的月份，仅 `validity_mode=2` 有值 |
| `validity_fixed_day` | TINYINT | 每年固定日期清零的日期，仅 `validity_mode=2` 有值 |
| `validity_notify_enabled` | TINYINT | 清零前消息提醒开关：0-不提醒，1-提醒，仅 `validity_mode=2` 有业务意义 |
| `validity_remind_days` | INT | 距清零提前提醒天数，仅 `validity_mode=2 且 validity_notify_enabled=1` 有值 |

不建议增加：

```text
expiry_effective_time
last_expiry_time
last_expiry_record_lid
task_cursor
task_status
```

原因：

- 用户已经明确不需要任务字段。
- 规则多次变更时，`expiry_effective_time` 会失去业务意义。
- 一开始没有配置规则时，这类字段没有值，反而增加空值分支。
- 当前规则和当前日期已经能决定是否需要执行年度清零。

### 5.3 枚举设计

建议新增枚举：

```text
CrmPointsValidityModeEnum
1 PERMANENT   永久有效
2 FIXED_DATE  每年固定日期清零
```

枚举 code 只新增，不复用旧 `CrmPointsExpiryModeEnum`、`CrmPointsExpiryFreqEnum`。

消息提醒开关复用现有 `CrmPointsSwitchEnum`：

```text
0 DISABLED  不提醒
1 ENABLED   提醒
```

## 6. 现有系统是否需要知道每年产生了多少积分

结论：本方案不需要系统知道每年产生了多少积分。

现有系统事实：

- `crm_card.points`：当前可用积分余额。
- `crm_card.sum_of_points`：累计积分或累计获得积分投影，是否用于成长值需要按现有业务确认。
- `crm_card_points_record`：积分流水事实源，正数代表获得，负数代表消费、撤销、转出、过期等减少。

年度余额清零只依赖：

```text
crm_card.points > 0
```

执行时：

```text
expireAmount = crm_card.points
```

然后写入：

```text
crm_card_points_record.amount = -expireAmount
crm_card_points_record.operation_model = PointsExpire
```

因此不需要月度汇总表，也不需要按年份聚合流水。

## 7. 过期执行设计

### 7.1 执行入口

建议新增一个简单定时任务：

```text
CrmPointsValidityTask
```

任务频率：

- XXL Job 调度频率不写死在 Java 业务代码里，由 XXL Admin 或部署脚本配置。
- 推荐 handler：`CrmPointsValidityTask`。
- 推荐 cron：`0 0 1 * * ?`，即每天 01:00 执行一次。
- 固定日期 `24:00` 语义由任务在次日凌晨处理。
- XXL Job 参数为空时，默认处理“昨天”这个清零日。
- 如需补跑，XXL Job 参数传清零日期，格式 `yyyy-MM-dd`，例如 `2026-12-31`；系统会按该日期匹配固定月日规则，并依赖积分流水幂等键避免重复清零。

服务方法命名：

```text
executeValidityRules()
```

不要恢复旧方法名：

```text
executeExpiryRules()
```

### 7.2 执行流程

整体流程：

```text
1. 查询启用积分规则且 validityMode = 每年固定日期清零 的积分规则。
2. 判断今天是否为该规则固定清零日期的次日。
3. 查询该规则对应会员方案下 crm_card.points > 0 的会员卡。
4. 单卡开启事务。
5. 读取并锁定会员卡当前积分余额。
6. expireAmount = 当前 crm_card.points。
7. 如果 expireAmount <= 0，跳过。
8. 检查本规则、本会员卡、本清零日期是否已有积分过期流水。
9. 如果没有，则写入一条 amount = -expireAmount 的积分过期流水。
10. 条件更新 crm_card.points = crm_card.points - expireAmount。
11. 更新后余额应为 0；单卡事务提交。
```

### 7.3 负数流水设计

过期时写入 `crm_card_points_record`：

| 字段 | 建议值 |
|---|---|
| `amount` | 负数，例如 `-100` |
| `balance_before` | 扣减前 `crm_card.points` |
| `balance_after` | `0` |
| `operation_model` | `OperationModelEnum.PointsExpire.name()` |
| `comment` | `积分有效期到期自动清零` |
| `is_third_party` | `false` |
| `if_deal_success` | `true` |
| `create_time` | 当前执行时间 |
| `year/month/day` | 当前执行日期 |
| `out_order_bill_id` | 幂等键 |

`crm_card.sum_of_points` 不建议扣减。

原因：

- `points` 表示当前可用积分余额，需要扣。
- `sum_of_points` 更像累计积分或成长积分投影，积分过期不是撤销历史获得行为。
- 如果产品明确要求“过期也减少累计积分”，再单独调整。

### 7.4 幂等设计

不新增任务表时，幂等放到现有积分流水中。

建议幂等键：

```text
POINTS_VALIDITY:{ruleLid}:{cardLid}:{cutoffDate}
```

写入前检查：

```text
同一 mid、cardId、operationModel=PointsExpire、outOrderBillId=幂等键 是否已存在
```

如果已存在：

- 跳过该会员卡本次清零处理。

注意：

- 如果数据库没有针对 `out_order_bill_id` 的唯一索引，幂等是应用层幂等，不是强数据库幂等。
- 不新增表的约束下，这是当前最小实现。
- 如果将来要强幂等，可以评估是否在现有流水表增加唯一索引，但本方案不要求。

### 7.5 并发保护

单卡处理必须避免扣成负数或重复扣减。

建议：

- 单卡事务内读取会员卡当前积分。
- 更新 `crm_card.points` 时带条件：

```text
where lid = ? and points >= expireAmount
```

- 如果更新失败，说明余额已被其他业务改变，本次单卡跳过或重试。
- 写流水和扣余额必须在同一个事务里完成。

### 7.6 清零前消息提醒执行口径

提醒执行和清零执行分离。

本轮配置口径：

```text
validityNotifyEnabled = ENABLED
validityRemindDays = N
```

提醒触发日：

```text
固定清零日期 - N 天
```

处理边界：

- 只有 `validityMode = FIXED_DATE` 且 `validityNotifyEnabled = ENABLED` 时才需要提醒。
- 提醒任务只读取规则和会员卡，不修改 `crm_card.points`。
- 本轮不新增提醒任务状态表；如果现有消息系统没有幂等能力，需要由消息发送链路使用业务幂等键控制重复发送。
- 建议提醒业务幂等键使用：

```text
POINTS_VALIDITY_REMIND:{ruleLid}:{cardLid}:{cutoffDate}:{validityRemindDays}
```

## 8. 规则变更处理

本方案不记录规则生效时间。

规则变更后：

- 下次任务按当前 `crm_points_rule` 配置判断是否需要年度清零。
- 不区分积分获得时使用的是哪版规则。
- 已经写入的过期流水不自动恢复。
- 如果运营把规则从“每年固定日期清零”改回“永久有效”，只影响后续任务，不恢复之前已经清掉的积分。

这样符合当前确认的口径：

```text
以当前规则为准；到清零日清当前可用积分余额；不做复杂追溯。
```

## 9. Convert 和校验规则

### 9.1 互斥字段清空

在 `CrmPointsRuleConvert.toEntity()` 中继续执行互斥字段清空。

规则：

```text
pointsRuleEnabled != ENABLED：
  清空所有积分获取、抵扣、有效期字段。

validityMode == PERMANENT 或 null：
  清空 fixedMonth、fixedDay、validityNotifyEnabled、validityRemindDays。

validityMode == FIXED_DATE：
  保留 fixedMonth、fixedDay。
  validityNotifyEnabled 关闭或为空时，清空 validityRemindDays。
```

### 9.2 Service 校验

校验只校验当前模式下有意义的字段。

永久有效：

- 不要求其他字段。

每年固定日期：

- `validityFixedMonth` 必填，范围 1-12。
- `validityFixedDay` 必填，且必须是该月份合法日期。
- 2 月最多支持 28 日。
- `validityNotifyEnabled` 可空；为空按不提醒处理。
- `validityNotifyEnabled=ENABLED` 时，`validityRemindDays` 必填，建议范围 1-365。

## 11. 现有 CRM 系统支持度

现有系统已支持：

- `crm_points_rule` 作为积分权益规则主表，可以承载少量新增配置字段。
- DTO、VO、Entity、Convert、Service 的积分规则保存链路。
- `crm_card_points_record` 积分流水。
- `crm_card.points` 当前积分余额。
- 负数积分流水模式，现有消费、撤销、转出等链路已经使用负数流水表达积分减少。
- `OperationModelEnum.PointsExpire` 历史操作类型。

现有系统暂不完整支持，需要新增：

- “积分过期设置”前端卡片。
- `validityMode`、`validityFixedMonth`、`validityFixedDay`、`validityNotifyEnabled`、`validityRemindDays` 字段的 DTO/VO/Entity/SyncVO/SQL/DBML。
- 新的有效期枚举。
- 保存时的字段互斥清空和校验。
- 定时任务入口 `CrmPointsValidityTask`。
- Service 执行方法 `executeValidityRules()` 和补跑重载 `executeValidityRules(LocalDate clearDate)`。
- 生产迁移脚本 `docs/sql/migration/V20260604__crm_points_rule_validity_columns.sql`。
- 年度余额清零负数流水写入逻辑。
- 扣减 `crm_card.points` 的单卡事务。
- 清零前消息提醒配置字段保留，前端入口禁用并提示“微信模板消息待接入此功能，暂不可开启。”。
- 清零前消息提醒发送入口待后续接入微信模板消息，本轮不新增提醒任务表。

## 12. 不纳入本轮的能力

本轮不做：

- 不新增积分批次表。
- 不新增任务状态表。
- 不做历史迁移。
- 不新增过期前短信、企微、小程序提醒任务表。
- 不做复杂提醒记录和提醒撤销。
- 不做按会员等级差异化有效期。
- 不做按门店、渠道、场景差异化有效期。
- 不做页面化运营手动补偿和过期撤销；仅支持运维通过 XXL Job 参数按清零日期补跑。
- 不做按自然月、当年年底、次年年底等复杂自然周期过期。
- 不做按积分获得批次精确过期。
- 不做前端复杂预估报表。

## 13. 五轮评审汇聚问题

### 13.1 必须处理

问题一：重复执行导致重复清零流水。

方案：

- 使用 `out_order_bill_id = POINTS_VALIDITY:{ruleLid}:{cardLid}:{cutoffDate}` 作为幂等键。
- 写流水前查同幂等键流水。
- 单卡事务内完成幂等检查、写流水、扣余额。

问题二：并发消费或加积分导致余额变化。

方案：

- 单卡事务内读取当前余额。
- 扣减时使用 `points >= expireAmount` 条件更新。
- 更新失败则跳过或重试，不允许扣成负数。

问题三：规则变更后的历史语义。

方案：

- 不增加规则生效时间字段。
- 按当前规则执行。
- 已清零积分不自动恢复。
- 前端可在规则旁提示“规则变更仅影响后续清零任务，已清零积分不会自动恢复”。

问题四：消息提醒重复发送。

方案：

- 配置层不新增提醒任务状态字段。
- 发送层使用业务幂等键 `POINTS_VALIDITY_REMIND:{ruleLid}:{cardLid}:{cutoffDate}:{validityRemindDays}`。
- 如果现有消息系统暂时没有可靠幂等，本轮先完成配置保存，提醒发送作为后续任务接入。

### 13.2 可暂缓

- 数据库强唯一索引：本轮不要求；如后续出现多实例重复执行风险，再评估在现有流水表增加唯一约束。
- 过期前提醒发送任务：本轮只做配置字段和校验；发送任务可后续接现有消息系统。
- 精确按获得年份过期：本轮不做，因为会重新引入积分消费归属问题。

## 16. 最终口径

本方案的本质是：

```text
用五个 validity 字段描述积分有效期和清零前提醒；
只支持永久有效、每年固定日期清零；
不保存规则生效时间；
不新增表；
不拆积分批次；
不计算消费归属；
到清零日直接读取 crm_card.points 当前余额；
写一条负数积分流水；
同步把会员卡当前积分余额扣到 0；
清零前提醒只保存开关和提前天数，不新增提醒状态表。
```

这比原来的 `expiry_*` 方案更容易理解，也更符合“不增加表、不增加复杂结构”的约束。
