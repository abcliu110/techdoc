# 储值相关创建人与修改人处理机制说明

## 1. 目的

本文说明 CRM 储值相关功能中：

- 创建人 `createdBy`
- 修改人 `updatedBy`
- 创建时间 `createdTime`
- 修改时间 `updatedTime`

在系统中的来源、赋值方式、写库方式，以及当前储值模块各业务的实际处理现状。


## 2. 核心结论

系统对“时间”和“人”的处理不是同一套机制：

- `createdTime` / `updatedTime`
  - 由 MyBatis-Plus 自动填充
- `createdBy` / `updatedBy`
  - 不是框架自动填充
  - 必须由业务代码手动赋值

如果业务代码没有手动给 `createdBy` / `updatedBy` 赋值：

- 数据库中的 `created_by` / `updated_by` 就会是 `null`
- 接口返回 JSON 时，这两个字段还可能被直接省略


## 3. 当前登录人是如何获取的

### 3.1 登录成功后如何写入 Session

商户登录成功后，平台会把当前登录管理员 `BizAdminVO` 写入 Sa-Token 的 `SaSession`。

关键代码：

- `StpBizUtil.login(...)`
- `session.set(CommonConstants.USER_IN_SA_TOKEN, vo)`

参考：

- `D:\mywork\nms4cloud\nms4cloud-app\1_platform\nms4cloud-platform\nms4cloud-platform-app\src\main\java\com\nms4cloud\platform\app\controller\biz\BizAuthController.java`


### 3.2 `getForceAdmin()` 为什么拿到的是当前登录人

`getForceAdmin()` 的逻辑是：

1. 先判断当前请求是否已登录：`StpBizUtil.isLogin()`
2. 再从当前请求绑定的 `SaSession` 中读取 `CommonConstants.USER_IN_SA_TOKEN`
3. 如果 `user` 为空，则回退读取 `CommonConstants.BOSS_IN_SA_TOKEN`

因此，`getForceAdmin()` 读取的不是“全局用户”，而是“当前请求上下文对应 Session 中的商户管理员对象”。

## 4. 时间字段如何赋值

时间字段由 `MetaObjectHandlerImpl` 自动处理：

### 4.1 新增时

执行 `insertFill()`：

- `createdTime = now`
- `updatedTime = now`

### 4.2 修改时

执行 `updateFill()`：

- `updatedTime = now`
- 不会改 `createdTime`

因此时间字段天然满足以下业务规则：

- 数据第一次新增
  - 有创建时间
  - 也有修改时间
  - 两者通常相同
- 已有记录再次修改
  - 创建时间保留
  - 修改时间刷新

参考：

- `D:\mywork\nms4cloud\nms4cloud-starter\nms4cloud-starter-mybatis\src\main\java\com\nms4cloud\mybatis\handlers\MetaObjectHandlerImpl.java`


## 5. 创建人和修改人如何赋值

### 5.1 框架不会自动填充人名

自动填充器只处理：

- `createdTime`
- `updatedTime`
- `lid`
- `deleted`
- `status_`

不会处理：

- `createdBy`
- `updatedBy`

所以人名字段必须由业务代码自行设置。


### 5.2 标准业务赋值方式

推荐模式如下：

#### 新增

```java
BizAdminVO admin = getForceAdmin();
entity.setCreatedBy(admin.getName());
entity.setUpdatedBy(admin.getName());
iService.save(entity);
```

#### 修改

```java
BizAdminVO admin = getForceAdmin();
entity.setUpdatedBy(admin.getName());
iService.update(entity, query);
```

对应业务含义：

- 数据库没有记录，执行新增
  - 创建人 = 当前登录人
  - 修改人 = 当前登录人
  - 创建时间、修改时间由框架自动填当前时间
- 数据库已有记录，执行修改
  - 创建人、创建时间保留原值
  - 修改人 = 当前登录人
  - 修改时间由框架自动刷新


## 6. 最终如何写入数据库

实体字段与数据库列的映射关系如下：

- `createdBy -> created_by`
- `createdTime -> created_time`
- `updatedBy -> updated_by`
- `updatedTime -> updated_time`

最终由 MyBatis-Plus 持久化：

- 新增：`iService.save(entity)`
- 修改：`iService.update(entity, query)` 或 `iService.updateById(entity)`

## 8. 为什么数据库为 null 时接口里看不到字段

系统当前 Web 输出使用 FastJson，且没有开启 `WriteNulls`。

因此当字段值为 `null` 时：

- 不会返回 `"createdBy": null`
- 而是整个字段直接不输出

所以接口里“看不到创建人/修改人”，不一定是 VO 没有字段，也可能只是数据库值为 `null`。

参考：

- `D:\mywork\nms4cloud\nms4cloud-starter\nms4cloud-starter-parent\src\main\java\com\nms4cloud\common\config\WebMvcConfiguration.java`


## 9. 对储值相关业务的统一规则建议

建议对储值相关模块统一采用以下规则：

### 9.1 新增

- 如果数据库没有记录，执行新增：
  - `createdBy = 当前登录人`
  - `updatedBy = 当前登录人`
  - `createdTime` 自动填充
  - `updatedTime` 自动填充

### 9.2 修改

- 如果数据库已有记录，执行修改：
  - 保留原 `createdBy`
  - 保留原 `createdTime`
  - `updatedBy = 当前登录人`
  - `updatedTime` 自动刷新

### 9.3 查询返回

- 历史数据如 `createdBy/updatedBy` 为 `null`
  - 可保留 `null`
  - 或在 VO 转换阶段兜底为 `""`
- 若不做兜底，在当前 FastJson 配置下字段会被省略


## 10. 当前建议的补齐顺序

建议先补齐储值主链路：

1. `CrmDepositAgreement`
   - 已补齐
2. `CrmDepositCategory`
   - 需补齐
3. `CrmDepositPlan`
   - 需补齐

推荐补齐方式：

1. Controller 新增/修改入口增加 `getForceAdmin()`
2. Service `add/update` 增加 `BizAdminVO admin` 参数
3. `add` 中写 `createdBy + updatedBy`
4. `update` 中写 `updatedBy`
5. 对查询返回按需要做空值兜底


## 11. 一句话总结

CRM 储值相关的“时间字段”是框架自动填的，“创建人/修改人”是业务手动填的。

谁没有手动写 `setCreatedBy/setUpdatedBy`，谁的数据库里这两个字段就可能长期为 `null`。
