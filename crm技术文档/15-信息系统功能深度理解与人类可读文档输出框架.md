# 信息系统功能深度理解与人类可读文档输出框架

> 本框架用于指导 AI 对任意现代信息系统中的某一个功能进行深度理解，并输出一份能被人类用于沟通、维护、测试、改造和风险评估的功能理解文档。
>
> 适用对象包括：企业后台、SaaS、移动应用、Web 前端 + API 后端、微服务系统、事件驱动系统、数据平台、低代码平台、集成系统等。
>
> 本框架不绑定具体技术栈。React、Vue、Spring、.NET、Node.js、Go、Python、移动端、微服务、单体系统都可以套用。

---

## 1. 最高原则

AI 分析功能的目标不是“解释代码”，而是“还原一个功能在系统中的真实行为”。

一个功能必须被理解为一条端到端系统切片：

```text
用户意图
 -> 交互入口
 -> 前端状态与事件
 -> API / 消息 / 任务契约
 -> 后端编排
 -> 业务规则
 -> 数据与状态变化
 -> 副作用与外部依赖
 -> 异常、并发、安全、性能影响
 -> 证据链与待确认问题
```

如果最终文档只能说明“代码调用了哪些类”，说明分析不合格。

如果最终文档能让一个没读过源码的高级工程师回答以下问题，说明分析合格：

- 这个功能为谁解决什么问题？
- 用户如何触发它？
- 系统内部如何完成它？
- 它读写了哪些状态？
- 它有哪些业务规则和不变量？
- 它失败、重复、并发、超时时会怎样？
- 它依赖哪些配置、权限、租户、缓存、消息、外部系统？
- 后续修改它时最危险的地方在哪里？
- 哪些结论有证据，哪些只是推断，哪些仍待验证？

## 2. 核心认知模型

### 2.1 功能闭环模型

任何业务功能都可以压缩成一个闭环：

```text
角色在某个场景下发起动作，系统读取状态、验证约束、改变状态、产生反馈和副作用。
```

分析时必须回答：

| 问题 | 含义 |
|---|---|
| 谁使用 | 用户、管理员、系统任务、第三方系统、内部服务 |
| 为什么用 | 用户目标、业务目标、系统目标 |
| 从哪里触发 | 页面、按钮、API、消息、定时任务、Webhook、脚本 |
| 读了什么 | DB、缓存、配置、上下文、权限、外部接口 |
| 验证什么 | 参数、权限、状态、额度、规则、幂等、租户、业务不变量 |
| 改了什么 | 表字段、状态机、缓存、索引、文件、外部系统 |
| 产生什么 | UI 反馈、消息事件、通知、审计、日志、异步任务 |
| 失败会怎样 | 回滚、重试、补偿、降级、阻断、脏数据、用户提示 |

### 2.2 证据优先模型

AI 不能把“看起来像”当作结论。所有关键结论都必须归类：

| 置信等级 | 定义 | 可接受证据 |
|---|---|---|
| 已证实 | 有直接证据支撑 | 代码实现、测试、运行结果、配置、DB 结构、接口定义 |
| 高概率推断 | 多个间接证据指向同一结论，但缺直接证据 | 命名、调用关系、文档、日志、历史提交、相邻模块模式 |
| 待验证 | 证据不足或证据冲突 | 缺代码、缺配置、缺运行环境、前后端不一致 |
| 已反证 | 有直接证据推翻原假设 | 代码分支、运行结果、测试失败、真实数据 |

证据优先级建议：

```text
运行实测 > 当前代码实现 > 自动化测试 > 接口/Schema/DDL > 配置 > 提交历史 > 文档 > 注释 > 命名推断 > 口头描述
```

注意：代码是当前行为的强证据，但不一定代表正确业务意图；文档能解释意图，但可能过时。

### 2.3 人类阅读模型

最终文档要按人类建立理解的顺序组织，而不是按源码文件顺序组织：

```text
先总览 -> 再用户路径 -> 再系统实现 -> 再规则状态 -> 再异常风险 -> 最后证据附录
```

禁止输出只适合 AI 自己看的零散笔记。

## 3. 分析前输入与边界

正式阅读源码前，AI 必须先建立分析边界。

### 3.1 最小输入

```md
功能名称：
系统名称/仓库路径：
已知入口：页面 / 菜单 / API / 任务 / 消息 / 类名 / 表名
用户角色：
使用场景：
分析目标：理解功能 / 准备改造 / 排障 / 测试设计 / 文档交接
特别关注：性能 / 安全 / 权限 / 数据一致性 / 并发 / 外部系统 / 兼容性
输出对象：产品 / 开发 / 测试 / 架构师 / 运维 / 新人
范围边界：本次要分析到哪里，不展开哪里
```

### 3.2 边界规则

AI 必须明确：

- 本次分析的主功能是什么；
- 哪些相邻功能只标记为外部依赖，不深入展开；
- 第三方 SDK、基础框架、通用组件是否只看接口契约；
- 是否需要覆盖前端、后端、数据、任务、消息、第三方系统；
- 是否需要查看 Git 历史和运行证据。

边界不清时，先按“能解释主功能闭环”的最小边界推进，并把边界假设写入文档。

## 4. 十二维分析框架

### 4.1 功能意图层

目标：理解这个功能为什么存在。

必须输出：

| 项 | 内容 |
|---|---|
| 一句话定义 | 这个功能用于什么场景，解决什么问题 |
| 使用角色 | 谁使用，谁受影响，谁维护 |
| 业务目标 | 提升效率、完成交易、控制风险、生成数据、触发流程等 |
| 成功标准 | 什么情况算完成，用户看到什么结果 |
| 失败标准 | 什么情况算失败，系统如何反馈 |
| 范围边界 | 包含什么，不包含什么 |

### 4.2 用户旅程层

目标：从真实使用路径理解功能，而不是从文件名理解功能。

必须输出：

| 步骤 | 用户动作 | 前端表现 | 系统行为 | 状态变化 | 证据 |
|---|---|---|---|---|---|

需要覆盖：

- 进入路径；
- 初始化加载；
- 用户输入；
- 前端校验；
- 提交动作；
- 成功反馈；
- 失败反馈；
- 刷新、返回、撤销、重复操作等次级路径。

### 4.3 入口地图层

目标：找全功能入口，避免只看主按钮。

入口类型包括：

| 优先级 | 入口类型 | 示例 | 风险 |
|---|---|---|---|
| P0 | 用户主路径 | 页面、按钮、移动端操作 | 影响最大 |
| P1 | 管理路径 | 后台操作、审批、配置 | 权限风险高 |
| P2 | 批量路径 | 导入、批处理、脚本 | 容易绕过校验 |
| P3 | 系统路径 | 定时任务、MQ、事件、Webhook | 隐蔽且难排查 |
| P4 | 内部路径 | 内部 API、服务调用、SDK | 契约漂移风险 |

必须输出入口地图：

| 优先级 | 入口类型 | 入口位置 | 触发方式 | 进入后调用 | 是否主路径 | 证据 |
|---|---|---|---|---|---|---|

### 4.4 交互与前端状态层

目标：理解用户界面如何表达系统状态。

需要检查：

- 页面路由、菜单、权限控制；
- 组件结构、弹窗、表单、列表、详情页；
- 本地状态、全局状态、缓存状态；
- 表单默认值、前端枚举、显示文案；
- 前端校验与后端校验是否一致；
- 加载、空态、错误、禁用、重复提交状态；
- 成功后刷新、跳转、局部更新、乐观更新；
- 多标签页、返回、刷新、弱网情况下的行为。

输出模板：

| 交互点 | 触发事件 | 前端状态变化 | 调用接口/消息 | 成功反馈 | 失败反馈 | 证据 |
|---|---|---|---|---|---|---|

### 4.5 契约层

目标：明确系统边界之间互相承诺了什么。

契约不限于 HTTP API，也包括消息、事件、RPC、文件、数据库共享表、第三方 Webhook。

必须输出：

| 契约 | 类型 | 调用方 | 提供方 | 请求/消息关键字段 | 响应/结果 | 错误语义 | 证据 |
|---|---|---|---|---|---|---|---|

字段级说明：

| 字段 | 类型 | 必填 | 含义 | 取值/枚举 | 默认值 | 兼容性要求 | 前端/调用方依赖 |
|---|---|---|---|---|---|---|---|

重点检查：

- 字段是否存在前后端含义不一致；
- 枚举值是否完整且一致；
- 错误码是否被前端特殊处理；
- 是否有隐式默认值；
- 是否存在兼容旧客户端或旧数据的逻辑；
- 是否有幂等键、版本号、时间戳、签名、租户上下文。

### 4.6 后端编排层

目标：理解后端如何组织业务动作。

不要只写 Controller -> Service -> DAO。必须标注责任和边界：

```text
入口适配
 -> 参数转换
 -> 认证/授权/租户/上下文
 -> 业务规则校验
 -> 领域对象/聚合处理
 -> 事务边界
 -> 持久化
 -> 缓存/搜索索引/文件/消息/外部调用
 -> 返回模型转换
```

输出模板：

| 步骤 | 组件/方法 | 职责 | 输入 | 输出 | 失败行为 | 证据 |
|---|---|---|---|---|---|---|

必须说明：

- 哪一层拥有业务规则；
- 哪一层只做适配、转换或持久化；
- 是否存在跨层越权；
- 是否存在重复规则或前后端双份规则；
- 是否有事务、锁、重试、补偿、降级。

### 4.7 数据与状态层

目标：理解功能改变了什么世界状态。

用以下四问穿透：

```text
读了哪些状态？
验证了哪些约束？
改变了哪些状态？
产生了哪些副作用？
```

数据读写表：

| 数据源 | 表/集合/缓存/索引/文件 | 字段 | 读/写 | 时机 | 事务保护 | 失败影响 | 证据 |
|---|---|---|---|---|---|---|---|

状态流转表：

| 当前状态 | 操作 | 前置条件 | 下一个状态 | 是否可回退 | 非法转换行为 | 证据 |
|---|---|---|---|---|---|---|

必须识别：

- 主实体和从属实体；
- 状态字段和状态机；
- 中间态、终态、失败态；
- 软删除、归档、冻结、取消、撤回等特殊状态；
- 历史数据、审计数据、流水数据；
- 缓存、搜索索引、报表、同步表是否一起变化。

### 4.8 业务规则与不变量层

目标：把代码条件翻译成人类可理解、可测试的业务规则。

规则表：

| 编号 | 业务规则 | 触发条件 | 系统行为 | 违反时结果 | 测试场景 | 证据 | 置信度 |
|---|---|---|---|---|---|---|---|

规则分类：

- 参数规则；
- 权限规则；
- 状态规则；
- 金额/数量/时间规则；
- 唯一性规则；
- 幂等规则；
- 互斥规则；
- 优先级规则；
- 配置开关规则；
- 合规/审计规则；
- 多租户/数据隔离规则。

不变量示例：

```text
同一订单不能同时处于“已取消”和“已支付”。
一笔支付成功后必须存在对应流水。
同一用户对同一资源只能有一个有效授权。
```

不变量比单个 if/else 更重要，因为它指导未来修改不能破坏什么。

### 4.9 副作用与集成层

目标：识别功能对系统其他部分的影响。

副作用包括：

- 发消息、发事件；
- 调第三方接口；
- 写缓存、删缓存、刷新索引；
- 发通知、短信、邮件、站内信；
- 写审计日志、操作日志、埋点；
- 生成文件、报表、单据；
- 触发工作流、审批流、任务流；
- 影响下游统计、计费、库存、积分、权限。

副作用矩阵：

| 副作用 | 触发时机 | 同步/异步 | 是否在事务内 | 失败处理 | 重试/补偿 | 下游消费者 | 证据 |
|---|---|---|---|---|---|---|---|

必须特别说明：数据库提交成功但副作用失败时，系统会处于什么状态。

### 4.10 异常、边界、并发层

目标：理解系统在非理想世界中的行为。

异常矩阵：

| 场景 | 触发条件 | 当前行为 | 用户可见结果 | 数据影响 | 恢复方式 | 风险等级 | 证据 |
|---|---|---|---|---|---|---|---|

必须覆盖：

- 参数非法；
- 权限不足；
- 数据不存在；
- 状态已变化；
- 重复提交；
- 并发修改；
- 超时；
- 下游失败；
- MQ 重复、乱序、积压；
- 缓存不一致；
- 部分成功；
- 事务回滚；
- 历史脏数据；
- 非法枚举；
- 时间边界、时区、过期；
- 多租户/多组织/多环境隔离；
- 旧版本客户端或旧数据兼容。

### 4.11 质量属性层

目标：从“能不能跑”提升到“能不能长期安全演进”。

质量雷达：

| 维度 | 评级 | 关键发现 | 影响 | 建议 |
|---|---|---|---|---|
| 可维护性 | 良好/警告/危险 |  |  |  |
| 可测试性 | 良好/警告/危险 |  |  |  |
| 一致性 | 良好/警告/危险 |  |  |  |
| 幂等性 | 良好/警告/危险 |  |  |  |
| 安全性 | 良好/警告/危险 |  |  |  |
| 性能 | 良好/警告/危险 |  |  |  |
| 可观测性 | 良好/警告/危险 |  |  |  |
| 可扩展性 | 良好/警告/危险 |  |  |  |
| 可恢复性 | 良好/警告/危险 |  |  |  |

评估时不要空泛评价，必须绑定证据或明确标记为推断。

### 4.12 演进历史层

目标：理解功能为什么变成现在这样。

可查看：

```bash
git log --oneline --follow -- <核心文件>
git show <commit> --stat
git blame <核心文件>
```

输出：

| 时间/提交 | 变更摘要 | 变更类型 | 可能原因 | 当前影响 | 证据 |
|---|---|---|---|---|---|

重点识别：

- 一次性设计还是长期叠加；
- 热修复和事故修复；
- 兼容旧逻辑；
- 半完成重构；
- 已废弃但仍保留的分支；
- 注释、TODO、临时开关；
- 文档与代码是否漂移。

没有 Git 历史时，不要编造，明确写“未获取到历史证据”。

## 5. 推荐执行 SOP

### 第 0 轮：任务定标

产出：功能定义卡。

退出标准：能用一句话说明“谁在什么场景下做什么，系统产生什么结果”。

### 第 1 轮：入口发现

产出：入口地图、候选 API/消息/任务清单。

退出标准：至少找到一条真实触发路径，并标记其他可能入口。

### 第 2 轮：主路径打通

产出：端到端主链路。

退出标准：能用 5 到 15 步讲清主路径，每一步有证据。

### 第 3 轮：契约与交互确认

产出：交互表、契约表、字段表。

退出标准：前端/调用方和后端/提供方的关键字段、错误、状态语义已对齐或标记冲突。

### 第 4 轮：数据与状态建模

产出：数据读写表、状态流转表、副作用矩阵。

退出标准：明确读写哪些状态，哪些变化在事务内，哪些是最终一致或无保障。

### 第 5 轮：业务规则抽取

产出：业务规则表、不变量表。

退出标准：关键 if/else、校验、权限、状态约束已经翻译成人类规则，并能对应测试场景。

### 第 6 轮：异常与质量分析

产出：异常矩阵、风险清单、质量雷达。

退出标准：覆盖失败、并发、重复、超时、外部依赖、缓存/消息不一致等关键风险。

### 第 7 轮：历史与演进分析

产出：演进时间线、技术债线索。

退出标准：能区分设计意图、历史兼容、补丁痕迹；无法确认的地方明确标记。

### 第 8 轮：证据复核与冲突裁决

产出：证据表、冲突清单、待验证问题。

退出标准：关键结论都有证据等级；证据冲突已裁决或降级为待验证。

### 第 9 轮：人类文档整编

产出：最终功能理解文档。

退出标准：文档能被产品、开发、测试、架构师分别使用，不需要读者重新翻源码才能理解主结论。

## 7. 多 Agent 协作框架

当系统复杂、前后端分离、微服务多、异步路径多时，可以拆成多个 Agent。

| 角色 | 职责 | 产出 | 禁止事项 |
|---|---|---|---|
| Lead Agent | 定边界、拆任务、合并、最终裁决 | 总计划、最终文档 | 不跳过证据复核 |
| Product/UX Agent | 用户旅程、交互、业务目标 | 用户旅程、交互状态 | 不猜后端实现 |
| Frontend Agent | 页面、路由、状态、API 调用 | 前端链路、契约调用点 | 不猜业务最终落库 |
| Backend Agent | API、服务编排、事务、异常 | 后端执行链路 | 不忽略前端契约 |
| Data Agent | 表、状态机、缓存、消息、副作用 | 数据状态模型 | 不单独下业务结论 |
| Integration Agent | 外部系统、消息、任务、Webhook | 集成与副作用矩阵 | 不把异步路径当主路径 |
| Security/Quality Agent | 权限、安全、性能、可观测性 | 风险和质量雷达 | 不提出无证据泛化评价 |
| Verifier Agent | 证据复核、冲突裁决、置信度 | 证据表、冲突清单 | 不产生未经验证的新结论 |

交接包模板：

```md
## 交接包 - <Agent> - <范围>

### 已证实结论
- 结论：
  证据：
  置信度：

### 高概率推断
- 推断：
  依据：
  缺失证据：

### 待验证问题
- 问题：
  建议验证方式：
  影响：

### 冲突或风险
- 冲突点：
  双方证据：
  建议裁决：
```

冲突裁决规则：

1. 先比较证据等级；
2. 再比较证据是否来自当前版本；
3. 再看是否存在运行实测；
4. 仍无法裁决时降级为待验证；
5. 最终文档必须写明冲突，不允许隐去。

## 8. 任务规模裁剪规则

不是所有功能都需要完整重型流程。按复杂度选择深度。

| 规模 | 适用场景 | 必做内容 |
|---|---|---|
| 轻量分析 | 单页面、单接口、无状态复杂度 | 摘要、入口、主链路、规则、证据 |
| 标准分析 | 常规前后端功能，有状态和权限 | 十二维中的前 10 维，附证据表 |
| 深度分析 | 金融、订单、库存、审批、权限、计费、跨服务、异步复杂功能 | 全量十二维 + 历史 + 质量雷达 + 测试场景 |
| 审计级分析 | 高风险安全/合规/数据一致性功能 | 全量十二维 + 运行验证 + 风险分级 + 复核记录 |

裁剪原则：可以减少篇幅，不能省略证据、边界、规则、状态、异常。

## 9. 质量门禁

最终文档交付前，AI 必须自检。

### 9.1 必须满足

- 已说明功能目标和边界；
- 已找到真实入口，不只依据名称猜测；
- 已打通至少一条端到端主路径；
- 已说明关键契约和字段语义；
- 已说明数据读写和状态流转；
- 已提炼业务规则，而不是复述 if/else；
- 已覆盖异常、重复、并发、下游失败；
- 已列出副作用和外部依赖；
- 已标注权限、安全、隔离、性能、可观测性风险；
- 已附证据表和待确认问题；
- 关键结论都有置信度；
- 文档先给摘要，再给细节，最后给证据。

### 9.2 不合格信号

- 只看 Controller、页面或一个 Service 就下结论；
- 只讲代码调用，不讲用户目标；
- 把技术条件直接当业务规则；
- 忽略异步、缓存、定时任务、外部系统；
- 不区分主路径、分支路径、异常路径；
- 没有字段级契约说明；
- 没有状态流转；
- 没有待验证问题；
- 结论全是“已证实”；
- 大量复制源码，但没有形成人类可读解释；
- 没有说明证据来自哪里。

## 10. 可直接交给其他 AI 的标准指令

```text
你现在是“信息系统功能深度理解分析师”。

任务：请对【系统/仓库】中的【功能名称】进行端到端深度理解，并输出一份供人类阅读的功能理解文档。

你不能只解释代码。你必须还原这个功能的真实系统行为：用户为什么使用它、从哪里触发、前端如何交互、契约如何定义、后端如何编排、数据和状态如何变化、有哪些业务规则、副作用、异常、并发、安全、性能和演进风险。

执行要求：
1. 先建立功能边界和分析假设，不要一开始随机读文件。
2. 按端到端功能闭环分析：用户意图 -> 入口 -> 交互 -> 契约 -> 后端编排 -> 数据状态 -> 业务规则 -> 副作用 -> 异常风险 -> 证据。
3. 所有关键结论必须附证据位置，不能只根据命名猜测。
4. 明确区分：已证实 / 高概率推断 / 待验证 / 已反证。
5. 先打通主路径，再补分支、异常、异步、外部依赖和历史演进。
6. 把 if/else 翻译成业务规则，把数据变化整理成状态流转，把风险整理成异常矩阵。
7. 输出文档必须先给人类可读摘要，再给技术详解，最后附证据表和待确认问题。

最终文档至少包含：
1. 一页摘要
2. 功能边界
3. 用户旅程
4. 入口地图
5. 交互与前端状态
6. API/RPC/消息/任务契约
7. 后端执行链路
8. 数据读写与状态流转
9. 业务规则与不变量
10. 副作用与外部依赖
11. 异常、边界、并发与幂等
12. 安全、权限与隔离
13. 性能与可观测性
14. 质量评估
15. 测试场景建议
16. 演进历史
17. 证据表
18. 待确认问题
19. 给后续开发者的修改提示

交付标准：一个没有读过源码的高级工程师，读完你的文档后，应能安全地修改这个功能、设计测试用例、评估上线风险。如果做不到，继续补证据和分析。
```

## 11. 最终判断标准

功能理解文档的价值不在于“覆盖了多少文件”，而在于它是否把复杂系统压缩成可决策的知识。

优秀文档应该做到：

```text
业务人员能看懂它解决什么问题；
产品能看懂功能边界和规则；
开发能看懂实现链路和修改风险；
测试能直接提取测试场景；
架构师能评估一致性、扩展性和技术债；
运维能知道失败和恢复路径；
新成员能快速建立系统心智模型。
```

这是 AI 做功能理解的最高标准。
