# 储值方案储值设计说明

## 1. 文档范围

本文描述“会员按储值方案储值”的设计，包括：

- 储值方案配置模型。
- 小程序/H5/后台与 POS 的储值 API。
- POS 本地账单与 CRM 云端入账的关系。
- 涉及的数据表、关键字段及赋值来源。
- 事务处理、幂等控制、异步赠券与失败边界。
- 与老普通储值方案的差异。

## 2. 总体设计目标

储值方案不是简单的“充多少送多少”，它把储值金额、赠送金额、赠送积分、赠券、券包、会员升级、可用组织、会员等级、购买限制、协议、渠道等规则集中在 `crm_deposit_plan` 中配置。

设计目标是：

1. 保留老普通充值的余额入账引擎和 POS 收款主时序。
2. 新增储值方案规则层，不直接改写老充值主逻辑。
3. POS 侧用本地账单承载支付事实，CRM 侧用交易任务承载会员余额事实。
4. 金额最终以 CRM 储值方案档位主数据为准，前端/POS 传入金额只作为展示和防篡改校验。
5. 赠券、升级、积分等后置权益不阻塞主余额入账。

## 3. 核心流程

### 3.1 小程序/H5/后台线上储值

线上储值主流程：

1. 前端查询可用储值方案。
2. 用户选择 `planLid + tierIndex`。
3. 调用 `/deposit-plan/charge/commit` 创建支付单、交易任务和购买记录。
4. 支付平台回调或测试环境 `mock-success` 触发支付成功处理。
5. CRM 更新会员余额、购买记录状态和方案销售统计。
6. 余额成功后异步处理赠券、券包、会员升级和积分。

对应 API 图见 [API、参数与用途关系图](./crm-deposit-plan-api-parameters-flow.svg)。

### 3.2 POS 储值方案充值

POS 储值方案主流程：

1. POS 页面调用本地接口查询可用方案。
2. POS 页面调用本地 `cardCharge` 创建或复用本地充值单。
3. 本地写入 `dwd_bill / dwd_food / dwd_pay`，并打上 `depositPlanLid / depositTierIndex` 标记。
4. POS 走老收款和 `checkOut` 收口流程。
5. `OrderServiceUtil.dealCharge` 根据 `dwd_bill.depositPlanLid` 分流。
6. 新储值方案进入 `DepositPlanChargeUtil`。
7. POS 同步调用 CRM `/deposit-plan/charge/commitByPos`。
8. CRM 完成交易任务、余额、购买记录、销售统计和后置权益处理。
9. POS 用 CRM 返回的 `CardBalanceVo` 回写本地账单和项目行。

关键点：`/member_deposit_plan_ops/cardCharge` 只是 POS 本地建单、收款和支付收口入口，真正的 CRM 入账 API 是 `/deposit-plan/charge/commitByPos`。

## 4. API 设计

### 4.1 储值方案查询 API

| 调用方 | API | 参数 DTO | 用途 |
| --- | --- | --- | --- |
| 后台/POS 云端 | `POST /crm_deposit_plan/listAvailable` | `CrmDepositPlanAvailableQueryDTO` | 查询当前门店、会员卡可用的储值方案和档位 |
| 小程序 | `POST /crm_deposit_plan/listByWx` | `CrmDepositPlanAvailableQueryDTO` | 小程序登录态查询可用方案，服务端从用户上下文补 `mid` |
| POS 本地页面 | `POST {PATH_PREFIX_MERCHANT}/member_deposit_plan_ops/listAvailableDepositPlans` | `CrmDepositPlanAvailableQueryDTO` | POS 本地入口，补 `mid` 后转调 CRM `/crm_deposit_plan/listAvailable` |

`CrmDepositPlanAvailableQueryDTO` 关键字段：

| 字段 | 含义 |
| --- | --- |
| `mid` | 商户 ID；部分入口由登录上下文补齐 |
| `sid` | 门店 ID，必填，用于过滤门店可用方案 |
| `cardLid` | 会员卡 lid，可用于按会员卡类型、等级过滤 |
| `brandLid` | 品牌过滤 |

### 4.2 线上储值提交 API

| API | 参数 DTO | 返回 | 用途 |
| --- | --- | --- | --- |
| `POST /deposit-plan/charge/commit` | `DepositPlanChargeCommitDTO` | `DepositPlanChargeCommitVO` | 小程序/H5/后台提交储值方案充值，创建支付单和任务 |
| `POST /deposit-plan/charge/query_charge_status` | `ChargeQueryStatus` | `Boolean` | 按 `taskLid` 查询储值任务状态 |
| `POST /deposit-plan/charge/mock-success` | `DepositPlanChargeMockSuccessDTO` | `String` | 测试环境模拟支付成功 |

`DepositPlanChargeCommitDTO` 关键字段：

| 字段 | 含义 |
| --- | --- |
| `planLid` | 储值方案 ID |
| `tierIndex` | 档位索引，从 0 开始 |
| `cardLid` | 会员卡 ID；不传时可取登录用户会员卡 |
| `appId` | 小程序 appId |
| `h5` | 是否 H5 支付 |
| `channelCode` | 渠道编码，例如 `POS / MINI_PROGRAM / H5` |
| `brandLid` | 品牌 ID |

线上 `commit` 的核心作用：

1. 校验会员卡、方案状态、档位、门店、会员等级、购买限制。
2. 按档位 `saveAmount` 调支付服务。
3. 创建 `crm_deal_task`。
4. 创建 `crm_deal_task_item`。
5. 创建 `crm_deposit_charge_record`。
6. 发送延时关单消息。

## 5. 微信支付流程

储值方案使用微信支付时，需要区分小程序/H5 链路和门店 POS 链路。两者最终都会进入 CRM 储值成功处理，但微信支付单的创建方、支付事实落点和入账触发点不同。

### 5.1 小程序/H5 微信支付流程

小程序/H5 的微信支付由 CRM 云端创建支付单，支付成功后由 CRM 自己完成入账。

```text
小程序/H5
  -> /crm_deposit_plan/listByWx 查询可用储值方案
  -> 用户选择 planLid + tierIndex
  -> /deposit-plan/charge/commit 提交储值
  -> CRM 根据 crm_deposit_plan.tier_rule[tierIndex].saveAmount 计算支付金额
  -> CRM 调支付服务创建微信支付单
  -> 返回 DepositPlanChargeCommitVO 与 taskLid
  -> 小程序/H5 拉起微信支付
  -> 微信支付成功回调或关单队列消费
  -> DepositPlanChargeService.onPaymentSuccess
  -> CardBalanceService.execute 更新会员余额
  -> 更新 crm_deposit_charge_record 成功
  -> 更新 crm_deposit_plan 销售统计
  -> 异步赠券、送积分、会员升级
```

`/deposit-plan/charge/commit` 内部按 `h5` 字段区分支付请求：

| 场景 | 请求对象 | 支付服务调用 | 说明 |
| --- | --- | --- | --- |
| 小程序支付 | `MiniPayRequest` | `dealReactiveFeign.miniPay(...)` | `h5=false` 时使用 |
| H5/公众号 JS 支付 | `JsPayRequest` | `dealReactiveFeign.jsPay(...)` | `h5=true` 时使用 |

支付金额来源：

```text
crm_deposit_plan.tier_rule[tierIndex].saveAmount * 100
```

这里乘以 `100` 是把元转换为分。小程序/H5 前端不应该自己决定最终支付金额，最终金额以 CRM 储值方案档位主数据为准。

小程序/H5 支付涉及的核心表：

| 阶段 | 表 | 写入或更新内容 |
| --- | --- | --- |
| 提交储值 | `crm_deal_task` | 创建支付中任务，`trade_state=USERPAYING`，`out_trade_no=支付服务订单号` |
| 提交储值 | `crm_deal_task_item` | 创建充值子项，金额来自档位规则 |
| 提交储值 | `crm_deposit_charge_record` | 创建购买记录，保存 `plan_lid`、`tier_index`、`tier_snapshot` |
| 支付成功 | `crm_card` | 更新余额、本金余额、赠送余额、积分余额 |
| 支付成功 | `crm_card_record` | 新增会员卡储值流水 |
| 支付成功 | `crm_deposit_charge_record` | `trade_state` 从支付中改为成功 |
| 支付成功 | `crm_deposit_plan` | `total_sold_count + 1`、`total_sold_amount + saveAmount` |
| 后置权益 | `crm_deposit_coupon_schedule` | 周期发券时创建或更新发券计划 |

### 5.2 门店 POS 微信支付流程

门店 POS 的微信支付由 POS 本地收款流程处理。CRM 不直接创建 POS 的微信支付单；CRM 只在 POS 支付成功收口后接收入账请求。

```text
POS 页面
  -> /member_deposit_plan_ops/listAvailableDepositPlans 查询方案
  -> 用户选择 planLid + tierIndex
  -> /member_deposit_plan_ops/cardCharge 创建或复用本地充值单
  -> 写 dwd_bill / dwd_food / dwd_pay
  -> POS 本地发起微信支付或扫码支付
  -> 微信支付成功
  -> POS CloseMpScHandler 收口
  -> OrderServiceUtil.checkOut(...)
  -> OrderServiceUtil.dealCharge(...)
  -> dwd_bill.depositPlanLid 不为空，分流到 DepositPlanChargeUtil
  -> POS 调 CRM /deposit-plan/charge/commitByPos
  -> CRM 同步创建任务、购买记录并执行余额入账
  -> CRM 返回 CardBalanceVo
  -> POS 回写 dwd_bill / dwd_food 的最终金额和余额字段
```

POS 微信支付事实先落在本地表：

| 表 | 作用 |
| --- | --- |
| `dwd_bill` | 本地充值账单主表，记录 `saas_order_key`、会员快照、方案档位、金额、余额回写结果 |
| `dwd_food` | 本地储值方案项目行，当前新方案只生成一行 |
| `dwd_pay` | 本地支付方式和支付金额，微信支付方式也记录在这里 |

POS 支付成功后，CRM 入账请求中的关键映射是：

```text
dwd_bill.saasOrderKey
  -> DepositPlanPosChargeDTO.orderId
  -> crm_deal_task.out_trade_no

dwd_bill.depositPlanLid
  -> DepositPlanPosChargeDTO.planLid
  -> crm_deposit_plan.lid

dwd_bill.depositTierIndex
  -> DepositPlanPosChargeDTO.tierIndex
  -> crm_deposit_plan.tier_rule[tierIndex]
```

POS 的微信支付和 CRM 入账是两个事实层：

- POS 负责微信收款事实：谁收款、用什么支付方式、支付是否关闭、退款状态等。
- CRM 负责会员储值事实：会员余额是否增加、赠送金额和积分是否入账、储值方案购买记录是否成功。

### 5.3 小程序/H5 与 POS 微信支付差异

| 环节 | 小程序/H5 | 门店 POS |
| --- | --- | --- |
| 微信支付单创建方 | CRM 调支付服务创建 | POS 本地收款流程创建 |
| 储值提交 API | `/deposit-plan/charge/commit` | `{PATH_PREFIX_MERCHANT}/member_deposit_plan_ops/cardCharge` |
| CRM 入账触发 | 微信支付回调或关单队列消费后 `onPaymentSuccess` | POS 支付收口后同步调用 `/deposit-plan/charge/commitByPos` |
| CRM 入账 API | 不直接暴露给前端，内部进入 `onPaymentSuccess` | `/deposit-plan/charge/commitByPos` |
| POS 本地账单 | 不涉及 | 涉及 `dwd_bill / dwd_food / dwd_pay` |
| 幂等关键 | CRM `taskLid / outTradeNo` | `dwd_bill.saasOrderKey = crm_deal_task.out_trade_no` |
| 最终金额口径 | CRM 档位主数据 | CRM 档位主数据，POS 本地成功后按 CRM 返回值回写 |

一句话总结：

```text
小程序/H5：CRM 先创建微信支付单，支付成功后 CRM 自己入账。
门店 POS：POS 本地先完成微信收款，支付收口时再通知 CRM 入账。
```

## 6. 表设计与字段赋值

### 6.1 配置表

| 表 | 作用 | 重要字段 |
| --- | --- | --- |
| `crm_deposit_plan` | 储值方案主表 | `lid`、`name`、`status`、`save_type`、`tier_rule`、`channels`、`applicable_orgs`、`member_level_rule`、`purchase_limit_rule`、`total_sold_count`、`total_sold_amount` |
| `crm_deposit_category` | 储值方案分类 | `lid`、`name`、`sort_order`、`brand_ids` |
| `crm_deposit_agreement` | 储值协议 | `lid`、`agreement_type`、`content`、`status` |
| `crm_card_type` | 会员方案 | 用于约束适用会员方案 |
| `crm_card_level` | 会员等级 | 用于约束适用会员等级和升级目标 |

`crm_deposit_plan` 是配置中心，应该放在配置图中心。分类、协议、会员方案、会员等级都是它的约束或引用对象。

### 6.2 CRM 交易表

| 表 | 作用 | 关键赋值 |
| --- | --- | --- |
| `crm_deal_task` | 交易主任务，余额入账引擎的核心输入 | `lid=tradeNo`；`card_lid=会员卡`；`operation_model=Charge`；`principal_amount=tier.saveAmount`；`give_amount=tier.giftAmount`；`give_point=tier.giftPoints`；`trade_state=USERPAYING/SUCCESS`；`out_trade_no=支付单号或 POS orderId`；`save_rule=plan.name`；`save_rule_code=plan.lid` |
| `crm_deal_task_item` | 交易任务子项 | `task_lid=crm_deal_task.lid`；`principal_amount=tier.saveAmount`；`give_amount=tier.giftAmount`；`amount=本金+赠送金额`；`give_point=tier.giftPoints` |
| `crm_deposit_charge_record` | 储值方案购买记录 | `plan_lid=plan.lid`；`tier_index=档位索引`；`card_lid=会员卡`；`task_lid=crm_deal_task.lid`；`save_amount=tier.saveAmount`；`gift_amount=tier.giftAmount`；`gift_points=tier.giftPoints`；`tier_snapshot=档位快照`；`channel_code=渠道`；`trade_state=USERPAYING/SUCCESS/CLOSED` |
| `crm_deposit_coupon_schedule` | 周期发券计划 | `plan_lid`、`tier_index`、`card_lid`、`charge_record_lid`、`tier_snapshot`、`issue_cycle_type`、`issue_time`、`total_issue_times`、`issued_times`、`next_issue_date`、`status` |
| `crm_card` | 会员卡余额和积分 | 由 `CardBalanceService.execute` 更新余额、本金余额、赠送余额、积分等 |
| `crm_card_record` | 会员卡操作记录 | 由老余额引擎记录储值流水，关联 `crm_deal_task.lid` |

### 6.3 POS 本地账单表

| 表 | 作用 | 关键赋值 |
| --- | --- | --- |
| `dwd_bill` | POS 本地充值账单主表 | `order_op_type=E`；`saas_order_key=POS 订单号`；`card_no=会员卡号`；`deposit_plan_lid=planLid`；`deposit_tier_index=tierIndex`；`paid_amount=前端本金`；`send_amount=前端赠送金额`；`card_task_lid=CRM taskLid`；余额字段由 CRM 返回值回写 |
| `dwd_food` | POS 本地充值项目行 | 新储值方案只生成 1 行；`food_name=方案名称或储值方案-档位N`；金额字段先按前端选择写入，收口后可由 CRM 返回值覆盖 |
| `dwd_pay` | POS 本地支付方式 | 来源于 `DepositPlanChargeExDTO.pays`，用于收款和拼接 `payWay` |

POS 本地账单的核心识别字段是 `dwd_bill.depositPlanLid`：

- 为空：老普通充值，继续走 `/crm_card_op/cardChargeSign`。
- 不为空：新储值方案，切到 `DepositPlanChargeUtil`，最终走 `/deposit-plan/charge/commitByPos`。

### 6.4 小程序与 POS 更新字段差异

小程序和 POS 最终都会更新 CRM 云端核心表，但 POS 额外更新本地账单表。两者最大的区别是：

- 小程序没有 `dwd_bill / dwd_food / dwd_pay` 本地账单层。
- POS 先写本地账单，再在支付收口时调用 CRM 入账，最后用 CRM 返回余额回写本地账单。

#### 6.4.1 CRM 云端表更新字段对比

| 表 | 小程序更新字段 | POS 更新字段 | 差异说明 |
| --- | --- | --- | --- |

#### 6.4.3 关键映射关系

小程序的关键映射：

```text
/deposit-plan/charge/commit
  -> 支付服务订单号
  -> crm_deal_task.out_trade_no
  -> crm_deposit_charge_record.task_lid = crm_deal_task.lid
```

POS 的关键映射：

```text
dwd_bill.deposit_plan_lid
  -> crm_deposit_plan.lid

dwd_bill.deposit_tier_index
  -> crm_deposit_plan.tier_rule[tierIndex]

dwd_bill.saas_order_key
  -> DepositPlanPosChargeDTO.orderId
  -> crm_deal_task.out_trade_no

dwd_bill.card_task_lid
  -> crm_deal_task.lid
```

因此，小程序和 POS 的 CRM 入账结果一致，但字段更新范围不同：

```text
小程序：只更新 CRM 云端表
POS：先更新 POS 本地表，再更新 CRM 云端表，最后回写 POS 本地表
```

## 7. 事务处理与幂等设计

### 7.1 线上 `commit` 事务

`/deposit-plan/charge/commit` 使用 `@Transactional(rollbackFor = Exception.class)`。

事务内主要完成：

1. 查询并校验会员卡、门店、方案、档位。
2. 执行方案可用性校验。
3. 执行购买限制校验。
4. 调支付服务创建支付单。
5. 创建 `crm_deal_task`。
6. 创建 `crm_deal_task_item`。
7. 创建 `crm_deposit_charge_record`。
8. 发送延时关单消息。

支付成功后的余额入账不在 `commit` 立即完成，而是在支付回调或测试 `mock-success` 中进入 `onPaymentSuccess`。

### 7.2 POS `commitByPos` 事务

`/deposit-plan/charge/commitByPos` 是 POS 同步入账接口，事务处理分为两层：

第一层：幂等与锁。

- Redis 锁 key：`depositPlan:pos:charge:{mid}:{orderId}`。
- 如果同一个 `mid + orderId` 已有 `crm_deal_task`：
  - `trade_state=SUCCESS`：直接返回当前余额 VO，视为幂等成功。
  - 非成功：提示订单处理中，禁止重复提交。

第二层：阶段性事务。

- 使用 `transactionTemplate` 创建 `crm_deal_task`、`crm_deal_task_item`、`crm_deposit_charge_record`。
- 这一步提交后，并发请求可以看到 `outTradeNo=orderId` 的任务，避免重复创建。
- 随后同步调用 `onPaymentSuccess` 完成余额入账。
- 入账后再次查询 `crm_deal_task.trade_state`，若不是 `SUCCESS`，抛出“储值方案余额入账失败”。

这里的关键幂等字段是：

```text
DepositPlanPosChargeDTO.orderId
= dwd_bill.saasOrderKey
= crm_deal_task.out_trade_no
```

### 7.3 `onPaymentSuccess` 事务

`onPaymentSuccess` 使用 `@Transactional(rollbackFor = Exception.class)`。

主事务内完成：

1. 调用 `CardBalanceService.execute(mid, taskLid, cardLid, false)`，复用老余额引擎更新会员余额。
2. 将 `crm_deposit_charge_record.trade_state` 从支付中更新为成功。
3. 更新 `crm_deposit_plan.total_sold_count / total_sold_amount`。

主事务后触发：

```text
CompletableFuture.runAsync(() -> handlePostSuccessBenefits(task, notifyAttach))
```

后置权益包括：

- 赠券和券包。
- 会员等级升级。
- 赠送积分。

异常处理策略：后置权益异常只记录日志，不影响余额主事务。

## 8. 赠券与延迟处理

赠券由 `DepositPlanCouponService.handleGiftIssuance` 处理。

| 发放模式 | 行为 | 表 |
| --- | --- | --- |
| `ONE_TIME` | 立即发放档位配置的优惠券和券包 | 券相关表，当前文档不展开 |
| `PERIODIC` | 创建周期发券计划，后续由定时任务发放 | `crm_deposit_coupon_schedule` |

周期发券任务：

- Job：`depositPlanCouponScheduleJob`。
- 查询条件：`status=1 AND next_issue_date <= today AND deleted=0`。
- 每条计划调用 `DepositPlanCouponService.executeScheduledIssuance(schedule)`。
- 发放后 `issued_times + 1`。
- 达到 `total_issue_times` 后 `status=0`，表示完成。
- 未完成则计算下一次 `next_issue_date`。

需要注意：

1. 余额入账和购买记录成功是主流程。
2. 发券是异步后置动作。
3. 异步动作没有阻塞余额主事务。
4. 如果发券过程异常，当前实现主要依赖日志和周期计划继续调度；一次性发券的失败不会自动回滚余额。

## 9. 撤销储值方案充值

POS 新储值方案有独立撤销入口：

```text
POST {PATH_PREFIX_MERCHANT}/member_deposit_plan_ops/revokeCharge
```

撤销思路与老普通充值一致，但增加了储值方案单隔离：

1. 根据 CRM `taskLid` 找到本地已关闭充值单。
2. 仅允许撤销 `depositPlanLid` 不为空的储值方案单。
3. 先撤销或退款 POS 本地账单。
4. 再调用 CRM 老接口 `/crm_card_op/revokeChargeSign` 撤销会员余额。

这个设计能避免把老普通充值单误当作储值方案单撤销。

但需要明确：当前撤销入口复用老 CRM 余额撤销接口，源码中没有看到“储值方案赠券自动取消”的完整闭环。也就是说：

- 余额、本金、赠送金的撤销依赖老 `revokeChargeSign`。
- 储值方案赠券是否撤销，需要看券系统是否按 `taskLid / chargeRecordLid` 建立可撤销关系。
- 从当前分析到的储值方案代码看，发券属于异步后置权益，撤销时没有在新入口中显式取消赠券。

## 10. 与老普通充值方案对比

| 对比项 | 老普通充值 | 新储值方案充值 |
| --- | --- | --- |
| POS 入账 API | `/crm_card_op/cardChargeSign` | `/deposit-plan/charge/commitByPos` |
| POS 本地入口 | `/member_ops/cardCharge` | `/member_deposit_plan_ops/cardCharge` |
| 本地账单类型 | `order_op_type=E` | 仍为 `order_op_type=E` |
| 新老分流字段 | 无 | `dwd_bill.depositPlanLid` |
| 金额来源 | POS/老规则传入 | CRM `crm_deposit_plan.tier_rule` 档位主数据 |
| 防篡改 | 依赖老充值校验 | `tier.saveAmount` 与请求 `saveAmount` 必须一致 |
| 交易任务 | `crm_deal_task` | 仍复用 `crm_deal_task` |
| 子任务 | `crm_deal_task_item` | 仍复用 `crm_deal_task_item` |
| 购买记录 | 无独立储值方案购买记录 | 新增 `crm_deposit_charge_record` |
| 赠券逻辑 | 老 `giveCouponId/giveCouponNum` 或老规则 | 新 `DepositPlanCouponService`，支持一次性和周期发放 |
| 周期发券 | 不在老普通充值主模型中 | `crm_deposit_coupon_schedule + XXL-Job` |
| 会员升级 | 老逻辑不按储值方案档位配置 | 储值档位可配置升级 |
| 积分 | 老余额引擎或老规则 | 储值档位配置 `giftPoints`，后置处理 |
| 撤销 | `/crm_card_op/revokeChargeSign` | POS 新入口限制储值方案单后，仍调用 `/crm_card_op/revokeChargeSign` |

设计上，新方案不是替换老充值，而是在老 POS 收款和老 CRM 余额引擎之上加一层储值方案规则：

```text
老 POS 收款主时序
  -> dwd_bill.depositPlanLid 分流
  -> 新储值方案 commitByPos
  -> 老 CardBalanceService.execute 余额引擎
  -> 新储值方案购买记录、统计、赠券、升级、积分
```

## 11. 当前设计边界与风险

### 11.1 主余额事务相对安全

主余额入账具备以下保护：

- Redis 锁防止同一 POS 单短时间重复提交。
- `crm_deal_task.out_trade_no = orderId` 做幂等识别。
- POS `saveAmount` 与 CRM 档位本金做防篡改校验。
- POS 成功返回前会检查 `crm_deal_task.trade_state=SUCCESS`。
- POS 本地最终金额和余额以 CRM 返回 `CardBalanceVo` 为准回写。

### 11.2 后置权益不是强事务

赠券、升级、积分通过 `CompletableFuture.runAsync` 异步执行：

- 优点：赠券异常不影响余额入账。
- 风险：余额已成功但赠券失败时，不会自动回滚余额。
- 周期发券有 `crm_deposit_coupon_schedule` 记录和定时任务，可继续扫描。
- 一次性发券失败主要依赖日志排查，是否有补偿入口需要继续结合券服务确认。

### 11.3 撤销赠券闭环需要补充确认

当前储值方案撤销入口没有显式调用“取消储值方案赠券”的逻辑。若业务要求撤销储值时必须收回赠券，需要补充设计：

- 建立赠券记录与 `taskLid / chargeRecordLid` 的明确关系。
- 撤销时先判断券是否已使用。
- 未使用券可取消。
- 已使用券需要定义阻断撤销、扣回等价金额，或走人工处理。
- 周期发券计划需要暂停或关闭。
