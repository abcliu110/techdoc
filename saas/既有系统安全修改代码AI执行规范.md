# 既有系统安全修改代码 AI 执行规范

## 1. 目标

本规范用于指导 AI 在维护既有系统、老系统、历史业务系统时安全修改代码。

最高优先级目标：

> 保留老业务逻辑。老逻辑无论看起来是否正确，只要线上或历史数据可能依赖它，AI 都必须先把它当成兼容契约，而不是当成错误主动修正。

AI 修改代码时，不能以“代码看起来不合理”“可以顺手优化”“我认为这是 bug”为理由直接改变旧行为。

### 1.1 术语定义

**老逻辑（Legacy Logic）**：指系统当前运行的、尚未通过明确废弃流程宣布废止的代码行为。包括但不限于：

- 已有线上数据依赖的业务分支。
- 被多个调用方引用的公共方法。
- 接口返回中已经存在的字段，无论当前代码中是否能找到调用方。
- 处理空值、默认值、特殊枚举值、历史状态值的分支。
- 定时任务、消息消费、同步任务、报表、导入导出等非接口入口中的历史行为。

**疑似冗余逻辑**：指代码中看起来不再使用，但无法证实没有实际依赖的逻辑。疑似冗余逻辑不等于可删除逻辑；在没有确认前，AI 必须保留它。

**明确错误（Confirmed Bug）**：指同时满足以下条件的旧行为：

1. 有明确技术错误证据，不只是“看起来不合理”。
2. 业务方、技术负责人或需求上下文明确允许修改。
3. 不存在依赖该行为的历史数据或调用方，或者依赖方确认可以接受变更。

以下情况不属于明确错误：

- 命名、代码风格、架构方式不符合 AI 偏好。
- AI 认为新逻辑更合理。
- 旧逻辑与新需求方向不一致。
- 当前文档没有记录该历史规则。

## 2. 适用范围

适用于以下场景：

- 老系统功能增强。
- 历史接口维护。
- 已上线业务流程修改。
- 兼容老数据的新需求开发。
- 订单、支付、计费、权限、审批、库存、会员、营销、同步任务等高风险业务代码修改。
- 无完整测试覆盖、业务规则沉淀在代码中的系统。

不适用于以下场景：

- 明确要求废弃旧逻辑的重构任务。
- 明确确认旧行为是错误且允许修复的 bug。
- 全新模块且无历史兼容要求的开发。

## 3. AI 总原则

AI 必须遵守以下原则：

1. **老逻辑优先保留**
   - 老代码当前行为就是第一版事实。
   - 不要先判断它是否合理。
   - 不要主动“修正”历史分支。

2. **新逻辑受控进入**
   - 新逻辑必须通过明确边界接入。
   - 优先使用包装、分支、开关、扩展点。
   - 不允许把新规则随意插入旧逻辑深处。

3. **先保护，再修改**
   - 修改前必须尽量补充特征化测试或可重复验证用例。
   - 测试目标是锁定旧行为，不是证明旧行为正确。

4. **小范围、可回退**
   - 每次修改只解决一个明确目标。
   - 不混入无关格式化、重命名、大重构。
   - 代码必须便于回滚、关闭或隔离。

5. **证据优先**
   - 修改前要确认入口、调用方、数据形态、配置影响。
   - 不确定时通过搜索、日志、测试、数据库样例、接口样例获取证据。

6. **等价重写必须受保护**
   - AI 不得以“行为不变”为理由重写旧逻辑。
   - 循环改 Stream、手写判空改 `Optional`、整数计算改 `BigDecimal`、异常处理重排，都可能改变边界行为。
   - 涉及金额、状态、权限、库存、数据迁移、同步任务等核心业务时，等价重写必须独立于功能修改，并有特征化测试或详细验证结果。

## 3.5 快速参考（每次修改前必看）

修改前：

```text
[ ] 已检查 git status，无冲突修改。
[ ] 已识别旧逻辑入口和调用方。
[ ] 已确认涉及的文件、接口、数据库字段、配置、定时任务、消息或同步任务。
[ ] 已选择包装、开关、并行验证、扩展点或版本化接口之一。
[ ] 核心业务已补充特征化测试或手工验证用例。
[ ] 不确定的依赖项已列为未确认项并说明。
[ ] 回滚方案已明确。
```

修改时禁止：

```text
[禁止] 因为“看起来不合理”而改旧逻辑。
[禁止] 以“行为等价”为由无保护重写旧代码。
[禁止] 默认开启新逻辑。
[禁止] 删除无确认的疑似冗余逻辑。
[禁止] 改旧字段、枚举 code、错误码、配置默认值。
```

修改后必须交付：

```text
[交付] 保留了哪些老逻辑。
[交付] 新逻辑的启用条件，包含开关配置。
[交付] 数据库、配置、接口、消息的兼容措施。
[交付] 验证结果，包含未覆盖风险。
[交付] 回滚方案。
[交付] 需要人类确认的未决项。
```

## 3.6 Java 文件编码规则

- `.java` 文件必须保存为 `UTF-8` 无 BOM；即使通用文本文件规则要求保留原编码或新文件使用 `utf-8-sig`，Java 源码也必须例外处理为 UTF-8 无 BOM。

## 4. 强制执行流程

AI 在修改既有系统代码前，必须按以下顺序执行。

### 4.0 修改前环境检查

AI 修改代码前应先确认当前工作环境：

- 查看当前未提交变更，避免覆盖人类或其他 AI 已做的工作。
- 查看近期相关提交或文档，确认是否已经有人处理过同一问题。
- 如果存在并行修改风险，必须缩小修改范围，或者在最终说明中明确冲突风险和未确认项。

推荐命令：

```text
git status
git log --oneline -10
```

### 4.1 确认旧行为

必须先回答：

- 当前入口在哪里？
- 当前老逻辑如何分支？
- 哪些接口、任务、页面、服务调用了它？
- 当前输入输出是什么？
- 是否存在历史数据、特殊枚举、空值、默认值、兼容字段？
- 新需求是否只影响一部分场景？

AI 不得在没有理解调用链和旧行为的情况下直接修改核心逻辑。

### 4.1.1 不确定时的处理

如果通过代码搜索、日志分析、数据库样例、接口测试等方式仍无法确认：

- 某段逻辑是否被调用方依赖。
- 某个分支是否处理历史数据。
- 某个字段是否被前端、第三方、报表、导出、定时任务或其他服务使用。
- 某个配置、枚举、状态值是否有历史含义。

AI 必须按以下方式处理：

1. 将该问题标记为“未确认项”。
2. 说明已经尝试过哪些验证手段。
3. 不得基于不确定假设直接改变旧行为。
4. 可以继续提供保守方案：保留旧逻辑、新增分支、默认关闭开关、补充注释。
5. 如果必须改变旧行为，必须等待业务方、技术负责人或用户明确确认。

未确认项示例：

```text
未确认项：订单 status=3 且 payTime 为空的历史数据是否仍存在。
已验证：搜索了服务代码、接口调用、测试用例，未找到完整证据。
保守处理：保留旧展示逻辑，新规则仅在 feature.xxx.enabled=true 时启用。
需要确认：是否允许改变该类历史订单的展示状态。
```

### 4.2 增加特征化测试或验证用例

AI 必须优先添加或补充特征化测试；如果无法自动化测试，必须形成可重复的手工验证用例。

测试重点：

- 保留老输入对应的老输出。
- 保留老状态流转。
- 保留老字段兼容。
- 保留老异常或默认行为。
- 覆盖新逻辑的启用条件。
- 覆盖新逻辑不启用时仍走旧逻辑。

测试命名建议：

```text
should_keep_legacy_behavior_when_xxx
should_use_legacy_result_when_feature_disabled
should_not_change_legacy_status_mapping
should_apply_new_rule_only_when_xxx
```

如果项目不适合立即写自动化测试，AI 必须至少形成可重复的手工验证用例：

```text
场景：
前置数据：
请求参数：
旧逻辑预期：
新逻辑预期：
验证命令或接口：
```

### 4.3 选择安全接入方案

优先级从高到低：

1. 包装旧逻辑。
2. 新旧分支隔离。
3. 功能开关控制。
4. 并行验证（影子模式，Shadow Mode）。
5. 扩展点或策略类。
6. 新增兼容接口或版本化接口。

禁止优先选择：

- 直接重写旧方法。
- 在旧方法中插入大量新条件。
- 顺手整理历史代码。
- 改旧字段含义。
- 改旧接口返回语义。

### 4.4 回滚方案确认

每次修改前必须明确回滚方案。回滚方案不要求复杂，但必须能回答：

- 回滚粒度是什么：关闭配置、关闭开关、恢复文件、回滚提交，还是回滚版本？
- 旧版本代码是否在 git 中可追溯？
- 功能开关关闭后，新写入的数据是否会造成旧逻辑无法处理？
- 数据库或配置变更是否有反向脚本、备份或兼容读取路径？
- 回滚后，定时任务、消息消费、同步任务重跑是否会意外触发新逻辑？

回滚方案必须作为最终交付说明的一部分。

### 4.4 修改后验证

修改后必须验证：

- 老场景仍然走老逻辑。
- 新场景只在明确条件下生效。
- 功能开关关闭时，新逻辑不影响旧业务。
- 老接口字段、枚举、错误码、状态码保持兼容。
- 历史特殊逻辑有注释说明。

最终交付时，AI 必须说明：

```text
保留了哪些老逻辑：
新增了哪些新逻辑：
新逻辑启用条件：
是否有功能开关：
老接口是否兼容：
测试或验证结果：
仍存在的风险：
回滚方案：
需要人类确认的未决项：
```

## 5. 方案：特征化测试

### 5.1 定义

特征化测试是为了记录系统当前行为的测试。

它不判断老逻辑是否业务正确，只确认：

> 代码修改后，旧输入仍然得到旧结果。

### 5.2 AI 必须做什么

AI 必须优先为以下内容补特征化测试：

- 历史状态判断。
- 金额、折扣、积分、库存计算。
- 权限判断。
- 审批流转。
- 定时任务筛选条件。
- 老接口字段兼容。
- 空值、默认值、特殊枚举值处理。
- 历史数据兼容分支。

### 5.3 测试写法要求

测试中应明确写出“保持历史行为”的目的。

示例：

```java
@Test
void shouldKeepLegacyPendingStatusWhenPayTimeIsNull() {
  // 保持历史兼容：旧订单 status=3 且 payTime 为空时，仍按待确认处理。
}
```

### 5.4 无法写测试时的替代方案

如果无法写自动化测试，AI 必须提供手工验证用例。

示例：

```text
验证场景：旧订单待确认状态兼容
前置数据：订单 status=3，payTime=null
操作：调用订单详情接口
预期：返回展示状态为待确认
目的：确认新逻辑未改变旧订单展示行为
```

### 5.5 测试和验证分级

AI 不得简单用“项目不好测”“现有代码没有测试框架”作为跳过验证的理由，应按风险分级处理。

#### 5.5.1 核心业务逻辑

核心业务包括：

- 订单金额、支付状态、退款、结算、计费。
- 库存扣减、积分、权益、优惠、会员等级。
- 权限、审批、状态流转。
- 数据同步、消息消费、定时任务、批量处理。
- 数据迁移、历史数据兼容。

要求：

- 优先补自动化测试。
- 无法自动化时，必须写清楚可重复的手工验证用例和验证结果。
- 无法验证时，只允许采用保守改法：保留旧逻辑、新增分支、默认关闭开关、小范围启用。
- 无法验证且必须改变旧行为时，必须列为未确认项，等待明确确认。

#### 5.5.2 辅助业务逻辑

辅助业务包括：

- 日志、通知、统计、查询筛选、导出字段、展示字段补充。

要求：

- 优先自动化测试。
- 可以接受手工验证用例。
- 必须确认不会改变旧接口结构和历史字段语义。

#### 5.5.3 低风险变更

低风险变更包括：

- 文档、注释、非行为性说明补充、局部命名补充。

要求：

- 至少通过代码审查确认无副作用。
- 不得混入业务行为修改。

### 5.6 跨服务场景

如果修改涉及 RPC、HTTP、MQ、定时任务、数据同步、网关、第三方回调等跨服务链路，AI 必须单独确认影响范围。

必须检查：

- 上游服务是否仍按旧请求格式调用。
- 下游服务是否能兼容新增字段、缺省字段、旧枚举 code。
- MQ 消息字段是否向下兼容，消费者是否能处理新旧消息。
- 定时任务修改后是否会重跑历史分区数据。
- 同步任务是否会重新触发，是否会重复写入或覆盖旧数据。
- 其他服务是否会和本服务同步发版；如果不能同步发版，必须保持版本兼容。

跨服务变更有独立影响范围，必须单独验证。AI 不得假设其他服务会同步更新。

## 6. 方案：包装旧逻辑

### 6.1 定义

包装旧逻辑是指不直接重写原有核心代码，而是在外层增加路由、适配或分支。

推荐结构：

```java
public Result handle(Request request) {
  if (newRuleEnabled(request)) {
    return newHandler.handle(request);
  }

  return legacyHandler.handle(request);
}
```

### 6.2 适用场景

适用于：

- 老方法复杂且缺少测试。
- 老逻辑被多个调用方复用。
- 新需求只影响部分客户、部分租户、部分状态或部分来源。
- 业务不能确认所有历史分支。

### 6.3 AI 必须遵守

- 不要在旧方法内部大量插入新逻辑。
- 不要删除旧分支。
- 不要改变旧方法默认返回。
- 新逻辑必须有明确启用条件。
- 旧逻辑入口必须仍然可被调用。

### 6.4 推荐命名

```text
LegacyXxxService
LegacyXxxHandler
XxxCompatService
NewXxxHandler
XxxRoutingService
```

命名必须让后续维护者能看出哪部分是历史兼容逻辑，哪部分是新逻辑。

## 7. 方案：功能开关

### 7.1 定义

功能开关用于控制新逻辑是否生效。

目标：

- 默认不影响老业务。
- 可以灰度启用。
- 出问题可以快速关闭。

### 7.2 适用场景

必须优先考虑功能开关的场景：

- 计费、支付、订单、权限、库存、审批等核心链路。
- 影响多个客户或租户。
- 线上数据复杂，无法一次覆盖所有场景。
- 新逻辑需要逐步验证。
- 回滚成本高。

### 7.3 AI 实现要求

功能开关关闭时：

- 必须完全走旧逻辑。
- 不得改变旧返回。
- 不得写入新字段导致旧流程异常。

功能开关开启时：

- 只影响明确范围。
- 必须有日志或可观测信息。
- 必须可以定位命中新逻辑的请求或数据。

### 7.4 配置命名建议

```text
feature.xxx.enabled
feature.xxx.tenantIds
feature.xxx.customerScope
feature.xxx.grayPercent
```

不要使用含义模糊的名称：

```text
new.switch
flag1
test.enabled
```

### 7.5 功能开关生命周期

功能开关不是永久垃圾代码入口，必须有生命周期说明。

上线建议：

1. 默认关闭，先全量走旧逻辑。
2. 测试环境验证开关开启后的行为。
3. 预发或灰度环境使用接近真实的数据验证。
4. 线上按租户、客户、门店、用户、渠道或比例小范围开启。
5. 稳定后再扩大范围。

稳定后处理：

- 不要在开关刚全量开启后立即删除旧逻辑。
- 删除旧路径前，必须确认无调用方、无历史数据依赖，并且有可回滚方案。
- 开关必须有可审计记录、默认值说明、启用范围和回滚方案。
- 开关配置可以在配置中心、数据库或配置文件中，但必须能追踪谁在何时因为什么启用或关闭。

禁止：

- 开关默认全量开启。
- 开关开启后立即删除旧逻辑。
- 多个开关控制同一逻辑但没有优先级说明。
- 只在本地代码中知道开关含义，配置处没有说明。

## 8. 方案：扩展点和策略类

### 8.1 定义

当新规则和旧规则并存，且未来可能继续增加规则时，可以使用扩展点或策略类。

推荐结构：

```text
XxxRule
LegacyXxxRule
TenantXxxRule
NewBusinessXxxRule
```

### 8.2 适用场景

适用于：

- 不同客户、租户、渠道、业务类型有不同规则。
- 后续预计会增加更多规则。
- 现有代码已经出现多处重复条件判断。
- 旧逻辑需要保留为一个独立策略。

### 8.3 AI 必须避免

- 只有一个新分支时过度抽象。
- 为了显得设计复杂而引入多层接口。
- 把所有逻辑都拆碎，导致调用链难以追踪。
- 把历史兼容逻辑隐藏到看不出语义的通用类里。

### 8.4 推荐判断

AI 可以使用以下判断：

```text
如果只是一个小条件：优先简单分支。
如果已有多个业务规则：可以策略化。
如果未来明确会继续增加规则：可以抽扩展点。
如果旧逻辑风险极高：先包装，再逐步策略化。
```

### 8.5 抽象程度判断标准

以下标准是辅助判断，不是机械硬规则。AI 应结合代码规模、风险、已有项目风格决定。

推荐方式：

```text
仅有一个新条件，且未来不会变化：简单 if/else。
有 2-3 个规则，且未来可能增加：可以使用策略类。
有 3 个以上规则，且按租户、客户、渠道、业务类型分流：可以使用扩展点或注册机制。
旧逻辑风险极高，无法一次性验证：包装旧逻辑 + 功能开关 + 灰度。
```

过度抽象信号：

- 新增多层接口或抽象类，但实际只有 1 个实现。
- 为了策略模式而创建空壳 `LegacyXxxStrategy`。
- 新增抽象层后，理解一次调用需要追踪多个无业务含义的文件。
- 新抽象比原始业务逻辑更难理解。

过度简单信号：

- 同一个方法中出现大量业务分支，且不同分支属于不同客户、租户、渠道或业务类型。
- 相同判断在多个位置重复。
- 功能开关判断散落在多个方法中，没有统一入口。
- 新旧逻辑混杂在一起，无法判断哪个分支保留历史行为。

## 9. 方案：并行验证（影子模式，Shadow Mode）

### 9.1 定义

并行验证是指新旧逻辑同时执行，对比结果后再决定是否切换。它的核心原则是：

> 运行新逻辑，但不让新逻辑直接影响线上结果。

不确定性高、结果可对比、影响核心业务时，AI 必须优先考虑并行验证。

### 9.2 适用场景

适用于：

- 新旧逻辑结果需要精确对比，例如金额、折扣、积分、库存、权益、计费。
- 无法通过代码审查确认所有分支等价。
- 灰度前需要使用真实数据验证。
- 新算法、新规则、新状态判断要替代旧逻辑。
- 旧逻辑复杂、缺少测试，但又不能直接重写。

### 9.3 实现要点

推荐流程：

```text
旧逻辑执行 -> 结果用于真实返回或真实写入
新逻辑执行 -> 只用于对比，不影响线上结果
差异记录 -> 日志、告警、差异表或审计记录
人工确认 -> 积累足够样本后判断是否可切换
逐步切换 -> 开关灰度，保留一段时间差异日志
```

代码形态示例：

```java
LegacyResult legacyResult = legacyCalculator.calculate(request);
NewResult shadowResult = newCalculator.calculate(request);

if (!sameBusinessResult(legacyResult, shadowResult)) {
  log.warn("shadow diff, bizId={}, legacy={}, shadow={}",
      request.getBizId(), legacyResult, shadowResult);
}

return legacyResult;
```

### 9.4 AI 必须遵守

- 并行验证阶段必须只返回旧逻辑结果作为线上响应。
- 差异必须记录，且能定位业务主键、输入条件、新旧结果。
- 差异日志不得泄露敏感信息。
- 必须设置观察周期或样本量目标。
- 切换前必须有人确认差异是否可接受。
- 切换后应保留一段时间差异日志用于回溯。

### 9.5 禁止事项

- 长期双跑但不收敛结论。
- 差异时不告警、不记录。
- 新逻辑在影子模式（Shadow Mode）下写入真实业务字段。
- 对有副作用的逻辑直接双写，例如重复扣库存、重复发券、重复发消息。
- 对无法幂等的操作并行执行，除非新逻辑只做模拟计算或写隔离审计表。

## 10. 方案：接口向后兼容

### 10.1 基本要求

AI 修改既有接口时，必须保持向后兼容。

禁止随意修改：

- 字段名。
- 字段类型。
- 枚举 code。
- 状态码。
- 错误码。
- 默认值。
- 空值语义。
- 排序规则。
- 分页结构。
- 返回层级。

### 10.2 推荐做法

优先：

- 新增字段，不删除旧字段。
- 新增接口，不破坏旧接口。
- 新增版本，不直接改老版本。
- 老字段继续返回。
- 老枚举 code 保持不变。
- 新字段允许老调用方忽略。

版本化示例：

```text
/api/order/detail      保持旧逻辑
/api/order/detail/v2   新逻辑
```

### 10.3 AI 必须检查

修改接口前必须检查：

- 前端是否使用该字段。
- 其他后端服务是否调用该接口。
- 第三方系统是否依赖该接口。
- 文档、测试、示例请求是否需要同步。
- 是否存在历史客户端无法升级。

如果无法确认所有调用方，AI 应默认接口有隐藏依赖。

## 11. 方案：数据库变更兼容

### 11.1 基本要求

数据库变更必须比代码变更更保守。字段、枚举 code、默认值、索引、迁移脚本都可能影响旧业务。

AI 必须优先采用“扩展、兼容、观察、再收缩”的方式：

```text
第一步：新增兼容结构，不删除旧结构。
第二步：代码同时兼容新旧字段或新旧数据。
第三步：回填或迁移历史数据。
第四步：观察线上运行和日志。
第五步：确认无依赖后，再考虑删除旧字段或旧逻辑。
```

### 11.2 禁止事项

AI 禁止：

- 修改已有枚举 code 的含义，或复用旧 code 表示新业务含义。
- 新增 `NOT NULL` 字段但不给默认值，也不确认所有写入路径。
- 删除旧字段前不确认代码、SQL、报表、导出、第三方同步是否引用。
- 修改字段单位或语义，例如金额单位、时间单位、比例含义。
- 直接修改历史数据含义来适配新代码。
- 不写迁移脚本或不说明迁移顺序。

数据库变更还必须遵守第 15 条的全局禁止事项。

### 11.3 推荐做法

新增字段：

- 优先 nullable。
- 如必须 `NOT NULL`，必须提供默认值，并确认历史数据和所有写入路径。
- 新旧字段并存期间，读取逻辑必须明确优先级。

枚举字段：

- 只扩增，不修改旧 code。
- 新 code 必须有注释和文档说明。
- 服务逻辑应比较枚举常量，不使用魔法数字。

删除字段：

- 先确认代码中无引用。
- 再确认 SQL、报表、导出、定时任务、同步任务、第三方接口无引用。
- 删除前必须确认无依赖，并且有回滚方案或备份方案。

### 11.4 数据库变更检查清单

```text
[ ] 新增字段是否 nullable 或已有默认值？
[ ] 历史数据是否满足新约束？
[ ] 所有写入路径是否已经兼容？
[ ] 枚举字段是否只新增 code，没有修改旧 code？
[ ] 是否确认 SQL、报表、导出、同步任务无旧字段依赖？
[ ] 是否有迁移脚本、回滚方案或备份方案？
[ ] 新旧字段并存时，读取优先级是否明确？
```

## 12. 方案：配置变更兼容

### 12.1 基本要求

配置变更也属于业务行为变更。Nacos、Apollo、环境变量、配置文件、数据库配置项的默认值变化，都可能影响旧业务。

AI 修改配置时必须确认：

- 配置 key 是否已有线上使用。
- 默认值是否会改变旧行为。
- 所有服务实例是否可能在滚动发布期间同时看到不同配置。
- 是否有回滚方式。
- 配置单位、含义、作用范围是否保持不变。

### 12.2 禁止事项

AI 禁止：

- 删除旧配置 key 但不确认引用。
- 改变旧配置默认值但不说明影响。
- 改变配置单位，例如秒改毫秒、元改分。
- 假设所有服务实例会同时加载新配置。
- 修改配置含义但沿用旧名称。
- 配置变更没有回滚方案。

### 12.3 推荐做法

- 新增配置 key 优先于修改旧 key 含义。
- 旧 key 保持旧语义。
- 配置说明必须写清楚默认值、启用范围、回滚方式。
- 多个配置共同控制一个逻辑时，必须说明优先级。
- 对核心业务配置，默认值应保持旧行为。

## 13. 方案：区分 bug 修复和业务变更

### 13.1 基本判断

既有系统中，很多看似 bug 的逻辑可能是历史业务规则。

AI 必须先判断：

```text
这是明确技术错误？
这是历史业务兼容？
这是特定客户规则？
这是历史数据修补逻辑？
这是旧前端依赖的返回格式？
```

### 13.2 AI 不得擅自认定

AI 不得仅凭以下理由修改旧逻辑：

- 命名不合理。
- 分支看起来多余。
- 状态值不符合常识。
- 空值处理看起来奇怪。
- 老接口返回重复字段。
- 老代码风格不统一。
- 旧规则和新需求冲突。
- 注释可能过时。
- 某个字段赋值后当前代码中未读取。
- 当前业务文档中没有该规则。
- 业务方口头说功能已废弃，但没有正式废弃流程。
- 日志中暂时没有发现该路径执行。
- 没发现接口调用记录。

以上列举的是典型不可靠证据。AI 也不应反过来把所有无法证明有依赖的情况都当成“安全可改”的理由；无法确认依赖时，默认视为有依赖，按未确认项处理。

### 13.3 推荐表达

如果要保留历史行为，交付说明或注释应使用：

```text
保留历史兼容行为
兼容旧接口调用方
兼容历史数据
新逻辑仅在 xxx 条件下启用
```

避免使用：

```text
修复错误逻辑
清理无用分支
优化旧代码
修正历史问题
```

除非业务已经明确确认旧行为可以改变。

## 14. 方案：历史兼容注释

### 14.1 什么时候必须写注释

以下情况必须写注释：

- 保留了看起来不合理的旧逻辑。
- 新逻辑必须绕开某个历史分支。
- 某个字段为空、特殊枚举、特殊状态有历史含义。
- 不能改变某个返回字段或错误码。
- 某个客户、租户、历史数据依赖旧行为。

### 14.2 注释必须包含什么

注释应包含：

- 为什么保留。
- 影响哪些数据或调用方。
- 不要随意修改什么。
- 如果知道，写清楚历史原因或业务方确认信息。

### 14.3 注释模板

```java
// 保留历史兼容：<说明旧行为>。
// 影响范围：<历史数据/调用方/客户/接口>。
// 修改风险：不要改为 <看似更合理的新行为>，否则会影响 <具体后果>。
```

示例：

```java
// 保留历史兼容：老订单 status=3 且 payTime 为空时，前端仍按“待确认”展示。
// 影响范围：2022 年前导入订单和旧版 POS 客户端。
// 修改风险：不要改为标准支付状态判断，否则旧订单详情展示会变化。
```

## 15. 禁止事项

AI 在维护既有系统时禁止：

- 未加验证就直接修改核心业务逻辑。
- 因为旧逻辑“不合理”而主动修正。
- 把新需求和大重构混在一起。
- 删除历史分支但没有证据证明无人依赖。
- 修改老接口字段含义。
- 修改老枚举 code 含义。
- 修改老错误码或状态码含义，除非业务方或技术负责人已书面确认。
- 把新逻辑默认全量开启。
- 在旧逻辑中堆叠大量无注释的新条件。
- 将“兼容历史行为”的修改描述成普通 bug 修复。
- 以“行为等价”为理由重写旧逻辑，但没有特征化测试或验证证据。
- 修改数据库字段、枚举 code、配置默认值，但不说明兼容影响。
- 删除疑似冗余逻辑，但没有确认无调用方、无历史数据、无配置或脚本依赖。
- 忽略未确认项，假设“应该没人在用”。

## 16. AI 修改代码前检查清单

AI 在动代码前必须自检：

```text
[ ] 我已经查看当前工作区状态，避免覆盖他人或其他 AI 的修改。
[ ] 我已经理解旧逻辑当前行为。
[ ] 我已经确认入口和主要调用方。
[ ] 我已经识别新逻辑的启用条件。
[ ] 我没有打算主动修正未确认的历史行为。
[ ] 我已经考虑特征化测试或手工验证用例。
[ ] 我已经选择包装、开关、并行验证、扩展点或兼容接口方案。
[ ] 我知道哪些接口字段、枚举、状态不能改。
[ ] 我知道是否涉及数据库字段、配置项、定时任务或同步任务。
[ ] 我知道哪些地方需要写历史兼容注释。
[ ] 我已经列出仍无法确认但可能影响旧业务的未确认项。
[ ] 我已经明确回滚方案。
```

## 17. AI 修改代码后检查清单

AI 完成修改后必须自检：

```text
[ ] 老场景仍然走旧逻辑。
[ ] 新场景只在明确条件下走新逻辑。
[ ] 功能开关关闭时行为不变。
[ ] 老接口响应结构保持兼容。
[ ] 老枚举 code、错误码、状态码未被修改。
[ ] 数据库字段、配置默认值、定时任务行为没有被无意改变。
[ ] 历史兼容逻辑已有注释。
[ ] 已执行相关测试或手工验证。
[ ] 回滚方案已写入最终说明。
[ ] 最终说明中列出了保留的老逻辑、剩余风险和未确认项。
```

## 18. 推荐交付说明模板

AI 完成任务后，最终回复应包含：

```text
变更目标：

保留的老逻辑：

新增的逻辑：

新逻辑启用条件：

兼容措施：

测试或验证：

回滚方案：

未覆盖风险（AI 已识别但尚未覆盖的风险）：

需要人类确认的未决项：
```

## 19. 决策规则

当 AI 不确定是否可以改变旧行为时，按以下优先级处理：

```text
P0：能保留旧行为，就保留旧行为。
P0：不确定项必须列出，不能自行决定改变旧行为。
P1：能新增分支，就不要改旧分支。
P1：能加开关，就不要默认全量启用。
P1：能新增字段，就不要改旧字段。
P1：能新增接口版本，就不要破坏老接口。
P2：核心业务无法确认等价时，优先并行验证。
P2（备选）：无法做并行验证时，只允许保守改法。
P3：必须保留但看起来奇怪的逻辑，要写注释。
```

## 20. 一句话原则

维护既有系统时，AI 的任务不是证明旧逻辑正确，而是确保新修改不会无意改变旧业务。

## 21. 适用范围声明

本规范约束 AI 在既有系统上执行的所有修改行为，包括但不限于：

- 直接修改代码。
- 生成或修改测试代码。
- 建议删除、移动或重命名代码。
- 代码评审（Review）中指出问题或提出修改建议。

也就是说，只要 AI 在既有代码库中做任意形式的修改建议，都必须遵守本规范。AI 不得以”这是 Review 不是修改”为由绕过本规范提出的流程和要求。

### 21.1 代码评审要求

在代码评审（Review）场景下，AI 必须主动检查候选修改是否违反本规范，并按以下方式处理：

**必须检查的内容：**

- 是否在未做特征化测试的情况下修改了核心业务逻辑。
- 是否删除了历史分支但没有证据证明无人依赖。
- 是否修改了老接口字段、枚举 code、错误码、状态码的含义。
- 是否有疑似冗余逻辑被删除但未经过依赖确认。
- 是否有新逻辑默认全量开启。
- 是否有”兼容历史行为”的修改被描述为普通 bug 修复或优化。
- 是否在旧逻辑中堆叠大量新条件但没有历史兼容注释。
- 是否有等价重写但缺少特征化测试保护。

**检查结果表达：**

如果发现疑似违规，AI 应在 Review 中明确说明：

```text
[规范检查] 疑似违反：<描述候选修改做了什么>
违反原因：<对应本规范哪一条>
建议：<保留旧行为 / 增加开关 / 补充测试 / 补充注释>
```

如果确认无误，AI 应在 Review 中说明：

```text
[规范检查] 本次修改已识别并保留了以下老逻辑：
<列出保留的老逻辑>
本次修改的新逻辑启用条件：
<列出启用条件>
```

AI 不得在 Review 中只说”代码风格问题””建议优化”而不说明是否影响老业务逻辑。
