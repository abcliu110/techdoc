# RocketMQ 在这个系统里到底是干什么的

## 先说一个最简单的比喻

**不用 RocketMQ 的问题：**

```
收银后台：我要给门店 POS 发一条打印指令
系统：好，现在发给 POS 终端... 咦，终端没联网，报错了
```

**用了 RocketMQ 的改进：**

```
收银后台：我要给门店 POS 发一条打印指令
系统：好，我先把指令放到一个"排队区"（RocketMQ），然后返回"发送成功"
       （门店 POS 什么时候联网，什么时候来排队区拿指令）
```

RocketMQ 本质上就是一个**可靠的排队区**——发消息的不需要等对方在线，收消息的不需要找发消息的，两边解耦了。

---

## 在这个系统里，具体是怎么用的

一句话：**后台想给 POS 终端发消息，但不知道终端在不在线，所以先把消息扔到 RocketMQ 里，等 POS 联网了再来拿。**

完整流程：

```
1. 后台操作员点击"打印"
2. 系统把打印指令写入 RocketMQ（立刻返回"成功"）
3. POS 终端联网后，来 RocketMQ 拉这条指令
4. POS 收到指令，开始打印
```

在这个系统里，RocketMQ 主要充当两个角色：

### 角色一：消息中转站

后台服务把消息扔进去，nms4cloud-mq 服务从里面取出来，转发给 POS。

```
后台服务 → RocketMQ → nms4cloud-mq → Netty/MQTT → POS 终端
```

### 角色二：定时闹钟（延迟消息）

用来检查"消息有没有被收到"。

```
发消息 → 等10秒 → 检查 Redis → 收到了？→ 不管
                    ↓
              没收到？→ 重新发
```

"等10秒后检查"这个动作，就是靠 RocketMQ 的延迟消息实现的。

---

## RocketMQ 的四个最基本概念（只需要记住这四个）

| 概念 | 相当于 | 在这个系统里 |
|------|--------|-------------|
| **Topic** | 排队区的名字 | `MQTT_SEND_MSG` = 要发给终端的消息 |
| **Producer** | 往排队区放东西的人 | 后台服务（收银后台） |
| **Consumer** | 从排队区取东西的人 | nms4cloud-mq 里的 SendMqttMsgConsumer |
| **Broker** | 排队区本身 | RocketMQ 服务器 |

就这么简单。

---

## 这个系统的两条实际链路

### 链路一：后台 → POS 推送（最常见）

```
收银后台
   │  发送打印指令
   ▼
RocketMQ（Topic: MQTT_SEND_MSG）
   │  被消费
   ▼
SendMqttMsgConsumer（nms4cloud-mq）
   │  根据终端 ID 查路由
   ▼
Netty 长连接 ──→ POS 终端（打印机出纸）
```

### 链路二：消息确认 + 重试（保证可靠送达）

```
发消息到 POS 终端
   │
   ├─ 同时写入 Redis（记录"这条消息还没被确认"）
   │
   └─ 发一条延迟消息到 RocketMQ（10秒后再检查）
          │
          ▼
     10秒后，检查 Redis
          │
          ├─ 消息已被确认 → 不用管
          │
          └─ 消息还在 Redis → POS 没收到 → 重新发一次
```

---

## 为什么不能没有 RocketMQ？

如果没有 RocketMQ，会有这些问题：

| 没有 RocketMQ | 有了 RocketMQ |
|--------------|--------------|
| 终端不在线时，后台会报错 | 后台直接返回"成功"，终端来了再推 |
| 后台要一直等着终端响应 | 后台不等待，干下一件事 |
| 高峰期消息太多，终端处理不过来 | 消息在 RocketMQ 里排队，慢慢处理 |
| 想重试？没地方记"发了没收到" | RocketMQ 延迟消息来触发重试 |

---

## 一句话总结

**RocketMQ = 一个可靠的排队区。**

后台把消息扔进去，不用等终端在线。
终端上线了，来排队区拿。
10秒后还没拿，就再发一次。

就这么简单。

# 深入技术原理

下面把之前没讲清楚的技术细节逐一说透。

---

## 一、MQTT 的 QoS 到底是怎么回事

### 1.1 三个 QoS 级别是什么意思

MQTT 是设备（POS 终端）和服务器之间常用的通信协议。QoS（Quality of Service）是它的消息可靠等级，分三级：

| 等级 | 名字 | 简单说 | 类比 |
|------|------|--------|------|
| **QoS 0** | At most once | 发一次，不管对方收到没有 | 发微信，只管发出去 |
| **QoS 1** | At least once | 发一次，对方必须回确认，没确认就重发 | 打电话，必须听到"喂"才挂 |
| **QoS 2** | Exactly once | 发一次，对方收到且只收到一次 | 挂号信，有回执，最可靠 |

### 1.2 QoS 0 的握手过程（最简单）

```
POS 终端                          MQTT Broker
    │                                 │
    │◄─────── CONNECT（建立连接）───────►│
    │                                 │
    │◄─────── CONNACK（确认连接）───────►│
    │                                 │
    │◄──── PUBLISH(topic, msg) ────────│  ← 发出去，不管回不回
    │                                 │
```

- Broker 发完 PUBLISH 就完事了
- POS 终端可能在线可能不在线，可能收到可能没收到
- **这个系统里用的就是 QoS 0**

### 1.3 QoS 1 的握手过程（发一次必须确认）

QoS 1 比 QoS 0 多了两步：发完要等 PUBACK（回执），没收到就重发。

```
POS 终端                          MQTT Broker
    │                                 │
    │◄──── PUBLISH(topic, msg) ────────│  ← 第一次发
    │                                 │
    │─────── PUBACK ─────────────────►│  ← POS 确认"收到了"
    │                                 │
    │ （如果没收到 PUBACK，Broker 会重发 PUBLISH）│
    │◄──── PUBLISH(topic, msg) ────────│  ← 重发
    │─────── PUBACK ─────────────────►│  ← 再次确认
    │                                 │
```

**注意**：QoS 1 的重发会导致一个副作用——**消息可能重复**。
Broker 没收到 PUBACK 就重发，但此时 POS 可能已经处理过了（比如打印了两张单）。

### 1.4 QoS 2 的握手过程（四步握手，最可靠）

QoS 2 保证"只收到一次"，代价是需要 4 步握手：

```
POS 终端                          MQTT Broker
    │                                 │
    │◄──── PUBLISH(topic, msg) ────────│  ← 第一步：发出去
    │─────── PUBREC ─────────────────►│  ← 第二步：POS 回"我收到了，还没处理"
    │                                 │
    │◄──── PUBREL ────────────────────│  ← 第三步：Broker 说"你可以处理了"
    │─────── PUBCOMP ────────────────►│  ← 第四步：POS 回"处理完了"
    │                                 │
```

为什么需要四步？因为要处理一个边界情况：Broker 重发 PUBLISH 时，POS 要能认出这是重复消息。

### 1.5 这个系统为什么用 QoS 0 而不是 QoS 1/2

| 方案 | 优点 | 缺点 |
|------|------|------|
| 用 QoS 1/2 | MQTT 协议层保证可靠性 | 握手慢，消息体积大，POS 端实现复杂 |
| **用 QoS 0 + 应用层自己保证** | **灵活，可定制重试策略，协议更轻量** | **需要额外实现确认机制** |

这个系统选择了**应用层保证可靠性**，原因是：
- POS 终端资源有限，MQTT 协议层越简单越好
- 应用层可以自己实现"10秒没确认就重试"这种业务逻辑
- 配合 Redis 记录消息状态，比 MQTT 协议层更可控

## 二、应用层可靠性：MQ QoS 0 + Redis + 延迟消息

### 2.1 为什么 QoS 0 不够用

MQTT QoS 0 发出去就完事了，POS 终端：
- 可能根本没联网，根本没收到
- 可能收到了但还没处理完就断电了
- 可能处理完了但回包丢了

所以必须有一套**应用层的可靠性机制**。

### 2.2 三层保证机制

这个系统用了三层保障，确保消息"发出去 + 对方收到 + 有回执"：

```
第一层：写入 Redis（记录"这条消息发出去了"）
第二层：发送 MQTT QoS 0（实际发到终端）
第三层：RocketMQ 延迟消息（10秒后检查"收到了吗"）
```

### 2.3 详细流程图

```
1. 后台点击"打印"
       │
       ▼
2. MqMqttMsgServicePlus.sendConfirmMsg()
       │
       ├─ ① 先写 Redis（ZSet + String）
       │      ZSet key: msgSet:{sid}   → 存消息 ID 列表（按时间排序）
       │      String key: msg:{lid}     → 存消息内容，TTL 1 天
       │
       ├─ ② 发 MQTT 消息到 RocketMQ
       │      Topic: MQ_SEND_MQTT_MSG
       │      → SendMqttMsgConsumer 消费
       │      → MQTT Gateway 发送到 POS 终端（QoS 0）
       │
       └─ ③ 发一条延迟消息
              Topic: MQTT_SEND_LOG
              延迟：10 秒
              → RetryMqttMsgConsumer 10 秒后取出来
                     │
                     ▼
              4. 检查 Redis
                     │
                     ├─ 消息不在 Redis 了 → POS 已确认，结束
                     │
                     └─ 消息还在 Redis → POS 没收到，执行重发
                            │
                            ▼
                      再次调用 sendConfirmMsg()（重新走这三步）
```

### 2.4 Redis 的两套数据结构

为什么 Redis 里用两套结构而不是一套？

**ZSet（有序集合）**：
```
key: msgSet:{sid}     ← sid = 门店 ID
value: 消息 ID，按 createdTime 排序
用途：快速知道"这个门店有哪些待确认消息"
```

**String（字符串）**：
```
key: msg:{lid}        ← lid = 消息的雪花算法 ID
value: 完整的消息内容（JSON）
TTL: 1 天（消息最多存 1 天）
用途：存消息正文，供重发时取用
```

为什么这样设计？
- POS 终端收到消息后，要从 Redis 里**删除**自己的待确认消息（告诉服务器"我收到了"）
- 如果只用一个 String Key，POS 怎么知道该删哪条？→ 需要一个索引 → 所以加一个 ZSet
- ZSet 天然支持"按时间排序"，方便做超时清理

### 2.5 POS 终端收到消息后怎么确认

POS 终端收到打印指令后：
```
1. 处理指令（打印小票）
2. 调用确认接口 /api/mq/pos/confirmMsg
3. 后端从 Redis 删除：
   - ZSet msgSet:{sid} 中移除该消息 ID
   - String msg:{lid} 直接过期（不再需要）
```

这样 RetryMqttMsgConsumer 10 秒后再查，消息就不在了，就知道 POS 已经收到了。

## 三、RocketMQ 延迟消息的内部原理

### 3.1 延迟消息 vs. 定时任务，有什么区别

很多人会把"延迟消息"和"定时任务"搞混：

| | 定时任务（XXL-Job） | 延迟消息（RocketMQ） |
|---|---|---|
| 触发方式 | 调度器定时执行 | 时间到了自动投递 |
| 精度 | 分钟级（Job 执行有延迟） | 秒级 |
| 扩展性 | Job 多时调度器压力大 | 消息多时 Broker 自动分配 |
| 可靠性 | Job 执行失败需要自己处理 | Broker 保证消息不丢 |
| 资源占用 | 需要部署调度服务 | 不需要 |

在这个系统里，检查"10秒后消息有没有被确认"这件事，用**延迟消息**比定时任务更合适：
- 消息量可能很大（成千上万台 POS），定时任务调度不过来
- 每条消息有各自的检查时间，定时任务不灵活
- RocketMQ 延迟消息是 Broker 本身在时间到了之后才投递，省心

### 3.2 RocketMQ 延迟消息的 10 个等级

RocketMQ 的延迟消息不是"任意时间"，而是固定 18 个等级：

| 等级 | 延迟时间 | 在这个系统里的用途 |
|------|---------|-----------------|
| 1 | 1 秒 | - |
| 2 | 5 秒 | - |
| 3 | 10 秒 | **重试检查（第一次）** |
| 4 | 30 秒 | 重试检查（第二次） |
| 5 | 1 分钟 | 重试检查（第三次） |
| 6 | 2 分钟 | - |
| 7 | 4 分钟 | - |
| ... | ... | - |
| 18 | 2 小时 | - |

这个系统里用的是**等级 3（10 秒）**。

### 3.3 延迟消息的内部流程

```
Producer（后台服务）
   │
   │ 发送消息，指定 delayLevel = 3（10秒）
   ▼
RocketMQ Broker
   │
   │ 普通消息直接写入 ConsumerQueue（消费者能立即拿到）
   │ 延迟消息写入 DelayQueue（消费者拿不到）
   │
   ▼
  Broker 内部有一个定时器（按等级分组）
   │
   │ 10 秒后，定时器触发，把消息从 DelayQueue 移回 ConsumerQueue
   ▼
Consumer（RetryMqttMsgConsumer）可以取到这条消息了
```

**关键点**：消息在延迟期间，**不在 ConsumerQueue 里**，消费者根本看不到它。只有时间到了，Broker 才把它搬出来。

### 3.4 延迟消息怎么保证不丢

RocketMQ 的延迟消息有以下保障：
- 消息在 Broker 端持久化（磁盘）
- Broker 重启后，延迟消息会从磁盘恢复继续计时
- 消费者没订阅到？不会丢，消费者下次订阅还能拿到

## 四、重试机制：指数退避

### 4.1 为什么需要指数退避

如果 POS 终端一直没联网，一直重发会怎么样？

**没有退避**：每次都 10 秒后重发，1 分钟内重发 6 次 → 浪费资源 → 可能把终端打爆

**有退避**：
```
第1次重试：10 秒后
第2次重试：30 秒后（比上次多等 20 秒）
第3次重试：1 分钟后
第4次重试：2 分钟后
...以此类推，越来越慢
```

### 4.2 这个系统的重试逻辑

根据 RocketMQ 的延迟等级：

```
发送消息（等级 3 = 10秒）
       │
       ▼
10秒后，检查 Redis
       │
       ├─ POS 确认了 → 完成 ✓
       │
       └─ 没确认 → 发延迟消息（等级 4 = 30秒）
                        │
                        ▼
                  30秒后，再次检查 Redis
                        │
                        ├─ 确认了 → 完成 ✓
                        │
                        └─ 没确认 → 发延迟消息（等级 5 = 1分钟）
                                         │
                                         ▼
                                   ... 继续下去直到成功或超时
```

实际代码中，每次 RetryMqttMsgConsumer 检查时，如果消息还在 Redis，会再次调用 `sendConfirmMsg()`，这会重新写一遍 Redis（刷新 TTL）+ 发一个新的延迟消息，所以延迟等级会逐次升高。

## 五、双通道冗余：Netty + MQTT 同时发

### 5.1 为什么有两套通道

这个系统有个特点：**同一份消息，可能同时走 Netty 通道和 MQTT 通道**。

| 通道 | 技术 | 适用场景 |
|------|------|---------|
| Netty | TCP 长连接，服务端主动推送 | 大部分 POS 终端 |
| MQTT | MQTT 协议，发布/订阅 | 网络不稳定的移动 POS |

两种终端可能连不同的通道，但最终都要能收到消息。

### 5.2 两套通道怎么选

终端注册上线时，nms4cloud-netty 会记录这个终端连的是哪个通道：
- 如果终端连的是 Netty，就用 Netty 推送
- 如果终端连的是 MQTT，就用 MQTT 推送

SendMqttMsgConsumer 消费 RocketMQ 消息后，会根据终端 ID 查路由表，决定用哪个通道发。

## 六、完整数据流总结

把上面的内容串起来，就是整个消息系统的完整数据流：

```
┌─────────────────────────────────────────────────────────────┐
│                     后台操作员点击"打印"                        │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  MqMqttMsgServicePlus.sendConfirmMsg()                       │
│  ① Redis ZSet msgSet:{sid} ← 加入消息 ID                    │
│  ② Redis String msg:{lid} ← 写入消息内容（TTL=1天）         │
│  ③ RocketMQ MQ_SEND_MQTT_MSG ← 触发实际发送                │
│  ④ RocketMQ MQTT_SEND_LOG ← 发延迟消息（10秒后检查）        │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  SendMqttMsgConsumer（RocketMQ 消费者）                       │
│  根据终端 ID 查路由 → 决定用 Netty 还是 MQTT 发送             │
│  MQTT Gateway.send(topic, 0, msg) ← QoS 0 发到 POS          │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  POS 终端收到消息 → 处理（打印小票）→ 调用 /api/mq/confirmMsg  │
│  后端删除 Redis 中的消息记录                                  │
└─────────────────────────────────────────────────────────────┘
                          ▲
                          │ （10秒后，如果 POS 没确认）
┌─────────────────────────────────────────────────────────────┐
│  RetryMqttMsgConsumer（RocketMQ 延迟消息消费者）              │
│  检查 Redis msg:{lid} 是否还在                               │
│  ├─ 不在 → POS 已确认，无事发生 ✓                            │
│  └─ 还在 → 再次调用 sendConfirmMsg() → 重发                  │
└─────────────────────────────────────────────────────────────┘
```

## 七、为什么这套设计是合理的

| 设计决策 | 原因 |
|---------|------|
| MQTT 用 QoS 0 | POS 终端资源有限，协议层越轻越好 |
| 应用层做可靠性 | 业务可定制（重试间隔、确认超时）比 MQTT 协议层更灵活 |
| Redis 记录状态 | 高速读写，支持大量 POS 并发，TTL 自动清理 |
| RocketMQ 延迟消息 | Broker 保证消息持久化，不丢；比定时任务更精确 |
| ZSet + String 两套结构 | ZSet 索引 + String 正文，分工明确 |
| 双通道（Netty + MQTT） | 兼容不同类型的 POS 终端 |

## 八、常见问题解答

**Q：为什么消息要存 1 天才过期？**
A：有些 POS 终端可能断网超过 10 分钟（超过了好几次重试间隔），但只要终端最终联网了，消息还是有效的。

**Q：为什么不用 MQ 的重试机制，而要自己写 Redis + 延迟消息？**
A：RocketMQ 自带的重试是"重新投递给消费者"，但我们需要的是"重新发到 POS"。业务逻辑不一样，所以自己实现。

**Q：10 秒是怎么来的，能改吗？**
A：可以改。改成更短，重试更及时但可能更耗资源；改成更长，资源节省但消息延迟更大。系统里写死在代码里（`delayLevel = 3`），如果业务需要可以改成可配置。

## 九、MQTT QoS 三级别的具体实现

> 本节以 Eclipse Paho MQTT 库（`org.eclipse.paho.client.mqttv3`）为例，说明三个 QoS 级别在代码层面分别是怎么实现的。

### 9.1 发送端统一入口

所有 MQTT 发送都经过 `MqttGateway.send()`，qos 作为参数传入：

```java
// 文件：nms4cloud-starter-mqtt/.../MqttGateway.java

public void send(String topic, int qos, String content) {
    MqttMessage message = new MqttMessage(content.getBytes());
    message.setQos(qos);          // ← 关键：设置 QoS 等级
    client.publish(topic, message); // ← 发出去，qos=0/1/2 行为各不相同
}
```

调用方（`MqttMsgRealTimeConsumer`）写死了 `qos = 0`：

```java
mqttGateway.send(topic, 0, message.getMsg()); // ← qos = 0 硬编码
```

### 9.2 QoS 0 实现：只管发，不管收

**代码调用**：`client.publish(topic, message)`，qos = 0

**协议层发生了什么（MQTT 报文）**：

```
┌──────────────────────────────────────────────────────────────┐
│                      QoS 0 握手过程（最简单）                  │
│                                                              │
│  MQTT Broker                                                │
│       │                                                      │
│       │◄─────────── CONNECT（建立连接）─────────────────────│
│       │──────────── CONNACK ────────────────────────────────►│
│       │                                                      │
│       │◄─────────── PUBLISH ────────────────────────────────│
│       │         (topic, msg, QoS=0, DUP=0)                  │
│       │                                                      │
│  完成。Broker 发完 PUBLISH 就完事。                          │
│  POS 终端在线就收到，不在线就没收到，谁也不知道。              │
└──────────────────────────────────────────────────────────────┘
```

- MQTT 报文里有一个 `DUP` 标志位（Duplicate），QoS 0 时固定为 0
- **没有确认机制**：`client.publish()` 发出后立刻返回，Eclipse Paho 不维护任何重发队列
- 消息可能丢，但**永远不会重复**（因为没有重发）

**这个系统里的调用**：

```java
mqttGateway.send(topic, 0, message.getMsg());
// qos = 0，MQTT 协议层没有任何确认
// 应用层自己用 Redis + 延迟消息保证可靠性（见第二节）
```

### 9.3 QoS 1 实现：发一次，必须回确认

**代码调用**：`client.publish(topic, message)`，qos = 1

**协议层发生了什么（MQTT 报文）**：

```
┌──────────────────────────────────────────────────────────────┐
│                   QoS 1 握手过程（至少一次）                  │
│                                                              │
│  MQTT Broker                    POS 终端                     │
│       │                              │                       │
│       │◄─── PUBLISH ────────────────│                       │
│       │   (topic, msg, QoS=1, DUP=0)│                       │
│       │──── PUBACK ─────────────────►│                       │
│       │   (packetId=123)             │                       │
│       │                              │                       │
│  [Broker 没收到 PUBACK？]                                     │
│       │◄─── PUBLISH (DUP=1) ─────────│ ← 重发，同一个包      │
│       │──── PUBACK ─────────────────►│                       │
│       │                              │                       │
│  完成。消息至少到达一次，但可能重复。                          │
└──────────────────────────────────────────────────────────────┘
```

- 关键字段：`packetId`（包 ID），Broker 和 POS 用它匹配"这条 PUBLISH 对应哪条 PUBACK"
- **可能重复**：Broker 没收到 PUBACK 就重发，POS 可能已经处理过了（打印两张单）
- Paho 库内部维护一个**发送中的消息队列**，没收到 PUBACK 就自动重发
- 代码里**不用写 PUBACK**，库帮你做了

### 9.4 QoS 2 实现：四步握手，只收到一次

**代码调用**：`client.publish(topic, message)`，qos = 2

**协议层发生了什么（MQTT 报文）**：

```
┌──────────────────────────────────────────────────────────────┐
│                   QoS 2 握手过程（恰好一次）                  │
│                                                              │
│  MQTT Broker                    POS 终端                     │
│       │                              │                       │
│       │◄─── PUBLISH ────────────────│  ①                    │
│       │   (QoS=2, DUP=0)             │                       │
│       │──── PUBREC ─────────────────►│  ②                    │
│       │   (packetId=456)              │                       │
│       │◄─── PUBREL ─────────────────│  ③                    │
│       │──── PUBCOMP ────────────────►│  ④                    │
│       │   (packetId=456)              │                       │
│       │                              │                       │
│  完成。消息恰好到达一次（不会丢，也不会重复）。                 │
└──────────────────────────────────────────────────────────────┘
```

为什么需要四步？来看一个边界情况：

```
Broker 发 PUBLISH → POS 收到处理了 → 但 PUBREC 回包丢了
     │
     ▼
Broker 以为 POS 没收到，重发 PUBLISH (DUP=1)
     │
     ▼
POS 收到重复的 PUBLISH，怎么知道这是重复的？
     │
     ▼
靠 packetId：如果这个 packetId 已经在"处理中"列表里，就跳过处理
```

**四步的实质**：
- `PUBREC`（Publisher Received）：POS 告诉 Broker"我收到了，还没处理"
- `PUBREL`（Publisher Release）：Broker 说"现在可以正式处理这条消息了"
- `PUBCOMP`（Publisher Complete）：POS 说"处理完了"

**QoS 2 的代价**：
- 消息体积比其他等级大（多带 packetId）
- 握手步骤多，往返延迟最高
- Paho 库内部维护状态机，实现最复杂

### 9.5 三个 QoS 级别完整对比

| 维度 | QoS 0 | QoS 1 | QoS 2 |
|------|-------|-------|-------|
| **代码写法** | `message.setQos(0)` | `message.setQos(1)` | `message.setQos(2)` |
| **协议握手** | 无确认 | 1 次往返（PUBLISH ↔ PUBACK） | 2 次往返（4 步） |
| **是否会丢消息** | 可能丢 | 不丢（重发保证） | 不丢 |
| **是否会重复** | 不重复 | 可能重复 | 不会重复 |
| **消息到达次数** | 最多 1 次 | 至少 1 次 | 恰好 1 次 |
| **Broker 侧存储** | 不存储 | 存到收到 PUBACK | 存到收到 PUBCOMP |
| **库内部复杂度** | 无状态 | 维护发送队列 | 维护状态机 |
| **适用场景** | 丢弃可接受的场景 | 大多数场景 | 金融/支付等不允许重复的场景 |

### 9.6 这个系统为什么用 QoS 0 + 应用层保证

看代码里的实际调用就清楚了：

```java
// MqttMsgRealTimeConsumer.java（MQTT 发送的入口）
mqttGateway.send(topic, 0, message.getMsg());  // ← qos = 0
```

**系统选择 QoS 0 的核心理由**：

| 考虑因素 | 说明 |
|---------|------|
| POS 终端性能 | POS 终端资源有限，MQTT 协议层越轻越好 |
| 应用层可定制 | 自己写 Redis 确认机制，可以控制重试间隔、TTL、最大重发次数 |
| 状态持久化 | QoS 1/2 的确认队列在 Paho 库内存里，断电丢失；Redis 里存消息状态，断电不丢 |
| 业务灵活性 | POS 端断网 1 天后再联网，消息仍然有效（1 天 TTL）；MQTT 协议层做不到 |

**应用层确认机制在代码里的体现**：

```java
// MqMqttMsgServicePlus.send()（三段式）
// ① 写 Redis（记"这笔账还没结"）
redisSecTemplatePlus.set(keyForMsg, JSON.toJSONString(entity), Duration.ofDays(1));

// ② 发 MQTT QoS 0（实际发出去，不管收到没有）
sendMqtt(entity.getSid(), strContent);

// ③ 发延迟消息（10秒后回来检查"结账了没"）
rocketMQTemplatePlus.asyncSendDelay(MQTT_SEND_LOG, msg, DelayTimeLevelEnum.TEN_SECOND);
```

**POS 端确认**（`confirm()` 接口）：

```java
// 从 Redis 删掉 = "我收到了"
redisSecTemplatePlus.delete(keyForMsg);
```

**10 秒后检查**（`existMqttMsg()`）：

```java
// 还在 Redis → 没确认 → 重发
// 不在 Redis → 已确认 → 不用管
return redisSecTemplatePlus.hasKey(keyForMsg);
```

### 9.7 如果改成 QoS 1，代码只需要改一个数字

如果将来想换成 MQTT QoS 1，只需要改这一行：

```java
// MqttMsgRealTimeConsumer.java
mqttGateway.send(topic, 0, message.getMsg());
//                        ↑ 改成 1 就行
```

但需要注意：
- POS 端也需要配合（发 PUBACK）
- 不再有应用层确认机制，可能导致重复打印
- Redis + 延迟消息的逻辑仍然可以保留（双重保障）

## 十一、各组件职责与关系

> 很多初学者容易把各组件的功能搞混，这一节把每个组件的职责说清楚，并解释它们之间的关系。

### 11.1 各组件的真实职责

| 组件 | 职责 | 不是什么 |
|------|------|---------|
| **RocketMQ** | 消息调度中心 + 延迟闹钟 | 不是消息内容的存储地，不是 POS 收发的通道 |
| **Redis** | 消息状态存储（有没有被确认） | 不是消息队列，不帮消费者交换 RocketMQ 的消息 |
| **MQTT** | 协议——把消息从服务器发给 POS 终端 | 本身不保证可靠性，QoS 0 发出去不管 |
| **Netty** | 另一个通道——同样把消息发给 POS 终端 | 是 TCP 长连接推送，与 MQTT 并列，互为冗余 |

### 11.2 Redis 是用来帮消费者交换 RocketMQ 消息的吗？

**不是。** 这是一个常见的误解。

RocketMQ 的消费者（`MqttMsgRealTimeConsumer`）消费的是**发往 MQTT 的消息本身**（Topic: `MQ_SEND_MQTT_MSG`），Redis 在整个流程里根本不参与 RocketMQ 的消息交换。

Redis 的作用只有一个：**记录"这条消息 POS 终端收到了没有"**。

```
Redis 里的数据结构：

msgSet:{sid}  (ZSet)
  ├── 消息ID1 (时间戳=score)
  ├── 消息ID2
  └── 消息ID3

msg:{lid}     (String)
  └── {"content": "打印指令", "lid": 12345}   ← TTL=1天
```

- ZSet 是索引：快速查"这个门店有哪些消息还没确认"
- String 是内容：重发时取出来重新发送

### 11.3 为什么不用 Redis 做消息队列

既然 Redis 这么快，为什么不把消息存在 Redis 里，让消费者直接从 Redis 取？

| 方案 | 问题 |
|------|------|
| Redis 做消息队列 | 无持久化（重启丢消息）、无消费追踪、无延迟消息能力 |
| RocketMQ 做消息队列 | 持久化到磁盘、支持消费确认、支持任意延迟级别 |

RocketMQ 擅长的是**可靠存储 + 延迟投递**，Redis 擅长的是**高速读写 + TTL 自动清理**，两者各司其职。

### 11.5 三个 Topic 分别是什么

RocketMQ 在这个系统里用到了三个 Topic，各有各的用途：

| Topic | 性质 | 用途 | 消费者 |
|-------|------|------|--------|
| `MQ_SEND_MQTT_MSG` | 普通消息 | 触发 MQTT 实际发送 | `MqttMsgRealTimeConsumer` |
| `MQTT_SEND_LOG` | **延迟消息**（延迟 10 秒） | 10 秒后检查消息有没有被确认 | `RetryMqttMsgConsumer` |
| `NETTY_SUBSCRIBE` | 普通消息 | 终端订阅日志记录 | 其他消费者 |

注意：延迟消息和普通消息在同一个 Topic 里的区别在于**消息属性**（`delayLevel`），Broker 会根据这个属性决定是立刻投递还是等时间到了再投递。

### 11.6 组件关系一句话总结

- **RocketMQ**：负责"什么时候发什么消息"——既触发实时发送，又触发延迟检查
- **Redis**：负责"记录发了没有"——ZSet 做索引，String 存内容，POS 确认后删除
- **MQTT / Netty**：负责"怎么把消息送到 POS"——两条并行的物理通道，互为冗余

理解了这四者的分工，就理解了整个消息系统的设计。
