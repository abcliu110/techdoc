# 192.168.1.119 远程连接与支付链路排查方法说明

## 1. 文档目的

本文记录两部分内容：

- 我是如何连接 `192.168.1.119` 并进入现场排查的
- 遇到 `p.clovia.top` 访问异常、支付二维码超时等问题时，应该按什么顺序排查

本文偏“操作手册”，目的是后续你或其他同事可以照着复现排查路径。

---

## 2. 本次连接方式

### 2.1 连接目标

- 目标主机：`192.168.1.119`
- 角色：K8s 节点 / 物理机
- 系统：Linux

### 2.2 使用方式

本次没有使用交互式手动 SSH，而是通过本机脚本方式远程执行命令。

使用思路：

1. 本机先确认 `22` 端口可达
2. 使用 Python 的 `paramiko` 建立 SSH 连接
3. 在远端执行：
   - 系统信息命令
   - `kubectl` 命令
   - `curl` 命令
   - 日志查询命令

这样做的好处是：

- 可以连续执行多条检查命令
- 可以稳定抓取输出
- 适合排查容器、K8s、Ingress、Nginx、MQ、Netty 这类多层链路问题

### 2.3 权限方式

登录账号本身不是 root，因此涉及：

- `kubectl`
- 查看 Pod 日志
- 进入容器

这些动作时，使用了：

```bash
sudo -S
```

也就是：

- SSH 登录普通用户
- 需要 root 权限时再提权

说明：

- 用户名、密码属于敏感信息
- 后续文档和脚本中建议使用占位符，不建议把明文密码长期写入文档或脚本

---

## 3. 本次实际排查总思路

这次问题不是一开始就能判断在前端、Nginx 还是支付服务，所以采用的是“由外到内、由静到动”的分层排查法。

排查顺序如下：

```text
1. 先确认主机可达
2. 再确认域名入口是否正常
3. 再确认 K8s Pod / Service / Ingress 是否正常
4. 再确认静态页面是否正常返回
5. 再确认前端实际调用了哪个支付接口
6. 再确认支付接口是入口超时，还是后端业务超时
7. 再确认 mq -> netty -> POS -> 回调 的异步链路
8. 最终定位根因
```

---

## 4. 连接 119 的标准排查步骤

### 第一步：先确认 SSH 端口是否通

本机执行：

```powershell
Test-NetConnection 192.168.1.119 -Port 22
```

判断标准：

- `TcpTestSucceeded : True`
  - 说明 22 端口可达，可以继续
- 如果不通：
  - 先查网络、防火墙、主机是否在线

---

### 第二步：确认能否登录

连接成功后先做最基础的系统识别：

```bash
hostname
whoami
uname -a
```

作用：

- 确认登录到的是不是目标机器
- 确认当前用户是谁
- 确认操作系统版本

---

### 第三步：确认这台机器到底跑了什么

先不要假设“是普通 Linux Nginx”还是“是 K8s 节点”，先查事实：

```bash
which kubectl
which ctr
which crictl
ps -ef | grep -E "nginx|kubelet|containerd"
systemctl is-active kubelet
systemctl is-active containerd
```

本次查出来的事实是：

- 这是 `rke2/k8s` 节点
- 存在 `kubelet`
- 存在 `containerd`
- 节点里跑着容器化的 `nginx`

也正因为这个事实，后续排查全部切换到了：

- `kubectl get pod`
- `kubectl logs`
- `kubectl exec`

而不是继续按传统单机 Nginx 思路查 `/etc/nginx`。

---

## 5. 针对访问异常的排查方法

### 5.1 先区分“域名入口问题”还是“业务页面问题”

对外部域名先做最轻量检查：

```bash
curl -I http://p.clovia.top
```

如果返回：

- `200`
  - 说明入口大概率正常
- `404`
  - 需要区分是 Ingress 404 还是业务 Nginx 404
- 超时
  - 先查 FRP / Ingress / 网关入口

本次一开始就是通过这一步确认：

- 域名可达
- 但一度返回 `404`

---

### 5.2 区分“业务 Nginx 正常”还是“入口路由错误”

如果已知业务 Service 的 NodePort，可直接在目标机器本地测试：

```bash
curl -I http://127.0.0.1:30080
curl -I -H 'Host: p.clovia.top' http://127.0.0.1:30080
```

作用：

- 不经过公网
- 不经过 FRP
- 不经过域名解析
- 直接测业务 Nginx 是否能正确返回页面

本次就是通过这一步确认：

- `nms4cloud/nginx` 自身页面是正常的
- 404 问题不在静态页面目录

---

### 5.3 查 K8s 资源而不是猜

标准命令：

```bash
kubectl get pods -A -o wide
kubectl get svc -A
kubectl get ingress -A
kubectl describe pod <pod> -n <namespace>
kubectl get deploy <name> -n <namespace> -o yaml
```

作用：

- 确认 Pod 在不在
- 确认 Service 端口暴露方式
- 确认 Ingress 有没有对应 Host 规则
- 确认 Deployment 是否真挂了卷、真配置了端口

本次就是靠这些命令确认：

- `nms4cloud/nginx` 是 NodePort `30080`
- `p.clovia.top` 对应的 Ingress 当时缺失
- 补完 Ingress 后域名入口恢复正常

---

## 6. 针对“支付页超时”的排查方法

### 6.1 先确认是不是首页本身超时

带完整 URL 测试：

```bash
curl -I "http://p.clovia.top/?S=...&M=...&B=..."
curl -w "start=%{time_starttransfer} total=%{time_total}" ...
```

如果首页 HTML 很快返回：

- 说明页面入口没问题
- 后面的超时多半发生在前端加载后调用的支付接口

本次就是这样确认的：

- 根页面正常
- 带参数首页也能快速返回 HTML
- 超时不在页面入口，而在后续支付接口

---

### 6.2 从前端代码反查实际调用哪个接口

当用户描述“带参数就超时”时，不要猜接口，直接查前端代码：

```bash
rg -n "URLSearchParams|S|M|B|get_qrcode_from_offline|isOpenIdNeeded" <frontend-src>
```

本次在 `uni4pay` 中查到：

- 先调 `/api/pay/pay_channel/isOpenIdNeeded`
- 再调 `/api/mq/pay/get_qrcode_from_offline`

这一步非常关键，因为它把问题从“页面问题”缩小成了“支付接口问题”。

---

### 6.3 对每个支付接口单独压测，不混在页面里

不要只靠浏览器页面感觉“像是超时”，要直接单独调接口：

```bash
curl -X POST http://.../api/pay/pay_channel/isOpenIdNeeded
curl -X POST http://.../api/mq/pay/get_qrcode_from_offline
```

本次结果：

- `isOpenIdNeeded`
  - 很快返回 `success=true`
- `get_qrcode_from_offline`
  - 一开始约 20 秒后返回 `over time`
  - 后来恢复到 1.85 秒左右返回 `qr_code`

这一步让问题精确收敛到：

- 不是 `payment` 的 openid 判断问题
- 是 `mq` 出码链路问题

---

## 7. 针对 mq / netty / POS 链路的排查方法

### 7.1 先确认 mq 是否真收到请求

查 `mq` 日志：

```bash
kubectl logs -n nms4cloud <mq-pod> | grep get_qrcode_from_offline
```

如果能看到：

```text
get_qrcode_from_offline: GenQrcodeForBill:netty_xxx
```

说明：

- 前端 -> gateway -> mq 这条链是通的

---

### 7.2 再确认 mq 等待的是不是 POS 回调

查 `mq` 回调日志：

```bash
kubectl logs -n nms4cloud <mq-pod> | grep receive_qrcode_from_offline
```

如果没有这条日志：

- 说明 POS 没有把二维码结果回传回来

如果有：

- 说明 POS 已经回包
- mq 侧问题基本解决

---

### 7.3 判断 topic 是否已经订阅成功

这是这次最关键的检查项。

查询 `mq` 本地 topic：

```bash
curl -X POST http://127.0.0.1:18120/getChannelHandlerContextByTopic   -H 'Content-Type: application/json'   --data '{"Topic":"1942885905090105345_server"}'
```

判断标准：

- 如果返回“没有被订阅”
  - 在 `useNettyService=false` 模式下，出码一定会超时
- 如果返回带有 `topics` 的连接信息
  - 说明 POS 已经挂在 mq 本地
  - 出码链路有机会成功

本次根因就是在这一层被找到的。

---

### 7.4 判断当前运行的是哪种 Netty 模式

查代码和配置：

- `useNettyService=false`
  - 使用 `mq` 本地 topic 链路
- `useNettyService=true`
  - 调独立 `nms4cloud-netty` 服务

本次现场最终确认：

- 当前 `mq` 本地已经有 topic
- 且出码已经恢复
- 说明此时 `false` 模式已经具备运行条件

---

## 8. 为什么这次前后结论会变化

本次排查过程中，现场状态发生过变化，所以不同时间点看到的结果不同。

前期状态：

- `mq` 本地查不到 `1942885905090105345_server`
- 出码接口返回 `over time`

后期状态：

- `mq` Pod 重启后变成新实例
- `mq` 本地已经能查到 `1942885905090105345_server`
- 出码接口恢复返回 `qr_code`

这说明：

- 问题并不是“某个接口永久坏掉”
- 而是“运行中订阅关系在某个时段没有建立好”

所以做链路排查时，必须注意：

- 不同时间的运行结果可能不同
- 排查结论要以当前最新实测为准

## 10. 后续建议的标准排查顺序

以后遇到同类问题，建议永远按下面顺序排查：

### 第 1 层：网络 / SSH

- 主机能不能连
- 22 端口通不通

### 第 2 层：入口和静态页

- 域名是否解析正确
- Ingress / FRP / Nginx 是否正常
- 静态首页是否快速返回

### 第 3 层：前端实际请求了谁

- 从代码确认真实接口
- 不凭感觉猜测

### 第 4 层：接口逐个测

- 哪个接口快
- 哪个接口慢
- 哪个接口报错

### 第 5 层：异步链路

- mq 是否收到请求
- mq 是否发出消息
- topic 是否已订阅
- POS 是否收到命令
- POS 是否回调
- mq 是否收到回调

---

## 11. 一句话总结

这次远程排查的关键方法不是“猜配置”，而是：

- 先连上主机
- 逐层拆链路
- 每一层都用实际命令验证
- 把“入口问题、页面问题、接口问题、异步消息问题”一层层排除

最终真正把问题定位到了：

- `mq` 本地是否已经拿到 POS 的 topic 订阅

也就是说，支付二维码能不能出来，最后不取决于页面本身，而取决于：

- `mq` 能不能把 `GenQrcodeForBill` 真实送到 POS
- POS 又能不能把二维码结果回调回来
