# V2 离线箱机制说明

本文专门解释：

- `D:\mywork\netty-mq-replica-suite\replica-v2\src\main\java\com\example\replica\v2`

里“离线箱”是怎么实现的。

核心结论先说：

> v2 的离线箱本质上是一个内存 Map。  
> 它按 `topic` 维度缓存离线消息。  
> 设备重新订阅后，再按 `topic` 取出这些消息重新发送。

也就是说，它不是数据库，不是 MQ，也不是 Redis。

它是：

> 教学版内存离线缓存

---

## 1. 离线箱的核心数据结构

离线箱定义在：

- `V2Stores`

核心代码：

```java
public static final Map<String, List<V2ReplicaMessage>> OFFLINE_BOX = new ConcurrentHashMap<>();
```

这行的含义是：

```text
topic -> 这个 topic 当前积压的离线消息列表
```

例如：

```text
2002_server -> [msgA, msgB, msgC]
```

你可以把它理解成：

- Map 的 key 是 `topic`
- Map 的 value 是这个 topic 下尚未补发的消息列表

所以离线箱不是“按设备对象存”

而是：

> 按订阅主题存

这和整套系统“按 topic 路由”的核心思想是一致的。

---

## 2. 为什么离线箱要按 topic 存，而不是按连接存

因为发送链路本身就是：

```text
业务侧 -> topic -> 路由表 -> 在线连接
```

业务系统不知道：

- 某个具体 socket 是谁

它只知道：

- 我要发给哪个 `topic`

所以离线缓存也应该和发送模型一致：

```text
topic -> 离线消息
```

这样设备上线时，只要重新订阅了某个 topic，就能把对应 topic 下的离线消息一起补出来。

---

## 3. 离线箱是在哪一步写入的

写入逻辑在：

- `V2ServerHandler.send(String topic, V2ReplicaMessage message)`

核心代码：

```java
Set<ChannelHandlerContext> contexts = V2Stores.TOPIC_TO_CONTEXT.get(topic);
if (contexts == null || contexts.isEmpty()) {
    System.out.println("[v2-server] no subscriber, store offline, topic=" + topic);
    V2Stores.OFFLINE_BOX
        .computeIfAbsent(topic, ignored -> new ArrayList<>())
        .add(cloneMessage(message));
    return false;
}
```

---

## 4. 这段写入逻辑怎么理解

它分 4 步。

### 第一步：先查这个 topic 有没有在线订阅者

```java
Set<ChannelHandlerContext> contexts = V2Stores.TOPIC_TO_CONTEXT.get(topic);
```

这句的意思是：

- 去在线路由表里查
- 这个 `topic` 当前有没有连接

如果查出来：

- `null`
- 或者空集合

那就说明：

```text
当前没人在线
```

---

### 第二步：没人在线时，不直接丢消息

判断条件：

```java
if (contexts == null || contexts.isEmpty())
```

进入这里说明：

- 这条消息此刻没法立即下发

但 v2 和 classic 的差别就在这里：

- classic：没人在线就返回失败
- v2：没人在线先缓存起来

所以日志里才会看到：

```text
[v2-server] no subscriber, store offline, topic=...
```

---

### 第三步：如果这个 topic 还没有离线列表，就先创建一个

代码：

```java
V2Stores.OFFLINE_BOX.computeIfAbsent(topic, ignored -> new ArrayList<>())
```

意思是：

- 如果 `OFFLINE_BOX` 里还没有这个 topic
- 就创建一个新的 `ArrayList`

例如原来：

```text
OFFLINE_BOX:
空
```

第一次有离线消息发给 `2002_server` 时，会变成：

```text
OFFLINE_BOX:
2002_server -> []
```

---

### 第四步：把这条消息追加进去

代码：

```java
.add(cloneMessage(message));
```

这句的意思是：

- 把当前消息保存到这个 topic 对应的离线列表里

例如：

```text
OFFLINE_BOX:
2002_server -> [msgA]
```

如果后面又来一条：

```text
OFFLINE_BOX:
2002_server -> [msgA, msgB]
```

所以当前实现是：

> 同一个 topic 下的离线消息按到达顺序追加到列表里

---

## 5. 为什么不是直接 `.add(message)`，而是 `cloneMessage(message)`

这点非常重要。

保存离线消息时用的是：

```java
cloneMessage(message)
```

而不是原对象本身。

对应方法：

```java
private V2ReplicaMessage cloneMessage(V2ReplicaMessage source)
```

这个方法会新建一个 `V2ReplicaMessage`，再把字段复制过去。

作用是：

> 把离线箱保存成消息快照，而不是共享引用

这样做的好处是：

1. 后面外部如果改原对象，不会污染离线箱
2. 多次重试时不会因为同一对象被复用导致状态串联
3. 离线箱里的消息更像“历史记录”

所以你可以把 `cloneMessage()` 理解成：

```text
入箱前先拍一个快照
```

---

## 6. 离线箱是在哪一步被补发的

补发触发点在：

- `V2ServerHandler.channelRead0()`

核心代码：

```java
if ("subscribe".equalsIgnoreCase(message.getCmd())) {
    subscribe(context, message.getTopic());
    flushOfflineMessages(message.getTopic());
    return;
}
```

这段逻辑的意思是：

设备一旦上线并发送 `subscribe`：

1. 先建立订阅关系
2. 再尝试补发这个设备刚刚订阅的 topic 下的离线消息

这就是为什么：

> 离线消息不是“连接一建立就补发”  
> 而是“topic 订阅建立后才补发”

因为补发依赖 topic 路由。

---

## 7. `flushOfflineMessages()` 是怎么补发的

核心代码：

```java
for (String topic : topics) {
    List<V2ReplicaMessage> offlineMessages = V2Stores.OFFLINE_BOX.remove(topic);
    if (offlineMessages == null || offlineMessages.isEmpty()) {
        continue;
    }
    System.out.println("[v2-server] flush offline messages, topic=" + topic + ", size=" + offlineMessages.size());
    for (V2ReplicaMessage offlineMessage : offlineMessages) {
        send(topic, offlineMessage);
    }
}
```

这段逻辑可以分成 4 步。

---

### 第一步：遍历本次订阅的所有 topic

```java
for (String topic : topics)
```

意思是：

- 当前设备如果订阅了多个 topic
- 那就逐个 topic 去查离线箱

---

### 第二步：把这个 topic 的离线消息整批取出来

```java
List<V2ReplicaMessage> offlineMessages = V2Stores.OFFLINE_BOX.remove(topic);
```

这句非常关键。

注意它不是：

```java
get(topic)
```

而是：

```java
remove(topic)
```

这表示：

> 不是只查看，而是“取出并删除”

为什么这样做？

因为：

- 这些消息现在准备补发了
- 不能还继续留在离线箱里

否则会重复补发。

所以这一步的语义是：

```text
把这个 topic 的离线积压整批出箱
```

---

### 第三步：如果没有离线消息，就跳过

```java
if (offlineMessages == null || offlineMessages.isEmpty()) {
    continue;
}
```

说明：

- 并不是每个 topic 都一定有离线积压
- 没有就跳过

---

### 第四步：一条条重新调用 `send()`

```java
for (V2ReplicaMessage offlineMessage : offlineMessages) {
    send(topic, offlineMessage);
}
```

这句的意义是：

> 离线补发并没有单独搞一套发送逻辑  
> 而是复用普通发送入口 `send()`

这很重要。

因为这样设计后：

- 在线直发
- 离线补发

最终都走同一个发送方法。

好处是：

1. 逻辑统一
2. 不容易出现两套发送规则不一致
3. 更容易阅读和维护

---

## 8. 为什么 `flushOfflineMessages()` 要在 `subscribe()` 后调用

顺序是：

```java
subscribe(context, message.getTopic());
flushOfflineMessages(message.getTopic());
```

不能反过来。

因为：

如果你先补发：

- 这时路由表里还没有这个新连接
- `send(topic, offlineMessage)` 查不到在线订阅者
- 消息又会重新回到离线箱

所以必须先：

```text
建立 topic -> context 路由
```

再：

```text
按 topic 补发离线消息
```

这是离线补发成立的前提。

---

## 9. `V2ReplicaMain` 是怎么制造离线场景的

在：

- `V2ReplicaMain`

里，顺序是故意这么写的：

### 第一步：先启动服务

```java
nettyServer.start();
httpBridgeServer.start();
```

### 第二步：客户端还没上线时先发一条消息

```java
businessService.sendNetty(2002L, "nms4cloud", "v2 offline message before client online");
```

这时：

- `TOPIC_TO_CONTEXT` 里没有 `2002_server`
- 消息会进入 `OFFLINE_BOX`

### 第三步：过一会儿客户端才上线订阅

```java
deviceClient.start();
```

客户端发送 `subscribe`

于是触发：

```java
flushOfflineMessages(...)
```

### 第四步：离线消息被补发

日志里会出现类似：

```text
[v2-server] flush offline messages, topic=2002_server, size=...
```

这就是离线箱出箱补发。

## 10. 为什么日志里可能会看到 `size=3`

这是当前教学实现的一个特点。

因为：

1. 第一次发送时没人在线
   - 消息入离线箱
2. retry 线程后面又拿到重试任务
   - 再次调用 `send()`
3. 如果此时还是没人在线
   - 又会再次入离线箱

所以同一条消息可能被多次追加进 `OFFLINE_BOX`

例如：

```text
OFFLINE_BOX:
2002_server -> [msgA, msgA, msgA]
```

这也是为什么设备上线时，日志可能显示：

```text
size=3
```

说明：

- 当前实现是“追加列表模型”
- 没有按 `msgId` 去重

所以它是：

> 教学版离线箱，不是生产级离线消息系统

## 11. 当前离线箱实现的特征总结

你可以把 v2 的离线箱总结为下面 5 点：

### 1. 存储介质

```text
内存 Map
```

不是数据库，不是 Redis，不是 MQ。

---

### 2. 分组维度

```text
按 topic 分组
```

不是按 socket，不是按设备对象。

---

### 3. 缓存内容

```text
List<V2ReplicaMessage>
```

保存的是消息快照列表。

---

### 4. 入箱时机

```text
send() 发现当前 topic 没有在线订阅者
```

---

### 5. 出箱时机

```text
设备发送 subscribe 后
```

并且是：

```text
先建路由，再按 topic 补发
```

---

## 12. 一句话总结离线箱模型

一句话理解：

> `OFFLINE_BOX` 是一个按 topic 分组的内存离线消息缓存；消息在发送时若无人在线则入箱，设备重新订阅后再按 topic 出箱补发。

再压缩成最短一句：

> 无人在线先入箱，重新订阅再出箱。
