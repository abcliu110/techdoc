# 微信点餐完整时序图

## 1. 说明

本文是对《微信点餐链路中 Netty、RocketMQ、WebSocket 使用分析》的补充，重点用时序视角描述：

1. 用户扫码进入点餐
2. 用户点餐后门店端如何联动
3. 用户支付后系统如何延迟关单与异步处理

本文全部使用纯文本流程图，确保在 VS Code 源码视图、Markdown 预览和普通控制台里都能直接查看。

---


## 3. 扫码进入点餐页时序

## 3.1 业务说明

扫码进入时，前端不会只靠 URL 自己判断是不是点餐码，而是：

1. 先完成微信登录
2. 再把 `mid + code(qrcodeKey)` 发给后端
3. 后端查二维码绑定表
4. 返回该码对应的业务类型 `businesstype`
5. 前端据此进入点餐页、外卖页、自提页等

## 3.2 时序图

```text
用户
  |
  | 扫码打开链接 / 小程序码
  v
前端(taro-mall / nms4pos-ui)
  |
  | Taro.login() / H5 授权
  v
微信
  |
  | 返回 code
  v
前端
  |
  | POST /wxMaLogin
  v
CrmAuthController
  |
  | 返回 session / 用户信息
  v
前端
  |
  | POST /pt_qr_bind/get { mid, qrcodeKey }
  v
PtQrBindController
  |
  v
PtQrBindServicePlus
  |
  | 查询 pt_qr_bind
  | 查询门店 / 桌台 / 支付方式 / 是否允许点餐
  v
数据库 / 门店基础数据
  |
  | 返回 PtQrBindVO
  v
前端
  |
  | 根据 businesstype 分流
  +--> order         -> 点餐页
  +--> takeout       -> 外卖页
  +--> self_lifting  -> 自提页
  +--> book_mgr      -> 预订页
```

## 3.3 关键点

- `code` 本质就是后端生成的 `qrcodeKey`
- `businesstype` 决定这是不是点餐码
- 前端是在拿到后端结果后才知道该跳哪个业务页

## 4. 用户点餐后的联动时序

## 4.1 业务说明

用户在点餐页加菜后，主链路还是 HTTP 提交订单。

真正复杂的是“点餐成功之后系统通知谁”：

1. 更新订单金额
2. 创建打印任务
3. 推送给门店前端刷新
4. 推送给 KDS 厨显刷新
5. 推送给打印端/设备端执行任务

## 4.2 时序图

```text
用户
  |
  | 加菜 / 点餐
  v
点餐前端
  |
  | POST addFood
  v
DwdBillOpsForBizController
  |
  v
DwdBillOpsService
  |
  v
DwdFoodOpsService
  |
  | 1. 保存菜品
  | 2. 重算订单金额
  | 3. 生成打印任务
  |
  +------------------> CalcOrderService
  |
  +------------------> PosPrnJobService
  |
  +------------------> MessageUtil.broadcastInfo(...)
  |                      |
  |                      v
  |                  本地 WebSocketServer
  |                      |
  |                      +--> 收银前端: RefreshOrder / RefreshTable
  |                      |
  |                      +--> KDS前端: RefreshKds
  |
  +------------------> PushNettyService.sendMessage(...)
                         |
                         v
                       Netty
                         |
                         v
                    打印端 / 设备端

最后返回:

DwdFoodOpsService
  -> DwdBillOpsService
  -> DwdBillOpsForBizController
  -> 点餐前端
```

## 4.3 这一步里三种技术的角色

### HTTP

负责：

- 提交点餐请求
- 返回点餐结果

### WebSocket

负责：

- 收银前端刷新
- KDS 厨显刷新
- 桌台数据刷新

### Netty

负责：

- 打印端消息推送
- 设备端消息推送
- topic 定向下发

## 5. 厨房出菜/撤菜时序

## 5.1 业务说明

厨房侧状态变化不一定重新下单，但一定要通知相关前端刷新。

典型场景：

- 出菜完成
- 撤菜
- 隐藏菜品

这些操作会触发 `RefreshKds` 广播。

## 5.2 时序图

```text
厨房操作端
  |
  | 完成 / 撤菜 / 隐藏
  v
DwdFoodMakingController
  |
  v
厨房业务 Service
  |
  | 更新菜品状态成功
  v
DwdFoodMakingController
  |
  | MessageUtil.broadcastInfo(RefreshKds)
  v
本地 WebSocket
  |
  +--> KDS前端: 刷新厨房显示
  |
  +--> 收银前端: 刷新订单状态
```

## 6. 用户支付后的时序

## 6.1 业务说明

支付相关主线：

1. 用户发起微信支付
2. 后端生成支付单
3. 同时写入 30 分钟延迟关单消息
4. 到时消费消息，判断订单是否支付
5. 未支付则重试或自动关闭
6. 已支付则继续券、分佣等异步逻辑

## 6.2 时序图

```text
用户
  |
  | 提交支付
  v
点餐前端
  |
  | 发起支付
  v
PayOrderServiceImpl
  |
  | 创建支付单
  v
支付渠道
  |
  | 返回支付信息
  v
PayOrderServiceImpl
  |
  | 发送 PT_ORDER_CLOSE_QUEUE (30分钟延迟)
  v
RocketMQ

30 分钟后:

RocketMQ
  |
  | 投递延迟关单消息
  v
PTOrderCloseQueueConsumer
  |
  | 查询支付状态
  v
支付渠道
  |
  +--> SUCCESS
  |      |
  |      +--> 更新订单状态
  |      +--> 触发券 / 分佣 / 后续异步任务
  |
  +--> USERPAYING / NOTPAY
         |
         +--> 如果仍需观察: 重新投递短延迟消息
         |
         +--> 如果超时未付: 自动取消订单
```

## 6.3 RocketMQ 在这一步的作用

RocketMQ 在支付链路里的主要职责：

- 延迟关单
- 重试检查
- 后续异步任务解耦

它不是面向用户前端的实时协议，而是后台流程控制器。

## 7. 打印任务下发时序

## 7.1 业务说明

打印任务常见流程：

1. 业务层创建打印任务
2. 组装设备 topic
3. 通过 Netty 推送给打印端
4. 如果 Netty 失败，按 IP 或 HTTP 兜底

## 7.2 时序图

```text
POS业务服务
  |
  | 创建打印任务
  v
打印任务服务
  |
  | sendMessage(topic, body)
  v
PushNettyService / PushNettyServerService
  |
  +--> 优先按 topic 发
  |      |
  |      +--> topic 在线 -> Netty -> 打印端 / 设备端
  |
  +--> topic 不在线时按 IP 发
  |      |
  |      +--> IP 在线 -> Netty -> 打印端 / 设备端
  |
  +--> 仍失败 -> HTTP 兜底发送
```

## 8. 本地 POS 前端连接时序

门店桌面前端的 WebSocket 连接是持续存在的，不是每次下单临时建连。

```text
pos4desktop 前端
  |
  | 读取 apiServerIp / uuid
  v
Electron 本地配置
  |
  | 返回 serverIp + uuid
  v
前端
  |
  | 建立 ws://serverIp:9180/api/ws/{uuid}
  v
pos3boot WebSocketServer
  |
  | 连接成功后常驻
  |
  +--> 推送 RefreshTable
  +--> 推送 RefreshKds
  +--> 推送 RefreshNetwork
  |
  v
前端页面实时刷新
```

## 9. 时序拆解总结

把整条微信点餐链路拆开后，职责非常清晰：

### 9.1 用户动作层

- 微信登录
- 扫码
- 点餐
- 支付

主要走 HTTP。

### 9.2 门店前端刷新层

- 桌台刷新
- KDS 刷新
- 订单刷新
- 网络状态刷新

主要走 WebSocket。

### 9.3 门店设备联动层

- 打印机
- 设备客户端
- topic 推送

主要走 Netty。

### 9.4 后台异步控制层

- 延迟关单
- 重试补偿
- 券
- 分佣
- MQTT 投递

主要走 RocketMQ。

---

## 10. 最后一张总图

```text
微信小程序 / H5
        |
        | HTTP
        v
登录 / 二维码 / 点餐 / 订单 / 支付服务
        |
        +----------------------> WebSocket 广播 ---------------------> 收银前端 / KDS前端
        |
        +----------------------> Netty 推送 -------------------------> 打印端 / 设备端
        |
        +----------------------> RocketMQ ----------------------------> 延迟关单 / 异步任务 / 补偿
```

最终可归纳为一句话：

**微信点餐主请求走 HTTP，页面刷新走 WebSocket，设备联动走 Netty，后台解耦走 RocketMQ。**
