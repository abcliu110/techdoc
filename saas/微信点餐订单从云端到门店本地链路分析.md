# 微信点餐订单从云端到门店本地链路分析

## 1. 目标

这份文档只回答一个问题：

**微信点餐订单，是如何从云端进入门店本地的？**

这里不再混入 Canal 基础资料同步。

---

## 2. 最终结论

当前源码最稳的结论是：

```text
微信点餐前端
  ->
云端订单服务
  ->
云端生成门店订单请求消息
  ->
门店本地订单处理器
  ->
创建或匹配本地 dwd_bill
  ->
继续落本地菜品/支付/打印/KDS
  ->
回告云端接单结果
```

这条链不是：

```text
Canal
  ->
Kafka
  ->
Netty
  ->
IncrementalSyncDataService
  ->
本地订单表
```

## 3. 云端侧：订单先进入云端订单服务

核心类：

[OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java)

### 3.1 `cashPostOrder(CashPostOrderDTO request)`

位置：

[OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java:1157)

这一段说明：

- 微信点餐订单先以 `CashPostOrderDTO` 形式进入云端
- 订单里含有：
  - `OrderBillAddDTO order`
  - `foodList`
  - `payList`
  - `msgId`
  - `orderId`
  - `tableNo`
  - `viewMode`
- 云端会把：
  - `mid`
  - `sid`
  转成负值保存到线上订单域
- 并把整单 JSON 放到 Redis：
  `OrderRedisConstants.getCashPostOderKey(saasOrderKey)`

所以这一层的数据载体是：

```text
HTTP JSON
  ->
CashPostOrderDTO
  ->
云端订单域对象 / Redis
```

### 3.2 这一层不是门店本地库

这里的 `OrderBill` / `OrderFood` 是云端订单服务自己的对象，不是门店本地 `DwdBill` / `DwdFood`。

直接证据：

- `order.setMid(-1 * request.getMerchantNo())`
- `order.setSid(-1 * request.getTerminalId())`

这说明它在维护线上订单空间，不是在操作门店本地表。

## 4. 云端侧：向门店发起订单请求

### 4.1 `getOfflineOrderInfo(GetOfflineOrderDTO request)`

位置：

[OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java:1318)

这一段是云端到门店的关键。

它会：

1. 生成一个 `msgId`
2. 注册 `monoResponsive` 回调
3. 调 `orderUtil.sendMessageToShop(...)`
4. 下发 `OrderMsgDTO`
5. 消息类型是 `OrderMsgTypeEnum.CASH_REQUEST`

### 4.2 这一层的数据如何传

数据载体是：

```text
GetOfflineOrderDTO
  ->
OrderMsgDTO
  ->
OrderDataDTO
```

`OrderDataDTO` 里至少带：

- `billId`
- `billName`
- `tblId`
- `sid`
- `cardId`
- `openId`
- `pull`
- `mustOfflineOpenTbl`
- `id(msgId)`

所以云端不是把数据库表同步过去，而是把“请门店处理这笔线上订单”的业务消息发过去。

## 5. 门店侧：DoOrderHandler 负责把线上订单变成本地订单

核心类：

[DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java)

### 5.1 `checkAndGetTbl(...)`

关键位置：

[DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:496)

这段代码说明门店本地收到线上订单后，会按业务规则判断：

- 是堂食、外卖还是自提
- 是否已经有本地单
- 是否需要新开本地单
- 是否按桌台合并
- 是否按微信用户合并
- 是否每次扫码生成新单

### 5.2 堂食单如何变成本地单

如果没有可复用的本地单，代码会构造：

- `DwdBillCreateDTO`

并设置：

- `mid`
- `sid`
- `billName`
- `personNum`
- `source = Customer`
- `tableLid`
- `openId`
- `orderSubType = DINE_IN`
- `pickNumber`
- `receive = true`
- `remark`
- `otherOrderKey(order.getSaasOrderKey())`

然后执行：

```java
DwdBill dwdBill = dwdBillServicePlus.create(dwdBillCreateDTO, user);
```

这就是本地 `dwd_bill` 的创建动作。

### 5.3 外卖/自提如何变成本地单

如果是外卖或自提，仍然不是资料同步，而是本地业务建单：

- 构造 `DwdBillCreateDTO`
- 写客户姓名、电话、地址、自提时间、取餐号
- 调 `dwdBillServicePlus.create(...)`

所以无论堂食还是外卖，这里都是门店业务建本地单，不是 Canal 写库。

## 6. 门店侧：如何判断复用旧单还是新建单

门店本地会查：

- `DwdBill::getSaasOrderKey`
- `DwdBill::getOtherOrderKey`
- `DwdBill::getTableLid`
- `DwdBill::getOpenId`

见：

- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:623)
- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:642)

这说明“云端订单 -> 本地订单”的关联关系，不是表同步，而是通过订单号、桌台、OpenId 进行业务匹配。

## 7. 本地接单后如何回告云端

关键类：

[Nms4cloudUtil.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/util/Nms4cloudUtil.java:408)

本地接单成功时会调用：

- `confirmSuccess(msgId, orderId, orgBillId, billName)`
- 最终走 `confirmOrder(...)`

回告载体是：

- `ConfirmOrderDTO`

里面包含：

- `msgId`
- `orderId`
- `confirmStatus`
- `desc`
- `orgBillId`
- `billName`

所以门店接单后，云端会知道：

- 这笔线上单有没有被门店接收
- 本地生成了哪个单号
- 接单失败时的原因是什么

## 8. 当前能确认的数据传递方式

### 第 1 步：前端到云端

载体：

- HTTP JSON
- `CashPostOrderDTO`

### 第 2 步：云端保存线上订单态

载体：

- `OrderBillAddDTO`
- `OrderFoodAddDTO`
- `OrderPayAddDTO`
- Redis `cashPostOrderKey`

### 第 3 步：云端到门店

载体：

- `OrderMsgDTO`
- `OrderDataDTO`
- `msgType = CASH_REQUEST`

### 第 4 步：门店本地建单

载体：

- `DoGetOrderVO`
- `DwdBillCreateDTO`
- `dwdBillServicePlus.create(...)`

### 第 5 步：门店到云端回告

载体：

- `ConfirmOrderDTO`

## 9. 当前最稳的纯文本时序图

```text
微信前端
  |
  | HTTP 提交订单
  v
云端 OrderBillServicePlus.cashPostOrder()
  |
  | 保存线上订单态 / Redis
  v
云端 OrderBillServicePlus.getOfflineOrderInfo()
  |
  | 发送 OrderMsgDTO(msgType=CASH_REQUEST)
  v
门店本地订单消息处理
  |
  | DoOrderHandler.checkAndGetTbl(...)
  | 查本地单 / 新建本地单
  v
本地 dwd_bill
  |
  | 后续菜品、打印、KDS、本地业务联动
  v
Nms4cloudUtil.confirmOrder(...)
  |
  | 回告云端接单结果
  v
云端
```

## 10. 不能过度下结论的地方

当前我还没有在这轮里继续抠出的点：

1. `orderUtil.sendMessageToShop(...)` 最终是通过哪条底层通道到门店
2. `DoOrderHandler` 后续哪一段代码精确把菜品明细写进 `dwd_food`
3. `CASH_REQUEST` 的门店消息接收入口完整调度链

所以这份文档当前最稳的边界是：

- 已能确认“云端不是通过 Canal 把微信点餐订单同步进门店本地库”
- 已能确认“云端通过订单请求消息，驱动门店本地业务创建 `dwd_bill`”
- 还没把“本地 `dwd_food` 的精确写入方法链”完整展开到每个方法
