# 微信点餐订单与基础资料同步方向差异说明

## 1. 结论先行

前面的分析里，容易混淆的是两条完全不同的链路：

1. 基础资料同步
2. 微信点餐订单同步

当前源码能明确支持的结论是：

- 基础资料同步：走 `Canal + Kafka + 云端 Netty + 门店本地消费 + 本地写库`
- 微信点餐订单：**不能归到这条 Canal 链里**
- 微信点餐订单更接近：`云端订单服务 -> 门店本地订单请求/接单处理 -> 本地 dwd_bill / dwd_food`

也就是说：

```text
基础资料:
云端 -> 门店本地库

微信点餐订单:
云端订单 -> 门店本地业务 -> 门店本地订单表
```

## 4. 微信点餐订单这条链，当前更合理的源码解释

### 4.1 云端订单服务并不是直接写门店本地库

云端订单服务更像“线上订单中心 + 向门店发起取单/接单流程”。

直接证据：

- 云端接收扫码点餐/后付订单请求：
  [OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java:1157)
- 云端在 `cashPostOrder()` 里并没有直接操作门店本地表，而是：
  - 规范化线上订单数据
  - 放 Redis
  - 返回给前端
  - 设置购物车/人数等线上态

这段逻辑里能看到：

- `order.setMid(-1 * request.getMerchantNo())`
- `order.setSid(-1 * request.getTerminalId())`
- `redisSecTemplatePlus.set(OrderRedisConstants.getCashPostOderKey(saasOrderKey), cashPostMsg, ...)`

说明它管理的是“线上订单态”，不是直接写门店本地 `dwd_bill`。

### 4.2 云端主动向门店发起“取单请求”

真正把线上订单推给门店的关键入口，是：

[OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java:1318)

`getOfflineOrderInfo(GetOfflineOrderDTO request)` 里会：

1. 生成一个 `msgId`
2. 注册 `monoResponsive` 回调
3. 调 `orderUtil.sendMessageToShop(...)`
4. 消息类型是 `OrderMsgTypeEnum.CASH_REQUEST`
5. 把 `billId / billName / tblId / openId / sid / msgId` 等数据发给门店

这说明云端到门店不是通过 Canal，而是通过“订单请求消息”。

数据载体大致是：

```text
GetOfflineOrderDTO
  ->
OrderMsgDTO(msgType=CASH_REQUEST, msgId, mid, sid, data=OrderDataDTO)
  ->
门店
```

## 5. 门店本地如何把线上订单落到本地订单表

### 5.1 门店收到请求后由 DoOrderHandler 处理

关键代码：

- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:496)

这里不是资料同步那种 JSON -> Entity -> Mapper 的通用更新链，而是门店订单业务逻辑：

1. 先取云端订单详情 `DoGetOrderVO`
2. 根据桌台、订单类型、是否已支付，判断是复用现有单还是新开本地单
3. 构造 `DwdBillCreateDTO`
4. 调 `dwdBillServicePlus.create(...)`
5. 再继续把线上订单内容映射到本地账单

可以直接看到的行为：

- 堂食单会查桌台 `PtTbl`
- 外卖/自提会新建 `DwdBillCreateDTO`
- 堂食新单会设置：
  - `tableLid`
  - `openId`
  - `orderSubType`
  - `pickNumber`
  - `otherOrderKey(order.getSaasOrderKey())`
- 然后调用：
  `dwdBillServicePlus.create(dwdBillCreateDTO, user)`

所以本地 `dwd_bill` 不是 Canal 落进去的，而是门店订单服务自己创建的。

### 5.2 本地单和线上单如何关联

从 `DoOrderHandler` 可以看到两组关键字段：

- `DwdBill::getSaasOrderKey`
- `DwdBill::getOtherOrderKey`

见：

- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:623)
- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:642)

这说明门店本地会用“线上订单号/外部订单号”去匹配或复用本地单，而不是直接把云端 `OrderBill` 表同步成 `DwdBill`。

## 6. 门店接单后还会回告云端

本地接单成功后，门店还会调用云端确认接口：

- [Nms4cloudUtil.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/util/Nms4cloudUtil.java:408)
- [Nms4cloudUtil.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/util/Nms4cloudUtil.java:443)

关键方法：

- `confirmSuccess(msgId, orderId, orgBillId, billName)`
- `confirmOrder(...)`

它会把这些数据回传给云端：

- `msgId`
- `orderId`
- `confirmStatus`
- `orgBillId`
- `billName`

所以订单专线是：

```text
云端发起取单请求
  ->
门店本地创建/匹配 dwd_bill
  ->
门店本地继续处理 dwd_food / 支付 / 打印 / KDS
  ->
门店回告云端接单结果
```

## 7. 当前最稳的订单专线图

```text
微信点餐前端
  ->
云端订单服务
  ->
云端 getOfflineOrderInfo()
  ->
OrderMsgDTO(msgType=CASH_REQUEST)
  ->
门店本地 DoOrderHandler
  ->
按桌台/订单类型查本地单或新建本地单
  ->
dwdBillServicePlus.create(...)
  ->
本地 dwd_bill
  ->
后续映射菜品、支付、打印、KDS
  ->
Nms4cloudUtil.confirmOrder(...)
  ->
回告云端
```

## 8. 修正后的最终说法

正确表述应该是：

- `Canal + Kafka + Netty + IncrementalSyncDataService` 这条链是**基础资料同步链**
- 微信点餐订单不是通过这条链进入门店本地库
- 微信点餐订单更接近：
  `云端订单服务 -> 订单请求消息 -> 门店本地 DoOrderHandler / DwdBillServicePlus -> 本地 dwd_bill / dwd_food`

如果继续往下深挖，下一步最值得查的是：

1. `orderUtil.sendMessageToShop(...)` 最终使用的是哪种通道
2. `DoOrderHandler` 后续把菜品明细写入 `dwd_food` 的具体调用链
3. `CASH_REQUEST` 之外是否还有专门的“订单变更”消息类型
