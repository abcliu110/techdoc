# 云端资料与微信点菜订单进入门店本地库的链路说明

## 1. 这份文档修正了什么

前面的版本把两条链路混在了一起：

1. 云端基础资料进入门店本地库
2. 微信点餐订单进入门店本地库

修正后的结论是：

- 云端基础资料：可以明确归到 `Canal + Kafka + Netty + 本地增量写库`
- 微信点餐订单：**不能写成 Canal 同步**
- 微信点餐订单应单独理解为：
  `云端订单服务 -> 门店订单请求处理 -> 本地 dwd_bill / dwd_food`

## 3. 微信点菜订单进入门店本地库

### 3.1 先纠正一个错误说法

错误说法：

```text
微信点菜订单
  ->
Canal
  ->
Kafka
  ->
Netty
  ->
IncrementalSyncDataService
  ->
本地 dwd_bill / dwd_food
```

当前代码证据不支持这个说法。

原因很直接：

- 本地增量同步白名单里没有 `dwd_bill`
- 也没有 `dwd_food`
- 也没有 `dwd_pay`

所以微信点餐订单不能继续写成这条链。

---

## 4. 微信点餐订单的更合理链路

### 4.1 云端订单服务先管理线上订单态

核心入口：

[OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java:1157)

`cashPostOrder(CashPostOrderDTO request)` 里能看到：

- 订单先以云端 `OrderBillAddDTO / OrderFoodAddDTO / OrderPayAddDTO` 的形式存在
- 会把 `mid/sid` 转成负值保存到线上订单域
- 会把整单 JSON 放进 Redis：
  `OrderRedisConstants.getCashPostOderKey(saasOrderKey)`
- 会设置扫码点餐相关的购物车/人数/支付模式

这一步更像：

```text
微信点餐前端
  ->
云端订单服务
  ->
线上订单态 / Redis / 订单服务域对象
```

### 4.2 云端再发起“向门店取单”的消息

关键入口：

[OrderBillServicePlus.java](D:/mywork/nms4cloud/nms4cloud-app/3_customer/nms4cloud-order/nms4cloud-order-service/src/main/java/com/nms4cloud/order/service/OrderBillServicePlus.java:1318)

`getOfflineOrderInfo(GetOfflineOrderDTO request)` 会：

1. 生成 `msgId`
2. 注册回调 `monoResponsive.put(...)`
3. 通过 `orderUtil.sendMessageToShop(...)` 发消息给门店
4. 消息体是：
   - `OrderMsgDTO`
   - `msgType = OrderMsgTypeEnum.CASH_REQUEST`
   - `data = OrderDataDTO`

`OrderDataDTO` 里带的核心数据包括：

- `billId`
- `billName`
- `tblId`
- `sid`
- `cardId`
- `openId`
- `pull`
- `id(msgId)`

所以这一步不是 Canal，而是“订单请求消息下发”。

## 5. 门店本地如何把线上订单变成本地订单

### 5.1 由 DoOrderHandler 接单

关键代码：

[DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:496)

这段逻辑非常关键，它做的是门店订单业务，而不是资料同步：

- 读取云端订单 `OrderBillVO`
- 判断是堂食、外卖还是自提
- 判断是否已经有本地单可复用
- 没有的话新建本地单
- 有的话复用当前本地单

### 5.2 如何新建本地单

`DoOrderHandler` 会构造 `DwdBillCreateDTO`，然后：

```java
DwdBill dwdBill = dwdBillServicePlus.create(dwdBillCreateDTO, user);
```

这说明本地 `dwd_bill` 是门店业务服务自己创建出来的，不是被 `IncrementalSyncDataService` 同步进去的。

创建时会把这些线上信息带进本地：

- `tableLid`
- `openId`
- `orderSubType`
- `pickNumber`
- `remark`
- `otherOrderKey(order.getSaasOrderKey())`

### 5.3 如何复用本地单

门店会按这些条件查现有本地单：

- `DwdBill::getSaasOrderKey`
- `DwdBill::getOtherOrderKey`
- `DwdBill::getTableLid`
- `DwdBill::getOpenId`

见：

- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:623)
- [DoOrderHandler.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/service/order/handler/DoOrderHandler.java:642)

这表示门店本地是在“匹配/创建本地单”，不是把云端订单表直接复制过来。

## 6. 订单接单后，门店还会回告云端

本地确认成功时会调用：

- [Nms4cloudUtil.java](D:/mywork/nms4pos/nms4cloud-pos2plugin/nms4cloud-pos2plugin-biz/src/main/java/com/nms4cloud/pos2plugin/util/Nms4cloudUtil.java:408)

核心方法：

- `confirmSuccess(msgId, orderId, orgBillId, billName)`
- `confirmOrder(...)`

回告内容包括：

- `msgId`
- `orderId`
- `confirmStatus`
- `orgBillId`
- `billName`

所以云端和门店之间的订单链是双向的：

```text
云端下发订单请求
  ->
门店创建/匹配本地单
  ->
门店回告接单结果
```

## 8. 修正后的最终结论

- 云端资料进入门店本地库：用的是 `Canal + Kafka + Netty + 本地增量写库`
- 微信点菜订单进入门店本地库：不能写成 Canal 同步
- 微信点菜订单更准确的描述应该是：
  `云端订单服务 -> 订单请求消息 -> 门店本地订单业务 -> dwd_bill / dwd_food`

后续如果继续深挖，最值得查的是：

1. `orderUtil.sendMessageToShop(...)` 最终走的是 MQTT、Netty 还是别的门店通道
2. `DoOrderHandler` 后续把菜品明细写入 `dwd_food` 的精确方法链
3. `CASH_REQUEST` 对应的门店消息接收器和调度入口
