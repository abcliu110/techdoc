## 一、总体现象

两次失败都发生在 WMS 模块构建阶段，失败模块都是：

```text
nms4cloud-wms-service
```

共同的直接失败点是 Maven 无法解析依赖：

```text
com.nms4cloud:nms4cloud-pos4cloud-feign
```

但两个日志的具体失败原因不同：

| 日志 | 使用版本 | 直接原因 |
| --- | --- | --- |
| `#57` | `0.0.1-20260507.071847-7` | 脚本把 SNAPSHOT 时间戳版本写进了依赖版本，Maven 按固定版本目录查找，找不到 |
| `#60` | `0.0.1-SNAPSHOT` | 当前脚本没有把依赖改成时间戳版本；失败更像是 Maven settings 没有显式声明可解析 SNAPSHOT 依赖的 repository/profile |

## 二、#57 日志失败原因

### 1. 直接报错

`#57 - all.txt` 中最终失败信息指向：

```text
Could not resolve dependencies for project com.nms4cloud:nms4cloud-wms-service
Could not find artifact com.nms4cloud:nms4cloud-pos4cloud-feign:jar:0.0.1-20260507.071847-7
```

也就是说，WMS 构建时使用了这个指定版本：

```text
0.0.1-20260507.071847-7
```

### 2. 技术原因

`0.0.1-20260507.071847-7` 是 Maven SNAPSHOT 解析出来的时间戳文件版本，不应该直接写进业务模块的 `pom.xml` 依赖版本。

Maven 正常解析 SNAPSHOT 时，业务 POM 里应该保持：

```xml
<version>0.0.1-SNAPSHOT</version>
```

或者保持项目原有写法：

```xml
<version>${project.version}</version>
```

Maven 会先读取：

```text
.../0.0.1-SNAPSHOT/maven-metadata.xml
```

再根据 metadata 找到真实文件名，例如：

```text
nms4cloud-pos4cloud-feign-0.0.1-20260507.071847-7.pom
nms4cloud-pos4cloud-feign-0.0.1-20260507.071847-7.jar
```

注意，这些真实文件仍然放在：

```text
.../0.0.1-SNAPSHOT/
```

而不是放在：

```text
.../0.0.1-20260507.071847-7/
```

如果脚本把依赖版本改成：

```xml
<version>0.0.1-20260507.071847-7</version>
```

Maven 会把它当作普通固定版本处理，去下面这种目录查找：

```text
.../com/nms4cloud/nms4cloud-pos4cloud-feign/0.0.1-20260507.071847-7/
```

远程检查 Nexus 时，这个固定版本目录下的 POM/JAR 返回的是 `404`。

所以 `#57` 的失败是脚本兜底逻辑方向错误：

```text
不能把 SNAPSHOT 时间戳文件名反写回业务 POM 的 version。
```

从 `192.168.1.119` 机器内部直接访问 Nexus，确认以下路径都可以访问：

metadata 当前指向的版本类似：

```text
0.0.1-20260507.095458-8
```

对应的时间戳 POM/JAR 在 `maven-public` 和 `maven-snapshots` 下都能访问：

```text
nms4cloud-pos4cloud-feign-0.0.1-20260507.095458-8.pom
nms4cloud-pos4cloud-feign-0.0.1-20260507.095458-8.jar
```

同时 Nexus API 显示：

```text

- maven-releases
```

所以 `#60/#61` 不能简单解释为：

```text
Nexus 中没有 nms4cloud-pos4cloud-feign
```

### 3. 已排除：不是 Jenkins Pod 本地缓存主因

远程连接 `192.168.1.119` 后查询 Kubernetes，确认 Jenkins 构建运行在临时 agent Pod 中。

`#60` 的 Pod 是：

```text
jenkinsfile-nms4cloud-60-c95qf-73lmp-bc7kv
```

Pod 模板关键内容：

```yaml
containers:
  - name: maven
    image: maven:3.9-eclipse-temurin-21
    volumeMounts:
      - mountPath: /var/jenkins_home
        name: jenkins-home
      - mountPath: /home/jenkins/agent
        name: workspace-volume

volumes:
  - name: jenkins-home
    persistentVolumeClaim:
      claimName: jenkins-pvc
  - name: workspace-volume
    emptyDir: {}
```

Maven 日志显示本地仓库写入的是：

```text
/root/.m2/repository
```

例如：

```text
Installing ... to /root/.m2/repository/com/nms4cloud/...
```

关键点：

1. 工作区 `/home/jenkins/agent` 是 `emptyDir`，Pod 删除后消失。
2. Maven 本地仓库是容器内 `/root/.m2/repository`。
3. 当前 Pod 模板没有把 `/root/.m2` 挂到 PVC。
4. 虽然 Pod 把 Jenkins PVC 挂到 `/var/jenkins_home`，但 Maven 没有使用 `/var/jenkins_home/.m2`。

因此，针对 `#60/#61`，Pod 本地 Maven 缓存不是主因。

`-U` 可以作为保险参数保留，但它不是这次问题的核心修复点。

### 4. 真正的高概率原因

当前运行脚本 `Jenkinsfile-nms4cloud-final copy` 生成的 Maven settings 只有 mirror 和 server：

```xml
<settings>
  <mirrors>
    <mirror>
      <id>nexus</id>
      <mirrorOf>*</mirrorOf>
      <url>${NEXUS_URL}/repository/${NEXUS_REPO_GROUP}/</url>
    </mirror>
  </mirrors>
  <servers>
    ...
  </servers>
</settings>
```

它没有显式声明一个启用 SNAPSHOT 的依赖仓库：

```xml
<repositories>
  <repository>
    <id>maven-snapshots</id>
    ...
    <snapshots>
      <enabled>true</enabled>
    </snapshots>
  </repository>
</repositories>
```

这就是 `#60/#61` 的核心风险点：

```text
settings.xml 只配置了“下载地址镜像”，没有配置“当前构建允许解析 SNAPSHOT 依赖的仓库策略”。
```

## 四、技术原因详解

### 1. Maven 的 mirror 不是 repository

Maven 里有两个容易混淆的概念：

```text
mirror      负责把某个仓库请求重定向到另一个 URL
repository 负责声明当前构建有哪些依赖仓库，以及这些仓库是否允许 SNAPSHOT
```

当前脚本里的：

```xml
<mirrorOf>*</mirrorOf>
```

含义是：

```text
所有仓库请求都改走 Nexus maven-public。
```

但它不会自动声明：

```text
当前构建启用了 SNAPSHOT 依赖解析。
```

Maven 默认内置的 `central` 仓库是 release 仓库，默认不解析 SNAPSHOT。即使 `central` 被 mirror 到了 `maven-public`，Maven 的仓库策略仍然可能来自原始 repository 定义，而不是 Nexus group 的实际成员配置。

所以仅有 mirror 不等价于：

```text
dependencies 可以解析内部 0.0.1-SNAPSHOT 依赖。
```

### 2. 为什么 curl 能访问，但 Maven 仍然失败

直接访问 Nexus：

```text
curl http://nexus.nexus:8081/repository/maven-public/com/nms4cloud/nms4cloud-pos4cloud-feign/0.0.1-SNAPSHOT/maven-metadata.xml
```

返回 `200`，只能证明：

```text
Nexus HTTP 路径可访问，制品文件存在。
```

但 Maven 解析 SNAPSHOT 依赖还需要满足：

1. 当前构建的 repository 列表里有可用于 SNAPSHOT 的仓库。
2. 该仓库的 `<snapshots><enabled>true</enabled></snapshots>` 生效。
3. Maven 能读取 `maven-metadata.xml` 并按 metadata 找到时间戳 POM/JAR。
4. mirror 映射没有让 SNAPSHOT 请求继承一个不允许 SNAPSHOT 的 repository 策略。

所以：

```text
curl 能拿到文件
```

和：

```text
Maven dependency resolution 能成功
```

不是同一个验证维度。

### 3. deploy 仓库和依赖下载仓库不是一回事

当前流水线 Maven 命令中有：

```text
-DaltSnapshotDeploymentRepository=nexus-snapshots::http://nexus.nexus:8081/repository/maven-snapshots/
```

这个参数只影响 deploy 阶段：

```text
本模块构建完成后上传到哪里。
```

它不影响依赖解析阶段：

```text
当前模块编译前从哪里下载依赖。
```

因此，日志里看到很多模块上传到了 `nexus-snapshots`，只能证明 deploy 配置有效，不能证明 dependency resolution 已经正确启用了 `maven-snapshots`。

本次失败发生在 `nms4cloud-wms-service` 编译前的依赖解析阶段，缺的是下载侧的 SNAPSHOT repository/profile。

### 4. 为什么其他 SNAPSHOT 看起来正常

日志中确实有很多内部模块和 `nexus-snapshots` 交互，例如：

```text
nms4cloud-starter-*
nms4cloud-platform-api
nms4cloud-biz-api
nms4cloud-bi-api
nms4cloud-wms-api
nms4cloud-wms-dao
```

但这些大多属于当前流水线前置阶段刚构建、刚安装、刚 deploy 的模块，或者发生在 deploy 生命周期中。

失败的依赖是：

```text
com.nms4cloud:nms4cloud-pos4cloud-feign:0.0.1-SNAPSHOT
```

它不是当前 WMS reactor 里正在构建的模块，也不是 `#60` 当前脚本前面刚构建出来的模块。它需要 Maven 在依赖解析阶段从远程 SNAPSHOT 仓库解析。

这类外部 SNAPSHOT 依赖最依赖 settings 中是否显式配置：

```text
repositories -> maven-snapshots -> snapshots enabled
```

### 5. #57 和 #60 的根因区别

`#57` 的问题是：

```text
脚本把 0.0.1-SNAPSHOT 解析后的时间戳文件版本，错误地写进 POM 依赖版本。
```

`#60/#61` 的问题是：

```text
当前脚本没有再写时间戳版本，但 settings.xml 没有显式启用 SNAPSHOT 依赖仓库。
```

两个问题都表现为 `pos4cloud-feign` 找不到，但技术链路不同。

## 五、推荐修复方案

### 1. 不要再改业务 POM 为时间戳版本

禁止把依赖版本改成：

```xml
<version>0.0.1-20260507.071847-7</version>
```

正确做法是保持：

```xml
<version>0.0.1-SNAPSHOT</version>
```

或保持项目原来的：

```xml
<version>${project.version}</version>
```

### 2. 当前 Jenkinsfile 生成的 settings.xml 增加 snapshots repository

推荐在当前脚本 `generateMavenSettings()` 里增加 profile：

```xml
<profiles>
  <profile>
    <id>nexus-snapshots-repo</id>
    <repositories>
      <repository>
        <id>maven-snapshots</id>
        <url>${NEXUS_URL}/repository/${NEXUS_REPO_SNAPSHOTS}/</url>
        <releases>
          <enabled>false</enabled>
        </releases>
        <snapshots>
          <enabled>true</enabled>
          <updatePolicy>always</updatePolicy>
        </snapshots>
      </repository>
    </repositories>
  </profile>
</profiles>

<activeProfiles>
  <activeProfile>nexus-snapshots-repo</activeProfile>
</activeProfiles>
```

### 3. mirror 建议排除 maven-snapshots

把 mirror 从：

```xml
<mirrorOf>*</mirrorOf>
```

改为：

```xml
<mirrorOf>*,!maven-snapshots</mirrorOf>
```

含义：

```text
普通 release 和第三方依赖继续走 maven-public；
maven-snapshots 按显式 repository 配置直接访问 snapshots 仓库；
避免 snapshots 仓库被 mirror 覆盖后继承不合适的 repository 策略。
```

### 4. Maven 参数可保留 `-U`

建议增加：

```text
-U
```

作用是强制刷新 SNAPSHOT 元数据。

但根据远程 Pod 检查，`#60/#61` 不是 Pod 本地 Maven 缓存主因，所以 `-U` 是保险项，不是核心修复。
