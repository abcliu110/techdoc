# 既有系统安全修改代码 AI 执行规范

## 1. 目标

本规范用于指导 AI 在维护既有系统、老系统、历史业务系统时安全修改代码。

最高优先级目标：

> 保留老业务逻辑。老逻辑无论看起来是否正确，只要线上或历史数据可能依赖它，AI 都必须先把它当成兼容契约，而不是当成错误主动修正。

AI 修改代码时，不能以“代码看起来不合理”“可以顺手优化”“我认为这是 bug”为理由直接改变旧行为。

## 2. 适用范围

适用于以下场景：

- 老系统功能增强。
- 历史接口维护。
- 已上线业务流程修改。
- 兼容老数据的新需求开发。
- 订单、支付、计费、权限、审批、库存、会员、营销、同步任务等高风险业务代码修改。
- 无完整测试覆盖、业务规则沉淀在代码中的系统。

不适用于以下场景：

- 明确要求废弃旧逻辑的重构任务。
- 明确确认旧行为是错误且允许修复的 bug。
- 全新模块且无历史兼容要求的开发。

## 3. AI 总原则

AI 必须遵守以下原则：

1. **老逻辑优先保留**
   - 老代码当前行为就是第一版事实。
   - 不要先判断它是否合理。
   - 不要主动“修正”历史分支。

2. **新逻辑受控进入**
   - 新逻辑必须通过明确边界接入。
   - 优先使用包装、分支、开关、扩展点。
   - 不允许把新规则随意插入旧逻辑深处。

3. **先保护，再修改**
   - 修改前必须尽量补充特征化测试或可重复验证用例。
   - 测试目标是锁定旧行为，不是证明旧行为正确。

4. **小范围、可回退**
   - 每次修改只解决一个明确目标。
   - 不混入无关格式化、重命名、大重构。
   - 代码必须便于回滚、关闭或隔离。

5. **证据优先**
   - 修改前要确认入口、调用方、数据形态、配置影响。
   - 不确定时通过搜索、日志、测试、数据库样例、接口样例获取证据。

## 4. 强制执行流程

AI 在修改既有系统代码前，必须按以下顺序执行。

### 4.1 确认旧行为

必须先回答：

- 当前入口在哪里？
- 当前老逻辑如何分支？
- 哪些接口、任务、页面、服务调用了它？
- 当前输入输出是什么？
- 是否存在历史数据、特殊枚举、空值、默认值、兼容字段？
- 新需求是否只影响一部分场景？

AI 不得在没有理解调用链和旧行为的情况下直接修改核心逻辑。

### 4.2 增加特征化测试或验证用例

只要条件允许，必须先添加或补充测试。

测试重点：

- 保留老输入对应的老输出。
- 保留老状态流转。
- 保留老字段兼容。
- 保留老异常或默认行为。
- 覆盖新逻辑的启用条件。
- 覆盖新逻辑不启用时仍走旧逻辑。

测试命名建议：

```text
should_keep_legacy_behavior_when_xxx
should_use_legacy_result_when_feature_disabled
should_not_change_legacy_status_mapping
should_apply_new_rule_only_when_xxx
```

如果项目不适合立即写自动化测试，AI 必须至少形成可重复的手工验证用例：

```text
场景：
前置数据：
请求参数：
旧逻辑预期：
新逻辑预期：
验证命令或接口：
```

### 4.3 选择安全接入方案

优先级从高到低：

1. 包装旧逻辑。
2. 新旧分支隔离。
3. 功能开关控制。
4. 扩展点或策略类。
5. 新增兼容接口或版本化接口。

禁止优先选择：

- 直接重写旧方法。
- 在旧方法中插入大量新条件。
- 顺手整理历史代码。
- 改旧字段含义。
- 改旧接口返回语义。

### 4.4 修改后验证

修改后必须验证：

- 老场景仍然走老逻辑。
- 新场景只在明确条件下生效。
- 功能开关关闭时，新逻辑不影响旧业务。
- 老接口字段、枚举、错误码、状态码保持兼容。
- 历史特殊逻辑有注释说明。

最终交付时，AI 必须说明：

```text
保留了哪些老逻辑：
新增了哪些新逻辑：
新逻辑启用条件：
是否有功能开关：
老接口是否兼容：
测试或验证结果：
仍存在的风险：
```

## 5. 方案一：特征化测试

### 5.1 定义

特征化测试是为了记录系统当前行为的测试。

它不判断老逻辑是否业务正确，只确认：

> 代码修改后，旧输入仍然得到旧结果。

### 5.2 AI 必须做什么

AI 必须优先为以下内容补特征化测试：

- 历史状态判断。
- 金额、折扣、积分、库存计算。
- 权限判断。
- 审批流转。
- 定时任务筛选条件。
- 老接口字段兼容。
- 空值、默认值、特殊枚举值处理。
- 历史数据兼容分支。

### 5.3 测试写法要求

测试中应明确写出“保持历史行为”的目的。

示例：

```java
@Test
void shouldKeepLegacyPendingStatusWhenPayTimeIsNull() {
  // 保持历史兼容：旧订单 status=3 且 payTime 为空时，仍按待确认处理。
}
```

### 5.4 无法写测试时的替代方案

如果无法写自动化测试，AI 必须提供手工验证用例。

示例：

```text
验证场景：旧订单待确认状态兼容
前置数据：订单 status=3，payTime=null
操作：调用订单详情接口
预期：返回展示状态为待确认
目的：确认新逻辑未改变旧订单展示行为
```

## 6. 方案二：包装旧逻辑

### 6.1 定义

包装旧逻辑是指不直接重写原有核心代码，而是在外层增加路由、适配或分支。

推荐结构：

```java
public Result handle(Request request) {
  if (newRuleEnabled(request)) {
    return newHandler.handle(request);
  }

  return legacyHandler.handle(request);
}
```

### 6.2 适用场景

适用于：

- 老方法复杂且缺少测试。
- 老逻辑被多个调用方复用。
- 新需求只影响部分客户、部分租户、部分状态或部分来源。
- 业务不能确认所有历史分支。

### 6.3 AI 必须遵守

- 不要在旧方法内部大量插入新逻辑。
- 不要删除旧分支。
- 不要改变旧方法默认返回。
- 新逻辑必须有明确启用条件。
- 旧逻辑入口必须仍然可被调用。

### 6.4 推荐命名

```text
LegacyXxxService
LegacyXxxHandler
XxxCompatService
NewXxxHandler
XxxRoutingService
```

命名必须让后续维护者能看出哪部分是历史兼容逻辑，哪部分是新逻辑。

## 7. 方案三：功能开关

### 7.1 定义

功能开关用于控制新逻辑是否生效。

目标：

- 默认不影响老业务。
- 可以灰度启用。
- 出问题可以快速关闭。

### 7.2 适用场景

必须优先考虑功能开关的场景：

- 计费、支付、订单、权限、库存、审批等核心链路。
- 影响多个客户或租户。
- 线上数据复杂，无法一次覆盖所有场景。
- 新逻辑需要逐步验证。
- 回滚成本高。

### 7.3 AI 实现要求

功能开关关闭时：

- 必须完全走旧逻辑。
- 不得改变旧返回。
- 不得写入新字段导致旧流程异常。

功能开关开启时：

- 只影响明确范围。
- 必须有日志或可观测信息。
- 必须可以定位命中新逻辑的请求或数据。

### 7.4 配置命名建议

```text
feature.xxx.enabled
feature.xxx.tenantIds
feature.xxx.customerScope
feature.xxx.grayPercent
```

不要使用含义模糊的名称：

```text
new.switch
flag1
test.enabled
```

## 8. 方案四：扩展点和策略类

### 8.1 定义

当新规则和旧规则并存，且未来可能继续增加规则时，可以使用扩展点或策略类。

推荐结构：

```text
XxxRule
LegacyXxxRule
TenantXxxRule
NewBusinessXxxRule
```

### 8.2 适用场景

适用于：

- 不同客户、租户、渠道、业务类型有不同规则。
- 后续预计会增加更多规则。
- 现有代码已经出现多处重复条件判断。
- 旧逻辑需要保留为一个独立策略。

### 8.3 AI 必须避免

- 只有一个新分支时过度抽象。
- 为了显得设计复杂而引入多层接口。
- 把所有逻辑都拆碎，导致调用链难以追踪。
- 把历史兼容逻辑隐藏到看不出语义的通用类里。

### 8.4 推荐判断

AI 可以使用以下判断：

```text
如果只是一个小条件：优先简单分支。
如果已有多个业务规则：可以策略化。
如果未来明确会继续增加规则：可以抽扩展点。
如果旧逻辑风险极高：先包装，再逐步策略化。
```

## 9. 方案五：接口向后兼容

### 9.1 基本要求

AI 修改既有接口时，必须保持向后兼容。

禁止随意修改：

- 字段名。
- 字段类型。
- 枚举 code。
- 状态码。
- 错误码。
- 默认值。
- 空值语义。
- 排序规则。
- 分页结构。
- 返回层级。

### 9.2 推荐做法

优先：

- 新增字段，不删除旧字段。
- 新增接口，不破坏旧接口。
- 新增版本，不直接改老版本。
- 老字段继续返回。
- 老枚举 code 保持不变。
- 新字段允许老调用方忽略。

版本化示例：

```text
/api/order/detail      保持旧逻辑
/api/order/detail/v2   新逻辑
```

### 9.3 AI 必须检查

修改接口前必须检查：

- 前端是否使用该字段。
- 其他后端服务是否调用该接口。
- 第三方系统是否依赖该接口。
- 文档、测试、示例请求是否需要同步。
- 是否存在历史客户端无法升级。

如果无法确认所有调用方，AI 应默认接口有隐藏依赖。

## 10. 方案六：区分 bug 修复和业务变更

### 10.1 基本判断

既有系统中，很多看似 bug 的逻辑可能是历史业务规则。

AI 必须先判断：

```text
这是明确技术错误？
这是历史业务兼容？
这是特定客户规则？
这是历史数据修补逻辑？
这是旧前端依赖的返回格式？
```

### 10.2 AI 不得擅自认定

AI 不得仅凭以下理由修改旧逻辑：

- 命名不合理。
- 分支看起来多余。
- 状态值不符合常识。
- 空值处理看起来奇怪。
- 老接口返回重复字段。
- 老代码风格不统一。
- 旧规则和新需求冲突。

### 10.3 推荐表达

如果要保留历史行为，提交说明或注释应使用：

```text
保留历史兼容行为
兼容旧接口调用方
兼容历史数据
新逻辑仅在 xxx 条件下启用
```

避免使用：

```text
修复错误逻辑
清理无用分支
优化旧代码
修正历史问题
```

除非业务已经明确确认旧行为可以改变。

## 11. 方案七：历史兼容注释

### 11.1 什么时候必须写注释

以下情况必须写注释：

- 保留了看起来不合理的旧逻辑。
- 新逻辑必须绕开某个历史分支。
- 某个字段为空、特殊枚举、特殊状态有历史含义。
- 不能改变某个返回字段或错误码。
- 某个客户、租户、历史数据依赖旧行为。

### 11.2 注释必须包含什么

注释应包含：

- 为什么保留。
- 影响哪些数据或调用方。
- 不要随意修改什么。
- 如果知道，写清楚历史原因或业务方确认信息。

### 11.3 注释模板

```java
// 保留历史兼容：<说明旧行为>。
// 影响范围：<历史数据/调用方/客户/接口>。
// 修改风险：不要改为 <看似更合理的新行为>，否则会影响 <具体后果>。
```

示例：

```java
// 保留历史兼容：老订单 status=3 且 payTime 为空时，前端仍按“待确认”展示。
// 影响范围：2022 年前导入订单和旧版 POS 客户端。
// 修改风险：不要改为标准支付状态判断，否则旧订单详情展示会变化。
```

## 12. 禁止事项

AI 在维护既有系统时禁止：

- 未加验证就直接修改核心业务逻辑。
- 因为旧逻辑“不合理”而主动修正。
- 把新需求和大重构混在一起。
- 删除历史分支但没有证据证明无人依赖。
- 修改老接口字段含义。
- 修改老枚举 code 含义。
- 修改老错误码或状态码含义。
- 把新逻辑默认全量开启。
- 在旧逻辑中堆叠大量无注释的新条件。
- 将“兼容历史行为”的修改描述成普通 bug 修复。

## 13. AI 修改代码前检查清单

AI 在动代码前必须自检：

```text
[ ] 我已经理解旧逻辑当前行为。
[ ] 我已经确认入口和主要调用方。
[ ] 我已经识别新逻辑的启用条件。
[ ] 我没有打算主动修正未确认的历史行为。
[ ] 我已经考虑特征化测试或手工验证用例。
[ ] 我已经选择包装、开关、扩展点或兼容接口方案。
[ ] 我知道哪些接口字段、枚举、状态不能改。
[ ] 我知道哪些地方需要写历史兼容注释。
```

## 14. AI 修改代码后检查清单

AI 完成修改后必须自检：

```text
[ ] 老场景仍然走旧逻辑。
[ ] 新场景只在明确条件下走新逻辑。
[ ] 功能开关关闭时行为不变。
[ ] 老接口响应结构保持兼容。
[ ] 老枚举 code、错误码、状态码未被修改。
[ ] 历史兼容逻辑已有注释。
[ ] 已执行相关测试或手工验证。
[ ] 最终说明中列出了保留的老逻辑和剩余风险。
```

## 15. 推荐交付说明模板

AI 完成任务后，最终回复或提交说明应包含：

```text
变更目标：

保留的老逻辑：

新增的逻辑：

新逻辑启用条件：

兼容措施：

测试或验证：

未覆盖风险：
```

## 16. 决策规则

当 AI 不确定是否可以改变旧行为时，按以下规则处理：

```text
能保留旧行为，就保留旧行为。
能新增分支，就不要改旧分支。
能加开关，就不要默认全量启用。
能新增字段，就不要改旧字段。
能新增接口版本，就不要破坏老接口。
能写特征化测试，就先写测试。
必须保留但看起来奇怪的逻辑，要写注释。
```

## 17. 一句话原则

维护既有系统时，AI 的任务不是证明旧逻辑正确，而是确保新修改不会无意改变旧业务。

## 一、必须修改（高优先级——可能引发 AI 误判或系统事故）

### 问题 1：缺乏"不确定时的升级机制"，AI 可能擅自做主

**位置**：第 4.1 节（确认旧行为）、第 10 节（区分 bug 修复和业务变更）

**问题描述**：文档要求 AI"证据优先"（第 3.5 节），但在 AI 穷尽搜索手段后仍无法确认旧行为性质时，没有任何升级路径。规范中没有出现过"需要人类确认""请咨询业务方""标记待确认"等表达。这会导致两种风险：
- AI 过自信：在证据不足时自行判断为"可改"或"应改"，而实际上影响了隐藏依赖。
- AI 过保守：将所有不理解的代码都标记为"保留"，导致合理需求无法推进。

**补充建议**（直接在第 4.1 节末尾增加）：

```markdown
### 4.1.1 不确定时的处理

如果通过代码搜索、日志分析、数据库样例、接口测试等手段仍无法确认：
- 该逻辑是否被任何调用方依赖。
- 该分支是否处理了某种历史数据状态。
- 该字段是否被前端或其他服务使用。

则 AI 必须：
1. 在回复中明确标注为「未确认项」，说明已尝试的验证手段和仍存疑的内容。
2. 不得基于不确定的假设直接修改代码。
3. 请求人类确认：提供具体问题（如："订单表中是否存在 status=3 且 payTime 为 null 的历史数据？"），并等待明确回复。
4. 如果时间紧迫，可在保留旧行为的前提下，提供「保守方案」（保留+开关）和「进取方案」（清理+重构）两种选项，由人类选择。
```

### 问题 2："特征化测试"的强制性表述模糊

**位置**：第 4.2 节标题为"必须先添加或补充测试"，但第 5.1 节的定义中写的是"只要条件允许，必须先添加或补充测试"（第 76 行），前后措辞不一致。"条件允许"的边界未定义，AI 可以将任何困难都解释为"条件不允许"，从而绕过测试要求。

**补充建议**：在第 5.4 节增加更明确的分级要求：

```markdown
### 5.5 测试充分性判断

AI 应按以下顺序判断测试是否充分：

1. **核心业务逻辑**（订单金额、支付状态、权限判断、库存扣减、计费规则）：
   - 必须有自动化测试覆盖。
   - 即使无法写自动化测试，也必须有详细的手工验证用例并记录结果。
   - 无法验证时，该修改不得执行。

2. **辅助业务逻辑**（日志、通知、统计、筛选条件）：
   - 优先自动化测试。
   - 可接受详细的手工验证用例。

3. **低风险变更**（仅修改文档、配置默认值、注释）：
   - 至少通过代码审查确认无副作用。
   - 手工验证用例可选。

AI 不得以「项目没有测试框架」「现有代码不友好」「时间有限」为由跳过核心业务逻辑的验证。如果确实无法满足，应将修改拆分为多次低风险提交，每次提交后验证无误再进行下一步。
```

### 问题 3：未定义"老逻辑"的范围边界

**位置**：第 1 节（目标）、第 10 节（区分 bug 修复和业务变更）

**问题描述**："只要线上或历史数据可能依赖它"这个条件中的"可能"没有量化标准。AI 无法区分以下几种情况：
- 确实存在线上数据依赖（如：历史订单中确实有 `pay_time = null` 的记录）
- 理论上可能存在但尚未发现证据
- 理论上不可能存在（如：新建系统的某个从未被调用的分支）

这会导致两种误用：
- 过度保留：将任何看似多余的代码都保留，导致技术债务无限累积
- 过度激进：AI 自行判断"线上不可能有这种情况"而改掉旧逻辑

**补充建议**：在第 1 节目标段落后增加定义：

```markdown
### 1.1 术语定义

**老逻辑（Legacy Logic）**：指在本规范适用范围内，系统当前运行的、尚未通过明确废弃流程宣布废止的代码行为。包括但不限于：
- 已有线上数据依赖的业务分支（通过数据库查询、日志分析、接口调用记录等可证实）
- 被多个调用方引用的公共方法
- 接口返回中携带的字段，无论其当前是否有调用方使用（因可能存在隐藏依赖）
- 处理空值、默认值、特殊枚举值的分支

**疑似冗余逻辑**：指代码中存在但无法通过上述方式证实有实际依赖的分支。对疑似冗余逻辑的处理方式见第 10.4 节。

**明确错误（Confirmed Bug）**：指同时满足以下条件的旧行为：
1. 有明确的技术错误证据（非仅"看起来不合理"）
2. 业务方已书面/口头确认可以修改
3. 不存在依赖该行为的历史数据或调用方，或依赖方已确认可以接受变更

以下情况**不属于**明确错误：
- 仅因命名、代码风格、架构不符合 AI 审美
- 仅因 AI 认为"新逻辑更合理"
- 仅因与新需求的设计方向不一致
```

### 问题 4：第 8.4 节的"过度抽象"判断标准缺乏可操作性

**位置**：第 8.4 节（推荐判断）

**问题描述**："只有一个新分支时优先简单分支"——问题是 AI 对"多少算一个分支"的判断非常主观。例如，AI 可能将 10 行 if-else 误判为"只是一个分支"而选择策略模式，也可能将一个分支误判为"已有多个业务规则"而过度抽象。缺少量化的或可观测的判断标准。

**补充建议**（替换第 8.4 节末尾的推荐判断部分）：

```markdown
### 8.5 抽象程度判断标准

AI 应按以下标准选择实现方式：

| 场景 | 推荐方式 |
|------|---------|
| 仅有一个新条件，且该条件未来不会变化 | 简单 if/else 分支，直接在调用处处理 |
| 有 2-3 个规则，且未来可能增加 | 策略类（Strategy Pattern），每个策略独立实现 |
| 有 3+ 个规则，且按租户/客户/渠道分流 | 扩展点（Extension Point），支持注册机制 |
| 旧逻辑风险极高，无法一次性验证 | 包装旧逻辑 + 功能开关，逐步灰度 |

**触发过度抽象的信号**（AI 应主动避免）：
- 引入 2+ 层新接口/抽象类，但实际只有 1-2 个实现
- 策略类中只有一个 `LegacyXxxStrategy` 而其他 slot 为空
- 为了抽象而抽象，导致调用链需要追踪 3+ 个文件才能理解
- 新增的抽象层比原有代码本身还复杂

**触发过度简单的信号**（AI 也应主动避免）：
- 在一个方法中出现 10+ 个 if-else 分支
- 相同判断逻辑在 3+ 个不同位置重复
- 开关逻辑散布在多个方法中，没有统一管理
```

## 二、建议修改（中优先级——规范完整性不足，但不至于导致严重事故）

### 问题 5：缺少数据库 Schema 变更的处理指引

**位置**：整篇文档未涉及数据库层面

**问题描述**：规范几乎完全聚焦于代码层，未涉及：
- ALTER TABLE 添加 nullable 字段 vs NOT NULL 字段的风险
- 枚举字段扩增 vs 修改已有 code 值
- 删除旧字段的时机和风险
- 索引变更对查询行为的影响
- 数据库迁移脚本与代码部署的顺序

这些场景在 CRM、ERP 等业务系统中极为常见，且影响往往是隐式和延迟的（代码行为没变，但查询性能变了）。

**补充建议**：在第 9 节（接口向后兼容）后增加一节：

```markdown
## 9B. 方案五（补充）：数据库变更兼容

### 9B.1 禁止事项

- 禁止直接修改已有枚举字段的 code 值（如将 `status=1` 从"待支付"改为"已取消"）
- 禁止在新增 NOT NULL 字段时不提供默认值（除非已确认所有写入路径都已更新）
- 禁止在删除旧字段前不确认代码中已无引用
- 禁止使用 DROP COLUMN 而不使用迁移工具记录

### 9B.2 推荐做法

**扩增枚举**：
```sql
-- 错误：修改已有 code
UPDATE order_status SET code = 5 WHERE name = '已取消';

-- 正确：新增 code
ALTER TABLE order_status ADD COLUMN new_code INT;
UPDATE order_status SET new_code = 5 WHERE name = '已取消';
```

**新增字段**：
- 优先新增 nullable 字段，提供合理的默认值
- 必须更新所有写入路径后再改为 NOT NULL
- 变更期间通过功能开关控制新旧字段的读取逻辑

**删除字段**：
- 阶段一：确认代码中已无引用，在字段上增加 `@Deprecated` 注释（如果有 ORM 注解支持）
- 阶段二：至少一个发布周期后，确认无异常，再执行物理删除
- 如有迁移脚本，必须先在测试环境验证对所有业务场景的影响

### 9B.3 检查清单

```text
[ ] 新增字段是否为 nullable 或已提供默认值？
[ ] 是否有历史数据无法满足 NOT NULL 约束？
[ ] 所有写入路径是否都已更新？
[ ] 枚举字段是否只扩增而未修改已有 code？
[ ] 是否已确认代码中无对该字段的引用？
[ ] 是否需要在迁移前备份该字段数据？
```

### 问题 6：功能开关缺乏生命周期管理

**位置**：第 7 节（功能开关）

**问题描述**：规范定义了功能开关的用途和命名，但缺少：
- 开关上线后何时可以删除或永久关闭
- 多个开关并存时的优先级冲突处理
- 开关配置如何同步到所有实例（尤其是在容器化/K8s 环境中）
- 开关是否需要纳入版本控制

**补充建议**（在第 7.4 节末尾增加）：

```markdown
### 7.5 开关生命周期管理

**上线流程**：
1. 开关默认关闭（`enabled = false`），全量走旧逻辑
2. 在测试环境验证开关开启后的行为
3. 在 staging/预发环境用真实数据灰度验证
4. 线上按租户或用户 ID 灰度开启
5. 确认稳定后全量开启

**稳定后处理**：
- 开关全量开启后，应在代码中保留对旧逻辑的路径（通过 `if (legacy)` 判断），不要直接删除旧逻辑
- 在下一个主版本发布时，可以将旧路径标记为 `@Deprecated`，再下一个版本删除
- 开关配置本身应纳入版本控制（如 `feature-flags.yaml`），而非仅存于数据库或配置中心

**禁止事项**：
- 开关开启后立即删除旧逻辑代码
- 开关配置不记录在版本控制中
- 多个开关控制同一逻辑时未定义优先级
```

**补充建议**（扩展第 10.2 节）：

```markdown
### 10.2 AI 不得擅自认定（扩展）

AI 不得仅凭以下理由修改旧逻辑（补充）：

- 注释说明了旧逻辑的历史原因，但注释可能已过时（应以代码实际行为为准，注释过时不等于行为可改）

- 该接口从未被发现被调用（可能通过代理、网关、动态路由被隐藏调用）
- 日志中未出现该代码路径被执行（可能因为边界条件在生产中罕见但确实存在）

## 三、可选优化（低优先级——锦上添花）

### 建议 A：增加决策规则的优先级排序

**位置**：第 16 节（决策规则）

当前 7 条规则按相同权重列出，AI 在遇到冲突时可能无法取舍。建议增加优先级：

```markdown
## 16. 决策规则（带优先级）

当规则之间冲突时，按从上到下优先级处理：

| 优先级 | 规则 |
|--------|------|
| P0 | 保留旧行为（任何不确定时，默认保留） |
| P0 | 不确定的未决项必须升级，不得自行决定 |
| P1 | 新增分支优先于修改旧分支 |
| P1 | 功能开关默认关闭，不得默认全量启用 |
| P2 | 简单分支优先于过度抽象 |
| P2 | 新增接口/字段优先于修改旧接口/字段 |
| P3 | 特征化测试优先于口头说明 |
| P3 | 注释必须写清楚"为什么保留"而非"做了什么" |

### 建议 B：增加多 AI 协作场景的处理

**位置**：整篇文档

如果团队中可能有多个 AI 实例同时修改同一模块，规范中缺少：
- 多个 AI 修改同一文件的冲突检测流程
- AI 修改前应检查 git recent commits 确认近期是否已有相关变更
- AI 在生成 patch 前应运行 `git status` 确认工作区状态

**补充建议**（在第 4 节强制执行流程开头增加）：

```markdown
### 4.0 修改前环境检查

在开始任何修改前，AI 必须：

1. 运行 `git status` 确认工作区干净或了解当前未提交状态
2. 运行 `git log --oneline -10` 确认近期是否已有相关修改（避免重复或冲突修改）
3. 检查是否有其他 AI 实例或人类开发者在修改相关模块（通过 git 分支、任务分配系统等确认）
4. 如果存在并行修改风险，先与相关方协调，或将修改范围限制在不冲突的区域

### 建议 C：增加对"行为一致但实现方式劣化"的识别

**位置**：第 3 节（AI 总原则）

规范目前聚焦于"改变旧行为"的风险，但未覆盖另一种常见风险：AI 以"不改变行为"为名，将旧逻辑用更简洁/更现代的方式重写，但重写过程中引入了细微差异（如浮点数精度、边界条件判断顺序、异常类型等）。

**补充建议**（在第 3 节末尾增加）：

```markdown
### 3.6 禁止等价重写

AI 不得以"保持行为不变"为由，对旧逻辑进行重写性修改，除非同时满足：
1. 有完整的特征化测试覆盖（自动化测试必须全部通过）
2. 业务方明确确认新实现与旧实现等价
3. 重写拆分为独立提交，不与功能修改混在一起

常见等价重写陷阱：
- 用 `BigDecimal.divide()` 替代整型除法：即使结果在大多数场景下一致，精度处理可能不同
- 用 Stream API 重写循环：边界条件、异常处理顺序可能不同
- 用 `Objects.requireNonNull()` 替代手写判空：抛出的异常类型不同
- 用 `Optional` 重写 null 判断：短路行为可能不同
```

### 建议 D：增加配置变更的处理

**位置**：整篇文档

规范专注于代码，但业务系统中的配置（如 Nacos 配置、Apollo 配置、环境变量）变更同样可能破坏旧行为，且更难追踪。

**补充建议**（在第 12 节禁止事项末尾增加）：

```markdown
### 12B. 配置变更禁止事项

- 禁止在配置变更时假设所有服务实例同时生效（考虑滚动发布窗口期）
- 禁止修改配置值的语义（如将 `timeout=30` 的单位从秒改为毫秒）
- 禁止删除旧配置 key 而不确认无引用
- 禁止将旧配置的默认值改为不同值（即使 key 不变）
- 配置变更必须记录变更原因和回滚方案
```

## 四、评审总结

### 是否建议作为 AI 维护老系统的执行规范使用？

**结论：建议使用，但需先解决以下 3 个必须修改的问题。**

### 通过标准

| 评估维度 | 评分（1-5） | 说明 |
|---------|-----------|------|
| 对 AI 的可执行性 | 4 | 条款清晰，但部分判断标准缺乏量化 |
| 对老业务逻辑的保护力度 | 3.5 | 方向正确，但"不确定时"的处理缺失 |
| 防止误改的能力 | 3.5 | 禁止事项全面，但边界条件覆盖不足 |
| 防止漏验的能力 | 3 | 特征化测试要求有歧义，数据库变更完全缺失 |
| 防止过度抽象的能力 | 3 | 判断标准主观，难以稳定执行 |
| 防止默认改旧行为的能力 | 4 | 决策规则和禁止事项覆盖较好 |
| 可操作性 | 4 | 清单和模板实用，流程清晰 |

### 必须修改项（使用前必须解决）

1. **增加不确定时的升级机制**（问题 1）——缺失此机制会导致 AI 在关键判断上独自做主，后果不可控
2. **明确特征化测试的强制性分级**（问题 2）——当前措辞允许 AI 规避核心验证
3. **定义"老逻辑"的范围边界**（问题 3）——"可能依赖"的无边界解读会摧毁规范的可操作性

### 建议修改项（使用中逐步完善）

4. 增加数据库 Schema 变更指引（问题 5）
5. 增加功能开关生命周期管理（问题 6）
6. 扩展"不得擅自认定"的理由清单（问题 7）
