# 世界级 UI 模型设计常用方法与底层思想

## 1. 总体原则

UI 设计不是把字段摆到页面上，也不是把功能入口堆成菜单。优秀的 UI 模型要表达用户任务、业务对象、状态变化、操作路径和反馈机制，让用户能以最低认知成本完成高价值工作。

世界级 UI 模型通常追求四个目标：

1. **任务清晰**：用户知道自己在哪里、能做什么、下一步是什么。
2. **业务可解释**：页面能解释对象、状态、金额、异常和历史过程。
3. **操作高效**：高频任务路径短，批量任务可控，异常处理有出口。
4. **系统可演进**：页面、组件、权限、状态和接口契约能支撑业务扩展。

底层思想：

> UI 不是视觉包装，而是业务模型、用户任务和系统能力的可视化表达。

## 2. 从用户任务设计，而不是从数据库字段设计

### 方法

先识别用户要完成的任务：

- 用户是谁。
- 他为什么进入系统。
- 他要判断什么。
- 他要操作什么。
- 他需要哪些信息才能决策。
- 操作成功后他要看到什么结果。

再决定页面、字段、按钮、状态和反馈。

### 为什么这样设计

用户不是为了填写数据库字段而使用系统，而是为了完成业务任务。数据库表是技术结构，任务才是用户心智结构。

### 解决的问题

- 避免一张表一个页面。
- 避免字段很多但用户不知道怎么做。
- 避免页面割裂，用户需要跨页面拼业务上下文。
- 避免前端暴露技术概念。

### 可能带来的问题

- 需要更多业务分析。
- 后端可能需要提供聚合接口。
- 页面不再和数据库表一一对应，实现成本更高。

### 蕴含思想

> 页面服务任务，不服务表结构。

## 3. 围绕业务对象组织页面

### 方法

先识别核心业务对象：

- 订单。
- 客户。
- 商品。
- 会员。
- 库存。
- 支付单。
- 审批单。
- 报表指标。

页面围绕对象组织：

- 列表页用于发现和筛选对象。
- 详情页用于理解对象和追踪历史。
- 编辑页用于创建或修改对象。
- 操作弹窗用于完成局部动作。
- 工作台用于聚合待办、异常和指标。

### 为什么这样设计

业务对象是用户理解系统的稳定锚点。用户需要知道“我现在处理的是哪张订单、哪个会员、哪笔库存、哪个客户”。

### 解决的问题

- 页面语义混乱。
- 用户不知道当前操作影响哪个对象。
- 多个功能入口围绕同一对象重复建设。
- 详情、日志、明细、状态分散。

### 可能带来的问题

- 详情页可能变复杂。
- 需要设计对象聚合视图。
- 后端需要聚合对象相关数据。

### 蕴含思想

> 对象是业务世界的名词，页面应该围绕名词组织。

## 4. 列表页服务发现、筛选和批量处理

### 方法

列表页应重点设计：

- 对象识别字段。
- 关键状态。
- 高频筛选条件。
- 排序方式。
- 批量操作。
- 行级操作。
- 异常标识。
- 汇总信息。

### 为什么这样设计

列表页的主要价值不是展示所有字段，而是帮助用户快速找到目标对象、判断状态、执行下一步。

### 解决的问题

- 列太多，用户找不到重点。
- 筛选项不对应真实工作方式。
- 批量操作没有风险控制。
- 状态和异常不明显。

### 可能带来的问题

- 需要取舍字段。
- 不同角色可能需要不同列表视图。
- 需要支持列配置或保存筛选条件。

### 蕴含思想

> 列表页不是表格展示器，而是对象发现和任务入口。

## 5. 详情页服务理解、追溯和决策

### 方法

详情页应包含：

- 主对象摘要。
- 当前状态。
- 关键金额或数量。
- 明细。
- 上下游关联。
- 操作记录。
- 审批记录。
- 流水或事件。
- 当前可执行动作。

### 为什么这样设计

复杂业务对象不能只展示静态字段。用户需要理解当前对象为什么是这个状态，下一步能做什么，过去发生了什么。

### 解决的问题

- 用户无法解释业务结果。
- 出问题时无法追溯。
- 操作按钮和对象状态脱节。
- 需要跨多个页面查历史。

### 可能带来的问题

- 信息量大。
- 首屏设计要求高。
- 后端详情接口复杂。

### 蕴含思想

> 详情页不是字段集合，而是对象的当前事实和历史证据。

## 6. 首屏表达最重要的业务判断

### 方法

首屏优先展示：

- 当前对象是谁。
- 当前处于什么状态。
- 是否异常。
- 关键金额、数量或进度。
- 用户下一步最可能做什么。

次要信息折叠或分区展示。

### 为什么这样设计

用户进入页面后的前几秒，必须建立判断。复杂系统如果首屏信息没有重点，用户会被迫阅读大量字段。

### 解决的问题

- 用户不知道页面重点。
- 关键信息被埋在下方。
- 高频操作路径长。
- 状态和异常不显眼。

### 可能带来的问题

- 需要区分主次信息。
- 不同角色对“重要”的定义不同。

### 蕴含思想

> 首屏不是装饰区域，而是决策区域。

## 7. 状态决定动作

### 方法

根据业务状态展示合法动作：

- 草稿：编辑、提交、删除。
- 待审核：审核、驳回。
- 已审核：下推、反审、关闭。
- 已支付：退款、开票、查看流水。
- 已关闭：查看、复制、归档。

禁止展示当前状态下不可执行的主操作。

### 为什么这样设计

状态代表对象生命周期，动作代表状态转换。UI 必须和状态机一致，否则用户会看到能点但不能做的按钮。

### 解决的问题

- 按钮可见但后端拒绝。
- 用户不知道为什么不能操作。
- 非法操作产生异常数据。
- 前后端状态逻辑重复且不一致。

### 可能带来的问题

- 前端需要状态和动作模型。
- 后端最好返回当前可执行动作。
- 权限、状态、业务规则需要统一判断。

### 蕴含思想

> 操作不是按钮集合，而是状态机的可视化。

## 8. 危险动作必须解释影响范围

### 方法

对高风险动作提供确认和影响说明：

- 删除。
- 作废。
- 反审。
- 反结账。
- 退款。
- 冲正。
- 批量修改。
- 关闭流程。

确认内容应说明：

- 会影响哪些数据。
- 是否可撤销。
- 是否影响报表。
- 是否生成逆向流水。
- 是否通知其他系统。

### 为什么这样设计

企业系统的操作会影响财务、库存、权益、审批、报表和外部系统。危险动作不能只用“确定吗”。

### 解决的问题

- 误操作。
- 用户不知道后果。
- 异常操作无法追责。
- 业务回滚成本高。

### 可能带来的问题

- 确认流程增加操作步骤。
- 过多确认会造成麻木。
- 需要区分真正危险和普通动作。

### 蕴含思想

> 高风险操作不是交互问题，而是责任问题。

## 9. 高频任务要短路径，低频任务要受控

### 方法

按任务频率和风险设计入口：

- 高频低风险：入口明显，步骤少。
- 高频高风险：入口明显，但确认严格。
- 低频低风险：收敛到更多操作。
- 低频高风险：隐藏在受控流程中，有权限和确认。

### 为什么这样设计

不同任务不应拥有同等视觉权重。企业系统需要同时追求效率和安全。

### 解决的问题

- 页面按钮过多。
- 高频任务路径太长。
- 危险操作太容易误触。
- 低频功能干扰主要工作。

### 可能带来的问题

- 需要真实使用数据或业务判断。
- 需要按角色区分任务频率。

### 蕴含思想

> 入口权重应该由任务频率和业务风险共同决定。

## 10. 信息分组应符合业务心智

### 方法

字段分组不按数据库顺序，而按业务理解分组：

- 基本信息。
- 交易信息。
- 金额信息。
- 收付款信息。
- 发货/库存信息。
- 权益/优惠信息。
- 审批信息。
- 操作记录。

### 为什么这样设计

用户理解业务时会按主题组织信息，而不是按字段创建顺序或表结构。

### 解决的问题

- 表单难读。
- 字段关系不清。
- 重要信息分散。
- 用户需要自己拼业务含义。

### 可能带来的问题

- 需要重新组织后端返回结构。
- 不同业务角色可能需要不同分组。

### 蕴含思想

> 信息架构应该匹配人的理解方式，而不是数据库存储方式。

## 11. 表单设计服务最小业务事实

### 方法

创建表单只要求用户输入创建对象所必需的最小业务事实。其他字段通过默认值、后续补充、系统计算或配置规则处理。

设计时区分：

- 必填字段。
- 可选字段。
- 系统计算字段。
- 后续流程字段。
- 只在特定类型下出现的字段。

### 为什么这样设计

表单越长，错误越多，完成率越低。很多字段并不属于创建动作，而属于后续流程或扩展信息。

### 解决的问题

- 创建成本高。
- 用户不知道哪些字段重要。
- 低频字段干扰高频录入。
- 表单校验复杂。

### 可能带来的问题

- 后续补充信息需要流程支持。
- 默认值和自动计算必须可靠。
- 过度简化可能导致后续返工。

### 蕴含思想

> 表单不是字段收集器，而是业务对象创建入口。

## 12. 报表 UI 围绕经营问题设计

### 方法

先定义报表要回答的问题：

- 今天卖得怎么样？
- 哪些门店异常？
- 哪些商品毛利低？
- 哪些客户贡献高？
- 库存是否健康？
- 退款是否异常？

再设计：

- 核心指标。
- 时间范围。
- 分析维度。
- 趋势图。
- 分布图。
- 明细钻取。
- 导出和口径说明。

### 为什么这样设计

报表不是把数据表展示出来，而是帮助管理者判断业务问题。

### 解决的问题

- 指标堆叠但没有结论。
- 用户不知道看什么。
- 同一指标口径不一致。
- 无法从汇总追溯到明细。

### 可能带来的问题

- 需要统一指标口径。
- 需要数据仓库或报表模型支持。
- 需要兼顾实时性和准确性。

### 蕴含思想

> 报表的核心不是展示数据，而是回答问题。

## 13. 工作台聚合待办、异常和关键指标

### 方法

工作台不应做成入口大全，而应聚合：

- 待处理任务。
- 异常提醒。
- 今日关键指标。
- 高频快捷动作。
- 个人关注对象。
- 风险预警。

### 为什么这样设计

用户进入系统时，最关心的是“我今天要处理什么”和“哪里出问题了”。

### 解决的问题

- 用户每天从菜单里找任务。
- 异常分散在多个模块。
- 管理者看不到关键经营状态。
- 高频动作没有统一入口。

### 可能带来的问题

- 需要跨模块聚合数据。
- 不同角色工作台差异大。
- 指标实时性和权限过滤更复杂。

### 蕴含思想

> 工作台不是导航页，而是任务和风险的聚合页。

## 14. 错误提示必须可操作

### 方法

错误提示应包含：

- 发生了什么。
- 为什么失败。
- 用户能做什么。
- 是否可以重试。
- 是否需要权限或人工处理。
- 影响范围是什么。

### 为什么这样设计

企业系统中的错误往往意味着业务被阻断。只提示“操作失败”无法帮助用户恢复流程。

### 解决的问题

- 用户不知道如何处理。
- 客服、实施、研发排查成本高。
- 同类错误反复出现。
- 用户误以为系统不可用。

### 可能带来的问题

- 后端需要提供结构化错误码和原因。
- 敏感错误不能暴露过多内部信息。
- 错误文案需要维护。

### 蕴含思想

> 好的错误提示不是通知失败，而是提供恢复路径。

## 15. 空状态要引导下一步

### 方法

空状态应说明：

- 为什么这里没有数据。
- 用户可以做什么。
- 是否需要调整筛选条件。
- 是否需要创建第一条数据。
- 是否没有权限。
- 是否等待同步。

### 为什么这样设计

空页面是用户体验的重要场景。空白或“暂无数据”不能解释原因，也不能推动任务继续。

### 解决的问题

- 用户不知道是没数据、没权限、筛选错还是系统错误。
- 新用户不知道怎么开始。
- 同步延迟被误解为 bug。

### 可能带来的问题

- 需要识别不同空状态原因。
- 不同角色的引导不同。

### 蕴含思想

> 空状态也是状态，必须被设计。

## 16. 反馈要覆盖同步和异步过程

### 方法

对操作提供完整反馈：

- 点击后处理中。
- 成功后结果明确。
- 失败后原因明确。
- 异步任务可查看进度。
- 长任务可离开页面。
- 批量任务有成功数、失败数和失败明细。

### 为什么这样设计

企业系统大量操作不是瞬时完成，例如导入、导出、同步、审核、批量处理、报表生成。

### 解决的问题

- 用户重复点击。
- 不知道任务是否开始。
- 长任务中断后无法追踪。
- 批量失败只能重试全部。

### 可能带来的问题

- 需要任务中心或异步记录。
- 后端需要任务状态模型。
- UI 需要处理更多状态。

### 蕴含思想

> 用户需要看到系统正在工作，而不是猜系统是否工作。

## 17. 权限和数据范围要可解释

### 方法

权限设计要在 UI 中清楚表达：

- 没有入口。
- 可见但不可操作。
- 操作被拒绝时说明缺少什么权限。
- 数据范围被过滤时说明范围。
- 敏感字段脱敏或隐藏。

### 为什么这样设计

企业系统中，很多“看不到、不能改、不能审批”来自权限和组织边界。不可解释的权限会被误认为系统错误。

### 解决的问题

- 用户不知道为什么不能操作。
- 管理员难以配置权限。
- 数据越权风险。
- 权限问题排查困难。

### 可能带来的问题

- 权限模型复杂。
- UI 需要和后端权限结果对齐。
- 不能泄露用户不该知道的信息。

### 蕴含思想

> 权限不是隐藏按钮这么简单，而是组织责任边界的表达。

## 18. 组件一致性服务认知一致性

### 方法

同类信息使用一致组件：

- 状态标签。
- 金额展示。
- 时间展示。
- 操作按钮。
- 筛选器。
- 表单校验。
- 确认弹窗。
- 明细表格。

### 为什么这样设计

一致性降低学习成本。用户在一个模块学会的交互，应能迁移到其他模块。

### 解决的问题

- 每个页面体验不同。
- 用户需要反复学习。
- 开发重复造组件。
- 设计质量不可控。

### 可能带来的问题

- 设计系统建设需要成本。
- 过度一致可能压制特殊业务表达。
- 组件需要持续治理。

### 蕴含思想

> 一致性不是审美要求，而是降低认知成本的工程手段。

## 19. 密度要适合企业系统

### 方法

企业系统通常需要较高信息密度，但必须有清晰层级：

- 关键信息突出。
- 次要信息收敛。
- 表格字段可配置。
- 分组清晰。
- 数字对齐。
- 状态醒目但不过度装饰。

### 为什么这样设计

企业用户需要扫描、比较、处理大量数据。过度留白会降低效率，过度拥挤会降低理解。

### 解决的问题

- 页面看起来漂亮但效率低。
- 数据很多但无法扫描。
- 长表格难比较。
- 关键信息被淹没。

### 可能带来的问题

- 需要平衡新手理解和老手效率。
- 移动端和桌面端需要不同密度策略。

### 蕴含思想

> 企业 UI 的高级感来自秩序、密度和可扫描性，不来自装饰。

## 20. 设计可配置能力，但不要牺牲默认体验

### 方法

支持配置：

- 列表列配置。
- 筛选条件保存。
- 打印模板。
- 审批流。
- 字段显示。
- 自定义字段。
- 报表维度。

同时提供优秀默认值。

### 为什么这样设计

企业客户差异大，必须支持一定配置。但配置不是让用户从零搭系统。

### 解决的问题

- 不同企业流程不同。
- 不同角色关注字段不同。
- 实施时需要适配客户。

### 可能带来的问题

- 配置复杂度高。
- 默认体验变差。
- 过度配置导致实施成本上升。

### 蕴含思想

> 配置能力是适配差异，默认体验是表达最佳实践。

## 21. UI 必须和接口契约共同设计

### 方法

页面设计时同步定义接口：

- 列表接口返回状态摘要和行级动作。
- 详情接口返回主对象、明细、历史和可执行动作。
- 操作接口返回新状态和影响结果。
- 报表接口返回指标、维度和口径。
- 枚举和权限由后端提供统一契约。

### 为什么这样设计

如果 UI 和接口分离设计，前端会被迫拼字段、猜状态、重复写业务规则。

### 解决的问题

- 前端逻辑过重。
- 状态判断前后端不一致。
- 页面需要多次请求才能拼完整信息。
- 业务规则散落。

### 可能带来的问题

- 接口设计需要更贴近页面任务。
- 后端需要支持读模型或聚合查询。

### 蕴含思想

> 好 UI 需要好接口支撑，前端不是业务规则的垃圾桶。

## 22. UI 模型必须反向校验数据模型

### 方法

用 UI 反推数据模型是否合理：

- 列表筛选项是否对应稳定维度。
- 详情页是否能解释状态和历史。
- 操作按钮是否对应状态转换。
- 报表指标是否能追溯到事实。
- 异常提示是否有规则和证据来源。

### 为什么这样设计

如果数据模型缺失，UI 会表现出明显症状：字段拼凑、状态解释不清、报表口径混乱、操作结果不可追溯。

### 解决的问题

- 数据模型和用户任务脱节。
- 页面展示依赖复杂前端计算。
- 报表和详情无法解释业务结果。

### 可能带来的问题

- 可能发现底层模型需要重构。
- 需要跨前端、后端、数据团队协作。

### 蕴含思想

> UI 是数据模型是否合理的外显症状。

## 23. 从竞品 UI 中提炼思想，而不是复制界面

### 方法

分析优秀产品时提炼：

- 它解决了什么任务。
- 它如何组织业务对象。
- 它如何表达状态。
- 它如何处理异常。
- 它如何支撑报表和钻取。
- 它的复杂度代价是什么。

不要直接复制：

- 视觉样式。
- 字段命名。
- 页面结构。
- 私有流程。
- 专有交互细节。

### 为什么这样设计

竞品的 UI 来自它自己的客户、历史、技术和商业定位。直接复制可能不适合自己的用户和业务阶段。

### 解决的问题

- 避免抄袭和同质化。
- 避免把竞品历史包袱当最佳实践。
- 保留可复用思想。

### 可能带来的问题

- 分析比照抄更费时间。
- 需要有能力判断哪些是思想，哪些是实现。

### 蕴含思想

> 借鉴不是复制结果，而是理解选择背后的原因。

## 24. 可访问性和可测试性是企业 UI 的基础设施

### 方法

设计时考虑：

- 明确文本标签。
- 键盘操作。
- 焦点顺序。
- 表单错误定位。
- 语义化结构。
- 稳定测试标识。
- 可观测操作结果。

### 为什么这样设计

企业系统需要长期维护、自动化测试和不同人群使用。不可测试、不可访问的 UI 会增加长期成本。

### 解决的问题

- 自动化测试不稳定。
- 错误难定位。
- 键盘用户效率低。
- 屏幕阅读器无法理解。
- UI 改动容易破坏业务流程。

### 可能带来的问题

- 前期规范成本增加。
- 组件库需要统一支持。

### 蕴含思想

> 可访问性和可测试性不是附加项，而是高质量 UI 的工程属性。

## 25. UI 设计评审清单

设计 UI 模型后，用以下问题评审：

```text
1. 这个页面服务哪个用户任务？
2. 这个页面围绕哪个业务对象？
3. 用户进入页面后 3 秒内能否知道重点？
4. 当前状态是否清晰？
5. 当前可执行动作是否清晰？
6. 危险动作是否说明影响范围？
7. 高频任务路径是否足够短？
8. 低频危险任务是否受控？
9. 信息分组是否符合业务心智？
10. 表单是否只要求最小必要业务事实？
11. 错误提示是否可操作？
12. 空状态是否说明原因和下一步？
13. 批量任务是否有结果反馈和失败明细？
14. 报表是否回答明确经营问题？
15. 是否能从汇总钻取到明细？
16. 权限和数据范围是否可解释？
17. UI 是否暴露了过多技术结构？
18. 接口是否支撑页面任务，而不是让前端拼业务？
19. 页面是否反向验证了数据模型的合理性？
20. 这个设计解决了什么问题，又引入了什么复杂度？
```

## 26. 总结

世界级 UI 设计的核心不是漂亮，而是把复杂业务变得可理解、可操作、可追溯、可扩展。

常用方法背后的共同思想是：

- 用任务对抗功能堆叠。
- 用对象对抗页面碎片。
- 用状态机对抗按钮混乱。
- 用信息架构对抗字段堆砌。
- 用反馈机制对抗不确定性。
- 用报表问题对抗数据堆叠。
- 用权限解释对抗操作困惑。
- 用一致组件对抗认知负担。
- 用接口契约对抗前端拼凑。
- 用可测试性对抗长期维护风险。

最终目标：

> 让用户不需要理解系统复杂度，也能安全、高效、可解释地完成复杂业务。
