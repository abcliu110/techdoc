# 架构评审工作流

> 版本：v2.0  
> 日期：2026-07-07  
> 适用场景：新系统架构设计、重大架构变更、技术选型

## 一、架构评审触发时机

```
触发条件（满足任一即触发）：
✓ 新系统/新模块的架构设计
✓ 技术栈重大变更（如更换数据库、框架）
✓ 架构模式变更（如单体→微服务、同步→异步）
✓ 引入新的外部依赖（如MQ、缓存、搜索引擎）
✓ 性能目标变更（如QPS从1000→10000）
✓ 安全等级提升（如公网暴露、处理敏感数据）
✓ 跨模块/跨系统集成方案
```

## 二、架构评审核心理念

### 1. 对抗性评审（Adversarial Review）

**每个架构决策都必须有反对者**

```
决策：使用Redis作为缓存
├─ 支持者（Proponent）：
│  └─ 理由：性能好、支持丰富数据结构
│
└─ 反对者（Devil's Advocate）：
   ├─ 质疑1：Redis不可用时怎么办？
   ├─ 质疑2：多实例缓存如何同步？
   ├─ 质疑3：为什么不用本地缓存+DB轮询？
   ├─ 质疑4：Redis内存不足时会影响主业务吗？
   └─ 质疑5：运维成本如何？

→ 强制双方辩论，记录权衡过程
```

### 2. 权衡矩阵法（Tradeoff Matrix）

**每个架构点必须列出≥3个候选方案**

| 架构点 | 候选A | 候选B | 候选C | 采纳 | 拒绝理由 |
|---|---|---|---|---|---|
| 元数据存储 | 少表+JSON聚合 | 子定义全部拆表 | JSON聚合+影子表 | A+影子表 | B事务成本高 |
| 业务数据 | 动态表 | EAV | JSON宽表 | A+SchemaSync | EAV查询复杂；JSON损失约束 |
| 权限模型 | RBAC | ABAC | RBAC+ABAC混合 | C+AccessView | A不支持数据范围；B过于复杂 |

**强制规则**：
- 必须有≥3个候选
- 每个落选方案必须有明确拒绝理由
- 记录到ADR（Architecture Decision Record）

### 3. 质量属性量化

**质量目标必须可度量、可验收**

```
❌ 错误：系统要"性能好"、"安全"、"可扩展"
✅ 正确：
- 性能：冷启动<2s，list P95<500ms，并发1000 QPS
- 安全：通过OWASP Top 10测试，无SQL注入/XSS漏洞
- 可扩展：新增字段类型不改核心代码，通过SPI注册
```

质量属性模板：

| 维度 | 定义 | 可验收标准 | 验证方式 |
|---|---|---|---|
| 正确性 | 元数据、物理结构、缓存、快照四方一致 | 发布后Reconciler对账无差异 | 集成测试 |
| 安全性 | 六层权限统一判定，恶意输入有测试集 | 通过14类安全测试 | 安全测试 |
| 性能 | 冷加载<2s，list P95<500ms | 基准数据集测试通过 | 性能测试 |
| 可扩展性 | 新增类型=实现SPI+注册 | 22种字段类型通过SPI实现 | 契约测试 |

### 4. 陷阱驱动评审

**基于历史陷阱清单，逐项检查防线**

```
评审流程：
1. 查阅《00-陷阱覆盖总表.md》
2. 定位本架构涉及的陷阱编号
3. 检查每个陷阱的防线是否存在

示例：
架构变更：引入动态DDL
涉及陷阱：
  - A1: DDL非原子
  - A2: DDL不可回滚
  - A3: 动态表爆炸
  - A5: MDL锁雪崩

检查防线：
  A1 → 是否有发布状态机？✓
  A1 → 是否持久化DDL Plan？✓
  A1 → 是否有Reconciler？✓
  A2 → 是否定义回滚边界？✓
  A3 → 是否有表数量上限？✓
  A5 → 是否有MDL预检？✓
```

## 三、架构评审流程

### Phase 1: 准备阶段（评审前1-2天）

**1.1 提交架构方案**

```markdown
# 架构方案文档结构（必需）

## 1. 背景与目标
- 业务背景
- 架构目标
- 非功能需求（性能、安全、可用性）

## 2. 现状分析
- 当前架构问题
- 技术债务
- 遗留约束

## 3. 架构决策
- 决策点清单
- 每个决策点的权衡矩阵（≥3候选）
- 采纳结论与拒绝理由

## 4. 目标架构
- 架构图（C4模型：Context/Container/Component/Code）
- 模块划分
- 依赖关系
- 数据流图
- 部署架构

## 5. 质量属性
- 8维质量目标（可度量）
- 验证方式

## 6. 风险与缓解
- 技术风险
- 陷阱覆盖清单
- 缓解措施

## 7. 实施计划
- 里程碑划分
- 依赖项
- 回滚方案
```

**1.2 分发材料**

```bash
# 材料清单
- 架构方案文档（PDF/Markdown）
- 架构图（draw.io/PlantUML）
- 权衡矩阵（Excel/Markdown表格）
- 陷阱覆盖表（标注涉及的陷阱编号）
- 原型/POC代码（如有）

# 分发时间
提前48小时分发，给评审者充足时间准备
```

**1.3 指定角色**

```
必需角色：
- 方案提出者（Proponent）：架构师/技术负责人
- 魔鬼代言人（Devil's Advocate）：指定反对者
- 安全专家：检查安全红线
- 性能专家：检查性能瓶颈
- DBA：检查数据模型
- 运维专家：检查可运维性
- 记录员：记录评审过程
```

### Phase 2: 评审会议（2-4小时）

**2.1 开场（10分钟）**

```
1. 主持人宣布评审规则
2. 介绍评审角色
3. 说明评审范围和目标
```

**2.2 方案陈述（30分钟）**

```
方案提出者陈述：
- 背景与目标（5分钟）
- 架构决策（15分钟）
  - 重点讲权衡过程
  - 为什么选A不选B/C
- 风险与缓解（10分钟）

禁止事项：
❌ 不要照本宣科读PPT
❌ 不要过度技术细节
✓ 重点讲决策理由
✓ 重点讲权衡过程
```

**2.3 魔鬼代言人质疑（30分钟）**

```
魔鬼代言人逐项质疑：

质疑模板：
1. "为什么不用方案B？"
   → 强制提出者解释拒绝理由

2. "如果XXX失败了怎么办？"
   → 强制提出者说明降级方案

3. "这个假设如果不成立会怎样？"
   → 挑战设计假设

4. "这个决策的成本是什么？"
   → 强制提出者说明代价

5. "有没有更简单的方案？"
   → 避免过度设计

记录员记录所有质疑和回答
```

**2.4 分组讨论（60分钟）**

```
按维度分组讨论：

组1：架构合理性
- 模块边界是否清晰？
- 依赖方向是否正确？
- 是否有循环依赖？

组2：安全性
- 是否覆盖所有安全红线？
- 是否有越权风险？
- 是否有注入风险？

组3：性能与可扩展性
- 是否有性能瓶颈？
- 是否有单点故障？
- 扩展策略是否清晰？

组4：可运维性
- 是否可监控？
- 是否可降级？
- 是否可回滚？

每组讨论30分钟，产出问题清单
```

**2.5 陷阱逐项检查（30分钟）**

```
使用《00-陷阱覆盖总表.md》逐项检查：

示例：
□ A1 DDL非原子
  - 是否有发布状态机？
  - 是否持久化DDL Plan？
  - 是否有Reconciler？
  - 机器防线：✓ DDL失败注入测试
  - 人工checklist：✓ 半应用状态可恢复

□ B1 租户上下文异步丢失
  - 异步任务是否显式传递tenantId？
  - 是否有fail-fast？
  - 机器防线：✓ 双租户异步测试
  
逐项打勾，未覆盖的陷阱记录为风险项
```

**2.6 投票表决（10分钟）**

```
架构评审结论：
APPROVE
  - 无阻断问题
  - 风险可控
  - 质量目标可达

COMMENT
  - 无架构 BLOCKER，但有中等风险或待补证据
  - 必须记录负责人和补齐期限

NEEDS HUMAN DECISION
  - 需要业务/安全/架构负责人裁决
  - 必须记录负责人和补齐期限

REQUEST CHANGES
  - 有重大架构缺陷
  - 需要重新设计

投票规则：
- 无 BLOCKER 且 2/3 以上同意 → APPROVE
- 存在架构 BLOCKER → REQUEST CHANGES
- 需要业务/安全/架构负责人裁决 → NEEDS HUMAN DECISION
- 无 BLOCKER 但有待补充事项 → COMMENT
```

## 四、架构评审检查清单

### 通用检查点

```
□ 业务目标
  □ 架构是否支持业务目标？
  □ 是否有明确的成功标准？

□ 架构决策
  □ 每个决策点是否有≥3个候选方案？
  □ 每个落选方案是否有明确拒绝理由？
  □ 是否记录到ADR？

□ 模块边界
  □ 模块职责是否单一？
  □ 模块间依赖方向是否正确？
  □ 是否有循环依赖？
  □ 是否有ArchUnit规则？

□ 质量属性
  □ 8个维度是否都有可度量目标？
  □ 每个目标是否有验证方式？

□ 陷阱覆盖
  □ 是否逐项检查了陷阱清单？
  □ 每个陷阱是否有防线？
  □ 是否有机器防线+人工checklist？
```

## 五、架构评审反模式

### ❌ 反模式1：橡皮图章评审

```
错误做法：
- 评审会上没人提问题
- 大家都说"看起来不错"
- 10分钟走过场

正确做法：
- 指定魔鬼代言人，强制质疑
- 每个决策必须辩论
- 记录所有质疑和回答
```

### ❌ 反模式2：权威压制

```
错误做法：
老板：我觉得用Redis
下属：（心里不同意，但不敢说）好的

正确做法：
- 技术问题用技术说话
- 鼓励提出反对意见
- 权威者最后发言
```

### ❌ 反模式3：事后评审

```
错误做法：
代码都写完了，才开评审会

正确做法：
- 设计阶段就评审
- 编码前评审通过
- 边界变更时重新评审
```

### ❌ 反模式4：完美主义

```
错误做法：
评审者：这个方案不够优雅，要重新设计
（方案已满足门禁，却因主观审美要求重做）

正确做法：
- 满足门禁即可通过
- 未阻断的技术债要记录责任人、补齐期限或风险接受依据
- 关注关键风险，而不是用主观审美替代评审标准
```

### ❌ 反模式5：形式主义

```
错误做法：
- PPT做得很漂亮
- 但没有权衡矩阵
- 没有ADR
- 没有陷阱覆盖

正确做法：
- 内容重于形式
- 必须有权衡过程
- 必须有ADR
- 必须标注陷阱覆盖
```

### 4. 架构约束检查

```java
// ArchUnit
@ArchTest
public static final ArchRule modules_should_not_have_cycles =
    slices().matching("com.lowcode.(*)..").should().beFreeOfCycles();

@ArchTest
public static final ArchRule service_should_not_depend_on_controller =
    noClasses().that().resideInPackage("..service..")
        .should().dependOnClassesThat().resideInPackage("..controller..");
```

## 七、架构评审度量

```
过程指标：
- 评审覆盖率 = 已评审架构点 / 总架构点
- 评审周期 = 从提交到通过的时间
- 质疑密度 = 质疑数 / 决策点数

结果指标：
- 架构变更率 = 评审后架构变更次数 / 总评审次数
- 架构缺陷密度 = 生产架构问题 / 已评审架构点
- ADR完整性 = 有ADR的决策 / 总决策数
```

### 案例1：低代码平台发布管线架构评审

**背景**：设计发布管线，将设计态元数据发布到运行态

**候选方案**：
- A: 直接修改元数据状态 + 执行DDL
- B: 发布状态机 + DDL Plan + 物理登记表
- C: 版本快照 + 蓝绿发布

**权衡过程**：
- A简单但不可恢复，拒绝
- C成本高，首版不需要，拒绝
- B采纳

**魔鬼代言人质疑**：
- Q: DDL失败了怎么办？
- A: 状态机记录失败点，可resume或abandon

- Q: 并发发布怎么办？
- A: DB行锁+fencing token互斥

**结论**：APPROVE，产出 ADR-LOWCODE-PUBLISH-001
