## 推荐使用路径

### 人工代码评审

```text
1. 读 00，理解整体分级和门禁。
2. 读 03，按 L1-L5 确认评审强度。
3. 用 05 做检查清单。
4. 需要工具时参考 06，但先验证项目是否已配置。
```

### AI代码评审

```text
1. 从 09 开始。
2. 按 09 加载既有系统安全修改规范和项目陷阱表。
3. 必要时参考 03、05、08。
4. 输出 09 规定的证据化评审报告。
```

### 架构评审

```text
1. 读 01。
2. 准备目标、约束、候选方案、权衡矩阵和 ADR。
3. 指定魔鬼代言人。
4. 输出 ADR、风险清单、验证方式和后续行动。
```

## 核心原则

```text
风险分层：不同风险用不同评审强度。
证据驱动：每条问题必须能追溯到文件、行号、规范或行为证据。
兼容优先：既有系统默认存在隐性契约，不能无证据破坏旧逻辑。
自动化优先：机器负责重复检查，人负责语义、业务和架构判断。
例外可审计：允许风险接受，但必须有负责人、期限和补偿措施。
持续改进：新问题要回写陷阱清单或知识库。
```

## 与项目规范的关系

当前目录是评审流程层，不是所有项目规范的源头。评审时仍需加载具体项目规范。

| 依赖 | 状态 | 用途 |
|---|---|---|
| `../工程规范/低代码平台规范/06-陷阱覆盖矩阵.md` | 已确认存在 | 定位低代码平台历史陷阱 |
| `../工程规范/*.md` | 按实际文件存在性读取 | 命名、安全、数据库、API、测试等细则 |
| `D:\mywork\techdoc\saas\既有系统安全修改代码AI执行规范.md` | 代码修改/评审必读 | 既有系统兼容、安全修改底线 |

如果某个规范文件不存在，评审报告必须说明，不得静默跳过。

## 工具说明

`06-评审工具与自动化.md` 中的 SonarQube、SpotBugs、Checkstyle、Semgrep、JaCoCo、GitHub Actions 等配置是参考模板。是否能作为门禁，取决于目标项目是否已经安装、配置并验证。

工具状态应分为：

```text
已验证可用：当前项目已配置并能运行。
模板示例：可参考，需要适配。
概念工具：仅表达方向，不能作为门禁依据。
```

## 通用性评价

这套评审体系的框架具备较强通用性，适合大多数企业级软件项目；但具体规则不应直接照搬到所有项目。

| 场景 | 通用性 |
|---|---|
| 多租户 SaaS、CRM、ERP、订单、库存、支付系统 | 高 |
| 普通后端 Web 项目 | 中高 |
| 前端应用 | 中，需要补充 UI、可访问性、状态管理和 E2E 细则 |
| 小脚本、Demo、一次性工具 | 中低，应裁剪为轻量流程 |
| 嵌入式、实时系统、算法系统 | 中低，需要替换风险域和验证方式 |

结论：流程框架通用，风险域、规范条款、工具链和门禁阈值需要按项目裁剪。
