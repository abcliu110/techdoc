# 自动化代码评审工作流 - AI执行入口

> 版本：v2.0  
> 日期：2026-07-07  
> 定位：AI执行代码评审的唯一入口 SOP  
> 适用范围：既有系统代码评审、PR评审、目录/文件级代码审查  
> 核心原则：风险分层、证据驱动、兼容优先、可审计、可回滚

## 0. SOP定位

本文件是 AI 执行代码评审时的入口文件。AI 读取本文件后，应按本文流程完成评审、输出证据化结论，并明确哪些问题阻断合并、哪些非阻断问题允许风险接受。

本 SOP 不替代人工责任。AI 负责初审、证据汇总、风险定位和修复建议；涉及业务取舍、风险接受、上线窗口、生产例外时，必须由对应负责人确认。

## 1. 先进评审模型

本 SOP 采用当前工程组织中更成熟的代码评审做法：

| 能力 | 执行方式 |
|---|---|
| Shift Left | 评审从需求、架构、设计、代码、测试逐层前移 |
| Risk-Based Review | 按风险层级决定逐行、重点、抽查或工具扫描 |
| Evidence-Based Review | 每个问题必须有文件、行号、规范、影响、验证建议 |
| Compatibility First | 既有系统默认存在隐性依赖，禁止无证据破坏旧逻辑 |
| Security by Default | 安全、租户、权限、资金、库存、订单默认高优先级 |
| Automation + Human Judgment | 机器覆盖重复规则，人处理语义、架构、业务取舍 |
| Exception Governance | 允许例外，但必须记录风险接受人、截止时间和补偿措施 |
| Continuous Learning | 评审发现的新坑要回写陷阱清单或知识库 |

## 2. 输入与输出

### 2.1 输入

评审开始前必须确认：

```text
评审范围：PR / 分支 diff / 单文件 / 目录
项目根目录：实际业务仓库根目录，不能用 D:\ 作为项目根
变更目标：需求、bug、重构、配置、测试或文档
运行环境：本地、测试、预发、生产修复
风险领域：金额、库存、订单、权限、多租户、支付、MQ、外部接口、数据库结构
```

### 2.2 输出

评审报告必须包含：

```text
评审结论：
评审范围：
风险层级：
保留的老逻辑：
新增的逻辑：
新逻辑启用条件：
兼容措施：
问题清单：
测试或验证：
回滚方案：
未覆盖风险：
需要人工确认的未决项：
```

## 4. 风险分层

### 4.1 代码层级

| 层级 | 范围 | 评审强度 |
|---|---|---|
| L1 安全边界 | SQL拼装、权限、租户、认证、支付凭证、敏感数据 | 逐行 + 双人/人工复核 |
| L2 核心状态 | 订单、库存、支付、MQ、事务、状态机、数据迁移 | 逐行 + 场景验证 |
| L3 核心业务 | Service、领域规则、计费、同步、外部接口 | 重点逐行 + 影响分析 |
| L4 普通业务 | CRUD、查询、后台配置、低风险接口 | 重点抽查 + 工具扫描 |
| L5 低风险代码 | DTO、VO、枚举、工具类、测试辅助 | 抽查 + 规范扫描 |

默认规则：无法判断层级时按更高风险处理。

### 4.2 问题严重级别

| 级别 | 合并策略 | 示例 |
|---|---|---|
| BLOCKER | 禁止合并，必须修复；无法立即修复时只能转人工裁决，不能自动通过 | SQL注入、越权、跨租户、资金/库存错误、破坏旧逻辑、核心路径无验证 |
| HIGH | 默认修复后合并；延期必须记录风险接受人和截止时间 | N+1导致核心接口不可用、并发风险、兼容性不明、测试缺失 |
| MEDIUM | 建议本次修复；可带记录合并 | 局部复杂度高、错误处理不足、日志不完整、非核心性能问题 |
| LOW | 可选优化 | 命名、注释、局部可读性、非关键重复代码 |

关键规则：不是“所有规范违反都阻断”，而是“违反红线规范或造成实质风险才阻断”。普通规范违反必须按影响分级。

## 5. 执行流程

### Step 1 确认评审范围

```text
1. 读取 git diff 或用户指定文件。
2. 列出变更文件、变更类型、涉及模块。
3. 标出新增、修改、删除、重命名、配置变更、数据库变更。
4. 检查工作区是否有无关改动，避免把用户改动误判为本次变更。
```

### Step 2 识别旧逻辑和隐性契约

必须回答：

```text
旧入口在哪里？
旧调用方有哪些？
旧数据字段、配置、定时任务、消息链路是否受影响？
旧接口行为是否改变？
历史数据是否兼容？
是否存在灰度、开关、回滚或兼容路径？
```

无法确认依赖时，结论必须写为“未知”，并给出最小验证动作。

### Step 3 定位陷阱和风险域

按变更内容映射陷阱：

```text
查询 / Mapper / 报表        -> 租户隔离、N+1、SQL注入、分页、索引
权限 / 菜单 / 角色          -> 越权、权限一致性、缓存失效
订单 / 支付 / 库存          -> 幂等、并发、状态机、补偿
MQ / 定时任务 / 同步        -> 重复投递、租户上下文、失败重试
DDL / 数据迁移              -> 回滚、兼容、锁表、历史数据
前端输入 / 富文本 / URL      -> XSS、CSRF、输入校验
```

### Step 4 执行红线检查

以下问题发现即 BLOCKER：

```text
跨租户读取或写入风险
权限绕过或未统一鉴权
SQL/NoSQL/表达式注入
金额、库存、支付、订单状态错误
无幂等的支付、订单、MQ消费写操作
无并发控制的金额/库存扣减
核心旧逻辑被删除或行为改变但无兼容说明
数据库结构变更无回滚或历史数据兼容说明
密钥、token、支付凭证硬编码
敏感信息明文日志
核心链路无任何可执行验证
```

### Step 5 执行质量检查

按影响分级，不自动阻断：

```text
函数过长、嵌套过深、重复逻辑
命名不一致但不影响契约
注释不足但代码语义明确
错误信息不够清晰
局部性能可优化
测试覆盖不足但非核心路径
```

### Step 6 验证充分性检查

评审报告必须说明已验证和未验证：

```text
是否有单元测试？
是否有集成测试？
是否有回归测试？
是否覆盖租户隔离？
是否覆盖幂等与重复请求？
是否覆盖并发？
是否覆盖异常与回滚？
是否运行 lint / typecheck / build / test？
```

AI 不得把“建议运行测试”写成“已验证”。

### Step 7 输出评审结论

合并建议只能使用以下四类：

```text
APPROVE：无 BLOCKER，HIGH 已修复或已正式接受，验证充分。
COMMENT：无 BLOCKER，但存在 MEDIUM/LOW 或轻量 HIGH，需要记录。
REQUEST CHANGES：存在 BLOCKER，或 HIGH 未处理且无风险接受。
NEEDS HUMAN DECISION：技术证据不足，需要业务/架构/安全负责人裁决。
```

### Step 8 评审-修复-再评审循环

评审不是单轮动作。只要发现可执行缺陷，必须进入闭环：

```text
1. 评审输出问题清单，并标注 BLOCKER/HIGH/MEDIUM/LOW。
2. 如果结论为 REQUEST CHANGES，开发者必须修复 BLOCKER 和未被正式接受的 HIGH。
3. 修复后重新评审同一变更，重点确认问题是否关闭、旧逻辑是否仍兼容、是否引入新风险。
4. 如果复审发现新的 BLOCKER/HIGH，继续修复并再次复审。
5. 不能因为“已经评审过一轮”而停止。
```

循环停止条件：

```text
- 不存在 BLOCKER。
- HIGH 已修复，或已有明确责任人正式接受风险。
- 必要验证已经运行，验证缺口已记录。
- 剩余 MEDIUM/LOW 已进入后续计划或技术债。
- 需要人类裁决的事项已明确列出。
- 最终建议为 APPROVE 或 COMMENT。
```

## 6. 例外与风险接受

允许 HIGH/MEDIUM/LOW 例外，但必须满足全部条件。BLOCKER 不走自动风险接受；确实无法立即修复时，结论必须是 NEEDS HUMAN DECISION。

```text
1. 说明为什么不能本次修复。
2. 说明影响范围和最坏后果。
3. 指定风险接受人。
4. 指定补救截止时间。
5. 给出临时防护或监控措施。
6. 记录到评审报告。
```

以下问题原则上不允许例外：跨租户、越权、SQL注入、资金/库存/支付错误、密钥泄露。

## 8. AI自检清单

评审完成前必须逐项确认：

```text
□ 是否读取了既有系统安全修改规范？
□ 是否确认了项目根目录和评审范围？
□ 是否区分了事实、推断、未知？
□ 是否识别了旧逻辑、调用方、配置、数据和回滚路径？
□ 是否按 L1-L5 做了风险分层？
□ 是否检查了租户、权限、SQL、幂等、并发、敏感日志？
□ 是否把普通规范问题按影响分级，而不是一律阻断？
□ 每个问题是否有文件/行号/证据/影响/修复建议？
□ 是否说明已验证、未验证和剩余风险？
□ 是否给出明确合并建议？
```

## 9. 通用性评价

本 SOP 具备较强通用性，但不是完全无差别通用。

| 适用性 | 评价 |
|---|---|
| 企业级后端系统 | 高。尤其适合多租户、订单、库存、支付、权限、数据一致性系统 |
| 既有系统维护 | 高。兼容优先、旧逻辑保留、回滚方案是核心设计 |
| 普通 Web / SaaS 项目 | 中高。可直接复用风险分层和报告模板 |
| 前端 UI 项目 | 中。需要补充可访问性、视觉回归、状态管理、端到端测试细则 |
| 小型脚本 / Demo | 中低。流程偏重，可裁剪为轻量版 |
| 嵌入式 / 实时系统 / 算法系统 | 中低。需替换风险域和验证标准 |
| 开源社区项目 | 中。可复用报告结构，但风险接受和角色职责需改成维护者模型 |

结论：本 SOP 的“框架”通用，具体规范、风险域、门禁阈值不应照搬。落地到其他团队时，应替换项目规范、风险域、工具链和审批角色。
