# 代码评审工作流

> 版本：v2.0  
> 日期：2026-07-07  
> 适用场景：所有代码变更、PR评审、日常开发

## 一、代码评审核心理念

### 1. 分层评审策略

**不同层次的代码用不同的评审强度**

```
┌────────────────────────────────────────────┐
│ 层级 │ 代码类型          │ 评审方式    │
├────────────────────────────────────────────┤
│ L1  │ 安全边界          │ 逐行+双人   │
│     │ - SqlAssembler    │             │
│     │ - AccessDecider   │             │
│     │ - TenantContext   │             │
│ L2  │ 事务边界/状态机    │ 逐行        │
│     │ - PublishService  │             │
│     │ - StateMachine    │             │
│ L3  │ 核心业务逻辑      │ 重点逐行    │
│ L4  │ 普通CRUD          │ 抽查20%     │
│ L5  │ 工具类/DTO        │ 抽查10%     │
│ ALL │ 所有代码          │ 工具扫描100%│
└────────────────────────────────────────────┘
```

### 2. 陷阱驱动评审

**基于《00-陷阱覆盖总表.md》，针对性检查防线**

```bash
# 第一步：定位陷阱
git diff origin/main --name-only
# → 修改了OrderService.java

# 第二步：查陷阱表
# OrderService涉及CRUD查询
# 可能涉及陷阱：B1(租户隔离)、C1(N+1查询)、H1(SQL注入)

# 第三步：逐项检查防线
□ B1: tenantId是否fail-fast？
□ C1: 循环中是否有查询？
□ H1: SQL是否参数化？
```

### 3. 契约测试驱动

**关键代码必须有契约测试覆盖**

```java
// 不只评审代码，更要评审测试

// L1/L2代码要求：
// - 行覆盖率 100%
// - 分支覆盖率 100%
// - 每个分支都有测试用例

@Test
void shouldRejectSqlInjection() {
    String malicious = "' OR '1'='1";
    assertThrows(BizException.class, () -> {
        sqlAssembler.buildWhere(
            FilterNode.eq("status", malicious)
        );
    });
}
```

### Phase 2: 工具自动评审（CI）

以下 CI 配置是模板。引入项目门禁前，必须确认工具已安装、规则已校准、失败处理可执行；知识图谱类工具还必须确认当前仓库已完成索引。

```yaml
# .github/workflows/code-review.yml
name: Code Review

on: [pull_request]

jobs:
  automated-review:
    runs-on: ubuntu-latest
    steps:
      # 1. 代码规范检查
      - name: Checkstyle
        run: mvn checkstyle:check
        
      # 2. 代码质量扫描
      - name: SonarQube
        run: mvn sonar:sonar
        
      # 3. 安全漏洞扫描
      - name: SpotBugs
        run: mvn spotbugs:check
        
      # 4. 依赖漏洞扫描
      - name: Dependency Check
        run: mvn dependency-check:check
        
      # 5. 架构约束检查
      - name: ArchUnit
        run: mvn test -Dtest=ArchitectureTest
        
      # 6. 测试覆盖率检查
      - name: JaCoCo
        run: mvn jacoco:check
        
      # 7. 自定义安全规则
      - name: Semgrep
        run: semgrep --config=.semgrep.yml
        
      # 8. 知识图谱分析
      - name: Impact Analysis
        run: |
          command -v code-review-graph >/dev/null 2>&1 || { echo "code-review-graph not installed"; exit 1; }
          code-review-graph detect-changes \
            --repo . \
            --base origin/main \
            --brief
```

**3.2 分层评审（30-120分钟）**

#### L1: 安全边界逐行评审（必须100%逐行）

```java
// ✓ 必须逐行：动态SQL组装
public class SqlAssembler {
    public SqlSegment buildWhere(FilterNode filters, ObjectDef objectDef) {
        // 第1行评审：table来源是否白名单？
        TableRef table = tableRegistry.requireTable(objectDef.code());
        
        // 第2行评审：op是否白名单枚举？
        String op = this.validateOp(filters.getOp());
        
        // 第3行评审：column是否从元数据获取？
        ColumnRef column = objectDef.requireReadableField(
            filters.getField()
        ).column();
        
        // 第4行评审：用户值是否参数化？
        sql.append(column.name()).append(" ").append(op).append(" ?");
        params.add(filters.getValue());
        
        return new SqlSegment(sql.toString(), params);
    }
}
```

**L1评审检查点**：
```
□ SQL组装（SqlAssembler）
  □ 表名是否白名单？
  □ 列名是否白名单？
  □ op是否枚举？
  □ 用户值是否参数化？
  □ data_scope是否AST层合并？

□ 权限判定（AccessDecider）
  □ 是否统一入口？
  □ 是否计算了六层链？
  □ 是否返回不可变视图？
  □ 是否记录metaHash？

□ 租户隔离（TenantDataService）
  □ tenantId是否fail-fast？
  □ 是否强制注入tenant_id？
  □ 异步是否显式传递？
```

#### L3: 核心业务重点评审

```java
// ○ 重点评审：核心业务逻辑
public void createOrder(OrderDTO dto) {
    // 重点看：
    // 1. 租户隔离
    Long tenantId = TenantContext.getTenantId();
    if (tenantId == null) {
        throw new BizException("租户ID缺失");
    }
    
    // 2. 权限检查
    AccessView access = accessDecider.decide(user, "order", ctx);
    if (!access.hasObjectPerm(ObjectPerm.CREATE)) {
        throw new BizException("无权限");
    }
    
    // 3. 数据校验
    validateOrder(dto);
    
    // 4. 业务逻辑
    Order order = orderConverter.toEntity(dto);
    order.setTenantId(tenantId);
    order.setStatus(OrderStatus.PENDING);
    
    // 5. 持久化
    orderMapper.insert(order);
    
    // 6. 审计日志
    auditLogger.log("创建订单", order.getLid());
}
```

**L3评审检查点**：

□ 权限检查
□ 数据校验
□ 枚举使用
□ 审计日志

## 七、评审效率优化

### 1. 小批量PR

```
❌ 一次提交2000行代码
✅ 分成4个PR，每个500行

优势：
- 评审更快
- 问题更容易发现
- 回退风险更小
```

### 2. 异步评审

```
传统同步：所有人一起开会评审（低效）
现代异步：
  1. 评审者在线上提交意见
  2. 开发者在线上回复
  3. 只对争议点开会
  4. 会议时间缩短80%
```

### 3. 评审时间盒

```
建议时间：
- 小PR (<200行)：20分钟
- 中PR (200-500行)：60分钟
- 大PR (500-1000行)：120分钟
- 超大PR (>1000行)：拆分成小PR
```

### 4. 评审优先级

```
高优先级：
- 安全边界代码（L1）
- 关键业务代码（L2）
- 修复生产问题的PR

低优先级：
- 文档更新
- 代码格式化
- 注释修改
```

## 八、评审度量

```
过程指标：
- 评审覆盖率 = 已评审代码行数 / 总代码行数
- 评审响应时间 = 提交PR到首次评审的时间
- 评审周期 = 提交到合并的时间

质量指标：
- 缺陷发现率 = 评审发现缺陷 / 总缺陷
- 缺陷逃逸率 = 生产缺陷 / 总缺陷
- 返工率 = 返工代码行 / 总代码行

目标：
- 评审响应时间 ≤ 24小时
- 缺陷发现率 ≥ 90%
- 返工率 ≤ 20%
```

## 九、评审文化

### 应该做的（DO's）

✅ **快速反馈**：24小时内给出评审意见  
✅ **建设性**：给出具体改进建议  
✅ **有数据**：引用规范条款、陷阱编号  
✅ **双向对话**：平等讨论，不是单向批评  
✅ **学习心态**：评审是学习机会  

### 不应该做的（DON'Ts）

❌ **不要拖延**：评审不超过48小时  
❌ **不要追求完美**：满足门禁即可通过，非阻断问题进入技术债或后续改进  
❌ **不要人身攻击**：评审代码，不评审人  
❌ **不要权威压制**：技术问题用技术说话  
❌ **不要形式主义**：要有实质内容
