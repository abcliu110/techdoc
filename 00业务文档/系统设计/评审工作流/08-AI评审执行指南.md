> 版本：v2.0  
> 日期：2026-07-07  
> 定位：AI评审原则、表达规范和边界说明  
> 说明：AI真正执行代码评审时，以 `09-自动化代码评审工作流-AI执行入口.md` 为唯一入口。

## 1. 与09的职责边界

| 文档 | 职责 |
|---|---|

## 2. AI评审原则

```text
证据优先：每个问题必须有文件、行号、代码片段或文档依据。
风险分层：普通规范问题不自动等于阻断，必须按影响分级。
兼容优先：既有系统默认存在隐性契约，不能轻易建议删除或重写旧逻辑。
事实/推断分离：证据不足时写“未知”或“待验证”，不能伪装确定。
修复建议保守：优先给最小修改方向，不引入无关重构。
验证闭环：评审报告必须说明已验证、未验证和剩余风险。
```

## 3. 严重级别表达

| 级别 | 使用条件 | 表达方式 |
|---|---|---|
| BLOCKER | 安全、租户、权限、资金、库存、订单、核心兼容或核心验证缺失 | “必须修复后才能合并” |
| HIGH | 明确风险但可通过风险接受延期 | “默认修复；若延期需风险接受” |
| MEDIUM | 可维护性、非核心性能、错误处理、测试不足等 | “建议本次修复或记录技术债” |
| LOW | 命名、注释、局部可读性、非关键重复 | “可选优化” |

禁止把所有规范违反都标为 BLOCKER。只有造成红线风险或实质影响的规范违反才阻断。

## 4. 必须检查的高风险域

```text
多租户：是否带 tenant/pid/mid/sid 等隔离条件。
权限：是否统一鉴权，是否存在绕过路径。
SQL/输入：是否参数化，外部输入是否校验。
幂等：支付、订单、MQ 消费、重复提交是否可重入。
并发：金额、库存、状态机是否有锁或版本控制。
事务：事务边界是否包住一致性操作，是否包含不该在事务内的外部调用。
日志：手机号、身份证、token、支付凭证是否脱敏。
兼容：旧接口、旧配置、历史数据、定时任务、消息链路是否被破坏。
验证：核心路径是否有可执行测试或明确验证记录。
```

## 5. 输出要求

AI评审报告必须包含：

```text
评审结论：
评审范围：
风险层级：
变更目标：
保留的老逻辑：
新增的逻辑：
新逻辑启用条件：
兼容措施：
问题清单：
测试或验证：
回滚方案：
未覆盖风险：
需要人工确认的未决项：
```

问题描述必须包含：

```text
位置：文件和行号
证据：代码或文档依据
影响：为什么这会造成风险
级别：BLOCKER/HIGH/MEDIUM/LOW
建议：最小修复方向
验证：如何证明修复有效
```

## 6. 禁止事项

```text
禁止无证据下结论。
禁止把推断写成事实。
禁止只说“建议优化”但不说明影响。
禁止把所有命名、注释、风格问题一律阻断。
禁止为了显得完整而建议无关重构。
禁止忽略旧逻辑兼容、回滚和验证。
禁止把未运行的测试写成已通过。
禁止生成可能破坏业务语义的“一键修复代码”。
```

## 7. 推荐评审话术

好的表达：

```text
BLOCKER：`OrderMapper.xml:42` 使用 `${status}` 拼接外部输入。证据是该参数来自接口入参，未做白名单或参数化，存在 SQL 注入风险。建议改为 `#{status}` 或白名单映射，并补充非法 status 的回归测试。
```

不好的表达：

```text
这里不太好，建议改一下。
```

好的表达：

```text
MEDIUM：`CustomerService.java:88` 方法包含查询、转换、权限过滤三类职责，当前未造成明确错误，但会增加后续修改成本。建议在本次或后续小重构中提取权限过滤逻辑，并保留现有返回结构不变。
```

不好的表达：

```text
这个方法太长，阻断。
```

## 8. 自检清单

```text
□ 是否从 09 进入执行流程？
□ 是否加载了既有系统安全修改规范？
□ 是否确认了评审范围和项目根目录？
□ 是否识别旧逻辑、调用方、配置、数据和回滚路径？
□ 是否按风险分层，而不是按个人偏好分级？
□ 每个问题是否有证据、影响、建议和验证方式？
□ 是否说明未验证项和人工确认项？
□ 是否避免了夸大、绝对化和无证据判断？
```
