## 2. 核心理念

### 2.1 质量前置

评审不只发生在代码合并前。需求、架构、设计、代码、测试和上线准备都应有相应门禁。

```text
需求 -> 需求评审 -> 架构/设计 -> 架构/设计评审 -> 编码 -> 代码评审 -> 验证 -> 发布评审
```

### 2.2 风险分层

不同风险使用不同评审强度：

| 层级 | 范围 | 评审强度 |
|---|---|---|
| L1 安全边界 | SQL、权限、租户、认证、支付凭证、敏感数据 | 逐行 + 双人/人工复核 |
| L2 核心状态 | 订单、库存、支付、MQ、事务、状态机、数据迁移 | 逐行 + 场景验证 |
| L3 核心业务 | Service、领域规则、计费、同步、外部接口 | 重点逐行 + 影响分析 |
| L4 普通业务 | CRUD、查询、后台配置、低风险接口 | 重点抽查 + 工具扫描 |
| L5 低风险代码 | DTO、VO、枚举、工具类、测试辅助 | 抽查 + 规范扫描 |

### 2.3 陷阱驱动

评审应优先覆盖历史上出过问题的区域：

```text
变更文件 -> 风险域 -> 陷阱编号 -> 防线 -> 测试/验证
```

项目应维护陷阱清单。新发现的问题如果具备复发风险，应回写到陷阱清单或知识库。

### 2.4 兼容优先

既有系统默认存在隐性契约。评审必须确认：

```text
旧入口是否保留？
旧调用方是否兼容？
历史数据是否兼容？
配置、定时任务、消息链路是否受影响？
是否有新逻辑启用条件？
是否有回滚方案？
```

### 2.5 自动化与人工判断结合

机器适合做重复、确定、可规则化的检查；人负责语义、业务、架构和风险接受。

```text
机器做：格式、静态扫描、依赖漏洞、测试覆盖、部分安全规则、架构约束。
人做：业务正确性、兼容性、架构取舍、风险接受、上线决策。
```

## 3. 评审阶段

| 阶段 | 输入 | 输出 | 主要检查 |
|---|---|---|---|
| 需求评审 | PRD、用户故事、业务规则 | 开放问题、验收标准、追溯表 | 完整性、一致性、可测试性、非功能需求 |
| 架构评审 | 架构方案、候选方案、约束 | ADR、权衡矩阵、风险清单 | 边界、依赖、质量属性、回滚、陷阱覆盖 |
| 设计评审 | 接口契约、数据模型、状态机 | 冻结清单、设计改进项 | 契约、异常、事务、幂等、兼容 |
| 代码评审 | PR、diff、测试结果 | 评审报告、问题清单、合并建议 | 风险分层、红线、测试、旧逻辑兼容 |
| UI评审 | 原型、页面、交互、前端代码 | UI问题清单、可访问性结论 | 可访问性、安全、响应式、性能、可测试性 |

## 4. 严重级别与合并门禁

| 级别 | 合并策略 | 示例 |
|---|---|---|
| BLOCKER | 禁止合并，必须修复；无法立即修复时只能转人工裁决，不能自动通过 | SQL注入、越权、跨租户、资金/库存错误、破坏旧逻辑、核心路径无验证 |
| HIGH | 默认修复后合并；延期必须记录风险接受人和截止时间 | N+1导致核心接口不可用、并发风险、兼容性不明、测试缺失 |
| MEDIUM | 建议本次修复；可带记录合并 | 局部复杂度高、错误处理不足、日志不完整、非核心性能问题 |
| LOW | 可选优化 | 命名、注释、局部可读性、非关键重复代码 |

合并建议只允许使用：

```text
APPROVE：无 BLOCKER，HIGH 已修复或已正式接受，验证充分。
COMMENT：无 BLOCKER，但存在 MEDIUM/LOW 或轻量 HIGH，需要记录。
REQUEST CHANGES：存在 BLOCKER，或 HIGH 未处理且无风险接受。
NEEDS HUMAN DECISION：技术证据不足，需要业务/架构/安全负责人裁决。
```

严禁把“存在 BLOCKER 但已有修复计划”标为通过。修复计划不是修复结果。

## 5. 例外与风险接受

允许 HIGH/MEDIUM/LOW 例外，但必须可审计。BLOCKER 不走自动风险接受；确实无法立即修复时，结论必须是 NEEDS HUMAN DECISION。

```text
原因：为什么本次不能修复
影响：影响范围和最坏后果
责任：风险接受人
期限：补救截止时间
防护：临时监控、开关、回滚或人工操作
记录：写入评审报告
```

原则上不接受例外：跨租户、越权、SQL注入、资金/库存/支付错误、密钥泄露。

## 6. 工具链状态

`06-评审工具与自动化.md` 是工具模板和配置参考。任何工具成为门禁前，必须先在目标项目验证。

| 状态 | 含义 |
|---|---|
| 已验证可用 | 当前项目已配置并能稳定运行 |
| 模板示例 | 可复制参考，但需要项目适配 |
| 概念工具 | 表达方向，不能作为门禁依据 |

## 7. 度量指标

建议跟踪：

```text
评审响应时间：目标 24 小时内首次反馈。
评审缺陷逃逸率：评审未发现但测试/生产发现的问题比例。
BLOCKER修复周期：从发现到关闭的时间。
高风险例外数量：越少越好，必须可追踪。
关键路径验证覆盖率：L1/L2 默认 100% 证据覆盖。
```

## 8. 成熟度模型

| 级别 | 状态 |
|---|---|
| Level 1 初始级 | 无稳定评审流程，依赖个人经验 |
| Level 2 可重复级 | 有流程和清单，但执行不稳定 |
| Level 3 标准化级 | 风险分层、模板、门禁规则统一 |
| Level 4 量化管理级 | 有度量、趋势和复盘机制 |
| Level 5 优化级 | 自动化、知识图谱、AI辅助、陷阱回写形成闭环 |

## 9. 快速开始

人工代码评审：

```text
1. 读 03。
2. 用 05。
3. 必要时参考 06。
4. 输出统一评审报告。
```

AI代码评审：

```text
1. 读 09。
2. 按 09 加载既有系统安全修改规范和项目陷阱表。
3. 必要时参考 03、05、08。
4. 输出证据化评审报告。
```

架构评审：

```text
1. 读 01。
2. 准备候选方案、权衡矩阵和 ADR。
3. 指定魔鬼代言人。
4. 输出 ADR、风险清单、验证方式和后续行动。
```

## 10. 通用性

本体系的框架通用，但具体规范、工具和风险域需要裁剪。

| 场景 | 通用性 |
|---|---|
| 多租户 SaaS、CRM、ERP、订单、库存、支付系统 | 高 |
| 普通后端 Web 项目 | 中高 |
| 前端应用 | 中，需要补充 UI、可访问性、状态管理和 E2E 细则 |
| 小脚本、Demo、一次性工具 | 中低，应裁剪 |
| 嵌入式、实时系统、算法系统 | 中低，需要替换风险域和验证方式 |

结论：风险分层、证据驱动、兼容优先、例外可审计这些原则通用；项目条款和门禁阈值不应照搬。
