# 整理版：如何正确生成和金蝶 ERP 一模一样的表单界面

## 1. 文档目的

本文回答一个具体问题：

```text
如何根据业务需求，稳定生成一个接近金蝶 ERP / KDesign / 金蝶云星空风格的 PC 后台表单界面？
```

这里说的“一模一样”，不是复制金蝶的截图、品牌、图标或私有组件，而是复刻它的企业级表单设计方法：

```text
业务对象清楚。
单据头和单据体清楚。
字段分组符合业务决策顺序。
状态驱动字段、按钮和流程。
上下游关系可追溯。
权限、审计、附件、导入导出、打印都进入页面设计。
视觉克制、高密度、低装饰。
错误能定位到字段、分录行、权限、状态或业务规则。
页面能被自动化测试稳定定位和断言。
```

禁止把“像金蝶”理解成：

```text
照抄截图。
仿冒金蝶品牌。
只模仿颜色、表格和按钮。
把字段按数据库列平铺出来。
只做普通 CRUD 表单。
忽略状态、流程、权限、审计、上下游和测试契约。
```

金蝶 ERP 表单的本质不是“好看的输入框集合”，而是一个业务对象生命周期的录入、校验、流转和证据界面。

## 2. 一句话原则

生成金蝶 ERP 风格表单时，必须按下面顺序思考：

```text
业务域 -> 业务对象 -> 单据身份 -> 状态机 -> 字段语义 -> 单据头/体 -> 上下游 -> 操作权限 -> 反馈错误 -> 视觉密度 -> 测试契约
```

不要从视觉开始。

正确顺序是：

```text
先定义业务对象，再生成表单界面。
先定义状态和规则，再放按钮。
先定义字段语义，再排输入框。
先定义分录和校验，再画表格。
先定义上下游和证据，再做详情入口。
```

## 5. 表单区域生成规则

### 5.1 顶部标题区

顶部必须让用户一眼识别业务对象。

必须包含：

```text
单据名称。
单据编号。
状态标签。
所属组织。
业务日期。
入口来源。
当前流程节点或下一步。
主操作按钮。
更多操作入口。
```

生成要求：

```text
单据编号保存前可以显示“保存后生成”。
状态标签文案必须稳定。
状态颜色必须稳定。
主操作只能有 1 个，通常是保存、提交或审核。
风险操作放入更多菜单或二次确认弹窗，不放成高亮主按钮。
```

错误示例：

```text
页面标题只写“新增”。
状态只靠按钮变化表达。
所有按钮并排高亮。
删除、作废和提交视觉优先级相同。
```

### 5.2 上下文条

金蝶 ERP 表单通常非常重视业务上下文。上下文条用于说明当前表单为什么是这个范围、这个组织、这个入口。

建议展示：

```text
当前组织。
当前账套或门店。
业务日期。
来源类型。
来源单据。
当前模式：新增 / 编辑 / 只读 / 审批。
权限提示。
```

设计要求：

```text
如果字段由上游带入，必须让用户知道来源。
如果当前状态导致不可编辑，必须显示原因。
如果当前用户无权限，不能伪装成普通禁用。
```

### 5.3 基本信息区

基本信息区放整单级字段。

常见字段：

```text
业务组织。
业务日期。
往来单位。
部门。
经办人。
单据类型。
来源单据。
结算方式。
币别。
备注。
```

字段排序原则：

```text
按用户决策顺序排列，不按数据库顺序排列。
先放决定业务范围的字段。
再放决定对象关系的字段。
再放决定金额、库存、结算、审批的字段。
最后放备注和低频扩展字段。
```

推荐布局：

```text
PC 宽屏：3-4 列表单。
普通后台：3 列优先。
字段名右对齐或固定宽度。
输入框高度 32px。
行距紧凑但不拥挤。
长文本、备注、附件单独成行。
```

### 5.4 业务信息区

业务信息区放会影响流程、计算、下游、报表的字段。

例如：

```text
采购类型。
销售类型。
结算方式。
税率口径。
库存方向。
是否含税。
项目。
成本中心。
业务分类。
凭证模板分类。
```

生成要求：

```text
高风险字段必须说明业务影响。
影响下游转换的字段必须可解释。
影响账期、凭证、成本、结算的字段必须在标签、提示或说明中表达。
```

不要只生成：

```text
业务分类：[下拉框]
```

应该生成：

```text
业务分类：[下拉框]
提示：用于匹配下游凭证模板和报表口径。
```

### 5.5 明细分录表

ERP 表单的核心通常是分录表。

分录表不是普通表格，而是行级业务事实录入区。

必须支持：

```text
新增行。
删除行。
复制行。
批量填充。
行级校验。
行级错误定位。
金额和数量自动计算。
合计。
来源行追踪。
```

典型列顺序：

```text
行号。
物料 / 商品 / 服务。
规格型号。
批次。
仓库。
单位。
数量。
单价。
金额。
税率。
税额。
来源单据行。
备注。
行操作。
```

生成要求：

```text
识别列在前：编码、名称、规格。
业务判断列居中：仓库、批次、单位、数量。
金额计算列靠后：单价、金额、税额。
行操作最后。
数字右对齐。
金额保留统一小数位。
行错误必须标到具体单元格。
```

错误反馈示例：

```text
第 3 行“实收数量”不能为空。
第 5 行“实收数量”不能大于应收数量。
第 2 行“仓库”无当前组织权限。
```

禁止只提示：

```text
保存失败。
参数错误。
明细不完整。
```

### 5.6 合计区

表单存在数量、金额、税额、折扣、已收、未收等字段时，必须有合计区。

推荐展示：

```text
数量合计。
金额合计。
税额合计。
价税合计。
已执行数量。
未执行数量。
已收 / 已付。
未收 / 未付。
```

设计要求：

```text
合计区固定在分录表下方。
长表格可吸底显示关键合计。
金额字段必须说明口径：含税 / 不含税 / 暂估 / 核算 / 应收 / 已收 / 未收。
```

### 5.7 附件区

附件是 ERP 单据证据，不是装饰功能。

必须定义：

```text
附件类型。
是否必传。
是否参与审批。
谁能查看。
谁能删除。
归档后是否只读。
```

常见附件：

```text
合同。
发票。
付款凭证。
收货照片。
报销票据。
质检报告。
审批材料。
```

### 5.8 关联关系区

金蝶式表单必须能追溯上下游。

应展示：

```text
来源单据。
来源单据行。
下游单据。
凭证。
结算记录。
库存影响。
反写状态。
```

生成要求：

```text
手工新增时也要预留关联区。
从上游下推生成时必须展示来源。
部分下推时必须展示已执行和未执行。
下游生成失败必须说明失败原因。
```

### 5.9 审批与日志区

审批区不是所有表单都默认展开，但必须有入口。

应包含：

```text
当前审批节点。
当前处理人。
审批历史。
审批意见。
驳回原因。
是否超期。
撤回 / 转交 / 加签入口。
```

日志区应记录：

```text
创建。
修改。
提交。
审核。
反审核。
作废。
删除。
导入。
导出。
打印。
接口同步。
权限变更。
```

## 6. 字段生成规则

### 6.1 字段不是数据库列

每个字段都必须有业务语义。

生成字段时必须回答：

```text
字段属于单据头还是单据体？
字段由用户录入、系统计算、上游带入还是接口同步？
字段什么时候必填？
字段什么时候可编辑？
字段影响哪个状态或流程？
字段影响下游转换吗？
字段影响报表、凭证、结算或库存吗？
字段是否受权限控制？
字段是否可导入、导出、打印？
```

### 6.2 字段分组标准

推荐分组：

```text
基本信息：对象身份、组织、日期、人员、往来单位。
业务信息：业务类型、结算方式、税率、仓库、项目、成本中心。
明细分录：行级对象、数量、金额、批次、来源行。
关联信息：来源、下游、凭证、结算、库存影响。
审批信息：节点、处理人、意见、时间。
附件备注：合同、票据、图片、说明。
系统信息：创建人、创建时间、修改人、版本号。
```

### 6.3 必填规则

ERP 表单必填不能只靠红星。

必须区分：

```text
新增时必填。
保存时必填。
提交时必填。
审核时必填。
下推时必填。
打印时必填。
生成凭证时必填。
```

示例：

```text
仓库：保存时可为空，提交前必填。
业务分类：保存时可为空，生成凭证前必填。
附件：提交报销时必传，普通暂存不必传。
```

### 6.4 编辑性规则

字段可编辑性必须受状态驱动。

示例：

```text
草稿：大部分字段可编辑。
已提交：业务字段只读，可撤回后编辑。
审核中：审批意见可编辑，单据字段只读。
已审核：字段只读，只允许下推、打印、反审核。
已关闭：所有业务字段只读。
已作废：只读，并显示作废原因。
```

## 7. 状态机与操作规则

### 7.1 常见表单状态

推荐状态全集：

```text
initial。
loading。
draft。
saved。
submitting。
submitted。
auditing。
approved。
rejected。
closed。
cancelled。
error。
noPermission。
dirty。
saving。
saveFailed。
```

### 7.2 状态驱动按钮

按钮不能只按角色硬编码，必须同时受状态、权限、数据和流程控制。

示例：

| 状态 | 可见主操作 | 可见次操作 | 风险操作 |
|---|---|---|---|
| draft | 保存 / 提交 | 暂存、导入 | 删除 |
| saved | 提交 | 编辑、打印 | 删除 |
| auditing | 审核 / 驳回 | 查看流程 | 撤回 |
| approved | 下推 | 打印、导出 | 反审核、作废 |
| closed | 无 | 打印、导出、查看日志 | 无 |
| cancelled | 无 | 查看日志 | 无 |

### 7.3 操作反馈

每个操作必须有完整反馈链：

```text
点击前：能否执行、为什么不能执行。
点击中：loading、禁用重复点击。
点击后成功：状态变化、结果说明、下一步入口。
点击后失败：具体原因、定位、恢复方式。
```

高风险操作必须：

```text
二次确认。
说明影响范围。
校验权限。
记录日志。
支持失败恢复。
```

## 8. 视觉与布局规则

### 8.1 总体视觉

目标风格：

```text
企业级。
克制。
高密度。
清晰。
专业。
低装饰。
强信息层级。
```

禁止：

```text
营销页风格。
大面积渐变。
大圆角卡片堆叠。
装饰插画主导。
移动端卡片硬搬到 PC。
所有区域都做成浮动卡片。
过度留白导致信息密度过低。
```

```text
页面左右边距：24px。
区域间距：12-16px。
表单项高度：32px。
```

```text
宽屏：4 列。

低频长字段：独占一行。
备注、附件、说明：独占一行。
明细表：独占整行。
```

### 8.5 按钮布局

```text
页面顶部放全局操作。
页面底部吸底放保存、提交、取消等表单动作。
主按钮最多一个。
次按钮使用普通样式。
风险按钮不用主色，放更多菜单或确认弹窗。
只读详情态不显示保存按钮。
```

## 9. 反馈和错误规范

### 9.1 反馈类型

| 场景 | 推荐反馈 |
|---|---|
| 保存成功 | 轻量 message |
| 提交成功 | message + 状态变化 |
| 字段错误 | 字段级错误 |
| 分录错误 | 单元格错误 + 行定位 |
| 批量失败 | 结果面板 + 失败明细 |
| 权限不足 | 禁用说明 / 无权限页 |
| 状态不允许 | 明确说明当前状态和所需状态 |
| 网络异常 | error panel + 重试 |
| 长任务 | 进度条 / 任务中心 |
| 风险确认 | modal |

### 9.2 错误结构

推荐错误契约：

```ts
type FormError = {
  code: string;
  scope: 'page' | 'field' | 'row' | 'action' | 'permission' | 'state' | 'network';
  field?: string;
  rowIndex?: number;
  action?: string;
  message: string;
  recoverable: boolean;
  retryAction?: string;
};
```

错误必须能回答：

```text
哪里错了？
为什么错？
怎么修？
能不能重试？
是否需要权限或状态变化？
```

## 10. 权限、审计和治理

表单必须考虑以下权限：

```text
页面访问权限。
```

```text
无权限页：页面级访问被拒绝。
```

## 12. 生成 Prompt 模板

可以直接使用下面模板生成金蝶 ERP 风格表单页。

```text
你是企业级 ERP 产品设计师和前端架构师。

请生成一个 PC 后台 ERP 表单界面，风格参考金蝶 ERP / KDesign / 金蝶云星空的企业管理界面方法，但不要复制金蝶品牌、截图、图标或私有资产。

业务信息：
- 页面名称：{pageName}
- 业务域：{businessDomain}
- 业务对象：{objectName}
- 页面模式：{create/edit/readonly/approve}
- 使用角色：{primaryUser}
- 入口来源：{entry}
- 状态全集：{states}
- 主要操作：{actions}
- 风险操作：{riskActions}
- 上游对象：{upstream}
- 下游对象：{downstream}

表单结构必须包含：
1. 顶部标题区：单据名称、单据编号、状态、组织、业务日期、主操作、更多操作。
2. 上下文条：来源类型、来源单据、当前模式、权限或状态提示。
3. 基本信息区：按业务决策顺序排列，不按数据库字段顺序。
4. 业务信息区：展示会影响流程、金额、库存、凭证、报表的字段。
5. 明细分录表：支持新增行、删除行、复制行、行级校验、单元格错误、合计。
6. 合计区：展示数量、金额、税额等关键口径。
7. 附件区：支持附件类型、权限、审批关联。
8. 关联关系区：展示来源单据、下游单据、凭证、结算、反写状态。
9. 审批和日志区：展示审批节点、处理人、历史意见、操作日志。
10. 底部吸底操作栏：取消、暂存、保存、提交。

视觉要求：
- 浅灰页面背景。
- 白色业务面板。
- 低对比边框。
- 蓝色主色。
- 小圆角 4-6px。
- 表单项高度 32px。
- 表格行高 40-44px。
- PC 高密度布局，优先 3-4 列表单。
- 主按钮最多一个。
- 风险操作降级，不要和主操作同等强调。

状态和权限要求：
- 状态必须驱动字段编辑性和按钮可用性。
- 保存、提交、审核、反审核、作废必须有不同反馈。
- 无权限、状态不允许、字段错误、分录错误、网络错误必须分开表达。
- 高风险操作必须二次确认、显示影响范围并留痕。

测试契约要求：
- 页面包含 data-page-id、data-page-type、data-pattern、data-object-type、data-mode、data-state。
- 字段包含 data-field、data-field-group、data-required-phase、data-editable、data-source。
- 分录行包含 data-line-index、data-line-id、data-row-state。
- 按钮包含 data-action、data-action-state、data-permission、data-risk-level。
- 错误包含 data-error-code、data-error-scope、data-error-field、data-error-row-index。

不要：
- 不要生成营销页。
- 不要大面积渐变。
- 不要大圆角卡片堆叠。
- 不要移动端卡片列表硬搬到 PC。
- 不要只做字段堆叠。
- 不要忽略状态、权限、上下游、审批、日志、附件和测试属性。
```

## 13. 表单生成后验收清单

生成后按下面清单检查。

```text
1. 是否能一眼识别业务对象？
2. 是否展示单据编号、状态、组织、业务日期和入口来源？
3. 是否区分单据头和单据体？
4. 字段是否按业务语义分组，而不是按数据库顺序？
5. 是否说明关键字段来源、必填阶段、编辑条件和业务影响？
6. 是否有明细分录表，并支持行级错误定位？
7. 是否有数量、金额、税额等合计区？
8. 状态是否驱动字段和按钮？
9. 保存、提交、审核、反审核、作废是否有不同操作反馈？
10. 是否展示来源单据、下游单据、凭证、结算或反写状态？
11. 是否包含附件、审批记录、操作日志入口？
12. 权限不足时是否有合理表现？
13. 风险操作是否二次确认、说明影响范围并留痕？
14. 错误是否能定位到字段、行、权限、状态或业务规则？
15. 视觉是否克制、高密度、企业级？

17. 是否包含稳定测试属性？

```

## 14. 常见错误和修正方式

### 14.1 只像普通 CRUD

错误：

```text
顶部一个标题。
中间一堆输入框。
底部保存按钮。
```

修正：

```text
补业务身份、状态、单据头、分录、合计、附件、上下游、审批、日志、权限、测试契约。
```

### 14.2 字段按数据库顺序排列

错误：

```text
id、tenantId、createTime、updateTime、supplierId、billDate...
```

修正：

```text
按业务决策顺序：组织 -> 日期 -> 往来单位 -> 业务类型 -> 明细 -> 金额 -> 附件 -> 备注。
```

### 14.3 状态只是标签

错误：

```text
显示“已审核”，但保存按钮仍可点击。
```

修正：

```text
已审核状态下业务字段只读，保存隐藏，只保留下推、打印、导出、反审核等符合状态的操作。
```

### 14.4 错误无法定位

错误：

```text
保存失败，请检查。
```

修正：

```text
第 3 行“仓库”不能为空；第 5 行“实收数量”不能大于应收数量；当前状态已审核，不允许编辑供应商。
```

### 14.5 缺少上下游

错误：

```text
只显示当前表单字段。
```

修正：

```text
展示来源采购订单、来源行号、已下推应付单、未下推数量、反写状态和凭证状态。
```

## 15. 最终结论

要生成接近金蝶 ERP 的表单界面，核心不是“模仿外观”，而是生成一套企业单据系统界面。

必须同时具备：

```text
业务对象。
单据身份。
单据头。
单据体。
字段语义。
状态机。
操作权限。
上下游关系。
审批日志。
附件证据。
导入导出打印。
错误恢复。
测试契约。
克制高密度视觉。
```

如果缺少这些，只是一个普通后台表单；即使颜色和按钮像金蝶，也不是真正的金蝶 ERP 风格。

真正的金蝶式表单应该让用户做到：

```text
看得懂对象。
填得对字段。
改得动状态。
追得出来源。
查得到下游。
错得明白。
批量做得稳。
权限管得住。
审计留得下。
测试跑得起。
```
