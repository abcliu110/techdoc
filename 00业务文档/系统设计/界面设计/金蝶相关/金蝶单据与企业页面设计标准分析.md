
## 3. 金蝶式单据的本质

金蝶产品手册对单据的定义很关键：业务单据是业务对象，用于保存业务流程中发生的数据，并承载业务流程之间的逻辑关系。

这句话可以拆成四层：

```text
业务对象：单据不是一个页面，而是一个业务对象。
过程数据：单据记录业务发生过程中的事实。
逻辑关系：单据之间有上游、下游、转换、关联、凭证、结算、库存、流程关系。
业务入口：单据页面只是业务对象的一个操作入口。
```

所以企业级单据设计不是“画表单”，而是设计一个业务对象的全生命周期界面。

一个合格的单据页面至少要回答：

```text
这是什么单据？
属于哪个业务域？
当前处于什么状态？
它从哪里来？
它能流向哪里？
谁可以操作？
哪些字段是业务事实？
哪些字段是系统计算？
哪些字段来自上游单据？
哪些字段影响下游单据？
哪些动作会改变状态？
哪些动作只是辅助查看？
如何审批？
如何打印？
如何导入导出？
如何追溯？
如何测试？
```

## 4. 金蝶单据模型拆解

从金蝶公开资料看，单据不是孤立页面，而是由一组元数据和业务能力组成。

### 4.1 单据基础身份

一个企业级单据必须先有身份。

| 维度 | 说明 | 页面要求 |
|---|---|---|
| 单据类型 | 如采购入库单、销售出库单、应收单、应付单、费用报销单 | 页面标题、路由、埋点、测试选择器必须能识别 |
| 单据编号 | 对外可沟通、可搜索、可追溯的业务编号 | 页面显著位置展示，支持复制，测试可读取 |
| 业务组织 | 单据归属哪个组织、门店、部门、公司 | 必须暴露并参与权限、筛选、审批 |
| 业务日期 | 业务发生日期，不一定等于创建日期 | 必须说明影响账期、结算、统计、报表 |
| 单据状态 | 保存、提交、审核、关闭、作废等 | 状态标签必须清晰、稳定、可自动化识别 |
| 来源类型 | 手工新建、上游下推、导入、系统生成、接口同步 | 影响字段可编辑性、回退方式和异常解释 |

小程序页面对应要求：

```text
页面必须定义 pageId。
页面必须定义 bizType。
页面必须暴露 bizNo 或可展示业务编号。
页面必须展示当前业务状态。
页面必须说明入口来源 entry。
页面必须让测试框架读取这些身份信息。
```

### 4.2 字段与字段分组

金蝶单据通过常规字段、通用控件、字段属性、控件属性来构造单据模板。这说明字段不是“摆上去就行”，字段必须有结构。

字段至少分为：

```text
主信息字段：单据编号、业务日期、业务组织、往来单位、经办人。
业务字段：物料、商品、客户、供应商、金额、数量、税率、仓库、门店。
状态字段：单据状态、审核状态、关闭状态、结算状态、核算状态。
来源字段：上游单据号、来源类型、来源行号、接口来源。
计算字段：合计金额、成本金额、毛利、税额、应收应付金额。
控制字段：是否必填、是否可编辑、是否可见、是否参与流程。
扩展字段：项目、备注、自定义字段、附件。
```

字段分组原则：

| 分组 | 放什么 | 不应放什么 |
|---|---|---|
| 单据头 | 影响整单的主数据和控制字段 | 分录行级字段 |
| 明细分录 | 数量、金额、物料、商品、仓库、批次等行级数据 | 整单审批意见 |
| 关联信息 | 上游单据、下游单据、凭证、结算记录 | 主要录入字段 |
| 审批信息 | 审批流、当前节点、审批意见、处理人 | 业务明细字段 |
| 附件备注 | 图片、票据、合同、备注 | 核心金额字段 |
| 系统信息 | 创建人、创建时间、修改人、版本、来源 | 用户高频录入字段 |

小程序页面对应要求：

```text
字段不能只是视觉排列，必须有信息分区。
核心字段必须优先展示。
次要字段可以折叠，但不能消失。
来源、状态、金额、业务对象字段必须可测试读取。
字段的编辑性必须和当前状态绑定。
```

### 4.3 单据状态

金蝶单据设计过程包含“单据状态”配置。企业单据的状态不是装饰标签，而是页面行为的主控因素。

典型状态包括：

```text
draft：草稿 / 暂存
saved：已保存
submitted：已提交
auditing：审核中
approved：已审核
rejected：已驳回
closed：已关闭
cancelled：已作废
converted：已下推 / 已转换
settled：已结算
posted：已生成凭证
```

状态决定：

```text
哪些字段可编辑。
哪些按钮可点击。
哪些错误可恢复。
哪些信息必须展示。
是否允许导入导出。
是否允许打印。
是否允许重新提交。
是否允许撤销或反审核。
是否允许下推下游单据。
```

这正好对应前面小程序规范中的“页面像一个小型状态机”。

一个单据页面的状态机可以这样表达：

```text
[loading]
   |
   v
[draft] --save--> [saved]
   |                |
   | submit         | submit
   v                v
[submitting] --> [auditing] --approve--> [approved] --close--> [closed]
                     |
                     | reject
                     v
                  [rejected] --edit--> [draft]

[approved] --convert--> [converted]
[approved] --postVoucher--> [posted]
[approved] --cancel--> [cancelled]
```

页面不能只写：

```text
if status == 1 show button A
if status == 2 show button B
```

企业级做法应该是：

```text
状态定义当前页面的可见内容、可执行动作、可编辑字段、错误恢复路径和测试断言。
```

### 4.4 单据菜单和操作

金蝶单据模板会设置菜单集合，客户端主控台会通过工具栏按钮对业务单据进行通用操作。这说明企业单据操作需要被标准化。

常见操作可分为：

| 类型 | 操作 | 设计要求 |
|---|---|---|
| 录入类 | 新增、保存、暂存、复制 | 明确保存结果，失败可恢复 |
| 流程类 | 提交、审核、驳回、撤回、反审核 | 必须展示状态变化和审批结果 |
| 转换类 | 下推、生成、关联、转换 | 必须展示上游/下游关系 |
| 财务类 | 生成凭证、结算、核销、获取成本 | 必须说明影响范围和来源 |
| 数据类 | 导入、导出、批量处理 | 必须支持结果反馈和失败明细 |
| 查看类 | 打印、套打、附件、日志、关联查询 | 不应改变主业务状态 |
| 风险类 | 作废、关闭、删除 | 必须二次确认、权限校验、留痕 |

小程序页面对应要求：

```text
主操作只能有一个。
次操作必须分组。
风险操作必须降低视觉优先级。
所有会改变状态的操作必须有 loading、success、fail、rollback 或 retry 状态。
测试框架必须能识别点击后页面进入了哪个状态。
```

### 4.5 单据转换和上下游关系

金蝶单据设计过程包含“单据转换”。社区问答中也出现了凭证模板业务分类、字段映射、上游单据日期映射、销售出库单下推其他应付款单等问题。

这说明企业单据页面必须显式处理上下游关系。

上下游关系包括：

```text
申请单 -> 订单
订单 -> 出库单
出库单 -> 应收单
应收单 -> 发票
应收单 -> 凭证
入库单 -> 应付单
费用申请 -> 费用报销 -> 付款
```

页面设计要求：

| 项目 | 设计要求 |
|---|---|
| 来源单据 | 明确展示来源单据号、来源类型、来源行 |
| 下游单据 | 明确展示已生成哪些下游对象 |
| 字段映射 | 用户能理解关键字段从哪里来 |
| 转换条件 | 不能让用户靠猜测判断为什么不能下推 |
| 失败原因 | 必须具体到字段、行、权限、状态或业务规则 |
| 链路追溯 | 支持查看单据关系图或关联记录 |

如果一个页面只显示“操作失败”，但不告诉用户是：

```text
缺少销售员。
业务分类不匹配。
状态不允许下推。
上游单据未审核。
字段映射缺失。
凭证模板没有覆盖当前分类。
```

那么它就不符合企业级单据设计标准。

### 4.6 单据视图

金蝶手册中单据模板设计后还需要创建单据视图。视图意味着同一个业务对象可以有不同呈现方式。

常见视图：

```text
新增视图：聚焦录入，减少干扰。
编辑视图：展示可修改字段和不可修改原因。
详情视图：展示完整事实、状态和关联关系。
审批视图：突出审批意见、风险字段、变更点。
打印视图：满足正式输出格式。
移动视图：保留关键字段，简化低频信息。
列表视图：用于检索、筛选、批量操作。
```

小程序页面对应要求：

```text
不要把录入、详情、审批、结果页混成一个杂乱页面。
同一个页面可以通过 mode 表达视图意图。
mode 必须进入页面契约和测试契约。
不同 mode 下字段、按钮、状态标签必须可验证。
```

示例：

```ts
type PageMode = 'create' | 'edit' | 'readonly' | 'approve' | 'result';

const pageContract = {
  pageId: 'expenseBill',
  modes: ['create', 'edit', 'readonly', 'approve', 'result'],
  defaultModeByEntry: {
    fromCreateButton: 'create',
    fromTodo: 'approve',
    fromList: 'readonly'
  }
};
```

## 5. 金蝶典型页面设计拆解

KDesign 明确覆盖 PC 典型页面和移动典型页面。公开可访问的信息中，列表页、表单页、详情页、工作台是最值得迁移到小程序规范中的模式。

### 5.1 列表页

KDesign 列表页强调：

```text
易读性：格式一致，突出对象识别关键信息。
可寻性：便于浏览、排序、查找。
分层展示：通过交互减少认知负荷。
```

企业列表页不是简单的 `table` 或卡片列表，而是“对象检索和批量处理中心”。

合格列表页必须包含：

| 能力 | 标准 |
|---|---|
| 对象识别 | 每一行能快速识别这是什么业务对象 |
| 关键状态 | 状态标签明确，颜色和文案稳定 |
| 关键字段 | 编号、名称、组织、日期、金额、负责人、状态优先展示 |
| 查询筛选 | 支持业务常用筛选，不只支持关键词 |
| 排序逻辑 | 默认排序符合业务任务，例如待处理优先、最新优先、异常优先 |
| 批量操作 | 批量操作前后有数量、范围、失败明细 |
| 空状态 | 告诉用户为什么为空，以及下一步能做什么 |
| 异常态 | 网络、权限、筛选无结果、数据加载失败要区分 |

文本图：

```text
┌──────────────────────────────────────┐
│ 页面标题 + 统计/状态摘要               │
├──────────────────────────────────────┤
│ 搜索框 | 筛选 | 状态 | 日期 | 更多条件   │
├──────────────────────────────────────┤
│ 批量操作栏 / 已选 N 条 / 风险提示        │
├──────────────────────────────────────┤
│ 编号     对象       状态    金额   操作  │
│ SO001   客户A      待审核  1000   查看  │
│ SO002   客户B      已驳回  300    修改  │
└──────────────────────────────────────┘
```

小程序卡片式列表：

```text
┌──────────────────────────────┐
│ 搜索 / 扫码 / 筛选             │
├──────────────────────────────┤
│ 待处理 12 | 已完成 88 | 异常 3  │
├──────────────────────────────┤
│ 单据号 SO001          待审核    │
│ 客户 A                         │
│ 金额 1000    日期 2026-07-03   │
│ [查看] [审批]                  │
└──────────────────────────────┘
```

### 5.2 表单页

KDesign 表单页强调字段按内容关联性分组，选择正确输入形式，并通过预填减少操作步骤。

企业表单页的目标不是让字段“摆得整齐”，而是让用户以最小错误成本完成一项业务录入。

合格表单页必须包含：

| 能力 | 标准 |
|---|---|
| 字段分组 | 按业务语义分区，不按数据库字段顺序堆叠 |
| 必填解释 | 关键必填项必须能解释为什么必填 |
| 默认值 | 可从用户、组织、入口、上游单据推导的字段应预填 |
| 控件匹配 | 日期用日期选择，金额用金额输入，枚举用选择器，人员用组织人员选择 |
| 行级录入 | 分录必须支持增删行、复制行、校验行、定位错误行 |
| 即时校验 | 格式错误、必填缺失、业务约束尽早提示 |
| 提交校验 | 跨字段、跨行、跨单据规则在提交时统一说明 |
| 草稿能力 | 长表单必须支持暂存、恢复、离开保护 |

表单页文本图：

```text
┌──────────────────────────────────────┐
│ 单据标题             草稿/待提交       │
├──────────────────────────────────────┤
│ 基本信息                              │
│ 客户       业务日期     经办人          │
│ 组织       部门         来源            │
├──────────────────────────────────────┤
│ 明细信息                              │
│ 商品      数量     单价     金额        │
│ 商品      数量     单价     金额        │
├──────────────────────────────────────┤
│ 附件 / 备注 / 扩展信息                 │
├──────────────────────────────────────┤
│ [暂存] [提交]                         │
└──────────────────────────────────────┘
```

### 5.3 详情页

KDesign 详情页用于展示一个对象的完整信息，也可以认为是表单录入结果呈现，主要用于信息浏览，并可承载审批、打印、导出等简单操作。

企业详情页的本质是“业务事实阅读器”。

合格详情页必须包含：

| 能力 | 标准 |
|---|---|
| 对象身份 | 标题、编号、状态、组织、日期必须清晰 |
| 事实完整 | 业务事实、分录、金额、附件、审批、日志完整展示 |
| 关联关系 | 上游、下游、凭证、结算、出入库、发票等可追溯 |
| 状态解释 | 当前状态为什么是这样，下一步能做什么 |
| 操作克制 | 详情页只放与当前状态相关的关键操作 |
| 审批友好 | 审批视图突出变更、风险、金额、异常、历史意见 |
| 输出能力 | 打印、导出、分享、复制编号等低风险操作可达 |

详情页文本图：

```text
┌──────────────────────────────────────┐
```

### 5.4 工作台

KDesign 典型页面中包含工作台。金蝶云苍穹强调角色、任务、实时、极简，这意味着工作台不是门户装饰，而是角色任务入口。

合格工作台必须包含：

| 能力 | 标准 |
|---|---|
| 角色化 | 不同角色看到不同任务、指标和入口 |
| 任务优先 | 待办、异常、预警、审批优先于宣传内容 |
| 数据摘要 | 关键数字直接支撑业务判断 |
| 快捷入口 | 高频业务可一跳进入 |
| 异常可达 | 预警、失败、超期、待处理能直接进入列表 |
| 可追溯 | 从指标能追到明细，从明细能追到单据 |

工作台文本图：

```text
├──────────────────────────────────────┤

├──────────────────────────────────────┤
│ 关键指标卡片                           │
│ 销售额 | 应收 | 库存预警 | 待结算        │
├──────────────────────────────────────┤
│ 待办列表 / 异常列表 / 最近单据           │
└──────────────────────────────────────┘
```

## 6. 金蝶社区单据问题反推的设计标准

金蝶社区公开问答中能看到很多真实企业用户遇到的问题。这些问题不是单纯的“不会操作”，而是能反推企业单据页面应该更强的地方。

### 6.1 凭证模板业务分类

社区问答中提到，同一个单据可能因为业务分类不同而使用不同凭证模板；如果某类业务没有配置凭证模板，凭证生成时可能选择不到该类单据。

这反推一个标准：

```text
当一个字段会影响下游流程、凭证、结账、报表或校验时，页面必须把该字段视为“流程控制字段”，不能只当普通字段。
```

页面要求：

| 项目 | 要求 |
|---|---|
| 业务分类 | 显示当前分类，并说明分类影响 |
| 凭证状态 | 显示是否已生成凭证、是否必须生成 |
| 模板覆盖 | 不能生成时说明当前分类没有模板或条件不匹配 |
| 测试覆盖 | 每个业务分类至少有一条下游流程测试 |

不符合标准的设计：

```text
用户点“生成凭证”后只提示“无可生成单据”。
```

符合标准的设计：

```text
当前单据业务分类为 C。
当前凭证模板仅覆盖 A、B。
C 不参与凭证生成，也不会触发结账检查。
```

### 6.2 批量处理和必填字段

社区问答中出现批量接收银行承兑汇票时，因为销售员、销售部门必填而需要逐张进入单据维护的问题。

这反推一个标准：

```text
企业页面不能只满足单条操作，还要评估批量场景是否顺畅。
```

页面要求：

| 项目 | 要求 |
|---|---|
| 批量前校验 | 批量操作前预检查缺失字段 |
| 批量补全 | 支持批量设置销售员、部门等共同字段 |
| 失败明细 | 批量失败必须返回具体单据、字段、原因 |
| 模板引入 | 支持导出模板、补充字段、再导入 |
| 必填策略 | 必填字段要区分创建必填、提交必填、审核必填、下推必填 |

不符合标准的设计：

```text
保存失败：必填字段为空。
```

符合标准的设计：

```text
共 50 张票据，37 张缺少销售员，37 张缺少销售部门。
可选择批量补全，也可导出失败明细。
```

### 6.3 业务日期

社区问答中提到业务日期表示业务发生日期，可能影响账期和应付账款到期日。

这反推一个标准：

```text
看似普通的日期字段，可能是账期、统计、报表、结算、到期计算的核心依据。
```

页面要求：

| 项目 | 要求 |
|---|---|
| 日期语义 | 明确创建日期、业务日期、审核日期、记账日期区别 |
| 默认来源 | 显示日期默认来自系统、上游单据还是用户选择 |
| 影响提示 | 修改业务日期时提示可能影响账期或到期日 |
| 测试覆盖 | 日期变更应覆盖账期计算和报表归属 |

不符合标准的设计：

```text
页面上只有一个“日期”，用户不知道它影响什么。
```

符合标准的设计：

```text
业务日期：用于账期和到期日计算。
创建日期：系统记录，不参与业务账期。
```

### 6.4 字段映射和下推

社区问答中出现销售出库单下推其他应付款单时，往来单位类型和往来单位需要通过字段映射或公式确定。

这反推一个标准：

```text
单据下推不是复制页面字段，而是执行一套可解释的字段映射规则。
```

页面要求：

| 项目 | 要求 |
|---|---|
| 映射可解释 | 关键字段来源可查看 |
| 映射失败可定位 | 失败要说明哪个字段无法映射 |
| 字段来源 | 显示字段来自用户录入、上游单据、系统公式还是默认配置 |
| 测试覆盖 | 下推链路必须验证字段映射结果 |

不符合标准的设计：

```text
下推失败。
```

符合标准的设计：

```text
下推失败：其他应付款单.往来单位类型无法确定。
来源单据字段：销售出库单.供应商。
建议：配置字段映射或补充供应商信息。
```

### 6.5 成本金额

社区问答中提到应收单上的成本金额可能用于利润分析，最终成本仍以核算后的成本金额为准。

这反推一个标准：

```text
金额字段必须区分“业务结算依据、展示分析依据、暂估值、最终核算值”。
```

页面要求：

| 项目 | 要求 |
|---|---|
| 金额口径 | 明确金额字段的业务口径 |
| 来源说明 | 标明来自录入、计算、核算、上游单据还是接口 |
| 是否影响主账 | 告诉用户是否影响结账、对账、报表 |
| 更新时间 | 对暂估或核算类金额显示更新时间或状态 |
| 测试覆盖 | 金额字段测试不能只断言显示值，要断言业务影响 |

不符合标准的设计：

```text
成本金额：100。
```

符合标准的设计：

```text
成本金额：100。
来源：出库单核算后带入。
用途：用于毛利分析；最终成本以存货核算结果为准。
```

## 7. 金蝶标准与我们页面评价体系的映射

| 金蝶设计要点 | 我们的小程序规范映射 | 测试规范映射 |
|---|---|---|
| 单据是业务对象 | 页面必须有 pageId、bizType、bizNo、bizStatus | 测试必须能读取业务身份 |
| 单据状态 | 页面必须是状态机 | 测试必须覆盖状态全集 |
| 单据视图 | 页面必须定义 mode | 测试必须覆盖 create/edit/detail/approve/result |
| 菜单集合 | 页面必须定义 actions | 测试必须断言动作可见、可点、结果状态 |
| 字段属性 | 页面必须定义字段分组、必填、编辑性、来源 | 测试必须覆盖字段校验和异常定位 |
| 单据转换 | 页面必须暴露上下游关系 | 测试必须覆盖下推、关联、映射 |
| 编码规则 | 页面必须展示业务编号 | 测试必须使用外部业务编号定位 |
| 工作流 | 页面必须展示审批节点和下一步 | 测试必须覆盖提交、审批、驳回 |
| 引入引出 | 页面必须支持批量反馈 | 测试必须覆盖批量成功、部分失败 |
| 附件 | 页面必须定义附件类型和约束 | 测试必须覆盖附件上传、预览、删除 |
| 通用过滤 | 列表页必须有查询筛选契约 | 测试必须覆盖筛选、排序、空态 |
| 个性化选项 | 页面扩展要有兼容边界 | 测试必须覆盖默认配置和扩展配置 |

### 7.1 经典页面骨架必须单独识别

前面的分析重点是单据对象、状态、流程、字段和企业治理。实际做原型设计时，还必须先判断页面应采用哪一种经典骨架。

企业系统常见骨架包括：

```text
左树右表：先选组织、分类、区域、仓库，再看右侧数据。
左树右详情：先选层级节点，再维护节点详情。
左列表右详情：先选待办、消息、工单，再处理当前对象。
上筛下表：先筛选，再查看列表结果。
主从表：上方主对象，下方明细分录。
三栏结构：队列、对象列表、详情处理并列。
分步向导：复杂任务按步骤完成。
工作台：角色任务、待办、指标、快捷入口组合。
看板：按状态列推进对象。
矩阵表：行列交叉表达规则、权限、价格或排班。
树表一体：表格行本身带层级。
图表钻取：从指标总览下钻到业务明细。
时间轴详情：按时间展示事件链。
双栏穿梭：从可选集合分配到已选集合。
```

其中“左树右表”是金蝶类企业系统、ERP、SaaS 后台中最常见的原型结构之一。它不是简单布局，而是表达一种业务逻辑：

```text
先确定业务范围，再处理范围内对象。
```

例如：

```text
组织树 -> 员工表
商品分类树 -> 商品表
门店区域树 -> 门店表
仓库树 -> 库存表
权限菜单树 -> 权限点表
```

经典原型结构已经单独整理到：

```text
D:\mywork\techdoc\00业务文档\系统设计\小程序相关\企业级经典页面原型结构模式.md
```

后续评价页面时，不能只问“页面字段是否完整”，还要先问：

```text
这个业务场景选的页面骨架对不对？
应该是左树右表，还是上筛下表？
应该是主从表，还是详情页明细分组？
应该是看板，还是普通列表？
应该是分步向导，还是一次性表单？
```

## 8. 是否符合企业 UI 标准的判断模型

判断一个单据或页面是否符合企业级标准，不能只看视觉，而要看 10 个维度。

### 8.1 对象身份是否清楚

A级：

```text
页面标题、业务类型、业务编号、组织、状态、来源都清楚。
用户一眼知道自己正在处理哪个对象。
测试框架能读取 pageId、bizType、bizNo、status。
```

C级：

```text
页面只有标题和字段，业务编号不明显。
状态不稳定，测试只能靠文本猜测。
```

F级：

```text
页面不知道是什么业务对象。
无法通过页面定位订单号、会员号、单据号。
```

### 8.2 信息架构是否正确

A级：

```text
核心摘要在前。
主信息、明细、关联、审批、附件、日志分区清晰。
用户先看决策字段，再看补充字段。
```

C级：

```text
字段都展示了，但顺序像数据库表。
用户需要自己找重点。
```

F级：

```text
关键金额、状态、对象、日期混在一起。
无法判断主次。
```

### 8.3 状态是否驱动页面

A级：

```text
不同状态下字段可编辑性、按钮、提示、下一步都明确。
状态转换可测试。
```

C级：

```text
有状态标签，但按钮控制散落在页面代码中。
```

F级：

```text
页面没有状态概念。
用户点击后不知道发生了什么。
```

### 8.4 操作是否可解释

A级：

```text
每个操作都有前置条件、执行反馈、成功结果、失败原因。
风险操作有确认和留痕。
```

C级：

```text
操作能点，但失败提示泛化。
```

F级：

```text
操作后没有明显反馈。
自动化只能 sleep 等待。
```

### 8.5 字段是否有语义

A级：

```text
字段来源、必填原因、编辑条件、业务影响清楚。
金额、日期、分类、状态字段尤其明确。
```

C级：

```text
字段名称存在，但用户不知道影响什么。
```

F级：

```text
字段只是堆砌，错误时无法定位。
```

### 8.6 上下游是否可追溯

A级：

```text
能从当前页面看到来源单据、下游单据、凭证、结算、审批、日志。
关键字段能追溯来源。
```

C级：

```text
只能看到部分关联信息。
下推失败原因不清晰。
```

F级：

```text
页面是孤岛。
无法知道数据从哪里来、到哪里去。
```

### 8.7 批量场景是否成立

A级：

```text
支持批量选择、批量校验、批量补全、失败明细、导入导出。
```

C级：

```text
支持批量操作，但失败后需要逐条排查。
```

F级：

```text
只能逐条处理，企业高频任务效率很低。
```

### 8.8 异常是否可恢复

A级：

```text
错误具体到字段、行、规则、权限、网络或状态。
用户知道下一步怎么修复。
```

C级：

```text
错误能提示，但不够具体。
```

F级：

```text
只提示失败、异常、系统错误。
```

### 8.9 测试是否可自动化

A级：

```text
页面暴露稳定选择器、业务编号、状态、动作状态、错误码。
自动化可以基于状态等待，不依赖固定 sleep。
```

C级：

```text
能自动化，但依赖文本、层级或固定等待。
```

F级：

```text
自动化无法稳定定位页面元素和业务对象。
```

### 8.10 是否有企业治理能力

A级：

```text
权限、审批、日志、附件、导入导出、打印、版本兼容、灰度、观测都有设计。
```

C级：

```text
有部分治理能力，但不系统。
```

F级：

```text
页面只完成当前录入，不支持企业运维和审计。
```

## 9. 标准评分表

建议用 100 分评价一个企业页面。

| 维度 | 分值 | A级标准 |
|---|---:|---|
| 对象身份 | 10 | pageId、bizType、bizNo、status、entry 清楚 |
| 信息架构 | 12 | 摘要、主信息、明细、关联、审批、日志结构清楚 |
| 状态机 | 12 | 状态全集、状态转换、按钮和字段权限明确 |
| 字段语义 | 10 | 字段来源、必填、编辑性、业务影响清楚 |
| 操作反馈 | 10 | 点击后状态可观察，成功失败可判断 |
| 上下游关系 | 10 | 来源、下推、转换、凭证、结算可追溯 |
| 批量效率 | 8 | 批量校验、批量补全、失败明细 |
| 异常恢复 | 8 | 错误可定位、可修复、可重试 |
| 自动化测试 | 10 | 稳定选择器、状态等待、业务证据 |
| 企业治理 | 10 | 权限、审批、日志、附件、打印、导入导出 |

评级：

```text
A：90-100，企业级优秀页面，可稳定上线和自动化回归。
B：80-89，基本符合企业标准，存在少量体验或测试短板。
C：70-79，能用但不够规范，测试和运维成本较高。
D：60-69，存在明显企业级缺陷，需要重构页面契约。
F：60 以下，不符合企业级页面标准。
```

## 10. 用金蝶标准反看小程序页面

小程序页面相比 PC 企业系统屏幕更小，但企业标准不能降低，只能换表达方式。

### 10.1 小程序单据页必须保留的能力

```text
业务身份不能省。
状态不能省。
关键金额不能省。
关键日期不能省。
上下游关系不能省。
审批/处理结果不能省。
错误恢复不能省。
测试选择器不能省。
```

可以移动到二级区域的能力：

```text
系统日志。
低频扩展字段。
历史版本。
完整审批轨迹。
附件列表详情。
非关键关联单据。
```

但这些能力不能完全消失。小程序可以折叠、分 Tab、分步骤、分详情页，但不能把企业单据的治理能力删掉。

### 10.2 小程序页面推荐结构

```text
┌──────────────────────────────┐
│ 标题 + 状态标签                │
│ 业务编号 + 复制                │
├──────────────────────────────┤
│ 关键摘要                       │
│ 金额 / 状态 / 日期 / 对象       │
├──────────────────────────────┤
│ 主信息                         │
│ 客户/会员/门店/组织             │
├──────────────────────────────┤
│ 明细卡片                       │
│ 商品/数量/金额                  │
├──────────────────────────────┤
│ 关联/审批/附件 Tabs             │
├──────────────────────────────┤
│ 主操作按钮 + 更多操作            │
└──────────────────────────────┘
```

### 10.3 小程序状态标签

状态标签必须遵循：

```text
状态文案稳定。
状态颜色稳定。
状态含义唯一。
状态能被自动化读取。
状态和按钮联动。
状态变更有反馈。
```

错误示例：

```text
页面上显示“处理中”，但按钮还可重复提交。
```

正确示例：

```text
状态：submitting。
按钮：提交按钮 loading，其他风险操作禁用。
提示：正在提交，请勿重复操作。
测试：等待 data-state="submitting" 变为 data-state="submitted"。
```

## 11. 页面是否符合标准的分析方法

拿到一个页面截图或页面实现时，按下面顺序分析。

### 11.1 第一步：识别页面类型

```text
这是列表页？
这是表单页？
这是详情页？
这是审批页？
这是工作台？
这是结果页？
这是异常处理页？
```

如果页面类型都不清楚，通常说明信息架构已经有问题。

### 11.2 第二步：识别业务对象

```text
对象是什么？
对象编号是什么？
对象状态是什么？
对象归属谁？
对象来源是什么？
对象下一步是什么？
```

如果页面无法回答这些问题，就不是合格企业页面。

### 11.3 第三步：识别状态机

```text
页面一共有多少状态？
当前状态是什么？
能从当前状态去哪些状态？
每个状态下有哪些按钮？
每个状态下哪些字段可编辑？
错误状态如何恢复？
```

如果测试框架不知道页面有多少状态，说明状态机没有被契约化。

### 11.4 第四步：识别字段语义

```text
哪些字段是用户输入？
哪些字段是系统计算？
哪些字段来自上游？
哪些字段影响下游？
哪些字段只是展示？
哪些字段是风险字段？
```

### 11.5 第五步：识别操作反馈

```text
点击后是否有 loading？
成功后状态是否变化？
失败后错误是否具体？
是否能重试？
是否防重复提交？
是否能被自动化等待？
```

### 11.6 第六步：识别企业治理

```text
权限在哪里？
审批在哪里？
日志在哪里？
附件在哪里？
导入导出在哪里？
打印在哪里？
上下游追溯在哪里？
异常告警在哪里？
```

## 12. 对金蝶单据和页面设计的总体评价

基于公开资料，可以给出以下评价。

### 12.1 符合企业级标准的部分

金蝶单据和页面设计在企业级标准上有明显优势：

| 维度 | 评价 |
|---|---|
| 业务对象建模 | 强。单据被定义为业务对象，而不是普通表单 |
| 元数据能力 | 强。字段、控件、整单属性、菜单、编码规则、状态都有配置位置 |
| 流程关系 | 强。单据转换、工作流、业务流程是标准能力 |
| 企业治理 | 强。附件、引入引出、套打、发布、部署、通用过滤都进入体系 |
| 复杂业务适配 | 强。通过动态领域模型、柔性配置、单据转换、流程配置支持复杂企业场景 |
| 设计系统 | 强。KDesign 提供统一原则、典型页面、组件和开发资源 |
| 角色和效率 | 较强。强调一致、高效、角色化、极简和减少认知负荷 |

按本文评分模型，金蝶标准单据体系在“对象身份、状态、流程、治理、追溯”上可以达到 A 级。

### 12.2 从社区问题反推的改进点

真实企业场景中仍会出现以下问题，这些问题不一定代表金蝶产品缺陷，但代表企业页面设计必须重点防范。

| 问题 | 反映的设计风险 |
|---|---|
| 业务分类影响凭证生成但用户不理解 | 流程控制字段解释不足 |
| 批量接收票据因必填字段逐条失败 | 批量校验和批量补全能力不足 |
| 业务日期影响账期但页面不一定充分解释 | 日期字段语义和影响范围不明显 |
| 下推其他应付款需要字段映射 | 字段来源和转换规则需要可解释 |
| 成本金额用途容易误解 | 金额口径、来源和业务影响需要标注 |

所以即使学习金蝶，也不能只学组件和布局，还要学习它背后的业务对象、状态、流程和治理；同时要在我们自己的小程序规范中补强“字段影响解释、批量反馈、测试契约”。

### 12.3 结论

金蝶的企业 UI 标准符合企业级软件的核心方向：

```text
以业务对象为中心。
以状态和流程为主线。
以角色效率为目标。
以元数据和配置支撑复杂变化。
以追溯、审计、导入导出、打印、附件支持企业治理。
```

但在我们落地小程序时，必须进一步工程化：

```text
页面契约化。
状态机显式化。
业务身份可读取。
字段语义可解释。
操作反馈可观察。
错误可恢复。
自动化测试可等待。
上下游证据可追踪。
```

## 13. 我们的小程序页面落地标准

以后任何小程序页面，如果要达到企业级 A 级，必须满足下面清单。

### 13.1 页面契约

```ts
export const pageContract = {
  pageId: 'receivableDetail',
  pageType: 'detail',
  bizType: 'receivableBill',
  bizNoField: 'billNo',
  entries: ['fromList', 'fromTodo', 'fromMessage', 'fromScan'],
  modes: ['readonly', 'approve', 'result'],
  states: [
    'loading',
    'loaded',
    'auditing',
    'approved',
    'rejected',
    'closed',
    'error'
  ],
  actions: [
    'approve',
    'reject',
    'print',
    'export',
    'viewSource',
    'viewVoucher'
  ]
};
```

### 13.2 页面结构

```text
页面头部：标题、状态、业务编号。
摘要区：关键金额、日期、对象、下一步。
主信息区：整单字段。
明细区：分录行。
关联区：上游、下游、凭证、结算。
流程区：审批节点、意见、日志。
附件区：票据、图片、合同。
操作区：主操作、次操作、风险操作。
```

### 13.3 数据属性

```html
<view
  data-page-id="receivableDetail"
  data-page-type="detail"
  data-biz-type="receivableBill"
  data-biz-no="{{bill.billNo}}"
  data-biz-status="{{bill.status}}"
  data-entry="{{entry}}"
>
```

这些属性不是给用户看的，而是给测试、埋点、日志和可观测系统看的。它们让页面从“视觉界面”升级为“可治理界面”。

### 13.4 状态等待

禁止测试这样写：

```ts
await sleep(3000);
expect(page.text()).toContain('已提交');
```

必须这样写：

```ts
await page.tapByTestId('submit-button');
await page.waitForState('submitting');
await page.waitForState('submitted');
await expect(page.getByBizNo(orderNo)).toHaveBizStatus('submitted');
```

页面必须暴露可等待状态：

```html
<view data-state="{{state}}" data-action-state="{{actionState}}">
```

### 13.5 错误设计

错误必须结构化：

```ts
type PageError = {
  code: string;
  scope: 'page' | 'field' | 'row' | 'action' | 'permission' | 'network';
  field?: string;
  rowIndex?: number;
  message: string;
  recoverable: boolean;
  retryAction?: string;
};
```

错误示例：

```text
错误码：FIELD_REQUIRED
位置：明细第 3 行，销售部门
原因：提交前销售部门必填
修复：补充销售部门或使用批量补全
```

## 15. 对“是否符合标准”的最终判断口径

一个页面是否符合企业级标准，不看它是否像金蝶，而看它是否具备金蝶式企业系统的底层能力。

符合标准的页面：

```text
看得懂对象。
看得懂状态。
看得懂字段。
看得懂流程。
看得懂上下游。
点得动。
错得明白。
批量做得快。
测得稳定。
查得出证据。
出了问题能追溯。
```

不符合标准的页面：

```text
只有 UI，没有业务对象。
只有字段，没有字段语义。
只有按钮，没有动作契约。
只有状态文案，没有状态机。
只有错误提示，没有恢复路径。
只有人工可看，没有自动化可测。
只有当前页面，没有上下游证据。
```

这就是学习金蝶单据和页面设计后，对我们小程序页面最重要的启发。
