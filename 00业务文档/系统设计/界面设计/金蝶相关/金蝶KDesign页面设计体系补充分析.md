## 3. KDesign 的定位

![KDesign 企业级页面设计体系](images/kdesign-pages/kdesign-system-map.svg)

金蝶社区公开文章明确提到，KDesign 是面向 ToB 的企业级产品设计系统，服务金蝶云苍穹、星瀚、星空、星辰及生态伙伴产品。

它不是单纯组件库。

它包含：

```text
设计原则。
设计规范。
设计资源。
设计工具。
前端组件库。
PC 端典型页面。
移动端典型页面。
可视化组件。
可视化大屏案例。
```

这说明金蝶的页面设计体系至少有四层：

```text
第一层：设计哲学和原则。
第二层：基础视觉规范。
第三层：组件级规范。
第四层：典型页面和业务场景模板。
```

我们的文档体系不能只学“单据”，还要学这四层。

## 4. KDesign 的角色模型

![KDesign 角色模型](images/kdesign-pages/role-model.svg)

金蝶社区文章提到 KDesign 设计原则包括：

```text
基于角色。
一致的。
高效的。
愉悦的。
```

其中“基于角色”非常关键。

ToB 端用户可以拆成四类：

```text
事务型：一线员工，主要做录入、处理、提交。
审批型：审批事务型角色提交的内容。
分析型：关注业务数据，通常是业务主管或运营人员。
决策型：通过数据洞察市场变化和资源动态，通常是管理层。
```

这四类角色对应不同页面：

| 角色 | 主要任务 | 典型页面 |
|---|---|---|
| 事务型 | 录入、查询、处理、批量作业 | 表单页、列表页、导入向导、左树右表 |
| 审批型 | 判断风险、审批、驳回、转交 | 详情页、审批页、三栏待办 |
| 分析型 | 看数据、找异常、追原因 | 报表页、图表钻取、工作台 |
| 决策型 | 看指标、看趋势、看风险 | 工作台、驾驶舱、可视化大屏 |

设计启发：

```text
页面不是按功能菜单设计，而是按角色任务设计。
```

同样是“订单”：

```text
事务型用户要的是录入效率和批量处理。
审批型用户要的是风险摘要和审批证据。
分析型用户要的是趋势、异常和下钻。
决策型用户要的是指标、风险和经营判断。
```

## 5. KDesign 典型页面体系

KDesign 的 PC 典型页面至少包括：

```text
工作台。
表单页。
详情页。
列表页。
报表页。
```

这不是随便列的五类页面，而是 ToB 应用的核心页面骨架。

从顶级设计师和架构师角度看，这五类页面分别承担不同系统职责：

| 页面类型 | 系统职责 | 解决的核心问题 |
|---|---|---|
| 工作台 | 角色任务入口 | 用户今天要处理什么 |
| 列表页 | 对象集合处理 | 如何查找、比较、批量处理对象 |
| 表单页 | 业务事实录入 | 如何低错误率录入数据 |
| 详情页 | 对象事实和证据聚合 | 如何理解一个对象的全貌 |
| 报表页 | 指标解释和追溯 | 如何从结果追到原因 |

这五类页面覆盖了企业后台的基本作业循环：

```text
工作台发现任务
-> 列表页定位对象
-> 表单页录入或修改事实
-> 详情页确认状态和证据
-> 报表页汇总结果并追溯原因
```

如果一个后台系统只做了列表和表单，没有工作台、详情证据和报表追溯，它只能算 CRUD 系统，还不能算成熟企业系统。

### 5.1 工作台

![工作台页面原型](images/kdesign-pages/workbench-page.svg)

工作台常作为应用首页，是应用交通枢纽。

它聚合：

```text
工作任务。
关键指标。
快捷入口。
异常预警。
业务概览。
```

工作台不是门户装饰。

它解决的是：

```text
不同角色进入系统后，如何快速知道当前状态、待办事项和下一步入口。
```

设计原则：

```text
基于角色。
任务优先。
异常优先。
指标可下钻。
入口服务高频任务。
```

页面模型：

```text
Role -> Tasks + Metrics + Alerts + Shortcuts
```

错误设计：

```text
所有角色看到同一个首页。
首页堆菜单和公告。
指标不能点击。
待办不能直达对象。
```

工作台的深层设计原则：

```text
不是让用户知道系统有哪些功能，而是让用户知道现在应该处理什么。
```

因此工作台必须按角色和任务建模：

```text
Role -> Todo + Alert + Metric + Shortcut
```

不要把菜单复制到首页。菜单回答“系统有什么”，工作台回答“我现在该做什么”。

### 5.2 列表页

![列表页页面原型](images/kdesign-pages/list-page.svg)

KDesign 列表页强调：

```text
帮助用户更高效地查看、处理、查找条目。
易读性。
突出有利于对象识别的关键信息。
```

列表页不是表格控件。

它解决的是：

```text
如何在大量对象中识别、筛选、比较、操作目标对象。
```

页面模型：

```text
ObjectCollection -> Query / Sort / Filter -> ResultRows -> RowAction / BatchAction
```

列表页必须包含：

```text
对象身份列。
状态列。
关键业务字段。
操作列。
筛选区。
工具栏。
分页。
批量操作。
空状态。
异常状态。
```

社区里的客户列表改造案例也印证了这个方向：它不是只改表格，而是补充过滤选项、多个界面窗口、导入能力、按使用组织查询页签、实时同步等能力。

列表页的深层设计原则：

```text
列表页不是数据展示页，而是对象集合的处理中心。
```

它必须同时支持：

```text
找到对象。
识别对象。
判断状态。
选择对象。
批量处理。
进入详情。
导入导出。
保留证据。
```

所以列表页的质量不能只看表格样式，而要看：

```text
默认筛选是否安全。
表格列是否服务判断。
状态是否可筛选。
批量失败是否有明细。
导出是否有范围说明。
```

### 5.3 表单页

![表单页页面原型](images/kdesign-pages/form-page.svg)

KDesign 表单页强调：

```text
按内容关联性分组。
选择正确输入框形式。
通过预填减少操作步骤。
及时反馈。
```

表单页不是字段堆叠。

它解决的是：

```text
用户如何以最小错误成本录入一组业务事实。
```

页面模型：

```text
BusinessFacts -> FieldGroups -> Validation -> Save / Submit
```

优秀表单页必须考虑：

```text
字段分组。
默认值。
输入控件匹配。
必填阶段。
字段可编辑状态。
即时校验。
提交校验。
错误定位。
草稿保存。
离开保护。
```

错误设计：

```text
字段按数据库顺序排列。
必填只靠红星。
错误只在顶部提示。
所有校验堆到提交时。
```

表单页的深层设计原则：

```text
表单不是字段容器，而是业务事实的构造过程。
```

设计师要思考：

```text
哪些事实必须先填？
哪些字段可以从上下文预填？
哪些字段影响下游流程？
哪些字段在保存时必填，哪些在提交时必填？
用户填错后如何定位和恢复？
```

架构师要思考：

```text
字段规则是否和后端校验一致？
字段状态是否受单据状态控制？
草稿和提交是否不同校验等级？
错误是否能结构化返回到字段和行？
```

### 5.4 详情页

![详情页页面原型](images/kdesign-pages/detail-page.svg)

KDesign 详情页用于展示一个对象的完整信息，也可以看作表单录入结果呈现，主要用于信息浏览，并可进行审批、打印、导出等简单操作。

详情页不是只读表单。

它解决的是：

```text
用户如何理解一个对象的完整事实、当前状态、历史过程和可执行动作。
```

页面模型：

```text
BusinessObject -> Facts + State + Relations + Logs + Actions
```

详情页必须包含：

```text
对象身份。
状态。
关键摘要。
主信息。
明细。
关联对象。
审批记录。
附件。
操作日志。
可执行动作。
```

对于 ERP 单据，详情页还要承载：

```text
上游单据。
下游单据。
凭证。
反写。
结算。
套打。
```

详情页的深层设计原则：

```text
详情页是业务事实和证据中心，不是只读表单。
```

一个成熟详情页必须回答：

```text
这个对象是什么？
当前状态是什么？
为什么是这个状态？
从哪里来？
到哪里去？
谁处理过？
还能做什么？
出了问题能追溯到什么证据？
```

因此详情页要把业务事实、流程状态、上下游关系、日志附件和可执行动作组织在一起。

### 5.5 报表页

![报表页页面原型](images/kdesign-pages/report-page.svg)

KDesign 报表页强调：

```text
大量数据下易查、易看、易懂。
```

报表页不是图表堆叠。

它解决的是：

```text
用户如何从大量业务事实中理解指标、定位异常、追溯原因。
```

页面模型：

```text
Metric -> Dimension -> Detail -> SourceObject
```

报表页必须包含：

```text
指标口径。
查询条件。
方案切换。
图表或透视。
明细表。
钻取路径。
导出。
原始对象追溯。
```

错误设计：

```text
只有图，没有口径。
只有指标，没有明细。
明细不能回到原始单据。
图表筛选和明细筛选不一致。
```

报表页的深层设计原则：

```text
报表不是图表页面，而是业务事实的解释系统。
```

报表设计必须有三层：

```text
指标层：告诉用户结果是什么。
维度层：告诉用户结果由哪些维度构成。
证据层：告诉用户这些数字来自哪些原始业务对象。
```

如果图表不能下钻，指标不能解释，明细不能回到原始对象，那么它只是展示，不是企业级报表。

## 6. KDesign 基础视觉规范

金蝶社区文章提到 KDesign 有色彩、阴影、字体、图标与插画、消息反馈、国际化等规范。

这些不是“视觉装饰”，而是企业后台可用性的基础。

### 6.1 色彩

社区文章提到 KDesign 提供常用基础色，并可通过品牌色生成对应色板。

设计启发：

```text
企业后台要支持品牌差异，但不能让品牌色破坏业务语义。
```

色彩要分层：

```text
品牌色：主按钮、选中态、链接、焦点。
状态色：成功、警告、危险、禁用、处理中。
中性色：背景、边框、正文、辅助文字。
图表色：分类和系列，不应污染状态语义。
```

### 6.2 阴影

社区文章提到 KDesign 通过低、中、高阴影区分内容层级。

设计启发：

```text
阴影不是为了高级感，而是为了表达层级和浮层关系。
```

使用原则：

```text
低阴影：卡片、静态容器。
中阴影：下拉、浮层。
高阴影：弹窗、独立内容。
```

后台页面不能滥用阴影，否则高密度信息会变脏。

### 6.3 字体

字体规范解决的是：

```text
长期阅读。
高密度表格。
中英文混排。
数字对齐。
层级识别。
```

后台字体重点：

```text
正文清晰。
数字易读。
表格不拥挤。
标题不夸张。
辅助文字可读但不抢主信息。
```

### 6.4 图标和插画

图标解决操作识别。

插画主要用于：

```text
空状态。
提示状态。
引导说明。
异常轻提示。
```

注意：

```text
后台图标要克制。
图标不能替代文字。
高风险操作不能只靠图标表达。
空状态插画不能盖过下一步行动。
```

### 6.5 消息与反馈

![消息反馈体系](images/kdesign-pages/feedback-system.svg)

社区文章提到 KDesign 的反馈规范会区分：

```text
信息提示。
过程反馈。
结果反馈。
重要程度。
紧急程度。
```

这非常重要。

后台反馈不能一律 toast。

应按场景选择：

| 场景 | 推荐反馈 |
|---|---|
| 轻量成功 | toast / message |
| 表单字段错误 | 字段级错误 |
| 批量失败 | 结果面板 + 失败明细 |
| 高风险确认 | modal |
| 长任务 | 进度 / 任务中心 |
| 系统异常 | alert / error panel |
| 权限不足 | 禁用说明 / 无权限页 |

### 6.6 国际化

国际化不是翻译。

它影响：

```text
文字长度。
菜单宽度。
表格列宽。
按钮宽度。
日期格式。
数字格式。
阅读方向。
货币和单位。
```

设计启发：

```text
后台页面不能把中文文案长度当作固定布局前提。
```

## 7. KDesign 组件级规范

金蝶社区文章提到 KDesign 包含组件级规范，覆盖基础组件、导航、表单、可视化组件等。

这说明金蝶页面设计不是只靠页面模板，而是：

```text
组件规范 + 典型页面 + 业务模型。
```

组件规范解决：

```text
一致性。
开发效率。
交互稳定。
测试稳定。
设计和前端协作。
```

典型组件：

```text
按钮。
分页。
滚动条。
导航菜单。
页签。
下拉菜单。
单选。
多选。
输入框。
选择器。
表格。
弹窗。
抽屉。
消息反馈。
图表。
```

设计启发：

```text
页面质量不是靠设计师每次自由发挥，而是靠稳定组件组合。
```

## 8. 社区实践中的页面设计信号

### 8.1 左树右表实践

金蝶社区有文章专门讲如何实现左树右表列表界面。

这说明左树右表不是我们自己抽象出来的，而是金蝶生态中真实存在的页面模式。

文章里的关键点：

```text
左树右表用于单据列表界面。
左侧树形控件承载分组数据。
右侧单据列表控件承载对象集合。
F7 选择时也要按树节点范围过滤。
可以利用平台提供的分组基础资料模板。
也可以自定义动态表单实现。
```

设计启发：

```text
左树右表不仅是视觉结构，还是平台模板、数据过滤、选择器、权限范围共同作用的结果。
```

### 8.2 客户列表改造实践

金蝶社区客户列表改造案例提到：

```text
提供过滤选项。
分多个界面窗口。
支持 Excel 导入。
新增按使用组织查询客户资料的页签。
全部资料和金蝶云星空实时同步。
```

这说明企业后台列表页不是普通列表。

它要同时解决：

```text
对象识别。
多组织分配。
过滤查询。
导入。
页签视图。
实时同步。
多窗口信息组织。
```

设计启发：

```text
真正的后台列表页经常是“列表 + 组织关系 + 导入 + 多视图 + 同步状态”的组合页面。
```

## 9. 页面设计体系与 ERP 单据体系的关系

金蝶体系可以拆成两条线：

```text
ERP 业务建模线：业务对象、单据、字段、状态、流程、转换、反写、权限、审计。
KDesign 页面体验线：角色、页面类型、组件、色彩、字体、反馈、典型页面、可视化。
```

两条线必须合并。

如果只有 ERP 业务建模：

```text
系统可能很强，但页面难用。
```

如果只有 KDesign 页面体验：

```text
页面可能漂亮，但承载不了复杂业务。
```

企业级后台设计必须同时满足：

```text
业务模型正确。
页面模型正确。
组件使用正确。
反馈机制正确。
视觉语义正确。
测试证据正确。
```

## 11. 最终结论

金蝶社区和 KDesign 的页面设计体系确实不止“单据”和“几个结构”。

它至少包含：

```text
角色模型。
设计原则。
基础视觉规范。
组件规范。
PC 典型页面。
移动端典型页面。
可视化规范。
社区实践案例。
前端组件库。
设计资源。
```

所以后续整理时应分成两大体系：

```text
ERP 业务建模体系：解决业务对象、流程、状态、权限、审计。
KDesign 页面设计体系：解决角色、页面类型、组件、反馈、视觉、可用性。
```

真正的企业级后端管理界面设计，要把这两套体系合在一起。
