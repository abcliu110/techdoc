后端管理界面。

## 3. 一句话总结金蝶 ERP 的精髓

金蝶 ERP 的精髓不是“页面做得像金蝶”，而是：

```text
用模型描述业务对象。
用单据承载业务事实。
用状态控制操作边界。
用流程连接上下游业务。
用元数据沉淀企业能力。
用权限、审计、日志、附件、打印、导入导出支撑企业治理。
用扩展和继承处理共性与个性。
用列表、表单、详情、报表、工作台、流程中心这些经典界面承载复杂企业协作。
```

从架构视角看，它不是一套 UI，而是一套企业业务操作系统。

## 4. 金蝶 ERP 的底层哲学

### 4.1 模型优先，而不是页面优先

普通后台系统经常是这样设计的：

```text
先画页面。
再定义接口。
再建表。
再补权限。
再补审批。
再补导出。
再补日志。
```

金蝶式 ERP 的顺序更接近：

```text
先定义业务对象。
再定义字段、状态、规则、流程、权限、操作。
再定义视图：列表、表单、详情、报表、移动端。
再发布到工作台、流程中心、主控台。
再配置套打、导入导出、附件、API、测试、部署。
```

这就是“模型优先”。

页面只是模型的一种视图。模型才是系统资产。

### 4.2 单据不是表单，而是业务对象

在 ERP 中，单据不是一个输入表单。

单据至少包含：

```text
业务身份：单据类型、单据编号、业务组织、单据日期、业务日期。
业务事实：字段、分录、金额、数量、对象、附件。
业务状态：保存、提交、审核、关闭、作废、下推、结算、记账。
业务动作：新增、保存、提交、审核、反审核、下推、选单、打印、导入、导出。
业务关系：上游单据、下游单据、凭证、库存、结算、报表、附件。
业务约束：权限、字段规则、编码规则、网络控制、流程规则。
业务证据：操作日志、审批轨迹、附件、打印模板、接口调用、反写记录。
```

所以 ERP 后端管理界面设计，不能把单据设计成“字段集合”，而要设计成“业务对象的生命周期界面”。

### 4.3 业务流和审批流要融合

很多后台系统只做审批流：

```text
提交 -> 审批 -> 结束
```

ERP 不够。ERP 还要做业务流：

```text
采购申请 -> 采购订单 -> 收料通知 -> 入库单 -> 应付单 -> 付款单 -> 凭证
销售订单 -> 出库单 -> 应收单 -> 发票 -> 收款单 -> 凭证
费用申请 -> 费用报销 -> 付款申请 -> 付款单 -> 凭证
```

审批流解决“谁批准”。

业务流解决“业务对象如何从一个环节流转到下一个环节”。

金蝶式 ERP 的关键能力是：

```text
审批流和业务流融合。
流程可视化。
上下游可追溯。
下推、选单、转换、反写可配置。
流程异常可监控。
```

### 4.4 共性与个性必须共存

ERP 面向很多行业、很多公司、很多组织。

如果只做标准产品，会不够灵活。

如果每个客户都定制一套，会不可维护。

金蝶动态领域模型强调“共性与个性”：

```text
标准产品沉淀共性模型。
行业产品扩展行业模型。
客户项目扩展客户个性。
通过分层模型、继承、扩展、插件、配置让它们共存。
```

这对后端管理界面设计非常重要。

页面不能只考虑当前客户，而要考虑：

```text
哪些是标准字段？
哪些是行业字段？
哪些是客户自定义字段？
哪些是标准动作？
哪些动作允许扩展？
哪些流程允许配置？
哪些规则允许覆盖？
哪些能力必须保持稳定？
```

## 5. 金蝶 ERP 的核心设计对象

### 5.1 业务领域和子系统

金蝶单据设计先选择业务领域和子系统。

这说明 ERP 设计不是从页面开始，而是从业务域开始。

常见业务域：

```text
财务会计。
供应链。
库存管理。
采购管理。
销售管理。
应收应付。
成本核算。
资产管理。
预算管理。
费用管理。
生产制造。
人力资源。
客户关系。
权限与组织。
报表分析。
```

后端管理界面设计必须先回答：

```text
这个页面属于哪个业务域？
这个页面服务哪个子系统？
这个页面操作的是哪个业务对象？
这个页面影响哪些下游业务域？
```

### 5.2 业务对象

业务对象是 ERP 的核心建模单位。

示例：

```text
采购订单。
销售订单。
入库单。
出库单。
应收单。
应付单。
收款单。
付款单。
费用报销单。
库存调拨单。
盘点单。
凭证。
客户。
供应商。
物料。
仓库。
组织。
员工。
角色。
```

业务对象必须有：

```text
对象类型。
对象编号。
对象状态。
对象字段。
对象动作。
对象权限。
对象流程。
对象关系。
对象审计。
对象视图。
```

如果页面没有清楚表达业务对象，就不符合 ERP 设计。

### 5.3 单据

单据是最重要的业务对象类型。

单据和普通主数据不同：

```text
主数据相对稳定，例如客户、供应商、物料、仓库。
单据记录业务过程，例如采购、销售、入库、出库、收款、付款。
```

单据设计必须包含：

```text
单据头。
单据体。
分录。
字段属性。
整单属性。
菜单集合。
编码规则。
单据状态。
网络控制。
元数据校验。
单据视图。
单据转换。
发布。
部署。
套打。
单据类型。
工作流。
业务流程。
通用操作。
通用过滤。
个性化设置。
附件。
引入引出。
```

这也是金蝶式单据设计和普通 CRUD 页面最大的差别。

### 5.4 字段

ERP 字段不是数据库列的简单投影。

字段必须有业务语义：

```text
字段属于单据头还是单据体？
字段是主数据引用还是自由输入？
字段是否必填？
字段是否可编辑？
字段是否影响状态？
字段是否影响下游转换？
字段是否参与金额计算？
字段是否参与报表统计？
字段是否参与权限控制？
字段是否可导入导出？
字段是否可套打？
字段是否允许个性化隐藏？
```

典型字段类型：

```text
业务编号字段。
业务日期字段。
组织字段。
往来单位字段。
人员字段。
部门字段。
物料字段。
仓库字段。
数量字段。
金额字段。
税率字段。
状态字段。
审批字段。
来源单据字段。
下游单据字段。
附件字段。
备注字段。
自定义字段。
```

字段设计的精髓是：

```text
字段不是展示项，而是规则、流程、权限、计算、追溯的节点。
```

### 5.5 状态

ERP 页面必须以状态为核心。

典型单据状态：

```text
暂存。
已保存。
已提交。
审核中。
已审核。
已驳回。
已关闭。
已作废。
已下推。
部分下推。
已结算。
部分结算。
已生成凭证。
部分生成凭证。
```

状态决定：

```text
字段是否可编辑。
按钮是否可见。
按钮是否可点击。
是否允许下推。
是否允许反审核。
是否允许删除。
是否允许打印。
是否允许导出。
是否允许生成凭证。
是否允许结算。
是否进入报表口径。
```

ERP 设计中，状态不是一个标签，而是页面的控制器。

## 6. 金蝶 ERP 的界面设计精髓

### 6.1 界面不是独立设计，而是模型视图

金蝶云苍穹使用动态领域模型，将业务对象、布局、规则、流程作为元数据进行描述和沉淀。

这意味着界面设计不是设计一个孤立页面，而是设计模型的视图。

一个业务对象可以有多个视图：

```text
列表视图。
新增视图。
编辑视图。
详情视图。
审批视图。
移动视图。
打印视图。
报表视图。
API 视图。
```

同一个单据在不同角色、不同状态、不同入口下可以有不同界面。

例如费用报销单：

```text
申请人新增视图：突出录入和附件。
申请人详情视图：突出状态和审批进度。
审批人视图：突出金额、风险、历史意见、同意/驳回。
财务视图：突出付款信息、核算科目、凭证状态。
报表视图：突出费用归属、部门、项目、预算占用。
打印视图：突出正式格式和签字栏。
```

### 6.2 列表页是业务对象检索和批量处理中心

ERP 列表页不是表格展示。

列表页必须承担：

```text
查询。
筛选。
排序。
分页。
状态识别。
批量操作。
快速入口。
导入导出。
下推。
选单。
打印。
查看详情。
权限控制。
个性化列设置。
```

典型 ERP 列表页结构：

```text
┌──────────────────────────────────────────────┐
│ 页面标题 / 单据类型 / 业务域                  │
├──────────────────────────────────────────────┤
│ 查询条件：编号、日期、组织、状态、客户、金额区间 │
├──────────────────────────────────────────────┤
│ 操作栏：新增、提交、审核、下推、导出、打印       │
├──────────────────────────────────────────────┤
│ 表格：编号、日期、对象、金额、状态、来源、操作   │
├──────────────────────────────────────────────┤
│ 分页 / 合计 / 选中数量                         │
└──────────────────────────────────────────────┘
```

设计关键：

```text
默认筛选必须符合业务高频场景。
状态列必须稳定。
批量操作必须先校验。
导出必须说明导出范围。
行操作必须受状态和权限控制。
```

### 6.3 表单页是业务事实录入和规则校验界面

ERP 表单页不是输入框集合。

表单页必须承担：

```text
字段分组。
默认值。
主数据选择。
分录录入。
行级校验。
整单校验。
附件上传。
暂存。
提交。
审批。
下推。
```

典型 ERP 表单页结构：

```text
┌──────────────────────────────────────────────┐
│ 单据标题 / 状态 / 单据编号                    │
├──────────────────────────────────────────────┤
│ 单据头：组织、日期、客户、部门、经办人          │
├──────────────────────────────────────────────┤
│ 分录：物料、仓库、数量、单价、金额、税率         │
├──────────────────────────────────────────────┤
│ 合计：数量合计、金额合计、税额合计              │
├──────────────────────────────────────────────┤
│ 附件 / 备注 / 扩展字段                         │
├──────────────────────────────────────────────┤
│ 操作：保存、提交、审核、下推、打印              │
└──────────────────────────────────────────────┘
```

设计关键：

```text
字段不能按数据库顺序排列，要按业务决策顺序排列。
必填不只是红星号，要说明在哪个阶段必填。
金额、日期、组织、往来单位是高风险字段。
分录行错误必须定位到具体行和具体字段。
保存、提交、审核要区分校验强度。
```

### 6.4 详情页是业务事实、流程和证据的聚合界面

详情页不是只读表单。

详情页必须让用户理解：

```text
这个对象是什么？
当前状态是什么？
为什么是这个状态？
从哪里来？
到哪里去？
谁处理过？
还能做什么？
有哪些附件、日志、审批、凭证、下游单据？
```

典型详情页结构：

```text
┌──────────────────────────────────────────────┐
│ 单据标题 / 编号 / 状态 / 组织 / 日期           │
├──────────────────────────────────────────────┤
│ 核心摘要：金额、对象、状态、下一步              │
├──────────────────────────────────────────────┤
│ 主信息                                       │
├──────────────────────────────────────────────┤
│ 分录明细                                     │
├──────────────────────────────────────────────┤
│ 上游 / 下游 / 凭证 / 结算 / 库存影响            │
├──────────────────────────────────────────────┤
│ 审批记录 / 操作日志 / 附件                     │
└──────────────────────────────────────────────┘
```

设计关键：

```text
详情页必须突出对象身份和状态。
关键事实不能折叠到看不见。
上下游关系必须可追溯。
审批记录和操作日志不能缺失。
风险操作必须克制。
```

### 6.5 报表页是指标口径和明细追溯界面

ERP 报表不是图表展示。

报表必须回答：

```text
指标口径是什么？
统计范围是什么？
数据来自哪些单据？
时间口径是什么？
组织口径是什么？
金额口径是什么？
能否下钻到明细？
能否回溯到原始单据？
```

典型报表结构：

```text
┌──────────────────────────────────────────────┐
│ 报表标题 / 口径说明                           │
├──────────────────────────────────────────────┤
│ 查询条件：组织、期间、业务类型、状态            │
├──────────────────────────────────────────────┤
│ 指标卡片 / 图表                               │
├──────────────────────────────────────────────┤
│ 明细表                                       │
├──────────────────────────────────────────────┤
│ 下钻路径 / 导出 / 打印                         │
└──────────────────────────────────────────────┘
```

报表设计的精髓是：

```text
报表不是结论，而是可追溯的业务证据视图。
```

### 6.6 工作台是角色任务中心

ERP 工作台不是首页装饰。

工作台必须围绕角色任务：

```text
财务看待付款、待记账、异常凭证、资金余额。
采购看待采购申请、待下单、待入库、供应商异常。
销售看待订单、应收、发货、客户跟进。
仓库看待待入库、待出库、库存预警、盘点任务。
审批人看待待审批、超期、驳回、已处理。
老板看待经营指标、风险预警、异常追踪。
```

工作台的结构：

```text
任务优先。
异常优先。
指标辅助。
入口服务任务。
所有指标能下钻。
所有待办能直达对象。
```

## 7. 金蝶 ERP 的经典页面结构

### 7.1 左树右表

这是企业管理界面最经典的结构之一。

适用：

```text
组织树 -> 员工列表。
部门树 -> 岗位列表。
商品分类树 -> 商品列表。
仓库树 -> 库存列表。
区域树 -> 门店列表。
菜单树 -> 权限点列表。
科目树 -> 凭证明细。
```

精髓：

```text
左侧先确定业务范围。
右侧处理该范围下的数据对象。
```

文本原型：

```text
┌──────────────┬───────────────────────────────┐
│ 组织/分类树    │ 查询条件 / 操作栏              │
│              ├───────────────────────────────┤
│ 全部          │ 数据表                         │
│ ├ 华东区      │ 编号  名称  状态  负责人  操作   │
│ │ ├ 上海      │ 001  门店A 启用  张三    编辑   │
│ │ └ 杭州      │ 002  门店B 停用  李四    查看   │
└──────────────┴───────────────────────────────┘
```

设计关键：

```text
当前树节点必须清楚。
右侧数据范围必须清楚。
切换节点后分页和筛选规则必须明确。
批量操作必须限制在当前范围或明确说明影响范围。
```

### 7.2 上筛下表

适用：

```text
订单列表。
单据列表。
流水列表。
库存列表。
凭证列表。
付款记录。
发票列表。
```

精髓：

```text
先定义查询口径，再展示结果集合。
```

设计关键：

```text
默认查询条件不能危险。
高级筛选不能淹没首屏。
状态筛选必须与业务状态一致。
导出必须明确当前页、选中项还是全部结果。
```

### 7.3 主从表

适用：

```text
订单 -> 订单明细。
入库单 -> 入库分录。
对账单 -> 对账流水。
客户 -> 联系人。
门店 -> 设备。
```

精髓：

```text
主对象决定从对象范围。
```

设计关键：

```text
主表当前行必须明显。
从表必须显示当前主对象。
主表切换时从表加载状态必须可观察。
从表为空时要说明原因。
```

### 7.4 树表一体

适用：

```text
菜单资源。
权限点。
会计科目。
组织层级。
物料 BOM。
```

精髓：

```text
表格本身表达层级对象。
```

区别：

```text
左树右表：左树是范围，右表是对象集合。
树表一体：每一行本身就是层级对象。
```

### 7.5 三栏工作界面

适用：

```text
审批中心。
工单中心。
消息中心。
异常处理中心。
客服中心。
```

精髓：

```text
左栏选队列。
中栏选对象。
右栏处理对象。
```

### 7.6 分步向导

适用：

```text
导入数据。
创建活动。
配置流程。
初始化系统。
批量生成单据。
```

精髓：

```text
把高复杂任务拆成可验证步骤。
```

## 8. 金蝶 ERP 的流程设计精髓

### 8.1 入口单据

业务流程通常从入口单据开始。

入口单据决定：

```text
流程版本。
流程字段。
流程生效条件。
后续下推路径。
审批路径。
业务消息。
```

后端管理界面设计时必须明确：

```text
这个页面是否是流程入口？
入口单据类型是什么？
入口单据字段是否足够支撑流程判断？
入口单据状态是否允许发起流程？
```

### 8.2 下推和选单

ERP 中“下推”和“选单”非常关键。

```text
下推：从当前单据生成下游单据。
选单：在下游单据中选择上游单据带入数据。
```

例子：

```text
采购订单下推入库单。
销售订单下推出库单。
出库单下推应收单。
应收单下推收款单。
```

设计关键：

```text
下推条件必须清楚。
下推结果必须可见。
字段映射必须可解释。
部分下推必须可追踪。
下推失败必须说明具体原因。
```

### 8.3 单据转换

单据转换不是复制字段。

它包含：

```text
转换路线。
转换规则。
字段映射。
行合并/拆分。
数量反写。
金额反写。
状态反写。
插件扩展。
异常处理。
```

设计要求：

```text
用户要知道当前单据可以转换成什么。
用户要知道为什么不能转换。
转换后要能回到上游。
上游要能看到下游。
部分转换要显示剩余数量、已转换数量。
```

### 8.4 反写

反写是 ERP 的核心能力之一。

例子：

```text
入库单反写采购订单已入库数量。
收款单反写应收单已收金额。
出库单反写销售订单已出库数量。
下游作废或删除时反向扣减上游累计值。
```

界面必须展示：

```text
已下推数量。
未下推数量。
已结算金额。
未结算金额。
已收款金额。
未收款金额。
反写状态。
```

如果不展示反写结果，用户就无法理解上下游业务是否一致。

### 8.5 审批流

审批流关注：

```text
谁审批。
按什么条件审批。
当前节点是谁。
历史审批意见是什么。
是否超期。
是否可以撤回。
是否可以转交。
是否可以加签。
```

ERP 界面必须把审批和业务事实结合：

```text
审批人不应该只看到“同意/驳回”。
审批人要看到金额、风险、预算、附件、历史、下游影响。
```

## 9. 金蝶 ERP 的治理设计精髓

### 9.1 权限

ERP 权限不是简单菜单权限。

至少包括：

```text
菜单权限。
按钮权限。
字段权限。
数据权限。
组织权限。
角色权限。
流程权限。
导出权限。
打印权限。
API 权限。
```

设计关键：

```text
无权限时不要只隐藏，要考虑是否需要解释。
危险操作必须有权限和二次确认。
导出、打印、删除、反审核、作废是高风险权限。
```

### 9.2 安全审计

企业系统必须可审计。

审计对象：

```text
谁创建了单据。
谁修改了字段。
谁提交了审批。
谁审核通过。
谁驳回。
谁反审核。
谁删除或作废。
谁导出。
谁打印。
谁调用 API。
谁修改流程或权限。
```

设计要求：

```text
关键动作必须有操作日志。
审批必须有审批轨迹。
字段变更最好有变更记录。
导出和打印必须可追踪。
接口调用必须有调用日志。
```

### 9.3 网络控制和并发

金蝶单据设计中包含网络控制。它反映了 ERP 高并发协作问题。

典型问题：

```text
两个人同时编辑同一张单据。
审批时单据已经被修改。
下推时上游数据已经变化。
导入时数据被其他用户占用。
库存数量被并发扣减。
```

界面必须处理：

```text
锁定状态。
占用人。
最后修改时间。
版本冲突。
刷新提示。
重新加载。
```

### 9.4 附件

ERP 附件不是附属品。

附件可能是：

```text
合同。
发票。
付款凭证。
收货照片。
报销票据。
质检报告。
审批材料。
```

设计要求：

```text
附件类型要清楚。
附件和单据状态要关联。
审批时要能快速查看附件。
归档后附件要可追溯。
敏感附件要控制权限。
```

### 9.5 套打、打印、导入、导出

ERP 不只是线上操作，还要和线下文档、财务凭证、审计材料、外部系统协作。

所以必须支持：

```text
套打。
正式打印。
导入模板。
导出结果。
失败明细。
批量处理。
```

设计关键：

```text
导入要有模板。
导入失败要有行号、字段、原因。
导出要说明范围。
打印要有模板和版本。
套打要和单据字段绑定。
```

## 10. 金蝶 ERP 的可扩展设计精髓

### 10.1 元数据驱动

金蝶式系统把业务对象、布局、规则、流程用元数据描述。

这带来几个能力：

```text
页面可以由模型生成。
字段可以配置。
流程可以配置。
规则可以配置。
视图可以配置。
权限可以配置。
API 可以由操作发布。
测试可以围绕模型生成。
```

这不是为了“少写代码”而低代码，而是为了企业能力资产化。

### 10.2 扩展和继承

ERP 标准产品很难一次满足所有客户。

所以必须支持：

```text
标准模型。
行业扩展。
客户扩展。
插件扩展。
脚本扩展。
字段扩展。
流程扩展。
视图扩展。
```

设计边界：

```text
扩展不能破坏标准模型。
扩展要能升级。
扩展要能追踪。
扩展要能关闭或回退。
扩展字段要进入权限、导入导出、打印、接口和测试。
```

### 10.3 多终端复用

金蝶公开资料提到业务建模与前端界面解耦，同一业务模型可支持 PC、移动端、手持设备等多终端。

这给我们的启发：

```text
模型是通用的。
页面是终端特定的。
PC 后台可以左树右表、宽表格、三栏。
移动端只承载高频任务和轻量审批。
手持设备承载扫码、盘点、收货、发货。
```

不要把移动端限制反过来削弱 ERP 模型。

也不要把 PC 复杂布局硬塞到移动端。

## 11. 金蝶 ERP 对后端管理界面的设计原则

### 11.1 通用原则

这些原则不只适用于 ERP，任何后端管理界面都应该遵守。

```text
对象清楚：页面必须让用户知道正在操作什么对象。
状态清楚：对象状态必须明确，并驱动可用操作。
入口清楚：用户从哪里来，当前处于什么业务上下文。
操作清楚：每个按钮的前置条件、结果、失败原因清楚。
反馈清楚：点击后必须有加载、成功、失败或可恢复状态。
错误清楚：错误要具体到字段、行、权限、状态或规则。
权限清楚：看不到、不能点、不能导出、不能审批都要有治理。
证据清楚：关键业务操作必须可追踪、可审计。
结构清楚：信息分组符合业务语义，不按数据库字段堆叠。
效率清楚：高频操作短路径，低频操作收纳，批量场景被支持。
```

### 11.2 ERP 特定原则

这些原则主要适用于 ERP、财务、供应链、单据系统。

```text
单据是业务对象，不是表单。
单据头和单据体要清楚。
业务日期、组织、往来单位、金额、数量是高风险字段。
状态控制字段编辑性和按钮可用性。
下推、选单、转换、反写必须可追溯。
审批流和业务流要融合。
来源单据和下游单据要可见。
部分执行状态必须可表达，例如部分入库、部分收款、部分结算。
金额口径必须清楚，例如暂估、核算、含税、不含税、已收、未收。
导入导出、套打、打印、附件是核心能力，不是附加能力。
```

### 11.3 金蝶类低代码平台特定原则

这些原则主要适用于金蝶苍穹、星空、BOS、低代码平台。

```text
业务对象、布局、规则、流程要元数据化。
标准模型和扩展模型要分层。
视图、字段、按钮、流程、权限、API 发布要可配置。
模型要能被测试、发布、部署、版本管理。
扩展要能继承标准能力，不破坏升级。
API 和界面操作要基于同一业务能力。
```

## 12. 金蝶 ERP 的页面评价标准

评价一个 ERP 后端管理页面，不看“是否好看”，先看下面 12 个问题。

### 12.1 是否识别了业务对象

必须回答：

```text
对象类型是什么？
对象编号是什么？
对象状态是什么？
对象属于哪个组织？
对象和哪些上游、下游有关？
```

### 12.2 是否识别了业务域

必须回答：

```text
这是财务、采购、销售、库存、生产、费用、权限还是报表？
页面影响哪些业务域？
页面是否跨组织、跨部门、跨账期？
```

### 12.3 是否有正确页面结构

必须回答：

```text
应该用左树右表，还是上筛下表？
应该用主从表，还是详情页？
应该用树表一体，还是左树右详情？
应该用工作台，还是普通列表？
应该用分步向导，还是一次性表单？
```

### 12.4 是否以状态驱动

必须回答：

```text
状态全集是什么？
当前状态是什么？
每个状态允许哪些字段编辑？
每个状态允许哪些按钮？
状态如何流转？
异常状态如何恢复？
```

### 12.5 是否解释了字段语义

必须回答：

```text
字段来源是什么？
字段影响什么？
字段什么时候必填？
字段什么时候可编辑？
字段是否影响下游？
字段是否影响报表？
```

### 12.6 是否支持上下游追溯

必须回答：

```text
来源单据是什么？
下游单据是什么？
是否部分下推？
是否反写？
是否生成凭证？
是否结算？
```

### 12.7 是否支持批量操作

必须回答：

```text
是否支持批量提交、审核、导出、打印、下推？
批量前是否校验？
部分失败如何展示？
失败明细能否导出？
```

### 12.8 是否支持权限和审计

必须回答：

```text
按钮权限在哪里？
字段权限在哪里？
数据权限在哪里？
谁操作过？
改了什么？
什么时候改的？
能否追踪导出和打印？
```

### 12.9 是否支持附件和正式输出

必须回答：

```text
附件是否分类？
附件是否参与审批？
打印模板是否清楚？
导入导出是否有模板和失败明细？
```

### 12.10 是否支持流程监控

必须回答：

```text
当前流程节点是什么？
当前处理人是谁？
是否超期？
是否异常？
能否查看流程实例？
能否查看流程效率？
```

### 12.11 是否支持扩展

必须回答：

```text
字段能否扩展？
按钮能否扩展？
流程能否扩展？
规则能否扩展？
扩展是否影响升级？
扩展是否进入权限、导出、打印、API？
```

### 12.12 是否可测试

必须回答：

```text
是否能按业务对象定位页面？
是否能按状态断言按钮和字段？
是否能测试下推、反写、审批、导入导出？
是否能生成证据？
是否能追溯失败原因？
```

## 13. 金蝶 ERP 的优秀设计不是组件，而是系统

很多人学习金蝶、ERP、企业后台，会学错方向。

错误学习方式：

```text
学它的表格长什么样。
学它的按钮放哪里。
学它的表单分几列。
学它的弹窗怎么做。
学它的颜色和间距。
```

正确学习方式：

```text
学它如何定义业务对象。
学它如何把单据作为企业流程节点。
学它如何让状态驱动操作。
学它如何让上下游业务可追溯。
学它如何处理共性与个性。
学它如何用元数据沉淀企业资产。
学它如何把 UI、流程、权限、规则、API、测试连成一个系统。
```

ERP 界面最终不是为了“美观”，而是为了：

```text
让企业业务可操作。
让业务过程可控制。
让业务结果可追溯。
让多人协作可审计。
让复杂变化可扩展。
让长期维护可治理。
```

## 14. 从金蝶 ERP 提炼出的后端管理界面总框架

后续整理文档时，可以按这个框架重写。

```text
第一层：通用后端管理界面原则
  对象、状态、入口、操作、反馈、错误、权限、审计、效率。

第二层：企业管理界面经典结构
  左树右表、上筛下表、主从表、三栏、工作台、树表一体、矩阵表。

第三层：ERP 单据设计规范
  单据头、单据体、状态、字段、编码、菜单、视图、转换、反写、附件、套打。

第四层：流程与审批设计规范
  入口单据、流程版本、审批节点、下推、选单、业务消息、流程实例。

第五层：权限与审计规范
  菜单、按钮、字段、数据、组织、导出、打印、日志、审批轨迹。

第六层：扩展与元数据规范
  模型、字段、视图、规则、流程、插件、脚本、API、版本、部署。

第七层：测试与证据规范
  状态测试、字段测试、流程测试、下推测试、反写测试、批量测试、审计证据。
```

这七层比“小程序页面规范”更准确，也更适合后端管理界面。

## 15. 金蝶 ERP 精髓清单

最后用一张清单总结。

```text
1. 业务域先行：页面属于明确业务域和子系统。
2. 业务对象先行：页面操作的是业务对象，不是数据库表。
3. 单据是核心：单据承载业务事实、状态、动作、流程和证据。
4. 元数据驱动：对象、布局、规则、流程、权限沉淀为模型资产。
5. 状态驱动界面：状态决定字段、按钮、流程、权限、报表口径。
6. 单据头/体分离：整单字段和分录字段必须分层。
7. 业务流与审批流融合：审批只是流程的一部分，业务下推才是 ERP 关键。
8. 上下游可追溯：来源、下游、反写、凭证、结算必须可查。
9. 共性与个性共存：标准模型、行业模型、客户扩展不能互相破坏。
10. 权限精细化：菜单、按钮、字段、数据、组织、导出、打印都要治理。
11. 审计内建：关键动作必须留痕。
12. 附件和套打是核心：企业业务需要正式材料和外部凭证。
13. 导入导出是核心：企业批量业务必须可处理、可失败定位。
14. 列表是处理中心：查询、筛选、批量、导出、下推都从列表发生。
15. 详情是证据中心：业务事实、状态、流程、日志、附件聚合。
16. 报表是追溯视图：指标必须能下钻到明细和原始单据。
17. 工作台是角色任务中心：按角色组织待办、异常、指标和入口。
18. 扩展必须可升级：插件、脚本、字段、流程扩展不能破坏标准能力。
19. API 与界面同源：能在界面操作的业务能力，应该能治理为服务能力。
20. 测试围绕模型：测试状态、流程、转换、反写、权限和证据，而不是只测按钮。
```

## 16. 最终结论

金蝶 ERP 的真正价值不在 UI 表面，而在企业业务复杂度的组织方式。

它把企业系统拆成：

```text
业务对象。
单据。
字段。
状态。
流程。
权限。
视图。
转换。
反写。
审计。
扩展。
发布。
测试。
```

然后用元数据和平台能力把这些东西连接起来。

因此，后续设计后端管理界面时，不能先问：

```text
这个页面怎么排版？
按钮放左边还是右边？
表格几列？
```

必须先问：

```text
这个页面属于哪个业务域？
它操作哪个业务对象？
它是不是单据？
它的状态机是什么？
它有哪些上下游？
它有哪些字段影响流程、报表、凭证、结算？
它的权限和审计怎么做？
它支持哪些批量业务？
它如何扩展？
它如何测试？
```

这才是金蝶 ERP 对后端管理界面设计最重要的启发。
