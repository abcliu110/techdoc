# 小程序信息架构模式与前端架构模式

## 1. 文档目的

本文整理企业级小程序常见的信息架构模式和前端工程架构模式。

信息架构模式解决：

```text
页面怎么组织信息？
用户先看什么、后看什么？
对象、状态、动作、结果之间如何关联？
```

前端工程架构模式解决：

```text
页面代码怎么组织？
状态、入口、接口、测试、证据如何治理？
如何让页面可测试、可追踪、可发布？
```

本文适用于：

- 小程序页面设计。
- POS / 收银 / 交易类页面评审。
- 企业级页面契约设计。
- 自动化测试可测性设计。
- 老页面重构前的信息架构分析。

## 2. 信息架构模式

### 2.1 列表 - 详情模式

也叫 Master-Detail。

```text
┌──────────────────────────────────────────────┐
│ 筛选 / 搜索 / 状态摘要                         │
├───────────────┬──────────────────────────────┤
│ 对象列表       │ 当前对象详情                   │
│ - 对象 A       │ 标题 / 状态 / 编号              │
│ - 对象 B 选中  │ 明细信息                        │
│ - 对象 C       │ 关联记录                        │
├───────────────┴──────────────────────────────┤
│ 对当前对象的操作：编辑 / 退款 / 打印 / 删除       │
└──────────────────────────────────────────────┘
```

适合：

```text
订单列表 + 订单详情
结账单列表 + 小票预览
会员列表 + 会员详情
退款列表 + 退款详情
```

关键要求：

```text
当前选中对象必须明确。
列表和详情必须强绑定。
详情区必须显示同一个业务编号。
操作按钮必须作用于当前选中对象。
```

### 2.2 工作台模式

```text
┌──────────────────────────────────────────────┐
│ 区域 / 搜索 / 系统状态 / 快捷工具               │
├──────────────────────────────────────────────┤
│                                              │
│ 业务对象网格 / 高密度操作区                    │
│                                              │
│ [对象1] [对象2] [对象3] [对象4]                │
│ [对象5] [对象6] [对象7] [对象8]                │
│                                              │
├──────────────────────┬───────────────────────┤
│ 状态筛选 / 数量统计   │ 高频操作 / 系统操作       │
├──────────────────────┴───────────────────────┤
│ 操作员 / 班次 / 营业日 / 设备状态               │
└──────────────────────────────────────────────┘
```

适合：

```text
桌台工作台
收银前台
厨房出餐屏
排队叫号
门店运营控制台
```

关键要求：

```text
对象状态必须一眼可见。
操作必须极快。
按钮必须按风险和频率分组。
当前选中对象必须明确。
系统运行状态必须可见。
```

### 2.3 查询筛选 - 结果模式

```text
┌──────────────────────────────────────────────┐
│ 查询条件                                      │
│ 日期 / 单号 / 状态 / 金额 / 更多筛选            │
├──────────────────────────────────────────────┤
│ 结果摘要                                      │
│ 共 N 条 / 合计金额 / 当前筛选条件               │
├──────────────────────────────────────────────┤
│ 结果列表                                      │
│ 表头                                          │
│ 行 1                                          │
│ 行 2                                          │
│ 行 3                                          │
├──────────────────────────────────────────────┤
│ 分页 / 导出 / 批量操作                         │
└──────────────────────────────────────────────┘
```

适合：

```text
交易流水
账单查询
库存查询
会员查询
报表明细
```

关键要求：

```text
筛选条件清楚。
当前生效条件可见。
结果数量可见。
合计口径明确。
空结果和错误状态明确。
```

### 2.4 表单向导模式

```text
┌──────────────────────────────────────────────┐
│ 步骤条：1 基础信息 -> 2 业务配置 -> 3 确认提交   │
├──────────────────────────────────────────────┤
│ 当前步骤标题                                  │
│ 当前步骤说明                                  │
├──────────────────────────────────────────────┤
│ 表单区域                                      │
│ 字段 A                                        │
│ 字段 B                                        │
│ 字段 C                                        │
├──────────────────────────────────────────────┤
│ 上一步 / 下一步 / 保存草稿 / 提交               │
└──────────────────────────────────────────────┘
```

适合：

```text
会员开卡
创建活动
申请退款
新增门店
复杂配置
```

关键要求：

```text
当前步骤明确。
每一步只处理一个任务。
已填信息可回看。
提交前有确认页。
失败后能返回修正。
```

### 2.5 单对象详情模式

```text
┌──────────────────────────────────────────────┐
│ 对象摘要                                      │
│ 编号 / 状态 / 关键金额 / 关键时间               │
├──────────────────────────────────────────────┤
│ 详情分组 A                                    │
│ 基础信息                                      │
├──────────────────────────────────────────────┤
│ 详情分组 B                                    │
│ 支付 / 退款 / 商品 / 会员等                     │
├──────────────────────────────────────────────┤
│ 操作区                                        │
│ 编辑 / 打印 / 退款 / 返回                       │
└──────────────────────────────────────────────┘
```

适合：

```text
订单详情
退款详情
会员详情
优惠券详情
门店详情
```

关键要求：

```text
对象编号突出。
对象状态突出。
关键金额或权限突出。
操作按钮按状态启用。
关联记录可追溯。
```

```text
│ 卡片 C     │           │ 卡片 J     │          │
```

```text

每列数量可见。
异常状态突出。
拖拽或操作必须可审计。
```

```text

│ 时间范围 / 门店 / 维度切换                     │
├───────────┬───────────┬───────────┬──────────┤
│ 核心指标1  │ 核心指标2  │ 核心指标3  │ 核心指标4 │
├───────────────────────┬──────────────────────┤

├───────────────────────┴──────────────────────┤
│ 明细列表 / 下钻入口                            │
└──────────────────────────────────────────────┘
```

适合：

```text
门店经营概览
销售看板
会员增长
库存预警
```

关键要求：

```text
指标口径明确。
时间范围明确。
异常指标可下钻。
图表不能只好看，要能解释业务。
```

### 2.8 任务流模式

```text
┌──────────────────────────────────────────────┐
│ 任务筛选 / 优先级 / 状态统计                   │
├───────────────┬──────────────────────────────┤
│ 待处理任务列表  │ 当前任务详情                   │
│ - 任务 A       │ 任务说明                        │
│ - 任务 B 选中  │ 影响对象                        │
│ - 任务 C       │ 处理记录                        │
├───────────────┴──────────────────────────────┤
│ 处理动作：接受 / 拒绝 / 完成 / 转交              │
└──────────────────────────────────────────────┘
```

适合：

```text
待退款
待审核
待接单
待出餐
待处理异常订单
```

关键要求：

```text
优先级清楚。
超时任务突出。
批量处理和单个处理区分。
处理结果可追踪。
```

### 2.9 搜索优先模式

```text

│ 大搜索框                                      │
│ 输入订单号 / 小票号 / 手机号 / 桌台号            │
├──────────────────────────────────────────────┤

│ 结果 1                                        │
│ 结果 2                                        │
│ 结果 3                                        │
└──────────────────────────────────────────────┘
```

适合：

```text
查订单
查会员
查小票
查桌台
```

关键要求：

```text
输入焦点明确。

搜索历史或最近对象可用。
```

```text
├───────────────┬──────────────────────────────┤

│ - 同步失败选中 │ 影响范围                        │
```

适合：


```text

影响对象明确。

处理结果可审计。
```

## 3. 前端工程架构模式

### 3.1 页面契约驱动架构

每个页面都有 contract。

```text
pageContract
  -> pageIdentity
  -> entries
  -> params
  -> stateMachine
  -> bizContext
  -> actions
  -> evidence
```

适合：

```text
企业级小程序
复杂 POS
交易系统
自动化测试要求高的项目
```

价值：

```text
开发、测试、日志、后端证据统一语言。
```

### 3.2 状态机驱动架构

```text
event -> transition -> pageState -> UI
```

适合：

```text
支付
退款
订单确认
登录
复杂表单
```

价值：

```text
避免布尔变量混乱。
测试可等待状态。
```

### 3.3 业务上下文驱动架构

```text
data-page-id
data-state
data-biz-type
data-biz-no
data-trace-id
```

适合：

```text
需要自动化测试和后端 evidence 的页面。
```

价值：

```text
UI 和后端业务结果能串起来。
```

### 3.4 Router Contract 架构

```text
router.open(pageId, entryId, params)
  -> validate contract
  -> navigate
  -> resolveEntryContext
```

适合：

```text
分享、扫码、订阅消息、支付回跳多的项目。
```

价值：

```text
入口可治理。
冷启动可测试。
```

### 3.5 Service + API 分层架构

```text
Page
  -> Service
    -> API
      -> request wrapper
```

适合：

```text
所有中大型项目。
```

价值：

```text
页面干净。
接口统一处理。
错误统一映射。
```

### 3.6 Evidence Driven Testing 架构

```text
UI action
  -> traceId
  -> bizNo
  -> backend evidence
  -> assertion
```

适合：

```text
支付、退款、订单、会员、积分、库存。
```

价值：

```text
页面成功和业务成功能区分。
```

### 3.7 Test Bridge 架构

测试包暴露测试桥，正式包剥离。

```text
getApp().__TEST_BRIDGE__
  -> pageState
  -> actionLog
  -> apiLog
  -> bizContext
```

适合：

```text
小程序 UI 自动化。
```

价值：

```text
自动化更稳定。
失败更好定位。
```

### 3.8 权限 / 能力适配层架构

```text
Page
  -> PlatformService
    -> wx.login / wx.pay / wx.scanCode
```

适合：

```text
登录、支付、扫码、定位、相机、蓝牙。
```

价值：

```text
兼容、授权、错误处理统一。
```

### 3.9 Feature Flag / 灰度架构

```text
feature.xxx.enabled
  -> old flow / new flow
```

适合：

```text
支付改造
退款改造
状态机改造
入口治理改造
```

价值：

```text
可灰度。
可回滚。
```

### 3.10 设计系统架构

```text
Design Tokens
  -> StatusTag
  -> ActionButton
  -> PageShell
  -> Empty/Error/Loading
```

适合：

```text
多页面、多团队项目。
```

价值：

```text
页面一致。
状态语义统一。
测试也能统一识别。
```

## 5. 选择口诀

```text
找对象：搜索优先 / 查询结果
看对象：单对象详情 / 明细分组
选对象：列表详情 / 卡片网格
处理对象：任务流 / 工作台
看状态：状态看板 / 监控告警
做高风险动作：决策确认
看动作结果：结果回执
核对差异：对账核对
做企业级治理：页面契约驱动
做自动化测试：状态机 + 业务上下文 + evidence
```

## 6. 模式选择表

| 问题 | 优先模式 |
|---|---|
| 要从很多对象中选一个看详情 | 列表 - 详情模式 |
| 要高频操作多个对象 | 工作台模式 / 卡片网格模式 |
| 要查数据 | 查询筛选 - 结果模式 |
| 要一步步填写 | 表单向导模式 |
| 要看单个对象全部信息 | 单对象详情 / 明细分组 |
| 要处理不同状态流转 | 状态看板模式 |
| 要看指标 | 仪表盘模式 |
| 要找一个对象 | 搜索优先模式 |
| 要处理异常 | 异常处理模式 |
| 要比对两边数据 | 对账 / 核对模式 |
| 要做高风险操作 | 决策确认模式 |
| 要展示操作结果 | 结果回执模式 |
| 要做自动化测试 | 状态机驱动 + 业务上下文 + evidence |
| 要治理入口 | Router Contract 架构 |
| 要治理页面整体 | 页面契约驱动架构 |

## 7. 最终结论

页面设计不要先问“这个页面怎么排版”，而要先问：

```text
用户是在找对象、看对象、选对象、处理对象，还是核对对象？
这个页面有没有高风险动作？
这个页面结果是否需要后端证据证明？
这个页面是否需要快速自动化测试？
```

信息架构模式决定页面如何被用户理解。

前端工程架构模式决定页面如何被开发、测试、日志、后端证据和发布系统治理。

## 8. 经典原型结构补充

本文重点讲信息架构和前端工程架构。企业系统中还必须单独识别一类更基础的页面骨架，也就是经典原型结构。

典型结构包括：

```text
左树右表。
左树右详情。
左列表右详情。
上筛下表。
上摘要下明细。
主从表。
三栏结构。
分步向导。
工作台。
看板。
矩阵表。
树表一体。
图表钻取。
时间轴详情。
双栏穿梭。
抽屉/弹窗辅助。
```

其中“左树右表”是企业系统非常经典的结构：

```text
左侧解决范围：组织、部门、分类、区域、仓库、权限节点。
右侧解决对象：当前范围下的数据列表、查询、批量操作。
```

