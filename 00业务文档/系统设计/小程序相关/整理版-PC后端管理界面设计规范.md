# 整理版：PC 后端管理界面设计规范

## 1. 文档定位

本文专门整理 PC 后端管理界面、ERP、SaaS 后台、业务中台、运营后台、金蝶类单据系统的设计规范。

它不面向小程序小屏场景。

本文关注的是：

```text
PC 大屏。
鼠标键盘。
高密度数据。
复杂筛选。
批量操作。
权限和审计。
ERP 单据。
报表和钻取。
导入导出。
多角色协同。
```

不包含：

```text
小程序页面栈。
tabBar。
分包。
授权弹窗。
扫码。
移动端底部操作栏。
移动端卡片列表。
```

## 2. PC 后台设计的核心目标

PC 后台不是营销页面，也不是移动端页面放大版。

PC 后台的核心目标是：

```text
让业务对象可识别。
让高密度数据可检索。
让批量任务可高效完成。
让复杂状态可判断。
让高风险操作可控制。
让多人协作可审计。
让异常问题可追溯。
让自动化测试可稳定执行。
```

PC 后台设计的关键词：

```text
对象。
状态。
范围。
筛选。
表格。
批量。
权限。
审计。
证据。
效率。
```

## 3. PC 后台和小程序的边界

| 维度 | PC 后台 | 小程序 / 移动端 |
|---|---|---|
| 屏幕 | 大屏，可多栏、宽表格 | 小屏，纵向流 |
| 输入 | 鼠标、键盘、快捷键、多选 | 触摸、扫码、拍照、定位 |
| 任务 | 批量处理、复杂配置、长时间作业 | 单对象、短流程、轻量处理 |
| 信息密度 | 高 | 低到中 |
| 常见结构 | 左树右表、主从表、矩阵、三栏 | 卡片列表、详情分组、底部操作 |
| 风险 | 批量误操作、导出泄漏、权限越界 | 重复提交、弱网、授权失败 |
| 测试重点 | 表格、筛选、批量、权限、审计 | 页面栈、授权、支付、弱网 |

PC 后台可以承载复杂结构，但不能因此变得混乱。

设计目标不是把信息都放上去，而是让用户在复杂信息中快速判断和操作。

## 4. PC 后台页面的基础骨架

一个标准 PC 后台页面通常由这些区域组成：

```text
页面标题区。
业务摘要区。
查询筛选区。
操作工具栏。
主内容区。
批量操作区。
分页区。
详情抽屉或弹窗。
审计/日志/附件区。
```

### 4.1 页面标题区

标题区必须告诉用户：

```text
当前页面是什么？
管理什么对象？
属于哪个模块？
是否有当前范围？
是否有当前状态视图？
```

示例：

```text
销售订单
库存明细
会员管理
商品分类
付款审批
应收单列表
```

错误做法：

```text
标题只写“列表”。
标题只写“管理”。
用户不知道当前模块和对象。
```

### 4.2 业务摘要区

不是所有页面都需要摘要。

适合有统计和待办的页面：

```text
订单总数。
待审核数量。
异常数量。
未结算金额。
库存预警数量。
待处理任务。
```

摘要区必须可下钻。

不能只展示数字：

```text
异常 12
```

要能点击进入异常列表，并带上筛选条件。

### 4.3 查询筛选区

查询区解决“如何找到对象”。

设计原则：

```text
高频筛选外露。
低频筛选折叠。
默认条件安全。
重置回到默认条件。
查询条件可回显。
筛选结果可解释。
```

高频筛选通常包括：

```text
关键词。
状态。
日期范围。
组织/门店。
业务类型。
负责人。
金额区间。
```

### 4.4 操作工具栏

工具栏解决“对当前对象集合做什么”。

操作分层：

```text
主操作：新增、导入、创建。
批量操作：批量审核、批量导出、批量停用。
辅助操作：刷新、列设置、打印。
风险操作：删除、作废、反审核。
```

原则：

```text
高频主操作外露。
风险操作不要放在显眼主按钮位置。
批量操作必须在选中对象后明确影响范围。
```

### 4.5 主内容区

主内容区通常是：

```text
表格。
树。
看板。
矩阵。
报表。
详情。
```

它必须服务页面主任务，而不是展示所有字段。

### 4.6 批量操作区

批量操作必须显示：

```text
已选数量。
影响范围。
是否跨页。
是否跨组织。
失败后如何处理。
```

示例：

```text
已选择 25 条，仅处理当前页。
已选择全部查询结果 1200 条，操作将影响所有筛选结果。
```

### 4.7 分页区

分页不仅是组件。

必须定义：

```text
默认每页数量。
最大每页数量。
切换页是否保留筛选。
切换页是否保留选中。
跨页选择策略。
```

## 5. 信息密度设计

PC 后台可以高密度，但高密度不是拥挤。

### 5.1 信息优先级

页面信息分四级：

```text
P0：决定用户能否识别对象的信息。
P1：决定用户能否判断状态和风险的信息。
P2：辅助用户完成操作的信息。
P3：低频查看、审计、扩展信息。
```

表格列示例：

```text
P0：单号、名称、客户、组织。
P1：状态、金额、日期、异常标识。
P2：负责人、来源、备注。
P3：创建人、修改时间、扩展字段。
```

### 5.2 首屏原则

PC 后台首屏必须优先承载：

```text
页面身份。
关键筛选。
主要操作。
结果列表。
当前状态或范围。
```

不应让说明文案、装饰图、低频统计占据主区域。

### 5.3 折叠原则

可以折叠：

```text
高级筛选。
低频字段。
审计日志。
附件详情。
扩展配置。
```

不应折叠：

```text
对象编号。
状态。
金额。
日期。
当前范围。
主操作。
错误原因。
```

## 6. 经典结构的问题模型与模式原理

PC 后台经典结构不是视觉布局套路，而是业务数据关系的外化。

设计页面前必须先识别：

```text
用户要解决什么问题？
业务对象之间是什么关系？
用户是先选范围，还是先查对象？
用户是处理单个对象，还是批量处理对象？
用户是看状态流转，还是看层级结构？
用户是配置规则，还是分析指标？
```

只有先识别问题模型，才能选对页面模式。

### 6.1 左树右表：范围选择模型

#### 6.1.1 它解决什么问题

左树右表解决的是：

```text
数据很多，必须先确定业务范围，再处理该范围下的数据对象。
```

在企业系统中，用户很少真的想看“全部数据”。

用户通常想看：

```text
某个组织下的员工。
某个分类下的商品。
某个仓库下的库存。
某个区域下的门店。
某个菜单节点下的权限。
某个科目下的明细。
```

如果只用普通筛选，用户需要在一堆条件里反复选择范围。

左树右表把范围常驻在左侧，让用户始终知道当前正在处理哪个范围。

#### 6.1.2 底层模型

左树右表的底层模型是：

```text
ScopeTree + ObjectTable

ScopeTreeNode -> QueryScope -> ObjectCollection
```

也可以写成：

```text
范围节点 -> 查询范围 -> 对象集合 -> 操作集合
```

例如：

```text
商品分类节点 -> categoryId -> 商品列表 -> 新增/编辑/导出/批量停用
```

#### 6.1.3 结构模式

```text
┌──────────────┬───────────────────────────────┐
│ Scope Tree   │ Current Scope + Query + Action │
│              ├───────────────────────────────┤
│ Tree Node    │ Object Table                   │
│ Tree Node    │ Row / Row / Row                │
└──────────────┴───────────────────────────────┘
```

信息流：

```text
选择树节点 -> 生成查询范围 -> 加载右侧表格 -> 对当前范围内对象操作
```

#### 6.1.4 适用条件

必须同时满足：

```text
存在稳定层级范围。
用户经常按层级范围处理数据。
范围切换频繁。
右侧对象数量较多。
当前范围会影响新增、批量、导出、权限。
```

典型：

```text
组织 -> 员工。
商品分类 -> 商品。
仓库 -> 库存。
区域 -> 门店。
菜单 -> 权限点。
```

#### 6.1.5 不适用条件

不适合：

```text
没有层级范围。
层级范围只是一个普通筛选条件。
用户主要靠搜索定位对象。
树节点很少，用下拉框即可。
树层级太深且没有搜索。
右侧数据和左侧树没有强约束关系。
```

误用示例：

```text
左侧放一个状态树：全部、待审核、已审核、已关闭。
右侧放订单表。
```

如果状态只是筛选条件，用状态 Tab 或筛选器更合适，不一定要左树右表。

#### 6.1.6 核心设计变量

左树右表必须定义这些变量：

```text
currentScope：当前范围。
includeChildren：是否包含子节点。
scopePermission：当前范围权限。
scopeCount：节点数量口径。
tableQuery：右表查询条件。
selectedRows：右表选中对象。
batchScope：批量操作范围。
exportScope：导出范围。
```

#### 6.1.7 典型状态

```text
treeLoading。
treeLoaded。
treeLoadFailed。
scopeSelected。
scopeNoPermission。
tableLoading。
tableLoaded。
tableEmpty。
tableLoadFailed。
batchSelecting。
batchOperating。
```

#### 6.1.8 设计关键

```text
左侧树当前节点必须明显。
右侧必须显示当前范围。
父节点是否包含子节点必须明确。
切换范围后分页和筛选规则必须明确。
新增对象默认归属当前范围。
批量操作必须说明是否影响子节点。
导出必须说明导出范围。
```

#### 6.1.9 设计本质

左树右表的本质不是左右布局。

它的本质是：

```text
业务范围控制对象集合。
```

只要这个关系不存在，就不该用左树右表。

### 6.2 上筛下表：查询结果模型

#### 6.2.1 它解决什么问题

上筛下表解决的是：

```text
用户需要通过多个条件从大量对象中找到目标集合。
```

它不是按层级范围浏览，而是按条件检索。

例如：

```text
查某个日期范围内的订单。
查某个状态的退款单。
查某个客户的应收单。
查某个金额区间的付款记录。
```

#### 6.2.2 底层模型

```text
QueryForm -> QueryCondition -> ResultSet -> RowAction / BatchAction
```

也就是：

```text
查询条件 -> 结果集合 -> 单行操作 / 批量操作
```

#### 6.2.3 结构模式

```text
┌──────────────────────────────────────────────┐
│ Query Conditions                              │
├──────────────────────────────────────────────┤
│ Actions                                       │
├──────────────────────────────────────────────┤
│ Result Table                                  │
└──────────────────────────────────────────────┘
```

#### 6.2.4 适用条件

```text
对象没有必须常驻的层级范围。
用户主要通过关键词、日期、状态、组织、金额等条件查询。
结果需要排序、分页、导出、批量处理。
```

#### 6.2.5 核心设计变量

```text
defaultQuery：默认查询。
currentQuery：当前查询。
dirtyQuery：已修改但未查询的条件。
resultSet：结果集合。
selectedRows：选中行。
exportScope：导出范围。
resetPolicy：重置策略。
```

#### 6.2.6 设计关键

```text
默认查询必须安全。
高级筛选必须折叠。
重置必须回到默认条件，而不是危险全量。
查询条件必须可回显。
导出范围必须明确。
空结果要区分无数据和筛选过窄。
```

#### 6.2.7 设计本质

上筛下表的本质是：

```text
查询条件定义结果集合。
```

如果用户不是通过条件找数据，而是通过范围或状态推进对象，就应考虑其他模式。

### 6.3 主从表：主对象约束明细模型

#### 6.3.1 它解决什么问题

主从表解决的是：

```text
用户需要同时查看主对象和该主对象下的明细集合。
```

例如：

```text
订单和订单明细。
入库单和入库分录。
对账单和流水。
门店和设备。
客户和联系人。
```

#### 6.3.2 底层模型

```text
MasterRecord -> DetailCollection
```

也就是：

```text
主对象决定从对象范围。
```

#### 6.3.3 结构模式

```text
┌──────────────────────────────────────────────┐
│ Master Table                                  │
│ Row A  Row B  Row C                           │
├──────────────────────────────────────────────┤
│ Detail Table for Selected Master              │
│ Detail / Detail / Detail                      │
└──────────────────────────────────────────────┘
```

#### 6.3.4 适用条件

```text
主对象数量较多。
每个主对象都有明细。
用户需要频繁切换主对象并查看明细。
明细和主对象强绑定。
```

#### 6.3.5 不适用条件

```text
用户主要查看单个对象完整详情。
明细字段非常复杂，适合详情页。
主对象数量少，直接详情页即可。
明细需要大量编辑，不适合压在下方小区域。
```

#### 6.3.6 设计关键

```text
主表当前行必须明显。
从表必须显示当前主对象。
主表切换后从表必须刷新。
从表为空要说明原因。
从表操作影响主表状态时必须反馈。
```

#### 6.3.7 设计本质

主从表的本质是：

```text
一个主对象控制一个明细集合。
```

### 6.4 树表一体：层级对象模型

#### 6.4.1 它解决什么问题

树表一体解决的是：

```text
对象本身有层级，并且每个层级对象都有属性需要展示或编辑。
```

例如：

```text
菜单资源。
权限点。
组织结构。
会计科目。
物料 BOM。
分类及其属性。
```

#### 6.4.2 底层模型

```text
HierarchicalObjectCollection

Node -> Children -> NodeAttributes -> NodeActions
```

#### 6.4.3 和左树右表的区别

```text
左树右表：左树是范围，右表是范围内对象。
树表一体：表格每一行就是层级对象。
```

例如：

```text
商品分类 -> 商品列表：左树右表。
商品分类本身的名称、编码、状态、排序维护：树表一体。
```

#### 6.4.4 设计关键

```text
层级缩进必须清楚。
展开收起状态稳定。
新增同级和新增下级必须区分。
父节点状态对子节点影响必须明确。
移动节点必须有校验。
```

#### 6.4.5 设计本质

树表一体的本质是：

```text
层级对象集合的属性维护。
```

### 6.5 三栏工作界面：队列处理模型

#### 6.5.1 它解决什么问题

三栏工作界面解决的是：

```text
用户需要连续处理大量待办对象，并在分类、列表、详情之间快速切换。
```

例如：

```text
审批中心。
工单中心。
客服工作台。
异常处理中心。
消息中心。
```

#### 6.5.2 底层模型

```text
QueueCategory -> QueueItems -> CurrentItemDetail -> Action
```

#### 6.5.3 结构模式

```text
┌──────────┬──────────────┬────────────────────┐
│ Category │ Queue Items  │ Current Detail      │
│          │              │ Actions             │
└──────────┴──────────────┴────────────────────┘
```

#### 6.5.4 适用条件

```text
用户需要连续处理。
任务有队列分类。
每个对象需要查看详情后操作。
处理完成后队列状态会变化。
```

#### 6.5.5 设计关键

```text
分类数量要可理解。
列表要展示优先级、状态、超期。
详情要突出决策信息。
处理完成后队列数量必须刷新。
是否自动进入下一条必须明确。
```

#### 6.5.6 设计本质

三栏工作界面的本质是：

```text
队列化连续处理。
```

### 6.6 矩阵表：交叉配置模型

#### 6.6.1 它解决什么问题

矩阵表解决的是：

```text
一个配置结果由行对象和列对象交叉决定。
```

例如：

```text
角色 x 权限。
门店 x 商品。
人员 x 日期。
客户等级 x 价格策略。
渠道 x 能力开关。
```

#### 6.6.2 底层模型

```text
RowDimension x ColumnDimension -> CellValue
```

#### 6.6.3 设计关键

```text
冻结首行首列。
高亮当前行列。
格子含义清楚。
批量勾选范围清楚。
保存前展示变更数量。
高风险配置有版本和回滚。
```

#### 6.6.4 设计本质

矩阵表的本质是：

```text
二维关系配置。
```

如果只是一组普通字段，不应强行做矩阵。

### 6.7 看板：状态流转模型

#### 6.7.1 它解决什么问题

看板解决的是：

```text
用户需要按状态阶段推进对象。
```

例如：

```text
线索阶段。
商机阶段。
工单状态。
售后处理。
招聘流程。
任务进度。
```

#### 6.7.2 底层模型

```text
StatusColumn -> ObjectCards
ObjectCard + Move -> StateTransition
```

#### 6.7.3 设计关键

```text
列就是状态。
拖拽就是状态变更。
状态变更必须有权限和校验。
失败必须回滚。
每列数量、超期、异常要可见。
```

#### 6.7.4 不适用条件

```text
对象没有明确阶段。
用户主要通过筛选查数据。
状态很多且复杂。
单个对象信息密度很高，卡片无法承载。
```

#### 6.7.5 设计本质

看板的本质是：

```text
状态驱动的对象推进。
```

### 6.8 图表钻取：指标追溯模型

#### 6.8.1 它解决什么问题

图表钻取解决的是：

```text
用户看到一个指标后，需要追溯它由哪些明细和原始对象构成。
```

例如：

```text
销售额下降 -> 哪些门店下降 -> 哪些订单下降。
库存异常 -> 哪些仓库异常 -> 哪些商品异常。
应收金额 -> 哪些客户 -> 哪些应收单。
```

#### 6.8.2 底层模型

```text
Metric -> DimensionBreakdown -> DetailRows -> SourceObject
```

#### 6.8.3 设计关键

```text
指标口径清楚。
当前钻取条件可见。
图表筛选和明细筛选一致。
明细能回到原始业务对象。
导出时保留钻取条件。
```

#### 6.8.4 设计本质

图表钻取的本质是：

```text
从经营结果追溯到业务事实。
```

### 6.9 导入向导：外部数据入库模型

#### 6.9.1 它解决什么问题

导入向导解决的是：

```text
外部批量数据进入系统前，必须经过格式、字段、业务规则和影响范围校验。
```

#### 6.9.2 底层模型

```text
File -> FieldMapping -> ValidationResult -> Confirm -> ImportJob -> ImportResult
```

#### 6.9.3 设计关键

```text
下载模板。
上传文件。
字段映射。
数据校验。
失败明细。
确认导入。
导入结果。
幂等处理。
```

#### 6.9.4 设计本质

导入向导的本质是：

```text
高风险批量写入的受控流程。
```

## 7. 经典结构设计规范

### 7.1 左树右表

#### 定义

左树右表用于表达：

```text
左侧确定业务范围，右侧处理该范围下的对象集合。
```

#### 适用场景

```text
组织树 -> 员工表。
部门树 -> 岗位表。
商品分类树 -> 商品表。
仓库树 -> 库存表。
区域树 -> 门店表。
菜单树 -> 权限点表。
科目树 -> 明细表。
客户分组树 -> 客户表。
```

#### 标准结构

```text
┌──────────────────────────────────────────────┐
│ 页面标题                                      │
├──────────────┬───────────────────────────────┤
│ 范围树        │ 当前范围 / 查询 / 操作栏         │
│              ├───────────────────────────────┤
│ 全部          │ 表格                           │
│ ├ 华东区      │ 编号  名称  状态  负责人  操作   │
│ │ ├ 上海      │ 001  门店A 启用  张三    编辑   │
│ │ └ 杭州      │ 002  门店B 停用  李四    查看   │
└──────────────┴───────────────────────────────┘
```

#### 必须定义的规则

```text
点击树节点后右表是否立即刷新？
父节点是否包含子节点数据？
切换节点后分页是否重置？
筛选条件是否保留？
新增对象是否默认归属当前节点？
批量操作是否包含子节点？
导出范围是当前节点还是当前筛选结果？
无权限节点是隐藏还是禁用？
```

#### 设计标准

```text
当前树节点必须明显。
右侧必须显示当前范围。
树节点过深要支持搜索。
树节点数量要有口径说明。
右表数据必须和当前范围一致。
批量操作必须受范围约束。
```

#### 反模式

```text
左树只是装饰，右表和树无关。
右表没有当前范围提示。
父节点是否包含子节点不清楚。
切换节点后筛选残留导致误判。
无权限节点和无数据节点表现一样。
```

### 7.2 上筛下表

#### 定义

上筛下表用于表达：

```text
先定义查询条件，再展示结果集合。
```

#### 适用场景

```text
订单列表。
单据列表。
流水列表。
会员列表。
凭证列表。
库存列表。
操作日志。
支付记录。
```

#### 标准结构

```text
┌──────────────────────────────────────────────┐
│ 查询条件：单号 | 日期 | 状态 | 门店 | 客户       │
├──────────────────────────────────────────────┤
│ 操作栏：新增 | 批量审核 | 导出 | 打印            │
├──────────────────────────────────────────────┤
│ 表格：单号  日期  客户  金额  状态  操作          │
├──────────────────────────────────────────────┤
│ 已选 N 条 | 共 M 条 | 分页                     │
└──────────────────────────────────────────────┘
```

#### 设计标准

```text
默认查询条件必须安全。
高级筛选折叠。
查询后条件可回显。
重置回到默认条件。
导出范围明确。
表格列必须服务业务判断。
```

#### 反模式

```text
筛选条件太多，占满首屏。
默认全量查询全公司数据。
重置后变成危险查询。
导出不知道是当前页还是全部结果。
```

### 7.3 主从表

#### 定义

主从表用于表达：

```text
一个主对象对应一组从属明细。
```

#### 适用场景

```text
订单 -> 订单明细。
入库单 -> 入库分录。
客户 -> 联系人。
门店 -> 设备。
对账单 -> 流水。
活动 -> 商品。
```

#### 设计标准

```text
主表当前行必须明显。
从表标题必须说明属于哪个主对象。
切换主表行后从表必须刷新或清空。
从表为空必须说明原因。
从表操作影响主对象时必须反馈。
```

### 7.4 树表一体

#### 定义

树表一体用于表达：

```text
表格行本身具有层级关系。
```

#### 适用场景

```text
菜单资源。
权限点。
会计科目。
组织层级。
物料 BOM。
分类带属性。
```

#### 设计标准

```text
层级缩进清楚。
展开收起状态稳定。
新增同级和新增下级区分。
父子状态关系明确。
拖拽移动必须有业务校验。
```

### 7.5 三栏工作界面

#### 定义

三栏工作界面用于高频处理：

```text
队列分类 -> 对象列表 -> 当前对象详情和操作。
```

#### 适用场景

```text
审批中心。
工单中心。
客服工作台。
异常处理中心。
消息处理中心。
```

#### 设计标准

```text
左栏分类数量可理解。
中栏列表展示优先级和状态。
右栏突出当前对象和主操作。
处理完成后队列、列表、详情同步刷新。
连续处理下一条规则明确。
```

### 7.6 矩阵表

#### 定义

矩阵表用于表达：

```text
行和列交叉形成配置结果。
```

#### 适用场景

```text
权限矩阵。
价格矩阵。
排班表。
门店商品配置。
规则配置。
```

#### 设计标准

```text
冻结首行首列。
悬停高亮当前行列。
批量勾选说明范围。
保存前展示变更数量。
支持恢复默认或回滚。
关键配置要有审计。
```

### 7.7 图表钻取

#### 定义

图表钻取用于表达：

```text
指标 -> 明细 -> 原始业务对象。
```

#### 设计标准

```text
指标口径清楚。
统计范围清楚。
钻取条件可见。
明细筛选和图表条件一致。
明细能回到原始单据。
```

### 7.8 导入向导

#### 定义

导入向导用于高风险批量数据进入系统。

标准流程：

```text
下载模板 -> 上传文件 -> 字段映射 -> 数据校验 -> 确认导入 -> 结果反馈
```

必须支持：

```text
失败行号。
失败字段。
失败原因。
失败明细下载。
重复导入处理。
成功和失败数量统计。
```

## 8. 表格设计规范

### 8.1 表格不是数据库表

表格列不是数据库字段平铺。

表格列必须服务用户判断：

```text
识别对象。
判断状态。
判断风险。
决定操作。
追踪来源。
```


架构师视角：

```text
表格是对象集合的决策界面，不是数据库记录浏览器。
```

设计师视角：

```text
用户扫表时不是逐字阅读，而是在寻找对象、状态、异常、金额、时间和可执行动作。
```

因此表格设计要先定义用户判断路径：

```text
第一眼识别对象。
第二眼判断状态。
第三眼发现异常。
第四眼决定操作。
```

如果表格列只是把接口字段全部铺出来，页面看似完整，实际会增加用户认知负担。
### 8.2 列设计

推荐列分组：

```text
身份列：编号、名称、客户、组织。
状态列：业务状态、审批状态、支付状态。
时间列：业务日期、创建时间、更新时间。
金额列：应收、已收、未收、成本、利润。
责任列：负责人、审批人、创建人。
来源列：来源单据、来源渠道、入口。
操作列：查看、编辑、审批、更多。
```


列设计要遵守“三类列”模型：

```text
识别列：帮助用户知道这行是什么。
判断列：帮助用户知道这行现在怎么样。
行动列：帮助用户知道接下来能做什么。
```

列顺序不是技术顺序，而是决策顺序。创建时间、修改人、扩展字段通常不应该放在前面抢占判断空间。
### 8.3 固定列

适合固定：

```text
左侧对象身份列。
右侧操作列。
```

不应固定过多列，否则横向滚动会混乱。

### 8.4 状态列

状态必须：

```text
文案稳定。
颜色语义稳定。
可筛选。
可测试。
和后端枚举一致。
```

不要：

```text
同一个状态不同页面叫法不同。
状态只靠颜色。
状态色和风险色冲突。
```

### 8.5 操作列

操作列原则：

```text
高频安全操作外露。
低频操作放更多。
风险操作二次确认。
不可用操作要说明原因，不能只灰掉。
```

## 9. 筛选和查询规范

### 9.1 筛选分类

筛选分为：

```text
关键词筛选。
日期范围。
状态筛选。
组织/门店筛选。
业务类型。
金额区间。
负责人。
高级条件。
```

### 9.2 默认筛选

默认筛选必须安全。

常见默认：

```text
最近 7 天。
当前组织。
当前门店。
待处理状态。
未归档数据。
```

不要默认查全量历史数据。


架构师视角：

```text
默认筛选是页面的数据安全边界。
```

设计师视角：

```text
默认筛选决定用户进入页面后的第一眼世界。
```

后台页面不应该默认把所有历史数据、所有组织数据、所有状态数据都查出来。默认条件要同时控制性能风险、权限风险、认知风险和批量误操作风险。
### 9.3 重置规则

重置不是一定清空。

重置应回到默认安全条件。

必须明确：

```text
重置后日期是什么？
组织是否保留？
状态是否回默认？
分页是否回第一页？
```

### 9.4 查询结果说明

查询结果必须显示：

```text
结果数量。
当前页。
当前筛选条件。
空结果原因。
```

## 10. 批量操作规范

### 10.1 批量操作分类

```text
批量审核。
批量提交。
批量导出。
批量打印。
批量停用。
批量删除。
批量分配。
批量下推。
```

### 10.2 批量操作前

必须显示：

```text
选中数量。
影响范围。
是否跨页。
是否跨组织。
是否包含不可操作对象。
```


架构师视角：

```text
批量操作是集合级写操作，风险远高于单行操作。
```

设计师视角：

```text
用户在批量操作前最需要知道的不是按钮在哪里，而是这次操作会影响谁。
```

因此批量操作前必须先回答：选中了多少条、是否跨页、是否跨组织、是否包含不可操作对象、是否可撤销、失败后如何恢复。
### 10.3 批量操作中

必须显示：

```text
处理中状态。
进度。
是否可取消。
是否允许离开页面。
```

### 10.4 批量操作后

必须显示：

```text
成功数量。
失败数量。
跳过数量。
失败明细。
失败原因。
下载失败报告。
```

### 10.5 批量反模式

```text
失败只提示“部分失败”。
不告诉哪几条失败。
不告诉失败字段和原因。
跨页选择不说明。
导出范围不说明。
```

## 11. ERP 单据设计规范

### 11.1 单据不是表单

ERP 单据是业务对象和流程节点。

单据必须包含：

```text
单据类型。
单据编号。
业务组织。
业务日期。
单据状态。
单据头。
单据体。
分录。
菜单操作。
编码规则。
状态流转。
上游单据。
下游单据。
单据转换。
反写。
审批。
附件。
套打。
导入导出。
审计日志。
```


架构师视角：

```text
单据是企业业务流中的状态化对象，不是一次性录入表单。
```

设计师视角：

```text
用户看单据时，真正关心的是这张单据是什么、到哪一步了、影响了什么、还能做什么。
```

所以单据页面必须同时表达事实、状态、关系、动作和证据。如果单据页面只展示字段和保存按钮，它就还停留在 CRUD 层，不是 ERP 层。
### 11.2 单据头

单据头承载整单字段：

```text
单据编号。
业务日期。
组织。
客户/供应商。
部门。
经办人。
业务类型。
币种。
备注。
```

### 11.3 单据体

单据体承载分录字段：

```text
物料/商品。
仓库。
批次。
数量。
单价。
金额。
税率。
行状态。
来源行。
```

### 11.4 单据状态

状态决定页面行为：

```text
草稿：可编辑、可保存、可提交。
已提交：不可随意编辑，可撤回或审批。
审核中：显示当前审批节点。
已审核：可下推、打印、生成凭证。
已驳回：可修改后重新提交。
已关闭：只读。
已作废：只读并标记风险。
```

### 11.5 下推、选单、转换

必须展示：

```text
可下推目标。
下推条件。
已下推数量。
未下推数量。
下游单据编号。
转换失败原因。
```

### 11.6 反写

反写结果必须可见：

```text
已入库数量。
未入库数量。
已收款金额。
未收款金额。
已结算金额。
未结算金额。
已生成凭证状态。
```

### 11.7 审批

审批视图必须展示：

```text
当前节点。
当前处理人。
历史意见。
附件。
风险字段。
金额。
预算。
同意、驳回、转交、加签。
```

## 12. 权限设计规范

PC 后台权限至少包括：

```text
菜单权限。
页面权限。
按钮权限。
字段权限。
数据权限。
组织权限。
导出权限。
打印权限。
审批权限。
API 权限。
```

### 12.1 无权限表现

可选策略：

```text
隐藏：用户完全不应知道该能力。
禁用：用户知道有该能力，但当前无权限或状态不允许。
提示：需要解释原因，例如无组织权限。
```

不能所有场景都简单隐藏。


权限表现的设计取决于用户是否需要理解“能力存在但自己不可用”。

```text
隐藏：适合用户不应该知道的能力，例如管理员配置入口。
禁用：适合能力存在但当前状态或权限不足，例如已审核单据不可编辑。
提示：适合用户需要知道原因，例如当前组织无导出权限。
```

权限设计也是协作设计。用户经常需要知道不能操作是权限问题、状态问题、组织范围问题，还是数据问题。
### 12.2 高风险权限

必须重点控制：

```text
删除。
作废。
反审核。
批量导出。
批量删除。
修改权限。
修改价格。
生成凭证。
付款。
退款。
```

## 13. 审计与日志规范

必须记录：

```text
谁操作。
什么时候。
操作对象。
操作动作。
操作前状态。
操作后状态。
失败原因。
IP / 设备 / 来源。
traceId。
```

页面应支持查看：

```text
操作日志。
审批日志。
字段变更记录。
导入导出记录。
打印记录。
接口调用记录。
```


审计不是事后补的日志，而是后台系统的信任基础。

架构师视角：

```text
凡是会改变业务事实、权限、资金、库存、流程状态的动作，都必须可追溯。
```

设计师视角：

```text
用户需要在出问题时回答：谁在什么时候做了什么，造成了什么结果。
```

因此审计入口不能藏得太深。高风险对象详情页应该能直接看到审批轨迹、操作日志、字段变更、导入导出记录、下游生成记录和反写记录。
## 14. 导入导出规范

### 14.1 导入

导入必须经过：

```text
下载模板。
上传文件。
字段映射。
数据校验。
确认导入。
结果反馈。
```

失败反馈必须包含：

```text
行号。
字段。
错误原因。
修复建议。
失败明细下载。
```

### 14.2 导出

导出必须明确：

```text
导出当前页。
导出选中项。
导出当前筛选结果。
导出全部。
```

高风险导出必须：

```text
权限校验。
二次确认。
日志记录。
脱敏。
异步任务。
```

## 15. 报表设计规范

报表必须回答：

```text
指标口径是什么？
统计范围是什么？
时间口径是什么？
组织口径是什么？
金额口径是什么？
数据来自哪些单据？
能否下钻到明细？
能否回到原始单据？
```

报表不是图表展示，而是可追溯的业务证据视图。


架构师视角：

```text
报表是业务事实的聚合视图，必须能回到事实来源。
```

设计师视角：

```text
用户看报表不是为了欣赏图形，而是为了发现问题、解释原因、做出决策。
```

报表必须有三层：指标层告诉用户结果是什么，分解层告诉用户结果由哪些维度构成，证据层告诉用户这些数字来自哪些明细和原始单据。
## 16. 工作台设计规范

工作台不是首页门户。

工作台必须围绕角色任务：

```text
我的待办。
我的异常。
我的超期。
关键指标。
快捷入口。
最近处理。
风险预警。
```

原则：

```text
任务优先。
异常优先。
指标能下钻。
入口服务高频任务。
不同角色看到不同工作台。
```


工作台的本质不是“导航集合”，而是“角色任务模型”。

架构师视角：

```text
工作台把待办、异常、指标、入口聚合成一个角色作业面。
```

设计师视角：

```text
用户打开工作台时，应该立刻知道今天要处理什么、什么最紧急、从哪里进入。
```

如果所有角色看到同一个首页，通常说明它不是工作台，只是菜单门户。
## 17. 弹窗、抽屉、详情页规范

### 17.1 弹窗

适合：

```text
确认。
简单编辑。
选择对象。
短表单。
```

不适合：

```text
复杂长表单。
多步骤流程。
高风险审批。
大量明细查看。
```

### 17.2 抽屉

适合：

```text
查看详情摘要。
快速编辑。
查看日志。
查看关联对象。
```

原则：

```text
不打断当前列表上下文。
复杂任务应跳详情页。
抽屉标题必须说明对象。
关闭前处理未保存变更。
```

## 18. 配色规范

PC 后台配色必须克制。

### 18.1 颜色角色

```text
中性色：背景、文字、边框、分割线。
品牌色：主按钮、选中态、链接、焦点。
成功色：已完成、启用、成功。
警告色：待处理、预警、临界风险。
危险色：失败、删除、作废、严重异常。
信息色：处理中、同步中。
图表色：分类和系列。
```

### 18.2 使用原则

```text
大面积用中性色。
品牌色只服务主操作和选中态。
状态色语义稳定。
风险色不能滥用。
图表色不能和状态色冲突。
状态不能只靠颜色表达。
```

### 18.3 表格配色

```text
表头轻微区分。
行分割线要弱。
hover 行要轻。
选中行要清楚。
异常行可以弱背景，但必须有文字原因。
金额不要一律红色。
```

## 19. 工程契约

PC 后台页面建议暴露：

```text
pageId。
pageType。
pattern。
objectType。
objectId/objectNo。
scopeType/scopeId。
state。
actionState。
permissionState。
errorCode。
```

示例：

```html
<div
  data-page-id="inventoryManage"
  data-page-type="list"
  data-pattern="tree-table"
  data-object-type="inventoryItem"
  data-scope-type="warehouse"
  data-scope-id="WH001"
  data-state="loaded"
>
```


工程契约不是给测试人员临时加的属性，而是页面设计的一部分。

架构师视角：

```text
页面契约把 UI、路由、权限、接口、日志、测试连接成同一个可治理对象。
```

设计师视角：

```text
当页面有稳定身份和状态时，团队才能跨设计、开发、测试、运维讨论同一个对象。
```

没有工程契约的后台页面会导致测试靠文本猜元素、埋点靠路径猜页面、日志无法定位业务对象、灰度发布无法按页面能力控制。
## 20. PC 后台测试规范

必须覆盖：

```text
页面身份。
入口参数。
默认筛选。
查询条件。
表格列。
排序分页。
范围树。
批量选择。
批量操作。
导入导出。
权限按钮。
字段权限。
状态流转。
审批。
下推。
反写。
报表钻取。
日志审计。
```

自动化测试必须避免：

```text
靠 sleep 等待。
靠颜色判断状态。
靠表格列位置猜字段。
靠按钮文本猜权限。
```

应等待：

```text
data-state。
data-action-state。
data-biz-status。
接口响应。
后端证据。
```

## 21. PC 后台页面评审清单

评审时逐项检查：

```text
1. 页面属于哪个业务域？
2. 页面操作哪个业务对象？
3. 页面选用了哪种经典结构？
4. 当前范围是否清楚？
5. 默认筛选是否安全？
6. 表格列是否服务业务判断？
7. 状态是否驱动按钮和字段？
8. 批量操作是否显示影响范围？
9. 导入导出是否有权限和审计？
10. 错误是否能定位到字段、行、对象？
11. 权限不可用是否有合理表现？
12. 审批和流程是否可追溯？
13. 报表是否能下钻到明细和原始对象？
14. 日志和审计是否完整？
15. 页面是否暴露测试契约？
```

## 22. PC 后台 A 级标准

达到 A 级必须满足：

```text
选对经典结构。
业务对象清楚。
页面身份清楚。
范围和筛选清楚。
默认查询安全。
表格列服务业务判断。
批量操作有范围和失败明细。
状态驱动按钮和字段。
权限和审计完整。
导入导出可追溯。
ERP 单据上下游清楚。
报表能下钻到明细和原始对象。
配色克制且语义稳定。
自动化测试能定位对象、范围、状态和操作结果。
```

## 23. 最终结论

PC 后端管理界面设计的核心不是页面好不好看，而是：

```text
业务对象是否清楚。
页面结构是否选对。
数据范围是否清楚。
状态和权限是否驱动操作。
批量和导出是否可控。
流程和审计是否可追溯。
测试是否能稳定验证。
```

如果一个 PC 后台页面只做到了“字段展示”和“按钮可点”，它还不是企业级页面。

企业级 PC 后台页面必须同时服务：

```text
业务效率。
风险控制。
团队协作。
长期维护。
自动化测试。
审计追溯。
```
