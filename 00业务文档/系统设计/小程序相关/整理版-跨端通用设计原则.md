# 整理版：跨端通用设计原则

## 1. 文档定位

本文整理同时适用于小程序和 PC 后端管理界面的通用设计原则。

不管页面运行在：

```text
PC 后台。
ERP 管理端。
SaaS 管理端。
业务中台。
微信小程序。
移动 H5。
Taro / uni-app。
```

只要它是业务系统页面，就必须遵守本文原则。

本文不讨论具体终端布局，例如左树右表、tabBar、分包、宽表格、底部操作栏。这些放到专用文档。

## 2. 通用原则总览

| 原则 | 说明 |
|---|---|
| 业务对象优先 | 页面必须让用户知道正在操作什么对象 |
| 页面身份明确 | 页面必须有稳定 pageId、pageType、bizType |
| 入口来源明确 | 页面必须知道用户从哪里进入 |
| 参数契约明确 | 页面参数必须可校验、可降级、可追踪 |
| 状态机驱动 | 页面状态决定内容、按钮、字段和测试 |
| 业务上下文明确 | 页面必须暴露业务编号、业务状态、组织、来源 |
| 操作反馈可观察 | 点击后必须有 loading、success、fail、pending |
| 错误可恢复 | 错误要具体到字段、行、权限、状态或规则 |
| 权限可解释 | 不可见、不可点、不可导出、不可审批要有治理 |
| 证据可追溯 | 关键动作必须能追到后端证据 |
| 自动化可测试 | 测试不能依赖固定 sleep 和视觉猜测 |
| 发布可兼容 | 新旧入口、字段、状态、接口必须兼容 |

## 3. 业务对象优先

任何业务页面都不是孤立 UI。

页面必须明确：

```text
这个页面操作什么对象？
对象类型是什么？
对象编号是什么？
对象状态是什么？
对象属于哪个组织、门店、会员、客户或订单？
对象和哪些上下游有关？
```

典型业务对象：

```text
订单。
会员。
商品。
门店。
优惠券。
库存。
支付单。
退款单。
审批单。
报销单。
应收单。
应付单。
```

如果页面没有业务对象，测试和运维就无法定位问题。

## 4. 页面身份

页面必须有稳定身份。

推荐字段：

```text
pageId：页面唯一标识。
pageType：列表、详情、表单、审批、结果、工作台。
bizType：业务对象类型。
module：业务模块。
version：页面契约版本。
```

页面身份不是给用户看的，而是给：

```text
路由治理。
埋点。
日志。
自动化测试。
问题定位。
灰度发布。
权限治理。
```

## 5. 页面入口

同一个页面可能从多个入口进入。

例如订单详情：

```text
从订单列表进入。
从支付结果进入。
从消息通知进入。
从扫码进入。
从审批待办进入。
从报表钻取进入。
```

入口影响：

```text
默认视图。
返回路径。
可见操作。
参数要求。
错误处理。
埋点归因。
测试用例。
```

因此页面必须定义入口契约：

```ts
type PageEntry =
  | 'fromList'
  | 'fromSearch'
  | 'fromMessage'
  | 'fromApproval'
  | 'fromReport'
  | 'fromScan'
  | 'directUrl';
```

## 6. 参数契约

页面参数必须被当作接口契约处理。

必须定义：

```text
参数名。
参数类型。
是否必填。
默认值。
来源入口。
非法值处理。
缺失值处理。
是否影响权限。
是否影响查询范围。
```

错误做法：

```text
页面直接读取 query.id，不校验。
参数缺失就白屏。
参数错误就请求接口失败。
```

正确做法：

```text
先解析入口。
再校验参数。
再解析业务上下文。
再决定进入 loaded、notFound、noPermission、invalidParams 状态。
```

## 7. 页面状态机

好的业务页面应该像一个小型状态机。

通用状态：

```text
initial。
loading。
loaded。
empty。
submitting。
success。
failed。
noPermission。
notFound。
offline。
expired。
```

状态决定：

```text
显示什么内容。
哪些按钮可见。
哪些按钮可点。
哪些字段可编辑。
是否允许离开。
自动化测试等待什么。
```

不要让状态散落在多个 boolean 中：

```text
isLoading。
isSubmit。
isError。
isEmpty。
isDisabled。
```

更好的方式是定义明确状态：

```ts
type PageState =
  | 'loading'
  | 'loaded'
  | 'empty'
  | 'submitting'
  | 'submitSuccess'
  | 'submitFailed'
  | 'noPermission'
  | 'notFound';
```

## 8. 业务上下文

页面必须暴露业务上下文。

推荐：

```text
bizType：业务类型。
bizNo：外部业务编号。
bizId：内部 ID，必要时使用。
bizStatus：业务状态。
orgId：组织。
storeId：门店。
memberNo：会员号。
orderNo：订单号。
traceId：链路 ID。
entry：入口来源。
```

原则：

```text
给用户看的优先是外部业务编号。
给系统内部关联的可以是内部 ID。
给测试和日志的必须稳定可读。
```

## 9. 操作反馈

所有动作都必须可观察。

动作至少有：

```text
idle。
confirming。
submitting。
success。
failed。
retrying。
```

点击后必须发生可观察变化：

```text
按钮 loading。
页面 actionState 变化。
状态标签变化。
结果提示。
错误提示。
后端证据更新。
```

自动化测试应等待状态，而不是固定 sleep。

## 10. 错误恢复

错误必须结构化。

```ts
type PageError = {
  code: string;
  scope: 'page' | 'field' | 'row' | 'action' | 'permission' | 'network';
  field?: string;
  rowIndex?: number;
  message: string;
  recoverable: boolean;
  retryAction?: string;
};
```

错误必须告诉用户：

```text
哪里错了。
为什么错。
如何修复。
是否能重试。
重试会不会重复提交。
```

## 11. 权限和审计

权限不是简单隐藏按钮。

跨端通用权限包括：

```text
页面访问权限。
数据查看权限。
按钮操作权限。
字段编辑权限。
导出权限。
审批权限。
删除/作废等高风险权限。
```

关键动作必须留痕：

```text
谁操作。
什么时候操作。
操作了什么对象。
操作前后状态。
失败原因。
traceId。
```

## 12. 测试契约

页面要天然可测试。

推荐暴露：

```html
<div
  data-page-id="orderDetail"
  data-page-type="detail"
  data-biz-type="order"
  data-biz-no="SO202607030001"
  data-biz-status="paid"
  data-state="loaded"
  data-entry="fromList"
>
```

测试应围绕：

```text
页面身份。
入口。
参数。
状态。
业务编号。
业务状态。
操作结果。
错误码。
后端证据。
```

## 13. 配色通用原则

小程序和 PC 后台都应遵守语义配色。

通用原则：

```text
颜色表达语义，不做装饰。
品牌色用于主动作和选中态。
成功、警告、危险、禁用语义稳定。
风险色不能滥用。
状态不能只靠颜色表达，必须有文字。
测试不能靠颜色判断业务状态。
```

## 14. 跨端通用评价标准

一个页面如果跨端达到 A 级，必须满足：

```text
业务对象清楚。
页面身份清楚。
入口和参数清楚。
状态机清楚。
业务上下文可读取。
操作反馈可观察。
错误可恢复。
权限可治理。
证据可追溯。
自动化可测试。
```
