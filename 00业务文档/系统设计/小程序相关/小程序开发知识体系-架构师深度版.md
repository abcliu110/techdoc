# 小程序开发知识体系：架构师深度版

> 本文面向需要设计、评审、开发、测试、维护微信小程序的技术负责人、架构师、前端负责人、测试负责人和高级开发者。
>
> 本文不是官方教程的复述，而是从架构、产品体验、工程质量、性能、安全、测试和团队协作角度，解释小程序为什么这样设计、应该如何建设、怎样判断一个小程序是否具备长期演进能力。

## 1. 小程序的本质

微信小程序不是“运行在微信里的网页”，也不是“缩水版 App”。它是一种运行在微信受控环境中的轻应用形态。

可以把小程序理解成：

```text
受控运行时
+ 声明式页面地图
+ 分离的逻辑层和渲染层
+ 微信托管的能力网关
+ 平台审核和分发机制
+ 受限制的网络、安全、权限和包体模型
```

这个定义比“用 WXML、WXSS、JS 写页面”更重要。因为所有小程序开发中的限制、最佳实践和坑，基本都来自这几个事实。

### 1.1 它首先是平台应用，不是自由应用

Web 应用的默认假设是“浏览器给你一个通用运行环境，业务自己做路由、网络、权限、发布”。小程序的默认假设不同：

```text
页面先声明
能力先授权
域名先配置
代码先审核
包体先约束
用户体验先受平台规则限制
```

因此，小程序架构的第一原则不是“我想怎么写”，而是：

```text
平台允许什么？
平台托管什么？
平台限制什么？
平台审核什么？
平台在不同端上的行为是否一致？
```

### 1.2 小程序适合解决什么问题

小程序适合：

- 高频入口但低安装意愿的服务，例如点餐、支付、会员、预约、查询。
- 强依赖微信生态能力的业务，例如微信登录、支付、订阅消息、扫码、分享。
- 需要快速触达用户的轻量业务。
- 用户使用路径短、任务明确、无需复杂后台运行能力的应用。
- 以门店、订单、会员、营销、服务预约为核心的业务闭环。

小程序不适合：

- 需要长时间后台运行的复杂任务。
- 高度自由的图形、复杂编辑器、重型办公场景。
- 对系统底层能力有大量依赖的 App。
- 需要绕过平台审核或频繁热更新核心业务逻辑的场景。
- 对网络、权限、进程、文件系统有完全控制诉求的场景。

架构师必须在立项阶段就判断：业务是否真的适合小程序。如果不适合，后面所有技术努力都是在平台边界里挣扎。

## 2. 小程序和 Web、App 的根本差异

### 2.1 运行环境差异

| 维度 | Web | 原生 App | 小程序 |
|---|---|---|---|
| 运行宿主 | 浏览器 | iOS/Android | 微信客户端 |
| 页面声明 | 自由路由 | App 自控 | 必须配置页面路径 |
| 渲染模型 | DOM | 原生 UI | 小程序渲染层 |
| 逻辑模型 | JS 主线程 | 原生线程 | 逻辑层 JS |
| 网络 | 浏览器安全模型 | App 自控 | 合法域名 + HTTPS |
| 发布 | 服务端部署 | 应用商店 | 上传 + 审核 + 发布 |
| 权限 | 浏览器授权 | 系统授权 | 微信授权 + 系统授权 + 平台隐私 |
| 调试 | 浏览器 DevTools | 原生工具 | 微信开发者工具 + 真机 |
| 自动化 | DOM 自动化成熟 | 原生控件自动化 | DevTools 自动化 + 小程序运行时契约 |

### 2.2 架构心智差异

Web 架构常常围绕：

```text
路由 -> 状态管理 -> 组件树 -> 网络层 -> 构建部署
```

小程序架构必须额外围绕：

```text
页面配置 -> 页面栈 -> tabBar -> 分包 -> 基础库兼容 -> 微信能力 -> 审核发布 -> 自动化契约
```

这意味着在小程序里，一个页面不是单纯的前端文件，而是一个平台可识别、可路由、可审核、可加载、可授权、可测试的运行单元。

## 3. 小程序的核心设计模型

### 3.1 配置先行

小程序的页面、窗口、tabBar、分包、权限、网络超时等能力，都需要通过配置声明。

这不是繁琐，而是平台治理需要：

- 微信需要提前知道页面地图，才能路由和审核。
- 微信需要知道 tabBar 页面，才能托管一级导航。
- 微信需要知道分包结构，才能控制下载和加载。
- 微信需要知道权限用途，才能做隐私合规。
- 微信需要知道网络域名，才能做安全边界。

因此 `app.json` 不是普通配置文件，而是小程序的“平台契约”。

架构上应把 `app.json` 看成：

```text
产品入口图
+ 页面地图
+ 导航结构
+ 能力声明
+ 包体边界
+ 审核边界
+ 运行策略
```

### 3.2 逻辑层和渲染层分离

小程序逻辑层和渲染层不是同一个自由共享内存的环境。数据从逻辑层到渲染层需要经过序列化、跨线程或跨进程传输、反序列化和渲染更新。

这解释了为什么：

- 不能像 Web 一样直接操作 DOM。
- 修改视图必须通过 `setData` 或框架抽象。
- `setData` 数据量越大越慢。
- 高频 `setData` 会造成卡顿。
- 页面隐藏后继续 `setData` 也会影响前台页面性能。

从架构角度看，`setData` 不是普通赋值，而是一次跨层通信。

```text
this.setData(...)
  -> 逻辑层计算
  -> 数据序列化
  -> 跨线程/进程通信
  -> 渲染层接收
  -> 虚拟树更新
  -> 真实视图更新
```

所以小程序性能治理的核心不是“少写代码”，而是“减少跨层数据通信、减少无意义渲染、缩小更新范围”。

### 3.3 平台能力通过 `wx.*` 网关访问

小程序访问登录、支付、扫码、定位、相机、文件、蓝牙等能力，都必须通过微信提供的 API。

这意味着：

- 能力调用受基础库版本限制。
- 能力调用受用户授权限制。
- 能力调用受平台审核和隐私规则限制。
- 能力调用在开发者工具和真机上可能有差异。
- 能力调用不能完全用普通函数 mock 代替。

架构上应把微信能力封装成“平台适配层”，而不是在页面里到处直接调用。

推荐分层：

```text
页面
  -> 业务服务
    -> 平台能力适配层
      -> wx.login / wx.requestPayment / wx.getLocation / wx.scanCode
```

这样做的价值：

- 统一兼容处理。
- 统一错误码。
- 统一日志和埋点。
- 便于测试 mock。
- 便于替换 Taro、uni-app 或原生实现差异。

## 4. 工程架构设计

### 4.1 一个可长期维护的小程序应具备的分层

推荐架构：

```text
app
  全局启动、登录态恢复、更新检查、全局错误入口

pages
  页面容器，只负责页面生命周期、路由参数、页面状态编排

components
  可复用 UI 和业务组件

services
  业务服务层，封装业务用例

api
  接口层，封装 HTTP 请求和接口声明

platform
  微信能力适配层，封装 wx.* API

store
  全局状态或领域状态

constants
  路由、枚举、业务状态、错误码

utils
  无业务副作用的工具函数

qa
  测试契约、页面状态、data-qa 常量
```

### 4.2 页面不应该承载所有业务逻辑

常见低质量写法：

```text
页面 onLoad
  -> 读参数
  -> 调 wx.login
  -> 拼 URL
  -> wx.request
  -> 判断 code
  -> setData
  -> 处理支付
  -> 写埋点
  -> 处理异常
```

这种页面会快速变成不可维护的“流程泥团”。

更好的结构：

```text
页面
  -> 读取参数
  -> 调用 use case / service
  -> 映射页面状态
  -> 绑定用户事件

service
  -> 编排业务流程
  -> 调用 api
  -> 调用 platform
  -> 输出业务结果

api
  -> 请求后端
  -> 统一错误处理
  -> 返回结构化结果
```

页面应该是“状态机的展示器”，而不是“所有逻辑的垃圾桶”。

### 4.3 好的小程序页面应像一个小型状态机

页面不能只靠几个布尔值拼状态：

```js
loading: false,
isError: false,
isEmpty: false,
isSubmitting: false
```

这种写法容易出现互相矛盾：

```text
loading = true
isError = true
isEmpty = true
```

推荐使用显式状态：

```text
idle
loading
ready
empty
error
submitting
success
expired
forbidden
```

页面状态机示例：

```text
idle
  -> loading
    -> ready
    -> empty
    -> error

ready
  -> submitting
    -> success
    -> error
```

这样带来的好处：

- 页面逻辑更清楚。
- 测试可以等待稳定状态。
- 错误恢复路径明确。
- UI 设计可以覆盖所有状态。
- 自动化不会依赖固定 sleep。

## 5. `app.json` 的架构意义

### 5.1 `pages` 是页面地图，不只是文件列表

`pages` 的顺序和内容影响：

- 哪些页面可被路由。
- 默认首页。
- 审核可见页面范围。
- 开发者工具页面生成。
- 页面路径合法性。

如果没有配置 `entryPagePath`，`pages` 第一项就是默认首页。这是很多项目启动页异常的根因。

### 5.2 `tabBar` 是产品信息架构

tabBar 不是 UI 装饰，而是产品的一级导航结构。

一个好的 tabBar 应该满足：

- 代表用户最高频任务。
- 数量控制在 2 到 5 个。
- 每个 tab 的心智互斥，不能职责重叠。
- tab 页可以独立作为一级入口存在。
- tab 页之间切换不依赖复杂上下文。

常见错误：

- 把低频功能放进 tabBar。
- tab 文案模糊，例如“服务”“功能”“更多”。
- tab 页面依赖上一个页面传参。
- tab 页里承载过重、过深的流程。

架构师评审 tabBar 时，应像评审产品导航一样，而不是只看配置是否能跑。

### 5.3 `window` 是全局体验基线

`window` 决定导航栏、背景色、下拉刷新、触底距离等基础体验。

如果不同页面各自随意覆盖，会导致：

- 导航体验不统一。
- 下拉刷新行为不一致。
- 自定义导航适配成本上升。
- 自动化截图对比困难。

建议：

- 全局定义统一视觉基线。
- 页面只覆盖确实不同的部分。
- 自定义导航栏需要统一组件和适配策略。

### 5.4 `networkTimeout` 是用户体验策略

默认 60 秒超时对用户体验通常太长。真实业务中，应根据接口类型设置合理超时：

| 类型 | 建议 |
|---|---|
| 普通查询 | 5 到 10 秒 |
| 提交类接口 | 10 到 20 秒 |
| 上传下载 | 按文件大小单独配置 |
| 支付状态查询 | 可短轮询，但要有总时长限制 |

超时不是技术细节，而是产品体验：用户到底等多久、看到什么提示、是否可以重试、重复提交怎么办。

## 6. 页面栈和路由设计

### 6.1 页面栈是用户路径的真实记录

小程序维护页面栈。页面跳转不是单纯改变 URL，而是在维护一组页面实例。

常见路由行为：

```text
navigateTo：普通页面入栈
navigateBack：页面出栈
redirectTo：替换当前页面
switchTab：切换 tabBar 页面并关闭非 tab 页面
reLaunch：清空页面栈并打开新页面
```

### 6.2 路由设计的核心问题

设计页面跳转时必须回答：

```text
这个页面是否是一级入口？
用户完成任务后应该回到哪里？
用户按返回键应该看到什么？
这个页面是否允许从分享直接进入？
这个页面是否允许从扫码直接进入？
这个页面是否依赖上一个页面的内存状态？
```

如果页面依赖“上一个页面刚好传过来的内存对象”，它就不是一个健壮页面。

### 6.3 页面参数应该小而稳定

页面 URL 参数只传启动页面所需的最小上下文：

```text
orderId
productId
storeId
scene
source
```

不要传：

- 完整订单对象。
- 完整商品对象。
- 用户敏感信息。
- 价格、权限、库存等权威业务值。
- 长 JSON 字符串。

原因：

- URL 长度和编码复杂。
- 分享和扫码入口难以维护。
- 敏感信息泄漏。
- 页面刷新后数据可能过期。
- 权威数据必须从服务端重新获取。

### 6.4 分享入口和扫码入口要按冷启动设计

很多项目只测试“从首页点进去”，不测试分享卡片、二维码、订阅消息、外部打开。

架构上必须要求：

```text
每个可分享页面都能独立冷启动
每个扫码页面都能独立冷启动
每个外部入口都能恢复登录态和业务上下文
```

这意味着页面不能依赖：

- 上一页内存变量。
- 全局临时对象。
- 仅在某个流程中才初始化的数据。
- 没有持久化的门店上下文。

## 7. 生命周期设计

### 7.1 生命周期不是背 API，而是分配职责

生命周期的核心不是记住哪个钩子先执行，而是明确什么逻辑放在哪里。

推荐职责：

| 生命周期 | 架构职责 |
|---|---|
| `App.onLaunch` | 全局初始化、登录态恢复、更新检查、日志初始化 |
| `App.onShow` | 处理场景值、前后台恢复、外部入口 |
| `App.onHide` | 记录离开、暂停全局任务 |
| `Page.onLoad` | 参数校验、页面一次性初始化 |
| `Page.onShow` | 每次展示时刷新必要数据 |
| `Page.onReady` | 依赖渲染完成的节点操作 |
| `Page.onHide` | 暂停页面任务 |
| `Page.onUnload` | 清理页面资源 |

### 7.2 `onLoad` 和 `onShow` 的边界

常见错误是把所有接口都放到 `onShow`。这样从详情页返回、切 tab、授权弹窗回来，都可能重复请求。

更好的策略：

```text
onLoad：
  校验参数
  初始化页面状态
  请求一次性基础数据

onShow：
  只刷新确实可能变化的数据
  例如购物车数量、订单状态、授权状态
```

判断一个请求应放哪里：

```text
只和页面参数有关，并且页面实例生命周期内不变 -> onLoad
每次回到页面都可能变化 -> onShow
依赖节点尺寸或渲染完成 -> onReady
用户动作触发 -> event handler
```

### 7.3 页面隐藏和卸载必须清理资源

小程序里常见内存和性能问题来自：

- 定时器不清理。
- 轮询不停止。
- Socket 不关闭。
- 事件监听不注销。
- 页面卸载后异步回调仍 `setData`。

标准做法：

```text
onHide:
  暂停定时器、轮询、动画、音视频

onUnload:
  清理定时器、监听器、Socket、Worker、请求回调标识
```

异步请求回调必须判断页面是否仍有效：

```text
页面已卸载 -> 不再 setData
请求已过期 -> 丢弃旧结果
```

## 8. 状态管理与数据流

### 8.1 小程序状态分四类

| 状态类型 | 示例 | 存放建议 |
|---|---|---|
| 页面渲染状态 | 列表、loading、错误提示 | 页面 `data` |
| 页面非渲染状态 | 请求锁、临时 ID、定时器句柄 | 页面实例字段 |
| 全局业务状态 | token、用户、门店 | store 或 globalData + 持久化 |
| 服务端权威状态 | 订单、库存、金额、权限 | 服务端，前端只展示 |

最常见架构错误是把所有状态都放进页面 `data`。

### 8.2 `data` 不是内存仓库

`data` 的本质是“需要同步给渲染层的数据”。把非渲染数据放入 `data` 会造成：

- 无意义跨层传输。
- 渲染开销增加。
- 敏感数据更容易暴露。
- 状态职责混乱。

推荐：

```js
Page({
  data: {
    pageState: 'loading',
    list: [],
    errorMessage: ''
  },

  requestSeq: 0,
  timer: null,
  currentOrderId: '',
})
```

### 8.3 服务端权威原则

前端可以缓存和展示，但不能成为权威来源。

必须以服务端为准的值：

- 商品价格。
- 库存。
- 优惠金额。
- 支付金额。
- 用户权限。
- 订单状态。
- 退款状态。
- 会员积分。
- 门店归属。

前端传给后端的金额、折扣、库存，只能作为展示或辅助信息，后端必须重新计算和校验。

### 8.4 Vue、React、Taro、uni-app 中的页面状态机落地

现代小程序开发通常不会直接写原生 `Page({ ... })`，而是使用 Vue、React、Taro、uni-app、Remax 等框架。状态机思想并不和这些框架冲突。相反，Vue 的组合式函数、React 的 Hook、Taro 的跨端组件模型、uni-app 的页面生命周期，都很适合把页面状态机封装成项目级架构能力。

要先明确一点：

```text
Vue / React / Taro / uni-app 本身不强制页面状态机。
但它们都可以很好地承载页面状态机。
```

也就是说，页面状态机不是必须来自框架官方，而是可以作为一层团队架构规范放在所有页面之上。

#### 8.4.1 所有页面都应该有状态机，但不要求状态完全一样

正确做法是：

```text
公共页面状态统一
+ 页面业务状态自定义
+ 测试暴露契约统一
```

不要把所有页面都强行塞进一个巨大状态枚举里。一个项目如果定义了一个包含 100 个状态的全局 `PageState`，通常很快会变成新的混乱来源，因为不同页面的业务流程完全不同。

更合理的方式是拆成两层。

第一层是所有页面共享的基础状态：

```text
idle           初始状态
loading        首次加载中
ready          数据已就绪，可展示或可操作
empty          无数据
error          通用错误
unauthorized   未登录或登录失效
forbidden      无权限
offline        网络不可用
```

第二层是页面自己的业务状态。

订单确认页可以扩展：

```text
submitting     正在提交订单
submitted      订单提交完成
paying         正在拉起支付或等待支付
paySuccess     支付成功
payFailed      支付失败
payPending     支付结果待确认
```

购物车页可以扩展：

```text
syncing            正在同步购物车
editing            编辑模式
checkoutDisabled   不满足结算条件
priceChanged       价格发生变化
stockChanged       库存发生变化
```

登录页可以扩展：

```text
requestingPhone    正在请求手机号授权
loggingIn          正在登录
bindingUser        正在绑定业务用户
loginSuccess       登录成功
loginFailed        登录失败
```

退款页可以扩展：

```text
confirming         等待用户确认退款
submittingRefund   正在提交退款
refundSuccess      退款成功
refundRejected     退款被拒绝
refundPending      退款处理中
```

最终原则是：

```text
统一架构，不统一死状态。
统一测试契约，不限制业务表达。
统一状态机写法，不强迫所有页面一样复杂。
```

#### 8.4.2 项目级推荐目录

在 Vue、React、Taro、uni-app 项目中，可以采用类似结构：

```text
src/
  machines/
    createPageMachine.ts
    basePageStates.ts
    basePageEvents.ts
    machineTypes.ts

  pages/
    order-confirm/
      machine.ts
      service.ts
      index.vue 或 index.tsx

    cart/
      machine.ts

      index.vue 或 index.tsx

  services/
    orderService.ts
    cartService.ts

  api/
    request.ts
    orderApi.ts

  qa/
    qaIds.ts
    pageStates.ts
```

职责划分：

| 文件 | 职责 |
|---|---|
| `createPageMachine.ts` | 提供轻量状态机运行器 |
| `basePageStates.ts` | 定义项目通用基础状态 |
| `machine.ts` | 定义某个页面自己的状态、事件、流转 |
| `service.ts` | 编排该页面业务动作 |

| `request.ts` | 统一请求、错误、登录失效和 trace |
| `qaIds.ts` | 统一关键元素测试标识 |

#### 8.4.3 轻量状态机运行器

普通小程序页面不需要一开始就引入复杂状态机库。可以先用一个轻量运行器。

```ts
type TransitionMap = Record<string, Record<string, string>>

export function createPageMachine(options: {
  initial: string
  transitions: TransitionMap
  onInvalidTransition?: (current: string, event: string) => void
}) {
  let state = options.initial

  return {
    getState() {
      return state
    },

    can(event: string) {
      return Boolean(options.transitions[state]?.[event])
    },

    dispatch(event: string) {
      const next = options.transitions[state]?.[event]

      if (!next) {
        options.onInvalidTransition?.(state, event)
        return state
      }

      state = next
      return state
    }
  }
}
```

这个运行器解决三个核心问题：

```text
当前是什么状态？
当前状态允许接收什么事件？
收到事件后应该进入哪个状态？
```

它不是为了炫技，而是为了禁止页面到处随意 `setData` 或随意改 `ref`，让状态变化有规则可查。

#### 8.4.4 页面状态机定义示例

订单确认页可以这样定义：

如果没有这张状态流转表，这些逻辑往往会散落在多个 `if`、多个回调、多个组件事件里。

#### 8.4.5 Vue 组合式函数落地

在 Vue 或 uni-app Vue 项目中，推荐把页面状态机放进组合式函数。

```ts
import { ref } from 'vue'
import { createPageMachine } from '@/machines/createPageMachine'
import { orderConfirmMachine } from './machine'
import { orderService } from './service'

export function useOrderConfirmPage() {
  const machine = createPageMachine({
    ...orderConfirmMachine,
    onInvalidTransition(current, event) {
      console.warn(`invalid transition: ${current} -> ${event}`)
    }
  })

  const pageState = ref(machine.getState())
  const order = ref<any>(null)
  const errorMessage = ref('')

  function dispatch(event: string) {
    pageState.value = machine.dispatch(event)
  }

  async function loadOrder(orderId: string) {
    dispatch('LOAD')
    errorMessage.value = ''

    try {
      const result = await orderService.getOrder(orderId)

      if (!result) {
        dispatch('LOAD_EMPTY')
        return
      }

      order.value = result
      dispatch('LOAD_SUCCESS')
    } catch (e: any) {
      errorMessage.value = e.message || '订单加载失败'

      if (e.code === 401) {
        dispatch('LOGIN_EXPIRED')
        return
      }

      if (e.code === 403) {
        dispatch('FORBIDDEN')
        return
      }

      dispatch('LOAD_FAIL')
    }
  }

  async function submitOrder() {
    if (!machine.can('SUBMIT')) {
      return
    }

    dispatch('SUBMIT')

    try {
      await orderService.submitOrder(order.value.id)
      dispatch('SUBMIT_SUCCESS')
    } catch (e: any) {
      errorMessage.value = e.message || '提交失败'
      dispatch(e.code === 401 ? 'LOGIN_EXPIRED' : 'SUBMIT_FAIL')
    }
  }

  return {
    pageState,
    order,
    errorMessage,
    loadOrder,
    submitOrder,
    dispatch
  }
}
```

页面模板只负责按状态渲染：

```vue
<template>
  <view data-qa="order.confirm.page" :data-state="pageState">
    <LoadingView v-if="pageState === 'loading'" />

    <EmptyView
      v-else-if="pageState === 'empty'"
      @retry="loadOrder(orderId)"
    />

    <ErrorView
      v-else-if="pageState === 'error'"
      :message="errorMessage"
      @retry="loadOrder(orderId)"
    />

    <LoginExpiredView
      v-else-if="pageState === 'unauthorized'"
      @login="loginAndReload"
    />

    <OrderConfirmView
      v-else-if="pageState === 'ready'"
      :order="order"
      @submit="submitOrder"
    />

    <SubmittingView v-else-if="pageState === 'submitting'" />
  </view>
</template>
```

这里的关键点不是语法，而是职责：

```text
组合式函数负责状态和业务动作。
模板负责展示状态。
service 负责业务调用。
api 负责请求细节。
```

#### 8.4.6 React Hook 落地

在 React 或 Taro React 项目中，可以把状态机封装成 Hook。

```tsx
import { useCallback, useRef, useState } from 'react'
import { createPageMachine } from '@/machines/createPageMachine'

export function usePageMachine(config: any) {
  const machineRef = useRef(
    createPageMachine({
      ...config,
      onInvalidTransition(current, event) {
        console.warn(`invalid transition: ${current} -> ${event}`)
      }
    })
  )

  const [pageState, setPageState] = useState(machineRef.current.getState())

  const dispatch = useCallback((event: string) => {
    setPageState(machineRef.current.dispatch(event))
  }, [])

  const can = useCallback((event: string) => {
    return machineRef.current.can(event)
  }, [])

  return {
    pageState,
    dispatch,
    can
  }
}
```

页面使用：

```tsx
function OrderConfirmPage() {
  const { pageState, dispatch, can } = usePageMachine(orderConfirmMachine)
  const [order, setOrder] = useState<any>(null)
  const [errorMessage, setErrorMessage] = useState('')

  async function loadOrder(orderId: string) {
    dispatch('LOAD')
    setErrorMessage('')

    try {
      const result = await orderService.getOrder(orderId)

      if (!result) {
        dispatch('LOAD_EMPTY')
        return
      }

      setOrder(result)
      dispatch('LOAD_SUCCESS')
    } catch (e: any) {
      setErrorMessage(e.message || '订单加载失败')
      dispatch(e.code === 401 ? 'LOGIN_EXPIRED' : 'LOAD_FAIL')
    }
  }

  async function submitOrder() {
    if (!can('SUBMIT')) {
      return
    }

    dispatch('SUBMIT')

    try {
      await orderService.submitOrder(order.id)
      dispatch('SUBMIT_SUCCESS')
    } catch (e: any) {
      setErrorMessage(e.message || '提交失败')
      dispatch('SUBMIT_FAIL')
    }
  }

  return (
    <View data-qa="order.confirm.page" data-state={pageState}>
      {pageState === 'loading' && <LoadingView />}
      {pageState === 'empty' && <EmptyView onRetry={loadOrder} />}
      {pageState === 'error' && (
        <ErrorView message={errorMessage} onRetry={loadOrder} />
      )}
      {pageState === 'ready' && (
        <OrderConfirmView order={order} onSubmit={submitOrder} />
      )}
      {pageState === 'submitting' && <SubmittingView />}
    </View>
  )
}
```

React 里特别要注意：状态机实例应保持稳定，不要每次 render 都重新创建，否则当前状态会被重置。所以上面使用 `useRef` 保存机器实例。

#### 8.4.7 什么情况下需要 XState

轻量状态机适合大多数页面。但如果页面进入以下复杂度，可以考虑引入成熟状态机库，例如 XState：

- 状态嵌套很深。
- 一个页面存在多个并行状态。
- 异步任务之间有复杂取消、重试和竞争关系。
- 支付、退款、核销等链路需要严格状态图。
- 需要状态图可视化。
- 需要更强的类型约束和测试能力。

示例：

```text
支付页
  页面状态：ready / submitting / paying / result
  支付状态：notStarted / invoking / pending / success / failed
  查询状态：idle / polling / timeout
  用户状态：loggedIn / expired
```

这种页面如果只靠简单字符串状态，可能会过度简化真实业务。成熟状态机库可以表达并行状态和嵌套状态。

但不建议全项目无差别引入重型状态机。更务实的策略是：

```text
普通页面：pageState 字符串
中等页面：State + Event + transitions + dispatch
复杂交易页面：考虑 XState
全项目：统一状态命名、测试暴露和错误分类
```

#### 8.4.8 状态机和全局 store 的关系

状态机不是替代 Pinia、Redux、Zustand、MobX 或 Vuex。它们解决的问题不同。

| 能力 | 适合解决的问题 |
|---|---|
| 页面状态机 | 当前页面处于什么阶段，允许什么事件，下一步到哪里 |
| 全局 store | 用户、token、门店、购物车数量等跨页面共享状态 |
| 请求缓存 | 接口数据缓存、失效、刷新 |
| service 层 | 业务流程编排 |

不要把页面流程塞进全局 store。全局 store 适合保存跨页面共享的业务事实，不适合保存某个页面临时处于 `submitting` 还是 `error`。

推荐边界：

```text
页面状态机：控制页面流程。
全局 store：保存跨页面共享事实。
service：执行业务动作。
api：负责通信。
```

#### 8.4.9 测试契约必须统一

即使每个页面状态不同，测试暴露方式必须统一。

推荐页面根节点统一暴露：

```text
data-qa="页面ID.page"
data-state="当前 pageState"
data-biz-id="当前业务对象 ID，可选"
```

示例：

```vue
<view
  data-qa="order.confirm.page"
  :data-state="pageState"
  :data-biz-id="order?.id || ''"
>
  ...
</view>
```

自动化测试就可以使用统一策略：

```text
进入页面
等待 data-qa="order.confirm.page"
等待 data-state="ready"
执行点击
等待 data-state="submitting"
等待 data-state="submitted" 或 "error"
查询后端业务结果
```

这就是状态机对“快速测试”的真正价值：测试不再猜页面什么时候加载完，而是读取页面主动暴露的状态。

#### 8.4.10 团队落地规范

项目级应该统一：

```text
pageState 字段名
基础状态语义
事件命名风格
dispatch 调用方式
非法流转日志
错误码到状态的映射
data-state 测试暴露方式
```

页面级可以自定义：

```text
具体业务状态
具体业务事件
具体状态流转表
具体 UI 表现
具体 service 编排
```

评审页面状态机时，直接问开发：

```text
这个页面有哪些状态？
每个状态显示什么 UI？
哪些事件会触发状态变化？
哪些状态允许点击提交？
接口失败进入哪个状态？
登录失效进入哪个状态？
自动化测试等待哪个状态？
非法状态流转是否有日志？
```

如果这些问题答不上来，说明页面还没有真正状态机化，只是把变量名换成了 `pageState`。

#### 8.4.11 测试框架应该知道固定页面的状态全集

如果页面被设计成状态机，那么对一个固定页面来说，它的状态集合应该是可枚举、可审查、可测试的。测试框架不应该只知道“当前页面有一个 `pageState`”，还应该知道：

```text
这个页面允许出现哪些状态？
初始状态是什么？
哪些状态是稳定状态？
哪些状态是中间状态？
哪些状态是错误状态？
哪些状态允许用户操作？
哪些状态是终态？
哪些事件会让状态发生变化？
哪些状态不应该在当前测试路径出现？
```

这意味着页面状态机不能只藏在页面实现里，还应该沉淀为测试框架可读取的“页面状态契约”。

##### 8.4.11.2 测试框架可以做哪些校验

当测试框架知道页面状态全集后，可以做很多传统 UI 自动化做不到的检查。

第一，校验当前状态是否合法：

```text
读取页面 data-state
-> 查询 order.confirm 的 states
-> 如果 data-state 不在 states 中，立即失败
```

这能发现拼写错误和未定义状态，例如：

```text
loadding
loadSuccess
submiting
```

第二，校验当前状态是否允许当前动作：

```text
当前状态 ready
-> 允许 submit
-> 可以点击提交

当前状态 submitting
-> allowActions 为空
-> 不允许再次点击提交
```

这能直接测试防重复提交是否成立。

第三，校验状态流转是否合法：

```text
ready -> submitting -> submitted
```

是合法流转。

```text
loading -> paySuccess
```

不是合法流转。即使页面最终看起来显示成功，也应该标记为状态流转异常。

第四，校验测试路径是否覆盖关键状态：

```text
订单确认页状态全集：
loading / ready / empty / error / unauthorized / submitting / submitted / paying / paySuccess / payFailed / payPending

当前自动化只覆盖：
loading / ready / submitting / submitted / paySuccess

未覆盖：
empty / error / unauthorized / payFailed / payPending
```

这样测试报告能直接给出状态覆盖率，而不是只说“跑了 10 条用例”。

第五，区分中间状态和稳定状态：

```text
loading、submitting、paying 是 transient 状态。
ready、empty、error、paySuccess 是 stable 状态。
```

测试等待时不应该长期停在 transient 状态。如果超过阈值还停在 `loading` 或 `submitting`，可以判断为卡死或接口未返回。

##### 8.4.11.3 自动化测试中的状态等待策略

测试框架应该避免这种写法：

```text
点击提交
等待 3 秒
检查页面文案
```

更合理的是：

```text
点击提交
等待页面状态从 ready 变成 submitting
再等待 submitting 结束
最终进入 submitted / error / unauthorized 中的一个稳定状态
根据最终状态继续断言
```

伪代码：

```ts
await page.waitForState('order.confirm', 'ready')
await page.tapQa('order.confirm.submit.button')

await page.waitForState('order.confirm', 'submitting')

const finalState = await page.waitForAnyStableState('order.confirm', {
  include: ['submitted', 'error', 'unauthorized'],
  timeout: 10000
})

expect(finalState).toBe('submitted')
```

这个写法比固定 sleep 更快、更稳定，也更容易定位失败。

##### 8.4.11.4 状态覆盖率应该进入测试报告

如果页面状态机是明确的，测试报告就可以输出状态覆盖率。

示例：

```json
{
  "pageId": "order.confirm",
  "totalStates": 13,
  "coveredStates": [
    "idle",
    "loading",
    "ready",
    "submitting",
    "submitted",
    "paying",
    "paySuccess"
  ],
  "missingStates": [
    "empty",
    "error",
    "unauthorized",
    "forbidden",
    "payFailed",
    "payPending"
  ],
  "coverage": "53.85%"
}
```

这比传统“页面自动化通过”更有价值。它能告诉团队：主路径通过了，但异常状态还没有覆盖。

##### 8.4.11.5 页面状态契约和开发规范的关系

一个页面如果宣称采用状态机，就必须至少交付三样东西：

```text
状态全集 states
状态流转 transitions
测试暴露 data-state
```

更完整的页面契约还应包含：

```text
pageId
qa 根节点
initialState
stableStates
transientStates
errorStates
finalStates
allowedActions
状态描述
关键业务 ID 暴露规则
```

这使得测试框架可以从“黑盒点页面”升级为“按页面行为契约验证页面”。

##### 8.4.11.6 固定页面的状态数量应可评审

对固定页面来说，状态不是越多越好，也不是越少越好。

状态太少的问题：

```text
loading 和 submitting 混在一起
error 不区分登录失效和业务失败
支付失败和支付待确认混在一起
测试无法判断真实业务阶段
```

状态太多的问题：

```text
每个接口都定义一个状态
每个按钮都定义一个状态
状态之间语义重叠
测试和开发都难以理解
```

合理状态数量取决于页面业务复杂度。

参考标准：

| 页面类型 | 常见状态数量 |
|---|---|
| 简单静态页 | 2 到 4 个 |
| 普通列表页 | 5 到 8 个 |
| 表单提交页 | 6 到 10 个 |
| 订单确认页 | 8 到 15 个 |
| 支付、退款、核销页 | 10 到 20 个 |
| 多步骤复杂流程页 | 需要拆页面或使用嵌套状态机 |

如果一个普通页面出现 20 个以上状态，要优先判断是不是页面职责过大，是否应该拆分页面、拆组件或拆业务流程。

##### 8.4.11.7 测试框架读取 contract 的方式

推荐把页面状态契约导出为普通 TypeScript/JSON 对象，使测试框架可以直接读取。

```text
src/pages/order-confirm/contract.ts
```

开发代码使用：

```ts
import { orderConfirmPageContract } from './contract'

const machine = createPageMachine({
  initial: orderConfirmPageContract.initialState,
  transitions: orderConfirmPageContract.transitions
})
```

测试代码使用：

```ts
import { orderConfirmPageContract } from '@/pages/order-confirm/contract'

await miniappPage.assertKnownState(orderConfirmPageContract)
await miniappPage.waitForState(orderConfirmPageContract, 'ready')
await miniappPage.assertAllowedAction(orderConfirmPageContract, 'submit')
```

这样开发和测试使用同一份契约，避免“开发认为有 8 个状态，测试脚本认为有 5 个状态”的不一致。

如果测试运行环境无法直接 import 前端源码，也可以在构建测试包时导出 contract JSON：

```text
dist-test/contracts/order-confirm.json
```

测试平台启动时加载：

```text
load contracts
-> connect DevTools
-> enter page
-> read data-state
-> validate state by contract
```

##### 8.4.11.8 最终结论

页面状态机不是只给开发看的代码结构，也应该成为测试框架理解页面的协议。

高质量页面应该满足：

```text
开发知道页面有哪些状态。
设计知道每个状态长什么样。
测试知道每个状态如何进入和断言。
自动化框架知道当前状态是否合法。
测试报告能展示状态覆盖率。
```

这时，小程序测试才真正从“模拟用户乱点”升级为“按页面行为模型验证系统”。

## 9. 网络与接口架构

### 9.1 请求层必须统一

小程序页面里不应该散落裸 `wx.request`。统一请求层至少负责：

- 域名和环境。
- token 注入。
- 超时。
- HTTP 状态码判断。
- 业务 code 判断。
- 登录失效处理。
- 错误提示映射。
- traceId。
- 请求日志。
- 自动化测试记录。
- 幂等 key。

标准数据流：

```text
页面事件
  -> service
    -> api method
      -> request wrapper
        -> wx.request
          -> normalize response
            -> service result
              -> page state
```

### 9.2 `success` 不等于成功

`wx.request` 的 `success` 表示收到 HTTP 响应，不代表业务成功。

必须判断三层：

```text
网络层：请求是否发出、是否超时、是否 interrupted
HTTP 层：statusCode 是否符合预期
业务层：业务 code、data、message 是否符合预期
```

错误处理也必须分层：

| 错误类型 | 处理 |
---|---|
| 断网 | 提示网络异常，可重试 |
| 超时 | 提示请求超时，可重试或稍后查看结果 |
| 401 | 清理登录态，进入重登 |
| 403 | 展示无权限 |
| 业务失败 | 展示业务提示 |
| 数据结构异常 | 记录错误，显示兜底 |

### 9.3 幂等是关键业务的底线

小程序天然容易出现重复操作：

- 用户连点。
- 弱网重试。
- 页面返回再提交。
- 支付回调延迟。
- 前端查询状态与后端回调不同步。

下列操作必须有幂等设计：

- 下单。
- 支付。
- 退款。
- 领券。
- 核销。
- 积分变更。
- 会员开卡。
- 购物车数量修改。

前端要做防重复提交，但真正的幂等必须在服务端完成。

### 9.4 traceId 是测试和排障的基础设施

每个关键动作最好形成链路：

```text
actionName
  -> requestId / traceId
    -> backend log
      -> businessId
        -> evidence
```

没有 traceId 的系统，一旦出现“页面显示成功但后台没数据”，只能靠猜。

## 10. 登录、身份和权限

### 10.1 微信登录不是业务登录

`wx.login` 只提供临时 code。业务系统必须用 code 到后端换取自己的登录态。

标准流程：

```text
wx.login
  -> code
    -> 后端 code2Session
      -> openid / unionid / session_key
        -> 后端创建或查找业务用户
          -> 返回业务 token
            -> 小程序持久化业务 token
```

小程序前端不应该接触：

- AppSecret。
- session_key。
- code2Session 的直接调用。

### 10.2 openid、unionid、业务 userId 的区别

| 标识 | 含义 |
|---|---|
| openid | 用户在某个小程序下的唯一标识 |
| unionid | 用户在同一微信开放平台账号下的统一标识 |
| userId | 业务系统自己的用户 ID |

业务系统不要把 openid 当作所有系统的最终用户主键。更合理的是：

```text
openid / unionid
  -> 绑定业务 userId
    -> userId 作为业务领域主键
```

### 10.3 登录态恢复策略

小程序冷启动时应：

```text
读取本地 token
  -> 校验是否存在
    -> 可选：静默校验 token
      -> 恢复用户和门店上下文
        -> 进入页面
```

不要让每个页面自己判断登录，否则会出现：

- 多个页面重复登录。
- 登录失效处理不一致。
- 用户被多次弹窗。
- 自动化测试不稳定。

### 10.4 权限必须前后端双重控制

前端权限控制只能决定“显示不显示按钮”，不能决定“能不能做业务”。

服务端必须校验：

- 用户身份。
- 商户归属。
- 门店归属。
- 角色权限。
- 数据权限。
- 操作状态。

## 11. 授权与隐私合规

### 11.1 授权是用户体验，不是弹窗技术

用户授权的关键不是“调 API”，而是：

```text
用户为什么要授权？
拒绝后还能不能继续？
什么时候请求授权最合理？
授权失败怎么恢复？
隐私指引是否一致？
```

不要在首页一进入就连续弹授权。授权应该尽量在用户触发相关能力时出现。

### 11.2 权限请求时机

| 能力 | 推荐时机 |
|---|---|
| 手机号 | 用户点击登录、绑定会员、下单联系时 |
| 位置 | 用户选择附近门店、配送地址、定位服务时 |
| 相机 | 用户点击扫码、拍照上传时 |
| 相册 | 用户点击选择图片时 |
| 麦克风 | 用户点击语音相关功能时 |

### 11.3 拒绝授权必须可恢复

授权拒绝不是异常终点，而是一个用户选择。

必须设计：

- 拒绝后的提示。
- 不授权是否能继续使用部分功能。
- 重新授权入口。
- 去设置页的路径。
- 测试用例覆盖拒绝路径。

## 12. 组件体系设计

### 12.1 组件不是为了复用而复用

组件的价值有四类：

```text
复用
隔离复杂度
缩小渲染范围
建立业务语义
```

如果一个组件只为了“看起来像架构”而存在，它会增加理解成本。

### 12.2 组件的输入输出要清晰

一个好组件应该像一个小型 API：

```text
输入：properties
内部状态：data
输出：events
错误：明确展示
空态：明确展示
禁用态：明确展示
加载态：明确展示
```

组件不应该偷偷修改父页面数据，也不应该依赖父页面的隐式全局变量。

### 12.3 业务组件要表达业务语言

低质量组件名：

```text
Box
Item
Panel
List
Popup
```

高质量组件名：

```text
ProductCard
StoreSelector
CouponPicker
OrderSummary
PaymentMethodList
RefundConfirmDialog
```

组件命名本质上是领域建模。好的命名能让代码结构反映业务结构。

## 13. UI 与体验设计原则

### 13.1 小程序体验应服务于短任务

多数小程序用户不是来“逛系统”的，而是来完成任务的：

- 点餐。
- 买单。
- 查订单。
- 领券。
- 预约。
- 领取会员权益。
- 扫码进入某个场景。

因此，小程序设计不应像大型后台系统那样层层导航，也不应像品牌官网那样堆大量展示内容。

### 13.2 首页不是万能入口

很多小程序把所有功能堆在首页，导致：

- 首屏复杂。
- 加载慢。
- 用户不知道下一步。
- 后续功能不断加入口，首页变成杂货铺。

优秀首页应该回答：

```text
用户是谁？
用户当前最可能要做什么？
当前场景下最重要的信息是什么？
下一步动作是什么？
异常状态是什么？
```

例如门店点餐首页，不应该先展示一堆营销 Banner，而应优先确认：

- 当前门店。
- 堂食/外带/配送。
- 是否营业。
- 是否已登录。
- 购物车状态。
- 下单入口。

### 13.3 状态设计比静态页面更重要

设计稿常常只画“有数据的成功态”，但真实系统里更多问题来自：

- 加载中。
- 空数据。
- 请求失败。
- 权限不足。
- 已过期。
- 已售罄。
- 已下架。
- 已关闭。
- 重复提交。
- 后端处理中。

架构师和设计师必须一起定义完整状态，而不是只交付漂亮静态图。

## 14. 性能设计

### 14.1 性能问题的根源

小程序性能问题常见根源：

- 主包过大。
- 首页接口过多。
- 首屏等待登录和多个接口串行。
- 图片过大。
- `setData` 数据量过大。
- 高频 `setData`。
- 长列表一次性渲染。
- 页面隐藏后仍更新。
- 组件拆分不合理。
- 分包设计不合理。

### 14.2 启动性能优化

启动优化的核心是让用户尽快看到有意义的内容。

手段：

- 控制主包大小。
- 非首屏业务进入分包。
- 首页接口并行化。
- 不阻塞首屏的接口延后。
- 使用骨架屏或轻量 loading。
- 静默恢复登录态。
- 缓存稳定配置。
- 图片资源压缩和懒加载。

反模式：

```text
进入首页
  -> 等登录接口
    -> 等用户接口
      -> 等门店接口
        -> 等营销接口
          -> 等商品接口
            -> 才显示页面
```

更好的方式：

```text
进入首页
  -> 立即显示基础框架
  -> 并行恢复登录态、门店、配置
  -> 优先展示核心可用内容
  -> 次要信息渐进加载
```

### 14.3 `setData` 优化

核心原则：

```text
只更新渲染相关数据
只更新变化的数据
降低更新频率
缩小更新范围
页面隐藏时停止无用更新
```

错误写法：

```js
this.setData(this.data)
```

错误原因：

- 把所有数据重新传输。
- 增加序列化和通信成本。
- 触发不必要渲染。
- 难以判断真实变化。

更好的写法：

```js
this.setData({
  'items[2].count': nextCount
})
```

### 14.4 长列表优化

长列表要考虑：

- 分页。
- 图片懒加载。
- 局部更新。
- 列表项组件化。
- 避免每次刷新替换整个数组。
- 避免列表项过重。
- 必要时使用虚拟列表或框架能力。

列表不是简单的 `map` 展示。列表是小程序性能问题最集中的区域。

## 15. 分包架构

### 15.1 分包不是补救措施，而是架构边界

很多项目等主包超限后才分包，这是错误的。分包应该在业务架构阶段设计。

分包适合按以下维度划分：

- 业务域：订单、会员、营销、售后。
- 使用频率：低频功能进分包。
- 团队边界：不同团队独立维护。
- 资源重量：图片、模型、复杂页面进分包。
- 入口独立性：可独立启动的场景做独立分包。

### 15.2 主包应该保持克制

主包只放：

- 默认启动页。
- tabBar 页面。
- 登录和请求基础设施。
- 全局样式和基础组件。
- 所有分包都必须依赖的公共代码。

不应放：

- 低频业务页面。
- 大型图片资源。
- 活动页。
- 一次性营销页面。
- 后台管理式功能。

### 15.3 分包预下载要有业务依据

预下载不是越多越好。它会消耗网络和资源。

适合预下载：

- 用户从首页高概率进入的下一步。
- 下单流程下一页。
- 详情页后高概率进入的确认页。

不适合预下载：

- 低概率活动页。
- 重资源但不确定访问的页面。
- 用户流量敏感场景。

## 16. 安全架构

### 16.1 小程序前端不可信

小程序运行在用户设备上。任何前端代码、请求参数、本地缓存都不能作为权威依据。

服务端必须重新校验：

- 身份。
- 权限。
- 金额。
- 库存。
- 优惠。
- 状态流转。
- 门店归属。
- 商户归属。
- 操作幂等。

### 16.2 敏感信息最小化

前端不应保存：

- AppSecret。
- session_key。
- 支付密钥。
- 服务端内部 token。
- 过多个人敏感信息。

前端日志不应打印：

- token。
- 手机号明文。
- 身份证号。
- 详细地址。
- 支付敏感信息。

### 16.3 业务安全比接口安全更重要

很多漏洞不是“接口被黑”，而是业务规则被绕过：

- 前端传 0 元金额。
- 重复提交领券。
- 修改门店 ID 查看别的门店订单。
- 修改订单 ID 操作别人的订单。
- 支付未完成就显示成功。
- 退款重复提交。

小程序前端必须把关键业务操作视为“请求服务端执行”，而不是“通知服务端我已经执行完”。

## 17. 可测试性架构

### 17.1 可测试性要在开发时设计

小程序测试困难通常不是测试工具不强，而是开发没有提供稳定测试契约。

可测试小程序必须具备：

```text
稳定元素定位
稳定页面状态
稳定路由入口
稳定测试数据
稳定接口追踪
稳定后端证据
稳定失败分类
```

### 17.2 `data-qa` 是测试契约，不是临时属性

关键元素应有稳定标识：

```text
home.search.input
home.search.submit
cart.submit.button
order.confirm.pay.button
refund.dialog.confirm
```

不要用文案做定位，因为文案会变。不要用层级做定位，因为结构会变。不要用坐标做定位，因为屏幕会变。

### 17.3 页面状态是自动化等待的基础

自动化测试不应写：

```text
点击按钮
等待 3 秒
断言页面
```

应该写：

```text
点击按钮
等待 pageState = success
查询后端业务结果
断言页面和后端一致
```

页面状态契约是测试稳定性的核心。

### 17.4 UI 测试必须和业务证据结合

只看页面显示成功是不够的。关键业务必须验证后端结果：

```text
页面显示支付成功
  -> 查询订单状态
  -> 查询支付单状态
  -> 查询回调记录
  -> 核对金额
```

这就是“证据驱动测试”。它比截图测试更可靠。

## 18. 发布与运维

### 18.1 小程序发布是平台流程

发布链路：

```text
开发版
  -> 体验版
    -> 提交审核
      -> 审核通过
        -> 全量发布或分阶段发布
```

架构上必须考虑：

- 版本号规则。
- 环境配置。
- 审核账号。
- 审核说明。
- 灰度策略。
- 回滚策略。
- 线上监控。

### 18.2 分阶段发布是风险控制工具

当小程序承载核心交易、支付、订单业务时，不应每次都直接全量发布。

适合灰度：

- 登录改造。
- 支付改造。
- 下单流程改造。
- 会员权益改造。
- 分包结构调整。
- 大范围 UI 重构。
- 网络请求封装调整。

灰度必须配合：

- 实时错误监控。
- 关键业务指标。
- 用户反馈。
- 回滚方案。

### 18.3 线上问题要能定位

线上问题定位依赖：

- 版本号。
- 用户 ID。
- 页面路径。
- 场景值。
- traceId。
- 请求日志。
- 后端业务日志。
- 设备和微信版本。
- 基础库版本。

如果没有这些信息，线上排障会退化成“复现玄学”。

## 19. 团队协作模型

### 19.1 小程序不是前端一个人的事

一个高质量小程序需要：

- 产品定义场景、入口、状态和异常。
- 设计定义完整状态和交互反馈。
- 前端实现页面、状态、路由和平台能力。
- 后端提供权威业务逻辑和证据查询。
- 测试定义用例、自动化契约和证据标准。
- 运维或平台负责人保障发布和监控。

### 19.2 需求评审必须问的问题

每个功能开始前必须问：

```text
入口在哪里？
是否可分享？
是否可扫码进入？
是否需要登录？
是否需要授权？
是否涉及隐私？
是否涉及支付？
是否涉及幂等？
是否有空态、错态、加载态？
是否需要分包？
是否支持自动化测试？
后端业务结果如何验证？
```

这些问题如果在开发后才问，成本会非常高。

### 19.3 设计评审必须看状态全集

设计稿至少应覆盖：

- 默认态。
- 加载态。
- 空态。
- 错误态。
- 禁用态。
- 提交中。
- 成功态。
- 权限不足。
- 授权拒绝。
- 网络失败。

如果设计稿只给成功态，开发会自行发挥，测试会无法判断，用户体验会不一致。

## 20. 常见反模式

### 20.1 页面巨大化

表现：

- 一个页面几千行。
- 所有接口和状态都在页面里。
- 多个弹窗、列表、表单混在一个文件。

后果：

- 难维护。
- 难测试。
- 难复用。
- 难排查。

修正：

- 抽业务 service。
- 抽稳定组件。
- 抽请求层。
- 明确页面状态机。

### 20.2 接口散落

表现：

- 页面里直接写 URL。
- 每个页面自己处理 token。
- 每个页面自己判断 code。

后果：

- 登录失效不一致。
- 错误提示不一致。
- trace 缺失。
- 自动化难留证。

修正：

- 建统一 request wrapper。
- 建 api 层。
- 统一错误码和异常处理。

### 20.3 只测成功路径

表现：

- 只点一遍主流程。
- 不测拒绝授权。
- 不测接口失败。
- 不测重复提交。
- 不查后端结果。

后果：

- 上线后问题集中爆发。
- 页面成功但业务失败。
- 用户卡死在异常状态。

修正：

- 用状态矩阵设计测试。
- 关键业务查后端证据。
- 自动化按页面状态等待。

### 20.4 过度全局状态

表现：

- 所有页面都读写 `globalData`。
- 页面之间靠全局临时对象传数据。
- 刷新或分享进入就丢状态。

后果：

- 冷启动失败。
- 分享入口失败。
- 页面互相污染。
- 测试不稳定。

修正：

- URL 只传最小 ID。
- 页面按 ID 重新拉权威数据。
- 全局状态只保存真正全局的信息。

## 21. 架构评审清单

### 21.1 页面与路由

```text
[ ] 页面已在主包或分包注册
[ ] 默认首页正确
[ ] tabBar 页面职责清晰
[ ] 普通页面和 tabBar 页面路由 API 使用正确
[ ] 分享、扫码、订阅消息入口可冷启动
[ ] 页面参数有校验和异常态
```

### 21.2 状态与体验

```text
[ ] loading、empty、error、success 状态完整
[ ] 提交中有防重复
[ ] 授权拒绝有兜底
[ ] 弱网和超时有提示
[ ] 后台返回后状态恢复正确
```

### 21.3 接口与安全

```text
[ ] 请求层统一封装
[ ] HTTP 和业务 code 都判断
[ ] 登录失效统一处理
[ ] traceId 可关联后端
[ ] 前端无敏感密钥
[ ] 服务端重新校验金额、权限、库存、状态
```

### 21.4 性能与包体

```text
[ ] 主包大小受控
[ ] 低频页面已分包
[ ] 首页接口不无意义串行
[ ] 没有大对象或高频 setData
[ ] 图片资源压缩和懒加载
[ ] 页面隐藏后停止无用更新
```

### 21.5 测试与发布

```text
[ ] 关键元素有 data-qa
[ ] 页面状态可被自动化等待
[ ] 关键动作可关联请求
[ ] 关键业务可查询后端证据
[ ] 测试包和正式包能力隔离
[ ] 发布前有门禁和回滚策略
```

## 22. 一个高质量小程序的最终形态

从架构师角度看，一个高质量小程序不是“页面漂亮、功能能跑”，而是：

```text
产品结构清晰
页面职责明确
路由行为可预期
状态模型完整
接口边界统一
登录授权安全
隐私合规可解释
性能可度量
分包有边界
测试可自动化
证据可复查
发布可灰度
问题可追踪
团队可协作
```

这套标准会让小程序不仅能上线，而且能长期迭代。

## 24. 使用本文的方法

如果你是开发负责人，用本文评审架构：

```text
先看页面地图
再看路由和生命周期
再看状态机
再看请求层
再看安全和权限
再看性能和包体
最后看测试契约
```

如果你是测试负责人，用本文判断可测性：

```text
页面能否稳定进入
元素能否稳定定位
状态能否稳定等待
接口能否追踪
业务结果能否查询
失败能否复盘
```

如果你是产品或设计负责人，用本文检查体验完整性：

```text
入口是否明确
任务是否短
状态是否完整
异常是否可恢复
授权是否合理
用户是否知道下一步
```

最终目标只有一个：小程序不是“能跑就行”，而是要在微信这个受控平台里，形成可维护、可测试、可演进、可运营的业务系统。
