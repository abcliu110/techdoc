# 工作流与审批模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

工作流与审批模型的本质，是企业决策过程的可执行证据链。它让“谁在什么规则下，看着什么内容，做出了什么决定”可以被执行、追溯和审计。

核心抽象：

```text
流程定义 -> 流程实例 -> 节点任务 -> 审批动作 -> 操作历史 -> 业务回调
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| wf_process_def 流程定义 | def_key, version, def_content(节点/连线/规则 JSON), status(DRAFT/PUBLISHED/DEPRECATED), published_at | 版本化模板；(def_key, version) 唯一，改版只影响新实例 |
| wf_process_instance 流程实例 | def_key, def_version(发起时固化), biz_type, biz_order_no, status(RUNNING/APPROVED/REJECTED/CANCELED/TERMINATED), variables(变量发起快照), started_by, started_at | 一次具体流转；永远跑在发起时的定义版本上 |
| wf_task 任务 | instance_id, node_key, assignee_id, candidate(候选解析结果快照), status(PENDING/CLAIMED/APPROVED/REJECTED/TRANSFERRED/TIMEOUT), action_at, comment | 人的待办与决定终态；candidate 快照解释"为什么是他" |
| wf_task_history 操作历史 | task_id, instance_id, operator_id, action(通过/驳回/转办/加签/撤回/催办/超时), comment, 附件, action_at | 只增不改的证据链流水，任务记终态历史记全过程 |
| wf_assignee_rule 审批人规则 | rule_type(角色/直属上级/分管领导/表单字段/会签组), rule_config, fallback_rule 兜底 | 晚绑定规则；解析为空走显式兜底，不让流程无声卡死 |
| wf_delegation 委托规则 | delegator_id, delegatee_id, scope 范围, valid_from, valid_to, status | 休假代理；任务保留原责任人+实际处理人双字段 |
| wf_biz_snapshot 送审内容快照 | instance_id, biz_order_no, content_hash, content_snapshot, created_at | 固化审批人看到的单据内容，通过时校验防篡改 |
| wf_instance_migration 实例迁移记录 | instance_id, from_version, to_version, approved_by, migrated_at, reason | 在途实例升版本的显式操作留痕，不允许悄悄改 |
| wf_cc_record 抄送记录 | instance_id, node_key, cc_user_id, read_at | 知会不阻塞，只产生已读记录不产生待办 |

### 2.2 实体关系

```text
wf_process_def (1) ──< wf_process_instance (N)   实例绑定 (def_key, def_version)
wf_process_instance (1) ──< wf_task (N)          每到一节点产生任务；会签每人一条
wf_task (1) ──< wf_task_history (N)              每个动作一条流水，只增不改
wf_process_instance (1) ──< wf_task_history (N)  历史同时挂实例便于整链回放
wf_process_instance (1) ── wf_biz_snapshot (1)   送审内容哈希快照
wf_process_instance (1) ──< wf_instance_migration (N)  显式迁移留痕
wf_process_instance (1) ──< wf_cc_record (N)     抄送已读记录
wf_delegation (1) ──< wf_task (N)                委托期任务派给受托人，记双人
外部引用：biz_type/biz_order_no → 各业务域单据（回调推进，引擎不直改业务表）；
         assignee_id/candidate → 组织权限域（晚绑定解析同一组织事实源）；
         附件 → 文件域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：(def_key, version) 定义唯一；(instance_id, node_key, assignee_id, 轮次) 任务唯一
版本固化：instance.def_version 发起时写死；variables 发起快照，
        改单据金额必须撤回重新发起，禁止中途重算路由
状态机约束：任务并发处理用 CAS（UPDATE ... WHERE status='PENDING'），
          或签一人处理后其余任务置 CANCELED；
          实例终态(APPROVED/REJECTED/CANCELED/TERMINATED)后禁止新任务
事务纪律：完成任务→计算下节点→生成新任务，事务内完成或走 Outbox，
        防"任务完成但下节点没生成"的卡死；配卡死检测（RUNNING 且无 PENDING）
留痕纪律：wf_task_history / wf_instance_migration 只增不改，人工修复也走留痕正门
```

## 3. 核心张力

最大张力是：

```text
流程定义变化 vs 在途实例稳定
```

企业审批规则会不断调整，但已经发起的审批必须按发起时的规则走完，否则审批依据会被后来规则污染。

另一个张力是：

```text
组织实时变化 vs 审批责任可追溯
```

审批人要按当前组织规则解析，但历史上为什么由某个人审批必须能解释。

## 4. 不变量

- 流程定义、流程实例、任务必须分离。
- 流程定义必须版本化。
- 实例必须固化发起时的定义版本。
- 流程变量必须使用发起时快照。
- 审批人规则晚绑定，但解析结果必须快照。
- 流程引擎不能直接修改业务单据。
- 所有审批动作只增不改，必须留痕。

## 5. 为什么这样设计

定义版本化是为了让新规则只影响新实例。在途实例按旧版本走完，避免中途改规则造成审批不公或审计失败。

任务独立建模，是为了支持待办、会签、或签、转办、加签、委托、超时等人机协作场景。

业务单据和流程状态分离，是为了保持业务域对单据的权威控制，流程只负责决策过程。

## 6. 失败模式

- 改了流程定义后，在途审批突然按新规则走。
- 审批人写死，组织变更后流程失效。
- 任务没有快照，事后无法解释为什么是这个人审批。
- 流程引擎直接 UPDATE 业务单据，业务状态和流程状态冲突。
- 任务完成后下个节点没生成，流程卡死。

## 7. 适用边界

适用：

- 采购审批、退款审批、调价审批、权限申请、费用报销、合同审批、人事审批。

可简化：

- 只有单一步骤、固定审批人的小功能可不用完整工作流引擎，但仍应记录审批人、意见、时间和结果。

## 8. 设计取舍

优先保证：

- 责任可追溯。
- 流程可审计。
- 在途实例稳定。
- 审批规则可演进。

牺牲：

- 流程配置复杂度。
- 待办查询成本。
- 实例迁移成本。

## 9. 同一模型的多业务形态建模

工作流模型要同时支撑多种审批形态，核心方法是：**统一定义/实例/任务三层骨架 + def_key 区分流程 + 差异下沉到流程定义和变量**。

### 9.1 统一骨架（所有业态不变的部分）

```text
流程定义（def_key + version）→ 流程实例（固化发起版本 + 变量快照 + biz_type/biz_order_no）→ 任务（候选解析快照）→ 操作历史 → 业务回调
不变量全集：三层分离、版本锁定、变量发起快照、审批人晚绑定+结果快照、引擎不直改业务表、动作只增不改
```

骨架决不为某个业态特化——任何审批的"按什么规则、谁在批、批到哪一步、为什么是他"都用同一套三层结构回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 采购审批 | 金额分档路由、加签财务 | 差异全在 def_content：条件分支用变量快照中的金额；biz_type=PURCHASE 回调采购域推进单据（标准形态，骨架原型） |
| 退款审批 | 时效敏感、小额自动通过 | 定义中首节点为自动节点（规则通过即批，产生系统任务留痕）；超时策略配置为自动升级而非仅提醒 |
| 调价审批 | 审批对象是"变更"而非单据 | 送审内容为价格变更前后对照快照（哈希防篡改）；通过回调价格域生成新版本价格，不直改价格表 |
| 合同用印 | 物理动作节点、证据附件强制 | 定义中加"用印执行"任务节点（非审批语义，完成需上传用印照片附件）；操作历史的附件字段承载证据 |
| 请假 OA | 按人事关系路由、代理高频 | 审批人规则用"直属上级/分管领导"晚绑定；委托规则表吸收休假代理，任务记原责任人+实际处理人 |
| 商家入驻审核 | 多角色并行审、驳回补材料循环 | 并行分支（资质审/风控审）+ 汇聚节点；驳回目标显式配置为"发起人"，重新提交产生新实例，历次实例全保留 |

### 9.3 建模决策规则

- **能用新 def_key + def_content 表达的，不改引擎表结构**：路由、会签、超时、加签差异全部是定义数据，三层表一套。
- **业务差异进流程变量，不进任务表字段**：金额、部门、风险分作为变量快照参与条件判断，任务表保持通用。
- **节点语义允许扩展类型（自动节点/执行节点）但不新增流转语义**：任何节点仍是"产生任务→完成→推进"，不发明骨架之外的推进方式。
- **拆独立域的信号**：当"流程"的本质是长周期业务编排（订单履约、生产工序）而非人的决策链，应由业务域状态机承载，不是审批流的一种定义。

```yaml
  - 组织实时变化 vs 审批责任可追溯

```
