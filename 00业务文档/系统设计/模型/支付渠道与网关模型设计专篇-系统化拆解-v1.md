# 支付渠道与网关模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

支付渠道与网关模型的本质，是内部业务订单世界和外部资金渠道世界之间的边界适配系统。它负责把内部支付意图转成渠道请求，并用回调、查询、对账确认真实资金状态。

核心抽象：

```text
业务订单 -> 支付单 -> 渠道交易 -> 回调/查询 -> 对账单 -> 退款单 -> 差错处理
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| pay_payment 支付单 | payment_no(唯一), biz_order_no, amount, currency, status(INIT/PAYING/SUCCESS/FAILED/CLOSED), expire_at, 已退金额 | 内部协调者：一单多次尝试/组合支付的收敛点，维护退款留量 |
| pay_channel_txn 渠道交易 | payment_id, channel_code, out_trade_no(唯一), channel_txn_no, request_amount, status(含 UNKNOWN), confirmed_by(回调/查询/对账), callback_at | 与外部每次真实交互的忠实记录，UNKNOWN 合法 |
| pay_refund 退款单 | refund_no, payment_id, channel_txn_id(原路), refund_amount, status, fail_reason, idempotent_key(唯一) | 独立异步链路，失败可人工处理，不是支付逆操作 |
| pay_channel_config 渠道配置 | channel_code, 商户号, 能力(支付/退款/分账), 费率, 限额, status, 证书有效期 | 渠道×商户号的能力与费率主数据 |
| pay_route_log 路由决策 | channel_txn_id, 路由规则, 命中商户号, 费率快照 | 固化"为什么走这个渠道"，成本核算依据 |
| pay_recon_bill 渠道对账单 | channel_code, 账单日期, 渠道流水号, 金额, 手续费, 原始文件引用 | 外部资金事实的终审底稿，解析入库只增不改 |
| pay_recon_diff 差错记录 | 差异类型(长款/短款/金额不符), channel_txn_no, 本地/渠道金额, 处理状态, 处理人 | 长短款的处理队列，不允许静默吞掉 |
| pay_sign_contract 签约协议 | 协议号(唯一), 用户标识, 签约渠道, 渠道回执, status, 失败重试策略 | 代扣/订阅的扣款授权，重试策略配置在协议层 |

### 2.2 实体关系

```text
业务订单 (1) ──< pay_payment (N)          一单可多次支付/组合支付
pay_payment (1) ──< pay_channel_txn (N)   重试、换渠道各一条交易
pay_payment (1) ──< pay_refund (N)        部分退款多次发起
pay_channel_txn (1) ──< pay_refund (N)    原路退回挂原渠道交易
pay_channel_config (1) ──< pay_channel_txn (N)  交易走某渠道商户号
pay_channel_txn (1) ── pay_route_log (1)  每笔交易一条路由决策
pay_recon_bill (1) ── pay_channel_txn (1) 按 channel_txn_no 逐笔匹配
pay_recon_bill / pay_channel_txn ──> pay_recon_diff  不平记录进差错队列
pay_sign_contract (1) ──< pay_payment (N) 代扣支付单挂协议号
外部引用：biz_order_no → 订单域；用户标识 → 会员域；
         对账结果 → 账务域凭证（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：payment_no、out_trade_no（渠道侧幂等契约）、
             refund idempotent_key、协议号 各自唯一
留量校验：退款原子 UPDATE 支付单已退金额
         WHERE 已退 + 本次 ≤ amount（防超退闸门）
恒等式：支付单 SUCCESS ⇔ Σ成功渠道交易金额 = amount（组合支付全部成功才推终态）；
       日终：本地成功交易 与 渠道账单逐笔平，手续费 = Σ费率快照核算值
状态机约束：渠道交易 CAS 推进 WHERE status IN ('REQUESTING','PENDING')，
           SUCCESS 终态不可改写；超时只能经查询推进，不得臆断；
           关单先关渠道再关本地
分片键：payment_no（支付主路径）；对账按 channel_code + 账单日期批处理
```

## 3. 核心张力

最大张力是：

```text
内部状态确定性 vs 外部渠道不可靠
```

业务系统希望立刻知道支付成功或失败；外部渠道可能超时、重复回调、乱序通知、隔天对账修正、退款失败。

## 4. 不变量

- 订单、支付单、渠道交易必须分离。
- UNKNOWN 是合法状态。
- 回调必须验签、核对金额、幂等处理。
- 状态只能向终态推进，不能随意回退。
- 渠道对账单是终审依据。
- 退款是独立链路，不是简单把支付改失败。
- 敏感支付要素不能随意落库。

## 5. 为什么这样设计

三层单据分离，是为了支持一单多次支付尝试、组合支付、渠道重试、渠道切换和掉单补偿。

UNKNOWN 状态是为了正确表达“请求发出去了，但结果未知”。如果把超时当失败，可能用户已付款但订单关闭；如果把超时当成功，可能没付款却发货。

对账终审是因为渠道是外部资金事实源，回调和查询都可能不完整，最终必须以渠道账单和银行流水校验。

## 6. 失败模式

- 订单直接挂渠道流水，多次支付和组合支付无法表达。
- 没有 UNKNOWN，超时被错误处理成成功或失败。
- 回调不验签、不核金额，存在伪造风险。
- 重复回调重复推进订单或重复记账。
- 退款和支付混在一张表，失败退款无法处理。
- 不做渠道对账，掉单和长短款长期隐藏。

## 7. 适用边界

适用：

- 微信、支付宝、银行卡、银联、国际卡、钱包、聚合支付、退款、分账。

可简化：

- 只接内部模拟支付的测试系统可简化渠道层，但真实资金系统必须保留支付单、渠道交易、对账和退款链路。

## 8. 设计取舍

优先保证：

- 资金状态可信。
- 外部失败可恢复。
- 重复通知安全。
- 对账可闭环。

牺牲：

- 支付流程复杂度。
- 状态数量。
- 补偿任务和对账任务成本。

## 9. 同一模型的多业务形态建模

支付网关要同时支撑多种业务形态，核心方法是：**统一三层单据骨架 + 渠道适配器插件化 + 支付方式差异下沉到渠道交易层**。

### 9.1 统一骨架（所有业态不变的部分）

```text
业务订单 → 支付单（pay_scene 标识形态）→ 渠道交易 → 回调/查询/对账三源 → 退款单 → 差错处理
不变量全集：三层分离、UNKNOWN 合法、回调验签幂等、状态只向终态推进、对账终审、退款独立链路
```

骨架决不为某个业态特化——任何业态的"这笔钱到底收到没有"都靠回调、查询、对账三源用同一套机制回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 线上聚合支付 | 多渠道路由、用户主动付款 | 标准形态（骨架原型）：渠道路由表按规则（费率/成功率/限额）选渠道，一支付单可挂多次渠道交易重试 |
| 线下扫码 POS | B 扫 C/C 扫 B、终端设备、当面性 | 渠道交易加终端号/门店/操作员字段；B 扫 C 走同步等待+轮询查询（回调可能不通）；撤单能力（当日 reverse）注册为渠道操作类型 |
| 代扣订阅扣款 | 无用户在场、先签约后扣款 | 新增签约协议表（协议号+状态+签约渠道回执）；扣款支付单挂协议号发起；失败重试策略配置在协议层，不污染支付单状态机 |
| 跨境收单 | 多币种、汇率、外卡 3DS、清算周期长 | 渠道交易加交易币种/结算币种/汇率快照三字段；3DS 验证作为渠道交易的中间状态（不新增终态）；对账周期按渠道配置 T+N |
| 对公转账 | 无回调、靠银行流水匹配、金额可能不符 | 收款走虚拟收款账号或附言匹配规则表；银行流水文件进对账管线，匹配成功才推进支付单；不符金额落差错处理 |
| 组合支付 | 一单多支付方式（余额+微信） | 骨架天然支持：一支付单拆多渠道交易，全部成功才推支付单终态；部分成功走自动退回补偿，退款按渠道原路拆分 |

### 9.3 建模决策规则

- **新渠道/新方式 = 新渠道适配器 + 渠道交易扩展字段**，不改支付单结构：支付单只关心"付没付成"，渠道差异全部止步于渠道交易层。
- **渠道特有动作注册为渠道操作类型**（撤单、预授权、签约），不在支付单状态机上加业态状态。
- **允许按业态裁剪三源但不允许放弃对账**：对公转账没有回调，查询和对账补位；任何业态最终都必须能被渠道账单终审。
- **拆独立域的信号**：当资金事实源不再是外部渠道（如内部虚拟币清算、账户间转账），那是账务域的事，不是支付网关的一种渠道。
