# 定时任务与批处理模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

定时任务与批处理模型的本质，是系统中可重复、可追踪、可补偿的离线执行机制。它让过期扫描、日切、对账、报表、补偿、重建索引等后台工作有定义、有实例、有进度、有失败恢复。

核心抽象：

```text
任务定义 -> 执行实例 -> 分片进度 -> 幂等结果 -> 补跑订正
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| job_def 任务定义 | job_key(唯一), cron_expr, timezone, timeout_sec, retry_policy, concurrency(FORBID/ALLOW/REPLACE), alarm_config, owner, status | "任务是什么"的唯一登记处；每个任务必有 owner 和告警 |
| job_instance 执行实例 | job_key, biz_date(业务日期), trigger_type(SCHEDULED/MANUAL/RETRY/BACKFILL), status(RUNNING/SUCCESS/FAILED/TIMEOUT/CANCELED), started_at/ended_at, stats, operator | 每次运行的档案；biz_date 来自参数支撑补跑 |
| job_shard_progress 分片进度 | instance_id, shard_key, status, checkpoint(断点), processed | 分片认领与断点续跑；失败只重跑失败分片 |
| job_dependency 任务依赖 | job_key, 上游 job_key, 依赖类型(同 biz_date SUCCESS) | 依赖显式声明成 DAG，不用时间差表达依赖 |
| job_data_ready 数据就绪标记 | 数据集标识, biz_date, 就绪时间, 来源 | 跨系统依赖的就绪信号，替代"猜他跑完了" |
| job_delay_task 延迟任务 | biz_type, biz_id, execute_at, idempotent_key(唯一), status | 一次性未来动作（超时关单）；DB 兜底可审计 |

### 2.2 实体关系

```text
job_def (1) ──< job_instance (N)          一个定义多次运行（含补跑）
job_instance (1) ──< job_shard_progress (N)  实例按分片推进，CAS 认领
job_def (1) ──< job_dependency (N)        依赖声明挂定义，运行时查上游实例状态
job_data_ready (1) ──< job_instance (N)   下游实例启动前查就绪标记
外部引用：biz_type/biz_id → 各业务域单据（延迟任务执行时 CAS 校验业务状态）；
         operator → 权限域用户（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：job_key 唯一；(instance_id, shard_key) 分片唯一；
             延迟任务 idempotent_key 唯一；产生流水的任务结果带幂等键落唯一索引
biz_date 参数化：实例的 biz_date 来自调度参数，任务代码禁止内部取 now()，
               这是补跑（BACKFILL 按区间生成实例）的前提
状态机约束：实例状态只进不退；FORBID 并发靠实例级互斥（分布式锁+状态检查）；
           状态迁移类任务用 CAS 条件更新保证重跑无害
断点/续跑落点：checkpoint 每批持久化，游标推进（WHERE id > checkpoint）
             替代深分页；每批一个小事务
订正边界：补跑触碰已出具快照必须走订正流程（新版本+差异说明），不静默覆盖；
         手工触发 operator 必填
分片键：shard_key（ID 区间/哈希桶/租户/仓库，配置放 job_def）
```

## 3. 核心张力

最大张力是：

```text
批量吞吐 vs 正确性和可续跑
```

批任务要处理大量数据，但不能漏跑、不能重复造成双份结果、不能失败后从头重做、不能拖垮在线系统。

## 4. 不变量

- 每次执行必须有实例记录。
- biz_date 必须来自参数，不能在任务内部取 now。
- 任务必须幂等，重跑无害。
- 大数据量必须分片和记录 checkpoint。
- 任务依赖必须显式声明。
- 补跑必须走正门。
- 手工触发必须留 operator。

## 5. 为什么这样设计

定义、实例、分片分离，是为了区分“任务是什么”“这次跑了什么”“每个分片跑到哪里”。biz_date 参数化，是为了支持补跑历史日期。幂等和 checkpoint 是为了让失败恢复成为常规能力。

## 6. 失败模式

- 任务内部用 now，当补跑三天前数据时仍处理今天。
- 调度重复触发，生成双份账单或双份流水。
- 大表深分页扫描，越跑越慢。
- 上游失败，下游按时间差照常运行，报表错。
- 任务漏跑没有告警。
- 手工补跑覆盖已出具快照。

## 7. 适用边界

适用：

- 日切、对账、券过期、积分过期、报表跑批、补偿重试、索引重建、账单生成。

可简化：

- 小任务可以不分片，但仍应有实例记录、biz_date、幂等和告警。

## 8. 设计取舍

优先保证：

- 可重跑。
- 可续跑。
- 可审计。
- 不漏跑。

牺牲：

- 实现简单性。
- 任务表和进度表维护成本。
- 调度治理复杂度。

## 9. 同一模型的多业务形态建模

批处理模型要同时支撑多种任务形态，核心方法是：**统一三层骨架 + 任务类型标识 + 差异下沉到定义参数与分片策略**。

### 9.1 统一骨架（所有形态不变的部分）

```text
job_def（job_type 标识形态）→ job_instance（biz_date 参数化）→ job_shard_progress（checkpoint）
不变量全集：执行必有实例、biz_date 来自参数、幂等重跑、分片续跑、依赖显式、手工触发留 operator
```

骨架决不为某类任务特化——任何形态的"这次跑的哪个业务日、跑到哪了、重跑安不安全"都用同一套表回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 日切对账批 | 强依赖上游文件、日切时点严格 | job_def 声明依赖（上游对账文件就绪任务）；biz_date=T-1 由调度传参；实例失败阻断下游而非跳过（标准形态，骨架原型） |
| 过期扫描批（券/积分） | 数据量大、可高频跑 | 按 ID 段分片进 job_shard_progress；游标推进不用深分页；幂等靠"状态机条件更新"（只把 active 改 expired），重跑天然无害 |
| 报表跑批 | 依赖链长、结果要固化 | job_def 依赖 DAG 显式声明；产出写快照表带 biz_date 版本；补跑生成新版本不覆盖已出具快照 |
| 延迟任务（超时关单） | 单条触发、非批量扫描 | 定义层复用 job_def，触发方式=延迟队列；每条延迟消息对应一次幂等执行（按订单号+动作幂等键）；兜底扫描批作为对账通道 |
| 补数回刷 | 历史区间、人工发起 | 走正门：按 biz_date 区间循环生成标准 job_instance，每天一实例；operator 必填；受"已出具快照不覆盖"规则约束 |
| 大促临时任务 | 一次性、临时资源 | 仍注册 job_def（带 expire_at 失效期）不写脚本裸跑；实例和分片记录照常留档，事后可审计可复盘 |

### 9.3 建模决策规则

- **能用 job_type + 定义参数解决的，不建新执行框架**：差异只在触发方式、分片键、依赖时，实例/进度/告警一套。
- **分片策略按任务垂直配置**（ID 段 / 哈希桶 / 文件块），配置放 job_def，不在执行代码写死。
- **执行流程允许按形态裁剪但不新增语义**：延迟任务跳过分片层，但不得跳过实例记录和幂等键。
- **拆独立域的信号**：当任务是长时流程编排（多步骤人机协作、可回滚 Saga），那是工作流域，不是批处理的一种类型。
