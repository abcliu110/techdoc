# 物流与履约模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

物流与履约模型的本质，是订单承诺到物理交付的执行和证据系统。它把订单中的“应交付”转化为履约单、出库单、包裹、运单、轨迹、签收和异常处理。

核心抽象：

```text
订单行 -> 履约单 -> 出库单 -> 包裹 -> 运单 -> 物流轨迹 -> 签收/异常 -> 逆向物流
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| ful_order 履约单 | 履约单号(唯一), order_id, fulfill_type, 收货地址快照, 承诺送达时间(下单固化), status | 订单的履约视角副本，承诺是数据不是文案 |
| ful_outbound 出库单 | 出库单号, fulfillment_id, warehouse_id, status(拣货/复核/打包), 计划/实际出库时间 | 指挥仓内作业；计划 vs 实际定位延误责任段 |
| ful_parcel 包裹 | 包裹号(唯一), fulfillment_id, 重量/体积, status, 运费分摊明细 | 物理包裹；合单运费按规则分摊到订单 |
| ful_parcel_item 包裹明细 | parcel_id, order_line_id, qty | 订单行 N:M 包裹的行级映射，履约域事实中枢 |
| tms_waybill 运单 | (carrier_code, waybill_no)(唯一), parcel_id, status(揽收/运输/签收/拒收/丢件), est_fee 预估运费, billed_fee 账单运费 | 承运商合同；预估与实付两数对照防资损 |
| tms_track 物流轨迹 | waybill_id, 节点时间, 地点, 动作, 推送原文, 去重键(唯一) | 外部事件只增不改，状态从轨迹推导 |
| carrier_price_card 承运商报价卡 | carrier_code, 线路, 重量段, 计费规则(首重续重/抛比), 生效区间 | 运费预估与月度对账的规则依据，带版本 |
| ful_exception_ticket 异常工单 | 关联运单/包裹/订单行, 异常类型(丢件/破损/拒收/错发), 责任初判(仓/承运商/商家), 处理动作, 赔付金额, status | 物理失败的统一差错户，双线独立结案 |
| ful_reverse 逆向物流单 | 原 order_line_id, 退货物流单号, 入库质检结果(回可售/残次), 退款时点策略, status | 退货物理链路，无头件进待归属池 |

### 2.2 实体关系

```text
订单 order (1) ──< ful_order (N)          拆单履约一对多
ful_order (1) ──< ful_outbound (N)        履约单驱动仓内作业
ful_order (1) ──< ful_parcel (N)          一次履约多包裹
ful_parcel (1) ──< ful_parcel_item (N)    订单行 N:M 包裹（经明细行）
ful_parcel (1) ──< tms_waybill (N)        跨境/补发一包裹多运单
tms_waybill (1) ──< tms_track (N)         轨迹只增
carrier_price_card (1) ──< tms_waybill (N) 运单按报价卡固化 est_fee
tms_waybill (1) ──< ful_exception_ticket (N) 异常挂运单，补发生成新履约单回指工单
订单行 (1) ──< ful_reverse (N)            逆向关联原订单行
外部引用：order_id/order_line_id → 订单域；warehouse_id → 库存/仓储域；
         carrier_code → 承运商主数据；退款触发 → 资金域
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：履约单号、包裹号 唯一；(carrier_code, waybill_no) 唯一；
             电子面单取号幂等（号段预取+废弃回收）；
             轨迹 (waybill_no, 节点时间, 动作) 唯一去重
留量校验：拆包原子校验 Σful_parcel_item.qty ≤ 订单行 qty - shipped_qty（防超发）；
         逆向入库 qty ≤ 原订单行已签收 qty
恒等式：订单"部分已发货"等展示状态 = 包裹状态聚合计算，不手工维护；
       运费三数分离对照：用户支付运费 / est_fee / billed_fee 月度逐单对账
状态机约束：运单状态从轨迹按业务时间推导，只进不退，容忍乱序推送；
           承诺送达时间下单固化，承诺引擎升级不回写历史；
           补发是新履约单引用原单+工单，不改原运单
分片键：order_id（用户"我的包裹"主路径）；轨迹按 waybill_no 路由
```

## 3. 核心张力

最大张力是：

```text
交易承诺确定性 vs 物理执行不确定性
```

订单承诺用户会收到货，但仓库、承运商、天气、地址、拒收、丢件、破损都会让执行过程不确定。

## 4. 不变量

- 订单与履约必须解耦。
- 一单可多包，一包可包含多订单行。
- 拆单和合单必须行级映射。
- 履约承诺必须下单时固化。
- 物流轨迹只增不改，状态只进不退。
- 异常件必须工单化。
- 逆向物流必须关联原订单行。

## 5. 为什么这样设计

订单表达交易合同，履约表达物理执行。两者解耦后，订单不用承载多包裹、多仓、改址、补发、拒收、丢件这些物理复杂度。

包裹和订单行 N:M 映射，是为了支持拆单、合单、部分发货和客服追踪。

轨迹只增不改，是因为承运商推送可能乱序、重复、延迟，原始事件必须保留。

## 6. 失败模式

- 订单表只存一个运单号，无法多包裹。
- 包裹不关联订单行，客服无法知道哪件商品在哪个包裹。
- 承诺送达时间不固化，延误考核无依据。
- 轨迹覆盖更新，历史节点丢失。
- 丢件、拒收、破损没有工单，责任和赔付无法闭环。

## 7. 适用边界

适用：

- 电商、零售、仓配、同城配送、B2B 发货、逆向退货。

可简化：

- 虚拟商品、无需物流的服务订单可省略包裹和运单，但仍要有交付状态和交付凭证。

## 8. 设计取舍

优先保证：

- 物理交付可追踪。
- 异常可闭环。
- 运费可对账。
- 订单行级履约清晰。

牺牲：

- 模型复杂度。
- 外部接口处理复杂度。
- 轨迹存储量。

## 9. 同一模型的多业务形态建模

履约模型要同时支撑多种履约形态，核心方法是：**统一履约单骨架 + 履约类型标识 + 承运与交付环节可替换**。

### 9.1 统一骨架（所有业态不变的部分）

```text
订单行 → 履约单（fulfill_type 标识形态，固化承诺）→ 包裹/交付载体（与订单行 N:M）→ 运单/执行记录 → 轨迹（只增不改）→ 签收/核销 → 异常工单 → 逆向
不变量全集：订单履约解耦、行级映射防超发、承诺下单固化、轨迹只增状态只进、异常工单化
```

骨架决不为某个形态特化——任何形态的"这行商品交付到哪一步、承诺兑现没有、出了问题谁负责"都用同一套履约单 + 行级映射 + 轨迹回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 快递电商件 | 仓发、承运商轨迹推送 | fulfill_type=EXPRESS，出库单+电子面单+承运商轨迹（标准形态，骨架原型） |
| 同城即时配 | 骑手运力、分钟级时效、轨迹是坐标流 | 运单对接运力平台，状态机加"骑手接单/到店/取货"节点（裁剪自骨架节点集）；承诺粒度从"日"改"分钟"，超时赔付规则挂承诺快照 |
| 门店自提 | 无运输段、核销制交付 | 包裹=门店备货记录，运单省略；签收替换为提货核销（提货码+核销人+时间），超期未提触发异常工单转退款 |
| 大件安装配送 | 送装一体、预约改约、两段轨迹 | 履约单拆干线段+送装段两张子单（行级引用同一订单行）；预约记录独立表（预约/改约留痕）；签收含安装完成凭证照片附件 |
| 跨境物流 | 清关节点、多段承运、头程尾程 | 一个包裹多张运单按段串联（段序号+段类型：头程/清关/尾程）；清关状态作为轨迹事件接入，单证（报关单号）挂包裹扩展表 |
| 虚拟履约 | 无实物、发放即交付 | 包裹/运单退化省略，履约单直连发放记录（卡密/券 ID/账号开通），交付凭证=发放流水；异常工单承载发放失败重试 |

### 9.3 建模决策规则

- **能用 fulfill_type + 环节裁剪解决的，不建第二套履约链**：差异在"有没有运输段、交付动作是什么"时，替换/省略环节，履约单和行级映射一套。
- **交付载体可替换（包裹/备货记录/发放流水），订单行 N:M 映射不变**：客服"这行东西在哪"永远查同一张映射表。
- **状态机按形态裁剪节点但不新增终态语义**：自提跳过运输中，但终态仍是签收（核销）/异常，不发明新终态。
- **拆独立域的信号**：当"履约"是长期持续服务（订阅内容按期解锁、SaaS 开通计费），交付是权益状态而非一次动作，应归权益/订阅域，不是履约单的一种类型。
