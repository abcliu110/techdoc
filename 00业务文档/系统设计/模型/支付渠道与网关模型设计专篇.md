# 支付渠道与网关模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦支付网关建模。账务篇管"内部的账"，本篇管"与外部资金世界的边界"：微信、支付宝、银行、国际卡组织。边界的本质难题是**你无法控制对方**——渠道会超时、会掉单、会退款失败、会隔天冲正，模型必须假设外部永远不可靠。

## 0. 一句话总纲

> 支付网关模型的全部使命：在"内部订单世界"和"外部渠道世界"之间维护一张永远可对平的映射，任何一笔钱的状态都以渠道对账单为终审、以查询确认为准绳、以幂等为底线。

## 1. 三层单据：订单、支付单、渠道单

```text
业务订单（order）        —— 业务域：买了什么
    ↓ 1:N（一单可多次支付尝试、可组合支付）
支付单（payment）        —— 支付域：这次收多少钱、什么方式
    ↓ 1:N（一次支付可能多次请求渠道：重试、换渠道）
渠道交易（channel_txn）  —— 渠道域：与外部的每一次真实交互
```

**为什么三层**：用户点三次支付按钮、第一次微信失败换支付宝、组合支付（余额 + 微信）——如果订单直接挂渠道流水，这些场景全部变成补丁。支付单是内部协调者，渠道交易是外部事实的忠实记录。

```sql
CREATE TABLE pay_payment (
  id            BIGINT PRIMARY KEY,
  payment_no    VARCHAR(32) NOT NULL,
  biz_order_no  VARCHAR(32) NOT NULL,
  amount        DECIMAL(18,4) NOT NULL,
  currency      CHAR(3) NOT NULL DEFAULT 'CNY',
  status        VARCHAR(16) NOT NULL,   -- INIT/PAYING/SUCCESS/FAILED/CLOSED
  expire_at     DATETIME NOT NULL,      -- 支付有效期
  UNIQUE KEY uk_no (payment_no)
);

CREATE TABLE pay_channel_txn (
  id            BIGINT PRIMARY KEY,
  payment_id    BIGINT NOT NULL,
  channel_code  VARCHAR(32) NOT NULL,   -- WXPAY/ALIPAY/UNIONPAY/...
  out_trade_no  VARCHAR(64) NOT NULL,   -- 我方送给渠道的单号（渠道侧幂等键）
  channel_txn_no VARCHAR(64),           -- 渠道返回的流水号（对账主键）
  request_amount DECIMAL(18,4) NOT NULL,
  status        VARCHAR(16) NOT NULL,   -- REQUESTING/PENDING/SUCCESS/FAILED/UNKNOWN
  callback_at   DATETIME,
  confirmed_by  VARCHAR(16),            -- CALLBACK回调/QUERY主动查询/RECON对账
  UNIQUE KEY uk_out_no (out_trade_no),
  KEY idx_payment (payment_id)
);
```

要点：

- `out_trade_no` 每次渠道请求**新生成**（重试也换新号，或按渠道幂等规则复用），与渠道的幂等契约要逐渠道确认——各渠道对"同单号重复请求"的行为不同。
- **UNKNOWN 是合法状态**：请求超时既不能当成功也不能当失败，只能查询确认。没有 UNKNOWN 状态的支付系统必然出现"用户付了钱订单没变"或"没付钱发了货"。

## 2. 状态推进的三个来源与优先级

支付结果有三个来源，可信度递增：

| 来源 | 时效 | 可信度 | 纪律 |
|---|---|---|---|
| 异步回调（notify） | 秒级 | 中（可伪造、可乱序、可重复） | 验签 + 幂等 + 以我方单号反查金额核对 |
| 主动查询（query） | 秒~分钟 | 高 | 超时/UNKNOWN 必查；定时任务扫 PENDING 单补查 |
| 对账单（recon file） | T+1 | 终审 | 以渠道账单为准修正一切 |

纪律：

- **回调处理三步**：验签 → 按 out_trade_no 定位渠道交易并核对金额 → CAS 推进状态（`WHERE status IN ('REQUESTING','PENDING')`）。重复回调命中已终态直接返回成功应答，**处理成功才应答成功**（应答了渠道就不再通知）。
- **状态只进不退**：SUCCESS 是终态，后续任何回调/查询不得改写；渠道侧的"支付后退款"是新的退款交易，不是把支付改回失败。
- **掉单补偿**：定时扫描超过 X 分钟仍 PENDING/UNKNOWN 的渠道交易主动查询；超过支付有效期仍无果的关单（向渠道发关单指令 + 本地 CLOSED），**先关渠道再关本地**，防止关单后用户恰好支付成功（若发生，走退款而非拒收）。

## 3. 退款：原路退回的独立链路

```sql
CREATE TABLE pay_refund (
  id             BIGINT PRIMARY KEY,
  refund_no      VARCHAR(32) NOT NULL,
  payment_id     BIGINT NOT NULL,       -- 退哪笔支付
  channel_txn_id BIGINT NOT NULL,       -- 原路：退到原渠道交易
  refund_amount  DECIMAL(18,4) NOT NULL,
  status         VARCHAR(16) NOT NULL,  -- INIT/PROCESSING/SUCCESS/FAILED
  fail_reason    VARCHAR(255),
  idempotent_key VARCHAR(64) NOT NULL,
  UNIQUE KEY uk_idem (idempotent_key)
);
```

- **原路退回**是默认且多数场景的合规要求：退款挂原渠道交易，部分退款多次发起时**原支付单上维护已退金额，原子校验 Σ退款 ≤ 支付额**（防超退的支付侧闸门）。
- 退款是**异步且会失败的**（用户卡注销、渠道余额不足）：FAILED 要有人工处理队列（换卡退、线下退），不能吞掉。
- 组合支付的退款要**按支付单拆分**（余额部分退余额户、微信部分原路退微信），拆分规则显式配置（先退现金渠道还是先退余额，影响用户感知和资损）。

## 4. 渠道路由与渠道管理

- **渠道配置表**：渠道 × 商户号 × 能力（支付/退款/分账）× 费率 × 限额 × 状态。一个渠道多商户号（分流/备用）是常态。
- **路由规则**：按金额、终端（App/H5/小程序）、成功率、费率、商户号限额选择渠道——路由决策记录在渠道交易上（为什么走了这个商户号），费率固化（算成本用当时费率）。
- 渠道**熔断降级**：成功率跌破阈值自动切备用商户号/渠道，切换事件留痕。
- 渠道密钥/证书带有效期管理，到期前告警——证书过期是支付事故排行榜常客。

## 5. 渠道对账：终审法庭

与账务篇三层对账衔接，这里是"信息流 vs 渠道资金流"的实操：

```text
每日：下载渠道对账单 → 解析入库（原始文件留存）→ 与本地渠道交易逐笔匹配
匹配键：channel_txn_no（首选）/ out_trade_no
结果：平 | 长款（渠道有我方无）| 短款（我方成功渠道无）| 金额不符
```

- **长款**：多数是掉单（回调丢失），补单处理（补渠道交易 + 推进支付单）；无法归属的挂差错户。
- **短款**：我方标成功渠道说没有——最危险（可能发了货），立即冻结相关订单人工核查。
- **手续费对账**：渠道账单含手续费，与费率配置核算的应收手续费比对，费率被渠道悄悄调整时第一时间发现。
- 对账结果驱动账务凭证（待清算户 → 渠道应收），渠道结算入账（D+1 打款）再与银行流水对，形成完整资金闭环。

## 6. 安全与合规红线

- **卡号等敏感要素不落库**（PCI-DSS）：走渠道 token 化，本地只存脱敏卡号（前 6 后 4）和 token。
- 签名密钥进密管系统（KMS/加密机），不进配置文件和代码库。
- 金额以**分**为单位与渠道交互的，与内部 DECIMAL 元的转换只在网关边界做一次，杜绝散落各处的 ×100。
- 回调地址、跳转地址白名单校验，防钓鱼篡改。

## 7. 支付网关评审清单

```text
1. 订单/支付单/渠道交易是否三层分离？组合支付和多次尝试是否天然支持？
2. out_trade_no 是否唯一键？各渠道幂等契约是否逐一确认？
3. UNKNOWN 是否为合法状态？超时是否只能通过查询推进？
4. 回调是否验签+核对金额+CAS 推进？状态是否只进不退？
5. 掉单补查和关单流程是否存在？先关渠道再关本地？
6. 退款是否原路、防超退、失败有人工队列？组合支付退款拆分规则是否显式？
7. 渠道路由决策和费率是否固化在交易上？熔断切换是否留痕？
8. 每日渠道对账是否逐笔匹配？长短款处理是否流程化？手续费是否核对？
9. 敏感要素是否不落库/token化？密钥是否进密管？
10. 分与元的转换是否只发生在网关边界？
```

## 8. 总结

支付网关建模五条纪律：

1. **三层分离**：订单管业务，支付单管协调，渠道交易管外部事实。
2. **不信任外部**：UNKNOWN 合法、回调要验、查询确认、对账终审。
3. **幂等与单向**：单号唯一、状态只进不退、重复通知安全吸收。
4. **退款独立成链**：原路、防超退、失败可处理，不是支付的逆操作。
5. **边界收敛**：金额转换、密钥、敏感数据全部收敛在网关一层。

> 支付网关是系统里唯一"对面坐着整个外部金融世界"的模块。它的谦卑——假设一切外部交互都会失败——恰恰是资金安全的最大底气。
