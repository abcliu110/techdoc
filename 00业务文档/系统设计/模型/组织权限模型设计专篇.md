# 组织权限模型设计专篇

> 本篇聚焦组织、角色、权限、数据范围、审批责任和多租户边界。组织权限模型决定企业系统中“谁能看、谁能做、谁负责”。

## 0. 一句话总纲

> 组织权限模型的核心，是把企业责任边界映射为系统中的可见范围、可操作范围和可审计责任。

## 1. 组织模型

企业组织可能包含：

- 集团。
- 公司。
- 区域。
- 门店。
- 部门。
- 仓库。
- 业务单元。

组织模型要支持：

- 层级。
- 生效失效。
- 历史调整。
- 多组织任职。
- 组织维度报表。

交易发生时应固化组织归属，避免组织调整污染历史报表。

## 2. 角色和权限

权限分三类：

- 菜单权限。
- 操作权限。
- 数据权限。

企业系统最容易漏的是数据权限。能进入订单页面，不代表能看所有门店订单。

## 3. 数据范围模型

常见数据范围：

- 本人。
- 本部门。
- 本部门及下级。
- 指定门店。
- 指定区域。
- 全部。
- 自定义数据范围。

数据范围应和业务表中的组织、门店、客户经理、仓库等字段对应。

## 4. 权限和状态结合

同一个按钮是否可用，通常由三类条件决定：

```text
角色权限 + 数据范围 + 业务状态
```

例如“反结账”不仅需要操作权限，还需要订单属于当前门店，且订单处于可反结账状态。

## 5. 审批责任

审批不是普通权限，而是责任链。

审批模型应记录：

- 审批人。
- 审批角色。
- 审批节点。
- 审批意见。
- 审批时间。
- 代理审批。
- 加签、转交、驳回。

## 6. 评审清单

```text
1. 组织层级是否支持历史生效和失效？
2. 交易是否固化发生时组织归属？
3. 菜单、操作、数据权限是否分离？
4. 数据范围是否能映射到业务字段？
5. 按钮可执行是否同时考虑权限、数据范围和业务状态？
6. 审批是否记录责任链？
7. 权限变更是否有审计？
```

## 7. 总结

组织权限建模五条纪律：

1. **组织有历史**：组织调整不能改写历史业务。
2. **权限分层**：菜单、操作、数据范围不是一回事。
3. **状态参与授权**：有权限不等于当前状态可操作。
4. **审批是责任链**：审批记录要能追责。
5. **安全默认拒绝**：无法确认权限时默认不可见或不可操作。
