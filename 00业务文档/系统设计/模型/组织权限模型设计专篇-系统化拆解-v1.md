# 组织权限模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

组织权限模型的本质，是把企业组织结构、岗位责任和数据边界映射成系统中的可见范围、可操作范围和审批责任。

核心抽象：

```text
租户 -> 组织 -> 岗位 -> 用户 -> 角色 -> 数据范围 -> 审批责任
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| org_unit 组织节点 | org_code, org_name, parent_id, org_path, org_type(集团/公司/区域/门店/部门/仓库/EXTERNAL/VIRTUAL), effective_from, effective_to, tenant_id | 带时间的单父组织树；节点只失效不删除，历史归属永远可解释 |
| org_legal_ext 法人扩展 | org_id, 税号, 开票主体, 核算币种 | org_type=LEGAL_ENTITY 的节点属性；法人是财务核算边界 |
| org_shop_ext 门店扩展 | org_id, 地址, 营业时间, 默认仓库映射 | 门店经营属性垂直下沉，不污染树结构 |
| org_position 岗位 | position_code, position_name, org_id, status | 授权的载体：岗挂角色，人不直接挂角色 |
| org_assignment 任职 | user_id, position_id, org_id, is_primary, assign_from, assign_to | 人×岗×组织×期间；多条任职吸收矩阵/兼岗，只增不改支撑审计回放 |
| auth_role 角色 | role_code, role_name, tenant_id, status | 菜单/操作权限的集合，租户内定义 |
| auth_position_role 岗位-角色 | position_id, role_id | 调动=任职变化，权限自动跟随岗位 |
| auth_data_scope 数据范围 | role_id, resource, scope_type(ALL/ORG_TREE/ORG/SELF/CUSTOM), custom_expr | 角色×资源→行过滤；映射到业务表的 org_id/owner_id 等归属字段 |
| approval_snapshot 审批责任快照 | 业务单据 id, 审批节点, approver_id, 当时岗位 id, 当时组织 id, 审批意见, 审批时间, 代理人 | 固化"当时谁以什么身份批的"，组织调整后责任链不漂移 |

### 2.2 实体关系

```text
org_unit (1) ──< org_unit (N)            parent_id 单父树；org_path 前缀过滤子树
org_unit (1) ──< org_legal_ext (1)       法人节点扩展（按 org_type 垂直挂）
org_unit (1) ──< org_shop_ext (1)        门店节点扩展
org_unit (1) ──< org_position (N)        岗位挂组织
org_position (1) ──< org_assignment (N)  一岗多任职；一人多条任职成矩阵
org_position (1) ──< auth_position_role (N) >── (1) auth_role  岗挂角色
auth_role (1) ──< auth_data_scope (N)    角色×资源→数据范围
org_assignment (1) ──< approval_snapshot (N)  审批固化当时任职
外部引用：user_id → 用户域；业务单据 org_id/legal_entity_id 归属字段 → 各业务域
        （交易发生时固化，不随组织调整改写）；审批节点 → 工作流域；
        tenant_id → 租户域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：(tenant_id, org_code) 唯一；(user_id, position_id, assign_from) 任职唯一；
       (role_id, resource) 数据范围唯一
时间约束：org_unit / org_assignment 带 effective 区间、只增不改；
        同一节点同一时刻只有一个父（org_path 才成立）
归属固化：交易落库时固化 org_id/legal_entity_id 快照，
        历史报表按发生时归属统计，组织调整零污染
树纪律：单父树，多重隶属靠任职多条，禁止组织多父；
       删除节点改为失效，禁止物理删除
tenant_id 纪律：组织树/岗位/角色/范围全部带 tenant_id，跨租户不可见
默认拒绝：无法确定权限时默认不可见不可操作
```

## 3. 核心张力

最大张力是：

```text
组织灵活调整 vs 历史业务归属稳定
```

企业组织会调整，但历史订单、库存、销售、会员归属不能被当前组织结构改写。

## 4. 不变量

- 组织有生效和失效时间。
- 交易发生时必须固化组织归属。
- 数据范围要映射到业务字段。
- 审批记录要保留当时审批人和岗位。
- 组织调整不能污染历史报表。

## 5. 为什么这样设计

组织模型不仅是树形结构，更是权限、报表、审批、责任和经营分析的共同基础。没有历史组织归属，所有历史经营数据都会随组织调整而漂移。

## 6. 失败模式

- 交易只引用当前组织 ID，组织调整后历史报表变化。
- 数据权限无法映射到门店、区域、仓库。
- 审批记录只存用户 ID，不存当时岗位和组织。
- 组织删除导致历史数据无法解释。

## 7. 适用边界

适用：

- 多公司、多门店、多部门、多区域、多仓库、多租户系统。

简化：

- 单组织系统可简化组织层级，但仍应保留数据归属字段。

## 8. 同一模型的多业务形态建模

组织权限模型要同时支撑多种组织形态，核心方法是：**统一组织树骨架 + 组织类型标识 + 任职关系吸收结构差异**。

### 8.1 统一骨架（所有业态不变的部分）

```text
租户 → 组织节点（org_type 标识形态，带生效/失效区间、org_path）→ 岗位 → 任职（人×岗×组织×期间）→ 角色 → 数据范围
不变量全集：组织有历史生效、交易固化发生时归属、数据范围映射业务字段、审批责任快照
```

骨架决不为某个形态特化——任何形态的"谁挂在哪、管到哪、历史归属是什么"都用同一棵带时间的组织树 + 任职记录回答。

### 8.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 集团多法人 | 法人是财务核算边界，跨法人交易要关联结算 | 组织节点 org_type=LEGAL_ENTITY，节点挂法人扩展表（税号/开票主体/币种）；单据固化 legal_entity_id，跨法人走内部结算单 |
| 连锁门店树 | 门店是经营和库存单元，层级=集团/区域/门店 | 门店节点挂门店扩展表（地址/营业时间/仓库映射）；数据范围按 org_path 前缀过滤区域下所有门店 |
| 加盟体系 | 加盟商是外部法人，总部只见脱敏经营数据 | 加盟商组织标记 owner_type=FRANCHISEE，归属外部法人主体；总部对加盟门店的数据范围走列权限脱敏，结算关系另建加盟合同 |
| 事业部矩阵（一人多岗） | 一人同时挂行政线和业务线 | 任职表天然多条（人×岗×组织），每条任职独立带角色和数据范围；主任职标记 is_primary 供默认展示，权限取多任职并集 |
| 外部合作方组织 | 供应商/ISV 人员进系统但不进内部编制 | 独立的外部组织子树（org_type=EXTERNAL），任职机制复用；数据范围强制 SELF/指定范围，禁止 ORG_TREE 向内部组织展开 |
| 虚拟团队/项目组 | 跨部门临时聚合，不改行政隶属 | 虚拟组织节点（org_type=VIRTUAL）+ 带起止时间的任职；只作数据范围和审批路由输入，不参与行政报表归属 |

### 8.3 建模决策规则

- **能用 org_type + 节点扩展表解决的，不建第二棵树**：法人/门店/部门差异在节点属性，树结构和任职机制一套。
- **多重隶属靠任职多条，不靠组织多父**：树保持单父（org_path 才能成立），矩阵关系全部下沉到任职表。
- **形态裁剪层级但不改时间语义**：单店系统可以只有一层，但节点仍带生效失效，交易仍固化归属。
- **拆独立域的信号**：当"组织"承载的是交易对手关系而非责任边界（如客户主数据、供应商主数据），应归属对应主数据域，不是组织树的节点类型。
