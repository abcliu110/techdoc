# 订单模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

订单模型的本质，是一次交易承诺的证据链。它记录用户买了什么、以什么价格成交、如何支付、如何履约、发生过哪些变更、退过什么、最终如何结算。

核心抽象：

```text
订单主表 -> 订单行 -> 金额分解 -> 支付记录 -> 履约记录 -> 逆向单据
```

订单不是一张表，而是交易事实的中心索引。

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| ord_order 主单 | order_no(唯一), buyer_id, order_type, status, 应付/实付/优惠/运费金额, 下单时间 | 交易的根，状态克制（≤8个终身状态） |
| ord_order_line 订单行 | order_id, sku_id, qty, deal_price, 商品快照(标题/规格/图), refunded_qty, refunded_amount, shipped_qty | 最小计算颗粒；快照+留量防超退超发 |
| ord_amount_detail 金额分解 | order_id, amount_type(商品/运费/税费/优惠), amount, currency | 恒等式的分项证据 |
| ord_discount 优惠分摊 | order_line_id, activity_id, rule_version, 分摊金额, cost_bearer 出资方 | 订单级优惠拆到行；出资方定结算 |
| ord_payment 支付记录 | order_id, payment_no, pay_way, amount, status, paid_at | 一单多次支付/组合支付天然支持 |
| ord_fulfillment 履约记录 | order_id, 履约单号, 类型(发货/核销/发放), status | 指向履约域，订单不直接管物流 |
| ord_reverse 逆向单据 | 原 order_id/line_id, 逆向类型(退款/退货/取消), qty, amount, status | 独立生命周期，行级引用原单 |
| ord_status_log 状态流水 | order_id, 旧状态, 新状态, 触发原因, operator, 时间 | 状态只进不退的证据链 |

### 2.2 实体关系

```text
ord_order (1) ──< ord_order_line (N)      主单-行：一对多
ord_order (1) ──< ord_amount_detail (N)   金额分解挂主单
ord_order_line (1) ──< ord_discount (N)   优惠分摊到行
ord_order (1) ──< ord_payment (N)         组合支付/多次尝试
ord_order (1) ──< ord_fulfillment (N)     拆单发货一对多
ord_order_line (1) ──< ord_reverse 明细 (N)  逆向行级引用原行
ord_order (1) ──< ord_status_log (N)      状态流水只增
外部引用：sku_id → 商品域（值靠快照固化）；buyer_id → 会员域；
         payment_no → 支付域；履约单号 → 履约域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：order_no 全局唯一；(order_id, sku_id, 行序号) 行唯一；
       逆向单幂等键唯一（防重复退款）
留量校验：Σ逆向 qty ≤ 行 qty - 已退 qty（原子 UPDATE 条件）；
         Σ包裹发货 ≤ 行 qty
恒等式：应付 = Σ行成交额 + 运费 + 税费 - Σ优惠；实付 = Σ成功支付
       落库前强校验，不满足即拒绝
分片键：buyer_id（买家查询主路径）；卖家查询走异构读模型
```

## 3. 核心张力

订单模型最大的张力是：

```text
交易主流程简洁 vs 交易变化和逆向复杂
```

业务希望订单流程简单：下单、支付、发货、完成。但真实系统存在组合支付、优惠分摊、部分发货、部分退款、发票、售后、拆单、改价、反结账等复杂情况。

因此订单模型必须让主单状态保持克制，把复杂度下沉到订单行、金额明细、支付记录、履约单和逆向单据。

## 4. 不变量

- 订单行是交易最小计算颗粒。
- 下单时商品、价格、规则必须快照。
- 订单金额必须满足恒等式。
- 订单级优惠必须分摊到行。
- 退款、退货、取消必须是独立单据。
- 主单状态不能承载所有组合状态。
- 跨域协作不能直接修改其他域数据。

## 5. 为什么这样设计

行级建模是为了支撑部分发货、部分退款、行级结算、行级开票、行级毛利和行级优惠分摊。

快照设计是为了防止商品、价格、会员等级、规则后续变化污染历史订单。

逆向单据独立，是为了让退款、退货、换货、取消有自己的生命周期，而不是把所有异常塞进订单状态。

## 6. 失败模式

- 订单只有主表和总金额，后续无法做部分退款。
- 商品信息只引用当前商品，历史订单展示被商品改名污染。
- 优惠不分摊到行，导致退款、结算、开票不准。
- 一个订单状态字段塞入几十个组合状态，没人敢改。
- 订单直接扣库存、改余额，导致跨域耦合。

## 7. 适用边界

适用：

- 电商、零售、POS、采购、B2B 交易、服务订单、订阅订单。

不适用：

- 极简单的一次性登记记录。但一旦涉及支付、履约、退款或报表，就应采用订单证据链模型。

## 8. 设计取舍

优先保证：

- 交易可追溯。
- 金额可解释。
- 逆向可处理。
- 报表可还原。

牺牲：

- 表数量。
- 查询复杂度。
- 初期实现速度。

## 9. 同一模型的多业务形态建模

订单模型要同时支撑多种业务形态，核心方法是：**统一骨架 + 类型标识 + 差异下沉到扩展表**。

### 9.1 统一骨架（所有业态不变的部分）

```text
主单（order_type 标识业态）→ 订单行 → 金额分解 → 支付记录 → 履约记录 → 逆向单据
不变量全集：行级颗粒、金额恒等式、快照固化、逆向独立单据
```

骨架决不为某个业态特化——任何业态的"买了什么、付了多少、履约到哪、退了什么"都用同一套表回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 电商订单 | 收货地址、物流履约、超时关单 | 地址快照表 + 履约单（标准形态，骨架原型） |
| POS 门店单 | 无地址、当场履约、一单一付、离线补传 | 履约记录退化为"当场交付"一条；门店/收银员/班次字段进扩展表；离线单带本地单号+幂等上传 |
| B2B 大宗单 | 合同价、账期、分批发货、对公支付 | 行级挂合同 ID+合同价版本；支付记录支持账期（应收挂账）；分批发货靠行级 shipped_qty（骨架已支持） |
| 服务订单 | 无物流、按次/按时段履约、核销制 | 履约记录 = 预约/核销记录；订单行挂服务 SKU + 服务时段扩展表 |
| 订阅订单 | 周期扣费、续费、按期退款 | 订阅是"母合同"独立建模，每期生成一张标准订单；退款按剩余期折算规则挂母合同 |
| 虚拟/卡券商品 | 即时履约、发放即交付 | 履约记录 = 发放记录（卡密/券 ID）；无逆向物流，退款规则按已使用量 |
| 预售订单 | 定金+尾款两段支付 | 支付记录天然多条（骨架已支持）；主单加预售阶段扩展表，定金不退规则显式配置 |

### 9.3 建模决策规则

- **能用 order_type + 扩展表解决的，不建新订单表**：差异只在字段和履约方式时，主链路一套（报表、对账、退款复用）。
- **扩展表按业态垂直拆**（order_pos_ext / order_presale_ext），不在主表堆可空字段——主表列稳定，业态扩展互不污染。
- **状态机允许按业态裁剪但不允许新增语义**：POS 单跳过"待发货"，但不得发明骨架之外的新终态。
- **拆独立域的信号**：当某业态的"订单行/金额/逆向"不变量本身不同（如金融交易、竞拍），才是独立域，不是订单的一种类型。

```yaml
type: model_decomposition

core_grain: 订单行

  - 金额恒等式成立
  - 交易快照不可被当前主数据污染

  - 主状态爆炸
  - 优惠无法退款分摊

  strategy: 统一骨架 + order_type + 业态垂直扩展表

```
