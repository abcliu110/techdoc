# 权限模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦权限建模。权限是企业系统的隐形骨架：设计得好没人注意，设计得差每个需求都要绕它走。核心难点不在"能不能点按钮"，而在**数据权限**——同一个列表，不同人看到不同的行。

## 0. 一句话总纲

> 权限模型回答两个正交的问题：你能做什么动作（功能权限），你能触达哪些数据（数据权限）。混在一起设计，两个都做不好。

## 1. 功能权限：标准 RBAC 及其边界

### 1.1 经典四表

```text
用户（user）→ 用户-角色（user_role）→ 角色（role）→ 角色-权限（role_permission）→ 权限点（permission）
```

```sql
CREATE TABLE auth_permission (
  id        BIGINT PRIMARY KEY,
  perm_code VARCHAR(64) NOT NULL,   -- order:refund:approve，代码引用的稳定标识
  perm_type VARCHAR(16) NOT NULL,   -- MENU菜单/BUTTON按钮/API接口
  parent_id BIGINT,                 -- 菜单树
  UNIQUE KEY uk_code (perm_code)
);
```

纪律：

- **权限点用语义编码**（`domain:resource:action`）而非自增 ID 被代码引用，权限点是"枚举"性质（总览篇 §18），随代码发布而增减。
- 前端按钮显隐只是体验，**API 层必须独立鉴权**——前端校验挡不住直接调接口。
- 角色是权限的集合，禁止"直接给用户挂权限点"作为常规通道（可留紧急授权口子，但要有有效期和审计）。

### 1.2 RBAC 的常见扩展

- **角色互斥与上限**（合规需求）：出纳与审批人不能同人（职责分离 SoD），用互斥角色组约束。
- **有效期授权**：user_role 带生效/失效时间，临时授权到期自动收回，替代"忘了回收"这一最大安全漏洞。
- **多租户下的角色**：平台预置角色模板 + 租户自定义角色，两层并存，租户不能越过平台给自己加权限点。

## 2. 数据权限：真正的难点

### 2.1 数据权限的本质是"行过滤规则"

同样是"查看订单列表"：

- 店员只能看本门店订单。
- 区域经理看本区域所有门店。
- 财务看全部但看不到客户手机号（列权限）。
- 销售只能看自己名下客户的订单（本人维度）。

抽象后，数据权限 = **给查询自动追加的过滤条件**，常见维度：

| 维度 | 过滤形态 |
|---|---|
| 组织维度 | org_id IN (我的组织及下级) —— 最常见 |
| 本人维度 | owner_id = 我 |
| 区域/渠道维度 | region_id IN (...) |
| 自定义规则 | 字段级条件表达式 |

### 2.2 落地模型

```sql
-- 数据权限策略：角色 × 资源 → 范围类型
CREATE TABLE auth_data_scope (
  id          BIGINT PRIMARY KEY,
  role_id     BIGINT NOT NULL,
  resource    VARCHAR(64) NOT NULL,   -- 作用的业务资源：order/customer/report...
  scope_type  VARCHAR(24) NOT NULL,   -- ALL全部/ORG_TREE本部门及以下/ORG本部门/SELF本人/CUSTOM
  custom_expr VARCHAR(500),           -- CUSTOM 时的规则表达式
  UNIQUE KEY uk_role_res (role_id, resource)
);
```

实施要点：

- **业务表必须带权限维度字段**：数据权限能不能做，取决于订单表上有没有 org_id / owner_id——这是建模期的决定（总览篇 §24"交易固化归属"在此处兑现）。权限系统只能过滤已存在的字段，不能变出维度。
- **组织树过滤用路径字段**：`org_path LIKE '/1/4/%'` 一次命中整棵子树（总览篇 §35），避免运行时递归查子部门。
- 过滤条件在**服务端查询构造层统一注入**（MyBatis 拦截器/ORM 全局 scope），不靠每个开发者记得手写 where——漏写一处就是数据泄露。
- 多个角色的数据范围取**并集**；数据权限与功能权限独立判断。

### 2.3 列权限与脱敏

- 列权限 = 资源 × 角色 × 字段 → 可见/脱敏/隐藏，配置化。
- 与总览篇 §29 分级联动：敏感级别高的字段默认脱敏，"查看明文"是独立权限点且**每次查看记审计日志**。
- 导出是独立权限且单独审计——泄露事故大多发生在导出环节。

## 3. 组织与岗位：权限的载体

成熟企业系统中，授权对象往往不是"人"而是**岗位**：

```text
人 → 任职（人×岗位×组织，带任职期间）→ 岗位 → 角色
```

- 人员调动 = 任职记录变化，权限自动跟随岗位，无需逐人调整。
- 兼岗（一人多岗）天然支持；任职历史（谁在什么期间任什么岗）满足审计"当时他有没有权限"——这是**权限的双时态问题**（总览篇 §12）：审计需要的是"操作发生时"的权限状态，任职和授权记录都要带时间区间、只增不改。
- 代理/委托（休假期间审批权临时给他人）：独立的委托记录（委托人、受托人、范围、期限），并在操作日志中同时记录"实际操作人 + 以谁的身份"。

## 4. 审批流中的权限

审批人的确定是权限模型的高频消费场景：

- 审批规则引用组织/岗位/角色（"申请人所在部门的负责人""金额>10万 加签财务总监"），不要写死人名。
- 审批记录固化**当时的审批人和其岗位**（快照），组织调整后历史审批依然可解释。
- 审批权与查看权分离：能审批某单不代表能看该类所有单。

## 5. 多租户隔离：权限之下的地基

- 租户隔离不是数据权限的一种，而是**更底层的硬边界**：tenant_id 在连接/会话层强制注入，任何查询逃不掉（行级安全策略或强制拦截器），数据权限在租户内部再做细分。
- 平台运营人员跨租户操作要走**特权通道**：明确的授权、操作理由、全程审计，而不是"超级账号"随意进出。

## 6. 权限评审清单

```text
1. 功能权限与数据权限是否分离设计？
2. 权限点是否语义编码、随代码管理？API 层是否独立鉴权？
3. 业务表是否带齐权限维度字段（org/owner/tenant）？交易是否固化当时归属？
4. 组织树过滤是否用路径字段而非运行时递归？
5. 数据范围过滤是否框架层统一注入而非手写 where？
6. 是否支持岗位授权、任职历史、有效期授权、委托代理？
7. 审计能否回答"操作发生时他有没有权限"（授权记录带时间区间）？
8. 列权限/脱敏/导出是否独立管控并审计？
9. 租户隔离是否在权限之下强制生效？
10. 职责分离（互斥角色）是否有约束？
```

## 7. 总结

权限建模五条纪律：

1. **功能与数据两条线**：RBAC 管动作，数据范围管行，列权限管字段。
2. **维度先入模型**：数据权限的上限由业务表上的归属字段决定。
3. **岗位承载授权**：人挂任职，岗挂角色，调动不重配。
4. **统一注入**：过滤靠框架强制，不靠自觉。
5. **权限可追溯**：授权、任职、查看敏感数据，全部带时间、可回放。

> 权限系统的失败有两种：管得太死业务绕开它，管得太松等于没有。出路只有一条——把权限维度当作业务模型的一部分来设计，而不是事后套壳。
