# 会员与积分模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

会员与积分模型的本质，是用户经营中的“身份 + 成长 + 类资金权益”模型。它既要识别会员，也要记录会员如何升级、如何获得权益、如何消费权益、如何回滚权益。

核心抽象：

```text
One ID -> 会员档案 -> 等级规则 -> 积分账户 -> 积分流水 -> 权益资产
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| mbr_customer 自然人身份 | customer_id(不变内部主键), status(正常/MERGED/注销), merged_to_id, 注册时间 | One ID 之根；所有业务表引用它，合并不物理删除 |
| mbr_credential 登录凭证 | customer_id, 凭证类型(手机/unionid/邮箱), 凭证哈希指纹, 绑定时间, 解绑时间 | 凭证与身份分离；绑定历史只增，回答"这个号 3 月属于谁" |
| mbr_membership 会员档案 | customer_id, membership_type(免费/付费/企业), 会员号, 当前等级, 会籍起止时间, status | 某体系内的会员身份；付费会籍带有效期与续费 |
| mbr_growth_flow 成长值流水 | customer_id, 行为类型(消费/签到/评价), biz_order_no, 成长值增减, 时间 | 等级计算的原料，只增不改 |
| mbr_level_change 等级变更记录 | customer_id, 旧等级, 新等级, 触发原因(升级/保级失败/人工), 规则版本, 生效时间, 审批单号 | 等级是计算结果的证据；人工调级走单据 |
| mbr_point_batch 积分批次 | customer_id, biz_type, biz_order_no, issue_idem_key(唯一), issued 发放量, remaining 剩余量, expire_at | 按批次记账；消耗和过期都从 remaining 扣，FIFO |
| mbr_point_flow 积分流水 | customer_id, 动作(发放/消耗/过期/冲正), batch_id, 数量, 关联单号, 时间 | 当前值的唯一解释来源；消耗记录跨批次明细 |
| mbr_stored_account 储值账户 | customer_id, 本金余额, 赠金余额, currency, status | 真实资金；本金赠金分户，退卡只退本金 |
| mbr_benefit_instance 权益实例 | customer_id, 权益定义ID, 有效期起止, 总次数, 已用次数, status | 资格类资产；次数型按周期发放、用一记一 |
| mbr_merge_order 合并单 | 主 customer_id, 被合并 customer_id, 合并时间, 操作依据, 风控结果 | 账户合并的证据；余额迁移走转账凭证不 UPDATE 相加 |

### 2.2 实体关系

```text
mbr_customer (1) ──< mbr_credential (N)        一人多凭证，绑定史只增
mbr_customer (1) ──< mbr_membership (N)        可入多个会员体系
mbr_customer (1) ──< mbr_growth_flow (N)       成长值流水
mbr_customer (1) ──< mbr_level_change (N)      等级变更留痕
mbr_customer (1) ──< mbr_point_batch (N)       积分按批次挂人
mbr_point_batch (1) ──< mbr_point_flow (N)     消耗/过期回指批次
mbr_customer (1) ──1 mbr_stored_account        储值账户（本金/赠金分列）
mbr_customer (1) ──< mbr_benefit_instance (N)  权益实例带有效期
mbr_customer (1) ──< mbr_merge_order (N)       被合并身份指向主身份
外部引用：biz_order_no → 订单域（消费赠积分/退款冲正的来源）；
         权益定义 → 营销域券模板；储值充值/退款凭证 → 资金账务域
         （引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：issue_idem_key 唯一（积分发放幂等）；凭证哈希指纹 + 生效状态唯一
       （一个手机号同时只属于一人）；(customer_id, 权益定义ID, 周期) 防重复发放
留量校验：积分消耗按 FIFO 原子扣批次：
         remaining = remaining - x WHERE remaining >= x；
         储值扣减 WHERE 余额 >= 金额，本金/赠金扣减顺序写进规则并落流水
恒等式：账户总余额 = Σ批次 remaining（每日对账）；
       积分负债余额 = Σ发放 - Σ消耗 - Σ过期（审计科目）；
       储值可退金额 = 本金余额 - 已享优惠追回
状态机约束：customer 正常 → MERGED/注销（终态，不回写历史交易）；
           权益实例 有效 → 已用尽/过期；负积分策略显式定义（挂负或后续抵扣）
版本化规则：等级 = f(窗口期成长值, 等级规则版本)，规则改版不重算历史；
           订单固化下单时等级
分片键：customer_id（会员数据全部按人聚合查询）
```

## 3. 核心张力

最大张力是：

```text
营销灵活性 vs 权益账务纪律
```

运营希望积分和权益玩法灵活，但积分、储值、券一旦可抵扣或可兑换，就具备准资金属性，必须遵守流水、幂等、冲正、过期和审计纪律。

## 4. 不变量

- One ID 是身份合并基础。
- 积分当前值必须由流水解释。
- 积分发放、消耗、过期、冲正必须有来源。
- 等级是规则计算结果，不应只是手工字段。
- 储值必须按资金模型治理。
- 权益资产必须有有效期、状态和核销记录。

## 5. 为什么这样设计

积分按准资金建模，是为了防止“送错、扣错、退错、过期错”无法解释。等级计算模型化，是为了避免人工字段和真实消费行为脱节。

## 6. 失败模式

- 积分只存余额，没有流水。
- 退款时只退钱，不冲正积分。
- 等级手工修改无记录。
- 储值当普通积分处理，造成财务风险。
- 权益没有状态机，核销和过期混乱。

## 7. 适用边界

适用：

- 会员积分、成长值、储值、等级、权益包、会员营销。

简化：

- 只做展示型会员资料且无权益资产时，可不引入完整账务纪律。

## 8. 同一模型的多业务形态建模

会员体系形态多样，但支撑它们的骨架只有一套：**One ID + 成长值流水 + 积分批次，形态差异下沉到会籍类型和账户类型**。

### 8.1 统一骨架（所有形态不变的部分）

```text
One ID → 会员档案（membership_type 标识形态）→ 成长值/积分流水 → 积分批次（先进先出过期）→ 权益资产 → 冲正记录
不变量全集：当前值由流水解释、发放消耗过期冲正有来源、等级是计算结果、储值按资金纪律
```

骨架决不为某个形态特化——任何形态的"这个人是谁、积了多少、怎么积的、退款怎么冲"都用同一套表回答。

### 8.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 免费等级会员 | 等级由消费行为计算 | 成长值流水 + 等级规则表驱动升降级（标准形态，骨架原型） |
| 付费会籍（年卡） | 等级靠购买获得、有有效期、可退 | 会籍订单 + 会籍有效期表（起止时间、续费记录）；等级判定改为"付费会籍优先于成长值等级"；退卡按剩余天数折算，走标准冲正 |
| 积分商城 | 积分作为支付手段消耗 | 兑换单独立建模（兑换订单 + 积分消耗流水，幂等键=兑换单号）；扣减按批次先进先出，兑换失败按批次原路返还 |
| 储值会员 | 余额是真实资金 | 储值账户与积分账户分表，储值走资金域纪律（充值/消费/退款流水、本金与赠金分列）；赠金是营销成本，退卡只退本金 |
| 企业会员 | 主体是组织、额度多人共享 | 会员主体挂客户域企业 ID；新增"成员-主体"关系表，积分/额度记在企业账户，流水带 operator_member_id 区分是谁消费 |
| 多品牌联合会员 | 积分跨品牌通兑、汇率不同 | 各品牌独立积分账户 + 通兑规则表（品牌间兑换比率、版本化）；通兑生成一出一入两条流水，双边幂等，不做余额直改 |

### 8.3 建模决策规则

- **能用 membership_type + 账户类型解决的，不建新会员表**：免费/付费/企业会员共用档案和流水表，差异在会籍表和账户维度。
- **账户按资产性质垂直拆**（积分账户/储值账户/成长值），不在一个 balance 字段里混装——储值是钱，积分是权益，纪律等级不同。
- **等级判定规则允许按形态换来源，但不允许绕过流水**：付费会籍直接给等级，也必须有会籍记录可追溯，不得手工改等级字段。
- **拆独立域的信号**：当"账户"需要计息、提现、对外清算（如钱包、预付卡监管），资金不变量已超出会员域，应交给资金账务域，会员域只留引用。

## 9. AI 知识库沉淀

```yaml
type: model_decomposition
domain: 会员与积分
essence: 身份 + 成长 + 类资金权益
core_tension: 营销灵活性 vs 权益账务纪律
invariants:
  - One ID
  - 积分流水
  - 退款冲正
  - 等级规则化
business_forms:
  strategy: One ID+流水+批次骨架 + membership_type/账户类型吸收形态差异 + 账户按资产性质垂直拆
  forms: [免费等级会员, 付费会籍, 积分商城, 储值会员, 企业会员, 多品牌联合会员]
  split_signal: 账户涉及计息提现清算等资金监管时交给资金账务域
```
