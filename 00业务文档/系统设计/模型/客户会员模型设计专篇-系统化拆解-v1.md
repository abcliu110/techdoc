# 客户会员模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

客户会员模型的本质，是经营“人”和“组织关系”的系统。它回答：这个对象是谁、属于谁、有什么权益、发生过什么行为、未来如何被服务。

核心抽象：

```text
用户身份 -> 会员/客户主体 -> 归属关系 -> 权益账户 -> 行为事件 -> 标签画像
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| cus_user 用户身份 | user_id(内部主键), 凭证类型(手机/openid/unionid), 凭证指纹, 绑定/解绑时间, status | 登录账号主体；凭证与身份分离，支持后续合并 |
| cus_customer 客户/会员主体 | customer_id, customer_type(个人/企业/经销商), 名称, 所属租户/品牌, status | 经营对象之根；B2C 与 B2B 共用主体表，类型定形态 |
| cus_contact 联系人 | customer_id, 姓名, 角色(采购/财务/收货), 联系方式, user_id 引用, status | 企业客户下的自然人；联系人不是客户本身 |
| cus_belonging 归属关系 | customer_id, 归属类型(门店/渠道/客户经理/推荐人), 归属对象ID, 生效起止时间 | 归属可变但历史可查；交易固化当时归属 |
| cus_benefit_account 权益账户 | customer_id, 账户类型(积分/储值/成长值), 当前值, 冻结值, status | 当前值容器；每种资产独立账户，纪律不同 |
| cus_benefit_flow 权益流水 | account_id, 动作(发放/消耗/冻结/过期/冲正), 数量, biz_order_no, 幂等键, 规则版本, 时间 | 当前值的唯一解释来源；退款/反结账靠冲正 |
| cus_level_record 等级记录 | customer_id, 旧等级, 新等级, 触发原因, 规则版本, 计算快照, 生效时间 | 等级是规则计算结果，变更全留痕 |
| cus_tag 标签 | customer_id, 标签ID, 标签类型(人工/规则/模型), 来源, 规则版本, 打标时间 | 每个标签有来源可解释 |
| cus_segment 分群 | segment_id, 分群规则, 规则版本, 计算时间, 命中人数, 关联活动ID | 解释"当时为什么这个会员被选中" |
| cus_event 行为事件 | customer_id, 事件类型(支付/退款/核销/过期), biz_order_no, 事件时间, 载荷 | 权益变化的事实依据，只增不改 |

### 2.2 实体关系

```text
cus_user (N) >──< cus_customer (M)          一用户可绑多会员身份（关系表带 role_type）
cus_customer (1) ──< cus_contact (N)        企业客户下挂多联系人
cus_customer (1) ──< cus_belonging (N)      归属关系带生效区间
cus_customer (1) ──< cus_benefit_account (N)  按资产类型分账户
cus_benefit_account (1) ──< cus_benefit_flow (N)  流水解释当前值
cus_customer (1) ──< cus_level_record (N)   等级变更只增
cus_customer (1) ──< cus_tag (N)            标签多值
cus_segment (N) >──< cus_customer (M)       分群命中名单（版本化）
cus_customer (1) ──< cus_event (N)          行为事件流
外部引用：biz_order_no → 订单域（订单固化当时等级/归属）；
         券资产 → 营销域；储值资金凭证 → 资金账务域；
         门店/客户经理 → 组织域（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：凭证指纹 + 生效状态唯一（一凭证同时只属一人）；
       权益流水幂等键唯一（防重复发放/冲正）；
       (customer_id, 归属类型, 生效区间) 同类型归属不重叠
留量校验：权益消耗原子 UPDATE：当前值 = 当前值 - x
         WHERE 当前值 - 冻结值 >= x；储值按资金域纪律另行治理
恒等式：账户当前值 = Σ流水净额（每日对账）；
       冻结值 = Σ未释放冻结流水
版本化规则：等级/标签/分群规则修改发新版本，命中记录固化规则版本；
           归属变更不回写历史订单，历史按交易时快照解释
隐私约束：手机号/证件加密存储 + 指纹检索，展示脱敏；
         访问与导出留审计记录；注销走匿名化流程，交易骨架保留
分片键：customer_id（按人聚合是主查询路径）
```

## 3. 核心张力

客户会员模型最大的张力是：

```text
统一识别 vs 多身份多归属
```

业务希望一个人能被统一识别，但真实世界中同一个人可能有多个手机号、微信身份、门店会员身份、企业客户联系人身份和不同归属关系。

## 4. 不变量

- 用户、会员、客户、联系人不能混为一张表。
- 权益当前值必须能由流水解释。
- 等级、标签、分群规则必须可追溯。
- 会员归属变化不能污染历史订单。
- 敏感信息访问必须受控。

## 5. 为什么这样设计

身份分层是为了支持 B2C 和 B2B 同时存在。会员权益账户化，是为了支持积分、储值、成长值、券、等级这些会变化且需要追溯的资产。

标签和分群版本化，是为了能解释当时为什么这个会员被营销活动命中。

## 6. 失败模式

- 用手机号直接当会员主键，后续无法合并身份。
- 积分只存当前值，退款和反结账无法解释。
- 客户、联系人、会员混用，B2B 场景崩塌。
- 标签没有来源和版本，营销命中不可解释。
- 敏感信息随意导出，造成合规风险。

## 7. 适用边界

适用：

- CRM、会员系统、零售、餐饮、B2B 客户管理、营销平台。

可简化：

- 只有匿名购买、无权益、无复购经营的场景可简化为用户档案。

## 8. 同一模型的多业务形态建模

客户形态跨越 C 端和 B 端，但支撑它们的骨架只有一套：**身份分层（凭证/渠道身份/会员身份）+ 主体类型标识 + 关系表吸收归属差异**。

### 8.1 统一骨架（所有形态不变的部分）

```text
凭证（手机号/openid/unionid）→ 渠道身份 → One ID → 客户/会员主体（customer_type 标识形态）→ 归属关系 → 权益账户 → 行为事件
不变量全集：用户/客户/联系人分表、权益由流水解释、归属变化快照化、敏感信息受控
```

骨架决不为某个形态特化——任何形态的"这是谁、通过什么渠道来的、属于谁、有什么权益"都用同一套分层回答。

### 8.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| C 端个人客户 | 一人一主体、身份即客户 | 凭证→One ID→个人主体一对一（标准形态，骨架原型） |
| B 端企业客户 | 主体是组织、下挂多个联系人 | customer_type=ENTERPRISE，企业主体表存工商信息；联系人独立建表挂企业 ID，联系人自己的登录身份走凭证层，角色（采购/财务/收货）字段区分职责 |
| 经销商 | 有层级、有授信、有区域 | 企业客户的子类型 + 经销商扩展表（上级经销商 ID、授信额度、授权区域）；层级用 parent_id 树，返利结算引用客户 ID 不复制主体 |
| 门店店主 | 既是客户又是渠道 | 一个 One ID 挂两个主体角色：个人会员身份 + 店主经营身份（关系表带 role_type）；进货订单归经营身份，个人消费归会员身份，权益账户分开 |
| 内部员工客户 | 员工价、额度限制、需防套利 | 个人主体加员工标识（关联 HR 工号，离职自动失效）；员工价资格由标识驱动，月度额度记专用权益账户，行为事件供风控审计 |

### 8.3 建模决策规则

- **能用 customer_type + 扩展表解决的，不建新客户表**：个人/企业/经销商共用主体表和凭证层，差异在类型字段和垂直扩展表。
- **"一人多角色"用关系表，不复制主体**：店主、员工、联系人都是 One ID 之上的角色关系，复制主体会让身份合并永久失效。
- **B 端的组织结构放扩展，不放身份层**：企业的部门、层级、授信是客户域业务属性；凭证和渠道身份层保持与形态无关。
- **拆独立域的信号**：当"客户"需要独立的合同、结算、绩效体系（如供应商、渠道商管理），经营关系复杂度已超出客户档案，应拆入采购/渠道域，客户域只留主体引用。
