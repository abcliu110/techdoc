# 审计日志与操作留痕模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

审计日志与操作留痕模型的本质，是系统的不可篡改黑匣子。它记录谁在什么时间、什么地点、对什么对象、做了什么动作，以及数据从什么变成什么。

核心抽象：

```text
操作审计 -> 数据审计 -> 访问审计 -> 登录审计 -> 防篡改存储 -> 审计查询留痕
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| audit_operation 操作审计 | trace_id, operator_id, operator_type(USER/ADMIN/SYSTEM/OPENAPI), on_behalf_of 代操作, op_time, ip, biz_type, biz_id, action, action_detail, result(SUCCESS/FAILED/DENIED), tenant_id | 五要素主表；DENIED/SYSTEM/代操作同样入册，只增不改 |
| audit_data_change 数据审计 | operation_id, table_name, record_id, change_type(INSERT/UPDATE/DELETE), before_data, after_data, changed_fields | 字段级改前改后证据；敏感字段脱敏后入册，挂操作审计 |
| audit_access 访问审计 | operator_id, 被访问主体 id, resource_type, field_category 字段类别, 用途, export_rows/export_condition(批量导出), access_time | 记"看"而非"改"；敏感资源注册触发，批量导出记条数与条件 |
| audit_login 登录审计 | user_id, event_type(登录/登出/失败/锁定/改密/OAuth授权), session_id, ip, device_info, result, event_time | 认证事件全记；失败连续多次是风控信号源，session_id 关联操作审计 |
| audit_action_dict 动作字典 | action_code(唯一), 含义, 所属域, detail_schema, 敏感级别, retention_period 保留期 | action 注册制：编码、脱敏与保留策略的唯一定义处 |
| audit_hash_chain 哈希链批次 | batch_no, prev_batch_hash, batch_hash, record_range, sealed_at, 外存凭证 | 按批哈希链+外部存证，让删改一条在密码学上可发现 |
| audit_archive_log 归档与销毁记录 | 数据类别, 归档范围, 目标存储, 操作人, archived_at, destroyed_at | 冷热分层与到期销毁的留痕，销毁本身也是审计事件 |
| audit_query_log 审计查询留痕 | 查询人, 查询条件, 命中范围, query_time | 审计的审计：防止内鬼用审计系统踩点 |

### 2.2 实体关系

```text
audit_operation (1) ──< audit_data_change (N)   一次操作产生多条字段变更
audit_operation (N) >── trace_id ──< audit_access/audit_login  同一请求靠 trace_id/session_id 串链
audit_action_dict (1) ──< audit_operation (N)   action 编码引用字典（注册制）
audit_hash_chain (1) ──< audit_operation (N)    批次封存范围覆盖记录区间
audit_archive_log (1) ──< 各审计表 (N)          归档/销毁按表按期执行留痕
audit_query_log ──> 各审计表                    查询行为单向留痕，不回写
外部引用：operator_id/on_behalf_of → 用户域；biz_type/biz_id → 各业务域单据；
         tenant_id → 租户域；登录失败事件 → 风控域（实时推送）
        （审计库与业务库物理隔离，引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键：action_code 字典唯一；(batch_no) 哈希链批次唯一；
       幂等键 trace_id + action + biz_id 防同一请求重复入册
只增不改：审计表对应用账号只授 INSERT/SELECT，无 UPDATE/DELETE；
        管理员操作审计库本身也被另一层审计
写入通道分级：资金类与业务变更同事务或走 Outbox（丢一条都不行）；
           普通操作异步；访问类批量缓冲——通道分级，五要素结构不分裂
脱敏纪律：before/after 中手机号/卡号等按 action_dict 敏感级别脱敏后存储
tenant_id 纪律：全部审计记录带 tenant_id，租户只能查询本租户审计
保留期：按 action_dict 的法规级别配置（资金 5~7 年），到期销毁留销毁记录
```

## 3. 核心张力

最大张力是：

```text
全量留痕诉求 vs 存储成本和隐私风险
```

企业希望所有关键行为都可追溯，但审计数据量巨大，且审计日志本身也可能包含敏感信息。

## 4. 不变量

- 审计记录只增不改。
- 操作审计必须有五要素。
- DENIED、SYSTEM、代操作也必须记录。
- 数据审计要记录改前改后。
- 敏感字段进入审计前要脱敏。
- 审计库应与业务库隔离。
- 审计查询本身也要留痕。

## 5. 为什么这样设计

操作审计回答“谁做了什么”，数据审计回答“值怎么变了”，访问审计回答“谁看过敏感数据”，登录审计回答“谁进入过系统”。四者分离，是因为写入频率、查询方式、保留期和合规要求不同。

## 6. 失败模式

- 只记成功操作，不记越权尝试。
- 只记动作，不记改前改后值。
- 审计日志可被应用账号 UPDATE 或 DELETE。
- 审计里存明文手机号、证件号，成为二次泄露源。
- 批量导出不留痕，拖库无法追踪。

## 7. 适用边界

适用：

- ERP、SaaS、金融、支付、CRM、权限、价格、账务、合同、审批系统。

可简化：

- 非敏感小工具可只保留操作审计，但涉及资金、权限、客户数据必须完整审计。

## 8. 设计取舍

优先保证：

- 责任可追溯。
- 不可篡改。
- 合规可证明。
- 事故可还原。

牺牲：

- 存储成本。
- 写入开销。
- 查询治理成本。

## 9. 同一模型的多业务形态建模

审计模型要同时支撑多种留痕形态，核心方法是：**统一五要素骨架 + 四类分治 + 差异下沉到写入时机与保留策略**。

### 9.1 统一骨架（所有形态不变的部分）

```text
四类分治（操作/数据/访问/登录）→ 五要素（谁/何时/何地/对象/动作）→ 只增不改 → 敏感脱敏 → 审计库隔离
不变量全集：只增不改、五要素齐全、DENIED 也记录、改前改后、审计查询留痕
```

骨架决不为某类审计特化——任何形态的"谁在什么时间对什么做了什么"都用同一套五要素回答。

### 9.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 后台操作审计 | 量大、异步可接受 | 写 audit_operation，拦截器统一采集五要素，异步落审计库（标准形态，骨架原型）；保留期按内控要求配置 |
| 资金动作审计（同事务） | 不允许丢一条 | 写入与业务变更同库同事务落 audit_data_change（改前改后值），事后异步搬运到审计库；不走可能丢失的异步通道 |
| 隐私访问审计 | 记录"看"而非"改"、量极大 | 独立访问审计表：记录访问者+被访问主体+字段类别+用途；批量导出记条数和条件而非逐行；保留期按隐私法配置 |
| 登录安全审计 | 关注失败与异常、驱动风控 | 独立登录审计表：成功/失败/锁定都记，含 IP、设备指纹；失败事件实时推送风控，而非仅落库 |
| OpenAPI 调用审计 | 主体是应用凭证、量巨大 | 操作者=app_id+凭证指纹而非用户；只记写操作全量+读操作抽样/聚合；限流拒绝（DENIED）必记 |

### 9.3 建模决策规则

- **能用四类之一 + 保留策略解决的，不建新审计表**：差异只在采集时机、主体类型、保留期时，五要素结构一套。
- **按写入一致性要求垂直分通道**：资金类同事务写、普通操作异步写、访问类可批量缓冲——通道分级，结构不分裂。
- **记录内容允许按形态裁剪但不新增语义**：访问审计不记改前改后，但不得省略五要素中的任何一项。
- **拆独立域的信号**：当留痕需要业务状态机和处理流程（工单化的合规事件、风控案件），那是案件域，审计只是它的证据来源。
