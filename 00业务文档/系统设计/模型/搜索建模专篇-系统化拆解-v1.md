# 搜索建模专篇 - 系统化拆解 v1

## 1. 本质模型

搜索建模的本质，是把规范化业务数据转换成面向查询的派生副本。数据库是权威源，搜索索引是为了快速查找、筛选、排序和展示而重塑的读模型。

核心抽象：

```text
权威源 DB -> 变更事件/CDC -> 文档构建 -> 搜索索引 -> 查询回源校验 -> 重建恢复
```

## 2. 逻辑模型

搜索域的特殊性：索引本身是派生副本不落 DB，DB 侧要建模的是"喂养索引的管道"和"搜索运营数据"。

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| srch_index_def 索引定义 | index_name, 文档粒度(SPU/订单/店铺), mapping 版本, 字段鲜度分级配置, 当前别名指向, status | 索引的元数据权威源；粒度=结果粒度，mapping 变更即重建 |
| srch_change_mapping 变更影响映射 | 源表名, 变更字段, 目标 index_name, 文档定位规则, 更新方式(整文档/partial), 优先级 | 显式配置"哪个表变了影响哪些文档"，避免隐式风暴 |
| srch_sync_task 同步任务 | 任务ID, index_name, 变更事件ID, doc_id, doc_version 文档版本, status(待处理/成功/失败/死信), retry_count | 增量同步的执行与重试证据；版本号防乱序覆盖 |
| srch_rebuild_job 重建任务 | index_name, 新旧索引名, 快照位点, 增量追平位点, 校验结果(条数/抽样), 切换时间, status | 双索引+别名切换的全过程留痕，可回滚 |
| srch_synonym 同义词表 | 词组ID, 同义词组, 类型(同义/纠错), 生效状态, 版本, 操作人 | 词表是数据不是配置文件；改动灰度可回滚 |
| srch_intervention 搜索干预 | 干预词, 干预类型(置顶/加权/屏蔽), 目标文档ID, 生效起止, 版本, operator | 回答"为什么这个商品排第一"的运营留痕 |
| srch_hot_word 热搜词 | 词, 来源(离线统计/人工), 审核状态, 生效区间, 排序权重 | 双通道产出，敏感词过滤后才可见 |
| srch_query_log 查询日志 | 查询词, index_name, 结果数, 点击文档ID, user_id, 时间 | 效果分析与无结果词运营的数据基础；进数仓不进业务库 |

### 2.2 实体关系

```text
srch_index_def (1) ──< srch_change_mapping (N)   一个索引接收多张源表的变更
srch_index_def (1) ──< srch_sync_task (N)        增量同步按索引分流
srch_index_def (1) ──< srch_rebuild_job (N)      重建历史只增
srch_index_def (1) ──< srch_intervention (N)     干预按索引配置
srch_synonym / srch_hot_word ──> 搜索引擎          词表版本化下发
外部引用：文档内容回查商品/价格/库存/订单等权威域（构建期拼装，值不回写）；
         doc_id → 各域主键（sku_id/spu_id/order_no）；user_id → 会员域
         （引用不落外键，应用层校验；索引是镜子，DB 是实物）
```

### 2.3 关键约束与不变量落点

```text
唯一键：index_name 唯一；(index_name, doc_id) 同步任务在途去重；
       同义词组内词条唯一，防循环映射
版本化规则：文档带 doc_version（源数据更新时间/版本号），
           索引写入用外部版本控制，旧版本写入被拒（乱序防护）；
           mapping/词表/干预修改发新版本，留生效记录
鲜度分级（按索引配置）：上下架/删除 秒级最高优先级；
           价格/有无库存 partial update 快道；描述 分钟级；统计 T+1
兜底约束：删除同步 = CDC 删除事件 + 每日全量比对（防幽灵文档）；
         价格/库存/上架状态下单前强制回源校验，精确库存数不进索引
重建约束：新索引构建 = DB 快照 + 增量追平，校验通过才切别名，
         旧索引保留回滚窗口；全量取数走只读副本，不打业务库
```

## 3. 核心张力

最大张力是：

```text
查询体验 vs 数据新鲜度和准确性
```

搜索希望快、宽、可筛选、可排序；交易系统要求价格、库存、上下架等关键数据准确。索引作为副本天然有延迟。

## 4. 不变量

- 文档粒度必须等于搜索结果粒度。
- 索引是派生副本，DB 是权威源。
- 索引必须可全量重建。
- 删除和下架必须有兜底同步。
- 不同字段要按鲜度分级。
- 交易关键数据必须回源校验。
- 搜索运营规则必须版本化留痕。

## 5. 为什么这样设计

索引文档为查询场景建模，是为了避免每次搜索跨商品、价格、库存、评价、营销等多表实时拼接。CDC 和版本控制是为了让索引能跟随权威源变化，同时防止乱序覆盖。

## 6. 失败模式

- 直接用数据库结构建索引，搜索结果字段不完整。
- 已下架商品仍在索引里，形成幽灵商品。
- 价格库存进索引后不回源，下单价格错误。
- 索引无法全量重建，事故后无法恢复。
- 运营干预无记录，无法解释排序结果。

## 7. 适用边界

适用：

- 商品搜索、订单搜索、客户搜索、合同搜索、知识库检索、后台复杂筛选。

可简化：

- 小数据量后台列表可直接用数据库查询，但一旦需要多条件筛选、全文检索、排序特征，就应引入搜索读模型。

## 8. 设计取舍

优先保证：

- 查询速度。
- 筛选体验。
- 搜索可运营。
- 索引可重建。

牺牲：

- 强一致性。
- 同步链路简单性。
- 存储成本。

## 9. 同一模型的多业务形态建模

搜索场景差别巨大，但支撑它们的骨架只有一套：**文档设计 + CDC 同步 + 分级鲜度，场景差异下沉到文档模型和鲜度分级参数**。

### 9.1 统一骨架（所有场景不变的部分）

```text
权威源 DB → CDC/变更事件 → 文档构建（粒度=结果粒度）→ 独立索引 → 查询（关键数据回源校验）→ 全量重建通道
不变量全集：DB 是权威源、索引可重建、删除下架有兜底、字段按鲜度分级、运营规则版本化
```

骨架决不为某个场景特化——任何场景的"文档怎么来、怎么保持新鲜、坏了怎么重建"都用同一条链路回答。

### 9.2 各场景差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 商品搜索 | 多表聚合宽文档、排序特征多、交易数据敏感 | 文档聚合商品+价格+库存+销量+评价（标准形态，骨架原型）；价格库存标注"低鲜度容忍"，下单前强制回源 |
| 订单搜索（客服） | 按买家/手机号/单号精确找、含敏感信息 | 文档粒度=订单主单，行信息作嵌套字段；手机号等敏感字段只存脱敏值+加密精确匹配字段，查询走权限过滤；鲜度要求高，CDC 直连订单库秒级同步 |
| 内容搜索 | 长文本、分词和相关性主导、更新少 | 文档主体是正文分词字段 + 标题加权；无需回源校验（内容无交易属性）；同步允许分钟级，重建成本低可全量重刷 |
| 店铺搜索 | 数据量小、地理位置排序 | 文档带 geo_point 字段做距离排序；店铺状态（营业/关闭）是高鲜度字段走实时更新，其余字段 T+1 即可 |
| 后台管理搜索 | 多条件组合筛选、要求与库一致、量小 | 文档=单表投影，不做跨表聚合；鲜度要求"写后可见"，用同步双写或事务后即时刷新；数据量小到一定程度直接退化为 DB 查询 |

### 9.3 建模决策规则

- **一个结果粒度一个索引，不共用文档模型**：商品、订单、店铺各建索引，文档结构为各自查询场景设计，禁止"一个大索引装所有实体"。
- **鲜度分级是每个索引自己的配置，不是全局值**：同一索引内价格（回源）、库存（准实时）、评价数（T+1）可以不同级；机制统一，参数各异。
- **敏感场景在文档构建期脱敏，不在查询期补救**：客服订单搜索的隐私字段进索引前就处理，索引泄露不等于明文泄露。
- **拆独立链路的信号**：当查询需要的不是"找到文档"而是语义理解与生成（如智能问答、向量召回），检索模型和评估方式都不同，应独立建向量检索链路，与倒排索引并行而非混装。
