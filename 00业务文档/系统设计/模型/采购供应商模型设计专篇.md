# 采购供应商模型设计专篇

> 本篇聚焦供应商、采购、收货、退供、应付和采购价格。采购模型连接商品、库存、资金账务、财务和供应链协同。

## 0. 一句话总纲

> 采购模型的核心，是把“向谁买、买什么、什么价、何时到、是否合格、是否应付”变成可追踪链路。

## 1. 供应商主数据

供应商不仅是名称和电话，还包括：

- 供应商编码。
- 资质。
- 合同。
- 账期。
- 结算方式。
- 供货范围。
- 价格表。
- 税率。
- 开票信息。
- 联系人。

供应商变更要有审核和历史记录。

## 2. 采购链路

标准链路：

```text
采购申请 -> 采购订单 -> 到货通知 -> 收货 -> 质检 -> 入库 -> 应付 -> 付款
```

不同企业可简化，但模型要知道哪些环节被合并，不能用一张采购单表达所有状态。

## 3. 采购订单和采购行

采购订单主表记录供应商、组织、仓库、币种、税率、状态。采购行记录 SKU、数量、单价、税额、预计到货时间、已收数量、已退数量。

行级很重要，因为部分收货、部分退货、部分开票、部分付款都在行上发生。

## 4. 收货、质检和入库

收货不等于入库：

- 收货：供应商送到了。
- 质检：是否合格。
- 入库：进入可用或待检库存。

标准做法：

```text
收货单 -> 质检单 -> 入库单
```

低复杂度系统可以合并，但要明确是否保留质检和差异记录。

## 5. 应付和付款

采购应付来源可能是：

- 采购订单。
- 收货单。
- 入库单。
- 发票。

标准做法应明确三单匹配：

```text
采购订单数量/价格
收货或入库数量
供应商发票金额
```

匹配后生成应付，付款冲减应付。

## 6. 评审清单

```text
1. 供应商主数据是否包含资质、账期、结算、开票和联系人？
2. 采购申请、订单、收货、质检、入库、应付、付款是否边界清楚？
3. 采购行是否支持部分收货、部分退货、部分开票？
4. 收货和入库是否区分？
5. 质检不合格是否有退供或报损路径？
6. 三单匹配是否支持？
7. 采购价格、税率、币种是否有历史快照？
```

## 7. 总结

采购供应商建模五条纪律：

1. **供应商是主数据**：资质、账期、结算、开票都要治理。
2. **采购行是颗粒**：收货、退货、开票、付款都离不开行。
3. **收货不等于入库**：实物到达、质检合格、库存可用是三件事。
4. **应付要有来源**：订单、收货、发票必须能匹配。
5. **价格税率快照**：历史采购成本必须可解释。
