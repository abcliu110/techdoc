# 国际化与多币种模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦国际化建模。出海不是"把界面翻译一下"：多币种让每个金额字段都要带币种和汇率快照，多时区让"今天"变成危险词，多语言让每段文案变成一组版本，多法域让税费和合规按国家分叉。国际化改造最贵的部分全在数据模型——这正是要在设计期埋好接口的原因。

## 0. 一句话总纲

> 国际化模型四根柱子：金额永远带币种、汇率下单即固化、时间存 UTC 按时区解释、内容与文案键值化多语言——四者都遵循同一底层思想：记录事实时固化"当时的上下文"。

## 1. 多币种：金额 = 数值 + 币种，缺一即错

### 1.1 铁律

- **每个金额字段旁边必须有币种字段**（ISO 4217 三字码），没有币种的金额是无意义的数字。同一张表出现多个金额时明确各自币种（或共享一个币种字段并写清约束）。
- **最小单位与精度按币种驱动**：JPY 无小数、KWD 三位小数、USD 两位——精度不是常量是**币种元数据表**（币种、小数位、舍入规则、符号、展示格式）。计算保留高精度（DECIMAL(18,6)），**对外呈现/入账时按币种精度舍入，舍入差额有去处**（分摊尾差纪律，订单篇同款）。
- 禁止在代码里硬编码 `×100 转分`——JPY 会错，改用币种元数据的 `minor_unit`。

### 1.2 交易币、结算币、记账币三分

```text
交易币（transaction currency）：用户看到并支付的币种（日本用户付 JPY）
结算币（settlement currency）：与渠道/商家结算的币种（渠道结 USD）
记账币（functional currency） ：公司账本币种（总部记 CNY）
```

一笔交易三个币种三个金额同时落库，各自带汇率与换算时点——审计要求任何金额能沿"交易币 → 结算币 → 记账币"链条追溯换算依据。

## 2. 汇率：版本化的事实源

```sql
CREATE TABLE fx_rate (
  id           BIGINT PRIMARY KEY,
  base_ccy     CHAR(3) NOT NULL,
  quote_ccy    CHAR(3) NOT NULL,
  rate_type    VARCHAR(16) NOT NULL,  -- PRICING标价用/SETTLE结算用/ACCOUNTING记账用
  rate         DECIMAL(18,8) NOT NULL,
  valid_from   DATETIME NOT NULL,     -- 生效起（左闭右开区间）
  source       VARCHAR(32) NOT NULL,  -- 汇率来源（央行/路透/渠道）
  created_at   DATETIME NOT NULL,
  KEY idx_pair (base_ccy, quote_ccy, rate_type, valid_from)
);
```

- **汇率只增不改**（双时态思想）：新汇率插入新区间，"3 月 5 日下单用的什么汇率"永远可查。
- **用途分型**：标价汇率（可含加成，商业策略）、结算汇率（渠道实际给的）、记账汇率（财务口径，常为月度固定价）——三者混用是汇兑损益算不清的根源。
- **订单固化汇率快照**：下单时的交易币金额、汇率、换算结果写死在订单上；退款按**原汇率退**（用户付了多少交易币就退多少），汇率波动产生的差额进**汇兑损益科目**（账务篇内部户思想）——汇差不是误差，是要单独核算的科目。

## 3. 时间与时区：UTC 存储、时区解释

- **存储一律 UTC**（总览篇物理底线），展示按用户/门店时区渲染——这在国际化下从"最佳实践"升级为"不这样做必错"。
- **"营业日"是本地概念**：日报表、日切、"今日订单"按**业务主体时区**（门店所在时区/站点时区）划界，报表表带 `biz_date`（按主体时区算好的业务日期）而不是让查询方各自换算。
- 关键实体（门店、仓库、站点、租户）**必须有时区属性**；促销活动的起止时间要明确"哪个时区的 0 点"（全球活动常用各站点本地时间生效——模型上是站点 × 活动的生效时间展开）。
- 夏令时：用 IANA 时区名（America/New_York）不用固定偏移（UTC-5），换算交给时区库。

## 4. 多语言内容：键值化与翻译工作流

两类多语言，方案不同：

**系统文案**（按钮、提示、邮件模板）：资源键值方案，`key × locale → text`，随代码/配置管理，缺失语言回退链（zh-HK → zh-TW → zh → 默认）。

**业务数据**（商品标题、类目名、品牌故事）：**翻译表模式**：

```sql
CREATE TABLE prd_spu_i18n (
  id          BIGINT PRIMARY KEY,
  spu_id      BIGINT NOT NULL,
  locale      VARCHAR(10) NOT NULL,    -- BCP 47：en-US, ja-JP
  title       VARCHAR(255) NOT NULL,
  selling_point VARCHAR(512),
  trans_status VARCHAR(16) NOT NULL,   -- MACHINE机翻/REVIEWED人审/NATIVE原生
  version     INT NOT NULL,            -- 源内容版本（源变了译文要重审）
  UNIQUE KEY uk_spu_locale (spu_id, locale)
);
```

- 主表存**源语言**内容 + 源语言标记，翻译表存各语言版本；查询按 locale 取、缺失回退源语言（回退要显式标记，避免日文站默默出现中文标题还没人知道）。
- **翻译状态与源版本联动**：源内容更新后译文标记"待重审"——过期译文是国际站最常见的内容事故。
- 搜索按语言分索引（分词器不同），排序规则（collation）按语言（德语 ä、瑞典语 å 的排序规则不同）。

## 5. 多法域：税、合规、地址按国家分叉

- **税费模型独立**：税种（VAT/GST/销售税）× 法域 × 类目税率表，带生效区间；订单行固化**税额与税率快照**、含税/不含税标价模式按站点配置（欧洲含税展示、美国税后加）——税是订单金额恒等式的一等成员，不是运费式的附加项。
- **地址模型弹性化**：各国地址结构不同（日本邮编定位到街区、有些国家无邮编），用"结构化字段（国家/一级区划/邮编）+ 弹性行（address_line1/2）"，各国校验规则配置化；行政区划表按国家维护、带版本。
- **合规分叉是配置不是分支**：实名要求、发票制度（金税/欧洲 VAT invoice）、退货法定期限（欧盟 14 天无理由）、数据驻留（数据必须存在本国）——按站点/法域配置驱动，代码里不出现 `if (country == "DE")`。
- 数据驻留要求严格的市场（如部分国家个人数据不出境）会把架构推向**分区域部署**：全球共享的（商品主数据）与区域私有的（用户、订单）在模型上明确划分，跨区同步只走脱敏/聚合数据。

## 6. 站点：国际化的组织单元

多数出海架构以**站点（site/market）**为经营单元：站点 = 法域 × 语言集 × 币种 × 时区 × 合规配置的组合。商品在哪些站点上架（渠道级上下架的扩展）、价格按站点定价（价格域的站点维度）、库存按站点/区域仓分配（渠道共享库存的扩展）——站点是把前述所有国际化维度收口的配置实体。

## 7. 国际化评审清单

```text
1. 每个金额是否带币种？精度与舍入是否由币种元数据驱动？
2. 交易/结算/记账三币是否分离、各带汇率快照、换算链可追溯？
3. 汇率是否只增不改、按用途分型？订单是否固化汇率？退款是否原汇率+汇差入科目？
4. 时间是否 UTC 存储？业务日是否按主体时区预算好？主体是否都有时区属性？
5. 文案是否键值化带回退链？业务数据是否翻译表+源版本联动+过期标记？
6. 税是否法域×类目税率表、订单行固化税额？含/不含税模式是否站点配置？
7. 地址是否弹性结构+分国校验？合规差异是否配置驱动而非代码分支？
8. 数据驻留是否明确全球共享与区域私有边界？
9. 站点是否作为收口配置实体（币种/语言/时区/合规/价格/上架）？
```

## 8. 总结

国际化建模五条纪律：

1. **金额带币种**：数值离开币种即无意义，精度舍入按币种元数据。
2. **汇率即快照**：三币三率下单固化，汇差是科目不是误差。
3. **UTC 加时区**：存储统一、解释本地，业务日预先算好。
4. **翻译是版本**：源变译废、缺失回退、状态可见。
5. **法域即配置**：税、地址、合规按站点配置分叉，代码保持一份。

> 国际化的本质是把"隐含的默认"全部显式化：默认的币种、默认的时区、默认的语言、默认的法律。模型里每消灭一个隐含默认，出海就少踩一个国家的坑。
