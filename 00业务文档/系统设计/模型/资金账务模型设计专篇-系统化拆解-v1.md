# 资金账务模型设计专篇 - 系统化拆解 v1

## 1. 本质模型

资金账务模型的本质，是一套“价值流动的证据系统”。它不只是记录余额，而是记录每一次价值从哪里来、经过哪里、到哪里去，以及这次流动为什么发生、由谁触发、是否可审计、是否可对账。

核心抽象：

```text
业务单据 -> 记账凭证 -> 借贷分录 -> 账户流水 -> 余额快照 -> 对账差异
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| acct_account 账户 | account_no, owner_type(USER/MERCHANT/PLATFORM), owner_id, account_type, currency, balance, frozen, balance_dir 余额方向, version | 当前值层，是流水的派生；可用=balance-frozen 不落库 |
| acct_voucher 记账凭证 | voucher_no, biz_type, biz_order_no, idempotent_key(唯一), amount, currency, acct_date(日切决定) | 一次业务动作一张凭证，幂等的物理保证 |
| acct_entry 分录 | voucher_id, account_id, direction(借/贷), amount(恒为正), currency, acct_date | 面向凭证的视角：借贷合计为零 |
| acct_flow 账户流水 | account_id, entry_id, voucher_no, change_amount(带符号), balance_after, acct_date | 面向账户的视角；balance_after 链式对账 |
| acct_freeze 冻结记录 | 冻结单号(唯一), account_id, amount, 过期时间, status(冻结/已确认/已释放) | 冻结不是记账；支撑三段式与超时释放 |
| acct_entry_rule 记账规则 | biz_type, 借方户类型, 贷方户类型, 金额来源/比例, rule_version, 生效区间 | 记账规则配置化+版本化，新业务零改核心 |
| acct_daily_snapshot 日终快照 | account_id, 账务日, 期初余额, 借方发生额, 贷方发生额, 期末余额 | 日切冻结后不可变；历史余额免回放 |
| recon_batch 对账批次 | (recon_date, channel)(唯一), bill_file, 我方笔数/金额, 渠道笔数/金额, diff_count, status | 每日渠道对账的执行单元 |
| recon_diff 差异明细 | batch_id, diff_type(长款/短款/金额不符), 我方/渠道单号与金额, handle_status, handle_way, handler | 长短款处置队列，差错户趋零的抓手 |

### 2.2 实体关系

```text
acct_voucher (1) ──< acct_entry (N)       一凭证 2~N 条分录，借贷合计为零
acct_account (1) ──< acct_entry (N)       分录动账户
acct_entry (1) ── acct_flow (1)           每条分录产生一条账户流水
acct_account (1) ──< acct_flow (N)        账户视角流水只增
acct_account (1) ──< acct_freeze (N)      冻结记录独立于分录
acct_entry_rule (1) ──< acct_voucher (N)  凭证按规则模板生成（rule_version 固化）
acct_account (1) ──< acct_daily_snapshot (N)  日终快照按账务日
recon_batch (1) ──< recon_diff (N)        差异明细挂批次
冲正：acct_voucher (1) ──< 冲正凭证 (N)   镜像凭证回指原凭证号，累计冲正 ≤ 原额
外部引用：biz_order_no → 订单/支付/结算等业务域；channel → 支付网关域；
         财务科目映射 → 财务总账（引用不落外键，应用层校验）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：voucher idempotent_key(=业务单号+动作) 唯一索引；
             account (owner_type, owner_id, account_type, currency) 唯一，
             开户 INSERT IGNORE 幂等；冻结单号唯一
留量校验：扣减原子 UPDATE balance = balance + ?
         WHERE id = ? AND balance + ? >= frozen（余额检查与扣减同一条 SQL）；
         冻结校验 balance - frozen >= X；累计冲正/退款 ≤ 原凭证金额
恒等式：每张凭证 Σ借 = Σ贷；全局 Σ所有账户余额(按方向) = 0；
       逐户 Σ流水变动 = 余额变化；balance_after 链条连续；
       任一失败即冻结日切并告警
状态机约束：凭证/分录/流水只 INSERT 不 UPDATE/DELETE，纠错走冲正凭证；
           快照出具后不可变，改错重出新版本；
           多账户更新按 account_id 排序加锁防死锁
分片键：account_id（记账主路径）；跨库记账经在途户拆两个本地凭证，
       在途户余额长期不为零即告警
```

## 3. 核心张力

资金账务最大的张力是：

```text
业务灵活性 vs 账务确定性
```

业务希望快速增加充值、支付、退款、分账、结算、储值、积分、提现等玩法；账务系统必须保证每一次价值变化都守恒、可追溯、可冲正、可对账。

因此设计上必须牺牲一部分简单性，换取账务可信度。

## 4. 不变量

资金账务模型必须守住以下不变量：

- 每一笔价值变化必须有凭证。
- 每一张凭证借贷必须平衡。
- 凭证、分录、流水只增不改。
- 错误必须通过冲正或红字处理。
- 当前余额必须能被流水解释。
- 重复请求不能重复记账。
- 渠道账、业务账、账务账必须能对账。

破坏这些不变量的后果，是资金错账、用户余额错误、商户结算不准、财务无法审计。

## 5. 为什么这样设计

三层分离是为了解决“业务发生、价值记账、真实资金流动”不是同一件事的问题。

```text
交易层：订单、退款、充值、提现等业务事实
账务层：账户、凭证、分录、流水等价值转移
资金层：银行、微信、支付宝等真实资金渠道
```

内部户是为了让每一分钱都有位置。没有渠道待清算户、在途户、差错户、手续费户，系统只能知道“用户多了 100”，却说不清这 100 从哪里来。

## 6. 失败模式

常见失败模式：

- 只存余额，不存流水，导致无法追溯。
- 只记业务单，不记账务凭证，导致财务无法核算。
- 直接修改历史凭证，导致审计链断裂。
- 没有幂等键，重复回调造成重复入账。
- 没有差错户，对账差异只能人工改数。
- 业务账和渠道账不分，支付成功但渠道未结算时无法表达。

## 7. 适用边界

适用：

- 支付、储值、钱包、积分、分账、结算、提现、商户账务。

不适用：

- 只做简单展示、不涉及价值变化的小型功能。

但只要涉及钱、类钱、权益余额，哪怕系统规模小，也应至少保留流水、幂等和冲正思想。

## 8. 设计取舍

优先保证：

- 守恒。
- 可追溯。
- 可对账。
- 可冲正。
- 可审计。

牺牲：

- 表结构简单性。
- 开发速度。
- 查询直观性。
- 运维复杂度。

这是账务系统必须接受的代价。

## 9. 同一模型的多业务形态建模

资金账务要同时支撑多种业务形态，核心方法是：**统一账本骨架 + 账户类型扩展 + 记账规则配置化**。业态变化落在"开什么户、配什么记账规则"上，而不是改账本结构。

### 9.1 统一骨架（所有业态不变的部分）

```text
账户（account_type 标识户别）→ 记账凭证 → 借贷分录 → 账户流水 → 余额快照 → 对账差异
不变量全集：借贷平衡、只增不改、冲正红字纠错、幂等防重、余额可由流水解释
```

骨架决不为某个业态特化——任何业态的"这笔钱从哪来、到哪去、为什么动"都用同一套凭证分录回答。

### 9.2 各业态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 电商平台账务 | 买家付款先入平台，再结算给商家 | 增设内部户（渠道待清算户、平台应付户）；支付成功/确认收货各配一条记账规则模板，钱始终有位置 |
| 储值余额 | 充值、消费、退回储值，本金与赠送金规则不同 | 用户储值账户（负债类）+ 冻结/解冻/扣减三段式；本金和赠送金拆子账户，消费按扣减顺序规则出分录 |
| 商家结算 | 账期、周期出账、结算前可调整 | 商家应付账户挂账；结算单作为业务单据驱动出账凭证；依赖日切保证结算周期边界清晰 |
| 分销佣金 | 多级计提、退款要回退佣金、提现 | 佣金应付账户 + 计提凭证；退款触发红字凭证回退佣金；提现走独立单据+冻结三段式 |
| 多币种账务 | 同一主体多币种余额、汇兑损益 | 账户粒度带币种（一户一币）；跨币种交易拆两张凭证 + 汇兑损益户吸收差额；汇率快照进凭证 |
| 营销费用账 | 券/红包成本要分清平台承担还是商家承担 | 营销费用户按承担方拆分；核销时按承担比例出分摊分录，费用直接落到费用户而非隐没在收入里 |

### 9.3 建模决策规则

- **新业态先问"钱从哪个账户到哪个账户"**：能用新增账户类型 + 新记账规则配置回答的，不新建账务体系。
- **记账规则配置化，不在业务代码里散写借贷**：每个业务事件对应一条 entry_rule 模板（借方户、贷方户、金额来源），业态扩展就是加模板。
- **差异优先用内部户吸收**：待清算户、在途户、差错户、费用户是业态差异的缓冲区，不允许业态差异侵入分录结构。
- **拆独立账本的信号**：当守恒规则或记账本位本身不同（如证券持仓按份额、积分非货币权益），才拆独立账本，不是账务的一种账户类型。

```yaml
type: model_decomposition

```
