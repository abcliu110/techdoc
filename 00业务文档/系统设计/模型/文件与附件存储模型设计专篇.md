# 文件与附件存储模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦文件建模。商品图片、合同扫描件、报表导出、发票 PDF、审批附件——每个业务域都产生文件。文件的建模难点：二进制在对象存储、元数据在 DB，两边要保持一致；引用满天飞，谁也不敢删；外链一泄露，权限全落空。

## 0. 一句话总纲

> 文件模型三分离：二进制（对象存储）、文件元数据(DB)、业务引用关系(DB)。文件本体以内容哈希去重且不可变，删除由引用计数与生命周期策略驱动，访问永远走带权限校验的临时授权。

## 1. 三层结构

```text
对象存储（OSS/S3）   —— 二进制本体：key = 内容哈希或规范路径
文件元数据（file）   —— 一行一个文件：哈希、大小、类型、状态、存储位置
业务引用（file_ref） —— 谁在用：业务域 × 单据 × 用途 → 文件
```

```sql
CREATE TABLE sys_file (
  id            BIGINT PRIMARY KEY,
  file_key      VARCHAR(255) NOT NULL,  -- 对象存储 key
  content_hash  CHAR(64) NOT NULL,      -- SHA-256：去重与完整性校验
  size_bytes    BIGINT NOT NULL,
  mime_type     VARCHAR(64) NOT NULL,
  storage       VARCHAR(16) NOT NULL,   -- 存储集群/区域（数据驻留相关）
  status        VARCHAR(16) NOT NULL,   -- UPLOADING/ACTIVE/ARCHIVED/DELETED
  uploaded_by   BIGINT NOT NULL,
  tenant_id     BIGINT NOT NULL,
  created_at    DATETIME NOT NULL,
  UNIQUE KEY uk_hash_tenant (content_hash, tenant_id),  -- 租户内去重
  KEY idx_status (status, created_at)
);

CREATE TABLE sys_file_ref (
  id          BIGINT PRIMARY KEY,
  file_id     BIGINT NOT NULL,
  biz_type    VARCHAR(32) NOT NULL,   -- SPU_IMAGE/CONTRACT/APPROVAL_ATTACH/INVOICE...
  biz_id      VARCHAR(64) NOT NULL,   -- 业务单据标识
  purpose     VARCHAR(32),            -- 主图/详情图/正本/副本
  sort_order  INT NOT NULL DEFAULT 0,
  created_at  DATETIME NOT NULL,
  deleted_at  DATETIME,               -- 解除引用（软删）
  KEY idx_biz (biz_type, biz_id),
  KEY idx_file (file_id)
);
```

要点：

- **内容哈希去重**：同一文件多处上传只存一份（商品篇内容地址存储的通用化）；哈希同时是完整性校验（下载/迁移后比对）。
- **业务表不直接存 URL**：存 file_id（或经引用表），URL 是**访问时生成的**（见 §4）——存 URL 的系统在更换存储域名/迁移集群那天全库刷 URL。
- 引用关系是删除决策的唯一依据："这张图还有没有人用"查引用表，不是猜。

## 2. 上传协议：先登记、再直传、后确认

```text
1. 业务方向文件服务申请上传：登记元数据（UPLOADING 状态）+ 返回预签名直传地址
2. 客户端直传对象存储（不过应用服务器，大文件分片/断点续传）
3. 回调/确认：校验哈希与大小 → 状态置 ACTIVE
4. 超时未确认的 UPLOADING 记录由清理任务处理（对象存储侧同步清理碎片）
```

- **UPLOADING 是合法中间态**（与支付 UNKNOWN 同构）：直传成功但确认丢失，靠对账任务（扫描对象存储 vs 元数据）收敛。
- 上传即校验：类型白名单（MIME + 魔数双验，防伪装扩展名）、大小上限、图片安全扫描（涉黄涉政）异步进行，扫描不通过转 FROZEN 并通知业务方——安全扫描状态也是文件状态机的一部分。

## 3. 生命周期：删除是策略不是操作

- **业务删除只解除引用**（file_ref 软删），文件本体不动。
- **回收任务**：零引用且超过保护期（如 30 天）的文件转 ARCHIVED（低频存储）或 DELETED（真删）——删除前二次校验引用，删除动作留流水（谁的策略、何时删的、哈希是什么）。
- **法定保留优先于零引用**：合同、发票、审批附件按法规保留年限（配置在 biz_type 维度上），到期才进入可删池——"没人引用"不等于"可以删"。
- 冷热分层：按最近访问时间批量转低频/归档存储，元数据记录当前层级，取归档文件的"解冻延迟"要在产品层有预期管理。

## 4. 访问控制：URL 即授权

- **私有读是默认**，公开读是例外（商品图等纯公开资产单独 bucket + CDN）。
- 私有文件访问流程：业务方校验业务权限（这个人能不能看这份合同——权限篇数据权限）→ 文件服务签发**短时效预签名 URL**（分钟级过期）→ 客户端访问。**URL 短命化是防扩散的核心**：泄露的链接几分钟后失效。
- 敏感文件（合同、证件照）加**访问日志**（谁、何时、看了哪个文件）——与敏感数据查询留痕（总览篇 §29）同一纪律。
- 防盗链、防遍历：file_key 不可猜测（含随机段），禁止自增 ID 直接当 key。

## 5. 图片处理与派生文件

- 缩略图、水印图、格式转换产物是**派生文件**：以"源文件 + 处理参数"为键按需生成 + CDN 缓存（主流对象存储原生支持），**不逐个登记进元数据表**——派生可重算，只登记事实源。
- 需要固化的例外：合同盖章版 PDF、加水印后对外发出的正式文件——这些是新的"事实"（出具即证据，快照特例规则），独立登记并与源文件建关联。

## 6. 文件评审清单

```text
1. 二进制/元数据/引用是否三层分离？业务表是否存 file_id 而非 URL？
2. 是否内容哈希去重 + 完整性校验？
3. 上传是否预签名直传 + 确认机制 + 碎片对账清理？
4. 类型是否 MIME+魔数双验？安全扫描是否纳入状态机？
5. 删除是否引用驱动 + 保护期 + 法定保留期优先 + 删除留痕？
6. 私有读是否默认？URL 是否短时效签发、key 不可猜测？
7. 敏感文件是否有访问日志？
8. 派生文件是否按需生成不登记，正式出具文件是否固化登记？
9. 存储区域是否满足数据驻留要求（国际化篇）？
```

## 7. 总结

文件建模五条纪律：

1. **三层分离**：对象存本体、元数据记档案、引用表定生死。
2. **哈希为本**：内容寻址去重，完整性随时可验。
3. **删除走策略**：引用计数 + 保护期 + 法定保留，删前留痕。
4. **访问即授权**：私有默认、短时效 URL、敏感留访问日志。
5. **派生不入册**：可重算的按需生成，出具的固化为证据。

> 文件是最容易被当作"存上就完了"的数据。直到某天要回答"这份合同的原件在哪、谁看过、为什么被删了"，才发现文件也需要一本严肃的账。
