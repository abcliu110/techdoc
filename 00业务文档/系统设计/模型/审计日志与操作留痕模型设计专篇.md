# 审计日志与操作留痕模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦审计建模。前面每一篇都在说"留痕"——本篇回答留痕本身怎么建模：操作日志、数据变更轨迹、登录审计、敏感访问记录。审计数据的特殊性：写入方是全系统、读取方是审计与司法、生命周期以年计、而且**它自己必须不可篡改**——给系统记账的账本，不能被记账的人改。

## 0. 一句话总纲

> 审计模型四类分明：操作审计（谁做了什么）、数据审计（值从什么变成什么）、访问审计（谁看了什么）、登录审计（谁进来过）。全部只增不改、带完整五要素（谁/何时/何地/对什么/做了什么），存储与业务库隔离，保留期由法规而非磁盘决定。

## 1. 四类审计，四种粒度

| 类型 | 记什么 | 典型消费者 |
|---|---|---|
| 操作审计 | 业务动作：改价、退款审批、导出报表 | 内审、客诉定责 |
| 数据审计 | 字段级变更：改前值→改后值 | 数据纠纷、事故还原 |
| 访问审计 | 敏感读取：查了谁的手机号、看了哪份合同 | 隐私合规（总览篇 §29） |
| 登录审计 | 认证事件：登录/登出/失败/异地 | 安全、盗号排查 |

四类混在一张表是常见错误：写入频率差三个数量级（访问审计 >> 操作审计）、保留期不同、查询模式不同——**分表分策略**。

## 2. 操作审计：五要素结构

```sql
CREATE TABLE audit_operation (
  id           BIGINT PRIMARY KEY,
  trace_id     VARCHAR(64) NOT NULL,   -- 全链路追踪号（串起同一请求的多条审计）
  operator_id  BIGINT NOT NULL,        -- 谁：操作人
  operator_type VARCHAR(16) NOT NULL,  -- USER/ADMIN/SYSTEM/OPENAPI —— 系统动作也要留痕
  on_behalf_of BIGINT,                 -- 代操作：客服代用户操作时记两个人
  op_time      DATETIME(3) NOT NULL,   -- 何时（毫秒）
  ip           VARCHAR(45),            -- 何地
  device_info  VARCHAR(128),
  biz_type     VARCHAR(32) NOT NULL,   -- 对什么：对象类型
  biz_id       VARCHAR(64) NOT NULL,   --         对象标识
  action       VARCHAR(32) NOT NULL,   -- 做了什么：动作编码（注册制，见 §5）
  action_detail JSON,                  -- 动作参数快照
  result       VARCHAR(16) NOT NULL,   -- SUCCESS/FAILED/DENIED —— 被拒绝的尝试更要记
  tenant_id    BIGINT NOT NULL,
  KEY idx_biz (biz_type, biz_id, op_time),
  KEY idx_operator (operator_id, op_time)
);
```

要点：

- **DENIED 也是审计事件**：越权尝试被权限系统拦下，恰恰是最该记的一条——审计不只记"发生了什么"，也记"有人试图让什么发生"。
- **代操作双记名**（on_behalf_of）：客服后台代用户改地址，责任人是客服，影响对象是用户，两个都要能查到。
- **SYSTEM 是合法操作者**：定时任务改状态、风控自动冻结，operator_type=SYSTEM + 任务/规则标识——"这个状态谁改的？系统改的？哪个任务凭什么改的"必须可答。
- trace_id 贯穿：一次退款审批产生的操作审计、数据审计、通知发送，靠 trace_id 串成完整故事。

## 3. 数据审计：改前改后值

```sql
CREATE TABLE audit_data_change (
  id          BIGINT PRIMARY KEY,
  operation_id BIGINT NOT NULL,        -- 归属哪次操作（挂操作审计）
  table_name  VARCHAR(64) NOT NULL,
  record_id   VARCHAR(64) NOT NULL,
  change_type VARCHAR(8) NOT NULL,     -- INSERT/UPDATE/DELETE
  before_data JSON,                    -- 改前（INSERT 为空）
  after_data  JSON,                    -- 改后（DELETE 为空）
  changed_fields JSON,                 -- 变更字段清单（快速筛"谁改过价格"）
  KEY idx_record (table_name, record_id)
);
```

- **采集方式选型**：应用层拦截（能关联操作人与业务语义，首选）vs binlog 解析（无遗漏但缺业务上下文，兜底）——严肃场景两路并用，互为校验。
- **不是所有表都值得字段级审计**：按敏感度分级——资金、价格、权限、主数据配置必须记；高频流水表本身就是审计（只增不改的表不需要再审计自己）。
- 敏感字段（手机号、卡号）在 before/after 中**脱敏存储**——审计日志成为敏感数据的二次泄露源是经典事故。

## 4. 访问与登录审计

- **访问审计按策略触发**：不是记所有 SELECT（不可能也无意义），而是敏感资源注册制——标记为敏感的接口/字段（客户手机号明文查看、合同下载、批量导出）触发记录；**批量导出单独重点审计**（导了多少行、什么条件、文件去向），拖库往往从合法导出功能开始。
- **登录审计**：成功/失败/登出/踢出/改密/授权（OAuth 授权给第三方），失败连续多次是安全信号源；会话标识入审计，与操作审计关联（这次操作出自哪次登录）。

## 5. 动作注册制与接入纪律

- **action 编码注册制**（与指标字典、特征字典同构）：动作编码、含义、所属域、detail 结构、敏感级别、保留期——自由字符串的 action 三个月后没人知道 `updOrd2` 是什么。
- 接入方式优先**声明式**（注解/切面自动采集 + 业务补充语义），关键资金动作**编程式显式记录**并与业务事务同库同事务或走 Outbox——资金操作的审计丢一条都不行；一般操作审计允许异步（丢失容忍度按动作敏感级别定）。

## 6. 审计数据自身的治理

审计系统的元问题——账本自己怎么被保护：

- **只增不改在权限层强制**：审计表对应用账号只授 INSERT/SELECT，无 UPDATE/DELETE 权限；管理员账号操作审计库本身也要被（另一层）审计。
- **防篡改升级手段**（合规要求高时）：按批哈希链（每批记录含上批哈希）、定期摘要外存（写入 WORM 存储/第三方存证）——让"事后删改一条记录"在密码学上可发现。
- **存储与生命周期**：审计写入量巨大，热数据（近 3~6 月）在 DB 支撑在线查询，冷数据归档到廉价存储（对象存储 + 可检索格式）；**保留期按法规与动作级别配置**（资金操作常要求 5~7 年），到期销毁也要留销毁记录。
- 与业务库**物理隔离**（独立库/集群）：审计写入洪峰不能拖垮交易；业务库被入侵不等于审计被抹掉。
- 审计查询本身留痕：谁在查审计日志、查了谁——审计的审计，防止内鬼用审计系统踩点。

## 7. 审计评审清单

```text
1. 操作/数据/访问/登录四类是否分表分策略？
2. 操作审计是否五要素齐全？DENIED、SYSTEM、代操作是否覆盖？
3. trace_id 是否贯穿一次请求的所有审计记录？
4. 数据审计是否记改前改后值？采集是否应用层+binlog 双路？
5. 审计中的敏感字段是否脱敏？批量导出是否重点审计？
6. action 是否注册制（含义/结构/敏感级/保留期）？
7. 资金动作审计是否与业务事务同保障（同事务/Outbox）？
8. 审计表是否权限层禁改禁删？高合规场景是否有哈希链/外部存证？
9. 是否与业务库物理隔离？冷热分层与法定保留期是否落实？
10. 审计查询是否自身留痕？
```

## 8. 总结

审计建模五条纪律：

1. **四类分治**：操作、数据、访问、登录各有其表、各有其策略。
2. **五要素齐全**：谁、何时、何地、对什么、做了什么，拒绝和系统动作同样入册。
3. **改前改后**：数据审计记两个值，敏感字段脱敏后再入册。
4. **账本不可改**：权限禁改、哈希链防篡、隔离部署、依法保留。
5. **审计有字典**：动作注册制，审计的查询也被审计。

> 审计日志是系统的黑匣子：平时没人看它，出事时它是唯一在场的证人。证人的价值取决于两点——它当时在场（覆盖完整），以及没人能收买它（不可篡改）。
