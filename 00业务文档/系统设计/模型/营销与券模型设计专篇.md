# 营销与券模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦促销与券建模。营销是变化最快的业务域（活动天天换玩法），也是资损事故的重灾区（薅羊毛、超发、重复核销）。模型要同时做到"玩法可扩展"和"资金可控"。

## 0. 一句话总纲

> 营销模型 = 可配置的规则（怎么优惠）+ 严格的资产（券/权益是准资金）+ 固化的证据（当时为什么便宜）。三者分层，缺一必出事故。

## 1. 三层结构：活动、规则、资产

```text
活动（activity）   —— 运营单元：名称、时间、预算、状态、目标人群
规则（rule）       —— 优惠逻辑：门槛、方式、力度、互斥、限购（版本化！）
资产（asset）      —— 发到用户手里的东西：券、资格、名额（准资金，账务纪律）
```

**为什么分层**：活动是运营视角（预算、周期、复盘），规则是计算视角（引擎执行），资产是资金视角（发了多少、用了多少、成本谁担）。混在一起的典型症状：改一次活动规则，已发的券行为跟着变（历史被污染）；或者活动结束了统计不出成本。

## 2. 优惠规则建模

### 2.1 规则的通用结构

绝大多数玩法都能归一为：

```text
适用范围（scope）：哪些商品/类目/店铺/渠道/人群
门槛（condition）：满 X 元 / 满 N 件 / 会员等级 / 首单
优惠方式（action）：减固定额 / 打折 / 一口价 / 换购 / 赠品 / 免邮
力度上限（cap）：最多优惠 X 元（打折封顶）
```

新玩法优先用"范围+门槛+方式"组合表达，而不是新增一张玩法表——玩法表膨胀到 20 张后没人能维护叠加逻辑。

### 2.2 规则必须版本化（总览篇 §17 的最重场景）

- 活动进行中改规则 = 生成新版本，新订单用新版本，**已下订单/已发券锁定旧版本**。
- 订单固化命中的 `activity_id + rule_version`（订单篇 §3.2），客诉"昨天明明减 20"时可精确还原。

### 2.3 叠加与互斥

多个优惠同时命中是营销计算的核心复杂度：

- 显式配置**互斥组**与**叠加顺序**（先商品级 → 再订单级 → 最后运费；同组互斥取最优）。
- 计算结果输出**优惠明细链**：每一步用了哪个活动、抵了多少、分摊到哪些行（订单篇 ord_discount）。
- 计算过程要**可重放**：同样输入 + 同版本规则 = 同样输出，营销引擎是纯函数，这是对账和客诉处理的基础。

## 3. 券：按准资金建模

券是发到用户手里的"未来资金承诺"，必须套用账务纪律（账务篇）：

### 3.1 券的三张核心表

```sql
-- 券模板（batch）：一批券的定义
CREATE TABLE mkt_coupon_template (
  id            BIGINT PRIMARY KEY,
  activity_id   BIGINT NOT NULL,
  coupon_type   VARCHAR(16) NOT NULL,   -- CASH立减/DISCOUNT折扣/EXCHANGE兑换/FREIGHT运费
  face_value    DECIMAL(18,4),          -- 面额
  threshold     DECIMAL(18,4),          -- 使用门槛
  scope_type    VARCHAR(16) NOT NULL,   -- 全场/类目/商品/店铺
  total_count   INT NOT NULL,           -- 发行总量（预算硬上限）
  issued_count  INT NOT NULL DEFAULT 0, -- 已发（原子累加，防超发）
  per_user_limit INT NOT NULL DEFAULT 1,
  valid_type    VARCHAR(16) NOT NULL,   -- 固定区间 / 领取后N天
  valid_start   DATETIME, valid_end DATETIME, valid_days INT,
  cost_bearer   VARCHAR(16) NOT NULL,   -- 出资方：平台/商家/分摊比例 —— 结算命脉
  status        VARCHAR(16) NOT NULL
);

-- 用户券（instance）：每张券一行，有完整生命周期
CREATE TABLE mkt_user_coupon (
  id           BIGINT PRIMARY KEY,
  coupon_no    VARCHAR(32) NOT NULL,
  template_id  BIGINT NOT NULL,
  user_id      BIGINT NOT NULL,
  status       VARCHAR(16) NOT NULL,    -- UNUSED/LOCKED/USED/EXPIRED/FROZEN
  issue_idem_key VARCHAR(64) NOT NULL,  -- 发放幂等键
  locked_order_no VARCHAR(32),          -- 锁定它的订单
  used_at      DATETIME,
  valid_start  DATETIME NOT NULL, valid_end DATETIME NOT NULL,
  UNIQUE KEY uk_no (coupon_no),
  UNIQUE KEY uk_issue_idem (issue_idem_key),   -- 防重复发放
  KEY idx_user_status (user_id, status, valid_end)
);

-- 券流水：发放/锁定/核销/退回/过期，只增不改
```

### 3.2 券的资金纪律

- **防超发**：`UPDATE template SET issued_count = issued_count + 1 WHERE id=? AND issued_count < total_count`——预算是硬约束，原子更新保证（并发热点时分桶，库存篇 §5 同款方案）。
- **核销三段式**：下单锁定（LOCKED + 记订单号）→ 支付成功核销（USED）→ 取消/超时退回（UNUSED）。与库存冻结、账务冻结完全同构。
- **一券一核销**：状态 CAS 更新 `WHERE status='LOCKED' AND locked_order_no=?`，唯一约束兜底。
- **退款时券的处理**是规则不是代码分支：整单退是否退券（通常券已过期则不退）、部分退不退券但优惠按比例回收——写进规则配置并在退款单上留痕。
- **成本入账**：券核销时按出资方记营销费用（账务篇营销费用户），平台券补贴商家的钱走结算，成本可按活动汇总复盘。

### 3.3 过期与对账

- 过期是定时任务驱动的状态迁移（记流水），亿级券表用 `(status, valid_end)` 索引分批扫。
- 每日对账：模板 issued_count = Σ用户券；Σ核销券成本 = 账务营销费用户发生额；LOCKED 超过 X 小时的券 = 泄露的锁（告警）。

## 4. 防薅羊毛：风控埋点在模型里

- **领取幂等 + 限领**：`uk_issue_idem` + per_user_limit 原子校验，同人多设备并发领取只成功一次。
- **资格与领取分离**：先算资格（人群包/新客判定）再发放，资格判定记录留痕，事后能审计"这批新客券发给了谁、依据什么"。
- 设备指纹、风控评分是外部系统，但**发放流水要预留风控字段**（设备号、IP、风控结果），出事后能圈定黑产批次并批量冻结（FROZEN 状态的存在意义）。

## 5. 营销评审清单

```text
1. 活动/规则/资产是否三层分离？
2. 规则是否版本化？订单是否固化命中版本？已发资产是否锁定旧规则？
3. 叠加互斥是否显式配置？计算是否纯函数可重放？优惠是否分摊到行并记出资方？
4. 券发放是否幂等 + 预算原子扣减防超发？
5. 券核销是否三段式 + CAS + 唯一约束？
6. 退款时券和优惠的回收规则是否配置化并留痕？
7. 券成本是否入账（营销费用户）？活动成本能否汇总复盘？
8. 过期任务是否可扩展到亿级？每日对账是否覆盖发行量/核销成本/僵尸锁定？
9. 发放流水是否预留风控字段？是否支持批量冻结？
```

## 6. 总结

营销建模五条纪律：

1. **三层分离**：活动管运营，规则管计算，资产管资金。
2. **规则纯函数**：版本化、可重放、结果固化到订单行。
3. **券即准资金**：发放有预算、核销走三段、成本要入账。
4. **幂等与原子**：领取、核销、退回全部幂等，预算与状态原子更新。
5. **为风控留痕**：谁、凭什么资格、在什么设备上领的，都查得到。

> 营销的玩法可以天马行空，但钱的纪律必须和账务一样死板。玩法层越灵活，资产层就要越严格。
