# 风控规则引擎模型设计专篇

> 本篇是《世界级数据库模型设计常用方法与底层思想》的深化分册，聚焦风控建模。营销篇、支付篇、会员篇都提到"预留风控字段"，本篇讲风控系统自身的模型：事件、特征、规则、决策、名单、处置。风控的建模难点是**决策必须可解释可回放**——拦了正常用户要能道歉并放行，放过了黑产要能复盘补漏。

## 0. 一句话总纲

> 风控模型五件套：事件（发生了什么）、特征（累计画像）、规则/模型（怎么判）、决策记录（判了什么、依据什么版本）、名单与处置（结果落到哪）。每一次决策都要能在事后完整重放。

## 1. 风控事件：一切判断的输入

```sql
CREATE TABLE rc_event (
  id           BIGINT PRIMARY KEY,
  event_type   VARCHAR(32) NOT NULL,  -- REGISTER/LOGIN/ORDER/PAY/COUPON_CLAIM/WITHDRAW...
  event_time   DATETIME NOT NULL,
  user_id      BIGINT,
  -- 环境要素（黑产识别的核心维度）
  device_id    VARCHAR(64),
  ip           VARCHAR(45),
  ip_region    VARCHAR(32),
  -- 业务要素
  biz_order_no VARCHAR(64),
  amount       DECIMAL(18,4),
  ext          JSON,                  -- 事件类型特有字段
  KEY idx_user_time (user_id, event_time),
  KEY idx_device (device_id, event_time),
  KEY idx_ip (ip, event_time)
);
```

- **事件只增不改**，是风控的 ODS：规则实时消费，特征离线加工，案件回溯全靠它。
- 环境要素（设备指纹、IP、渠道）在**业务接入时强制采集**——事后想补黑产设备维度，历史事件里没有就是永远没有。
- 事件接入是标准化契约：业务方按 schema 上报，缺关键字段的接入不予放行——风控的质量上限由事件质量决定。

## 2. 特征：可累计、可回放的画像

特征 = 对事件的聚合计算，供规则引用：

```text
计数窗口特征：该设备 24h 内注册账号数、该用户 1h 内下单次数、该 IP 当日领券数
关联图特征：设备关联的账号数、收货地址关联的账号数（黑产团伙的核心信号）
状态特征：账号年龄、实名等级、历史拒付次数
外部特征：三方信用分、运营商在网时长（调用记录留痕，含费用）
```

纪律：

- **特征定义注册制**：特征编码、口径、窗口、数据来源、负责人——与指标字典（报表篇 §3）同构，"同名不同数"在风控里就是误判。
- 实时特征走流计算 + KV 存储（Redis/HBase），但**特征取值在决策时刻的快照要落进决策记录**（见 §4）——否则事后无法重放"当时它的 24h 注册数是多少"。
- 外部数据调用记录（调了什么、返回什么、花了多少钱）留存，既是成本核算也是合规要求。

## 3. 规则与策略：版本化的判断逻辑

```text
规则（rule）：条件表达式 + 分数/动作。例：设备24h注册数 > 3 → +40分
策略（policy）：场景 × 规则集 × 组合方式（评分累加/最坏单条/决策表）+ 阈值动作
  例：领券场景 → 总分 ≥ 80 拒绝，60–79 挑战（验证码），< 60 放行
模型（model）：机器学习评分，作为一种特殊"规则"接入，模型版本同样登记
```

铁律：

- **规则和策略全部版本化**（总览篇 §17 最严场景）：改规则 = 新版本 + 生效时间 + 审批，决策记录固化命中版本——"上周为什么拦他"必须按上周的规则解释。
- **灰度与试运行（shadow mode）**：新规则先只记分不动作，观察命中率和误伤率达标后再转正——风控规则直接上线全量生效，等价于拿真实用户做实验。
- 规则要记**命中统计**（命中量、拦截量、事后确认的误报/漏报），没有效果度量的规则只会越堆越多，没人敢删。

## 4. 决策记录：风控的账本

```sql
CREATE TABLE rc_decision (
  id            BIGINT PRIMARY KEY,
  event_id      BIGINT NOT NULL,
  scene         VARCHAR(32) NOT NULL,
  policy_version VARCHAR(32) NOT NULL, -- 固化策略版本
  hit_rules     JSON NOT NULL,         -- 命中规则列表：[{ruleId, version, score}]
  feature_snapshot JSON NOT NULL,      -- 决策时刻的特征取值快照
  score         INT,
  decision      VARCHAR(16) NOT NULL,  -- PASS/CHALLENGE/REVIEW人工/REJECT
  final_result  VARCHAR(16),           -- 挑战通过/人工结论 回填
  created_at    DATETIME NOT NULL,
  KEY idx_event (event_id)
);
```

- **每次决策一条记录，包含版本 + 命中明细 + 特征快照**——这三样齐了，任何决策可完整重放。客诉申诉、监管质询、误报复盘全靠它。
- 决策是同步链路（下单/支付等着答案），性能预算严格（p99 几十毫秒），决策记录**异步落库**但不可丢（Outbox/本地队列）。
- **降级预案是显式配置**：风控系统不可用时各场景默认放行还是拒绝（支付场景通常放行 + 事后审，提现场景拒绝）——这是业务决策，要写进配置而不是 catch 块。

## 5. 名单体系：结果的沉淀

```text
名单（list）：名单类型（黑/白/灰）× 维度（用户/设备/IP/手机号/卡号/收货地址）× 场景范围
名单条目：值、来源（哪条规则/哪个案件/人工）、入单时间、有效期、操作人
```

- **入单必有来源、出单必有记录**：名单是处置结果不是判断本身，每个条目能追到"为什么进来的"；黑名单永久化要谨慎（设备会转手、IP 会重新分配），**默认带有效期**。
- 白名单是双刃剑（内部测试号、大客户），条目要定期复核——泄露的白名单是黑产的最爱。
- 名单查询是决策链路最高频操作，走缓存（布隆过滤器 + KV），DB 是权威源（派生副本纪律）。

## 6. 案件与处置：人工闭环

- **人工审核队列**（REVIEW 决策落入）：案件 = 决策 + 上下文聚合展示，审核结论（通过/拒绝/加黑）回填决策记录并可能生成名单条目——结论是规则效果评估的标注数据。
- **处置动作单据化**：冻结账户、批量冻券（营销篇 FROZEN）、取消订单，都走各业务域的正式单据/接口并留痕，风控系统不直改业务表（与工作流引擎同一纪律）。
- **误报救济通道**：申诉单（用户提交 → 复核 → 解除处置 + 决策记录标记误报）——误报率是风控的核心健康指标，没有申诉通道就没有误报数据。

## 7. 风控评审清单

```text
1. 事件是否只增不改、强制采集环境要素、接入有 schema 契约？
2. 特征是否注册制（口径/窗口/负责人）？外部调用是否留痕计费？
3. 规则/策略/模型是否版本化 + 审批 + 灰度试运行？
4. 决策记录是否固化版本、命中明细、特征快照，可完整重放？
5. 决策链路性能预算是否明确？记录落库是否异步不丢？
6. 降级预案（放行/拒绝）是否按场景显式配置？
7. 名单是否带来源、有效期？白名单是否定期复核？
8. 人工审核结论是否回填并成为规则效果标注？
9. 处置是否走业务域正式单据？申诉与误报统计是否闭环？
```

## 8. 总结

风控建模五条纪律：

1. **事件为本**：只增不改、要素齐全，事件质量决定风控上限。
2. **特征有典**：口径注册、决策时快照，同名同义可重放。
3. **规则如法**：版本化、审批、灰度、效果度量，无度量不上线。
4. **决策留证**：版本 + 命中 + 快照三件套，每个决定经得起质询。
5. **处置走正门**：名单有来源有效期，动作走单据，误报有救济。

> 风控的权力是拦截用户，权力越大留痕要越重。一个不能解释自己决定的风控系统，和它要对抗的黑产一样不可信。
