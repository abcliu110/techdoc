# 报表与数仓建模专篇 - 系统化拆解 v1

## 1. 本质模型

报表与数仓模型的本质，是把业务系统产生的原始事实，经过清洗、建模、汇总和语义治理，转化成可复用、可追溯、可分析的数据资产。

核心抽象：

```text
交易库 -> ODS -> DWD -> DWS -> ADS -> 报表/指标/API
```

## 2. 逻辑模型

### 2.1 实体清单

| 实体 | 关键字段 | 职责 |
|---|---|---|
| ods_* 贴源表 | 源表全字段原样, 删除标记, 变更时间, 抽取批次 | 原始保留不加工，是全链路可重算的根 |
| dwd_pay_order_line 明细事实表(例) | 业务键, biz_date(分区键), 度量(成交金额/数量), 各维度业务键(store_id/sku_id...), 事件时间 | 一行=一个业务事实；粒度一句话声明，度量标注可加性 |
| dim_store 门店维(拉链) | store_key(代理键), store_id(业务键), 门店名, 所属大区, 生效日, 失效日, 是否当前 | SCD2 默认；事实关联"事发当时"版本，防组织调整污染历史 |
| dim_date 日期维 | 日期, 财周, 节假日标记, 促销日标记 | 物化日期属性，不在 SQL 里现算 |
| dws_sales_day 主题汇总表(例) | biz_date, 主题维度组合(门店/品类), 聚合度量, 产出任务 ID | 只从 DWD 聚合；聚合口径固化在此层 |
| ads_report_* 应用结果表 | 报表维度, 指标值, biz_date, refreshed_at | 面向具体看板；只从 DWS/DWD 取数，禁自造口径 |
| metric_dict 指标字典 | 指标编码(唯一如 GMV_PAID), 业务定义, 计算口径, 数据来源表, 维度支持, 负责人, version, 生效日期 | "同名不同数"的唯一解药；口径变更走新版本 |
| report_submit_batch 报送批次 | 批次号(唯一), 报送模板, 数据快照, 报送时间, 回执 | 对外/监管出具即冻结；重报生成新批次 |
| dq_check_log 质量监控记录 | 表名, biz_date, 检查项(行数波动/主键重复/空值率/金额比对), 结果, 是否阻断 | 错误数字宁可晚到不可到达的执行点 |

### 2.2 实体关系

```text
ods_* (1) ──< dwd 明细事实 (N)            清洗标准化，逐层只从上游取数
dwd 明细事实 (N) >── dim_store/dim_date (1)  按业务键+日期区间关联拉链版本
dwd 明细事实 (1) ──< dws 主题汇总 (N)      聚合口径固化在 DWS
dws 主题汇总 (1) ──< ads_report_* (N)     一个主题服务多张报表
metric_dict (1) ──< ads_report_* (N)      报表指标必须引用字典编码
ads_report_* (1) ──< report_submit_batch (N)  对外报送落批次快照
dq_check_log ──> 各层表                    质量红灯阻断下游产出任务
外部引用：源表 → 各业务域交易库（CDC/批量抽取单向流入）；
         产出任务 → 批处理域 job_def；负责人 → 权限域用户
         （引用不落外键，应用层校验；在线业务禁止反向查询数仓）
```

### 2.3 关键约束与不变量落点

```text
唯一键/幂等键：指标编码唯一；(业务键, 生效日) 维度版本唯一；报送批次号唯一；
             事实表按 biz_date 分区重刷幂等（先清本分区再写）
版本固化：维度 SCD2 拉链（代理键与业务键分离）；指标口径变更=新版本+生效日期，
         不改历史定义；报送批次快照永久保留不重算覆盖
分层约束：每层只从上游层取数（DWS 不回查 ODS，ADS 不自造口径）；
         任一层损坏可从上游重建，ODS 历史保留足够
迟到/订正落点：分区键用业务日期而非处理日期，迟到事实入对应分区重刷；
             账期关账后数据冻结，差异走调整记录追加
质量闸门：行数波动/主键重复/空值率/金额与源库比对，红灯阻断下游任务
分片键：biz_date（分区主键）+ 主题维度（门店/商家等，多租户报表带 merchant_id）
```

## 3. 核心张力

最大张力是：

```text
业务实时变化 vs 分析口径稳定
```

业务数据实时变化，管理报表需要稳定、可解释、可复核。数仓分层就是为了把变化中的交易数据变成稳定的分析资产。

## 4. 不变量

- 每一层只做一类事情。
- ODS 保留原始数据。
- DWD 表达明细事实。
- DWS 做主题汇总。
- ADS 服务具体应用。
- 指标口径必须统一。
- 报表不能反向依赖业务库写逻辑。

## 5. 为什么这样设计

数仓分层是为了防止所有报表都从交易库临时拼接。维度建模是为了让指标能够按时间、组织、商品、会员、渠道等稳定维度分析。

## 6. 失败模式

- 每个报表自己写 SQL，口径不一致。
- 明细层缺失，汇总无法追溯。
- 报表直接压业务库。
- 指标被重复定义。
- 没有数据血缘，数字错了无法定位。

## 7. 适用边界

适用：

- 多报表、多指标、多业务域、需要经营分析和历史追溯的系统。

简化：

- 初期可用轻量宽表或汇总表替代完整数仓，但必须保留指标口径和血缘意识。

## 8. 同一模型的多业务形态建模

报表消费形态差异极大，但支撑它们的骨架只有一套：**分层 + 事实表 + 维度 + 指标字典，消费差异下沉到 ADS 层和服务方式**。

### 8.1 统一骨架（所有形态不变的部分）

```text
ODS（原始保留）→ DWD（明细事实）→ DWS（主题汇总）→ ADS（按应用出）→ 报表/大屏/API
不变量全集：每层只做一类事、明细可追溯、指标口径统一（指标字典唯一定义）、不反向依赖业务库
```

骨架决不为某个报表特化——ODS/DWD/DWS 与消费形态无关，所有形态差异都在 ADS 层和链路时效上吸收。

### 8.2 各形态差异建模

| 业务形态 | 差异点 | 建模方式 |
|---|---|---|
| 经营日报 | T+1、口径稳定、按组织下钻 | 标准形态：DWS 日粒度主题表 → ADS 日报宽表，维度用组织快照维（防组织调整污染历史） |
| 实时大屏 | 秒级鲜度、允许近似、只看当天 | 独立实时链路（CDC→流计算→实时汇总表），不经过批量分层；指标必须引用同一指标字典口径，屏上标注"实时口径，以日结为准" |
| 财务对账报表 | 分毫不差、可复核、有截止时间 | DWD 用事务事实表 + 账期快照（关账后数据冻结不可重算）；差异调整走调整记录追加，不改历史行 |
| 商家侧报表 | 多租户、只能看自己、并发查询高 | ADS 表带 merchant_id 分区 + 行级过滤在服务层强制注入；预聚合到商家粒度，避免明细层直接暴露给外部 |
| 监管报送 | 固定格式、报送后不可变、留痕 | ADS 层按报送模板建专表，生成即落"报送批次表"（快照 + 报送时间 + 回执）；重报生成新批次，旧批次永久保留 |

### 8.3 建模决策规则

- **能在 ADS 层出新表解决的，不动 DWD/DWS**：新报表形态是新的消费视图，明细和汇总层保持与消费无关。
- **实时与离线是两条链路、一个口径**：实时链路可以另建，但指标定义必须引用同一指标字典，禁止大屏自造"实时销售额"。
- **对外/对监管的形态必须加"冻结快照"机制**：商家结算、监管报送的数据一旦出去就不可被重算覆盖，用批次快照而非直接查 ADS。
- **拆独立域的信号**：当消费方需要的不是聚合分析而是逐笔事实回放（如审计取证、风控回溯），那是数据服务/审计域的需求，不应在数仓 ADS 里硬造明细出口。
