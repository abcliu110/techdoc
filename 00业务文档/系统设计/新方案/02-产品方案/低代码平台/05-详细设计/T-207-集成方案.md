## 二、集成策略

### 方案：扩展现有packages（推荐）

**原则**：
- ✅ 复用现有的 `renderer`（已有渲染能力）
- ✅ 扩展 `builder`（增加表单设计能力）
- ✅ 新增 `form-core`（表单专用的BO/Model层）
- ✅ 新增 `form-components`（表单组件库）

**理由**：
1. 不破坏现有架构
2. 复用已有的pnpm workspace配置
3. 与现有builder/renderer协同工作
4. 保持项目统一性

```typescript
import { FormModel } from '@lowcode/form-core';

export const FormRenderer: React.FC<{
  formModel: FormModel
}> = observer(({ formModel }) => {
  return (
    <Form>
      {Array.from(formModel.fields.values()).map(field => (
        <FieldRenderer key={field.fieldId} field={field} />
      ))}
    </Form>
  );
});
```

## 八、优势分析

### vs 独立项目

| 维度 | 独立项目 | 集成方案 |
|------|---------|---------|
| 开发速度 | 慢（从零开始） | 快（复用现有基础） |
| 代码复用 | 低 | 高 |
| 维护成本 | 高（两个项目） | 低（统一项目） |
| 技术栈 | 可能不一致 | 完全一致 |
| 团队协作 | 分散 | 集中 |

### 复用现有能力

✅ 复用pnpm workspace配置  
✅ 复用TypeScript配置  
✅ 复用Vite构建配置  
✅ 复用ESLint/Prettier配置  
✅ 复用CI/CD流程  
✅ 复用已有的shared工具  
✅ 复用已有的renderer基础
