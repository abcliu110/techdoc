## 1. 设计目标

强表格与分录设计要回答一个核心问题：

```text
设计器如何让实施顾问设计复杂企业单据中的分录、明细、列编辑、冻结、汇总、行操作和一层母子结构。
```

T-207 的表格不是普通 HTML table，也不是 Excel 替代品。它必须支撑采购、销售、库存、财务、审批等企业单据中高密度、可编辑、可校验、可授权、可测试的分录表格。

强表格必须做到：

1. 从 BO 分录自动生成 EntryTable 和 TableColumn。
2. 支持行内 CellEditor、只读渲染、校验提示和列级权限。
3. 支持列宽、冻结列、行高、密度、分页、虚拟滚动和汇总行。
4. 支持 P0 一层 DetailPanel / DetailTable，不支持 P0 多层嵌套。
5. 支持表格设计态抓手，让自动化测试平台能模拟点击、拖拽、调整列宽、展开明细。
6. 支持 Analyzer 检查列绑定、明细层级、冻结冲突、性能风险和发布阻断问题。
7. 保持与 RuntimeRenderer 同源渲染，设计态不是截图，也不是报表画布。

## 2. 参考产品吸收

| 产品 | 可吸收能力 | T-207 转化 |
|---|---|---|
| 金蝶 BOS / 云苍穹 | 业务对象、单据体、分录、基础资料、行级规则 | EntryTable 以 BO Entry 为源，表格列保留业务语义 |
| AG Grid Enterprise | Master-Detail、虚拟滚动、列固定、CellEditor 生命周期 | TableEngine 的运行态交互模型 |
| DevExpress VCL / Web Grid | Level Tree、Object Inspector、主从表设计 | Table Level Tree + Inspector 分区 |
| Kendo / Telerik Grid | Hierarchy Grid、detailTemplate、列菜单 | P1 列菜单和层级扩展参考 |
| Syncfusion Grid | childGrid、虚拟滚动、列选择器 | P0 一层明细 + P1 列选择器 |
| Handsontable | 键盘录入、单元格编辑生命周期、复制粘贴 | P1 Excel-like 批量录入参考 |
| SpreadJS | 复杂表头、合并单元格、打印套打 | P1/P2 复杂表头和打印能力参考 |

T-207 不直接嵌入某一家表格产品的 UI。设计器应抽象出自己的 `TableSchema`、`TableEngine`、`TableDesignerOverlay` 和 `TableAnalyzer`，具体实现可以在运行态复用成熟网格库，但 Schema 不能绑定到某个第三方库。

## 3. 核心定位

### 3.1 强表格是什么

| 定义 | 说明 |
|---|---|
| 企业分录设计器 | 用于设计单据体、子表、明细表和行内字段编辑 |
| TableSchema 模型 | 描述列、编辑器、行操作、冻结、汇总、明细和性能策略 |
| TableEngine | 运行态表格渲染、编辑、校验、虚拟滚动和行状态管理 |
| TableDesignerOverlay | 设计态列抓手、投放点、宽度标尺、冻结线、明细展开区 |
| TableAnalyzer | 检查绑定、列宽、层级、冻结、性能、权限和发布风险 |

### 3.2 强表格不是什么

| 非目标 | 原因 |
|---|---|
| 报表设计器 | 报表以打印和分页为核心，T-207 表格以业务录入和运行态交互为核心 |
| Excel 替代品 | P0 不做公式网格、合并单元格、自由单元格格式 |
| 任意嵌套表格 | P0 只支持 Level 1，P1 Level 2，P2 Level 3 |
| 第三方 Grid 配置透传 | 会锁死实现并破坏 Schema 可治理性 |
| 移动端表格设计器 | P0 聚焦 PC Web 单据 |
| 自由 JS 单元格 | 无法静态分析、无法发布门禁、无法自动化测试 |

| 禁止 | 阶段 | 原因 |

| 单元格自由合并 | P0 禁止 | 会破坏行列数据绑定和自动化测试 |

## 5. 总体架构

### 5.1 模块结构

```text
BusinessObject.Entry
  ↓
EntryTableResolver
  ├─ TableSchemaFactory
  ├─ TableColumnFactory
  ├─ CellEditorResolver
  ├─ TablePermissionResolver
  ├─ TableSummaryResolver
  ├─ DetailSchemaResolver
  └─ TableAnalyzer
      ↓
EntryTableSchema
      ↓
TableEngine
      ↓
RuntimeRenderer
      ↓
TableDesignerOverlay
```

### 5.2 模块职责

| 模块 | 职责 | P0 |
|---|---|---|
| EntryTableResolver | 从 BO 分录生成表格 Schema | 是 |
| TableSchemaFactory | 生成 EntryTableSchema 默认结构 | 是 |
| TableColumnFactory | 生成列、绑定、编辑器、只读渲染 | 是 |
| CellEditorResolver | 复用字段体系 EditorRegistry 选择单元格编辑器 | 是 |
| TablePermissionResolver | 合并表格和列的静态权限 | 是 |
| TableSummaryResolver | 生成汇总行配置 | 是 |
| DetailSchemaResolver | 生成 Level 1 DetailPanel / DetailTable | 是 |
| TableEngine | 渲染、编辑、校验、行状态和虚拟滚动 | 是 |
| TableDesignerOverlay | 列抓手、投放点、冻结线、明细区设计辅助 | 是 |
| TableAnalyzer | 表格发布质量门禁 | 是 |

### 5.3 双层设计

T-207 表格必须分为设计态和运行态：

| 层 | 说明 |
|---|---|
| TableSchema | 产品自己的表格声明模型 |
| TableEngine | 运行态表格渲染和交互，内部可适配第三方 Grid |
| TableDesignerOverlay | 设计态辅助层，负责选中框、列宽抓手、投放线、冻结线 |
| Inspector | 当前表格、列、行操作、明细区属性配置 |
| Analyzer | 发布前检查和问题定位 |

第三方 Grid 只允许在 `TableEngine` 内部作为实现细节，不能泄漏到 Schema 和 Inspector。

## 6. EntryTableSchema

### 6.1 Schema 示例

```json
{
  "id": "entry_sales_items",
  "type": "EntryTable",
  "name": "salesOrderEntries",
  "title": "销售订单分录",
  "dataSource": {
    "type": "boEntry",
    "boId": "SalesOrder",
    "entryCode": "entries",
    "rowKey": "entryId"
  },
  "layout": {
    "height": 360,
    "density": "middle",
    "rowHeight": 36,
    "headerHeight": 36
  },
  "columns": [],
  "features": {
    "inlineEdit": true,
    "rowSelection": "multiple",
    "rowNumber": true,
    "pagination": true,
    "virtualScroll": true,
    "summaryRow": true
  },
  "freeze": {
    "left": 1,
    "right": 0
  },
  "detail": {
    "enabled": true,
    "detailDepth": 1,
    "type": "DetailPanel",
    "schemaRef": "detail_batch"
  },
  "design": {
    "selectable": true,
    "draggable": true,
    "testId": "designer-node-entry_sales_items"
  }
}
```

### 6.2 属性说明

| 属性 | 说明 | P0 |
|---|---|---|
| id | 表格节点 ID | 是 |
| type | 固定为 EntryTable | 是 |
| name | 技术名称 | 是 |
| title | 展示标题 | 是 |
| dataSource | BO 分录数据源 | 是 |
| layout | 表格尺寸、密度、行高 | 是 |
| columns | TableColumn 数组 | 是 |
| features | 编辑、选择、分页、虚拟滚动等开关 | 是 |
| freeze | 左右冻结列配置 | 是 |
| detail | 一层明细配置 | 是 |
| design | 设计态元数据 | 是 |

### 6.3 dataSource

| 属性 | 说明 | P0 |
|---|---|---|
| type | boEntry / staticSample | 是 |
| boId | 业务对象 ID | 是 |
| entryCode | 分录编码 | 是 |
| rowKey | 行唯一键 | 是 |
| sampleRows | 示例数据 | 是 |

P0 EntryTable 必须有 `rowKey`。如果 BO 分录没有显式主键，Schema 生成时必须创建稳定 `_rowId`，但发布前应提示实施顾问确认行主键策略。

### 6.4 层级术语

| 术语 | 说明 | 阶段 |
|---|---|---|
| Level 0 | EntryTable 自身 | P0 |
| Level 1 | EntryTable 行展开的一层 DetailPanel 或 DetailTable | P0 |
| Level 2 | 明细内再嵌套子明细 | P1 |
| Level 3 | 更深层业务嵌套 | P2 |

`detailDepth=1` 等价于 Level 1。P0 Analyzer 必须阻断 `detailDepth > 1`。

### 7.3 列类型

| 列类型 | 说明 | P0 |
|---|---|---|
| DataColumn | 绑定 BO 分录字段 | 是 |
| RowNumberColumn | 行号列 | 是 |
| SelectionColumn | 选择列 | 是 |
| ActionColumn | 行操作列 | 是 |
| DetailToggleColumn | 明细展开列 | 是 |
| ComputedColumn | 计算列 | P1 |
| ColumnGroup | 列组 / 复杂表头 | P1 |

### 7.4 dataIndex 规范

`dataIndex` 始终是行内字段名：

| 正确 | 错误 |
|---|---|
| `materialId` | `entries[0].materialId` |
| `qty` | `salesOrder.entries.qty` |
| `price` | `components.entry.columns[2]` |

运行时由 TableEngine 根据当前行上下文把 `dataIndex` 映射到真实数据路径。

## 8. CellEditor 与编辑生命周期

### 8.1 编辑器映射

TableColumn 复用字段体系的 EditorRegistry：

| BO 字段类型 | 默认 CellEditor | P0 |
|---|---|---|
| string | TextInput | 是 |
| integer / decimal | NumberInput | 是 |
| money | MoneyInput | 是 |
| date | DatePicker | 是 |
| datetime | DateTimePicker | 是 |
| enum | Select | 是 |
| boolean | Checkbox / Switch | 是 |
| reference | ReferencePicker | 是 |
| treeReference | TreePicker | 是 |
| attachment | AttachmentUploader | 是 |

### 8.2 编辑生命周期

```text
cell.focus
  -> editor.open
  -> value.input
  -> value.parse
  -> value.validate
  -> rowState.patch
  -> summary.recompute
  -> analyzer.incrementalCheck
  -> editor.commit / editor.cancel
```

| 阶段 | P0 行为 |
|---|---|
| focus | 单击选中单元格，双击或 Enter 进入编辑 |
| open | 创建 CellEditor，读取当前行值 |
| input | 用户通过真实 DOM 输入 |
| parse | 按字段类型解析 |
| validate | required、范围、格式校验 |
| patch | 写入 RuntimeState，不直接改 Schema |
| commit | Enter / Tab / 失焦提交 |
| cancel | Esc 回滚当前编辑 |

### 8.3 键盘行为

| 操作 | P0 |
|---|---|
| Enter | 进入编辑 / 提交编辑 |
| Esc | 取消当前编辑 |
| Tab | 提交并移动到下一可编辑单元格 |
| Shift+Tab | 提交并移动到上一可编辑单元格 |
| Arrow | 单元格导航 |
| Delete | 清空当前可编辑单元格 |
| Ctrl+C / Ctrl+V | P1 Excel-like 复制粘贴 |

P0 支持单元格级键盘导航，不支持多区域复制粘贴和批量填充。

## 9. 行模型与行操作

### 9.1 RowState

```json
{
  "rowKey": "row_001",
  "status": "editing",
  "dirty": true,
  "selected": false,
  "expanded": false,
  "errors": {
    "materialId": ["物料不能为空"]
  }
}
```

| 状态 | 说明 |
|---|---|
| clean | 未修改 |
| editing | 正在编辑 |
| dirty | 已修改未保存 |
| new | 新增行 |
| deleted | 待删除行 |
| error | 行内存在错误 |
| expanded | 明细展开 |

### 9.2 P0 行操作

| 操作 | 说明 |
|---|---|
| addRow | 新增一行 |
| insertRowAbove | 在当前行上方插入 |
| insertRowBelow | 在当前行下方插入 |
| copyRow | 复制当前行 |
| deleteRow | 删除当前行 |
| moveRowUp | 上移当前行 |
| moveRowDown | 下移当前行 |
| expandDetail | 展开 Level 1 明细 |
| collapseDetail | 收起 Level 1 明细 |

行操作必须进入 CommandStack，支持撤销/重做。批量行操作归 P1。

### 9.3 ActionColumn

ActionColumn 是 P0 行操作入口：

| 属性 | 说明 |
|---|---|
| actions | 行操作集合 |
| displayMode | icon / menu |
| confirm | 删除等危险操作二次确认 |
| permission | 行操作静态权限入口 |

P0 ActionColumn 不支持任意 JS 动作。提交、审核、反审核等业务状态动作归后续规则权限流程文档。

## 10. 冻结、列宽和布局

### 10.1 列宽

| 属性 | 说明 | P0 |
|---|---|---|
| width | 当前列宽 px | 是 |
| minWidth | 最小列宽 px | 是 |
| maxWidth | 最大列宽 px | P1 |
| resizable | 是否可拖拽调整 | 是 |
| autoFit | 根据内容自适应 | P1 |

列宽调整必须通过设计态抓手完成：

```text
data-testid="table-column-resize-{tableId}-{columnId}"
```

### 10.2 冻结列

P0 支持左冻结和右冻结：

| 配置 | 说明 |
|---|---|
| freeze.left | 左侧冻结列数 |
| freeze.right | 右侧冻结列数 |
| column.layout.freeze | none / left / right |

冻结规则：

1. 行号列和选择列默认可左冻结。
2. ActionColumn 默认可右冻结。
3. 左右冻结列宽总和不能超过表格可视宽度的 60%。
4. 冻结列不能与隐藏列冲突。

### 10.3 密度和行高

| 密度 | rowHeight | 适用场景 |
|---|---|---|
| compact | 30 | 数据录入密集场景 |
| middle | 36 | 默认企业单据 |
| comfortable | 44 | 审批查看或低密度表格 |

P0 不允许单行自由高度。附件、多行文本等复杂 CellEditor 需要通过弹窗或明细区承载。

## 12. DetailPanel / DetailTable

### 12.1 DetailPanel

DetailPanel 用于展开单行的附加信息，例如批次、序列号、交付计划、审批备注。

```json
{
  "type": "DetailPanel",
  "id": "detail_batch",
  "parentTableId": "entry_sales_items",
  "detailDepth": 1,
  "content": {
    "type": "GridLayout",
    "children": []
  }
}
```

### 12.2 DetailTable

DetailTable 用于一行下面的一层子表，例如物料批次明细。

```json
{
  "type": "DetailTable",
  "id": "detail_batch_table",
  "parentTableId": "entry_sales_items",
  "detailDepth": 1,
  "dataSource": {
    "type": "boEntryChild",
    "entryCode": "entries.batchDetails",
    "rowKey": "batchRowId"
  },
  "columns": []
}
```

### 12.3 P0 约束

| 约束 | P0 |
|---|---|
| detailDepth 最大值 | 1 |
| 每个 EntryTable 可有 DetailPanel 或 DetailTable | 是，二选一为默认建议 |
| DetailPanel 内允许 GridLayout | 是 |
| DetailPanel 内允许再放 EntryTable | 否 |
| DetailTable 内允许再放 DetailTable | 否 |
| 多层展开同时打开 | P1 |

Analyzer 必须输出 `TABLE_DETAIL_DEPTH_EXCEEDED` 阻断 P0 多层嵌套。

### 15.1 Analyzer 输出结构

#### 15.1.1 schemaPath 扩展

本文沿用 04-表单元素与字段体系的 `schemaPath` 规范，并补充 EntryTable / TableColumn 路径：

| 场景 | schemaPath 示例 | 说明 |
|---|---|---|
| EntryTable 根 | `components.entry_sales_items` | 表格节点 |
| EntryTable 数据源 | `components.entry_sales_items.dataSource` | 表格数据源 |
| EntryTable 明细层级 | `components.entry_sales_items.detail.detailDepth` | Level 1/2/3 检查 |
| TableColumn | `components.entry_sales_items.columns[3]` | 第 4 列 |
| TableColumn 编辑器 | `components.entry_sales_items.columns[3].editor.type` | 列编辑器 |
| TableColumn 宽度 | `components.entry_sales_items.columns[3].layout.width` | 列宽 |
| SummaryRow | `components.entry_sales_items.summaryRow.items[0]` | 汇总项 |

Analyzer Quick Fix 必须使用上述路径定位到表格、列或汇总项，不能用列标题或显示名作为定位依据。

### 15.2 P0 规则清单

| 规则 | 级别 | 说明 |
|---|---|---|
| TABLE_DATASOURCE_MISSING | Error | 表格缺少数据源 |
| TABLE_ROWKEY_MISSING | Error | 表格缺少 rowKey |
| TABLE_COLUMN_BINDING_MISSING | Error | 列缺少 dataIndex 或 businessField |
| TABLE_COLUMN_EDITOR_MISMATCH | Error | 列字段类型和 CellEditor 不兼容 |
| TABLE_DETAIL_DEPTH_EXCEEDED | Error | 明细层级超过 Level 1 |
| TABLE_FREEZE_WIDTH_EXCEEDED | Error | 冻结列总宽超过可视宽度 60% |
| TABLE_FREEZE_HIDDEN_CONFLICT | Warning | 冻结列同时被隐藏 |
| TABLE_COLUMN_WIDTH_TOO_SMALL | Warning | 列宽小于 minWidth |
| TABLE_COLUMN_COUNT_RISK | Warning | 列数超过建议阈值 50 |
| TABLE_DEFAULT_ROWS_RISK | Warning | 默认展示行数超过建议阈值 100 且未开虚拟滚动 |
| TABLE_SUMMARY_TYPE_INVALID | Error | 汇总类型与列字段类型不兼容 |
| TABLE_ACTION_PERMISSION_CONFLICT | Warning | 行操作可见但不可执行 |

### 15.3 Quick Fix

| 问题 | Quick Fix |
|---|---|
| 缺少 rowKey | 选择 BO 主键或生成 `_rowId` |
| 列缺少绑定 | 绑定到原 BO 分录字段 |
| 编辑器不兼容 | 切回默认 CellEditor |
| 明细层级超限 | 移除 Level 2+ 明细 |
| 冻结宽度超限 | 取消最右侧冻结列 |
| 列宽过小 | 调整到 minWidth |
| 未开虚拟滚动 | 开启 virtualScroll |

Error 级问题不可忽略，必须修复后才能发布。

## 17. 性能与规模约束

| 指标 | P0 建议 |
|---|---|
| 单表列数 | 建议 ≤ 50 |
| 默认渲染行数 | 建议 ≤ 100 |
| 虚拟滚动行数 | 可支持 1000+，具体由运行时验证 |
| 冻结列总宽 | 不超过可视宽度 60% |
| 明细层级 | Level 1 |
| 同页 EntryTable 数量 | 建议 ≤ 5 |
| 单表 CellEditor 类型数 | 建议 ≤ 12 |

超过建议值时 Analyzer 输出 Warning；超过 P0 硬约束时输出 Error。
