> 评审结论：仍需截图级复核。2026-07-09 已按本记录修复一轮可见原型；源码静态检查显示三列、48px + 288px、说明下沉为 tooltip 已到位，但由于 file:// 页签被浏览器安全策略阻止自动刷新，尚未完成用户可见截图复评。

## 1. 为什么之前评审没有发现问题

之前评审主要检查“是否有左侧栏、是否有组件库、是否有字段入口、是否有 data-testid”，属于文档一致性和可测试性视角；没有引入专业前端/UX 对“视觉密度、信息层级、首屏承载量、说明文字常驻成本、企业 IDE 工具箱效率”的专项判断。

本次暴露的问题是典型专业 UI 问题：组件库存在不等于组件库好用；三列大卡片存在不等于满足高频设计器效率；规则门禁通过不等于视觉密度和交互层级合理。

## 2. 金蝶参考结论

依据知识库 `UI-KINGDEE-FORM-002-表单设计器像素级参考.md`：

| 项 | 金蝶参考值 |
|---|---|
| 左控件栏宽度 | 约 265-290px |
| 控件项宽度 | 约 70-85px |
| 控件排列 | 三列排列 |
| 控件形态 | 高密度图标网格，不是大卡片 |
| 文案 | 12-13px，位于图标下方 |

因此，不能说“列数越多就越强”。更准确的吸收结论是：金蝶采用窄栏三列高密度控件库；自研设计器采用约 288px Dock Panel 内的三列紧凑图标卡，优先保证中文名称可读。T-207 的超越点应来自字段入口分离、停靠折叠、状态合同、复杂表格和自动化抓手，而不是单纯增加列数。

## 4. 修复决策

| 决策 | 内容 |
|---|---|
| 组件面板宽度 | Activity Bar 约 48px；组件 Dock Panel 约 288px |

## 7. 复核标准

1. 组件 Dock Panel 默认约 288px，Activity Bar 约 48px，整体左侧不超过约 336px；超过该宽度视为挤压中央画布。
2. 每个可见组件分组三列显示，优先保证中文名可读。
3. 卡片默认只保留图标、中文名称、拖拽抓手。
4. 说明文字不常驻，进入 tooltip/详情区。
5. 六个 Activity Bar 入口切换后的面板内容在原型和文档中一致。
6. 后续每次复杂 UI 原型修改，都必须留下专业前端/UX 专项评审记录。
7. 业务流程门禁只负责阶段阻断，不能作为“UI 已专业 / 已超过竞品”的证据；UI 结论必须来自可见原型、截图对比和专业前端/UX 复评。
