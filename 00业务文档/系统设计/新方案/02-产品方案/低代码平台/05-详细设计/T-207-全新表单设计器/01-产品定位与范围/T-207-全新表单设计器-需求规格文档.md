## 1. 背景与目标

### 1.1 背景

企业低代码平台需要一个能够设计真实 PC Web 单据、复杂表单、主数据维护页、审批页和业务工作台的可见即可得设计器。现有普通表单 Builder 通常只能完成字段拖拽和简单布局，无法满足企业页面中的单据头体、强表格、母子明细、复杂布局、规则权限、版本发布和自动化质量门禁要求。

T-207 全新表单设计器的目标不是复制金蝶、DevExpress、Visual Studio Form Designer 或 AG Grid，而是吸收其优秀结构：

| 参考来源 | 吸收能力 | T-207 转化 |
|---|---|---|
| 金蝶 BOS / 云苍穹 | 单据、分录、业务对象、规则、权限、发布 | BusinessObject、HeaderForm、EntryTable、Rule/Permission、PublishAnalyzer |
| DevExpress Web Designer | 工具箱、字段列表、属性面板、分析器 | 五区 IDE 工作台、Resource Dock、Inspector、Analyzer |
| Visual Studio Form Designer | Toolbox、Properties、Outline、Snaplines | 拖拽、选中框、父级选择、吸附线、错误列表 |
| Oracle APEX / ServiceNow UI Builder | 页面树、数据源、声明式运行时 | PageSchema、DataBinding、RuntimeRenderer |
| AG Grid / DevExtreme / Kendo / Syncfusion / SpreadJS | 强表格、Master-Detail、虚拟滚动、复杂表格能力 | TableEngine、EntryTable、DetailPanel、Level 1 明细 |
| Webflow / Plasmic | Grid/Flex 可视化布局、组件注册 | LayoutEngine、ComponentRegistry、受控 Web 布局 |

### 1.2 产品目标

T-207 全新表单设计器是：

```text
面向企业 PC Web 单据、复杂表单、主数据维护、审批页和业务工作台的
IDE 级可见即可得页面设计器。
```

它必须让实施顾问和开发团队能够从业务对象出发，设计一个真实可运行的 PC Web 页面，并经过保存、校验、预览、发布、回滚和自动化回归验证。

### 1.3 成功标准

| 编号 | 成功标准 | 可量化验收 |
|---|---|---|
| S-001 | 用户能识别产品定位 | 5 名目标用户中至少 4 名在 3 秒内判断为企业 PC Web 单据设计器 |
| S-002 | 可设计复杂单据 | P0 销售订单样例包含 HeaderForm、EntryTable、SummaryBar、ActionBar、RightAssistPanel |
| S-003 | Web 布局能力充分 | P0 覆盖 Grid、Flex、Section、Tabs、Split、Scroll、Sticky 7 类布局容器 |
| S-004 | 强表格可用 | P0 支持列宽、冻结列、行操作、汇总行、Level 1 DetailPanel 或 DetailTable |
| S-005 | 运行态同源 | 同一 PageSchema 在设计态、预览态、运行态布局结构一致，设计态只额外显示 DesignerOverlay |
| S-006 | 发布可治理 | Analyzer Error 存在时不能发布；发布成功必须生成 schemaHash 和版本记录 |
| S-007 | 自动化可模拟 | Playwright 可通过稳定 testId 完成拖拽字段、调整 span、调整列宽、点击 Analyzer 问题 |
| S-008 | 性能达标 | 设计器冷启动 <= 3 秒，运行态页面加载 <= 2 秒，热更新 <= 5 秒，拖拽帧率 >= 30 fps |

## 3. 非功能性需求

### 3.1 性能要求

| 编号 | 要求 | 验收标准 |
|---|---|---|
| NFR-001 | 设计器冷启动 | 打开已有页面到可编辑 <= 3 秒 |
| NFR-002 | Schema 保存 | 单页 <= 200 组件、<= 500KB Schema 时保存 <= 2 秒 |
| NFR-003 | 运行态加载 | 已发布页面加载 <= 2 秒 |
| NFR-004 | 热更新 | 发布后运行态重新拉取 Schema 并生效 <= 5 秒 |
| NFR-005 | 拖拽流畅度 | 字段拖拽、span 调整、列宽调整过程中帧率 >= 30 fps |
| NFR-006 | 表格规模 | 1000 行运行态表格虚拟滚动无明显卡顿；P0 默认展示行数建议 <= 100 |
| NFR-007 | 结构规模 | 单页总组件 <= 200，布局容器 <= 100，容器嵌套深度 <= 8，单容器子节点 <= 80 |

### 3.2 安全要求

| 编号 | 要求 | 验收标准 |
|---|---|---|
| NFR-008 | 禁止任意脚本 | P0/P1 默认禁止任意 JS、任意 CSS、直接 DOM 修改 |

## 4. 约束与假设

### 4.1 技术约束

| 编号 | 约束 | 说明 |
|---|---|---|
| C-001 | 主体必须是 PC Web | P0 不做移动端优先设计 |
| C-002 | 主体不使用自由绝对定位 | 页面主体采用 Grid/Flex/Split/Scroll/Sticky 等 Web 布局 |
| C-003 | Schema 是运行时契约 | 设计器输出的 PageSchema 直接驱动运行态渲染 |
| C-004 | PageSchema 结构固定 | P0 以 `root + components` 为权威结构 |
| C-005 | EntryTable 结构固定 | P0 表格列使用 `columns`，明细使用 `detail`，不使用 `children` 表达列 |
| C-006 | 版本语义固定 | `version` 是正整数；`status` 是 `draft/published/archived`；业务状态使用 `bizState` |
| C-007 | P0 动态规则受限 | P0 仅提供规则入口和静态状态/权限；完整动态 RuleSchema 为 P1 |
| C-008 | P0 嵌套表格深度受限 | P0 只允许 Level 1 明细；Level 2 为 P1，Level 3 为 P2 |

### 4.2 业务假设

| 编号 | 假设 | 说明 |
|---|---|---|
| A-001 | 平台已有业务对象元数据 | T-207 从 BO、Field、EntryModel、StateMachine 获取设计输入 |
| A-002 | 平台已有用户和权限基础能力 | P0 使用静态权限入口，P1 才接入真实角色权限引擎 |
| A-003 | 平台允许声明式运行时 | 运行态能加载 PageSchema 并由 RuntimeRenderer 渲染 |
| A-004 | 实施顾问是主要设计者 | 产品优先保障企业实施效率，而不是面向普通终端用户 |
| A-005 | 自动化测试平台可使用 Playwright 类工具 | 抓手和 testId 面向真实 DOM 操作设计 |

## 6. 复杂度等级判定

### 6.1 判定等级

```text
qualityLevel = L4
```

### 6.2 触发条件

| L3/L4 触发条件 | 是否命中 | 依据 |
|---|---|---|
| 涉及 3 个以上用户角色 | 是 | 业务顾问、实施顾问、前端开发、后端开发、测试/运维、平台管理员 |
| 涉及设计态 / 预览态 / 运行态 | 是 | RuntimeRenderer 同源渲染三模式 |

| 涉及复杂 UI 工作台 | 是 | 五区 IDE、画布、资源区、Inspector、Analyzer |

## 8. 核心业务对象与术语表

| 术语 | 定义 | 不允许混用 |
|---|---|---|
| BusinessObject | 业务对象，字段和分录的元数据来源 | 不等同于 PageSchema |
| Field | 业务字段或页面字段节点，需区分 BO Field 与 FieldNode | 不等同于 TableColumn |
| EntryTable | 企业单据分录表格组件 | 不等同于普通 HTML table |
| TableColumn | EntryTable 的列定义，存放在 `columns[]` | 不作为顶层 `components.{id}` |
| PageSchema | 页面运行时元数据，包含 root、components、rules、permissions 等 | 不等同于设计器 UI 状态 |
| RuntimeRenderer | 根据 PageSchema 渲染设计态、预览态、运行态页面的同源渲染器 | 不等同于 DesignerOverlay |
| DesignerOverlay | 设计态辅助层，绘制选中框、抓手、投放点、吸附线 | 不改变业务 DOM 语义 |
| Analyzer | 发布前质量分析器，输出 Error/Warning/Suggestion | 不等同于测试执行报告 |
| SchemaNormalizer | 保存/发布前归一化器 | 不执行业务规则求值 |
| version | Schema 正整数版本号 | 不写 `draft` / `published` |
| status | Schema 生命周期状态：draft/published/archived | 不表示业务单据状态 |
| bizState | 运行态业务状态，如 draft/submitted/approved/archived | 不表示 Schema 版本状态 |
| Level 0/1/2/3 | 表格嵌套层级；Level 0 是主表，Level 1 是一层明细 | P0 不允许 Level 2 |
| PT/IX/PS | 阶段 2 原型图、交互流、页面状态标识 | 不等同于测试用例 TC |
| data-testid | 自动化测试稳定定位 | 不依赖中文标题或 CSS class |

## 9. P0 业务闭环样例

### 9.1 样例一：销售订单页面从设计到发布

| 环节 | 描述 |
|---|---|
| 入口 | 实施顾问打开 T-207，选择业务对象 `SalesOrder` |
| 数据 | BO 包含单据头字段、分录 `entries`、状态 `draft/submitted/approved` |
| 设计 | 系统生成 PageRoot、StickyBar、SplitPane、HeaderForm、EntryTable、SummaryBar、ActionBar |
| 布局 | 顾问使用 Grid 调整基础字段，用 Split 设置右侧审批/附件面板，用 Sticky 固定底部操作 |
| 表格 | 顾问配置物料、数量、单价、金额列，设置冻结列、汇总行和 Level 1 批次明细 |
| 权限 | 顾问配置金额字段只读、提交按钮可见、归档状态不可编辑 |
| 质量 | Analyzer 检查字段绑定、表格层级、布局溢出、权限冲突、Schema size |
| 发布 | 无 Error 后保存 published 版本，生成 schemaHash 和发布证据包 |
| 运行 | RuntimeRenderer 加载 published Schema，终端用户填写销售订单 |
| 可验收输出 | `page_sales_order` published 版本、无 Analyzer Error、Playwright 可完成字段拖入/列宽调整/发布阻断用例 |

### 9.2 样例二：表格 Level 2 被阻断

| 环节 | 描述 |
|---|---|

### 9.3 样例三：自动化测试模拟真实用户
