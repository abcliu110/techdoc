## 1. 设计目标

表单元素与字段体系要回答一个核心问题：

```text
设计器如何把业务对象字段，转化为可布局、可绑定、可校验、可授权、可运行的企业级 Web 字段控件。
```

T-207 的字段体系不是简单控件库，也不是彩色组件墙。它必须支撑企业单据、主数据、审批页和业务工作台中的真实字段表达。

字段体系必须做到：

1. 能从业务对象字段自动生成合适的编辑器。
2. 能在 `GridLayout` 中表现为表单字段。
3. 能在 `EntryTable` 中表现为表格列和单元格编辑器。
4. 能承载数据绑定、校验、权限、状态、格式化和默认值。
5. 能在设计态、预览态、运行态保持一致。
6. 能被 Analyzer 检查绑定缺失、类型不匹配、权限冲突和交互风险。
7. 能提供稳定抓手和测试标识，让自动测试平台模拟点击、输入、选择、上传和校验。

## 2. 核心定位

### 2.1 字段体系是什么

| 定义 | 说明 |
|---|---|
| 业务字段到 UI 编辑器的映射层 | 根据 BO 字段类型、业务属性和场景选择编辑器 |
| 企业字段组件库 | TextInput、NumberInput、MoneyInput、DatePicker、ReferencePicker、AttachmentUploader 等 |
| 字段状态系统 | required、readonly、visible、disabled、error、warning、loading |
| 数据绑定入口 | 字段绑定 BO 字段、静态值或计算值 |
| 规则权限承载点 | 后续规则、权限、流程状态作用到字段节点 |
| 可测试交互单元 | 每个字段有稳定 `data-testid`、输入区、错误区和可操作抓手 |

### 2.2 与工作台规则入口的边界

字段体系只承担 P0 字段级基础能力，不把完整业务规则引擎提前塞进字段配置：

| 能力 | P0 所属 | 说明 |
|---|---|---|
| 字段基础校验 | `FieldNode.validation` | required、长度、数值范围、格式、错误文案 |
| 字段静态状态 | `FieldNode.state` | visible、required、readonly、disabled、hidden |
| 字段静态权限 | `FieldNode.permission` / Inspector 权限 Tab | visible、editable、readonly、mask |
| 完整业务规则 | P1 `RuleSchema` | visibleWhen、requiredWhen、readonlyWhen、computeValue、crossFieldValidation |
| 规则影响分析 | P1 规则权限流程文档 | 字段被哪些规则、角色、流程状态影响 |

Analyzer 在 P0 只合并字段基础校验、静态状态和静态权限；进入 P1 后再合并 `RuleSchema` 的动态状态驱动结果。这样可以避免 P0 变成不可控的规则平台，同时保留后续扩展点。

### 2.3 字段体系不是什么

| 非目标 | 原因 |
|---|---|
| 通用前端组件市场 | P0 只做企业表单字段，不做任意 UI 组件 |
| 表格列完整设计 | TableColumn、复杂表头、列菜单归强表格文档 |
| 规则引擎完整实现 | visibleWhen、readonlyWhen、calculation 归规则权限流程文档 |
| 移动端字段库 | P0 聚焦 PC Web |
| 任意 CSS 样式编辑器 | 字段视觉由设计系统和 Token 控制 |
| 自由脚本控件 | 字段行为必须可分析、可治理、可发布 |

## 3. 设计原则

### 3.1 业务语义优先

字段不是孤立输入框，而是业务对象属性在页面上的表现。

| 业务语义 | UI 表现 |
|---|---|
| 客户 | ReferencePicker |
| 组织 | OrgPicker / ReferencePicker |
| 日期 | DatePicker |
| 金额 | MoneyInput |
| 数量 | NumberInput |
| 状态 | StatusSelect / TagDisplay |
| 附件 | AttachmentUploader |

### 3.2 编辑器自动选择优先

从 BO 字段拖入页面时，系统应根据字段元数据自动选择编辑器，实施顾问可以在受控范围内替换。

```text
BO Field
  -> FieldTypeResolver
  -> EditorRegistry
  -> FieldNode
  -> RuntimeRenderer
```

### 3.3 受控替换优先

用户不能把任意编辑器绑定到任意字段类型。替换必须满足类型兼容：

| 字段类型 | 允许编辑器 |
|---|---|
| string | TextInput、Textarea、CodeInput(P1) |
| number | NumberInput、PercentInput |
| money | MoneyInput |
| date | DatePicker、DateRangePicker(P1) |
| enum | Select、RadioGroup、Segmented(P1) |
| boolean | Switch、Checkbox |
| reference | ReferencePicker、TreePicker |
| attachment | AttachmentUploader |

### 3.4 同一 FieldNode，多场景表现

同一个业务字段在不同容器中表现不同：

| 场景 | 表现 |
|---|---|
| GridLayout | 标签 + 输入控件 + 校验提示 |
| FlexLayout | P0 不直接承载字段；若迁移或旧 Schema 中出现字段，降级为紧凑状态文本或轻量输入 |
| EntryTable | TableColumn + CellEditor |
| Readonly Preview | 文本、标签、链接、金额格式化 |

### 3.5 状态可叠加

字段状态不是互斥的。一个字段可能同时是 required、readonly、warning、permission-limited。

| 状态 | 说明 |
|---|---|
| visible | 是否渲染 |
| required | 是否必填 |
| readonly | 是否只读但可见 |
| disabled | 不可操作且弱化显示 |
| error | 阻断性校验错误 |
| warning | 非阻断性风险 |
| loading | 引用数据或远程校验加载中 |
| permissionDenied | 权限禁止查看或编辑 |

### 3.6 自动化测试优先

每个字段必须提供稳定测试标识：

| 对象 | 定位 |
|---|---|
| 字段根节点 | `data-testid="field-node-{fieldId}"` |
| 字段标签 | `data-testid="field-label-{fieldId}"` |
| 字段输入区 | `data-testid="field-editor-{fieldId}"` |
| 字段错误区 | `data-testid="field-error-{fieldId}"` |
| 字段帮助区 | `data-testid="field-help-{fieldId}"` |
| 字段权限状态 | `data-testid="field-permission-{fieldId}"` |
| 编辑器下拉面板 | `data-testid="field-popup-{fieldId}"` |

| 能力 | P2 |
|---|---|

### 4.4 明确禁止

| 禁止 | 阶段 | 原因 |
|---|---|---|
| 任意字段绑定任意编辑器 | P0/P1/P2 禁止默认开放 | 类型不安全、不可分析 |
| 字段级任意 JS | P0 禁止 | 无法治理和发布校验 |
| 字段级任意 CSS | P0 禁止 | 破坏企业视觉一致性 |
| 把表格列完整设计塞入本文件 | P0/P1/P2 禁止 | 强表格文档负责 |
| 字段直接操作 DOM | P0/P1/P2 禁止 | 破坏 Schema 驱动 |

## 5. 总体架构

### 5.1 字段体系结构

```text
BusinessObject
  └─ BusinessField
      ↓
FieldTypeResolver
  ├─ FieldTypeRegistry
  ├─ EditorRegistry
  ├─ BindingResolver
  ├─ ValidationProfileResolver
  ├─ PermissionStateResolver
  └─ FieldAnalyzer
      ↓
FieldNode / TableColumn
      ↓
RuntimeRenderer
      ↓
DesignerOverlay
```

### 5.2 模块职责

| 模块 | 职责 | P0 |
|---|---|---|
| FieldTypeRegistry | 注册业务字段类型和字段元数据 | 是 |
| EditorRegistry | 注册编辑器、默认属性、Inspector 配置 | 是 |
| FieldTypeResolver | 根据字段类型选择编辑器 | 是 |
| BindingResolver | 解析字段绑定路径、静态值、默认值 | 是 |
| ValidationProfileResolver | 生成基础校验规则 | 是 |
| PermissionStateResolver | 合并字段静态权限和字段状态 | 是 |
| EditorCompatibilityMatrix | 校验编辑器与字段类型的兼容性，支持插件扩展注册 | 是 |
| FieldAnalyzer | 检查字段绑定、类型、状态、校验风险 | 是 |

P0 的 `PermissionStateResolver` 只处理字段自身的静态权限和静态状态，不读取角色、流程实例或业务数据状态。角色权限、流程状态权限、数据状态权限由 P1 `PermissionEngine` 求值后再回写到运行态字段状态。

### 5.3 字段生成链路

```text
用户拖入 BO 字段
  -> FieldTypeResolver 选择默认 editor
  -> 生成 FieldNode
  -> DropRuleEngine 放入 GridLayout / Section
  -> BindingResolver 补齐 binding
  -> LayoutNormalizer 补齐 FieldNode.layout
  -> RuntimeRenderer 渲染字段
  -> Analyzer 检查字段绑定和布局风险
```

这里的 `LayoutNormalizer` 引用 03-Web 布局引擎中的同名模块，只负责补齐字段布局默认值，例如 `span`、`minWidth`、`labelWidth`、`labelPosition`，并清理非法布局属性。字段绑定不属于布局引擎职责，必须由 `BindingResolver` 负责。

## 6. FieldNode 基础模型

### 6.1 Schema

```json
{
  "id": "field_customer",
  "type": "Field",
  "name": "customerId",
  "title": "客户",
  "businessField": {
    "boId": "SalesOrder",
    "fieldCode": "customerId",
    "fieldType": "reference",
    "required": true
  },
  "editor": {
    "type": "ReferencePicker",
    "props": {
      "displayField": "name",
      "valueField": "id",
      "searchable": true
    }
  },
  "binding": {
    "type": "boField",
    "path": "salesOrder.customerId"
  },
  "layout": {
    "span": 2,
    "minWidth": 320,
    "labelPosition": "left",
    "labelWidth": 96
  },
  "state": {
    "required": true,
    "readonly": false,
    "visible": true,
    "disabled": false
  },
  "validation": [
    {
      "type": "required",
      "message": "客户不能为空"
    }
  ],
  "design": {
    "selectable": true,
    "draggable": true,
    "resizable": true,
    "testId": "field-node-field_customer"
  }
}
```

### 6.2 属性说明

| 属性 | 说明 | P0 |
|---|---|---|
| id | 字段节点唯一 ID | 是 |
| type | 固定为 Field | 是 |
| name | 技术名称 | 是 |
| title | 展示标题 | 是 |
| businessField | BO 字段来源 | 是 |
| editor | 编辑器类型和属性 | 是 |
| binding | 数据绑定 | 是 |
| layout | 字段布局属性 | 是 |
| state | 字段基础状态 | 是 |
| validation | 字段基础校验 | 是 |
| design | 设计态元数据 | 是 |

### 6.3 状态与校验语义

`state.required` 和 `validation.required` 不等价：

| 项 | 职责 | 运行效果 |
|---|---|---|
| `state.required` | 显示必填标记和字段状态 | 标签前显示必填标记，参与只读/隐藏冲突分析 |
| `validation.required` | 提交和失焦校验 | 空值时产生错误并阻断发布/提交 |

如果字段显示为必填但没有 required 校验，Analyzer 输出 `FIELD_REQUIRED_NO_VALIDATION`，级别为 Error。原因是用户会看到必填标记，但系统不会阻断空值提交，这在企业单据里属于数据质量风险。

当顾问在状态 Tab 勾选 `required=true` 时，设计器应自动在 `FieldNode.validation` 中补一条 required 校验；当顾问在校验 Tab 添加 required 校验时，设计器也应自动把 `state.required` 置为 true。`FIELD_REQUIRED_NO_VALIDATION` 用于兜底检查导入旧 Schema、手工编辑 Schema 或插件生成的不规范节点。

`displayMode` 与 `state.readonly` 的关系：

| 配置 | 语义 | 结果 |
|---|---|---|
| `displayMode=edit` | 优先使用可编辑控件 | 若 `state.readonly=true`，仍强制切到只读展示 |
| `displayMode=readonly` | 强制只读展示 | 无论权限是否可编辑，都显示只读 renderer |
| `displayMode=auto` | 根据权限、状态、预览模式自动选择 | P0 默认推荐 |
| `state.readonly=true` | 字段当前状态只读 | 强制使用 readonly renderer，不能仅禁用 input |

Readonly renderer 必须是可读展示，不应只是灰色禁用输入框。

P0 中 `state.readonly` 和 `permission.readonly` 都属于静态只读配置，最终只读状态按并集求值：

```text
finalReadonly =
  state.readonly
  OR permission.readonly
  OR PermissionEngine.dynamicReadonly(P1)
```

P0 不接入 `PermissionEngine.dynamicReadonly`，但字段渲染层必须预留该覆盖入口。这样状态 Tab 和权限 Tab 不会形成两套互相竞争的只读逻辑。

### 6.4 businessField

| 属性 | 说明 | P0 |
|---|---|---|
| boId | 业务对象 ID | 是 |
| fieldCode | 字段编码 | 是 |
| fieldType | 业务字段类型 | 是 |
| required | 元数据必填 | 是 |
| length | 字符串长度 | 是 |
| precision | 数字精度 | 是 |
| scale | 小数位 | 是 |
| enumCode | 枚举编码 | 是 |
| referenceBo | 引用对象 | 是 |

### 6.5 binding

| 绑定类型 | P0 | 说明 |
|---|---|---|
| boField | 是 | 绑定业务对象字段 |
| staticValue | 是 | 静态展示或默认值 |
| expression | P1 | 公式表达式 |
| context | P1 | 当前用户、组织、日期等上下文 |
| remote | P1 | 远程数据服务 |

P0 数据 Tab 只展示 `boField` 和 `staticValue`。`expression`、`context`、`remote` 归 P1 数据 Tab 扩展，不允许在 P0 Inspector 中出现可编辑入口。

P0 中 FieldNode 必须有 `binding` 或 `staticValue`。未绑定字段只能作为设计态占位，发布时 Analyzer Error。

## 7. P0 编辑器设计

### 7.1 编辑器矩阵

| 编辑器 | 适用字段 | P0 | 说明 |
|---|---|---|---|
| TextInput | string | 是 | 单行文本 |
| Textarea | string | 是 | 多行备注 |
| NumberInput | number / integer | 是 | 数量、比例 |
| MoneyInput | money / decimal | 是 | 金额，带币别和格式化 |
| DatePicker | date | 是 | 日期 |
| DateTimePicker | datetime | 是 | 日期时间 |
| Select | enum | 是 | 下拉枚举 |
| RadioGroup | enum | 是 | 少量枚举 |
| Checkbox | boolean | 是 | 布尔选择 |
| Switch | boolean | 是 | 状态开关 |
| ReferencePicker | reference | 是 | 基础资料、客户、组织等引用 |
| TreePicker | reference / tree | 是 | 分类、组织树 |
| AttachmentUploader | attachment | 是 | 附件上传和查看 |
| ReadonlyText | any | 是 | 只读展示 |
| LinkDisplay | url / reference | 是 | 链接或关联对象跳转 |

TagDisplay 和 StatusBadge 在 P0 中不作为独立 `editor.type` 注册，而是 ReadonlyText 的展示变体，由字段类型、枚举状态和 `displayMode=tag/status` 控制。后续如需独立样式和交互，再在设计系统组件库中拆分。

TagDisplay 和 StatusBadge 的渲染逻辑：

| 模式 | 触发方式 | Schema 影响 |
|---|---|---|
| text | 默认只读展示 | `editor.type=ReadonlyText` |
| tag | 枚举字段或 `displayMode=tag` | 不新增 editor.type，只切换 ReadonlyText 渲染模式 |
| status | 状态字段或 `displayMode=status` | 不新增 editor.type，只切换 ReadonlyText 渲染模式 |

视觉差异由设计系统组件内部实现，不改变 FieldNode 结构。

### 7.2 TextInput

| 属性 | P0 | 说明 |
|---|---|---|
| maxLength | 是 | 最大长度 |
| placeholder | 是 | 占位提示 |
| trim | 是 | 保存前去首尾空格 |
| uppercase | 是 | 编码类字段可转大写 |
| lowercase | 是 | 邮箱、URL、编码类字段可转小写 |

Analyzer：

| 规则 | 级别 | 说明 |
|---|---|---|
| TEXT_LENGTH_OVER_LIMIT | Error | maxLength 超过 BO 字段长度 |
| TEXT_REQUIRED_EMPTY | Error | 必填字段为空 |

### 7.3 NumberInput

| 属性 | P0 | 说明 |
|---|---|---|
| min | 是 | 最小值 |
| max | 是 | 最大值 |
| precision | 是 | 总精度 |
| scale | 是 | 小数位 |
| step | 是 | 步长 |
| roundingMode | 是 | 四舍五入策略，P0 固定为 half-up |

P0 数值格式化规则：

| 项 | 规则 |
|---|---|
| 小数位 | 按 `scale` 固定展示和提交 |
| 舍入 | 使用 half-up 四舍五入 |
| 空值 | 不显示 0，除非 defaultValue 明确为 0 |
| 非法输入 | 保留输入态并显示校验错误，不自动吞掉字符 |

Analyzer：

| 规则 | 级别 | 说明 |
|---|---|---|
| NUMBER_PRECISION_MISMATCH | Error | 编辑器精度超过字段精度 |
| NUMBER_RANGE_INVALID | Error | min 大于 max |

### 7.4 MoneyInput

| 属性 | P0 | 说明 |
|---|---|---|
| currencyField | 是 | 币别字段 |
| precision | 是 | 金额精度 |
| scale | 是 | 小数位 |
| thousandSeparator | 是 | 千分位 |
| showCurrency | 是 | 是否显示币别 |
| roundingMode | 是 | P0 固定为 half-up |

MoneyInput 的 `precision` 和 `scale` 以 `businessField.precision / businessField.scale` 为唯一权威来源。Inspector 中可展示这两个值，但 P0 不允许覆盖 BO 字段精度，也不回写 BO 元数据。若编辑器 props 与 BO 元数据不一致，Analyzer 输出 `MONEY_SCALE_MISMATCH` 或 `NUMBER_PRECISION_MISMATCH`。

MoneyInput 必须支持企业财务格式：

```text
1,234,567.89 CNY
```

Analyzer：

| 规则 | 级别 | 说明 |
|---|---|---|
| MONEY_CURRENCY_MISSING | Warning | 金额字段未绑定币别 |
| MONEY_SCALE_MISMATCH | Error | 小数位和 BO 字段不一致 |

MoneyInput 保存和展示均按 BO 字段 `precision/scale` 处理，P0 舍入策略固定为 half-up。税额、折扣额等更复杂财务舍入规则由后续领域规则承接，不能在 P0 字段控件里开放任意公式。

### 7.9 AttachmentUploader

| 属性 | P0 | 说明 |
|---|---|---|
| maxFiles | 是 | 最大文件数 |
| maxFileSize | 是 | 单文件大小 |
| accept | 是 | 文件类型 |
| uploadMode | 是 | immediate / submit |
| preview | 是 | 是否预览 |

`uploadMode` 语义：

| 模式 | P0 行为 | 失败处理 |
|---|---|---|
| `immediate` | 选择文件后立即串行上传，成功后写入附件临时 ID | 单文件失败只标记该文件错误，不影响已成功文件 |
| `submit` | 选择文件后进入本地待提交队列，随表单提交时串行上传 | 任一必传附件失败时阻断提交，保留队列和错误原因 |

`submit` 模式的 P0 提交流程：

```text
1. 用户选择文件，前端只记录本地待提交队列。
2. 用户提交表单时，前端先调用附件临时上传接口，逐个上传并获得 tempAttachmentId。
3. 全部附件上传成功后，表单保存请求携带 tempAttachmentId 列表。
4. 后端在表单保存事务中把 tempAttachmentId 绑定到单据。
5. 如果附件上传失败，表单保存不发起，错误显示在附件字段。
6. 如果表单保存失败，tempAttachmentId 保持临时状态，可重试或由服务端定时清理。
```

P0 只要求串行上传、临时附件 ID 和基础预览，断点续传、并发上传、病毒扫描回调、在线编辑归 P1/P2。

#### 7.9.1 tempAttachmentId 后端生命周期

| 项目 | 约束 |
|---|---|
| 有效期 | `tempAttachmentId` 生成后 24 小时内有效 |
| 清理机制 | 服务端定时任务每日清理超过 24 小时且未绑定的临时附件 |
| 绑定时机 | 表单保存事务中，`tempAttachmentId` 绑定到业务单据后变为已持久化 |
| 放弃编辑 | 用户关闭页面或切换页面后，未绑定附件由清理任务回收 |
| 重试策略 | 上传失败的本地队列项可重新上传并获得新的 `tempAttachmentId` |

Analyzer：

| 规则 | 级别 | 说明 |
|---|---|---|
| ATTACHMENT_SIZE_TOO_LARGE | Warning | 文件大小限制过大 |
| ATTACHMENT_TYPE_UNSAFE | Error | 允许不安全文件类型 |

### 7.10 ReadonlyText / LinkDisplay

ReadonlyText 和 LinkDisplay 属于 P0 只读展示编辑器，用于运行态只读、审批只读、引用跳转和 URL 展示。

| 编辑器 | 适用字段 | P0 属性 | Analyzer |
|---|---|---|---|
| ReadonlyText | any | formatter、emptyText、copyable | `READONLY_FORMATTER_INVALID` Warning |

以下编辑器出现在 P1 能力表中，但不进入 P0 编辑器矩阵。本文只定义骨架，详细交互在 P1 字段扩展设计中展开。

### 8.2 EditorCompatibilityMatrix（编辑器兼容性矩阵）

编辑器兼容性矩阵定义每种 BO 字段类型允许使用的编辑器集合。系统按矩阵校验替换操作，插件扩展编辑器时也必须注册兼容字段类型，不能绕过 Analyzer。

| 替换 | 是否允许 | 说明 |
|---|---|---|
| TextInput -> Textarea | 是 | string 类型可替换 |
| Select -> RadioGroup | 是 | 枚举项较少时 |
| ReferencePicker -> TreePicker | 条件 | referenceBo 必须具备树结构 |
| NumberInput -> MoneyInput | 否 | money 字段才允许 MoneyInput |
| AttachmentUploader -> TextInput | 否 | 类型不兼容 |
| DatePicker -> TextInput | 否 | 防止日期失去类型校验 |

### 8.3 EditorRegistry 结构

```json
{
  "editorType": "ReferencePicker",
  "fieldTypes": ["reference"],
  "defaultProps": {
    "searchable": true,
    "popupColumns": ["code", "name"]
  },
  "inspector": {
    "tabs": ["props", "binding", "validation", "permission"],
    "props": {
      "editable": ["placeholder", "displayField", "popupColumns"],
      "readonly": ["valueField", "referenceBo"]
    }
  },
  "runtime": {
    "component": "ReferencePickerRuntime",
    "readonlyRenderer": "ReferenceReadonlyText"
  }
}
```

`EditorRegistry.inspector` 只声明编辑器希望暴露的属性集合，最终 Tab 名称、分组、权限和阶段边界以工作台 Inspector 规范为准：

| 属性 | 说明 |
|---|---|
| `tabs` | 编辑器适用的 Inspector Tab 标识，映射到工作台的属性/数据/校验/权限等 Tab |
| `props.editable` | 运行态可由实施顾问配置的编辑器属性 |
| `props.readonly` | 只读展示或由 BO 元数据决定的属性，例如 referenceBo、valueField、precision、scale |
| `props.stage` | P1 后可扩展，用于声明属性阶段和插件来源 |

P0 中 Inspector 具体控件布局由工作台统一生成，字段体系不单独定义一套 UI 表单 Schema。

### 8.4 左侧资源标签与编辑器名称对照

工作台左侧资源面板可以使用更短的业务标签，但 Schema 与运行态必须使用稳定编辑器名称：

| 资源面板标签 | Schema 编辑器 / 组件 | 说明 |
|---|---|---|
| Attachment | AttachmentUploader | 文件上传字段 |
| TextDisplay | ReadonlyText | 通用只读文本展示 |
| Link | LinkDisplay | URL 或引用对象跳转 |
| Tag | TagDisplay / ReadonlyText variant | 枚举或状态只读展示 |
| StatusBadge | StatusBadge / ReadonlyText variant | 状态字段展示 |
| HelpText | FieldWrapper.helpText | 字段辅助说明，不是独立 FieldNode |

```text
  ├─ Label

```

| 属性 | P0 | 说明 |
|---|---|---|
| span | 是 | 占 Grid 列数 |

| minWidth | 是 | 最小宽度 |

| showRequiredMark | 是 | 显示必填标记 |

### 9.2 EntryTable 中的字段

字段进入 EntryTable 时不直接生成 FieldNode，而是生成 TableColumn：

```text
BusinessField
  -> TableColumn
    -> CellEditor
    -> CellReadonlyRenderer
```

| FieldNode 概念 | EntryTable 对应 |
|---|---|
| title | column.title |
| editor.type | column.editor.type |
| binding.path | column.dataIndex |
| validation | column.validation |
| permission | column.permission |
| readonlyRenderer | column.readonlyRenderer |

TableColumn 不是 FieldNode，但必须携带字段编辑所需的受控子集：

| TableColumn 子集 | 来源 | P0 要求 |
|---|---|---|
| `businessField` | BO 字段 | 必须保留字段类型、精度、枚举、引用对象 |
| `editor` | EditorRegistry | 与 FieldNode 使用同一编辑器兼容规则 |
| `binding/dataIndex` | EntryTable 数据源 | 必须指向分录行字段 |
| `validation` | FieldNode.validation 子集 | required、范围、格式 |
| `permission` | 字段静态权限子集 | visible、editable、readonly、mask |
| `displayMode` | FieldNode.displayMode | 若表格整体只读则强制 readonlyRenderer |
| `readonlyRenderer` | EditorRegistry | 表格只读态使用 |

EntryTable 的 `column.dataIndex` 使用行字段名，例如 `materialId`、`qty`、`price`，不包含 `entries` 前缀和数组索引。运行时由 TableEngine 根据当前编辑行注入行上下文。

P0 中 `EntryTable.detailDepth` 最大为 1。`DetailPanel` 或 `DetailTable` 只能作为一层明细承载，超过一层时由 Analyzer 输出 `TABLE_DETAIL_DEPTH_EXCEEDED`。多层母子表和复杂嵌套归 P1。

层级术语对照：

| 术语 | 含义 |
|---|---|
| Level 0 | 主分录表，即 EntryTable 自身 |
| Level 1 | 一层明细，`detailDepth=1`，DetailPanel 或 DetailTable 第一层 |
| Level 2 | 子明细，`detailDepth=2`，P1 才支持 |

EntryTable 中的字段分析路径：

```text
路径 A：从 FieldNode 拖入 EntryTable
  FieldNode
    -> FieldTypeResolver 选择 CellEditor
    -> TableColumn 子集生成器
    -> TableColumn
    -> FieldAnalyzer 分析 TableColumn 子集

路径 B：从 BO 分录字段直接生成 TableColumn
  BO EntryField
    -> TableColumn 生成器
    -> TableColumn
    -> FieldAnalyzer 分析 TableColumn
```

FieldAnalyzer 在 EntryTable 场景输入的是 TableColumn，但 TableColumn 保留了 `editor.type`、`binding/dataIndex`、`validation`、`permission` 等受控子集，因此主表单字段和表格列可复用同一套基础检查逻辑。

FieldAnalyzer 在表格场景分析的是 `TableColumn`，不是 `FieldNode`。列宽、冻结列、行操作、汇总行和明细表格归强表格文档展开。

### 9.3 Readonly 展示

字段只读时不应简单禁用输入框，而应切换为企业可读展示：

| 字段类型 | 只读展示 |
|---|---|
| string | 文本 |
| number | 数字格式 |
| money | 金额 + 币别 |
| date | 格式化日期 |
| enum | 文本 / 标签 |
| reference | 展示名称，可跳转 |

## 11. 字段 Analyzer

### 11.1 Analyzer 输出结构

```json
{
  "severity": "error",
  "code": "FIELD_BINDING_MISSING",
  "message": "客户字段缺少数据绑定。",
  "nodeId": "field_customer",
  "schemaPath": "components.field_customer.binding",
  "quickFix": "绑定到 salesOrder.customerId"
}
```

#### 11.1.1 schemaPath 格式规范

`schemaPath` 使用 JSONPath 风格，表示问题在 PageSchema 中的位置，从 PageSchema 根开始，不使用 `$` 前缀。

| 场景 | schemaPath 示例 | 说明 |
|---|---|---|
| 主表单字段 | `components.field_customer.binding` | components 下的顶层 FieldNode |
| Section 内字段 | `components.section_basic.children[0].binding` | children 数组中的节点 |
| GridLayout 子字段 | `components.grid_basic.children[2]` | children 数组下标 |
| EntryTable 列 | `components.entry_table.columns[3]` | EntryTable columns 数组下标 |
| EntryTable 列编辑器属性 | `components.entry_table.columns[3].editor.props.displayMode` | 列编辑器属性路径 |

规则：

1. 顶层组件优先使用 `components.{nodeId}`。
2. 数组子节点使用 `[0]`、`[1]` 形式。
3. 命名属性使用点号，例如 `.binding`、`.editor`、`.layout`。
4. Analyzer 必须使用同一格式，便于画布、Inspector 和 Analyzer 三向定位。

### 11.2 P0 规则清单

| 规则 | 级别 | 说明 |
|---|---|---|
| FIELD_BINDING_MISSING | Error | 发布字段缺少 binding 或 staticValue |
| FIELD_TYPE_EDITOR_MISMATCH | Error | 字段类型和编辑器不兼容 |
| FIELD_REQUIRED_NO_VALIDATION | Error | 必填字段缺少 required 校验 |
| FIELD_LABEL_MISSING | Warning | 字段标题缺失 |
| FIELD_HELP_TOO_LONG | Warning | helpText 过长 |
| FIELD_PERMISSION_CONFLICT | Warning | hidden+required、readonly+required 且无默认值、mask+editable 等静态冲突 |
| FIELD_DISPLAYMODE_CONFLICT | Warning | displayMode=edit 但 readonly/permission 强制只读，或 displayMode=readonly 但仍配置可编辑交互 |
| FIELD_READONLY_REQUIRED_RISK | Warning | 只读必填字段可能无法提交 |
| REFERENCE_TARGET_MISSING | Error | 引用对象缺失 |
| ENUM_OPTIONS_EMPTY | Error | 枚举字段没有选项 |
| TREE_PICKER_SOURCE_INVALID | Error | treeSource 不是 BO treeReference 或系统内置树 |
| ATTACHMENT_TYPE_UNSAFE | Error | 附件允许危险类型 |

字段宽度不足不在 FieldAnalyzer 中重复实现。它统一由布局引擎 LayoutAnalyzer 输出 `GRID_FIELD_MIN_WIDTH_RISK`，触发条件为字段实际宽度小于 `FieldNode.layout.minWidth` 或编辑器推荐最小宽度。FieldAnalyzer 只消费该结果并在字段问题列表中关联展示。

P0 Analyzer 对权限相关规则的求值范围：

| 规则 | P0 求值范围 | P1 扩展 |
|---|---|---|
| FIELD_PERMISSION_CONFLICT | 只检查静态冲突：hidden+required、readonly+required 且无默认值、mask+editable | 合并角色、流程、数据状态后的动态冲突 |
| FIELD_DISPLAYMODE_CONFLICT | 只检查 displayMode 与 `state.readonly` / `permission.readonly` 的静态冲突 | 合并 PermissionEngine 动态只读结果 |

### 11.3 Quick Fix 边界

P0 Quick Fix 只允许单条受控修复：

| 问题 | Quick Fix |
|---|---|
| 缺少 binding | 绑定回原 BO 字段 |
| 必填缺少校验 | 添加 required 校验 |
| label 缺失 | 使用 BO 字段显示名 |
| editor 不兼容 | 切回默认编辑器 |
| 枚举选项为空 | 绑定 enumCode |

批量修复、批量忽略和影响分析归 P1。Error 级问题不可忽略，必须修复后才能发布。

## 14. P0 验收标准

| 标准 | 验收方式 |
|---|---|
| BO 字段可拖入生成字段 | 从销售订单 BO 拖入客户、日期、金额等字段 |
| 编辑器自动选择正确 | 客户=ReferencePicker，金额=MoneyInput，日期=DatePicker |
| GridLayout 字段可布局 | 字段可设置 span、labelWidth、labelPosition |
| EntryTable 可生成基础 CellEditor | 字段拖入表格生成 TableColumn + CellEditor |
| 字段绑定可发布 | 所有发布字段有 binding 或 staticValue |
| 校验状态可见 | required、error、warning 可在画布和预览态展示 |
| 权限入口存在 | 字段可配置 visible/editable/readonly/mask |
| Analyzer 阻断错误 | 缺绑定、类型不兼容、枚举无选项等 Error 阻断发布 |
| 自动化测试可操作 | Playwright 可定位字段、输入、选择、上传、校验 |
| P0/P1 边界清晰 | 不把复杂引用、多语言、级联字段、远程校验、角色预览当作 P0 完成 |
