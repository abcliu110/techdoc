# ADR-LOWCODE-CONVERSION-001: 单据转换、反写与引用追踪

> 状态：proposed
> 日期：2026-07-06

## 背景

企业业务系统常见从销售订单生成发货单、从发货单反写订单数量、从发票追溯源单等链路。若把这些能力隐藏在按钮脚本、规则 JSON 或页面跳转中，运行时无法统一保证幂等、权限、审计、失败补偿和引用追踪。

## 决策

将 `Conversion`、`WriteBack`、`LinkTrace` 作为一等元数据：

- `Conversion` 描述源对象、目标对象、字段映射、行映射、过滤条件和幂等策略。
- `WriteBack` 描述反写方向、聚合口径、触发时机、冲突策略和失败补偿。
- `LinkTrace` 描述上查/下查的引用链和权限裁剪策略。
- M0 只保存 DTO 和版本快照，不执行运行时转换。

## 理由

单据链路是业务不变量的一部分，不是 UI 动作。将它建模为一等元数据，才能做发布前校验、升级兼容、权限裁剪、审计回放和故障恢复。

## 否决方案

- 用普通 action 脚本实现转换：表达快，但权限、审计、幂等和升级兼容不可控。
- 用 formula/fetch_from 表达反写：混淆读时计算、写入快照和副作用，容易造成循环和写放大。
- 只保留 link 字段不建 LinkTrace：能存关系，但不能回答上查/下查、权限裁剪和历史追溯。

## 后果

- M2/M3 运行时前必须明确事务边界、一致性级别、失败重放和撤销策略。
- 转换和反写必须走统一写入口，不得绕过权限、审计和规则。
- 引用追踪展示必须按当前用户权限裁剪。

## 验证

- 重复转换不重复生成目标单或副作用。
- 反写失败可重试且幂等。
- 无权限用户通过上查/下查不能探测目标单是否存在。
