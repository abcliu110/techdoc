## 1. 产品定位

**独立商用的企业业务低代码平台**，对标金蝶云苍穹/Frappe 路线（业务元模型平台），而非 Appsmith 类内部工具构建器（依据 BIZ-LOWCODE-001 三类路线分析）。

- 目标用户：企业应用团队、ISV、有业务系统建设需求的中大型客户
- 核心场景：单据类业务系统（订单、审批、主数据）、业务对象全生命周期管理
- 部署形态：多租户 SaaS + 私有化部署（同一套代码，部署模式可切换）
- 核心主张：**业务对象元模型是平台内核，页面是模型的视图**（ADR-LOWCODE-DM-001）

### 1.1 设计依据收敛

本节收敛早期金蝶式业务低代码对比分析的有效结论；早期分析底稿不再作为活跃入口。

设计基准不是“拖拽页面 + 通用 CRUD”，而是金蝶式企业业务低代码能力：

```text
业务对象 / 元数据层
+ 基础资料层
+ 单据头体层
+ 操作 / 状态层
+ 规则 / 事件 / 服务层
+ 单据转换层
+ 反写 / 引用追踪层
+ 流程 / 权限 / 组织层
```

因此本平台必须把以下能力作为一等语义或明确的里程碑预留：

```text
MasterDataSchema
DocumentSchema
EntrySchema
ActionSchema
RuleSchema / EventSchema
WorkflowSchema
PermissionSchema
PageSchema
FlexFieldSchema
OrgRelation / OrgSchema
CodeRuleSchema
LinkConfig / Conversion / WriteBack / LinkTrace
Plugin / Extension lifecycle
```

金蝶侧的关键设计启发：

- KDDM / BOS 的价值在于用元数据描述业务对象、布局、规则、流程、权限和组织，而不是只描述页面。
- DynamicObject 的启发是：运行时数据载体必须带类型、校验和业务语义，不能退化为弱类型 Map / JSON。
- 单据转换、关联配置、反写规则、引用追踪是企业业务低代码区别于内部工具构建器的关键边界。
- 弹性域、多组织业务关系、复杂编码规则属于金蝶式平台的一等业务语义，M0 可以只预留 DTO 和阻断误执行，但不能被普通 JSON 字段、单一 `org_id` 或自增编号替代。

开源项目分工：

| 参考对象 | 借鉴点 | 不照搬点 |
|---|---|---|
| Frappe / ERPNext | DocType、Link、Fetch From、Child Table、Workflow、Hooks | 不直接采用 Python/Frappe 技术栈 |
| NocoBase | Collection、Association、UI Schema、插件体系 | 业务单据和 ERP 语义需要自研补齐 |
| Directus | Fields、Role/Policy/Permission、ItemsService、自动 API | License 和竞品使用限制决定不能作为代码基座 |
| Appsmith / ToolJet / Lowcoder | UI Builder、数据源、查询绑定、内部工具体验 | 不作为业务低代码内核；JS binding 不能承载核心业务规则 |

## 2. 技术选型（本文档新增决策，需评审）

### ADR-LOWCODE-TECH-001：后端 Java 21 + Spring Boot 3.4，前端 React 18 + Ant Design

| 项 | 选择 | 理由 |
|---|---|---|
| 后端 | Java 21 / Spring Boot 3.4.x / Maven 多模块 | 与团队现有能力一致（nms4cloud 同栈），企业级生态成熟，私有化部署客户接受度高 |
| 前端 | React 18 + TypeScript + Ant Design 5 | 团队现有前端栈（React/AntD Pro），设计器类产品生态最丰富 |
| 数据库 | MySQL 8.0（首版唯一支持） | 团队最熟；元数据与业务数据同库不同表；多数据库适配放入 DataSource/Connector 后续版本 |
| 缓存 | Redis | 元数据缓存、会话、分布式锁 |
| ORM | MyBatis-Plus（平台自身元数据表） + 自研动态数据访问层（业务对象数据） | 业务对象表是运行时动态建的，静态 ORM 无法覆盖，必须自研动态层（参照 Frappe Database 层与 NocoBase Repository，见 E-FRAPPE-SRC / E-NOCOBASE-SRC 系列） |
| 表达式引擎 | Aviator（沙箱模式，禁反射/禁 IO） | 规则、默认值、显隐条件、校验都需要表达式；**此项为暂定，对应知识库 L0 缺口"表达式引擎"，需补拆解后终审** |

**否决方案**：Node.js + TS（与 NocoBase 同栈可借鉴生态，但团队 Java 能力无法复用，长期维护成本高）；Python/Frappe 同栈（同理）。

**代价**：Frappe/NocoBase 的代码不能直接复用，只能复用其元模型设计思想；动态数据访问层需完全自研。

## 3. 总体架构

```text
┌─────────────────────────────────────────────────────────┐
│  前端（React）                                            │
│  ├─ 业务建模构建器（对象/字段/关系/状态/规则/权限）          │
│  ├─ 页面编排构建器（列表/表单/详情/布局覆盖）               │
│  └─ 运行时渲染器（按元数据+页面Schema渲染业务应用）          │
├─────────────────────────────────────────────────────────┤
│  API 网关层（认证、租户解析、限流）                         │
├─────────────────────────────────────────────────────────┤
│  设计态服务 (lowcode-designer)     运行态服务 (lowcode-runtime) │
│  ├─ 元模型管理 API                 ├─ 动态数据 API (/api/data/*) │
│  ├─ 页面 Schema 管理               ├─ 状态机与动作执行            │
│  ├─ 版本与发布                     ├─ 规则引擎（表达式求值）       │
│  └─ 应用导入导出                   ├─ 权限判定链                  │
│                                   └─ 工作流引擎（M3）            │
├─────────────────────────────────────────────────────────┤
│  元模型内核 (lowcode-metamodel)：19 对象定义、元数据加载与缓存、  │
│  动态 DDL（Schema Sync）、元数据校验                          │
├─────────────────────────────────────────────────────────┤
│  存储：MySQL（meta_* 元数据表 + 动态业务表） + Redis          │
└─────────────────────────────────────────────────────────┘
```

**关键架构原则**（来源：DM-LOWCODE-001 关键判断 + Frappe DocType 内聚性证据 E-FRAPPE-SRC-001/002）：

1. **元数据与运行时一致**：运行态的一切行为（API、校验、权限、状态流转）都由元数据驱动，不允许"页面配置绕过模型"（ADR-LOWCODE-UI-001 验证条件）。
2. **设计态与运行态分离部署**：私有化客户可只部署运行态；设计态变更通过"发布"进入运行态。
3. **业务对象内聚**：字段、权限、状态、规则都挂在 BusinessObject 下（借鉴 Frappe DocType 内聚 fields+permissions 的设计），而不是散落的全局配置。

## 4. 仓库结构（Maven 多模块）

```text
lowcode-platform/
├── lowcode-common/            # 通用工具、异常、租户上下文
├── lowcode-metamodel/         # 元模型内核：19 对象、元数据仓库、Schema Sync、缓存
├── lowcode-runtime/           # 运行态：动态数据 API、状态机、规则、权限判定
├── lowcode-designer/          # 设计态：元模型管理 API、页面 Schema、版本发布、导入导出
├── lowcode-expression/        # 表达式引擎封装（沙箱、函数库、白名单）
├── lowcode-plugin/            # 插件 SPI 边界，M0 仅空模块
├── lowcode-workflow/          # 工作流扩展边界，M0 仅空模块
├── lowcode-app/               # 启动装配（单体启动，模块化边界为后续拆微服务留余地）
└── lowcode-web/               # React 前端（builder + renderer 两个包）
    ├── packages/builder/
    ├── packages/renderer/
    └── packages/shared/
```

首版**单体部署**（一个 Spring Boot 进程 + 一个前端应用），模块边界按上述划分，避免过早微服务化。`lowcode-plugin` 与 `lowcode-workflow` 在 M0 只作为空边界模块存在，不实现插件系统或工作流引擎。

## 5. 多租户方案（首版基线已定，独立库扩展待 ADR）

### ADR-LOWCODE-TENANT-001（暂定，risk: high，单向门）

- **首版采用共享库 + tenant_id 行级隔离**：这是已拍板的 M0/M1 产品基线。所有 meta_* 表和动态业务表都带 `tenant_id` 列，运行时通过租户上下文自动注入过滤条件（MyBatis-Plus 多租户插件 + 动态层强制注入）。
- 私有化部署 = 单租户模式（tenant_id 固定），同一套代码。
- **后续扩展**：Frappe 的 site 级隔离（每租户独立库）隔离性最强但运维成本高，作为后续"VIP 租户独立库"扩展预留；动态层的数据源路由接口从第一天就抽象出来。
- **实现门禁**：共享库行级隔离可以进入 M0/M1；VIP 独立库、跨库迁移、单租户恢复和备份恢复能力实现前必须补 ADR-LOWCODE-TENANT-001 终审、压测和恢复演练。

## 6. 与竞品的差异化（来源：BIZ-KINGDEE-001 差异化机会 + OPP-LOWCODE-001）

1. 保留金蝶级的业务元模型与治理能力，但**降低建模入口复杂度**（默认约定优于配置：建对象即得默认列表/表单/详情/API）。
2. **逃逸通道透明**：插件、脚本、API 的边界从第一版就明确定义（对应 MT-LOWCODE-001 的"脚本逃逸率"指标——逃逸率高说明平台表达力不足）。
3. 中国企业场景优先：组织/权限模型对标金蝶的企业基础架构，而非欧美 workspace 模型。

## 8. ADR 清单与里程碑门禁

### ADR-LOWCODE-ID-001：内部雪花 ID + 外部 ULID 逻辑 ID（accepted）

**决策**：数据库内部主键统一使用雪花 ID（`bigint id`），跨环境、API、导入导出、页面 Schema、插件引用统一使用 ULID（`varchar(26) lid`）或稳定 `code`。

| 标识 | 类型 | 用途 | 是否可外部引用 |
|---|---|---|---|
| `id` | bigint（雪花 ID） | 数据库主键、内部索引、事务更新、性能敏感关联 | 否 |
| `lid` | varchar(26)（ULID） | 业务记录跨环境稳定引用、导入导出、API record id、插件/页面 schema 引用 | 是 |
| `code` | varchar(64) | 元模型对象、字段、动作、状态、插件等人定义稳定编码 | 是 |

**理由**：雪花 ID 适合 MySQL 主键和高性能索引；ULID 不依赖 workerId，适合跨环境迁移和外部引用。低代码平台同时有动态表、导入导出、插件和私有化迁移，因此必须双 ID。

**约束**：
- 任何导入导出包不得引用数据库 `id`。
- 动态业务表必须包含 `id bigint` 与 `lid varchar(26)`。
- link 字段默认存目标记录 `lid`，列名为 `{field_code}_lid`；如为了查询性能维护 `{field_code}_id`，只能作为可重建派生列，不作为跨环境契约。
- 元模型对象优先用 `code` 互相引用；运行时业务记录优先用 `lid` 互相引用。
