# 低代码平台 · 元模型设计

> 版本：v0.2（M0 开工版；M1/M2/M3 能力按里程碑 ADR 继续终审）
> 日期：2026-07-05
> 依据：ADR-LOWCODE-DM-001（最小业务元模型 19 对象，accepted for M0）；字段级设计参照 DM-FRAPPE-SRC-001/002（DocType/DocField）、DM-NOCOBASE-SRC-001/002（Collection/Field/Relation）、PM-FRAPPE-SRC-002（DocPerm）
> 本文档是 ADR-LOWCODE-DM-001 从"对象清单"到"字段级定义"的细化；M0 冻结元模型承载结构，M1/M2/M3 的权限、页面、工作流、插件和 Connector 行为仍按对应 ADR 终审。

## 1. 元模型总览（19 对象与层级关系）

### 1.1 标识策略（ADR-LOWCODE-ID-001）

本平台采用双 ID：

```text
id  = bigint，雪花 ID，数据库内部主键和高性能索引使用
lid = varchar(26)，ULID，运行时业务记录跨环境/API/导入导出引用使用
code = varchar(64)，元模型对象、字段、动作、状态等设计态稳定编码
```

规则：

1. 元模型之间优先用 `code` 建引用，例如 page schema 引用 field_code/action_code。
2. 业务记录之间优先用 `lid` 建引用，例如订单 link 客户时存 `customer_lid`。
3. 数据库 `id` 不进入导入导出包，不作为页面 schema、插件、外部 API 的长期引用。
4. 如为了查询性能维护 `{field_code}_id`，它只能作为可重建派生列，真实契约仍是 `{field_code}_lid`。

```text
Tenant（租户）
└─ Workspace（工作区）
   └─ App（应用）
      ├─ BusinessObject（业务对象）★内核
      │  ├─ Field（字段）
      │  ├─ Relation（关系）
      │  ├─ State（状态）+ StateTransition（流转）
      │  ├─ Action（动作）
      │  ├─ Rule（规则）
      │  ├─ Event / EventSubscription（业务事件与订阅，M1/M3）
      │  └─ ObjectPermission（对象权限，挂 Role）
      ├─ Workflow（工作流，M3）
      ├─ OptionSet（全局选项集，M2）
      ├─ ObjectExtension（对象扩展层，M2/M3）
      ├─ ConvertRule / Conversion / WriteBack / LinkTrace（单据下推、反写、引用追踪，M2/M3）
      ├─ FlexField（弹性域/辅助属性，M2/M3）
      ├─ Page（页面）
      │  └─ View（视图：列表/表单/详情/看板）
      │     └─ Component（组件配置）
      ├─ SavedView（保存的筛选视图，M2）
      ├─ Report / PrintTemplate / Menu（报表、打印、菜单，M2/M3）
      ├─ Comment / Follow / Message（协作、关注、站内消息，M2/M3）
      ├─ Role（应用内角色）
      ├─ IdentityProvider / ApiToken（身份集成与开放 API，M1/M3）
      ├─ DataSource（外部数据源）
      │  └─ Connector（连接器类型）
      ├─ Version（应用版本）
      └─ Plugin（插件挂载）
Permission（平台级权限，跨 App，见 02 文档）
```

设计依据：Frappe 证明业务对象应**内聚**字段与权限（DocType.fields=Table(DocField)、DocType.permissions=Table(DocPerm)，E-FRAPPE-SRC-001/002）；NocoBase 证明关系需显式建模 target/foreignKey/sourceKey/targetKey（DM-NOCOBASE-SRC-002）。

### 2.1.1 业务对象扩展与单据继承边界

M0/M1 不实现无限继承，也不允许通过继承自动合并动态表。首版采用更可控的两层模型：

```text
标准对象/行业模板对象
  + 受控扩展层（客户字段、客户规则、客户页面覆盖、插件扩展）
  -> 发布时合成为一个运行态 ObjectSnapshot
```

规则：

1. 标准对象和客户扩展必须保留 `source_kind`、`base_object_code`、扩展来源、扩展版本和安装包来源。
2. 客户扩展不得覆盖平台禁止覆盖字段、系统列、租户字段、权限不变量、发布状态机和审计字段。
3. 扩展字段、扩展规则、扩展页面必须进入发布流程、`lc_meta_ref`、Version.snapshot 和兼容检查。
4. 平台升级、行业模板升级或插件升级时，必须先生成扩展合并报告；冲突不得静默覆盖客户扩展。
5. 如后续引入父子单据类型继承，必须先补 ADR，明确字段、状态、权限、规则、页面、动态表、转换规则和升级合并策略。

### 2.2 Field（字段）——元模型中最重要的对象

字段类型集合（fieldtype）取 Frappe DocField.fieldtype 选项集与 NocoBase FieldType 的并集裁剪（依据 DM-FRAPPE-SRC-002：Frappe fieldtype 含 Data/Date/Datetime/Link/Table/Table MultiSelect/JSON/Code/Read Only 等）：

**首版字段类型（M0 22 种，M2 增 rollup 后为 23 种）**：

```text
文本类：text(单行) / textarea(多行) / richtext / code
数值类：integer / decimal(必须指定精度，禁float) / percent / currency(含币种)
日期类：date / datetime / time
选择类：select(单选) / multiselect(多选) / checkbox(布尔)
关系类：link(多对一引用) / table(子表,一对多) / multilink(多对多)
系统类：autonumber(编码规则) / user(人员) / org(组织) / attachment(附件) / formula(公式,只读) / rollup(汇总,M2)
```

### 2.2.1 字段类型 Handler SPI 与能力矩阵

字段类型行为不允许散落在转换器、DDL 映射、前端组件和导入导出代码中。M0 起使用 `FieldTypeHandler` 注册表作为唯一入口，详见 `ADR/ADR-LOWCODE-FIELDTYPE-SPI-001-field-type-handler-spi.md`。

```text
FieldTypeHandler:
  typeCode()
  ddlMapping(FieldDef) -> ColumnSpec
  converter() -> 输入合法形态/非法拒绝/NULL/空串语义
  validator(FieldDef) -> options 合法性
  queryCapability() -> 可过滤 op / 可排序 / 可聚合
  rendererKey() -> 前端组件注册键
  capabilities() -> unique/title_field/in_filter/M0可发布 等能力
```

能力矩阵首版口径：

| 类型族 | M0 可发布 | 可过滤/排序 | 可聚合 | 说明 |
|---|---|---|---|---|
| text/textarea/richtext/code | text 可过滤/排序；大文本默认不可排序 | text/select 类可按能力声明 | 否 | richtext/code 必须经过安全处理 |
| integer/decimal/percent/currency | 是 | 是 | 是 | decimal/currency 只用 BigDecimal 语义 |
| date/datetime/time | 是 | 是 | 否 | 查询边界按租户时区策略转换 |
| select/checkbox | 是 | 是 | 可 count | option code 只增不复用 |
| multiselect | M0 可保存 JSON；`in_filter=true` 阻断发布 | M0 不承诺高性能过滤 | 否 | 后续倒排/关联表能力补齐 |
| link/user/org | 是 | 是 | 否 | 存 `{field_code}_lid`，写入和展开必须同租户校验 |
| table | 是 | 列表默认不展开 | 否 | 子表对象独立动态表 |
| multilink | 草稿可保存，M0 发布阻断 | 否 | 否 | 后续 through 表能力补齐 |
| autonumber | 是 | 是 | 否 | 允许跳号，不以连续性作为审计依据 |
| attachment | M0 可保存 JSON | 否 | 否 | 下载/预览走权限与生命周期规则 |
| formula | 读时求值可用；`persisted=true` 阻断 | M0 不支持过滤/排序/聚合 | 否 | 循环依赖和字段权限由 Handler 声明并校验 |
| rollup | M2 前阻断发布，M2 作为 FieldTypeHandler 实现 | 只允许按被汇总字段能力过滤 | sum/count/avg/max/min | 子表或关联对象聚合，默认物化并声明重算策略 |

| 字段 | 类型 | 说明 | 竞品对照 |

### 2.3 Relation（关系）

依据 DM-NOCOBASE-SRC-002（RelationField 必须显式建模五要素）：

|---|---|

### 2.5 Action（动作）/ Rule（规则）

Action（页面按钮/API 动作的统一抽象）：

```text
action_type: create / update / delete / transition(状态流转) / custom(调用规则链) / script(M3)
scope: single(单条) / batch(批量) / object(无记录上下文)
confirm_text: 二次确认文案
visible_condition: 显示条件表达式
```

Rule（规则，事件驱动）：

```text
trigger: before_save / after_save / before_delete / after_transition / field_change / schedule / event
rule_type: validate(校验) / assign(赋值) / create_record / update_record / notify(通知,M2) / call_api(M3)
condition: 触发条件表达式
config: json（规则类型相关配置）
```

规则动作边界：

1. `create_record/update_record` 必须声明目标对象、字段映射、幂等键、执行时机和最大级联深度。
2. 跨对象写入必须走 `DataWriteService`，触发目标对象自身的权限、规则、审计和 outbox。
3. `schedule` 触发必须走统一调度器，声明租户时区、错过补跑策略和租户配额。
4. 规则仍不是自由脚本；表达力不足时先扩展规则动作或 ConvertRule。

**首版明确不做**：自由 JS 脚本注入（对应"脚本逃逸率"控制，MT-LOWCODE-001）。表达力不足的场景记录为需求，而不是开脚本口子。

## 3. 元数据存储设计

### 3.1 存储策略决策（ADR-LOWCODE-STORE-001）：少表 + JSON 聚合文档

**决策：元数据按"聚合根一行、子定义进 JSON 列"存储，元数据表控制在 9 张以内，不做子对象级关联表。**

理由（Java + MySQL 技术栈下的性能与复杂度权衡）：

1. **元数据是读多写少、整体加载的数据**：运行态只在 MetaGraph 冷加载/发布时读一次，之后全内存；不存在"按字段行级查询元数据"的热路径。规范化成 17 张表只会让加载变成多次 JOIN、让保存变成多表事务，收益为零。
2. **对象定义天然是文档**：一个 BusinessObject 的字段/状态/动作/规则是强内聚整体，几乎总是整体读、整体存（设计器保存即整对象提交）。竞品佐证：Frappe 的 DocType 本质就是一份 JSON 文档（doctype.json 内聚 fields+permissions，E-FRAPPE-SRC-001）；NocoBase 的 Collection options 也以 JSON 为主。
3. **少表 = 少迁移负担**：元模型演进（给 Field 加属性）只改 JSON 内容和代码，不动 DDL。

**元数据表（9 张）**：

| 表 | 结构化列（用于查询/唯一约束） | JSON 聚合列 |
|---|---|---|
| lc_meta_tenant | code、name、plan、status、isolation_mode | config |
| lc_meta_workspace | tenant_id、code、name | config |
| lc_meta_app | workspace_id、code、name、current_version、status | config |
| **lc_meta_object** | app_id、code、name、table_name、object_type、status | **fields**（字段数组）、**relations**、**states**（含 transitions）、**actions**、**rules**、options |
| lc_meta_page | app_id、code、name、page_type、bind_object | **views**（列表/表单/详情 Schema 数组） |
| lc_meta_role | app_id、code、name、role_type | **permissions**（对象权限/数据范围/字段权限/动作权限数组，即 02 文档 §3.2 结构） |
| lc_meta_datasource | app_id、code、type、status | config（密钥字段单独加密） |
| lc_meta_version | app_id、version_no、publish_status、published_at | **snapshot**（应用全量元数据快照） |
| lc_meta_plugin | app_id、code、status | config |

**配套规则**：

1. 全部带 `tenant_id` + 标准列（revision/deleted/deleted_at/delete_token/create_time/create_by/update_time/update_by）；结构化列只保留**需要 SQL 查询、唯一约束或列表展示**的属性，其余一律进 JSON。
2. **每个 JSON 聚合列内含 `_v` 结构版本号**，代码侧用带版本的 Java DTO 反序列化（Jackson），元模型演进时写升级器（JsonUpgrader）做惰性迁移——禁止无版本裸 JSON。
3. 并发控制：对象保存 = 整行更新 + `revision` 乐观锁（设计器单对象编辑场景足够，无需行级锁字段）。
4. **交叉引用查询用派生索引表 `lc_meta_ref`**（source_object_code, ref_type, target_code, detail）：保存对象时由代码同步重建，解决"哪些对象 link 了客户对象/哪些页面绑定了该对象"的影响分析查询——它是**可重建的缓存表**，不是关系模型的一部分，损坏可全量重算。
5. JSON 列上不建 MySQL 函数索引、不在业务 SQL 里用 `JSON_EXTRACT` 查询（需要查的属性要么升级为结构化列，要么走 lc_meta_ref / MetaGraph 内存查询）——防止 JSON 查询性能陷阱。

**代价与边界**：

- 失去子对象级 SQL 可查性 → 由 MetaGraph 内存图 + lc_meta_ref 补偿；
- 单行 JSON 有大小上限 → 单对象字段数上限 500（元数据校验器强制，远超实际需要）；
- 01 文档 §2 的字段级定义**依然全部有效**——它们是 JSON 内的 Java DTO 结构（FieldDef/StateDef/ActionDef/RuleDef 等），只是持久化载体从"表列"变为"JSON 属性"。

注意与业务数据的区分：**业务数据仍走动态真实表（§3.2），绝不用 JSON 存业务记录**——JSON 聚合只适用于"整体读写、无行级查询"的元数据。

### 3.2 业务数据存储：动态真实表（Schema Sync）

**决策：每个实体 BusinessObject 生成一张真实物理表**（`lc_{app}_{object}`），不用 EAV/宽表。依据：Frappe（tabXxx 表）与 NocoBase（collection→table + migration 能力，DM-NOCOBASE-SRC-001）均为真实表路线；EAV 的查询与索引代价在单据类场景不可接受。

Schema Sync 引擎规则：

1. 对象发布时对比元数据与 information_schema，生成并执行 DDL（建表/加列/改列长度）。
2. **只做加法**：不自动删列、不自动缩短长度、不自动改类型（危险变更生成工单提示人工处理）——既有数据安全优先。
3. 每张动态表自动附加系统列：`id, tenant_id, app_id, object_id, lid, state_code(启用状态机时), owner_user_lid, owner_dept_lid, owner_org_path, revision, deleted, deleted_at, delete_token, create_time, create_by, update_time, update_by`。
4. link 字段默认落库为 `{field_code}_lid` 列 + 普通索引，值为目标记录 ULID；不建物理外键（依赖运行时 on_delete 逻辑）。如需性能优化，可额外维护 `{field_code}_id` 派生列，但不得作为 API、导入导出或页面 Schema 的引用契约。

### 3.3 元数据缓存

- 应用元数据全量加载为内存对象图（MetaGraph），Redis 存版本号，本地缓存按版本号失效。
- 运行态只加载**已发布 Version 的快照**，设计态改动不影响运行中的应用。

## 4. 与 ADR-LOWCODE-DM-001 的验证对齐

该 ADR 的验证条件是："用一个真实业务对象验证：能完成字段建模、关联关系、列表/表单生成、状态动作、权限校验和版本升级"。本设计以**「客户-订单-审批」样例**作为 M0-M2 的贯穿验收用例（与知识库升级指导书中的实测样例一致，未来可与竞品实测行为直接对比）。
