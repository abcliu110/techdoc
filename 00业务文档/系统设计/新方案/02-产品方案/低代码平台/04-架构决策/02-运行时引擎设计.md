> 版本：v0.1（M1 候选设计；M1 权限、表达式、Outbox 实现前按 ADR 终审）

> 依据：PM-FRAPPE-SRC-001（权限判定链路）、PM-FRAPPE-SRC-002（DocPerm/UserPermission）、PM-NOCOBASE-SRC-001（ACL 判定链路）、SM-FRAPPE-SRC / SM-NOCOBASE-SRC 状态机卡、ADR-LOWCODE-PERM-001

## 1. 动态数据 API

### 1.1 API 表面（对象自动生成）

每个已发布的实体 BusinessObject 自动获得统一 API（全部 POST，沿用团队接口习惯）：

```text
POST /api/data/{app}/{object}/list      # 分页查询：filters + sorts + fields + page
POST /api/data/{app}/{object}/get       # 单条：id + fields（含 link 展开）
POST /api/data/{app}/{object}/add       # 新增
POST /api/data/{app}/{object}/update    # 更新（带 revision 乐观锁）
POST /api/data/{app}/{object}/del       # 逻辑删除
POST /api/data/{app}/{object}/action    # 执行动作：action_code + ids + params
POST /api/data/{app}/{object}/meta      # 返回该对象对当前用户生效的元数据（字段权限裁剪后）
POST /api/data/{app}/{object}/suggest   # link/user/org 选项查询，复用 AccessView 与 data_scope
```

设计依据：NocoBase/Directus/Frappe 均为"模型即 API"（BIZ-LOWCODE-001：数据模型驱动平台的核心能力）。`/meta` 接口是前端渲染器的唯一元数据来源，保证"权限裁剪在服务端完成"。

### 1.2 ExecutionContext（请求级版本钉住）

运行时入口在认证与租户解析之后构造不可变 `ExecutionContext`。后续 service 不得重新读取 latest MetaGraph，也不得裸读 ThreadLocal。

```text
ExecutionContext:
  tenantId / userId / roles
  metaGraph: 本请求钉住的已发布 MetaGraph 引用（含 metaHash）
  accessViews: 按对象惰性缓存的 AccessView map
  traceId
  idempotencyKey
  requestTime（Clock 注入）
```

规则：

- 初始化时机：认证 Filter 之后、业务 Handler 之前。
- 写接口必须校验前端提交的 metaHash/metaVersion；过旧则返回刷新错误，不继续提交旧 schema。
- 异步任务、outbox 消费、调度任务必须落库 ctx 摘要（tenantId、userId、metaHash、traceId），消费侧重建 context。
- 动态 SQL 组装处保留 fail-fast：tenant_id 缺失直接抛异常。

### 1.3 请求处理管线（每个写请求固定流水线）

```text
1. 构造 ExecutionContext  → tenant/user/metaGraph/metaHash/traceId/idempotencyKey
2. 对象定位               → context.metaGraph.getObject(app, object)
3. AccessView 判定         → 见 §3，不通过记录 PERMISSION_DENY 事件
4. 数据反序列化与类型转换  → 按 Field.field_type 严格转换，未知字段拒绝
5. before_save 规则       → 校验规则(validate) → 赋值规则(assign) → fetch_from 带出
6. 状态机检查             → 当前状态 doc_editable？动作是否合法流转？
7. 持久化                 → 动态 SQL（参数化，列名白名单来自元数据，防注入）
8. 写审计与 outbox         → 审计主记录同事务，notify/call_api/webhook 只写 outbox
9. 异步投递               → 统一调度器投递 outbox，重试/死信/幂等统一处理
```

**安全红线**：动态 SQL 的表名/列名只能来自元数据注册表（白名单机制），用户输入永远只作为参数化值；filters 中的字段名必须存在于元数据且通过字段级读权限校验。

### 1.4 查询引擎

filters 结构（前端与规则共用）：

```json
{ "and": [
    {"field": "state_code", "op": "in", "value": ["draft","submitted"]},
    {"or": [
      {"field": "amount", "op": "gte", "value": 1000},
      {"field": "customer_id.level", "op": "eq", "value": "VIP"}
    ]}
]}
```

- op 白名单：eq/ne/gt/gte/lt/lte/in/nin/like/is_null/not_null/between
- `link字段.目标字段` 支持一层跨对象过滤（生成 JOIN），首版最多一层，防查询爆炸
- list 默认注入：`tenant_id = ?` + `deleted = 0` + 数据范围权限条件（§3.4），**任何代码路径不得绕过**

### 1.5 统一写入口与 outbox

所有写入动态业务表的路径必须经过 `DataWriteService`：

```text
API add/update/del
批量导入
级联 cascade/set_null
规则 assign
工作流写入
系统修复任务
```

动态表的 INSERT/UPDATE/DELETE 只允许出现在该服务包内。导入或级联如果需要豁免某类规则，必须通过显式配置并写审计事件，禁止调用方绕过管线。

外部副作用统一进入本地 outbox 表：notify、call_api、webhook、缓存广播、通知通道投递。outbox 与业务写同事务落库，由统一调度器执行重试、退避、死信和人工重放。

## 2. 状态机与动作执行

参照 Frappe Workflow（State/Transition/allowed role/condition，SM-FRAPPE-SRC 卡）：

1. 启用状态机的对象，新增时自动置 initial 状态；`state_code` 由平台管理，**不允许通过 update 接口直接改**，只能走 transition 动作。
2. transition 执行序：动作存在性 → 当前状态匹配 from_state → 角色检查（allowed_roles ∩ 用户角色）→ condition 表达式求值 → 写 state_code → post_rules → 审计。
3. 状态与编辑联动：`doc_editable=false` 的状态下，update/del 被拒绝（系统级规则，优先于对象权限）。
4. 全部流转记录入 `lc_rt_transition_log`（含操作人、前后状态、时间、参数）——对应竞品缺口"执行日志"（知识库 DoD 缺口清单）。

## 3. AccessView 权限判定内核（ADR-LOWCODE-PERM-001 的实现设计）

### 3.1 权限层级（判定顺序自上而下，先粗后细，任一层拒绝即终止）

```text
L1 租户/工作区成员资格     → 不是成员直接 403
L2 应用访问权限            → design（可进设计器）/ runtime（可运行应用）
L3 对象权限                → 按 Role × BusinessObject × 操作(read/create/update/delete/action) 的授权矩阵
L4 数据范围权限            → 本人 / 本部门 / 本部门及以下 / 全部 / 自定义条件表达式 → 转为 filters 注入查询
L5 字段权限                → perm_level>0 的字段需 Role 显式授权；read 裁剪返回值，write 拒绝提交
L6 动作权限                → Action 级授权 + StateTransition.allowed_roles
```

设计依据：Frappe 判定链 = Role Permission(DocPerm) → User Permission(数据范围) → permlevel(字段级)（PM-FRAPPE-SRC-001/002）；NocoBase ACL = role → resource → action + scope（PM-NOCOBASE-SRC-001）。本设计合并两者：**L3/L5 学 Frappe 的 permlevel 分层，L4 学 NocoBase 的 scope 条件化**。

### 3.2 权限元数据

存储于 `lc_meta_role.permissions` JSON 聚合列（存储策略见 01 文档 §3.1），数组元素结构：

```text
PermissionDef:
  object_code
  can_read / can_create / can_update / can_delete  (boolean)
  data_scope: 枚举 self/dept/dept_tree/all/custom
  scope_condition: 表达式（custom 时，可用变量 $user.id/$user.dept/$user.roles）
  field_perms: {"perm_level_1": "read", "perm_level_2": "none"}
  action_codes: 允许的动作列表
```

运行时判定不查库：权限随 MetaGraph 整体加载进内存，按 (role, object) 建哈希索引。

### 3.3 AccessView 模型

请求内对每个对象只允许通过 `AccessDecider` 计算一次不可变 AccessView：

```text
AccessView:
  objectPerms: read/create/update/delete/actions[]
  fieldView: fieldCode -> NONE | READ | WRITE
  scopeAst: data_scope 折算出的 filters AST
  metaHash: 本次判定依据的元数据摘要
  permVersion: 用户/角色/组织树/字段权限/数据范围版本组合
```

消费规则：

- `/meta`、list、get、add、update、del、action、suggest、link 展开、导出、导入、级联写只消费 AccessView。
- data_scope 只能在 AST 层合并：`RootAnd(scopeAst, userFilters)`。
- link 跨对象展开必须对目标对象再取 AccessView；无 read 则展开字段整体不返回。
- 权限拒绝写 PlatformEvent，记录 denialReason、metaHash、permVersion、traceId。

### 3.4 实现要点

- 判定结果按 `(user, app, object, metaHash, permVersion)` 缓存，角色/权限/组织树变更必须更新 permVersion。
- 设计器权限与运行权限分离（role_type=design/runtime）：设计者可改页面但**读不到 DataSource 密钥明文**（ADR-LOWCODE-PERM-001 验证条件），密钥只在运行态服务内解密。
- 平台管理员/租户管理员的越权入口显式建模（super 角色），审计全量记录。
- 管理端提供 `explain(user, object, field?)` 权限解释接口，返回六层链逐层判定过程，供排障使用。

## 4. 规则引擎与表达式

### 4.1 表达式引擎（lowcode-expression 模块，暂定 Aviator）

- 沙箱约束：禁反射、禁类加载、禁 IO、执行超时 100ms、白名单函数库。
- 内置函数分类：字符串/数值/日期/逻辑 + 上下文变量 `$record`（当前记录）、`$user`、`$old`（变更前）、`$env`（租户/应用）。
- 统一用于：Field.default_value(`=`前缀)、depends_on、validate_rule、Rule.condition、StateTransition.condition、data_scope 的 custom 条件。
- **一处语法，六处复用**——这是表达式引擎成为单向门决策的原因（换引擎=全部配置失效），因此 ADR-LOWCODE-EXPR-001 终审前，表达式必须通过 `lowcode-expression` 的门面接口调用，禁止业务代码直接 import Aviator。

### 4.2 规则执行

- before_save 内规则按 sort_no 串行，validate 失败即终止并返回字段级错误。
- assign 规则修改 `$record` 后重新触发关联 fetch_from（最多一轮，防循环）。
- 规则执行日志入 `lc_rt_rule_log`（触发器、耗时、结果），支撑后续"脚本逃逸率/规则复杂度"指标（MT-LOWCODE-001）。

## 4.3 业务事件中心与订阅模型

业务事件是 Rule、Workflow、Webhook、Plugin 和通知的统一底座，不再让各能力各自定义触发器。

内置事件：

```text
record.created
record.updated
record.deleted
state.transitioned
action.executed
version.published
import.completed
export.completed
schedule.fired
```

事件 payload 固定包含：

```text
tenantId / appId / objectCode / recordLid
eventCode / eventTime / metaHash / traceId
actor / sourceAction / diffSummary
payloadVersion / dataClassifications
```

订阅模型：

| 订阅者 | M1/M2/M3 边界 | 执行方式 |
|---|---|---|
| rule | M1 内置事件 + 规则消费者化 | before 同事务，after 可 outbox |
| workflow | M3 | after 事件触发流程实例 |
| webhook | M3 | outbox 投递，受出网治理 |
| plugin handler | M3 | 插件生命周期与隔离通过后启用 |
| notification | M2 站内消息，M3 外部通道 | outbox 投递 |

同步订阅只能用于 before 类校验和补充字段，必须有超时和循环检测；异步订阅必须进入 outbox，不得在业务事务内直接调用外部系统。

## 4.4 跨对象规则动作与定时自动化

`create_record/update_record` 是受控规则动作，不是脚本逃逸口。

执行规则：

1. 目标对象、字段映射和默认值必须来自元数据，字段映射表达式只能使用 lowcode-expression 门面。
2. 目标对象写入必须走 `DataWriteService`，触发目标对象的 AccessView、类型转换、规则、状态机、审计和 outbox。
3. 级联深度默认最大 3 层，单个事件最多创建/更新 1000 条目标记录；超限进入阻断或异步任务。
4. 幂等键默认由 `sourceEventId + targetObject + ruleCode` 组成。

定时自动化：

1. schedule trigger 声明 cron/周期、租户时区、目标对象 filters、动作和错过补跑策略。
2. 执行由统一调度器触发，调度任务必须携带 ExecutionContext 摘要。
3. 单租户 schedule 任务频率、并发和执行时长纳入配额。

## 4.5 rollup 汇总字段

rollup 是 FieldTypeHandler 的字段类型实现，表达“子表或关联对象聚合到当前对象字段”的业务语义。

M2 首版语义：

| 项 | 决策 |
|---|---|
| 聚合函数 | count/sum/avg/max/min |
| 源 | table 子表或一层 link/multilink 关联对象 |
| 存储 | 默认物化到当前对象动态表列 |
| 重算时机 | 子表/关联记录经 DataWriteService 写入后同事务或 outbox 可靠重算，具体由 ADR 终审 |
| 权限 | rollup 结果可见性不得高于源字段权限；无源字段 read 权限时结果不可见或按脱敏聚合策略显示 |
| 风险 | 必须有重算风暴限流、循环依赖检测和失败补偿 |

M2 前 rollup DTO 可保存但发布阻断；M2 实现前必须补 rollup ADR，明确同步物化还是异步物化。

## 4.6 ConvertRule 单据下推

ConvertRule 是单据下推的一等模型，不等同于普通 `create_record`。

M3 首版只承诺 1:1 单级下推：

```text
source object + source record
  -> ConvertRule(field_mappings, line_mappings, source_ref_policy)
  -> target object + target record + LinkTrace
```

规则：

1. 下推动作挂在 Action 上，执行前必须校验源单 read/action 权限和目标单 create 权限。
2. 目标记录必须记录 source_lid 和 conversion_code，形成 LinkTrace。
3. 下推必须走 DataWriteService，复用类型转换、规则、状态、审计和 outbox。
4. 1:N、合并下推、拆分下推、跨应用下推进入 M4。

## 5. 平台事件、审计与数据版本

- 所有写操作记录 `lc_rt_audit_log`：tenant/app/object/record_id/操作类型/操作人/字段级 diff(json)/trace_id。
- `enable_version=true` 的对象：每次 update 前将旧记录快照入 `lc_rt_record_version`，支持查看历史与（管理员）恢复。

统一平台事件模型：

```text
PlatformEvent:
  eventType:
    DATA_WRITE / PERMISSION_DENY / PUBLISH_STEP / RECONCILE_DIFF /
    BREAK_GLASS / EXPORT / NOTIFY_DELIVERY / DDL_EXEC
  tenantId / appId / objectCode / recordLid
  actor / onBehalfOf
  traceId / metaHash / permVersion
  payload（字段 diff、拒绝原因、步骤结果；敏感字段脱敏）
```

规则：

- metaHash 与 permVersion 必须进入 DATA_WRITE、PERMISSION_DENY、EXPORT、NOTIFY_DELIVERY 事件。
- 发布步骤、DDL 执行和 Reconciler 差异必须能通过 traceId 串联到一次发布任务。
- break-glass 审计不可被同一权限主体修改或删除。

## 5.1 站内消息、全局搜索与开放访问

站内消息：

- notify 的默认通道是站内消息中心，外部邮件/企微/钉钉属于 M3 通道扩展。
- 消息只保存脱敏摘要、来源事件、目标引用和跳转信息，不保存敏感字段明文。
- @ 提及、关注记录和规则通知都必须通过 EventSubscription 触发。

全局搜索：

- M3 首版按 `title_field`、autonumber 和 `search_fields` 做跨对象 LIKE 检索。
- 搜索候选对象和字段必须来自 MetaGraph，并逐对象应用 AccessView。
- 全文索引、相关性排序、拼音分词和跨应用搜索进入 M4。

开放 API Token：

- API token 鉴权后必须构造 ExecutionContext，并复用 AccessView。
- token scope 至少包含 app、object、operation、field mask、expiresAt、rateLimit。
- token 使用、失败、撤销和轮换必须写审计，不得复用浏览器 cookie 会话。

IdentityProvider SPI：

- M1 只预留 SPI 和本地用户模型映射接口。
- M3 支持钉钉/企微/OIDC 等至少一个落地实现前，必须补组织同步冲突策略和历史组织快照测试。

## 6. 性能与非功能（首版目标，来源 NFR-LOWCODE-001）

| 项 | 目标 |
|---|---|
| 元数据加载 | 应用级 MetaGraph 冷加载 < 2s，热路径全内存 |
| list API | 万级数据分页 P95 < 500ms（依赖动态索引：in_filter 字段自动建索引） |
| 权限判定 | 缓存命中时 < 1ms |
| 并发写 | revision 乐观锁，冲突返回明确错误码 |
| 隔离 | 任何 SQL 必含 tenant_id 条件（动态层单元测试强制断言） |
