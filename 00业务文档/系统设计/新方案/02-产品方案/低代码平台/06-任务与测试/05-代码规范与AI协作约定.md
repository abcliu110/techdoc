## 2. Java 代码规范

### 2.1 包结构与分层

根包：`com.lowcode`（已冻结；首行代码后不得改名）。

```text
com.lowcode.{module}            # module = common/metamodel/runtime/designer/expression/plugin/workflow/app
├── api/          # 对外 REST Controller + DTO/VO（仅 designer/runtime 有）
├── service/      # 业务服务（接口 + impl）
├── domain/       # 领域模型（元模型对象、值对象、枚举）
├── dao/          # Entity + Mapper
├── config/       # Spring 配置
└── support/      # 模块内工具（禁止跨模块引用 support）
```

分层依赖规则（ArchUnit 强制）：

- `api → service → dao`，禁止逆向、禁止 api 直连 dao
- `runtime`/`designer` 可依赖 `metamodel`/`expression`/`common`；`metamodel` 只可依赖 `common`；**任何模块禁止依赖 `app`**
- 业务代码禁止直接 import `com.googlecode.aviator.*`（必须走 lowcode-expression 门面）
- Controller 禁止出现 SQL/Mapper 调用；Entity 禁止出现在 api 层出入参

### 2.2 命名与形态

- 类名后缀：`Controller / Service / ServiceImpl / Mapper / Entity / DTO / VO / Enum / Config / Handler`
- 接口方法命名：`add / update / get / list / del`（禁 create/getDetail/delete）；接口全 POST
- 2 空格缩进；注解一行一个纵向排列；Lombok 允许 `@Data/@Builder/@Slf4j`，禁 `@SneakyThrows`
- Java 源码 UTF-8 无 BOM（Spotless 强制）
- 枚举端到端：固定选项集合字段（状态/开关/模式/类型）必须建枚举，DTO/VO/Entity/Service 用同一枚举类型，API 与 DB 传 code 值，逻辑比较用枚举常量禁魔法值（1/2/3）
- 金额/精度字段用 `BigDecimal`，禁 float/double（Checkstyle 自定义规则）

### 2.3 异常与错误码体系

```text
统一响应：Result<T> { code: String, message: String, data: T, traceId: String }
错误码格式：LC-{模块}-{四位数}，如 LC-META-1001
  LC-COMM-xxxx 通用（参数/认证/权限/限流）
  LC-META-xxxx 元模型（校验失败/编码冲突/悬空引用）
  LC-DATA-xxxx 运行时数据（乐观锁冲突/状态非法/规则校验失败）
  LC-PERM-xxxx 权限（分层判定链每层独立错误码，便于排查）
  LC-EXPR-xxxx 表达式（语法/超时/沙箱拦截）
```

- 业务异常统一 `BizException(errorCode, args)`，全局 `@RestControllerAdvice` 兜底；**禁止 catch 后吞异常或仅 log 继续**
- 校验类错误必须携带字段定位：`{"field": "amount", "code": "LC-DATA-2001", "message": "金额超出范围"}`
- 错误码注册为枚举 `ErrorCode`，新增错误码 = 加枚举项（禁字符串散落）

### 2.4 日志规范

- `@Slf4j` + 占位符（禁字符串拼接）；每请求注入 `traceId`（MDC），审计/规则/流转日志均带 traceId 串联
- 级别：ERROR=需要人处理；WARN=自动降级/重试成功；INFO=关键业务动作（发布/流转/权限变更）；DEBUG=开发排查
- **禁输出**：DataSource 密钥、用户密码/token、完整业务数据行（只 log id 与字段名）
- 动态 SQL 执行日志：DEBUG 级输出 SQL 模板 + 参数摘要，便于排查但不泄数据

## 3. 注释规范（重点）

原则：注释解释"**为什么**"，不复述代码"做了什么"；宁可少而准，不要多而废。

### 3.1 必须写注释的场景（Checkstyle/评审强制）

| 场景 | 要求 |
|---|---|
| 对外 REST 接口方法 | Javadoc：业务目的、入参含义、返回结构、错误码清单 |
| Service 公开方法 | Javadoc：业务语义、事务边界、幂等性说明 |
| DTO/VO 类 | 中文类注释：使用场景、请求/响应方向 |
| DTO/VO 字段 | 中文字段注释：含义、枚举 code 含义、单位、格式、是否必填、默认值 |
| 枚举类与枚举项 | 每个枚举项注释业务含义；调用处传枚举参数时说明选择原因 |
| 元数据白名单/安全过滤代码 | 说明防的是什么攻击、白名单来源 |
| 动态 DDL/Schema Sync 的每类变更分支 | 说明为什么允许/拒绝该类变更（"只做加法"依据） |
| 权限判定链每一层 | 说明该层判什么、对应竞品参照（如"permlevel 机制，参照 Frappe DocPerm"） |
| 表达式沙箱配置 | 每个禁用项注释风险（反射→任意代码执行等） |
| 复杂算法/递归（元数据图遍历、fetch_from 传播） | 说明终止条件与防循环机制 |

### 3.2 特殊注释标记

```java
// KB-REF: DM-FRAPPE-SRC-002 —— 设计依据指向知识库卡片（涉及竞品借鉴的实现必须标）
// ADR-PENDING: ADR-LOWCODE-EXPR-001 —— 该实现受暂定 ADR 影响，终审后可能重构
// SECURITY: —— 安全关键代码，修改必须人工评审
// TODO(T-xxx): —— 必须挂任务卡编号，禁止无主 TODO（CI 扫描裸 TODO 报警）
```

### 3.3 不要写的注释

- 复述代码：`// 遍历字段列表` ❌
- 注释掉的死代码（直接删，git 有历史）❌
- 自动生成的空 Javadoc 模板（`@param x x`）❌

## 4. 前端规范（lowcode-web）

- TS `strict: true`；禁 `any`（确需时 `// eslint-disable` 单行并注明原因）
- 组件：函数组件 + hooks；文件名 PascalCase；一个组件一个文件；props 定义 interface 并写字段注释
- 状态：服务端状态用 TanStack Query，客户端状态用 zustand；禁 prop-drilling 超 2 层（提 context/store）
- API 层：统一 `request.post(path, body)` 封装（错误码→AntD message 映射、traceId 透传）；接口类型从后端 OpenAPI 生成，禁手写重复类型
- 字段组件注册表：新增 field_type 组件必须实现统一接口 `FieldComponent { view, edit, validate }` 并注册，禁 switch-case 散落
- ESLint + Prettier + stylelint 配置入库，CI 强制 `lint --max-warnings=0`

## 5. 数据库与 Flyway 规范

- 迁移文件：`V{yyyyMMddHHmm}__{描述}.sql`，只增不改（已合并的迁移文件禁止修改）
- 表名：元数据 `lc_meta_*`、运行时系统表 `lc_rt_*`、动态业务表 `lc_{app}_{object}`（引擎生成，禁手工建）
- 每表必带标准列（见 01 文档）+ `uk_` 前缀唯一索引 / `idx_` 普通索引命名
- 布尔列 `is_` 前缀 TINYINT 注释 `0-否，1-是`；JSON 列注明结构示例；所有列必须有 COMMENT（CI 扫描无注释列）

## 6. 测试规范

| 类型 | 要求 | 门禁 |
|---|---|---|
| 单元测试 | service/domain 层核心逻辑；命名 `方法_场景_期望`（如 `addField_duplicateCode_throwsBizException`） | JaCoCo：metamodel/runtime/expression 模块行覆盖 ≥ 70%，安全关键类（白名单/沙箱/权限链）≥ 90% |
| 集成测试 | Testcontainers 起真 MySQL/Redis，**禁 H2 模拟**（方言差异会掩盖动态 DDL bug） | CI 必跑 |
| 架构测试 | ArchUnit：§2.1 全部依赖规则 + "Controller 禁 Mapper" + "禁直接 import Aviator" | CI 必跑 |
| 安全测试集 | SQL 注入用例集（字段名/op/值三个注入面）、表达式沙箱逃逸用例集、tenant_id 缺失断言 | 固定测试类，任何 PR 不得删改用例（改需人工评审） |
| 前端 | vitest 组件测试：字段组件 22 种类型渲染/校验；表达式前端求值器与后端语义对照用例 | CI 必跑 |

测试数据：统一 `TestDataBuilder`（建租户/应用/对象/字段的流式构造器），禁止测试里手写大段 insert。

## 7. Git 与任务执行流程

```text
分支：main（保护）← feature/T-001-project-skeleton（一任务卡一分支）
提交：T-001: 简述改动目的（中文，首行 ≤ 50 字）
PR：一任务卡一 PR；PR 描述使用交付说明模板（见 §7.1）；CI 全绿 + 人工评审后合并
禁止：跨任务卡混提；无关格式化混入；force-push main
```

### 7.1 交付说明模板（每个任务卡 PR 必填）

```text
任务卡：T-xxx
变更目标：
实现要点与设计文档对应章节：
新增依赖及理由：
验收标准自检：（逐条 ✅/❌，❌ 必须说明）
测试证据：（测试类清单 + 覆盖率截图/数字）
未覆盖风险：
需要人类确认的未决项：
```

### 7.2 多 AI 并行协作规则

1. **接口契约先行**：有依赖关系的任务（如 T-102 依赖 T-005 的 MetaGraph），必须按三步冻结：①接口 PR（interface + Javadoc + DTO + 空实现 + 契约测试桩）先合入；②下游任务在 24 小时内补 stub 测试或明确无影响；③契约冻结标记写入任务卡和 PR 描述后再实现。冻结后接口变更必须开兼容 PR，列出影响任务、下游负责人和迁移窗口。
2. **模块所有权**：一个任务卡只改自己模块的文件；需要改共享模块（common/metamodel 接口）时，在 PR 中单列并说明影响面。
3. **冲突处理**：发现他人代码有问题 → 报告，不顺手改；自己被 rebase 冲突 → 以 main 为准手工解决，禁 `--theirs/--ours` 一把梭。
4. **ADR 入仓**：`../04-架构决策/ADR/` 目录存放已冻结、已进入实现或会影响代码替换成本的 ADR（从知识库同步 + 新增技术 ADR）；代码中 `ADR-PENDING` 标记必须能在架构设计、详细设计总纲或 `../04-架构决策/ADR/` 找到对应决策编号和终审门禁，决策不留在聊天记录里。
