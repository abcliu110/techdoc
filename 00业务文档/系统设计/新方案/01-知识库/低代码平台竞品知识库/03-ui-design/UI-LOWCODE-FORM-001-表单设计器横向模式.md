# UI 设计：低代码平台表单设计器横向模式

## 本质模型

表单设计器不是字段拖拽器，而是把对象字段、填写路径、布局容器、条件规则、提交动作、权限边界和发布形态统一建模的可视化工作台。

## 横向竞品模式

| 模式 | 代表竞品 | 观察结论 | 适用边界 |
|---|---|---|---|
| 对象字段驱动表单 | Frappe、NocoBase、Airtable | 表单字段来自 DocType、Collection 或 Table 字段，天然贴近业务数据模型 | 适合企业业务录入；若缺少独立布局层，会受底层字段结构限制 |
| 规则驱动动态表单 | Jotform、SurveyJS | 条件规则控制字段、页面、选项、只读、必填、计算、通知等行为 | 适合复杂采集；规则冲突、顺序和可维护性是主要风险 |
| 对话式路径表单 | Typeform | 以问题路径和分支为中心，强调填写者参与度和完成率 | 适合问卷、线索、调研；不天然适合密集后台录入 |
| 表单到流程 | Formstack | 表单提交后进入审批、评论、退回和状态追踪 | 适合企业请求、审批、合同、HR 等流程入口 |
| Schema/JSON 双通道 | Form.io、SurveyJS | 可视化配置和 JSON/schema 编辑并存，支持嵌入、复制、版本化和高级扩展 | 适合开发者平台；对纯业务用户门槛更高 |

## 推荐信息架构

```text
顶部：保存、预览、发布、版本、设备/展示模式切换
左侧：字段/组件库、对象字段、布局容器、模板
中间：表单画布、分页/步骤、弹窗/内嵌/门户预览
右侧：字段属性、校验、默认值、权限提示、条件规则
底部或独立页：全局规则、提交动作、流程、集成、审计
```

## 关键取舍

表单设计器有两条路线：

```text
业务用户优先：对象字段 -> 默认表单 -> 少量布局和规则覆盖
开发者优先：组件库 -> schema/JSON -> 表达式/脚本 -> 高度自定义
```

企业业务低代码不应直接选择纯自由画布路线。更稳妥的首版路径是“对象字段驱动默认表单 + 受控布局覆盖 + 可视化规则 + 发布预览”，把 JSON/schema 编辑作为高级能力。

## 设计原则

1. 字段、布局、规则、权限、提交动作必须分层建模。
2. 条件显隐不能等同权限；敏感字段必须由权限模型约束。
3. 规则既要能在字段属性里配置，也要能在全局规则视图集中审计。
4. 表单必须支持预览和测试，尤其是条件逻辑、分支路径和审批流。
5. 内部管理表单、外部门户表单、对话式表单、审批表单可以共享对象字段，但运行时边界不同。

## 不宜照搬

- 不照搬 Typeform 的全对话式体验作为后台主录入形态；它适合单问题路径，不适合密集业务字段录入。
- 不照搬 Airtable 的条件隐藏作为安全控制；隐藏只是视觉状态。
- 不把 Form.io 的 JSON 编辑暴露为普通用户主路径；它应作为高级诊断、迁移和扩展入口。

## 待验证

- NocoBase/Frappe 本地样例中，表单字段权限、条件显隐和工作流状态变化之间的运行时优先级。
- 复杂规则冲突时，是否需要规则图、规则排序、循环依赖检测和影响分析。
- 门户表单是否需要匿名访问、草稿保存、防重复提交、反垃圾和提交频率限制。
