## 2. 总体结论

低代码平台可以分成三类：

1. 业务元模型平台：金蝶 AI 苍穹、Frappe。
2. 内部工具构建器：Appsmith、ToolJet、Budibase、Lowcoder。
3. 数据模型驱动平台：NocoBase、NocoDB、Directus。

如果目标是建设自己的企业业务低代码平台，应采用“业务元模型优先”的路线：

```text
BusinessObject
→ Field / Relation
→ State / Action
→ Rule / Workflow
→ Permission / Data Scope
→ Page / View / Component
→ API / Integration
→ Version / Audit
```

不建议直接从自由画布、组件拖拽和外部数据源绑定开始。

## 3. 金蝶路线的启发

金蝶低代码平台的关键启发是：

```text
用元数据描述业务对象、布局、规则、流程和权限。
```

它的低代码能力不是孤立的页面搭建器，而是企业级业务基础架构的一部分，和组织、人员、客商、权限、流程、数据、AI、RPA 等能力结合。

这意味着自研平台不能只做：

```text
页面组件 + 数据源 + 查询绑定
```

而要做：

```text
业务对象 + 状态动作 + 规则流程 + 组织权限 + 页面生成 + 集成扩展
```

## 4. GitHub 开源项目启发

| 项目 | 可借鉴点 | 不宜照搬点 |
|---|---|---|
| Appsmith | Widget、DataSource、Query、JS Binding、内部工具体验 | 缺少业务对象生命周期和 ERP 语义 |
| ToolJet | 组件、数据源、工作流、RBAC、AI/Agent 方向 | 仍偏内部工具，业务建模不深 |
| Budibase | Data Source、Screen、Automation 的组合 | 业务规则和对象生命周期较弱 |
| Lowcoder | 组件体系、自由布局、查询绑定 | 容易停留在 UI Builder |
| NocoBase | Collection、Association、Workflow、Plugin、模型驱动 UI | 需要补强企业级单据语义 |
| NocoDB | 表格体验、多视图、API、层级权限 | 更像数据表协作，不是复杂业务平台 |
| Directus | 数据模型、权限、策略、自动 API | 不是完整低代码业务流程平台 |
| Frappe | DocType、表单、列表、权限、工作流、ERPNext 基座 | UI 构建体验不如现代 builder |

## 5. 推荐产品方向

### 5.1 首版定位

首版不要做“万能低代码”，而是做：

```text
面向企业业务系统的模型驱动低代码平台
```

核心场景：

- 业务对象建模。
- 列表、表单、详情自动生成。
- 状态动作配置。
- 规则和流程配置。
- 角色和数据权限配置。
- 数据源/API 集成。
- 模型版本发布和回滚。

### 5.2 首版最小能力

```text
Tenant / Workspace
App
BusinessObject
Field
Relation
State
Action
Rule
Workflow
Page
View
Component
Role
Permission
DataSource
Connector
Version
Plugin
```

### 5.3 不做事项

首版不建议优先做：

- 大而全自由画布。
- 复杂 BI 报表。
- 完整 RPA。
- 复杂 AI Agent 编排。
- 面向所有行业的通用模板市场。

原因：这些能力会稀释业务元模型内核，导致平台变成组件和插件集合。

## 6. 关键 ADR

| ADR | 决策 |
|---|---|
| ADR-LOWCODE-001 | 采用业务元模型优先路线 |
| ADR-LOWCODE-DM-001 | 首版最小元模型 |
| ADR-LOWCODE-UI-001 | 采用业务建模 + 页面编排的双层构建器 |
| ADR-LOWCODE-PERM-001 | 采用多层权限模型 |

## 7. 关键风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 过度 UI Builder 化 | 只学 Appsmith/ToolJet，会忽略业务对象和规则 | 业务元模型先行 |
| 元模型过度设计 | 首版设计太大，无法落地 | 用一个真实业务对象验证闭环 |
| 权限不足 | 低代码暴露数据源和流程，权限不完整风险高 | 多层权限和审计前置 |
| 脚本逃逸 | 平台能力不足导致大量 JS/SQL 脚本 | 监控脚本逃逸率 |
| 版本不可控 | 运行中应用配置被直接覆盖 | 模型/页面/规则全部版本化 |

## 8. 下一步建议

1. 选择一个真实业务对象作为试点，例如“门店商品”“订单”“会员等级”。
2. 用 `DM-LOWCODE-001` 的最小元模型建模。
3. 自动生成列表、表单、详情、状态动作和权限。
4. 用 `BR-LOWCODE-001` 的治理规则做发布校验。
5. 用 `MT-LOWCODE-001` 的指标验证平台是否降低开发成本。

## 9. 证据边界

本知识库基于公开资料和已有本地对比文档形成：

- 金蝶没有公开源码，内部实现只做推断。
- GitHub 项目未做源码级调用链分析，本版侧重产品和平台设计拆解。
- 能力矩阵是基于官方公开资料的经验评分，后续应通过实际安装试用和源码分析校正。
