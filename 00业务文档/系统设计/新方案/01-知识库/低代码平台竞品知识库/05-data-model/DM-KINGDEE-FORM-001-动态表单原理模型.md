# 数据模型：金蝶动态表单原理模型

## 证据边界

金蝶苍穹为闭源商业平台。本卡只基于官方公开资料、产品手册和公开报道抽象原理模型；不能证明内部表结构、类名、接口名或运行时源码实现。

## 本质模型

金蝶动态表单可抽象为企业业务界面的基类模型：

```text
动态表单元模型
  -> 单据
  -> 基础资料
  -> 过滤条件框
  -> 系统参数
  -> 列表界面
  -> 主控台界面
  -> 操作结果对话框
```

表单设计器则是这个元模型的可视化编辑器，负责把业务对象、字段、控件、布局、状态、权限、操作、流程和发布信息写入元数据。

## 运行原理推断

```text
设计期：
业务对象/动态表单 -> 字段/控件/布局/属性/权限/状态/规则 -> 元数据校验 -> 测试

发布期：
元数据 -> 视图/菜单/主控台 -> 部署包/版本 -> 流程/转换/套打等关联配置

运行期：
动态解释引擎加载元数据 -> 生成界面和行为 -> 插件参与扩展逻辑 -> 权限/状态/流程共同约束用户操作
```

## 不变量

1. 表单定义必须可追溯到业务对象，否则单据状态、流程、权限和编码规则无法统一。
2. 字段和控件不是纯 UI 属性，必须参与数据、权限、校验和运行时解释。
3. 已发布表单和草稿表单需要隔离，否则元数据变更会影响生产运行。
4. 多页面布局共享同一业务对象时，字段权限和操作权限必须在服务端统一判定。
5. 插件扩展必须有注册、生命周期和作用范围，否则会破坏模型可解释性。
