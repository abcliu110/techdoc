# 知识库驱动新产品创建与迭代流程（V 模型）

> 版本：v0.7
> 日期：2026-07-06
> 总纲关系：本文属于《00-总纲-知识库与产品交付双循环研发流程.md》的循环 2（产品交付循环），按经典软件工程 V 模型组织：左臂逐层分解（业务需求 → 需求规格 → 架构设计 → 详细设计），右臂逐层验证（单元 → 集成 → 系统 → 验收），每个左臂产出在产生时就定义右臂验证标准。所有 accepted ADR 和最终设计决策只在本流程产生。AI 执行本文时必须遵守总纲第 6 章"AI 自动执行契约"。
> 适用范围：使用领域知识库创建新产品方案，并从产品蓝图推进到设计、实现、上线和后续迭代的半自动化流程。
> 核心原则：知识库是证据来源，不是需求本身；AI 是草案生成器和一致性检查器，不是最终决策者。
> 命名说明：本文的核心不是泛化的“产品工厂”概念，而是“知识库 → 新产品方案 → 上线 → 知识库回写”的工作流；当该工作流稳定后，才可产品化为“产品工厂模块”。
> 风险方法：产品方案必须引用《03-方法论-系统级失败模式与设计约束清单.md》《03A-方法论-失败模式与设计约束清单建设.md》《04-清单-通用系统级失败模式与设计约束.md》《04A-清单-复杂平台级失败模式与设计约束.md》，并为具体领域生成领域实例清单。

## 0. 执行入口

### 0.1 最小输入契约

启动本流程时，用户只需要提供：

```text
产品名称或产品方向（必须）
知识库名称（必须；对应 ../01-知识库/{知识库名称}/）
```

产品方案目录约定：

```text
../02-产品方案/{产品名称}/
```

执行者必须自行完成存在性检查和分支判定，不得反问用户"是新建还是迭代"：

```text
输入：产品名称 + 知识库名称
→ 检查 ../01-知识库/{知识库名称}/ 是否存在
   → 不存在：停止，提示先执行《01A-工作流-知识库建设与持续完善.md》建库
   → 存在但成熟度不足（见阶段 1 盘点门禁）：只允许进入调研补证，不进入 PRD
→ 检查 ../02-产品方案/{产品名称}/ 是否存在
   → 不存在：从阶段 0 开始新产品流程
   → 已存在：扫描已有产物（Brief / 蓝图 / PRD / ADR / 详细设计 / 任务卡 / 测试规格），
     判定当前处于哪个阶段，从第一个未完成的阶段断点续跑；
     若知识库基线快照已过期，先走阶段 1A 变更再同步
```

### 0.2 给 AI 的启动提示

```text
请按《02-流程-知识库驱动新产品创建与迭代.md》执行本轮产品创建或迭代。
我只指定产品名称和知识库名称；你需要自行检查产品方案目录是否已存在，
不存在则从阶段 0 新建，已存在则扫描已有产物断点续跑。
执行时必须遵守《00-总纲》第 6 章 AI 自动执行契约：
每个动作归入自动执行 / 执行后送审 / 必须人工决策三级之一；
遇到必须人工决策的点（产品定位、首版范围、单向门 ADR、风险接受、上线放行等）
停下并输出决策材料，禁止代替我决策后继续。
所有产出必须走 5.0 产出质量保障工作流（五步流水线），登记追溯矩阵。
关键设计点（架构、功能、数据模型、UI、UI 组件）必须走 5A 统一设计决策工作流，
三路候选（知识库证据 + 行业标准 + 自主创新）比较后才能形成 accepted ADR。
未通过阶段 6A 开发启动门禁，不得进入编码。
```

### 0.3 与 01A 的分工速查

```text
想扩充 / 完善 / 深挖知识库 → 执行《01A-工作流-知识库建设与持续完善.md》
想用知识库做产品、写代码   → 执行本文
知识库不存在或成熟度不足   → 先 01A 后本文
产品上线复盘后            → 本文阶段 9 回写知识库，触发 01A 的下一轮补证
```

## 1. 目标

本文定义一套从领域知识库到新产品上线的完整流程，先作为团队内部半自动化作业流程运行，后续再沉淀为低代码平台的“产品工厂模块”。

第一版目标：

```text
知识库输入
→ 知识库变更识别
→ 产品机会识别
→ 产品蓝图
→ PRD
→ ADR
→ 系统级失败模式与设计约束清单
→ 详细设计
→ 任务卡
→ 测试规格
→ 上线门禁
→ 复盘回写
```

第一版不直接做成平台功能，也不允许 AI 绕过证据成熟度、ADR、人工评审和测试门禁直接进入实现。

## 2. 总体架构：V 模型 + 三轨协同

### 2.0 V 模型阶段映射

本流程的阶段按经典软件工程 V 模型组织。左臂逐层分解，右臂逐层验证；每个左臂产出在产生时就必须定义右臂对应层级的验证标准。

```text
左臂（分解）                              右臂（验证）
P1 业务需求（产品蓝图 / BRD）    ←验收测试→  P6 验收与上线运营
P2 需求规格（PRD / SRS）        ←系统测试→  P5 系统测试
P3 架构设计（架构+数据模型+ADR） ←集成测试→  P4b 集成测试
P4 详细设计（模块+接口+UI+组件） ←单元测试→  P4a 实现与单元测试

P0 准备（启动 / 知识库盘点 / 机会识别）
P7 复盘与知识库回写（闭环回循环 1）
```

本文阶段编号与 V 模型阶段对应关系：

| 本文阶段 | V 模型阶段 | 性质 |
|---|---|---|
| 阶段 0 启动与范围确认 | P0 准备 | 左臂入口 |
| 阶段 1 知识库盘点 | P0 准备 | 左臂入口 |
| 阶段 1A 知识库变更再同步 | 横切 | 变更控制的一部分 |
| 阶段 2 机会识别 | P0 准备 | 左臂入口 |
| 阶段 3 产品蓝图 | P1 业务需求 | 左臂 |
| 阶段 4 PRD 生成 | P2 需求规格 | 左臂 |
| 阶段 5 架构与 ADR | P3 架构设计 | 左臂 |
| 阶段 5A 统一设计决策工作流 | 横切 | 适用 P1-P4 所有关键设计点 |
| 阶段 6 详细设计与任务卡 | P4 详细设计 | 左臂 |
| 阶段 6A 开发启动门禁 | P4 出口门禁 | 左右臂交汇点 |
| 阶段 7 实现与验证 | P4a 单元 / P4b 集成 / P5 系统 | 右臂 |
| 阶段 8 上线与运营 | P6 验收与上线 | 右臂 |
| 阶段 9 复盘与知识库回写 | P7 复盘 | 闭环 |

V 模型规则：

```text
左臂每个阶段产出时必须同步产出对应右臂的验证标准：
  P1 产品蓝图 → 同步定义验收准则（成功指标、验收场景）
  P2 需求规格 → 同步定义系统测试要点（每条需求的验收标准）
  P3 架构设计 → 同步定义集成测试要点（模块边界、契约、失败模式验证）
  P4 详细设计 → 同步定义单元测试规格
左臂上游未通过评审基线，下游不得开工。
右臂每层测试失败时，缺陷回溯到对应左臂层级修复，不允许在右臂"就地打补丁"绕过左臂产物。
```

### 2.0A 三轨协同

该流程同时由三条协同轨道保障：产品生成轨推进交付物，证据治理轨约束证据质量，知识库变更同步轨负责在知识库变化后反查产品影响。

### 2.1 产品生成轨

产品生成轨负责把知识库转化为产品交付物。

```text
知识库盘点
→ 机会识别
→ 产品蓝图
→ PRD
→ 架构与 ADR
→ 详细设计与任务卡
→ 实现与验证
→ 上线与运营
```

### 2.2 证据治理轨

证据治理轨负责防止把未验证的竞品观察误当成产品事实。

```text
证据卡
→ 成熟度评分
→ 风险门禁
→ 变更影响分析
→ 实测 / 源码补证 / ADR 风险接受
→ 验证结果回写知识库
```

三条轨道在每个阶段交叉校验。产品生成轨负责推进，证据治理轨负责约束，知识库变更同步轨负责发现变化并反查产品覆盖。

### 2.3 知识库变更同步轨

知识库不是一次性输入。只要证据卡、能力矩阵、痛点、竞品结论、源码分析或实测结果发生变化，就必须启动同步轨，判断已有产品文档、任务卡和测试规格是否需要更新。

```text
知识库基线快照
→ 知识库变更检测
→ 变更分类
→ 影响范围定位
→ 产品功能覆盖审计
→ 产品文档差异检查
→ 决策：同步 / 延后 / 风险接受 / 作废
→ 更新 PRD、ADR、详细设计、任务卡、测试规格
→ 记录变更追踪
```

原则：

1. 知识库更新不会自动改需求；必须先做影响分析。
2. 证据等级上升，可能把候选能力推进到 PRD 或实现门禁。
3. 证据等级下降，必须冻结相关实现、补 ADR 风险接受或降级需求。
4. 竞品新增能力不是自动需求，只能进入机会池或差距清单。
5. 已上线产品受到影响时，必须走变更评审、回归测试和发布计划。

### 2.4 知识库变化检测机制

变化检测不能只靠人读目录。每次产品生成或迭代开始前，必须为知识库生成一份可比对的基线快照。

基线快照包含：

```text
文件路径
内容 hash
标题
证据等级
竞品名称
能力标签
业务对象标签
流程标签
风险标签
适用产品域
引用的截图/源码/链接
最后更新时间
```

检测输入：

```text
上一次 KnowledgeSnapshot.json
当前知识库目录
当前 CapabilityMatrix
当前 Evidence-Maturity-Matrix
当前 Open-Questions
```

检测输出：

```text
KnowledgeDiff.md
KnowledgeDiff.json
Knowledge-Change-Log.md
```

变更类型：

| 类型 | 判定方式 | 示例 |
|---|---|---|
| 新增 | 当前快照有，上一版无 | 新增 Frappe print format 证据卡 |
| 删除 | 上一版有，当前无 | 移除某竞品能力卡 |
| 内容修改 | 文件 hash 变化 | 能力描述、截图、源码证据更新 |
| 证据等级变化 | evidenceLevel 字段变化 | 未验证 -> 直接事实 |
| 能力标签变化 | capabilityTags 变化 | 从“报表”扩展到“打印/归档” |
| 矩阵变化 | CapabilityMatrix 单元格变化 | NocoBase 新增 calendar view |
| 冲突变化 | 同一能力出现相反结论 | 文档说支持，实测显示不支持 |

最低实现可以先用文件 hash + front matter/表格解析；成熟后再把证据卡结构化为 JSON/YAML。

### 2.5 产品功能覆盖审计机制

知识库变化被检测出来后，不能直接问“要不要做”。必须先问：当前产品是否已经覆盖了这个能力，覆盖到什么程度，证据在哪里。

每个产品方案必须维护一张能力追踪矩阵：

```text
CapabilityTraceMatrix:
  capabilityId
  capabilityName
  sourceEvidenceIds
  evidenceLevel
  productDecision: must / should / later / reject / not-applicable
  coverageStatus: covered / partial / missing / rejected / planned
  prdRefs
  adrRefs
  designRefs
  taskRefs
  testRefs
  releaseRefs
  lastCheckedAt
```

覆盖状态口径：

| 状态 | 含义 | 允许交付吗 |
|---|---|---|
| covered | PRD、设计、任务、测试均有承接，测试规格可验证；高风险能力还必须有 ADR | 可以 |
| partial | 只有 PRD 或设计，缺任务/测试/ADR | 不可声明完成 |
| missing | 知识库有能力，产品无任何承接 | 必须进入差距清单或明确拒绝 |
| rejected | 明确不做，有理由和替代方案 | 可以，但必须在 PRD 不做事项或 ADR 中可追溯 |
| planned | 已列后续版本，有里程碑和门禁 | 可以，但不能冒充首版能力 |

覆盖审计流程：

```text
读取 KnowledgeDiff
→ 提取新增/变化 capabilityId
→ 查询 CapabilityTraceMatrix
→ 扫描 PRD/ADR/详细设计/任务卡/测试规格引用
→ 给出覆盖状态
→ 对 missing/partial 生成差距任务
→ 对 rejected/planned 检查边界声明是否仍成立
```

审计输出：

```text
Product-Capability-Coverage-Audit.md
CapabilityTraceMatrix.md
Gap-To-Product-Tasks.md
```

## 4. 证据成熟度分级

任何从知识库进入产品设计的结论，都必须标注证据等级。

| 等级 | 定义 | 是否可进入需求 |
|---|---|---|
| 直接事实 | 官方文档、源码、可复现实测、截图、日志、运行导出 | 可进入 PRD 和设计 |
| 源码初证 | 已读源码，但未完成运行时验证 | 可进入设计草案，进入实现前需补验证或 ADR |
| 高可信推断 | 多来源材料一致，但缺源码或实测 | 只能进入假设或候选方案 |
| 未验证 | 只有公开描述、营销材料、二手资料或能力矩阵猜测 | 不得写成事实，只能进入待验证清单 |
| 风险接受 | 证据不足但业务决定先推进，并已记录 ADR 后果 | 可推进，但必须有回滚和监控 |

强制规则：

```text
未验证能力不能直接进入 PRD 的“必须交付”。
高可信推断不能直接变成不可逆架构决策。
源码初证进入实现前必须有实测、源码复核或 ADR 风险接受。
```

## 4.1 知识库变更分级

知识库变化按影响强度分级处理。

| 等级 | 变化类型 | 处理要求 |
|---|---|---|
| K0 资料补充 | 修正文案、补链接、补截图，不改变结论 | 记录变更即可，不触发产品文档同步 |
| K1 证据增强 | 证据等级上升，如“未验证”变“源码初证”或“直接事实” | 检查机会池、PRD 候选项、实现门禁是否可推进 |
| K2 证据削弱 | 证据被推翻、来源失效、实测不成立 | 立即标记受影响需求和任务，不得继续实现相关不可逆设计 |
| K3 竞品能力变化 | 竞品新增/删除/调整功能或商业策略 | 进入机会识别和差距分析，不自动进入 PRD |
| K4 架构/安全冲击 | 新证据影响单向门、权限、租户、数据、合规、License | 必须重开 ADR 或补风险接受，必要时冻结实现 |
| K5 上线事故/用户反馈 | 真实用户反馈、生产问题或上线复盘回写 | 必须更新知识库、风险清单、测试规格和下一轮路线 |

同步产物：

```text
Knowledge-Change-Log.md
ImpactAssessment-YYYYMMDD-主题.md
Affected-Artifacts.md
Sync-Decision.md
```

## 5. 阶段流程

### 5.0 横切：产出质量保障工作流

本流程任何阶段的任何产出，一律走同一条五步流水线。AI 执行者必须把该流水线内化为每个阶段的默认执行方式。

```text
1. 入口检查：上游基线产物齐备且已通过评审；无基线不开工。
2. 生产：按模板生产；产出必须引用上游追溯编号。
3. 自检：对照该产物的 DoD 检查单逐项核对，输出自检结果——
   完整性：模板必填项无缺失；
   一致性：与上游基线产物、ADR、术语表无冲突；
   可测试性：每条需求 / 设计点有可执行的验收或验证标准；
   可追溯性：上游编号、证据引用、RTM 登记齐全。
4. 评审：AI 预审（一致性 / 完整性 / 追溯检查）+ 人工评审
   → 结论：通过 / 有条件通过（记录补齐期限）/ 打回。
5. 基线：通过后纳入基线；此后修改只能走变更控制；同步更新追溯矩阵。
```

追溯矩阵（RTM）：

```text
维护一份端到端追溯矩阵（CapabilityTraceMatrix.md 即其载体）：
业务需求 ↔ 需求规格 ↔ 架构/ADR ↔ 详细设计 ↔ 代码 ↔ 测试用例。
每个产出通过五步流水线时必须登记 RTM。
任何一层变更必须先查 RTM 波及面，再改动。
```

变更控制：

```text
基线后的修改必须走：变更申请 → 影响分析（查 RTM + 知识库变更同步轨）
→ 评审 → 更新基线 → 更新 RTM。
禁止绕过变更流程直接改基线产物。
阶段 1A（知识库变更再同步）是变更控制在"知识库侧变化"场景下的具体实现。
```

门禁：

```text
没有通过五步流水线的产出，不得作为下游阶段的输入。
没有登记 RTM 的产出，不得声明完成。
```

| 覆盖状态 | 处理 |
|---|---|

| rejected | 复核拒绝理由是否仍成立；不成立则重开机会识别 |
| planned | 检查里程碑、边界声明和测试门禁是否仍匹配 |

### 5.7A 阶段 5A（横切）：统一设计决策工作流

本工作流回答一个问题：当生成新的软件时，如何证明架构、数据模型、业务规则、权限、流程、UI/组件等关键设计是“当前约束下最合适”的，而不是 AI 主观生成的。

本工作流不是一个串行阶段，而是横切于 P1-P4 的统一决策方法：架构设计、功能设计、数据模型、UI 设计、UI 组件设计五类设计问题，凡属关键设计点，一律走同一条六步决策链：

```text
1. 定义问题：设计问题陈述 + 质量属性场景 + 约束条件。
2. 收集候选：三路候选，候选总数 ≥ 2——
   a. 知识库路：最佳设计决策输入包中竞品证据支撑的候选；
   b. 行业路：行业标准与最佳实践（独立于竞品的行业基准）；
   c. 创新路：至少提出一个试图超越现有候选的自主创新方案。
3. 权衡分析：权衡矩阵（收益 / 代价 / 风险 / 失败模式 / 回滚成本 / 演进成本）。
4. 降险验证：高风险候选做 Spike / 原型 / PoC，用证据说话。
```

```text

有决策：通过 ADR 记录采纳方案、拒绝方案、后果和回滚方式。
有承接：PRD、详细设计、任务卡和测试规格能追溯到该决策。
有验证：测试、评审、原型、性能压测、安全审查或可用性验证覆盖关键风险。
有复审：后续证据变化时能定位并重开该决策。
```

## 6. AI 半自动化工作方式

本章是《00-总纲》第 6 章"AI 自动执行契约"在本流程中的落地。AI 执行本流程时，每个动作必须归入总纲定义的三级之一：自动执行 / 执行后送审 / 必须人工决策。遇到"必须人工决策"点时，AI 停下并输出决策材料（候选、证据、权衡、建议），不得代替人决策后继续。支持断点续跑：恢复执行时先扫描已有产物判定当前阶段，从第一个未完成阶段继续。

### 6.1 AI 可自动执行

```text
读取知识库
汇总证据
生成草案

做缺口分析
识别知识库变更并生成影响评估草案
生成 KnowledgeSnapshot 和 KnowledgeDiff
定位受影响 PRD/ADR/详细设计/任务卡/测试规格
生成产品功能覆盖审计草案
生成模板
生成测试矩阵
生成上线清单
生成复盘草案
```

### 7.2 追踪编号和引用规则

为了让知识库变化能自动反查产品文档，所有产物必须使用稳定编号，不允许只靠标题或自然语言匹配。

编号规则：

```text
capabilityId: CAP-领域-三位序号，例如 CAP-LCDP-001
evidenceId: EV-来源-三位序号，例如 EV-FRAPPE-012
requirementId: REQ-模块-三位序号，例如 REQ-WF-003
adrId: ADR-三位序号，例如 ADR-007
designId: DES-模块-三位序号，例如 DES-WF-002
taskId: TASK-模块-三位序号，例如 TASK-WF-006
testId: TEST-类型-三位序号，例如 TEST-E2E-004
```

引用规则：

```text
PRD 必须引用 capabilityId 和 evidenceId。
ADR 必须引用 requirementId；如果决策来自知识库证据，还必须引用 evidenceId。
详细设计必须引用 requirementId 和 adrId。
任务卡必须引用 requirementId 和 designId。
测试规格必须引用 requirementId 和 taskId。
CapabilityTraceMatrix 必须反向列出 PRD/ADR/设计/任务/测试引用。
```

覆盖审计匹配规则：

```text
只有明确编号引用才算强匹配。
只有标题、关键词或相似描述，不能判定 covered，最多判定 partial。
引用编号存在但文件或章节不存在，判定 partial。
测试规格没有 expectedResult 或 evidenceOutput，不能支撑 covered。
高风险能力缺 ADR，不能判定 covered。
rejected 必须能追溯到 PRD 不做事项或 ADR 决策。
planned 必须有里程碑、边界声明和下一次复查时间。
```

## 10. 最小试运行建议

建议用当前低代码平台项目做第一次试运行。

试运行范围：

```text
输入：低代码平台竞品知识库
产品方向：企业业务低代码平台
首版样例：客户-订单-审批
输出：蓝图、PRD、ADR、详细设计、任务卡、测试规格、上线门禁
回写：补充知识库缺口清单和实测证据要求
变更演练：任选一条竞品证据等级变化，跑一次 ImpactAssessment 和 Sync-Decision
覆盖审计：任选 5 条新增或变化能力，检查产品是否有 PRD/ADR/设计/任务/测试承接
```

成功标准：

```text
每条核心需求能追溯到证据或业务决策
每个单向门有 ADR
每个任务有详细设计和测试规格
每个上线门禁有验证证据
未验证竞品结论没有被写成事实
知识库变更后能定位受影响需求、ADR、任务卡和测试规格
新增能力能输出 covered/partial/missing/rejected/planned 状态
```

### 10.1 低代码平台知识库变化演练样例

场景：低代码平台竞品知识库新增一条证据，显示某竞品支持“打印模板可视化配置”，证据等级从未验证提升为源码初证。

知识库变化：

```text
changeType: evidence-level-changed
capabilityId: CAP-LCDP-PRINT-001
capabilityName: 打印模板可视化配置
evidenceId: EV-COMPETITOR-PRINT-003
oldEvidenceLevel: 未验证
newEvidenceLevel: 源码初证
riskLevel: K1
```

覆盖审计：

```text
读取 CapabilityTraceMatrix
检查 PRD 是否有 REQ-PRINT-* 引用 CAP-LCDP-PRINT-001
检查 ADR 是否存在打印模板数据模型、表达式安全或导出边界决策
检查详细设计是否有 DES-PRINT-* 承接模板结构、预览、渲染和权限
检查任务卡是否有 TASK-PRINT-* 拆分
检查测试规格是否有 TEST-E2E-* 或 TEST-SEC-* 验证打印预览、权限和表达式注入
```

可能结果：

| 审计项 | 结果 |
|---|---|
| PRD | 有“打印模板”候选描述，但没有 requirementId |
| ADR | 无 |
| 详细设计 | 无 |
| 任务卡 | 无 |
| 测试规格 | 无 |
| coverageStatus | partial |

同步决策：

```text
decision: 延后
reason: 源码初证不足以进入首版 must；当前客户-订单-审批 MVP 不依赖打印模板。
action:
  1. 在 Opportunity-Backlog 增加 CAP-LCDP-PRINT-001。
  2. 在 CapabilityTraceMatrix 标记 planned。
  3. 补 Open-Questions：打印模板表达式安全、数据权限、导出格式、预览一致性。
  4. 下一轮若进入 P1，先补 ADR-打印模板模型与表达式安全，再生成 PRD/设计/任务/测试。
```

如果业务决定首版必须支持：

```text
decision: 同步
requiredFix:
  1. PRD 增加 REQ-PRINT-001，并引用 CAP-LCDP-PRINT-001 / EV-COMPETITOR-PRINT-003。
  2. 新增 ADR-打印模板数据模型与表达式安全。
  3. 新增 DES-PRINT-001，覆盖模板结构、字段绑定、预览、权限和失败模式。
  4. 新增 TASK-PRINT-001..n，并写清验收标准。
  5. 新增 TEST-E2E-PRINT-001 和 TEST-SEC-PRINT-001。
  6. 更新 Development-Ready-Checklist，未补齐前不得进入阶段 7。
```
