# 03A-方法论-失败模式与设计约束清单建设

> 版本：v1.0
> 日期：2026-07-06
> 定位：软件开发业界通用的建设方法论，用于从需求、设计、知识库、代码、测试、事故和运维材料中建设或更新“系统级失败模式与设计约束清单”。
> 上位方法论：《03-方法论-系统级失败模式与设计约束清单.md》。

本文不是一次性的执行工作流，而是清单建设的方法规范：规定如何取材、筛选、改写、分级、承接和验证。具体项目或 AI 执行工作流可以引用本文，但不得把本文替代为领域清单本身。

---

## 1. 使用时机

在以下场景使用本文方法：

1. 新项目启动，需要建立首版清单。
2. 已有项目准备做大版本设计，需要补齐系统性风险。
3. 竞品学习、用户反馈、事故复盘或源码阅读发现新风险。
4. 需求、架构、实现、测试或上线阶段发现清单缺项。
5. 已有清单需要升级、合并、降级或废弃条目。

## 2. 输入材料

按可获得程度读取以下材料：

```text
需求材料
业务流程
核心对象和状态说明
接口或交互说明
数据样例
权限和角色说明
部署和运行约束
测试用例与缺陷记录
用户反馈
事故复盘
竞品或外部证据
已有设计文档
已有风险清单
```

缺少材料时不要编造结论，应把缺口写入“待补证问题”。

## 3. 建设步骤

### 步骤 1：确定系统边界

输出：

```text
系统名称：
目标用户：
核心能力：
核心对象：
外部依赖：
不在范围内：
```

要求：

- 先说明系统解决什么问题。
- 明确哪些用户、对象、流程和依赖在本轮清单范围内。
- 范围不清时，先生成候选清单，不直接定稿。

### 步骤 2：从通用问题族筛选候选

读取：

```text
04-清单-通用系统级失败模式与设计约束.md
```

对每个问题族标注：

```text
适用 / 不适用 / 待确认
```

输出表：

| 通用问题族 | 判断 | 原因 | 需要补充的领域信息 |
|---|---|---|---|

规则：

- 不适用必须写原因。
- 待确认必须写需要什么证据。
- 适用项必须进入步骤 3。

### 步骤 3：改写为领域实例

把通用问题族改写成当前系统的领域语言。

模板：

```text
通用问题族：
领域对象：
领域动作：
触发条件：
失败后果：
候选标题：
```

示例：

```text
通用问题族：重复执行造成重复结果
领域对象：订单支付
领域动作：提交支付
触发条件：用户重复点击或支付网关重试
失败后果：重复扣款或订单状态不一致
候选标题：重复支付导致资金和订单状态不一致
```

### 步骤 4：补充证据来源

每个候选条目必须至少标注一种来源：

```text
需求推导
设计推导
代码或源码证据
测试缺陷
用户反馈
事故复盘
竞品证据
行业经验
待补证假设
```

只有“待补证假设”的条目可以保留为候选，但不得作为已确认风险进入上线门禁。

### 步骤 5：分级

按后果分级：

| 等级 | 判断问题 |
|---|---|
| 高 | 是否会造成不可恢复数据错误、严重安全问题、合规问题、大面积不可用或重大业务损失？ |
| 中 | 是否会造成局部不可用、错误结果、明显体验损害、恢复成本较高或维护风险？ |
| 低 | 是否主要造成长期维护成本、理解成本、局部体验问题或可控返工？ |

要求：

- 等级依据必须写“后果”，不能写“实现很复杂”。
- 拿不准时先标“待确认”，并写需要什么证据。

### 步骤 6：写设计约束

每个已确认条目必须回答：

```text
为了避免这个失败，系统必须满足什么条件？
哪些边界必须写清？
哪些状态或关系必须保持不变？
哪些异常必须可恢复？
哪些行为必须被拒绝、降级、补偿或审计？
```

设计约束不能只写“加强校验”“注意异常”“做好测试”。

### 步骤 7：定义验证方式

为每个条目选择至少一种验证方式：

```text
需求评审
设计评审
单元测试
集成测试
端到端测试
契约测试
并发测试
故障演练
安全测试
回归测试
监控告警
人工检查清单
风险接受记录
```

高等级条目如果没有可执行验证方式，必须明确写“风险接受”或“待补证”，不能默认为已解决。

### 步骤 8：指定承接责任

每个条目必须指定承接位置：

```text
需求文档
设计文档
实现任务
测试规格
发布清单
运维手册
复盘记录
人工评审清单
```

没有承接位置的条目状态必须是：

```text
未闭环
```

### 步骤 9：形成清单

条目格式：

```text
### 编号 等级 标题

**失败模式**：
**触发条件**：
**后果**：
**设计约束**：
**验证方式**：
**承接责任**：
**证据来源**：
**状态**：
```

编号建议：

```text
A/B/C... 表示问题族
数字表示序号
```

### 步骤 10：评审和收敛

评审问题：

```text
是否把领域问题误写成通用问题？
是否只写了“风险名称”，没有写失败方式？
是否只写了“加强测试”，没有写验证方式？
是否存在没有承接责任的高等级条目？
是否存在重复条目？
是否存在应该下沉到专门清单或领域实例的条目？
```

评审结果：

```text
保留
合并
拆分
降级
升级
下沉到领域实例
标记待补证
废弃
```

## 4. 更新已有清单

更新已有清单时，不要直接追加。

先做四项检查：

1. 新风险是否已经被旧条目覆盖。
2. 新风险是否只是旧条目的领域实例。
3. 新风险是否需要升级旧条目等级。
4. 新风险是否暴露了承接或验证缺口。

更新记录必须写明：

```text
变更类型：新增 / 合并 / 拆分 / 升级 / 降级 / 废弃
变更原因：
证据来源：
影响的承接文档：
后续动作：
```

## 5. 产物清单

建设完成后至少产出：

```text
系统级失败模式与设计约束清单.md
适用性筛选表
待补证问题清单
未闭环条目清单
变更记录
```

如果是平台类软件，还应参考：

```text
04A-清单-复杂平台级失败模式与设计约束.md
```

## 6. 完成标准

完成必须满足：

- 已读取通用问题族清单。
- 适用、不适用、待确认均有理由。
- 领域条目使用当前系统语言，不是照抄通用条目。
- 高等级条目有验证方式或风险接受。
- 每个已确认条目都有承接责任。
- 未闭环条目单独列出。
