# AI UI 质量保障流程规范

> 版本：v2.0
> 日期：2026-07-10
> 类型：横切质量规范 / AI 原型生成 + UI 视觉审核 + 设计门禁一体化流程
> 适用对象：Web、桌面端、移动端的商业软件页面（含复杂工作台、数据密集型界面、交互式设计器、后台管理系统等）
> 上游规范：`05-研发流程规范.md`（P2-P5 的 11 个工程活动 + AI 执行变体 L1-L4 复杂度分级）

## 0. 一句话定义

**本规范的核心目标只有两个：**

```text
1. 设计的原型必须比竞品和行业标杆强，且强的地方有真实任务证据支撑
2. 原型没有视觉缺陷（几何、Token、响应式、可访问性、交互状态均无缺陷）
```

本规范是一套 AI 原型生成 + 设计门禁 + 运行态视觉审核的一体化质量保障流程。核心是"边生成、边检查、边修复"循环（生成阶段，第二部分第 25 节），配合标杆证据驱动设计门禁（设计阶段，第一部分 S0-S7）和只读审核门禁（审核阶段，第二部分 U0-U9），形成完整闭环：

```text
设计前：标杆准入 + 证据采集（S0-S2）
设计中：概念设计 + 渲染自检（S3-S4）
设计后：独立 Critic + 自动化 Verifier（S5-S7）
生成时：预防性检查 + 内嵌自检自修循环（第二部分第 25 节）
审核时：只读审核，职责分离，禁止修改（第二部分第 1-24 节）
```

### 0.1 AI 最小输入与模式选择

```text
最小输入：被评审页面/原型/设计稿的位置 + 本轮目标。
AI 自动发现：启动命令、页面路由、视口、状态、测试数据、设计基线和既有证据包。
模式选择：新设计/重大改版 → S0-S7；实现还原或视觉回归 → U0-U9；生成原型 → 第 25 节后必须进入 U0-U9。
轻量条件：仅低风险局部视觉变更且不影响任务流、布局结构、设计 Token、响应式或可访问性。
人工决策：S0 Brief 冻结、基线批准、UI-R3 终审、风险接受和流程降级。
```

无法启动真实页面、无法触发关键状态或没有批准基线时，结论必须为 `EVIDENCE_INSUFFICIENT`，不得以静态代码阅读替代运行态证据。

### 0.2 AI 执行状态与幂等规则

S0-S7、U0-U9 和第 25 节的每一步均使用总纲 6.5 的统一状态。`artifact_hash + spec_version + visual_contract_hash` 构成幂等键；哈希未变时复用有效证据，任一哈希变化时只使受影响证据失效并重跑对应门禁，不得无差别重跑全部流程。

每项结论必须区分：运行事实、视觉/交互推断、审核决定。AI 在 PASS 前必须主动检查长文本、空数据、错误态、最窄视口、键盘路径和至少一个业务失败态；未覆盖项写入 `residual_risks`。

核心原则：

```text
没有冻结版本，不审核。
没有真实运行，不通过。
没有状态证据，不算覆盖。
没有批准基线，不得声称像素级通过。
没有独立审核，不得自评放行。
发现问题只出报告，审核过程中禁止修改被审核原型、设计稿或代码。
设计阶段（S0-S7）：标杆证据驱动，角色分离，独立 Critic
生成阶段（第二部分第 25 节）：每个组件边生成、边检查、边修复
审核阶段（第二部分第 1-24 节）：只读审核，职责分离，禁止修改
```

```

## 第一部分：设计门禁流程（S0-S7）

### 1. 为什么需要设计门禁

AI 设计商业软件 UI 时常见失败模式：

```text
学到表层风格：卡片、圆角、渐变、留白
忽略任务结构：主任务、批量操作、异常恢复、状态转换
忽略真实复杂度：权限、脏数据、并发、异步失败、超长文本
静态截图看起来不错，真实操作链路断裂
自评时用"现代 / 高级 / 简洁"替代证据
用结构化文档包装幻觉，形成"看似完整"的假证据链
```

因此，本流程把 UI 设计质量拆成四个底座：

```text
证据真实性：每条标杆结论必须能追溯到原始来源，且来源不可由 Designer 自行伪造
任务真实性：必须在真实任务链路中比较，而不是只看截图
评审独立性：Designer 不能自我裁判，Critic 必须能独立抽查证据且与 Designer 物理隔离
反作弊约束：所有量化指标必须绑定任务价值，防止为了刷分而堆字段、堆状态、堆 artifact
```

### 2. 适用与不适用

#### 2.1 适用场景

```text
复杂业务工作台
数据密集型管理界面（列表、详情、表单、报表等）
交互式设计器（表单设计器、流程设计器、报表设计器等）
数据分析、监控、运营后台
需要长期高频使用的企业软件页面
需要证明"不能低于直接竞品"的关键 UI / 交互
```

#### 2.2 不适用场景

```text
一次性营销页、品牌海报、纯视觉创意稿
没有明确任务目标的探索性视觉练习
只需要轻量样式调整的小改动
没有可访问 benchmark 或无法采集证据的场景
```

不适用并不表示可以无标准设计，而是应改用更轻的视觉评审或设计系统规范，不启用本文的完整门禁。

#### 2.3 分级执行标准

页面风险等级决定走完整流程还是轻量流程，判定必须由人类在 S0 确认，AI 不得自行降级。

| 等级 | 判定标准（满足任一即为核心） | 流程 |
|---|---|---|
| 核心页面 | 涉及金额 / 权限 / 数据修改 / 批量操作 / 多步流程 / 合规审计 | 完整流程 S0-S7 |
| 普通页面 | 纯展示 / 配置 / 只读列表 / 低频操作 | 轻量版流程 |

### 3. 角色与职责边界

推荐压缩成 4 个核心角色，避免前段职责重叠。

| 角色 | 职责 | 禁止事项 | 关键产物 |
|---|---|---|---|
| Benchmark Analyst | 选择功能等价标杆，采集证据，抽取任务模型、状态清单、模式 / 反模式 | 不得给最终设计建议，不得改写 brief | benchmark shortlist、任务等价矩阵、证据台账、任务模型、状态清单、模式库 |
| Product Designer | 基于约束包和证据包设计信息架构、任务流、布局原则和交互方案 | 不得自评通过，不得修改 benchmark 证据 | IA、主任务流、布局原则、设计 rationale、非目标、风险清单 |
| Critic | 独立红队评审，抽查原始证据，输出缺陷分类、严重级别和回退目标状态 | 不得直接重画方案，不得只消费 Designer 摘要 | 缺陷单、P0 判定、回退路由、二次审计建议 |
| Verifier | 执行自动化门禁：任务回放、状态覆盖、a11y、截图布局、错误注入 | 不做审美判断，不修改评分标准 | 自动化验证报告、截图包、状态覆盖报告、失败门禁清单 |

在需要可运行原型时，可增加 `Visual Implementer`，负责把设计落成真实界面或原型；它只能实现已批准方案，不得扩大产品范围。

#### 3.1 角色分离的执行机制

必须分离的职责：

```text
Designer / Critic / Verifier 必须分离
```

原因：一个负责创造，一个负责独立挑战，一个负责硬门禁。三者合并会导致自评、自证和自我放行。

**角色体系统一说明**：本规范分两部分，角色名称不同但对应关系固定。第一部分（S0-S7 设计门禁）使用设计阶段角色名，第二部分（U0-U9 视觉审核）使用审核阶段角色名。执行时同一人/AI在不同阶段切换角色名，但职责分离约束跨阶段生效。

| 第一部分角色（设计阶段） | 第二部分角色（审核阶段） | 是否同一人/AI | 职责连续性 |
|---|---|---|---|
| Product Designer | Submitter | 是（Designer 生成完成后切换为 Submitter 提交审核） | Designer 不得自评放行；Submitter 不得批准自己的产物 |
| Critic | UX Critic | 是（同一独立角色，设计阶段叫 Critic，审核阶段叫 UX Critic） | 跨阶段保持独立，不得读取 Designer/Submitter 的 rationale |
| Verifier | Visual Verifier | 是（同一自动化角色，设计阶段叫 Verifier，审核阶段叫 Visual Verifier） | 跨阶段执行自动化门禁，不做审美判断 |
| 产品负责人 | Human Approver | 是 | S7 终审和 U6 UI-R3 签字由同一人担任 |
| Benchmark Analyst | （无对应） | 仅在 S1-S2 阶段存在 | S2 完成后退出 |
| （无对应） | Audit Owner | 仅在 U0-U9 阶段存在 | 可由产品负责人或指定审核协调人担任；但若产品负责人同时担任 Human Approver，则 Audit Owner 不得由同一人兼任——Audit Owner 产出"最终结论"，Human Approver 产出"最终放行"，两者必须分离 |

"分离"不是口头声明，必须有执行机制：

```text
人类场景：
  Designer / Critic / Verifier 必须由不同人担任。
  资源受限时 Critic 可由外部顾问或跨团队成员担任，但不得由 Designer 的上级兼任（上级倾向放行）。

AI 场景：
  Critic 和 Verifier 必须由不同 AI 模型实例或不同 session 执行。
  Critic 不能读取 Designer 的 rationale，只能读取原始证据和最终产物。
  若资源受限无法使用不同模型，必须由人类承担 Critic 角色或至少对 Critic 结论做抽样复核。
  H0 / H1 / H2 人类审批门禁必须确认 Critic 和 Designer 不是同一 AI 实例。
```

#### 3.2 小团队降级方案

| 团队规模 | 角色合并规则 | 约束 |
|---|---|---|
| 5 人以上 | 4 角色全部独立 | 无特殊约束 |
| 3-4 人 | 允许一人兼任 Benchmark Analyst + Verifier | Critic 必须独立 |
| 1-2 人 | 允许 Designer / Critic 合并 | 必须引入"延迟自审"：间隔至少 24 小时或切换 context 后再审；Verifier 完全自动化 |
| AI 单 agent | 禁止同一实例兼任 Designer + Critic | 必须用不同 session 或不同模型实例；人类对 Critic 结论做抽样复核 |

### 4. 流程状态机（S0-S7）

```text
S0 Brief Freeze
  Owner：产品负责人（人类）
  输入：用户目标、业务背景、平台约束
  输出：约束化 brief
  门禁：H0 人类批准

S1 Benchmark Scope
  Owner：Benchmark Analyst
  输出：benchmark shortlist、任务等价矩阵、排除理由
  门禁：A1 完整性检查；benchmark 改变问题定义时走 H1 人类批准

S2 Evidence Synthesis
  Owner：Benchmark Analyst
  输出：证据台账、任务模型、状态清单、模式 / 反模式库、密度基线、设计输入包
  门禁：A2 证据覆盖与溯源检查

S3 Concept Design
  Owner：Product Designer
  输出：IA、主任务流、布局原则、设计 rationale、风险清单、保守方案、差异化方案、设计决策日志
  门禁：H2 人类批准单向门决策

S4 Visual / Prototype Render
  Owner：Product Designer 或 Visual Implementer
  输出：真实界面 / 可运行原型、关键状态截图序列、交互录屏
  门禁：A3 渲染完整性检查 + 逐迭代即时视觉门禁

S5 Independent Critique
  Owner：Critic
  输出：benchmark defect / concept defect / implementation defect / scope-change request
  门禁：按缺陷类型自动路由

S6 Automated Verify
  Owner：Verifier
  输出：任务回放、状态覆盖、a11y、响应式、布局 / 数据密度、错误注入结果
  门禁：Pass 或回退

S7 Final Accept
  Owner：产品负责人（人类）
  输出：验收结论、已知风险、可回写知识
  门禁：高风险页面必须人类终审
```

#### 4.1 回退规则

```text
benchmark 错配 / 等价性不足       -> 回 S1
证据缺失 / 无法溯源              -> 回 S2
任务流 / IA / 优先级错误         -> 回 S3
视觉层级 / 响应式 / 状态渲染错误 -> 回 S4
Critic 发现实现缺陷              -> 回 S4（修实现）或 S5（重新独立评审）
Verifier 门禁失败                -> 回 S4（修渲染）或 S6（重新验证）
a11y 失败                        -> 回 S4
数据密度失败                     -> 回 S3（修 IA）或 S4（修渲染）
S7 终审不通过                    -> 回 S3（设计问题）或 S0（Brief 问题）/ 人类裁决
指标定义或门禁冲突               -> 回 S0 / 人类裁决
```

回退分为两类：

```text
根因回环（回 S0 / S1 / S2 / S3）：
  每次必须人类审批，不受 implementation loop 上限限制。
  第 2 次根因回环前必须提交"根因假设"文档，未通过则直接升级人工裁决。

实现回环（回 S4 / S5 / S6）：
  受 implementation loop 上限（3 次）限制。
```

#### 4.2 循环控制

```text
同一类 P0 连续 2 次复发，禁止继续自动循环，升级人工裁决
  "同类 P0"由 Critic 按根因分类（证据缺失类 / 状态缺失类 / 任务断裂类 / 布局崩溃类），
  不由 Designer 自填缺陷描述决定。
每个 concept 最多 2 次根因回环
每个 concept 下最多 3 次实现回环
每次打回必须附带：失败假设、责任归属、退出条件
Critic 和 Verifier 只能引用已批准 rubric，不能在循环中临时改标准
总耗时超过预算 2 倍时强制降级到轻量版（见第一部分第 15 节）
```

### 5. S0：约束化 Brief 门禁

启动前必须形成约束包。没有约束包，Designer 不得出稿。

#### 5.1 必填字段

```text
产品 / 模块名称
目标用户与角色权限
核心任务：1 个主任务 + 2 到 5 个次任务
业务目标：1 到 3 个可观察结果
非目标：本轮明确不做什么
数据对象与状态复杂度
权限 / 审计 / 合规约束
错误代价和风险等级
性能 / 延迟 / 设备 / 浏览器约束
设计系统 / 组件库约束
可访问性门槛
验证标准和 P0 触发条件
页面风险等级（核心 / 普通）— 由人类判定
设计令牌约束（spacing scale / 字号阶 / 颜色 token / 断点）
边界数据契约（最长文本长度 / 空值表现 / 特殊字符 / 最大记录数）
```

#### 5.2 禁止写法

```text
做一个高级、现代、简洁的页面
参考世界优秀软件
体验要超过竞品
界面要更专业
```

这些表述只能作为愿景，不得作为可执行 brief。

### 6. S1：Benchmark 准入门禁

Benchmark 不是"看起来优秀的软件"，而是"功能等价的任务证据来源"。

#### 6.1 任务等价准入表

每个候选 benchmark 必须填写：

| 字段 | 说明 | 阻断条件 |
|---|---|---|
| 用户角色 | 是否同类权限、同类操作责任 | 核心角色不同则无效 |
| 核心目标 | 是否解决同一类业务任务 | 目标不同则无效 |
| 数据对象 | 实体类型、关系复杂度是否相近（轻：实体≤3 / 中：4-10 / 重：>10） | 数据模型过轻则无效 |
| 操作集 | 以 S0 核心任务为锚，列出任务所需操作类型清单，覆盖度 = benchmark 支持的操作类型数 / 清单总数（分母由 S0 冻结后不得更改） | 核心操作覆盖不足 60% 则无效 |
| 状态复杂度 | 状态数量、异常态、权限态是否相近（低：≤5 / 中：6-15 / 高：>15） | 状态复杂度过低则无效 |
| 数据规模 | 数据量级、分页、筛选、并发使用量级 | 量级明显过低则降级参考 |
| 风险等级 | 错误后果、审计要求、恢复成本 | 风险等级不等价则需说明 |
| 平台约束 | Web / 移动 / 桌面 / 内嵌系统 | 平台差异影响交互时需降级 |
| 合规可采集性 | 是否可合法采集截图 / 录屏（见第一部分第 8 节） | 不可合法采集则不得作为证据来源 |
| 获取成本 | 账号类型 / 费用 / 等待时间 / 法律风险 | 成本超阈值时降级为公开截图 + 文档分析 |

准入判定由 Benchmark Analyst 提出数据、由 Critic 复核，不得由 Designer 自行判定。

#### 6.2 对照集组成

完整流程至少包含：

```text
1 个顶级样本：同类场景中的优秀标杆
1 个主流样本：行业常见成熟做法
1 个失败样本：明确识别不要学习什么
```

只选优秀样本容易导致 AI 学到"行业平均最优皮肤"，但看不到失败边界。

失败样本获取困难时，允许使用"已知的反模式文档"替代实际软件。

### 7. S2：证据采集与溯源门禁

每条证据必须绑定来源。没有来源锚点的判断不得进入评分。

#### 7.1 证据台账字段

```text
source_id
benchmark_name
product_version / observed_date
URL 或访问入口
screenshot_id / recording_id
screenshot_hash              # 截图文件哈希，用于防篡改（注意：此字段为 Part 1 证据台账专用；
                           # Part 2 审核阶段的 manifest.json 用 screenshots_sha256 数组
                           # 汇总 artifact 所有截图哈希，用于批量签名校验）
collected_by                 # 采集者（不得为 Designer）
verified_by                  # 核验者（必须非采集者）
viewport / device
操作步骤
原始观察
推断结论
置信度
备注
```

#### 7.2 必采证据

| 证据类型 | 内容 |
|---|---|
| 任务路径 | 起点、终点、步骤数、点击数、决策点、回退成本 |
| 状态矩阵 | 状态名、触发条件、视觉表现、可选动作、恢复路径 |
| 数据密度 | 首屏可见记录数、字段数、空白率、截断率、有效决策信息 |
| 交互机制 | 上下文操作、批量操作、快捷键、拖拽、inline edit、undo、auto-save |
| 错误恢复 | 校验失败、网络错误、权限不足、部分成功、重复提交、恢复重试 |
| 动态证据 | 关键路径录屏或逐状态截图序列 |
| 可访问性 | 键盘路径、focus 顺序、对比度、语义标签、错误提示可感知性 |

#### 7.3 证据分级

```text
直接证据：截图、录屏、可复现操作、可运行任务脚本
强推断：多条直接证据一致支持
弱推断：单一截图或有限观察推得
假设：没有直接证据，只能作为待验证方向
```

评分和 P0 判定只能基于直接证据和强推断。弱推断和假设可用于设计 rationale 和缺陷分析，但不得用于 P0 判定。

#### 7.4 证据真实性锚点

防止 AI 自行伪造证据链：

```text
1. 截图必须绑定 screenshot_hash，由 Verifier 校验哈希一致性。
2. 截图采集者（collected_by）不得为 Designer，核验者（verified_by）必须非采集者。
3. Verifier 随机抽取 10% 证据要求给出采集操作录屏，无法复现则整批证据降级为"假设"。
4. 明确禁止 AI 自生成 benchmark 截图，截图必须来自真实软件运行录屏。
5. 证据台账的 source_id 与 screenshot_id 必须双向可查，Critic 抽查时反查截图是否真实存在。
```

### 8. 证据采集合规边界

证据采集必须守住法律和道德底线。

#### 8.1 可采集与不可采集

| 类型 | 是否可采集 | 说明 |
|---|---|---|
| 公开帮助文档截图 | 可以 | 公开面向用户的文档 |
| 公开演示页截图 | 可以 | 无需登录即可访问的页面 |
| 注册试用后截图 | 可以 | 遵守试用协议，不超范围使用 |
| 录屏 | 需评估 | 部分软件 ToS 禁止录屏，需逐项确认 |
| 反编译 / 抓包 | 不可以 | 违反软件著作权和许可协议 |
| 内测泄漏数据 | 不可以 | 使用未经授权的泄漏信息 |
| 竞品员工内部信息 | 不可以 | 涉及商业秘密 |

#### 8.2 使用约束

```text
截图 / 录屏仅用于内部对比分析，不公开发布。
失败样本采集须匿名化处理，不指名道姓。
对有明确禁止截图 ToS 的软件，只做文字描述不截图。
知识库访问权限受控，仅限项目相关人员。
```

#### 8.3 数据隐私合规

证据采集过程中可能涉及用户数据，必须遵守数据隐私合规要求：

```text
1. 用户数据脱敏：
   - 采集截图 / 录屏中若包含真实用户数据（姓名、手机号、邮箱、地址等），必须在采集后立即脱敏处理
   - 脱敏方式包括：马赛克、模糊化、替换为测试数据等，确保无法还原原始数据
   - 不得以"内部使用"为由跳过脱敏步骤

2. 禁止用于 AI 训练：
   - 采集的竞品截图、录屏、用户数据严禁用于训练 AI 模型
   - 证据库的访问日志必须留存，确保可追溯是否有异常批量下载行为
   - 项目结束后，含有敏感数据的证据按数据保留策略定期清理

3. 跨境采集合规：
   - 若竞品为境外产品，采集前必须确认数据跨境传输的合规要求
   - 遵守《数据安全法》《个人信息保护法》及目标国的数据保护法规（如 GDPR）
   - 跨境采集的证据存储位置必须符合数据出境规定，不得将境外用户数据传输至未获批准的存储位置
```

### 9. S2 续：逆向抽象与设计输入包

Benchmark Analyst 必须把证据转成设计输入包，不得直接给"做成某某风格"的建议。

#### 9.1 固定产物

```text
任务分解树：用户目标 -> 任务 -> 操作步骤 -> 决策点
信息架构图：视图 -> 内容块 -> 字段 -> 关系
状态机：状态、触发事件、转换条件、副作用、恢复路径
交互模式库：模式、触发条件、反馈形式、正例、反例
模式 / 反模式库：什么值得学，什么必须避免
密度基线：有效决策信息密度，而不是简单字段数量
冲突裁决表：多个 benchmark 之间的取舍冲突和本项目优先级
```

以上产物统称"设计输入包"，进入 S3 Concept Design 作为 Designer 的输入。

#### 9.2 冲突裁决表

多个优秀产品之间必然存在取舍冲突。Designer 不得事后择优引用。

必须先裁决：

```text
效率优先 vs 可学性优先
高密度优先 vs 低认知负担优先
专家用户优先 vs 新手引导优先
审计可追踪优先 vs 操作速度优先
强约束流程优先 vs 灵活探索优先
```

裁决结果进入设计输入包。

### 10. S3：Concept Design 门禁

Designer 的目标不是"整体超过竞品"，而是：

```text
在已批准的核心任务上不低于 benchmark 下限；
在明确痛点上，有证据支持地超越 benchmark；
在非目标和无证据领域，不增加复杂度。
```

#### 10.1 必须输出两个方向

```text
保守方案：尽量贴近 benchmark 下限，保证不低于成熟做法
差异化方案：只在明确痛点上创新，写清牺牲什么、换来什么、如何验证净收益
```

#### 10.2 设计决策日志

每条关键设计决策必须包含：

```text
decision_id
决策内容
基准依据：source_id / screenshot_id / task_id
对应业务目标
反事实：如果不这样做，用户会遇到什么问题
取舍：保什么、牺牲什么
验证方式
不遵循 benchmark 的理由
```

禁止使用"业界最佳实践""更现代""更高级"替代证据。

### 11. S4：渲染与动态材料门禁

只看静态截图不允许通过视觉维度。

#### 11.1 必交材料

```text
关键屏静态截图
关键任务录屏或交互序列
状态切换截图序列
不同 viewport 截图
错误态与恢复路径截图
键盘流截图或记录
```

#### 11.2 逐迭代即时视觉门禁

复杂 UI 不允许在整页或整阶段完成后集中补做视觉评审。必须以"一个可独立验收的组件或工作区"为最小迭代单元，执行以下闭环：

```text
修改组件 / 工作区
  -> 立即在真实浏览器刷新
  -> 在批准的主 viewport 与紧凑 viewport 渲染
  -> 截取初始态、关键交互态、错误态 / 恢复态
  -> 专业前端 / UX 评审布局、密度、层级、裁切、重叠和可发现性
  -> Verifier 执行真实点击 / 拖拽与 DOM 状态断言
  -> Critic 独立检查"声明能力 > 实际实现"和"低于标杆"问题
  -> P0 阻断为 0 且 P1 降为 0 后才允许进入下一组件
```

P1 在逐组件迭代中为阻断条件（必须修复才能进入下一组件），但在全流程级不阻断（P1 不必回退到 S3 重新设计，只需在 S4 内修复）。

##### 11.2.1 修改分级与闭环强度

为避免每次微调都跑全量闭环，区分结构性修改和微调修改：

| 修改类型 | 触发条件 | 闭环强度 |
|---|---|---|
| 结构性修改 | 影响布局、尺寸、可见性、层级、定位、交互反馈 | 全闭环（7 步） |
| 微调修改 | 纯文案、纯颜色 token、纯间距 token（不改变布局结构） | 轻闭环：刷新 + 截图 + Verifier 自动断言（跳过人工 Critic） |

批量窗口：允许 30 分钟内的连续微调修改累积一次人工评审，而非逐次。但结构性修改不得使用批量窗口。

##### 11.2.2 硬规则

1. `HTML 已生成`、`data-testid 已存在`、`静态检查通过`均不能替代运行态视觉评审。
2. 结构性修改必须重新执行全闭环；微调修改可执行轻闭环。不得沿用修改前截图或结论。
3. 不得将多个复杂组件全部生成后再批量补审。批量补审发现的问题必须回退到最早引入问题的组件迭代。
4. 不同 viewport 必须使用同一任务和同一数据状态比较；至少覆盖目标产品的主桌面 viewport 和最窄受支持桌面 viewport。
5. 每个 viewport 必须自动检查关键元素的 `overflow`、`overlap`、离屏、遮挡、文字截断和滚动可达性；自动检查通过后仍需专业前端 / UX 视觉判断。
6. 点击命令返回成功不等于交互成功。必须在交互后断言可见结果、DOM 状态、错误码或 Schema 是否写入，防止离屏点击、选择器误命中和重复 `data-testid` 造成假阳性。
7. 与标杆比较时必须使用功能等价、状态等价、viewport 等价的截图或录屏；无法形成同条件证据时只能标记"证据不足"，不得宣称超过竞品。

##### 11.2.3 阻断条件

```text
未在真实浏览器渲染
缺少修改后的新截图
主 viewport 或最窄受支持 viewport 未覆盖
出现裁切、重叠、不可达滚动、错误定位或不可辨识状态
交互仅有按钮或文案而无真实状态变化
专业前端 / UX、Verifier 或 Critic 任一结论仍为阻断
```

#### 11.3 状态覆盖门禁

状态门禁不看"状态数量是否达标"，而看"是否覆盖高风险断点类别，并且每类至少 1 个状态且有可恢复路径"。

必须覆盖的风险断点类别：

```text
加载态类：loading / skeleton
空态类：empty / no-match filter
成功态类：success / partial success
校验态类：validation error
网络异常类：network error / timeout
权限态类：permission denied / disabled / read-only
数据态类：oversize / overflow / conflict
操作态类：draft / undo / recover / bulk action / destructive confirm / import / export
```

按优先级分层：

| 优先级 | 类别 | 首版要求 |
|---|---|---|
| P0 必做 | 加载态、空态、成功态、校验态、网络异常、权限态、destructive confirm | 必须有独立 UI + 恢复路径 |
| P1 次轮 | no-match、timeout、disabled、partial success、draft | 首版可降级为提示文案，但需进入 backlog |
| P2 后续 | undo、conflict、oversize、bulk、import / export | 首版可缺省，但需在风险清单中声明 |

轻量版流程只要求 P0 必做类别。

反 Goodhart 约束：状态数量本身不触发门禁，状态的可恢复路径完整性才触发门禁。如果状态数远超 P0 类别但多数无独立恢复路径，触发反向扣分。

### 12. S5：独立 Critic 门禁

Critic 是独立审计者，不是设计者的辩护律师。

#### 12.1 Critic 必须具备的权限

```text
查看 benchmark 原始来源
抽查 source_id 对应截图 / 录屏（反查 screenshot_hash 一致性）
独立重建部分关键任务路径
提出隐藏对抗场景
按已批准 rubric 判定 P0
触发二次审计
```

#### 12.2 Critic 输出格式

```text
缺陷类型：benchmark defect / evidence defect / concept defect / implementation defect / scope-change request
根因分类：证据缺失类 / 状态缺失类 / 任务断裂类 / 布局崩溃类
严重级别：P0 / P1 / P2
证据引用：source_id / screenshot_id / task_id
失败原因：哪个门禁未通过
影响：对任务、状态、错误恢复、可访问性或业务目标的影响
回退目标状态：S1 / S2 / S3 / S4 / S5 / S6 / H 人工裁决
退出条件：修到什么程度才允许复评
```

Designer 不能推翻 Critic，但可以触发二次审计。二次审计只基于原始证据，不接受话术辩论。

### 13. S6：自动化 Verifier 门禁

Verifier 负责硬门禁，不负责审美结论。

#### 13.1 可自动验证项

| 门禁 | 验证方式 | 通过标准 | 失败动作 |
|---|---|---|---|
| 任务回放 | Playwright 跑核心任务脚本 | 关键路径 100% 可达，无 dead-end | 回 S3（任务流问题）或 S4（渲染问题） |
| 状态覆盖 | 状态矩阵逐项渲染 | P0 类别状态均有独立 UI 且有可恢复路径 | 回 S4 |
| 错误恢复 | 网络拦截 / 故障注入 | 每类错误有重试、取消、返回或修正路径 | 回 S3 / S4 |
| 可访问性 | axe-core + 键盘遍历 | 无 critical / serious 问题，关键任务键盘可完成 | 回 S4 |
| 视觉纪律 | 截图 diff、overflow、overlap、token 审计 | 无关键越界、遮挡、未授权 token | 回 S4 |
| 数据密度 | 统计信息承载率、截断率、空白率 | 位于目标区间，且信息有任务必要性 | 回 S3 / S4 |
| 证据完整性 | manifest 校验 + screenshot_hash 校验 | 每个结论能追溯到 source_id，截图哈希一致 | 回 S2 |
| 证据真实性 | 随机抽取 10% 证据要求采集录屏复现 | 无法复现则整批降级 | 回 S2 |

#### 13.2 人工验证项

```text
真实用户效率
心智模型匹配
品牌调性
视觉美感
错误文案是否足够体面
屏幕阅读器真实体验
是否值得为差异化方案承担复杂度
```

自动化只能证明"不明显坏"，不能单独证明"真的好"。

#### 13.3 工具链最小集

| 阶段 | 工具 | 用途 | 必须 |
|---|---|---|---|
| MVP | Playwright | 任务回放、DOM 断言、截图 | 是 |
| MVP | @axe-core/playwright | a11y 自动扫描 | 是 |
| MVP | 手动截图对照 | viewport 视觉检查 | 是 |
| 进阶 | Percy / Chromatic | 截图 diff 自动化 | 否 |
| 进阶 | Storybook | 组件隔离渲染 | 否 |
| 进阶 | Mock Service Worker | 故障注入 | 否 |
| 完整 | 自定义 manifest 校验脚本 | 证据完整性 | 否 |

无工具降级策略：

```text
无 Playwright：退回人工任务回放（按任务清单逐步操作并截图）
无 axe-core：退回键盘遍历 checklist（Tab 顺序、focus 可见、对比度手动检查）
无截图 diff：退回 before/after 手动对照
```

### 14. S7：Final Accept 门禁

S6 通过后进入 S7 最终验收。

#### 14.1 验收内容

```text
S5 Critic 缺陷单全部关闭或降级
S6 Verifier 门禁全部通过
P0 = 0，P1 = 0（核心页面）或 P1 ≤ 阈值（普通页面）
已知风险清单已记录
可回写知识已整理（标杆模式、反模式、本项目的 ADR）
```

#### 14.2 验收门禁

```text
核心页面：必须人类终审，产品负责人签字
普通页面：Critic + Verifier 双通过即可，人类抽审
高风险页面（涉及金额 / 权限 / 合规）：必须人类终审 + 合规复核
```

#### 14.3 知识回写

验收通过后，以下内容回写到知识库：

```text
标杆设计模式（值得学的）
反模式（必须避免的）
本项目的设计决策和取舍理由
验证中发现的真实失败模式
状态覆盖清单和恢复路径设计
```

#### 14.4 设计决策回溯

知识回写不仅记录"成功的决策"，还必须建立设计决策回溯机制，以便在后续阶段发现设计决策错误时进行追溯标记。

```text
回溯触发条件：
  - 后续阶段（S5 Critic 评审 / S6 Verifier 验证 / 第 25 节生成 / U0-U9 审核）
    发现设计决策本身存在错误或缺陷
  - 回退根因属于"概念错误"或"方向错误"而非"实现缺陷"

回溯操作：
  1. 在 S3 设计决策日志中找到对应的 decision_id
  2. 在该 decision_id 下追加标记："决策失效"
  3. 记录失效原因：哪个后续阶段发现了什么问题
  4. 记录发现阶段、发现时间、发现人/Agent
  5. 如果该决策已被知识回写收录，在知识库中同步标记"决策失效"

回溯后的处理：
  - "决策失效"标记的决策不得被后续项目作为参考
  - 失效决策及其失效原因一并回写到知识库的"反模式"分类
  - 若同类型决策多次失效，触发设计方法论审查
```

设计决策回溯与第 4.1 节回退规则的关系：回退规则处理"修复后重新执行"，设计决策回溯处理"标记历史决策的问题以便知识库更新"。两者并行执行。

### 15. 轻量版流程

普通页面或低风险 UI 不执行完整流程，使用轻量版。

#### 15.1 轻量版流程

```text
L0：明确主用户、主任务、非目标、页面风险等级
L1：选 1 个功能等价 benchmark + 1 个失败样本
L2：采集 1 条核心任务链 + 1 组 P0 类别边界状态链
L3：生成设计并渲染 P0 类别关键状态
L4：Critic 抽查证据并给 P0 / P1 / P2
L5：执行任务回放、状态覆盖、a11y、布局检查
L6：验收（普通页面 Critic + Verifier 双通过即可）
```

轻量版 L4（Critic）在 L5（Verifier）之前，与完整流程 S5→S6 顺序一致。

#### 15.2 轻量版保留与裁剪

```text
保留：
  - benchmark 准入（L1，缩减为 2 个样本）
  - 证据溯源（L2，缩减为 1 条任务链）
  - Critic 独立抽查（L4，至少抽 1 项）
  - P0 类别状态覆盖（L2/L3）
  - 反 Goodhart 检查（不得完全砍掉）

裁剪：
  - 冲突裁决表（单 benchmark 无需）
  - 逐迭代全闭环（改为里程碑检查点）
  - 双方案（保守 / 差异化）——只要求 1 个方案
  - S7 人类终审（普通页面免，高风险页面仍需）
```

#### 15.3 超轻量版（1 天内完成）

低风险页面可进一步降级：

```text
L0-Lite：主任务 + 非目标
L1-Lite：1 个参考截图（无需注册操作）
L2-Lite：1 条主任务路径截图
L3-Lite：渲染 P0 类别 3 个状态
L4-Lite：仅 a11y 自动扫描（无任务回放）
L5-Lite：自查 P0 清单（无独立 Critic）
```

超轻量版适用条件：不涉及金额 / 权限 / 数据修改 / 批量操作 / 多步流程的纯展示页，且由人类在 L0 确认。

超轻量版视觉审核：超轻量版页面无需执行第二部分完整视觉审核（U0-U9），但必须执行第二部分 4.1 节轻量版审核的进一步降级——仅要求 G2（运行完整性）+ G8 a11y critical 检查 + 人工目视确认。

### 16. P0 / P1 / P2 分级

#### 16.1 P0：阻断级失败

```text
benchmark 未通过任务等价准入
核心证据无法溯源
证据真实性校验失败（screenshot_hash 不一致 / 无法复现采集）
核心任务无法回放
关键状态缺失（P0 类别状态无独立 UI 或无恢复路径）
错误态无恢复路径
权限 / 只读 / 禁用状态不明确
a11y critical / serious 问题阻断关键任务
关键布局越界、遮挡、文字不可读
Critic 发现 Designer 引用未采集证据或伪造证据
指标被刷分且无法说明任务价值
```

#### 16.2 P1：必须修复但不必重走全流程

```text
局部状态表现不清晰
部分错误文案不可操作
低频任务路径偏长
数据密度偏离目标区间但未阻断主任务
响应式局部问题
部分 token 偏差
```

P1 在逐组件迭代中为阻断条件（必须修复才能进入下一组件），但在全流程级不阻断（不必回退到 S3 重新设计，只需在 S4 内修复）。

#### 16.3 P2：可进入后续迭代

```text
轻微视觉一致性问题
非关键文案优化
低风险微交互优化
非核心状态的提示增强
```

### 17. 反 Goodhart 规则

以下指标不得直接代表质量：

```text
字段数
行数
截图数量
artifact 数量
布局比例
状态数量
组件数量
打分总分
```

每个指标必须绑定任务价值：

```text
这个信息是否在当前任务时刻必须可见？
这个操作是否减少真实步骤或错误成本？
这个状态是否对应真实风险断点？
这个视觉层级是否帮助用户做正确决策？
这个自动化分数是否能被原始结果复算？
```

无法证明任务价值的指标不计分；为了刷指标而增加复杂度时，反向扣分。

反 Goodhart 规则优先级高于状态覆盖门禁：状态数量本身不触发门禁，状态的可恢复路径完整性才触发门禁。

### 18. AI 视觉缺陷防御体系

AI 生成 UI 的视觉缺陷是系统性问题，需要四层防御。

#### 18.1 第一层：生成前约束

在 AI 生成代码前注入结构化约束，从源头减少缺陷：

```text
设计令牌锁：
  - 明确 spacing scale（如 4/8/12/16/24/32/48px，禁止任意值）
  - 明确字号阶（如 12/14/16/20/24/32px）
  - 明确容器最大宽度和断点
  - AI 只能引用 token，不得硬编码数值

布局模式锁：
  - 预定义 approved 布局模式（flex column / grid / sidebar+main）
  - AI 必须声明使用的布局模式，不得自创布局
  - 复杂区域必须用已有组件，不得手写

数据契约锁：
  - 提供真实数据样本（最长文本、最大数字、空值、特殊字符）
  - AI 必须针对每种数据样本验证布局不崩溃
  - 禁止用 Lorem Ipsum 或空字符串测试

状态清单锁：
  - 生成前先列出必须覆盖的状态
  - AI 必须逐状态生成，不得只生成初始态
```

以上约束在 S0 Brief 中冻结，Designer 不得自行变更。

#### 18.2 第二层：生成中拆分

禁止一次性生成整页，必须按原子组件 → 独立验证 → 组合验证 → 整页验证：

```text
步骤1：生成单个组件
  → 立即渲染
  → 用真实数据填充
  → 截图初始态
  → AI 自检：有没有溢出？有没有重叠？文本是否完整？

步骤2：注入边界数据
  → 最长文本（200 字符的名称）
  → 空值
  → 特殊字符（emoji、超长 URL、换行符）
  → 最大数量（100 条记录）
  → 截图边界态
  → AI 自检：布局是否崩溃？

步骤3：注入状态变化
  → loading（骨架屏或 spinner）
  → empty（空状态引导）
  → error（错误提示 + 恢复操作）
  → 截图每个状态
  → AI 自检：每个状态是否可辨识？是否有恢复路径？

步骤4：组合到页面
  → 多组件组合
  → 检查组件间间距、对齐、层级
  → 截图组合态

步骤5：多 viewport 验证
  → 主桌面（1920px）
  → 标准桌面（1440px）
  → 最窄受支持（1024px 或 768px）
  → 逐 viewport 截图
  → AI 自检：每个 viewport 下是否有溢出、重叠、不可达？
```

#### 18.3 第三层：生成后自动验证

AI 生成后，用工具自动检测视觉缺陷：

```text
1. 溢出检测
   - 遍历所有元素，检查 scrollWidth > clientWidth 或 scrollHeight > clientHeight
   - 检查元素是否超出父容器边界（getBoundingClientRect）
   - 输出溢出元素列表和位置

2. 重叠检测
   - 遍历所有可见元素，检查 bounding rect 是否相交
   - 排除故意重叠（如 dropdown、tooltip、modal）
   - 输出意外重叠的元素对

3. 不可达检测
   - 检查所有可交互元素是否在 viewport 内或可滚动到达
   - 检查 tab 顺序是否合理
   - 检查是否有元素被 z-index 遮挡且无法交互

4. 文本截断检测
   - 检查所有文本节点是否被 CSS 截断（text-overflow: ellipsis）
   - 区分"故意截断"（如表格单元格）和"意外截断"（如按钮文字被切）
   - 用真实最长数据测试

5. 对比度检测
   - axe-core 自动检测文字与背景对比度
   - 检查所有状态下的对比度（包括 disabled、hover）

6. 布局一致性检测
   - 检查同层级元素的间距是否一致
   - 检查对齐基线是否一致
   - 检查是否使用了未授权的间距值

7. 状态完整性检测
   - 对照状态清单，检查每个状态是否有对应的 UI
   - 检查状态切换是否有视觉反馈
   - 检查错误态是否有恢复操作
```

#### 18.4 第四层：AI 视觉自审

AI 完成生成后，用自审 prompt 重新审视自己的产出：

```text
你是视觉缺陷审查员。请对刚才生成的 UI 逐一回答以下问题，
任何一项回答"不确定"或"是"都必须修复后重新提交：

1. 当文本字段达到 S0 边界数据契约定义的最大长度时，该列/该区域是否会溢出或截断？
2. 当列表为空时，页面是否显示了有意义的空状态引导？
3. 当数据加载中时，用户是否能看到明确的加载指示？
4. 当网络请求失败时，用户是否有重试操作可用？
5. 当用户在 1024px 宽度下查看时，是否有元素溢出或重叠？
6. 当用户用键盘 Tab 导航时，是否能到达所有可交互元素？
7. 当弹窗打开时，背景内容是否被正确遮罩且不可交互？
8. 当表单校验失败时，错误提示是否在相关字段旁可见？
9. 当文本内容超长时，是否使用了受控截断（tooltip 展示全文）而非意外截断？
10. 当页面有多个操作按钮时，主操作是否在视觉上突出于次操作？
11. disabled 状态的元素是否在视觉上明确区别于 enabled？
12. 当滚动时，是否有固定元素（如表头、操作栏）保持可见？
13. 所有间距是否来自设计令牌，而非任意值？
14. 所有颜色是否来自设计令牌，而非硬编码？
15. 你是否有任何状态没有生成对应 UI？
```

### 18.5 AI 生成幻觉与证据伪造的识别与防御

AI 在 UI 设计过程中会产生特定类型的幻觉，这些幻觉与普通代码 bug 不同，需要专门的识别和防御机制。

#### 18.5.1 幻觉的七种典型形态

```text
1. "截图幻觉"：AI 描述了一个实际不存在的 UI 状态或交互，
   并给出看似合理的位置、大小、颜色数据，但这些数据无法在代码中找到对应实现。
   识别特征：描述的具体数值（如"第3个按钮距左边 247px"）在代码中没有精确对应。

2. "能力幻觉"：AI 声称某个组件"已经支持"某个功能，
   但代码中缺少关键的事件监听、状态更新或 DOM 操作。
   识别特征：功能描述中有"可以""会""支持"等词汇，但代码中没有对应逻辑。

3. "状态幻觉"：AI 说"当数据为空时显示空状态"，但没有生成空状态 UI，
   或状态存在但没有真实触发条件（只是注释或 TODO）。
   识别特征：状态存在文档/注释中，但 render 方法中没有对应条件分支。

4. "Benchmark 幻觉"：在 S1/S2 阶段，AI 编造截图证据或任务路径描述，
   声称来自真实软件，但这些"证据"无法与实际产品对应。
   识别特征：证据描述过于流畅和完整，缺乏"需要实际操作才能确认"的不确定性。

5. "Token 幻觉"：AI 使用了一个不存在的 design token 名称
   （如 --color-primary-darker），该 token 在实际的 token 文件中不存在。
   识别特征：代码中使用了 token，但实际 CSS 变量文件中没有定义。

6. "状态覆盖幻觉"：AI 在 findings 中声称"所有 P0 状态已覆盖"，
   但实际只生成了 initial/hover/active 等基础状态，缺少空态、错误态、加载态。
   识别特征：状态数量充足但缺少高风险断点类别的真实 UI。

7. "交互成功幻觉"：AI 说"用户可以保存表单"，
   但只实现了 form submit 事件，没有检查数据校验结果，没有处理网络错误，
   没有断言 success 状态是否真实可见。
   识别特征：命令/API 返回 200 即认为交互成功，但 DOM 中没有对应变化。
```

#### 18.5.2 防御策略

针对每种幻觉形态，对应防御策略：

```text
截图幻觉防御：
  → Verifier 必须对 AI 描述的数值进行 DOM 查询验证
  → AI 的任何具体描述（"按钮在 x=200px"）必须在代码或 DOM 中有精确对应
  → 禁止 AI 用模糊描述（"按钮在合适位置"）替代精确描述

能力幻觉防御：
  → 交互验证必须检查可见结果，不只是"事件被触发"
  → 断言 DOM 状态变化、aria 属性变化、可见反馈出现
  → 禁止仅凭"事件监听器存在"判断功能可用

状态幻觉防御：
  → 状态必须同时满足：有 UI 代码 + 有触发条件 + 可被 Verifier 真实触发
  → 三者缺一即不算覆盖

Benchmark 幻觉防御：
  → 所有证据必须绑定 screenshot_hash（SHA-256）
  → 采集者不得为 Designer，核验者必须非采集者
  → Verifier 随机抽取 10% 证据要求采集录屏复现
  → AI 不得自生成 benchmark 截图，必须来自真实软件运行录屏

Token 幻觉防御：
  → 生成前必须读取实际的 token 文件，列出可用 token
  → 生成后必须运行 token 审计脚本，检查是否存在未定义 token
  → 禁止 AI 自创 token 名称

状态覆盖幻觉防御：
  → 状态覆盖矩阵必须由 Verifier 逐项验证，不接受 Designer 自填
  → 每个 P0 类别必须有真实 UI + 可恢复路径，缺一不算覆盖

交互成功幻觉防御：
  → 每次交互后必须断言：可见结果 AND DOM 状态 AND 错误码
  → 禁止仅凭 API 返回成功判断交互成功
```

#### 18.5.3 跨 Agent 协作中的幻觉传播

在多 Agent 协作场景中，幻觉可能在 Agent 之间传播并放大：

```text
Designer Agent 产生能力幻觉（声称某功能已实现）
  → Generator Agent 接受 Designer 输出，继承幻觉
  → Verifier Agent 若只检查表面，容易被幻觉带偏

防御链：
  每个 Agent 在接受上游输出时，必须独立验证：
  1. 关键数据是否有源码对应（不是摘要）
  2. 关键能力是否可在真实浏览器中触发
  3. 关键状态是否有完整实现（三者：UI + 触发 + 恢复路径）
  4. 关键引用（截图/token/benchmark）是否可独立验证
```

#### 18.5.4 幻觉防御与门禁检查项的关联

18.5.1 节的七种幻觉防御不是独立内容，已嵌入以下门禁检查项：

| 幻觉形态 | 嵌入的门禁位置 | 检查方式 |
|---|---|---|
| 截图幻觉 | S6 Verifier 视觉纪律门禁（第一部分第 13.1 节）+ U4 G4 几何门禁（第二部分第 12 节） | DOM 查询验证 AI 描述的数值 |
| 能力幻觉 | S6 任务回放门禁（第一部分第 13.1 节）+ U4 G3 核心任务门禁（第二部分第 12 节） | 交互后断言可见结果 + DOM 状态变化 |
| 状态幻觉 | S6 状态覆盖门禁（第一部分第 13.1 节）+ U4 G3 状态门禁 | 状态必须同时有 UI + 触发条件 + 可被真实触发 |
| Benchmark 幻觉 | S2 证据真实性锚点（第一部分第 7.4 节）+ S6 证据完整性门禁 | screenshot_hash 校验 + 采集者隔离 + 随机复现 |
| Token 幻觉 | S6 视觉纪律门禁 token 审计（第一部分第 13.1 节）+ U4 G5 Token 门禁（第二部分第 12 节） | 生成前读取 token 文件 + 生成后 token 审计脚本 |
| 状态覆盖幻觉 | S6 状态覆盖门禁（第一部分第 13.1 节）+ U4 G3 状态门禁 | Verifier 逐项验证，不接受 Designer 自填 |
| 交互成功幻觉 | S6 任务回放门禁（第一部分第 13.1 节）+ U4 G3 核心任务门禁 + 第 25.2.1 节组件自检 | 交互后断言可见结果 AND DOM 状态 AND 错误码 |

#### 18.5.5 幻觉复发监控

幻觉防御不仅是单次检测，还需要监控同类幻觉是否反复出现。当同类幻觉在同一项目或同一 AI 模型实例中出现 2 次以上时，说明防御机制本身可能存在系统性缺陷，需要触发 AI 配置审查。

```text
监控规则：
  1. 按 18.5.1 节七种幻觉形态分类记录每次幻觉事件
  2. 同类幻觉在同一项目周期内出现 2 次以上 → 触发 AI 配置审查
  3. AI 配置审查范围：
     - 检查生成 prompt 是否缺少关键约束（如未注入 design token 文件路径）
     - 检查 system prompt 是否被不当修改或漂移
     - 检查是否使用了不适当的 AI 模型版本（如降级为不支持某些能力的版本）
     - 检查上下文窗口是否因过长导致关键约束被截断
  4. 审查结果必须记录并存档，修正后重新执行该幻觉类型对应的门禁验证
  5. 连续 3 次以上同类幻觉复发 → 上报人工裁决，考虑更换 AI 模型或调整流程
```

幻觉复发监控与第 4.2 节循环控制的关系：循环控制关注"同一类 P0 连续 2 次复发"（按根因分类），幻觉复发监控关注"同一幻觉形态出现 2 次以上"（按幻觉类型分类）。两者维度不同，互为补充。

```text

本流程配合《05-研发流程规范.md》使用——05 提供 P2-P5 的 11 个工程活动（含 L1-L4 复杂度分级），本规范作为 UI/交互专项子流程，提供 S0-S7 设计门禁与第二部分审核流程。

- 核心任务（1 主 + 2-5 次）
- 非目标
- 业务结果指标
- 权限 / 审计 / 性能 / 平台约束

然后进入 S1 Benchmark Scope：

- 准入判定由 Benchmark Analyst 提出数据、由 Critic 复核
- 未通过等价准入的 benchmark 不得用于评分
- 标注每个 benchmark 的获取成本和合规可采集性

进入 S2 Evidence Synthesis：
- 每条结论必须绑定 source_id / screenshot_id / recording_id / task_id
- 截图必须绑定 screenshot_hash，采集者不得为 Designer
- 没有来源锚点的判断只能标为假设，不得进入评分
- 评分和 P0 判定只能基于直接证据和强推断
- Verifier 会随机抽取 10% 证据要求采集录屏复现

- Designer 只能提交方案和依据，不能自评通过
- 必须输出保守方案和差异化方案
- 差异化方案必须说明牺牲什么、换来什么、如何验证净收益
- 每条设计决策必须记录 decision_id 和基准依据

进入 S4 Visual / Prototype Render：
- 必须逐组件生成并在真实浏览器渲染，禁止一次性生成整页
- 每个组件必须用真实数据和边界数据验证
- 结构性修改触发全闭环，微调修改可用轻闭环
- 必须覆盖 P0 类别状态（加载态 / 空态 / 成功态 / 校验态 / 网络异常 / 权限态 / destructive confirm）
- 每个 viewport 必须检查 overflow / overlap / 离屏 / 遮挡 / 截断
- 交互后必须断言可见结果和 DOM 状态，不能只看点击返回成功

- Critic 独立抽查原始证据，反查 screenshot_hash
- 按根因分类判定 P0
- Designer 不能推翻 Critic，只能触发二次审计

进入 S6 Automated Verify：
- Verifier 执行任务回放、状态覆盖、错误恢复、a11y、截图布局检查、证据完整性校验
- 任一 P0 触发，按缺陷类型回退到最近责任状态
- 回退分为根因回环（S0-S3，需人类审批）和实现回环（S4-S6，受 3 次上限限制）
- 同类 P0 连续两次复发（由 Critic 按根因分类），停止自动循环，升级人工裁决
- 每个 concept 最多 2 次根因回环、3 次实现回环
- 总耗时超预算 2 倍强制降级到轻量版

进入 S7 Final Accept：
- 核心页面必须人类终审
- 验收通过后回写知识库

S7 通过后进入第 25 节 AI 原型生成 SOP：
- 基于 S7 批准的设计方案生成最终 UI 代码
- 逐组件生成-检查-修复，边生成边自检
- 生成完成后计算 artifact_hash 并提交独立角色二次验证

第 25 节完成后进入 U0-U9 视觉审核：
- U0 确定审核模式、风险等级和通过标准
- U1 冻结 artifact_hash 和环境
- U2-U5 执行审核矩阵、运行态采证、自动门禁和独立视觉审核
- U6 输出 PASS / FAIL / EVIDENCE_INSUFFICIENT
- UI-R3 页面必须 Human Approver 签署
- FAIL 时进入 FIX_MODE 修复（最多 2 轮/会话，3 次/全局）

全程遵守反 Goodhart 规则（第一部分第 17 节）：
- 状态数量不直接代表质量，可恢复路径完整性才代表质量
- 无法证明任务价值的指标不计分
- 为刷指标而增加复杂度时反向扣分

全程遵守合规边界（第一部分第 8 节）：
- 只采集公开资料，不反编译、不入侵、不使用泄漏数据
- 截图 / 录屏仅用于内部对比，不公开发布
```

### 20.5 全流程三审分立与跨阶段衔接

本节说明设计门禁（Part 1）、AI 生成（SOP 第 25 节）和视觉审核（Part 2）三个阶段如何通过角色分离和证据传递形成完整闭环。

#### 20.5.1 三层角色分离体系

三套流程共用了同一套角色模型，但每层的职责边界不同：

| 角色 | Part 1 设计门禁中的职责 | 第 25 节生成 SOP 中的职责 | Part 2 视觉审核中的职责 |
|---|---|---|---|
| Product Designer / Generator | 基于证据包设计，输出保守+差异化双方案 | 按第 25 节生成每个组件，边生成边自检自修 | **禁止参与**：审核阶段已切换为 Submitter 角色 |
| Critic / UX Critic | 独立红队，按根因分类判定 P0，可触发二次审计 | 生成者自检（生成阶段内控，非独立 Critic） | 独立审核信息架构、密度、层级、标杆差距 |
| Verifier / Visual Verifier | 执行自动化门禁（任务回放、状态覆盖、a11y、布局） | 生成者执行自动检查（溢出、重叠、对比度、Token） | 执行 G1-G8 硬门禁 |

**关键约束**：同一 AI 实例不得在多个阶段担任需要"独立"的角色。具体限制：
```text
Designer session ≠ Critic session（无论在哪个流程中）
Designer session ≠ Verifier final judgment
第 25 节生成者自检 ≠ Part 2 的 UX Critic
```

#### 20.5.2 跨阶段证据传递

每个阶段完成后，必须向下一阶段传递以下证据：

```
Part 1 S7 验收完成 → 交付：
  ├── S0 冻结的 brief（约束包）
  ├── S2 的证据台账（source_id → screenshot_id 双向可查）
  ├── S3 的设计输入包（任务分解树 / IA / 状态机 / 模式库 / 冲突裁决表）
  ├── S3 的双方案（保守方案 + 差异化方案）含 decision_id
  ├── S4 的渲染材料（关键屏截图 / 状态序列 / 交互录屏）
  ├── S6 Verifier 自动化验证报告
  ├── artifact_hash（设计稿或原型版本）
  └── 已知风险清单

**artifact_hash 跨阶段传递规则**：
  S4 产出设计原型时计算并记录 artifact_hash（设计稿/原型截图的哈希）。
  S7 验收时传递该 artifact_hash 作为设计基线（design baseline），S7 本身不产生新的 artifact_hash。
  第 25 节生成阶段会产生新的 artifact_hash（最终 UI 代码/构建产物的哈希）。
  U1 冻结的是第 25 节生成的 artifact_hash，不是 S4/S7 的。
  两个哈希的关系：S4/S7 的 artifact_hash 作为设计基线（design baseline），
  第 25 节的 artifact_hash 作为审核对象（audit target）。
  如果 S7 后到第 25 节前设计文件被修改，必须回 S7 重新验收。

第 25 节生成完成 → 交付：
  ├── artifact_hash（生成产物的哈希）
  ├── 每个组件的检查点记录（25.5 节）
  ├── 每个组件的修复日志（问题→修复→验证）
  ├── 全局 token 使用声明（无硬编码的声明）
  └── 未修复的问题清单（如有）+ 二次验证报告

Part 2 视觉审核 U6 结论 → 交付：
  ├── manifest.json（含 verdict、p0/p1/p2、screenshots_sha256）
  ├── findings.json（完整 issue 清单 + 复验退出条件）
  ├── coverage-matrix.json（审核矩阵完成度）
  ├── diff 热力图 + 截图包
  ├── Human Approver 签署（如 UI-R3）
  └── U7 归档确认
```

#### 20.5.3 阶段衔接的硬约束

**S4 与第 25 节生成 SOP 的关系**：

S4（Visual / Prototype Render）和第 25 节（AI 原型生成 SOP）是两个不同阶段，执行顺序为 S0-S7 → 第 25 节 → U0-U9：

```text
S4 渲染的是设计原型：
  - 目的是验证设计概念的可视性（信息架构、布局原则、关键状态是否可渲染）
  - 产物是设计原型截图序列、交互录屏
  - 逐迭代视觉门禁（第一部分第 11.2 节）在此阶段执行
  - S4 通过后进入 S5/S6/S7 完成设计门禁

第 25 节生成的是最终 UI 实现：
  - 目的是基于已批准的设计方案（S7 交付物），生成可交付的 UI 代码
  - 产物是 HTML / 组件代码 + artifact_hash
  - 生成 SOP 的"边生成-边检查-边修复"循环（第二部分第 25.2 节）在此阶段执行
  - 第 25 节完成后进入 U0-U9 视觉审核

简言之：S4 验证"设计能不能看"，第 25 节生成"最终可交付的 UI"，U0-U9 审核"最终 UI 有没有缺陷"。
```

```text
S7 未通过，不得进入第 25 节生成
  → S7 终审未通过意味着设计本身有问题，继续生成只会放大问题

第 25 节未完成（artifact_hash 未记录），不得进入 U0-U9 审核
  → 没有版本哈希意味着没有可复现的审核对象

U6 结论非 PASS，不得进入研发下一阶段
  → 上游研发流程规范在对应阶段强制调用本规范的门禁

FIX_MODE 累计 3 次 FAIL（含同 artifact_hash 下），必须人工裁决
  → 人工裁决范围：继续修复 / 调整范围 / 废弃原型；无论哪种决策必须书面记录
```

#### 20.5.4 多 Agent 协作中的防串话机制

当使用 Claude Code、Codex 或类似多 Agent 系统时，以下是必须遵守的防串话规则：

```text
1. 证据隔离：Designer/Generator 的 session 不得读取 Critic 的 session history
   → 证据只能通过 manifest.json、findings.json 等文件传递，不得通过共享 context 传递

2. 哈希作为版本契约：
   → artifact_hash 是跨 Agent 的唯一版本标识
   → 任何 Agent 修改文件后必须重新计算哈希并通知其他 Agent
   → 基于旧哈希的结论在哈希变化后立即失效

3. Critic 独立性：
   → Critic 必须以独立 system prompt 启动，不继承 Designer 的 context
   → Critic 只能读取：原始证据（截图/录屏）、设计稿/原型文件、已批准 rubrics
   → Critic 不得读取 Designer 的设计 rationale，只能读取决策日志中的 source_id

4. 修复者的证据来源：
   → FIX_MODE 修复者只能基于 findings.json 开始修复
   → 不得基于"与 Designer 的对话"来理解问题，必须基于可执行的问题描述
```

## 第二部分：视觉审核规范与 AI 原型生成 SOP

### 1. 规范目标

本规范解决以下问题：

```text
AI 画完界面即声称完成
把 data-testid 存在误当成功能可用
只检查一张桌面截图，不检查窄屏、状态和交互
只做全页像素差，关键区域错误被平均值掩盖
把像素一致误当成设计优秀
截图、原型、交互流、状态图和测试合同彼此漂移
审核者一边评审一边修改，导致证据版本失效
生成者自己调整阈值、扩大遮罩并自评通过
原型不如竞品但无证据、无基准参照
有竞品参照但无法证明超越还是低于
```

本规范优先保证：

```text
原型比竞品强的地方有真实任务证据（不是主观感受）
原型不低于竞品下限（没有明显低于行业成熟做法的设计缺陷）
审核对象版本唯一
视觉证据可复现
核心任务真实可达
关键状态真实可触发
像素和几何差异可量化
视觉质量由独立角色裁决
任何通过结论均可追溯到原始证据
```

### 2. 规范用语

本文使用以下约束词：

| 用语 | 含义 |
|---|---|
| 必须 / MUST | 不满足即阻断，不得放行 |
| 禁止 / MUST NOT | 违反即判定审核无效或失败 |
| 应 / SHOULD | 原则上必须执行；偏离时需记录理由和批准人 |
| 可 / MAY | 按项目风险和成本选择执行 |

关键术语：

| 术语 | 定义 |
|---|---|
| 被审核对象 | 本轮固定版本的设计稿、HTML 原型、前端页面或应用构建产物 |
| 视觉基线 | 经批准、带版本哈希的参考设计稿或参考截图 |
| 任务基线 | 目标用户、核心任务、操作步骤、成功结果和恢复路径 |
| 关键区域 ROI | 主操作、核心表单、关键数据、错误反馈、权限提示等不得被全页平均值稀释的区域 |
| 稳定区域 | 在冻结数据和环境下，内容和布局应完全稳定的截图区域 |
| 动态区域 | 时间、随机数、动画、实时数据等经批准后允许精确遮罩的区域 |
| 视觉合同 | 对视口、状态、DOMRect、设计 Token、截图、交互结果和恢复路径的可执行约束 |
| 像素级通过 | 实现相对已批准视觉基线，在冻结环境下满足几何、Token、ROI 和截图差异阈值 |
| 门禁 | 质量检查关卡，上一阶段未通过门禁不得进入下一阶段；本规范中门禁包括设计阶段门禁（S0-S7 中的 H0/H1/H2、A1/A2/A3）和审核阶段门禁（G1-G8 硬门禁、U6 结论门禁） |

### 3. 审核模式

UI 审核必须先声明模式。三种模式不能互相替代。

#### 3.1 设计质量审核 `DESIGN_QUALITY`

适用于尚未形成批准视觉基线的新页面、新组件或新原型。

审核目标：

```text
信息架构是否正确
视觉层级是否清楚
密度和可读性是否适合目标用户
核心任务和状态是否完整
是否达到功能等价标杆下限
```

此模式可以审核尺寸、间距、字体、颜色和一致性，但禁止输出"像素级还原通过"。

#### 3.2 实现还原审核 `FIDELITY`

适用于设计稿到 HTML、Figma 到代码、批准原型到正式页面的实现检查。

前置条件：

```text
存在已批准视觉基线
基线具有 baseline_hash
审核对象具有 artifact_hash
运行环境已冻结
```

审核目标：实现是否像素级忠于批准基线。

#### 3.3 视觉回归审核 `REGRESSION`

适用于版本升级、代码修改、组件库升级、浏览器升级后的非预期视觉变化检查。

审核目标：批准变化之外的视觉差异必须为零；所有预期变化必须绑定变更单和新基线。

#### 3.4 组合审核

复杂商业软件、设计器、核心工作台和发布前页面必须同时通过：

```text
DESIGN_QUALITY
AND FIDELITY（存在批准基线时）
AND REGRESSION（已进入持续开发时）
```

像素级一致只能证明"实现忠于基线"，不能证明"基线本身是优秀设计"。设计质量必须由任务证据、标杆证据和独立视觉审核单独证明。

#### 3.5 修复模式 `FIX_MODE`

适用于 AI 在审核失败后进行自修复的场景。

前置条件：

```text
已完成至少一次完整审核（DESIGN_QUALITY / FIDELITY / REGRESSION）
存在 findings.json 或等价的 issue 清单
已记录修复前的 artifact_hash（FIX_MODE 过程中允许 artifact_hash 变化）
```

> **说明**：修复必然改变 artifact_hash。"artifact_hash 未变化"是审核阶段的前置条件，不是 FIX_MODE 的约束。FIX_MODE 正确的前置条件是"已记录修复前的 artifact_hash"，修复后生成新的 artifact_hash 并在新的哈希上重新验证。

核心原则：

```text
修复前必须先完成审核，获取 issue 清单
修复前记录当前 artifact_hash
按 P0 → P1 → P2 顺序逐项修复
每项修复后重新计算 artifact_hash
修复完成标准：所有 P0/P1 消除
修复后必须重新进入审核流程，不能自行放行
每轮修复后重新运行相关 G 层门禁验证
不得修复 A 导致 B 退化
```

**修复轮次限制**：

```text
单次 FIX_MODE 会话最多执行 2 轮修复（修复→验证→修复→验证）
第 1 轮修复后重新审核仍 FAIL → 可进入第 2 轮
第 2 轮修复后重新审核仍 FAIL → 强制退出 FIX_MODE，必须上报人工处理

全局升级规则（跨 FIX_MODE 会话）：
触发条件（必须同时满足）：
  1. 同一个 artifact_hash（修复前的原始哈希）下
  2. 累计 3 次独立 FIX_MODE 会话均以 FAIL 结束
  3. 每次 FAIL 由独立 AI 上下文独立审核得出

升级后的处理：
  - 修复前 artifact_hash（第一个会话开始时的哈希）→ 3 次 FAIL 后，该哈希对应原型强制上报人工处理
  - FIX_MODE 可继续尝试修复，但需在新 artifact_hash（修复后的哈希）下重新计数
  - 人工处理决定：是否继续修复、是否调整需求范围、是否废弃当前原型
```

**FIX_MODE 升级闭环规范**：

```text
触发升级的情形：
  - 第 2 轮修复后重新审核仍 FAIL → 退出 FIX_MODE，上报人工
  - 跨会话全局 3 次 FAIL → 上报人工

升级上报内容（必须包含）：
  - artifact_hash（修复前的原始哈希）
  - 所有 3 次会话的 findings.json 摘要
  - 每次会话的失败原因归类（是工具链问题、实现问题还是设计问题）
  - 未修复 P0 清单及各自的影响说明
  - FIX_MODE 已尝试的修复措施及其效果

人工处理决策（必须在上报后 5 个工作日内给出）：
  - 继续修复：指定新的修复范围或策略
  - 调整范围：缩减或重新定义本次审核范围
  - 废弃原型：放弃当前原型，重新开始设计/实现
  - 无论哪种决策，均必须给出书面记录存档
```

> **"连续 3 次 FAIL"的含义澄清**：指跨不同 FIX_MODE 会话的独立审核 FAIL，而非同一会话内的轮次计数。

**FIX_MODE 轮次计数器与全局升级审计规范**：

```text
轮次计数必须由外部系统持有，不得由修复者自行维护：
  - 单次会话轮次：必须在外部记录（如 audit DB、独立文件或 CI/CD 变量），修复者仅读取
  - 跨会话全局 FAIL 计数：必须在 audit DB 或独立文件中维护，修复者无法自行重置
  - 轮次计数器必须有操作日志（谁在何时执行了什么操作）

全局升级触发条件（必须同时满足）：
  1. 同一个 artifact_hash（修复前的原始哈希）下
  2. 累计 3 次独立 FIX_MODE 会话均以 FAIL 结束
  3. 每次 FAIL 由独立 AI 上下文独立审核得出

防止轮次计数器被绕过的规则：
  - 修复者不得读取或修改轮次计数器状态
  - 会话重启（重新对话）不得清除轮次计数
  - 任何尝试绕过轮次限制的行为必须记录为 P0 问题
```

**FIX_MODE 轮次计数器最小实现方案**：

团队无需建设复杂数据库，以下三种方案按成本递增选择：

```text
方案 A（最小成本，单文件）：
  在项目目录 .ai-workflow/ 下维护 fix_mode_state.json：
  文件创建时机：首次进入 FIX_MODE 时由 Audit Owner 创建，修复者不得创建或删除该文件。
  {
    "original_artifact_hash": "...",  // 修复前的原始哈希，跨会话计数的锚点
    "current_artifact_hash": "...",   // 最新修复后的哈希
    "session_count": 0,
    "sessions": [
      {"session_id": 1, "started_at": "...", "ended_at": "...", "verdict": "FAIL", "rounds": 2}
    ],
    "escalated": false
  }
  文件权限：修复者只读，审核者可写。
  适用场景：单团队、单项目。

方案 B（Git 仓库，可追溯）：
  将 fix_mode_state.json 纳入 Git，每次更新产生 commit。
  commit message 格式：fix-mode: session N, verdict FAIL/PASS, artifact_hash XXXX。
  历史不可篡改（Git 历史），修复者无法 delete + recreate 不留痕迹。
  适用场景：需要审计追溯的团队。

方案 C（CI/CD 集成，强制执行）：
  在 CI pipeline 中加入 fix-mode-gate job：
  - 读取 fix_mode_state.json
  - 若 session_count >= 3 且 escalated = true，阻断后续 pipeline
  - 若 session_count >= 2 且当前会话 FAIL，设置 escalated = true
  适用场景：已接入 CI/CD 的团队。
```

FIX_MODE 的审核者与被修复对象不得为同一 AI 上下文。AI 修复者协议见第二部分第 20 节。

> **"独立"的具体执行要求**：在 Claude Code 等多 Agent 系统中，"不得为同一 AI 上下文"指修复会话和审核会话必须是两个独立的 Agent 实例，分别以不同 system prompt 启动，不得共享同一 conversation history。

### 4. 风险分级

| 等级 | 适用对象 | 最低审核要求 |
|---|---|---|
| UI-R1 低风险 | 文案、非关键图标、轻微样式调整 | 变更区域截图、主视口、相关状态、Token 检查 |
| UI-R2 标准 | 普通列表、详情、表单、弹窗、抽屉 | 主视口、最窄视口、状态矩阵、交互回放、像素/布局/a11y 门禁 |
| UI-R3 严格 | 交互式设计器、强表格、复杂工作台、核心业务页面 | 全流程、全关键状态、断点前后视口、关键 ROI、错误注入、独立 UX 审核、最终人工放行 |

满足以下任一条件，最低为 UI-R3：

```text
包含画布、拖拽、Inspector、结构树或强表格
包含保存、预览、发布、回滚或版本冲突
包含权限差异、只读、禁用或审计状态
包含金额、权限、不可逆操作等高风险业务展示
包含三个以上工作区域或三个以上关键状态分支
```

#### 4.1 轻量版视觉审核

当第一部分走轻量版流程（第一部分第 15 节）时，视觉审核也应轻量化。轻量版审核与完整审核的对应关系：

| 完整审核 | 轻量版审核 | 裁剪说明 |
|---|---|---|
| U0 Scope Freeze | 简化为"确认审核范围 + 风险等级" | 无需完整合同构建 |
| U1 Environment Freeze | 保留（artifact_hash 仍必须冻结） | 不得裁剪 |
| U2 Visual Contract | 简化为"1 条核心任务 × 主视口 × P0 状态" | 覆盖矩阵缩减 |
| U3 Runtime Capture | 保留（核心任务必须真实回放） | 截图数量减少 |
| U4 Automated Gates | 仅执行 G2（运行完整性）+ G3（核心任务）+ G8（a11y critical） | G4/G5/G6/G7 跳过 |
| U5 Independent Critique | UX Critic 抽查 1 项（信息架构或标杆差距） | 非全维度审核 |
| U6 Verdict | PASS 标准降低：P0=0，P1≤2 | P1 允许带病通过但需记录 |
| U7 Archive | 保留 | 不得裁剪 |

轻量版审核不适用于 UI-R3（高风险）页面。

### 5. 角色和职责

| 角色 | 职责 | 禁止事项 |
|---|---|---|
| Submitter | 提交冻结版本、基线、状态矩阵和运行说明 | 不得批准自己的产物 |
| Visual Verifier | 执行浏览器、截图、DOMRect、Token、像素差、overflow 和 a11y 自动检查 | 不做主观美学放行；不得修改审核对象 |
| UX Critic | 独立审核信息架构、密度、层级、任务效率、视觉专业度和标杆差距 | 不得只读取 Submitter 的总结；不得直接修改产物 |
| Audit Owner | 确认范围、判级、证据完整性和最终结论 | 不得用平均分覆盖硬门禁失败 |
| Human Approver | UI-R3 或高风险页面最终放行 | 不得在缺少证据时口头放行 |

**Human Approver 授权规范**：

```text
适用范围：
  - UI-R3：必须由 Human Approver 签署方可输出 PASS
  - UI-R1 / UI-R2：推荐由 Human Approver 签署；若未签署，审核报告必须包含声明"本次审核未包含人工终审，结论基于自动化审核给出，请相关责任人在使用前确认"

授权条件：
  - 必须由人工（自然人）担任，不得由 AI 代理
  - 必须持有本次审核的完整证据包（manifest.json、截图、报告）
  - 必须在 UI-R3 审核报告的 Human Approver 字段签署
  - UI-R3 审核报告无 Human Approver 签字 → verdict 不得为 PASS
  - UI-R1 / UI-R2 未签署时 → verdict 可为 PASS，但必须附加"未含人工终审"声明

批准记录必须包含：
  - Approver 姓名/ID
  - 批准时间（ISO 8601）
  - 审核 artifact_hash
  - 批准范围（通过 / 有条件通过 + 条件清单）
  - Approver 对核心证据的直接查阅确认（不能只凭摘要）

禁止：
  - AI 代替 Human Approver 签署
  - 仅凭口头/聊天记录放行
  - 在证据包不完整时提前放行
  - 事后补签（必须在进入下一阶段前完成）
```

职责分离要求：

```text
Submitter != UX Critic
Submitter != 最终 Approver
Visual Verifier 不得调整失败阈值以帮助通过
同一个 AI 可以执行多个技术步骤，但最终 Critic 必须使用独立上下文或独立子代理重新审阅原始证据
```

### 6. 审核输入

开始审核前必须具备以下输入：

```text
audit_id
审核模式和风险等级
被审核对象路径或 URL
artifact_hash
视觉基线路径和 baseline_hash（FIDELITY / REGRESSION 必填）
目标用户、核心任务和非目标
支持的浏览器、设备和最小/最大视口
主题、语言、DPR 和浏览器缩放要求
PT / IX / PS 或等价的页面、交互、状态清单
设计 Token 或批准的 computed style 基线
固定测试数据和状态触发方法
P0/P1/P2 判定标准
```

缺少以下任一项时，审核结论只能是 `EVIDENCE_INSUFFICIENT`：

```text
artifact_hash
审核范围
核心任务
支持视口
状态清单
FIDELITY / REGRESSION 模式下的 baseline_hash
```

**工具链失败归因规范**：

```text
工具链自身故障（如 puppeteer 崩溃、无头浏览器无法启动）：
  → 归因：工具链问题，非被审核对象问题
  → 处理：EVIDENCE_INSUFFICIENT，修复工具链后重试
  → 不得因此降低对被审核对象的评分

被审核对象导致的故障（如 JS 错误导致页面崩溃）：
  → 归因：被审核对象问题
  → 处理：G2 FAIL，记录为 P0/P1（取决于是否影响核心任务）
  → 不得因此判定为 EVIDENCE_INSUFFICIENT

无法区分归因时：
  → 必须先修复工具链，再用已知良好工具重试
  → 重试后仍失败 → 归因被审核对象
  → 重试后通过 → 归因工具链，记录后通过
```

### 7. 审核状态机

```text
U0 Scope Freeze
  确定审核模式、风险等级、对象、任务和通过标准

U1 Artifact & Environment Freeze
  冻结 artifact_hash、baseline_hash、浏览器、字体、视口、DPR、主题和测试数据

  ### 7.1 artifact_hash 与 baseline_hash 计算规范

  **artifact_hash（审核对象哈希）**

  被审核对象的版本标识，必须满足以下全部条件：

  ```text
  内容指纹：被审核 HTML / 页面 / 构建产物的完整内容哈希
    推荐算法：SHA-256 或 MurmurHash3
    输入范围：
      - HTML 原型：完整 HTML 文本（不含动态注入的运行时状态）
      - 前端页面：构建产物（dist）的完整内容哈希
      - 设计稿文件：文件二进制内容哈希

  时间截断：截断到分钟（YYYYMMDDHHMM），不得包含秒和毫秒
    原因：artifact_hash 的核心用途是唯一标识"某个时刻冻结的内容版本"
    格式：{内容哈希前16位}_{YYYYMMDDHHMM}
    冲突解决规则：同一分钟内生成两个不同内容版本时，以 artifact_hash 前16位内容哈希差异区分

  禁止的输入：
    - 动态数据、随机数、时间戳、session ID
    - 注释中的临时说明文字
    - 未冻结的外部依赖（CDN 资源需记录版本）

  SPA / 动态内容场景的特殊处理：
    - SPA 的 artifact_hash 应基于源代码快照（commit hash 或构建产物 hash）
    - 多文件原型：必须使用确定性拼接顺序——按文件名字节序升序拼接
    - 禁止使用 CDN 浮动版本号，必须解析为具体版本号并记录在 cdn_locks 字段
  ```

  **baseline_hash（视觉基线哈希）**

  已批准参考设计的版本标识，必须使用完整 SHA-256 哈希：

  ```text
  格式：{SHA-256完整哈希}_{YYYYMMDDHHMM}
  禁止将 baseline_hash 截断（会导致碰撞风险）
  ```

  **FIX_MODE 中的 artifact_hash 行为**

  ```text
  修复前：记录当前 artifact_hash
  修复中：每项修复后重新计算 artifact_hash
  验证时：使用修复后的新 artifact_hash 重新审核
  审计：修复记录必须包含旧哈希 → 新哈希的映射关系
  ```

  > 注意：artifact_hash 变化是 FIX_MODE 的正常行为，不是审核失败条件。

  **artifact_hash 变更时的证据生命周期规则**：

  ```text
  artifact_hash 在 U1-U6 审核阶段发生变化（FIX_MODE 之外）：
    → 本次审核任务立即终止
    → U1-U6 所有已采集证据（截图、DOM、报告）标记为 stale
    → 必须重新从 U1 开始，以新 artifact_hash 重新执行审核
    → 原因：非预期变更意味着被审核对象已不在冻结状态，证据失效

  FIX_MODE 中 artifact_hash 变化：
    → 修复前 artifact_hash 与修复后 artifact_hash 均需记录
    → 修复前证据保留在原证据包（标记为"pre-fix"）
    → 修复后新证据存入新证据包
    → 不得将修复前后的证据混合使用

  artifact_hash 变更记录规范：
    必须记录：
      - 变更前的 artifact_hash
      - 变更后的 artifact_hash
      - 变更原因（人工修改 / FIX_MODE 修复 / 外部同步 / 重新构建）
      - 变更执行者（AI / 人工）
      - 变更时间戳（ISO 8601）
  ```

U2 Visual Contract Build
  建立页面 × 任务 × 状态 × 视口 × ROI 审核矩阵（"合同" = 审核矩阵约定的期望行为，
  包括：路由/页面存在性、状态流转路径、关键元素位置、Token 数值等）

U3 Runtime Capture
  在真实浏览器中逐项触发状态，采集 before / after / diff、DOM、ARIA、控制台和业务结果

U4 Automated Gates
  执行合同、运行、几何、Token、像素、响应式、布局和 a11y 硬门禁

U5 Independent Visual Critique
  UX Critic 独立检查信息架构、层级、密度、可读性、可发现性、专业感和标杆差距

U6 Verdict
  汇总证据，输出 PASS / FAIL / EVIDENCE_INSUFFICIENT

U7 Archive
  归档证据包、问题单、结论、版本哈希和未覆盖风险
  退出条件（依 verdict 分三种路径）：
    - PASS：证据包已归档且签名完成 → 审核任务正式结束
    - FAIL：证据包已归档 + 问题单已输出 + 已进入 U8 Findings Tracking → 审核任务结束，转整改
    - EVIDENCE_INSUFFICIENT：证据包标记为不足 + 已通知委托方 → 审核任务强制终止
  → FIX_MODE 不属于 U0-U7 状态机——它是独立于审核状态机的修复工作流

U8 Findings Tracking（仅 verdict = FAIL 或 EVIDENCE_INSUFFICIENT 时进入）
  记录未关闭的 P0/P1/P2，建立整改追踪条目
  门禁：无（U8 不是审核阶段，是审核后的整改管理阶段）
  退出条件：所有 P0/P1 已修复并通过重新审核，或已上报人工裁决

U9 Re-audit（仅在 FIX_MODE 修复完成后或整改完成后触发）
  以新 artifact_hash 重新执行 U1-U6
  新审核的 baseline_hash 继承上次 U1 的值（除非基线本身被修改）
  门禁：重新满足 U6 通过公式
  退出条件：U6 输出 PASS，或再次触发升级裁决
```

**FIX_MODE 与 U7 并发互斥规则**：

```text
同一 artifact_hash 下，FIX_MODE 和 U7 不得并发执行：
  - FIX_MODE 开始时：必须锁定该 artifact_hash 下的 U7 入口
  - U7 开始归档时：必须检查是否存在未完成的 FIX_MODE 会话
  - 并发检测机制：audit DB 必须记录 artifact_hash 的当前状态
```

回退规则：

```text
范围或合同错误       -> 回 U0 / U2
artifact_hash 变更   -> 当前证据全部失效，回 U1
环境不一致           -> 回 U1
状态无法触发         -> FAIL，回产品/原型实现阶段
自动门禁失败         -> FAIL，输出问题单
独立视觉审核失败     -> FAIL，输出问题单
证据缺失             -> EVIDENCE_INSUFFICIENT
```

审核任务在 U6 输出结论后结束。修复属于新的实现任务；审核者不得在本审核任务中修改被审核对象。

#### 11.3 状态触发方法规范

状态覆盖不只是"列出状态名"，必须能够**在真实浏览器中触发并验证**。

**交互状态**

| 状态 | 触发方法 | 验证点 |
|---|---|---|
| hover | CSS `:hover` 或 JS `mouseenter` 事件 | 视觉变化可观察（颜色/阴影/光标等） |
| focus-visible | Tab 键导航 + 截图 | focus ring 可见且不依赖鼠标 |
| selected | 点击选择或 checkbox/radio 状态 | 选中态与未选中态有明显视觉差异 |
| pressed / active | 点击时 mousedown + 截图 | active 态可见，松手后恢复 |
| disabled | 表单 disabled 属性或 view 层 disabled | 视觉上明显不可交互（opacity/灰度/光标），且点击无效 |

**数据状态**

| 状态 | 触发方法 | 验证点 |
|---|---|---|
| loading | 真实等待或 Mock 请求拦截 | 显示加载指示器，界面不卡死 |
| empty | fixture 设置空数据渲染 | 显示有意义的空状态文案/插图 |
| no-result | fixture 设置无匹配数据 | 显示有意义的空结果提示 |
| success | Mock 成功响应 | 反馈文案和视觉标识正确 |
| validation error | 提交非法值触发校验 | 错误提示可见且指向正确字段 |
| network error | Mock 网络错误或断网 | 错误提示可见，恢复路径明确 |
| timeout | Mock 延迟超时 | 超时提示可见，不卡死界面 |
| permission denied | 模拟无权限角色登录 | 权限不足态可见，不泄露受保护内容 |
| partial success | Mock 部分成功响应 | 明确告知部分成功/部分失败 |
| conflict / stale | Mock 并发冲突或数据过期 | 冲突提示可见，恢复路径明确 |
| long text / overflow | fixture 设置超长文本 | 无意外截断、无布局破坏 |
| extreme data | fixture 设置边界数据 | 无崩溃、无截断、无溢出 |

> **关键原则**：状态触发必须使用**真实交互或可复现的 Mock 数据**，不得仅凭代码结构推断"应该能触发"。

### 12. 八层审核门禁

八层门禁串联执行，任一硬门禁失败即总失败，不允许用总分抵消。

#### G1 合同一致性门禁

检查：

```text
页面、导航、PT、IX、PS、测试用例数量一致
所有链接和锚点存在
按钮、状态和说明与实际行为一致
设计稿、原型、测试规格和审核矩阵使用同一合同
```

通过标准：合同不一致数为 `0`。

#### G2 运行完整性门禁

检查：

```text
页面身份正确
页面非空白
无框架错误覆盖层
无未解释的 console error
核心资源加载成功
```

通过标准：运行错误和未解释控制台错误为 `0`。

#### G3 核心任务和状态门禁

检查每个点击、输入、拖拽、保存、预览、发布等操作后：

```text
可见状态是否改变
DOM / ARIA 状态是否改变
业务状态是否改变
失败时是否有可操作反馈
是否存在恢复路径
```

通过标准：

```text
核心任务成功率 = 100%
强制状态覆盖率 = 100%
dead-end = 0
```

**"核心任务成功"的定义**（必须同时满足）：

```text
1. 操作完成后，预期业务状态在 DOM/ARIA 中可见且正确反映
2. 无错误提示，或有明确恢复路径
3. 下一级用户目标可达（不得出现不可恢复的死胡同）
```

> 禁止以下作为"成功"的判定依据：命令返回 200、元素获得焦点、`data-testid` 存在、页面未崩溃。

> **组件视觉状态 vs 任务状态覆盖的区分**（防止 AI 混淆）：
> - "组件视觉状态"（hover / focus-visible / disabled / loading / skeleton / 空状态 / 错误态）
>   由 Section 25.2.1 组件自检清单在生成阶段覆盖，不由 G3 检查。
> - "任务状态覆盖"（点击 → 可见状态改变 / DOM-ARIA 改变 / 业务状态改变 / 失败反馈 / 恢复路径）
>   由 G3 门禁在审核阶段检查，Section 25.2.1 不覆盖此维度。
> 两个维度独立，必须分别通过 Section 25.2.1 和 G3，不允许互相替代。

#### G4 几何与布局门禁

检查：

```text
关键元素 DOMRect
对齐线
组件尺寸
间距
文字基线
容器比例
sticky / fixed / overlay 位置
```

默认阈值：

| 项目 | UI-R1/R2 | UI-R3 / 关键 ROI |
|---|---:|---:|
| x/y 位置偏差 | <= 2 CSS px | <= 1 CSS px |
| width/height 偏差 | <= 2 CSS px | <= 1 CSS px |
| 文字基线偏差 | <= 1 CSS px | <= 1 CSS px |
| 意外 overlap | 0 | 0 |
| 意外 clipping | 0 | 0 |
| 离屏关键操作 | 0 | 0 |
| 无策略整页横向滚动 | 0 | 0 |

任何导致换行、截断、错列或操作不可达的偏差，即使小于阈值也必须失败。

#### G5 Token 与排版门禁

以下属性必须命中批准设计 Token 或视觉合同：

```text
font-family
font-size
font-weight
line-height
letter-spacing
foreground / background color
border color / width
spacing
radius
shadow
opacity
icon size
```

通过标准：关键 ROI 未授权 Token 偏差为 `0`。

字体、颜色和间距应优先读取 computed style；禁止仅凭截图像素推断。

#### G6 像素差异门禁

像素差异只适用于 `FIDELITY` 和 `REGRESSION` 模式。

默认阈值：

| 区域 | 默认阈值 |
|---|---:|
| 稳定关键 ROI | `maxDiffPixelRatio <= 0.1%` |
| 稳定普通区域 | `maxDiffPixelRatio <= 0.3%` |
| 不应变化的回归区域 | 超阈值即失败 |

关键规则：

```text
关键 ROI 必须单独计算
禁止只看全页平均 diff
diff 热力图必须保留
像素 diff 必须与 DOMRect 和 computed style 结果一起解释

并发审核防伪规则：
  - 每个 G 层门禁执行前必须记录当前 artifact_hash
  - 若 G1-G8 任一层执行时 artifact_hash 与 U1 记录不一致，该层结论无效
```

#### G7 响应式和内容压力门禁

必须检查：

```text
断点前后布局是否按合同变化
导航、主内容、Inspector、弹窗和表格是否可达
长中文、长英文、长数字、空值、极值数据
字体缩放后的换行和遮挡
scroll container 是否正确
是否存在滚动陷阱
```

通过标准：意外溢出、遮挡、不可读、不可达和无提示裁切为 `0`。

#### G8 可访问性与视觉语义门禁

自动检查：

```text
axe critical / serious
键盘焦点顺序
focus-visible
颜色对比度
表单标签
ARIA 状态
动态结果 aria-live / role=status / role=alert
禁用、只读和权限态的可感知性
```

人工检查：

```text
视觉层级是否帮助用户做正确决策
危险操作是否足够醒目但不过度干扰
状态是否只依赖颜色表达
错误提示是否指出问题和下一步动作
```

通过标准：

```text
critical = 0
serious = 0
核心任务键盘可完成
核心状态可被辅助技术感知
```

### 13. 独立视觉审核

自动化只能证明"不明显坏"，不能证明"设计优秀"。U5 必须由独立 UX Critic 审核：

| 维度 | 必须回答的问题 |
|---|---|
| 信息架构 | 用户能否理解区域、模式、面板和状态之间的关系 |
| 视觉层级 | 主任务、主信息、辅助信息和低频信息是否分层清楚 |
| 视觉密度 | 首屏信息量、行高、字号、间距是否适合长期使用 |
| 可读性 | 文本、表格、表单、状态和错误是否清楚 |
| 可发现性 | 高频入口、当前选择、可拖拽、可编辑和可恢复动作是否明确 |
| 一致性 | 同类组件、状态、间距和反馈是否一致 |
| 专业感 | 是否符合目标商业软件类别 |
| 标杆差距 | 是否低于功能等价标杆；差异是否有任务收益证据 |

禁止使用以下话术作为通过依据：

```text
看起来不错
整体比较现代
风格很高级
大体一致
应该没问题
```

每个结论必须引用 `case_id / screenshot_id / ROI / benchmark source_id`。

### 13.5 AI 自检与独立审核的本质区别

这是本规范最核心的概念澄清之一。AI 生成者自检（Part 2 附第 25 节）与独立审核（Part 2 U5/G 层）不是同一件事，不能互相替代：

| 维度 | AI 生成者自检（第 25 节） | 独立审核（U5 / G 层） |
|---|---|---|
| 执行者 | 生成者自己（Design Generator） | 独立 UX Critic / Visual Verifier |
| 依据 | 设计 Token、spacing scale、状态清单 | 原始证据（截图/录屏）、rubric、benchmark |
| 检查范围 | 组件级：溢出、重叠、token 命中、状态生成 | 页面级：任务可达性、信息架构、标杆差距 |
| 发现问题的概率 | 高（熟悉自己代码） | 高（无先入之见，盲测） |
| 结论性质 | 生成阶段内控，不是质量证明 | 独立质量证明，可进入 U6 verdict |
| 能否替代对方 | 禁止 | 禁止 |
| 能否互相补充 | 是（两者都必须执行） | 是（两者都必须执行） |

**为什么禁止互相替代**：

AI 生成者自检的盲区：
```text
1. 确认偏误：生成者知道"应该是什么"，容易把"差不多"看成"正确"
2. 上下文丢失：生成者在生成过程中已消耗大量 token context，
   对"正常应该什么样"的判断基准已经漂移
3. 无法发现根本性设计错误：生成者已深度投入当前方案，
   很难质疑"整体方向是否正确"
```

独立审核的盲区：
```text
1. 无法读取生成者脑中的意图：独立审核者只能看到最终产物，
   无法知道"这个组件为什么要这样设计"
2. 无法快速定位问题根因：发现问题后需要大量猜测才能定位代码位置
3. 无法处理边缘情况：某些问题只在特定数据条件下触发，
   独立审核者不知道哪些边界数据是关键
```

**正确做法**：两者都必须执行。先自检（减少显而易见的问题），再提交独立审核（验证整体质量）。自检发现的问题越多，独立审核的效率就越高；自检放行的问题越多，独立审核的"兜底价值"就越大。

### 14. 动态区域与遮罩规则

允许遮罩的典型内容：

```text
当前时间
随机生成 ID
实时监控值
用户头像或第三方图片
不可冻结的光标或视频帧
```

遮罩必须满足：

```text
按稳定 selector 或 ROI 精确定位
记录 mask_id 和理由
遮罩范围最小化
不得覆盖关键操作、错误反馈、布局边界或主要内容
```

禁止：

```text
为了降低 diff 比例遮罩整个面板
失败后临时扩大遮罩
遮罩核心业务数据或关键状态
未记录理由的任意遮罩
```

### 15. 问题分级

本节定义审核阶段的 P0/P1/P2 分级，与第一部分第 16 节设计阶段的 P0/P1/P2 分级是同一套体系在不同阶段的应用：设计阶段侧重设计输入问题（如 benchmark 准入、证据溯源），审核阶段侧重运行态问题（如合同不一致、像素偏差）。两阶段的 P0 均为阻断级，P1 均为必须修复，P2 均为可放行后续优化。当两节定义有差异时，审核阶段以本节为准，设计阶段以第一部分第 16 节为准。

#### 15.1 P0 阻断

```text
核心任务无法完成或无真实状态变化
关键页面、区域或状态缺失
错误态无恢复路径
权限、只读、禁用或危险操作含义不清
关键内容被遮挡、裁切、离屏或不可读
关键 ROI 超出像素/几何阈值并影响任务
最窄支持视口无法完成核心任务
a11y critical 或 serious 阻断核心任务
合同、导航、页面、状态和测试不一致
基线或证据被伪造、复用或跨版本混用
```

#### 15.2 P1 必须修复

```text
非核心但高频路径视觉层级混乱
局部响应式问题
局部 Token、间距或对齐超阈值
错误文案无法指导下一步
低频状态缺少可操作反馈
自动化抓手不稳定或不唯一
非核心键盘路径或标签缺失
```

#### 15.3 P2 后续优化

```text
不影响任务的轻微视觉差异
非关键文案优化
低风险微交互优化
非核心状态提示增强
```

P0 和 P1 不允许用总分抵消。P2 可以在记录责任人和后续计划后放行。

### 16. 审核结论

只允许以下结论：

| 结论 | 条件 |
|---|---|
| PASS | 证据完整，全部硬门禁通过，P0=0，P1=0，独立 Critic 通过 |
| FAIL | 任一硬门禁失败，或存在 P0/P1 |
| EVIDENCE_INSUFFICIENT | 缺少基线、哈希、状态、视口或关键证据，无法做出有效判断 |

最终通过公式：

```text
PASS =
  verdict != EVIDENCE_INSUFFICIENT
  AND artifact_hash 固定
  AND 环境可复现
  AND 审核矩阵完整
  AND G1-G8 全部通过
  AND 核心任务成功率 = 100%
  AND 强制状态覆盖率 = 100%
  AND P0 = 0
  AND P1 = 0
  AND Independent Critic = PASS
  AND (risk_level != "UI-R3" OR Human_Approver = APPROVED)
```

> `UI-R3` 为高风险等级，审核者不是 Human Approver，不得输出 PASS。

**EVIDENCE_INSUFFICIENT 追踪规范**：

```text
触发 EVIDENCE_INSUFFICIENT 时，必须同时完成以下动作：
  1. 在 manifest.json 的 verdict_reason 字段中填写缺失证据的具体描述
  2. 在 findings.json 中创建 EVIDENCE_INSUFFICIENT 专属 issue
  3. 在 audit DB 或独立追踪文件中记录本次 EVIDENCE_INSUFFICIENT 事件
  4. 明确下次审核截止时间（建议不超过 3 个工作日）
```

### 17. 证据包规范

每次审核必须生成独立证据目录：

```text

### 19. 反作弊与反 Goodhart 规则

以下行为直接使本次审核无效：

```text
用 data-testid 数量证明可测试性
用截图数量证明状态完整
用点击成功证明业务成功
用全页平均 diff 掩盖关键 ROI 失败
复用旧 artifact_hash 的截图
审核期间修改被审核文件但不重启审核
失败后由生成者自行放宽阈值
失败后扩大动态遮罩
删除失败用例以提高覆盖率
Designer 自己担任最终 Critic
无 source_id 却声称超过竞品
用总分抵消 P0/P1
```

以下指标不能独立代表质量（完整规则见第一部分第 17 节反 Goodhart 规则，此处仅列审核阶段补充项）：

```text
截图数量
data-testid 数量
像素相似度总分
自动化通过用例数量
```

以上指标必须回答：它证明了哪个真实任务、风险断点或用户决策？无法证明的指标不计分，为刷指标而增加复杂度时反向扣分。

### 20. AI 强制执行协议

向 AI 下达 UI 审核任务时，必须附加以下协议：

```text
你当前是 UI 审核者，不是修复者。
禁止修改被审核 HTML、设计稿、代码、配置和测试数据。

1. 先计算并记录 artifact_hash；若有基线，同时记录 baseline_hash。
2. 声明审核模式、风险等级、浏览器、视口、主题、语言和 fixture。
3. 建立页面 × 任务 × 状态 × 视口 × ROI 审核矩阵。
4. 使用真实浏览器触发状态，保存 before / actual / diff、DOM、ARIA、console 和业务结果。
5. 按 G1-G8 顺序执行，任一硬门禁失败即判定 FAIL，不得用平均分覆盖。
6. 像素 diff 只作为定位证据；必须同时检查 DOMRect、computed style、任务和状态。
7. 发现问题只记录 issue、证据、影响、回退阶段和复验退出条件，不得修改被审核对象。
8. 审核期间 artifact_hash 改变时，立即停止并判定现有证据失效。
9. 没有批准基线时，不得声称"像素级通过"。
10. 最终只输出 PASS、FAIL 或 EVIDENCE_INSUFFICIENT。
```

### AI 修复者协议

向 AI 下达 UI 修复任务时，必须附加以下协议：

```text
你当前是 UI 修复者，不是审核者。
你必须先完成一次完整审核获取 issue 清单，再开始修复。
禁止在未读取 findings.json 的情况下开始修复。

1. 读取 findings.json 或等价的 issue 清单，按 P0 → P1 → P2 排序。
2. 对每条 issue，定位到具体元素、代码行和 artifact_hash。
3. 按以下优先级修复：
   P0：必须修复，不修复不得放行
   P1：必须修复，不修复不得放行
   P2：记录原因后可推迟，但必须有后续计划
4. 每项修复后重新计算 artifact_hash，确保变化可追溯。
5. 修复后重新运行相关 G 层门禁验证。
6. 不得修复 A 导致 B 退化；每轮修复后检查所有已通过的门禁。
7. 无法自行修复的 issue 必须上报，说明原因和需要的决策。
8. 修复完成后不得自行放行，必须重新进入审核流程。
9. 修复时优先使用 design token，不得自行发明新值。
10. 视觉问题（间距、颜色、字体）优先使用规则化布局修复，
    不得通过微调像素值"凑合通过"。
11. 轮次限制（单次 FIX_MODE 会话最多 2 轮：修复→验证→修复→验证；
    第 2 轮修复后重新审核仍 FAIL → 强制退出，上报人工处理）。
12. 全局升级条件（同一原始 artifact_hash 下跨会话累计 3 次 FAIL →
    必须上报人工裁决，不得继续自动修复）。
```

### 24. 设计门禁未执行时的回退处理

第一部分 S0-S7 设计门禁是**设计前质量门禁**，不是第二部分视觉审核的硬前置。实际项目中可能因时间、资源或轻量化需求跳过了设计门禁。这种情况下仍可执行视觉审核，但必须明确以下约束：

> **优先级声明**：本节跳过规则为例外机制，优先级低于第 20.5.3 节硬约束。即当本节的跳过规则与第 20.5.3 节阶段衔接硬约束冲突时，以第 20.5.3 节为准。

**允许跳过设计门禁的场景**

```text
- 内部工具、低风险页面、一次性原型
- 竞品证据无法采集（竞品无公开可访问 UI、无版本记录）
- 快速验证概念，不做正式发布级设计
- 已有经批准的设计系统约束包，无需再对标外部竞品
```

**跳过设计门禁后，视觉审核中的受影响项**

| 视觉审核中的位置 | 影响说明 | 回退处理 |
|---|---|---|
| 综合竞品优势（第一部分第 9 节设计输入包） | 无 benchmark 证据可综合 | 降级为轻量化行业研究（读 2-3 个同类产品截图），不得声称"已综合竞品" |
| G1 门禁（合同审核） | 无法引用 benchmark 任务等价性证明 | 仅以用户提供的业务需求文档为合同基准；需在 findings 中注明"无 benchmark 等价性证据" |
| FIX_MODE 修复策略 | 常见修复依赖 benchmark 参照 | 修复时不得引用"比 benchmark 更好"作为决策依据；仅依赖 design token、间距系统、WCAG 等客观标准 |
| P0 判定中的"低于标杆" | 无法判断是否低于竞品 | 该维度的 P0 降级为建议项（不阻断审核通过），但必须在 findings 中注明 |

**禁止跳过设计门禁的场景**

```text
- 高风险核心页面（涉及金额、权限、不可逆操作、合规审计的页面）
- 竞品存在且可访问，但以"节省时间"为由跳过
- 需要向外部客户、监管或合规方证明设计质量的场景
```

**回退处理的核心原则**

无论是否执行了设计门禁，视觉审核结论的表述必须准确：
- 执行了设计门禁：`"设计不低于 benchmark 下限，在 X 维度有证据支持的改进"`
- 未执行设计门禁：`"本结论未包含竞品对标；设计基于[需求文档/设计系统约束]做出"`

## 第二部分附：AI 原型生成标准操作规程（第 25 节）

### 25. AI 原型生成 SOP

本节定义 AI 生成 UI 原型时的执行规程：边生成、边检查、边修复。每个组件、每个区域完成后立即自检，不将问题积累到末尾。

> **重要声明**：本节的检查属于**生成阶段内部质量控制**，不替代第二部分第 1-24 节正式审核。
> - 本节检查由生成者执行，不受 Submitter ≠ UX Critic 约束。
> - 本节输出的 PASS/FAIL 不计入第 16 节 PASS 公式。
> - 所有组件和页面生成完成后，必须独立启动第二部分第 1-24 节审核流程。
> - 禁止将"第 25 节已检查通过"作为跳过第 1-24 节审核的依据。

> **外部验证强制要求**：第 25 节完成后，生成者必须将以下内容提交给独立角色进行二次验证：
> - artifact_hash（必须与检查点记录中的哈希一致）
> - 未修复的问题清单（如有）
>
> 二次验证的作用是确认检查清单本身是否完整、不得代替正式审核。

#### 25.1 生成前：准备阶段

AI 开始生成第一个组件之前，必须完成以下准备。未完成不得开始生成代码。

**设计系统声明：**

```text
[ ] 读取项目的 design token 文件（token.json / design-tokens.js / CSS 变量文件）
[ ] 提取并声明：
    - spacing system（间距层级，如 4/8/12/16/24/32/48px）
    - typographic scale（字体层级，如 12/14/16/20/24/32px）
    - color palette（颜色列表 + 语义用途）
    - elevation / shadow / border-radius 规则
[ ] 声明每个层级的 token 变量名
[ ] 声明每个语义颜色的使用场景
```

**布局策略声明：**

```text
[ ] 声明页面整体布局模式（flex / grid / stack）
[ ] 声明容器 max-width、padding 和响应式策略
[ ] 声明响应式断点（如 640/768/1024/1280/1536px）
[ ] 声明每个断点的主要布局变化策略
```

**组件状态计划：**

```text
[ ] 列出本页面/原型中所有交互组件
[ ] 对每个组件声明将生成哪些状态：
    按钮：default / hover / focus-visible / active / disabled / loading
    输入框：default / focus / valid / invalid / disabled / readonly
    卡片/列表项：default / hover / selected
    链接：default / hover / focus-visible / active
    模态框/抽屉：open / close 过渡
    加载态：spinner / skeleton
    空状态：插图 + 文案
    错误状态：视觉反馈 + 可操作提示
```

**任务和状态矩阵：**

```text
[ ] 确认核心用户任务
[ ] 列出核心任务涉及的所有页面状态
[ ] 确认每个状态的数据条件和预期视觉表现
```

#### 25.2 生成中：每个组件的"生成-检查-修复"循环

每个组件完成后必须执行自检，发现问题立即修复，修复后再生成下一个组件。

```
生成组件代码
  ↓
立即自检（见 25.2.1 节）
  ↓
发现问题？── 是 ──→ 立即修复（见 25.2.2 节）
  ↓ 否
记录检查结果
  ↓
下一个组件
```

##### 25.2.1 组件自检清单

每个组件生成后，逐一回答以下问题。任一问题为"否"时立即修复。

**间距与对齐：**

```text
[ ] margin/padding 是否来自 spacing token？
[ ] 组件与相邻元素间距是否符合设计系统？
[ ] 组件内部元素是否对齐？
[ ] 组件宽度是否受容器约束（max-width / flex）？
[ ] 是否有意外的 overflow（超出容器）？
```

**字体层级：**

```text
[ ] 所有文本 font-size 是否命中 typographic scale？
[ ] 标题、正文、说明文字是否有视觉层级区分？
[ ] 是否有不在 scale 中的 font-size 硬编码值？
```

**颜色使用：**

```text
[ ] 所有 color 值是否来自 color token？
[ ] 语义颜色使用是否正确（成功不用红色、错误不用蓝色）？
[ ] 背景色、前景色、边框色是否协调？
```

**组件状态：**

```text
[ ] 按钮：是否有 hover / focus-visible / disabled / loading 状态？
[ ] 输入框：是否有 focus / valid / invalid / disabled 状态？
[ ] disabled 状态是否同时包含：opacity + pointer-events + cursor 变化？
[ ] loading 状态是否显示 spinner 或 skeleton？
```

**可访问性：**

```text
[ ] 按钮/链接是否有 focus-visible 可见样式？
[ ] 表单是否有 label 或 aria-label？
[ ] 对比度是否满足：普通文本 ≥ 4.5:1，大文本 ≥ 3:1？
```

**响应式：**

```text
[ ] 组件在最窄支持视口下是否有横向滚动风险？
[ ] 组件内容是否会被裁切或截断？
[ ] 长文本是否使用了 text-overflow: ellipsis？
```

##### 25.2.2 组件修复规则

**修复原则：**

```text
1. 发现问题立即修复，不带病继续生成下一个组件
2. 优先使用 design token 修复，不得自行发明新值
3. 修复后重新检查该组件，确认问题已消除
4. 记录修复内容：问题描述 → 修复方式 → 验证结果
```

**修复优先顺序：**

```text
溢出/裁切 > 错位/不对齐 > 状态缺失 > 颜色/字体偏差 > a11y 问题
```

**禁止的修复方式：**

```text
- 不得用 margin: -1px 或 overflow: hidden 掩盖布局问题
- 不得用固定宽度"凑对齐"，必须用 flex/grid 布局
- 不得降低 contrast 来"解决"对比度问题
- 不得跳过未完成的组件状态，只生成 default 态
```

**常见错误的标准修复：**

| 错误 | 修复 |
|---|---|
| margin: 23px（硬编码） | margin: var(--spacing-lg) |
| width: 500px（固定宽度超出容器） | width: 100%; max-width: 500px |
| font-size: 18px（不在 scale 中） | font-size: var(--font-size-body) |
| color: #3b82f6（硬编码 hex） | color: var(--color-primary) |
| 错误提示用蓝色背景 | 替换为 var(--color-error) |
| .btn:disabled { opacity: 0.5 } | 添加 pointer-events: none; cursor: not-allowed |
| .btn:hover { background: #1d4ed8 } | 添加 box-shadow / transform / border 变化 |
| .btn:focus { outline: none } | 添加 :focus-visible { outline: 2px solid token } |
| 无 label 的 input | 添加 <label> 或 aria-label |
| 长文本无省略 | 添加 text-overflow: ellipsis; white-space: nowrap |

#### 25.3 区域完成后：区域级检查

当一个页面区域（Header / Sidebar / Content / Footer）完成所有组件生成后，执行区域检查。

```text
[ ] 区域内所有组件间距一致、对齐整齐
[ ] 区域内字体层级清晰（标题 > 正文 > 说明）
[ ] 区域内语义颜色一致（成功/错误/警告统一）
[ ] 区域在最窄视口下无布局问题
[ ] 区域在容器内（无意外超出）
```

#### 25.4 页面完成后：最终检查

当整个页面/原型生成完成后，执行最终检查。

```text
[ ] 页面在主视口和最窄视口下无横向滚动
[ ] 页面在每个响应式断点下布局正确变化
[ ] 所有硬编码 hex / font-size / spacing 已替换为 token
[ ] 所有交互组件状态完整
[ ] 核心任务流程可走通（点击/输入/提交/反馈）
[ ] 每个声明的状态（空/加载/错误/成功）有视觉表现
[ ] artifact_hash 已计算并记录
```

每个检查点完成后必须记录：

```text

检查时间：[ISO 8601 时间戳]
检查结果：PASS / FAIL

  1. [问题] → [修复] → [验证]

#### 25.6 生成过程中的常见陷阱

| 陷阱 | 症状 | 应对 |
|---|---|---|
| 边生成边累积问题 | 10 个小问题互相影响，修复成本高 | 每个组件完成后立即修复，不带病继续 |
| 用固定值"快速对齐" | 文字变长时溢出或换行断裂 | 使用 flex/grid 布局对齐，禁止固定宽度凑对齐 |
| hover 只是颜色变深 | 用户感知不到交互 | hover 态必须有结构性变化（shadow/transform/border） |
| 只生成 default 态 | 审核时 G3 失败，修改成本高 | 生成组件时同步生成所有状态 |
| 直接写 hex 不查 token | token 变更后页面颜色不一致 | 所有颜色必须来自 token，不得硬编码 hex |
| 用 overflow: hidden 掩盖溢出 | 内容被裁切，用户看不到 | 通过 flex-wrap / max-width 调整布局，不用 hidden 掩盖 |

## 附录 C：关键门禁检测脚本示例

本附录提供 G4/G5/G7 和生成后自动验证（第一部分第 18.3 节）的核心检测脚本片段，AI 可直接在浏览器 DevTools 或 Playwright 中执行。

### C.1 溢出检测（G4 + 第 18.3 节第 1 项）

```javascript
// 检测所有超出父容器边界的元素
const overflows = [];
document.querySelectorAll('*').forEach(el => {
  const rect = el.getBoundingClientRect();
  const parent = el.parentElement;
  if (!parent) return;
  const parentRect = parent.getBoundingClientRect();
  if (rect.right > parentRect.right + 1 || rect.bottom > parentRect.bottom + 1) {
    overflows.push({
      element: el.tagName + (el.className ? '.' + el.className.split(' ')[0] : ''),
      rect: { x: rect.x, y: rect.y, w: rect.width, h: rect.height },
      parentRect: { x: parentRect.x, y: parentRect.y, w: parentRect.width, h: parentRect.height },
      overflow: {
        right: rect.right - parentRect.right,
        bottom: rect.bottom - parentRect.bottom
      }
    });
  }
});
console.table(overflows);
```

### C.2 意外重叠检测（G4 + 第 18.3 节第 2 项）

```javascript
// 检测可见元素之间的意外重叠（排除 dropdown/tooltip/modal 等故意重叠）
const visibleElements = [...document.querySelectorAll('*')].filter(el => {
  const style = getComputedStyle(el);
  return style.display !== 'none' && style.visibility !== 'hidden' && style.opacity !== '0';
});

const overlaps = [];
for (let i = 0; i < visibleElements.length; i++) {
  for (let j = i + 1; j < visibleElements.length; j++) {
    const a = visibleElements[i].getBoundingClientRect();
    const b = visibleElements[j].getBoundingClientRect();
    const overlap = !(a.right < b.left || a.left > b.right || a.bottom < b.top || a.top > b.bottom);
    if (overlap) {
      // 排除父子关系
      const isParent = visibleElements[i].contains(visibleElements[j]) || visibleElements[j].contains(visibleElements[i]);
      if (!isParent) {
        const aStyle = getComputedStyle(visibleElements[i]);
        const bStyle = getComputedStyle(visibleElements[j]);
        // 排除 position: fixed/absolute 的覆盖层（modal/tooltip/dropdown）
        if (aStyle.position === 'static' && bStyle.position === 'static') {
          overlaps.push({
            a: visibleElements[i].tagName,
            b: visibleElements[j].tagName,
            overlapArea: Math.max(0, Math.min(a.right, b.right) - Math.max(a.left, b.left)) *
                         Math.max(0, Math.min(a.bottom, b.bottom) - Math.max(a.top, b.top))
          });
        }
      }
    }
  }
}
console.table(overlaps);
```

### C.3 Token 审计（G5 + 第 18.3 节第 6 项）

```javascript
// 检测未使用 design token 的硬编码值
const violations = [];

### C.5 对比度检测（G8 + 第 18.3 节第 5 项）

```javascript
// 检测文字与背景的对比度（简化版，精确版建议用 axe-core）
function getContrastRatio(fg, bg) {
  const parse = (color) => {
    const match = color.match(/rgb\((\d+),\s*(\d+),\s*(\d+)\)/);
    if (!match) return null;
    return [parseInt(match[1]), parseInt(match[2]), parseInt(match[3])].map(v => {
      v /= 255;
      return v <= 0.03928 ? v / 12.92 : Math.pow((v + 0.055) / 1.055, 2.4);
    });
  };
  const fgRgb = parse(fg), bgRgb = parse(bg);
  if (!fgRgb || !bgRgb) return null;
  const l1 = 0.2126 * fgRgb[0] + 0.7152 * fgRgb[1] + 0.0722 * fgRgb[2];
  const l2 = 0.2126 * bgRgb[0] + 0.7152 * bgRgb[1] + 0.0722 * bgRgb[2];
  const lighter = Math.max(l1, l2), darker = Math.min(l1, l2);
  return (lighter + 0.05) / (darker + 0.05);
}

const contrastIssues = [];
document.querySelectorAll('p, span, td, th, label, button, a, h1, h2, h3, h4, h5, h6').forEach(el => {
  const style = getComputedStyle(el);
  const ratio = getContrastRatio(style.color, style.backgroundColor);
  if (ratio !== null && ratio < 4.5) {
    contrastIssues.push({
      element: el.tagName + (el.className ? '.' + el.className.split(' ')[0] : ''),
      text: el.textContent.substring(0, 30),
      color: style.color,
      background: style.backgroundColor,
      ratio: ratio.toFixed(2),
      required: '4.5'
    });
  }
});
console.table(contrastIssues);
```

## 附录 D：P0 状态 / 必测视口 / 关键 ROI 配置方法

本附录提供通用配置方法，指导团队为任意页面类型自行确定 P0 状态集、必测视口和关键 ROI。本规范不预设特定业务场景的配置——特定业务的预设应放在项目执行指导书中，本规范只提供配置方法论。

### D.1 P0 状态集配置方法

P0 状态是页面必须覆盖的最低状态集合（见第一部分第 11.3 节）。配置步骤：

```text
步骤1：从核心任务反推必经状态
  - 列出 S0 冻结的核心任务
  - 每个任务至少经历：起始态 → 过程态 → 结果态
  - 起始态和结果态中涉及数据获取的，必须覆盖 loading / success / empty / error

步骤2：按风险断点类别补全（第一部分第 11.3 节 P0 类别）
  逐类判断该页面是否涉及该风险断点，涉及则必须配置对应状态：
  - 加载态：页面有异步数据获取 → 必配
  - 空态：列表/集合可能为空 → 必配
  - 成功态：所有页面 → 必配
  - 校验态：有用户输入 → 必配
  - 网络异常：有网络请求 → 必配
  - 权限态：有权限控制 → 必配
  - 危险确认：有删除/不可逆操作 → 必配

步骤3：按页面交互复杂度判断是否升级为复杂设计器状态集
  若页面满足以下任一条件，按第二部分第 11.2 节补全全部状态：
  - 含拖拽编辑
  - 含多面板联动
  - 含画布缩放
  - 含实时协作
  - 含三栏及以上布局

步骤4：排除不适用状态并记录理由
  对每个 P0 类别，若判断不适用，必须在 S0 约束包中记录排除理由
  排除理由不得为"首版不需要"——必须是"该页面架构上不涉及此断点"
```

### D.2 必测视口配置方法

必测视口是 U3 采证和 G7 响应式门禁必须覆盖的浏览器尺寸。配置步骤：

```text
步骤1：确定设计基准视口（必配，1个）
  - 根据产品目标用户的主流设备确定
  - 通常为 1920×1080 或 1440×900

步骤2：确定标准工作视口（必配，1个）
  - 用户实际工作中最常用的尺寸
  - 通常为 1440×900 或 1366×768

步骤3：确定最窄支持视口（必配，1个）
  - 产品声明支持的最低分辨率
  - Web 端通常为 1024×768 或 1280×720
  - 若支持移动端，增加 375×812（手机）或 768×1024（平板）

步骤4：判断是否需要断点视口（按需）
  若页面有响应式断点，每个断点上下限各测一个视口

步骤5：记录到 S0 约束包
  每个视口标注：分辨率、对应设备类型、是否为设计基准
```

### D.3 关键 ROI 配置方法

关键 ROI（Region of Interest）是 G4/G6 门禁中必须独立检查、不能用全页平均稀释的核心区域。配置步骤：

```text
步骤1：识别用户核心操作区域（必选）
  - 用户完成核心任务必须交互的区域
  - 通常包括：主操作入口、核心数据展示区、主操作按钮
  - 每个页面至少 1 个，通常 3-5 个

步骤2：识别高风险布局区域（按需）
  - 容易溢出的区域（如长文本列、动态宽度列）
  - 容易重叠的区域（如浮动操作栏、sticky 表头）
  - 容易遮挡的区域（如弹窗触发按钮、下拉菜单触发器）

步骤3：识别数据密度敏感区域（按需）
  - 信息承载率关键区域（如表格行、卡片网格）
  - 截断率关键区域（如标签栏、状态列）

步骤4：识别权限/状态敏感区域（按需）
  - 角色不同显示不同的区域
  - 状态变化时视觉反馈关键区域

步骤5：记录到 S0 约束包
  每个 ROI 标注：区域名称、选择理由（核心操作/高风险/密度敏感/状态敏感）、对应门禁
  ROI 数量建议 3-7 个，过少则覆盖不足，过多则失去焦点
```

### D.4 特殊注意事项配置方法

特殊注意事项是该页面容易出问题的特定场景，需在 S3 设计决策日志中逐条回应。配置步骤：

```text
步骤1：从 S0 边界数据契约提取
  - 最长文本字段 → 是否截断、是否有 tooltip
  - 最大记录数 → 分页/虚拟滚动性能
  - 空值/特殊字符 → 是否有占位符或防御性渲染

步骤2：从权限模型提取
  - 不同角色看到的数据范围差异
  - 不同角色的操作权限差异
  - 权限切换时的 UI 反馈

步骤3：从业务规则提取
  - 并发操作冲突场景（如两人同时编辑）
  - 状态流转的视觉反馈
  - 超时/自动处理的提示

步骤4：从交互复杂度提取
  - 拖拽/缩放/联动等复杂交互的边界场景
  - 多面板状态同步
  - 预览态/编辑态/运行态一致性

步骤5：记录到 S0 约束包
  每条注意事项标注：场景描述、风险等级、对应设计决策 ID
```

### D.5 配置产出模板

将上述四项配置汇总为一份页面配置清单，在 S0 冻结时纳入约束包：

```text
页面配置清单
页面名称：[填写]
页面风险等级：UI-R1 / UI-R2 / UI-R3（见第二部分第 4 节）
配置日期：[填写]
配置人：[填写]

P0 状态集：
  [状态1] — [触发条件] — [恢复路径]
  [状态2] — [触发条件] — [恢复路径]
  ...
  排除状态及理由：[如有]

必测视口：
  [视口1] [分辨率] — [设备类型] — [是否设计基准]
  [视口2] [分辨率] — [设备类型]
  [视口3] [分辨率] — [设备类型]
  断点视口：[如有]

关键 ROI：
  [ROI1] — [选择理由] — [对应门禁]
  [ROI2] — [选择理由] — [对应门禁]
  ...

特殊注意事项：
  [注意1] — [风险等级] — [决策ID]
  [注意2] — [风险等级] — [决策ID]
  ...
```

### D.6 配置质量自检

配置完成后，用以下问题自检配置质量：

```text
□ P0 状态集是否覆盖了所有风险断点类别（涉及的部分）？
□ 排除的状态是否都有"架构上不涉及"的理由（而非"首版不需要"）？
□ 必测视口是否至少包含设计基准、标准工作、最窄支持三个？
□ 关键 ROI 是否覆盖了用户核心操作路径？
□ 关键 ROI 数量是否在 3-7 个之间？
□ 特殊注意事项是否覆盖了边界数据、权限、业务规则、交互复杂度四类？
□ 每条特殊注意事项是否都有对应的设计决策 ID？
□ 该配置是否由人类在 S0 确认（不得由 AI 自行配置后直接执行）？
```

特定业务场景的预设配置不属于本规范范畴，应由各项目团队在执行指导书中根据本附录方法自行沉淀和维护。
