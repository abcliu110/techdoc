# 知识库建设与持续完善执行工作流

> 版本：v0.9
> 日期：2026-07-06
> 总纲关系：本文属于《00-总纲-知识库与产品交付双循环研发流程.md》的循环 1（知识供给循环）。知识库只做三件事：采证据、做抽象、出候选；唯一出口是"最佳设计决策输入包"；不为任何具体新软件做最终设计决策。AI 执行本文时必须遵守总纲第 6 章"AI 自动执行契约"。
> 适用范围：围绕某个业务领域或产品方向，通过竞品官网、产品截图、试用实测、开源项目源码和人工/AI 分析，建设可支撑自研产品设计的领域竞品知识库。
> 核心原则：竞品只是证据来源，知识库最终沉淀的是领域模型、业务规则、设计原则、竞品优缺点和自研产品机会。
> 文档关系：本文是《01-方法论-多竞品逆向学习与产品设计知识库》的完整执行版，覆盖新建知识库和存量知识库增量完善流程；执行本文不得漏掉方法论中的强制要求。
> 风险清单关系：若知识库用于产品设计，执行本文时必须同步参考《03-方法论-系统级失败模式与设计约束清单.md》《03A-方法论-失败模式与设计约束清单建设.md》《04-清单-通用系统级失败模式与设计约束.md》《04A-清单-复杂平台级失败模式与设计约束.md》，维护领域失败模式与设计约束清单。

## 0. 执行入口

实际执行一轮竞品知识库建设或完善时，优先按本文逐阶段执行。

执行者不需要先把方法论文档重写成计划；本文已经把方法论中的原则、证据要求、知识库结构、拆解流程、质量标准、合规要求、版本治理、AI 辅助边界、工具承载、量化评估和产物清单转成可执行流程。

使用规则：

```text
日常执行：
读本文，按阶段跑。

完整性要求：
方法论中的每个强制要求都必须执行；确实不适用的要求，必须在成熟度声明或缺口清单中标注“不适用”及理由。

遇到证据强度、成熟度、Schema、合规、版本治理、AI/CI、产品决策标准不明确的地方：
回查《01-方法论-多竞品逆向学习与产品设计知识库》，并以方法论为准。

如果知识库用于产品设计：
回查《03-方法论-系统级失败模式与设计约束清单.md》和《03A-方法论-失败模式与设计约束清单建设.md》，并以《04-清单-通用系统级失败模式与设计约束.md》和《04A-清单-复杂平台级失败模式与设计约束.md》作为领域失败模式清单的起点。
```

给 AI 或人机协作团队的启动提示：

```text
请按《01A-工作流-知识库建设与持续完善.md》执行本轮竞品知识库建设或完善。
我只指定知识库名称；你需要自行检查该知识库是否已经存在。
如果不存在，按本文流程新建知识库。
如果已经存在，先评估现有成熟度、缺口和过期内容，再按本文流程增量完善。
如遇到证据强度、成熟度、Schema、合规、版本治理、AI/CI 或产品决策标准不明确的地方，回查《01-方法论-多竞品逆向学习与产品设计知识库.md》，并以方法论为准。
如果本知识库将用于产品设计、架构设计、详细设计、编码规范或测试规格，必须同步维护领域失败模式与设计约束清单；竞品缺点、源码边界、错误提示、社区 issue、用户反馈和事故线索必须转成失败模式候选卡、非目标、风险接受或 ADR。
如果本知识库将用于软件架构、企业软件架构、领域模型、设计模式、详细设计、UI 设计或 UI 组件设计，必须同步建立或更新行业标准与最佳实践库；不得只从竞品反推设计，也不得只凭自身经验生成设计。
执行完成前，必须按本文“方法论完整覆盖检查”逐项确认；不得只执行最小流程后声明完成。
```

### 0.1 最小输入契约

启动本工作流时，用户只需要提供知识库名称。

约定知识库根目录：

```text
../01-知识库/
```

路径基准是本文所在目录：

```text
00-方法论与流程/
```

因此，某个知识库的目标目录必须是：

```text
../01-知识库/{知识库名称}/
```

示例：

```text
用户输入：低代码平台竞品知识库
目标目录：../01-知识库/低代码平台竞品知识库/
```

执行者必须自行完成存在性检查：

```text
输入：知识库名称
→ 在 ../01-知识库/{知识库名称}/ 查找同名知识库
→ 不存在：创建新知识库
→ 已存在：进入存量知识库完善流程
```

不得要求用户先判断“新建还是完善”。这是执行工作流的内部职责。

### 0.2 新建与完善分支

新建知识库：

```text
创建目录结构
→ 生成 README / 领域定义 / 研究范围 / 术语表
→ 进入阶段 1：竞品与开源项目筛选
→ 按阶段 2-11 完成第一轮建设
```

完善已有知识库：

```text
读取现有 README、成熟度声明、能力矩阵、缺口清单、证据卡和 ADR
→ 判断当前成熟度等级
→ 复核领域边界、竞品分级和证据模型是否仍可用
→ 找出证据不足、过期证据、未验证假设、缺失模块和未闭环决策
→ 生成本轮补证计划
→ 按阶段 0-2 补齐缺失基础资料
→ 按阶段 3-11 定向补证、修正结论、更新矩阵和回写知识库
```

如果已有知识库缺少成熟度声明、能力矩阵或缺口清单，先把这些基础治理文件补齐，再进入补证。完善已有知识库时，禁止推倒重来；除非证据链、目录结构或 Schema 已经错误到无法追溯，否则默认做增量完善。

### 0.3 再次学习与增量深化

当用户说“再次学习”“重新学习”“继续学习”“更新知识库”“学习金蝶低代码平台和 GitHub 开源项目”等表达时，必须进入存量知识库完善流程，而不是只复读旧结论或只整理已有卡片。

再次学习的目标是提高知识库成熟度，核心手段是：

```text
扩大竞品与开源项目候选池
→ 重新分级 S/A/B
→ 对重点对象增加证据渠道
→ 对开源项目做源码级阅读
→ 对商业闭源竞品深挖公开文档和可见界面证据
→ 更新能力矩阵、成熟度、缺口清单和产品决策
```

执行者必须先检查现有知识库里已经覆盖了哪些竞品、开源项目、模块和证据渠道，然后输出本轮增量深化计划。

增量深化计划至少包含：

```text
本轮新增候选竞品 / 开源项目清单
本轮升级为 S/A 级深拆的对象
每个对象新增的证据渠道
每个开源项目要阅读的源码模块
每个商业竞品要深挖的公开资料范围
预计更新哪些矩阵、卡片、ADR、缺口清单
```

对于 GitHub 开源项目，禁止只读 README、官网介绍或截图后就写结论。凡是进入 S/A 级的开源项目，必须执行源码级阅读：

```text
clone 或读取仓库源码
→ 阅读目录结构和核心模块
→ 阅读数据模型 / Schema / 元数据定义
→ 阅读权限、角色、租户或访问控制逻辑
→ 阅读流程、状态机、自动化或工作流实现
→ 阅读插件、扩展点、连接器或集成机制
→ 阅读前端页面 / 渲染器 / 组件配置机制
→ 阅读 API、服务层、持久化层和测试 / 示例
→ 阅读 Issue、Release Note、Changelog 里的关键设计变化
→ 生成源码证据卡和机制拆解卡
```

源码阅读不是为了复制代码，而是为了理解机制、边界、取舍和失败模式。开源项目如果没有源码级证据，不能作为 S/A 级深度参考，只能降级为 B 级方向参考。

对于金蝶等商业闭源低代码平台，不能因无法读取源码就停留在官网摘要。必须深挖所有合法公开资料：

```text
官网产品页
帮助中心 / 用户手册
开发者文档 / API 文档
产品白皮书 / 解决方案
公开演示 / 视频 / 直播课 / 培训资料
更新日志 / Release Note
客户案例 / 行业方案
部署、安全、权限、集成、扩展相关公开资料
公开截图、Demo、公开视频中的页面与交互证据
社区、公开问答、招聘 JD、技术文章中的侧面证据
```

商业闭源竞品的内部机制结论必须按证据强度标注为“直接事实 / 高可信推断 / 假设”。仅由官方宣传支持的单竞品能力断言，不得写成成熟能力；必须进入待验证清单或通过其他独立来源交叉验证。

再次学习完成前，必须回答：

```text
本轮比上一轮新增了哪些竞品或开源项目？
哪些开源项目完成了源码级阅读？
金蝶等重点商业竞品新增了哪些公开资料证据？
哪些旧结论被升级、修正或降级？
哪些能力矩阵格子的证据强度发生变化？
下一轮还需要增加哪些项目或深挖哪些模块？
```

### 0.4 功能专题深挖

当用户指定“低代码平台的某一个功能”并要求深挖金蝶低代码平台和 GitHub 开源项目时，必须进入功能专题深挖分支。

功能专题可以是但不限于：

```text
数据建模 / 元数据模型
页面设计器 / 表单设计器 / 列表设计器
权限与组织 / 多租户 / 数据范围
工作流 / 审批流 / 自动化
插件机制 / 扩展点 / 连接器
发布版本 / 应用打包 / 环境迁移
导入导出 / 数据集成
表达式 / 脚本 / 规则引擎
报表 / 指标 / 看板
移动端 / 多端适配
```

执行者必须先把用户给出的功能名归一化为专题名称，并检查现有知识库中该专题已有的证据、对象、规则、状态机、UI 模式、源码证据、能力矩阵和 ADR。

功能专题深挖必须输出专题计划：

```text
专题名称：
专题边界：
不研究的相邻功能：
金蝶公开资料深挖范围：
GitHub 开源项目候选清单：
本轮 S/A 级深拆项目：
每个开源项目要阅读的源码模块：
需要补的页面 / 流程 / 对象 / 状态机 / 规则 / 权限 / 指标 / 非功能性卡片：
预计影响的能力矩阵和 ADR：
```

对金蝶等商业闭源低代码平台，专题深挖必须按该功能定向采集公开资料：

```text
该功能在官网 / 白皮书 / 解决方案中的定位
该功能在帮助中心 / 用户手册中的操作路径
该功能在开发者文档 / API 文档中的扩展点和接口
该功能在公开视频 / Demo / 截图中的页面结构和交互
该功能在更新日志中的演进记录
该功能与权限、租户、部署、集成、安全相关的公开说明
该功能在客户案例、社区问答、技术文章、招聘 JD 中的侧面证据
```

对 GitHub 开源项目，专题深挖必须围绕该功能阅读源码，而不是泛读仓库：

```text
定位该功能相关目录和模块
阅读该功能的数据模型 / Schema / 元数据结构
阅读该功能的服务层、API、权限校验和持久化逻辑
阅读该功能的前端页面、配置器、渲染器或组件实现
阅读该功能的状态流转、工作流、事件、插件或扩展点
阅读该功能的测试、示例、模板和默认配置
阅读该功能相关 Issue、PR、Release Note、Changelog
生成该功能的源码证据卡、机制拆解卡和失败模式卡
```

专题深挖完成前，必须更新或新增：

```text
专题证据卡
专题页面拆解卡
专题流程卡
专题对象 / 数据模型卡
专题状态机卡
专题业务规则卡
专题权限模型卡
专题 UI 模式卡
专题非功能性卡
专题能力矩阵行 / 列
专题机会点和风险清单
专题 ADR 或 ADR 草案
专题缺口清单和下一轮补证计划
```

专题深挖完成前，必须回答：

```text
金蝶在该功能上有哪些直接事实证据？
哪些结论只是高可信推断或假设？
哪些 GitHub 开源项目完成了该功能的源码级阅读？
各项目在该功能上的对象模型、流程、权限、扩展点和取舍有什么差异？
哪些设计原则可以迁移到自研产品？
哪些竞品做法不能照搬，原因是什么？
该专题还缺哪些证据，下一轮如何补？
```

禁止只把该功能做成竞品功能清单。专题结论必须沉淀为领域对象、流程、规则、状态机、权限边界、UI 模式、设计取舍、风险和 ADR 输入。

### 0.5 方法论完整覆盖检查

本文是方法论的执行版，不是方法论的摘要。执行完成前，必须逐项检查《01-方法论-多竞品逆向学习与产品设计知识库.md》的所有章节是否已经落到产物或缺口清单。

覆盖关系如下：

| 方法论章节 | 本文执行落点 | 完成判据 |
|---|---|---|
| 1. 目标 | 阶段 0-11、阶段 15 | 知识库能从证据支撑产品蓝图 / PRD / ADR / 详细设计，而不是只保存资料。 |
| 2. 基本原则 | 总体原则、关键门禁、成熟度评审 | 不抄功能；事实、推断、假设、决策分离；按领域对象组织；功能性与非功能性并重；合规通过；知识原子化、结构化、可机读。 |
| 3. 知识库总体结构 | 推荐目录结构、证据模型设计、领域抽象 | 十二类知识库或其等价目录齐备；缺失项必须写入缺口清单。 |
| 4. 竞品拆解流程 | 阶段 1-8、阶段 10-11 | 完成竞品选择与分级、证据采集、页面/流程/对象/状态机/UI 模式拆解、多竞品对比、验证纠偏。 |
| 5. 驱动产品全新设计 | 阶段 15 | 能输出产品蓝图、模块设计和研发输入；不从页面或代码倒推产品范围。 |
| 6. 驱动已有产品升级 | 0.2 完善已有知识库、阶段 15 | 已有知识库或已有产品场景必须完成现状盘点、差距表、机会表、产品决策表。 |
| 7. 知识库到代码 | 阶段 15、最终产物清单 | 业务对象、状态机、规则、权限、UI、流程、指标、非功能性和 ADR 能映射到研发产物。 |
| 8. 盲区与陷阱 | 成熟度评审、迭代补证 | 看不到后台、烟雾弹、版本错位、幸存者偏差、推断错误、语境盲区、目标用户差异均被检查或列为风险。 |
| 9. 法律与合规边界 | 官网与公开资料采集、源码分析、关键门禁 | 只采集允许来源；不反编译、不入侵、不用泄漏数据；License 与 UI/代码抄袭风险通过自检。 |
| 10. 知识库质量标准 | 成熟度评审、关键门禁 | 成熟度分级明确；知识条目满足可追溯、结构化、可验证、可更新要求。 |
| 11. 角色分工与协作机制 | AI 与人工分工 | AI 草案、人工评审、决策归档职责清楚；高风险决策有人类确认。 |
| 12. 版本与时效管理 | 迭代补证、推荐目录结构、成熟度评审 | 证据有采集时间和版本；过期证据有标记；知识库有变更日志和下一轮更新计划。 |
| 13. AI 辅助拆解 | AI 与人工分工、关键门禁 | AI 只做辅助与草案；不得把 AI 推断当事实；AI 产出必须可追溯、可校验。 |
| 14. 工具与承载形式 | 证据模型设计、推荐目录结构 | 采用 Markdown / Git / Schema / 图谱 / RAG / CI 时，结构、命名、元数据和检索关系清楚。 |
| 15. 量化评估与优先级 | 能力矩阵、优缺点与机会提炼 | 能力评分、机会优先级、差距严重度、RICE / Kano 或等价评估已完成；未采用的竞品共性有理由。 |
| 16. 推荐工作流 | 阶段 1-15、最小启动顺序 | 单竞品拆解、多竞品融合、自研设计、持续反馈均已覆盖。 |
| 17. 最终产物清单 | 完整产物验收 | 26 类最终产物已交付或明确标注不适用 / 待补。 |
| 18. 一句话总结 | 成熟度声明、README | README 能说明知识库如何从竞品学习转化为产品设计决策系统。 |
| 附录 A. 轻量级快速启动指南 | 首轮启动顺序、方法论完整覆盖检查 | 可用附录 A 启动，但完成前必须扩展到完整方法论覆盖，或把未覆盖项写入缺口清单。 |

检查规则：

```text
每个方法论章节
→ 已执行：标注对应产物路径
→ 不适用：说明原因
→ 未完成：写入缺口清单和下一轮补证计划
```

禁止把“已阅读方法论”当成“已执行方法论”。执行证据必须落到知识库文件、矩阵、卡片、ADR、缺口清单或评审记录中。

## 1. 建设目标

领域竞品知识库不是资料夹，也不是链接清单。它的目标是把分散资料转化为可复用、可追溯、可评审、可用于产品设计的领域知识。

最终应支撑：

```text
领域定义
→ 竞品筛选
→ 多来源证据采集
→ 截图拆解 / 实测验证 / 源码分析
→ 领域抽象
→ 能力矩阵
→ 优缺点提炼
→ 自研产品机会
→ 成熟度评审
→ 缺口清单
→ 下一轮定向补证
→ 知识库回写
→ 产品蓝图 / PRD / ADR / 详细设计
```

## 2. 总体原则

### 2.1 按领域建库

知识库必须围绕领域建立，而不是围绕零散竞品建立。

```text
业务领域 / 产品方向
→ 领域问题空间
→ 竞品和开源项目
→ 证据采集
→ 领域抽象
→ 产品机会
```

例如：

```text
领域：企业业务低代码平台
核心问题：业务对象建模、页面生成、权限、流程、插件、发布治理
竞品：金蝶云苍穹、Frappe、NocoBase、Appsmith、Directus
开源项目：Frappe、NocoBase、Directus、Appsmith
```

### 2.2 证据先于结论

任何知识库结论都必须能追溯到证据。

```text
没有证据卡，不下结论。
没有截图拆解，不评价交互。
没有实测，不把能力标为成熟。
没有源码或实测，不做单向门架构决策。
```

### 2.3 吸收优点，避免缺点

知识库的最终目的不是抄竞品，而是：

```text
吸收竞品优点
避免竞品缺点
识别竞品边界
形成更有竞争力的自研产品
```

### 2.4 多来源、多轮次、循环补证

领域竞品知识库不是一次性调研报告，而是持续演化的证据系统。

同一个关键结论应尽量经过多来源交叉验证：

```text
官网 / 官方文档
→ 产品截图 / Demo / 视频
→ 试用实测
→ 源码 / API / 配置
→ 社区 / Issue / Release Note / 用户反馈
→ 人工评审
```

每一轮建设都必须形成闭环：

```text
本轮证据
→ 本轮结论
→ 本轮缺口
→ 下一轮补证计划
→ 新证据回写
→ 结论修正或升级
```

禁止把第一轮采集结果视为最终结论。竞品版本变化、实测发现、源码证据、用户反馈和人工评审都可能触发知识库更新。

## 4. 阶段 1：竞品与开源项目筛选

不是所有竞品都同等研究。建议按研究深度分级。

| 等级 | 研究深度 | 适用对象 |
|---|---|---|
| S 级 | 官网 + 文档 + 截图 + 实测 + 源码 | 直接影响自研产品核心设计的竞品或开源项目 |
| A 级 | 官网 + 文档 + 截图 + 试用 + 部分源码 | 重要参考对象 |
| B 级 | 官网 + 文档 + 能力矩阵 | 方向参考 |
| C 级 | 趋势记录 | 只观察，不深入 |

输出文件：

```text
02-competitors/竞品清单.md
02-competitors/竞品分级.md
```

竞品清单字段：

| 字段 | 说明 |
|---|---|
| 竞品名称 | 产品或项目名称 |
| 类型 | 商业产品 / 开源项目 / SaaS / 私有化 / 平台 |
| 等级 | S / A / B / C |
| 研究原因 | 为什么值得研究 |
| 证据来源 | 官网、文档、截图、试用、源码、社区 |
| License 风险 | 是否可参考源码、是否有竞品限制 |
| 预期借鉴点 | 要学习什么 |
| 不宜照搬点 | 要避免什么 |

人工门禁：

```text
S/A 级对象必须说明研究价值。
License 不清楚的开源项目不能进入源码借鉴。
```


证据强度分级：

| 等级 | 含义 |
|---|---|
| 直接事实 | 官方文档、源码、实测截图、运行日志、产品导出 |
| 源码初证 | 已读源码，但尚未运行验证 |
| 高可信推断 | 多来源一致，但缺源码或实测 |

```text

弱证据不能支撑产品设计。
```

目标：为架构设计、详细设计、UI 设计和 UI 组件设计建立独立于竞品观察的行业基准。

采集范围：


推荐起点：

```text

企业软件架构：分层架构、模块化单体、六边形架构、DDD 分层、事件驱动、微服务、插件化架构

```

执行规则：

```text

对每条最佳实践标注采纳、部分采纳、拒绝或待验证。
拒绝或偏离行业最佳实践时，必须写清自身约束和代价。
与竞品观察冲突时，优先进入 ADR，而不是直接选择竞品做法。
```

```text
每个候选方案必须说明收益、代价、风险、拒绝理由和需要补充的验证。

知识库阶段不得定稿具体新软件的架构设计、详细设计、UI 规范或组件规范。
```

```text

汇总架构、数据模型、业务规则、权限、流程、UI/组件等关键设计点的：
  候选输入 + 证据 + 行业对照 + 风险 + 待验证问题。

最终取舍不在本工作流执行；必须移交《02-流程-知识库驱动新产品创建与迭代.md》
的"统一设计决策工作流"（见《00-总纲》第 5 章），在具体产品语境下完成三路候选比较、
权衡定稿和 accepted ADR 归档。
```

```text
样例场景：
操作步骤：
截图或录屏：
成功路径：
失败路径：
限制：
体验问题：
与官网描述是否一致：
可借鉴点：
不宜照搬点：
```

门禁：

```text
S 级竞品没有统一样例实测，不能标记为成熟参考。
实测失败必须记录原因，不允许只记录成功路径。
```

## 9. 阶段 6：开源项目源码分析

源码分析不是为了复制代码，而是为了理解机制、边界和取舍。

优先分析模块：

```text
数据模型
权限模型
页面 schema
工作流
插件机制
迁移机制
表达式/脚本
API 层
审计日志
多租户隔离
License 边界
```

源码证据卡模板：

```text
证据编号：
项目：
版本 / commit：
文件路径：
类 / 函数 / 配置：
它解决的问题：
关键字段：
调用关系：
设计取舍：
可借鉴点：
不适合照搬点：
License 风险：
证据强度：源码初证 / 直接事实
```

输出目录：

```text
01-evidence/源码证据/
05-data-model/
06-state-machine/
08-permission/
12-non-functional/
```

门禁：

```text
只读源码不能等同于实测。
License 受限项目只能抽象思想，不能复制代码。
源码结论进入产品设计前要标注“源码初证”。
```

## 10. 阶段 7：领域抽象

这是知识库建设中最关键的一步。目标是把竞品现象变成领域知识。

从证据中抽象：

```text
业务对象库
流程库
状态机库
规则库
权限模型库
页面模式库
指标库
非功能要求库
痛点库
设计原则库
架构模式库
```

输出目录：

```text
02-business-design/
04-process/
05-data-model/
06-state-machine/
07-business-rule/
08-permission/
09-pain-point/
11-metrics/
12-non-functional/
```

抽象示例：

```text
竞品现象：
Frappe DocType 同时包含 fields 和 permissions。

领域抽象：
业务对象不只是表结构，还应内聚字段、权限、页面、状态和规则引用。
```

```text
竞品现象：
NocoBase Collection 支持 relation 字段。

领域抽象：
关系必须成为一等元数据，而不是页面查询配置。
```

门禁：

```text
每条领域抽象必须引用证据卡。
无法追溯证据的抽象只能标记为假设。
```

## 11. 阶段 8：能力矩阵

能力矩阵用于比较竞品、识别机会，不是简单打分。

输出：

```text
matrices/领域能力矩阵.md
matrices/证据成熟度矩阵.md
```

常见能力维度：

```text
业务建模
页面构建
数据源连接
查询绑定
权限
流程
插件
多租户
发布版本
导入导出
License
部署
运维
可观测
安全合规
```

评分建议：

| 分数 | 标准 |
|---|---|
| ★★★★★ | 直接事实 + 实测验证 |
| ★★★★ | 直接事实 |
| ★★★ | 源码初证 |
| ★★ | 高可信推断 |
| ★ | 弱证据 |
| - | 未体现或未验证 |

门禁：

```text
能力矩阵每个高分项必须能追溯到证据卡。
能力矩阵必须标注“未验证”。
```

## 12. 阶段 9：优缺点与机会提炼

针对每个竞品生成：

```text
值得学习的优点
必须避免的缺点
不适合照搬的地方
可迁移的设计原则
自研产品机会
```

输出：

```text
09-pain-point/
10-decision/
Opportunity-Backlog.md
Opportunity-Scorecard.md
```

机会评分维度：

| 维度 | 说明 |
|---|---|
| 客户价值 | 是否解决真实问题 |
| 竞品缺口 | 是否明显优于竞品 |
| 实现难度 | 是否可在首版落地 |
| 证据成熟度 | 是否有足够证据 |
| 商业风险 | License、定价、竞争限制 |
| 安全风险 | 权限、租户、密钥、数据 |

门禁：

```text
竞品优点必须转成设计原则，不能直接转成需求。
竞品缺点必须转成风险清单、非目标或改进机会。
```

其中会造成数据损坏、越权、停机、性能崩溃、合规风险或长期演进腐化的缺点，必须按《03-方法论-系统级失败模式与设计约束清单.md》和《03A-方法论-失败模式与设计约束清单建设.md》转成失败模式候选卡，并检查是否需要更新领域失败模式与设计约束清单。

## 13. 阶段 10：知识库成熟度评审

每轮知识库都要评审成熟度。

成熟度等级：

| 等级 | 定义 |
|---|---|
| L0 | 方向级资料，有官网和公开文档 |
| L1 | 有截图拆解和初步能力矩阵 |
| L2 | 有统一样例实测 |
| L3 | 有源码证据和运行时验证 |
| L4 | 有跨竞品对比、失败模式、设计原则 |
| L5 | 能稳定支撑产品 PRD、ADR、详细设计 |

输出：

```text
README.md
本轮缺口清单.md
S级DoD进度表.md
```

每轮必须回答：

```text
哪些结论是直接事实？
哪些只是推断？
哪些能力还没实测？
哪些源码还没读？
哪些结论不能用于产品决策？
下一轮补证优先级是什么？
```

## 14. 阶段 11：知识库迭代补证

如果对知识库不满意，不是继续无目标搜索，而是进入定向补证。

触发条件：

```text
能力矩阵证据不足
PRD 需要但知识库没有
ADR 决策证据不足
竞品截图无法解释机制
源码结论与实测冲突
人工评审质疑
```

补证路径：

```text
官网补采
产品截图补采
试用环境实测
源码定向阅读
社区 / Issue / Release Note 查证
竞品版本差异确认
License 复核
```

补证后必须：

```text
新增证据卡
更新能力矩阵
更新成熟度
更新缺口清单
更新设计原则
```

## 15. 知识库到自研产品的交接

知识库成熟后，不能直接抄竞品功能，也不在知识库侧继续推进产品设计。本工作流到"最佳设计决策输入包"为止；后续全部产品活动移交《02-流程-知识库驱动新产品创建与迭代.md》按 V 模型执行。

```text
循环 1（本工作流）：
领域知识 → 产品机会候选 → 最佳设计决策输入包
                              │ 交接
                              ▼
循环 2（02-流程，V 模型）：
P1 业务需求 → P2 需求规格 → P3 架构设计 → P4 详细设计
→ 实现与分层验证 → P6 验收上线 → P7 复盘
                              │ 证据回写
                              ▼
循环 1：新证据 → 结论修正 → 知识库升级
```

转化规则：

```text
吸收竞品优点 → 抽象成设计原则
避免竞品缺点 → 写入风险清单和非目标
竞品未验证能力 → 进入待验证假设
竞品 License 受限 → 只能借鉴思想，不复制代码和 UI 表达
```

## 17. 首轮启动顺序

第一轮建议按以下顺序启动。该顺序只是启动顺序，不是删减版方法论；完成前仍必须通过“0.5 方法论完整覆盖检查”。

```text
1. 定义领域
2. 选 3 个核心竞品 + 2 个开源项目
3. 采集官网和官方文档
4. 拆解 10 个关键页面截图
5. 做一个统一样例实测
6. 阅读 3 个核心源码模块
7. 建能力矩阵
8. 提炼优缺点和机会
9. 写成熟度声明和缺口清单
10. 用知识库生成产品蓝图
11. 按方法论完整覆盖检查补齐缺口或标注不适用
```

## 18. 关键门禁

```text
没有领域定义，不建库。
没有证据卡，不下结论。
没有截图拆解，不谈交互优劣。
没有实测，不把能力写成成熟参考。
没有源码或实测，不做单向门架构决策。
没有 License 复核，不引用开源项目实现。
没有缺口清单，不允许说知识库完成。
没有回写机制，知识库会很快过时。
没有通过方法论完整覆盖检查，不允许说本轮完成。
```

## 19. AI 与人工分工

AI 适合做：

```text
资料读取
证据卡草案
截图拆解草案
源码结构梳理
能力矩阵初稿
优缺点提炼
缺口分析
模板生成
一致性检查
```

人工必须做：

```text
领域边界确认
竞品优先级确认
试用账号和环境确认
截图真实性确认
源码 License 判断
产品机会选择
风险接受
最终产品决策
```

AI 禁止：

```text
把营销话术写成直接事实
把未验证能力写成成熟能力
绕过 License 复核引用源码
绕过人工决策生成产品范围
```

## 21. 一句话压缩

领域竞品知识库建设的本质，是把“看竞品”升级成一条有领域边界、有证据卡、有截图拆解、有实测验证、有源码分析、有成熟度门禁、有缺口回写的知识生产线，并用这条知识线支撑更有竞争力的自研产品。
