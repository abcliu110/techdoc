## 术语表

| 术语 | 含义 |
|---|---|
| 证据 | 竞品公开资料中的原始材料，包括截图、文档、视频等 |
| 直接事实 | 截图、文档、视频中明确出现的内容 |
| 高可信推断 | 多个证据共同支持的推断 |
| 假设 | 暂未验证的推断，只能作为后续验证方向 |
| 设计判断 | 基于事实和推断形成的产品决策 |
| 领域对象 | 业务核心实体，如订单、客户、商品 |
| 状态机 | 对象状态与流转规则的集合 |
| ADR | Architecture Decision Record，产品决策记录 |
| 证据强度 | 知识条目的可信程度分级 |
| 非功能性维度 | 性能、并发、缓存、国际化、可访问性等不直接对应业务功能的设计维度 |
| 知识卡片 | 原子化的知识条目，一卡只说一件事，带结构化元数据，是知识库的最小单元 |
| 知识图谱 | 以知识卡片为节点、以证据支持/引用/依赖为边构建的关系网络，用于追溯与影响分析 |
| RAG | Retrieval-Augmented Generation，检索增强生成，让 AI 基于知识库内容回答与产出 |
| GraphRAG | 在 RAG 基础上沿知识图谱扩展关联条目作为上下文的检索方式 |
| JTBD | Jobs to be Done，以"情境-动机-期望结果"表述用户任务的方法 |
| 事件风暴 | Event Storming，通过梳理领域事件反推命令与聚合的建模方法 |
| 聚合 | DDD 术语，一组被整体创建、修改、删除的对象群，聚合根决定事务与接口边界 |

## 1. 目标

本文定义一套面向产品全新设计与升级的竞品学习方法。

核心目标不是简单记录"竞品有什么功能"，而是把多个竞品的公开资料、社区文档、产品截图、演示视频和用户反馈，系统拆解成可复用的产品设计知识库，再用该知识库驱动自己的业务设计、UI 设计、数据模型、业务规则、研发实现和持续升级。

最终形成的不是资料库，而是产品设计决策系统。

```text
竞品公开资料
→ 证据采集
→ 页面/流程拆解
→ 业务对象抽取
→ 数据模型推断
→ UI 模式抽取
→ 业务规则/状态机抽取
→ 多竞品横向对比
→ 设计原则沉淀
→ 自己产品方案生成
→ 代码实现与验证
→ 用户反馈反哺知识库
```

## 2. 基本原则

### 2.1 竞品学习不是抄功能

竞品分析的低级形态是功能清单：

```text
竞品 A 有功能 1、功能 2、功能 3。
竞品 B 有功能 1、功能 4、功能 5。
```

这种材料不能直接指导产品设计，因为它回答不了：

- 这个功能解决什么问题？
- 它依赖哪些业务对象？
- 它背后有哪些状态和规则？
- 它的 UI 为什么这样组织？
- 它牺牲了什么？
- 用户是否仍有痛点？
- 我们应该借鉴、改进，还是放弃？

正确方式是把竞品表象转化为设计判断：

```text
竞品观察
→ 业务意图
→ 对象模型
→ 状态和规则
→ UI 模式
→ 设计取舍
→ 我们的产品决策
```

### 2.2 必须区分事实、推断和决策

竞品没有源码时，不能把推断写成事实。知识库中每条知识必须标注证据强度。

| 类型 | 含义 | 示例 |
|---|---|---|
| 直接事实 | 截图、文档、视频中明确出现的内容 | 订单列表出现"订单号、状态、金额、退款按钮" |
| 高可信推断 | 多个证据共同支持的推断 | 系统存在订单对象，订单有状态流转 |
| 假设 | 暂未验证，只能作为后续验证方向 | 可能存在独立的退款表或退款记录对象 |
| 已验证 | 经实测、试用、灰度或用户访谈验证的推断 | 实际试用确认退款直接到账，无审批环节 |
| 设计判断 | 基于事实和推断形成的产品决策 | 我们首版需要独立退款记录，而不是只在订单上记录退款状态 |

说明：前四类是知识卡片的证据强度，描述的是"这条知识有多可信"；"设计判断"不是知识卡片强度，而是产品决策记录（ADR）的属性，放在此表是为了说明从证据到决策的完整链条。第 14.5 节 Schema 的 strength 字段取值为前四类之一。

知识库必须保留证据链：

```text
观察到什么
→ 从中推断出什么
→ 为什么这样推断
→ 对自己产品有什么启发
→ 最终采用什么设计决策
```

在定性分级之外，关键推断建议附加数值置信度（0-1），并遵守交叉验证规则：

```text
单一来源推断：置信度 ≤ 0.5
两个独立来源支持：可升至 0.7
三个及以上独立来源，或经实测/试用验证：方可 ≥ 0.9
```

独立来源指不同渠道（如官方截图与用户社区讨论互为独立），同一渠道多次出现不算独立。数值置信度将直接参与第 15.4 节 RICE 优先级计算，使"仅凭猜测"的机会天然被降权。

两条防钻空子的补充规则：

- 来源独立性按结论的主语计算：对"竞品 X 具备能力 Y"的断言，只有关于竞品 X 的不同渠道证据才算独立来源；引用多个竞品的证据可以支撑"行业共性"结论，但不能提升对任何单个竞品推断的置信度。
- 仅有竞品官方渠道（官网、官方文档、官方宣传）支撑的单竞品能力断言，置信度不得超过 0.6——官方资料天然是营销视角（见 8.2 节烟雾弹），必须有社区、试用或实测类证据交叉后方可更高。

### 2.3 多竞品学习要围绕领域对象，而不是围绕竞品名称

错误组织方式：

```text
竞品 A 分析
竞品 B 分析
竞品 C 分析
```

正确组织方式：

```text
领域对象：订单
- 竞品 A 如何设计订单
- 竞品 B 如何设计订单
- 竞品 C 如何设计订单
- 共同点是什么
- 差异点是什么
- 各自取舍是什么
- 我们应该采用哪种模型
```

最终知识库要围绕业务对象、用户任务、状态机、业务规则、UI 模式、权限、指标、异常场景和设计取舍组织，而不是围绕竞品截图堆材料。

### 2.4 功能性与非功能性维度并重

竞品学习容易只关注"有什么功能"，而忽略性能、并发、缓存策略、离线能力、多端协同、国际化、可访问性、安全策略等非功能性维度。

这些维度对 B 端和平台型产品尤为关键：

- 一个报表页面加载 3 秒和 30 秒，体验天差地别，但截图看不出来。
- 一个支持万人并发的审批流和一个只支持百人并发的审批流，架构完全不同，但功能清单一样。
- 国际化、多时区、多语言的支持程度，决定了产品能否出海。

因此知识库必须同时覆盖功能性和非功能性维度，不能只拆功能不拆质量。

### 2.5 合规是底线，不是可选项

竞品学习必须守住法律和道德边界，详见第 9 章。核心原则：

- 只采集公开资料，不反编译、不入侵、不使用泄漏数据。
- 学设计模式和业务模型，不抄 UI 表达和代码实现。
- 用户数据采集遵守隐私法规。

### 2.6 知识必须原子化、结构化、可机读

传统竞品分析产出长文档，长文档只能被人通读，无法被检索、统计、追溯和被 AI 消费。知识库要成为"决策系统"而非"资料库"，必须满足三个工程属性：

- 原子化：一条知识只说一件事（一个规则、一个状态机、一条证据），通过唯一 ID 互相链接，而不是堆在一篇大文档里。
- 结构化：每条知识带统一元数据（见 14.5 节 Schema），机器可校验、可过滤、可统计。
- 可机读：结构化的知识可以被脚本构建成知识图谱（14.6 节）、被 AI 通过 RAG 消费（13.5 节）、被 CI 自动做质量门禁（14.8 节）。

判断标准很简单：如果无法用一条查询回答"订单对象的所有业务规则中，哪些还是未验证的假设"，说明知识库只是文档堆，不是知识库。

### 2.7 知识库必须沉淀失败模式与设计约束

知识库不只沉淀功能、对象、流程、页面和指标，也必须沉淀失败模式、设计约束、反例、边界条件和风险防线。

当知识库用于新产品设计或已有产品升级时，必须按《03-方法论-系统级失败模式与设计约束清单.md》和《03A-方法论-失败模式与设计约束清单建设.md》同步维护领域实例清单。通用起点是《04-清单-通用系统级失败模式与设计约束.md》；企业级平台、SaaS、低代码、CRM、ERP、工作流和数据平台类产品还必须参考《04A-清单-复杂平台级失败模式与设计约束.md》。

竞品缺点、源码限制、错误提示、用户反馈、社区 issue、事故复盘和不可验证假设，都不能只写成“待观察”，必须进入以下三类产物之一：

- 失败模式候选卡
- 领域失败模式与设计约束清单
- 非目标、风险接受或 ADR

### 2.8 设计不能只来自竞品，还必须对照行业标准和最佳实践

竞品学习只能说明“别人怎么做”，不能自动证明“这是当前最好的设计”。当知识库用于软件架构、领域模型、设计模式、详细设计、UI 设计或 UI 组件设计时，必须额外建立“行业标准与最佳实践”证据链。

必须覆盖：

```text
软件架构：架构描述标准、质量属性、可演进性、可观测性、安全边界、接口契约、部署与运维模式。
企业软件架构：分层架构、模块化单体、六边形架构、DDD 分层、事件驱动、微服务、插件化、权限体系、组织模型、审计、报表、配置、集成和扩展机制。
领域模型：限界上下文、聚合、实体、值对象、领域服务、领域事件、一致性边界、生命周期和不变量。
企业软件模式：企业应用架构模式、领域模式、集成模式、事务脚本、表模块、领域模型、活动记录、数据映射器、Repository、Unit of Work、CQRS、Saga、Outbox、规格模式、策略、状态、模板方法。
设计模式：架构模式、企业应用模式、集成模式、领域模式、并发与一致性模式、前端状态与组件模式。
详细设计：模块边界、领域模型、状态机、接口契约、错误处理、幂等、并发、数据一致性、测试策略。
UI 设计：目标平台设计系统、信息架构、表单、列表、导航、反馈、空状态、错误状态、响应式和可访问性。
UI 组件设计：组件语义、键盘交互、焦点管理、ARIA 语义、设计 token、状态组合、禁用/加载/错误/只读状态。
```

来源优先级：

```text
1. 官方标准或规范：如 ISO/IEC/IEEE、W3C、OpenAPI、OWASP 等。
2. 官方设计系统和平台指南：如 Material Design、Apple Human Interface Guidelines、Microsoft Fluent 等。
3. 经典软件设计知识体系：如 DDD、企业应用架构模式、企业集成模式、微服务模式、云原生架构模式等。
4. 主流开源项目和成熟组件库的实现实践。
5. 权威技术书籍、论文、工程博客或大型团队公开实践。
6. 竞品观察和社区经验。
```

每条行业最佳实践都必须写清：

```text
来源名称：
来源链接或出处：
检索日期：
版本或发布日期：
适用范围：
核心主张：
对本产品的启发：
采纳 / 部分采纳 / 拒绝：
理由：
落点：PRD / ADR / 架构设计 / 详细设计 / UI 规范 / 组件规范 / 测试规格
```

过期规则：

- 架构、安全、接口、可访问性、平台设计系统类来源，进入关键设计前必须重新确认是否有新版本。
- “最新理念”不能只写成口号，必须转成设计原则、约束、组件规则、ADR 或测试门禁。
- 与自身产品目标、用户群、技术约束冲突的最佳实践可以拒绝，但必须写拒绝理由。
- 没有行业标准或最佳实践对照表的软件架构、领域模型、设计模式、详细设计、UI 设计和组件设计，只能算草案，不能进入实现。

关键设计必须进行候选方案比较，不能只给一个方案：

```text
领域模型至少比较 2 个候选模型：例如以流程为中心、以核心业务对象为中心、以角色/权限为中心、以事件为中心。
企业软件架构至少比较 2 个候选架构：例如分层架构、模块化单体、六边形架构、DDD 分层、事件驱动、微服务、插件化架构或平台内核 + 扩展点。
企业软件模式至少比较 2 个候选模式：例如事务脚本、领域模型、表模块、活动记录、数据映射器、Repository、Unit of Work、规格模式、CQRS、Saga、Outbox 等。
UI 组件设计至少比较 2 个候选组件模式：例如表格 vs 卡片、向导 vs 单页表单、受控组件 vs 非受控组件、集中状态 vs 局部状态。
```

候选比较必须输出：

```text
候选方案：
适用条件：
不适用条件：
解决的问题：
引入的复杂度：
失败模式：
与行业最佳实践的关系：
与竞品实践的关系：
采纳 / 拒绝理由：
最终落点：ADR / 详细设计 / UI规范 / 组件规范 / 测试规格
```

所谓“最佳”只能表示“在当前业务目标、用户规模、团队能力、技术栈、交付周期、演进预期和风险约束下最合适”。不得把某个流行架构、设计模式或 UI 理念无条件写成最佳实践。

架构和设计阶段必须形成权衡矩阵，至少覆盖：

```text
业务目标适配度
领域模型表达力
用户任务与体验适配度
质量属性：性能、可用性、安全性、可维护性、可测试性、可观测性、可扩展性
企业软件共性能力：权限、组织、审计、配置、报表、集成、扩展、数据治理
实现复杂度
团队能力匹配度
交付周期影响
长期演进成本
数据一致性和事务边界
外部依赖和集成风险
失败模式和回滚成本
行业标准或最佳实践符合度
竞品实践支持度
```

权衡矩阵模板：

```text
候选方案：
业务适配：
领域模型适配：
质量属性得失：
企业软件共性能力得失：
UI/组件体验得失：
实现复杂度：
演进成本：
主要风险：
失败模式：
采纳结论：
拒绝其他方案的理由：
必须补充的验证：
承接 ADR：
承接详细设计：
承接测试规格：
```

知识库阶段的候选比较和权衡矩阵只能提供决策输入，不能替具体新软件做最终取舍。没有具体产品目标、用户范围、团队能力、交付周期、技术栈、存量系统和上线约束时，不得把候选方案标记为“最佳设计”，也不得支撑 accepted 状态的 ADR 或详细设计定稿。最终决策必须在具体产品创建流程中完成。

### 3.1 证据库

证据库保存原始材料，是所有分析的地基。

```text
证据编号：E-COMP-A-ORDER-001
竞品：竞品 A
模块：订单管理
证据类型：截图
来源：官网演示页
采集时间：2026-07-05
观察内容：订单列表包含订单号、客户、状态、金额、创建时间、操作按钮
证据强度：直接事实
附件路径：/evidence/competitor-a/order-list.png
```

要求：

- 原始截图、链接、文档片段必须保留。
- 每条证据必须标注来源和采集时间。
- 不在证据库里写产品结论，只写观察事实。

### 3.2 业务设计库

业务设计库回答"这个产品如何理解业务"。

记录内容：

- 用户角色
- 使用场景
- 核心任务
- 模块边界
- 功能结构
- 业务对象关系
- 业务闭环
- 设计取舍

模板：

```text
业务主题：客户跟进
用户角色：销售、销售主管、管理员
核心问题：客户沟通过程容易丢失，后续跟进缺少提醒
竞品共性：客户详情页通常包含跟进记录、联系人、标签、下次跟进时间
差异点：部分竞品强调自动提醒，部分竞品强调手工记录灵活性
设计启发：客户管理不能只保存客户资料，还必须保存跟进行为
关联ADR：ADR-CUSTOMER-001（首版实现客户资料、跟进记录、下次跟进提醒，暂不做复杂自动化营销）
```

核心任务建议用 JTBD 句式表述，避免把功能当任务：

```text
当…（情境），我想要…（动机），以便…（期望结果）。

反例：任务 = "使用跟进记录功能"
正例：当我今天上班打开系统时（情境），我想立即知道该联系哪些客户（动机），以便不遗漏任何承诺过的回访（期望结果）。
```

JTBD 表述的价值：竞品功能只是任务的一种解法。围绕任务组织知识，才能发现"竞品用功能 X 解决，我们可以用更简单的 Y 解决"的差异化机会。

### 3.3 页面与 UI 设计库

UI 设计库不是收藏截图，而是抽取页面模式和交互逻辑。

页面与 UI 设计库必须基于可见界面证据。源码、数据库 Schema、元数据配置和前端组件代码可以解释实现机制，但不能替代截图、录屏、官方 demo、官方文档截图、公开视频或公开产品截图所呈现的信息架构、交互流程和视觉层级。只读源码得到的 UI 结论，应归类为 UI 实现机制证据，不得冒充页面拆解卡。

记录内容：

- 页面目标
- 信息架构
- 布局结构
- 主要操作
- 表单设计
- 列表设计
- 筛选与搜索
- 状态展示
- 空状态和异常状态
- 操作反馈
- 多竞品 UI 差异

模板：

```text
页面：订单列表页
页面目标：帮助运营人员快速定位订单并执行后续操作
主要区域：筛选区、状态标签、列表区、批量操作区、分页区
关键字段：订单号、客户、金额、状态、创建时间、操作
主要操作：查看详情、取消订单、退款、发货
UI 模式：列表 + 筛选 + 状态标签 + 行内操作
适用场景：对象数量大、需要高频检索和处理
设计代价：字段过多时扫描压力大，移动端适配复杂
关联ADR：ADR-ORDER-UI-001（后台端采用高密度列表，移动端采用卡片式摘要）
```

### 3.4 流程库

流程库把页面串成完整业务过程。

模板：

```text
流程：创建订单
角色：客户、商家、系统
入口：商品详情页、后台手动创建
主路径：选择商品 → 填写信息 → 提交订单 → 支付 → 状态变更
异常路径：库存不足、支付失败、订单超时、用户取消
关键状态：待支付、已支付、已取消
关键规则：待支付订单可取消，已支付订单进入履约流程
设计启发：订单创建流程必须同时考虑用户提交、支付结果和超时关闭
关联ADR：ADR-ORDER-FLOW-001（订单创建流程需同时实现用户提交、支付回调、超时关闭三个触发路径）
```

### 3.5 数据模型库

数据模型库从页面字段、文档说明、操作流程和状态变化中反推领域对象。

模板：

```text
对象：Order
来源证据：订单列表、订单详情、支付流程、退款流程
直接字段证据：订单号、客户、金额、状态、创建时间
推断字段：paidAt、cancelledAt、refundStatus
关联对象：Customer、OrderItem、Payment、Refund
对象关系：Order 包含多个 OrderItem；Order 关联一个 Customer；Order 可关联多个 Payment/Refund 记录
生命周期：创建 → 支付 → 履约 → 完成/取消/退款
证据强度：字段为直接事实；对象关系为高可信推断；具体数据库表结构为假设
关联ADR：ADR-ORDER-DM-001（实现 Order、OrderItem、Payment、Refund 四个核心对象，避免把支付和退款信息全部塞入订单表）
```

数据模型推断顺序：

```text
页面名
→ 页面任务
→ 字段
→ 操作按钮
→ 状态
→ 业务对象
→ 对象关系
→ 业务规则
→ 自己产品的数据模型
```

反推建议引入 DDD（领域驱动设计）战术工具，提升模型质量：

| DDD 概念 | 竞品中的观察信号 | 反推价值 |
|---|---|---|
| 限界上下文 | 竞品的一级导航、模块边界、独立子系统 | 划分我们的模块和服务边界 |
| 聚合 | 被整体创建/删除/展示的对象群（订单与订单明细） | 聚合根决定接口粒度和事务边界，避免把明细做成独立入口 |
| 领域事件 | 操作记录、消息通知、状态提示中的动词（"已支付""已发货"） | 事件清单直接导出状态机和异步流程设计 |
| 值对象 | 无独立列表页、总是附属展示的信息（地址、金额） | 避免为附属信息建多余的实体和管理页面 |

配套方法是反向事件风暴（Event Storming）：先从竞品的操作日志、通知文案、状态标签中穷举领域事件，再为每个事件反推"谁触发（命令/角色）、改变了什么（聚合）、引发了什么（后续事件/规则）"。事件视角比字段视角更容易发现隐藏对象——例如看到"退款已到账"事件，即可推断存在独立的退款聚合，而不只是订单上的一个状态字段。

### 3.6 状态机库

状态机库记录核心对象的状态与流转，是从竞品学习系统设计的关键。

模板：

```text
对象：Order
状态：待支付、已支付、已发货、已完成、已取消、退款中、已退款
状态转移：
待支付 → 已支付：支付成功
待支付 → 已取消：用户取消或超时
已支付 → 已发货：商家发货
已支付 → 退款中：用户申请退款
退款中 → 已退款：退款成功
不变量：已取消订单不能再次支付；已退款订单不能重复退款
竞品证据：不同状态下显示不同操作按钮
关联ADR：ADR-ORDER-SM-001（订单状态和退款状态分离，避免单一状态字段承载过多含义）
```

### 3.7 业务规则库

业务规则库记录操作条件、校验逻辑、异常处理和边界约束。

模板：

```text
规则：只有待支付订单允许用户主动取消
证据：竞品 A 待支付状态显示取消按钮，已完成状态不显示；竞品 B 文档说明支付后需申请退款
证据强度：高可信推断
适用边界：管理员后台强制取消场景未验证
关联ADR：ADR-ORDER-RULE-001（普通用户只能取消待支付订单；管理员可在受控条件下关闭异常订单，并记录操作日志）
```

### 3.8 权限模型库

权限模型库记录角色、数据范围和操作边界。

模板：

```text
角色：销售
可见数据：自己负责的客户
可执行操作：新增跟进记录、编辑客户备注、设置下次跟进时间
不可执行操作：删除客户、修改客户归属、查看全部销售客户
竞品证据：页面中存在"负责人""团队客户""全部客户"等筛选项
关联ADR：ADR-CUSTOMER-PERM-001（客户数据按负责人隔离，主管可查看团队范围，管理员可查看全部）
```

### 3.9 痛点与机会库

痛点与机会库记录竞品未解决好的问题，以及自己产品的差异化机会。

模板：

```text
主题：客户跟进提醒
竞品现象：多数竞品支持下次跟进时间，但提醒方式弱，容易被忽略
用户痛点：销售需要知道今天该跟进谁，而不是手动筛选所有客户
机会判断：可以设计"今日待跟进"工作台
优先级：高
原因：直接影响销售日常工作效率
关联ADR：ADR-CUSTOMER-OPP-001（首版首页提供今日待跟进列表，暂不做复杂自动化规则引擎）
```

### 3.10 产品决策库

产品决策库是最终产物，记录自己产品采用什么、不采用什么、为什么。

模板：

```text
决策编号：ADR-CUSTOMER-001
主题：客户跟进记录是否独立建模
背景：多个竞品都在客户详情页展示历史跟进，说明跟进过程是客户管理的核心信息
备选方案：
方案 A：只在客户表保存最后一次跟进信息
方案 B：建立独立 FollowUpRecord 对象
决策：采用方案 B
理由：需要保留完整沟通历史，支持后续统计和提醒
代价：数据模型更复杂，需要额外列表、接口和权限控制
不采用原因：方案 A 简单但无法支持历史追踪
验证方式：销售能查看完整跟进历史，并按下次跟进时间收到提醒
```

说明：本模板是 ADR 卡片的正文内容；决策状态、依据卡片、否决方案等结构化元数据以第 14.5.3 节 ADR 卡片 Schema 为准，两处不重复维护。

### 3.11 指标库

指标库记录竞品报表和看板中体现的统计指标及其计算口径。指标口径是竞品学习的重要产物——报表里统计什么、怎么算，直接反映产品的业务理解和运营导向。

记录内容：

- 指标名称
- 业务含义
- 计算口径
- 展示形式
- 统计周期
- 数据来源
- 竞品证据
- 证据强度
- 我们决策

模板：

```text
指标：客户转化率
业务含义：衡量客户从线索到成交的转化效率
计算口径：已成交客户数 / 新增客户数
展示形式：漏斗图 + 趋势折线
统计周期：按月统计
数据来源：客户状态变更记录
竞品证据：竞品 A 看板中出现"转化率"指标，竞品 B 在客户分析报表中展示
证据强度：高可信推断（计算口径为推断，指标存在为直接事实）
适用边界：不同竞品对"成交"定义可能不同，转化率口径需统一
关联ADR：ADR-CUSTOMER-MT-001（首版实现客户转化率指标，成交定义为客户状态变为"已成交"，按月统计）
```

### 3.12 非功能性设计库

非功能性设计库记录竞品在性能、并发、缓存、离线、多端协同、国际化、可访问性、安全等方面的设计策略。这些维度从截图看不出来，需要从文档、更新日志、用户反馈、技术博客中推断。

记录内容：

- 维度名称
- 竞品表现
- 推断策略
- 证据来源
- 证据强度
- 对用户体验的影响
- 我们的约束
- 我们决策

模板：

```text
维度：报表加载性能
竞品表现：竞品 A 大数据量报表可在 3 秒内加载完成
推断策略：可能采用预聚合 + 异步查询 + 缓存
证据来源：用户社区评论、更新日志中提到"性能优化"
证据强度：假设（无法看到后端实现）
对用户体验的影响：直接影响运营人员日常使用效率
我们的约束：首版数据量较小，可暂不做预聚合，但接口需支持分页和异步
关联ADR：ADR-REPORT-NFR-001（首版采用分页查询 + 按需缓存；数据量增长后引入预聚合任务）
```

常见非功能性维度清单：

- 性能与响应时间
- 并发与吞吐量
- 缓存策略
- 离线能力与弱网处理
- 多端协同与数据同步
- 国际化与多语言
- 多时区处理
- 可访问性
- 安全策略（认证、加密、审计）
- 数据备份与容灾
- 扩展性与集成能力

## 4. 竞品拆解流程

### 4.0 第零步：竞品选择与分级

在采集证据之前，必须先确定拆哪些竞品、为什么拆、拆到什么深度。无选择地堆砌竞品会导致精力分散、重点模糊。

#### 4.0.1 竞品分类

| 类型 | 含义 | 示例 |
|---|---|---|
| 直接竞品 | 同赛道、同目标用户、同核心场景 | 两款 CRM SaaS |
| 间接竞品 | 不同赛道但解决相似用户任务 | CRM 与企业微信客户管理 |
| 跨界参照 | 不同领域但交互模式或架构可借鉴 | 游戏化设计参照游戏产品 |

#### 4.0.2 选择标准

```text
市场地位：是否代表行业主流做法
用户重叠度：与我们的目标用户重叠程度
设计成熟度：产品是否经过充分迭代
可观测性：公开资料是否充足
战略价值：是否对我们的差异化有启发
```

#### 4.0.3 分级策略

| 级别 | 深度 | 投入 |
|---|---|---|
| S 级（重点深拆） | 完整七步拆解 + 持续跟踪 | 全量证据采集、全模块拆解 |
| A 级（标准对照） | 核心模块七步拆解 | 重点模块证据采集 |
| B 级（浅度参照） | 只做能力矩阵和 UI 模式对照 | 截图级采集 |

建议：S 级 1-2 个，A 级 2-3 个，B 级 3-5 个。数量不宜过多，避免稀释深度。

各级别必须满足量化完成度下限（Definition of Done），否则不得声明该级别已拆解完成：

| 项目 | S 级（每核心模块） | A 级（每核心模块） | B 级（整体） |
|---|---|---|---|
| 证据卡片数 | ≥ 20 | ≥ 8 | ≥ 3 |
| 证据渠道种类 | ≥ 3 种（必须含深度渠道，见下表） | ≥ 2 种（建议含深度渠道） | ≥ 1 种 |
| 可见 UI 证据 | 若竞品存在可见界面，页面/UI 模块必含截图、录屏、官方 demo、官方文档截图、公开视频或公开产品截图证据 | 关键页面必须有可见界面证据 | 若做 UI 对照，至少引用公开截图或演示画面 |
| 页面拆解卡 | ≥ 10 | ≥ 5 | 不要求 |
| 数据模型/对象卡（字段级） | ≥ 3 | ≥ 1 | 不要求 |
| 状态机卡 | ≥ 1 | ≥ 1 | 不要求 |
| 业务规则卡 | ≥ 5 | ≥ 2 | 不要求 |

**深度渠道按竞品可观测性类型确定**——不同类型竞品能做到的最深采集手段不同，DoD 按"该类型可达的最深渠道"检查，而不是一律要求安装试用：

| 竞品类型 | 可达的深度渠道（按优先级） | 说明 |
|---|---|---|
| 开源竞品 | ① 源码阅读（clone 仓库，读数据模型定义、权限逻辑、状态流转代码）② 可见界面采集（本地截图/官方 demo/官方文档截图/演示视频/公开产品截图）③ 本地安装实测（可选增强） | 源码是内部机制最强证据（直接事实），AI 可直接执行；页面/UI 模块必须补可见界面证据；安装实测用于验证运行时行为，需要环境时可选 |
| 闭源但可免费试用/有演示环境 | ① 注册试用 + 人工操作截图 ② 演示视频逐帧分析 | 试用操作与截图通常需人工配合，AI 负责拆解截图与录屏 |
| 闭源且无试用渠道 | ① 官方文档/开发者文档深挖 ② 公开演示视频、社区帖、离职员工分享、招聘 JD | 无法实测，所有内部机制结论只能是"高可信推断/假设"，confidence 上限按 2.2 节执行 |

补充规则：

- "字段级"指对象卡必须列出具体字段、类型和关系语义，仅有对象名称清单不算数据模型卡。
- **开源竞品若未做源码级分析即完成拆解，整体级别自动降为 B 级参照**——源码就摆在那里却只读 README 和官网文档，是不可接受的偷懒。安装实测是增强项而非门槛。
- **源码证据不能替代可见 UI 证据**——若竞品有可见界面，页面/UI 模块必须引用截图、录屏、官方 demo、官方文档截图、公开视频或公开产品截图；没有可见界面证据时，页面拆解卡数量不得计入 S/A 级 UI DoD。
- 开源竞品源码阅读是数据模型、权限、状态机、扩展机制等内部机制拆解的强制项；页面/UI 结论还必须补充用户实际能看到的界面证据，二者缺一不可。
- 闭源无试用渠道的竞品不因"缺实测"降级，但其能力矩阵格子必须标"未验证"（4.7.1 节），内部机制类卡片不得写成"直接事实"。
- 未达 DoD 的拆解可以入库，但知识库必须如实声明当前成熟度级别（见 10.1 节），不得据此支撑高级别决策。

#### 4.7.1 多竞品能力矩阵

横向对比必须产出能力矩阵，用于一目了然地比较各竞品在各个维度的覆盖情况。

矩阵模板：

```text
能力维度    | 竞品 A | 竞品 B | 竞品 C | 我们现状 | 我们目标
-----------|--------|--------|--------|---------|--------
客户管理    | ★★★   | ★★★★  | ★★★   | ★★     | ★★★
跟进记录    | ★★★   | ★★★★  | ★★    | ★      | ★★★★
自动提醒    | ★★    | ★★★★  | ★     | -      | ★★★
数据分析    | ★★★   | ★★    | ★★★★  | ★★     | ★★★
移动端支持  | ★★★   | ★★    | ★★★★  | ★★     | ★★★
```

说明：

- ★★★★★ 表示该维度表现优秀且设计成熟。
- ★ 表示有基础支持但体验较弱。
- "-" 表示该维度缺失。
- "我们目标"列不是照搬最高分，而是基于自身定位和资源确定合理目标。

能力矩阵的要求：

- 维度按领域对象或用户任务组织，不按功能清单罗列。
- 评分必须基于证据，不能凭感觉。
- 矩阵中标注"未验证"项，表示证据不足。
- 矩阵定期更新，随竞品版本迭代刷新。

### 4.8 第八步：验证与纠偏

逆向推断天然存在偏差，必须有验证和纠偏环节，否则假设会长期以"事实"身份误导设计。

#### 4.8.1 验证手段

| 验证手段 | 适用场景 | 可信度提升 |
|---|---|---|
| 用户访谈 | 验证业务流程和痛点推断 | 假设 → 高可信推断 |
| 灰度测试 | 验证 UI 模式和交互设计 | 假设/高可信推断 → 已验证；ADR 决策状态 → validated |
| API 探测 | 验证数据模型和接口设计（合规前提下） | 假设 → 直接事实 |
| 试用体验 | 注册竞品试用账号实际操作 | 推断 → 直接事实 |
| 销售沟通 | 通过竞品售前沟通获取信息 | 假设 → 高可信推断 |
| 社区验证 | 在用户社区验证推断是否符合实际 | 假设 → 高可信推断 |

API 探测的合规边界：

- 只能使用竞品公开发布的 API 文档中提供的接口。
- 只能在竞品许可协议允许的范围内使用。
- 不得对竞品接口进行压力测试或安全扫描。
- 不得绕过竞品的访问限制和认证机制。
- 如有疑虑，优先使用其他验证手段。

#### 4.8.2 纠偏机制

```text
发现矛盾证据
→ 标记受影响的知识条目
→ 重新评估证据强度
→ 更新推断或降级为假设
→ 通知依赖该条目的决策
→ 记录纠偏日志
```

纠偏日志模板：

```text
纠偏编号：FIX-001
日期：2026-07-05
原条目：E-COMP-A-ORDER-015（推断订单有独立的退款审批流）
原证据强度：高可信推断
矛盾证据：实际试用发现退款直接到账，无审批环节
纠正后强度：假设（审批可能仅在特定金额以上触发）
影响决策：ADR-REFUND-002（原计划实现退款审批流，现调整为可选审批）
```

## 6. 如何驱动已有产品升级

已有产品升级时，用"现状盘点 - 差距 - 机会 - 决策"模型。

```text
竞品知识库
→ 自身软件现状盘点
→ 差距表
→ 机会表
→ 产品决策表
→ 升级路线图
→ 研发任务
→ 研发实现与验证反馈
```

竞品有拆解方法，自身软件也必须有盘点方法，两者用相同的领域对象和维度对齐，差距分析才可靠。

### 6.0 第零步：自身软件现状盘点

升级前必须先盘清自身软件现状，否则差距分析无根基。自身软件有源码和文档，盘点比竞品拆解更深入，但也更容易陷入细节，需聚焦与竞品对比所需的维度。

#### 6.0.1 盘点原则

- 用与竞品拆解相同的十二类知识库结构盘点自身软件，保证可对齐。
- 自身软件的盘点可信度高于竞品（有源码、有文档、可实测），证据强度可直接标注为"直接事实"。
- 盘点时同步记录技术债务、历史决策原因、已知缺陷，这些是竞品分析中看不到的内部语境。
- 盘点产物单独存放于"自身软件现状库"，不与竞品知识库混在一起。

#### 6.0.2 盘点内容

```text
01-自身证据库：现有页面截图、接口文档、数据库结构、配置项
02-自身业务设计：现有用户角色、场景、任务、模块边界
03-自身页面与UI：现有页面结构、字段、操作、UI 模式
04-自身流程：现有业务流程、主路径、异常路径
05-自身数据模型：现有表结构、对象关系、字段定义
06-自身状态机：现有状态字段、流转逻辑
07-自身业务规则：现有校验逻辑、约束、异常处理
08-自身权限模型：现有角色、数据范围、操作边界
09-自身指标体系：现有报表、统计口径、数据看板
10-自身非功能性现状：性能、并发、缓存、安全等现状
11-自身技术债务：已知缺陷、历史包袱、待优化项
12-自身历史决策：为什么当前这样设计、曾经否决过什么方案
```

#### 6.0.3 自身软件现状盘点模板

以模块为单位盘点：

```text
模块：客户管理
盘点日期：2026-07-05
盘点人：产品 + 研发

业务对象：

缺失对象：FollowUpRecord、Reminder（当前跟进信息以备注字段存在）

数据模型：
现有表：customer（含 follow_up_remark、next_follow_time 字段）
问题：跟进历史无法保留，只能存最后一次

状态机：
现有状态：新客户、已成交、已流失
缺失状态：跟进中
问题：缺少跟进中状态，无法区分正在跟进和新增未触达

业务规则：
现有规则：仅负责人可编辑客户
缺失规则：跟进记录权限、提醒触发规则

UI 现状：
现有页面：客户列表、客户详情（含备注区）
缺失页面：跟进记录时间线、今日待跟进工作台

权限现状：
现有角色：销售、管理员
缺失：团队数据范围、主管查看权限

指标现状：
现有报表：客户总数统计
缺失：转化率、跟进活跃度

技术债务：
- 客户表字段冗余，跟进信息耦合在客户表
- 无操作日志，无法追溯客户修改历史

历史决策：
- 首版为快速上线，跟进功能简化为备注字段
- 曾考虑独立跟进表，因排期推迟
```

#### 6.0.4 对齐对比方法

自身软件现状库与竞品知识库按相同维度对齐，形成差距：

```text

→ 竞品知识库中的最佳实践
→ 自身软件现状库中的现状
→ 差距（缺失、落后、领先、持平）
→ 差距原因（历史决策、技术债务、资源限制）
→ 是否需要弥补
```

#### 6.0.5 研发实现与验证反馈

升级任务交付研发后，研发实现结果必须反馈到知识库，形成闭环：

```text

→ 验证知识库推断是否正确（状态机、规则、模型）
→ 推断正确 → 证据强度升级为"已验证"
→ 推断错误 → 触发第 4.8 节纠偏机制
→ 记录实现中发现的新规则、新约束
→ 补充到自身软件现状库
→ 产品上线后进入第 16.4 节持续反馈
```


## 8. 逆向学习的盲区与陷阱

逆向学习天然有局限，必须明确认知，避免把推断当事实、把表象当本质。

### 8.1 看不到后台

竞品的前端只是冰山一角，以下内容只能推断，无法确认：

- 数据库表结构
- 服务端架构
- 性能瓶颈
- 缓存策略
- 异步任务
- 定时任务
- 消息队列

应对策略：

- 标注为"假设"，不做设计依据。
- 通过技术博客、开源项目、招聘信息侧面印证。
- 自己的产品设计不依赖竞品后台实现，只借鉴业务模型。

### 8.2 竞品"烟雾弹"

营销页和演示页可能与实际产品不符：

- 演示页功能比实际更丰富。
- 截图经过美化，实际体验可能有差异。
- 销售材料宣传的功能可能尚未上线或为路线图项。

应对策略：

- 区分"已上线功能"和"宣传功能"。
- 通过实际注册试用验证。
- 用户社区评论是去伪存真的重要渠道。

### 8.3 版本错位

看到的可能是竞品旧版本：

- 截图采集时间与竞品实际版本不对应。
- 文档可能滞后于产品实际更新。
- 竞品可能在采集后做了重大改版。

应对策略：

- 每条证据标注采集时间和竞品版本号（如可知）。
- 定期复查关键证据，标记是否过期。
- 过期证据不删除，标记为"历史版本参考"。

### 8.4 幸存者偏差

只能看到活下来的设计，看不到被放弃的：

- 竞品迭代中删除的功能，可能是踩过的坑。
- 看不到竞品 A/B 测试中被淘汰的方案。
- 看不到竞品内部否决的设计提案。

应对策略：

- 关注竞品更新日志中的"移除""下线"记录。
- 从用户社区中寻找"为什么取消了这个功能"的讨论。
- 对竞品共性保持警惕——共同做法不一定是最优，可能是路径依赖。

### 8.5 推断错误

状态机、规则、数据模型的推断可能完全错误：

- 从按钮变化推断状态机，可能漏掉中间状态。
- 从字段推断数据模型，可能把一个表拆成两个或把两个表合成一个。
- 从文档推断业务规则，可能理解偏差。

应对策略：

- 推断必须标注证据强度。
- 关键推断必须经过第 4.8 节验证。
- 多个独立证据交叉验证后才可升级为高可信推断。

### 8.6 语境盲区

竞品的设计决策可能基于我们不知道的约束：

- 竞品的历史技术债务。
- 竞品的组织架构限制。
- 竞品的特定客户需求。
- 竞品的合规要求。

应对策略：

- 不盲目照搬竞品设计。
- 理解设计意图而非复制设计结果。
- 结合自身约束做决策。

### 8.7 目标用户差异

竞品的设计是面向其目标用户群体的，如果我们的目标用户不同，直接借鉴可能水土不服：

- 竞品面向大型企业，设计复杂且配置项多；我们的用户是中小企业，需要开箱即用。
- 竞品面向专业用户，术语专业、操作密度高；我们的用户是新手，需要引导和简化。
- 竞品面向特定行业，业务规则深度定制；我们是通用产品，不能照搬行业规则。

应对策略：

- 拆解竞品时同步记录其目标用户画像。
- 横向对比时增加"目标用户差异"维度。
- 借鉴设计前先判断：我们的用户是否与竞品用户一致？不一致时如何调整？
- 对目标用户差异大的竞品，降级为 B 级参照，不做深度拆解。

## 9. 法律与合规边界

竞品学习必须守住法律和道德底线。本章是红线，不可逾越。

### 9.1 可采集与不可采集

| 类型 | 是否可采集 | 说明 |
|---|---|---|
| 官网公开内容 | 可以 | 截图、公开文档、公开演示 |
| 帮助文档 | 可以 | 公开面向用户的文档 |
| 更新日志 | 可以 | 公开发布的版本记录 |
| 社区公开帖子 | 可以 | 公开论坛、社区中的用户讨论 |
| 公开 API 文档 | 可以 | 面向开发者的公开文档 |
| 反编译产品代码 | 不可以 | 违反软件著作权和许可协议 |
| 内测泄漏数据 | 不可以 | 使用未经授权的泄漏信息 |
| 入侵或爬取非公开接口 | 不可以 | 违反网络安全法 |
| 购买竞品账号进行超出许可的逆向 | 不可以 | 违反用户协议 |
| 通过竞品员工获取内部信息 | 不可以 | 涉及商业秘密 |

### 9.2 抄袭风险规避

核心原则：学设计模式，不学表达形式。

- 可以学习业务模型、状态机、业务规则、UI 模式等抽象设计。
- 不可以复制竞品的 UI 视觉设计、图标、文案、代码实现。
- 可以借鉴交互逻辑，不可以像素级模仿界面。
- 竞品的专利技术方案需进行专利排查，避免侵权。

### 9.3 用户数据合规

采集用户反馈和评价时：

- 只采集公开发布的用户评价。
- 不采集个人身份信息。
- 不通过欺骗手段获取竞品用户数据。
- 遵守数据隐私相关法规（如《个人信息保护法》《GDPR》）。

### 9.4 商标与品牌名称使用

在知识库中使用竞品名称和截图时需注意：

- 知识库仅限项目内部使用，不对外公开发布。
- 引用竞品名称时使用"竞品 A""竞品 B"等代号，内部映射表单独管理。
- 截图仅用于内部分析，不用于对外宣传或营销材料。
- 如需在对外文档中引用竞品，需获得竞品授权或使用公开可引用的材料。
- 知识库访问权限受控，仅限项目相关人员。

### 9.5 合规自检清单

每次知识库更新前，进行合规自检：

```text
□ 所有证据来源均为公开渠道
□ 未使用反编译、入侵、泄漏等手段
□ 未复制竞品 UI 视觉表达和代码实现
□ 未采集个人隐私数据
□ 竞品引用标注了来源
□ 涉及专利的技术方案已做专利排查
□ 知识库访问权限受控，仅限项目内部使用
```

合规自检与证据卡片状态联动：

```text
人工合规自检
→ 通过 → 将证据卡片 compliance_status 更新为 approved
→ 存疑 → 标记为 pending-review，暂不作为知识卡片依据
→ 不通过 → 标记为 rejected，CI 门禁自动阻断其被引用（见 14.8 节）
→ 自检结果记录到证据卡片正文或变更日志
```

## 10. 知识库质量标准

### 10.1 知识库成熟度分级

质量标准回答"卡片好不好"，成熟度分级回答"整库到什么程度、能支撑什么决策"。两者缺一不可：一个每张卡片都格式合规的知识库，整体深度仍可能只够选方向。知识库必须在 README 中声明当前成熟度级别，用低级别知识库支撑高级别决策必须显式标注风险并经产品负责人确认。

| 级别 | 名称 | 能回答的问题 | 达标门槛 | 可支撑的决策 | 不可支撑的决策 |
|---|---|---|---|---|---|
| L0 | 方向级 | 选哪条路线、对标谁 | 竞品分类分级完成；能力矩阵初稿；核心路线 ADR | 立项方向、竞品选择、投入决策 | 元模型/架构设计、研发排期 |
| L1 | 设计级 | 核心模块怎么设计 | S 级竞品达到 4.0.3 节 DoD；核心对象有字段级数据模型 + 状态机 + 规则卡；关键推断含深度渠道证据（源码/实测，按 4.0.3 节竞品类型） | 产品蓝图（5.2）、模块设计（5.3）、单向门 ADR | 接口/表结构定稿、测试用例 |
| L2 | 研发输入级 | 这个功能具体怎么实现 | 5.4 节研发输入可直接生成；状态机/规则/权限卡可映射为代码与测试用例；假设验证率 ≥ 70% | PRD、接口设计、数据库设计、测试用例 | — |
| L3 | 持续运营级 | 竞品和自己产品的最新差距 | 16.4 节反馈闭环运转；证据过期率 < 20%；研发验证反馈持续回填 | 升级路线图、持续迭代 | — |

关键约束：

- 高风险单向门 ADR（如元模型定义、技术选型）最低要求 L1 支撑；仅有 L0 时，ADR 的 decision_status 只能是 proposed，不得 accepted。
- 成熟度评估对象是"领域对象/模块"，不是整库：允许核心模块 L2、边缘模块 L0，但必须逐模块声明。
- 参考规模感：一个中等复杂度产品领域，L0 约需数十张卡片，L1 约需 100-300 张，L2 通常 300 张以上。卡片数只是必要信号不是充分条件，但低于该量级却声明高级别，基本可判定为虚标。

### 10.2 知识条目质量标准

一个可用的竞品知识库必须满足以下标准：

```text
1. 每条结论能追溯到证据。
2. 事实、推断、假设、决策分开记录。
3. 多竞品对比围绕领域对象和用户任务，而不是围绕截图堆叠。
4. 能输出自己的产品蓝图、模块设计和研发输入。
5. 能说明哪些竞品设计被采用，哪些没有采用，原因是什么。
6. 能持续更新，而不是一次性报告。
7. 能支持产品全新设计，也能支持已有产品升级。
8. 覆盖功能性和非功能性维度，不只拆功能。
9. 关键推断经过验证，假设有明确的验证计划。
10. 合规自检通过，无法律风险。
11. 竞品选择有明确分级，重点突出。
12. 证据有时效标记，过期证据有处理机制。
13. 知识条目原子化且通过 Schema 校验，决策可沿知识图谱一键追溯到证据。
14. 证据纠偏时能自动定位受影响的推断和决策（影响分析）。
15. 知识库可被 AI 检索和消费，支撑设计、编码、评审的自动化。
```

低质量知识库的表现：

```text
只保存截图，没有拆对象。
只列功能，没有分析业务规则。
只说竞品好，没有说明适用边界。
只抄 UI，没有理解用户任务。
只记录字段，没有区分直接事实和推断。
只做一次报告，不能持续沉淀。
不能转成 PRD、数据模型、接口和测试用例。
只拆功能，不关注性能、安全等非功能性维度。
推断未经验证就当事实使用。
不区分竞品版本，新旧证据混在一起。
没有合规自检，存在法律风险。
```

## 11. 角色分工与协作机制

知识库的建设和使用需要多角色协作，职责不清会导致知识库沦为个人笔记。

### 11.1 角色职责

| 角色 | 职责 |
|---|---|
| 产品经理 | 竞品选择、业务设计库、流程库、业务规则库、产品决策库 |
| UI/UX 设计师 | 页面与 UI 设计库、UI 模式抽取、交互逻辑分析 |
| 前端工程师 | UI 模式可落地性评估、前端架构方案 |
| 后端工程师 | 数据模型库、状态机库、非功能性设计库 |
| 数据工程师 | 指标库、报表口径、数据模型评审 |
| 测试工程师 | 测试场景拆解、状态机验证用例 |
| 产品负责人 | 决策评审、优先级排序、范围把控 |

### 11.2 协作流程

```text
产品经理发起竞品拆解
→ 各角色按职责采集证据并拆解
→ 产品经理汇总多竞品横向对比
→ 产品负责人组织评审
→ 形成产品决策（ADR）
→ 决策归档并通知研发
→ 研发实现后反馈验证结果
→ 更新知识库证据强度
```

### 11.3 评审机制

知识库评审分三层：

| 评审层级 | 频率 | 参与者 | 评审内容 |
|---|---|---|---|
| 证据评审 | 每次采集后 | 产品 + 设计 | 证据质量、证据强度标注 |
| 模型评审 | 每模块拆解完成后 | 产品 + 设计 + 研发 | 数据模型、状态机、业务规则 |
| 决策评审 | 每次产品决策前 | 产品负责人 + 核心成员 | ADR 备选方案、取舍、验证方式 |

### 11.4 决策归档机制

- 每条产品决策（ADR）必须有编号、版本、评审记录。
- ADR 一经决策不可删除，只能通过新 ADR 废止。
- 废止的 ADR 保留并标注"已废止"及废止原因。
- ADR 与知识库条目双向关联，便于追溯。

## 12. 版本与时效管理

知识库是"活的"，必须有版本管理和时效机制，否则会积累大量过期信息。

### 12.1 证据时效管理

```text
证据采集
→ 标注采集时间和竞品版本
→ 定期复查（建议每季度）
→ 竞品已更新 → 标记旧证据为"历史版本参考"
→ 竞品已下线该功能 → 标记为"已废弃"
→ 重新采集新证据
→ 更新关联的知识条目
```

证据状态：

| 状态 | 含义 |
|---|---|
| 有效 | 当前竞品版本中仍存在 |
| 历史版本参考 | 竞品已更新，旧证据仅供历史参考 |
| 已废弃 | 竞品已下线该功能 |
| 待复查 | 超过复查周期未验证 |

### 12.2 知识库版本管理

- 知识库整体版本号：主版本.次版本（如 v2.1）。
- 主版本：知识库结构重大调整时递增。
- 次版本：新增或修改知识条目时递增。
- 每次版本更新记录变更日志。

变更日志模板：

```text
版本：v2.1
日期：2026-07-05
变更人：产品经理
变更内容：
- 新增竞品 D 证据 12 条
- 更新订单状态机（竞品 A 新增"部分退款"状态）
- 标记 5 条证据为"历史版本参考"
- 新增 ADR-ORDER-008
```

### 12.3 更新频率建议

| 更新类型 | 频率 | 说明 |
|---|---|---|
| 全量复查 | 每季度 | 核心竞品所有证据复查一遍 |
| 增量更新 | 每月 | 跟踪竞品更新日志，采集新证据 |
| 重点跟踪 | 实时 | S 级竞品重大版本更新时立即跟踪 |
| 决策复查 | 每半年 | 复查已有 ADR 是否需要调整 |

### 12.4 存量知识库升级模式：增量升级 vs 推倒重建

当方法论版本升级（如新增 DoD、成熟度分级）或发现已有知识库质量不达标时，面临"重新执行一遍"还是"在存量上升级"的选择。决策规则如下：

**核心判断：知识库的问题是"错"还是"浅"。**

| 问题类型 | 典型表现 | 处理模式 |
|---|---|---|
| 浅（深度不足） | 结构合规、结论方向正确，但证据数量、渠道种类、实测覆盖未达 DoD | **增量升级**：保留存量卡片，补充采集与实测 |
| 错（结构性错误） | Schema 混乱、编号冲突、大量结论被证伪、领域对象划分根本性错误 | **推倒重建**：按新版方法论重新执行 |
| 混合 | 部分模块错、部分模块浅 | 按模块拆分：错的模块重建，浅的模块增量 |

增量升级的理由：深度永远来自实测、安装、源码分析等真实采集动作，而不是来自方法论文档本身——重新执行一遍并不会让同样的公开资料拆解变深，只会重复支付已付过的成本。方法论升级（如量化 DoD）的作用是**堵死"浅活冒充完成"的路**，强制后续执行必须完成深度采集。

增量升级的标准动作序列：

1. **存量审计**：按新版质量规则逐卡审计（置信度是否违规、ADR 状态是否越级、矩阵是否缺"未验证"标注），先修正存量违规，再补新内容。
2. **成熟度定位**：按 10.1 节对各模块评级，输出与 4.0.3 节 DoD 的差距清单。
3. **增量采集**：按差距清单补证据卡与知识卡，沿用存量 Schema 与编号规则，不得另起一套。
4. **重新验收**：按 DoD 与成熟度门槛重新评估，更新 README 成熟度声明。

**配套要求：通用执行工作流 + 领域实例化指导书。** 本方法论是与领域无关的通用规范；通用执行入口是《01A-工作流-知识库建设与持续完善.md》。执行某个具体领域项目（如低代码平台、CRM 竞品分析）时，执行者只需要拿到知识库名称即可启动；若该领域需要固定竞品分级、核心模块、分阶段任务或验收判据，执行者应在该领域目录下生成一次性的《执行指导书》，完成规则到项目的实例化：

- 竞品 S/A/B 定级结果与理由（套用 4.0 节规则）；
- 该领域的核心模块清单（DoD 按模块检查的对象）;
- 存量修正任务清单（若为升级场景）；
- 分阶段任务与验收判据（套用 4.0.3 节 DoD 与 10.1 节分级）；
- 交付物清单与给执行 AI 的启动提示语。

实例化决策（哪个竞品定几级、核心模块是哪些）只写入执行指导书，不回写本方法论，以保持方法论的领域无关性。

## 13. AI 辅助拆解

在 AI 时代，竞品拆解的多个环节可以借助 AI 提效，但必须有人工把关。

### 13.1 AI 可辅助的环节

| 环节 | AI 能力 | 人工把关点 |
|---|---|---|
| 证据采集 | OCR 识别截图文字、文档解析 | 校验识别准确性 |
| 页面拆解 | 自动提取字段、按钮、筛选项 | 校验完整性、补充隐藏规则 |
| 对象抽取 | 从文本中识别业务对象和属性 | 校验对象边界和关系 |
| 状态机推断 | 从状态标签和按钮变化推断状态转移 | 校验状态完整性和不变量 |
| 横向对比 | 多竞品信息汇总和差异分析 | 校验对比维度和结论 |
| 指标抽取 | 从报表截图提取指标和口径 | 校验口径准确性 |
| 文档生成 | 自动生成模板填充初稿 | 人工评审和修正 |

### 13.2 AI 产出的校验原则

- AI 产出的所有内容必须标注"AI 辅助生成"。
- AI 产出的推断默认为"假设"，需人工验证后才能升级。
- AI 可能产生幻觉（编造不存在的信息），所有 AI 产出必须与原始证据交叉核对。
- AI 不做最终设计决策，决策由人做。
- AI 产出的内容需经过第 11.3 节评审流程。

### 13.3 AI 辅助的边界

AI 不能替代的环节：

- 竞品选择和分级（需要战略判断）
- 设计取舍决策（需要结合自身约束）
- 合规判断（需要法律专业知识）
- 用户痛点的深度理解（需要用户访谈和共情）
- 证据真实性的最终确认

### 13.4 多智能体拆解流水线

单点使用 AI（让 AI 看一张截图）价值有限。规模化拆解应组织成多智能体流水线，各环节职责单一、产出结构化：

```text
采集 Agent
（浏览器自动化批量截图、抓取公开文档、监控竞品更新日志）
→ 拆解 Agent
（按 4.2-4.6 模板逐页拆解，输出符合 14.5 节 Schema 的结构化知识卡片）
→ 对比 Agent
（按领域对象聚合多竞品卡片，产出横向对比初稿与能力矩阵初稿）
→ 校验 Agent（LLM-as-Judge）
（检查证据链完整性、模板字段完整性、是否把推断越界写成事实、置信度标注是否合规；
  对照 4.0.3 节 DoD 检查覆盖度——格式合规但深度不足时，输出缺口清单而不是放行）
→ 人工评审（第 11.3 节评审流程）
→ 入库（通过 14.8 节 CI 质量门禁后合并）
```

流水线要点：

- 每个 Agent 的输出必须是结构化卡片，不是自由文本，否则无法进入下一环节和 CI 门禁。
- 校验 Agent 与拆解 Agent 必须使用独立提示词（条件允许时用不同模型），避免自我确认偏差。
- 竞品更新监控可完全自动化：定时抓取竞品 changelog 与帮助文档，diff 出变化内容，自动生成"待拆解"任务并关联受影响的既有卡片。
- 采集 Agent 必须内置第 9 章合规约束：只访问公开页面、遵守 robots 协议、不绕过认证、控制访问频率。批量截图与抓取需设置合理间隔（建议单竞品单页不少于 3 秒），避免触发反爬机制和给竞品服务器造成压力；如竞品明确禁止自动化访问，则降级为人工采集。

### 13.5 知识库反向服务 AI

知识库不仅由 AI 帮助构建，建成后应作为 AI 的上下文源，这是结构化知识库相对文档堆的核心增值：

| 阶段 | 用法 |
|---|---|
| 设计阶段 | 将领域对象、状态机、业务规则卡片作为上下文，AI 生成模块设计初稿和 PRD 初稿 |
| 研发阶段 | 编码 Agent（如 Claude Code）通过 RAG 检索知识库（14.7 节），直接引用状态机和业务规则生成实体、状态校验、权限校验和测试用例，把第 7 章"知识库到代码"的映射自动化 |
| 评审阶段 | AI 对照业务规则库和状态机库，检查 PRD 与代码是否遗漏规则、是否引入非法状态转移 |
| 升级阶段 | AI 对照差距表和 ADR，检查新需求是否与既有决策冲突 |

约束：AI 引用知识库时必须携带卡片 ID 与证据强度；证据强度为"假设"的卡片不得作为代码生成依据，只能生成待验证任务。

## 14. 工具与承载形式建议

### 14.1 承载形式对比

### 14.4 检索与关联

- 每条知识条目通过编号唯一标识。
- 知识条目之间通过编号互相引用（如 ADR 引用证据编号）。
- 建议建立索引文件，按领域对象汇总所有关联条目。
- Git 仓库可借助全文搜索快速检索。

### 14.5 知识卡片与元数据 Schema

这是知识库构造的核心工程决策：每条知识存为一个独立 Markdown 文件（一卡一事），文件头部用 YAML frontmatter 承载统一元数据，正文承载第 3 章对应模板中不适合结构化的解释内容。

Schema 分三类，不混用：

- 证据卡片：保存原始来源与采集信息，是判断独立来源和证据时效的依据。
- 普通知识卡片：保存对象、流程、状态机、规则、权限、指标、非功能性推断等知识。
- ADR 卡片：保存产品设计决策，不使用 `strength` 表示决策质量，而使用 `decision_status` 表示决策状态。

#### 14.5.1 证据卡片 Schema

```yaml
---
id: E-COMPA-ORDER-001
type: evidence
competitor: A
module: order
source_channel: official-demo       # official-doc/official-demo/community/video/review/trial/interview/changelog
source_type: screenshot             # screenshot/doc/video/post/trial-note/api-observation/interview-note
source_url: https://example.com/demo
source_owner: competitor-official    # competitor-official/user-community/third-party/self-trial
captured_at: 2026-07-05
valid_until: 2026-10-05
license_note: public-page            # public-page/trial-account/user-provided/permission-required
compliance_status: approved          # approved/pending-review/rejected
status: active                       # active/deprecated/pending-review
owner: 产品经理
ai_generated: false
---

（正文：只记录原始观察，不写产品结论。）
```

#### 14.5.2 普通知识卡片 Schema

```yaml
---
id: SM-ORDER-001              # 全局唯一 ID：类型前缀-领域对象-序号
type: state-machine           # business/ui/process/data-model/state-machine/
                              # rule/permission/pain-point/metric/nfr
domain_object: Order          # 关联领域对象（图谱聚合的主键）
competitors: [A, B]           # 涉及竞品代号
evidence: [E-COMPA-ORDER-001, E-COMPB-ORDER-003]   # 证据链（图谱的"支持"边）
strength: 高可信推断           # 直接事实/高可信推断/假设/已验证
confidence: 0.7               # 数值置信度（见 2.2 节交叉验证规则）
status: active                # active/deprecated/pending-review
collected_at: 2026-07-05
valid_until: 2026-10-05       # 到期自动进入"待复查"（14.8 节定时扫描）
links: [ADR-ORDER-002, BR-ORDER-011]   # 关联条目（图谱的"引用"边）
owner: 产品经理
ai_generated: true            # 是否 AI 辅助生成（13.2 节要求）
---

（正文：按第 3 章对应模板填写解释、推理链、边界和反例；不重复维护已进入 frontmatter 的元数据。）
```

#### 14.5.3 ADR 卡片 Schema

```yaml
---
id: ADR-CUSTOMER-001
type: adr
domain_object: Customer
module: customer-management
decision_status: proposed       # proposed/accepted/rejected/validated/deprecated
basis: [BIZ-CUSTOMER-003, SM-CUSTOMER-001, BR-CUSTOMER-004]
evidence: [E-COMPA-CUSTOMER-002, E-COMPB-CUSTOMER-006]
rejected_options: [ADR-CUSTOMER-001-A, ADR-CUSTOMER-001-B]
risk: medium                    # low/medium/high
review_at: 2026-10-05
valid_until: 2027-01-05
links: [PRD-CUSTOMER-001, DM-CUSTOMER-002]
owner: 产品负责人
ai_generated: false
---

（正文：记录背景、备选方案、决策、理由、代价、回滚方式和验证方式。ADR 不填写 strength。）
```

Schema 规则：

- Schema 是知识库的"数据库表结构"，必须先定义后填充，字段变更走知识库主版本升级。
- ID 前缀约定：E-证据、BIZ-业务设计、UI-页面、FLOW-流程、DM-数据模型、SM-状态机、BR-业务规则、PM-权限、OPP-机会、ADR-决策、MT-指标、NFR-非功能。
- 普通知识卡片必须填写 `strength` 和 `confidence`；ADR 卡片不得填写 `strength`，必须填写 `decision_status`。
- 第 3 章模板中的字段按下表映射，避免正文与 frontmatter 两处不一致。

| 第 3 章模板字段 | frontmatter 字段 | 正文是否重复维护 |
|---|---|---|
| 竞品证据/来源证据 | `evidence` | 不重复，只在正文解释证据如何支撑结论 |
| 证据强度 | `strength` | 不重复 |
| 数值置信度 | `confidence` | 不重复 |
| 我们决策/关联ADR | 普通知识卡片的 `links` + ADR 卡片的 `decision_status` 和正文决策段落 | 普通知识卡片只链接 ADR，不写最终决策；最终决策只在 ADR 维护 |
| 适用边界/设计代价/失败场景 | 无统一元数据字段 | 写正文 |
| 备选方案/不采用原因/回滚方式 | ADR 正文 | 写正文 |

### 14.6 知识图谱与影响分析

结构化卡片之上即可构建知识图谱：节点是卡片，边由 frontmatter 的 evidence 和 links 字段解析而来。

```text
证据(E) --支持--> 推断类卡片(DM/SM/BR/PM/MT/NFR)
推断类卡片 --输入--> 横向对比结论
对比结论 --依据--> ADR
ADR --产出--> PRD/模块设计/研发任务
领域对象 --拥有--> 状态机/规则/页面/指标（按 domain_object 聚合）
```

图谱的三大用途：

- 追溯：任何 ADR 一键回溯到原始证据，把第 10 章质量标准第 1 条从"人工自觉"变成"图查询"。
- 影响分析：证据被纠偏（4.8.2 节）时，沿"支持"边自动找出所有受影响的推断和 ADR 并批量标记 pending-review，不再依赖人工记忆。这是纠偏机制能否真正执行的关键。
- 盲区发现：批量查出无证据支持的推断、无验证计划的假设、被零引用的孤儿证据、被"假设"直接支撑的 ADR。

实现建议：不必上重型图数据库。用脚本解析全部 frontmatter 即可在内存构建图谱并输出报告；卡片规模上千、查询复杂化后再迁移 Neo4j 等图数据库。

### 14.7 混合检索与 GraphRAG

知识库检索按需分四层，逐层增强：

| 层级 | 手段 | 回答的问题 |
|---|---|---|
| 结构化过滤 | 解析 frontmatter 按字段筛选 | "Order 对象所有未验证的假设""竞品 B 的全部状态机" |
| 全文检索 | ripgrep / ElasticSearch | 关键词命中 |
| 语义检索 | 卡片正文向量化（embedding） | "竞品如何处理退款超时"这类自然语言查询 |
| GraphRAG | 语义命中卡片后沿图谱边扩展关联证据、规则、ADR 一并作为上下文 | 供 AI 消费的完整上下文包（13.5 节） |

落地建议：中小团队先做前两层即可覆盖 80% 场景；接入 AI 设计/编码工作流后再补语义检索与 GraphRAG。向量索引应由 CI 在卡片变更时增量重建，避免检索结果落后于知识库。

### 14.8 KB-as-Code：CI 自动化质量门禁

知识库入 Git 后，用 CI 把第 10 章质量标准与第 12 章时效管理从"人工自觉"变成"自动门禁"：

```text
提交时（PR 合并前阻断）：
- Schema 校验：frontmatter 字段完整、类型合法、ID 唯一、前缀与 type 匹配
- 断链检测：evidence/links 引用的 ID 必须存在
- 证据卡片校验：source_channel/source_type/source_url/source_owner/captured_at/license_note/compliance_status 必须完整；
  compliance_status=rejected 的证据不得被任何知识卡片或 ADR 引用
- 强度一致性：strength=直接事实/已验证 的卡片必须关联至少一条证据；
  strength=假设 的普通知识卡片不得被 ADR 作为 accepted/validated 决策依据引用（只能作为验证方向）
- ADR 校验：type=adr 的卡片不得填写 strength；必须填写 decision_status、basis、rejected_options、risk、review_at；
  decision_status=accepted/validated 时，basis 不得全部来自 strength=假设 的知识卡片
- 置信度校验：confidence 与独立来源数量符合 2.2 节交叉验证规则；
  独立来源由被引用证据卡片的 source_channel + source_owner + source_url 去重判断，不能只数 evidence ID
- 页面/UI 门禁：竞品存在可见 UI 时，页面拆解卡必须引用截图、录屏、官方 demo、官方文档截图、公开视频或公开产品截图；仅引用源码、Schema 或元数据的 UI 卡只能计为实现机制证据，不得通过页面拆解数量门禁
- 合规检查：证据来源字段非空；命中禁用来源关键词（反编译、泄漏等）时阻断并人工复核

定时任务（每日/每周）：
- 时效扫描：valid_until 过期的卡片自动标记 pending-review 并生成复查任务
- 孤儿扫描：零引用证据、无验证计划的假设，输出清单
- 竞品 changelog 监控：diff 出竞品更新，生成待拆解任务（联动 13.4 节采集 Agent）
- 影响分析报告：本周被纠偏证据波及的推断与 ADR 清单
- 统计看板：各库卡片数、证据强度分布、假设验证率、证据过期率、AI 生成占比
- 成熟度评估：按模块对照 4.0.3 节 DoD 与 10.1 节分级自动评估，结果写入 README；
  检测"低成熟度支撑高级别决策"（如 L0 模块存在 accepted 状态的高风险 ADR）并告警
- 置信度审计：单竞品能力断言若仅有官方渠道证据支撑却 confidence > 0.6，标记违规（2.2 节规则）
```

效果：第 12 章的季度人工复查降级为"处理 CI 报出的异常清单"，第 4.8 节纠偏从"发现矛盾靠运气"变成"到期与冲突自动浮出"。

## 15. 量化评估与优先级框架

### 15.1 竞品能力量化评估

能力矩阵中的星级评分需要统一标准，避免主观随意。

评分标准：

| 分值 | 标准 |
|---|---|
| ★★★★★ | 该维度设计成熟、体验优秀、有创新 |
| ★★★★ | 该维度完整覆盖、体验良好 |
| ★★★ | 该维度有基础支持、体验一般 |
| ★★ | 该维度有雏形、体验较弱 |
| ★ | 该维度勉强支持、有明显缺陷 |
| - | 该维度缺失 |

评估维度建议：

```text
功能完整度：是否覆盖该领域的核心场景
体验成熟度：交互是否流畅、信息架构是否清晰
创新性：是否有差异化设计
灵活性：是否支持自定义和扩展
性能表现：响应速度和并发能力
移动端支持：移动端体验是否完善
```

### 15.2 机会优先级评估框架

痛点与机会库中的"优先级"字段不应凭感觉，建议用影响面 × 频率 × 成本的矩阵评估。

```text
优先级 = 影响面 × 影响频率 × （1 / 实现成本）
```

| 因素 | 评分 | 说明 |
|---|---|---|
| 影响面 | 1-5 | 影响多少用户（1=少数，5=全部） |
| 影响频率 | 1-5 | 多久遇到一次（1=低频，5=每日） |
| 实现成本 | 1-5 | 开发工作量（1=数日，5=数月） |

优先级矩阵：

```text
高优先级：影响面 ≥ 4 且 频率 ≥ 4 且 成本 ≤ 3
中优先级：不满足高优先级，但 影响面 × 频率 ≥ 12
低优先级：影响面 × 频率 < 12
```

### 15.3 差距严重程度评估

差距表中的"严重程度"字段量化标准：

| 严重程度 | 标准 |
|---|---|
| P0 | 核心流程受阻，用户无法完成关键任务 |
| P1 | 关键体验受损，用户需要大量 workaround |
| P2 | 体验不佳，但用户可以完成基本任务 |
| P3 | 体验可优化，不影响核心使用 |

### 15.4 RICE 评分：把证据可信度纳入优先级

15.2 节的公式有一个缺陷：不区分机会点的证据可信度。竞品学习产出的机会点很多来自推断，推断可能是错的，优先级却没有惩罚项。建议对进入排期的机会点升级为 RICE 评分：

```text
RICE = (Reach × Impact × Confidence) / Effort

Reach：单位周期内触达的用户数
Impact：对单个用户的影响程度（3=巨大，2=高，1=中，0.5=低，0.25=极小）
Confidence：信心系数，直接与证据强度挂钩
Effort：实现成本（人月）
```

Confidence 与证据强度的映射：

| 机会点主要依据的证据强度 | Confidence |
|---|---|
| 直接事实（截图/文档中明确出现） | 100% |
| 已验证（用户访谈/试用/灰度确认） | 100% |
| 高可信推断 | 80% |
| 假设 | 50% |

口径统一说明：若机会点已卡片化，Confidence 直接取其依据卡片的 `confidence` 字段（0-1，见 2.2 节交叉验证规则）；上表仅作为未卡片化时按强度档位速查的近似映射，两者不得混用。

效果：优先级天然惩罚"仅凭竞品猜测"的机会，倒逼团队先执行第 4.8 节验证再投入开发——验证通过后 Confidence 提升，优先级自动上浮。

### 15.5 Kano 分类：纠正"竞品都有 = 必做"误判

机会点在量化排序前先做 Kano 分类，避免把竞品共性误当用户必需（呼应 8.4 节幸存者偏差）：

| Kano 类型 | 含义 | 竞品学习中的判断信号 |
|---|---|---|
| 基础型 | 没有就流失 | 所有竞品都有，且用户社区存在"因缺失而弃用"的抱怨 |
| 期望型 | 越强越满意 | 竞品重点宣传，评分与该能力体验正相关 |
| 兴奋型 | 有则惊喜 | 仅个别竞品有，用户自发好评传播 |
| 无差异型 | 有无皆可 | 竞品有，但用户社区无人讨论、无人使用反馈 |
| 反向型 | 有反而反感 | 用户社区抱怨"太复杂""被强制打开" |

决策规则：

- 基础型：必须补齐，不受 RICE 排序约束。
- 兴奋型：差异化机会的主要来源，优先投入验证。
- 无差异型：即使所有竞品都有，也应放弃——竞品共性可能只是路径依赖。
- 反向型：写入"不做事项"，并在 ADR 中记录不做的理由。

## 18. 一句话总结

竞品知识库的价值，不是证明"别人怎么做"，而是帮助我们回答：

```text
用户真正的问题是什么？
竞品为什么这样设计？
这种设计的对象、状态、规则和 UI 模式是什么？
它有什么代价和边界？
哪些推断已验证，哪些仍是假设？
我们应该采用、改进、差异化，还是放弃？
最终如何转成自己的产品模型、页面、数据结构、业务规则和代码实现？
```

只有能持续回答这些问题的知识库，才能真正驱动产品全新设计和持续升级。

## 附录 A：轻量级快速启动指南

完整方法论适用于中大型团队和长期项目。对于小团队或快速调研场景，可先用最小可行版本启动，逐步扩展。

### A.1 最小可行版本

只做三件事：

```text
1. 选 1 个核心竞品，注册试用，截图关键页面。
2. 用页面拆解模板（第 4.2 节）拆 5-10 个核心页面。
3. 用横向对比模板（第 4.7 节）产出一页纸的对比结论。
```

### A.2 最小知识库结构

小团队可暂只建四个目录：

```text
competitor-kb-lite/
├── evidence/          # 截图和链接
├── pages/             # 页面拆解
├── decisions/         # 产品决策
└── summary.md         # 一页纸总结
```

### A.3 从轻量版扩展到完整版

当团队规模增长或项目变复杂时，按以下顺序逐步扩展：

```text
轻量版
→ 补充数据模型库和状态机库（研发对接需要）
→ 补充业务规则库和权限模型库（复杂业务需要）
→ 补充指标库和非功能性设计库（运营和架构需要）
→ 补充竞品选择分级和验证纠偏（多竞品需要）
→ 补充版本管理和合规自检（长期维护需要）
→ 补充知识图谱、RAG 检索和 CI 门禁（规模化与 AI 协作需要）
→ 完整版
```

重要提示：轻量版可以省掉图谱、RAG 和 CI，但不要省 frontmatter。从第一张卡片开始就按 14.5 节 Schema 写元数据，成本几乎为零；等卡片积累到几百张再补结构，需要人工回填全部历史条目，代价极高。结构先行，工具后补。

### A.4 轻量版适用判断

| 场景 | 建议版本 |
|---|---|
| 1-2 人团队、单模块调研 | 轻量版 |
| 创业初期 MVP 设计 | 轻量版 + 数据模型库 |
| 5 人以上团队、多模块产品 | 完整版 |
| 已有产品大版本升级 | 完整版 |
| 对外合规要求高的行业 | 完整版（含合规自检） |
| 需要AI辅助设计/编码/评审 | 完整版 + 14.5 Schema + 14.7 GraphRAG（轻量版 frontmatter 可先行，AI 消费时再补检索） |
| 竞品数量多、需定期监控更新 | 完整版 + 13.4 多智能体流水线 + 14.8 CI 门禁 |
