# 03-方法论-系统级失败模式与设计约束清单

> 版本：v1.3
> 日期：2026-07-10
> 定位：软件开发业界通用的方法论，用于识别、描述、分级、承接和持续维护软件系统的高后果失败模式。
> 建设方法规范：具体如何从需求、设计、代码、测试和事故材料中建设或更新清单，见《03A-方法论-失败模式与设计约束清单建设.md》。
> 通用问题族清单：《04-清单-通用系统级失败模式与设计约束.md》；平台类专门清单：《04A-清单-复杂平台级失败模式与设计约束.md》。

## 0. AI 读取与执行入口

```text
最小输入：待评审系统/产物 + 可获得的需求、设计、代码、测试或运行证据。
自动动作：识别候选失败模式，区分事实与推断，生成触发条件、后果、约束、验证和承接建议。
送审动作：失败模式候选卡、领域清单草案、覆盖缺口。
人工决策：风险等级争议、风险接受、条目废弃和上线豁免。
输出：按第 7 节结构生成条目，并附 evidence_refs、confidence、open_questions。
完成判据：高后果条目均有可执行验证或明确进入 awaiting_human；不得以“加强注意”代替约束。
```

AI 声明条目闭环前必须主动构造至少一个反例或故障注入场景，验证当前约束是否真的能阻止失败。

## 1. 这类文档解决什么问题

软件系统失败通常不是因为“某一行代码写错”这么简单，而是因为团队在需求、设计、实现、测试、发布或运维阶段，没有提前识别某类高后果失败。

这类文档要解决的问题是：

```text
在开发前和开发中，系统性识别软件可能如何失败；
把这些失败转成必须回答的设计约束、验证要求和交付门禁；
避免团队只覆盖正常路径，而漏掉会造成严重后果的异常路径。
```

它不是：

- 功能清单
- 任务排期
- 代码风格规范
- 普通测试用例列表
- 某个技术栈的最佳实践

它是：

```text
失败模式 → 后果 → 约束 → 验证 → 承接责任
```

## 2. 推荐命名

推荐名称：

```text
系统级失败模式与设计约束清单
```

也可以根据组织习惯命名为：

- 高后果失败模式清单
- 系统性风险与设计约束清单
- 失败模式与验证门禁清单
- 风险驱动设计约束清单

不建议只叫“架构风险清单”，因为很多失败不只来自架构，也来自需求语义、边界条件、数据状态、权限、并发、交付、运维和团队协作。

## 3. 核心概念

| 概念 | 通用定义 |
|---|---|
| 失败模式 | 系统在某种条件下偏离预期并造成不可接受后果的方式。 |
| 触发条件 | 让失败发生的输入、状态、时序、规模、权限、依赖、环境或操作条件。 |
| 后果 | 失败造成的用户损失、业务损失、数据错误、安全问题、合规风险、可用性问题或维护成本。 |
| 设计约束 | 为避免失败，需求、设计、实现、测试、发布或运维必须满足的条件。 |
| 验证方式 | 证明约束被满足的方式，可以是测试、评审、演练、监控、审计或检查清单。 |
| 承接责任 | 哪个交付物、角色或流程负责回答和验证该条目。 |
| 证据来源 | 支撑该失败模式真实存在的依据来源，可以是需求推导、设计推导、代码证据、测试缺陷、用户反馈、事故复盘、竞品证据、行业经验或待补证假设。 |
| 状态 | 条目当前所处的生命周期阶段，常见取值：候选 / 已确认 / 已承接 / 已验证 / 风险接受 / 废弃。 |

## 4. 什么时候使用方法论

方法论用于**建立或更新清单**。

适用时机：

1. **新项目启动**：识别首版系统性风险。
2. **需求评审**：检查需求是否隐含不可接受的失败后果。
3. **方案设计**：把失败模式转成设计约束。
4. **开发前任务拆分**：明确每个任务必须覆盖的风险和验证方式。
5. **测试设计**：从正常路径扩展到异常路径、边界路径和恢复路径。
6. **上线前评审**：确认关键失败模式都有验证或风险接受。
7. **事故复盘**：把真实失败回写成新条目。
8. **重大变更前**：当系统边界、用户规模、数据规模、依赖、部署方式或团队流程发生变化时重新检查清单。
9. **AI 辅助开发引入时**：检查 AI 生成代码是否覆盖已知失败模式。

## 5. 什么时候使用清单

清单用于**执行具体工作**。

| 阶段 | 清单的用法 |
|---|---|
| 需求 | 检查是否存在未定义的边界、异常、角色、状态、数据或承诺。 |
| 设计 | 逐条回答相关失败模式的约束和取舍。 |
| 实现 | 避免绕过已确定的约束、入口和不变量。 |
| 测试 | 生成边界、异常、并发、恢复、安全和回归用例。 |
| 发布 | 检查迁移、回滚、监控、告警和应急处理是否准备好。 |
| 运维 | 用清单定位高风险链路、监测指标和演练场景。 |
| 复盘 | 将事故原因转成可复用条目，而不是只修当次问题。 |

一句话：

```text
方法论用于发现和维护问题；
清单用于约束设计、实现、验证和交付。
```

## 6. 如何识别失败模式

从以下问题开始：

1. **对象会错吗**：关键对象的状态、关系、归属、数量、时间是否可能错误。
2. **边界会破吗**：权限、所有权、输入范围、外部依赖、资源限制是否可能被突破。
3. **时序会乱吗**：并发、重试、超时、乱序、重复、部分成功是否会造成错误。
4. **语义会漂吗**：同一个词、字段、状态、动作在不同地方是否含义不一致。
5. **规模会变吗**：数据量、用户量、请求量、依赖数量增长后，原假设是否失效。
6. **失败能恢复吗**：中断、异常、依赖失败、错误操作后是否能恢复到可接受状态。
7. **人会误用吗**：用户、运营、管理员、开发者是否容易按错误方式使用系统。
8. **变化会破坏吗**：版本升级、配置变更、流程调整、组织变化是否破坏历史行为。
9. **证据可靠吗**：需求、竞品、用户反馈、线上数据或推断是否有足够证据支撑。

## 7. 条目结构

每个条目应尽量回答以下内容：

```text
编号：
标题：
等级：
失败模式：系统如何坏。
触发条件：什么情况下会坏。
后果：坏了造成什么不可接受结果。
设计约束：为了避免它，方案必须满足什么。
验证方式：如何证明约束被满足。
承接责任：由哪个文档、角色、任务或流程负责。
证据来源：为什么认为这个失败模式真实存在。
状态：候选 / 已确认 / 已承接 / 已验证 / 风险接受 / 废弃。
```

完整填写示例：

```text
编号：D1
标题：重复执行造成重复结果
等级：高
失败模式：同一意图因为重复点击、重试、刷新、超时、消息重放或人工重复操作而执行多次，导致产生重复的副作用。
触发条件：用户重复点击提交按钮；网络超时后前端自动重试；支付网关重试回调；消息中间件重放。
后果：重复扣款、重复创建订单、重复发送通知、重复审批、重复结算，造成资金损失和业务数据不一致。
设计约束：带不可重复副作用的操作必须基于幂等键或唯一请求标识识别“同一次意图”；重复请求返回已有结果而非重新执行；关键操作必须有去重或锁定机制。
验证方式：并发测试模拟重复请求；端到端测试覆盖重试和超时场景；回归测试验证幂等键生效。
承接责任：设计文档定义幂等方案；实现任务包含去重逻辑；测试规格覆盖重复场景。
证据来源：行业经验（支付重复扣款事故普遍）；事故复盘（同类系统已发生重复扣款案例）；需求推导（支付场景天然存在重试）。
状态：已确认
```

## 8. 分级标准

| 等级 | 通用含义 | 处理要求 |
|---|---|---|
| 高 | 可能造成不可恢复的数据错误、严重安全问题、合规问题、大面积不可用或重大业务损失。 | 必须有明确设计约束和验证方式；无法验证时必须显式风险接受。涉及合规问题时，验证方式必须包含审计追溯能力。 |
| 中 | 可能造成局部不可用、错误结果、明显体验损害、恢复成本较高或维护风险。 | 必须有预防、检测或恢复方案。 |
| 低 | 主要造成长期维护成本、理解成本、局部体验问题或可控返工。 | 可以通过规范、评审、文档或后续治理承接。 |

等级不是技术复杂度，而是后果严重度。

## 9. 从通用到领域实例

通用方法论不能替代领域清单。

正确流程：

```text
通用识别框架
→ 通用问题族
→ 当前系统适用性筛选
→ 领域语言改写
→ 具体设计约束
→ 验证方式
→ 交付物承接
```

示例：

```text
通用问题族：重复操作造成重复副作用
领域实例：重复付款、重复发货、重复审批、重复发送通知、重复生成订单
```

## 10. 维护规则

1. 条目必须描述“怎么坏”，不能只写“注意风险”。
2. 条目必须说明后果，否则无法分级。
3. 条目必须有承接责任，否则无法验收。
4. 通用条目不能写成某个产品、行业或技术栈的专属问题。
5. 领域条目可以具体，但要明确它属于哪个通用问题族。
6. 事故复盘后应检查是否需要新增条目，而不是只修补代码。
7. 当条目长期没有承接或验证，应标记为未闭环，而不是默认已解决。
8. AI 生成的代码或设计应经过失败模式清单交叉检查。

## 11. 合格标准

一份合格清单应满足：

- 能说明系统最可能如何失败。
- 能区分高后果失败和普通缺陷。
- 能指导需求、设计、实现、测试、发布和运维。
- 能追踪每个高风险问题由谁承接、如何验证。
- 能随着真实反馈和事故复盘持续更新。
