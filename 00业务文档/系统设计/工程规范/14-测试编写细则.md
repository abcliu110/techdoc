## 1. 测试分层

规则 14-001：单元测试覆盖 domain/service 纯逻辑，不依赖 Spring 容器优先。

规则 14-002：集成测试使用 Testcontainers MySQL/Redis，禁止用 H2 替代动态 DDL 行为。

规则 14-003：架构测试使用 ArchUnit，覆盖模块依赖、层级、SQL 入口、表达式门面、权限门面。

规则 14-004：安全测试集不可删除；修改必须人工评审。

## 2. 命名与数据

规则 14-010：测试方法命名 `方法_场景_期望`。
正例：`publishVersion_ddlFails_marksPartialApplied`。

规则 14-011：测试数据统一 `TestDataBuilder`，禁止散落大段 SQL insert。

规则 14-012：核心测试数据必须包含双租户、双应用、两个角色、一个字段级权限、一个 data_scope。

## 3. 必测矩阵

规则 14-020：动态 SQL 注入测试覆盖字段名、op、sort、link path、分页、批量 ids、导出查询、统计查询、聚合查询（group by/having）。

规则 14-021：权限测试覆盖 /meta、list、get、update 三处一致性。

规则 14-022：租户隔离测试必须双租户并发执行，断言不能串数据。

规则 14-023：MetaGraph 测试必须覆盖发布中请求、旧版本请求、新版本请求、多实例收敛窗口。

规则 14-024：动态 DDL 测试覆盖 22 种类型建表、加列、危险变更拒绝、重复发布幂等。

规则 14-025：类型转换测试覆盖 null、空串、非法日期、跨时区、超精度 decimal、boolean 非法形态。

规则 14-026：action/transition 测试覆盖重复提交、并发审批、超时后重试、批量部分成功。

规则 14-027：规则副作用测试覆盖 outbox、重试、死信、幂等键、循环检测。

规则 14-028：导入包安全测试覆盖表达式注入、路径穿越、超大 JSON、重复 code、跨租户 id、未知字段。

规则 14-029：Connector 测试覆盖 localhost/私网/cloud metadata 拒绝、超时、限流、脱敏日志。

规则 14-030：attachment 测试覆盖 MIME/扩展名双校验、跨租户下载拒绝、软删除后访问、超大文件。

规则 14-031：组织数据范围测试必须覆盖员工调岗、部门删除、部门合并、组织树移动后历史记录可见性；断言对象级可配置的 `CURRENT_ORG` / `OWNER_SNAPSHOT` 行为，并以 `OWNER_SNAPSHOT` 作为默认值。
强制方式：权限集成测试，对应 B7。

规则 14-031a：link/user/org 跨租户测试必须覆盖写入他租户 lid 被拒绝、suggest 不返回他租户记录、展开查询 SQL 注入目标表 tenant_id 三类场景。
强制方式：双租户安全测试，对应 B4/B7。

规则 14-032：租户级资源隔离测试必须覆盖单租户大导出、规则死循环、慢 Connector、批量导入、频繁 DDL 发布时，其他租户核心 CRUD 仍可用。
强制方式：压测脚本或演练报告，对应 C5。

规则 14-033：工作流兼容测试必须覆盖旧流程实例遇到新版定义、节点删除、角色删除、字段删除、等待节点定时任务重复触发。
强制方式：工作流实例回放测试，对应 F8。

规则 14-034：前端安全与动态类型测试必须覆盖富文本 XSS、schema props 注入、附件文件名注入、旧 meta/新前端、新 meta/旧前端、权限裁剪字段缺失。
强制方式：Vitest/Playwright + 契约测试，对应 G4/G5。

规则 14-035：DDL 锁测试必须覆盖目标表存在长事务/慢查询时发布被阻断、`lock_wait_timeout` 超时后状态落库、DDL 日志记录锁等待耗时。
强制方式：Testcontainers MySQL 集成测试，对应 A5。

规则 14-036：Redis 与发布锁测试必须覆盖 Redis 闪断、DB 已提交但 Redis 版本未更新、锁过期双持有者、旧 fencing token 继续执行被拒绝。
强制方式：Testcontainers Redis + 并发集成测试，对应 A6。

规则 14-037：调度测试必须覆盖双实例同一任务只执行一次、实例宕机后必须补跑任务被补跑、跨租户时区的按日任务触发点正确。
强制方式：调度集成测试，对应 C6。

规则 14-038：formula 测试必须覆盖循环依赖、跨对象 link 深度上限、源字段删除/改类型发布阻断、无权限源字段导致结果不可见、过滤/排序/聚合能力阻断。
强制方式：元数据校验 + 权限集成测试，对应 D4。

规则 14-039：fetch_from 测试必须覆盖写入快照、源记录更新后历史值不漂移、源记录软删除后的显示口径、无权限用户不得通过快照泄露。
强制方式：运行时集成测试，对应 D5。

规则 14-040：multilink/级联/ID 测试必须覆盖中间表 DDL 与登记表、集合 diff 审计、cascade/set_null 全链预检、级联行数超限阻断、`Abc`/`abc` code 判重、雪花时钟回拨、禁止按 lid 排序。
强制方式：DDL 生成测试 + 集成测试，对应 D6/D7/D8。

规则 14-041：元数据 JSON 并发测试必须覆盖两个用户同时编辑不同字段，后提交者因 revision CAS 失败而不能覆盖先提交者；`lc_meta_ref` 重建失败时对象 JSON 与引用索引一起回滚。
强制方式：并发集成测试，对应 A4/B5/B12。

规则 14-042：认证安全测试必须覆盖 CSRF、禁用用户旧 token、改密旧会话失效、跨租户 token 重放、会话固定防护。
强制方式：安全集成测试，对应 F9。

规则 14-043：业务导入导出测试必须覆盖部分失败重导幂等、导入统一写入口、越权字段导出裁剪、大文件流式解析、CSV 公式注入、导出文件下载鉴权和保留期。
强制方式：导入导出安全/性能测试，对应 F10。

规则 14-044：通知测试必须覆盖无权限收件人、敏感字段模板、通道超时重试不重复发送、死信可重放、租户通知频率配额。
强制方式：通知集成测试，对应 F11。

规则 14-045：设计态预览测试必须覆盖预览态写操作阻断、草稿元数据不污染已发布 MetaGraph、预览权限裁剪和预览审计标识。
强制方式：Playwright E2E + API 契约测试，对应 G6。

规则 14-046：错误信息泄露测试必须覆盖不存在/无权限响应归一、错误响应体不含 SQL/物理列名/堆栈/内部类名/路径/连接串、traceId 可用于内部日志关联。
强制方式：安全测试 + 响应扫描，对应 G7。

规则 14-047：OpenAPI 与动态对象兼容测试必须覆盖平台固定 DTO breaking change 被 OpenAPI diff 阻断、动态 record 容器 breaking change 被阻断、record 内业务字段变化通过 metaVersion/runtime schema 处理。
强制方式：契约测试，对应 B11/G5。

规则 14-048：tombstone 测试必须覆盖软删目标有权限返回 title、无目标对象 read 返回空 title、无 title_field read 返回空 title，且响应结构固定为 `{lid, deleted, deletedAt, title}`。
强制方式：API 契约测试，对应 B10。

## 6. Testcontainers 基准模板

规则 14-061：动态 DDL、事务、锁、JSON、索引相关测试必须使用真 MySQL。

```java
@Testcontainers
@SpringBootTest
class SchemaSyncIntegrationTest {

  @Container
  static final MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0")
      .withDatabaseName("lowcode_test")
      .withUsername("test")
      .withPassword("test");

  @DynamicPropertySource
  static void datasource(DynamicPropertyRegistry registry) {
    registry.add("spring.datasource.url", mysql::getJdbcUrl);
    registry.add("spring.datasource.username", mysql::getUsername);
    registry.add("spring.datasource.password", mysql::getPassword);
  }
}
```

规则 14-062：Redis 缓存、分布式锁、MetaGraph 版本收敛测试必须使用真 Redis 容器。

## 7. TestDataBuilder 约定

规则 14-070：测试数据必须表达业务结构，不得把元数据 JSON 大段硬编码在测试方法里。

正例：

```java
LowcodeFixture fixture = testDataBuilder
    .tenant("t_a")
    .app("sales")
    .object("order")
    .field("amount", FieldTypeEnum.DECIMAL, f -> f.precision(18, 4).required())
    .field("customer", FieldTypeEnum.LINK, f -> f.target("customer"))
    .role("sales", r -> r.canRead("order").dataScope(DataScopeEnum.SELF))
    .publish();
```

反例：

```java
jdbcTemplate.execute("insert into lc_meta_object(fields) values ('...一大段 JSON...')");
```

## 8. 关键测试类骨架

### 8.1 DDL 半应用测试

```java
@Test
void publishVersion_thirdDdlFails_marksPartialAppliedAndCanReconcile() {
  PublishPlan plan = fixtureBuilder.orderApp().withThreeObjectChanges().buildPlan();
  ddlExecutor.failOn(3);

  assertThatThrownBy(() -> publishService.publish(plan.command()))
      .isInstanceOf(BizException.class);

  PublishState state = publishRepository.getState(plan.versionId());
  assertThat(state).isEqualTo(PublishState.PARTIAL_APPLIED);
  assertThat(ddlLogRepository.listByVersion(plan.versionId())).hasSize(3);
  assertThat(reconciler.detect(plan.appId())).isNotEmpty();
}
```

### 8.2 权限 OR 短路测试

```java
@Test
void list_withTopLevelOrFilter_doesNotBypassDataScope() {
  fixture.twoUsersInSameTenantWithSelfScope();
  QueryRequest request = QueryRequest.or(
      Filter.eq("ownerId", "other-user"),
      Filter.gt("amount", "0")
  );

  PageResult<RecordVO> result = dataApi.list(userA, "sales", "order", request);

  assertThat(result.records()).allMatch(row -> row.ownerId().equals(userA.id()));
}
```

### 8.3 幂等 action 测试

```java
@Test
void transition_sameIdempotencyKey_executesSideEffectOnce() {
  String key = "idem-001";
  actionApi.transition(user, orderId, "submit", key);
  actionApi.transition(user, orderId, "submit", key);

  assertThat(transitionLogRepository.countByRecord(orderId)).isEqualTo(1);
  assertThat(outboxRepository.countByRecord(orderId)).isEqualTo(1);
}
```

强制方式：组织权限集成测试，对应 D11/B7。

规则 14-090：复杂编码规则测试必须覆盖租户隔离、组织维度隔离、日期重置、并发取号、允许跳号和导入保留历史编号的边界。
强制方式：并发取号测试，对应 D11/C4。

规则 14-091：报表测试必须覆盖字段权限裁剪、数据范围注入、聚合口径、导出防公式注入和大数据量异步生成。
强制方式：报表 API/安全/性能测试，对应 G8/F10/B2/B3。

规则 14-092：打印模板测试必须覆盖模板版本回放、权限裁剪、预览/下载一致性、模板变量缺失和历史单据按历史快照渲染。
强制方式：打印 E2E + 契约测试，对应 G8/C3。

规则 14-093：菜单、门户和移动端测试必须覆盖菜单权限裁剪、深链访问后端拒绝、桌面组件移动端阻断或降级、不同端 schema 能力声明。
强制方式：Playwright E2E + schema lint，对应 G8/G1。

规则 14-094：多语言测试必须覆盖对象、字段、动作、状态、菜单、错误消息、报表标题的 key 解析和 fallback 顺序；缺失翻译必须进入发布报告。
强制方式：i18n 契约测试，对应 G8。
