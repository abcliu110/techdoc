## 1. 提示词工程基础

### 1.1 无效提示与有效提示的对比

**无效提示**（形容词堆砌，触发风格输出而非推理）：

```
"请你站在世界顶级架构师的高度，用企业级的标准，
提供一个优雅的、可扩展的、符合业界最佳实践的解决方案"
```

这种提示的问题在于：AI 看到"世界顶级"后倾向于输出一个看起来"大而全"的方案，而不是针对具体问题的最优解。形容词只影响输出风格，不触发推理深度。

**有效提示**（约束触发推理，具体约束引导最优路径）：

```
"这是一个电商订单模块。用户下单后需要：
1. 库存预占（扣减可用库存，保留锁定库存）
2. 优惠券核销（一券一用，需幂等）
3. 积分计算（按商品分类梯度计算）
4. 支付回调前允许取消订单，释放预占库存
请按以下约束设计：
- 使用领域驱动设计，给出 Entity、Value Object、Domain Service
- 订单聚合根在支付完成前后的状态流转必须用状态机表达
- 库存操作必须与其他操作在同一事务内，不接受最终一致性
- 不得引入分布式事务框架
- 参照 23-运行时数据语义细则 处理并发冲突
请给出完整的类图和关键方法签名"
```

**核心原则：形容词激发风格，约束激发思考。** 约束越具体，AI 的推理路径越清晰，输出的模型越接近最优解。

### 1.2 提示词结构模板

按以下顺序组织提示词，各部分缺一不可：

```
# 背景
[系统的业务背景和技术栈]

# 需求
[要解决的具体问题，用"subject-verb-object"描述]

# 约束（最关键）
[技术约束：事务边界、性能要求、兼容性要求、禁止使用的技术]
[业务约束：状态流转规则、异常边界、历史数据处理]
[参考规范：参照本工程规范的具体章节]

# 输出要求
[要求输出的模型类型：类图、状态机、序列图、DDL、API契约]
[要求分析的维度：调用链、并发场景、失败回滚]

# 验证标准
[功能验证：哪些场景必须覆盖]
[非功能验证：性能、并发、兼容性的验证方式]
```

## 2. 领域建模技术

### 2.1 核心概念辨析

**Entity（实体） vs Value Object（值对象）**

| 特征 | Entity | Value Object |
|---|---|---|
| 身份标识 | 有唯一标识，通过 ID 判等 | 无标识，按属性值判等 |
| 可变性 | 可变状态 | 不可变，设计为不可变类 |
| 生命周期 | 有创建、更新、删除 | 随宿主对象使用，不独立生命周期 |
| 典型示例 | Order、Customer、Product | Money、Address、PhoneNumber |
| 建模判断 | "如果两个对象属性完全相同但不是同一个，你关心吗？" 回答是，则为 Entity | 回答否，则为 Value Object |

**Entity 建模示例**：

```java
// 实体：订单，有唯一身份
public class Order extends Entity {

    private OrderId id;           // 唯一标识，通过 ID 判等
    private CustomerId customerId;
    private OrderStatus status;   // 枚举，不得使用裸 Integer
    private List<OrderLine> lines;
    private Money totalAmount;    // 值对象，金额必须用 Money 而非 Double

    // 聚合根方法：只能在聚合根上调用，禁止外部直接修改 lines 或 status
    public void cancel(CancelReason reason) {
        if (!status.canTransitionTo(CANCELLED)) {
            throw new OrderStateException("当前状态不允许取消");
        }
        this.status = OrderStatus.CANCELLED;
        this.cancellationReason = reason;
        this.cancelledAt = Instant.now();
        // 发布领域事件
        DomainEvents.publish(new OrderCancelledEvent(this.id, reason));
    }
}
```

**Value Object 建模示例**：

```java
// 值对象：金额，不可变，按值判等
public final class Money {

    private final BigDecimal amount;   // 禁止使用 float/double
    private final Currency currency;   // 币种语义

    public Money(BigDecimal amount, Currency currency) {
        this.amount = amount.setScale(currency.getScale(), RoundingMode.HALF_UP);
        this.currency = Objects.requireNonNull(currency);
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("币种不一致");
        }
        return new Money(this.amount.add(other.amount), this.currency);
    }
}
```

### 2.2 聚合设计原则

聚合是维护一致性的最小单元。设计聚合时遵循以下原则：

1. **聚合内强一致性**：聚合内部的对象状态变更必须在同一事务内完成
2. **聚合间最终一致性**：聚合之间通过领域事件异步同步，不在同一个事务内
3. **聚合边界最小化**：只将真正需要强一致性的实体放入同一聚合，避免大聚合带来的并发瓶颈
4. **聚合根是唯一入口**：外部只能通过聚合根 ID 引用聚合，只能调用聚合根的方法修改状态

**反面示例（大聚合）**：

```java
// 错误：将所有相关对象放入同一聚合，导致并发冲突范围过大
public class BigOrder extends Entity {
    private Customer customer;        // 不需要强一致性的关联对象
    private List<OrderLine> lines;
    private Payment payment;           // 支付与订单不属于同一聚合
    private Inventory inventory;      // 库存更不应在同一聚合
}
```

**正面示例（边界清晰的聚合）**：

```java
// 聚合根：Order
public class Order { ... }

// 独立聚合根：Payment
public class Payment { ... }

// 独立聚合根：InventoryReservation
public class InventoryReservation { ... }

// 聚合间通过领域事件协作
// OrderService 在支付完成后发布 OrderConfirmedEvent
// InventoryReservationService 订阅该事件，确认库存预占
```

### 2.3 贫血模型 vs 富血模型

**贫血模型**（反模式，Entity 只有 getter/setter，业务逻辑全在 Service）：

```java
// 反模式：不推荐
public class Order extends Entity {
    private OrderId id;
    private BigDecimal amount;
    // 只有 getter/setter，无行为
}

public class OrderService {
    public void cancel(OrderDTO dto) {
        if (order.getStatus() != OrderStatus.PENDING) {
            throw new IllegalStateException("只有待支付订单可取消");
        }
        order.setStatus(OrderStatus.CANCELLED);
        orderRepository.save(order);
        // 库存释放、积分退回逻辑全部散落在 Service
        inventoryService.release(dto.getInventoryReservationId());
        pointsService.revert(dto.getCustomerId(), dto.getPointsUsed());
    }
}
```

**富血模型**（推荐，业务逻辑内聚到聚合根）：

```java
// 推荐模式
public class Order extends Entity {

    private OrderStatus status;

    public void cancel(CancelReason reason, CancelPolicy policy) {
        // 取消逻辑内聚，状态机约束在聚合根内
        if (!status.canTransitionTo(CANCELLED)) {
            throw new OrderStateException(
                "当前状态 [" + status.getName() + "] 不允许取消。允许取消的状态：" +
                status.getAllowedTransitionsTo(CANCELLED));
        }
        this.status = OrderStatus.CANCELLED;

        // 按策略处理副作用，返回需要外部协调的命令
        return policy.execute(new CancellationContext(
            this,
            inventoryReservationId,
            pointsUsed,
            couponId
        ));
    }
}
```

### 2.4 领域事件

领域事件用于解耦聚合之间的协作。建模规则：

1. 事件命名使用过去式：`OrderPlaced`、`InventoryReserved`、`PaymentFailed`
2. 事件携带足够的上下文数据（聚合根 ID、时间戳、操作者 ID），但不携带完整聚合快照
3. 事件消费方必须幂等处理
4. 事件顺序不保证时，消费方必须处理乱序场景

```java
// 领域事件建模
public record OrderPlacedEvent(
    OrderId orderId,
    CustomerId customerId,
    List<ReservedInventory> reservations,
    Money totalAmount,
    Instant occurredAt
) {}

// 事件发布
public class Order extends Entity {
    public static Order create(OrderDraft draft) {
        Order order = new Order();
        order.status = OrderStatus.PENDING;
        // 发布领域事件
        DomainEvents.publish(new OrderPlacedEvent(
            order.id,
            draft.customerId(),
            draft.reservations(),
            draft.totalAmount(),
            Instant.now()
        ));
        return order;
    }
}
```

## 3. 业务规则建模

### 3.1 规则树（组合模式）

适用于：多个规则按优先级组合，最终输出单一结果的场景，如折扣计算、资格校验。

```
规则树结构：
         [Root Rule]
            / | \
     [RuleA] [RuleB] [RuleC]
       / \      |       / \
   [A1] [A2]  [B1]  [C1] [C2]
```

**示例：优惠券计算规则树**：

```java
public abstract class CouponRule {

    protected final int priority;

    protected CouponRule(int priority) {
        this.priority = priority;
    }

    public abstract CouponResult evaluate(CouponContext context);

    // 组合方法
    public CouponResult evaluateWithNext(CouponContext context, CouponRule next) {
        CouponResult current = this.evaluate(context);
        if (!current.isApplicable()) {
            return current;
        }
        CouponResult nextResult = next.evaluateWithNext(context, current);
        return nextResult;
    }
}

// 叶子规则：满减
public class AmountThresholdRule extends CouponRule {
    @Override
    public CouponResult evaluate(CouponContext context) {
        if (context.getOrderAmount().compareTo(threshold) < 0) {
            return CouponResult.notApplicable("订单金额未达到满减门槛");
        }
        return CouponResult.applicable(discount);
    }
}

// 叶子规则：会员等级折扣
public class MemberLevelDiscountRule extends CouponRule {
    @Override
    public CouponResult evaluate(CouponContext context) {
        int discountPercent = switch (context.getMemberLevel()) {
            case GOLD -> 10;
            case PLATINUM -> 15;
            case DIAMOND -> 20;
            default -> 0;
        };
        return CouponResult.applicable(context.getOrderAmount()
            .multiply(discountPercent / 100));
    }
}

// 规则链注册（参照 10-Java编码细则.md 的校验链模板）
@Configuration
public class CouponRuleConfig {
    @Bean
    public CouponRule couponRuleChain() {
        return new MemberLevelDiscountRule(10)
            .setNext(new AmountThresholdRule(20))
            .setNext(new FirstOrderRule(5));
    }
}
```

### 3.2 规则链（职责链模式）

适用于：按顺序处理，每个处理器决定是否继续传递的场景，如订单创建流程、审批流程。

```
请求 → [HandlerA] → [HandlerB] → [HandlerC] → 成功
            ↓          ↓          ↓
          拒绝        拒绝        拒绝
```

**示例：订单创建流程链**：

```java
public interface OrderCreationHandler {
    /**
     * 处理订单创建的前置校验或操作。
     * @return 若处理成功返回 null，继续传递给下一个 Handler；
     *         若拒绝则返回错误结果，不再继续。
     */
    OrderCreationResult handle(OrderCreationContext context);

    default OrderCreationHandler setNext(OrderCreationHandler next) {
        this.next = next;
        return next;
    }
}

public class InventoryReservationHandler implements OrderCreationHandler {
    @Override
    public OrderCreationResult handle(OrderCreationContext context) {
        try {
            context.setInventoryReservation(
                inventoryService.reserve(context.getOrderItems())
            );
            return null; // 继续传递
        } catch (InsufficientInventoryException e) {
            return OrderCreationResult.rejected(
                "库存不足，商品：" + e.getSkuCode() + "，可用：" + e.getAvailable()
            );
        }
    }
}

public class CouponValidationHandler implements OrderCreationHandler {
    @Override
    public OrderCreationResult handle(OrderCreationContext context) {
        if (context.getCouponId() != null) {
            ValidationResult result = couponService.validate(
                context.getCouponId(),
                context.getCustomerId()
            );
            if (!result.isValid()) {
                return OrderCreationResult.rejected(result.getReason());
            }
            context.setCoupon(result.getCoupon());
        }
        return null;
    }
}
```

### 3.3 状态机模式

适用于：对象有明确生命周期状态，且状态转换有前置条件和副作用的场景。

**建模要求**：

1. 状态和转换事件建模为枚举（`OrderStatus`、`PaymentEvent`）
2. 状态机转换规则建模为配置或表驱动，不硬编码在 if-else 中
3. 每个转换事件必须声明：源状态、目标状态、前置条件、副作用（事件发布）

```java
// 状态机配置（配置化，禁止硬编码状态分支）
public enum OrderStatusTransition {

    // 订单状态转换
    PENDING(PENDING, PAYMENT_INITIATED, PaymentInitiatedEvent.class, null),
    PAYMENT_INITIATED(PAYMENT_INITIATED, PAID, PaymentReceivedEvent.class,
        InventoryConfirmCommand.class),
    PAYMENT_INITIATED(PAYMENT_INITIATED, CANCELLED, OrderCancelledEvent.class, null),
    PAID(PAID, SHIPPED, ShipmentDispatchedEvent.class, null),
    PAID(PAID, REFUND_REQUESTED, RefundRequestedEvent.class, null),
    SHIPPED(SHIPPED, COMPLETED, DeliveryConfirmedEvent.class, null);

    private final OrderStatus from;
    private final OrderStatus to;
    private final Class<? extends DomainEvent> triggerEvent;
    private final Class<? extends Command> sideEffectCommand;

    public boolean matches(OrderStatus current, OrderStatus target) {
        return this.from == current && this.to == target;
    }
}

// 状态机执行器
public class OrderStateMachine {

    public void transition(Order order, OrderStatus target, Object trigger) {
        OrderStatusTransition transition = findTransition(order.getStatus(), target);

        if (transition == null) {
            throw new OrderStateException(
                "状态 [" + order.getStatus().getName() +
                "] 不允许转换到 [" + target.getName() + "]"
            );
        }

        // 执行转换
        order.setStatus(target);

        // 发布领域事件
        DomainEvents.publish(
            transition.getTriggerEvent().getDeclaredConstructor(Order.class, Object.class)
                .newInstance(order, trigger)
        );

        // 发送副作用命令
        if (transition.getSideEffectCommand() != null) {
            CommandBus.send(transition.getSideEffectCommand().newInstance(order));
        }
    }
}
```

### 3.4 批量操作策略

按业务需求选择以下三种策略，不得混用：

| 策略 | 语义 | 适用场景 | 事务边界 |
|---|---|---|---|
| `ALL_OR_NOTHING` | 全成功或全回滚 | 财务类、库存扣减 | 单事务 |
| `PARTIAL_SUCCESS` | 记录成功/失败清单，返回处理结果 | 批量发消息、批量通知 | 每条独立事务，汇总报告 |
| `FAIL_STOP` | 遇到第一条失败立即停止 | 不可中断的顺序依赖操作 | 失败时已处理的全部回滚 |

**PARTIAL_SUCCESS 建模示例**：

```java
public record BatchOperationResult<T>(
    int total,
    int successCount,
    int failureCount,
    List<BatchSuccessItem<T>> successes,
    List<BatchFailureItem<T>> failures
) {
    public boolean hasFailures() {
        return failureCount > 0;
    }

    public BatchOperationResult<T> requireAllSuccess() {
        if (hasFailures()) {
            throw new BatchOperationException(
                "批量操作存在失败项：" + failures.size() + " / " + total,
                this
            );
        }
        return this;
    }
}
```

## 4. 复杂业务表达方法

### 4.1 主体-动作-客体（Subject-Verb-Object）分解

将业务描述拆解为原子操作单元：

| 自然语言 | Subject | Verb | Object | 额外约束 |
|---|---|---|---|---|
| "客户下单后自动预占库存" | 客户 | 下单 | 订单 | 预占库存作为订单创建的副作用 |
| "订单支付成功后发放积分" | 系统 | 发放 | 积分 | 支付成功是前置事件 |
| "库存不足时拒绝下单" | 系统 | 拒绝 | 订单创建 | 库存校验先于支付 |
| "会员享受折上折" | 系统 | 计算 | 折扣 | 按会员等级分层叠加 |

### 4.2 场景脚本法

用结构化场景脚本穷举业务路径：

```
场景脚本模板：
Given[前置条件] + When[触发动作] + Then[预期结果] + And[副作用]

示例（订单取消场景）：

场景1：待支付订单可正常取消
  Given 订单状态 = PENDING，创建时间 = 当前时间 - 30分钟
  When 客户发起取消
  Then 订单状态 → CANCELLED
    And 预占库存释放
    And 优惠券退回
    And 积分返还

场景2：已支付订单取消需审批
  Given 订单状态 = PAID，支付时间 = 当前时间 - 10分钟
  When 客户发起取消
  Then 订单状态 → CANCEL_PENDING
    And 发送取消审批通知给客服

场景3：已发货订单不可取消
  Given 订单状态 = SHIPPED
  When 客户发起取消
  Then 返回错误："已发货订单不可取消，请申请退货"
    And 不变更订单状态

场景4：超过可取消时限的待支付订单自动关闭
  Given 订单状态 = PENDING，创建时间 = 当前时间 - 24小时
  When 系统定时任务扫描
  Then 订单状态 → EXPIRED
    And 预占库存释放
```

### 4.3 领域场景矩阵

用矩阵结构表达多维度业务规则：

```
规则：不同支付方式 × 不同会员等级 → 不同折扣率

              | 普通会员 | 银卡会员 | 金卡会员 |
|-------------|---------|---------|---------|
| 微信支付     | 9.8折   | 9.5折   | 9折     |
| 支付宝      | 9.8折   | 9.5折   | 9折     |
| 银行卡      | 无折扣   | 9.8折   | 9.5折   |
| 储值卡      | 无折扣   | 9.8折   | 9.5折   |
| 混合支付     | 折上折   | 折上折   | 折上折   |

建模为：
Map<PaymentMethod, Map<MemberLevel, DiscountPolicy>>
```

## 5. 常用业务模型模式

### 5.1 订单处理模型

```
核心聚合：
Order（聚合根）
  ├── OrderLine（订单明细，实体）
  ├── Payment（支付子聚合，独立聚合根，通过 OrderConfirmedEvent 关联）
  ├── InventoryReservation（库存预占子聚合，独立聚合根）
  └── CouponUsage（优惠券使用记录）

状态机：
PENDING → PAYMENT_INITIATED → PAID → SHIPPING → SHIPPED → COMPLETED
   ↓           ↓              ↓         ↓          ↓
 CANCELLED  CANCELLED    REFUND_PENDING  REFUND_PENDING  REFUND_PENDING

并发控制：
- 订单创建使用乐观锁（revision 字段）
- 库存预占使用数据库行锁（SELECT FOR UPDATE）
- 支付回调幂等键 = orderId + paymentId
```

### 5.2 审批流模型

```
核心概念：
- WorkflowDefinition（工作流定义，模板，不实例化）
- WorkflowInstance（工作流实例，按模板创建）
- ApprovalTask（审批任务，分配给审批人）
- ApprovalDecision（审批决定：同意/拒绝/转交）

状态机（审批任务）：
PENDING → APPROVED → COMPLETED
       → REJECTED → COMPLETED
       → DELEGATED → PENDING（新审批人）
       → EXPIRED → COMPLETED（自动跳过）

关键约束：
- 审批人不得审批自己发起的申请（自我审批禁止）
- 节点配置可指定审批人、审批人组、会签/或签策略
- 驳回（Reject）可配置驳回到哪一步
- 审批超时后按配置自动同意或自动拒绝
```

### 5.3 折扣计算模型

```
折扣叠加规则（按优先级取最高，不叠加）：

Layer 1：会员等级折扣（按会员等级直接打折）
Layer 2：促销活动折扣（满减、阶梯价、限时折扣）
Layer 3：优惠券折扣（满减券、折扣券、兑换券）
Layer 4：积分抵扣（积分抵现金，按比例上限）

最终金额 = 原始金额 × 折扣1 → 结果 × 折扣2 → 结果 - 优惠3

关键约束：
- 折扣后金额不得低于 0.01 元
- 每个 Layer 的上限由配置控制
- 折扣计算过程需要完整日志（审计要求）
- 历史订单的折扣金额必须快照，不可重算
```

### 5.4 多租户数据隔离模型

```
隔离层级：
1. tenant_id 列级隔离（最常用）
2. 租户专属 schema（高安全场景）
3. 租户专属数据库（VIP 客户，可选）

跨租户操作规则：
- 任何查询必须强制带上 tenant_id 条件（安全底线）
- 聚合根的 ID 生成策略必须包含租户前缀
- 异步任务（MQ/定时）必须携带租户上下文，不得从 ThreadLocal 读取
- 导出数据必须校验导出范围，跨租户导出必须阻断

参照 17-安全编码细则.md 的租户隔离规则执行。
```

## 6. 数据模型最佳实践

### 6.1 第三范式（3NF）建模步骤

1. **识别实体**：找出业务中的核心名词（客户、商品、订单）
2. **识别关系**：一对多（Customer → Order）、多对多（Order ↔ Product，中间表）
3. **识别属性**：将属性分配给正确的实体，避免数据冗余
4. **应用 3NF**：
   - 第一范式：每个字段原子，不可再分（地址拆分为省、市、区、详细地址）
   - 第二范式：非主键字段完全依赖主键（消除部分函数依赖）
   - 第三范式：非主键字段之间不存在传递依赖（客户等级名称不应存储在订单表中）

### 6.2 索引设计原则

1. **高频查询条件建索引**：WHERE、JOIN ON、ORDER BY 中的列
2. **复合索引遵循最左前缀**：`INDEX idx_tenant_status (tenant_id, status, created_time)`
3. **避免冗余索引**：相同列组合的索引只保留一个
4. **区分度低的列不放索引**：性别、状态枚举不适合单独建索引
5. **唯一约束本身就是索引**：需要唯一性时直接声明 UNIQUE INDEX
6. **禁止不加索引就上线**：上线前必须有慢查询分析和索引验证

### 6.3 ID 生成策略

| 场景 | 推荐策略 | 说明 |
|---|---|---|
| 全局唯一业务 ID | 雪花算法（Snowflake） | 时间 + 机器 ID + 序列号，高并发下不冲突 |
| 租户内唯一 ID | 租户前缀 + 雪花 | `TENANT_{tenantId}_{snowflake}`，支持跨租户合并场景 |
| 短链/code | 随机数 + 校验位 | 纯数字 6 位验证码、邀请码等 |
| 不可推测 ID | UUID v4 | 防爬、防猜测，但不建议用作业务主键（索引不友好） |
| 顺序敏感 ID | 数据库自增 / 号段模式 | 按创建时间自然排序，但分布式场景需要号段服务 |

参照 11-数据库设计细则.md 规则执行。

## 7. 架构与代码结构模式

### 7.1 分层架构

```
Controller层（*-app）
  └── 接收请求、参数校验、调用Service、封装响应
  └── 不得包含业务逻辑

Service层（*-service）
  └── 业务逻辑编排、事务边界、领域事件发布
  └── 不直接操作数据库（委托给 DAO 层）

DAO层（*-dao）
  └── 数据持久化、复杂查询组装
  └── 不得包含业务判断

Entity / VO / DTO
  └── Entity：数据库表映射，有唯一ID
  └── VO：值对象，不可变
  └── DTO：跨层数据传输
```

### 7.2 依赖方向规则

**核心原则：依赖永远指向稳定的方向。高层不依赖低层。**

```
稳定方向：Controller → Service → DAO
               ↓          ↓
            实体/VO   实体/VO

禁止反向依赖：DAO 不依赖 Service，Service 不依赖 Controller
禁止跨层调用：Controller 不直接调 DAO
```

参照 19-模块边界与依赖治理细则.md 执行。

### 7.3 模块边界控制

1. **按业务域划分模块**：`nms4cloud-crm`、`nms4cloud-order`、`nms4cloud-product`
2. **按层级划分子模块**：`*-api`、`*-app`、`*-service`、`*-dao`
3. **跨域调用走 Feign**：不得在模块内直接引用其他模块的 Service
4. **禁止硬编码依赖**：不得在通用模块中 import 业务模块的类

## 8. 给 AI 的建模检查清单

当向 AI 询问建模方案时，建议在提示词末尾追加以下检查要求：

```
请在输出前逐一确认：
□ 所有状态字段建模为枚举（不裸用 Integer）
□ 所有金额字段使用 BigDecimal 或专用值对象（不使用 Double）
□ 聚合根是唯一的状态修改入口
□ 聚合间通过领域事件协作，不跨聚合事务
□ 状态机转换规则配置化，不硬编码 if-else
□ 批量操作声明了明确的失败策略（ALL_OR_NOTHING / PARTIAL_SUCCESS / FAIL_STOP）
□ 并发场景有乐观锁或悲观锁处理
□ 幂等性有 idempotencyKey 保障
□ 租户隔离字段（tenant_id）存在于所有业务表
□ 敏感数据操作有审计日志
□ 参照了本工程规范相关细则（[列出章节]）
```
