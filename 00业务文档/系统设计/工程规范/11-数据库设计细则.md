## 1. MySQL 基础规则

规则 11-001：字符集统一 `utf8mb4`；排序规则首选 `utf8mb4_0900_ai_ci`，需要大小写敏感的 code 字段使用二进制校验或 `_bin` collation。

规则 11-002：所有表必须有 COMMENT，所有列必须有 COMMENT。
强制方式：SQL 扫描。

规则 11-003：禁止物理外键、触发器、存储过程、MySQL ENUM 列。
理由：动态元数据迁移、跨环境导入导出、私有化升级需要应用层可控。

规则 11-004：时间列用 `datetime(3)`，业务时区转换在应用层处理。

规则 11-005：金额用 `decimal(18,4)` 起步，字段 options 可声明更高精度；禁止 float/double。

## 2. 主键与标准列

规则 11-010：静态表和动态表主键统一 `id bigint`，值由雪花 ID 生成；动态业务记录同时有稳定业务标识 `lid varchar(26)`，值由 ULID 生成。导入导出、API、页面 schema、插件引用 `lid/code`，不引用数据库 `id`。

规则 11-011：所有租户隔离表必须有 `tenant_id bigint not null`，核心查询索引必须包含 tenant_id。

规则 11-012：标准列：`revision` 乐观锁，`deleted` 软删标记，`deleted_at` 软删时间，`delete_token` 软删唯一约束墓碑列，`create_time/create_by/update_time/update_by` 审计列。

## 3. 元数据表

规则 11-020：元数据表控制在 9 张 `lc_meta_*`，子定义进入 JSON 聚合列，不新增字段级明细表，除非 ADR 变更。

规则 11-021：JSON 聚合列必须包含 `_v`，COMMENT 必须说明结构版本和 DTO 名称。

规则 11-022：禁止业务查询使用 `JSON_EXTRACT`；需查询属性必须提升为结构化列或进入 `lc_meta_ref`。

规则 11-023：`lc_meta_ref` 是可重建索引表，必须覆盖 object/field/action/state/rule/page/permission/plugin/connector 引用。

规则 11-024：元数据 JSON 聚合列禁止局部 JSON_SET/JSON_REMOVE 方式更新。保存对象定义必须读出完整 DTO、通过 JsonUpgrader 升级到当前版本、整体验证后以 `revision` CAS 整行写回；CAS 失败必须返回冲突并要求刷新。
强制方式：并发保存字段丢更新测试，对应 A4/B5。

规则 11-025：`lc_meta_ref` 重建必须与对象 JSON 写回处于同一事务；事务内先生成新引用集，再替换旧引用集，禁止对象 JSON 已提交但引用索引仍是旧图。
强制方式：引用重建失败回滚测试，对应 B12/G1。

## 4. 动态业务表

规则 11-030：动态表名由 `lc_{app_code}_{object_code}` 生成，必须经过关键字、长度、字符、冲突校验。

规则 11-031：Schema Sync 只允许建表、加列、加索引、扩大长度/精度；删列、缩列、改类型生成危险变更工单，不自动执行。

规则 11-032：22 种 field_type 到 MySQL 类型必须由 ADR-LOWCODE-FIELDTYPE-SPI-001 的 `FieldTypeHandler` 注册表维护，DDL 生成、运行时转换和导入导出共用同一处理器能力声明。

规则 11-033：link 字段默认落库为 `{field_code}_lid varchar(26)`，必须建普通索引，不建物理外键。`{field_code}_id bigint` 只能作为可重建派生列用于内部查询优化，不得作为跨环境契约。

规则 11-033a：link/user/org 字段写入时必须校验目标记录与当前记录同 `tenant_id`，展开查询目标表时也必须注入目标表 `tenant_id = currentTenant`。平台管理面跨租户引用默认禁止，确需支持必须单独 ADR。
强制方式：跨租户目标 lid 写入拒绝测试 + 展开查询 SQL 断言，对应 B4。

规则 11-034：table 子表使用独立动态表，带父记录引用列；列表查询默认不展开。

## 5. 索引规则

规则 11-040：多租户业务查询索引 tenant_id 必须在前缀内；典型顺序：`tenant_id, deleted, object-specific-filter, id`。

规则 11-041：单表普通索引数量 M0/M1 默认不超过 8 个；超过必须有查询证据和 ADR/评审记录，达到 12 个直接阻断发布。

规则 11-042：`in_filter=true` 字段自动建索引有上限，超过上限需用户选择高频筛选字段。

规则 11-043：软删除唯一约束统一采用业务键 + `delete_token` 墓碑列方案：未删除记录 `delete_token=0`，删除时更新为该行 `id` 或删除批次唯一值。禁止把 nullable `deleted_at` 放入唯一索引，因为 MySQL 唯一索引允许多个 NULL，会绕过未删除记录唯一性。

规则 11-044：multilink/many_to_many 必须生成独立中间表，命名 `lc_rel_{app_code}_{object_code}_{field_code}`；中间表必须包含 `id/tenant_id/app_id/object_id/source_lid/target_lid/revision/deleted/deleted_at/delete_token/create_time/create_by/update_time/update_by`。
强制方式：中间表 DDL 生成测试，对应 D6。

规则 11-045：中间表必须登记进 `lc_rt_physical_schema`，并进入备份恢复对象清单；唯一约束使用 `(tenant_id, app_id, source_lid, target_lid, delete_token)`。
强制方式：Schema Sync 登记表测试 + 备份清单扫描，对应 D6/F6。

规则 11-046：code 唯一性必须统一为大小写敏感语义：元数据 code 列使用 `_bin` collation 或应用层二进制判重，Java 判重必须与 DB 一致。禁止依赖默认 `ci` collation 判断 code 唯一。
强制方式：`Abc`/`abc` 判重测试，对应 D8。

规则 11-047：雪花 ID 生成器必须定义时钟回拨策略，首版采用“短回拨等待、超过阈值拒绝服务并告警”；ULID/lid 只作为稳定标识，不承诺全局时间排序，业务排序必须使用 create_time 或业务时间列。
强制方式：时钟回拨单测 + 禁止按 lid 排序扫描，对应 D8。

## 6. Flyway 与 DDL

规则 11-050：Flyway 只管理平台静态表；动态表由 Schema Sync 管理。

规则 11-051：已合并 migration 禁止修改，只能新增。

规则 11-052：迁移文件名 `V{yyyyMMddHHmm}__{english_description}.sql`，禁止中文文件名。

规则 11-053：生产 DDL 必须先生成 DDL Plan 并写 DDL 日志。

## 7. 容量与备份

规则 11-060：M0/M1 共享库容量基线固定为：单租户动态表预警线 300 张、阻断线 500 张；单 MySQL 实例动态表预警线 20000 张、阻断线 50000 张；单动态表行数预警线 500 万、阻断线 1000 万；单表普通索引预警线 8 个、阻断线 12 个；单对象元数据 JSON 预警线 1MB、阻断线 4MB。
强制方式：容量配置测试 + 发布前容量检查，对应 A3/C5。

规则 11-061：备份恢复对象必须包含 `lc_meta_*`、`lc_rt_*`、动态业务表、物理结构登记表、插件表、文件存储引用。

规则 11-062：恢复后必须强制运行 Reconciler，差异未处理前运行态只读或阻断。

## 8. field_type 到 MySQL 类型映射

| field_type | 默认 MySQL 类型 | 索引策略 | 说明 |
|---|---|---|---|
| text | varchar(255) | 可按 in_filter 建普通索引 | maxLength 可调，最大 1024 |
| textarea | text | 不建普通索引 | 仅全文检索场景建 ft |
| richtext | mediumtext | 不建索引 | 必须 XSS 清洗 |
| code | mediumtext | 不建索引 | 代码文本，不是业务 code |
| integer | bigint | 可索引 | 溢出拒绝 |
| decimal | decimal(p,s) | 可索引 | 默认 decimal(18,4) |
| percent | decimal(9,4) | 可索引 | 0~100 |
| currency | decimal(18,4) + `{field}_currency` varchar(8) | 金额可索引 | 币种列按需生成 |
| date | date | 可索引 | 查询边界应用层处理 |
| datetime | datetime(3) | 可索引 | 存储统一时区 |
| time | time(3) | 少用索引 | 仅时间点 |
| select | varchar(64) | 常建索引 | 存 option code |
| multiselect | json | 不直接索引 | 需要筛选时另建倒排/关联表，首版可限制 |
| checkbox | tinyint | 可索引 | 0/1 |
| link | varchar(26) | 必建索引 | 指向目标记录 ULID，即 lid |
| table | 子表 | 父 lid 必建索引，父 id 可作派生优化 | 不在父表存 JSON |
| multilink | 中间表 | 两端索引 | 不存 JSON 数组 |
| autonumber | varchar(64) | 唯一索引 | 业务单号允许跳号 |
| user | varchar(26)，列名 `{field_code}_lid` | 可索引 | 指向用户记录 lid |
| org | varchar(26)，列名 `{field_code}_lid` | 可索引 | 指向组织记录 lid |
| attachment | json 或关联表 | 不直接索引 | 存附件 id 列表/引用 |
| formula | 不落库或按结果类型 | 视结果而定 | 首版默认不落库 |

规则 11-080：本映射表是人工可读索引，不是第二套事实源；机器事实源是 `FieldTypeHandler` 注册表和 `test-contracts/field-type-conversion` 契约资产。Schema Sync、运行时转换、导入导出必须引用同一处理器注册表。

## 9. 标准表结构模板

### 9.1 元数据表模板

```sql
create table lc_meta_object (
  id bigint primary key comment '主键',
  tenant_id bigint not null comment '租户ID',
  app_id bigint not null comment '应用ID',
  code varchar(64) not null comment '对象编码',
  name varchar(128) not null comment '对象名称',
  table_name varchar(128) not null comment '物理表名',
  object_type tinyint not null comment '对象类型',
  status tinyint not null comment '状态',
  fields json not null comment '字段定义JSON，含_v',
  relations json null comment '关系定义JSON，含_v',
  states json null comment '状态定义JSON，含_v',
  actions json null comment '动作定义JSON，含_v',
  rules json null comment '规则定义JSON，含_v',
  options json null comment '对象配置JSON，含_v',
  revision bigint not null default 0 comment '乐观锁版本',
  deleted tinyint not null default 0 comment '是否删除',
  deleted_at datetime(3) null comment '删除时间',
  delete_token bigint not null default 0 comment '软删除唯一约束墓碑；未删除为0，删除后写唯一值',
  create_time datetime(3) not null comment '创建时间',
  create_by bigint not null comment '创建人',
  update_time datetime(3) not null comment '更新时间',
  update_by bigint not null comment '更新人',
  unique key uk_lc_meta_object_app_code_alive (tenant_id, app_id, code, delete_token),
  key idx_lc_meta_object_tenant_app_status (tenant_id, app_id, status)
) comment '元数据对象表';
```

## 10. 物理结构登记表

规则 11-090：Schema Sync 必须维护物理结构登记表，避免每次全量扫 information_schema。

```sql
create table lc_rt_physical_schema (
  id bigint primary key comment '主键',
  tenant_id bigint not null comment '租户ID',
  app_id bigint not null comment '应用ID',
  object_code varchar(64) not null comment '对象编码',
  table_name varchar(128) not null comment '物理表名',
  column_name varchar(128) not null comment '列名',
  column_type varchar(128) not null comment '列类型',
  nullable_flag tinyint not null comment '是否可空',
  field_code varchar(64) null comment '字段编码',
  schema_version varchar(64) not null comment '结构版本',
  create_time datetime(3) not null comment '创建时间',
  update_time datetime(3) not null comment '更新时间',
  unique key uk_lc_rt_physical_schema_col (tenant_id, app_id, table_name, column_name)
) comment '动态物理结构登记表';
```

规则 11-091：登记表损坏时可由 information_schema 重建，但重建操作必须审计。
