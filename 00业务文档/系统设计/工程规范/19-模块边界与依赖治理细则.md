# 19-模块边界与依赖治理细则

> 版本：v0.1（初稿）
> 地位：约束低代码平台模块边界、依赖方向和架构腐蚀防线。本文必须与 `06-规范补全任务书.md`、`07-架构陷阱与高难度问题清单.md` 一起使用。

## 陷阱覆盖表

| 07 条目 | 本文覆盖点 | 强制/检测方式 |
|---|---|---|
| B3 /meta 裁剪与实际执行一致性 | 权限判定器唯一入口，禁止多处复制判定逻辑 | ArchUnit + 代码扫描 + 权限契约测试 |
| C3 MetaGraph 多实例一致性 | 请求级 MetaGraph 引用固定，禁止运行中重新拉取版本 | ArchUnit + 集成测试 |

## 1. 模块职责边界

| 模块 | 允许职责 | 禁止职责 |
|---|---|---|
| `common` | 通用错误、Result、基础工具、不可携带业务语义的基础类型 | 元模型规则、权限规则、动态 SQL、租户特判 |
| `metamodel` | 元对象、字段、关系、版本、引用图、MetaGraph | REST 控制器、动态数据读写、页面渲染 |
| `runtime` | 动态数据 CRUD、查询 AST、权限执行、规则执行、审计 | 设计器页面 schema 编辑、插件安装 UI |
| `designer` | 设计器 REST、页面 schema、发布入口、应用导入导出 | 直接拼接 SQL、绕过 runtime 写业务数据 |
| `expression` | 表达式门面、函数注册、沙箱上下文 | 直接查库、持久化业务数据 |
| `plugin` | 插件生命周期、扩展点注册、兼容检查 | 直接绕过模块 SPI 调用内部实现 |
| `workflow` | 流程定义、实例、任务、流转事件 | 修改元模型物理结构 |

规则 19-001：模块只能通过公开 API/SPI 通信，禁止跨模块引用 `support` 包。
强制方式：ArchUnit `noClasses().that().resideOutsideOfPackage("..{module}..").should().accessClassesThat().resideInAPackage("..{module}.support..")`。

规则 19-002：`runtime` 不得依赖 `designer`，`metamodel` 不得依赖 `runtime/designer/plugin/workflow`。
强制方式：ArchUnit 模块依赖规则。

规则 19-003：`api` 层不得出现 Entity、Mapper、SQL、MyBatis 类型。
强制方式：ArchUnit + Checkstyle import 禁止清单。

## 2. 分层对象使用边界

| 类型 | 允许出现位置 | 禁止出现位置 |
|---|---|---|
| Entity | dao、service 内部持久化转换 | api 出入参、前端契约 |
| DTO | api 请求/响应 | dao mapper 参数 |
| Command | service 写入入口 | controller 直接持久化 |
| Query | 查询条件入口 | 动态 SQL 字符串拼接 |
| Domain Object | 领域规则、状态机、不变量维护 | 序列化成长期兼容契约 |

规则 19-010：Controller 只接收 DTO/Query/Command，不直接接收 Entity。
强制方式：ArchUnit 检查 `..api..` 包方法签名。

规则 19-011：Service 公开方法必须声明事务语义、幂等语义和租户语义。
强制方式：Javadoc 扫描 + 人工评审。

## 3. 架构腐蚀防线

规则 19-020：引擎代码中禁止出现具体租户、应用、对象 code 的业务特判。
反例特征：`if ("tenantA".equals(tenantId))`、`if ("sales_order".equals(objectCode))`。
正规出口：登记需求，评估进入元模型、插件扩展点或配置开关。
强制方式：扫描 `tenantId ==`、`objectCode.equals("...")`、`appCode.equals("...")`，命中必须人工解释。

规则 19-021：表达式、权限、数据范围、字段可见性必须有唯一入口。
强制方式：ArchUnit 限制调用方只能依赖 `PermissionEvaluator`、`DataScopeMerger`、`FieldAccessPolicy` 门面。

规则 19-022：新增跨模块调用必须在 PR 中写明调用方向、契约稳定性和回退方案。
强制方式：PR checklist。

## 4. 依赖准入

规则 19-030：新增依赖必须提交依赖准入表：用途、替代方案、License、CVE 状态、退出成本、运行时影响。
强制方式：PR 模板 + `28-依赖与License合规细则.md` 门禁。

规则 19-031：不得为了一个模块内部需求把依赖提升到父 POM 或 common。
强制方式：依赖变更评审 + Maven dependency tree 检查。

## 5. 验收清单

- [ ] ArchUnit 覆盖模块依赖、层级依赖、跨 support 引用。
- [ ] 扫描脚本覆盖租户/应用/对象字面量特判。
- [ ] 权限、数据范围、字段可见性只有一个判定入口。
- [ ] 新增依赖均可追溯到准入表。
