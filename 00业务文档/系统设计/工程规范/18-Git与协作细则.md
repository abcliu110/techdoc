## 1. 分支与提交

规则 18-001：一任务卡一分支：`feature/T-001-project-skeleton`。

规则 18-002：提交首行写目的，不只写动作；中文不超过 50 字。

规则 18-003：禁止跨任务卡混提，禁止无关格式化混入。

规则 18-004：禁止 force-push main。

## 2. PR 模板

规则 18-010：PR 必填：

```text
任务卡：
变更目标：
设计文档对应章节：
新增依赖及理由：
兼容性影响：
安全影响：
测试证据：
未覆盖风险：
需要人类确认：
回滚方案：
```

规则 18-011：涉及 API、元模型 JSON、插件 SPI、导入导出包的变更必须附兼容性说明。

规则 18-012：涉及单向门决策必须附 ADR。

## 3. 契约先行

规则 18-020：有下游依赖的任务先提交接口契约 PR：interface、DTO、Javadoc、契约测试桩。

规则 18-021：契约冻结后变更必须通知下游任务卡，并更新影响面。

规则 18-022：下游不得绕过未冻结契约直接依赖实现类。

## 4. 多 AI 协作

规则 18-030：每个执行 AI 开工前声明任务卡、预计修改文件、依赖接口。

规则 18-031：发现他人代码问题只报告或开独立任务，不顺手改无关文件。

规则 18-032：共享模块变更必须单独列出影响面。

规则 18-033：冲突解决以 main 当前契约为准，禁止 `--ours/--theirs` 一把梭。

## 5. 评审重点

规则 18-040：评审先看风险：租户隔离、权限、动态 SQL、DDL、兼容性、幂等、错误恢复。

规则 18-041：评审必须检查是否新增未证明抽象，能删则删。

规则 18-042：测试证据必须覆盖任务风险；核心链路缺双租户/并发/失败路径不得通过。

## 6. 回滚与 hotfix

规则 18-050：每个 PR 必须写回滚方案；涉及 DDL 时必须说明物理结构不可回滚边界。

规则 18-051：hotfix 必须从 main 切分支，修复后补测试和事故说明。

规则 18-052：回滚后必须运行相关 Reconciler/兼容测试，不能只 revert 代码。
