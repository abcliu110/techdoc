# 20-扩展点与插件开发细则

> 版本：v0.1（初稿）
> 地位：约束平台扩展能力，防止商业交付退化为硬编码项目制代码。

## 1. 扩展点目录

| 扩展点 | 稳定级别 | 注册方式 | 禁止事项 |
|---|---|---|---|
| 字段类型 | 稳定 SPI | `FieldTypeHandler` | 绕过统一校验/审计/DDL 映射 |
| 动作按钮 | 稳定 SPI | `ActionPlugin` | 直接写动态表 |
| 校验器 | 稳定 SPI | `ValidatorPlugin` | 读取未授权字段 |
| 表达式函数 | 受限 SPI | `ExpressionFunctionPlugin` | IO、查库、反射、线程创建 |
| 页面块 | 稳定前端扩展 | `BlockRegistry` | 注入全局 CSS 污染 |
| 工作流节点 | 实验 SPI | `WorkflowNodePlugin` | 绕过流程审计 |
| 权限策略 | 受限 SPI | `PermissionPolicyPlugin` | 替换核心租户隔离 |
| 通知通道 | 稳定 SPI | `NotificationPlugin` | 泄露敏感字段 |

规则 20-000：字段类型扩展点以 ADR-LOWCODE-FIELDTYPE-SPI-001 的 `FieldTypeHandler` 为唯一后端 SPI；M0 只允许内置 22 种处理器，M3 前不得开放第三方后端字段类型插件。
强制方式：注册表白名单 + ArchUnit。

规则 20-001：所有扩展点必须有上下文对象，不允许插件自行从 ThreadLocal 或全局单例读取租户/用户。
强制方式：SPI 接口签名检查 + 插件测试。

规则 20-002：扩展点上下文只能暴露白名单字段，禁止直接暴露 Entity、DataSource、Mapper、HttpServletRequest。
强制方式：ArchUnit + API review。

### 1.1 字段类型扩展契约

规则 20-003：字段类型扩展必须实现 `FieldTypeHandler`，不得只注册前端组件或只补 DDL 映射。Handler 必须同时声明 DDL 映射、物理列兼容比较、能力矩阵、options 校验、运行时转换、前端组件 schema 和契约向量。
强制方式：SPI 加载测试 + `test-contracts/field-type-conversion` 契约向量测试。

规则 20-004：新增字段类型必须先提交契约向量，再提交实现；契约向量至少覆盖 DDL 映射、API 合法/非法输入、空串、NULL、前端组件能力和导入导出语义。
强制方式：CI 检查 Handler 与契约向量一一对应。

规则 20-005：字段类型能力由 `FieldTypeCapabilities` 决定，包括 `filterable/sortable/exportable/importable/persisted/auditable/searchable`。调用方只能消费能力标记，不得按 field_type 字符串自行分支。
强制方式：ArchUnit 禁止 runtime/designer 中散落字段类型 switch，例外必须列入 ADR。

## 2. 插件生命周期

规则 20-010：插件必须支持 `install -> enable -> disable -> upgrade -> rollback -> uninstall` 生命周期。
每个阶段必须记录审计日志，包含 pluginId、version、tenantId、operator、traceId、结果。
强制方式：插件生命周期集成测试。

规则 20-011：插件升级必须声明数据迁移计划和兼容范围。
强制方式：插件清单 schema 校验。

规则 20-012：插件禁用后不得留下激活菜单、路由、字段组件、表达式函数。
强制方式：禁用集成测试 + 前端注册表断言。

## 3. 插件隔离

规则 20-020：插件数据库表必须使用 `lc_plg_{plugin_code}_*` 前缀，且带 `tenant_id`。
强制方式：Flyway SQL 扫描。

规则 20-021：插件前端资源必须注册到插件命名空间，禁止覆盖平台全局组件 key。
强制方式：注册表 key 冲突检测。

规则 20-022：插件配置必须声明类型、默认值、敏感标记和租户覆盖规则。
强制方式：插件 manifest schema 校验。

## 4. 安全边界

规则 20-030：插件不得直接访问动态业务表，只能通过 runtime 门面执行数据操作。
强制方式：ArchUnit 禁止插件包引用 Mapper/SQL builder。

规则 20-031：表达式函数插件默认不得查库；确需数据时，由 runtime 预取并注入只读 DTO。
强制方式：白名单评审 + 函数单测。

规则 20-032：插件不得关闭租户隔离、字段权限、数据范围条件。
强制方式：安全测试集。

## 5. 兼容与测试

规则 20-040：插件 manifest 必须声明 `minPlatformVersion`、`maxTestedPlatformVersion`、`apiLevel`。
强制方式：manifest schema 校验。

规则 20-041：每个插件必须提供安装、升级、禁用、卸载、双租户隔离、权限拒绝测试。
强制方式：插件 CI 模板。

## 6. 插件 manifest 草案

规则 20-060：插件必须提供 `plugin.json`，字段不得少于：

```json
{
  "pluginCode": "crm-plus",
  "name": "CRM Plus",
  "version": "1.2.0",
  "minPlatformVersion": "1.0.0",
  "maxTestedPlatformVersion": "1.3.x",
  "apiLevel": "stable-1",
  "license": "Commercial",
  "entrypoints": {
    "backend": "com.lowcode.plugins.crm.CrmPlugin",
    "frontend": "src/index.ts"
  },
  "extensionPoints": [
    {"type": "fieldType", "code": "customer_score"},
    {"type": "action", "code": "assign_sales"}
  ],
  "permissions": [
    "runtime.record.read",
    "runtime.record.update"
  ],
  "configSchema": {
    "scoreThreshold": {"type": "integer", "default": 80}
  },
  "migrations": [
    {"version": "1.1.0", "script": "V110__add_score_table.sql"}
  ]
}
```

规则 20-061：manifest 中的 extension point code 必须带插件命名空间或全局唯一前缀。

## 7. 后端 SPI 模板

```java
public interface LowcodePlugin {
  PluginDescriptor descriptor();

  void register(PluginRegistry registry);

  default void onInstall(PluginInstallContext context) {}

  default void onEnable(PluginRuntimeContext context) {}

  default void onDisable(PluginRuntimeContext context) {}
}
```

规则 20-070：SPI 上下文必须包含 tenantId、appCode、pluginVersion、traceId；不得暴露 DataSource、Mapper、HttpServletRequest。

## 8. 前端注册模板

```ts
export function registerPlugin(registry: PluginRegistry) {
  registry.fieldTypes.register(customerScoreField);
  registry.actions.register(assignSalesAction);
}
```

规则 20-080：插件前端注册失败不得阻断整个平台启动，应隔离到插件并告警。

规则 20-081：字段类型前端组件注册必须引用后端 `FieldTypeHandler` 输出的组件 schema，禁止前端自行扩展未被后端 Handler 声明的 props。
强制方式：前后端契约测试。

## 9. 插件迁移规则

规则 20-090：插件迁移只能操作插件自有表 `lc_plg_{plugin_code}_*`，不得直接修改平台核心表和动态业务表。

规则 20-091：插件如需扩展业务对象，必须通过元模型 API 提交变更，并进入发布流程。

规则 20-092：插件卸载默认不物理删除数据，只禁用能力并保留归档；物理清理必须人工确认。

## 10. 业务对象扩展与应用包生命周期

规则 20-100：业务对象扩展必须通过 ObjectExtension 元模型提交，禁止插件或项目代码直接修改标准对象 JSON。扩展必须声明 `baseObjectCode`、`sourceKind`、`packageCode`、`packageVersion`、`extensionType` 和冲突策略。
强制方式：元模型 API 校验 + 插件迁移扫描。

规则 20-101：禁止覆盖系统字段、租户隔离字段、审计字段、发布状态字段、权限不变量和标准对象声明的 `final` 字段。命中必须阻断发布。
强制方式：禁止覆盖清单测试。

规则 20-102：标准对象、行业模板或插件升级前必须生成扩展合并报告，列出新增、覆盖、冲突、废弃和需人工处理项；冲突不得静默自动合并。
强制方式：升级回放测试。

规则 20-103：M0/M1 只允许模板复制和受控扩展层，不允许无限层级继承；父子单据类型继承必须等待 ADR-LOWCODE-OBJECT-EXT-001。
强制方式：元模型校验拒绝多层继承。

规则 20-104：扩展字段、扩展规则、扩展页面、扩展动作必须进入发布流程、Version.snapshot、lc_meta_ref 和兼容检查；禁止安装后直接运行。
强制方式：发布前引用检查 + 快照断言。

规则 20-110：应用包、行业模板包、插件包必须有 manifest，声明 packageCode、version、dependencies、license、migrations、objects、extensions、menus、reports、permissions 和 compatibility。
强制方式：manifest schema 校验。

规则 20-111：包安装、升级、回滚、卸载必须记录审计事件，包含 packageCode、sourceVersion、targetVersion、operator、tenantId、traceId 和结果。
强制方式：包生命周期集成测试。

规则 20-112：License 校验不得作为绕过审计或删除数据的理由。License 过期只能进入降级策略，不得破坏已有业务数据和审计链。
强制方式：License 过期降级测试。

规则 20-113：第三方后端代码默认 M0/M1 禁止；如 M3 开放，必须先补隔离、签名、License、依赖、漏洞扫描和权限边界 ADR。
强制方式：包安装拒绝未签名后端代码。

规则 20-114：应用包安装不得直接写动态业务表或绕过发布流程；包内对象、字段、权限、菜单、报表和打印模板必须先进入草稿元数据，再通过发布生成运行态快照。
强制方式：包安装流程集成测试。

规则 20-115：包依赖必须在安装和升级前解析完整，包括平台版本、apiLevel、插件依赖、行业模板依赖和 License 能力依赖；缺失依赖必须阻断，不得部分启用。
强制方式：依赖解析测试。

规则 20-116：包卸载默认只禁用能力和解除菜单入口，不物理删除业务数据、审计、历史单据和客户扩展；物理清理必须有单独任务、备份和人工确认。
强制方式：卸载预检和数据保留测试。

## 10. 插件安全测试

| 测试 | 断言 |
|---|---|
| 插件直接访问 Mapper | ArchUnit 失败 |
| 插件函数查库 | 白名单评审失败 |
| 插件禁用 | 菜单、路由、字段组件、表达式函数全部不可用 |
| 双租户安装 | A 租户插件配置不影响 B 租户 |
| 插件升级失败 | 回到升级前启用状态或进入失败待修复状态 |
