# 22-发布迁移与回滚细则

> 版本：v0.1（初稿）
> 地位：约束元数据发布、动态 DDL、环境迁移和生产回滚。

## 陷阱覆盖表

| 07 条目 | 本文覆盖点 | 强制/检测方式 |
|---|---|---|
| A1 DDL 非原子 | 发布状态机、DDL 日志、Reconciler | 集成测试 + 运维演练 |
| A2 DDL 不可回滚 | 回滚语义与阻断检查 | 回滚预检器 |
| A3 动态表爆炸 | 物理结构登记表、容量上限 | 指标告警 |
| A4 JSON 演进 | 快照升级与发布兼容 | 回放测试 |
| A5 DDL 的 MDL 元数据锁雪崩 | 长事务预检、锁等待上限、低峰窗口 | 锁等待集成测试 + 运维演练 |
| A6 发布分布式锁正确性 | 发布互斥、fencing token、旧持有者识别 | 并发发布测试 |
| E3 环境漂移 | 导出/导入唯一合法通道 | 发布门禁 |

## 1. 发布状态机

规则 22-001：发布流程必须落库状态，禁止只靠内存流程推进。
状态集合：`DRAFT`、`VALIDATING`、`DDL_PLANNED`、`DDL_RUNNING`、`PARTIAL_APPLIED`、`PUBLISHED`、`FAILED_NEED_REPAIR`、`ROLLED_BACK`。
强制方式：状态机单测。

规则 22-002：任何失败状态必须能回答：已执行哪些 DDL、元数据版本是什么、缓存版本是什么、下一步自动恢复还是人工介入。
强制方式：失败注入测试。

## 2. DDL 计划与执行

规则 22-010：Schema Sync 执行前必须生成 DDL Plan，不允许边 diff 边执行。
DDL Plan 必含：目标表、操作类型、SQL、风险等级、是否在线、预估耗时、回滚说明。
强制方式：发布接口契约测试。

规则 22-011：生产 DDL 必须先写入 `lc_rt_ddl_log`，成功/失败都更新结果。
强制方式：集成测试断言日志。

规则 22-012：大表 DDL 必须判断 `INSTANT/INPLACE/COPY` 能力，COPY 类默认阻断。
强制方式：DDL Plan 检查器。

规则 22-013：生产 DDL 前必须执行 MDL 风险预检，检查目标表上的长事务、慢查询和未提交事务；命中阈值时阻断发布或延迟到低峰窗口。
强制方式：Testcontainers MySQL 中模拟长事务持锁，再发起发布并断言被阻断，对应 A5。

规则 22-014：DDL 会话必须显式设置 `lock_wait_timeout`，首版上限为 10 秒；超时后发布状态必须进入 `PARTIAL_APPLIED` 或 `FAILED_NEED_REPAIR`，不得在请求线程无限等待。
强制方式：锁等待超时集成测试 + DDL 日志断言，对应 A5。

规则 22-015：DDL Plan 必须标注大表阈值、低峰窗口建议、租户 DDL 配额消耗和预估锁等待风险；共享库下任一租户 DDL 都按租户级噪声资源计入 C5。
强制方式：DDL Plan 契约测试 + 发布 checklist，对应 A5。

规则 22-016：同一 app 的发布必须带 fencing token 或等价单调发布序号；拿到旧锁但 token 过期的执行者不得继续写元数据状态或执行后续 DDL。
强制方式：两个发布者并发、首个发布者 GC/暂停后恢复的集成测试，对应 A6。

## 3. Reconciler 对账器

规则 22-020：发布前、启动后、发布失败后必须运行结构对账。
对账对象：元数据、物理结构登记表、information_schema。
强制方式：启动检查 + 发布检查。

规则 22-021：差异分为可自动补齐、需阻断发布、需人工工单三类。
强制方式：差异分类单测。

规则 22-022：Schema Sync 不得每次全量扫描 information_schema，必须优先使用物理结构登记表。
强制方式：代码审查 + 性能测试。

## 4. 回滚语义

规则 22-030：回滚只承诺元数据、页面、规则、权限回到旧版本；已执行 DDL 和已写入新格式数据不承诺物理回滚。
强制方式：用户文档 + 回滚接口响应说明。

规则 22-031：回滚前必须检查跨越版本内是否包含不可回滚变更。
不可回滚变更：字段类型收窄、选项集收窄、精度降低、删除引用目标、插件数据迁移。
强制方式：回滚预检器。

## 5. 环境迁移

规则 22-040：环境间应用迁移唯一合法通道是导出/导入包，禁止手工 SQL 搬运动动态表结构。
强制方式：发布流程制度 + 审计。

规则 22-041：导入前必须输出预检报告：将新增/修改哪些对象、字段、索引、页面、规则、权限、插件依赖。
强制方式：导入接口契约测试。

## 6. 发布状态机明细

| 当前状态 | 事件 | 下一状态 | 必须落库信息 |
|---|---|---|---|
| `DRAFT` | 提交发布 | `VALIDATING` | operator、traceId、sourceVersion |
| `VALIDATING` | 校验通过 | `DDL_PLANNED` | 校验报告、引用完整性摘要 |
| `VALIDATING` | 校验失败 | `FAILED_NEED_REPAIR` | 错误列表、定位路径 |
| `DDL_PLANNED` | 开始执行 | `DDL_RUNNING` | DDL Plan 版本、预计风险 |
| `DDL_RUNNING` | 部分 DDL 失败 | `PARTIAL_APPLIED` | 已执行序号、失败 SQL id、错误码 |
| `DDL_RUNNING` | 全部成功 | `PUBLISHED` | publishedVersion、metaVersion |
| `PARTIAL_APPLIED` | Reconciler 自动修复成功 | `PUBLISHED` | 修复记录 |
| `PARTIAL_APPLIED` | 无法自动修复 | `FAILED_NEED_REPAIR` | 工单 id、人工建议 |
| `PUBLISHED` | 元数据回滚 | `ROLLED_BACK` | 回滚到版本、不可回滚项说明 |

规则 22-060：状态机更新必须使用 CAS 条件，禁止两个发布流程同时推进同一 app。

```sql
update lc_meta_version
set publish_status = ?, revision = revision + 1
where id = ? and publish_status = ? and revision = ?;
```

## 7. DDL 日志表草案

规则 22-070：必须建立 DDL 执行日志表，字段不得少于：

```sql
create table lc_rt_ddl_log (
  id bigint primary key,
  tenant_id bigint not null comment '租户ID',
  app_id bigint not null comment '应用ID',
  version_id bigint not null comment '发布版本ID',
  plan_id varchar(64) not null comment 'DDL计划ID',
  step_no int not null comment '步骤序号',
  object_code varchar(64) not null comment '对象编码',
  table_name varchar(128) not null comment '物理表名',
  ddl_type varchar(32) not null comment 'DDL类型',
  ddl_sql text not null comment '执行SQL',
  risk_level varchar(16) not null comment '风险等级',
  execute_status varchar(32) not null comment '执行状态',
  error_code varchar(32) null comment '错误码',
  error_message varchar(1024) null comment '脱敏错误信息',
  started_at datetime(3) null comment '开始时间',
  finished_at datetime(3) null comment '结束时间',
  duration_ms bigint null comment '耗时毫秒',
  trace_id varchar(64) not null comment '追踪ID',
  create_time datetime(3) not null comment '创建时间'
) comment '动态DDL执行日志';
```

规则 22-071：`ddl_sql` 只允许记录结构 SQL，不得包含业务数据值或密钥。

## 8. Reconciler 差异分类

| 差异类型 | 示例 | 处理 |
|---|---|---|
| `MISSING_TABLE` | 元数据有对象，物理表不存在 | 未发布版本阻断；发布失败后可自动补建 |
| `MISSING_COLUMN` | 元数据有字段，物理列不存在 | 可自动补列，但必须写 DDL 日志 |
| `EXTRA_COLUMN` | 物理列存在，元数据已无字段 | 保留，标记 orphan，不自动删除 |
| `TYPE_NARROWED` | 元数据精度小于物理列或反向不兼容 | 阻断，人工工单 |
| `INDEX_MISSING` | in_filter 字段索引缺失 | 可自动补索引 |
| `REGISTRY_DRIFT` | 登记表与 information_schema 不一致 | 阻断发布，先修登记表 |

规则 22-080：Reconciler 每次运行必须产出报告，报告可被发布流程、运维界面、测试断言读取。

## 9. 回滚用户可见文案要求

规则 22-090：回滚接口返回必须包含能力边界。

```json
{
  "rollbackVersion": "v12",
  "metadataRolledBack": true,
  "physicalDdlRolledBack": false,
  "notRollbackItems": [
    {"type": "COLUMN_ADDED", "object": "order", "field": "approveNote"}
  ],
  "message": "已回滚元数据、页面、规则和权限。已执行的物理表结构变更不会删除。"
}
```

规则 22-091：如果跨越不可兼容字段语义变更，回滚必须阻断，而不是只提示。
