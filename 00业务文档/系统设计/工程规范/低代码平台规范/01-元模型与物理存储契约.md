## 1. 契约边界

1. `MUST` 表示在与规则匹配的保存、发布、安装、构建、评审或运行时控制点阻断；不能自动检测时必须阻断验收并登记门禁缺口。
2. `SHOULD` 表示原则上遵守，偏离时必须在 ADR 或任务卡中记录理由。
3. 本文的映射表是人工可读索引；可执行事实源是 `FieldTypeHandler` 注册表和 `test-contracts/field-type-conversion` 契约向量。
4. 本文不把 MySQL、API 安全、通用代码生成规则重写一遍；只规定低代码平台特有的稳定契约。

## 2. 稳定标识和业务词表

### 2.1 `id`、`lid` 和 `code`

| 标识 | 用途 | 稳定性 | 禁止用法 |
|---|---|---|---|
| `id` | 数据库内部物理主键 | 仅当前环境内稳定 | 不得进入跨环境包、页面 Schema 或插件长期契约 |
| `lid` | 运行时业务记录稳定标识 | 跨环境稳定 | 不得承诺时间排序语义 |
| `code` | 设计态元模型稳定编码 | 创建后默认不可改 | 不得用显示名或数据库 id 代替 |

- `MUST`：对象、字段、动作、状态、规则、页面、权限、插件和连接器间的长期引用使用 `code`。
- `MUST`：业务记录的跨环境引用使用 `lid`；link 物理列默认为 `{field_code}_lid`。
- `MUST`：显示名使用 `name`，说明使用 `description`；`titleField` 仅表示对象标题字段配置。
- `MUST`：元模型 code 的大小写判重必须与数据库一致，使用二进制校验或 `_bin` collation，不得依赖默认 `ci` collation。

### 2.2 低代码业务词

| 中文 | 稳定英文 | 语义边界 |
|---|---|---|
| 租户 | `tenant` | 不用 customer/account 表示租户 |
| 工作区 | `workspace` | 不与 app 混用 |
| 应用 | `app` | 业务字段不用 application |
| 对象 | `object` | 不用 entity/model/doc 表示平台对象 |
| 字段 | `field` | 不用 column/property/attr 表示元模型字段 |
| 页面 | `page` | PageSchema 仅是页面的结构契约 |
| 状态 | `state` | `status` 仅表示启停状态 |
| 表达式 | `expression` | `formula` 是字段类型 |
| 插件 | `plugin` | 不与通用 extension/addon 混用 |

### 2.3 物理命名

- 元数据表：`lc_meta_*`。
- 运行时系统表：`lc_rt_*`。
- 动态业务表：`lc_{app_code}_{object_code}`。
- 多选关系表：`lc_rel_{app_code}_{object_code}_{field_code}`。
- 插件自有表：`lc_plg_{plugin_code}_*`。
- 应用包：`lowcode-app-{appCode}-{versionNo}-{yyyyMMddHHmm}.zip`。
- 插件包：`lowcode-plugin-{pluginCode}-{pluginVersion}.zip`。

## 3. 元模型存储

### 3.1 JSON 聚合边界

1. `MUST`：核心元数据表使用受控的 `lc_meta_*` 少表 + JSON 聚合方案；新增字段级明细表必须先有 ADR。
2. `MUST`：每个 JSON 聚合列包含 `_v`，列 COMMENT 写明结构版本和 DTO 名。
3. `MUST`：读取时通过 `JsonUpgrader` 升级历史版本；升级器幂等，旧快照可回放。`_v` 高于当前服务支持版本时，写入必须失败关闭；读取若选择兼容降级，未知字段必须原样保留，禁止“忽略后全量写回”导致新字段丢失。**读取降级期间禁止触发任何写回操作**——降级读取产出的 DTO 仅用于只读展示，必须设置 `readOnly=true` 标记；保存流程检测到该标记时直接返回失败并提示“元模型版本降级中，暂不可保存”，避免把降级后的不完整结构全量写回造成数据损坏。
4. `MUST`：禁止局部 `JSON_SET/JSON_REMOVE` 更新。保存必须读完整 DTO、升级、全量校验，再以 `revision` CAS 整行写回；CAS 失败返回冲突。读取降级标记的 DTO 不得进入保存流程（见 3.1.3）。
5. `MUST`：业务查询不使用 `JSON_EXTRACT`。需检索的属性提升为结构化列，或纳入 `lc_meta_ref`。

### 3.2 引用索引

- `MUST`：`lc_meta_ref` 覆盖 object/field/action/state/rule/page/permission/plugin/connector 引用。
- `MUST`：引用集重建与对象 JSON 写回在同一事务内；先生成新引用集，再替换旧引用集。
- `MUST`：发布前做全图引用完整性校验，保存草稿成功不等于可发布。
- `MUST`：对象、字段、动作和状态 code 创建后默认禁止修改；改显示名不改 code。

## 4. 动态物理存储

### 4.1 基本结构

- `MUST`：动态表名经过关键字、长度、字符和冲突校验，不得直接拼接用户输入。表名仅允许 `[a-z0-9_]` 字符（白名单），总长度不超过 64 字节（MySQL 标识符限制），且必须以字母开头；`app_code` 和 `object_code` 在注册时即校验合规，DDL 拼接前再做一次防御性校验，不合规直接失败关闭。
- `MUST`：动态记录同时保有 `id bigint`、`lid varchar(26)`、`tenant_id`、`app_id`、`object_id`、`revision`、软删除列和审计列。
- `MUST`：租户业务索引以 `tenant_id` 作为前缀；典型顺序为 `tenant_id, deleted, business_filter, id`。
- `MUST`：软删除唯一性使用业务键 + `delete_token`。未删除时为 `0`，删除时同事务写入行 id 或删除批次唯一值；不得用 nullable `deleted_at` 规避唯一约束。**单条删除**时 `delete_token` 写入被删行 id（保证唯一键释放且可追溯）；**批量删除**时同一批次内所有行写入相同的批次唯一值（如 `{batchId}` 或 `{tenantId}-{batchSeq}`），批次 id 在批次开始时生成并记录审计；两种模式的判定依据是删除请求是否包含主键列表且数量为 1（单条）或多条（批量），不得由调用方自行选择 token 值。
- `MUST`：link/user/org 写入和展开时同时校验源、目标记录的 `tenant_id`。
- `MUST`：`table` 子表使用独立动态表；`multilink/many_to_many` 使用独立中间表，不在主表存 JSON 数组。

### 4.2 字段类型物理映射

| `field_type` | 默认物理形态 | 关键约束 |
|---|---|---|
| `text` | `varchar(255)` | 物理列默认 `varchar(255)`，`maxLength` 配置项控制业务层最大输入字符数、默认上限 1024；两者职责不同：`varchar(N)` 是 DB 层物理存储上限（建表时根据 `maxLength` 候选值向上取整到常用档位），`maxLength` 是运行时校验上限。`maxLength` 超过 255 时物理列同步扩大（取 `varchar(maxLength)` 或 `text`），但单字段 `maxLength` 不得超过 1024，更长文本使用 `textarea`。 |
| `textarea` | `text` | 默认不建普通索引 |
| `richtext` | `mediumtext` | 按产品保真要求规范化/净化，渲染时仍做输出净化与协议白名单 |
| `code` | `mediumtext` | 表示代码文本，不是元模型 code |
| `integer` | `bigint` | 溢出拒绝 |
| `decimal` | `decimal(p,s)` | 默认 `decimal(18,4)` |
| `percent` | `decimal(9,4)` | 0~100 |
| `currency` | `decimal(18,4)` + 币种列 | 币种列按契约生成 |
| `date` | `date` | 查询边界由应用层转换 |
| `datetime` | `datetime(3)` | 存储时区口径统一 |
| `time` | `time(3)` | 仅时间点 |
| `select` | `varchar(64)` | 存 option code |
| `multiselect` | `json` 或关联表 | 默认不直接索引 |
| `checkbox` | `tinyint` | 0/1 |
| `link` | `{field_code}_lid varchar(26)` | 必建索引，不建物理外键 |
| `table` | 独立子表 | 父 lid 必建索引 |
| `multilink` | 独立中间表 | 两端索引和软删除唯一约束 |
| `autonumber` | `varchar(64)` | 唯一索引，允许跳号 |
| `user` | `{field_code}_lid varchar(26)` | 用户 lid |
| `org` | `{field_code}_lid varchar(26)` | 组织 lid |
| `attachment` | `json` 或关联表 | 不直接索引 |
| `formula` | 默认不落库 | 物化前必须有 ADR |

- `MUST`：Schema Sync、运行时转换和导入导出消费同一 `FieldTypeHandler` 注册表；前端消费由该注册表生成并版本化的能力描述，不复制一份手工类型真值表。能力描述以版本化 JSON Schema 或 TypeScript 声明文件（`.d.ts`）形式导出，文件名包含 `metaVersion` 或 `schemaVersion`，前端按 `metaVersion` 精确匹配加载对应版本；新增/变更字段类型时必须同步更新导出文件版本号并在 CI 中校验前后端版本一致。
- `MUST`：新增 field type 先提交契约向量，覆盖 DDL、合法/非法 API 输入、空串、NULL、组件能力和导入导出语义，再提交实现。
- `MUST`：调用方依据 `filterable/sortable/exportable/importable/persisted/auditable/searchable` 能力标记，不得散落 field type 字符串分支。

## 5. Schema Sync 和物理登记

1. `MUST`：Schema Sync 先生成完整 DDL Plan，不得边 diff 边执行；发布和回滚见 `05-发布安全运维与测试契约.md`。
2. `MUST`：自动变更仅允许建表、加列、加索引、扩大长度/精度。删列、缩列、改类型生成危险变更工单。
3. `MUST`：`lc_rt_physical_schema` 记录 `tenant_id/app_id/object_code/table_name/column_name/column_type/nullable_flag/field_code/schema_version`，唯一键覆盖租户、应用、表和列。
4. `MUST`：动态表、子表、多选关系表均登记并进入备份恢复清单。
5. `MUST`：日常对账优先使用登记表，不每次全量扫描 `information_schema`。登记表可从 `information_schema` 重建，但重建必须审计。

## 6. 元模型兼容和扩展

- `MUST`：元模型 JSON DTO 只允许新增可选字段；禁止删除、改名、改类型或复用旧语义。
- `MUST`：枚举 code 只增不删，不得复用或改变含义。
- `MUST`：历史 Version.snapshot、旧页面 Schema、旧导出包在承诺的兼容窗口内可回放。
- `MUST`：启用“标准对象 + 客户扩展”能力时，标准定义与客户增量必须分层记录，来源和归属可追溯；不得直接修改标准对象 JSON。具体 DTO 名称、字段和存储形态由 accepted ADR 或版本化项目契约决定，本规范不预先锁定 `ObjectExtension` 实现。
- `MUST`：系统字段、租户隔离字段、审计字段、发布状态、权限不变量和 `final` 字段禁止覆盖。
- `MUST`：来源分类、合并优先级和冲突策略必须由 accepted ADR 或版本化项目契约显式规定并可测试，不得由偶然加载顺序决定；ADR 未 accepted 前不得把候选枚举写入公共契约。
- `条件规则`：实现标准对象升级回放、扩展合并或多层继承前，必须先有 accepted 专项 ADR、兼容矩阵和冲突恢复测试；只做字段预留或发布阻断时，不得宣称扩展运行时已经定型。

## 7. 容量与验收

### 7.1 M0/M1 候选护栏与校准门槛

下列数值是当前规划的共享 MySQL 8 部署画像下的候选初始值，不是已批准 ADR、已完成容量认证的结论，也不是 MySQL 或所有私有化部署的通用上限。容量基线或 ADR 验收前，部署必须配置保守护栏，但不得引用本表宣称已证明容量；验收后以版本化部署配置和容量报告为可执行事实源。上线前必须用实际行宽、索引宽度、租户分布、备份恢复时间和峰值查询压测校准。

| 指标 | 预警线 | 阻断线 |
|---|---:|---:|
| 单租户动态表 | 300 | 500 |
| 单 MySQL 实例动态表 | 20,000 | 50,000 |
| 单动态表行数 | 500 万 | 1,000 万 |
| 单表普通索引 | 8 | 12 |
| 单对象元数据 JSON | 1 MB | 4 MB |

超过当前部署已批准的阻断线不得依靠普通人工确认硬发布；必须先有分库、归档、拆对象或独立库方案，并完成恢复演练。候选表格尚未转成已批准部署基线时只能用于预警和压测规划，不能直接充当发布阻断证据。容量门禁还必须报告当前值、预测增长、测量时间和数据来源。

### 7.2 必测证据

- code 大小写判重，时钟回拨，不按 lid 时间排序。
- 元数据 JSON 并发保存不丢更新，`lc_meta_ref` 重建失败整体回滚。
- 22 种 field type 的 DDL 和运行时契约向量一致。
- link 跨租户写入拒绝，展开查询带目标表租户条件。
- 多选关系表 DDL、登记表和备份清单一致。
- 标准对象升级产生扩展合并报告，冲突不静默覆盖。
