# 数据与 API 规范

> 适用范围：关系型数据库、数据迁移、HTTP API、消息或文件形式的外部契约。
> 来源：原 `11-数据库设计细则.md`、`12-API设计细则.md` 的通用部分。
> 不包含：低代码动态表、字段类型注册表、平台元数据图、页面 schema 和固定平台 API 路径。
>
> 版本：v1.1 | 更新日期：2026-07-10

## 1. 约束等级

本规范沿用 `01-命名与编码规范.md` 第1节定义的约束等级：

- `MUST`：违反会破坏契约、安全、可维护性或跨环境兼容，必须阻断合并。
- `SHOULD`：默认遵守；偏离时需在评审中记录理由。
- `条件规则`：仅在项目采用对应数据库、框架或基础设施时生效。

## 2. 设计原则

1. 数据模型是长期兼容资产，API 是调用方依赖的公开契约。两者的破坏性变化必须有版本、迁移和回滚方案。
2. 数据完整性、租户隔离、权限和幂等由服务端保证，不依赖前端约定。
3. 项目级规范必须记录数据库版本、字符集、时区、主键策略、分页策略和兼容窗口。

## 3. 数据库基础规则

1. `MUST`：统一字符集和排序规则；业务 code 的大小写语义必须在应用与数据库中一致。
2. `SHOULD`：生产业务表和非自解释列在数据库或版本化数据字典中具有可追溯说明。不得用注释掩盖模糊命名，也不得为了满足形式要求生成无信息量注释。
3. `MUST`：金额和其他精确数值使用定点数类型，禁止使用浮点类型承载金额。
4. `MUST`：时间字段明确时区语义和精度。存储、接口和展示之间的转换由统一组件处理，不得散落在业务代码中。
5. `MUST`：租户隔离表具有非空租户键，核心查询和唯一约束包含正确的隔离维度。
6. `MUST`：并发更新的数据具有 revision、版本号或等价的乐观/悲观并发控制。
7. `MUST`：软删除策略同时定义查询过滤、唯一性、恢复、审计和清理规则。不得用 nullable 删除时间直接推断唯一性可靠。
8. `SHOULD`：除非项目 ADR 明确证明收益和运维能力，避免触发器、存储过程、数据库枚举和跨服务物理外键。

## 4. 主键、稳定标识与引用

1. `MUST`：需要跨环境迁移、离线交换、合并数据源或隐藏存储拓扑时，内部主键与外部稳定标识分离。仅在当前系统内使用且已承诺稳定的数据库 ID 可以作为外部标识，但必须记录其作用域和兼容边界。
2. `MUST`：引用目标时明确使用内部 ID、稳定 ID 或业务 code；写入和查询采用相同语义。
3. `MUST`：跨租户引用默认禁止。确需跨租户的管理能力必须有单独授权、审计和 ADR。
4. `条件规则`：分布式或时间相关 ID 生成器必须定义冲突、时钟回拨、节点漂移和降级策略；任何 ID 均不得仅凭其实现细节的隐含时间顺序作为业务排序，除非该顺序已成为显式、可测试的契约。

## 5. 索引、容量与备份

1. `MUST`：索引来自真实查询、唯一性或排序需求。多租户核心索引必须把租户维度放在能实际生效的位置。
2. `MUST`：新增索引前评估写放大、存储、选择性和现有重复索引；超出项目阈值需查询证据和评审记录。
3. `MUST`：大批量查询限制分页大小、`IN` 参数数量、导出规模和最大结果集。
4. `条件规则`：承载需恢复的生产数据时，备份清单覆盖恢复目标所需且允许保留的主表、关系表、配置、迁移元数据和对象存储；恢复演练验证业务一致性，而非只验证文件存在。审计数据是否进入同一备份按合规保留与职责分离要求决定。
5. `条件规则`：数据量会持续增长或资源存在明确上限时，容量阈值、增长率、归档与清理策略进入监控和发布评审。

## 6. Schema 迁移

1. `MUST`：生产 schema 变化通过版本化迁移工具执行，禁止手工改表作为正式发布步骤。
2. `MUST`：迁移脚本不可修改已在共享环境执行过的历史版本；修复使用新版本迁移。
3. `MUST`：删列、缩短长度、收窄精度、改变类型、添加非空约束等危险操作必须分阶段迁移，并提供数据扫描、兼容窗口和回滚方案。
4. `MUST`：迁移前评估锁等待、表规模和执行时长；失败必须落入可诊断状态，记录迁移版本与 trace 信息。
5. `MUST`：涉及历史数据转换时提供校验查询或对账结果；仅“迁移命令成功”不构成数据正确性证据。
6. `SHOULD`：DDL 与应用发布解耦为 expand、migrate、contract 阶段，先兼容读写，再清理旧结构。例如：将 NOT NULL 列先以 nullable 新增并赋予默认值（expand），应用切换到读写新列（migrate），最后在兼容窗口后收紧为 NOT NULL 并移除旧列（contract）。

## 7. API 契约

1. `MUST`：API 请求与响应使用稳定、文档化的 schema；数据库实体不得直接作为外部 DTO。
2. `MUST`：机器消费的 API 使用稳定错误码和安全消息；存在跨进程诊断链路时提供 traceId、requestId 或等价关联标识。不得暴露 SQL、堆栈、文件路径、连接串或内部类名。
3. `MUST`：输入校验覆盖类型、长度、范围、枚举、集合大小和嵌套深度；字段错误应包含可定位的路径。
4. `MUST`：HTTP 状态表达协议层结果，业务错误码表达稳定业务语义；不得永远返回成功状态掩盖失败。
5. `MUST`：分页协议明确页码或游标语义、排序稳定性、最大 page size 和总数口径。
6. `MUST`：过滤、排序、字段选择、关联路径和聚合能力采用白名单；用户输入只能作为参数值，不能成为 SQL 或表达式结构。
7. `MUST`：权限和数据范围由服务端合并，不能被客户端过滤条件覆盖或通过 OR 短路。
8. `SHOULD`：API 命名围绕资源与业务动作保持一致；不要仅为贴合内部方法名频繁增加同义 endpoint。

## 8. 写操作、幂等与并发

1. `MUST`：支付、订单、库存、状态流转、导入、消息消费及可重试外部调用在可能重复到达且重复执行会产生错误副作用时具有幂等机制。
2. `MUST`：幂等作用域包含业务资源和隔离维度；同一 key、同一语义重复请求返回首次结果或稳定冲突，不重复产生副作用。
3. `MUST`：同一幂等 key 携带不同请求摘要时拒绝复用。
4. `MUST`：并发更新使用 revision/CAS、行锁或等价机制；冲突返回明确结果，不得静默覆盖。
5. `条件规则`：使用持久化幂等记录的 API 明确保留期、处理中重试语义和失败后的可重试条件；使用唯一约束或状态条件更新时记录等价冲突语义。
6. `条件规则`：需要可靠异步副作用时采用 outbox、事务消息或项目已有等价机制；至少一次投递的消费者必须容忍重复，业务可能乱序时必须定义顺序策略。重试不得绕过权限与租户校验。

## 9. 兼容演进

1. `MUST`：删除字段、改名、改变类型或必填性、缩小枚举集合、改变默认值和错误语义均视为潜在 breaking change。
2. `MUST`：新增必填字段必须提供默认、兼容读写或版本化 endpoint；调用方完成迁移前不得直接收紧。
3. `MUST`：废弃流程包含标记、调用方盘点、迁移窗口、观测和最终移除条件。
4. `MUST`：OpenAPI、JSON Schema、AsyncAPI 或项目等价契约进入版本控制，并在 CI 中做 breaking-change diff。
5. `条件规则`：会持久化、跨进程交换或生命周期可能长于单次部署的动态/半结构化数据必须具有可判定 schema 版本或等价兼容标识，并定义升级、未知字段和降级策略；进程内短生命周期数据不为套规范增加版本字段。

## 10. 导入导出

1. `MUST`：导入复用正常写入口或实现经同一契约测试证明等价的类型校验、权限、租户、审计、不变量和幂等语义；性能优化不得通过批量 SQL 绕过业务边界。
2. `MUST`：导出执行与在线查询一致的字段权限和数据范围裁剪。
3. `MUST`：文件导入限制大小、记录数、压缩比、嵌套深度和允许类型；防止路径穿越、压缩炸弹、CSV 公式注入和恶意表达式。
4. `MUST`：批量任务明确原子提交、遇错即停或部分成功语义，并提供可追踪批次号。允许部分成功时提供逐项结果和安全重试能力；全量回滚时提供可定位的失败原因。
5. `MUST`：导出文件有访问控制、有效期、保留期和删除机制，不记录敏感下载 URL。

## 11. 验收证据

- [ ] Schema 迁移在真实数据库或等价测试环境执行，危险操作有分阶段方案。
- [ ] 数据隔离、引用完整性、并发冲突和软删除唯一性及恢复策略已有自动化测试。
- [ ] API 契约生成、校验和 breaking-change diff 通过。
- [ ] 恶意过滤、越权、超限分页和错误信息泄露测试通过。
- [ ] 幂等重放、同 key 异请求、处理中重试和消息重复测试通过。
- [ ] 导入导出权限、大小限制、部分失败与文件安全测试通过。
- [ ] 备份恢复或数据对账证据可追溯。
