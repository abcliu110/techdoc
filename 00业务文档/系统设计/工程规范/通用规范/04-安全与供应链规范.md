# 安全与供应链规范

> 适用范围：应用安全、数据保护、依赖、开源协议、构建与制品供应链。
> 来源：原 `17-安全编码细则.md`、`28-依赖与License合规细则.md` 的通用部分。
> 不包含：特定平台表达式、插件和应用包市场的具体实现契约。
>
> 版本：v1.1 | 更新日期：2026-07-10

## 1. 约束等级与术语

本规范沿用 `01-命名与编码规范.md` 第1节定义的约束等级（`MUST`、`SHOULD`、`条件规则`）。以下为本文件使用的补充术语：

| 术语 | 定义 |
|---|---|
| SSRF（Server-Side Request Forgery） | 服务端请求伪造，攻击者诱导服务端向非预期目标发起网络请求的攻击手法，可探测内网、云 metadata 或其他受限资源。 |
| CSRF（Cross-Site Request Forgery） | 跨站请求伪造，攻击者诱导已认证用户的浏览器在不知情下发起写操作的攻击手法。 |
| ADR（Architecture Decision Record） | 架构决策记录，以版本化文档记录架构决策的背景、方案、理由和后果。 |
| SBOM（Software Bill of Materials） | 软件物料清单，以标准格式记录制品所含全部依赖及其版本、来源和 License 的可追溯清单。 |
| break-glass | 紧急绕过常规安全控制的人工干预机制，须有完整审计和事后审查。 |

## 2. 信任边界

1. `MUST`：外部请求、文件、消息、配置、数据库内容和第三方响应均视为不可信输入，按类型、长度、范围、结构和业务权限校验。
2. `MUST`：认证只确认身份，授权必须在每个服务端资源入口执行；前端隐藏、网关路由和 URL 不构成授权。
3. `MUST`：安全控制集中在少量可审计入口，业务模块不得复制后逐渐产生不同判定语义。
4. `MUST`：失败默认拒绝访问。身份、租户或权限上下文缺失时不得降级为全量查询或匿名高权限。

## 3. 注入与查询安全

1. `MUST`：SQL 值使用参数化绑定。表名、列名、排序、运算符、函数和关联路径等结构部分必须来自服务端白名单。
2. `MUST`：数据范围在结构化查询树中作为不可移除的根约束合并，防止 OR 短路和括号优先级绕过。
3. `MUST`：分页、批量 ID、`IN` 集合、查询深度、关联数量和导出规模设置上限。
4. `MUST`：命令、模板、表达式、路径、LDAP、XML 和日志等其他解释器边界采用对应的安全 API，不得字符串拼接不可信输入。
5. `MUST`：反序列化只接受允许的类型和字段，限制嵌套深度、大小和多态能力。

## 4. 多租户与权限

1. `条件规则`：多租户共享数据时，所有租户业务查询与写入携带租户或等价隔离条件；异步、定时和消息任务显式传播或可靠恢复隔离上下文，不依赖调用线程的裸 ThreadLocal。单租户、租户独占存储和明确全局数据需记录其归属与访问边界，不强制增加租户字段。
2. `MUST`：跨租户管理入口与普通业务入口分离，实施最小授权、原因记录、时效控制和不可篡改审计。
3. `MUST`：对象、字段、动作和数据范围的读写判定使用同一授权策略源。
4. `MUST`：读取关联对象、搜索建议、导出和通知时重新校验目标资源权限，不能复用源资源授权。
5. `MUST`：用户禁用、移出租户、调岗、角色撤销或权限变更后，token 与权限缓存按项目声明的时限失效。
6. `MUST`：超级管理员不得默认绕过租户隔离、字段脱敏、密钥不可见和审计。

## 5. 认证与会话

1. `MUST`：会话生命周期定义签发、过期、续期、撤销和强制失效策略。
2. `MUST`：浏览器 cookie 会话的写操作启用 CSRF 防护；Cookie 设置适当的 `HttpOnly`、`Secure` 和 `SameSite` 属性。
3. `MUST`：登录包含暴力破解防护、会话固定防护和安全审计；具体阈值由风险与身份系统能力决定。
4. `MUST`：开放 API token 与浏览器会话分离，使用最小 scope、过期、轮换、限流和审计。
5. `MUST`：密码和凭据只通过成熟算法与密钥管理服务处理，禁止自制加密和可逆存储密码。

## 6. 密钥、个人信息与日志

1. `MUST`：密钥、token、连接凭据和支付凭证不得硬编码或提交到仓库；使用受控配置和密钥管理系统，运行时按最小范围解密。
2. `MUST`：个人信息和秘密字段分级登记，并定义采集用途、访问主体、脱敏、保留期和删除/匿名化策略。
3. `MUST`：日志、指标标签、追踪、审计、导出和前端错误上报不得记录密码、token、密钥、完整身份证件、完整支付信息或无必要的完整业务对象。
4. `MUST`：错误响应不得泄露 SQL、堆栈、路径、连接串和内部类名；当资源存在性属于敏感信息或可被用于枚举时，对外归一“不存在/无权限”语义。内部诊断通过关联标识获取详情。
5. `MUST`：安全关键、权限、管理和高风险业务操作的审计记录包含适用的操作者/代理身份、隔离范围、动作、资源、理由、时间、结果和关联标识；不存在的维度不填充伪值。审计记录必须防止被受审计的同一权限主体静默篡改。

## 7. 文件、富文本与外部连接

1. `MUST`：上传同时校验扩展名、MIME 和实际内容，限制大小与数量；存储 key 由服务端生成，原始文件名只作为不可信展示元数据。
2. `MUST`：下载和预览重新鉴权，使用短期授权；不得在日志或响应中暴露内部 bucket、物理路径或长期签名 URL。
3. `MUST`：富文本和不可信 HTML 使用统一净化策略；附件预览隔离执行环境，阻断脚本、事件属性和危险 URL。
4. `MUST`：外部 HTTP 连接防止 SSRF，默认阻断本机、链路本地、云 metadata、保留地址和 DNS rebinding，重定向及重新解析后再次校验。确需访问私网的内部 Connector 必须走独立类型、目标 allowlist、受控网络出口和审计，不能由普通外部 URL 配置绕过默认策略。
5. `MUST`：外部调用按协议、数据规模和故障模型界定连接/读取/总超时、响应体上限以及并发或限流边界。自动重试只用于可安全重放的操作，并设置次数和总预算；仅在依赖故障模型与调用规模确有需要时采用熔断，并明确打开、探测和恢复语义。

## 8. 导入、导出与通知

1. `MUST`：导入执行与正常写入相同的租户、权限、类型、审计和业务规则，不得直接写库绕过边界。
2. `MUST`：导出执行字段权限、数据范围和敏感字段脱敏，防止 CSV 公式注入；下载文件需要独立鉴权和保留期。
3. `MUST`：通知模板字段使用白名单，敏感字段默认不得出域；收件人必须具有对应资源访问权，除非产品契约明确把通知定义为受审计披露。
4. `条件规则`：承担业务义务或可能自动重试的通知具有幂等、重试上限、失败查询/处置、频率限制和关联标识；允许丢弃的提示类通知必须明确 best-effort 等级，不为其强制建设死信基础设施。

## 9. 依赖准入

新增 Maven、npm、容器镜像、可执行二进制以及会进入分发物的字体、图标、图片或模板资产前必须完成与风险相称的准入。低风险资产可由自动清单记录来源、版本和 License；运行时代码、构建插件、二进制和有网络/文件/执行权限的依赖使用完整记录：

```text
名称与版本：
用途和真实调用点：
现有能力或标准库替代：
License 与分发义务：
维护状态与来源：
已知漏洞：
运行时权限、网络和文件访问：
体积与性能影响：
退出、降级和回滚方案：
```

1. `MUST`：未完成对应风险等级准入的依赖或分发资产不得合并。
2. `MUST`：禁止为了少量工具函数引入大依赖；核心路径依赖必须有退出或降级方案。
3. `MUST`：依赖、插件、构建工具和生产基础镜像锁定版本；生产镜像优先锁定 digest。禁止 floating version。
4. `MUST`：只从受信任仓库获取制品并校验完整性或签名；禁止个人网盘、来源不明 release 和未校验 URL。
5. `MUST`：未经审查的 jar、war、zip、exe、dll 等二进制不得提交仓库；例外需 ADR、来源、哈希和扫描证据。

## 10. License 与知识产权

1. `MUST`：组织维护允许、限制、默认禁止和需法务确认的协议清单，并由自动化工具检查。
2. `MUST`：强 copyleft、服务端网络分发条款、来源不明或自定义商业协议不得凭工程人员判断进入闭源商用核心，必须经过法务确认。
3. `MUST`：前端资产、字体、图标、示例数据、模板和文案与代码依赖同等纳入 License 清单。
4. `MUST`：允许学习公开架构思想和交互模式，禁止复制受限制源码、独特注释、测试样例、图片、文案和可识别表达。
5. `SHOULD`：竞品研究先沉淀为记录来源、抽象原则、不可复制内容和重新设计方案的知识卡，再用于独立实现。

## 11. 构建与漏洞响应

1. `MUST`：会构建、部署或对外分发的软件制品在 CI/发布流程中执行适用的秘密、漏洞、License 和完整性检查，并生成或汇总可追溯依赖清单；组织或客户要求 SBOM 时必须生成标准 SBOM。纯文档和无依赖制品不为套规范增加空 SBOM。
2. `MUST`：发布制品可追溯到源码提交、构建环境、依赖锁文件和 SBOM；构建过程不得从未登记位置动态下载依赖。
3. `MUST`：组织定义漏洞分级与响应 SLA。Critical/High 漏洞必须在 SLA 内完成影响评估，并给出修复、缓解或有到期时间的风险接受。
4. `MUST`：依赖升级运行契约、关键业务和安全回归；不得仅凭编译通过完成升级。
5. `MUST`：无法立即修复时记录受影响版本、暴露面、补偿控制、负责人和到期时间，并持续监测。

## 12. 安全最低测试

- [ ] 注入：SQL 结构、排序、过滤、命令、路径、模板和恶意反序列化。
- [ ] 隔离：双租户读写、关联、缓存、异步、导出和通知。
- [ ] 权限：对象、字段、动作、数据范围、关联目标和角色撤销。
- [ ] 认证：CSRF、会话固定、暴力破解、旧 token 失效和跨租户重放。
- [ ] 数据保护：日志、错误、审计、导出、前端上报和保留期。
- [ ] 文件与内容：MIME、路径穿越、超大文件、压缩炸弹、XSS、下载鉴权。
- [ ] 外部连接：SSRF、DNS rebinding、超时、限流和敏感日志。
- [ ] 供应链：秘密、License、CVE、SBOM、锁版本和制品来源。

## 13. AI 工具特有的安全边界

代码和测试的安全门禁由变更风险决定，不因作者是人或 AI 而改变。使用 AI 工具时仅叠加以下工具特有边界：

1. `MUST`：提示词、上下文、附件和工具参数不得包含真实密钥、生产凭据或未经批准出域的敏感数据；模型服务、插件和工具调用遵守数据分级、最小授权和保留期要求。
2. `MUST`：AI 建议的依赖、命令、配置和外部内容在验证前视为不可信输入，仍执行第5节解释器边界和第9节依赖准入，不得自动获得执行或发布权限。
3. `MUST`：AI 输出不得直接触发生产写入、权限变更、密钥操作或不可逆外部副作用；此类动作必须经过与代码生成分离的授权、预览和审计边界。

## 14. 例外与验收

任何安全或供应链例外必须记录：规则、范围、业务理由、威胁分析、补偿控制、批准人、负责人和到期时间。到期未关闭即阻断下一次发布。

- [ ] 威胁边界、数据分级和权限模型已更新。
- [ ] 密钥与敏感信息扫描无未处理发现。
- [ ] 依赖准入、License、CVE 和 SBOM 证据齐全。
- [ ] 对应安全测试在本次提交或发布上通过。
- [ ] 所有例外均未过期并有补偿控制。
