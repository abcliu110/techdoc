## 1. 类型转换总则

规则 23-001：运行时不得猜测用户输入。非法格式必须拒绝，不得静默转换。
强制方式：22 种 field_type 参数化测试。

规则 23-002：空字符串与 NULL 必须按 field_type 真值表处理，禁止各接口自行决定。
强制方式：动态数据 API 契约测试。

规则 23-003：decimal 超精度直接拒绝，不四舍五入入库。
强制方式：decimal 边界测试。

## 2. 时间、金额、布尔

规则 23-010：API 时间统一 ISO-8601 字符串，入库统一按系统约定时区转换，租户时区只影响展示和查询边界。
强制方式：时区测试。

规则 23-011：金额和高精度数值 API 传字符串，后端使用 BigDecimal。
强制方式：OpenAPI lint + Java 类型扫描。

规则 23-012：布尔 API 只接受 true/false，数据库存 TINYINT 0/1，禁止同时接受 `"0"`、`0`、`"false"` 多形态。
强制方式：请求校验测试。

## 3. 查询语义

规则 23-020：用户 filters 与 data_scope 必须在 AST 层合并，data_scope 作为根 AND 的强制子节点。
强制方式：构造 OR 短路越权测试。

规则 23-021：排序字段、过滤字段、返回字段必须来自元数据白名单。
强制方式：SQL 注入测试集。

规则 23-022：深分页必须设上限；超过上限使用游标分页。
强制方式：API 参数校验测试。

规则 23-023：数据范围时间语义必须按对象声明为 `CURRENT_ORG` 或 `OWNER_SNAPSHOT`；默认值为 `OWNER_SNAPSHOT`。对象未显式声明时按 `OWNER_SNAPSHOT` 执行，显式声明为 `CURRENT_ORG` 时按当前组织关系执行。
强制方式：元数据校验 + 权限集成测试，对应 B7；M1 权限链实现前必须覆盖默认值和对象级覆盖两类用例。

规则 23-024：记录创建、流转、转交时必须保存 `owner_user_lid`、`owner_dept_lid`、`owner_org_path` 快照；组织迁移不得直接重写历史快照，除非通过批量修复流程并写审计。
强制方式：调岗/部门合并/组织树移动测试，对应 B7。

## 4. 展开语义

规则 23-030：列表页 link 展开必须批量化，禁止逐行点查。
强制方式：SQL 次数断言测试。

规则 23-031：默认展开深度最多一层；超过必须显式声明且受对象种类和总行数上限约束。
强制方式：API 契约测试。

规则 23-032：table 子表列表场景默认不加载，详情场景按需加载。
强制方式：接口响应测试。

## 5. 软删除与引用

规则 23-040：软删除记录默认不参与列表查询；link 指向已删记录时返回 tombstone 摘要，不返回完整敏感字段。
强制方式：软删除集成测试。

规则 23-041：unique 约束必须使用业务键 + `delete_token` 墓碑列；未删除记录 `delete_token=0`，软删除时同事务写入该行 `id` 或删除批次唯一值。禁止使用 nullable `deleted_at` 参与唯一索引。
强制方式：DDL 生成测试 + 软删除后重建同业务键集成测试。

规则 23-042：object_code、field_code 创建后默认禁止修改，只允许修改显示名。
强制方式：元数据服务校验测试。

规则 23-043：tombstone 摘要结构固定为 `{lid, deleted:true, deletedAt, title}`；无目标对象 read 权限或无 title_field 读权限时 `title` 返回空字符串，不返回完整记录或敏感字段。
强制方式：软删除 link 展开契约测试，对应 B10。

规则 23-060：新增 field_type 必须先补 `FieldTypeHandler` 契约向量，再补本表、11 数据库映射、13 字段组件、14 测试矩阵；禁止只改文档或只改代码。

## 7. 查询语义示例

规则 23-070：所有 list 默认条件固定为：

```text
tenant_id = currentTenant
AND deleted = 0
AND data_scope_filter
AND user_filters
```

其中 `data_scope_filter` 与 `user_filters` 必须在 AST 层合并，生成 SQL 时保持括号语义。

规则 23-071：link 展开只读当前用户对目标对象有 read 权限且目标字段可读的字段。

示例：订单列表展开 `customer.level`：

| 条件 | 返回 |
|---|---|
| 有 customer read + level read | 返回 level |
| 无 customer read | 不返回展开字段，并记录权限拒绝摘要 |
| 有 customer read 但无 level read | 不返回 level |
| customer 已软删 | 返回 tombstone 摘要，不返回敏感字段 |

## 8. autonumber 语义

规则 23-080：首版 autonumber 采用租户级 DB 行锁计数器，计数器唯一键为 `(tenant_id, app_id, object_code, field_code, scope_key)`；`scope_key` 默认为 `global`，后续若支持按年/月/组织分段必须进入字段 options 并纳入唯一键。

规则 23-081：业务单号允许跳号；原因包括事务回滚、并发失败、号段预留。首版不采用号段缓存，因此单对象取号吞吐以 DB 行锁为上限；产品文案必须明确“不以连续无间隔作为审计依据”。

规则 23-082：取号必须在业务保存同一事务中记录使用结果，失败可追踪。

规则 23-083：autonumber 只能由统一写入口生成，导入、批量创建和插件写入不得传入外部单号覆盖系统生成值；确需保留历史单号必须使用单独迁移字段。
强制方式：导入/插件写入契约测试，对应 B3/C4/F10。

## 9. 审计 diff 语义

规则 23-090：审计 diff 只记录发生变化的字段，字段名使用 field_code，显示时再按当前或历史元数据解析 name。

规则 23-091：richtext、attachment、大 JSON 字段 diff 必须截断或摘要化，禁止记录两份全文。

规则 23-092：字段无读权限的用户查看审计时，也不得通过 diff 泄露字段值。

## 10. formula 与 fetch_from 语义

规则 23-100：M0 formula 默认采用读时求值，不落库，不承诺过滤、排序、分页、导出和聚合下推；任何在这些场景使用 formula 的配置必须在发布前被能力检查器阻断或降级为不可选字段。
强制方式：formula 过滤/排序/导出阻断测试，对应 D4。

规则 23-101：formula 保存时必须构建依赖图并检测循环依赖，依赖边必须进入 `lc_meta_ref`；跨对象 link 链最大深度首版为 2，超过阻断发布。
强制方式：formula 引用 formula、跨对象 link 循环测试，对应 D4/G1。

规则 23-102：formula 引用的源字段被删除、改类型、改权限时，发布必须阻断并输出影响报告；不得运行时静默返回 null。
强制方式：源字段变更发布阻断测试，对应 D4。

规则 23-103：formula 结果权限遵循最小权限原则：任一引用字段对当前用户不可读，则 formula 结果不可见，除非 formula 明确声明为脱敏聚合且通过安全评审。
强制方式：字段权限组合测试，对应 D4/B3。

规则 23-104：如后续版本引入物化 formula，必须补 ADR，明确重算时机、批量事务边界、失败状态、告警和回滚边界；未补 ADR 前禁止 `persisted=true`。
强制方式：元数据校验测试，对应 D4/E1。

规则 23-110：fetch_from 首版语义定为“写入时快照落列”；源记录后续变化不会自动改写历史记录。产品文案必须说明历史数据按快照展示。
强制方式：源记录改名后历史记录不漂移测试，对应 D5。

规则 23-111：fetch_from 快照列命名 `{link_field_code}_{source_field_code}_snap`，类型按源字段结果类型映射；快照列不得作为源对象权限绕过入口。
强制方式：DDL 生成测试 + 字段权限测试，对应 D5。

规则 23-112：源记录软删除后，已落地 fetch_from 快照可继续显示，但必须按当前记录字段权限裁剪；用户无当前记录 read 权限时不得借快照泄露。
强制方式：软删目标/无权限用户显示测试，对应 D5/B3/D2。

规则 23-113：fetch_from 链首版只允许一层，不允许 fetch_from 引用另一个 fetch_from；需要更深链路必须进入显式任务卡和性能评审。
强制方式：元数据校验测试，对应 D5/C1。

## 11. multilink 与级联语义

规则 23-120：multilink 修改的审计只记录集合 diff（added/removed 的 target_lid 列表和显示摘要），不得逐行暴露中间表实现细节。
强制方式：审计日志断言测试，对应 D6。

规则 23-121：on_delete=cascade/set_null 必须先做全链预检，再执行任何写入；预检发现 restrict、权限不足、级联深度超限或行数超限时整体阻断。
强制方式：混合链 cascade->restrict 预检测试，对应 D7。

规则 23-122：级联深度首版最大 3 层，单次级联影响行数默认上限 1000；超限必须阻断并提示人工处理，不得静默截断。
强制方式：超限级联阻断测试，对应 D7/C5。

规则 23-123：级联操作默认走统一写入口，触发权限、审计、规则和 revision；性能豁免必须列入白名单并写审计，不得由调用方自行绕过。
强制方式：级联触发审计/规则测试，对应 D7/F1。

规则 23-124：cascade 首版执行软删除，set_null 写空引用列；硬删除仅允许清理过期系统垃圾数据，不作为业务级联策略。
强制方式：级联软删除集成测试，对应 D7/D2。

## 12. 业务数据导入导出

规则 23-130：业务数据导入必须声明部分成功语义。首版采用分批事务，每批最多 500 行；失败行进入错误报告，成功批次保留，重导必须通过幂等键去重。
强制方式：部分失败重导幂等测试，对应 F10/F1。

规则 23-131：导入必须走统一写入口，复用 field_type 真值表、权限判定、租户注入、状态 CAS、审计和规则策略；任何导入豁免规则必须进入审计。
强制方式：导入绕过校验/权限的安全测试，对应 F10/H1。

规则 23-132：导入文件必须流式解析，禁止全量读入内存；单次导入行数、文件大小和并发任务数纳入租户级配额。
强制方式：大文件导入内存上限测试 + 租户配额测试，对应 F10/C5。

规则 23-133：导出必须复用 B3 字段判定器和 B2 data_scope 注入；导出链路不得另写权限裁剪逻辑。
强制方式：越权字段导出裁剪测试，对应 F10/B2/B3。

规则 23-134：CSV/Excel 导出必须防公式注入，单元格以 `=`, `+`, `-`, `@` 开头时按安全策略转义；导出文件必须有保留期、下载鉴权和审计。
强制方式：恶意 CSV 公式注入测试，对应 F10/B6。

## 13. 单据转换、反写与引用追踪

规则 23-150：单据转换必须建模为 `Conversion` 元数据，不得用普通 action 脚本或页面按钮 JSON 隐式表达。`Conversion` 至少声明 source_object、target_object、field_mapping、line_mapping、filter_condition、idempotency_key_strategy 和 permission_policy。
强制方式：元数据校验测试，对应 D10。

规则 23-151：M0 只允许保存 Conversion/WriteBack/LinkTrace DTO 和进入版本快照，不执行转换、不生成反写、不展示上查下查结果；任何运行时执行入口必须被能力检查器阻断。
强制方式：发布阻断测试，对应 D10/DEC-REQ-007。

规则 23-152：单据转换执行必须走统一写入口，复用租户隔离、字段类型转换、权限判定、状态 CAS、规则引擎、审计和幂等键；禁止转换链路直接写动态表。
强制方式：转换绕过统一写入口的 ArchUnit/集成测试，对应 D10/F1/B3。

规则 23-153：反写必须建模为 `WriteBack` 元数据，并显式声明反写方向、聚合口径、触发时机、幂等键、冲突策略和失败补偿策略；禁止在 formula、fetch_from 或触发器中隐藏反写。
强制方式：反写幂等和失败补偿测试，对应 D10/C2/F2。

规则 23-154：引用追踪必须建模为 `LinkTrace`，上查/下查结果必须按当前用户对源单、目标单、字段和数据范围的权限裁剪；不得因为存在引用链而暴露无权限单据是否存在。
强制方式：上查/下查越权裁剪测试，对应 D10/B2/B3/G7。

规则 23-155：转换链路必须限制深度、宽度和单次影响行数；超过阈值必须阻断或转异步任务，不能在同步请求中递归生成无限单据链。
强制方式：链路深度/行数超限测试和指标告警，对应 D10/C1/C5。

规则 23-156：单据转换、反写与引用追踪进入 M2/M3 运行时前必须补 ADR，明确事务边界、反写一致性级别、失败重放、撤销策略、历史版本回放和升级兼容口径；未补 ADR 前不得把 DTO 接入运行时。
强制方式：ADR 门禁 + 运行时入口扫描，对应 D10/E1。

## 14. 弹性域、多组织业务关系与编码规则

规则 23-160：弹性域必须建模为 `FlexField` 元数据，不得把行业字段、客户扩展字段或辅助属性直接塞入无 schema 的 JSON 大字段。`FlexField` 至少声明适用对象、维度、值类型、可见性、校验规则和升级归属。
强制方式：元数据校验测试，对应 D11/D9。

规则 23-161：M0 只允许保存 FlexField/OrgRelation/CodeRule DTO 和进入版本快照，不执行维度组合校验、不参与运行时查询、不生成物理列；任何把弹性域当普通字段运行的配置必须被阻断。
强制方式：发布阻断测试，对应 D11/DEC-REQ-008。

规则 23-162：弹性域参与运行时前必须有维度组合唯一性、默认值、权限裁剪、导入导出、审计 diff 和历史快照语义；不得只实现表单展示。
强制方式：维度组合、导入导出和审计测试，对应 D11/F10/C2。

规则 23-163：多组织业务关系必须建模为 `OrgRelation`，显式区分行政组织、业务组织、核算组织、库存组织、销售组织等关系类型；禁止复用单一 `org_id` 字段表达所有业务组织语义。
强制方式：组织关系元数据校验和权限测试，对应 D11/B7。

规则 23-164：涉及组织关系的查询、编码、权限和报表必须声明使用当前组织还是历史快照；默认不得覆盖 23-023/23-024 的 `OWNER_SNAPSHOT` 历史数据范围语义。
强制方式：组织迁移后历史数据可见性测试，对应 D11/B7。

规则 23-165：编码规则必须建模为 `CodeRule`，并显式声明租户、应用、对象、字段、组织维度、日期维度、流水策略、跳号策略和并发策略；不得用普通 autonumber options 伪装复杂业务编码。
强制方式：组织隔离取号和并发取号测试，对应 D11/C4。

规则 23-166：弹性域、多组织业务关系和复杂编码规则进入 M2/M3 运行时前必须补 ADR，明确物理存储、索引策略、查询下推边界、升级兼容和报表口径；未补 ADR 前不得作为商用运行时能力验收。
强制方式：ADR 门禁 + 查询入口扫描，对应 D11/E1。
