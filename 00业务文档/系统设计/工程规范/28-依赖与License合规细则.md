# 28-依赖与License合规细则

> 版本：v0.1（初稿）
> 地位：约束第三方依赖、开源协议、竞品源码学习边界和供应链安全。

## 陷阱覆盖表

| 07 条目 | 本文覆盖点 | 强制/检测方式 |
|---|---|---|
| B4 沙箱逃逸 | 表达式/插件依赖需安全评估 | 依赖准入评审 |
| E1 兼容纪律 | 依赖升级不得破坏 API/SPI | 回归测试 |
| E2 架构腐蚀 | 禁止复制竞品受限源码形成不可维护分叉 | License 审查 |
| F12 应用包、市场与 License 生命周期 | 应用包依赖、市场分发、授权降级、第三方资产合规 | 包准入 + License 测试 |

## 1. 依赖准入

规则 28-001：新增 jar/npm/docker image 必须提交依赖准入表。
准入表包含：用途、替代方案、License、维护状态、CVE 状态、体积、运行时权限、退出成本。
强制方式：PR checklist + CI 依赖 diff。

规则 28-002：禁止为了少量工具函数引入大依赖。
强制方式：依赖评审。

规则 28-003：运行时核心路径依赖必须有替代或降级方案。
强制方式：架构评审。

## 2. License 策略

规则 28-010：允许协议、限制协议、禁止协议必须列表化并进入 CI。
强制方式：license-check。

规则 28-011：GPL、AGPL、SSPL、强 copyleft 或云服务限制协议默认禁止进入商用闭源核心代码。
强制方式：license-check + 人工法务评审。

规则 28-012：前端图标、图片、字体、示例模板也必须纳入 License 审查。
强制方式：资产清单扫描。

## 3. 竞品源码学习边界

规则 28-020：允许学习公开架构思想、交互模式、API 设计、数据模型思想和测试思路。
规则 28-021：禁止复制受限制协议源码、注释、测试、图片资源、文案和独特表达。
规则 28-022：竞品源码分析输出必须沉淀为抽象知识卡，不得直接形成可粘贴代码片段。
强制方式：知识库卡片审查 + PR KB-REF 检查。

## 4. 漏洞响应

规则 28-030：依赖漏洞按严重等级设响应时限：Critical 24h 内评估，High 72h 内给修复/缓解方案。
强制方式：CVE 扫描 + 安全工单。

规则 28-031：依赖升级必须跑契约测试、插件兼容测试、关键安全测试。
强制方式：CI。

## 5. 锁版本与供应链

规则 28-040：Maven、pnpm、Docker image、JDK、Node、MySQL 都必须锁版本；生产镜像建议锁 digest。
强制方式：CI 扫描。

规则 28-041：构建产物必须能生成 SBOM。
强制方式：CI 产出 SBOM。

规则 28-042：禁止提交未经审查的二进制依赖。
强制方式：文件扫描 + 人工评审。

## 6. 依赖准入表模板

规则 28-060：新增依赖 PR 必须填写：

```text
依赖名称：
生态：Maven / npm / Docker / binary
版本：
用途：
是否运行时依赖：
替代方案：
为什么不用 JDK/现有依赖：
License：
是否传染性协议：
CVE 状态：
维护状态：
包体/镜像大小影响：
权限/网络/文件访问能力：
退出成本：
回滚方式：
```

规则 28-061：未填写准入表的依赖变更不得合并。

## 7. License 分级

| 等级 | 协议 | 策略 |
|---|---|---|
| 允许 | MIT, BSD, Apache-2.0, ISC | 可用，仍需登记 |
| 限制 | MPL-2.0, EPL-2.0, LGPL | 需评估链接方式、修改分发义务 |
| 默认禁止 | GPL, AGPL, SSPL, BUSL 未授权 | 不进入商用闭源核心 |
| 需法务 | 自定义商业协议、来源不明资产 | 法务确认后再用 |

规则 28-070：前端图片、字体、图标、示例模板同样按 License 分级，不得只查 jar/npm。

## 8. 竞品源码学习合规

规则 28-080：竞品源码分析输出必须转化为抽象卡片：

```text
证据来源：
观察到的设计：
抽象出的模型/规则：
可借鉴原则：
不可复制内容：
License 风险：
用于本项目的重新设计：
```

规则 28-081：禁止在本项目提交中出现来自竞品的连续源码片段、独特注释、测试样例、图片资源、文案。

规则 28-082：如果实现中引用知识库，使用 `KB-REF` 指向抽象卡片，不指向竞品源码行作为复制依据。

## 9. SBOM 与漏洞响应

规则 28-090：每次 release 必须产出 SBOM，至少覆盖 Maven、npm、Docker image。

规则 28-091：CVE 响应 SLA：

| 严重级别 | 评估时限 | 修复/缓解时限 |
|---|---|---|
| Critical | 24 小时 | 72 小时 |
| High | 72 小时 | 7 天 |
| Medium | 7 天 | 下个小版本 |
| Low | 30 天 | 计划处理 |

规则 28-092：依赖升级必须运行 OpenAPI 契约测试、插件兼容测试、安全测试集。

## 10. 禁止项

规则 28-100：禁止提交 `.jar/.war/.zip/.exe/.dll` 等二进制依赖，除非有专门 ADR 和制品校验。

规则 28-101：禁止使用 floating version：`LATEST`、`+`、未锁 digest 的生产基础镜像。

规则 28-102：禁止从个人网盘、未知 GitHub release、未校验 URL 下载构建依赖。

## 11. 应用包、市场与 License 生命周期

规则 28-110：应用包、行业模板、插件包、报表包和打印模板包必须登记来源、供应商、版本、License、依赖包、资产清单和升级策略；禁止以手工 SQL、压缩包拷贝或未登记 JSON 导入替代包管理。
强制方式：包 manifest 校验 + PR checklist，对应 F12。

规则 28-111：应用包 manifest 必须区分 system/vendor/customer 三类归属层；vendor 包升级不得覆盖 customer 扩展层，冲突必须进入升级报告。
强制方式：包升级回放测试，对应 F12/D9。

规则 28-112：市场分发包不得包含来源不明的图片、字体、图标、示例数据、模板文案或二进制文件；所有资产必须进入 SBOM 或资产 License 清单。
强制方式：资产清单扫描 + license-check，对应 F12/B6。

规则 28-113：License 校验必须支持私有化离线授权；不得把运行时核心能力绑定到必须访问公网的授权服务。在线校验可作为增量能力，但失败时必须有明确降级策略。
强制方式：离线授权测试 + 无外网演练，对应 F12/DEC-REQ-004。

规则 28-114：License 过期或降级不得删除业务数据、元数据和审计；只能限制新增、发布、安装、外部访问或高级能力，并必须给出可审计的用户提示。
强制方式：授权降级集成测试，对应 F12。

规则 28-115：第三方后端代码包默认不允许进入共享 SaaS 运行时；如后续允许，必须先补插件沙箱 ADR、安全评审、资源配额和退出机制。
强制方式：包准入阻断 + ADR 门禁，对应 F12/B4/C5。

规则 28-116：竞品开源代码学习产出的行业模板或应用包必须从知识卡重新设计，不得携带竞品源码、测试样例、图片资源、文案或独特字段命名。
强制方式：KB-REF 审查 + 相似片段扫描，对应 F12/E2。
