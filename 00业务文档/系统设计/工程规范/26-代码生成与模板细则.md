# 26-代码生成与模板细则

> 版本：v0.1（初稿）
> 地位：约束代码、SQL、schema、类型定义、测试样板等生成物的可维护性和安全性。

## 陷阱覆盖表

| 07 条目 | 本文覆盖点 | 强制/检测方式 |
|---|---|---|
| A1 DDL 非原子 | DDL 生成必须先出计划再执行 | 生成器测试 |
| D1 类型转换损坏 | field_type 到代码/SQL 映射稳定 | 快照测试 |

## 1. 生成物分类

规则 26-001：生成物必须归类：可覆盖、需合并、禁止人工修改。
强制方式：生成物头部标记扫描。

规则 26-002：所有生成物必须记录模板版本、来源元数据版本、生成时间、生成器版本。
强制方式：生成结果扫描。

规则 26-003：同一输入必须稳定输出，禁止 Map 无序遍历导致 diff 抖动。
强制方式：重复生成快照测试。

## 2. 模板治理

规则 26-010：模板必须集中存放，禁止在业务代码里拼接大段模板字符串。
强制方式：代码扫描。

规则 26-011：模板变更必须附带至少一个输入样例和输出快照。
强制方式：模板测试。

规则 26-012：模板不得生成未审查依赖、密钥、绝对路径、环境私有配置。
强制方式：敏感词扫描。

## 3. SQL 与 Schema 生成

规则 26-020：DDL SQL 生成只输出计划，不直接执行。
强制方式：接口边界测试。

规则 26-021：动态字段列名必须经过关键字、长度、冲突、字符集校验。
强制方式：列名生成参数化测试。

规则 26-022：field_type 到 MySQL 类型映射必须由 ADR-LOWCODE-FIELDTYPE-SPI-001 的 `FieldTypeHandler` 注册表驱动，生成器不得维护第二套映射表。
强制方式：Handler 覆盖测试 + 生成快照测试。

## 4. 前后端类型生成

规则 26-030：前端 API 类型从 OpenAPI 生成，禁止手写重复类型。
强制方式：lint + 文件扫描。

规则 26-031：生成类型的破坏性变更必须被 OpenAPI diff 捕获。
强制方式：契约 CI。

## 5. 覆盖策略

规则 26-040：可覆盖生成物必须可一键重建；人工可编辑区域必须有明确 marker。
强制方式：生成器回归测试。

规则 26-041：生成前必须做 diff 摘要，涉及删除/重命名必须阻断并人工确认。
强制方式：生成命令测试。
