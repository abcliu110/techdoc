# ERP 低代码系统拆解方法论

## 1. 一句话结论

ERP 低代码平台不是简单的“拖表单 + 生成 CRUD”，而是把企业信息系统中反复出现的业务结构抽象成一组可配置的元模型：

```text
业务对象 + 业务单据 + 状态流程 + 业务规则 + 权限边界 + 页面视图 + 报表指标 + 系统集成 + 扩展代码
```

这套拆解可行，是因为 ERP 和信息系统的大部分复杂性并不是随机的，而是稳定落在“对象、关系、状态、操作、约束、统计、协作、例外”这些维度上。

## 2. 为什么 ERP 可以被系统化拆解

ERP 的本质是企业经营过程的数字化记录和约束系统。

它要回答几类稳定问题：

```text
有什么业务对象？
发生了什么业务动作？
当前处于什么状态？
谁能操作？
操作是否符合规则？
操作后影响哪些库存、资金、账务、积分或指标？
结果如何统计？
如何与外部系统同步？
低代码覆盖不了的地方如何扩展？
```

这些问题在采购、销售、库存、财务、会员、生产、人事等模块中反复出现，所以可以抽象成平台能力。

## 3. 核心拆解框架

### 3.1 业务对象：系统管理的稳定名词

业务对象回答：

```text
业务世界里有什么？
```

典型业务对象：

```text
客户
供应商
商品
仓库
门店
员工
部门
会员
账户
设备
```

业务对象的特点：

```text
相对稳定
生命周期较长
会被多个业务过程反复引用
描述实体的当前属性
通常属于主数据或基础数据
```

例如商品对象：

```text
商品编号
商品名称
分类
单位
条码
售价
成本价
是否启用
```

商品本身不代表一次业务发生，它只是销售、采购、入库、出库、盘点等业务过程会引用的对象。

### 3.2 业务单据：一次业务动作的结构化记录

业务单据回答：

```text
发生了什么业务动作？
```

典型业务单据：

```text
销售订单
采购订单
入库单
出库单
调拨单
盘点单
收款单
付款单
退款单
发票
```

业务单据的特点：

```text
记录一次业务事件或业务流程
通常有单据头和单据明细
通常有状态流转
通常会影响库存、金额、账务、积分、业绩等结果
通常需要审核、反审、关闭、作废等操作
```

例如销售订单：

```text
单据头：客户、门店、销售员、下单时间、订单状态、总金额
单据明细：商品、数量、单价、折扣、金额
状态：草稿、已提交、已支付、已发货、已完成、已取消
影响：库存减少、收入增加、会员积分变化
```

业务对象和业务单据的区别可以压缩为：

```text
业务对象 = 名词，描述“它是谁 / 它是什么”
业务单据 = 动作证据，描述“发生了什么 / 做了什么”
```

### 3.3 状态流程：业务单据的生命周期

ERP 单据一般不是保存后就结束，而是会经历一组状态。

例如采购订单：

```text
草稿 -> 已提交 -> 审批中 -> 已审核 -> 部分入库 -> 已完成 -> 已关闭
```

低代码平台需要抽象：

```text
State 状态
Action 操作
Transition 状态流转
Condition 流转条件
Approval 审批节点
Rollback 回退路径
```

每个状态应该约束可执行操作：

```text
草稿：编辑、提交、删除
审批中：审批、驳回
已审核：生成入库单、关闭
已完成：查看、打印、归档
```

状态流程的作用是防止业务被随意修改，保证过程可控、可追踪、可回滚。

### 3.4 业务规则：系统中的约束和计算

规则回答：

```text
什么情况下允许做？
应该怎么计算？
哪些字段要显示或隐藏？
哪些操作必须被拦截？
```

常见规则：

```text
校验规则：数量必须大于 0
计算规则：金额 = 数量 * 单价 - 折扣
库存规则：库存不足禁止出库
权限规则：只有财务能审核付款单
显示规则：选择个人客户时隐藏公司税号
联动规则：选择客户后带出价格等级
编码规则：订单号按日期流水生成
```

低代码平台通常会把规则抽象成：

```text
表达式
公式
条件
决策表
脚本
规则集
```

规则是 ERP 复杂性的主要来源。平台能否做深，取决于规则模型是否能覆盖真实业务。

### 3.5 权限边界：谁能看、谁能改、能操作哪些数据

ERP 是多人、多组织、多角色协作系统，所以权限不是附加功能，而是基础能力。

权限通常拆成：

```text
菜单权限：能不能进入功能
操作权限：能不能新增、修改、审核、导出
字段权限：字段是否可见、是否可编辑
数据权限：能看哪些门店、部门、客户、仓库
流程权限：谁能处理哪个审批节点
租户权限：SaaS 环境下的数据隔离
```

一个成熟低代码 ERP 平台不能只做角色权限，还必须支持数据范围和字段级权限。

### 3.6 页面视图：业务模型的操作界面

页面不是系统的起点，而是业务模型的投影。

常见页面类型：

```text
列表页
详情页
编辑页
审批页
看板页
报表页
配置页
导入导出页
```

列表页抽象：

```text
查询条件
列表列
排序
筛选
分页
批量操作
导出
```

表单页抽象：

```text
字段
控件
布局
默认值
校验
联动
子表
按钮
```

如果平台只从页面开始建模，很容易退化成“页面搭建器”。ERP 低代码应该是模型驱动页面，而不是页面驱动模型。

### 3.7 报表指标：业务结果的统计口径

ERP 最终要服务管理决策，所以必须把业务过程沉淀为指标。

报表模型通常包括：

```text
指标 Metric
维度 Dimension
过滤条件 Filter
分组 GroupBy
聚合 Aggregation
下钻 DrillDown
图表 Chart
导出 Export
```

例如销售分析：

```text
指标：销售额、毛利、客单价、订单数
维度：门店、商品、客户、日期、业务员
过滤：时间范围、门店范围、品类范围
```

报表的关键不是图表样式，而是指标口径一致。否则同一个“销售额”会在不同页面算出不同结果。

### 3.8 系统集成：与外部系统协作

ERP 往往要和多个系统连接：

```text
POS
支付系统
财务系统
仓储系统
电商平台
物流系统
微信 / 钉钉 / 企微
短信 / 邮件
```

集成能力通常抽象为：

```text
API Connector 接口连接器
Webhook 回调
Event 事件
Message 消息
Job 定时任务
Mapping 字段映射
Retry 重试
Idempotency 幂等
```

系统集成的核心不是“能调用接口”，而是要处理失败、重试、幂等、数据映射和状态一致性。

### 3.9 扩展代码：低代码覆盖不了的例外

低代码不可能覆盖所有复杂业务。

因此平台必须提供扩展边界：

```text
脚本扩展
插件扩展
自定义服务
自定义按钮动作
自定义校验
自定义报表数据源
自定义集成适配器
```

低代码的合理目标不是消灭代码，而是把高频、稳定、可配置的部分平台化，把低频、复杂、强业务差异的部分留给代码扩展。

## 4. 元模型视角：这些元素如何组成系统

可以把 ERP 低代码平台理解为一组元模型之间的协作。

```text
Entity 定义业务对象
Bill 定义业务单据
Relation 定义对象和单据之间的关系
Workflow 定义状态流转
Rule 定义约束、计算和联动
Permission 定义访问边界
Page 定义用户操作入口
Report 定义经营结果
Event 定义系统协作
Extension 定义例外逻辑
```

一个销售订单系统可以这样组合：

```text
客户、商品、门店、会员              -> 业务对象
销售订单                            -> 业务单据
销售订单头 + 销售订单明细            -> 单据结构
草稿、已提交、已支付、已完成、已取消 -> 状态流程
价格、折扣、库存、积分               -> 业务规则
销售员、店长、财务                   -> 权限边界
销售订单列表、详情、编辑、审批        -> 页面视图
销售额、订单数、毛利、客单价          -> 报表指标
支付系统、库存系统、会员系统          -> 系统集成
特殊促销、特殊结算、特殊审批          -> 扩展代码
```

## 5. 拆解一个 ERP 模块的执行顺序

分析一个 ERP 模块时，建议按下面顺序执行。

### 第一步：识别业务域

先判断模块属于哪个业务域：

```text
销售
采购
库存
财务
会员
生产
人事
资产
售后
```

业务域决定核心对象、核心单据和关键规则。

### 第二步：找业务对象

列出该模块依赖的稳定实体：

```text
这个模块管理哪些人、物、组织、地点、账户、资源？
哪些数据会被多个单据引用？
哪些数据生命周期明显长于单据？
```

### 第三步：找业务单据

列出业务动作留下的记录：

```text
哪些动作会产生一张单？
哪些记录有明细行？
哪些记录会影响库存、资金、账务、积分或业绩？
哪些记录需要审核、关闭、作废？
```

### 第四步：拆状态流程

对每张核心单据拆：

```text
有哪些状态？
每个状态允许哪些操作？
哪些操作会改变状态？
哪些状态不可逆？
哪些操作需要审批？
哪些操作要影响下游数据？
```

### 第五步：拆业务规则

按规则类型整理：

```text
字段校验规则
金额计算规则
库存影响规则
权限判断规则
状态流转规则
单据生成规则
异常拦截规则
```

### 第六步：拆权限

确认：

```text
谁能看？
谁能新增？
谁能修改？
谁能审核？
谁能导出？
谁能看哪些组织、门店、仓库、客户的数据？
哪些字段需要隐藏或只读？
```

### 第七步：拆页面

不要先画页面，先从业务动作反推页面：

```text
用户需要查询什么？
用户需要录入什么？
用户需要处理什么待办？
用户需要查看什么结果？
用户需要导入导出什么数据？
```

### 第八步：拆报表

确认管理者关心的结果：

```text
核心指标是什么？
指标按哪些维度分析？
指标口径是什么？
数据来自哪些单据？
是否需要下钻到明细？
```

### 第九步：拆集成和扩展

确认平台边界：

```text
需要同步哪些外部系统？
哪些动作要发事件？
哪些接口需要幂等？
哪些业务规则低代码表达不了？
哪些逻辑必须写代码扩展？
```

## 6. 判断业务对象和业务单据的快速方法

| 判断问题 | 更像业务对象 | 更像业务单据 |
|---|---|---|
| 它是在描述“谁/什么”还是“发生了什么” | 谁/什么 | 发生了什么 |
| 生命周期长还是短 | 长 | 相对短，围绕一次业务过程 |
| 是否有明细行 | 通常没有 | 经常有 |
| 是否有状态流转 | 可能有启用/停用 | 通常有完整业务状态 |
| 是否影响库存、金额、账务 | 通常不直接影响 | 通常直接影响 |
| 是否被其他业务反复引用 | 是 | 也会被引用，但更多作为过程证据 |
| 典型例子 | 商品、客户、仓库 | 销售单、采购单、入库单 |

压缩判断：

```text
它是谁 / 它是什么？      -> 业务对象
发生了什么 / 做了什么？  -> 业务单据
```

## 7. 低代码 ERP 的关键取舍

### 7.1 灵活性 vs 规范性

低代码希望让业务快速配置，但 ERP 又要求流程、权限、数据一致性严格受控。

因此平台不能无限自由，必须用元模型限制业务表达方式。

### 7.2 配置能力 vs 代码能力

平台应该把高频稳定场景做成配置：

```text
字段
表单
列表
状态
审批
权限
公式
校验
报表
```

但复杂差异要交给代码：

```text
复杂促销
复杂库存分摊
特殊财务口径
第三方系统特殊协议
高性能批处理
强一致交易
```

### 7.3 快速交付 vs 长期可维护

只追求快速搭建，会产生大量不可治理的配置。

成熟平台必须管理：

```text
版本
发布
回滚
审计
测试
权限变更
指标口径
配置依赖
```

## 8. 失败模式

### 8.1 把低代码做成页面搭建器

表现：

```text
先拖页面，再补数据表，再补规则
```

后果：

```text
页面之间逻辑重复
状态不一致
权限散落
报表口径混乱
后期难维护
```

正确方向：

```text
先建业务模型，再生成页面视图
```

### 8.2 只有 CRUD，没有单据模型

表现：

```text
所有东西都按表增删改查处理
```

后果：

```text
审核、反审、作废、库存影响、金额重算都变成硬编码
```

正确方向：

```text
把订单、入库单、出库单等建模为有状态、有明细、有操作、有影响的业务单据
```

### 8.3 规则没有统一模型

表现：

```text
前端写一点校验
后端写一点校验
流程里写一点条件
报表里再写一套口径
```

后果：

```text
同一个规则多处实现
修改困难
结果不一致
问题难追踪
```

正确方向：

```text
把规则集中建模，明确规则类型、执行时机、作用范围和优先级
```

### 8.4 没有扩展边界

表现：

```text
所有复杂业务都试图用配置硬塞
```

后果：

```text
配置变成另一种代码
难调试
难测试
难复用
```

正确方向：

```text
明确哪些适合配置，哪些必须进入脚本、插件或自定义服务
```

## 9. 成熟度检查清单

评估一个 ERP 低代码平台，可以检查：

```text
是否支持业务对象建模？
是否支持主从表单据？
是否支持单据状态机？
是否支持审批流？
是否支持字段校验和业务规则？
是否支持计算公式？
是否支持编码规则？
是否支持数据权限？
是否支持字段权限？
是否支持导入导出？
是否支持报表指标口径？
是否支持事件和消息？
是否支持外部接口集成？
是否支持版本、发布和回滚？
是否支持脚本或代码扩展？
是否支持多租户隔离？
是否支持审计和操作日志？
```

如果一个平台只支持拖表单和生成 CRUD，它适合简单信息采集，不适合复杂 ERP。

## 10. 最终压缩

ERP 低代码系统的拆解逻辑可以压缩为：

```text
用业务对象描述稳定实体；
用业务单据记录业务动作；
用状态流程约束生命周期；
用业务规则表达校验、计算和联动；
用权限边界控制谁能看、谁能改、能改哪些数据；
用页面视图承载用户操作；
用报表指标沉淀经营结果；
用系统集成连接外部世界；
用扩展代码承接低代码覆盖不了的复杂差异。
```

这套方法的关键不是多列几个模块，而是始终追问：

```text
它抽象了什么？
它约束了什么？
它牺牲了什么？
它在什么边界下可行？
```
