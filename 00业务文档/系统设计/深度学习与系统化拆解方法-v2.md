> 本文用于指导如何系统学习一个新事物、新技术、新业务、新系统或新领域。
> 设计原则：**核心框架只有一套，定义处与引用处一字不差；方法可独立检索；文档须能通过自身的检查清单。**

## 1. 目标

核心目标不是"记住更多知识点"，而是建立可迁移的理解模型，最终达到：

- 能清楚定义它是什么。
- 能解释它为什么存在。
- 能拆出它的核心模型、机制、边界和取舍。
- 能判断它什么时候适用、什么时候不适用。
- 能把它迁移到其他问题中使用。
- 能用自己的话重新组织、讲解甚至最小重建它。

一句话总结（四问递进）：

```text
普通学习问：它怎么用？
深入学习问：它为什么这样设计？
高手学习问：在什么约束下，它不得不这样设计？
融会贯通问：这个模型还能解释什么？
```

---

## 2. 核心心智模型

任何对象，本质都是【在约束下，为解决某问题，做的一组取舍】。

学习的核心始终围绕三个锚点收敛，其余方法都是为它们服务：

```text
锚点1  本质模型：它把现实抽象成了什么？（对象 + 关系 + 状态转移）
锚点2  核心张力：它内部最大的矛盾是什么？优先保谁、牺牲谁？
锚点3  适用边界：它在什么条件下成立、什么条件下失效？能迁移到哪里？
```

判断"真懂"的唯一标准（与三锚点逐项对应）：

```text
能说清 本质模型 + 核心张力/取舍 + 适用边界（含可迁移性）。
```

> 说明：取舍是核心张力的展开（把"保谁/牺牲谁"具体化），迁移是适用边界的延伸（原则能搬到哪）。二者是锚点的支撑要素，不单列。

## 3. 深度分档

不是所有对象都要拆到最深。先定目标深度，避免简单问题过度分析、复杂问题只停表面。

```text
L1 速懂：一句话定义 + 主要用途 + 一个关键边界。
L2 标准：定义 + 问题 + 核心模型 + 机制 + 主要取舍 + 适用边界。
L3 深挖：L2 + 不变量 + 核心张力 + 失败模式 + 反例 + 关键对比。
L4 融通：L3 + 演化脉络 + 跨领域迁移 + 最小重建 + 预测校验。
```

深度档与学习阶段一一对应（深度档 = 该阶段产出的理解深度）：

```text
L1 速懂  ↔ 入门阶段：建立基本地图
L2 标准  ↔ 理解阶段：知道它如何工作
L3 深挖  ↔ 深挖阶段：理解设计原因和边界
L4 融通  ↔ 融会贯通阶段：迁移、预测、重建
```

判断标准：

```text
只要能用即可：L1
需要讲清楚：L2
需要判断取舍：L3
需要迁移和创造：L4
```

停止条件（防分析瘫痪）：

```text
做完目标深度档的必做层，并通过第9节对应检查项，即停止。
继续深挖属于过度分析——深度是成本，不是荣誉；需要更深时再回来升档。
```

## 4. 主干：10 层拆解法

学习任何对象，按下面这条主干拆解。10 层与三锚点的对应关系：

```text
锚点1 本质模型 ← 层1 定义 + 层3 模型 + 层4 机制
锚点2 核心张力 ← 层6 取舍 + 层7 张力
锚点3 适用边界 ← 层5 边界 + 层8 失败 + 层9 对比
跨锚点         ← 层2 问题（动因）+ 层10 迁移（延伸）
```

```text
层1  定义  一句话说本质，给类别与同类根本区别。
层2  问题  为解决什么痛点而生？替代了什么、引入什么新成本？
层3  模型  核心对象/关系/状态/状态转移。（最重要）
层4  机制  输入→处理→输出→异常→恢复。数据流、控制流、常见使用模式和组合方式。
层5  边界  适用/不适用场景，失效条件，规模临界点，误用方式。
层6  取舍  约束？优先保什么、牺牲什么？代价谁承担？规模变大哪个约束先爆？哪些决策是改不动的单向门？（最重要）
层7  张力  内部最大冲突是什么？优先解决哪边？如何缓解？代价转到哪里？
层8  失败  常见错、隐蔽错、"看似正常实则已错"、检测/恢复/预防。
层9  对比  与最相似 1–2 个对象比"模型和取舍"，给选 A 还是 B 的判据。
层10 迁移  背后通用原则是什么？能迁移到哪些领域？能否最小重建？
```

各深度的必做层：

```text
L2+ 必做：层3（模型）、层6（取舍）。
L3+ 追加：层5（边界）、层7（张力）、层8（失败）、层9（对比），并按对象类型补不变量分析。
L4 追加：层10（迁移，含最小重建）。
```

> 说明：相比 v1，把"取舍"从隐含项升为独立层（层6），把"张力"升为独立层（层7），把原"模式层"（讲通常怎么用）并入层4机制或层5边界的实践部分，因为它的优先级低于取舍。

## 5. 方法选择矩阵

方法不是越多越好。根据对象类型选 3–5 个关键方法组合，叠加到 10 层主干上。

| 对象类型 | 优先方法 | 主要产出 |
|---|---|---|
| 概念 / 理论 | 第一性原理、反例驱动、对比分析 | 本质定义、成立条件、反例 |
| 技术 / 工具 | 操作符拆解、黑箱灰箱白箱、对比分析、最小重建 | 使用模型、工作机制、选型判据 |
| 业务 / 领域 | 问题空间法、视角切换、不变量分析、张力分析 | 领域地图、核心对象、业务规则 |
| 系统 / 架构 | 黑箱灰箱白箱、张力分析、不变量分析、失败模式分析、多尺度分析、架构师思维透镜 | 系统边界、协作关系、风险点、决策清单 |
| 流程 / 制度 | 反向工程、视角切换、约束取舍分析、失败模式分析 | 流程图、责任边界、例外处理 |
| 陌生新事物 | 5W1H、地图路径地形、问题空间法 | 入门地图、学习路径、关键问题 |

补充规则：

```text
先用少数方法建主干；
发现边界不清时补反例驱动；
发现机制不清时补黑箱灰箱白箱；
发现判断困难时补取舍和对比分析；
发现无法迁移时补最小重建和预测校验。
```

## 6. 对象类型执行路径

**执行路径是 10 层主干在该类型上的特化版，与主干二选一，不叠加。** 走路径时，步骤名对应主干同名层的语义；未列的层按深度档要求补齐。

```text
技术/工具：  定义 → 问题 → 模型 → 操作符拆解 → 机制 → 取舍 → 边界 → 对比分析 → 最小重建
业务/系统：  业务域 → 核心对象 → 对象关系 → 流程/状态 → 业务规则 → 不变量分析 → 权限边界 → 集成边界
架构方案：   目标问题 → 约束 → 核心张力 → 组件边界 → 数据流/控制流 → 不变量分析 → 失败模式分析 → 替代方案
概念/理论：  一句话定义 → 第一性事实 → 最小例子 → 反例驱动 → 边界 → 对比分析 → 可迁移原则
流程/制度：  参与角色 → 输入输出 → 状态流转 → 决策规则 → 异常路径 → 责任边界 → 审计/回滚
陌生新事物： 5W1H → 定义 → 问题 → 地图路径地形 → 学习路径 → 关键问题清单
```

> 说明：相比 v1，补齐了"流程/制度"和"陌生新事物"两条路径，与第5节矩阵的6类对齐。

## 7. 方法库

以下方法可独立选用，叠加到 10 层主干上。方法名以本节定义为准。

### 7.1 5W1H

快速分析入口，适合刚接触新对象时建第一层框架。

```text
What：它是什么？
Why：为什么需要它？
Where：在哪些场景用？
When：什么时候用，什么时候不用？
Who：谁依赖它，谁受它影响？
How：它怎么工作，怎么落地？
```

### 7.2 第一性原理

去掉术语、包装和惯性说法，追问最底层事实。

```text
它最底层依赖什么事实？
如果去掉所有术语和包装，它还剩什么？
它为什么必须这样，而不是另一种方式？
```

示例：学数据库事务，不先背 ACID，先问——多个人同时改数据会怎样？中途失败怎么知道回到哪里？读到一半别人改了算不算对？本质是"在不可靠环境里维护一致状态"。

### 7.3 反向工程

从结果倒推原因和前置条件。适合学框架、业务系统、算法、产品设计、流程机制。

```text
最终现象：
直接前置条件：
间接前置条件：
隐藏假设：
失败路径：
恢复机制：
```

### 7.4 黑箱灰箱白箱

学复杂系统时分三层推进，避免一开始陷入实现细节。

```text
黑箱：只看输入输出，理解它表现成什么。
灰箱：通过实验和边界测试推断内部规律。
白箱：阅读内部结构、源码、算法和实现。
顺序：先黑箱用会 → 再灰箱验证边界 → 最后白箱读实现。
```

### 7.5 操作符拆解

列对象支持的基本操作。适合学数据结构、工具、协议。

```text
创建、读取、更新、删除、组合、拆分、转换、比较、校验、回滚、同步、扩展。
```

示例：Git 基本操作 = 记录状态 / 比较状态 / 移动指针 / 合并历史 / 解决冲突 / 回滚变化 / 同步远端。

### 7.6 不变量分析

找出无论系统怎么变化都必须保持不变的东西。适合账务、库存、并发、状态机、一致性系统。

```text
什么一定不能变？
什么变了系统就坏？
它如何维持这个不变量？
谁负责检查？
失败后怎么恢复？
```

示例：账户系统不变量 = 钱不能凭空增加或消失；库存系统 = 不能超卖或要有补偿；Git = commit 内容确定后不可变。

### 7.7 张力分析

复杂系统围绕内部张力设计，而非围绕功能设计。

```text
常见张力：速度vs准确、灵活vs规范、简单vs完整、一致性vs可用性、短期效率vs长期维护、自动化vs可控性。
关键问题：内部最大冲突是什么？优先解决哪边？如何缓解？代价转到哪里？
```

### 7.8 失败模式分析

真正懂一个东西，要知道它怎么坏。适合可靠性相关对象。

```text
正常路径 / 异常路径 / 边界条件 / 误用方式 / 故障信号 / 修复手段 / 预防机制。
关键问题：最常见错？最隐蔽错？什么情况表面正常但内部已错？如何检测/恢复/预防？
```

### 7.9 反例驱动

学一个规则时，立刻找它不成立的地方。

```text
什么时候它是错的？最小反例是什么？边界在哪里？为什么这个反例能破坏它？
```

真正懂一个概念，往往是从知道它的失败边界开始。

### 7.10 多尺度分析

从不同尺度观察同一对象。

```text
微观：一个函数、动作、概念如何工作？
中观：模块之间如何协作？
宏观：它在整个系统、行业、历史里扮演什么角色？
```

示例：学 Spring——微观看 Bean 如何创建，中观看 IOC/AOP/事务如何协作，宏观看它为什么成为 Java 企业开发基础设施。

### 7.11 视角切换

同一对象从不同角色观察，会看到不同结构。

```text
使用者：我怎么用？
实现者：它怎么做？
维护者：它怎么坏？
设计者：为什么这样设计？
竞争者：我如何替代它？
攻击者：我如何绕过它？
教学者：我如何讲清它？
```

### 7.12 问题空间法

不只从答案学，还要从问题空间学。适合陌生领域、宏观认知。

```text
这个领域有哪些问题？哪些是核心？哪些是衍生？哪些互相冲突？哪些目前无解？
```

示例：学 AI 不只学模型/Prompt/RAG，而看问题空间——表示/推理/记忆/对齐/泛化/可控性/成本/评估。

### 7.13 最小重建

强迫自己做一个玩具版本，暴露真实复杂度。

```text
不用框架，能不能实现最小版本？去掉高级功能后核心还剩什么？哪些地方一实现就发现很难？
```

示例：学 React 手写最小响应式渲染；学数据库写简单 KV 存储；学消息队列写内存队列+ack；学编译器写表达式解析器。

### 7.14 迁移类比

用已知类比未知，但必须紧跟差异和边界。

```text
这个东西像哪个我已经懂的东西？相似点？不同点？这个模型能迁移到哪里？
```

示例：缓存像备忘录，索引像书的目录，消息队列像排队窗口，Git commit 像时间线快照（更准确说是有向无环图中的节点）。

> 类比不是为了偷懒，而是为了建立跨领域模型。类比之后必须继续找差异和边界。

### 7.15 地图路径地形

学一个领域时先建学习地图。

```text
地图：这个领域有哪些板块？
路径：学习顺序是什么？
地形：哪里容易、哪里困难、哪里坑多？
```

很多人学不好，不是因为笨，而是没有地图，在局部知识里打转。

### 7.16 对比分析

通过比较建立清晰边界。不要只比 API 或表面功能，要比模型和取舍。

```text
A 和 B 的本质区别是什么？分别优先保什么？分别牺牲什么？什么场景选 A，什么场景选 B？
```

### 7.17 演化分析

看历史能理解设计取舍。

```text
旧方案有什么问题？新方案解决了什么？新方案又引入了什么复杂度？版本演化背后的主要矛盾是什么？
```

### 7.18 预测校验

学完后做预测，用预测检验模型是否成立。

```text
如果输入变了，输出会怎样？规模扩大 10 倍先坏哪里？约束取消有没有更简单方案？换一个场景还适用吗？
```

能预测，说明模型进入了自己的思维系统。

### 7.19 生成式学习

判断是否真懂，不看能不能复述，而看能不能生成。

```text
能不能自己举例？能不能造简化版本？能不能教给别人？能不能预测新场景？能不能改造它解决别的问题？
最高检查：能不能不用原文术语，把它重新发明一遍？
```

### 7.20 层级压缩

把复杂知识压缩成不同长度。如果只能长篇解释，说明还没压缩出核心。

```text
一句话版 / 三句话版 / 一页纸版 / 十分钟讲解版 / 完整体系版。
```

### 7.21 元问题法

不同类型的知识需要不同学习方式。先问：我现在到底在学事实、概念、技能、系统，还是审美？评价标准是什么？什么样的人算真正懂它？我和专家的差距在哪？

```text
事实：记忆和检索
概念：边界和反例
技能：练习和反馈
系统：模型和关系
审美：大量案例和品味校准
```

### 7.22 约束取舍分析

任何技术、制度、工具、模型，本质都是在约束下做取舍。

```text
它受哪些约束？优先保证什么？牺牲了什么？代价由谁承担？规模变大后哪个约束最先爆？
```

示例：Redis 快因为以内存为主，代价是容量、持久化风险、成本和数据一致性复杂度。看到取舍，理解就从"知道用法"进入"知道为什么"。

### 7.23 证据链

每个结论都要追证据，避免"听起来懂了"。证据按强度分 5 档（全文档统一使用）：

```text
（直接事实）源码、数据、可复现实验、官方文档。
（主流共识）多来源稳定一致的行业/学术/工程共识。
（经验判断）基于常见实践，但依赖场景。
（推断）     由已知证据推出，需说明推理链。
（假设）     暂未验证，只能作为待验证方向；可加注"待验证"。
```

规则：高强度证据可支撑结论；低强度证据只能支撑倾向或假设。

### 7.24 领域逆向

从既有系统/竞品表象抽取对象、流程、状态、规则、不变量。适合"分析既有系统并设计新系统"，详见第8节。

### 7.25 架构师思维透镜

来自顶级架构师和设计师群体的高阶判断工具。它不替代 10 层主干，而是在层3/5/6/7 上追加更锋利的问题。每条透镜 = 一条定律 + 一个必问问题。

```text
盖尔定律      能工作的复杂系统，必然从能工作的简单系统演化而来；从零设计的复杂系统无法工作。
              问：它的最小可工作前身是什么？我能否从简单版本重走一遍它的演化？

康威定律      系统结构映射设计它的组织的沟通结构。
              问：这个设计里哪些边界其实是组织边界？换一个组织，它还会这样切分吗？

切斯特顿栅栏  在弄清一个设计为何存在之前，不要拆除它。
              问：这个"看似多余"的部分，当初防的是什么？它对应的约束消失了吗？

海勒姆定律    只要使用者足够多，系统的一切可观察行为终将被依赖，无论是否承诺过。
              问：除了承诺的接口，它还有哪些隐性契约？改动会伤到谁？

单向门/双向门 决策分两类：可逆（双向门）快速试错即可；不可逆（单向门）必须审慎。
              架构 = 那组难以改变的决策的集合。
              问：它哪些决定一旦做出就改不动了？审慎资源是否集中在单向门上？

简单 ≠ 容易   简单是"不缠绕"（客观：职责单一、正交、可独立推理）；容易是"熟悉"（主观）。
              问：这个方案是真的简单，还是只是对我来说熟悉？解耦了吗，还是只是顺手？

杠杆点        干预系统的杠杆从低到高：改参数 < 改反馈回路 < 改规则 < 改目标 < 改范式。
              问：它真正的高杠杆点在哪？我看到的问题是症状，还是回路本身？

二阶效应      每个方案都会引发"对策的对策"，参与者会适应性地改变行为。
              问：这样改之后，然后会发生什么？谁会绕过它、滥用它、依赖它？

古德哈特定律  指标一旦成为目标，就不再是好指标——人会优化指标本身，而非指标背后的目的。
              问：这个指标被当成考核目标后，会被怎样操纵？它还代表原来的目的吗？

变更成本分布  好架构不是预测对未来，而是让预测错误的代价变低。
              问：它把"改起来贵"的部分放在哪？变化最频繁的维度是否被隔离在便宜处？

第二系统效应  重写和"这次好好设计"的新系统，最容易过度设计。
              问：我加的每个"这次一定要有"的特性，证据是什么？
```

使用时机：

```text
学习系统/架构类对象：层5/6/7 做完后，用透镜逐条追问一遍，产出"单向门决策清单 + 隐性契约清单"。
走第8节逆向协议：进入 8.4 生成新架构前必须过盖尔定律和第二系统效应；8.5 反向校验时逐条对照。
```

### 7.26 费米估算（数量级推演）

顶级架构师谈方案前先算量级。不求精确，只求数量级正确——数量级错了，方案讨论毫无意义。

```text
先估规模：用户量、并发、数据量、增长率，精确到数量级即可（10³ 还是 10⁶）。
再推临界：当前方案在 10 倍、100 倍规模下，哪个资源先爆？（存储/带宽/计算/人力/配置复杂度）
后定方案：方案复杂度必须与量级匹配——为百万级设计的方案用在千级，就是第二系统效应。
```

示例：单表 1 万行不需要分库分表；日均 100 单不需要消息队列削峰；3 个人的团队不需要微服务。先算，再设计。

## 8. 逆向到架构生成协议（专项分支）

**触发**：目标是分析既有系统/竞品/标杆，并据此设计新系统时，启用本协议。它替代常规的 10 层主干流程，有自己的 6 步链路，但最终仍须过第9节检查清单和第10节反模式检查。

主链路：

```text
既有/第三方系统 → 领域逆向 → 抽取领域知识 → 形成知识库 → 生成新架构 → 反向校验
```

核心原则：

```text
不要把竞品功能当答案，要把竞品功能当证据。
不要停在页面和按钮，要抽象到对象、关系、状态、规则、不变量和边界。
不要只记录痛点，要把痛点转成新系统设计原则。
不要轻视存量设计：先过切斯特顿栅栏（它当初防什么）和海勒姆定律（谁在依赖它的隐性行为），再谈替换。
不要犯第二系统效应：新架构的每一处新增复杂度都必须有证据支撑，而不是"这次一定要有"。
```

### 8.1 逆向分析：从表象还原业务结构

先收集可观察证据：

```text
页面和菜单、用户角色、业务流程、表单字段、列表筛选、按钮动作、
审批节点、状态变化、报表指标、接口和集成、异常提示、权限限制。
```

再做抽象转换：

```text
页面 → 用户任务         按钮表单字段 → 业务对象属性
列表筛选 → 查询维度     审批流 → 状态机和责任边界
校验提示 → 业务规则     报表 → 指标口径
接口 → 系统边界         异常处理 → 不变量和失败模式
竞品缺陷 → 新系统设计原则
```

产出不是功能清单，而是：领域对象图、核心流程图、状态机、业务规则表、权限边界、指标口径、外部集成图、痛点和缺陷清单。

### 8.2 抽取领域知识：从功能拔高到知识

知识抽取维度：

```text
业务对象：系统管理哪些稳定名词？
业务过程：用户围绕这些对象完成什么任务？
状态模型：关键对象和单据如何流转？
业务规则：哪些校验、计算、权限、限制在起作用？
不变量：什么绝对不能被破坏？
指标口径：系统最终沉淀哪些经营结果？
系统边界：哪些能力属于外部系统或上下游系统？
失败模式：哪里复杂、绕路、低效、容易错？
设计取舍：它优先保证什么，牺牲了什么？
```

抽取规则：

```text
同一概念在多个系统中反复出现 → 提炼为领域对象或通用能力。
同一规则在多个入口重复出现 → 提炼为规则模型。
同一状态被多个页面修改 → 提炼为状态机。
同一指标多处计算 → 提炼为统一指标口径。
同一痛点多次出现 → 提炼为新系统设计原则。
```

### 8.3 形成知识库：让分析结果可复用

知识库不是资料堆积，而是新系统设计的输入。建议结构：

```text
领域对象库：客户、商品、订单、库存、账户、会员、供应商、仓库。
业务过程库：采购、销售、入库、出库、调拨、盘点、收款、付款、退款。
状态机库：订单状态、库存单状态、审批状态、支付状态、发票状态。
规则库：库存规则、价格规则、优惠规则、权限规则、结算规则、税务规则。
不变量库：库存不能无依据变化；金额不能凭空产生或消失；状态不能非法跳转；权限不能越界；指标口径不能多处不一致。
竞品证据库：截图、页面路径、操作步骤、字段、接口、报表、异常提示。
痛点和缺陷库：流程绕路、规则散落、权限后补、配置不可回滚、指标不一致。
设计原则库：模型优先于页面；状态机统一维护；规则集中治理；指标口径统一定义；扩展点和核心逻辑分层。
架构模式库：主数据模型、单据模型、状态机模型、规则引擎模型、权限模型、指标模型、事件集成模型、插件扩展模型。
```

每条知识至少包含：知识名称、所属领域、来源证据、抽象结论、适用条件、不适用条件、关联对象、关联规则、设计启发、待验证项。

### 8.4 生成新系统架构：从知识库到设计

知识库要能直接推导新系统设计：

```text
领域对象库 → 领域模型            业务过程库 → 用例模型/应用服务
状态机库   → 单据状态流转设计    规则库 → 规则引擎/决策表/校验中心
不变量库   → 事务边界/一致性策略/审计机制
权限库     → RBAC + 数据权限 + 字段权限
指标库     → 指标层/报表模型/数据仓库口径
竞品缺陷库 → 新系统设计原则      架构模式库 → 系统分层和扩展机制
```

新系统设计顺序：

```text
1. 定义目标用户和目标场景。  2. 定义业务域边界。
3. 定义核心业务对象。        4. 定义核心业务过程和业务单据。
5. 定义状态机。              6. 定义业务规则模型。
7. 定义权限模型。            8. 定义指标和报表口径。
9. 定义事件和集成模型。      10. 定义扩展点。
11. 定义质量属性与可观测性。 12. 最后定义页面和交互。
13. 为关键决策写 ADR（架构决策记录）。
```

> 页面是领域模型的投影，不是系统设计的起点。
> 第11步说明：质量属性 = 性能/可用性/安全/可维护性目标（用费米估算先定量级）；可观测性 = 日志、指标、链路追踪、审计——系统必须能回答"现在发生了什么、刚才发生过什么"。

决策治理规则：

```text
用"单向门/双向门"给每个设计决策分级：
  双向门（可逆：页面布局、报表样式、规则参数）→ 快速决定，允许试错。
  单向门（不可逆：核心对象模型、单据/状态机结构、元模型 Schema、对外契约、数据口径）
    → 必须写 ADR（背景约束 → 备选方案 → 决策 → 后果与撤销成本），并经过评审。
变更成本分布设计：把高频变化放进便宜层（配置/规则/页面），把低频变化放进昂贵层（对象模型/状态机/契约）。
遵循盖尔定律：先设计能跑通的最小系统，再演化，不追求首版大而全。
风险驱动投入：架构精力与风险成正比——高风险处（账务一致性、超卖、数据迁移、对外契约）深挖到 L3/L4，低风险处（页面样式）走默认方案，不平均用力。
最后责任时刻：能推迟的单向门决策尽量推迟——用双向门方案先顶住，直到"不决策的成本开始超过决策风险"时再落锤；推迟期间积累的信息就是决策质量。
先实验后落锤：单向门决策落锤前，用最小实验（Spike/POC）验证其关键假设，把赌注变成证据。
警惕古德哈特定律：考核指标与经营指标分开治理；任何指标被当成目标后，须重新校验它的口径和可操纵性。
```

ADR 模板（每条单向门决策一份）：

```text
标题：ADR-001 单据状态机集中在平台层维护
背景与约束：哪些业务/技术/组织约束逼出了这个决策？
备选方案：至少 2 个，各自优先保什么、牺牲什么。
决策：选了哪个，为什么。
后果：得到什么、牺牲什么、撤销成本多大（单向门/双向门）。
状态：提议 / 已接受 / 已废弃（被 ADR-xxx 替代）。
```

适应度函数示例（架构特性 → 可自动执行的校验）：

```text
状态机合法性 → 每日扫描单据表，非法状态跳转数必须为 0。
指标口径统一 → 同一指标在报表层与明细层重算，偏差超阈值即告警。
性能护栏     → 核心单据提交 P99 < 1s，压测纳入发布流水线。
权限不越界   → 自动化用例遍历"角色 × 敏感操作"矩阵，越权数必须为 0。
```

### 8.5 反向校验：新设计是否真的更好

回到原系统和竞品中校验：

```text
是否覆盖了核心用户任务？
是否守住了领域不变量？
是否减少了旧系统的绕路流程？
是否把散落规则收敛到了统一模型？
是否明确了配置和代码扩展的边界？
是否统一了指标口径？
是否支持审计、回滚、灰度和迁移？
是否能解释为什么比原方案更合理？
─── 架构师透镜校验（对照 7.25）───
单向门决策是否都有 ADR 且经过评审？
是否识别了旧系统的隐性契约（海勒姆定律），并给出兼容或迁移方案？
被移除的旧设计是否都弄清了"当初防什么"（切斯特顿栅栏）？
组件边界是否与团队边界匹配（康威定律）？错配之处是否有意为之？
是否定义了适应度函数——把关键架构特性（性能阈值、口径一致、状态机合法性、权限不越界）变成可自动执行的校验？
新增复杂度是否逐项有证据，而非"这次一定要有"（第二系统效应）？
首版是否是能跑通的最小系统（盖尔定律），而非大而全蓝图？
```

如果回答不了"为什么更合理"，说明只是重新组合了功能，不是完成了架构设计。

### 8.6 最终交付物

```text
1. 既有系统/竞品分析报告    2. 证据库和截图索引
3. 领域知识库               4. 领域对象模型
5. 核心业务流程图           6. 状态机设计
7. 业务规则清单             8. 不变量和风险清单
9. 能力地图                 10. 新系统设计原则
11. 新系统架构方案          12. 分阶段建设路线图
13. 架构决策记录（ADR）集   14. 适应度函数清单
```

一句话压缩：

```text
从已有系统中提取领域知识，从领域知识中识别不变量和设计原则，再从设计原则中生成更优系统架构。
```

## 9. 检查清单与质量评分

### 9.1 检查清单

学习一个对象时，逐项检查。前 10 项对应 10 层主干（必检），其后按深度档追加：

```text
□ 层1 定义：它是什么？（一句话+类别+核心差异）
□ 层2 问题：它解决什么？（替代了什么、引入什么代价）
□ 层3 模型：它如何抽象？（对象/关系/状态转移）——L2+必检
□ 层4 机制：它如何运行？（输入/处理/输出/异常/恢复）
□ 层5 边界：什么时候不用它？——L3+必检
□ 层6 取舍：它牺牲什么换来什么？——L2+必检
□ 层7 张力：内部有哪些冲突？——L3+必检
□ 层8 失败：它怎么坏？——L3+必检
□ 层9 对比：它和相似东西差在哪？（比模型和取舍）——L3+必检
□ 层10 迁移：它能解释别的什么？——L4必检
─────────────────────────────────────
□ 反例：什么时候规则不成立？（L3+）
□ 不变量：什么必须保持不变？（系统类必检）
□ 透镜：单向门决策和隐性契约识别了吗？（系统/架构类 L3+，对照 7.25）
□ 证据：我凭什么相信这些结论？是否按 5 档标注？
□ 视角：不同角色怎么看它？
□ 预测：我能否推演新情况？（L4）
□ 重建：我能否造最小版本？（L4）
□ 压缩：我能否一句话讲透？
```

检查失败时的修复动作：

```text
定义不清 → 补一句话定义和所属类别。
模型不清 → 补核心对象、关系、状态变化。
取舍不清 → 补约束、优先保证、牺牲内容、代价承担方。
张力不清 → 补内部最大冲突、优先解决方、代价转移去向。
边界不清 → 补反例和不适用场景。
对比不清 → 补最相似的 1–2 个对象，比模型和取舍，给选型判据。
不变量缺失（系统类）→ 补必须恒真的性质 + 破坏后果。
证据不足 → 降档标注为"推断/假设"，不要写成"直接事实"。
迁移困难 → 做最小重建和预测校验。
只能长篇 → 做层级压缩，至少压缩出一句话版。
```

### 9.2 理解质量评分（0-3 分）

| 维度 | 0 分 | 1 分 | 2 分 | 3 分 |
|---|---|---|---|---|
| 定义 | 说不清 | 只能复述术语 | 能用自己的话定义 | 能说出类别和核心差异 |
| 问题 | 不知道为何存在 | 只知道用途 | 能说明替代了什么 | 能说明新旧方案的成本变化 |
| 模型 | 只知道功能 | 能列对象 | 能说清对象关系和状态 | 能画出最小模型并解释派生概念 |
| 机制 | 只会使用 | 知道大概流程 | 能说明输入、处理、输出 | 能说明异常路径和恢复机制 |
| 取舍 | 只说优点 | 知道一些缺点 | 能说明牺牲了什么 | 能判断规模变化后哪个约束先爆 |
| 边界 | 不知道什么时候不用 | 能列少量限制 | 能给反例 | 能给适用/不适用判据 |
| 迁移 | 只能照搬 | 能做简单类比 | 能迁移到相似场景 | 能预测新场景下的行为 |

```text
平均 1 分：只是听过或用过。
平均 2 分：可以独立使用和解释。
平均 3 分：可以做判断、教学、迁移和设计。
```

## 10. 反模式（禁止的九类劣质学习与设计）

```text
反模式1 只背定义：能复述术语，但说不清核心对象、关系和状态变化。
反模式2 只学用法：知道调 API，但不知道为什么这样设计，不知道代价由谁承担。
反模式3 不看失败：教程里能跑通，遇到异常、边界、并发、规模变化就失效。
反模式4 假类比：用熟悉事物解释陌生概念，但不指出相似点和差异点。
反模式5 不压缩：笔记很长，但不能用一句话、三句话、一页纸讲清。
反模式6 伪确定：没有证据来源，不区分直接事实、主流共识、经验判断、推断、假设。
反模式7 抄竞品：把竞品功能清单直接当新系统需求，只记录痛点不提炼设计原则，或只说"更灵活/更强大"而不说强在哪、牺牲什么、边界在哪。
反模式8 无敬畏改造：没弄清旧设计当初防什么就拆除（违反切斯特顿栅栏），没识别隐性契约就改动（违反海勒姆定律）。
反模式9 第二系统效应：借"重新设计"之名过度设计——首版就要大而全，新增复杂度没有证据，只有"这次一定要有"。
```

## 11. 三问快速穿透

高度压缩的高手问题组，与三锚点逐项对应，可作为 L2 的内部脚手架：

```text
本质模型是什么？（它抽象成了什么）
核心张力是什么？（它内部最大的冲突，优先保谁、牺牲谁）
适用边界是什么？（什么时候成立、什么时候失效、能迁移到哪里）
```

> 说明：相比 v1，第三问从"主要取舍"改为"适用边界"，与三锚点对齐；取舍作为核心张力的展开并入第二问。

示例（缓存）：

```text
本质模型：用空间换时间的副本系统。
核心张力：快 vs 准（数据新鲜度）；性能 ↔ 一致性 ↔ 成本 ↔ 复杂度。
适用边界：适合读多写少、可容忍短暂不一致的场景；不适合强一致核心账务路径；可迁移到任何"用副本换访问速度"的场合。
```

示例（公司）：

```text
本质模型：多人协作的资源配置系统。
核心张力：个人目标和组织目标不完全一致。
适用边界：适合需要规模化协作的资源配置；不适合个人或小规模自发协作；可迁移到任何"集体目标与个体目标需对齐"的组织形态。
```

## 14. 完整案例：用本方法拆解 ERP 低代码平台

> 本案例演示 10 层主干全流程，并标注每层对应的深度档。

### 14.1 层1 定义（L1）

```text
ERP 低代码平台不是简单的页面搭建器，而是用元数据描述企业经营对象、业务单据、流程状态、规则约束、权限边界、报表指标和系统集成的应用构建平台。
类别：应用构建平台（低代码）。
核心差异：相比表单工具，它能表达单据/审批/库存/账务/权限/集成；相比定制开发，它把高频稳定结构平台化。
```

### 14.2 层2 问题（L1）

```text
传统定制开发：灵活，但成本高、周期长、重复劳动多。
简单表单工具：交付快，但难以表达单据、审批、库存、账务、权限和集成。
ERP 低代码：把高频稳定结构平台化，把复杂差异留给扩展代码。
新成本：元模型设计成本、规则治理成本、配置版本管理成本、复杂业务和平台边界的判断成本。
```

### 14.3 层3 模型（L2，最重要）

```text
业务对象：客户、商品、仓库、门店、供应商、会员。
业务单据：销售单、采购单、入库单、出库单、调拨单、收付款单。
状态流程：草稿、提交、审批、审核、执行、完成、关闭、作废。
业务规则：字段校验、金额计算、库存影响、权限判断、状态流转。
页面视图：列表、表单、详情、审批页、报表页。
报表指标：销售额、毛利、库存周转、应收应付、客单价。
系统集成：支付、POS、财务、仓储、物流、微信。
扩展代码：复杂促销、特殊结算、特殊库存分摊、第三方协议适配。
```

### 14.4 层4 机制（L2）

```text
输入：业务人员通过页面配置元数据（对象/字段/规则/流程/权限）。
处理：平台引擎解析元数据 → 生成表单/列表/审批流 → 执行规则校验 → 更新单据状态 → 触发库存/账务影响 → 记录审计。
中间状态：草稿→提交→审批→审核→执行→完成；异常时作废或回滚。
输出：可运行的业务页面 + 业务单据 + 报表 + 集成事件。
异常路径：规则校验失败、状态非法跳转、库存不足、权限越界、外部接口超时。
恢复机制：配置回滚、单据反审、库存冲红、接口重试+幂等。
```

### 14.5 层5 边界（L3）

```text
适用：高频稳定结构（对象/单据/流程/规则/权限/报表/集成）可元数据化的业务。
不适用：算法密集型（推荐/风控模型）、极致性能场景、非结构化业务、高度定制交互。
规模临界点：元模型数量超过一定阈值后，配置复杂度可能超过手写代码。
误用：把不可配置的复杂业务硬塞进配置，导致配置变成另一种难维护代码。
```

### 14.6 层6 取舍（L2，最重要）

```text
约束：业务结构稳定性、配置表达能力上限、扩展代码与配置的边界。
优先保证：高频稳定结构的快速交付和一致性。
牺牲：极致灵活性和极致性能。
代价承担方：有特殊定制需求的业务方（需写扩展代码）。
规模变大后先爆的约束：元模型复杂度管理（配置爆炸）。
```

### 14.7 层7 张力（L3）

```text
灵活配置 vs 业务规范
快速交付 vs 长期可维护
页面自由 vs 模型一致
低代码表达能力 vs 复杂业务代码扩展
局部效率 vs 全局数据一致性
优先解决：模型一致（以模型约束页面，而非反之）。
代价转到：有特殊需求的业务方需写扩展代码。
```

### 14.8 层8 失败（L3）

```text
常见错：只做页面搭建，不做业务模型 → 后期规则散落、状态混乱。
隐蔽错：只有 CRUD 没有单据模型 → 审核/反审/作废/库存影响全变硬编码。
"看似正常实则已错"：前端/后端/流程/报表各算一套规则，数据表面一致实则口径不同。
检测：指标口径校验、状态机合法性校验、配置版本 diff。
预防：规则集中治理、状态机统一维护、指标口径统一定义。
```

### 14.9 层9 对比（L3）

```text
vs 简单表单工具（如宜搭/简道流）：
  本质区别：表单工具以表单为中心，ERP低代码以业务对象+单据+状态机为中心。
  分别优先保：表单工具保交付速度，ERP低代码保业务一致性。
  选 A 还是 B：轻量审批流选表单工具，涉及库存/账务/权限/集成的经营业务选 ERP 低代码。

vs 传统定制开发：
  本质区别：定制开发以代码为中心，ERP低代码以元模型为中心。
  分别牺牲：定制开发牺牲交付速度和一致性，ERP低代码牺牲极致灵活性。
  选 A 还是 B：稳定通用业务选低代码，算法密集或极致性能选定制开发。
```

### 14.10 层10 迁移（L4）

```text
可迁移原则：把领域稳定结构抽象为元模型，把差异留给扩展点。
跨领域类比：可迁移到任何"稳定结构 + 差异扩展"的领域——工作流引擎（流程为元模型）、规则引擎（规则为元模型）、BI 平台（指标为元模型）。
最小重建思路：用 JSON 描述对象/字段/规则，写一个能解析 JSON 生成 CRUD 页面+校验+状态流转的最小引擎，会发现难点在规则治理和状态机一致性，而非页面生成。
预测：若业务对象超过 200 个、规则超过 1000 条，纯配置会触顶，需引入规则引擎和领域特定语言（DSL）。
```

### 14.11 透镜检验（L3–L4，对照 7.25）

```text
盖尔定律：先做"对象+单据+状态机"能跑通的最小平台，规则引擎、DSL、插件市场按需演化；首版全配置化是最常见死法。
单向门：元模型 Schema 一旦被大量租户配置依赖即成单向门，升级必须自带迁移方案；页面布局、报表样式是双向门，可快速试错。
海勒姆定律：配置导出格式、字段默认值、单据编号规则等"未承诺行为"会被集成方依赖，变更须按契约管理。
变更成本分布：高频变化（规则、页面、报表）放配置层，低频变化（对象模型、状态机、对外契约）放平台层——预测错误的代价被隔离。
康威定律：平台组与业务交付组的边界，决定了"平台能力"和"扩展代码"的边界；边界错配会表现为扩展代码里塞满本应平台化的能力。
二阶效应：配置能力开放后，业务方会把配置当代码用（配置爆炸），需要提前设计配置的版本、评审和回滚机制。
古德哈特定律：把"配置化率"当团队 KPI 后，本应写代码的复杂逻辑会被硬塞进配置——考核指标必须与"配置/代码边界"设计原则对齐。
费米估算：日均千单量级不需要分库分表和异步削峰；先按 10 倍增长推临界，够用即止（防第二系统效应）。
```

### 14.12 一句话压缩（L4）

```text
ERP 低代码的本质，是把企业系统中稳定重复的"对象、单据、流程、规则、权限、报表、集成"抽象成元模型，并为不可配置的复杂差异留下代码扩展边界。
```

## 15. 最重要的三个方法

如果只能记住三个方法，优先记住：

```text
找不变量
找核心张力
做最小重建
```

原因：
- 不变量让你看到系统必须守住的底线。
- 核心张力让你看到设计为什么不得不取舍。
- 最小重建让你发现真正的复杂度在哪里。

三者关系（明确说明，避免与三问/最终判断混淆）：

```text
三问（第11节）是学习切入点，回答"从哪三个角度入手"。
三个方法是深挖工具，回答"用什么手段把锚点挖深"。
最终判断标准（第2节）是验收标准，回答"怎么算学到位了"。
三者是同一套心智模型的不同用途，不矛盾。
```

最终判断标准：

```text
我是否知道它抽象了什么？（本质模型）
我是否知道它牺牲了什么？（核心张力/取舍）
我是否知道它能迁移到哪里？（适用边界/迁移）
```

## 16. 本文和 AI 执行版的关系

本文是面向人的完整版，目标是训练理解力、迁移力和判断力。

`系统化拆解方法论-AI执行版.md` / `系统化拆解方法论-AI执行版-v2.md` 是面向 AI 的压缩执行协议，目标是让 AI 在回答分析类问题时稳定遵守：

```text
先建模，再回答；
先说明本质、取舍、边界；
区分直接事实、主流共识、经验判断、推断、假设；
按问题难度选择回答深度；
避免罗列式、复读式、无边界式回答。
```

二者关系：

```text
人类学习本文，用来建立方法论；
AI 使用执行版，用来稳定产出结构化分析；
遇到复杂问题时，先用本文理解全貌，再把关键流程压缩进 AI 执行版。
两份文档的术语、深度档、证据标签、三问表述保持一致。
```

用 AI 辅助拆解的实操建议——让 AI 当对手盘，而不是答案机：

```text
视角切换（7.11）：让 AI 扮演竞争者/攻击者/维护者，专门攻击你的结论。
反例驱动（7.9）：把你总结的规则给 AI，要求生成最小反例。
预测校验（7.18）：你先写下自己的预测，再让 AI 独立推演，对比分歧点——分歧处就是模型薄弱处。
证据链（7.23）：要求 AI 对每个结论按 5 档标注证据强度，专查"推断被写成事实"。
最小重建（7.13）：自己先写玩具版本，再让 AI 评审，而不是让 AI 直接生成——生成的代码不产生理解。
```

> 警惕：AI 流畅的输出极易制造"听起来懂了"的伪确定（反模式6）。所有关键结论仍须自己过第9节检查清单。
