## 树与树表视觉协议

| 状态 | 视觉要求 | 交互要求 |
|---|---|---|
| 层级 | 稳定缩进、连接关系、展开图标、可选路径提示 | 深层节点提供路径/面包屑，搜索结果保留祖先上下文 |
| 焦点 | 键盘焦点轮廓独立于选择底色 | 上下移动、左右展开折叠，焦点不自动改写业务值 |
| 选择/勾选 | 单选、高亮、复选、半选使用不同符号 | 明确父子联动范围和未加载后代语义 |
| 加载 | 节点级骨架或加载标记，不冻结整棵树 | 可取消/重试；并发展开结果不得串节点 |
| 拖拽 | 源节点、目标节点、前/内/后落点明确 | 实时阻止自身后代、错误类型和无权限目标，说明拒绝原因 |
| 编辑 | 行或单元格进入编辑态时保持层级上下文 | 提交、取消、校验错误和并发冲突可恢复 |
| 空态/错误态 | 区分无数据、无搜索结果、无权限、加载失败 | 每种状态提供对应恢复动作，不能统一显示“暂无数据” |

## 企业表单设计器应形成的独立方案

### 设计态

- 业务对象树与页面组件树分栏或分模式呈现，避免把实体层级和视觉嵌套混成一棵树。
- 画布使用语义容器和约束网格；复杂容器显示边界、名称、层级路径和允许内容。
- 属性面板分为数据、业务、布局、视觉、交互、状态权限、设备适配和高级配置。
- 提供新增、修改、查看、提交、审核、不同角色和设备的上下文预览。
- 调试区显示数据查询、公式/规则求值、组件状态、事件时间线和性能问题。

### 运行态

- 基础资料采用紧凑字段布局、左树/列表/详情主从结构和清晰的权限反馈。
- 单据采用稳定命令栏、主表字段区、分录 TreeGrid/DataGrid、汇总和附件区域。
- 动态表单允许引导式 Section/Steps，也允许高密度面板，但不在同一页面无节制混用。
- PC 与移动共享语义和业务规则；移动重新安排字段顺序、命令入口、树导航和分录呈现。

## 第二轮视觉校正

### 已确认

- Power Apps 当前官方资料足以确认其多区工作台和“画布 + 公式 + 属性”并行编辑模式，但不应把某张版本截图的具体尺寸固化为设计规范。
- DevExpress TreeList 的展开、选择、编辑、过滤、拖拽和分页是相互独立的视觉状态与操作入口；TreeGrid 不能只表现为“带缩进的表格”。
- Appian 官方组件分类以业务语义组织，Tree Browser 与 Column Browser 是两种任务模型，不是同一控件的皮肤切换。

### 降级处理

- Retool、Appian 的旧官方截图发生页面迁移时，只保留仍由当前文档支持的结构结论，不保留旧截图的色彩、间距和尺寸结论。
- Mendix、OutSystems 缺少当前官方视觉证据的部分，继续标为待验证；不依据搜索缩略图或二手截图补造视觉事实。
- 第三方“易用、现代、复杂”等主观评价没有场景和版本支撑时，不进入设计决策矩阵。

### 对自研视觉方案的约束

1. 视觉规范必须从任务模型出发：Picker、Browser、Column Browser、TreeGrid 分别设计，不靠换皮复用。
2. 高密度不等于元素拥挤；要通过稳定区域、对齐、层级和状态分离提高单位屏幕信息量。
3. 品牌色只承担主操作和关键选择，不同时承担焦点、校验、加载、拖拽和权限状态。
4. 版本易变的竞品像素值不进入设计令牌；只吸收经多源确认的结构、状态和交互原则。

## 第三轮视觉深化：Power Apps 与 Retool

- Power Apps 的关键不是“三栏外观”，而是 Tree view、画布、属性、公式和上下文命令共享同一选中对象；自研设计器需要统一 selection context，禁止各面板各自维护选中状态。
- Power Apps 上下文命令能降低噪声，但全量替换命令会造成位置跳动；采用稳定全局区加上下文区更适合高频企业设计任务。
- Retool 当前官方定位已经从传统内部工具画布扩展为提示、可视编辑、查询/函数、运行历史、返回数据和代码协同的工作台。
- Retool 官方产品页的黑白粗边框属于营销视觉，不能直接抄入产品；应吸收其调试信息贴近构建过程、可观测性是一等入口的原则。
- 两款产品共同说明：可视设计器不能把表达式、查询和运行状态藏到二级页面，否则复杂表单的数据联动排错成本会快速上升。
