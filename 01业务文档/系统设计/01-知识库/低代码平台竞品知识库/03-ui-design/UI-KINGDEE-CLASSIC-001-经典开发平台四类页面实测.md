# UI 设计：金蝶经典开发平台四类页面实测

## 本质模型

经典开发平台不是一个通用画布，而是一组共享元数据编辑内核、按业务语义分化的页面设计器：

```text
业务云 -> 应用 -> 功能分组 -> 页面类型 -> 模板 -> 多设计面 -> 属性/插件/权限动作
```

页面类型决定默认数据行为、操作语义、关联设计面和模板约束；控件画布只是其中一层。

## 四类页面对比

| 页面类型 | 数据职责 | 默认设计面 | 默认业务语义 | 扩展重点 |
|---|---|---|---|---|
| 基础资料 | 可编码、可引用的主数据/档案 | 表单、列表、移动表单、移动列表 | 启停、分配、组织、控制策略 | 多端视图、引用、组织管控 |
| 单据 | 流水型交易数据 | 表单、列表、移动表单、移动列表 | 提交、审核、反审核、打印、下推、分录 | 状态机、分录、流程、转换 |
| 动态表单 | 自定义交互页，默认不负责持久化 | 单一页面设计面 | 无固定业务操作 | 插件、门户、弹窗、向导、自由组合 |
| 报表 | 查询、过滤和分析结果 | 单一报表设计面 | 查询、刷新、引出、报表配置 | 数据源、计算字段、过滤、分组、轻分析 |

## 共同工作台

```text
顶部：设计面 / XML / 预览 / 保存
左侧：控件 | 大纲 | 实体 [移动端增加场景卡片]
中部：页面画布 + 上下文工具栏
右侧：业务 | 样式 | 布局
页面级：插件、权限动作、运行时按钮集合
```

这套布局的核心张力是“统一编辑体验”与“不同业务对象的强语义”之间的平衡。金蝶通过共享控件/画布/属性结构，同时给不同页面类型注入模板、默认字段、操作集和关联设计面来解决。

## 可迁移设计原则

1. 创建入口必须先选择业务语义，再进入画布；不能让所有场景都从自由页面起步。
2. 同一业务对象的桌面表单、桌面列表、移动表单和移动列表应是关联视图，不是四个互不相干的页面。
3. 模板必须携带数据、权限和操作语义，而不只是视觉布局。
4. 运行时命令应作为可配置控件存在，并拥有稳定标识、权限项和展示属性。
5. 页面级插件、操作集合和权限动作需要独立建模，不能混成按钮的前端事件。
6. 报表应单独建模“过滤条件 + 数据控件 + 计算字段 + 用户方案”，不能退化为普通表格组件。

## 对自研平台原型的直接差距

- 现有原型已有对象、页面、规则、权限和发布工作区，但缺少“应用 -> 功能分组 -> 页面类型 -> 模板”的创建链路。
- 现有页面画布需要补充表单/列表/移动表单/移动列表的关联视图模型。
- 单据已有表头与分录，但操作集、状态机、打印/下推和组织隔离仍过浅。
- 动态表单应作为明确的自由页面类型，而不是所有页面的默认形态。
- 报表需要独立的过滤方案、计算字段、数据控件和报表配置模型。

## 与新版开发平台的关系

同一业务对象可以在经典平台和新版平台中打开。经典平台以“页面类型 + 多设计面 + 属性”为主，新版平台进一步把实体、布局、页面规则、插件管理和批量编辑提升为并列工作区。

```text
共享业务对象元数据
  -> 经典平台：表单/列表/移动表单/移动列表 + XML/预览/保存
  -> 新版平台：实体/布局 + 页面设计/规则/插件/批量编辑/基本信息
```

这意味着自研平台不应把新版工作台理解为另一套数据模型。更稳妥的抽象是：一个业务对象元模型可以被多个专业编辑器投影和编辑，编辑器之间必须共享稳定标识、字段、实体、插件和规则引用。

## 边界

以上结论来自社区版当前环境的产品实测。插件服务端生命周期、物理表、权限优先级和发布快照仍未验证，不能据此声明内部实现完全一致。
