# 命名与编码规范

> 适用范围：所有项目的源码、数据库对象、API、配置、事件和文档。
> 来源：原 `09-统一命名规范.md`、`10-Java编码细则.md`、`13-前端编码细则.md` 的通用部分。
> 不包含：特定平台固定包名、模块白名单和平台动态对象命名。
>
> 版本：v1.1 | 更新日期：2026-07-10

## 1. 约束等级

- `MUST`：违反会破坏契约、安全、可维护性或跨环境兼容，必须阻断合并。
- `SHOULD`：默认遵守；偏离时需在评审中记录理由。
- 条件规则：仅在项目采用对应语言、框架或基础设施时生效。

项目可以制定更具体的词表、目录和格式规则，但不得改变已公开标识的语义或降低本规范中的安全底线（详见04-安全与供应链规范.md）。

### 1.1 术语表

| 术语 | 定义 |
|---|---|
| ADR（Architecture Decision Record） | 架构决策记录，以版本化文档记录某项架构决策的背景、方案、理由和后果，可关联需求与验证证据。 |
| bounded context（限界上下文） | 领域驱动设计中的概念，指某一业务术语和模型具有明确、一致语义的边界范围；跨上下文同名的概念可能含义不同，不得强行合并。 |
| trace 上下文 | 在分布式调用链中传播的追踪关联信息（如 traceId、spanId），用于跨进程、跨线程关联同一请求的诊断链路；异步、线程池和消息任务必须显式传播或可靠恢复。 |
| 稳定标识 | 跨环境迁移、离线交换或对外暴露时保持不变且语义承诺稳定的标识，与内部主键分离；其作用域和兼容边界必须记录。 |

## 2. 业务词表与稳定标识

1. `MUST`：同一业务概念在同一 bounded context 或公开契约中使用一个规范术语。跨上下文同名异义或同义异名必须通过上下文边界、映射或兼容别名显式说明，不得强行合并不同领域语义。
2. `MUST`：需要跨环境迁移、离线交换或避免暴露存储拓扑时，区分内部主键、跨环境稳定标识和可读业务编码。仅在当前系统内公开且已承诺稳定的数据库 ID 可以作为外部标识，但必须记录不可跨环境复用、枚举防护和兼容边界。
3. `MUST`：公开字段名、枚举 code、事件名、错误码一经发布即视为兼容契约；改名必须遵循兼容演进流程。
4. `SHOULD`：显示名称、说明、标题等相邻概念使用项目词表定义的固定字段名，避免 `name/label/title/displayName` 无边界混用。
5. `SHOULD`：布尔值使用能直接表达判断或能力的名称，如 `is/has/can/enable/allow/required/disabled`；集合使用复数名。
6. `SHOULD`：缩写只使用项目白名单中的常见技术缩写；禁止拼音、无上下文单字母缩写和自造缩写（初始白名单示例：ID, URL, HTTP, JSON, XML, API, DTO, SQL）。

## 3. 代码命名

1. `MUST`：包、模块和目录名表达业务边界或技术职责；不得使用 `new/old/temp/common2` 等时间性或模糊名称承载长期代码。
2. `SHOULD`：类型后缀与项目已定义的职责词表一致。常见情况下 `Controller` 表示入口适配，`Repository/数据访问层（如MyBatis Mapper）` 表示持久化访问，`DTO/Command/Query` 表示边界数据；不得把示例后缀当成所有架构的唯一分层方式。
3. `SHOULD`：领域方法使用明确的业务动词，避免用 `process/doSomething` 掩盖真实职责；框架回调、事件处理器等约定入口可使用 `handle`，但类型或参数必须说明所处理对象。
4. `SHOULD`：Java 类型用 PascalCase，方法和变量用 camelCase，常量和枚举项用 UPPER_SNAKE_CASE，包名全小写。
5. `SHOULD`：前端组件文件用 PascalCase；hook 以 `use` 开头；事件属性用 `onXxx`；渲染函数用 `renderXxx`；受控组件优先使用 `value/onChange`。
6. `MUST`：禁止用 `data/info/temp/obj/manager/util` 作为缺少限定词的长期业务名称。框架约定或极短生命周期局部变量除外，但上下文必须明确。

## 4. 数据库、API 与基础设施命名

1. `MUST`：数据库库名、表名、列名使用项目统一的大小写和分隔规则；同一 schema 内不得混用。
2. `MUST`：数据库列名与 API 字段名由明确映射层转换，不得把物理存储字段未经审查直接暴露为外部契约。
3. `MUST`：索引和约束采用可识别的稳定模板，例如 `pk_{table}`、`uk_{table}_{columns}`、`idx_{table}_{columns}`；超过数据库长度限制时使用稳定截断规则。
4. `MUST`：外键或引用列名必须表达所引用标识的类型，如内部 ID、稳定 ID 或业务 code，不得混用。
5. `MUST`：缓存 key、锁 key、幂等 key 必须包含资源类型和隔离维度；多环境、多租户或多版本系统还必须包含对应维度。
6. `MUST`：领域事件名表达已经发生的业务事实，命令名表达请求执行的动作；事件、命令和通知不得用同一泛化名称混淆语义。具体时态遵循项目语言和命名约定。
7. `MUST`：错误码由稳定模块段和稳定编号组成；同一错误码的语义不得复用或漂移。

## 5. Java 编码规则（条件规则）

项目使用 Java 时：

1. `MUST`：源码使用 UTF-8；仓库应由格式化和静态检查工具统一 import、缩进和换行。
2. `MUST`：工具类不得持有可变静态状态；纯工具类应阻止实例化。
3. `MUST`：`Optional` 仅用于表达可缺失返回值，不作为实体字段、DTO 字段或方法参数。
4. `MUST`：事务边界放在业务用例层的可代理公开方法；不得依赖同类自调用触发事务。
5. `MUST`：禁止空 `catch`，也禁止捕获宽泛异常后仅记录日志并返回 `null/false`。异常必须继续抛出、转换为稳定业务错误，或落入可恢复状态。
6. `MUST`：线程池显式配置名称、容量、队列和拒绝策略，并传播租户、认证和 trace 上下文；禁止业务代码使用无界线程池或默认公共线程池执行关键任务。
7. `SHOULD`：record 用于不可变 DTO、命令、查询和值对象，不用于需要持久化代理或可变生命周期的实体。
8. `SHOULD`：当 Stream、`var` 或 Lombok 降低业务逻辑可读性时，改用显式类型和直接控制流。

## 6. TypeScript 与前端编码规则（条件规则）

项目使用 TypeScript 或组件化前端时：

1. `MUST`：启用严格类型检查。禁止无说明的 `any`；外部未知输入先使用 `unknown`，再做运行时校验或类型收窄。
2. `MUST`：API 类型应由契约生成或从单一 schema 导出，不得由多个消费者各自手写副本。
3. `MUST`：枚举 code 和协议常量集中维护或由契约生成，禁止散落魔法字符串。
4. `MUST`：服务端状态、跨组件 UI 状态和组件局部状态分别放在符合其生命周期的层级，不得无边界复制同一状态。
5. `MUST`：权限相关的隐藏或禁用只改善交互，不能成为安全边界；服务端必须独立鉴权。
6. `MUST`：富文本和其他不可信 HTML 只能经统一净化器渲染；禁止直接透传可执行事件、`srcdoc` 或任意样式表达式。
7. `SHOULD`：组件保持单一职责。项目应通过 lint 或评审设置复杂度阈值，但不得仅按行数机械拆分。
8. `SHOULD`：异步提交期间保持按钮状态和布局稳定，防止重复操作；关键写请求仍需服务端幂等。

## 7. 复杂度与抽象准入

1. `MUST`：新增公共抽象前必须指出真实重复、明确变化点或边界隔离收益。只有一个调用方且没有明确变化点时优先保持局部实现。
2. `MUST`：不得为了通过单一行数阈值而制造无语义的转发层、工具类或包装器。
3. `SHOULD`：项目为方法长度、参数数量、嵌套深度、圈复杂度和组件复杂度设置可执行阈值；超限时先判断职责是否混杂，再选择拆分方式。
4. `MUST`：格式化、重命名和结构调整不得混入无关业务变更。

## 8. 文件、配置与文档命名

1. `MUST`：迁移脚本、构建脚本和跨平台工具文件使用 ASCII 稳定名，避免依赖本机区域设置。
2. `MUST`：配置项名称包含作用域和语义；禁止用 `flag1/enableNew` 等无法长期解释的名称。
3. `MUST`：ADR、任务、分支和发布物使用项目定义的可追踪编号规则，能关联需求、风险和验证证据。
4. `SHOULD`：文件名直接描述主要职责，不附加无意义的版本尾缀。历史版本应交由版本控制或明确归档目录管理。

## 9. 门禁与验收

- [ ] 项目词表和禁用词已更新，公开标识没有无兼容方案的改名。
- [ ] 格式化、lint、静态检查和命名扫描通过。
- [ ] 数据库、API、缓存、锁、事件和错误码的隔离维度与稳定性已检查。
- [ ] Java/TypeScript 条件规则按实际技术栈执行。
- [ ] 新增抽象有真实调用方或明确边界收益。
- [ ] 所有偏离 `SHOULD` 的项目已有评审记录。
