# 模块、兼容与发布规范

> 上位治理：`../00-工程治理总纲.md`；涉及公共契约、迁移和发布的 R3/R4 决策必须记录可逆性、承诺期限、退出成本和复审触发器。

> 适用范围：存在模块边界、对外契约、数据迁移或生产发布的软件系统。
> 来源：原 `19-模块边界与依赖治理细则.md`、`21-兼容性与演进细则.md`、`22-发布迁移与回滚细则.md` 的跨项目规则。
> 迁出边界：低代码元模型、页面 schema、插件包、动态 DDL 状态机和专用表结构进入低代码平台规范。

## 1. 约束等级与适用规则

- `MUST`：在适用范围和条款前提命中时违反即阻断合并或发布。
- `SHOULD`：默认遵守；偏离时记录理由、风险和替代控制。
- `条件规则`：仅在所述部署、版本或风险前提成立时生效。

项目可以采用等价机制，但必须证明兼容目标、失败语义和恢复能力。工具名与机制清单是可选实现，不得脱离项目事实强制引入事件、SPI、功能开关、分布式锁或灰度平台。

## 2. 模块边界

1. `MUST`：具有独立职责或演进边界的模块通过已声明契约通信，例如接口、API、事件、SPI 或项目已有等价机制；禁止绕过边界访问另一模块的内部实现。单体内部的普通代码组织不必伪装成远程或插件契约。
2. `MUST`：模块依赖遵守项目声明的有向关系并避免未受控循环；基础能力不得反向依赖使用它的具体应用层或界面层。确有双向协作需求时通过提取契约、回调或上移协调职责消除循环，不把“所有依赖只能同一方向”误解为只能形成单链。
3. `MUST`：对外或承诺稳定的接口不得暴露持久化对象、ORM 类型或数据库实现细节；模块内部接口按项目边界治理，不为形式分层复制无价值对象。
4. `MUST`：新增或改变跨模块调用时在评审中说明调用方向、契约稳定等级和失败语义；只有调用可降级、切换或回退时才要求回退方案，不得虚构不可行的 fallback。
5. `MUST`：不得因单个模块需要而把依赖或业务能力上提到全局公共模块。
6. `SHOULD`：持久化对象、传输对象、命令、查询和领域对象按职责隔离；不得把内部对象直接固化为长期外部契约。
7. `条件规则`：项目存在架构测试工具时，将依赖方向、内部包访问和层级对象边界固化为自动检查；没有工具时必须进行人工影响面评审。

## 3. 兼容等级与稳定契约

1. `MUST`：每类契约声明兼容范围和稳定性等级，例如稳定、实验或内部；名称由项目决定，不得默认所有内部结构都承诺兼容。
2. `MUST`：稳定契约的兼容版本演进只允许新增可选能力；删除、改名、改类型或改变既有语义必须使用新版本、兼容层或明确迁移窗口。本文中"兼容窗口"指新旧版本或新旧结构可同时被消费方接受的并存期；"迁移窗口"指从旧契约/旧数据结构切换到新契约/新数据结构、完成消费者迁移的执行期。两者可能重叠但关注点不同：兼容窗口面向消费方可用性，迁移窗口面向切换操作计划。
3. `MUST`：稳定契约中的枚举值、错误码和状态码不得复用旧值表达新含义；新增值时必须验证接收方的未知值策略，不能仅验证生产方序列化成功。
4. `MUST`：日期、时区、金额、分页、错误响应和标识符格式属于高稳定语义；破坏性变化必须启用新版本或兼容层。
5. `MUST`：新增响应字段必须允许旧消费者忽略；新增请求字段不得让旧客户端无法继续调用。
6. `MUST`：废弃契约保留读取或调用能力直到约定窗口结束，并给出替代项和移除条件。
7. `条件规则`：消息、离线文件、持久化快照或其他生命周期可能长于当前部署的序列化格式必须有可判定版本；读取端保留仍处兼容窗口内的历史样例和升级路径。
8. `条件规则`：有 OpenAPI、JSON Schema、消息 Schema 或数据库契约时，使用 diff 和历史样例回放阻断破坏性变化。

## 4. 既有系统演进

1. `MUST`：现有行为先视为兼容契约；无法确认依赖时保留旧路径，不得自行清理。
2. `MUST`：已确认的安全漏洞、越权、持续数据破坏或违法处理不能仅以兼容为由永久保留；必须设计隔离、迁移、补偿或版本化替代，并明确受影响调用方和回退边界。
3. `MUST`：新逻辑必须有明确启用边界且默认不改变非目标场景。存在未确认消费者、需灰度或不能立即回滚时，使用包装、隔离分支、功能开关（特性开关的登记、灰度和回滚要求参见 `07-配置错误恢复与可观测性规范.md` 第3节）、扩展点或版本化接口等项目已有机制；简单局部变更不得为套规范额外引入平台或抽象层。
4. `MUST`：修改前锁定旧行为，优先使用特征化测试；无法自动化时提供可重复的手工验证。
5. `MUST`：核心行为的所谓等价重写必须与功能变更拆分，并有明确验证证据。
6. `SHOULD`：高风险替换在切换前采用影子模式对比新旧结果；影子结果不得影响真实响应或写入。
7. `MUST`：已确认的安全、租户隔离、权限、数据完整性或合规缺陷不得以兼容为由继续暴露为默认可用路径；修复仍须记录证据、受影响消费者、历史数据处置、启用边界和回退/前向修复方案。

## 5. 数据库和数据迁移

1. `MUST`：结构或数据迁移在执行前产出计划，至少包含目标、操作、风险等级、预计耗时、兼容窗口、验证和回滚边界。
2. `MUST`：生产迁移按可追踪步骤执行，记录执行人、版本、时间、结果和脱敏错误；禁止边发现差异边无计划修改。
3. `MUST`：大表或锁敏感变更必须预检长事务、锁等待、存储空间和数据库算法能力，并设置有界超时。
4. `条件规则`：滚动发布、零停机发布或存在新旧版本并行窗口时，破坏性迁移采用“扩展、迁移、切换、收缩”顺序；在消费者完成切换并有证据前不得删除旧结构。允许停机且无并行版本时，也必须显式记录停机窗口和不可逆边界。
5. `MUST`：迁移脚本具有幂等或明确的断点恢复语义；部分成功时必须能识别已执行步骤并阻止盲目重试。
6. `MUST`：不可逆操作在执行前确认恢复目标、备份覆盖范围和最近可用的恢复验证证据，并明确代码回滚不能恢复已经改变的数据或物理结构。
7. `条件规则`：多实例或并发发布必须使用单调版本、CAS、fencing token 或等价机制，防止过期执行者继续推进。
8. `条件规则`：大批量回填必须分批、有界、可暂停和可续跑，记录游标、失败项和完成度；限速与批大小来自容量证据，不硬编码通用阈值。
9. `条件规则`：迁移期存在双写或双读时，必须声明唯一真值源、写入顺序、失败补偿、对账方式、切换条件和停止双写的时间点，避免把临时同步变成永久数据分叉。最小声明模板如下：
   ```text
   唯一真值源：<数据源/表/字段>
   写入顺序：<先写A后写B / 并行写 / ...>
   失败补偿：<补偿机制，如对账任务、补偿事务、告警阈值>
   对账方式：<对账频率、对账口径、差异处理流程>
   切换条件：<从双写切单写的判定标准>
   停止双写时间点：<明确的计划停止时间或版本里程碑>
   ```

## 6. 发布流程

1. `MUST`：生产发布必须有持久化或可审计的执行记录，能够回答版本、操作者、步骤、状态、失败位置和下一步恢复动作。
2. `MUST`：发布前验证构建产物、配置、依赖、契约兼容、迁移计划、监控和回滚方案。
3. `条件规则`：多步骤、异步或包含迁移的发布失败后必须进入明确的可恢复或需人工处理状态，不得只依赖请求线程或临时日志判断结果。
4. `MUST`：环境间迁移使用版本化、可校验的制品和正式流程，禁止用未审计手工操作替代发布通道。
5. `条件规则`：发布工具能够推导变更集，或发布包含数据/契约/权限变化时，导入或部署前输出变更预览；涉及删除、重命名、类型收窄或权限变化时阻断并显式批准。纯制品替换至少提供版本和提交差异来源。
6. `SHOULD`：高风险发布采用灰度或分批切换，并把观测指标、观察窗口、停止条件和扩大范围条件写入发布计划。

## 7. 回滚与恢复

1. `MUST`：每次发布明确回滚粒度、触发条件、执行人、预计时间和回滚后验证。
2. `MUST`：区分应用版本回滚、配置回滚、契约兼容回退与数据恢复；不得笼统承诺“可一键回滚”。
3. `MUST`：回滚前检查新写数据、异步消息、定时任务和外部副作用是否仍能被旧版本处理。
4. `MUST`：跨越不可兼容数据变化时阻断普通回滚，改走恢复或前向修复流程。
5. `MUST`：回滚完成后运行契约、数据一致性和关键业务回归，不能只确认进程启动成功。
6. `条件规则`：高风险或高撤销成本发布在首次进入生产前必须演练回滚或恢复路径；无法演练的步骤记录原因、替代验证和批准人，不得只凭书面步骤宣称可恢复。替代验证至少包括可在非生产环境执行的等价检查。

## 8. 验证矩阵

只执行本次变更实际命中的条目，未采用的能力标记为不适用并说明判断依据：

- 旧客户端对新服务、新客户端对旧服务。
- 滚动窗口内新旧生产者、消费者和数据库结构的组合兼容。
- 历史序列化样例、历史数据和旧版本制品回放。
- 发布部分失败、超时、重入和并发执行。
- 功能开关关闭时旧行为不变，开启时仅命中目标范围。
- 数据迁移断点恢复、重复执行和回滚边界。
- 双写或回填期间的对账、切换和停止同步条件。
- 回滚后异步任务、消息和外部副作用的一致性。

具体低代码发布状态、动态结构对账、元模型版本和应用包兼容矩阵不在本文定义，统一由低代码平台规范（参见 `../低代码平台规范/README.md` 及其所列契约）约束。
