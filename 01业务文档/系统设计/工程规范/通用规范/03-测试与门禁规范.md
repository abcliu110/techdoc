# 测试与门禁规范

> 适用范围：所有项目的测试策略、持续集成和人工评审门禁。
> 来源：原 `14-测试编写细则.md`、`15-门禁配置清单.md` 的通用部分。
> 说明：本文规定必须证明什么，不宣称任何具体项目已经实现这些门禁。
>
> 版本：v1.2 | 更新日期：2026-07-10
> 上位治理：`../00-工程治理总纲.md`；R3/R4 变更同时执行风险分级、架构适应度函数和保证论证要求。

## 1. 约束等级与术语

本规范沿用 `01-命名与编码规范.md` 第1节定义的约束等级（`MUST`、`SHOULD`、`条件规则`）。以下为本文件使用的补充术语：

| 术语 | 定义 |
|---|---|
| SLO（Service Level Objective） | 服务水平目标，量化定义系统可用性、延迟、错误率等质量指标的期望值与可接受范围。 |
| SBOM（Software Bill of Materials） | 软件物料清单，以标准格式记录制品所含全部依赖及其版本、来源和 License 的可追溯清单。 |
| 变异测试 | 通过对被测代码注入微小变更（变异体）验证测试套件能否检测到这些变更，以此评估断言质量的测试方法。 |

## 2. 门禁状态

门禁条目必须标记为以下状态之一：

- `要求`：风险决定必须具备，但尚未提供自动化实现证据。
- `已实现`：仓库中存在可执行配置或脚本，且 CI 会调用。
- `已验证`：最近一次执行记录通过，并可追溯到提交或发布。
- `人工评审`：当前无法可靠机器化，已有负责人、清单和留痕位置。

不得把示例 YAML、测试类名、伪代码或计划清单描述为“门禁已落地”。

## 3. 测试分层

1. `MUST`：存在纯业务规则、边界转换或状态流转时，单元测试覆盖关键正常、边界和失败分支。
2. `MUST`：使用数据库事务、迁移、锁、缓存、消息或外部适配器时，集成测试覆盖当前变更依赖的真实框架行为。
3. `MUST`：存在 API、事件、文件格式或共享 schema 等稳定边界时，契约测试覆盖兼容性。
4. `MUST`：认证、授权、租户隔离、解释器边界、敏感信息或不可信内容进入变更范围时，安全测试覆盖对应风险。
5. `MUST`：端到端测试只承载关键用户旅程和跨组件协作，不重复堆叠可由低层测试稳定证明的排列组合。
6. `SHOULD`：性能、容量和故障恢复测试围绕明确 SLO、数据规模和资源条件执行。

## 4. 行为锁定与测试数据

1. `MUST`：修复缺陷或重构已有行为前，先用最窄的回归测试锁定兼容契约；无法自动化时记录可重复的人工步骤和证据。
2. `MUST`：缺陷回归必须保留修复前失败证据和修复后通过证据。优先用测试完成红绿验证；无法安全恢复旧实现或问题不稳定时，可使用原始失败日志、最小复现或故障注入作为修复前证据，不得只提交与缺陷无关的永绿断言。
3. `MUST`：测试名称说明场景、动作和期望结果，不使用 `test1/works/normal` 等名称。
4. `MUST`：测试数据显式包含当前场景相关的租户、用户、状态、版本和关键业务值；禁止依赖共享环境中的偶然数据，也不为无关测试机械填充全部维度。
5. `MUST`：测试相互隔离、可重复执行，时间、随机数、ID 和外部系统使用可控边界。
6. `SHOULD`：复杂业务数据由 builder 或 fixture 构造，但不得抽象到看不出测试意图。

## 5. 风险驱动最低矩阵

按任务实际涉及的风险执行对应测试，不要求无关项目机械实现全部类型。

| 风险 | 最低验证 |
|---|---|
| 多租户 | 至少两个租户混测；读、写、关联、缓存、异步任务均不串位 |
| 权限 | 允许、拒绝、角色撤销、数据范围 OR 短路、字段裁剪一致性 |
| 金额/库存/订单 | 并发、重复请求、事务回滚、精度和状态非法流转 |
| API/事件契约 | 新旧消费者兼容、breaking-change diff、未知字段策略 |
| 数据迁移 | 真实数据库执行、历史数据、锁等待、半失败、回滚或前滚 |
| 消息/定时任务 | 重复、乱序、超时、死信、宕机恢复、多实例只执行一次 |
| 外部调用 | 超时、SSRF、防重复副作用，以及故障模型命中的重试、限流或熔断 |
| 文件/富文本 | MIME、大小、路径、压缩炸弹、XSS、CSV 公式注入、下载鉴权 |
| 缓存/分布式锁 | 缓存失效、版本收敛、锁过期、双持有者、fencing token |
| 敏感数据 | 日志、错误响应、审计、导出、保留期和脱敏 |
| 前端 | loading/重复点击、错误展示、权限不作为安全边界、响应式布局 |

## 6. 真实基础设施与替身边界

1. `MUST`：事务、SQL 方言、索引、DDL、锁和字符排序相关测试使用与生产兼容的真实数据库实例或容器。
2. `MUST`：缓存原子性、分布式锁、过期和故障恢复使用真实兼容缓存实例或容器。
3. `MUST`：mock/stub 可用于隔离单元边界和稳定失败注入，但不能作为数据库方言、事务、序列化、网络协议或外部适配行为的唯一证据；核心业务规则不得因过度 mock 而跳过必要的真实协作测试。
4. `MUST`：前后端、生产者与消费者涉及同一语义时复用同一版本化测试向量或契约资产，禁止各写一套相似样例。

## 7. 覆盖率与质量标准

1. `SHOULD`：已有稳定自动化测试体系的项目设置与风险相称的覆盖率基线，并在 CI 中防止无理由下降；新接管的遗留项目可先建立基线和增量门禁，不因追求全量数字阻断更高价值的特征化测试。
2. `MUST`：安全、金额、权限、迁移和状态机关键类采用更高分支覆盖要求。
3. `MUST`：覆盖率不能替代场景矩阵、断言质量、变异测试或缺陷回归证据。
4. `SHOULD`：对高风险纯逻辑采用变异测试或等价手段验证断言能发现错误。
5. `MUST`：不得通过排除核心文件、无断言调用、过度快照或只验证状态码来虚增覆盖率。

## 8. 持续集成阶段

项目按技术栈实现以下逻辑阶段；可合并执行，但证据必须可区分：

1. 仓库与编码检查：脏文件、生成物漂移、UTF-8、秘密和大文件扫描。
2. 依赖与构建检查：版本收敛、锁文件、可复现构建、License、CVE、SBOM。
3. 静态质量检查：格式化、lint、类型检查、复杂度、危险 API 和架构边界。
4. 单元与组件测试。
5. 集成测试：数据库、缓存、消息和迁移。
6. 契约与兼容性检查。
7. 安全测试和敏感信息扫描。
8. 构建产物、启动烟测和必要的端到端测试。
9. 发布前迁移、回滚、配置与可观测性证据检查。

任一必需阶段失败不得合并。临时豁免必须有负责人、原因、风险、到期时间和补偿措施；口头豁免无效。

## 9. 静态与架构门禁

1. `MUST`：格式化、lint 和类型检查以零错误为基线；warning 是否阻断由项目记录，但不得无限增长。
2. `条件规则`：项目已声明模块依赖、事务、鉴权或数据访问边界时，使用架构测试或等价静态检查固化可机器验证的部分；其余进入人工评审。不得为了通过本条凭空引入“统一 SQL 入口”等项目没有批准的架构。
3. `MUST`：扫描空 catch、默认公共线程池、硬编码密钥、拼接 SQL、浮点金额、未界定超时和危险反序列化。
4. `MUST`：数据库迁移扫描命名、注释、危险 DDL 和历史脚本篡改。
5. `MUST`：生成代码或契约生成物检查漂移，避免手写副本与源契约分叉。

## 10. 人工评审

无法稳定机器化的条款进入版本化清单，每项至少包含：规则、适用条件、评审人、证据链接、结论和日期。重点包括：

- 业务术语与领域边界是否准确；
- 新抽象和新依赖是否必要；
- 兼容窗口、数据迁移和回滚是否可执行；
- 错误消息、日志和用户提示是否泄露信息；
- 运维动作、break-glass 和人工补偿是否可审计；
- 性能阈值和容量假设是否有数据依据。

## 11. 验收证据格式

每次发布或高风险变更至少记录：

```text
变更/发布标识：
提交或制品版本：
执行环境：
必需门禁及状态：
测试命令或 CI 链接：
通过/失败数量：
迁移与回滚验证：
未执行项及原因：
剩余风险与负责人：
```

## 12. 验收清单

- [ ] 所有“已实现”门禁都能定位到仓库文件和 CI 调用点。
- [ ] 所有“已验证”门禁都有本次提交或发布的执行记录。
- [ ] 风险矩阵与变更范围对应，没有用覆盖率代替高风险场景。
- [ ] 集成测试使用了必要的真实基础设施。
- [ ] 回归测试具有先失败后通过的证据。
- [ ] 豁免和人工评审均有负责人、到期时间和留痕。
- [ ] 关键架构性质已转成可持续执行的 Fitness Function 或有证据的人工评审。
- [ ] R4 变更已形成 Assurance Case，并处理会推翻放行主张的 defeater。
