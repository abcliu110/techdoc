## 1. 边界与类型

- `MUST`：`builder` 负责设计态编辑，`renderer` 负责运行态渲染，`shared` 只放稳定契约；禁止互相依赖具体实现。
- `MUST`：固定平台 DTO 从 OpenAPI 生成；动态 record 由 `metaVersion + runtime schema` 校验，不伪装成静态生成类型。
- `MUST`：使用静态类型语言且启用最严格类型检查；若使用 TypeScript 则启用 `strict` 模式（`strict: true`，含 `noImplicitAny`、`strictNullChecks`、`strictFunctionTypes`、`strictPropertyInitialization`）。业务代码不使用未收窄的 `any`。外部数据先作为 `unknown` 校验。
- `MUST`：组件不得自行发明字段类型、权限或表达式语义，统一消费 `FieldTypeHandler` 暴露的前端能力描述。能力描述以版本化 TypeScript 声明文件（`.d.ts`）或 JSON Schema 形式从后端注册表导出，前端按 `metaVersion` 精确匹配加载；组件代码不得硬编码字段类型字符串分支，统一通过能力标记（`filterable/sortable/exportable/importable/persisted/auditable/searchable`）判断行为。

## 2. 字段组件

每种字段组件至少声明 `render/edit/validate/serialize/deserialize` 能力，以及只读、禁用、必填、错误和无权限状态。组件能力声明必须实现以下 TypeScript 接口骨架（字段类型扩展时同步更新）：

```typescript
/** 字段组件能力契约骨架（参考实现，实际以版本化 .d.ts 为准） */
interface FieldComponentCapability {
  fieldType: string;               // 对应 FieldTypeHandler 注册的 field_type
  metaVersion: string;             // 绑定的元模型版本
  render(props: RenderProps): JSX.Element;
  edit(props: EditProps): JSX.Element;
  validate(value: unknown, ctx: ValidationContext): ValidationResult;
  serialize(value: unknown): string;       // 提交格式
  deserialize(raw: string): unknown;       // 回显格式
  states: {
    readonly: boolean;
    disabled: boolean;
    required: boolean;
    error?: { code: string; message: string };
    noPermission: boolean;
  };
  capabilities: {
    filterable: boolean;
    sortable: boolean;
    exportable: boolean;
    importable: boolean;
    persisted: boolean;
    auditable: boolean;
    searchable: boolean;
  };
}
```

- `MUST`：link/multilink 选择器传稳定 `lid`，展示值与真实值分离；搜索、分页和权限由后端契约控制。
- `MUST`：普通文本默认按文本编码输出；richtext、code、模板和 Schema 渲染采用白名单组件/属性，URL 校验协议与目标，禁止任意 HTML、脚本或事件属性注入。CSP 是纵深防御，不能替代输出编码和净化。
- `MUST`：金额、日期、时区、空串和 NULL 的显示/提交语义与后端契约向量一致。

## 3. 表达式与 Schema

- `MUST`：前端表达式仅用于提示和预览，后端是最终裁决者；双端读取同一 JSON 契约样例。
- `MUST`：页面 Schema 只引用注册组件和白名单 props。未知的纯展示组件可用不可交互占位安全降级并记录诊断；未知必填字段、写入组件、校验器或动作必须阻断提交/发布，禁止以"降级渲染"静默丢数据或绕过规则。
- `MUST`：保存草稿允许的局部错误和发布必须阻断的错误分别定义；保存成功不等于配置可执行。
- `MUST`：发布前执行全图校验，覆盖字段引用、表达式依赖、动作、权限、页面和插件能力。

## 4. 大对象与并发编辑

- `MUST`：大对象设计器使用虚拟化、局部订阅和稳定 key，禁止每次输入重建整棵 Schema。
- `MUST`：保存携带 `revision`；冲突 UI 基于 base/current/submitted 三份版本计算真实差异，展示冲突字段、对方修改时间和合并/刷新动作，禁止在缺少基线快照时伪造字段级合并或静默覆盖。冲突差异计算的数据来源为服务端返回的三份 DTO 快照（base=用户开始编辑时的版本、submitted=用户当前提交的版本、current=DB 最新版本），前端不得自行推断或缓存 base；差异算法采用字段级 deep diff，对嵌套 JSON 按 `_v` 版本升级后再比较，未知字段标记为"无法合并"而非自动丢弃。
- `SHOULD`：拖拽、撤销重做和属性面板采用可测试的纯状态操作，视图层不直接修改共享模型。

## 5. 预览与运行态隔离

- `MUST`：草稿预览与已发布 `MetaGraph` 隔离，明确使用沙箱数据还是经权限裁剪的真实数据。草稿预览渲染器独立于运行态 renderer，消费草稿 `MetaGraph`（含未发布变更），渲染前必须完成与已发布版本相同的 Schema 全图校验（字段引用、表达式依赖、动作、权限、页面和插件能力）；校验失败时降级为错误占位并记录诊断，禁止渲染半配置页面。草稿预览的 `metaVersion` 标记为 `draft`，与已发布 `metaVersion` 严格隔离，不得混入运行态缓存。
- `MUST`：预览默认禁止真实写操作、外部通知、Connector 调用和不可逆副作用。例外能力必须通过独立 allowlist、明确环境/租户、二次确认、幂等键和审计启用；仅显示"预览"标识不足以授权真实副作用。预览渲染器在请求层拦截所有写操作（POST/PUT/DELETE）和副作用触发器（formula 执行、autonumber 生成、事件发布），未命中 allowlist 时返回模拟成功响应并标注"预览模式——未实际执行"，避免用户误判操作已生效。
- `MUST`：renderer 只消费发布版本；设计态错误不得泄漏为运行时半配置。运行态 renderer 检测到 `metaVersion` 为 `draft` 或 Schema 校验失败时，必须回退到上一个已发布版本并记录告警，不得使用草稿配置渲染用户可见页面。

## 6. 独立能力模型

报表、打印、菜单、门户、移动端和国际化不得被简化为页面 Schema 的几个属性：

以下要求按 PRD 里程碑生效。M1-M3 对移动端和完整国际化只要求结构预留与不阻断后续演进，不得据此宣称已交付完整移动端/H5、多语言切换或语言包能力；对应运行能力进入 M4 或后续批准里程碑后，才启用其完整测试与发布门禁。

- 报表必须有数据集、指标口径、权限裁剪和导出边界。
- 打印模板必须版本化并可回放历史数据。
- 菜单和门户必须按用户权限裁剪。
- 移动端进入实现里程碑时必须定义能力差异和降级方案；此前页面与组件契约不得绑定仅桌面可用的固定假设。
- 国际化进入实现里程碑时必须定义 key、fallback 和缺失翻译检测；此前界面文案至少从可演进的资源入口读取。

## 7. 最低证据

- 同一表达式样例的前后端契约测试。
- XSS/Schema 注入、无权限组件、旧 meta/新前端和预览写阻断测试。
- 大对象渲染性能、revision 冲突和 builder/renderer 依赖方向测试。
