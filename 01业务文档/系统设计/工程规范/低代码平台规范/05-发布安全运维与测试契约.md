# 发布、安全、运维与测试契约

> 状态：低代码平台强制契约
> 版本：v1.2（2026-07-10 修订：规则编号 L05-XX-YY、方法论引用、AI执行指引）
> 约束等级：本文件沿用 `./README.md` §0.1 定义的规则编号（L05-XX-YY）和 `../通用规范/01-命名与编码规范.md` §1 定义的约束等级。
> 方法论原则：遵循 `./README.md` §0 方法论声明。

## 1. 发布状态机

发布至少区分 `DRAFT -> VALIDATING -> PLANNED -> APPLYING -> PUBLISHED`，并保留 `VALIDATION_FAILED`、`APPLY_FAILED`、`PARTIAL_APPLIED`、`FAILED_NEED_REPAIR` 状态。

- `MUST`：发布前完成全图校验、权限检查、兼容检查和完整 DDL Plan；状态与日志持久化。
- `MUST`：DDL 逐步执行并记录前后快照、耗时、错误和 traceId。以下以 MySQL 为参考实现；使用其他数据库时须证明等价语义（包括 DDL 原子性、MDL 行为、锁等待和回滚能力）。MySQL DDL 不假设事务原子性。
- `MUST`：DDL Plan 生成后不可变并保存摘要；每一步定义前置条件、期望后置条件和幂等恢复策略。断点恢复先核对实际结构与步骤结果，禁止仅凭任务状态盲目重放 DDL。
- `MUST`：Reconciler 将元模型、物理登记表和实际结构分类为可自动修复、需人工确认或不可修复；自动修复只允许已证明安全、幂等的白名单动作，每次运行产出可读取报告。
- `MUST`：DDL Plan 评估表大小、预期 algorithm/lock、MDL 等待、长事务和磁盘余量；超出在线变更能力时采用扩展-迁移-收缩、受控维护窗口或经 ADR 选定的在线变更工具。
- `MUST`：删列、缩窄类型/精度、收窄选项或删除引用目标不可宣称自动回滚；回滚前置检查和用户文案必须说明边界。
- `MUST`：环境迁移只走应用包导入导出，禁止手工 SQL 搬动态结构。

### 1.1 DDL 日志契约

DDL 日志至少记录：`id`、`tenant_id`、`app_id`、`version_id`、`plan_id`、`step_no`、`object_code`、`table_name`、`ddl_type`、`ddl_sql`、`risk_level`、`execute_status`、`error_code`、脱敏 `error_message`、`started_at`、`finished_at`、`duration_ms`、`trace_id` 和 `create_time`。

- `MUST`：`plan_id + step_no` 唯一定位一个执行步骤，状态可用于断点恢复和 Reconciler 对账。
- `MUST`：`ddl_sql` 只记录结构 SQL，不包含业务数据值、密钥或凭证。
- `MUST`：失败和部分应用状态持久化，不能只留在进程日志中。
- `MUST`：DDL SQL、结构快照和错误详情按运维敏感数据授权访问并设置保留期；对外错误和普通业务日志不得复制完整 DDL。

## 2. 租户、权限与认证

- `MUST`：异步、MQ、定时、规则和插件上下文显式携带 tenantId；缺失时 fail-fast，禁止查全量或依赖 ThreadLocal 猜测。
- `MUST`：权限条件经 AST 合并；超级管理员仍受租户、审计、密钥和 break-glass 底线约束。
- `MUST`：组织变更后的历史数据范围按对象配置，并在 accepted 权限 ADR 中明确实时组织、owner 快照或其他语义。`OWNER_SNAPSHOT` 仅是当前候选默认，不得在 ADR 接受前写成已批准事实；M1 权限运行时发布前必须完成 ADR 和调岗/合并/树移动测试。
- `MUST`：浏览器会话防 CSRF；token 绑定签发方、受众和租户上下文，租户切换重新授权。禁用用户、撤销角色和租户切换通过版本失效信号加有界 TTL/轮询兜底，在 SLA 内使旧 token/权限缓存失效。

## 3. 配置、缓存和调度

- `MUST`：配置分平台、环境、租户和应用层，并记录来源；密钥不进入普通配置和日志。
- `MUST`：特性开关有 owner、启用条件、默认值、观测指标和删除期限；不得用永久租户特判替代元模型或插件能力。
- `MUST`：发布互斥使用由持久化状态校验的单调 fencing token 或等价机制，旧持有者不得继续推进状态或执行 DDL；租约续期只能改善可用性，不能单独作为正确性保证。Redis 不可用且操作正确性依赖其锁、版本或权限状态、又没有经验证的 DB/本地替代路径时，阻断对应发布或写操作；能够继续执行的普通 CRUD 必须仍通过 DB 版本、权限、租户和幂等校验，不做无证据的全平台一刀切。
- `MUST`：调度任务声明租户时区、业务幂等键、错过执行、补跑、并发和失败语义；系统按“投递可能重复”设计，以持久化认领/fencing 和业务幂等保证效果不重复，不宣称双实例下绝对不会重复投递。

## 4. 错误与恢复

- `MUST`：对外错误仅给安全分类、可处理信息和 traceId；SQL、表列、堆栈、内部类、路径、连接串和签名 URL 不出域。
- `MUST`：不存在与无权限在探测场景归一；字段校验可返回安全的精确错误。
- `MUST`：自动重试仅用于幂等操作，并设置次数/总时限、指数退避和抖动；DDL 失败先 Reconciler 分类，禁止盲目重试。
- `MUST`：审计、规则、通知声明同事务、可靠异步或可丢弃等级；可靠异步机制以 accepted ADR 为准，候选 outbox 不能在批准前成为唯一实现要求。存在死信时必须可查询、重放、放弃并审计，重放保留原 tenant、业务幂等键和因果 trace，不能生成一份无关联的新副作用。
- `MUST`：通知在发送时重新校验租户、收件人和通道权限，模板只读取声明字段并按敏感级别脱敏；审计记录模板版本、通道、收件人摘要和幂等键，不记录完整敏感正文。

## 5. 可观测性与容量

- `MUST`：traceId 贯穿 REST、异步、规则、DDL、插件和前端上报。
- `MUST`：采集动态查询耗时/SQL 数、MetaGraph 版本滞后、DDL 锁等待、Reconciler 差异、outbox/调度积压、表达式超时、权限拒绝和取号冲突。
- `MUST`：指标标签禁止 recordId、userId 等高基数值；日志携带租户、应用、对象和版本但对个人信息脱敏。
- `MUST`：动态表、行数、索引和元数据 JSON 使用当前部署已批准并版本化的预警/阻断线；`01-元模型与物理存储契约.md` 中尚未校准的候选值只能用于预警和压测规划。越过生效阻断线时由能力检查器阻断。
- `MUST`：提供只读模式、Reconciler、权限排查和备份恢复 Runbook；启用插件、outbox/等价可靠异步机制后，再分别提供插件禁用和对应重放 Runbook，不为未落地机制编写虚假操作步骤。
- `MUST`：为表达式、动态查询、导入导出、Connector、调度和插件设置租户级资源预算与并发/队列上限；超限采用可观测的拒绝或公平排队，禁止单个噪声租户耗尽共享线程池、连接池或存储。
- `MUST`：备份覆盖元数据、发布快照、物理登记、动态表、插件表、文件引用及其对象存储内容；对象内容由外部存储独立保障时，必须有可验证的同水位恢复契约。各介质使用一致性时间点/水位关联，备份加密且最小授权。恢复后先运行 Reconciler 和抽样业务校验，再解除只读；RPO/RTO 按部署模式配置并通过演练证明。

### 5.1 指标命名与告警

指标命名采用 `lowcode_{domain}_{metric}`，`{domain}` 为稳定小写英文标识（如 `query`、`schema_sync`、`reconciler`、`outbox`、`auth`），禁止使用 recordId、userId 等高基数值作为标签。核心指标至少包括 API/动态查询耗时、单请求 SQL 数、MetaGraph 加载和版本滞后、Schema Sync 耗时和锁等待、Reconciler 差异、outbox/调度积压、表达式超时、权限拒绝、取号等待和元模型保存冲突。

以下事件必须告警：租户上下文缺失、`PARTIAL_APPLIED`、`FAILED_NEED_REPAIR`、Reconciler 持续差异、MetaGraph 超过部署基线定义的收敛 SLO、DDL 锁等待超时、Redis 触发受限降级、敏感信息扫描失败和 break-glass 会话开启。具体阈值与 P0/P1/P2 等级由版本化部署告警策略维护。

### 5.2 数据生命周期与隐私

- `MUST`：建立敏感数据注册表，覆盖动态字段、审计 diff、日志、导出、附件、通知和插件配置，记录分类、用途、责任人、保留期、脱敏方式、允许出域通道和删除/匿名化策略；未知动态字段默认按更严格级别处理，直到完成分类。
- `MUST`：采集、日志、审计和导出遵循目的限制与最小化；仅因“便于排障”不能保存完整请求、对象快照或长期下载链接。生产问题需要临时提级采样时，必须有审批、范围、期限、访问审计和自动恢复。
- `MUST`：租户注销、用户删除、字段降级和保留期到期触发可追踪的数据发现与处置任务，覆盖主数据、软删数据、搜索索引、缓存、导出产物、附件和异步副本。法律保留或安全审计要求阻止物理删除时，记录依据和解除条件，并优先删除业务载荷或匿名化主体标识。
- `MUST`：备份中的到期数据通过备份轮换自然淘汰或具备受控重写方案；恢复旧备份后必须重放删除/匿名化水位，禁止让已完成处置的数据永久复活。
- `MUST`：隐私请求、租户导出和注销任务使用稳定任务号保证幂等，输出逐数据域结果、失败重试和未覆盖项；不得仅删除一张主表后宣称完成。

## 6. 最低测试与门禁

至少覆盖：

- 双租户、组织变更、权限短路、CSRF、旧 token、SSRF、XSS、恶意包和错误信息扫描。
- 敏感字段默认分类、保留期清理、租户注销、旧备份恢复后删除水位重放和法律保留解除。
- DDL 半应用、MDL 锁等待、Redis 闪断、锁双持有者、多实例版本收敛和恢复演练。
- 动态 SQL 全操作符、内置字段类型（清单见 `01-元模型与物理存储契约.md` §4.2）、软删除/级联、并发取号、规则副作用、导入重导和通知幂等。
- builder/renderer 契约、大对象性能、旧版本回放、应用包升级和 SaaS/私有化双模式。

门禁状态必须标成 `已实现`、`候选` 或 `人工评审`；只有仓库中的测试/配置和实际运行结果可以证明已实现，文档或代码骨架不能作为通过证据。

测试证据至少记录代码版本、配置/部署模式、数据集或 fixture、执行命令、执行时间和结果链接。SaaS 与私有化差异必须显式配置并进入兼容矩阵；禁止通过未登记的代码分支形成两套行为契约。
