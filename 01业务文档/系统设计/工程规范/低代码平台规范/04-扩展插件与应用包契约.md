# 扩展、插件与应用包契约

> 状态：低代码平台强制契约
> 版本：v1.2（2026-07-10 修订：规则编号 L04-XX-YY、方法论引用、AI执行指引、信任边界模型、SPI声明模板）
> 适用范围：仅低代码平台项目叠加生效；通用工程底线以 `../通用规范/` 为准，安全底线不得降级。
> 通用底线：本契约的 SPI、插件、应用包和 License 条款在安全、隔离、兼容和可回滚方面不得低于 `../通用规范/04-安全与供应链规范.md` 和 `../通用规范/06-模块兼容与发布规范.md`。
> 约束等级：本文件沿用 `./README.md` §0.1 定义的规则编号（L04-XX-YY）和 `../通用规范/01-命名与编码规范.md` §1 定义的约束等级。
> 方法论原则：遵循 `./README.md` §0 方法论声明（尤其是"扩展点经济性"和"信任边界不可越权"）。
> AI 执行指引：按 `./README.md` §4 任务路由表加载相关文件，PR 中引用 L04-XX-YY 编号，完成后执行 `../通用规范/00-AI执行检查清单.md` §4。
> 信任边界模型：T0 平台内核 > T1 内置插件 > T2 一方扩展 > T3 第三方应用包 > T4 不可信输入。低信任级别代码不得通过反射、序列化、ThreadLocal 等路径访问高信任级别资源。

## 0.1 SPI 声明模板

每个可执行或有副作用的 SPI 必须在仓库中维护声明文件，路径为 `spi-declarations/{spi-name}.yaml`：

```yaml
spi_name: ""                    # SPI 唯一标识
version: ""                     # 语义化版本号
compatibility_promise: ""       # major-breaking | minor-backward-compatible | patch-only
lifecycle:
  init_phase: ""                # 初始化时机
  destroy_phase: ""             # 销毁时机
  states: []                    # 生命周期状态枚举
thread_model:
  model: ""                     # stateless | stateful-thread-local | stateful-singleton
  thread_safety: ""             # thread-safe | single-thread | externally-synchronized
  concurrency_limit: 0          # 最大并发调用数，0=无限制
tenant_context:
  required: true                # 是否需要租户上下文
  propagation: ""               # explicit-parameter | thread-local | context-object
  missing_behavior: ""          # fail-fast | reject | anonymous-readonly
permissions:
  required: []                  # 所需权限列表
  grant_model: ""               # declarative | implicit | admin-only
timeout:
  default_ms: 5000              # 默认超时
  max_ms: 30000                 # 允许配置的最大超时
  on_timeout: ""                # fail-and-isolate | fail | default-value
resource_quota:
  max_memory_mb: 128            # 单次调用最大堆内存
  max_cpu_time_ms: 2000         # 单次调用最大 CPU 时间
  max_db_queries: 10            # 单次调用最大 DB 查询数
failure_isolation:
  on_failure: ""                # isolate | fail | default-value
  max_failure_rate: 0.1         # 最大失败率，超过则隔离
  recovery_strategy: ""         # circuit-breaker | retry | manual-restart
deprecated: false               # 是否已废弃
deprecation_path: ""            # 废弃迁移路径（引入 SPI 时必须声明）
```

## 1. 扩展点准入

- `MUST`：扩展点由业务需求和稳定契约证明，不为未来可能性预留通用 SPI（Service Provider Interface，服务提供者接口）。
- `MUST`：每个 SPI 声明版本和兼容承诺；可执行或有副作用的 SPI 还必须按实际能力声明生命周期、线程模型、租户上下文、权限、超时、资源配额和失败隔离。不适用项显式标为 N/A 并说明理由，禁止为满足清单伪造无意义接口。
- `MUST`：插件只能通过公开上下文和服务门面访问平台，不得直接访问内部 Entity、Mapper、动态表或线程本地租户信息。
- `MUST`：插件注册、启用、升级、禁用和卸载均审计；失败不得破坏核心发布状态。

## 2. 字段类型 SPI

- `MUST`：`FieldTypeHandler` 是字段类型唯一后端 SPI，负责 DDL、转换、查询能力、审计和导入导出契约。每个 `FieldTypeHandler` 必须遵守 §1 中声明的版本、生命周期、线程模型、租户上下文、权限、超时、资源配额和失败隔离要求。
- `MUST`：M0 只允许内置字段类型处理器（具体清单见 `01-元模型与物理存储契约.md` §4.2），M3 前不开放第三方后端字段类型插件。
- `MUST`：新增处理器先提交契约向量与兼容评估，再实现；缺任一消费端能力时不得宣称完整支持。

## 3. 安全隔离

- `MUST`：表达式函数、Connector 和插件能力白名单化，禁止任意 IO、进程、反射、文件系统和未批准网络访问。
- `MUST`：Connector 防 SSRF，包括协议、域名/IP、DNS rebinding、重定向、超时、响应大小和租户级限流。每次解析及每次重定向都重新校验最终 IP，默认拒绝本机、链路本地、云 metadata 和保留地址；IPv6 地址必须先规范化（展开缩写形式、统一大小写、剥离 scope id）后再做范围和保留段匹配。私网目标仅允许通过独立内部 Connector 类型、目标 allowlist、受控网络出口和审计访问，不能由普通外部 URL 配置绕过。
- `MUST`：第三方后端代码默认不得进入共享 SaaS 运行时；开放前需插件沙箱 ADR、安全评审、资源隔离和退出机制。
- `MUST`：插件密钥使用平台批准的受控密钥存储或托管密钥服务，日志和错误不暴露凭证或内部路径；不得因组织尚无独立密钥服务而把密钥退回普通配置或源码。

## 4. 兼容与升级

- `MUST`：插件 SPI 和页面组件契约版本化；破坏性升级提供并行版本或迁移器，不原地改变旧语义。
- `MUST`：旧插件、旧 Schema 和旧应用包在承诺窗口内回放；升级前输出影响和冲突报告。
- `MUST`：启用标准对象升级时，必须使用“基准版本 + 独立客户增量”或 accepted ADR 规定的等价机制，不覆盖客户扩展；合并顺序、冲突判定和人工恢复入口必须确定且可审计。

### 4.1 工作流与长运行实例

- `MUST`：在途工作流、审批和长运行任务固定定义版本；发布新版本默认只影响新实例，不得让在途实例隐式切换语义。
- `MUST`：迁移在途实例必须有兼容预检、可重复 dry-run、逐实例结果、失败恢复和审计；字段、状态、角色或插件能力被删除时，发布影响报告必须列出受影响实例。
- `MUST`：回调、定时唤醒和人工任务携带实例版本与幂等标识，旧版本处理器在承诺窗口内可回放；超出窗口需先完成迁移或明确终止策略。

## 5. 应用包生命周期

本节仅在应用包、行业模板、插件包、报表/打印模板包或 License 能力进入产品范围时生效。产品需求可以确认必须治理的生命周期和安全不变量；包模型、市场流程、授权状态机及字段结构仍由 accepted ADR 或版本化项目契约决定，不得从 proposed ADR 推导为既定实现。

- `MUST`：应用、行业模板、插件、报表和打印模板包登记来源、供应商、版本、License、依赖、资产清单和升级策略。
- `MUST`：包包含 manifest、版本范围、依赖、逐文件校验和、信任级别、导入预检和卸载/回滚边界；外部或跨环境包必须签名并记录签名者。禁止用手工 SQL 或未登记 JSON 替代包管理。
- `MUST`：导入前校验压缩包大小、解压后总大小、文件数、压缩比（异常低压缩比提示可疑嵌入，阈值由部署画像配置）、签名、签名证书/密钥撤销状态、路径穿越、符号链接、引用完整性、权限、危险配置和目标环境兼容性；校验与解压使用同一规范化路径，防止检查与落盘不一致。
- `MUST`：License 过期的降级行为可预测且不破坏已有数据；续期、撤销和审计可追踪。

## 6. 最低证据

- 生命周期、超时、重复注册、插件异常隔离和资源配额测试。
- 恶意包、旧包升级、依赖缺失、License 过期和客户扩展冲突测试。
- SPI/API/Schema diff、依赖与 License 扫描以及 SBOM。
