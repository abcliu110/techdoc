## 1. 质量目标体系（评审与设计的统一标尺）

所有设计决策用以下八个维度评价。每个维度给出**可度量/可验收**的定义——不可度量的质量目标不算目标。

| 维度 | 定义与可验收标准 | 当前设计的主要短板 |
|---|---|---|
| **正确性/一致性** | 元数据、物理结构、缓存、快照四方在任意故障点后可收敛到一致；不变量（租户隔离、权限、只加法 DDL）有机器防线 | 发布非原子无状态机（A1）、lc_meta_ref 事务边界不清、JSON 并发 RMW 丢更新 |
| **安全性** | 六层权限收敛为单一判定入口；每条安全红线（注入/越权/沙箱/SSRF/XSS）有对应恶意用例测试集 + 静态检查 | 判定逻辑三处分写（/meta、返回裁剪、写校验）；认证/CSRF 维度整体缺失（07a-F9） |
| **可扩展性** | 新增字段类型/连接器/规则动作 = 实现 SPI + 注册，不改引擎核心代码；扩展点有能力描述符与兼容承诺 | 22 种字段类型行为散落在转换/DDL/校验/渲染四处 switch，加类型要改四处 |
| **性能** | 每个性能目标（冷加载<2s、list P95<500ms）绑定到具体任务卡验收 + 基准测试数据集；热路径零查库 | 指标只在 PRD/02 出现，任务卡验收未绑定；无基准数据集定义 |
| **可测试性** | 关键语义以"真值表/契约向量"形式定义，测试直接由表驱动生成；所有时间/ID/DDL 执行点有可替身接缝 | 类型转换、表达式前后端语义、权限组合均为散文描述，无法表驱动 |
| **可审计性** | 一切决策（权限拒绝、发布步骤、对账差异、越权操作）是结构化事件，含 traceId + 元数据版本；审计不可被同权限主体篡改 | 审计只覆盖业务写；权限拒绝/发布/对账无事件模型 |
| **易用性** | 建模→可用分钟级闭环；失败必有"人话"报告（发布预检、引用完整性、权限解释）定位到具体对象/字段 | 预检报告/权限解释无设计；错误码体系双层口径未定 |
| **可理解性/可演进** | 单一写路径、单一判定入口、显式上下文——读一处代码即可理解全链路；模块依赖方向有 ADR + ArchUnit 双保险 | 写路径存在旁路风险（导入/级联/规则未收口）；模块依赖 ADR 缺失 |

**执行 AI 义务**：在 00-总体架构 新增"§质量属性与验收锚点"一节，落上表内容，并将每个指标绑定到 04 任务卡的验收标准（见 §3-P 组）。

## 2. 七项核心架构改进

### ARCH-1 发布管线：持久化状态机 + DDL Plan 一等公民 + 物理登记表 + Reconciler

**现状问题**：发布 = "写快照 + Schema Sync DDL + 更新状态"三段非原子（陷阱 A1）；T-004 只有 DDL 日志无恢复能力；并发发布无带保护的互斥；diff 每次扫 information_schema（A3 隐患）；Reconciler 只在细则 22 中被提及但修复权限、触发时机、与登记表的关系全部不闭环。

**目标架构**：

```text
发布请求
  → [1] 全图校验（引用完整性，产出人话报告）
  → [2] 生成 DDL Plan（持久化 lc_meta_publish_task + plan JSON，含逐步骤 DDL、
        危险变更阻断项、预估锁影响；plan 可被人工查看=预检报告）
  → [3] 获取发布锁（DB 行锁 lc_meta_app.publish_lock + fencing token，
        不依赖 Redis 锁的存活性）
  → [4] 逐步骤执行 DDL：每步执行前写 journal(step, ddl, status=RUNNING)，
        执行后更新 status + 同事务更新物理登记表 lc_meta_physical
  → [5] 写 Version.snapshot（含 meta_hash 内容摘要）
  → [6] 激活版本（app.current_version 原子切换）+ 广播版本号
失败于任意步 → 状态停留在 FAILED_AT(step)，提供 resume/abandon 两个管理入口；
resume 前强制 Reconciler 对账（元数据 vs 登记表 vs information_schema 三方）。
```

**关键决策点（各写入 ADR-LOWCODE-PUBLISH-001）**：
- 状态机状态集：`VALIDATING → PLANNING → LOCKED → EXECUTING(step_n) → SNAPSHOTTING → ACTIVATING → DONE | FAILED_AT(step_n) | ABANDONED`，每个状态落库。
- **物理登记表 lc_meta_physical** 是 diff 的唯一日常数据源（表/列/索引/字符集/精度），information_schema 只在 Reconciler 对账和登记表重建时使用——同时解决 A3 的表数量扫描问题。
- Reconciler 差异分类与修复权限定死：`REGISTRY_DRIFT`（登记表≠information_schema：以 information_schema 为准自动修登记表）、`MISSING_COLUMN`（元数据有列物理无：阻断运行 + 工单）、`EXTRA_COLUMN/派生列`（登记不删除）、`TYPE_NARROWED`（工单）。触发时机：启动时、发布前强制、resume 前强制。
- 列类型对比决策表（解决详细设计问题 #2/#13）：逐项对比 类型/长度/精度/scale/字符集/nullable/默认值，结论枚举 `IDENTICAL / WIDEN(执行) / NARROWED(阻断) / TYPE_CHANGED(阻断) / COLLATION_CHANGED(告警工单)`，含 MySQL information_schema 实际返回格式的对照用例。
- 发布前 MDL 预检：检测目标表长事务/慢查询，DDL 会话 lock_wait_timeout 上限（07a-A5）。

**落地文档**：T-004 重写 §执行流程与恢复；T-002 增加 lc_meta_publish_task、lc_meta_physical 表 DDL；细则 22 的 Reconciler 章节按上述分类重写；细则 11 增加登记表规则；M0-测试规格增加"发布中断于每一步 + resume"的故障注入矩阵；新增 ADR-LOWCODE-PUBLISH-001。

### ARCH-2 统一访问决策内核：AccessView（一次判定、处处消费）

**现状问题**：字段可见性在 /meta 裁剪、list/get 返回裁剪、update 写校验三处分写（陷阱 B3）；data_scope 与 filters 的合并点、link 展开权限、导出/导入/规则各链路可能各自实现判定——漂移即越权。

**目标架构**：请求入口一次性计算不可变的 **AccessView**：

```text
AccessView（per request, per object，不可变）:
  objectPerms:  {read, create, update, delete, actions[]}
  fieldView:    每字段 → NONE | READ | WRITE   （六层链 L5 的最终折算结果）
  scopeAst:     data_scope 折算出的 filters AST 片段（强制作为根 AND 子节点）
  metaHash:     本次判定依据的元数据版本摘要
  permVersion:  权限缓存版本（用户/角色/组织树版本组合）
```

规则：
- 判定器唯一入口 `AccessDecider.decide(user, object, ctx) → AccessView`，**/meta、list、get、add、update、del、action、link 展开、suggest、导出、数据导入、级联写全部只消费 AccessView**，禁止任何链路自行判定——ArchUnit 禁止其他包引用权限元数据类。
- scope 合并只发生在 AST 层：`RootAnd(scopeAst, userFilters)`（陷阱 B2 的结构性解法）。
- link 跨对象展开：对目标对象再取一份 AccessView，无 read 则该展开字段整体不返回（定死，不报错）。
- AccessView 进入审计：权限拒绝事件记录 `denialReason + permVersion + metaHash`，可解释"当时为什么拒"。
- **权限解释器（易用性投资）**：管理端接口 `explain(user, object, field?) → 六层链逐层判定过程`，管理员排障"为什么他看不到这个字段"从猜测变成一次查询。

**落地文档**：02 §3 重写"实现要点"为 AccessView 设计；细则 10 增加判定器唯一入口 + ArchUnit 规则；细则 17 的 17-004 与本设计合并；细则 12 新增 suggest 接口（link 选项查询，复用 AccessView，解决细则评审问题 #6）；细则 27 权限拒绝日志字段按上述定义（解决问题 #18）。

### ARCH-3 请求级 ExecutionContext：版本钉住 + 租户 + 幂等的显式载体

**现状问题**：TenantContext 用 ThreadLocal 在异步链路丢失（B1）；MetaGraph 请求中途可能换版本（C3）；幂等键与 revision 双机制关系不清（细则评审问题 #15）；T-005 的"请求级版本固定"没有定初始化时机。

**目标架构**：入口 Filter 构造一个**不可变 ExecutionContext**，全链路显式传参：

```text
ExecutionContext:
  tenantId / userId / roles
  metaGraph:  本请求钉住的已发布 MetaGraph 引用（含 metaHash）
  accessViews: 惰性缓存的 AccessView map
  traceId / idempotencyKey / requestTime(Clock 注入)
```

规则：
- 初始化时机定死：认证 Filter 之后、业务 Handler 之前的 ContextFilter；ArchUnit 禁止 runtime 业务包直接调用 MetaGraphProvider / TenantContext.set。
- 异步边界（@Async、outbox 消费、调度任务）必须**序列化 ctx 摘要（tenantId/userId/metaHash/traceId）随任务落库**，消费侧重建 ctx——不依赖 ThreadLocal 传播（结构性消灭 B1，而不是修补传播器）。
- 动态 SQL 组装处保留 fail-fast 兜底：tenant_id 缺失直接抛异常。
- 幂等与并发分层定死：`idempotencyKey`（API 层去重，同 user+tenant+key 重复请求返回首次结果）→ `revision`（数据层 CAS）→ 状态流转带 from_state 条件更新。顺序：先幂等去重，再乐观锁。
- 写接口校验 metaHash：前端提交的 metaHash 过旧（跨发布）→ 明确错误码要求刷新（C3 陷阱2 的解法）。

**落地文档**：02 §1.2 管线首步改为"构造 ExecutionContext"；T-005 §7 补初始化时机 + ArchUnit 规则 + 伪代码；细则 12 幂等规则补分层说明（12-033）；细则 10 补异步上下文重建模板；M0/M1 测试规格补"发布中在途请求版本不混用""双租户并发"用例。

### ARCH-4 统一写管线 + Outbox 副作用总线

**现状问题**：写路径潜在旁路多——业务数据导入、级联删除、规则 assign、未来工作流各自持久化则 H1/E2 必然发生；notify/call_api 的可靠性（F2）、通知治理（07a-F11）各自为政。

**目标架构**：
- **单一写入口 `DataWriteService`**：API add/update/del、批量导入、级联操作、规则赋值、工作流写全部经过 02 §1.2 的同一条管线（权限→转换→规则→状态机→持久化→审计）。豁免必须显式声明（如导入豁免某类规则需配置 + 审计标记）。ArchUnit：动态表的 INSERT/UPDATE/DELETE 只允许出现在该服务包。
- **Outbox 副作用总线**：所有外部副作用（notify、call_api、webhook、缓存广播、通知）统一写本地 outbox 表（同业务事务），由调度器投递：重试/退避/死信/幂等键统一一套，不允许每类副作用各建机制。审计主记录同事务、规则日志走 outbox——C2 写放大的分级策略由此定死。
- **统一调度器**（07a-C6 的架构答案）：outbox 投递、autonumber 重置、归档、清理共用一个"DB 抢锁 + 心跳"的轻量调度组件，任务表含 tenantId 上下文摘要，禁止散落 @Scheduled。

**落地文档**：02 新增"§统一写管线与副作用总线"；细则 25（错误处理）与细则 22 的重试/补偿章节收敛引用 outbox；细则 19 增加 ArchUnit 依赖规则；新增 ADR-LOWCODE-OUTBOX-001。

### ARCH-5 字段类型 SPI 化（可扩展性的核心投资）

**现状问题**：22 种字段类型的行为散落四处（类型转换、DDL 映射、校验、前端组件映射），加第 23 种类型要改四处引擎代码；M3 插件想扩展字段类型无从挂接；D1 真值表、T-004 DDL 策略、13 前端组件表实为同一事物的三份手工副本，必然漂移。

**目标架构**：**FieldTypeHandler SPI**——每种字段类型一个自包含实现：

```text
FieldTypeHandler:
  typeCode()            // "decimal"
  ddlMapping(FieldDef)  → ColumnSpec（类型/长度/精度/索引建议）
  converter()           → 输入合法形态/非法拒绝/NULL 语义（= D1 真值表的代码化）
  validator(FieldDef)   → options 合法性（如 select 选项、decimal 精度范围）
  queryCapability()     → 可过滤 op 子集 / 可排序 / 可聚合（formula 声明不可下推）
  rendererKey()         → 前端组件注册键
  capabilities()        → 可 unique / 可作 title_field / 可进 in_filter …
```

- 引擎（转换管线、SchemaSync、查询构建器、/meta 输出）只面向 SPI 注册表编程，**零 switch-on-type**。
- **真值表即测试**：每个 Handler 附带表驱动契约用例（合法输入/非法输入/NULL/边界），D1 的"22 种类型真值表"不再是文档而是测试资产；前端组件表由 /meta 输出 rendererKey 驱动，消灭第三份副本。
- M3 插件扩展字段类型 = 实现 SPI + 声明 capabilities，平台按 capabilities 自动裁剪设计器选项——扩展点从第一天就有形状。
- formula/multilink 等未实现类型：Handler 注册但 capabilities 声明 M0 不可发布，SchemaSync 按能力阻断到**字段级**（对象其他字段照常发布，解决详细设计问题 #10）。

**落地文档**：01 §2.2 增加"字段类型 SPI 与能力矩阵"节 + 22 类型逐一的能力/属性清单（解决 PRD 层问题 #13）；T-004 §5 改为消费 SPI 的 ddlMapping；细则 23 的类型语义条款改为"真值表由 SPI 契约用例承载，文档只写原则"；细则 20（扩展点）挂接此 SPI；新增 ADR-LOWCODE-FIELDTYPE-SPI-001。

### ARCH-6 契约测试体系：语义用数据定义，不用散文定义

**现状问题**：类型转换、前后端表达式语义（G2）、权限组合（样例应用的角色×状态×字段矩阵）、恶意输入集全部以文字描述存在，无法表驱动、无法证明双端一致；M0 测试规格偏主路径（详细设计问题 #12）。

**目标架构**——四类**契约资产**（版本化 JSON/CSV，前后端/多模块共用）：
1. **类型转换向量**：`{type, input, expect: value|REJECT, note}` ×22 类型——后端 converter 与导入链路共用。
2. **表达式求值向量**：`{expr, context, expect}`——后端 Aviator 门面与前端镜像求值器**跑同一份向量**，G2 的"禁止大致兼容"变成 CI 断言。
3. **权限组合矩阵**：样例应用"客户-订单-审批"的 角色×状态×操作×字段 期望结果表（解决 PRD 层问题 #4），M1 骨架用手写元数据加载、M2 用设计器配置复跑同一矩阵（解决 M1/M2 验收依赖问题 #14）。
4. **恶意输入集**：注入 filters/sort/op、短路 scope、恶意导入包、XSS 载荷、CSV 公式注入——细则 14/15/17 引用同一资产库而不是各自描述。

配套接缝要求：Clock、IdGenerator、DdlExecutor、OutboxDispatcher 必须是可替身接口（写入细则 10/14）；核心链路测试矩阵强制含双租户/双实例/并发/发布中请求（H3 落地）。

**落地文档**：细则 14 新增"契约资产"章节与目录约定；M0-测试规格补 DDL 逐步骤故障注入 + 双租户发布；04 各任务卡验收改为引用对应契约资产（见 §3-P 组）。

### ARCH-7 审计事件流与可观测锚点

**现状问题**：审计只覆盖业务写；权限拒绝、发布步骤、对账差异、break-glass、导出、通知投递各自无结构化事件；无"每条事件锚定元数据版本"的设计，事后无法回答"当时系统认为的规则是什么"。

**目标架构**：统一**平台事件模型**（一张 lc_rt_audit_log 扩展 or 分类表，结构统一）：

```text
PlatformEvent:
  eventType（DATA_WRITE / PERMISSION_DENY / PUBLISH_STEP / RECONCILE_DIFF /
             BREAK_GLASS / EXPORT / NOTIFY_DELIVERY / DDL_EXEC）
  tenantId / actor / onBehalfOf（代客操作）
  traceId / metaHash / permVersion
  payload（字段级 diff / 拒绝原因 / 步骤结果，敏感字段按 B6 分级脱敏）
```

- **metaHash 与 permVersion 进入每条事件**——审计从"发生了什么"升级为"在什么规则下发生了什么"，这是低代码平台（规则本身会变）区别于普通系统的审计刚需。
- 审计写入路径与 super 角色权限隔离（B5：同一主体不可删改自身审计）。
- 可观测最小指标集定死并进细则 27：发布状态机各状态计数/时长、Reconciler 差异数、权限拒绝率（按租户）、outbox 积压、MetaGraph 版本收敛延迟、动态表总数、租户配额水位——每个 🟠 级运行陷阱至少一个先行指标。

**落地文档**：02 §5 重写为平台事件模型；细则 27 指标清单具体化（含阈值数字）；细则 17 补审计防篡改条款。
