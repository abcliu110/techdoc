# T-207 全新表单设计器：工具栏与组件面板商业产品参考研究

版本：v1.0  
日期：2026-07-09  
范围：表单设计器左侧 Activity Bar、组件工具箱、字段入口、结构入口、右侧属性面板、顶部命令栏的产品参考。

## 1. 结论先行

商业级设计器的工具栏不是单一“组件列表”，而是一个工作台信息架构：

| 层级 | 商业软件常见做法 | T-207 应吸收 |
|---|---|---|
| 顶部命令栏 | 保存、撤销、预览、发布、运行、对齐等高频命令集中放置 | 保留顶部命令栏，不把全局命令塞进左侧组件面板 |
| 左侧活动入口 | 用窄栏、页签或侧边导航切换组件、字段、结构、数据、状态等模式 | Activity Bar 固定显示：组件、字段、结构、问题、版本、数据 |
| 组件工具箱 | 分类、搜索、拖拽，按当前设计器上下文显示可用控件 | 组件入口只展示 UI 组件，分组显示，默认三列紧凑图标卡 |
| 字段/数据入口 | 数据源、字段、查询、参数独立成树，字段可拖入画布生成绑定控件 | 字段入口展示 BO Field Tree，不能混入组件库 |
| 结构入口 | 页面/报表/控件层级独立成树，用于定位、嵌套编辑和重排 | 结构入口展示 Page Outline Tree |
| 右侧属性面板 | 选中对象后显示属性，支持分类、搜索、说明、表达式或绑定 | 右侧 Inspector 按业务、布局、样式、规则、权限、事件拆分 |
| 底部状态/问题 | 发布、校验、调试、运行状态集中展示 | Bottom Analyzer 展示问题、Quick Fix、发布 Gate 和定位 |

因此，T-207 的工具栏设计不能只是“左边放一堆组件”。正确目标是：左侧 Activity Bar 负责工作模式切换，Dock Panel 负责当前模式的资产或结构，中央画布负责 WYSIWYG 编辑，右侧 Inspector 负责选中对象的深配置，底部 Analyzer 负责问题和发布状态。

## 2. 金蝶表单设计器

来源：本地知识库 `UI-KINGDEE-FORM-002-表单设计器像素级参考.md`，证据来自金蝶开发者社区公开截图的像素级观察。

### 工具栏实现方式

| 观察项 | 金蝶做法 |
|---|---|
| 顶部工具栏 | 约 50-58px 高，蓝色底，白色线性图标加文字，高频编辑命令集中在顶部 |
| 左控件栏 | 约 265-290px 宽，承载搜索、分类和控件网格 |
| 控件项 | 约 70-85px 宽，一行三列，图标约 24-32px，文字 12-13px |
| 分类 | 分类标题约 28-32px 高，带展开箭头 |
| 属性面板 | 右侧约 275-300px，按业务、样式、布局等页签组织 |
| 画布 | 中央白底，占主要空间；选中态使用蓝色边框和快捷操作 |
| 层级定位 | 底部有约 24-28px 面包屑层级路径 |

### 对 T-207 的吸收

1. 吸收高密度、企业工具感、图标网格、搜索、分类折叠。
2. 吸收金蝶一行三列的中文可读性和扫描效率。T-207 默认采用约 304px Dock Panel + 三列紧凑图标卡，不靠列数超越，而靠字段入口分离、停靠折叠、可测试状态合同和复杂表格设计能力超越。
3. 控件卡默认只显示图标、中文名称、拖拽抓手。说明进入 tooltip 或详情，不常驻占用空间。
4. 字段入口和组件入口必须拆开。金蝶的“对象字段/通用控件”启发我们要有双来源，但 T-207 需要用 Activity Bar 明确分离。

## 3. DevExpress Web Report Designer

参考来源：

- Field List：https://docs.devexpress.com/XtraReports/17567/web-reporting/end-user-report-designer-for-web/interface-elements/field-list
- Report Explorer：https://docs.devexpress.com/XtraReports/115045/web-reporting/end-user-report-designer-for-web/interface-elements/report-explorer
- Properties Panel：https://docs.devexpress.com/XtraReports/17563/web-reporting/end-user-report-designer-for-web/interface-elements/properties-panel

### 工具栏实现方式

| 面板 | DevExpress 做法 | 设计价值 |
|---|---|---|
| Field List | 展示报表数据源、字段、参数；字段可拖到设计面生成绑定控件；支持搜索、数据源操作、计算字段、关系编辑 | 字段/数据不是普通组件，应是独立资产树 |
| Report Explorer | 用树形结构展示报表控件、Band、样式和组件 | 复杂嵌套必须有结构树辅助定位 |
| Properties Panel | 通过画布选中、结构树选中或下拉选择对象进入属性；属性支持分类/字母模式、搜索、说明 icon、表达式编辑 | 属性面板必须强大，不能只做简单 key-value 表 |
| Smart Tag | 元素附近可打开常用属性快捷面板 | 画布选中态可以提供就地快捷配置，不替代右侧 Inspector |

### 对 T-207 的吸收

1. `字段`入口参考 Field List，展示 BO、分录、字段、参数、计算字段。
2. `结构`入口参考 Report Explorer，展示 PageRoot、Section、Grid、Tabs、EntryTable、Column、DetailPanel。
3. Inspector 需要支持属性搜索、分类/高级模式、说明 tip、表达式或绑定入口。
4. 画布控件可有 smart tag，但只能放高频属性和快捷操作，深配置仍在右侧 Inspector。

## 4. Visual Studio Windows Forms Designer

参考来源：

- Toolbox：https://learn.microsoft.com/en-us/visualstudio/ide/toolbox
- Windows Forms Designer Tutorial：https://learn.microsoft.com/en-us/visualstudio/designers/walkthrough-windows-forms-designer

### 工具栏实现方式

| 观察项 | Visual Studio 做法 | 设计价值 |
|---|---|---|
| Toolbox | 显示当前设计器可用控件，支持拖放到设计面 | 组件工具箱必须上下文敏感 |
| 停靠能力 | Toolbox 默认可折叠在 IDE 左侧，可 pin、undock、hide | 左侧面板要可收起、可固定，保护画布空间 |
| 分组和管理 | 可 List View、Show All、Choose Items、Add Tab、排序、重命名 | 组件库要支持扩展、分组、筛选和自定义 |
| 布局控件 | TableLayoutPanel、SplitContainer 等作为控件进入工具箱 | 布局能力本身要组件化，而不是只靠自由拖拽 |
| 辅助设计 | snaplines、smart tags、Document Outline、Properties Window | 专业设计器依靠辅助线、智能标签、结构树和属性窗协同 |

### 对 T-207 的吸收

1. Component Palette 必须根据当前画布、容器、权限和阶段显示可投放状态。
2. 左侧 Dock Panel 支持折叠、固定、宽度约束。
3. 布局容器是一等组件，包括网格布局、分栏容器、弹性布局、页签容器、折叠容器、吸附容器等。
4. 组件注册要支持分类、图标、中文名称、允许投放目标、默认 schema、属性集和自动化抓手。

## 5. Microsoft Power Apps Studio

参考来源：

- Power Apps Studio：https://learn.microsoft.com/en-us/power-apps/maker/canvas-apps/power-apps-studio
- Canvas Components：https://learn.microsoft.com/en-us/power-apps/maker/canvas-apps/create-component

### 工具栏实现方式

Power Apps Studio 左侧不是一个单纯工具箱，而是一组工作入口：

| 入口 | 官方说明中的职责 | 对 T-207 的启发 |
|---|---|---|
| Tree view | 显示当前 app 的 screens 和 controls | 结构入口必须独立存在 |
| Insert | 向屏幕添加控件 | 组件入口只负责添加 UI 控件 |
| Data | 添加数据和连接器，例如表 | 数据入口必须独立存在 |
| Variables | 编辑变量和集合 | T-207 可在后续扩展表达式、变量和上下文数据入口 |
| Search | 搜索媒体、公式、文本等 | 大型设计器需要全局搜索或命令面板 |
| Components | 在 Tree View 的 Components tab 中创建和管理组件 | 组件不仅是物料，也可以是可复用资产 |

### 对 T-207 的吸收

1. Activity Bar 使用“工作入口”而不是“筛选 chip”：组件、字段、结构、问题、版本、数据。
2. 字段/数据/结构不应混在组件库中。
3. 搜索应分两类：组件面板内搜索和全局命令搜索。
4. 可复用组件和模板片段需要独立生命周期，不能只当普通控件。

## 6. OutSystems Service Studio

参考来源：

- Designing Screens：https://success.outsystems.com/documentation/11/reference/outsystems_language/interfaces/designing_screens/
- OutSystems UI：https://success.outsystems.com/documentation/11/building_apps/user_interface/

### 工具栏实现方式

| 观察项 | OutSystems 做法 | 设计价值 |
|---|---|---|
| Toolbox | Widgets 出现在 Service Studio 左侧 Toolbox，可拖到 UI Screens | 左侧工具箱是主要投放入口 |
| Widget | 输入、按钮、图片、布局、模式等都是 UI 构建块 | T-207 组件库要覆盖布局、录入、展示、动作、业务、表格、模板 |
| 属性配置 | 拖入 widget 后设置相关属性 | 拖放只解决放置，属性配置必须通过 Inspector 完成 |

1. 组件库要区分基础控件和业务模板片段。
2. 模板片段应生成一组 schema 节点，而不是简单单控件。
3. 组件投放后立即进入可配置状态，Inspector 应自动切换到最关键属性。

- View Menu：https://docs.mendix.com/refguide/view-menu/

### 工具栏实现方式

| 观察项 | Mendix 做法 | 设计价值 |

## 8. Retool Classic App IDE

参考来源：

- Classic App IDE：https://docs.retool.com/apps/concepts/ide
- Components：https://docs.retool.com/apps/concepts/components/

### 工具栏实现方式

| 区域 | Retool 做法 | 设计价值 |
|---|---|---|
| Left panel | 包含 components、queries、transformers、constants 等多个 tab | 左侧不是纯 UI 组件，而是应用资产工作区 |
| Add UI tab | 可搜索并添加 components，也可添加 modules | 组件入口需要搜索和模块化能力 |

| Status bar | 环境、releases、runtime、debug | 发布和调试状态应有固定区域 |

### 对 T-207 的吸收

1. T-207 的 `数据`入口不能只列字段，还要显示 BO、样例数据、绑定摘要和依赖。
2. 组件、字段、结构、数据、问题、版本是同级工作入口。
3. 对于复杂设计器，后续可补“依赖视图”：字段绑定、事件、查询、权限、规则之间的引用关系。

参考来源：

- Quickstart：https://docs.appsmith.com/getting-started/tutorials/the-basics/work-with-data-in-ui
- Widgets：https://docs.appsmith.com/reference/widgets

### 工具栏实现方式

| 观察项 | Appsmith 做法 | 设计价值 |
|---|---|---|
| Entity Explorer | 左侧 Entity Explorer 的 UI tab 展示可用 widgets | 组件入口属于应用实体管理的一部分 |
| New UI element | 点击后从 widget 列表选择并拖到 canvas | 组件选择和投放是分步动作 |
| Widgets 分类 | 输入、按钮、选择、展示、布局、媒体、开关等分类 | 组件分组要按用户认知和投放场景组织 |

1. 组件面板按中文分类展示：布局容器、基础表单、展示与状态、操作命令、业务套件、表格明细、模板片段。
2. 属性面板固定在右侧，浮层只用于临时选择、提示和快捷操作。
3. 组件注册模型必须包含：中文名称、图标、分类、标签、默认 schema、属性 schema、事件、允许投放目标、禁用原因、自动化 testId。

## 10. T-207 工具栏目标设计

### 10.1 左侧结构

| 区域 | 推荐尺寸 | 说明 |
|---|---:|---|
| Activity Bar | 48-56px | 只放工作入口：组件、字段、结构、问题、版本、数据 |
| Component Dock Panel | 约 304px | 默认组件库，三列紧凑图标卡 |
| Field Dock Panel | 320-360px | BO Field Tree、分录字段、系统字段、计算字段、关联字段 |
| Outline Dock Panel | 320-360px | PageRoot 到叶子节点的结构树 |
| Analyzer Dock Panel | 320-360px | 问题列表、筛选、定位 |
| Version Dock Panel | 320-360px | draft、published、archived、rollback |
| Data Dock Panel | 320-360px | BO、样例数据、绑定摘要 |

### 10.2 组件入口分组

| 分组 | 放置内容 |
|---|---|
| 布局容器 | 网格布局、分栏容器、弹性布局、页签容器、折叠容器、吸附容器、固定区、间距块 |
| 基础表单 | 单行文本、多行文本、数字输入、日期选择、基础资料选择、下拉选择、复选框、附件上传 |
| 展示与状态 | 只读字段、状态标签、统计摘要、提示文本、进度状态、图片展示、链接文本、分隔线 |
| 操作命令 | 主按钮、次按钮、按钮组、行操作、更多菜单、审批动作、导入导出、刷新按钮 |
| 业务套件 | 单据标题、单据头容器、汇总栏、审批信息面板、附件面板、流程轨迹、操作记录、打印区 |
| 表格明细 | 分录表格、表格工具栏、明细面板、明细表格、列操作、分页器、冻结列提示、表格汇总 |
| 模板片段 | 销售订单头模板、分录区模板、审批布局模板、左右分栏单据模板、折叠区模板、只读详情模板 |

### 10.3 组件卡形态

| 项 | 规则 |
|---|---|
| 默认信息 | 图标、中文名称、拖拽抓手 |
| 默认布局 | 三列紧凑图标卡 |
| 说明文字 | 不常驻，hover/focus 显示 tooltip，或在右侧详情区显示 |
| 禁用态 | 卡片可见但置灰，必须显示禁用原因 |
| 可投放性 | 拖拽过程中按目标容器显示 legal/illegal/reject reason |
| 自动化抓手 | 每个入口、卡片、拖拽柄、投放点、状态都必须有稳定 `data-testid` |

## 11. 不应采用的做法

1. 不要把字段资产混进组件库。字段属于 BO Field Tree，组件属于 Component Palette。
2. 不要在组件卡上常驻长说明。说明应放在 tooltip、详情区或文档链接中。
3. 不要做大卡片墙。表单设计器是高频企业工具，不是营销展示页。
4. 不要用收藏、最近、业务域 chip 代替正式分组。初版只保留搜索和七组组件。
5. 不要让属性面板浮动遮挡画布。主属性配置固定在右侧。
6. 不要只靠坐标做自动化测试。必须提供稳定抓手。
7. 不要让组件库宽度超过必要范围。组件面板越宽，画布越弱。

## 12. 验收门禁

| 门禁 | 判断标准 |
|---|---|
| IA-GATE-001 | 左侧 Activity Bar 必须有组件、字段、结构、问题、版本、数据六个入口 |
| IA-GATE-002 | 切换每个入口后，Dock Panel 内容必须完全不同，不能只是筛选同一列表 |
| UI-GATE-001 | 组件入口默认桌面态必须三列紧凑图标卡 |
| UI-GATE-002 | 组件卡默认不得常驻长说明文字 |
| UI-GATE-003 | 组件显示名必须全部中文 |
| UI-GATE-004 | 组件库不得出现字段资产 |
| UX-GATE-001 | 组件面板默认宽度约 304px，Activity Bar 不超过 56px，整体左侧默认不超过约 360px |
| TEST-GATE-001 | Activity 切换、组件拖拽、字段拖拽、非法投放、结构选中、属性编辑均可用 `data-testid` 黑盒回放 |
