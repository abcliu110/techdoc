## 1. 原型目标

本阶段产出 T-207 全新表单设计器的 PC Web 原型和交互图，目标是让实施顾问能够像在 Visual Studio Form Designer、DevExpress Web Designer、金蝶动态表单设计器中一样，在一个企业 IDE 风格工作台中完成复杂 Web 单据页面设计。

本原型不是报表设计器、不是数据大屏、不是营销页面。它必须表现真实 Web 表单/单据页面的可见即可得设计能力，包括强 Web 布局、丰富表单元素、强表格与分录、Analyzer 门禁、发布回滚和自动化测试抓手。

---

## 2. 视觉禁区

| 禁区 | 禁止原因 |
|---|---|
| KPI 卡片堆叠式首页 | 会把设计器误导成报表/仪表盘 |
| 大面积渐变、霓虹、插画、装饰性 hero | 不符合企业实施工具的长期使用场景 |
| 卡片嵌套卡片 | 降低设计器信息密度和可扫描性 |
| 过大圆角和低密度留白 | 不适合复杂单据设计和属性编辑 |
| 仅用坐标标注拖拽 | 不满足自动化测试平台模拟用户操作 |

## 4. 人工检查要求

用户人工检查时至少确认：

1. 是否像企业级 PC Web 表单/单据设计器，而不是报表或大屏。
2. 五区工作台是否清楚：Top、Left、Center、Right、Bottom。
3. 强 Web 布局是否可见：Grid、Flex、Section、Tabs、Split、Scroll、Sticky。
4. 强表格是否足够：分录、列设计、冻结列、汇总行、Level 1 明细、大表分页和虚拟滚动。
5. 左侧是否是可预测的 Activity Bar + Component Palette + BO Field Tree + Outline，而不是混用的普通树列表。
6. Component Palette 是否按布局容器、基础表单、展示与状态、操作命令、业务套件、表格明细、模板片段七组中文组件展示，并以三列紧凑图标卡呈现；说明文字必须进入 tooltip 或详情区，不能常驻挤占卡片；顶部不得出现收藏、最近、业务域筛选 chip；字段资产必须在“字段”工作栏独立展示。
7. Inspector 是否能配置字段、列、布局、规则、权限、状态。
8. 发布闭环是否完整：Analyzer、保存、预览、发布、stale draft、回滚、证据包。
9. 自动化测试抓手是否便于模拟 Activity 切换、组件拖拽、字段绑定、非法投放、列宽调整、分页滚动和性能定位。

## 4.1 本轮必须重做的左侧与建模主路径

当前原型已补齐下列回退图件，但仍不能作为最终完成版本；人工检查时必须逐项确认：

| 对象 | 必须展示 |
|---|---|
| Activity Bar | components、fields、outline、issues、versions、data 六个模式及 active/hover/collapsed 状态 |
| Component Palette 默认态 | 搜索、七个中文组件分组、三列紧凑图标卡；无收藏/最近/业务域筛选 chip |
| 左侧工具栏详细原型 | Activity Bar、组件入口、字段入口、结构入口、问题入口、版本入口、数据入口、七组组件、三列图标卡、tooltip、拖拽抓手、禁止项说明 |
| 组件卡片 | 默认只显示图标、中文名称、拖拽抓手；用途、能力标签、投放约束进入 tooltip/详情区；覆盖 disabled/unavailable/no-permission |
| BO Field Tree | Header、Entries、系统字段、计算字段、关联字段，必须独立于组件面板 |
| Outline Tree | PageRoot、Section、Field、EntryTable、Column、DetailPanel 层级；选中同步、展开折叠、empty、readonly、no-permission、invalid-move |
| 合法拖拽 | drag ghost、legal dropzone、投放完成节点、Inspector schemaPath |
| 非法拖拽 | illegal dropzone、reject reason、释放后不生成节点 |
| 端到端路径 | 从 Component Palette 拖布局容器和分录表格，从 BO Field Tree 拖业务字段生成字段控件与表格列，再配置 Inspector 并通过 Analyzer |

### 4.2 专业前端/UX 专项评审门禁

规则门禁不能替代专业评审。本阶段所有复杂 UI 原型，尤其是左侧 Activity Bar、Component Palette、BO Field Tree、Outline、Inspector、强表格设计区，必须附带专业前端/UX 专项评审结论。

| 评审维度 | 必须检查 |
|---|---|
| 信息架构 | Activity Bar 每个入口切换后显示什么内容，是否把组件、字段、结构、问题、版本、数据混淆 |
| 视觉密度 | 面板宽度、卡片列数、字号、行高、图标尺寸是否符合企业 IDE 工具箱 |
| 组件层级 | 主信息是否只保留图标和中文名，解释文字是否进入 tooltip/详情区 |
| 竞品吸收 | 金蝶参考左控件栏约 265-290px、控件项约 70-85px、三列高密度；本设计采用约 304px 三列紧凑图标卡，优先保证中文名称可读，并通过字段入口分离、状态合同、停靠折叠和表格设计能力实现超越 |
| 可测试性 | 每个入口、卡片、拖拽抓手、状态和 tooltip 是否有稳定 DOM 抓手或可观察状态 |

若专项评审发现信息层级、密度、可拖拽抓手或竞品吸收结论不成立，即使 PT/IX/PS 数量和普通门禁通过，也必须回退修改原型。
