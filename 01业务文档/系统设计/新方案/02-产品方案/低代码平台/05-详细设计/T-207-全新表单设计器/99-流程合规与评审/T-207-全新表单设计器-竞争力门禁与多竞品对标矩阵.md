## 1. 为什么要加竞争力门禁

表单设计器属于平台核心能力，不是普通表单页面。仅检查“是否有左侧工具栏、是否有组件、是否有属性面板”会导致形式通过，但产品能力弱于成熟商业软件。

本门禁要求：凡是设计器核心界面、组件工具栏、字段入口、画布、属性面板、表格设计、发布校验、自动化测试抓手等关键能力，必须与竞品进行对照，并证明至少在目标维度上达到或超过竞品。不能证明时，不得通过阶段 2 功能设计、阶段 4 自我评审和阶段 5 测试规格。

## 3. 竞争力评分规则

每个关键维度用 0-3 分评估：

| 分值 | 含义 |
|---:|---|
| 0 | 缺失，竞品有而我们没有 |
| 1 | 基础可用，但弱于多数竞品 |
| 2 | 达到主流竞品水平 |
| 3 | 已通过原型截图、交互图、测试断言和专项评审共同证明超过主流竞品 |

通过标准：

1. P0 核心维度不得为 0。
2. P0 核心维度平均分必须 >= 2.3。
3. 至少 5 个 P0 维度必须达到 3 分。
4. 对金蝶直接对标维度，T-207 不能低于金蝶，且至少 3 项必须超过金蝶。
5. 对 DevExpress / Visual Studio 直接对标维度，T-207 必须说明吸收点和不采用点。
6. 若某维度不适用，必须写 N/A 理由和替代竞争力证明，不能直接跳过。

### 3.1 证据驱动评分加严规则

引入 `06-流程-AI商业软件UI标杆证据驱动设计门禁.md` 后，竞争力评分必须遵守以下上限：

| 证据状态 | 最高评分 |
|---|---:|
| 没有 `source_id / screenshot_id / task_id` | 1 |
| 只有静态说明或隐藏契约，没有可见原型 | 1 |
| 有静态可见原型，但不能点击 / 切换 / 回放 | 2 |
| 有可见原型和浏览器点击切换验证，但缺竞品截图级对比 | 2 |
| 同时具备竞品证据、可见原型、交互回放、独立评审、自动验证和人工终审 | 3 |

因此，“超过竞品”不能只靠设计描述。任一维度缺少可追溯证据时，必须降级为“目标方向正确 / 达到候选形态”，不得写成“已超过”。

## 5. 阶段门禁规则

### 5.1 阶段 1：需求规格

需求规格必须声明：

1. 目标竞品清单。
2. 每个竞品要学习的能力。
3. 本产品必须超过竞品的维度。
4. 不照搬竞品的边界。

未声明时，阶段 1 不得完成。

### 5.2 阶段 2：功能设计

功能设计必须包含《多竞品对标矩阵》。

每个 P0 页面原型必须能回答：

1. 它参考了哪个竞品或哪类设计器。
2. 它在哪些方面达到竞品。
3. 它在哪些方面超过竞品。
4. 它在哪些方面暂时弱于竞品，是否阻断。

如果 P0 原型弱于竞品且没有补救方案，阶段 2 必须失败。

### 5.3 阶段 3：详细设计

详细设计必须把“超过竞品”的能力落到架构：

1. 组件注册模型如何支持分组、禁用、投放规则、属性 schema、自动化抓手。
2. 画布引擎如何支持布局能力、吸附、选中、层级、dropzone。
3. Inspector 如何支持属性分层、搜索、说明、表达式、绑定。
4. Analyzer 如何支持问题定位、发布 Gate、Quick Fix。

只在原型中表现、架构无支撑时，阶段 3 不得通过。

### 5.4 阶段 4：自我评审

自我评审必须增加“竞争力专项评审”：

| 检查项 | 必须回答 |
|---|---|
| 竞品覆盖 | 是否至少覆盖金蝶、DevExpress、Visual Studio / Power Apps，以及 3 个其他商业低代码产品 |
| 证据引用 | 是否引用本地知识库或官方文档 |
| 对标矩阵 | 是否逐项评分，是否有 0/1 分阻断项 |
| 超越证明 | 是否明确哪些维度超过竞品 |
| 原型一致性 | 对标结论是否已体现在 PT 图中 |
| 修复闭环 | 发现问题是否已回改上游文档和原型 |

任一 P0 维度为 0，或阻断维度未修复，阶段 4 不得通过。

### 5.5 阶段 5：测试规格

测试规格必须把竞争力门禁转成可执行断言：

1. Activity Bar 六入口存在且切换内容不同。
2. 组件工具箱默认三列紧凑图标卡。
3. 默认展开分组数量和折叠策略可断言。
4. 字段资产不混入组件库。
5. 右侧 Inspector 有属性分层、搜索和 schemaPath。
6. 左侧面板可折叠 / 恢复。
7. 组件拖拽、字段拖拽、非法投放、Analyzer 定位均可通过真实 DOM 操作回放。

不能转成测试断言的“超过竞品”声明，不得作为通过依据。
