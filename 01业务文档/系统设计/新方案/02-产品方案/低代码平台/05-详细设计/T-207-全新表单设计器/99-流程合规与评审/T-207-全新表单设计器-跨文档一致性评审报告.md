## 1. 评审概述

### 1.1 评审目标

验证 T-207 文档体系中跨文档关键概念的一致性，确保：
- Schema 结构定义（顶层命名空间、字段类型、路径风格）在所有文档中一致
- testId / schemaPath 命名规范在设计态、运行态、测试态三方统一
- EntryTable / 分录设计约束在布局引擎、表单体系、Schema架构中不自相矛盾
- Analyzer / Normalizer 职责边界清晰，输出可映射到测试用例矩阵

### 1.2 评审结论摘要

| 严重等级 | 问题数 | 说明 |
|---|---|---|
| 阻断（发布前必须修复） | 2 | EntryTable children 结构冲突；version/status 字段类型混淆 |
| 高风险（设计态/运行态功能损坏） | 4 | Rule ID 命名空间、Field testId 不完整、TestIdNormalizer 遗漏、Inspector Tab 语义冲突 |
| 中风险（维护/扩展隐患） | 4 | TableDesignerOverlay 前缀、状态机驱动边界、命名不一致、措辞模糊 |
| 建议（质量提升） | 5 | Analyzer 测试矩阵缺失、FlexLayout children 矛盾、required 同步未闭环、DetailPanel 子类型、schemaPath 覆盖缺口 |

总计：15 个跨文档一致性问题。

## 2. 阻断问题（发布前必须修复）

### 2.1 【阻断-01】EntryTable children 结构 vs columns/detail 冲突

| 项目 | 内容 |
|---|---|
| 严重程度 | 阻断 |
| 影响文档 | Doc 03（Web布局引擎）、Doc 05（强表格与分录设计）、Doc 07（Schema与运行时架构） |
| 问题描述 | Doc 03 第 6.2.1 节 EntryTable 示例使用 `"children": []`，与 Doc 05 明确规定的 P0 约束"EntryTable 使用 `columns[]` 和 `detail`，不使用 `children`"直接矛盾。Doc 07 第 7.1 节 SchemaNormalizer 子 Normalizer 列表中，TableNormalizer 职责定义为"迁移 EntryTable.children 为 columns/detail"，说明 Doc 07 也认定 `children` 是旧格式需迁移。 |
| 影响 | 设计态在布局引擎层级看到 `children`，但 Schema 存储层和 Analyzer 层要求 `columns`/`detail`，导致结构断裂——设计器能看、发布后 Analyzer 报错或运行态崩溃。 |
| 修复建议 | **Doc 03**：将第 6.2.1 节 EntryTable 示例中的 `"children": []` 替换为 `"columns": []` 和 `"detail": { "content": { "children": [] } }`。同时检查第 10.5 节示例 Schema 中所有 EntryTable 节点，确保无 `children` 残留。**Doc 05**：在第 12.3 节 P0 约束中增加"禁止在 EntryTable 顶层出现 `children` 属性，一经发现由 TableNormalizer 迁移，无法迁移时输出 Error"的明确约束。**Doc 07**：在 TableNormalizer 职责描述中补充"若 `children` 包含不可迁移节点（如 EntryTable），输出 Error 并阻断发布"的具体行为。 |

### 2.2 【阻断-02】version / status 字段类型混淆

| 项目 | 内容 |
|---|---|
| 严重程度 | 阻断 |
| 影响文档 | Doc 00（参考产品拆解）、Doc 01（产品定位与范围）、Doc 07（Schema与运行时架构） |
| 问题描述 | Doc 00 第 9.3 节 Schema 版本管理示例使用 `"version": "draft"`（字符串），而 Doc 01 和 Doc 07 明确将 `version` 定义为正整数（版本号），将 `"draft"` 赋给 `status` 字段（枚举字符串）。两个字段的命名、类型、语义完全不同。Doc 00 的示例将两者混用，会导致实现者误解 API 返回结构。 |
| 影响 | 若实现者参照 Doc 00 示例开发版本管理功能，会将 `version` 存为字符串 `"draft"/"published"`，与 Doc 07 要求的正整数版本号冲突，运行态加载、版本Diff、破坏性变更检测均无法正常工作。 |
| 修复建议 | **Doc 00**：将第 9.3 节示例中的 `"version": "draft"` 改为 `"version": 1` 和 `"status": "draft"` 两个字段，并在文字说明中明确 `version` 是正整数、`status` 是枚举字符串。**Doc 01/07**：在 PageSchema 示例中增加注释说明 `"version"` 和 `"status"` 的区别，防止后续读者混淆。 |

## 3. 高风险问题（设计态/运行态功能损坏）

### 3.1 【高风险-01】Rule ID 命名空间前缀不一致

| 项目 | 内容 |
|---|---|
| 严重程度 | 高风险 |
| 影响文档 | Doc 06（规则权限流程集成）、Doc 07（Schema与运行时架构） |
| 问题描述 | Doc 06 第 5.2 节 RuleSchema 示例使用 `"id": "rule_credit_required"`（带 `rule_` 前缀）。Doc 06 第 10.4 节 schemaPath 示例为 `rules.rule_credit_required.actions[0]`，即路径前缀 `rules.` 与 ID 前缀 `rule_` 完全一致。但 Doc 07 第 5.2 节标准 schemaPath 表只写 `rules.rule_credit_required`，没有说明 `rule_` 前缀是 ID 的一部分还是独立前缀。若 ID 改为 `credit_required`（无前缀），则路径变为 `rules.credit_required`，与 Doc 06 示例不一致。 |
| 影响 | 若不同模块按不同约定命名 Rule ID，Analyzer 的引用校验（RulePermissionNormalizer 检查 target/transition/serviceRef 可解析性）可能误报"目标不存在"，或在不同文档中描述同一 Rule 时路径不一致。 |
| 修复建议 | **Doc 06**：在 RuleSchema 定义中明确 Rule ID 的命名规范（如"以 `rule_` 开头的小写蛇形"）。**Doc 07**：在 schemaPath 标准表中增加一条 Rule schemaPath 的完整说明，包括 `rules.{id}` 中 `id` 是否包含 `rule_` 前缀的规范。**统一结论**：所有文档统一采用 Doc 06 示例的约定——ID 带 `rule_` 前缀，schemaPath 为 `rules.rule_xxx`。 |

### 5.3 【建议-03】state.required 与 validation.required 双向同步逻辑未闭环

| 项目 | 内容 |
|---|---|
| 严重程度 | 建议 |
| 影响文档 | Doc 04（表单元素与字段体系） |
| 问题描述 | Doc 04 第 6.3 节定义了 FieldNode 的 `state.required` 和 `validation.required` 之间的双向同步规则：修改 `state.required` 时同步到 `validation.required`，修改 `validation.required` 时同步到 `state.required`。但文档没有说明两个字段同时被修改（由不同操作在极短时间间隔内触发）时的冲突处理策略，也没有说明同步的方向优先级。当 `state.required` 由权限系统覆盖（Rule/Permission 设置只读但不影响必填）时，`validation.required` 是否应保持不变？ |
| 影响 | 极端并发场景下，双向同步可能导致 required 状态在两个方向上被反复覆盖，最终状态不可预测。实现者需要自己推断处理策略，可能导致行为不一致。 |
| 修复建议 | **Doc 04**：在第 6.3 节双向同步规则的末尾增加"并发处理"说明："当同一 FieldNode 的 `state.required` 和 `validation.required` 在同一事件循环内被不同操作修改时，以后写入者为准。Rule/Permission 设置只读时不影响 `validation.required`；`validation.required` 仅由用户在 Inspector 中直接修改或由字段类型默认值同步。" |
