# ADR-LOWCODE-FLEXORG-001: 弹性域、多组织业务关系与复杂编码规则

> 状态：proposed
> 日期：2026-07-06

## 背景

商用业务平台需要处理行业扩展字段、辅助属性、多组织职能和按组织/日期维度编码。若这些能力被简化为普通 JSON 字段或单个 `org_id`，后续权限、查询、报表、导入导出和历史数据范围都会失真。

## 决策

将 `FlexField`、`OrgRelation`、`CodeRule` 作为一等元数据：

- `FlexField` 表达维度、值类型、可见性、校验和升级归属。
- `OrgRelation` 区分行政组织、业务组织、核算组织、库存组织、销售组织等关系。
- `CodeRule` 表达组织维度、日期维度、流水策略、跳号策略和并发策略。
- M0 只保存 DTO 和版本快照，不生成物理列、不执行运行时查询。

## 理由

弹性域和多组织不是“多几个字段”，而是会改变权限、唯一性、编码、报表和历史口径的领域语义。提前一等建模可以避免后续用不可治理的 JSON 逃逸。

## 否决方案

- 把弹性域直接放 JSON 大字段：保存容易，但查询、权限、导入导出、审计和升级都不可控。
- 用单一 `org_id` 表达所有组织：实现简单，但无法表达业务组织、核算组织、库存组织等不同口径。
- 复用 autonumber options 表达复杂编码：会让简单取号和业务编码混杂，难以做并发和组织隔离。

## 后果

- M2/M3 前必须补物理存储、索引策略、查询下推、权限裁剪和报表口径设计。
- M0/M1 不得把弹性域当普通字段运行。
- 复杂编码不得绕过 23-080~083 的取号并发和跳号语义。

## 验证

- 维度组合唯一性和权限裁剪正确。
- 组织关系变化后历史数据范围按声明策略展示。
- 编码规则按租户和组织维度隔离，并发下不重复。
