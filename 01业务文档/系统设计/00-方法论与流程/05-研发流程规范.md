# 研发流程规范

> 本文档定义了研发全流程的阶段划分、交付物标准、质量门槛和角色职责。
>
> **核心原则**：每个阶段必须有明确产出物，下一阶段必须以上一阶段的产出物为输入。任何环节缺失"血肉"视为未完成。
>
> **适用场景**：
> - 人工执行时：以本文档为基准，各阶段由人工完成自检/评审。
> - AI 执行时：AI 作为主要执行者，每个阶段完成自检后进入下一阶段；关键节点（需求确认、变更评审、风险接受、License 判断、上线放行）必须有人工决策。AI/人工协作节点、L1-L4 复杂度分级和变更处理均以本文为准；自动执行与断点续跑遵循《00-总纲-知识库与产品交付双循环研发流程.md》第 6 章。
> - 阶段映射：本文 11 个阶段是《02-流程-知识库驱动新产品创建与迭代.md》P2-P5 的工程执行展开，不是另一套产品生命周期。产品机会、上线运营和复盘回写仍执行 02 的 P0-P1、P6-P7。

## 0. AI 执行入口

### 0.1 最小输入与自动判定

```text
最小输入：项目/产品目录（必须）+ 本轮目标（可从任务、缺陷或变更记录推断）。
上位阶段：先按 02 的 P0-P7 定位；本文只执行 P2-P5 对应活动。
起点判定：扫描 11 类交付物、评审记录和验证证据，从第一个未通过门禁的阶段继续。
复杂度：按本文第六章 Q2 的 L1-L4 自动建议；L3/L4 或跨安全/数据边界时送人工确认。
```

AI 不得因文档缺失就从阶段 1 重做全部工作；先确认是否存在等价产物，并记录映射。已通过阶段仅在输入基线、上位决策或规则版本变化影响它时重开。

### 0.2 阶段执行记录

每个阶段必须记录以下字段，可写入项目既有执行日志：

```yaml
stage_id: S1-S11
status: not_started|in_progress|blocked|awaiting_human|passed|failed|superseded
inputs: []
input_hashes: []
actions: []
outputs: []
gate_checks: []
evidence_refs: []
human_decisions: []
next_stage: S2
reopen_condition: 输入或规则变化条件
```

合法迁移：`not_started → in_progress → passed/failed/blocked/awaiting_human`；问题消除后 `blocked/failed/awaiting_human → in_progress`；基线被新决策替代时 `passed → superseded → in_progress`。禁止跳过 `in_progress` 直接声明 `passed`。

### 0.3 完成与升级

- 阶段通过：交付物存在、质量门槛逐项通过、RTM（需求追溯矩阵，定义见《02-流程-知识库驱动新产品创建与迭代.md》第 5.0 节）已更新、反例/失败路径已验证、验证证据为本轮新鲜结果。
- 人工决策：需求确认、范围取舍、accepted ADR、风险接受、License 判断、重大变更和上线放行。
- 阻塞升级包：问题、事实、推断、候选、权衡、建议、缺失信息/权限、恢复条件。
- 全流程完成：11 阶段均 `passed`，且 02 对应 P2-P5 门禁通过；上线仍须进入 P6，不得由本文自动宣称上线完成。

## 一、流程全景图

```
需求规格 ──→ 功能设计 ──→ 详细设计 ──→ 设计评审 ──→ 测试规格 ──→ 任务分解 ──→ 实现 ──→ 代码评审 ──→ 单元测试 ──→ 集成测试 ──→ 回归测试
   ↓            ↓            ↓            ↓            ↓            ↓            ↓          ↓           ↓           ↓           ↓
 需求文档    功能规格    技术设计    评审记录    测试用例    任务列表     代码       CR报告    测试报告    测试报告    测试报告
```

**交付链条**：需求规格 → 技术设计 → 实现 → 《测试规格文档》（人工可读）→ 测试用例 → 执行报告

**阶段编号**：共 11 个阶段（阶段 1-11），不含部署验证、上线、文档更新（属运维/上线管理范畴，可另行扩展）。

## 二、阶段定义与交付物

### 阶段 1：需求规格

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **做什么**——解决什么问题，给谁用 |
| 主要产出 | 《需求规格文档》 |
| 执行角色 | 产品经理 / 业务方 |
| 评审角色 | 产品负责人、技术负责人 |

#### 交付物标准：《需求规格文档》

```
必须包含：
├── 1. 背景与目标
│   ├── 解决的问题是什么
│   ├── 目标用户是谁
│   └── 成功标准（可量化）
├── 2. 功能需求清单
│   ├── 功能点编号（F-001, F-002...）
│   ├── 功能点名称
│   ├── 功能点描述（做什么、输入、输出）
│   └── 优先级（P0/P1/P2）
├── 3. 非功能性需求
│   ├── 性能要求（响应时间、并发量）
│   ├── 安全要求（权限、数据隔离）
│   └── 兼容性要求
├── 4. 约束与假设
│   ├── 依赖的系统/接口
│   ├── 限制条件
│   └── 假设前提
└── 5. 超出范围
    └── 明确不做的部分
```

**质量门槛**：
- [ ] 每个功能点有唯一编号
- [ ] 每个功能点描述清楚输入/输出/业务规则
- [ ] 优先级有明确依据
- [ ] 不做的部分已明确排除

---

### 阶段 2：功能设计

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **业务逻辑怎么跑**——流程、规则、状态 |
| 主要产出 | 《功能规格文档》 |
| 执行角色 | 技术负责人 / 架构师 |
| 评审角色 | 产品确认业务逻辑、技术评审可行性 |

#### 交付物标准：《功能规格文档》

```
必须包含：
├── 1. 功能点映射
│   ├── 需求功能点编号 → 设计功能点编号
│   └── 新增/修改/删除标识
├── 2. 业务流程图
│   ├── 每个功能点的流程图（泳道图或时序图）
│   ├── 异常流程与处理
│   └── 并行/分支/循环逻辑
├── 3. 业务规则清单
│   ├── 规则编号（R-001...）
│   ├── 规则内容（IF-THEN 格式）
│   └── 规则来源（需求哪条）
├── 4. 状态机设计（如涉及状态流转）
│   ├── 状态定义
│   ├── 状态转换条件
│   └── 转换触发动作
├── 5. 数据流设计
│   ├── 关键数据实体
│   └── 数据生命周期
└── 6. 与既有功能的边界
    ├── 调用既有模块的接口
    ├── 被既有模块调用的接口
    └── 需要兼容的旧逻辑
```

**质量门槛**：
- [ ] 每个需求功能点都有对应的设计功能点
- [ ] 业务流程覆盖正常、异常、边界三种情况
- [ ] 所有业务规则有编号，可追溯到需求
- [ ] 与旧逻辑的边界已明确标注

---

### 阶段 3：详细设计

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **技术怎么实现**——接口、表、类、方法 |
| 主要产出 | 《技术设计文档》 |
| 执行角色 | 开发工程师 |
| 评审角色 | 技术负责人 / 架构师 |

#### 交付物标准：《技术设计文档》

```
必须包含：
├── 1. 系统架构
│   ├── 模块划分与职责
│   ├── 类图/包图
│   └── 调用依赖关系
├── 2. 接口设计
│   ├── 对外接口（API）定义
│   │   ├── 请求参数（含字段说明、枚举值、必填/可选）
│   │   ├── 响应参数
│   │   ├── 错误码定义
│   │   └── 调用示例
│   └── 内部接口（模块间）
├── 3. 数据库设计
│   ├── ER 图
│   ├── 表结构设计（含字段说明、类型、约束、索引）
│   └── 旧表修改的影响分析
├── 4. 核心逻辑设计
│   ├── 核心方法的伪代码/流程图
│   ├── 事务边界设计
│   └── 并发控制策略
├── 5. 技术选型说明
│   ├── 新增依赖说明
│   └── 技术方案对比（如有）
└── 6. 风险点与应对
    ├── 技术风险
    ├── 性能风险
    └── 兼容性风险
```

**质量门槛**：
- [ ] 所有接口有完整字段定义（含类型、说明、枚举值）
- [ ] 表结构设计与功能规格的数据流一致
- [ ] 核心逻辑有伪代码或流程图
- [ ] 风险点有应对方案

---

### 阶段 4：设计评审

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **设计对不对、完不完整** |
| 主要产出 | 《设计评审记录》 |
| 执行角色 | 评审小组（技术负责人 + 相关开发 + 产品） |
| 评审依据 | 功能规格文档 + 技术设计文档 |

#### 交付物标准：《设计评审记录》

```
必须包含：
├── 1. 评审基本信息
│   ├── 评审时间、参与者
│   ├── 评审版本
│   └── 评审结论（通过/有条件通过/不通过）
├── 2. 完整性检查
│   ├── 需求功能点覆盖率：X/Y
│   ├── 业务规则覆盖率：X/Y
│   └── 接口字段覆盖率：X/Y
├── 3. 问题清单
│   ├── 阻断问题（必须修复才能通过）
│   ├── 高风险问题（建议修复）
│   └── 建议优化（可选）
├── 4. 遗留问题跟踪
│   ├── 问题描述
│   ├── 责任人
│   ├── 修复计划
│   └── 验证方式
└── 5. 评审签字
    ├── 产品负责人
    ├── 技术负责人
    └── 开发工程师
```

**质量门槛**：
- [ ] 所有需求功能点已覆盖
- [ ] 阻断问题已修复
- [ ] 评审有签字记录
- [ ] 遗留问题有跟踪机制
- [ ] 涉及用户数据/权限/支付的设计已通过安全审查

### 阶段 5：测试规格

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **怎么算完成**——测试标准、验收条件 |
| 主要产出 | 《测试规格文档》（人工可读） |
| 执行角色 | 测试工程师 / 开发工程师 |
| 评审角色 | 测试负责人 + 产品确认 |

#### 交付物标准：《测试规格文档》

> **核心原则**：《测试规格文档》必须是"人工可读的"，不是代码，是自然语言描述的验收条件。

```
必须包含：
├── 1. 测试范围
│   ├── 功能点测试映射（F-001 → TC-001...）
│   ├── 不测试的范围说明
│   └── 测试环境说明
├── 2. 用例清单（表格形式）
│   ├── 用例编号（TC-001, TC-002...）
│   ├── 所属功能点（F-XXX）
│   ├── 前置条件
│   ├── 测试步骤（编号的步骤序列）
│   ├── 预期结果（明确的、可验证的）
│   ├── 测试数据类型（正常/异常/边界）
│   └── 优先级（P0/P1/P2）
├── 3. 测试数据类型定义
│   ├── 正常数据用例
│   ├── 异常数据用例
│   └── 边界数据用例
├── 4. 验收条件
│   ├── 功能验收条件（逐功能点）
│   ├── 性能验收条件
│   └── 兼容性验收条件
└── 5. 挂起/恢复条件
    ├── 哪些情况可以挂起测试
    └── 挂起后如何恢复
```

**质量门槛**：
- [ ] 每个功能点至少有一个 P0 用例
- [ ] 每个业务规则有对应用例覆盖
- [ ] 异常和边界情况有用例覆盖
- [ ] 预期结果可验证（不是"功能正常"这种模糊描述）
- [ ] 产品确认验收条件

### 阶段 6：任务分解

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **谁在什么时候做什么** |
| 主要产出 | 《开发任务列表》 |
| 执行角色 | 技术负责人 / 项目经理 |
| 评审角色 | 团队确认 |

#### 交付物标准：《开发任务列表》

```
必须包含：
├── 1. 任务基本信息
│   ├── 任务编号（T-001, T-002...）
│   ├── 关联功能点（F-XXX）
│   ├── 任务描述
│   ├── 负责人
│   ├── 预计工时
│   └── 截止日期
├── 2. 任务依赖关系
│   ├── 前置任务
│   └── 阻塞任务
├── 3. 验收标准
│   └── 关联测试用例（TC-XXX）
└── 4. 进度跟踪
    ├── 状态（待做/进行中/完成）
    ├── 实际工时
    └── 完成时间
```

**质量门槛**：
- [ ] 每个功能点有对应任务
- [ ] 每个任务有关联测试用例
- [ ] 任务有明确负责人和截止日期

### 阶段 7：实现（编码）

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **代码写对了吗** |
| 主要产出 | 代码 + 《代码实现记录》 |
| 执行角色 | 开发工程师 |
| 评审角色 | 自检 + Code Review |

#### 交付物标准：《代码实现记录》

```
必须包含：
├── 1. 实现对照
│   ├── 功能点（F-XXX）→ 实现的文件/类/方法
│   └── 业务规则（R-XXX）→ 对应的代码实现
├── 2. 与详细设计的差异说明
│   └── 如果实现与设计不一致，说明原因
├── 3. 新增/修改的代码清单
│   ├── 新增文件列表
│   ├── 修改文件列表（附 diff 摘要）
│   └── 删除文件列表
├── 4. 数据库变更
│   ├── DDL 脚本
│   └── 数据迁移（如有）
└── 5. 配置变更
    └── 新增配置项说明
```

**质量门槛**：
- [ ] 实现与设计一致（差异已说明）
- [ ] 代码符合编码规范
- [ ] 复杂逻辑有注释说明
- [ ] 数据库变更有 DDL 脚本

---

### 阶段 8：代码评审

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **代码质量合格吗** |
| 主要产出 | 《Code Review 报告》 |
| 执行角色 | 评审者（除作者外的开发） |
| 评审依据 | 代码 + 详细设计文档 |

#### 交付物标准：《Code Review 报告》

```
必须包含：
├── 1. 评审信息
│   ├── 评审人、被评审人
│   ├── 评审日期
│   └── 评审范围（文件列表）
├── 2. 问题清单
│   ├── 阻断问题（必须修复）
│   ├── 高风险问题（建议修复）
│   └── 低风险问题（可选优化）
├── 3. 评审结论
│   ├── 通过
│   ├── 有条件通过（遗留问题）
│   └── 不通过（需重提）
└── 4. 遗留问题跟踪
```

**质量门槛**：
- [ ] 阻断问题已修复
- [ ] 高风险问题有处理方案
- [ ] 评审意见已归档
- [ ] 新增依赖已确认 License 兼容性（开源协议类型、是否与商业用途冲突）

**AI 代码特有检查项**（当代码由 AI 生成时必须增加以下检查）：
- [ ] **幻觉代码检测**：检查是否存在不存在的 API 调用、不存在的库函数引用、编造的方法名或类名
- [ ] **引用验证**：检查所有 import/require 引用的模块是否真实存在，引用路径是否正确
- [ ] 检查代码中是否存在看似合理但实际不存在的配置项或环境变量
- [ ] 检查 AI 生成的代码注释是否与实际代码逻辑一致

### 阶段 9：单元测试

| 属性 | 内容 |
|-----|------|
| 回答的问题 | **功能实现正确吗** |
| 主要产出 | 测试用例代码 + 《单元测试执行报告》 |
| 执行角色 | 开发工程师 |
| 评审角色 | 自检 |

#### 交付物标准：《单元测试执行报告》

```
必须包含：
├── 1. 测试执行摘要
│   ├── 执行用例数
│   ├── 通过数 / 失败数 / 阻塞数
│   ├── 执行时间
│   └── 测试覆盖率
├── 2. 用例执行明细
│   ├── TC-XXX → 通过/失败/阻塞
│   └── 失败用例的失败原因
├── 3. Bug 清单
│   ├── Bug 编号
│   ├── 关联用例
│   ├── 严重程度
│   ├── 状态（打开/修复中/已修复/延期）
│   └── 修复说明
└── 4. 测试结论
    └── 是否可以进入下一阶段
```

**质量门槛**：
- [ ] P0 用例 100% 通过
- [ ] P1 用例通过率 ≥ 95%
- [ ] 所有失败用例有 Bug 跟踪
- [ ] 行覆盖率 ≥ 80%，分支覆盖率 ≥ 70%

### 阶段 10：集成测试

| 属性 | 内容 |
|-----|------|

## 四、质量门禁（Gate）

每个阶段是下一阶段的门禁，上一阶段未通过不能进入下一阶段。

```
需求规格 ──[通过]──→ 功能设计 ──[通过]──→ 详细设计 ──[通过]──→ 设计评审
                                                                        ↓ [通过]
测试规格 ←─────────────────────────────────────────────────────── 任务分解
    ↑                                                              ↓
    └────────────────────────[通过]────────────────────────────────┘
                                                                        ↓
测试执行 ──[通过]──→ 集成测试 ──[通过]──→ 回归测试
```

**门禁检查点**：
- 阶段1 需求 → 阶段2 功能设计：功能点覆盖率 100%
- 阶段2 功能设计 → 阶段3 详细设计：业务流程完整、规则覆盖
- 阶段3 详细设计 → 阶段4 设计评审：接口完整、核心逻辑清晰
- 阶段4 设计评审 → 阶段5 测试规格：所有设计已评审通过
- 阶段5 测试规格 → 阶段6 任务分解：验收条件已确认
- 阶段6 任务分解 → 阶段7 实现：任务有明确负责人和截止日期
- 阶段7 实现 → 阶段8 代码评审：实现与设计一致、代码符合规范
- 阶段8 代码评审 → 阶段9 单元测试：阻断问题已修复、高风险问题有处理方案
- 阶段9 单元测试 → 阶段10 集成测试：P0 用例 100% 通过、行覆盖率达标
- 阶段10 集成测试 → 阶段11 回归测试：所有集成点测试通过、接口一致性验证通过

## 六、常见问题

### Q1：文档太重，流程太长怎么办？

**答**：可以按项目复杂度分级：
- **重大项目**：完整走全流程
- **常规迭代**：简化部分文档，但关键门禁（设计评审、测试报告）不能省
- **紧急修复**：跳过设计文档，但必须有简单的任务卡片 + 测试用例 + 测试报告

**判定矩阵**：

| 判定条件 | 分级 | 流程要求 |
|---------|------|---------|
| 涉及核心业务流程变更 | 重大项目 | 完整走全流程 |
| 涉及数据模型变更 | 重大项目 | 完整走全流程 |
| 功能增强 | 常规迭代 | 简化部分文档，关键门禁不能省 |
| 缺陷修复 | 常规迭代 | 简化部分文档，关键门禁不能省 |
| 线上故障修复 | 紧急修复 | 跳过设计文档，但必须有任务卡片 + 测试用例 + 测试报告 |

**底线**：交付链条不能断。每个阶段必须有产出物，下一阶段以上一阶段产出物为输入。

### Q2：AI 参与时，人工节点怎么保证？

**答**：在 AI 流程中，以下节点必须有人工确认：
1. **需求规格** → 你确认需求理解正确
2. **测试规格** → 你确认测试覆盖完整
3. **代码评审** → 你确认实现符合预期
4. **回归测试** → 你确认旧功能无退化

#### L1-L4 复杂度分级定义

AI 研发流程规范根据变更范围和影响面将任务分为 L1-L4 四个等级，用于决定流程裁剪程度和评审严格度：

| 等级 | 名称 | 定义 | 典型场景 |
|------|------|------|---------|
| **L1** | 单文件修复 | 仅涉及单个文件的修改，无接口/数据模型变更 | Bug 修复、文案修改、样式微调 |
| **L2** | 单模块开发 | 在单个模块内新增或修改功能，可能涉及多文件但不跨模块 | 新增页面组件、扩展模块内功能 |
| **L3** | 多模块功能 | 跨多个模块的功能开发，涉及模块间接口协作 | 跨模块业务流程、新增子系统 |
| **L4** | 跨系统架构 | 涉及跨系统/跨服务的架构变更，影响面广 | 系统集成、架构重构、核心流程改造 |

### Q3：如何避免"骨架有、血肉没有"？

**答**：每个交付物都有**质量门槛**。检查交付物时，逐条核对质量门槛，未达标视为未完成。

### Q4：旧功能改造时要注意什么？

**答**：
- 明确哪些旧逻辑必须保留（兼容性约束）
- 设计阶段标注与旧逻辑的边界
- 回归测试覆盖旧功能
- 上线后验证旧功能正常

## 七、需求变更处理

> 当需求发生变更（增加、修改、删除）时，必须触发变更流程，确保交付链条的完整性不被破坏。

### 7.1 变更类型定义

| 变更类型 | 定义 | 影响范围 |
|---------|------|---------|
| **需求增加** | 新增功能点，不影响既有功能 | 扩展新链路 |
| **需求修改** | 变更既有功能点的业务逻辑 | 可能影响既有功能 |
| **需求删除** | 移除已有功能点 | 可能影响依赖方 |

### 7.2 变更处理流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  变更发起                                                                        │
│      ↓                                                                          │
│  变更影响分析                                                                    │
│      ↓                                                                          │
│  变更评审（产品 + 技术）                                                          │
│      ↓                                                                          │
│  更新需求规格文档                                                                │
│      ↓                                                                          │
│  评估当前阶段，决定回退还是继续                                                   │
│      ↓                                                                          │
│  顺向补全（从当前阶段往前补）                                                     │
│      ↓                                                                          │
│  验证通过后继续后续阶段                                                           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 7.3 变更影响分析

收到变更请求后，必须分析以下影响：

```
变更影响分析报告必须包含：
├── 1. 变更内容描述
│   ├── 变更类型（增加/修改/删除）
│   ├── 涉及的功能点（F-XXX）
│   └── 变更前后的对比
├── 2. 影响范围分析
│   ├── 受影响的既有功能点
│   ├── 受影响的接口
│   ├── 受影响的数据库表
│   ├── 受影响的定时任务/消息
│   └── 受影响的依赖方
├── 3. 回归影响评估
│   ├── 需要回归测试的既有功能
│   └── 回归测试用例预估
├── 4. 工作量评估
│   ├── 各阶段新增/修改工作量
│   └── 预计延期影响
└── 5. 变更决策建议
    └── 接受/拒绝/延期 + 理由
```

### 7.4 不同阶段的变更处理策略

#### 阶段 1-2（需求规格/功能设计阶段）

| 变更类型 | 处理策略 |
|---------|---------|
| 需求增加 | 直接在需求文档中新增功能点，从功能设计开始顺向补全 |
| 需求修改 | 更新需求文档 + 功能设计文档，从功能设计开始顺向补全 |
| 需求删除 | 在需求文档中标注"已删除"，从功能设计开始顺向补全 |

#### 阶段 3-4（详细设计/设计评审阶段）

| 变更类型 | 处理策略 |
|---------|---------|
| 需求增加 | 需求规格 + 功能设计 + 详细设计，全部顺向补全 |
| 需求修改 | 重新评审详细设计，更新受影响的接口/表结构 |
| 需求删除 | 删除相关设计，更新设计评审记录 |

#### 阶段 5-6（测试规格/任务分解阶段）

| 变更类型 | 处理策略 |
|---------|---------|
| 需求增加 | 需求 + 功能设计 + 详细设计 + 《测试规格文档》 + 任务分解，全部顺向补全 |
| 需求修改 | 重新评估《测试规格文档》，更新受影响的测试用例和任务 |
| 需求删除 | 删除相关测试用例和任务 |

#### 阶段 7-11（实现到回归测试阶段）

| 变更类型 | 处理策略 |
|---------|---------|
| 需求增加 | 评估影响范围，可能需要回退到设计阶段 |
| 需求修改 | 评估影响范围，可能需要回退到测试阶段甚至设计阶段 |
| 需求删除 | 删除已实现的代码，执行回归测试 |

### 7.5 回退决策矩阵

| 当前阶段 | 变更类型 | 建议处理 |
|---------|---------|---------|
| 需求规格 | 任意变更 | 原地修改，无需回退 |
| 功能设计 | 任意变更 | 原地修改，无需回退 |
| 详细设计 | 任意变更 | 原地修改，评审通过后继续 |
| 设计评审 | 任意变更 | 原地修改，重新评审 |
| 测试规格 | 任意变更 | 更新《测试规格文档》，重新评审 |
| 任务分解 | 任意变更 | 更新任务列表 |
| 实现 | 需求增加/修改 | **回退到详细设计** |
| 实现 | 需求删除 | 删除代码，回归测试 |
| 代码评审 | 任意变更 | 原地修改，重新评审 |
| 单元测试 | 任意变更 | 原地修复，重新测试 |
| 集成测试 | 任意变更 | 评估影响，回退到对应阶段 |
| 回归测试 | 任意变更 | 评估影响，回退到对应阶段 |


### 7.6 变更后的顺向补全

当变更触发回退后，必须执行顺向补全：

```
顺向补全步骤：
├── 1. 定位回退阶段（受变更影响的最高阶段）
├── 2. 在该阶段完成变更内容的更新
├── 3. 执行该阶段的自检/评审
├── 4. 通过后进入下一阶段
├── 5. 重复步骤 3-4，直到通过当前阶段
└── 6. 继续后续阶段
```

**顺向补全原则**：
- 每次只能向前（正向）补全一个阶段
- 每个阶段必须完成自检/评审才能进入下一阶段
- 已完成且未受影响的阶段保持不变

### 7.7 变更记录

每次变更必须记录以下信息：

```
需求变更记录必须包含：
├── 1. 变更基本信息
│   ├── 变更编号（CHG-001, CHG-002...）
│   ├── 变更日期
│   ├── 变更原因
│   ├── 变更申请人
│   └── 变更审批人
├── 2. 变更内容
│   ├── 变更类型（增加/修改/删除）
│   ├── 涉及的功能点
│   └── 变更前后对比
├── 3. 影响分析
│   ├── 影响的功能点
│   ├── 影响的设计文档
│   ├── 影响的测试用例
│   └── 影响的工作量
├── 4. 变更决策
│   ├── 决策结果（接受/拒绝/延期）
│   ├── 回退到的阶段
│   └── 决策理由
└── 5. 变更执行记录
    ├── 更新的文档列表
    ├── 新增的任务
    ├── 删除的任务
    └── 回归测试结果
```
