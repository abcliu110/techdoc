#### C-1：DROP TABLE IF EXISTS 导致生产数据销毁风险

| 属性 | 值 |
|------|-----|
| **文件** | `储值方案.sql` |
| **影响表** | crm_deposit_category、crm_deposit_agreement、crm_deposit_plan、crm_deposit_charge_record、crm_deposit_coupon_schedule（共 5 张） |
| **风险等级** | CRITICAL |
| **影响范围** | 数据安全 |

**问题描述**：

所有 5 张新表的 SQL 建表语句均以 `DROP TABLE IF EXISTS` 开头。SQL 迁移脚本在正式环境如果因部署重试、灰度回滚等原因被执行第二次，已写入的数据将被永久清除且不可回滚。

```sql
-- 现有代码（危险）
DROP TABLE IF EXISTS crm_deposit_category;
CREATE TABLE crm_deposit_category (...);
```

**根因**：开发阶段使用 `DROP IF EXISTS` 便于反复调试，但上线前未替换为安全写法。

**修复方案**：

```sql
-- 改为 CREATE TABLE IF NOT EXISTS（安全写法）
CREATE TABLE IF NOT EXISTS crm_deposit_category (
    lid BIGINT PRIMARY KEY,
    sid BIGINT NOT NULL,
    ...
) COMMENT '储值方案分类表';

CREATE TABLE IF NOT EXISTS crm_deposit_agreement (...);
CREATE TABLE IF NOT EXISTS crm_deposit_plan (...);
CREATE TABLE IF NOT EXISTS crm_deposit_charge_record (...);
CREATE TABLE IF NOT EXISTS crm_deposit_coupon_schedule (...);
```

**验证方法**：
- 脚本执行两次，第二次应报 `Table already exists` 但不丢失数据
- 对比新旧脚本，确认无任何 `DROP TABLE` 语句

#### C-2：depositPlanIssue 字段可被外部调用方伪造，绕过券领取限额

| 属性 | 值 |
|------|-----|
| **文件** | `CrmCouponOpServicePlus.java`（第 ~338 行） |
| **关联文件** | `CouponOpAddDTO.java`（第 ~28 行） |
| **风险等级** | CRITICAL |
| **影响范围** | 业务安全 / 券超发 |

**问题描述**：

`depositPlanIssue` 作为 boolean 字段由 POS 调用方通过 DTO 传入，服务端 `CrmCouponOpServicePlus` 直接信任其值用于跳过领取校验：

```java
// CrmCouponOpServicePlus.java ~338行
boolean skipReceiveLimitCheck = notNeedCheck || couponOpAddDTO.isDepositPlanIssue();
// ↑ isDepositPlanIssue() 的值由外部调用方控制，服务端无任何来源验证
```

这意味着：任何调用方（内部或外部）只要构造请求时设置 `depositPlanIssue=true`，即可绕过所有券模板配置的日领上限和总领上限，实现无限批量发券。

**根因**：安全边界混淆——"是否储值方案发券"应由服务端根据业务上下文判断，而非由客户端声称。

**修复方案（分两步）**：

**Step 1**：移除 `CouponOpAddDTO.java` 中暴露的 `depositPlanIssue` 字段：

```java
// CouponOpAddDTO.java
// 删除：
@Schema(title = "是否储值方案发券")
private boolean depositPlanIssue;
// （不替换为其他公开字段）
```

**Step 2**：在 `CrmCouponOpServicePlus` 内部封装专用的储值发券方法，不接受外部跳限请求：

```java
// CrmCouponOpServicePlus.java 新增内部方法
/**
 * 储值方案内部发券 — 不受领取限额限制
 * 仅限 depositPlanChargeService 内部调用，不对外暴露
 */
private void couponOpAddForDepositPlan(CouponOpAddDTO dto, Long mid, Long sid) {
    CouponOpAddDTO internalDto = CouponOpAddDTO.builder()
        .templateLid(dto.getTemplateLid())
        .mid(mid)
        .sid(sid)
        // 不设置 depositPlanIssue，全部由内部逻辑控制
        .build();
    couponOpAdd(internalDto, true); // 第二个参数 true = skipReceiveLimitCheck
}

// 原有 depositPlanChargeService.onPaymentSuccess() 调用改为：
// couponOpAddForDepositPlan(dto, mid, sid);
```

**审计建议**：在 `couponOpAdd` 方法入口增加来源追踪日志：

```java
if (skipReceiveLimitCheck) {
    log.info("券领取限额被跳过: mid={}, templateLid={}, operator=INTERNAL_DEPOSIT_PLAN",
             mid, templateLid);
}
```

**问题描述**：

`tierRule` 在 VO 中定义为 `Object` 类型，数据库 JSON 列无 schema 校验。无效的 JSON 结构（如空数组、缺失必填字段）可直接写入，生产环境储值方案配置将处于不可用状态但无任何告警。

**修复方案**：在 Service 层 save/update 时增加 JSON 结构校验：

```java
// 在 CrmDepositPlanServiceImpl.save() 或 update() 中
private void validateTierRule(Object tierRule) {
    if (tierRule == null) return;
    try {
        JsonNode node = new ObjectMapper().readTree(tierRule.toString());
        if (!node.isArray() || node.isEmpty()) {
            throw new BizException("tierRule 必须是有效的非空数组结构");
        }
        for (JsonNode tier : node) {
            if (!tier.has("tierIndex")) {
                throw new BizException("tierRule 每个元素必须包含 tierIndex 字段");
            }
        }
    } catch (JsonProcessingException e) {
        throw new BizException("tierRule JSON 格式非法: " + e.getMessage());
    }
}
```

#### H-2：periodic coupon scheduler 未实现，券发放计划无法自动执行

| 属性 | 值 |
|------|-----|
| **文件** | `储值方案.sql`（crm_deposit_coupon_schedule 表） |
| **根因** | 数据库表已创建但无对应的定时任务消费者 |
| **风险等级** | HIGH |

**问题描述**：

`crm_deposit_coupon_schedule` 表定义了周期性券发放的调度计划，但整个代码库中没有对应的定时任务扫描和执行到期待发放的记录。配置的定期券将永远不会被自动发放。

**修复方案**：新增 Spring @Scheduled 定时任务消费者：

```java
// 建议路径：nms4cloud-module-crm-deposit-biz/src/main/java/.../DepositCouponScheduleJob.java
@Component
@Slf4j
public class DepositCouponScheduleJob {

    @Autowired private CrmDepositCouponScheduleService scheduleService;

    // 每分钟扫描一次到期的发放计划
    @Scheduled(cron = "0 * * * * ?")
    @Transactional
    public void triggerDueCouponSchedule() {
        List<CrmDepositCouponSchedule> dueSchedules = scheduleService.listDueSchedules();
        for (CrmDepositCouponSchedule schedule : dueSchedules) {
            try {
                scheduleService.executeIssue(schedule.getLid());
            } catch (Exception e) {
                log.error("执行券发放计划失败, scheduleLid={}", schedule.getLid(), e);
                scheduleService.markFailed(schedule.getLid(), e.getMessage());
            }
        }
    }
}
```

`services.executeIssue()` 内部必须处理幂等：检查 `lastFireTime`，若已执行则跳过。

```java
}
```

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public CheckOutVO checkOut(...) { ... }
```
