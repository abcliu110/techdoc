# 会员储值与营销（发券/券包）系统功能白皮书

## 1. 系统全貌与核心概念

本系统主要解决门店日常运营中的**会员储值**及**储值赠送（发券、送积分、券包）**的业务闭环。

### 1.1 新旧系统说明（非常重要）
目前系统正处于平滑过渡期，存在两套储值配置：
*   **旧版规则 (`CrmSaveRule`)**：目前的运行主力，规则相对固定（充X元送Y元+送Z券）。
*   **新版方案 (`CrmDepositPlan`)**：未来的运行主力，使用JSON存储，支持极度复杂的规则（指定门店、指定时段、阶梯档位、购买限制等）。目前主要在后台管理，正在与主流程对接中。

### 1.2 核心业务对象
*   **会员与会员卡**：人员基本信息（`Member`）与资金/业务载体（`Card`）分离。**所有储值和发券操作均针对会员卡 (`Card`) 进行。**
*   **优惠券 (`Coupon`)**：普通的满减/折扣券。
*   **券包 (`Coupon Package`)**：一组优惠券的集合。**在系统设计中，券包本身也是一种“优惠券”**，只需通过一个标识位区分，它会自动展开为多张子券发给用户。

## 2. 核心业务流程全景图

### 2.1 会员储值全流程 (ChargeService)
储值是一个涉及资金安全的核心链路，分为“前端提交”和“支付成功异步回调”两步。

```text
[用户发起储值] 
   │
   ├─> 1. 验证用户/卡/规则有效性，计算本金、赠送金额、赠送积分
   ├─> 2. 生成支付订单，创建【储值任务 (CrmDealTask)】
   │
[用户完成支付] ──> MQ消息触发异步回调
   │
   ├─> 3. 更新【会员卡余额】(本金、赠送金额分别记账)
   ├─> 4. 发放【积分】(更新卡积分，写变动流水)
   ├─> 5. 发放【优惠券/券包】(调用发券服务，标记渠道为"储值赠送")
   ├─> 6. 判定并执行【会员升级】(如达到门槛)
   └─> 7. 发送微信/短信通知，任务标记SUCCESS
```

### 2.2 发券/发券包全流程 (CrmCouponOpServicePlus)
无论是储值赠送、后台群发还是活动发券，底层都走这个统一的引擎。

```text
[触发发券动作]
   │
   ├─> 1. 前置校验：是否有手机号？券是否上架？是否在领取时间内？
   ├─> 2. 额度校验：会员每日限领次数？总限领次数？
   ├─> 3. 库存校验与扣减：(使用数据库行级锁+防超发SQL直接扣减)
   │
   ├─> 4. 判断类型：
   │     ├─ 普通券 -> 直接生成【发券记录 (CrmCouponOrder)】
   │     └─ 券包   -> 查出所有子券 -> 扣减子券库存 -> 批量生成子券的发券记录
   │
   └─> 5. 提交事务，发送领券通知
```

## 3. 三大关键架构设计 (研发必读)

本系统有三个极其巧妙且需要特别注意的设计。

### 💡 设计一：券和券包同表设计（大道至简）
**设计理念**：不单独建“券包表”，券包本质上就是一张 `isPkg = true` 的优惠券。
*   **关系结构**：
    *   主表 `crm_coupon` (包含普通券、券包)。
    *   关联表 `crm_coupon_map` (仅券包使用，记录“A券包”包含2张“B券”和1张“C券”)。
*   **好处**：审核、上下架、有效期管理、库存管理的代码完全复用。

### 💡 设计二：独立的双重库存扣减机制
券包的库存，和它里面包含的“子券”的库存是**完全独立**的。
*   **扣减公式**：发放1个券包A（含2张子券B）
    *   券包A库存：-1
    *   子券B库存：-2
*   **木桶效应**：券包本身没库存了，或者**任意一张**子券没库存了，都会导致整个券包发放失败，事务回滚。

### 💡 设计三：库存防超发的原子操作
绝不在内存中计算库存后写入，而是直接委托给数据库的原子能力，防止高并发超发：
```sql
-- 核心扣减SQL，注意 WHERE 里的库存保护条件
UPDATE crm_coupon
SET received_number = IFNULL(received_number, limit_number) - {发放数量}
WHERE lid = {券ID} 
  AND status_ = 1
  -- 关键防线：扣减后必须 >= 0
  AND IFNULL(received_number, limit_number) - {发放数量} >= 0 
```

## 4. 数据库核心表关系图

系统采用领域驱动设计的思想，表结构分为三个主要领域：

### 4.1 会员领域 (Member Domain)
*   `crm_member` (会员档案)：存姓名、手机号、生日等稳定的人员信息。
*   `crm_card` (会员卡)：**最核心的业务表**。存余额、积分、等级，所有的交易流水都挂在这个ID下。一个Member可以有多张Card。
*   `crm_card_type` / `crm_card_level`：会员卡类型与等级体系。
*   `crm_card_op_record` / `crm_card_points_record`：资金与积分的变动流水表。

### 4.2 营销/发券领域 (Coupon Domain)
*   `crm_coupon` (券模板表)：定义这是什么券（或券包），面值多少，总库存多少。
*   `crm_coupon_map` (券包配置表)：记录券包与子券的包含关系。
*   `crm_coupon_order` (券实例/领券记录表)：**用户钱包里真正看到的券**。记录哪张会员卡，领了哪张券，何时到期，是否已核销。

### 4.3 储值/交易领域 (Recharge Domain)
*   `crm_deal_task` (储值任务主表)：记录一笔储值订单的金额、支付状态。
*   `crm_deal_task_item` (储值任务明细)：记录这笔订单应该赠送的券和积分明细。
*   `crm_deposit_plan` / `crm_save_rule`：定义怎么充、送什么。

## 5. 常见问题排查指南 (QA & 运维必读)

当发生业务客诉时，按以下逻辑排查：

**Q1：用户付了钱，但没收到储值的钱？**
*   排查 `crm_deal_task` 状态是否停留在 `USERPAYING`。如果是，说明支付回调丢失，需排查支付网关/MQ。

**Q2：储值成功了，钱到了，但没发优惠券？**
1.  **查用户**：该会员绑定手机号了吗？（系统硬性规定：发券必须有手机号）。
2.  **查规则**：旧版 `crm_save_rule` 里确实配置了送券吗？
3.  **查库存**：`crm_coupon` 表里的 `receivedNumber` 是不是 <= 0 了？
4.  **查限制**：是不是用户当天领券次数/总次数达到了上限？
5.  **查日志**：查看 `CardBalanceService` 异步发券时的Exception日志。

**Q3：用户领取券包失败？**
*   只要有一张子券的库存不足以满足本次发放（发券包数量 × 子券配置数量），或者券包本身没库存，都会直接抛出异常导致失败。去 `crm_coupon_map` 顺藤摸瓜查所有子券库存。

**Q4：为什么同一个优惠券，有的用户马上能用，有的明天才能用？**
*   检查券的有效期模式。如果是“固定天数模式 (GDTS)”，系统支持配置 `beginValidDays`（领券后第N天生效）。

## 6. 后续演进路线建议

1.  **架构融合**：尽早完成 `ChargeService` 从旧版 `CrmSaveRule` 切换到新版 `CrmDepositPlan`，消灭两套规则并行的技术债。
2.  **性能优化**：如果是群发券包场景，当前针对每一张子券执行 `INSERT` 会导致数据库压力较大，建议升级为 `MyBatis Plus` 的批量插入机制。
3.  **并发升级**：应对大促秒杀储值，当前依赖数据库行锁防超发，未来可引入 Redis Lua脚本 或 Redisson 分布式锁 提前拦截库存校验请求，保护数据库。

# 会员储值与营销资产系统：高维业务与架构抽象

整个系统可以抽象为三大核心领域：**账户与身份领域**（谁在交易）、**商品与规则领域**（卖什么、怎么卖）、**资产与履约领域**（交易达成后下发什么）。

## 一、 账户与身份领域（Identity & Account Domain）

系统在底层设计上，将“人”与“资金载体”做了严格的解耦，这是理解所有业务的前置条件。

### 1. “人”与“账户”的解耦（Member vs Card）
*   **自然人（CrmMember）**：代表物理世界的一个人，拥有唯一属性（姓名、手机号、身份证、微信号等）。属于**静态档案数据**。
*   **资金账户/业务载体（CrmCard）**：系统所有资金变动、营销动作的**唯一锚点**。一个自然人可以开通多张卡（多账户）。储值、消费、领券、升降级，本质上都是在操作这张“卡”，而不是操作这个“人”。
*   **性能考量（适度冗余）**：为了避免高频交易时频繁连表，`CrmCard` 表适度冗余了 `CrmMember` 的基础信息（如手机号、姓名），兼顾了规范与性能。

### 2. 会员体系的“三维立面”
会员等级不是扁平的，而是**“体系(Type) -> 级别(Level) -> 实例(Card)”**的三层结构：
*   **体系（CardType）**：定义了卡的物种（如：普通储值卡、付费权益卡）。
*   **级别（CardLevel）**：依附于体系，定义了升降级规则（按消费额、次数、储值额等）和对应权益（折扣、积分翻倍）。
*   **动态演进**：用户的升级降级，只是在改变 `CardLevel`；而用户换卡/购买新体系，则是改变 `CardType`。

---

## 二、 商品与规则领域（Product & Rule Domain）

“储值”在业务本质上，是向用户售卖一个**“资金+权益的资产包”**。系统正经历从“硬编码规则”向“高度配置化方案”的演进。

### 1. 储值方案（DepositPlan）的积木式封装
新版 `CrmDepositPlan` 使用 JSON 实现了极度灵活的规则引擎。一个储值方案包含了：
*   **准入规则**：谁能买（指定会员等级）、在哪买（指定门店/品牌/渠道）、什么时候能买（时间段、排除节假日）。
*   **限购规则**：买多少（单人周期限购、总库存限购）。
*   **阶梯定价（TierRule）**：核心的 SKU 列表。充100送20+送A券；充500送100+送券包+直接升级为金卡。
*   **诱饵机制**：支持“首充特定奖励”、“可选券（用户N选1）”、“霸王餐储值（特殊账户体系）”。

### 2. 营销资产的“组合模式”设计（普通券 vs 券包）
系统对“优惠券”和“券包”的设计极为精妙，采用了软件工程中的**组合模式（Composite Pattern）**：
*   **同源同表**：系统中不存在独立的券包表。券包（Package）本质上也是一张优惠券记录（存放在 `crm_coupon`），仅仅通过 `isPkg = true` 标识。
*   **树形展开**：通过映射表（`crm_coupon_map`），一个券包可以挂载多张子券。
*   **多态复用**：因为同表，券包自动继承了普通券的所有生命周期管理能力（上下架、库存限制、领券次数限制、有效期校验等），极大降低了系统复杂度。

---

## 三、 资产履约与流转领域（Fulfillment & Asset Pipeline）

这是系统的核心引擎。当用户发起一次储值并支付成功后，系统需要保证多种资产（资金、积分、券）的**一致性、安全性和幂等性**。

### 1. 储值履约的标准SOP
储值并不是简单的“改个余额”，而是一个严密的异步流：
1.  **契约锁定（主任务）**：生成待支付订单和储值任务（`CrmDealTask`）。
2.  **资金清算（分账记账）**：支付成功后，本金与赠送金额**分离记账**（`principal_balance` / `give_balance`），这是为了后续退款和财务合规做铺垫。
3.  **资产瀑布式下发**：
    *   **下发积分**：写入积分变更流水，更新卡积分。
    *   **下发实物/虚拟营销资产**：调用券引擎，下发优惠券或券包，并打上来源标记（`CouponChannel=Recharge`）。
    *   **身份跃迁**：检测是否满足当前规则配置的升级门槛，或是否包含“升级工本费”，触发会员等级的变更并记录流水。

### 2. 营销引擎（发券引擎）的安全护城河
发券（`CrmCouponOpServicePlus`）是高危操作，系统在此构筑了三道防线：
*   **防线一：准入校验（内存态）**。券是否生效？用户是否有手机号（强制风控要求）？用户每日/总计领券次数是否超标？
*   **防线二：双层库存独立计算（逻辑态）**。
    *   券包和其包含的子券，库存是**完全独立**管理的。
    *   发一个券包，既要扣券包本身的库存，又要遍历扣减所有子券的库存。符合**“木桶效应”**：只要有一张子券没库存，整个券包的发放事务就会彻底回滚。
*   **防线三：原子防超发（数据库态）**。绝对不在代码层做 `库存 = 库存 - 1` 的计算，而是下沉到 SQL 层，利用数据库行级排他锁和 `WHERE 余量 - 扣减量 >= 0` 进行兜底，确保大并发下的绝对安全。

### 3. 资产实例的具象化（从 Template 到 Order）
*   `CrmCoupon` 是**资产模板（Template）**，定义了券的面值和规则（“商品”）。
*   `CrmCouponOrder` 是**资产实例（Instance）**，是发放到用户账户里的真实资产（“订单”）。
*   实例的有效期分为“绝对时间（固定日期）”和“相对时间（领取后N天生效，存活M天）”，在具象化（生成 Order）的那一刻，系统会根据时间引擎计算出这个实例的最终消亡时间（`deadline`）。

## 四、 架构演进与技术债（现状洞察）

从这份文档可以看出，该系统是一套**演进中的系统**，新老架构正在交替：

1.  **规则引擎的升级（过渡期）**：
    *   **Old（`CrmSaveRule`）**：传统的关系型表结构，规则固化在数据库列中（如 `give_point`, `give_coupon_id`），灵活性差，目前仍在跑主流程。
    *   **New（`CrmDepositPlan`）**：全面拥抱 JSON，将限次、限店、时段等复杂逻辑全部 JSON 化，具备极高的扩展性，但与主交易链路（ChargeService）的集成仍在待完成状态（Todo）。
2.  **重度依赖数据库事务**：
    *   当前的库存扣减、业务防超发、多表状态更新，强依赖数据库的 `@Transactional` 事务和行级锁。
    *   文档提及的长期优化建议中包含了“分布式锁”，说明系统目前在应对突发性大流量（如秒杀级发券/储值）时，数据库可能存在瓶颈，未来需要引入 Redis 拦截前置流量。
