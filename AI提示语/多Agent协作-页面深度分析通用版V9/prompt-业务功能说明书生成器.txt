## 一、功能全景

### 1.1 这个功能解决什么问题

用一段话说清楚：
- 谁在用这个功能（角色）
- 解决什么业务痛点
- 期望达到什么效果

### 1.2 功能定位图

用一句话定位这个功能在整个系统中的位置：
- 属于哪个业务模块
- 与哪些上下游功能关联
- 在业务流程中处于什么环节

### 1.3 一句话概括

> 用一句话概括这个功能的核心价值，格式：「让{角色}能够{做什么}，从而{达到什么效果}」

## 二、用户操作流程

### 2.1 主流程（Happy Path）

用编号步骤描述用户从进入页面到完成操作的完整路径：

```
步骤1: {用户做什么} → {系统反馈什么}
步骤2: {用户做什么} → {系统反馈什么}
...
最终: {用户点击保存} → {系统完成什么}
```

★ 每个步骤用用户能理解的动词：填写、选择、点击、确认、查看
★ 不用"提交请求体""调用API"等技术语言

### 2.2 分支流程

列出主流程中的关键分岔点：

```
在步骤N，如果用户选择了「{选项A}」：
  → 接下来会出现{什么}，需要额外{做什么}

在步骤N，如果用户选择了「{选项B}」：
  → {什么}会被隐藏，直接进入下一步
```

### 2.3 编辑 vs 新增的差异

用对比表格说明编辑模式与新增模式的区别：
- 哪些信息可以修改
- 哪些信息创建后不可更改
- 修改时的特殊行为（如明细的增删改）

## 三、信息结构（用业务语言）

### 3.1 这个功能管理哪些信息

不说"实体""表""字段"，而是用业务语言描述：

```
{功能名}包含以下信息：

■ 基本信息
  - 活动名称（必填，最多100个字）
  - 活动日期（两种模式：指定时间段 或 永久有效）
  - ...

■ {子信息名称}（可添加多条，最多N条）
  每条包含：
  - ...

■ 关联信息
  - 适用品牌（可多选）
  - ...
```

### 3.2 信息之间的关系

用业务语言解释信息的层级关系：

> 一个「储值套餐」可以包含多个「储值档位」，每个档位可以配置不同的「赠送礼品」。
> 套餐是主体，档位是套餐的组成部分，礼品挂在档位下面。
> 删除套餐时，其下的所有档位和礼品一并删除。

### 3.3 引用信息 vs 自有信息

解释哪些信息是"自己创建的"，哪些是"从其他地方选来的"：

> 「适用品牌」不是在本页面创建的，而是从品牌库中选取的。
> 即使品牌信息后来变了（比如改了名字），这里显示的也会自动更新。
> （如果是快照场景）这里记录的是选择时的品牌信息，即使后来品牌改名了，这里仍显示当时的名称。

## 四、业务规则说明

### 4.1 智能联动（用户可感知的）

用"当...时，系统会..."的格式描述所有联动规则：

```
规则1: 当选择「指定日期」时，下方会出现开始日期和结束日期输入框
规则2: 当选择「永久有效」时，日期输入框消失，表示不限时间
规则3: ...
```

★ 只描述用户可感知的界面变化，不说"前端渲染""后端校验"

### 4.2 自动计算

描述系统自动算出的值：

```
- 「合计金额」= 所有档位的储值金额之和（系统自动计算，不可手动修改）
- 「档位数」= 当前已添加的档位条数
```

### 4.3 填写约束（什么必须填、怎么填才对）

用表格说明约束，语言面向用户：

| 信息项 | 是否必填 | 填写要求 | 不满足时提示 |
|--------|---------|---------|-------------|
| 活动名称 | 必填 | 最多100个字 | "请输入活动名称" |
| 储值金额 | 必填 | 大于0的金额 | "储值金额必须大于0" |
| 开始日期 | 选了指定日期时必填 | 不能晚于结束日期 | "开始日期不能晚于结束日期" |

### 4.4 业务限制

用自然语言说明不可违反的业务规则：

```
- 至少要有1个档位，否则无法保存
- 同一套餐下的档位金额不能重复
- 启用状态的套餐不能删除，需要先停用
```

## 七、设计理念提炼

★ 这是本文档最有价值的部分——从具体功能设计中提炼出一般性的设计原则。

### 7.1 本功能体现的设计理念

从需求文档的设计决策中，提炼出3-5条设计理念，格式：

```
理念1: {一句话概括}
  体现在: {具体设计点}
  为什么: {背后的思考}
  一般化: {这条理念可以推广到什么场景}
```

示例：

```
理念1: 渐进式披露 — 只展示当前需要的信息
  体现在: 选择「永久有效」时隐藏日期选择器；选择「不赠送」时隐藏礼品配置区
  为什么: 减少用户的认知负担，避免用不相关的选项干扰判断
  一般化: 任何表单设计，当某些字段仅在特定条件下有意义时，应默认隐藏

理念2: 主从聚合 — 相关数据作为一个整体保存
  体现在: 套餐和档位一起保存，不需要分两次操作
  为什么: 保证数据一致性（不会出现"套餐存了但档位没存"的中间状态）
  一般化: 业务上属于同一操作的数据，技术上应在同一事务中处理

理念3: 引用而非复制 — 选择已有数据而不是重新录入
  体现在: 品牌、优惠券从已有数据中选择，而不是重新填写
  为什么: 保证数据一致性（品牌改名后，所有引用自动更新）
  一般化: 当多个功能需要同一份数据时，建立引用关系而非各存一份副本
```

### 7.2 设计哲学总结

从所有理念中提炼出更高层次的设计哲学（1-3条）：

```
哲学: {一句话}
解释: {2-3句话展开}
```

示例：

```
哲学: 让简单的事情简单，让复杂的事情可能
解释: 基础功能（如填写活动名称）不需要任何额外操作。
     高级功能（如配置多档位赠送礼品）通过渐进式展开实现——
     只有需要的用户才会看到复杂的选项，其他人不受干扰。
```

### 7.3 设计模式识别

识别本功能中使用的通用设计模式，帮助读者理解"这不是随意设计的"：

| 设计模式 | 在本功能中的体现 | 解决什么问题 |
|---------|----------------|-------------|
| 主从表单 | 套餐-档位-礼品三级结构 | 一对多关系的数据录入 |
| 条件展示 | 按日期类型显隐字段 | 减少不必要的输入 |
| 引用选择 | 品牌/优惠券从库中选取 | 避免数据冗余 |
| 整体保存 | 主表+子表一次保存 | 保证数据一致性 |
| 差异更新 | 编辑时只改变化的部分 | 减少数据冲突 |
| ... | ... | ... |

1. 禁止使用以下技术术语（用括号中的替代词）：
