## 架构哲学（所有规则的源头，未覆盖场景回到此处判断）

P-1 单向依赖，禁止逆流
  Controller → Service → Mapper/Repository，下层不反向依赖上层。Service间可互调但禁止循环依赖。Controller不出现Mapper；Entity不出现DTO/VO。

P-2 契约驱动，代码即文档
  前后端唯一契约=DTO/VO结构+@Schema注解。写完代码=写完文档=Apifox可直接导入。联调零口头沟通。

P-3 显式优于隐式
  枚举值写明含义；条件校验写在代码里；类型转换在Convert层；错误提示是完整中文句子。

P-4 变更局部化
  改一个业务规则只改一个文件。枚举/校验/转换逻辑集中。禁止同一逻辑散布三层。

P-5 防御性编码，快速失败
  入口@Valid+JSR-303拦截非法数据→Service校验业务规则→数据库约束是最后防线非唯一防线。

P-6 安全零信任
  不信任前端任何数据（ID可能篡改、枚举可能越界、身份可能伪造）。每层假设上层可能失守。数据归属必须校验。

P-7 可演进性
  枚举可加值不改旧值、接口可加字段不删旧字段、表可加列不改旧列。向后兼容为默认，破坏兼容需显式版本号。

P-8 多租户隔离
  每张业务表强制包含 mid（商户ID）+ sid（门店ID），数据访问层自动拼接租户条件。跨租户数据不可见、不可改。

P-9 枚举闭环落地
  一旦识别出业务字段属于固定选项，必须在 DTO、VO、Entity、Service 判断、接口文档五处一致落地，禁止只定义枚举类而调用链继续使用裸 `Integer`。

## 项目结构

项目采用多模块 Maven 架构，分为三个主要层次：

### 1. nms4cloud-starter（基础设施层）
通用 starter 模块，提供可复用的基础能力：
- `nms4cloud-starter-parent`: 父依赖（Web, AOP, Validation, Actuator, Sa-Token, Lombok, Fastjson2, Hutool, Forest, MapStruct 等）
- `nms4cloud-starter-redis`: Redis 集成
- `nms4cloud-starter-mybatis-flex`: MyBatis-Flex 集成
- `nms4cloud-starter-oss`: 对象存储
- `nms4cloud-starter-rocketmq`: RocketMQ 集成
- `nms4cloud-starter-mongodb`: MongoDB 集成
- `nms4cloud-starter-xxjob`: XXL-Job 定时任务
- `nms4cloud-starter-cloud`: Spring Cloud 集成
- 其他：mqtt, pdf, shardingsphere 等

### 2. nms4cloud-app（应用层）
业务应用模块，按业务域划分：

| 层级 | 服务 | 说明 |
|------|------|------|
| 1_platform | nms4cloud-platform, nms4cloud-netty, nms4cloud-mq, nms4cloud-reg, nms4cloud-wechat | 平台核心 |
| 2_business | nms4cloud-biz, nms4cloud-crm, nms4cloud-mall, nms4cloud-payment, nms4cloud-pos, nms4cloud-product, nms4cloud-scm | 业务服务 |
| 3_customer | nms4cloud-order | 客户侧服务 |

### 服务模块四层结构（每个服务统一遵循）
```
nms4cloud-{service}/
  {service}-api/       ← 对外接口定义（DTO, VO, Feign 接口）
  {service}-dao/       ← 数据访问层（Entity, Mapper）
  {service}-service/   ← 业务逻辑层（Service 实现）
  {service}-app/       ← 应用启动层（Controller, 启动类, 配置文件）
```

★ 跨服务调用通过 api 模块的 Feign 接口，不跨服务访问 Mapper。

## 分层职责（严格执行，不可越层）

Controller层：接收请求→参数校验(@Valid)→调用Service→包装响应。禁止写业务逻辑、直接调Mapper、做数据转换。Controller是薄的"接线员"。
  ↓ 只传DTO，不传Entity/VO
Service层：业务逻辑、事务管理、条件校验、调用Mapper/其他Service。所有业务规则唯一归属地。返回Entity或内部对象，不返回VO。
  ↓ 只操作Entity
Mapper层：数据访问，SQL执行。禁止写业务逻辑、抛业务异常。纯数据通道。

Convert层（横切）：DTO↔Entity↔VO对象转换（MapStruct）。★ 枚举code→label转换在此层统一处理。禁止BeanUtils.copyProperties（无编译期检查）。

## 包结构

```
com.nms4cloud.{module}
  common/              ← 全局基础设施
    result/            ← NmsResult、PageResult
    exception/         ← BusinessException、GlobalExceptionHandler
    config/            ← 全局配置（Jackson、MyBatis-Flex、Swagger）
    util/              ← 工具类（严格限制数量）
  module/
    {业务模块}/         ← 如 product、order、member
      controller/ service/impl/ mapper/ entity/ dto/ vo/ convert/ enums/
  Application.java
```

★ 每个业务模块内部自治，模块间通过Service接口通信，不跨模块访问Mapper。

## 枚举建模规范（平台统一）

### 必须使用枚举的字段类型

- 状态类字段
- 开关类字段
- 模式类字段
- 范围类字段
- 类型类字段

### 不应使用枚举的字段类型

- 主键、逻辑编号、租户编号、门店编号等 ID 字段
- 金额、比例、数量、次数、天数、倍率等数值字段
- 名称、备注、说明、日期字符串、逗号分隔字符串等文本字段
- 列表、明细、时间段等集合字段

### 代码落地要求

- DTO / VO / Entity 对同一业务枚举字段必须保持类型一致
- MyBatis-Flex / MyBatis-Plus 均应以枚举 `code` 持久化，不存枚举名
- Service / Convert 中禁止使用魔法值 `1/2/3` 直接判断业务语义
- 枚举类和枚举值都必须补中文注释

### 前端协议约定

- 接口传参统一传枚举 `code`
- 响应默认返回枚举 `code`
- 不以枚举常量名作为接口协议

### 文档约定

- OpenAPI / Apifox 中必须写明“该字段使用哪个枚举的编码”
- 同时列出编码值与中文含义，避免只写数字不写语义

### 评审关注点

- 是否存在“枚举类已定义，但 DTO/Entity/Service 未接入”的半完成状态
- 是否残留魔法值判断
- 是否只改后端代码而未同步接口文档

## 主键与ID策略（pid / mid / sid / lid 四ID体系）

nms4cloud 采用四种ID，每张表必须全部包含：

| ID | 类型 | 生成方式 | 用途 |
|----|------|---------|------|
| pid | BIGINT AUTO_INCREMENT | 数据库自增 | 物理主键，仅用于内部索引，不暴露给前端 |
| mid | BIGINT | 登录上下文注入 | 商户ID，多租户隔离 |
| sid | BIGINT | 登录上下文注入 | 门店ID，门店数据隔离 |
| lid | BIGINT UNIQUE | Yitter IdGenerator 雪花算法 | 逻辑编号，全局唯一，作为业务层面的主键 |

★ 关键规则：
- 前端看到的"ID"一律是 lid，不是 pid
- 表间外键关联必须使用 lid，禁止使用 pid
- lid 必须同时在字段定义标记 UNIQUE + 建立 UNIQUE INDEX uk_lid (lid)
- mid/sid 必须建立普通索引 INDEX idx_mid (mid), INDEX idx_sid (sid)

Entity 对应注解：
```java
@Id(keyType = KeyType.Auto)
private Long pid;

private Long mid;
private Long sid;

// lid 由 Yitter IdGenerator 生成，插入前赋值
private Long lid;
```

## 审计字段模板

每张业务表必须包含以下审计字段：

```sql
created_by VARCHAR(100) COMMENT '创建人',
created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
updated_by VARCHAR(100) COMMENT '更新人',
updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除(0-否,1-是)',
revision INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
```

对应Entity字段（MyBatis-Flex）：
```java
private String createdBy;        // 创建人，插入时从上下文注入
private LocalDateTime createdTime;  // @Column(onInsertValue = "now()")
private String updatedBy;        // 更新人，更新时从上下文注入
private LocalDateTime updatedTime;  // @Column(onInsertValue = "now()", onUpdateValue = "now()")
@Column(isLogicDelete = true)
private Integer deleted;         // 逻辑删除
@Column(version = true)
private Integer revision;        // 乐观锁版本号
```

★ 与旧规范的区别：
- created_by / updated_by 为 VARCHAR(100)（非 BIGINT），存储用户名
- 强制 revision（乐观锁），每张表必须有
- 无 create_by（用 created_by），无 update_time（用 updated_time）

## 多租户规范

### 数据隔离
- 每张业务表必须包含 mid（商户ID）和 sid（门店ID）
- 查询时自动拼接 mid/sid 条件（通过拦截器或 MyBatis-Flex 的多租户插件）
- 写入时自动从登录上下文注入 mid/sid

### 安全规则
- UPDATE/DELETE 操作必须先查数据 → 校验 mid/sid 归属 → 再操作
- 跨租户数据不可见、不可改
- 禁止通过 SQL 跳过租户隔离（如手动拼接不带 mid 的查询）

### Entity 示例
```java
@Table("product")
public class Product {
    @Id(keyType = KeyType.Auto)
    private Long pid;
    private Long mid;      // 商户ID
    private Long sid;      // 门店ID
    private Long lid;      // 逻辑编号(雪花算法)

    // 业务字段...

    private String createdBy;
    private LocalDateTime createdTime;
    private String updatedBy;
    private LocalDateTime updatedTime;
    @Column(isLogicDelete = true)
    private Integer deleted;
    @Column(version = true)
    private Integer revision;
}
```

## 物理模型优化规则

★ 从逻辑模型到物理模型的优化策略，用于「物理模型决策」步骤。
核心问题：每张表=Entity+Mapper+Service+Controller全套，表过多导致启动慢、事务复杂、维护成本高。目标：每功能模块3-5张表。

合并为JSON的条件（满足任一即可合并）：
- 纯ID列表（多选关联ID）→ brand_ids JSON [1,2,3]
- 只有1-2个业务字段的子记录 → periods JSON [{"start":"09:00","end":"12:00"}]
- 简单值列表（日期、编码等）→ exclude_dates JSON ["2026-04-05"]
- 数量有限(<20)且不需独立查询
- 只随主表读写，不被其他模块引用

必须独立建表的条件（满足任一即不可合并）：
- 子记录有3+个业务字段（不含pid/FK/审计字段）
- 需按子记录字段做条件查询或报表统计
- 子记录有独立状态或生命周期
- 子记录数量可能很大(100+)
- 子记录被其他模块通过FK引用

实现要点：
- JSON字段用MySQL JSON类型
- MyBatis-Flex 用 @Column(typeHandler = JacksonTypeHandler.class) 映射
- DTO/VO结构不变（前端仍提交嵌套JSON），只是存储层合并
- 合并的子数据不需DELETE+INSERT，直接更新父表JSON列
- 逻辑模型实体关系仍成立，只是物理存储方式变了

## 契约规范（前后端零沟通成本）

★ 后端写完→Apifox导入→前端直接开发，无需口头沟通。

统一响应：所有接口返回 NmsResult.success(data) / NmsResult.fail(ErrorCode.XXX)
```json
{"code": 0, "message": "操作成功", "data": { ... }}
```

DTO/VO分离（入参出参严格分开）：
- DTO接收请求：XxxAddDTO / XxxUpdateDTO / XxxQueryDTO
- VO返回前端：XxxListVO / XxxVO
- 禁止：Entity直接暴露给前端；同一类既做入参又做出参

枚举值一致性（★ DTO与VO必须对称）：
- DTO入参枚举字段：Integer类型，字段名xxxCode（如dateTypeCode）
- VO出参枚举字段：同时返回xxxCode(Integer)+xxxLabel(String)
- ★ DTO的xxxCode与VO的xxxCode必须完全同名
- 嵌套DTO/VO中枚举字段递归遵循同样规则
- code→label转换统一在Convert层

字段序列化：
| 字段类型 | 处理方式 | 原因 |
|---------|---------|------|
| Long ID | @JsonSerialize(using = ToStringSerializer.class) | JS精度丢失 |
| BigDecimal | @JsonSerialize(using = ToStringSerializer.class) | JS精度丢失 |
| LocalDateTime | @JsonFormat(pattern="yyyy-MM-dd HH:mm:ss", timezone="GMT+8") | 格式统一 |
| 枚举 | xxxCode(Integer) + xxxLabel(String) | 前端直接展示 |
| Boolean | isXxx命名 | 语义清晰 |

分页规范：
- 列表查询继承 PageReqDTO，响应用 PageResult<T> 包装
- PageReqDTO：pageNum(int,默认1)、pageSize(int,默认10,最大100)
- PageResult<T>：total(long)、list(List<T>)、pageNum(int)、pageSize(int)

API文档注解（不可省略）：
- Controller类：@Tag(name, description)
- Controller方法：@Operation(summary, description) + @ApiResponse
- DTO/VO类：@Schema(description)
- DTO/VO每个字段（无一例外）：@Schema(description, example, requiredMode)
- 枚举字段description必须列出所有值
- 嵌套对象说明约束（如"最多5个"）

错误码规范：
- 统一枚举管理，格式：模块编号(4位)+序号(3位)，如1001001
- 错误信息必须是完整中文句子
- 禁止硬编码错误信息字符串

## 枚举类型规范

### 数据库存储
枚举值在数据库中存储数字编码，多语言翻译由前端语言包或后端 i18n 资源文件负责：
```sql
-- ✅ 正确：数据库只存数字编码
status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0-禁用，1-启用'
```

### Java 枚举定义
所有枚举实现 CommonEnum 接口（如项目已定义），包含：
- code(int)、label(String)、构造器、getByCode、getLabelByCode
- 加@Schema+Javadoc
- Entity中枚举字段用Integer

```java
@Getter
@AllArgsConstructor
public enum ProductStatusEnum {
    DISABLED(0, "禁用"),
    ENABLED(1, "启用");

    private final int code;
    private final String label;

    public static ProductStatusEnum getByCode(int code) {
        return Arrays.stream(values())
            .filter(e -> e.code == code)
            .findFirst()
            .orElseThrow(() -> new IllegalArgumentException("未知的状态编码: " + code));
    }

    public static String getLabelByCode(int code) {
        return getByCode(code).label;
    }
}
```

### 最佳实践
```sql
-- ✅ 推荐：完整枚举值描述
tenant_type TINYINT NOT NULL COMMENT '租户类型：1-平台，2-服务商，3-商户'

-- ❌ 不推荐：只有一个枚举值
is_admin TINYINT COMMENT '是否管理员：1-是'

-- ❌ 不推荐：枚举值描述不完整
tenant_type TINYINT COMMENT '租户类型'

-- ❌ 不推荐：没有数字编码
status TINYINT COMMENT '状态：禁用、启用、暂停'
```

## 国际化（i18n）规范

### 核心原则
数据库中存储的是数据，不是展示文本。需要多语言展示的场景：

### 枚举的多语言
枚举的 label 展示由前端语言包负责，数据库和后端只存数字编码：
```
// 前端语言包
common.enum.status.enabled = '启用' / 'Enabled'
common.enum.status.disabled = '禁用' / 'Disabled'
```

### 需要多语言的文本字段

**方案一：多字段方案（推荐，字段少时使用）**
```sql
CREATE TABLE category (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '物理编号',
  lid BIGINT NOT NULL UNIQUE COMMENT '逻辑编号(雪花算法)',
  name VARCHAR(100) NOT NULL COMMENT '名称(中文)',
  name_en VARCHAR(100) COMMENT '名称(英文)'
) COMMENT='分类表';
```

**方案二：JSON 字段方案（字段多时使用）**
```sql
CREATE TABLE product (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '物理编号',
  lid BIGINT NOT NULL UNIQUE COMMENT '逻辑编号(雪花算法)',
  name_i18n JSON NOT NULL COMMENT '名称(多语言JSON: {"zh-CN":"中文名","en-US":"English Name"})',
  description_i18n JSON COMMENT '描述(多语言JSON)'
) COMMENT='商品表';
```

### 不需要多语言的字段
系统内部字段、编码字段、技术字段不需要多语言：
```sql
code VARCHAR(50) NOT NULL COMMENT '编码'           -- 系统编码，不展示给用户
created_by VARCHAR(100) COMMENT '创建人'            -- 内部记录
remark VARCHAR(500) COMMENT '备注'                  -- 管理员内部备注
```

## 代码质量规范

参数校验三道防线：
1. Controller入口（@Valid+JSR-303）：DTO加@NotNull/@NotBlank/@Size/@Min/@Max，message必须中文如@NotBlank(message="活动名称不能为空")
2. Service业务校验：条件必填(RULE规则)用if校验，冲突抛BusinessException(ErrorCode.XXX)
3. 数据库约束（最后防线）：NOT NULL、UNIQUE、外键

事务管理：
- 写操作：@Transactional(rollbackFor = Exception.class)
- 只读：@Transactional(readOnly = true)
- ★ @Transactional只加在Service实现类public方法上
- ★ 事务方法内禁止调外部HTTP/RPC
- ★ 大批量数据操作拆分小事务

数据访问（MyBatis-Flex）：
- 链式查询优先：QueryChain.of(mapper).where(...).list()
- 链式更新：UpdateChain.of(mapper).set(...).where(...).update()
- 批量插入 saveBatch()，批量更新 updateBatch()
- 删除子表用 QueryChain 构造条件删除
- ★ 禁止SELECT *，查列表用select(字段列表)
- ★ 分页必须用 MyBatis-Flex 的 Page 对象
- ★ 防N+1查询：子表数据用IN批量查

并发安全：
- 库存/配额/余额：Redis INCR + LUA脚本保证原子性
- 关键数据：revision 乐观锁（每张表已强制包含）或 SELECT FOR UPDATE
- 幂等接口：唯一requestId + Redis去重

异常处理：
- 业务异常：throw new BusinessException(ErrorCode.XXX)
- 系统异常：log.error()记完整堆栈后返回通用错误提示
- ★ 禁止空catch

日志规范：
- 类加@Slf4j，用{}占位符，禁止打印敏感信息
- 格式：log.info("[{操作名}] {关键参数}={},...", value)
- 必须打日志：Service写操作入口和完成、关键业务分支、状态变更（前后状态）、外部调用（调用前+后/异常）、异常处理（业务WARN+系统ERROR带堆栈）

注释规范：
- 必须有：类Javadoc、Entity对应表名、Service每个public方法Javadoc、枚举类用途
- 禁止：// getter setter、代码能自解释处不加注释

安全编码：
- 数据归属校验：UPDATE/DELETE先查数据→校验 mid/sid 归属
- 多租户：Entity含 mid/sid，查询自动拼接
- 敏感数据：VO返回脱敏格式
- 防注入：QueryChain/LambdaQuery 优先，自定义SQL用#{}
- 接口防刷：写接口加频率限制

逻辑删除查询陷阱：
- 唯一性校验考虑已删除数据：UNIQUE(name, deleted)或删除时改名
- JOIN子表需手动加AND sub.deleted = 0

领域事件：
- applicationEventPublisher.publishEvent(new XxxEvent(id))
- @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)

依赖注入：
- ★ 必须构造器注入(@RequiredArgsConstructor)，禁止@Autowired字段注入
- 依赖超5个时考虑拆分Service
- 禁止@Lazy掩盖循环依赖

嵌套DTO校验传递：
```java
@Valid
@NotEmpty(message = "至少需要一个档位")
@Size(max = 5, message = "档位最多5个")
private List<TierAddDTO> tiers;
```
★ 多层嵌套每层都要加@Valid

定时任务：多实例部署必须用分布式锁（Redisson），每次记录日志：开始+结束+处理数量。XXL-Job 作为任务调度平台。

接口幂等：
- 方案一：Token令牌（表单提交）— GET /api/idempotent/token + Header Idempotent-Token
- 方案二：业务唯一键 — UNIQUE约束 + catch DuplicateKeyException
- ★ 每个POST写接口必须选其中一种

CORS：前后端分离必须配置，生产环境allowedOriginPatterns配具体域名。

全局请求拦截：AccessLogInterceptor自动记录访问日志+慢接口告警(>3s)。

可演进性：
- 枚举：新增值加末尾，不改已有code
- 接口：新增可选字段，不删旧字段
- 数据库：加列给默认值，禁止删列/改列类型

## 各层生成规范

Entity层（MyBatis-Flex）：
- @Table("product")  （表名无 t_ 前缀）
- 主键：@Id(keyType = KeyType.Auto) private Long pid;
- 必含四ID：pid, mid, sid, lid
- 必含审计字段：createdBy, createdTime, updatedBy, updatedTime, deleted(@Column(isLogicDelete=true)), revision(@Column(version=true))
- lid 字段加 @JsonSerialize(using = ToStringSerializer.class)
- 加@Schema类注解+Javadoc
- ★ 每个[OWN]实体生成独立Entity
- ★ 子表Entity含父表 lid 作为外键字段
- ★ [REF]引用实体不生成Entity
- ★ 逻辑删除+唯一约束：UNIQUE(name, deleted)

DTO层：
- 每个类和字段加@Schema
- 枚举字段用Integer，字段名xxxCode
- 校验注解：BLOCK级→@NotNull/@NotBlank；范围→@Min/@Max/@Size
- message必须中文
- 条件必填在Service层校验
- ★ 嵌套结构按接口数据结构生成
- ★ UpdateDTO的明细列表元素含可选lid（用于区分新增/更新）

枚举层：
- ★ 按枚举值定义表为每个枚举字段生成独立枚举类
- 含：code(int)、label(String)、构造器、getByCode、getLabelByCode
- 加@Schema+Javadoc
- Entity中枚举字段用Integer

VO层：
- 每个类和字段加@Schema
- ★ 枚举字段同时返回xxxCode+xxxLabel
- xxxLabel加accessMode=READ_ONLY
- 详情VO嵌套子表VO的List
- XxxListVO 用于分页列表，XxxVO 用于详情

Convert层（MapStruct）：
- 每个[OWN]实体一个Convert接口
- ★ 枚举code→label转换在此层
- DTO→Entity、Entity→ListVO、Entity→VO各一个方法

Service层：
- 写操作@Transactional(rollbackFor = Exception.class)
- 查询@Transactional(readOnly = true)
- ★ 保存按事务边界：主表操作→子表整体替换→关联表整体替换
- ★ 后端业务层规则在此实现
- ★ UPDATE/DELETE先查数据→校验 mid/sid 归属→再操作
- ★ lid 在 create 方法中通过 IdGenerator 生成并赋值

Controller层：@Tag+@Operation+@ApiResponse，参数加@Valid

## 建表脚本模板（MySQL方言）

标准表模板（所有业务表必须遵循）：
```sql
CREATE TABLE {module}_{business_name} (
  pid BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '物理编号',
  mid BIGINT NOT NULL COMMENT '商户ID',
  sid BIGINT NOT NULL COMMENT '门店ID',
  lid BIGINT NOT NULL UNIQUE COMMENT '逻辑编号(雪花算法)',

  -- 业务字段...
  name VARCHAR(100) NOT NULL COMMENT '名称',
  status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0-禁用，1-启用',

  created_by VARCHAR(100) COMMENT '创建人',
  created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_by VARCHAR(100) COMMENT '更新人',
  updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除(0-否,1-是)',
  revision INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',

  INDEX idx_mid (mid),
  INDEX idx_sid (sid),
  UNIQUE INDEX uk_lid (lid)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='表说明';
```

通用SQL→MySQL翻译规则：
```sql
-- BIGINT PRIMARY KEY          → BIGINT AUTO_INCREMENT PRIMARY KEY
-- INTEGER NOT NULL DEFAULT 1  → TINYINT NOT NULL DEFAULT 1 （枚举字段用TINYINT）
-- INTEGER NOT NULL            → INT NOT NULL （普通整数用INT）
-- BOOLEAN NOT NULL DEFAULT FALSE → TINYINT NOT NULL DEFAULT 0
-- TIMESTAMP NOT NULL          → DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
-- 更新时间字段追加            → ON UPDATE CURRENT_TIMESTAMP
-- 字段注释                    → COMMENT '...'
-- 表注释                      → COMMENT='...'
-- 引擎                        → ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
```

索引命名规范：
| 索引类型 | 命名规则 | 示例 |
|----------|----------|------|
| 普通索引 | idx_{字段名} | idx_mid |
| 唯一索引 | uk_{字段名} | uk_lid |
| 复合索引 | idx_{字段1}_{字段2} | idx_mid_status |

## 数值与类型安全规范（AI生成代码必须遵守）

★ 本章节是 AI 代码生成的高频陷阱清单，生成每一段业务代码前必须自检。

### 一、金额/浮点数空值语义

**规则：前端传入的 BigDecimal 金额字段，null 和 0 均视为"未传值/无意义"。**

```java
// ✅ 正确：金额有效性判断
boolean hasAmount = amount != null && amount.compareTo(BigDecimal.ZERO) > 0;

// ❌ 错误：只判 null，0 会被误认为有效金额
boolean hasAmount = amount != null;

// ❌ 错误：不能用 > 运算符（BigDecimal 不支持）
boolean hasAmount = amount > 0;
```

适用场景：储值金额、优惠门槛、折扣金额、封顶金额等一切金额字段的条件判断。

### 二、BigDecimal 比较必须用 compareTo

```java
// ✅ 正确
a.compareTo(b) == 0   // 相等
a.compareTo(b) > 0    // a > b

// ❌ 错误：equals 在 scale 不同时返回 false（1.0 != 1.00）
a.equals(b);
```

### 三、BigDecimal 构造必须用字符串或 valueOf

```java
// ✅ 正确
new BigDecimal("0.1")
BigDecimal.valueOf(0.1)

// ❌ 错误：浮点精度丢失
new BigDecimal(0.1)
```

### 四、BigDecimal 除法必须指定精度和舍入模式

```java
// ✅ 正确
a.divide(b, 2, RoundingMode.HALF_UP)

// ❌ 错误：无限小数时抛 ArithmeticException
a.divide(b)
```

### 五、金额累加全程使用 BigDecimal

```java
// ✅ 正确
BigDecimal total = BigDecimal.ZERO;
for (Item item : items) {
    total = total.add(item.getPrice());
}

// ❌ 错误：double 累加有精度误差
double total = 0;
for (Item item : items) {
    total += item.getPrice().doubleValue();
}
```

### 六、负数金额必须拦截

```java
// ✅ DTO 层：注解拦截
@DecimalMin(value = "0.01", message = "金额必须大于0")
private BigDecimal amount;

// ✅ Service 层：业务校验
if (amount != null && amount.compareTo(BigDecimal.ZERO) <= 0) {
    throw new BusinessException(ErrorCode.INVALID_AMOUNT);
}
```

### 七、Long ID 序列化防精度丢失

JS Number 最大安全整数为 2^53，雪花算法生成的 Long 超出后截断。

```java
// ✅ VO 中所有 Long 类型 ID 必须加
@JsonSerialize(using = ToStringSerializer.class)
private Long id;
```

### 八、枚举 null 必须显式兜底

```java
// ✅ 正确：getByCode 返回 null 时抛异常
StatusEnum status = StatusEnum.getByCode(dto.getStatusCode());
if (status == null) {
    throw new BusinessException(ErrorCode.INVALID_PARAM, "无效的状态值: " + dto.getStatusCode());
}

// ❌ 错误：null 静默传递，后续 NPE
entity.setStatus(StatusEnum.getByCode(dto.getStatusCode()));
```

### 九、逻辑删除唯一索引必须含 deleted 列

```sql
-- ✅ 正确：软删除后同名记录可重新创建
UNIQUE KEY uk_name_deleted (name, deleted)

-- ❌ 错误：软删除后 name 仍被唯一索引占用
UNIQUE KEY uk_name (name)
```

### 十、字符串空值判断用 isNotBlank

```java
// ✅ 正确：同时覆盖 null、""、"   "
if (StringUtils.isNotBlank(dto.getName())) { ... }

// ❌ 错误：前端传 "" 或 "   " 时漏过校验
if (dto.getName() != null) { ... }
```

字符串入参在 Service 层统一 trim，或 DTO 用 @NotBlank 自动拒绝纯空格。

### 十一、字符串转 BigDecimal 必须 catch 异常

```java
// ✅ 正确
try {
    BigDecimal amount = new BigDecimal(str);
} catch (NumberFormatException e) {
    throw new BusinessException(ErrorCode.INVALID_PARAM, "金额格式错误");
}

// ❌ 错误：Double 中转损失精度
BigDecimal amount = BigDecimal.valueOf(Double.parseDouble(str));
```

### 十二、分页 pageSize 必须有上限

```java
// ✅ DTO 层
@Max(value = 100, message = "每页最多查询100条")
private Integer pageSize;

// ✅ Service 层兜底（防绕过）
if (pageSize > 100) pageSize = 100;
```

### 十三、批量操作 List 必须限制数量

```java
// ✅ DTO 层
@Size(max = 500, message = "批量操作最多500条")
@NotEmpty(message = "列表不能为空")
private List<Long> ids;

// ✅ Service 层：空集合直接返回
if (CollectionUtils.isEmpty(ids)) return Collections.emptyList();
```

### 十四、数组/集合入参空值语义

**规则：前端传入的数组/集合字段，null 和空集合（size=0）均视为"用户未输入"，语义等价。后端校验必须同时覆盖两种情况。**

与金额规则对称：金额字段 null/0 → 未传值；集合字段 null/[] → 未传值。

```java
// ✅ 正确：用 CollectionUtils.isEmpty 一次性覆盖 null 和 []
if (CollectionUtils.isEmpty(dto.getTagIds())) {
    throw new BusinessException(ErrorCode.INVALID_PARAM, "请至少选择一个标签");
}

// ❌ 错误：只判 null，前端传 [] 时漏过校验，后续 IN 查询报错或语义错误
if (dto.getTagIds() == null) {
    throw new BusinessException(...);
}

// ❌ 错误：直接 .size() 未判 null，前端传 null 时 NPE
if (dto.getTagIds().size() == 0) { ... }
```

适用场景：标签列表、门店列表、商品列表、权限列表等一切集合/数组类入参字段的**必填校验**和**条件筛选**。

### 十五、空集合 IN 查询必须前置判断

```java
// ✅ 正确：避免 WHERE id IN () 语法错误
if (CollectionUtils.isEmpty(ids)) {
    return Collections.emptyList();
}

// ❌ 错误：ids 为空时直接查询，SQL 报错或全表扫描
```

### 十五、列表查询禁止返回 null

```java
// ✅ 正确
return list != null ? list : Collections.emptyList();

// ❌ 错误：返回 null，调用方 NPE
```

### 十六、开始/结束日期必须校验顺序

```java
// ✅ Service 层必须校验
if (dto.getStartDate() != null && dto.getEndDate() != null
        && dto.getStartDate().isAfter(dto.getEndDate())) {
    throw new BusinessException(ErrorCode.INVALID_PARAM, "开始日期不能晚于结束日期");
}
```

### 十七、批量操作必须校验数据归属（防越权）

```java
// ✅ 正确：WHERE 条件带租户标识，只操作本租户数据
wrapper.in(Entity::getId, ids)
       .eq(Entity::getMid, currentMid)   // ★ 必须带租户条件
       .eq(Entity::getSid, currentSid);

// ❌ 错误：仅按 id 批量操作，可能越权访问其他租户数据
```

### 十八、对外接口禁止用自增 pid 做查询条件

```java
// ✅ 正确：对外只暴露雪花 ID
public Result getById(Long lid) {
    wrapper.eq(Entity::getLid, lid);
}

// ❌ 错误：pid 自增可遍历，暴露给前端存在安全风险
```

## 多值字段 JSON 数组化规范

### 适用场景

当一个字段需要存储"多选"业务值时（如多选场景、多选渠道、多选星期、多选日期），必须使用 JSON 数组存储，禁止使用逗号分隔字符串。

### 判断规则

| 情况 | 存储方式 |
|------|---------|
| 多选枚举值（场景、渠道、星期、日期等） | JSON 数字数组 `[1, 2, 3]` |
| 单选枚举值 | 枚举 code（TINYINT） |
| 逗号分隔字符串 | ❌ 禁止使用 |

### Java 落地要求

**Entity（MyBatis-Plus）：**
```java
// ✅ 正确
@TableName(value = "crm_xxx", autoResultMap = true)  // autoResultMap=true 必须
public class CrmXxx {
    @TableField(value = "scene_limit", typeHandler = JacksonTypeHandler.class)
    private List<Integer> sceneLimit;
}
```

**DTO / VO：**
```java
// ✅ 正确：前端传数字数组，@Schema 说明合法值
@Schema(title = "场景限制，传数字数组，合法值：1-堂食，2-外卖，3-自提，4-外带")

```

**Service 校验：**
```java
// ✅ 正确：数字范围校验
if (!CollectionUtils.isEmpty(sceneLimit)) {
    for (Integer scene : sceneLimit) {
        Assert.notNull(scene, "场景值不能为null");
        Assert.isTrue(scene >= 1 && scene <= 4, "场景值非法：" + scene);
    }
}
```

### 前后端协议约定

- 前端传参：`"sceneLimit": [1, 3]`（数字数组）
- 禁止传字符串枚举名：`"sceneLimit": "DINE_IN,DELIVERY"`
- 空值语义：`null` 和 `[]` 均表示"不限制"

### 数据库约定

## 互斥字段清空规范

### 核心原则

保存时，由主控字段决定哪些从属字段有意义。**无意义的从属字段必须在 Convert/Service 层清空（置 null），不应该校验它们，也不应该把前端传来的脏数据存入数据库。**

### 典型场景

```
// ❌ 错误：earningMode=1 时校验商品列表 → 报"商品不存在"
if (!CollectionUtils.isEmpty(earningSpecifiedProductLids)) {
    validateProducts(earningSpecifiedProductLids);
}

// ✅ 正确：earningMode != BY_PRODUCT 时直接清空，不进入校验
if (earningMode != BY_PRODUCT) {
    entity.setEarningProductScopeType(null);
    entity.setEarningSpecifiedProductLids(null);
}
```

### 实现位置

互斥字段清空逻辑放在 **Convert 层的 `toEntity()` 方法**中（或 Service 保存前），在字段映射之后、校验之前执行。

### 规则

1. 主控字段为关闭/不适用时，所有从属字段置 null
2. 校验方法只校验"有值时的合法性"，不校验互斥字段
3. 清空逻辑按层级递进：上层关闭则下层全部清空，不需要逐一判断

### 评审检查项

- 是否存在"主控字段关闭但从属字段仍被校验"的情况
- 数据库中互斥字段是否存的是 null
- 校验报错信息是否因互斥字段脏数据导致误报


## 接口文档规范

### 核心原则

接口文档（OpenAPI JSON + 接口逻辑 MD）必须与代码保持严格同步，做到"文档即契约"。

### 请求体规范

1. **全量字段**：请求体示例必须列出 DTO 中所有字段
2. **互斥字段说明**：有主控/从属关系的字段，必须在注释或 description 中说明互斥规则
   - 格式：`// 仅当 xxx=N 时必填；其他情况后端自动清空`
3. **枚举字段**：必须列出所有合法值及中文含义
4. **多值字段**：说明数组格式和合法值范围

### 响应体规范

1. **全量字段**：响应示例必须列出 VO 中所有字段
2. **条件说明**：每个字段必须说明在什么情况下有值、什么情况下为 null
   - 格式：`// 仅当 xxx=N 时有值；其他情况为 null`

### DTO 字段命名规范

**GetDTO / UpdateDTO / DelDTO** 中用于定位记录的 `lid` 字段，必须命名为 `{业务名}Lid`，禁止直接使用裸 `lid`：

| DTO 类型 | 规则 | 示例 |
|---------|------|------|
| `CrmXxxUpdateDTO` | `xxxLid`（必填，定位待修改记录） | `pointsRuleLid`、`depositPlanLid` |
| `CrmXxxDelDTO` | `xxxLid`（必填，定位待删除记录） | `couponLid`、`cardLid` |
| `CrmXxxGetDTO` | `xxxLid`（可选，精准查询用） | `pointsRuleLid`、`memberLid` |
| `CrmXxxAddDTO` | 保持 `lid`（新增时可选传入） | `lid` |
| `CrmXxxVO` | 保持 `lid`（响应字段），`@Schema` 注释说明业务含义 | `lid` |

**原因**：裸 `lid` 在 GetDTO/UpdateDTO/DelDTO 中语义不明确，前端开发者无法直接判断这个 `lid` 指的是哪个业务实体的编号。

### 多用例规范

同一接口有多种典型场景时，必须提供多个独立用例，每个用例对应一种场景，请求体只包含该场景下有意义的字段。
