## 复杂业务系统页面深度分析框架 V2.0

要指导 AI 进行"像素级"的页面分析，不能只让它"看图说话"，必须建立一套**结构化的解构框架**。
本框架经过实战提炼，覆盖从 UI 元素到领域模型、从业务规则到工程落地的完整分析链路。
将文末的 **Master Prompt** 直接发给任何 AI，即可产出专家级的业务分析报告。

# 复杂业务系统页面深度分析框架 V2.0（AI 指导手册）

## 第零步：前置识别 (Pre-Analysis Context) ⭐

> **目的：** 在分析开始前建立坐标系，避免脱离上下文的盲目推导。

**指令：** "在分析页面细节之前，先回答三个定位问题。"

*   **0-1 系统角色识别**
    *   **操作者是谁？** 运营人员 / 商家管理员 / 超级管理员 / 财务审计员
    *   **终态受益者是谁？** C 端消费者 / 门店收银员 / 财务人员
    *   **所属子系统？** 营销中心 / 会员中心 / 财务中心 / 供应链…

*   **0-2 表单形态识别**
    *   **交互模式：** 向导式多步骤（Wizard）/ 单页长表单 / 弹窗表单 / 行内编辑
    *   **保存机制：** 是否支持草稿（决定数据完整性约束的触发时机）
    *   **新增 vs 编辑：** 两种模式下字段约束可能不同，需分别定义

*   **0-3 多租户边界识别**
    *   页面首字段是否存在**租户标识符**（如：品牌、组织、门店）
    *   该字段是"过滤器"（只影响查询范围）还是"归属权声明"（决定数据写入权限）
    *   是否存在**跨租户操作**（如：品牌多选 → 意味着向多品牌写入数据，权限模型需特殊处理）

## 第一步：空间解构 (Functional Zoning)

**指令：** "请将页面划分为独立的职能区块，识别每个区块的业务目标，并分析区块间的信息流向。"

*   **1-1 区块职能分区**
    *   **区块定义：** 如"基础信息区（定义身份）"、"策略核心区（制定对价）"、"准入限制区（发布约束）"
    *   **业务逻辑链：** 识别页面如何引导用户从"定义是什么"→"配置给什么"→"限制给谁用"

*   **1-2 视觉层级作为业务信号** ⭐ 新增
    *   **缩进与分组：** 字段缩进在另一字段下方 = 父子条件依赖关系
        *   示例：协议选择器缩进在协议 Toggle 下方 → 说明 Toggle 开启时该字段才生效
    *   **颜色语义：** 绿色选中 = 当前激活状态；灰色 = 禁用条件未满足
    *   **标签措辞：** "推荐"vs"指定"vs"限制" = 业务设计意图的编码，反映产品价值判断

*   **1-3 负空间分析（页面刻意缺失的内容）** ⭐ 新增
    *   **缺失的操作按钮暗示的约束：**
        *   没有"立即生效"按钮 → 说明修改有审批流或延迟生效机制
        *   没有"复制活动"按钮 → 活动不支持克隆，每次须从头配置
    *   **缺失的字段暗示的系统假设：**
        *   没有"货币类型"字段 → 隐含单币种假设，未来国际化需改造
        *   没有"库存数量"字段 → 该业务品类不做库存管控

*   **1-4 页面导航结构分析** ⭐ 新增
    *   **面包屑路径 = 数据层级归属：** 首页 > 会员中心 > 储值活动 > 新增储值套餐 → 数据隶属于会员中心下的储值活动模块
    *   **锚点导航 vs 步骤向导：** 同页锚点 = 各区块强关联；步骤向导 = 各步骤可独立校验，数据分批提交
    *   **按钮语义区分：**"保存" = 草稿持久化；"发布" = 状态跃迁；"提交" = 触发审批流

## 第二步：原子模型提炼 (Atomic Domain Modeling)

**指令：** "基于页面字段，推导出底层的实体模型（ER）及其关联关系，建立字段级数据字典。"

*   **2-1 核心实体与值对象识别**
    *   **核心实体（Core Entities）：** 识别主表（如 `StoredValueActivity`）与从表（如 `StoredValueTier`），标注主从关系与数量约束（1:N max=5）
    *   **值对象（Value Objects）：** 不需要独立 ID、与实体同生命周期的嵌套结构
        *   示例："周期规则（`CycleRule`）"= `{cycle_type, time_start, time_end}` 合并为值对象
    *   **映射关系：** 标注 1:1 / 1:N / M:N，并注明关联的业务语义

*   **2-2 字段类型与存储策略推导**
    *   为每个 UI 元素建立字段数据字典：字段名 / 数据类型 / 存储形式 / 是否必填
    *   **多选字段的存储决策对比：**

        | 存储方式 | 优点 | 缺点 | 适用场景 |
        |---------|------|------|---------|
        | 位运算（bitmask） | 查询快，单字段存储 | 可读性差，上限受位数限制 | 枚举固定、选项 ≤ 32 个 |
        | JSON 数组 | 灵活，无上限 | 不可索引，无法做 JOIN | 选项动态、数量不定 |
        | 关联中间表 | 规范，可关联查询 | 有 JOIN 代价，写入复杂 | 需要反查、选项频繁变动 |

*   **2-3 跨区字段依赖图** ⭐ 新增
    *   识别在 A 区定义、被 B/C 区消费的**跨区驱动字段**，建立依赖拓扑：
        ```
        示例：
        Activity.cycle_type（A区定义）
            ├── 驱动 → B区：amount_limit_type 的重置周期窗口
            └── 驱动 → C区：periodic_sales_limit 的统计周期
        ```
    *   依赖方向决定后端校验的执行顺序：被依赖字段必须先于依赖字段完成校验

*   **2-4 判别字段识别（Discriminator Fields）** ⭐ 新增
    *   识别控制其他字段**显示/必填/计算逻辑**的"主控字段"
    *   判别字段必须显式存储，不可由其他字段推导，否则会导致数据一致性问题
        ```
        示例：date_type（指定日期 | 永久有效）= 判别字段
             → 决定 start_date / end_date 的必填性
             → 决定 excluded_dates 合法性校验的范围边界
        ```

*   **2-5 默认值语义分析** ⭐ 新增
    *   **默认全选（Opt-out 模型）：** 系统鼓励宽泛适用，运营需主动收窄范围
    *   **默认不选（Opt-in 模型）：** 系统要求运营主动声明，防止误操作扩大影响
    *   **默认值 = 产品团队对最佳实践的编码**，折射业务设计哲学：
        *   示例："协议 Toggle 默认开启" → 产品认为协议保护是必要的标准做法
        *   示例："推荐 默认选中" → 产品希望新创建的活动默认获得曝光

## 第三步：业务规则挖掘 (Business Rule Extraction)

**指令：** "针对每个输入元素，提取其背后的显式与隐式业务规则，建立完整的规则清单。"

*   **3-1 隐式文本挖掘（Implicit Text Mining）** ⭐ 新增
    *   **Tooltip 文字 = 操作后果的系统承诺**，必须转化为精确业务规则
        *   示例："储值当日限制时，冻结金额即日解冻" → 定义了资金冻结/解冻的精确时机
    *   **Placeholder = 字段的格式期望和合法示例数据**
    *   **帮助/警告文字 = 系统主动风控提示**
        *   示例："建议开启" → 不强制，但违反有业务风险；需在文档中标注该风险
    *   **按钮文字语义差异：**"保存"≠"发布"≠"提交"，每个对应不同的状态变更和副作用

*   **3-2 UI 条件渲染规则（UI State Machine）** ⭐ 新增
    *   系统地提取所有"当 X 变化时，Y 如何响应"的规则，形成完整 IF-THEN 映射表：
        ```
        WHEN agreement_enabled = ON
          THEN 协议选择器 → 显示 AND 必填

        WHEN channel_flags.includes('微信小程序') OR '支付宝小程序'
          THEN 适用小程序选择器 → 显示

        WHEN org_scope_type = 指定门店
          THEN 门店多选列表 → 显示 AND 必填（至少选 1 个）

        WHEN tier.upgrade_level_id IS NULL
          THEN 等级延期复选框 → 禁用（置灰）
        ```

*   **3-3 触发逻辑与计算逻辑（原有，强化）**
    *   **触发逻辑：** 某个开关打开后，哪些后续字段变为必填或联动变化
    *   **计算逻辑：** 字段间的数值依赖关系
        *   示例：赠送率 = `bonus_amount / deposit_amount × 100%`（派生字段，实时计算展示，不存储）
    *   **优先级逻辑：** 当多个规则冲突时，系统的判定优先顺序
        *   示例：排除日期与可用时段冲突时，排除日期优先（日维度 > 时维度）

*   **3-4 业务不变量（Business Invariants）** ⭐ 新增
    *   **不变量 ≠ 验证规则**：验证规则在提交时检查；不变量在系统任意时刻都必须成立，违反即视为数据损坏
    *   常见不变量模式：
        ```
        bonus_amount < deposit_amount            （防套利漏洞，任何时刻必须成立）
        tier.tier_no 在同活动内唯一              （档位标识不变量）
        已签署的协议版本内容不可修改              （法律义务不变量）
        activity.status 只能单向前进，不可逆回    （状态机不变量）
        ```

## 第四步：多维约束定义 (Multi-dimensional Constraints)

**指令：** "从技术与合规角度，分层定义该页面的输入约束与系统级校验。"

*   **约束分级体系** ⭐ 强化

    | 约束级别 | 说明 | 校验时机 | 示例 |
    |---------|------|---------|------|
    | 类型约束 | 数据类型、精度、长度 | 前端实时 | `DECIMAL(10,2)`、`VARCHAR(50)` |
    | 范围约束 | min/max、枚举值 | 前端实时 | `deposit_amount > 0` |
    | 必填约束 | 静态必填 vs 条件必填 | 提交时 | `agreement_enabled=ON` 时协议必填 |
    | 跨字段约束 | 多字段联合校验 | 提交时 | `end_date >= start_date` |
    | 唯一性约束 | 业务层面防重 | 提交时（异步） | 同品牌进行中活动名称不重复 |
    | 不变量约束 | 任意时刻必须成立 | 写入前（服务层） | `bonus_amount < deposit_amount` |
    | 并发约束 | 需要原子操作保证 | 运行时 | 限购计数器不超卖 |

## 第五步：外部系统联动分析 (Cross-System Integration)

**指令：** "识别该页面与后台其他子系统的交互依赖，区分同步/异步调用，建立完整的系统集成地图。"

*   **5-1 外部系统依赖地图（原有，强化）**
    *   为每个字段/操作标注：调用的下游服务 + 调用方式 + 失败处理策略
        ```
        保存活动   → 同步写入活动DB          失败 → 事务回滚，提示用户
        发放积分   → 异步消息队列（MQ）       失败 → 补偿队列重试，不阻断主流程
        发放优惠券 → 异步消息队列（MQ）       失败 → 补偿队列，需幂等设计
        会员等级升级→ 同步调用会员服务        失败 → 事务回滚，储值记录标记待处理
        ```

*   **5-2 读写模型分离（CQRS 视角）** ⭐ 新增
    *   **写模型（Admin Form）：** 面向运营配置，字段全、约束严、格式原始
    *   **读模型（Consumer Display）：** 面向消费者，字段精简、格式友好、多语言
        ```
        写模型字段：excluded_dates: ["2025-01-01","2025-02-14"]
        读模型展示：「春节及情人节不参与活动」（需反查日历名称映射）
        ```
    *   建立「写入 DTO → 领域模型 → 展示 VO」的三层映射关系

*   **5-3 数据生命周期与级联规则** ⭐ 新增
    *   父实体变更/删除时，子实体的处理策略：

        | 操作 | 影响判断 | 建议策略 |
        |------|---------|---------|
        | 删除协议 | 有活动正在引用 | 禁止删除（硬约束），提示引用活动数量 |
        | 删除档位 | 用户已购买该档位 | 软删除，历史记录保留，不影响已购权益 |
        | 修改活动日期 | 活动进行中 | 需二次确认，修改后推送通知给已购用户 |
        | 禁用会员等级 | 有档位配置了该等级升级 | 警告提示，不阻断禁用，但档位升级逻辑降级处理 |

*   **5-4 内联创建的 API 模式** ⭐ 新增
    *   页面中的"+ 新增分类"、"+ 添加储值协议"等内联按钮，隐含特定 API 交互模式：
        ```
        POST /categories  → 返回 {id, name} → 前端立即回填并选中
        ```
    *   内联创建失败不影响主表单，弹出轻提示后用户可重试

## 第六步：异常处理与风控建模 (Exception & Risk Modeling)

**指令：** "提炼该页面涉及的风控点、并发争用点及异常处理流程。"

*   **6-1 反作弊约束（原有，强化）**
    *   单人限购：Redis `INCR` + 用户维度 Key（`sva:user:{activityId}:{userId}`）
    *   周期配额：Redis `INCR` + 时间窗口 Key（`sva:counter:{activityId}:{cycleKey}`），周期边界原子重置
    *   赠送比例预警：`bonus_amount / deposit_amount > 阈值` 时触发人工审核

*   **6-2 并发与一致性建模** ⭐ 新增
    *   识别高并发下会产生争用的字段，制定防超卖方案：

        | 争用点 | 风险 | 方案 |
        |-------|------|------|
        | 周期售卖总量 | 超卖 | Redis 原子 INCR，达上限返回失败 |
        | 单人购买次数 | 重复购买 | 分布式锁（用户粒度） |
        | 会员等级升级 | 并发升至错误等级 | 乐观锁（version 字段） |
        | 协议版本绑定 | 用户签署旧版协议 | 快照 + 版本号，购买时锁定当前版本 |

*   **6-3 资金安全建模** ⭐ 新增
    *   **账务隔离：** 储值本金 vs 赠送虚拟金额必须分账存储，消费时分别核销
    *   **消费优先级：** 通常先扣赠送金额，后扣本金（降低商家实际成本）
    *   **退款拆分规则：** 部分消费后退款，剩余金额须按本金/赠送比例拆分退回
    *   **最大财务敞口计算：** `max(bonus_amount/deposit_amount) × periodic_sales_limit = 单周期最大营销亏损`

*   **6-4 版本快照策略（原有，强化）**
    *   **触发时机：** 用户支付成功瞬间，对活动配置做不可变快照
    *   **快照内容：** 活动完整配置 + 协议文本版本号
    *   **快照用途：** 退款计算 / 权益纠纷 / 审计追溯，保证"买时是什么就是什么"

## 第七步：操作行为与生命周期 (Lifecycle & Audit)

**指令：** "分析该业务实体的状态流转、字段级操作权限，以及审计留存要求。"

*   **7-1 状态机（原有，强化）**
    *   为每个状态迁移明确：触发条件 / 执行者（系统自动 vs 人工操作）/ 副作用 / 可逆性
        ```
        草稿 ──[提交发布]──► 待开始 ──[到达开始日期，系统自动]──► 进行中
                                                                       │
                                                          [到达结束日期] │ [手动禁用]
                                                                       ▼         ▼
                                                                    已结束     已禁用
        ```

*   **7-2 字段级操作权限矩阵** ⭐ 新增，强化原有"操作权限"
    *   按「活动状态 × 字段」建立可编辑性矩阵（✏️可编辑 / 🔒只读保护 / 👁️仅查看）：

        | 字段 | 草稿 | 待开始 | 进行中 | 已结束 | 已禁用 |
        |-----|------|-------|-------|-------|-------|
        | 活动名称 | ✏️ | ✏️ | ✏️ | 👁️ | 👁️ |
        | 活动日期 | ✏️ | ✏️ | 🔒 | 👁️ | 👁️ |
        | 储值档位金额 | ✏️ | ✏️ | 🔒 | 👁️ | 👁️ |
        | 赠送内容 | ✏️ | ✏️ | 🔒 | 👁️ | 👁️ |
        | 购买渠道 | ✏️ | ✏️ | ✏️ | 👁️ | 👁️ |
        | 协议绑定 | ✏️ | 🔒 | 🔒 | 👁️ | 👁️ |

    *   🔒 标记的字段：进行中修改会影响已购用户权益，须强制走版本快照或禁止修改

*   **7-3 审计日志要求** ⭐ 新增
    *   **审计标准：** 凡涉及金额、用户权益、法律协议的字段变更 = 必须记录审计日志
    *   **必须审计的字段：**
        *   储值金额 / 赠送金额修改（财务影响）
        *   协议内容变更（法律影响）
        *   活动状态变更（业务影响）
        *   适用用户范围变更（权益影响）
    *   **审计日志字段标准：** `operator_id` + `timestamp` + `client_ip` + `old_value` + `new_value` + `change_reason`

*   **7-4 通知与消息触发点**
    *   活动临期（N 天前）→ 通知运营人员
    *   周期配额耗尽 → 通知运营人员（库存预警）
    *   活动下架 → 通知已购但未使用的用户（权益告知）

## 附加维度：工程落地视角 (Engineering Dimensions)

### E-1 财务对账视角（原有，强化）
*   **消费优先级：** 实付本金 vs 赠送虚币的消耗先后顺序（通常：赠送先扣，本金后扣）
*   **分账规则：** 多门店活动下，储值金如何在门店间分配（按收银门店归属）
*   **退款资产回滚顺序：** 先退赠送（核销赠送金额），后退本金（原路返回支付渠道）

### E-2 数据埋点建议（原有，强化）
*   **用户行为漏斗（转化率分析）：** 档位曝光 → 档位点击 → 支付发起 → 支付成功
*   **管理员操作热度（产品优化）：** 哪个配置项最频繁被修改？哪个字段最多触发校验错误？
*   **必须记录 Audit Log 的字段：** 凡涉及金额、状态、权益的变更（见 7-3）

### E-3 可扩展性设计建议（原有，强化）
*   **识别当前写死的枚举，预判扩展方向：**
    *   `channel_flags` 用位运算设计 → 新增渠道只需新增 bit 位，零改造
    *   `rule_strategy`（固定金额档位）用策略模式 → 未来支持自定义金额、阶梯比例
*   **预留扩展接口：** 赠送礼品类型扩展（当前：优惠券/券包；未来：实物礼品/联名会员权益）

### E-4 API 设计推导 ⭐ 新增
*   从页面交互反推 API 边界，避免前后端接口设计割裂：

    | 页面交互 | 对应 API | 幂等要求 |
    |---------|---------|---------|
    | 页面初始化（编辑模式） | `GET /activities/{id}` | - |
    | 分类下拉加载 | `GET /categories?brand_id=xxx` | - |
    | 保存草稿 | `POST /activities`（status=DRAFT） | 需防重（唯一请求ID） |
    | 发布活动 | `PATCH /activities/{id}/publish` | 需幂等（重复发布不报错） |
    | 内联新建分类 | `POST /categories` → 返回 `{id,name}` 立即回填 | - |
    | 用户购买储值 | `POST /deposits` | **强幂等**（防重复扣款） |

### E-5 性能与容量规划提示 ⭐ 新增
*   **热路径识别：** 用户购买时的"活动有效性校验"是高频读操作 → 建议本地缓存 TTL 30s，活动修改时广播失效
*   **增长型数据识别：** 购买记录、销量计数器 → 需要分区/归档策略
*   **周期重置的定时任务：** 周期销量计数器在周期边界重置，需考虑跨时区场景

## 给 AI 的 Master Prompt（可直接复制使用）

**使用方法：** 将下方内容作为 System Prompt 或对话开头发给任何支持图片的 AI，然后附上页面截图。

---

> 你是一位资深系统架构师，同时具备产品经理、后端工程师和安全工程师的复合视角。
> 请按照《复杂业务系统页面深度分析框架 V2.0》，对我提供的页面截图进行像素级解构，产出可直接用于开发需求文档的专业分析报告。
>
> **请严格按以下 7 个分析层 + 5 个工程维度输出，不可省略任何层次：**
>
> **【第零层：前置识别】**
> - 明确操作者角色、终态受益者、所属子系统模块
> - 识别表单交互模式（向导式/单页/弹窗）及草稿保存机制
> - 识别租户边界字段，说明其是"过滤器"还是"归属权声明"
>
> **【第一层：结构解构】**
> - 划分页面职能区块，描述区块间的业务信息流向
> - 分析视觉层级（缩进/颜色/分组）作为业务规则的隐式编码
> - 分析页面"刻意缺失"的元素及其暗示的系统假设（负空间分析）
>
> **【第二层：数据建模】**
> - 提炼核心实体、值对象、主从表关系，建立字段级数据字典（含类型、存储策略）
> - 建立跨区字段依赖拓扑图，标注依赖方向
> - 识别所有判别字段（Discriminator），分析默认值背后的产品设计哲学
> - 区分"存储字段"与"派生计算字段"
>
> **【第三层：规则提炼】**
> - 挖掘 Tooltip、帮助文字、Placeholder、按钮文字中的隐式业务规则
> - 建立完整的 UI 条件渲染规则映射表（`WHEN X THEN Y` 格式，穷举所有联动）
> - 区分"业务不变量"（任意时刻必须成立）与"验证规则"（提交时校验）
> - 按「阻断类 / 逻辑类 / 警告类」三级建立约束规则清单
>
> **【第四层：系统集成】**
> - 绘制外部系统依赖地图，标注每个调用的同步/异步方式及失败处理策略
> - 建立读模型（消费者视图）与写模型（管理员视图）的字段映射差异
> - 分析数据级联规则（父实体变更/删除时各子实体的处理策略）
>
> **【第五层：风险建模】**
> - 识别所有并发争用点，设计原子操作保障方案
> - 建立资金安全模型（本金/赠送金隔离、退款拆分、最大财务敞口计算）
> - 按「活动状态 × 字段」建立操作权限矩阵（可编辑/只读保护/仅查看）
>
> **【第六层：生命周期】**
> - 绘制完整状态机（含触发条件、执行者、副作用、可逆性）
> - 定义审计日志字段级标准（哪些字段变更必须记录 old_value/new_value）
> - 识别系统通知触发点及通知对象
>
> **【工程视角（5个）】**
> 1. 财务对账：本金/赠送金消费优先级、退款资产回滚顺序
> 2. 数据埋点：转化率漏斗埋点、管理员操作热度埋点
> 3. 可扩展性：识别当前写死的枚举及其扩展预留方案
> 4. API 推导：从页面交互反推 API 边界及幂等要求
> 5. 性能规划：热路径缓存策略、增长型数据归档策略
>
> **输出格式要求：**
> - 每个分析点需包含：UI 元素描述 → 业务语义 → 技术实现建议
> - 约束规则需标注严重级别（BLOCK / LOGIC / WARN）
> - 最终产出须可直接作为开发需求文档的验收标准使用
