# 09-解耦架构说明：需求分析 vs 代码实现

> 本文档说明 V6 版本的核心架构改进——将需求分析和代码实现解耦为两个独立阶段，
> 通过精确定义的"中间表示"（IR）实现可插拔的多技术栈支持。

---

## 一、问题：为什么要解耦

### 原始状态

`prompt-all-in-one-需求分析.txt` 将三件事混在一起：

```
1. 需求分析（UI→业务→数据模型）         ← 与技术栈无关
2. 数据建模（逻辑模型 + 物理模型优化）    ← 逻辑模型无关，物理优化有关
3. 代码实现导向（MySQL方言、Java引用等）  ← 完全绑定技术栈
```

具体的耦合点：

| 耦合点 | 原文位置 | 耦合内容 |
|--------|---------|---------|
| UI→数据模式速查 | lines 40-68 | TINYINT、Java Enum、DTO、DELETE+INSERT |
| 字段字典 | lines 381-398 | 判别控制列中的 Java 引用 |
| 枚举定义表 | lines 400-414 | "Java 枚举类"引用 |
| DDL 预览 | lines 436-463 | AUTO_INCREMENT、COMMENT=、ENGINE=、ON UPDATE |
| 保存事务边界 | lines 416-427 | INSERT/UPDATE/DELETE+INSERT SQL动词 |
| 规则执行层 | lines 479-491 | Service/Validator、DDL/Entity、Job/EventHandler |
| 接口数据结构 | lines 492-558 | DTO 字样 |
| 物理表优化 E-5 | lines 604-643 | MyBatis-Plus typeHandler、JSON合并 |

### 耦合的代价

1. **无法复用**：同一份需求文档无法对接 .NET、Go、Vue3 等其他技术栈
2. **职责混乱**：需求分析师需要懂 MySQL 方言和 Java 框架
3. **变更扩散**：换数据库（MySQL→PostgreSQL）需要改需求分析 prompt
4. **认知负担**：一个 prompt 承载太多关注点，AI 容易遗漏

## 二、解决方案：编译器架构

### 核心洞见

借鉴编译器设计的思想（详见「更高维度思维方法-超越修补Prompt.md」第四章）：

```
源语言（UI截图）──→ 前端编译器 ──→ 中间表示 IR ──→ 后端编译器 ──→ 目标代码
                     ↑                 ↑                 ↑
            prompt-需求分析-通用    需求文档标准格式   prompt-代码生成器
                                                    + code-style-xxx
```

### 解耦后的文件体系

```
┌─────────────────────────────────────────────────────────────┐
│  前端（技术无关）                                             │
│                                                              │
│  prompt-需求分析-通用.txt ───→ 需求文档（遵循标准格式）        │
│  需求文档标准格式.md          ← 契约/类型系统                  │
└──────────────────────────────┬──────────────────────────────┘
                               │ 标准需求文档（IR）
              ┌────────────────┼────────────────┐
              ↓                ↓                ↓
┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐
│  代码生成（可插拔）│ │  业务文档生成      │ │  其他消费者...    │
│                  │ │                  │ │                  │
│ prompt-代码生成器 │ │ prompt-业务功能   │ │ prompt-测试用例  │
│ + code-style-xxx │ │  说明书生成器     │ │  生成器（未来）  │
│    ↓             │ │    ↓             │ │                  │
│ 目标代码         │ │ 业务功能说明书    │ │                  │
└──────────────────┘ └──────────────────┘ └──────────────────┘
```

### 文件清单与职责

| 文件 | 角色 | 类比 |
|------|------|------|
| `prompt-需求分析-通用.txt` | 前端编译器 | gcc 的 parser |
| `需求文档标准格式.md` | IR 规范 | LLVM IR spec |
| `prompt-代码生成器-通用框架.txt` | 后端编译器骨架 | LLVM codegen |
| `code-style-springboot-alibaba.txt` | 目标平台规范 | x86 target |
| `prompt-业务功能说明书生成器.txt` | IR→业务文档翻译器 | 反编译器/文档生成 |
| `code-style-dotnet.txt`（未来） | 另一个目标平台 | ARM target |

## 三、改造详情

### 3.1 需求分析 prompt 改造点

| 改造点 | 原文内容 | 改造后 | 精度影响 |
|--------|---------|--------|---------|
| 数据类型 | MySQL 方言（TINYINT、DATETIME） | 通用SQL标准（INTEGER、TIMESTAMP、BOOLEAN） | 无损——类型精度（长度、小数位）全部保留 |
| 枚举引用 | "Java Enum 类" | "代码生成时映射为对应语言枚举类型" | 无损——枚举定义表内容不变 |
| 推导链 | "DTO 嵌套为 List<子DTO>" | "保存接口中嵌套为子对象列表" | 无损——嵌套关系语义不变 |
| 关联策略 | "DELETE+INSERT 策略" | "随主表整体替换（先清后写）" | 无损——业务语义不变，具体SQL策略移到代码生成 |
| DDL 预览 | MySQL DDL 全语法 | 通用 SQL 逻辑建表 | 无损——字段名、类型精度、约束、索引全部保留 |
| 事务边界 | INSERT/UPDATE/DELETE SQL动词 | 业务语义描述（新增/更新/替换） | 无损——业务操作语义不变 |
| 规则执行层 | "Service/Validator"等框架引用 | "后端业务层"等通用层级名 | 无损——分层位置语义不变 |
| 接口结构 | DTO 字样 | "数据对象" | 无损——JSON结构本身不变 |
| 物理表优化 | E-5 整节 | 移除→移到代码生成阶段 | 按设计移走——纯实现决策 |

### 3.2 输出结构对比

```
原结构（13章 + 物理优化）              解耦后（13章，物理优化移走）
─────────────────────              ─────────────────────────
术语表                              术语表
一、业务概述                         一、业务概述
二、页面结构                         二、页面结构
三、数据模型                         三、数据模型（技术无关逻辑模型）
  3.1-3.6 不变                        3.1-3.6 不变
  3.7 保存事务边界(含SQL)              3.7 保存事务边界(业务语义)  ← 去SQL动词
  3.8 ER图                            3.8 ER图
  3.9 DDL预览(MySQL)                  3.9 逻辑建表预览(通用SQL)  ← 去方言
四、业务规则                         四、业务规则
  4.4 执行层(框架引用)                  4.4 执行层(通用层级)      ← 去框架引用
  4.5 接口结构(DTO)                    4.5 接口结构(数据对象)    ← 去DTO字样
五-八 不变                           五-八 不变
九、工程维度                         九、工程建议
  9.4 物理表优化(MyBatis-Plus)          ← 移除，移到代码生成阶段
十-十三 不变                         十-十三 不变
```

### 3.3 代码生成器新增的步骤

物理表优化从需求分析移到代码生成的第一步：

```
原流程：
  需求分析 ──→ [含物理优化的需求文档] ──→ 代码生成

解耦后流程：
  需求分析 ──→ [逻辑模型需求文档] ──→ 代码生成器
                                          ↓
                                    步骤0: 读取校验
                                    步骤1: 物理模型决策 ← 从需求分析移入
                                    步骤2-10: 按层生成代码
```

代码生成器的「精度传递保障协议」确保每一条需求信息都被准确翻译到代码中。

## 四、精度保障

### 4.1 精度自检矩阵

| 精度维度 | 原文档 | 新需求文档 | 代码规范文件 | 生成的代码 |
|---------|--------|-----------|------------|-----------|
| VARCHAR(100) | ✅ MySQL | ✅ 通用SQL | ✅ 映射表 | ✅ MySQL DDL |
| DECIMAL(10,2) | ✅ MySQL | ✅ 通用SQL | ✅ 映射表 | ✅ BigDecimal + DDL |
| 枚举值 1=指定日期 | ✅ 枚举表 | ✅ 枚举表(不变) | ✅ 生成规则 | ✅ Java Enum |
| OWN/REF 分类 | ✅ 容器类型 | ✅ 容器类型(不变) | ✅ 生成规则 | ✅ Entity vs Long |
| 必填/默认值 | ✅ 字段字典 | ✅ 字段字典(不变) | ✅ 校验规则 | ✅ @NotNull + DDL |
| RULE 穷举 | ✅ 规则层 | ✅ 规则层(不变) | ✅ 执行层分类 | ✅ Service 校验 |
| 约束矩阵(7级) | ✅ 约束层 | ✅ 约束层(不变) | ✅ 校验注解 | ✅ @Min/@Max |
| 事务边界 | ✅ SQL动词 | ✅ 业务语义 | ✅ 事务规范 | ✅ @Transactional |
| 嵌套JSON结构 | ✅ DTO结构 | ✅ 数据结构(不变) | ✅ DTO命名 | ✅ 嵌套DTO |
| 索引设计 | ✅ DDL索引 | ✅ 逻辑建表索引 | ✅ 建表模板 | ✅ MySQL索引 |
| MySQL DDL方言 | ✅ 内嵌 | ❌ 移除 | ✅ 建表模板 | ✅ 目标DDL |
| 物理表优化 | ✅ E-5 | ❌ 移除 | ✅ 优化规则 | ✅ 物理模型 |

**结论**：最后两行（DDL方言、物理优化）按设计移到代码规范文件，不是丢失而是重新安置。
其余 10 个精度维度完全无损。

### 4.2 信息不丢失的保证机制

```
需求文档中的信息 ───→ 代码生成器的精度传递检查清单 ───→ 逐条验证
                          ↑
              prompt-代码生成器-通用框架.txt 第六章
              定义了 16 项精度传递验证规则
```

## 五、使用方式

### 场景 A：Spring Boot 后端开发（现有流程）

```
Step 1: 新对话 → 粘贴 prompt-需求分析-通用.txt → 发送截图
        ↓ 一问一答
        产出：标准需求文档.md

Step 2: 新对话 → 粘贴 prompt-代码生成器-通用框架.txt
        → 附上 标准需求文档.md
        → 附上 code-style-springboot-alibaba.txt
        ↓
        产出：Spring Boot 全套代码 + MySQL DDL + Apifox 测试用例
```

### 场景 B：切换技术栈

```
同一份 标准需求文档.md
  + code-style-dotnet.txt     → .NET 代码 + SQL Server DDL
  + code-style-go.txt         → Go 代码 + PostgreSQL DDL
  + code-style-vue3.txt       → Vue3 前端代码
  + code-style-flutter.txt    → Flutter 移动端代码
```

### 场景 C：前后端同时生成

```
标准需求文档.md
  ├─ + code-style-springboot-alibaba.txt  → 后端代码
  └─ + code-style-vue3-element.txt        → 前端代码（基于同一份接口结构）
```

### 与分步模式（多 Agent）的关系

```
原有 7 个 Agent（00-06）                    解耦后
────────────────────                    ────────
Agent 0-4: 分步需求分析                  → prompt-需求分析-通用.txt（合并版）
Agent 5:   代码生成师(Spring Boot耦合)   → prompt-代码生成器-通用框架.txt
                                           + code-style-springboot-alibaba.txt
Agent 6:   红队审查员                    → 不变，仍可独立使用
All-in-One: 需求分析(Spring Boot耦合)    → prompt-需求分析-通用.txt（替代）
```

## 六、扩展指南：编写新的代码规范文件

要支持新技术栈，只需创建一个新的 `code-style-{tech}.txt`，包含以下章节：

```
必须包含：
  □ 技术栈声明（语言、框架、ORM、版本）
  □ 分层职责（对应通用框架的 10 个生成步骤）
  □ 包/模块结构
  □ 命名规范
  □ 数据类型映射表（通用SQL → 目标数据库 → 目标语言）
  □ 主键策略
  □ 审计字段模板
  □ 建表脚本模板（目标数据库方言）

可选包含：
  □ 物理模型优化规则
  □ 架构哲学
  □ 代码质量规范
  □ 微服务规范
  □ 测试工具适配
```

## 八、设计决策记录

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|
| 逻辑建表用通用SQL | 通用SQL标准子集 | 无SQL / 纯表格 | 通用SQL是技术人员的"最大公约数"，可被任何数据库方言翻译 |
| 事务边界用业务语义 | "新增或更新""整体替换" | 保留SQL动词 | SQL动词（DELETE+INSERT）是实现策略，不是业务语义 |
| 物理优化移到代码生成 | 代码规范文件定义规则 | 保留在需求文档 | 物理优化是"编译器优化 pass"，不是"IR 的一部分" |
| 枚举引用改为技术无关 | "映射为对应语言枚举类型" | 保留 Java Enum | 枚举定义表的精度不依赖于目标语言 |
| 保留 [M][D][P][G][J] 分类法 | 不变 | 简化为主/从/关联 | 五分类法精度更高，且与 UI 结构一一对应 |
