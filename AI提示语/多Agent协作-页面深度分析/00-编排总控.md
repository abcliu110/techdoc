# 复杂业务系统页面深度分析 — 多 Agent 协作编排方案 V1.0

## 一、方案概述

将原始的单体 Master Prompt 拆解为 **4 个专职 Agent**，每个 Agent 聚焦自身领域，通过**结构化中间产物**进行协作，最终由整合者产出完整需求文档。

### 为什么要拆？

| 问题 | 单体 Prompt | 多 Agent 协作 |
|------|------------|--------------|
| 注意力分配 | 1500 字指令，中间层执行力下降 | 每个 Agent ≤ 500 字指令，聚焦深入 |
| 输出质量 | 面面俱到但每层浅尝辄止 | 各层深度分析，专业度更高 |
| 可调试性 | 整体输出不满意需全部重来 | 哪一层不满意只重跑那个 Agent |
| 灵活组合 | 只能全量执行 | 可按需选择 Agent 组合（如只做建模） |
| 上下文利用 | 单窗口上下文被指令占满 | 每个 Agent 的上下文专注于自身分析 |

                         │
                         ▼
              ┌─────────────────────┐
              │  Agent-1 业务分析师   │  第0-2层
              │  Business Analyst   │  语境 + 结构 + 建模
              └──────────┬──────────┘
                         │ 产出：业务上下文报告 + 数据模型

              ┌─────────────────────┐

              └──────────┬──────────┘
                         │ 产出：规则清单 + 约束矩阵

              ┌─────────────────────┐

                         │ 产出：架构分析报告

              ┌─────────────────────┐

              └─────────────────────┘

## 三、中间产物规范

Agent 之间通过**结构化 JSON/Markdown 中间产物**传递信息，确保下游 Agent 能准确消费上游输出。

### 3.1 Agent-1 → Agent-2 的交接产物

```markdown
## 业务上下文摘要（供下游 Agent 使用）

### 系统定位
- 操作者角色：{角色}
- 所属子系统：{子系统}
- 表单模式：{向导式/单页/弹窗}

### 核心实体清单
- 主实体：{实体名}（字段列表）
- 从实体：{实体名}（字段列表，与主实体关系：1:N）

### 判别字段清单
- {字段名} → 控制 {受控字段列表} 的 {显隐/必填/计算逻辑}

### 跨区依赖字段
- {字段A}（定义于{区块X}）→ 驱动 {字段B}（消费于{区块Y}）
```

### 3.2 Agent-2 → Agent-3 的交接产物

```markdown
## 规则与约束摘要（供下游 Agent 使用）

### UI 联动规则清单
- WHEN {条件} THEN {效果}（共 N 条）

### 业务不变量
- {不变量表达式}（违反 = 数据损坏）

### 约束分级表
- BLOCK 级：{N} 条
- LOGIC 级：{N} 条
- WARN 级：{N} 条

### 计算/派生字段
- {字段名} = {计算公式}（不存储）
```

### 3.3 Agent-3 → Agent-4 的交接产物

```markdown
## 架构分析摘要（供整合使用）

### 外部系统依赖
- {操作} → {下游服务}（同步/异步）→ 失败策略：{策略}

### 并发争用点
- {争用点} → 方案：{方案}

### 状态机
- {状态转换图}

### API 清单
- {交互} → {API endpoint}（幂等要求：{是/否}）

### 权限矩阵
- {状态 × 字段} 矩阵
```

## 四、执行流程

### 4.1 串行执行（简单场景）

```
1. 用户 → Agent-1：发送截图 + 业务背景
2. Agent-1 输出 → 人工审核 → 确认无误
3. Agent-1 输出 + 截图 → Agent-2
4. Agent-2 输出 → 人工审核 → 确认无误
5. Agent-1 + Agent-2 输出 + 截图 → Agent-3
6. Agent-3 输出 → 人工审核 → 确认无误
7. Agent-1 + Agent-2 + Agent-3 全部输出 → Agent-4
8. Agent-4 产出最终文档 → 人工终审
```

### 4.2 半并行执行（效率优化）

Agent-2 和 Agent-3 的部分工作可以并行：
- Agent-3 的"外部系统依赖分析"只需要 Agent-1 的实体模型，不依赖 Agent-2
- Agent-3 的"并发建模"需要 Agent-2 的规则输出

```
           Agent-1
          ┌───┴───┐
     Agent-2    Agent-3（部分：系统集成、生命周期）
          └───┬───┘
       Agent-3（部分：风控建模，消费 Agent-2 输出）
              │
          Agent-4
```

### 4.3 人工检查点

每个 Agent 输出后设置人工检查点（Gate），确保：
- [ ] 实体识别完整，无遗漏字段
- [ ] 规则无矛盾（不存在互斥的 WHEN-THEN）
- [ ] 不变量表达式可执行（非自然语言描述）
- [ ] API 列表覆盖所有页面交互
- [ ] 状态机无死锁（每个状态都有出路或是终态）

## 五、适用平台

| 平台 | 使用方式 |
|------|---------|
| **Claude**（对话） | 4 轮对话，每轮切换 System Prompt |
| **ChatGPT**（对话） | 4 轮对话，或使用 4 个自定义 GPTs |
| **Claude Code / Cursor** | 4 个 Agent 子任务并行/串行调度 |
| **Dify / Coze** | 工作流节点编排，自动传递中间产物 |
| **AutoGen / CrewAI** | 代码编排多 Agent，全自动执行 |
| **LangGraph** | 状态图编排，支持条件分支和人工审核节点 |
