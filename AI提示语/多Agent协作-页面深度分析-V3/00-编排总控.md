# 复杂业务系统页面深度分析 — 多 Agent 协作编排方案 V3.1

## 一、V3.1 核心升级

相比 V2 的线性流水线，V3 引入三项架构级改进，V3.1 在此基础上新增隐性知识提取增强：

| 改进 | V2 | V3 | V3.1 新增 |
|------|----|----|----------|
| 输入质量 | 截图 + 一句话背景 | Agent-0 结构化补全后的增强输入包 | Agent-0 新增技术资产扫描阶段，从代码/DDL/API文档中提取隐性约束 |
| 信息流拓扑 | 单向管道 1→2→3→4 | 带反馈环的有向图，支持逆向修正 | — |
| 验证机制 | Agent-4 被动交叉验证 | Agent-5 主动对抗性攻击 | Agent-5 新增探针验证维度 |
| 中间产物 | 交接摘要（有损压缩） | 交接摘要 + 推理备忘录（保留不确定推断） | — |
| 知识可信度 | 无区分 | `[⚠️ 假设]` 标记 | 四级知识来源分类（📸截图/👤用户/📄资产/🤖推断） |
| 隐性知识 | 无 | Agent-0 提问缓解 | Agent-1/2/3 隐性知识探针（Tacit Knowledge Probe） |

## 三、反馈环机制

### 3.1 反馈环触发条件

Agent-5（红队审查员）在审查过程中，如果发现以下情况，将生成 FEEDBACK 指令：

| 触发条件 | 反馈目标 | 示例 |
|---------|---------|------|
| 信息缺口（需要用户确认） | Agent-0 | "用户未说明该下拉框的选项" |
| 实体/字段遗漏 | Agent-1 | "截图中有一个隐藏 Tab 的字段未被建模" |
| 规则缺失或矛盾 | Agent-2 | "两条联动规则在特定组合下冲突" |
| 架构方案不足 | Agent-3 | "并发方案未覆盖跨服务事务场景" |
| 文档格式/一致性问题 | Agent-4 | "术语表中的术语与正文不一致" |

### 3.2 反馈环执行流程

```
Agent-5 发现问题
       │
       ▼
生成 FEEDBACK 指令（含目标 Agent、内容、优先级）
       │
       ├── P0 反馈 → 必须执行
       ├── P1 反馈 → 建议执行
       └── P2 反馈 → 可选（记录到未决问题）
       │
       ▼
重跑目标 Agent（附上 FEEDBACK 内容作为补充指令）
       │
       ▼
目标 Agent 产出补充内容
       │
       ▼
重跑 Agent-4（增量整合）→ 重跑 Agent-5（验证修复）
```

### 3.3 反馈环终止条件

- Agent-5 无新的 🔴 Critical 发现
- 反馈环已执行 2 轮（防止无限循环）
- 人工判断剩余问题可接受

## 四、技术资产接入规范（V3.1 新增）

### 4.1 什么是技术资产接入

截图只包含显性信息，但业务系统 80% 的复杂度藏在截图看不到的地方——后端已有的校验逻辑、与其他模块的隐式耦合、历史遗留的特殊处理。Agent-0 通过提问缓解了"用户知道但没说"的部分，但无法触及"用户自己也不清楚"的部分。

技术资产接入让用户可以选择性地提供已有的技术资产，Agent-0 从中提取与截图页面相关的约束和规则，补全截图看不到的信息。

### 4.2 支持的技术资产类型

| 资产类型 | 示例 | 提取目标 |
|---------|------|---------|
| 代码片段 | Controller/Service/Entity/前端组件 | 已实现的校验逻辑、字段约束、业务规则 |
| 数据库 DDL | 建表语句 | 字段类型、长度、索引、外键、唯一约束 |
| API 文档 | Swagger/OpenAPI JSON/YAML | 接口定义、参数校验、枚举值 |
| 需求文档/PRD | 已有的产品文档 | 业务规则、状态流转、角色权限 |
| 配置文件 | 枚举字典、权限配置、错误码表 | 枚举值域、角色权限矩阵、错误处理规范 |

### 4.3 技术资产摘要格式

Agent-0 对技术资产做结构化提取后，输出"技术资产摘要"作为增强输入包的一部分：

```markdown
### 技术资产摘要（Technical Asset Summary）

#### 资产来源
- 提供的资产类型：{列表}
- 资产与截图页面的关联度评估：{高/中/低}

#### 已实现的约束（从代码/DDL 提取）
- {字段名}：{约束描述}（来源：{代码文件/DDL}）

#### 已定义的枚举值域（从代码/配置 提取）
- {枚举名}：{值列表}（来源：{配置文件/代码}）

#### 已有的业务规则（从PRD/代码 提取）
- {规则描述}（来源：{PRD/Service代码}）

#### 已有的接口定义（从API文档 提取）
- {接口} {方法} {路径}：{参数校验规则}

#### 资产与截图的差异点
- {截图显示X，但代码/DDL中为Y}（需人工确认）
```

关键：这不是让 AI 做代码审计，而是从已有资产中提取**与截图页面相关的约束和规则**，补全截图看不到的信息。

## 五、知识来源分类规范（V3.1 新增）

### 5.1 四级知识来源

V3.1 要求 Agent-1/2/3 的核心输出表格增加"知识来源"分类列，精确标注每条信息的可信度：

| 标记 | 含义 | 可信度 | 说明 |
|------|------|-------|------|
| 📸 截图可见 | 直接从截图中读取的信息 | 最高 | 可通过截图验证 |
| 👤 用户确认 | 用户在 Agent-0 提问环节确认的信息 | 高 | 来自增强输入包中的用户回答 |
| 📄 技术资产 | 从代码/DDL/API文档等技术资产中提取的信息 | 高 | 来自增强输入包中的技术资产摘要 |
| 🤖 AI 推断 | AI 基于经验和上下文推断的信息 | 需验证 | 原来的 `[⚠️ 假设]`，需人工或技术资产验证 |

### 5.2 使用规则

- 每条字段定义、规则、约束都必须标注知识来源
- 同一条信息如果有多个来源，标注可信度最高的来源
- 当截图信息与技术资产信息矛盾时，标注为 `[⚠️ 待确认]` 并列出两种说法
- 下游 Agent 和人工审核者可据此精确判断每条信息的可信度

## 六、隐性知识探针机制（V3.1 新增）

### 6.1 什么是隐性知识探针

Agent-1/2/3 在完成本层分析后，基于分析结果反向推导："要让这个系统正确运行，还必须存在哪些我没看到的东西？"

探针的输出不是答案（AI 不知道答案），而是**结构化的待验证问题清单**，标注为 `[🔎 探针]`，由人工或通过技术资产验证。

### 6.2 各 Agent 的探针方向

| Agent | 探针方向 | 示例探针问题 |
|-------|---------|------------|
| Agent-1（业务分析师） | 数据来源与消费 | "这个实体的数据从哪里初始化？是否有批量导入？" |
| Agent-1 | 数据流向 | "这个页面的数据被哪些其他页面/报表消费？" |
| Agent-1 | 后台任务 | "是否有定时任务会修改这些数据？" |
| Agent-2（规则工程师） | 规则实现现状 | "这些校验规则在后端是否已经实现？实现方式是什么？" |
| Agent-2 | 规则管理 | "是否有规则引擎/配置中心管理这些规则？" |
| Agent-2 | 历史事故 | "历史上是否有因为规则缺失导致的线上事故？" |
| Agent-3（系统架构师） | 调用链 | "这个服务的上下游调用链是什么？" |
| Agent-3 | 发布策略 | "是否有灰度发布/AB测试影响这个页面的行为？" |
| Agent-3 | 运维问题 | "监控告警中与这个模块相关的 Top 问题是什么？" |

### 6.3 探针输出格式

```markdown
## 隐性知识探针（Tacit Knowledge Probe）

### 待验证问题清单

| 编号 | 探针问题 | 探针方向 | 关联分析点 | 验证方式 | 优先级 |
|------|---------|---------|-----------|---------|-------|
| TKP-01 | {问题} | {方向} | {本层哪个分析结论依赖此答案} | 人工确认/查代码/查配置 | P0/P1/P2 |
```

### 6.4 探针的消费规则

- 人工审核时，优先验证 P0 探针问题
- 如果技术资产中已包含答案，Agent-0 在技术资产摘要中标注
- 已验证的探针答案可反馈给对应 Agent，触发增量分析
- Agent-4 整合时，将未验证的探针问题汇总到"未决问题清单"
- Agent-5 审查时，将未验证的探针问题作为攻击线索来源

## 七、推理备忘录规范

V3 要求 Agent-1/2/3 在输出交接摘要的同时，附加一份**推理备忘录**。

### 7.1 什么是推理备忘录

交接摘要是"结论"，推理备忘录是"推理过程中的不确定信号"。它记录：

- 被排除的备选假设（"我考虑过 X 方案但因为 Y 原因排除了"）
- 不确定的推断（"这个字段可能是从外部系统同步的，但截图无法确认"）
- 需要关注的异常信号（"这个按钮的措辞暗示可能有审批流程"）
- 与上游信息的矛盾点（"Agent-0 说是单租户，但页面上有门店选择器"）

### 7.2 推理备忘录格式

```markdown
## 推理备忘录（Reasoning Memo）

### 被排除的假设
- [RM-01] 考虑过 {假设A}，排除原因：{原因}
- [RM-02] ...

### 不确定推断
- [RM-03] {推断内容}，置信度：{高/中/低}，验证方式：{如何确认}
- [RM-04] ...

### 异常信号
- [RM-05] {观察到的异常}，可能含义：{推测}
- [RM-06] ...

### 与上游信息的矛盾
- [RM-07] 上游说 {X}，但本层分析发现 {Y}，建议：{处理方式}
```

### 7.3 推理备忘录的消费规则

- 下游 Agent **优先消费交接摘要**
- 遇到歧义或需要做判断时，**查阅推理备忘录**
- Agent-4 整合时，将高价值的推理备忘录条目转化为 `[⚠️ 待确认]` 标记
- Agent-5 审查时，将推理备忘录作为攻击线索来源

## 八、中间产物规范

### 8.1 Agent-0 → 全体下游

**增强输入包**（详见 01-Agent-信息补全者.md）

## 九、执行流程

### 9.1 完整执行（推荐）

```
1. 用户 → Agent-0：发送截图 + 业务背景 + 技术资产（可选）
2. Agent-0 输出提问清单 → 用户回答 → Agent-0 扫描技术资产 → Agent-0 输出增强输入包（含技术资产摘要）
3. 增强输入包 + 截图 → Agent-1
4. Agent-1 输出 → 人工审核 → 确认无误
5. Agent-1 交接摘要 + 推理备忘录 + 增强输入包 + 截图 → Agent-2
6. Agent-2 输出 → 人工审核 → 确认无误
7. Agent-1 + Agent-2 交接摘要 + 推理备忘录 + 增强输入包 + 截图 → Agent-3
8. Agent-3 输出 → 人工审核 → 确认无误
9. Agent-1 + Agent-2 + Agent-3 全部输出 → Agent-4
10. Agent-4 产出需求文档 → 人工初审
11. 需求文档 + 截图 + 增强输入包 → Agent-5
12. Agent-5 产出红队报告
    ├── 无 🔴 发现 → 合入 FIX 建议 → 文档交付 ✅
    └── 有 🔴 发现 → 执行反馈环 → 重跑对应 Agent → 回到步骤 9
```

### 9.2 半并行执行

```
                Agent-0（信息补全）
                    │
                Agent-1（业务分析）
               ┌────┴────┐
          Agent-2      Agent-3（部分：系统集成、生命周期）
               └────┬────┘
            Agent-3（部分：风控建模，消费 Agent-2 输出）
                    │
                Agent-4（文档整合）
                    │
                Agent-5（红队审查）
                    │
               ┌────┴────┐
           通过 ✅     反馈环 🔄
```

### 9.3 人工检查点

每个 Agent 输出后设置人工检查点（Gate）：

| Agent | 检查重点 |
|-------|---------|
| Agent-0 | 增强输入包是否覆盖关键信息，P0 问题是否都已回答，技术资产摘要是否准确 |
| Agent-1 | 实体识别完整，判别字段无遗漏，探针问题有价值 |
| Agent-2 | 联动规则穷举，不变量与验证规则区分清楚，探针问题有价值 |
| Agent-3 | 状态机无死锁，API 列表完整，并发方案具体，探针问题有价值 |
| Agent-4 | 术语统一，交叉验证通过，编号体系一致 |
| Agent-5 | 🔴 发现已处理，反馈环指令已执行或标记为可接受 |
