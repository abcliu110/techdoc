第零层：前置语境识别

0-1 系统角色识别

第一层：空间解构

1-1 区块职能分区
- 将页面划分为独立的职能区块，每个区块用一句话描述其业务目标
- 识别区块间的信息流向（用户如何从一个区块流转到下一个）
- 区块命名要反映业务语义，不要用"上中下"或"左右"等位置描述

1-2 视觉层级解读
- 缩进与分组：字段缩进关系通常代表父子条件依赖
- 颜色/状态语义：选中态、禁用态、警告态各代表什么业务含义
- 标签措辞分析："推荐" vs "指定" vs "限制" 背后的产品意图

1-3 负空间分析
- 页面刻意缺失的操作按钮暗示了什么约束？
- 缺失的字段暗示了什么系统假设或前置条件？
- 这些缺失对未来扩展有何影响？
- 将推测性结论记入推理备忘录，不要在正文中断言

1-4 导航结构分析
- 面包屑路径 = 数据层级归属关系
- 按钮语义区分："保存" / "发布" / "提交" / "确认" 各自对应什么状态变更
- 是否有步骤指示器（Wizard）或锚点导航


第二层：数据建模（面向数据库设计）

★ 核心原则：UI 层级 = 数据表层级。字段必须按所属数据库实体分组，不按 UI 区块分组。
★ 使用增强输入包中的「数据容器识别」结果作为建表依据。若增强输入包中无此内容，需自行执行数据容器识别。

★ UI模式→数据模式 速查规则（机械执行，不需要推理）：

  文本输入框         →  VARCHAR 列, 直接存储
  数字输入框         →  INT/DECIMAL 列, 直接存储
  日期选择器         →  DATE/DATETIME 列, 直接存储
  单选 Radio         →  TINYINT 枚举列 + 枚举值定义表条目
  单选下拉框         →  TINYINT/INT 枚举列 或 FK→引用表
  多选 Checkbox      →  关联表 [J], 每个选中值一行
  多选下拉框(标签)   →  关联表 [J]
  开关 Toggle        →  TINYINT(0/1) 列
  时间范围(可多条)   →  时间段子表 [D], FK→父表
  多日期选择器       →  日期子表 [D], FK→父表
  可重复卡片(+添加)  →  明细表 [D], FK→父表
  按钮触发的弹窗     →  子实体表 [P], FK→触发区域所属表
  弹窗内可编辑表格   →  嵌套子表 [G], FK→弹窗表
  文件上传(多文件)   →  附件表 [D], FK→父表
  富文本编辑器       →  TEXT/LONGTEXT 列, 直接存储

  识别后自动推导链：
    [D]/[P]/[G] → 需要外键 → 在父表事务内 → DTO 嵌套为 List<子DTO>
    [J]         → 需要关联表 → DELETE+INSERT 策略 → DTO 用 List<Long>
    枚举列      → 枚举值定义表 → 判别字段检查 → Java Enum 类

2-1 实体层级与关系
- 基于 5 类数据容器（[M]主表单/[D]可重复明细/[P]弹窗子实体/[G]弹窗内表格/[J]多选关联），输出实体层级图
- 每个实体包含：英文表名、PK、FK → 父表、数量约束、UI来源
- 用树形缩进表示父子关系
- ★ 区分"拥有实体"与"引用实体"：
  [OWN] 拥有实体：本页面创建和管理的数据 → 需生成 Entity + Mapper + Service
  [REF] 引用实体：本页面仅通过 ID 引用的已有数据 → 不生成代码，只存 ID
- 区分核心实体（有独立 ID，独立生命周期）与值对象（与实体同生命周期，无独立 ID）

2-2 字段数据字典（★ 按实体分表输出）
- ★ 关键要求：每个数据库实体单独一张字段表，不要把所有字段混在一起
- 每张表必须包含：主键(id)、外键、审计字段(create_time, update_time, create_by)
- 每张表头部标注：父表、关系类型(1:N/N:N)、UI来源(页面区域/弹窗)
- 新增"判别控制"列：标注该字段如何影响其他字段或子表的存在性
- UI 元素类型：文本框 / 下拉选择 / 单选 / 多选 / 日期选择 / 数字输入 / 开关 / 文件上传 / 富文本
- 存储建议：直接存储 / JSON数组 / 位运算 / 关联中间表 / 不存储（派生）

2-3 实体关系图（ER图文本表示）
- 用标准 ER 表示法输出所有实体间关系，包含基数
- 格式：实体A ||──o{ 实体B : "1:N (约束说明)"

2-4 保存事务边界
- 明确哪些表在同一个事务中保存（前端提交一次 = 后端一个事务）
- 标注每张表的写入策略：INSERT/UPDATE/DELETE+INSERT

2-5 跨区字段依赖图
- 识别在 A 区定义、被 B/C 区消费的跨区驱动字段
- 用拓扑结构描述依赖方向（依赖方向决定后端校验的执行顺序）

2-6 判别字段识别（含数据层影响）
- 识别所有控制其他字段显隐/必填/计算逻辑的"主控字段"
- 格式表格含三列：判别字段 | UI控制效果 | 数据层影响
- "数据层影响"说明该判别字段如何影响数据库存储（字段是否允许NULL、子表是否有数据等）
- 判别字段必须显式存储，不可由其他字段推导

2-7 枚举值定义表（★ 面向代码生成，必须输出）
- 为所有 Radio、单选下拉框、Toggle、状态字段定义明确的数值映射
- 格式：字段名 | 所属表 | 枚举值(int) | 含义 | 说明(默认值等)
- 数值从 0 或 1 开始，不要用中文作为存储值
- 代码生成器将直接用此表生成 Java 枚举类
