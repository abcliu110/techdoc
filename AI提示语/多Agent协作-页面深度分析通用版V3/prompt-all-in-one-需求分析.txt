★ 核心认知：UI 的嵌套层级 = 数据库的主从表层级。

★ UI模式→数据模式 速查规则（机械执行，不需要推理）：

  文本输入框         →  VARCHAR 列, 直接存储
  数字输入框         →  INT/DECIMAL 列, 直接存储
  日期选择器         →  DATE/DATETIME 列, 直接存储
  单选 Radio         →  TINYINT 枚举列 + 枚举值定义表条目
  单选下拉框         →  TINYINT/INT 枚举列 或 FK→引用表
  多选下拉框(标签)   →  关联表 [J], 每个选中值一行

    已选值为标签(tag)形式（带×删除）= 多选 → 关联表[J]

  ★ 下拉框特殊值规则：
    选项中含「自定义」「其他」「不限」「指定XXX」等 → 可能触发额外UI
    每个特殊值 = 潜在的额外数据库字段 → 必须逐个切换确认
  开关 Toggle        →  TINYINT(0/1) 列
  时间范围(可多条)   →  时间段子表 [D], FK→父表
  多日期选择器       →  日期子表 [D], FK→父表
  可重复卡片(+添加)  →  明细表 [D], FK→父表
  按钮触发的弹窗     →  子实体表 [P], FK→触发区域所属表

★ 关键要求：不要把所有字段混在一起。每个数据库实体单独一张字段表。
★ 每张表必须包含：主键(id)、外键、审计字段(create_time, update_time, create_by)
★ 判别字段必须在"判别控制"列标注它控制哪些字段的存在性

格式：

#### 表名：{中文名}({英文表名})
父表：{父表名} | 关系：{1:N / N:N} | UI来源：{页面区域/弹窗}

| 字段名 | 列名(英文) | 数据类型 | 必填 | 默认值 | 说明 | 判别控制 |

"判别控制"列示例：

2-7 DDL 预览（★ 验证数据模型正确性的最佳手段）

基于字段字典和枚举表，为每个 [OWN] 实体输出 CREATE TABLE 语句。
作用：1) 验证数据模型设计无遗漏 2) 代码生成器可直接使用 3) 开发可直接执行建表

要求：
- 表名前缀 t_（如 t_stored_value_package）
- 主键 id BIGINT AUTO_INCREMENT
- 外键字段加注释标明关联表
- 枚举字段加注释标明枚举值含义
- 必须包含 create_time, update_time, create_by, deleted 审计字段
- 为外键字段添加索引（idx_{表名}_{字段名}）

格式示例：
CREATE TABLE t_stored_value_package (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  activity_name VARCHAR(100) NOT NULL COMMENT '活动名称',
  date_type TINYINT NOT NULL DEFAULT 1 COMMENT '活动日期类型: 1=指定日期 2=永久有效',
  start_date DATE NULL COMMENT '开始日期, date_type=1时必填',
  ...
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  create_by BIGINT NOT NULL COMMENT '创建人ID',
  deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除: 0=未删除 1=已删除'
) COMMENT='储值套餐主表';

CREATE INDEX idx_svp_create_by ON t_stored_value_package(create_by);

3-1 隐式文本：Tooltip→规则、Placeholder→约束、帮助文字→WARN/BLOCK、按钮→副作用

3-2 UI 条件渲染（穷举）：
RULE-{编号}: WHEN {条件} THEN {效果} ELSE {默认} 来源：📸/👤/🤖

3-5 规则执行层分类（★ 面向代码生成，必须输出）

将所有 RULE 和 INVARIANT 按执行层分类，直接指导代码生成：

| 规则编号 | 规则描述 | 前端UI层 | 后端校验层 | 数据库层 | 说明 |
|---------|---------|---------|-----------|---------|------|

分类标准：
- 「前端UI层」：仅影响界面渲染（显隐、禁用、选项联动）→ 生成前端代码
- 「后端校验层」：需要服务端二次校验或业务逻辑处理 → 生成 Service/Validator 代码
- 「数据库层」：影响表结构设计（字段可空性、约束、索引）→ 生成 DDL/Entity
- 「定时任务/事件」：需要异步处理 → 生成 Job/EventHandler 代码

3-6 保存接口请求体结构（★ 面向代码生成，必须输出）

基于实体层级关系，输出保存接口的嵌套 JSON 结构预览。
嵌套层级必须与 2-1 的实体层级完全对应。
代码生成器可以直接从此结构生成 DTO 类。

★ 必须同时输出三种结构：

A. 新增请求体 (POST)：明细记录没有 id
B. 编辑请求体 (PUT)：明细记录有 id（已有记录）或没有 id（新增记录），传空数组 = 全部删除
C. 详情响应体 (GET)：返回完整嵌套数据供前端回填表单

--- A. 新增请求 ---
POST /api/{主实体}
{
  // ─── 主表字段 ───
  "field1": "value",

  // ─── 明细表：{明细名称}列表 ───
  "details": [
    {
      "detailField1": "value",
      // ─── 子明细：{子明细名称} ───
      "subDetails": [ { ... } ]
    }
  ],

  // ─── 关联表：{关联名称} ───
  "relatedIds": [1, 2, 3]
}

--- B. 编辑请求（差异说明）---
PUT /api/{主实体}/{id}
{
  // 主表字段同上
  "details": [
    { "id": 101, "detailField1": "修改值" },  // 有id = 更新已有记录
    { "detailField1": "新值" }                  // 无id = 新增记录
    // 原有 id=102 的记录未出现 = 删除
  ]
}

--- C. 详情响应（★ 前端回填表单所需）---
GET /api/{主实体}/{id}
{
  "id": 1,
  // ─── 主表字段 ───
  "field1": "value",
  "field1Label": "展示值",  // 枚举字段同时返回 label

  // ─── 明细表 ───
  "details": [
    {
      "id": 101,
      "detailField1": "value",
      "subDetails": [ { "id": 201, ... } ]
    }
  ],

  // ─── 关联数据（含名称，不只是 ID）───
  "brands": [ { "id": 1, "name": "品牌A" } ],
  "orgIds": [1001, 1002],

  // ─── 审计信息 ───
  "createTime": "2024-01-01 00:00:00",
  "updateTime": "2024-01-02 00:00:00"
}

★ 详情响应与新增请求的区别：
- 所有记录都有 id
- 枚举字段同时返回 code 和 label
- 关联数据返回完整对象（含名称），不只是 ID 数组
- 包含审计字段
