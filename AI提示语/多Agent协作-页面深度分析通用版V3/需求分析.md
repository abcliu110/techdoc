# 储值套餐需求分析

## 一、业务概述

- **操作角色**：运营/活动运营在全来店后台配置余额套餐，终态受益者为门店及消费者，归属于会员/营销子系统。交互为单页长表单（“新增储值套餐”第一步基础信息），支持草稿保存，生命周期为草稿→启用→停用，保存即生效，无审批流程。来源：截图 & 用户确认。
- **表单形态**：单步编辑，所有字段同时展现；“储值规则”区只提供固定金额档位，未体现自定义金额；范围控制通过组织/品牌筛选，未贴出多租户字段。

## 二、页面结构

- **区块划分**
  1. 基础信息：名称/分类（可新增）、品牌、备注、启用开关等。
  2. 储值规则：固定金额档位配置、赠送金额/积分、适用渠道、小程序白名单。
  3. 购买限制：组织范围（全部/指定）、渠道勾选、可售渠道（POS、小程序、商户中心、自助大屏、码牌收银）、适用小程序白名单。
- **信息流**：基础信息决定规则组合，渠道/组织筛选提供约束，提交时联动校验。
- **负空间**：缺少自定义金额和新增渠道入口，表明当前仅支持固定档/精选通道。
- **导航**：左侧步骤导航 + 顶部“草稿/上一步/下一步”按钮。

## 三、数据模型

- **核心实体**
  - `bonus_package`（主表，含名称、分类、状态、生命周期、渠道）。
  - `package_organization`（从表，记录“指定门店/区域”选择）。
  - `package_channel`（值对象，当前渠道集固定，存储勾选状态）。
  - `mini_program_entry`（值对象，白名单两个小程序）。
- **字段字典（示例）**
  1. `package_name`：字符串（UI 标签“活动名称/套餐名称”），品牌/组织范围内唯一，必填，提交层校验“名称重复”。来源：截图+确认。
  2. `category_id`：下拉，支持新增（弹窗填写名称/品牌/备注）。来源：截图。
  3. `fixed_amounts`：固定金额档数组。当前仅允许此模式（无其他选项）。来源：截图+确认。
  4. `applicable_channels`：枚举集合（门店 POS、微信小程序、支付宝小程序、商户中心、自助大屏、码牌收银），默认全选但可手动取消。来源：截图。
  5. `mini_program_whitelist`：白名单，仅“全来店智慧餐厅（微信/支付宝）”。来源：截图。
  6. `organization_scope`：枚举（全部门店/指定门店/指定区域），指定门店需展开“选择组织”弹窗，顶部筛选项（品牌/门店分组/标签/经营方式/门店类型/城市）均可用，并可多选门店。来源：截图+确认。
  7. `state`：草稿/启用/停用，保存即生效。来源：截图+确认。

- **判别字段**：`organization_scope` 决定是否展示组织筛选弹窗；`applicable_channels` 选择小程序需同步约束小程序白名单。
- **跨区依赖**：渠道选择影响小程序白名单展示；组织筛选影响购买限制区可售范围。

## 四、业务规则

- **RULE-BR-001**：WHEN `organization_scope`=“指定门店” THEN 启用“选择组织”弹窗并允许使用所有筛选项（品牌/分组/标签/经营方式/类型/城市），ELSE 隐藏。来源：截图+确认。
- **RULE-BR-002**：WHEN 编辑“储值规则” THEN 只能配置“固定金额档位”，其他模式不可见。来源：截图+确认。
- **RULE-BR-003**：WHEN 选中小程序渠道 THEN `mini_program_whitelist` 必须从两个白名单项（微信/支付宝）中选择。来源：截图。
- **INV-001**：`package_name` 在品牌/组织范围内唯一，违反时阻止保存。来源：截图+确认。
- **计算字段**：无派生字段。

## 五、约束矩阵

- **C-001**：`package_name` 级别 BLOCK，提交校验，错误提示“套餐名称重复”，来源：📸+👤。
- **C-002**：`fixed_amounts` 仅允许固定金额档，级别 LOGIC，前端实时校验提示“当前仅支持固定金额档位”，来源：📸+👤。
- **C-003**：`organization_scope` 若选“指定门店”且未选任何门店，则提交报错“请选择门店”，级别 BLOCK，提交时校验，来源：📸+👤。
- **C-004**：`applicable_channels` 需至少勾选一个渠道（默认全选），无渠道时提示“请至少选择一个适用渠道”，级别 LOGIC，前端/提交校验，来源：📸。

## 六、系统集成

- **系统依赖**：组织筛选调用组织结构/门店列表接口，筛选项（品牌/标签等）由同一服务提供，失败需提示“组织加载失败”。渠道列表由配置接口返回。
- **CQRS**：写模型为套餐表及关联表；读模型提供列表/筛选/状态展示，字段保持一致。
- **数据级联**：套餐删除为软删，其组织/渠道记录同步失效，避免历史混淆。
- **API**
  1. `POST /api/bonus-packages`：幂等接口，提交草稿/启用/停用状态。
  2. `GET /api/organizations?filters=...`：支持品牌/分组/标签/经营方式/类型/城市。
  3. `GET /api/channels`：返回固定渠道集合及小程序白名单。

## 七、风险与安全

- 并发：多人编辑同一套餐需乐观锁（如 version 字段），防止覆盖。
- 数据安全：涉及金额与积分，操作需走权限校验，仅运营角色可动作。
- 防重/幂等：提交时再校验套餐名称唯一，并记录状态，避免重复保存。
- 版本策略：每次启用/停用写入审计轨迹，便于追踪。

## 八、生命周期

- **状态机**：草稿（可编辑）→ 启用（立即生效）→ 停用（仅停用但数据保留）。保存即变更状态，无审批通知。来源：确认。
- **字段权限**：草稿可编辑所有字段；启用后仅允许调整渠道/组织；停用后仅能切换回启用。
- **审计**：记录名称/渠道/组织/状态变更，供运营/风控复盘。
- **通知**：无额外通知/审批。

## 九、工程维度

- **性能**：组织筛选为热路径，需分页/懒加载门店；渠道列表可缓存，下拉白名单提前加载。
- **可扩展性**：渠道与小程序字段为枚举，应支持新增渠道（通过配置界面扩容）。
- **埋点**：记录每次保存/启用/停用操作的字段快照，便于后续营销分析。

## 十、验收标准

- **AC-001**：GIVEN 草稿状态，WHEN 填写唯一名称 + 固定档 + 指定组织 THEN 保存成功且进入启用。
- **AC-002**：GIVEN 套餐启用，WHEN 点击停用按钮 THEN 变为停用且渠道/门店立即失效。
- **AC-003**：GIVEN 组织范围为“指定门店”，WHEN 未选门店 THEN 提交报错“请选择门店”。
- **AC-004**：GIVEN 小程序渠道启用，WHEN 选择入口 THEN 仅允许“全来店智慧餐厅（微信/支付宝）”两个项。

## 十一、开发任务拆解建议

1. 表单基础布局（名称/分类/固定档）——中等复杂度（约12h）。
2. 组织选择及筛选弹窗——高复杂度（约24h）。
3. 渠道+小程序白名单配置——中等（约18h）。
4. 生命周期与状态控制（草稿/启用/停用）——高（约20h）。
5. 接口与幂等/校验（唯一名+状态检查）——高（约24h）。
