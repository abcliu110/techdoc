## 术语表

| 术语 | 定义 | 来源 |
|------|------|------|
| 储值套餐 | 商户创建的储值营销活动，顾客按固定金额档位充值可获赠送 | 📸 |
| 档位 | 储值套餐中的一个充值金额档次，含储值金额、赠送金额/积分/礼品 | 📸 |
| 可用周期 | 套餐在一周/一月中的哪些日期可用 | 📸 |
| 可用时段 | 套餐在一天中的哪些时间段可用 | 📸 |
| 排除日期 | 套餐不可使用的特定日期 | 📸 |
| 储值协议 | 约定商家与用户之间权益和义务的法律文本 | 📸 |
| 可用金额限制 | 储值后资金的使用限制方式（不限制/当日限制/当餐限制） | 📸 |

## 一、业务概述

### 1.1 系统定位

- **系统**：全来店商户中心 📸
- **所属子系统**：会员中心 > 储值活动 📸
- **操作者**：商户管理员 🤖
- **受益者**：商户（促进顾客预充值），最终用户（获得赠送优惠） 🤖

### 1.2 表单形态

- **交互模式**：分步表单 + 长页面混合。左侧有「基础信息」「储值规则」「购买规则」三个步骤锚点 📸
- **保存机制**：右上角「保存」按钮统一提交 📸
- **新增 vs 编辑差异**：当前为新增页面，编辑页面差异待确认 [⚠️ 假设]
- **导航按钮**：「取消」「上一步」「保存」 📸

## 二、页面结构

### 2.1 区块分区

| 区块 | 业务目标 | UI 位置 |
|------|---------|--------|
| 步骤导航 | 引导用户分步填写 | 左侧固定 📸 |
| 活动基础信息 | 配置套餐的基本属性（品牌、分类、日期、周期、时段、协议等） | 主内容区上部 📸 |
| 储值推荐规则 | 配置充值档位和赠送方案 | 主内容区中部 📸 |
| 购买规则 | 配置购买条件和限制 | 主内容区下部 📸（仅露出标题） |
| 操作栏 | 取消/上一步/保存 | 右上角固定 📸 |

### 2.2 信息流向

```
品牌选择 → 约束活动分类范围（分类关联品牌）
活动日期类型 → 控制日期选择器显隐
可用周期类型 → 控制周/月选择器显隐
可用时段类型 → 控制时间段选择器显隐
储值协议Toggle → 控制协议选择区域显隐
赠送礼品类型 → 控制「+ 加入礼品」按钮显隐
档位数据 → 汇总到套餐整体
```

### 2.3 负空间

- 未见「适用门店/组织范围」选择，可能在「购买规则」步骤中或后台默认全部门店 [🤖 AI 推断]
- 未见「适用会员等级」筛选条件 [🤖 AI 推断]
- 未见「活动图片/封面」上传 [🤖 AI 推断]
- 底部「我已阅读并同意《法律声明》」暗示保存前需勾选同意 📸

## 三、数据模型

### 3.1 数据容器识别（UI层级 → 表层级映射）

```
[M] 主实体：「新增储值套餐」→ stored_value_package
  [D] 明细1：「档位」→ stored_value_tier（最多5个）
    [P] 子实体：「加入礼品」→ tier_gift（FK→stored_value_tier）
  [D] 明细2：「可用时段」→ package_available_period（可多条，指定时段时）
  [D] 明细3：「排除日期」→ package_exclude_date（可多条）
  [J] 关联1：「品牌」→ package_brand（多选）
  [J] 关联2：「储值协议」→ package_agreement（多选）
[REF] 活动分类(activity_category) — 仅存 category_id
[REF] 储值协议(user_agreement) — 仅存关联ID
[REF] 优惠券(coupon) — 仅存 coupon_id（礼品引用）
```

### 3.2 实体层级与关系图

```
储值套餐(stored_value_package) [M][OWN]
  PK: id
  数量约束: -
  UI来源: 整个页面主表单

  ├─ 储值档位(stored_value_tier) [D][OWN]
  │   PK: id
  │   FK: package_id → stored_value_package
  │   数量约束: 最多5个
  │   UI来源: 「储值推荐规则」区域的档位卡片
  │
  │   └─ 档位礼品(tier_gift) [P][OWN]
  │       PK: id
  │       FK: tier_id → stored_value_tier
  │       数量约束: 不限
  │       UI来源: 「加入礼品」弹窗
  │
  ├─ 可用时段(package_available_period) [D][OWN]
  │   PK: id
  │   FK: package_id → stored_value_package
  │   数量约束: 不限（指定时段时）
  │   UI来源: 「可用时段」-「指定时段」下的时间段
  │
  ├─ 排除日期(package_exclude_date) [D][OWN]
  │   PK: id
  │   FK: package_id → stored_value_package
  │   数量约束: 不限
  │   UI来源: 「排除日期」多选日期标签
  │
  ├─ 套餐品牌关联(package_brand) [J][OWN]
  │   PK: id
  │   FK: package_id → stored_value_package
  │   FK: brand_id → brand
  │   UI来源: 「品牌」多选标签选择器
  │
  └─ 套餐协议关联(package_agreement) [J][OWN]
      PK: id
      FK: package_id → stored_value_package
      FK: agreement_id → user_agreement
      UI来源: 「请选择协议」多选下拉框

品牌(brand) [REF] — 仅存 brand_id
活动分类(activity_category) [REF] — 仅存 category_id
用户协议(user_agreement) [REF] — 仅存 agreement_id
优惠券(coupon) [REF] — 仅存 coupon_id
```

| effect_time_type | tier_gift | =99显示自定义天数输入 👤 | =99时effect_custom_days必填 |
| count_as_revenue | tier_gift | =1显示实收金额下拉 👤 | =0时revenue相关字段全NULL |
| revenue_type | tier_gift | =1无额外输入；=2百分比；=3金额 👤 | 控制revenue_percent或revenue_amount |

### 3.6 跨区依赖

```
「品牌」→「活动分类」：分类关联品牌，品牌变更可能影响分类可选范围 🤖
「储值协议」Toggle → 「请选择协议」+「+ 添加储值协议」：开关控制整个协议区 📸
```

### 3.7 保存事务边界

事务1「保存储值套餐」：
- stored_value_package (INSERT 或 UPDATE)
- stored_value_tier (DELETE+INSERT 或逐条对比)
- tier_gift (DELETE+INSERT，随 tier 一起保存)
- package_available_period (DELETE+INSERT)
- package_exclude_date (DELETE+INSERT)
- package_brand [J] (DELETE+INSERT)
- package_agreement [J] (DELETE+INSERT)

> 所有表在同一个事务中提交，前端点击「保存」= 后端一个事务。

### 4.1 UI联动规则（RULE穷举）

| 编号 | 条件 | 效果 | 默认 | 来源 |
|------|------|------|------|------|
| RULE-01 | WHEN date_type=指定日期 | THEN 显示开始/结束日期选择器 | 显示 | 📸 |
| RULE-02 | WHEN date_type=永久有效 | THEN 隐藏日期选择器 | - | 👤 |
| RULE-03 | WHEN cycle_type=按日 | THEN 无额外选择器 | 默认 | 📸 |
| RULE-04 | WHEN cycle_type=按周 | THEN 显示周一~周日多选按钮 | - | 👤 |
| RULE-05 | WHEN cycle_type=按月 | THEN 显示1号~31号多选按钮 | - | 👤 |
| RULE-06 | WHEN time_type=全天 | THEN 无时间段选择 | 默认 | 📸 |
| RULE-07 | WHEN time_type=指定时段 | THEN 显示时间范围+「+」添加按钮 | - | 👤 |
| RULE-08 | WHEN agreement_enabled=开启 | THEN 显示协议下拉+添加按钮 | 默认 | 📸 |
| RULE-09 | WHEN agreement_enabled=关闭 | THEN 隐藏协议区域 | - | 👤 |
| RULE-10 | WHEN 勾选「优惠券」 | THEN 取消「不赠送」，显示「+ 加入礼品」 | - | 👤 |
| RULE-11 | WHEN 勾选「不赠送」 | THEN 取消「优惠券」「券包」，隐藏加入按钮 | 默认 | 📸 |
| RULE-12 | WHEN validity_type=相对有效期 | THEN 显示有效天数输入 | 默认 | 👤 |
| RULE-13 | WHEN validity_type=固定有效期 | THEN 显示日期范围选择器 | - | 👤 |

## 七、风险与安全

### 7.1 并发争用

- 同一套餐多人同时编辑：使用乐观锁（update_time 或 version 字段），提交时检查版本冲突 🤖
- 新增场景无并发争用

### 7.2 数据安全

- 多租户隔离：按商户（广州赛群电子科技有限公司）过滤，所有查询需带租户 ID 🤖
- 金额字段应防止负值注入，后端二次校验 🤖

### 7.3 幂等

- PUT 编辑接口天然幂等（全量替换）
- POST 新增接口需防重复提交：前端按钮防抖 + 后端幂等 Key 🤖

## 八、生命周期

### 8.1 状态机 [⚠️ 假设]

```
草稿(DRAFT) → 启用(ACTIVE) → 停用(INACTIVE)
                                    ↓
                              启用(ACTIVE)（可重新启用）
```

### 8.2 权限矩阵 [⚠️ 假设]

| 字段/操作 | 草稿 | 启用 | 停用 |
|---------|------|------|------|
| 所有字段 | 可编辑 | 部分可编辑 | 只读 |
| 档位 | 可增删改 | 不可修改 | 不可修改 |
| 删除套餐 | 可 | 不可（需先停用） | 可 |

### 8.3 审计

- 必审字段：活动名称、储值金额、赠送金额变更、状态变更 🤖
- 理由：涉及资金和用户权益 🤖

### 8.4 通知

- 套餐启用时通知相关门店 [⚠️ 假设]
- 套餐到期前提醒管理员 [⚠️ 假设]

## 九、工程维度

### 9.1 性能

- 热路径：保存操作涉及 7 张表，需在同一事务内完成，注意事务时间
- 档位最多 5 个，每个礼品数量有限，数据量可控
- 品牌/分类/优惠券列表可缓存

### 9.2 扩展性

- rule_type 预留扩展（当前仅"固定金额档位"，未来可能增加"自定义金额"）
- amount_limit_type 预留扩展
- gift_source_type 预留扩展（优惠券/券包之外可能增加实物礼品）

### 9.3 埋点

- 保存成功/失败次数
- 各档位配置分布统计
- 「+ 加入礼品」使用率
