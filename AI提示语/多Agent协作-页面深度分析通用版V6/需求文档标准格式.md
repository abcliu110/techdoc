# 需求文档标准格式（契约规范）

> 本文档定义需求分析产出文档的标准格式。
> 它是需求分析 prompt 和代码生成器之间的"类型系统"——
> 需求分析的输出必须符合此格式，代码生成器的输入也基于此格式解析。
>
> 同一份符合此格式的需求文档，可对接任何技术栈的代码生成器。

---

## 设计哲学

```
需求分析 prompt ──产出──→ 需求文档（遵循本格式） ──输入──→ 代码生成器
     ↑                        ↑                           ↑
  "前端编译器"            "中间表示 IR"              "后端编译器"
  (UI截图→IR)          (技术无关的精确规范)        (IR→目标代码)
```

本格式的设计目标：
1. **精度完备**：包含代码生成所需的全部信息（7 个核心数据结构）
2. **技术无关**：不绑定任何编程语言、框架或数据库
3. **可机械验证**：每个章节的内容可以被程序化地交叉校验
4. **单向依赖**：后面的章节引用前面的章节，不反向引用

---

## 7 个核心数据结构

从代码生成的角度倒推，一份完整的需求文档必须包含以下 7 个核心数据结构。
**缺一不可**——少一个就无法完成代码生成。

| 序号 | 数据结构 | 所在章节 | 代码生成用途 |
|------|---------|---------|------------|
| 1 | 实体层级图 | 3.2 | 确定需要生成哪些实体、它们的父子关系和外键 |
| 2 | 字段字典 | 3.3 | 确定每个实体的字段名、类型、约束 |
| 3 | 枚举值定义表 | 3.4 | 生成枚举类/常量 |
| 4 | 保存事务边界 | 3.7 | 确定事务范围和写入策略 |
| 5 | 三态接口数据结构 | 4.5 | 生成输入/输出数据对象的嵌套结构 |
| 6 | 规则执行层分类 | 4.4 | 确定每条规则在哪一层实现 |
| 7 | API 概要设计 | 6.5 | 确定接口路径、方法、幂等性 |

## 各章节格式规范

### 3.2 实体层级与关系图

```
{实体中文名}({英文表名}) [{容器类型}][{归属类型}]
  PK: {主键字段}
  FK: {外键字段} → {父表名}
  数量约束: {1:N 的 N 上限}
  UI来源: {对应的页面区域/弹窗名称}
```

**容器类型**：[M] 主表单 | [D] 可重复明细 | [P] 弹窗子实体 | [G] 弹窗内表格 | [J] 多选关联
**归属类型**：[OWN] 本页面创建管理 | [REF] 仅引用已有数据

用树形缩进表示父子关系。示例：
```
储值套餐(stored_value_package) [M][OWN]
  └─ 储值档位(stored_value_tier) [D][OWN]
      └─ 档位礼品(stored_value_tier_gift) [P][OWN]
品牌(brand) [REF] — 仅存 brand_id
```

### 3.3 字段字典

每个 [OWN] 实体单独一张表：

```
#### 表名：{中文名}({英文表名})
父表：{父表名} | 关系：{1:N / N:N} | UI来源：{页面区域}

| 字段名 | 列名(英文) | 数据类型 | 必填 | 默认值 | 说明 | 写入方 | 判别控制 |
|--------|-----------|---------|------|--------|------|--------|---------|
```

**写入方**取值：
- **用户输入**：用户在表单中手动填写/选择
- **后端自动**：后端根据上下文自动填入（如create_by取当前用户）→ 不出现在请求体(POST/PUT)中
- **引用携带**：选择引用实体后自动带入的快照值 → 标注来源实体
- **系统生成**：系统自动生成的值（如自增编号）→ 不出现在请求体中

**字段归属规则**（字段字典中只包含以下两类字段）：
- **自有字段**：用户手动输入/选择的值 → 正常建模
- **快照字段**：选择后填充、需冻结的值 → 建模并在说明列标注「冗余快照，来源={引用实体}」

**不得出现**在字段字典中的：
- **携带字段**：选择引用实体后自动填充的只读展示信息（如选券包后显示的子券名称）
  → 当前表只存引用ID（如 coupon_package_id），携带字段在响应体中通过关联查询返回

**数据类型规范**（通用SQL标准，不含任何数据库方言）：

| 类型 | 格式 | 用途 |
|------|------|------|
| VARCHAR(N) | 标注最大长度 N | 变长字符串 |
| INTEGER | 无需标注位数 | 整数、枚举值 |
| BIGINT | 无需标注位数 | 主键、外键 |
| DECIMAL(p,s) | 标注总位数 p 和小数位 s | 精确小数（金额等）|
| BOOLEAN | 无需标注 | 布尔值 |
| DATE | 无需标注 | 日期 |
| TIMESTAMP | 无需标注 | 时间戳 |
| TEXT | 无需标注 | 长文本 |

**必须包含的字段**（每个 [OWN] 实体）：
- id (BIGINT, PK)
- 外键字段（如 package_id BIGINT）
- create_time (TIMESTAMP)
- update_time (TIMESTAMP)
- create_by (BIGINT)
- deleted (BOOLEAN)

### 3.4 枚举值定义表

```
| 字段名 | 所属表 | 枚举值 | 含义 | 说明 |
|--------|-------|--------|------|------|
```

规则：
- 每个 Radio、单选下拉框（固定枚举）、Toggle 字段必须列入
- 数值从 0 或 1 开始
- 不用中文作为存储值

### 3.7 保存事务边界

```
事务1「{操作名}」：
  - {实体中文名} — 新增或更新
  - {子实体} — 随主实体整体替换（先清后写）
  - {关联实体} — 关联 ID 列表整体替换
```

**技术无关**：仅描述业务语义，不指定 INSERT/UPDATE/DELETE 等 SQL 动词。

### 3.9 逻辑建表预览

使用通用 SQL 标准语法，**不得包含**以下数据库方言：
- ❌ AUTO_INCREMENT（改用注释"自增策略由实现决定"）
- ❌ ENGINE=InnoDB
- ❌ COMMENT='...'（改用 SQL 标准 `--` 注释）
- ❌ ON UPDATE CURRENT_TIMESTAMP
- ❌ TINYINT（改用 INTEGER 或 BOOLEAN）
- ❌ DATETIME（改用 TIMESTAMP）

**必须保留**的精度：
- ✅ 字段名和英文列名
- ✅ 类型精度（VARCHAR(100), DECIMAL(10,2)）
- ✅ NOT NULL 约束
- ✅ DEFAULT 值
- ✅ 索引设计（CREATE INDEX）
- ✅ 字段注释（用 -- 行注释）

### 4.4 规则执行层分类

```
| 规则编号 | 规则描述 | 前端UI层 | 后端业务层 | 数据存储层 | 异步任务层 | 说明 |
|---------|---------|---------|-----------|-----------|-----------|------|
```

层级定义（技术无关）：
- 前端UI层：界面渲染逻辑（显隐、禁用、选项联动）
- 后端业务层：服务端业务校验和逻辑处理
- 数据存储层：表结构约束（NOT NULL、UNIQUE、CHECK）
- 异步任务层：定时任务、事件处理

### 4.5 保存接口数据结构预览

必须同时输出三种结构：
- A. 新增请求体 (POST)
- B. 编辑请求体 (PUT)
- C. 详情响应体 (GET)

JSON 结构本身是技术无关的。注释中不得出现 DTO/VO 等框架术语，改用：
- "数据对象"
- "请求体"
- "响应体"
- "子对象列表"

## 交叉验证矩阵

需求文档输出后，必须执行以下交叉验证确保一致性：

```
            字段字典  枚举表  ER图   建表预览  接口结构  规则分类
字段字典      -       ✓字段   ✓表    ✓列      ✓字段    ✗
枚举表       ✓字段    -      ✗      ✓注释    ✓类型    ✗
ER图         ✓表     ✗       -      ✓FK     ✓嵌套    ✗
建表预览     ✓列     ✓注释   ✓FK    -       ✗       ✓约束
接口结构     ✓字段   ✓类型   ✓嵌套  ✗        -       ✗
规则分类     ✗       ✗      ✗      ✓约束    ✗        -

✓ = 必须交叉验证一致    ✗ = 无直接关联
```

关键验证点：
1. **字段字典 ↔ 建表预览**：每个字段在两处的类型、可空性、默认值必须一致
2. **枚举表 ↔ 建表预览**：枚举字段的注释中必须包含完整枚举值说明
3. **ER图 ↔ 接口结构**：实体层级嵌套关系必须完全对应
4. **规则分类 ↔ 建表预览**：标注为"数据存储层"的规则必须体现为表约束
5. **字段字典 ↔ 携带字段**：字段字典中不得出现携带字段（只存引用ID），快照字段必须标注来源
6. **字段字典 ↔ 计算字段**：计算字段不入字段字典，在4.3章节定义公式；若需冗余存储须标注公式
7. **有序列表 ↔ sort_order**：所有可拖拽排序/有序号的明细表必须有 sort_order 字段
8. **列表页筛选 ↔ 索引设计**：列表页的每个筛选条件字段必须在建表预览中有对应索引
9. **行操作按钮 ↔ 状态机**：列表页行操作按钮集合必须与状态机(8.1)的转换动作一致

## 标记规范

| 标记 | 含义 | 处理方式 |
|------|------|---------|
| 📸 截图可见 | 信息来源于截图 | 高置信度 |
| 👤 用户确认 | 信息来源于用户回答 | 高置信度 |
| 🤖 AI 推断 | AI 基于上下文推断 | 需验证 |
| [⚠️ 假设] | 未确认的假设 | 代码生成时加注释标注 |
| [🔍 补漏] | 交叉验证发现的缺失 | 需补充完善 |

## 编号规范

| 编号前缀 | 用途 | 示例 |
|---------|------|------|
| RULE- | UI联动规则 | RULE-01: WHEN date_type=1 THEN show start_date |
| INV- | 不变量 | INV-01: tiers.length >= 1 |
| VAL- | 校验规则 | VAL-01: start_date <= end_date |
| C- | 约束 | C-01: activityName VARCHAR(100) NOT NULL |
| API- | API 编号 | API-01: POST /api/packages |
| AC- | 验收标准 | AC-01: GIVEN ... WHEN ... THEN ... |

## 版本兼容性

本格式为 v1.0。后续版本承诺：
- 新增章节不影响已有章节的解析
- 已有字段不改名、不删除
- 新增字段提供默认值

代码生成器应对未识别的章节做静默跳过处理，不报错。
