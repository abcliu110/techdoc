# 代码规范：Spring Boot + MyBatis-Plus + 阿里巴巴 Java 开发规范

可插拔代码规范文件，供代码生成器读取。替换为其他规范文件可对接不同技术栈。

## 技术栈

Java 17+ / Spring Boot 3.x / Spring Cloud（如涉及） / MyBatis-Plus / Springdoc OpenAPI (io.swagger.v3.oas.annotations) / MapStruct / Maven或Gradle / MySQL 8.x

## 架构哲学（所有规则的源头，未覆盖场景回到此处判断）

P-1 单向依赖，禁止逆流
  Controller → Service → Mapper/Repository，下层不反向依赖上层。Service间可互调但禁止循环依赖。Controller不出现Mapper；Entity不出现DTO/VO。

P-2 契约驱动，代码即文档
  前后端唯一契约=DTO/VO结构+@Schema注解。写完代码=写完文档=Apifox可直接导入。联调零口头沟通。

P-3 显式优于隐式
  枚举值写明含义；条件校验写在代码里；类型转换在Convert层；错误提示是完整中文句子。

P-4 变更局部化
  改一个业务规则只改一个文件。枚举/校验/转换逻辑集中。禁止同一逻辑散布三层。

P-5 防御性编码，快速失败
  入口@Valid+JSR-303拦截非法数据→Service校验业务规则→数据库约束是最后防线非唯一防线。

P-6 安全零信任
  不信任前端任何数据（ID可能篡改、枚举可能越界、身份可能伪造）。每层假设上层可能失守。数据归属必须校验。

P-7 可演进性
  枚举可加值不改旧值、接口可加字段不删旧字段、表可加列不改旧列。向后兼容为默认，破坏兼容需显式版本号。

## 分层职责（严格执行，不可越层）

Controller层：接收请求→参数校验(@Valid)→调用Service→包装响应。禁止写业务逻辑、直接调Mapper、做数据转换。Controller是薄的"接线员"。
  ↓ 只传DTO，不传Entity/VO
Service层：业务逻辑、事务管理、条件校验、调用Mapper/其他Service。所有业务规则唯一归属地。返回Entity或内部对象，不返回VO。
  ↓ 只操作Entity
Mapper层：数据访问，SQL执行。禁止写业务逻辑、抛业务异常。纯数据通道。

Convert层（横切）：DTO↔Entity↔VO对象转换（MapStruct）。★ 枚举code→label转换在此层统一处理。禁止BeanUtils.copyProperties（无编译期检查）。

## 包结构

com.{company}.{project}
  common/              ← 全局基础设施
    result/          ← Result、PageRespVO、ErrorCode
    exception/       ← BusinessException、GlobalExceptionHandler
    config/          ← 全局配置（Jackson、MyBatis-Plus、Swagger）
    util/            ← 工具类（严格限制数量）
  module/
    {业务模块}/       ← 如 storedvalue、order、member
      controller/ service/impl/ mapper/ entity/ dto/ vo/ convert/ enums/
  Application.java

★ 每个业务模块内部自治，模块间通过Service接口通信，不跨模块访问Mapper。

## 主键策略

MySQL AUTO_INCREMENT + @TableId(type = IdType.AUTO)

## 审计字段模板

每个[OWN]实体表必须包含：

```sql
create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
create_by BIGINT NOT NULL COMMENT '创建人ID',
deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除: 0=未删除 1=已删除'
```

对应Entity字段：
- createTime: LocalDateTime, @TableField(fill = FieldFill.INSERT)
- updateTime: LocalDateTime, @TableField(fill = FieldFill.INSERT_UPDATE)
- createBy: Long
- deleted: Integer, @TableLogic

## 物理模型优化规则

★ 从逻辑模型到物理模型的优化策略，用于「物理模型决策」步骤。
核心问题：MyBatis-Plus每张表=Entity+Mapper+Service+Controller全套，表过多导致启动慢、事务复杂、维护成本高。目标：每功能模块3-5张表。

合并为JSON的条件（满足任一即可合并）：
- 纯ID列表（多选关联ID）→ brand_ids JSON [1,2,3]
- 只有1-2个业务字段的子记录 → periods JSON [{"start":"09:00","end":"12:00"}]
- 简单值列表（日期、编码等）→ exclude_dates JSON ["2026-04-05"]
- 数量有限(<20)且不需独立查询
- 只随主表读写，不被其他模块引用

必须独立建表的条件（满足任一即不可合并）：
- 子记录有3+个业务字段（不含id/FK/审计字段）
- 需按子记录字段做条件查询或报表统计
- 子记录有独立状态或生命周期
- 子记录数量可能很大(100+)
- 子记录被其他模块通过FK引用

实现要点：
- JSON字段用MySQL JSON类型
- MyBatis-Plus用 @TableField(typeHandler = JacksonTypeHandler.class) 映射
- DTO/VO结构不变（前端仍提交嵌套JSON），只是存储层合并
- 合并的子数据不需DELETE+INSERT，直接更新父表JSON列
- 逻辑模型实体关系仍成立，只是物理存储方式变了

## 契约规范（前后端零沟通成本）

★ 后端写完→Apifox导入→前端直接开发，无需口头沟通。

统一响应：所有接口返回 Result.success(data) / Result.fail(ErrorCode.XXX)
```json
{"code": 0, "message": "操作成功", "data": { ... }}
```

DTO/VO分离（入参出参严格分开）：
- DTO接收请求：XxxCreateReqDTO / XxxUpdateReqDTO / XxxQueryReqDTO
- VO返回前端：XxxRespVO / XxxDetailRespVO
- 禁止：Entity直接暴露给前端；同一类既做入参又做出参

枚举值一致性（★ DTO与VO必须对称）：
- DTO入参枚举字段：Integer类型，字段名xxxCode（如dateTypeCode）
- VO出参枚举字段：同时返回xxxCode(Integer)+xxxLabel(String)
- ★ DTO的xxxCode与VO的xxxCode必须完全同名
- 嵌套DTO/VO中枚举字段递归遵循同样规则
- code→label转换统一在Convert层

字段序列化：
| 字段类型 | 处理方式 | 原因 |
|---------|---------|------|
| Long ID | @JsonSerialize(using = ToStringSerializer.class) | JS精度丢失 |
| BigDecimal | @JsonSerialize(using = ToStringSerializer.class) | JS精度丢失 |
| LocalDateTime | @JsonFormat(pattern="yyyy-MM-dd HH:mm:ss", timezone="GMT+8") | 格式统一 |
| 枚举 | xxxCode(Integer) + xxxLabel(String) | 前端直接展示 |
| Boolean | isXxx命名 | 语义清晰 |

分页规范：
- 列表查询继承PageReqDTO，响应用PageRespVO<T>包装
- PageReqDTO：pageNum(int,默认1)、pageSize(int,默认10,最大100)
- PageRespVO<T>：total(long)、list(List<T>)、pageNum(int)、pageSize(int)

API文档注解（不可省略）：
- Controller类：@Tag(name, description)
- Controller方法：@Operation(summary, description) + @ApiResponse
- DTO/VO类：@Schema(description)
- DTO/VO每个字段（无一例外）：@Schema(description, example, requiredMode)
- 枚举字段description必须列出所有值
- 嵌套对象说明约束（如"最多5个"）

错误码规范：
- 统一枚举管理，格式：模块编号(4位)+序号(3位)，如1001001
- 错误信息必须是完整中文句子
- 禁止硬编码错误信息字符串

## 代码质量规范

参数校验三道防线：
1. Controller入口（@Valid+JSR-303）：DTO加@NotNull/@NotBlank/@Size/@Min/@Max，message必须中文如@NotBlank(message="活动名称不能为空")
2. Service业务校验：条件必填(RULE规则)用if校验，冲突抛BusinessException(ErrorCode.XXX)
3. 数据库约束（最后防线）：NOT NULL、UNIQUE、外键

事务管理：
- 写操作：@Transactional(rollbackFor = Exception.class)
- 只读：@Transactional(readOnly = true)
- ★ @Transactional只加在Service实现类public方法上
- ★ 事务方法内禁止调外部HTTP/RPC
- ★ 大批量数据操作拆分小事务

数据访问（MyBatis-Plus）：
- 批量插入saveBatch()，批量更新updateBatchById()
- 删除子表用remove(Wrappers.<T>lambdaQuery().eq(T::getParentId, parentId))
- ★ 禁止SELECT *，查列表用select(字段列表)
- ★ 分页必须用MyBatis-Plus的Page对象
- ★ 防N+1查询：子表数据用IN批量查

并发安全：
- 库存/配额/余额：Redis INCR + LUA脚本保证原子性
- 关键数据：数据库乐观锁(@Version)或悲观锁(SELECT FOR UPDATE)
- 幂等接口：唯一requestId + Redis去重

异常处理：
- 业务异常：throw new BusinessException(ErrorCode.XXX)
- 系统异常：log.error()记完整堆栈后返回通用错误提示
- ★ 禁止空catch

日志规范：
- 类加@Slf4j，用{}占位符，禁止打印敏感信息
- 格式：log.info("[{操作名}] {关键参数}={},...", value)
- 必须打日志：Service写操作入口和完成、关键业务分支、状态变更（前后状态）、外部调用（调用前+后/异常）、异常处理（业务WARN+系统ERROR带堆栈）

注释规范：
- 必须有：类Javadoc、Entity对应表名、Service每个public方法Javadoc、枚举类用途
- 禁止：// getter setter、代码能自解释处不加注释

安全编码：
- 数据归属校验：UPDATE/DELETE先查数据→校验归属
- 多租户：Entity含tenant_id，查询自动拼接
- 敏感数据：VO返回脱敏格式
- 防注入：LambdaQueryWrapper优先，自定义SQL用#{}
- 接口防刷：写接口加频率限制

逻辑删除查询陷阱：
- 唯一性校验考虑已删除数据：UNIQUE(name, deleted)或删除时改名
- JOIN子表需手动加AND sub.deleted = 0

领域事件：
- applicationEventPublisher.publishEvent(new XxxEvent(id))
- @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)

依赖注入：
- ★ 必须构造器注入(@RequiredArgsConstructor)，禁止@Autowired字段注入
- 依赖超5个时考虑拆分Service
- 禁止@Lazy掩盖循环依赖

嵌套DTO校验传递：
```java
@Valid
@NotEmpty(message = "至少需要一个档位")
@Size(max = 5, message = "档位最多5个")
private List<TierCreateReqDTO> tiers;
```
★ 多层嵌套每层都要加@Valid

定时任务：多实例部署必须用分布式锁（Redisson/ShedLock），每次记录日志：开始+结束+处理数量。

接口幂等：
- 方案一：Token令牌（表单提交）— GET /api/idempotent/token + Header Idempotent-Token
- 方案二：业务唯一键 — UNIQUE约束 + catch DuplicateKeyException
- ★ 每个POST写接口必须选其中一种

CORS：前后端分离必须配置，生产环境allowedOriginPatterns配具体域名。

全局请求拦截：AccessLogInterceptor自动记录访问日志+慢接口告警(>3s)。

可演进性：
- 枚举：新增值加末尾，不改已有code
- 接口：新增可选字段，不删旧字段
- 数据库：加列给默认值，禁止删列/改列类型

## 微服务规范（如涉及）

服务间通信：
- 同步：OpenFeign，接口命名XxxClient
- 异步：消息队列(RocketMQ/RabbitMQ)，消费者做幂等
- 禁止微服务间共享数据库

弹性设计：远程调用必须配超时+关键调用加熔断+非关键调用加降级

配置管理：yml→profile yml→配置中心。敏感配置放配置中心或环境变量。用@ConfigurationProperties绑定。

健康检查与优雅停机：/actuator/health + server.shutdown=graceful

链路追踪：Micrometer Tracing，日志含traceId

## 各层生成规范

Entity层：
- @TableName("t_{业务名}")
- ID：@TableId(type = IdType.AUTO) + @JsonSerialize(using = ToStringSerializer.class)
- 必含：create_time、update_time(@TableField(fill=...))、deleted(@TableLogic)
- 多租户含tenant_id
- 加@Schema类注解+Javadoc
- ★ 每个[OWN]实体生成独立Entity
- ★ 子表Entity含父表外键字段
- ★ [REF]引用实体不生成Entity
- ★ 逻辑删除+唯一约束：UNIQUE(name, deleted)

DTO层：
- 每个类和字段加@Schema
- 枚举字段用Integer，字段名xxxCode
- 校验注解：BLOCK级→@NotNull/@NotBlank；范围→@Min/@Max/@Size
- message必须中文
- 条件必填在Service层校验
- ★ 嵌套结构按接口数据结构生成
- ★ UpdateReqDTO的明细列表元素含可选id

枚举层：
- ★ 按枚举值定义表为每个枚举字段生成独立枚举类
- 含：code(int)、label(String)、构造器、getByCode、getLabelByCode
- 加@Schema+Javadoc
- Entity中枚举字段用Integer

VO层：
- 每个类和字段加@Schema
- ★ 枚举字段同时返回xxxCode+xxxLabel
- xxxLabel加accessMode=READ_ONLY
- 详情VO嵌套子表VO的List

Convert层（MapStruct）：
- 每个[OWN]实体一个Convert接口
- ★ 枚举code→label转换在此层
- DTO→Entity、Entity→VO、Entity→DetailVO各一个方法

Service层：
- 写操作@Transactional(rollbackFor = Exception.class)
- 查询@Transactional(readOnly = true)
- ★ 保存按事务边界：主表操作→子表整体替换→关联表整体替换
- ★ 后端业务层规则在此实现
- ★ UPDATE/DELETE先查数据→校验归属→再操作

Controller层：@Tag+@Operation+@ApiResponse，参数加@Valid

## 建表脚本模板（MySQL方言）

通用SQL→MySQL翻译规则：
```sql
-- BIGINT PRIMARY KEY          → BIGINT AUTO_INCREMENT PRIMARY KEY
-- INTEGER NOT NULL DEFAULT 1  → TINYINT NOT NULL DEFAULT 1 （枚举字段用TINYINT）
-- INTEGER NOT NULL            → INT NOT NULL （普通整数用INT）
-- BOOLEAN NOT NULL DEFAULT FALSE → TINYINT NOT NULL DEFAULT 0
-- TIMESTAMP NOT NULL          → DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
-- 更新时间字段追加            → ON UPDATE CURRENT_TIMESTAMP
-- 字段注释                    → COMMENT '...'
-- 表注释                      → COMMENT='...'
-- 引擎                        → ENGINE=InnoDB DEFAULT CHARSET=utf8mb4

-- 示例：
CREATE TABLE t_stored_value_package (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  activity_name VARCHAR(100) NOT NULL COMMENT '活动名称',
  date_type TINYINT NOT NULL DEFAULT 1 COMMENT '活动日期类型: 1=指定日期 2=永久有效',
  start_date DATE NULL COMMENT '开始日期, date_type=1时必填',
  end_date DATE NULL COMMENT '结束日期, date_type=1时必填',
  status TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0=草稿 1=启用 2=停用',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  create_by BIGINT NOT NULL COMMENT '创建人ID',
  deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除: 0=未删除 1=已删除'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='储值套餐主表';

CREATE INDEX idx_svp_create_by ON t_stored_value_package(create_by);
```

## 代码生成顺序
