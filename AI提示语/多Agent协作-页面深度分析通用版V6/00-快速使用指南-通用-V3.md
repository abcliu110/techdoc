# 快速使用指南（通用版 V6）

## 30 秒了解 V3

V3 相比 V2 的核心变化：
- All-in-One 模式从「两轮交互」升级为 **逐题追问（一问一答）**，降低用户负担
- 支持用 **截图回答** 问题（如展开下拉框、点击按钮后截图），信息采集更精准
- 新增 **深度追踪机制**：Checkbox/Radio 切换后出现的新界面（弹窗、新字段等）会被**强制生成子清单并逐项追问**，确保二级界面的每个交互元素都不遗漏
- Prompt 结构简化，兼容低智能模型

V2 已有的能力保留不变：
- **Agent-0（信息补全者）**：在分析前通过结构化提问补全截图缺失的信息
- **Agent-6（红队审查员）**：在整合后对文档做对抗性攻击测试
- Agent-1/2/3 新增**推理备忘录**输出，保留不确定推断供下游参考
- 支持**反馈环**：Agent-6 发现问题可触发上游 Agent 定向重跑

---

## 适用场景

任何 B 端业务页面：订单管理、用户管理、商品管理、审批流程、报表配置、权限设置等。
不预设业务领域，AI 根据原型图自行识别业务语境。

## 快速模式：All-in-One（推荐新手使用）

**一个对话窗口、逐题追问**，即可完成从截图到完整需求文档的全流程。

### 与分步模式的对比

| 对比项 | 分步模式（5 个 Agent） | 快速模式（All-in-One） |
|--------|----------------------|----------------------|
| 对话窗口数 | 5 个 | **1 个** |
| 截图传递次数 | 5 次 | **1 次初始 + 按需补充** |
| 信息补全方式 | 批量提问 | **逐题追问，支持截图回答** |
| 手动复制粘贴 | 4 次交接 | **0 次** |
| 中间产物审核 | 可逐步审核 | 跳过，直出终稿 |

### 使用步骤

**第一步 — 发送截图：**
```
[粘贴 prompt-all-in-one-需求分析.txt 的内容]

这是 [系统名称] 的 [页面名称] 页面。
[附上截图]
[可选：补充业务背景]
```

AI 输出截图初扫结果，然后开始逐题追问（每次只问 1 个问题）。

**第二步 — 逐题回答：**

AI 每次只问 1 个问题，你可以用以下方式回答：
- 选字母（如 `A`）
- 文字说明
- **发截图**（如展开下拉框、点击按钮后的截图）
- 回复 `跳过` 表示不清楚
- 回复 `开始分析` 提前结束提问，进入分析阶段
- 回复 `暂停` 或 `下一步验证` 生成阶段进度包（含阶段状态码，可保存下次续写，模板见 `进度包模板.md`）

**第三步 — 获取文档：**

所有问题回答完毕（或提前结束）后，AI 直接输出完整需求分析文档（与 Agent-4 输出格式一致）。

**后续（如需）：**
- 代码生成 → 另开对话，用 `prompt-05-代码生成师.txt`
- 红队审查 → 另开对话，用 `prompt-06-红队审查员.txt`

### 适用场景

- 截图较完整、业务背景较清晰
- 想要通过截图补充交互细节（展开下拉框、切换 Tab 等）
- 想要快速出文档，不需要逐步审核中间产物
- 新手用户，希望减少操作步骤
- 需要“严格模式但允许中途验证与续写”的场景

### 不适用场景

- 需要精细控制每步分析结果 → 用下方分步模式
- 截图特别多、业务极复杂 → 上下文窗口可能不够
- 需要反馈环机制 → 必须用分步模式

> 详细说明见 `08-All-in-One-需求分析-通用-V6.md`

## 分步模式：完整执行链路

```
原型图 + 业务背景
    │
    ▼
Agent-0（信息补全者）── 产出：增强输入包（结构化补充信息）
    │
    ▼
Agent-1（业务分析师）── 产出：实体清单、字段定义、判别字段 + 推理备忘录
    │
    ▼
Agent-2（规则工程师）── 产出：联动规则、约束矩阵、不变量 + 推理备忘录
    │
    ▼
Agent-3（系统架构师）── 产出：API清单、状态机、并发方案 + 推理备忘录
    │
    ├──► Agent-4（文档整合者）── 产出：完整需求文档（给产品/测试/前端）
    │
    └──► Agent-5（代码生成师）── 产出：可运行的 Spring Boot 代码
              │
              ▼
Agent-6（红队审查员）── 产出：风险报告 + 反馈环指令
              │
              ▼（如有 🔴 Critical 发现）
         反馈环 → 重跑上游 Agent → 重跑 Agent-4 → 重跑 Agent-6
```

- Agent-0 在最前面执行，产出增强输入包供后续所有 Agent 使用
- Agent-1 → 2 → 3 串行执行，每步传**交接摘要 + 推理备忘录**给下游
- Agent-4 和 Agent-5 可以**并行执行**，输入都是 Agent-1+2+3 的输出
- Agent-6 在 Agent-4 完成后执行，对需求文档进行对抗性审查

## 反馈环操作指南

当 Agent-6 的报告中包含 FEEDBACK 指令时：

1. **按优先级排序**：P0 必须执行，P1 建议执行，P2 可选
2. **重跑目标 Agent**：在对应 Agent 的对话中追加 FEEDBACK 内容
   ```
   以下是红队审查员的反馈，请针对性补充分析：

   FEEDBACK-001:
     类型：补充分析
     内容：{具体内容}
   ```
3. **重跑 Agent-4**：将补充后的输出重新发给 Agent-4 整合
4. **重跑 Agent-6**：验证修复是否到位
5. **最多 2 轮**：反馈环最多执行 2 轮，防止无限循环

## 灵活组合

不是每次都需要跑完全部步骤：

| 目标 | 使用 Agent |
|------|-----------|
| **快速出需求文档** | **All-in-One（1 个对话搞定）** |
| 只需信息梳理 | Agent-0 |
| 只需数据库表设计 | Agent-0 → Agent-1 |
| 只需接口文档 | Agent-0 → Agent-1 → Agent-3 |
| 只需校验规则 | Agent-0 → Agent-1 → Agent-2 |
| 只要需求文档 | Agent-0 → 1 → 2 → 3 → 4 |
| 只要后端代码 | Agent-0 → 1 → 2 → 3 → 5 |
| 文档 + 代码都要 | Agent-0 → 1 → 2 → 3 → 4（并行）5 |
| 完整 + 对抗验证 | Agent-0 → 1 → 2 → 3 → 4（并行）5 → 6 |

## 常见问题

### Q: Agent-0 可以跳过吗？

可以，但不推荐。跳过 Agent-0 意味着：
- 下游 Agent 会产生更多 `[⚠️ 假设]` 标记
- Agent-6 红队审查时会发现更多信息缺口
- 最终可能还是要回来补信息，不如一开始就做

如果你的截图非常完整（覆盖所有状态、所有 Tab），且你对业务非常熟悉，可以自己写一份简化版的增强输入包替代 Agent-0。

### Q: 推理备忘录太长怎么办？

推理备忘录是给下游 Agent 的"参考资料"，不是必读内容。如果上下文窗口紧张：
- 优先传递交接摘要（必须）
- 推理备忘录只传递标记为"高置信度"和"异常信号"的条目
- Agent-4 整合时会自动筛选高价值条目

### Q: 反馈环要跑几轮？

通常 1 轮就够。设计上限为 2 轮，防止无限循环。如果 2 轮后仍有 🔴 发现，说明输入信息本身不足，应该回到 Agent-0 补充信息而非继续循环。

### Q: 能用同一个对话窗口完成吗？

**可以！使用 All-in-One 模式**（`prompt-all-in-one-需求分析.txt`），一个对话、逐题追问即可完成需求文档。详见上方「快速模式」章节。

如果使用分步模式，则不推荐在同一对话中运行多个 Agent，原因：
- 上下文窗口会被前面的输出占满，后面 Agent 的质量下降
- 分开对话可以并行
- 分开对话方便单独重跑某个 Agent（特别是反馈环场景）
