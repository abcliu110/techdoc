1.5 数据容器识别（面向数据建模，必须执行）

扫描页面，识别5类数据容器并标注层级关系。
★ 核心认知：UI的嵌套层级 = 数据实体的主从层级。

★ UI模式→数据模式 速查规则（机械执行）：

  文本输入框         → VARCHAR(N), 标注最大长度
  数字输入框         → INTEGER / DECIMAL(p,s), 标注精度
  日期选择器         → DATE / TIMESTAMP
  单选Radio          → INTEGER枚举列 + 枚举值定义表条目
  单选下拉框         → INTEGER枚举列 或 外键→引用表
  多选下拉框(标签)   → 关联表[J], 每个选中值一行
  多选Checkbox       → 关联表[J], 每个选中值一行
  ★ 下拉框单/多选判断：
    已选值为标签(tag)形式（带×删除）= 多选 → 关联表[J]
    已选值为纯文本 = 单选 → 枚举列或外键
    无法判断 → 必须列入清单向用户确认
  ★ 下拉框特殊值规则：
    选项含「自定义」「其他」「不限」「指定XXX」等 → 可能触发额外UI
    每个特殊值 = 潜在额外数据字段 → 必须逐个切换确认
  开关Toggle         → BOOLEAN
  时间范围(可多条)   → 时间段子表[D], 外键→父表
  多日期选择器       → 日期子表[D], 外键→父表
  可重复卡片(+添加)  → 明细表[D], 外键→父表
  按钮触发的弹窗     → 子实体表[P], 外键→触发区域所属表
  弹窗内可编辑表格   → 嵌套子表[G], 外键→弹窗表
  文件上传(多文件)   → 附件表[D], 外键→父表
  富文本编辑器       → TEXT

  ★ 高级UI模式识别（常见但易遗漏）：

  树形选择器（分类/部门/区域等带层级的选择）
    → 引用实体有 parent_id 自引用外键，形成树结构
    → 当前表只存叶子节点ID（如 category_id）
    → 若支持多选树节点 → 关联表[J]
    → 必须追问：选择的是叶子节点还是任意层级？是否支持多选？

  搜索型输入框（带搜索/自动补全的输入框，看似文本框实为引用选择器）
    → ★ 这不是文本字段！它产出的是引用实体的ID（如 customer_id）
    → 识别特征：有搜索图标、输入时弹出下拉候选、选中后显示名称但实际存ID
    → 建模：当前表存外键ID → 引用实体[REF]
    → ❌ 错误：建成 VARCHAR 存名称文本

  可切换的表单区域（选择类型后，表单字段完全不同）
    → 多态实体模式：同一个表根据 type 字段呈现完全不同的字段集
    → 必须追问：共有多少种类型？每种类型有哪些独有字段？
    → 建模选择（列入未决问题，由代码生成阶段决定）：
      - 单表模式：所有类型字段放同一张表，非当前类型字段允许NULL
      - 多表模式：公共字段一张表 + 每种类型一张扩展表
    → 需求文档中：为每种类型分别列出字段集，标注「多态：共有字段」和「多态：{类型名}独有」

  向导/分步表单（Stepper，多步骤完成一个实体的创建）
    → 所有步骤最终创建同一个实体，不要每步建一张表
    → 必须追问：每一步保存还是最后一步统一保存？是否有草稿/暂存？能否返回上一步修改？
    → IF 每步保存 → 需要 status 字段标记完成到第几步
    → IF 最后统一保存 → 请求体包含所有步骤字段，验证在最后一步执行

  文件/图片上传区域
    → 需确认关联时机：先上传后关联（文件先存到临时区，保存主表时关联）还是随主表一起上传
    → 单文件 → 主表直接存 file_url VARCHAR(500)
    → 多文件 → 附件子表[D]（id, 主表外键, file_name, file_url, file_size, sort_order）
    → 必须追问：文件类型限制？大小限制？是否需要预览？

  自动推导链（技术无关）：
    [D]/[P]/[G] → 需要外键 → 在父实体事务内 → 保存接口中嵌套为子对象列表
    [J]         → 需要关联表 → 随主表整体替换（先清后写）→ 保存接口用ID列表
    枚举列      → 枚举值定义表 → 判别字段检查 → 代码生成时映射为对应语言枚举类型

5类数据容器：
[M] 主表单区域 — 页面标题含"新增/编辑"、整个表单围绕同一业务对象
[D] 可重复明细 — 有「+添加」「删除」按钮、有序号、卡片式可复制
[P] 弹窗子实体 — 弹窗内有自己的表单字段和保存/确定按钮
[G] 弹窗内表格 — 弹窗内有表格结构，行可编辑
[J] 多选关联   — 多选下拉框、Checkbox组、标签选择器等产生的N:N关系

★ 携带字段识别规则（强制执行）：

用户选择/关联已有实体后，UI可能自动填充该实体属性。这些叫「携带字段」——来自引用实体[REF]，不属于当前实体。

识别方法：
  灰色/只读/不可编辑        → 携带字段（选择时自动带入）
  选择后自动填充、不可修改   → 携带字段
  选择后自动填充、可修改     → 需确认：携带后可覆盖 还是 独立字段

建模规则：
  ✅ 当前表只存引用ID（如coupon_package_id），携带字段不入当前表
  ❌ 把携带字段建到当前表（造成数据冗余和不一致）
例外（必须向用户确认）：
  - 快照需求：值需"冻结"（如订单中商品价格=下单时价格）→ 标注「冗余快照」
  - 性能需求：高频查询避免JOIN → 标注来源

★ 每个"选择后自动填充"的场景，必须列入交互探索清单逐个确认。

★ UI→数据模型 9类深度识别规则（逐条机械检查）：

TRAP-1 计算字段 vs 存储字段
  识别：值=其他字段的函数（合计=数量×单价）| 父级汇总（"共3个档位"）
  规则：计算字段不入字段字典，在4.3章定义公式，响应体中实时计算返回
  例外：快照金额、高频查询聚合值 → 标注「冗余计算值，公式={xxx}」

TRAP-2 隐含字段 — UI上看不见但必须存在
  强制补充清单：
    有序列表（可拖拽/有序号）     → sort_order
    存在"启用/停用"操作          → status
    有"版本号"或"最后修改"显示   → version乐观锁
    自动生成编号（如PKG-001）     → code → 追问生成规则
    多租户系统                    → tenant_id
  ★ 每个隐含字段列入交互探索清单向用户确认

TRAP-3 一个UI控件 → 多个数据库字段
  常见模式：
    日期范围选择器 → start_date + end_date
    地址选择器     → province + city + district + address_detail
    金额+币种     → amount + currency
    时间段控件     → start_time + end_time

TRAP-4 列表页筛选/排序 → 索引设计线索
  映射：
    筛选条件下拉框/输入框   → WHERE字段 → 需要索引
    可排序表头（排序箭头）   → ORDER BY → 可能需索引
    搜索框                  → 模糊查询LIKE → 注意性能
    Tab切换（全部/启用/停用）→ status筛选
  建模影响：高频筛选字段→建表预览加索引，组合筛选→联合索引

TRAP-5 表格行操作按钮 → 状态机 + API线索
  按钮→推导：
    编辑/删除       → 基础CRUD
    启用/停用       → 状态机转换
    复制            → 深拷贝API（含子表？）
    审批/提交/驳回  → 审批流状态机
    导出            → 导出API + 异步任务
  ★ 根据按钮组合推导完整状态机（8.1章）

TRAP-6 显示格式 ≠ 存储格式
  常见误判：
    显示"¥1,000.00"    → 存DECIMAL(10,2)，不存字符串
    显示"138****5678"   → 存完整手机号，展示时脱敏
    显示"张三(管理员)"  → 存user_id，展示时关联查询
    显示百分比"85%"     → 存DECIMAL原始值0.85
  规则：字段字典中的数据类型 = 存储类型，不是显示格式

TRAP-7 同一字段在不同状态下语义不同
  常见模式：
    新增可编辑→编辑时只读       → 字段权限矩阵(8.2)
    草稿可改→提交后锁定         → 状态驱动的字段锁定
    普通用户不可见→管理员可见    → 角色驱动的字段可见性
  ★ 遇到编辑页面必须追问：哪些字段变只读？哪些被锁定？

TRAP-8 批量操作 → 额外API和数据流
  按钮→推导：
    全选Checkbox → 批量删除/批量状态变更API
    导入按钮     → 批量创建API → 追问模板格式、校验规则、失败处理
    导出按钮     → 导出API（大数据量需异步）
    拖拽排序     → 批量更新排序API
  ★ 批量操作影响事务设计（分批提交）和错误处理策略

TRAP-9 跨实体约束 — 规则跨越父子或兄弟实体
  问题：有些业务规则不在单个实体内部，而是跨越多个实体
  常见模式：
    父子约束：所有档位金额之和不得超过套餐上限
    兄弟约束：同一套餐下的档位金额不能重复
    引用约束：选择的优惠券必须在有效期内
    数量约束：明细行数不能超过N条（这是集合级约束，不是字段级）
  ★ 在约束矩阵中标注为「跨实体」，在规则执行层分类中归入「后端业务层」
  ★ 每发现一个"整体不能超过""不能重复""至少N条"的描述，就是跨实体约束

TRAP-10 有效期/时间版本 — 同一实体不同时段有不同值
  问题：价格、费率、规则等可能按时间段变化，不是简单的一个值
  识别：UI中出现"生效日期""失效日期""版本"或"历史记录"Tab
  建模选择：
    简单场景：实体本身有 effective_date + expiry_date → 同一实体可有多条不同时段的记录
    复杂场景：需要版本表（entity_id + version + effective_date + 值字段）
  ★ 必须追问：历史版本是否需要保留？新版本生效时旧版本如何处理？

TRAP-11 条件性实体创建/销毁 — 选择某选项时创建子实体，取消时删除
  问题：不仅是字段的显隐，而是整个子实体的存在性取决于父实体的某个判别字段
  示例：勾选"赠送礼品"→ 创建礼品子实体；取消勾选 → 删除所有礼品子记录
  与TRAP-7的区别：TRAP-7是字段级的显隐，这是实体级的存在性
  建模影响：
    - 判别字段汇总(2-6)中必须标注"控制{子实体}的存在性"
    - 保存逻辑：判别字段变为"不赠送"时，需清除所有礼品子记录
    - 字段字典中子实体的外键不能设 NOT NULL（因为父实体可能没有子记录）

TRAP-12 数据流方向 — 谁是数据的主人
  问题：页面上展示的数据，必须搞清楚"谁写入、谁读取、谁是权威来源"
  常见误判：
    列表页展示的字段 ≠ 当前实体的字段（可能JOIN查询来的）
    表单中"所属部门"是当前用户自动带入 还是 手动选择的？
    "创建人"字段是后端自动写入 还是 前端可选择？
  建模影响：
    后端自动写入的字段 → 不出现在请求体(POST/PUT)中，只出现在响应体(GET)中
    前端传入但后端覆盖的字段（如create_by强制取当前用户）→ 请求体可选或不传
  ★ 在字段字典中增加"写入方"标注：用户输入 / 后端自动 / 引用携带

输出格式（缩进表示父子关系）：

## 数据容器识别

[M] 主实体：「{页面标题}」→ {建议表名}
  [D] 明细1：「{区域标题}」→ {建议表名}（{数量约束}）
    [P] 子实体1.1：「{弹窗标题}」→ {建议表名}
      [G] 嵌套表格1.1.1：「{表格标题}」→ {建议表名}
  [J] 关联1：「{元素标题}」→ {建议表名}
  [J] 关联2：「{元素标题}」→ {建议表名}

★ 层级嵌套直接反映主从关系：缩进越深 = 外键层级越深。

2. 生成交互探索清单

从截图顶部到底部，逐个扫描每个UI元素。对每个元素问"我能看到它的全部信息吗？"——不能则列入清单。

必须穷举，一个不漏：
- 每个下拉框 → 列入，确认三件事：①选项列表 ②单选/多选 ③是否有特殊值
  ★ 表格内下拉框必须单独列项：逐列确认选项，测试不同选项是否增删/显隐其他列
- 每个未选中的Checkbox → 每个选项单独列入
- 每个未选中的Radio选项 → 每个选项单独列入
- 每个按钮（「+新增」「添加」「选择」等）→ 列入
- 每个Toggle/开关 → 列入
- 每个未激活的Tab → 单独列入
- 每个折叠区域 → 列入
- ★ 每个灰色/只读/不可编辑的字段或表格列 → 列入，确认：①数据从哪个实体携带 ②携带后能否修改 ③是否需冗余存储/快照
  ★ 灰色列 = 极可能是携带字段 → 不建模到当前实体，只存引用ID
- ★ 每个"选择后自动填充"的交互行为 → 列入，确认：①什么触发填充 ②填充了哪些字段 ③填充字段是否可修改
- ★ 每个带搜索图标/自动补全的输入框 → 列入，确认：①这是自由文本还是引用选择器？②选中后存的是ID还是文本？③候选数据来自哪个实体？
- ★ 每个树形选择器（分类/部门/区域） → 列入，确认：①只能选叶子节点还是任意层级？②单选还是多选？③树结构数据来自哪个引用表？
- ★ 如果是向导/分步表单（有步骤条Stepper） → 列入，确认：①共几步？②每步保存还是最后统一保存？③能否返回上一步？④是否有草稿暂存？
- ★ 每个文件/图片上传区域 → 列入，确认：①单文件还是多文件？②文件类型和大小限制？③上传时机（先传后关联/随表单提交）？

★ 下拉框特殊值深度规则（强制执行）：

选项中出现以下类型值，必须逐个切换并截图确认界面变化：

特殊值类型：
  「自定义」「自定义金额」「自定义时间」等 → 通常弹出输入框让用户手动填写
  「其他」「其他原因」等               → 通常弹出文本输入框
  「不限」「无限制」「永久」等          → 通常隐藏关联约束字段
  「指定XXX」「选择XXX」等             → 通常弹出选择器或新输入区域
  任何含动态语义的选项                 → 可能触发条件渲染

处理方式：
  1. 先展开下拉框截图确认所有选项
  2. 识别其中的特殊值
  3. 每个特殊值作为独立清单项，逐个切换确认界面变化
  4. 切换后出现新交互元素，按深度追踪规则处理

交互元素列完后，追加业务逻辑问题（生命周期状态、编辑权限、删除策略等）。

## 第二层：数据建模（技术无关逻辑模型）

★ 核心原则：UI层级 = 数据实体层级。字段必须按所属实体分组，不按UI区块分组。

2-1 实体层级与关系

基于「数据容器识别」结果，输出实体层级图（含PK/FK/基数/UI来源/归属类型）：

格式：
{实体中文名}({英文表名}) [{容器类型}]
  PK: {主键字段}
  FK: {外键字段} → {父表名}
  数量约束: {1:N的N上限}
  UI来源: {对应页面区域/弹窗名称}

树形缩进表示父子关系，层级与数据容器识别一致。

★ 区分"拥有实体"与"引用实体"：
- [OWN] 拥有实体：本页面创建和管理的数据，需生成完整数据实体和数据访问代码
- [REF] 引用实体：本页面仅通过ID引用的已有数据（如品牌、会员方案），不需生成代码

格式示例：
储值套餐(stored_value_package) [M][OWN]

品牌(brand) [REF] — 仅存brand_id

2-2 字段字典（★ 按实体分表输出，每个实体单独一张表）

★ 不要把所有字段混在一起。每个实体单独一张字段表。
★ 每张表必须含：主键(id)、外键、审计字段(create_time, update_time, create_by)
★ 判别字段必须在"判别控制"列标注它控制哪些字段的存在性
★ 数据类型使用通用SQL标准类型，不用特定数据库方言

★ 携带字段 vs 自有字段（强制执行）：
  页面上看到的字段不一定属于当前实体，必须逐字段判断归属：
  - 自有字段：用户手动输入或选择的值 → 建模到当前表
  - 引用ID：用户选择已有实体 → 当前表只存该实体ID
  - 携带字段：选择后自动填充的只读信息 → 不建模到当前表 → 字段字典中不出现 → 响应体(GET)中通过关联查询返回，标注「携带展示，不存储」
  - 快照字段：选择后填充且需冻结的值 → 建模到当前表，标注「冗余快照，来源={实体}」

  判断流程：
    该字段可编辑吗？
      → 可编辑 → 自有字段
      → 不可编辑（灰色/只读）→ 来自哪个引用实体？
        → 已确认来源 → 携带字段，不建模
        → 需要冻结快照吗？→ 需要 → 快照字段，标注来源 | 不需要 → 携带字段，不建模

格式：

#### 表名：{中文名}({英文表名})
父表：{父表名} | 关系：{1:N / N:N} | UI来源：{页面区域/弹窗}

| 字段名 | 列名(英文) | 数据类型 | 必填 | 默认值 | 说明 | 写入方 | 判别控制 |
|--------|-----------|---------|------|--------|------|--------|---------|

写入方取值：
  用户输入 — 用户在表单中手动填写/选择
  后端自动 — 后端根据上下文自动填入（如create_by取当前用户）→ 不出现在请求体中
  引用携带 — 选择引用实体后自动带入的快照值 → 标注来源实体
  系统生成 — 系统自动生成的值（如自增编号、UUID）→ 不出现在请求体中

数据类型参考：VARCHAR(100), DECIMAL(10,2), INTEGER, BIGINT, BOOLEAN, DATE, TIMESTAMP, TEXT

"判别控制"列示例：
- date_type: "=1时 start_date/end_date 必填"
- gift_type: "决定 tier_gift / tier_gift_bundle 是否有数据"
- org_scope: "=全部门店时 package_organization 表无数据"

★ 使用通用SQL标准语法，不用任何数据库方言特有语法。

要求：
- 表名前缀t_（如t_stored_value_package）

- 枚举字段加注释标明枚举值含义

格式示例：
CREATE TABLE t_stored_value_package (
  id BIGINT PRIMARY KEY,                        -- 主键，自增策略由实现决定
  activity_name VARCHAR(100) NOT NULL,           -- 活动名称
  date_type INTEGER NOT NULL DEFAULT 1,          -- 活动日期类型: 1=指定日期 2=永久有效
  start_date DATE,                               -- 开始日期, date_type=1时必填
  end_date DATE,                                 -- 结束日期, date_type=1时必填
  status INTEGER NOT NULL DEFAULT 0,             -- 状态: 0=草稿 1=启用 2=停用
  create_time TIMESTAMP NOT NULL,                -- 创建时间
  update_time TIMESTAMP NOT NULL,                -- 更新时间
  create_by BIGINT NOT NULL,                     -- 创建人ID
  deleted BOOLEAN NOT NULL DEFAULT FALSE         -- 逻辑删除

CREATE INDEX idx_svp_create_by ON t_stored_value_package(create_by);

★ 不含的内容（由代码规范文件在代码生成阶段补充）：
  - 数据库特有语法（AUTO_INCREMENT、ENGINE=、ON UPDATE CURRENT_TIMESTAMP等）
  - COMMENT=表注释语法（改用SQL标准注释--）

★ 保留精度：字段名、类型精度(如DECIMAL(10,2))、NOT NULL、DEFAULT、索引设计、字段注释

## 第三层：业务规则

3-1 隐式文本：Tooltip→规则、Placeholder→约束、帮助文字→WARN/BLOCK、按钮→副作用

3-2 UI条件渲染（穷举）：
RULE-{编号}: WHEN {条件} THEN {效果} ELSE {默认} 来源：📸/👤/🤖

3-3 计算逻辑：派生字段 = 公式（不存储）；冲突优先级

3-4 不变量 vs 验证：
INVARIANT-{编号}: {条件} 违反后果：{描述}
VALIDATION-{编号}: {条件} 校验时机：前端/提交/异步

3-5 规则执行层分类（★ 必须输出）

将所有RULE和INVARIANT按执行层分类：

| 规则编号 | 规则描述 | 前端UI层 | 后端业务层 | 数据存储层 | 异步任务层 | 说明 |
|---------|---------|---------|-----------|-----------|-----------|------|

分类标准：
- 前端UI层：仅影响界面渲染（显隐、禁用、选项联动）
- 后端业务层：需服务端二次校验或业务逻辑处理
- 数据存储层：影响表结构设计（字段可空性、约束、索引）
- 异步任务层：需异步处理

3-6 保存接口数据结构预览（★ 必须输出）

基于实体层级关系，输出保存接口的嵌套JSON结构预览。嵌套层级必须与2-1实体层级完全对应。代码生成器可直接从此结构生成输入/输出数据对象。

★ 必须同时输出三种结构：

A. 新增请求体(POST)：明细记录没有id
B. 编辑请求体(PUT)：明细记录有id（已有）或没有id（新增），传空数组=全部删除
C. 详情响应体(GET)：返回完整嵌套数据供前端回填表单

--- A. 新增请求 ---
POST /api/{主实体}
{
  // 主表字段
  "field1": "value",

  // 明细：{明细名称}列表
  "details": [
    {
      "detailField1": "value",
      // 子明细：{子明细名称}
      "subDetails": [ { ... } ]
    }
  ],

  // 关联：{关联名称}
  "relatedIds": [1, 2, 3]
}

--- B. 编辑请求（差异说明）---
PUT /api/{主实体}/{id}
{
  // 主表字段同上
  "details": [
    { "id": 101, "detailField1": "修改值" },  // 有id = 更新已有记录
    { "detailField1": "新值" }                  // 无id = 新增记录
    // 原有id=102的记录未出现 = 删除
  ]
}
