■ 0-1 需求文档完整性校验

检查需求文档是否包含以下 7 个核心数据结构（缺一不可）：

  □ 实体层级图（含 PK/FK/OWN/REF 标注）    ← 来自 3.2
  □ 按实体分表的字段字典（含类型/可空/判别控制） ← 来自 3.3
  □ 枚举值定义表                             ← 来自 3.4
  □ 保存事务边界                             ← 来自 3.7
  □ 三态接口数据结构（POST/PUT/GET）          ← 来自 4.5
  □ 规则执行层分类                           ← 来自 4.4
  □ API 概要设计                             ← 来自 6.5

若缺少任何一项，立即提示用户补充，不继续生成。

■ 0-2 实体识别

从实体层级图中识别：
  - [OWN] 实体 → 需要生成完整代码（数据实体 + 数据访问 + 业务逻辑 + 接口）
  - [REF] 实体 → 仅引用，不生成代码，只在输入/输出数据对象中用 ID 字段

■ 0-3 代码规范文件读取

从代码规范文件中确认以下关键约定：
  - 目标语言和框架版本
  - 分层架构和包/模块结构
  - 命名规范（类名、方法名、字段名、表名）
  - 通用SQL类型 → 目标数据库类型 → 目标语言类型的三段映射
  - 主键策略
  - 审计字段模板
  - 物理模型优化规则（如有）

三、测试用例生成规则（通用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

代码生成完毕后，必须基于需求文档的业务规则和约束，输出结构化测试用例集。

■ 3-1 测试用例分类

每个接口输出以下分类的测试用例：

A. 正向用例（Happy Path）
  - 最简请求：只填必填字段，其余全省略 → 验证默认值和非必填处理
  - 完整请求：所有字段填写，嵌套对象完整 → 验证全量保存
  - 各枚举分支：每个枚举值至少一条（如 dateTypeCode=1 一条, =2 一条）

B. 必填校验用例（★ 每个必填字段至少一条）
  - 字段缺失（不传该字段）→ 预期返回校验错误 + 字段路径 + 错误提示
  - 字段为 null → 同上
  - 字符串字段为空串 "" → 如果用 NotBlank 类校验应失败

C. 类型与格式用例
  - 数字字段传字符串 → 预期 400 类型错误
  - 日期字段传非法格式（如 "2026-13-01"）→ 预期格式错误
  - ID 字段传负数 → 预期校验失败
  - 枚举字段传不存在的值（如 dateTypeCode=99）→ 预期业务校验失败

D. 边界值用例（★ 来自约束矩阵，每个约束至少一条）
  - 数字字段：最小合法值→成功 | 低于最小值→失败 | 最大合法值→成功 | 超出最大值→失败
  - 字符串字段：最大长度刚好→成功 | 超出1字符→失败 | 特殊字符测试
  - 集合字段：空数组→验证必填 | 恰好上限→成功 | 超出上限→失败
  - 日期字段：开始=结束→验证是否允许 | 开始>结束→失败

E. 条件联动用例（★ 来自每条 RULE，逐条生成）
  - 每条 RULE 生成 2 条用例：
    1. 条件触发 + 条件必填字段缺失 → 预期失败 + 错误信息
    2. 条件不触发 + 条件字段为 null → 预期成功

F. 业务不变量用例（来自每条 INVARIANT）
  - 每条 INVARIANT 至少 1 条违反用例

G. 数据归属与权限用例
  - 操作不属于当前用户/租户的数据 → 预期 403/404
  - 未登录状态访问需鉴权接口 → 预期 401

H. 幂等用例
  - 同一请求连续提交 2 次 → 第 2 次预期失败或幂等返回
  - 不同请求体但相同幂等键 → 预期被拦截

I. 编辑模式特殊用例
  - 明细列表含有 id 的项（更新）+ 无 id 的项（新增）+ 原有项未传（删除）→ 验证差异更新
  - 传空数组 [] → 验证是否清空所有子记录

■ 3-2 测试用例输出格式

每条测试用例格式：

TC-{编号}: {用例名称}
  分类: {A/B/C/D/E/F/G/H/I}
  接口: {Method} {Path}
  请求体:
  ```json
  { ... }
  ```
  预期状态码: {200/400/401/403}
  预期响应:
  ```json
  {
    "code": "{成功码或错误码}",
    "message": "预期提示信息"
  }
  ```
  覆盖规则: {RULE-xx / CONSTRAINT-xx / INVARIANT-xx / 无}

■ 3-3 用例数量要求

  - A 正向用例：至少 3 条（最简 + 完整 + 各枚举分支）
  - B 必填校验：每个必填字段 1 条
  - C 类型格式：每个非字符串字段 1 条类型错误用例
  - D 边界值：每个有范围约束的字段 2 条（刚好合法 + 刚好超出）
  - E 条件联动：每条 RULE 2 条（触发+未触发）
  - F 不变量：每条 INVARIANT 1 条
  - G 权限：至少 1 条越权用例
  - H 幂等：至少 1 条重复提交用例
  - I 编辑特殊：至少 2 条（差异更新 + 清空子记录）

★ 请求体中未测试的字段用正向用例的合法值填充，确保一次只测一个变量。
★ 预期的错误信息必须与代码中的校验消息一致。
★ 嵌套对象的校验错误路径必须包含索引（如 tiers[0].storeAmount）。

■ 3-4 覆盖率自检

测试用例输出后，必须附带覆盖率自检表：

| 检查项 | 覆盖情况 |
|--------|---------|
| 每个 RULE 规则 | RULE-01 → TC-E01,E02; RULE-02 → TC-E03,E04; ... |
| 每个 CONSTRAINT | C-01 → TC-D01; C-02 → TC-D02; ... |
| 每个 INVARIANT | INV-01 → TC-F01; ... |
| 每个必填字段 | activityName → TC-B01; categoryId → TC-B02; ... |
| 每个枚举字段 | dateTypeCode → TC-A02,A03,C01; ... |
| 安全防护 | 越权 → TC-G01; 幂等 → TC-H01 |

★ 如果某条规则/约束没有对应的测试用例，必须标红并补充。

六、精度传递保障协议
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

★ 本协议确保需求文档中的每一条精度信息都被准确翻译到代码中，无信息丢失。

■ 精度传递检查清单（代码生成完成后逐条验证）

| 精度来源 | 代码目标 | 验证方法 |
|---------|---------|---------|
| 字段字典.数据类型 VARCHAR(100) | 数据实体字段类型 + 建表脚本列类型 | 类型和精度完全匹配 |
| 字段字典.数据类型 DECIMAL(10,2) | 数据实体字段类型 + 建表脚本列类型 | 总位数和小数位匹配 |
| 字段字典.必填 = 是 | 输入数据对象非空校验注解 | 每个必填字段有对应校验 |
| 字段字典.默认值 | 建表脚本 DEFAULT 子句 | 默认值完全一致 |
| 枚举定义表.枚举值 | 枚举类的每个成员 | 值和含义一一对应 |
| 枚举定义表.所属表 | 枚举类关联的实体 | 枚举用在正确的实体字段上 |
| 实体层级.PK/FK | 数据实体的主键/外键字段 | 关联关系正确 |
| 实体层级.[OWN] | 生成了完整代码 | 每个OWN实体有完整代码 |
| 实体层级.[REF] | 仅用 ID 引用 | REF实体无独立代码 |
| 事务边界 | 业务逻辑层事务范围 | 同一事务内的实体在同一方法中处理 |
| 规则执行层.后端业务层 | 业务逻辑层校验代码 | 每条规则有对应校验 |
| 规则执行层.数据存储层 | 建表脚本约束 | 每条规则有对应约束 |
| 接口数据结构.嵌套层级 | 数据对象嵌套层级 | 层级完全对应 |
| 约束矩阵.BLOCK 级 | 校验注解/业务校验 | 每个BLOCK约束有对应代码 |
| API概要设计 | 接口层路由和方法 | 路径和HTTP方法一致 |
| 逻辑建表预览.索引 | 建表脚本索引 | 索引完全保留 |

★ 发现不一致时，以需求文档为准，调整代码。
  需求文档中标记为 [⚠️ 假设] 的内容，在代码中加 // [⚠️ 假设] 注释。
