# 从 UI 逆向到数据模型与代码：思考过程与方法论

> 本文档记录了对"多 Agent 协作需求分析 Prompt 体系"进行三轮深度改进的完整思考过程，
> 包括问题诊断、理论依据、技术方法、以及每一步决策的推理链。

---

## 一、问题的本质：信息链断裂

### 1.1 起点观察

用户反馈：用 AI 从 UI 截图生成的需求文档"没有给出合理的数据库设计，丢失了太多东西，比如二级界面和一级界面的关系，主表和明细表的关系没有体现出来"。

### 1.2 问题抽象

这不是"AI 不够聪明"的问题，而是 **Prompt 的信息结构与目标产物的信息结构不对齐**。

用一个类比来说明：

```
现状：
  UI 截图  →  [按 UI 区块平铺字段]  →  代码生成器  →  ???

  信息在中间环节被"压扁"了。
  一个三维的数据结构（主表→明细表→子明细表）
  被投影成了一维的字段列表。

期望：
  UI 截图  →  [按数据库实体分层组织]  →  代码生成器  →  正确的代码
```

### 1.3 根因定位方法：逆向追溯法

我没有从"怎么改 Prompt"开始，而是从**最终产物（代码）倒推**需要什么信息：

```
Java Entity 类需要什么？  →  表名、字段名、字段类型、FK 关系
Java DTO 类需要什么？     →  嵌套结构（主表 DTO 包含 List<子表DTO>）
Service 层需要什么？      →  事务边界、校验规则归属层
DDL 需要什么？            →  完整的 CREATE TABLE 语句

← 这些信息在需求文档中存在吗？
← 需求文档的哪个环节应该产出这些信息？
← Prompt 的哪一步指导 AI 产出这些信息？
```

逐层倒推后发现，**断裂点在"第二层：数据建模"——Prompt 只用了两行话描述，AI 自然产出的是粗糙的平铺结构**。

## 二、理论基础

### 2.1 领域驱动设计（DDD）— 聚合根与实体边界

**核心理论**：Eric Evans 的领域驱动设计中，"聚合"（Aggregate）是一组紧密关联的对象集合，有一个"聚合根"（Aggregate Root）作为入口。聚合内部的对象不能被外部直接访问，只能通过聚合根操作。

**在本问题中的应用**：

```
储值套餐 = 聚合根（Aggregate Root）
  ├── 储值档位 = 实体（Entity，属于聚合内部）
  │     ├── 赠送礼品 = 实体
  │     └── 赠送券包 = 实体
  │           └── 券包礼品实收配置 = 值对象（Value Object）
  ├── 套餐品牌关联 = 值对象
  └── 套餐渠道关联 = 值对象
```

DDD 的聚合边界 **天然对应数据库的事务边界**：一个聚合内的所有变更在一个事务中完成。

**Prompt 改进的体现**：
- "保存事务边界"（2-4）直接来自 DDD 的聚合概念
- [OWN]（拥有实体）= 聚合内部的实体/值对象
- [REF]（引用实体）= 聚合外部的引用
- "一个主 Service 注入所有子表 Mapper"= 通过聚合根操作聚合内部

### 2.2 关系数据库范式理论 — ER 模型

**核心理论**：Peter Chen 的实体-关系模型（ER Model），将现实世界抽象为实体（Entity）、属性（Attribute）、关系（Relationship）三个要素。关系的基数（1:1, 1:N, M:N）决定了表结构和外键设计。

**在本问题中的应用**：

原始 Prompt 让 AI 识别"实体关系"，但没有教 AI **如何从 UI 中识别实体和关系**。

我发现了一个关键映射：

| UI 模式 | ER 对应 | 数据库实现 |
|---------|---------|-----------|
| 页面主表单 | 主实体 | 主表 |
| 可重复卡片/行 | 1:N 关系的"N"端 | 明细表（FK→主表） |
| 弹窗表单 | 子实体 | 子表（FK→明细表） |
| 弹窗内表格 | 孙实体 | 孙表（FK→子表） |
| 多选下拉/Checkbox | M:N 关系 | 关联表/中间表 |

这就是"数据容器识别"（[M][D][P][G][J]）的理论来源。**UI 的嵌套层级 = ER 模型的实体层级**。

### 2.3 信息论 — 无损传输

**核心理论**：Shannon 信息论中，信息在传输过程中如果经过"有损压缩"，接收方就无法还原原始信息。

**在本问题中的应用**：

需求分析是一个信息传输过程：

```
信源：UI 截图（包含丰富的层级、嵌套、交互信息）
信道：Prompt 指导的分析流程
信宿：代码生成器（需要精确的表结构、字段类型、规则归属）
```

原始 Prompt 的"第二层数据建模"是一个**有损压缩器**——它把三维的实体层级压扁为一维的字段列表，丢失了：
- 字段归属哪张表
- 表之间的 FK 关系
- 枚举字段的数值映射
- 规则的执行层归属

改进的核心就是**让信道无损**：每一步的输出格式都精确对齐下一步的输入需求。

### 2.4 编译原理 — 中间表示（IR）

**核心理论**：编译器将源代码（高级语言）翻译为目标代码（机器语言）时，不会一步到位，而是先翻译为"中间表示"（Intermediate Representation），然后从 IR 生成目标代码。IR 的设计决定了编译器的灵活性和正确性。

**在本问题中的应用**：

需求分析文档就是 UI 截图和最终代码之间的"中间表示"。

```
UI 截图（源语言）
    ↓ 前端（需求分析 Prompt）
需求文档（中间表示 / IR）
    ↓ 后端（代码生成 Prompt）
Java 代码（目标语言）
```

好的 IR 设计原则：
1. **与源语言和目标语言都无关**（通用性）
2. **保留足够信息**不丢失语义（无损性）
3. **结构规整**便于后续处理（可机器消费性）

原始的需求文档作为 IR 不合格：
- 字段字典是给人看的表格，不是给代码生成器消费的结构化数据
- 业务规则只描述 UI 效果，不描述代码层归属
- 没有 DDL 这样的"接近目标语言"的中间产物

改进后的 IR：
- 按实体分表的字段字典 → 直接映射到 Entity 类
- 枚举值定义表 → 直接映射到 Enum 类
- 请求体/响应体结构 → 直接映射到 DTO/VO 类
- DDL → 直接可执行的 SQL
- 规则执行层分类 → 直接决定代码放在哪一层

### 2.5 关注点分离（Separation of Concerns）

**核心理论**：Dijkstra 提出的软件工程原则——将程序分解为不同的部分，每个部分解决一个独立的关注点。

**在本问题中的应用**：

原始 Prompt 的业务规则混淆了多个关注点：

```
RULE-001: WHEN dateType = 永久有效 THEN 隐藏日期范围选择器
```

这一条规则实际上跨越了三个关注点：
1. **前端关注点**：隐藏日期选择器（UI 渲染逻辑）
2. **后端关注点**：dateType=指定日期时，startDate/endDate 必须非空（校验逻辑）
3. **数据库关注点**：startDate/endDate 允许 NULL（DDL 约束）

"规则执行层分类"（3-5）的本质就是**将一条混合规则拆分为不同关注点**，让每个关注点映射到对应的代码层。

### 2.6 契约式设计（Design by Contract）

**核心理论**：Bertrand Meyer 提出的 DbC——每个软件组件之间通过"契约"（前置条件、后置条件、不变量）来交互。

**在本问题中的应用**：

多 Agent 协作本质上是一个"流水线"，每个 Agent 是流水线上的一个工位。工位之间需要明确的契约：

```
Agent-0 的输出契约 → Agent-1 的输入契约
Agent-1 的输出契约 → Agent-2 的输入契约
...
Agent-4 的输出契约 → Agent-5 的输入契约
```

原始问题：Agent-1 输出的"实体清单、字段定义"是一个模糊契约——它没有规定字段是按实体分组还是按 UI 区块分组。Agent-5 期望按实体分组，但 Agent-1 按 UI 区块分组，契约不匹配。

改进：为每个 Agent 之间的交接定义精确契约：
- Agent-1 必须输出"实体层级关系"（含 PK/FK/OWN/REF）
- Agent-1 必须输出"枚举值定义表"
- Agent-2 必须输出"规则执行层分类"
- Agent-2 必须输出"保存接口请求体结构"

## 三、技术方法

### 3.1 UI 容器-数据表映射法（本次原创方法）

**核心思想**：UI 界面的物理嵌套结构直接映射为数据库的逻辑层级结构。

**方法步骤**：

```
Step 1: 识别 UI 中的 5 类数据容器
  [M] 主表单  [D] 可重复区域  [P] 弹窗  [G] 弹窗内表格  [J] 多选

Step 2: 建立层级关系（缩进 = FK 深度）
  [M] 储值套餐
    [D] 档位
      [P] 加入礼品弹窗
      [P] 加入券包弹窗
        [G] 券包礼品实收表格
    [J] 品牌多选
    [J] 渠道Checkbox

Step 3: 为每个容器命名英文表名
Step 4: 标注 [OWN] 或 [REF]
Step 5: 按此层级输出字段字典（每个容器一张表）
```

**理论依据**：UI 设计的层级嵌套不是随意的——产品经理设计弹窗和可重复区域时，天然在表达"这是一个独立的数据单元"。因此 UI 层级是数据模型的"影子"。

**适用范围**：B 端 CRUD 表单页面（新增/编辑页面）。不适用于纯展示页面、仪表盘、流程图等。

### 3.2 规则执行层分类法

**核心思想**：每条业务规则都需要在至少一个代码层中实现。将规则按执行层分类，就能直接生成对应层的代码。

**分类维度**：

| 执行层 | 判断标准 | 代码归属 |
|--------|---------|---------|
| 前端 UI 层 | 只影响界面渲染（显隐、禁用、选项联动） | Vue/React 条件渲染 |
| 后端校验层 | 需要服务端二次校验或涉及多字段/多表联合校验 | Service + Validator |
| 数据库层 | 影响表结构（NOT NULL、DEFAULT、CHECK） | DDL + Entity 注解 |
| 定时任务层 | 需要异步/定期执行 | Job + EventHandler |

**理论依据**：分层架构（Layered Architecture）的每一层有不同的职责。规则应该放在最合适的层执行，而不是全部堆在一层。

### 3.3 三态 DTO 设计法

**核心思想**：同一个数据实体在创建、编辑、查询三种场景下的数据结构不同，需要分别定义。

```
创建(POST)：明细没有 id，关联用 id 数组
编辑(PUT)：明细有可选 id（有=更新, 无=新增, 缺失=删除）
查询(GET)：所有记录有 id，枚举有 label，关联有完整对象
```

**理论依据**：CQRS（Command Query Responsibility Segregation）——命令模型和查询模型应该分离。写操作（POST/PUT）的 DTO 结构和读操作（GET）的 VO 结构服务于不同的消费者，不应混为一谈。

### 3.4 DDL 作为验证手段

**核心思想**：DDL（CREATE TABLE 语句）是数据模型正确性的终极验证——如果 DDL 能正确执行且覆盖所有字段，数据模型就是完整的。

```
字段字典 ←→ DDL 字段
枚举定义 ←→ DDL COMMENT
实体关系 ←→ DDL 外键/索引
判别控制 ←→ DDL NULL/NOT NULL
```

**理论依据**：形式化验证（Formal Verification）的简化版——用一种精确的形式语言（SQL DDL）来验证一种半结构化的描述（需求文档）。如果两者能对齐，说明需求文档是自洽的。

### 3.5 OWN/REF 分类法

**核心思想**：区分"本页面创建和拥有的数据"与"本页面仅引用的已有数据"。

```
[OWN] 储值套餐、档位、礼品配置  → 需要生成 Entity + Mapper + DDL
[REF] 品牌、会员方案、优惠券     → 只在 DTO 中用 Long xxxId 引用
```

**理论依据**：DDD 中的"限界上下文"（Bounded Context）——每个限界上下文只负责自己领域内的实体。品牌、会员方案属于其他限界上下文，在储值套餐的上下文中只是一个引用。

**实际价值**：防止代码生成器为已有实体重复生成 Entity 类，造成代码冗余和命名冲突。

## 四、思考过程的三轮迭代

### 4.1 第一轮：结构性缺陷修复

**思考起点**：为什么字段字典是"平铺"的？

**推理链**：
1. 字段字典平铺 ← Prompt 说"为每个 UI 元素建立字段记录"（按 UI 元素，不按实体）
2. 没有实体层级 ← Prompt 说"识别主从关系"但只用了一行，没有给出方法论
3. 弹窗没有建表 ← AI 把弹窗当作"交互细节"而非"数据容器"
4. 多选没有中间表 ← 没有识别 M:N 关系的机制

**解决方案**：
- 教 AI 一个认知框架：UI 容器-数据表映射法（[M][D][P][G][J]）
- 在阶段一（初扫）就执行数据容器识别
- 强制字段字典按实体分表输出
- 新增保存事务边界
- 新增规则执行层分类
- 新增保存接口请求体结构

**改动文件**：8 个

### 4.2 第二轮：代码生成链路对齐

**思考起点**：第一轮改了需求分析的输出，但代码生成器（Agent-5）知道怎么消费吗？

**推理链**：
1. Agent-5 的输入声明还是"实体清单、字段定义" ← 没有更新
2. Agent-5 假设每个页面一个主实体 ← 代码生成顺序是线性的
3. 没有枚举类生成 ← 没有枚举值定义表
4. DTO 是平铺的 ← 不知道嵌套结构
5. 创建和编辑 DTO 相同 ← 没有 id 的差异处理
6. 深度追踪发现新弹窗时不更新数据容器 ← 流程断裂

**解决方案**：
- 更新 Agent-5 输入声明，新增 V3 增强指令
- 代码生成顺序改为：枚举 → Entity(按层级) → 嵌套DTO → 嵌套VO → Mapper → ...
- 新增枚举层生成规范
- 新增 [OWN]/[REF] 分类
- 深度追踪增加第 0 步：回更新数据容器

**改动文件**：3 个新增修改

### 4.3 第三轮：全链路完整性审查

**思考起点**：站在整个流水线的角度，还有哪些 Agent 没有跟上？

**推理链**：
1. 只定义了写入结构（POST/PUT），没定义查询结构（GET）← 前端无法回填表单
2. Agent-3（系统架构师）不知道实体层级 ← API 设计和级联删除都是猜测
3. Agent-6（红队）不检查新章节 ← 新增的改进没有验证机制
4. 旧示例文档会误导 AI ← 示例的力量大于指令
5. 缺少 DDL 作为中间验证手段 ← 无法验证数据模型自洽性

**解决方案**：
- 请求体结构扩展为三态（POST + PUT + GET 详情响应）
- Agent-3 输入声明增加 V3 变化说明
- Agent-6 新增"数据模型完整性审查"维度（DM-1~DM-6）
- 重命名旧示例文档
- 新增 DDL 预览输出

**改动文件**：5 个新增修改

## 五、核心洞见总结

### 洞见一：UI 是数据模型的"影子"

产品经理设计 UI 时，虽然不画 ER 图，但他们通过弹窗、可重复卡片、多选控件等 UI 模式，隐式地表达了数据的层级和关系。**从 UI 逆向数据模型的关键，不是分析字段，而是分析容器**。

### 洞见二：Prompt 工程的本质是"信息架构设计"

Prompt 不只是"告诉 AI 做什么"，更是**设计信息的流动路径和变换方式**。
好的 Prompt = 好的中间表示（IR）设计。
IR 必须满足：无损性（不丢信息）、可消费性（下游能直接使用）、可验证性（能自查正确性）。

### 洞见三：多 Agent 协作是一条生产流水线

每个 Agent 是流水线上的一个工位，工位之间需要精确的接口契约。
**流水线的产出质量取决于最弱的一个接口**——如果 Agent-1 和 Agent-5 之间的接口是模糊的，整条线都会出问题。

### 洞见四：规则必须分层才能生成代码

一条业务规则如果只描述"WHEN X THEN Y"，代码生成器不知道该生成前端代码还是后端代码还是数据库约束。
**规则的执行层归属是代码生成的必要信息**。

### 洞见五：验证机制必须与改进同步

每次改进输出结构，如果不同步改进验证机制（红队审查），改进就是不可靠的。
**DDL 预览 + 红队数据模型审查**形成了双重验证：DDL 是自动化验证（SQL 能不能跑），红队是对抗性验证（有没有遗漏）。

## 六、方法论的适用边界

### 适用场景
- B 端 SaaS 系统的 CRUD 表单页面（新增/编辑）
- 主从表结构明确的业务（订单+订单明细、合同+合同条款、活动+活动规则）
- 需要生成 Spring Boot / MyBatis-Plus 后端代码的项目

### 不适用场景
- 纯展示页面（仪表盘、报表）— 没有"写入"操作，数据容器识别不适用
- 流程型页面（审批流、工作流）— 主要逻辑在状态机而非数据结构
- 高度动态的页面（低代码平台、表单引擎）— 数据结构是运行时配置的，不是设计时固定的
- 非关系型数据库项目 — DDL 验证法不适用

### 可能的局限
- Prompt 长度增加可能影响弱模型的执行质量
- 对于极其简单的页面（只有一张主表），数据容器识别等步骤是多余的
- 枚举值需要人工确认数值映射，AI 的猜测可能不符合已有系统的编码规范
