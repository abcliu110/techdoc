## 术语表

| 术语 | 定义 | 来源 |
|------|------|------|
| 储值套餐(stored_value_package) | 商户创建的储值营销活动，顾客按固定金额档位充值可获赠送金额、积分、礼品等 | 📸 截图可见 |

## 一、业务概述

### 1.1 系统定位

- 系统：全来店商户中心 📸 截图可见
- 所属模块：会员中心 > 储值活动 📸 截图可见
- 当前页面：新增储值套餐（create 模式） 📸 截图可见
- 操作者：商户管理员 🤖 AI 推断
- 受益者：商户（促进顾客预充值）、最终用户（获得赠送优惠） 🤖 AI 推断

### 1.2 表单形态

- 交互模式：长页面 + 左侧锚点导航混合 [⚠️ 假设：三步骤为同页锚点，非独立步骤页]
- 左侧导航：3个步骤锚点 —「基础信息」[active]、「储值规则」、「购买规则」 📸 截图可见
- 右上角按钮：「取消」「上一步」「保存」 📸 截图可见
- 底部：「我已阅读并同意《法律声明》」checkbox 📸 截图可见
- 保存机制：右上角「保存」按钮统一提交 📸 截图可见
- 新增 vs 编辑差异：当前为新增页面，编辑页面差异待确认 [⚠️ 假设：启用后档位不可修改]

### 1.3 业务流程概述

```
商户管理员 → 进入「储值活动」列表 → 点击「新增储值套餐」
  → 填写「基础信息」（品牌、分类、日期、周期、时段、协议等）
  → 配置「储值规则」（档位金额、赠送方案、礼品、会员升级等）
  → 配置「购买规则」（渠道、门店、用户限制、购买次数等）
  → 勾选「我已阅读并同意《法律声明》」
  → 点击「保存」→ 创建储值套餐（草稿状态）
```

## 三、数据模型

### 3.1 实体关系总览

```
[M] stored_value_package (主实体)
 ├── [D] stored_value_tier (档位, 1:N, 最多5)
 │    ├── [P] tier_gift (优惠券礼品, 1:N)
 │    ├── [P] tier_gift_bundle (券包, 1:N)
 │    │    └── [G] tier_gift_bundle_item (券包明细计入实收, 1:N)
 │    └── [G] tier_upgrade_level (升级会员等级, 1:N)
 ├── [D] package_available_period (可用时段, 1:N)
 ├── [D] package_exclude_date (排除日期, 1:N)
 ├── [J] package_brand (品牌关联, M:N)
 ├── [J] package_agreement (储值协议关联, M:N)
 ├── [J] package_channel (适用渠道, 1:N)
 └── [J] package_miniprogram (适用小程序, M:N)

[REF] activity_category — 活动分类(可内联创建)
[REF] user_agreement — 储值协议(跳转新页面创建)
[REF] coupon — 优惠券
[REF] coupon_bundle — 券包
[REF] member_plan — 会员方案
[REF] member_level — 会员等级
[REF] organization — 门店
[REF] area — 区域
[REF] miniprogram — 小程序
```

## 七、风险与安全

### 7.1 并发控制

| 场景 | 风险 | 方案 |
|------|------|------|
| 多管理员同时编辑同一套餐 | 后保存者覆盖前保存者的修改 | 乐观锁(updated_at 或 version 字段)，保存时比对版本号，冲突时返回 409 提示刷新 |
| 高频创建套餐 | package_code 重复 | 使用数据库 UNIQUE 约束 + 应用层雪花算法/UUID 生成 |
| 启用/停用并发操作 | 状态不一致 | 数据库行锁 SELECT ... FOR UPDATE 或状态机 CAS 更新 |

### 7.2 安全

| 威胁 | 防护措施 |
|------|---------|
| XSS | package_name、category_name 等文本字段入库前 HTML 转义 |
| SQL 注入 | 全部使用参数化查询(MyBatis #{}) |
| 越权访问 | tenant_id 强制注入，查询/修改时附加 tenant_id 条件 |
| 金额篡改 | store_amount / gift_amount 等金额字段后端二次校验，不信任前端传值范围 |
| CSRF | 接口层统一 CSRF Token 校验 |
| 批量枚举 | 列表/搜索接口限流(Rate Limit)，分页强制 size ≤ 100 |

### 7.3 幂等性

| 接口 | 幂等方案 |
|------|---------|
| POST 新增 | 前端生成 request_id，后端以 (tenant_id, request_id) 做去重表，10 分钟内重复提交返回已创建的 id |
| PUT 编辑 | 天然幂等（同一 body 多次提交结果一致），配合乐观锁防止旧版本覆盖 |
| PUT 启用/停用 | 状态机约束，重复调用返回当前状态，不报错 |
| DELETE 删除 | 逻辑删除天然幂等，已删除时返回成功 |

### 8.1 状态机

```
         ┌────────────────────┐
         │                    │
         ▼                    │
    ┌─────────┐  启用    ┌─────────┐
    │  草稿   │────────▶│  启用   │

    │  已删除  │         │  停用   │

                              │

                              ▼
                         ┌─────────┐
                         │  已删除  │
                         │(逻辑)   │
                         └─────────┘
```

状态转换规则：
- **草稿 → 启用**：所有必填字段校验通过，legal_agreed=1
- **启用 → 停用**：立即生效，已购买用户权益不受影响
- **草稿 → 删除**：直接逻辑删除
- **停用 → 删除**：需校验无已购买记录（A-05，置信度 70%）
- **启用 → 编辑**：部分字段可能只读（A-04，如档位不可修改，置信度 50%）
