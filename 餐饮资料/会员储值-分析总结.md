# 会员储值功能 - 分析总结

## 📋 分析范围

本分析覆盖 nms4cloud-crm 会员储值功能的完整业务流程和所有相关数据表。

## 🎯 核心内容总结

### 1. 业务流程（3个关键流程）

#### 流程1：储值方案配置
- 商户创建储值方案，定义规则（有效期、周期、时段、渠道等）
- 配置储值档位（多个金额档位，每个档位有不同的赠送）
- 配置购买限制（单人限制、总量限制）
- 设置赠送内容（金额、积分、优惠券、券包、会员升级）

#### 流程2：会员储值执行
- 会员查询可用的储值方案（系统多条件过滤）
- 选择储值档位和支付方式
- 支付成功后，系统自动更新余额并发放赠送

#### 流程3：余额消费
- 会员购物结账时选择使用储值
- 系统优先扣除赠送金额，再扣除本金
- 记录完整的交易流水

### 2. 数据库表（7张表）

| 表名 | 用途 | 关键特点 |
|------|------|---------|
| **crm_deposit_plan** | 储值方案配置 | 包含8个JSON字段，最复杂的表 |
| **crm_deposit_category** | 方案分类 | 支持多级分类管理 |
| **crm_deposit_agreement** | 储值协议 | 用户需要同意的条款 |
| **crm_card_balance** | 会员卡余额 | 使用乐观锁防止并发问题 |
| **crm_card_balance_task** | 余额操作记录 | 记录每次储值/消费操作 |
| **crm_card_record** | 交易流水 | 详细的对账信息 |
| **crm_member_gift_record** | 礼品发放记录 | 记录优惠券等赠品 |

### 3. 复杂的JSON字段（8个）

#### 核心JSON字段

1. **tier_rule** - 储值档位规则（最复杂）
   - 定义多个储值档位
   - 每个档位包含：本金、赠送金额、赠送积分、优惠券、券包等
   - 支持一次性和周期性发放

2. **purchase_limit_rule** - 购买限制规则
   - 单人总限：总共只能买N次
   - 单人周期限：每天/周/月最多买N次
   - 全局总限：全局总共只能卖N次
   - 全局周期限：每天/周/月全局最多卖N次

3. **cycle_rule** - 周期规则
   - WEEKLY: 仅周一到周五
   - MONTHLY: 每月的1号、15号等

4. **time_periods** - 时段规则
   - 指定几点到几点可用
   - 支持多个时段

5. **exclude_dates** - 排除日期
   - 这些日期方案不可用（如国定假日）

6. **channels** - 渠道规则
   - POS、小程序、H5等不同渠道的支持

7. **applicable_orgs** - 门店/地区规则
   - 指定门店或地区可用

8. **member_level_rule** - 会员等级规则
   - 限制特定会员等级可购买

### 4. 关键业务概念

#### 余额组成
```
总余额(total) = 本金(principal) + 赠送(gift)

例：会员储值100元，赠送50元
→ principal = 100, gift = 50, total = 150
```

#### 消费顺序
```
优先扣除赠送金额 → 再扣除本金

例：余额 principal=100, gift=50, total=150
消费120元 → 先扣gift(50)，再扣principal(70)
结果：principal=30, gift=0, total=30
```

#### 乐观锁防护
```
UPDATE crm_card_balance
SET principal = principal + ?,
    gift = gift + ?,
    revision = revision + 1
WHERE lid = ? AND revision = ?  ← 关键：版本号必须匹配
```

防止并发冲突，如果版本号不匹配说明有其他线程已更新。

### 5. 可用方案过滤顺序（9个过滤条件）

```
1. 状态必须启用 (status=1)
   ↓
2. 在有效期内 (begin_time ≤ now ≤ end_time)
   ↓
3. 满足周期规则 (今天是否在 cycle_rule 的 values 中)
   ↓
4. 不在排除日期中 (今天是否在 exclude_dates 中)
   ↓
5. 满足时段规则 (现在时间是否在 time_periods 中)
   ↓
6. 满足渠道规则 (当前渠道是否在 channels 中)
   ↓
7. 满足门店规则 (当前门店是否在 applicable_orgs 中)
   ↓
8. 满足品牌规则 (当前品牌是否在 brand_rule 中)
   ↓
9. 满足会员等级规则 (会员等级是否在 member_level_rule 中)
   ↓
返回可用方案
```

## 🔑 最重要的5个点

### 1. crm_deposit_plan 的 8个 JSON 字段
这是整个系统的核心，定义了所有的规则。开发者必须理解这8个字段的结构和含义。

### 2. 乐观锁 (revision)
在 crm_card_balance 中必须正确使用乐观锁，否则在并发场景下会导致余额错误。

### 3. 余额组成：principal + gift
理解本金和赠送的概念，消费时优先扣除赠送。

### 4. 可用方案的9个过滤条件
系统必须按照指定顺序进行过滤，任何一个条件不满足都要排除该方案。

### 5. 一次性和周期性发放
理解优惠券发放的两种模式，实现中需要分别处理。

## 🛠 应用场景

### 场景1：新人首充
- 配置 `isFirstChargeOnly=true`
- 设置 `perUserTotalLimit=1`（只能购买1次）
- 提供更高的赠送比例

### 场景2：周末促销
- 配置 `cycle_rule={"type":"WEEKLY","values":[6,7]}`
- 配置 `time_periods`（指定营业时段）

### 场景3：VIP会员专享
- 配置 `member_level_rule` 限制会员等级
- 提供专属的赠送内容

### 场景4：渠道差异化
- 分别为 POS、小程序、H5 配置不同的 channels
- 不同渠道提供不同的赠送

### 场景5：限量秒杀
- 配置 `totalSaleLimit=100`（全局只能卖100次）
- 配置 `time_periods`（指定秒杀时间）
- 配置 `perUserTotalLimit=1`（每人最多买1次）

## ⚠️ 常见坑点

1. **忘记使用乐观锁** - 导致并发扣减不正确
2. **混淆 principal 和 gift** - 导致消费顺序错误
3. **未按顺序检查过滤条件** - 导致方案被意外隐藏
4. **JSON 字段解析错误** - 导致规则不生效
5. **未更新统计字段** - 导致数据不一致
