# 储值方案（CrmDepositPlan）充值集成 - 实现文档

> 版本：1.0
> 日期：2026-03-19
> 模块：nms4cloud-crm

---

## 一、概述

### 1.1 背景

CrmDepositPlan（储值方案）此前仅有配置管理能力（CRUD + 可用方案查询），未接入实际储值支付流程。旧系统通过 `ChargeService` + `CrmSaveRule` 驱动储值，在生产环境稳定运行。

### 1.2 目标

新建一套 **完全独立** 的储值接口，基于 CrmDepositPlan 驱动：

- 新入口（`/deposit-plan/charge/*`）
- 新规则匹配（CrmDepositPlan + TierConfig）
- 复用旧的支付通道（`dealReactiveFeign`）
- 复用旧的余额引擎（`CardBalanceService.execute()`）
- 新的赠品发放逻辑（`DepositPlanCouponService`）

### 1.3 核心策略

```
新入口 → 新规则匹配 → 旧的支付通道 → 旧的余额引擎 → 新的赠品发放逻辑
```

旧接口 `/charge/*` 完全不修改，`ChargeService` 零修改，`CardBalanceService` 零修改。

## 二、架构总览

```
新接口（完全隔离）                    共享层（不修改）                    旧接口（不触碰）
─────────────────                    ──────────                         ──────────────
DepositPlanChargeController          dealReactiveFeign (支付)           CrmChargeController
  ↓                                    ↑                                  ↓
DepositPlanChargeService             CardBalanceService (余额)          ChargeService
  ↓                                    ↑                                  ↓
CrmDepositPlanServicePlus (规则)     CrmDealTask (共享表)              CrmSaveRule (旧规则)
  ↓
DepositPlanCouponService (发券)
  ↓
CrmDepositCouponSchedule (周期发券)
```

### 隔离机制

新旧流程通过 `ChargeNotifyAttach.depositPlan` 字段路由：

```java
// CrmOrderCloseQueueConsumer.onMessage() 中
if (Objects.nonNull(notifyAttach) && Boolean.TRUE.equals(notifyAttach.getDepositPlan())) {
    depositPlanChargeService.onPaymentSuccess(task, notifyAttach);
    return;  // 不走旧逻辑
}
```

旧流程的 attach 中不包含 `depositPlan` 字段（值为 null），因此 **永远不会进入新分支**。

## 四、数据库表设计

### 4.1 crm_deposit_charge_record（储值方案购买记录）

用于追踪哪个会员购买了哪个方案的哪个档位，支撑 **购买限制检查**。

```sql
CREATE TABLE crm_deposit_charge_record (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '物理编号',
  mid BIGINT NOT NULL COMMENT '商户ID',
  sid BIGINT NOT NULL COMMENT '门店ID',
  lid BIGINT NOT NULL UNIQUE COMMENT '逻辑编号(雪花算法)',
  name VARCHAR(100) NOT NULL COMMENT '记录名称',
  plan_lid BIGINT NOT NULL COMMENT '储值方案ID(FK→crm_deposit_plan.lid)',
  tier_index INT NOT NULL COMMENT '档位索引(从0开始)',
  card_lid BIGINT NOT NULL COMMENT '会员卡ID(FK→crm_card.lid)',
  task_lid BIGINT NOT NULL COMMENT '交易任务ID(FK→crm_deal_task.lid)',
  save_amount DECIMAL(18,2) NOT NULL COMMENT '储值本金',
  gift_amount DECIMAL(18,2) NOT NULL DEFAULT 0 COMMENT '赠送金额',
  gift_points DECIMAL(18,2) NOT NULL DEFAULT 0 COMMENT '赠送积分',
  channel_code VARCHAR(50) COMMENT '渠道编码(POS/MINI_PROGRAM/H5等)',
  trade_state TINYINT NOT NULL DEFAULT 6 COMMENT '状态:2-成功,4-已关闭,6-支付中',
  created_by VARCHAR(100) COMMENT '创建人',
  created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_by VARCHAR(100) COMMENT '更新人',
  updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除(0-否,1-是)',
  revision INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
  INDEX idx_mid (mid),
  INDEX idx_sid (sid),
  UNIQUE INDEX uk_lid (lid),
  INDEX idx_plan_lid (plan_lid),
  INDEX idx_card_lid (card_lid),
  INDEX idx_task_lid (task_lid)
) COMMENT='储值方案购买记录，用于购买限制检查';
```

**索引说明：**
- `idx_plan_lid` + `idx_card_lid`：购买限制查询的核心索引
- `idx_task_lid`：支付回调时根据 taskLid 更新状态

### 4.2 crm_deposit_coupon_schedule（周期发券计划）

储值成功后如果是 PERIODIC 发放模式，创建此记录。定时任务每天扫描并执行发放。

```sql
CREATE TABLE crm_deposit_coupon_schedule (
  pid BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '物理编号',
  mid BIGINT NOT NULL COMMENT '商户ID',
  sid BIGINT NOT NULL COMMENT '门店ID',
  lid BIGINT NOT NULL UNIQUE COMMENT '逻辑编号(雪花算法)',
  name VARCHAR(100) NOT NULL COMMENT '计划名称',
  plan_lid BIGINT NOT NULL COMMENT '储值方案ID',
  tier_index INT NOT NULL COMMENT '档位索引',
  card_lid BIGINT NOT NULL COMMENT '会员卡ID',
  charge_record_lid BIGINT NOT NULL COMMENT '购买记录ID(FK→crm_deposit_charge_record.lid)',
  tier_snapshot JSON NOT NULL COMMENT '档位快照(TierConfig的完整JSON)',
  issue_cycle_type VARCHAR(20) NOT NULL COMMENT '周期类型:DAILY/WEEKLY/MONTHLY',
  issue_cycle_values JSON COMMENT '周期值:[1,3,5]等',
  issue_time TIME COMMENT '发放时间(HH:mm:ss)',
  total_issue_times INT NOT NULL COMMENT '总发放次数',
  issued_times INT NOT NULL DEFAULT 0 COMMENT '已发放次数',
  first_issue_immediate TINYINT NOT NULL DEFAULT 0 COMMENT '首次是否立即发放:0-否,1-是',
  next_issue_date DATE COMMENT '下次发放日期',
  status TINYINT NOT NULL DEFAULT 1 COMMENT '状态:0-已完成,1-进行中,2-已暂停',
  created_by VARCHAR(100) COMMENT '创建人',
  created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_by VARCHAR(100) COMMENT '更新人',
  updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除(0-否,1-是)',
  revision INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
  INDEX idx_mid (mid),
  INDEX idx_sid (sid),
  UNIQUE INDEX uk_lid (lid),
  INDEX idx_card_lid (card_lid),
  INDEX idx_status_next_date (status, next_issue_date)
) COMMENT='储值方案周期发券计划';
```

**关键字段说明：**
- `tier_snapshot`：购买时的 TierConfig JSON 快照。**方案修改后不影响已购买的发放计划**。
- `idx_status_next_date`：定时任务查询核心索引（`WHERE status=1 AND next_issue_date <= today`）。

## 五、接口定义

### 5.1 POST `/deposit-plan/charge/commit`

提交储值方案充值。

**请求体** `DepositPlanChargeCommitDTO`：

```json
{
  "planLid": 100001,            // [必填] 储值方案ID
  "tierIndex": 0,               // [必填] 档位索引（从0开始）
  "cardLid": 200001,            // [可选] 会员卡ID，默认取当前登录用户的 cardLid
  "appId": "wx123...",          // [可选] 小程序appId，默认自动获取
  "h5": false,                  // [可选] 是否H5支付，默认 false
  "channelCode": "MINI_PROGRAM" // [可选] 渠道编码
}
```

**字段验证：**
- `planLid`：`@NotNull`，不能为空
- `tierIndex`：`@NotNull` + `@Min(0)`，不能为空且不能小于 0
- `mid` / `sid`：继承自 `BizBaseRequest`，由拦截器注入

**响应体** `DepositPlanChargeCommitVO`：

```json
{
  "payAppId": "wx...",
  "payTimeStamp": "1679000000",
  "payNonceStr": "abc123",
  "payPackage": "prepay_id=...",
  "paySignType": "RSA",
  "paySign": "...",
  "terminalTrace": "1234567890",   // 终端流水号
  "outTradeNo": "WX2024...",       // 支付平台单号
  "taskLid": 1234567890,           // 充值任务编号，用于轮询状态
  "payInfoForHalfScreen": {}       // 半屏模式数据
}
```

### 5.2 POST `/deposit-plan/charge/query_charge_status`

轮询充值状态，前端在支付后调用此接口判断是否到账。

**请求体** `ChargeQueryStatus`（复用旧的）：

```json
{
  "taskLid": 1234567890  // 由 commit 返回的任务编号
}
```

**响应体**：`NmsResult<Boolean>`，`true` 表示充值已成功到账。

### 5.3 权限与限流

- 所有接口均需 `@SaCheckLogin` 登录校验
- `commit` 接口附加 `@ForbidRepeatSubmit` 防重提交

### 6.3 赠品发放逻辑

#### ONE_TIME 一次性发放

```
issueOneTime(task, tier, card)
  ├─ 遍历 tier.coupons[]
  │    └─ 调用 CrmCouponOpServicePlus.couponOpAdd()
  │         ├─ CouponChannel = Recharge
  │         ├─ orderId = task.lid
  │         └─ couponCode + couponNum 来自 CouponItem
  │
  └─ 遍历 tier.couponPackages[]
       └─ 每个 pkg 按 packageQty 重复发放
            └─ 发放 pkg.coupons[] 内的每张券
```

#### PERIODIC 周期性发放

```
createPeriodicSchedule(task, tier, card, planLid, tierIndex, chargeRecordLid)
  ├─ 创建 CrmDepositCouponSchedule 记录
  │    ├─ tierSnapshot = JSON(tier)  // 快照防止方案修改影响
  │    ├─ issueCycleType = DAILY / WEEKLY / MONTHLY
  │    ├─ issueCycleValues = [1, 15]  // 每月1号和15号
  │    ├─ totalIssueTimes = 12
  │    └─ status = 1（进行中）
  │
  ├─ 如果 firstIssueImmediate = true
  │    ├─ 立即调用 issueOneTime()
  │    └─ issuedTimes = 1
  │
  └─ 计算 nextIssueDate
```

#### 周期发券定时任务

```
DepositPlanCouponScheduleJob (XXL-Job: depositPlanCouponScheduleJob)
  每天执行一次
  ├─ 查询 status=1 AND next_issue_date <= today AND deleted=0
  ├─ 遍历每条 schedule
  │    ├─ 从 tier_snapshot 反序列化 TierConfig
  │    ├─ 执行 issueOneTime() 发放本期券
  │    ├─ issuedTimes += 1
  │    ├─ 如果 issuedTimes >= totalIssueTimes → status=0（完成）
  │    └─ 否则计算下次 nextIssueDate
  └─ 单条失败不影响其他记录
```

## 七、CrmDealTask 字段映射

新旧流程共享 `crm_deal_task` 表，但通过不同的字段值区分：

| CrmDealTask 字段 | 旧流程（CrmSaveRule） | 新流程（CrmDepositPlan） |
|---|---|---|
| `saveRule` | 规则名称 | 方案名称（`plan.name`） |
| `saveRuleCode` | CrmSaveRule.lid | CrmDepositPlan.lid |
| `comment` | "线上储值" | `{"depositPlan":true,"tierIndex":0}` |
| `operationModel` | Charge | Charge |
| `principalAmount` | ∑ rule.save_amount | tier.saveAmount |
| `giveAmount` | ∑ rule.give_amount | tier.giftAmount |
| `givePoint` | ∑ rule.give_point | tier.giftPoints |
| `asUpgradeCost` | 来自 CrmSaveRule | false（新流程不使用） |
| `payWayCode` | -1 | -1（线上微信） |

**发券隔离：** 旧流程在 `CrmDealTaskItem.giveCouponId` 中写入优惠券信息，由 `CardBalanceService.dealOtherItemTask()` 处理。新流程 **不写入** `giveCouponId`，由 `DepositPlanCouponService` 在 `onPaymentSuccess` 中独立处理。两套发券逻辑完全不冲突。

## 八、购买限制校验

`DepositPlanPurchaseLimitChecker` 支持 4 种购买限制校验，数据来源为 `crm_deposit_charge_record` 表：

### 8.1 单人购买总次数（perUserTotalLimit）

```sql
SELECT COUNT(*) FROM crm_deposit_charge_record
WHERE mid=? AND card_lid=? AND plan_lid=? AND trade_state IN (2, 6)
```

统计该会员在该方案下的所有成功/支付中记录数，与 `limit` 对比。

### 8.2 单人周期购买次数（perUserPeriodLimit）

同上，但增加 `created_time BETWEEN ? AND ?` 的时间范围过滤：
- **DAILY**：当天 00:00:00 ~ 次日 00:00:00
- **WEEKLY**：本周一 00:00:00 ~ 下周一 00:00:00
- **MONTHLY**：本月 1 号 00:00:00 ~ 下月 1 号 00:00:00

### 8.3 销售总量（totalSaleLimit）

直接读取 `crm_deposit_plan.total_sold_count`，与 `limit` 对比。

### 8.4 周期售卖总量（totalSalePeriodLimit）

```sql
SELECT COUNT(*) FROM crm_deposit_charge_record
WHERE mid=? AND plan_lid=? AND trade_state IN (2, 6)
  AND created_time BETWEEN ? AND ?
```

按周期类型计算时间范围。

## 九、ChargeNotifyAttach 扩展

新增 3 个字段，旧流程不设置这些字段（均为 null），不影响旧逻辑：

```java
@Schema(title = "是否储值方案流程")
private Boolean depositPlan;       // 新流程设为 true

@Schema(title = "储值方案ID")
```

## 十、CrmOrderCloseQueueConsumer 修改详解

仅在支付成功确认后、余额更新前，增加一个路由分支：

```java
// 原有代码：第二次解析 attach（支付成功后）
attach = payOrder.getAttach();
if (StrUtil.isNotBlank(attach)) {
    notifyAttach = JSON.parseObject(attach, ChargeNotifyAttach.class);

    // 原有逻辑：打赏流程
    if (Objects.nonNull(notifyAttach) && Boolean.TRUE.equals(notifyAttach.getPosGiveBill())) {
        giveBillService.finish(sid, notifyAttach.getOrderId(), payOrder);
        return;
    }

    // ===== 新增：储值方案流程路由 =====
    if (Objects.nonNull(notifyAttach) && Boolean.TRUE.equals(notifyAttach.getDepositPlan())) {
        depositPlanChargeService.onPaymentSuccess(task, notifyAttach);
        return;
    }
}
// 以下旧逻辑不变：cardBalanceService.execute() ...
```

**安全分析：**
- 新分支在 `posGiveBill` 判断之后，不影响打赏流程
- 新分支使用 `return` 直接返回，不会走到旧的 `cardBalanceService.execute()`
- 旧流程的 attach 中 `depositPlan` 为 null，`Boolean.TRUE.equals(null)` 返回 false

## 十二、风险控制

| 风险点 | 控制措施 | 说明 |
|--------|---------|------|
| 新流程影响旧流程 | attach 标记路由 | 旧流程 attach 中无 `depositPlan` 字段，条件永远为 false |
| CardBalanceService 行为变化 | 零修改复用 | 新流程 CrmDealTaskItem 不写 `giveCouponId`，旧的 `dealOtherItemTask` 不会处理新流程的券 |
| 支付回调重复处理 | Redis 幂等检查 | 复用 `getLockKeyForOrderId` 机制，2 小时 TTL |
| 赠品发放失败影响余额 | try-catch 隔离 | 赠品发放在 `onPaymentSuccess` 中用 try-catch 包裹，失败不影响余额更新 |
| 周期发券漏发 | schedule 表 + 定时任务 | `next_issue_date` 索引 + 每日扫描 + 单条失败不中断 |
| 方案修改影响已购买的周期发放 | `tier_snapshot` 字段 | 购买时保存 TierConfig 的完整 JSON 快照 |
| 并发购买限制不准 | Redisson 分布式锁 | `CardBalanceService.execute()` 的锁粒度为 cardLid |
| 周期发券日期计算错误 | 边界处理 | WEEKLY 处理跨周、MONTHLY 处理月末天数不足（`Math.min(dom, lengthOfMonth)`） |
