## 系统概述

旧的会员储值系统是 nms4cloud-crm 中的核心功能之一，支持：
- 商户定义多种储值套餐
- 会员选择套餐进行储值
- 系统计算并发放赠送内容（金额、积分、优惠券）
- 完整的交易审计和对账

**设计特点：**
- 使用 `CrmSaveRule` 定义储值规则（简单扁平化设计，相比新的 CrmDepositPlan）
- 使用 `CrmCard` 存储会员卡的余额汇总
- 使用 `CrmCardBalance` 记录每次的余额变动
- 使用 `CrmCardRecord` 和 `CrmDealTask` 提供完整的审计路径
- 所有金额分为 `principal`（本金）和 `give`（赠送）两部分

**统计字段：**

| 字段名 | 说明 |
|--------|------|
| **sum_of_save_times** | 储值总次数 |
| **sum_of_save** | 储值总本金 |

## 业务流程

### 流程1：储值规则配置

```
商户管理员
    ↓
创建储值规则 (CrmSaveRule)
    ├─ 输入：储值金额、赠送金额/积分、优惠券
    ├─ 输入：生效期、适用门店
    ├─ 输入：升卡选项、佣金配置
    └─ 输入：是否首充限制、是否可选券
    ↓
系统验证
    ├─ 规则名称唯一性
    ├─ 优惠券是否存在
    └─ 日期范围有效性
    ↓
规则启用 (status_=1)
    ↓
可供会员选择
```

---

### 流程2：会员储值（核心流程）

```
会员选择储值 (UI: 选择套餐)
    ↓
查询可用规则
    ├─ 过滤：status_=1 (启用)
    ├─ 过滤：begin_time ≤ now ≤ end_time
    ├─ 过滤：门店在 apply_to_store 中
    ├─ 过滤：会员卡存在
    └─ 过滤：未过期
    ↓
会员选择一个或多个规则
    ├─ 如果 optional=true，还要选择优惠券
    └─ 可选：使用优惠券/分期等选项
    ↓
[ChargeService.commit] - 提交储值
    ├─ 验证所选规则
    ├─ 计算总金额：
    │  ├─ principalAmount = ∑ save_amount
    │  ├─ giveAmount = ∑ give_amount
    │  ├─ givePoint = ∑ give_point
    │  └─ unpaid = ∑ (unpaid_amount 或 unpaid_rate)
    ├─ 创建支付订单 (out_trade_no)
    ├─ 创建 CrmDealTask
    │  └─ status: USERPAYING
    ├─ 创建 CrmDealTaskItem (每个规则一条)
    └─ 返回支付链接 (WeChat)
    ↓
会员支付
    ├─ 跳转到 WeChat 支付页面
    ├─ 输入密码/确认
    └─ 支付成功
    ↓
支付回调 [ChargeService.success]
    ├─ 验证支付成功 (trade_state=SUCCESS)
    ├─ 更新 CrmDealTask status=SUCCESS
    ├─ 更新 CrmCard 余额：
    │  ├─ balance += (principal + gift)
    │  ├─ principal_balance += principal
    │  ├─ give_balance += gift
    │  ├─ sum_of_save += principal
    │  ├─ sum_of_save_give += gift
    │  └─ sum_of_save_times += 1
    ├─ 创建 CrmCardBalance 记录
    ├─ 创建 CrmCardRecord 审计记录
    ├─ 发放优惠券 (CrmConsumerCoupon)
    ├─ 发放积分 (更新 CrmCard.points)
    ├─ 如需升卡，触发升卡流程
    └─ 返回成功提示
    ↓
储值完成 ✓
```

**计算逻辑详细例子：**

假设会员选择了2个规则：
- 规则A：save_amount=100, give_amount=20, give_point=50
- 规则B：save_amount=500, give_amount=80, give_point=200

计算结果：
```
principalAmount = 100 + 500 = 600
giveAmount = 20 + 80 = 100
givePoint = 50 + 200 = 250
totalAmount = 600 + 100 = 700

CrmCard 更新：
  balance: 0 → 700
  principal_balance: 0 → 600
  give_balance: 0 → 100
  points: 0 → 250
  sum_of_save: 0 → 600
  sum_of_save_give: 0 → 100
  sum_of_save_times: 0 → 1
```

### 流程3：会员消费

```
会员购物结账
    ↓
收银员扫描会员卡
    ↓
系统查询卡余额 (CrmCard.balance)
    ↓
会员选择使用储值
    ├─ 输入使用金额
    └─ 或选择全部使用
    ↓
系统验证
    ├─ balance >= 使用金额
    └─ card_lid 有效
    ↓
扣除余额（优先级：先扣 gift，再扣 principal）
    ├─ IF give_balance >= 使用金额:
    │    give_balance -= 使用金额
    │    principal_amount -= 0
    └─ ELSE:
         give_balance = 0
         principal_balance -= (使用金额 - 原give_balance)
    ↓
更新 CrmCard：
    ├─ balance -= 使用金额
    ├─ give_balance 或 principal_balance 减少
    ├─ sum_of_consume += 使用金额
    └─ sum_of_consume_times += 1
    ↓
创建 CrmCardBalance 记录 (Consume)
    ↓
创建 CrmCardRecord 审计记录
    ├─ balance_before/after
    ├─ principal_before/after
    └─ give_amount_before/after
    ↓
完成消费 ✓
```

**消费逻辑例子：**

初始状态：
```
principal_balance = 600
give_balance = 100
total balance = 700
```

消费150元：
```
1. 先扣 gift: give_balance = 100 → 0 (扣了100)
2. 再扣 principal: principal_balance = 600 → 550 (扣了50)
3. 结果：
   principal_balance = 550
   give_balance = 0
   total balance = 550
```

---

### 流程4：退款/撤销

```
会员申请退款
    ↓
系统检查
    ├─ trade_state 是否为 SUCCESS
    ├─ 是否在可退款期限内
    └─ balance >= 退款金额
    ↓
标记为退款
    ├─ CrmDealTask.canceled = true
    ├─ CrmDealTask.canceled_amount = 退款金额
    ├─ CrmDealTask.cancel_time = now()
    ├─ CrmDealTask.cancel_by = 操作人
    └─ CrmDealTask.trade_state = REFUND
    ↓
扣除卡余额
    ├─ balance -= 退款金额
    ├─ principal_balance 和 give_balance 按比例扣除
    └─ sum_of_back += 退款金额
    ↓
创建审计记录
    └─ CrmCardRecord (operation_model=Refund)
    ↓
发起退款到支付账户
    ↓
退款完成
```

## 服务层

### ChargeService（核心服务）

**路径：** `nms4cloud-crm-service/src/main/java/com/nms4cloud/crm/service/charge/ChargeService.java`

**关键方法：**

#### 1. `commit(ChargeCommitRequest)`
提交储值请求
- 入参：
  - cardLid: 会员卡ID
  - saveRuleCodes: 选择的规则ID列表
  - selectCoupons: 选择的优惠券（如果 optional=true）
  - unpaidMoneyAmount: 分期金额（可选）
- 出参：paymentUrl (WeChat 支付链接)
- 流程：
  1. 验证卡存在
  2. 查询规则详情
  3. 计算总金额和分期
  4. 创建支付订单
  5. 创建 CrmDealTask
  6. 创建 CrmDealTaskItem

#### 2. `success(ChargeQueryRequest)`
支付成功回调处理
- 入参：out_trade_no (支付订单号)
- 流程：
  1. 查询 CrmDealTask
  2. 更新状态为 SUCCESS
  3. 更新 CrmCard 余额
  4. 创建 CrmCardBalance
  5. 创建 CrmCardRecord
  6. 发放优惠券和积分

#### 3. `queryChargeStatus(ChargeQueryRequest)`
查询支付状态
- 入参：out_trade_no
- 出参：trade_state (NOTPAY/SUCCESS/REFUND等)

## 数据一致性保证

### 1. 乐观锁（Optimistic Lock）

`CrmCard.revision` 字段用于防止并发更新冲突

```sql
UPDATE crm_card
SET balance = balance + ?,
    principal_balance = principal_balance + ?,
    revision = revision + 1
WHERE lid = ? AND revision = ?
```

如果版本号不匹配，更新失败，需要重试。

### 2. 交易原子性

一次储值操作涉及多个表的更新：
- 更新 CrmCard 余额
- 创建 CrmCardBalance 记录
- 创建 CrmCardRecord 审计
- 发放优惠券

使用数据库事务保证这些操作的原子性。

### 3. 审计完整性

每次变动都会记录到 CrmCardRecord，包括：
- 变动前的余额
- 变动后的余额
- 变动原因
- 操作人
- 操作时间

**旧系统特点：**
- ✓ 简单直观，扁平化设计
- ✓ 单一规则对应多档位（通过 CrmSaveRuleInterval）
- ✓ 完整的审计和对账机制
- ✗ 规则配置不够灵活（相比新的 CrmDepositPlan）
- ✗ 不支持复杂的限制条件（周期、时段、渠道等）

- 旧系统：简单、成熟、生产环境使用
