# 储值方案充值集成 - 可行性评估与修订建议

> 日期：2026-03-20
> 评估对象：`储值方案充值集成-实现文档.md`
> 评估范围：方案设计、现有代码承接度、上线可行性、主要风险与修订建议

---

## 一、结论

该方案整体 **技术可行**，并且当前仓库中已经存在主要承接代码，说明它不是停留在概念层面，而是已经进入可落地状态。

但按当前实现和文档状态判断：

1. **可开发、可联调**
2. **可做测试环境验证**
3. **暂不建议直接上线生产**

核心原因不是主链路走不通，而是仍存在若干 **上线阻塞项**：

1. 并发限购与销量统计存在竞争条件
2. 未支付关单后购买记录状态未闭环
3. 品牌 / 区域 / 渠道可用性校验不完整
4. 周期发券的时间语义未完整兑现
5. 会员等级延期仍是 TODO

结论可归纳为一句话：

**方向正确，隔离思路成立，主流程可跑，但上线前必须补齐一致性与规则闭环。**

---

## 二、现有方案的可行点

### 2.1 新旧流程隔离思路是成立的

方案通过 `ChargeNotifyAttach.depositPlan` 区分新旧流程，支付成功后在 `CrmOrderCloseQueueConsumer` 中分流：

1. 旧流程继续走原有 `cardBalanceService.execute()`
2. 新流程走 `depositPlanChargeService.onPaymentSuccess()`
3. 命中新分支后直接 `return`，不会落入旧逻辑

这个设计的优点：

1. 对旧接口 `/charge/*` 改动很小
2. 对旧的 `ChargeService` 基本零侵入
3. 支付通道和余额引擎复用充分
4. 旧生产流程回归成本较低

### 2.2 主流程链路是闭合的

现有实现已经覆盖以下环节：

1. 独立提交接口 `POST /deposit-plan/charge/commit`
2. 储值方案与档位校验
3. 购买限制校验
4. 调用旧支付通道发起支付
5. 创建 `CrmDealTask`、`CrmDealTaskItem`
6. 创建 `crm_deposit_charge_record`
7. 支付成功后复用 `CardBalanceService.execute()`
8. 支付成功后处理赠品发放、周期发券、会员升级
9. 提供 `query_charge_status` 给前端轮询到账状态

这说明从接口到支付再到到账，链路并不缺主干能力。

### 2.3 周期发券的建模方向合理

`crm_deposit_coupon_schedule` 使用快照字段 `tier_snapshot` 保存购买时档位配置，这是合理设计。

它解决了一个关键问题：

1. 方案后续被修改时，不会影响已购买用户的发放权益
2. 周期任务可以基于快照独立执行
3. 已购用户权益更稳定，可追溯性更好

### 2.4 最大程度复用了旧的稳定能力

该方案尽量复用现有稳定资产：

1. 支付继续走 `dealReactiveFeign`
2. 余额变更继续走 `CardBalanceService.execute()`
3. 订单任务继续使用 `CrmDealTask`
4. 充值状态轮询沿用旧模型

这比完全新建一套支付到账引擎要稳健得多，实施成本也更低。

## 三、上线阻塞项

以下问题如果不修，不建议直接上线。

### 3.1 并发限购控制并没有真正成立

当前方案在 `commit()` 中先做购买限制校验，再创建购买记录，再发起支付。

问题在于：

1. 限购判断和记录落库之间没有方案级并发控制
2. 文档把并发风险寄托在 `CardBalanceService.execute()` 的 Redisson 锁上
3. 但该锁发生在支付成功后，不是在提交购买时
4. 该锁粒度是 `cardLid`，只对同一会员生效，不能限制不同会员同时抢同一方案

这会带来两个风险：

1. `totalSaleLimit` 和 `totalSalePeriodLimit` 可能被超卖
2. 同一会员快速并发点击时，也可能在支付前通过多次校验

结论：

**当前“并发购买限制不准”的风险控制描述不充分，实际尚未闭环。**

### 3.2 关单后购买记录状态没有回写

当前延时关单逻辑只关闭了支付单和 `CrmDealTask` 状态，但没有同步把 `crm_deposit_charge_record.trade_state` 从 `6-支付中` 改为 `4-已关闭`。

直接后果：

1. 购买限制查询把 `trade_state in (2,6)` 都计入占用名额
2. 失效订单会长期占用购买次数
3. 周期限购和总量限购都会被脏数据污染

这不是小优化，而是业务正确性问题。

结论：

**这是明确的上线阻塞项。**

### 3.3 方案可用性校验未真正覆盖品牌 / 区域 / 渠道

当前充值时调用 `isPlanAvailable()`：

1. `brandLid` 传的是 `null`
2. 指定品牌规则实际上不会生效
3. 指定区域规则当前直接 `return true`
4. 渠道校验仍停留在文档 TODO

这意味着如果运营配置了：

1. 仅某品牌可买
2. 仅某区域门店可买
3. 仅某渠道可买

当前实现仍可能放行不符合条件的请求。

结论：

**只要业务启用了这些规则，当前实现就不满足上线要求。**

### 3.4 周期发券的“时间”语义没有真正落地

虽然计划表存了 `issueTime`，但当前执行逻辑只按 `next_issue_date <= today` 扫描，`calculateNextIssueDate()` 也只计算日期，不计算具体时间。

这表示：

1. 方案支持“哪一天发”
2. 但还不真正支持“哪一天几点发”

如果业务接受“每天定时任务统一发当天到期记录”，问题不大。
如果业务要求精确到时分秒，当前实现不够。

结论：

**这项是否阻塞上线，取决于业务是否真的依赖精确时间。**

### 3.5 会员等级延期仍未实现

会员升级部分已经落了等级切换，但 `isLevelDelay` 仍只有日志，没有真正改会员有效期字段。

若储值方案的卖点包含：

1. 升级会员等级
2. 同时延长有效期

则当前实现只兑现了一半。

结论：

**如果运营配置会使用等级延期，这也是上线阻塞项。**

## 四、中高风险问题

### 4.1 销量统计存在丢失更新风险

当前 `updatePlanSoldStats()` 是先查出 `totalSoldCount / totalSoldAmount`，再计算新值，再执行更新。

在并发支付成功时，可能出现：

1. 两个线程读到相同旧值
2. 分别计算相同新值
3. 后写覆盖前写
4. 最终销量被少记

建议：

1. 改为数据库原子更新
2. 或使用乐观锁 `revision`
3. 或使用 `setSql("total_sold_count = total_sold_count + 1")`

### 4.2 周期发券任务缺少并发抢占保护

`DepositPlanCouponScheduleJob` 每天扫描到期记录后逐条执行。

如果出现以下情况：

1. XXL-Job 重复触发
2. 多实例同时跑任务
3. 手工重试叠加自动执行

则同一条 `schedule` 有重复发放风险。

当前更像“尽量单实例运行”，而不是“代码层面确保幂等”。

建议：

1. 执行前给 `schedule` 加锁
2. 或更新时带状态条件与版本号
3. 或引入“处理中”状态防重复消费

### 4.3 周期发券使用的任务标识不够严谨

`executeScheduledIssuance()` 中构造最小化 `CrmDealTask` 时，将 `task.lid` 设置成了 `chargeRecordLid`。

这虽然能勉强作为发券 `orderId` 使用，但语义上并不是交易任务编号。

风险：

1. 后续排障时容易混淆“购买记录号”和“交易任务号”
2. 发券记录、日志、幂等排查时不够清晰

建议：

1. 周期发券使用明确的 scheduleLid 作为业务流水
2. 或在表中额外保留 originTaskLid

### 4.4 `query_charge_status` 依赖旧引擎 Redis 标记

当前轮询状态接口依旧通过 `CardBalanceService.getLockKeyForOrderId(taskLid, false)` 对应的 Redis 标记判断成功。

优点是复用旧逻辑。

但要注意：

1. 它本质上是“订单已处理”的信号
2. TTL 为 2 小时
3. 并不是严格意义上的支付状态表

如果前端轮询只发生在支付后的短时间内，可接受。
如果以后要支持更长时间查询，建议基于 `CrmDealTask.tradeState` 或 `crm_deposit_charge_record.trade_state` 补正式状态查询。

## 五、建议修订方案

### 5.1 第一优先级：补业务正确性闭环

建议优先补这 4 项：

1. 关单时同步更新 `crm_deposit_charge_record.trade_state = 4`
2. `updatePlanSoldStats()` 改为原子更新或乐观锁更新
3. 在 `commit()` 增加方案级并发控制
4. 把品牌 / 区域 / 渠道可用性校验补完整

这是最小上线集合。

### 5.2 第二优先级：补权益完整性

建议补这 3 项：

1. 实现会员等级延期
2. 明确积分是否完全由旧引擎处理
3. 明确周期发券是否需要精确到 `issueTime`

### 5.3 第三优先级：补运维与排障能力

建议补这 4 项：

1. 为周期发券增加幂等和抢占保护
2. 为 `crm_deposit_charge_record` 增加按 `trade_state`、`created_time` 的排障查询索引视图或统计 SQL
3. 增加关键日志：方案号、档位号、购买记录号、schedule号
4. 增加失败重试与告警口径

## 六、推荐实现方式

### 6.1 方案级并发控制

建议在 `commit()` 中对以下维度加分布式锁之一：

1. `mid + planLid`
2. `mid + planLid + cardLid`

建议区分两种锁：

1. 校验总量时用 `mid + planLid`
2. 校验单人购买次数时用 `mid + planLid + cardLid`

这样可以兼顾：

1. 防超卖
2. 防同会员重复抢购
3. 降低锁冲突范围

### 6.2 销量统计改为原子更新

推荐不要再走“先查后更”，改成数据库原子递增。

建议方式：

1. `total_sold_count = total_sold_count + 1`
2. `total_sold_amount = total_sold_amount + saveAmount`

若项目已有 `revision` 体系，则优先走乐观锁更新并失败重试。

### 6.3 关单回写购买记录

建议在 `closePay(task)` 成功后追加：

1. 更新 `crm_deposit_charge_record.trade_state = 4`
2. 条件限制 `task_lid = task.lid and trade_state = 6`

这样可以保证：

1. 只回写支付中记录
2. 不误改成功记录
3. 限购统计及时释放

### 6.4 可用性校验补全

建议把 `commit()` 前置查询补足：

1. 当前门店所属品牌
2. 当前门店所属区域
3. 当前请求渠道 `channelCode`

再统一传入 `isPlanAvailable()` 或扩展成新的校验方法。

### 6.5 周期发券明确业务语义

建议先做一个业务决策：

1. 如果只要求“按日发”，保留当前实现，但文档要明确 `issueTime` 暂不生效
2. 如果要求“按具体时间发”，则应把任务扫描条件改为日期时间级别，并把 `nextIssueTime` 作为真正调度字段

## 七、建议测试清单

上线前至少覆盖以下场景：

### 7.1 正常流程

1. 方案启用、档位有效，支付成功，余额到账
2. ONE_TIME 礼品发放成功
3. PERIODIC 计划创建成功
4. 首次充值限定档位校验成功

### 7.2 异常流程

1. 支付超时未成功，延时关单后购买记录状态变为关闭
2. 支付回调重复到达，余额不重复入账
3. 周期发券单条失败，不影响其他计划执行
4. 发券异常不影响余额更新

### 7.3 并发流程

1. 同一会员重复提交同一方案
2. 多会员并发抢最后一个限量名额
3. 周期限购边界值并发测试
4. 销量统计并发准确性测试

### 7.4 规则流程

1. 指定门店可买，非指定门店不可买
2. 指定品牌可买，非指定品牌不可买
3. 指定区域可买，非指定区域不可买
4. 指定渠道可买，非指定渠道不可买

### 7.5 回归流程

1. 旧接口 `/charge/commit` 正常储值
2. 旧充值发券逻辑正常
3. 打赏流程 `posGiveBill` 不受影响
4. 订单支付闭环不受影响

## 八、最终建议

### 8.1 是否值得继续做

**值得。**

原因：

1. 方案隔离思路对
2. 复用旧能力多
3. 系统侵入性低
4. 已有实现基础，不是从零开始

### 8.2 是否可以直接上线

**不建议直接上线。**

### 8.3 上线前最低要求

至少完成以下事项后，再进入提测或灰度：

1. 修复关单回写
2. 修复销量并发更新
3. 增加方案级并发锁
4. 补齐品牌 / 区域 / 渠道校验
5. 明确并实现等级延期或明确禁用该配置
6. 补齐核心单元测试和并发验证

---

## 九、总评

本方案不是“不可行”，而是“已经基本可行，但离生产可用还差最后一段工程化闭环”。

从架构角度看，它是一个比较务实的方案：

1. 不推翻旧链路
2. 不重写支付与余额
3. 把差异点集中在规则、赠品和限购层

这条路线是对的。

真正需要补的，不是重新设计架构，而是把以下几类问题收口：

1. 并发一致性
2. 状态闭环
3. 规则完整性
4. 权益兑现完整性
5. 调度幂等性

补完这些后，该方案可以进入灰度上线阶段。
