# 储值方案充值集成开发实现与风险说明

> 范围：`/deposit-plan/charge/*` 新充值链路。
>
> 本文整合了实现文档、审查报告、可行性评估、日期校验修改说明四类内容。
>
> 目标：让接手开发的人员一次看清“已经做了什么、没做什么、风险在哪、上线前还差什么”。

---

## 1. 当前结论

这条链路的整体方向是正确的，但当前状态更适合描述为：

1. 主流程基本闭合
2. 代码已具备开发联调基础
3. 可以做测试环境验证
4. 还不适合作为“已完全工程化的生产方案”直接上线

核心判断可以压缩为一句话：

```text
隔离思路成立，主链路可跑，但业务正确性闭环和规则完整性仍需补齐。
```

---

## 2. 设计目标与总体思路

新链路的目标不是重写整套储值系统，而是：

1. 以 `CrmDepositPlan` 作为新的规则来源
2. 使用新的充值入口
3. 尽量复用旧支付与余额能力
4. 将差异集中在规则判断、购买记录、赠品发放、周期发券等环节

总体策略可以概括为：

```text
新入口 -> 新规则匹配 -> 旧支付通道 -> 旧余额引擎 -> 新赠品逻辑
```

这个方向的优势非常明确：

1. 旧接口 `/charge/*` 不必大改
2. 旧 `ChargeService` 不必重写
3. 支付通道和余额引擎沿用成熟能力
4. 对存量生产链路的侵入较小

---

## 3. 新旧链路如何隔离

当前隔离核心是 `ChargeNotifyAttach.depositPlan`。

基本思路：

1. 新流程在提交支付时，将 `depositPlan=true` 写入 attach
2. 支付成功回调进入 `CrmOrderCloseQueueConsumer`
3. 消费者根据 attach 判断是否转入 `depositPlanChargeService.onPaymentSuccess()`
4. 命中新分支后直接返回，不再进入旧储值逻辑

这意味着：

1. 新流程和旧流程共用支付回调入口
2. 但通过 attach 字段完成路由分流
3. 旧流程 attach 中没有该标记，因此不会误入新逻辑

这是当前方案最合理、也最值得保留的设计点之一。

## 4. 已有实现能力

## 4.1 接口层

当前主文档中明确存在的接口包括：

1. `POST /deposit-plan/charge/commit`
2. `POST /deposit-plan/charge/query_charge_status`

它们分别负责：

1. 提交新储值方案购买请求
2. 供前端轮询支付到账状态

## 4.2 数据模型与表变更

当前方案引入的关键新表：

| 表 | 用途 |
|---|---|
| `crm_deposit_charge_record` | 购买记录，用于记录会员买了哪个方案的哪个档位，并支撑限购统计 |
| `crm_deposit_coupon_schedule` | 周期发券计划，用于后续按计划兑现权益 |

## 4.3 主要服务职责

| 组件 | 作用 |
|---|---|
| `DepositPlanChargeService` | 主提交流程、支付成功后主处理 |
| `DepositPlanCouponService` | 赠品发放、周期计划处理、会员升级相关能力 |
| `DepositPlanPurchaseLimitChecker` | 购买限制校验 |
| `DepositPlanChargeController` | 新链路控制器 |
| `DepositPlanCouponScheduleJob` | 周期发券定时任务 |

## 4.4 主流程已经覆盖到的能力

从已有文档可确认，当前代码至少覆盖了下面这些环节：

1. 独立提交充值请求
2. 校验方案与档位
3. 校验购买限制
4. 调旧支付通道发起支付
5. 创建交易任务
6. 创建购买记录
7. 支付成功后复用旧余额引擎入账
8. 支付成功后执行赠品发放和周期发券计划处理
9. 给前端提供查询支付状态的接口

也就是说，主链路不是概念设计，而是已经具备实装基础。

## 5. 核心流程说明

## 5.1 `commit` 提交流程

建议把它理解成 4 个阶段：

### 阶段 1：接收入参并定位购买上下文

需要拿到至少这些信息：

1. 当前会员
2. 方案 ID
3. 档位索引
4. 当前渠道
5. 当前门店
6. 可能还包括品牌或区域上下文

### 阶段 2：方案与档位校验

至少要确认：

1. 方案存在
2. 方案启用
3. 档位存在
4. 当前时间、渠道、门店、品牌、会员等级是否可买
5. 限购是否允许

### 阶段 3：落交易与购买记录

通常要完成：

1. 创建 `CrmDealTask`
2. 创建 `CrmDealTaskItem`
3. 创建 `crm_deposit_charge_record`
4. 调支付通道生成支付单

### 阶段 4：返回支付参数

前端拿到支付参数后发起支付，并再通过状态查询接口轮询结果。

## 5.2 `onPaymentSuccess` 支付成功处理流程

该流程的职责是把“方案购买成功”兑现为“账户权益到账”。

至少包含：

1. 读取交易任务和 attach
2. 确认该单属于 `depositPlan` 新链路
3. 调用旧余额引擎更新余额
4. 更新购买记录状态
5. 发放一次性赠品
6. 为周期权益创建或推进发放计划
7. 必要时执行会员升级
8. 更新方案销量统计

这里的关键难点不是串起来，而是保证幂等和状态闭环。

## 5.3 周期发券逻辑

当前建模方向本身是合理的，因为它使用快照保存购买时档位配置。

这样做的好处：

1. 方案后续修改不会污染已购用户权益
2. 周期任务可以按快照独立执行
3. 追责和排障时更容易解释“为什么这个用户拿到的是这个权益”

但要注意：建模合理，不等于时间语义已经完整落地。

---

## 6. 已实现与未实现要明确分开

## 6.1 可以视为已经落地的能力

1. 新旧充值流程的路由隔离
2. 独立充值提交接口
3. 独立充值状态查询接口
4. 新购买记录表与周期发券计划表
5. 复用旧支付通道
6. 复用旧余额引擎
7. 新赠品服务与新定时任务骨架

## 6.2 文档和实现中已经暴露的未完成项

以下内容不能按“已完整支持”理解：

1. 等级延期逻辑仍有 TODO
2. 渠道校验未必真正执行到位
3. 品牌 / 区域规则未必真正进入运行时校验
4. 关单后购买记录状态回写未闭环
5. 周期发券的发放时间语义没有完全兑现
6. 一些券项高级有效期能力未完全落地

这部分一定要在接手时区分清楚，否则最容易把“有模型字段”误判为“已实现能力”。

---

## 7. 当前上线阻塞项

以下问题在已有评估中都属于不建议忽略的阻塞项。

## 7.1 并发限购并没有真正闭环

问题本质：

1. 常见实现是先校验限购，再创建记录
2. 如果没有方案级并发控制，不同请求可以同时通过校验
3. 只在支付成功后锁会员卡，无法解决“提交阶段超卖”

影响：

1. `totalSaleLimit` 可能被超卖
2. `totalSalePeriodLimit` 可能在高并发下失真
3. 同一会员快速重复点击，也可能在支付前绕过限购

## 7.2 关单后购买记录状态未回写

问题本质：

1. 延时关单可能只关闭支付单或任务单
2. 没有同步将 `crm_deposit_charge_record.trade_state` 改成已关闭

影响：

1. 支付中残留记录长期占用名额
2. 周期限购和总量限购被脏数据污染
3. 规则统计逐渐偏离真实业务

## 7.3 品牌 / 区域 / 渠道规则没有完全进入实际校验

典型表现：

1. `brandLid` 传 `null`
2. 区域规则直接放过
3. 渠道字段已在 DTO 里，但未必真正参与 `isPlanAvailable()`

影响：

1. 运营配置了规则，运行时却不生效
2. 前端误以为系统支持差异化售卖，实际未落地

## 7.4 周期发券的“时间”语义不完整

问题本质不是“会不会发”，而是“是否按配置的时间准确发”。

如果只有计划表和任务，没有明确的调度语义，容易出现：

1. 发券时间偏早或偏晚
2. 周期边界不一致
3. 重试与重复执行难以解释

## 7.5 会员等级延期仍未完整实现

如果档位支持“升级并延期”，但代码只做升级、不做延期，会导致配置语义和实际权益不一致。

---

## 8. 中高风险项

除了明确阻塞项，下面这些问题也需要提前正视。

## 8.1 销量统计存在丢失更新风险

如果 `total_sold_count`、`total_sold_amount` 采用普通读改写，在并发下会丢失更新。

## 8.2 周期任务缺少并发抢占保护

如果同一批计划可能被多个调度实例同时消费，会出现重复发放或幂等复杂度上升的问题。

## 8.3 `query_charge_status` 依赖旧引擎状态语义

这不是不能用，而是要明确：

1. 新链路查询能力可能仍依赖旧 Redis 标记或旧任务状态
2. 排障时不能只看新表，必须回看旧交易引擎状态

---

## 9. 推荐修订方向

## 9.1 第一优先级：补业务正确性

建议先做这几件事：

1. 关单后同步回写购买记录状态
2. 增加方案级并发控制
3. 将销量统计改为原子更新
4. 保证支付成功回调幂等

## 9.2 第二优先级：补规则完整性

建议补齐：

1. 渠道规则校验
2. 品牌规则校验
3. 区域/门店规则校验
4. 等级延期逻辑

## 9.3 第三优先级：补调度与排障能力

建议补齐：

1. 周期任务并发保护
2. 核心日志与监控
3. 排障 SQL 与状态查询工具

---

## 10. 日期校验修改应如何理解

此前已有单独文档说明“开始日期和结束日期为同一天时不应报错”。这部分应视为本链路规则正确性的一部分，而不是孤立的改单。

## 10.1 变更内容

从语义上看，校验规则应从：

```text
结束时间必须严格晚于开始时间
```

调整为：

```text
结束时间不能早于开始时间
```

这样才能支持：

1. 同一天活动
2. 同一时刻开始和结束的特殊活动
3. 单日促销配置

## 10.2 影响范围

主要影响：

1. 储值方案创建
2. 储值方案修改

不影响：

1. 查询类能力
2. 删除类能力
3. 既有正确数据

这类变更虽然小，但很典型地说明：规则配置层的语义要和真实业务保持一致，不能为了简化实现而限制运营能力。

## 11. 建议测试清单

## 11.1 正常流程

1. 正常提交流程
2. 正常支付成功到账
3. 一次性赠品正常发放
4. 周期计划正常创建

## 11.2 异常流程

1. 支付超时后自动关单
2. 重复回调不重复入账
3. 发券失败不影响余额入账
4. 非法档位或非法方案被正确拒绝

## 11.3 并发流程

1. 同一会员短时间重复提交
2. 多会员并发抢最后一个限量名额
3. 周期限购边界并发测试
4. 销量统计并发准确性测试

## 11.4 规则流程

1. 指定渠道可买、非指定渠道不可买
2. 指定品牌可买、非指定品牌不可买
3. 指定门店可买、非指定门店不可买
4. 指定会员等级可买、非指定等级不可买

## 11.5 回归流程

1. 旧接口 `/charge/*` 不受影响
2. 旧充值发券逻辑不受影响
3. 旧余额引擎行为不被新链路破坏

---

## 12. 上线前最低要求

如果要进入灰度或生产，建议至少满足以下条件：

1. 关单状态回写完成
2. 方案级并发控制完成
3. 销量统计原子更新完成
4. 品牌 / 区域 / 渠道规则校验补齐
5. 等级延期明确实现或明确禁用
6. 核心单测、联调测试、并发测试补齐

如果这些条件还没满足，那么最合理的状态描述是“可联调、可验证、暂不建议直接上线”。

---

## 13. 接手开发时的阅读顺序

建议按以下顺序看代码和文档：

1. 先看 `01-新会员储值系统开发文档.md`，理解方案模型和规则语义
2. 再看本文，理解接入链路、实现现状与风险
3. 然后看 `02-旧会员储值系统开发文档.md`，理解被复用的旧支付与余额能力

这样能避免一上来就陷入单个 service 或单张表里。

---

## 14. 一句话总结

这条充值集成链路目前最大的优点是“新旧隔离清晰、复用旧能力务实”，最大的短板是“规则完整性和状态闭环还不够硬”。接手开发时，优先补一致性、限购、关单、规则传参，再谈上线。
