# 新会员储值系统开发文档

> 范围：基于 `crm_deposit_plan` 的新会员储值体系。
>
> 目标读者：后端开发、联调开发、测试、排障人员。
>
> 不包含：新链路支付接入细节的实现现状和风险，这部分请看 `03-储值方案充值集成-开发实现与风险说明.md`。

---

## 1. 系统定位

新会员储值系统的核心特点不是“只是多一个储值接口”，而是把储值规则建模成了可扩展的方案体系：

1. 用 `crm_deposit_plan` 表达方案主体。
2. 用多个 JSON 规则字段表达时间、渠道、品牌、门店、会员等级、购买限制、档位权益。
3. 允许一个方案下配置多个档位，每个档位对应不同的本金、赠送、积分、优惠券、券包、会员升级等权益。
4. 方案本身是“规则定义层”，支付、入账、余额扣减、赠品发放属于“执行层”。

这套模型相比旧系统更灵活，但复杂度明显更高，开发重点不在 CRUD，而在“规则判断是否完整、执行是否与规则语义一致”。

## 2. 必须先理解的核心概念

### 2.1 方案与档位

- `crm_deposit_plan` 表示一个储值方案，例如“新人首充礼”“周末加赠”“VIP 专享储值”。
- `tier_rule` 表示该方案下的多个档位，例如：
  - 储值 100 赠 20
  - 储值 500 赠 150
  - 储值 1000 赠 300 并送券包

方案负责定义“可不可以买”，档位负责定义“买了给什么”。

### 2.2 本金、赠送、总余额

新系统的金额模型仍然沿用储值领域的基本公式：

```text
total = principal + gift
```

- `principal`：会员实付形成的本金余额
- `gift`：活动赠送形成的赠送余额
- `total`：可用总余额

消费时必须遵守既定顺序：

```text
优先扣 gift，再扣 principal
```

### 2.3 一次性发放与周期性发放

档位权益既可以一次性到账，也可以按周期分批发放。

- 一次性发放：支付成功后立即发放
- 周期性发放：支付成功后生成后续发放计划，由定时任务逐次发放

开发时要区分两层语义：

1. 余额是否立即到账
2. 赠券/券包/积分是否立即到账

### 2.4 方案可用性与购买限制不是一回事

方案能否购买，至少要过两类检查：

1. 可用性规则：当前时间、当前渠道、当前门店、当前品牌、当前会员等级是否符合方案定义
2. 购买限制：当前会员或全局是否已经达到购买次数/销售次数上限

很多问题都是把这两层混在一起导致的。

## 3. 核心表与关系

## 3.1 配置层

| 表 | 作用 | 关键点 |
|---|---|---|
| `crm_deposit_plan` | 储值方案主表 | 最核心；包含多个 JSON 规则字段 |
| `crm_deposit_category` | 储值方案分类 | 用于方案展示和管理分组 |
| `crm_deposit_agreement` | 储值协议 | 方案可要求用户确认协议 |

## 3.2 执行层

| 表 | 作用 | 关键点 |
|---|---|---|
| `crm_card_balance` | 会员卡余额汇总 | `principal`、`gift`、`total` 与 `revision` 是核心字段 |
| `crm_card_balance_task` | 余额操作任务 | 记录一次储值/消费/退款的任务上下文 |
| `crm_card_record` | 余额交易流水 | 最适合对账和排查前后余额变化 |
| `crm_member_gift_record` | 赠品发放记录 | 记录优惠券等权益发放 |

## 3.3 关键关系

```text
crm_deposit_plan
  ├─ category_lid -> crm_deposit_category.lid
  ├─ agreement_lids[] -> crm_deposit_agreement.lid
  └─ tier_rule / purchase_limit_rule / channels / member_level_rule ...

支付成功后
  -> crm_card_balance 更新余额
  -> crm_card_balance_task 记录本次操作
  -> crm_card_record 记录前后余额变化
  -> crm_member_gift_record 记录赠券等权益
```

需要特别注意：新方案表本身并不直接承担资金账务，它定义规则；真正资金变化仍在卡余额体系中落账。

## 4. `crm_deposit_plan` 的关键字段

`crm_deposit_plan` 是整个新系统最值得重点阅读的表。开发接手时，优先确认下面几类字段。

### 4.1 基础字段

| 字段 | 说明 |
|---|---|
| `mid` / `sid` / `lid` | 商户、门店、业务主键 |
| `name` | 方案名 |
| `description` | 方案说明或营销文案 |
| `image_url` | 前端展示图 |
| `sort_no` | 展示排序 |
| `plan_type` | 方案类型 |
| `category_lid` | 分类 |
| `status` | 0 禁用，1 启用 |

### 4.2 日期与协议

| 字段 | 说明 |
|---|---|
| `date_type` | 1 指定时间范围，2 永久有效 |
| `begin_time` / `end_time` | 生效区间 |
| `agreement_enabled` | 是否启用协议 |
| `agreement_lids` | 关联协议列表 |

### 4.3 JSON 规则字段

| 字段 | 作用 |
|---|---|
| `tier_rule` | 档位与权益配置，最核心 |
| `purchase_limit_rule` | 购买限制规则 |
| `cycle_rule` | 周期规则 |
| `time_periods` | 时段规则 |
| `exclude_dates` | 排除日期 |
| `channels` | 渠道规则 |
| `applicable_orgs` | 组织/门店/区域规则 |
| `brand_rule` | 品牌规则 |
| `member_level_rule` | 会员等级规则 |

### 4.4 统计与展示字段

| 字段 | 说明 |
|---|---|
| `total_sold_count` | 已售次数 |
| `total_sold_amount` | 已售金额 |
| `recommended` | 是否推荐 |
| `created_time` / `updated_time` | 审计字段 |

## 5. 方案可用性的判断顺序

开发实现 `listAvailable()` 或购买前校验时，建议固定采用下面顺序，避免漏判：

1. 方案状态必须为启用
2. 若是指定日期范围，当前时间必须在 `begin_time ~ end_time` 之间
3. 当前日期必须命中 `cycle_rule`
4. 当前日期不能出现在 `exclude_dates`
5. 当前时间必须命中 `time_periods`
6. 当前渠道必须命中 `channels`
7. 当前门店/区域必须命中 `applicable_orgs`
8. 当前品牌必须命中 `brand_rule`
9. 当前会员等级必须命中 `member_level_rule`
10. 通过后，再进入 `purchase_limit_rule` 检查

如果系统里出现“配置了规则但用户仍然可以买”的问题，优先排查的是第 6 到第 9 步，因为这些规则最容易出现“文档有、模型有、运行时没真正传参”的情况。

## 6. 最重要的 JSON 规则

## 6.1 `tier_rule`

这是新系统的核心中的核心，决定档位和权益。

一个档位通常至少包含：

1. 储值本金
2. 赠送金额
3. 赠送积分
4. 赠送优惠券或券包
5. 是否升卡
6. 是否延期
7. 发放方式：一次性或周期性

开发上要重点关注三类问题：

1. 档位索引是否稳定
2. 支付成功后是否按购买时快照兑现，而不是按当前最新方案兑现
3. 周期发放配置是否真的被执行引擎完整支持

## 6.2 `purchase_limit_rule`

这类规则通常包含四层限制：

1. `perUserTotalLimit`：单个会员总购买上限
2. `perUserPeriodLimit`：单个会员按天/周/月的购买上限
3. `totalSaleLimit`：全局总销售上限
4. `totalSalePeriodLimit`：全局按天/周/月的销售上限

这里最容易出事故的是并发：

1. 先查再写时可能被并发绕过
2. 只锁会员维度无法防止全局超卖
3. 支付中记录如果不闭环，会污染限购统计

## 6.3 `cycle_rule`

常见语义：

- `DAILY`
- `WEEKLY`
- `MONTHLY`

它解决的是“哪些日期可买”，不是“哪些时间点发券”。

## 6.4 `time_periods`

用于控制某天中的可买时段，例如：

- 09:00-12:00
- 15:00-18:00

开发时要统一分钟制或统一时间对象，避免边界值判断不一致。

## 6.5 `exclude_dates`

用于表达黑名单日期，例如法定假日、停业日、特殊不可售卖日。

## 6.6 `channels`

常见渠道包括：

- POS
- 小程序
- H5

注意：文档配置了渠道，不代表实现已经真正用请求中的 `channelCode` 做过滤；这一点需要结合接入实现再核查。

## 6.7 `applicable_orgs` 与 `brand_rule`

这两类规则经常决定“某品牌可买、某门店不可买”，但在开发实现里最容易被临时跳过或传 `null`。

## 6.8 `member_level_rule`

用于限定会员等级。它和“支付成功后升级等级”属于前后两个阶段：

1. 购买前判断是否有资格买
2. 购买后判断是否要升级会员等级

## 7. 核心业务流程

## 7.1 方案配置流程

```text
创建方案
  -> 填基础信息
  -> 配有效期
  -> 配周期/时段/排除日期
  -> 配渠道/门店/品牌/等级
  -> 配购买限制
  -> 配档位权益
  -> 保存并启用
```

这一阶段的重点不是页面字段能不能保存，而是“配置项是否能被后续购买链路完整解释”。

## 7.2 可用方案查询流程

```text
接收会员、门店、品牌、渠道、当前时间
  -> 逐条过滤可用性规则
  -> 过滤限购规则
  -> 返回可买档位和展示信息
```

开发排查“为什么方案不显示”时，建议把每一步过滤结果打日志，而不是只看最终布尔值。

## 7.3 支付成功后的入账流程

支付成功后，至少要完成以下动作：

1. 更新 `crm_card_balance`
2. 记录 `crm_card_balance_task`
3. 记录 `crm_card_record`
4. 发放赠送金额、积分、优惠券、券包
5. 必要时触发升卡
6. 更新方案统计

如果是周期发放方案，不能把“余额已到账”和“后续券已排程”混为一谈，二者需要分别确认。

## 7.4 余额消费流程

```text
会员消费
  -> 检查 total 是否足够
  -> 优先扣 gift
  -> 再扣 principal
  -> 更新汇总余额
  -> 记录 task 和 record
```

这条规则必须在所有消费入口上保持一致，否则账务会出现“汇总对、分项不对”的情况。

## 8. 典型业务场景

### 8.1 新人首充礼

特点：

1. 只允许首次购买
2. 一般会用到 `perUserTotalLimit = 1`
3. 常附带送券、升卡或加赠金额

### 8.2 周末促销

特点：

1. 使用 `cycle_rule = WEEKLY`
2. 周六周日可用
3. 常配合 `time_periods` 限时售卖

### 8.3 VIP 分层储值

特点：

1. 使用 `member_level_rule`
2. 不同会员等级看到不同方案
3. 购买后还可能触发升级

### 8.4 渠道差异化储值

特点：

1. 小程序可买，POS 不可买
2. H5 和小程序文案一致，但支付链路可能不同
3. 需要严格核实渠道参数是否真正进入规则判断

### 8.5 限量储值

特点：

1. 使用 `totalSaleLimit`
2. 高并发场景下最容易出现超卖
3. 需要原子更新或方案级并发控制

## 9. 开发最容易踩的坑

1. 只校验了方案状态和日期，没有校验品牌、门店、渠道、会员等级。
2. 购买限制采用“先查后写”，并发下不准确。
3. 关闭未支付订单后，没有把占用名额释放。
4. 周期发放只建模了计划，没有真正兑现时间语义。
5. 方案更新后，已购用户权益被新配置污染，没有使用购买时快照。
6. 余额汇总更新了，但流水或赠品记录没同步落库。
7. 只看 `total`，没核对 `principal` 和 `gift` 的拆分是否正确。

## 10. 排障建议

### 10.1 查某方案为什么买不到

建议按下面顺序查：

1. 方案是否启用
2. 日期是否在有效期
3. 当前日期是否命中周期规则
4. 当前时间是否命中时段规则
5. 当前渠道、门店、品牌、会员等级是否都命中
6. 是否触发个人限购或全局限购

### 10.2 查某次充值后余额是否正确

至少核对：

1. `crm_card_balance`
2. `crm_card_balance_task`
3. `crm_card_record`
4. 赠品记录表

### 10.3 常用 SQL 思路

```sql
-- 查方案基础信息
select lid, name, status, begin_time, end_time, total_sold_count, total_sold_amount
from crm_deposit_plan
where lid = ?;

-- 查会员当前余额
select lid, principal, gift, total, revision
from crm_card_balance
where lid = ?;

-- 查最近余额流水
select task_lid, operation_model, amount, principal_amount_before, principal_amount_after,
       give_amount_before, give_amount_after, balance_before, balance_after, created_time
from crm_card_record
where card_lid = ?
order by created_time desc;
```

## 11. 与旧系统的关系

新系统不是推翻旧余额体系，而是“新规则模型 + 旧稳定能力”的组合：

1. 规则配置使用 `crm_deposit_plan`
2. 部分执行层仍复用旧支付和余额能力
3. 因此理解新系统时，旧系统文档仍然有价值

建议配合阅读：

1. `02-旧会员储值系统开发文档.md`
2. `03-储值方案充值集成-开发实现与风险说明.md`

---

## 12. 一句话总结

新会员储值系统的难点不在表结构本身，而在“规则定义是否被执行链路完整兑现”。只要开发能把可用性、限购、入账、赠品、快照、并发这几层理顺，这套系统就能读懂也能接手。
