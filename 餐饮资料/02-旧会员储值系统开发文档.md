# 旧会员储值系统开发文档

> 范围：旧生产储值链路，核心规则实体为 `CrmSaveRule`。
>
> 目标读者：需要维护旧接口、排查旧储值问题、理解新系统复用能力来源的开发人员。

---

## 1. 系统定位

旧会员储值系统是当前生产稳定能力的主要来源。新储值方案链路虽然引入了新的规则模型，但支付、余额、部分交易状态查询能力仍然明显继承或复用旧体系。

旧系统的特点是：

1. 规则模型较扁平，以 `CrmSaveRule` 为中心。
2. 支付、到账、余额流水、退款、对账链路已经比较稳定。
3. 业务可扩展性不如新系统，但执行闭环更成熟。

如果开发目标是“快速定位线上余额或订单问题”，旧系统往往比新系统更应该优先读。

---

## 2. 核心表链路

## 2.1 规则层

| 表 | 作用 |
|---|---|
| `crm_save_rule` | 储值规则主表，定义本金、赠送、优惠券、有效期、门店限制等 |
| `crm_save_rule_interval` | 阶梯式储值规则，用于按金额区间给不同赠送 |

## 2.2 账户与流水层

| 表 | 作用 |
|---|---|
| `crm_card` | 会员卡余额汇总表 |
| `crm_card_balance` | 单次余额变动记录 |
| `crm_card_record` | 最完整的交易审计流水 |

## 2.3 交易任务层

| 表 | 作用 |
|---|---|
| `crm_deal_task` | 交易主任务 |
| `crm_deal_task_item` | 交易明细 |

## 2.4 关系速记

```text
crm_save_rule / crm_save_rule_interval
  -> ChargeService.commit()
  -> crm_deal_task / crm_deal_task_item
  -> 支付成功
  -> crm_card / crm_card_balance / crm_card_record
  -> 优惠券 / 积分 / 升卡等权益
```

---

## 3. 关键业务概念

### 3.1 余额拆分

旧系统常见字段命名与新系统略有区别，但本质相同：

```text
balance = principal_balance + give_balance
```

- `principal_balance`：本金余额
- `give_balance`：赠送余额
- `balance`：总可用余额

消费顺序仍然是：

```text
先扣 give_balance，再扣 principal_balance
```

### 3.2 储值规则与档位规则

旧系统常见两种表达方式：

1. 单条 `CrmSaveRule` 直接表达一个套餐
2. `crm_save_rule_interval` 用于表达区间赠送规则

与新系统相比，旧系统的规则更扁平，灵活度较弱，但运行时解释成本更低。

### 3.3 交易任务是排查主线

排查旧储值问题时，`crm_deal_task` 往往是最佳起点，因为它连接了：

1. 规则
2. 支付状态
3. 会员卡
4. 流水
5. 退款/撤销

---

## 4. 核心流程

## 4.1 规则配置流程

```text
商户配置 CrmSaveRule
  -> 设置本金、赠送、优惠券、有效期、适用门店等
  -> 启用规则
  -> 前端列表可见
```

旧系统规则字段不少，但总体属于“单规则单套餐”的思路，复杂度低于新系统的多 JSON 组合规则。

## 4.2 会员储值流程

```text
会员选择储值规则
  -> ChargeService.commit()
  -> 计算本金/赠送/积分/优惠券
  -> 创建 crm_deal_task
  -> 创建 crm_deal_task_item
  -> 发起支付
  -> 支付成功回调
  -> ChargeService.success()
  -> 更新 crm_card
  -> 写 crm_card_balance
  -> 写 crm_card_record
  -> 发放券/积分/升级权益
```

这是旧系统最重要的一条链路，也是很多线上问题的排查主路径。

## 4.3 余额消费流程

```text
会员消费
  -> 检查 card 余额
  -> 优先扣赠送余额
  -> 再扣本金余额
  -> 更新 crm_card
  -> 记录变动和流水
```

消费逻辑如果不统一，最常见后果就是：

1. 汇总余额对不上
2. 本金/赠送拆分错位
3. 对账时无法解释

## 4.4 退款/撤销流程

```text
发起退款或撤销
  -> 标记交易状态
  -> 扣回卡余额
  -> 触发支付退款
  -> 记录退款流水
```

退款问题排查时，要同时看支付状态、任务状态、卡余额和流水，不要只看单张表。

## 5. 核心服务理解方式

## 5.1 `ChargeService`

这是旧储值主链路的核心服务，通常至少关注这几个能力：

1. `commit(...)`：提交储值
2. `success(...)`：支付成功回调处理
3. `queryChargeStatus(...)`：查询支付状态

理解旧系统时，优先把这三个方法的输入输出和落表动作搞清楚。

## 5.2 `CrmSaveRuleServicePlus`

主要负责规则管理：

1. 规则列表
2. 规则启停
3. 小程序/结账场景的规则查询
4. CRUD 管理

它更偏管理面，而不是支付执行面。

## 6. 关键字段速查

## 6.1 `crm_save_rule`

重点字段建议优先记住：

| 字段 | 说明 |
|---|---|
| `lid` | 规则主键 |
| `name` | 规则名称 |
| `status_` | 启停状态 |
| `save_amount` | 本金 |
| `give_amount` | 赠送金额 |
| `give_point` | 赠送积分 |
| `give_coupon_id` | 赠送券 |
| `begin_time` / `end_time` | 有效期 |
| `apply_to_store` | 适用门店 |
| `upgrade_card_type_code` | 升卡目标 |

## 6.2 `crm_card`

重点不是字段多，而是要记住哪些字段互相约束：

| 字段 | 说明 |
|---|---|
| `balance` | 总余额 |
| `principal_balance` | 本金余额 |
| `give_balance` | 赠送余额 |
| `points` | 当前积分 |
| `sum_of_save` | 累计储值本金 |
| `sum_of_save_give` | 累计赠送 |
| `sum_of_save_times` | 累计储值次数 |
| `revision` | 乐观锁版本号 |

## 6.3 `crm_card_record`

这是对账最关键的表。重点关注：

1. `operation_model`
2. `balance_before` / `balance_after`
3. `principal_amount_before` / `principal_amount_after`
4. `give_amount_before` / `give_amount_after`
5. `amount`
6. `task_lid`

## 7. 常用枚举与状态

## 7.1 操作类型

常见会出现在 `operation_model` 或类似字段中的值：

1. `CHARGE`
2. `CONSUME`
3. `REFUND`
4. 以及其他充值、消费、撤销衍生类型

开发排查时，不要只看金额，还要看操作类型是否符合预期。

## 7.2 支付状态

常见会出现在 `trade_state` 或任务状态中的值：

1. 支付中
2. 支付成功
3. 支付失败
4. 已关闭
5. 已退款

不同表的状态字段不一定完全同名，但语义必须能对齐。

---

## 8. 数据一致性为什么相对稳定

旧系统相对可依赖，主要因为这几层做得更成熟：

### 8.1 乐观锁

余额汇总更新依赖 `revision` 之类的版本号控制，防止并发更新覆盖。

### 8.2 任务与流水较完整

旧系统通常能同时留下：

1. 交易任务
2. 任务明细
3. 单次余额变动
4. 前后余额流水

因此更容易审计和回放问题。

### 8.3 规则执行语义更简单

旧规则虽然没那么灵活，但“配置出来的能力”和“运行时真正执行的能力”之间差距更小。

---

## 9. 与新系统的差异

| 维度 | 旧系统 | 新系统 |
|---|---|---|
| 规则模型 | 扁平，围绕 `CrmSaveRule` | 复杂，围绕 `crm_deposit_plan` 多 JSON 规则 |
| 灵活性 | 较低 | 高 |
| 执行闭环成熟度 | 高 | 取决于新接入是否补齐 |
| 适合场景 | 稳定线上能力 | 复杂营销和细粒度规则 |

理解这点很重要：新系统强在“规则表达能力”，旧系统强在“执行闭环成熟度”。

## 10. 线上排障建议

### 10.1 查一笔旧储值为什么没到账

建议顺序：

1. 查 `crm_deal_task`
2. 查 `crm_deal_task_item`
3. 查 `crm_card`
4. 查 `crm_card_balance`
5. 查 `crm_card_record`

### 10.2 查余额为什么不对

先判断是下面哪一类：

1. 汇总没更新
2. 流水没落
3. 本金/赠送拆分错误
4. 退款冲正没做完整

### 10.3 常用 SQL 思路

```sql
-- 查会员卡余额
select lid, balance, principal_balance, give_balance, revision
from crm_card
where lid = ?;

-- 查最近储值任务
select lid, out_trade_no, trade_state, amount, created_time
from crm_deal_task
where card_lid = ?
order by created_time desc;

-- 查最近交易流水
select task_lid, operation_model, amount, balance_before, balance_after,
       principal_amount_before, principal_amount_after,
       give_amount_before, give_amount_after, created_time
from crm_card_record
where card_lid = ?
order by created_time desc;
```

## 11. 什么时候必须回头看旧系统

以下场景建议优先查旧系统逻辑：

1. 新链路复用了旧支付状态查询
2. 新链路复用了旧余额执行引擎
3. 线上实际问题发生在卡余额更新之后
4. 需要判断新链路是否破坏旧链路稳定性

---

## 12. 一句话总结

旧会员储值系统的核心价值不是“规则很多”，而是“账务执行闭环成熟、状态和流水相对完整”。新系统接手时，只要涉及复用支付、余额、到账、退款能力，都应回头对照旧系统实现。
